# deadliner - DEA_021

**deadliner** isn’t just another calendar widget. It’s a deterministic, CLI-first, API‑driven deadline engine designed for developers, DevOps leads, and remote‑first squads who need to track, cascade, and enforce deadlines across microservices, sprints, and delivery pipelines—without bloated UI or manual sync.

---

## 🔧 Three Hardcore Features That Set deadliner Apart

### 1. **Dependency‑Aware Deadline Cascading**
Define deadlines that automatically recalculate when upstream tasks slip. deadliner uses a directed acyclic graph (DAG) to model task dependencies and propagates changes in real‑time via webhook triggers—no manual re‑estimation, no spreadsheet drift.

### 2. **Git‑Backed Audit Trail with `deadliner diff`**
Every deadline update, status change, or reassignment is stored as a timestamped, signed commit in a local Git‑like object store. Run `deadliner diff <task_id> --since 2d` to see exactly who changed what, when, and why—ideal for post‑mortems and compliance.

### 3. **Unix‑Style Piping & CI/CD Integration**
deadliner outputs JSON, YAML, or plain text. Pipe it into `jq`, `grep`, or your CI pipeline:  
```bash
deadliner list --status overdue --format json | jq '.[].task_id'
```
Trigger Slack alerts, update Jira tickets, or block a merge if a deadline is missed—all from a single, composable command.

## 🚀 Three Professional Steps to Get Productive in 5 Minutes

### Step 1: Initialize a Project Workspace
```bash
deadliner init --project my-service --timezone UTC --cascade-depth 3
```
This creates a `.deadliner/` directory with a `manifest.yml` and a local Git‑backed store. The `--cascade-depth` flag pre‑configures how many levels of dependency to auto‑recalculate—critical for multi‑team delivery chains.

### Step 2: Define Tasks with Dependency Graph
```yaml
# tasks/deploy.yml
tasks:
  - id: "backend-api-v2"
    due: "2025-05-20T14:00:00Z"
    depends_on: ["db-migration", "auth-refactor"]
    on_slip: "notify:slack:channel=#dev-alerts"
    on_complete: "trigger:ci:pipeline=deploy-staging"
```
Run `deadliner validate` to check for circular dependencies and enforce ISO 8601 timestamps. deadliner will reject any task with a self‑referencing `depends_on`.

### Step 3: Integrate with Your CI/CD Pipeline
Add this to your `.github/workflows/deadline-check.yml`:
```yaml
- name: Deadline Gate
  run: |
    deadliner check --task backend-api-v2 --fail-if-overdue
    deadliner report --format markdown >> $GITHUB_STEP_SUMMARY
```
If `backend-api-v2` is overdue, the pipeline fails immediately. The report is appended to the GitHub Actions summary—no extra dashboard needed.

## 🏆 Two Industry Best Practices from Production Deployments

### Practice 1: **Use `deadliner watch` for Real‑Time Slack/Teams Integration**
Instead of polling, run `deadliner watch --webhook https://hooks.slack.com/...` in a background process. deadliner emits events (task created, slipped, completed) as structured JSON payloads. This reduces latency from minutes to milliseconds and eliminates cron‑based checkers.

### Practice 2: **Adopt the “Three‑Layer Deadline” Pattern**
- **Layer 1 (External)**: Public milestones (client deliverables, go‑live dates).  
- **Layer 2 (Internal)**: Engineering milestones (code freeze, QA sign‑off).  
- **Layer 3 (Hidden)**: Technical debt deadlines (dependency upgrades, security patches).  
Use `deadliner tag --layer hidden` to keep Layer 3 invisible in standard reports but enforce blocking gates in CI. This prevents scope creep from surfacing as missed client deadlines.
