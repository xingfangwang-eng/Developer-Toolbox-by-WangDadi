# deadliner - DEA_007

## Overview

deadliner is a precision-engineered deadline orchestration tool built for developers who refuse to let project timelines slip. It combines deterministic scheduling with intelligent buffering to keep your delivery dates locked in, even when dependencies shift. This is not a generic to-do list—it's a time-bound execution framework designed for CI/CD pipelines, sprint planning, and multi-repo coordination.

## Core Features

### 1. Dependency-Aware Deadline Propagation
Deadliner automatically recalculates downstream deadlines when a blocker is resolved or an upstream task shifts. Instead of manually adjusting every linked subtask, the engine propagates changes through a directed acyclic graph (DAG) of dependencies, respecting slack time and critical path constraints.

### 2. Slack-Buffer Auto-Tuning
The tool analyzes historical velocity data from your team’s commits and PR merge times to dynamically adjust buffer windows per task. It doesn't just add 20% overhead—it applies a weighted regression model to predict realistic completion windows, then flags tasks where slack is dangerously low.

### 3. Deadline Drift Alerts with Root-Cause Tracing
When a deadline is at risk, deadliner generates an alert that includes a traceback of contributing factors: delayed upstream tasks, excessive WIP, or unplanned context switches. Each alert links directly to the relevant commit or ticket, so you can jump straight into remediation without digging through logs.

## How to Use

### 1. Initialize a Deadline DAG
```bash
deadliner init --project "Q4-release" --deadline "2025-12-15T18:00:00Z" --deps "backend:2025-11-30,frontend:2025-12-05"
```
This creates a `.deadliner.yml` manifest in your repo root. The engine parses the dependency graph and calculates the critical path. Run `deadliner status` to see a Gantt-style output of every task with its slack time and buffer-adjusted deadline.

### 2. Integrate with Your CI/CD Pipeline
Add a post-merge hook to your GitHub Actions or GitLab CI:
```yaml
- name: Update deadline status
  run: deadliner track --task "deploy-v2.1" --status "completed" --duration 45m
```
The engine updates the DAG and recalculates all downstream deadlines. If a sibling task’s slack drops below 10%, deadliner pushes a comment to the associated PR thread with a recommended reprioritization.

### 3. Generate a Recovery Plan on Drift
When a deadline alert fires, run:
```bash
deadliner recover --task "auth-service" --options "swap,reduce,skip"
```
The tool outputs three ranked recovery strategies based on your current velocity and dependency constraints. Each strategy includes a revised deadline and a list of tasks to deprioritize or parallelize. Apply a strategy with `deadliner apply --strategy 2` to update the manifest and notify all stakeholders via webhook.

## Best Practices

### Practice 1: Use Deadline Locks for External Dependencies
If your project relies on a third-party API or a vendor milestone, mark that task with `--lock true` in the manifest. Deadliner will treat it as an immovable anchor and never auto-adjust its deadline. This prevents cascading schedule creep when external factors change, forcing the team to buffer internally instead.

### Practice 2: Run a Weekly Slack Audit
Every Friday, execute `deadliner audit --threshold 15`. This scans your active projects and surfaces tasks where slack has dropped below 15% of the original buffer. Use the output to identify systemic bottlenecks (e.g., a specific team member consistently overloaded) and redistribute work before deadlines become critical. Combine this with a 5-minute standup to review the audit results and adjust priorities.
