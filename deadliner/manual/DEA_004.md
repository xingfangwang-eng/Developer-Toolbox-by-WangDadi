## What is deadliner?

deadliner is a lightweight, API-first deadline orchestration engine built for developers who are tired of flaky calendar hacks and bloated project management overlays. It decouples deadline logic from your app’s core, giving you a deterministic, timezone-aware backbone for scheduling, escalation, and dependency resolution. Think of it as a cron job on steroids, but purpose-built for human-facing deadlines.

## Core Features

### 1. Dynamic Deadline Recalculation with Dependency Graph
deadliner automatically recalculates all downstream deadlines when a parent task shifts. Using a directed acyclic graph (DAG) model, it propagates changes through your task chain and surfaces critical path violations in real time. No more manual “if this slips, push everything” logic.

### 2. Timezone-Aware Escalation Engine
Built-in escalation policies that respect local business hours, holidays, and timezone offsets. You define rules like “escalate after 2 hours of inactivity during working hours,” and deadliner triggers webhooks, Slack messages, or custom callbacks—without you writing a single scheduling daemon.

### 3. Immutable Deadline Audit Trail
Every deadline mutation is logged as an immutable event in a structured audit log. This isn’t just for compliance—it’s for replaying historical states, debugging why a deadline blew up, and generating precise “time-to-close” analytics for your team.

## How to Use

### 1. Initialize a Deadline Context with Custom Timezone
```python
from deadliner import DeadlineContext, TimezoneConfig

config = TimezoneConfig(
    timezone="America/New_York",
    business_hours=(9, 18),
    holidays=["2025-12-25", "2025-01-01"]
)

ctx = DeadlineContext(
    project_id="proj_alpha",
    config=config,
    dependencies=["task_01", "task_02"]
)
```
This creates a fully isolated deadline environment. All subsequent operations—creation, escalation, recalculation—are scoped to this context.

### 2. Attach a Deadline with Conditional Escalation
```python
deadline = ctx.create_deadline(
    task_id="task_03",
    due_at="2025-04-15T14:00:00Z",
    escalation_policy={
        "type": "relative",
        "trigger": "overdue_by_minutes",
        "threshold": 120,
        "actions": ["webhook:https://hooks.slack.com/...", "email:ops@team.com"]
    }
)
```
The engine validates the due date against your timezone config, checks for conflicts in the dependency graph, and registers the escalation policy as a state machine. No polling required.

### 3. Simulate a Deadline Shift and Review Critical Path
```python
ctx.shift_deadline(task_id="task_01", delta_minutes=45)
impact_report = ctx.critical_path_analysis()
print(impact_report.affected_tasks)  # ["task_02", "task_03"]
print(impact_report.new_risk_score)  # 0.87
```
This triggers a full recalculation. The critical path analysis returns a structured report with affected tasks, new risk scores, and suggested buffer adjustments. All mutations are recorded in the audit trail.

## Best Practices

### 1. Separate Deadline Logic from Business Logic
Don’t embed deadline calculations inside your service layer. Use deadliner as a dedicated sidecar service or library. This gives you a single source of truth for all timing-related decisions, makes testing trivial, and allows you to swap scheduling policies without touching application code.

### 2. Use Immutable Audit Trails for Post-Mortem Analysis
When a deadline misses, don’t guess why. Configure deadliner’s audit log to emit to a structured sink (e.g., PostgreSQL, Elasticsearch). Then run queries like “which tasks had their deadline shifted more than twice” or “what was the state of the dependency graph 24 hours before failure.” This turns operational chaos into measurable patterns.
