# deadliner - DEA_012

## Overview

deadliner is not another calendar app. It's a lightweight, CLI-first, event-driven deadline management engine built for developers who hate missing ship dates, compliance windows, or sprint cutoffs. It integrates directly into your existing CI/CD pipeline, Git hooks, and shell environment—no bloated UI, no manual data entry.

## Core Features

### 1. Git-Aware Deadline Propagation
deadliner hooks into your commit history and branch structure. When a PR is merged, it automatically recalculates dependent deadlines across linked tasks. If `release/v2.1` has a hard code freeze at 2025-06-15T18:00Z, the tool recursively traces all open feature branches and flags any that still have unresolved commits. No more "I didn't know the deadline moved."

### 2. Timezone-Native Cron Scheduler with Graceful Drift Detection
Each deadline is stored as a UTC epoch with an attached timezone descriptor. deadliner runs a lightweight daemon that checks deadline proximity every 60 seconds. If a deadline is missed within a configurable grace window (e.g., 2 hours), it triggers a webhook to your Slack, PagerDuty, or custom HTTP endpoint—but only after verifying the system clock drift is < 500ms against NTP. False alarms eliminated.

### 3. Declarative Deadline DSL via YAML
Define deadlines in a `deadliner.yml` file at your repo root. Supports relative offsets (`+3d`, `-2h`), recurring patterns (`every last Friday of month`), and conditional activation (`if branch == 'main'`). The parser validates the schema at pre-commit hook time. Invalid syntax blocks the commit.

## How to Use

### 1. Bootstrap a Deadline Set in Your Repository
```bash
# Initialize deadliner in the current directory
deadliner init --project "API v3 migration" --deadline 2025-07-01T09:00:00Z --timezone America/New_York
```
This creates a `.deadliner/` directory with a `manifest.json` and a default `deadliner.yml`. The manifest is gitignored by default—only the YAML goes into version control.

### 2. Attach a Deadline to a Specific Branch
```bash
# Bind the deadline to a feature branch
deadliner attach --branch feat/api-v3 --deadline-id dln_abc123 --hard
```
The `--hard` flag means the deadline is non-negotiable; the tool will emit a non-zero exit code in CI if the branch is still open past the cutoff. Use this for regulatory or contractual deadlines.

### 3. Monitor Deadline Health in Real-Time
```bash
# Watch all active deadlines for the current project
deadliner watch --interval 30 --alert webhook
```
This spawns a daemon that polls every 30 seconds. If a deadline is within 1 hour, it sends a pre-alert to the configured webhook. If missed, it sends a breach notification with a diff of what changed since the deadline was set.

## Best Practices

### 1. Always Pin Deadline Definitions to the Main Branch
Keep your `deadliner.yml` on `main` only. Feature branches should never override base deadlines—they should inherit them. This prevents drift when multiple developers edit the same deadline file in parallel. Use `deadliner diff` before merging to see if any deadline boundaries shifted.

### 2. Use Grace Windows for Internal Milestones, Not External Commitments
Set a grace window of 0 for client-facing or compliance deadlines. For internal sprint goals, use a 4-hour grace window combined with a `--soft` flag. This allows the team to absorb minor delays without triggering alerts, while still enforcing hard boundaries for external deliverables.
