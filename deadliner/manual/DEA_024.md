## What is deadliner?

**deadliner** is a high-precision deadline orchestration engine designed for developers, project leads, and remote-first teams who need to track, visualize, and enforce task timelines without the overhead of bloated project management suites. It’s built for the terminal and the browser—lightweight, scriptable, and API-first.

---

## Core Features

### 1. **Smart Dependency Graph with Critical Path Detection**  
deadliner automatically parses task dependencies and computes the **critical path** in real time. When you shift a deadline, it instantly recalculates downstream impacts across your entire project tree—no manual Gantt charting required.

### 2. **Git-Integrated Deadline Anchors**  
Each deadline can be pinned to a specific Git commit, branch, or tag. deadliner monitors your repo and triggers alerts when code changes threaten a deadline (e.g., a merge that introduces 3 new blockers). This turns deadline tracking into a **version-controlled, auditable workflow**.

### 3. **Multi-Time Zone Scheduler with Slack/Webhook Bridge**  
deadliner natively handles UTC, local, and team-specific time zones. It exposes a webhook endpoint that integrates with Slack, Discord, or any custom CI/CD pipeline. You can set **auto-escalation rules** (e.g., "If a deadline is 2 hours away and task is 0% complete, ping the lead and the repo owner").

## How to Use

### 1. **Initialize a Project with a Deadline Manifest**  
Create a `deadliner.yml` file in your project root. Define tasks, owners, estimates (in hours or epochs), and dependencies.  
```yaml
tasks:
  - id: auth-module
    owner: @alice
    estimate: 16h
    depends_on: [db-schema]
  - id: db-schema
    owner: @bob
    estimate: 8h
```
Run `deadliner init` to generate a local SQLite timeline database. deadliner will parse the manifest and build a dependency graph.

### 2. **Track Progress via CLI or API**  
Use `deadliner check` to view a live table of tasks with remaining time, risk level (green/yellow/red), and critical path status.  
For programmatic access, hit the REST API:  
```bash
curl -X GET https://your-deadliner-instance/api/v1/projects/abc123/timeline
```
Returns JSON with task IDs, deadlines, and a `blockers` array listing any unresolved dependencies.

### 3. **Set Automated Escalation Rules**  
Define rules in `deadliner.rules.yml`:  
```yaml
escalations:
  - condition: deadline_within_2h AND progress < 50%
    action: notify_slack("#deadline-alerts", "⚠️ @here: {task.name} is at risk")
  - condition: deadline_passed
    action: webhook("https://hooks.zapier.com/...", { "task": task.id, "status": "overdue" })
```
Run `deadliner watch` to start the background monitor. It checks every 60 seconds and fires webhooks/slack messages when conditions are met.

## Best Practices

### 1. **Anchor Deadlines to Milestones, Not Dates**  
Avoid hard-coding absolute dates. Instead, use relative milestones like `release-v2.0` or `sprint-12-end`. deadliner resolves these to actual timestamps based on your project calendar. This makes replanning painless when priorities shift.

### 2. **Use the Critical Path as Your Daily Standup Guide**  
Instead of asking "What are you working on?", ask "What’s on the critical path?" deadliner surfaces this automatically. Share the output in your team chat every morning. This keeps everyone aligned on the **true bottlenecks**, not just busy work.
