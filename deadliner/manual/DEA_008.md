## What is deadliner?

`deadliner` is not your grandfather’s task scheduler. It is a **deterministic deadline orchestration engine** built for developers who treat time as a first-class dependency. Unlike bloated project management suites, `deadliner` operates as a **CLI-first, API-native, conflict-aware timeline manager** that integrates directly into your CI/CD pipeline, git hooks, and calendar backends. It eliminates the guesswork from deadline management by enforcing **hard constraints, soft warnings, and automated escalation policies**—all from a single YAML or TOML configuration file.

## 3 Hardcore Functional Differentiators

### 1. **Dependency-Aware Deadline Propagation (DADP)**
Deadlines are rarely independent. `deadliner` automatically recalculates all downstream deadlines when a predecessor shifts. Using a **directed acyclic graph (DAG) model**, it propagates changes through your project’s timeline with configurable slack buffers. No more manual “if this slips, then that slips” spreadsheets. The engine emits a **diff log** showing exactly which deadlines moved and by how much.

### 2. **Time-Zone–Resilient Deadline Anchoring**
When your team spans San Francisco, Berlin, and Tokyo, “end of day” is meaningless. `deadliner` anchors every deadline to a **canonical UTC timestamp** but renders it in each developer’s local TZ via the `deadliner show` command. It also supports **floating deadlines** (e.g., “48 hours from now”) and **absolute anchors** (e.g., “2025-11-30T23:59:59Z”). The engine warns you if a deadline falls into a weekend or holiday in any team member’s locale.

### 3. **Git-Hook–Integrated Escalation Pipeline**
Deadlines are useless if nobody gets notified. `deadliner` ships with a **pre-commit hook** that scans your commit message for deadline tags (e.g., `DEADLINE:2025-12-01`). If a commit pushes past the deadline, the hook can:
- Block the commit (hard mode)
- Append an automatic “late” label to the commit (soft mode)
- Trigger a webhook to Slack, PagerDuty, or a custom endpoint

This turns your version control system into a **real-time compliance enforcement layer**.

## 3 Detailed Operational Workflows

### Workflow 1: Bootstrap a Multi-Phase Project with Slack Buffers

```bash
# Initialize a new project timeline
deadliner init --project "API v2 Migration" --tz America/New_York

# Define phases with slack buffers (in days)
deadliner phase add --name "Database Schema Migration" \
  --duration 14d --slack 3d --depends-on ""

deadliner phase add --name "Backend Refactor" \
  --duration 21d --slack 5d --depends-on "Database Schema Migration"

deadliner phase add --name "Frontend Integration" \
  --duration 10d --slack 2d --depends-on "Backend Refactor"

# Set a hard deadline for the final phase
deadliner deadline set --phase "Frontend Integration" \
  --absolute 2025-12-15T23:59:59Z --escalation slack

# View the propagated timeline
deadliner timeline --format table
```

**Output:** The engine calculates start/end dates for each phase, applies slack, and shows the critical path. If `Database Schema Migration` slips by 2 days, `deadliner` automatically shifts `Backend Refactor` and `Frontend Integration` accordingly.

### Workflow 2: Integrate with GitHub Issues for Real-Time Sync

```bash
# Link a GitHub milestone to a deadliner phase
deadliner link --phase "Backend Refactor" \
  --github-repo "myorg/api-v2" \
  --milestone "M5: Refactor Core" \
  --token $GITHUB_TOKEN

# Automatically update deadline when milestone due date changes
deadliner watch --phase "Backend Refactor" --interval 5m

# Trigger a webhook when deadline is within 48 hours
deadliner deadline hook --phase "Backend Refactor" \
  --condition "within 48h" \
  --url "https://hooks.slack.com/services/T00/B00/xxx" \
  --payload '{"text": "⚠️ Backend Refactor deadline approaching in 48 hours"}'
```

**Result:** Every time the GitHub milestone due date changes (via UI or API), `deadliner` updates its internal DAG and recalculates all downstream phases. The Slack webhook fires automatically.

### Workflow 3: Enforce Deadline Compliance via Git Pre-Commit Hook

```bash
# Install the deadliner pre-commit hook
deadliner hook install --scope global

# In your project’s .git/hooks/pre-commit, add:
#!/bin/sh
deadliner hook pre-commit --project "API v2 Migration" \
  --mode soft \
  --label "LATE_COMMIT" \
  --message "This commit is past the deadline for phase: %PHASE%"

# Make it executable
chmod +x .git/hooks/pre-commit
```

**Behavior:** When a developer commits after the `Backend Refactor` deadline, the hook:
- Does **not** block the commit (soft mode)
- Automatically adds a `LATE_COMMIT` label to the commit
- Prints a warning: `⚠️ This commit is past the deadline for phase: Backend Refactor`

## 2 Industry Best Practices

### Practice 1: **Treat Slack as a First-Class Resource, Not an Afterthought**
Most teams allocate zero buffer between phases, then scramble when something slips. With `deadliner`, **explicitly define slack per phase** (e.g., 20% of duration). This creates a probabilistic timeline: the engine can tell you the **probability of hitting the final deadline** based on historical slip rates. Use `deadliner risk --confidence 95` to surface phases that are statistically likely to overrun. This is how elite engineering teams at FAANG manage complex multi-team launches.

### Practice 2: **Automate Deadline Escalation with Conditional Webhooks**
Don’t rely on human memory for escalation. Configure `deadliner` to fire webhooks at **three thresholds**: 7 days out (heads-up), 48 hours out (warning), and 0 hours (breach). Each webhook should target a different channel: the first goes to the team Slack, the second to the engineering manager, the third to the VP. Use `deadliner escalation chain` to define this cascade. This pattern eliminates the “I thought someone else was watching it” failure mode and is standard practice in high-reliability DevOps environments.
