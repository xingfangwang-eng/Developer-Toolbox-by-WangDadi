### What is deadliner?

**deadliner** is a developer-first, lightweight deadline orchestration engine designed to eliminate the chaos of fragmented task tracking. Unlike bloated project management suites, deadliner focuses on a single, brutally efficient workflow: **define, track, and enforce deadlines across distributed systems and teams**. It integrates directly into your CI/CD pipeline, CLI, and IDE, giving you real-time visibility into what's slipping before it breaks the sprint.

### Core Features

1. **Multi-Repository Deadline Graph**  
   deadliner doesn't just track dates—it builds a directed acyclic graph (DAG) of dependencies across your repos. When a PR in `backend/auth` blocks `frontend/payment`, the deadline for the frontend task auto-adjusts with a cascading alert. No more manual cross-team syncs.

2. **Git-Aware Time Budgeting**  
   Each task gets a time budget (e.g., `5d`, `12h`). deadliner parses your commit history, branch activity, and PR merge velocity to calculate *remaining budget in real-time*. If you're burning through budget faster than planned, it triggers a warning in your Slack channel or terminal.

3. **Deadline-Driven CI/CD Gate**  
   Configure your pipeline to fail builds if associated deadlines are missed. deadliner injects a `DEADLINE_STATUS` environment variable into your CI runner. If the deadline is breached, the build halts—forcing the team to either renegotiate the deadline or fix the blocker before merging.

### How to Use

1. **Initialize a Deadline in Your Repo**  
   Run `deadliner init --project=payment-v2 --deadline=2025-04-15T18:00:00Z --budget=40h`. This creates a `.deadliner.yml` file in the root of your repo. The file is version-controlled, so deadline changes are tracked like any other code change. Any push to the repo updates the remote deadline state.

2. **Link Tasks to Branches**  
   Inside `.deadliner.yml`, map branches to tasks:
   ```yaml
   tasks:
     - name: "Implement payment gateway"
       branch: "feature/payment-gateway"
       depends_on: ["auth-refactor"]
       budget: "20h"
   ```
   deadliner watches the branch for commits and PR activity. If `feature/payment-gateway` is idle for more than 2 days within the budget, an alert fires. Use `deadliner status --verbose` to see per-task burn-down.

3. **Automate Deadline Gates in CI**  
   Add a step in your `Jenkinsfile` or `.github/workflows/ci.yml`:
   ```bash
   - name: Check deadline compliance
     run: deadliner gate --project=payment-v2 --branch=${{ github.head_ref }}
   ```
   If the deadline is missed, the step exits with code 1, failing the pipeline. The output includes a link to the deadline graph and the exact time slippage.

### Best Practices

- **Use Time Budgets for Estimation Audits**  
  Don't just set deadlines—set budgets. After each sprint, run `deadliner audit --project=<name>` to compare estimated budget vs. actual time spent (parsed from your commit timestamps). This gives you a data-driven baseline for future estimation accuracy, especially useful in distributed teams where "progress" is often opaque.

- **Enforce Deadline Gates Only on Critical Paths**  
  Not every task needs a CI gate. Use deadliner's `--critical` flag on tasks that block the next release. For non-critical tasks, just use alerts. This prevents developer frustration from false positives while keeping the release train on schedule.
