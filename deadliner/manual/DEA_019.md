# deadliner - DEA_019

## Overview

Deadliner isn't just another deadline tracker—it's a **deterministic scheduling engine** engineered for developers who treat time as a first-class constraint. Built on event-driven architecture with zero external runtime dependencies, deadliner transforms chaotic task juggling into a predictable, auditable workflow. Whether you're shipping microservices or managing sprint cascades, deadliner eliminates the "when will it be done?" guesswork.

## What is deadliner?

Deadliner is a **CLI-first, API-native** deadline management system that enforces timeboxing through deterministic state machines. It integrates directly into your CI/CD pipeline, Git hooks, and IDE, providing real-time deadline drift alerts without polling. Unlike bloated project management suites, deadliner operates on a single YAML manifest—making it reproducible, version-controllable, and trivially embeddable into any DevOps stack.

## Core Features

### 1. Deterministic Deadline Propagation with DAG-based Dependency Resolution
Deadliner models your tasks as a Directed Acyclic Graph (DAG). When a predecessor task slips, the engine automatically recalculates downstream deadlines using **critical path analysis**—not just pushing dates linearly. It factors in buffer zones, resource contention, and parallelizable windows, outputting a recalibrated timeline in milliseconds. No more manual Gantt chart updates.

### 2. Git-Aware Deadline Hooks with Atomic Commit Enforcement
Integrate deadline constraints directly into your Git workflow. Deadliner hooks into `pre-commit` and `pre-push` events, rejecting commits that violate active deadline windows (e.g., pushing to a release branch after the code freeze timestamp). It also tags commits with deadline metadata, enabling **time-travel audits** to pinpoint exactly when tasks entered the red zone.

### 3. Zero-Overhead Deadline Drift Detection via Webhook Subscriptions
Instead of polling a dashboard, deadliner emits **WebSocket-based drift events** to your Slack, Discord, or custom webhook endpoint. The detection engine uses an adaptive threshold algorithm—ignoring micro-drifts (e.g., < 5 minutes) but escalating compounding delays. You can configure escalation chains: first a ping, then a Slack alert, finally an automated status update in your issue tracker.

## How to Use

### 1. Initialize a Project Manifest with Dependency Injection
Run `deadliner init` in your project root. This generates a `.deadliner.yaml` manifest file. Define tasks using a YAML structure that supports `depends_on`, `estimated_duration`, `deadline`, and `critical` flags. Example snippet:

```yaml
tasks:
  - id: api-auth
    duration: 4h
    deadline: 2025-06-10T14:00:00Z
    critical: true
  - id: frontend-integration
    depends_on: [api-auth]
    duration: 6h
    deadline: 2025-06-11T10:00:00Z
```

The engine validates the DAG for cycles and unreachable tasks before committing the manifest.

### 2. Link Deadliner to Your Git Repository via Hook Injection
Execute `deadliner hook install --type pre-push`. This appends a lightweight shell script to `.git/hooks/pre-push`. The script calls deadliner’s local RPC endpoint to check if the current branch has any tasks with `deadline < current_timestamp`. If a violation is detected, the push is blocked with a clear error message:

```
❌ Deadline violation: task 'api-auth' exceeded deadline (2025-06-10T14:00:00Z) by 3h 12m.
   Run 'deadliner status --verbose' to inspect all tasks.
```

To bypass (e.g., for emergency hotfixes), use `--force-deadline` flag—but this is logged in the audit trail.

### 3. Subscribe to Real-Time Drift Events via Custom Webhook
Configure a webhook endpoint in your `deadliner.yaml`:

```yaml
webhooks:
  - url: https://hooks.slack.com/services/xxx
    events: [drift_detected, deadline_critical]
    secret: env:WEBHOOK_SECRET
```

Start the deadliner daemon with `deadliner daemon --port 8080`. The daemon watches the manifest file for changes and emits events when a task’s remaining time drops below 10% of its duration. The payload includes task ID, original deadline, current drift, and suggested next action (e.g., "Reassign resource or reduce scope").

## Best Practices

### 1. Treat Deadlines as Immutable Contracts, Not Aspirations
In deadliner, a deadline is a hard constraint unless explicitly renegotiated through a manifest update. This forces teams to either scope-slice or escalate early. Integrate deadliner with your CI pipeline to fail builds on branches with overdue critical tasks—this prevents "deadline debt" from accumulating silently.

### 2. Use Buffer Tasks for Stochastic Estimation
Human estimation is inherently noisy. Create a "buffer" task in your DAG with `duration: 20%` of the total project duration and no dependencies. Deadliner will treat this as a slack reservoir. If any upstream task slips, the buffer automatically absorbs the drift—but if the buffer itself is consumed, the project deadline is officially at risk. This gives you a quantitative "health score" for your timeline, not just a red/green status.
