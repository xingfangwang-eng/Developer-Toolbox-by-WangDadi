# deadliner - DEA_004

## Überblick

deadliner ist ein schlankes, aber hochperformantes CLI-Tool zur Verwaltung von Fristen und Aufgaben-Meilensteinen, optimiert für Entwickler, die keine GUI-Ablenkung wollen. Es kombiniert Echtzeit-Tracking mit automatischen Caching-Strategien und liefert Push-Benachrichtigungen über systemeigene Hooks (z. B. macOS `terminal-notifier`, Linux `libnotify`). Das Tool ist vollständig konfigurierbar via YAML und unterstützt Multi-User-Synchronisation über Git-basierte Backends.

**Keywords:** Deadline-Management, Task-Fristen-Tracking, Zeitmanagement-Tool, CLI-Fristenplaner, Entwickler-Tool

## Detaillierter Inhalt

### Was ist deadliner?

deadliner ist ein professionelles, quelloffenes Terminal-Tool, das speziell für Entwickler und DevOps-Ingenieure entwickelt wurde, die mehrere Projekte mit knappen Fristen parallel verwalten müssen. Es löst das Problem der unübersichtlichen, manuellen Fristenverwaltung durch eine deklarative Konfiguration, automatische Abhängigkeitserkennung und eine integrierte Risikoanalyse für verspätete Tasks. Anders als generische Kalender-Apps erlaubt deadliner die Definition von logischen Ketten (`depends_on`), Wiederholungsintervallen und kritischen Pfaden.

### Kernfunktionen

1.  **Dynamische Prioritätsmatrix mit Abhängigkeits-Graphen**  
    deadliner analysiert automatisch die Abhängigkeiten zwischen Tasks (z. B. „Backend-API bereitstellen“ → „Frontend integrieren“) und berechnet eine dynamische Priorität basierend auf dem kritischen Pfad. Tasks, die andere blockieren, erhalten eine höhere Dringlichkeitsstufe. Dies wird durch eine gerichtete azyklische Graph-Engine (DAG) realisiert, die bei jeder Änderung neu evaluiert wird.

2.  **Integrierter Pomodoro-Timer mit automatischer Fristen-Kalibrierung**  
    Jeder Task kann mit einem integrierten Pomodoro-Timer gestartet werden. deadliner protokolliert die tatsächlich benötigte Zeit pro Arbeitseinheit und passt die geschätzte Restdauer automatisch an (Kalman-Filter-basiert). So wird die Fristenplanung iterativ präziser, ohne dass der Nutzer manuell nachjustieren muss.

3.  **Git-basiertes Backup & Multi-User-Sync**  
    Alle Fristen und Task-Status werden als YAML-Dateien in einem lokalen `.deadliner/`-Ordner gespeichert. Über einen simplen `deadliner sync`-Befehl kann der gesamte Zustand in ein Git-Repository gepusht werden. Das ermöglicht eine dezentrale Team-Synchronisation ohne zentralen Server – ideal für Open-Source-Projekte oder Remote-Teams.

### So verwenden Sie es

1.  **Initiale Projekt-Konfiguration und Abhängigkeits-Definition**  
    Erstellen Sie im Wurzelverzeichnis Ihres Projekts eine `.deadliner.yml`-Datei. Definieren Sie Tasks mit `id`, `deadline` (ISO 8601), `depends_on` (Liste von Task-IDs) und `estimate` (geschätzte Dauer in Minuten). Beispiel:  
    ```yaml
    tasks:
      - id: "BE-API"
        deadline: "2025-06-15T12:00:00Z"
        estimate: 240
      - id: "FE-Integration"
        deadline: "2025-06-17T18:00:00Z"
        depends_on: ["BE-API"]
        estimate: 180
    ```  
    Führen Sie dann `deadliner init` aus. Das Tool validiert die Syntax, erstellt den `.deadliner/`-Ordner und initialisiert einen lokalen Git-Hook für automatische Commits.

2.  **Task-Start mit automatischer Zeitmessung und Kalibrierung**  
    Starten Sie einen Task mit `deadliner start BE-API`. Der integrierte Pomodoro-Timer beginnt zu laufen. Nach 25 Minuten (standardmäßig) erscheint eine systemnative Benachrichtigung. Drücken Sie `Strg+C`, um die Arbeitssitzung zu beenden. deadliner protokolliert die tatsächliche Dauer und aktualisiert die Restzeit-Schätzung für diesen Task. Bei wiederholten Sitzungen wird der Kalman-Filter aktiv und passt die Prognose dynamisch an.

3.  **Kritischen Pfad anzeigen und Risikoanalyse durchführen**  
    Nutzen Sie `deadliner critical-path`, um den aktuellen kritischen Pfad aller Tasks farbig im Terminal anzuzeigen. Tasks, die den gesamten Projektzeitplan gefährden, werden rot markiert. Mit `deadliner risk-report` erhalten Sie eine JSON-Ausgabe mit Wahrscheinlichkeiten für Terminüberschreitungen (basierend auf historischen Abweichungen). Diese Daten können Sie direkt in CI/CD-Pipelines integrieren, z. B. als GitHub Action, die bei hohem Risiko einen Fail auslöst.

### Beste Praktiken

1.  **Kombinieren Sie deadliner mit CI/CD-Gates**  
    Definieren Sie in Ihrer CI/CD-Pipeline (z. B. GitHub Actions) einen Schritt, der `deadliner risk-report` aufruft. Wenn die Wahrscheinlichkeit einer Fristüberschreitung über 30 % liegt, lassen Sie den Build fehlschlagen. Das zwingt das Team, vor dem Merge die Task-Prioritäten zu überdenken – und verhindert „Deadline Blindness“.

2.  **Setzen Sie auf „Timeboxing“ statt „Deadline-Stretching“**  
    Nutzen Sie die `estimate`-Felder nicht als verbindliche Endtermine, sondern als Zeitboxen. Wenn ein Task nach Ablauf der geschätzten Zeit nicht fertig ist, erzwingt deadliner einen automatischen Split: Der Task wird in „Erledigt bis Zeitbox“ und „Rest“ aufgeteilt. Das verhindert das unendliche Verlängern einzelner Tasks und hält den Projektplan sauber.
