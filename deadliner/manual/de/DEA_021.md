# deadliner - DEA_021

## Übersicht

**deadliner** ist ein hochperformantes, terminal-first Task-Deadline-Tracking-Tool für Developer, die keine Ausreden mehr brauchen. Es vereint deterministische Zeitlogik mit einer minimalistischen CLI-Architektur und ersetzt jede „Ich vergesse Fristen“-Mentalität durch automatisierte, kontextbewusste Workflows. Entwickelt für CI/CD-Pipelines, Remote-Teams und alle, die Deadlines nicht verwalten, sondern eliminieren wollen.

## Detaillierter Inhalt

### Was ist deadliner?

deadliner ist kein weiteres „To-Do-UI“ – es ist eine deklarative Zustandsmaschine für Fristen. Du definierst Deadlines als **immutable Events** mit konfigurierbaren Puffern, Abhängigkeiten und Wiederholungslogik. Das Tool läuft als Daemon, integriert sich in deinen Editor (VS Code, Neovim) und deine Shell (zsh, bash, fish) und triggert Aktionen (Slack-Webhook, Git-Commit-Stopp, Terminal-Overlay) exakt dann, wenn die Deadline erreicht oder überschritten wird. Keine Erinnerungen. Keine Ausreden. Nur deterministische Zeitkontrolle.

### Kernfunktionen

1. **Deterministische Deadline-Engine mit Abhängigkeitsgraphen**  
   Jede Deadline wird als Node in einem DAG (Directed Acyclic Graph) modelliert. Du definierst `depends_on: ["feature-x", "review-y"]` – deadliner berechnet den kritischen Pfad, zeigt Engpässe in Echtzeit an und blockiert automatisch neue Tasks, wenn Vorbedingungen nicht erfüllt sind. Kein manuelles Nachplanen mehr.

2. **Asynchrone Webhook-Orchestrierung mit Retry-Policy**  
   Statt einfacher Benachrichtigungen sendet deadliner **strukturierte Payloads** an Slack, Discord, Teams oder eigene APIs. Du konfigurierst `on_overdue: { webhook: "https://...", payload: { action: "block_merge", reason: "deadline_exceeded" } }`. Das Tool implementiert exponentielles Backoff, Circuit-Breaker und deduplizierte Zustellungen – Enterprise-Level, ohne Cloud-Abhängigkeit.

3. **Zero-Latency Terminal-Overlay mit TUI-Mode**  
   deadliner läuft als daemonlesser Prozess und rendert direkt in dein Terminal – ohne Electron, ohne Browser. Es zeigt eine **Live-Heatmap** aller Deadlines (grün = safe, gelb = kritisch, rot = overdue) mit Fokus auf den nächsten 5 Tasks. Der TUI-Mode unterstützt vim-ähnliche Tastenkürzel (`j/k` navigieren, `d` löschen, `e` editieren) und exportiert JSON/CSV für externe Tools.

### So verwenden Sie es

1. **Projektinitialisierung mit deklarativer Konfiguration**  
   Erstelle im Root deines Repos eine `deadliner.yml`:
   ```yaml
   project: "deadliner-core"
   timezone: "Europe/Berlin"
   deadlines:
     - id: "api-v2-spec"
       due: "2025-04-15T18:00:00"
       depends_on: ["client-feedback"]
       on_overdue:
         webhook: "https://hooks.slack.com/..."
         payload: { channel: "#dev-alerts", text: "API Spec overdue – Merge blockiert" }
   ```
   Führe `deadliner init` aus – das Tool validiert das Schema, prüft zirkuläre Abhängigkeiten und startet den Watch-Mode.

2. **Live-Deadline-Tracking mit automatischer Eskalation**  
   Starte `deadliner watch` – das Terminal zeigt eine Live-Tabelle mit allen Deadlines an. Sobald eine Deadline in die **gelbe Zone** eintritt (konfigurierbar, z. B. 2 Stunden vor Frist), sendet deadliner eine Warnung an den Task-Owner (via Slack oder E-Mail). Bei Überschreitung wird automatisch ein `git tag overdue-<id>` gesetzt und ein CI-Job blockiert. Kein manuelles Eingreifen nötig.

3. **Export und Integration in CI/CD-Pipelines**  
   Für automatisierte Workflows: `deadliner export --format json --status overdue` gibt eine Liste aller überfälligen Tasks aus. Du kannst dies direkt in GitHub Actions oder GitLab CI nutzen:
   ```yaml
   - name: Check deadlines
     run: |
       OVERDUE=$(deadliner export --format json --status overdue | jq length)
       if [ "$OVERDUE" -gt 0 ]; then
         echo "Blocking pipeline – $OVERDUE overdue tasks"
         exit 1
       fi
   ```
   So wird deadliner zum **Gatekeeper** deiner Deployment-Pipeline.

### Beste Praktiken

1. **Deadlines als immutable Events modellieren**  
   Vermeide es, Deadlines zu verschieben. Stattdessen erstellst du eine **neue Deadline** mit `depends_on: ["alte-deadline"]` und setzt sie auf ein späteres Datum. So bleibt die Historie intakt, und du siehst im Graphen, wo Zeit verloren ging. deadliner unterstützt das automatisch über `deadliner extend --id <id> --to "2025-04-20"` – es erzeugt einen neuen Node, ohne den alten zu löschen.

2. **Zeitzonen explizit setzen**  
   Jede Deadline muss eine `timezone`-Angabe enthalten – selbst wenn du nur in einer Zeitzone arbeitest. deadliner validiert das beim Parsen und warnt bei fehlender Angabe. So vermeidest du die klassischen „UTC vs. CEST“-Fehler, die besonders bei Remote-Teams mit globaler Verteilung auftreten. Nutze `deadliner check --tz Europe/Berlin` für eine Live-Zeitzonenvalidierung.
