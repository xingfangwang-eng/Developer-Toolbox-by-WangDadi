# deadliner - DEA_023

## Überblick

deadliner ist kein weiterer simpler Aufgabenplaner. Es ist ein **agentenbasiertes Fristen-Framework** für Entwickler, die mit komplexen, mehrstufigen Projekten arbeiten – inklusive automatisierter Risikoprüfung, CI/CD-Integration und Live-Kollaboration über Markdown. Entwickelt für Teams, die Deadlines nicht nur setzen, sondern auch **einhalten** wollen.

## Hardcore-Features

### 1. **Rekursive Abhängigkeitsanalyse mit Deadlock-Erkennung**
deadliner analysiert deine Aufgaben topologisch. Erkennt zirkuläre Abhängigkeiten (z. B. „Task A wartet auf B, B auf C, C auf A“) und schlägt **automatisch alternative Sequenzen** vor – inklusive simuliertem kritischem Pfad. Ideal für Microservice-Meilensteine oder Release-Phasen.

### 2. **Auto-Triage mit Time-Budget-Propagation**
Jeder Task bekommt ein Zeitbudget. deadliner propagiert Verzögerungen **rekursiv** durch den gesamten Graphen. Falls ein Task sein Budget überschreitet, triggert das Tool automatisch eine **Risiko-Eskalation** (Slack-Webhook, Git-Issue, E‑Mail). Kein manuelles Nachhalten mehr.

### 3. **Markdown-native Schnittstelle + CLI-First**
Definiere Deadlines direkt in `DEADLINE.md` – YAML-Frontmatter und Markdown-Aufgabenliste. Das CLI-Interface erlaubt `deadliner check`, `deadliner push` und `deadliner triage` in der Pipeline. Kein GUI-Zwang, volle Git-Versionierung.

## Detaillierte Anleitung (für Profis)

### 1. **Projekt initialisieren & Struktur definieren**
```bash
mkdir my-project && cd my-project
deadliner init
```
Es wird eine `DEADLINE.md` erstellt. Ergänze im Frontmatter die Projekt-ID, Zeitzone und Standard-Eskalationskanal. Lege dann Tasks als Markdown-Liste an – mit `[ ]` oder `[x]`, plus Metadaten via `<!-- deadline:2025-04-15 -->`. Nutze `depends-on: task-id` für Abhängigkeiten.

### 2. **Abhängigkeitsgraph visualisieren & testen**
```bash
deadliner graph --format mermaid
```
Erzeugt ein Mermaid-Diagramm deines Task-Graphen. Mit `--check-cycles` prüfst du auf Deadlocks. Falls ein Zyklus erkannt wird, schlägt deadliner eine **alternative Reihenfolge** vor (z. B. „Task C vor B verschieben“). Fixe den Graphen, bis `deadliner validate` grün ist.

### 3. **Automatische Triage in der CI-Pipeline**
Integriere deadliner in deine GitHub Actions:
```yaml
- name: Fristen-Check
  run: |
    deadliner check --fail-on-overdue
    deadliner triage --auto-escalate
```
Bei jedem Push wird geprüft, ob Tasks überfällig sind. Falls ja, erstellt deadliner automatisch einen **GitHub Issue** mit Verantwortlichem und Eskalationsstufe. Der Status wird in der `DEADLINE.md` aktualisiert – alles versioniert.

## Best Practices (aus der Architektenpraxis)

### 1. **Task-Granularität an Sprint-Länge anpassen**
Teile große Meilensteine in **3–5 Tages-Tasks** auf. deadliner propagiert Verzögerungen dann granular genug, um frühzeitig zu reagieren. Vermeide Tasks > 7 Tage – sie machen die Risikoanalyse stumpf.

### 2. **Eskalationsregeln mit Team-Rollen verknüpfen**
Definiere pro Task `assignee` und `escalation-to`. deadliner kann dann automatisch **Stufen eskalieren**: Zuerst an den Entwickler, nach 2 Tagen an den Tech Lead, nach 5 Tagen an den Projektmanager. Das entlastet das Management und hält den Druck auf der richtigen Ebene.
