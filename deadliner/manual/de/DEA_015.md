# deadliner - DEA_015

## Überblick

deadliner ist kein weiterer simpler Aufgabenplaner. Es ist eine **präzise Engine für zeitkritische Workflows**, die speziell für Entwickler und technische Projektmanager entwickelt wurde, die mit komplexen Abhängigkeiten und knappen Release-Zyklen jonglieren. Vergessen Sie manuelles Nachverfolgen – deadliner automatisiert die Priorisierung und Eskalation.

## Detaillierter Inhalt

### Was ist deadliner?

deadliner ist ein Open-Source-Kommandozeilen-Tool und API-Gateway, das sich nahtlos in Ihre CI/CD-Pipeline und Ihren bestehenden Tech-Stack (Jira, GitLab, Slack) integriert. Es transformiert passive Fristen in **aktive, durchsetzbare Metriken**. Statt nur zu erinnern, blockiert, eskaliert und protokolliert deadliner – immer deterministisch und regelbasiert.

### Kernfunktionen

- **Deterministische Deadline-Propagation**  
  Verknüpfen Sie Deadlines über Abhängigkeitsgraphen hinweg (z.B. `design-review` → `feature-implementierung` → `code-review` → `deployment`). Wenn eine Deadline im Graphen ausgelöst wird, berechnet deadliner automatisch die kritischen Pfade neu und passt die abhängigen Fristen an. Kein manuelles Verschieben mehr.

- **Policy-basierte Eskalations-Engine**  
  Definieren Sie im `deadliner.yaml` Regeln wie: `Wenn Deadline < 48h und Status != 'in_review', dann blockiere Merge-Request und sende @channel im Slack-Channel #release-alerts`. Die Engine ist idempotent und protokolliert jeden Schritt für das Compliance-Audit.

- **Zero-Overhead CLI & API**  
  Setzen Sie Deadlines direkt aus Ihrem Terminal mit `deadliner set --task feat-123 --deadline 2024-03-15T18:00:00Z --depends-on review-456`. Oder nutzen Sie die REST-API für die Integration in Ihre eigene CI/CD-Orchestrierung. Der Output ist immer maschinenlesbar (JSON) und menschenlesbar (tabellarisch).

### So verwenden Sie es

1. **Projekt initialisieren & Regeln definieren**  
   Führen Sie `deadliner init` im Root Ihres Repos aus. Dadurch wird eine `deadliner.yaml` erstellt. Definieren Sie dort Ihre Eskalationspolicies und Standard-Workflows:
   ```yaml
   policies:
     - if: deadline < 24h && status == 'open'
       then: block_pr
     - if: deadline < 1h && status == 'in_progress'
       then: notify_slack(channel: '#urgent', message: 'Deadline für {{task}} in 60 Minuten.')
   ```

2. **Tasks mit Abhängigkeiten erfassen**  
   Erstellen Sie einen Task mit Abhängigkeit:
   ```bash
   deadliner add --name "Backend-API-Doku" --deadline "2024-03-20" --depends-on "Feature-X-Review" --priority critical
   ```
   deadliner validiert sofort, ob die Abhängigkeit zyklisch ist oder ob die Deadline der Abhängigkeit nach Ihrer liegt (und warnt Sie).

3. **Pipeline-Integration & automatische Eskalation**  
   Fügen Sie in Ihrer CI/CD-Konfiguration (z.B. `.gitlab-ci.yml`) einen einfachen Schritt hinzu:
   ```yaml
   after_script:
     - deadliner check --fail-on-blocked
   ```
   Wenn ein Task blockiert ist oder eine Policy ausgelöst wurde, schlägt der Job fehl. So wird die Deadline-Disziplin direkt in den Build-Prozess eingebaut.

### Beste Praktiken

- **Deadlines als Code (DaC)**  
  Halten Sie Ihre `deadliner.yaml` und alle Task-Definitionen im gleichen Repository wie Ihren Code. Nutzen Sie Code-Reviews für Änderungen an Deadlines. So wird die Zeitplanung versioniert, nachvollziehbar und Teil Ihrer Entwicklungsdisziplin, nicht ein separates Spreadsheet.

- **Granulare Eskalationsstufen**  
  Definieren Sie nicht nur eine harte Deadline. Nutzen Sie die Policy-Engine für weiche (Warning) und harte (Block) Deadlines. Ein typischer Workflow: `72h vorher → Benachrichtigung`, `24h vorher → Slack-Erwähnung`, `6h vorher → Ticket blockieren`. So haben alle Beteiligten mehrere Chancen zu reagieren, bevor der Merge-Request blockiert wird.
