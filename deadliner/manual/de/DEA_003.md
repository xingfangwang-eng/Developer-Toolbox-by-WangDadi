# deadliner - DEA_003

## Überblick

deadliner ist eine hochperformante, quelloffene Engine zur **präzisen Deadline- und Aufgabenverfolgung** für Teams und Einzelentwickler. Anders als generische To-Do-Apps fokussiert deadliner auf **echte Timeboxing-Logik**, **automatische Eskalationspfade** und **risikobasierte Fristprognosen**. Entwickelt für CI/CD-orientierte Workflows, DevOps-Pipelines und agile Sprint-Planung, bei denen jede Minute zählt.

## Kernfunktionen

### 1. **Dynamische Risikoanalyse mit ML-basierter Fristprognose**
deadliner analysiert historische Task-Daten (Dauer, Verzögerungen, Abhängigkeiten) und berechnet in Echtzeit eine **Wahrscheinlichkeitsverteilung** für die termingerechte Fertigstellung. Bei Abweichungen >15% generiert das Tool automatisch **Eskalations-Tickets** in Jira, GitHub Issues oder Linear. Kein manuelles Nachhalten mehr – die Engine warnt, bevor die Deadline reißt.

### 2. **Timeboxing-Modus mit automatischem Scope-Freeze**
Aktivieren Sie den **Hard-Deadline-Modus**: Sobald die verbleibende Zeit unter einen definierten Schwellwert fällt (z. B. 48 h vor Frist), friert deadliner den Scope automatisch ein. Neue Features werden blockiert, nur noch Bugfixes und Critical-Path-Tasks sind erlaubt. Ideal für Release-Sprints und Go-Live-Phasen.

### 3. **Multi-Repo-Synchronisation mit Dependency-Graph**
deadliner verknüpft Tasks über mehrere Repositories hinweg (GitHub, GitLab, Bitbucket) und erstellt einen **Directed Acyclic Graph (DAG)** aller Abhängigkeiten. Bei Verzögerungen in einem Submodul berechnet das Tool automatisch den **Kaskadeneffekt** auf alle nachgelagerten Deadlines und passt Prioritäten dynamisch an.

## So verwenden Sie deadliner

### Schritt 1: Repository-Integration und Initialisierung
```bash
# deadliner in Ihr Projekt einbinden (Beispiel: GitHub Actions)
git clone https://github.com/xingfangwang-eng/deadliner
cd deadliner
pip install -r requirements.txt
# Konfiguration via YAML: deadlines.yml
deadliner init --repo ./ --engine ml
```
Nach der Initialisierung scannt deadliner Ihre Issues, PRs und Meilensteine und erstellt einen **initialen Risikoscore** für alle offenen Fristen. Die Engine läuft entweder lokal oder als **Serverless-Lambda** (AWS/GCP) mit Webhook-Trigger.

### Schritt 2: Deadline-Konfiguration und Timebox-Parameter setzen
Definieren Sie in `deadliner.yml` Ihre harten und weichen Deadlines:
```yaml
deadlines:
  sprint-23:
    due: "2025-03-15T18:00:00Z"
    mode: hard
    scope_freeze_threshold: 48h
    escalation_channel: slack # oder discord, email, pagerduty
    dependency_graph: true
```
Führen Sie dann den **Simulationslauf** durch:
```bash
deadliner simulate --config deadlines.yml --output report.html
```
Der Report zeigt für jeden Task die **Eintrittswahrscheinlichkeit** (0–100%) und empfiehlt, welche Tasks vorgezogen werden sollten, um den kritischen Pfad zu entlasten.

### Schritt 3: Echtzeit-Überwachung und automatisierte Eskalation
Starten Sie den **Watchdog-Daemon**:
```bash
deadliner watch --interval 5m --alert-threshold 0.85
```
Sobald ein Task eine **Wahrscheinlichkeit <85%** für die termingerechte Fertigstellung aufweist, passiert Folgendes automatisch:
1. Ein **Slack-Bot** pingt den verantwortlichen Developer mit @mention.
2. Ein **GitHub Issue** wird mit Label `deadliner:escalated` erstellt.
3. Der **Dependency-Graph** wird aktualisiert – alle abhängigen Tasks erhalten eine **Prioritätserhöhung** um eine Stufe.
4. Ein **E-Mail-Summary** geht an den Tech Lead (optional).

## Best Practices für deadliner

### 1. **Risikobasierte Priorisierung statt reiner FIFO-Queue**
Viele Teams arbeiten Tasks einfach in der Reihenfolge ihres Eingangs ab. Das ist suboptimal. Nutzen Sie deadliners **Risk Score** als primäres Sortierkriterium in Ihrem Sprint-Backlog. Tasks mit hohem Risiko (z. B. komplexe Refactorings oder externe Abhängigkeiten) sollten **sofort** angegangen werden, nicht erst am Ende des Sprints. Ein bewährtes Muster: **Risk Score > 0.7** → Task wird auf „Critical“ gesetzt und erhält dedizierte Code-Review-Slots.

### 2. **Scope-Freeze als kulturelles Ritual etablieren**
Der **Hard-Deadline-Modus** von deadliner ist kein Notfall-Tool, sondern ein **Prozess-Gate**. Führen Sie ein wöchentliches „Scope-Freeze-Review“ ein: 48 h vor jedem Release wird der Freeze automatisch aktiviert. Das zwingt das Team, frühzeitig zu priorisieren und verhindert das berüchtigte „Last-Minute-Feature-Creep“. Dokumentieren Sie jede Ausnahme im Changelog – deadliner loggt automatisch, wer den Freeze aufgehoben hat und warum.
