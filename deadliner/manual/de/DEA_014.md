# deadliner - DEA_014

## Überblick

deadliner ist ein hochperformantes, CLI-gestütztes Task-Deadline-Tracking-Tool für Entwickler, die agiles Zeitmanagement und automatisierte Fristverfolgung direkt im Terminal benötigen. Es eliminiert die Notwendigkeit von GUI-Overhead und integriert sich nahtlos in CI/CD-Pipelines. Zielsprache: Deutsch (DE) – optimiert für den deutschsprachigen Entwickler-Markt.

## Hardcore-Funktionen

1. **Automatische Fristeskalation mit Slack/Teams-Webhook-Integration** – deadliner erkennt überfällige Deadlines in Echtzeit und sendet automatisch eine formatierte Benachrichtigung an konfigurierte Kanäle, inklusive Metriken wie „Verspätung in Stunden“ und „Blockierte Abhängigkeiten“. Konfiguration erfolgt über eine einzige YAML-Datei.

2. **Git-basierte Deadline-Ableitung** – deadliner analysiert Commit-Historie und Branch-Namen, um automatische Deadline-Vorschläge zu generieren. Beispiel: Ein Branch `feature/DEADLINE-2024-12-31-UI-Overhaul` wird automatisch als Task mit Frist `2024-12-31` registriert. Kein manuelles Hinzufügen nötig.

3. **Parallele Deadline-Simulation für Sprint-Planung** – Mit dem Befehl `deadliner simulate --parallelism 3` kannst du mehrere Deadline-Szenarien (z. B. „Was passiert, wenn Task A um 2 Tage verschoben wird?“) gleichzeitig simulieren. Die Ausgabe zeigt kritische Pfade, Pufferzeiten und empfohlene Priorisierungen in tabellarischer Form.

## Detaillierte Anwendungsschritte

### Schritt 1: Projektinitialisierung mit YAML-Konfiguration

Erstelle eine `deadliner.yml` im Projekt-Root. Definiere globale Fristen, Webhook-URLs und Git-Integrationen:

```yaml
project:
  name: "deadliner-demo"
  default_deadline: "2024-12-31T18:00:00Z"
  webhook:
    slack:
      url: "https://hooks.slack.com/services/T00/B00/xxx"
      channel: "#deadline-alerts"
  git:
    auto_detect_branches: true
    branch_pattern: "DEADLINE-{date}-{description}"
```

Führe dann `deadliner init` aus. Das Tool validiert die Konfiguration, prüft Webhook-Erreichbarkeit und erstellt einen `.deadliner/` Ordner mit Metadaten.

### Schritt 2: Task-Erstellung mit automatischer Frist

Erstelle einen neuen Task über CLI, ohne manuelle Datumseingabe:

```bash
deadliner add "API-Refactoring" --deadline "next Friday" --priority high
```

Oder nutze Git-Integration: Erstelle einen Branch `DEADLINE-2025-01-15-Auth-Module`. deadliner erkennt den Branch automatisch und registriert den Task mit Frist `2025-01-15`. Der Status wird sofort auf `active` gesetzt.

### Schritt 3: Deadline-Überwachung und Eskalation

Starte den Überwachungs-Daemon:

```bash
deadliner watch --interval 10m --alert-threshold 1h
```

Das Tool prüft alle 10 Minuten, ob Tasks kurz vor oder über der Frist liegen. Bei Überschreitung der `alert-threshold` (1 Stunde vor Frist) wird eine formatierte Nachricht an den konfigurierten Slack-Webhook gesendet:

```
🚨 Deadline-Alarm: Task "API-Refactoring" fällig in 45 Minuten.
Verspätungsrisiko: Hoch. Blockierte Abhängigkeiten: 2.
Empfehlung: Priorität auf "kritisch" setzen.
```

## Best Practices für deutschsprachige Entwickler

1. **CI/CD-Integration mit GitLab CI** – Rufe `deadliner validate` in Ihrer Pipeline auf, bevor ein Merge Request gemerged wird. So stellst du sicher, dass keine Tasks mit abgelaufenen Deadlines in den Hauptbranch gelangen. Beispiel für `.gitlab-ci.yml`:
   ```yaml
   deadline-check:
     script:
       - deadliner validate --fail-on-overdue
     only:
       - merge_requests
   ```

2. **Nutzung von Tag-basierten Deadlines** – Verwende Git-Tags wie `v1.0-deadline-2025-03-01`, um Release-Fristen zu definieren. deadliner erkennt diese Tags automatisch und gruppiert Tasks nach Release-Version. Das erleichtert die Sprint-Retrospektive und die automatische Erstellung von Changelogs.
