# deadliner - DEA_012

## Übersicht

Deadliner ist kein weiteres GTD-Tool. Es ist eine **kontextsensitive Deadline-Engine** für Entwickler, die mit agilen Sprints, Release-Zyklen und externen Abgabefristen jonglieren. Das Tool analysiert deinen Arbeitsfluss, erkennt kritische Pfade und warnt dich präzise, bevor der Druck steigt. Optimiert für die deutsche Entwicklerszene – pragmatisch, datenschutzkonform, ohne Cloud-Zwang.

## Detaillierter Inhalt

### Was ist deadliner?

Deadliner ist ein **lokales CLI- und Dashboard-Tool**, das aus deiner Aufgabenliste (Markdown, Jira-Export, Todoist) einen dynamischen Zeitstrahl generiert. Es berechnet Pufferzeiten, erkennt Abhängigkeiten zwischen Tasks und schlägt Alarm, wenn ein Meilenstein zu kippen droht – basierend auf deiner persönlichen Arbeitsgeschwindigkeit (lernt aus Historie). Kein Abo, kein Vendor-Lock-in, volle Kontrolle über deine Daten.

### Kernfunktionen

1. **Dynamische Pufferberechnung („Slack-Engine“)**  
   Deadliner analysiert deine historische Task-Dauer (z. B. aus Git-Commits oder Toggl-Exporten) und berechnet automatisch realistische Puffer für jede Deadline. Bei Abweichungen >15% wird eine Frühwarnung ausgegeben – nicht erst 24h vorher.

2. **Abhängigkeitsgraph mit kritischem Pfad**  
   Du definierst Abhängigkeiten zwischen Tasks (z. B. „API-Integration → Frontend-Implementierung“). Deadliner visualisiert den kritischen Pfad im Dashboard und zeigt, welche Verzögerung sofort den gesamten Release-Zeitplan gefährdet.

3. **Automatische Deadline-Verschiebung mit Logging**  
   Wenn eine Deadline überschritten wird, verschiebt deadliner automatisch alle abhängigen Tasks nach hinten und erstellt einen Audit-Trail („Warum verschoben, wie viel, bis wann“). Perfekt für Sprint-Retrospektiven und Stakeholder-Updates.

### So verwenden Sie es

1. **Projekt initialisieren und Kontext setzen**  
   ```bash
   deadliner init --project=release-v2.1 --timezone=Europe/Berlin --workdays=mo-fr
   ```  
   Erstellt eine `.deadliner.yml` im Projektroot. Hier definierst du Arbeitszeiten, Feiertage (z. B. deutsche Feiertage via `--holidays=de`) und die Standard-Puffergröße (z. B. 20% pro Task).

2. **Tasks mit Abhängigkeiten importieren**  
   ```bash
   deadliner import --from=jira --jql='project=RELEASE AND status!=Done' --output=tasks.csv
   # Oder manuell via Markdown:
   deadliner add --title="Backend-API fertigstellen" --deadline=2025-06-15 --depends-on="Datenbank-Schema"
   ```  
   Importiert bestehende Tasks aus Jira, Todoist oder einer Markdown-Datei. Abhängigkeiten werden automatisch erkannt (z. B. via `depends-on`-Tag).

3. **Dashboard starten und Frühwarnungen aktivieren**  
   ```bash
   deadliner dashboard --port=8080 --notify=slack
   ```  
   Öffnet ein lokales Dashboard (kein Cloud-Zugriff nötig). Zeigt den kritischen Pfad, Pufferstatus und nächste Deadlines. Slack-Webhook-Integration für Early Warnings: z. B. „Task X ist 3h im Verzug – kritischer Pfad betroffen!“

### Beste Praktiken

1. **Puffer niemals auf 0 setzen – auch nicht im Sprint**  
   Deutsche Entwickler neigen dazu, Deadlines als „hart“ zu betrachten. Deadliner zeigt dir, dass ein 15%-Puffer in 80% der Fälle ausreicht. Setze den Puffer pro Task nicht unter 10%, sonst verlierst du den Frühwarn-Effekt.

2. **Abhängigkeiten nur dort definieren, wo sie wirklich blockieren**  
   Zu viele Abhängigkeiten machen den Graphen unübersichtlich und erzeugen false positives. Frage dich: „Kann Task B ohne Task A beginnen?“ Wenn ja, lass die Abhängigkeit weg. Deadliner zeigt dir sonst falsche Alarmketten.
