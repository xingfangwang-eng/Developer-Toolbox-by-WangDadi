# deadliner - DEA_020

## Überblick

deadliner ist ein speziell für agile Entwicklungsteams und Freelancer konzipiertes CLI-Tool zur präzisen Deadlinesteuerung, das auf asynchrone Workflows optimiert ist. Es eliminiert manuelle Kalenderpflege und bietet eine API-first-Architektur für CI/CD-Pipelines.

## Hardcore-Funktionen

1. **Dynamische Zeitpuffer-Analyse**  
   Das Tool berechnet automatisch einen Risikopuffer basierend auf historischen Sprint-Daten und Task-Komplexität (z.B. Fibonacci-Punkte). Es warnt, wenn ein Deadline-Risiko > 15 % liegt, und schlägt Neupriorisierung vor.

2. **Git-basierte Deadline-Verknüpfung**  
   Jeder Task wird direkt mit einem Git-Branch oder Commit-Hash verknüpft. Bei Merge-Konflikten oder Verzögerungen im Commit-Log wird die Deadline automatisch neu berechnet und eine Benachrichtigung an den Slack-Webhook gesendet.

3. **Multi-Repository-Synchronisation**  
   deadliner aggregiert Deadlines aus mehreren GitHub-Repositories, GitLab-Projekten und Bitbucket-Workspaces in einer zentralen Zeitachse. Es erzeugt eine `deadline_matrix.json`, die in CI/CD-Pipelines als Artefakt verwendet werden kann.

## Detaillierte Anwendungsschritte

1. **Initialisierung und Konfiguration**  
   Führe `deadliner init` im Wurzelverzeichnis deines Projekts aus. Das Tool erstellt eine `.deadliner.yml` mit default-Werten. Trage dort deine Git-Provider-Tokens (z.B. `GITHUB_TOKEN`) und den Pfad zu deiner `package.json` oder `Cargo.toml` ein. Beispiel:  
   ```yaml
   repositories:
     - name: backend
       url: https://github.com/team/backend
       token: ${GITHUB_TOKEN}
     - name: frontend
       url: https://gitlab.com/team/frontend
   ```

2. **Task-Erstellung mit Deadline-Constraint**  
   Verwende `deadliner add --task "Implementiere OAuth" --deadline 2025-06-30 --priority high --buffer 20%`. Der Puffer wird als prozentualer Aufschlag zur Basiszeit berechnet. Der Task wird automatisch mit dem aktuellen Git-Branch (`git rev-parse --abbrev-ref HEAD`) verknüpft.

3. **Monitoring und Eskalation**  
   Starte den Watchdog mit `deadliner watch --interval 30m`. Das Tool überprüft alle 30 Minuten den Git-Status und berechnet die Abweichung von der geplanten Deadline. Bei Überschreitung von 80 % des Zeitpuffers wird ein `deadline_alert.json` in das `./reports/`-Verzeichnis geschrieben und optional ein Webhook an dein Team-Notification-System (z.B. Discord, Mattermost) gesendet.

## Best Practices

1. **Deadline als Code**  
   Definiere alle Deadlines in einer `deadline.yml` und committe sie ins Repository. So bleibt die Deadline-Historie versioniert und nachvollziehbar. Verwende `deadliner validate` in der CI, um sicherzustellen, dass keine Deadline-Konflikte zwischen Branches bestehen.

2. **Asynchrone Benachrichtigungen**  
   Verknüpfe deadliner mit deinem Issue-Tracker (z.B. Linear, Jira) über die REST-API. Wenn ein Task seine Deadline überschreitet, setze automatisch den Status auf "Blocked" und füge einen Kommentar mit dem genauen Verzug hinzu. So bleibt der Workflow transparent, ohne dass jemand manuell nachfassen muss.
