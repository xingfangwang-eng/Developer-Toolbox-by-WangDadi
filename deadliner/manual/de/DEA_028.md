# deadliner - DEA_028

## Überblick

deadliner ist ein präzises, ereignisgesteuertes Fristen-Tracking-Tool für Entwickler und Projektleiter, die keine Ausreden mehr für verpasste Deadlines dulden. Es kombiniert ISO-konforme Zeitlogik mit einem minimalistischen CLI- und API-First-Ansatz – ideal für CI/CD-Pipelines, agile Sprint-Reviews und automatisierte Benachrichtigungen.

**Keywords:** Fristenmanagement, Aufgaben-Tracking, Zeitmanagement-Tool, Deadline-Überwachung, DevOps-Deadline

## Detaillierter Inhalt

### Was ist deadliner?

deadliner ist ein professionelles, quelloffenes Tool zur Verwaltung und Überwachung von Aufgabenfristen in Echtzeit. Es basiert auf einer strikten, zeitzonenbewussten Kernlogik (ISO 8601) und bietet eine REST-API sowie eine native CLI. Anders als generische Projektmanagement-Tools fokussiert deadliner auf die **automatische Eskalation** und **granulare Benachrichtigungssteuerung** – ideal für Teams, die mit harten Deadlines in verteilten Umgebungen arbeiten.

### Kernfunktionen

1.  **Automatische Fristen-Eskalation mit Stufenlogik**  
    Sobald eine Deadline unterschritten wird (z. B. < 24 Stunden), startet deadliner eine konfigurierbare Eskalationskaskade: E-Mail → Slack → PagerDuty. Jede Stufe löst automatisch eine neue Benachrichtigung aus, bis die Aufgabe abgeschlossen oder die Frist manuell bestätigt wird. Kein manuelles Nachfassen mehr.

2.  **Zeitzonen-resiliente Fristenberechnung**  
    Jede Deadline wird mit einer expliziten IANA-Zeitzone (z. B. `Europe/Berlin`, `America/New_York`) gespeichert. Bei Abfragen oder Benachrichtigungen wird automatisch in die Systemzeit des Clients konvertiert. So vermeidest du die berüchtigten „UTC-Midnight-Fallstricke“ und arbeitest immer mit der korrekten lokalen Zeit.

3.  **Git-Integration mit automatischen Fristen-Commits**  
    Über einen Git-Hook (Pre-Commit oder Pre-Push) kannst du deadliner so konfigurieren, dass es automatisch den aktuellen Commit-Hash und die verbleibende Zeit bis zur Deadline in ein `deadline.json`-File schreibt. So wird jede Code-Änderung mit dem Fristenstatus verknüpft – ideal für Audit-Trails und automatisierte Sprint-Reviews.

### So verwenden Sie es

1.  **Initialisierung und Konfiguration einer Deadline**  
    Führe im Projektverzeichnis `deadliner init` aus. Das Tool erstellt eine `.deadliner.yml`-Konfigurationsdatei. Definiere dort die erste Deadline:  
    ```yaml
    deadlines:
      - name: "Sprint-Review-Demo"
        due: "2024-12-15T10:00:00"
        timezone: "Europe/Berlin"
        escalation:
          - type: "email"
            target: "team@example.com"
            threshold: "24h"
          - type: "slack"
            target: "#sprint-reviews"
            threshold: "6h"
    ```  
    Danach `deadliner validate` ausführen, um die Konfiguration auf Syntax- und Zeitzonenfehler zu prüfen.

2.  **Automatische Deadline-Überwachung im CI/CD**  
    Binde deadliner in deine GitHub Actions oder GitLab CI ein. Beispiel für eine GitHub Action:  
    ```yaml
    - name: Deadline Check
      run: |
        npx deadliner check --config .deadliner.yml --fail-on-overdue
    ```  
    Diese Action schlägt fehl, wenn eine Deadline überschritten ist – und blockiert so den Merge oder Deployment. So stellst du sicher, dass keine Fristen stillschweigend verstreichen.

3.  **API-gesteuerte Fristenaktualisierung**  
    Nutze die REST-API, um Deadlines programmatisch zu aktualisieren – z. B. aus einem Jira-Webhook oder einem Slack-Befehl:  
    ```bash
    curl -X PATCH https://api.deadliner.dev/v1/deadlines/sprint-review-demo \
      -H "Authorization: Bearer $DEADLINER_TOKEN" \
      -H "Content-Type: application/json" \
      -d '{"due": "2024-12-16T14:00:00", "reason": "Product Owner krank"}'
    ```  
    Die API protokolliert jede Änderung mit einem Audit-Trail – inklusive Begründung und Zeitstempel.

### Beste Praktiken

1.  **Fristen nicht in UTC speichern, sondern mit IANA-Zeitzone**  
    Viele Tools speichern Deadlines standardmäßig in UTC. Das führt zu Verwirrung bei Teams in unterschiedlichen Zeitzonen. deadliner erzwingt die Angabe einer IANA-Zeitzone. **Best Practice:** Definiere jede Deadline mit der lokalen Zeit des verantwortlichen Teams und lasse deadliner die Konvertierung für Benachrichtigungen übernehmen. So vermeidest du „Deadline-Verschiebungen“ durch Zeitzonenfehler.

2.  **Eskalationsstufen frühzeitig testen**  
    Konfiguriere die Eskalationsschwellen nicht erst kurz vor der Deadline. **Best Practice:** Setze für jede Deadline eine Test-Eskalation (z. B. mit `--dry-run` oder einer separaten Test-Deadline) und prüfe, ob alle Benachrichtigungskanäle korrekt funktionieren. So stellst du sicher, dass im Ernstfall keine Benachrichtigung stumm bleibt.
