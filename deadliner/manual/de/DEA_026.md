# deadliner - DEA_026

## Überblick

deadliner ist ein schlankes, aber hochperformantes CLI-Tool und Library-Set für Entwickler, die Deadlines in agilen Workflows nicht nur verwalten, sondern aktiv **steuern** wollen. Es richtet sich an DevOps-Ingenieure, Scrum Master und Backend-Entwickler, die Termintracking direkt in ihre CI/CD-Pipeline oder Codebase integrieren möchten – ohne zusätzliche SaaS-Abonnements oder GUI-Overhead.

## Detaillierter Inhalt

### Was ist deadliner?

deadliner ist kein weiterer Kalender. Es ist ein **deterministisches Deadline-Engine**, das auf einem eigens entwickelten `deadline-graph` basiert. Statt einfacher Fristen werden Abhängigkeiten, Pufferzeiten und Wiederholungszyklen als gerichteter azyklischer Graph (DAG) modelliert. Das Ergebnis: eine präzise, maschinenlesbare Prognose, wann ein Task tatsächlich fertig sein muss – inklusive Risikobewertung durch Monte-Carlo-Simulation.

### Kernfunktionen

1. **Deadline-Graph-Engine mit Topologischer Sortierung**  
   Definiere Tasks mit Abhängigkeiten (`depends_on`) und Puffern (`lag`). deadliner berechnet automatisch den kritischen Pfad und gibt den spätestmöglichen Startzeitpunkt für jeden Task aus. Ideal für Microservice-Release-Planung oder Sprint-Backlog-Synchronisation.

2. **Git-Integrated Deadline Hooks**  
   Hänge Deadlines direkt an Branches oder Commits via `deadliner hook --branch feature/xyz --deadline 2025-04-15`. Der Hook blockiert Merges in `main`, falls die Deadline überschritten ist – oder triggert einen CI-Job für automatische Eskalation.

3. **Monte-Carlo-Risikoanalyse für Termintreue**  
   `deadliner simulate --tasks 50 --iterations 10000` liefert eine Wahrscheinlichkeitsverteilung der Fertigstellungstermine. Ausgabe als JSON oder Plot (via Matplotlib-Export). Perfekt für Stakeholder-Updates ohne Bullshit-Bingo.

### So verwenden Sie es

1. **Initialisierung eines Deadline-Graphen**  
   ```bash
   deadliner init --project my-release --tasks "build,test,deploy" --dependencies "test:build,deploy:test"
   ```  
   Legt eine `deadliner.yaml` an, die den DAG mit Standard-Puffern (z. B. 10 % der Task-Dauer) erstellt. Optional: `--timezone UTC` für globale Teams.

2. **Integration in CI/CD (GitHub Actions / GitLab CI)**  
   ```yaml
   - name: Check Deadline Compliance
     run: |
       deadliner check --branch ${{ github.head_ref }} --deadline 2025-04-15
       if [ $? -ne 0 ]; then
         echo "::error::Deadline überschritten. Merge blockiert."
         exit 1
       fi
   ```  
   Der Hook parst `deadliner.yaml`, berechnet den kritischen Pfad und vergleicht den aktuellen Branch-Zeitstempel mit der Deadline. Bei Überschreitung wird ein nicht-null Exit-Code zurückgegeben.

3. **Risiko-Simulation für Release-Planung**  
   ```bash
   deadliner simulate --input deadliner.yaml --iterations 5000 --output risk_report.json
   ```  
   Liefert ein JSON-Objekt mit `p50`, `p80`, `p95` und `p99`-Terminen. Nutze `jq` zur Extraktion:  
   ```bash
   jq '.p95' risk_report.json
   ```  
   Das Ergebnis: „95 % Wahrscheinlichkeit, dass der Release bis zum 12. Mai fertig ist.“ – perfekt für Management-Reporting.

### Beste Praktiken

1. **Deadlines als Code versionieren**  
   Lege `deadliner.yaml` im Repository-Root ab und versioniere sie mit Git. So wird jede Deadline-Änderung nachvollziehbar – inklusive Code-Review. Vermeide manuelle Eingriffe in der GUI; deadliner lebt von deterministischen, wiederholbaren Konfigurationen.

2. **Puffer nicht linear, sondern exponentiell wählen**  
   Statt fixer 20 % Puffer auf jeden Task: Nutze `lag: exponential` in der YAML, um Puffer basierend auf Task-Komplexität (z. B. Lines of Code oder Story Points) zu skalieren. Das verhindert, dass kleine Tasks übermäßig gepuffert werden und große Tasks untergehen.
