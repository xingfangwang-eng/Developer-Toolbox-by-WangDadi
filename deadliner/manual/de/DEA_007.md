# deadliner - DEA_007

## Übersicht

deadliner ist ein hochperformantes, ereignisgesteuertes Task-Deadline-Tracking-System, das speziell für agile Entwicklungsteams und DevOps-Pipelines entwickelt wurde. Es eliminiert manuelle Deadline-Verwaltung durch automatisierte Zeitstrafe-Kalkulation, dynamische Prioritätsverschiebung und native CI/CD-Integration. Anders als herkömmliche Tools wie Trello oder Jira, die auf statische Boards setzen, arbeitet deadliner mit einem **ereignisbasierten Zustandsautomaten** – jeder Task durchläuft Phasen wie `CREATED → IN_PROGRESS → AT_RISK → OVERDUE → BLOCKED`, wobei Aktionen wie Slack-Benachrichtigungen, automatische Eskalation oder Ressourcen-Neuzuweisung ausgelöst werden. Es ist das Schweizer Taschenmesser für alle, die Deadlines nicht nur verwalten, sondern **aktiv erzwingen** wollen.

## Kernfunktionen

### 1. Dynamische Zeitstrafe-Kalkulation mit exponentieller Verzögerung
deadliner berechnet automatisch die verbleibende Zeit bis zur Deadline und wendet eine **exponentielle Verzögerungsfunktion** an: Sobald ein Task 80% seiner Deadline erreicht hat, wird die Priorität um den Faktor `2^(t/T)` erhöht, wobei `t` die vergangene Zeit und `T` die Gesamtlaufzeit ist. Dies verhindert das typische "Last-Minute-Chaos" und zwingt Teams, frühzeitig zu handeln. Die Berechnung erfolgt in Echtzeit über einen integrierten **Time-Warp-Engine**, die sogar Zeitzonen und Feiertage (z. B. deutsche Brückentage) berücksichtigt.

### 2. Native CI/CD-Integration mit Pipeline-Trigger
deadliner lässt sich direkt in Ihre CI/CD-Pipeline einbinden (z. B. GitHub Actions, GitLab CI, Jenkins). Jeder Commit oder Merge Request aktualisiert automatisch den Task-Status. Wenn ein Task in den `OVERDUE`-Zustand gerät, kann deadliner einen **Rollback-Job** auslösen oder einen **Deployment-Freeze** aktivieren. Dies ist besonders nützlich für Teams, die **Continuous Delivery** mit strengen Deadlines kombinieren (z. B. Sprint-Reviews oder Release-Zyklen).

### 3. KI-gestützte Ressourcen-Prognose mit Monte-Carlo-Simulation
Statt einfacher Gantt-Diagramme nutzt deadliner eine **Monte-Carlo-Simulation**, um die Wahrscheinlichkeit zu berechnen, dass ein Task bis zur Deadline fertig wird. Basierend auf historischen Daten (z. B. durchschnittliche Bearbeitungszeit, Fehlerrate) generiert das Tool eine **Heatmap** der Engpässe und schlägt automatisch Ressourcen-Umverteilungen vor. Dies ist ein Game-Changer für Projektmanager, die mit **unsicheren Schätzungen** umgehen müssen.

## So verwenden Sie es

### Schritt 1: Task-Erstellung mit benutzerdefinierten Deadlines und Metriken
Erstellen Sie einen neuen Task über die CLI oder API. Geben Sie die Deadline im ISO-8601-Format an (z. B. `2025-03-15T18:00:00+01:00`). Verwenden Sie den Parameter `--penalty-mode`, um die Zeitstrafe-Kalkulation zu aktivieren. Optional können Sie Metriken wie `--estimated-hours` und `--complexity-score` (1-10) hinzufügen, um die Monte-Carlo-Simulation zu verbessern. Beispiel:
```bash
deadliner create --title "API-Integration abschließen" --deadline "2025-03-15T18:00:00+01:00" --penalty-mode exponential --estimated-hours 8 --complexity-score 7
```

### Schritt 2: Automatische Statusüberwachung und Eskalation
Nach der Erstellung überwacht deadliner den Task kontinuierlich. Sobald der Task 80% der Deadline erreicht hat, wird automatisch eine **Slack-Benachrichtigung** an den Verantwortlichen gesendet (konfigurierbar über `--notify-channel`). Wenn der Task in den `OVERDUE`-Zustand gerät, wird eine **Eskalationsstufe** ausgelöst: Zuerst wird der Team-Lead benachrichtigt, nach weiteren 24 Stunden der Product Owner. Sie können die Eskalationslogik über eine YAML-Datei anpassen:
```yaml
escalation:
  levels:
    - after: 0h
      notify: [team-lead]
    - after: 24h
      notify: [product-owner, cto]
  action: pause-pipeline
```

### Schritt 3: CI/CD-Integration und automatische Pipeline-Trigger
Binden Sie deadliner in Ihre CI/CD-Pipeline ein. Fügen Sie in Ihrer `.github/workflows/deadliner.yml` folgende Schritte hinzu:
```yaml
- name: Check Task Status
  run: |
    deadliner check --task-id ${{ github.event.issue.number }} --status
- name: Block Deployment if Overdue
  if: steps.check.outputs.status == 'OVERDUE'
  run: |
    deadliner trigger --action block-deployment --reason "Task is overdue"
```
Dies stellt sicher, dass kein Code deployed wird, solange kritische Tasks überfällig sind. Für GitLab CI verwenden Sie analog `deadliner check --task-id $CI_MERGE_REQUEST_IID`.

## Beste Praktiken

### 1. Nutzen Sie die Monte-Carlo-Simulation für Sprint-Planung
Statt sich auf Bauchgefühl zu verlassen, lassen Sie deadliner die Wahrscheinlichkeit berechnen, dass Ihr Sprint-Ziel erreicht wird. Führen Sie vor jedem Sprint eine Simulation durch: `deadliner simulate --sprint-id "Sprint-12" --runs 10000`. Das Tool gibt eine **Wahrscheinlichkeitsverteilung** aus (z. B. "80% Wahrscheinlichkeit, dass Task A und B fertig werden, aber nur 30% für Task C"). Nutzen Sie diese Daten, um Tasks neu zu priorisieren oder Ressourcen umzuverteilen. Dies ist besonders wertvoll in **verteilten Teams**, wo Abhängigkeiten schwer zu erkennen sind.

### 2. Kombinieren Sie deadliner mit Ihrer Incident-Management-Plattform
Viele Teams nutzen PagerDuty oder Opsgenie für Incidents. deadliner lässt sich nahtlos integrieren: Konfigurieren Sie einen Webhook, der bei `OVERDUE`-Status automatisch einen Incident erstellt. Dies stellt sicher, dass Deadlines nicht nur im Task-Board, sondern auch im **Incident-Management-Workflow** sichtbar sind. Beispiel-Webhook-Konfiguration:
```bash
deadliner config set --webhook-url "https://api.pagerduty.com/incidents" --api-key "your-key" --trigger-on overdue
```
So wird aus einem verpassten Deadline ein **P1-Incident**, der sofortige Aufmerksamkeit erfordert. Dies ist ein Paradebeispiel für **Shift-Left-Deadline-Management**.
