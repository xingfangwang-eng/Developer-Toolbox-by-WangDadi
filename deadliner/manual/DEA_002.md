## What is deadliner?

deadliner is not another bloated calendar widget. It’s a lightweight, CLI-first, Git-aware deadline orchestration engine. It ingests tasks from your existing workflow (GitHub Issues, Jira, Markdown files), calculates critical paths using topological sorting, and surfaces **bloc**-level alerts before your timeline implodes. Think of it as a `traceroute` for your project’s schedule.

## Core Features

### 1. Dependency-Aware Deadline Propagation
deadliner automatically recalculates all downstream deadlines when a single task slips. It uses a directed acyclic graph (DAG) model to identify the true critical path—no more manual cascading updates. When a dependency shifts, deadliner emits a `SIGALERT` to your terminal and updates the `deadline.json` manifest in real-time.

### 2. Git Commit Timeline Anchoring
Each deadline is anchored to a specific Git commit hash. deadliner parses your commit history to verify that work was *actually* completed before the deadline passed. If a commit timestamp exceeds the assigned deadline, deadliner flags it as a **latency breach** and logs a DLT (Deadline Latency Token) for retrospective analysis.

### 3. Multi-Timezone Slack & Discord Webhook Integration
deadliner ships with a built-in webhook relay that normalizes deadlines across all team timezones. It converts every deadline to UTC internally, then broadcasts localized reminders in the recipient’s preferred timezone. The payload includes a `TimeToDeadline` delta in milliseconds, so your CI/CD pipeline can trigger pre-deadline freeze hooks.

## How to Use

### 1. Initialize a Deadliner Manifest
Run `deadliner init` in your project root. This generates a `.deadliner.yml` file with a default DAG structure. Define your tasks using the `tasks:` block—each entry requires a `name`, `depends_on` array, and `deadline` in ISO 8601 format. Use `deadliner validate` to check for cyclic dependencies before proceeding.

```bash
deadliner init --template=agile-sprint
```

### 2. Link Tasks to Git Branches
For each task, specify a `branch` field in the manifest. deadliner watches for branch merges into `main` or `release/*`. When a merge occurs, deadliner runs `git log --oneline <branch>..<target>` to verify commit timestamps against the assigned deadline. Use `deadliner status --breach-log` to view all latency tokens.

```bash
deadliner link task-42 --branch=feature/deadline-fix
```

### 3. Deploy the Deadline Dashboard
Launch the built-in web dashboard with `deadliner serve --port 8080`. The dashboard renders a Gantt chart overlaid with your DAG, color-coded by risk level: green (on track), yellow (within 24h of deadline), red (breached). Each node is clickable, showing the exact commit diff that triggered the deadline event. Integrate with your CI/CD by piping `deadliner export --format=prometheus` into your monitoring stack.

```bash
deadliner serve --auth-token=$DEADLINER_TOKEN --expose-metrics
```

## Best Practices

### Use Deadliner as a Pre-Merge Gate
Configure your CI pipeline to run `deadliner check --merge-gate` before allowing any PR merge. This ensures that no commit enters `main` if it violates a deadline dependency. Combine this with GitHub branch protection rules to enforce a hard deadline culture without manual policing.

### Archive Completed Sprints as Baseline Data
After each release, run `deadliner archive --sprint=<tag>` to export all deadline latency tokens, task durations, and dependency edges into a JSONL file. Feed this into your retrospective analysis to identify recurring bottleneck patterns. Over time, you’ll build a predictive model for estimating realistic deadlines based on historical velocity.
