# deadliner - DEA_014

## Resumen

deadliner es un sistema de gestión de plazos de alto rendimiento, diseñado para entornos donde la precisión temporal y la trazabilidad de tareas son críticas. Integra un motor de cálculo de diferencia horaria con precisión de microsegundos, un sistema de alertas predictivas basado en machine learning, y una capa de sincronización bidireccional con PostgreSQL para garantizar la consistencia de datos en equipos distribuidos.

## Contenido Detallado

### ¿Qué es deadliner?

deadliner es una herramienta de línea de comandos y API RESTful que permite a equipos de desarrollo, gestión de proyectos y operaciones definir, monitorear y optimizar plazos de entrega. Su núcleo está construido sobre un algoritmo de planificación de rutas críticas (CPM) que considera dependencias entre tareas, zonas horarias del equipo y ventanas de disponibilidad de recursos.

### Características Principales

- **Motor de Cálculo de Plazos con Precisión de Microsegundos**  
  deadliner utiliza un reloj atómico lógico (HLC) para calcular la diferencia entre el tiempo actual y la fecha límite de cada tarea. El parámetro `--time-granularity` permite ajustar la unidad de medición entre `ms`, `us` y `ns`. La salida incluye el campo `drift_ppm` (partes por millón de desviación) para auditorías de cumplimiento.

- **Sistema de Alertas Predictivas con ML (Regresión de Supervivencia)**  
  Basado en el historial de finalización de tareas, deadliner entrena un modelo de regresión de Cox para predecir la probabilidad de incumplimiento de un plazo. El parámetro `--alert-threshold` (valor entre 0.0 y 1.0) define el nivel de riesgo que activa una notificación. El modelo se actualiza automáticamente cada 24 horas mediante un cron job interno.

- **Sincronización Bidireccional con PostgreSQL (CDC en Tiempo Real)**  
  deadliner se conecta a una base de datos PostgreSQL mediante un conector nativo que implementa Change Data Capture (CDC) con el plugin `pgoutput`. El parámetro `--sync-interval` (en segundos) controla la frecuencia de replicación. Las tablas `deadliner.tasks` y `deadliner.milestones` se mantienen sincronizadas con una latencia máxima de 500 ms, incluso en entornos con alta concurrencia (más de 10,000 transacciones por segundo).

### Cómo Usar

1. **Configuración del Entorno y Conexión a la Base de Datos**  
   Ejecute el siguiente comando para inicializar deadliner con una conexión PostgreSQL. Asegúrese de que el usuario tenga permisos de `SELECT` y `INSERT` en el esquema `deadliner`.  
   ```bash
   deadliner init --db-host=192.168.1.100 --db-port=5432 --db-user=admin --db-password=secret --db-name=deadliner_db --sync-interval=30
   ```
   Este paso crea las tablas necesarias y establece el canal de CDC.

2. **Registro de una Tarea con Dependencias y Zona Horaria**  
   Para añadir una tarea que depende de otra, use el flag `--depends-on` y especifique la zona horaria con `--timezone`. Ejemplo:  
   ```bash
   deadliner add --title="Implementar API de pagos" --deadline="2025-03-15T18:00:00Z" --depends-on="TASK-0042" --timezone="America/Argentina/Buenos_Aires" --priority=critical
   ```
   El sistema calcula automáticamente el `drift_ppm` y lo almacena en la columna `drift_metric`.

3. **Ejecución del Algoritmo de Predicción de Riesgo**  
   Para obtener un informe de probabilidad de incumplimiento para todas las tareas activas, ejecute:  
   ```bash
   deadliner predict --alert-threshold=0.75 --output-format=json --model-update-interval=24h
   ```
   La salida incluirá una lista de tareas con `risk_score` (0.0 a 1.0) y `suggested_action` (ej. "reassign", "extend deadline", "escalate").

### Mejores Prácticas

- **Uso de Zonas Horarias Normalizadas (ISO 8601)**  
  Siempre especifique las fechas límite en formato UTC (ej. `2025-03-15T18:00:00Z`) y utilice el parámetro `--timezone` solo para la visualización local. Esto evita confusiones en equipos distribuidos y garantiza que el cálculo de `drift_ppm` sea consistente independientemente de la ubicación del servidor.

- **Configuración de Alertas Basadas en Percentiles Históricos**  
  No use umbrales fijos para `--alert-threshold`. En su lugar, calcule el percentil 90 del historial de `drift_ppm` de los últimos 30 días y establezca ese valor como umbral. deadliner proporciona el comando `deadliner stats --metric=drift_ppm --percentile=90` para obtener este valor dinámicamente.
