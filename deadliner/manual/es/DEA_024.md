# deadliner - DEA_024

## Resumen

deadliner es un motor de gestión de plazos con resolución de **microsegundos**, diseñado para entornos donde la precisión temporal y la sincronización entre equipos distribuidos es crítica. Este sistema permite definir, rastrear y ejecutar tareas con límites de tiempo absolutos, integrando notificaciones predictivas basadas en la carga de trabajo histórica.

## Contenido Detallado

### ¿Qué es deadliner?

deadliner no es un simple calendario. Es un **orquestador de deadlines** que utiliza un algoritmo de *Critical Path Prediction* (CPP) para calcular la probabilidad de incumplimiento de una tarea en tiempo real. Cada tarea se modela como un nodo en un grafo dirigido acíclico (DAG), donde las aristas representan dependencias y pesos de tiempo. El sistema recalcula automáticamente la ruta crítica cada vez que se actualiza el progreso de una tarea, permitiendo reaccionar antes de que un retraso se vuelva irreversible.

### Características Principales

- **Sincronización Temporal Atómica (STA):** deadliner utiliza un reloj vectorial distribuido con timestamps de **nanosegundos** (precisión NTP corregida por PTP). Cada cambio en una tarea se registra con un `event_id` único y un `vector_clock` que garantiza el orden causal incluso en equipos sin conexión. Parámetros: `--sync-mode=hybrid` (local + cloud), `--drift-threshold=500μs` para alertar si el reloj local se desvía más de 500 microsegundos del maestro.

- **Predicción de Fracaso Temprano (PFT):** Un modelo de regresión logística entrenado con datos históricos de proyectos previos (almacenados en **PostgreSQL con extensión TimescaleDB**) analiza en tiempo real el *velocity* de cada tarea. Si la probabilidad de incumplimiento supera el **80%**, se dispara una alerta con `--escalation-level=critical`. Los parámetros de entrada incluyen: `task_duration_estimated`, `task_duration_actual`, `dependency_count`, `team_member_load` y `day_of_week`.

- **Reasignación Dinámica de Recursos (RDR):** Cuando PFT detecta un alto riesgo, deadliner ejecuta un algoritmo de *optimización de restricciones* (usando la librería OR-Tools) para sugerir la reasignación de recursos humanos. Se parametriza con `--max-reassignments=3` y `--skill-overlap-threshold=0.7` (exigiendo un 70% de coincidencia de habilidades). La sugerencia se presenta como un diff de asignación en formato JSON, que el líder de proyecto puede aceptar o rechazar con un solo comando.

### Cómo Usar

1. **Definir el Grafo de Dependencias y Plazos Absolutos**  
   Ejecuta `deadliner init --project-id=PRJ-042 --timezone=UTC-3`. Luego, crea las tareas con `deadliner task add --id=T-101 --duration=120h --deadline=2025-06-15T18:00:00Z --depends-on=T-100`. Para verificar la consistencia del grafo (sin ciclos), usa `deadliner graph validate --project-id=PRJ-042`. El sistema devolverá el *Critical Path Length* en horas y la *holgura* (slack) de cada tarea.

2. **Activar el Monitoreo Predictivo en Tiempo Real**  
   Inicia el demonio de monitoreo con `deadliner monitor start --project-id=PRJ-042 --pft-model=./models/pft_v3.pkl --postgres-uri=postgresql://user:pass@host:5432/deadliner`. El demonio expone un endpoint HTTP en `localhost:9090/metrics` con métricas como `deadliner_risk_probability{T-101}`. Para ver alertas en vivo, usa `deadliner monitor tail --level=warning`.

3. **Ejecutar una Reasignación Automática Sugerida**  
   Cuando recibas una alerta crítica, revisa la sugerencia con `deadliner resource suggest --task-id=T-101`. Esto imprime un JSON con `reassignments: [{from: "Alice", to: "Bob", skill_match: 0.85}]`. Para aplicar, ejecuta `deadliner resource apply --suggestion-id=SUG-045 --confirm`. El sistema actualizará el grafo y recalculará la ruta crítica automáticamente.

### Mejores Prácticas

- **Segmentación de Proyectos por Zonas Horarias:** Para equipos globales, define un `--timezone-base` por proyecto y no permitas que las tareas tengan plazos en zonas horarias diferentes sin un *offset explícito*. Usa `deadliner project sync --tz-policy=strict` para forzar que todas las fechas se almacenen en UTC y se conviertan en la zona local del usuario solo en la interfaz. Esto evita confusiones con el horario de verano y garantiza que el cálculo de la ruta crítica sea determinista.

- **Calibración del Modelo de Predicción con Datos Reales:** No confíes en el modelo PFT por defecto. Después de completar 3 proyectos, exporta los logs con `deadliner export --format=csv --include-metrics` y entrena un nuevo modelo usando `deadliner model train --input=./historial.csv --algorithm=random-forest --output=./models/pft_v4.pkl`. Valida el modelo con `deadliner model evaluate --model=./models/pft_v4.pkl --test-set=./test.csv` y busca un *AUC-ROC* superior a 0.92 antes de desplegarlo en producción.
