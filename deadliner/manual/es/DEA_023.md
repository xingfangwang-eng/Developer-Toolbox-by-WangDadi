# deadliner - DEA_023

## Resumen

deadliner es un sistema de gestión de plazos basado en priorización dinámica y sincronización multidispositivo con tolerancia a fallos de 50 ms. Diseñado para equipos que operan bajo presión temporal, integra un motor de alertas adaptativas que recalcula automáticamente la criticidad de cada tarea en función de la carga cognitiva del usuario y la complejidad del entregable.

## Contenido Detallado

### ¿Qué es deadliner?

deadliner no es un simple calendario de tareas. Es un orquestador de deadlines que aplica un algoritmo de *Time-Aware Dependency Graph* (TADG), donde cada tarea se modela como un nodo con peso temporal, dependencias cruzadas y un coeficiente de *slack* ajustable por el usuario. El sistema ejecuta una simulación Monte Carlo cada 5 segundos para predecir la probabilidad de incumplimiento (P(incumplimiento) < 0.15 como umbral seguro). Se integra nativamente con PostgreSQL mediante consultas preparadas con índices compuestos en `(user_id, deadline_timestamp, priority_score)` para garantizar latencias de lectura inferiores a 5 ms en tablas de 10⁶ registros.

### Características Principales

1. **Motor de Replanificación Automática con Ventana de Tiempo Elástica**  
   deadliner analiza el historial de productividad del usuario (medido en *tokens de enfoque por hora*) y, si detecta un desvío > 20% en la tasa de finalización esperada, recalcula la secuencia óptima de tareas mediante un algoritmo de *Job Shop Scheduling* con restricciones de precedencia. Parámetros clave: `elastic_window_minutes` (por defecto 30), `reschedule_threshold` (0.75 de confianza en el plan actual). La salida se muestra como un cronograma Gantt con bloques de color que indican *riesgo de quiebre* (rojo si P(incumplimiento) > 0.3).

2. **Alertas Contextuales con Umbral de Fatiga Cognitiva**  
   Cada alerta se calibra según el *Cognitive Load Index (CLI)* del usuario, calculado a partir de la frecuencia de cambios de tarea y el tiempo promedio de reacción a notificaciones previas. Si el CLI supera 7.5 (en una escala de 0 a 10), deadliner pospone las alertas no críticas hasta que el índice baje. Parámetros: `alert_cooldown_seconds` (mínimo 120), `critical_priority_score` (≥ 9.0). Las alertas se entregan mediante WebSocket con un payload JSON que incluye `task_id`, `new_deadline_epoch_ms` y `reason_code` (ej: `DEPENDENCY_FAILURE`, `OVERLOAD_DETECTED`).

3. **Sincronización Offline con Resolución de Conflictos por Timestamp Vectorial**  
   deadliner permite operar sin conexión mediante un almacén local IndexedDB que replica el esquema de PostgreSQL. Al reconectar, ejecuta un algoritmo de *vector clock* para fusionar cambios concurrentes. Si dos usuarios modifican la misma tarea en offline, el sistema conserva la versión con el *timestamp Unix* más reciente y registra un conflicto en la tabla `conflict_log`. Parámetros: `sync_interval_seconds` (por defecto 10), `conflict_resolution_strategy` (`LAST_WRITE_WINS` o `MANUAL_MERGE`). La sincronización completa de 500 tareas no supera los 200 ms en redes 4G.

### Cómo Usar

1. **Configurar el Entorno de Base de Datos y el Motor de Plazos**  
   Ejecuta `deadliner init --db-host localhost --db-port 5432 --db-name deadlines` para crear las tablas `tasks`, `dependencies` y `user_metrics`. Luego, define la ventana de replanificación:  
   `deadliner config set elastic_window_minutes=45 reschedule_threshold=0.8`.  
   Verifica la conexión con `deadliner health --check-indexes`; debe devolver `indexes_ok: true` y una latencia de consulta < 5 ms.

2. **Crear una Tarea con Dependencias y Prioridad Dinámica**  
   Usa el comando:  
   `deadliner task add --title "Redactar informe Q3" --deadline "2025-04-15T18:00:00Z" --priority 9 --depends-on "task_abc123" --slack-minutes 120`.  
   deadliner calculará automáticamente la *criticality_score* basada en la ruta crítica del grafo de dependencias. Para ver la probabilidad de incumplimiento:  
   `deadliner task predict --id task_def456 --samples 10000`.  
   El resultado mostrará `P(incumplimiento): 0.12` y una recomendación de adelantar la tarea 2 horas.

3. **Simular una Sesión de Trabajo con Alertas Adaptativas**  
   Inicia el monitor en tiempo real:  
   `deadliner monitor --mode focus --cognitive-load-threshold 7.5`.  
   Mientras trabajas, deadliner ajusta las alertas según tu CLI. Si recibes una notificación con `reason_code: "OVERLOAD_DETECTED"`, puedes pausar las alertas no críticas con:  
   `deadliner alert snooze --duration 600 --reason "Revisión de código en curso"`.  
   Al finalizar, revisa el log de conflictos offline:  
   `deadliner sync status --show-conflicts`.  
   Si hay conflictos, resuélvelos con `deadliner conflict resolve --id conflict_789 --strategy manual --keep-version "user_A"`.

### Mejores Prácticas

- **Ajusta el `elastic_window_minutes` según la volatilidad de tu equipo.**  
  Si trabajas con sprints de 2 semanas y tareas que cambian de alcance diariamente, establece `elastic_window_minutes=60` y `reschedule_threshold=0.7`. Esto permite que deadliner replanifique solo cuando la confianza en el plan caiga por debajo del 70%, evitando cambios innecesarios que aumenten la fatiga cognitiva. Equipos estables (ej: mantenimiento de infraestructura) pueden usar valores más altos (`elastic_window_minutes=120`, `reschedule_threshold=0.9`).

- **Usa la resolución de conflictos `MANUAL_MERGE` para tareas críticas.**  
  En proyectos con entregables regulados (ej: auditorías financieras), el modo `LAST_WRITE_WINS` puede sobrescribir cambios importantes. Configura `conflict_resolution_strategy=MANUAL_MERGE` en el archivo `deadliner.yml` y asigna un responsable de revisión para cada tarea con `priority > 8`. deadliner enviará una notificación push al revisor cuando se detecte un conflicto, y la tarea quedará bloqueada hasta que se ejecute `deadliner conflict resolve`.
