# deadliner - DEA_026

## Resumen

deadliner es un orquestador de plazos críticos diseñado para entornos donde cada milisegundo cuenta. Integra un motor de reglas temporales con validación de dependencias en cascada, permitiendo gestionar desde sprints ágiles hasta cronogramas regulatorios con precisión nanométrica. Su núcleo utiliza un algoritmo de planificación basado en el *Critical Path Method* (CPM) con tolerancia a la deriva temporal, ideal para equipos que operan bajo presión de tiempo real.

## Contenido Detallado

### ¿Qué es deadliner?

deadliner no es un simple calendario. Es un sistema de **gestión proactiva de plazos** que evalúa continuamente el riesgo de retraso mediante un modelo de predicción estocástica. Cada tarea se asigna a un nodo en un grafo dirigido acíclico (DAG), donde las dependencias se resuelven en microsegundos. Está diseñado para integrarse directamente con pipelines CI/CD, ERPs y sistemas de ticketing, utilizando PostgreSQL como almacén de estado transaccional y Redis para la cola de eventos en tiempo real.

### Características Principales

- **Motor de Plazos con Tolerancia a la Deriva (Drift-Tolerant Scheduler)**  
  Calcula automáticamente la ruta crítica de cada proyecto utilizando el algoritmo **CPM++**. Permite definir ventanas de holgura (slack) en milisegundos y ajusta dinámicamente las prioridades cuando una tarea supera el 85% de su tiempo asignado. Parámetros clave: `slack_window_ms`, `drift_threshold_ratio`, `critical_path_depth`.

- **Validación de Dependencias en Cascada con Bloqueo Temporal**  
  Implementa un sistema de **bloqueo preventivo** que impide que una tarea se inicie si su predecesora no ha cumplido con un hito intermedio. Utiliza un mecanismo de `timestamp locking` con precisión de nanosegundos (basado en `CLOCK_MONOTONIC`). Soporta dependencias de tipo `finish-to-start`, `start-to-start` y `finish-to-finish` con retardos configurables (`lag_ms`).

- **Analítica Predictiva de Riesgo de Retraso (Delay Risk Engine)**  
  Evalúa en tiempo real la probabilidad de incumplimiento de plazos mediante un modelo de regresión logística entrenado con datos históricos de rendimiento del equipo. Genera alertas proactivas cuando el riesgo supera el umbral del 70%. Parámetros: `risk_threshold_pct`, `history_window_days`, `model_confidence_interval`.

### Cómo Usar

1. **Configuración del Proyecto y Definición de Plazos Críticos**  
   - Ejecuta `deadliner init --project-name "Sprint-Q4" --timezone "America/Mexico_City"` para crear el archivo de configuración `deadliner.yml`.  
   - Define las tareas con sus duraciones exactas en milisegundos:  
     ```yaml
     tasks:
       - id: "TASK-001"
         duration_ms: 86400000  # 24 horas
         deadline_utc: "2025-03-15T18:00:00Z"
         depends_on: []
         slack_window_ms: 3600000
     ```  
   - Aplica la validación de dependencias con `deadliner validate --dag-graph --output-format=dot`.

2. **Ejecución del Motor de Plazos y Monitoreo en Tiempo Real**  
   - Inicia el scheduler con `deadliner run --engine-mode=drift-tolerant --redis-uri=redis://localhost:6379`.  
   - Monitorea la ruta crítica en vivo usando `deadliner watch --critical-path --refresh-interval-ms=500`.  
   - Si una tarea supera el umbral de deriva, el sistema genera automáticamente un `drift_event` y reasigna prioridades.

3. **Análisis de Riesgo y Generación de Informes**  
   - Ejecuta `deadliner risk --project-id "Sprint-Q4" --threshold=70` para obtener una matriz de probabilidad de retraso por tarea.  
   - Exporta un informe en formato JSON con `deadliner report --format=json --output=./reports/risk_analysis.json`.  
   - Integra los resultados con tu dashboard de Grafana mediante el endpoint `POST /api/v1/events` con payload `{ "task_id": "TASK-001", "risk_pct": 85 }`.

### Mejores Prácticas

- **Segmentación de Plazos con Ventanas de Holgura Dinámicas**  
  No asignes un solo deadline monolítico. Divide cada tarea en hitos intermedios con ventanas de holgura de al menos el 10% de la duración total. Por ejemplo, para una tarea de 7 días, configura `slack_window_ms = 604800000`. Esto permite absorber retrasos menores sin afectar la ruta crítica.

- **Integración Continua con el Pipeline de CI/CD**  
  Conecta deadliner directamente con tu sistema de integración continua (Jenkins, GitLab CI). Cuando un commit rompe una dependencia temporal, el motor debe bloquear automáticamente el despliegue hasta que se resuelva. Usa el hook `post-commit` para ejecutar `deadliner validate --fail-fast --exit-code`.
