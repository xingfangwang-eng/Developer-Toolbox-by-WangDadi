# deadliner - DEA_021

## Resumen

deadliner es un motor de gestión de plazos con resolución de conflictos en tiempo real, diseñado para entornos de alta concurrencia donde cada milisegundo cuenta. Integra un planificador predictivo basado en ventanas de tiempo ajustables y un sistema de notificaciones con priorización dinámica.

## Contenido Detallado

### ¿Qué es deadliner?

deadliner es una infraestructura de orquestación de tareas que, mediante un algoritmo de calendarización basado en restricciones (CP-SAT), asigna deadlines a partir de dependencias, recursos y cargas de trabajo históricas. Su núcleo opera con una precisión de ±1 ms en la detección de vencimientos, utilizando un bus de eventos asíncrono sobre Redis Streams y persistencia dual en PostgreSQL (capa relacional) y MongoDB (capa de series temporales). Está pensado para equipos que necesitan coordinar cientos de miles de tareas diarias sin perder trazabilidad ni tolerancia a fallos.

### Características Principales

- **Planificador de ventanas de tiempo con colisión cero**: deadliner implementa un algoritmo de *interval tree* que evalúa solapamientos entre tareas en O(log n). Cada tarea se define con parámetros `start_time` (timestamp Unix en microsegundos), `duration_ms` (duración en milisegundos) y `hard_deadline` (límite absoluto). Si se detecta un conflicto, el motor reubica automáticamente la tarea en la primera ventana libre disponible, respetando prioridades numéricas (1-100, donde 1 es crítica). La reasignación se registra en una tabla `task_reschedules` con `reason_code` y `delta_ms`.

- **Notificaciones con escalado exponencial y umbral de fatiga**: Cada deadline activa un pipeline de alertas multicanal (Slack, correo SMTP, webhook REST). El sistema calcula un *fatigue score* basado en la frecuencia de recordatorios previos (parámetro `reminder_interval_ms` mínimo de 30.000 ms). Si el score supera el umbral configurable `max_fatigue` (por defecto 0.85), se omite la notificación y se eleva un evento `ALERT_SUPPRESSED` al log central. Las métricas de entrega se almacenan en `notification_logs` con campos `channel`, `status_code` y `latency_ms`.

- **Dashboard de carga predictiva con ventanas deslizantes**: deadliner expone una API REST que devuelve, para cualquier intervalo `from_ts` / `to_ts` (en ISO 8601), una matriz de densidad de tareas agrupada por buckets de `bucket_size_ms` (mínimo 60.000 ms). El endpoint `GET /api/v1/load?bucket_size_ms=300000&from_ts=2025-04-01T00:00:00Z` devuelve un JSON con `buckets[]`, cada uno conteniendo `count`, `avg_duration_ms`, `p95_duration_ms` y `conflict_count`. Esta información permite a los gestores redistribuir cargas antes de que ocurran cuellos de botella.

### Cómo Usar

1. **Inicializar el motor de deadlines con persistencia dual**  
   Ejecutar `deadliner init --db-postgres "postgres://user:pass@host:5432/deadliner?sslmode=require" --db-mongo "mongodb://user:pass@host:27017/deadliner?replicaSet=rs0" --redis "redis://host:6379/0"`. Este comando crea las tablas relacionales (`tasks`, `dependencies`, `reschedules`) y las colecciones de series temporales (`task_metrics`, `notification_logs`). Verificar la conectividad con `deadliner health --timeout 5000`.

2. **Registrar una tarea con dependencias y prioridad**  
   Usar `deadliner task add --title "Despliegue v3.2" --start-time 1711929600000000 --duration-ms 450000 --hard-deadline 1711933200000000 --priority 5 --depends-on "uuid-de-la-tarea-previa" --tags "produccion,bloqueante"`. El sistema devuelve un UUID único y, si hay colisiones, un array `conflicts` con las tareas afectadas y la nueva ventana sugerida. Para ver el estado en tiempo real: `deadliner task status <uuid> --format json`.

3. **Configurar alertas multicanal con umbral de fatiga**  
   Crear un archivo `alert_rules.yaml`:
   ```yaml
   channels:
     - type: slack
       webhook: "https://hooks.slack.com/services/T00/B00/xxxx"
       cooldown_ms: 60000
     - type: smtp
       host: "smtp.example.com"
       port: 587
       from: "deadliner@example.com"
       to: "team@example.com"
   fatigue:
     max_fatigue: 0.85
     window_ms: 3600000
   ```
   Luego cargarlo con `deadliner alerts apply --config alert_rules.yaml --dry-run` para validar la sintaxis. Sin `--dry-run`, se activan las reglas en el demonio de alertas.

### Mejores Prácticas

- **Segmentar deadlines por criticidad y usar ventanas de tiempo asimétricas**: Asigna prioridades 1-3 a tareas que afectan a clientes (con `hard_deadline` estricto) y prioridades 8-10 a tareas internas (con `duration_ms` flexible). deadliner, por defecto, nunca reubica tareas de prioridad 1-3; en su lugar, escala el conflicto a un gestor humano mediante un evento `ESCALATION_REQUIRED`. Esto evita que tareas críticas sean desplazadas por cargas secundarias.

- **Monitorear la métrica `conflict_rate` semanalmente**: Ejecuta `deadliner metrics --metric conflict_rate --window 7d --bucket-size-ms 86400000`. Si la tasa de conflictos supera el 5% en algún bucket, revisa las ventanas de tiempo asignadas y considera aumentar `bucket_size_ms` o redistribuir tareas manualmente. Un `conflict_rate` alto suele indicar que el equipo está sobrecargando ciertos horarios; deadliner puede sugerir ventanas alternativas mediante `deadliner suggest-slot --duration-ms 300000 --priority 5`.
