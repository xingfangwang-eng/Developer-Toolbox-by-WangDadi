# deadliner - DEA_022

## Resumen

deadliner es un sistema de gestión de plazos con precisión de microsegundos, diseñado para entornos donde el incumplimiento de una fecha crítica implica penalizaciones contractuales o pérdida de reputación. Integra motores de cálculo temporal asíncronos, alertas predictivas basadas en machine learning y sincronización bidireccional con calendarios empresariales (Google Calendar, Outlook, iCal). El proyecto está orientado a equipos de desarrollo, consultorías legales y gestores de proyectos con alta exigencia de trazabilidad.

## Contenido Detallado

### ¿Qué es deadliner?

deadliner no es un simple recordatorio. Es una arquitectura de cumplimiento de plazos que opera sobre una base de datos PostgreSQL con índices temporales compuestos (fecha_hora + huso horario + prioridad). Cada tarea se registra con un identificador UUID v7 y un timestamp de creación en nanosegundos. El sistema evalúa en tiempo real la holgura de cada deadline, calculando la ruta crítica de dependencias entre tareas. Si una tarea padre se retrasa más de 500 ms, se dispara una alerta escalable a todo el árbol de dependencias.

### Características Principales

- **Motor de Cálculo de Holgura Temporal con Precisión de Microsegundos**  
  deadliner implementa un algoritmo de *critical path method* (CPM) modificado, que evalúa la holgura total y libre de cada tarea en microsegundos. El parámetro `--tolerance-threshold` permite definir un límite de tolerancia (por defecto 1000 μs). Si la holgura libre cae por debajo de ese umbral, se genera automáticamente una tarea de mitigación con prioridad `P0`. El sistema se conecta a PostgreSQL mediante `pgx` (Go) y utiliza tablas particionadas por mes para garantizar consultas en menos de 10 ms incluso con 10 millones de registros.

- **Sistema de Alertas Predictivas con Red Neuronal Ligera**  
  Basado en un modelo LSTM entrenado con 50.000 historiales de proyectos, deadliner predice la probabilidad de incumplimiento de un deadline con una ventana de 72 horas. El parámetro `--prediction-window` (en horas) ajusta el horizonte de predicción. La salida incluye un score de riesgo (0.0 a 1.0) y una recomendación de acción (ej. *reasignar recurso*, *notificar stakeholder*). Las alertas se envían por webhook, Slack o correo electrónico con una latencia máxima de 200 ms.

- **Sincronización Bidireccional con Calendarios Empresariales mediante Webhooks**  
  deadliner expone un endpoint REST `/sync/calendar` que recibe eventos de Google Calendar, Outlook o iCal y los transforma en tareas dentro del sistema. Cada evento se mapea con los campos `calendar_id`, `event_uid`, `start_time` (en ISO 8601 con zona horaria), `end_time` y `recurrence_rule` (RRULE). La sincronización es bidireccional: cualquier cambio en deadliner (reprogramación, cancelación, priorización) se refleja en el calendario origen en menos de 5 segundos. El parámetro `--sync-interval` define el intervalo de polling (por defecto 30 s).

### Cómo Usar

1. **Configuración del Entorno y Conexión a PostgreSQL**  
   Ejecuta el binario `deadliner` con las siguientes variables de entorno:
   ```bash
   export DEADLINER_DB_HOST=localhost
   export DEADLINER_DB_PORT=5432
   export DEADLINER_DB_USER=admin
   export DEADLINER_DB_PASSWORD=secret
   export DEADLINER_DB_NAME=deadliner_prod
   export DEADLINER_TOLERANCE_THRESHOLD=1000
   export DEADLINER_PREDICTION_WINDOW=72
   export DEADLINER_SYNC_INTERVAL=30
   ```
   Luego inicia el servicio con:
   ```bash
   ./deadliner --mode server --port 8080
   ```
   Verifica la conexión consultando el endpoint `/health`:
   ```bash
   curl http://localhost:8080/health
   ```
   Debe devolver `{"status":"ok","db_latency_ms":2}`.

2. **Creación de una Tarea con Dependencias y Holgura Personalizada**  
   Usa el endpoint `/tasks` con una solicitud POST:
   ```json
   {
     "title": "Entrega del informe Q3",
     "description": "Incluir análisis de ventas y proyecciones",
     "deadline": "2025-10-15T18:00:00-05:00",
     "priority": "P1",
     "dependencies": [
       {"task_id": "0192f3a1-...", "lag_microseconds": 500000}
     ],
     "tolerance_threshold": 500
   }
   ```
   El sistema validará que la tarea dependiente exista y que el `lag_microseconds` sea un número entero positivo. La respuesta incluirá el `task_id` (UUID v7) y la holgura calculada en microsegundos.

3. **Configuración de Alertas Predictivas y Sincronización con Calendario**  
   Para activar la predicción de incumplimiento, ejecuta:
   ```bash
   ./deadliner --enable-prediction --prediction-window 72 --webhook-url https://hooks.slack.com/services/...
   ```
   Para sincronizar con Google Calendar, primero autentícate mediante OAuth 2.0:
   ```bash
   ./deadliner --auth-google --client-id <CLIENT_ID> --client-secret <CLIENT_SECRET> --redirect-uri http://localhost:8080/oauth/callback
   ```
   Luego, asocia un calendario con el comando:
   ```bash
   ./deadliner --sync-calendar --calendar-id primary --sync-interval 30
   ```
   El sistema comenzará a sincronizar eventos cada 30 segundos.

### Mejores Prácticas

- **Segmentación de Tareas por Zona Horaria y Prioridad**  
  En equipos distribuidos globalmente, asigna un huso horario explícito a cada tarea usando el campo `timezone` (formato IANA, ej. `America/Mexico_City`). deadliner calculará automáticamente la diferencia horaria y ajustará la holgura. Para tareas críticas (P0), establece un `tolerance_threshold` de 100 μs; para tareas de baja prioridad (P3), usa 5000 μs. Esto evita falsos positivos en alertas y optimiza el uso de recursos.

- **Auditoría Continua con Exportación de Logs a Elasticsearch**  
  deadliner genera logs estructurados en formato JSON que incluyen `task_id`, `event_type` (created, updated, alerted, synced), `timestamp` (en nanosegundos) y `latency`. Configura la exportación automática a Elasticsearch mediante el parámetro `--log-output elasticsearch://localhost:9200/deadliner-logs`. Esto permite construir dashboards en Kibana para monitorear la salud del sistema, identificar cuellos de botella y generar informes de cumplimiento auditables.
