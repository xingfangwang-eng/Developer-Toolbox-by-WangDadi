# deadliner - DEA_030

## Resumen

deadliner es una herramienta de gestión de plazos de alto rendimiento, diseñada para sincronizar cronogramas complejos con precisión de milisegundos. Integra un motor de planificación predictiva que anticipa cuellos de botella en flujos de trabajo multidisciplinarios. Ideal para equipos que manejan dependencias críticas entre tareas y requieren una visibilidad granular de la carga de trabajo.

## Contenido Detallado

### ¿Qué es deadliner?

deadliner es un sistema de orquestación de plazos que combina un planificador basado en grafos dirigidos con un motor de alertas adaptativas. A diferencia de herramientas convencionales, deadliner calcula la ruta crítica de cada proyecto en tiempo real, ajustando automáticamente las prioridades según la disponibilidad de recursos y la holgura de cada tarea. Utiliza PostgreSQL como backend transaccional, garantizando consistencia en entornos con alta concurrencia de ediciones.

### Características Principales

- **Planificador de Ruta Crítica con Ventana Temporal:** deadliner modela cada proyecto como un grafo acíclico dirigido (DAG). Cada nodo representa una tarea con parámetros `duración_estimada` (en milisegundos), `dependencias` (lista de UUIDs de tareas predecesoras) y `holgura_mínima` (en segundos). El motor recalcula la ruta crítica cada vez que se modifica un nodo, actualizando el campo `fecha_límite_absoluta` en la tabla `tareas` de PostgreSQL mediante disparadores (triggers) optimizados con índices compuestos en `(proyecto_id, fecha_límite_absoluta)`.

- **Alertas Predictivas de Desviación:** Implementa un modelo de regresión lineal simple sobre el historial de progreso de cada tarea, almacenado en la tabla `progreso_historico` (columnas: `tarea_id`, `timestamp`, `porcentaje_completado`). Cuando la pendiente de la regresión indica que la fecha de finalización estimada superará la `fecha_límite_absoluta` en más de un umbral configurable (por defecto, 5% de la duración total), deadliner genera una alerta de tipo `ALTA`, `MEDIA` o `BAJA` en la tabla `alertas`. La alerta incluye un campo `desviación_estimada` en milisegundos y un enlace directo a la tarea.

- **Simulación de Escenarios "Qué Pasaría Si":** Permite al usuario clonar un proyecto completo en una tabla `simulaciones` (con un `UUID` único y una copia profunda de todas las tareas asociadas). El usuario puede modificar parámetros como `duración_estimada` o `dependencias` en la simulación sin afectar el proyecto real. deadliner ejecuta entonces el algoritmo de ruta crítica sobre la simulación y compara los resultados con el proyecto original, generando un informe de diferencias que incluye el delta de tiempo en la `fecha_límite_absoluta` de cada tarea y el cambio en la holgura total del proyecto.

### Cómo Usar

1. **Configuración Inicial de un Proyecto y sus Tareas:**
   - Ejecuta `deadliner create-project --nombre "Lanzamiento Q3" --zona-horaria "America/Mexico_City"`. Esto crea un registro en la tabla `proyectos` con un `UUID` generado automáticamente y establece la zona horaria para todos los cálculos de fechas.
   - Añade tareas con `deadliner add-task --proyecto-id <UUID> --nombre "Redactar manual" --duracion-estimada 86400000 --dependencias "UUID-tarea-A,UUID-tarea-B" --holgura-minima 3600`. El parámetro `duracion-estimada` se expresa en milisegundos (86400000 ms = 24 horas). `dependencias` acepta una lista separada por comas de UUIDs de tareas que deben completarse antes. `holgura-minima` define el tiempo de amortiguación en segundos.
   - Verifica la estructura del grafo con `deadliner graph --proyecto-id <UUID> --formato dot`. Esto genera un archivo DOT que puedes visualizar con herramientas como Graphviz para inspeccionar dependencias y la ruta crítica.

2. **Monitoreo y Actualización de Progreso:**
   - Actualiza el progreso de una tarea: `deadliner update-progress --tarea-id <UUID> --porcentaje 45 --timestamp "$(date +%s%3N)"`. El timestamp debe ser en milisegundos desde la época Unix. deadliner inserta este registro en `progreso_historico` y dispara el cálculo de la alerta predictiva.
   - Consulta el estado actual del proyecto con `deadliner status --proyecto-id <UUID> --formato json`. La salida JSON incluye un array `tareas` con campos como `fecha_limite_absoluta`, `progreso_actual`, `alerta_asociada` (si existe) y `desviacion_estimada`. También incluye un objeto `ruta_critica` con la lista de UUIDs de tareas en la ruta crítica y el `tiempo_total_proyecto` en milisegundos.

3. **Ejecución de una Simulación de Escenario:**
   - Clona el proyecto para simular: `deadliner simulate clone --proyecto-id <UUID> --nombre-simulacion "Q3 con retraso en manual"`. Esto crea un nuevo registro en `simulaciones` con un `UUID` propio y una copia de todas las tareas.
   - Modifica parámetros en la simulación: `deadliner simulate update-task --simulacion-id <UUID-simulacion> --tarea-id <UUID-tarea-copiada> --duracion-estimada 172800000`. Esto cambia la duración de esa tarea específica en la simulación a 48 horas.
   - Genera el informe de diferencias: `deadliner simulate diff --simulacion-id <UUID-simulacion> --proyecto-id <UUID-proyecto>`. El comando devuelve una tabla en la terminal con las columnas: `tarea_nombre`, `fecha_original`, `fecha_simulada`, `delta_ms`, `impacto_en_ruta_critica` (booleano).

### Mejores Prácticas

- **Uso de Holgura Mínima para Amortiguar Incertidumbres:** Asigna un valor de `holgura_mínima` a cada tarea basado en la complejidad histórica. Por ejemplo, para tareas de desarrollo de software, una holgura del 20% sobre la duración estimada suele ser efectiva. deadliner utilizará este valor para calcular la holgura total del proyecto y alertar cuando esta se reduzca por debajo de un umbral crítico (configurable en la tabla `configuracion_proyecto`, campo `umbral_holgura_critica` en milisegundos). Esto previene que una sola tarea consuma todo el tiempo de reserva del proyecto.

- **Integración Continua con Sistemas de CI/CD:** Configura deadliner para que actualice automáticamente el progreso de las tareas asociadas a commits de código. Por ejemplo, un hook en tu pipeline de Jenkins o GitHub Actions puede ejecutar `deadliner update-progress --tarea-id <UUID> --porcentaje 100 --timestamp $(date +%s%3N)` al fusionar un pull request que cierra una issue. Además, puedes usar el comando `deadliner check-blockers --proyecto-id <UUID>` en tu pipeline para que falle si hay tareas en la ruta crítica con una desviación estimada mayor a cero, impidiendo así despliegues que dependan de trabajo no completado.
