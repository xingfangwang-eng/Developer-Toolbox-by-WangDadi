# deadliner - DEA_013

## Resumen

deadliner es un motor de gestión de plazos críticos diseñado para entornos de alta presión donde el incumplimiento de una fecha implica penalizaciones contractuales severas. El sistema opera con una precisión de ±1 milisegundo en el cálculo de vencimientos, integrando zonas horarias UTC+0 a UTC+14, y sincroniza automáticamente con calendarios corporativos vía API REST. Cada tarea se evalúa en tiempo real contra umbrales de riesgo (Rojo: <24h, Naranja: 24-72h, Verde: >72h), generando alertas push con prioridad P0-P3.

## Contenido Detallado

### ¿Qué es deadliner?

deadliner es un planificador determinista que resuelve el problema de la fragmentación de plazos en equipos distribuidos. A diferencia de herramientas genéricas como Trello o Asana, deadliner implementa un algoritmo de **Deadline Propagation Graph (DPG)** que calcula automáticamente las dependencias entre tareas usando un grafo acíclico dirigido (DAG). Cada nodo del grafo contiene un campo `deadline_epoch_ms` (timestamp Unix en milisegundos), y el sistema recalcula en cascada cualquier cambio en la ruta crítica. La base de datos subyacente es PostgreSQL con índices GiST para consultas de rango temporal (`tsrange`), permitiendo búsquedas de plazos superpuestos en menos de 50ms para 10⁶ registros.

### Características Principales

- **Motor de Colisión de Plazos (Deadline Collision Engine)**  
  Detecta automáticamente conflictos cuando dos tareas asignadas al mismo recurso humano tienen plazos que se superponen en un intervalo menor a 2 horas. Usa un algoritmo de *interval tree* con complejidad O(log n + m) para identificar solapamientos. Parámetros configurables: `min_overlap_minutes=120`, `resource_idle_time_ms=300000`. La salida es un JSON con `collision_id`, `task_ids[]`, `severity_score` (0.0-1.0) y `suggested_reschedule_epoch`.

- **Simulador de Escenarios "What-If"**  
  Permite ejecutar un modelo de Monte Carlo con 10,000 iteraciones para predecir la probabilidad de cumplir un plazo dado un conjunto de tareas. Cada iteración varía la duración estimada según una distribución log-normal con parámetros `mu` y `sigma` extraídos del historial de la tarea. El resultado incluye: `p_completion` (probabilidad 0-1), `p50_epoch`, `p95_epoch`, y `risk_exposure_hours`. La simulación se ejecuta en un worker de Celery con prioridad `HIGH` y timeout de 120 segundos.

- **Sincronización Bidireccional con Calendarios Externos (iCal/CalDAV)**  
  Conecta con Google Calendar, Outlook y cualquier servidor CalDAV mediante OAuth 2.0. Cada evento de calendario se mapea a una tarea de deadliner con campos: `external_event_id`, `last_sync_epoch_ms`, `sync_direction` (push/pull/bidirectional). El conflicto de edición se resuelve con *Last Writer Wins* (LWW) basado en `updated_at` del evento. La latencia de sincronización es menor a 5 segundos para 100 eventos concurrentes.

### Cómo Usar

1. **Configurar el Entorno de Simulación de Plazos**  
   Ejecuta el comando `deadliner init --project-id PROJ_2025 --timezone America/Santiago --calendar-backend caldav --calendar-url https://caldav.example.com/principal/`. Esto crea un archivo `deadliner_config.yaml` con parámetros como `simulation_iterations=10000`, `collision_min_overlap=120`, `sync_interval_seconds=300`. Verifica que PostgreSQL esté corriendo con `deadliner check-db --host localhost --port 5432 --dbname deadliner_prod`.

2. **Crear una Tarea con Dependencias Explícitas**  
   Usa `deadliner task create --title "Implementar módulo de facturación" --deadline 2025-12-31T23:59:59-03:00 --estimated-hours 40 --priority P1 --depends-on TASK_0042,TASK_0043 --resource-id juan.perez@corp.com`. El sistema generará un `task_id` UUID v4, calculará la ruta crítica usando el DPG, y devolverá `critical_path_ms` y `slack_time_ms`. Para ver la ruta: `deadliner task graph --task-id TASK_0045 --format dot | dot -Tpng > critical_path.png`.

3. **Ejecutar una Simulación What-If**  
   Lanza `deadliner simulate --task-id TASK_0045 --iterations 10000 --confidence-interval 0.95 --output json`. El proceso devuelve un JSON con `p_completion=0.87`, `p50_epoch=1735689599000`, `p95_epoch=1735693199000`, y `risk_exposure_hours=4.2`. Si `p_completion < 0.9`, el sistema sugiere automáticamente reasignar recursos con `deadliner resource rebalance --task-id TASK_0045 --max-utilization 0.8`.

### Mejores Prácticas

- **Establecer un Umbral de Riesgo Automatizado**  
  Configura una regla en el archivo `deadliner_config.yaml` con `risk_threshold: 0.85` y `auto_escalation: true`. Cuando `p_completion` caiga por debajo de 0.85, el sistema debe enviar automáticamente un mensaje a Slack con formato `@channel ⚠️ Riesgo de incumplimiento en tarea {task_id} (probabilidad actual: {p_completion})`. Además, programa un webhook POST a `https://alertmanager.corp.com/api/v1/alerts` con el payload `{"labels": {"alertname": "DeadlineRisk", "task_id": "TASK_0045"}, "annotations": {"summary": "Riesgo de deadline", "runbook_url": "https://runbooks.corp.com/deadliner-risk"}`.

- **Usar el Historial de Tareas para Mejorar Estimaciones**  
  Cada 30 días, ejecuta `deadliner analyze --period "last-90-days" --metric accuracy --output csv`. Esto genera un archivo con columnas `task_id`, `estimated_hours`, `actual_hours`, `variance_hours`, `distribution_type`. Con esos datos, entrena un modelo de regresión lineal simple (pendiente `m`, intersección `b`) para ajustar automáticamente las estimaciones futuras: `adjusted_estimate = m * raw_estimate + b`. Incorpora este ajuste en el pipeline de creación de tareas usando `deadliner config set --key estimation.adjustment --value "linear_regression"`.
