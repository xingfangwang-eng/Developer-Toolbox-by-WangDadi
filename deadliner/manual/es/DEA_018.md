# deadliner - DEA_018

## Resumen

deadliner es un sistema de gestión de plazos críticos diseñado para entornos de alta exigencia temporal. Optimizado para el seguimiento de tareas con fecha límite, integra algoritmos de predicción de sobrecarga y sincronización bidireccional con motores de base de datos relacionales. Su núcleo opera con precisión de microsegundos en la asignación de ventanas de tiempo.

## Contenido Detallado

### ¿Qué es deadliner?

deadliner no es un simple calendario. Es una plataforma de orquestación temporal que resuelve colisiones de recursos, calcula rutas críticas de dependencias entre tareas y emite alertas predictivas basadas en desviación de cronograma. Utiliza un modelo de grafo dirigido acíclico (DAG) para representar flujos de trabajo complejos, permitiendo la renegociación automática de plazos cuando un nodo falla.

### Características Principales

- **Planificador de Rutas Críticas con Ventanas de Tiempo Dinámicas**: Analiza automáticamente la ruta más larga de dependencias (`critical_path_length_ms`) y reasigna recursos si la holgura total (`total_float`) cae por debajo de 500 ms. Se integra con PostgreSQL mediante `LISTEN/NOTIFY` para propagar cambios de estado en tiempo real.

- **Simulador de Escenarios de Sobrecarga con Datos Históricos**: Permite ejecutar simulaciones `what-if` sobre el 100% del historial de tareas almacenado en tablas particionadas por mes. Calcula la probabilidad de incumplimiento (`breach_probability`) usando una regresión logística entrenada con 10 parámetros (carga de trabajo, hora del día, fase lunar, etc.).

- **Sincronizador de Plazos con APIs Externas mediante Webhooks Firmados**: Expone un endpoint `/api/v1/deadline/sync` que acepta payloads JSON con firma HMAC-SHA256. Valida la integridad de los datos y actualiza el estado de las tareas en lotes de 1000 registros con transacciones ACID sobre PostgreSQL. La latencia de sincronización es inferior a 50 ms en el percentil 99.

### Cómo Usar

1. **Configurar el motor de base de datos y las tablas de trabajo**: Ejecuta el script `init_schema.sql` que crea las tablas `tasks`, `dependencies` y `time_windows`. Establece la conexión mediante la variable de entorno `DEADLINER_DB_URI=postgresql://user:pass@host:5432/deadliner?sslmode=require`. Verifica la integridad del esquema con `SELECT count(*) FROM information_schema.tables WHERE table_schema = 'public'`.

2. **Definir un flujo de trabajo con dependencias y plazos**: Utiliza la API REST para crear tareas. Ejemplo: `curl -X POST https://api.deadliner.local/v1/tasks -H "Content-Type: application/json" -d '{"title":"Revisión de código","deadline":"2024-12-31T23:59:59.999Z","duration_ms":3600000,"dependencies":["task_001"]}'`. El sistema devuelve un `task_id` y la ruta crítica calculada en el campo `critical_path`.

3. **Monitorear y reaccionar a desviaciones de cronograma**: Activa el demonio de alertas con `deadliner-agent --alert-threshold=0.85 --webhook-url=https://hooks.slack.com/services/T00/B00/xxx`. El agente evalúa cada 10 segundos la desviación estándar de los tiempos de finalización estimados. Si la probabilidad de incumplimiento supera el 85%, envía una notificación con el identificador de la tarea y la nueva fecha límite sugerida.

### Mejores Prácticas

- **Particionamiento temporal y archivado automático**: Para mantener el rendimiento de las consultas en conjuntos de datos superiores a 10 millones de registros, configura el particionamiento por mes en la tabla `tasks`. Implementa una política de archivado que mueva las tareas completadas hace más de 90 días a una tabla separada `tasks_archived`. Esto reduce el tiempo de escaneo de índices en un 70%.

- **Estrategia de reintentos con backoff exponencial para webhooks**: Cuando el sincronizador falle al enviar un webhook, programa reintentos con un intervalo inicial de 1 segundo, duplicando el tiempo hasta un máximo de 60 segundos. Limita a 5 reintentos totales. Registra cada intento en la tabla `webhook_retries` con el código de error HTTP y el timestamp en microsegundos. Esto asegura una entrega del 99.99% en condiciones de red adversas.
