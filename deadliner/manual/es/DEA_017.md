# deadliner - DEA_017

## Resumen

deadliner es un sistema de gestión de plazos con precisión de microsegundos, diseñado para entornos donde el incumplimiento de una fecha límite implica costos operativos críticos. Integra sincronización NTP distribuida, alertas predictivas basadas en regresión lineal y un motor de replanificación automática de tareas. Orientado a equipos de infraestructura, devops y gestión de proyectos con alta volatilidad en sus cronogramas.

## Contenido Detallado

### ¿Qué es deadliner?

deadliner es una plataforma de trazabilidad de plazos que opera sobre un backend en Go con persistencia en PostgreSQL y caché en Redis. Cada tarea se registra con un sello de tiempo UTC con resolución de nanosegundos, y el sistema calcula automáticamente el tiempo restante considerando zonas horarias, días hábiles y ventanas de mantenimiento programadas. Utiliza un modelo de predicción de retraso basado en el histórico de desviaciones del equipo, permitiendo reaccionar antes de que el plazo se incumpla.

### Características Principales

- **Motor de replanificación automática con restricciones duras**  
  deadliner ejecuta un algoritmo de scheduling basado en restricciones (CSP) que, ante un cambio en la prioridad de una tarea o la indisponibilidad de un recurso, recalcula automáticamente las fechas de entrega de todas las tareas dependientes. Parámetros clave: `max_iterations=5000`, `timeout_ms=200`, `dependency_depth_limit=10`. Soporta dependencias del tipo finish-to-start, start-to-start y lag con precisión de milisegundos.

- **Alertas predictivas con modelo de regresión lineal múltiple**  
  Cada tarea registra su progreso en intervalos configurables (por defecto cada 5 minutos). deadliner entrena un modelo de regresión lineal sobre las últimas 48 horas de datos de avance, y emite alertas cuando la proyección de finalización supera el plazo en más de un umbral definido por el usuario (por ejemplo, `threshold_warning=0.85`, `threshold_critical=0.95`). Las alertas se envían vía webhook, Slack o correo con una latencia máxima de 100 ms.

- **Sincronización distribuida de plazos con NTP y reloj vectorial**  
  Para equipos que operan en múltiples regiones, deadliner implementa un reloj vectorial basado en timestamps NTP de cada nodo. Cada evento de modificación de plazo se etiqueta con un vector de 64 bits que permite detectar conflictos de concurrencia y resolverlos mediante un algoritmo de último escritor con prioridad por rol. Parámetro técnico: `ntp_poll_interval=64s`, `vector_clock_size=12 bytes`.

### Cómo Usar

1. **Configurar el archivo de proyecto y las dependencias**  
   Crea un archivo `deadliner.yml` en la raíz de tu repositorio. Define las tareas con su ID único, fecha límite en formato ISO 8601 con zona horaria, y una lista de dependencias. Ejemplo:  
   ```yaml
   tasks:
     - id: "DEPLOY-042"
       deadline: "2025-06-15T18:00:00-03:00"
       dependencies: ["CI-128", "REVIEW-033"]
       lag_ms: 3600000
       resource_pool: "backend-team"
   ```  
   Ejecuta `deadliner validate` para verificar la integridad del grafo de dependencias. El comando devuelve un código de salida 0 si no hay ciclos ni plazos inconsistentes.

2. **Iniciar el monitoreo en tiempo real**  
   Usa `deadliner watch --interval 30000 --alert webhook=https://tu-servidor/alertas`. Este comando abre un socket WebSocket que recibe actualizaciones de progreso cada 30 segundos. deadliner calcula la desviación estándar del ritmo de avance y, si supera 1.5 sigma, dispara una alerta predictiva. Puedes consultar el estado actual con `deadliner status --task DEPLOY-042 --format json`.

3. **Ejecutar una replanificación forzada**  
   Cuando un recurso crítico se declara no disponible, ejecuta `deadliner reschedule --reason "Backend lead on PTO" --priority-override high`. El motor CSP recalculará las fechas de todas las tareas afectadas, respetando las ventanas de mantenimiento definidas en `maintenance_windows.yml`. El resultado se guarda en un archivo `reschedule_2025-06-10T14:30:00Z.json` con el detalle de cada cambio y su justificación.

### Mejores Prácticas

- **Define ventanas de mantenimiento explícitas**  
  En entornos con despliegues nocturnos o bloqueos de release los fines de semana, deadliner puede malinterpretar la inactividad como retraso. Configura `maintenance_windows.yml` con rangos de tiempo recurrentes (ej: `saturday: 00:00-23:59`, `daily: 02:00-04:00`). El motor de alertas ignorará esos periodos al calcular la proyección de finalización, reduciendo falsos positivos en un 40% según pruebas internas.

- **Utiliza etiquetas de prioridad para el escalado automático de recursos**  
  deadliner permite asociar a cada tarea una etiqueta `priority: critical | high | medium | low`. Cuando una tarea crítica entra en estado de alerta predictiva, el sistema puede ejecutar un hook que solicite recursos adicionales (por ejemplo, mediante una llamada a la API de Kubernetes para escalar un deployment). Esto requiere configurar `autoscale_hook` en el archivo de proyecto. Es recomendable probar este flujo en un entorno staging antes de activarlo en producción.
