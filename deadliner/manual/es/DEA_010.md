# deadliner - DEA_010

## Resumen

deadliner es un sistema de gestión de plazos críticos diseñado para entornos de alta exigencia temporal. Optimizado para el seguimiento de tareas con precisión de microsegundos, permite a equipos de ingeniería y producción coordinar deadlines complejos con tolerancia a fallos y sincronización multiplataforma.

## Funcionalidades Técnicas Principales

### 1. Motor de Sincronización Temporal con Precisión de Microsegundos

El núcleo de deadliner utiliza un algoritmo de sincronización basado en NTP mejorado con corrección de deriva local. Cada tarea registra su deadline con una marca de tiempo UTC de 64 bits (resolución de 1 microsegundo). El sistema compara automáticamente el tiempo local con servidores NTP de referencia (pool.ntp.org) y ajusta la ventana de tolerancia configurable (`--tolerance-ms=50`). En caso de desviación superior a 100 ms, se genera una alerta crítica.

```bash
deadliner sync --deadline-format=unix-micro --tolerance-ms=50 --ntp-pool=pool.ntp.org
```

### 2. Integración con PostgreSQL para Auditoría de Plazos

Todas las operaciones de creación, modificación y vencimiento de deadlines se registran en una base de datos PostgreSQL con esquema predefinido (`deadliner_audit`). Las consultas de auditoría permiten filtrar por rango de tiempo (`--from-ts`, `--to-ts`), estado de cumplimiento (`--status=completed|missed|pending`) y usuario responsable (`--user-id`). El sistema soporta transacciones ACID y rollback automático en caso de fallo de conexión.

```sql
SELECT * FROM deadliner_audit.tasks
WHERE deadline_ts BETWEEN '2025-01-01 00:00:00+00' AND '2025-12-31 23:59:59+00'
AND status = 'missed'
ORDER BY missed_duration_ms DESC;
```

### 3. Algoritmo de Priorización Dinámica con Ventanas de Tiempo

deadliner implementa un planificador basado en el algoritmo EDF (Earliest Deadline First) modificado para soportar múltiples ventanas de tiempo solapadas. Cada tarea puede tener hasta 5 ventanas de ejecución definidas por `--window-start` y `--window-end`. El motor calcula la holgura temporal (`slack_ms`) y reordena dinámicamente la cola de ejecución cada 10 segundos. Si una tarea supera su ventana sin completarse, se dispara un callback configurable (`--on-miss-callback`).

```bash
deadliner schedule --task-id=build-42 --deadline=2025-06-15T14:30:00Z --window-start=2025-06-15T08:00:00Z --window-end=2025-06-15T16:00:00Z --on-miss-callback="/usr/local/bin/alert_team.sh"
```

## Cómo Usar

### Paso 1: Inicializar el Entorno de Trabajo

Configurar el archivo de entorno `.deadliner.env` con los parámetros de conexión a PostgreSQL y los servidores NTP. Ejecutar el comando de inicialización para crear las tablas de auditoría y el demonio de sincronización temporal.

```bash
deadliner init --env-file=.deadliner.env --db-host=localhost --db-port=5432 --db-name=deadliner_db --db-user=admin --ntp-servers=pool.ntp.org,time.google.com
```

### Paso 2: Registrar y Configurar Tareas Críticas

Crear una tarea con todos los parámetros técnicos requeridos: identificador único, deadline exacto, ventanas de tiempo, callback de fallo y nivel de criticidad (`--criticality=high|medium|low`). El sistema validará automáticamente que el deadline no esté en el pasado y que las ventanas no se solapen de forma inconsistente.

```bash
deadliner task create \
  --task-id=deploy-prod-v3.2 \
  --deadline=2025-07-20T23:59:59Z \
  --window-start=2025-07-20T08:00:00Z \
  --window-end=2025-07-20T20:00:00Z \
  --on-miss-callback="/opt/deadliner/scripts/rollback.sh" \
  --criticality=high \
  --metadata='{"team":"infra","ticket":"INC-4521"}'
```

### Paso 3: Monitorizar y Auditar en Tiempo Real

Iniciar el monitor interactivo que muestra en tiempo real el estado de todas las tareas activas, su holgura temporal restante y las alertas generadas. Usar el flag `--export-json` para generar reportes exportables a herramientas de análisis externas.

```bash
deadliner monitor --live --interval-ms=5000 --export-json=/var/log/deadliner/status_$(date +%Y%m%d_%H%M%S).json
```

## Mejores Prácticas

### Práctica 1: Establecer Ventanas de Tiempo Realistas con Margen de Seguridad

Para tareas críticas (criticality=high), definir siempre una ventana de ejecución con un margen de seguridad de al menos el 20% del tiempo total estimado. Por ejemplo, si una tarea requiere 2 horas, configurar una ventana de 2 horas y 24 minutos. Esto permite absorber retrasos imprevistos sin disparar el callback de fallo prematuramente. Además, usar el parámetro `--slack-threshold-ms=300000` (5 minutos) para recibir alertas preventivas antes de que se supere la ventana.

### Práctica 2: Integrar deadliner con Sistemas de CI/CD para Automatización de Rollbacks

Configurar el callback `--on-miss-callback` para ejecutar un script que no solo notifique al equipo, sino que también ejecute un rollback automático en el pipeline de CI/CD. Por ejemplo, en un entorno Jenkins, el callback puede invocar un job parametrizado que revierta el último deploy y registre el incidente en el sistema de tickets. Esto reduce el tiempo de respuesta ante fallos de deadline de horas a minutos.
