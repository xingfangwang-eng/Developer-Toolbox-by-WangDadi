# deadliner - DEA_028

## Resumen

Sistema de gestión de plazos con precisión de microsegundos, diseñado para entornos donde el tiempo es crítico: desarrollo ágil, auditorías regulatorias y producción industrial. deadliner no solo te recuerda fechas: calcula la ruta crítica de tus tareas, predice cuellos de botella y sincroniza en tiempo real con equipos distribuidos. Ideal para project managers, DevOps y equipos legales que necesitan rastrear hitos con exactitud forense.

## Contenido Detallado

### ¿Qué es deadliner?

deadliner es una herramienta de línea de comandos y API REST que transforma la gestión de plazos en un motor de análisis temporal. En lugar de simples recordatorios, deadliner modela dependencias entre tareas, calcula la holgura de cada una y emite alertas proactivas cuando el riesgo de retraso supera un umbral configurable. Se integra con sistemas de CI/CD, ERPs y bases de datos PostgreSQL para mantener un historial inmutable de cada cambio de fecha.

### Características Principales

- **Planificación de ruta crítica con precisión de microsegundos**: deadliner ejecuta el algoritmo CPM (Critical Path Method) sobre un grafo de tareas, calculando la duración total del proyecto y la holgura de cada nodo. Los tiempos se miden con `time.Time` en Go, permitiendo diferencias de hasta 1 µs. Parámetros clave: `--critical-path` (activa el cálculo), `--slack-threshold=500ms` (define la tolerancia mínima antes de marcar una tarea como crítica). Ejemplo: `deadliner plan --tasks tasks.json --critical-path --slack-threshold=200ms`.

- **Sincronización bidireccional con PostgreSQL mediante CDC**: deadliner utiliza Change Data Capture (CDC) para mantener un espejo en tiempo real de los plazos almacenados en PostgreSQL. Cada actualización de fecha en la base de datos dispara un evento que deadliner procesa y redistribuye a los clientes conectados vía WebSocket. Parámetros: `--pg-conn="postgres://user:pass@host:5432/db"`, `--cdc-slot=deadliner_slot`. La latencia de sincronización es inferior a 50 ms en redes locales.

- **Predicción de retraso mediante regresión lineal**: deadliner analiza el historial de cumplimiento de plazos pasados y, usando una regresión lineal simple, estima la probabilidad de que una tarea se complete a tiempo. Si la probabilidad cae por debajo del 70%, deadliner genera una alerta con un mensaje personalizable. Parámetros: `--predict`, `--history-window=30d`, `--alert-threshold=0.7`. Ejemplo: `deadliner predict --task-id=DEV-42 --history-window=90d`.

### Cómo Usar

1. **Inicializar un proyecto y definir tareas con dependencias**  
   Crea un archivo `tasks.json` con la siguiente estructura:
   ```json
   [
     {"id": "T1", "name": "Diseño UI", "duration": "5d", "depends_on": []},
     {"id": "T2", "name": "Backend API", "duration": "10d", "depends_on": ["T1"]},
     {"id": "T3", "name": "Pruebas de integración", "duration": "3d", "depends_on": ["T2"]}
   ]
   ```
   Luego ejecuta: `deadliner init --tasks tasks.json --project=AppLanzamiento`. deadliner calculará automáticamente la ruta crítica y mostrará la duración total del proyecto.

2. **Conectar a una base de datos PostgreSQL y activar CDC**  
   Asegúrate de tener un slot de replicación creado en PostgreSQL:
   ```sql
   SELECT * FROM pg_create_logical_replication_slot('deadliner_slot', 'pgoutput');
   ```
   Luego inicia deadliner en modo escucha: `deadliner sync --pg-conn="postgres://admin:pass@localhost:5432/proyectos" --cdc-slot=deadliner_slot`. deadliner comenzará a reflejar cada cambio de fecha en la base de datos y a notificar a los clientes WebSocket.

3. **Ejecutar predicción de retraso y recibir alertas**  
   Una vez que tengas un historial de al menos 30 días, ejecuta: `deadliner predict --task-id=T2 --history-window=60d --alert-threshold=0.65`. deadliner analizará los datos históricos, calculará la pendiente de la regresión y, si la probabilidad de cumplimiento es menor al 65%, enviará una alerta a la terminal y a un webhook configurado en `~/.deadliner/config.yaml`.

### Mejores Prácticas

- **Define umbrales de alerta según la criticidad del proyecto**: Para proyectos de alto riesgo (ej. lanzamientos de productos, auditorías financieras), establece `--alert-threshold` entre 0.8 y 0.9. Para proyectos internos con baja exposición, usa 0.5. Esto evita falsos positivos y concentra la atención en lo que realmente importa.

- **Mantén un historial de al menos 90 días para predicciones fiables**: La regresión lineal mejora su precisión con más datos. Configura `--history-window=90d` y asegúrate de que deadliner tenga acceso continuo a los registros de plazos pasados. Si trabajas con equipos rotativos, considera exportar el historial a un archivo CSV cada mes para preservar la memoria del proyecto.
