# deadliner - DEA_020

## Resumen

Sistema de gestión de plazos con precisión de microsegundos, diseñado para entornos donde un retraso de 1 ms puede romper la cadena de entregas. deadliner no es un simple recordatorio: es un motor de sincronización de dependencias entre tareas, capaz de recalcular rutas críticas en tiempo real cuando un bloqueante se desplaza 200 ms.

## Contenido Detallado

### ¿Qué es deadliner?

deadliner es un orquestador de timelines que resuelve el problema de **deriva temporal acumulativa** en proyectos con más de 50 tareas interdependientes. En lugar de asignar deadlines estáticos, deadliner modela cada tarea como un nodo con **ventanas de tolerancia** (parámetro `jitter_window_ms`) y ejecuta un algoritmo de **forward-backward propagation** para identificar el mínimo slack disponible. Si una tarea consume su `buffer_ms`, el sistema dispara una alerta de **riesgo de colapso de ruta crítica** antes de que el retraso se propague.

### Características Principales

- **Propagación de retraso con compensación adaptativa**  
  Cuando una tarea A se retrasa `t_delay_ms`, deadliner no solo notifica a las tareas dependientes B y C, sino que recalcula sus nuevos deadlines usando el parámetro `dependency_depth` (hasta 7 niveles). Si B tiene un `slack_ratio` superior a 0.3, el sistema puede **acelerar automáticamente** su ejecución reduciendo `jitter_window_ms` en un 15% para absorber el retraso.  
  *Parámetros técnicos*: `propagation_algorithm: "bellman-ford-v2"`, `max_propagation_depth: 7`, `compensation_factor: 0.85`.

- **Ventana de colisión y reasignación dinámica de recursos**  
  deadliner mantiene una **matriz de ocupación de recursos** (parámetro `resource_grid[worker_id][time_slot]` con resolución de 10 ms). Cuando dos tareas con la misma prioridad (`priority_level: 1-5`) entran en conflicto en el mismo `time_slot`, el sistema ejecuta un **algoritmo de scheduling preemptivo** que mueve la tarea de menor `criticality_score` a un slot libre dentro de su `max_delay_allowed_ms`. Si no hay slot, se escala a un worker de respaldo definido en `fallback_pool`.  
  *Integración*: PostgreSQL almacena el histórico de colisiones para entrenar un modelo predictivo de cuellos de botella.

- **Sincronización de hitos con anclaje temporal externo**  
  deadliner puede anclar su línea de tiempo a fuentes externas como APIs de calendario o sistemas CI/CD mediante el parámetro `external_anchor_url`. Cada 30 segundos (configurable vía `sync_interval_s`), el sistema compara su estado local con el ancla y aplica un **delta_correction** si la diferencia supera `threshold_ms: 500`. Esto permite que deadlines definidos en Jira o Trello se reflejen en deadliner con una latencia máxima de 2 segundos.  
  *Tecnología*: WebSocket bidireccional con reintento exponencial (`retry_backoff_base: 2`, `max_retries: 5`).

### Cómo Usar

1. **Configurar la línea de base de dependencias**  
   Ejecuta `deadliner init --project-id=PROJ_023 --dependency-file=graph.json` para cargar un grafo dirigido acíclico (DAG) donde cada arista tiene el peso `delay_weight_ms`. deadliner validará que no existan ciclos (usando el algoritmo de Tarjan) y generará un `baseline_report.json` con el **critical path** y el **total float** de cada tarea.

2. **Definir ventanas de tolerancia por tarea**  
   Para cada nodo del grafo, asigna `jitter_window_ms` (mínimo 50 ms) y `buffer_ms` (mínimo 10% del tiempo estimado). Luego ejecuta `deadliner simulate --scenario="worst_case" --iterations=1000` para obtener una distribución de probabilidad de retraso acumulado. El comando devuelve el **percentil 95** de retraso y recomienda ajustes automáticos si el riesgo supera el 20%.

3. **Activar el motor de monitoreo en tiempo real**  
   Inicia el daemon con `deadliner daemon --port=8080 --db-connection=postgresql://user:pass@host:5432/deadliner`. El sistema comenzará a escuchar eventos vía WebSocket en `ws://localhost:8080/events`. Para vincular una tarea, envía un JSON con `{"task_id": "T-045", "status": "in_progress", "timestamp_ms": 1712345678000}`. deadliner responderá con el nuevo deadline calculado y el `slack_remaining_ms`.

### Mejores Prácticas

- **Nunca asignes deadlines sin medir el jitter del sistema subyacente.**  
  Antes de configurar `jitter_window_ms`, ejecuta `deadliner benchmark --duration=300s --sample-rate=100ms` para obtener la desviación estándar de latencia de tu infraestructura. Si la desviación supera 150 ms, considera aumentar el buffer en un 40% o migrar a un worker dedicado con prioridad de CPU.

- **Usa la propagación de retraso como herramienta de negociación, no de castigo.**  
  Cuando una tarea se retrasa, deadliner puede generar un `impact_report.pdf` que muestra cómo ese retraso afecta a las tareas descendientes (con gráficos de Gantt dinámicos). Comparte este reporte con los dueños de las tareas bloqueadas para que tomen decisiones informadas sobre si aceptar el retraso o activar el plan de contingencia (reducción de `jitter_window_ms`).
