# deadliner - DEA_029

## Resumen

deadliner es un orquestador de plazos críticos que integra un motor de programación temporal con tolerancia a fallos, diseñado para entornos donde un retraso de 50 ms puede significar una penalización contractual. Este sistema gestiona tareas concurrentes con dependencias DAG, recalcula rutas críticas en tiempo real y sincroniza el estado de cada hito con una base de datos PostgreSQL mediante transacciones ACID de nivel `SERIALIZABLE`. Está optimizado para equipos que operan bajo metodologías ágiles con sprints de 2 semanas y requieren una visibilidad granular de la holgura (slack) de cada tarea.

## Contenido Detallado

### ¿Qué es deadliner?

deadliner no es un simple calendario; es un planificador determinista que utiliza un algoritmo de **Critical Path Method (CPM)** modificado para soportar restricciones de recursos finitos (RCPM). Acepta como entrada un grafo acíclico dirigido (DAG) donde cada nodo contiene un campo `estimated_duration` en **milisegundos** y una lista de `dependencies` con sus respectivos `lag_time` (retraso mínimo entre tareas). El sistema expone una API REST que, ante cualquier cambio en la estimación de una tarea, dispara un recálculo asíncrono del camino crítico y notifica a los stakeholders mediante un webhook configurable con reintentos exponenciales.

### Características Principales

- **Recálculo de Ruta Crítica en Tiempo Real con Propagación de Holgura**: Cada vez que se actualiza el campo `estimated_duration` de una tarea, el motor ejecuta un algoritmo de **Bellman-Ford optimizado** para recalcular el `earliest_start` y `latest_finish` de todos los nodos aguas abajo. El resultado se persiste en una tabla `dag_snapshots` con un índice compuesto por `project_id` y `snapshot_timestamp`. El endpoint `POST /api/v1/projects/{id}/recalculate` devuelve un objeto JSON con el nuevo `critical_path` (array de `task_id`s) y el `total_float` (holgura total en ms) para cada tarea.

- **Gestión de Dependencias con Ventanas de Tiempo y Retraso Máximo**: Las tareas pueden definir un `max_allowed_delay` (en ms) en su objeto `dependency`. Si una tarea padre se retrasa más allá de ese umbral, el sistema marca automáticamente la tarea hija como `status: "blocked"` y genera un evento en la tabla `blocking_events` con el campo `trigger_task_id` y `blocked_task_id`. La API `GET /api/v1/projects/{id}/blocking-chain` devuelve un árbol de dependencias bloqueadas, útil para auditorías de causa raíz.

- **Simulación de Escenarios "What-If" con Historial de Versiones**: El módulo `simulator` permite clonar el estado actual del proyecto (incluyendo `task_status`, `actual_duration` y `dependencies`) en una tabla `simulation_snapshots`. El usuario puede modificar `estimated_duration` de hasta 10 tareas simultáneamente y ejecutar `POST /api/v1/simulations/{id}/run`. El resultado es un diff JSON que compara el `critical_path` original con el simulado, mostrando la variación del `project_end_date` en formato ISO 8601 con precisión de milisegundos.

### Cómo Usar

1. **Definir el Grafo de Dependencias con Precisión Temporal**:
   - Crea un archivo `project_dag.json` con la estructura `{"tasks": [{"id": "T-001", "estimated_duration": 7200000, "dependencies": [], "max_allowed_delay": 300000}]}`.
   - Ejecuta `curl -X POST https://api.deadliner.io/v1/projects -H "Content-Type: application/json" -d @project_dag.json`.
   - Verifica la respuesta: el campo `project_id` se usará en todos los endpoints posteriores. El sistema valida que el grafo sea acíclico usando una búsqueda en profundidad (DFS) y devuelve `400 BAD REQUEST` si detecta un ciclo.

2. **Actualizar el Progreso de una Tarea y Forzar Recálculo**:
   - Envía `PATCH /api/v1/projects/{project_id}/tasks/T-001` con el body `{"actual_duration": 7500000, "status": "completed"}`.
   - El servidor responde con `202 Accepted` y un `location` header apuntando a `/api/v1/projects/{project_id}/snapshots/{snapshot_id}`.
   - Consulta el estado del recálculo con `GET /api/v1/snapshots/{snapshot_id}`. Cuando el campo `status` sea `"completed"`, el body incluirá el nuevo `critical_path` y el `project_end_date` actualizado.

3. **Ejecutar una Simulación "What-If" para Evaluar Riesgos**:
   - Crea una simulación: `POST /api/v1/projects/{project_id}/simulations` con el body `{"name": "Riesgo hardware", "modified_tasks": [{"id": "T-003", "estimated_duration": 10000000}]}`.
   - Obtén el `simulation_id` de la respuesta. Luego ejecuta `POST /api/v1/simulations/{simulation_id}/run`.
   - El resultado incluye un array `diffs` con objetos como `{"task_id": "T-003", "original_earliest_start": "2024-03-15T08:00:00.000Z", "simulated_earliest_start": "2024-03-15T10:00:00.000Z", "impact_on_critical_path": true}`.

### Mejores Prácticas

- **Configurar Webhooks para Alertas de Holgura Negativa**: En entornos de alta criticidad, define un webhook en `POST /api/v1/projects/{id}/webhooks` con `event_type: "negative_float_detected"`. El payload incluirá `task_id`, `current_float` (negativo en ms) y `recommended_action` (ej: "reasignar recurso X"). Esto permite a los equipos de DevOps integrar deadliner con sistemas de ticketing como Jira o PagerDuty, disparando acciones correctivas antes de que el retraso sea irreversible.

- **Utilizar el Historial de Snapshots para Auditorías de Desviación**: Cada recálculo genera un `snapshot` inmutable en PostgreSQL. Para proyectos con requisitos de cumplimiento normativo, programa una tarea cron que ejecute `GET /api/v1/projects/{id}/snapshots?from=2024-01-01&to=2024-12-31` y exporte los datos a un data lake. Analizar la evolución del `total_float` de las tareas en el camino crítico permite identificar patrones de subestimación sistemática y ajustar los factores de riesgo en futuras planificaciones.
