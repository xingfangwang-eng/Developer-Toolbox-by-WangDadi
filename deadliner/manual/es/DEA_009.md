# deadliner - DEA_009

## Resumen

deadliner es un sistema de gestión de plazos de alto rendimiento diseñado para entornos donde cada milisegundo cuenta. Optimizado para el seguimiento de tareas críticas con sincronización en tiempo real, deadliner permite a equipos distribuidos coordinar deadlines con precisión atómica. Este proyecto integra un motor de recalculo automático basado en husos horarios y dependencias dinámicas.

## Contenido Detallado

### ¿Qué es deadliner?

deadliner es una plataforma de orquestación de plazos que resuelve la colisión de dependencias entre tareas en proyectos multi-equipo. Su núcleo utiliza un algoritmo de *pathfinding temporal* que calcula automáticamente la ruta crítica del proyecto, ajustando los deadlines en función de la carga de trabajo real y las ventanas de disponibilidad de los recursos humanos. Está diseñado para integrarse con sistemas CI/CD y ERPs mediante webhooks firmados con JWT.

### Características Principales

- **Motor de Recalculo Predictivo (PRM)**: Analiza el historial de desviaciones de cada tarea (parámetro `drift_history_ms`) y aplica un modelo de regresión lineal para predecir la fecha de finalización real. Si la desviación supera el umbral configurable `threshold_pct` (por defecto 15%), el sistema dispara alertas proactivas y sugiere reasignaciones de recursos. La precisión del modelo se reporta en la métrica `prediction_accuracy_mae` (error absoluto medio en milisegundos).

- **Sincronización Cuántica de Husos Horarios (QTS)**: deadliner almacena todos los deadlines en formato UNIX timestamp (precisión de nanosegundos) y los convierte automáticamente al huso horario local del usuario mediante la base de datos IANA. El parámetro `sync_mode` permite elegir entre `strict` (rechaza solapamientos) y `flexible` (permite solapamientos con priorización por peso `task_weight`). La sincronización se realiza a través de una cola Redis con confirmación ACK en menos de 50 ms.

- **Simulador de Escenarios "¿Qué pasaría si...?" (WSS)**: Permite al usuario modificar virtualmente la duración estimada de una tarea (parámetro `estimated_duration_hours`) o la fecha de inicio (`start_date_unix`) y observar instantáneamente cómo se desplazan todos los deadlines dependientes. El sistema renderiza un gráfico Gantt interactivo en menos de 200 ms para 500 tareas, y exporta el resultado en formato JSON o CSV con el campo `delta_ms` que muestra la variación en milisegundos respecto al plan original.

### Cómo Usar

1. **Configuración inicial del proyecto**: Ejecuta `deadliner init --project-id DEA_009 --timezone "America/Mexico_City" --sync-mode strict`. Esto creará un archivo `deadliner.config.yml` con los parámetros `drift_history_ms` y `threshold_pct`. Luego, ejecuta `deadliner import-tasks --from jira --jql "project = DEA_009 AND status != Done"` para traer todas las tareas de Jira. El sistema mapeará automáticamente los campos `duedate` y `timeestimate` a `deadline_unix` y `estimated_duration_hours`.

2. **Definición de dependencias y pesos**: Usa el comando `deadliner link --task-id TASK-123 --depends-on TASK-045 --weight 0.8`. El parámetro `weight` (0.0 a 1.0) indica el grado de bloqueo. Con `weight=0.8`, si TASK-045 se retrasa 10 horas, TASK-123 se retrasará 8 horas. Para ver el impacto visual, ejecuta `deadliner simulate --delay TASK-045 --hours 10` y observa el delta en milisegundos en la salida JSON.

3. **Monitoreo y alertas en tiempo real**: Inicia el agente de monitoreo con `deadliner watch --alert-threshold 15000ms --webhook-url "https://hooks.slack.com/services/xxx"`. Esto crea un proceso en segundo plano que cada 60 segundos ejecuta el PRM. Si la predicción de finalización se desvía más de 15 segundos (15000 ms) del deadline original, se envía una alerta a Slack con la tarea, el nuevo deadline estimado y el delta en milisegundos. Para detener el agente, usa `deadliner unwatch`.

### Mejores Prácticas

- **Integración con CI/CD para actualización automática**: Configura un webhook en tu pipeline de GitLab CI/CD que, al finalizar un build, ejecute `deadliner update --task-id TASK-XXX --actual-duration $DURATION_SECONDS`. Esto alimenta el historial de `drift_history_ms` del PRM, mejorando la precisión de las predicciones futuras. Se recomienda usar `threshold_pct` entre 10% y 20% para evitar falsos positivos en entornos ágiles.

- **Sincronización periódica con PostgreSQL para auditoría**: Programa un cron job que ejecute `deadliner export --format sql --output /backups/deadliner_$(date +%Y%m%d).sql`. Este comando genera un script SQL con todas las tareas, deadlines y dependencias, incluyendo el campo `predicted_end_unix` del último cálculo del PRM. Luego, importa este archivo en una base de datos PostgreSQL separada para generar reportes de cumplimiento históricos. Usa el índice `CREATE INDEX idx_deadline_unix ON tasks (deadline_unix);` para optimizar consultas de rango temporal.
