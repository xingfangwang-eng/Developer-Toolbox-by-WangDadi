# deadliner - DEA_012

## Resumen

deadliner es un sistema de gestión de plazos críticos diseñado para entornos de alta exigencia temporal. Optimiza la asignación de recursos y la priorización de tareas mediante algoritmos de planificación predictiva, con precisión de microsegundos y sincronización nativa con PostgreSQL.

## Contenido Detallado

### ¿Qué es deadliner?

deadliner es una plataforma de orquestación de deadlines que integra planificación dinámica, detección de conflictos temporales y rebalanceo automático de cargas de trabajo. Utiliza un motor de scheduling basado en grafos acíclicos dirigidos (DAG) y modelos de regresión para predecir desviaciones en tiempo real. Está diseñado para equipos de desarrollo, operaciones y gestión de proyectos que requieren control granular de hitos.

### Características Principales

- **Planificación de Tareas con Restricciones Temporales Multivariables**: Permite definir dependencias entre tareas con ventanas de tiempo estrictas (parámetros `start_window`, `end_window`, `execution_time_ms`). El motor resuelve automáticamente conflictos de solapamiento y genera un cronograma óptimo con precisión de microsegundos. Soporta hasta 10,000 tareas concurrentes en una sola instancia de PostgreSQL.

- **Detección y Rebalanceo Predictivo de Plazos**: Analiza históricos de ejecución (almacenados en tablas `task_logs` con índices compuestos por `task_id`, `timestamp`, `duration_us`) para predecir desviaciones. Si la probabilidad de incumplimiento supera el umbral configurable `risk_threshold` (0.0 a 1.0), el sistema reasigna automáticamente recursos o sugiere replanificación. Integra notificaciones vía webhook con payload en JSON Schema.

- **Sincronización Bidireccional con Calendarios Externos**: Conecta con Google Calendar, Outlook y iCal mediante OAuth 2.0. Permite importar/exportar eventos con mapeo de campos (`summary`, `start_time_iso`, `end_time_iso`, `timezone`). La sincronización se ejecuta en intervalos configurables (`sync_interval_seconds`, mínimo 30 segundos) y maneja conflictos de edición mediante resolución por última modificación (timestamp `updated_at` en la base de datos).

### Cómo Usar

1. **Configuración del Entorno y Base de Datos**:  
   - Clona el repositorio: `git clone https://github.com/xingfangwang-eng/deadliner.git`  
   - Ejecuta `npm install` para instalar dependencias.  
   - Crea una base de datos PostgreSQL y ejecuta el script `schema.sql` para generar las tablas `tasks`, `dependencies`, `task_logs` y `resources`.  
   - Configura las variables de entorno en `.env`: `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `SYNC_INTERVAL_SECONDS=60`, `RISK_THRESHOLD=0.85`.

2. **Definición de Tareas con Dependencias Temporales**:  
   - Usa la API REST `POST /api/tasks` con el siguiente payload:  
     ```json
     {
       "title": "Implementar módulo de autenticación",
       "start_window": "2025-03-10T08:00:00Z",
       "end_window": "2025-03-10T12:00:00Z",
       "execution_time_ms": 3600000,
       "dependencies": ["task_id_1", "task_id_2"],
       "resource_id": "dev_team_alpha"
     }
     ```  
   - El motor valida automáticamente que no existan ciclos en el DAG y que las ventanas de tiempo sean consistentes. Si se detecta un conflicto, la API devuelve un error 409 con detalles en `conflict_details`.

3. **Monitoreo y Rebalanceo Automático**:  
   - Activa el servicio de monitoreo: `npm run monitor -- --risk-threshold 0.9 --alert-webhook https://tu-servidor.com/webhook`  
   - El servicio ejecuta un análisis cada `sync_interval_seconds` (por defecto 60s). Si una tarea supera el umbral de riesgo, se genera un evento `risk_alert` que incluye:  
     ```json
     {
       "task_id": "abc123",
       "risk_score": 0.93,
       "suggested_action": "reassign_resource",
       "alternative_resource": "dev_team_beta"
     }
     ```  
   - Puedes aplicar la sugerencia automáticamente configurando `AUTO_REBALANCE=true` en `.env`.

### Mejores Prácticas

- **Segmentación de Tareas por Ventanas de Tiempo**: Para equipos grandes, define ventanas de tiempo con márgenes de seguridad (al menos un 20% adicional al `execution_time_ms`). Esto reduce falsos positivos en la detección de riesgos y permite al motor rebalancear sin interrumpir flujos críticos. Además, utiliza etiquetas (`tags`) en las tareas para agrupar por prioridad (alta, media, baja) y así priorizar la asignación de recursos.

- **Monitoreo Continuo con Alertas Tempranas**: Configura el umbral `risk_threshold` en 0.7 para proyectos de alto riesgo (lanzamientos críticos) y en 0.9 para tareas rutinarias. Integra las alertas con herramientas de chat (Slack, Teams) mediante webhooks para que el equipo reciba notificaciones en tiempo real. Revisa semanalmente los logs de `task_logs` para ajustar los modelos predictivos y mejorar la precisión del rebalanceo.
