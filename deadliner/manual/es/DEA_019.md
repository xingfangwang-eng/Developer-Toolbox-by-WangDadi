# deadliner - DEA_019

## Resumen Técnico

deadliner es un motor de planificación temporal diseñado para entornos donde cada milisegundo cuenta. Integra calendarios distribuidos, colas de prioridad con notificaciones push en tiempo real y un sistema de auditoría de cumplimiento basado en PostgreSQL. Ideal para equipos de desarrollo ágil, gestión de proyectos legales o cualquier flujo donde un deadline perdido implique costos directos.

## Funcionalidades Técnicas Principales

### 1. Algoritmo de Propagación de Dependencias con Ventana de Tiempo

Implementa un grafo acíclico dirigido (DAG) de tareas donde cada nodo contiene un campo `estimated_duration_ms` (en milisegundos) y un `hard_deadline_timestamp`. El motor calcula automáticamente la **ruta crítica** y ajusta las fechas de inicio temprano/tardío mediante un algoritmo de relajación de bordes. Soporta restricciones del tipo `FS` (fin a inicio), `SS` (inicio a inicio) y `FF` (fin a fin) con desfases personalizados en formato ISO 8601.

**Parámetros clave:**
- `dependency_graph: List[Tuple[uuid, uuid, int, str]]` — aristas con desfase en ms y tipo de restricción.
- `tolerance_window_ms: int` — ventana de tolerancia antes de marcar una tarea como "en riesgo".
- `recalc_trigger: str` — evento que dispara el recálculo (ej: `on_task_complete`, `on_priority_change`).

### 2. Sistema de Notificaciones Predictivas con Umbrales Dinámicos

No solo notifica cuando un deadline está próximo. Utiliza un modelo de regresión lineal simple sobre el historial de `completion_rate` (tasa de finalización por hora) para predecir la probabilidad de incumplimiento con 48 horas de antelación. Las alertas se envían vía WebSocket (con reconexión automática) y se almacenan en una cola Redis con persistencia en disco.

**Parámetros clave:**
- `prediction_horizon_hours: int` — horizonte de predicción (por defecto 48).
- `confidence_threshold: float` — umbral de confianza para disparar alerta (0.0 a 1.0).
- `notification_channels: List[str]` — canales habilitados: `slack_webhook`, `email_smtp`, `pushover`.

### 3. Motor de Auditoría de Cumplimiento con Integración PostgreSQL

Cada cambio de estado en una tarea (`pending`, `in_progress`, `completed`, `overdue`) se registra en una tabla `deadliner_audit_log` con los campos: `task_id`, `previous_status`, `new_status`, `changed_by`, `changed_at` (timestamp con zona horaria). Permite generar informes de cumplimiento histórico con consultas SQL parametrizadas, incluyendo métricas como `avg_delay_per_user` o `overdue_rate_per_project`.

**Parámetros clave:**
- `audit_retention_days: int` — días de retención de logs (por defecto 365).
- `pg_connection_string: str` — cadena de conexión a PostgreSQL.
- `index_strategy: str` — estrategia de indexación: `btree` (por defecto) o `brin` para tablas de alta cardinalidad temporal.

## Cómo Usar deadliner en un Entorno de Producción

### Paso 1: Configurar el Grafo de Dependencias Inicial

Define la estructura de tareas con sus dependencias y plazos duros. Ejemplo en JSON:

```json
{
  "tasks": [
    {"id": "a1", "name": "Diseño de API", "estimated_duration_ms": 7200000, "hard_deadline": "2025-06-15T18:00:00Z"},
    {"id": "b1", "name": "Implementación", "estimated_duration_ms": 14400000, "hard_deadline": "2025-06-20T18:00:00Z"}
  ],
  "dependencies": [
    {"from": "a1", "to": "b1", "lag_ms": 0, "type": "FS"}
  ]
}
```

Ejecuta el motor de propagación:

```bash
deadliner propagate --input tasks.json --output schedule.json --tolerance 3600000
```

### Paso 2: Activar el Sistema de Notificaciones Predictivas

Inicia el servicio de predicción y alertas:

```bash
deadliner predict --horizon 48 --confidence 0.85 \
  --channels slack_webhook,email_smtp \
  --redis-uri redis://localhost:6379/0
```

El servicio se mantendrá en background escuchando cambios en la base de datos y emitiendo alertas cuando la probabilidad de incumplimiento supere el umbral.

### Paso 3: Consultar el Historial de Auditoría

Conéctate a PostgreSQL y ejecuta una consulta para obtener el tiempo promedio de retraso por usuario en el último trimestre:

```sql
SELECT 
  changed_by AS usuario,
  AVG(EXTRACT(EPOCH FROM (changed_at - (SELECT hard_deadline FROM tasks WHERE id = task_id)))) AS retraso_promedio_segundos
FROM deadliner_audit_log
WHERE new_status = 'overdue'
  AND changed_at >= NOW() - INTERVAL '90 days'
GROUP BY usuario
ORDER BY retraso_promedio_segundos DESC;
```

## Mejores Prácticas para Equipos de Ingeniería

### 1. Establecer una Ventana de Tolerancia por Proyecto, No Global

Cada proyecto tiene una criticidad distinta. Configura `tolerance_window_ms` a nivel de proyecto en lugar de usar un valor global. Por ejemplo, para proyectos de compliance legal usa 0 ms (tolerancia cero); para features internas usa 3600000 ms (1 hora). Esto evita falsos positivos en notificaciones y mantiene la atención del equipo donde realmente importa.

### 2. Indexar la Tabla de Auditoría con BRIN para Alto Volumen

Si tu equipo genera más de 100 000 eventos de auditoría al día, usar un índice `BRIN` en la columna `changed_at` reduce drásticamente el tamaño del índice y acelera las consultas de rango temporal. Ejemplo:

```sql
CREATE INDEX idx_audit_changed_at_brin ON deadliner_audit_log USING brin (changed_at) WITH (pages_per_range = 32);
```

Esto es especialmente útil para informes mensuales o trimestrales donde solo se consultan ventanas de tiempo específicas.
