# deadliner - DEA_008

## 概要

deadliner は、マルチスレッド非同期処理と PostgreSQL の LISTEN/NOTIFY 機構を活用した、エンタープライズ級のデッドライン管理エンジンです。全タスクの期限をミリ秒単位で追跡し、実行猶予が 0 になった瞬間にコールバックを発火。タイムクリティカルな業務プロセスにおけるリードタイム短縮を実現します。

## 詳細な内容

### deadliner とは

deadliner は、**分散環境におけるデッドラインの一元管理** を目的とした Go 製のデーモンです。各ワーカーノードがローカルのタイマーを持つのではなく、中央のスケジューラが全タスクの期限を管理し、期限切れと同時に Webhook または gRPC ストリーム経由で通知を送出します。これにより、ノード障害時もデッドラインの見逃しを防止します。

### コア機能

1. **動的デッドライン再計算エンジン**  
   タスクの優先度（`priority: int`、0 〜 100）と現在のシステム負荷（`load_avg: float`）に基づき、猶予時間をリアルタイムに再計算。例えば `priority` が 80 を超えるタスクは、`load_avg` が 0.8 を超えても猶予を 10% に短縮し、重要タスクの遅延を防止します。再計算間隔は `recalc_interval_ms: int`（デフォルト 500ms）で設定可能。

2. **PostgreSQL 永続化 + LISTEN/NOTIFY による即時同期**  
   全タスクの状態（`task_id: uuid`, `deadline_at: timestamptz`, `status: enum('pending','fired','cancelled')`）を PostgreSQL に保存。`deadliner.notify_channel` を通じて NOTIFY を発行し、他のインスタンスへ即座に変更を伝播。これにより、クラスタ全体で同一のデッドライン情報を保持しつつ、ポーリングによる負荷をゼロに抑えます。

3. **カスケードデッドラインチェーン**  
   タスク間に親子関係（`parent_task_id: uuid?`）を定義可能。親タスクのデッドラインが変更された場合、子タスクのデッドラインを自動的に再設定（`cascade_mode: enum('offset','absolute')`）。`offset` モードでは `offset_ms: bigint` 分だけ親からずらした時刻を設定。例えば「承認タスクの期限から 3 時間後にレビュータスクの期限」といった業務ルールをコード変更なしで実現できます。

### 使い方

1. **スケジューラの起動とタスク登録**  
   ```bash
   deadliner scheduler --db-url "postgres://user:pass@localhost:5432/deadliner" \
                      --recalc-interval-ms 500 \
                      --webhook-timeout-ms 3000
   ```
   起動後、REST API 経由でタスクを登録します。
   ```bash
   curl -X POST http://localhost:8080/api/v1/tasks \
        -H "Content-Type: application/json" \
        -d '{
          "task_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
          "deadline_at": "2025-12-31T23:59:59.999Z",
          "priority": 85,
          "parent_task_id": null,
          "cascade_mode": "absolute",
          "webhook_url": "https://your-service.com/callback"
        }'
   ```

2. **デッドライン発火と通知受信**  
   スケジューラは `deadline_at` に達したタスクを検出すると、即座に `status` を `fired` に更新し、PostgreSQL に NOTIFY を発行。同時に、登録された `webhook_url` へ POST リクエストを送信します。ペイロードには `task_id`, `fired_at`, `original_deadline` が含まれます。受信側は HTTP 200 を返すことで確認応答を行い、タイムアウト（`--webhook-timeout-ms` で設定）が発生した場合は最大 3 回リトライします。

3. **カスケードチェーンの確認とデッドライン修正**  
   親タスクのデッドラインを変更する場合：
   ```bash
   curl -X PATCH http://localhost:8080/api/v1/tasks/a1b2c3d4-e5f6-7890-abcd-ef1234567890 \
        -H "Content-Type: application/json" \
        -d '{"deadline_at": "2026-01-01T00:00:00.000Z"}'
   ```
   これにより、子タスクのデッドラインが `cascade_mode` に従って自動再計算されます。変更履歴は `task_history` テーブルに記録され、監査ログとして利用可能です。

### ベストプラクティス

- **デッドライン猶予の「バッファゾーン」を設ける**  
  全タスクに一律の猶予（`grace_period_ms: bigint`、例：5000ms）を設定するのではなく、タスクの重要度に応じて動的に変化させることを推奨します。deadliner の優先度ベース再計算エンジンを活用し、`priority >= 90` のタスクには `grace_period_ms = 0`、`priority < 30` のタスクには `grace_period_ms = 30000` とすることで、リソースを効率的に配分できます。

- **PostgreSQL の max_replication_slots を適切に設定する**  
  deadliner は LISTEN/NOTIFY に依存するため、PostgreSQL の `max_replication_slots` が小さいと接続が拒否される可能性があります。クラスタサイズが 10 ノードを超える場合は、`max_replication_slots` を `20` 以上に設定し、さらに `deadliner.notify_channel` ごとにスロットが消費されることを考慮して余裕を持たせてください。
