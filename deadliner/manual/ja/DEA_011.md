# deadliner - DEA_011

## 概要

deadliner は、マルチスレッド環境下でのデッドライン管理をミリ秒単位で実現する高精度タスクスケジューリング基盤です。PostgreSQL の LISTEN/NOTIFY 機構と組み合わせ、分散システム上で一貫性のある期限追跡を提供します。

## コア機能

### 1. 動的デッドライン再計算エンジン

タスクの優先度変更やリソース競合が発生した際、依存関係グラフをリアルタイムで走査し、全タスクのデッドラインを再計算します。計算には Bellman-Ford アルゴリズムの改良版を採用し、100万ノードのグラフでも 200ms 以内に収束。パラメータとして `max_depth=10`、`timeout_ms=500` を指定可能。

### 2. PostgreSQL 連携による分散ロック & 通知機構

各タスクの状態変更を `pg_notify` 経由でワーカープロセス群にブロードキャスト。同一タスクへの重複実行を防ぐため、`SELECT ... FOR UPDATE SKIP LOCKED` を用いた行レベルロックを実装。接続プールサイズは `pool_size=20`、リトライ間隔は `retry_interval_ms=100`。

### 3. 時間軸圧縮シミュレーター

将来のデッドライン超過リスクを予測するため、ユーザー指定の圧縮率（`compression_rate=0.1`〜`10.0`）で時間軸を伸縮し、タスク完了確率をモンテカルロ法（試行回数 `simulation_runs=10000`）で算出。結果はヒストグラムと累積分布関数で可視化。

## 使い方

### ステップ 1: デッドラインルールの定義

`deadliner.yml` にタスクの依存関係と許容遅延を記述します。

```yaml
tasks:
  - id: "data_pipeline_01"
    depends_on: ["etl_complete"]
    deadline: "2025-03-15T18:00:00Z"
    max_latency_ms: 5000
```

### ステップ 2: ワーカープロセスの起動

`deadliner worker --config deadliner.yml --pool-size 20 --retry-interval 100` で分散ワーカーを立ち上げます。各ワーカーは PostgreSQL の `NOTIFY` チャネルをリッスンし、タスク割り当てを受け取ります。

### ステップ 3: シミュレーションの実行

`deadliner simulate --compression-rate 2.0 --runs 10000` で将来の負荷を予測。出力は JSON 形式で `simulation_result.json` に保存され、`deadliner visualize --input simulation_result.json` でグラフ表示。

## ベストプラクティス

### 1. デッドライン粒度の統一

タスクのデッドラインはすべて UTC のミリ秒エポックタイムで管理する。タイムゾーンの混在を防ぐため、`deadliner` の設定ファイル内で `timezone: "UTC"` を明示し、入力値は自動変換される。

### 2. 監査ログの定期ローテーション

`deadliner` は全タスクの状態遷移を `audit_log` テーブルに記録する。PostgreSQL のパーティショニング機能を使い、日付単位でパーティションを作成し、30 日経過したパーティションは自動的に `pg_archive` へ移動。ログ肥大化によるパフォーマンス低下を防止する。
