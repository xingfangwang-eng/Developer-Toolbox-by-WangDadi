# deadliner - DEA_029

## 概要

deadliner は、複数プロジェクトにまたがる納期をミリ秒単位で管理し、タスクのクリティカルパスを動的に再計算する次世代型デッドライン管理エンジンです。PostgreSQL の非同期トリガーと連携し、依存関係の変更をリアルタイムで検知してスケジュールを自動調整します。

## 詳細な内容

### deadliner とは

deadliner は、納期管理における「計画と実績の乖離」をゼロにすることを目的としたアーキテクチャです。タスクの開始条件、所要時間、リソース制約をパラメータ化し、ガントチャート上のクリティカルパスを動的に再計算します。特に、マルチエージェント環境での分散タスク調整を想定し、各エージェントの処理能力（CPU コア数、メモリ使用量）を考慮したスケジューリングが可能です。

### コア機能

- **動的クリティカルパス再計算エンジン**  
  タスクの依存関係グラフを DAG（有向非巡回グラフ）で保持し、任意のタスクの開始遅延や所要時間変更が発生した場合、`O(E log V)` の計算量で全タスクの納期への影響をミリ秒単位で再計算します。パラメータとして `max_parallel_tasks`（最大並列タスク数）と `resource_pool_size`（リソースプールサイズ）を指定可能。

- **PostgreSQL 非同期トリガー連携によるリアルタイム同期**  
  PostgreSQL の `LISTEN/NOTIFY` 機構を利用し、タスクテーブルに変更が生じると即座に deadliner がイベントを捕捉。`pg_notify` チャンネル経由で受信した変更をバッファリングし、`sync_interval_ms`（デフォルト 100ms）で一括処理します。これにより、データベースの整合性を保ちながら、更新頻度を抑制します。

- **リソース競合検出と自動スケジュール調整**  
  各タスクに `required_cpu_cores`（必要 CPU コア数）と `required_memory_mb`（必要メモリ容量）を設定可能。全タスクのリソース使用量を時系列でプロファイリングし、競合が発生した場合は優先度（`priority` パラメータ、1〜10）に基づいてタスクの開始時刻を自動的に後ろ倒しします。競合解消アルゴリズムには、重み付き最大マッチングを採用。

### 使い方

1. **初期設定とデータベース接続**  
   `deadliner init --db-host localhost --db-port 5432 --db-name deadline_db --sync-interval-ms 100` を実行し、PostgreSQL との接続を確立します。このコマンドは内部で `CREATE TABLE IF NOT EXISTS tasks (...)` を発行し、必要なスキーマを自動生成します。また、`LISTEN deadline_channel` を開始し、変更通知の待受を開始します。

2. **タスクの登録と依存関係の定義**  
   `deadliner add-task --id T001 --description "API 設計書作成" --duration-ms 3600000 --required-cpu-cores 2 --required-memory-mb 512 --priority 5` でタスクを追加。依存関係は `deadliner add-dependency --task-id T001 --depends-on T000` で設定します。この際、循環依存を自動検出し、`cyclic_dependency_error` を返します。

3. **スケジュールの実行とモニタリング**  
   `deadliner schedule --start-time "2025-03-01T09:00:00Z" --max-parallel-tasks 4 --resource-pool-cpu-cores 8 --resource-pool-memory-mb 4096` でスケジュールを計算。結果は JSON 形式で標準出力に出力され、各タスクの `start_time`、`end_time`、`slack_time_ms`（余裕時間）が確認できます。`deadliner monitor --interval-ms 5000` でリアルタイムに進捗を監視可能。

### ベストプラクティス

- **タスクの粒度は 1 時間単位を基準に**  
  `duration-ms` が 3600000（1 時間）未満のタスクは、オーバーヘッドが大きくなりスケジューリングの精度が低下します。タスクは 1 時間以上に分割し、必要に応じてサブタスクとして管理することを推奨します。これにより、クリティカルパスの再計算コストも低減されます。

- **リソース競合を避けるためのプロファイリング実施**  
  初回のスケジュール実行前に、`deadliner profile --sample-duration-ms 600000` で実際のリソース使用量をサンプリングします。このデータを基に `required_cpu_cores` と `required_memory_mb` を調整することで、競合検出の精度が向上します。特に、メモリ使用量が急増するタスク（例：大規模データ処理）は、事前に `resource_pool_size` を拡張しておくことが有効です。
