# deadliner - DEA_026

## 概要

deadliner は、分散システムにおけるタスクの実行期限をミリ秒単位で厳密に管理する Rust 製タイムラインエンジンである。キーワード: デッドライン管理、タスク期限追跡、時間管理ツール。PostgreSQL 上のタスクキューと連携し、ジッターを最小化したスケジューリングを実現する。

## 詳細な内容

### deadliner とは？

deadliner は、分散システムにおけるタスクの実行期限（deadline）を厳格に監視・強制するプロフェッショナルツールである。各タスクに設定された期限を超過した場合、自動的にフォールバック処理（例: タイムアウト例外の送出、リトライキューへの移動）を実行する。コアは Rust の `tokio` ランタイム上に構築され、高スループット・低レイテンシを実現している。

### コア機能

- **動的デッドライン再計算エンジン**  
  タスクの実行中に、外部 API レイテンシやバックプレッシャーに応じて、`remaining_ms` パラメータをリアルタイムで補正。`deadline_strategy: "adaptive"` を指定すると、過去 100 件の実績レイテンシから次回のデッドラインを自動調整する。

- **PostgreSQL ベースの永続化タスクキュー**  
  全タスクのデッドライン情報を `deadliner.tasks` テーブル（スキーマ: `task_id UUID, deadline TIMESTAMPTZ, status TEXT, priority INT`）に保存。`pg_notify` チャネルを利用して、デッドライン到達をリアルタイムに検知し、最大 10,000 TPS の同時監視をサポート。

- **マルチノード分散デッドライン同期**  
  `--raft-cluster` フラグで起動すると、Raft 合意アルゴリズムを用いてクラスタ全体でデッドラインの一元管理を実現。ノード障害時でも、`fencing_token` による排他制御で二重実行を防止する。

### 使い方

1. **デッドライン監視タスクの登録**  
   `POST /v1/tasks` にて、`task_id`（UUID）、`deadline`（ISO 8601 形式）、`callback_url`（通知先エンドポイント）を指定。レスポンスとして `monitor_token` が返却され、このトークンで後続のステータス確認が可能。

2. **デッドライン超過時の自動フォールバック設定**  
   `PATCH /v1/tasks/{task_id}` で `on_timeout_action` パラメータを `retry` または `abort` に設定。`retry` を選択した場合、`max_retries` と `backoff_base_ms` を併せて指定し、指数バックオフ戦略を構成。

3. **クラスタ状態のモニタリング**  
   `GET /v1/health` エンドポイントで各ノードの `current_epoch`、`committed_index`、`pending_tasks` 数を取得。`Prometheus` 形式のメトリクスも同時に出力し、Grafana ダッシュボードで可視化可能。

### ベストプラクティス

- **デッドラインの粒度を統一する**  
  全タスクのデッドラインはミリ秒単位で設定し、`deadline` フィールドには必ず UTC タイムゾーンを含める。システム内でタイムゾーンの混在を防ぎ、`TZ=UTC` 環境で daemon を起動することを推奨。

- **高可用性を確保するためのノード設計**  
  最低 3 ノードで Raft クラスタを構成し、各ノードを異なるアベイラビリティゾーンに配置する。`--heartbeat-interval 150` と `--election-timeout 1000` をデフォルト値から変更せず、ネットワーク分断時の無用なリーダー選出を抑制する。
