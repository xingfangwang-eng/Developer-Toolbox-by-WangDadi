# deadliner - DEA_019

## 概要

deadliner は、マルチスレッド環境下で動作する高精度タスクデッドライン管理エンジンです。PostgreSQL 15+ との連携により、タスクの生成から期限超過までの全行程をミリ秒単位で追跡・制御します。本プロジェクトは、CI/CD パイプラインやリアルタイム通知システムとの統合を前提とした、プロフェッショナル向けデッドライン管理基盤を提供します。

## 詳細な内容

### とは deadliner?

deadliner は、タスクのデッドラインを「単なる日時」ではなく、**動的なリスクスコア**として扱うアーキテクチャを採用しています。各タスクには、優先度（P0〜P4）、依存関係グラフ、およびスラックタイム（余裕時間）が紐づけられ、エンジニアリングチーム全体のワークロードをリアルタイムで可視化します。特に、**クリティカルパス分析**と**バーンダウンチャートの自動生成**により、プロジェクト遅延の兆候を早期に検出します。

### コア機能

1. **動的スラックタイム計算エンジン**  
   各タスクの依存関係をトポロジカルソートし、タスクごとに `slack_ms`（ミリ秒単位のスラックタイム）を算出。`critical_path_weight` パラメータを 0.0〜1.0 で設定可能で、値を高くするほどクリティカルパス上のタスクが強調表示されます。PostgreSQL の `ltree` 拡張を使用して階層的依存関係を管理し、変更があった場合は `NOTIFY/LISTEN` 機構で即時再計算をトリガーします。

2. **マルチエポックデッドライン比較ビュー**  
   同一プロジェクト内で、異なるスプリント（例：Sprint 1、Sprint 2）のデッドライン達成率を横断比較します。`epoch_id` パラメータで比較対象エポックを指定し、`completion_ratio`（0.0〜1.0）と `overdue_threshold_ms`（超過許容ミリ秒）を用いて、各エポックの健全性スコアを自動算出。結果はヒートマップ形式で出力され、チームのパフォーマンス傾向を一目で把握できます。

3. **リアルタイムデッドラインアラートパイプライン**  
   WebSocket 経由で、デッドライン接近タスク（`remaining_ms < warning_threshold_ms`）を即座に通知。`escalation_level`（1〜5）を設定可能で、レベルが上がるごとに通知先（Slack、PagerDuty、メール）が自動拡張されます。また、`auto_pause_on_overdue` フラグを `true` にすると、デッドライン超過タスクが自動的に一時停止状態になり、チームリーダーによる承認がない限り進行不可となります。

### 使い方

1. **環境初期化とデータベースセットアップ**  
   `deadliner init --db-host localhost --db-port 5432 --db-name deadline_db --extension ltree`  
   を実行し、PostgreSQL に必要なテーブル（`tasks`、`dependencies`、`epochs`）とトリガー関数を作成します。`--slack-default-ms 3600000` で全タスクのデフォルトスラックタイムを 1 時間に設定可能。

2. **タスク登録と依存関係の定義**  
   ```bash
   deadliner task add \
     --title "フロントエンドAPI実装" \
     --priority P1 \
     --deadline "2025-03-15T18:00:00Z" \
     --depends-on "task-uuid-abc123" \
     --depends-on "task-uuid-def456" \
     --slack-ms 1800000
   ```
   これにより、タスクが登録され、依存関係が `ltree` パスとして保存されます。`--depends-on` は複数指定可能で、循環依存は自動検出されエラーとなります。

3. **デッドライン監視開始とアラート設定**  
   `deadliner watch --project-uuid proj-789 --warning-threshold-ms 600000 --escalation-level 3 --auto-pause-on-overdue`  
   を実行すると、WebSocket サーバーが起動し、5 分以内に期限が切れるタスクがあれば Slack に通知。`--escalation-level 3` は、超過後 10 分で PagerDuty にエスカレーションする設定です。監視を停止するには `deadliner unwatch` を使用します。

### ベストプラクティス

1. **スラックタイムの動的調整をチームのペロシティに連動させる**  
   各スプリント終了時に、`deadliner epoch analyze --epoch-id sprint-12 --velocity-adjustment 0.95` を実行し、実際の完了時間とスラックタイムの乖離を分析。次スプリントの `slack_default_ms` に自動反映させることで、現実的なデッドライン設定が可能になります。これにより、過剰なバッファ（バッファスナイピング）を防止します。

2. **クリティカルパスを可視化し、リソース再配分のトリガーとする**  
   `deadliner path critical --project-uuid proj-789 --output-format dot | dot -Tpng > critical_path.png`  
   でクリティカルパスを画像出力。毎日朝会でこのグラフを確認し、クリティカルパス上のタスクが遅延している場合は、即座に `deadliner task reassign --task-uuid <uuid> --new-owner <user>` で担当者を変更するルールをチームに導入します。
