# deadliner - DEA_027

## 概要

deadliner は、タスクのデッドライン管理をミリ秒単位で精密に制御するプロフェッショナル向けタイムライン最適化エンジンです。マルチスレッド環境下での依存関係グラフ解析と、PostgreSQL との連携による永続化スケジュール管理を中核とし、複数プロジェクトのデッドライン競合を動的に解決します。

## 詳細な内容

### deadliner とは

deadliner は、タスク間のクリティカルパスを自動抽出し、各タスクの開始時刻・終了時刻をリアルタイムに再計算するタイムライン最適化ツールです。特に、複数のデッドラインが衝突する複雑なプロジェクトにおいて、リソース競合を回避しながら最適なスケジュールを自動生成します。

### コア機能

- **依存関係グラフによるクリティカルパス自動抽出**  
  タスク間の先行関係（`predecessorId`、`lagTime`）を有向非巡回グラフ（DAG）として解析し、クリティカルパスをミリ秒単位で算出。`--critical-path-only` フラグを指定すると、全体のスケジュールに影響を与える必須タスクのみを抽出して表示します。

- **デッドライン競合の動的解決（Conflict Resolver）**  
  複数のタスクが同一リソース（`resourcePool`）を要求する場合、優先度（`priority` パラメータ、1〜100）と残り余裕時間（`slackTime`）に基づいて最適な実行順序を自動決定。競合が検出された場合は `conflict-resolution-strategy`（`preemptive` または `fair-share`）に従ってスケジュールを再調整します。

- **PostgreSQL 連携によるスケジュール永続化と履歴追跡**  
  タスクの作成・更新・削除のたびに、`schedule_snapshots` テーブルにタイムスタンプ付きでスナップショットを保存。`--revision-history` オプションで過去の任意時点のスケジュールを復元可能。`pg_stat_statements` を活用したクエリパフォーマンス監視も標準対応。

### 使い方

1. **依存関係を定義したタスクリストの作成**  
   `tasks.json` に各タスクの `id`、`duration`（ミリ秒）、`deadline`（UNIX タイムスタンプ）、`predecessors`（先行タスク ID の配列）、`resourcePool`（リソースグループ名）を記述。  
   例:  
   ```json
   {
     "tasks": [
       {"id": "T1", "duration": 3000, "deadline": 1740000000000, "predecessors": [], "resourcePool": "dev"},
       {"id": "T2", "duration": 5000, "deadline": 1740005000000, "predecessors": ["T1"], "resourcePool": "dev"}
     ]
   }
   ```

2. **クリティカルパス解析の実行**  
   次のコマンドでスケジュールを最適化:  
   ```bash
   deadliner --input tasks.json --critical-path-only --conflict-resolution-strategy preemptive --output schedule.optimized.json
   ```  
   出力には各タスクの `startTime`、`endTime`、`slackTime` がミリ秒単位で含まれます。

3. **PostgreSQL へのスケジュール保存と履歴確認**  
   解析結果をデータベースに保存:  
   ```bash
   deadliner --input schedule.optimized.json --db-connection "postgresql://user:pass@localhost:5432/deadliner" --store-snapshot
   ```  
   過去のスナップショットを確認するには:  
   ```bash
   deadliner --db-connection "postgresql://user:pass@localhost:5432/deadliner" --revision-history --revision-id 20250315_120000
   ```

### ベストプラクティス

- **優先度と余裕時間の定期的な再評価**  
  プロジェクト進行中にタスクの優先度（`priority`）や所要時間（`duration`）が変わるたびに、`deadliner --re-evaluate` を実行してスケジュールを再最適化する。特に、クリティカルパス上のタスクに遅延が発生した場合は即座に再計算を行い、他のタスクへの影響を最小化する。

- **リソースプールの適切な設計**  
  同一リソースを共有するタスクは同じ `resourcePool` に属させる。ただし、競合が頻発する場合は、プールを細分化（例：`dev-frontend`、`dev-backend`）して並列実行の機会を増やす。`--conflict-resolution-strategy fair-share` を併用すると、すべてのタスクが均等にリソースを利用できるようになる。
