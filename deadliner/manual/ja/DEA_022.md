# deadliner - DEA_022

## 概要

deadliner は、複数プロジェクトにまたがるタスクのデッドラインをマイクロ秒単位で管理し、依存関係のクリティカルパスを動的に再計算するプロフェッショナル向けスケジューリングエンジンです。PostgreSQL の非同期通知機能と組み合わせることで、チーム全体の進捗をリアルタイムに可視化します。

## 詳細な内容

### deadliner とは

deadliner は、タスクの開始時刻・終了時刻・バッファ時間を厳密に定義し、リソース競合を自動解決する高精度なデッドライン追跡ツールです。特に、アジャイル開発におけるスプリント計画や、複数チームが並行稼働する大規模プロジェクトのガントチャート管理を想定しています。

### コア機能

- **動的クリティカルパス再計算エンジン**  
  タスクの依存関係グラフ（DAG）を保持し、任意のタスクの遅延が発生した際に、影響を受ける後続タスクのデッドラインを `O(V+E)` アルゴリズムで再計算。再計算結果は即座に PostgreSQL の `NOTIFY/LISTEN` 経由で全クライアントにプッシュ通知される。  
  パラメータ例: `--recalc-interval 500ms`（再計算間隔をミリ秒単位で指定可能）

- **リソース競合解決 & 自動スロットリング**  
  同一リソース（担当者名・マシン名など）に複数のタスクが割り当てられた場合、タスクの優先度（`priority` パラメータ：0〜100、デフォルト50）と残りバッファ時間をもとに、最適な実行順序を自動決定。競合が解消できない場合は警告ログを出力し、管理者に手動介入を促す。

- **タイムラインスナップショット & 差分比較**  
  任意の時点における全タスクのデッドライン状態をスナップショットとして保存（`snapshot` コマンド）。過去のスナップショットと現在のスケジュールを比較し、どのタスクがどれだけ遅延/前倒しされたかを `diff` 形式で表示。  
  出力例: `Task #42: deadline 2025-03-15T14:30:00.000Z → 2025-03-15T15:45:00.000Z (+75min)`

### 使い方

1. **PostgreSQL 接続設定 & 初期スキーマ作成**  
   ```bash
   deadliner init --db-host localhost --db-port 5432 --db-name deadline_db --schema public
   ```  
   上記コマンドで、タスクテーブル（`tasks`）、依存関係テーブル（`dependencies`）、スナップショットテーブル（`snapshots`）が自動生成される。接続情報は `~/.deadliner/config.toml` に保存され、以降のコマンドで再利用可能。

2. **タスクの一括インポートと依存関係の定義**  
   ```bash
   deadliner import --format csv --file tasks.csv --delimiter ","
   ```  
   CSV は以下のカラムを必須とする：  
   `id, title, start_at, end_at, buffer_minutes, priority, resource`  
   インポート後、依存関係を追加する場合は `deadliner link --from 42 --to 56 --lag 10m` のように指定。`--lag` はタスク完了から次のタスク開始までの最小インターバル（例：`10m` = 10分）。

3. **リアルタイムモニタリング & アラート設定**  
   ```bash
   deadliner watch --threshold 80% --notify slack --webhook https://hooks.slack.com/services/xxx
   ```  
   全タスクの残りバッファ時間がしきい値（`--threshold`）を下回った場合、Slack へ Webhook 経由で警告が飛ぶ。`--notify email` に変更すれば SMTP 経由のメール通知も可能。  
   モニタリング画面は `http://localhost:8080` で起動し、クリティカルパス上のタスクが赤色でハイライト表示される。

### ベストプラクティス

- **バッファ時間の設計は「見積もりの20%ルール」を採用せよ**  
  各タスクの `buffer_minutes` には、見積もり作業時間の20%を最低限設定する。これにより、予期せぬ遅延が発生してもクリティカルパス全体への影響を吸収できる。deadliner の自動スロットリング機能は、バッファ残量が10%を切ったタスクを最優先で処理する。

- **スナップショットはスプリント終了時およびマイルストーン到達時に必ず取得せよ**  
  スナップショットを取得しておけば、後日「なぜそのスプリントで遅延が発生したのか」を定量的に分析できる。`deadliner diff --from snapshot_2025-03-01 --to snapshot_2025-03-15` のように比較することで、原因タスクを特定し、次のスプリント計画にフィードバックできる。
