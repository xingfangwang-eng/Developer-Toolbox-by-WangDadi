# deadliner - DEA_014

## 概要

`deadliner` は、タスクの絶対締切と相対締切を同時に管理する分散型スケジューリングエンジンです。PostgreSQL の `TIMESTAMPTZ` 型と `INTERVAL` 型を活用し、タイムゾーンを跨いだプロジェクトでも 1 ミリ秒単位の整合性を保証します。

## コア機能

### 機能 1: 動的バッファ付きデッドライン計算エンジン
タスクの依存関係グラフを走査し、各ノードに最適なバッファ（デフォルト: 作業時間の 15 %）を自動挿入。  
- パラメータ: `--buffer-ratio 0.15`（0.0 〜 1.0 の浮動小数点）  
- 内部で `pg_safe_dependency_walk()` を呼び出し、循環参照を検出した場合は即座にエラーコード `E_DEP_CYCLE` を返却します。

### 機能 2: マルチタイムゾーン対応の絶対・相対締切同期
タスク作成時に `UTC` 絶対時刻と `JST` 相対時刻を同時に記録。  
- パラメータ: `--timezone Asia/Tokyo`（IANA タイムゾーン識別子）  
- PostgreSQL の `AT TIME ZONE` 句を用いて、クエリ実行時に自動変換。  
- 例: `2025-12-31 23:59:59 UTC` を `2026-01-01 08:59:59 JST` として表示可能。

### 機能 3: イベント駆動型リマインダー・エスカレーション機構
締切時刻の 24 時間前、1 時間前、10 分前の 3 段階で通知を発火。  
- パラメータ: `--reminder-offset '24h,1h,10m'`（カンマ区切り、単位は h/m/s）  
- 未対応タスクが締切を過ぎた場合、エスカレーションチェーンを起動し、上位管理者に `pg_notify` 経由で即時アラートを送信。

## 使い方

### ステップ 1: データベーススキーマの初期化
```bash
deadliner init-db \
  --host localhost \
  --port 5432 \
  --database deadline_db \
  --user admin \
  --password-secret env:DB_PASS
```
- 内部で `CREATE EXTENSION IF NOT EXISTS pgcrypto;` を実行し、タスク ID に UUID v7 を割り当てます。

### ステップ 2: タスクと依存関係の登録
```bash
deadliner add-task \
  --id "TASK-001" \
  --title "バックエンドAPI実装" \
  --deadline-abs "2025-12-20T18:00:00Z" \
  --deadline-rel "P5D" \
  --depends-on "TASK-000" \
  --buffer-ratio 0.2 \
  --timezone "Asia/Tokyo"
```
- `--deadline-rel` は ISO 8601 期間形式（`P5D` = 5 日間）。  
- 依存タスクが未存在の場合はエラー `E_DEP_NOT_FOUND` が返ります。

### ステップ 3: スケジュールの実行と監視
```bash
deadliner run-scheduler \
  --interval 5000 \
  --reminder-offset '24h,1h,10m' \
  --log-level debug
```
- `--interval 5000` は 5000 ミリ秒（5 秒）ごとにポーリング。  
- リマインダー発火時に標準出力へ `[REMINDER] TASK-001: 残り1時間` の形式でログ出力。

## ベストプラクティス

### プラクティス 1: バッファ率はプロジェクトの複雑度に応じて調整する
単一チームの小規模プロジェクトでは `--buffer-ratio 0.1`（10 %）、複数チームが絡む大規模プロジェクトでは `0.25`（25 %）を推奨。  
- 実績データを `deadliner stats --period 30d` で取得し、過去の遅延率に基づいてバッファ率を動的にチューニングすると効果的。

### プラクティス 2: タイムゾーンはプロジェクト開始時に統一する
全タスクの絶対時刻を `UTC` で記録し、表示時のみ各メンバーのローカルタイムゾーンに変換する運用を徹底。  
- `deadliner config --default-timezone UTC` でデフォルトを固定。  
- 日本チームと欧州チームが混在する場合でも、PostgreSQL の `TIMESTAMPTZ` 型が自動変換を行うため、手動計算ミスがゼロになる。
