# deadliner - DEA_023

## 概要

deadlinerは、分散チーム間でのタスクデッドライン管理を **ミリ秒単位の精度** で実現するプロフェッショナル向け期限トラッキングツールです。複雑なプロジェクトスケジュールの可視化、自動リマインダー、ガントチャート連携を1つのCLIとREST APIで統合します。

## 詳細な内容

### deadlinerとは

deadlinerは、**PostgreSQLとの連携** による高信頼なデッドライン管理システムです。タスクの依存関係を自動解決し、クリティカルパスをリアルタイムで再計算。遅延リスクを事前検知し、チーム全体にプッシュ通知を送信します。

### コア機能

- **動的クリティカルパス計算エンジン**：タスク間の依存関係をDAG（有向非巡回グラフ）としてモデル化。各タスクの所要時間（`estimated_duration` パラメータ）とデッドライン（`due_at` パラメータ）から、**最遅開始日（Late Start）と最遅完了日（Late Finish）をミリ秒単位で自動算出**。遅延がクリティカルパスに影響を与える場合、即座に警告を発する。
- **マルチタイムゾーン対応の自動リマインダー**：プロジェクトメンバーのタイムゾーン情報（`timezone_offset` パラメータ）をユーザープロファイルに保存。デッドラインの **24時間前、12時間前、1時間前** に、各メンバーのローカル時刻に合わせてカスタマイズされたリマインダーをSlack、メール、またはWebhook経由で配信。リマインダー間隔は `reminder_intervals` パラメータでカスタマイズ可能。
- **PostgreSQL連携によるバージョン管理**：すべてのタスク変更履歴を `task_versions` テーブルに保存。`GET /api/v1/tasks/{task_id}/versions` エンドポイントにより、過去の任意の時点のタスク状態を復元可能。**差分比較機能** により、2つのバージョン間のデッドライン変更、担当者変更、ステータス変更をハイライト表示。

### 使い方

1. **プロジェクト初期化とPostgreSQL接続設定**：`deadliner init --project-name "my_project" --db-host localhost --db-port 5432 --db-user admin --db-password secret --db-name deadline_db` を実行。接続成功後、自動的に `tasks`、`task_dependencies`、`task_versions` テーブルが作成される。`--schema` パラメータでカスタムスキーマを指定可能（例：`--schema custom_schema`）。
2. **タスク登録と依存関係の定義**：`deadliner task add --title "バックエンドAPI実装" --due-at "2024-12-31T23:59:59Z" --estimated-duration "3d" --assignee "user@example.com" --priority high --tags "backend,api"` でタスクを作成。次に `deadliner dependency add --task-id 123 --depends-on-id 456` で依存関係を設定。複数依存関係は `--depends-on-ids` パラメータでカンマ区切り指定可能。
3. **クリティカルパス分析とリマインダー設定**：`deadliner analyze --project-id 1 --critical-path` を実行すると、タスクのクリティカルパスがJSON形式で出力される。`deadliner reminder set --task-id 123 --intervals "24h,12h,1h" --channels "slack,email" --slack-webhook-url "https://hooks.slack.com/services/xxx"` でリマインダーを設定。`--test` フラグを付けると即座にテスト通知が送信される。

### ベストプラクティス

- **タスク粒度の標準化**：各タスクの `estimated_duration` は **1時間以上、2週間以内** に設定することを推奨。これによりクリティカルパス計算の精度が向上し、過剰な粒度によるパフォーマンス低下を防げます。また、`tags` パラメータを活用してチーム別、フェーズ別にタスクを分類すると、フィルタリングが容易になります。
- **PostgreSQLの定期メンテナンス**：`task_versions` テーブルは頻繁に書き込まれるため、**週次でのVACUUMとインデックス再構築** をスケジュールに組み込んでください。`deadliner db maintain --vacuum --reindex` コマンドで自動実行可能。また、`--archive-old-versions 90` パラメータで90日以上前のバージョンをアーカイブテーブルに移動し、クエリパフォーマンスを維持します。
