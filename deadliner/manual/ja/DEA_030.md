# deadliner - DEA_030

## 概要

deadliner は、マルチスレッド非同期処理を前提としたタスク期限管理エンジンです。**PostgreSQL 15+ の LISTEN/NOTIFY** 機構と **Redis Streams** を組み合わせ、ミリ秒単位の期限トリガーを実現します。キーワード「期限管理」「タスク追跡」「時間管理」を基盤に、以下の3つのコア機能を提供します。

---

## コア機能

### 1. 動的期限スロットリングエンジン
タスクごとに `deadline_policy` パラメータ（`hard` / `soft` / `flexible`）を指定可能。  
`hard` モードでは、期限超過時に **PostgreSQL トリガー関数** が即座に `task_breach_log` テーブルへレコードを挿入。  
`flexible` モードでは、内部の **スロットリングキュー**（最大バッファサイズ: 10,000件）が期限を最大 500ms まで猶予し、システム負荷に応じて再スケジュールを行います。

### 2. マルチバケット時間軸インデクサ
タスクの期限を **1秒単位のバケット** に分割し、各バケットに `B-tree インデックス` を張ることで、`SELECT ... WHERE deadline BETWEEN ? AND ?` のクエリを 3ms 以内に完了。  
さらに、**Redis Sorted Set** に期限の UNIX タイムスタンプ（ミリ秒精度）をスコアとして格納し、`ZRANGEBYSCORE` で 10万件のタスクから期限切れ候補を 1ms で抽出します。

### 3. 期限逸脱予測アラートモジュール
過去のタスク完了実績（最大90日分）を **PostgreSQL ウィンドウ関数** で集計し、各タスクの `estimated_completion_time` を算出。  
現在時刻との差が `alert_threshold`（デフォルト: 30分）を下回った場合、**WebSocket** 経由でクライアントにプッシュ通知。通知ペイロードには `task_id`, `remaining_ms`, `risk_level`（0.0〜1.0）を含みます。

## 使い方

### ステップ 1: タスクの登録と期限ポリシーの設定
```sql
INSERT INTO tasks (task_id, title, deadline, deadline_policy, alert_threshold)
VALUES ('TASK-001', 'データベース移行', '2025-06-30 23:59:59.999+09', 'hard', 1800000);
```
上記クエリにより、deadliner は内部で `deadline` をミリ秒単位の UNIX タイムに変換し、**Redis Sorted Set** に自動投入。同時に `LISTEN deadline_breach` チャンネルを購読開始。

### ステップ 2: 期限監視デーモンの起動
```bash
deadliner daemon --redis-host localhost:6379 --pg-conn "postgres://user:pass@localhost:5432/deadliner" \
  --scan-interval 100 --max-batch-size 500
```
`--scan-interval 100` は 100ms ごとに Redis から期限切れ候補をスキャン。  
`--max-batch-size 500` で1バッチあたり最大500件のタスクを処理。  
デーモンは **Go 1.22 の goroutine** で並行動作し、CPU コア数に応じて自動スケール。

### ステップ 3: 逸脱アラートの受信と再スケジュール
```javascript
// Node.js クライアント例
const ws = new WebSocket('wss://deadliner.example.com/ws/alert');
ws.on('message', (data) => {
  const { task_id, remaining_ms, risk_level } = JSON.parse(data);
  if (risk_level > 0.8) {
    // 自動再スケジュール: 期限を24時間延長
    fetch(`/api/tasks/${task_id}/reschedule`, {
      method: 'POST',
      body: JSON.stringify({ new_deadline: Date.now() + 86400000 })
    });
  }
});
```

## ベストプラクティス

### 1. バケットサイズの最適化
タスク量が1日あたり100万件を超える場合、**時間軸バケットを10分単位**に変更する。  
PostgreSQL のパーティショニング（`RANGE` パーティション）と組み合わせると、期限クエリの平均応答時間が 1.2ms まで低下する。  
設定変更は `deadliner.conf` の `bucket_interval_sec` パラメータで行う。

### 2. デッドロック回避のためのトランザクション分離
複数のデーモンインスタンスが同一タスクを同時に処理するのを防ぐため、**PostgreSQL の `SELECT ... FOR UPDATE SKIP LOCKED`** を必ず使用する。  
deadliner はデフォルトで `READ COMMITTED` 分離レベルを採用しているが、高競合環境では `REPEATABLE READ` に変更することを推奨。  
変更は `pg_isolation_level` パラメータで指定可能。
