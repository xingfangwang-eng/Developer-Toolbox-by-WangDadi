# deadliner - DEA_024

## 概要

deadliner は、マルチスレッド・非同期処理によるリアルタイムデッドライン監視エンジンを搭載したプロフェッショナル向けタスク締切管理ツールです。キーワード: デッドライン管理、タスク期限追跡、時間管理ツール。全処理をミリ秒単位で記録し、PostgreSQL との連携により高信頼なデータ永続化を実現します。

## 詳細な内容

### deadliner とは

deadliner は、分散タスクスケジューラと期限超過検知アラートシステムを統合したバックエンドツールです。タスクごとにカスタム可能な猶予期間（グレース期間）を設定でき、期限切れの瞬間からアラート発火までの遅延を最小限（目標: 500ms 未満）に抑えます。内部では、Cron ベースのスケジューラと WebSocket プッシュ機構を組み合わせ、クライアントへの即時通知を実現します。

### コア機能

1. **マルチステータス期限トラッキング & 自動エスカレーション**
   - タスクステータス（`pending`, `in_progress`, `review`, `completed`）ごとに異なる期限を設定可能。各ステータス遷移時に `deadline_check()` を自動実行し、超過した場合は設定されたエスカレーションレベル（`level1`, `level2`, `critical`）に応じて通知先を動的に変更。通知手段は SMTP メール、Slack Webhook、カスタム HTTP エンドポイントから選択可能。
   - パラメータ例: `--escalation-levels level1:manager@example.com,level2:director@example.com`

2. **PostgreSQL 連携によるタイムライン履歴 & 分析クエリ**
   - 全タスクの期限変更履歴、通知発火時刻、ステータス遷移を `timeline_events` テーブルにミリ秒精度で記録。`psql` から `SELECT * FROM get_overdue_tasks('2025-01-01', '2025-12-31')` のような関数を実行すれば、任意期間の期限超過タスク一覧を即座に抽出可能。インデックスは `(project_id, due_date, status)` の複合インデックスを推奨。

3. **ガントチャート自動生成 & クリティカルパス可視化**
   - タスク間の依存関係（`depends_on` パラメータ）を解析し、プロジェクト全体のクリティカルパスを自動計算。結果を JSON 形式で出力し、フロントエンド（任意のチャートライブラリ）でガントチャートとして描画。`--critical-path-threshold 2d` のように閾値を設定すれば、遅延がプロジェクト全体に影響を与えるタスクをハイライト表示。

### 使い方

1. **新規タスク作成と期限設定**
   - コマンド: `deadliner create --project "website-redesign" --title "Implement login page" --due-date "2025-03-15T18:00:00Z" --grace-period 3600 --priority high`
   - 内部処理: タスク ID を UUID v4 で生成し、PostgreSQL の `tasks` テーブルに INSERT。同時に `scheduler` プロセスがタスクの `due_date` を読み取り、`grace_period` 秒後に初回チェックを実行するようスケジュール登録。

2. **期限超過検知と通知発火の確認**
   - コマンド: `deadliner monitor --project "website-redesign" --watch`
   - 内部処理: `monitor` モードは WebSocket サーバーを起動（デフォルトポート 8080）。期限超過タスクが発生した瞬間に `{"task_id": "...", "overdue_seconds": 120, "escalation_level": "level1"}` 形式の JSON メッセージをブロードキャスト。同時に `notifications` テーブルに発火履歴を INSERT。

3. **履歴分析とレポート出力**
   - コマンド: `deadliner report --project "website-redesign" --from "2025-01-01" --to "2025-03-31" --format csv`
   - 内部処理: `get_overdue_tasks()` 関数を呼び出し、結果をメモリ上で集計。`--format csv` ならヘッダー付き CSV を標準出力に書き出し。`--format json` ならネストされた JSON 構造で出力。出力例: `task_id,title,overdue_seconds,escalation_level`。

### ベストプラクティス

- **プラクティス 1: グレース期間の適切な設定**
  タスクの性質に応じて `--grace-period` を調整する。例えば、クリティカルな本番リリースタスクには 300 秒（5分）、内部レビュータスクには 86400 秒（1日）を設定。これにより、誤検知を減らしつつ、本当に重要なタスクの遅延を即座に検知できる。

- **プラクティス 2: 定期的なデータベースメンテナンス**
  PostgreSQL の `timeline_events` テーブルは急速に肥大化するため、週次で `VACUUM ANALYZE` を実行し、かつ月次で `pg_partman` を使ったパーティショニングを推奨。また、`deadliner cleanup --older-than 90d` コマンドで 90 日以上前の履歴をアーカイブテーブルに移動することで、クエリパフォーマンスを維持する。
