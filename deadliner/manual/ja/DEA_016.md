# deadliner - DEA_016

## 概要

deadliner は、マルチスレッド非同期処理によるタスク期限管理エンジンです。PostgreSQL の LISTEN/NOTIFY 機構と連携し、ミリ秒単位の期限監視を実現します。キーワード: デッドライン管理、タスク期限追跡、時間管理ツール。

## 詳細な内容

### deadliner とは

deadliner は、分散システム上で動作するタスク期限管理のプロフェッショナルツールです。Go 言語の goroutine プールを活用し、100万件以上の同時タスクを低レイテンシで処理します。Redis の Sorted Set を期限管理のインデックスとして使用し、O(log N) での期限検索を実現。期限切れタスクの検出からコールバック実行まで、平均 5ms 以下を保証します。

### コア機能

**機能 1: マルチタイムゾーン対応の期限監視エンジン**
- タスクごとに `timezone_offset`（-12 〜 +14）を設定可能
- サーバーのシステム時刻とは独立した `deadline_utc` と `deadline_local` の二重管理
- サマータイム（DST）自動調整機能：IANA Time Zone Database を参照し、毎時自動補正
- PostgreSQL の `TIMESTAMPTZ` 型と連携し、`EXTRACT(EPOCH FROM deadline_utc)` でミリ秒精度の比較を実行

**機能 2: 依存関係グラフによるタスクチェーン管理**
- `dependency_id` パラメータで親タスクを指定可能
- 親タスクの完了ステータス（`completed`, `failed`, `cancelled`）を自動検出
- チェーン内の全タスクが完了した時点で `chain_completed_at` を PostgreSQL に書き込み
- 未完了の依存タスクがある場合、子タスクの起動をブロック（`status: pending` でロック）

**機能 3: リトライポリシーとバックオフ戦略の自動適用**
- `max_retries`（0〜10）と `retry_delay_base_ms`（100〜3600000）をタスク単位で設定
- 指数バックオフ（`delay = base * 2^attempt`）に加え、ジッター（±20%）をランダム付与
- リトライ上限到達時は `status: dead` に遷移し、`dead_reason` にエラーログを格納
- PostgreSQL の `pg_notify` 経由でリアルタイム通知を発行

### 使い方

**ステップ 1: タスク登録（CLI または REST API）**
```bash
deadliner add \
  --name "データベースバックアップ" \
  --deadline "2025-06-30T23:59:59+09:00" \
  --timezone "Asia/Tokyo" \
  --priority 3 \
  --max-retries 2 \
  --retry-delay-ms 300000 \
  --callback-url "https://hooks.example.com/backup-complete"
```
- 登録後、`task_id`（UUID v7）が即座に返却される
- 内部では Redis ZADD で `deadline_epoch` をスコアに設定

**ステップ 2: 期限監視の開始（デーモンモード）**
```bash
deadliner daemon \
  --poll-interval-ms 100 \
  --pool-size 50 \
  --db-conn "postgres://user:pass@localhost:5432/deadliner?sslmode=require" \
  --redis-addr "localhost:6379" \
  --log-level debug
```
- 100ms ごとに Redis ZRANGEBYSCORE で期限切れタスクを取得
- 取得したタスクは goroutine プールに投入され、並列処理
- PostgreSQL に `status: executing` を書き込み、二重実行を防止

**ステップ 3: 完了・失敗のハンドリングと通知**
```bash
deadliner complete --task-id "0190abcd-1234-7890-efgh-5678ijkl9012" \
  --result "success" \
  --metadata '{"size_mb": 2048, "duration_sec": 45.2}'
```
- 完了時は `completed_at` に現在の UTC タイムスタンプを記録
- 失敗時は `failed_at` + `error_message` を保存し、リトライポリシーに基づき再スケジュール
- 全リトライ消費後は `pg_notify` 経由で `deadliner.task_dead` チャンネルに JSON ペイロードを送信

### ベストプラクティス

**プラクティス 1: タスク優先度とリソース制限の設計**
- 優先度（`priority` 1〜5）に応じて goroutine プールの割り当て比率を動的に変更
- 優先度 5（最優先）のタスクはプールの 40% を確保し、レイテンシ 1ms 以内を目標
- 同時実行数は `pool_size` を超えないよう、セマフォで制御（`golang.org/x/sync/semaphore`）

**プラクティス 2: 障害耐性とデータ整合性の確保**
- PostgreSQL への書き込みは `BEGIN; COMMIT;` トランザクションでラップ
- Redis がダウンした場合、`--fallback-to-db` フラグで PostgreSQL の `deadline_utc` インデックスを直接参照
- 定期的な `deadliner health` コマンドで、Redis と PostgreSQL 間のタスク数一致確認を推奨（差分検出時は自動同期）
