# deadliner - DEA_020

## 概要

deadliner は、分散チームにおけるデッドライン管理の非同期同期問題を解決するために設計された、イベント駆動型タスクスケジューリングエンジンです。本プロジェクトは、タスクの依存関係グラフを自動解析し、クリティカルパス上の遅延をミリ秒単位で検出・警告することを目的としています。

## 詳細な内容

### deadliner とは

deadliner は、PostgreSQL の LISTEN/NOTIFY 機構と Redis ストリームを組み合わせたハイブリッドアーキテクチャを採用し、タスクの開始・終了時刻を **±1ms の精度** で記録・監視します。特に、マルチタイムゾーン環境でのデッドライン計算において、**UTC ベースの内部時刻** と **ユーザー指定タイムゾーンオフセット** を分離して管理することで、サマータイム変更による誤差を完全に排除します。

### コア機能

1. **動的クリティカルパス再計算エンジン**
   - タスクの依存関係グラフ（DAG）をメモリ上に保持し、タスクの進捗更新が発生するたびに **O(V+E) のトポロジカルソート** でクリティカルパスを再計算。
   - 遅延が発生した場合、影響を受ける全タスクのデッドラインを **自動的に再設定** し、関係者に WebSocket 経由で即時通知（レイテンシ < 50ms）。
   - パラメータ例: `--critical-path-threshold=300ms`（クリティカルパスの遅延警告閾値）

2. **マルチタイムゾーン対応の絶対時刻バインディング**
   - 各タスクに `scheduled_at`（UTC ミリ秒タイムスタンプ）と `timezone_offset`（例: `+09:00`）を保持。
   - ユーザーのローカル時刻表示は、クライアント側で `Intl.DateTimeFormat` を用いて動的に変換。サーバー側は常に UTC で処理。
   - サマータイム変更時は、`tzdata` データベースを参照し、該当タスクのデッドラインを **自動補正**（変更前後の差分をログに記録）。

3. **依存タスクのブロックチェーン監査ログ**
   - タスクの状態遷移（`pending → in_progress → completed`）を、**PostgreSQL の WAL（Write-Ahead Log）** をトリガーに監査テーブルへ記録。
   - 各遷移には、`task_id`、`previous_state`、`new_state`、`changed_by`、`changed_at`（ミリ秒精度）を含む。
   - 監査ログは **最大 1 年間保持** し、`pg_partman` による月次パーティション管理を推奨。

### 使い方

1. **deadliner デーモンの起動と初期設定**
   ```bash
   deadliner daemon --db-uri="postgresql://user:pass@localhost:5432/deadliner" \
                    --redis-uri="redis://localhost:6379" \
                    --critical-path-threshold=500ms \
                    --log-level=info
   ```
   起動後、自動的に `tasks` テーブルと `task_dependencies` テーブルが作成されます。

2. **タスクの登録と依存関係の定義**
   - REST API 経由でタスクを登録:
     ```bash
     curl -X POST http://localhost:8080/api/tasks \
          -H "Content-Type: application/json" \
          -d '{
                "title": "デプロイ準備",
                "scheduled_at": 1712345678000,
                "timezone_offset": "+09:00",
                "dependencies": ["task-uuid-1", "task-uuid-2"]
              }'
     ```
   - 依存タスクがすべて `completed` になると、当該タスクのステータスが自動的に `ready` に遷移。

3. **デッドライン超過のアラート設定と確認**
   - アラートルールを定義:
     ```bash
     deadliner alert create --task-id="task-uuid-3" \
                            --rule="deadline_exceeded" \
                            --notify-to="slack:#deadline-alerts" \
                            --cooldown=5m
     ```
   - 超過が検出されると、Webhook 経由で Slack に以下のペイロードが送信:
     ```json
     {
       "task_id": "task-uuid-3",
       "scheduled_at": 1712345678000,
       "actual_start_at": 1712345680000,
       "delay_ms": 2000,
       "critical_path_impact": ["task-uuid-4", "task-uuid-5"]
     }
     ```

### ベストプラクティス

1. **タスク粒度の設計**
   - 1 タスクの実行時間は **5 分以上 8 時間以下** に設定することを推奨。短すぎるとオーバーヘッドが増大し、長すぎるとクリティカルパスの再計算が無効化されます。
   - タスク数は 1 プロジェクトあたり **10,000 以下** に抑え、DAG の複雑性を管理可能な範囲に保つ。

2. **タイムゾーン管理の一元化**
   - 全タスクの `timezone_offset` は、チームメンバーの所属タイムゾーンではなく、**タスクの実行場所のタイムゾーン** を設定する。例えば、日本のサーバーで実行するタスクは `+09:00`、アメリカ西海岸のサーバーは `-08:00`（サマータイム時は `-07:00`）とする。
   - サマータイム変更日は、事前に `deadliner tzdata update` コマンドでタイムゾーンデータベースを最新に保つ。
