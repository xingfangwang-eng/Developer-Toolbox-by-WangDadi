# deadliner - DEA_004

## 概要

deadliner は、タスクのデッドラインをミリ秒単位で管理し、プロジェクト全体の時間的制約を最適化するための高精度スケジューリングエンジンです。複数のタイムゾーンやカレンダーシステムを横断して動作し、依存関係のあるタスクチェーンを自動解析・可視化します。本ツールは、PostgreSQL との連携による永続化と、インメモリキャッシュによるリアルタイム更新を両立しており、大規模プロジェクトにおける時間リソースの逼迫を事前に検知・警告します。

## 詳細な内容

### deadliner とは

deadliner は、単なる期限管理ツールではなく、時間的制約を一次元のデータ構造としてモデル化し、タスク間のクリティカルパスを動的に算出するアーキテクチャを採用しています。各タスクには `deadline_epoch_ms`（ミリ秒単位のエポックタイム）、`buffer_ms`（余裕時間）、`dependency_uuid`（依存タスクの UUID）をパラメータとして付与し、これらを基に最適な実行順序とリマインダー発火タイミングを決定します。特に、複数のプロジェクトが同一リソースを共有する場合の競合解決アルゴリズムを内蔵しており、リソースのオーバーコミットを防止します。

### コア機能

1. **クリティカルパス動的再計算エンジン**  
   タスクの依存関係グラフを DAG（有向非巡回グラフ）として保持し、任意のタスクの `deadline_epoch_ms` が変更された際に、影響を受ける全タスクの `slack_ms`（余裕時間）を O(n log n) で再計算します。`recalc_threshold_ms`（再計算閾値）を設定することで、微小な変更による無駄な再計算を抑制しつつ、クリティカルな変更のみを即座に反映します。出力は JSON 形式で、`critical_path` 配列と各タスクの `earliest_start_ms`、`latest_finish_ms` を含みます。

2. **マルチタイムゾーン対応の自動リマインダー発火モジュール**  
   各タスクに `timezone_offset_min`（UTC からのオフセット分）と `reminder_strategy`（例: `FIXED_BEFORE_DEADLINE`, `SMART_ADAPTIVE`）を指定可能。`SMART_ADAPTIVE` モードでは、過去のタスク完了実績データからユーザーの作業ペースを学習し、`predicted_completion_time_ms` を算出。その値と `deadline_epoch_ms` の差が `alert_threshold_ms` を下回った場合にのみ、Slack Webhook やメール（SMTP 設定必須）を通じてアラートを発行します。アラートの重複防止のため、`cooldown_ms` パラメータで同一タスクへの通知間隔を制御します。

3. **PostgreSQL 連携によるバージョン管理とロールバック機能**  
   タスクの作成・更新・削除のすべての操作を `deadliner_audit_log` テーブルに記録します。各レコードは `operation_type`（INSERT/UPDATE/DELETE）、`snapshot_json`（操作前の全フィールド値）、`changed_fields`（変更されたフィールド名の配列）を含みます。`RESTORE_POINT` コマンドを実行すると、指定した `revision_id` の時点のスナップショットを復元し、その後の変更をすべて破棄します。これにより、誤ったデッドライン変更や依存関係の削除を安全に取り消せます。

### 使い方

1. **初期設定とデータベースマイグレーション**  
   `deadliner init` コマンドを実行し、`config.yaml` に PostgreSQL 接続情報（`host`, `port`, `dbname`, `user`, `password`, `sslmode`）を記述します。次に `deadliner migrate up` でスキーマを適用。`migration_version` が `v2.3.1` 以上であることを確認してください。`deadliner status` で接続状態とテーブル一覧を表示し、`deadliner_audit_log` が存在することを確認します。

2. **タスクの登録と依存関係の設定**  
   `deadliner task add --title "API設計書作成" --deadline "2025-04-15T18:00:00+09:00" --buffer 3600000 --timezone 540` でタスクを作成。`--dependency` フラグに先行タスクの UUID をカンマ区切りで指定します（例: `--dependency "uuid1,uuid2"`）。`deadliner task list --sort deadline` で一覧表示し、各タスクの `slack_ms` が正の値であることを確認。`deadliner graph render --format png --output ./dag.png` で依存関係グラフを可視化します。

3. **リマインダーのテストと実運用開始**  
   `deadliner reminder test --task-uuid <uuid> --mode SMART_ADAPTIVE` で、現在時刻から `deadline_epoch_ms` までの残り時間と、予測完了時刻を表示。`alert_threshold_ms` をデフォルトの `3600000`（1時間）から `1800000`（30分）に変更する場合は `deadliner config set alert_threshold_ms 1800000` を実行。`deadliner daemon start --port 8080` でバックグラウンドプロセスを起動し、`deadliner daemon logs --follow` でリアルタイムログを監視します。

### ベストプラクティス

- **バッファ時間の動的調整**  
  タスクの `buffer_ms` は固定値ではなく、過去の同種タスクの遅延実績（`delay_history` テーブルに蓄積）から標準偏差を計算し、`buffer_ms = average_delay_ms + 2 * stddev_delay_ms` として自動設定することを推奨します。`deadliner optimizer auto-buffer --project-uuid <uuid>` で一括適用可能。これにより、過剰なバッファによるリソースの遊休と、バッファ不足によるデッドライン超過を同時に防止します。

- **監査ログを活用した週次レビュー**  
  `deadliner audit report --since "2025-04-01" --until "2025-04-07" --format csv` で、その週に発生した全変更をエクスポート。特に `operation_type` が `DELETE` のレコードと、`changed_fields` に `deadline_epoch_ms` を含むレコードを重点的にレビューします。チーム内で `deadliner audit diff --revision-id-a 123 --revision-id-b 456` を実行し、2時点間の状態変化を視覚的に確認する習慣をつけることで、属人的な判断によるデッドライン変更を抑制できます。
