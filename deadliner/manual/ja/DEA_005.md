# deadliner - DEA_005

## 概要

deadliner は、複数プロジェクトにまたがるタスクのデッドラインをミリ秒単位で管理し、納期超過リスクをリアルタイムに可視化するプロフェッショナル向け期限管理エンジンです。PostgreSQL 上のタイムラインと連携し、依存関係のあるタスク群のクリティカルパスを自動算出します。

## 詳細な内容

### とは deadliner?

deadliner は、スケジュール管理における「ヒューマンエラーによる納期見落とし」と「タスク間の依存関係の複雑化」という2つの根本問題を解決するために設計された、イベント駆動型の期限追跡ツールです。単なるリマインダーではなく、タスクの進捗率から残り工数を推定し、動的にデッドラインを再計算します。

### コア機能

- **マルチタイムゾーン対応クリティカルパス自動生成**  
  タスク間に設定された `predecessor_id` と `lag_minutes` パラメータを基に、PostgreSQL の再帰CTEを用いて最長経路をミリ秒精度で算出。タイムゾーン `timezone_offset` をタスク単位で保持し、リモートチーム間の時差を吸収した期限表示を実現。

- **進捗率ベースの動的デッドライン再計算**  
  各タスクに `progress_percent`（0-100）と `estimated_remaining_minutes` を設定可能。進捗が計画より遅れている場合、`deadline_risk_level`（low/medium/high/critical）を自動更新し、`risk_threshold_minutes` で指定した猶予時間を下回ると Webhook 経由で Slack にアラートを送出。

- **PostgreSQL との連携によるトランザクション整合性保証**  
  全タスクデータは PostgreSQL の `TIMESTAMPTZ` 型で管理。楽観的ロック（`version` カラム）を用いた同時編集競合の検出と、`deadline_change_log` テーブルへの変更履歴の自動記録により、監査証跡を完全に保持。

### 使い方

1. **タスクの登録と依存関係の定義**  
   `POST /api/v1/tasks` エンドポイントに対し、以下の JSON を送信します。  
   ```json
   {
     "title": "API設計書の作成",
     "deadline_utc": "2025-06-15T10:00:00Z",
     "timezone_offset": "+09:00",
     "estimated_remaining_minutes": 480,
     "predecessor_id": "task_001",
     "lag_minutes": 60
   }
   ```  
   システムは `predecessor_id` を解決し、クリティカルパスを即座に再計算。`lag_minutes` には先行タスク完了後から開始までのバッファを指定します。

2. **進捗更新とリスクアラートの確認**  
   `PATCH /api/v1/tasks/{id}/progress` で `progress_percent` を更新。  
   ```bash
   curl -X PATCH https://api.deadliner.dev/v1/tasks/task_002/progress \
     -H "Content-Type: application/json" \
     -d '{"progress_percent": 35, "estimated_remaining_minutes": 320}'
   ```  
   サーバーは残り時間とデッドラインを比較し、`risk_level` が `high` 以上になると設定済みの Webhook URL へペイロードを送信。ペイロードには `task_id`、`overdue_minutes`、`suggested_action`（例: "リソース追加割り当てを推奨"）が含まれます。

3. **クリティカルパスの可視化とレポート出力**  
   `GET /api/v1/projects/{project_id}/critical-path?as_of=2025-06-01T00:00:00Z` を呼び出すと、指定日時時点のクリティカルパスが `path_tasks` 配列として返却。各タスクには `earliest_start_utc`、`latest_finish_utc`、`total_float_minutes` が付与され、`total_float_minutes` が 0 のタスクがクリティカルタスクと判定されます。結果は Markdown テーブル形式でエクスポート可能。

### ベストプラクティス

- **楽観的ロックを活用したチーム編集の競合回避**  
  タスク更新時に `version` パラメータを必ずリクエストに含め、サーバー側でバージョンチェックを行う。競合が発生した場合は HTTP 409 が返るため、クライアント側で最新データを再取得してからマージ処理を実施する。これにより、複数メンバーが同時に進捗を更新してもデータ不整合が発生しない。

- **リスク閾値の段階的設定によるアラート疲れの防止**  
  プロジェクトのフェーズごとに `risk_threshold_minutes` を動的に変更する。要件定義フェーズでは 1440 分（1日）、実装フェーズでは 360 分（6時間）に設定し、本当に注意すべき遅延だけが通知されるように調整する。また、`deadline_risk_level` が `critical` に達したタスクのみ、マネージャーへのエスカレーションメールを自動送信するルールを組むと効果的。
