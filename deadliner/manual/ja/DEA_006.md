# deadliner - DEA_006

## 概要

deadliner は、複数プロジェクトに跨るタスクのデッドラインを **ミリ秒単位** で管理し、リソース競合を自動検出するアーキテクチャです。PostgreSQL の `TIMESTAMPTZ` と `INTERVAL` 型を利用した厳密な時刻管理により、タイムゾーン依存のバグをゼロにします。

## 詳細な内容

### deadliner とは

deadliner は、タスクの依存関係グラフを DAG（有向非巡回グラフ）として保持し、クリティカルパスをリアルタイムに算出するプロフェッショナルツールです。プロジェクトマネージャーが手動で調整していた「タスクの先行関係」と「リソースの割り当て」を、アルゴリズムで自動最適化します。

### コア機能

**1. 動的クリティカルパス・エンジン（CPE）**
- パラメータ: `task_id`, `dependency_ids[]`, `estimated_duration_ms`, `resource_pool_id`
- タスクの依存関係が変更されるたびに、**O(n log n)** でクリティカルパスを再計算。遅延が発生した場合、影響を受ける後続タスクを即座にハイライトし、納期への影響を **ミリ秒単位** で表示します。

**2. リソース競合検出と自動スケジュール調整**
- パラメータ: `resource_id`, `max_concurrent_tasks`, `time_slot_start`, `time_slot_end`
- 同一リソース（担当者・マシン）に割り当てられたタスクの時間帯が重複した場合、競合を `conflict_level: HIGH/MEDIUM/LOW` で評価。ユーザーの許可なく、最適なスロットにタスクを再配置する「自動コンフリクト解決モード」を搭載。

**3. PostgreSQL 連携による永続化と履歴追跡**
- テーブル設計: `tasks`（`task_id UUID PK`, `deadline TIMESTAMPTZ`, `status task_status_enum`, `version INT`）
- 全てのタスク変更は `task_history` テーブルに INSERT オンリーで記録。任意の時点のスケジュールを `SELECT * FROM task_history WHERE task_id = $1 AND recorded_at <= $2 ORDER BY recorded_at DESC LIMIT 1` で再現可能。

### 使い方

1. **タスクの投入と依存関係の定義**
   ```sql
   INSERT INTO tasks (task_id, deadline, estimated_duration_ms, resource_pool_id)
   VALUES ('a1b2c3d4', '2025-06-30T17:00:00+09:00', 3600000, 'dev-team-01');
   ```
   その後、依存関係を `task_dependencies` テーブルに登録するだけで、CPE が自動的に最適な着手日を算出します。

2. **競合の確認と自動調整の実行**
   ```bash
   deadliner detect-conflicts --resource-pool dev-team-01 --auto-resolve
   ```
   コマンドを実行すると、競合が検出されたタスクの `scheduled_start` が自動更新され、変更ログが `task_history` に追記されます。

3. **過去のスケジュール状態の再現**
   ```sql
   SELECT * FROM reconstruct_schedule_at('2025-06-01T00:00:00+09:00');
   ```
   このストアドプロシージャにより、特定日時点の全タスク状態を完全再現。監査や「なぜあの時遅延したのか」の分析に利用します。

### ベストプラクティス

- **プラクティス 1: タスク粒度の統一**
  `estimated_duration_ms` は 30 分（1,800,000 ms）未満のタスクに分割しない。細かすぎるタスクは CPE の計算負荷が増大し、スケジュールの安定性が低下します。最低でも 1 時間（3,600,000 ms）単位での登録を推奨します。

- **プラクティス 2: リソースプールの設計**
  リソースプールは「スキルセット」と「稼働時間帯」で分割する。例: `backend-dev-jp`（日本時間 10:00-19:00）、`backend-dev-us`（米国東部時間 9:00-18:00）。これにより、deadliner はタイムゾーンを跨いだ最適なタスク割り当てを自動計算できます。
