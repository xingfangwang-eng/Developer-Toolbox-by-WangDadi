# deadliner - DEA_003

## 概要

deadlinerは、複数プロジェクトにまたがるタスクのデッドラインをミリ秒単位で管理するための高精度タイムラインエンジンです。PostgreSQLのタイムスタンプ型と連携し、タイムゾーン自動補正、リマインダーのジッター制御、依存タスクのクリティカルパス自動算出を実現します。キーワード：デッドライン管理、タスク期限追跡、時間管理ツール。

## 詳細な内容

### deadlinerとは

deadlinerは、プロジェクト管理における**期限逸脱リスクをリアルタイムで定量化**するプロフェッショナルツールです。PostgreSQLの`TIMESTAMPTZ`型をバックエンドに採用し、全タスクの開始・終了時刻をUTCで統一保存。フロントエンドからはユーザーのロケールに応じてJST/EST/PSTなどへ自動変換し、夏時間(DST)の切り替えによるずれを防止します。さらに、タスク間の依存関係をDAG（有向非巡回グラフ）として保持し、クリティカルパス法(CPM)に基づく最短完了時刻と余裕日数(フロート)を自動計算。これにより、どのタスクが全体の納期に直結するかを可視化します。

### コア機能

- **高精度タイムラインエンジン**：全タスクの開始・終了時刻をPostgreSQLの`TIMESTAMPTZ`型（マイクロ秒精度）で保持。リマインダー送信時には、サーバー側で`NOW()`との差分をミリ秒単位で計算し、指定されたリードタイム（例：`-P1DT2H30M`）に対して±50ms以内のジッターでトリガー。タイムゾーンは`pg_timezone_names`テーブルから動的に解決し、ユーザー設定に応じて自動変換。

- **依存タスクのクリティカルパス自動算出**：タスク間に`depends_on`パラメータ（UUID配列）を設定。エンジンはDAGを構築し、トポロジカルソート後に各タスクの最早開始日(ES)、最遅完了日(LC)、トータルフロート(TF)をO(V+E)で計算。フロートが0のタスクをクリティカルパスとして強調表示。例えば、`Task A (duration: 3d) → Task B (duration: 2d)` の依存関係がある場合、`Task A`の完了が1日遅れると`Task B`の開始も1日遅れ、全体の納期に直結することを自動検出。

- **バーンダウンチャートと期限超過リスクスコア**：全タスクの残り作業時間（`estimated_remaining_hours`パラメータ）を日次で取得し、PostgreSQLのウィンドウ関数を用いて累積実績曲線を生成。理想線（バーンダウン）との乖離から、期限超過確率をパーセンテージで算出（例：`risk_score: 87.3%`）。閾値を超えた場合、Slack/TeamsへのWebhook通知を自動発火。

### 使い方

1. **PostgreSQLスキーマの初期化とタスク登録**  
   `deadliner init --db-url postgresql://user:pass@localhost:5432/deadliner` を実行。自動で`tasks`テーブル（カラム: `id UUID PK`, `title TEXT`, `start_at TIMESTAMPTZ`, `end_at TIMESTAMPTZ`, `depends_on UUID[]`, `estimated_hours NUMERIC(5,1)`, `timezone TEXT DEFAULT 'UTC'`）と`reminders`テーブルが作成される。次に、`deadliner task add --title "API設計書作成" --start-at "2025-03-10T09:00:00+09:00" --end-at "2025-03-12T18:00:00+09:00" --depends-on "uuid-of-prev-task" --timezone "Asia/Tokyo"` でタスクを登録。`--depends-on`は省略可能で、複数指定はカンマ区切り。

2. **クリティカルパスの可視化とリスク評価**  
   `deadliner analyze --project-id <UUID>` を実行。エンジンがDAGを構築し、全タスクのES/LC/TFを計算。結果はJSON形式で出力され、`critical_path`フィールドにフロート0のタスクUUIDリストが格納される。同時に`risk_score`（0.0〜100.0）が算出され、例えば`risk_score: 87.3`の場合、`deadliner alert --threshold 80 --webhook-url https://hooks.slack.com/services/...` でSlackに「⚠️ 期限超過リスク87.3%: タスク「API設計書作成」の完了が遅れると全体納期に影響」と通知。

3. **リマインダーのカスタムジッター制御とログ検証**  
   `deadliner reminder set --task-id <UUID> --lead-time "-P1DT2H30M" --jitter-ms 50` で、タスク終了1日2時間30分前にリマインダーを設定。`--jitter-ms`でトリガー時刻のばらつきを±50msに制御。設定後、`deadliner log --task-id <UUID> --since "2025-03-09T00:00:00Z"` でPostgreSQLの`reminder_logs`テーブルから実際のトリガー時刻（`triggered_at TIMESTAMPTZ`）を取得し、期待時刻との差分をミリ秒単位で検証可能。

### ベストプラクティス

- **タイムゾーンはタスク単位で設定し、DB保存はUTCに統一する**：`start_at`と`end_at`は常にUTCで保存し、`timezone`カラムにユーザーのローカルタイムゾーン（例：`Asia/Tokyo`）を保持。これにより、夏時間の切り替えやチームメンバーのタイムゾーン分散による誤差を防止。表示時はフロントエンドで`timezone`に基づき変換。PostgreSQLの`AT TIME ZONE`構文を活用。

- **依存関係は循環参照を避け、フロート値を監視する**：`depends_on`に設定するUUIDは必ず既存タスクのIDとし、`deadliner validate --project-id <UUID>` で循環依存（例：A→B→A）を事前チェック。トポロジカルソートが失敗した場合はエラーを返す。また、`analyze`結果の`total_float`が0に近いタスク（クリティカルパス）は定期的に`deadliner alert --float-threshold 0.5` で監視し、フロートが0.5日を切ったら即座にエスカレーションする。
