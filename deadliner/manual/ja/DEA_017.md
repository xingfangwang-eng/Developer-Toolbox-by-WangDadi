# deadliner - DEA_017

## 概要

deadliner は、タスクの依存関係グラフをリアルタイムで解析し、納期超過リスクを確率論的に評価するエンタープライズ向け期限管理基盤です。PostgreSQL の LISTEN/NOTIFY による非同期待機型イベント駆動アーキテクチャを採用し、分散環境下でのタスク進捗をミリ秒単位で追跡。クリティカルパス上のボトルネックを特定し、プロジェクト全体の納期遵守率を動的に可視化します。

## コア機能

### 1. DAG ベース動的クリティカルパス再計算エンジン

タスク間の依存関係（`depends_on`）と推定残作業時間（`remaining_hours`）を入力として、有向非巡回グラフ（DAG）を構築。トポロジカルソートにより最長経路を特定し、各タスクの余裕日数（`slack_days`）をミリ秒精度で算出します。`slack_days` が負値になった場合、PostgreSQL の非同期トリガー（`NOTIFY deadline_critical`）を即座に発行。更新間隔は 5 秒、計算の時間計算量は O(V+E)（V: タスク数, E: 依存関係数）を保証。

**パラメータ**:
- `depends_on`: UUID 配列（最大 32 件）
- `remaining_hours`: float（0.1〜999.9）
- `slack_days`: float（ミリ秒単位で算出、負値許容）

### 2. IANA タイムゾーン絶対時刻同期モジュール

全タスクの開始・終了時刻を UTC で一元管理し、ユーザーごとの `timezone_offset`（±12:00）と `iana_timezone`（例: `Asia/Tokyo`）に基づき表示を変換。サマータイム（DST）の開始・終了ルールを内蔵し、年間スケジュールを自動補正。PostgreSQL の `TIMESTAMPTZ` 型と組み合わせることで、タイムゾーン起因の期限認識ズレを完全排除。

**パラメータ**:
- `timezone_offset`: string（例: `+09:00`）
- `iana_timezone`: string（例: `America/New_York`）
- 内部保持: TIMESTAMPTZ（マイクロ秒精度）

### 3. ベイズ推定による確率的期限超過予測モデル

過去のタスク完了実績（`completion_history` テーブル）から、タスク種別ごとの完了時間分布（正規分布近似）を学習。現在の進捗率（`progress_percentage`）と残時間（`remaining_seconds`）を入力とし、ベイズ推定により `overdue_probability`（0.0〜1.0）を出力。`overdue_probability` が 0.85 を超えた場合、`alert_log` テーブルにレコードを作成し、SMTP 経由でエスカレーションメールを自動送信。

**パラメータ**:
- `progress_percentage`: integer（0〜100）
- `remaining_seconds`: integer（0〜86400）
- `overdue_probability`: float（95%信頼区間付き）
- 学習データ: `completion_history`（最低 30 レコード必須）

## 3 つの詳細な操作手順

### 手順 1: タスク登録と依存関係グラフの構築

`POST /api/v1/tasks` エンドポイントに以下の JSON ペイロードを送信します。`deadline` は ISO 8601 形式（例: `2025-12-31T23:59:59Z`）、`depends_on` には先行タスクの UUID を配列で指定。PostgreSQL 側では `INSERT` トリガーが発火し、`task_dag` テーブルにエッジ情報が自動格納されます。登録後、`GET /api/v1/tasks/{uuid}/critical_path` でクリティカルパスを確認可能。

```json
{
  "title": "データベースマイグレーション完了",
  "deadline": "2025-12-31T23:59:59Z",
  "depends_on": ["uuid-of-prerequisite-task"],
  "remaining_hours": 8.5
}
```

### 手順 2: 進捗更新とリアルタイム WebSocket 同期

`PATCH /api/v1/tasks/{uuid}/progress` に `progress_percentage`（0〜100）と `remaining_seconds` を送信。サーバー側では、変更を PostgreSQL の `NOTIFY deadline_update` チャネルにパブリッシュし、全接続クライアントが即座に同期。同時に、クリティカルパス解析エンジンが再計算を実行し、`slack_days` が更新された場合は WebSocket 経由でプッシュ通知。更新間隔は 5 秒、レイテンシは 100ms 以内。

```json
{
  "progress_percentage": 65,
  "remaining_seconds": 7200
}
```

### 手順 3: 超過確率の確認とアラート閾値の調整

`GET /api/v1/tasks/{uuid}/risk` で、`overdue_probability` とその根拠となる `confidence_interval_95`（95%信頼区間）を取得。アラート閾値は環境変数 `OVERDUE_THRESHOLD`（デフォルト 0.85）で設定可能。閾値を超えた場合、`alert_log` テーブルにレコードが作成され、登録されたメールアドレスに SMTP 経由で自動通知。`confidence_interval_95` が 0.1 を超える場合は、学習データ不足の警告がログに出力されます。

## ベストプラクティス

### 1. タスク粒度の統一とサブタスク分割

1 タスクあたりの `remaining_hours` は 2〜40 時間に収めることを推奨。これを超えるタスクはサブタスクに分割し、`depends_on` で直列依存関係を明示することで、クリティカルパス解析の精度が向上。また、`completion_history` の学習データが均質化され、確率予測モデルのバイアスが低減。分割後は、`GET /api/v1/tasks/{uuid}/subtasks` でサブタスク一覧を確認可能。

### 2. 定期的なモデル再学習の自動化

`cron` ジョブで毎日午前 3 時に `REFRESH MATERIALIZED VIEW completion_stats` を実行し、`overdue_probability` 計算に用いる分布パラメータ（平均、標準偏差）を更新。これにより、プロジェクトの進行に伴うタスク特性の変化にモデルが追従し、予測精度が長期的に維持。`completion_history` テーブルに 30 レコード未満の場合は、モデル再学習はスキップされ、警告がログに出力。
