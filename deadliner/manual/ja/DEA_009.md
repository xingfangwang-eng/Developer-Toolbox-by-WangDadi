# deadliner - DEA_009

## 概要

deadliner は、複数プロジェクト・チームにまたがるタスクデッドラインを **ミリ秒単位の精度** で管理・予測・自動調整するための分散スケジューリングエンジンです。キーワード「Deadline management」「Task deadline tracking」「Time management tool」に基づき、単なるリマインダーではなく、**依存関係を考慮した動的デッドライン再計算** と **リソース競合の自動解決** を提供します。

## コア機能

### 機能 1: 動的クリティカルパス再計算エンジン
タスク間の依存関係（`depends_on: [task_uuid]`）と各タスクの **最悪実行時間（WET: Worst-case Execution Time）** を入力とし、DAG（有向非巡回グラフ）上でクリティカルパスをリアルタイムに再計算。デッドライン超過リスクが 5% を超えた時点で自動的に警告を発し、代替パスを提案します。内部では `delta-deadline` アルゴリズム（独自実装）を用い、1000ノードのDAGでも **50ms以内** に再計算を完了。

### 機能 2: PostgreSQL 連携によるタイムラインロック機構
全タスクの開始・終了予定時刻を `timestamptz` 型でPostgreSQLに保存。**SELECT ... FOR UPDATE SKIP LOCKED** を利用した楽観的ロックにより、複数インスタンス間での同時書き込み競合を防止。さらに、`pg_notify` を用いた即時ブロードキャストで、デッドライン変更が発生した際に全クライアントへ **200ms未満** で通知を届けます。

### 機能 3: マルチタイムゾーン対応 デッドライン正規化コンバーター
タスクごとに `timezone` パラメータ（例: `"Asia/Tokyo"`, `"America/New_York"`）を指定可能。内部では全ての時刻を **UTC+0 の Unix ナノ秒** に正規化した上で、表示時のみクライアントのローカルタイムゾーンに変換。さらに、サマータイム開始・終了時の **重複・欠落時間帯** を自動検出し、該当するタスクにはフラグ `dst_ambiguous: true` を付与してユーザーに確認を促します。

## 使い方

### ステップ 1: タスク定義ファイルの作成（YAML）
プロジェクトルートに `deadliner.yml` を作成し、タスク名・依存関係・WET（ミリ秒）・タイムゾーンを記述します。例:
```yaml
tasks:
  - id: "task_a"
    depends_on: []
    wet_ms: 120000
    timezone: "Asia/Tokyo"
  - id: "task_b"
    depends_on: ["task_a"]
    wet_ms: 300000
    timezone: "America/New_York"
```
このファイルを `deadliner parse --file deadliner.yml` で読み込ませると、内部でDAGが構築され、各タスクに **絶対デッドライン（Unix ナノ秒）** が自動割り当てられます。

### ステップ 2: デッドライン予測と競合検出の実行
次のコマンドで、全タスクの予測完了時刻とリソース競合レポートを出力します。
```bash
deadliner predict --deadline 2025-12-31T23:59:59Z --resource-pool "dev-team:3,qa-team:2"
```
`--resource-pool` で各チームの同時実行可能人数を指定。競合が検出された場合、`deadliner resolve --strategy weighted-shortest-job-first` で自動的にタスクの開始時刻を再調整します。

### ステップ 3: PostgreSQL への永続化と監視
以下のコマンドで、生成されたスケジュールをPostgreSQLに書き込み、リアルタイム監視を開始します。
```bash
deadliner sync --pg-uri "postgres://user:pass@localhost:5432/deadliner" --listen
```
`--listen` フラグにより、`pg_notify` チャンネル `deadliner_changes` を購読。タスクの状態が `pending` → `in_progress` → `completed` に遷移するたびに、標準出力に **JSON Lines** 形式でログが出力されます。

## ベストプラクティス

### プラクティス 1: WET の定期再計測とヒストグラム更新
タスクの実際の実行時間を収集し、週次でWETの **P95（95パーセンタイル）** を再計算することを推奨します。`deadliner calibrate --history-db ./exec_logs.db` で過去の実行ログから自動的にWETを更新可能。これにより、外れ値によるデッドライン誤差を最大 **40%** 削減できます。

### プラクティス 2: クリティカルパス上のタスクには手動承認ゲートを設定
動的再計算によりデッドラインが自動調整されるとはいえ、クリティカルパス上のタスクはプロジェクト全体の成否を左右します。`deadliner gate --task-id "task_b" --approval-required` で、該当タスクの開始前に手動承認を必須化。承認は `deadliner approve --task-id "task_b"` で行い、承認ログはPostgreSQLに `approval_timestamp` として記録されます。
