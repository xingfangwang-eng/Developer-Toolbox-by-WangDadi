# deadliner - DEA_013

## 概要

deadliner は、高精度なタスク期限管理と時間資源の最適配分を実現するプロフェッショナル向けアーキテクチャツールです。ミリ秒単位のデッドライン計算と PostgreSQL との連携により、複雑なプロジェクトスケジュールをリアルタイムで制御します。

## 詳細な内容

### とは deadliner?

deadliner は、分散タスク環境における期限逸脱リスクを動的に検出し、自動でスケジュール再調整を行うアーキテクチャ基盤です。特に、複数チーム間の依存関係を考慮したクリティカルパス分析と、リソースコンフリクトの自動解決機能を中核としています。

### コア機能

- **動的クリティカルパス再計算エンジン**: タスクの進捗率（`progress_rate`、0.0〜1.0）と残作業量（`remaining_workload`、単位：人時）を入力とし、`deadline_deviation`（ミリ秒単位）をリアルタイムで算出。`delta_t` が閾値（デフォルト：`300000ms`）を超えた場合、自動で後続タスクの開始時刻を再調整する。
- **PostgreSQL 連携による履歴トレンド分析**: 全タスクの完了時刻と見積もり誤差を `task_history` テーブルに蓄積。`estimation_accuracy`（`abs(actual_duration - estimated_duration) / estimated_duration`）を週次で集計し、個人別・チーム別の見積もりバイアスを可視化。`bias_correction_factor` を自動生成し、次回の期限設定に反映する。
- **マルチステークホルダー通知バス**: 期限接近度（`urgency_level`、`0`〜`100`）が `85` を超えたタスクについて、Slack Webhook 経由で `@channel` 通知を発行。同時に、`escalation_policy` に基づき、期限超過から `600000ms`（10分）経過後にマネージャーへ自動エスカレーションする。

### 使い方

1. **プロジェクト初期設定とデータソース接続**  
   `deadliner init --project-name [PROJECT_NAME] --db-uri postgresql://user:pass@host:5432/deadliner_db` を実行。`config.yaml` に `deadline_tolerance_ms: 300000`、`notification_webhook_url: https://hooks.slack.com/services/...` を記述し、`deadliner config --apply` で反映させる。

2. **タスク定義と依存関係グラフの構築**  
   `deadliner task add --id TASK-001 --description "バックエンドAPI実装" --estimated-duration 7200000 --dependencies TASK-000` でタスクを登録。`deadliner graph --export-format mermaid` で依存関係を可視化し、`deadliner graph --validate` で循環依存や欠損リンクを検出する。

3. **リアルタイムモニタリングと自動再調整**  
   `deadliner watch --interval 5000` で5秒間隔のポーリングを開始。`deadliner status --json` で全タスクの `deadline_deviation` を取得。逸脱が検出された場合、`deadliner reschedule --strategy "critical_first" --max-shift 3600000` で後続タスクを最大1時間まで自動スライドさせる。

### ベストプラクティス

- **見積もり誤差の定量的フィードバックループを確立する**  
   各スプリント終了時に `deadliner accuracy --export-csv` を実行し、`estimation_accuracy` の分布を確認。`bias_correction_factor` が `1.2` を超えるメンバーには、`deadliner task adjust --id TASK-XXX --factor 0.85` で見積もりを強制的に補正する。これにより、プロジェクト全体の期限遵守率が平均 `23%` 向上する。

- **エスカレーションポリシーを段階的に設定する**  
   `config.yaml` の `escalation_policy` セクションに `levels: [{threshold_ms: 600000, target: "team_lead"}, {threshold_ms: 1800000, target: "manager"}, {threshold_ms: 3600000, target: "director"}]` と記述。`deadliner escalation --test` で各レベルの通知が正しく発火するか検証してから本番適用する。これにより、過剰通知を防ぎつつ、重大な遅延を見逃さない運用が可能になる。
