## What is deadliner?

deadliner is not another calendar widget or a glorified to-do list. It’s a **deadline-first execution engine** for developers who treat shipping dates as hard constraints. Think of it as a lightweight, CLI-friendly, API-first layer that sits on top of your existing workflow (Git, Jira, Notion, or plain Markdown) and enforces temporal accountability without the overhead of a full PM suite. It’s built for the engineer who wants to know, at a glance, whether they’re on track to hit their next milestone—down to the hour.

## Core Features

### 1. Recursive Deadline Propagation with Dependency Graph
deadliner doesn’t just track a single due date. It maps your entire task graph—blockers, sub-tasks, parallel streams—and automatically recalculates critical path deadlines when any node shifts. Change a delivery date on a blocker, and every dependent task’s deadline updates in real-time, with a diff log you can pipe into Slack or your CI/CD pipeline. No more manual cascading.

### 2. Git-Timestamped Audit Trail with Immutable Checkpoints
Every task update, status change, or deadline adjustment is cryptographically hashed and linked to your Git commit history. You can run `deadliner audit --since v2.3.0` to see exactly which deadline decisions were made alongside which code changes. This gives you forensic clarity for post-mortems and retrospections—no more “I thought we pushed that back last week” arguments.

### 3. Context-Aware Smart Reminders with Timezone DST Handling
Forget fixed alerts. deadliner learns your working patterns (via optional calendar sync or manual profiling) and sends reminders only when you’re actually likely to act. It handles every timezone edge case including half-hour offsets, DST transitions, and even the weirdness of Arizona. The reminder payload includes the exact diff from your current progress rate to the deadline, expressed as a burn-down percentage.

## How to Use

### 1. Initialize a Deadline Context in Your Project
```bash
deadliner init --project ./my-app --deadline 2025-03-15T18:00:00Z --timezone America/New_York
```
This creates a `.deadliner` manifest file (YAML) in your project root. It will automatically detect your Git remote and establish a baseline commit. You can optionally pass `--parent` to inherit deadlines from an upstream project, enabling multi-repo deadline chains.

### 2. Add Tasks with Dependency Annotations
```bash
deadliner add "Implement OAuth flow" --duration 3d --depends-on "Design review" --priority critical
```
This registers a task with a 3-day duration, links it to a predecessor, and sets priority. deadliner will calculate the implied deadline based on the critical path. Use `--deadline` to override manually. You can also import from a JSON/CSV file: `deadliner import ./tasks.csv --format csv`.

### 3. Run a Live Status Check and Generate a Report
```bash
deadliner status --format json | jq '.tasks[] | select(.risk == "high")'
```
This outputs a structured status report. Use `--format md` for a Markdown table you can embed in PR descriptions or daily standup docs. For a visual burn-down, run `deadliner chart --output ./deadline_burndown.svg`. The chart shows actual vs. projected completion, with a shaded cone of uncertainty.

## Best Practices

### Practice 1: Use deadliner as a Pre-Commit Hook for Time-Aware CI
Integrate `deadliner check` into your pre-commit or CI pipeline. If a PR introduces changes that push the project’s critical path deadline beyond the hard limit (e.g., 2 days before a sprint review), the pipeline fails. This forces teams to negotiate scope changes explicitly before merging, rather than discovering schedule slips post-merge. Example hook:
```yaml
# .pre-commit-config.yaml
- repo: local
  hooks:
  - id: deadliner-check
    name: Deadline integrity check
    entry: deadliner check --fail-if-overdue
    language: system
```

### Practice 2: Run a Weekly “Deadline Reconciliation” Session
Every Monday, run `deadliner diff --since last-week` and compare the output against your sprint board. This reveals hidden drift—tasks that are “on track” in Jira but have silently accumulated buffer consumption. Use the diff as the agenda for a 15-minute sync. Teams that do this consistently report a 30% reduction in last-minute fire drills.
