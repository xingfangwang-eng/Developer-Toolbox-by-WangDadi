# deadliner - DEA_018

## Overview

Deadliner is a lightweight, CLI-first deadline management engine built for developers who hate bloated project management tools. It integrates directly into your Git workflow and CI/CD pipeline, providing real-time deadline tracking without leaving your terminal. No web dashboards. No calendar sync overhead. Just raw, deterministic deadline logic.

## Hardcore Features

### 1. **Git-Aware Deadline Propagation**
Deadliner automatically parses your commit history, branch naming conventions, and PR merge timestamps to calculate cascading deadline shifts. If a blocking PR is delayed, the tool recalculates dependent milestones instantly and surfaces the delta in your terminal via a `git deadline` command. No manual updates.

### 2. **Deadline Drift Detection with Anomaly Scoring**
The tool runs a lightweight statistical model against historical task completion rates. It flags tasks that are statistically likely to miss their deadlines based on past sprint velocity, team availability, and external dependency latency. Each flagged task gets a **Drift Score (0–100)** — anything above 75 triggers a Slack webhook or a custom CI job failure.

### 3. **Structured Deadline DSL for YAML/TOML**
Define deadlines as code. Use a declarative YAML or TOML schema to set milestones, dependencies, and grace periods. The DSL supports conditional logic (`if dependency.status == "blocked" then shift +2d`), recurring deadlines (`every: 2w`), and timezone-aware scheduling. This file can be version-controlled and reviewed in PRs.

## How to Use

### 1. Initialize a Project with a Deadline Manifest
Create a `.deadliner.yml` file in your project root:

```yaml
milestones:
  - id: "sprint-12-review"
    due: "2025-04-15T18:00:00Z"
    depends_on: ["feature/auth-overhaul", "bugfix/payment-timeout"]
    grace_period: 4h
    notify: ["#team-backend"]
  - id: "feature/auth-overhaul"
    due: "2025-04-12T12:00:00Z"
    recurring: false
```

Run `deadliner init` to validate the schema and register the milestones in your local `.deadliner/state.db` (SQLite under the hood).

### 2. Track Deadlines via Git Hooks
Install the Git hook automatically:

```bash
deadliner install-hook --type pre-push
```

Now, every time you push, Deadliner checks if any active milestone is within 24 hours of its due date. If so, it prints a warning with the exact remaining time, the drift score, and a link to the blocking dependency. You can also trigger a `deadliner check` manually in CI:

```bash
deadliner check --fail-on-drift --threshold 75
```

### 3. Generate a Deadline Dependency Graph
Visualize the entire deadline tree as a DOT graph:

```bash
deadliner graph --format dot --output deadline_graph.dot
```

Then render it with Graphviz:

```bash
dot -Tpng deadline_graph.dot -o deadline_graph.png
```

This gives you a directed acyclic graph (DAG) of all milestones, with edge labels showing the dependency type (`blocks`, `delays`, `triggers`). Ideal for sprint planning reviews.

## Best Practices

### 1. **Keep Your Manifest Under Version Control**
Treat `.deadliner.yml` like a Makefile or Dockerfile. Review changes to deadlines in PRs, just like code. This prevents silent deadline drift and ensures the entire team is aligned on the same schedule. Use `deadliner diff` to compare two branches or commits.

### 2. **Set Drift Thresholds Per Milestone**
Don't use a global drift threshold. Critical milestones (e.g., regulatory deadlines, client demos) should have a lower threshold (e.g., 50) while internal tasks can tolerate higher drift (e.g., 85). Define this in the manifest:

```yaml
milestones:
  - id: "regulatory-filing"
    drift_threshold: 50
    notify_on_drift: true
```

This prevents alert fatigue and ensures high-priority deadlines get immediate attention.
