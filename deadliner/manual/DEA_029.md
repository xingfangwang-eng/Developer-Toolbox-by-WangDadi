# deadliner - DEA_029

**deadliner** is not another bloated calendar app. It’s a lightweight, CLI-first, API-ready deadline engine built for developers who need to track, alert, and optimize task timelines without leaving their terminal or CI/CD pipeline. Think `cron` meets `git log` with a zero-friction UX.

## 🚀 3 Hardcore Features That Set deadliner Apart

### 1. **Declarative Deadline DSL via `.deadliner.yml`**
Define deadlines as code. Use YAML to set relative/absolute deadlines, recurrence rules (ISO 8601 RRULE), and conditional triggers (e.g., "if PR merged, shift deadline by 2d"). deadliner parses the DSL and syncs with your repo’s commit history to auto-adjust timelines.

### 2. **Git-Aware Blocker Detection**
deadliner hooks into your Git hooks (`pre-push`, `post-commit`) and cross-references open issues, unmerged branches, and stale PRs. If a deadline is at risk due to unresolved blockers, it surfaces them with direct links and estimated delay impact — no manual status meetings required.

### 3. **Webhook Relay & Slack/Teams/Telegram Push**
Instead of polling, deadliner fires JSON payloads to your custom webhooks or built-in integrations. Set up `on: deadline:missed` or `on: deadline:resolved` actions to trigger CI jobs, auto-assign tasks, or post to your team’s channel with context (commit hash, assignee, delta).

## 🛠️ How to Use deadliner in 3 Steps

### Step 1: Initialize a Project Timeline
```bash
deadliner init --project "Q3 Launch" --deadline "2025-09-30T18:00:00Z" \
  --recurrence "FREQ=WEEKLY;BYDAY=MO,WE,FR" \
  --blockers "issue:123,PR:456"
```
This creates a `.deadliner.yml` and registers a local `.deadliner.db` (SQLite). deadliner will start tracking commits against these milestones from the current branch.

### Step 2: Integrate with Your CI/CD
Add a pre-deploy check in your pipeline:
```yaml
# .github/workflows/deadliner-check.yml
- name: Check deadline health
  run: |
    deadliner check --branch ${{ github.head_ref }} --output json
    deadliner alert --webhook ${{ secrets.DEADLINER_WEBHOOK }}
```
If the check fails (e.g., deadline passed or blockers unresolved), the pipeline can fail or warn — your call. deadliner exits with code 1 on missed deadlines.

### Step 3: Query & Visualize with SQL
deadliner stores all events in a local SQLite DB. You can query it directly:
```sql
SELECT project, deadline, status, last_commit_sha
FROM deadlines
WHERE status = 'at_risk' AND assignee = 'maria';
```
Or use the built-in `deadliner stats --format table` for a terminal dashboard. Export to CSV/JSON for Notion or Grafana.

## 📈 2 Industry Best Practices for deadliner

### Practice 1: **Treat Deadlines as Code, Not Calendar Events**
Store your `.deadliner.yml` in the repo root alongside `Dockerfile` and `Makefile`. This makes deadlines versioned, reviewable, and auditable — just like any other config. When a deadline changes, it’s a PR, not a Slack message.

### Practice 2: **Use Blocker Detection as a Retrospective Signal**
After a sprint, run `deadliner audit --range "2 weeks ago"` to see which deadlines were missed due to blockers vs. poor estimation. Feed this data into your sprint retro to identify systemic bottlenecks (e.g., “PRs blocked by code review > 48h”). deadliner’s output can be piped directly into Jira or Linear.
