## Detailed Content

### What is deadliner?

deadliner is a lean, CLI-first deadline orchestration engine built for engineers who hate context switching. It ingests task metadata from Markdown, JSON, or YAML, computes critical paths and slack windows in real-time, and surfaces exactly what needs attention—without the noise of traditional Gantt charts or bloated project tools.

### Core Features

- **Dynamic Critical Path Recalculation** – Automatically recomputes the longest path and slack time every time a task status or dependency changes. No manual refresh, no stale views. Built on a lightweight DAG engine with O(n + m) topological sort under the hood.

- **Git-Aware Timeline Anchoring** – Automatically syncs task deadlines with Git commit timestamps and branch activity. If a PR for a dependent task is merged late, deadliner adjusts the downstream deadlines and flags the drift in real-time.

- **Zero-Dependency Markdown DSL** – Define tasks, dependencies, and deadlines using a simple, human-readable syntax in your existing `.md` files. No database, no API, no cloud lock-in. Just a single binary that parses, validates, and visualizes your timeline.

### How to Use

1. **Initialize a project timeline from your existing Markdown**  
   Run `deadliner init --from ./tasks.md` to scan all Markdown files for task definitions. The tool auto-detects headers with `deadline:` and `depends_on:` metadata, builds a dependency graph, and outputs a JSON timeline. Use `--format dot` to export a Graphviz graph for visual review.

2. **Track drift with Git-aware timeline snapshots**  
   Execute `deadliner check --git-ref HEAD~3` to compare the current timeline against a historical Git reference. deadliner diffs the computed deadlines, highlights tasks that shifted beyond a configurable threshold (default: 4 hours), and outputs a diff report in JSON or terminal-friendly table format. Integrate this into CI with `deadliner ci --alert-on-drift`.

3. **Visualize your critical path in the terminal**  
   Run `deadliner viz --critical-only --output terminal` to render a color-coded ASCII Gantt chart directly in your terminal. Tasks on the critical path are highlighted in red, slack windows shown in dimmed gray. Use `--export svg` for documentation or `--export png` for sharing.

### Best Practices

- **Use semantic versioning for your timeline files** – Treat your `deadliner.yaml` or `tasks.md` as a first-class artifact. Version it alongside your code, and run `deadliner check` in CI to catch timeline drift before it reaches production. This prevents silent deadline creep in distributed teams.

- **Anchor deadlines to real events, not arbitrary dates** – Instead of "deadline: 2025-04-15", use `deadline: release/v2.3+2d` to bind the deadline to a Git tag or branch creation date. This makes timelines resilient to schedule shifts and keeps your DAG aligned with actual development cadence.
