# deadliner - DEA_022

## Overview

deadliner isn't just another todo list or calendar widget. It's a lightweight, event-driven deadline management engine built for developers who need to track, visualize, and enforce time-sensitive tasks across distributed workflows. Think of it as a cron scheduler with a human-readable UI, CI/CD hooks, and real-time escalation logic. No fluff, just deterministic deadline execution.

## Core Features

### 1. Deadline Chain with Automatic Escalation
deadliner supports recursive deadline chaining: if Task A depends on Task B, and B slips, A's deadline is automatically recalculated using a configurable buffer (e.g., `+2h` or `+1d`). Escalation rules trigger Webhooks, Slack DMs, or PagerDuty alerts when a deadline is breached by more than 5% of the original duration. No manual status checks needed.

### 2. Git-Aware Time Tracking
Integrates directly with GitHub, GitLab, and Bitbucket. deadliner parses commit timestamps, PR merge times, and issue close dates to auto-update task progress. If a PR is merged 3 hours before a deadline, the task is marked as "completed" with a timestamp delta. If the PR is late, the system automatically logs a latency metric and adjusts the next dependent deadline.

### 3. Deadline as Code (DaC)
Define deadlines in YAML or JSON, commit them to your repo, and let deadliner parse them via a CLI or API. Supports timezone-aware expressions (e.g., `"2025-06-15T14:00:00Z"` or `"next Friday 17:00 PST"`). The engine validates syntax, prevents overlapping deadlines, and outputs a dependency graph in DOT or Mermaid format. Perfect for infrastructure-as-code enthusiasts.

## How to Use

### 1. Initialize a Deadline File
Create a `.deadliner.yml` in your project root:
```yaml
tasks:
  - id: "api-v2-launch"
    deadline: "2025-07-01T09:00:00Z"
    depends_on: ["auth-refactor", "docs-v2"]
    buffer: "4h"
    escalation:
      - type: "slack"
        channel: "#dev-alerts"
        threshold: "0.8"  # 80% of deadline elapsed
```
Run `deadliner init` to validate syntax and generate a dependency graph. The engine will output any circular dependency warnings immediately.

### 2. Start the Daemon in Watch Mode
Execute `deadliner daemon --watch --interval 60` to run the engine as a background process. It polls Git status every 60 seconds, recalculates deadlines based on new commits or PR merges, and triggers escalations if thresholds are crossed. The daemon exposes a REST API at `localhost:9090` for real-time status queries.

### 3. Query and Visualize the Deadline State
Use the CLI to get a snapshot:
```bash
deadliner status --format json --filter "status:overdue"
```
Or generate an SVG Gantt chart:
```bash
deadliner visualize --output chart.svg --timescale "hours"
```
This renders all tasks with their current progress, remaining time, and dependency arrows. The SVG can be embedded in CI dashboards or Notion pages.

## Best Practices

### 1. Enforce Deadline Budgets in CI/CD
Integrate deadliner's validation step into your pre-merge pipeline. Before a PR is merged, run `deadliner check --task-id <id>` to verify that the proposed changes don't push any dependent deadlines beyond their buffer. If they do, the pipeline fails with a detailed reason (e.g., "Task 'api-v2-launch' would be delayed by 3h due to this PR"). This prevents scope creep from silently destroying your timeline.

### 2. Use Timezone-Normalized Deadlines for Distributed Teams
Always define deadlines in UTC and let deadliner convert to local time for each team member via their profile settings. This avoids the classic "meeting at 9 AM" confusion across timezones. deadliner also logs conversion mismatches (e.g., if a user's local time zone changes mid-sprint) and flags them in the escalation report. For remote-first teams, this is a game-changer.
