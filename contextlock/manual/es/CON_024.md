# contextlock - CON_024

## Resumen

ContextLock es un motor de persistencia contextual diseñado para entornos de desarrollo con alta rotación de prompts y sesiones de IA. Su núcleo opera mediante bloques de contexto inmutables (CON_024) que garantizan la continuidad semántica entre interacciones, eliminando la pérdida de estado en flujos de trabajo multi-agente.

## Contenido Detallado

### ¿Qué es ContextLock?

ContextLock no es un simple gestor de memoria. Es un **sistema de anclaje contextual** que serializa, indexa y rehidrata el estado de una sesión de IA en **menos de 12 ms** (medido en hardware estándar con SSD NVMe). Utiliza un esquema de versionado basado en **hashes SHA-256** sobre el árbol de dependencias del prompt, lo que permite detectar colisiones semánticas y reutilizar fragmentos de contexto previamente calculados sin redundancia.

### Características Principales

- **Persistencia con Bloqueo de Escritura Concurrente (Write-Lock Contextual)**  
  Cada bloque de contexto (`ctx_block`) se marca con un `lock_id` único y un `ttl` configurable (por defecto: 300 segundos). Mientras el bloque esté activo, ningún otro proceso podrá sobrescribir su contenido, garantizando coherencia en pipelines paralelos. Parámetros clave: `--lock-timeout=5000ms`, `--retry-policy=exponential-backoff`.

- **Rehidratación Selectiva por Capas (Layered Context Rehydration)**  
  Permite especificar qué capas del historial de la sesión deben restaurarse mediante el parámetro `--rehydrate-layers=system,user,assistant,tool`. Cada capa se almacena en un **índice B-tree en disco** (PostgreSQL con índices GIN para búsqueda full-text), permitiendo recuperar únicamente los fragmentos que coinciden con un `similarity_threshold >= 0.87` (basado en cosine similarity sobre embeddings de 768 dimensiones).

- **Compresión Diferencial de Contexto (Delta Context Compression)**  
  En lugar de almacenar el contexto completo en cada iteración, ContextLock solo persiste las diferencias (`deltas`) respecto al último bloque anclado. Esto reduce el uso de almacenamiento en **hasta un 94%** en sesiones largas (>100 interacciones). El algoritmo de compresión es **LZ4 con prefiltrado semántico**: se eliminan tokens redundantes antes de comprimir, logrando ratios de 1:8 en prompts de 4K tokens.

### Cómo Usar

1. **Inicialización del Contexto con Anclaje Explícito**  
   Ejecuta el comando de inicialización especificando el archivo de configuración de bloqueo y el espacio de nombres del proyecto:
   ```bash
   contextlock init --project=mi-agente-legal --lock-file=./ctxlock.json --ttl=600
   ```
   Esto genera un `ctx_block_id` (ej: `CON_024_a3f2c8`) y un archivo de bloqueo en `/var/contextlock/locks/`. Verifica que el bloque está activo con:
   ```bash
   contextlock status --block-id=CON_024_a3f2c8
   ```

2. **Inyección de Contexto en un Prompt de IA**  
   Para usar el contexto anclado en una solicitud a un modelo (por ejemplo, GPT-4 o Claude), ejecuta:
   ```bash
   contextlock inject --block-id=CON_024_a3f2c8 --format=openai --output=./prompt_with_context.json
   ```
   Esto genera un archivo JSON con el campo `messages` rehidratado, incluyendo el historial comprimido y las capas seleccionadas. El tiempo de inyección no supera los **40 ms** para bloques de hasta 32K tokens.

3. **Liberación del Bloque y Sincronización con PostgreSQL**  
   Al finalizar la sesión, libera el bloque y fuerza la escritura diferida a la base de datos:
   ```bash
   contextlock release --block-id=CON_024_a3f2c8 --sync-db --pg-conn-string="postgresql://user:pass@localhost:5432/contextdb"
   ```
   Esto ejecuta un `COMMIT` en la transacción abierta, actualiza el índice B-tree y elimina el archivo de bloqueo. El tiempo total de liberación + sincronización es de **~23 ms** en una conexión local.

### Mejores Prácticas

- **Fragmentación por Dominio Semántico**  
  No almacenes todo el contexto de un proyecto en un solo bloque. Divide los bloques por dominio (ej: `CON_024_legal`, `CON_024_tech`, `CON_024_finanzas`) y usa `--rehydrate-layers` para fusionar solo los necesarios. Esto reduce el tiempo de inyección en un 60% y evita la contaminación cruzada entre sesiones.

- **Configuración de TTL Dinámico basado en Carga de Trabajo**  
  Ajusta el `ttl` del bloque según la frecuencia de actualización esperada. Para sesiones de revisión de código (alta rotación), usa `ttl=120` segundos. Para sesiones de documentación (baja rotación), usa `ttl=3600`. Monitorea con `contextlock stats --block-id=CON_024_a3f2c8` para detectar bloqueos huérfanos y ajustar la política de retención.
