# contextlock - CON_015

## Resumen

ContextLock es una herramienta de gestión de contexto persistente para desarrolladores que trabajan con asistentes de IA. Diseñada para eliminar la pérdida de memoria contextual en sesiones largas, garantiza que cada interacción con modelos como GPT-4, Claude o Gemini mantenga el estado completo del proyecto, sin importar cuántas veces se reinicie el flujo de trabajo.

## Contenido Detallado

### ¿Qué es ContextLock?

ContextLock no es un simple gestor de prompts. Es un middleware de capa de estado que serializa, versiona y reinyecta el contexto de tu sesión de desarrollo en tiempo real. Si trabajas con arquitecturas de microservicios, pipelines de CI/CD o debugging multicapa, sabes que el mayor cuello de botella no es el modelo, sino la pérdida de información entre turnos. ContextLock resuelve eso con un sistema de bloques de contexto inmutables, checksums criptográficos y rehidratación automática.

### Características Principales

1. **Contexto Versionado con Bloqueo de Estado (State Locking)**: Cada fragmento de contexto se almacena como un bloque inmutable con un hash SHA-256. Puedes "bloquear" un estado específico (por ejemplo, la configuración de tu pipeline de CI) y rehidratarlo en cualquier momento, incluso después de 50 iteraciones de debugging. El sistema detecta automáticamente desviaciones y te alerta si el contexto ha sido modificado externamente.

2. **Inyección Contextual Diferencial (Diff Injection)**: En lugar de reenviar todo el contexto en cada llamada a la API, ContextLock calcula la diferencia entre el estado actual y el último estado conocido. Solo envía los cambios, reduciendo el consumo de tokens hasta en un 70% en sesiones largas. Ideal para equipos que trabajan con modelos de pago por token.

3. **Persistencia Híbrida Local-Remota con Sincronización Asíncrona**: El contexto se almacena localmente en un archivo `.contextlock` en el directorio del proyecto, pero también se sincroniza de forma asíncrona con un bucket S3 o un servidor Redis. Esto permite que múltiples desarrolladores compartan el mismo contexto sin colisiones, y que el contexto sobreviva a un `git reset --hard` o un cambio de rama.

### Cómo Usar

1. **Inicialización y Enlace al Proyecto**:
   ```bash
   contextlock init --project mi-proyecto --provider openai --model gpt-4-turbo
   ```
   Esto genera un archivo `.contextlock` en la raíz del proyecto. Luego, enlaza el contexto a tu sesión de desarrollo con:
   ```bash
   contextlock attach --session-id debug-pipeline-01
   ```
   El sistema crea automáticamente un snapshot del estado actual de tu terminal, variables de entorno y archivos de configuración relevantes.

2. **Captura y Bloqueo de Contexto Crítico**:
   Durante una sesión de debugging, ejecuta:
   ```bash
   contextlock capture --label "config-db-prod" --lock
   ```
   Esto captura el contexto actual (incluyendo logs, variables de entorno, y el historial de la conversación con la IA) y lo bloquea. Cualquier intento de modificar ese bloque generará una advertencia. Para rehidratarlo después de un reinicio:
   ```bash
   contextlock restore --label "config-db-prod"
   ```

3. **Inyección Diferencial Automática en Pipeline de CI**:
   Integra ContextLock en tu pipeline de GitHub Actions o GitLab CI:
   ```yaml
   - name: Inject context diff
     run: |
       contextlock diff --from latest --to current --output context_diff.json
       contextlock inject --file context_diff.json --target $OPENAI_API_ENDPOINT
   ```
   Esto permite que tu asistente de IA en CI reciba solo los cambios desde la última ejecución, reduciendo costos y mejorando la precisión de las respuestas.

### Mejores Prácticas

1. **Usa etiquetas semánticas para el bloqueo de contexto**: En lugar de etiquetas genéricas como "debug-1", usa nombres como `config-staging-v2` o `error-handler-auth`. Esto facilita la rehidratación en equipos grandes y evita colisiones. Además, combina el bloqueo con un `--ttl` (time-to-live) para que los contextos expiren automáticamente después de 24 horas si no se usan.

2. **Versiona el archivo `.contextlock` en tu repositorio, pero con cuidado**: Incluye el archivo en el control de versiones solo si contiene contexto no sensible (como configuraciones de desarrollo o plantillas de prompts). Para contextos con secretos (API keys, tokens), usa el flag `--encrypt` y almacena la clave de cifrado en un gestor de secretos como HashiCorp Vault o AWS Secrets Manager. Nunca subas contextos cifrados sin documentar la ubicación de la clave.
