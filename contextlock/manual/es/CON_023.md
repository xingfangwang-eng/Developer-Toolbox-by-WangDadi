# contextlock - CON_023

## Resumen

ContextLock es un sistema de gestión de contexto persistente para asistentes de IA, diseñado para desarrolladores que necesitan mantener estado coherente entre sesiones de trabajo, ejecuciones de pipelines y cambios de rama. Elimina la pérdida de contexto entre prompts y garantiza que cada interacción con la IA retenga el historial completo de decisiones técnicas, configuraciones de entorno y dependencias activas.

## Contenido Detallado

### ¿Qué es ContextLock?

ContextLock es un motor de memoria contextual con checkpointing en tiempo real. Permite que herramientas como Claude, GPT o Copilot recuerden no solo el último prompt, sino el estado completo del proyecto: variables de entorno, archivos modificados, resultados de tests fallidos, decisiones de arquitectura y hasta logs de errores previos. Funciona como un middleware que intercepta la salida de cada comando y la almacena en una base de datos PostgreSQL local, indexada por hash de proyecto y timestamp de sesión. La latencia de escritura es inferior a 2 ms por evento, y la recuperación de contexto completo no supera los 50 ms incluso con 10.000+ registros.

### Características Principales

- **Memoria de Sesión con Checkpointing Automático**: Cada 5 segundos o cada 3 comandos ejecutados (lo que ocurra primero), ContextLock genera un snapshot delta del contexto actual. Los snapshots se almacenan en una tabla `context_snapshots` con columnas `project_hash`, `session_id`, `timestamp (TIMESTAMPTZ)`, `delta (JSONB)`, `parent_snapshot_id (UUID)`. La restauración de un snapshot específico se realiza mediante una consulta recursiva CTE que reconstruye el estado completo en menos de 10 ms.

- **Vinculación de Contexto por Rama Git**: ContextLock detecta automáticamente la rama activa mediante `git rev-parse --abbrev-ref HEAD` y asocia cada sesión a esa rama. Si cambias de rama, el contexto se aísla automáticamente. Puedes forzar un merge de contexto entre ramas usando `contextlock merge --source feature/foo --target main --strategy=union`, que combina los contextos priorizando el más reciente por timestamp. El comando `contextlock diff --branch feature/foo` muestra las diferencias de contexto entre ramas en formato JSON patch.

- **Exportación de Contexto a Plantillas de Prompt**: ContextLock puede generar un bloque de contexto formateado listo para pegar en cualquier prompt de IA. El comando `contextlock export --format markdown --include-vars --include-logs --max-entries 50` produce un documento con secciones `## Variables de Entorno`, `## Últimos Comandos Ejecutados`, `## Resultados de Tests`, `## Decisiones de Arquitectura`. Cada entrada incluye timestamp, duración (en milisegundos), y un enlace al snapshot correspondiente. El formato también soporta `--format json` para integración con pipelines CI/CD.

### Cómo Usar

1. **Inicializar ContextLock en un proyecto existente**:
   - Ejecuta `contextlock init --project-dir . --db-uri postgresql://user:pass@localhost:5432/contextlock --auto-detect-git`
   - ContextLock creará la tabla `context_events` y `context_snapshots` en PostgreSQL, y generará un archivo `.contextlock/config.yml` con `poll_interval_ms: 5000`, `max_snapshots: 1000`, `compression: zstd`.
   - Verifica la conexión con `contextlock status --verbose`, que debe devolver `DB connected`, `Git detected: true`, `Branch: main`, `Snapshots: 0`.

2. **Registrar un evento de contexto manualmente**:
   - Después de ejecutar un comando crítico (ej. `npm run build --production`), usa `contextlock log --event-type command --payload '{"cmd":"npm run build --production","exit_code":0,"duration_ms":1234}' --tags build,production`
   - ContextLock almacenará el evento en `context_events` con un `event_id (UUID v4)`, `session_id` actual, `timestamp` automático, y `payload` en JSONB.
   - Para ver el evento recién creado: `contextlock get --event-id <UUID> --format json | jq .`

3. **Restaurar contexto desde un snapshot específico**:
   - Lista los snapshots disponibles: `contextlock list --snapshots --limit 5 --order-by timestamp DESC`
   - Elige un snapshot por su `snapshot_id` y restaura: `contextlock restore --snapshot-id <UUID> --output-file context_restore.json`
   - ContextLock reconstruirá el estado completo en un archivo JSON con estructura `{"snapshot_id": "...", "parent_id": "...", "delta": {...}, "full_state": {...}}`.
   - Para cargar ese contexto directamente en un prompt de Claude: `cat context_restore.json | contextlock export --from-stdin --format markdown --include-full-state | pbcopy`

### Mejores Prácticas

1. **Integración con hooks de Git para captura automática de contexto**: Configura un hook `post-commit` que ejecute `contextlock log --event-type git-commit --payload '{"message":"$COMMIT_MSG","files_changed":$FILES_CHANGED}' --tags commit.` Esto garantiza que cada commit quede registrado con su contexto completo. Además, en el hook `pre-push`, usa `contextlock snapshot --force` para crear un checkpoint antes de enviar cambios al remoto. Esto permite revertir el contexto si un push rompe algo.

2. **Uso de ContextLock en pipelines CI/CD para depuración post-mortem**: Al final de cada pipeline, ejecuta `contextlock export --format json --include-logs --include-vars --max-entries 200 > context_$CI_PIPELINE_ID.json` y súbelo como artefacto. Cuando un pipeline falla, los desarrolladores pueden descargar ese archivo y restaurar el contexto localmente con `contextlock import --file context_$CI_PIPELINE_ID.json`. Esto elimina la necesidad de reproducir errores manualmente, ya que el contexto incluye variables de entorno exactas, comandos ejecutados y sus salidas.
