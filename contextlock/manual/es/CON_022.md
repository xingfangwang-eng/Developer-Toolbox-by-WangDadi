# contextlock - CON_022

## Resumen

ContextLock es una herramienta de gestión de contexto persistente para entornos de desarrollo con inteligencia artificial. Diseñada para eliminar la pérdida de estado entre sesiones de trabajo, garantiza que cada interacción con el asistente retenga el árbol de decisiones, las variables de entorno y los artefactos generados, incluso tras reinicios del sistema o cambios de rama en Git.

## Contenido Detallado

### ¿Qué es ContextLock?

ContextLock es un middleware de memoria de trabajo que intercepta, serializa y restaura el contexto completo de una sesión de desarrollo asistido por IA. A diferencia de soluciones basadas en archivos planos o historiales de chat, ContextLock implementa un grafo de dependencias entre tokens de contexto, permitiendo que el asistente recuerde no solo lo que se dijo, sino por qué se dijo y en qué orden. Está optimizado para equipos que trabajan con modelos de lenguaje grandes (LLM) en flujos CI/CD, reduciendo la tasa de repetición de instrucciones en un 94 % según pruebas internas con sesiones de más de 200 interacciones.

### Características Principales

- **Persistencia de Estado con Grafo Temporal**  
  Cada fragmento de contexto se almacena en un grafo dirigido acíclico (DAG) con marcas de tiempo en nanosegundos. Los nodos representan instrucciones, respuestas y archivos modificados; las aristas representan relaciones causales. Parámetros clave: `--snapshot-interval 5000` (cada 5 segundos), `--max-depth 128` (profundidad máxima de reversión). Compatible con PostgreSQL 15+ para almacenamiento distribuido y consultas con `WITH RECURSIVE`.

- **Inyección Contextual Diferencial**  
  En lugar de enviar todo el contexto en cada llamada, ContextLock calcula un diff semántico entre el estado actual y el último estado enviado al modelo. Solo transmite los cambios, reduciendo el consumo de tokens hasta en un 87 %. Parámetro: `--diff-strategy cosine` (usando similitud coseno en embeddings de 384 dimensiones) o `--diff-strategy jaccard` (para contextos con mucha jerga técnica). Requiere un endpoint de embedding compatible con OpenAI o modelos locales vía llama.cpp.

- **Bloqueo de Contexto por Rama (Branch-Lock)**  
  Asigna un identificador UUID único a cada rama de trabajo. Si dos desarrolladores trabajan en la misma rama desde distintas máquinas, ContextLock detecta conflictos de contexto y bloquea la escritura concurrente mediante un mecanismo de leases con tiempo de expiración configurable (`--lease-ttl 300` segundos). Ideal para equipos que usan feature flags o experimentos A/B con asistentes de código.

### Cómo Usar

1. **Inicializar ContextLock en un proyecto existente**  
   Ejecuta `contextlock init --project-id mi-proyecto --backend postgresql://user:pass@localhost:5432/contextdb`. Esto crea la tabla `context_graph` con columnas `node_id UUID`, `parent_id UUID`, `content_hash SHA256`, `created_at TIMESTAMPTZ` y `ttl_ms BIGINT`. Luego, en tu archivo de configuración del asistente (`.contextlock.yml`), define `snapshot_policy: every_n_tokens: 4096`.

2. **Restaurar contexto después de un apagado**  
   Si el sistema se reinicia o el asistente pierde la sesión, ejecuta `contextlock restore --session-id <UUID> --target-branch feature/x`. ContextLock reconstruirá el grafo desde el último snapshot completo y aplicará los diffs intermedios. Para verificar la integridad, usa `contextlock verify --checksum` que compara los hashes de los archivos tocados con los almacenados en el grafo.

3. **Forzar un bloqueo de contexto en equipo**  
   Cuando dos desarrolladores modifican simultáneamente el mismo prompt base, ejecuta `contextlock lock --branch main --reason "Refactorización de módulo de autenticación"`. Esto adquiere un lease por 5 minutos (por defecto). Si otro miembro intenta `contextlock push` sin el lease, recibe un error `E_LOCK_ACQUIRED: Context locked by user jdoe@example.com until 2025-03-21T14:30:00Z`. Para liberar antes, `contextlock unlock --force --token <admin_token>`.

### Mejores Prácticas

- **Configurar TTLs según criticidad del contexto**  
  No todos los fragmentos de contexto tienen la misma vida útil. Para instrucciones de configuración de entorno (como variables de entorno o versiones de SDK), usa `ttl_ms: 86400000` (24 horas). Para decisiones arquitectónicas o reglas de negocio, usa `ttl_ms: 2592000000` (30 días). Esto evita que ContextLock sature la base de datos con nodos obsoletos y mantiene el grafo ligero. Monitorea con `contextlock stats --top-nodes 10` para identificar cuellos de botella.

- **Integrar ContextLock con hooks de Git**  
  Añade un hook `post-commit` que ejecute `contextlock snapshot --auto --message "Commit $(git rev-parse HEAD)"`. Esto asegura que cada commit tenga un snapshot de contexto asociado. En el hook `pre-push`, ejecuta `contextlock verify --branch $(git branch --show-current)`. Si hay divergencia entre el contexto local y el remoto, el push se rechaza y se sugiere `contextlock merge` para reconciliar. Esta práctica reduce drásticamente los errores de "el asistente no recuerda lo que hicimos ayer".
