# contextlock - CON_027

## Resumen

ContextLock es un sistema de gestión de contexto persistente para asistentes de IA, diseñado para mantener coherencia semántica en sesiones de desarrollo prolongadas. Optimizado para entornos de productividad con memoria de trabajo parametrizable y sincronización en tiempo real.

## Contenido Detallado

### ¿Qué es ContextLock?

ContextLock actúa como una capa de estado intermedio entre el desarrollador y el modelo de lenguaje, permitiendo retener variables de contexto, decisiones arquitectónicas y fragmentos de código relevantes sin depender de la ventana de tokens del LLM. Utiliza un esquema de bloqueo por hash semántico para evitar colisiones entre contextos concurrentes.

### Características Principales

- **Bloqueo de Contexto por Hash Semántico (SHL-256)**: Cada fragmento de contexto se indexa mediante un hash SHA-256 del contenido más un metadato de sesión (`session_id`, `project_root`, `timestamp`). Permite recuperar instantáneamente el estado exacto de cualquier punto del desarrollo, incluso tras reinicios del entorno. Soporta hasta 10,000 bloques concurrentes con una latencia de búsqueda inferior a 2 ms en PostgreSQL.

- **Sincronización Asíncrona con Cache Local y Remoto**: Implementa un buffer de escritura diferida (write-back) con un umbral configurable de `sync_interval_ms` (por defecto 500 ms). Los cambios se almacenan primero en un archivo `.contextlock/cache.json` local y luego se replican a una base de datos PostgreSQL remota mediante conexiones SSL/TLS. La integridad se verifica con checksums CRC32 en cada operación de escritura.

- **Resolución Automática de Conflictos por Timestamp Vectorial**: Cuando dos instancias de ContextLock modifican el mismo bloque concurrentemente, se emplea un vector de relojes lógicos (Lamport clocks) para determinar el orden causal. El bloque ganador se fusiona con un merge recursivo de campos clave (`key`, `value`, `ttl_seconds`), y se notifica al usuario mediante logs estructurados en formato JSON con nivel `WARN`.

### Cómo Usar

1. **Inicializar el Contexto del Proyecto**: Ejecuta `contextlock init --project-dir /ruta/del/proyecto --backend postgresql://user:pass@localhost:5432/contextlock_db --cache-ttl 3600`. Esto crea el directorio `.contextlock/` con el archivo de configuración `config.json` que incluye parámetros como `hash_algorithm: sha256`, `max_blocks: 10000` y `sync_interval_ms: 500`. Verifica con `contextlock status --verbose` que la conexión a la base de datos esté activa y que el cache local esté vacío.

2. **Bloquear un Fragmento de Contexto**: Usa `contextlock lock --key "arquitectura_servicio" --value "{\"patron\": \"CQRS\", \"lenguaje\": \"Go\", \"version\": \"1.22\"}" --ttl 7200 --session-id "dev-session-20250315"`. El sistema genera un hash SHA-256 del valor y lo almacena en la base de datos con un timestamp UTC. Para verificar, ejecuta `contextlock get --key "arquitectura_servicio"` y observa la salida JSON con los campos `hash`, `created_at` y `expires_at`.

3. **Resolver un Conflicto Manualmente**: Si recibes un mensaje de conflicto (ej: `CONFLICT detected for key "db_schema"`), ejecuta `contextlock resolve --key "db_schema" --strategy merge --merge-mode recursive`. El sistema muestra las dos versiones en colisión con sus vectores de tiempo y aplica la fusión automática. Luego, confirma con `contextlock commit --key "db_schema" --force` para sobrescribir el bloque en PostgreSQL.

### Mejores Prácticas

- **Segmentar Contextos por Módulo Funcional**: En proyectos con múltiples microservicios, asigna un `session_id` único por módulo (ej: `auth-service`, `payment-api`). Esto evita que el contexto de un servicio contamine a otro y reduce la latencia de búsqueda en PostgreSQL al particionar los bloques por `session_id` con índices compuestos. Además, define un `ttl_seconds` de 86400 (24 horas) para contextos de larga duración y 600 (10 minutos) para fragmentos transitorios como logs de depuración.

- **Automatizar la Limpieza de Bloques Expirados**: Configura un cron job que ejecute `contextlock purge --older-than 604800` (7 días) cada medianoche. Este comando elimina bloques cuyo `expires_at` sea anterior al timestamp actual y compacta la tabla de PostgreSQL con `VACUUM FULL`. Para entornos críticos, habilita la retención de logs de eliminación en un archivo `.contextlock/purge.log` con formato `[%Y-%m-%d %H:%M:%S] PURGED key=%s session=%s`.
