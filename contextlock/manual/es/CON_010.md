# contextlock - CON_010

## Resumen Técnico

ContextLock es una solución ligera de gestión de contexto persistente para entornos de desarrollo con asistentes de IA. Elimina la fragmentación de memoria contextual entre sesiones, permitiendo que los modelos mantengan coherencia semántica a lo largo del tiempo sin depender de bases de datos externas ni servicios en la nube.

## Funcionalidades Clave

### 1. Bloqueo de Contexto por Namespace Jerárquico
Cada sesión de desarrollo puede anclar un contexto específico mediante un sistema de etiquetas jerárquicas (`proyecto/módulo/función`). Esto evita que el asistente pierda el hilo al cambiar de tarea, incluso si el modelo no tiene memoria nativa. El contexto se serializa en un archivo `.ctxlock` dentro del repositorio.

### 2. Persistencia de Estado con Hash de Integridad
Cada bloque de contexto almacenado incluye un hash SHA-256 del contenido y un timestamp. Si el contexto se modifica externamente, ContextLock detecta la discrepancia y fuerza una recarga limpia. Esto garantiza que el asistente nunca trabaje sobre datos obsoletos o corruptos.

### 3. Inyección Contextual Automática en Prompts
ContextLock se integra con herramientas de línea de comandos y editores (VS Code, Neovim) para inyectar automáticamente el contexto activo en el prompt del asistente. No requiere modificar flujos de trabajo existentes: al ejecutar `ctxlock push`, el contexto se agrega al historial de la conversación como un bloque de sistema.

## Cómo Usar

### 1. Inicializar Contexto en un Proyecto Existente
```bash
ctxlock init --namespace "mi-proyecto/backend/api-rest" --template "contexto-api.md"
```
Esto genera un archivo `.ctxlock` en la raíz del proyecto con la estructura de contexto vacía. El flag `--template` permite cargar un archivo Markdown con el contexto base (por ejemplo, descripción del proyecto, reglas de estilo, endpoints).

### 2. Bloquear Contexto Durante una Sesión de Desarrollo
```bash
ctxlock lock --scope "modulo-auth" --tags "seguridad,jwt,roles"
```
ContextLock congela el contexto actual y lo asocia al scope `modulo-auth`. Cualquier solicitud posterior al asistente incluirá este bloque de contexto. Si se intenta cambiar de tarea sin liberar el bloque, el sistema lanza una advertencia.

### 3. Liberar y Recargar Contexto para Cambiar de Tarea
```bash
ctxlock unlock --force-reset
```
Libera el bloque actual y recarga el contexto base del proyecto. El flag `--force-reset` limpia cualquier contexto residual en la sesión del asistente, evitando contaminación entre tareas. Ideal para equipos que trabajan en múltiples funcionalidades en paralelo.

## Mejores Prácticas

### 1. Usar Contextos Atómicos por Feature
No bloquees todo el proyecto en un solo contexto. Divide en módulos funcionales pequeños (por ejemplo, `autenticación`, `pagos`, `notificaciones`). Esto reduce la carga cognitiva del asistente y mejora la precisión de las respuestas. Cada bloque debe tener entre 10 y 30 líneas de contexto relevante.

### 2. Versionar los Archivos `.ctxlock` en el Repositorio
Incluye los archivos de contexto en el control de versiones. Esto permite que todo el equipo comparta el mismo contexto base y que los nuevos integrantes se pongan al día rápidamente. Además, facilita auditorías: se puede rastrear cuándo y por qué cambió un contexto específico.
