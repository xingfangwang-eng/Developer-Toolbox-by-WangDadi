# contextlock - CON_014

## Resumen Ejecutivo

ContextLock es la navaja suiza definitiva para desarrolladores que trabajan con asistentes de IA: un gestor de contexto persistente que elimina el síndrome de "cerebro de pez" en tus sesiones de trabajo. Diseñado para entornos CI/CD, pair programming con IA y debugging complejo, este toolkit garantiza que tu asistente recuerde decisiones arquitectónicas, configuraciones críticas y patrones de código sin importar cuántos reinicios de sesión sufras. Optimizado para **AI Context Management**, **Persistent AI Memory** y **Developer Productivity Tool**.

## Contenido Detallado

### ¿Qué es ContextLock?

ContextLock no es otro wrapper de prompts. Es un middleware de memoria contextual que serializa el estado cognitivo de tu sesión de IA en un archivo `.contextlock` cifrado y versionable. Cuando tu asistente "olvida" una variable de entorno, una dependencia o una regla de negocio, ContextLock lo restaura automáticamente. Está construido sobre un grafo de dependencias en Rust con bindings para Node.js, Python y Go, lo que lo hace ideal para pipelines de CI/CD, monorepos y entornos de desarrollo distribuido.

### Características Principales (Hardcore)

1. **Snapshot Diferencial con Merge Semántico**  
   ContextLock no guarda el contexto completo cada vez; aplica un algoritmo de diff semántico basado en AST. Detecta cambios en variables de entorno, configuraciones de linter, reglas de ESLint/Prettier, y hasta en la estructura de imports de TypeScript. Cuando haces `git pull` y el contexto de tu IA se desincroniza, ContextLock ejecuta un merge automático que prioriza las reglas del proyecto sobre las locales, sin sobrescribir configuraciones sensibles.

2. **Persistencia Cross-Session con Hash de Identidad**  
   Cada sesión de IA se identifica mediante un hash SHA-256 generado a partir del `package.json`, `.env`, `tsconfig.json` y el último commit. Si cambias de rama, cierras la terminal o incluso cambias de máquina, ContextLock restaura el contexto exacto de la sesión anterior. Ideal para equipos que trabajan con pair programming asíncrono o debugging de incidentes en producción.

3. **Inyección Contextual en Tiempo Real vía WebSocket**  
   ContextLock expone un endpoint WebSocket en `ws://localhost:8484/context` que permite a cualquier asistente de IA (Copilot, Cursor, Claude Code, etc.) suscribirse a cambios de contexto. Cuando modificas un archivo de configuración o instalas una dependencia, ContextLock empuja el nuevo contexto automáticamente al asistente, sin necesidad de reiniciar la sesión. Soporta rate limiting y filtrado por regex para evitar sobrecarga.

### Cómo Usar (Pasos Detallados)

1. **Inicialización y Configuración del Contexto Base**  
   Ejecuta `contextlock init` en la raíz de tu proyecto. Esto genera un archivo `.contextlock.yml` donde defines las variables de entorno críticas (`NODE_ENV`, `API_KEY`), los archivos de configuración a monitorizar (`tsconfig.json`, `.eslintrc.js`) y las reglas de exclusión (`node_modules`, `dist`). Luego, corre `contextlock snapshot --tag v2.3.1` para crear el primer snapshot. ContextLock analizará el AST de tu código y generará un archivo `.contextlock/snapshots/v2.3.1.ctxl` cifrado con AES-256-GCM.

2. **Vinculación con tu Asistente de IA**  
   En tu terminal, ejecuta `contextlock attach --assistant copilot`. Esto inicia el servidor WebSocket en segundo plano. Luego, en tu IDE, agrega el siguiente snippet en el prompt inicial de tu asistente:  
   `[ContextLock] Conectado a ws://localhost:8484/context. Usa 'contextlock status' para ver el contexto activo.`  
   Cada vez que el asistente necesite recordar una variable de entorno o una regla de linting, ContextLock inyecta el contexto correspondiente sin que tú tengas que escribir nada. Para verificar, usa `contextlock status --verbose`.

3. **Restauración de Contexto en CI/CD**  
   En tu pipeline de GitHub Actions o GitLab CI, agrega el siguiente step:  
   ```yaml
   - name: Restaurar contexto de IA
     run: |
       contextlock restore --from-snapshot v2.3.1 --env production
       contextlock attach --assistant claude-code
   ```  
   Esto asegura que Claude Code recuerde la configuración exacta de producción (variables de entorno, reglas de base de datos, endpoints) sin tener que leer el `README.md` cada vez. Si el snapshot está corrupto, ContextLock intentará reconstruirlo desde el último commit válido usando `git log --diff-filter=M`.

### Mejores Prácticas para Desarrolladores

1. **Versiona tus Snapshots como si fueran commits**  
   No guardes un snapshot por cada cambio trivial. En lugar de eso, asócialos a versiones semánticas de tu proyecto (`v1.0.0`, `v2.3.1`). Esto permite que en code reviews, el revisor pueda restaurar el contexto exacto que usó el autor del PR. Además, usa `contextlock diff --from v1.0.0 --to v2.3.1` para ver qué variables de entorno o reglas de linting cambiaron entre versiones.

2. **Implementa ContextLock en entornos de staging antes que en producción**  
   El WebSocket de ContextLock puede ser un vector de ataque si no se asegura. En staging, prueba con `contextlock attach --ssl --cert ./certs/localhost.pem` para habilitar TLS. Además, configura un firewall de rate limiting con `contextlock config --rate-limit 10/s` para evitar que un asistente malicioso inunde el contexto con peticiones. En producción, usa el modo `--readonly` para que el asistente solo pueda leer el contexto, no modificarlo.
