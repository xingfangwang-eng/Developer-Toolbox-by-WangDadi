# contextlock - CON_012

## Overview

ContextLock is a high-performance, persistent AI context management engine engineered for developer productivity. It eliminates session volatility by locking and restoring task-specific context across LLM invocations, with sub-millisecond retrieval latency and zero-copy state serialization.

## Core Features

### 1. **Deterministic Context Snapshot Engine**
- Captures full conversation state, including system prompts, tool call histories, and token-level embeddings, into a compressed binary snapshot (`.ctxlock` format).
- Supports **incremental diff snapshots** (delta size ≤ 4 KB per change) for real-time collaboration scenarios.
- Snapshot integrity verified via SHA-256 hash; rollback to any previous state in under 50 ms.

### 2. **Multi-Threaded Context Locker with Priority Queues**
- Manages up to 1,024 concurrent context sessions using a lock-free ring buffer and priority-based eviction policy (LRU + token cost weight).
- Each session lock holds a **TTL (time-to-live)** configurable from 30 seconds to 24 hours; expired locks auto-release to prevent memory leaks.
- Exposes a gRPC endpoint (`LockContext`, `UnlockContext`, `GetContext`) with average round-trip time < 2 ms.

### 3. **PostgreSQL-Backed Persistent Index**
- All context snapshots are indexed in a PostgreSQL database with a custom `context_vector` column using `pgvector` for semantic similarity search.
- Supports querying by **context ID**, **session UUID**, or **embedding cosine similarity** (threshold > 0.85 for recall).
- Index rebuilds at 100,000 records per second; automatic vacuum and analyze triggered every 5 minutes.

## How to Use

### Step 1: Initialize a Context Session
```bash
contextlock init --session-id "dev-session-20250315" --ttl 3600 --storage postgresql://user:pass@localhost:5432/contextdb
```
- The CLI returns a `session_token` (JWT-encoded) and a unique `context_id`.
- All subsequent operations require the `session_token` for authentication.

### Step 2: Lock and Store Context
```python
from contextlock import ContextClient

client = ContextClient(endpoint="localhost:50051")
client.lock_context(
    session_token="<token>",
    context_id="ctx-abc123",
    payload={
        "system_prompt": "You are a code reviewer...",
        "tool_history": [{"tool": "lint", "output": "No errors"}],
        "embedding": model.encode("current state").tolist()
    },
    metadata={"project": "contextlock", "branch": "main"}
)
```
- The payload is serialized using Protocol Buffers; maximum payload size is 16 MB.
- Returns a `lock_id` and `timestamp` (nanosecond precision).

### Step 3: Retrieve and Resume Context
```bash
contextlock get --session-token "<token>" --context-id "ctx-abc123" --output json
```
- Output includes `status`, `payload` (base64-encoded), and `lock_expires_at` (ISO 8601).
- If the lock has expired, the tool automatically requests a new lock with the same TTL.
- For semantic search: `contextlock search --query "code review state" --threshold 0.85`

## Best Practices

### 1. **Use Short-Lived Locks for Interactive Sessions**
- Set TTL to 300 seconds for real-time chat workflows. This reduces contention and ensures context is released promptly if the user disconnects.
- Combine with a heartbeat mechanism (ping every 60 seconds) to extend the lock automatically.

### 2. **Batch Snapshot Deltas During Long-Running Tasks**
- Instead of locking after every tool call, batch incremental snapshots every 10 invocations or 30 seconds, whichever comes first.
- This reduces PostgreSQL write amplification by 70% and keeps snapshot size under 1 MB for most developer workflows.
