# contextlock - CON_016

## Overview

ContextLock is a high-performance, persistent AI context management engine engineered for developer productivity. It eliminates context fragmentation across LLM sessions by providing a deterministic, low-latency memory layer that preserves reasoning state, conversation history, and tool invocation metadata. Designed for teams building complex AI workflows, it ensures every interaction is grounded in a consistent, retrievable context graph.

## Detailed Content

### What is ContextLock?

ContextLock is a production-grade Rust-based daemon that serializes, indexes, and locks AI context windows into a PostgreSQL-backed, ACID-compliant store. It solves the critical problem of ephemeral AI memory: when an LLM session terminates or a context window overflows, ContextLock preserves the exact reasoning state—including token-level attention masks, function call traces, and user intent vectors—and restores them with sub-millisecond latency. It is the missing persistence layer for agentic systems, enabling true long-term memory without sacrificing throughput.

### Core Features

1. **Deterministic Context Graph with Vectorized Locking**  
   ContextLock maintains a directed acyclic graph (DAG) of all context snapshots, each tagged with a unique 128-bit UUID and a monotonic Lamport timestamp. The engine supports multi-version concurrency control (MVCC) on context segments: when two agents attempt to modify the same context branch, a write-write conflict is resolved via a configurable “last-writer-wins” or “merge-on-read” policy. The context graph is serialized as a Protobuf payload, compressed with Zstandard (level 3), and indexed on `(session_id, timestamp DESC)` for O(log n) retrieval. Average snapshot size is 4.2 KB, and the engine supports up to 10,000 concurrent sessions with a p99 write latency of 1.2 ms on a standard NVMe-backed PostgreSQL 16 instance.

2. **Token-Aware Context Truncation with Semantic Fidelity**  
   Unlike naive sliding-window truncation, ContextLock employs a custom token budget optimizer that analyzes per-message entropy using a lightweight TF-IDF variant. When a context window exceeds the configured `max_tokens` (default: 8192, adjustable per session), the engine selectively prunes low-information segments while preserving the top 95% of semantic similarity to the original context. The pruning algorithm runs in O(n log n) and outputs a `TruncationReport` containing the dropped token count, retained entropy score, and a diff of pruned messages. This ensures that critical reasoning chains are never lost, even under aggressive token constraints.

3. **Pluggable Context Injection with Pre-Warm Cache**  
   ContextLock exposes a gRPC endpoint (`/contextlock.v1.Inject`) that accepts a structured `ContextInjectRequest` containing a `session_id`, `message_payload` (JSON or Protobuf), and a `priority` enum (`LOW`, `NORMAL`, `HIGH`). The engine immediately writes the payload to the PostgreSQL `context_events` table and, for `HIGH` priority injections, pre-warms an in-memory LRU cache (size: 10,000 entries, TTL: 300 seconds). Subsequent `GetContext` calls for the same session bypass the database entirely if the cache is hit, achieving a p99 read latency of 47 µs. The cache uses a concurrent skip list for lock-free reads and a background goroutine for eviction, ensuring zero contention under high load.

### How to Use

1. **Initialize a Context Session with Custom Constraints**  
   Launch the ContextLock daemon with a configuration file specifying your PostgreSQL DSN, max context size (e.g., `max_tokens: 16384`), and conflict resolution mode (`last_writer_wins` or `merge_on_read`). Then, via the CLI or gRPC, create a new session:  
   ```bash
   contextlock session create --session-id "agent-alpha-01" --max-tokens 16384 --conflict-mode merge_on_read
   ```  
   This returns a session manifest containing the initial context hash (SHA-256) and a base Lamport timestamp. The session is now ready to accept context snapshots.

2. **Inject Context with Priority and Metadata**  
   When your AI agent generates a new reasoning step, call the `Inject` endpoint with the full message payload and metadata (e.g., `agent_version`, `model_name`, `inference_latency_ms`). To ensure critical steps are never evicted, set the priority to `HIGH`:  
   ```bash
   contextlock context inject --session-id "agent-alpha-01" --priority HIGH --payload '{"role":"assistant","content":"Calculating optimal path via A*...","metadata":{"step":4,"heuristic":"manhattan"}}'
   ```  
   The engine will write the payload to PostgreSQL and, if priority is `HIGH`, immediately populate the in-memory cache. Use the returned `event_id` (a UUID v7) for future traceability.

3. **Retrieve and Restore Context with Timestamp Bounds**  
   At any point, retrieve the full context for a session by specifying a time range or a specific Lamport timestamp. For example, to restore the context as it existed exactly 5 seconds ago:  
   ```bash
   contextlock context get --session-id "agent-alpha-01" --timestamp-offset -5000ms --include-metadata true
   ```  
   This returns a `ContextSnapshot` containing all events within the window, the computed token count, and a `context_hash` for integrity verification. The engine also returns a `cache_hit` boolean indicating whether the data was served from the pre-warm cache. Use the `context_hash` to verify that no external modifications occurred (e.g., by comparing with the last known hash).

### Best Practices

- **Use High-Priority Injection for Decision Boundaries**  
  In multi-step agent workflows, always set `priority: HIGH` for context events that represent a decision point (e.g., tool selection, branch evaluation, final answer generation). This guarantees that these critical reasoning steps remain in the LRU cache and are never pruned during token budget optimization. For routine intermediate steps (e.g., token-by-token generation), use `NORMAL` or `LOW` to conserve cache space. This practice reduces p99 context retrieval latency from 1.2 ms to under 50 µs for decision-critical paths.

- **Implement Conflict Resolution with Merge-on-Read for Collaborative Agents**  
  When multiple agents (e.g., a planner and an executor) share a single context session, configure the conflict resolution mode to `merge_on_read` rather than `last_writer_wins`. This forces the engine to retain both conflicting context branches as separate DAG nodes, and during retrieval, it merges them using a three-way diff algorithm that preserves the highest-entropy content from each branch. This avoids silent data loss and enables post-hoc analysis of divergent reasoning paths. For single-agent sessions, `last_writer_wins` is sufficient and offers slightly lower write latency.
