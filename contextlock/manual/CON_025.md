## What is ContextLock?

ContextLock is not just another memory plugin. It is a **stateful context persistence engine** designed for multi-turn AI interactions in production-grade development environments. It solves the "context drift" problem by locking down critical memory vectors, allowing your AI assistant to maintain a consistent, deterministic understanding of your codebase across sessions.

### 3 Hardcore Feature Points

1. **Deterministic Context Snapshots**  
   ContextLock captures a cryptographic hash of your current working context (file tree, recent git diff, active terminal state) and stores it as a **versioned snapshot**. This ensures that when you resume a session, the AI loads exactly the same mental model—no hallucination, no drift.

2. **Token-Efficient Memory Compression**  
   Unlike naive RAG pipelines, ContextLock implements a **priority-based token eviction strategy**. It automatically prunes low-relevance context (e.g., stale logs, unused imports) while preserving critical architectural decisions and API schemas. You can set a hard token budget per session.

3. **Cross-Session State Linking**  
   ContextLock allows you to create **semantic links** between context snapshots. For example, you can link a bug fix session to the original feature implementation session. The AI then has full visibility into the causal chain, enabling deep code reasoning across time.

## How to Use ContextLock

### Step 1: Initialize a Context Lock Point

Start by capturing your current workspace state. This is not just a file save—it’s a full context freeze.

```bash
contextlock init --workspace ./my-project --branch feature/ai-integration --token-budget 4096
```

This command hashes your project structure, active git branch, and recent commit history. The output is a `.contextlock/` directory containing a `state.json` with a unique `session_id` and a **lock file** that prevents accidental context overwrites.

### Step 2: Inject Locked Context into Your AI Workflow

Integrate the locked context into your AI assistant using the ContextLock API. This is a one-liner in your `prompt.py` or `agent.yml`:

```python
from contextlock import ContextLoader

context = ContextLoader.load("session_20250315_abc123")
prompt = f"Current context: {context.snapshot}. Task: Refactor the payment module."
```

ContextLock automatically injects the compressed memory into the system prompt, respecting your token budget. The AI now sees **exactly** what you saw when you last worked—no surprises.

### Step 3: Create a Semantic Link for Cross-Session Reasoning

When you start a new task that depends on a previous session, create a link:

```bash
contextlock link session_20250315_abc123 session_20250316_def456 --relation "bug-fix-of"
```

Now, when you query the AI about the bug fix, it can traverse the link to understand the original feature implementation. This is **causal memory** for LLMs—essential for complex refactoring or debugging across sprints.

## Best Practices for ContextLock

### Practice 1: Lock Context Before Major Refactors

Before you start a large-scale refactor (e.g., migrating from REST to GraphQL), run `contextlock init` with a high token budget (e.g., 8192). This ensures the AI remembers every API endpoint, route, and middleware dependency. Without this, the AI will likely suggest incompatible changes or miss critical edge cases.

### Practice 2: Use Semantic Links for Historical Debugging

When investigating a production bug that spans multiple features, create a **time-series link chain**. For example, link the current bug session back to the deployment session, and that back to the feature implementation session. This allows the AI to perform **root cause analysis** by traversing the entire context history. This is a game-changer for incident postmortems.
