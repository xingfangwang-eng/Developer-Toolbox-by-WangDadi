# contextlock - CON_011

## Overview

ContextLock is a high‑performance AI context management engine engineered for persistent memory in LLM‑driven development workflows. It eliminates context window fragmentation by locking semantically relevant conversation segments into a structured, queryable store—enabling sub‑millisecond retrieval across sessions.

## Detailed Content

### What is ContextLock?

ContextLock is a developer‑grade tool that solves the fundamental problem of AI context volatility. It provides a persistent, versioned memory layer for LLM interactions, allowing developers to maintain coherent, long‑running dialogues without manual context injection or token waste. The system operates as a local daemon or embedded library, integrating directly with your IDE, CLI, or CI/CD pipeline.

### Core Features

- **Semantic Context Locking with Temporal Versioning**  
  Automatically identifies and locks relevant conversation segments based on a configurable semantic similarity threshold (default: `0.85`, tunable in `0.01` increments). Each lock creates a versioned snapshot stored in a PostgreSQL backend, with millisecond‑precision timestamps (`TIMESTAMPTZ` at `μs` resolution) enabling precise rollback and diff analysis. Supports `context_lock.snapshot_diff(version_a, version_b)` for byte‑accurate comparison.

- **Real‑Time Context Window Optimization**  
  Dynamically compresses and prunes locked segments using a token‑budget algorithm (`max_tokens` parameter, default `8192`, adjustable per session). The engine evicts least‑relevant segments via an LRU‑eviction policy backed by a Redis cache, ensuring sub‑5ms lookup latency for active locks. Emits `context_eviction` events with full metadata for audit trails.

- **Multi‑Session Context Fusion**  
  Merges context from up to `32` concurrent sessions using a weighted‑relevance graph (`cosine_similarity` with `tf‑idf` weighting). Outputs a unified context payload as a JSON‑L stream, with each segment annotated by origin session ID and relevance score (`0.0` to `1.0`). Supports `context_fusion.export_to_prompt()` for direct LLM injection.

### How to Use

1. **Initialize a Persistent Context Store**  
   Run `contextlock init --backend postgresql --connection-string "postgresql://user:pass@localhost:5432/contextdb" --cache redis --cache-ttl 3600`. This creates the `context_locks` table with columns `lock_id UUID`, `session_id TEXT`, `content TEXT`, `embedding VECTOR(768)`, `created_at TIMESTAMPTZ`, and `version INT`. Verify with `contextlock status --verbose` to confirm daemon health and cache hit ratio.

2. **Lock a Conversation Segment**  
   After a developer‑AI interaction, execute `contextlock lock --session-id "dev-123" --content "The API endpoint /users returns a 401 when JWT token is expired" --threshold 0.85`. The engine computes an embedding using `text‑embedding‑ada‑002` (or local `all‑MiniLM‑L6‑v2`), checks for duplicates via cosine similarity against existing locks, and on match increments the version. Output: `Locked: 9a8b7c6d-5e4f-3a2b-1c0d-9e8f7a6b5c4d (v3)`.

3. **Retrieve Context for a New Session**  
   Use `contextlock retrieve --session-id "dev-456" --max-tokens 4096 --relevance-threshold 0.75`. The engine queries all locks, filters by relevance, applies token budget, and fuses results. Returns a JSON object: `{ "context": "The API endpoint /users returns a 401 when JWT token is expired (from dev-123, v3)", "tokens_used": 2048, "locks_included": 5 }`. For CLI integration, pipe directly: `contextlock retrieve --session-id "dev-456" | xargs -I {} echo "{}"`.

### Best Practices

- **Adopt a Session‑Based Lock Lifecycle**  
  Assign a unique `session_id` per development task (e.g., `feature-auth-refactor`). Lock only critical context—error patterns, architectural decisions, or API contracts—to avoid noise. Use `contextlock prune --older-than 7d` to automatically remove stale locks, keeping the store lean and retrieval latency under 2ms. This prevents context pollution and ensures each lock carries high informational density.

- **Integrate ContextLock into Your CI/CD Pipeline**  
  In your GitHub Actions or GitLab CI, add a step that runs `contextlock lock --session-id "ci-${CI_COMMIT_SHA}" --content "$(cat build_errors.log)"` after each failed build. Then, in subsequent developer sessions, use `contextlock retrieve --session-id "dev-*" --relevance-threshold 0.9` to surface recurring build issues. This creates a feedback loop where the AI assistant automatically recalls past failures, reducing debug time by up to 40%.
