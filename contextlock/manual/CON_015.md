# contextlock - CON_015

## Overview

ContextLock is a hardened AI context management layer designed for developers who need persistent, thread-safe memory across LLM sessions. Unlike ephemeral chat histories that evaporate on tab close, ContextLock snapshots, tags, and restores context with surgical precision—no more repeating your entire architecture to GPT-4 every Monday morning.

## Core Features

### 1. Deterministic Context Snapshots with Hash-Linked Diffs
Every context state is hashed and chained like a Git commit. You can rollback to any previous context frame, diff against another snapshot, or fork a context branch for A/B experimentation. The hash chain ensures tamper-proof audit trails for compliance-heavy workflows.

### 2. Multi-Model Context Sync via gRPC Streams
ContextLock exposes a bi-directional gRPC stream that synchronizes context across OpenAI, Anthropic, and local Ollama models in real-time. No more manual copy-paste between Claude and GPT—your system prompt, conversation history, and injected data stay consistent across providers.

### 3. Context-Aware Injection Hooks (Middleware Pattern)
Define middleware hooks that automatically inject relevant context fragments based on the current user action or codebase state. For example, when you open a Python file, ContextLock pulls the last 5 related Stack Overflow threads and your project’s dependency graph into the LLM’s system prompt—zero manual effort.

## How to Use

### 1. Initialize a Context Lock File
Run `contextlock init --project my-ai-app` in your terminal. This creates a `.contextlock/` directory with a `context.yaml` manifest and a SQLite-backed history store. The manifest defines your context schema: system prompt, allowed injection sources (e.g., `./docs/`, `./src/`), and model routing rules.

### 2. Capture a Context Snapshot
After a productive LLM session, execute `contextlock capture --tag "auth-module-v3"`. This freezes the current conversation state, computes a SHA-256 hash, and appends it to the local chain. You can also pass `--auto` to enable event-driven capture on terminal close or file save.

### 3. Restore and Diff Across Sessions
To resume work, run `contextlock restore --tag "auth-module-v3"`. This replays the exact context into your LLM of choice. For debugging, use `contextlock diff --from "auth-module-v2" --to "auth-module-v3"` to see which context fragments changed—critical for understanding why an LLM suddenly shifted its output behavior.

## Best Practices

### 1. Tag Contexts by Feature Branch, Not by Date
Avoid generic tags like “2025-03-01”. Instead, use semantic tags matching your Git branches (e.g., `feat/payment-gateway`, `fix/rate-limit`). This lets you correlate LLM context with code changes, making regression debugging trivial.

### 2. Set Up a CI/CD Pipeline for Context Validation
Integrate ContextLock into your CI pipeline: after each PR merge, run `contextlock validate` to ensure the new codebase state doesn’t break any pre-existing context snapshots. This catches silent context drift before it reaches production.
