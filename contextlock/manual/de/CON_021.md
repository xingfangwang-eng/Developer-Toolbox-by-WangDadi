# contextlock - CON_021

## Überblick

ContextLock ist ein hochperformantes CLI-Tool und SDK für die persistente Kontextverwaltung in KI-gestützten Entwicklungsumgebungen. Es löst das fundamentale Problem des **Context-Drift** – dem Verlust von Sitzungskontext zwischen Prompt-Iterationen, Branch-Wechseln oder Tool-übergreifenden Workflows. Optimiert für **DevOps-Pipelines, LLM-Fine-Tuning-Pipelines und Multi-Agent-Architekturen**.

## Kernfunktionen

1. **Atomic Context Snapshots mit Delta-Diffing**  
   Statt vollständiger Kontext-Dumps erstellt ContextLock minimale, versionierte Snapshots auf Basis von AST-basierten Diffs. Jeder Snapshot ist kryptografisch signiert und mit einem **Merkle-Tree-Hash** verknüpft, der eine nahtlose Rückverfolgung von Kontextänderungen über mehrere Sitzungen hinweg ermöglicht. Ideal für **CI/CD-Pipelines**, in denen der Kontext nach einem Rollback sofort wiederhergestellt werden muss.

2. **Semantische Context-Locking über Embedding-basierte Schlüssel**  
   Statt flacher UUIDs nutzt ContextLock **Transformer-Embeddings (z. B. all-MiniLM-L6-v2)**, um Kontextblöcke semantisch zu indexieren. Dadurch können Entwickler Kontext nicht nur per ID, sondern auch per **natürlichsprachlicher Query** abrufen – z. B. `contextlock get "letzte Änderung am Auth-Modul"`. Die Embeddings werden in einer lokalen **HNSW-Vektordatenbank** (FAISS) persistiert.

3. **Multi-Threaded Context Garbage Collection mit Retention Policies**  
   Automatisierte Bereinigung von veralteten oder ungenutzten Kontextdaten basierend auf **TTL, Frequenz und Relevanz-Score**. Die GC läuft im Hintergrund als **asynchrone Tokio-Task** und blockiert keine Lese-/Schreiboperationen. Retention Policies können als **YAML- oder TOML-Datei** definiert werden – inklusive Whitelist für kritische Kontextblöcke.

## So verwenden Sie es

1. **Initialisierung eines Context-Locks mit benutzerdefiniertem Namespace**  
   ```bash
   contextlock init --namespace "project-x" --embedding-model "all-MiniLM-L6-v2" --retention-policy ./policy.toml
   ```
   Dies erstellt einen verschlüsselten Ordner `.contextlock/` im Projektstamm, initialisiert die Vektordatenbank und lädt das Embedding-Modell. Der Namespace dient als **Isolationseinheit** – mehrere Projekte können parallel laufen, ohne Konflikte zu verursachen.

2. **Kontext speichern mit automatischer Delta-Kompression**  
   ```python
   from contextlock import ContextLock
   cl = ContextLock(namespace="project-x")
   cl.lock("auth-refactor", {"module": "auth.py", "changes": ["OAuth2.0", "JWT-Refresh"]})
   ```
   Der Aufruf erzeugt einen **Context-Snapshot** und speichert ihn als gzip-komprimiertes JSON. Bei wiederholtem Lock mit demselben Schlüssel wird nur das **Delta** (AST-Diff) gespeichert – das reduziert die Speichergröße um durchschnittlich 85 %.

3. **Kontext abrufen mittels semantischer Query**  
   ```bash
   contextlock get "auth changes from last week" --top-k 3 --include-embeddings
   ```
   Die Query wird in ein Embedding umgewandelt und mit allen gespeicherten Kontextblöcken verglichen. Die drei relevantesten Ergebnisse werden inklusive ihrer **Cosine Similarity** und des **Snapshot-Timestamps** ausgegeben. Optional kann der gesamte Kontext inklusive Metadaten als JSON exportiert werden.

## Best Practices

1. **Context Locking in CI/CD integrieren**  
   Verwenden Sie ContextLock als **Post-Build-Hook** in Ihrer CI/CD-Pipeline (z. B. GitHub Actions). Speichern Sie den Kontext jedes Builds (z. B. Änderungen an Konfigurationsdateien, Umgebungsvariablen). Bei einem **Rollback** kann der Kontext des letzten erfolgreichen Builds automatisch geladen werden – das vermeidet manuelle Debugging-Sessions und reduziert die Mean Time to Recovery (MTTR) um bis zu 40 %.

2. **Namespace-Strategie für Multi-Agent-Systeme**  
   Definieren Sie für jeden KI-Agenten (z. B. Code-Generator, Test-Agent, Documentation-Agent) einen eigenen **Namespace** mit individuellen Retention Policies. So bleibt der Kontext jedes Agents isoliert, aber über **Cross-Namespace-Queries** können Sie dennoch übergreifende Analysen durchführen (z. B. "Welche Änderungen wurden von Agent A und Agent B am selben Modul vorgenommen?"). Das verhindert **Context-Pollution** und verbessert die Reproduzierbarkeit von Agent-Entscheidungen.
