# contextlock - CON_007

## Überblick

ContextLock ist eine hochperformante, speicherresidente Context-Management-Engine für KI-gestützte Entwicklungsumgebungen. Es eliminiert flüchtige Kontextverluste und ermöglicht persistente, thread-übergreifende AI-Memory-Sessions. Entwickelt für DevOps-Ingenieure, die mit LLM-gestützten Workflows arbeiten und reproduzierbare, kontextbewusste Automatisierung benötigen.

## Detaillierter Inhalt

### Was ist ContextLock?

ContextLock ist kein weiterer Chat-Wrapper. Es ist eine Zustandsmaschine für KI-Kontexte. Es serialisiert und deserialisiert Kontextgraphen in einer In-Memory-Datenbank (Redis-kompatibel) und stellt sicher, dass jede AI-Interaktion auf dem exakten vorherigen Zustand aufbaut – ohne Halluzinationen durch fehlende oder überschriebene Informationen. Ideal für CI/CD-Pipelines, Code-Review-Agenten und Multi-Step-Refactoring-Prozesse.

### Kernfunktionen

1. **Context-Graph-Snapshotting**: Jeder AI-Chat oder jede Pipeline-Interaktion wird als gerichteter, azyklischer Graph (DAG) gespeichert. Du kannst zu jedem beliebigen Snapshot zurückkehren, ohne den gesamten Verlauf neu aufbauen zu müssen. Der Snapshot enthält Metadaten wie Token-Verbrauch, Zeitstempel und Abhängigkeits-Hashes.

2. **Deterministisches Context-Rollback**: Falls ein AI-Agent fehlerhafte Änderungen vorschlägt, führst du ein atomic rollback auf den letzten validierten Kontextknoten aus. Das System stellt automatisch die exakte Umgebung (Variablen, Dateien, Prompt-History) wieder her – in Millisekunden.

3. **Multi-Thread-Context-Synchronisation**: ContextLock synchronisiert Kontexte zwischen parallel laufenden AI-Instanzen (z. B. Code-Generator, Test-Generator und Dokumentations-Agent). Keine Race Conditions, keine inkonsistenten Zustände. Jeder Thread hat eine isolierte, aber synchronisierte Sicht auf den globalen Kontext.

### So verwenden Sie es

1. **ContextLock initialisieren und Kontext-Volumes mounten**  
   ```bash
   contextlock init --project ./my-ai-project --storage redis://localhost:6379
   ```  
   Dies erzeugt ein `.contextlock/`-Verzeichnis mit einer `context.graph`-Datei und mountet einen Redis-Backend-Store. Der initiale Kontext wird aus dem aktuellen Arbeitsverzeichnis (`.env`, `prompts/`, `config.yaml`) automatisch extrahiert.

2. **Kontext-Snapshot manuell auslösen**  
   ```bash
   contextlock snapshot --label "refactor-v2.1" --tags "auth,db-migration"
   ```  
   Der Befehl serialisiert den gesamten Context-Graph, berechnet einen SHA-256-Hash über alle Knoten und speichert ihn als Snapshot-ID. Du erhältst eine Ausgabe wie:  
   `Snapshot gespeichert: 0x7a3f... (2.4 MB, 12 Knoten)`.

3. **Rollback auf einen vorherigen Snapshot durchführen**  
   ```bash
   contextlock rollback --snapshot 0x7a3f --restore-env
   ```  
   Das System stellt den gesamten Kontext aus dem Snapshot wieder her, inklusive Umgebungsvariablen, Dateiänderungen und Prompt-History. Ein `--dry-run`-Flag zeigt die geplanten Änderungen vor der Ausführung an.

### Beste Praktiken

1. **Kontext-Snapshots in CI/CD-Pipelines versionieren**  
   Binde `contextlock snapshot` in jeden erfolgreichen Pipeline-Step ein. So kannst du bei einem fehlgeschlagenen Build oder einer fehlerhaften AI-generierten Änderung exakt den Zustand vor dem Fehler reproduzieren. Verwende Git-Tags als Snapshot-Label, um die Rückverfolgbarkeit zu gewährleisten.

2. **Context-Garbage-Collection regelmäßig ausführen**  
   Da ContextLock jeden Snapshot persistent speichert, kann der Speicherverbrauch bei intensiver Nutzung exponentiell wachsen. Führe wöchentlich `contextlock gc --keep-last 50` aus, um alte, unbenutzte Snapshots zu entfernen. Definiere eine Retention-Policy in der `contextlock.yaml`, z. B. `retention: 30d` für automatische Bereinigung.
