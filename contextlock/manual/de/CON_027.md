# contextlock - CON_027

**ContextLock** ist kein weiteres KI-Tool, das nach jedem Prompt vergisst, worum es geht. Es ist eine schlanke, kontextbewusste Middleware für Entwickler, die LLM-gestützte Workflows in CI/CD, Code-Review oder Agenten-Pipelines betreiben. Statt flüchtiger Sessions liefert ContextLock deterministische Kontextpersistenz über Threads, Branches und sogar Deployment-Zyklen hinweg.

---

## 🔧 Hardcore-Features, die den Unterschied machen

### 1. **Lock-by-Context-Hash** – Kein Halluzinieren durch veralteten Kontext
Jeder Kontextblock wird über einen SHA-256-Hash referenziert. Änderst du Code, Dokumentation oder Prompt-Templates, erzeugt ContextLock automatisch einen neuen Lock. Das verhindert, dass dein LLM mit veralteten oder inkonsistenten Informationen arbeitet – selbst bei parallelen Branch-Wechseln.

### 2. **Multi-Layer Memory Stack** – Session, Branch, Global
ContextLock speichert Kontext auf drei Ebenen:
- **Session**: flüchtig, für interaktive Debugging-Sessions
- **Branch**: persistent pro Git-Branch, ideal für Feature-Entwicklung
- **Global**: projektweit, für Architekturregeln oder Coding-Standards

Jede Ebene überschreibt nur die relevanten Keys – keine redundanten Daten, keine Konflikte.

### 3. **Lock-Export als JSON-Schema** – Portabel, validierbar, versionierbar
Exportiere den gesamten aktuellen Kontext als JSON-Schema inklusive Metadaten (Timestamp, Git-Hash, LLM-Modell). Damit kannst du Kontextstände reproduzieren, in CI-Artefakten ablegen oder in anderen Tools validieren – z. B. in ESLint-Plugins oder Pre-Commit-Hooks.

## 🧠 Detaillierte Anwendungsschritte (für echte Developer)

### 1. **ContextLock initialisieren und an Git binden**
```bash
# Projekt initialisieren
contextlock init --project ./my-app

# Automatische Verknüpfung mit Git-Hooks
contextlock hook install --hook post-checkout,pre-commit
```
Danach wird bei jedem Branch-Wechsel automatisch der passende Kontext geladen. Nutze `contextlock status` um zu sehen, welcher Lock aktiv ist.

### 2. **Kontext für ein Feature definieren und sperren**
```bash
# Kontext für ein neues Feature setzen
contextlock set --layer branch --key "auth_module" --value "OAuth2 mit JWT, Refresh-Token in Redis, Expiry 15min"

# Lock erzwingen (verhindert Überschreiben durch andere Branches)
contextlock lock --layer branch --reason "Feature-spezifischer Kontext für PR #42"
```
Jetzt kann kein anderer Branch diesen Lock überschreiben – außer du hebst ihn explizit mit `contextlock unlock` auf.

### 3. **Kontext in CI/CD Pipeline auslesen und validieren**
```yaml
# .github/workflows/context-check.yml
jobs:
  validate-context:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: contextlock export --layer global --format json > context.json
      - run: npx contextlock-validator context.json --schema ./context-schema.json
```
Damit stellst du sicher, dass jeder Build den korrekten, validierten Kontext verwendet – keine stillschweigenden Abweichungen mehr.

## 🏆 Best Practices aus der Praxis

### 1. **Lock-Wechsel mit Branch-Namen synchronisieren**
Verwende Branch-Namen als Lock-Identifier. Das verhindert Kontext-Kollisionen und macht den Workflow debugbar. Beispiel: Statt `feature/login` → Lock-Name: `feature/login`. So weißt du immer, welcher Kontext zu welchem Branch gehört.

### 2. **Globalen Kontext für Architekturregeln nutzen**
Definiere im globalen Lock projektweite Regeln wie „Keine direkten DB-Zugriffe in Controllern“ oder „Alle API-Endpunkte müssen versioniert sein“. Dadurch bleiben LLM-generierte Vorschläge konsistent mit deiner Architektur – auch bei wechselnden Entwicklern oder Tools.
