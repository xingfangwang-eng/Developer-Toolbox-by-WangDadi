# contextlock - CON_006

## Überblick

ContextLock ist ein **kontextbasiertes Speicher-Framework** für KI-gestützte Entwicklungsworkflows. Es eliminiert das lästige Wiederholen von Kontext in Chats, indem es Sitzungszustände, Projektstrukturen und Entwicklerabsichten persistiert – ideal für Teams, die mit ChatGPT, Claude oder lokalen LLMs arbeiten. Die Lösung richtet sich an **Backend-Entwickler, DevOps-Ingenieure und KI-Prompt-Ingenieure**, die ihre Produktivität durch intelligente Kontextverwaltung steigern wollen.

## Kernfunktionen (Hardcore-Features)

### 1. **Kontextuelle Zustandsmaschine (CSM)**
ContextLock implementiert eine **deterministische Zustandsmaschine**, die den gesamten Kontext einer KI-Sitzung in einer JSON-Struktur speichert – inklusive Dateipfade, Variablenbelegungen und Abhängigkeitsgraphen. Die CSM synchronisiert sich automatisch mit Git-Commits, sodass Kontextänderungen versioniert und rückverfolgbar sind.

### 2. **Autonome Kontext-Injektion (ACI)**
Statt manueller Prompt-Erweiterung analysiert ACI die aktuelle Projektstruktur (z. B. `package.json`, `requirements.txt`, `Dockerfile`) und injiziert **relevante Codefragmente, Fehlermeldungen und Abhängigkeitshistorie** direkt in den KI-Kontext. Dies reduziert Halluzinationen bei Code-Generierung um bis zu 40%.

### 3. **Multi-Modal Context Bridge (MMCB)**
ContextLock unterstützt nicht nur Text, sondern auch **Bild-, Audio- und Log-Dateien** als Kontextquellen. Die Bridge normalisiert diese Formate in ein einheitliches Embedding-Modell (z. B. via `sentence-transformers`), sodass Entwickler z. B. Screenshots von Fehlermeldungen direkt in den Kontext einfließen lassen können.

## So verwenden Sie es (Detaillierte Schritte)

### Schritt 1: Kontext initialisieren und an Projekt binden
Führen Sie im Projektstammverzeichnis aus:
```bash
contextlock init --project my-app --mode persistent
```
Dies erstellt eine `.contextlock`-Datei, die den Projektnamen, die aktuelle Git-Branch und einen MD5-Hash des Dateibaums speichert. Anschließend können Sie den Kontext mit `contextlock attach` an eine KI-Sitzung binden (z. B. via `OPENAI_API_KEY` oder lokale Ollama-Instanz).

### Schritt 2: Kontext dynamisch erweitern mit ACI
Wenn Sie eine neue Funktion implementieren, injizieren Sie automatisch relevante Kontextfragmente:
```bash
contextlock inject --scope current --depth 3 --include "src/**/*.py"
```
ACI extrahiert dann alle Python-Dateien im angegebenen Pfad, filtert Duplikate und komprimiert sie in eine **Token-optimierte Struktur** (max. 4096 Tokens). Sie können auch `--exclude "tests/"` setzen, um Testdateien auszuschließen.

### Schritt 3: Kontext über Git-Commits hinweg synchronisieren
Um Kontextänderungen nach einem Commit zu speichern:
```bash
git commit -m "Fix: Login-Validierung hinzugefügt"
contextlock sync --commit-hash HEAD
```
ContextLock erstellt dann einen **Snapshot** des aktuellen Kontexts (inklusive Prompt-Historie und letzten KI-Antworten) und verknüpft ihn mit dem Git-Commit. Später können Sie mit `contextlock restore --commit-hash <hash>` den exakten Kontext einer früheren Sitzung wiederherstellen.

## Best Practices (Industrieempfehlungen)

### 1. **Kontext-Grenzen für große Projekte definieren**
Bei Monorepos oder Projekten mit über 1000 Dateien sollten Sie `contextlock config --max-depth 2` setzen, um die Kontextgröße zu begrenzen. Kombinieren Sie dies mit **manuellen Tags** (z. B. `contextlock tag --name "auth-module" --paths "src/auth/"`), um nur relevante Teile zu laden. Dies verhindert Token-Überläufe und beschleunigt die KI-Antwortzeit.

### 2. **Kontext-Rollback für Debugging nutzen**
Wenn eine KI-Halluzination fehlerhaften Code generiert, können Sie mit `contextlock rollback --to-last-stable` den Kontext auf den letzten **validierten Zustand** zurücksetzen. Kombinieren Sie dies mit einem CI/CD-Trigger (z. B. GitHub Actions), der bei fehlgeschlagenen Tests automatisch einen Rollback ausführt. Das spart Stunden manueller Fehlersuche.
