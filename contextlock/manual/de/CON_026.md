# contextlock - CON_026

## Überblick

ContextLock ist ein **deterministisches AI-Kontextmanagement-Tool**, das persistente Kontextbindungen zwischen Entwickler-Workflows und LLM-Sessions herstellt. Es eliminiert das allgegenwärtige Problem des „Kontext-Vergessens“ bei KI-gestützter Entwicklung. Statt flüchtiger Chat-Verläufe liefert ContextLock eine **zustandsbehaftete, referenzierbare Kontextschicht**, die über Sitzungen, Branches und sogar verschiedene KI-Modelle hinweg konsistent bleibt. Dieses Tool richtet sich an Developer, die mit KI-Copiloten, Code-Reviews und automatisierten Refactoring-Pipelines arbeiten und eine reproduzierbare, nachvollziehbare Kontextbasis benötigen.

## Kernfunktionen (Hardcore-Features)

### 1. **Deterministische Kontext-Snapshots mit Git-Integration**
ContextLock erstellt **immutable Context-Snapshots**, die direkt an Git-Commits gebunden sind. Jeder Snapshot enthält den gesamten relevanten Kontext (z. B. aktuelle Codebasis, Fehlerlogs, vorherige KI-Entscheidungen) und wird als **`.contextlock/`-Verzeichnis** im Repository versioniert. So können Sie bei einem `git checkout` oder `git bisect` automatisch den exakten Kontext wiederherstellen, der zum Zeitpunkt eines Commits gültig war. Das verhindert „Phantom-Kontext“ und sorgt für reproduzierbare KI-Interaktionen.

### 2. **Multi-Modell-Kontext-Synchronisation**
ContextLock synchronisiert Kontextdaten **bidirektional zwischen verschiedenen KI-Modellen** (z. B. GPT-4, Claude, Gemini) über eine einheitliche API. Wenn Sie einen Kontext in einer Session mit GPT-4 aufbauen, kann derselbe Kontext nahtlos in einer Claude-Session weitergenutzt werden. Die Synchronisation erfolgt über ein **differenzbasiertes Delta-Update**, das nur geänderte Kontextblöcke überträgt – ideal für Teams, die verschiedene Modelle für verschiedene Aufgaben (Code-Generierung, Security-Audit, Dokumentation) einsetzen.

### 3. **Context-Locking mit Rule-Based Constraints**
Sie definieren **harte und weiche Constraints** für den Kontext. Harte Constraints (z. B. „API-Key nie in Kontext übernehmen“) werden automatisch durchgesetzt und führen bei Verletzung zu einem **ContextLock-Fehler**. Weiche Constraints (z. B. „Bevorzuge TypeScript über JavaScript“) werden als Prioritäten markiert. ContextLock erstellt daraufhin einen **optimierten Kontext**, der die Constraints erfüllt, ohne die Informationsdichte zu reduzieren. Dieses Feature ist essenziell für sicherheitskritische und compliance-relevante Entwicklungsumgebungen.

## Detaillierte Anwendungsschritte (Professionelle Workflows)

### Schritt 1: Initialen Kontext-Snapshot erstellen und an Branch binden
Führen Sie `contextlock init --branch feature/neue-auth` aus. ContextLock scannt das aktuelle Arbeitsverzeichnis, extrahiert alle relevanten Dateien (z. B. `src/`, `config/`, `*.md`, `package.json`) und erstellt einen **deterministischen Hash** des gesamten Kontexts. Dieser Hash wird als **ContextLock-Anchor** in der `.contextlock/`-Metadaten-Datei gespeichert. Der Snapshot wird dann mit dem aktuellen Git-Commit verknüpft. Spätere `contextlock restore`-Aufrufe stellen genau diesen Zustand wieder her – unabhängig von zwischenzeitlichen Änderungen.

### Schritt 2: Kontext zwischen KI-Modellen synchronisieren mit Delta-Updates
Nachdem Sie mit GPT-4 einen Code-Review-Kontext aufgebaut haben (z. B. mit `contextlock attach --model gpt-4`), führen Sie `contextlock sync --target claude` aus. ContextLock berechnet ein **Delta zwischen dem aktuellen Kontext und dem letzten synchronisierten Zustand** (gespeichert in `.contextlock/sync_state.json`). Nur die geänderten Kontextblöcke (z. B. neue Code-Blöcke, geänderte Fehlerbeschreibungen) werden an Claude übertragen. Das spart Tokens und verhindert Kontextüberladung. Sie können den Sync-Vorgang mit `--dry-run` testen, bevor Sie ihn ausführen.

### Schritt 3: Rule-Based Constraints definieren und validieren
Erstellen Sie eine `.contextlock/rules.yaml`-Datei mit folgendem Inhalt:
```yaml
hard_constraints:
  - "kein_plaintext_passwort"
  - "kein_api_key_in_kontext"
soft_constraints:
  - "bevorzuge_typescript"
  - "nutze_functional_composition"
```
Führen Sie `contextlock validate --rules .contextlock/rules.yaml` aus. ContextLock prüft den aktuellen Kontext gegen alle harten Constraints. Bei Verstoß wird ein **ContextLockError** mit genauer Angabe der verletzten Regel und der betroffenen Kontextblöcke ausgegeben. Bei weichen Constraints erhalten Sie eine **Prioritäts-Warnung** mit Vorschlägen zur Optimierung. Bei erfolgreicher Validierung wird der Kontext als **compliant** markiert und kann für KI-Interaktionen freigegeben werden.

## Branchen-Best Practices

### 1. **Kontext-Isolation für parallele Feature-Entwicklung**
In Teams, die an mehreren Features gleichzeitig arbeiten, sollten Sie für jedes Feature einen eigenen **ContextLock-Branch** erstellen. Verwenden Sie `contextlock branch --name feature-xyz`, um einen isolierten Kontext zu erzeugen, der nur die relevanten Dateien und Abhängigkeiten enthält. Das verhindert Kontext-Kollisionen und stellt sicher, dass KI-Interaktionen für Feature A nicht versehentlich den Kontext für Feature B beeinflussen. Kombinieren Sie dies mit **ContextLock-Tags** (z. B. `contextlock tag --name "release-2.3"`), um Kontext-Snapshots für Releases zu archivieren.

### 2. **Automatisierte ContextLock-Validierung in CI/CD-Pipelines**
Integrieren Sie ContextLock in Ihre CI/CD-Pipeline (z. B. GitHub Actions, GitLab CI). Fügen Sie einen Schritt hinzu, der vor jedem KI-gestützten Code-Review `contextlock validate --rules .contextlock/rules.yaml` ausführt. Bei Fehlschlag (z. B. wegen eines API-Keys im Kontext) wird der Pipeline-Schritt abgebrochen und ein detaillierter Report generiert. Das verhindert, dass sensible Daten in KI-Modelle gelangen und stellt sicher, dass die Code-Review-Qualität konstant bleibt. Verwenden Sie `contextlock status --json` für maschinenlesbare Ausgaben, die Sie in Dashboards visualisieren können.
