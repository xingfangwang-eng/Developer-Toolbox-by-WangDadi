# contextlock - CON_012

## Überblick

ContextLock ist kein weiteres KI-Tool – es ist die **Context-Layer-Engine** für Developer, die mit LLMs, Agenten oder RAG-Systemen arbeiten. Es löst das zentrale Problem flüchtiger Sitzungen: **Kontextverlust zwischen Prompts, Sessions und Pipelines**. Mit ContextLock persistierst du Kontextzustände als versionierte Snapshots, die von jeder Umgebung aus referenziert werden können – lokal, CI/CD oder Cloud.

## Kernfunktionen

### 1. **Context-Compression via Semantic Pruning**
ContextLock komprimiert aktive Konversationen oder Code-Kontext durch **semantische Deduplizierung** und **Token-bewusstes Clipping**. Es entfernt redundante Informationen, ohne die semantische Kohärenz zu verlieren. Ergebnis: Bis zu 60 % Token-Reduktion bei gleichbleibender Antwortqualität.

### 2. **Multi-Session Context Linking**
Du kannst mehrere Sitzungen (z. B. Code-Review, Bug-Analyse, Architekturdiskussion) über **Context-IDs** miteinander verknüpfen. ContextLock erzeugt automatisch eine **Context-Graph-Struktur**, die Abhängigkeiten und zeitliche Abfolgen abbildet. Dadurch können LLMs später auf einen vollständigen Projektkontext zugreifen, ohne dass du erneut prompten musst.

### 3. **Context Locking & Rollback**
Jeder Kontext-Snapshot wird mit einem **kryptografischen Hash** versehen und in einer **immutablen Context-DB** (SQLite + optional S3) gespeichert. Du kannst jederzeit auf einen früheren Stand zurücksetzen – ideal für Experimente mit verschiedenen Prompts oder Agenten-Konfigurationen. Kein „Prompt-Verlust“ mehr.

## So setzt du ContextLock ein (3 professionelle Workflows)

### Workflow 1: **Context-Persistenz für CI/CD-Pipelines**
1. **Initialisierung**: Installiere ContextLock via `pip install contextlock` und initialisiere ein Context-Repository mit `contextlock init --path ./context-store`.  
2. **Snapshot in Pipeline**: Füge in deiner CI/CD-Pipeline (z. B. GitHub Actions) einen Schritt ein: `contextlock snapshot --id build-123 --meta "Commit: $GITHUB_SHA"`. Dadurch wird der gesamte Build-Kontext (Logs, Fehler, Umgebungsvariablen) als Snapshot gespeichert.  
3. **Wiederherstellung**: Bei einem Fehler in der nächsten Pipeline rufst du `contextlock restore --id build-123` auf. Der LLM-Assistent erhält exakt denselben Kontext wie beim ersten Build – inklusive vorheriger Fehlermeldungen und Lösungsversuche.

### Workflow 2: **Agenten-Kontext für Multi-Step-RAG**
1. **Context-Erzeugung**: Starte einen Agenten mit `contextlock wrap --agent "my-rag-agent"`. ContextLock fängt automatisch alle Prompts, Retrieval-Ergebnisse und Antworten.  
2. **Context-Verkettung**: Verwende `contextlock link --from session-1 --to session-2`, um zwei Sitzungen zu verknüpfen. Der Agent kann nun auf Daten aus Sitzung 1 zugreifen, ohne erneut zu retrieven.  
3. **Export für Audit**: Exportiere den gesamten Context-Graph als JSON mit `contextlock export --format json --output audit.json`. Perfekt für Compliance oder Debugging.

### Workflow 3: **Context-bewusstes Prompt-Engineering**
1. **Context-Load**: Lade einen bestehenden Kontext in dein Prompt-Template: `contextlock load --id "arch-discussion-42"`.  
2. **Prompt-Injection**: Füge den geladenen Kontext als `{context}` in dein Prompt-Template ein. ContextLock sorgt dafür, dass nur relevante Teile (basierend auf Token-Limit) injiziert werden.  
3. **Lock & Test**: Nach erfolgreichem Prompt speicherst du den neuen Kontext mit `contextlock lock --id "prompt-v3"`. Du kannst später mit `contextlock diff --id1 "prompt-v2" --id2 "prompt-v3"` die Änderungen analysieren.

## Best Practices für Enterprise-Entwickler

1. **Context-Lebenszyklus definieren**: Nutze ContextLock nicht nur für Ad-hoc-Sitzungen, sondern integriere es in dein **Incident-Management**. Jeder Bug-Report erhält eine Context-ID. Verknüpfe sie mit dem Code-Review-Kontext. So entsteht ein **durchsuchbarer Context-Graph** für dein gesamtes Team.  
2. **Token-Budget pro Snapshot festlegen**: Setze mit `contextlock config --max-tokens 4000` ein hartes Limit pro Snapshot. ContextLock komprimiert dann automatisch, wenn das Limit überschritten wird. Das verhindert, dass LLMs mit zu viel Kontext überfordert werden und die Antwortqualität leidet.
