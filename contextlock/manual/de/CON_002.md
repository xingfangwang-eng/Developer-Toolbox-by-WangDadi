# contextlock - CON_002

## Übersicht

ContextLock ist das erste Open-Source-Framework, das **persistente KI-Kontexte** zwischen Sitzungen, Tools und Teams synchronisiert – ohne Cloud-Zwang, ohne Vendor-Lock-in. Entwickelt für Developer, die LLM-gestützte Workflows deterministisch und reproduzierbar halten müssen.

**Keywords:** AI Context Management, Persistent AI Memory, Developer Productivity Tool, LLM State Persistence, Context Locking

## Detaillierter Inhalt

### Was ist ContextLock?

ContextLock löst das fundamentale Problem der **Kontextflüchtigkeit** in KI-gestützten Entwicklungsumgebungen. Statt dass jeder Prompt, jede Chain und jeder Agent bei Null anfängt, friert ContextLock den gesamten Reasoning-Zustand ein – inklusive Embeddings, Session-Verlauf, Tool-Calls und User-Intents. Das Ergebnis: **deterministische Wiederholbarkeit** und **echte Multi-Session-Kohärenz**.

### Kernfunktionen

1. **Deterministischer Context Snapshot**  
   Erzwingt einen vollständigen, serialisierbaren Zustandsdump aller LLM-Interaktionen inklusive Token-Statistiken, Embedding-Vektoren und Metadaten. Jeder Snapshot ist kryptografisch signiert und kann mittels `contextlock verify --hash <sha256>` auf Integrität geprüft werden.

2. **Cross-Session Locking mit Diff-Merge**  
   Sperrt Kontextblöcke über mehrere Sitzungen hinweg – ideal für CI/CD-Pipelines oder Code-Review-Assistenten. Änderungen werden als granularer Patch gespeichert, kein vollständiger Rebuild nötig. Merge-Konflikte werden automatisch via `contextlock merge --strategy timestamp` gelöst.

3. **Context-Aware Tool Orchestration**  
   Bindet externe Tools (z. B. `grep`, `jq`, `curl`) direkt in den Kontext-Graphen ein. Tool-Outputs werden als eigenständige Context-Nodes gespeichert und können später referenziert werden – ohne erneute Ausführung. Ermöglicht **lazy evaluation** und **incremental computation**.

### So verwenden Sie es

1. **Initialen Kontext definieren und sperren**  
   ```bash
   contextlock init --workspace ./projekt --lockfile .contextlock.lock
   contextlock freeze --reason "Initialer Projekt-Kontext für Refactoring"
   ```  
   Erzeugt eine `.contextlock.lock`-Datei, die alle relevanten Dateien, Metadaten und den aktuellen LLM-Session-Status enthält. Der `--reason`-Flag wird als Commit-Message im internen Context-Log gespeichert.

2. **Kontext inkrementell aktualisieren und diffen**  
   ```bash
   contextlock update --add ./src/neu.js --reason "Neue Business-Logik hinzugefügt"
   contextlock diff --from HEAD~1 --to HEAD
   ```  
   Zeigt nur die geänderten Context-Nodes an (z. B. neue Embeddings, geänderte Prompt-Historie). Kein vollständiger Rebuild, kein Token-Verlust.

3. **Kontext in CI/CD integrieren**  
   ```yaml
   # .github/workflows/contextlock-ci.yml
   - name: ContextLock CI
     run: |
       contextlock checkout --ref ${{ github.sha }}
       contextlock verify --hash ${{ env.CONTEXTLOCK_HASH }}
       contextlock merge --strategy timestamp
   ```  
   Stellt sicher, dass jeder Build den exakt gleichen Kontext hat wie der letzte erfolgreiche Run. Ermöglicht reproduzierbare KI-gestützte Tests.

### Beste Praktiken

1. **Context Locking als Teil des Git-Workflows**  
   Führen Sie `contextlock freeze` nach jedem `git commit` aus. So bleibt der KI-Kontext immer synchron mit dem Code-Stand. Nutzen Sie `contextlock diff` im Code-Review, um zu sehen, ob ein PR den Kontext verändert hat – das verhindert unerwartete Side-Effects in KI-gestützten Tools.

2. **Granulare Context-Nodes für Microservices**  
   Definieren Sie pro Service einen eigenen ContextLock-Workspace. Verhindern Sie, dass ein fehlerhafter Kontext aus Service A in Service B überschwappt. Nutzen Sie `contextlock export --format json` für die serviceübergreifende Kommunikation – ohne Shared State, ohne Race Conditions.
