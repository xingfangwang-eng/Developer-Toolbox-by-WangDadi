# contextlock - CON_020

## Überblick

ContextLock ist ein KI-gestütztes Tool zur persistenten Kontextverwaltung, das speziell für Developer entwickelt wurde, die mit großen Sprachmodellen (LLMs) arbeiten. Es eliminiert das lästige Neuladen von Kontexten und sorgt für eine konsistente, wiederherstellbare Arbeitsumgebung – selbst über Sitzungsgrenzen hinweg. Optimiert für AI Context Management, Persistent AI Memory und maximale Developer Productivity.

## Detaillierter Inhalt

### Was ist ContextLock?

ContextLock ist kein weiterer Prompt-Manager. Es ist eine **kontextuelle Zustandsmaschine**, die den gesamten Dialogverlauf, System-Prompts und benutzerdefinierte Variablen in einem verschlüsselten, versionierten Speicher ablegt. Entwickler können so nahtlos zwischen Projekten, Branches oder sogar verschiedenen KI-Modellen wechseln, ohne den Kontext neu aufbauen zu müssen. Die Architektur basiert auf einer **Event-Sourcing-ähnlichen Logik**, bei der jeder Kontext-Change als unveränderliches Event protokolliert wird – ideal für Debugging und Auditing.

### Kernfunktionen

1. **Deterministisches Context-Rollback**: Jeder Kontext-Snapshot wird mit einem SHA-256-Hash signiert. Du kannst zu jedem beliebigen Zeitpunkt zurückkehren, ohne dass sich der Kontext durch latente Modelländerungen verschiebt. Perfekt für Regressionstests oder das Nachvollziehen von Fehlern in CI/CD-Pipelines.

2. **Multi-Model Context Bridging**: Wechsle nahtlos zwischen GPT-4, Claude 3 oder lokalen Modellen (z. B. Llama 3), während ContextLock die unterschiedlichen Token-Limits und Prompt-Formate automatisch normalisiert. Kein manuelles Umschreiben von System-Prompts mehr – das Tool passt die Kontextstruktur dynamisch an das Zielmodell an.

3. **Context-as-Code (CaC)**: Definiere Kontexte in YAML-Dateien, die in dein Repository eingecheckt werden können. Jeder Kontext enthält Metadaten (Projekt-ID, Branch, Benutzerrolle) und kann über CI/CD automatisch geladen werden. So bleibt der Kontext zwischen Teammitgliedern synchron – ohne manuelles Kopieren von Chat-Logs.

### So verwenden Sie es

1. **Initialen Kontext definieren und persistieren**  
   Erstelle eine `contextlock.yaml` im Stammverzeichnis deines Projekts. Definiere darin System-Prompts, Variablen (z. B. `{{project_name}}`) und den gewünschten Modell-Typ. Führe dann `contextlock init` aus – das Tool generiert einen eindeutigen Kontext-Hash und speichert ihn in einer lokalen SQLite-Datenbank. Der Kontext ist sofort für alle Sitzungen verfügbar.

2. **Kontext über Branch-Wechsel hinweg synchronisieren**  
   Angenommen, du wechselst von `feature/login` zu `hotfix/crash`. Normalerweise müsstest du den gesamten KI-Kontext neu aufbauen. Mit ContextLock reicht `contextlock switch --branch hotfix/crash`. Das Tool lädt automatisch den zuletzt gespeicherten Kontext für diesen Branch – inklusive aller benutzerdefinierten Variablen und System-Prompts. Falls kein Kontext existiert, wird ein neuer auf Basis der YAML-Definition erstellt.

3. **Context-Rollback für Debugging durchführen**  
   Wenn ein KI-generierter Code plötzlich fehlerhaft ist, führe `contextlock rollback --hash <hash>` aus. Das Tool stellt den exakten Kontext wieder her, der zum Zeitpunkt des fehlerhaften Outputs galt. Du kannst dann den Prompt-Verlauf analysieren und die Ursache isolieren. Der Rollback ist atomar und hinterlässt keine Seiteneffekte im aktuellen Kontext.

### Beste Praktiken

- **Kontext-Versionierung in CI/CD integrieren**: Binde `contextlock snapshot` in deine Pipeline ein, vor jedem Deployment. So hast du einen nachvollziehbaren Kontext für jeden Build. Falls ein Rollback nötig ist, kannst du den Kontext aus dem entsprechenden Snapshot laden – ohne auf die Erinnerungen deines Teams angewiesen zu sein.

- **Kontext-Sharing über Team-Grenzen hinweg standardisieren**: Lege für jedes Microservice-Team eine eigene `contextlock.yaml` an, die nur die relevanten System-Prompts und Variablen enthält. Teammitglieder können dann ihren persönlichen Kontext (z. B. persönliche API-Keys) in einer `.env`-Datei überschreiben, ohne die zentrale Konfiguration zu ändern. Das reduziert Konflikte und beschleunigt das Onboarding neuer Developer.
