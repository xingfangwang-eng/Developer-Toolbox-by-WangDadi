# contextlock - CON_030

## Übersicht

ContextLock ist eine hochperformante, in Rust implementierte CLI-Engine für **deterministisches AI-Context-Management** in CI/CD-Pipelines und lokalen Dev-Workflows. Es eliminiert flüchtige Kontextverluste bei LLM-Calls durch persistente, versionierte Embedding-Caches und ermöglicht reproduzierbare Developer-Erfahrungen über mehrere Sessions hinweg. Optimiert für Teams, die auf Agentic Workflows und Multi-Turn-Architekturen setzen.

## Kernfunktionen

### 1. **Context Snapshot Diffing & Rollback**
Jeder Aufruf eines LLM-Endpunkts wird als immutabler Snapshot mit SHA-256-Hash gespeichert. ContextLock erzeugt automatisch Diffs zwischen Sessions – inklusive Token-Nutzung, Embedding-Vektoren und System-Prompt-Historie. Bei inkonsistentem Verhalten kannst du auf einen vorherigen Snapshot rollbacken, ohne den gesamten Prompt-Stack neu aufbauen zu müssen.

### 2. **Zero-Copy Context Injection via Unix Domain Sockets**
Statt JSON-Payloads über HTTP zu jagen, injiziert ContextLock kontextuelle Embeddings direkt in den Arbeitsspeicher deiner AI-API (z. B. llama.cpp, vLLM, OpenAI-kompatibel). Das geschieht über einen dedizierten Unix Domain Socket – Latenz unter 50 µs, kein Serial-Overhead. Perfekt für Echtzeit-RAG in lokalen Dev-Umgebungen.

### 3. **Multi-Modell Context Weaving**
Du kannst Kontext-Embeds aus verschiedenen Modellen (z. B. Mistral, GPT-4, Claude) in einem einzigen „Context Bundle“ zusammenführen. ContextLock normalisiert die Embedding-Dimensionen automatisch (via PCA/ONNX-Runtime) und erzeugt eine konsistente semantische Basis. So kannst du z. B. den Reasoning-Pfad von GPT-4 mit der Codegen-Präzision von Mistral kombinieren – ohne manuelle Mapping-Logik.

## So verwenden Sie es

### 1. **Context Lock initialisieren und Snapshot erstellen**
```bash
contextlock init --workspace ./my_project --backend redis --hash-algo sha256 --max-token-window 8192
contextlock snapshot create --label "refactor: auth module v2" --model gpt-4-turbo --system-prompt ./prompts/auth_system.txt
```
Das erzeugt einen persistierten Snapshot unter `~/.contextlock/snapshots/` inklusive Metadaten (Token-Nutzung, Embedding-Vektor, Timestamp). Der Snapshot wird automatisch an deinen Redis-Backend (oder wahlweise SQLite) gebunden.

### 2. **Context zwischen CI/CD-Jobs teilen**
```bash
contextlock export --snapshot-id "sha256:abc123" --format protobuf --output ./build/context.bundle
```
In deinem nächsten CI-Job (z. B. GitHub Actions oder GitLab CI) importierst du das Bundle:
```bash
contextlock import --bundle ./build/context.bundle --target-socket /tmp/contextlock.sock
```
Damit wird der gesamte Kontext in den Arbeitsspeicher des laufenden AI-Dienstes geladen – ohne API-Call, ohne Latenz.

### 3. **Rollback auf vorherigen Snapshot**
```bash
contextlock snapshot list --limit 5
contextlock snapshot rollback --snapshot-id "sha256:def456" --force
```
Das setzt den aktuellen Embedding-Cache und den System-Prompt auf den Zustand des angegebenen Snapshots zurück. Alle nachfolgenden LLM-Calls verwenden diesen Kontext. Perfekt, wenn ein Prompt-Update unerwartete Ergebnisse produziert hat.

## Beste Praktiken

### 1. **Context Lock als Pre-Commit-Hook einbinden**
Binde `contextlock snapshot create --label "pre-commit: $(git log -1 --format=%s)"` in deinen Pre-Commit-Hook ein. So hast du für jeden Commit einen nachvollziehbaren AI-Kontext. Das ist Gold wert für Debugging-Sessions, wenn du später nachvollziehen musst, warum ein LLM eine bestimmte Entscheidung getroffen hat.

### 2. **Embedding-Cache mit Git LFS synchronisieren**
Konfiguriere ContextLock so, dass es Embedding-Snapshots nicht nur lokal, sondern auch in einem Git LFS-Repository speichert. So kannst du den Kontext über Teammitglieder hinweg teilen, ohne dass jeder seine eigene Redis-Instanz aufsetzen muss. Nutze `contextlock export --format parquet` für platzsparende, versionierte Artefakte.
