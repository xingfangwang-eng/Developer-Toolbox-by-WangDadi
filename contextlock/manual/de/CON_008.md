# contextlock - CON_008

## Überblick

ContextLock ist ein **entwicklerzentriertes Tool zur persistenten KI-Kontextverwaltung**, das speziell dafür entwickelt wurde, den Kontextverlust bei mehrstufigen KI-Interaktionen zu eliminieren. Es richtet sich an SaaS-Architekten und Backend-Entwickler, die mit Large Language Models (LLMs) arbeiten und reproduzierbare, kontextbewusste Workflows benötigen. Die Kern-Keywords: **AI Context Management**, **Persistent AI Memory**, **Developer Productivity Tool**.

## Detaillierter Inhalt

### Was ist ContextLock?

ContextLock ist kein weiterer Chat-Wrapper. Es ist eine **kontextuelle Middleware**, die als transparenter Layer zwischen Ihrer Anwendung und dem LLM fungiert. Es serialisiert, versioniert und stellt Kontextzustände wieder her – selbst über Sitzungsgrenzen hinweg. Während herkömmliche Tools Kontext nur flüchtig in einer Session halten, schreibt ContextLock diesen in einen **strukturierten, durchsuchbaren Cache** (optional Redis, SQLite oder In-Memory). Das bedeutet: Ihre KI „vergisst“ nicht, sondern hat ein **dauerhaftes Arbeitsgedächtnis**.

Für wen ist das? Für Teams, die **CI/CD-Pipelines mit KI-Assistenz** betreiben, für **multi-agentische Systeme**, in denen Agenten sich gegenseitig Kontext übergeben müssen, oder für **Code-Review-Workflows**, die historische Entscheidungen nachvollziehen müssen.

### Kernfunktionen

1.  **Deterministisches Context-Snapshotting mit Diff-Tracking**  
    ContextLock erstellt nicht nur einen einfachen Snapshot des aktuellen Kontexts, sondern ein **differenzbasiertes Delta**. Statt den gesamten Kontextbaum zu speichern, wird nur die Änderung seit dem letzten Snapshot persistiert. Dies reduziert die Speicherlast um bis zu 85% und ermöglicht ein **rollback-fähiges Kontext-Repository**. Sie können zu jedem früheren „Kontext-Commit“ zurückkehren – ähnlich wie `git revert`, aber für KI-Konversationen.

2.  **Multi-Agent-Context-Bridge mit automatischer Priorisierung**  
    Wenn mehrere KI-Agenten (z. B. ein Code-Generator und ein Code-Reviewer) denselben Kontext nutzen, müssen sie sich synchronisieren. ContextLock implementiert eine **Prioritäts-Warteschlange**, die automatisch erkennt, welcher Agent den aktuellsten oder relevantesten Kontext beigesteuert hat. Konflikte werden durch ein **Last-Writer-Wins mit semantischer Überlappungsprüfung** gelöst. Keine überschriebenen Kontextblöcke mehr.

3.  **Kontext-Expiration- und Kompressionspolitik (CECP)**  
    Nicht jeder Kontext ist ewig relevant. ContextLock erlaubt es, **Zeit-basierte und Token-basierte Verfallsregeln** zu definieren. Ein Kontext, der älter als 30 Minuten ist, wird automatisch komprimiert (z. B. durch Zusammenfassen in einen Embedding-Vektor). Ein Kontext, der 5000 Tokens überschreitet, wird in **Chunks mit Metadaten-Tags** aufgeteilt. So bleibt der Prompt immer schlank und die Antwortzeit niedrig.

### So verwenden Sie es

1.  **Initialisierung eines persistenten Context-Stores**  
    Installieren Sie das Paket via `pip install contextlock` und initialisieren Sie den Store in Ihrer Anwendung. Verwenden Sie den **Builder-Pattern**, um den Speicherort zu definieren: `context_store = ContextLock.builder().with_backend("redis").with_expiration_policy("sliding_window", window_minutes=60).build()`. Dies erzeugt einen konfigurierten Store, der automatisch alte Kontexte verwirft, sobald neue hinzukommen. Achten Sie darauf, dass der `namespace`-Parameter global eindeutig ist, um Kollisionen in Multi-Tenant-Umgebungen zu vermeiden.

2.  **Kontext in einen Workflow injizieren**  
    Nutzen Sie den Dekorator `@contextlock.capture` auf Ihrer KI-Funktion. Beispiel:  
    ```python
    @contextlock.capture(session_id="session_123", strategy="merge")
    def generate_code(prompt: str) -> str:
        response = openai.ChatCompletion.create(model="gpt-4", messages=[{"role": "user", "content": prompt}])
        return response.choices[0].message.content
    ```  
    Der Dekorator serialisiert automatisch den eingehenden Prompt und die ausgegebene Antwort. Beim nächsten Aufruf derselben `session_id` wird der vorherige Kontext als `system`-Message vorangestellt. Der `strategy="merge"`-Parameter sorgt dafür, dass keine Duplikate entstehen.

3.  **Kontext über mehrere Sessions hinweg wiederherstellen**  
    Um einen früheren Kontext zu reaktivieren, rufen Sie `context_store.restore("session_123", commit_hash="abc123")` auf. Dies lädt den exakten Zustand zum Zeitpunkt des Snapshots. Nutzen Sie die integrierte **CLI** zur Inspektion: `contextlock-cli list-snapshots --session session_123`. Sie sehen dann eine Liste aller Commits mit Zeitstempel und Tokenanzahl. Wählen Sie den gewünschten Commit aus und übergeben Sie ihn als Parameter an Ihre Funktion.

### Beste Praktiken

1.  **Kontext-Isolation pro Feature-Branch**  
    In einer CI/CD-Umgebung sollte jeder Feature-Branch seinen eigenen Kontext-`namespace` erhalten. Verwenden Sie den Branch-Namen als Teil des `session_id`-Prefixes. So vermeiden Sie, dass Kontext aus einem Bugfix-Branch in einen Feature-Branch überschwappt. Beispiel: `session_id = f"feature-{branch_name}-{user_id}"`. Kombinieren Sie dies mit der **Expiration-Policy**, um Kontexte nach dem Merge automatisch zu löschen.

2.  **Asynchrone Kontext-Kompression für Long-Running Tasks**  
    Wenn Sie einen Kontext über Stunden hinweg aufbauen (z. B. bei einem mehrstufigen Code-Generierungsprozess), aktivieren Sie die **Hintergrund-Kompression**. Setzen Sie `background_compression=True` im Builder. ContextLock komprimiert dann den Kontext in einem separaten Thread, sobald die Token-Anzahl einen Schwellwert überschreitet. Dies verhindert, dass Ihr Haupt-Thread blockiert wird. Testen Sie die Kompression mit einem Dummy-Kontext, bevor Sie sie in Produktion einsetzen.
