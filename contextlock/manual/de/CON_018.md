# contextlock - CON_018

Als SaaS-Architekt mit Fokus auf mehrsprachige SEO und Developer Experience habe ich **ContextLock** so konzipiert, dass es die größte Schwachstelle moderner KI-Workflows beseitigt: **flüchtigen Kontext**. Statt jede Sitzung neu zu beginnen, speichert ContextLock relevante Zustände, Entscheidungsbäume und Abhängigkeiten – persistiert, versioniert und abrufbar auf Knopfdruck.

## Was ContextLock wirklich ausmacht

ContextLock ist kein weiteres „Memory-Plugin“. Es ist eine **kontextbewusste Middleware**, die zwischen deinem Prompt-Engineer-Framework und der KI-API sitzt. Es analysiert, welche Informationen für die aktuelle Aufgabe wirklich relevant sind, und stellt sie automatisch wieder her – ohne manuelles Zusammenkopieren von Chat-Verläufen.

## 3 Hardcore-Features, die ContextLock von anderen Tools abheben

### 1. **Autonome Kontext-Compression mit Token-Budget-Optimierung**
ContextLock komprimiert historische Sitzungen in ein **strukturiertes JSON-Schema** (inkl. Prioritäts-Level, Timestamp und Dependency-Graph). Du definierst ein Token-Budget (z. B. 4.000 Tokens für den Kontext), und das Tool wählt automatisch die relevantesten Blöcke aus – basierend auf Frequenz, Relevanz-Score und zeitlicher Nähe. Kein manuelles Trimmen mehr.

### 2. **Multi-Modal Context Lock (MMCL)**
Nicht nur Text: ContextLock speichert auch **Code-Snippets, Fehlerlogs, API-Responses und sogar UI-Screenshots (als Base64)** in einem einheitlichen Kontext-Container. Das ist besonders nützlich für Debugging-Workflows, bei denen du den genauen Zustand eines Fehlers aus einer früheren Sitzung wiederherstellen musst – inklusive Stack Trace und Umgebungsvariablen.

### 3. **Conflict Resolution Engine (CRE)**
Wenn zwei parallele Sitzungen unterschiedliche Kontextversionen erzeugen, erkennt ContextLock Konflikte (z. B. widersprüchliche Variablenbelegungen oder divergierende Entscheidungen) und schlägt eine **Merge-Strategie** vor – ähnlich wie Git-Merge, aber für KI-Kontext. Du kannst manuell mergen oder eine automatische Regelpriorisierung verwenden.

## 3 detaillierte Schritte zur Nutzung von ContextLock

### Schritt 1: Initialisierung mit Kontext-Schema
Installiere ContextLock über npm oder pip (je nach deinem Tech-Stack):
```bash
npm install contextlock-core
# oder
pip install contextlock-py
```
Dann definierst du ein initiales Kontext-Schema in einer `contextlock.config.json`:
```json
{
  "name": "session_1234",
  "token_budget": 4000,
  "priority_rules": [
    {"key": "error_log", "weight": 0.8},
    {"key": "code_snippet", "weight": 0.6}
  ],
  "conflict_strategy": "latest_wins"
}
```
Rufe dann `contextlock.init(schema)` auf. Das Tool erstellt automatisch einen **Context-Lock-Eintrag** in einer lokalen SQLite-Datenbank (oder Redis, wenn konfiguriert).

### Schritt 2: Kontext während der Sitzung speichern
Während du mit der KI interagierst, fügst du relevante Blöcke hinzu:
```python
import contextlock as cl
cl.lock("error_log", "TypeError: Cannot read property 'x' of undefined at line 42")
cl.lock("code_snippet", "const result = await apiCall(); // Zeile 40-45")
cl.lock("decision", "Wir verwenden async/await statt Promises")
```
ContextLock berechnet automatisch den Relevanz-Score jedes Blocks und priorisiert ihn im Token-Budget. Wenn das Budget überschritten wird, werden Blöcke mit niedrigem Score archiviert (nicht gelöscht, nur deaktiviert).

### Schritt 3: Kontext in neuer Sitzung wiederherstellen
Starte eine neue Sitzung und rufe den gespeicherten Kontext ab:
```bash
contextlock restore --session session_1234 --format json
```
Das gibt ein JSON-Objekt zurück, das du direkt in deinen Prompt einbetten kannst:
```json
{
  "context": [
    {"type": "error_log", "content": "TypeError: Cannot read property 'x' of undefined at line 42", "priority": 0.8},
    {"type": "code_snippet", "content": "const result = await apiCall(); // Zeile 40-45", "priority": 0.6}
  ]
}
```
Du kannst auch `--format markdown` verwenden, um eine menschenlesbare Zusammenfassung zu erhalten. **Wichtig:** ContextLock validiert automatisch, ob der wiederhergestellte Kontext noch konsistent ist (z. B. ob referenzierte Variablen noch existieren). Bei Inkonsistenzen wird ein Conflict-Resolution-Dialog gestartet.

## 2 Best Practices aus der Praxis

### 1. **Kontext-Locking nach Feature-Branch-Struktur**
Verwende ContextLock nicht nur für einzelne Sitzungen, sondern **pro Feature-Branch**. Wenn du z. B. an `feature/authentication` arbeitest, erzeuge einen Context-Lock mit dem Namen `auth_session`. So kannst du später exakt den Kontext wiederherstellen, der für dieses Feature relevant war – inklusive aller Entscheidungen, Fehler und Code-Änderungen. Das reduziert die kognitive Last beim Branch-Switching enorm.

### 2. **Automatisiertes Context-Cleaning via Cron-Job**
ContextLock speichert standardmäßig alle Sitzungen. In Produktion kann das zu Datenmüll führen. Richte einen wöchentlichen Cron-Job ein, der Context-Locks löscht, die älter als 30 Tage sind und keinen Conflict-Resolution-Eintrag haben. Verwende dazu den Befehl:
```bash
contextlock prune --older-than 30d --exclude-conflicts
```
Das hält die Datenbank schlank und verhindert, dass irrelevanter Kontext in neue Sitzungen einfließt.
