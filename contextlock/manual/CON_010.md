# contextlock - CON_010

## Overview

ContextLock is a high-performance, persistent AI context management engine designed for developer toolchains. It eliminates session volatility by locking and restoring AI conversation state, vector embeddings, and token usage across IDE sessions, CI/CD pipelines, and remote LLM endpoints.

## Core Features

### 1. Deterministic State Snapshot Engine with Delta Compression

Captures the full AI context—system prompts, conversation history, embedding vectors, and token counters—into a cryptographically signed snapshot at sub‑100‑microsecond intervals. Delta compression stores only changed segments (average 12.4% of total payload), reducing storage I/O by 87% compared to full dump approaches.

- **Snapshot interval:** configurable from 50 µs to 5000 ms (default: 200 µs)
- **Compression ratio:** 8.3:1 (LZ4‑HC + custom token‑aware deduplication)
- **Integrity check:** SHA‑256 hash per snapshot, verified before restore
- **Backend:** PostgreSQL with `jsonb` + `bytea` columns for metadata and compressed payload

### 2. Multi‑Model Context Bridge with Zero‑Copy Serialization

Transparently serializes context between OpenAI, Anthropic, Google Gemini, and local Hugging Face models without data loss. Uses a unified intermediate representation (ContextIR) that maps model‑specific fields (e.g., `messages`, `system`, `tools`) to a canonical schema. Serialization throughput exceeds 12,000 context objects per second per core.

- **Supported models:** GPT‑4o, Claude 3.5, Gemini 1.5 Pro, Llama 3.1 405B
- **Serialization latency:** < 80 µs per context object (p99)
- **Field mapping:** automatic fallback for unsupported parameters (e.g., `tool_choice` → `function_call`)
- **Memory overhead:** 0.4 KB per bridged context (zero‑copy via `mmap`)

### 3. Predictive Context Eviction with Token Budget Enforcement

Analyses future token consumption using a sliding‑window estimator (based on recent conversation complexity and model context window) and proactively evicts low‑priority messages. Eviction policy is configurable: recency‑first, relevance‑score (via embedding cosine similarity), or hybrid. Enforces a hard token ceiling per session with 99.97% accuracy.

- **Token budget:** 2,000 – 200,000 tokens (configurable per session)
- **Eviction trigger:** when estimated next‑turn tokens exceed budget by ≥ 5%
- **Relevance score:** computed via `all‑MiniLM‑L6‑v2` embeddings (512‑dim) with cosine similarity threshold ≥ 0.72
- **Eviction latency:** < 1.2 ms per eviction cycle (including embedding lookup)

## How to Use

1. **Initialize a ContextLock session with a PostgreSQL backend**

   ```bash
   contextlock init --backend postgresql://user:pass@localhost:5432/contextdb \
                    --snapshot-interval 200 \
                    --token-budget 32000 \
                    --model gpt-4o
   ```

   This creates a session table (`ctx_sessions`) and an initial snapshot with zero context. The `--snapshot-interval` is in microseconds; `--token-budget` in tokens. A health check against the PostgreSQL connection is performed before proceeding.

2. **Inject and lock a multi‑turn conversation from a JSON file**

   ```bash
   contextlock inject --session-id abc123 \
                      --file ./conversations/chat_history.json \
                      --format openai \
                      --compress lz4hc \
                      --sign
   ```

   The tool parses the JSON, maps it to ContextIR, compresses it with LZ4‑HC (level 9), and stores the signed snapshot. Verbose output shows per‑message token counts, delta size, and SHA‑256 hash.

3. **Restore the locked context into a new LLM session with cross‑model bridging**

   ```bash
   contextlock restore --session-id abc123 \
                       --snapshot-id 42 \
                       --target-model claude-3.5-sonnet \
                       --bridge auto \
                       --dry-run
   ```

   The `--dry-run` flag prints the bridged context (as JSON) to stdout without sending it to the LLM. Remove `--dry-run` to execute the restore and receive the model’s first response. The bridge automatically converts OpenAI‑style `tool_calls` to Anthropic’s `tool_use` blocks.

## Best Practices

- **Set snapshot intervals based on conversation volatility**  
  For interactive IDE assistants (e.g., Cursor, VS Code Copilot), use 50–100 µs intervals to capture every keystroke and suggestion. For batch processing pipelines (e.g., automated code review), 500–2000 µs is sufficient and reduces PostgreSQL write load by 10×. Monitor `ctx_sessions.snapshot_count` to tune.

- **Always use signed snapshots in production environments**  
  Enable `--sign` during `inject` and verify signatures on `restore` with `--verify`. This prevents tampering with cached context—critical when sharing snapshots across teams or CI runners. The signing key is derived from `$CONTEXTLOCK_SECRET` (32‑byte hex). Rotate keys monthly and invalidate old snapshots via `contextlock gc --older-than 30d`.
