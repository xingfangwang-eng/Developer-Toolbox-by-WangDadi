# contextlock - CON_002

## 概要

ContextLock は、AI 開発における**コンテキスト永続化**と**メモリ管理**の課題を解決する、次世代型デベロッパーツールです。大規模言語モデル（LLM）との対話において、セッションを超えたコンテキスト保持が可能になり、開発生産性を劇的に向上させます。本ドキュメントでは、その中核機能と実践的な活用方法を詳述します。

## 詳細な内容

### ContextLock とは

ContextLock は、AI エージェントやチャットボットの開発現場で頻発する「コンテキスト消失問題」を解決するために設計された、**プロフェッショナル向け永続メモリ管理ツール**です。従来のセッションベースのメモリ管理では、会話のたびにコンテキストがリセットされ、開発者は同じ情報を何度も入力する必要がありました。ContextLock は、**セマンティックベクトルインデックス**と**トランザクションログベースの差分更新**を組み合わせることで、コンテキストの完全性とパフォーマンスを両立します。

### コア機能

1. **セマンティック・コンテキストロック**
   コンテキストデータをベクトル化し、**意味的類似性**に基づいて自動的にロック・解放を行います。例えば、開発中のAPI仕様をロックすると、その仕様に関連する会話が行われた際に自動的にコンテキストが復元されます。これにより、手動でのコンテキスト切り替えが不要になります。

2. **トランザクションログベースの差分更新**
   コンテキスト全体を毎回保存するのではなく、**変更差分のみをトランザクションログとして記録**します。これにより、ストレージ消費を最小限に抑えつつ、過去の任意の時点のコンテキスト状態を正確に復元できます。Git のような履歴管理が可能で、開発中のデバッグやロールバックが容易です。

3. **マルチテナント・コンテキスト分離**
   複数のプロジェクトや複数のAIエージェントが同時に動作する環境でも、**コンテキストを完全に分離**して管理します。テナントごとに独立したベクトルインデックスとトランザクションログを持ち、データの混在やリークを防止します。これにより、マイクロサービスアーキテクチャでの利用が安全に行えます。

### 使い方

1. **コンテキストロックの作成と設定**
   ```bash
   contextlock lock --name "api_spec_v2" --scope "project:myapp" --ttl 3600
   ```
   上記コマンドで、`api_spec_v2` という名前のコンテキストロックを作成します。`--scope` でスコープ（プロジェクトやテナント）を指定し、`--ttl` で有効期限（秒）を設定します。作成後、ロックは自動的にベクトル化され、セマンティック検索の対象となります。

2. **コンテキストデータの投入と更新**
   ```python
   from contextlock import ContextLockClient

   client = ContextLockClient(api_key="your_api_key")
   client.push_context(
       lock_id="api_spec_v2",
       data={"endpoint": "/users", "method": "POST", "params": ["name", "email"]},
       operation="upsert"
   )
   ```
   この例では、Python SDK を使用してコンテキストデータを投入しています。`operation="upsert"` により、既存データがあれば更新、なければ新規作成を行います。トランザクションログには差分のみが記録されるため、大量データの更新でもパフォーマンスが低下しません。

3. **コンテキストの復元と検証**
   ```bash
   contextlock restore --lock "api_spec_v2" --timestamp "2025-03-15T10:00:00Z" --dry-run
   ```
   `--dry-run` フラグを使用すると、実際に復元せずに復元される差分をプレビューできます。これにより、意図しない状態への復元を防止します。復元後は、`contextlock diff` コマンドで現在のコンテキストとの差分を確認可能です。

### ベストプラクティス

1. **コンテキストロックの粒度を適切に設定する**
   ロックの粒度が細かすぎると管理コストが増大し、粗すぎるとコンテキストの復元精度が低下します。推奨される粒度は、**1つのAPIエンドポイント**や**1つのビジネスロジックユニット**単位です。例えば、ユーザー管理機能全体を1つのロックにするのではなく、`/users` エンドポイントと `/orders` エンドポイントで別々のロックを作成します。これにより、部分的な更新や復元が容易になります。

2. **トランザクションログのクリーンアップ戦略を策定する**
   トランザクションログは蓄積され続けるため、定期的なクリーンアップが必要です。`contextlock cleanup --older-than 30d` コマンドで、30日以上前のログを自動的にアーカイブまたは削除できます。ただし、監査要件がある場合は、削除前にログを外部ストレージにエクスポートすることを推奨します。また、`--compress` オプションを使用して、古いログを圧縮保存することも可能です。
