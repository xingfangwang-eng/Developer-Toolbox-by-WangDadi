# contextlock - CON_007

## 概要

ContextLock は、AI エージェントのコンテキストウィンドウをセッション間で永続化し、推論の一貫性を担保するためのメモリ管理基盤です。キーワード：AI Context Management, Persistent AI Memory, Developer Productivity Tool。

## 詳細な内容

### ContextLock とは

ContextLock は、大規模言語モデル（LLM）がセッションを跨いで失いがちな「文脈情報」を、PostgreSQL 上のタイムスタンプ付きベクトルインデックスとして保存・復元する専用ミドルウェアです。開発者が明示的に指定したコンテキストスコープ（例：プロジェクトID、スレッドID）に基づき、推論時のプロンプトに自動注入することで、AI の応答品質を安定させます。

### コア機能

1. **コンテキストスッキングエンジン**
   - パラメータ：`--context-ttl 3600000`（ミリ秒単位の生存期間）、`--max-tokens 8192`（最大トークン数）
   - 技術指標：シャーディングによる水平分散対応、PostgreSQL の `pgvector` 拡張を用いた類似度検索（cosine similarity, 0.0〜1.0）
   - 説明：セッション終了時に直近の会話履歴を圧縮・ベクトル化し、次回起動時に自動復元。復元精度は 97.3% 以上（内部ベンチマーク値）。

2. **コンテキストロック機構**
   - パラメータ：`--lock-scope namespace:project:thread`、`--conflict-resolution last-write-wins`
   - 技術指標：Redis ベースの分散ロック（TTL 30秒、リトライ間隔 100ms）、書き込み競合時の自動マージ機能
   - 説明：複数の AI エージェントが同一コンテキストを同時編集する際、排他制御を実施。競合が発生した場合、タイムスタンプが最新の変更を優先し、古い変更は差分ログとして保存。

3. **コンテキストエクスポート/インポートパイプライン**
   - パラメータ：`--format jsonl`、`--include-metadata true`、`--compress gzip`
   - 技術指標：エクスポート速度 250MB/秒（SSD 環境）、インポート時のバリデーションチェック（スキーマ不一致検出）
   - 説明：コンテキストデータを JSONL 形式で出力し、別環境への移行やバックアップを可能に。メタデータには作成日時、更新回数、ロック状態が含まれる。

### 使い方

1. **初期化とコンテキストスコープの設定**
   ```bash
   contextlock init --project my-ai-app --namespace production --pg-uri postgresql://user:pass@host:5432/contextdb
   ```
   - 内部処理：指定された PostgreSQL に `context_vectors` テーブルを作成し、`pgvector` 拡張を有効化。スコープごとにユニークな UUID を発行。

2. **コンテキストの書き込みとロック**
   ```python
   from contextlock import ContextLock
   cl = ContextLock(host="localhost", port=6379)
   with cl.lock(scope="namespace:project:thread", timeout=5000):
       cl.write({"user_input": "最新の要件を教えて", "response": "以下が要件です..."})
   ```
   - 内部処理：Redis にロックキー `lock:namespace:project:thread` を設定。書き込み成功後、PostgreSQL にベクトル化されたデータを upsert。

3. **コンテキストの復元と検証**
   ```bash
   contextlock restore --scope namespace:project:thread --similarity-threshold 0.85
   ```
   - 内部処理：指定スコープ内の全ベクトルを cosine similarity でスコアリングし、閾値以上のレコードをプロンプトに注入。復元後、整合性ハッシュを出力して改ざん検知。

### ベストプラクティス

- **プラクティス 1：コンテキストの有効期限をワークフローに合わせて設定する**
  短期タスク（例：コードレビュー）では `--context-ttl 300000`（5分）に設定し、長期プロジェクト（例：ドキュメント管理）では `--context-ttl 86400000`（24時間）に設定。TTL を短くしすぎると復元精度が低下し、長くしすぎるとストレージコストが増大するため、タスクの性質に応じて調整する。

- **プラクティス 2：競合解決戦略を事前にチームで合意する**
  複数エージェントが同時に同一コンテキストを更新する場合、`--conflict-resolution last-write-wins` はシンプルだが、重要な情報が失われるリスクがある。代わりに `merge` モード（デフォルトでは未実装、カスタムプラグインが必要）を採用し、競合箇所を自動マージする戦略を推奨。チーム内で「何をロックするか」のルールをドキュメント化する。
