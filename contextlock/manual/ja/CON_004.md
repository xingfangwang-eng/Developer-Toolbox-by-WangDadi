# contextlock - CON_004

## 概要

ContextLock は、大規模言語モデル（LLM）の推論コンテキストをロックし、セッション間で一貫性のある永続メモリを提供する開発者向けプロダクティビティツールです。AI エージェントの状態管理における「コンテキストドリフト」問題を解消し、長期間の対話や複雑なタスクチェーンでも精度を維持します。

## 詳細な内容

### ContextLock とは

ContextLock は、AI の推論コンテキストを「ロック」し、特定のワークフローやプロジェクトに対して静的なメモリ領域を確保するアーキテクチャです。従来のプロンプトエンジニアリングでは扱えなかった、**コンテキストウィンドウの断片化**や**メモリ競合**を解決します。PostgreSQL のトランザクション分離レベルに着想を得た設計で、複数の AI エージェントが同一のコンテキストを安全に共有・更新できます。

### コア機能

#### 1. コンテキストスナップショットロック（CSL）

特定の時点のコンテキスト全体を **ミリ秒単位** でスナップショットし、不変のロックとして保存します。これにより、後続の推論が過去の状態に依存する場合でも、**タイムトラベルデバッグ** が可能です。スナップショットは `snapshot_id`（UUIDv7）、`locked_at`（Unix タイムスタンプ、ミリ秒精度）、`context_hash`（SHA-256）で管理され、PostgreSQL の `pg_repack` と連携してストレージフラグメンテーションを防止します。

#### 2. セマンティックコンテキストマージ（SCM）

複数のコンテキストロックを、**ベクトル類似度（コサイン類似度閾値: 0.92）** に基づいて自動マージする機能です。重複する情報を排除しつつ、矛盾が生じた場合は `conflict_resolution_mode` パラメータ（`latest`、`majority`、`manual`）に従って解決します。マージ結果は `merge_log` テーブルに JSONB 形式で記録され、後からトレーサビリティを確保できます。

#### 3. コンテキストアンカーチェーン（CAC）

長期間のワークフローにおいて、コンテキストロックをチェーン状に連結する機能です。各ノードは `anchor_id`、`parent_anchor_id`、`ttl`（Time-To-Live、デフォルト: 3600 秒）を持ち、チェーン全体の整合性は **Merkle Tree** で検証されます。これにより、AI エージェントが途中でコンテキストを失っても、チェーンを再構築して中断箇所から再開できます。

### 使い方

1. **コンテキストロックの生成とバインド**  
   `POST /api/v1/contextlock/lock` エンドポイントに、`project_id`（必須）、`context_type`（`chat`、`code`、`workflow` のいずれか）、`initial_context`（JSONB）を送信します。レスポンスとして `lock_id` と `snapshot_id` が返却されます。生成後は、`lock_id` を HTTP ヘッダー `X-ContextLock-ID` にセットして、後続の API 呼び出しにバインドします。

2. **コンテキストの部分更新とロック競合解決**  
   `PATCH /api/v1/contextlock/{lock_id}` で、`delta`（差分 JSON）と `conflict_resolution_mode` を指定して更新します。同時に複数のエージェントが同一ロックを更新しようとした場合、**楽観的ロック**（`version` フィールドによる楽観的同時実行制御）が適用され、競合が発生した場合は `409 Conflict` が返ります。その際、`conflict_resolution_mode` に基づいて自動解決するか、手動での解決を促します。

3. **コンテキストチェーンの検証とロールバック**  
   `GET /api/v1/contextlock/{lock_id}/chain` でチェーン全体を取得し、`/verify` エンドポイントで Merkle Tree の整合性を検証します。不正なノードが検出された場合、`rollback_to_anchor` パラメータで特定のアンカーまでロールバックできます。ロールバック後は、`rollback_log` にイベントが記録され、監査証跡として利用可能です。

### ベストプラクティス

- **コンテキストロックの粒度をタスク単位に設計する**  
  プロジェクト全体を1つのロックで管理するのではなく、**個々のタスクやサブワークフロー** ごとにロックを分割してください。例えば、コード生成タスクとドキュメント作成タスクで別々のロックを使用することで、コンテキスト競合を最小限に抑えられます。推奨されるロック数は、1プロジェクトあたり **5〜20** です。

- **TTL を適切に設定し、古いコンテキストを自動クリーンアップする**  
  コンテキストアンカーチェーンの `ttl` パラメータは、ワークフローの特性に応じて設定します。短期間のチャットセッションでは **300 秒**、長期間の開発プロジェクトでは **86400 秒（24時間）** が推奨されます。期限切れのロックは、バックグラウンドジョブ（cron で 5 分ごと実行）によって自動的にアーカイブされ、`archived_locks` テーブルに移動されます。
