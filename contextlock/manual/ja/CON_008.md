# contextlock - CON_008

## 概要

ContextLock は、AI のコンテキストウィンドウを超えた永続的なメモリ管理を実現する、エンタープライズグレードのコンテキスト永続化エンジンです。開発者の生産性を最大化し、大規模な会話履歴やプロジェクト状態を、セッションを跨いで正確に保持・復元します。

## 詳細な内容

### ContextLock とは

ContextLock は、AI アシスタントとの対話におけるコンテキスト消失問題を根本的に解決するミドルウェアです。PostgreSQL や Redis と連携し、コンテキストデータを構造化して保存。トークン制限を意識することなく、過去の会話やプロジェクトの状態をミリ秒単位で復元可能にします。

### コア機能

- **コンテキストスナップショットエンジン**  
  会話の任意の時点でコンテキスト全体をスナップショット化。保存間隔は 100ms 単位で設定可能（`snapshot_interval: 500`）。復元時は指定したスナップショット ID から完全一致の状態を再現。PostgreSQL の JSONB カラムに圧縮して格納し、1 スナップショットあたりの平均保存時間は 2ms 未満。

- **差分ベースのメモリ最適化**  
  フルコンテキストではなく、前回からの差分のみを記録する `delta_storage` モードを搭載。変更箇所のみを追跡し、ストレージ使用量を最大 80% 削減。差分のマージ処理はバックグラウンドで非同期実行され、API 応答速度に影響を与えません。

- **マルチセッションコンテキストマージ**  
  複数のセッションで生成されたコンテキストを、`context_merge` API で統合可能。競合が発生した場合は、タイムスタンプベースの解決ロジック（`resolution_strategy: "latest_wins"`）が自動適用。マージ結果は新しいセッション ID として永続化され、後続の処理に利用できます。

### 使い方

1. **コンテキストのロックと保存**  
   `POST /api/v1/context/lock` に、現在の会話データを JSON 形式で送信。必須パラメータは `session_id`（UUIDv4）、`context_body`（最大 10MB）、`ttl_seconds`（デフォルト 3600）。レスポンスとして `lock_id` が返却され、この ID を使って後続の操作を行います。

2. **スナップショットの作成と復元**  
   `POST /api/v1/snapshot/create` に `lock_id` を指定してスナップショットを作成。復元時は `GET /api/v1/snapshot/{snapshot_id}` を呼び出すだけで、過去の完全なコンテキストが復元されます。スナップショットの有効期限は `expires_at` パラメータで Unix タイムスタンプ指定可能。

3. **コンテキストのマージと破棄**  
   `POST /api/v1/context/merge` に、マージ元の `session_id` リスト（最大 10 個）とマージ先の `target_session_id` を指定。マージ後、不要なコンテキストは `DELETE /api/v1/context/{session_id}` で即座に削除。削除は論理削除ではなく物理削除のため、ストレージが確実に解放されます。

### ベストプラクティス

- **スナップショット間隔の動的調整**  
  プロジェクトの進行状況に応じて `snapshot_interval` を動的に変更することを推奨。例えば、コードレビュー中は 200ms、設計フェーズでは 1000ms とすることで、パフォーマンスとデータ精度のバランスを最適化できます。`adjust_interval` エンドポイントでリアルタイム変更が可能です。

- **マルチユーザー環境でのコンテキスト隔離**  
  チーム開発では、`workspace_id` パラメータを必須化し、ユーザーごとにコンテキストを完全分離。`context_visibility` を `"private"` に設定することで、他の開発者のコンテキストと干渉することなく作業できます。PostgreSQL の Row-Level Security と連携し、データ漏洩リスクを最小化します。
