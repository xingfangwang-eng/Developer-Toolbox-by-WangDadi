# contextlock - CON_013

## 概要

ContextLock は、AI コンテキストの永続化と管理を極限まで最適化するための開発者向けツールです。AI セッション間でのメモリ断片化を解消し、長期的なプロジェクトコンテキストを維持することで、開発効率を飛躍的に向上させます。本プロジェクトは「AI Context Management」「Persistent AI Memory」「Developer Productivity Tool」をキーワードに、SEO 対策も徹底しています。

## 詳細な内容

### ContextLock とは？

ContextLock は、AI アシスタントとの対話における「コンテキスト消失問題」を根本的に解決するプロフェッショナルツールです。従来の AI チャットでは、セッションを跨ぐたびにコンテキストがリセットされ、同じ説明を繰り返す無駄が発生していました。ContextLock は、トークンレベルのコンテキストスナップショットを外部ストレージに永続化し、必要なときに瞬時に復元します。これにより、大規模リファクタリングや複数フェーズにわたるコードレビューでも、一貫性のある AI アシスタンスを実現します。

### コア機能

1. **コンテキストスナップショットの自動差分管理**  
   各セッション終了時に、現在のコンテキストを前回のスナップショットと比較し、差分のみを保存します。これにより、ストレージ消費を最小限に抑えつつ、過去の任意の時点のコンテキストを完全に復元可能です。内部的には Merkle Tree 構造を利用し、整合性チェックを自動実行します。

2. **トークンアロケーションベースのコンテキストプライオリティ設定**  
   プロジェクト内のファイルやモジュールごとにトークン割り当て上限を設定可能。AI が優先すべきコンテキストを明示的に指定できるため、大規模コードベースでも必要な情報だけを効率的に保持します。例えば、`contextlock.json` に `"priority": {"src/core/": 0.7, "tests/": 0.3}` と記述することで、コアロジックに 70% のトークンを割り当てます。

3. **マルチセッションコンテキストマージ機能**  
   複数の AI セッションで得たコンテキストを、コンフリクト解決アルゴリズムを用いて自動マージします。例えば、Session A で定義した関数 `validateUser()` と Session B で修正した `handleAuth()` を、依存関係を解析しながら統合。マージ結果は YAML 形式でエクスポート可能です。

### 使い方

1. **初期セットアップとコンテキストキャプチャ**  
   プロジェクトルートで `contextlock init` を実行し、`.contextlock/` ディレクトリを作成します。次に、AI アシスタントとの対話開始時に `contextlock capture --session "refactor-auth"` でスナップショットを取得。このコマンドは、現在の作業ディレクトリの全ファイルのハッシュと、AI との直近の会話ログをバンドルして保存します。保存先はデフォルトで `~/.contextlock/snapshots/` ですが、環境変数 `CONTEXTLOCK_STORAGE` で任意のパスに変更可能です。

2. **コンテキストの復元と差分適用**  
   新しいセッションを開始する際、`contextlock restore --session "refactor-auth" --timestamp "2023-10-15T14:30:00Z"` で過去のコンテキストを復元します。復元後、`contextlock diff --session "refactor-auth"` を実行すると、現在のファイル状態とスナップショット間の差分がカラーコード付きで表示されます。この差分を AI プロンプトに直接埋め込むことで、コンテキストの継続性を担保します。

3. **コンテキストのエクスポートと共有**  
   チームメンバーとコンテキストを共有する場合、`contextlock export --session "refactor-auth" --format json` で JSON ファイルとしてエクスポート。エクスポートされたファイルには、トークン使用量、スナップショットサイズ、依存関係グラフが含まれます。受信側は `contextlock import ./shared_context.json` で即座にインポート可能。さらに、`contextlock verify --session "refactor-auth"` で整合性チェックを実行し、改ざんや破損を検出します。

### ベストプラクティス

1. **プロジェクトフェーズごとにセッションを分割する**  
   大規模なリファクタリングや機能追加では、フェーズごとに異なるセッション名を付けてスナップショットを管理します。例えば、`init`, `core-refactor`, `test-coverage`, `deploy-prep` のように分割。これにより、特定のフェーズのみをロールバックしたり、他のフェーズのコンテキストとマージする際の依存関係が明確になります。また、`contextlock list --status active` で進行中のセッションを一覧表示し、不要なスナップショットは `contextlock prune --older-than 30d` で自動削除することを推奨します。

2. **トークンアロケーションをコードベースのホットパスに合わせる**  
   `contextlock.json` の `priority` 設定は、プロジェクトのホットパス（頻繁に変更されるコアモジュール）に重点的に割り当てます。例えば、`src/core/` に 60%、`src/utils/` に 20%、`tests/` に 20% と設定。これにより、AI が最も重要なコンテキストを優先的に保持し、不要なノイズを排除します。また、定期的に `contextlock analyze --session "refactor-auth"` を実行し、トークン使用量のトレンドを可視化。閾値を超えた場合は自動でアラートを発する `--threshold 0.8` オプションを併用すると効果的です。
