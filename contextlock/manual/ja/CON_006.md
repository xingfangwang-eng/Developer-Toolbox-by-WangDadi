# contextlock - CON_006

## 概要

ContextLock は、AI コンテキストの永続化と整合性を担保するためのエンタープライズ向けツールです。開発チームが LLM との対話セッションを跨いで、コンテキストの一貫性を維持することを可能にします。特に、マルチエージェントシステムや長時間実行されるタスクの途中で発生するコンテキストドリフトを、PostgreSQL を用いたトランザクション管理とミリ秒単位のロック機構で防止します。

## 詳細な内容

### とは ContextLock?

ContextLock は、AI ワークフローにおけるコンテキストの断絶問題を解決するために設計された、プロフェッショナルツールです。従来のセッション管理では、LLM が過去のやり取りを忘れたり、複数のスレッドで矛盾した情報が混在する「コンテキスト汚染」が発生します。ContextLock は、各コンテキストに一意のロックをかけ、排他制御とバージョン管理を実現します。これにより、AI エージェントが常に最新かつ整合性の取れた状態で動作することを保証します。

### コア機能

- **コンテキストアトミックロック (CAL)**: コンテキストの読み取りから書き込みまでの一連の操作を、PostgreSQL の `SELECT ... FOR UPDATE` とカスタムロックテーブルを用いてアトミックに実行します。ロックのタイムアウトは 500 ミリ秒単位で設定可能で、デッドロックを自動検出し、ロールバックします。パラメータ `lock_timeout_ms` で調整可能。
- **コンテキストバージョニングエンジン (CVE)**: 各コンテキストに自動インクリメントのバージョン番号を割り当て、変更履歴をすべて保存します。`context_id` と `version` の複合キーで検索可能。これにより、過去の任意の時点のコンテキストを復元でき、A/B テストや障害時のロールバックが可能です。`max_versions_per_context` パラメータで保持バージョン数を制限。
- **コンテキストマージング API**: 複数のエージェントが並行して生成したコンテキストの差分を、3-way マージアルゴリズムで統合します。競合が発生した場合は、`conflict_resolution_strategy` パラメータ (`latest_wins`, `manual`, `custom_function`) で動作を指定可能。マージ結果は、`merge_result` として JSON 形式で返却され、コンフリクト箇所は `conflict_list` フィールドにリストアップされます。

### 使い方

1. **コンテキストの作成とロック取得**:
   ```bash
   contextlock lock --context-id "session_001" --lock-timeout 2000
   ```
   このコマンドは、`session_001` に対する排他的ロックを取得します。ロックが取得できない場合は、2 秒後にタイムアウトエラーが返ります。成功すると、`lock_token` と `current_version` が返されます。

2. **コンテキストの更新とバージョン管理**:
   ```bash
   contextlock update --context-id "session_001" --lock-token "abc123" --data '{"user_query": "新しい質問"}' --version 3
   ```
   更新時には、必ず `lock_token` と現在の `version` を指定します。サーバー側でバージョンの競合チェックが行われ、不整合があれば `409 Conflict` が返ります。成功すると、新しい `version` 番号 (例: 4) が返却されます。

3. **コンテキストのマージと競合解決**:
   ```bash
   contextlock merge --context-id "session_001" --branch "agent_b" --conflict-resolution "latest_wins"
   ```
   分岐したコンテキスト `agent_b` をメインブランチにマージします。`latest_wins` を指定すると、競合時は常に `agent_b` のデータが優先されます。マージ後、`merge_version` が発行され、コンフリクトの詳細が `merge_report` として出力されます。

### ベストプラクティス

- **ロックの粒度を適切に設計する**: コンテキストロックは、セッション全体ではなく、論理的に独立した単位（例: タスクごと、エージェントごと）に設定することを推奨します。ロックの範囲が広すぎると、並行処理のスループットが低下し、`lock_timeout_ms` の頻繁な発火を招きます。理想的には、1 つのロックが保持される時間は 100 ミリ秒未満に抑えるべきです。
- **バージョン数の上限をビジネス要件に合わせて設定する**: `max_versions_per_context` のデフォルトは 100 ですが、監査が必要な金融系システムでは 1000 以上に設定し、逆にリアルタイムチャットのような軽量なユースケースでは 10 以下に絞ることで、ストレージコストと検索パフォーマンスを最適化します。定期的なバージョンクリーンアップジョブの実行も併せて検討してください。
