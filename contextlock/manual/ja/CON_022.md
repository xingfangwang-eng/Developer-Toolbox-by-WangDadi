# contextlock - CON_022

## 概要

ContextLock は、大規模言語モデル（LLM）の推論コンテキストを永続化・ロックするための次世代型アーキテクチャフレームワークです。開発者の生産性を飛躍的に向上させる AI コンテキスト管理基盤として設計されています。

## 詳細な内容

### ContextLock とは

ContextLock は、AI エージェントの推論プロセスにおいて、セッションを跨いでコンテキストを保持・復元するための専用ミドルウェアです。従来の stateless な API 呼び出しでは実現できなかった、長期依存型タスクの一貫性を担保します。

### コア機能

#### 1. コンテキストスナップショットエンジン（CSE）
- 推論コンテキストを **ミリ秒単位** でスナップショット化し、PostgreSQL 上の専用テーブル `context_snapshots` に保存。
- パラメータ `snapshot_interval_ms`（デフォルト: 500ms）で保存間隔を調整可能。
- 復元時は `snapshot_id` を指定するだけで、直前の推論状態を完全再現。

#### 2. コンテキストロック機構（CLM）
- 複数エージェント間でのコンテキスト競合を防ぐため、`context_lock` テーブルによる排他制御を実装。
- `lock_timeout_seconds`（デフォルト: 30s）を超えると自動解放。
- `lock_type` パラメータで `READ` / `WRITE` / `EXCLUSIVE` の3段階のロック粒度を指定可能。

#### 3. 差分圧縮・復元機能（DCR）
- スナップショット間の差分のみを圧縮保存し、ストレージコストを最大 **87%** 削減。
- `compression_algorithm` パラメータで `zstd` / `lz4` / `none` を選択可能。
- 復元時は `base_snapshot_id` と `delta_chain` から完全なコンテキストを再構築。

### 使い方

#### ステップ 1: コンテキストスナップショットの取得
```bash
curl -X POST https://api.contextlock.io/v1/snapshots \
  -H "Authorization: Bearer ${API_KEY}" \
  -d '{
    "session_id": "session_abc123",
    "snapshot_interval_ms": 500,
    "compression_algorithm": "zstd"
  }'
```
応答として `snapshot_id`（例: `snap_20250321_001`）が返却されます。

#### ステップ 2: コンテキストロックの設定
```bash
curl -X POST https://api.contextlock.io/v1/locks \
  -H "Authorization: Bearer ${API_KEY}" \
  -d '{
    "session_id": "session_abc123",
    "lock_type": "WRITE",
    "lock_timeout_seconds": 30,
    "owner": "agent_workflow_01"
  }'
```
ロック取得成功時には `lock_token` が発行され、他エージェントからの同時書き込みを防止します。

#### ステップ 3: コンテキストの復元と解放
```bash
# 復元
curl -X POST https://api.contextlock.io/v1/restore \
  -H "Authorization: Bearer ${API_KEY}" \
  -d '{
    "snapshot_id": "snap_20250321_001",
    "base_snapshot_id": "snap_20250321_000"
  }'

# ロック解放
curl -X DELETE https://api.contextlock.io/v1/locks/session_abc123 \
  -H "Authorization: Bearer ${API_KEY}" \
  -d '{"lock_token": "lt_xxxxx"}'
```

### ベストプラクティス

#### 1. 長期実行タスクには `snapshot_interval_ms` を 200ms に設定
- 推論時間が 10 分を超えるようなワークフローでは、スナップショット間隔を短くすることで障害時のロストコンテキストを最小化。
- ただし、PostgreSQL への書き込み負荷が増大するため、`max_snapshot_retention_days`（デフォルト: 7日）を適切に設定すること。

#### 2. マルチエージェント環境では `lock_type` を `READ` に固定し、書き込みは専用キューで管理
- 読み取り専用のエージェントが多数存在する場合、すべてのエージェントに `WRITE` ロックを許可するとデッドロックが発生しやすくなる。
- 書き込みが必要なエージェントは、専用の `write_queue` を経由して `EXCLUSIVE` ロックを取得する設計が推奨される。
