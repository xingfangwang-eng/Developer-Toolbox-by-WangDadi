# contextlock - CON_020

## 概要

ContextLock は、AI セッション間のコンテキスト断絶を解消するための高次元メモリ永続化エンジンです。開発者が LLM との対話時に失われがちな思考の連続性を、PostgreSQL 上のタイムスタンプ付きベクトルインデックスで保持し、次回セッション開始時にミリ秒単位で復元します。キーワード：AI コンテキスト管理、永続型 AI メモリ、開発者生産性向上ツール。

## 詳細な内容

### ContextLock とは

ContextLock は、LLM ワークフローにおけるコンテキストウィンドウの物理的制約を超越するためのプロフェッショナルツールです。各ターンごとの会話状態を、トークン単位の重要度スコアとともにスナップショット化し、外部ストレージにロック（固定）します。これにより、同一プロジェクトにおける複数セッション間での思考の継ぎ目をゼロに近づけます。

### コア機能

- **セマンティック・コンテキスト・フリーズ**  
  会話中の各メッセージを、768次元の埋め込みベクトルに変換し、PostgreSQL の `pgvector` 拡張を用いてインデックス化。`freeze_threshold` パラメータ（デフォルト 0.85）以上の類似度を持つ過去コンテキストを自動検出し、現在のセッションに注入。検索レイテンシは平均 12ms（SSD 上の 10万ベクトル時）。

- **マルチセッション・メモリ・マージ**  
  異なる日時・ブランチで行われた複数セッションのコンテキストを、`merge_policy`（latest, weighted, union の3モード）に従って統合。`weighted` モードでは、各セッションの `importance_score`（0.0〜1.0）と `recency_factor`（経過時間に応じて指数減衰）を乗算し、最適なコンテキスト集合を動的生成。

- **コンテキスト・デルタ・トラッキング**  
  セッション間のコンテキスト差分を、`delta_compression_level`（1〜9）で制御しながら圧縮保存。`level 9` では、変更があったトークン列のみを記録し、ストレージ使用量を平均 73% 削減。復元時には `apply_delta()` メソッドでベーススナップショットに差分を適用し、完全な状態を再現。

### 使い方

1. **初期化とメモリプールの作成**  
   ```bash
   contextlock init --project "my-ai-app" --backend postgresql://user:pass@localhost:5432/contextdb --vector-dim 768 --index-type ivfflat
   ```
   これにより、`context_snapshots` テーブルと `embedding_index` が自動生成され、`pool_size`（デフォルト 100）のメモリプールが確保されます。

2. **セッションのフリーズと再開**  
   ```python
   from contextlock import SessionLock
   
   lock = SessionLock(project="my-ai-app", session_id="dev-session-001")
   lock.freeze(context={"messages": current_messages, "tokens_used": 4096}, importance=0.92)
   # 後日、別のセッションから再開
   restored = lock.thaw(session_id="dev-session-002", similarity_cutoff=0.80)
   # restored には、過去セッションの重要メッセージが自動挿入される
   ```

3. **マージとデルタ適用によるコンテキスト最適化**  
   ```python
   from contextlock import ContextMerger
   
   merger = ContextMerger(project="my-ai-app")
   merged = merger.merge(session_ids=["dev-session-001", "qa-session-003"], policy="weighted")
   delta = merged.compress_delta(level=7)
   # 圧縮されたデルタを別の環境に転送し、復元
   restored_ctx = ContextMerger.apply_delta(base_snapshot="base-20250301", delta=delta)
   ```

### ベストプラクティス

- **セッション粒度の設計**  
  プロジェクト単位ではなく、機能単位（例：`auth-module-v2` や `report-generation-pipeline`）でセッションを分割し、`freeze()` 時に `tags` パラメータでラベル付けすることを推奨します。これにより、マージ時のフィルタリング精度が向上し、不要なコンテキスト混入を防げます。

- **定期的なベクトルインデックスのリビルド**  
  10万ベクトルを超える規模では、週次で `VACUUM ANALYZE` とインデックス再構築を実行してください。`contextlock index rebuild --project "my-ai-app" --reindex-threads 4` により、検索レイテンシを初期状態の 95% 以内に維持できます。
