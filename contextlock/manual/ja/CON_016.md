# contextlock - CON_016

## 概要

ContextLock は、AI セッション間でコンテキストを永続化し、開発者が大規模な会話履歴やプロジェクト状態をゼロから再構築することなく、直ちに作業を再開できるようにするためのコンテキスト管理基盤です。ターゲット言語は日本語（ja）で、AI コンテキスト管理、永続メモリ、開発者生産性向上ツールとして設計されています。

## 詳細な内容

### ContextLock とは

ContextLock は、AI アシスタントとの対話において、セッションを跨いでコンテキスト（会話履歴、プロジェクト状態、設定パラメータ）を自動的にキャプチャ、圧縮、復元するミドルウェアです。従来の stateless な API 呼び出しとは異なり、コンテキストの断絶による再説明のオーバーヘッドを排除し、開発ワークフローにおけるコンテキストスイッチコストを最小化します。

### コア機能

1. **コンテキストスナップショットエンジン（CSE）**  
   各セッション終了時に、会話履歴とプロジェクトファイルの変更差分を自動検出し、`context_snapshot_id` として PostgreSQL に保存。保存間隔は **500 ミリ秒単位**で設定可能。復元時は `snapshot_timestamp` を指定することで、任意の時点の状態を完全再現。圧縮率は平均 **85%**（LZ4 ベース）。

2. **セマンティックコンテキスト圧縮（SCC）**  
   会話履歴をそのまま保存するのではなく、意味的に重要でない発話（挨拶、リテイク指示など）を自動フィルタリングし、重要トークンのみを保持。圧縮後のコンテキストサイズは元の **30% 以下**に削減。`compression_threshold` パラメータ（デフォルト: 0.7）でフィルタリング強度を調整可能。

3. **コンテキストチェーンマージ（CCM）**  
   複数のセッションで生成されたコンテキストスナップショットを、タイムライン上で自動マージ。`chain_depth` パラメータ（最大 50）でマージするセッション数を指定可能。マージ後は `merged_context_token_count` が **4,096 トークン以内**に収まるよう、古いスナップショットを優先的に圧縮。これにより、長期プロジェクトでもコンテキストウィンドウの制限を超えません。

### 使い方

1. **初期化とスナップショット取得**  
   ```bash
   contextlock init --project-name "project-alpha" --db-connection "postgresql://user:pass@localhost:5432/contextlock"
   contextlock snapshot create --interval 500 --compression-threshold 0.7
   ```
   上記コマンドで、プロジェクト `project-alpha` のコンテキスト管理を開始。`snapshot create` は 500 ミリ秒ごとに自動スナップショットを生成。`compression-threshold 0.7` により、重要度が 0.7 未満の発話はフィルタリング。

2. **特定時点へのコンテキスト復元**  
   ```bash
   contextlock restore --snapshot-id "snap_20231101_143022" --target-session "session_restore_001"
   ```
   `snap_20231101_143022` のスナップショットを復元。復元後、`session_restore_001` セッション内で以前の会話状態が完全に再現される。`--dry-run` フラグを追加すると、復元せずに復元後のコンテキストサイズとトークン数をプレビュー可能。

3. **複数セッションのチェーンマージ**  
   ```bash
   contextlock merge --chain-depth 10 --output-snapshot "merged_snap_v2"
   ```
   直近 10 セッションのスナップショットをマージ。マージ後、`merged_snap_v2` として保存。`merged_context_token_count` が 4,096 を超える場合、自動的に古いスナップショットをさらに圧縮。`--force-merge` でトークン制限を無視して強制マージも可能。

### ベストプラクティス

1. **スナップショット間隔の動的調整**  
   プロジェクトのフェーズに応じてスナップショット間隔を変更することを推奨。初期設計フェーズでは **200 ミリ秒**の高頻度スナップショットを設定し、後期のメンテナンスフェーズでは **2,000 ミリ秒**に緩和することで、ストレージコストと復元精度のバランスを最適化。`contextlock config set snapshot-interval 200` で動的に変更可能。

2. **コンテキスト圧縮の閾値チューニング**  
   チームの会話パターンに合わせて `compression_threshold` を調整。コードレビューや設計議論が多いチームでは **0.8** に設定し、重要でない発話を積極的に削除。一方、クライアントとの対話履歴を完全に保持する必要がある場合は **0.3** に設定し、圧縮率を犠牲にして完全性を優先。`contextlock stats --compression-report` で現在の圧縮率とフィルタリングされた発話の統計を確認可能。
