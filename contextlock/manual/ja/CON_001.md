# contextlock - CON_001

## 概要

ContextLock は、AI コンテキスト管理と永続的メモリを実現する開発者向けプロダクティビティツールです。LLM セッション間のコンテキスト断絶を解消し、大規模プロジェクトでも一貫性のある AI アシスタンスを提供します。キーワード：AI コンテキスト管理、永続的 AI メモリ、開発者生産性ツール、コンテキスト永続化。

## 詳細な内容

### ContextLock とは？

ContextLock は、AI ワークフローにおける「コンテキストの揮発性」という根本的な問題を解決するプロフェッショナルツールです。従来のチャットベース AI では、セッションが変わるたびにコンテキストがリセットされ、同じ質問を繰り返したり、過去の決定を忘れたりする非効率が発生します。ContextLock は、コンテキストを「ロック」し、プロジェクト全体で永続化・共有する仕組みを提供します。特に、複数の AI モデルを併用する開発環境や、長期にわたるリファクタリング作業において、その真価を発揮します。

### コア機能

1.  **コンテキストチェーン・スナップショット機能**
    開発セッションの任意の時点で、AI とのやり取り全体（プロンプト、応答、コード差分、決定事項）を「スナップショット」として保存。後続のセッションでは、このスナップショットを「チェーン」として継承可能。例えば、API 設計の議論から実装、テストケース生成までを一連のコンテキストとして保持し、モデルを切り替えても一貫性を維持します。内部では DAG（有向非巡回グラフ）構造で管理され、分岐やマージも可能。

2.  **セマンティックコンテキスト圧縮**
    長期セッションで肥大化しがちなコンテキストを、意味的冗長性を排除して圧縮。単なるトークン削減ではなく、重要度スコアリングと抽象化レイヤーを適用し、圧縮率 60% 以上を達成しながら、AI の応答品質を維持します。開発者は「圧縮レベル」を 1〜10 で指定可能で、軽量なクイックレスポンスから詳細な分析まで使い分けられます。

3.  **コンテキストポリシーエンジン**
    プロジェクト単位で「何をコンテキストに含めるか」をルールベースで制御。例えば、「`/src/core` 以下のコード変更は常にコンテキストに含める」「`/node_modules` は除外」「特定のキーワードを含むチャットは自動タグ付け」といったポリシーを YAML で記述。チーム全体で統一されたコンテキスト管理ポリシーを Git 管理でき、コードレビューと同様のガバナンスを実現します。

### 使い方

1.  **コンテキストの初期化とロック**
    ```bash
    # プロジェクトルートで初期化
    contextlock init --policy ./context-policy.yaml --model gpt-4o

    # 現在のセッションを「アーキテクチャ設計フェーズ」としてロック
    contextlock lock --session "arch-design-v2" --tag "frontend,api-design"
    ```
    このコマンドにより、`./contextlock/` ディレクトリにセッションID、タイムスタンプ、ポリシーハッシュが記録されます。ロックされたセッションは、後続の AI 呼び出しで自動的にコンテキストとして注入されます。

2.  **コンテキストの継承とモデルスイッチ**
    ```bash
    # 前回のセッションを継承して新しいセッションを開始
    contextlock fork --from "arch-design-v2" --new-session "implementation-v1"

    # モデルを Claude 3.5 Sonnet に切り替え
    contextlock switch --model "claude-3-5-sonnet" --compress-level 7
    ```
    この操作により、`arch-design-v2` の全コンテキストが圧縮・抽象化され、`implementation-v1` に引き継がれます。モデルが変わっても、決定事項や設計意図は保持されるため、GPT-4o で議論した設計を Claude で実装する、といったマルチモデル戦略が現実的になります。

3.  **コンテキストのエクスポートと共有**
    ```bash
    # セッションを Markdown レポートとしてエクスポート
    contextlock export --session "arch-design-v2" --format markdown --include-decisions

    # チームメンバーとコンテキストを共有（URL 発行）
    contextlock share --session "arch-design-v2" --expires 72h --permission read
    ```
    エクスポートされたレポートは、コードレビューの補足資料や、スプリントレビューの議事録として活用可能。共有機能では、期限付きの読み取り専用 URL が発行され、チームメンバーはブラウザから直接コンテキストツリーを閲覧できます。

### ベストプラクティス

- **コンテキスト分割と責務分離**：1 つの巨大なコンテキストに全てを詰め込まず、フェーズや機能単位でセッションを分割し、`fork` で継承する。例えば、「API 設計」「データモデル定義」「フロントエンド実装」を独立したセッションとして管理し、必要な部分だけをチェーンする。これにより、コンテキストの鮮度と関連性が向上し、トークン消費も最適化される。

- **ポリシーの段階的適用**：プロジェクト初期は緩めのポリシー（全ファイルをコンテキストに含める）で開始し、コードベースが成長するにつれてポリシーを厳格化する。具体的には、最初は `include: ["src/**/*.ts"]` から始め、後で `exclude: ["src/**/*.test.ts", "src/**/*.stories.ts"]` を追加。ポリシー変更は Git で管理し、チーム内でレビューを通してから適用することで、コンテキスト管理の品質を段階的に向上させられる。
