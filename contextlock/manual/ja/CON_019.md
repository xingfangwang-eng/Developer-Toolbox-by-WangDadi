# contextlock - CON_019

## 概要

ContextLock は、大規模言語モデル（LLM）のコンテキストウィンドウを動的にロック・管理し、セッションを跨いで永続的な AI メモリを実現するプロフェッショナルツールです。本プロジェクトは、AI コンテキスト管理、永続化メモリ、開発者生産性向上をキーワードに設計されています。

## 詳細な内容

### とは ContextLock?

ContextLock は、LLM との対話における「コンテキスト消失問題」を解決するために開発された、エンタープライズグレードのコンテキスト管理基盤です。セッション間で重要なコンテキスト情報を自動的に抽出・保存・復元し、開発者が手動でプロンプトを再構築する手間を完全に排除します。PostgreSQL による永続化ストレージと、ミリ秒単位のコンテキスト再構築レイテンシを実現しています。

### コア機能

1. **コンテキストスナップショットエンジン（CSE）**  
   - セッション内の全てのメッセージを解析し、`context_weight`（0.0〜1.0）パラメータに基づいて重要度を自動スコアリング。  
   - スコアが閾値（デフォルト: 0.75）を超えたコンテキストブロックのみを PostgreSQL の `context_snapshots` テーブルに保存。  
   - 復元時は、保存されたスナップショットを元に `max_tokens` を動的に計算し、LLM に最適なプロンプトを再構築。復元レイテンシは平均 45ms（P99: 120ms）。

2. **メモリチェーンリンカー（MCL）**  
   - 複数セッションに跨るコンテキストを `session_chain_id` で紐付け、ツリー構造として管理。  
   - 各チェーンノードには `timestamp`（ISO 8601）と `context_hash`（SHA-256）が付与され、変更検知とバージョン管理が可能。  
   - `chain_depth` パラメータ（デフォルト: 5）を超えた古いチェーンは自動アーカイブし、クエリパフォーマンスを維持。

3. **コンテキストフラグメントガベージコレクター（CFGC）**  
   - 未使用または重複したコンテキストフラグメントを定期的にスキャンし、`retention_days`（デフォルト: 30）を超えたデータを自動削除。  
   - 削除前に `dry_run` モードで影響範囲をレポート出力可能。  
   - PostgreSQL の `VACUUM` と連携し、ストレージ使用量を最小化。

### 使い方

1. **初期セットアップとスナップショット取得**  
   ```bash
   # ContextLock をインストール後、PostgreSQL 接続情報を設定
   contextlock init --db-host localhost --db-port 5432 --db-name contextlock_db

   # 現在のセッションのコンテキストスナップショットを強制取得
   contextlock snapshot --session-id "session_abc123" --force --context-weight 0.8
   ```
   - `--force` フラグを指定すると、自動スコアリングをスキップし、全コンテキストを保存。

2. **チェーンリンクによるマルチセッション復元**  
   ```bash
   # 特定のチェーン ID からコンテキストを復元
   contextlock restore --chain-id "chain_xyz789" --max-tokens 4096 --session-id "session_def456"

   # 復元時に特定のタイムスタンプ範囲を指定
   contextlock restore --chain-id "chain_xyz789" --from "2025-01-01T00:00:00Z" --to "2025-01-15T23:59:59Z"
   ```
   - 復元されたコンテキストは、LLM へのプロンプトとして自動挿入。`--dry-run` でプレビュー可能。

3. **ガベージコレクションの定期実行**  
   ```bash
   # ドライランモードで削除対象を確認
   contextlock gc --dry-run --retention-days 30

   # 実際に削除を実行
   contextlock gc --retention-days 30 --vacuum-full
   ```
   - `--vacuum-full` を指定すると、PostgreSQL の `VACUUM FULL` を実行し、ストレージを完全に最適化。

### ベストプラクティス

1. **コンテキスト重みの動的調整**  
   - プロジェクトの初期フェーズでは `context_weight` を 0.5 に設定し、自動スコアリングの精度を確認。  
   - 重要な会話が頻繁に発生するフェーズでは 0.8 以上に引き上げ、スナップショットの網羅性を向上。  
   - 定期的に `contextlock stats --session-id` でスコアリング分布を確認し、閾値をチューニング。

2. **チェーン深さの適切な設計**  
   - `chain_depth` はセッションの複雑さに応じて設定。単一トピックの長期対話では 10 以上、複数トピックが混在する場合は 3〜5 を推奨。  
   - アーカイブされたチェーンは `contextlock chain-archive --restore` で後から復元可能。必要に応じて `retention_days` を 90 日に延長。
