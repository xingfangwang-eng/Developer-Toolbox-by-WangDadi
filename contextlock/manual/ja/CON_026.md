# contextlock - CON_026

## 概要

ContextLock は、AI セッションにおけるコンテキスト消失問題を根本的に解決する、次世代の永続型コンテキスト管理ツールです。開発者が LLM（大規模言語モデル）と対話する際、過去の会話や設定がリセットされる「コンテキスト切れ」を防ぎ、プロジェクト全体で一貫した AI メモリを維持します。特に、複数セッションにまたがる長期的な開発タスクや、チームでのナレッジ共有において真価を発揮します。

## コア機能

### 1. セッション横断型コンテキストレイヤー
ContextLock は、単一の会話に閉じず、プロジェクト全体で共有可能な「永続コンテキストレイヤー」を提供します。これにより、異なるブランチやタスク間で AI の記憶をシームレスに継承。例えば、API 設計の議論を終えた後、実装フェーズに移行しても、前回の決定事項や制約条件を自動的に参照します。

### 2. 動的コンテキストプライオリティエンジン
すべてのコンテキスト情報に対して、自動的に優先順位（重要度スコア）を付与。頻繁に参照されるルールやプロジェクトのコア設定は常に最上位に保持され、古いログや一時的な指示は自動的にアーカイブされます。これにより、トークン制限に引っかかることなく、最も重要な情報だけが AI に渡されます。

### 3. バージョン管理付きコンテキストスナップショット
コンテキストの状態を任意のタイミングでスナップショットとして保存可能。Git ライクな履歴管理により、過去のコンテキスト状態にワンクリックでロールバックできます。デバッグや実験的なプロンプト変更の影響を検証する際に、不可欠な機能です。

## 使い方

### ステップ 1: プロジェクトルートにコンテキストファイルを初期化
まず、ターミナルでプロジェクトディレクトリに移動し、以下のコマンドを実行します。
```bash
contextlock init --project my-ai-project --lang ja
```
これにより、`.contextlock/` ディレクトリが生成され、デフォルトのコンテキストルールが設定されます。`--lang ja` フラグにより、日本語での応答を優先する設定が自動注入されます。

### ステップ 2: セッション間でコンテキストを固定（ロック）する
AI との対話中に、特定の会話内容を永続化したい場合、以下のコマンドでコンテキストをロックします。
```bash
contextlock lock --scope "architecture:microservices" --priority 9
```
`--scope` でタグを指定し、`--priority` で優先度を設定（1-10）。これにより、そのセッションの議論内容が「アーキテクチャ:マイクロサービス」として固定され、以降の全セッションで自動的に参照されます。

### ステップ 3: コンテキストの競合を解決する
複数の開発者が異なるコンテキストをロックした場合、自動的に競合検出が行われます。競合が発生したら、以下のコマンドでマージします。
```bash
contextlock merge --branch feature/context-a --branch hotfix/context-b
```
コンフリクトが生じた箇所は、インタラクティブに解決可能。解決後、`contextlock push` でリモートリポジトリに同期します。

## ベストプラクティス

### 1. コンテキストの粒度は「モジュール単位」で設計せよ
プロジェクト全体に単一のコンテキストを適用するのではなく、機能モジュールやマイクロサービスごとに独立したコンテキストレイヤーを定義することを推奨します。例えば、`contextlock init --scope auth` で認証モジュール専用のコンテキストを作成し、認証ロジックの変更が他のモジュールに影響を与えないようにします。これにより、コンテキストの肥大化を防ぎ、AI の応答精度が向上します。

### 2. 定期的なコンテキストクリーンアップを CI/CD に組み込め
コンテキストスナップショットが増えすぎると、パフォーマンスに影響が出る可能性があります。週次またはリリース前に、以下のコマンドを CI パイプラインに追加し、不要なスナップショットを自動削除しましょう。
```bash
contextlock prune --older-than 30d --keep-latest 5
```
これにより、直近 5 件のスナップショットを保持しつつ、30 日以上前の古いコンテキストをクリーンアップ。常に最新の状態を維持できます。
