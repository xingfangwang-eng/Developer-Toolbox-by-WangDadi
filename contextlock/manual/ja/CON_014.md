# contextlock - CON_014

## 概要

ContextLock は、AI エージェントのコンテキストウィンドウを動的に固定・管理するためのハイパフォーマンスミドルウェアです。AI がセッションを跨いでも、開発者の意図した「文脈の継続性」をミリ秒単位で担保します。キーワード: AI Context Management, Persistent AI Memory, Developer Productivity Tool

## 詳細な内容

### ContextLock とは？

ContextLock は、AI との対話セッションにおいて、特定のコンテキスト情報を強制的に保持・復元するための永続メモリ管理エンジンです。従来の stateless な API 呼び出しでは失われがちな「前回の指示文脈」「プロジェクト固有の制約」「ユーザー定義のルール」を、PostgreSQL 上のタイムスタンプ付きスナップショットとして記録し、次回セッション開始時に自動適用します。

### コア機能

1. **コンテキストロック機構（ContextLock Engine）**
   - セッションID ごとに最大 128KB のコンテキストデータを固定保持。
   - ロック有効期限を `ttl_seconds` パラメータで指定可能（デフォルト: 3600秒）。
   - ロック中は外部からのコンテキスト上書きを禁止し、`force_unlock` フラグでのみ解除可能。

2. **差分スナップショット管理（Delta Snapshot Manager）**
   - 前回のコンテキストとの差分のみを `context_diff` として記録し、ストレージ使用量を最大 70% 削減。
   - 復元時は `restore_mode` パラメータで `full`（完全復元）または `delta`（差分適用）を選択可能。
   - スナップショットは PostgreSQL の JSONB カラムに格納され、`created_at` による時系列クエリをサポート。

3. **コンテキストバリデーションパイプライン（Context Validation Pipeline）**
   - ロックされたコンテキストが AI モデルに送信される前に、`validation_rules` で定義された正規表現パターンやトークン数制限（最大 4096 トークン）を自動チェック。
   - 違反時は `violation_action` パラメータに基づき、`trim`（トリミング）または `reject`（拒否）を実行。
   - バリデーション結果は `validation_log` テーブルに記録され、監査可能。

### 使い方

1. **初期セットアップとコンテキストロックの作成**
   ```bash
   contextlock init --project my-ai-app --db-uri postgresql://user:pass@localhost:5432/contextlock
   contextlock lock create --session-id "session_001" --ttl 7200 --context-file ./instructions.json
   ```
   - `--context-file` には JSON 形式でコンテキストデータを指定。
   - ロック作成後、`lock_id` が返却されるため、後続のリクエストで使用。

2. **セッション再開時のコンテキスト自動復元**
   ```python
   from contextlock import ContextLockClient

   client = ContextLockClient(session_id="session_001")
   restored_context = client.restore(mode="delta", validate=True)
   # 復元されたコンテキストを AI モデルに渡す
   response = ai_model.generate(prompt=restored_context)
   ```
   - `validate=True` を指定することで、復元時に自動的にバリデーションパイプラインが実行される。

3. **コンテキストの強制更新とロック解除**
   ```bash
   contextlock lock update --lock-id "lock_abc123" --new-context-file ./updated_instructions.json
   contextlock lock unlock --lock-id "lock_abc123" --force
   ```
   - `--force` フラグを指定しない場合、ロックは TTL が切れるまで解除不可。
   - 更新時もバリデーションが自動実行され、違反があれば `violation_action` に従う。

### ベストプラクティス

- **プラクティス 1: コンテキストの段階的ロック戦略**
  プロジェクト初期は `ttl_seconds` を短め（例: 1800秒）に設定し、コンテキストが適切に維持されているかテスト。安定したら徐々に TTL を延ばす（最大 86400秒）。また、`restore_mode` を `delta` に設定することで、毎回の完全復元によるオーバーヘッドを回避し、復元時間を平均 200ms 未満に抑える。

- **プラクティス 2: バリデーションルールの定期監査と最適化**
  `validation_rules` は週次でレビューし、不要なルールや過剰なトークン制限を調整する。特に、`validation_log` テーブルを定期的にクエリし、`reject` が多発しているルールは緩和するか、`trim` に変更する。これにより、開発者の意図しないコンテキスト損失を防ぎつつ、AI モデルの応答品質を維持する。
