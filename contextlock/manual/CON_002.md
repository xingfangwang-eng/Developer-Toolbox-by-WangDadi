### What is ContextLock?

ContextLock is a **production-grade context persistence engine** that solves the fragmentation problem in AI-assisted development workflows. It eliminates the "forgetfulness" of LLMs by providing a structured, version-controlled memory layer that survives across sessions, agents, and toolchains. Unlike naive chat history dumps, ContextLock uses semantic indexing and priority decay to ensure the most relevant context is always injected, reducing token costs by up to 40%.

### Core Features

- **Deterministic Context Snapshots with Semantic Diffing**  
  Instead of storing raw conversation logs, ContextLock creates immutable snapshots of your AI workspace—including code snippets, agent instructions, and environmental variables. It then uses semantic diffing to detect what actually changed between sessions, allowing you to revert, fork, or merge context states like a Git branch. This is critical for debugging multi-turn agent workflows.

- **Token-Aware Priority Injection Engine**  
  The built-in priority router dynamically ranks context fragments based on recency, task relevance, and user-defined weights. It automatically trims low-priority tokens from the context window before API calls, ensuring that your LLM always sees the most actionable information without exceeding token limits. This is especially useful in RAG pipelines where irrelevant documents bloat the prompt.

- **Cross-Platform Context Sync via Lockfile Protocol**  
  ContextLock introduces a `.contextlock` lockfile that can be shared across IDEs, CI/CD pipelines, and cloud agents. This file acts as a signed, tamper-evident manifest of your AI session state. It enables teams to collaborate on the same AI context—for example, a developer in VS Code can hand off a debugging session to a GitHub Actions runner with full context integrity.

### How to Use

1. **Initialize a ContextLock Workspace**  
   Run `contextlock init --project my-ai-app` in your project root. This generates a `.contextlock/` directory with a `config.yaml` where you define your context sources (e.g., `./src/`, `./docs/`, `./prompts/`). Use the `--priority` flag to set default weights for each source. Example:  
   ```bash
   contextlock init --project my-ai-app --priority src=0.8,docs=0.5,prompts=1.0
   ```

2. **Capture a Context Snapshot with Semantic Tags**  
   After a productive AI session, run `contextlock snapshot --tag "bugfix-issue-42" --message "Fixed race condition in auth module"`. This locks the current context into an immutable snapshot. Use `contextlock diff --from bugfix-issue-41 --to bugfix-issue-42` to see semantic changes (not line-by-line diffs) in the context structure.

3. **Inject Context into an LLM Call**  
   In your Python script, import the ContextLock client:  
   ```python
   from contextlock import ContextLock
   cl = ContextLock(project="my-ai-app", snapshot="bugfix-issue-42")
   context_prompt = cl.build_prompt(max_tokens=4096, strategy="priority")  # Auto-trims low-priority tokens
   response = openai.ChatCompletion.create(model="gpt-4", messages=[{"role": "user", "content": context_prompt}])
   ```  
   The `build_prompt()` method returns a string that is guaranteed to fit within your token budget, with the most important context at the top.

### Best Practices

- **Use Context Locking as a Pre-Commit Hook**  
  Integrate `contextlock snapshot` into your pre-commit or pre-push hooks. This ensures that every code change is accompanied by a corresponding context snapshot. When debugging a regression weeks later, you can replay the exact AI context that generated the code, avoiding "I don't remember why I did that" moments.

- **Implement Context Rotation for Long-Running Agents**  
  For agents that run for hours or days (e.g., automated PR reviewers), set up a cron job that rotates context snapshots every 1000 tokens of new input. Use `contextlock merge` to combine old and new snapshots while discarding duplicates. This prevents context bloat and keeps inference latency low.
