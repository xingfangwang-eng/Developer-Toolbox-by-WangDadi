# contextlock - CON_024

## Overview

ContextLock is a developer-first persistent context management engine designed to eliminate AI "amnesia" in multi-session workflows. Unlike ephemeral chat windows or brittle prompt hacks, ContextLock provides deterministic context retention across IDE, CLI, and API surfaces — purpose-built for teams shipping complex software with LLM-assisted pipelines.

## Why ContextLock Matters

Every AI-assisted developer knows the pain: you explain your architecture in session A, then session B forgets your stack, your constraints, and your naming conventions. ContextLock solves this by enforcing a **versioned, lockable context graph** that persists across tools, sessions, and team members. It’s not a memory wrapper — it’s a structured context layer that your entire toolchain respects.

## Core Features

### 1. Deterministic Context Locking with Dependency Pinning
ContextLock introduces **`lockfiles`** — think `package-lock.json` but for your AI context. Each lockfile captures the exact state of your project’s context graph: imported modules, architectural decisions, API contracts, and even custom heuristics. Once locked, the AI cannot deviate from the pinned context unless explicitly unlocked. This guarantees reproducible AI behavior across builds, deploys, and team members.

### 2. Multi-Session Context Reconciliation Engine
When working across IDE, terminal, and CI/CD pipelines, context drift is inevitable. ContextLock’s **reconciliation engine** automatically detects conflicts between sessions (e.g., a refactor in your editor vs. a commit message in your terminal) and merges them using a three-way merge strategy. You get a unified, conflict-free context tree without manual copy-pasting or “remember this for later” notes.

### 3. Context-Aware Hook System for CI/CD Integration
ContextLock exposes a **pre-commit and post-merge hook API** that injects context directly into your pipeline. For example, when a PR is merged, ContextLock can automatically update the context graph to reflect the new codebase state — no more stale context causing hallucinated deprecation warnings. Hooks are written in TypeScript or JSON and integrate with GitHub Actions, GitLab CI, and Jenkins.

## How to Use

### Step 1: Initialize a Context Lockfile in Your Project Root
Run `contextlock init` in your project directory. This scans your `package.json`, `tsconfig.json`, and any `.contextlockignore` files to generate an initial `context.lock.json`. The lockfile includes your dependency tree, runtime environment variables, and a user-defined “context schema” (e.g., “We use React 18 with Next.js 14, no class components, all components are functional.”).  
```bash
contextlock init --schema ./context-schema.yaml
```

### Step 2: Pin a Session’s Context to the Lockfile
When you start a new AI-assisted session (e.g., in VS Code via the ContextLock extension), run `contextlock pin --session "refactor-auth-module"`. This binds the session to the current lockfile version. The AI will only see the pinned context — any attempt to reference out-of-scope APIs or deprecated patterns triggers a warning. You can verify the pin status with `contextlock status`.  
```bash
contextlock pin --session "refactor-auth-module" --lockfile context.lock.json
```

### Step 3: Reconcile Context Across Sessions After a Merge
After merging a feature branch, run `contextlock reconcile --from main --into feature-x`. The engine compares the lockfiles from both branches, detects divergent context entries (e.g., “auth module uses JWT” vs. “auth module uses OAuth2”), and presents a three-way merge interface. Accept the merged context with `contextlock commit`. The updated lockfile is then ready for the next AI session.  
```bash
contextlock reconcile --from main --into feature-x --output merged.lock.json
contextlock commit merged.lock.json
```

## Best Practices

### 1. Lock Context Before Heavy Refactoring Sessions
Before you ask an AI to rewrite a module or migrate a dependency, run `contextlock pin` with a descriptive session name. This ensures the AI operates within the exact boundaries of your current architecture — no accidental suggestions to replace your custom hook with a library you’ve already deprecated. After the refactor, `contextlock unlock` and reconcile to capture the new state.

### 2. Use Context Lockfiles as Onboarding Artifacts
When a new developer joins your team, share the latest `context.lock.json` as part of their onboarding. They can run `contextlock init --from-lockfile context.lock.json` to bootstrap their local context. This eliminates the “what’s our API contract again?” loop and ensures every AI session they start is aligned with the team’s current decisions.
