# contextlock - CON_014

## Overview

ContextLock is a deterministic AI context persistence engine designed for developers who demand zero-loss memory across multi-turn LLM interactions. It eliminates context drift, token waste, and hallucination cascades by locking critical state variables at the nanosecond level.

## Core Features

### 1. Nanosecond-Precision Context Snapshot Engine
Captures the full LLM conversation state (system prompt, user messages, tool call stack, and embedding vectors) into a compressed binary snapshot every 50ms. Supports rollback to any previous snapshot ID within a session. Snapshot size averages 2.3KB per 10k tokens, with PostgreSQL-backed storage for historical query replay.

**Parameters:** `--snapshot-interval=50` (ms), `--max-snapshots=5000`, `--storage-backend=postgres://user:pass@host:5432/contextlock`

### 2. Deterministic Context Lock via Hash-Chain Validation
Each context snapshot is cryptographically signed using SHA-3-256 and chained to the previous snapshot via a Merkle tree. The tool verifies integrity on every LLM response, rejecting any context that fails hash validation. This guarantees that no external process or middleware can silently mutate the conversation state.

**Parameters:** `--hash-algorithm=sha3-256`, `--chain-depth=128`, `--validation-mode=strict`

### 3. Multi-Session Context Merging with Conflict Resolution
Allows merging two or more independent context sessions into a single coherent state. Uses a three-way merge algorithm with token-level diffing and automatic conflict resolution based on timestamp priority. Supports merging up to 16 sessions simultaneously, with a merge latency under 200ms for sessions up to 50k tokens each.

**Parameters:** `--merge-strategy=timestamp-priority`, `--max-merge-sessions=16`, `--merge-timeout=200` (ms)

## How to Use

### Step 1: Initialize a ContextLock Session with Custom Snapshot Interval
```bash
contextlock init --session-id=dev-session-01 \
                 --snapshot-interval=50 \
                 --storage-backend=postgres://localhost:5432/contextlock \
                 --hash-algorithm=sha3-256
```
This creates a new session with nanosecond-level snapshotting and cryptographic integrity validation. The PostgreSQL backend stores all snapshots with a TTL of 7 days by default.

### Step 2: Inject a System Prompt and Lock the Context Boundary
```bash
contextlock inject --session-id=dev-session-01 \
                   --role=system \
                   --content="You are a code reviewer. Only respond with JSON." \
                   --lock=true
```
The `--lock=true` flag freezes the system prompt and all subsequent context modifications must pass through the hash-chain validator. Any attempt to modify the context externally will be detected and rejected.

### Step 3: Merge Two Production Sessions and Replay the Result
```bash
contextlock merge --session-ids=prod-session-01,prod-session-02 \
                  --merge-strategy=timestamp-priority \
                  --output-session=merged-session-final \
                  --replay=true
```
This merges two sessions with automatic conflict resolution, then replays the entire merged context to the LLM to verify coherence. The replay output is written to `merged-session-final.json` with full snapshot history.

## Best Practices

### 1. Use ContextLock as a Preprocessing Layer for All LLM Calls in CI/CD Pipelines
Integrate ContextLock as a middleware between your CI/CD runner and the LLM API. For example, in a GitHub Actions workflow, initialize a session before each code review step. This ensures that every review request has a deterministic, replayable context that can be audited later. Set `--validation-mode=strict` to automatically fail the pipeline if any context tampering is detected.

### 2. Implement a Context Retention Policy with Tiered Storage
For production deployments, configure ContextLock to use a dual-storage strategy: in-memory for active sessions (TTL: 1 hour) and PostgreSQL for archived sessions (TTL: 30 days). Use the `--snapshot-interval` parameter to reduce storage overhead for long-running sessions. For example, set `--snapshot-interval=500` for sessions longer than 10 minutes to balance precision with storage cost.
