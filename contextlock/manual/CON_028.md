## What is ContextLock?

**ContextLock** is a developer-first, persistent AI context management tool designed to eliminate session drift and memory loss across LLM interactions. It enables engineering teams to inject structured, version-controlled context into any AI workflow—whether you're building agents, fine-tuning prompts, or debugging complex pipelines. Think of it as `git` for your AI session state.

---

## Core Features

### 1. **Immutable Context Snapshots with Diff Tracking**
Every AI interaction context is captured as an immutable snapshot, indexed by timestamp and session hash. You can roll back to any prior state, compare diffs between context versions, and replay exact conditions that produced a specific output. This eliminates "black box" debugging and ensures full reproducibility.

### 2. **Context Injection via Lockfiles (`.ctxlock`)**
Define your AI context declaratively using `.ctxlock` files—YAML-based manifests that bundle system prompts, conversation history, tool schemas, and environment variables. Lockfiles can be checked into version control, shared across teams, and automatically merged with runtime variables. This turns context from a hidden state into a first-class artifact.

### 3. **Zero-Latency Context Rehydration**
ContextLock uses a local-first, embedded database (powered by SQLite + memory-mapped indexes) to rehydrate context in under 5ms, even for sessions with 100k+ tokens. It supports lazy loading of large context blocks and automatic compression of redundant data, making it suitable for real-time agent loops and CI/CD pipelines.

## How to Use

### 1. Initialize a ContextLock Repository
```bash
contextlock init --project my-ai-agent --backend local
```
This creates a `.contextlock/` directory with an embedded database, a default `.ctxlock` template, and a `contextlock.json` config file. The `--backend` flag supports `local`, `redis`, or `s3`—choose `local` for single-machine development, `redis` for distributed teams.

### 2. Capture and Lock a Context Snapshot
```python
from contextlock import ContextLock

cl = ContextLock(project="my-ai-agent")
snapshot = cl.capture(
    session_id="sess_abc123",
    system_prompt="You are a senior Rust engineer...",
    history=[{"role": "user", "content": "Explain ownership."}],
    tools=["read_file", "execute_command"],
    metadata={"branch": "feature/borrow-checker"}
)
snapshot.lock()  # Locks the snapshot: immutable, versioned, and hash-signed
```
The `lock()` call generates a SHA-256 hash of the entire context bundle, signs it with your local key, and writes it to the immutable store. Any subsequent modification will produce a new snapshot—no silent overwrites.

### 3. Rehydrate and Replay a Previous Context
```bash
contextlock rehydrate --snapshot-id 0x7a9f3c --output ./replay_context.json
```
This extracts the exact context from snapshot `0x7a9f3c` and writes it to a JSON file. You can then pipe it directly into your LLM API call:
```bash
cat replay_context.json | openai api chat_completions -m gpt-4o
```
Alternatively, use the `--replay` flag to automatically execute the same tool calls and log the output for regression testing.

## Best Practices

### 1. **Adopt Lockfiles as Part of Your CI/CD Pipeline**
Treat `.ctxlock` files like `package-lock.json` or `Cargo.lock`. Commit them to your repository and run `contextlock verify` in CI to ensure that every AI interaction is reproducible. This is critical for auditing LLM behavior in production—especially when you need to prove that a specific output was generated under a known set of conditions.

### 2. **Use Context Branches for A/B Testing Prompts**
Create separate `.ctxlock` branches (e.g., `main`, `experiment/shorter-prompt`, `experiment/rag-enabled`) and run the same input through each branch. Use ContextLock's diff viewer to compare outputs side-by-side. This gives you a rigorous, version-controlled way to optimize prompts without losing track of what changed—and why.
