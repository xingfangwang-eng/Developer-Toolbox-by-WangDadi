# contextlock - CON_021

## Overview

**ContextLock** is a developer-first persistent memory layer for AI-assisted workflows. It eliminates context drift across sessions, IDE instances, and terminal repls by locking semantic state into a lightweight, portable index. Designed for engineers who live in the terminal, ContextLock hooks into your shell, editor, and AI agent to preserve intent across reboots, branch switches, and model swaps.

Keywords: AI Context Management, Persistent AI Memory, Developer Productivity Tool

---

## Core Features

### 1. **Session-Pinned Semantic Snapshots**
ContextLock captures the full semantic payload of your current development session — open files, recent git diffs, terminal scrollback, and AI chat history — and materializes it into a deterministic, versioned snapshot. Each snapshot is hashed and indexed locally, enabling instant recall across any tool that supports the ContextLock API. No more “what was I doing?” after a `git stash` or a lunch break.

### 2. **Zero-Latency Context Injection via Unix Domain Sockets**
Instead of polling files or parsing logs, ContextLock exposes a lightweight IPC endpoint over Unix domain sockets. Any process (Neovim, VS Code, tmux, `llm` CLI, custom scripts) can request the current context block in under 1ms. The payload is a compressed, schema-agnostic JSON blob that includes file paths, cursor positions, linting errors, and the last 50 lines of terminal output. Designed for sub-millisecond reads even with 10k+ indexed snapshots.

### 3. **Multi-Model Memory Bridge**
ContextLock acts as a bidirectional memory bridge between local LLMs (Ollama, llama.cpp) and cloud APIs (OpenAI, Anthropic, Gemini). It transparently injects locked context into the system prompt of any model, and records model responses back into the index. This means you can switch from `gpt-4` to `codellama` mid-session without losing the thread — the context is model-agnostic and persistent.

## How to Use

### 1. **Initialize a Locked Context in Your Shell**
```bash
# Install via Homebrew or cargo
brew install contextlock/tap/cl

# Initialize a new context session in the current directory
cl init --project "my-microservice" --scope "refactoring auth module"

# Lock the current state (files, git diff, terminal history)
cl lock --snapshot --auto-inject
```
This creates a `.contextlock/` directory in your project root. The daemon (`cld`) runs in the background, monitoring file changes and terminal output. Every `cl lock` generates a new snapshot ID (e.g., `snap_a1b2c3d4`) that you can reference later.

### 2. **Inject Context into Any AI Tool**
```bash
# Inject context into an LLM CLI (e.g., llm)
cl inject --snap snap_a1b2c3d4 | llm -m gpt-4 "explain the logic in the last 5 diffs"

# Or pipe directly into a Neovim buffer
cl inject --snap latest | nvim - -c "setfiletype markdown"
```
The `inject` command outputs a structured context block. You can filter by scope (`--scope "auth"`), time range (`--since "30m ago"`), or file pattern (`--files "*.rs"`). The output is plaintext by default, but supports JSON and YAML for programmatic consumption.

### 3. **Restore a Previous Context Across Machines**
```bash
# Export a snapshot as a portable archive
cl export --snap snap_a1b2c3d4 --format tar.gz --output context_backup.tar.gz

# On another machine, import and restore
cl import context_backup.tar.gz
cl restore --snap snap_a1b2c3d4 --workspace ~/projects/my-microservice
```
This restores file states, cursor positions, and even open terminal panes (if you use `tmux` or `zellij`). The restore operation is idempotent — it merges with existing files and warns on conflicts. Perfect for pairing sessions, CI debugging, or moving between a desktop and laptop mid-task.

## Best Practices

### 1. **Use Scoped Snapshots for Task Isolation**
Don’t lock the entire project. Instead, scope your snapshots to the specific subsystem you're working on (e.g., `--scope "payment gateway"`). This keeps context injection fast and prevents irrelevant file changes from polluting your AI agent’s attention. For monorepos, always pass `--scope` or `--file-pattern` to avoid blowing up the snapshot size.

### 2. **Automate Snapshots on Git Hooks**
Set up a `post-checkout` or `pre-commit` hook that runs `cl lock --auto`. This ensures every branch switch or commit creates a recoverable context checkpoint. Combine with `cl prune --keep-last 50` to manage storage (snapshots are tiny, but can accumulate). For CI pipelines, use `cl export` before `git stash` and `cl import` after `git stash pop` to preserve debugging context across rebases.
