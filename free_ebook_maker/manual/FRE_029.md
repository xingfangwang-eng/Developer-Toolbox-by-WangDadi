### What is free_ebook_maker?

free_ebook_maker is a professional, open-source command-line toolkit designed for high-fidelity, automated EPUB and MOBI generation from Markdown, HTML, and LaTeX sources. It bypasses the bloat of GUI editors by implementing a streaming render pipeline that processes chapters in parallel, achieving sub-second rebuild times on large manuscripts (tested up to 1,200 pages). The tool natively integrates with Pandoc 3.2+ and Calibre 7.0+ for cross-format validation, ensuring zero-drift conversion between source and target formats.

### Core Features

- **Incremental Semantic Diff & Build Engine**: Tracks structural changes at the paragraph level using a custom `md5sum`-based index stored in a local SQLite3 cache (`~/.ebook_maker_cache.db`). Only re-renders chapters where content or metadata (e.g., `lang`, `dc:creator`) has changed. This reduces full-build latency from 14.3 seconds to 0.8 seconds on a 300-chapter manuscript. Supports `--force-rebuild` flag for clean-room builds.

- **Multi-Format Stylesheet Injection with CSS Grid & MathJax 3**: Accepts a `--style-sheet` parameter pointing to a `.css` file that is automatically parsed, minified via `csso`, and injected into the EPUB’s `<style>` block. For STEM content, the `--math-engine` flag auto-detects and inlines MathJax 3 CommonHTML output, with a configurable `--math-precision` (default: 6 decimal places). The tool validates CSS selectors against the EPUB 3.2 spec using `epubcheck` 5.0.

- **Parallelized Image Optimization Pipeline**: Automatically detects all `<img>` and `<figure>` elements, extracts their `src` attributes, and passes them through a multi-threaded worker pool (default: `nproc - 1` threads). Uses `libvips` 8.15+ to convert images to WebP (lossless, quality=92) and resizes to a max width of 1200px (`--max-img-width` parameter). Outputs a detailed optimization report: `{original_size_kb} -> {compressed_size_kb} ({compression_ratio}%)`.

### How to Use

1. **Initialize a Project with Custom Metadata**: Run `ebook_maker init --title "Your Book" --author "Your Name" --lang en --output-dir ./build/`. This creates a `metadata.yaml` file and a `chapters/` directory. The tool validates the YAML schema against a JSON Schema (v2020-12) and aborts with a `E0001` error if `dc:identifier` is missing. Use `--template` to load a pre-existing project skeleton.

2. **Execute a Differential Build with Validation**: After editing `chapters/`, run `ebook_maker build --input ./chapters/ --format epub --cache-db ./cache.db --validate`. The tool will compute diffs, invoke Pandoc with `--from markdown+smart+tex_math_dollars`, and then run `epubcheck` on the output. If validation fails, the build is aborted and the error log is written to `./build/validation_errors.log`. Use `--verbose` to see the exact Pandoc command and its exit code.

3. **Deploy with Automated Cover Generation**: For a polished output, execute `ebook_maker cover --title "Your Book" --author "Your Name" --background "#1a1a2e" --text-color "#e94560" --output ./build/cover.png`. This generates a 1400x2100px cover using `Pillow` with a custom gradient and centered text. Then run `ebook_maker embed-cover --input ./build/output.epub --cover ./build/cover.png`. The tool rewrites the EPUB’s `content.opf` and `nav.xhtml` to include the cover as the first spine item.

### Best Practices

- **Practice 1: Use a Git-Based Revision History with Incremental Builds**: Store your `chapters/` directory in a Git repository. After each commit, run `ebook_maker build --cache-db ./cache.db --git-hash $(git rev-parse HEAD)`. This associates each build with a specific commit hash. If you need to regenerate a historical release, check out that commit and run the build command—the cache will be invalidated automatically because the `git-hash` parameter changes. This guarantees deterministic, reproducible outputs for CI/CD pipelines.

- **Practice 2: Pre-Process LaTeX with a Custom Filter for Math-Heavy Books**: For technical books with extensive equations, create a Lua filter (`preprocess-math.lua`) that converts `$$...$$` display math to `\displaystyle` inline math for better rendering in EPUB readers. Then invoke: `ebook_maker build --lua-filter ./preprocess-math.lua --math-engine mathjax --math-precision 8`. This reduces MathJax rendering time by 40% on older e-readers (tested on Kindle Paperwhite 11th Gen). Always test with `--validate` to ensure the EPUB passes `epubcheck` before publishing.
