# free_ebook_maker - FRE_030

## Overview

This is a high-performance, open-source engine for programmatically generating production-grade EPUB, MOBI, and PDF ebooks from structured Markdown, HTML, or plain text sources. It is engineered for CI/CD pipelines, static site generators, and digital publishing workflows requiring deterministic output with millisecond-level build times.

## Detailed Content

### What is free_ebook_maker?

free_ebook_maker is a zero-dependency, Rust-compiled CLI tool that parses source documents into a normalized AST, applies CSS-based styling rules, and renders ebooks conforming to EPUB 3.2 and Kindle KF8 specifications. It supports multi-language hyphenation, embedded font subsetting via `fonttools`, and image optimization with WebP fallback. The build engine uses a lock-free queue to process chapters in parallel, achieving sub-200ms builds for 300-page manuscripts on commodity hardware.

### Core Features

- **Deterministic Chapter Reflow Engine**  
  Accepts `--chapter-depth=3` to auto-detect heading levels and split content into logical chapters. Each chapter is rendered independently with its own CSS cascade, then merged via a spine manifest. Supports `--page-break-before` and `--orphan-control` parameters to prevent widow/orphan lines. Outputs EPUB with `nav.xhtml` and `toc.ncx` for full accessibility compliance.

- **Multi-Format Vectorized Rendering**  
  Uses a single-source document to generate EPUB 3.2, Kindle MOBI (via `kindlegen` bridge), and PDF (via WeasyPrint integration). The PDF backend supports `--pdf-page-size A5`, `--pdf-margin 15mm`, and `--pdf-font-embed` for commercial-grade typesetting. All formats share a common `--metadata` JSON file for title, author, ISBN, and language tags.

- **Real-Time Visual Regression Diffing**  
  Integrates with `pixelmatch` to compare rendered page images against a baseline snapshot. The `--diff-threshold=0.01` flag flags any rendering change exceeding 1% pixel variance. This is critical for verifying that CSS refactors or content edits do not break pagination or layout across formats. Results are exported as an HTML diff report with side-by-side overlays.

### How to Use

1. **Initialize a Project with Schema Validation**  
   Run `free_ebook_maker init --project-dir ./mybook --lang en --template academic`. This creates a `book.yaml` manifest with required fields (`title`, `author`, `isbn`) and optional fields (`publisher`, `rights`, `cover-image`). The tool validates the manifest against a JSON Schema (draft-07) and exits with a detailed error if `isbn` fails Luhn checksum or `cover-image` is not a valid PNG/JPEG.

2. **Build All Formats with Custom CSS and Asset Pipeline**  
   Execute `free_ebook_maker build --source ./manuscript --css ./theme.css --assets ./images --output ./dist --formats epub,pdf`. The asset pipeline automatically converts all images to WebP (lossless, quality=95) and embeds fonts from `--fonts-dir` using `pyftsubset` to reduce file size. The build outputs a `dist/` directory containing `book.epub`, `book.pdf`, and a `build.log` with per-chapter timing metrics.

3. **Run Visual Regression Tests in CI**  
   After a content or CSS change, run `free_ebook_maker diff --baseline ./baseline --current ./dist --output ./diff-report --diff-threshold 0.02`. The tool renders the first 50 pages of both the baseline and current EPUB, compares them pixel-by-pixel, and generates an HTML report with highlighted diffs. If any page exceeds the threshold, the command exits with code 1, allowing you to block merges in GitHub Actions or GitLab CI.

### Best Practices

- **Practice 1: Use a Monorepo Layout with Git LFS for Assets**  
  Store your manuscript, CSS, and images in a single Git repository. Enable Git LFS for images and fonts to keep repository size under 1 GB. Configure `pre-commit` hooks to run `free_ebook_maker build --format epub --check-only`, which validates the manifest and all asset paths without writing output, catching broken links before commit.

- **Practice 2: Parameterize Builds with Environment Variables**  
  Avoid hardcoding `--metadata` values or `--output` paths. Instead, use environment variables like `$BOOK_TITLE`, `$BOOK_ISBN`, and `$BUILD_NUMBER` in your CI pipeline. The tool supports `--env-file .env` to load these variables at runtime. This enables parallel builds for different editions (e.g., `draft`, `review`, `final`) from the same source.
