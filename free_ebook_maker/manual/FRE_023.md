# free_ebook_maker - FRE_023

## Overview

`free_ebook_maker` is a zero-dependency, CLI-first toolkit designed for automated EPUB 3.2 / PDF generation from structured Markdown and YAML front-matter. It targets static-site authors, technical writers, and CI/CD pipelines that require reproducible, DRM-free eBook artifacts. The engine parses raw content into a validated book spine, injects semantic metadata, and exports paginated files with precise typographic control—all without a GUI.

## Detailed Content

### What is free_ebook_maker?

`free_ebook_maker` is a deterministic build system for eBooks. It ingests a source directory containing `toc.yml`, `metadata.yml`, and a series of `.md` chapters, then outputs a compliant EPUB or print-ready PDF. Unlike generic converters, it enforces a strict chapter hierarchy, supports custom CSS injection per section, and performs real-time validation against the EPUB 3.2 schema (OPF, NCX, and XHTML conformance). It is designed for headless environments—no browser, no Electron, no network calls during build.

### Core Features

1. **Multi-Format Spine Compilation with Granular Pagination Control**  
   The engine reads a `toc.yml` file that defines exact chapter order, sub-chapter breaks, and page-break-before rules. It compiles the spine into a single EPUB container or split PDF chapters. The `--page-margin` parameter accepts values in `mm` (e.g., `--page-margin 12.7mm`) and `--font-base` sets the root em size (default: `14pt`). The PDF renderer uses a custom Harfbuzz-based layout engine that handles CJK, Latin, and mixed-script kerning with sub-millimeter precision.

2. **Semantic Metadata Injection with ISBN & DOI Validation**  
   The `metadata.yml` parser extracts fields like `isbn`, `doi`, `language`, `rights`, and `subject`. It validates ISBN-13 checksums (mod 10) and DOI prefixes (e.g., `10.xxxx`). The `--strict-metadata` flag enforces that all required fields (`title`, `creator`, `identifier`) are non-empty. The resulting EPUB’s `content.opf` includes a `<dc:identifier>` with `opf:scheme="ISBN"` and a `<meta property="dcterms:modified">` timestamp accurate to milliseconds.

3. **PostScript-Level Hyphenation & Ligature Engine**  
   A built-in hyphenation dictionary (based on Liang’s algorithm) supports 12 languages (en, de, fr, es, it, pt, ru, ja, zh-CN, ko, ar, hi). The `--ligatures` flag (default: `true`) enables discretionary ligatures (fi, fl, ffi, ffl) in the PDF output. The hyphenation threshold is configurable via `--hyphen-min-chars 3` (minimum characters before a break). This is not a regex hack—it operates at the glyph level via a custom OpenType shaper.

### How to Use

1. **Initialize a Book Project with Validation Scaffold**  
   Run `free_ebook_maker init mybook --author "Jane Doe" --lang en --template academic`. This generates a skeleton directory with `toc.yml`, `metadata.yml`, and a `chapters/` folder. The `--template` flag selects a pre-configured CSS profile (e.g., `academic` uses 11pt Garamond, 1.5 line-height, justified alignment). The command also creates a `.ebookmakerrc` JSON config file with default values for `page-margin`, `font-base`, and `hyphen-min-chars`. If `--strict-metadata` is set, the tool immediately validates `metadata.yml` against the EPUB 3.2 schema and exits with error code 1 on failure.

2. **Compile the eBook with Parallel Page Rendering**  
   Execute `free_ebook_maker build --input ./mybook --output ./dist --format epub --threads 4`. The engine parses `toc.yml` to build a directed acyclic graph (DAG) of chapters, then renders each chapter’s Markdown to XHTML in parallel using a thread pool (controlled by `--threads`, max 16). The PDF path uses `--format pdf --dpi 300 --pdf-engine harfbuzz`. The build process logs per-chapter timing (in milliseconds) and total memory usage (peak RSS). The output file includes a SHA-256 checksum appended to the filename (e.g., `mybook_epub_a1b2c3d4.epub`).

3. **Inject Custom CSS and Override Page Breaks**  
   Place a `styles/custom.css` file in the project root. Use the `--css-override` flag: `free_ebook_maker build --css-override ./styles/custom.css --page-break-before each-chapter`. The engine merges the default CSS with the override, then applies it to each XHTML page. For fine-grained control, add `<!-- pagebreak -->` comments in Markdown to force a break at a specific paragraph. The tool also supports `--first-page-number 3` to start page numbering from an arbitrary integer (useful for prefaces).

### Best Practices

- **Use a Monorepo with Git-LFS for Image Assets**  
  Store all cover images, diagrams, and embedded fonts in a `assets/` directory tracked by Git-LFS. In `metadata.yml`, reference images via relative paths (e.g., `cover: assets/cover.png`). This keeps the repository lean and ensures reproducible builds across CI runners. Always set `--strict-metadata` in CI to catch missing cover art or broken links before deployment.

- **Profile Hyphenation for Multi-Language Documents**  
  For books containing mixed scripts (e.g., English body with Japanese blockquotes), run `free_ebook_maker profile --lang ja --sample ./chapters/ --output hyphenation_stats.json` before the full build. This generates a report of hyphenation misses and ligature conflicts per language. Adjust `--hyphen-min-chars` or disable ligatures for specific sections via `<!-- noligatures -->` comments. This prevents over-hyphenation in narrow columns and ensures CJK characters are never broken.
