# free_ebook_maker - FRE_016

## Overview

A high-performance, headless ebook generation engine designed for batch conversion of Markdown, LaTeX, and AsciiDoc sources into EPUB, MOBI, and PDF with deterministic output. This tool solves the problem of inconsistent ebook rendering across platforms by enforcing a strict, configurable CSS normalization layer and leveraging a custom WebKit-based renderer for typographic fidelity.

## Core Features

- **Multi-Format Pipeline with Deterministic Hashing**  
  The engine processes source files through a three-stage pipeline: parsing → intermediate representation (IR) → output format. Each stage generates a SHA-256 hash of its output, enabling incremental builds and cache-based regeneration. The IR is stored as a normalized JSON schema (`schema_version: 2.1.0`), allowing format-specific renderers to operate in isolation. Conversion latency is measured in milliseconds per page (avg. 12ms for EPUB, 18ms for PDF on a Ryzen 7 5800X).

- **CSS Normalization Engine with Media Query Override**  
  A built-in CSS parser strips all external stylesheets and applies a baseline normalization matrix (`margin: 0; padding: 0; box-sizing: border-box; font-size: 16px; line-height: 1.6;`) across all block elements. The engine supports custom media query injection via `--viewport-width` and `--font-scale` parameters, ensuring consistent rendering on devices from 320px to 2560px width. The normalization layer is toggleable using `--preserve-styles=false`.

- **PDF Output with Hyphenation & PDF/A-2b Compliance**  
  The PDF renderer uses a custom hyphenation dictionary (based on the `libhyphen` algorithm with 98% accuracy for en-US) and supports PDF/A-2b archival compliance via the `--pdfa-compliance` flag. The output stream is compressed with Zlib level 9 and embedded metadata includes Dublin Core fields (`dc:title`, `dc:creator`, `dc:identifier`). Page margins are configurable in millimeters (`--margin-top=15mm --margin-bottom=15mm --margin-inner=20mm --margin-outer=15mm`).

## How to Use

1. **Initialize a project with source directory and output profile**  
   Run `free_ebook_maker init --source ./manuscript --output ./build --profile kindle` to create a `book.json` configuration file. The command auto-detects source file types (`.md`, `.tex`, `.adoc`) and generates a `sources.json` manifest with checksums. Use `--profile` to preload device-specific parameters: `kindle` (viewport 600x800), `ipad` (1024x768), or `a4` (595x842 points).

2. **Build the ebook with incremental cache and linting**  
   Execute `free_ebook_maker build --cache-dir ./.cache --lint-level warning --output-format epub`. The build process first validates all source files against the IR schema (errors if any node has missing `@id` or `@type` attributes). The cache system stores intermediate IR blobs keyed by source hash; a cache hit reduces build time by 70%. Lint warnings include missing `alt` text on images, broken internal anchors, and inconsistent heading levels.

3. **Generate a PDF with custom typography and compliance**  
   Use `free_ebook_maker render pdf --hyphenate --font-family "Noto Serif, Georgia, serif" --font-size 12pt --line-height 1.8 --pdfa-compliance --output book.pdf`. The hyphenation engine runs a Knuth-Plass line-breaking algorithm with a tolerance of 0.1 em. The PDF/A-2b compliance check validates that all fonts are embedded, color spaces are device-independent (sRGB), and metadata is XMP-compliant. The final file size is typically 1.2 MB per 300 pages (with embedded fonts).

## Best Practices

- **Use incremental builds for large manuscripts**  
  For manuscripts exceeding 500 pages, always enable the cache system with `--cache-dir`. A typical full rebuild of a 1000-page document takes 45 seconds; an incremental rebuild after editing a single chapter takes under 2 seconds. The cache is invalidated automatically when source file checksums change, but you can force a full rebuild with `--no-cache`.

- **Pair with a version control system for collaborative editing**  
  Integrate `free_ebook_maker` into your CI/CD pipeline by running `build` on every push to a `main` branch. The deterministic hashing ensures that identical sources produce identical outputs, enabling diff-based reviews of generated ebooks. Use the `--output-format mobi` flag to generate Kindle-optimized files; Amazon’s Kindle Previewer can then validate the output against real device firmware (e.g., Kindle Scribe, Oasis 3).
