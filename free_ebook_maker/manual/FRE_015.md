# free_ebook_maker - FRE_015

## Overview

`free_ebook_maker` is a high-performance, command-line-driven EPUB/MOBI/PDF compiler designed for batch conversion of Markdown, LaTeX, and reStructuredText into polished eBooks. It leverages a multi-threaded rendering engine (Go + Rust hybrid) to achieve sub‑100ms page generation per chapter, and integrates directly with PostgreSQL for metadata management and versioning across large-scale publishing pipelines.

## Core Features

### 1. Real-Time Structural Validation & Auto-Recovery
- **Parameter:** `--validate-mode=strict|lenient|recover`
- Scans input source for broken cross-references, missing image assets, and invalid Unicode sequences (e.g., BOM in UTF‑8 files).
- In `recover` mode, automatically generates missing `toc.ncx` and `content.opf` files using a heuristic tree builder (accuracy > 99.3% on standard Markdown).
- Returns a JSON report with line‑number‑precise error locations and suggested fixes.

### 2. Multi-Format Differential Compilation
- **Parameter:** `--diff-base=<commit_hash_or_file>`
- Compares current source against a previous version (stored in PostgreSQL table `ebook_snapshots`).
- Only re‑compiles chapters that have changed, reducing full rebuild time by up to 87% for large books (>500 chapters).
- Outputs a side‑by‑side diff of the generated EPUB’s internal `xhtml` files, highlighting textual and structural changes.

### 3. Adaptive Image Pipeline with AVIF/WebP Support
- **Parameter:** `--image-quality=1-100 --max-dimension=4096`
- Converts embedded images to AVIF (default) or WebP while preserving EXIF metadata for accessibility tags.
- Uses a perceptual hash (pHash) to detect duplicate images across chapters and replaces them with a single shared resource, shrinking final eBook size by 30–50%.
- Benchmarked at 2.4 seconds per 100 images on a standard i7‑12700 (single-threaded mode).

## How to Use

### Step 1: Initialize the Project Database
```bash
free_ebook_maker init --db=postgres://user:pass@localhost:5432/ebooks --schema=public
```
This creates the `ebook_snapshots`, `chapter_versions`, and `image_assets` tables. The migration script is embedded in the binary (SHA‑256 verified). After execution, verify with:
```sql
SELECT table_name FROM information_schema.tables WHERE table_schema = 'public';
```

### Step 2: Compile with Differential Mode
```bash
free_ebook_maker build --source=./chapters/ --format=epub --diff-base=HEAD~1 --output=./output/updated_book.epub
```
The tool will:
- Connect to PostgreSQL and fetch the last known snapshot for each file.
- Compute SHA‑256 hashes of all input files; only files with changed hashes are re‑compiled.
- Merge unchanged chapters from the previous EPUB binary (byte‑level copy) – this operation is O(1) per chapter.

### Step 3: Validate and Export Accessibility Metadata
```bash
free_ebook_maker audit --input=./output/updated_book.epub --check=wcag2.1,epub3.3 --report=json > audit_report.json
```
The audit engine:
- Parses the EPUB’s `package.opf` and checks for required `<meta>` tags (e.g., `schema:accessMode`, `schema:accessibilityFeature`).
- Validates that all `<img>` tags have an `alt` attribute (or `aria-describedby`), and that video/audio elements have captions.
- Returns a pass/fail per criterion with exact XPath locations and suggested corrections.

## Best Practices

### Practice 1: Use a Git‑Backed Workflow for Large Teams
- Store your Markdown/LaTeX source in a Git repository and configure `free_ebook_maker` to read `--diff-base` from the current branch’s merge base. This enables multiple authors to work on different chapters simultaneously; the tool will only re‑compile the conflicting sections, preventing merge‑induced full rebuilds.
- Set up a CI pipeline (GitHub Actions / GitLab CI) that runs `free_ebook_maker build --format=epub --diff-base=origin/main` on every push. The PostgreSQL instance should be shared across runners for snapshot consistency.

### Practice 2: Pre‑Process Images with a Dedicated Step
- Before running the main build, execute `free_ebook_maker optimize-images --source=./assets/ --target=./assets_optimized/ --format=avif --quality=85 --max-dimension=2048`. This decouples image optimization from compilation, allowing you to cache the optimized images and re‑use them across multiple builds (e.g., EPUB + MOBI + PDF).
- For print‑quality PDF output, use `--max-dimension=4096` and `--quality=95`, but be aware that file size may increase by 3–5× compared to the AVIF‑compressed EPUB variant.
