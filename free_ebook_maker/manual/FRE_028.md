# free_ebook_maker - FRE_028

## Overview

`free_ebook_maker` is a high‑performance, CLI‑first ebook generation engine for developers and publishers who need to batch‑produce EPUB3, PDF, and MOBI files from Markdown source trees. It uses a **two‑stage pipeline** (compilation → packaging) with **sub‑millisecond incremental rebuilds** and native `PostgreSQL` metadata caching. Designed for CI/CD workflows, it supports **parallel rendering** across 16+ cores and outputs **validated, accessibility‑compliant** ebooks without relying on heavyweight GUI tools.

## Core Features

### 1. Incremental AST Compilation with PostgreSQL Backed Cache
- **Parameter**: `--cache-dsn postgresql://user:pass@host/ebook_cache`  
- **Metric**: Subsequent builds of a 500‑chapter book complete in **≤ 2.3 ms** (cold build: 1.2 s).  
- **Mechanism**: Each Markdown file is parsed into an Abstract Syntax Tree (AST). File hashes and AST diffs are stored in a `PostgreSQL` table (`ebook_ast_cache`). On rebuild, only changed nodes are recompiled; unchanged nodes are fetched from the cache, reducing disk I/O by **94%** in large repositories.

### 2. Multi‑Format Output with Structural Fidelity Validation
- **Parameter**: `--formats epub3,pdf,mobi --validation-level strict`  
- **Metric**: Outputs pass **W3C EPUB3 validator** (no errors) and **PDF/A‑2b** conformance.  
- **Mechanism**: The tool uses a **unified intermediate representation** (IR) that preserves semantic structure (headings, code blocks, math, footnotes). Each format backend maps the IR to its native spec. A post‑generation step runs `epubcheck` and `verapdf`; if any error exceeds a configurable threshold (`--max-errors 0`), the build fails with a detailed report.

### 3. Parallel Asset Pipeline with Sub‑second Image Optimization
- **Parameter**: `--assets-dir ./images --max-concurrent 16 --optimize jpeg:85,webp:lossless`  
- **Metric**: 1,200 high‑resolution images processed in **3.4 s** (vs. 47 s sequential).  
- **Mechanism**: Each image is hashed, resized to target dimensions (configurable via `--max-width 1600`), and re‑encoded in parallel using a **worker pool** (one per CPU core). Metadata (EXIF, color profile) is stripped for privacy. The pipeline auto‑generates `srcset` attributes for HTML‑based formats.

## How to Use

### Step 1: Initialize the Project and Seed the Cache
```bash
free_ebook_maker init --project-dir ./my_book \
  --cache-dsn postgresql://admin:secret@localhost:5432/ebook_cache \
  --title "Advanced Rust Patterns" \
  --author "Jane Doe"
```
- Creates the directory structure (`./my_book/src/`, `./my_book/assets/`).  
- Connects to PostgreSQL and creates the `ebook_ast_cache` table if missing.  
- Scans `src/` for `.md` files and builds the initial AST cache (cold build).

### Step 2: Compile with Incremental Mode and Validate
```bash
free_ebook_maker build --project-dir ./my_book \
  --formats epub3,pdf \
  --validation-level strict \
  --incremental \
  --verbose
```
- Re‑parses only files whose modification timestamps or content hashes differ from the cache.  
- Runs `epubcheck` and `verapdf` on the generated files.  
- Outputs a JSON summary to stdout: `{"duration_ms": 2.3, "errors": 0, "warnings": 2}`.

### Step 3: Distribute the Artifacts with Metadata Injection
```bash
free_ebook_maker package --project-dir ./my_book \
  --output-dir ./dist \
  --metadata "isbn=978-0-123-45678-9" \
  --metadata "rights=CC-BY-NC-4.0" \
  --sign-key /path/to/private.pem
```
- Injects ISBN, rights, and a digital signature into the EPUB3 OPF and PDF metadata streams.  
- Generates a `checksums.txt` file (SHA‑256) for each artifact.  
- Creates a `release.json` manifest for downstream CI/CD consumption.

## Best Practices

### Practice 1: Use PostgreSQL Cache in CI/CD for Sub‑second Builds
In a multi‑branch pipeline, share a single `PostgreSQL` instance (e.g., via Docker) to cache ASTs across runs. Configure `--cache-ttl 86400` to purge stale entries. This reduces average build time from **12 s to < 100 ms** for books with 95% unchanged content—critical for PR‑based ebook previews.

### Practice 2: Enforce Structural Fidelity with Pre‑commit Hooks
Integrate `free_ebook_maker validate` as a Git pre‑commit hook. Use `--validation-level strict` to reject commits that introduce broken cross‑references, missing alt texts, or malformed math blocks (`$...$`). Pair this with a `--max-errors 0` policy to ensure every commit produces a validatable ebook.
