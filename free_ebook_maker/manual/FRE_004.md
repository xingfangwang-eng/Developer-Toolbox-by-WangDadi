# free_ebook_maker - FRE_004

## Overview

This is the core architecture documentation for the **free_ebook_maker** project, an open-source, high-performance engine designed to convert raw Markdown, LaTeX, and HTML content into professionally formatted EPUB, MOBI, and PDF ebooks. The system is built for batch processing, with sub-100ms per-page rendering and native support for multilingual Unicode typesetting (including CJK, Arabic, and RTL scripts).

## Core Features

### 1. Zero-Copy Markdown-to-EPUB Pipeline with Incremental Compilation  
- **Parameter:** `--incremental-hash` (SHA-256 of source directory)  
- **Spec:** Detects only changed `.md` files since last build; reuses pre-compiled CSS and image assets from a local SQLite cache. Reduces full rebuild time from 12s to under 0.8s for a 300-page document.  
- **Technical detail:** Uses a custom `TreeSitter` grammar to parse Markdown AST and applies a `lazy-evaluation` model for image embedding (base64 inlined only when output format supports it).

### 2. Multi-Format Typesetting Engine with Font Subsetting  
- **Parameter:** `--font-strategy=subset|embed|link`  
- **Spec:** Automatically extracts only the glyphs used in the document (via Unicode range detection) and generates a subsetted WOFF2 font (average size 4KB per language). Supports `harfbuzz` shaping for complex scripts like Devanagari and Thai.  
- **Technical detail:** Integrates with `fonttools` to programmatically modify `cmap` tables; the subsetter runs in a separate `asyncio` worker to avoid blocking the main pipeline.

### 3. Real-Time Cross-Platform Validation via EPUB 3.3 Conformance Checker  
- **Parameter:** `--validate-level=strict|standard|lax`  
- **Spec:** Runs `epubcheck` in a sandboxed Docker container (image: `ghcr.io/w3c/epubcheck:latest`) and parses the output into a structured JSON report. Validates against 47 mandatory checks from the EPUB 3.3 spec, including `nav` order, `dc:identifier` uniqueness, and XHTML well-formedness.  
- **Technical detail:** The validator is triggered as a `pre-commit` hook in the CI pipeline; any violation with severity `ERROR` blocks the build with exit code 1 and a human-readable diff of the offending lines.

## How to Use

### Step 1: Initialize a Project with Git-Based Versioning  
```bash
free_ebook_maker init --project-name "my_ebook" --source-dir ./chapters --output-dir ./build
```
This creates a `free_ebook_maker.json` config file with `"incremental": true` and `"version": "0.1.0"`. It also generates a `.free_ebook_maker_cache/` directory (ignored by `.gitignore`) that stores the last-known SHA-256 hashes for incremental builds.

### Step 2: Compile with Font Subsetting and Validation  
```bash
free_ebook_maker build --font-strategy subset --validate-level strict --output-format epub
```
The engine first scans `./chapters/` for all `.md` files, orders them by a numeric prefix (e.g., `01_intro.md`, `02_chapter1.md`), and runs the incremental hash check. Then it invokes the typesetter: for each paragraph, `harfbuzz` shapes the glyphs, the subsetter extracts only those glyphs from the source font (e.g., `NotoSansSC-Regular.ttf`), and the result is written into a temporary `OEBPS/` directory. Finally, the EPUB 3.3 validator runs inside Docker—if any `ERROR` is found, the build halts and prints the exact XPath to the invalid node.

### Step 3: Deploy the Generated Ebook to Multiple Distribution Channels  
```bash
free_ebook_maker deploy --target amazon-kdp --meta-file ./metadata.yaml --sign-key ./gpg-private.asc
```
This step packages the validated EPUB into a `.zip` with a signed manifest (PGP signature embedded in `META-INF/signatures.xml`). For Amazon KDP, it also automatically converts to MOBI via `kindlegen` (if installed) and applies a KDP-specific cover image resize to 2560x1600 pixels. The deployment script outputs a `deployment_report.json` with timestamps, file sizes, and checksums.

## Best Practices

- **Practice 1: Always enable incremental builds for CI/CD pipelines.**  
  When `--incremental-hash` is active, the engine skips 95% of the compilation work for unchanged files. For a team of 5+ editors, this reduces the average CI build time from 45 seconds to 3 seconds. Combine with a `pre-commit` hook that runs `--validate-level strict` on only the modified files to catch errors before they reach the main branch.

- **Practice 2: Use font subsetting with a fallback chain for multilingual projects.**  
  Define a `fonts` block in the config file with a prioritized list: `["NotoSansSC-Regular.ttf", "NotoNaskhArabic-Regular.ttf", "NotoSansMath-Regular.ttf"]`. The engine will automatically detect the Unicode ranges used (e.g., `U+0600–U+06FF` for Arabic) and subset only the relevant fonts. This reduces the final EPUB size by up to 80% while maintaining perfect rendering across all scripts.
