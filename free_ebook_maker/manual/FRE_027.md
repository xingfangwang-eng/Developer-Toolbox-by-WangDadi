# free_ebook_maker - FRE_027

## Overview

**free_ebook_maker** is a production-grade CLI & GUI toolkit designed to automate the entire lifecycle of EPUB 3.2 / MOBI / PDF generation from raw Markdown, LaTeX, or HTML sources. It is optimized for batch processing, multi-language metadata injection, and deterministic output reproducibility — ideal for CI/CD pipelines, self-publishers, and digital archiving teams.

## Core Features

### 1. Multi-Format Rendering Engine with Strict EPUB 3.2 Compliance

The core renderer (`febm-render`) converts source documents into EPUB 3.2, MOBI (KF8), and PDF (via WeasyPrint or Prince XML). It validates against the official EPUB 3.2 schema using the `epubcheck` 5.1.0 library, ensuring zero validation errors before output.  
- **Parameters:** `--output-format epub,mobi,pdf`, `--epub-version 3.2`, `--validate-schema true`  
- **Performance:** Renders a 300‑page document in under 2.8 seconds on a mid‑range CPU (Intel i7‑12700, 16 GB RAM).  
- **Dependency:** Uses `lxml` 4.9+ for XHTML/CSS parsing and `Pillow` 10.0+ for image optimization (JPEG compression @ 85% quality, PNG quantisation to 256 colors).

### 2. Metadata Injection & Multi-Language Localisation Engine

Automatically injects Dublin Core metadata (title, author, identifier, language, rights) and supports up to 12 concurrent language variants per book. The engine reads a `metadata.yaml` file and merges it with per‑chapter `lang` attributes.  
- **Parameters:** `--metadata metadata.yaml`, `--lang en,de,fr,ja`, `--rights "CC BY-NC-SA 4.0"`  
- **Integration:** Reads from PostgreSQL 16 (via `asyncpg`) or a local SQLite 3.40+ database for dynamic metadata lookups — e.g., author pseudonyms, series order, or ISBN‑13 validation.  
- **Latency:** Metadata merging for 50 chapters takes less than 40 milliseconds.

### 3. Deterministic Build System with Caching & Incremental Compilation

Every build is fully deterministic: given the same source tree and `--seed 0`, the tool produces byte‑identical output (SHA‑256 hash). It uses a content‑addressable cache (BLAKE3) to skip unchanged chapters, reducing rebuild time by up to 94% for large projects.  
- **Parameters:** `--cache-dir ~/.febm-cache`, `--seed 0`, `--incremental true`  
- **Metrics:** Cache hit ratio typically > 85% on CI pipelines; cache size capped at 2 GB with LRU eviction.  
- **Backend:** Written in Rust (core engine) with Python bindings for the CLI; the cache uses `mmap`‑based key‑value storage for zero‑copy reads.

## How to Use

### Step 1: Prepare Source Tree & Metadata

Create a directory with your Markdown files (one per chapter) and a `metadata.yaml` at the root:

```bash
mkdir my-book
cd my-book
cat > metadata.yaml <<EOF
title: "Introduction to Distributed Systems"
author: "Jane Doe"
language: en
identifier: "urn:isbn:978-0-123-45678-9"
rights: "CC BY 4.0"
EOF
```

Place chapter files as `01-intro.md`, `02-architecture.md`, etc. Each file must start with a `#` heading (H1). The engine will auto‑generate the spine order based on numeric prefixes.

### Step 2: Run the Full Build Pipeline

Execute the build command with explicit format and validation flags:

```bash
febm-render . \
  --output-format epub,mobi,pdf \
  --epub-version 3.2 \
  --validate-schema true \
  --metadata metadata.yaml \
  --cache-dir ~/.febm-cache \
  --incremental true \
  --seed 0
```

The tool will:
1. Parse all `.md` files and extract H1 headings as chapter titles.
2. Merge metadata from `metadata.yaml` and validate ISBN‑13 checksum.
3. Render each chapter to XHTML 1.1 + CSS 2.1, then assemble the EPUB container.
4. Run `epubcheck` on the generated `.epub` file; if validation fails, the build exits with code 1.
5. Convert the EPUB to MOBI via `kindlegen` (v2.9) and to PDF via WeasyPrint (with `@page { size: 6x9in; margin: 1in; }`).
6. Write the final files as `my-book.epub`, `my-book.mobi`, `my-book.pdf`.

### Step 3: Verify Output Integrity & Cache Status

After the build, verify the cache hit ratio and output hash:

```bash
febm-cache-stats --cache-dir ~/.febm-cache
# Output: "Cache entries: 47, Hits: 42, Misses: 5, Hit ratio: 89.36%"

sha256sum my-book.epub
# Compare against a known‑good hash from a previous build.
```

To force a full rebuild (ignoring cache), add `--no-cache`:

```bash
febm-render . --no-cache --seed 0
```

## Best Practices

### Practice 1: Always Use a Fixed Seed for CI/CD Reproducibility

Set `--seed 0` (or any constant integer) in your build scripts. This ensures that the order of chapters, image compression parameters, and table of contents generation are identical across every build. Without a fixed seed, the tool may randomise certain internal sort orders, leading to non‑deterministic output — a common source of downstream QA failures. Integrate this into your `Makefile` or `justfile`:

```makefile
build:
	febm-render . --seed 0 --output-format epub --validate-schema true
```

### Practice 2: Leverage Incremental Caching for Large Repositories

When working with a repository of 500+ chapters (e.g., a technical manual series), always enable `--incremental true` and point `--cache-dir` to a persistent volume (e.g., `/tmp/febm-cache` on Linux or a Docker volume). For CI runners, mount the cache directory across builds to avoid re‑rendering unchanged chapters. Monitor cache hit ratio via `febm-cache-stats`; if it drops below 70%, consider increasing the cache size with `--max-cache-size 4G`.
