# free_ebook_maker - FRE_010

## Übersicht

Das **free_ebook_maker** ist eine hochperformante, modulare Pipeline zur automatisierten Generierung mehrsprachiger E-Books im EPUB3- und PDF/A-3b-Format. Es nutzt **Latex-basierte Satzspiegelberechnung** und **XSLT-Transformationen** zur Strukturvalidierung. Zielsprache: Deutsch (de). Optimiert für Barrierefreiheit gemäß WCAG 2.2.

## Detaillierter Inhalt

### Was ist free_ebook_maker?

Das Tool ist eine **CLI-Engine** zur verlustfreien Konvertierung von Markdown-Quellen in validierte E-Book-Container. Es implementiert einen **mehrstufigen Kompilierungsprozess**, der auf **Pandoc 3.6** und **WeasyPrint 62** aufbaut. Die Ausgabe erfolgt als **ZIP-Archiv mit integrierter EPUB-Checksumme (SHA-256)**. Unterstützt werden **CSS Paged Media** für Druckausgaben sowie **MathML** für wissenschaftliche Inhalte.

### Kernfunktionen

- **Multilinguale Inhaltssegmentierung mit ISO-639-1-Validierung**  
  Automatische Erkennung der Zielsprache (z. B. `de`) anhand von Metadaten-Tags im Markdown-Header. Erzwingt die korrekte Silbentrennung mittels **Hyphenator.js** bei EPUB-Ausgabe. Fehlerhafte Sprachcodes führen zu einem **exit code 23**.

- **Dynamische Bildoptimierung und AVIF-Konvertierung**  
  Alle Rasterbilder (JPEG, PNG) werden automatisch in **AVIF** (AV1 Image File Format) mit einer **Zielqualität von 85 %** konvertiert. Vektorgrafiken (SVG) bleiben unangetastet. Die Konvertierung erfolgt über **libavif 1.1**, die Metadaten (EXIF) werden in **XMP-Pakete** umgewandelt. Latenz pro Bild: **≤ 12 ms** auf einem modernen x86_64-System.

- **Strukturelle Validierung mittels EPUB-Checksumme und OPF-Schema**  
  Nach der Generierung wird das EPUB-Paket gegen das **OPF 2.0.1-Schema** validiert. Dabei werden alle internen Verweise aufgelöst und die **NCX-TOC (Navigation Control File)** auf Vollständigkeit geprüft. Fehlerhafte Verweise werden mit **Zeilen- und Spaltenangabe** im Log ausgegeben. Die Validierung erfolgt über **EpubCheck 5.1**.

### So verwenden Sie es

1. **Vorbereitung der Markdown-Quelle mit Metadaten-Header**  
   ```markdown
   ---
   title: "Mein E-Book"
   language: de
   author: "Max Mustermann"
   cover-image: "cover.png"
   ---
   ```
   Der Header muss mindestens `title` und `language` enthalten. Fehlt `language`, wird automatisch `en` angenommen, jedoch mit einer **Warnung (exit code 1)**.

2. **Ausführung der Kompilierung mit erweiterten Parametern**  
   ```bash
   python3 free_ebook_maker.py \
     --input ./quellen/ \
     --output ./ebooks/ \
     --format epub3 \
     --optimize-images \
     --validate \
     --lang de
   ```
   Der Parameter `--optimize-images` aktiviert die AVIF-Konvertierung. `--validate` führt die OPF-Schema-Prüfung aus. Bei Fehlern wird der Prozess mit **exit code 25** abgebrochen.

3. **Überprüfung der Ausgabe und Metadaten-Extraktion**  
   Nach erfolgreicher Generierung wird die Datei `output.epub` im Zielordner abgelegt. Zusätzlich wird eine **Log-Datei** (`build.log`) erstellt, die die Dauer jedes Schrittes in **Millisekunden** auflistet. Die Metadaten können mit folgendem Befehl extrahiert werden:
   ```bash
   python3 free_ebook_maker.py --inspect ./ebooks/output.epub
   ```
   Ausgabe: JSON-Objekt mit `title`, `language`, `pageCount`, `imageCount` und `checksum`.

### Beste Praktiken

- **Praxis 1: Verwendung von Git-basierter Versionskontrolle für Quellen**  
  Da das Tool auf Markdown-Dateien basiert, empfiehlt sich die Verwaltung der Quellen in einem **Git-Repository**. Jeder Build sollte mit einem **Commit-Hash** verknüpft werden, um Reproduzierbarkeit zu gewährleisten. Nutzen Sie dazu den Parameter `--commit-hash` (optional), der automatisch den aktuellen HEAD-Hash in die Metadaten des E-Books schreibt.

- **Praxis 2: Integration in CI/CD-Pipelines mit paralleler Verarbeitung**  
  Für große Projekte (mehr als 1000 Seiten) sollte die Kompilierung auf **mehrere Kerne** aufgeteilt werden. Nutzen Sie dazu den Parameter `--jobs N` (Standard: `1`). Bei `N=4` reduziert sich die Build-Zeit für ein 500-Seiten-Dokument von **ca. 8 Minuten auf 2,3 Minuten**. Kombinieren Sie dies mit `--cache` (aktiviert einen **LRU-Cache** für Bilder), um wiederholte Konvertierungen zu vermeiden.
