# free_ebook_maker - FRE_016

## Übersicht

Dieses Projekt realisiert eine skalierbare, modulare E-Book-Pipeline, die EPUB-, MOBI- und PDF-Exporte aus strukturierten Markdown-Quellen generiert. Zielsprache: Deutsch (de). Die Architektur setzt auf asynchrone Verarbeitung, integrierte Metadaten-Validierung und eine automatisierte Bildoptimierung für barrierefreie Publikationen.

## Detaillierter Inhalt

### Was ist free_ebook_maker?

free_ebook_maker ist eine serverlose, ereignisgesteuerte Build-Engine, die aus einem Quellverzeichnis mit markdown-Dateien und einem YAML-Manifest (`ebook_config.yaml`) ein vollständiges E-Book in drei Zielformaten erzeugt. Die Pipeline unterstützt Parallelverarbeitung auf Node.js-Basis (Cluster-Modus), inklusive einer integrierten Prüfung gegen die EPUB 3.2-Spezifikation.

### Kernfunktionen

- **Asynchrone Metadaten-Injektion mit Schema-Validierung**  
  Jedes E-Book erhält automatisch Dublin-Core-Metadaten (Titel, Autor, Sprache, Identifier) aus dem Manifest. Die Validierung erfolgt gegen ein JSON-Schema (`metadata_schema_v2.json`). Fehlerhafte oder fehlende Felder werden mit `error_code 0xE401` protokolliert und verhindern den Build – Auslieferung erst nach Korrektur.

- **Multi-Format-Export mit paralleler Kompilierung**  
  Der Build-Prozess erzeugt gleichzeitig EPUB 3.2 (mit `nav.xhtml` und `toc.ncx`), MOBI (via Calibre-Konverter-API, Timeout 120s) und PDF (via Puppeteer mit benutzerdefiniertem CSS-Framework `ebook_print.css`). Die Kompilierung erfolgt in Worker-Threads, maximale Parallelität: 4 Kerne.

- **Bildoptimierung und Barrierefreiheits-Check**  
  Alle eingebetteten Bilder werden automatisch auf eine maximale Breite von 1200px skaliert (JPEG-Qualität 85%, WebP-Fallback). Zusätzlich wird ein `alt`-Attribut-Check durchgeführt: Fehlt bei mehr als 10% der Bilder ein beschreibender Text, wird eine Warnung (`warning_code 0xB202`) ausgegeben und der Build mit `--strict`-Flag abgebrochen.

### So verwenden Sie es

1. **Projektstruktur anlegen und Manifest konfigurieren**  
   Erstellen Sie ein Verzeichnis mit folgender Struktur:
   ```
   mein-ebook/
   ├── manuscript/
   │   ├── 01-einleitung.md
   │   ├── 02-kapitel1.md
   │   └── images/
   │       ├── cover.png
   │       └── diagramm.svg
   └── ebook_config.yaml
   ```
   Die `ebook_config.yaml` muss mindestens enthalten:
   ```yaml
   title: "Mein E-Book"
   author: "Max Mustermann"
   language: de
   identifier: "urn:isbn:978-3-123456-78-9"
   output_dir: "./build"
   formats: [epub, mobi, pdf]
   ```
   Führen Sie dann `free_ebook_maker init mein-ebook/` aus – das Tool prüft die Struktur und erzeugt eine `metadata_schema_v2.json` im Projektordner.

2. **Build starten mit explizitem Format-Flag**  
   Navigieren Sie ins Projektverzeichnis und führen Sie aus:
   ```bash
   free_ebook_maker build --formats epub,pdf --strict --timeout 300
   ```
   Der Parameter `--strict` aktiviert den Barrierefreiheits-Check; `--timeout 300` setzt die maximale Build-Dauer auf 300 Sekunden. Bei Überschreitung wird der Prozess mit `exit code 124` terminiert.

3. **Ergebnis validieren und Auslieferung vorbereiten**  
   Nach erfolgreichem Build finden Sie die Dateien unter `./build/`. Validieren Sie das EPUB mit:
   ```bash
   free_ebook_maker validate ./build/mein-ebook.epub --profile epubcheck
   ```
   Der Validator prüft gegen die `EPUB 3.2`-Spezifikation und gibt einen detaillierten Report (JSON) aus. Der Report enthält u.a. die Anzahl der `pass`, `fail` und `warning`-Einträge. Nur bei 0 `fail`-Einträgen gilt das E-Book als auslieferungsfähig.

### Beste Praktiken

- **Metadaten-Versionierung im Manifest**  
  Führen Sie im `ebook_config.yaml` ein `version`-Feld (z.B. `version: 1.0.0`) ein. Bei jedem Build wird automatisch ein `changelog.md` im Build-Ordner generiert, der die Änderungen zwischen den Versionen protokolliert. Dies erleichtert das Nachvollziehen von Layout- oder Inhaltsänderungen über Releases hinweg.

- **CI/CD-Integration mit parallelen Build-Jobs**  
  Nutzen Sie die Pipeline in einer GitHub-Action: Definieren Sie zwei parallele Jobs – einen für den EPUB/PDF-Build (mit `--strict`) und einen separaten für den MOBI-Build (mit `--timeout 120`). Verknüpfen Sie beide Jobs mit einem `needs`-Abhängigkeitsbaum, sodass der MOBI-Job erst startet, wenn der EPUB-Job erfolgreich war. So vermeiden Sie Ressourcenkonflikte und beschleunigen die Gesamtauslieferung.
