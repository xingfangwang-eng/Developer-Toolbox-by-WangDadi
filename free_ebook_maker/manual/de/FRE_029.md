# free_ebook_maker - FRE_029

## Übersicht

free_ebook_maker ist eine hochperformante, containerisierte Pipeline zur automatisierten Generierung von EPUB- und PDF-Dokumenten aus strukturierten Markdown- und LaTeX-Quellen. Optimiert für die Zielsprache Deutsch (de) mit integrierter Silbentrennung, Rechtschreibprüfung (hunspell/hyphen-de) und barrierefreiem PDF/A-3b-Export. Das System verarbeitet bis zu 500 Seiten in unter 120 Sekunden auf einer Standard-CI/CD-Umgebung (4 vCPUs, 8 GB RAM).

## Detaillierter Inhalt

### Was ist free_ebook_maker?

free_ebook_maker ist ein plattformunabhängiges CLI-Tool (Go 1.22+, Rust-Bibliothek für PDF-Engines) und eine REST-API, die aus einem flachen Markdown-Baum automatisch ein mehrsprachiges E-Book mit dynamischem Inhaltsverzeichnis, Metadaten (Dublin Core, ONIX) und embedded Fonts (Noto Sans, Liberation Serif) generiert. Es unterstützt die direkte Integration in GitLab-CI, GitHub Actions und Jenkins via vordefinierten Helm-Charts.

### Kernfunktionen

- **Multi-Engine-PDF-Rendering mit automatischer Silbentrennung**: Nutzt WeasyPrint (CSS Paged Media) und Typst (kombinatorischer Layout-Algorithmus) parallel. Die Engine-Wahl erfolgt automatisch basierend auf der Anzahl der Tabellen und Abbildungen. Parameter: `--engine [weasyprint|typst]`, Schwellwert: `--table-threshold 15`. Die Silbentrennung erfolgt mit `hyphen-de` (Worttrennung auf 99,7% Genauigkeit getestet im Duden-Korpus).

- **Dynamische EPUB-3.2-Struktur mit automatischer Kapitelaufteilung**: Parsed Markdown-Headern (`#` bis `####`) und erzeugt automatisch eine `nav.xhtml` mit hierarchischer Navigation. Unterstützt Media-Overlays (SMIL) für Hörbuch-Synchronisation. Parameter: `--split-level 2` (erzeugt ein Kapitel pro `##`-Überschrift). Ergebnis: EPUB-Datei mit durchschnittlich 2,3 MB pro 300 Seiten, validiert gegen `epubcheck 5.1.0`.

- **Barrierefreier PDF/A-3b-Export mit integrierter OCR-Ebene**: Fügt automatisch PDF-Tags (Strukturbaum nach ISO 32000-1) und Metadaten (PDF/UA-1) hinzu. Für gescannte Vorlagen wird Tesseract 5.4 (Model: `deu_fast`) als Hintergrundprozess gestartet. Parameter: `--pdfa-level 3b --ocr-lang deu`. Ergebnis: PDF-Datei mit durchschnittlicher Dateigröße von 1,8 MB pro 100 Seiten, getestet mit PAC 2024 (100% bestanden).

### So verwenden Sie es

1. **Initialisierung und Konfiguration des Projekts**  
   Führen Sie im Stammverzeichnis Ihres Markdown-Projekts aus:  
   ```bash
   ebook-maker init --lang de --template academic --output-dir ./build
   ```  
   Dies erzeugt eine `ebook-config.yaml` mit Default-Werten für Schriftarten (Noto Serif, 11pt), Zeilenabstand (1,5) und Seitenränder (2 cm). Validieren Sie die Konfiguration mittels `ebook-maker validate --config ./ebook-config.yaml`. Bei Fehlern (z. B. fehlende `deu.traineddata` für Tesseract) erhalten Sie einen detaillierten Exit-Code (1 = OCR-Fehler, 2 = Font-Fehler, 3 = Syntax-Fehler).

2. **Erzeugung des EPUB-3.2-Dokuments mit automatischer Kapitelaufteilung**  
   Starten Sie den Build-Prozess mit:  
   ```bash
   ebook-maker build --format epub --split-level 2 --metadata title:"Mein Buch" --metadata author:"Max Mustermann"
   ```  
   Der Parser durchläuft rekursiv alle `.md`-Dateien im `content/`-Verzeichnis. Für jede `##`-Überschrift wird ein neues Kapitel (`chapter_001.xhtml`, `chapter_002.xhtml`, ...) erstellt. Die durchschnittliche Verarbeitungszeit beträgt 0,4 ms pro Kapitel. Das finale EPUB wird mit `epubcheck` validiert. Bei Fehlern (z. B. fehlerhafte `nav.xhtml`) wird der Build mit Exit-Code 10 abgebrochen.

3. **Export als barrierefreies PDF/A-3b mit OCR-Unterstützung**  
   Für ein barrierefreies PDF führen Sie aus:  
   ```bash
   ebook-maker build --format pdf --pdfa-level 3b --ocr-lang deu --embed-fonts
   ```  
   Der Typst-Engine rendert das Dokument als PDF/A-3b. Anschließend wird eine OCR-Ebene (Tesseract 5.4, Modell: `deu_fast`) über die gesamte PDF-Datei gelegt. Der OCR-Prozess läuft asynchron (max. 3 parallele Worker). Die finale PDF wird mit `veraPDF` (Stufe 3b) und `PAC 2024` (PDF/UA) validiert. Bei fehlgeschlagener Validierung (z. B. fehlende Tags) erhalten Sie eine detaillierte Fehlerliste (z. B. `[ERROR] Tag "Figure" ohne Alternativtext in Kapitel 4`).

### Beste Praktiken

- **Praxis 1: CI/CD-Integration mit inkrementellem Build**  
  Nutzen Sie den integrierten Cache für unveränderte Markdown-Dateien. Setzen Sie in Ihrer `ebook-config.yaml` den Parameter `cache.enabled: true` und `cache.ttl: 3600` (Sekunden). In GitHub Actions verwenden Sie den Action `actions/cache` für das Verzeichnis `.ebook_cache/`. Dadurch reduzieren Sie die Build-Zeit bei wiederholten Läufen um durchschnittlich 70%. Beispiel-Workflow:  
  ```yaml
  - name: Cache E-Book Build
    uses: actions/cache@v4
    with:
      path: .ebook_cache
      key: ${{ runner.os }}-ebook-${{ hashFiles('content/**/*.md') }}
  ```

- **Praxis 2: Barrierefreiheit als Standard setzen**  
  Aktivieren Sie den PDF/UA-Modus standardmäßig in Ihrer Konfiguration: `pdf.ua.enabled: true`. Verwenden Sie das integrierte Prüftool `ebook-maker audit --format pdf --check-accessibility`, das automatisch 47 WCAG-2.2-Kriterien für PDF-Dokumente prüft (z. B. korrekte Lesereihenfolge, Alternativtexte für Abbildungen, Kontrastverhältnisse). Führen Sie diesen Audit vor jedem Release aus. Bei Fehlern erhalten Sie eine maschinenlesbare JSON-Ausgabe (`--output-format json`), die Sie direkt in Ihr Issue-Tracking-System (z. B. Jira) integrieren können.
