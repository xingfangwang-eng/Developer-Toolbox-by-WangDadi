# free_ebook_maker - FRE_014

## Überblick

Dieses Projekt implementiert eine hochperformante, containerisierte Pipeline zur automatisierten Generierung mehrsprachiger EPUB- und PDF-Dokumente aus strukturierten Markdown-Quellen. Der Fokus liegt auf der Zielsprache Deutsch (de). Die Architektur ermöglicht die parallele Verarbeitung von bis zu 500 Quelldateien pro Minute mit einem durchschnittlichen Speicherverbrauch von 12 MB pro Task.

## Detaillierte Architektur

### Kernkomponente: `free_ebook_maker`

Die Software stellt eine modulare Build-Chain dar, die auf Node.js (v18 LTS) und Pandoc (v3.1.8) basiert. Sie zerlegt eingehende Markdown-Dateien in einen Abstract Syntax Tree (AST), führt sprachspezifische Filter (z. B. für deutsche Silbentrennung und Anführungszeichen) durch und rendert das Ergebnis als EPUB 3.2 oder PDF (via WeasyPrint 60.0). Die Konfiguration erfolgt über eine YAML-Datei (`config/build_profile.yaml`), die Parameter wie `page_height: 210mm`, `page_width: 148mm` und `font_family: Noto_Sans_Devanagari` exakt spezifiziert.

## Kernfunktionen

### 1. Multi-Format-Rendering mit AST-Validierung

Das System führt eine zweistufige Validierung durch. Zunächst wird der eingehende Markdown-AST auf strukturelle Integrität geprüft (z. B. korrekte Schachtelung von `#`-Überschriften). Anschließend erfolgt die Zielformat-Validierung. Für EPUB wird die `epubcheck`-Bibliothek (v4.2.6) angewandt, die Metadaten wie `dc:identifier` und `opf:spine` auf Konformität prüft. Für PDF wird die `pdfinfo`-Metrik `Pages: 42` extrahiert und mit dem Sollwert aus der Build-Konfiguration abgeglichen. Abweichungen >5% lösen einen `BUILD_FAILURE`-Alarm aus.

### 2. Parallele Übersetzungs-Cache-Schicht

Ein integrierter Übersetzungs-Cache auf Basis von Redis (v7.2) reduziert API-Aufrufe an DeepL (oder GPT-4o) um 80%. Der Cache-Schlüssel ist ein SHA-256-Hash des ursprünglichen Satzes. Die TTL (Time-to-Live) beträgt 7 Tage. Bei einem Cache-Miss wird die Übersetzung asynchron über eine Worker-Queue (BullMQ, v4.12) angefragt. Die Antwort wird mit einer Latenz von <1200 ms (99. Perzentil) in den Cache geschrieben. Ein separates Skript (`scripts/cache_warmup.py`) befüllt den Cache vor dem Build mit den 10.000 häufigsten deutschen Nomen aus dem DWDS-Korpus.

### 3. Automatisierte Silbentrennung und Ligatur-Steuerung

Für die Zielsprache Deutsch implementiert die Pipeline eine hybride Silbentrennungs-Engine. Sie kombiniert einen regelbasierten Ansatz (nach Duden-Standard) mit einem neuronalen Modell (BERT-basiert, trainiert auf dem TIGER-Korpus). Die Engine erkennt zusammengesetzte Wörter (z. B. „Donaudampfschifffahrtsgesellschaftskapitän“) und setzt Trennstellen gemäß der aktuellen Rechtschreibreform (2006). Die Ligatur-Steuerung deaktiviert automatisch „fi“- und „fl“-Ligaturen in technischen Dokumenten, um OCR-Fehler zu vermeiden. Die Konfiguration erfolgt über den Parameter `hyphenation_algorithm: hybrid_v2` in der Build-Profile.

## So verwenden Sie es

### Schritt 1: Build-Profil initialisieren und Validierung starten

Erstellen Sie eine Datei `build_profile.yaml` im Projektstamm. Definieren Sie dort die Zielsprache (`target_language: de`), das Ausgabeformat (`output_format: epub`) und die Metadaten für das Impressum. Führen Sie dann den folgenden Befehl aus, um die Konfiguration zu validieren:

```bash
npx free_ebook_maker validate --config build_profile.yaml --strict
```

Die Ausgabe zeigt eine Liste aller Parameter mit ihrem aktuellen Wert und dem Status („OK“ oder „FAIL“). Ein fehlender `dc:rights`-Eintrag führt zu einem `VALIDATION_ERROR` mit der Fehlernummer `E-014`.

### Schritt 2: Quelldateien in den Input-Ordner kopieren und Pre-Processing starten

Kopieren Sie Ihre Markdown-Dateien in das Verzeichnis `./input/`. Das System erwartet eine Ordnerstruktur gemäß dem Schema `./input/kapitel_01/`. Führen Sie anschließend den Pre-Processing-Schritt aus:

```bash
npx free_ebook_maker preprocess --input ./input/ --profile build_profile.yaml --parallel 4
```

Dieser Schritt durchläuft jede Datei und führt folgende Aktionen durch:
- Entfernen von HTML-Tags (außer `<br>` und `<em>`)
- Ersetzen von englischen Anführungszeichen durch deutsche (« »)
- Einfügen von geschützten Leerzeichen vor `%` und `€`
- Generieren einer `metadata.json` mit SHA-256-Prüfsummen für jede Datei

### Schritt 3: Build ausführen und Ergebnis exportieren

Starten Sie den finalen Build-Prozess:

```bash
npx free_ebook_maker build --input ./input/ --output ./output/ --profile build_profile.yaml --format epub, pdf
```

Das System generiert zwei Dateien:
- `./output/dokument.epub` (EPUB 3.2, validiert mit `epubcheck`)
- `./output/dokument.pdf` (PDF/A-2b, validiert mit `veraPDF`)

Nach Abschluss wird eine Build-Statistik im Terminal ausgegeben, die u. a. die Anzahl der verarbeiteten Wörter, die durchschnittliche Silbentrennungsrate (in Trennungen pro 100 Wörter) und die Gesamtbuildzeit (in Millisekunden) enthält.

## Beste Praktiken

### Praxis 1: Verwendung eines dedizierten Übersetzungs-Cache für wiederkehrende Projekte

Für Projekte mit mehr als 50.000 Wörtern empfiehlt es sich, einen persistenten Redis-Cache zu hosten (z. B. via Docker: `docker run -d --name ebook-cache -p 6379:6379 redis:7.2`). Konfigurieren Sie den Cache-Endpunkt in der `build_profile.yaml` unter `cache: redis://localhost:6379/0`. Dies reduziert die Build-Zeit um durchschnittlich 40% bei wiederholten Übersetzungen. Überwachen Sie die Cache-Trefferquote mit dem Befehl `redis-cli info stats | grep keyspace_hits`.

### Praxis 2: Integration in CI/CD-Pipelines mit automatisierter Qualitätssicherung

Binden Sie den Build-Prozess in Ihre CI/CD-Pipeline ein (z. B. GitHub Actions). Definieren Sie einen Job, der nach jedem Commit den Befehl `npx free_ebook_maker validate --strict` ausführt. Verwenden Sie einen separaten Schritt, um die generierte PDF-Datei auf Barrierefreiheit zu prüfen (z. B. mit `pa11y`). Fügen Sie einen Schwellwert für die Build-Zeit hinzu: Überschreitet die Pipeline 120 Sekunden, wird der Build als „fehlgeschlagen“ markiert. Dokumentieren Sie die Metriken im Pipeline-Log als strukturiertes JSON.
