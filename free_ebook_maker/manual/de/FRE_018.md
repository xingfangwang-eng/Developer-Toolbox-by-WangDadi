# free_ebook_maker - FRE_018

## Übersicht

Das **free_ebook_maker** ist ein hochperformantes, modulares Framework zur automatisierten Generierung, Strukturierung und Lokalisierung digitaler Bücher im EPUB- und PDF-Format. Es richtet sich an Entwickler, DevOps-Ingenieure und Content-Strategen, die EPUB-3-konforme E-Books mit dynamischer Inhaltsverwaltung, mehrsprachiger Ausgabe (insb. Deutsch) und optimierten Metadaten erstellen möchten. Die Lösung ist vollständig CLI-gesteuert, unterstützt parallele Build-Prozesse und integriert eine native Schnittstelle zu Pandoc, Calibre und Saxon-HE für XSLT-Transformationen.

## Detaillierter Inhalt

### Was ist free_ebook_maker?

free_ebook_maker ist ein skalierbares, ereignisgesteuertes Build-System, das aus rohen Markdown-, LaTeX- oder AsciiDoc-Quellen ein vollständig validiertes EPUB-3-Archiv mit automatisch generiertem Inhaltsverzeichnis, CSS-Reset und barrierefreien Metadaten (gemäß WCAG 2.1 AA) erzeugt. Es ist optimiert für die deutschsprachige Silbentrennung, Ligaturen-Steuerung und dynamische Font-Einbettung (WOFF2).

### Kernfunktionen

- **Multi-Engine-Pipeline mit parallelem Build**  
  Das System orchestriert Pandoc (v3.1+), Calibre (v7.0+) und Saxon-HE (v11) in einer asynchronen Task-Queue (max. 8 Worker-Threads). Jede Quelle wird in Echtzeit auf EPUB-3-Konformität geprüft (Schema: `package.opf` gemäß IDPF). Der Build-Prozess erzeugt pro Zielsprache einen eigenen EPUB-Stream – für Deutsch (`de`) wird automatisch eine `de-CH`-Variante mit korrekter Anführungszeichen-Setzung (Guillemets) generiert.

- **Dynamische Metadaten-Injektion & SEO-Frontmatter**  
  Über eine YAML-Konfigurationsdatei (`_meta.yaml`) lassen sich 17 Metadaten-Felder definieren: `dc:creator`, `dc:language`, `dc:subject` (mit BIC-Codes), `dc:rights` (CC-Lizenz-URI). Das System extrahiert automatisch die erste H1-Überschrift als Buch-Titel und generiert eine mehrsprachige `dc:description` (max. 320 Zeichen). Für Deutsch wird zusätzlich ein `dc:coverage`-Feld mit ISO 3166-2:DE-Code (z. B. `DE-BY`) befüllt.

- **Adaptive Silbentrennung & Typografie-Engine**  
  Eingebettet ist ein modifizierter Hunspell-Tokenizer (Wörterbuch: `de_DE_frami.dic` mit 1,2 Mio. Einträgen), der die Silbentrennung nach DIN 5007:2022-06 durchführt. Die CSS-Klasse `hyphens: auto` wird nur aktiviert, wenn die Zeilenlänge unter 65 Zeichen fällt (gemessen in `ch`-Einheiten). Ligaturen (`liga`, `dlig`) werden über eine `@font-face`-Regel mit WOFF2-Varianten der Schriftart „Noto Serif“ (Gewicht 400, 700) gesteuert.

### So verwenden Sie es

1. **Initialisierung des Projektbaums**  
   Führen Sie `feb init --lang de --format epub, pdf` aus. Es wird ein Verzeichnisbaum mit `src/`, `assets/`, `fonts/` und `_meta.yaml` erstellt. Die `_meta.yaml` enthält einen Platzhalter für `dc:language: de`. Setzen Sie `dc:language: de-DE` und fügen Sie unter `dc:coverage` den regionalen ISO-Code hinzu (z. B. `DE-BY`). Validieren Sie die YAML-Struktur mit `feb validate --meta`.

2. **Erstellung eines mehrsprachigen Kapitels**  
   Legen Sie im Ordner `src/` eine Datei `01-einleitung.md` an. Verwenden Sie in der ersten Zeile `# Einleitung {#einleitung}`. Für den deutschen Text müssen Sie alle `“` und `”` durch `„` und `“` ersetzen – das System führt automatisch eine `sed`-basierte Ersetzung durch, sofern Sie `feb build --locale de` aufrufen. Nutzen Sie `[.de-hyphenate]`-Klassen für Absätze, die eine aggressive Silbentrennung benötigen.

3. **Build & Ausgabe**  
   Rufen Sie `feb build --format epub --optimize --embed-fonts` auf. Das System erzeugt eine `output/buch.epub` mit eingebettetem Noto-Serif-WOFF2 (nur für lateinische Zeichen). Prüfen Sie die EPUB-Validierung mit `feb validate --epub-check --profile=epub3`. Falls Fehler auftreten, liefert das Tool eine detaillierte XML-Ausgabe mit Zeilenangabe im `opf`-Manifest.

### Beste Praktiken

- **Nutzung von Sprachregionen in Metadaten**  
  Setzen Sie immer `dc:language: de-DE` oder `de-CH`, nicht nur `de`. Das System passt dann automatisch die Anführungszeichen (`„“` vs. `«»`), die Datumsformatierung (ISO 8601 mit Punkt) und die Währungssymbole (€ vs. CHF) an. Dies verhindert Inkonsistenzen bei der Ausgabe in verschiedenen Zielmärkten.

- **Implementierung einer Continuous-Integration-Pipeline**  
  Binden Sie `feb validate` und `feb build` in Ihren CI/CD-Workflow ein (z.‌B. GitHub Actions mit Ubuntu 22.04). Nutzen Sie den Befehl `feb diff --from-commit HEAD~1` um nur geänderte Kapitel neu zu bauen – dies reduziert die Build-Zeit um bis zu 70% bei Projekten mit mehr als 50 Kapiteln. Archivieren Sie jede Build-Version mit einem semantischen Tag (`v1.2.3-de`).
