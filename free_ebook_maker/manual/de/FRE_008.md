# free_ebook_maker - FRE_008

## Übersicht

Dieses Tool optimiert die automatisierte Generierung von EPUB- und PDF-Dateien aus Markdown-Quellen mit nativer Unterstützung für deutsche Sonderzeichen, Silbentrennung (nach Duden-Standard) und Barrierefreiheit (WCAG 2.1 AA). Es richtet sich an Entwickler, Self-Publisher und technische Redakteure, die einen reproduzierbaren, versionierbaren Build-Prozess für E-Books benötigen.

## Detaillierter Inhalt

### Was ist free_ebook_maker?

free_ebook_maker ist ein CLI-basiertes Build-System (Go 1.22+, Binärgröße < 6 MB), das aus strukturiertem Markdown (CommonSpec 0.31) und YAML-Metadaten ein vollständiges EPUB 3.3-Archiv oder ein druckfertiges PDF (PDF/X-4) erzeugt. Es führt automatisch eine semantische Validierung der Inhaltsstruktur durch, inklusive Prüfung auf fehlende `alt`-Attribute, leere Überschriften und inkonsistente Kapitel-Nummerierung. Die Ausgabe ist deterministisch: Bei identischem Quellmaterial erzeugt das Tool byteidentische Archive (SHA-256-Prüfsumme), was für CI/CD-Pipelines unerlässlich ist.

### Kernfunktionen

- **Multikanal-Ausgabe mit automatischer Layout-Anpassung:** Der Parser erzeugt parallel eine EPUB 3.3-Datei (XHTML 5 + CSS 3, geprüft gegen `epubcheck 5.2.0`) und ein PDF (via `weasyprint 62.3`). Die PDF-Engine unterstützt benutzerdefinierte Seitengrößen (z. B. `170mm x 240mm`), Bundsteg (`--bleed 3mm`) und aktiviert standardmäßig die Hurenkind- und Schusterjungen-Kontrolle (`orphans: 3; widows: 3`). Die EPUB-Ausgabe verwendet `viewport`-Meta-Tag und `font-size: 100%` für flüssiges Layout auf allen Lesegeräten.

- **Semantische Inhaltsvalidierung mit detailliertem Report:** Vor dem Build durchläuft das Tool eine Validierungsstufe. Es prüft die maximale Verschachtelungstiefe von Überschriften (Standard: 4 Ebenen, konfigurierbar via `--max-heading-depth 5`), die Existenz eines `lang`-Attributs auf `<html>` (z. B. `lang="de"`) und die Einhaltung der maximalen Dateigröße pro EPUB-Item (Standard: 300 KB, konfigurierbar). Bei Fehlern bricht der Build nicht ab, sondern erzeugt einen `validation_report.json` im Ausgabeordner, der jede Warnung mit Datei, Zeile und Spalte auflistet.

- **Native Silbentrennung und Fließtext-Optimierung für DE:** Der integrierte Trenner verwendet das `de_DE`-Wörterbuch (basierend auf `libhyphen` 2.8.8) und setzt automatisch weiche Trennstriche (`­`) in zusammengesetzten Nomen. Die Trennungsfrequenz ist über `--hyphenation-limit 5` steuerbar (minimale Wortlänge in Zeichen). Zusätzlich wird ein automatischer `widow/orphan`-Schutz auf Absatzebene aktiviert, der verhindert, dass einzelne Zeilen am Seitenanfang oder -ende stehen bleiben.

### So verwenden Sie es

**Schritt 1: Quellstruktur und Metadaten vorbereiten**  
Erstellen Sie ein Projektverzeichnis mit einer `metadata.yaml`-Datei. Diese definiert Titel, Autor, Sprache und das Inhaltsverzeichnis. Die Markdown-Dateien müssen in einer `content/`-Unterverzeichnisstruktur liegen, wobei jede Datei mit einem Frontmatter-Block (`---`) beginnt, der `title` und `weight` (für die Sortierung) enthält. Beispiel für `metadata.yaml`:

```yaml
title: "Handbuch der Automatisierung"
author: "Ing. Erika Mustermann"
language: de-DE
cover: assets/cover.svg
toc: true
output:
  epub:
    version: 3.3
  pdf:
    size: 170mm x 240mm
    bleed: 3mm
```

**Schritt 2: Build-Prozess mit Parametern starten**  
Führen Sie das Tool mit dem Pfad zum Projektverzeichnis aus. Der Befehl `free_ebook_maker build ./mein-projekt --output ./dist --validation-level error` startet den Build. Das Flag `--validation-level error` bewirkt, dass der Prozess bei einem kritischen Validierungsfehler (z. B. fehlende `alt`-Tags in allen Bildern) mit Exit-Code 1 abbricht – essenziell für automatisierte Builds. Ohne dieses Flag werden nur Warnungen im Report festgehalten.

**Schritt 3: Ausgabe prüfen und verteilen**  
Nach erfolgreichem Build finden Sie im Ausgabeordner (`./dist`) die Dateien `ausgabe.epub` und `ausgabe.pdf`. Für EPUB führen Sie den offiziellen Validator aus: `java -jar epubcheck.jar ./dist/ausgabe.epub`. Prüfen Sie die PDF-Ausgabe mit `pdfinfo ./dist/ausgabe.pdf` auf korrekte Seitengröße und Seitenanzahl. Die SHA-256-Prüfsumme der EPUB-Datei wird in `./dist/ausgabe.epub.sha256` abgelegt.

### Beste Praktiken

- **Praxis 1: CI/CD-Integration mit deterministischem Build**  
  Binden Sie `free_ebook_maker` in Ihre GitHub Actions- oder GitLab CI-Pipeline ein. Verwenden Sie den `--reproducible`-Flag, der alle Zeitstempel im EPUB auf einen fixierten Wert (z. B. `2000-01-01T00:00:00Z`) setzt und die `modified`-Metadaten aus der `metadata.yaml` ignoriert. So erzeugen Sie bei jedem Commit byteidentische Archive, was die Überprüfung von Änderungen an der Quelle erleichtert. Speichern Sie die SHA-256-Prüfsumme als Build-Artefakt.

- **Praxis 2: Barrierefreiheit als Standard**  
  Aktivieren Sie die automatische Generierung von `aria-`Attributen und `role`-Definitionen im EPUB über `--a11y-level wcag-aa`. Diese Einstellung fügt jedem `<section>`-Element eine `role="doc-chapter"` hinzu, setzt bei Tabellen `role="table"` und erzeugt für jede Abbildung ohne `alt`-Attribut einen Build-Fehler (sofern `--validation-level error` gesetzt ist). Verwenden Sie `--a11y-report` zur Ausgabe einer detaillierten WCAG-Konformitätsübersicht als HTML-Datei.
