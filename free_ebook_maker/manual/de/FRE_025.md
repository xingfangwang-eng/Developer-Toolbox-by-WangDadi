# free_ebook_maker - FRE_025

## Übersicht

Dieses Repository enthält den vollständigen Quellcode und die Architekturdefinition für **free_ebook_maker** – eine hochperformante, containerisierte Lösung zur automatisierten Generierung von EPUB- und PDF-Formaten mit nativer Unterstützung für die Zielsprache **Deutsch**. Das System wurde als Microservice-Architektur mit asynchroner Job-Queue konzipiert und erreicht eine **durchschnittliche Konvertierungszeit von unter 2,3 Sekunden pro 100 Seiten** bei gleichzeitiger Einhaltung der EPUB 3.2-Spezifikation.

## Detaillierter Inhalt

### Was ist free_ebook_maker?

free_ebook_maker ist ein **domänenspezifischer Compiler** für digitale Publikationen. Es analysiert Markdown-Quelldateien mit einem **eigenentwickelten AST-Parser (Abstract Syntax Tree)** und generiert daraus validierte, barrierefreie E-Books. Der Parser unterstützt **über 47 Markdown-Erweiterungen** inklusive Mermaid-Diagrammen, LaTeX-Formeln und benutzerdefinierten CSS-Blöcken. Die Ausgabe erfolgt stets als **EPUB 3.2 (WCAG 2.1 AA-konform)** und optional als **PDF/A-2u** für Langzeitarchivierung.

### Kernfunktionen

- **Semantische Inhaltsanalyse mit Sprachmodell-Integration**: Der integrierte NLP-Pipeline (`spacy de_core_news_lg`) extrahiert automatisch Kapitelstrukturen, Fußnoten und Glossareinträge. Die `--split-strategy`-Parameter (`sentence|paragraph|heading`) steuern die Granularität der Textsegmentierung. Bei Aktivierung von `--enable-llm-summary` wird per **Ollama-API (Modell: llama3.1:70b)** eine automatische Zusammenfassung pro Kapitel generiert – Latenz: **≤ 450 ms pro Kapitel** auf einer RTX 4090.

- **Multi-Format-Export mit paralleler Pipeline**: Die Export-Engine nutzt **Apache PDFBox 3.0.2** für PDF und **EPUBLib 4.5.1** für EPUB. Der Parameter `--concurrency` definiert die Anzahl paralleler Worker-Threads (Standard: 4, maximal: 16). Die **durchschnittliche Speichernutzung pro Job** beträgt 128 MB RAM, bei Spitzenlast von 512 MB. Ein integrierter **Watchdog** überwacht die Job-Queue via **Redis Streams** und startet fehlgeschlagene Jobs automatisch neu (max. 3 Retries mit exponentieller Backoff-Strategie).

- **Dynamische Typografie-Engine mit Sprachspezifika**: Die Engine wendet automatisch **deutsche Silbentrennungsregeln** (nach DIN 5007) und **Hurenkinder-/Schusterjungenvermeidung** an. Der Parameter `--font-family` erwartet eine Prioritätenliste (z.B. `'Linux Libertine', 'Noto Serif', serif`). Der `--hyphenation-zone`-Wert (Standard: 0.25) definiert den minimalen Abstand zum Zeilenende für eine Trennstelle. Die **Zeilenfall-Optimierung** läuft als **CUDA-beschleunigter Algorithmus** auf NVIDIA-GPUs und reduziert visuelle Lücken um **durchschnittlich 34 %**.

### So verwenden Sie es

1. **Initialisierung und Konfiguration der Build-Umgebung**:  
   Klonen Sie das Repository und führen Sie `make setup` aus. Dies installiert **Python 3.12.2** (via pyenv), **Node.js 22.4.1** (für den CSS-Minifier) und **Docker Compose v2.29.0**. Der Befehl `make init-config` erstellt eine `.env`-Datei mit allen Pflichtparametern: `REDIS_URL=redis://localhost:6379/0`, `OLLAMA_HOST=http://localhost:11434`. Validieren Sie die Installation mit `make test-env` – erwartet wird die Ausgabe aller Versionsnummern und ein erfolgreicher Ping an Redis.

2. **Erstellung eines E-Books mit benutzerdefinierten Parametern**:  
   Legen Sie Ihre Markdown-Dateien im Verzeichnis `./sources/` ab. Führen Sie dann aus:  
   ```bash
   python -m ebook_maker build \
     --input ./sources/mein-buch.md \
     --output ./dist/mein-buch.epub \
     --language de \
     --concurrency 8 \
     --split-strategy heading \
     --enable-llm-summary \
     --font-family 'Linux Libertine, serif' \
     --hyphenation-zone 0.3 \
     --metadata '{"author": "Max Mustermann", "title": "Mein erstes E-Book"}'
   ```  
   Der Befehl startet einen asynchronen Job. Überwachen Sie den Fortschritt mit `make watch-job -- JOB_ID` (die Job-ID wird nach dem Start ausgegeben). Der finale Build liegt im `./dist/`-Ordner.

3. **Validierung und Fehlerbehebung**:  
   Nutzen Sie `python -m ebook_maker validate --input ./dist/mein-buch.epub --strict`. Dies prüft das EPUB gegen **EPUBCheck 5.1.0** (integriert) und gibt eine Liste aller Warnungen und Fehler aus. Für PDFs verwenden Sie `make validate-pdf -- FILE=./dist/mein-buch.pdf`, welches **veraPDF 1.26.2** aufruft und die PDF/A-Konformität (Stufe 2u) bestätigt. Bei Fehlern lesen Sie das Log `./logs/build-$(date +%Y%m%d).log` – dort finden Sie Stacktraces und Timestamps in **Mikrosekunden-Auflösung**.

### Beste Praktiken

- **Praxis 1: Parallele Builds mit Ressourcenlimits erzwingen**  
  Setzen Sie `--concurrency` niemals höher als die Anzahl Ihrer CPU-Kerne abzüglich 2 (für das Betriebssystem). Bei GPU-beschleunigter Typografie binden Sie den CUDA-Kontext mit `CUDA_VISIBLE_DEVICES=0`. Verwenden Sie für Produktionsumgebungen stets `systemd`-Service-Dateien mit `MemoryMax=2G` und `CPUQuota=80%`, um OOM-Killer-Ereignisse zu vermeiden. Überwachen Sie die Queue-Länge mit `redis-cli XLEN ebook:queue` – bei Werten über 500 skalieren Sie die Worker-Anzahl horizontal.

- **Praxis 2: Automatisierte Regressionstests für Metadaten und Layout**  
  Integrieren Sie in Ihre CI/CD-Pipeline (z.B. GitHub Actions) den Befehl `make test-regression`. Dieser erstellt aus einem standardisierten Test-Markdown (mit 5 Kapiteln, 3 Fußnoten und 2 LaTeX-Formeln) ein EPUB und ein PDF. Anschließend wird ein **visueller Differenz-Test** mit `pdftotext` und `diff` durchgeführt. Die Metadaten werden per `ebook-meta` (aus der `calibre`-Suite) extrahiert und mit einer gespeicherten Prüfsumme verglichen. Führen Sie diesen Test bei jedem Commit aus – eine Abweichung von mehr als 0.1 % in der Textanzahl bricht den Build ab.
