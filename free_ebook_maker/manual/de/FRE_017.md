# free_ebook_maker - FRE_017

## Übersicht

Dieses Projekt stellt eine hochperformante, modulare E-Book-Pipeline zur Verfügung, die Rohdaten (Markdown, HTML, LaTeX, EPUB) in ein standardisiertes, mehrsprachiges Ausgabeformat (EPUB 3.2, PDF/A-4, Mobi) transformiert. Die Zielsprache ist Deutsch (de). Der Fokus liegt auf deterministischer Reproduzierbarkeit, minimaler Latenz bei der Metadaten-Extraktion und automatischer Barrierefreiheitsprüfung gemäß WCAG 2.2.

## Kernfunktionen

- **Dynamische Kapitel-Neugenerierung & Layout-Engine**  
  Nutzt einen auf `lxml` und `pandoc 3.1.12` basierenden Parser, der Kapitelstrukturen rekursiv analysiert und in ein internes DOM überführt. Parameter: `--chapter-depth=3` (maximale Verschachtelungstiefe), `--page-break-strategy=auto` (automatische Seitenumbrüche bei `<h2>`-Elementen). Die Engine berechnet die optimale Font-Größe (in `pt`) basierend auf dem Ziel-Medium (Tablet vs. E-Reader) und führt eine *Kerning-Korrektur* in Mikrosekunden pro Glyphe durch.

- **Semantische Metadaten-Extraktion & Validierung**  
  Extrahiert automatisch Dublin Core (DCMI) und ONIX 3.0 Metadaten aus dem Quelltext. Unterstützt `dc:title`, `dc:creator`, `dc:identifier` (ISBN/DOI). Validiert die Metadaten gegen ein lokales Schema (JSON-Schema-Datei: `schemas/metadata_v2.json`). Bei Fehlern wird ein detaillierter Report im Format `validation_report_<timestamp>.json` generiert. Die Extraktion erfolgt in < 50 ms pro 1000 Zeilen Quelltext.

- **Multilinguale Textnormalisierung & Silbentrennung**  
  Implementiert eine angepasste Version von `pyphen` für die deutsche Sprache mit einem spezifischen Wörterbuch (`de_DE_frami.dic`). Normalisiert Unicode-Zeichen (z. B. `ä` → `ae` in bestimmten Kontexten) und führt eine automatische Ligatur-Erkennung (`ff`, `fi`, `fl`) durch. Parameter: `--hyphenation-min-chars=5` (minimale Zeichenanzahl vor einem Trennstrich), `--ligature-mode=smart` (nur in Fließtext, nicht in Code-Blöcken).

## Detaillierte Anwendungsschritte

1. **Konfiguration der Pipeline**  
   Erstellen Sie eine `config.yaml` mit folgendem Inhalt:
   ```yaml
   source: "./input/mein_buch.md"
   output: "./output/mein_buch.epub"
   language: de
   metadata:
     title: "Mein erstes E-Book"
     creator: "Max Mustermann"
     identifier: "urn:isbn:978-3-16-148410-0"
   options:
     chapter-depth: 3
     page-break-strategy: auto
     hyphenation: true
   ```
   Führen Sie dann `free_ebook_maker --config config.yaml` aus. Der Parser validiert die YAML-Struktur und startet die Extraktion.

2. **Durchführung der Metadaten-Validierung**  
   Nutzen Sie das Flag `--validate-only`, um die Metadaten vor der eigentlichen Generierung zu prüfen:
   ```bash
   free_ebook_maker --validate-only --metadata-file "path/to/metadata.xml"
   ```
   Das Tool liefert einen Exit-Code `0` bei Erfolg, `1` bei Fehlern (mit detaillierter Ausgabe in `stderr`). Beispiel für eine fehlerhafte ISBN: `ERROR: ISBN-13 checksum mismatch. Expected '0', got '3'.`

3. **Generierung mit benutzerdefinierten Silbentrennungsregeln**  
   Überschreiben Sie die Standard-Silbentrennung mit einer eigenen `.dic`-Datei:
   ```bash
   free_ebook_maker --config config.yaml --hyphenation-dict ./custom_de.dic
   ```
   Das Tool lädt die Datei und führt eine *In-Memory*-Optimierung durch: Es berechnet die *Trennwahrscheinlichkeit* für jede Silbe und speichert sie in einem LRU-Cache (Größe: 10.000 Einträge). Die Verarbeitungszeit steigt dadurch um maximal 2 % im Vergleich zur Standard-Engine.

## Best Practices aus der Industrie

- **Reproduzierbare Builds mit Docker**  
  Stellen Sie sicher, dass die gesamte Pipeline in einem Docker-Container läuft, um Abhängigkeitskonflikte zu vermeiden. Verwenden Sie ein Image mit `python:3.11-slim` und installieren Sie `free_ebook_maker` sowie `pandoc 3.1.12` explizit. Nutzen Sie `docker build --no-cache` und `docker run --rm -v $(pwd):/data` für konsistente Ergebnisse. Dies verhindert Abweichungen durch unterschiedliche Systembibliotheken (z. B. `libxml2` vs. `libxslt`).

- **Inkrementelle Generierung für große Projekte**  
  Bei E-Books mit > 500 Seiten oder > 50 Kapiteln aktivieren Sie die *inkrementelle Pipeline* mit dem Flag `--incremental`. Das Tool speichert Zwischenergebnisse (geparstes DOM, Metadaten-Cache) in einer SQLite-Datenbank (`cache.db`). Bei erneuter Ausführung werden nur geänderte Dateien neu verarbeitet, basierend auf dem `mtime` der Quelldatei. Dies reduziert die Generierungszeit um bis zu 80 % bei wiederholten Builds.
