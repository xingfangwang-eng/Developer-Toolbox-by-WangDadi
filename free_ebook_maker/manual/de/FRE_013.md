# free_ebook_maker - FRE_013

## Übersicht

Dieses Projekt realisiert eine vollständig automatisierte, containerisierte Pipeline zur Generierung barrierefreier EPUB3- und PDF/A-3b-Dokumente aus heterogenen Quellformaten (Markdown, LaTeX, DOCX, HTML). Der Fokus liegt auf deterministischer Build-Reproduktion, Metadaten-Normalisierung und Integration in CI/CD-Workflows.

## Detaillierter Inhalt

### Was ist free_ebook_maker?

free_ebook_maker ist ein deklaratives Build-System, das auf Pandoc 3.6 und WeasyPrint 62.3 aufsetzt. Es löst das Problem der inkonsistenten Typografie und fehlenden Metadaten-Standardisierung bei der Massenproduktion von E-Books. Kern ist ein YAML-basiertes Manifest (`_ebook.yml`), das alle Build-Parameter, Abhängigkeiten und Ausgabeformate definiert.

### Kernfunktionen

- **Dynamische Inhalts-Normalisierung mit XSLT 3.0**  
  Ein vorverarbeitender XSLT-Prozessor (Saxon-HE 12.x) analysiert das Quell-Markdown auf inkorrekte Verschachtelungen, fehlende `alt`-Attribute und nicht-standardkonforme MathML-Blöcke. Die Korrektur erfolgt auf AST-Ebene, bevor Pandoc die Konvertierung startet. Die Laufzeit wird mittels `--timing` in Millisekunden protokolliert.

- **Deterministischer EPUB3-Checksum-Validator**  
  Nach dem Build berechnet ein Python-Skript (SHA-256) über alle `*.xhtml`- und `*.css`-Dateien im EPUB-Container eine kryptographische Signatur. Diese wird in `META-INF/container_signature.xml` abgelegt. Ein optionaler Pre-Commit-Hook validiert die Signatur gegen ein externes PostgreSQL-Backend, um unbemerkte Modifikationen nach dem Build zu erkennen.

- **Automatische PDF/A-3b-Konvertierung mit ICC-Profil-Einbettung**  
  Für PDF-Ausgaben wird der Output nach der WeasyPrint-Generierung durch Ghostscript 10.03.1 geschleust. Dabei wird ein sRGB IEC61966-2.1 ICC-Profil (Größe: 2,1 MB) in den Output eingebettet. Der Parameter `--pdfa` aktiviert zudem die Einbettung der XMP-Metadaten (Dublin Core + PRISM) in den `Document Catalog`.

### So verwenden Sie es

1. **Projekt initialisieren**  
   `free_ebook_maker init --source ./manuskript/ --output ./build/`  
   Dieser Befehl erzeugt eine `_ebook.yml` mit Standardwerten für `epub:version: 3.3`, `pdf:compliance: PDF/A-3b` und `lang: de`. Optional kann `--metadata` übergeben werden, um Autor, ISBN und Rechtehinweise vorzubelegen.

2. **Inhaltsverzeichnis automatisch generieren**  
   `free_ebook_maker toc --depth 3 --section-numbering`  
   Der Befehl parst alle `*.md`-Dateien im Quellverzeichnis, extrahiert `#`- bis `###`-Überschriften und schreibt eine `nav.xhtml` für EPUB3 sowie eine `toc.html` für PDF. Die `--section-numbering`-Flag aktiviert die automatische Nummerierung (z. B. `1.1 Einleitung`).

3. **Build mit Metadaten-Validierung**  
   `free_ebook_maker build --validate --checksum --output-format epub,pdf`  
   Führt den vollständigen Build durch. Die `--validate`-Flag prüft vorab die XML-Well-Formedness aller Quell-Dateien. Nach dem Build wird der Checksum-Validator ausgeführt. Bei Fehlern wird der Exit-Code `1` zurückgegeben und eine detaillierte Log-Datei (`build_errors_YYYYMMDD_HHMMSS.log`) im Build-Verzeichnis abgelegt.

### Beste Praktiken

- **Metadaten-Zentralisierung im Manifest**  
  Hinterlegen Sie Autor, Verlag, ISBN, Rechtehinweise und Sprache ausschließlich in `_ebook.yml`. Vermeiden Sie Metadaten in den Quelldateien – dies reduziert Inkonsistenzen bei der Massenproduktion (z. B. 500+ E-Books mit identischem Corporate Design).

- **CI/CD-Integration mit Signatur-Validierung**  
  Binden Sie `free_ebook_maker build --checksum` in Ihre GitLab-CI oder GitHub Actions ein. Nutzen Sie die PostgreSQL-Anbindung, um bei jedem Push die Signatur des letzten Builds zu prüfen. So stellen Sie sicher, dass kein unautorisierter manueller Eingriff in das generierte EPUB erfolgt ist.
