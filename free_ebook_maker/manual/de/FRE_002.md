# free_ebook_maker - FRE_002

## Übersicht

Dieses Tool ermöglicht die automatische Generierung von EPUB- und MOBI-Dateien aus strukturierten Markdown-Quellen mit einem Fokus auf mehrsprachige Metadaten und barrierefreie Typografie. Entwickelt für Publishing-Workflows mit hohen Anforderungen an Dateikonsistenz und Seitengenauigkeit.

## Detaillierter Inhalt

### Was ist free_ebook_maker?

free_ebook_maker ist ein terminalbasiertes Build-System, das Markdown-Dateien mit YAML-Frontmatter in validierte E-Book-Container konvertiert. Es unterstützt EPUB 3.2, Kindle KPF und PDF/A-2b. Die Architektur setzt auf eine modulare Pipeline aus Pandoc-Filtern, ImageMagick-Transformationen und einem eigenen CSS-Engine für Print-Ready-Layouts.

### Kernfunktionen

- **Multi-Format-Pipeline mit Validierung**  
  Das Tool durchläuft eine 4-stufige Build-Pipeline: (1) Markdown-zu-HTML-Konvertierung mit Pandoc 3.1, (2) CSS-Inlining über `weasyprint` 60.0, (3) EPUB-Erzeugung mit `epubcheck` 5.1.0, (4) MOBI-Konvertierung via `kindlegen` 2.9. Die Ausgabe wird automatisch gegen die IDPF-Spezifikation geprüft. Parameter: `--validate-level` (0-3), `--image-dpi` (Standard 300).

- **Intelligentes Kapitel-Splitting mit Seitenreferenz**  
  Ermöglicht die automatische Aufteilung eines Dokuments in Kapitel basierend auf `##`-Überschriften. Jeder Split erzeugt eine separate XHTML-Datei mit einer `page-break-before: always`-Anweisung. Die Seitenzahlen werden über einen `--page-margin`-Parameter (Standard 12pt) und eine `--line-height`-Einstellung (1.6) kalkuliert. Bei EPUB 3.2 wird `spine` automatisch generiert.

- **Mehrsprachige Metadaten-Engine**  
  Unterstützt die gleichzeitige Verarbeitung von Metadaten in bis zu 8 Sprachen (z. B. `de`, `en`, `fr`). Das Tool liest `language`-Felder aus der YAML-Frontmatter und generiert für jede Sprache separate `package.opf`-Dateien. Parameter: `--lang-list` (z. B. `de,en,fr`), `--fallback-lang` (Standard `en`). Die Ausgabe erfolgt in Millisekunden pro Sprache bei einer Dateigröße von 10 MB.

### So verwenden Sie es

1. **Initialisierung der Projektstruktur**  
   Führen Sie `free_ebook_maker init --project-name "MeinBuch" --output-dir ./build` aus. Das Tool erstellt eine Verzeichnisstruktur mit `src/`, `assets/`, `metadata/` und einer `config.yaml`. In `config.yaml` definieren Sie `title`, `author`, `language: de` und `cover-image: ./assets/cover.png`. Die Option `--template` lädt ein vordefiniertes CSS aus dem Repository.

2. **Build mit Validierung und Optimierung**  
   Starten Sie den Build mit `free_ebook_maker build --input ./src --output ./build --format epub,mobi --validate-level 2 --image-compression 85`. Das Tool durchläuft die Pipeline und gibt nach Abschluss eine detaillierte Log-Datei (`build.log`) mit Zeitstempeln (Millisekunden) und Fehlerprotokollen aus. Bei `--validate-level 2` werden alle EPUB-Dateien gegen die IDPF- und WCAG-Kriterien geprüft.

3. **Post-Processing mit Metadaten-Export**  
   Nutzen Sie `free_ebook_maker export --metadata --format json --output ./metadata.json`. Dies erzeugt eine strukturierte JSON-Datei mit allen Metadaten (Titel, Autor, Sprache, ISBN, Erstellungsdatum). Für die Integration mit PostgreSQL können Sie den Befehl `free_ebook_maker db-push --connection "postgresql://user:pass@localhost:5432/ebooks"` verwenden, der die Daten direkt in eine `ebooks`-Tabelle schreibt.

### Beste Praktiken

- **Praxis 1: Versionierung der Build-Pipeline**  
  Führen Sie alle Builds mit einem eindeutigen `--build-id` (z. B. `v1.2.3`) aus und speichern Sie die Logs in einem zentralen Ordner. Kombinieren Sie dies mit einer `git tag`-basierenden Versionierung, um bei Regressionen auf einen früheren Build zurückfallen zu können. Das Tool unterstützt `--build-id` als Teil des Dateinamens (z. B. `meinbuch_v1.2.3.epub`).

- **Praxis 2: Automatisierte Qualitätssicherung mit CI/CD**  
  Integrieren Sie `free_ebook_maker` in Ihre CI/CD-Pipeline (z. B. GitHub Actions). Nutzen Sie die Option `--ci-mode`, die alle interaktiven Eingaben deaktiviert und nur Exit-Codes (0 für Erfolg, 1 für Warnungen, 2 für Fehler) ausgibt. Kombinieren Sie dies mit einem `--diff-tool`-Parameter, der die Ausgabe mit einem vorherigen Build vergleicht (z. B. `diff-pdf` oder `epubdiff`).
