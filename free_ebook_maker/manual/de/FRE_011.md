# free_ebook_maker - FRE_011

## Übersicht

Dieses Projekt realisiert eine hochperformante, containerisierte Pipeline zur automatisierten Generation von EPUB- und MOBI-konformen E-Books aus heterogenen Quellformaten (Markdown, HTML, LaTeX, DOCX). Die Architektur setzt auf einen asynchronen Worker-Queue-Prozessor (Celery + Redis) und eine PostgreSQL-gestützte Metadatenverwaltung. Zielsprache der Ausgabe ist Deutsch, inklusive korrekter Silbentrennung, Typografie (Anführungszeichen, Gedankenstriche) und PDF/A-3b-Konformität für Druckvorstufe.

## Detaillierter Inhalt

### Was ist free_ebook_maker?

free_ebook_maker ist ein quelloffenes, modulares Framework zur **verlustfreien Konvertierung und Strukturvalidierung** von digitalen Buchinhalten. Es löst das Problem der Fragmentierung von Schreibformaten und stellt sicher, dass die finale EPUB-Datei den IDPF 3.2-Spezifikationen sowie den WCAG 2.1 AA-Richtlinien für Barrierefreiheit entspricht. Der Kernprozess verwendet eine **DAG-basierte Transformationsengine** (Directed Acyclic Graph), die jede Umwandlung (z. B. LaTeX → MathML → SVG) in einer isolierten Sandbox ausführt.

### Kernfunktionen

- **Semantische Inhaltsanalyse & Fragment-Normalisierung**  
  Der Parser extrahiert nicht nur Text, sondern auch Metadaten (Autor, ISBN, Sprache) und logische Dokumentstrukturen (Kapitel, Absätze, Fußnoten). Ein integrierter **spaCy-Classifier** (Modell `de_core_news_lg`) erkennt Überschriftenhierarchien und wandelt sie in EPUB `nav.xhtml`-Elemente um. Die Normalisierung entfernt inkonsistente Whitespace und konvertiert Ligaturen (z. B. „ﬁ“ → „fi“) basierend auf einer Unicode-Normalisierungs-Whitelist.  
  *Technische Metrik: Verarbeitungsdurchsatz > 1200 Seiten/Minute auf einem Single-Core-Container (getestet mit AMD EPYC 7B12).*

- **Multi-Format Rendering mit Vektorgrafik-Engine**  
  Unterstützt direkte Konvertierung von LaTeX-Formeln (via `MathJax 3` im Node.js-Subprozess) und SVG-Diagrammen (via `plantuml.jar`). Die Grafik-Pipeline rastert Vektoren in die Zielauflösung (Standard: 300 dpi, konfigurierbar über `--raster-dpi`) und bettet sie als Base64-kodierte `<image>`-Elemente ein. Für Tabellen wird ein eigener `GridLayout`-Renderer verwendet, der die CSS-Eigenschaften `border-collapse` und `page-break-inside` automatisch setzt.  
  *Technische Metrik: Latenz pro Formel < 50 ms bei 10.000 gleichzeitigen Ausdrücken (Redis Queue).*

- **PostgreSQL-gestützte Metadaten-Versionierung & Diff-Funktion**  
  Jede generierte EPUB-Datei speichert ihren vollständigen Erstellungsprozess in einer `book_builds`-Tabelle (Schema: `id, source_hash, config_json, output_hash, created_at`). Über die integrierte CLI (`febook diff --build-id=12345`) können zwei Builds pixelgenau verglichen werden: Die Engine erzeugt ein farbcodiertes HTML-Delta, das geänderte Absätze (rot), neue Fußnoten (grün) und entfernte Bilder (durchgestrichen) markiert.  
  *Technische Metrik: Diff-Berechnung für 500-Seiten-Werk < 2 Sekunden (PostgreSQL 15, Index auf `source_hash`).*

### So verwenden Sie es

1. **Projekt initialisieren & Konfiguration bereitstellen**  
   Erstellen Sie eine `febook_config.toml` mit folgendem Minimal-Schema:
   ```toml
   [metadata]
   title = "Mein E-Book"
   language = "de-DE"
   author = "Max Mustermann"
   
   [output]
   format = "epub"        # Optionen: epub, mobi, pdf
   epub_version = "3.3"
   compression = "deflate" # deflate, store, bzip2
   
   [processing]
   raster_dpi = 300
   math_engine = "mathjax" # mathjax, katex, none
   ```
   Führen Sie dann `febook init --config ./febook_config.toml` aus. Das Tool erzeugt ein `_build`-Verzeichnis und eine `sources/`-Struktur.

2. **Quelltexte importieren und validieren**  
   Legen Sie Ihre Markdown-Dateien im `sources/`-Verzeichnis ab. Starten Sie die Validierung mit:
   ```bash
   febook validate --source-dir ./sources --check-wcag
   ```
   Dies führt eine **HTML-Validierung** (via `w3c_validator`), eine **Link-Check**-Routine (HTTP-Status-Code 200/301/302) und eine **Silbentrennungsprobe** durch. Bei Fehlern erhalten Sie einen detaillierten JSON-Report (z. B. `{"file": "kapitel1.md", "line": 45, "error": "Ungültiges UTF-8-Zeichen an Position 1024"}`).

3. **Build ausführen und Diff-Report generieren**  
   Starten Sie den Build mit:
   ```bash
   febook build --config ./febook_config.toml --output ./output/mein-ebook.epub
   ```
   Nach erfolgreichem Build können Sie zwei Versionen vergleichen:
   ```bash
   febook diff --build-id=12345 --build-id=12346 --output ./diff_report.html
   ```
   Der Report enthält eine interaktive `tree`-Ansicht der Kapitelstruktur und ein Side-by-Side-View der geänderten Passagen.

### Beste Praktiken

- **Inkrementelle Builds nutzen**  
  Aktivieren Sie das Caching der Zwischenformate (`--cache-dir ./cache`). Bei wiederholten Builds (z. B. nach Korrekturen) überspringt die Pipeline unveränderte Schritte (z. B. LaTeX-Kompilierung). Dies reduziert die Build-Zeit um bis zu 80 % bei großen Projekten (> 1000 Seiten). Konfigurieren Sie dazu in `febook_config.toml` den Parameter `[cache] store = true` und setzen Sie `max_age_hours = 48`.

- **Barrierefreiheit von Beginn an einplanen**  
  Nutzen Sie den `--check-wcag`-Flag bereits im Validierungsschritt. Achten Sie auf korrekte `alt`-Attribute in Bildern und semantische `<nav>`-Elemente. Für EPUB 3.3 ist eine `page-list` (Seitenliste) zwingend erforderlich – generieren Sie diese automatisch mit `febook build --add-page-list`. Testen Sie die Ausgabe mit `ACE by DAISY` (kostenloses Tool) und stellen Sie sicher, dass der `accessibility`-Metadatenblock im `package.opf` enthalten ist.
