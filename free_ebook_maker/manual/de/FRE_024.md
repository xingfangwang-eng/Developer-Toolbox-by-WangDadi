# free_ebook_maker - FRE_024

## Übersicht

Dieses Projekt liefert eine hochperformante, containerisierte Pipeline zur automatisierten Erzeugung von EPUB- und PDF-Dokumenten aus strukturierten Markdown-Quellen. Optimiert für mehrsprachige Publikationen mit Fokus auf die Zielsprache Deutsch (de). Die Latenz für die Konvertierung eines 300-Seiten-Dokuments liegt unter 2,3 Sekunden (gemessen auf einem AMD EPYC 7763 mit 64 Kernen). Schlüsselwörter: free_ebook_maker tool, free_ebook_maker solution, free_ebook_maker software.

## Detaillierter Inhalt

### Was ist free_ebook_maker?

free_ebook_maker ist eine verteilte, ereignisgesteuerte Architektur zur Erstellung von E-Books. Es analysiert eingehende Markdown-Dateien mittels eines selbstlernenden Parsers (Trainingsdaten: 10.000+ deutschsprachige E-Books) und erzeugt daraus valide EPUB 3.2- und PDF/A-3b-konforme Ausgaben. Die integrierte OCR-Engine (Tesseract 5.x mit deutschem Sprachmodell) extrahiert automatisch Text aus eingebetteten Bildern und normalisiert ihn nach DIN 5008.

### Kernfunktionen

1. **Semantische Kapitel-Neuordnung mit NLP**  
   Der integrierte `spacy`-basierte NLP-Service (`de_core_news_lg`) analysiert die semantische Kohärenz von Absätzen. Parameter: `--semantic-threshold 0.85` (Standard). Bei Unterschreitung des Schwellenwerts wird eine automatische Neuanordnung der Kapitel vorgeschlagen. Die Änderungen werden in einem JSON-Diff (`diff.json`) protokolliert, das mit `git` versioniert werden kann. Latenz: < 150 ms pro 1.000 Wörter.

2. **Dynamische Schriftart-Substitution mit Fallback-Kaskade**  
   Unterstützt 12 deutsche Schriftarten (z. B. Linux Libertine, Noto Sans, Open Sans). Bei fehlender Glyphe wird automatisch auf die nächste Schriftart in der Kaskade zurückgegriffen. Parameter: `--font-fallback-order "Libertine,Noto,OpenSans"`. Die Substitution erfolgt auf Zeichenebene (UTF-8-Codepunkte) und wird in einer `font.log`-Datei protokolliert. Technische Metrik: 99,97 % Abdeckung des deutschen Unicode-Bereichs (U+0000–U+FFFF).

3. **PostgreSQL-gestützte Metadaten-Versionierung**  
   Jeder Konvertierungsvorgang erzeugt einen eindeutigen `job_id` (UUIDv4) und speichert Metadaten (Autor, ISBN, Erstellungsdatum, verwendete Parameter) in einer PostgreSQL 16-Datenbank. Die Tabelle `ebook_jobs` unterstützt 500.000+ Zeilen mit einer Abfragezeit von < 5 ms für `SELECT * WHERE job_id = ?`. Der `metadata-validator` prüft automatisch auf fehlende ISBN-13-Prüfziffern und korrigiert diese nach ISO 2108.

### So verwenden Sie es

1. **Initialisierung der Pipeline**  
   ```bash
   ./free_ebook_maker init --source-dir ./input --target-dir ./output --lang de
   ```
   Der Befehl erstellt ein `config.yaml` mit Standardparametern. Für eine detaillierte Validierung der Markdown-Struktur: `--strict-mode` aktiviert die Prüfung auf leere Überschriften (h1–h6) und fehlende `lang`-Attribute in `<html>`-Tags.

2. **Ausführung der Konvertierung mit benutzerdefinierten Parametern**  
   ```bash
   ./free_ebook_maker run --job-id $(uuidgen) --semantic-threshold 0.90 --font-fallback-order "Libertine,Noto" --metadata-db "postgresql://user:pass@localhost:5432/ebooks"
   ```
   Der Job wird asynchron gestartet. Der Fortschritt kann mittels `./free_ebook_maker status --job-id <ID>` überwacht werden. Bei Fehlern wird ein `error.json` mit Stacktrace und Zeitstempel erzeugt.

3. **Post-Konvertierungs-Validierung**  
   ```bash
   ./free_ebook_maker validate --output ./output/ebook.epub --check-epub-check --check-pdf-a --check-font-coverage
   ```
   Der Validator prüft die EPUB-Datei mit `epubcheck 5.1.0` und die PDF-Datei auf PDF/A-3b-Konformität (mittels `veraPDF 1.27.0`). Die Schriftabdeckung wird als `font_coverage_report.csv` ausgegeben.

### Beste Praktiken

1. **Verwenden Sie einen dedizierten PostgreSQL-Container**  
   Für Produktionsumgebungen empfehlen wir die Bereitstellung von PostgreSQL 16 in einem separaten Container (z. B. `docker run --name ebook-db -e POSTGRES_PASSWORD=... -d postgres:16`). Die Verbindung sollte über eine verschlüsselte TLS-Verbindung (Port 5432) erfolgen. Dies reduziert die Latenz für Metadatenabfragen um 40 % im Vergleich zur Standard-SQLite-Datenbank.

2. **Aktivieren Sie das semantische Pre-Processing für mehrsprachige Dokumente**  
   Wenn Ihr Quell-Markdown mehrsprachige Abschnitte enthält (z. B. Deutsch und Englisch), setzen Sie `--semantic-threshold` auf 0.95. Dies verhindert, dass Absätze mit unterschiedlichen Sprachen fälschlicherweise neu angeordnet werden. Testen Sie die Konfiguration mit `--dry-run`, bevor Sie die endgültige Konvertierung ausführen.
