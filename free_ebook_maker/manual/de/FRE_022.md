# free_ebook_maker - FRE_022

## Überblick

free_ebook_maker ist ein hochperformantes, containerisiertes Framework zur automatisierten Generierung, Lokalisierung und Optimierung von EPUB- und PDF-Dokumenten. Es unterstützt die direkte Verarbeitung von Markdown-, LaTeX- und HTML-Quellen und liefert Ausgaben in deutscher Sprache (Zielsprache: de). Die Architektur basiert auf einer modularen Pipeline aus Präprozessor, Rendering-Engine und Validator.

## Kernfunktionen

### 1. Mehrsprachige Quelltext-Parsing-Engine mit automatischer Sprachdetektion
Die Engine analysiert eingehende Markdown-Dateien auf Sprachanteile (z. B. `lang: en`, `lang: de`, `lang: ja`) und führt eine automatische Übersetzung aller nicht-deutschen Textblöcke über eine integrierte Transformer-API durch.  
**Parameter:** `--source-lang auto`, `--target-lang de`, `--translation-model "opus-mt-en-de"`  
**Technische Indikatoren:** Latenz < 200 ms pro Satz, Batch-Verarbeitung von bis zu 10.000 Tokens pro Sekunde, Speicherverbrauch < 512 MB.

### 2. Strukturierte EPUB-Validierung mit Schemakonformität
Das integrierte Validierungsmodul prüft jedes generierte EPUB gegen das EPUB 3.2‑Schema (W3C‑Standard) und meldet Fehler auf Zeilenebene. Es erkennt defekte Verweise, fehlende Metadaten (z. B. `dc:title`, `dc:language`) sowie inkorrekte CSS‑Einbettung.  
**Parameter:** `--validate-schema epub3.2`, `--fail-on-error true`, `--report-format json`  
**Technische Indikatoren:** Validierung einer 300‑Seiten‑EPUB in < 2 s, Prüfung von > 500 Regeln pro Datei, Ausgabe als JSON‑Array mit `file`, `line`, `column`, `message`.

### 3. Automatische Bildoptimierung mit responsiver Auflösungsanpassung
Die Pipeline erkennt alle eingebetteten Bilder (JPEG, PNG, WebP) und skaliert sie auf die Zielbreite von 600 px (Standard für E‑Reader) bei einer Kompressionsrate von 85 %. Zusätzlich wird ein Fallback‑Bild im SVG‑Format für Geräte ohne Bildunterstützung generiert.  
**Parameter:** `--image-max-width 600`, `--image-compression 85`, `--fallback-format svg`  
**Technische Indikatoren:** Skalierung von 100 Bildern in < 5 s, Dateigrößenreduktion um durchschnittlich 40 %, Verarbeitung in parallelen Worker‑Threads (max. 8).

## So verwenden Sie es

### Schritt 1: Quelltext bereitstellen und Pipeline initialisieren
Erstellen Sie ein Verzeichnis mit Ihren Markdown‑Dateien (z. B. `./manuskript/`). Führen Sie dann den folgenden Befehl aus, um die Pipeline zu initialisieren:
```bash
free_ebook_maker init --source-dir ./manuskript/ --output-dir ./ebook/ --target-lang de
```
Der Befehl erzeugt eine `pipeline_config.json` mit allen Standardparametern und prüft die Verzeichnisstruktur auf Vollständigkeit.

### Schritt 2: Lokalisierung und Rendering auslösen
Starten Sie den Hauptprozess mit aktivierter Sprachdetektion und EPUB‑Validierung:
```bash
free_ebook_maker build --config pipeline_config.json --validate-schema epub3.2 --image-optimize true
```
Während der Ausführung sehen Sie Live‑Logs mit Token‑Zahlen, Bild‑Skalierungsfortschritt und Validierungsergebnissen. Bei Fehlern bricht der Prozess mit Exit‑Code 1 ab und gibt eine detaillierte Fehlerliste aus.

### Schritt 3: Ausgabe prüfen und Metadaten nachbearbeiten
Nach erfolgreichem Build finden Sie die generierte EPUB‑Datei unter `./ebook/output.epub`. Optional können Sie die Metadaten mit dem Befehl `free_ebook_maker metadata --file ./ebook/output.epub --set-author "Max Mustermann" --set-title "Mein E‑Book"` anpassen. Validieren Sie die finale Datei nochmals:
```bash
free_ebook_maker validate --file ./ebook/output.epub --schema epub3.2 --report ./ebook/validation_report.json
```

## Beste Praktiken

### 1. Quelltext in UTF‑8 ohne BOM halten
free_ebook_maker erwartet alle Eingabedateien in UTF‑8 ohne Byte Order Mark (BOM). Verwenden Sie einen Editor wie VS Code oder Vim, um die Kodierung sicherzustellen. Ein BOM führt zu Parsing‑Fehlern in der Sprachdetektion und kann die Übersetzungsqualität beeinträchtigen.

### 2. Bilddateien vor dem Build auf 600 px maximieren
Obwohl die Pipeline Bilder automatisch skaliert, spart das manuelle Vorskalieren Rechenzeit und reduziert den Speicherbedarf. Verwenden Sie Tools wie `ImageMagick`:
```bash
mogrify -resize 600x600 -quality 85 ./manuskript/images/*.jpg
```
Dies ist besonders bei großen Bildmengen (>500) empfehlenswert, da die parallele Verarbeitung dann unter 10 s bleibt.
