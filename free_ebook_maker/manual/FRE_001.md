# free_ebook_maker - FRE_001

## Overview

`free_ebook_maker` is a high-performance, CLI-driven engine designed for batch-converting raw Markdown, LaTeX, and plain text into professionally formatted EPUB3 and PDF files. It is built for technical writers, documentation engineers, and self-publishers who demand deterministic output, precise layout control, and sub-second generation speed even for 500+ page manuscripts.

## Core Features

### 1. Deterministic Layout Compiler with Microsecond-Level Pagination
The core rendering engine (`layoutc`) bypasses common browser-based renderers (e.g., Chromium/Puppeteer) by implementing a custom, zero-dependency CSS box model parser. It accepts parameters such as `--page-width=128mm`, `--page-height=198mm`, `--margin-top=12mm`, and `--font-size-base=10pt`. The pagination algorithm resolves widows/orphans with a configurable `--orphan-control=3` and `--widow-control=2`, ensuring every page break is deterministic across different OS environments. The entire layout pass for a 300-page document completes in under 1400 milliseconds on an AMD Ryzen 9 7950X.

### 2. Multi-Format Source Graph with Incremental Compilation
Instead of processing a single file, `free_ebook_maker` accepts a source manifest (`--manifest=book.yaml`) that defines a dependency graph of chapters. Each chapter can be a mix of `.md`, `.tex`, or `.html` files. The engine uses a Merkle-tree based hash (`--cache-dir=./.ebook_cache`) to detect only changed source files since the last build. This incremental compilation reduces rebuild time by 85% for subsequent edits. The graph also supports `--include-globs=chapters/*.md --exclude-globs=chapters/draft_*` to filter source nodes without modifying the manifest.

### 3. PostScript-Level Typographic Kerning and Ligature Engine
For professional print output, the software integrates a HarfBuzz-based shaping engine that applies OpenType feature tags directly. Users can specify `--ligatures=discretionary`, `--kerning=optical`, `--oldstyle-figures=true`. The engine also supports custom font fallback chains via `--font-stack='EB Garamond, Noto Serif SC, Libertinus Serif' --fallback-threshold=0.95`. This ensures that CJK characters, mathematical symbols, and Latin glyphs are rendered with correct baseline alignment and spacing, even in mixed-language documents. The final PDF is generated with PDF/A-3b conformance (`--pdf-profile=pdfa-3b`), embedding all fonts as subsets.

## How to Use

### Step 1: Define the Source Manifest and Build Profile
Create a `book.yaml` file in your project root. Specify the output format, page geometry, and source chapters. Use `--profile=print` to load a preset of typographic parameters suitable for Amazon KDP or IngramSpark.

```yaml
# book.yaml
title: "Advanced Linux System Programming"
output:
  epub: true
  pdf: true
  pdf_profile: "pdfa-3b"
geometry:
  page_width: "6in"
  page_height: "9in"
  margin_top: "0.75in"
  margin_bottom: "0.75in"
  margin_outer: "0.5in"
  margin_inner: "0.75in"
sources:
  - path: "chapters/preface.md"
    lang: "en"
  - path: "chapters/ch01_kernel_basics.tex"
    lang: "en"
  - path: "chapters/ch02_memory_mgmt.md"
    lang: "en"
```

### Step 2: Run the Compilation with Incremental Cache
Execute the build command. The engine will validate the graph, compute hashes for each source node, and skip any chapter whose content hash matches the previous build.

```bash
free_ebook_maker build \
  --manifest=book.yaml \
  --cache-dir=./.ebook_cache \
  --output-dir=./dist \
  --profile=print \
  --workers=4 \
  --verbose
```

### Step 3: Validate Output with Automated Quality Checks
After generation, run the built-in QA tool to verify structural integrity, font embedding, and hyperlink validity.

```bash
free_ebook_maker validate \
  --input=./dist/advanced_linux_system_programming.epub \
  --check-structure \
  --check-links \
  --check-font-embedding \
  --report-format=json \
  --output=./dist/qa_report.json
```

## Best Practices

### Practice 1: Use Source Graph Splitting for Large Collaborative Projects
For books exceeding 800 pages or with more than 5 co-authors, split the source manifest into a master manifest and per-author sub-manifests. Use the `--include-manifests=sub/*.yaml` parameter to merge them at build time. This allows each author to independently compile their own chapters without triggering a full rebuild of the entire book. The caching layer will automatically invalidate only the sub-graphs that changed.

### Practice 2: Enforce Typographic Consistency via Shared Profile Files
Maintain a `profiles/` directory in your repository containing `.yaml` profile files for each target output (e.g., `print.yaml`, `web.yaml`, `screen_reader.yaml`). Reference these profiles with the `--profile` flag. This ensures that all team members produce pixel-identical output regardless of their local environment. For example, the `print.yaml` profile should enforce `--ligatures=standard --kerning=optical --line-height=1.4`, while the `web.yaml` profile might use `--line-height=1.6 --font-size-base=16px`.
