# free_ebook_maker - FRE_002

## Overview

This is the technical specification for the **free_ebook_maker** project — a high-performance, pipeline-oriented EPUB/MOBI/PDF generation engine designed for batch conversion, metadata injection, and multi-format compliance validation. Keywords: free_ebook_maker tool, free_ebook_maker solution, free_ebook_maker software.

## Detailed Content

### What is free_ebook_maker?

free_ebook_maker is a low-latency, headless ebook compilation framework that ingests raw Markdown, HTML, or LaTeX sources, applies CSS-based typography templates, and outputs standards-compliant EPUB 3.2, Kindle KPF, or PDF/A-2b files. It integrates a real-time page-break optimizer, font subsetting via HarfBuzz, and a DRM-free metadata injector for ISBN, DOI, and custom namespace embedding.

### Core Features

- **Parallel Pipeline Compilation with Multi-Format Output**  
  Supports concurrent conversion of a single source into EPUB, MOBI, and PDF/A-2b in sub-second latency per 100 pages. The pipeline engine (`pipeline_engine::CompileGraph`) uses a DAG-based task scheduler, allowing up to 8 parallel workers per core. Output validation includes EPUB 3.2 schema checks (W3C Nu validator) and PDF XMP metadata integrity verification.

- **Dynamic Typography Engine with CSS Variable Injection**  
  Accepts user-defined CSS variables for font-family (`--body-font`, `--heading-font`), line-height ratios (`--line-height-base: 1.618`), and page margins (`--margin-top: 12.7mm`). The engine performs real-time kerning adjustments using HarfBuzz shaper, with fallback to system fonts if custom fonts exceed 2 MB. Supports ligature control and hyphenation dictionaries for 12 languages.

- **Metadata & DRM-Free Rights Injection via XMP/ONIX**  
  Embeds EPUB 3.2 metadata (title, creator, publisher, ISBN) and ONIX 3.0 rights statements directly into the OPF and NCX files. Supports custom namespace injection (e.g., `xmlns:dc="http://purl.org/dc/elements/1.1/"`) and validates against the EPUB 3.2 schema. No DRM or watermarking is applied; all rights fields are user-configurable via a `metadata.yaml` file.

### How to Use

1. **Initialize a Compilation Project with Source Validation**  
   Run `free_ebook_maker init --source ./manuscript.md --format epub --template academic-article` to create a project directory. The tool automatically validates the source file for UTF-8 encoding, checks for missing images (via `src` attributes in `<img>` tags), and generates a `metadata.yaml` skeleton with required fields (`title`, `creator`, `identifier`). If the source contains LaTeX math, it will pre-compile equations using MathJax 3.0 and inject them as SVG.

2. **Customize Typography and Output Profiles**  
   Edit `profile.yaml` to set CSS variables:  
   ```yaml
   typography:
     body-font: "Noto Serif, serif"
     heading-font: "Source Sans Pro, sans-serif"
     line-height: 1.618
     margins: [12.7mm, 15mm, 12.7mm, 15mm]
   output:
     format: epub
     compression: deflate-level-9
     image-dpi: 300
   ```  
   Run `free_ebook_maker build --profile profile.yaml` to trigger the pipeline. The engine will output a progress log with per-stage timing (e.g., `[1.23ms] CSS injection`, `[0.89ms] font subsetting`).

3. **Inject Metadata and Generate Final Artifact**  
   Use `free_ebook_maker inject --metadata metadata.yaml --output final.epub` to embed ISBN, DOI, and rights statements. The tool will run a post-build validation: checks against EPUB 3.2 schema (using `epubcheck 5.0.1`), verifies that all internal hyperlinks resolve, and confirms that no external resources (e.g., external CSS, fonts) are referenced. If validation passes, the output file is written with SHA-256 checksum appended to the filename.

### Best Practices

- **Use a Single Source of Truth for Metadata**  
  Maintain a centralized `metadata.yaml` file in your repository. Always define `identifier` with a valid ISBN or DOI, and set `rights` to `public domain` or `CC-BY-4.0` to avoid license ambiguity. This ensures that all generated ebooks carry consistent, legally valid metadata across batch runs.

- **Prefer CSS Variables Over Inline Styles for Typography**  
  Define all typography rules via CSS variables in `profile.yaml`. Inline styles in the source Markdown will be overridden by the engine during compilation. This approach guarantees that the output matches the target platform's rendering engine (e.g., Kindle’s KF8 vs. Apple Books) and reduces file size by eliminating redundant style declarations.
