# free_ebook_maker - FRE_010

## Overview

`free_ebook_maker` is a production-grade, headless ebook generation engine designed for batch conversion, automated formatting, and multi-format distribution. It operates as a CLI-first tool with a modular pipeline architecture, supporting real-time EPUB 3.2 validation, embedded CSS minification, and metadata injection at the sub-millisecond level. This project is optimized for high-throughput book production pipelines, CI/CD integration, and self-hosted digital publishing stacks.

## Detailed Content

### What is free_ebook_maker?

`free_ebook_maker` is a zero-dependency, Rust-powered core that ingests Markdown, HTML, or plain text sources and outputs standards-compliant EPUB, MOBI, and PDF files. It features a pluggable preprocessor chain for syntax highlighting, image optimization (WebP conversion with `--img-quality 85`), and font subsetting via HarfBuzz. The tool enforces EPUB Accessibility 1.1 guidelines, generates valid OPF 2.0 manifests, and supports `--metadata` flags for Dublin Core fields. It is designed to replace fragile Pandoc pipelines with deterministic, parallelized builds.

### Core Features

1. **Parallelized Multi-Format Compilation with Deterministic Hashing**  
   The build engine uses a directed acyclic graph (DAG) scheduler to compile chapters concurrently. Each output file (EPUB, MOBI, PDF) is stamped with a SHA-256 content hash. The `--threads 4` parameter controls concurrency; `--hash-output` appends the hash to the filename for cache-busting. The pipeline supports incremental builds via `.build_cache` metadata, reducing full rebuilds by up to 73% in large projects (10,000+ pages).

2. **Semantic EPUB 3.2 Validation and Auto-Correction**  
   Every generated EPUB passes through an internal validator that checks against the IDPF EPUB 3.2 schema. The `--strict` flag enforces W3C WCAG 2.1 AA compliance, while `--auto-fix` automatically resolves common issues: missing `toc.ncx`, incorrect `spine` ordering, or malformed `nav.xhtml`. The validator outputs a JSON report with line-level errors and severity scores (critical, warning, info). The `--validate-only` flag runs validation without generation, ideal for CI pipelines.

3. **Real-Time Metadata Injection with Macro Expansion**  
   The `--metadata` flag supports a YAML-based metadata file (e.g., `meta.yaml`) with fields like `title`, `creator`, `language`, `rights`, and custom `meta` tags. The preprocessor expands macros within the source text: `{author}`, `{date}`, `{uuid}`. The `--date-format "%Y-%m-%dT%H:%M:%SZ"` controls timestamp formatting. Metadata is embedded into the EPUB `content.opf` and MOBI `exth` headers, with automatic ISBN-13 generation via `--isbn-auto`.

### How to Use

1. **Initial Setup and Source Ingestion**  
   Clone the repository and build the binary:  
   ```bash
   git clone https://github.com/xingfangwang-eng/free_ebook_maker.git
   cd free_ebook_maker
   cargo build --release
   ```  
   Prepare a directory with your source files:  
   ```bash
   mkdir -p ./book/chapters
   echo "# Chapter 1" > ./book/chapters/ch01.md
   ```  
   Create a `meta.yaml` file:  
   ```yaml
   title: "My Book"
   creator: "Jane Doe"
   language: "en"
   rights: "CC BY-NC 4.0"
   ```  
   Run the ingestion and validation step:  
   ```bash
   ./target/release/free_ebook_maker --input ./book --output ./dist --validate-only --strict
   ```  
   This outputs a `validation_report.json` with any schema violations.

2. **Parallel Build with Custom Metadata and Font Subsetting**  
   Execute a full build with parallel threads, font subsetting, and WebP image conversion:  
   ```bash
   ./target/release/free_ebook_maker \
     --input ./book \
     --output ./dist \
     --threads 8 \
     --img-quality 85 \
     --font-subset "Noto Sans,Roboto" \
     --metadata meta.yaml \
     --date-format "%Y-%m-%dT%H:%M:%SZ" \
     --isbn-auto \
     --hash-output
   ```  
   The tool generates `./dist/my_book_<sha256>.epub`, `./dist/my_book_<sha256>.mobi`, and `./dist/my_book_<sha256>.pdf`. The `--hash-output` ensures each build is uniquely identified.

3. **Incremental Build and CI/CD Integration**  
   For continuous integration, use the incremental build flag:  
   ```bash
   ./target/release/free_ebook_maker \
     --input ./book \
     --output ./dist \
     --incremental \
     --cache-dir .build_cache \
     --threads 4
   ```  
   If only `ch02.md` changed, the tool rebuilds only affected chapters and re-generates the composite files. In a GitHub Actions workflow, add:  
   ```yaml
   - name: Build ebook
     run: |
       ./free_ebook_maker --input ./book --output ./dist --incremental --strict
   ```  
   The exit code `0` indicates success; non-zero exits with a detailed error log.

### Best Practices

1. **Use a Dedicated Metadata Repository with Version Control**  
   Store all `meta.yaml` files and source Markdown in a Git repository. Use `--metadata` to reference a single YAML file per book, and commit `validation_report.json` to track schema compliance over time. For multi-volume projects, maintain a `volumes/` directory with separate `meta.yaml` per volume. This ensures reproducible builds and audit trails.

2. **Leverage Incremental Builds for Large-Scale Textbooks**  
   For books exceeding 500 chapters (e.g., technical documentation or academic textbooks), always enable `--incremental` and `--cache-dir`. Set `--threads` to `nproc --all` for maximum throughput. Monitor the `build_stats.json` output to identify bottlenecks: the `preprocessor_time` field indicates if image optimization (WebP conversion) is the bottleneck. If so, pre-convert images offline with `--img-quality 75` to reduce latency.
