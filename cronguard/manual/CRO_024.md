# cronguard - CRO_024

Cronguard is a lightweight, developer-first cron job observability and alerting engine built for distributed systems. It doesn’t just tell you when a job fails — it surfaces silent failures, execution drift, and scheduling conflicts before they escalate.

---

## 3 Hardcore Features That Set Cronguard Apart

### 1. Execution Drift Detection with Adaptive Baselines
Cronguard automatically learns the historical execution window of every cron job (e.g., “this job normally finishes in 1.2–2.4 seconds”). If a job finishes 3× faster or slower than its learned baseline — even if it exits with code 0 — you get a precision alert. No false positives from normal variance.

### 2. Multi-Instance Job Locking via Distributed Lease
When the same cron job runs across multiple replicas (K8s pods, EC2 instances), Cronguard enforces a distributed lease mechanism using a lightweight consensus layer. Only one instance executes the job per schedule tick. No duplicate runs, no race conditions, no manual fencing.

### 3. Real-Time Dependency Graph with Backpressure Visualization
Define job DAGs (e.g., `job_b` depends on `job_a`). Cronguard renders a live dependency graph and detects backpressure: if `job_a` runs late, it auto-calculates the cascading delay on `job_b` and `job_c`, then surfaces the impact in milliseconds. You see the blast radius before the alert fires.

## 3-Step Production Setup (No Fluff)

### Step 1: Instrument Your Cron Jobs with the Agent SDK
Install the Cronguard agent via a single binary or container sidecar. Wrap your existing cron command with the SDK wrapper:

```bash
cronguard run --job-id "data_pipeline_ingest" --schedule "*/5 * * * *" -- ./ingest.sh
```

The agent captures exit codes, wall-clock time, and stdout/stderr tail. It pushes telemetry to the Cronguard backend via gRPC with automatic retry and backoff. No config file needed — environment variables override defaults.

### Step 2: Define Alert Policies in the Dashboard (or via YAML)
Go to the Cronguard dashboard → **Alerts** → **Create Policy**. Or push a `cronguard-policy.yaml` to your repo:

```yaml
policies:
  - name: "ingest_slowdown"
    condition: "execution_duration > 3.5 * baseline_avg"
    severity: "warning"
    channels: ["slack", "pagerduty"]

  - name: "ingest_missing"
    condition: "no_heartbeat_for > 90s"
    severity: "critical"
    channels: ["pagerduty"]
```

The system evaluates conditions every 10 seconds. No polling. No SQL.

### Step 3: Connect to Your Observability Stack via Webhook or Export
Cronguard exposes a Prometheus-compatible `/metrics` endpoint and a structured webhook sink. Forward alerts to Datadog, Grafana OnCall, or Opsgenie with a single toggle:

```bash
cronguard config set webhook.url "https://hooks.datadoghq.com/api/v2/webhook/..."
cronguard config set webhook.headers "X-API-Key: $DD_API_KEY"
```

Each webhook payload includes the job ID, execution context, and a precomputed impact score (0–100). Your on-call engineer knows exactly how bad it is before opening Slack.

## 2 Industry Best Practices for Cronguard

### 1. Always Set a Heartbeat Timeout (Even for Idempotent Jobs)
Many teams assume idempotent jobs are safe to miss. They’re not — a missed cron can mean stale data, broken SLAs, or silent drift. With Cronguard, set a heartbeat timeout equal to 1.5× the job’s maximum historical interval. If the heartbeat misses, the system auto-escalates to critical after 2 consecutive failures. This catches both crashed pods and network partitions.

### 2. Use Staggered Schedules to Avoid Thundering Herd
When you have 50+ cron jobs all firing at `:00`, the database gets hammered. Cronguard’s **Schedule Optimizer** (Pro feature) analyzes your job timing and suggests offset shifts (e.g., `:03`, `:17`, `:42`). This reduces peak load by up to 40% and prevents cascading timeouts. Apply the changes with a single click — no manual cron table editing.
