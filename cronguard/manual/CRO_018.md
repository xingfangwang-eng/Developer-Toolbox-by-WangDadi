# cronguard - CRO_018

## Overview

cronguard is a production-grade cron job monitoring, failover, and observability engine designed for high-stakes SaaS environments. It eliminates silent failures, detects scheduling drift, and provides real-time alerting across distributed cron workers. Built for DevOps teams who refuse to let a missed job cascade into a revenue event.

## Core Features

### 1. Predictive Execution Drift Detection
cronguard continuously profiles each cron job’s historical execution time, resource consumption, and completion latency. Using a lightweight statistical model, it predicts when a job will drift outside its acceptable SLA window — and triggers a preemptive alert before the timeout fires. This prevents cascading delays in dependent pipeline stages.

### 2. Multi-Region Graceful Failover with Idempotency Locks
When a cron job fails in one region, cronguard automatically reassigns execution to a healthy replica using distributed idempotency locks (backed by Redis or PostgreSQL advisory locks). No duplicate executions, no state corruption. The failover is atomic and respects the original job’s concurrency policy.

### 3. Real-Time Execution Trace with Structured Logs
Every cron run is captured as a structured trace (OpenTelemetry-compatible), including start/end timestamps, exit codes, stdout/stderr, and resource metrics. You can replay any failed execution locally or stream logs to your existing observability stack (Datadog, Grafana, ELK). No more grepping through syslog.

## How to Use

### 1. Bootstrap the Sentinel Agent
Deploy the cronguard agent as a sidecar or standalone daemon. Configure your cron job definitions in a YAML manifest:

```yaml
jobs:
  - name: nightly-db-cleanup
    schedule: "0 2 * * *"
    command: "./cleanup.sh"
    timeout: 300
    retry_policy:
      max_retries: 3
      backoff: exponential
    alert_on: [failure, drift, timeout]
```

Run `cronguard init --config ./cronguard.yml` to register the jobs and start the monitoring loop.

### 2. Integrate with Your Existing Cron Daemon
cronguard wraps your existing cron entries without modifying the system crontab. Use the `cronguard wrap` command to prefix your cron commands:

```
* * * * * /usr/local/bin/cronguard run --job-id health-check -- /opt/healthcheck.sh
```

This injects the sentinel wrapper that captures exit codes, timing, and resource usage. The original command runs unmodified.

### 3. Set Up Slack/PagerDuty Alerting with Conditional Escalation
Define alert conditions in the manifest with escalation rules:

```yaml
alerting:
  channels:
    - type: slack
      webhook_url: "https://hooks.slack.com/services/..."
    - type: pagerduty
      integration_key: "your-key"
  escalation:
    - condition: "failure_count > 2"
      notify: pagerduty
    - condition: "drift_seconds > 60"
      notify: slack
```

Then run `cronguard agent --daemon` to start the background watcher. It will automatically evaluate conditions and route alerts.

## Best Practices

### 1. Always Set Idempotency Keys for Critical Jobs
For any job that mutates state (database writes, file system operations, API calls), define a unique idempotency key in the job manifest. This prevents double execution during failover or manual retries. cronguard enforces the key via its distributed lock mechanism, so even if the same job fires twice, only one execution proceeds.

### 2. Use Staggered Schedules for Resource-Intensive Jobs
When multiple heavy jobs (e.g., data aggregation, ETL, cache warming) share the same cron slot, cronguard’s drift detection can trigger false positives due to resource contention. Mitigate this by introducing a 30–60 second offset between jobs. cronguard’s `stagger` directive can automatically adjust schedules based on historical resource usage profiles.
