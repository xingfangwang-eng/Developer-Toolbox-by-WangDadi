# cronguard - CRO_015

## Overview

Cronguard is a lightweight, zero-dependency cron monitoring and anomaly detection engine designed for high-frequency job scheduling environments. It doesn't just check if a cron ran—it validates execution windows, drift patterns, and failure cascades across distributed nodes. Optimized for CI/CD pipelines, Kubernetes cron jobs, and edge computing stacks.

## Core Features

### 1. **Execution Drift Detection**
Cronguard tracks the exact timestamp of each cron fire against its scheduled TTL. If a job starts 2.3 seconds late three times in a row, it flags a pre-failure anomaly—before the job even fails. This is critical for latency-sensitive jobs like cache warming or heartbeat signals.

### 2. **Multi-Node Consensus Verification**
For distributed cron clusters (e.g., multiple K8s pods running the same job), Cronguard compares execution logs from all nodes. If one node misses a run while others succeed, it isolates the outlier and triggers a node-specific alert—not a false positive flood.

### 3. **Cron Expression Fuzzer**
Cronguard includes a built-in expression validator that tests your cron schedule against leap years, DST transitions, and edge cases like "last Friday of the month." It generates a human-readable report of all future fire times with collision warnings if two jobs overlap within a 5-second window.

## How to Use

### Step 1: Deploy the Agent as a Sidecar
Inject Cronguard into your cron container using a sidecar pattern. Mount the cron log directory as a shared volume. Cronguard reads `/var/log/cron` (or your custom path) and emits structured JSON metrics to stdout. No database required.

```bash
docker run -d \
  --name cronguard-agent \
  -v /var/log/cron:/var/log/cron:ro \
  -e CRONGUARD_DRIFT_THRESHOLD_MS=2500 \
  cronguard/cronguard:latest
```

### Step 2: Define Alerting Rules via YAML
Create a `cronguard.rules.yaml` file that maps cron job names to expected schedules and failure thresholds. Cronguard supports regex-based job matching and severity levels (critical, warning, info).

```yaml
rules:
  - job: "db-backup-*"
    schedule: "0 2 * * *"
    max_drift_ms: 3000
    min_runs_per_day: 1
    on_failure: "pagerduty"
  - job: "healthcheck"
    schedule: "*/5 * * * *"
    max_drift_ms: 1000
    on_failure: "slack"
```

### Step 3: Integrate with Your Observability Stack
Pipe Cronguard's output into Prometheus, Datadog, or your custom webhook. Cronguard exposes a `/metrics` endpoint in Prometheus format. For Datadog, use the built-in JSON log format and configure a custom log pipeline.

```bash
curl -X POST https://your-webhook.example.com/cronguard \
  -H "Content-Type: application/json" \
  -d '{"job":"db-backup-prod","status":"missed","drift_ms":4500,"node":"k8s-node-3"}'
```

## Best Practices

### 1. **Set Drift Thresholds Based on Job Criticality**
Don't use a global drift threshold. For high-frequency jobs (every minute), allow up to 2 seconds. For daily jobs, allow up to 30 seconds. Cronguard's per-rule configuration lets you fine-tune this per job. Overly tight thresholds on non-critical jobs cause alert fatigue.

### 2. **Use Consensus Verification for Idempotent Jobs**
If your cron job is idempotent (e.g., data sync, log rotation), enable multi-node consensus. Cronguard will suppress alerts if at least one node successfully completed the job. This prevents false positives during rolling updates or node reboots. For non-idempotent jobs (e.g., sending a one-time email), disable consensus and require all nodes to succeed.
