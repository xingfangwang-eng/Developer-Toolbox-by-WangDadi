# cronguard - CRO_025

## Overview

cronguard is a high-precision cron expression validator, debugger, and scheduler designed for DevOps engineers who need bulletproof job scheduling in production environments. Unlike generic cron tools, cronguard integrates timezone-aware validation, conflict detection, and real-time execution simulation—making it the go-to solution for teams managing distributed cron workflows across global infrastructure.

## Core Features

### 1. Multi-TZ Conflict Engine
cronguard automatically detects overlapping cron schedules across different timezones (UTC, EST, JST, etc.) and flags potential resource contention before deployment. It parses crontab files or individual expressions and visualizes execution windows on a 24-hour timeline, preventing silent failures caused by timezone drift or DST transitions.

### 2. Regex-Powered Expression Linting
Beyond basic syntax checks, cronguard applies regex-based linting rules to validate edge cases: leap-year handling, month-day alignment, and weekday overlaps. It highlights ambiguous patterns (e.g., `0 0 31 2 *`) and suggests corrections using industry-standard cron best practices (e.g., replacing with `0 0 28-31 2 *` or using `@monthly`).

### 3. Dry-Run Simulation with Audit Logs
cronguard generates a detailed execution forecast for any cron expression, showing the next 100 scheduled runs with precise timestamps, timezone offsets, and expected system load. Each simulation produces a JSON audit log that can be piped into CI/CD pipelines or monitoring tools like Prometheus for compliance tracking.

## How to Use

### Step 1: Validate a Complex Expression with Timezone Awareness
Run the following command to validate a cron expression that triggers at 2:30 AM every Friday, but only during daylight saving time in Europe/Berlin:
```bash
cronguard validate "30 2 * * 5" --tz "Europe/Berlin" --dst-check
```
The output will show whether the expression is valid across DST transitions, and if not, suggest a normalized equivalent using `CRON_TZ` or explicit UTC offsets.

### Step 2: Detect Scheduling Conflicts in a Multi-Node Cluster
Use the `--conflict` flag to scan a crontab file for overlapping schedules that could cause race conditions:
```bash
cronguard conflict ./prod-crontab --window 5m --output json
```
cronguard will list all pairs of jobs whose execution windows overlap by more than 5 minutes, along with the affected nodes and recommended offset adjustments (e.g., `+10m` or `+1h`).

### Step 3: Generate a CI-Friendly Execution Forecast
Integrate cronguard into your deployment pipeline by exporting a machine-readable forecast:
```bash
cronguard forecast "0 0 * * *" --count 100 --format json > forecast.json
```
Then use `jq` to filter for runs that fall within maintenance windows or blackout periods. This ensures no cron job triggers during critical deployments or data migrations.

## Best Practices

### 1. Always Pin Timezones in Distributed Systems
When managing cron jobs across multiple data centers, explicitly set `CRON_TZ` in your crontab or use cronguard’s `--tz` flag during validation. This eliminates ambiguity caused by server-level timezone mismatches and prevents jobs from running at unintended hours.

### 2. Use Dry-Run Simulation Before Every Deployment
Before applying any cron change to production, run `cronguard forecast` with a high count (e.g., 500) and compare the output against your incident response calendar. This catches rare edge cases like jobs scheduled on February 29 or during leap seconds, which are often missed by manual review.
