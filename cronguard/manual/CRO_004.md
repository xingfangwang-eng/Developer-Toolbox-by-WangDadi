## What is cronguard?

cronguard is a **deterministic cron expression validator and scheduler simulator** that parses, normalizes, and tests cron expressions against real-world edge cases — including leap seconds, DST transitions, and system clock skews. Unlike basic online validators, cronguard integrates with CI/CD pipelines to catch malformed expressions before they hit production, and provides a sandboxed runtime to preview next N fire times with nanosecond precision.

---

## Core Features

### 1. Deterministic Execution Preview with Timezone Drift Detection

cronguard simulates the next `N` fire times of any cron expression across multiple timezones simultaneously. It automatically detects **timezone drift** — where the same expression yields different actual execution windows due to DST shifts or regional offset changes — and flags them with actionable warnings. This is critical for globally distributed systems where a job scheduled at `0 2 * * *` in UTC may fire at different wall-clock hours in `America/New_York` during March.

### 2. Cron Syntax Linting with Multi-Dialect Support

cronguard supports **Vixie, Quartz, and AWS EventBridge** cron dialects. It lints expressions for common pitfalls: overlapping ranges, non-orthogonal day-of-month and day-of-week combinations, and out-of-bounds values. The linter also detects **ambiguous expressions** (e.g., `* * 31 2 *`) and suggests normalized alternatives. Output includes line-numbered error reports compatible with GitHub Actions annotations.

### 3. Sandboxed Execution Trace with Failure Injection

Beyond validation, cronguard offers a **sandboxed trace mode** that simulates job execution under failure conditions: missed fire, retry exhaustion, and resource contention. You can inject synthetic delays or resource limits to observe how your cron job behaves under stress. The trace output is a JSON timeline that can be fed into monitoring tools like Prometheus or Grafana for post-mortem analysis.

## How to Use

### Step 1: Validate a Cron Expression with Multi-Timezone Preview

```bash
cronguard validate "*/15 9-17 * * 1-5" --timezones UTC,America/New_York,Asia/Tokyo --next 10
```

This command parses the expression, checks it against the Vixie dialect, and prints the next 10 fire times for each specified timezone. If DST drift is detected (e.g., a job that fires at 09:00 UTC in winter may fire at 10:00 UTC in summer for `America/New_York`), cronguard outputs a `WARN` with the offset difference. Use `--format json` for machine-readable output.

### Step 2: Lint a Crontab File with Dialect-Specific Rules

```bash
cronguard lint ./crontab --dialect quartz --strict
```

cronguard reads a standard crontab file (or a custom schedule definition) and applies dialect-specific linting rules. With `--strict`, it treats warnings as errors (exit code 1). For example, it will flag `0 0 1 * *` because the day-of-month and day-of-week are both wildcarded — a common source of double-fire. The linter outputs a table with severity, line number, and suggested fix.

### Step 3: Run a Sandboxed Execution Trace

```bash
cronguard trace "0 2 * * *" --duration 24h --failure-rate 0.05 --max-retries 3 --output trace.json
```

This command simulates a 24-hour execution window for the given cron expression. It injects a 5% failure rate (random misses or crashes) and models up to 3 retries per failure. The output `trace.json` contains a timeline of fire attempts, successes, failures, and retry delays. Use `--visualize` to generate an ASCII timeline in the terminal for quick inspection.

## Best Practices

### Practice 1: Always Validate Cron Expressions in CI/CD with Multi-Dialect Linting

Integrate `cronguard lint` into your CI pipeline (GitHub Actions, GitLab CI, Jenkins) to catch malformed expressions before deployment. Use `--dialect` to match your target scheduler (e.g., `quartz` for Spring Boot, `aws` for EventBridge). Set `--strict` to fail the build on any warning — this prevents silent drift and double-fire issues that are notoriously hard to debug in production.

### Practice 2: Use Timezone-Aware Execution Preview for Global Job Scheduling

For distributed systems with instances in multiple regions, always run `cronguard validate` with `--timezones` covering all deployment zones. Compare the fire times across timezones: if a job is supposed to run at 02:00 local time in every region, ensure the cron expression is written relative to UTC or uses a fixed offset. cronguard’s drift detection will alert you if the same expression yields different local times due to DST — a common cause of missed maintenance windows.
