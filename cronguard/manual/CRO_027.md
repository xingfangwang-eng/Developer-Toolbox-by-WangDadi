# cronguard - CRO_027

**Cronguard** is a lightweight, polyglot-aware cron job supervisor engineered for distributed environments where uptime, auditability, and failure isolation are non-negotiable. It wraps your existing cron expressions with runtime guards, anomaly detection, and idempotency enforcement — no DSL, no lock-in.

## Hardcore Features

### 1. **Idempotency Locks with TTL & Partition Awareness**
Cronguard prevents duplicate execution across horizontally scaled workers without relying on a centralized database. Each job instance acquires a distributed lease via Redis or etcd, with configurable TTL (minimum 100ms granularity) and shard-key support (e.g., `partition: us-east-1a`). If a worker crashes mid-execution, the lease auto-expires within 500ms — no manual cleanup required. Lease acquisition uses optimistic concurrency control via CAS operations, ensuring zero contention under 10,000 concurrent job registrations.

### 2. **Runtime Anomaly Detection via Execution Drift Analysis**
Instead of static timeouts, Cronguard profiles each job's historical runtime distribution (P50, P95, P99) using a sliding window of 1000 recent executions. It triggers alerts when a current execution exceeds the expected window by a configurable multiplier (e.g., 2.5× P99). The drift engine runs in-memory with O(1) lookup per job, supporting up to 5000 jobs per daemon instance. Alerts are emitted as structured JSON over stdout/stderr, consumable by Prometheus Alertmanager or PagerDuty webhooks.

### 3. **Audit Trail with Immutable Execution Logs**
Every cron run is recorded in a structured, append-only log (JSON Lines, shipped to S3/GCS or local disk with 256-bit AES-GCM encryption). Each entry includes job name, trigger timestamp (nanosecond precision), exit code, stdout/stderr SHA-256 hash, lease metadata, and worker hostname. Logs are queryable via `jq` or your SIEM pipeline — no database needed. Retention is configurable per job (default: 90 days), with automatic rotation and compression (gzip level 6).

## How to Use

### Step 1: Define a Job with Guards
Create a YAML configuration file that declares your cron schedule, command, and guard policies. Example `jobs/daily-report.yaml`:

```yaml
name: daily-report
schedule: "0 6 * * *"
command: "python3 /app/generate_report.py"
guards:
  max_duration: 300s
  retry: 3
  idempotency:
    enabled: true
    ttl: 600s
    namespace: prod-us-east-1
    shard_key: "{{ .Hostname }}"
  drift_alert:
    enabled: true
    deviation_threshold: 2.5
    sliding_window_size: 1000
```

### Step 2: Register the Job with Cronguard Daemon
Start the supervisor daemon and register your job file. The daemon will validate the cron expression against the POSIX standard, check for syntax errors, and load the guard policies into memory. Use `--log-level debug` to see lease acquisition details.

```bash
cronguard daemon start --config-dir ./jobs/ --log-level info
cronguard job register --file jobs/daily-report.yaml --daemon localhost:9090
```

Confirm the job is active with `cronguard job list --daemon localhost:9090`. The daemon emits structured logs to stdout by default — pipe them to `jq` for real-time monitoring:

```bash
cronguard daemon start --config-dir ./jobs/ | jq 'select(.level == "warn")'
```

### Step 3: Monitor Execution and Review Audit Logs
After the job runs, inspect its execution history using the built-in CLI with pagination and filtering:

```bash
cronguard history --job daily-report --last 10 --format json | jq '.[] | {execution_id, duration, exit_code}'
```

Each entry shows:
- `execution_id` — UUIDv4 unique per run
- `started_at`, `completed_at` — RFC3339 with nanosecond precision
- `duration` — milliseconds, with P50/P95/P99 percentiles
- `exit_code` and `error_message` (if any)
- `lease_holder` — worker hostname and PID

To dive deeper, stream the audit log directly with decryption and decompression:

```bash
cronguard logs --job daily-report --execution-id <id> --decrypt-key ./key.pem | gunzip | jq '.stdout'
```

## Industry Best Practices

### 1. **Always Set `max_duration` to 1.5× Your P99 Runtime**
Underestimating the ceiling leads to premature kills during legitimate spikes (e.g., database replication lag). Use `cronguard history --job <name> --stats --percentiles 99` to extract P99 values after a week of runs, then update your guard. For example, if P99 is 200s, set `max_duration: 300s`. This prevents false positives while still catching stuck processes. Monitor drift alerts weekly to adjust thresholds as job complexity evolves.

### 2. **Use Separate Lease Namespaces for Production vs. Staging**
When testing idempotency in parallel environments, prefix your job names or set the `namespace` field in your config to avoid cross-environment lease collisions. Example: `namespace: prod-us-east-1` for production, `namespace: staging-us-east-1` for staging. This ensures staging jobs don't accidentally block production executions. Additionally, configure separate Redis/etcd instances per environment to isolate failure domains — a staging cluster crash should never impact production lease availability.
