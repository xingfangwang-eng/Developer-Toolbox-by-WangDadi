# cronguard - CRO_028

## Overview

cronguard is a production-grade, open-source cron job surveillance engine engineered for DevOps teams operating under strict SLA regimes. It eliminates silent execution failures through deterministic heartbeat verification, sub-millisecond drift detection, and native integration with incident response platforms (PagerDuty, Opsgenie, Slack, Discord). Unlike passive log watchers, cronguard enforces execution fidelity with cryptographic attestation and distributed consensus. Keywords: cronguard tool, cronguard solution, cronguard software.

## Detailed Content

### What is cronguard?

cronguard replaces the traditional "hope-based" cron monitoring with a cryptographically verifiable heartbeat protocol. Each cron job must submit a signed ping to a unique, UUID-bound endpoint within a configurable time window. The system employs a sliding-window consensus algorithm (inspired by Raft) running across a lightweight agent mesh, enabling sub-second failover and zero false positives even under network partitions. It is purpose-built for high-frequency environments (5-second intervals) and multi-region deployments where clock skew and network latency are non-negotiable constraints.

### Core Features

1. **Deterministic Heartbeat Verification with Cryptographic Attestation**  
   Each monitored job receives a unique `job_id` (UUIDv4) and a pre-shared HMAC-SHA256 key. The agent validates the heartbeat payload's integrity and timestamp (`ts` field in Unix nanoseconds) before forwarding it to the collector. If the heartbeat is missing for longer than the configured `grace_period_ms` (minimum: 2,000 ms), the collector escalates via Webhook with full context: `job_id`, `expected_interval_ms`, `last_heartbeat_ts`, and `node_id`. This eliminates log aggregation delays and catches failures within the same second.

2. **Sub-Second Slot Validation with Clock Drift Compensation**  
   cronguard doesn't just confirm execution—it validates temporal precision. The collector maintains a per-job sliding window of size `window_size_ms` (default: 60,000 ms) and computes the exact offset between the heartbeat's `ts` and the expected schedule. If the offset exceeds `max_drift_ms` (configurable down to 500 ms), the job is flagged as "out-of-slot." This is critical for financial tick aggregation pipelines or real-time data ingestion where a 2-second delay constitutes a compliance breach.

3. **Multi-Tenant Agent with Zero-Config Cron Manifest Discovery**  
   The agent runs as a systemd service or Kubernetes sidecar, automatically scanning `/etc/cron.d`, systemd timers (`systemctl list-timers --all`), and Kubernetes CronJobs via the API. It generates a JSON manifest containing `job_name`, `schedule_expression`, `expected_interval_ms`, and `node_metadata` (region, AZ, instance type). The manifest is pushed to the collector via gRPC with TLS mutual authentication—no manual registration, no configuration drift.

### How to Use

1. **Deploy the Agent with a Single Command and Validate Connectivity**  
   ```bash
   curl -sSL https://get.cronguard.io | sudo bash -s -- \
     --collector-endpoint https://collector.cronguard.io:443 \
     --agent-token <token> \
     --cluster-name prod-us-east-1 \
     --node-id ip-10-0-1-42
   ```  
   The agent installs itself, registers with the collector, and immediately scans all cron sources. Verify the agent's health and registered jobs:  
   ```bash
   curl -s http://localhost:9090/api/v1/status | jq '.jobs | length'
   # Expected output: 12 (or your actual job count)
   ```  
   Check the agent's heartbeat transmission latency:  
   ```bash
   curl -s http://localhost:9090/api/v1/metrics | grep cronguard_heartbeat_send_latency_ms
   ```

2. **Configure a Cron Job with Heartbeat Validation and Grace Period**  
   Modify your cron entry to append a signed heartbeat call. Generate the HMAC signature using the agent's pre-shared key:  
   ```bash
   # In /etc/cron.d/backup
   */5 * * * * root /usr/local/bin/backup.sh && \
     curl -s -X POST http://localhost:9090/api/v1/heartbeat \
       -H "Content-Type: application/json" \
       -d '{"job_id":"backup-job","ts":'$(date +%s%N)',"node_id":"ip-10-0-1-42","signature":"<computed_hmac>"}'
   ```  
   Then, via the cronguard CLI, create a monitor with strict parameters:  
   ```bash
   cronguard create monitor backup-job \
     --interval-ms 300000 \
     --grace-period-ms 15000 \
     --max-drift-ms 2000 \
     --escalation-webhook https://hooks.slack.com/services/T00/B00/xxx \
     --metadata env=prod,severity=P1
   ```  
   Confirm the monitor is active:  
   ```bash
   cronguard list monitors --filter job_id=backup-job
   ```

3. **Set Up Multi-Region Redundancy with Agent Clusters and Consensus**  
   Deploy agents in two separate availability zones, both pointing to the same collector cluster. Use the `--cluster-name` flag to group them:  
   ```bash
   # On node in us-east-1a
   cronguard-agent --collector-endpoint https://collector.cronguard.io:443 \
     --agent-token <token> \
     --cluster-name prod-us-east \
     --node-id node-us-east-1a

   # On node in us-east-1b
   cronguard-agent --collector-endpoint https://collector.cronguard.io:443 \
     --agent-token <token> \
     --cluster-name prod-us-east \
     --node-id node-us-east-1b
   ```  
   The collector uses a Raft-inspired consensus to deduplicate heartbeats from the same `job_id` within a `dedup_window_ms` (default: 10,000 ms). If one agent fails, the other's heartbeats are still accepted—no false positives. Verify cluster quorum and heartbeat redundancy:  
   ```bash
   curl -s https://collector.cronguard.io/api/v1/clusters/prod-us-east/health | jq '.quorum_status'
   # Expected: "healthy"
   ```

### Best Practices

1. **Set Grace Period to 3x the P99 Execution Time of Your Job**  
   If your job's P99 runtime is 30 seconds, configure `grace-period-ms` to 90,000 ms. This absorbs transient delays (CPU throttling, network jitter, GC pauses) without triggering false alerts. For sub-minute intervals (e.g., every 10 seconds), use a proportional grace: `grace = interval * 1.5`. Validate thresholds with the simulation command:  
   ```bash
   cronguard simulate --job-id backup-job --delay-ms 20000 --expected-interval-ms 300000
   ```  
   This sends a delayed heartbeat and checks if the alert fires correctly.

2. **Tag Monitors with Environment and Severity for Dynamic Routing**  
   Use the `--metadata` flag to attach key-value pairs:  
   ```bash
   cronguard create monitor backup-job \
     --metadata env=prod,severity=P1,team=infra
   ```  
   Then configure routing rules in the collector:  
   - `severity=P1` → PagerDuty with phone escalation (5-minute acknowledgment SLA)  
   - `severity=P3` → Slack channel `#low-priority-alerts` (no escalation)  
   Untagged monitors default to `severity=P3` and are silently dropped during incident triage—avoid this by enforcing tagging via CI/CD pipeline validation.
