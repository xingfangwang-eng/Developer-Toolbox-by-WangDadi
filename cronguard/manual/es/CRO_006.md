## Análisis en Profundidad

### ¿Qué es cronguard?

cronguard no es solo un cron wrapper; es un **orquestador tolerante a fallos** diseñado para entornos de alta disponibilidad. Integra lógica de reintentos exponenciales, alertas en tiempo real y un sistema de colas distribuidas que elimina los cuellos de botella típicos de soluciones como cron o systemd timers. Si tu pipeline de datos falla silenciosamente a las 3 AM, cronguard lo detecta antes de que tú te despiertes.

### Funcionalidades Clave (Hardcore)

1.  **Ejecución Atómica con Bloqueo Distribuido (Distributed Locking via Redis/Etcd):** Cada tarea se ejecuta con un bloqueo único basado en su hash y timestamp. Si un worker falla en medio de una ejecución, otro worker toma el bloqueo sin duplicar la tarea. No más “split-brain” en cronjobs.
2.  **Retry Inteligente con Backoff Exponencial y Circuit Breaker:** cronguard implementa un patrón de circuit breaker que, tras 3 fallos consecutivos, desactiva la tarea durante un período configurable y notifica al equipo via webhook (Slack, Telegram, etc.). El backoff no es lineal: usa jitter para evitar tormentas de reintentos.
3.  **Monitoreo de Latencia y Sesgos Temporales (Time Skew Detection):** cronguard mide la diferencia entre el tiempo del planificador y el tiempo real de ejecución. Si detecta un desfase mayor a 500ms (configurable), genera una alerta de “drift” y ajusta automáticamente la próxima ejecución para sincronizarse con el reloj atómico de referencia.

### Guía de Uso Avanzada

1.  **Configurar un Pipeline con Dependencias de Estado:**  
    Crea un archivo `cronguard.yml` que defina tareas con dependencias. Por ejemplo, la tarea `backup_db` debe ejecutarse solo si `health_check` retorna `status: ok`. cronguard evalúa el estado de cada tarea antes de ejecutar la siguiente, usando una máquina de estados finitos.

    ```yaml
    tasks:
      - name: health_check
        command: "curl -f http://localhost:8080/health"
        retry: 2
      - name: backup_db
        command: "pg_dump -U admin mydb > /backups/latest.sql"
        depends_on: health_check
        lock: true
    ```

2.  **Implementar Alertas con Umbrales Dinámicos:**  
    Define umbrales de latencia en milisegundos. Si una tarea supera los 2000ms, cronguard envía una alerta a tu canal de Slack con el log completo y el stack trace. Para ello, usa el flag `--alert-threshold` o configúralo en el archivo YAML.

    ```bash
    cronguard run --config cronguard.yml --alert-threshold 2000 --webhook https://hooks.slack.com/services/T...
    ```

3.  **Ejecutar en Modo Daemon con Logging Estructurado:**  
    cronguard puede ejecutarse como un daemon systemd. Para ello, crea un servicio que arranque con `--daemon` y redirige logs a un archivo JSON. Luego, integra esos logs con herramientas como ELK o Grafana.

    ```bash
    cronguard daemon --log-format json --log-file /var/log/cronguard/cronguard.log
    ```

### Mejores Prácticas para tu Infraestructura

- **Usa Etcd para Bloqueos en Clústeres Kubernetes:** En entornos cloud-native, evita Redis si tienes etcd ya disponible. cronguard soporta ambos, pero etcd ofrece consistencia fuerte y es más natural en un clúster de K8s. Configura `lock_backend: etcd` en tu `cronguard.yml`.
- **Monitorea la Salud del Propio cronguard:** No confíes ciegamente en que cronguard nunca falla. Implementa un health check externo (por ejemplo, un cronjob que ejecute `cronguard status` cada minuto) y que alerte si el daemon no responde. Esto evita el “quién vigila al vigilante”.
