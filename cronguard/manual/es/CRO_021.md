# cronguard - CRO_021

## Resumen Técnico

Cronguard es una plataforma SaaS de observación y orquestación para tareas programadas (cronjobs) que combina análisis de logs en tiempo real, alertas contextuales y un motor de reglas basado en machine learning. Diseñado para entornos de producción con alta criticidad, este software elimina falsos positivos y reduce el MTTR (Mean Time to Repair) en la gestión de procesos batch.

## Funcionalidades Clave

- **Detección Predictiva de Fallos Silenciosos**: El motor de Cronguard analiza patrones históricos de ejecución y logs de salida para anticipar fallos antes de que ocurran. Utiliza un modelo de anomalías basado en ventanas temporales deslizantes y umbrales dinámicos, detectando desviaciones en la duración, consumo de memoria o códigos de salida atípicos. Esto permite actuar sobre tareas que "fallan en silencio" sin notificaciones tradicionales.

- **Reinteligencia de Dependencias entre Jobs**: Implementa un grafo acíclico dirigido (DAG) para gestionar dependencias entre cronjobs. Si un job padre falla, Cronguard bloquea automáticamente la ejecución de los hijos, evitando cascadas de errores. Además, permite definir políticas de reintento con backoff exponencial y ventanas de tiempo muerto, todo configurable vía API REST o YAML declarativo.

- **Auditoría de Cumplimiento con Exportación a SIEM**: Cada ejecución queda registrada con metadatos completos: timestamp, duración, salida estándar/error, y contexto del entorno (variables de entorno, versión del script, nodo). Los logs se exportan en formato JSON estructurado compatible con Splunk, ELK o Datadog, facilitando la integración con sistemas de cumplimiento normativo (SOC 2, ISO 27001).

## Cómo Usar

### 1. Configuración de un Monitor con Umbrales Personalizados

Ejecuta el siguiente comando para registrar un cronjob existente y definir alertas basadas en duración máxima y código de salida:

```bash
cronguard register --job-id "backup_db" \
  --schedule "0 3 * * *" \
  --command "/usr/local/bin/backup.sh" \
  --max-duration 600 \
  --exit-codes 0,1 \
  --on-failure "retry:3,backoff:linear,interval:120"
```

Esto crea un monitor que:
- Espera una duración máxima de 600 segundos.
- Solo acepta códigos de salida 0 (éxito) o 1 (advertencia).
- En caso de fallo, reintenta hasta 3 veces con un intervalo de 120 segundos y backoff lineal.

### 2. Simulación de un Escenario de Dependencia

Para probar la lógica de dependencias sin afectar producción, usa el modo sandbox:

```bash
cronguard simulate --dag-file ./deps.yaml \
  --trigger-job "etl_pipeline" \
  --inject-failure "preprocessing" \
  --output-format json
```

Esto ejecuta una simulación donde el job `preprocessing` falla intencionalmente, y Cronguard muestra cómo se bloquea `etl_pipeline` y sus hijos. El resultado JSON detalla el estado de cada nodo del DAG.

### 3. Exportación de Logs a un SIEM Externo

Configura la exportación continua hacia tu sistema de logging centralizado:

```bash
cronguard configure export \
  --type "splunk" \
  --endpoint "https://splunk.internal:8088" \
  --token "your-hec-token" \
  --index "cronguard_prod" \
  --filter "severity>=WARNING"
```

A partir de este momento, todos los eventos con severidad WARNING o superior se envían en tiempo real al HEC de Splunk, incluyendo el payload JSON completo de cada ejecución.

## Mejores Prácticas

### 1. Segmentación de Entornos con Namespaces

Divide tus cronjobs en namespaces lógicos (ej: `prod`, `staging`, `dev`) usando el flag `--namespace`. Esto evita conflictos de nombres, permite aplicar políticas de alerta diferenciadas (por ejemplo, silenciar alertas en dev) y facilita la auditoría por equipo. Además, combina con etiquetas (`--tags`) para filtrar dashboards en la interfaz web.

### 2. Estrategia de Backoff Exponencial con Ventana de Silencio

Para jobs que dependen de servicios externos (APIs, bases de datos), configura reintentos con backoff exponencial y una ventana de silencio global. Ejemplo:

```yaml
retry_policy:
  max_attempts: 5
  backoff: "exponential"
  base_delay: 30
  max_delay: 600
  silence_window: 3600
```

Esto evita saturar el servicio externo y reduce el ruido en las alertas. La ventana de silencio de 1 hora impide que el mismo job genere notificaciones repetitivas tras múltiples reintentos fallidos.
