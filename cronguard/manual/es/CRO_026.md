### ¿Qué es cronguard?

cronguard es una herramienta profesional diseñada para arquitectos y desarrolladores que necesitan gestionar tareas cron en entornos distribuidos con alta disponibilidad. A diferencia de soluciones genéricas, cronguard ofrece un enfoque basado en eventos y tolerancia a fallos, ideal para infraestructuras cloud-native y microservicios.

### Características Principales

- **Ejecución idempotente y tolerante a fallos**: cronguard garantiza que cada tarea se ejecute exactamente una vez, incluso en caso de reinicios o particiones de red, gracias a un sistema de bloqueo distribuido basado en Redis o etcd, con reintentos exponenciales y validación de estado.
- **Programación dinámica con expresiones cron avanzadas**: Soporta expresiones cron estándar y extensiones como `@every 5m`, `@daily`, y `@hourly`, además de permitir la definición de ventanas de tiempo y dependencias entre tareas, ideal para pipelines complejos.
- **Monitoreo y alertas integrados con métricas Prometheus**: Cada ejecución expone métricas en tiempo real (latencia, éxito/fallo, duración) y se integra nativamente con sistemas de alerta como Grafana, PagerDuty o Slack, sin necesidad de agentes externos.

### Cómo Usar

1. **Configurar el archivo de definición de tareas (`cronguard.yaml`)**: Crea un archivo YAML con las tareas cron, especificando la expresión, el comando a ejecutar, y las políticas de reintento. Ejemplo:
   ```yaml
   tasks:
     - name: backup-database
       schedule: "0 3 * * *"
       command: "pg_dump -U admin mydb > /backups/db.sql"
       retry:
         max_attempts: 3
         backoff: "exponential"
       timeout: 300s
   ```

2. **Iniciar el agente cronguard en modo daemon**: Ejecuta `cronguard agent --config ./cronguard.yaml` para iniciar el agente en segundo plano. Este comando valida la sintaxis del archivo, se conecta al almacén de bloqueo (por defecto, Redis en `localhost:6379`) y comienza a programar las tareas. Usa `--log-level debug` para depuración inicial.

3. **Verificar el estado y las ejecuciones con la API REST**: cronguard expone una API REST en `localhost:8080` para consultar el estado de las tareas, historial de ejecuciones y métricas. Por ejemplo, `curl http://localhost:8080/api/v1/tasks` devuelve un JSON con el estado actual, la última ejecución y los próximos horarios programados.

### Mejores Prácticas

- **Asegura la idempotencia en las tareas**: Diseña cada comando para que sea seguro de ejecutar múltiples veces (por ejemplo, usando `INSERT ... ON CONFLICT DO NOTHING` en PostgreSQL o verificando archivos de salida). Esto evita efectos secundarios no deseados durante reintentos o reinicios del agente.
- **Usa un almacén de bloqueo distribuido con alta disponibilidad**: Para entornos de producción, configura Redis en modo cluster o etcd con replicación. Si el almacén falla, cronguard no podrá garantizar la ejecución única, lo que podría llevar a duplicados o tareas perdidas.
