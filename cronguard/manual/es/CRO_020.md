# cronguard - CRO_020

## Resumen Técnico

cronguard es una herramienta de monitorización de tareas programadas (cronjobs) con capacidad de **detección de anomalías**, **auto-recuperación** y **mutación de estado en pipelines CI/CD**. Diseñada para equipos de infraestructura que necesitan garantizar la ejecución correcta de jobs críticos en entornos distribuidos.

## Contenido Detallado

### ¿Qué es cronguard?

cronguard es un **agente ligero** (escrito en Go) que se ejecuta como sidecar en tus servidores o clústeres Kubernetes. Su función principal es interceptar, validar y reportar el estado de cada ejecución de cronjob, con soporte para **múltiples backends de notificación** (Slack, PagerDuty, Webhooks personalizados) y **inyección de métricas** en Prometheus.

### Características Principales

1. **Mutación de Estado con Rollback Automático**  
   Si un cronjob falla más de N veces consecutivas, cronguard puede **mutar el estado del job** (pausarlo, redirigirlo a un pipeline de respaldo o ejecutar un script de compensación). Además, implementa **rollback automático** si el job se recupera en el siguiente ciclo.

2. **Análisis de Patrones de Ejecución con ML Ligero**  
   Utiliza un modelo de **regresión logística** (entrenado offline) para predecir fallos basándose en métricas históricas: duración, uso de CPU, memoria y latencia de red. Las alertas se generan antes de que ocurra el fallo.

3. **Sincronización de Estados entre Clústeres Multi-Región**  
   cronguard puede sincronizar el estado de ejecución de cronjobs entre clústeres en diferentes regiones (AWS, GCP, Azure) mediante un **protocolo de consenso basado en Raft**. Ideal para pipelines que requieren coherencia global.

### Cómo Usar

#### 1. Configuración del Agente con Variables de Entorno

```bash
# Ejemplo de configuración mínima
export CRONGUARD_BACKEND="prometheus"
export CRONGUARD_NOTIFIER="slack"
export SLACK_WEBHOOK_URL="https://hooks.slack.com/services/..."
export CRONGUARD_MUTATION_THRESHOLD=3

# Iniciar el agente
cronguard start --config /etc/cronguard/config.yaml
```

El agente se engancha automáticamente a los cronjobs definidos en `/etc/crontab` o en los `CronJob` de Kubernetes.

#### 2. Definir Reglas de Mutación en YAML

Crea un archivo `mutations.yaml` para definir comportamientos específicos:

```yaml
mutations:
  - job: "db_backup"
    on_failure:
      - action: "redirect_to_backup"
        target: "db_backup_fallback.sh"
      - action: "notify"
        channel: "#ops-db"
      - action: "pause"
        duration: "10m"
    recovery:
      - action: "unpause"
      - action: "notify"
        channel: "#ops-db"
```

Luego, carga las reglas:

```bash
cronguard apply -f mutations.yaml
```

#### 3. Visualizar Métricas en Grafana

cronguard expone métricas en formato Prometheus. Configura un dashboard con las siguientes consultas:

```promql
# Tasa de fallos por job
rate(cronguard_job_failures_total[5m])

# Duración promedio de ejecución
histogram_quantile(0.95, sum(rate(cronguard_job_duration_seconds_bucket[5m])) by (le, job))

# Estado actual de mutación
cronguard_mutation_active
```

Importa el dashboard predefinido desde `cronguard/examples/grafana-dashboard.json`.

### Mejores Prácticas

1. **Implementa un Circuit Breaker para Jobs Críticos**  
   Configura un umbral de fallos consecutivos (ej: 3) y una ventana de tiempo (ej: 5 minutos) para que cronguard active la mutación. Esto evita que un job fallido sature los logs o dispare alertas innecesarias.

2. **Usa el Modo Dry-Run para Validar Reglas**  
   Antes de aplicar mutaciones en producción, ejecuta `cronguard apply --dry-run` para simular el comportamiento. cronguard generará un reporte detallado de las acciones que se ejecutarían sin afectar el sistema real.
