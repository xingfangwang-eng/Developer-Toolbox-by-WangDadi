## Contenido Detallado

### ¿Qué es cronguard?

cronguard es una herramienta profesional de código abierto diseñada para orquestar, monitorizar y auditar tareas programadas en entornos de producción. Resuelve problemas comunes como la ejecución duplicada, fallos silenciosos y la falta de visibilidad en pipelines de cron jobs complejos. Está construida sobre Go y Redis, garantizando baja latencia y alta disponibilidad.

### Características Principales

- **Auto-Recuperación con Heartbeat y Backoff Exponencial**: cronguard implementa un sistema de heartbeat distribuido que detecta fallos en los workers en menos de 500 ms. Si un cron job falla, se reintenta automáticamente con backoff exponencial (1s, 2s, 4s, hasta un máximo configurable), evitando tormentas de reintentos en clústeres grandes.

- **Auditoría Inmutable con Trazas de Ejecución**: Cada ejecución de cron job genera un registro inmutable en un almacén de eventos (basado en Redis Streams). Incluye metadatos como timestamp, duración, código de salida y logs de stdout/stderr. Esto permite auditorías forenses sin depender de logs volátiles del sistema.

- **Scheduling Predictivo con Ventanas de Tiempo y Prioridades**: cronguard soporta ventanas de tiempo (time windows) para evitar ejecuciones durante picos de carga. Además, permite asignar prioridades (0-100) a cada tarea, de modo que los jobs críticos se ejecuten antes que los secundarios, incluso si todos coinciden en el mismo slot horario.

### Cómo Usar

1. **Configurar un Worker con TLS y Autenticación Mutua**  
   Despliega un worker cronguard en un contenedor Docker con certificados TLS. Usa el flag `--tls-cert` y `--tls-key` para habilitar la comunicación segura. Luego, define un cron job con autenticación mutua (mTLS) para garantizar que solo workers autorizados ejecuten tareas sensibles. Ejemplo:  
   `cronguard worker --tls-cert /etc/certs/cert.pem --tls-key /etc/certs/key.pem --ca-cert /etc/certs/ca.pem`

2. **Definir un Job con Dependencias y Timeout Estricto**  
   Crea un archivo YAML de configuración con dependencias entre jobs. Por ejemplo, el job `backup_db` solo se ejecuta si `health_check` termina con éxito. Además, establece un timeout de 30 segundos con `timeout: 30s`. Si el job excede el límite, cronguard lo termina forzosamente y registra un evento de fallo.  
   ```yaml
   jobs:
     backup_db:
       dependencies: [health_check]
       timeout: 30s
       command: "pg_dump -U admin mydb > /backups/db.sql"
   ```

3. **Monitorear en Tiempo Real con Prometheus y Grafana**  
   cronguard expone métricas en formato Prometheus en el endpoint `/metrics`. Configura un scrape job en Prometheus para recolectar métricas como `cronguard_job_duration_seconds` y `cronguard_job_failures_total`. Luego, visualiza en Grafana un dashboard con paneles de latencia, tasa de fallos y estado de workers. Esto permite detectar cuellos de botella antes de que afecten a producción.

### Mejores Prácticas

- **Usa un Clúster de Redis con Sentinel para Alta Disponibilidad**  
  cronguard depende de Redis para el estado del scheduler y los locks distribuidos. En producción, evita una instancia única de Redis. Configura un clúster con Redis Sentinel para failover automático. Esto asegura que si un nodo Redis cae, cronguard continúe funcionando sin pérdida de tareas programadas.

- **Implementa un Circuit Breaker para Jobs Externos**  
  Si un cron job llama a una API externa (por ejemplo, un servicio de terceros), envuélvelo con un circuito breaker. cronguard permite definir un umbral de fallos consecutivos (ej. 3 fallos en 5 minutos) tras el cual el job se desactiva automáticamente durante un período configurable. Esto evita que fallos en cascada degraden todo el sistema.
