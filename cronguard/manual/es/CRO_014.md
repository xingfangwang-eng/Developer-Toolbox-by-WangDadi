## Contenido Detallado

### ¿Qué es cronguard?

cronguard es una plataforma de observabilidad para cron jobs que va más allá del simple registro de logs. Resuelve problemas críticos como la **deriva temporal** en entornos serverless, la **pérdida de ejecuciones** en pipelines de datos y la **gestión de dependencias** entre tareas asíncronas. A diferencia de soluciones genéricas, cronguard ofrece un motor de reconciliación que compara el estado esperado de las tareas con el estado real, disparando acciones correctivas automáticas.

### Características Principales

- **Motor de Reconciliación de Estados (State Reconciliation Engine):** cronguard mantiene un modelo de estado declarativo para cada cron job. Si una tarea falla o se salta una ejecución, el motor la reencola automáticamente en el próximo slot disponible, evitando la pérdida de datos en pipelines críticos. Esto es esencial para sistemas que dependen de la integridad de eventos, como ETLs financieros o sincronización de bases de datos.
- **Análisis de Deriva Temporal (Time Drift Analysis):** Detecta desviaciones en los tiempos de inicio de las tareas en comparación con el cron schedule original. cronguard calcula una métrica de "salud temporal" (Temporal Health Index) y alerta si la latencia supera un umbral configurable. Esto es vital para entornos con nodos en diferentes zonas horarias o con relojes de sistema inconsistentes.
- **Sistema de Dependencias con Backpressure (Dependency Graph with Backpressure):** Define un grafo acíclico dirigido (DAG) para tus cron jobs. cronguard gestiona la ejecución secuencial de tareas, aplicando backpressure si una tarea upstream se retrasa. Esto evita colapsos en colas de mensajes y garantiza que las tareas downstream no se ejecuten con datos incompletos.

### Cómo Usar

1. **Configuración del Manifiesto de Cron Jobs usando YAML con Validación de Esquema (JSON Schema):**
   Crea un archivo `cronguard.yaml` en la raíz de tu proyecto. Define cada cron job con su expresión cron, el comando a ejecutar (shell, script, o endpoint HTTP) y el contenedor Docker asociado. cronguard valida el esquema automáticamente para evitar errores de sintaxis.
   ```yaml
   version: "2.0"
   jobs:
     - name: "sincronizar_usuarios"
       schedule: "0 */6 * * *"
       command: "python3 /app/sync_users.py"
       container: "my-org/sync-service:1.2.3"
       retry_policy:
         max_attempts: 3
         backoff_factor: 2
       dependencies: []
   ```
   Ejecuta `cronguard init` para registrar el manifiesto en el sistema de monitoreo.

2. **Implementación de Alertas Basadas en Métricas de Salud (Health Metrics):**
   cronguard expone métricas en formato Prometheus. Para configurar una alerta que notifique si un cron job falla más de 2 veces consecutivas, usa el endpoint de alertas integrado:
   ```bash
   cronguard alert add --name "Fallo_Consecutivo" --condition "failure_count > 2" --action "slack_webhook" --webhook-url "https://hooks.slack.com/services/..."
   ```
   También puedes definir alertas basadas en el **Temporal Health Index** (THI). Por ejemplo, si el THI cae por debajo de 0.8, cronguard envía una notificación a PagerDuty.

3. **Ejecución de un Diagnóstico de Dependencias (Dependency Audit):**
   Si tienes un DAG complejo, usa el comando `audit` para visualizar el flujo y detectar cuellos de botella:
   ```bash
   cronguard audit --dag --format=html --output=report.html
   ```
   Esto genera un gráfico interactivo donde cada nodo muestra el tiempo de ejecución promedio, el número de reintentos y el estado actual. cronguard también resalta los caminos críticos (critical paths) que ralentizan todo el pipeline.

### Mejores Prácticas

- **Aplica el Principio de "Fail Fast" con Backpressure:** En entornos de microservicios, no dejes que las colas de mensajes se llenen con tareas fallidas. Configura cronguard para que, ante un error en una tarea upstream, detenga inmediatamente la ejecución de las tareas downstream. Esto se logra definiendo un `timeout` estricto en el manifiesto y usando el sistema de backpressure integrado. Por ejemplo, si la tarea "sincronizar_usuarios" falla, cronguard no ejecutará "generar_reportes" hasta que se resuelva el error, evitando datos inconsistentes.
- **Implementa un Ciclo de Vida de Retención de Logs con Rotación Automática:** cronguard almacena logs de ejecución en un bucket S3 o en un volumen persistente. Para evitar costos de almacenamiento innecesarios, configura una política de retención basada en el tiempo de vida (TTL). Usa el comando `cronguard logs prune --ttl=30d` para eliminar logs de más de 30 días. Además, habilita la rotación automática de logs por tamaño: si un archivo de log supera los 100 MB, cronguard lo comprime y lo archiva. Esto es crucial para equipos que gestionan cientos de cron jobs en producción.
