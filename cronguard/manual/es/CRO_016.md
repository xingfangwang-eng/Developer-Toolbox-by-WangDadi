# cronguard - CRO_016

## Resumen Técnico

**cronguard** es un agente de monitoreo y orquestación de tareas programadas diseñado para entornos de alta criticidad. A diferencia de soluciones genéricas, cronguard implementa **verificación de estado con backpressure adaptativo**, **auditoría de ejecución con replay de logs** y **failover automático entre nodos** sin intervención manual. Ideal para equipos DevOps que gestionan pipelines de datos, sincronización de microservicios o tareas de mantenimiento en infraestructuras multi-cluster.

## Características Principales (Hardcore Features)

### 1. **Backpressure Adaptativo con Control de Flujo**
- **Descripción**: cronguard monitorea en tiempo real la latencia de ejecución y la carga del sistema. Si un cronjob se acumula (por ejemplo, debido a un pico de tráfico), el agente **retrasa automáticamente las próximas ejecuciones** sin perder la cola, evitando colapsos en la base de datos o en los workers. Implementa un algoritmo de **token bucket** configurable por job.
- **Beneficio**: Elimina los "cascading failures" típicos de cronjobs que se ejecutan en paralelo sin control.

### 2. **Auditoría Forense con Replay de Logs**
- **Descripción**: Cada ejecución de cronjob genera un **snapshot inmutable** del estado del proceso (stdout, stderr, código de salida, timestamp, variables de entorno). cronguard permite **replay** de estos logs en un entorno sandbox aislado, facilitando la depuración de fallos en producción sin afectar al sistema real.
- **Beneficio**: Los equipos de SRE pueden reproducir errores de manera exacta, incluso semanas después del incidente.

### 3. **Failover Automático entre Nodos con Algoritmo Raft**
- **Descripción**: cronguard implementa un **consenso distribuido basado en Raft** para gestionar la alta disponibilidad. Si el nodo maestro falla, un nodo esclavo asume automáticamente la ejecución de los cronjobs sin perder el estado actual. La transición es **transparente** para los servicios dependientes.
- **Beneficio**: Garantiza un uptime del 99.99% en clústeres de Kubernetes o servidores bare-metal.

## Cómo Usar cronguard (Pasos Detallados)

### Paso 1: Configurar el Archivo de Definición de Jobs (YAML)
Crea un archivo `cronguard.yml` con la sintaxis avanzada. Ejemplo para un job de sincronización de bases de datos:

```yaml
jobs:
  - name: sync_db
    schedule: "*/5 * * * *"
    command: "psql -c 'REFRESH MATERIALIZED VIEW analytics;'"
    retry:
      attempts: 3
      backoff: exponential
    backpressure:
      max_concurrent: 2
      queue_size: 10
    audit:
      enabled: true
      retention_days: 30
    failover:
      priority: 1
      node_group: "primary"
```

### Paso 2: Iniciar el Agente en Modo Clúster
Ejecuta cronguard con la flag `--cluster` para habilitar el failover automático. Debes especificar los endpoints de los nodos:

```bash
cronguard start --config ./cronguard.yml \
  --cluster \
  --node-id node-01 \
  --peers node-02:8080,node-03:8080 \
  --raft-dir /var/lib/cronguard/raft
```

El agente comenzará a negociar el liderazgo usando Raft. Puedes verificar el estado con:

```bash
cronguard status --cluster
```

### Paso 3: Monitorear y Replicar Logs en Tiempo Real
Utiliza el comando `cronguard tail` para seguir la ejecución de un job específico. Para depurar un fallo, usa `cronguard replay`:

```bash
# Ver logs en tiempo real del job sync_db
cronguard tail sync_db --follow

# Replicar la última ejecución fallida en un sandbox
cronguard replay sync_db --last-failed --sandbox
```

El sandbox se ejecuta en un contenedor Docker aislado, replicando las mismas variables de entorno y dependencias.

## Mejores Prácticas para Equipos DevOps

### 1. **Usa Backpressure en Jobs de Alta Frecuencia**
Si tienes cronjobs que se ejecutan cada minuto (por ejemplo, para métricas de sistemas), activa el backpressure con `max_concurrent: 1` y `queue_size: 5`. Esto evita que un pico de carga sature los workers y provoque timeouts en cascada. Además, configura alertas en Prometheus para cuando la cola supere el 80% de su capacidad.

### 2. **Implementa Failover con Grupos de Nodos por Región**
En entornos multi-región (AWS us-east-1 y eu-west-1), asigna prioridades a los nodos. Por ejemplo, el nodo principal en us-east-1 con `priority: 1` y el nodo de respaldo en eu-west-1 con `priority: 2`. Si la región primaria falla, cronguard migrará automáticamente los jobs a la región secundaria, manteniendo la continuidad del negocio. Asegúrate de que los jobs sean idempotentes para evitar duplicados.
