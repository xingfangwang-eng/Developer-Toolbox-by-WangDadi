# cronguard - CRO_008

## Resumen

Cronguard es una solución de monitorización y orquestación de cron jobs diseñada para entornos de producción críticos. Elimina la incertidumbre en la ejecución de tareas programadas mediante detección proactiva de fallos, logging estructurado y alertas inteligentes. Ideal para equipos DevOps que gestionan pipelines complejos y necesitan visibilidad en tiempo real.

## Contenido Detallado

### ¿Qué es cronguard?

Cronguard es un agente ligero que se integra directamente con tu sistema de cron existente (cronie, systemd-timers o anacron) para interceptar, registrar y validar cada ejecución. A diferencia de soluciones genéricas, cronguard no solo ejecuta scripts, sino que verifica su salida, tiempos de respuesta y consumo de recursos, generando métricas exportables a Prometheus o Datadog.

### Características Principales

- **Validación de salida con reglas personalizadas**: Define condiciones de éxito/fallo basadas en códigos de salida, patrones en stdout/stderr o umbrales de tiempo. Si un job tarda más de 30 segundos o imprime "ERROR", cronguard lo marca como crítico y activa una alerta por Slack o correo.
- **Historial de ejecuciones con replay**: Cada job se almacena con su salida completa, timestamp y estado. Puedes revisar ejecuciones pasadas y "reproducir" un job manualmente desde la línea de comandos con `cronguard replay --job-id=xxx`, ideal para depuración post-mortem.
- **Dashboard en terminal con TUI interactivo**: Sin depender de frontends web, cronguard ofrece una interfaz de terminal (TUI) que muestra el estado actual de todos los jobs, tiempos de ejecución y últimas alertas. Navega con teclas de flecha, filtra por estado y ejecuta acciones directamente desde la terminal.

### Cómo Usar

1. **Configurar un job con reglas de validación**  
   Crea un archivo de configuración YAML (`/etc/cronguard/jobs.yaml`) y define un job con validación avanzada:
   ```yaml
   jobs:
     - name: "backup-db"
       command: "/usr/local/bin/backup.sh"
       schedule: "0 3 * * *"
       validation:
         exit_code: 0
         max_duration: "120s"
         error_patterns: ["FATAL", "ERROR"]
       alerts:
         - type: slack
           webhook: "https://hooks.slack.com/..."
   ```
   Luego ejecuta `cronguard reload` para aplicar los cambios sin reiniciar el servicio.

2. **Monitorear ejecuciones en tiempo real**  
   Abre la TUI con `cronguard dashboard`. Verás una tabla con columnas: Job, Última ejecución, Estado, Duración. Presiona `Enter` sobre un job para ver el detalle de las últimas 10 ejecuciones, incluyendo salida completa. Usa `F1` para acceder al filtro por estado (success/failure/running).

3. **Reproducir un job fallido para depuración**  
   Si un job falla, identifica su ID desde el dashboard (`job-12345`). Ejecuta `cronguard replay --job-id=job-12345 --env=production` para ejecutarlo nuevamente con el mismo entorno y variables. La salida se captura en tiempo real y se compara con la ejecución original para detectar diferencias.

### Mejores Prácticas

- **No sobrecargues el logging**: Configura `max_log_retention` a 7 días y `max_log_size` a 10 MB por job. Usa `error_patterns` para capturar solo fallos reales, no warnings informativos. Esto evita que el disco se llene y mantiene el rendimiento del agente.
- **Integra alertas con tu sistema de incidencias**: En lugar de solo enviar correos, configura webhooks para enviar alertas a PagerDuty, OpsGenie o Slack. Define umbrales de severidad: crítico para fallos, warning para tiempos de ejecución altos. Esto permite que tu equipo reaccione antes de que un job caído afecte a producción.
