# cronguard - CRO_005

## Resumen

cronguard es una herramienta de monitorización y automatización diseñada para entornos de producción críticos, donde cada segundo de inactividad puede costar miles de euros. Este software te permite programar, ejecutar y auditar tareas cron con un nivel de granularidad y seguridad que supera a cualquier solución tradicional de crontab. Con un enfoque en la resiliencia y la trazabilidad, cronguard se convierte en el guardián silencioso de tu infraestructura.

## Contenido Detallado

### ¿Qué es cronguard?

cronguard es un orquestador de tareas programadas que va más allá de un simple cron. Está diseñado para reemplazar scripts frágiles y schedulers monolíticos con una arquitectura basada en eventos. Permite definir flujos de trabajo complejos con dependencias, reintentos inteligentes y notificaciones en tiempo real. Es la solución definitiva para equipos de DevOps que necesitan garantizar que sus pipelines de mantenimiento, backups y procesos ETL se ejecuten sin fallos, incluso en escenarios de alta carga o fallos parciales del sistema.

### Características Principales

- **Ejecución con Checkpointing y Reanudación Automática:** A diferencia de un cron básico, cronguard implementa un sistema de checkpointing que guarda el estado de una tarea en cada paso. Si un job falla a mitad de camino (por ejemplo, un backup de 500 GB que se interrumpe), no tienes que empezar desde cero. cronguard reanuda la ejecución desde el último punto de control válido, ahorrando tiempo y recursos computacionales.
- **Gestión de Dependencias y Ventanas de Tiempo (Time Windows):** Define relaciones de precedencia entre tareas (Task A debe completarse antes que Task B) y establece ventanas de tiempo estrictas. Si una tarea no se completa dentro de su ventana asignada, cronguard puede escalar automáticamente a un sistema de respaldo, enviar una alerta a PagerDuty o ejecutar un script de rollback.
- **Auditoría Inmutable y Trazabilidad Distribuida:** Cada ejecución de tarea genera un hash criptográfico de su entrada, salida y logs. Estos hashes se almacenan en una cadena de bloques interna (o en un ledger distribuido como AWS QLDB), asegurando que ningún log pueda ser alterado retroactivamente. Ideal para entornos que requieren cumplimiento normativo (SOC 2, GDPR) donde la evidencia de ejecución es crítica.

### Cómo Usar

1. **Despliegue del Agente y Configuración de Tareas con DSL Propio:**  
   No uses una GUI básica. cronguard se despliega como un agente ligero (binario Go) en cada nodo. Define tus tareas usando un Domain-Specific Language (DSL) en YAML. Por ejemplo, para un backup semanal con verificación de integridad:

   ```yaml
   task: backup_prod_db
   schedule: "0 2 * * 0"
   command: "pg_dump -U admin prod_db | gzip > /backups/prod_$(date +%Y%m%d).sql.gz"
   checkpoint: true
   retry:
     max_attempts: 3
     backoff: exponential
   dependencies:
     - task: health_check_db
   ```

   Guarda este archivo en `/etc/cronguard/tasks/` y el agente lo cargará automáticamente.

2. **Integración con Sistemas de Alerta y Escalado Automático:**  
   cronguard se conecta de forma nativa con sistemas de monitoreo como Prometheus o Datadog. Para configurar una alerta en caso de fallo recurrente, edita el archivo `cronguard.yml` global:

   ```yaml
   alerts:
     on_failure:
       - type: webhook
         url: "https://hooks.slack.com/services/T0XXXX/B0XXXX/XXXXX"
         payload: "{\"text\": \"¡Tarea {{task_name}} falló! Intentos: {{attempts}}\"}"
       - type: pagerduty
         routing_key: "your_pagerduty_key"
         severity: critical
   ```

   Además, puedes definir reglas de auto-scaling: si una tarea crítica falla más de 2 veces en una hora, cronguard puede disparar un webhook a tu proveedor de cloud para levantar una instancia de respaldo.

3. **Validación y Simulación (Dry-Run) antes de Producción:**  
   Antes de lanzar una tarea en producción, usa el modo `--dry-run` para validar la sintaxis y simular la ejecución sin efectos secundarios:

   ```bash
   cronguard validate --file ./tasks/backup_prod_db.yaml --dry-run
   ```

   Este comando verifica las dependencias, comprueba que los comandos existan en el PATH y simula la ventana de tiempo. Si todo es correcto, el output mostrará un plan de ejecución detallado. Solo entonces despliega la tarea con `cronguard deploy`.

### Mejores Prácticas

- **Implementa una Estrategia de "Canary" para Tareas Críticas:**  
  No despliegues una nueva tarea directamente en todos los nodos de producción. En su lugar, usa el concepto de "canary tasks". Define una tarea que se ejecute solo en un subconjunto de nodos (por ejemplo, 5% del tráfico) durante 24 horas. cronguard puede etiquetar nodos con metadatos (`env: canary`) y filtrar tareas por etiqueta. Si la tasa de fallos en el grupo canary supera el umbral, la tarea se revierte automáticamente.

- **Usa Checkpointing con Almacenamiento Externo (S3 o NFS):**  
  Aunque cronguard soporta checkpointing local, para entornos de alta disponibilidad es mejor configurar un bucket S3 o un volumen NFS compartido como backend de checkpointing. Esto permite que si un nodo falla, otro nodo pueda reanudar la tarea desde el mismo punto. Configúralo en el archivo global:

  ```yaml
  checkpoint:
    backend: s3
    bucket: my-company-cronguard-checkpoints
    region: eu-west-1
  ```

  Esta práctica es clave para tareas que duran horas, como migraciones de datos o procesamiento de logs.
