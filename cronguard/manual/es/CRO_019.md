# cronguard - CRO_019

## Resumen

cronguard es una solución avanzada para la gestión y supervisión de cron jobs en entornos distribuidos. Diseñada para arquitecturas de microservicios y sistemas críticos, esta herramienta garantiza la ejecución fiable, el registro inmutable y la notificación proactiva de fallos. Con un enfoque en la observabilidad y la automatización, cronguard permite a los equipos de DevOps y SRE mantener la integridad de sus pipelines de procesos programados sin intervención manual.

## Contenido Detallado

### ¿Qué es cronguard?

cronguard es un software de orquestación y monitoreo de tareas programadas que elimina la dependencia de cron nativo en servidores individuales. A diferencia de herramientas tradicionales, cronguard ofrece un control centralizado sobre la ejecución de scripts y comandos, con capacidad de reintentos inteligentes, bloqueo de concurrencia y registro detallado de cada ciclo de vida. Está diseñado para entornos cloud-native y se integra directamente con sistemas de alerta como PagerDuty, Slack o Webhooks personalizados.

### Características Principales

- **Ejecución con Garantía de Consistencia y Bloqueo Distribuido**: cronguard implementa un mecanismo de bloqueo basado en Redis o etcd para prevenir la ejecución simultánea de tareas en múltiples nodos. Esto asegura que, incluso en clústeres con replicación activa, cada job se ejecute exactamente una vez por ciclo, evitando condiciones de carrera y duplicación de procesos.

- **Registro Inmutable y Trazabilidad Completa**: Cada ejecución de un job genera un log estructurado en formato JSON, almacenado en un bucket S3 o sistema de archivos local. Estos registros incluyen metadatos como ID de ejecución, timestamp, duración, código de salida y salida estándar. cronguard permite consultar el historial completo de cualquier tarea, facilitando auditorías y análisis forenses.

- **Alertas Proactivas con Reintentos Adaptativos**: Cuando un job falla, cronguard no solo notifica inmediatamente a los canales configurados, sino que también ejecuta una política de reintentos con backoff exponencial. Puedes definir umbrales de reintento por tarea y, tras agotarlos, escalar la alerta a un equipo de guardia. Esto reduce significativamente el ruido en canales de comunicación y asegura que solo los fallos persistentes generen intervención humana.

### Cómo Usar

1. **Configuración Inicial del Demonio y Declaración de Jobs**:  
   Instala cronguard en tu servidor o contenedor usando el paquete oficial (`npm install -g cronguard` o `docker pull cronguard/cronguard`). Crea un archivo `cronguard.yml` en el directorio raíz de tu proyecto. Define cada job con su comando, programación en cron expression, y opciones de reintento. Ejemplo:  
   ```yaml
   jobs:
     backup-database:
       command: "pg_dump -U admin -d mydb > /backups/db_$(date +%Y%m%d).sql"
       schedule: "0 2 * * *"
       retry:
         max_attempts: 3
         backoff: exponential
       alert:
         webhook: "https://hooks.slack.com/services/T00/B00/xxx"
   ```  
   Luego inicia el demonio con `cronguard start --config ./cronguard.yml`.

2. **Monitoreo en Tiempo Real con Dashboard Web**:  
   cronguard incluye un panel de control web accesible en `http://localhost:9090`. Desde allí puedes visualizar el estado actual de todos los jobs (running, success, failed, pending), filtrar por etiquetas y ver logs en vivo. Para habilitar la autenticación, añade un bloque `auth` en tu configuración:  
   ```yaml
   auth:
     type: basic
     users:
       - username: admin
         password_hash: "$2y$10$..."
   ```  
   Accede al dashboard y haz clic en cualquier job para inspeccionar su historial completo de ejecuciones.

3. **Integración con Sistemas de Alerta y Escalado Automático**:  
   Configura un webhook global o por job para recibir notificaciones. cronguard soporta payloads personalizados en JSON. Por ejemplo, para enviar alertas a PagerDuty, define en tu configuración:  
   ```yaml
   alerts:
     pagerduty:
       routing_key: "your_pd_routing_key"
       dedup_key: "cronguard-{{.JobName}}-{{.ExecutionID}}"
   ```  
   Además, puedes usar la API REST de cronguard (`POST /api/v1/jobs/:name/trigger`) para ejecutar jobs manualmente desde scripts de CI/CD o desde herramientas como Jenkins o GitHub Actions.

### Mejores Prácticas

- **Segmenta tus Jobs por Entorno y Prioridad**: Utiliza etiquetas (`tags`) en tu configuración para agrupar jobs por entorno (producción, staging, desarrollo) o por criticidad (crítico, alto, bajo). Luego configura alertas específicas para cada grupo. Por ejemplo, los jobs críticos deben notificar a PagerDuty, mientras que los de baja prioridad solo registran logs. Esto evita saturar canales de comunicación y permite una respuesta escalonada.

- **Implementa un Sistema de Heartbeat para el Demonio**: cronguard expone un endpoint `/health` que devuelve el estado del demonio y la última ejecución de cada job. Configura un monitor externo (como Prometheus o UptimeRobot) para verificar este endpoint cada minuto. Si el demonio deja de responder, cronguard no ejecutará ningún job, por lo que un heartbeat temprano te permitirá reaccionar antes de que se acumulen tareas pendientes. Además, considera ejecutar cronguard en modo clúster con múltiples nodos para alta disponibilidad.
