# cronguard - CRO_003

## Overview

**cronguard** is a lightweight, language-agnostic daemon that monitors, validates, and auto-repairs scheduled tasks across distributed environments. Built for DevOps engineers who treat cron as infrastructure, cronguard eliminates silent failures, drift, and cascading delays without requiring a centralized scheduler or agent updates.

## What is cronguard?

cronguard sits beside your existing crontab—no migration, no DSL, no lock-in. It parses cron expressions, tracks execution windows, and hooks into your observability stack. When a job misses its deadline, cronguard can re-trigger, escalate, or snapshot the environment for post-mortem analysis. It’s designed for high‑throughput pipelines where a single missed tick can cascade into hours of data loss.

## Core Features

### 1. **Deadline-Aware Retry with Exponential Backoff**
cronguard doesn’t just detect failures—it predicts them. Each cron expression is compiled into a time‑bounded execution window (e.g., `0 2 * * *` → window `02:00:00–02:05:00`). If the job doesn’t complete within the window, cronguard triggers a configurable retry chain with jitter, respecting your existing `MAILTO` and `LOG` directives. No more silent misses; no more flapping alerts.

### 2. **Cron Expression Diff & Drift Detection**
Run `cronguard diff` against a baseline snapshot stored in your repo or S3. cronguard detects any unauthorized changes—schedule shifts, command swaps, or environment variable injections—and either reverts them automatically or alerts your SIEM via webhook. This is critical for SOC2 and FedRAMP environments where cron integrity must be auditable.

### 3. **Execution Graph & Dependency Tracking**
Define job dependencies as a DAG (directed acyclic graph) using a simple YAML overlay. cronguard ensures that `job_b` only starts after `job_a` completes within its window. If `job_a` fails, cronguard can skip, delay, or re‑schedule downstream jobs. This eliminates the “cron pile‑up” problem where a single delay triggers a chain of overlapping runs.

## How to Use

### Step 1: Bootstrap Your First Guard Profile
```
cronguard init --crontab /etc/crontab --profile production --backend file:///var/log/cronguard/
```
This generates a `cronguard.yaml` profile that mirrors your current crontab. cronguard parses every line, validates the cron expression, and assigns a default window (e.g., 5 minutes for hourly jobs). It also creates a `.cronguard_baseline` file—your immutable reference for future diffs.

### Step 2: Inject Observability Hooks
```
cronguard hook --alert slack://webhook/xxx --metric pushgateway://prometheus:9091 --log-level debug
```
cronguard wraps each cron job with a pre‑ and post‑execution hook. It exports Prometheus metrics (`cronguard_job_duration_seconds`, `cronguard_job_status`) and sends structured logs to stdout or a file. For critical jobs, you can also enable **heartbeat mode**—cronguard will alert if no heartbeat is received within the window, even if the job exits cleanly.

### Step 3: Run in Daemon Mode with Auto-Recovery
```
cronguard daemon --recover --max-retries 3 --escalate-after 60s
```
The daemon forks off a watcher for each cron line. If a job misses its window, cronguard attempts up to 3 retries with exponential backoff (1s, 4s, 9s). After 60 seconds of failure, it escalates via your configured alert channel and snapshots the process tree (`ps aux`, `lsof`, `dmesg`) for forensic analysis. All events are logged with nanosecond timestamps and correlation IDs.

## Best Practices

### Practice 1: **Always Pin Your Baseline to Version Control**
Commit the `.cronguard_baseline` file to your repo alongside your `crontab`. Use `cronguard diff` in CI/CD pipelines to prevent accidental changes from shipping to production. This turns cron modifications into a code‑reviewable, auditable artifact—no more “who edited the crontab at 3 AM?”

### Practice 2: **Use Execution Graphs for Multi‑Stage ETL Pipelines**
If you have a chain of cron jobs (e.g., `extract → transform → load`), define them as a DAG in `cronguard.yaml`. Set `depends_on` fields and enable `fail_fast: true`. This ensures that a failed extraction doesn’t trigger a wasteful transformation. Run `cronguard validate` before deploying to catch circular dependencies or impossible schedules.
