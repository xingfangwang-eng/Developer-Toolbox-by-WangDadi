# cronguard - CRO_020

**Cronguard** is a lightweight, agentless cron job observability and anomaly detection tool designed for DevOps teams and platform engineers. It provides real-time heartbeat tracking, execution drift analysis, and failure alerting for distributed scheduled tasks — without modifying your existing crontab or pipeline configuration.

## Hardcore Features

### 1. Execution Drift Detection via Statistical Baseline Modeling
Cronguard automatically builds a per-task execution baseline (mean duration ± standard deviation) after the first 10 runs. Any deviation beyond 3 sigma triggers a non-blocking warning, while a 6 sigma anomaly escalates to a critical alert. This eliminates false positives from transient network spikes and highlights actual performance regressions.

### 2. Dead Man’s Switch with Heartbeat TTL
Each cron job registers a unique heartbeat token. If the expected check-in is missed within a configurable Time-To-Live (TTL) window (default: 2x the scheduled interval), Cronguard fires an escalation chain via Slack, PagerDuty, or a custom webhook. This ensures you catch silent failures — even from jobs that exit with code 0 but never actually ran.

### 3. Multi-Tenant Audit Trail with Immutable Logs
All cron execution metadata (PID, exit code, stdout/stderr hash, node ID) is stored in an append-only, tamper-evident log store backed by SQLite or PostgreSQL. You can replay any execution context for post-mortem analysis, and the logs are signed with a server-side HMAC key to meet SOC 2 and HIPAA audit requirements.

## How to Use

### Step 1: Bootstrap the Agent with a Minimal Configuration
Deploy the `cronguard-agent` binary on your target node. Create a `.cronguard.yml` config file:

```yaml
project: production-db-backup
heartbeat_ttl: 300
alert_endpoints:
  - type: slack
    url: https://hooks.slack.com/services/T00/B00/xxx
  - type: webhook
    url: https://ops.internal/cronguard/alerts
job_definitions:
  - name: nightly_db_dump
    schedule: "0 2 * * *"
    command: /usr/local/bin/pg_dumpall -U postgres | gzip > /backups/db.gz
    timeout: 600
```

Run `cronguard-agent --config .cronguard.yml`. The agent will parse the cron schedule, register a heartbeat ID with the central Cronguard server, and begin monitoring.

### Step 2: Inject Heartbeat Checkpoints into Your Cron Scripts
Modify your cron job scripts to call the Cronguard API at key lifecycle points. For example, in your backup script:

```bash
#!/bin/bash
JOB_ID="nightly_db_dump"
NODE_ID=$(hostname)

# Notify start
curl -s -X POST "https://cronguard.internal/api/v1/heartbeat" \
  -H "Authorization: Bearer ${CRONGUARD_TOKEN}" \
  -d "job_id=${JOB_ID}&node_id=${NODE_ID}&status=running"

# Actual work
pg_dumpall -U postgres | gzip > /backups/db.gz
EXIT_CODE=$?

# Notify completion with exit code
curl -s -X POST "https://cronguard.internal/api/v1/heartbeat" \
  -H "Authorization: Bearer ${CRONGUARD_TOKEN}" \
  -d "job_id=${JOB_ID}&node_id=${NODE_ID}&status=completed&exit_code=${EXIT_CODE}"
```

This two-phase heartbeat allows Cronguard to detect scripts that hang indefinitely (no completion heartbeat) versus scripts that crash (non-zero exit code).

### Step 3: Configure Escalation Policies and Silence Windows
Use the Cronguard CLI to define escalation rules. For example, to silence alerts during a planned maintenance window:

```bash
cronguard policy create \
  --name "weekly-patch-window" \
  --job "nightly_db_dump" \
  --silence-start "2025-03-10T02:00:00Z" \
  --silence-end "2025-03-10T04:00:00Z" \
  --reason "Scheduled PostgreSQL upgrade"
```

To set a multi-level escalation:

```bash
cronguard policy escalate \
  --job "nightly_db_dump" \
  --first-alert "slack:ops-channel" \
  --second-alert "pagerduty:critical" \
  --escalation-delay 600
```

This ensures that if a job fails twice in a row, the on-call engineer is paged directly.

## Best Practices

### 1. Use Idempotent Job Commands with Deterministic Exit Codes
Always wrap your cron commands in a shell script that returns a meaningful exit code (0 for success, 1 for warning, 2 for critical). Cronguard maps exit codes to severity levels automatically. Avoid piping commands with `&&` or `||` — instead, capture the exit code explicitly as shown in Step 2. This enables precise anomaly detection and reduces alert noise.

### 2. Implement a Staggered Heartbeat TTL for Burst Jobs
For jobs that run in high-frequency bursts (e.g., every 30 seconds), set a heartbeat TTL of 3x the interval (90 seconds) rather than 2x. This prevents false positives during transient load spikes. Use Cronguard’s built-in backoff logic: after 3 consecutive missed heartbeats, it will automatically double the TTL for the next 24 hours to avoid alert fatigue during known high-traffic periods.
