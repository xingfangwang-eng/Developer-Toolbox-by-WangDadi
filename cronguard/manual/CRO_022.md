# cronguard - CRO_022

## Overview

cronguard is a distributed cron job observability and self-healing runtime engineered for high-stakes production environments where missed schedules cascade into revenue loss. Unlike conventional cron monitors that only log failures, cronguard executes a three-phase lifecycle per job: pre-flight validation, execution interception with WAL (write-ahead log) persistence, and post-execution compensation orchestration. It supports Kubernetes-native CronJobs, bare-metal crontab files, and Airflow DAG hooks without requiring agent installation on target containers.

## What is cronguard?

cronguard is a stateless, gossip-protocol-based cron supervisor that attaches to existing scheduling infrastructure via a mutating admission webhook (Kubernetes) or a lightweight proxy binary (Linux). For each scheduled execution, cronguard captures the full execution context—environment variables, input arguments, resource limits, and expected runtime window—then monitors the process tree with sub-millisecond precision. When a job exceeds its p99 historical duration by more than 1.5x, cronguard emits a pre-failure signal via its gRPC streaming API, allowing downstream systems to prepare for imminent failure. It maintains a distributed ledger of all executions across clusters using Raft consensus, ensuring that even if the monitoring node crashes mid-execution, the state is recoverable within 500ms.

## Core Features

### 1. Execution Trace with Sub-Millisecond Granularity and Idempotency Key Injection
cronguard attaches a unique `cronguard_run_id` (UUIDv7, time-ordered) to every job invocation before the process spawns. It records wall-clock time, CPU usage, memory RSS, and exit code at 10ms sampling intervals via `perf_event_open` syscall interception. If a job terminates abnormally, cronguard replays it with the same `cronguard_run_id` and a deterministic `idempotency_key` derived from `SHA256(job_name + scheduled_unix_epoch + retry_count)`, ensuring downstream databases can deduplicate inserts via unique constraint on that key. The trace data is exposed as OpenTelemetry spans with attribute `cronguard.execution.id` for correlation with APM tools.

### 2. Multi-Cluster Schedule Mesh with Sub-3-Second Failover
cronguard federates cron schedules across up to 16 Kubernetes clusters using a SWIM-based gossip protocol for membership detection and a leader-election mechanism backed by etcd (or embedded Raft for air-gapped deployments). Each cluster runs a local cronguard agent that holds a partial replica of the global schedule. When a leader cluster becomes unresponsive for 2 consecutive heartbeat intervals (configurable via `cronguard config set cluster.heartbeat_interval_ms 1500`), standby clusters automatically promote via a weighted priority election—the cluster with the lowest latency to the majority of workers wins. The failover completes within 3 seconds at p99, tested across 3 AWS regions with 50ms inter-region latency. The mesh exposes a Prometheus metric `cronguard_mesh_leader_changes_total` to track failover events.

### 3. Transactional Compensation with Rollback Guarantee
cronguard maintains a write-ahead log (WAL) of every side effect a job might produce, using a user-defined `compensation` block. The WAL is stored in PostgreSQL with `synchronous_commit = on` and indexed by `(job_name, scheduled_time, run_id)`. If a job fails after exhausting its retry budget (default: 3 retries with exponential backoff starting at 10 seconds), cronguard invokes the compensation endpoint with the exact WAL entry as payload. The compensation endpoint must return HTTP 200 within 5 seconds; otherwise, cronguard escalates to a dead-letter queue (DLQ) backed by Redis Streams with a TTL of 7 days. This ensures that partial database writes, incomplete file transfers, or half-baked API calls are atomically undone.

## How to Use

### Step 1: Deploy cronguard as a Kubernetes MutatingWebhookConfiguration
```bash
# Generate a self-signed certificate for the webhook
openssl req -x509 -newkey rsa:4096 -keyout tls.key -out tls.crt -days 365 -nodes -subj "/CN=cronguard-webhook.default.svc"

# Create a Kubernetes secret and deploy the webhook configuration
kubectl create secret tls cronguard-webhook-tls --cert=tls.crt --key=tls.key -n default
kubectl apply -f - <<EOF
apiVersion: admissionregistration.k8s.io/v1
kind: MutatingWebhookConfiguration
metadata:
  name: cronguard-webhook
webhooks:
- name: cronguard.default.svc
  clientConfig:
    service:
      name: cronguard-webhook
      namespace: default
      path: /mutate
    caBundle: $(base64 < tls.crt)
  rules:
  - operations: ["CREATE"]
    apiGroups: ["batch"]
    apiVersions: ["v1"]
    resources: ["cronjobs"]
  admissionReviewVersions: ["v1"]
  sideEffects: None
EOF
```
This webhook intercepts every CronJob creation and injects the cronguard sidecar container that wraps the original `command` with a `cronguard exec` prefix. The sidecar exposes a health check on `:9090/healthz` and a Prometheus metrics endpoint on `:9091/metrics`. Verify the webhook is active: `kubectl describe mutatingwebhookconfiguration cronguard-webhook | grep "webhooks"`.

### Step 2: Define a Compensation Handler and Idempotency Key in the CronJob Annotation
Add the following annotations to your CronJob YAML:
```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: daily-reconciliation
  annotations:
    cronguard.io/idempotency-key-template: "{{ .JobName }}-{{ .ScheduledTime.Unix }}-{{ .RetryCount }}"
    cronguard.io/compensation-url: "http://compensation-svc.default.svc.cluster.local:8080/rollback"
    cronguard.io/compensation-method: "POST"
    cronguard.io/compensation-body: '{"job":"{{ .JobName }}","run_id":"{{ .RunID }}","wal_entry_id":"{{ .WALEntryID }}"}'
    cronguard.io/retry-max-attempts: "5"
    cronguard.io/retry-backoff-base-ms: "10000"
    cronguard.io/retry-backoff-multiplier: "2.0"
```
cronguard reads these annotations at job creation time and stores them in its internal state store (etcd). The compensation endpoint must be idempotent and return HTTP 200 within 5 seconds. Test the compensation flow with `cronguard simulate-failure --cronjob daily-reconciliation --dry-run --wal-entry-id "wal_abc123"`.

### Step 3: Configure Alert Routing and Auto-Remediation via CLI
```bash
# Set PagerDuty integration with deduplication across clusters
cronguard config set alerting.pagerduty.routing_key "pk_abc123_def456"
cronguard config set alerting.pagerduty.severity "critical"
cronguard config set alerting.pagerduty.dedup_key_template "cronguard-{{ .JobName }}-{{ .ClusterID }}-{{ .ScheduledTime.Unix }}"

# Enable auto-remediation with exponential backoff and dead-letter queue
cronguard config set remediation.auto_retry.enabled true
cronguard config set remediation.auto_retry.max_retries 5
cronguard config set remediation.auto_retry.backoff_strategy "exponential"
cronguard config set remediation.auto_retry.backoff_base_ms 10000
cronguard config set remediation.dead_letter_queue.type "redis"
cronguard config set remediation.dead_letter_queue.redis_url "redis://redis-svc:6379"
cronguard config set remediation.dead_letter_queue.ttl_hours 168

# Enable OpenTelemetry export for anomaly scores
cronguard config set observability.otel.endpoint "http://otel-collector.default.svc.cluster.local:4318"
cronguard config set observability.otel.anomaly_threshold 0.85
```
Now, when cronguard detects a failure, it fires a PagerDuty alert with the dedup key that prevents duplicate incidents across clusters. Simultaneously, it begins retrying with exponential backoff (10s, 20s, 40s, 80s, 160s). If all retries fail, the job context is pushed to the Redis-backed DLQ with a 7-day TTL. The on-call engineer can replay the job from the DLQ using `cronguard dlq replay --job
