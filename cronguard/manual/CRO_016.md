# cronguard - CRO_016

## Overview

cronguard is a battle-tested cron job monitoring and anomaly detection toolkit engineered for DevOps teams managing distributed scheduling pipelines. It eliminates silent failures, drift detection gaps, and alert fatigue in production-grade cron environments. Built for engineers who demand deterministic execution guarantees across multi-region, multi-timezone infrastructures.

## Hardcore Features

### 1. **Deterministic Execution Watermarking**
cronguard injects cryptographically signed execution watermarks into every cron run. These watermarks are validated against a distributed ledger (backed by etcd or Redis Streams) to detect missed, duplicated, or misordered runs—even across node restarts or clock skews. No more guessing if a job actually ran.

### 2. **Adaptive Anomaly Detection with Drift Compensation**
Leveraging a sliding-window statistical model, cronguard learns the typical execution time, resource footprint, and exit code distribution for each cron job. It then flags deviations (e.g., a job that normally takes 2s suddenly spiking to 30s) and automatically triggers compensatory actions—like retrying with backoff or escalating to PagerDuty—without manual threshold tuning.

### 3. **Multi-Tenant Audit Trail with Immutable Logging**
Every cron event is recorded in an append-only, SHA-256 chained log (compatible with AWS CloudTrail or GCP Audit Logs). This gives you a tamper-proof, searchable history of all job executions, config changes, and alert triggers. Perfect for SOC 2 or FedRAMP audits.

## Detailed Usage Steps

### Step 1: Initialize the Guard with a Custom Watermark Policy
```bash
cronguard init --config /etc/cronguard/config.yaml \
  --watermark-algorithm sha256 \
  --ledger-backend redis://cronguard-redis:6379 \
  --drift-sensitivity high
```
This creates a local `.cronguard` state directory and registers the watermarking scheme. The `--drift-sensitivity high` flag enables aggressive anomaly detection for latency-critical jobs.

### Step 2: Attach the Guard to an Existing Cron Job
Wrap your cron command with `cronguard execute` to inject monitoring:
```bash
0 3 * * * /usr/local/bin/cronguard execute --job-id backup_db \
  --timeout 300 \
  --retry-policy exponential \
  --on-failure "slack://ops-channel?message=backup_failed" \
  -- /opt/scripts/db_backup.sh
```
cronguard will capture stdout/stderr, measure execution duration, and compare against the learned baseline. If the job exceeds `--timeout` or exits with a non-zero code, it triggers the retry policy and sends the alert.

### Step 3: Query the Audit Trail for Post-Mortem Analysis
```bash
cronguard audit --job-id backup_db --from "2025-01-15T00:00:00Z" \
  --to "2025-01-15T23:59:59Z" \
  --filter "exit_code!=0" \
  --format json | jq '.events[] | {id, timestamp, watermark, duration_ms, error}'
```
This returns a JSON array of all non-zero exit events within the timeframe, including the cryptographic watermark and exact duration. You can pipe this into your incident management dashboard for root cause analysis.

## Industry Best Practices

### 1. **Integrate with Your Existing Observability Stack**
Don’t treat cronguard as a silo. Configure it to emit OpenTelemetry spans and Prometheus metrics (`cronguard_job_duration_seconds`, `cronguard_watermark_mismatch_total`). This allows you to correlate cron health with other system signals in Grafana or Datadog, enabling proactive capacity planning.

### 2. **Implement a Graceful Degradation Strategy**
Set up a fallback cron guard on a secondary region or cloud provider. If the primary cronguard instance goes down, the secondary automatically promotes itself and replays the last 10 minutes of missed watermarks. Use `cronguard failover --standby <backup_region>` to enable this—ensuring your critical jobs never go unmonitored.
