# cronguard - CRO_002

**cronguard** is a lightweight, battle-tested daemon that watches your cron jobs, detects silent failures, and auto-remediates them before they turn into production incidents. Designed for DevOps teams who refuse to let a missed schedule bring down their pipeline.

---

## 🔥 Hardcore Features

### 1. **Anomaly-Aware Execution Profiler**
cronguard builds a behavioral baseline for every cron job (execution duration, exit codes, output size). When a job deviates beyond 3σ from its historical profile—e.g., finishes 5x faster or emits zero output—it flags the run as suspicious and triggers a pre-configured rollback or retry logic *before* the next scheduled tick.

### 2. **Self-Healing Retry with Exponential Backoff & Circuit Breaker**
Unlike naive retry loops, cronguard implements a **circuit-breaker pattern** per job. If a job fails 3 consecutive times within a sliding 5-minute window, the circuit opens—halting further retries and escalating to your PagerDuty/Slack webhook. Once the circuit resets (configurable TTL), it resumes with a clean state. No infinite loops, no alert fatigue.

### 3. **Declarative CronGuardfile with YAML + Jinja2 Templating**
Define your entire cron ecosystem in a single `CronGuardfile` using YAML syntax with Jinja2 template variables. Inject environment-specific values (e.g., `{{ env.DB_HOST }}`) without touching the runtime config. Supports multi-stage validation (lint → dry-run → apply) to catch syntax errors before they hit production.

## 🛠️ How to Use (3-Step Expert Workflow)

### Step 1: Bootstrap the CronGuardfile with Auto-Discovery
```bash
cronguard init --scan /etc/cron.d --output ./CronGuardfile
```
This scans all existing cron entries, extracts their schedules, commands, and shell environments, then generates a fully annotated `CronGuardfile`. Each job gets a unique UUID, a default retry policy (3 retries, 10s backoff), and a placeholder for the anomaly profile. Review and tweak thresholds in `./CronGuardfile` before moving to the next step.

### Step 2: Validate & Apply with Dry-Run Mode
```bash
cronguard validate --file ./CronGuardfile --dry-run
```
This parses the entire file, checks for circular dependencies (e.g., job A depends on job B which depends on job A), validates cron expressions against the `croniter` library, and simulates the next 10 execution cycles. Outputs a JSON report showing which jobs would fire, their dependencies, and any potential overlaps. Only after a clean dry-run should you apply:
```bash
cronguard apply --file ./CronGuardfile --daemonize
```

### Step 3: Watch Real-Time Metrics via Embedded HTTP Endpoint
```bash
cronguard serve --port 9090 --metrics
```
cronguard exposes a Prometheus-compatible `/metrics` endpoint (default port 9090) showing per-job execution count, failure rate, retry attempts, and circuit state. Integrate with Grafana for dashboards or set up alerting rules directly in Prometheus:
```yaml
# prometheus.yml
alerting:
  alertmanagers:
    - static_configs:
        - targets: ['localhost:9090']
```
Use `cronguard status --json` for a CLI snapshot of all job states.

## 💡 Industry Best Practices

### 1. **Always Set a `max_retries` with Exponential Backoff**
Never use infinite retries—they hide transient failures and can cascade into DDoS on downstream services. Configure `max_retries: 3` with `backoff_factor: 2` (retries at 10s, 20s, 40s). For idempotent jobs, enable `idempotent: true` to allow safe retries without side effects.

### 2. **Isolate Critical Jobs with Dependency Groups**
Group jobs that share a common resource (e.g., DB connection, API rate limit) into a `dependency_group`. cronguard will enforce a **staggered start** (configurable `start_delay` in seconds) to avoid thundering herd problems. Example:
```yaml
jobs:
  - name: "data_sync"
    dependency_group: "db_pool"
    start_delay: 5
```
