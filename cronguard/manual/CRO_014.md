### What is cronguard?

cronguard is a battle-tested, open-source cron job monitoring and alerting toolkit designed for DevOps engineers and SREs who need zero-downtime job execution visibility. It hooks directly into your existing cron infrastructure, providing real-time failure detection, execution latency tracking, and automated retry logic—without requiring a full observability stack.

### Core Features

1. **Intelligent Anomaly Detection with Adaptive Thresholds**  
   cronguard learns your job execution patterns (duration, frequency, exit codes) and automatically flags outliers. Instead of static timeouts, it uses rolling window statistics to detect when a cron job runs 3x slower than its historical median or exits with a non-zero code after a silent failure. This reduces false positives by up to 80% compared to hardcoded limits.

2. **Distributed Execution State Machine**  
   Each cron job is modeled as a finite state machine (IDLE → SCHEDULED → RUNNING → SUCCESS/FAILURE). cronguard maintains a distributed lock via etcd or Redis to prevent duplicate executions across multiple nodes, ensuring exactly-once semantics even in Kubernetes clusters with pod restarts. It also exposes a Prometheus-compatible metrics endpoint for granular per-job SLI tracking.

3. **Self-Healing Webhook Pipeline**  
   When a job fails, cronguard doesn't just alert—it triggers a configurable webhook chain. You can define escalation policies (e.g., retry with exponential backoff, then notify PagerDuty, then auto-restart the job on a different node). The pipeline supports conditional branching based on error code or execution context, effectively turning cron into an event-driven automation engine.

### How to Use

1. **Initialize the cronguard Agent**  
   Deploy the cronguard binary as a sidecar in your Kubernetes pod or as a systemd service on bare metal. Run `cronguard init --backend=etcd --cluster-id=prod-us-west` to generate a unique agent ID and register it with the control plane. This creates a local configuration file (`/etc/cronguard/cronguard.yml`) that defines the heartbeat interval (default: 30s) and the alerting endpoint (e.g., Slack webhook URL).

2. **Wrap Your Cron Jobs with cronguard Hooks**  
   Modify your crontab entries by prepending `cronguard run --job-id=nightly-backup --timeout=600s --` before the actual command. For example:  
   `0 3 * * * /usr/local/bin/cronguard run --job-id=nightly-backup --timeout=600s -- /opt/scripts/backup.sh`  
   This injects a pre-execution health check (ensures the job isn't already running) and a post-execution callback that sends the exit code, stdout/stderr, and execution duration to the cronguard API.

3. **Configure Alerting Rules via YAML**  
   Create an `alerting-rules.yml` file in the cronguard config directory:  
   ```yaml
   rules:
     - name: "Backup Failure Escalation"
       condition: "exit_code != 0 OR duration > 540"
       actions:
         - type: retry
           max_attempts: 3
           backoff: exponential
           base_interval: 60s
         - type: notify
           channel: slack
           webhook: "https://hooks.slack.com/services/T00/B00/xxx"
           message: "Nightly backup failed after {{ .Attempts }} retries"
         - type: webhook
           url: "https://api.pagerduty.com/v2/trigger"
           headers: { "Authorization": "Bearer {{ .PD_TOKEN }}" }
   ```  
   Apply the rules with `cronguard apply -f alerting-rules.yml`. The agent will hot-reload the configuration without downtime.

### Best Practices

- **Use cronguard as a Cron Wrapper in CI/CD Pipelines**  
   Instead of embedding cron directly in your Docker images, run cronguard as a sidecar container that intercepts all scheduled jobs. This decouples job scheduling from application logic, making it easier to audit, rollback, and monitor across environments. For example, in a GitLab CI pipeline, add a `cronguard run` step before your deployment job to validate that the previous cron execution succeeded.

- **Implement Canary Deployments for Cron Job Updates**  
   When modifying a critical cron job (e.g., database vacuum or log rotation), use cronguard’s `--canary` flag to run the new version on a single node first. Monitor the execution metrics in Prometheus/Grafana for 24 hours. If the canary job’s P99 latency and error rate are within acceptable bounds, promote it to all nodes using `cronguard promote --job-id=log-rotation --version=2.1`. This prevents widespread failures from a single misconfigured cron.
