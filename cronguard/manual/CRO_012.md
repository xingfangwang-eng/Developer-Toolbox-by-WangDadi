# cronguard - CRO_012

Cronguard is a distributed cron job supervisor engineered for teams that treat scheduled tasks as mission-critical infrastructure. It eliminates ghost processes, enforces idempotent execution, and surfaces execution telemetry without requiring a message broker. Built for high-compliance environments where a missed job is a P0 incident.

## Three Hardcore Features That Define Cronguard

**1. Dead-Letter Replay with Deterministic State Machine**  
Every cron execution passes through a finite-state machine (Pending → Acquired → Running → Complete / Failed). If a worker crashes mid-execution, Cronguard automatically transitions the job to `DeadLetter` and replays it on the next available node with configurable backoff (exponential jitter, max retries, circuit-breaker threshold). No orphaned locks, no manual cleanup.

**2. Preemptive Lease Expiration & Watchdog Heartbeat**  
Instead of relying on fixed TTLs, Cronguard implements a lease-based locking mechanism. Each worker sends a lightweight heartbeat (UDP multicast or Redis Pub/Sub) every 500ms. If the heartbeat misses three consecutive intervals, the lease is revoked and the job is reassigned within <2 seconds. This prevents the classic "zombie scheduler" problem seen in naive crontab wrappers.

**3. Execution Graph with Dependency Injection**  
Define cron jobs as directed acyclic graphs (DAGs) with explicit preconditions and postconditions. Cronguard evaluates dependencies at runtime and skips, delays, or forks execution based on upstream failures. For example: `backup_db` only runs if `health_check` returns HTTP 200, and `notify_oncall` runs unconditionally but with a 30-second debounce window.

## Three Steps to Production-Ready Cron Orchestration

**Step 1: Declare Your Job Manifest in YAML**  
Create a `cronguard.yml` at your project root. Each job entry must include a unique `id`, a `schedule` expression (standard cron or extended with seconds), a `command` with absolute path, and an optional `dependency` block. Cronguard validates the syntax at startup and rejects malformed manifests with a structured error log.

```yaml
jobs:
  - id: data_pipeline_etl
    schedule: "0 3 * * *"
    command: "/usr/local/bin/etl_runner --config /etc/etl/prod.toml"
    dependencies:
      - type: http
        endpoint: "https://api.internal/health"
        expected_status: 200
        timeout_ms: 5000
    retry:
      max_attempts: 3
      backoff: "exponential"
      jitter: true
```

**Step 2: Deploy the Agent as a Sidecar or Systemd Unit**  
Cronguard runs as a single binary with no external runtime dependencies (Go compiled). Install it via `curl -sfL https://cronguard.sh/install | sh`. For Kubernetes, inject it as a sidecar container with the `--store=etcd` flag for distributed coordination. The agent exposes a Prometheus metrics endpoint at `:9090/metrics` with counters for `cronguard_job_runs_total`, `cronguard_lease_errors_total`, and `cronguard_execution_duration_seconds`.

**Step 3: Monitor Execution Traces via the Built-In Web UI**  
Open `http://localhost:8080` to view the live dashboard. Each job run is logged with a unique trace ID, start/stop timestamps, exit code, and stdout/stderr tail (last 100 lines). Filter by status (`failed`, `dead_letter`, `running`) or search by job ID. Export the last 10,000 runs as JSON for audit compliance. The UI also shows the current lease holder for each job (node hostname + PID).

## Two Industry Best Practices for Cron Reliability

**1. Treat Every Cron Job as an Idempotent API Call**  
Wrap your cron scripts with a locking guard (e.g., `flock` on Linux) to prevent overlapping executions, but Cronguard’s lease system already enforces this at the scheduler level. Go further: design your job to be safely replayed. If `data_pipeline_etl` fails mid-run, Cronguard will retry it; ensure your script can detect partial writes and roll back gracefully. Use the `CRONGUARD_RUN_ID` environment variable (injected automatically) to track idempotency keys.

**2. Implement Gradual Rollout via Canary Schedules**  
Before pushing a schedule change to production, define a canary job with the same command but a different schedule (e.g., every 5 minutes instead of hourly). Monitor the canary’s execution duration, error rate, and resource usage on the Cronguard dashboard. Only promote the schedule to the main job after the canary has passed 10 consecutive runs without a dead-letter event. This pattern catches silent failures (e.g., memory leaks, API deprecations) before they impact SLAs.
