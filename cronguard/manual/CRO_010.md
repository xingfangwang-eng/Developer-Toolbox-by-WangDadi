# cronguard - CRO_010

**cronguard** is a lightweight, multi-language cron orchestration and compliance engine built for DevOps engineers who refuse to let scheduled jobs fail silently. Unlike generic cron wrappers, cronguard enforces **execution guarantees**, **runtime introspection**, and **failure forensics** across distributed environments—without requiring a centralized scheduler or a complex message broker.

If you are managing cron jobs across Kubernetes pods, bare-metal VMs, or hybrid edge nodes, cronguard is your single source of truth for scheduled task observability.

---

## 🔧 Hardcore Features That Set cronguard Apart

### 1. **Execution Fingerprint & Drift Detection**
Every cron run generates a deterministic **execution fingerprint** (hash of command, environment variables, host metadata, and timestamp window). cronguard compares this fingerprint against a historical baseline to detect **silent configuration drift**—e.g., a changed `PATH`, missing dependency, or environment variable injection—before the job fails in production.

### 2. **Graceful Degradation with Fallback Chains**
You can define a **fallback chain** of up to 5 alternative commands or scripts. If the primary job exits with a non-zero code, cronguard automatically executes the next fallback within the same scheduling window. This is not a retry—it is a **semantically different recovery path** (e.g., try primary API, fallback to cached data, fallback to mock service). Each fallback is logged with its own fingerprint and exit code.

### 3. **Distributed Lock-Free Mutual Exclusion**
For jobs that must run exactly once across a cluster, cronguard uses a **lease-based mutual exclusion** built on etcd or Redis (configurable). No distributed lock, no contention—just a time-bound lease that auto-expires. If the primary node dies mid-execution, any other node with the same cron schedule can claim the lease and resume, guaranteed by the lease’s TTL and cronguard’s heartbeat mechanism.

## 🛠️ How to Use cronguard Like a Pro

### Step 1: Initialize a cronguard Managed Cron Environment
```bash
cronguard init --project my-ebank-sync \
  --backend redis://10.0.0.1:6379 \
  --fingerprint-dir /var/log/cronguard/fingerprints \
  --fallback-chain "primary:./sync.sh,fallback1:./sync_cached.sh,fallback2:./sync_mock.sh"
```
This creates a `.cronguard.yaml` manifest in your project root, registers the backend, and precomputes the fingerprint baseline for all commands in the fallback chain. The `--fingerprint-dir` stores historical fingerprints for drift analysis.

### Step 2: Schedule a Job with Lease-Based Mutual Exclusion
```yaml
# .cronguard.yaml
jobs:
  - name: "daily-reconciliation"
    schedule: "0 2 * * *"
    command: "./reconcile.py --env production"
    fingerprint: true
    lease:
      backend: "etcd"
      ttl: 600  # seconds
      key: "/cronguard/leases/daily-reconciliation"
    fallback_chain:
      - "./reconcile.py --env fallback"
      - "./reconcile.py --dry-run"
```
Run `cronguard apply` to deploy the job. cronguard will automatically register the lease key in etcd. On each cron tick, the node that acquires the lease (via `cronguard lease-try`) executes the job. Other nodes skip this tick.

### Step 3: Forensic Analysis After a Failure
```bash
cronguard inspect --job daily-reconciliation --run-id 2025-03-21T02:00:00Z \
  --show-fingerprint \
  --show-drift \
  --show-fallback-chain
```
This outputs a JSON report including:
- **Fingerprint hash** vs. baseline (drift score)
- **Lease status** (acquired by which node, TTL remaining)
- **Full fallback chain execution log** (exit codes, stdout/stderr per fallback)
- **Host metadata** (kernel version, cronguard version, uptime)

Use this to determine whether the failure was due to drift, lease contention, or a genuine bug in the primary command.

## 🏆 Industry Best Practices for cronguard

### 1. **Always Enable Fingerprint Baseline Updates After Successful Runs**
Do not let the baseline become stale. After a successful job execution, run:
```bash
cronguard update-baseline --job daily-reconciliation
```
This updates the stored fingerprint to reflect the current environment. Without this, a legitimate configuration change (e.g., a new Python package) will trigger a false drift alert on the next run.

### 2. **Use Fallback Chains for Stateful Jobs, Not Stateless Ones**
Fallback chains are expensive in terms of log storage and execution time. Reserve them for jobs that **cannot afford to fail**—e.g., database replication checks, payment reconciliation, or certificate renewal. For stateless jobs (e.g., log rotation), rely on cronguard’s **automatic retry with exponential backoff** (configured via `retry: 3` in the job spec) instead of a fallback chain.
