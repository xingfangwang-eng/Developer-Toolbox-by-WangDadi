## What is cronguard?

cronguard is a **production-grade cron job monitoring and alerting engine** designed for distributed systems, microservices, and serverless architectures. It provides real-time visibility into scheduled task execution, failure detection, and automated remediation—without requiring any infrastructure changes or complex integrations.

---

## Core Features

### 1. **Deadline-Aware Heartbeat Escalation**
Unlike simple “up/down” monitors, cronguard tracks expected execution windows (e.g., “must run every 5 minutes, but allowed a 30-second grace period”). If a heartbeat is missed or delayed beyond a configurable deadline, it triggers a multi-channel alert (PagerDuty, Slack, email) with contextual logs and failure stack traces. This prevents false alarms from transient network blips while ensuring zero tolerance for actual failures.

### 2. **Idempotency Key Injection via Sidecar Proxy**
cronguard automatically injects idempotency keys into each cron job invocation using a lightweight sidecar container. This guarantees that even if a job is retried due to node failure or network partition, it won’t cause duplicate side effects. The sidecar intercepts HTTP/gRPC calls and validates keys against a Redis-backed deduplication store—critical for financial, billing, or database migration tasks.

### 3. **Declarative Cron Policy as Code (CPaC)**
Define cron job behavior, alerting rules, and retry strategies in a YAML/JSON manifest stored in your Git repo. cronguard’s operator reconciles the desired state with actual execution, automatically pausing jobs during maintenance windows, scaling concurrency across nodes, or enforcing runtime limits (e.g., kill jobs exceeding 10 minutes). This enables full GitOps workflow for cron management.

## How to Use

### Step 1: Deploy the cronguard Agent as a DaemonSet
Run the following command to deploy the agent across your Kubernetes cluster. The agent will automatically discover cron jobs via the Kubernetes API and register them with the cronguard backend.

```bash
kubectl apply -f https://cronguard.example.com/deploy/daemonset.yaml
```

Verify agent health:
```bash
cronguardctl status --namespace default
```

Expected output:
```
Agent healthy | 12 cron jobs discovered | 3 jobs with missing heartbeats
```

### Step 2: Attach a CronJob Manifest with CPaC Annotations
Add cronguard-specific annotations to your existing CronJob YAML. This example sets a 5-minute heartbeat deadline, enables idempotency keys, and defines a retry policy with exponential backoff.

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: data-export
  annotations:
    cronguard.io/heartbeat-deadline: "5m"
    cronguard.io/idempotency-enabled: "true"
    cronguard.io/retry-max-attempts: "3"
    cronguard.io/retry-backoff: "exponential"
spec:
  schedule: "*/10 * * * *"
  jobTemplate:
    spec:
      template:
        metadata:
          annotations:
            cronguard.io/sidecar-inject: "true"
        spec:
          containers:
          - name: exporter
            image: myrepo/exporter:latest
          - name: cronguard-sidecar
            image: cronguard/sidecar:latest
```

Apply the manifest:
```bash
kubectl apply -f cronjob-export.yaml
```

### Step 3: Monitor and Tune Using the cronguard Dashboard
Access the web UI at `https://cronguard.local:8443` (default). The dashboard provides:

- **Execution timeline**: See each cron job’s start, end, and any missed or delayed runs.
- **Alert history**: Review all triggered alerts with raw logs and failure context.
- **Policy editor**: Modify CPaC rules in real-time (e.g., increase retry attempts for a flaky job).

To test alerting, manually trigger a heartbeat failure:
```bash
cronguardctl simulate-failure --job=data-export --reason="network timeout"
```

Check the alert in Slack/PagerDuty within 10 seconds.

## Best Practices

### 1. **Always Set a Grace Period for Heartbeat Deadlines**
For jobs with variable execution times (e.g., data pipelines that may take 2–10 minutes), set a grace period of 20–30% of the expected runtime. This avoids false positives during normal variance while still catching genuine stalls. Example: if a job runs every 15 minutes, set `heartbeat-deadline` to 18 minutes.

### 2. **Use Idempotency Keys for All Mutating Cron Jobs**
If your cron job writes to a database, sends emails, or triggers external APIs, enable idempotency key injection. This prevents duplicate records or double charges during retries. Store the idempotency data in Redis with a TTL equal to the cron job’s schedule interval (e.g., for a job running every hour, set TTL to 60 minutes). This balances memory usage with safety.
