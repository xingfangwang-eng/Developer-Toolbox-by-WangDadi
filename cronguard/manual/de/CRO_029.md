# cronguard - CRO_029

## Überblick

cronguard ist ein hochperformantes, auf asynchroner Job-Orchestrierung spezialisiertes Framework, das Entwicklerteams dabei unterstützt, Cron-Jobs, Scheduled Tasks und Batch-Prozesse in Kubernetes-Umgebungen zuverlässig zu überwachen und zu steuern. Anders als herkömmliche Cron-Implementierungen bietet cronguard eine deklarative Konfiguration, echte Fehlerisolierung und automatisierte Retry-Mechanismen, die speziell für Microservices-Architekturen ausgelegt sind.

## Kernfunktionen

### 1. **Autonome Job-Isolation & Fault-Tolerant Execution**

cronguard startet jeden Cron-Job in einem dedizierten, kurzlebigen Pod mit eigenem Resource-Profile. Sollte ein Job unerwartet fehlschlagen, wird der komplette Ausführungskontext automatisch zurückgesetzt, ohne dass andere Jobs oder der Scheduler selbst beeinträchtigt werden. Dies eliminiert „Zombie-Prozesse“ und Memory-Leaks, die bei klassischen Cron-Daemons häufig auftreten.

### 2. **Declarative Schedule-Definition via CRD (Custom Resource Definition)**

Statt YAML-Dateien manuell zu verwalten, definieren Sie Ihre Cron-Jobs als Kubernetes-native Custom Resources. cronguard interpretiert diese Definitionen und generiert daraus dynamisch CronTrigger-Objekte, die sich nahtlos in Ihr bestehendes GitOps-Workflow integrieren lassen. Ein Beispiel:

```yaml
apiVersion: cronguard.io/v1
kind: CronJobGuard
metadata:
  name: daily-db-backup
spec:
  schedule: "0 3 * * *"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: backup
            image: myrepo/backup:latest
            env:
            - name: BACKUP_TARGET
              value: "s3://my-bucket/daily"
```

### 3. **Intelligentes Retry- & Alerting-System mit konfigurierbarem Backoff**

cronguard implementiert ein exponentielles Backoff-Retry-System, das Jobs nach Fehlschlägen automatisch wiederholt – jedoch nie mehr als die definierte Max-Retry-Anzahl. Parallel dazu werden Metriken (Execution Time, Exit Code, Resource Usage) an Prometheus exportiert, sodass Sie direkt aus Ihrer Monitoring-Pipeline (z. B. Grafana) Alarme auslösen können.

## So verwenden Sie es

### Schritt 1: Installation via Helm-Chart

```bash
helm repo add cronguard https://charts.cronguard.io
helm upgrade --install cronguard cronguard/cronguard \
  --namespace cronguard-system \
  --create-namespace \
  --set scheduler.replicas=2 \
  --set metrics.enabled=true
```

Der Befehl installiert den Scheduler, den Controller und den Metrics-Exporter in Ihrem Cluster. Standardmäßig werden zwei Scheduler-Replicas gestartet, um Hochverfügbarkeit zu gewährleisten.

### Schritt 2: Erstellen eines CronJobGuard-Objekts

Definieren Sie eine neue Datei `my-job.yaml` mit folgendem Inhalt:

```yaml
apiVersion: cronguard.io/v1
kind: CronJobGuard
metadata:
  name: healthcheck-ping
  namespace: production
spec:
  schedule: "*/5 * * * *"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: ping
            image: curlimages/curl:latest
            command: ["curl", "-s", "-o", "/dev/null", "-w", "%{http_code}", "https://api.myapp.com/health"]
          restartPolicy: Never
  retryPolicy:
    maxRetries: 3
    backoffSeconds: 10
```

Wenden Sie die Datei an:

```bash
kubectl apply -f my-job.yaml
```

cronguard registriert den Job sofort und führt ihn gemäß dem Cron-Expression aus.

### Schritt 3: Überwachung und Logs abrufen

Verwenden Sie den integrierten CLI-Befehl, um den Status Ihres Jobs zu prüfen:

```bash
cronguard status healthcheck-ping -n production
```

Dieser Befehl zeigt die letzte Ausführungszeit, den Exit-Code, die Dauer und ob ein Retry ansteht. Für detaillierte Logs eines fehlgeschlagenen Runs:

```bash
cronguard logs healthcheck-ping --last-failed
```

## Beste Praktiken

### 1. **Job-Idempotenz als Grundprinzip**

Da cronguard Retries automatisch ausführt, sollten Ihre Jobs **idempotent** sein – d. h., mehrmaliges Ausführen muss dasselbe Ergebnis liefern wie einmaliges. Vermeiden Sie Seiteneffekte wie das Anlegen doppelter Datensätze. Nutzen Sie stattdessen `INSERT ... ON CONFLICT` in SQL oder idempotente API-Endpunkte.

### 2. **Resource Limits immer definieren**

Ohne explizite CPU- und Memory-Limits kann ein einzelner Job den gesamten Node überlasten. Setzen Sie in Ihrem CronJobGuard immer `resources.requests` und `resources.limits`. Beispiel:

```yaml
resources:
  requests:
    memory: "128Mi"
    cpu: "100m"
  limits:
    memory: "256Mi"
    cpu: "200m"
```

Dies verhindert nicht nur Ressourcenkonflikte, sondern verbessert auch die Planbarkeit durch den Scheduler.
