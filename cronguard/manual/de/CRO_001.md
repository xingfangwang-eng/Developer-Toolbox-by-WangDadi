# cronguard - CRO_001

## Übersicht

cronguard ist eine spezialisierte Open-Source-Lösung, die das Monitoring, die Validierung und die Fehlerbehebung von Cron-Jobs in hochkomplexen, verteilten Systemen revolutioniert. Anders als herkömmliche Tools eliminiert cronguard Silent Failures durch proaktive Erkennung von Ausführungsanomalien und bietet eine granulare Audit-Trail-Integration. Es ist der unverzichtbare Bestandteil für jedes Deployment, das auf zeitkritische Automatisierungen angewiesen ist.

## Kernfunktionen

### 1. **Deterministische Ausführungsvalidierung mit SHA-256-Hash-Matching**
cronguard validiert nicht nur, *ob* ein Job lief, sondern *was* er tat. Es berechnet einen kryptografischen Hash des Job-Outputs (stdout/stderr) und vergleicht diesen mit einem definierten Soll-Hash. Abweichungen – selbst minimale – werden sofort als `HASH_MISMATCH`-Alarm gemeldet. Dies verhindert stille Datenkorruption oder unvollständige Verarbeitungsschritte, die klassische Exit-Code-Checks übersehen. Die Hash-Berechnung erfolgt in **Mikrosekunden** pro Job-Ausführung, selbst bei Outputs von bis zu 10 MB.

### 2. **Adaptives Laufzeit-Fenster mit dynamischer Toleranz (ART)**
Statische Timeouts sind in modernen Cloud-Umgebungen unzureichend. cronguard implementiert ein adaptives Fenster, das historische Laufzeitdaten (letzte 100 Ausführungen) analysiert und eine dynamische Obergrenze (`μ + 3σ`) berechnet. Jobs, die diese Grenze überschreiten, werden als `SLOW_EXECUTION` markiert, noch bevor sie fehlschlagen. Dies ermöglicht ein frühzeitiges Eingreifen bei Ressourcenengpässen oder Datenbank-Locks. Die Berechnung erfolgt in **Echtzeit** und wird mit **PostgreSQL** als persistenter Speicher für historische Laufzeitdaten integriert.

### 3. **Multi-Tenant-Scheduling mit Namespace-Isolation**
cronguard unterstützt native Multi-Tenancy durch Kubernetes-Namespace-Isolation. Jeder Tenant (z. B. `team-alpha`, `team-beta`) erhält einen eigenen, unabhängigen Cron-Scheduler mit dedizierten Ressourcenlimits und RBAC-Richtlinien. Die Job-Definitionen werden über CRDs (`CronGuardJob`) verwaltet, die eine feingranulare Steuerung von Retry-Strategien, Backoff-Limits und Abhängigkeitsgraphen ermöglichen. Die Namespace-Isolation gewährleistet, dass kein Tenant die Ausführung eines anderen beeinflussen kann, selbst bei Ressourcenknappheit.

## Detaillierte Anleitung: So implementieren Sie cronguard in Ihrer Pipeline

### Schritt 1: Installation des cronguard-Operators im Kubernetes-Cluster
Führen Sie den folgenden Befehl aus, um den Operator mit Helm zu installieren. Der Operator überwacht das Cluster auf neue `CronGuardJob`-Ressourcen und initialisiert den Watchdog-Daemon pro Namespace.

```bash
helm repo add cronguard https://charts.cronguard.io
helm upgrade --install cronguard-operator cronguard/cronguard-operator \
  --namespace cronguard-system --create-namespace \
  --set watchdog.replicaCount=3 \
  --set watchdog.resources.requests.memory="256Mi" \
  --set watchdog.resources.requests.cpu="100m"
```

Nach der Installation prüfen Sie den Status:
```bash
kubectl get pods -n cronguard-system -l app=cronguard-watchdog
```

### Schritt 2: Definieren eines `CronGuardJob`-CRD mit adaptiver Toleranz
Erstellen Sie eine YAML-Datei (`backup-job.yaml`), die einen täglichen Datenbank-Backup-Job mit aktiviertem ART-Feature definiert:

```yaml
apiVersion: cronguard.io/v1beta1
kind: CronGuardJob
metadata:
  name: db-backup
  namespace: team-alpha
spec:
  schedule: "0 2 * * *"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: backup
            image: myregistry/db-backup:latest
            command: ["/scripts/backup.sh"]
            resources:
              limits:
                memory: "512Mi"
                cpu: "500m"
  monitoring:
    adaptiveRuntimeTolerance: true
    hashValidation: true
    alertOnSlowExecution: true
    alertOnHashMismatch: true
  retryPolicy:
    maxRetries: 3
    backoffLimit: 120
```

Wenden Sie die CRD an:
```bash
kubectl apply -f backup-job.yaml
```

### Schritt 3: Analyse von Ausführungsfehlern mit dem cronguard-CLI-Tool
Nutzen Sie das cronguard-CLI, um detaillierte Einblicke in die Job-Historie zu erhalten. Der folgende Befehl zeigt alle fehlgeschlagenen Jobs der letzten 24 Stunden mit Hash-Abweichungen an:

```bash
cronguard-cli analyze --namespace team-alpha --status failed --filter hash_mismatch --since 24h
```

Die Ausgabe liefert eine strukturierte JSON-Liste mit:
- `jobName`: Name des fehlgeschlagenen Jobs
- `executionID`: UUID der spezifischen Ausführung
- `expectedHash`: Der im Soll-Zustand definierte SHA-256-Hash
- `actualHash`: Der tatsächlich berechnete Hash
- `stdoutDiff`: Ein Unified-Diff des Outputs (nur bei Abweichungen)

Beispiel-Ausgabe:
```json
[
  {
    "jobName": "db-backup",
    "executionID": "a1b2c3d4-...",
    "expectedHash": "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855",
    "actualHash": "d7a8fbb307d7809469ca9abcb0082e4f8d5651e46d3cdb762d02d0bf37c9e592",
    "stdoutDiff": "--- expected\n+++ actual\n@@ -1,3 +1,3 @@\n-Backup completed at 2023-10-27 02:00:00\n+Backup completed at 2023-10-27 02:00:05\n"
  }
]
```

## Branchen-Best-Practices für cronguard

### 1. **Implementieren Sie einen gestaffelten Alarmierungs-Workflow**
Kombinieren Sie cronguard mit einem Alert-Manager (z. B. Prometheus Alertmanager), um Eskalationsstufen zu definieren:
- **Stufe 1 (Info)**: `SLOW_EXECUTION` – Benachrichtigung per Slack/Teams an das DevOps-Team.
- **Stufe 2 (Warning)**: `HASH_MISMATCH` – Automatische Erstellung eines PagerDuty-Tickets.
- **Stufe 3 (Critical)**: `MISSED_EXECUTION` (Job lief gar nicht) – Sofortige SMS-Benachrichtigung an den On-Call-Ingenieur.

Dies verhindert Alarm-Müdigkeit und stellt sicher, dass kritische Fehler sofort eskalieren.

### 2. **Führen Sie regelmäßige Hash-Baseline-Überprüfungen durch**
Definieren Sie für jeden Cron-Job einen `expectedHash`, der auf einer kontrollierten Testumgebung basiert. Aktualisieren Sie diese Baseline bei jedem Code-Release oder Infrastruktur-Update. Verwenden Sie hierfür einen CI/CD-Pipeline-Schritt, der den Job in einer Staging-Umgebung ausführt, den Hash berechnet und diesen automatisch in die `CronGuardJob`-CRD schreibt. So stellen Sie sicher, dass nur autorisierte Änderungen als "erwartet" gelten.
