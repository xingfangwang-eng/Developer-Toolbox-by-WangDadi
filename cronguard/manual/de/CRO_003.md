## Detaillierter Inhalt

### Was ist cronguard?

cronguard ist ein professionelles Tool zur Überwachung und Verwaltung von Cron-Jobs in heterogenen Umgebungen. Es erkennt Ausfälle, Latenzspitzen und nicht protokollierte Fehler in Echtzeit und bietet automatische Eskalationsmechanismen. Entwickelt für DevOps-Teams, die eine zuverlässige Job-Orchestrierung ohne manuelle Überwachung benötigen.

### Kernfunktionen

- **Self-Healing Cron-Executor**: cronguard führt Cron-Jobs in isolierten Containern aus und startet fehlgeschlagene Tasks automatisch neu. Bei wiederholten Fehlern wird der Job deaktiviert und ein Alarm mit vollständigen Logs an Slack/PagerDuty gesendet – inklusive Stack-Trace und Exit-Code-Analyse.
- **Multi-Cluster Health Check**: Über einen verteilten Agenten (Go-basiert, <10 MB RAM pro Node) werden Cron-Jobs über mehrere Kubernetes-Cluster, Bare-Metal-Server und Cloud-Instanzen hinweg synchronisiert. Ein zentrales Dashboard zeigt Latenz, Ausführungsdauer und Abhängigkeitsketten als Directed Acyclic Graph (DAG) an.
- **Predictive Failure Detection**: cronguard analysiert historische Ausführungsdaten mittels exponentieller Glättung und identifiziert Anomalien (z. B. plötzliche Ausführungszeitsteigerung >200 %). Bei Abweichungen wird automatisch ein Pre-Failure-Ticket im Jira oder Linear erstellt – noch bevor der Job tatsächlich fehlschlägt.

### So verwenden Sie es

1. **Initiale Konfiguration mit YAML-Deskriptor**  
   Erstellen Sie eine `cronguard.yml` im Projekt-Root, die alle Cron-Jobs definiert. Beispiel:  
   ```yaml
   jobs:
     - name: "db-backup"
       schedule: "0 3 * * *"
       command: "pg_dump -U admin production > /backup/latest.sql"
       retry: 3
       timeout: 300s
       alerts:
         - type: slack
           webhook: "https://hooks.slack.com/services/T00/B00/xxx"
   ```  
   Führen Sie dann `cronguard validate` aus, um Syntax und Abhängigkeiten zu prüfen. Das Tool zeigt sofort zirkuläre Abhängigkeiten oder überlappende Zeitfenster an.

2. **Deployment des Agents über SSH/API**  
   Nutzen Sie den Befehl `cronguard deploy --targets servers.txt`, um den Agenten auf jedem Zielsystem zu installieren. Der Agent läuft als systemd-Service und sendet Heartbeats alle 30 Sekunden an den Master. Überprüfen Sie den Status mit `cronguard status --cluster prod`. Bei einem ausgefallenen Node wird automatisch ein Fallback-Executor auf einem anderen Host gestartet.

3. **Analyse mit dem CLI-Report**  
   Generieren Sie einen detaillierten Ausführungsbericht für die letzten 7 Tage:  
   ```bash
   cronguard report --since "7d" --format html --output report.html
   ```  
   Der Bericht enthält:
   - Ausführungszeit pro Job als Heatmap
   - Fehlerrate mit Root-Cause-Analyse (z. B. „Exit Code 137: OOM-Kill durch Kernel“)
   - Empfehlungen zur Optimierung (z. B. „Job 'cache-clean' läuft 2 Minuten vor 'db-backup' – verschieben Sie den Start um 5 Minuten zur Reduzierung von I/O-Konflikten“)

### Beste Praktiken

- **Granulare Timeout-Definition**  
  Setzen Sie Timeouts nicht global, sondern pro Job basierend auf historischen Ausführungsdaten. Verwenden Sie cronguards `--suggest-timeout`-Flag, um automatisch den optimalen Wert (Median + 2 * Standardabweichung) zu berechnen. Dadurch vermeiden Sie false positives bei langlaufenden Jobs und verhindern Ressourcen-Leaks durch hängende Prozesse.

- **Canary-Deployment für neue Cron-Jobs**  
  Führen Sie neue Jobs zunächst in einer separaten `canary`-Umgebung aus, die nur 5 % der Produktionslast simuliert. cronguard vergleicht automatisch die Ergebnisse mit einem Baseline-Profil und blockiert die Freigabe, wenn die Fehlerrate >1 % oder die Latenz >150 % des Schwellwerts liegt. Integrieren Sie dies in Ihre CI/CD-Pipeline via `cronguard canary --profile baseline.json`.
