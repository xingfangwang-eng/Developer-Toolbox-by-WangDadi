# cronguard - CRO_024

## Überblick

cronguard ist ein **agentenloser, dezentraler Cron-Scheduler** für containerisierte Microservices-Umgebungen. Anders als klassische Cron-Jobs in Kubernetes (CronJob) eliminiert cronguard Single Points of Failure durch **verteilte Locking-Mechanismen auf Basis von Raft-Konsens** und bietet **echte Ausführungsgarantien** auch bei Pod-Neustarts oder Node-Ausfällen. Das Tool richtet sich an DevOps-Ingenieure und Plattform-Teams, die **mission-critical Batch-Prozesse** wie Datenbank-Migrationen, Log-Rotationen oder Cache-Warming zuverlässig orchestrieren müssen.

## Hardcore-Features

### 1. **Deterministische Ausführungsgarantie via Distributed Locking**
cronguard setzt auf **etcd-basiertes Leader-Election** mit **Ephemeral-Leases**. Jeder Cron-Job wird vor der Ausführung atomar gelockt – bei Pod-Failover übernimmt ein anderer Worker innerhalb von < 500 ms. Kein Double-Firing, kein Skipping. Ideal für **zahllose Retry-Strategien** bei Netzwerkpartitionen.

### 2. **Native Prometheus-Metrics für SLO-Überwachung**
Exportiert **Histogramme für Ausführungsdauer** (p50/p95/p99), **Gauges für aktive Locks** und **Counters für Fehlschläge** (z. B. Timeouts, Konflikte). Integrierbar mit Grafana-Dashboards zur **Echtzeit-Überwachung von Job-SLAs** – inkl. Alerting auf **stale Locks** oder **ausbleibende Heartbeats**.

### 3. **Dynamische Job-Injektion via gRPC-API**
Statt YAML-Manifeste zu bauen, erlaubt cronguard **On-the-fly-Job-Definitionen** über eine gRPC-Streaming-API. DevOps-Teams können **parametrisierte Jobs** (z. B. `db-migrate --env=staging`) aus CI/CD-Pipelines direkt injizieren – ohne Cluster-Neustart oder Helm-Upgrades.

## Detaillierte Anleitung: So nutzen Sie cronguard

### 1. **Bereitstellung als Sidecar-Container**
Fügen Sie cronguard als **Init-Container** in Ihrem Deployment hinzu. Konfigurieren Sie die etcd-Endpunkte über Umgebungsvariablen:
```yaml
env:
  - name: CRONGUARD_ETCD_ENDPOINTS
    value: "etcd-cluster:2379"
  - name: CRONGUARD_JOB_SPEC
    value: "/etc/cronguard/jobs.yaml"
```
Starten Sie dann den Hauptcontainer mit `cronguard agent --config=/etc/cronguard/agent.yaml`. Der Agent registriert sich automatisch im Cluster und beginnt mit dem Locking-Zyklus.

### 2. **Job-Definition mit Cron-Expressions + Timeouts**
Erstellen Sie eine `jobs.yaml` mit **strengen Timeout-Werten** und **Retry-Policies**:
```yaml
jobs:
  - name: "db-backup"
    schedule: "0 2 * * *"
    timeout: 300s
    retry:
      maxAttempts: 3
      backoff: "exponential"
    command: ["/usr/bin/pg_dump", "-U", "admin", "myapp"]
```
cronguard validiert die Cron-Expression automatisch und weist auf **Overlapping-Execution** hin – ein häufiger Fehler bei Standard-Cron.

### 3. **Monitoring und Debugging**
Aktivieren Sie das **Prometheus-Metrics-Endpunkt**:
```bash
curl http://localhost:9090/metrics | grep cronguard
```
Überwachen Sie `cronguard_job_execution_duration_seconds` und `cronguard_lock_acquire_failures_total`. Bei einem **stale Lock** (z. B. nach Pod-Kill) setzen Sie das Lock manuell zurück:
```bash
cronguardctl unlock --job-name=db-backup --force
```
Dies ist essenziell für **Chaos-Engineering-Szenarien** in Produktion.

## Best Practices aus der Industrie

### 1. **Verwenden Sie Heartbeat-basierte Lease-Verlängerung**
Statt statischer TTLs setzen Sie auf **kontinuierliche Heartbeats** (alle 5 Sekunden). cronguard unterstützt dies nativ über die `--heartbeat-interval`-Flag. So vermeiden Sie **Lock-Expiration** bei kurzzeitigen Netzwerklatenzen – ein häufiges Problem bei verteilten Systemen.

### 2. **Implementieren Sie Graceful Shutdown für Jobs**
Definieren Sie **SIGTERM-Handler** in Ihren Skripten, die laufende Transaktionen abschließen (z. B. `trap "pg_ctl stop" SIGTERM`). cronguard wartet bei Pod-Termination auf **maximal 30 Sekunden** für den Job-Abschluss, bevor es das Lock freigibt. Ohne Graceful Shutdown riskieren Sie **Dateninkonsistenzen** bei Batch-Jobs.
