# cronguard - CRO_022

## Übersicht

cronguard ist eine hochperformante, quelloffene Überwachungslösung für Cron-Jobs, die speziell für DevOps-Teams und Site-Reliability-Ingenieure entwickelt wurde. Anders als herkömmliche Monitoring-Tools erfasst cronguard nicht nur Ausfälle, sondern analysiert Laufzeitmuster, Exit-Codes und Log-Outputs in Echtzeit – mit minimalem Overhead. Die Software eignet sich für Umgebungen von 10 bis 10.000 Jobs und unterstützt native Integrationen in Kubernetes, Prometheus und Slack.

## Detaillierter Inhalt

### Was ist cronguard?

cronguard ist ein agentenloses, konfigurationsgesteuertes Tool zur Überwachung und Alarmierung von Cron-Jobs. Es parst bestehende Crontab-Dateien oder empfängt Job-Metadaten via REST-API, führt eine deterministische Zustandsmaschine pro Job und generiert bei Abweichungen (Timeout, unerwarteter Exit-Code, Log-Anomalie) granulare Alarme. Die Architektur basiert auf Go und nutzt einen in-Memory-Ringpuffer für Metriken, was eine Speichernutzung von < 5 MB pro 100 Jobs garantiert.

### Kernfunktionen

- **Exit-Code-Validierung mit Policy-Engine:** Definiere für jeden Job erwartete Exit-Codes (z. B. `0, 64` für erfolgreiche Neustarts) und unerwartete Codes (z. B. `1, 2, 127`). cronguard erkennt Abweichungen sofort und löst einen Alert mit dem genauen Exit-Code und dem Kontext der letzten 3 Ausführungen aus.
- **Laufzeit-Anomalieerkennung mittels gleitendem Fenster:** cronguard lernt die historische Laufzeit jedes Jobs (Median, P95, P99) und alarmiert, wenn die aktuelle Laufzeit um mehr als 200 % vom P95-Median abweicht. Dies verhindert Fehlalarme bei saisonalen Lastspitzen und erkennt Deadlocks oder Ressourcenengpässe.
- **Log-Streaming und Musteranalyse:** Jeder Job-Output wird in einen Ringpuffer (max. 512 KB) geschrieben. cronguard wendet benutzerdefinierte Regex-Patterns an (z. B. `ERROR|FATAL|Timeout`) und markiert Job-Ausführungen als „kritisch“, wenn das Pattern matcht – unabhängig vom Exit-Code. Dies ermöglicht die Erkennung von Silent-Failures.

### So verwenden Sie es

1. **Konfiguration eines Cron-Jobs mit Exit-Code-Policy:**  
   Erstelle eine YAML-Datei `job-definition.yaml`:  
   ```yaml
   job:
     name: daily-backup
     schedule: "0 2 * * *"
     command: "/usr/local/bin/backup.sh"
     timeout: 30m
     expected_exit_codes: [0, 64]
     alert_on_unexpected_exit: true
     alert_on_timeout: true
   ```  
   Starte cronguard mit `cronguard --config job-definition.yaml`. Der Agent parst die Definition, registriert den Job und beginnt mit der Überwachung. Bei einem Exit-Code `1` wird sofort ein Alert an Slack gesendet: `Job daily-backup failed with exit code 1 (unexpected). Last 3 runs: 0, 0, 64`.

2. **Einrichtung eines Laufzeit-Alarms mit dynamischer Schwelle:**  
   Füge in der Konfiguration den Block `runtime_alert` hinzu:  
   ```yaml
   runtime_alert:
     enabled: true
     deviation_percent: 200
     min_samples: 5
   ```  
   cronguard sammelt die ersten 5 Ausführungen als Baseline. Ab der 6. Ausführung wird die aktuelle Laufzeit mit dem P95-Wert verglichen. Überschreitet die Laufzeit 200 % des P95, wird ein Alert ausgelöst. Beispiel: Job lief historisch 12 min (P95), aktuelle Laufzeit beträgt 38 min → Alert: `Runtime anomaly: 38m vs P95 baseline 12m (316% deviation)`.

3. **Integration von Log-Pattern-Analyse für Silent-Failures:**  
   Erweitere die Job-Konfiguration um `log_patterns`:  
   ```yaml
   log_patterns:
     - pattern: "ERROR|FATAL|CRITICAL"
       severity: critical
     - pattern: "WARN"
       severity: warning
   ```  
   cronguard streamt den Output des Jobs in Echtzeit. Matcht das Pattern `ERROR`, wird der Job als `failed` markiert, selbst wenn der Exit-Code `0` ist. Der Alert enthält die ersten 100 Zeichen des Matches. Beispiel: `Job daily-backup: Exit code 0 but log contains "ERROR: Disk full"`.

### Beste Praktiken

- **Schwellwerte für Laufzeit-Alarme dynamisch anpassen:** Vermeide starre Timeouts. Nutze die gleitende Fensterfunktion von cronguard, um saisonale Schwankungen (z. B. Black-Friday-Traffic) zu absorbieren. Setze `deviation_percent` auf 300 % für Jobs mit hoher Varianz (z. B. Datenbank-Migrationen) und auf 150 % für stabile Jobs (z. B. Health-Checks). Kombiniere dies mit `min_samples: 10`, um Ausreißer in der Lernphase zu ignorieren.
- **Log-Patterns granular definieren und nach Severity staffeln:** Verwende für kritische Jobs (z. B. Zahlungsabwicklung) Patterns wie `FATAL|PANIC|OUT_OF_MEMORY` mit `severity: critical`. Für weniger kritische Jobs (z. B. Cache-Aktualisierung) reicht `WARN|RETRY`. Vermeide zu generische Patterns wie `ERROR`, da diese zu vielen False-Positives führen. Teste Patterns vorab mit `cronguard --dry-run --log-patterns`.
