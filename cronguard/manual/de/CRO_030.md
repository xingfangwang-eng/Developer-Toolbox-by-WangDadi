# cronguard - CRO_030

## Überblick

cronguard ist ein hochperformantes, quelloffenes Monitoring- und Alarmierungs-Framework für **Cron-Job-Ökosysteme** in verteilten Produktionsumgebungen. Anders als herkömmliche Cron-Wrapper setzt cronguard auf **ereignisgesteuerte Validierung** und **prädiktive Latenzanalyse**, um Ausfälle und Fehlkonfigurationen proaktiv zu erkennen, bevor sie sich auf die Geschäftslogik auswirken. Das Tool ist speziell für DevOps-Teams und SRE-Ingenieure konzipiert, die eine granular kontrollierbare, asynchrone Ausführungsüberwachung benötigen – ohne schwergewichtige Abhängigkeiten wie Kubernetes oder Prometheus.

## Kernfunktionen

### 1. **Deterministische Execution-Trace-Analyse mit automatischer Anomalie-Korrelation**
cronguard protokolliert jeden Cron-Job-Lauf als strukturierten **Execution-Trace** (inkl. PID, Exit-Code, stdout/stderr, Laufzeit, System-CPU-Load). Bei Abweichungen vom historischen Baseline-Profil (z. B. Laufzeit > 3σ) wird automatisch eine **Anomalie-Korrelation** durchgeführt: Das Tool verknüpft den Fehler mit vorhergehenden Job-Abhängigkeiten, System-Logs und Metriken (z. B. Memory Pressure, I/O-Wait). Ergebnis: Kein Blindflug mehr bei sporadischen Failures – cronguard liefert direkt den Root-Cause-Kontext.

### 2. **Multi-Instance-Synchronisation über dezentrales Consensus-Gateway**
Standard-Cron-Monitore scheitern an Multi-Node-Umgebungen (z. B. mehrere Worker hinter einem Load Balancer). cronguard implementiert ein **dezentrales Consensus-Gateway** auf Basis eines Raft-ähnlichen Protokolls: Jede Instanz meldet ihren geplanten Job-Start und -Ende. Tritt ein Timeout auf (z. B. Job läuft auf Node A, aber Node B startet ihn doppelt), wird der Job **automatisch terminiert** und ein Escalation-Event ausgelöst. Ideal für **High-Availability-Cron-Cluster**, in denen jeder Job genau einmal ausgeführt werden muss.

### 3. **Prädiktive Scheduler-Optimierung via Machine-Learning-basierter Latenzprognose**
cronguard analysiert historische Ausführungsdaten und erstellt ein **Latenzprognosemodell** (LightGBM-basiert, läuft on‑device). Das Modell sagt für jeden Job die erwartete Dauer unter aktuellen Systembedingungen (CPU-Auslastung, Disk-IO, Netzwerklatenz) voraus. Überschreitet die Prognose einen konfigurierbaren Schwellwert (z. B. > 120 % der Baseline), wird der Job automatisch **auf einen späteren Slot verschoben** oder auf einen weniger ausgelasteten Node umgeleitet. So vermeiden Sie Ressourcenkonflikte und reduzieren Latenzspitzen in kritischen Pipelines.

## So verwenden Sie es

### 1. **Initiale Konfiguration und Job-Registrierung mit YAML-basierter DSL**
Erstellen Sie eine Konfigurationsdatei `cronguard.yml` im Stammverzeichnis Ihres Projekts. Definieren Sie jeden Cron-Job als Entry mit Metadaten wie `schedule` (cron‑Ausdruck), `command`, `timeout` und `dependencies`. Beispiel:

```yaml
jobs:
  - name: data_ingestion
    schedule: "0 */2 * * *"
    command: "/usr/local/bin/ingest.sh"
    timeout: 300
    dependencies:
      - db_connection_check
    alerts:
      - type: email
        recipients: ["ops@example.com"]
      - type: webhook
        url: "https://hooks.slack.com/..."
```

Führen Sie dann `cronguard init` aus – das Tool validiert die Syntax, prüft auf zirkuläre Abhängigkeiten und registriert alle Jobs im lokalen Execution-Store (SQLite oder PostgreSQL).

### 2. **Live-Monitoring mit Echtzeit-Dashboard und Alerting-Pipeline**
Starten Sie den cronguard-Agenten im Hintergrund: `cronguard run --daemon`. Der Agent öffnet einen HTTP-Endpunkt auf Port `:8080` (konfigurierbar). Greifen Sie über `http://localhost:8080/dashboard` auf das Web-UI zu. Dort sehen Sie:
- **Job-Status** (Running, Completed, Failed, Skipped)
- **Laufzeit-Histogramm** mit Anomalie-Markern
- **Dependency-Graph** (interaktiv, D3.js-basiert)

Jeder Fehler löst sofort eine **Alerting-Pipeline** aus: cronguard sendet über konfigurierbare Kanäle (Slack, E‑Mail, PagerDuty, Webhook) eine strukturierte Nachricht mit Job‑ID, Exit‑Code, Fehlerkontext und Korrelations‑ID. Sie können zudem **Escalation-Rules** definieren (z. B. nach 3 Fehlern innerhalb von 5 Minuten → Ticket im ITSM‑System öffnen).

### 3. **Prädiktive Optimierung aktivieren und automatische Rescheduling-Richtlinien festlegen**
Aktivieren Sie die ML-Komponente mit `cronguard enable-ml --model-path /opt/cronguard/models`. Der Agent trainiert automatisch ein erstes Modell auf Basis der letzten 500 Job-Läufe. Definieren Sie in der Konfiguration die **Rescheduling-Richtlinie**:

```yaml
optimization:
  enabled: true
  prediction_threshold: 1.2  # 120% der Baseline
  fallback_node: "worker-03"
  max_reschedules_per_job: 3
```

Sobald ein Job die Prognose überschreitet, verschiebt cronguard ihn auf den nächsten verfügbaren Slot (z. B. +10 Minuten) oder leitet ihn an den Fallback-Node weiter. Alle Rescheduling-Ereignisse werden protokolliert und sind im Dashboard als **Scheduled Events** sichtbar.

## Beste Praktiken

### 1. **Granulare Timeout- und Retry-Strategien für geschäftskritische Pipelines**
Setzen Sie für jeden Job einen **individuellen Timeout** (nicht den globalen Standard). Für Datenbank-Migrationsjobs z. B. 600 Sekunden, für Health-Checks 30 Sekunden. Kombinieren Sie dies mit einer **exponentiellen Backoff-Retry-Strategie** (z. B. 1×, 4×, 16× Sekunden). Dies verhindert, dass ein einzelner Job das gesamte Cron-System blockiert, und reduziert die Belastung bei temporären Netzwerkfehlern.

### 2. **Asynchrone Abhängigkeitsauflösung mit Dead-Letter-Queue für fehlgeschlagene Jobs**
Nutzen Sie cronguards **Dependency-Graph**, um Jobs in einer Topologie anzuordnen. Definieren Sie für jeden Job eine **Dead-Letter-Queue** (DLQ): Falls ein abhängiger Job dauerhaft fehlschlägt (z. B. nach 3 Retries), wird der nachgelagerte Job nicht gestartet, sondern in die DLQ verschoben. Dort kann er manuell oder durch ein automatisches Recovery-Skript erneut ausgelöst werden. Dies verhindert kaskadierende Fehler und sorgt für deterministische Ausführungsketten.
