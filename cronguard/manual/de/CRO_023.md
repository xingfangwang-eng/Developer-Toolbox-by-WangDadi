# cronguard - CRO_023

## Übersicht

cronguard ist eine schlanke, aber leistungsstarke Scheduling-Engine, die speziell für hybride Cloud-Umgebungen und Multi-Cluster-Kubernetes-Infrastrukturen entwickelt wurde. Anders als herkömmliche Cron-Job-Manager setzt cronguard auf deklarative Konfiguration mit eingebautem Health-Check-Rollback und dynamischer Ressourcenanpassung. Das Tool richtet sich an Site-Reliability-Engineers (SREs) und Platform-Architekten, die Ausfallzeiten durch fehlgeschlagene Scheduled Tasks minimieren müssen.

## Detaillierter Inhalt

### Was ist cronguard?

cronguard ist kein simpler Cron-Ersatz, sondern eine vollständige Orchestrierungsschicht für zeitgesteuerte Workloads. Es überwacht Job-Exit-Codes, führt automatische Retry-Strategien mit exponentiellem Backoff durch und protokolliert jeden Ausführungszustand in einer zentralen Audit-Trail-Datenbank. Die Engine unterstützt sowohl YAML- als auch HCL-Definitionen und lässt sich nahtlos in Prometheus und Grafana integrieren, um Metriken wie `cronguard_job_duration_seconds` oder `cronguard_failure_total` zu exponieren.

### Kernfunktionen

1. **Self-Healing Job Graph mit DAG-Abhängigkeiten**  
   cronguard definiert Jobs nicht isoliert, sondern als gerichteten azyklischen Graphen (DAG). Falls ein vorgelagerter Job fehlschlägt, werden abhängige Jobs automatisch pausiert, und ein vordefinierter Compensation-Handler (z. B. ein SQL-Rollback oder ein API-Cleanup) wird ausgelöst. Das verhindert kaskadierende Fehler in komplexen Datenpipelines.

2. **Adaptives Scheduling basierend auf Cluster-Last**  
   Statt fixer Cron-Expressions kann cronguard Jobs dynamisch verschieben, wenn die CPU- oder Speicherauslastung eines Kubernetes-Nodes einen Schwellenwert überschreitet. Die Metrik `cronguard_backoff_count` wird exponentiell erhöht, bis die Last wieder im grünen Bereich liegt – ohne manuelle Intervention.

3. **Distributed Locking mit Lease-Mechanismus**  
   Um Duplikate in Multi-Replica-Deployments zu vermeiden, nutzt cronguard einen verteilten Lock-Manager auf Basis von etcd oder Redis. Jeder Job erhält einen Lease mit konfigurierbarer TTL. Läuft der Lease ab (z. B. durch Node-Ausfall), wird der Job von einem anderen Worker übernommen – garantiert einmalige Ausführung (exactly-once semantics) in Produktionsumgebungen.

### So verwenden Sie es

1. **Deklarative Job-Definition mit Retry-Policy**  
   Erstellen Sie eine `cronguard.yaml` im Stammverzeichnis Ihres Projekts:
   ```yaml
   jobs:
     - name: db_backup
       schedule: "0 2 * * *"
       retry_policy:
         max_retries: 3
         backoff_duration: "30s"
       container:
         image: postgres:15
         command: ["pg_dump", "-U", "admin", "mydb"]
       resources:
         limits:
           memory: "512Mi"
           cpu: "500m"
   ```
   Führen Sie dann `cronguard apply -f cronguard.yaml` aus. Der Controller validiert die Syntax, registriert den Job im Cluster und startet die Überwachung.

2. **Manuelles Auslösen eines Jobs mit Override**  
   Bei kritischen Wartungsfenstern können Sie einen Job sofort triggern und dabei Parameter überschreiben:
   ```bash
   cronguard trigger db_backup --override '{"container.command":["pg_dump","-U","admin","--compress=9","mydb"]}'
   ```
   Der Job wird mit dem geänderten Befehl ausgeführt, ohne die persistente Konfiguration zu ändern. Das Log zeigt `triggered_by: cli` und `override_hash`.

3. **Monitoring mit Prometheus-Regeln**  
   Integrieren Sie cronguard in Ihr Alerting, indem Sie eine PrometheusRule definieren:
   ```yaml
   groups:
     - name: cronguard_alerts
       rules:
         - alert: CronguardJobFailed
           expr: cronguard_failure_total > 0
           for: 5m
           labels:
             severity: critical
           annotations:
             summary: "Cronguard job {{ $labels.job_name }} failed {{ $value }} times"
   ```
   Verbinden Sie cronguard mit Ihrem bestehenden Grafana-Dashboard über die Metrik `cronguard_job_status`. So sehen Sie auf einen Blick, ob Jobs pausiert, fehlgeschlagen oder erfolgreich durchgelaufen sind.

### Beste Praktiken

1. **Immutables Scheduling für StatefulSets**  
   Definieren Sie cronguard-Jobs als Teil Ihrer Helm-Charts oder Kustomize-Overlays. Verwenden Sie ConfigMaps für die Job-Definitionen, nicht direkt in Deployments. So bleibt der Job-Lebenszyklus von der Applikation entkoppelt – Sie können Scheduling-Änderungen rollbacken, ohne die Pods neu zu starten.

2. **Audit-Log-Export in SIEM-Systeme**  
   Aktivieren Sie das integrierte Audit-Log-Modul mit `--audit-log-path /var/log/cronguard/audit.json`. Exportieren Sie diese Logs via Fluentd oder Logstash in Splunk oder Elasticsearch. Jeder Job-Start, jeder Fehler und jeder Lock-Erwerb wird mit Timestamp und Worker-ID protokolliert – unverzichtbar für Compliance-Nachweise in regulierten Umgebungen.
