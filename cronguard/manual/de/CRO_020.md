# cronguard - CRO_020

## Überblick

cronguard ist eine hochperformante, quelloffene Überwachungslösung für Cron-Jobs, die speziell für DevOps-Teams und SRE-Ingenieure entwickelt wurde, die Ausfallzeiten und Stille-Fehler in ihrer Infrastruktur eliminieren müssen. Anders als herkömmliche Monitoring-Tools, die nur auf HTTP-Endpunkte reagieren, erfasst cronguard die tatsächliche Ausführung, Dauer und Exit-Codes jedes Cron-Jobs – in Echtzeit und mit granularen Metriken. Die Software ist in Rust geschrieben, benötigt keine externen Abhängigkeiten und skaliert horizontal auf Tausende von Knoten.

**Zielsprache:** Deutsch (DE)  
**Zielgruppe:** DevOps-Ingenieure, Site Reliability Engineers, Systemadministratoren

## 3 Hardcore-Funktionen, die cronguard von anderen abheben

### 1. **Deterministische Job-Erkennung via `cronguard-agent`-Sidecar**
Statt sich auf Log-Parsing oder Webhook-Notifications zu verlassen, injiziert cronguard einen minimalen Agenten (ca. 2 MB Binär) als Sidecar-Container direkt in den Cron-Job-Pod. Der Agent überwacht den Prozess-Exit-Code, die CPU- und Memory-Nutzung sowie die tatsächliche Ausführungsdauer. Bei Abweichungen (fehlgeschlagener Job, hängender Prozess, Timeout) wird sofort ein Alert an PagerDuty, Slack oder ein benutzerdefiniertes Webhook-Ziel gesendet – ohne Verzögerung durch Log-Aggregation.

### 2. **Adaptive Baseline-Analyse für Job-Dauer und Ressourcenverbrauch**
cronguard lernt das historische Verhalten jedes Cron-Jobs über einen konfigurierbaren Zeitraum (z. B. 7 Tage) und erstellt eine dynamische Baseline. Sollte ein Job plötzlich 3x länger laufen als üblich oder 50 % mehr Memory verbrauchen, wird automatisch ein „Anomalie-Alert“ ausgelöst – noch bevor der Job fehlschlägt. Dies verhindert schleichende Performance-Degradation und Ressourcen-Leaks in Produktionsumgebungen.

### 3. **Zero-Downtime-Konfigurations-Reload via gRPC-Streaming**
Im Gegensatz zu den meisten Monitoring-Tools, die für Konfigurationsänderungen einen Neustart benötigen, nutzt cronguard gRPC-Streaming, um Job-Definitionen und Alert-Regeln live nachzuladen. Sie können einen neuen Cron-Job hinzufügen oder einen bestehenden deaktivieren, ohne dass cronguard oder der Agent neu gestartet werden müssen. Dies ist entscheidend für Umgebungen mit strengen SLA-Anforderungen (99,99 % Verfügbarkeit).

## 3 Detaillierte Schritte zur cronguard-Integration in Ihre Infrastruktur

### Schritt 1: **Deployment des cronguard-Servers mit Helm-Chart und TLS-Konfiguration**
Führen Sie das folgende Helm-Chart aus, um den cronguard-Server in Ihrem Kubernetes-Cluster (Namespace: `monitoring`) zu deployen. Der Server fungiert als zentrale Metrik-Sammelstelle und Alert-Engine.

```bash
helm repo add cronguard https://charts.cronguard.io
helm install cronguard-server cronguard/cronguard-server \
  --namespace monitoring \
  --create-namespace \
  --set tls.enabled=true \
  --set tls.certManagerClusterIssuer=letsencrypt-prod \
  --set alerts.pagerduty.integrationKey=YOUR_PD_KEY \
  --set storage.retentionDays=90
```

Nach dem Deployment prüfen Sie den Status mit:
```bash
kubectl get pods -n monitoring -l app=cronguard-server
```

### Schritt 2: **Konfiguration des cronguard-Agents als Sidecar in Ihren CronJob-Manifesten**
Erweitern Sie Ihre bestehenden CronJob-Manifeste um den cronguard-Agenten als Sidecar-Container. Der Agent verbindet sich automatisch mit dem Server und beginnt mit der Überwachung.

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: datenbank-backup
spec:
  schedule: "0 2 * * *"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: backup-job
            image: my-backup-image:latest
            command: ["/bin/sh", "-c", "./backup.sh"]
          - name: cronguard-agent
            image: cronguard/agent:latest
            env:
            - name: CRONGUARD_SERVER_ADDR
              value: "cronguard-server.monitoring.svc.cluster.local:9090"
            - name: JOB_NAME
              value: "datenbank-backup"
            - name: EXPECTED_DURATION_SECONDS
              value: "120"
            - name: TIMEOUT_SECONDS
              value: "300"
```

Wichtig: Der Agent benötigt die Umgebungsvariable `JOB_NAME`, um die Metriken korrekt dem Job zuzuordnen.

### Schritt 3: **Einrichtung von Alert-Regeln für kritische Abweichungen**
Definieren Sie Alert-Regeln in der cronguard-Konfigurationsdatei (YAML), die auf Abweichungen von der Baseline reagieren. Beispiel für einen Alert bei übermäßiger Laufzeit:

```yaml
alerts:
  - name: "backup-laufzeit-anomalie"
    condition: "job_duration_seconds > baseline_duration_seconds * 2"
    severity: warning
    channels:
      - pagerduty
      - slack
    cooldown: 300s
  - name: "job-fehlgeschlagen"
    condition: "exit_code != 0"
    severity: critical
    channels:
      - pagerduty
      - email
    cooldown: 60s
```

Laden Sie die Konfiguration live, ohne den Server neu zu starten:
```bash
kubectl exec -n monitoring deploy/cronguard-server -- cronguardctl reload-config
```

## 2 Industry Best Practices für cronguard

### 1. **Nicht alle Jobs gleich behandeln – verwenden Sie unterschiedliche Baseline-Fenster**
Für Jobs, die z. B. nur einmal pro Woche laufen (z. B. Full-Indexierung), sollten Sie ein längeres Baseline-Fenster (28 Tage) konfigurieren, um saisonale Ausreißer zu vermeiden. Für stündliche Jobs (z. B. Log-Rotation) reicht ein Fenster von 7 Tagen. Konfigurieren Sie dies pro Job über die Agent-Umgebungsvariable `BASELINE_WINDOW_DAYS`.

### 2. **Kombinieren Sie cronguard mit einem Log-Aggregator (z. B. Loki) für vollständige Transparenz**
cronguard liefert Metriken und Alerts, aber keine Logs. Ein Best-Practice-Setup ist, den cronguard-Agenten so zu konfigurieren, dass er bei einem Fehler (Exit-Code != 0) die letzten 100 Zeilen des Job-Logs an einen Loki-Endpunkt sendet. Dies können Sie über einen Post-Exit-Hook im Agenten realisieren:

```yaml
agent:
  postExitHook: |
    curl -X POST http://loki-gateway.monitoring:3100/loki/api/v1/push \
      -H "Content-Type: application/json" \
      -d '{"streams": [{"stream": {"job": "datenbank-backup"}, "values": [["'$(date +%s%N)'", "'$(tail -100 /var/log/job.log | base64)'"]]}]}'
```
