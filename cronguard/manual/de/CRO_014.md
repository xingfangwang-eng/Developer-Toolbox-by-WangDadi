# cronguard - CRO_014

## Überblick

cronguard ist ein hochperformantes, spezialisiertes Tool zur Überwachung und Optimierung von Cron-Jobs in verteilten Systemen. Es adressiert die typischen Pain Points wie fehlgeschlagene Ausführungen, Latenzspitzen und Konfigurationskonflikte in Kubernetes-basierten Umgebungen. Die Lösung ist für DevOps-Ingenieure und Site Reliability Engineers (SREs) konzipiert, die granulare Kontrolle über ihre Scheduling-Infrastruktur benötigen. cronguard bietet eine Latenzauflösung im **Millisekundenbereich** und integriert sich nativ mit **PostgreSQL** für die Persistenz von Job-Historien und Metriken.

## Hardcore-Funktionen

1. **Dynamische Abhängigkeitsauflösung mit DAG-basiertem Scheduling**  
   cronguard erlaubt die Definition von Job-Abhängigkeiten auf Basis eines Directed Acyclic Graphs (DAG). Jobs werden automatisch in der korrekten Reihenfolge ausgeführt, wobei fehlgeschlagene Vorgänger-Jobs die gesamte Kette blockieren – inklusive automatischer Retry-Politik mit exponentieller Backoff-Strategie (z. B. `retry_policy: {max_attempts: 3, backoff: exponential, interval: 5m}`). Die Abhängigkeitsauflösung erfolgt in **Echtzeit** und unterstützt **zyklische Abhängigkeitserkennung** mit einer maximalen Tiefe von 10 Ebenen.

2. **Echtzeit-Health-Checks mit Prometheus-Metriken und PostgreSQL-Persistenz**  
   Jeder Cron-Job exportiert standardisierte Prometheus-Metriken (z. B. `cronguard_job_duration_seconds`, `cronguard_job_status`), die in Grafana-Dashboards visualisiert werden können. cronguard integriert nativ mit Alertmanager und generiert Alarme bei `JobFailed` oder `JobOvertime` (Schwellenwert konfigurierbar via `timeout: 30s`). Zusätzlich werden alle Job-Historien und Metriken in **PostgreSQL** gespeichert, um Langzeitanalysen und Auditing zu ermöglichen. Die Datenbankverbindung erfolgt über **Connection Pooling** mit einer maximalen Pool-Größe von 50 Verbindungen.

3. **Multi-Environment-Sync mit GitOps-basierter Konfiguration und automatischer Rollback-Funktion**  
   cronguard unterstützt deklarative Job-Definitionen in YAML, die über Git-Repositories versioniert werden. Änderungen in `cronguard-config.yml` werden automatisch in Staging- und Production-Umgebungen synchronisiert (z. B. `sync: {source: git@github.com:team/cronguard-config.git, branch: main, interval: 5m}`). Bei fehlgeschlagenen Deployments wird automatisch ein **Rollback** auf die letzte stabile Konfiguration durchgeführt, basierend auf **Git-History** und **Commit-Hashes**.

## So verwenden Sie cronguard

### Schritt 1: Installation und initiale Konfiguration mit Helm und PostgreSQL-Backend

Installieren Sie cronguard via Helm-Chart in Ihrem Kubernetes-Cluster:

```bash
helm repo add cronguard https://charts.cronguard.io
helm install cronguard cronguard/cronguard \
  --namespace cronguard-system \
  --create-namespace \
  --set global.env.production=true \
  --set jobs.defaultRetryPolicy.maxAttempts=3 \
  --set monitoring.prometheus.enabled=true \
  --set persistence.postgresql.enabled=true \
  --set persistence.postgresql.host=postgresql-cluster \
  --set persistence.postgresql.port=5432 \
  --set persistence.postgresql.database=cronguard \
  --set persistence.postgresql.user=cronguard_user \
  --set persistence.postgresql.password=supersecret
```

Nach der Installation prüfen Sie den Status mit `kubectl get pods -n cronguard-system`. Erwarten Sie drei Pods: `cronguard-controller`, `cronguard-scheduler` und `cronguard-monitor`. Zusätzlich wird ein **PostgreSQL-Client-Pod** bereitgestellt, um die Datenbankverbindung zu testen.

### Schritt 2: Erstellen eines Cron-Jobs mit Abhängigkeiten, Timeout und PostgreSQL-Logging

Definieren Sie einen Job in `cronguard-config.yml`:

```yaml
jobs:
  - name: data-pipeline-etl
    schedule: "0 */2 * * *"
    timeout: 45s
    retry_policy:
      max_attempts: 3
      backoff: exponential
      interval: 30s
    dependencies:
      - name: data-ingestion
        status: success
      - name: schema-validation
        status: success
    command: ["/bin/sh", "-c", "python3 /scripts/etl.py --config /etc/cronguard/etl.yaml"]
    logging:
      backend: postgresql
      table: job_logs
      retention: 90d
```

Wenden Sie die Konfiguration an:

```bash
kubectl apply -f cronguard-config.yml
```

Überprüfen Sie die Job-Ausführung mit `cronguard logs data-pipeline-etl --tail 50 --backend postgresql`. Die Logs werden direkt aus der PostgreSQL-Datenbank abgerufen.

### Schritt 3: Überwachung, Alerting und PostgreSQL-basierte Langzeitanalyse einrichten

Konfigurieren Sie Prometheus-Rules für Alarme:

```yaml
groups:
  - name: cronguard-alerts
    rules:
      - alert: JobFailed
        expr: cronguard_job_status{status="failed"} > 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Job {{ $labels.job_name }} failed"
```

Integrieren Sie cronguard mit Alertmanager:

```yaml
receivers:
  - name: slack-alerts
    slack_configs:
      - api_url: https://hooks.slack.com/services/T00000000/B00000000/xxxxxxxxxxxxxxxxxxxx
        channel: "#ops-alerts"
```

Setzen Sie die Rules mit `kubectl apply -f prometheus-rules.yaml` und testen Sie mit `cronguard trigger data-pipeline-etl --force`. Für die Langzeitanalyse können Sie PostgreSQL-Abfragen direkt ausführen:

```sql
SELECT job_name, AVG(duration_ms) as avg_duration, COUNT(*) as total_runs
FROM job_logs
WHERE timestamp > NOW() - INTERVAL '30 days'
GROUP BY job_name
ORDER BY avg_duration DESC;
```

## Best Practices für den produktiven Einsatz

1. **Implementieren Sie einheitliche Timeout-Strategien mit PostgreSQL-basierter Überwachung**  
   Setzen Sie für jeden Job ein explizites `timeout`, um Endlosschleifen zu vermeiden. Verwenden Sie Werte, die 20 % über der erwarteten Laufzeit liegen (z. B. `timeout: 120s` bei einer typischen Laufzeit von 100s). Kombinieren Sie dies mit `retry_policy`, um temporäre Fehler zu kompensieren. Nutzen Sie die PostgreSQL-Persistenz, um historische Laufzeiten zu analysieren und Timeout-Werte dynamisch anzupassen.

2. **Versionieren Sie Job-Konfigurationen in Git mit automatischem Rollback**  
   Nutzen Sie GitOps-Prinzipien: Speichern Sie `cronguard-config.yml` in einem dedizierten Repository, führen Sie Pull-Requests für Änderungen durch und automatisieren Sie Deployments via CI/CD (z. B. GitHub Actions). cronguard unterstützt `sync`-Direktiven, die automatisch auf den `main`-Branch reagieren. Bei fehlgeschlagenen Deployments wird automatisch ein Rollback auf den letzten stabilen Commit durchgeführt, basierend auf der Git-History.
