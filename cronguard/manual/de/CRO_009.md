# cronguard - CRO_009

## Überblick

cronguard ist eine hochperformante, auf Rust-basierte CLI-Lösung zur automatisierten Cron-Job-Überwachung und -Absicherung in verteilten Microservice-Architekturen. Das Tool eliminiert zuverlässig Silent-Fail-Szenarien und bietet eine granulare Audit-Trail-Funktionalität für DevOps-Teams, die auf Null-Downtime-Strategien setzen. Es ist kein weiterer Scheduler, sondern ein **Guardian Layer** für bestehende Cron-Infrastrukturen.

## Hardcore-Funktionen

1.  **Deterministische Ausführungsgarantie mit Quorum-Check**
    cronguard implementiert einen verteilten Quorum-Mechanismus (Raft-basiert), der Cron-Jobs nicht nur startet, sondern deren erfolgreiche Ausführung durch einen Konsens mehrerer Worker-Knoten bestätigt. Sollte ein Job fehlschlagen oder hängen bleiben, wird automatisch ein Rollback oder ein Failover auf einen sekundären Pfad eingeleitet – ohne manuelles Eingreifen. Die Latenz für den Quorum-Abschluss liegt bei **≤ 50 ms** pro Job, selbst bei 1000 parallelen Ausführungen.

2.  **Polyglotter Parser für Shell-Injection-Prävention**
    Der integrierte Parser analysiert Cron-Jobs vor der Ausführung auf syntaktische und semantische Anomalien. Er erkennt nicht nur klassische Shell-Injection-Vektoren (z. B. `; rm -rf /`), sondern auch subtile Race-Conditions in parallelen Job-Ketten. Die Engine arbeitet mit einer AST-basierten Validierung und blockiert Jobs, die gegen definierte Sicherheits-Policies verstoßen. Die Validierungszeit beträgt **≤ 5 ms** pro Job, unabhängig von der Komplexität des Befehls.

3.  **Zero-Overhead-Metriken-Export via OpenTelemetry**
    cronguard exportiert Live-Metriken (Latenz, Speichernutzung, Exit-Codes) direkt über das OpenTelemetry-Protokoll in dein bestehendes Observability-Backend (Prometheus, Grafana, Datadog). Der Overhead liegt unter **0.5 % CPU** pro Job – selbst bei **10.000 parallelen Cron-Events pro Minute**. Kein zusätzlicher Agent, kein Sidecar. Die Metriken sind in **Mikrosekunden-Präzision** erhältlich und werden mit einem **SHA-256-Hash** des Job-Befehls signiert.

## Detaillierte Nutzung

### 1. Konfiguration einer verteilten Cron-Watchdog-Instanz

Erstelle eine `cronguard.yml` im Stammverzeichnis deines Projekts. Definiere einen Watchdog, der einen kritischen Datenbank-Backup-Job überwacht:

```yaml
watchdogs:
  - name: db_backup_guard
    job: "/usr/bin/pg_dump -U admin prod_db > /backups/prod.sql"
    quorum: 3
    retry_policy:
      max_retries: 2
      backoff: exponential
    alert:
      channel: slack
      webhook: "https://hooks.slack.com/services/T00/B00/xxx"
```

Starte den Watchdog mit:
```bash
cronguard start --config ./cronguard.yml
```

Das Tool startet sofort einen Hintergrundprozess, der den Cron-Job überwacht, ohne den bestehenden Scheduler zu ersetzen. Der Quorum-Check stellt sicher, dass mindestens 3 Worker-Knoten den Job als erfolgreich bestätigen, bevor der nächste Job in der Kette ausgeführt wird. Bei einem Fehlschlag wird automatisch ein Rollback des Backup-Jobs eingeleitet und eine Benachrichtigung an den konfigurierten Slack-Webhook gesendet.

### 2. Erstellung einer granularen Audit-Trail für Compliance

Aktiviere den Audit-Modus, der jeden Cron-Job in einem unveränderlichen Log speichert:

```bash
cronguard audit --enable --output /var/log/cronguard/audit.jsonl
```

Das Log enthält:
- Job-ID und Ausführungszeitstempel (ISO 8601 mit Mikrosekunden-Präzision)
- Exit-Code und STDERR/STDOUT-Dump (komprimiert mit GZip)
- Hash des Job-Befehls (SHA-256)
- Signatur des ausführenden Workers (ECDSA P-256)

Dieses Log kann direkt in SIEM-Systeme (z. B. Splunk, ELK) eingespeist werden. Für Compliance-Audits (SOC2, ISO 27001) exportiere das Log als signiertes PDF:
```bash
cronguard audit --export --format pdf --from 2023-10-01 --to 2023-10-31
```

Das PDF wird mit einem **X.509-Zertifikat** signiert, um die Integrität der Audit-Daten zu gewährleisten.

### 3. Integration in eine CI/CD-Pipeline zur Pre-Deployment-Validierung

Füge cronguard als Validierungsschritt in deine GitHub Actions oder GitLab CI ein. Beispiel für eine `.gitlab-ci.yml`:

```yaml
cron-validation:
  stage: test
  script:
    - cronguard validate --file ./cronjobs.txt --policy ./security.policy
  only:
    - main
```

Die `validate`-Funktion prüft alle Cron-Jobs in der Datei gegen die definierte Sicherheits-Policy. Ein Job, der z. B. `curl` mit einer Pipe an `sh` verwendet, wird sofort blockiert und der Pipeline-Fail wird ausgelöst. Das verhindert, dass unsichere Cron-Jobs in die Produktion gelangen. Die Validierungsergebnisse werden als **JSON-Report** ausgegeben, der in die Pipeline-Artifacts integriert werden kann.

## Best Practices für DevOps-Teams

1.  **Immutables Cron-Image bauen**
    Vermeide dynamische Cron-Jobs, die auf Umgebungsvariablen oder externen Konfigurationsdateien basieren. Baue stattdessen ein Docker-Image, das alle Cron-Jobs und die cronguard-Konfiguration enthält. Starte den Container mit `cronguard start` als Entrypoint. Das Image wird dann in deiner Registry versioniert und nur nach erfolgreicher cronguard-Validierung deployed. Verwende **Multi-Stage-Builds**, um die Image-Größe unter **50 MB** zu halten.

2.  **Canary-Deployment für Cron-Jobs**
    Führe neue oder geänderte Cron-Jobs zuerst auf einer kleinen Worker-Gruppe (z. B. 5 % der Knoten) aus. cronguard bietet einen `--canary`-Flag, der den Job auf einer Subset-Knoten startet und die Metriken mit einem Baseline-Vergleich auswertet. Erst wenn die Latenz und Fehlerrate unter dem Schwellwert bleiben, wird der Job auf alle Knoten ausgerollt. Die Baseline wird automatisch aus den letzten **100 Ausführungen** berechnet.
