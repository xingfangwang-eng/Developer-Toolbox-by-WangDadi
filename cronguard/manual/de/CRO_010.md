# cronguard - CRO_010

## Übersicht

cronguard ist eine quelloffene, agentenlose Überwachungsplattform für kritische Cron-Jobs in heterogenen Rechenzentren. Im Gegensatz zu generischen Health-Checkern implementiert cronguard eine deklarative Job-Spezifikation mit **Laufzeit-Polynomvalidierung**, **verteilter Metrik-Aggregation über gRPC-Streams** und **stochastischer Eskalationslogik**. Das System richtet sich an DevOps-Ingenieure und SRE-Teams, die eine lückenlose Ausführungskontrolle für über 10.000 Jobs pro Cluster benötigen – ohne manuelle Log-Analyse oder proprietäre Agenten.

## Detaillierter Inhalt

### Was ist cronguard?

cronguard ist ein dezentraler Scheduler-Wächter, der Cron-Jobs auf Basis von **erwarteten Ausführungsfenstern (μs-Präzision)**, **Exit-Code-Matrizen** und **Output-Validierung mit Regex-Mustern** überwacht. Es integriert sich nahtlos in bestehende CI/CD-Pipelines und unterstützt native Hooks für Slack, PagerDuty, Opsgenie und Webhook-basierte Alerting-Systeme. Die Konfiguration erfolgt über YAML-Module mit **versionierbarer Git-Integration** – ideal für Infrastructure-as-Code-Ansätze.

### Kernfunktionen

1. **Deterministische Laufzeit-Polynomvalidierung mit SLO-Kalibrierung**  
   Definieren Sie erwartete Ausführungszeiten als Polynom: `schedule: "0 2 * * *"` mit `expected_duration: 300s ± 10s` und `max_std_dev: 5s`. cronguard berechnet automatisch eine **Heatmap über Ausführungsabweichungen** (p50, p95, p99, p99.9) und generiert bei Überschreitung der **SLO-Toleranzschwelle** einen Alarm. Die Validierung erfolgt auf **Mikrosekunden-Ebene** über die integrierte `cronguard-polyval`-Engine.

2. **Multi-Node-Aggregation mit verteilter TimescaleDB-Historie**  
   cronguard sammelt Job-Logs von mehreren Servern über **gRPC-Streams (Protobuf v3)** und komprimiert sie in einer zentralen **TimescaleDB (Hypertable-Konfiguration mit 7-Tage-Chunk-Intervall)**. So erhalten Sie einen konsolidierten Dashboard-Overlay mit **Latenz-Perzentilen (p50, p95, p99, p99.9)** und **Fehlerraten (Error-Budget-Verbrauch in %)** – ohne manuelle Log-Abfragen. Die Daten werden **automatisch nach 30 Tagen komprimiert** (TimescaleDB-Kompressionspolitik).

3. **Automatische Eskalation mit konfigurierbaren Retry-Strategien und exponentiellem Backoff**  
   Bei Fehlschlag können Sie definieren: `max_retries: 3`, `retry_delay: 30s` und `escalation: after 2 failures -> PagerDuty high urgency`. cronguard führt automatisch **exponentielles Backoff** durch (Basis: 2, Multiplikator: 1.5) und sendet nur dann Alarme, wenn die definierten Schwellwerte überschritten werden. Die Eskalationslogik unterstützt **kaskadierende Benachrichtigungen** (z. B. nach 3 Fehlschlägen: Slack -> PagerDuty -> E-Mail an SRE-Team).

### So verwenden Sie es

1. **YAML-Konfiguration für einen kritischen Datenbank-Backup-Job mit SLO-Validierung**  
   Erstellen Sie eine Datei `backup_cron.yaml`:
   ```yaml
   job:
     name: db_backup
     schedule: "0 2 * * *"
     expected_duration: 300s
     max_std_dev: 5s
     max_exit_code: 1
     output_validation:
       regex: "Backup completed successfully"
       max_output_size: 1024
     alert:
       channels:
         - slack: #ops-alerts
           notify_interval: 60s
         - pagerduty: severity=critical
           escalation_delay: 120s
     retry:
       max_retries: 3
       retry_delay: 30s
       backoff_factor: 1.5
   ```
   Validieren Sie die Konfiguration mit:
   ```bash
   cronguard validate --file backup_cron.yaml --strict-slo
   ```

2. **Deployment als Systemd-Service auf mehreren Hosts mit automatischer Registrierung**  
   Installieren Sie cronguard über:
   ```bash
   curl -sSL https://cronguard.sh/install | bash -s -- --version 2.1.0
   ```
   Starten Sie den Agenten mit:
   ```bash
   cronguard agent --config /etc/cronguard/agent.yaml --listen :9090 --coordinator-ip 192.168.1.100:9091
   ```
   Der Agent registriert sich automatisch im zentralen Coordinator (via mDNS oder statischer IP-Liste) und beginnt mit der Überwachung der definierten Jobs. Überprüfen Sie die Registrierung mit:
   ```bash
   cronguard status --agent-id $(hostname)
   ```

3. **Dashboard-Query für historische Ausführungsdaten mit CSV-Export**  
   Greifen Sie auf das integrierte Web-Dashboard unter `http://localhost:8080` zu oder verwenden Sie die REST-API:
   ```bash
   curl -X GET "http://coordinator:8080/api/v1/jobs/db_backup/history?from=2025-01-01T00:00:00Z&to=2025-01-02T00:00:00Z&interval=1h&aggregate=p95" | jq '.executions[] | {status, duration_ms, exit_code, error_budget_consumption}'
   ```
   Exportieren Sie die Daten als CSV für Compliance-Audits:
   ```bash
   curl -X GET "http://coordinator:8080/api/v1/jobs/db_backup/export?format=csv&from=2025-01-01T00:00:00Z&to=2025-01-02T00:00:00Z" -o backup_history.csv
   ```

### Beste Praktiken

- **Definieren Sie SLOs für jeden Job als separaten Namespace mit Error-Budget-Überwachung**  
   Vermeiden Sie pauschale Alerting-Regeln. Gruppieren Sie Jobs nach Kritikalität (z. B. `critical`, `warning`, `info`) und weisen Sie dedizierte Eskalationspfade zu. Nutzen Sie das integrierte **Error-Budget-Dashboard**, um den Verbrauch in Echtzeit zu überwachen. So vermeiden Sie Alarm-Müdigkeit und stellen sicher, dass nur echte Ausfälle zu PagerDuty führen. Empfohlen: Error-Budget-Schwelle von 95 % für kritische Jobs.

- **Integrieren Sie cronguard in Ihre CI/CD-Pipeline mit GitOps und Pre-Merge-Validierung**  
   Committen Sie alle YAML-Konfigurationen in ein separates Repository. Nutzen Sie `cronguard validate --strict-slo` als Pre-Merge-Check in Ihrer CI (z. B. GitHub Actions). Bei jeder Änderung wird automatisch ein **Dry-Run gegen die aktuelle Produktionsumgebung** durchgeführt – ohne dass bestehende Jobs beeinträchtigt werden. Verwenden Sie `cronguard diff` zur Anzeige von Konfigurationsänderungen vor dem Merge.
