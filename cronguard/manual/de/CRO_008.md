# cronguard - CRO_008

## Übersicht

cronguard ist eine hochperformante SaaS-Lösung zur Überwachung und Absicherung von Cron-Jobs in verteilten Systemen, speziell entwickelt für DevOps-Teams, die auf Null-Ausfallzeit und granulare Kontrolle angewiesen sind. Die Plattform unterstützt native Kubernetes CronJobs, AWS EventBridge und GCP Cloud Scheduler mit einer gemeinsamen Konsole für bis zu 10.000 Jobs pro Cluster.

## Detaillierter Inhalt

### Was ist cronguard?

cronguard automatisiert die Überwachung, Protokollierung und Fehlerbehandlung von Cron-Jobs in Cloud-nativen Umgebungen. Es eliminiert manuelle Checks, reduziert Alarmmüdigkeit und bietet eine zentrale Konsole für die Verwaltung von über 10.000 Jobs pro Cluster. Mit nativer Unterstützung für Kubernetes CronJobs und AWS EventBridge wird cronguard zum unverzichtbaren Bestandteil jeder CI/CD-Pipeline.

### Kernfunktionen

- **Distributed Lock mit Quorum-basierter Ausführungsgarantie**  
  cronguard implementiert einen Raft-basierten Lock-Mechanismus, der sicherstellt, dass jeder Cron-Job genau einmal ausgeführt wird – selbst bei Node-Failover. Die Retry-Engine mit konfigurierbarer exponentieller Backoff-Strategie (Basis: 2s, Faktor: 2, maximale Verzögerung: 300s) und Slack-Webhook-Integration verhindert Fehlalarme und reduziert manuelle Eingriffe um 80%. Der Lock-Timeout beträgt standardmäßig 30s, konfigurierbar über `cronguard.lock_ttl`.

- **Prometheus-basiertes Echtzeit-Monitoring mit Anomalieerkennung**  
  Jeder Job wird mit Prometheus-Metriken instrumentiert (Histogramme für Laufzeit in Millisekunden, Speichernutzung in MB, Exit-Codes). cronguard erstellt automatisch Heatmaps und benachrichtigt nur bei Abweichungen vom historischen Baseline-Muster (P95, P99). Die Anomalieerkennung nutzt einen saisonalen ARIMA-Algorithmus, der Drift erkennt, bevor er zum Ausfall wird. Standard-Schwellenwert: Abweichung > 20% vom P95 über 3 aufeinanderfolgende Ausführungen.

- **Unveränderliches Audit-Log mit rollenbasierter Zugriffssteuerung auf Namespace-Ebene**  
  Jede Aktion (Start, Stop, Konfigurationsänderung) wird in einem unveränderlichen Audit-Log gespeichert, das mit SHA-256 gehasht und in PostgreSQL persistiert wird. Dank RBAC auf Namespace-Ebene (standardmäßig 5 Rollen: Admin, Editor, Operator, Viewer, Auditor) können Teams eigenständig Jobs verwalten, ohne globale Sicherheitsrichtlinien zu umgehen. Ideal für SOC2- und ISO27001-konforme Umgebungen. Die Aufbewahrungsdauer beträgt standardmäßig 365 Tage.

### So verwenden Sie es

1. **Job-Definition als YAML mit cronguard-Spezifikation deployen**  
   Erstellen Sie eine `cronguard.yaml` im Projektstamm, die Ihre Cron-Jobs beschreibt. Beispiel:  
   ```yaml
   cronjobs:
     - name: db-backup
       schedule: "0 3 * * *"
       command: ["pg_dump", "-U", "admin", "prod_db"]
       retry:
         max_attempts: 3
         backoff: exponential
         base_delay: 2s
         max_delay: 300s
       alerts:
         - type: webhook
           url: https://hooks.slack.com/...
         - type: pagerduty
           severity: critical
       metrics:
         enabled: true
         histogram_buckets: [0.1, 0.5, 1, 2.5, 5, 10]
   ```  
   Führen Sie `cronguard apply --namespace=production` aus. Der Agent parst die Spezifikation, validiert die Syntax (inklusive Cron-Expression-Validierung) und erstellt die entsprechenden Kubernetes CronJobs inklusive Sidecar-Container für Metriken. Der gesamte Vorgang dauert weniger als 500ms.

2. **Alerting-Schwellenwerte und Eskalationsmatrix konfigurieren**  
   Rufen Sie das Dashboard unter `cronguard dashboard --port=9090` auf. Navigieren Sie zu "Alert Policies" und definieren Sie Regeln wie:  
   - Wenn Laufzeit > 2x historisches P95 (berechnet über 30 Tage) → Slack-Benachrichtigung  
   - Wenn Exit-Code != 0 → PagerDuty-Eskalation nach 2 Minuten (konfigurierbar)  
   - Wenn Job 3x hintereinander fehlschlägt → automatischer Rollback auf letzte stabile Version (definiert in `cronguard.rollback_version`)  
   - Wenn Speichernutzung > 80% des Pod-Limits → E-Mail-Benachrichtigung  
   Speichern Sie die Konfiguration. cronguard übernimmt sofort die Überwachung und validiert die Regeln gegen die aktuelle Job-Konfiguration.

3. **Retrospective Analyse mit cronguard logs**  
   Nutzen Sie das CLI-Tool zur gezielten Fehlersuche:  
   ```bash
   cronguard logs db-backup --since 24h --status failed --format json --namespace=production | jq '.'
   ```  
   cronguard aggregiert Logs aus allen Replicas, filtert nach Exit-Codes und zeigt die genaue Fehlerursache (z. B. Timeout nach 30s, OutOfMemory mit 512MB Limit). Exportieren Sie die Ergebnisse als CSV für Compliance-Audits:  
   ```bash
   cronguard logs db-backup --since 7d --format csv --output failed_jobs.csv
   ```  
   Die CSV enthält Spalten: Job-Name, Ausführungszeit, Exit-Code, Fehlermeldung, Dauer (ms), Speichernutzung (MB), Node-Name.

### Beste Praktiken

- **Definieren Sie für jeden Job eine eindeutige ID und einen Health-Check-Endpunkt**  
  In verteilten Systemen kann ein Job auf mehreren Nodes gleichzeitig starten. Verwenden Sie `cronguard.lock_id` in der YAML-Definition (z. B. `lock_id: "db-backup-lock-${NAMESPACE}"`), um Kollisionen zu vermeiden. Kombinieren Sie dies mit einem externen Health-Check (z. B. HTTP 200 auf einem internen Endpunkt wie `http://health-check.internal:8080/ready`), den cronguard vor dem Start eines Jobs prüft. So stellen Sie sicher, dass abhängige Dienste verfügbar sind. Der Health-Check-Timeout beträgt standardmäßig 5s.

- **Nutzen Sie die integrierte Kapazitätsplanung für saisonale Lastspitzen**  
  cronguard bietet einen "Load Forecast"-Modus, der historische Ausführungsdaten (standardmäßig 90 Tage) analysiert und vorhersagt, wann ein Job mehr Ressourcen benötigt (z. B. am Monatsende). Aktivieren Sie diese Funktion im Dashboard unter "Optimization" > "Load Forecast". cronguard skaliert dann automatisch die Pod-Ressourcenlimits hoch (CPU und Memory), bevor der Job startet – ohne manuelle Anpassung. Die Vorhersagegenauigkeit beträgt typischerweise > 85% bei wöchentlichen Mustern.
