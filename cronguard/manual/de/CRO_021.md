# cronguard - CRO_021

## Überblick

cronguard ist eine hochperformante, quelloffene Monitoring-Engine für Cron-Jobs, die speziell für DevOps-Teams entwickelt wurde, die latenzkritische Workloads in Kubernetes-Umgebungen betreiben. Anders als traditionelle Cron-Manager setzt cronguard auf einen ereignisgesteuerten Checkpoint-Mechanismus, der Ausfälle mit subsekundengenauer Alarmierung erfasst – ideal für CI/CD-Pipelines und Batch-Jobs mit SLA-Anforderungen.

## Kernfunktionen

### 1. **Distributed Checkpoint Reconciliation (DCR)**
Statt einfacher Heartbeat-Prüfungen nutzt cronguard einen konsensbasierten Checkpoint-Ansatz: Jeder Cron-Job hinterlegt nach erfolgreicher Ausführung einen kryptografisch signierten Checkpoint in einer verteilten Key-Value-Store (z. B. etcd oder Redis Cluster). Fehlschläge werden nicht nur erkannt, sondern automatisch mit dem letzten gültigen Checkpoint abgeglichen, um falsch positive Alarme aufgrund von Netzwerkpartitionen zu vermeiden.

### 2. **Adaptive Execution Latency Profiling (AELP)**
cronguard analysiert die historische Ausführungsdauer jedes Jobs und erstellt ein dynamisches Latenzprofil. Weicht die aktuelle Ausführungszeit um mehr als 3 Standardabweichungen vom gleitenden Mittelwert ab, wird ein **SLO-Verletzungsalarm** ausgelöst – ohne dass feste Timeout-Schwellwerte manuell definiert werden müssen. Dies reduziert Rauschen und optimiert die Alarmkorrelation für Batch-Verarbeitungsfenster.

### 3. **Multi-Cloud Cron Audit Trail mit Immutable Logging**
Jeder Cron-Event wird in einem append-only Log (Apache Kafka oder AWS Kinesis) persistiert, der mit HMAC-SHA256 signiert ist. cronguard stellt eine `cronctl audit`-CLI bereit, die es erlaubt, Ausführungsverläufe über mehrere Cloud-Regionen hinweg zu replizieren und auf Compliance (z. B. SOC2, GDPR) zu prüfen.

## Detaillierte Anwendungsschritte

### 1. **Integration in bestehende Kubernetes-CronJobs via Sidecar-Pattern**
Fügen Sie cronguard als Sidecar-Container zu Ihrem vorhandenen CronJob-Manifest hinzu. Konfigurieren Sie die Umgebungsvariable `CRONGUARD_CHECKPOINT_URI` auf Ihren etcd-Endpunkt. Das folgende Snippet zeigt eine minimalistische Konfiguration:

```yaml
apiVersion: batch/v1
kind: CronJob
spec:
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: cron-task
            image: your-cron-image:latest
          - name: cronguard-sidecar
            image: cronguard/sidecar:latest
            env:
            - name: CRONGUARD_CHECKPOINT_URI
              value: "etcd://etcd-cluster:2379/cronguard/checkpoints"
            - name: CRONGUARD_ALERT_WEBHOOK
              value: "https://hooks.slack.com/services/T00/B00/xxx"
```

Nach dem Deployment startet cronguard automatisch die Überwachung aller CronJobs im selben Namespace.

### 2. **Alarmregeln für kritische Jobs definieren**
Erstellen Sie eine `cronguard-policy.yaml`, die spezifische Schwellwerte für einzelne Jobs setzt. Verwenden Sie die `cronctl policy apply`-CLI, um die Regeln zu laden:

```yaml
apiVersion: cronguard.io/v1
kind: AlertPolicy
metadata:
  name: production-batch-policy
spec:
  jobs:
    - name: "data-export-daily"
      maxLatencySeconds: 300
      retryOnFailure: true
      alertChannels:
        - pagerduty
        - email
    - name: "cleanup-temp-files"
      maxLatencySeconds: 60
      retryOnFailure: false
      alertChannels:
        - slack
```

Führen Sie dann aus:  
`cronctl policy apply -f cronguard-policy.yaml --namespace=production`

### 3. **Audit-Log exportieren und in SIEM-Integration einbinden**
Exportieren Sie die letzten 7 Tage an Audit-Logs als JSON-Lines in Ihr zentrales Log-Management:

```bash
cronctl audit export --since "7d" --format jsonlines | \
  kafka-console-producer --broker-list kafka-cluster:9092 --topic cronguard-audit
```

Konfigurieren Sie anschließend in Ihrem SIEM (z. B. Splunk oder Elastic) einen Input-Connector für dieses Kafka-Topic. So erhalten Sie eine manipulationssichere Historie aller Cron-Ausführungen.

## Best Practices aus der Industrie

### 1. **Checkpoint-Strategie für Stateful Workloads**
Vermeiden Sie, dass Checkpoints auf demselben Knoten wie der Cron-Job liegen. Nutzen Sie stattdessen einen dedizierten etcd-Cluster mit Multi-Zone-Replikation. So stellen Sie sicher, dass selbst bei einem vollständigen Node-Ausfall die Checkpoint-Daten erhalten bleiben und der Alarmmechanismus nicht durch Split-Brain-Szenarien gestört wird.

### 2. **Staggered Rolling Updates für Cron-Job-Konfigurationen**
Wenn Sie Änderungen an der `cronguard-policy.yaml` vornehmen, wenden Sie diese nicht global an, sondern rollen Sie sie schrittweise über Namespaces aus. Nutzen Sie die `--dry-run`-Option von `cronctl policy apply`, um die Auswirkungen auf die Alarmierung zu simulieren – insbesondere bei kritischen Jobs mit PagerDuty-Integration. Dies verhindert eine Alarmflut bei fehlerhaften Konfigurationen.
