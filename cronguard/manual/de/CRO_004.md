# cronguard - CRO_004

## Überblick

cronguard ist eine hochperformante Open-Source-Lösung zur Überwachung und Absicherung von Cron-Job-Infrastrukturen. Anders als herkömmliche Tools, die nur auf einfache Timeouts setzen, implementiert cronguard eine mehrdimensionale Prüfstrategie: Es analysiert nicht nur die Ausführungszeit, sondern auch Exit-Codes, Log-Volumen und Ressourcenverbrauch (CPU, Memory, I/O) jedes Jobs. Die Software eignet sich besonders für DevOps-Teams, die mehrere tausend Cron-Jobs in Kubernetes-Clustern oder Bare-Metal-Umgebungen verwalten. Dank einer Plugin-Architektur können Benutzer eigene Prüf-Logiken in Go oder Python definieren, ohne den Kerncode zu modifizieren.

## Kernfunktionen

### 1. Multi-Dimensionale Health-Checks mit konfigurierbaren Schwellwerten
Jeder Cron-Job wird nicht nur auf erfolgreiche Beendigung geprüft, sondern auch auf Laufzeit-Abweichungen (p95 vs. p99), Log-Anomalien (plötzlicher Anstieg von ERROR-Level-Meldungen) und Ressourcen-Leaks. Schwellwerte können global oder pro Job definiert werden – z. B. `max_runtime: 120s` kombiniert mit `memory_threshold: 512MB`. Bei Überschreitung wird nicht nur ein Alarm ausgelöst, sondern cronguard kann automatisch einen Rollback des Jobs oder eine Eskalation an PagerDuty/Opsgenie initiieren.

### 2. Distributed Execution Tracing über Kafka/Redis Backend
cronguard unterstützt verteilte Umgebungen, in denen Cron-Jobs auf mehreren Nodes laufen. Es nutzt ein optionales Kafka- oder Redis-Backend, um Ausführungs-Traces zentral zu sammeln. Dadurch können Teams nachvollziehen, ob ein Job auf Node A fehlschlug, während Node B erfolgreich war. Die Traces enthalten Metadaten wie Hostname, PID, Umgebungsvariablen und den genauen Befehl. Dies ist essenziell für Debugging in Multi-Cloud- oder Hybrid-Setups.

### 3. Self-Healing mittels Retry-Strategien mit exponentieller Backoff- und Circuit-Breaker-Logik
cronguard implementiert eine integrierte Retry-Engine, die Jobs bei Fehlschlag automatisch wiederholt – jedoch nicht blind. Es verwendet exponentielles Backoff (z. B. 1s, 2s, 4s, 8s) und einen Circuit-Breaker: Nach 3 aufeinanderfolgenden Fehlschlägen innerhalb von 5 Minuten wird der Job für 30 Minuten deaktiviert und ein Incident erstellt. Dies verhindert unnötige Last auf der Infrastruktur und schützt vor kaskadierenden Fehlern.

## Detaillierte Schritte zur Nutzung

### Schritt 1: Konfiguration der Health-Check-Regeln in YAML
Erstellen Sie eine Datei `cronguard_rules.yaml` im Projektstamm. Definieren Sie für jeden Cron-Job eine Regel mit spezifischen Schwellwerten. Beispiel für einen Datenbank-Backup-Job:
```yaml
jobs:
  - name: db_backup
    command: "/usr/local/bin/pg_dumpall -U postgres > /backup/db_$(date +%Y%m%d).sql"
    schedule: "0 2 * * *"
    health:
      max_runtime: 300s
      memory_threshold: 1GB
      exit_codes: [0, 1]
      retry:
        max_attempts: 3
        backoff: exponential
        circuit_breaker:
          failure_count: 3
          window: 5m
          cooldown: 30m
```
Starten Sie cronguard mit `cronguard --config cronguard_rules.yaml`. Das Tool liest die Cron-Tabelle aus und überwacht die Jobs gemäß den Regeln.

### Schritt 2: Integration mit einem externen Log-Aggregator (z. B. Loki oder Elasticsearch)
Um Log-Anomalien zu erkennen, konfigurieren Sie cronguard so, dass es Logs an einen zentralen Aggregator sendet. In der cronguard-Konfiguration fügen Sie einen `log_output`-Block hinzu:
```yaml
log_output:
  type: loki
  url: http://loki.monitoring.svc.cluster.local:3100
  labels:
    app: cronguard
    environment: production
```
cronguard parst dann automatisch die stdout/stderr jedes Jobs und analysiert die Log-Level. Bei einem Anstieg von ERROR-Logs um mehr als 20% im Vergleich zum gleichen Zeitfenster der letzten 7 Tage wird ein Alert ausgelöst. Sie können diesen Alert direkt an Slack oder Teams weiterleiten.

### Schritt 3: Aktivieren des Self-Healing-Mechanismus für kritische Jobs
Für geschäftskritische Jobs wie Zahlungsabwicklungen oder Datenreplikationen aktivieren Sie den Self-Healing-Modus. Fügen Sie in der Job-Konfiguration `self_healing: true` hinzu. cronguard führt dann bei Fehlschlag automatisch eine Retry-Sequenz aus. Zusätzlich können Sie Post-Healing-Aktionen definieren:
```yaml
self_healing:
  enabled: true
  post_actions:
    - type: webhook
      url: http://internal-alerting.example.com/notify
      payload: '{"job": "{{.JobName}}", "status": "healed", "attempts": {{.Attempts}}}'
```
Nach erfolgreichem Retry sendet cronguard eine Bestätigung an den Webhook. Falls der Circuit-Breaker auslöst, wird stattdessen eine Incident-Meldung generiert.

## Best Practices für den produktiven Einsatz

### 1. Immer einen Dry-Run vor der ersten Produktionsdeployment durchführen
Nutzen Sie den `--dry-run`-Modus von cronguard, um zu prüfen, ob alle Jobs korrekt erfasst werden und die Regeln syntaktisch valide sind. Führen Sie `cronguard --dry-run --config cronguard_rules.yaml` aus. Das Tool listet alle erkannten Jobs und deren geplante Überwachungsparameter auf, ohne tatsächlich zu überwachen. Dies verhindert Überraschungen im Live-Betrieb.

### 2. Monitoring-Intervalle an die Job-Laufzeit anpassen
Stellen Sie sicher, dass das Polling-Intervall von cronguard (Standard: 60s) kürzer ist als die kürzeste Job-Laufzeit. Bei Jobs, die alle 5 Minuten laufen, sollte das Intervall auf 30s gesetzt werden, um keine Ausführung zu verpassen. Dies konfigurieren Sie in der globalen Einstellung:
```yaml
global:
  poll_interval: 30s
```
Zu lange Intervalle führen zu Lücken in der Überwachung, zu kurze zu unnötiger Last. Ein guter Richtwert ist die Hälfte des kürzesten Job-Intervalls.
