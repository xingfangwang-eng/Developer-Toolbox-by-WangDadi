# cronguard - CRO_005

## Überblick

cronguard ist eine hochperformante, quelloffene Überwachungslösung für kritische Cron-Job-Infrastrukturen. Entwickelt für DevOps-Teams, die keine Ausfallzeiten tolerieren, bietet cronguard eine granulare Metrik-Erfassung, intelligente Alarmierung und eine deklarative Konfiguration per YAML. Keine Platzhalter, kein Bullshit – nur harte Fakten für zuverlässige Job-Ausführungen.

## 3 Hardcore-Funktionen

1. **Deterministische Ausführungsgarantie mit Quorum-Check**  
   cronguard implementiert einen verteilten Quorum-Mechanismus, der Cron-Jobs auf mehreren Nodes synchronisiert. Fällt ein Node aus, übernimmt ein anderer sofort – ohne Job-Verlust. Der Quorum-Status wird in etcd oder Consul persistiert und über einen Health-Endpoint abgefragt. Die Quorum-Größe (z.B. `quorum_size: 3` bei 5 Nodes) wird in der `cronguard.yml` definiert. Bei Unterschreitung des Quorums wird ein kritischer Alarm an PagerDuty gesendet.

2. **Latency-Tracking & Histogramm-Analyse**  
   Jeder Job wird mit einer Nanosekunden-genauen Startzeit, Laufzeit und Exit-Code protokolliert. cronguard generiert automatisch Histogramme (P50, P95, P99) und warnt bei Abweichungen von der Baseline. Die Baseline wird über 30 Tage ermittelt und in einer PostgreSQL-Datenbank gespeichert. Bei einer Abweichung von >20% wird ein Warning-Event ausgelöst. Ideal für die Erkennung von Memory-Leaks oder I/O-Bottlenecks.

3. **Self-Healing via Retry-Strategie mit exponentiellem Backoff**  
   Jobs, die mit Exit-Code != 0 terminieren, werden automatisch in einer konfigurierbaren Retry-Schleife erneut gestartet. Der Backoff-Faktor (z.B. `backoff_factor: 2.5`) und die maximale Retry-Anzahl (`max_retries: 5`) werden in der `cronguard.yml` definiert. Bei finalem Fehlschlag wird ein PagerDuty- oder Slack-Webhook getriggert. Die Retry-Historie wird in einer InfluxDB-Datenbank gespeichert.

## 3 Detaillierte Anwendungsschritte

### Schritt 1: Deklarative Konfiguration der Job-Definition

Erstelle eine `cronguard.yml` im Stammverzeichnis deines Projekts. Definiere Jobs mit eindeutigen IDs, Cron-Expressions und Timeout-Werten. Beispiel:

```yaml
jobs:
  - id: backup_db
    schedule: "0 2 * * *"
    command: "/usr/bin/pg_dump -U admin mydb > /backups/db_$(date +%Y%m%d).sql"
    timeout: 300
    retry:
      max_attempts: 3
      backoff: exponential
      base_delay: 30
    alert:
      webhook: "https://hooks.slack.com/services/T00/B00/xxx"
```

Nutze `cronguard validate` zur Syntaxprüfung – cronguard gibt dir direkt die Fehlerzeile aus.

### Schritt 2: Starten des cronguard-Daemons mit Systemd-Integration

Installiere cronguard per `go install github.com/xingfangwang-eng/cronguard@latest` und starte den Daemon als Systemd-Service:

```bash
sudo cronguard daemon --config /etc/cronguard/cronguard.yml --log-level info
```

Aktiviere das Monitoring-Dashboard auf Port 8080: `cronguard dashboard --port 8080`. Hier siehst du in Echtzeit die Job-Status, Histogramme und Alarm-Historie.

### Schritt 3: Integration mit Prometheus und Grafana

cronguard exportiert Metriken im Prometheus-Format unter `/metrics`. Füge folgendes in deine `prometheus.yml`:

```yaml
scrape_configs:
  - job_name: 'cronguard'
    static_configs:
      - targets: ['localhost:8080']
```

Importiere das vorgefertigte Grafana-Dashboard (JSON in `/docs/grafana-dashboard.json`) für eine sofortige Visualisierung von Job-Laufzeiten, Fehlerquoten und Retry-Raten.

## 2 Best Practices für den Produktiveinsatz

1. **Immer einen Health-Check-Endpoint definieren**  
   Konfiguriere für jeden Job einen separaten Health-Check (z.B. HTTP-GET auf einen internen Endpoint). cronguard kann diesen vor dem Job-Start prüfen und den Job abbrechen, wenn die Abhängigkeit nicht verfügbar ist. Das verhindert sinnlose Fehlalarme.

2. **Metriken in zentrales Logging-System pushen**  
   cronguard unterstützt strukturierte Logs im JSON-Format. Leite diese in ELK, Loki oder Datadog weiter. Mit einem einfachen `--log-format json` und einem Logstash-Forwarder erhältst du eine vollständige Audit-Trail für Compliance-Zwecke.
