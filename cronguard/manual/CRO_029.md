# cronguard - CRO_029

**cronguard** is a distributed cron job supervisor engineered for mission-critical production environments. It combines runtime guardrails, anomaly detection, and immutable audit trails into a single binary—no external dependencies required.

---

## Why cronguard?

Traditional cron leaves you blind: silent failures, overlapping executions, and zero accountability. cronguard wraps every scheduled task with a safety harness. It intercepts `SIGKILL`, enforces concurrency locks, and streams structured logs to any sink (S3, Elasticsearch, or local disk). Built for teams that treat cron as infrastructure—not an afterthought.

---

## Core Features

### 1. Pre-Execution Gate with Distributed Quarantine (etcd + Raft Consensus)
Before a job starts, cronguard acquires a distributed lock via etcd using **Raft-based consensus** with a configurable TTL (default: `lock_ttl: 15s`). If the previous instance is still running (stuck, zombie, or retrying), the new execution enters a **quarantine queue**—not killed, but buffered with a `skip_reason` tag and `quarantine_ts` microsecond timestamp. You can configure `max_overlap: 1` (strict serial) or `max_overlap: 0` (no overlap allowed). The lock key format is `cronguard/locks/{job_name}/{node_id}`. If a node crashes, the lock auto-expires after `lock_ttl` + `heartbeat_interval: 5s`. No more double-billing or duplicate API calls.

### 2. Immutable Audit Trail with Merkle DAG (Directed Acyclic Graph)
Every execution produces a signed, hash-linked log entry. The chain includes: `job_id → node_id → start_ts (nanosecond precision) → exit_code → stdout_digest (sha256) → stderr_digest (sha256) → prev_hash (sha256) → signature (Ed25519)`. This creates a tamper-evident **Merkle DAG** ideal for SOC 2 Type II or FedRAMP compliance. Export the chain to an S3 bucket with Object Lock enabled (retention: `retention_days: 90`). Use `cronguard verify --job nightly-db-backup` to replay the chain and detect any hash mismatch. A mismatch means the log has been tampered with.

### 3. Adaptive Retry with Fibonacci Backoff + Jitter (Sub-Millisecond Precision)
Instead of fixed retries, cronguard uses a **Fibonacci backoff** (1s, 1s, 2s, 3s, 5s, 8s, 13s, 21s…) with ±10% jitter. It respects the job’s `SIGTERM` timeout (default: `sigterm_timeout: 30s`) and only retries if the exit code matches a whitelist (e.g., `retry_on_exit_codes: [1, 2, 130]` for transient errors, `0` for success—no retry). Configurable `max_retries: 5` and `backoff_cap: 60s` prevent runaway loops. The backoff is calculated in **microseconds** (`backoff_us`) for sub-second precision. If a job exceeds `max_duration: 3600s`, cronguard sends `SIGTERM`, waits `sigterm_timeout`, then issues `SIGKILL` and logs `kill_reason: duration_exceeded`.

## How to Use

### Step 1: Define a Guarded Job with Resource Limits
Create a YAML manifest with resource constraints:

```yaml
job:
  name: nightly-db-backup
  command: pg_dump -U admin mydb | gzip > /backups/db-$(date +%Y%m%d).gz
  schedule: "0 2 * * *"
  concurrency: forbid
  lock_ttl: 15s
  heartbeat_interval: 5s
  retry:
    max_attempts: 3
    backoff: fibonacci
    jitter: 0.1
    backoff_cap: 60s
    retry_on_exit_codes: [1, 2, 130]
  audit:
    sink: s3://my-bucket/audit-logs/
    sign: true
    retention_days: 90
  resource_limits:
    max_memory_mb: 512
    max_cpu_percent: 80
    max_duration: 3600s
    sigterm_timeout: 30s
  env:
    PGPASSWORD: ${PGPASSWORD}
    AWS_REGION: us-east-1
```

Drop this into `/etc/cronguard/jobs/`. The daemon auto-detects changes via inotify with a `cooldown: 2s` to avoid rapid reloads.

### Step 2: Start the Daemon with Remote Logging and Metrics
```bash
cronguard run \
  --config-dir /etc/cronguard/jobs \
  --log-sink tcp://logs.internal:5000 \
  --lock-backend etcd://etcd-cluster:2379 \
  --node-id prod-web-01 \
  --metrics-addr :9090 \
  --heartbeat-interval 5s \
  --audit-sink s3://cronguard-audit-prod/ \
  --sign-key /etc/cronguard/keys/ed25519-private.pem
```

This launches the supervisor in the foreground. It registers itself in etcd as a worker with key `cronguard/workers/prod-web-01` and TTL `15s`. Every execution is streamed to the remote log sink in JSON Lines format with nanosecond timestamps. Metrics are exposed at `:9090` for Prometheus scraping (e.g., `cronguard_job_duration_seconds`, `cronguard_retry_count_total`).

### Step 3: Inspect Execution History with Merkle Verification
```bash
cronguard history nightly-db-backup --limit 10 --format json --verify
```

Output includes:
```json
{
  "executions": [
    {
      "id": "a1b2c3d4",
      "start": "2025-03-21T02:00:00.123456789Z",
      "exit_code": 0,
      "duration_ms": 12345,
      "stdout_digest": "sha256:abc123...",
      "stderr_digest": "sha256:def456...",
      "hash": "sha256:ghi789...",
      "prev_hash": "sha256:jkl012...",
      "signature": "ed25519:valid_signature_here",
      "verified": true
    }
  ],
  "chain_integrity": "valid"
}
```

Use `--verify` to check the Merkle DAG integrity. A mismatch means the log has been tampered with. The `verified` field is `true` only if the hash chain and Ed25519 signature are both valid.

## Industry Best Practices

### 1. Use a Dedicated etcd Cluster for Locking with Low TTL
Don’t share your Kubernetes etcd or application database. Spin up a small 3-node etcd cluster (t3.medium or equivalent, 2 vCPU, 4GB RAM, 20GB gp3 SSD) dedicated to cronguard locks. Set `lock_ttl: 10s` and `heartbeat_interval: 3s` so a crashed node releases the lock quickly. Enable etcd authentication with `--auth-token` and restrict access to cronguard nodes only via firewall rules (port 2379/tcp). This isolates cron coordination from other system load and prevents lock contention.

### 2. Route All Audit Logs to an Immutable S3 Bucket with Object Lock
Enable Object Lock on your audit bucket with a retention period of 90 days (governance mode). Configure cronguard to write logs with `--audit-sink s3://cronguard-audit-prod/` and set `sign: true` with an Ed25519 key. This creates a forensically sound record: even root cannot delete or modify logs without leaving a trace. Pair with AWS CloudTrail (enable data events for S3) and AWS Config (track bucket policy changes) for full accountability. Use `cronguard verify --all --since 2025-01-01` weekly to ensure chain integrity.
