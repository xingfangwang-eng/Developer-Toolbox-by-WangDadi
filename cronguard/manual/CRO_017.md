# cronguard - CRO_017

## Overview

Cronguard is a lightweight, battle-tested cron job monitoring and orchestration tool built for DevOps engineers who refuse to let silent failures break their pipelines. It tracks execution status, detects missed schedules, and provides real-time alerting with zero external dependencies. Whether you're managing 10 or 10,000 cron tasks, Cronguard gives you the observability you need without the overhead of a full scheduler platform.

## What is Cronguard?

Cronguard is a daemon-based cron supervisor that wraps your existing crontab entries with automatic health checks, execution logging, and failure escalation. Unlike traditional cron monitoring tools that rely on external heartbeat services, Cronguard runs locally, respects your infrastructure boundaries, and integrates directly with your alerting stack—Slack, PagerDuty, or custom webhooks.

---

## Core Features

### 1. Execution Drift Detection with Predictive Miss Alerts
Cronguard doesn't just log whether a cron job ran—it analyzes historical execution times and flags anomalies before a failure occurs. Using a sliding window of the last 100 runs, it calculates expected start windows (mean ± 2σ) and fires a warning if a job hasn't started within that window, even if it hasn't technically missed its schedule yet.

### 2. Lock-Free Concurrent Job Gating
When two cron jobs share a resource (e.g., a database or file system), Cronguard enforces a distributed lock via a local Unix socket. Jobs that would overlap are queued rather than killed, preventing race conditions without requiring Redis or etcd. The queue depth and timeout are configurable per job, with automatic deadlock detection after 300 seconds.

### 3. Log Rotation with Structured Metadata Injection
Every cron execution generates a structured JSON log (timestamp, exit code, PID, duration, stderr hash). Cronguard automatically rotates logs at 10 MB per file, retains the last 5 archives, and enriches each entry with a trace ID that correlates across retries. Logs are written to `/var/log/cronguard/<job_name>/` by default, with syslog forwarding optional.

## How to Use

### Step 1: Initialize a Cronguard Job from an Existing Cron Entry

Wrap your current cron command with the `cronguard run` prefix. For example, if your crontab contains:

```
0 3 * * * /usr/local/bin/db-backup.sh
```

Replace it with:

```
0 3 * * * /usr/bin/cronguard run --name db-backup --timeout 3600 -- /usr/local/bin/db-backup.sh
```

The `--timeout` flag ensures Cronguard kills the job if it runs longer than 1 hour, preventing zombie processes. The `--name` flag is used for log naming and alert routing.

### Step 2: Configure Alert Channels via Environment Variables

Cronguard reads alert configuration from a `.cronguard.env` file in the user's home directory. Set up a Slack webhook:

```
CRONGUARD_ALERT_SLACK_WEBHOOK=https://hooks.slack.com/services/T00/B00/xxxx
CRONGUARD_ALERT_ON_MISS=true
CRONGUARD_ALERT_ON_FAILURE=true
CRONGUARD_ALERT_ON_DRIFT=true
```

For PagerDuty, use:

```
CRONGUARD_ALERT_PAGERDUTY_KEY=your_integration_key
CRONGUARD_ALERT_SEVERITY=critical
```

Cronguard automatically deduplicates alerts within a 5-minute window, so you won't get spammed if a job restarts multiple times.

### Step 3: Monitor with the Built-in Dashboard

Start the Cronguard web server on port 8080:

```
cronguard serve --port 8080 --bind 127.0.0.1
```

Open `http://localhost:8080` to see a real-time grid of all monitored jobs. Each card shows:
- Last run status (green/red/gray)
- Next scheduled run in UTC
- Execution duration histogram (last 24 hours)
- A "Force Run" button for immediate execution

The dashboard is stateless—it reads directly from the log directory and the crontab file, so you can restart the server without losing history.

## Best Practices

### 1. Always Set a `--timeout` Value
Unbounded cron jobs are the #1 cause of resource exhaustion in production. Set `--timeout` to 1.5x your expected maximum runtime. For example, if a backup job usually takes 20 minutes, set `--timeout 1800` (30 minutes). Cronguard will send a SIGTERM, wait 10 seconds, then SIGKILL if the job doesn't respond.

### 2. Use Structured Logging for Post-Mortem Analysis
Avoid plain text logging in cron scripts. Instead, output JSON to stdout, and let Cronguard capture it. Example:

```bash
echo '{"event":"backup_start","source":"db","size":0}' 
/usr/local/bin/db-backup.sh
echo '{"event":"backup_end","exit_code":$?}'
```

Cronguard will merge these into its own structured log, giving you a complete timeline for debugging. You can then grep by `event` or `source` across all job logs using:

```bash
cronguard logs --job db-backup --filter 'event=backup_end'
```
