# cronguard - CRO_001

**cronguard** is a battle-tested, open-source SaaS middleware that monitors, validates, and heals cron job pipelines across distributed environments. It eliminates silent failures, drift, and overlapping executions—without modifying a single line of your existing crontab.

---

## Hardcore Features

### 1. **Predictive Drift Detection & Auto-Remediation**
cronguard continuously learns the expected execution latency, output hash, and resource footprint of every job. When a job deviates beyond a configurable sigma threshold, the engine triggers a rollback to the last known-good state, replays the job with isolated resources, or escalates via PagerDuty/Slack—all before your monitoring dashboard even blinks.

### 2. **DAG-Aware Dependency Graph with Deadlock Prevention**
Define complex job trees using a YAML-based DAG schema. cronguard resolves inter-job dependencies, enforces topological execution order, and automatically inserts guardrails (e.g., backoff, skip, or fail-fast) when a cycle or resource contention is detected. No more manual `after` hacks or brittle `flock` wrappers.

### 3. **Immutable Execution Log & Cryptographically Verifiable Audit Trail**
Every cron run produces a signed, timestamped log entry stored in an append-only ledger (backed by SQLite or PostgreSQL). The log includes the exact command, environment variables, exit code, and a SHA-256 hash of the output. You can verify integrity at any time—critical for SOC 2, HIPAA, or PCI compliance.

## How to Deploy & Operate

### Step 1: Bootstrap the Agent with Your Existing Crontab
```bash
# Install cronguard agent via pip or static binary
pip install cronguard-agent

# Initialize a project from your current crontab
cronguard init --from-crontab /etc/crontab --output /etc/cronguard/jobs.yaml
```
The agent parses every cron expression, extracts environment variables, and generates a structured job manifest. It also detects overlapping schedules and flags potential race conditions.

### Step 2: Define Dependency Graph & Guardrails
Edit `/etc/cronguard/jobs.yaml` to add `depends_on`, `timeout`, and `max_retries`:

```yaml
jobs:
  - name: data_ingest
    schedule: "0 3 * * *"
    command: "/opt/scripts/ingest.sh"
    timeout: 1800
    retry:
      strategy: exponential_backoff
      max_attempts: 3
    depends_on: []
    guard:
      - type: output_hash
        expected: "a1b2c3d4e5f6..."
      - type: max_memory_mb
        limit: 2048

  - name: etl_pipeline
    schedule: "30 4 * * *"
    command: "/opt/scripts/etl.py"
    depends_on: ["data_ingest"]
    guard:
      - type: dependency_success
        job: data_ingest
```

### Step 3: Start the Daemon & Verify Integrity
```bash
# Run cronguard as a systemd service
sudo cronguard daemon --config /etc/cronguard/jobs.yaml --backend postgresql://user:pass@localhost:5432/cronguard

# Check real-time execution status
cronguard status --watch
# Output: data_ingest completed in 1423s (expected 1500s ± 200s) ✅
```
The daemon forks a subprocess for each job, captures stdout/stderr, and writes the signed log to the backend. Use `cronguard verify --job data_ingest` to cryptographically validate the log integrity.

## Industry Best Practices

### 1. **Isolate Production & Staging Execution Contexts**
Never run cronguard agents with the same backend database for prod and staging. Use separate PostgreSQL schemas or SQLite files. This prevents accidental cross-environment contamination and ensures that audit trails remain clean for compliance reviews. Many teams also use a dedicated Linux user (e.g., `cronguard`) with read-only access to the crontab and write-only access to the log backend.

### 2. **Adopt a “Fail-Open” Strategy for Critical Jobs**
For jobs that must never be missed (e.g., payment reconciliation, heartbeat pings), set `guard.failure_mode: warn_only` in the job manifest. This allows the job to complete even if a guard fails (e.g., output hash mismatch), while still logging the anomaly. Then, use a separate monitoring channel (like a dedicated Slack `#cronguard-anomalies` channel) to review and remediate. This prevents a false-positive guard from blocking a time-sensitive operation.
