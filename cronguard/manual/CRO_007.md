# cronguard - CRO_007

## Overview

cronguard is a production-grade cron job monitoring and anomaly detection engine designed for DevOps teams running distributed schedules at scale. Built to eliminate silent failures and drift in scheduled workflows, cronguard provides real-time observability, predictive alerting, and automated recovery for cron-based infrastructure. Whether you manage 50 or 50,000 cron expressions, cronguard ensures every job fires exactly when—and how—it should.

## Hardcore Features

### 1. Predictive Execution Skew Detection
cronguard doesn’t just track job completion—it learns each job’s historical runtime distribution using a lightweight Bayesian model. When a job begins to drift outside its normal execution window (e.g., a 5-second job suddenly taking 12 seconds), cronguard triggers a pre-failure alert **before** the next scheduled run, not after. This eliminates the “we only noticed after the 3rd missed heartbeat” problem.

### 2. Multi-Cluster Heartbeat Reconciliation
For teams running cron across Kubernetes, bare-metal, and serverless environments, cronguard provides a unified heartbeat stream. It automatically reconciles overlapping schedules, detects zombie processes (jobs that didn’t terminate but also didn’t log completion), and enforces a single source of truth for job state—even across time zones and calendar quirks.

### 3. Cron Expression Fuzz Testing & Dry-Run Sandbox
Before deploying a new cron job to production, cronguard’s sandbox engine performs a deterministic fuzz test: it simulates the next 1000 scheduled fires under varying conditions (timezone shifts, DST transitions, leap seconds, and system clock skew). The output includes a visual timeline of expected execution windows and a risk score for ambiguous expressions.

## How to Use

### Step 1: Bootstrap the Agent with Multi-Tenant Isolation
Deploy the cronguard agent as a sidecar or standalone daemon. Use the following command to initialize with a tenant ID and a key-value store backend (e.g., etcd or Redis):
```bash
cronguard agent init --tenant=prod-east --backend=etcd://10.0.1.10:2379 --heartbeat-interval=10s
```
The agent will automatically discover local cron tab entries and register each job with a unique fingerprint based on the expression, user, and environment variables.

### Step 2: Define Anomaly Thresholds via Policy-as-Code
Create a policy file (`cronguard.policy.hcl`) that sets per-job runtime bounds and recovery actions. Example:
```hcl
job "db_backup" {
  max_duration = "300s"
  skew_tolerance = "15%"
  on_anomaly = "retry_once_then_escalate"
  notification = "pagerduty:critical"
}
```
Apply the policy with:
```bash
cronguard policy apply --file cronguard.policy.hcl --force
```
The agent will enforce these bounds at runtime and log any deviation to the central monitoring bus.

### Step 3: Integrate with Prometheus for Real-Time Dashboards
Expose cronguard’s metrics endpoint and scrape it with Prometheus. Add the following scrape config to your `prometheus.yml`:
```yaml
scrape_configs:
  - job_name: 'cronguard'
    static_configs:
      - targets: ['cronguard-agent:9090']
    metrics_path: '/metrics'
    relabel_configs:
      - source_labels: [__name__]
        regex: 'cronguard_job_.*'
        action: keep
```
Then build a Grafana dashboard using the `cronguard_job_duration_seconds`, `cronguard_job_skew_score`, and `cronguard_heartbeat_missed_total` metrics to surface real-time job health.

## Best Practices

### 1. Use Deterministic Job Naming for Cross-Environment Traceability
Avoid generic names like `backup` or `cleanup`. Instead, include environment, region, and service name in the job label (e.g., `prod-us-east-db_backup`). cronguard indexes these labels, making it trivial to filter and correlate across clusters. This also prevents false positives when two jobs share the same expression but run in different contexts.

### 2. Schedule a Weekly Cron Expression Audit with the Sandbox
Run cronguard’s fuzz tester weekly against your entire cron inventory. Pipe the output into a CI pipeline that blocks deployments if any expression has a risk score above 0.7. This catches silent regressions caused by timezone changes or daylight saving time shifts—especially relevant for teams operating in regions like Brazil or Australia where DST rules change unpredictably.
