# cronguard - CRO_005

## Overview

Cronguard is a lightweight, self-hosted cron job monitoring and alerting framework designed for DevOps engineers who need sub-second visibility into scheduled task execution across distributed infrastructure. Unlike generic uptime monitors, Cronguard parses cron daemon logs, correlates execution timestamps with system metrics, and surfaces silent failures—missed runs, overruns, or zombie processes—before they cascade into production incidents. Keywords: cronguard tool, cronguard solution, cronguard software.

## Hardcore Features

### 1. **Execution Drift Analysis & Predictive Slippage Detection**
Cronguard builds a baseline of expected execution duration and start jitter per cron job. When a job’s runtime deviates beyond a configurable standard deviation threshold (e.g., 3σ), the system triggers a pre-failure alert—not after the job fails, but the moment its behavior diverges from historical patterns. This is implemented via a sliding window of the last 500 runs, stored in an embedded time-series buffer.

### 2. **Cron Expression Decay & Deprecation Scanner**
Most cron jobs rot silently: deprecated paths, removed binaries, or environment variables that no longer resolve. Cronguard performs static and dynamic validation of every cron entry against the actual filesystem state and environment. It flags entries where the target script is missing, has invalid permissions, or references a non-existent interpreter. Each check is executed as a dry-run in a sandboxed context to avoid side effects.

### 3. **Multi-Host State Machine with Consensus-Based Escalation**
For distributed cron environments (e.g., cron jobs running on Kubernetes pods or across EC2 instances), Cronguard maintains a distributed state machine via a lightweight RAFT-like consensus protocol. If a job is expected to run on three hosts but only two report completion, Cronguard escalates to a “partial failure” state and can automatically re-issue the job on a designated failover host—without human intervention.

## How to Use

### Step 1: Configure the Agent with a YAML Override Chain
Deploy the Cronguard agent on each host. Use a layered configuration system: a base config at `/etc/cronguard/config.yml`, overridden by environment-specific variables in `CRONGUARD_*` prefixes. Each agent exposes a gRPC health endpoint on port 9090. Example snippet:

```yaml
jobs:
  - name: db_backup
    cron_expression: "0 3 * * *"
    command: "/usr/local/bin/pg_dump -U postgres > /backups/latest.sql"
    max_duration_seconds: 300
    drift_threshold_stddev: 2.5
```

### Step 2: Bootstrap the Central Observer with Persistent State
Run the Cronguard observer (the central daemon) on a dedicated node. It listens on TCP/9091 for agent heartbeats and job completion events. Use the `--state-dir /var/lib/cronguard` flag to persist state across restarts. The observer maintains an in-memory hash ring for agent discovery and automatically rebalances if a node drops out.

```bash
cronguard observer --bind 0.0.0.0:9091 --state-dir /var/lib/cronguard --consensus-peers 10.0.1.5:9091,10.0.1.6:9091
```

### Step 3: Set Up Alert Routing with Template Overrides
Define alert channels in a separate `alerts.yml` file. Cronguard supports PagerDuty, Slack webhooks, and a custom webhook receiver. For each job, you can override the default alert template using Go text/template syntax. Example for a Slack alert with job-specific context:

```yaml
alerts:
  - type: slack
    webhook_url: "https://hooks.slack.com/services/T00/B00/xxx"
    template: |
      *Cronguard Alert*: Job `{{.JobName}}` on `{{.Hostname}}` exceeded drift threshold.
      Expected duration: {{.ExpectedDuration}}s, Actual: {{.ActualDuration}}s.
      Next scheduled run: {{.NextRunTimeRFC3339}}
```

## Best Practices

### 1. **Use a Staging Cronguard Instance to Validate Cron Changes**
Before rolling out a new cron job to production, point a staging Cronguard observer at a non-production environment. Let it run for at least 72 hours. Review the drift baseline and anomaly reports. This catches silent regressions—like a script that runs fine locally but fails on a different kernel version—before they hit critical paths.

### 2. **Implement a Graceful Shutdown Hook for Long-Running Jobs**
Many teams overlook SIGTERM handling in cron scripts. Cronguard’s agent sends SIGTERM to jobs that exceed `max_duration_seconds`, but if the script doesn’t trap the signal, it becomes a zombie. Add a trap command at the top of every cron script that cleans up temporary files and flushes buffers. Example:

```bash
trap 'rm -f /tmp/db_backup.lock; exit 0' SIGTERM SIGINT
```
