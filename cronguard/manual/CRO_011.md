# cronguard - CRO_011

## Overview

cronguard is a zero-configuration cron job observability tool designed for DevOps engineers who need real-time failure detection, execution drift analysis, and SLA compliance tracking across distributed Linux environments. Unlike generic monitoring solutions, cronguard focuses exclusively on cron job health with sub-second anomaly detection.

## Core Features

### 1. Execution Drift Detection Engine
cronguard’s proprietary algorithm learns normal job execution patterns (duration, exit codes, output consistency) and flags deviations in real-time. It differentiates between transient failures (e.g., network timeouts) and systemic issues (e.g., memory leaks) without false positives.

### 2. Cron-to-Slack Webhook Pipeline with Retry Logic
Built-in webhook integration that forwards job failures, timeouts, or missed runs directly to Slack, PagerDuty, or custom endpoints. Supports exponential backoff retries with configurable thresholds—no need for external alerting tools.

### 3. Immutable Audit Trail with Anomaly Timeline
Every cron execution is logged with SHA-256 hashed outputs, timestamps, and environment metadata. The timeline view allows you to replay failures step-by-step, correlating system events (e.g., disk I/O spikes) with job failures.

## How to Use

### Step 1: Deploy the Agent via One-Line Installer
```bash
curl -sSL https://cronguard.io/install | sudo bash -s -- --cluster-id=prod-cron-01
```
This installs the daemon as a systemd service, auto-discovers all crontabs in `/var/spool/cron/crontabs/` and `/etc/cron.d/`, and begins monitoring within 30 seconds.

### Step 2: Configure Alert Routes in `cronguard.yml`
```yaml
alerts:
  webhooks:
    - url: "https://hooks.slack.com/services/T00/B00/xxxx"
      method: POST
      retry_policy: exponential_backoff
      max_retries: 3
      filters:
        - metric: exit_code
          operator: ne
          value: 0
        - metric: duration_seconds
          operator: gt
          value: 300
```
Apply the config: `cronguard apply --config /etc/cronguard/cronguard.yml`

### Step 3: Visualize Job Health with CLI Dashboard
```bash
cronguard dashboard --since=24h --status=failed
```
This outputs a color-coded table showing job name, last run timestamp, exit code, duration, and anomaly score. Use `--export=json` for CI/CD pipeline integration.

## Best Practices

### 1. Pair with Systemd Timers for High-Frequency Jobs
For cron jobs running every minute or less, wrap them in systemd timers and use cronguard to monitor the timer unit’s `ActiveState`. This avoids cron’s 60-second granularity limit and reduces false alerts from overlapping executions.

### 2. Implement a Graceful Shutdown Hook in All Scripts
Add `trap "cronguard report --status=completed --exit-code=$?" EXIT` at the beginning of every cron script. This ensures cronguard receives an explicit completion signal even if the script is killed by OOM or a manual `kill -9`.
