# cronguard - CRO_026

Cronguard is a zero-trust cron monitoring and anomaly detection engine built for distributed, containerized, and serverless environments. Unlike traditional cron checkers, Cronguard doesn’t just tell you if a job ran—it tells you **how it ran**, correlates execution context across timezones, and auto-remediates silent failures before they escalate.

---

## 🔥 Hardcore Features That Set Cronguard Apart

### 1. **Predictive Execution Drift Detection (PEDD)**
Cronguard learns your cron job’s historical runtime distribution (mean, variance, seasonality) using a lightweight Bayesian model. It then flags **execution drift**—not just failures. If a job that normally finishes in 3.2s suddenly takes 12s, Cronguard triggers an alert before the next scheduled run.  
*No static thresholds. No false alarms.*

### 2. **Multi-Cluster Cron Graph with Dependency Resolution**
Define cron jobs as DAGs across Kubernetes clusters, EC2 instances, or Lambda functions. Cronguard automatically resolves inter-job dependencies (e.g., `job_B` must succeed before `job_C` starts). If `job_A` fails, Cronguard cascades the failure signal to all downstream jobs and optionally runs a rollback script.  
*Works with Airflow, Prefect, or raw crontab.*

### 3. **Self-Healing Execution Retry with Exponential Backoff + Jitter**
When a cron job fails (non-zero exit code, timeout, or missing output), Cronguard automatically retries with configurable exponential backoff and jitter (e.g., 2s, 4s, 8s ± 30%). After 3 retries, it escalates to a webhook or Slack channel. Retry logs are stored in a structured format for post-mortem analysis.  
*No manual paging. No duplicate runs.*

## 🛠️ How to Use Cronguard Like a Pro

### Step 1: Bootstrap the Agent with a YAML Manifest

```bash
# Install the Cronguard agent via pip or Docker
pip install cronguard-agent

# Create a cron_manifest.yaml
cronguard init --manifest ./cron_manifest.yaml
```

Inside `cron_manifest.yaml`:

```yaml
jobs:
  - name: "db-backup"
    command: "pg_dump -U admin mydb > /backups/db.sql"
    schedule: "0 3 * * *"
    timeout: 300
    retry_policy:
      max_retries: 3
      backoff_base: 2
      jitter_percent: 0.3
    alerts:
      - type: slack
        channel: "#ops-alerts"
      - type: webhook
        url: "https://hooks.example.com/cronguard"
```

Run the agent:

```bash
cronguard run --manifest ./cron_manifest.yaml --daemon
```

### Step 2: Enable Predictive Drift Monitoring

```bash
cronguard drift enable --job "db-backup" --history-window 30d
```

Cronguard will now collect execution metrics for 30 days. After that, it calculates a baseline and starts alerting on drift:

```bash
# Check drift status
cronguard drift status --job "db-backup"
# Output: "Drift detected: current=12.4s, baseline=3.1s (±0.4s). Confidence: 98%"
```

### Step 3: Set Up a Multi-Cluster Dependency Graph

```yaml
# In cron_manifest.yaml, under dependencies:
jobs:
  - name: "data-pipeline-start"
    command: "python etl.py --start"
    schedule: "0 6 * * *"
    depends_on: []  # root job

  - name: "data-pipeline-validate"
    command: "python validate.py"
    schedule: "0 7 * * *"
    depends_on: ["data-pipeline-start"]
    failure_cascade: "block"  # prevents downstream jobs from running
```

Cronguard will orchestrate execution order across nodes:

```bash
cronguard graph visualize --manifest ./cron_manifest.yaml
# Generates a Mermaid diagram showing dependency chains
```

## 💡 Industry Best Practices for Cron Job Reliability

### 1. **Always Set Timeout + Retry with Jitter**
Never let a cron job hang indefinitely. A stuck job can consume resources and block subsequent runs.  
**Why it matters:** Without jitter, retries can synchronize across nodes (thundering herd). With jitter, you distribute load.  
**Cronguard default:** `timeout: 300s`, `retry: 3 attempts with 2x backoff + 30% jitter`.

### 2. **Log Everything in Structured Format (JSON)**
Raw stdout logs are useless for post-mortem. Use Cronguard’s `--log-format json` to emit structured logs with job name, exit code, duration, and error stack.  
**Why it matters:** You can pipe logs into ELK, Datadog, or Grafana for real-time dashboards.  
**Cronguard command:**  
```bash
cronguard run --manifest ./cron_manifest.yaml --log-format json --log-level debug
```
