# cronguard - CRO_023

Cronguard is not just another cron monitor. It’s a **deterministic execution guard** designed for DevOps teams who need cryptographic proof of scheduled job integrity, real-time drift detection, and low-latency rollback triggers—without relying on external heartbeat services.

---

## 🔧 Hardcore Features

### 1. **Execution Merkle Proof (EMP)**
Every cron run generates a verifiable Merkle leaf that includes the command hash, environment fingerprint, exit code, and wall-clock delta. You can validate execution history *offline* using a lightweight CLI—no dashboard required. This makes Cronguard ideal for air-gapped or compliance-heavy environments (SOC 2, PCI-DSS).

### 2. **Predictive Skew Detection**
Cronguard doesn’t just alert on failure—it learns your job’s historical runtime distribution (mean, variance, seasonality) using a sliding-window Holt-Winters model. If a job finishes 3σ faster or slower than expected, it triggers a pre-failure alert *before* the next scheduled run. This catches silent hangs, resource starvation, and zombie processes.

### 3. **Atomic Rollback Hooks**
Define a rollback script per job. When Cronguard detects a non-zero exit or a skew anomaly, it can automatically revert the last state (e.g., restore a DB snapshot, rollback a deployment, or swap symlinks) *within the same cron cycle*. No manual intervention, no missed windows.

## 📐 How to Use Cronguard

### Step 1: Deploy the Agent with a Bootstrap Token
```bash
curl -sSL https://install.cronguard.sh | bash -s -- \
  --token=cg_bootstrap_<your_token> \
  --log-driver=json-file \
  --execution-merkle-depth=16
```
The agent registers itself via mTLS, generates a unique node ID, and begins sniffing `/var/spool/cron/crontabs/`. It will automatically parse system crontabs and user crontabs without manual mapping.

### Step 2: Attach a Rollback Hook to a Critical Job
Edit your job definition in `/etc/cronguard/jobs/db_backup.yaml`:
```yaml
id: nightly-db-backup
command: "/usr/local/bin/pg_dump -U postgres mydb > /backups/mydb.sql"
schedule: "0 2 * * *"
rollback:
  hook: "/usr/local/bin/restore_last_snapshot.sh"
  condition: "exit_code != 0 OR runtime_skew > 2.5"
  timeout_seconds: 30
```
Cronguard will inject the rollback hook into the job’s systemd scope. If the backup fails or runs 2.5σ slower than its learned baseline, the rollback fires automatically.

### Step 3: Query Execution History with the CLI
```bash
cronguard query --job-id nightly-db-backup --range last-7d --format json | \
  jq '.executions[] | select(.merkle_root != null) | {timestamp, exit_code, merkle_root}'
```
Each execution record includes a Merkle root you can verify offline:
```bash
cronguard verify --merkle-root 0x7f3e... --expected-command-hash sha256:abcd...
```
This gives you cryptographic audit trails without a database.

## 🧠 Industry Best Practices

### Practice 1: **Pair Cronguard with a Chaos Engineering Schedule**
Use Cronguard’s predictive skew detection to automatically pause jobs during chaos experiments. For example, if you run `chaos-mesh` during off-peak hours, configure a Cronguard rule to skip or delay non-critical jobs when CPU throttling or memory pressure exceeds a threshold. This prevents false positives from resource contention.

### Practice 2: **Use EMP for Compliance Audits**
Instead of storing raw logs for years (expensive and PII-heavy), store only the Merkle roots and the command hash. When an auditor asks for proof that a job ran at 02:00 UTC on a specific date, you can reconstruct the leaf from the agent’s local state and compare it against the stored root. This reduces log storage costs by ~80% while maintaining cryptographic verifiability.
