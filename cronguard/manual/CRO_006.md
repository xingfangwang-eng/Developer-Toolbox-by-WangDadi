# cronguard - CRO_006

Cronguard is not just another cron job checker. It’s a **declarative, idempotent, and audit-ready** cron job supervisor designed for SaaS teams that need to enforce SLA compliance across distributed schedules. Whether you're running 5 or 5,000 cron jobs, Cronguard gives you deterministic failure detection and automated remediation without the noise.

---

## Core Features (Hardcore)

### 1. **Deterministic Grace Period with Adaptive Jitter**

Cronguard doesn't just wait for a timeout. It uses **adaptive jitter** based on historical execution variance. Each cron job defines a `grace_window` in seconds, and Cronguard dynamically adjusts the alert threshold using a rolling median of the last 100 runs. This eliminates false positives from transient network spikes while catching true failures within 2 standard deviations of the baseline.

```yaml
cron_jobs:
  - name: "data-pipeline-sync"
    schedule: "*/5 * * * *"
    grace_window: 30
    adaptive_jitter: true
    alert_on_miss: true
```

### 2. **Idempotent Heartbeat with Cryptographic Nonce**

Every cron job push includes a **nonce** signed with HMAC-SHA256. Cronguard verifies the nonce against a rolling window to prevent replay attacks and duplicate executions. This is critical for financial or billing pipelines where double execution would be catastrophic.

```bash
curl -X POST https://cronguard.yourdomain.com/api/v1/heartbeat \
  -H "X-Signature: <hmac>" \
  -d '{"job_id": "pipeline-001", "nonce": "e3b0c44298fc1c14", "status": "success"}'
```

### 3. **Multi-Region Dead Man's Switch**

Cronguard supports a **dead man's switch** across three geographically independent regions. If the primary heartbeat region goes silent, the secondary region takes over alerting within 5 seconds (configurable). This is built for teams running cron jobs across AWS, GCP, and on-prem simultaneously.

```yaml
dead_mans_switch:
  regions:
    - us-east-1
    - eu-west-2
    - ap-southeast-1
  failover_threshold: 5s
  notify_on_failover: true
```

## How to Use (3 Steps)

### Step 1: Declare Your Cron Job Contract

Create a `cronguard.yaml` manifest in your repository root. This file defines every cron job with its schedule, grace window, and escalation policy. Cronguard will parse this file on startup and validate against your actual cron daemon.

```yaml
version: "1.0"
jobs:
  - name: "user-sync"
    schedule: "0 */6 * * *"
    grace_window: 120
    adaptive_jitter: true
    escalation:
      first: slack
      second: pagerduty
      third: sms
```

### Step 2: Instrument Your Cron Script with the Heartbeat SDK

Embed the Cronguard heartbeat call at the end of your cron script. Use the official Python or Go SDK to automatically generate the nonce and signature.

```python
from cronguard import Heartbeat

hb = Heartbeat(
    job_id="user-sync",
    api_key=os.getenv("CRONGUARD_API_KEY"),
    endpoint="https://cronguard.yourdomain.com"
)
hb.send(status="success", metadata={"rows_processed": 1234})
```

### Step 3: Configure Escalation Channels in Dashboard

Go to the Cronguard dashboard → **Settings → Escalation**. Add your Slack webhook, PagerDuty service key, and SMS number. Cronguard will automatically test each channel on save and report latency.

```bash
cronguard test-channel --type slack --webhook "https://hooks.slack.com/..."
```

## Best Practices (2 Industry Recommendations)

### 1. **Always Set an Idempotency Key for Billing Jobs**

If your cron job processes payments or invoices, add an `idempotency_key` to the heartbeat payload. Cronguard will reject duplicate nonces, preventing double billing even if the job runs twice due to a network retry.

```json
{
  "job_id": "billing-run",
  "nonce": "abc123",
  "idempotency_key": "invoice-cycle-2025-03-21",
  "status": "success"
}
```

### 2. **Use Adaptive Jitter for High-Frequency Jobs**

For jobs running every minute or less, enable `adaptive_jitter` and set a `grace_window` of 15–30 seconds. This reduces alert fatigue while still catching real failures. Cronguard automatically recalculates the baseline every 500 runs.
