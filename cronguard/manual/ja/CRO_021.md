# cronguard - CRO_021

## 概要

cronguard は、cron ジョブの監視・管理に特化した次世代型 SaaS ツールです。特に「ジョブの未実行検知」「実行履歴の可視化」「アラートの多層化」に強みを持ち、DevOps エンジニアや SRE チームが求める高信頼性 cron 運用を実現します。本ドキュメントでは、cronguard の技術的な核心と実践的な活用方法を解説します。

## 詳細な内容

### cronguard とは？

cronguard は、cron ジョブの実行状態をリアルタイムで監視し、障害発生時に即座に通知するプロフェッショナルツールです。従来の cron 監視ツールと異なり、**ジョブのスケジュール逸脱検知**、**実行時間の異常値検出**、**マルチチャネルアラート** を標準搭載。特にマイクロサービスアーキテクチャや Kubernetes 環境での cron ジョブ管理に最適化されています。

### コア機能

**1. スケジュール逸脱検知エンジン**
cronguard は、cron 式の解析と実績実行時刻を比較し、ジョブが予定から 30 秒以上遅延した場合に自動検知。さらに、ジョブの実行間隔が設定値から ±10% 以上乖離した場合もアラートを発報します。これにより、バッチ処理のタイムラグやジョブの重複実行を未然に防止します。

**2. 実行履歴のタイムライン可視化**
各ジョブの実行結果を、Gantt チャート形式で可視化。成功・失敗・タイムアウトを色分け表示し、ジョブの依存関係や実行順序を直感的に把握可能。また、実行時間のヒストグラムを自動生成し、パフォーマンスボトルネックを特定できます。

**3. マルチチャネルアラートとエスカレーションポリシー**
Slack、PagerDuty、Microsoft Teams、メール、Webhook に対応。さらに、特定のジョブが 3 回連続で失敗した場合にのみエスカレーションするなど、**条件付きアラートルール** を設定可能。重大度に応じて通知先を動的に切り替えることで、ノイズを最小化します。

### 使い方

**ステップ 1: API キーの発行とジョブ登録**
cronguard ダッシュボードでプロジェクトを作成し、API キーを発行します。次に、監視対象の cron ジョブを登録します。ジョブの定義は、cron 式・実行コマンド・期待実行時間（秒）を指定。例: `0 3 * * * /usr/bin/backup.sh` の場合、期待実行時間を `300` 秒と設定。登録後、`cronguard-agent` をジョブ実行前に呼び出すよう cron ファイルに追記します。

```bash
# crontab の例
0 3 * * * curl -X POST https://api.cronguard.io/v1/jobs/start -H "Authorization: Bearer YOUR_API_KEY" && /usr/bin/backup.sh && curl -X POST https://api.cronguard.io/v1/jobs/end -H "Authorization: Bearer YOUR_API_KEY"
```

**ステップ 2: アラートルールのカスタマイズ**
ダッシュボードの「アラート設定」で、以下の 3 つのルールを定義します。
- **未実行検知**: ジョブが実行開始されないまま 5 分経過したらアラート
- **タイムアウト検知**: 期待実行時間の 2 倍を超えたらアラート
- **異常値検知**: 実行時間が過去 7 日間の平均 ±3σ を超えたらアラート

各ルールに Slack チャンネルやメールアドレスを紐付け、通知の重要度（critical/warning/info）を設定します。

**ステップ 3: ダッシュボードで運用監視**
「ジョブ一覧」タブで、全ジョブのステータス（正常/警告/エラー）を一覧表示。各ジョブをクリックすると、過去 30 日間の実行履歴がタイムライン表示されます。異常が発生したジョブは、赤色ハイライトで即座に識別可能。また、週次レポート機能で、全ジョブの成功率・平均実行時間・最大遅延時間を CSV エクスポートできます。

### ベストプラクティス

**ベストプラクティス 1: ジョブの依存関係を明示的に定義する**
cron ジョブが複数存在する場合、`depends_on` フィールドで依存関係を設定することを推奨します。例えば、バックアップジョブが成功した後にのみレポートジョブを実行するよう定義します。cronguard は依存関係を自動解決し、親ジョブが失敗した場合に子ジョブの開始をブロックします。これにより、不完全なデータに基づく後続処理を防止します。

**ベストプラクティス 2: 実行時間のベースラインを週次で再計算する**
システム負荷やデータ量の変動により、ジョブの実行時間は変化します。cronguard の「動的ベースライン機能」を有効にし、毎週月曜日の午前 3 時に過去 7 日間の実行時間を再計算するよう設定します。これにより、季節変動やプロジェクトフェーズの変化に自動追従し、誤検知を最大 40% 削減できます。
