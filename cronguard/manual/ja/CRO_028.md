# cronguard - CRO_028

## 概要

cronguard は、高頻度ジョブスケジューリングとタスク実行の完全監視を実現する、エンタープライズグレードの cron ガードシステムです。ミリ秒単位の実行精度、PostgreSQL による分散ロック管理、および異常検知アラートを統合し、cron ジョブの欠落や重複実行をゼロにします。

## 詳細な内容

### cronguard とは

cronguard は、従来の crontab では不可能だった「実行履歴の永続化」「ジョブ間の依存関係制御」「実行時間の統計解析」を実現する次世代 cron 管理基盤です。特に、複数ノードで同一の cron 定義を共有しながらも、二重実行を完全防止する分散排他機構が中核機能です。

### コア機能

1. **分散ロックベースの排他実行制御**  
   PostgreSQL の `pg_advisory_lock` または Redis の `SET NX PX` を利用し、同一ジョブが複数ワーカーで同時に起動するのを防止。ロック獲得に失敗したワーカーは即座にスキップし、実行履歴テーブルに `status=skipped` を記録。ロックタイムアウトは設定可能（デフォルト 300 秒）。

2. **実行履歴の完全監査とレイテンシ計測**  
   全ジョブ実行ごとに、開始時刻（`started_at`）、終了時刻（`finished_at`）、終了コード（`exit_code`）、標準出力/エラー出力のスナップショットを PostgreSQL に保存。`cronguard query --job-id backup_db --since 7d` で過去 7 日間の実行時間分布（P50/P95/P99）をミリ秒単位で表示可能。

3. **ジョブチェーンと依存関係解決**  
   YAML 設定で `depends_on` フィールドを指定可能。例：`job_b` は `job_a` の成功完了後にのみ起動。依存ジョブが失敗した場合、後続ジョブは自動的に `status=blocked` となり、実行されない。チェーン全体の DAG を `cronguard graph` コマンドで可視化。

### 使い方

1. **初期設定とジョブ定義ファイルの作成**  
   ```bash
   cronguard init --db-url postgresql://user:pass@localhost:5432/cronguard --lock-type postgresql
   ```
   上記コマンドで `cronguard.yml` が生成される。次にジョブを定義：
   ```yaml
   jobs:
     backup_db:
       schedule: "0 3 * * *"
       command: "pg_dump mydb > /backups/db_$(date +%Y%m%d).sql"
       timeout: 600
       retry: 2
       alert_on_failure: true
   ```

2. **ワーカーノードの起動と登録**  
   ```bash
   # ノード1
   cronguard worker --node-id node-01 --config ./cronguard.yml
   # ノード2
   cronguard worker --node-id node-02 --config ./cronguard.yml
   ```
   各ワーカーは起動時に PostgreSQL の `workers` テーブルに自身のノードIDと最終ハートビート（`last_heartbeat`）を登録。デフォルト 10 秒間隔でハートビートを更新し、障害検知に利用。

3. **実行状況のリアルタイム監視とアラート設定**  
   ```bash
   cronguard watch --interval 5s --alert-webhook https://hooks.slack.com/services/xxx
   ```
   5 秒ごとにジョブの実行状態をポーリング。ジョブがタイムアウト（`timeout` 超過）した場合、またはリトライ上限に達した場合、Slack Webhook にアラートを送信。アラートペイロードにはジョブ名、ノードID、最終エラーログの先頭 500 文字を含む。

### ベストプラクティス

- **ジョブごとに個別の PostgreSQL トランザクション分離レベルを設定**  
  高負荷環境ではジョブの実行履歴書き込みとロック解放が競合する可能性がある。`cronguard.yml` の `execution_settings` で `isolation_level: repeatable_read` を指定し、履歴テーブルの更新競合を回避。特にレイテンシ計測の正確性が求められる金融系処理に推奨。

- **ワーカーノードの冗長化とヘルスチェック閾値の調整**  
  ワーカーが 3 台以上存在する場合、`cronguard.yml` の `worker_settings` で `heartbeat_timeout: 30`（秒）を設定。30 秒以上ハートビートが停止したノードは自動的に「死亡」とみなされ、そのノードが保持していたロックは強制解放される。これにより、単一ノード障害時のジョブ実行機会損失を最小化。
