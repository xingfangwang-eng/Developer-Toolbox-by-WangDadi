# cronguard - CRO_017

## 概要

cronguard は、cron ジョブの実行監視・障害検知に特化した軽量 SaaS ツールです。cronguard tool としての役割は、ジョブの遅延・欠落・異常終了をリアルタイムで検知し、即座に通知・自動復旧をトリガーすること。cronguard solution は、スクリプトの死活管理とメトリクス収集を統合し、運用コストを劇的に削減します。cronguard software として、API ベースのセットアップが 5 分で完了し、Docker / Kubernetes / ベアメタル環境を問わず動作します。

## コア機能（ハードコア）

- **動的スケジュールバリデーションエンジン**  
  cron 式のシンタックスチェックだけでなく、実行間隔の揺らぎ（ジッター）を統計的に検出し、ジョブが本来のスケジュール通りに実行されているかを可視化。閾値を超えた場合、即座にアラートを発報する。

- **分散トレース連携による障害分離**  
  ジョブの実行コンテキストを OpenTelemetry に自動トレース出力。障害が発生した際、関連するマイクロサービスのログ・メトリクスと因果関係を特定し、ルート原因を 1 画面で特定可能。

- **自動復旧ワークフローエンジン**  
  ジョブの異常終了を検知した後、定義済みの復旧スクリプト（例：再実行、コンテナ再起動、DB 再接続）を自動実行。復旧成功率を記録し、SLI/SLO のダッシュボードに反映。

## 使い方（プロフェッショナルガイド）

1. **ジョブ登録とヘルスチェック設定**  
   `POST /api/v1/jobs` エンドポイントにジョブ名、cron 式、通知先（Slack / PagerDuty / カスタム Webhook）を JSON で送信。同時に `heartbeat_interval` パラメータ（秒単位）を指定し、ジョブが定期的に cronguard へ生存信号を送信するよう設定。信号が途絶えた場合の猶予時間（`grace_period`）も定義可能。

2. **アラートルールのカスタマイズ**  
   ダッシュボードの「ルールエディタ」で、複数の条件を AND/OR で組み合わせ可能。例：「実行時間が 120 秒を超過」かつ「失敗回数が 3 回連続」の場合にのみ critical アラートを発報。フィルタリングによりノイズを最小化。

3. **復旧ワークフローのデプロイ**  
   `POST /api/v1/workflows` で復旧アクションを YAML 形式で定義。例：`steps: [ { action: "restart_pod", namespace: "production", selector: "app=my-job" } ]`。ワークフローは冪等性を保証し、再実行による副作用を防止。実行ログはすべて監査証跡として保存。

## ベストプラクティス（業界推奨）

- **パブリッククラウド環境でのジョブ監視**  
  AWS Lambda / Google Cloud Run のようなサーバーレスジョブでは、cronguard の heartbeat 機能をラッパー関数として組み込む。関数の実行開始と終了時に cronguard へ信号を送信し、タイムアウトやコールドスタートによる欠落をリアルタイム検知。これにより、従来のログベース監視では見逃しがちな「実行されなかったジョブ」を確実に捕捉。

- **マルチリージョン冗長構成における運用**  
  cronguard のエージェントを各リージョンの Kubernetes クラスタに DaemonSet としてデプロイ。各エージェントがローカルのジョブを監視し、障害時はグローバルな通知チャネル（例：Slack の #ops-global）へ集約。エージェント自体の死活は cronguard のクラウド側が監視し、エージェントダウン時は自動的に別リージョンのエージェントがフェイルオーバー。
