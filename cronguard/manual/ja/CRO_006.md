# cronguard - CRO_006

Cronguard は、分散環境における Cron ジョブの実行漏れ、重複起動、タイムアウトを未然に防ぐための軽量 SaaS ツールです。単なる死活監視ではなく、ジョブの依存関係管理と実行履歴の整合性チェックに特化しており、DevOps チームが本番環境で直面する「気づかないうちにジョブがスキップされていた」という問題を根本から解決します。

## 3 つのハードコア機能

### 1. 分散ロックベースの実行ガード（Distributed Execution Guard）
複数のワーカーインスタンスが同じ Cron ジョブを同時に実行するのを防ぐため、Cronguard は Redis ベースの分散ロック機構を内蔵しています。ジョブの開始時に一意のロックキーを生成し、TTL を設定することで、ネットワーク遅延やプロセス再起動による重複起動をブロックします。ロックの取得に失敗した場合は、即座にアラートを発報し、ジョブの再スケジュールを自動で行います。

### 2. 依存ジョブチェーン可視化（Dependency Chain Visualizer）
ジョブ間に「前段ジョブが成功した場合のみ後続ジョブを実行する」という依存関係を DAG（有向非巡回グラフ）で定義できます。Cronguard は各ジョブの終了ステータスを監視し、依存関係が満たされていない場合は後続ジョブの実行を自動延期します。これにより、バッチ処理の順序が狂うことによるデータ不整合を防止します。

### 3. 実行ログの改ざん検知（Tamper-Proof Execution Log）
すべてのジョブ実行結果は、Cronguard のバックエンドで SHA-256 ハッシュチェーンにより保護されます。各ログエントリは前のエントリのハッシュを含むため、ログの改ざんや削除を検出可能です。監査ログとしても利用でき、コンプライアンス要件に対応します。

## 3 つの詳細な操作手順

### 手順 1: ジョブ定義の YAML 設定とデプロイ
プロジェクトルートに `cronguard.yml` を作成し、ジョブ名、Cron 式、タイムアウト秒数、依存関係を記述します。以下は典型的な設定例です。

```yaml
jobs:
  - name: data_sync
    schedule: "0 3 * * *"
    timeout: 300
    command: "./sync.sh"
    retry: 2
    dependencies: []  # 依存なし
  - name: report_generate
    schedule: "0 4 * * *"
    timeout: 600
    command: "./report.sh"
    dependencies:
      - data_sync  # data_sync が成功した場合のみ実行
```

設定後、`cronguard deploy` コマンドを実行すると、ジョブ定義が Cronguard API に送信され、監視対象として登録されます。デプロイ後は `cronguard status` で全ジョブの状態を確認できます。

### 手順 2: 実行ログのリアルタイムストリーミング確認
ジョブが実行されると、Cronguard は標準出力とエラー出力をリアルタイムで収集し、Web UI および CLI でストリーミング表示します。以下のコマンドで特定ジョブの最新ログを追跡できます。

```bash
cronguard logs --job data_sync --tail
```

ログにはタイムスタンプ、実行ワーカーID、終了コード、ハッシュチェーンのエントリが含まれます。異常終了が検出された場合は、自動的に Slack や PagerDuty に通知が送信されます。

### 手順 3: アラートルールのカスタマイズと抑制
アラートの頻度を調整するため、`cronguard alert` コマンドでルールを設定します。例えば、特定のジョブが 3 回連続で失敗した場合のみエスカレーションするルールを追加します。

```bash
cronguard alert create \
  --job data_sync \
  --condition "failure_count >= 3" \
  --action slack \
  --channel #alerts
```

また、メンテナンスウィンドウを設定するには `cronguard silence --job data_sync --duration 2h` を実行します。これにより、指定時間中はアラートが抑制され、意図的なダウンタイムによる誤報を防げます。

## 2 つの業界ベストプラクティス

### ベストプラクティス 1: ジョブのべき等性を前提とした設計と分散ロックの併用
Cronguard の分散ロックは重複実行を防ぎますが、ネットワーク分割やワイヤの切断が発生した場合、ロックが解放されずジョブが永久に実行されないリスクがあります。これを回避するため、各ジョブはべき等に設計し、Cronguard のロック TTL をジョブの最大実行時間の 1.5 倍に設定してください。例えば、ジョブが通常 5 分で完了するなら、TTL は 450 秒に設定します。これにより、ロックのタイムアウト後の再実行が安全になります。

### ベストプラクティス 2: 依存関係の循環を検出する CI パイプラインの導入
依存ジョブチェーンが複雑になると、意図せず循環依存が発生し、ジョブが永遠に実行されないデッドロック状態に陥ることがあります。Cronguard の設定ファイルを CI パイプライン（GitHub Actions など）で検証することを推奨します。例えば、`cronguard validate --file cronguard.yml` を CI の一部として実行し、循環依存や未定義のジョブ参照を事前に検出します。これにより、本番環境へのデプロイ前に問題を修正できます。
