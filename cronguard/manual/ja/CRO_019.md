# cronguard - CRO_019

## 概要

cronguard は、cron ジョブの実行履歴をリアルタイムで監視し、障害検知・パフォーマンス最適化・セキュリティ監査を自動化するエンタープライズ向け SaaS ツールです。本プロジェクトは、特に大規模分散環境におけるジョブスケジューリングの信頼性と可視性を向上させるために設計されています。

## 詳細な内容

### cronguard とは

cronguard は、cron ジョブの実行状態を**リアルタイムに収集・分析**し、異常終了、実行遅延、リソースリーク、セキュリティ脆弱性を自動検出します。従来の cron 監視ツールではカバーできない「ジョブ間依存関係の整合性チェック」や「実行履歴の機械学習による異常予測」を実現。DevOps チームが運用負荷を減らしながら、ジョブ障害の MTTR（平均復旧時間）を 90% 削減することを目指します。

### コア機能（3 つのハードコア機能）

#### 1. ジョブ依存関係グラフとクリティカルパス分析
- cron ジョブ同士の依存関係を自動解析し、DAG（有向非巡回グラフ）として可視化。
- 障害が発生した場合、影響範囲を即座に特定し、クリティカルパス上のジョブを優先的に再実行するリカバリオプションを提示。
- ジョブチェーン全体のレイテンシを測定し、ボトルネックを特定。

#### 2. 異常検知エンジン（機械学習ベース）
- 過去 30 日間の実行履歴から、ジョブごとの正常な実行時間、リソース消費量、終了コードパターンを学習。
- 学習モデルに基づき、実行時間が標準偏差の 2 倍を超えた場合や、リソース使用量が急激に増加した場合にアラートを発報。
- 異常検知の感度はユーザーが調整可能（高感度モード/標準モード/低感度モード）。

#### 3. セキュリティ監査とコンプライアンスレポート
- cron ジョブ内のシェルスクリプトやコマンドを静的解析し、インジェクション脆弱性、権限昇格のリスク、機密情報のハードコードを検出。
- PCI-DSS、SOC2、HIPAA などのコンプライアンス要件に基づいた監査レポートを自動生成。
- 検出された脆弱性には推奨修正コードを提示。

### 使い方（3 つの詳細な操作手順）

#### 手順 1: cronguard エージェントのインストールと初期設定
```bash
# 1. cronguard エージェントをダウンロード
curl -fsSL https://cronguard.io/install.sh | bash

# 2. エージェントを cron ジョブ実行環境にデプロイ
sudo cronguard agent install --config /etc/cronguard/config.yaml

# 3. config.yaml に監視対象の cron ジョブを定義
cat <<EOF > /etc/cronguard/config.yaml
monitoring:
  cron_path: /var/spool/cron/crontabs
  log_path: /var/log/cron
  alert:
    email: team@example.com
    slack_webhook: https://hooks.slack.com/services/xxx
  anomaly_detection:
    sensitivity: standard
EOF

# 4. エージェント起動
sudo systemctl start cronguard-agent
sudo systemctl enable cronguard-agent
```

#### 手順 2: ジョブ依存関係の自動マッピングと可視化
```bash
# 1. cronguard CLI で依存関係スキャンを実行
cronguard scan --dependency --output json > dependency_graph.json

# 2. 生成された JSON を cronguard ダッシュボードにアップロード
cronguard dashboard import --file dependency_graph.json

# 3. ダッシュボード上で依存関係グラフを確認
#    - ノード: 各ジョブ（色分け: 正常/警告/エラー）
#    - エッジ: 依存関係（実線: 正常、点線: 未解決）
#    - クリティカルパス: 赤色ハイライト

# 4. クリティカルパス上のジョブにアラートルールを設定
cronguard alert create --job "data_pipeline" --condition "execution_time > 300" --action "slack"
```

#### 手順 3: 異常検知モデルのトレーニングとアラート設定
```bash
# 1. 過去 30 日間の実行履歴をエクスポート
cronguard history export --days 30 --format csv > training_data.csv

# 2. 機械学習モデルをトレーニング
cronguard ml train --data training_data.csv --model anomaly_model_v1.pkl

# 3. トレーニング済みモデルを cronguard に適用
cronguard ml deploy --model anomaly_model_v1.pkl

# 4. 異常検知アラートのしきい値を調整
cronguard alert threshold --metric execution_time --std_dev_multiplier 2.5

# 5. テスト実行: モデルが正常に動作するか確認
cronguard ml test --job "batch_processing" --expected_time 120 --actual_time 350
# 出力: "異常検知: 実行時間が期待値の 291% です。アラートを発報します。"
```

### ベストプラクティス（2 つの業界推奨）

#### 1. ジョブ実行履歴の長期保存と定期監査
- cronguard はデフォルトで 90 日間の実行履歴を保持しますが、**コンプライアンス要件に応じて 1 年以上の保存**を推奨します。特に金融業界や医療業界では、監査証跡としての利用が必須です。
- 毎月 1 回、cronguard の「セキュリティ監査レポート」を自動生成し、DevOps チームとセキュリティチームでレビューするサイクルを確立してください。

#### 2. 段階的な異常検知モデルのチューニング
- 初回導入時は「低感度モード」で運用し、誤検知を最小限に抑えてください。2 週間後に「標準モード」に切り替え、さらに 1 ヶ月後には「高感度モード」を試すことを推奨します。
- モデルの精度を向上させるため、**異常と判断されたジョブのうち、実際に障害だったものの割合（適合率）** を毎週確認し、しきい値を調整してください。
