# cronguard - CRO_003

## 概要

cronguard は、分散 Cron ジョブの実行状態をリアルタイムで可視化し、障害発生時に即座に検知・通知する SaaS 型監視ツールです。従来の「動いているかどうか分からない」という Cron ジョブ管理の課題を、シグネチャベースの健全性チェックとメトリクス集約により解決します。本プロジェクトは、SRE チームや DevOps エンジニアが複数サーバー・複数環境にまたがるジョブスケジュールを統合管理するための専用ソリューションです。

## コア機能

### 1. 分散ジョブシグネチャ照合エンジン

各 Cron ジョブに対して、実行開始時と終了時に一意のシグネチャ（ハッシュ値＋タイムスタンプ）を生成。cronguard エージェントがこれを収集し、クラウド側で期待値と照合します。ジョブが予定時刻に開始されなかった、異常終了した、出力が欠損しているなどの状態を、ミリ秒単位で検出します。従来の「プロセスが生きているか」だけでは見逃されるケース（例: ジョブが起動したが初期化でハング）も、シグネチャ未到達として確実にキャッチします。

### 2. ジョブ依存関係グラフによるクリティカルパス可視化

複数のジョブが連鎖するワークフロー（例: ETL → 集計 → レポート生成）において、各ジョブの依存関係を DAG（有向非巡回グラフ）として自動生成。各ノードに実行時間、リトライ回数、エラーレートを重ねて表示します。異常が発生した場合、その影響範囲（下流のジョブブロック数）を即座に計算し、クリティカルパス上のボトルネックを特定。スケジューリングの最適化ポイントを具体的に提示します。

### 3. マルチクラスタエージェント自動スケーリング

Kubernetes クラスタや EC2 Auto Scaling Group など、動的にノード数が変動する環境でも、cronguard エージェントが自動的に検出・登録されます。各エージェントはジョブ実行コンテキスト（環境変数、実行ユーザー、カレントディレクトリ）を収集し、クラウド側で一元管理。ノード追加時にはエージェントが自動的にジョブスケジュールを取得し、即座に監視を開始します。エージェントの死活監視は gRPC 双方向ストリームで行われ、切断時には即座にアラートが発報されます。

## 使い方

### 1. エージェント設定ファイルの作成とジョブシグネチャ登録

まず、監視対象サーバーに cronguard エージェントをインストールします。次に、`/etc/cronguard/agent.yml` を作成し、監視する Cron ジョブの定義を記述します。各ジョブには `signature_expected` として、正常終了時に出力されるべき文字列の SHA256 ハッシュ値を設定します。例えば、バックアップジョブが `Backup completed at 2024-01-01` と出力する場合、そのハッシュを事前計算して設定します。エージェントはジョブ実行後に実際の出力ハッシュと照合し、不一致があれば即座に API 経由でアラートを送信します。

### 2. 依存関係グラフの生成と閾値設定

Web コンソールの「ジョブ依存関係」タブで、各ジョブの `depends_on` フィールドに先行ジョブのジョブ名を指定します。例えば、`data_export` ジョブが `db_backup` の完了を待つ場合、`depends_on: ["db_backup"]` と設定します。cronguard はこれを解析し、自動的に DAG を生成。各エッジに「最大待機時間（秒）」を設定することで、先行ジョブの遅延が後続に与える影響を数値化します。閾値を超えた場合は、クリティカルパス上の全ジョブに対してアラートが発報されます。

### 3. アラートルーティングルールのカスタマイズ

`/etc/cronguard/rules.yml` に、アラートの重要度に応じた通知先を定義します。例えば、`critical` レベルのアラートは PagerDuty と Slack の両方に送信し、`warning` レベルは Slack のみ、`info` レベルはログ出力のみ、といったルーティングが可能です。また、`aggregation_window: 60` を設定することで、同一ジョブからの連続アラートを 60 秒間集約し、通知ノイズを低減します。各ルールには `escalation_policy` を指定でき、初回通知から 5 分以内に確認応答がない場合、上位の管理者へ自動エスカレーションします。

## ベストプラクティス

### 1. ジョブ実行履歴に基づく動的閾値の導入

静的なタイムアウト値ではなく、過去 7 日間の実行時間の平均と標準偏差から動的な閾値を算出することを推奨します。cronguard のメトリクス API から過去の実行時間を取得し、`平均 + 3σ` をアラート閾値として設定します。これにより、週末のバッチ処理や月末の決算処理など、負荷変動が大きいジョブでも誤検知を最小限に抑えられます。実装例として、Python スクリプトで cronguard API を叩き、閾値を自動更新する CI パイプラインを構築すると効果的です。

### 2. カナリアリリースと連動したジョブ検証

新しい Cron ジョブを本番環境に投入する前に、cronguard の「ドライランモード」を活用します。ドライランモードでは、ジョブのシグネチャ照合と依存関係チェックのみ実行し、実際のジョブは起動しません。これにより、設定ミス（例えば、存在しないジョブ名を `depends_on` に指定した場合）や、循環依存の発生を事前に検出できます。カナリアリリースのパイプラインにこのドライランチェックを組み込むことで、本番障害のリスクを大幅に低減できます。
