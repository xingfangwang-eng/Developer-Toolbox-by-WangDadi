# cronguard - CRO_018

## 概要

cronguard は、ジョブスケジューリングと Cron タスク管理を極限まで自動化・可視化する次世代型監視ツールです。本プロジェクトは、SaaS アーキテクチャに特化した軽量バックエンドサービスとして設計され、複数ノード間のジョブ整合性、障害検知、リトライ機構を統合管理します。キーワード: cronguard ツール, cronguard ソリューション, cronguard ソフトウェア。

## 詳細な内容

### cronguard とは？

cronguard は、従来の Cron ジョブが抱える「実行漏れ」「重複起動」「ログ散乱」といった運用課題を解決するために設計された、エンタープライズグレードのジョブスケジューリングプラットフォームです。分散環境においても、単一障害点（SPOF）を排除し、高可用性（HA）を実現。RESTful API による外部連携、メトリクスダッシュボード、Slack / PagerDuty へのリアルタイムアラート通知を標準搭載しています。

### コア機能

1. **ジョブ依存関係グラフによる DAG スケジューリング**  
   cronguard は、単純な Cron 式だけでなく、ジョブ間の依存関係を DAG（有向非巡回グラフ）として定義できます。ジョブ A の成功後にジョブ B を起動、失敗時はジョブ C にフォールバック — といった複雑なワークフローを、YAML または JSON で宣言的に記述可能です。依存解決はトポロジカルソートで自動実行され、並列実行可否も制御できます。

2. **実行履歴の完全トレースとリトライポリシーエンジン**  
   全ジョブ実行は、開始・終了・エラー・リトライの各イベントがタイムスタンプ付きで永続化されます。リトライポリシーは指数バックオフ、固定間隔、カスタム条件（例: HTTP 500 のみリトライ）をサポート。実行ログはストリーム形式でリアルタイム取得可能で、障害解析時に過去の実行をタイムラインで確認できます。

3. **分散ロックによる重複起動防止とノード間同期**  
   複数ワーカーノードが同一ジョブを同時に実行しないよう、Redis または PostgreSQL をバックエンドとした分散ロック機構を実装。リーダーエレクションによりアクティブノードを自動選出し、ノード障害時には即座に別ノードがジョブを引き継ぎます。これにより、cronguard はクラスタ構成でも完全なジョブ整合性を保証します。

### 使い方

1. **ジョブ定義ファイルの作成とデプロイ**  
   プロジェクトルートに `cronguard.yml` を作成し、以下のようにジョブを定義します。
   ```yaml
   jobs:
     - name: data_ingestion
       schedule: "0 */2 * * *"
       command: "python ingest.py"
       retry: 3
       retry_delay: "30s"
       dependencies: []
     - name: data_validation
       schedule: null
       command: "python validate.py"
       dependencies: ["data_ingestion"]
   ```
   その後、`cronguard deploy --file cronguard.yml` を実行すると、定義がサーバーにアップロードされ、即座にスケジューリングが開始されます。デプロイ後は `cronguard status` で全ジョブの状態を一覧表示できます。

2. **リアルタイムダッシュボードでの監視とアラート設定**  
   `cronguard dashboard` を起動すると、ブラウザ上でジョブ実行状況が可視化されます。各ジョブの実行時間、成功/失敗回数、平均レイテンシが表示され、異常なジョブは赤色でハイライトされます。アラート設定は `cronguard alert add --type slack --webhook https://hooks.slack.com/...` で追加可能。閾値を超えたエラー率や実行遅延が発生した場合、即座に通知が送られます。

3. **障害時の手動リカバリとジョブ再実行**  
   万が一ジョブが失敗した場合、`cronguard rerun --job data_ingestion --force` で任意のジョブを強制再実行できます。`--force` フラグを付けると、依存関係を無視して単体実行が可能。また、`cronguard pause --job data_validation` で特定ジョブを一時停止し、`cronguard resume` で再開できます。全ジョブの停止は `cronguard stop --all` で一括制御可能です。

### ベストプラクティス

1. **ジョブ定義はバージョン管理し、CI/CD パイプラインに統合せよ**  
   cronguard のジョブ定義ファイル（YAML/JSON）は、Git リポジトリで管理することを強く推奨します。プルリクエストによるレビューを経て `cronguard deploy` を CI パイプラインの一部として実行することで、本番環境へのデプロイミスを防げます。また、定義変更の履歴が残るため、障害発生時の原因特定が容易になります。

2. **ジョブ実行時間の SLA を設定し、異常検知を自動化せよ**  
   各ジョブには期待実行時間（例: 5 分以内）をメタデータとして設定し、cronguard のアラート機能で SLA 違反を自動検知するのが効果的です。`cronguard alert add --type webhook --condition "duration > 300"` のように設定すれば、実行が 5 分を超えた時点で即座に Slack や PagerDuty に通知が飛びます。これにより、障害の初動対応が劇的に向上します。
