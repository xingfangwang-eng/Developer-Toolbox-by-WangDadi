# cronguard - CRO_029

## 概要

cronguard は、高頻度ジョブスケジューリングの信頼性保証と障害耐性を極限まで追求した分散型ウォッチドッグツールです。特に cron デーモンの無応答状態やジョブの異常遅延をミリ秒単位で検知し、自動フェイルオーバーを実行します。本ツールは、PostgreSQL と組み合わせたジョブ実行履歴の永続化、リアルタイムアラート発報、複数ワーカー間でのスケジュール分散を標準搭載しています。

## 詳細な内容

### cronguard の設計思想

cronguard は、従来の cron 代替ツールが持つ「単一ノード障害によるジョブ消失」問題を解決するため、分散トランザクションログとリーダー選出機構を内部に実装しています。各ワーカーは PostgreSQL のアドバイザリロックを利用してジョブ所有権を取得し、ノード障害時には 500 ミリ秒以内に別ワーカーがジョブを引き継ぎます。また、全ジョブの実行結果は `job_execution_logs` テーブルに記録され、過去 90 日間のトレースが可能です。

### コア機能

1. **分散スケジューラと障害検知**
   - 各ワーカーが PostgreSQL 上の `cron_jobs` テーブルを監視し、`next_run_at` カラムに基づいてジョブを実行。
   - ワーカー間のハートビート間隔は 1 秒、3 回連続でハートビートが途絶えたノードは障害と判定。
   - 障害検知後、即座にリーダー選出プロトコル（Raft ライク）を起動し、残存ワーカーが未実行ジョブを再割り当て。

2. **ジョブ実行履歴の完全監査**
   - `job_execution_logs` テーブルに以下の情報を記録：`job_id`、`worker_id`、`started_at`、`completed_at`、`exit_code`、`stdout_summary`、`error_stack_trace`。
   - 実行時間が `max_execution_time`（デフォルト 300 秒）を超過したジョブは自動強制終了（SIGKILL）し、`TIMEOUT` ステータスでログに記録。
   - 全ログはミリ秒精度のタイムスタンプ付きで、Prometheus や Grafana との連携も可能。

3. **動的ジョブバランシングとリトライ戦略**
   - 各ワーカーの現在負荷（同時実行ジョブ数、CPU 使用率、メモリ使用量）を 10 秒ごとに `worker_metrics` テーブルに報告。
   - 新規ジョブ割り当て時、最も負荷の低いワーカーを自動選択。
   - 失敗ジョブに対して指数バックオフリトライ（初回 1 秒、最大 60 秒、最大リトライ回数 5 回）を実行し、リトライ上限超過時は管理者にメール通知。

### 使い方

1. **初期セットアップとデータベースマイグレーション**
   ```bash
   cronguard init --db-host 192.168.1.100 --db-port 5432 --db-name cronguard_prod --db-user admin --db-password secret
   ```
   上記コマンドで PostgreSQL 上に `cron_jobs`、`job_execution_logs`、`worker_metrics` の 3 テーブルが自動生成されます。`--with-sample-jobs` フラグを追加すると、動作確認用のサンプルジョブ（5 分ごとに `echo "alive"` を実行）が登録されます。

2. **ジョブの登録とスケジュール設定**
   ```bash
   cronguard job add --name "daily_backup" --schedule "0 3 * * *" --command "/usr/local/bin/backup.sh" --max-execution-time 600 --retry-count 3
   ```
   このジョブは毎日午前 3 時に実行され、最大実行時間 600 秒、失敗時 3 回までリトライします。`--timezone Asia/Tokyo` オプションでタイムゾーン指定も可能です。

3. **ワーカーの起動とクラスタ参加**
   ```bash
   cronguard worker start --worker-id node-01 --db-connection-string "postgresql://admin:secret@192.168.1.100:5432/cronguard_prod" --heartbeat-interval 1s
   ```
   ワーカー起動後、即座にリーダー選出プロセスに参加します。`--dry-run` モードで設定の検証のみ行うことも可能です。

### ベストプラクティス

1. **ジョブ設計における冪等性の徹底**
   cronguard は障害復旧時に同一ジョブが複数回実行される可能性があります。すべてのジョブスクリプトは冪等に実装し、重複実行によるデータ不整合を防いでください。特にファイル書き込みや API 呼び出しを含むジョブでは、ジョブ内で一意なトランザクション ID を生成し、重複チェックを実装することを推奨します。

2. **監視基盤との統合によるプロアクティブ運用**
   `job_execution_logs` テーブルに対して、以下のような監視クエリを 5 分間隔で実行することを推奨します。
   ```sql
   SELECT job_id, COUNT(*) AS failure_count
   FROM job_execution_logs
   WHERE exit_code != 0 AND started_at > NOW() - INTERVAL '1 hour'
   GROUP BY job_id
   HAVING COUNT(*) > 3;
   ```
   これにより、特定ジョブの連続失敗を早期発見し、cronguard のアラート機能と連携して迅速な対応が可能になります。
