# cronguard - CRO_030

## 概要

cronguard は、cron ジョブの異常終了・スケジュール競合・リソースリークを高頻度監視し、障害発生から **200ms 以内** に検知・通知する国産ジョブガードエンジンです。PostgreSQL 15+ とのネイティブ連携により、ジョブ実行履歴をミリ秒単位で記録し、過去 90 日間のパターン分析から次回の実行遅延を予測します。

## コア機能

### 1. マイクロ秒精度のスケジュール競合検出
- 内部パラメータ `--conflict-window=50ms` により、同一ノード上で 2 つ以上の cron ジョブが 50 ミリ秒以内に重複起動しようとした場合、即座に競合フラグを立てる。
- 競合発生時には `cronguard-agent` が `SIGSTOP` を送出し、優先度の低いジョブを一時停止。PostgreSQL の `pg_advisory_lock` と連動してデッドロックを防止する。

### 2. 履歴ベースの異常実行時間検知（ヒューリスティックエンジン）
- 各ジョブの実行時間を `cronguard_history` テーブルに保存し、平均値 ± 3σ を超える異常値を検出。
- パラメータ `--anomaly-threshold=3.0` で標準偏差倍率を調整可能。検出後は `webhook` 経由で Slack / PagerDuty に **1.2 秒以内** にアラートを送信。

### 3. ジョブチェーン依存関係の自動解決とリトライ戦略
- `cronguard.yml` に `depends_on: [job_a, job_b]` を記述すると、前段ジョブの終了コードが 0 でない場合、後段ジョブを自動スキップ。
- リトライ設定 `--retry=3 --retry-interval=5s` により、失敗ジョブを 5 秒間隔で最大 3 回再実行。再実行時は `X-Cronguard-Retry: 2` ヘッダを HTTP コールバックに付与。

## 使い方

### ステップ 1: cronguard デーモンの起動と PostgreSQL スキーマ初期化
```bash
cronguard init --db-host=localhost --db-port=5432 --db-name=cronguard_db \
  --schema-file=/etc/cronguard/schema_v3.sql
```
- 上記コマンドで `cronguard_jobs`、`cronguard_history`、`cronguard_locks` の 3 テーブルが自動生成される。
- 初期化後、`cronguard daemon --log-level=debug --conflict-window=50ms` でデーモンを起動。

### ステップ 2: 監視対象ジョブの登録と依存関係定義
```bash
cronguard register --job-name=db_backup --cron="0 3 * * *" \
  --command="/usr/local/bin/pg_dumpall > /backup/db.sql" \
  --depends-on=health_check --retry=2 --retry-interval=10s \
  --timeout=300s --anomaly-threshold=2.5
```
- `--timeout=300s` で最大実行時間を設定。超過時は `SIGKILL` を送出し、`cronguard_history` に `status=timeout` を記録。
- 依存ジョブ `health_check` が失敗した場合、`db_backup` は自動スキップされ、`skipped_by_dependency` フラグが立つ。

### ステップ 3: リアルタイムダッシュボードの起動とアラート確認
```bash
cronguard dashboard --port=8080 --webhook-url=https://hooks.slack.com/services/T00/B00/xxx \
  --pagerduty-routing-key=abc123
```
- ダッシュボードは WebSocket 経由でジョブ実行状況を **100ms 単位** で更新。競合検出時には画面上に赤色のフラッシュが表示される。
- Slack 通知のペイロード例（自動生成）:
```json
{
  "job_name": "db_backup",
  "status": "conflict",
  "conflict_with": "log_rotate",
  "detected_at": "2025-04-11T03:00:00.123Z",
  "resolution": "paused_low_priority"
}
```

## ベストプラクティス

### プラクティス 1: 競合ウィンドウの適正化による誤検知抑制
- 同一ホスト上で cron ジョブが密集している環境では、`--conflict-window` を **100ms** に緩和することを推奨。
- 一方、コンテナ環境（Kubernetes Pod 内）ではノイズが少ないため **20ms** に設定し、高精度な競合検出を維持する。設定変更後は必ず `cronguard stats --conflict-false-positive-rate` で誤検知率を確認すること。

### プラクティス 2: 履歴保持期間とパーティショニング戦略
- `cronguard_history` テーブルは月次パーティションを推奨。例: `cronguard_history_2025_04`。
- 保持期間は **90 日** を標準とし、`cronguard purge --older-than=90d` で定期的にクリーンアップ。パーティション単位で `DROP` することで、VACUUM の負荷を軽減できる。
- インデックスは `(job_name, started_at DESC)` に複合インデックスを張ると、異常検出クエリが 3 倍高速化する。
