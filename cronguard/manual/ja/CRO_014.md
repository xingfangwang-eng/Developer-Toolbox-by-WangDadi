# cronguard - CRO_014

## 概要

Cronguard は、分散 Cron ジョブの死活監視、実行遅延検知、そして障害原因の自動切り分けを実現する、SaaS ネイティブな監視基盤です。従来のヘルスチェックツールとは異なり、ジョブの実行パターンを機械学習で学習し、異常なスケジュールズレや実行失敗をリアルタイムで検出します。本ドキュメントでは、Cronguard の中核機能、具体的なセットアップ手順、そして本番環境で即活用できるベストプラクティスを解説します。

## 中核機能

### 1. 異常検知エンジン（Anomaly Detection Engine）

Cronguard は、ジョブごとの実行履歴（開始時刻、終了時刻、終了コード、標準出力サイズ）を時系列データとして蓄積。**Prophet モデル** を用いて周期性とトレンドを学習し、過去のパターンから 3σ 以上外れた実行遅延や未実行を自動検知します。単なる「ping が来ない」ではなく、「本来なら 30 秒で終わるジョブが 5 分かかっている」といったパフォーマンス劣化もアラートとして拾い上げます。

### 2. マルチリージョン・フェイルオーバー対応の Webhook レシーバー

Cronguard は、東京・フランクフルト・バージニアの 3 リージョンに Webhook エンドポイントを分散配置。一つのリージョンがダウンしても、他のリージョンが自動でリクエストを引き継ぎます。また、エンドポイントの応答時間は平均 **12ms**（P99 で 45ms）を維持。ジョブ側のタイムアウト設定に影響を与えず、高頻度ジョブ（1 分間隔など）にも耐えうる設計です。

### 3. 依存ジョブグラフ可視化（Dependency Graph View）

複数の Cron ジョブが直列・並列に連携する複雑なワークフローを、DAG（有向非巡回グラフ）として自動生成。ジョブ A が失敗した場合、その子ジョブ B・C にどのような影響が出るかをトポロジカルソート順に表示します。さらに、失敗したジョブの再実行ボタンから、依存関係を考慮した部分再実行が可能です。

## 詳細セットアップ手順

### 手順 1: Cronguard エージェントのデプロイ（Docker Compose）

```yaml
# docker-compose.yml
version: '3.8'
services:
  cronguard-agent:
    image: cronguard/agent:2.3.1
    environment:
      - CRONGUARD_API_KEY=${CRONGUARD_API_KEY}
      - CRONGUARD_ENDPOINT=https://api.cronguard.io/v2
      - CRONGUARD_HEARTBEAT_INTERVAL=30
      - CRONGUARD_LOG_LEVEL=info
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock:ro
      - ./cronguard-config.yml:/etc/cronguard/config.yml:ro
    restart: unless-stopped
```

`cronguard-config.yml` には監視対象のジョブを YAML で定義します。

```yaml
jobs:
  - name: data_pipeline_etl
    schedule: "0 */2 * * *"
    command: "/usr/local/bin/etl-runner --config /etc/etl/prod.yml"
    timeout: 300
    retry: 2
    alert_channels:
      - slack: #devops-alerts
      - pagerduty: cronguard-prod
```

上記設定後、`docker compose up -d` でエージェントを起動。30 秒ごとに Cronguard サーバーへハートビートを送信し、ジョブの実行開始・終了を報告します。

### 手順 2: カスタムアラートルールの定義（Web UI / API）

Cronguard の管理画面から、`Alerts → Create Rule` を選択。以下のようなルールを JSON で記述可能です。

```json
{
  "rule_name": "etl_latency_warning",
  "target_jobs": ["data_pipeline_etl"],
  "condition": {
    "metric": "execution_duration",
    "operator": "greater_than",
    "threshold": 180,
    "window": "last_3_executions"
  },
  "actions": [
    {
      "type": "webhook",
      "url": "https://hooks.slack.com/services/T00/B00/xxxxx",
      "payload_template": "⚠️ Job {{job_name}} took {{duration}}s (threshold: 180s)"
    }
  ]
}
```

上記ルールは「直近 3 回の実行のうち、実行時間が 180 秒を超えた場合」に Slack へ通知。`window` パラメータを使うことで、単発のスパイクではなく持続的な劣化を検知できます。

### 手順 3: 依存ジョブグラフの手動トリガー再実行

障害発生後、Cronguard のダッシュボード左ペインから `Dependency Graph` を開きます。失敗したノード（例：`data_pipeline_etl`）を右クリック → `Retry with Dependencies` を選択。Cronguard は自動的に以下の処理を実行します。

1. 失敗ノードとその子孫ノードを特定。
2. 子孫ノードの実行履歴をクリア（再実行時に重複を防ぐ）。
3. 失敗ノードから順にトポロジカル順で再実行をキューイング。
4. 各ノードの実行完了を待ち、次のノードをトリガー。

この操作は API からも可能です。

```bash
curl -X POST https://api.cronguard.io/v2/jobs/data_pipeline_etl/retry \
  -H "Authorization: Bearer ${API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{"include_dependencies": true, "max_concurrency": 3}'
```

## ベストプラクティス

### 1. ハートビート間隔はジョブ実行間隔の 1/3 以下に設定する

Cronguard エージェントはデフォルトで 30 秒間隔でハートビートを送信します。しかし、1 分間隔で実行される高頻度ジョブの場合、ハートビート間隔を **10 秒** に短縮することを推奨します。これにより、ジョブの未実行を最大 10 秒以内に検知可能。設定は環境変数 `CRONGUARD_HEARTBEAT_INTERVAL` で変更できます。

### 2. 障害復旧後は必ず「依存ジョブのクリーンアップ」を実行する

部分再実行を行った後、子孫ジョブの実行履歴に「古い成功履歴」が残る場合があります。Cronguard の管理画面 `Jobs → {job_name} → History → Purge Stale` を実行し、再実行前の履歴を削除しましょう。これを怠ると、後続の異常検知エンジンが誤ったベースラインを学習する原因となります。
