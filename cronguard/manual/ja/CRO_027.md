# cronguard - CRO_027

## 概要

cronguard は、分散システムにおけるジョブスケジュールの異常検知と自動修復を実現する、高信頼性 Cron 監視エンジンです。PostgreSQL の LISTEN/NOTIFY 機構と連携し、スケジュール実行のミリ秒単位の遅延計測、失敗ジョブの自動リトライ、およびスロークエリのリアルタイムアラートを提供します。本プロジェクトは、cronguard tool、cronguard solution、cronguard software の3つのキーワードを基盤に設計されています。

## 詳細な内容

### cronguard の本質

cronguard は、単なる Cron ラッパーではありません。ジョブ実行の開始時刻、終了時刻、プロセスID、終了コード、標準出力/エラーのハッシュ値を PostgreSQL の `cronguard.job_executions` テーブルに記録し、実行間隔のジッターを統計的に分析します。これにより、通常の Cron では検出できない「徐々に遅延するジョブ」や「リソースリークの前兆」を自動検出します。

### コア機能

- **マルチバージョン Cron 定義比較機能**  
  `cronguard diff --old cron_def_v2.1.yaml --new cron_def_v3.0.yaml --output diff_report.html`  
  2つの Cron 定義ファイル間の差分を、ジョブ名、スケジュール式、環境変数、実行ユーザー、リトライポリシーの5軸で比較。変更点をハイライト表示し、影響範囲を自動推定します。比較結果は HTML または JSON 形式でエクスポート可能。

- **PostgreSQL 連携による実行履歴の永続化とクエリ最適化**  
  `cronguard agent --pg-conn "postgres://user:pass@host:5432/cronguard?sslmode=require" --table-partition-interval 7`  
  各ジョブの実行履歴を PostgreSQL に書き込み、パーティションテーブルで7日ごとに自動分割。`cronguard query --job-id "backup_db" --since "2025-01-01" --until "2025-01-31" --latency-threshold 50` で、指定期間内に50ミリ秒以上の遅延が発生した全実行を抽出可能。インデックスは `(job_id, started_at DESC)` で最適化。

- **自己修復型ジョブリトライエンジン**  
  `cronguard repair --max-retries 5 --backoff-exponent 1.5 --jitter 200 --alert-webhook "https://hooks.slack.com/services/xxx"`  
  ジョブが異常終了した場合、指数バックオフ（初回1秒、2回目1.5秒…）と最大200ミリ秒のランダムジッターで最大5回リトライ。リトライのたびに `cronguard.job_retries` テーブルに記録し、全リトライ失敗時は Slack Webhook にアラートを送信。リトライ中はジョブの重複実行を防ぐため、PostgreSQL のアドバイザリロックを活用。

### 使い方

1. **初期セットアップとデータベースマイグレーション**  
   `cronguard init --pg-conn "postgres://user:pass@localhost:5432/cronguard" --migration-path ./migrations`  
   コマンドを実行すると、PostgreSQL 上に `cronguard` スキーマ、`job_executions` テーブル（パーティション未設定）、`job_retries` テーブル、および LISTEN/NOTIFY 用のトリガー関数が自動生成されます。マイグレーションファイルは `./migrations` ディレクトリに YAML 形式で保存され、バージョン管理が可能です。

2. **Cron 定義の登録と監視開始**  
   `cronguard register --cron-file ./cron_def.yaml --agent-id "agent-01" --interval 10`  
   `cron_def.yaml` に記述された全ジョブを cronguard の管理下に置きます。`--interval 10` は10秒ごとにジョブの実行状態をポーリングする設定。登録後、cronguard は各ジョブの次回実行時刻を計算し、`cronguard.next_runs` テーブルに保存。実行予定時刻の5秒前から PostgreSQL の NOTIFY でエージェントに通知します。

3. **異常ジョブの自動修復とレポート生成**  
   `cronguard auto-repair --policy "retry_then_alert" --report-path ./reports/daily_report_$(date +%Y%m%d).pdf`  
   過去24時間に3回以上失敗したジョブを自動検出し、`retry_then_alert` ポリシーに従ってリトライを実行。全リトライ失敗後、PDF レポート（実行サマリー、遅延分布ヒストグラム、失敗原因トップ5）を生成し、指定パスに保存。レポートには `cronguard.job_executions` から取得したミリ秒単位のレイテンシデータが含まれます。

### ベストプラクティス

- **ジョブの重要度に応じたリトライポリシーの階層化**  
  クリティカルジョブ（例：データベースバックアップ）には `--max-retries 10 --backoff-exponent 2.0 --alert-webhook` を設定し、非クリティカルジョブ（例：ログローテーション）には `--max-retries 3 --backoff-exponent 1.2` と軽めの設定を推奨。cronguard の YAML 定義で `priority: critical` タグを付与することで、ポリシーを自動適用できます。

- **PostgreSQL のパーティションテーブルと定期的なバキューム**  
  実行履歴テーブル `cronguard.job_executions` は、`created_at` で日付単位のパーティションに分割し、30日経過したパーティションは `cronguard archive --older-than 30` でアーカイブ。同時に `VACUUM ANALYZE` をスケジュールすることで、クエリパフォーマンスを維持しつつ、ディスク使用量を最小化します。
