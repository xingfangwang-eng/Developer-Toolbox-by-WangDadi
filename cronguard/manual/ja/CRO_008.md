# cronguard - CRO_008

## 概要

Cronguard は、分散 Cron ジョブの実行ログを統合監査し、障害検知・パフォーマンス劣化をリアルタイムで可視化するエンジンです。従来の `cron` 管理では見逃されがちな「実行スキュー」「ジョブ間依存の循環」「リソースリーク」を自動検出し、運用チームに即座にアラートを送ります。Cron ジョブの可観測性と信頼性を飛躍的に向上させる、SaaS ネイティブなソリューションです。

## ハードコアな 3 つの機能

### 1. 実行スキュー検出エンジン (Execution Skew Detector)
指定されたスケジュールに対して、実際のジョブ開始時刻のズレ（スキュー）をミリ秒単位でトラッキング。スキューが閾値（例: `500ms`）を超えた場合、自動で根本原因を特定し、ジョブの再スケジュールまたは代替インスタンスへのフェイルオーバーを提案します。これにより、深夜バッチの遅延連鎖を未然に防止します。

### 2. ジョブ依存グラフの動的循環検出 (Dynamic Cyclic Dependency Detection)
複数の Cron ジョブ間に定義された依存関係（例: `jobA → jobB → jobC → jobA`）をリアルタイムで解析。循環依存が発生した瞬間に、影響範囲を可視化し、該当ジョブの実行をブロックまたはキューイングします。これにより、無限ループによるリソース枯渇やデータ不整合を防ぎます。

### 3. リソースリークプロファイラー (Resource Leak Profiler)
各ジョブの実行コンテキスト（メモリ、ファイルディスクリプタ、データベース接続）を監視し、終了後にリークが残っていないか自動チェック。リークが検出された場合、該当ジョブのコード行レベルでのトレースを提供し、修正を促します。これにより、長時間稼働サーバーのメモリ使用量増大や接続プール枯渇を根本から解決します。

## 3 つの詳細な操作手順

### 手順 1: スキュー検出の閾値設定と通知ルール作成
`cronguard.yaml` に以下の設定を追加し、スキュー検出を有効にします。

```yaml
skew_detection:
  enabled: true
  threshold_ms: 500
  actions:
    - type: webhook
      url: "https://hooks.slack.com/services/T00/B00/xxxx"
    - type: retry
      max_retries: 3
      backoff: exponential
```

次に、Cronguard CLI で設定を反映します。

```bash
cronguard config apply --file cronguard.yaml
```

これで、全ジョブの開始時刻が 500ms 以上ズレた場合、Slack 通知と指数バックオフリトライが自動実行されます。

### 手順 2: 循環依存検出の有効化と可視化
ジョブ定義ファイル（例: `jobs.json`）に依存関係を記述します。

```json
{
  "jobA": { "depends_on": ["jobB"] },
  "jobB": { "depends_on": ["jobC"] },
  "jobC": { "depends_on": ["jobA"] }
}
```

Cronguard の Web UI で「依存グラフ」タブを開き、`Scan` ボタンをクリックします。循環依存が検出されると、グラフ上で該当ノードが赤色でハイライトされ、`jobC → jobA` のリンクに警告アイコンが表示されます。この状態で、`jobC` の `depends_on` を `jobD` に変更し、再度スキャンすることで循環が解消されたことを確認できます。

### 手順 3: リソースリークレポートの取得と修正
Cronguard Agent を各ジョブ実行環境にデプロイします。

```bash
cronguard agent install --token <your_token>
```

ジョブ実行後、以下のコマンドでリークレポートを取得します。

```bash
cronguard report leaks --job-id jobA --since 2025-03-01T00:00:00Z
```

出力例:
```
Leak detected in jobA (PID 12345):
- File descriptor leak: 5 unclosed handles (open_files: /tmp/tmpXXXX)
- Database connection leak: 2 unclosed connections (pool: main-db)
Suggested fix: Add `close()` calls in lines 42-45 of /opt/jobA/worker.py
```

このレポートに従い、該当コードを修正後、再実行してリークが解消されたことを確認します。

## 2 つの業界ベストプラクティス

### ベストプラクティス 1: スキュー検出を SLA 監視と統合する
Cronguard のスキュー検出結果を、既存の SLA ダッシュボード（Datadog、Prometheus など）にメトリクスとして送信することを推奨します。具体的には、`cronguard metrics export` コマンドで OpenMetrics 形式のエンドポイントを公開し、Prometheus がスクレイピングする設定を行います。これにより、Cron ジョブの遅延が SLA 違反に直結する前に、運用チームが予防措置を取れます。

### ベストプラクティス 2: 循環依存検出を CI/CD パイプラインに組み込む
ジョブ定義ファイルの変更が行われるたびに、Cronguard の循環依存チェックを CI パイプライン（GitHub Actions、GitLab CI など）で自動実行することを推奨します。例えば、`.github/workflows/cronguard-check.yml` で以下のジョブを追加します。

```yaml
jobs:
  dependency-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: cronguard check cycles --file jobs.json
```

循環依存が検出された場合、パイプラインを失敗させることで、本番環境へのデプロイ前に問題を発見できます。
