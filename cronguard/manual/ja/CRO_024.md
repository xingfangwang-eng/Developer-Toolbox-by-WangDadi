# cronguard - CRO_024

## 概要

cronguard は、SaaS インフラのクラッシュ耐性とジョブスケジュール異常をリアルタイムで検出・自動修復する次世代型プロフェッショナルツールです。本プロジェクトは、CRO_024 コードネームで開発され、特に cron ジョブのガード（監視＋保護）に特化。キーワード：cronguard tool、cronguard solution、cronguard software。

## 詳細な内容

### cronguard とは

cronguard は、従来の cron 監視ツールが抱える「障害検知の遅延」「手動復旧の依存」「ジョブ間依存関係のブラックボックス化」を根本的に解決する SaaS アーキテクチャです。Go 言語による軽量エージェントと、Kubernetes ネイティブなバックエンドが連携し、秒単位の異常検知と自動ロールバックを実現します。

### コア機能

1. **自動ジョブ依存グラフ解析と動的リカバリ**  
   各 cron ジョブの依存関係を DAG（有向非巡回グラフ）として自動構築。障害発生時、影響範囲を特定し、依存チェーン全体を考慮した最適な再実行またはスキップ戦略を動的に決定。手動インターベンション不要で、複雑なジョブチェーンでも安定稼働。

2. **マルチリージョン・フェイルオーバー型ジョブディスパッチャ**  
   AWS / GCP / Azure の複数リージョンにまたがるジョブ実行を一元管理。プライマリリージョン障害時、自動でセカンダリリージョンにジョブを再割り当て。レイテンシ変動に応じた動的スケジューリングアルゴリズム（DSA）を搭載し、SLA 99.99% を担保。

3. **コードレスな異常パターン学習と自己修復ルールエンジン**  
   過去のジョブ実行ログを機械学習（異常検知モデル：Isolation Forest + 時系列分析）で学習。未定義の異常パターンでも「類似障害履歴」から自動で修復アクション（再試行、リソーススケールアップ、代替ジョブ起動）を生成。ユーザーが明示的にルールを書く必要はありません。

### 使い方

1. **エージェントデプロイと初期設定**  
   `curl -sL https://get.cronguard.io/install.sh | bash` でエージェントを対象サーバーにインストール。その後、`cronguard init --config /etc/cronguard/config.yaml` で監視対象の cron ジョブリストを自動スキャン。エージェントは `systemd` ユニットとして常駐し、ジョブ実行のたびにメタデータ（開始時刻、終了コード、出力ハッシュ）を収集します。

2. **異常検知閾値のカスタマイズとポリシー設定**  
   ダッシュボード（`https://<your-instance>.cronguard.io/policies`）で、ジョブごとに「許容最大実行時間」「再試行回数」「フェイルオーバー先リージョン」を定義。例えば、`job_backup.sh` が 300 秒を超えた場合、自動で GCP us-central1 のインスタンスに切り替え、`--force` フラグ付きで再実行するポリシーを YAML で記述可能。設定は即座にエージェントにプッシュされます。

3. **障害発生時の自動リカバリとレポート確認**  
   障害がトリガーされると、cronguard は Slack / PagerDuty / Email にアラートを送信。同時に、`cronguard recover --job-id <ID>` 相当の自動プロセスが実行され、障害原因とリカバリ結果が JSON 形式でログ保存されます。`cronguard report --last 24h` で、全ジョブの実行ステータス、異常発生数、修復成功率を一覧表示。レポートは Markdown または PDF でエクスポート可能。

### ベストプラクティス

1. **ジョブ実行履歴を少なくとも 30 日間保持し、機械学習モデルの再学習サイクルを週次で実行する**  
   異常検知モデルは直近のデータに強く依存します。特に、季節性のあるジョブ（月末バッチ処理など）では、古いデータを削除せず保持することで、周期性パターンを正確に学習できます。`cronguard model retrain --retention 30d` で自動化可能。

2. **本番環境へのポリシー適用前に、ステージング環境で「カオスエンジニアリング」テストを実施する**  
   cronguard の強みは自動修復ですが、誤ったポリシーが逆に障害を拡大するリスクがあります。`cronguard chaos run --policy <policy-name> --duration 10m` で、意図的にジョブを失敗させ、修復プロセスが正しく動作するか検証。このテスト結果はダッシュボードに記録され、本番適用の判断材料として利用できます。
