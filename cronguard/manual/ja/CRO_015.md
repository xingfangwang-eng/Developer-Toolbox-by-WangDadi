# cronguard - CRO_015

## 概要

cronguard は、分散 Cron ジョブの実行履歴をリアルタイムにキャプチャし、スケジュールの不整合やリソース競合を検出する「Cron ジョブ監査ツール」です。従来のログ監視とは異なり、ジョブの依存関係グラフを自動生成し、クリティカルパス上の遅延を可視化します。本ドキュメントでは、**cronguard tool** の導入から運用最適化までを詳細に解説します。

## ハードコア機能（3選）

### 1. ジョブチェーン依存関係グラフの自動生成
cronguard は、`crontab` 内のコメントアウトされた依存タグ（例: `#@after: job_alpha`）を解析し、ジョブ同士の実行順序とブロッキング条件を DAG（有向非巡回グラフ）として可視化します。これにより、複数サーバーにまたがるジョブチェーンのボトルネックを特定可能です。

### 2. スケジュールドリフト検出エンジン
各ジョブの実際の実行開始時刻と `crontab` で定義されたスケジュールを比較し、**ドリフト率**（遅延 / 早期実行の頻度）を計算。閾値（デフォルト 5%）を超えたジョブを自動でアラートし、タイムゾーン設定ミスやシステムクロックのずれを特定します。

### 3. リソースコンフリクトヒートマップ
過去 30 日間のジョブ実行ログを基に、CPU・メモリ・ディスク I/O の使用率を時間軸でマッピング。同一時間帯にリソースを競合するジョブペアをスコアリングし、推奨スケジュール調整案を出力します。

## 詳細な操作手順（3ステップ）

### ステップ 1: エージェントデプロイと初期設定
```bash
# cronguard エージェントを全ターゲットサーバーにデプロイ
curl -fsSL https://cronguard.io/install.sh | bash -s -- --agent-key=YOUR_AGENT_KEY

# 設定ファイルを編集：監視対象の cron パスと除外パターンを指定
vim /etc/cronguard/agent.yaml
# 例：
# watch_paths:
#   - /var/spool/cron/crontabs
#   - /etc/cron.d
# exclude_patterns:
#   - "*.bak"
#   - "legacy_*"
```
**ポイント**: `agent-key` はマスターサーバーで発行したトークンを使用。初回起動時に自動で既存の crontab をスキャンし、ジョブリストを中央DBに登録します。

### ステップ 2: 依存関係タグの埋め込み
既存の crontab に依存タグを追加し、cronguard に実行順序を認識させます。
```bash
# 例：ジョブBはジョブAの完了後にのみ実行する
#@after: job_alpha
*/5 * * * * /usr/local/bin/job_beta.sh

# 例：ジョブCはジョブBの成功ステータスに依存
#@depends-on: job_beta:exit_code=0
0 */2 * * * /usr/local/bin/job_gamma.sh
```
**補足**: タグは `crontab` のコメント行に記述します。cronguard は `cron` デーモンとは独立して解析するため、既存のスケジューリングに影響を与えません。

### ステップ 3: ダッシュボードでドリフト分析とアラート設定
Web UI にアクセスし、`/dashboard/drift` を確認します。
```bash
# 例：ドリフト率が 10% を超えたジョブに Slack 通知を設定
cronguard alert create \
  --metric=drift_rate \
  --threshold=10 \
  --action=slack \
  --webhook-url=https://hooks.slack.com/services/YOUR_WEBHOOK
```
**詳細**: ドリフト率の計算式は `(実際の実行開始時刻 - スケジュール時刻) / スケジュール間隔 × 100`。閾値を超えたジョブは一覧でハイライト表示され、タイムライン上に実行遅延の分布が表示されます。

## 業界ベストプラクティス（2選）

### 1. ジョブチェーンに「タイムアウトガード」を組み込む
依存関係を持つジョブには、`#@timeout: 300` タグを追加し、指定秒数以上ジョブが完了しない場合に強制的に失敗ステータスを出力する設定を推奨します。これにより、チェーン全体が無限待ちになるデッドロックを防止できます。

### 2. 週次で「スケジュール最適化レポート」を自動生成
cronguard の `report weekly` コマンドを crontab に組み込み、毎週月曜にリソースコンフリクトヒートマップとドリフト率のサマリをメール送信する運用を推奨します。これにより、負荷ピークを事前に調整し、深夜バッチの失敗率を低減できます。
