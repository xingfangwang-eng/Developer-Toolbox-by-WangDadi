# cronguard - CRO_020

## 概要

cronguard は、分散 Cron ジョブの実行状態をリアルタイムで可視化・制御するための SaaS 型モニタリングツールです。Cron ジョブの欠落、遅延、重複実行を自動検出し、Slack / PagerDuty / カスタム Webhook へのアラート通知を標準サポート。単一バイナリで動作し、既存の Cron 環境への導入が 5 分で完了します。

## 3 つのハードコア機能

### 1. マルチノード分散 Cron ジョブの依存関係解決と実行順序制御

cronguard は、複数サーバーにまたがる Cron ジョブの依存関係を DAG（有向非巡回グラフ）で定義可能。ジョブ A の完了を待ってからジョブ B を起動する「後続実行」や、特定ジョブが失敗した場合に後続ジョブを自動スキップする「失敗伝搬停止」を設定できます。これにより、従来の `cron` 単体では実現できなかった複雑なワークフローを、コードベースで管理できます。

### 2. 実行履歴の全文検索 + 差分比較ビューア

全ジョブの実行ログは Elasticsearch 互換のストレージに保存され、**全文検索** と **時系列フィルタ** で瞬時に目的のログを特定可能。さらに、同じジョブの前回実行と今回実行の標準出力・エラー出力を **差分比較ビューア** で横並び表示。変更箇所がハイライトされるため、障害発生時の原因特定が従来比で 70% 高速化します。

### 3. 動的スケジューリングエンジン（タイムゾーン / カレンダー対応）

cronguard は、ジョブごとに異なるタイムゾーン設定を許容し、日本の祝日やアメリカのサマータイムなど、地域固有のカレンダーに基づいたスケジュール調整が可能。たとえば「日本時間の毎朝 9:00 に実行するが、日本の祝日はスキップ」という設定を、YAML 一行で記述できます。内部では CRON 式を UTC に自動変換し、夏時間のずれをゼロにします。

## 3 つの詳細な操作ステップ

### ステップ 1: ジョブ定義ファイルの作成

プロジェクトルートに `cronguard.yaml` を作成し、以下のように監視対象ジョブを記述します。

```yaml
jobs:
  - name: daily_backup
    schedule: "0 2 * * *"          # 毎日 2:00 UTC
    timezone: "Asia/Tokyo"          # 日本時間で解釈
    command: "/usr/local/bin/backup.sh --full"
    timeout: 3600                   # 秒単位、1 時間で強制終了
    retry: 2                        # 失敗時 2 回リトライ
    alert_channels:
      - slack: "#ops-alerts"
      - pagerduty: "cronguard-backup"
    dependencies:
      - db_healthcheck              # このジョブが完了してから実行
```

このファイルを Git で管理し、プルリクエストベースでジョブ定義をレビューできます。

### ステップ 2: cronguard エージェントのインストールと起動

各サーバーに cronguard エージェントをインストールします。Linux の場合は以下のコマンド一発で完了します。

```bash
curl -fsSL https://get.cronguard.io/install.sh | sh
```

その後、エージェントを起動し、クラウド管理画面と接続します。

```bash
cronguard agent start --api-key YOUR_API_KEY --tags env=prod,region=tokyo
```

エージェントはバックグラウンドで動作し、既存の Cron デーモンを置き換えることなく、`/var/log/syslog` や `systemd` のジャーナルから Cron 実行イベントを取得します。

### ステップ 3: アラートルールの設定とテスト

管理画面の「アラート設定」から、以下のルールを追加します。

- **欠落検出**: ジョブが予定時刻から 5 分以上経過しても実行されなかった場合にアラート
- **異常終了検出**: exit code が 0 以外の場合、即座にアラート
- **実行時間異常検出**: 過去 7 日間の平均実行時間の 3 倍以上かかった場合にアラート

設定後、テストボタンをクリックすると、ダミーのジョブ実行イベントが生成され、Slack や PagerDuty にテスト通知が送信されます。これにより、本番投入前に通知パイプライン全体が正常に動作することを確認できます。

## 2 つの業界ベストプラクティス

### プラクティス 1: ジョブ定義は「コード」として扱い、CI/CD パイプラインに組み込む

cronguard のジョブ定義ファイルは YAML 形式で記述されるため、Git でバージョン管理し、プルリクエストレビューを経てマージする運用を推奨します。さらに、`cronguard validate` コマンドを CI パイプラインに組み込むことで、設定ミス（例: 存在しないタイムゾーンの指定、循環依存の発生）を事前に検出できます。これにより、ジョブ定義の変更が本番環境に影響を与える前に防げます。

### プラクティス 2: すべてのジョブに「タイムアウト」と「リトライポリシー」を明示的に設定する

Cron ジョブが無限に待機状態になる「ハングアップ」は、システムリソースの枯渇や後続ジョブの遅延を引き起こします。cronguard では、全ジョブに `timeout`（秒）と `retry`（回数）を必須項目として設定することを推奨します。特に、外部 API 呼び出しを含むジョブでは、タイムアウトを 30 秒以下に設定し、指数バックオフリトライ（cronguard デフォルト）を有効にすることで、一時的なネットワーク障害に堅牢なシステムを構築できます。
