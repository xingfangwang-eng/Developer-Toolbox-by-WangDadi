# cronguard - CRO_010

## 概要

cronguardは、分散環境におけるCronジョブの実行漏れ・遅延・重複実行をリアルタイムに検知し、アラートと自動リカバリを提供するSaaS型監視エンジンです。従来の死活監視では捉えきれなかった「ジョブが実行されたが結果が期待通りでない」ケースまでカバーし、DevOpsチームの運用負荷を大幅に削減します。

## ハードコア機能

### 1. 実行パターン異常検知エンジン
従来の「実行したか／していないか」の二値監視ではなく、ジョブの実行時間、終了コード、出力ログのサイズ・内容、実行間隔のゆらぎを機械学習ベースで学習。過去7日間のパターンから逸脱した挙動（例：通常5秒で終わるジョブが突如30秒かかっている、終了コードは0だが出力が空）を自動検出し、**擬陽性を最小化**しながらアラートを発報します。

### 2. マルチクラスタ・マルチタイムゾーン対応の分散チェックポイント機構
複数のKubernetesクラスタやオンプレサーバに分散したCronジョブに対し、cronguard Agentが各ノードでチェックポイントを生成。中央のSaaSバックエンドはこれらのチェックポイントを**ベクトルクロック方式で統合**し、タイムゾーンやネットワーク遅延に依存しない正確なジョブ実行順序と依存関係を可視化します。

### 3. セルフヒーリングワークフロー
ジョブ失敗を検知した際、事前定義されたリカバリポリシーに基づき、自動で再実行、依存ジョブのキャンセル、またはフォールバックジョブへの切り替えを実施。リカバリの成否は**トランザクションID**で追跡可能で、すべてのアクションは監査ログに記録されます。

## 詳細なセットアップ手順

### 手順1: 分散エージェントのデプロイと登録
各監視対象サーバにcronguard Agentをインストールします。以下のコマンドでエージェントを起動し、APIキーを指定してSaaSバックエンドに登録します。

```bash
curl -fsSL https://cronguard.io/install.sh | sh
cronguard-agent register --api-key YOUR_API_KEY --cluster-name production-tokyo
```

登録後、エージェントは自動的に `/etc/crontab` および `kubectl get cronjobs --all-namespaces` をスキャンし、既存のジョブ定義をインポートします。

### 手順2: ジョブ監視ルールの定義（YAMLベース）
監視対象ジョブごとに、期待される実行間隔、タイムアウト、リトライポリシーをYAMLで定義します。以下は典型的な設定例です。

```yaml
# cronguard-rules.yaml
jobs:
  - name: "daily-report-generator"
    schedule: "0 3 * * *"
    timeout_seconds: 300
    expected_duration_seconds: 45
    retry_policy:
      max_retries: 2
      backoff_seconds: 60
    anomaly_detection:
      enabled: true
      sensitivity: "medium"   # low / medium / high
    alert_channels:
      - slack
      - pagerduty
```

このファイルをcronguard CLIで適用します。

```bash
cronguard apply -f cronguard-rules.yaml
```

### 手順3: アラートと自動リカバリのテスト
実際にジョブを強制的に失敗させ、アラートが正しく発報され、リカバリが動作するか確認します。

```bash
# テスト用にジョブの実行をブロック
cronguard test-failure --job-name daily-report-generator --failure-type timeout

# アラートがSlackに届くことを確認
# リカバリが自動実行されるのをログで確認
cronguard logs --job-name daily-report-generator --tail 50
```

## ベストプラクティス

### プラクティス1: ジョブの「正常終了」を終了コードだけで判断しない
シェルスクリプトやコンテナジョブでは、エラーが発生しても `exit 0` で終了するケースが多発します。cronguardでは、**出力ログに特定のエラーパターンが含まれていないか**、**標準出力と標準エラーの比率**を監視条件に追加することを推奨します。具体的には、ルール定義に `output_match: "ERROR|FATAL|Exception"` を追加し、マッチした場合に警告を出す設定を行います。

### プラクティス2: リカバリポリシーは冪等性を前提に設計する
自動リカバリによる再実行は、ジョブが**冪等**であることを前提に設計します。特にデータベースへの書き込みや外部APIへのPOSTを行うジョブでは、再実行によって重複データや二重課金が発生しないよう、ジョブ内で一意なリクエストID（idempotency key）を発行し、cronguardのトランザクションIDと紐付けることを強く推奨します。
