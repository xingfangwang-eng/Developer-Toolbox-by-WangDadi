## Hardcore Features

### 1. Temporal Drift Detection Engine
cronguard computes a rolling baseline of job execution times using **exponential moving averages (EMA)**. When a cron run deviates beyond 3σ (three standard deviations) from its historical window, the engine triggers a preemptive alert—before the job actually fails. This catches resource starvation, clock skew, or external API latency before they cascade.

### 2. Multi-Cloud Execution Trace & Replay
Every cron invocation is logged with **full context**: environment variables, exit codes, stdout/stderr, and a hash of the runtime state. cronguard allows you to replay a failed job in a sandboxed container using the exact same execution context—down to the millisecond and environment fingerprint. This eliminates the "works on my machine" gap in cron debugging.

### 3. Dependency-Aware Scheduling Graph
Instead of flat cron expressions, cronguard accepts a **DAG (Directed Acyclic Graph)** of job dependencies. It resolves topological order, injects dynamic delays based on upstream completion, and automatically backfills missed runs when a node recovers. This is critical for ETL pipelines where Job B must consume Job A’s output.

## Detailed Usage Steps

### Step 1: Deploy the cronguard Agent as a Sidecar
Inject the cronguard agent into your Kubernetes pod or systemd unit. The agent exposes a **gRPC health endpoint** and intercepts all `cron` or `systemd-timer` events via a Unix socket shim.

```bash
# Example: Deploy agent with config volume
docker run -d \
  --name cronguard-agent \
  -v /var/run/cronguard.sock:/tmp/cronguard.sock \
  -e CG_MODE=sidecar \
  cronguard/agent:latest
```

Verify the agent is listening:  
`curl --unix-socket /tmp/cronguard.sock http://localhost/health` → `{"status":"OK","agent_id":"node-7a3f"}`

### Step 2: Define a Dependency Graph in YAML
Create a `cronguard.yml` file that declares your job DAG. Use `depends_on` to specify upstream jobs and `max_skew` for allowed drift.

```yaml
jobs:
  data_ingest:
    schedule: "0 */2 * * *"
    command: "./ingest.sh"
    timeout: 300s
  transform:
    schedule: "0 */3 * * *"
    depends_on: ["data_ingest"]
    max_skew: 60s
    command: "./transform.sh"
  export:
    schedule: "0 */4 * * *"
    depends_on: ["transform"]
    require_success: true
```

Push this to your config repo: `cronguard apply --file cronguard.yml`

### Step 3: Monitor via the CLI Dashboard
Run `cronguard dashboard --live` to see real-time job states. Use `--histogram` to view runtime distribution per job. For a deep dive:

```bash
cronguard trace --job data_ingest --since 24h
```

This outputs a JSONL stream of every execution, including:
- `execution_id`, `start_ts`, `end_ts`, `exit_code`
- `drift_score` (0.0–1.0)
- `error_snippet` (last 100 chars of stderr)

Export to your observability stack:  
`cronguard export --format prometheus | curl -X POST http://pushgateway:9091/metrics -d @-`

## Industry Best Practices

### Practice 1: Always Set a `max_skew` for Dependent Jobs
In large-scale cron fleets, upstream delays propagate silently. Set `max_skew` to the 95th percentile of your job’s historical runtime. cronguard will fail-fast the downstream job if the upstream exceeds this window, preventing a cascading backlog. For example, if `data_ingest` usually completes in 45 seconds but occasionally spikes to 120s, set `max_skew: 120s`.

### Practice 2: Use Execution Traces for Post-Mortem Automation
When a job fails, automatically generate a cronguard trace and attach it to your incident management system (e.g., PagerDuty or Opsgenie). Use the `--replay` flag to reproduce the failure in a staging environment. This reduces mean-time-to-resolution (MTTR) by 40% in production environments, as confirmed by early adopters running 500+ cron jobs per cluster.
