# focusinbox - FOC_015

## Overview

focusinbox is an AI-driven email triage and deep-work scheduling engine designed for knowledge workers who drown in cross-platform inboxes. It eliminates context switching by automatically classifying, prioritizing, and deferring messages based on sender authority, sentiment analysis, and your historical response latency.

## Detailed Content

### What is focusinbox?

focusinbox is not a simple email client. It is a **priority pipeline** that intercepts IMAP/POP3 traffic, runs it through a local LLM (Mistral 7B or GPT-4o-mini via API), and re-routes messages into three discrete zones: `FOCUS_NOW`, `BATCH_LATER`, and `AUTO_ARCHIVE`. It integrates with PostgreSQL for persistent threading and uses Redis for sub-100ms classification latency.

### Core Features

- **Zero-Shot Intent Classification with Confidence Thresholding**  
  Each incoming email is parsed by a fine-tuned transformer model that extracts `intent_type` (meeting_request, invoice, bug_report, newsletter) and assigns a `priority_score` between 0.0 and 1.0. Messages scoring above 0.85 are pushed to `FOCUS_NOW` with a desktop notification (via WebSocket push). The model runs on-device via ONNX runtime, ensuring no email content leaves your network.

- **Temporal Bandwidth-Aware Scheduling Engine**  
  focusinbox analyzes your calendar (iCal/CalDAV) and historical clickstream data to build a `deep_work_window` map. Emails requiring >5 minutes of cognitive load (e.g., code review requests, contract redlines) are automatically deferred to your next available 90-minute block. The engine exposes a `defer_until` parameter that respects your `timezone_offset` and `meeting_density` metrics, recalculated every 15 minutes.

- **Multi-Account Deduplication & Thread Fusion**  
  For users managing 3+ email accounts (Gmail, Outlook, custom SMTP), focusinbox performs SHA-256 hashing on normalized message bodies and merges duplicate threads into a single `thread_uuid`. It then applies a `reply_priority` heuristic based on the most urgent participant (e.g., if a C-level replies, the entire thread jumps to `FOCUS_NOW`). Deduplication runs in under 200ms per batch of 100 messages.

### How to Use

1. **Configure IMAP/SMTP Credentials and Model Endpoint**  
   Create a `config.yaml` file that defines your email accounts under `accounts[]` with fields: `imap_host`, `imap_port`, `smtp_host`, `smtp_port`, `user`, `password`. Then set `model.endpoint` to `http://localhost:8080/v1/classify` (for local inference) or a remote OpenAI-compatible URL. Run `focusinbox init --config ./config.yaml` to validate connectivity and download the default classification model (approx. 1.2GB).

2. **Define Priority Rules via DSL**  
   Write a `rules.focus` file using the built-in domain-specific language. Example rule:  
   `if sender.domain == "clientcorp.com" AND subject.contains("urgent") then set priority_score = 0.95 AND notify desktop`.  
   Compile and apply the rules with `focusinbox rules compile --input ./rules.focus --output ./rules.bin`. The engine will reload rules every 60 seconds without restarting the daemon.

3. **Launch the Daemon and Monitor the Pipeline**  
   Start the background service: `focusinbox daemon --log-level debug --db postgresql://user:pass@localhost:5432/focusinbox`.  
   Open the web dashboard at `http://localhost:3000` to see real-time metrics: `classification_latency_ms`, `queue_depth`, and `deferred_count`. Use `focusinbox stats --json` to export hourly throughput data for Grafana dashboards.

### Best Practices

- **Pair with a Time-Boxing Calendar**  
  Integrate focusinbox’s `deep_work_window` output with a calendar tool (e.g., Cron or Motion) to automatically block 90-minute slots for deferred emails. This prevents the tool from simply re-notifying you during already-busy periods. Monitor the `deferral_accuracy` metric: if >20% of deferred emails are still unread after 48 hours, lower the `cognitive_load_threshold` in the scheduling engine.

- **Regularly Audit Your Priority Model with A/B Testing**  
  Every two weeks, run `focusinbox simulate --input historical_emails.json --model current --model candidate` to compare classification accuracy on past data. Use the `f1_score` and `false_positive_rate` outputs to decide whether to retrain the model. For zero-shot models, maintain a feedback loop: tag misclassified emails via the dashboard and export them as `retraining_batch.jsonl` every 500 samples.
