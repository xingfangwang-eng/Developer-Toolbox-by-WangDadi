# focusinbox - FOC_028

## Overview

focusinbox is a zero-latency email triage and priority engine designed for knowledge workers drowning in inbox overload. It leverages a hybrid rule-based + ML scoring model to surface only the messages that require human cognition, while auto-archiving or deferring the rest. The system operates on a sub-100ms classification cycle per email, directly integrating with IMAP and Microsoft Graph API.

## Core Features

- **Adaptive Priority Scoring Engine (APSE)**  
  Each inbound email is evaluated against a multi-dimensional tensor: sender reputation score (0.0–1.0), thread velocity (messages/hour), semantic urgency detected via NLP (e.g., “ASAP”, “blocked”, “deadline”), and your personal calendar density (from Outlook/Google Calendar API). The final priority score is computed as `P = α·sender_score + β·urgency_score + γ·calendar_conflict`, with α, β, γ tunable per user. Emails below a configurable threshold (default: 0.35) are automatically moved to a "Digest" folder and summarized every 4 hours.

- **Temporal Attention Windows**  
  You define "focus sessions" (e.g., 09:00–11:00, 14:00–16:00) during which only emails with priority score > 0.7 are shown in the main view. All other messages are silently deferred to the next available slot. The system tracks your historical response latency per sender and dynamically adjusts the threshold: if you typically reply to a colleague within 2 minutes, their email will bypass the window. This is stored in a local SQLite database with a 50ms read latency guarantee.

- **Cross‑Account Thread Unification**  
  If you manage multiple inboxes (Gmail, Outlook, custom IMAP), focusinbox deduplicates and merges threads by Message-ID and In-Reply-To headers. It then presents a single timeline per conversation, annotated with the source account and a "last seen" timestamp. The merge engine runs as a background worker, processing up to 1,200 emails per minute per account, and stores the unified state in PostgreSQL with a JSONB column for thread metadata.

## How to Use

1. **Configure IMAP/Graph API credentials**  
   Create a `config.yaml` in the project root:
   ```yaml
   accounts:
     - provider: gmail
       imap_server: imap.gmail.com
       username: user@gmail.com
       app_password: "xxxx xxxx xxxx xxxx"
       sync_interval_sec: 30
     - provider: outlook
       graph_tenant_id: "your-tenant-id"
       graph_client_id: "your-client-id"
       graph_client_secret: "your-client-secret"
   priority_threshold: 0.35
   focus_windows:
     - start: "09:00"
       end: "11:00"
     - start: "14:00"
       end: "16:00"
   ```
   Run `focusinbox init` to validate connectivity and perform a one‑time full sync. The system will report per‑account latency in milliseconds.

2. **Tune your priority model**  
   After 7 days of usage, run `focusinbox analyze --profile` to generate a JSON report of your actual response patterns. Use the output to adjust α, β, γ in `config.yaml`:
   ```json
   {
     "alpha_current": 0.5,
     "beta_current": 0.3,
     "gamma_current": 0.2,
     "suggested_alpha": 0.6,
     "suggested_beta": 0.25,
     "suggested_gamma": 0.15,
     "reason": "Your response to urgent emails is 40% faster than to high‑sender‑score emails."
   }
   ```
   Apply the suggestion with `focusinbox apply --profile report.json`.

3. **Activate a focus session**  
   Execute `focusinbox focus --start` to immediately enter the next configured window. The CLI will lock the inbox view to only high‑priority threads and display a live counter: `[Focus active] 3 urgent emails • 12 deferred • next digest in 2h14m`. To exit early, run `focusinbox focus --stop`. All deferred emails will be re‑evaluated within 10 seconds.

## Best Practices

- **Set a low priority threshold during onboarding**  
  Start with `priority_threshold: 0.25` for the first two weeks. This allows you to observe the system’s classification accuracy without missing legitimate emails. Gradually raise it to 0.35–0.45 as you verify the model’s precision via the `focusinbox analyze --confusion-matrix` command, which outputs a 2×2 table of true/false positives/negatives.

- **Combine focus windows with calendar blocking**  
  Use the `focus_windows` to mirror your existing calendar events (e.g., “Deep Work” blocks). Since focusinbox reads your calendar via OAuth, it can automatically suggest windows: `focusinbox suggest-windows --from-calendar`. This synchronizes your email triage with your time‑management system, reducing context‑switching overhead by an estimated 23% based on internal telemetry.
