# focusinbox - FOC_011

## Overview

This is a production-grade email management engine designed for professionals handling 500+ daily inbound messages. Built on a zero-inbox philosophy, focusinbox eliminates noise through real-time priority scoring, automated triage, and cross-platform synchronization.

## Detailed Content

### What is focusinbox?

focusinbox is a server-side email orchestration layer that integrates with IMAP, Microsoft Graph, and Gmail API. It ingests raw email streams, applies a multi-dimensional priority matrix (sender reputation, thread velocity, keyword entropy, attachment weight), and surfaces only the top 7% of actionable messages per session. Designed for latency-critical workflows, it processes each email in under 180 milliseconds and stores decision logs in PostgreSQL for auditability.

### Core Features

- **Adaptive Priority Scoring Engine**  
  Each incoming email is evaluated against a 12-parameter model: `sender_reputation_index` (0–1, trained from historical reply rates), `thread_urgency_score` (calculated via exponential decay of reply latency), `keyword_entropy` (TF-IDF against user-defined watchlists), and `attachment_mime_weight` (PDF/DOCX > HTML > plain text). The composite score is normalized to a 0–100 scale and updated in real-time as new replies arrive, with a full re-evaluation every 2.3 seconds per active thread.

- **Automated Triage & Folder Distribution**  
  Based on the priority score, focusinbox automatically routes emails into one of three folders: `Critical` (score ≥ 78), `Review` (score 45–77), or `Archive` (score < 45). Routing rules are configurable via a YAML-based policy file with support for regex-based sender whitelisting, time-window-based suppression (e.g., ignore marketing emails between 22:00–06:00 UTC), and attachment size thresholds (default: >5MB flagged). The entire triage operation completes in under 50ms per email, with a PostgreSQL transaction log for rollback.

- **Cross-Platform Sync with Conflict Resolution**  
  focusinbox maintains a bidirectional sync with up to 3 email providers simultaneously. It uses a vector clock-based conflict detection algorithm to handle concurrent modifications (e.g., marking an email as read on Gmail while it's being archived on Outlook). The sync runs every 60 seconds by default, with a configurable `sync_interval_ms` parameter. Conflict resolution follows a last-writer-wins strategy with a 15-second grace period, ensuring zero data loss.

### How to Use

1. **Initialize the Priority Model**  
   Run `focusinbox init --config ./focusinbox.yaml` to bootstrap the engine. This command scans your last 90 days of email history, extracts sender patterns, reply latency distributions, and keyword frequencies, and generates a baseline `priority_model.json`. The initialization takes approximately 4.7 seconds per 1,000 emails on a standard NVMe SSD. After completion, verify the model with `focusinbox validate --model ./priority_model.json` to ensure a minimum F1-score of 0.85 on the training set.

2. **Configure Triage Policies**  
   Edit the `focusinbox.yaml` file to define your routing rules. For example, to auto-archive all emails from `newsletter@*` with attachments smaller than 1MB, add:  
   ```yaml
   rules:
     - name: "auto-archive-newsletters"
       condition: "sender matches '^newsletter@' AND attachment_size < 1048576"
       action: "archive"
   ```  
   Then reload the configuration without downtime using `focusinbox reload --policy ./focusinbox.yaml`. The engine will apply the new rules to the next 500 incoming emails and log any rule violations in `focusinbox_policy_errors.log`.

3. **Monitor Sync Health**  
   Use `focusinbox status --sync` to view real-time sync metrics: `last_sync_timestamp`, `pending_conflicts_count`, `average_sync_latency_ms`, and `provider_health` (OK/DEGRADED/DOWN). If conflicts exceed 10 in a 5-minute window, run `focusinbox resolve --conflict --strategy last-writer-wins --force` to force a resolution. For persistent issues, inspect the sync log with `focusinbox logs --sync --level warn` to identify provider-specific API rate limits or token expiry.

### Best Practices

- **Practice 1: Implement a Weekly Model Retraining Schedule**  
  Email patterns drift over time—sender reputations change, new keywords emerge, and thread urgency thresholds shift. Schedule a cron job to run `focusinbox retrain --model ./priority_model.json --data ./email_history_7d.json` every Sunday at 02:00 UTC. This retrains the priority engine on the latest 7 days of data, ensuring the F1-score remains above 0.90. Always validate the new model against the previous week's data before deployment, and keep the last 3 model snapshots in `/var/focusinbox/models/` for rollback.

- **Practice 2: Use a Dedicated PostgreSQL Instance for Audit Trails**  
  focusinbox generates approximately 2.3 GB of log data per 10,000 emails processed, including triage decisions, sync conflicts, and model predictions. To avoid performance degradation, configure a separate PostgreSQL database (e.g., `focusinbox_audit`) with a 30-day retention policy. Set `log_retention_days: 30` in the YAML config and enable automatic vacuuming every 6 hours. This separation ensures the main `focusinbox` database (used for real-time scoring) remains under 1ms query latency.
