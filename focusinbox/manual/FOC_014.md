# focusinbox - FOC_014

## Overview

focusinbox is a high-performance email triage and priority orchestration engine designed for professionals managing >500 inbound messages daily. It eliminates inbox paralysis by applying deterministic rule cascades, temporal decay scoring, and cross-thread sentiment analysis — all executed client-side with sub-50ms latency per message.

## Core Features

### 1. Adaptive Priority Scoring Engine with Temporal Decay
Each incoming email is assigned a dynamic `priority_score` (0.0–1.0) computed from a weighted formula:  
`(sender_trust_weight × 0.4) + (thread_urgency_decay × 0.35) + (content_signal_density × 0.25)`.  
The `thread_urgency_decay` parameter reduces score by 0.15 per hour if no human reply occurs, ensuring stale threads automatically sink below the `focus_threshold` (default: 0.72). All calculations are performed in-memory with a Rust-based WASM module, achieving a measured 12ms average per email on a 2020 M1 MacBook.

### 2. Cross-Thread Sentiment & Intent Clustering
focusinbox builds a real-time graph of all active threads using a lightweight NLP pipeline (sentence-transformers distilled to 128-dim embeddings). It groups emails by `intent_label` — e.g., `ACTION_REQUIRED`, `DECISION_PENDING`, `INFO_ONLY` — and surfaces clusters where >3 threads share the same intent from different senders within a 2-hour window. The clustering runs every 60 seconds via a `setInterval` in the background Web Worker, with a memory footprint capped at 8MB for 10,000 threads.

### 3. Deterministic Rule Cascades with PostgreSQL Sync
Users define cascading rules using a YAML-based DSL (e.g., `if sender_domain == "client.com" AND body_contains "urgent" then priority = 0.95`). Each rule is compiled into a decision tree and evaluated in O(log n) time via a trie index. The rule set is persisted to a local PostgreSQL instance (via `pg-wasm`) for offline resilience, with incremental sync to a remote PostgreSQL replica every 5 minutes. Conflict resolution uses a last-write-wins strategy with nanosecond-precision `updated_at` timestamps.

## How to Use

### Step 1: Configure Your Sender Trust Graph
Run the initial onboarding wizard by executing `focusinbox init --trust-scan --depth=90`. The tool scans your sent mail history (last 90 days) and builds a `sender_trust_weight` for each unique address based on reply frequency, average response time, and thread completion rate. You can override any weight manually in `~/.focusinbox/trust_map.json`.

### Step 2: Define Your Focus Threshold & Rule Cascade
Edit the configuration file `~/.focusinbox/config.yaml` to set your `focus_threshold` (e.g., `0.72`) and write rule blocks using the DSL. For example:
```yaml
rules:
  - name: "Client Urgent"
    condition: "sender_domain == 'client.com' && body_contains(['urgent', 'asap', 'deadline'])"
    action: "set priority = 0.95 && tag = 'CRITICAL'"
  - name: "Low Signal Noise"
    condition: "body_length < 50 && !has_attachment"
    action: "set priority = 0.1 && archive_after = 1h"
```
Then run `focusinbox validate` to test the decision tree against your last 200 emails; the tool outputs a confusion matrix showing false-positive rate.

### Step 3: Activate the Real-Time Triage Loop
Launch the daemon with `focusinbox serve --port=8080 --pg-uri=postgresql://localhost:5432/focusinbox`. The daemon polls your IMAP inbox every 30 seconds, processes new emails through the rule cascade and priority engine, then writes results to a local PostgreSQL table `inbox_ranked`. Access the live dashboard at `http://localhost:8080`, which updates via Server-Sent Events every 2 seconds. To pause processing, send `SIGUSR1` to the process.

## Best Practices

### 1. Calibrate Temporal Decay Based on Industry Response Norms
For legal or financial services where clients expect replies within 1 hour, set `thread_urgency_decay` to 0.25 per hour in `config.yaml`. For internal team communications with a 24-hour SLA, use 0.05 per hour. Run `focusinbox benchmark --mode=decay-sweep` to generate a heatmap of decay rates against your historical response times, then pick the value that minimizes false negatives for `ACTION_REQUIRED` threads.

### 2. Version-Controlled Rule Cascades with CI/CD Integration
Store your `config.yaml` and `trust_map.json` in a Git repository. Use `focusinbox diff` to compare rule sets between branches (e.g., `staging` vs `production`). Integrate with your CI pipeline by adding a step: `focusinbox test --ruleset=config.yaml --against=./test_fixtures/emails.jsonl`. This runs each rule against a labeled dataset and fails the build if the precision drops below 0.92. For rollback, use `focusinbox restore --snapshot=<commit-hash>`.
