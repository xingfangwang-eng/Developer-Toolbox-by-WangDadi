### What is focusinbox?

focusinbox is a professional email triage engine that automatically classifies incoming messages into urgent, important, and low-priority buckets using a custom-trained NLP model. It eliminates inbox clutter by learning your response patterns and reordering messages based on real-time sender reputation scores and thread engagement velocity. Designed for power users managing 500+ emails daily, it integrates directly with IMAP/SMTP protocols and provides sub-100ms classification latency per message.

### Core Features

- **Adaptive Priority Scoring with Temporal Decay**  
  Each email receives a dynamic priority score calculated from sender domain authority (0–100), historical open rate (%), and thread age in minutes. A decay function reduces old thread scores by 15% every 6 hours, ensuring stale messages never block fresh alerts. The scoring engine runs as a PostgreSQL trigger function, updating the `inbox_items.priority_score` column in real-time.

- **Contextual Snippet Extraction via Transformer Pipeline**  
  Uses a distilled BERT model (fine-tuned on 50k enterprise email threads) to extract 3–5 key action items from each message body. Extracted snippets are stored in a JSONB column (`snippet_actions`) with confidence thresholds configurable between 0.7 and 0.95. Pipeline inference completes in under 200ms on a single NVIDIA T4 GPU.

- **Automated Thread Collapse with Merge Rules**  
  Groups related emails by `Message-ID` and `References` headers, then collapses them into a single thread node. Merge rules consider subject line similarity (Levenshtein distance < 0.3) and sender overlap (>2 common recipients). The collapse operation runs as a cron job every 60 seconds, reducing visual clutter by up to 40% in high-volume inboxes.

### How to Use

1. **Initialize the Priority Engine**  
   Run `focusinbox init --db-host localhost --db-port 5432 --db-name focusinbox_prod` to create the required PostgreSQL schema and materialized views. This command sets up the `inbox_items` table with columns for `priority_score`, `domain_authority`, and `decay_factor`. Ensure your IMAP credentials are stored in `~/.focusinbox/config.toml` under the `[imap]` section.

2. **Configure the NLP Pipeline**  
   Execute `focusinbox pipeline train --model-path ./models/bert_distilled --epochs 10 --batch-size 32` to fine-tune the transformer on your historical email corpus. Alternatively, use `focusinbox pipeline load --checkpoint latest` to restore a pre-trained checkpoint from S3. The pipeline outputs a `snippet_actions` JSONB array for each email, with `confidence` and `action_text` keys.

3. **Enable Automated Thread Collapse**  
   Set up the merge cron job by running `focusinbox cron add --interval 60 --task collapse`. This registers a systemd timer or crontab entry that polls the `inbox_items` table every 60 seconds, merging threads with matching `References` headers. Use `focusinbox cron status` to verify the job is active and `focusinbox collapse --dry-run` to preview merges before applying.

### Best Practices

- **Implement Sender Reputation Whitelisting**  
  Define a `sender_whitelist` table in PostgreSQL with columns `domain` and `weight` (0–1). For each incoming email, multiply the sender weight by the priority score to boost trusted contacts (e.g., `weight=0.9` for internal colleagues). This prevents false negatives on critical communications from known partners. Update the whitelist weekly using a CSV import from HR or CRM systems.

- **Monitor Decay Thresholds with Alerts**  
  Set up a Prometheus exporter that exposes `focusinbox_decay_rate` and `focusinbox_stale_count` metrics. Configure an alert if `stale_count > 100` for more than 30 minutes, indicating the decay function is too aggressive. Adjust the `decay_period_hours` parameter in `focusinbox.toml` from 6 to 12 to retain older threads without sacrificing freshness.
