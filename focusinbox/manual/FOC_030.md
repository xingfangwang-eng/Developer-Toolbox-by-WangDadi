# focusinbox - FOC_030

## Overview

focusinbox is a zero‑latency email triage engine designed for high‑volume inboxes. It eliminates context switching by automatically classifying, ranking, and surfacing only the messages that require immediate action — with a guaranteed sub‑50ms classification latency per email.

## Core Features

### 1. Adaptive Priority Scoring with Temporal Decay

Each incoming email is assigned a dynamic priority score (`P_score`) computed in real‑time using a weighted combination of sender reputation, thread depth, keyword urgency, and a custom exponential decay factor (`λ = 0.15`). The algorithm re‑evaluates scores every 120 seconds against a PostgreSQL‑backed history table, ensuring that stale threads are automatically demoted without manual intervention.

### 2. Multi‑Channel Actionable Digest (MCAD)

Instead of a flat inbox, focusinbox generates a structured digest grouped by action type: `requires_reply`, `needs_approval`, `waiting_on_you`, and `info_only`. Each group is limited to a configurable cap (`max_items: 12`). The digest is refreshed on a 90‑second cycle using a Redis‑backed job queue, and supports push delivery to Slack, Microsoft Teams, or a custom webhook endpoint.

### 3. Semantic Thread Collapsing with Conflict Detection

All messages within a thread are collapsed into a single entry, but the system also performs cross‑thread semantic analysis using a lightweight TF‑IDF vectorizer. If two or more collapsed threads reference the same project ID, customer name, or deadline timestamp, they are flagged as `conflicting_threads` in the metadata. The conflict indicator is exposed via the API as a boolean field (`has_conflict`) and triggers a visual warning in the UI.

## How to Use

### Step 1: Configure Inbound Connector and Scoring Rules

Connect your email provider via IMAP or Microsoft Graph API. Then define your scoring rules in the `focusinbox.toml` configuration file:

```toml
[scoring]
sender_whitelist = ["ceo@company.com", "legal@company.com"]
keyword_boost = { "urgent" = 2.5, "security" = 3.0, "p0" = 4.0 }
decay_rate = 0.15
```

Run `focusinbox validate` to test the rules against the last 500 emails in your inbox.

### Step 2: Deploy the Digest Scheduler

Start the background worker with:

```bash
focusinbox worker start --digest-interval 90 --max-digest-items 12
```

This spawns a long‑running process that polls your inbox every 90 seconds, re‑calculates scores, and pushes the MCAD digest to your configured channels. Monitor the worker logs at `/var/log/focusinbox/worker.log`.

### Step 3: Integrate the Conflict Detection Webhook

In your CRM or project management tool, set up a webhook that listens to the focusinbox `/api/v1/conflicts` endpoint. Example `curl` request:

```bash
curl -X POST https://your-focusinbox-instance/api/v1/conflicts \
  -H "Authorization: Bearer <your_api_key>" \
  -H "Content-Type: application/json" \
  -d '{"threshold": 0.8, "callback_url": "https://your-crm.com/webhook"}'
```

focusinbox will send a JSON payload whenever a new conflict is detected, including the `thread_ids` and the overlapping metadata fields.

## Best Practices

### Practice 1: Establish a Weekly Scoring Audit

Schedule a recurring review every Monday to inspect the `scoring_effectiveness` metric exposed via the `/api/v1/metrics` endpoint. Compare the number of false positives (emails scored high but actually irrelevant) against your team’s manual re‑classification rate. Adjust the `keyword_boost` values and `decay_rate` accordingly. Over time, maintain a false‑positive rate below 5%.

### Practice 2: Use Conflict Detection as a Pre‑Merge Gate

Before merging any pull request that touches shared configuration (e.g., feature flags, environment variables), require a zero‑conflict report from focusinbox. Integrate this check into your CI/CD pipeline by calling the `/api/v1/conflicts` endpoint with a `scope: "config"` parameter. This prevents email‑based miscommunication from causing silent production issues.
