### What is focusinbox?

focusinbox is a professional tool that solves the core problem of email overwhelm by applying a triage-based, zero-inbox methodology. It automatically categorizes, prioritizes, and archives messages using a multi-factor urgency engine, ensuring that only the most critical communications reach your primary attention span.

### Core Features

- **Adaptive Priority Scoring Engine**: Each inbound message is analyzed in real-time using a `priority_score` metric calculated from sender relationship weight (based on historical reply latency < 300ms), keyword density for project-specific triggers, and embedded deadline detection (parses ISO 8601 timestamps within the body). Messages exceeding a configurable `critical_threshold` (default: 85/100) are instantly surfaced to the top of the queue, bypassing all filters.

- **Zero-Latency Smart Archive Pipeline**: Emails flagged as "Read & Done" are automatically moved to a compressed archive bucket within 50ms of user action. The system uses a `archival_policy` parameter (options: `age_based`, `action_complete`, `thread_terminated`) to trigger background cleanup jobs. Archived data is stored in a PostgreSQL-backed vector store, enabling full-text and semantic search with sub-100ms query times.

- **Multi-Modal Notification Throttling**: Based on the `focus_mode` state (active, deep_work, idle), the engine dynamically adjusts push notification frequency. In `deep_work` mode, all non-critical notifications are suppressed, and a `cooldown_interval` of 300 seconds is enforced between any two alerts, preventing context switching while maintaining a guaranteed delivery window for emergency messages.

### How to Use

1. **Configure Priority Thresholds**:
   - Open the Settings panel and navigate to the "Scoring" tab.
   - Set the `critical_threshold` parameter to 90 for high-precision filtering.
   - Define a custom `sender_whitelist` (e.g., `["ceo@company.com", "urgent-alerts@infra.internal"]`) to automatically boost `priority_score` by 30 points for those senders.
   - Execute a dry-run validation using the `/api/v1/score/test` endpoint to verify your rules against a sample of 100 recent emails.

2. **Enable Smart Archival with Retention Policy**:
   - Activate the "Auto-Archive" toggle in the Workflow section.
   - Select `action_complete` as your `archival_policy` to archive only after you mark an email as "Done."
   - Set the `retention_days` parameter to 180. Emails older than this will be moved to a cold storage bucket (S3 Glacier tier) to reduce PostgreSQL load.
   - Run the `/api/v1/archive/force` command to trigger an immediate archival of all eligible messages, monitoring the `archival_duration_ms` metric to ensure it stays under 200ms.

3. **Activate Deep Work Mode for Notification Control**:
   - From the system tray icon, toggle the `focus_mode` to `deep_work`.
   - Confirm the `cooldown_interval` is set to 300 seconds in the Notification Settings.
   - Add a `bypass_keyword` (e.g., "SEV1") to the emergency list. Any email containing this keyword will still trigger an immediate alert, regardless of the cooldown.
   - Verify the suppression is active by checking the `/api/v1/notifications/status` endpoint; it should return `{"blocked": 0, "delayed": 0, "allowed": 0}`.

### Best Practices

- **Implement a Strict "Inbox Zero" Cadence**: Schedule two 15-minute intervals per day (e.g., 10:00 AM and 3:00 PM) to process your focusinbox queue. During these windows, apply the "4D" method (Delete, Delegate, Do, Defer) to every message. Use the `archival_policy` set to `action_complete` to ensure your inbox remains empty between sessions, reducing cognitive load by up to 40%.

- **Leverage Historical Data for Sender Weight Calibration**: Regularly review the `sender_weight` history in the PostgreSQL database. If you notice a recurring low-priority sender who consistently sends critical updates, adjust their `priority_boost` factor manually. For maximum efficiency, schedule a monthly cron job that runs `/api/v1/weight/recalibrate` to automatically update these weights based on your last 30 days of interaction patterns (reply rate, response time, and thread depth).
