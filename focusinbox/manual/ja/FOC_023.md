# focusinbox - FOC_023

## 概要

focusinbox は、メールトリアージと並列処理のレイテンシをミリ秒単位で最適化するプロフェッショナル向けメール整理システムです。PostgreSQL をバックエンドに据え、機械学習による送信者スコアリングと動的優先度キューを実装。1日あたり10万件以上の受信トレイをゼロベースで再構築することを前提としています。

## 詳細な内容

### focusinbox とは

focusinbox は、受信メールの重要度をリアルタイムに評価し、ユーザーの作業コンテキストに応じて最適な処理順序を動的に割り出すメール管理ツールです。従来のルールベースフィルタを超え、送信者の関係性スコア、スレッドの経過時間、添付ファイルの種類とサイズ、過去の返信パターンを統合した **マルチモーダル優先度エンジン** を搭載しています。

### コア機能

1. **動的トリアージエンジン（DTE）**
   - 各メールに対し `priority_score`（0.0〜1.0、小数点以下4桁）を算出し、`urgency_decay_factor` に基づいて24時間経過後にスコアを自動減衰。
   - バックエンドは PostgreSQL の `BRIN` インデックスと `LISTEN/NOTIFY` 機構を利用し、新着メール到着から **平均 12 ミリ秒** で優先度キューに反映。
   - パラメータ例：`sender_reputation_weight: 0.45`, `thread_depth_bonus: +0.15`, `attachment_size_penalty: -0.02 per MB`

2. **コンテキストアウェア・スヌーズ（CAS）**
   - カレンダーイベント（iCal 形式）と連携し、会議中・集中作業中は `do_not_disturb_mode` を自動有効化。その間のメールは `hold_queue` に格納され、解除後 **30 秒以内** に再スコアリングが走る。
   - スヌーズ設定は `snooze_until: '2025-03-15T14:00:00+09:00'` のように絶対時刻指定可能。複数条件の OR/AND 結合にも対応。

3. **バッチアクションAPI（BAA）**
   - RESTful エンドポイント `POST /api/v2/batch/actions` により、選択したメール群に対して一括で `archive`, `delegate`, `categorize` を実行。
   - リクエストボディは `{ "email_ids": ["uuid1","uuid2"], "action": "delegate", "assignee": "user@domain" }`。レスポンスは **平均 45 ミリ秒** で完了、トランザクションは PostgreSQL の `SAVEPOINT` で保護。

### 使い方

1. **初期セットアップと送信者スコアリングの調整**
   - `focusinbox config --init` を実行し、PostgreSQL 接続文字列（例：`postgresql://user:pass@localhost:5432/focusinbox?sslmode=require`）を設定。
   - `focusinbox train --source imap://imap.example.com --limit 5000` で過去5000通のメールから送信者関係性グラフを構築。学習後、`sender_reputation` テーブルが自動更新される。
   - `focusinbox priority --test` でサンプルメール10通のスコアを表示し、`sender_reputation_weight` や `urgency_decay_factor` を微調整。

2. **動的スヌーズルールの作成**
   - `focusinbox rule add --name "会議中は静かに" --condition "calendar_event.status == 'busy'" --action "snooze_until: event_end + 5min"` でルールを登録。
   - `focusinbox rule list --verbose` で全ルールの評価順序（`priority_order`）を確認。必要に応じて `--priority 10` で順位を変更。
   - テスト実行: `focusinbox rule simulate --email_id "uuid"` で該当メールがどのルールにマッチし、どのアクションが発動するかを **マイクロ秒単位** でログ出力。

3. **バッチアクションの自動化パイプライン構築**
   - `focusinbox pipeline create --name "週次アーカイブ" --cron "0 3 * * 0"` で毎週日曜午前3時に実行するパイプラインを作成。
   - パイプラインにステップ追加: `focusinbox pipeline step add --pipeline_id 1 --action "archive" --filter "priority_score < 0.2 AND age > 7d"`。
   - 実行結果は `pipeline_execution_log` テーブルに記録され、失敗時には `retry_count: 3`（各リトライ間隔 5 秒）で自動再試行。

### ベストプラクティス

- **プラクティス 1: スコアリングモデルの定期的な再学習**
  送信者の行動パターンは時間とともに変化します。`sender_reputation` テーブルは最低でも **月1回**、理想的には **週1回** の再学習を推奨。`focusinbox train --incremental --limit 1000` で差分学習を行い、全件再学習によるダウンタイムを回避してください。特に新規ドメインからのメールが急増した場合は、`anomaly_detection_threshold` を 0.85 から 0.90 に引き上げることで誤検知を抑制できます。

- **プラクティス 2: バッチアクションの冪等性保証**
  `batch/actions` API は `idempotency_key` ヘッダー（UUID v4）を必須とします。同一キーでのリクエストは必ず同じ結果を返すため、ネットワーク再送による二重実行を防止できます。また、大量のメール（1000通以上）を処理する際は、`chunk_size: 200` を指定して分割実行し、PostgreSQL の `ROW EXCLUSIVE` ロック競合を避けてください。処理後は必ず `focusinbox stats --recent` でレイテンシ分布（P50/P95/P99）を確認し、ボトルネックを特定します。
