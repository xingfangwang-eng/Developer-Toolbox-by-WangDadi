# focusinbox - FOC_002

## 概要

focusinbox は、受信トレイのノイズをゼロにし、真に重要なメールだけをミリ秒単位で抽出する次世代型メール優先順位管理システムです。機械学習ベースのシグナルスコアリングと、PostgreSQL上のリアルタイムトランザクション処理により、1日10,000通以上の受信環境でも、最重要メールの取りこぼしを99.97%削減します。

## 詳細な内容

### とは focusinbox

focusinbox は、単なるフィルタリングツールではありません。メールヘッダー、ボディのN-gram解析、送信者の過去のインタラクション履歴（開封率、返信時間、クリックスルー率）を多次元ベクトル化し、独自開発の「緊急度推定エンジン」が0.0～1.0のスコアをリアルタイム付与。ユーザーの業務コンテキスト（カレンダーの予定、Slackのスレッドアクティビティ）とクロスリファレンスすることで、本当に今アクションすべきメールだけを前面に押し出します。

### コア機能

- **マルチシグナル優先順位エンジン**: 送信者のドメイン reputation、メール内のアクションキーワード（「至急」「承認」「締切」）、添付ファイルのMIMEタイプとファイルサイズ（例: 1MB以上のPDFは高優先度フラグ）を重み付けスコアリング。PostgreSQL上のトリガー関数が新着メール到着から平均12msでスコアを算出し、受信トレイを自動再ソート。
- **コンテキストアウェアフォーカスモード**: Google Calendar API と連携し、現在の予定の「空き時間」「会議中」「集中時間」を検知。会議中は、送信者が直属の上司または緊急度スコア0.85以上のメールのみをポップアップ表示。それ以外はサイレントアーカイブ。設定は `focusinbox.conf` 内の `contextual_filter_threshold` パラメータ（デフォルト値: 0.75）で調整可能。
- **インタラクション履歴ベースの動的学習**: ユーザーが「スヌーズ」「アーカイブ」「即返信」したメールの特徴量（曜日、時間帯、送信者の部署、メールの文字数）を毎晩バッチ処理で学習。モデルはONNX形式でエクスポートされ、次回のスコアリングに自動反映。学習サイクルは `retrain_interval_hours` で設定（推奨値: 24時間）。

### 使い方

1. **初期セットアップとAPI連携の確立**: `focusinbox init --provider=office365 --mailbox=user@company.com` を実行。OAuth2.0認証後、`~/.focusinbox/config.yaml` に `imap_server`, `imap_port`, `webhook_url` が自動生成される。次に、`focusinbox sync --full` で過去90日分のメールメタデータをPostgreSQLにインポート。初回同期の完了後、`focusinbox status` で `last_sync_timestamp` と `total_emails_indexed` を確認。
2. **優先順位ルールのカスタマイズ**: `focusinbox rule add --name "Critical Client" --condition "from_domain eq 'client.com' AND attachment_size_mb gt 5" --priority 0.95` でルールを追加。複数ルールは `focusinbox rule list` で確認でき、`--order` フラグで評価順序を指定可能（例: ルールAが0.9、ルールBが0.8の場合、先にマッチしたルールのスコアが適用される）。
3. **フォーカスモードの起動と閾値調整**: `focusinbox focus --mode=deep --threshold=0.8` で起動。CLIは常時バックグラウンドで動作し、新着メールを監視。`threshold` を0.9に上げると、表示されるメールは全体の約5%に絞られる。動作中は `focusinbox focus --stats` で `emails_filtered`, `emails_shown`, `average_latency_ms` をリアルタイム表示可能。終了は `Ctrl+C`。

### ベストプラクティス

- **段階的な閾値調整とA/Bテスト**: 初週は `threshold=0.6` で運用し、取りこぼしが発生していないかを `focusinbox audit --missed` で毎日レビュー。2週目以降、0.05刻みで閾値を上げ、誤検知（False Positive）率を1%未満に保つ。重要なメールを逃した場合は `focusinbox rule add --exception` で例外ルールを即時追加する。
- **学習データの定期的なクレンジング**: 毎月1日、`focusinbox model prune --older-than=90` をcronジョブで実行。90日以上前のインタラクションデータを削除し、モデルのドリフトを防止。また、`focusinbox model validate` で最新の100件のユーザーアクションに対する予測精度（Precision, Recall）を確認し、精度が95%を下回った場合は `focusinbox model retrain --force` で強制再学習を実施する。
