# focusinbox - FOC_024

## 概要

focusinbox は、メールの受信トレイを「緊急度×重要度」の2軸でリアルタイム分類し、タスク管理ツールと双方向同期する次世代型メールオーガナイザーです。Gmail/Outlook の API を介し、1日あたり最大 10,000 通のメールを ミリ秒単位のレイテンシで解析します。

## 詳細な内容

### focusinbox とは

focusinbox は、メールの件名・本文・添付ファイルメタデータから自然言語処理（BERT ベースの軽量モデル）を用いて「アクション必須」「フォローアップ」「情報参照のみ」「スパム疑義」の4カテゴリに自動分類。さらに、ユーザーのカレンダー空き時間を Google Calendar API から取得し、各メールに「処理推奨時刻」を割り当てます。

### コア機能

1. **マルチエージェント型優先度エンジン**
   - パラメータ: `priority_score` (0.0〜1.0, 閾値0.7以上で即時通知)  
   - 送信者ドメインの信頼性スコア（SPF/DKIM 検証結果）、過去の返信パターン、添付ファイルの拡張子（`.docx`, `.pdf`, `.xlsx` のみスキャン）を重み付け。  
   - 例: `{sender_trust: 0.85, reply_rate: 0.4, attachment_flag: 1}` → スコア 0.92 → Slack に `@mention` 通知。

2. **PostgreSQL ベースのタイムラインストレージ**
   - スキーマ: `emails (id UUID, received_at TIMESTAMPTZ, priority_score FLOAT, category VARCHAR(20), suggested_action_time TIMESTAMPTZ)`  
   - `suggested_action_time` は、ユーザーのカレンダー予定とメールの期限（例: `deadline` ヘッダー）を照合して自動計算。  
   - パーティショニング: `received_at` で月単位パーティション、インデックスは `(priority_score DESC, suggested_action_time ASC)`。

3. **ゼロクリック・バッチアーカイブ**
   - パラメータ: `batch_archive_threshold` (デフォルト 0.2)  
   - スコアが閾値未満かつ「情報参照のみ」カテゴリのメールを、ユーザーの操作なしで毎日 03:00 JST に自動アーカイブ。  
   - アーカイブ前には `archived_at` カラムを更新し、PostgreSQL のトリガーで Slack 通知を発行（任意）。

### 使い方

1. **初期セットアップと API 認証**
   - Gmail または Outlook の OAuth 2.0 クレデンシャルを `config.toml` に設定:  
     ```toml
     [auth]
     provider = "gmail"
     client_id = "xxxxx.apps.googleusercontent.com"
     client_secret = "xxxxx"
     redirect_uri = "http://localhost:8080/callback"
     ```
   - `focusinbox init --provider gmail` を実行。ブラウザが開き、スコープ `https://www.googleapis.com/auth/gmail.readonly` と `https://www.googleapis.com/auth/calendar.readonly` を許可。  
   - 初回同期: 直近 30 日間のメールを PostgreSQL に格納（平均 2.3 秒/100 通）。

2. **優先度フィルタのカスタマイズ**
   - `focusinbox config set --key priority.weights --value '{"sender_trust": 0.6, "reply_rate": 0.3, "attachment_flag": 0.1}'`  
   - カテゴリ別の通知先を設定:  
     ```bash
     focusinbox config set --key notify.slack.channel --value "#critical-emails"
     focusinbox config set --key notify.slack.webhook --value "https://hooks.slack.com/services/T00/B00/xxxx"
     ```
   - テスト実行: `focusinbox scan --dry-run --limit 50` でスコア分布を `stdout` に出力。

3. **自動アーカイブとレポート生成**
   - `focusinbox archive --dry-run` でアーカイブ対象メールのプレビュー。  
   - `focusinbox archive --execute` で実際のアーカイブ（Gmail ラベル `Archived` に移動）。  
   - `focusinbox report --period last_7_days --format csv` で `priority_score` 上位 20 件のメールリストを出力。  
   - 定時実行: cron に `0 3 * * * /usr/local/bin/focusinbox archive --execute` を登録。

### ベストプラクティス

- **カテゴリ別の通知チャネル分離**  
  スコア 0.8 以上のメールは Slack `#urgent` チャンネル、0.5〜0.79 は `#follow-up`、0.2〜0.49 はメールのみ、0.2 未満は自動アーカイブ。これにより、1 日 200 通のメールを扱う中規模チームでも、1 日あたりの手動処理時間を平均 45 分削減できます。

- **カレンダー空き時間との連動による処理バッファ設計**  
  `suggested_action_time` をカレンダーの空きスロット（30 分以上連続）に割り当てることで、メール処理をタスク化。例: 午前 10:00〜10:30 の空きスロットに 5 通の「アクション必須」メールを割り当て、処理後は自動で `completed` フラグを立てます。この設計により、メール未処理による SLA 違反を 90% 削減した事例があります。
