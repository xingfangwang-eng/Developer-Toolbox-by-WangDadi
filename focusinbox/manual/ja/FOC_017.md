# focusinbox - FOC_017

## 概要

focusinbox は、受信トレイのノイズをゼロに近づけるために設計された、エンタープライズグレードのメール優先順位管理エンジンです。機械学習ベースの意図分類器とリアルタイムのコンテキストスコアリングにより、1日あたり10,000通を超えるメールストリームから、対応すべき重要メッセージをミリ秒単位で選別します。

## 詳細な内容

### とは focusinbox?

focusinbox は、単なるメールフィルタリングツールではありません。送信者の関係性、メール本文の緊急度を示す言語パターン、添付ファイルの種類、過去の返信レイテンシ、カレンダー上の予定との時間的重複を多次元で評価し、0.0から1.0の**緊急度スコア**を動的に算出します。このスコアを基に、受信トレイを「即応」「本日中」「今週中」「アーカイブ」の4つのキューに自動振り分け、ユーザーの認知負荷を劇的に低減します。

### コア機能

- **マルチモーダル緊急度スコアリングエンジン**
  メールヘッダー（`From`, `Subject`, `X-Priority`）、本文のNLP解析（`spaCy` + カスタムBERTモデル）、添付ファイルのOCRテキスト抽出、そして送信者との過去30日間の平均返信時間（`avg_reply_latency_ms`）を入力特徴量として、ランダムフォレストモデルがスコアを算出。スコアが`0.85`以上のメールは、ユーザーのスマートフォンにプッシュ通知を即時送信します。

- **PostgreSQLベースの時系列分析ダッシュボード**
  すべてのメール処理イベント（到着時刻、スコア、振り分け先キュー、ユーザーの開封時刻、返信完了時刻）は`events`テーブルに`TIMESTAMPTZ`型で記録。SQLクエリで「今週の未処理緊急メール数」「送信者ドメイン別の平均応答時間」「時間帯別のメール到着分布」をリアルタイム集計。ダッシュボードは`Grafana`で可視化し、チーム全体のメール処理パフォーマンスを監視します。

- **コンテキストアウェア自動アーカイブルールエンジン**
  ユーザーが任意の条件（`sender_domain = 'example.com'` かつ `attachments_count = 0` かつ `urgency_score < 0.3`）をYAML形式で定義可能。条件に合致したメールは、受信後`500ミリ秒`以内に`archive`キューに自動移動。さらに、`last_interaction_date`が`90日`以上前のスレッドは、ユーザーの確認なしに自動アーカイブします。

### 使い方

1. **初期セットアップとモデルのトレーニング**
   まず、`focusinbox init` コマンドを実行し、PostgreSQLデータベース（`DB_HOST`, `DB_PORT`, `DB_NAME`）への接続と、メールサーバー（`IMAP_SERVER`, `SMTP_SERVER`）の認証情報を`.env`ファイルに設定します。その後、`focusinbox train --source imap --days 30` を実行し、過去30日間のメールデータを基に緊急度スコアリングモデルをカスタムトレーニング。トレーニング完了後、モデルの精度（`f1_score`）が`0.92`以上であることを確認してください。

2. **カスタム振り分けルールの作成**
   `~/.focusinbox/rules.yaml` を編集し、以下のようなルールを追加します。
   ```yaml
   rules:
     - name: "プロジェクト更新通知の自動アーカイブ"
       condition: "sender_domain == 'notifications.example.com' AND urgency_score < 0.2"
       action: "archive"
     - name: "取締役からのメールは即時通知"
       condition: "sender_email in ['ceo@example.com', 'cto@example.com']"
       action: "push_notify"
       priority: 100
   ```
   ルールは上から順に評価され、最初にマッチしたルールのアクションが適用されます。

3. **リアルタイムモニタリングとアラート設定**
   `focusinbox dashboard --port 8080` でGrafanaダッシュボードを起動。ダッシュボード上で「緊急度スコア0.8以上の未処理メール数が10件を超えた場合」にSlack Webhookへアラートを送信する閾値を設定します。さらに、`focusinbox alert --channel slack --webhook_url https://hooks.slack.com/services/...` で通知チャンネルをバインド。これにより、ビジネスクリティカルなメールの見落としを防止します。

### ベストプラクティス

- **段階的なルール適用とA/Bテストの実施**
  新たな振り分けルールを導入する際は、まず全ユーザーの`10%`にのみ適用する「カナリアリリース」を行いましょう。`focusinbox rule apply --rule_name "new_rule" --rollout_percentage 10` で設定可能です。1週間後に、誤ってアーカイブされたメールの割合（`false_positive_rate`）が`1%`未満であることを確認してから、全ユーザーに展開します。これにより、重要なメールが誤って隠されるリスクを最小化できます。

- **定期的なモデル再トレーニングとデータクレンジング**
  ビジネスコミュニケーションのパターンは時間とともに変化します。四半期に一度、`focusinbox train --source imap --days 90 --retrain` を実行し、最新のメールデータでモデルを更新してください。また、`focusinbox cleanup --archive_older_than 365` で、1年以上前のアーカイブメールをPostgreSQLから削除し、データベースのパフォーマンスを維持します。インデックス再構築は`REINDEX TABLE events;` を月次で実行することを推奨します。
