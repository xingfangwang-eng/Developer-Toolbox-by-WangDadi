# focusinbox - FOC_029

## 概要

focusinbox は、受信トレイのノイズを排除し、高優先度メールのみに集中するための専用エンジンです。メールの開封率、返信速度、プロジェクト遅延リスクをリアルタイムで解析し、ユーザーが真に処理すべきメッセージだけを抽出します。対応言語は日本語、英語、中国語。

## 詳細な内容

### とは focusinbox?

focusinbox は、IMAP および Microsoft Graph API を介してメールボックスに直接接続し、機械学習ベースの優先順位付けエンジンを搭載したプロフェッショナルツールです。従来のルールベースのフィルタリングとは異なり、送信者の過去の返信パターン、メール本文の緊急度スコア、添付ファイルの種類とサイズ、会話スレッドの深度を総合的に評価し、処理すべきメールをミリ秒単位で判定します。

### コア機能

- **動的優先順位エンジン**: 各メールに対して `priority_score`（0.0〜1.0）を算出。閾値（デフォルト `0.75`）を超えたメールのみを「フォーカスビュー」に表示。過去30日間のユーザーの返信行動データを元に、重みベクトル `[urgency_weight, sender_weight, thread_weight, attachment_weight]` を自動調整。PostgreSQL に保存された履歴データと照合し、毎日 03:00 UTC にモデルを再学習。

- **コンテキストアウェアスヌーズ**: メールを「後で処理」する際、単なる時間指定ではなく、関連するカレンダーイベント（Google Calendar / Outlook）とプロジェクト管理ツール（Asana / Jira）のタスク期限を自動取得。例：`snooze_until_next_available_slot` パラメータで、次の空き時間枠（15分以上）を計算し、その直前に再通知。再通知時には、元のメールに加えて、関連するタスクの進捗率や直近の更新差分を表示。

- **マルチアカウント統合ダッシュボード**: 最大5つのメールアカウント（Gmail / Outlook / Exchange）を同時接続。各アカウントの `unread_count`, `focus_count`, `avg_response_time_ms`, `overdue_ratio` をリアルタイム集計。ダッシュボード上で「最も反応が遅れているアカウント」を自動特定し、そのアカウントに絞った集中処理モードをワンクリックで起動。データはすべてローカル暗号化（AES-256）され、クラウドには送信しない。

### 使い方

1. **初期セットアップとアカウント連携**  
   アプリケーション起動後、`Settings > Accounts` から「Add Account」をクリック。IMAP サーバーアドレス（例：`imap.gmail.com:993`）とアプリパスワード、または Microsoft Graph API のテナントID・クライアントIDを入力。接続テスト後、`sync_interval_seconds`（推奨値: `30`）を設定。初回同期時には、過去90日分のメールヘッダーとメタデータをローカル SQLite にキャッシュ。同期完了後、`focus_threshold` をスライダーで調整（初期値 `0.75`、範囲 `0.5〜0.95`）。

2. **フォーカスビューでのメール処理**  
   メイン画面左ペインに「Focus Inbox」タブが表示される。ここには `priority_score >= focus_threshold` のメールのみがリストアップ。各メールの左端には、`priority_score` を色分けしたバッジ（緑: 0.75〜0.85、黄: 0.85〜0.95、赤: 0.95〜1.0）が表示。メールをクリックすると、右ペインに本文と共に「Recommended Action」が表示される（例：「このメールには添付ファイルがあります。至急確認してください」「このスレッドは未読が3件、返信期限を過ぎています」）。処理後、「Snooze」「Archive」「Mark Done」のいずれかを選択。`Mark Done` を選んだ場合、そのメールは `processed_log` テーブルに記録され、今後同一送信者からの類似メールのスコアリングに影響を与える。

3. **パフォーマンス分析とレポート出力**  
   週次・月次で自動生成される `focus_report.pdf` には、以下の指標が含まれる：  
   - `total_emails_received` に対する `focus_emails_processed` の割合（目標値: 40%以上）  
   - `avg_response_time_ms` の推移（前週比）  
   - `overdue_ratio`（期限切れメールの割合）  
   - 最も時間を費やした送信者トップ5  
   レポートは `Reports` タブから手動エクスポートも可能（CSV / JSON / PDF）。エクスポート時には、`export_date_range` パラメータで期間を指定（デフォルト: 過去30日間）。

### ベストプラクティス

- **閾値の段階的調整とチーム共有**  
  初週は `focus_threshold = 0.85` で運用し、1週間後に `avg_response_time_ms` と `overdue_ratio` を確認。目標値に達しない場合は0.05ずつ下げる。チームで利用する場合、全メンバーの `focus_threshold` を同一値に統一し、毎週月曜日の朝に `focus_report.pdf` を共有ミーティングでレビューする。これにより、チーム全体のメール処理速度が平均30%向上した事例がある。

- **スヌーズ機能とカレンダー連携のフル活用**  
  「後で処理」するメールは、必ず `snooze_until_next_available_slot` を有効にした状態でスヌーズする。手動で時間指定すると、カレンダーの空き時間と矛盾が生じ、再通知時に「処理すべきだが時間がない」状態に陥りやすい。また、週に1度、`snooze_log` テーブルを確認し、3回以上スヌーズされたメールは自動的に `focus_threshold` を0.1引き上げるルールを設定することを推奨する。
