# focusinbox - FOC_019

## 概要

focusinbox は、受信トレイのノイズをゼロベースで排除し、真に重要なメールのみをミリ秒単位で抽出する、エンタープライズグレードのメール優先順位管理エンジンです。PostgreSQL 上のメタデータインデックスと連携し、1日10万通以上のトラフィックを処理可能な設計を採用しています。

## 詳細な内容

### focusinbox とは

focusinbox は、SMTP/IMAP プロトコルを介してメールボックスに直接接続し、機械学習ベースの重要度スコアリングとコンテキストクラスタリングを実行する、プロフェッショナル向けメールオーガナイゼーションツールです。従来のルールベースフィルタとは異なり、送信者の行動履歴、返信パターン、プロジェクト内の位置づけを動的に解析し、受信トレイを「今対応すべき」と「後で確認」に二分します。

## コア機能

### 1. 動的優先順位スコアリングエンジン（DPS Engine）

- **パラメータ**: `priority_score` は 0.0 ～ 1.0 の浮動小数点値で、閾値 `threshold_urgent`（デフォルト 0.75）を超えたメールのみをフォーカスビューに表示。
- **技術指標**: 各メールのスコアリング処理は平均 12 ミリ秒で完了。PostgreSQL 上の `email_metadata` テーブルにリアルタイムで書き込み、インデックス最適化により 50 万件のレコードでも検索レスポンスは 200 ミリ秒未満。
- **実装詳細**: 過去 30 日間の送信者とのやり取り頻度、添付ファイルの有無、メール本文のキーワード密度（例: 「緊急」「締切」）を重み付けし、`tf-idf` 類似度スコアと組み合わせて最終スコアを算出。

### 2. コンテキストクラスタリングによるスレッド自動統合

- **パラメータ**: `cluster_window_minutes`（デフォルト 120 分）と `subject_similarity_threshold`（デフォルト 0.85）を設定可能。
- **技術指標**: 同一プロジェクト内のメールを、件名のレーベンシュタイン距離と送信者リストの Jaccard 係数でグループ化。クラスタリング処理は 1 メールあたり 8 ミリ秒で完了し、誤統合率は 0.3% 未満。
- **実装詳細**: クラスタリング結果は `thread_clusters` テーブルに保存され、`parent_thread_id` を用いた再帰クエリで階層構造を再構築。ユーザーは統合されたスレッドを「展開/折りたたみ」可能。

### 3. タイムゾーン対応の応答期限アラート

- **パラメータ**: `deadline_buffer_minutes`（デフォルト 30 分）と `timezone_auto_detect`（デフォルト true）を環境変数で制御。
- **技術指標**: メール本文から `自然言語解析` で期限日時を抽出（精度 94.2%）。抽出された期限は `deadline_utc` として保存され、期限の 30 分前に WebSocket 経由でプッシュ通知を送信。
- **実装詳細**: タイムゾーンは `IANA Time Zone Database` に基づき、送信者のメールヘッダー `Received` フィールドから自動判定。複数タイムゾーンにまたがるチームでも、各ユーザーのローカル時間で期限が表示される。

## 使い方

### ステップ 1: IMAP 接続設定と初回インデックス構築

1. 設定ファイル `config/focusinbox.yaml` に以下のパラメータを記述:
   ```yaml
   imap_server: "imap.gmail.com"
   imap_port: 993
   auth_method: "oauth2"
   sync_interval_seconds: 300
   db_connection: "postgresql://user:pass@localhost:5432/focusinbox"
   ```
2. `focusinbox init --full-sync` を実行。初回同期では、過去 90 日間の全メールを `email_metadata` テーブルに取り込み、`priority_score` を一括計算。10 万件のメールで約 4.2 秒で完了。
3. 同期完了後、`focusinbox status --index-stats` でインデックスサイズと未処理メール数を確認。

### ステップ 2: カスタムルールとスコアリング重みの調整

1. `rules/custom_rules.json` にルールを追加:
   ```json
   {
     "rules": [
       {
         "field": "sender_domain",
         "operator": "equals",
         "value": "client-company.com",
         "weight_boost": 0.3
       },
       {
         "field": "subject_contains",
         "operator": "regex",
         "value": "(urgent|asap|critical)",
         "weight_boost": 0.5
       }
     ]
   }
   ```
2. `focusinbox reload-rules` でルールをホットリロード。ルール変更は 1 秒未満で反映され、既存メールのスコアはバックグラウンドで再計算される。
3. `focusinbox test-rule --email-id 12345` で特定メールに対するルール適用結果をプレビュー。

### ステップ 3: フォーカスビューの活用とアラート設定

1. Web ダッシュボード（`http://localhost:8080`）にアクセスし、デフォルトのフォーカスビューで `priority_score >= 0.75` のメールのみを表示。
2. サイドバーから「アラート設定」を開き、`deadline_alert` を有効化。`webhook_url` に Slack や Teams のエンドポイントを指定。
3. `focusinbox alert --test` でテスト通知を送信。通知にはメールの件名、`priority_score`、抽出された期限（例: `2025-03-15T14:00:00+09:00`）が含まれる。

## ベストプラクティス

### プラクティス 1: ドメインホワイトリストとブラックリストの併用によるノイズ削減

エンタープライズ環境では、`sender_domain` ベースのホワイトリスト（例: `@company.com`）とブラックリスト（例: `@newsletter.com`）を `rules/custom_rules.json` に定義する。これにより、マーケティングメールの `priority_score` を一律 0.2 に抑制し、クライアントからのメールを 0.9 以上にブーストできる。初期設定では `blacklist_weight_penalty` を -0.4 に設定し、1 週間のログを確認しながらチューニングすることを推奨。

### プラクティス 2: PostgreSQL のパーティショニングと定期的なバキューム

`email_metadata` テーブルは `received_at` で月単位のパーティショニングを行う。`CREATE TABLE email_metadata_2025_03 PARTITION OF email_metadata FOR VALUES FROM ('2025-03-01') TO ('2025-04-01');` のように設定し、`autovacuum` を有効にして古いパーティションのデッドタプルを自動回収する。これにより、100 万件を超えるデータでもクエリパフォーマンスを 300 ミリ秒未満に維持できる。また、`focusinbox vacuum --partition 2024_12` で手動バキュームもサポート。
