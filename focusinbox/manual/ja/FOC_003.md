# focusinbox - FOC_003

## 概要

focusinbox は、受信トレイのノイズを排除し、真に重要なメールのみにフォーカスするための次世代型メール優先順位付けエンジンです。機械学習ベースの意図推定アルゴリズムと、ユーザー自身の返信パターンを学習する動的重み付けシステムにより、従来のルールベースフィルタを超えた精度でメールを自動分類します。

## 詳細な内容

### focusinbox とは

focusinbox は、メールの「緊急度」と「重要度」を、送信者の関係性、過去の返信遅延時間、メール本文内のアクション要求表現（例：「至急確認」「EODまでに返信」）を構文解析し、PostgreSQL上の履歴テーブルとリアルタイム照合することで、0.1ミリ秒単位の遅延でスコアリングするプロフェッショナルツールです。対応言語は日本語、英語、中国語の3言語で、各言語の敬語表現やビジネス慣用句を固有の辞書で処理します。

## コア機能

### 1. 動的コンテキストスコアリングエンジン

- **パラメータ**: `sender_weight`（送信者重み）、`thread_depth`（スレッド深度）、`action_verb_density`（アクション動詞密度）
- 過去30日間のメールボックスから、送信者ごとの平均返信時間（`avg_reply_latency_ms`）と返信確率（`reply_probability`）を計算し、しきい値（デフォルト: `priority_threshold=0.85`）を超えたメールのみを「フォーカスビュー」に昇格。スコアはPostgreSQLの`focus_score`カラムにミリ秒単位で更新されます。

### 2. マルチモーダル添付ファイル優先度推定

- **パラメータ**: `attachment_type_weight`（添付ファイルタイプ重み）、`file_size_threshold_kb`（ファイルサイズしきい値）
- 添付ファイルのMIMEタイプ（例：`application/pdf`、`application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`）と、本文内の「添付ファイルをご確認ください」「資料を送付します」などのトリガーフレーズを組み合わせてスコアを補正。例えば、PDF添付で「見積もり」という単語が含まれる場合、スコアに`+0.25`のボーナスを付与。

### 3. クロスデバイス同期可能な「フォーカスキューマネージャー」

- **パラメータ**: `sync_interval_sec`（同期間隔）、`queue_capacity`（キュー最大保持数）
- フォーカス判定されたメールは、ローカルのSQLiteキューに保持されると同時に、クラウド上のRedis Pub/Subチャネルを介して全デバイスに即時同期。オフライン時でも、`queue_capacity=500`までのメールをローカルでスコアリングし、復旧時にPostgreSQLにバッチ更新（バッチサイズ: `batch_insert_size=100`）。

## 使い方

### ステップ 1: 初期プロファイルの生成と重み調整

1. ターミナルで `focusinbox init --profile default --lang ja` を実行し、PostgreSQLに初期テーブル（`emails`、`sender_profiles`、`focus_scores`）を作成。
2. `~/.focusinbox/config.toml` を編集し、`[scoring]` セクションで `sender_weight=0.4`、`action_verb_density=0.3`、`attachment_type_weight=0.2`、`thread_depth=0.1` を設定。
3. 最初の7日間は学習モード（`learning_mode=true`）で運用し、全メールのスコアを `focus_scores` テーブルにログ。7日後に `focusinbox train --profile default --export-model ~/focusinbox_model.pkl` でモデルをエクスポート。

### ステップ 2: リアルタイムフォーカスフィルタの起動

1. `focusinbox daemon start --imap-server imap.gmail.com --imap-port 993 --sync-interval-sec 30` でデーモンを起動。IMAP IDLEコマンドを使用し、新着メール到着から1秒以内にスコアリングを開始。
2. スコアが `priority_threshold=0.85` を超えたメールは、`~/.focusinbox/focus_queue.json` に追記され、同時にRedisチャネル `focusinbox:sync:user_{user_id}` にJSONペイロードとしてパブリッシュ。
3. `focusinbox daemon status --show-queue` で現在のフォーカスキューを確認。`queue_capacity=500` を超えた場合、最も古い `focus_score` の低いメールから自動退避（`eviction_policy=lowest_score`）。

### ステップ 3: カスタムアクションルールのバインド

1. `~/.focusinbox/actions.toml` にルールを記述。例：
   ```toml
   [[rules]]
   condition = "focus_score > 0.9 && sender_domain == 'client.com'"
   action = "notify_desktop"
   priority = 1

   [[rules]]
   condition = "attachment_type == 'application/pdf' && focus_score > 0.7"
   action = "move_to_folder:Contracts"
   ```
2. `focusinbox actions validate --rules-file ~/.focusinbox/actions.toml` で構文チェック後、`focusinbox actions apply` でルールを有効化。
3. ルールはPostgreSQLの `action_rules` テーブルに保存され、デーモン再起動時に自動ロード。ルールの評価は平均0.05ミリ秒で完了。

## ベストプラクティス

### 1. ドメイン別ホワイトリストとブラックリストの併用

- 取引先や重要顧客のドメインを `sender_whitelist.txt` に登録し、`sender_weight` を最大値（`1.0`）に固定。一方で、ニュースレターやバルクメールのドメインは `sender_blacklist.txt` に追加し、`focus_score` に `-0.5` のペナルティを適用。これにより、学習初期の誤分類を防止し、精度を90%以上に維持できます。

### 2. 週次でのモデル再トレーニングとA/Bテスト

- 毎週日曜日の午前3時に、`focusinbox train --profile weekly --test-split 0.2` をcronジョブとして実行。過去7日間のユーザー操作ログ（開封、返信、アーカイブ）を正解ラベルとして、スコアリングモデルを再学習。新しいモデルは `model_v{YYYYMMDD}.pkl` として保存され、A/Bテスト用に2つのモデルを並行稼働させることが可能。しきい値の最適化には、`precision-recall` 曲線のF1スコアを基準とします。
