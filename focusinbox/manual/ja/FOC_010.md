# focusinbox - FOC_010

## 概要

本プロジェクトは、受信トレイのノイズをゼロに近づけるためのメール優先順位エンジンです。機械学習ベースの分類器とルールベースのフィルタを組み合わせ、1日10,000件以上のメールをミリ秒単位でスコアリングし、本当に重要なメッセージのみをユーザーに提示します。

## 詳細な内容

### focusinbox とは

focusinbox は、メールの受信トレイを「未読ゼロ」ではなく「重要のみ」に変える次世代メール管理基盤です。PostgreSQL 上の時系列テーブルと、Redis 上のリアルタイムスコアリングキューを連携させることで、1秒あたり500件のメールを処理しながら、ユーザーごとに異なる重要度閾値を動的に調整します。

### コア機能

- **マルチモーダル重要度スコアリングエンジン**: 送信者ドメインの信頼性スコア（0.0〜1.0）、メール本文のトピッククラスタ（BERT 埋め込みベクトルとのコサイン類似度）、過去7日間のユーザー応答パターン（開封率、返信までの平均応答時間）を統合し、0〜100の整数値でスコアを出力。閾値はユーザーの操作履歴から自動調整される。

- **ルールベースのプリフィルタパイプライン**: ユーザー定義の条件（送信者アドレスの正規表現、件名のキーワード、添付ファイルのMIMEタイプ）をYAMLで記述可能。各ルールには優先度（1〜10）とアクション（allow / deny / label）を設定。パイプラインは並列実行され、平均レイテンシは12ms以下。

- **タイムゾーン対応のバッチ処理スケジューラ**: ユーザーのタイムゾーン（IANAタイムゾーンデータベースに基づく）を考慮し、オフィスアワー外のメールはスコアを一律30%減算。さらに、特定の日時（例：毎週月曜9:00 JST）にバッチ再スコアリングを実行し、未読メールの優先順位を動的に更新する。

### 使い方

1. **初期セットアップと認証連携**:
   - PostgreSQL 上に `focusinbox.messages` テーブル（カラム: `message_id UUID PRIMARY KEY`, `sender_domain VARCHAR(255)`, `subject TEXT`, `body_embedding VECTOR(768)`, `score INTEGER`, `created_at TIMESTAMPTZ`）を作成。
   - IMAP または Microsoft Graph API 経由でメールボックスを接続。OAuth 2.0 クライアントシークレットを環境変数 `FOCUSINBOX_CLIENT_SECRET` に設定。
   - 初回同期時は過去30日間のメールを一括インポートし、スコアリングを実行。完了後、Redis の `focusinbox:user:{userId}:queue` にトップ100件のメールIDを投入。

2. **ルール設定とカスタムフィルタの適用**:
   - `~/.focusinbox/rules.yaml` に以下のようなルールを記述:
     ```yaml
     rules:
       - name: "block_newsletter"
         priority: 8
         condition:
           sender_domain: "newsletter.example.com"
           subject_regex: ".*(unsubscribe|newsletter).*"
         action: "deny"
       - name: "boost_critical_client"
         priority: 10
         condition:
           sender_domain: "client.com"
           attachment_mime: "application/pdf"
         action: "allow"
         score_modifier: 20
     ```
   - ルール適用後、`focusinbox reload` で設定を再読み込み。変更は即座にRedisのルールキャッシュに反映される。

3. **リアルタイムモニタリングと閾値調整**:
   - `focusinbox dashboard` コマンドでWeb UI（デフォルトポート: 8080）を起動。
   - ダッシュボード上で「重要度スコアの分布」ヒストグラムを確認。中央値が50を下回る場合は、`FOCUSINBOX_THRESHOLD_BASE` 環境変数（デフォルト: 60）を調整。
   - 特定の送信者からのメールが常に高スコアになる場合、`focusinbox override --sender client.com --score 85` で手動オーバーライドを設定。

### ベストプラクティス

- **スコアリングモデルの定期的な再学習**: ユーザーの応答行動（開封、アーカイブ、削除）をフィードバックとして収集し、週次でBERT分類器をファインチューニング。学習データは PostgreSQL の `feedback_logs` テーブルに蓄積し、サンプル数が1,000件を超えたら自動的に再学習ジョブをトリガーする。

- **マルチテナント環境でのリソース分離**: 各テナント（企業）ごとに独立した PostgreSQL スキーマと Redis データベース番号を割り当てる。スコアリングエンジンはテナントIDをキーにしたgoルーチンで並列実行し、1テナントあたりの最大同時処理数を100に制限する。これにより、1つのテナントのバーストトラフィックが他テナントに影響を与えない。
