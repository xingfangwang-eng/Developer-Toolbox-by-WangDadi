# focusinbox - FOC_005

## 概要

本プロジェクトは、メールトリアージと集中処理に特化した高精度フィルタリングエンジンです。機械学習による重要度スコアリングと、ユーザー定義のルールベース制御を組み合わせ、受信箱のノイズをミリ秒単位で除去します。

## 詳細な内容

### とは focusinbox?

focusinbox は、メールの受信ストリームをリアルタイムで解析し、重要度に応じて自動分類・振り分けを行うアーキテクチャです。PostgreSQL の NOTIFY/LISTEN 機構と連携し、着信即座に処理パイプラインが起動します。対応言語は日本語・英語・中国語を含む12言語、エンコーディング自動判別機能を標準搭載しています。

### コア機能

- **マルチモーダル重要度スコアリングエンジン**  
  送信者履歴、返信頻度、キーワード出現パターン、添付ファイルの有無とタイプを特徴量として、LightGBM モデルが0.0〜1.0のスコアを算出。スレッド単位での会話コンテキストを保持し、返信期限が迫っているメールは自動で優先度が+0.15補正されます。

- **カスタムルールチェーンと条件分岐アクション**  
  `sender_domain`, `subject_regex`, `body_contains`, `attachment_size_kb` などのパラメータを AND/OR/NOT で組み合わせ、最大64階層の入れ子ルールを定義可能。条件に合致したメールは、指定のフォルダへ移動、ラベル付与、Slack/Teams への通知、または自動アーカイブまで実行します。

- **集中処理モード「フォーカスセッション」**  
  タイマー（25分/50分/カスタム）を設定すると、その間は重要度スコア0.7以上のメールのみを表示。残りは「後で見る」キューに格納され、セッション終了後に一括処理できます。キューイングは Redis 上の ZSET で管理され、期限切れアラートを発行します。

## 使い方

1. **初期設定とモデルトレーニング**  
   `focusinbox init --engine lightgbm --lang ja` を実行し、メールボックス（IMAP/POP3）を接続。初回は過去30日分のメールをスキャンし、ユーザーが手動で「重要」「不要」のラベルを各100件以上付与することで、個人適応モデルを生成します。

2. **カスタムルールの追加**  
   `focusinbox rule add --name "取引先緊急" --condition "sender_domain contains 'partner.com' AND subject_regex '緊急|至急'" --action move_to_folder "urgent" --priority 10`  
   上記のように、CLIまたはYAMLファイルでルールチェーンを定義。ルールはホットリロード可能で、変更後5秒以内に反映されます。

3. **フォーカスセッションの起動と終了処理**  
   `focusinbox focus --duration 25 --min-score 0.7` でセッションを開始。セッション中は新着メールもスコアリングされ、閾値未満は自動キューイング。終了後、`focusinbox queue process --batch-size 50` でキュー内メールを一括処理（既読/アーカイブ/削除）します。

## ベストプラクティス

- **ルールの優先度設計**  
  ルールには0〜100の優先度を設定可能で、数値が大きいほど先に評価されます。緊急度の高いルール（例：障害通知）は90以上、定型的な振り分けルールは10〜30に設定し、評価順序を最適化してください。ルール数が50を超える場合は、`focusinbox rule optimize` で競合チェックと評価順の自動調整を推奨します。

- **モデルの定期的な再学習**  
  個人適応モデルは週次で再学習を行うことを推奨します。`focusinbox model retrain --force --retention-days 90` により、過去90日分のラベルデータを保持したまま最新のメールパターンに適応。再学習後は必ず `focusinbox model validate --test-ratio 0.2` で精度（目標: F1スコア0.85以上）を確認してください。
