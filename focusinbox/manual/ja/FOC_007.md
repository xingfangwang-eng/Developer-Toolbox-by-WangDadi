# focusinbox - FOC_007

## 概要

focusinboxは、受信トレイのノイズをゼロに近づけるためのメール優先順位管理エンジンです。IMAP/Exchange ActiveSyncプロトコルを介してメールボックスに直接接続し、機械学習による意図推定とコンテキストスコアリングをリアルタイムで実行します。

## 詳細な内容

### とは focusinbox?

focusinboxは、メールボックス内の全メッセージに対し、送信者関係性、返信頻度、キーワード密度、カレンダーイベントとの時間的重複を多次元で解析し、0から100までの緊急度スコアをミリ秒単位で算出するプロフェッショナルツールです。PostgreSQL上の時系列テーブルと連携し、過去30日間のスコア推移を可視化します。

### コア機能

- **マルチモーダル優先順位エンジン**: メール本文のTF-IDFベクトルと送信者プロファイル（直属上司フラグ、プロジェクトメンバーフラグ）を重み付けし、`priority_score = (0.4 * sender_weight) + (0.35 * content_relevance) + (0.25 * recency_factor)` の式でスコアリング。閾値70以上を即時通知対象とし、応答時間の中央値を従来比42%短縮。
- **タイムブロック自動仕分け**: カレンダーAPI（Google Calendar API v3 / Microsoft Graph API）と連携し、予定の開始・終了時刻を取得。空き時間スロットに応じてメールを「今すぐ処理」「30分以内」「本日中」の3バケットに動的再割り当て。バケット切り替えのレイテンシは平均200ms以下。
- **コンテキストベースのスヌーズチェーン**: メールをスヌーズする際、次に表示すべき条件（例：`sender_domain = "client.com" AND attachments_count > 0`）をSQLライクなDSLで指定可能。条件が満たされた瞬間にIMAP IDLEコマンドでプッシュ通知を再送。スヌーズ成功率は95.3%（n=10,000）。

### 使い方

1. **OAuth2.0認証と初期スコアリング設定**: まず`focusinbox-cli auth --provider google --scope https://www.googleapis.com/auth/gmail.modify` を実行。認証後、`config.yaml` に `priority_threshold: 70` と `timezone: Asia/Tokyo` を記述。`focusinbox-cli scan --mailbox INBOX --history 30d` で過去30日間の全メールをスキャンし、PostgreSQLの `email_scores` テーブルに初期スコアを書き込む。スキャン完了までに約1.2秒/1000メール（テスト環境: 8コア、16GB RAM）。

2. **タイムブロックルールの動的調整**: `focusinbox-cli rule add --name "client_meeting" --condition "calendar_event.title CONTAINS '定例会'" --bucket "今すぐ処理" --priority 90` でルールを追加。`focusinbox-cli daemon start --interval 5s` で常時監視デーモンを起動。5秒ごとにカレンダー差分をポーリングし、該当メールのバケットを自動移動。ログは `/var/log/focusinbox/daemon.log` にJSON形式で出力。

3. **スヌーズチェーンのテストとデプロイ**: `focusinbox-cli snooze create --message-id <msg_id> --until "condition: sender.email = 'urgent@partner.com'" --dry-run` で条件評価をシミュレーション。`dry-run` 結果が `match: true` を返すことを確認後、`--deploy` フラグで本番適用。スヌーズ状態はRedisのキー `snooze:<msg_id>` にTTL付きで保存され、条件トリガー時にPub/Subでデーモンに通知。

### ベストプラクティス

- **スコアリングモデルの定期再学習**: 毎週日曜日午前3時に `focusinbox-cli model retrain --feedback-db feedback_logs --epochs 10` をcronジョブで実行。ユーザーが手動で「重要」「不要」とマークしたフィードバックをロジスティック回帰モデルに反映させ、スコアリング精度を継続的に向上させる。再学習中のダウンタイムはゼロ（モデルはホットスワップ可能）。
- **バケット閾値のA/Bテスト**: 2つのPostgreSQLインスタンス（`focusinbox_prod_a`、`focusinbox_prod_b`）を用意。インスタンスAには閾値65、Bには75を設定し、`focusinbox-cli stats compare --instance-a prod_a --instance-b prod_b --metric response_time_p95` で95パーセンタイル応答時間を比較。統計的有意差（p < 0.05）が出た閾値を全インスタンスにロールアウトする。
