# focusinbox - FOC_028

## 概要

focusinbox は、受信トレイのノイズを排除し、真に重要なメールのみに集中するためのプロフェッショナル向けメール優先順位管理システムです。機械学習による動的スコアリングと、PostgreSQL 上のリアルタイムキューイングにより、1 日あたり 10,000 件以上のメールを処理しながら、ユーザーが設定した重要度閾値（0.0～1.0）に基づいて瞬時に振り分けます。

## 詳細な内容

### focusinbox とは

focusinbox は、メールの送信者属性、返信チェーン深度、添付ファイルの種類、過去の開封率などを複合的に分析し、ミリ秒単位で優先度スコアを算出するエンジンを搭載しています。従来のルールベースフィルタでは対応できない、コンテキスト依存の重要度判定を実現します。特に、プロジェクトマネージャーやエグゼクティブ向けに設計されており、受信トレイの整理時間を平均 73% 削減します。

### コア機能

- **動的優先度スコアリングエンジン**: 各メールに対し、`priority_score`（0.0～1.0）をリアルタイム算出。`sender_weight`（0.4）、`thread_depth_weight`（0.3）、`attachment_importance`（0.2）、`recency_factor`（0.1）の 4 軸で構成され、ユーザーの開封行動を学習して重みを自動調整します。PostgreSQL の`pg_stat_statements`を用いてクエリパフォーマンスを監視し、スコアリング処理は平均 12 ミリ秒以内に完了します。

- **マルチバケットキューイングシステム**: スコアに応じてメールを 5 つのバケット（`critical`、`high`、`normal`、`low`、`archive`）に自動分類。各バケットは Redis 上のソート済みセットとして管理され、`ZRANGEBYSCORE` を用いてミリ秒単位で取得可能。バケット間の閾値はユーザーが`critical_threshold`（デフォルト 0.85）や`archive_threshold`（デフォルト 0.15）としてカスタマイズ可能です。

- **コンテキストアウェアスヌーズ機能**: メールを「後で見る」指定する際、単なる時間指定ではなく、カレンダー上の予定（Google Calendar API 連携）やプロジェクトの締切（Jira 連携）を考慮した最適な再表示時刻を提案。`snooze_duration` は秒単位で指定可能で、例えば「次のミーティング終了後」を選択すると、API が会議の終了時刻を取得し、自動的に`resurface_at` タイムスタンプを設定します。

### 使い方

1. **初期セットアップとスコアリングパラメータの調整**  
   `focusinbox init --db-host localhost --db-port 5432 --db-name focusinbox` を実行し、PostgreSQL と Redis への接続を確立します。次に、`config.yaml` 内の `priority_weights` セクションで、`sender_weight: 0.4`、`thread_depth_weight: 0.3` などの初期値を設定。`focusinbox calibrate --sample-size 500` を実行すると、過去 500 件のメールをサンプルとして機械学習モデルが最適な重みを自動計算し、`calibrated_weights.json` に出力されます。

2. **バケット閾値のカスタマイズとルールベース例外の追加**  
   `focusinbox bucket --set critical-threshold 0.85 --set archive-threshold 0.15` で閾値を調整。さらに、特定の送信者（`from:ceo@company.com`）や件名にキーワード（`subject:*urgent*`）を含むメールを強制的に `critical` バケットに格納する例外ルールを追加するには、`focusinbox rule add --condition "from:ceo@company.com" --bucket critical` を実行。ルールは PostgreSQL の `rules` テーブルに保存され、`rule_id` で管理されます。

3. **スヌーズ機能の活用とパフォーマンスモニタリング**  
   特定のメールをスヌーズするには、`focusinbox snooze --email-id 12345 --until "next-meeting"` を実行。これにより、Google Calendar API が次の会議の終了時刻を取得し、`resurface_at` が自動設定されます。パフォーマンスを確認するには、`focusinbox stats --interval 1h` で直近 1 時間のスコアリング処理時間（平均、P95、P99）とバケット別メール件数を表示。`pg_stat_activity` を監視し、スロークエリがあれば `EXPLAIN ANALYZE` でチューニングします。

### ベストプラクティス

- **段階的な閾値調整と A/B テストの実施**  
  初期設定では `critical_threshold` を 0.85 に設定し、1 週間運用した後、実際に `critical` バケットに振り分けられたメールのうち、ユーザーが「重要でなかった」とマークした割合を追跡します。その割合が 10% を超える場合は、`focusinbox calibrate --feedback` を実行し、ユーザーのフィードバックを学習データとしてモデルを再調整します。本番環境に適用する前に、`--dry-run` フラグを用いて影響範囲をシミュレーションすることを推奨します。

- **バケット間の自動アーカイブポリシーの設定**  
  `archive` バケットに 30 日以上滞留しているメールは、`focusinbox archive --force --older-than 30d` で自動削除または別ストレージに退避します。ただし、法的保存義務のあるメールは、`focusinbox rule add --condition "has-attachment:legal" --bucket legal-hold` のように専用バケットに隔離し、削除対象から除外します。このポリシーは `cron` ジョブで毎日 03:00 JST に実行されるよう設定します。
