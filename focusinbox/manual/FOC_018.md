# focusinbox - FOC_018

## Overview

focusinbox is an email triage and priority engine designed for high-volume inboxes. It eliminates the noise by applying deterministic, rule-based filtering combined with lightweight NLP to surface only the messages that demand immediate action or decision-making. The system is built for professionals managing 200+ daily emails across multiple accounts.

## Detailed Content

### What is focusinbox?

focusinbox is a self-hosted email prioritization framework that ingests IMAP streams, applies a multi-layered scoring model (sender reputation, thread depth, keyword urgency, and attachment type), and outputs a ranked, actionable queue. It integrates with PostgreSQL for persistent state and supports sub-millisecond classification through an in-memory cache layer.

### Core Features

- **Dynamic Priority Scoring Engine**  
  Each email is scored against a configurable matrix of parameters: `sender_trust_score` (0.0–1.0), `thread_activity_delta` (hours since last reply), `keyword_urgency_vector` (predefined terms like "ASAP", "breach", "deadline"), and `attachment_risk_factor` (PDF/DOCX vs. executable). The system computes a composite `priority_index` in under 50 microseconds per message using a Rust-based classification pipeline.

- **Multi-Account IMAP Aggregation with Deduplication**  
  Connects to up to 10 IMAP endpoints simultaneously (Gmail, Outlook, custom servers). Uses `message_id` and `received_timestamp` (nanosecond precision) to deduplicate cross-account forwards. The aggregation layer supports OAuth2 and app-specific passwords, with automatic reconnection on TLS failure (retry interval: 30 seconds, exponential backoff).

- **Time-Series Inbox Analytics & SLO Monitoring**  
  Logs every classification decision to a PostgreSQL time-series table (`focusinbox.decisions`). Exposes a REST API at `/api/v1/metrics` returning `p50`, `p95`, and `p99` response times for the scoring engine. Supports real-time dashboards via Prometheus integration, with alerts triggered when `p95_latency_ms` exceeds 200ms.

### How to Use

1. **Initialize the Configuration File**  
   Create `config/focusinbox.yaml` with your IMAP credentials and scoring weights. Example:  
   ```yaml
   accounts:
     - host: imap.gmail.com
       port: 993
       user: user@gmail.com
       password: ${GMAIL_APP_PASSWORD}
   scoring:
     sender_trust_weight: 0.4
     keyword_urgency_weight: 0.3
     thread_activity_weight: 0.2
     attachment_risk_weight: 0.1
   ```
   Run `focusinbox validate --config config/focusinbox.yaml` to verify connectivity and parse the YAML schema.

2. **Execute a Dry-Run Classification**  
   Launch the engine in dry-run mode to inspect scoring without modifying any email state:  
   ```bash
   focusinbox classify --dry-run --limit 500 --output results.json
   ```  
   This processes the 500 most recent emails per account, writes the `priority_index` and `decision_reason` to `results.json`, and prints a summary table showing the top 10 items with their scores.

3. **Deploy the Persistent Queue**  
   Start the daemon with PostgreSQL backend to continuously monitor and reclassify new emails:  
   ```bash
   focusinbox daemon --config config/focusinbox.yaml --db postgresql://user:pass@localhost:5432/focusinbox
   ```  
   The daemon polls every 60 seconds, updates the `focusinbox.queue` table, and exposes an HTTP endpoint at `http://localhost:8080/api/v1/queue` for external clients (e.g., a custom Slack bot or CLI) to fetch the current top-20 priority items.

### Best Practices

- **Tune Sender Trust Incrementally**  
  Start with a conservative `sender_trust_weight` (≤0.3) and a low `keyword_urgency_weight` (≤0.2). Run dry-runs against historical data for 7 days. Analyze false positives (e.g., marketing emails scored as urgent) and adjust the keyword dictionary by removing terms like "limited time" or "exclusive offer". Recalibrate the `sender_trust_score` by seeding a whitelist of known VIP domains (e.g., `*.board.com`, `*.legal.com`).

- **Isolate Classification from Delivery**  
  Never let focusinbox automatically move or delete emails. Instead, write the `priority_index` to a custom IMAP flag (e.g., `$PriorityHigh`) and let your email client sort by that flag. This prevents accidental loss and allows manual override. Use the `/api/v1/metrics` endpoint to monitor classification speed; if `p99_latency_ms` exceeds 500ms, reduce the `thread_activity_weight` or increase the cache TTL for sender reputation lookups.
