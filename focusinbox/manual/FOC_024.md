### What is focusinbox?

focusinbox is a professional tool that solves the critical problem of email overload by implementing a zero-inbox workflow based on real-time priority scoring, automated triage, and cross-platform synchronization. It is designed for power users who manage 200+ emails daily and require sub-second decision latency on message importance.

### Core Features

- **Adaptive Priority Scoring Engine (APSE)**: Dynamically assigns a `priority_score` (0.0–1.0, updated every 500ms) to each incoming email by analyzing sender reputation, thread depth, keyword frequency, and historical open rates. The engine uses a lightweight ML model (XGBoost) trained on your past 30 days of inbox behavior, with a cold-start fallback to a heuristic rule set. Scores are recalculated in-memory using Redis Streams, ensuring < 50ms latency per message.

- **Contextual Snooze & Batch Resurfacing**: Allows per-email `snooze_until` timestamps with microsecond precision (stored as Unix nanoseconds in PostgreSQL). When an email resurfaces, the system re-evaluates its priority against the current inbox state and can automatically merge it into a "batch digest" if the new priority falls below a configurable threshold (default: 0.3). The batch digest is delivered as a single compressed Markdown email at 9:00 AM and 3:00 PM local time, reducing inbox noise by up to 60%.

- **Cross-Platform Action Mirroring**: Enables real-time synchronization of email actions (archive, flag, delete, move to folder) across Gmail, Outlook, and ProtonMail via their respective IMAP IDLE and API webhooks. The system maintains a conflict-resolution log in a separate `action_sync_log` table, with automatic rollback if a mirror operation fails within 2 seconds. The sync latency is guaranteed under 1.2 seconds for 95% of operations, measured end-to-end from source action to destination inbox update.

### How to Use

**Step 1: Initial Inbox Profiling & Model Training**
1. Run the `focusinbox init` CLI command with your email provider credentials (OAuth 2.0 tokens, stored encrypted via AES-256-GCM).
2. The system downloads the last 30 days of email metadata (subject, sender, timestamp, read status, folder path) into a local SQLite cache.
3. Execute `focusinbox train --model-type xgboost --max-depth 6 --learning-rate 0.1` to train the priority model. The training process outputs a `model_accuracy` metric (target: > 85%) and a `feature_importance` JSON file. If accuracy is below 80%, the system prompts you to manually label 50 emails as "critical" or "low" via an interactive TUI.

**Step 2: Configure Priority Thresholds & Batch Rules**
1. Edit the YAML configuration file at `~/.focusinbox/config.yaml`. Set `priority_critical_threshold: 0.7` and `priority_low_threshold: 0.25`.
2. Define a batch rule under `batch_rules:` using a cron-like syntax. For example:
   ```yaml
   batch_rules:
     - name: "daily_digest"
       schedule: "0 9,15 * * 1-5"
       min_priority_to_batch: 0.3
       max_emails_per_batch: 15
   ```
3. Validate the configuration with `focusinbox validate --config ~/.focusinbox/config.yaml`. The validator checks for syntax errors, schedule conflicts, and threshold overlaps.

**Step 3: Enable Real-Time Sync & Start Monitoring**
1. Start the daemon: `focusinbox daemon --sync-interval 500ms --log-level info`. The daemon opens three persistent IMAP IDLE connections (one per provider) and begins streaming email events into Redis Streams.
2. Open the web dashboard at `http://localhost:8080`. The dashboard shows a live-updating table of the top 20 emails by `priority_score`, with columns for sender, subject, score, and `snooze_until`.
3. To test the sync, archive an email in Gmail. Within 1.2 seconds, the same email should be archived in Outlook and ProtonMail. Check the `action_sync_log` in PostgreSQL for any conflicts: `SELECT * FROM action_sync_log WHERE status = 'rolled_back';`.

### Best Practices

- **Practice 1: Periodic Model Retraining for Drift Prevention**
  Email behavior changes over time (e.g., new projects, seasonal campaigns). Schedule a weekly retraining job using `cron` that runs `focusinbox train --incremental --window-days 7`. This retrains the XGBoost model on the latest 7 days of data while retaining the previous model's tree structure, preventing catastrophic forgetting. Monitor the `model_accuracy` metric; if it drops below 80%, trigger a full retrain with `--full-retrain`.

- **Practice 2: Granular Snooze Strategy for High-Volume Inboxes**
  For inboxes receiving > 500 emails/day, avoid snoozing individual emails. Instead, use the "batch resurfacing" feature with a `min_priority_to_batch` of 0.4. Set the batch schedule to three times per day (e.g., 8 AM, 12 PM, 5 PM). This reduces the cognitive load of checking snoozed emails manually. Additionally, enable the `auto_merge_duplicates` flag in the config to collapse email threads with identical subject lines into a single batch entry, saving an average of 12% inbox space per month.
