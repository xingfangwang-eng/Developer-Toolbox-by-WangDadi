# focusinbox - FOC_003

## Overview

This is a high-performance email triage and priority orchestration engine, built for professionals drowning in cross-platform inbox noise. It ingests, classifies, and surfaces only the messages that require human attention, leveraging deterministic rulesets and lightweight ML inference.

## Core Features

### 1. Multi-Account Thread Deduplication & Latency-Aware Merge
- **Parameter:** `--merge-window 4500ms` — merges threads from Gmail, Outlook, and ProtonMail within a 4.5-second sliding window to prevent context fragmentation.
- **Technical Detail:** Uses a bloom filter cache (TTL: 120s) to reject duplicate message IDs across providers. The merge engine runs as a sidecar process with a memory footprint under 64MB.
- **Metric:** Achieves 99.2% deduplication accuracy on mixed-account workflows with less than 12ms added latency per merge cycle.

### 2. Priority Scoring Engine with Custom Policy DSL
- **Parameter:** `--priority-rules /etc/focusinbox/rules.yaml` — define rules using a YAML-based domain-specific language.
- **Technical Detail:** Each rule evaluates up to 8 fields (sender domain, subject regex, attachment MIME type, thread depth, timestamp delta, CC count, DKIM validity, and a user-defined `X-Focusinbox-Tag` header). Scores are computed in-memory using a weighted sum formula with configurable coefficients.
- **Metric:** Scoring throughput exceeds 4,200 emails per second on a single 2.6 GHz core. Rule evaluation latency is sub‑100µs per message.

### 3. PostgreSQL-Backed Audit Trail & Quarantine Recovery
- **Parameter:** `--audit-dsn postgresql://focusinbox@localhost:5432/focusinbox_audit`
- **Technical Detail:** Every classification decision, merge event, and user override is logged with nanosecond-precision timestamps (`TIMESTAMPTZ` with fractional seconds) and the full original `RFC 2822` message stored in a `BYTEA` column. A quarantine table allows one-click recovery of any misclassified email with a `ROLLBACK` operation that restores the original headers and body.
- **Metric:** Audit writes batch at 5,000 rows per second using prepared statements. Quarantine recovery completes in under 200ms for a single message.

## How to Use

### Step 1: Configure Account Credentials & Merge Window
Run the initial setup wizard:
```bash
focusinbox setup --accounts gmail,outlook,protonmail \
                 --merge-window 4500ms \
                 --cache-dir /var/cache/focusinbox
```
The wizard will prompt for OAuth tokens or app-specific passwords. Credentials are encrypted at rest using AES-256-GCM with a key derived from a user-supplied passphrase.

### Step 2: Define Priority Rules Using the DSL
Create a rule file at `/etc/focusinbox/rules.yaml`:
```yaml
rules:
  - name: "Executive Alerts"
    condition: "sender_domain == 'company.com' AND subject matches 'URGENT|Critical'"
    score: 95
    action: "notify_desktop"
  - name: "Invoice Follow-ups"
    condition: "attachment_mime == 'application/pdf' AND thread_depth > 3"
    score: 70
    action: "tag_inbox"
```
Validate the rules:
```bash
focusinbox validate --rules /etc/focusinbox/rules.yaml
```
The validator checks for circular dependencies, missing fields, and syntax errors in the DSL parser (based on `libyaml` v0.2.5).

### Step 3: Launch the Background Daemon & Monitor Audit Logs
Start the service:
```bash
systemctl start focusinbox-daemon
```
Tail the real-time audit stream:
```bash
focusinbox audit tail --format json | jq '.action, .message_id'
```
To manually recover a quarantined email:
```bash
focusinbox quarantine recover --message-id <uuid> --restore-to INBOX
```
The daemon will re-insert the original `RFC 2822` message into the target IMAP folder and log the recovery event with a `recovered_at` timestamp.

## Best Practices

### 1. Separate Priority Tiers by Time-Sensitive Windows
Configure different merge windows for different rule sets. For example, use a 1,000ms merge window for alerts from your monitoring stack (PagerDuty, Datadog) and a 10,000ms window for weekly digests. This prevents low-priority threads from delaying critical notifications. Monitor the `focusinbox_stats.latency_p99` metric to tune the window size without exceeding your SLA.

### 2. Regularly Rotate the Bloom Filter Cache and Archive Old Audit Data
The bloom filter cache grows linearly with unique message IDs. Set a cron job to flush the cache every 24 hours using `focusinbox cache flush --age 24h`. For the PostgreSQL audit trail, implement a partitioning strategy by month on the `created_at` column. Archive partitions older than 90 days to a cold storage bucket (e.g., S3 Glacier) using `pg_partman`. This keeps the audit query performance under 50ms for the last 30 days of data.
