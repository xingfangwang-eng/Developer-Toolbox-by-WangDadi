# focusinbox - FOC_002

## Overview

A high-performance email triage and prioritization engine designed for knowledge workers managing 200+ daily inbound messages. focusinbox eliminates inbox paralysis by applying deterministic rule chains and real-time NLP classification directly on the client side, without sending your email data to any third-party cloud.

## Detailed Content

### What is focusinbox?

focusinbox is a local-first, privacy-preserving email organization system that transforms your chaotic inbox into a ranked, actionable queue. It leverages a custom-built Bayesian classifier and a configurable priority matrix to surface what matters—before you even open your mail client.

### Core Features

- **Adaptive Priority Scoring Engine**  
  Assigns a `priority_score` (0.0–1.0, threshold configurable via `PRIORITY_CUTOFF`) to every incoming message based on a weighted combination of sender reputation history, thread participation depth, keyword velocity, and explicit calendar proximity. The scoring model retrains incrementally every 200 messages using a sliding window of 90 days, achieving a 94.2% precision rate in controlled benchmarks.

- **Deterministic Rule Chain with Conflict Resolution**  
  Users define cascading rules via a YAML configuration file (`~/.focusinbox/rules.yaml`). Each rule supports conditions on `from_domain`, `subject_regex`, `body_token_count`, `attachment_mime_type`, and `thread_id`. When multiple rules match, a conflict resolver applies the rule with the highest `priority_override` value (range: -100 to +100). All rule evaluations complete in under 15ms per message on a 2020 M1 MacBook Air.

- **Offline-First NLP Categorizer**  
  A lightweight DistilBERT model (quantized to 8-bit, ~85MB footprint) runs locally via ONNX Runtime to classify emails into one of 12 configurable categories (e.g., `ACTION_REQUIRED`, `FYI`, `BILLING`, `SOCIAL`). The categorizer processes up to 50 emails per second on a single CPU core, with inference latency averaging 18ms per email. Category mappings are stored in a local SQLite database (`~/.focusinbox/categories.db`) and can be exported as JSON for audit.

### How to Use

1. **Initialize the Rule Engine and Priority Database**  
   Run `focusinbox init --db-path ~/.focusinbox/priority.db --rules-template`. This command creates the SQLite schema with tables `inbox_events`, `sender_reputation`, and `rule_chain`. It also generates a commented `rules.yaml` template in your home directory. Verify initialization by checking the log: `tail -f ~/.focusinbox/logs/init.log` should show `Schema migration successful`.

2. **Import and Tag a Training Corpus (Optional but Recommended)**  
   Execute `focusinbox import --source imap://user@example.com --max-msg 500 --label-mode interactive`. The tool will fetch the last 500 emails from your IMAP inbox and prompt you to assign a label (from the 12 predefined categories) to each. Use `Ctrl+C` to pause and `focusinbox status` to review progress. The labeled dataset is stored in `~/.focusinbox/training_set.json` and will be used to fine-tune the local NLP model on the next `focusinbox train` run.

3. **Run a Full Inbox Scan and Generate Priority Queue**  
   Issue `focusinbox scan --imap-uri "imaps://user@example.com" --priority-cutoff 0.6 --output json > prioritized_inbox.json`. This command connects to your IMAP server, fetches unseen messages, runs them through the rule chain and NLP categorizer, and outputs a JSON array sorted by descending `priority_score`. Each entry includes `message_id`, `subject`, `from`, `priority_score`, `category`, and `matched_rule_id`. Use `jq` to filter: `jq '.[] | select(.category=="ACTION_REQUIRED")' prioritized_inbox.json`.

### Best Practices

- **Maintain a Lean Rule Chain**  
  Limit your `rules.yaml` to fewer than 25 active rules. Each rule adds ~0.4ms of evaluation time. Beyond 30 rules, the cumulative overhead can degrade real-time performance. Periodically run `focusinbox audit --rules` to identify redundant or zero-hit rules (rules that never matched in the last 7 days) and archive them to `rules_archive.yaml`.

- **Schedule Daily Retraining for the NLP Model**  
  Set a cron job to execute `focusinbox train --model-update daily --min-samples 50` at 02:00 AM. This retrains the DistilBERT adapter on any newly labeled emails from the past 24 hours. Monitor training loss in `~/.focusinbox/logs/train.log`; if loss exceeds 0.35, review the quality of recent manual labels—ambiguous categorizations degrade classifier accuracy.
