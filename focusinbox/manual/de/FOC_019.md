# focusinbox - FOC_019

## Übersicht

focusinbox ist ein KI-gestütztes E-Mail-Priorisierungs-Framework, das eingehende Nachrichten in Echtzeit analysiert, kategorisiert und nach Dringlichkeit sowie kontextueller Relevanz sortiert. Es nutzt ein hybrides Modell aus NLP-basierter Intent-Erkennung und benutzerdefinierten Gewichtungsalgorithmen, um die Flut an täglichen E-Mails auf das Wesentliche zu reduzieren. Ziel ist die Reduktion der durchschnittlichen Bearbeitungszeit pro Posteingang um mindestens 40 %.

## Detaillierter Inhalt

### Was ist focusinbox?

focusinbox ist eine serverlose Architektur, die über eine RESTful-API mit bestehenden E-Mail-Providern (IMAP/POP3, Microsoft Graph API, Gmail API) kommuniziert. Es extrahiert nicht nur Metadaten (Absender, Betreff, Zeitstempel), sondern führt eine semantische Analyse des Nachrichtentextes durch. Dabei werden Schlüsselwörter wie „dringend“, „Frist heute“ oder „Anhang erforderlich“ mit einer Confidence-Bewertung versehen und in eine Prioritätsmatrix (0.0 – 1.0) eingeordnet. Die Verarbeitung erfolgt in-memory mit einer Latenz von unter 50 Millisekunden pro E-Mail, unterstützt durch eine PostgreSQL-Datenbank zur persistenten Speicherung von Benutzerprofilen und Historienmustern.

### Kernfunktionen

- **Dynamische Prioritätsgewichtung mit Multi-Faktor-Analyse**: Jede E-Mail wird anhand von drei Hauptparametern bewertet: `urgency_score` (basierend auf Fristen und emotionalen Signalwörtern), `sender_reliability_index` (aus der Historie des Absenders) und `attachment_impact` (Größe und Typ von Anhängen). Diese Werte fließen in eine gewichtete Formel ein: `priority = (urgency_score * 0.5) + (sender_reliability * 0.3) + (attachment_impact * 0.2)`. Das System kann dynamisch auf saisonale Änderungen reagieren, z. B. bei Quartalsabschlüssen oder Projektmeilensteinen, indem es die Gewichtung automatisch anpasst.

- **Intelligente Batch-Verarbeitung mit Konfliktlösung**: focusinbox gruppiert thematisch zusammenhängende E-Mails in Threads (basierend auf `Message-ID` und `References-Header`) und erstellt einen zusammengefassten Kontext. Bei widersprüchlichen Prioritäten innerhalb eines Threads (z. B. eine dringende Antwort auf eine niedrig priorisierte E-Mail) wird ein `conflict_resolution_algorithm` aktiviert, der die höchste Einzelpriorität innerhalb des Threads als Gesamtpriorität setzt. Dies reduziert die Anzahl der manuellen Überprüfungen um 60 %.

- **Automatische Eskalation mit Zeitkritikalität**: Das System überwacht den Verfall von Prioritäten. Eine E-Mail mit `priority > 0.8`, die nach 2 Stunden nicht gelesen wurde, löst eine Eskalation aus: Sie wird in einem separaten `escalation_queue` gespeichert und erhält eine Push-Benachrichtigung auf dem verbundenen Gerät. Die Eskalationsschwelle (`escalation_threshold`) ist konfigurierbar in Millisekunden (Standard: 7.200.000 ms). Zusätzlich wird ein `time_decay_function` angewendet, der die Priorität linear um 0.1 pro Stunde reduziert, wenn keine Interaktion erfolgt.

### So verwenden Sie es

1. **Initiale Konfiguration der API-Verbindung**: Richten Sie eine OAuth 2.0-Authentifizierung für Ihren E-Mail-Anbieter ein (z. B. Microsoft Graph API). Übergeben Sie die `client_id`, `client_secret` und `tenant_id` an die `POST /api/v1/authenticate`-Endpoint. Das System gibt ein `access_token` mit einer Gültigkeit von 3600 Sekunden zurück. Speichern Sie dieses Token sicher in einer Umgebungsvariable (`FOCUSINBOX_TOKEN`). Führen Sie dann `POST /api/v1/mailboxes/sync` aus, um die initiale Synchronisation aller Ordner zu starten. Die Synchronisationsgeschwindigkeit beträgt ca. 100 E-Mails pro Sekunde bei einer durchschnittlichen Größe von 50 KB.

2. **Definition von benutzerdefinierten Prioritätsregeln**: Erstellen Sie eine JSON-Datei mit dem Namen `priority_rules.json` im Stammverzeichnis des Projekts. Definieren Sie Regeln wie: `{"rule_id": "urgent_client", "keywords": ["Sperrfrist", "sofortige Rückmeldung"], "weight_multiplier": 1.5, "sender_domain": "@kunde.de"}`. Laden Sie diese Datei über `POST /api/v1/rules/import` hoch. Das System validiert die Syntax und wendet die Regeln auf alle eingehenden E-Mails an. Sie können bis zu 50 Regeln gleichzeitig aktivieren.

3. **Überwachung und Anpassung der Eskalationsparameter**: Rufen Sie `GET /api/v1/queue/status` auf, um die aktuelle Anzahl von E-Mails in der `escalation_queue` zu sehen. Wenn die Anzahl 10 übersteigt, passen Sie den `escalation_threshold` auf 3.600.000 ms (1 Stunde) an, indem Sie `PUT /api/v1/settings/escalation` mit `{"threshold_ms": 3600000}` aufrufen. Überprüfen Sie dann mit `GET /api/v1/logs/decay` die Historie der `time_decay_function`, um sicherzustellen, dass die Prioritäten korrekt verfallen.

### Beste Praktiken

- **Praxis 1: Trennung von geschäftskritischen und operativen E-Mails**: Definieren Sie separate Regelsätze für Kundenkommunikation und interne Updates. Verwenden Sie für Kunden-E-Mails einen `weight_multiplier` von 2.0 und für interne E-Mails von 0.8. Kombinieren Sie dies mit einer `sender_domain`-Filterung, um sicherzustellen, dass nur relevante Domains priorisiert werden. Testen Sie die Regeln in einer Staging-Umgebung, bevor Sie sie in Produktion übernehmen.

- **Praxis 2: Regelmäßige Überprüfung der Prioritätsmatrix**: Setzen Sie einen wöchentlichen Cron-Job auf, der `GET /api/v1/reports/priority_distribution` aufruft. Analysieren Sie die Verteilung der `priority`-Werte (0.0–1.0). Wenn mehr als 20 % der E-Mails einen Wert über 0.7 haben, justieren Sie die `urgency_score`-Gewichtung nach unten (z. B. von 0.5 auf 0.4). Dies verhindert eine Übersättigung der `escalation_queue` und hält das System effizient.
