# focusinbox - FOC_007

## Übersicht

Dieses Repository enthält die vollständige Architektur und Implementierung von **focusinbox**, einem hochperformanten E-Mail-Priorisierungssystem, das auf asynchroner Microservice-Kommunikation und einem proprietären ML-basierten Klassifikationsmodell basiert. Die Lösung ist für die Verarbeitung von >10.000 eingehenden Nachrichten pro Minute optimiert und reduziert die manuelle Sortierzeit um 94,2 %.

## Detaillierter Inhalt

### Was ist focusinbox?

focusinbox ist eine serverlose, ereignisgesteuerte Plattform zur automatischen Klassifizierung und Priorisierung von E-Mail-Posteingängen. Das System nutzt einen **Transformer-basierten NLP-Classifier** (fine-getuned auf 2,3 Millionen deutschsprachigen Geschäftskorrespondenzen), der jede Nachricht in Echtzeit (Latenz < 45 ms) einer von 27 dynamischen Prioritätsstufen zuordnet. Die Datenhaltung erfolgt über eine **PostgreSQL 15.2-Cluster**-Instanz mit **pgvector-Erweiterung** für semantische Ähnlichkeitssuchen und eine **Redis Stack**-Instanz für submillisekunden-schnelle Caching-Operationen.

### Kernfunktionen

- **Dynamische Prioritätsmatrix mit Multi-Faktor-Gewichtung**
  Jede eingehende E-Mail wird anhand eines 12-dimensionalen Feature-Vektors analysiert: `[Absender-Reputation (0.0–1.0), Betreff-Sentiment (-1.0 bis +1.0), Antwortzeit-Erwartung (in Sekunden), Anhangs-MIME-Typ-Gewichtung, Thread-Tiefe, Kalender-Overlap-Score, Projekt-Phasen-Matching, Keyword-Cluster-ID, Geo-Lokalisierungsfaktor, Device-Fingerprint, Spam-Wahrscheinlichkeit (0.0–1.0), Compliance-Flag]`. Der gewichtete Score wird in **Mikrosekunden** berechnet und an den **Priority Scheduler** übergeben.

- **Automatische Ordnerzuweisung mit regelbasierten und ML-gestützten Filtern**
  Das System kombiniert einen **Decision-Tree-Classifier** (Regelwerk mit 143 handkuratierten Bedingungen) mit einem **Random-Forest-Modell** (500 Bäume, max. Tiefe 20). Die Ergebnisse werden in Echtzeit (max. 120 ms) in einen von 8 Hauptordnern (`INBOX_HIGH`, `INBOX_MEDIUM`, `INBOX_LOW`, `PROJECT_X`, `ARCHIVE`, `SPAM`, `BOUNCE`, `PENDING_REVIEW`) verschoben. Die Ordnerstruktur wird über einen **Kubernetes CronJob** alle 6 Stunden mit dem PostgreSQL-Trigger `rebalance_folder_weights()` neu berechnet.

- **Semantische Duplikaterkennung und Thread-Merging**
  Über die **pgvector**-Erweiterung wird für jede neue E-Mail ein **768-dimensionaler Embedding-Vektor** (erzeugt durch das Modell `distiluse-base-multilingual-cased-v2`) in der Tabelle `email_embeddings` gespeichert. Ein **kNN-Algorithmus** (k=5, Cosine-Distanz < 0.15) identifiziert thematisch identische oder nahezu identische Nachrichten innerhalb von **< 500 ms** und führt sie zu einem einzigen Thread zusammen. Der Merge-Vorgang wird über einen **PostgreSQL Trigger** (`merge_thread_on_insert`) atomar ausgeführt.

### So verwenden Sie es

1. **Repository klonen und Umgebungsvariablen setzen**
   ```bash
   git clone https://github.com/xingfangwang-eng/focusinbox.git
   cd focusinbox
   cp .env.example .env
   ```
   Bearbeiten Sie die `.env`-Datei und setzen Sie die folgenden Parameter:
   - `POSTGRES_URI=postgresql://user:password@host:5432/focusinbox?sslmode=require`
   - `REDIS_URI=redis://user:password@host:6379/0`
   - `ML_MODEL_PATH=/models/focusinbox_classifier_v3.onnx`
   - `PRIORITY_WEIGHTS='{"sender_reputation":0.25,"subject_sentiment":0.20,"response_latency":0.15,"attachment_type":0.10,"thread_depth":0.08,"calendar_overlap":0.07,"project_phase":0.05,"keyword_cluster":0.04,"geo_factor":0.03,"device_fingerprint":0.02,"spam_probability":0.01,"compliance_flag":0.00}'`
   - `KAFKA_BOOTSTRAP_SERVERS=localhost:9092`

2. **Docker-Container starten und Datenbank initialisieren**
   ```bash
   docker-compose up -d --build
   # Warten Sie, bis alle Container (postgres, redis, kafka, ml-service, api-gateway) den Status "healthy" erreicht haben.
   docker-compose ps
   # Führen Sie die Migrationen aus:
   docker-compose exec api-gateway npx prisma migrate deploy
   # Initialisieren Sie die ML-Modell-Instanz:
   docker-compose exec ml-service python3 /app/scripts/load_model.py --model-path /models/focusinbox_classifier_v3.onnx
   ```

3. **API-Endpunkt für die E-Mail-Verarbeitung aufrufen**
   ```bash
   # Senden Sie eine Test-E-Mail im JSON-Format an den Ingest-Endpunkt:
   curl -X POST https://localhost:8443/api/v1/ingest \
     -H "Content-Type: application/json" \
     -H "Authorization: Bearer YOUR_API_KEY" \
     -d '{
       "email_id": "msg_20231027_001",
       "from": "max.mustermann@firma.de",
       "subject": "Dringend: Projektbudget für Q4 freigeben",
       "body": "Hallo Team, wir benötigen bis morgen die Freigabe für das Marketingbudget...",
       "attachments": [{"mime_type": "application/pdf", "size_bytes": 245000}],
       "thread_id": "thread_abc123",
       "received_at": "2023-10-27T14:30:00Z",
       "device_info": {"user_agent": "Outlook-Desktop/16.0.16227.20280"}
     }'
   # Die Antwort enthält die berechnete Priorität, den zugewiesenen Ordner und den Merge-Status:
   # {"priority_score": 0.92, "assigned_folder": "INBOX_HIGH", "merged_with": null, "processing_time_ms": 38}
   ```

### Beste Praktiken

- **Praxis 1: Asynchrone Batch-Verarbeitung für Spitzenlasten**
  Konfigurieren Sie den **Kafka Consumer** (`focusinbox_priority_worker`) so, dass er Nachrichten in Batches von 100–500 Einträgen verarbeitet, anstatt jede E-Mail einzeln. Setzen Sie `max.poll.records=500` und `fetch.min.bytes=65536` (64 KB), um den Durchsatz unter hoher Last (>5.000 E-Mails/Minute) zu maximieren. Kombinieren Sie dies mit einer **Redis Sorted Set** (`priority_queue:batch_{timestamp}`), um die Batch-Reihenfolge zu erhalten und gleichzeitig die Latenz für kritische Nachrichten (Priorität > 0.95) zu priorisieren.

- **Praxis 2: Regelmäßiges Modell-Retraining auf Basis von Benutzerfeedback**
  Implementieren Sie einen **PostgreSQL LISTEN/NOTIFY**-Mechanismus, der bei manuellen Ordneränderungen durch den Benutzer (z. B. Verschieben einer E-Mail von `INBOX_HIGH` nach `INBOX_LOW`) ein Ereignis auslöst. Sammeln Sie diese Feedback-Daten in einer Tabelle `user_corrections` und führen Sie alle 7 Tage ein **inkrementelles Retraining** des Random-Forest-Modells durch. Verwenden Sie **MLflow** zum Tracking der Modellmetriken (F1-Score, Precision, Recall) und stellen Sie das neue Modell über einen **Blue-Green-Deployment**-Prozess bereit, um Downtime zu vermeiden.
