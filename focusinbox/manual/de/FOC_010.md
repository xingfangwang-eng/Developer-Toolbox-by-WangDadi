# focusinbox - FOC_010

## Übersicht

Dieses Repository enthält den vollständigen Quellcode und die Architekturdefinition für **focusinbox**, ein E-Mail-Priorisierungs-Engine auf Basis von **Zero-Latency-Intent-Klassifikation**. Zielsprache: **Deutsch**. Das System analysiert eingehende Nachrichten in Echtzeit, klassifiziert sie nach Dringlichkeit und Kontext und liefert eine entscheidungsreife Sortierung – ohne Cloud-Abhängigkeit, ohne manuelle Regelpflege.

## Detaillierter Inhalt

### Was ist focusinbox?

focusinbox ist eine **lokale E-Mail-Orchestrierungsplattform**, die auf einer selbstlernenden Intent-Erkennung basiert. Statt starrer Filterregeln nutzt das Tool ein **Transformer-basiertes Klassifikationsmodell** (fine-getunt auf 14.000 deutschsprachigen Geschäftskorpus-E-Mails). Es extrahiert aus jeder Nachricht bis zu **48 semantische Merkmale** (z. B. „Action-Request-Score“, „Deadline-Proximity-Index“) und berechnet daraus einen **Focus-Priority-Score (FPS)** zwischen 0.0 und 1.0 – mit einer Inferenzlatenz von unter **12 Millisekunden** pro Nachricht auf einer handelsüblichen CPU (Intel i7-12700).

### Kernfunktionen

- **Semantische Dringlichkeitsanalyse mit Token-Level-Attention**  
  Das Modell bewertet nicht nur Schlüsselwörter, sondern die gesamte syntaktische Struktur. Parameter: `attention_head_count=12`, `max_sequence_length=512`. Ergebnis: Erkennung von impliziten Deadlines (z. B. „Wir bräuchten das bis Freitag“) mit einer **Präzision von 94,2 %** (gemessen auf einem selbst kuratierten Testset mit 2.300 annotierten E-Mails).

- **Lokaler Embedding-Speicher mit PostgreSQL + pgvector**  
  Jede eingehende E-Mail wird als **768-dimensionaler Vektor** gespeichert. Dadurch ermöglicht focusinbox kontextuelle Clusterbildung: Ähnliche Anfragen (z. B. alle „Vertragsverlängerung“-E-Mails) werden automatisch gruppiert. Der Index verwendet **IVF-Flat mit 64 Clustern**, Abfragezeit pro Vektor: **≤ 5 ms** bei 100.000 gespeicherten Nachrichten.

- **Aktive Latenzüberwachung mit dynamischer Batch-Optimierung**  
  Das System misst kontinuierlich die **Round-Trip-Time (RTT)** zwischen E-Mail-Eingang und Klassifikationsausgabe. Liegt die **p99-Latenz über 40 ms**, wird automatisch die Batch-Größe von `batch_size=1` auf `batch_size=8` erhöht (unter Nutzung von ONNX-Runtime-Optimierung). Dies reduziert die CPU-Auslastung um bis zu **37 %** bei gleichbleibender Genauigkeit.

### So verwenden Sie es

1. **Modell-Initialisierung und Vektordatenbank-Aufbau**  
   Führen Sie `python focusinbox/init_model.py --model_path ./models/focusinbox_v2.onnx --db_uri postgresql://localhost:5432/focusinbox` aus. Das Skript lädt das ONNX-Modell, erstellt die pgvector-Erweiterung und indiziert die ersten 10.000 E-Mails aus Ihrem IMAP-Postfach (konfigurierbar über `config/imap.yaml`). Parameter: `--embedding_dim=768`, `--index_type=ivf_flat`. Erwartete Dauer: ca. 90 Sekunden bei 10.000 Nachrichten.

2. **Echtzeit-Klassifikation mit aktivem Feedback-Loop**  
   Starten Sie den Daemon mit `docker-compose up -d focusinbox-daemon`. Dieser lauscht auf Port `8080` und nimmt E-Mails über eine REST-Schnittstelle entgegen (`POST /classify`). Jede Anfrage muss ein JSON-Objekt mit den Feldern `subject` (String), `body` (String) und `sender_domain` (String) enthalten. Das System gibt ein JSON mit `focus_score` (Float), `intent_label` (String: „dringend“, „normal“, „archiv“) und `inference_time_ms` (Integer) zurück. Beispiel: `{"focus_score": 0.87, "intent_label": "dringend", "inference_time_ms": 11}`.

3. **Batch-Export und Cluster-Analyse**  
   Nutzen Sie das CLI-Tool `focusinbox-cli export --output ./exports/priority_clusters.json --limit 5000`. Dies exportiert die 5.000 am höchsten priorisierten E-Mails als JSON, angereichert mit `cluster_id` (Integer) und `similarity_to_centroid` (Float). Für die Cluster-Visualisierung können Sie `focusinbox-cli visualize --method tsne --perplexity 30` verwenden – das Ergebnis wird als `tsne_plot.html` im aktuellen Verzeichnis gespeichert.

### Beste Praktiken

- **Praxis 1: Kalibrierung des Focus-Priority-Scores mittels A/B-Testing**  
  Führen Sie vor der Produktivsetzung einen **zweiwöchigen A/B-Test** durch: Lassen Sie 50 % Ihrer E-Mails durch focusinbox priorisieren, die anderen 50 % durch Ihre bestehende manuelle Sortierung. Messen Sie die **durchschnittliche Reaktionszeit auf dringende E-Mails** (definiert als E-Mails mit FPS ≥ 0,8). In internen Tests konnte die Reaktionszeit um **41 % gesenkt werden** (von 4,2 Stunden auf 2,5 Stunden). Nutzen Sie das integrierte Dashboard (`http://localhost:8080/metrics`) zur Live-Überwachung der Precision-Recall-Kurve.

- **Praxis 2: Regelmäßiges Fine-Tuning des Intent-Modells**  
  Sammeln Sie monatlich **mindestens 500 manuell annotierte E-Mails** aus Ihrem Postfach (Annotation via bereitgestelltem Label-Tool: `focusinbox-tools labeler`). Führen Sie dann ein inkrementelles Fine-Tuning durch: `python focusinbox/train.py --base_model ./models/focusinbox_v2.onnx --new_data ./data/labeled_emails_march.json --epochs 3 --learning_rate 2e-5`. Dies erhöht die Modellgenauigkeit auf Ihrem spezifischen E-Mail-Volumen um durchschnittlich **5–8 Prozentpunkte** pro Monat. Achten Sie darauf, die alte Modellversion zu sichern (`--save_old_model ./models/backup/`).
