# focusinbox - FOC_009

## Übersicht

focusinbox ist eine hochperformante E-Mail-Priorisierungs-Engine, die auf einem hybriden Bayes-Transformer-Modell basiert. Entwickelt für Wissensarbeiter mit >500 eingehenden Nachrichten pro Tag, reduziert es die kognitive Last durch echtzeitfähige Intent-Klassifikation und kontextbezogene Warteschlangensteuerung. Zielsprache: Deutsch (de).

## Detaillierter Inhalt

### Was ist focusinbox?

focusinbox ist eine serverlose Architektur, die eingehende E-Mails mittels eines mehrstufigen NLP-Pipelines (Tokenisierung, POS-Tagging, Dependency Parsing) in eine von 12 priorisierten Kategorien einordnet. Die Latenz pro Nachricht beträgt < 50 ms bei einer Trefferquote von 94,2 % im A/B-Test gegen einen reinen Regelwerk-Ansatz. Die Integration erfolgt über IMAP-IDLE-Push oder MQTT-Bridge.

### Kernfunktionen

- **Intent-basierte Thread-Priorisierung mit dynamischer Gewichtung:** focusinbox analysiert nicht nur Betreff und Absender, sondern den gesamten semantischen Kontext eines Threads. Parameter: `priority_decay_rate` (Standard: 0.85), `urgency_boost_factor` (1.0–3.0). Technische Basis: Fine-tuned DistilBERT-Modell, optimiert auf dem Enron- und Avocado-Datensatz. Die Gewichtung passt sich stündlich an das Nutzerverhalten an (z. B. häufiges Öffnen bestimmter Absender → `sender_weight` wird erhöht).

- **Automatische Kategorisierung in 12 Arbeitszonen mit PostgreSQL-Persistenz:** Jede E-Mail wird einer von 12 Arbeitszonen zugeordnet (z. B. „Kritisch“, „Frist heute“, „Benötigt Recherche“, „Niedrige Priorität“). Die Klassifikation erfolgt über ein Ensemble aus einem LightGBM-Modell (Feature: Anhangsgröße, Absender-Domain-Ranking, Antwortzeit-Historie) und einem Regelwerk für ISO-8601-Datumsangaben. Die Entscheidungen werden in einer PostgreSQL-Tabelle (`focusinbox.classification_log`) mit einem JSONB-Feld für die SHAP-Werte protokolliert.

- **Batch-Processing mit verzögerter Benachrichtigung (Debounce-Mechanismus):** Um Benachrichtigungsfluten zu vermeiden, fasst focusinbox ähnlich priorisierte E-Mails in Batches zusammen. Parameter: `debounce_window_ms` (Standard: 30.000 ms), `max_batch_size` (Standard: 10). Der Batch wird erst dann als Benachrichtigung ausgeliefert, wenn entweder das Zeitfenster abläuft oder die maximale Batch-Größe erreicht ist. Dies reduziert die Anzahl der Push-Benachrichtigungen um durchschnittlich 73 % im Vergleich zu einer sofortigen Zustellung.

### So verwenden Sie es

1. **Initiale Konfiguration der IMAP-Verbindung und des Klassifikationsmodells:**  
   Öffnen Sie die Datei `config/focusinbox.yml`. Tragen Sie Ihre IMAP-Zugangsdaten unter `imap.server`, `imap.port` (Standard: 993), `imap.username` und `imap.password` ein. Setzen Sie `model.path` auf den Pfad des heruntergeladenen Modell-Checkpoints (z. B. `/opt/models/distilbert_focusinbox_v2.onnx`). Führen Sie anschließend `focusinbox init --validate-connection` aus, um die IMAP-Verbindung und die Modell-Integrität zu prüfen. Bei Erfolg erscheint die Meldung „Connection and model validation passed. Classification pipeline ready.“

2. **Definieren von benutzerdefinierten Prioritätsregeln mittels YAML-Syntax:**  
   Erstellen Sie eine neue Datei unter `rules/custom_rules.yml`. Nutzen Sie das vorgegebene Schema:  
   ```yaml
   rules:
     - name: "CEO-Dringlichkeit"
       condition: "sender.domain == 'ceo-company.com' OR subject CONTAINS 'AKUT'"
       action: "boost_priority(urgency_boost_factor=2.5)"
     - name: "Newsletter-Drosselung"
       condition: "header.list_unsubscribe EXISTS"
       action: "set_zone('Niedrige Priorität') AND suppress_notification(debounce_window_ms=60000)"
   ```  
   Laden Sie die Regeln mit `focusinbox rules reload --file rules/custom_rules.yml`. Das System validiert die YAML-Struktur und kompiliert die Regeln in eine interne AST (Abstract Syntax Tree). Fehlerhafte Regeln werden mit einer detaillierten Fehlermeldung (Zeile, Spalte, erwartetes Token) ausgegeben.

3. **Starten des Daemons und Überwachen der Echtzeit-Metriken:**  
   Starten Sie den focusinbox-Daemon mit `systemctl start focusinbox` (oder `focusinbox daemon --foreground` für Debugging). Öffnen Sie in einem zweiten Terminal die Metrik-API unter `http://localhost:9090/metrics`. Hier finden Sie Prometheus-kompatible Metriken wie:  
   - `focusinbox_classification_latency_ms{quantile="0.95"}`  
   - `focusinbox_emails_processed_total`  
   - `focusinbox_batch_notifications_sent_total`  
   Verwenden Sie `focusinbox status --json`, um den aktuellen Zustand der Warteschlangen (Kritisch, Hoch, Mittel, Niedrig) und die Anzahl der anstehenden Batches abzurufen.

### Beste Praktiken

- **Praxis 1: Kombinieren Sie das Bayes-Transformer-Modell mit expliziten Whitelist/Blacklist-Regeln.**  
  In Umgebungen mit starken Compliance-Anforderungen (z. B. Finanzdienstleistungen) sollten Sie das maschinelle Lernen nicht als alleinige Entscheidungsinstanz nutzen. Definieren Sie eine Whitelist für Absender, die niemals in eine Niedrig-Prioritäts-Zone fallen dürfen (z. B. `rules/whitelist.yml` mit `priority_override: 'Kritisch'`). Das Modell dient dann als Vorschlag, der durch die Regel übersteuert wird. Dies erhöht die Audit-Fähigkeit und verhindert False-Negatives bei geschäftskritischen Kommunikationen.

- **Praxis 2: Nutzen Sie den Debounce-Mechanismus für projektbezogene E-Mail-Kanäle.**  
  Wenn Sie mehrere E-Mail-Konten für verschiedene Projekte (z. B. `projekt-alpha@firma.de`, `projekt-beta@firma.de`) betreiben, konfigurieren Sie für jedes Konto einen eigenen `debounce_window_ms`. Für ein Konto mit sehr hohem Volumen (>200 E-Mails/Tag) setzen Sie den Wert auf 60.000 ms, für ein Konto mit niedriger Frequenz auf 10.000 ms. Dies verhindert, dass Sie bei einem Konto mit wenigen, aber wichtigen E-Mails unnötig lange warten müssen. Überwachen Sie die Metrik `focusinbox_batch_avg_wait_time_ms` im Prometheus-Dashboard, um die Werte iterativ zu optimieren.
