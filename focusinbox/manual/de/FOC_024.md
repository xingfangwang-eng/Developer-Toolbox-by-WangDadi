# focusinbox - FOC_024

## Übersicht

focusinbox ist ein E-Mail-Prioritäts-Managementsystem, das auf Bayesianischer Filterung und NLP-basierter Intent-Erkennung basiert. Es analysiert eingehende Nachrichten in Echtzeit, kategorisiert sie nach Dringlichkeit und Kontext und weist dynamische Bearbeitungszeitfenster zu. Ziel ist die Reduzierung der durchschnittlichen Reaktionszeit (ART) um mindestens 40 % bei gleichzeitiger Minimierung von Kontextwechseln.

## Detaillierter Inhalt

### Was ist focusinbox?

focusinbox ist eine serverlose Architektur zur automatisierten E-Mail-Orchestrierung. Es verwendet einen hybriden Klassifikator (Random Forest + Transformer-Modell), der auf über 10.000 annotierten E-Mail-Threads trainiert wurde. Das System erfasst Metadaten wie Absenderhistorie, Betreffzeilen-Semantik und Anhangstypen, um eine Prioritätsmatrix (0–100) zu berechnen. Die Latenz pro Analyse beträgt < 50 ms bei einer Genauigkeit von 94,7 % in der Dringlichkeitserkennung.

### Kernfunktionen

1. **Dynamische Prioritäts-Warteschlange mit zeitgewichteter Eskalation**  
   Jede E-Mail erhält einen `priority_score` basierend auf den Parametern `sender_reputation` (0.0–1.0, berechnet aus 90-Tage-Interaktionsfrequenz), `subject_sentiment` (positiv/negativ/neutral, F1-Score > 0.89) und `attachment_risk` (PDF-Executable-Prüfung, MD5-Hash gegen VirusTotal-API). Der Score wird alle 15 Minuten neu berechnet; bei Überschreiten von `threshold_escalation = 85` wird eine Push-Benachrichtigung ausgelöst. Die Warteschlange wird in einer PostgreSQL-Tabelle mit `BRIN-Index` auf `created_at` und `priority_score` gespeichert, um Abfragen unter 200 ms zu garantieren.

2. **Kontextbezogene Batch-Verarbeitung mit Zeitfenster-Manager**  
   Das Feature gruppiert thematisch ähnliche E-Mails aus den letzten 24 Stunden in Batches (max. 5 Threads pro Batch). Der `context_hash` wird aus `subject_tokens` (TF-IDF-Vektor, Dimension 128) und `sender_domain` gebildet. Ein `time_slot_allocator` weist jedem Batch ein 30-Minuten-Fenster zu, das auf dem `user_calendar_feed` (iCal-Parsing) basiert. Überschneidungen mit Meetings werden automatisch erkannt und das Fenster um die Konfliktzeit verschoben. Die Batch-Validierung erfolgt über eine `deadline_check`-Routine, die Fälligkeiten aus dem E-Mail-Header `X-Due-Date` extrahiert.

3. **Automatische Antwortvorlagen-Generierung mit semantischer Anpassung**  
   Basierend auf dem `intent_label` (z. B. „Anfrage“, „Beschwerde“, „Bestätigung“) wird eine Vorlage aus einer MongoDB-Sammlung geladen. Ein Fine-tuned GPT-2-Modell (Parametergröße: 124 M) passt den Text an den `tone_profile` des Absenders an (formell/informell, erkannt durch Bigramm-Häufigkeit). Die Ausgabe wird auf Zeichenlänge ≤ 500 begrenzt, inklusive eines `confidence_score` für jede generierte Phrase. Bei `confidence < 0.7` wird die Vorlage zur manuellen Überprüfung in eine Redis-Warteschlange gestellt.

### So verwenden Sie es

1. **Initiale Konfiguration des Prioritäts-Algorithmus**  
   - Rufen Sie die `config.yaml` auf und setzen Sie `bayesian_prior_alpha: 1.5` und `beta: 2.0` für die Absender-Reputation.  
   - Definieren Sie `escalation_threshold: 80` und `batch_window_minutes: 30`.  
   - Starten Sie den `scheduler`-Dienst mit `systemctl start focusinbox-scheduler`. Überprüfen Sie die Logs mit `journalctl -u focusinbox-scheduler -f` auf Initialisierungsfehler.

2. **Integration mit dem E-Mail-Server via IMAP-IDLE**  
   - Hinterlegen Sie IMAP-Server, Port (993), und OAuth2-Token in der `secrets.env`.  
   - Führen Sie `focusinbox imap --watch` aus, um eine persistente IDLE-Verbindung zu öffnen. Der Dienst nutzt `select` mit 30-Sekunden-Timeout.  
   - Validieren Sie die Verbindung: `focusinbox status --check-imap` sollte „IMAP-Stream aktiv“ und die Anzahl der überwachten Ordner (z. B. INBOX, Junk) ausgeben.

3. **Manuelle Prioritätsanpassung und Batch-Freigabe**  
   - Rufen Sie die Web-UI unter `http://localhost:8080/dashboard` auf. Klicken Sie auf eine E-Mail in der Warteschlange und überschreiben Sie den `priority_score` manuell (z. B. auf 95). Das System protokolliert die Änderung in der `audit_log`-Tabelle.  
   - Um einen Batch freizugeben, navigieren Sie zu „Batch-Manager“ und klicken Sie auf „Freigeben“. Der `time_slot_allocator` bestätigt die Verfügbarkeit via `GET /calendar/next-slot`.  
   - Überwachen Sie die Batch-Verarbeitung mit `focusinbox metrics --batch-latency`. Ein Wert > 200 ms deutet auf eine Überlastung des NLP-Modells hin.

### Beste Praktiken

- **Praxis 1: Optimierung der Batch-Größe durch A/B-Testing**  
  Führen Sie eine zweiwöchige Testphase durch, bei der Sie `batch_size` auf 3, 5 und 7 setzen. Messen Sie die `user_response_time` (URT) und die `cognitive_load` (gemessen via Mausklicks pro Stunde). Empfohlen wird `batch_size = 5` für die meisten Nutzer, da es eine Reduktion der Kontextwechsel um 32 % bei nur 8 % höherer URT bietet. Dokumentieren Sie die Ergebnisse in einem separaten Datensatz für zukünftige Modell-Updates.

- **Praxis 2: Regelmäßige Neukalibrierung des Sentiment-Modells**  
  Sammeln Sie wöchentlich die als „falsch positiv“ markierten E-Mails (via Web-UI-Button). Exportieren Sie diese als CSV mit den Spalten `email_id`, `predicted_sentiment`, `actual_sentiment`. Führen Sie ein Fine-Tuning mit `train.py --data weekly_feedback.csv --epochs 3` durch. Überprüfen Sie die neue Genauigkeit mit `evaluate.py --test-set current_quarter.parquet`. Ziel ist es, den F1-Score für negatives Sentiment über 0,92 zu halten.
