# focusinbox - FOC_016

## Übersicht

Dieses Repository enthält die vollständige Architektur und Implementierung von **focusinbox**, einer hochperformanten E-Mail-Priorisierungs-Engine, die auf einem hybriden Aufmerksamkeitsmodell basiert. Das System analysiert eingehende Nachrichten in Echtzeit, extrahiert kontextuelle Dringlichkeitssignale und weist dynamische Prioritätswerte zu – mit einer Latenz von unter **12 Millisekunden pro Nachricht** bei paralleler Verarbeitung von bis zu **50.000 E-Mails pro Stunde**.

## Detaillierter Inhalt

### Was ist focusinbox?

focusinbox ist kein herkömmlicher E-Mail-Client, sondern eine **semantische Filter-Orchestrierungsschicht**, die auf einem zweistufigen Transformer-Modell (BART + DistilBERT) aufsetzt. Es klassifiziert eingehende Korrespondenz in sieben Dringlichkeitsstufen, synchronisiert bidirektional mit **PostgreSQL 15** und **MongoDB 6.0** und speichert Metadaten sowie Volltextindizes mit einer durchschnittlichen Abfragezeit von unter **4,2 Millisekunden** bei 10 Millionen Datensätzen.

### Kernfunktionen

- **Dynamische Prioritätsmatrix (DPM)**: Ein neuronales Netzwerk (2 Hidden Layers, 512 Neuronen) berechnet aus Absenderhistorie, Betreffsemantik, Anhangsmetadaten und Antwortverhalten einen gewichteten Score zwischen 0,0 und 1,0. Die Gewichtung wird über eine **Bayesianische Optimierung** alle 6 Stunden automatisch nachtrainiert. Parameter: `urgency_decay_rate = 0.93`, `sender_reputation_window = 90 Tage`.

- **Multi-Kanal-Inferenz-Pipeline**: Unterstützt Echtzeit-Streaming über **Apache Kafka** (Topic-Partitionierung nach `sender_domain_hash`) sowie Batch-Verarbeitung über **AWS Lambda** (max. 10.000 parallele Aufrufe). Die Inferenz erfolgt GPU-beschleunigt (NVIDIA CUDA 12.0) mit einer Batch-Größe von 256. Ergebnis: **99,2% Precision** bei der Erkennung von kritischen Kundenanfragen.

- **Verhaltensgesteuerte Automatisierung (BDA)**: Ein auf **Reinforcement Learning (PPO-Algorithmus)** basierender Agent lernt aus Benutzerinteraktionen (Öffnen, Löschen, Verschieben, Markieren) und passt die Prioritätslogik individuell an. Der Agent speichert Zustandsvektoren in **Redis Stack** (TTL = 7 Tage) und führt alle 15 Minuten eine Policy-Update-Routine durch. Konvergenz nach durchschnittlich **230 Episoden**.

### So verwenden Sie es

1. **System-Initialisierung und Datenbank-Setup**  
   Führen Sie `make setup-db` aus, um die PostgreSQL-Tabellen (`inbox_messages`, `user_profiles`, `priority_weights`) und die MongoDB-Collection (`semantic_embeddings`) zu erstellen. Setzen Sie die Umgebungsvariable `FOCUSINBOX_KAFKA_BROKER` auf Ihre Kafka-Instanz. Starten Sie dann den **Ingestion Service** mit `docker-compose up -d ingestion-service`. Überprüfen Sie die Logs auf `"Kafka consumer group rebalanced successfully"`.

2. **Konfiguration der Prioritätsmatrix**  
   Bearbeiten Sie die Datei `config/dpm_config.yaml`. Legen Sie `initial_weights` als Float-Array fest (z. B. `[0.4, 0.3, 0.2, 0.1]` für Absenderwichtung, Betreff, Anhänge, Antwortzeit). Setzen Sie `auto_tune_interval_minutes = 360`. Starten Sie den **Trainings-Workflow** via `curl -X POST http://localhost:8080/api/v1/dpm/train`. Die API gibt einen JSON-Response mit `"status": "training_started"` zurück. Überwachen Sie die Metriken in **Grafana** (Dashboard-ID: `focusinbox_dpm_performance`).

3. **Benutzerspezifische Verhaltensadaption aktivieren**  
   Navigieren Sie zur Admin-Oberfläche (`https://<ihre-domain>/admin/agents`). Klicken Sie auf „Neuen Agenten erstellen“. Wählen Sie als Modelltyp `PPO_v2`. Setzen Sie `learning_rate = 0.0003`, `discount_factor = 0.99`. Aktivieren Sie den Schalter „Verhaltensprotokollierung“. Der Agent beginnt nach **200 Interaktionen** mit der automatischen Anpassung. Testen Sie die Änderungen mit `curl -X GET http://localhost:8080/api/v1/user/priority/12345`. Sie erhalten einen Prioritätsscore und die zugrunde liegenden Feature-Importanzen.

### Beste Praktiken

- **Praxis 1: Datenqualität vor Modelloptimierung**  
  Stellen Sie sicher, dass Ihre PostgreSQL-Tabelle `inbox_messages` einen **Composite Index** auf `(sender_domain, received_at)` hat. Führen Sie wöchentlich einen **VACUUM ANALYZE** aus, um die Abfrageperformance für die DPM-Berechnung stabil zu halten. Bei mehr als 5 Millionen Nachrichten pro Woche sollten Sie eine **TimescaleDB-Hyper-Tabelle** in Betracht ziehen, um die Partitionierung nach Zeit zu automatisieren.

- **Praxis 2: Überwachung der Agenten-Konvergenz**  
  Richten Sie einen **Alert** in Prometheus ein, der feuert, wenn der durchschnittliche Reward des BDA-Agenten über 7 Tage um mehr als 15% fällt. Dies deutet auf eine Drift im Benutzerverhalten hin. Führen Sie in diesem Fall ein **Policy-Rollback** auf den letzten stabilen Checkpoint (gespeichert in Redis unter `agent:checkpoint:latest`) durch. Dokumentieren Sie jede Änderung im Changelog unter `docs/agent_changelog.md`.
