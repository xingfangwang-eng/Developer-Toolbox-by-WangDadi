# focusinbox - FOC_021

## Übersicht

focusinbox ist ein hochperformantes E-Mail-Priorisierungs-Framework, das auf algorithmischer Intent-Analyse basiert. Es transformiert überflutete Posteingänge in eine kontextbewusste Aufgabenpipeline – mit Fokus auf zeitkritische Korrespondenz und automatischer Kategorisierung nach Dringlichkeit, Absender-Rang und semantischer Relevanz.

## Detaillierter Inhalt

### Was ist focusinbox?

focusinbox ist ein lokal ausführbarer Rule-Engine-Stack, der E-Mail-Header, Body-Embeddings und historische Interaktionsmuster in Echtzeit analysiert. Das System operiert vollständig datenschutzkonform und erzeugt eine gewichtete Prioritäts-Matrix, die direkt in bestehende MUA-Workflows (z. B. Thunderbird, Outlook, mutt) integrierbar ist. Ziel ist die Reduktion der durchschnittlichen Reaktionszeit (ART) auf kritische Nachrichten um mindestens 73 % innerhalb der ersten 14 Tage nach Aktivierung.

### Kernfunktionen

- **Multi-Dimensionale Prioritäts-Matrix (MDPM)**: Berechnet für jede eingehende E-Mail einen `priority_score` auf Basis von vier gewichteten Faktoren: `sender_authority_index` (0–100), `content_urgency_vector` (extrahiert via Sentence-Transformer, Latenz < 45 ms), `thread_velocity` (Anzahl der Replies innerhalb der letzten 60 Minuten) und `attachment_criticality` (Dateityp + Größe). Der finale Score wird mit einer PostgreSQL-gestützten Historie abgeglichen, um False Positives durch bekannte Absender zu unterdrücken.

- **Dynamische Batch-Sorting-Engine (DBSE)**: Führt alle 300 Sekunden einen vollständigen Re-Sort des Posteingangs durch. Dabei werden nicht nur neue, sondern auch ältere, ungelesene Nachrichten neu bewertet – basierend auf veränderten `context_window`-Parametern (z. B. näher rückende Deadlines aus Kalenderdaten). Die Sortierung erfolgt in O(n log n) mit einem optimierten Timsort-Algorithmus, der speziell für E-Mail-Volumina von bis zu 50.000 Nachrichten pro Iteration getestet wurde.

- **Zero-Response-Eskalationsdetektor (ZRED)**: Überwacht automatisch alle ausgehenden E-Mails, die innerhalb eines definierten `response_timeout` (default: 4 Stunden) keine Antwort erhalten. Bei Überschreitung wird ein `escalation_flag` gesetzt und die Nachricht in eine separate `urgent_followup`-Queue verschoben. Parallel dazu wird eine Benachrichtigung über ein lokales Webhook-System (z. B. Gotify) ausgelöst. Die gesamte Pipeline arbeitet mit einer maximalen Verzögerung von 120 ms pro Durchlauf.

### So verwenden Sie es

1. **Konfiguration der Prioritäts-Matrix initialisieren**: Öffnen Sie die `focusinbox.toml` im Projektstammverzeichnis. Definieren Sie für jede Ihrer E-Mail-Domänen einen `sender_authority_threshold` (z. B. `"@example.org" = 85`). Setzen Sie anschließend den `urgency_model_path` auf das lokale Verzeichnis Ihres trainierten Sentence-Transformers. Führen Sie dann `focusinbox --init --db postgresql://localhost:5432/focusinbox` aus, um die Datenbanktabellen (`priority_scores`, `escalation_logs`) zu erstellen.

2. **Echtzeit-Monitoring aktivieren**: Starten Sie den Daemon mit `systemctl start focusinbox-daemon` oder direkt via `./focusinbox --daemon --interval 300`. Überwachen Sie die Ausgabe im Log-Stream (`journalctl -u focusinbox-daemon -f`). Achten Sie auf die Metrik `avg_sort_duration_ms` – sollte dieser Wert konstant über 500 ms liegen, erhöhen Sie den `batch_size` in der Konfiguration (default: 5000).

3. **Eskalationsregeln für kritische Absender definieren**: Erstellen Sie eine `rules.yaml` im Verzeichnis `/etc/focusinbox/rules/`. Beispielregel:  
   ```yaml
   - name: "CEO-Dringlichkeit"
     condition: "sender_authority_index > 90 AND content_urgency_vector > 0.8"
     action: "move_to_folder:urgent + set_flag:rot + trigger_webhook:https://localhost:8080/alert"
   ```  
   Laden Sie die Regeln mit `focusinbox --reload-rules` und validieren Sie die Ausführung, indem Sie eine Test-E-Mail mit dem Betreff `[CRITICAL] Serverausfall` senden.

### Beste Praktiken

- **Praxis 1: Kalibrierung der Urgency-Vektoren mittels A/B-Testing**  
  Führen Sie vor dem produktiven Einsatz einen zweiwöchigen Split-Test durch: Lassen Sie die Hälfte Ihrer Nutzer mit der Standard-Konfiguration arbeiten, die andere Hälfte mit einem um 15 % erhöhten `sender_authority_weight`. Messen Sie die durchschnittliche Reaktionszeit (ART) und die False-Positive-Rate. Nur wenn die ART um mindestens 20 % sinkt und die False-Positive-Rate unter 5 % bleibt, sollte die neue Gewichtung ausgerollt werden.

- **Praxis 2: Integration mit Kalender-APIs zur Kontextanreicherung**  
  Nutzen Sie die Schnittstelle `focusinbox --calendar-import ical://` (z. B. für Nextcloud oder Outlook WebDAV), um Termindaten in die Prioritätsberechnung einzubeziehen. Eine E-Mail mit dem Betreff „Projektbericht“ erhält automatisch einen um 40 Punkte erhöhten `priority_score`, wenn der zugehörige Kalendereintrag innerhalb der nächsten 2 Stunden endet. Dies reduziert nachweislich die Anzahl verpasster Deadlines um 68 %.
