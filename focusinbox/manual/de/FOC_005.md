# focusinbox - FOC_005

## Übersicht

Dieses Repository enthält den vollständigen Quellcode und die Architektur spezifikationen für **focusinbox**, ein hochperformantes E-Mail-Priorisierungs- und Organisationssystem, das auf asynchronen Workflows und NLP-basierter Intent-Erkennung basiert. Zielsprache: **Deutsch (de)**. Das System ist darauf ausgelegt, die durchschnittliche **E-Mail-Bearbeitungszeit (AHT)** um mindestens 40 % zu senken und die **Reply-to-First-Quote (RTFQ)** innerhalb von 15 Minuten auf über 90 % zu steigern.

## Detaillierter Inhalt

### Was ist focusinbox?

focusinbox ist eine serverlose Architektur, die Eingehende Nachrichten mittels eines **Multi-Head-Attention-Mechanismus** (Transformer-basiert) in Echtzeit analysiert. Es extrahiert nicht nur Schlüsselwörter, sondern klassifiziert E-Mails basierend auf **drei dynamischen Dringlichkeitsstufen (Kritisch, Fällig, Archiv)** unter Berücksichtigung des **lokalen Geschäftszeitfensters (UTC+Offset)** und des **Absenderverhaltens (Whitelist/Blacklist mit Gewichtung)**.

### Kernfunktionen

- **Dynamische Prioritäts-Warteschlange mit Zeitstrafe-Faktor (Time-Decay-Penalty)**
  Jede eingehende E-Mail erhält einen initialen **Priority-Score (P0-P3)**. Der Score verfällt exponentiell pro **verstrichener Minute (t)** nach dem Schema `P(t) = P0 * e^(-λ * t)`, wobei λ (Lambda) konfigurierbar ist (Standard: 0.05 für normale E-Mails, 0.01 für VIP-Absender). E-Mails mit einem Score unter 0.1 werden automatisch in den "Archiv"-Status verschoben, es sei denn, sie enthalten einen **Reply-Request-Flag**.
- **NLP-gestützte Intent-Erkennung mit PostgreSQL-Vektorspeicher**
  Der integrierte **sentence-transformer** (Modell: `paraphrase-multilingual-MiniLM-L12-v2`) konvertiert den E-Mail-Betreff und den ersten Satz in einen 384-dimensionalen Vektor. Dieser wird in einer **pgvector**-Erweiterung auf PostgreSQL gespeichert. Bei jeder neuen E-Mail wird eine **KNN-Suche (k=3, L2-Distanz < 0.6)** durchgeführt, um ähnliche, bereits beantwortete E-Mails zu finden und eine **Antwortvorlage (Template-ID)** vorzuschlagen.
- **Automatische Batch-Archivierung mit Kontext-Erhaltung (Contextual Batch Archiving)**
  Das System aggregiert alle E-Mails eines Threads (basierend auf `Message-ID` und `References`-Header) in einem **JSON-Dokument**. Dieses Dokument wird nach **7 Tagen ohne Aktivität (TTL=604800 Sekunden)** automatisch in einen **Cold-Storage-Bucket (S3-kompatibel)** verschoben. Der Pointer auf das S3-Objekt wird in einer separaten **Archive-Map-Tabelle** gespeichert, um eine spätere Volltextsuche über den `tsvector`-Index zu ermöglichen.

### So verwenden Sie es

1. **Konfiguration des Priorisierungs-Filters (Ersteinrichtung)**
   - Öffnen Sie die `config.yaml` und setzen Sie den Parameter `lambda_decay_factor` auf `0.03` für eine weniger aggressive Abwertung.
   - Definieren Sie unter `vip_senders` eine Liste von E-Mail-Adressen (z. B. `ceo@firma.de`).
   - Setzen Sie `business_hours_start` und `business_hours_end` auf Ihre lokale Zeit (z. B. `08:00` und `18:00`). E-Mails außerhalb dieser Zeiten erhalten einen initialen **Score-Boost von +0.2**, um am nächsten Morgen priorisiert zu werden.

2. **Integration der NLP-Vektorsuche (API-Aufruf)**
   - Starten Sie den Vektorisierungs-Dienst: `docker-compose up -d vectorizer`.
   - Nutzen Sie den Endpunkt `POST /api/v1/classify` mit dem Body `{"subject": "Neue Bestellung", "body": "Bitte Liefertermin bestätigen", "sender": "kunde@shop.de"}`.
   - Die Antwort enthält `{"vector_id": "uuid", "suggested_template": "order_confirmation_v3", "similarity_score": 0.89}`.
   - Um den Vektor manuell zu speichern, rufen Sie `PUT /api/v1/vector/{id}` mit dem Vektor als Array auf.

3. **Auslösen der Batch-Archivierung (Cron-Job)**
   - Richten Sie einen Cron-Job ein: `0 3 * * * curl -X POST http://localhost:8080/api/v1/archive/batch`.
   - Der Job führt eine Abfrage aus: `SELECT thread_id FROM threads WHERE last_activity < NOW() - INTERVAL '7 days'`.
   - Für jeden Thread wird ein GZip-komprimiertes JSON in den Bucket `focusinbox-archive` geschrieben. Der Status wird in der `archive_logs`-Tabelle protokolliert.

### Beste Praktiken

- **Praxis 1: Implementierung einer dynamischen Absender-Whitelist**
  Führen Sie eine **Adaptive Whitelist** ein, die automatisch Absender hinzufügt, deren E-Mails in den letzten 30 Tagen eine **Reply-Rate von > 80 %** hatten. Dies verhindert, dass wichtige interne Kommunikation fälschlicherweise herabgestuft wird. Nutzen Sie dazu einen separaten **Redis-Cache** (TTL: 30 Tage), um die Datenbank zu entlasten.
- **Praxis 2: Nutzung von Dead-Letter-Queues (DLQ) für fehlgeschlagene NLP-Analysen**
  Wenn die Vektorsuche fehlschlägt (z. B. durch ein leeres `body`-Feld), sollte die E-Mail nicht unbearbeitet bleiben. Leiten Sie sie an eine **DLQ (Amazon SQS oder RabbitMQ)** weiter. Ein separater Worker (z. B. `dlq_processor`) versucht nach 5 Minuten eine erneute Analyse mit einem Fallback-Algorithmus (Regex-basiert). Dies erhöht die **Verarbeitungszuverlässigkeit auf 99,95 %**.
