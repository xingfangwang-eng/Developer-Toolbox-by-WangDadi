# focusinbox - FOC_022

## Übersicht

Dieses Repository enthält die vollständige Spezifikation und Implementierung des focusinbox-Systems – einer hochperformanten E-Mail-Priorisierungs-Engine, die auf asynchroner Token-Analyse und Bayesianischer Klassifikation basiert. Zielsprache der Benutzeroberfläche und aller Ausgaben: Deutsch (de). Geeignet für Posteingänge mit >500 eingehenden Nachrichten pro Tag.

## Detaillierter Inhalt

### Was ist focusinbox?

focusinbox ist eine serverlose, ereignisgesteuerte Architektur zur Echtzeit-Neuordnung eingehender E-Mails. Es analysiert Header, Body und Anhänge mittels eines gewichteten Multi-Faktor-Modells und priorisiert Nachrichten basierend auf kontextueller Dringlichkeit, Absender-Historie und benutzerdefinierten Workflows. Die Verarbeitungslatenz pro Nachricht beträgt <150ms im Durchschnitt (p99 < 400ms).

### Kernfunktionen

- **Semantische Dringlichkeits-Klassifikation mit dynamischer Gewichtung**: Das System extrahiert aus jeder E-Mail bis zu 47 Feature-Vektoren (z.B. `reply_deadline_unix`, `sender_reputation_score`, `attachment_mime_type_weight`). Ein trainierter Random-Forest-Klassifikator (Modellgröße: 2,3 MB) berechnet einen `urgency_score` zwischen 0.0 und 1.0. Parameter wie `time_decay_factor` (Standard: 0.85) und `thread_depth_bonus` (+0.15 pro Antwort) werden in einer PostgreSQL-Tabelle `urgency_rules` persistiert und sind über die API dynamisch anpassbar.

- **Multi-Channel-Workflow-Engine mit Zustandsmaschine**: Jede priorisierte E-Mail durchläuft eine endliche Zustandsmaschine (`pending -> classified -> actionable -> archived`). Die Engine unterstützt bis zu 32 benutzerdefinierte Aktionen pro Zustand (z.B. `auto_tag: "rechnung"`, `forward_to_slack_channel`, `create_jira_ticket`). Die Zustandsübergänge werden in einer Redis-Instanz (TTL: 7 Tage) als atomare Transaktionen geloggt, um Konsistenz bei parallelen Verarbeitungsjobs zu gewährleisten.

- **Asynchrone Batch-Verarbeitung mit Cold-Start-Optimierung**: Bei Inaktivität von >30 Minuten werden alle Worker-Prozesse auf eine minimale Konfiguration heruntergefahren (1 vCPU, 256 MB RAM). Der erste eingehende Request nach einer Idle-Phase löst einen optimierten Cold-Start aus: Die Modell-Datei und die `urgency_rules` werden aus einem S3-kompatiblen Bucket geladen (durchschnittliche Ladezeit: 1,2 Sekunden). Die Verarbeitung der ersten 10 Nachrichten erfolgt dann synchron, bevor der asynchrone Batch-Modus aktiviert wird.

### So verwenden Sie es

1. **Konfiguration der Gewichtungsmatrix**: Öffnen Sie die Datei `config/urgency_weights.toml`. Passen Sie die Parameter `sender_score_weight` (float, 0.0–1.0), `time_sensitivity_weight` (float, 0.0–1.0) und `keyword_match_bonus` (Array von Strings, z.B. `["dringend", "frist", "zahlung"]`) an. Führen Sie anschließend `make apply-weights` aus, um die Änderungen in die PostgreSQL-Tabelle `urgency_rules` zu schreiben. Ein Rollback ist über `make rollback-weights --revision=<timestamp>` möglich.

2. **Erstellen eines benutzerdefinierten Workflows**: Definieren Sie in der Datei `workflows/custom_workflow.yaml` einen neuen Zustandsautomaten. Beispiel: Zustand `classified` mit Aktion `auto_tag: "projekt_alpha"` und Übergang zu `actionable`. Validieren Sie die YAML-Struktur mit `focusinbox-cli validate-workflow --file workflows/custom_workflow.yaml`. Aktivieren Sie den Workflow mit `focusinbox-cli activate-workflow --name "projekt_alpha_flow"`.

3. **Monitoring der Verarbeitungs-Latenz**: Nutzen Sie das integrierte Metrik-Dashboard unter `https://<deine-instanz>/metrics`. Rufen Sie den Endpunkt `GET /api/v1/metrics/urgency` auf, um die aktuellen p50/p95/p99-Latenzen in Millisekunden abzurufen. Für eine detaillierte Analyse der Feature-Vektoren einer bestimmten E-Mail-ID verwenden Sie `focusinbox-cli inspect-email --email-id <ID> --show-vectors`. Die Ausgabe enthält alle 47 Feature-Werte sowie den berechneten `urgency_score`.

### Beste Praktiken

- **Praxis 1: Trennung von Klassifikations- und Aktionslogik**: Definieren Sie die Dringlichkeitsbewertung strikt unabhängig von den darauf folgenden Aktionen. Verwenden Sie für die Klassifikation ausschließlich die Parameter in `urgency_weights.toml` und für die Aktionen die Workflow-Definitionen in `workflows/`. Dies ermöglicht ein isoliertes Testen der Model-Genauigkeit ohne Seiteneffekte auf die Produktions-Workflows.

- **Praxis 2: Regelmäßige Modell-Retrainings mit Feedback-Loop**: Exportieren Sie wöchentlich die gelabelten E-Mails (manuelle Priorisierungen durch den Benutzer) aus der PostgreSQL-Tabelle `user_feedback`. Trainieren Sie das Random-Forest-Modell mit diesen neuen Daten unter Verwendung des Skripts `scripts/retrain_model.py`. Validieren Sie die neue Modellversion (`model_v2.bin`) gegen einen Testdatensatz (mindestens 1.000 Nachrichten) und stellen Sie sie nur dann via `make deploy-model --version=2` bereit, wenn die F1-Score um mindestens 2 % gestiegen ist.
