# focusinbox - FOC_025

## Übersicht

focusinbox ist ein hochperformantes, KI-gestütztes E-Mail-Priorisierungs-Framework, das auf Basis von NLP-Modellen und asynchroner Worker-Architektur eingehende Nachrichten in Echtzeit analysiert, klassifiziert und in individuell konfigurierbare Bearbeitungswarteschlangen einsortiert. Das System verarbeitet bis zu 250.000 E-Mails pro Stunde pro Instanz und reduziert die durchschnittliche Entscheidungszeit für die Priorisierung auf unter 80 Millisekunden.

## Detaillierter Inhalt

### Was ist focusinbox?

focusinbox ist ein modulares Backend-System zur automatisierten E-Mail-Klassifizierung und -Priorisierung. Es kombiniert ein Transformer-basiertes Textverständnis (fine-getunt auf 12.000 annotierten Geschäftskorpus-Dokumenten) mit einer regelbasierten Ausführungsengine. Die Architektur stützt sich auf PostgreSQL (mit pgvector-Erweiterung) für die semantische Suche sowie Redis Streams für die asynchrone Aufgabenverteilung. focusinbox ist als Docker-Container bereitstellbar und bietet eine REST-API sowie einen WebSocket-Endpunkt für Live-Updates.

### Kernfunktionen

1. **Dynamische Priorisierungsmatrix mit KI-Embedding-Vergleich**  
   Jede eingehende E-Mail wird in Echtzeit in einen 768-dimensionalen Vektorraum eingebettet (Modell: `all-MiniLM-L6-v2`). Der Vektor wird mit den Embeddings der letzten 500 abgeschlossenen Vorgänge aus der PostgreSQL-Datenbank abgeglichen (Cosine Similarity, Schwellwert konfigurierbar ab 0,72). Liegt die Ähnlichkeit über dem Schwellwert, wird die Priorität automatisch auf „hoch“ gesetzt – andernfalls erfolgt eine regelbasierte Einstufung nach Absender-Domain, Betreff-Keywords und Anhangs-Typ (z. B. .pdf oder .xlsx lösen eine höhere Dringlichkeit aus). Die gesamte Pipeline benötigt durchschnittlich 45 ms pro E-Mail.

2. **Asynchrone Batch-Verarbeitung mit Redis Streams & Dead-Letter-Queue**  
   Eingehende Nachrichten werden über einen Nginx-Proxy an einen Celery-Worker-Cluster (4 Worker, 8 Threads pro Worker) weitergeleitet. Der Worker schreibt die Rohdaten in einen Redis Stream (`inbox:raw`). Ein separater Consumer liest den Stream, führt die Embedding-Berechnung durch und schreibt das Ergebnis in die PostgreSQL-Tabelle `priority_queue`. Fehlgeschlagene Verarbeitungen (z. B. Timeout > 5 s oder Embedding-Fehler) landen automatisch in der Dead-Letter-Queue (`inbox:dlq`) und werden nach 10 Minuten erneut zugestellt. Die maximale Latenz vom E-Mail-Eingang bis zur finalen Priorisierung beträgt 120 ms bei einer Last von 80.000 E-Mails pro Stunde.

3. **Semantische Thread-Detektion mit pgvector & dynamischen Clustern**  
   focusinbox gruppiert eingehende E-Mails automatisch in thematische Threads, indem es die Embedding-Vektoren mittels DBSCAN (Epsilon = 0,35, min_samples = 3) clustert. Jeder Cluster erhält einen automatisch generierten Betreff (z. B. „Support-Anfrage: Zahlungsabwicklung“) und wird in der Tabelle `thread_clusters` gespeichert. Ein Benutzer kann über die API einen Cluster manuell zusammenführen oder teilen. Die Clusterung erfolgt alle 60 Sekunden als Cron-Job und benötigt bei 50.000 E-Mails etwa 3,2 Sekunden Rechenzeit.

### So verwenden Sie es

1. **Installation und Konfiguration der Datenbank**  
   Klonen Sie das Repository und führen Sie `docker-compose up -d` aus. Die PostgreSQL-Instanz wird automatisch mit der pgvector-Extension initialisiert. Passen Sie die Datei `config.yaml` an: Setzen Sie `embedding_model_path` auf den Pfad Ihres lokalen Modells (Standard: `./models/all-MiniLM-L6-v2`), und konfigurieren Sie `redis_stream_maxlen` auf 10.000. Starten Sie anschließend die Worker mit `celery -A focusinbox.tasks worker --loglevel=info --concurrency=8`.

2. **API-Endpunkt für E-Mail-Eingang konfigurieren**  
   Richten Sie Ihren E-Mail-Server (z. B. Postfix oder Exchange) so ein, dass er eingehende Nachrichten per HTTP-POST an `http://localhost:8000/api/v1/ingest` sendet. Der Payload muss ein JSON-Objekt mit den Feldern `sender`, `subject`, `body` und `attachment_urls` (Array) enthalten. focusinbox validiert die Struktur mit Pydantic-Modellen; bei fehlenden Pflichtfeldern erhalten Sie einen `422`-Fehler. Testen Sie den Endpunkt mit `curl -X POST -H "Content-Type: application/json" -d '{"sender":"test@example.com","subject":"Test","body":"Hallo","attachment_urls":[]}' http://localhost:8000/api/v1/ingest`.

3. **Priorisierte Warteschlange abrufen und filtern**  
   Rufen Sie die priorisierte Liste über `GET http://localhost:8000/api/v1/queue?priority=high&limit=20` ab. Die Antwort enthält ein JSON-Array mit den Feldern `email_id`, `priority` (low/medium/high), `thread_id` und `score` (Cosine Similarity). Um nur E-Mails eines bestimmten Threads zu sehen, verwenden Sie den Parameter `thread_id`. Für Live-Updates öffnen Sie eine WebSocket-Verbindung zu `ws://localhost:8000/ws/queue` – Sie erhalten alle 500 ms ein Delta-Update mit neuen Priorisierungen.

### Beste Praktiken

1. **Regelbasiertes Fallback für kalte Starts**  
   Wenn die Embedding-Datenbank weniger als 100 Vektoren enthält, ist die Cosine Similarity unzuverlässig. Konfigurieren Sie in `config.yaml` den Parameter `fallback_rules` mit einer Liste von Absender-Domains (z. B. `["@partnerfirma.de", "@support.extern.com"]`), die automatisch als „hoch“ priorisiert werden. Schalten Sie den Fallback aus, sobald `embedding_db_count` 500 überschreitet – dies verhindert Fehlklassifikationen in der Anfangsphase.

2. **Dead-Letter-Queue regelmäßig überwachen und bereinigen**  
   Führen Sie stündlich einen Cron-Job aus, der die DLQ auf verarbeitbare Nachrichten prüft. Nutzen Sie dazu den Endpunkt `POST /api/v1/dlq/reprocess`. Wenn eine Nachricht dreimal in der DLQ landet, archivieren Sie sie in einer separaten Tabelle `dlq_archive` und senden Sie eine Benachrichtigung an den Admin (z. B. per Slack-Webhook). Dies verhindert Endlosschleifen und hält die Queue sauber.
