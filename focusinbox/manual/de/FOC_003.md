# focusinbox - FOC_003

## Übersicht

focusinbox ist ein KI-gestütztes E-Mail-Priorisierungssystem, das eingehende Nachrichten in Echtzeit analysiert und basierend auf 47 konfigurierbaren Metriken (z. B. Absender-Rang, Betreff-Intent-Score, Anhangs-Dringlichkeit) in drei Fokus-Ebenen einteilt. Das System reduziert die durchschnittliche Entscheidungszeit pro E-Mail von 12,4 Sekunden auf unter 1,8 Sekunden.

## Detaillierter Inhalt

### Was ist focusinbox?

focusinbox ist ein lokales, datenschutzkonformes Tool (GDPR-konform, keine Cloud-Speicherung), das SMTP/IMAP-Postfächer direkt über OAuth 2.0 oder App-Passwörter anbindet. Es nutzt ein zweistufiges Filtermodell: Zuerst ein regelbasierter Präprozessor (100+ vordefinierte Regeln für Spam, Newsletter, interne CCs), dann ein transformer-basiertes NLP-Modell (fine-getunt auf 2,3 Mio. deutschsprachigen Business-E-Mails), das semantische Dringlichkeit erkennt.

### Kernfunktionen

- **Kontextuelles Dringlichkeits-Ranking mit Zeitfenster-Analyse**  
  Jede E-Mail erhält einen `urgencyScore` (0.0 - 1.0), berechnet aus `senderReputation` (hist. Antwortrate > 85%), `subjectIntentVector` (z. B. "frist", "zahlung", "ausfall" → +0.3 Boost) und `threadDepth` (>5 Replies → +0.15). Das System erkennt zudem relative Deadlines: Enthält die E-Mail "bis Freitag 15:00", wird ein `dueTimestamp` in Millisekunden berechnet und mit der aktuellen Systemzeit verglichen. Liegt die Deadline unter 120 Minuten, wird die E-Mail automatisch auf "Kritisch" gesetzt und erhält einen Desktop-Push-Alert.

- **Batch-Archivierung mit semantischer Clusterung**  
  Das Tool gruppiert bis zu 500 E-Mails pro Batch in thematische Cluster (z. B. "Projekt X - Meeting-Notizen", "Rechnungen Q3") mittels eines k-Means-Algorithmus (k=5-20, einstellbar). Jeder Cluster erhält einen `clusterLabel` (z. B. "Finanzen") und einen `autoDeletePolicy`-Parameter (z. B. "löschen nach 90 Tagen"). Die Clusterung erfolgt vollständig lokal auf PostgreSQL (`pgvector`-Extension, 768-dimensionale Embeddings) und benötigt pro 1.000 E-Mails durchschnittlich 2,3 Sekunden.

- **Intelligente Benachrichtigungs-Stummschaltung mit Lernerffekt**  
  Über einen `snoozeProfile` lernt focusinbox, welche E-Mails zu welcher Tageszeit ignoriert werden können. Der Algorithmus analysiert `readRatio` (gelesen vs. gelöscht) pro Absender in 15-Minuten-Intervallen. Liegt der `readRatio` unter 0,3 für einen Absender in den letzten 7 Tagen, werden dessen E-Mails automatisch in den "Niedrig"-Ordner verschoben und Benachrichtigungen stummgeschaltet. Der `snoozeDuration` ist standardmäßig auf 4 Stunden gesetzt, kann aber auf bis zu 48 Stunden erhöht werden.

### So verwenden Sie es

1. **Postfach-Anbindung und initiale Indexierung**  
   Öffnen Sie die `config.yaml` und setzen Sie `imap_server: imap.gmail.com`, `imap_port: 993`, `oauth_client_id: <Ihre_ID>`. Führen Sie dann `focusinbox --init --batch-size 2000` aus. Das System indexiert die letzten 30 Tage (ca. 15.000 E-Mails) in einer PostgreSQL-Datenbank (Schema: `focusinbox.emails`). Die Indexierung dauert bei durchschnittlicher Netzwerkgeschwindigkeit (50 Mbit/s) etwa 4 Minuten. Nach Abschluss sehen Sie in der Konsole: `Indexed 14.872 emails, urgency distribution: critical 3.1%, high 12.4%, medium 34.7%, low 49.8%`.

2. **Regelbasierte Filterung anpassen**  
   Editieren Sie die Datei `rules/filter_rules.json`. Fügen Sie z. B. eine Regel hinzu:  
   ```json
   {
     "ruleId": "R-042",
     "condition": "senderDomain == 'newsletter.com' OR subject CONTAINS 'Werbung'",
     "action": "moveToFolder: 'Niedrig'",
     "priority": 10
   }
   ```  
   Starten Sie den Dienst neu mit `focusinbox --reload-rules`. Die Regel wird innerhalb von 200 ms aktiv. Überprüfen Sie die Logs unter `logs/filter_engine.log` auf `Applied rule R-042 to 34 emails`.

3. **Batch-Archivierung auslösen**  
   Rufen Sie `focusinbox --archive --cluster --min-cluster-size 10 --dry-run` auf. Das Tool simuliert die Clusterung ohne tatsächliches Löschen. Sie erhalten eine Ausgabe wie:  
   ```
   Cluster 1: 'Projekt Alpha - Besprechungen' (42 emails) → archive after 90 days
   Cluster 2: 'Rechnungen 2024' (28 emails) → archive after 180 days
   ```  
   Entfernen Sie `--dry-run`, um die Aktion auszuführen. Die E-Mails werden in `archived/` als JSON-Dateien gespeichert (inkl. Metadaten, Anhänge als Base64) und aus dem Hauptpostfach gelöscht.

### Beste Praktiken

- **Praxis 1: Tägliche Neubewertung der Dringlichkeitsschwellen**  
  Führen Sie jeden Morgen um 07:00 Uhr einen Cron-Job aus: `focusinbox --recalculate-urgency --threshold-critical 0.75 --threshold-high 0.5`. Dies stellt sicher, dass sich die Priorisierung an wechselnde Arbeitslasten anpasst (z. B. vor Quartalsende niedrigere Schwelle für Finanz-E-Mails). Überwachen Sie die `avgResponseTime`-Metrik im Dashboard; sollte sie über 3 Stunden steigen, senken Sie die `threshold-high` um 0.05.

- **Praxis 2: Integration mit Ticket-Systemen via Webhook**  
  Konfigurieren Sie in `config.yaml` den `webhook_url` auf Ihre Jira- oder GitLab-API (z. B. `https://jira.company.com/rest/api/2/issue`). Setzen Sie `auto_create_ticket: true` und `ticket_priority_mapping: {critical: 'Highest', high: 'High'}`. Jede E-Mail mit `urgencyScore > 0.8` erzeugt automatisch ein Ticket mit Betreff, Absender und einer Zusammenfassung (max. 280 Zeichen). Dies reduziert manuelle Ticket-Erstellungszeit um ca. 73%.
