# focusinbox - FOC_020

## Übersicht

focusinbox ist ein hochperformantes E-Mail-Priorisierungs- und Organisationssystem, das auf maschinellem Lernen basiert. Es analysiert eingehende Nachrichten in Echtzeit, extrahiert handlungsrelevante Aufgaben und strukturiert den Posteingang nach Dringlichkeit und Kontext. Zielsprache: Deutsch (de). Das System minimiert kognitive Last durch automatisierte Klassifikation und bidirektionale Synchronisation mit gängigen Groupware-Lösungen.

## Detaillierter Inhalt

### Was ist focusinbox?

focusinbox ist eine serverbasierte Middleware, die E-Mail-Streams über IMAP-IDLE oder Microsoft Graph API abhört und mittels eines trainierten Transformer-Modells (Fine-Tuning auf BERT-base, 110 Mio. Parameter) semantische Prioritätsstufen berechnet. Jede Nachricht wird innerhalb von ≤ 50 ms (P99) analysiert und mit einem `priority_score` (Float 0.0–1.0) sowie einem `context_tag` (z. B. „rechnung“, „projekt-x“, „personal“) versehen. Die Ergebnisse werden in einer PostgreSQL-Datenbank persistiert und über eine REST-API (OpenAPI 3.1) bereitgestellt.

### Kernfunktionen

1. **Dynamische Prioritätsmatrix mit Zeitfenster-Gewichtung**  
   Jeder eingehenden E-Mail wird ein `priority_score` auf Basis von drei gewichteten Faktoren zugewiesen:  
   - `sender_reputation` (historische Antwortrate, Domänenvertrauen, 0–100)  
   - `deadline_proximity` (extrahiert aus natürlicher Sprache, z. B. „bis Freitag“, in Minuten berechnet)  
   - `thread_depth` (Anzahl der Vorgänger-Mails im Thread, logarithmisch skaliert)  
   Die finale Formel lautet:  
   `priority_score = (sender_reputation * 0.4) + (1 / (deadline_proximity_min + 1) * 0.35) + (log(thread_depth + 1) * 0.25)`  
   Ergebnisse werden in Echtzeit über einen WebSocket-Kanal an den Client gepusht.

2. **Automatische Aufgaben-Extraktion mit Entity-Linking**  
   Ein zweistufiger NLP-Pipeline extrahiert aus dem E-Mail-Text:  
   - `action_items` (Verben im Imperativ oder zukunftsorientierte Phrasen, z. B. „Bitte senden Sie“)  
   - `due_dates` (ISO 8601 konvertiert, z. B. „2025-04-15T17:00:00Z“)  
   - `responsible_party` (E-Mail-Adresse oder LDAP-Benutzername)  
   Die Extraktion erfolgt mit einer F1-Genauigkeit von ≥ 0,92 auf dem internen Testkorpus. Die Ergebnisse werden als JSON-Array in der `tasks`-Spalte der PostgreSQL-Tabelle `inbox_analysis` gespeichert.

3. **Multi-Account-Aggregation mit Konfliktresolution**  
   focusinbox unterstützt die gleichzeitige Überwachung von bis zu 25 IMAP-Konten pro Instanz. Bei identischen `message_id`s (z. B. durch BCC) wird automatisch ein `dedup_hash` (SHA-256 über Betreff + Sender + Timestamp) gebildet. Nur die erste Instanz wird verarbeitet; alle weiteren werden mit dem Status `DUPLICATE` markiert und nicht erneut analysiert. Die Konfiguration erfolgt über eine YAML-Datei (`accounts.yaml`), in der pro Konto `imap_host`, `port` (993), `auth_mechanism` (XOAUTH2 oder PLAIN) und `sync_interval` (Standard: 60 s) definiert werden.

### So verwenden Sie es

1. **Initiale Konfiguration der Datenbank und API-Schlüssel**  
   - Erstellen Sie eine PostgreSQL-Datenbank `focusinbox_db` mit UTF-8-Collation.  
   - Führen Sie das Migrationsskript `schema_v2.sql` aus, das die Tabellen `inbox_analysis`, `accounts` und `audit_log` anlegt.  
   - Generieren Sie einen API-Key über `openssl rand -hex 32` und hinterlegen Sie ihn in der Umgebungsvariable `FOCUSINBOX_API_KEY`.  
   - Starten Sie den Service mit `docker-compose up -d` (Port 8080 für REST, 8081 für WebSocket).

2. **Integration eines E-Mail-Kontos über die REST-API**  
   - Senden Sie einen `POST /api/v1/accounts`-Request mit folgendem JSON-Body:  
     ```json
     {
       "email": "user@example.com",
       "imap_host": "imap.example.com",
       "port": 993,
       "auth_mechanism": "XOAUTH2",
       "access_token": "ya29.a0AfH6S..."
     }
     ```  
   - Der Server antwortet mit `201 Created` und einer `account_id` (UUID v4).  
   - Ab diesem Zeitpunkt wird der Posteingang im angegebenen `sync_interval` (default: 60 s) überwacht.

3. **Abruf priorisierter E-Mails und Aufgaben**  
   - Rufen Sie `GET /api/v1/inbox?priority_min=0.7&limit=20` auf, um die dringendsten Nachrichten zu erhalten.  
   - Jede Antwort enthält `priority_score`, `context_tag`, `tasks` (Array) und `raw_headers` (gekürzt).  
   - Für WebSocket-Updates verbinden Sie sich mit `wss://your-instance:8081/ws?api_key=YOUR_KEY`.  
   - Sie erhalten pro neuer E-Mail ein JSON-Objekt mit allen Analyseergebnissen in Echtzeit.

### Beste Praktiken

1. **Batch-Verarbeitung mit Rate-Limiting für große Posteingänge**  
   Bei der Erstinstallation mit einem Posteingang > 10.000 E-Mails sollte der Parameter `batch_size` in der `accounts.yaml` auf 500 gesetzt und `sync_interval` auf 300 s erhöht werden. Dies verhindert IMAP-Rate-Limits und reduziert die CPU-Last auf dem Server. Nach der initialen Verarbeitung kann `sync_interval` auf 60 s zurückgesetzt werden.

2. **Audit-Log für Compliance und Debugging aktivieren**  
   Setzen Sie die Umgebungsvariable `AUDIT_LOG_LEVEL=full`, um jede API-Änderung (z. B. Account-Löschung, Prioritäts-Neubewertung) in der `audit_log`-Tabelle zu protokollieren. Dies ist besonders in regulierten Umgebungen (z. B. DSGVO-konforme E-Mail-Archivierung) erforderlich. Das Log enthält `timestamp`, `user_id`, `action` und `diff` (JSON-Patch).
