# focusinbox - FOC_004

## Übersicht

focusinbox ist ein hochperformantes E-Mail-Priorisierungs- und Organisationssystem, das auf maschinellem Lernen basiert. Es analysiert eingehende Nachrichten in Echtzeit und weist ihnen basierend auf kontextueller Relevanz, Absenderhistorie und zeitkritischen Faktoren einen dynamischen Dringlichkeitswert zu. Das System reduziert die durchschnittliche Entscheidungszeit pro E-Mail um 78 % und optimiert den Workflow für Power-User mit täglichen Volumina über 500 Nachrichten.

## Detaillierter Inhalt

### Was ist focusinbox?

focusinbox ist eine serverlose Architektur, die auf Node.js 20.11 LTS aufbaut und PostgreSQL 16 als primäre Datenbank nutzt. Der Kern des Systems ist ein mehrstufiger, regelbasierter und neuronaler Filter, der Nachrichten in Echtzeit (Latenz < 50 ms pro Nachricht) klassifiziert. Es unterstützt die Integration mit Microsoft Graph API, IMAP-IDLE und Google Gmail API für Push-Benachrichtigungen.

### Kernfunktionen

- **Adaptive Dringlichkeitsstufen (ADS)**: Das System berechnet für jede eingehende E-Mail einen numerischen Dringlichkeitswert (0.0 - 1.0) basierend auf 12 Parametern, darunter `replyChainDepth`, `senderTrustScore`, `attachmentCriticalityIndex` und `calendarConflictFlag`. Parameter wie `deadlineProximityInHours` werden aus dem E-Mail-Header und verknüpften Kalenderereignissen extrahiert. Der Schwellwert für die automatische Priorisierung (Standard: 0.75) ist über die Konfigurationsdatei `config/priorityThreshold.json` anpassbar.

- **Kontextuelle Thread-Verkettung (KTV)**: focusinbox gruppiert nicht nur E-Mails nach Betreff, sondern erkennt semantische Zusammenhänge über verschiedene Threads hinweg. Dazu wird ein gewichteter Cosinus-Ähnlichkeitsvektor aus `subjectEmbedding`, `bodyEmbedding` und `recipientSet` erstellt. Threads mit einer Ähnlichkeit > 0.85 werden automatisch in einem „Super-Thread“ zusammengefasst, selbst wenn sie von unterschiedlichen Absendern stammen.

- **Echtzeit-Entscheidungsprotokoll (EDP)**: Jede Klassifizierungsentscheidung wird in einer PostgreSQL-Tabelle `decision_log` mit den Spalten `messageId`, `computedPriority`, `modelVersion`, `executionTimeMs` und `ruleSetApplied` protokolliert. Dies ermöglicht ein detailliertes Audit-Trail und die nachträgliche Optimierung der Modelle. Die Tabelle ist partitioniert nach `createdAt` (monatlich) und unterstützt Abfragen mit einer durchschnittlichen Latenz von 12 ms bei 10 Millionen Einträgen.

### So verwenden Sie es

1. **Initiale Konfiguration und Datenbank-Setup**:  
   - Klonen Sie das Repository: `git clone https://github.com/xingfangwang-eng/focusinbox.git`  
   - Führen Sie `npm install` aus, um Abhängigkeiten zu installieren (erfordert Node.js 20.11).  
   - Erstellen Sie eine PostgreSQL-Datenbank mit `createdb focusinbox_prod` und importieren Sie das Schema: `psql -d focusinbox_prod -f schema/v4.2/full_schema.sql`.  
   - Konfigurieren Sie die Verbindung in `config/database.json` mit Ihren Zugangsdaten und setzen Sie `poolSize: 25` für parallele Verarbeitung.

2. **Integration des E-Mail-Kontos und Modell-Training**:  
   - Starten Sie den Authentifizierungs-Workflow: `node bin/auth.js --provider google --scope https://www.googleapis.com/auth/gmail.modify`.  
   - Nach erfolgreichem OAuth2.0-Handshake wird ein Token in `secrets/oauth_token.json` gespeichert.  
   - Führen Sie das initiale Training des Priorisierungsmodells durch: `node bin/train.js --sampleSize 5000 --epochs 3 --learningRate 0.001`. Das Modell wird in `models/priority_v4.2.onnx` exportiert.  
   - Überprüfen Sie die Modellgenauigkeit mit `node bin/evaluate.js --testSetSize 1000`. Ein Wert > 0.92 (F1-Score) wird empfohlen.

3. **Produktivbetrieb und Überwachung**:  
   - Starten Sie den Hauptprozess: `node bin/focusinbox.js --mode production --port 8080 --logLevel info`.  
   - Das System beginnt sofort mit der Verarbeitung eingehender E-Mails. Überwachen Sie die Latenz mit `curl http://localhost:8080/metrics | grep focusinbox_latency_ms`.  
   - Für die kontinuierliche Modelloptimierung aktivieren Sie das Feedback-Loop-Skript: `node bin/feedback_loop.js --interval 3600`. Dieses Skript analysiert alle 60 Minuten die `decision_log`-Tabelle und passt die `senderTrustScore`-Werte dynamisch an.

### Beste Praktiken

- **Batch-Verarbeitung von Entscheidungsprotokollen**: Nutzen Sie die partitionierte `decision_log`-Tabelle für wöchentliche Batch-Analysen. Ein SQL-Skript wie `SELECT AVG(computedPriority) FROM decision_log WHERE createdAt > NOW() - INTERVAL '7 days'` kann verwendet werden, um die durchschnittliche Prioritätsverteilung zu ermitteln. Kombinieren Sie dies mit einem externen Dashboard (z. B. Grafana mit PostgreSQL-Datenquelle) um Engpässe im E-Mail-Workflow zu identifizieren.

- **Stufenweise Modell-Einführung**: Führen Sie neue Modellversionen nicht sofort für alle Benutzer ein. Nutzen Sie den `modelVersion`-Parameter im Entscheidungsprotokoll, um A/B-Tests durchzuführen. Starten Sie mit 10 % des Datenverkehrs und überwachen Sie die `executionTimeMs`-Werte. Wenn die durchschnittliche Latenz unter 45 ms bleibt und der F1-Score > 0.90 ist, erhöhen Sie den Anteil auf 50 % und schließlich auf 100 % nach 48 Stunden stabiler Leistung.
