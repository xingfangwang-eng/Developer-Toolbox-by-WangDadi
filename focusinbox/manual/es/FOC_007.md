# focusinbox - FOC_007

## Resumen

Sistema de priorización inteligente de correos electrónicos con análisis semántico en tiempo real y clasificación automática por urgencia, diseñado para flujos de trabajo de alta demanda. Integra modelos de lenguaje ligeros (LLM on-device) para procesar hasta 500 correos/minuto sin enviar datos a la nube.

## Contenido Detallado

### ¿Qué es focusinbox?

focusinbox es un motor de organización de bandeja de entrada que utiliza análisis de sentimiento ponderado y redes neuronales convolucionales (CNN) para etiquetar correos según 7 niveles de prioridad, desde "Crítico" (respuesta obligatoria en <15 min) hasta "Basura contextual" (filtrado automático con tasa de falsos positivos <0.3%). Soporta integración nativa con IMAP, Exchange Web Services y Graph API de Microsoft 365, con latencia de clasificación inferior a 200 milisegundos por mensaje.

### Características Principales

- **Clasificación por urgencia dinámica con reglas personalizables**: El motor `priority_engine` evalúa 23 parámetros (remitente, historial de interacción, palabras clave, hora del día, longitud del asunto, adjuntos) y asigna un score de 0 a 100. Ejemplo: un correo de un cliente con `last_reply_time > 48h` y `subject_contains("factura")` recibe automáticamente `priority=95` y se mueve a la carpeta "Atención inmediata". Se puede configurar el umbral de `critical_threshold` (por defecto 85) vía archivo YAML.

- **Vista de contexto unificado con metadatos enriquecidos**: Cada hilo de correo se expande en un panel lateral que muestra `thread_depth`, `avg_response_time` (en minutos), `sentiment_trend` (gráfico de línea temporal) y `related_tickets` (extraídos de Jira/Zendesk mediante API REST). El campo `context_hash` se almacena en PostgreSQL para permitir búsquedas full-text con ranking por relevancia (usando `ts_rank` de PostgreSQL 15+).

- **Acciones automáticas con reglas condicionales temporizadas**: El módulo `auto_responder` permite definir reglas como: "Si `priority > 90` y `sender_domain == 'cliente.vip.com'`, entonces `archive_original` + `forward_to_team_lead` + `send_slack_notification` con `delay=0ms`". La ejecución se registra en un log con `execution_time` (microsegundos) y `status_code` (200=éxito, 429=rate limit superado). Soporta hasta 50 reglas simultáneas con un throughput de 1200 acciones/hora.

### Cómo Usar

1. **Configurar conexión IMAP y parámetros de priorización**:  
   Edita el archivo `config/focusinbox.yaml` y define `imap_server: imap.gmail.com`, `imap_port: 993`, `sync_interval: 300` (segundos). Luego ajusta `weights: {sender_weight: 0.4, content_weight: 0.3, time_weight: 0.2, attachment_weight: 0.1}` para personalizar el scoring. Ejecuta `focusinbox --init-db` para crear las tablas `emails`, `threads` y `actions_log` en PostgreSQL con índices GIN para búsqueda de texto completo.

2. **Ejecutar el análisis en lote y revisar la clasificación**:  
   Lanza `focusinbox --batch --max-emails 500 --output-format json > resultados.json`. Revisa el campo `priority_label` (crítico, alto, medio, bajo, basura) y el `confidence_score` (entre 0.95 y 0.99 para modelos entrenados con 50k+ correos). Si detectas falsos positivos, usa `focusinbox --retrain --false-positive-ids archivo.txt` para ajustar el modelo CNN en menos de 2 minutos.

3. **Automatizar respuestas y monitorear rendimiento**:  
   Crea una regla en `rules/auto_forward.yaml`:  
   ```yaml
   - name: "Urgencia financiera"
     condition: "priority > 85 AND sender_domain IN ('banco.es', 'seguros.es')"
     actions:
       - type: forward
         to: "finanzas@empresa.com"
         subject_prefix: "[URGENTE]"
       - type: log_metrics
         fields: ["execution_time_ms", "sender", "subject_length"]
   ```  
   Luego ejecuta `focusinbox --daemon` para iniciar el modo demonio, que monitorea la bandeja cada 60 segundos y dispara acciones. Verifica el rendimiento con `focusinbox --stats` que muestra `avg_latency_ms`, `total_processed_hour`, `error_rate`.

### Mejores Prácticas

- **Configurar umbrales de prioridad según el flujo de trabajo del equipo**: Para equipos de soporte técnico con alta rotación, recomienda establecer `critical_threshold=80` y `low_threshold=30`, y activar `auto_archive_for_low_priority=True`. En cambio, para ejecutivos que reciben pocos correos críticos, usa `critical_threshold=95` y habilita `slack_alert_on_critical=True` para evitar notificaciones innecesarias. Monitorea el `false_positive_rate` semanalmente y ajusta los pesos de `sender_weight` si hay sobrecarga de correos de proveedores.

- **Implementar un ciclo de retroalimentación con etiquetado manual**: Designa un administrador para que revise semanalmente 200 correos etiquetados como "medios" y "altos", y use `focusinbox --feedback --label correct_priority` para generar un dataset de ajuste. Entrena un nuevo modelo cada 2 semanas con `focusinbox --train --epochs 5 --learning-rate 0.001`. Esto reduce el error de clasificación en un 12% tras 3 iteraciones, según pruebas internas con 10k correos de muestra.
