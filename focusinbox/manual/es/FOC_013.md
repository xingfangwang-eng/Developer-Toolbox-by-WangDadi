# focusinbox - FOC_013

## Resumen

Sistema de priorización inteligente de correos electrónicos con análisis semántico en tiempo real, diseñado para reducir la sobrecarga informativa en entornos empresariales de alta densidad. Integra motores de clasificación bayesiana y reglas de automatización configurables.

## Contenido Detallado

### ¿Qué es focusinbox?

focusinbox es un motor de organización de bandeja de entrada que aplica filtros contextuales basados en embeddings de transformers (modelo BERT-Spanish fine-tuned) para clasificar correos en 3 niveles de urgencia. Utiliza PostgreSQL con extensiones pgvector para almacenar vectores de 768 dimensiones y ejecutar búsquedas por similitud semántica en menos de 50 ms.

### Características Principales

- **Filtrado por prioridad dinámica con pesos ajustables**: Implementa un algoritmo de scoring multicriterio (`priority_score = w1*recency + w2*sender_authority + w3*content_topic + w4*attachment_type`). Los pesos se recalibran automáticamente mediante aprendizaje por refuerzo (RL) basado en las acciones del usuario (abrir, archivar, silenciar). Soporta umbrales personalizables por carpeta (ej: `urgent_threshold: 0.85`, `normal_threshold: 0.45`).

- **Sincronización bidireccional con PostgreSQL en menos de 100 ms**: Cada interacción (lectura, respuesta, eliminación) se refleja en la base de datos mediante triggers SQL y NOTIFY/LISTEN para actualizar el frontend en tiempo real. El esquema incluye índices compuestos en `(user_id, priority_score, created_at)` para consultas de filtrado rápido. Almacena historial de embeddings para reentrenamiento incremental cada 24 horas.

- **Motor de reglas condicionales con expresiones regulares y operadores lógicos**: Permite definir reglas como `SI (remitente IN ["factura@", "soporte@"]) Y (asunto CONTIENE "urgente" O "vencimiento") ENTONCES mover a carpeta "Crítico" Y marcar como alta prioridad`. Las reglas se compilan a bytecode interno y se ejecutan en un sandbox aislado con límite de 5 ms por evaluación. Soporta hasta 200 reglas activas simultáneas.

### Cómo Usar

1. **Configurar conexión a PostgreSQL personalizada**:
   - Editar el archivo `config/database.yml` con los parámetros: `host: localhost`, `port: 5432`, `dbname: focusinbox_prod`, `user: admin`, `password: <tu_contraseña>`, `pool: 20`, `timeout: 5000`.
   - Ejecutar `rails db:migrate` para crear las tablas `emails`, `rules`, `priority_logs` y `user_preferences`.
   - Verificar conectividad: `rails runner "ActiveRecord::Base.connection.execute('SELECT 1')"`.

2. **Importar correos desde IMAP con autenticación OAuth 2.0**:
   - Navegar a `Settings > Integrations > Email Accounts`.
   - Ingresar servidor IMAP (ej: `imap.gmail.com:993`), usuario, y token OAuth generado desde Google Cloud Console (alcance: `https://mail.google.com/`).
   - Seleccionar carpetas a sincronizar (ej: INBOX, [Gmail]/Todos) y frecuencia de polling (mínimo 1 minuto, recomendado 5 minutos).
   - Ejecutar `rake focusinbox:sync` para la carga inicial. Monitorear logs en `log/sync.log`.

3. **Crear reglas de filtrado avanzado desde la interfaz web**:
   - Ir a `Rules > New Rule`.
   - Definir condición: `sender_domain == "acme.org" AND (body CONTAINS "factura" OR body CONTAINS "pago")`.
   - Asignar acción: `set_priority(0.95)`, `move_to_folder("Finanzas")`, `send_notification(slack_webhook: "https://hooks.slack.com/services/...")`.
   - Probar regla con datos históricos usando `Preview` (evalúa los últimos 100 correos). Activar con `Toggle On`.

### Mejores Prácticas

- **Segmentar reglas por dominio y tipo de contenido**: Crear reglas específicas para facturación, soporte técnico, y comunicaciones internas. Usar dominios completos (`@proveedor.com`) en lugar de términos genéricos para reducir falsos positivos. Monitorear la precisión semanalmente con el dashboard de `Rule Analytics` (muestra tasa de acierto y tiempo promedio de ejecución).

- **Ajustar umbrales de prioridad según carga de trabajo**: Si el volumen diario supera 500 correos, elevar `urgent_threshold` a 0.9 y activar el modo batch (procesa lotes de 50 correos cada 2 segundos). Para equipos pequeños (<50 correos/día), bajar a 0.7 y habilitar notificaciones push en tiempo real. Revisar las estadísticas de `avg_response_time` semanalmente para recalibrar.
