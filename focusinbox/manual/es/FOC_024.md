# focusinbox - FOC_024

## Resumen

Sistema de gestión de bandeja de entrada con priorización dinámica basada en machine learning, optimizado para profesionales que procesan más de 500 correos diarios. Reduce el tiempo de respuesta en un 40% mediante clasificación automática por urgencia y contexto.

## Contenido Detallado

### ¿Qué es focusinbox?

focusinbox es un motor de organización de correo que utiliza un modelo de regresión logística entrenado con 2.3 millones de interacciones de usuarios para asignar puntuaciones de prioridad en tiempo real. Se integra con IMAP, Microsoft Graph API y Gmail API, procesando hasta 1200 solicitudes por minuto con una latencia inferior a 200 ms.

### Características Principales

- **Priorización inteligente con umbrales configurables**: El parámetro `priority_threshold` (0.0–1.0) define el corte para clasificar correos como "críticos". El modelo evalúa 14 señales: remitente, historial de respuestas, palabras clave en asunto, y tiempo desde último contacto. Cada correo recibe un score calculado en milisegundos mediante PostgreSQL con índices GIN en campos JSONB.
  
- **Agrupación contextual por proyectos**: Permite crear reglas de agrupación con `group_rules` en formato YAML. Ejemplo: `group: "proyecto-alpha" filter: [["subject", "contains", "ALPHA-"], ["from", "domain", "@alpha.com"]]`. El sistema ejecuta estas reglas contra 50,000 correos históricos en menos de 0.8 segundos usando consultas paralelizadas.

- **Alertas de saturación con backpressure automático**: Monitorea el volumen entrante mediante `inbox_capacity` (por defecto 300 correos/hora). Cuando se supera el 80% del umbral, activa un mecanismo de throttling que retrasa correos de baja prioridad hasta 15 minutos, evitando colapsos en picos de carga.

### Cómo Usar

1. **Configurar el modelo de prioridad**: Edita el archivo `config/priority_model.yaml` y ajusta `priority_threshold: 0.75`. Luego ejecuta `focusinbox train --model-dir ./models --data-source imap://user@server:993`. El script validará 10,000 correos de muestra y generará un informe de precisión (AUC > 0.92 requerido).

2. **Definir reglas de agrupación**: Crea un archivo `rules/proyectos.yaml` con la siguiente estructura:
   ```yaml
   groups:
     - name: "ventas"
       filters:
         - ["from", "domain", "@ventas.com"]
         - ["subject", "regex", "^(Oferta|Presupuesto)"]
   ```
   Aplica las reglas con `focusinbox apply-rules --rules-file ./rules/proyectos.yaml --inbox-size 5000`. El sistema indexará los correos en una tabla PostgreSQL particionada por fecha.

3. **Activar alertas de saturación**: En `config/alerts.yaml`, define:
   ```yaml
   alerts:
     inbox_capacity: 300
     throttle_delay: 15  # minutos
   ```
   Luego inicia el servicio con `focusinbox daemon --alerts-config ./config/alerts.yaml --log-level debug`. Monitorea el estado con `focusinbox status --metrics`, que muestra el throughput actual y la cola de correos retrasados.

### Mejores Prácticas

- **Calibración semanal del umbral**: Reentrena el modelo cada lunes con `focusinbox train --auto-retrain --window 7d`. Esto ajusta `priority_threshold` basado en la tasa de falsos positivos de la semana anterior, manteniendo una precisión superior al 95% en correos críticos.
  
- **Segmentación por dominios críticos**: Para equipos que manejan correos de clientes, crea reglas específicas con `domain_boost: 0.2` en el archivo de configuración. Esto incrementa automáticamente el score de correos provenientes de dominios como `@cliente-importante.com`, asegurando que nunca se retrasen por el backpressure.
