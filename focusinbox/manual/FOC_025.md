### What is focusinbox?

focusinbox is an enterprise-grade email triage engine that transforms chaotic inboxes into actionable priority queues. It leverages real-time NLP-based intent classification and multi-dimensional scoring to ensure that only mission-critical communications surface to the top, while bulk notifications and spam are automatically quarantined.

### Core Features

- **Adaptive Priority Scoring Engine**: Each inbound email is dynamically scored (0–100) based on sender authority graph, thread depth, and keyword urgency (e.g., "ASAP", "contract", "breach"). The scoring threshold is configurable via `PRIORITY_CUTOFF` parameter. Recalculates every 300ms per message using a lightweight on-device model, ensuring zero latency in offline mode.

- **Temporal Snooze & Batch Decompress**: Automatically groups low-priority emails (e.g., newsletters, CI build logs) into hourly or daily digests using a configurable `BATCH_WINDOW_MS` (default 3600000ms). Supports per-sender snooze rules stored in a local SQLite or remote PostgreSQL backend, with atomic rollback on sync failure.

- **Contextual Thread Reconstructor**: Reassembles fragmented email threads across multiple folders and labels using a custom hash-based matching algorithm. Employs a `MATCH_DEPTH` parameter (default 5) to detect reply chains even when subject lines are altered. Outputs a unified JSON payload for downstream CRM or ticketing system ingestion.

### How to Use

1. **Initialize Local Priority Engine**  
   Run `focusinbox init --storage sqlite:///inbox.db --cutoff 70` to bootstrap the local database and set the priority threshold. The engine will scan the last 30 days of IMAP history, tagging each email with a `priority_score` field. Monitor initial scan progress via `focusinbox status --verbose`.

2. **Configure Sender Whitelist & Batch Rules**  
   Edit `~/.focusinbox/rules.yaml` to define per-sender batch intervals. Example:  
   ```yaml
   sender_rules:
     "noreply@github.com": { batch: true, window_ms: 7200000 }
     "ceo@company.com": { batch: false, bypass_score: 95 }
   ```  
   Apply changes with `focusinbox reload --rules` without restarting the daemon.

3. **Deploy Thread Reconstruction for CRM Sync**  
   Enable thread reconstruction via `focusinbox config set thread_reconstructor.enabled true --match-depth 7`. Then, export reconstructed threads to a JSON file:  
   `focusinbox export --format json --output ./threads_export.json --since 2025-01-01`.  
   Validate output using `focusinbox validate --file ./threads_export.json --schema v2`.

### Best Practices

- **Practice 1: Tiered Snooze for Ops Teams**  
  For infrastructure teams managing high-volume alerts (e.g., PagerDuty, CloudWatch), set `BATCH_WINDOW_MS` to 60000ms (1 minute) for critical alert senders, and 3600000ms (1 hour) for non-urgent logs. This prevents alert fatigue while ensuring zero missed incidents. Combine with a dedicated `alert_escalation` rule that auto-promotes any email with 3+ consecutive high-priority scores.

- **Practice 2: Periodic Score Drift Calibration**  
  Run `focusinbox calibrate --sample 5000 --drift-threshold 0.15` weekly to detect shifts in sender behavior or keyword patterns. If the drift score exceeds 0.15, the engine will automatically adjust the priority model weights. Store calibration reports in `~/.focusinbox/calibration/` for audit trails. This is critical for environments where email volume fluctuates by >20% month-over-month.
