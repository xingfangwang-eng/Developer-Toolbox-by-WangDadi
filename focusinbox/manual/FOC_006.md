# focusinbox - FOC_006

## Overview

focusinbox is a high-performance email triage engine designed for knowledge workers managing 200+ daily messages across multiple inboxes. It leverages real-time priority scoring ( < 50ms per message ) and cross-platform synchronization to eliminate inbox overload.

## Core Features

### 1. Adaptive Priority Scoring Engine with Temporal Decay
- **Parameter**: `decay_factor` (default: 0.85), `urgency_threshold` (configurable: 0.0–1.0).
- **Technical metric**: Each inbound message is scored within 45ms using a weighted combination of sender reputation (via local PostgreSQL cache), keyword clustering (TF-IDF over last 30 days), and a temporal decay function that reduces priority for messages older than 4 hours. The engine supports custom `boost_rules` (e.g., `from:ceo@company.com` → score +0.3).
- **Hardcore detail**: Scores are persisted in a Redis sorted set with TTL of 72 hours for fast retrieval and batch re-ranking.

### 2. Multi-Account Threading & Conflict Detection
- **Parameter**: `thread_window` (default: 600 seconds), `merge_threshold` (default: 0.75 similarity via Levenshtein distance).
- **Technical metric**: Automatically groups related emails across Gmail, Outlook, and custom IMAP accounts by subject fingerprint (SHA-256 hash of normalized subject line) and sender chain. Detects conflicting replies (e.g., two people assigning different priorities to the same thread) and flags them with a `conflict_score` > 0.9.
- **Hardcore detail**: Thread merging uses a lock-free concurrent map (Go `sync.Map`) and writes results to a SQLite journal for crash recovery.

### 3. Inbox Zero Automation with Granular Snooze Rules
- **Parameter**: `snooze_duration` (in milliseconds, default: 3600000 for 1 hour), `auto_archive_after` (default: 7 days).
- **Technical metric**: Allows defining conditional snooze rules such as “snooze all emails from `newsletter@*` with `unsubscribe` header for 24 hours” or “auto-archive if `priority_score` < 0.2 and `age` > 48 hours”. The engine processes up to 500 rules per second using a trie-based matcher.
- **Hardcore detail**: All actions are logged to a dedicated PostgreSQL table with latency < 10ms per operation, enabling full audit trails.

## How to Use

### Step 1: Configure Your Mailbox Integration
1. Open the **focusinbox CLI** and run `focusinbox auth add --provider gmail --client-id YOUR_CLIENT_ID --client-secret YOUR_CLIENT_SECRET`.
2. For IMAP, use `focusinbox auth add --provider imap --host imap.example.com --port 993 --ssl --username user@domain.com`.
3. Verify the connection: `focusinbox auth test --provider gmail` should return `status: connected` with a latency report in milliseconds.

### Step 2: Set Up Priority Scoring Rules
1. Edit the configuration file at `~/.focusinbox/config.yaml`:
   ```yaml
   scoring:
     decay_factor: 0.85
     urgency_threshold: 0.6
     boost_rules:
       - from: "boss@company.com"
         boost: 0.4
       - subject_contains: "urgent"
         boost: 0.2
   ```
2. Apply the changes: `focusinbox config reload`.
3. Run a dry-run scan: `focusinbox scan --dry-run --limit 100` to see the scored results in JSON format without modifying your inbox.

### Step 3: Automate Inbox Cleanup with Snooze Rules
1. Add a snooze rule via the CLI: `focusinbox rule add --type snooze --match "from:newsletter@*" --duration 86400000 --label "Newsletter"`.
2. Enable auto-archive: `focusinbox rule add --type auto-archive --match "priority_score < 0.2" --age 48h`.
3. Start the daemon: `focusinbox daemon start --interval 300` (checks every 5 minutes). Monitor logs with `focusinbox daemon logs --tail 50`.

### Practice 2: Regularly Audit Your Decay Factor
- **Why**: The default `decay_factor` of 0.85 works for most users, but if you handle time-sensitive deals (e.g., sales or support), lower it to 0.7 to deprioritize older emails faster. Run `focusinbox stats decay --days 7` to see the distribution of scores over time and adjust accordingly. A misconfigured decay factor can cause critical emails to be buried within 2 hours.
