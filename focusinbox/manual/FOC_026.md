# focusinbox - FOC_026

## Overview

focusinbox is an intelligent email triage and priority orchestration engine designed for high-volume inbox environments. It eliminates inbox overload by applying multi-layered Bayesian classification, temporal decay scoring, and cross-thread dependency mapping.

## Detailed Content

### What is focusinbox?

focusinbox is a server-side email processing agent that ingests IMAP streams, applies real-time priority scoring using a configurable decision tree, and re-routes messages into user-defined action lanes. It operates on a sub-100ms decision loop per message, with full offline queue support and PostgreSQL-backed state persistence.

### Core Features

- **Adaptive Priority Scoring Engine** — Each incoming email is analyzed across 12+ weighted parameters: sender reputation score (0–100), thread depth, keyword density, and temporal urgency delta (in milliseconds since last reply). The engine outputs a `priority_score` (float, 0.0–1.0) and an `action_label` (e.g., `urgent`, `defer`, `archive`). Scoring rules are hot-reloadable via a YAML config file without restarting the daemon.

- **Cross-Thread Dependency Mapper** — Detects implicit relationships between emails by computing cosine similarity of TF-IDF vectors on subject lines and body fragments. It builds a directed acyclic graph (DAG) of dependencies, flagging emails that require prior actions (e.g., “Awaiting approval from user X before proceeding”). The graph is serialized to PostgreSQL with a 30-second TTL for real-time updates.

- **Time-Decay Escalation Scheduler** — Assigns each deferred email a `decay_start` timestamp and an escalation interval (default: 4 hours). If the email remains unactioned beyond the `decay_deadline`, the system auto-escalates by bumping its priority score by 0.15 and sending a webhook to a configurable Slack channel. Escalation events are logged with microsecond precision.

### How to Use

1. **Configure IMAP Ingest and Scoring Rules**  
   Edit `config/inbox.yaml` to set your IMAP server credentials, polling interval (minimum: 5 seconds), and priority threshold (`priority_cutoff: 0.65`). Run `focusinbox validate` to check syntax and connectivity. Example:  
   ```bash
   focusinbox validate --config ./config/inbox.yaml
   ```

2. **Deploy the Scoring Engine with a Custom Ruleset**  
   Create a ruleset file `rules/priority_rules.yaml` defining weighted parameters. For example:  
   ```yaml
   rules:
     - name: "sender_priority"
       weight: 0.4
       source: "sender_score"
       condition: "score > 75"
     - name: "thread_depth_boost"
       weight: 0.3
       source: "thread_count"
       condition: "count > 5"
   ```  
   Apply it via:  
   ```bash
   focusinbox apply-rules --rules ./rules/priority_rules.yaml
   ```

3. **Monitor Escalation Events and Adjust TTL**  
   Tail the escalation log in real time:  
   ```bash
   focusinbox tail --log escalation --format json
   ```  
   To change the default escalation interval, set `escalation_interval_minutes: 240` in `config/inbox.yaml` and reload:  
   ```bash
   focusinbox reload --config ./config/inbox.yaml
   ```

### Best Practices

- **Use Sender Reputation Whitelisting** — Maintain a curated whitelist of high-priority senders (e.g., direct reports, critical vendors) with a `sender_score` floor of 90. This prevents false negatives from the Bayesian classifier and ensures urgent messages never get deferred. Store the whitelist in a separate PostgreSQL table for fast lookup with a 10-second cache.

- **Implement a Two-Tier Escalation Policy** — For emails that remain unactioned past the first escalation, trigger a second escalation after 8 hours with a higher priority bump (+0.25) and a phone call notification via Twilio API. This prevents inbox drift for truly critical threads while avoiding alert fatigue for low-urgency items.
