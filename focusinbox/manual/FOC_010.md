### What is focusinbox?

focusinbox is a professional email triage engine that processes incoming messages through a multi-stage priority pipeline, leveraging Bayesian classification and real-time sender reputation scoring to reduce inbox noise by up to 87%. It integrates with any IMAP/SMTP backend and provides sub‑100ms latency for rule evaluation.

### Core Features

- **Adaptive Priority Scoring Engine** – Each email is assigned a `priority_score` (0.0‑1.0) based on a weighted combination of sender history (`sender_reputation`), content urgency detection (`urgency_lexicon_match`), and thread depth (`thread_depth > 2`). The threshold for auto‑archiving is configurable via `PRIORITY_CUTOFF: 0.35`, and the engine recalculates scores every 300 seconds using a rolling window of the last 500 interactions.

- **Context‑Aware Snooze & Batch Dispatch** – Users can define `snooze_rules` with ISO‑8601 duration strings (e.g., `"PT2H"`, `"P1D"`). The system groups snoozed emails into a `batch_dispatch_queue` and releases them when the cumulative `batch_weight` exceeds 10 or the earliest snooze expires. Dispatch uses a dedicated SMTP connection pool (max 5 concurrent) to avoid blocking the main fetch loop.

- **Zero‑Click Action Templates** – Pre‑configured `action_templates` (e.g., `"mark_read_and_archive"`, `"forward_to_slack"`) are triggered by regex patterns on the subject line or body. Each template defines `action_type`, `target_endpoint` (e.g., `https://hooks.slack.com/services/...`), and `retry_policy` (3 retries with exponential backoff, 1000ms base). Execution is tracked in a PostgreSQL `action_log` table with `latency_ms` and `status_code`.

### How to Use

1. **Initialize the Priority Pipeline** – Clone the repository and run `./focusinbox init --config ./config.yaml`. The `config.yaml` must contain your IMAP credentials (`imap_server`, `imap_port`, `imap_username`, `imap_password`), the `PRIORITY_CUTOFF` value, and a `postgresql_dsn` for persistent storage. The init command creates the `priority_scores` and `action_log` tables automatically.

2. **Define Snooze Rules via CLI** – Execute `focusinbox snooze --rule '{"pattern": "newsletter|digest", "duration": "P3D", "batch_weight": 5}'`. This adds a rule to the `snooze_rules` JSON column in PostgreSQL. Verify with `focusinbox snooze --list`. The daemon will apply the rule to all incoming messages matching the pattern, storing them in the `snoozed_emails` table with `release_at` computed as `NOW() + duration`.

3. **Create an Action Template** – Write a new template file in `./templates/slack_notify.yaml`:
   ```yaml
   action_type: webhook
   target_endpoint: "https://hooks.slack.com/services/T00/B00/xxx"
   trigger_pattern: "urgent|critical|p0"
   retry_policy:
     max_attempts: 3
     base_delay_ms: 1000
   ```
   Then run `focusinbox template --apply ./templates/slack_notify.yaml`. The system parses the YAML and inserts a row into the `action_templates` table. On the next fetch cycle, any email with subject or body matching the regex will be forwarded to Slack with a `latency_ms` metric logged.

### Best Practices

- **Tune the `PRIORITY_CUTOFF` incrementally** – Start with a conservative value (e.g., 0.45) and monitor the `auto_archived_count` metric over 48 hours. Use the `focusinbox stats --histogram` command to visualize the distribution of `priority_score`. Adjust the cutoff downward only after confirming that false positives (important emails archived) remain below 2% of your total daily volume.

- **Isolate production and staging PostgreSQL databases** – Use separate `postgresql_dsn` values for testing (`focusinbox_dev`) and production (`focusinbox_prod`). The `action_log` table can grow rapidly; set a TTL index on `created_at` and run `VACUUM` weekly. For high‑volume mailboxes (>5000 messages/day), enable connection pooling with `PGPOOL_SIZE: 10` in the config to avoid `too many clients` errors.
