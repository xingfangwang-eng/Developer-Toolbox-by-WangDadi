# focusinbox - FOC_023

## Overview

focusinbox is a zero-latency email triage engine designed for knowledge workers drowning in asynchronous communication. It ingests IMAP, Microsoft Graph, and custom SMTP streams to compute an **urgency score** per message using a hybrid Bayesian + heuristic model, then surfaces only the 5–12 emails that demand action within the next 90 minutes. The system operates entirely in-memory via Redis Streams with a 200ms P99 ingestion latency, and persists metadata to PostgreSQL for daily trend analysis.

## Core Features

### 1. Real‑Time Priority Scoring via Multi‑Factor Attention Decay

Each inbound email is parsed against a configurable `priority_weight` matrix that evaluates sender relationship (internal/external, direct report, C‑suite), thread depth, keyword density (e.g., “ASAP”, “blocker”, “deadline”), and attachment MIME type. A **log‑linear decay function** recalculates the score every 15 minutes based on elapsed time since receipt. Emails with a score above the dynamic threshold (auto‑tuned via a rolling 7‑day F1‑score against user “snooze” behavior) are promoted to the focus queue. The entire pipeline runs in under 50ms per message using a Rust‑based tokenizer and a pre‑compiled ONNX classifier.

### 2. Multi‑Account Thread Coalescing with Conflict Detection

focusinbox merges threads across up to 12 separate mailboxes by normalizing `Message‑ID`, `References`, and `In‑Reply‑To` headers into a single directed acyclic graph (DAG). When two replies diverge (e.g., a manager and a peer both respond to the same thread with contradictory instructions), the system flags a **conflict node** and surfaces both messages side‑by‑side with a diff of actionable items extracted via a fine‑tuned BERT NER model. The DAG is stored in a Neo4j graph database with a query latency of <10ms for threads with up to 500 messages.

### 3. Automated Snooze & Follow‑Up Scheduling with Calendar Integration

Users can snooze an email by specifying a `remind_at` timestamp (supports natural language like “next Tuesday 3pm” via a Duckling‑based parser). The system then creates a hidden calendar event in Google Calendar or Outlook with a 15‑minute buffer, and at the exact moment, re‑injects the email into the focus queue with a “⏰ Follow‑up” tag. If the sender replies before the snooze fires, the event is automatically deleted and the thread is re‑scored. The scheduler runs on a 30‑second cron cycle with a jitter of ±5 seconds to avoid API rate limits.

## How to Use

### Step 1: Configure Inbound Mailboxes & Priority Rules

Run the initial setup wizard via CLI:

```bash
focusinbox configure --imap imap.gmail.com:993 --user alice@example.com --auth-oauth2 ./token.json
```

Then define a custom priority weight file (`weights.json`):

```json
{
  "sender_boost": 0.4,
  "keyword_boost": 0.3,
  "attachment_boost": 0.2,
  "thread_depth_boost": 0.1,
  "decay_half_life_minutes": 30
}
```

Apply the configuration:

```bash
focusinbox apply-weights --file weights.json --mailbox-id 1
```

### Step 2: Run the Focus Queue Daemon

Start the daemon in foreground mode to see real‑time scoring:

```bash
focusinbox daemon --port 8080 --redis-stream focus:inbox --pg-conn "postgres://user:pass@localhost:5432/focusinbox"
```

The daemon will output a JSON stream every 2 seconds with the current top‑5 emails. To filter by a minimum score of 0.75:

```bash
focusinbox daemon --min-score 0.75
```

### Step 3: Snooze an Email via the REST API

Send a PATCH request to snooze a specific email until tomorrow at 9:00 AM UTC:

```bash
curl -X PATCH http://localhost:8080/api/v1/emails/msg_abc123 \
  -H "Content-Type: application/json" \
  -d '{"action": "snooze", "remind_at": "2025-04-11T09:00:00Z", "note": "Check contract terms"}'
```

The API returns a 200 OK with the new `next_scan_at` timestamp. To verify the scheduled event, query the calendar integration status:

```bash
focusinbox status --email-id msg_abc123
```

## Best Practices

### 1. Decouple Notification Channels from Inbox Triage

Do not rely on push notifications (Slack, mobile alerts) to decide urgency. Instead, configure focusinbox to **only** surface emails that exceed the dynamic threshold, and treat all other inbox traffic as a daily batch review. This reduces context‑switching by ~40% based on internal A/B tests. Use the `--min-score` flag per mailbox to enforce a strict lower bound for high‑volume accounts (e.g., support@).

### 2. Regularly Retrain the Priority Model on User Feedback

The default decay and keyword weights are generic. Once per quarter, export user “snooze” and “archive before reading” events from PostgreSQL and run the retraining script:

```bash
focusinbox retrain --feedback-table user_feedback --output-model ./v2.onnx --epochs 50
```

Then hot‑swap the model without restarting the daemon via a `SIGUSR1` signal. This ensures the urgency score aligns with evolving communication patterns (e.g., a new project with unique keywords).
