# focusinbox - FOC_001

## Overview

This is a high-performance email triage and prioritization engine designed for knowledge workers, executive assistants, and DevOps engineers who need to process 500+ inbound messages per day with sub‑second classification latency. focusinbox eliminates inbox paralysis by applying deterministic rules, machine‑learned intent detection, and attachment‑level threat scoring before the user ever opens a message.

## Detailed Content

### What is focusinbox?

focusinbox is a local‑first, privacy‑preserving email orchestration layer that sits between your IMAP/SMTP server and your MUA (mail user agent). It ingests raw RFC 5322 messages, extracts structured metadata (DKIM‑signature‑validated sender domain, thread depth, attachment MIME‑type inventory), and assigns a **priority delta** (a float from -1.0 to +1.0) per message. The delta is computed using a hybrid pipeline: a deterministic rule engine (configurable via YAML) plus a lightweight ONNX‑runtime neural classifier trained on 50,000 labeled corporate email threads. All processing happens in‑memory with PostgreSQL for persistent state; no email content ever leaves your infrastructure.

### Core Features

- **Temporal‑Attention Priority Scoring**  
  Each inbound message is scored using a custom `priority_score = α · urgency_decay(t) + β · sender_authority + γ · thread_depth_weight`. The `urgency_decay` function applies a logarithmic half‑life (configurable from 30 minutes to 72 hours) so that a message from your CEO flagged as “urgent” at 2:00 AM is still ranked above a routine CI‑build notification at 9:00 AM. The sender authority is derived from a local LDAP or CSV import of organizational hierarchy. Thread depth weighting prevents shallow “reply‑all” chains from drowning out original high‑impact emails.

- **Attachment‑Level Risk & Action Classification**  
  Every attachment is unpacked (up to 3 levels of nested ZIP/TAR/GZ) and scanned for executable MIME types, macro‑enabled Office documents, and known hash signatures via a local ClamAV socket. If risk_score > 0.7, the email is quarantined into a `SUSPICIOUS` folder with a `x-focusinbox-risk` header injected. Separately, the classifier assigns an action label (e.g., `NEEDS_REPLY`, `FOR_ARCHIVE`, `ACTION_REQUIRED_BY_DEADLINE`) with a confidence threshold of 0.92. All classification results are stored in a `focusinbox.messages` table with a `classification_latency_ms` metric exposed via Prometheus.

- **Deterministic Snooze & Batch Digest Engine**  
  Users define snooze rules via a JSON schema: `{ "match": { "sender_domain": "jira.internal", "subject_contains": "sprint" }, "snooze_until": "next_monday_09:00", "digest": true }`. When a message matches, focusinbox suppresses the notification and queues it into a per‑user digest buffer. Every `digest_interval_seconds` (minimum 300), the engine flushes the buffer as a single Markdown‑formatted summary email. The digest includes a `X‑FocusInbox‑Digest‑Count` header and a `List‑Unsubscribe` link for one‑click opt‑out. Latency from match to digest delivery is guaranteed under 2 seconds for buffer sizes up to 200 messages.

### How to Use

1. **Initialize the PostgreSQL Schema & Configuration**  
   Run `focusinbox init --pg-uri postgresql://user:pass@localhost:5432/focusinbox --org-chart ./org.csv`. This creates the `focusinbox` schema with tables `messages`, `snooze_rules`, `sender_authority`, and `classification_log`. The `org.csv` must contain columns `email,manager_email,role_weight`. The CLI validates the file and prints a summary of imported authority scores (e.g., “Imported 1,234 senders, mean authority 0.43”). If the `--pg-uri` is omitted, focusinbox falls back to a local SQLite file at `~/.focusinbox/focusinbox.db`.

2. **Connect to Your IMAP Server & Start the Classification Pipeline**  
   Execute `focusinbox daemon --imap-host imap.example.com --imap-user you@example.com --imap-pass-env IMAP_PASS --inbox INBOX --priority-threshold 0.6 --action-confidence 0.92`. The daemon opens an IDLE connection, listens for `EXISTS` responses, and fetches unseen messages. For each message, it runs the full pipeline: header parsing → attachment extraction → ClamAV scan → priority scoring → action classification. Every classified message is moved to a `focusinbox/INBOX` folder (created automatically) with IMAP flags `$label1` (high priority), `$label2` (medium), `$label3` (low). The daemon logs per‑message latency to stdout in JSON format: `{"msg_id": "abc123", "latency_ms": 187, "priority_score": 0.84, "action": "NEEDS_REPLY"}`.

3. **Query & Export Classification Results**  
   Use `focusinbox query --since "2025-04-01T00:00:00Z" --until "2025-04-07T23:59:59Z" --priority-min 0.7 --format csv` to export a CSV with columns: `message_id, sender, subject, priority_score, action_label, risk_score, classification_latency_ms`. The output is written to stdout by default; pipe to a file with `> weekly_report.csv`. For real‑time monitoring, run `focusinbox metrics --prometheus-listen :9090` to expose a `/metrics` endpoint with histograms for `focusinbox_classification_latency_seconds` and `focusinbox_messages_processed_total`.

### Best Practices

- **Tune Priority Thresholds Based on Historical Response Patterns**  
  Instead of guessing `priority_threshold`, run a one‑week dry‑run with `focusinbox daemon --dry-run --log-dir /var/log/focusinbox`. Analyze the log with `jq` to compute the distribution of `priority_score` for messages that you actually replied to within 4 hours. Set `--priority-threshold` to the 85th percentile of that distribution. This ensures that the “high priority” folder contains only emails that historically required immediate action, reducing false positives by up to 40%.

- **Integrate Snooze Rules with Calendar Event Extraction**  
  Use the `--snooze-rules-file` option to load a JSON array that watches for `ICALENDAR` attachments or `X‑Calendar‑Event` headers. For example, a rule like `{ "match": { "mime_type_contains": "text/calendar" }, "snooze_until": "event_start_minus_15min", "digest": false }` will automatically snooze all calendar‑invite emails until 15 minutes before the event starts. This prevents your inbox from being cluttered with “Accepted” / “Declined” notifications while still surfacing the invite at the moment it becomes actionable.
