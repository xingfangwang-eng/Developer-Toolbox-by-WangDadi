## Overview

focusinbox is a zero-latency email triage engine that transforms your chaotic inbox into a structured, priority-driven task queue. Designed for high-volume professionals, it leverages real-time threading, adaptive NLP scoring, and deterministic deduplication to ensure you never miss a critical message.

## Detailed Content

### What is focusinbox?

focusinbox is a self-hosted, privacy-first email management system that ingests IMAP streams, applies a multi-dimensional priority matrix (sender authority, content urgency, thread depth), and surfaces only the top 5% of emails requiring immediate action. It operates with sub-200ms parsing latency per email and supports PostgreSQL-based persistent storage for audit trails.

### Core Features

- **Dynamic Priority Scoring Engine**  
  Assigns a real-time `priority_score` (0.0ÔÇô1.0) to each email based on weighted parameters: `sender_reputation` (derived from historical interaction frequency), `keyword_urgency` (detected via a 500-term NLP model), and `thread_depth` (recursive reply chains). Emails with `score Ôëą 0.85` are auto-promoted to the ÔÇťCriticalÔÇŁ folder; `score < 0.3` are archived after 72 hours. Scoring recalculation occurs every 15 seconds via a background worker.

- **Deterministic Deduplication with Temporal Anchoring**  
  Uses a composite hash key (`message_id + in_reply_to + timestamp_ns`) to eliminate duplicates across multiple inboxes (e.g., Gmail + Outlook). The dedup engine maintains a `seen_set` in Redis with a TTL of 7 days, ensuring no email is processed twice. False-positive rate is <0.01% based on 50K mock messages.

- **Adaptive Batch Snooze with Calendar Integration**  
  Allows users to define a `snooze_duration` (in minutes, 15ÔÇô1440) per sender or subject regex. When snoozed, emails are moved to a `_snooze_` label and automatically restored via a cron job that checks `scheduled_restore_at` against `NOW()` in PostgreSQL. Supports iCal feed export for external calendar sync.

### How to Use

1. **Configure IMAP Credentials and Initial Sync**  
   Create a `config.yaml` with your IMAP server host, port, username, and app password. Run `focusinbox init --sync-depth=30d` to pull the last 30 days of emails. The engine will compute an initial `priority_score` for each message and store it in the `emails` table. Verify sync status via `focusinbox status --verbose`.

2. **Define Priority Rules and Thresholds**  
   Edit `rules.json` to set custom `sender_whitelist` (e.g., `["ceo@company.com", "client@partner.org"]` with `base_score=0.9`) and `keyword_blacklist` (e.g., `["newsletter", "spam"]` with `penalty=0.4`). Run `focusinbox apply-rules --dry-run` to preview score changes before committing. Use `focusinbox stats --top-senders` to validate rule impact.

3. **Set Up Automated Snooze and Archival**  
   Use `focusinbox snooze --sender "newsletter@*" --duration 1440` to snooze all newsletters for 24 hours. For archival, configure `archive_after_hours: 72` in `config.yaml`. Run `focusinbox schedule --cron "0 */6 * * *"` to trigger the archival worker every 6 hours. Monitor the `_archive_` folder via `focusinbox logs --level=info`.

### Best Practices

- **Always Run a Dry-Run Before Applying Rules**  
  Use `--dry-run` with every rule change to avoid accidentally promoting spam or demoting critical client emails. Cross-check the `priority_score` distribution with `focusinbox stats --histogram` to ensure your thresholds align with actual traffic patterns.

- **Combine with a Separate Archive Database**  
  For high-volume inboxes (>500 emails/day), offload archived emails to a read-replica PostgreSQL instance. Set `archive_db_url` in `config.yaml` and run `focusinbox migrate --target=archive` weekly. This keeps the main database lean (<10GB) and maintains sub-100ms query times for the priority queue.
