# focusinbox - FOC_020

## Overview

focusinbox is a high-performance email triage engine designed for professionals managing 500+ daily inbound messages. It leverages a custom NLP pipeline to classify, prioritize, and surface only the emails that require immediate action, reducing inbox noise by up to 73% in controlled benchmarks. The system is built on a zero-configuration rule engine that adapts to user behavior within 48 hours of deployment.

## Core Features

### 1. Dynamic Priority Scoring with Temporal Decay
Each email is assigned a `priority_score` between 0.0 and 1.0 using a weighted combination of sender reputation (based on historical response latency), thread depth, and keyword sentiment analysis. A temporal decay factor (`λ = 0.15 per hour`) ensures that older, unactioned messages automatically drop below the `action_threshold` (default: 0.65). This prevents inbox paralysis by ensuring only time-sensitive items remain visible. The scoring engine processes 2,800 emails per second on a single `c6i.2xlarge` instance, with a 99.7% accuracy rate against manual labeling.

### 2. Contextual Thread Reconstruction with Cross-Reference Linking
Instead of flat threading, focusinbox builds a directed acyclic graph (DAG) of all related messages, including those from different senders or subjects. It uses a `reference_hash` computed from the `Message-ID`, `In-Reply-To`, and `References` headers, plus a fuzzy match on subject line tokens (Levenshtein distance < 3). The system then surfaces a `context_summary` block that includes the last 5 critical messages, regardless of chronological order. This feature reduces time spent on context-switching by 42% in enterprise trials.

### 3. Automated Action Extraction with Calendar Integration
The NLP parser extracts actionable items (meeting requests, deadlines, approval workflows) into a structured `action_item` object with fields: `due_utc`, `action_type` (enum: `reply`, `approve`, `schedule`, `delegate`), and `confidence` (0.0–1.0). If `confidence > 0.85` and the action type is `schedule`, focusinbox automatically creates a Google Calendar event via OAuth 2.0, with a 15-minute pre-meeting buffer. Conflicts are flagged in the UI with a `calendar_conflict` boolean. Latency from email receipt to calendar creation averages 1.2 seconds.

## How to Use

### Step 1: Configure Sender Reputation Baseline
Run the initial scan to build a reputation matrix. Execute `focusinbox --scan --depth 90d` to analyze all emails in the last 90 days. The system will output a `reputation.json` file containing a `sender_score` for each unique address. Manually override any scores using `focusinbox --override sender@example.com --score 0.95`. This step must complete before priority scoring activates.

### Step 2: Set Action Threshold and Notification Rules
Edit the `config.yaml` file located at `~/.focusinbox/config.yaml`. Set `action_threshold: 0.70` for stricter filtering, or `0.55` for a broader net. Define `notification_rules` as a list of JSON objects, e.g., `[{ "min_priority": 0.85, "method": "push", "channels": ["slack", "desktop"] }]`. Validate the config with `focusinbox --validate`; the parser checks for schema compliance and will reject malformed `due_utc` formats (must be ISO 8601).

### Step 3: Deploy the Listener and Monitor Performance
Launch the background daemon: `focusinbox --daemon --port 8080`. This opens a WebSocket endpoint at `ws://localhost:8080/events`. Use `curl -X GET http://localhost:8080/metrics` to query real-time stats: `emails_processed_total`, `actions_created_total`, and `avg_latency_ms`. To verify thread reconstruction, inspect a specific thread via `focusinbox --thread <message-id> --output json`. The output includes the full DAG with `parent_hash` and `child_hash` fields.

## Best Practices

### 1. Implement a Weekly Reputation Recalibration Cycle
Email patterns shift—new stakeholders join, old contacts become inactive. Schedule a cron job to run `focusinbox --scan --depth 7d` every Sunday at 02:00 UTC. Merge the new `reputation.json` with the existing one using `focusinbox --merge --source latest_reputation.json --target current_reputation.json`. This prevents priority drift and ensures that a once-important sender does not permanently dominate the inbox.

### 2. Use Action Extraction as a Pre-Filter for CRM Integration
Instead of manually logging every client email, configure focusinbox to forward all `action_item` objects with `action_type: 'reply'` and `confidence > 0.9` to a webhook endpoint. Use this to auto-create tasks in tools like Jira or Asana. For example, set `webhook_url: https://your-crm.com/api/tasks` in `config.yaml` under `action_forwarding`. This reduces manual data entry errors by 90% and ensures zero missed follow-ups.
