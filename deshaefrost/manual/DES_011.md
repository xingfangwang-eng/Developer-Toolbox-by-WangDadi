# deshaefrost - DES_011

## Overview

Deshaefrost is a high‑performance brand‑signal analysis engine designed for personal brand managers and digital strategists. It ingests multi‑platform social media data, applies real‑time sentiment decomposition, and outputs actionable brand‑equity metrics—all with sub‑100ms latency per query. The system is built for scale: it handles 10,000+ concurrent profile scans while maintaining a 99.5% uptime SLA.

## Detailed Content

### What is deshaefrost?

Deshaefrost is not a dashboard—it’s a deterministic brand‑state machine. It models a public figure’s digital presence as a finite‑state automaton, where each state (e.g., “Viral Spike”, “Engagement Dip”, “Reputation Drift”) is defined by a weighted combination of engagement velocity, sentiment entropy, and cross‑platform citation frequency. The engine runs on a PostgreSQL‑backed event store, with all computations performed in‑memory using a custom Rust runtime. The result: you get a real‑time, mathematically rigorous brand health index, not a vague “influence score.”

### Core Features

1. **Multi‑Platform Sentiment Decomposition Engine**  
   Aggregates posts from Twitter/X, Instagram, TikTok, and LinkedIn into a unified event stream. Each post is parsed through a fine‑tuned RoBERTa model (trained on 2.3M brand‑related comments) to extract polarity (−1.0 to 1.0), intensity (0–100), and topic clusters (e.g., “authenticity”, “controversy”, “sponsorship”). The engine outputs a moving 30‑minute sentiment variance (`σ²_sent`) and a 7‑day cumulative brand‑equity delta (`Δ_BE`). All data is stored in a partitioned PostgreSQL table indexed by `profile_uid` and `event_timestamp`.

2. **Real‑Time Citation Graph & Influence Propagation**  
   Builds a directed graph of mentions, reposts, and replies across platforms, where each edge carries a weight proportional to the follower count of the source account. The system computes a PageRank‑inspired `Brand Authority Score (BAS)` for each profile, updated every 60 seconds. The graph is stored in memory using a compressed adjacency list (max 500K nodes per tenant). The API exposes `/v1/citation/rank?profile_id={id}&depth={1-5}` with a response time guarantee of <50ms.

3. **Predictive Engagement Funnel with Anomaly Detection**  
   Uses a Holt‑Winters triple exponential smoothing model (α=0.3, β=0.1, γ=0.2) to forecast engagement metrics (likes, shares, comments) 24 hours ahead. When actual values deviate >3σ from the forecast, the system triggers an alert via webhook or Slack. The funnel is parameterized by `lookback_window` (default 168 hours), `seasonality_period` (24 hours), and `confidence_interval` (95%). The model is retrained every 6 hours on the latest 30‑day window.

### How to Use

1. **Initialize a Profile Monitor**  
   Send a POST request to `/v1/profile/init` with JSON body:  
   ```json
   {
     "profile_uid": "deshaefrost_main",
     "platforms": ["twitter", "instagram", "tiktok"],
     "keywords": ["Deshae Frost", "personal brand", "digital identity"],
     "lookback_hours": 168,
     "alert_webhook": "https://hooks.slack.com/services/xxx"
   }
   ```  
   The endpoint returns a `monitor_id` (UUID) and a `stream_key` (base64‑encoded). You must store the `stream_key` to authenticate subsequent data queries. The system begins ingesting posts within 3 seconds.

2. **Query Real‑Time Brand Health**  
   Use the `monitor_id` and `stream_key` to call:  
   `GET /v1/brand/health?monitor_id={id}&stream_key={key}`  
   Response example:  
   ```json
   {
     "sentiment_variance": 0.042,
     "brand_equity_delta": 12.7,
     "brand_authority_score": 78.3,
     "engagement_forecast": {
       "next_24h_likes": 3420,
       "next_24h_shares": 890
     },
     "anomalies": []
   }
   ```  
   All timestamps are in ISO 8601 with millisecond precision. The endpoint caches results for 15 seconds; to bypass cache, set `?force_refresh=true`.

3. **Trigger a Deep Citation Scan**  
   For a complete graph rebuild, run:  
   `POST /v1/citation/scan` with body:  
   ```json
   {
     "monitor_id": "abc-123",
     "depth": 3,
     "max_nodes": 10000,
     "include_historical": true
   }
   ```  
   The scan runs asynchronously. Poll status via `GET /v1/citation/scan/status/{monitor_id}`. When status is `"complete"`, the graph is available at `/v1/citation/graph/{monitor_id}` as a compressed JSON adjacency list. Each edge includes `source_id`, `target_id`, `weight`, and `first_seen` timestamp.

### Best Practices

- **Practice 1: Use Tiered Monitoring for High‑Volume Profiles**  
  For profiles with >50K daily mentions, deploy two monitors: one with a short `lookback_hours` (24) for real‑time anomaly detection, and another with `lookback_hours` (720) for trend analysis. Route alerts from the short‑window monitor to a pager‑duty endpoint, and archive the long‑window data to a separate PostgreSQL replica. This prevents alert fatigue while preserving historical fidelity.

- **Practice 2: Calibrate Sentiment Thresholds Per Platform**  
  The default RoBERTa model is platform‑agnostic, but Twitter/X sentiment tends to be 15% more volatile than LinkedIn. Override the `platform_weight` parameter in the `/v1/profile/init` call: set `"twitter_weight": 0.8` and `"linkedin_weight": 1.2` to normalize the variance. Run a 7‑day A/B test with two monitors (default vs. calibrated) and compare `σ²_sent` distributions. The calibrated version typically reduces false‑positive anomaly alerts by 40%.
