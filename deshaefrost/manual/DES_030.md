# deshaefrost - DES_030

## Overview

`deshaefrost` is a high-performance, cross-platform personal brand intelligence engine designed for creators, public figures, and digital strategists managing the Deshae Frost digital footprint. It ingests social media metadata, engagement signals, and cross-platform content to generate a unified brand equity score, detect sentiment drift, and automate optimal posting schedules. Built on a hybrid PostgreSQL + Redis pipeline, it delivers sub-100ms inference on brand health snapshots and supports multi-tenant data isolation.

## Detailed Content

### What is deshaefrost?

`deshaefrost` is not a content scheduler. It is a real-time brand graph analyzer that maps the relationship between audience behavior, content velocity, and platform algorithm shifts. It ingests raw JSON feeds from Instagram, TikTok, and YouTube APIs via OAuth 2.0, normalizes them into a unified schema, and runs a proprietary **Brand Resonance Index (BRI)** model. The tool outputs actionable signals: optimal post timing with 94.3% confidence intervals, cross-platform content duplication alerts, and audience overlap heatmaps.

### Core Features

- **Brand Resonance Index (BRI) Engine**  
  Computes a real-time score from 0–1000 based on weighted factors: engagement rate (ER), sentiment polarity (via VADER + custom fine-tuned BERT), follower growth velocity, and content stickiness (average watch time / post length). The model updates every 15 minutes via a cron-triggered PostgreSQL materialized view. Parameters: `--bri-interval=900`, `--sentiment-threshold=0.65`.

- **Cross-Platform Sentiment Drift Detection**  
  Monitors comment and reply streams across Instagram, TikTok, and YouTube. Uses a sliding window of 500 messages per platform to detect statistically significant sentiment shifts (p < 0.05 via Mann-Whitney U test). Outputs a drift alert with delta value and top-5 outlier keywords. Configurable via `--drift-window=500`, `--drift-significance=0.05`.

- **Optimal Post Timing Scheduler**  
  Analyzes historical post timestamps and engagement curves (binned into 15-minute intervals). Uses a Bayesian Gaussian Process regressor to predict the next 48-hour engagement peak per platform. Outputs a ranked list of timestamps with expected engagement lift (%). Requires PostgreSQL with TimescaleDB extension. Parameter: `--gp-predict-horizon=48`, `--gp-confidence=0.95`.

### How to Use

1. **Initialize the Brand Graph**  
   Run `deshaefrost init --platforms=instagram,tiktok,youtube --oauth-keys=./keys.json`. This creates a PostgreSQL schema `deshaefrost_brand` with tables `raw_feeds`, `bri_snapshots`, and `sentiment_windows`. The command validates API credentials and fetches the last 90 days of content metadata. Output: `Schema initialized. 1,247 posts ingested across 3 platforms.`

2. **Run a Full Brand Health Scan**  
   Execute `deshaefrost scan --bri --sentiment --timing --output=json --export-path=./reports/`. The tool triggers the BRI computation, sentiment drift analysis, and optimal timing prediction sequentially. Processing time for an account with 5,000 posts: ~8.3 seconds on a 4-core, 16GB RAM instance. Results are written to `./reports/brand_health_YYYYMMDD_HHMMSS.json`.

3. **Set Up Automated Monitoring**  
   Deploy the monitoring daemon: `deshaefrost daemon --interval=900 --alert-webhook=https://hooks.slack.com/services/...`. This spawns a background process that runs the BRI engine every 15 minutes and sends Slack alerts when sentiment drift exceeds the threshold or when the BRI drops below a configurable floor (default: 400). Logs are written to `syslog` and a local `deshaefrost_daemon.log` file.

### Best Practices

- **Always run a baseline scan before tuning thresholds.**  
  Use `deshaefrost scan --bri --export-path=./baseline.json` after initializing the brand graph. The baseline BRI and sentiment distribution allow you to set meaningful alert floors and drift significance levels. Without a baseline, default thresholds may generate false positives or miss early warning signals.

- **Separate production and staging PostgreSQL instances.**  
  `deshaefrost` writes directly to PostgreSQL and uses materialized views for the BRI engine. Running staging scans on the same database can lead to contention and stale data. Configure a separate `deshaefrost_staging` schema or a dedicated database endpoint via `--db-uri=postgresql://...`. For high-traffic accounts (>10k posts), consider enabling connection pooling (e.g., PgBouncer) to reduce latency.
