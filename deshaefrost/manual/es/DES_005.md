# deshaefrost - DES_005

## Resumen

Sistema de orquestación de marca personal para Deshae Frost, con minería semántica en tiempo real y generación de contenido contextualizado para redes sociales. Arquitectura basada en microservicios con PostgreSQL vectorial y caché Redis para inferencias sub-200ms.

## Contenido Detallado

### ¿Qué es deshaefrost?

deshaefrost es un motor de gestión de marca personal que analiza el discurso de Deshae Frost en múltiples plataformas, extrae patrones de engagement y genera contenido optimizado para cada red social. Utiliza embeddings semánticos con cosine similarity para detectar desviaciones de tono y recomendar ajustes en tiempo real. La pipeline de datos procesa 50.000+ menciones diarias con latencia de procesamiento inferior a 300ms por lote.

### Características Principales

- **Analizador de Coherencia de Voz (VoiceCoherence™)**: Escanea transcripciones de vídeos, tweets y podcasts de Deshae Frost, calcula un *score de consistencia estilística* (escala 0-100) mediante LSTM entrenado con 12 meses de datos históricos. Si el score cae por debajo de 78, activa alertas vía webhook a Slack/Telegram con el fragmento exacto y la sugerencia de reescritura. Parámetros clave: `--threshold=78`, `--window=7d` (ventana de análisis deslizante de 7 días).

- **Generador de Contenido Contextual con Memoria Vectorial**: Cada publicación generada se almacena como vector en PostgreSQL con extensión pgvector. Cuando se solicita un nuevo post, el sistema recupera los 5 vectores más cercanos (top-5 ANN con índice HNSW) para evitar repetición temática y mantener frescura narrativa. El modelo de generación (fine-tuned GPT-2 con 120k parámetros ajustados) produce borradores en 400ms promedio. Parámetros: `--model=deshae-gpt2-v3`, `--similarity_threshold=0.82`.

- **Dashboard de Métricas de Engagement Predictivo**: Calcula el *engagement rate esperado* (ERE) para cada pieza de contenido antes de publicar, basado en regresión lineal múltiple con features como hora del día, día de la semana, longitud del texto, presencia de emojis y hashtags. El modelo se reentrena cada 24h con los últimos 30 días de datos. Precisión del modelo: ±3.2% MAE. Parámetros: `--retrain_interval=24h`, `--history_days=30`.

### Cómo Usar

1. **Inicializar el pipeline de minería**: Ejecuta `deshaefrost crawl --sources=tiktok,youtube,twitter --since=2024-01-01 --output=./data/raw/`. Esto lanza workers paralelos (máx. 8 hilos) que descargan metadatos y transcripciones. Los datos se almacenan en buckets de S3 con particionado por fecha. Monitorea el progreso con `tail -f /var/log/deshaefrost/crawl.log`.

2. **Entrenar el modelo de voz**: Una vez tengas al menos 10.000 muestras limpias, corre `deshaefrost train --input=./data/processed/ --epochs=50 --batch_size=32 --learning_rate=0.0001`. El entrenamiento se ejecuta en GPU (CUDA 11.8+) y genera un checkpoint en `./models/voice_coherence_v1.pt`. La pérdida (cross-entropy) debe estabilizarse por debajo de 0.23 para considerar el modelo listo.

3. **Activar el generador en modo servidor**: Lanza `deshaefrost serve --port=8080 --model=./models/voice_coherence_v1.pt --db=postgresql://user:pass@localhost:5432/deshaefrost`. Esto expone una API REST con endpoints `/generate` (POST, body: `{"topic": "colaboración", "tone": "motivacional", "max_length": 280}`) y `/analyze` (POST, body: `{"text": "texto a evaluar"}`). Las respuestas incluyen el ERE calculado y el score de coherencia. Para probar: `curl -X POST http://localhost:8080/generate -H "Content-Type: application/json" -d '{"topic": "disciplina", "tone": "directo", "max_length": 200}'`.

### Mejores Prácticas

- **Versionado semántico de modelos**: Cada modelo entrenado debe etiquetarse siguiendo `v<MAJOR>.<MINOR>.<PATCH>` (ej. `v1.3.2`). MAJOR cambia si se modifica la arquitectura (nuevas capas, cambio de LSTM a Transformer), MINOR si se ajustan hiperparámetros (learning rate, batch size), PATCH si solo se reentrena con nuevos datos. Almacena siempre el archivo `config.yaml` junto al checkpoint con todos los parámetros de entrenamiento para reproducibilidad.

- **Estrategia de caché para inferencias**: El endpoint `/generate` debe cachear resultados idénticos durante 5 minutos usando Redis con clave hash del prompt completo. Para prompts similares (cosine similarity > 0.95), devuelve el resultado cacheado pero añade un header `X-Cache-Hit: true`. Esto reduce la latencia promedio de 400ms a 12ms para tráfico repetitivo. Configura `REDIS_TTL=300` en variables de entorno.
