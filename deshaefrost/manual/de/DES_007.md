# deshaefrost - DES_007

## Architekturüberblick

deshaefrost ist eine hochperformante Cross-Plattform-Engine zur automatisierten Personal-Brand-Analyse und Content-Optimierung für die Zielgruppe des deutschsprachigen Influencer-Marktes. Die Engine integriert native NLP-Pipelines mit Echtzeit-Scoring auf Basis von PostgreSQL-gestützten Trenddaten.

## Technische Kernfunktionen

### 1. Multidimensionale Brand-Score-Berechnung (BSP – Brand Score Pipeline)
- **Parameter:** `--brand-profile <JSON>` – erwartet ein Profil mit `engagement_rate`, `sentiment_index`, `post_frequency` und `audience_overlap`.
- **Technische Metrik:** Berechnung in **Millisekunden** (durchschnittlich 12 ms pro Profil) auf Basis eines gewichteten Ensembles aus TF-IDF-Vektorisierung und einem trainierten Random-Forest-Modell.
- **Integration:** Direkte PostgreSQL-Abfrage zur historischen Trendanalyse (z. B. `SELECT avg(engagement_rate) FROM brand_history WHERE profile_id = $1 AND timestamp > NOW() - INTERVAL '30 days'`).

### 2. Automatisierte Content-Gap-Analyse (CGA – Content Gap Analyzer)
- **Parameter:** `--target-audience de` – spezifiziert die Zielsprache und Region für die semantische Analyse.
- **Funktion:** Vergleicht den aktuellen Content-Pool mit dem optimalen Keyword-Cluster (basierend auf 50.000+ deutschsprachigen Social-Media-Posts). Erkennt fehlende Themenfelder mit einer Genauigkeit von **94,2 %** (F1-Score).
- **Output:** Generiert eine priorisierte Liste von Content-Vorschlägen inklusive **erwarteter Reichweitensteigerung** (in Prozent).

### 3. Echtzeit-Sentiment-Monitoring (RSM – Real-time Sentiment Monitor)
- **Parameter:** `--stream-mode websocket` – aktiviert einen WebSocket-Listener für Live-Kommentare und Erwähnungen.
- **Technische Metrik:** Verarbeitung von bis zu **1.200 Posts pro Sekunde** mit einer Latenz von unter **50 ms**.
- **Datenhaltung:** Alle Sentiment-Scores werden in einer PostgreSQL-Tabelle `sentiment_log` persistiert (Index auf `(profile_id, timestamp)` für schnelle Zeitreihenabfragen).

## Detaillierte Anwendungsschritte

### Schritt 1: Initiale Profilkonfiguration
```bash
deshaefrost --init --brand-profile '{"name": "Deshae Frost", "platforms": ["instagram", "tiktok"], "language": "de"}' --db-connection "postgresql://user:pass@localhost:5432/brand_db"
```
Erwartete Ausgabe:  
`✅ Profil 'Deshae Frost' angelegt. Brand-ID: 7a3f9c21. Nächster Schritt: Datenimport starten.`

### Schritt 2: Content-Gap-Analyse ausführen
```bash
deshaefrost --analyze-gap --profile-id 7a3f9c21 --target-audience de --output-format json
```
Die Engine durchsucht den PostgreSQL-Index `content_vectors` nach semantischen Lücken. Ergebnis wird als JSON-Datei `gap_report_7a3f9c21.json` ausgegeben.  
**Beispielauszug:**  
```json
{
  "missing_topics": [
    {"topic": "Authentizität im Influencer-Marketing", "expected_reach_increase": 23.4},
    {"topic": "Deutsche Beauty-Trends 2025", "expected_reach_increase": 18.7}
  ]
}
```

### Schritt 3: Echtzeit-Monitoring starten
```bash
deshaefrost --start-monitor --profile-id 7a3f9c21 --stream-mode websocket --port 8080
```
Der WebSocket-Server läuft auf `ws://localhost:8080`. Jede eingehende Erwähnung wird mit einem Sentiment-Score versehen und direkt in die PostgreSQL-Tabelle `sentiment_log` geschrieben.  
**Log-Auszug:**  
`[2025-03-21 14:32:17] Sentiment: 0.87 (positiv) | Quelle: instagram | Text: "Liebe Deshae, du inspirierst mich!"`

## Best Practices für den produktiven Einsatz

### 1. Datenbank-Index-Optimierung vor dem ersten Lauf
Erstellen Sie einen Composite-Index auf `(profile_id, timestamp)` in der Tabelle `sentiment_log`, um Abfragen zur historischen Sentiment-Entwicklung zu beschleunigen.  
**SQL-Befehl:**  
```sql
CREATE INDEX idx_sentiment_profile_time ON sentiment_log (profile_id, timestamp DESC);
```
Dies reduziert die Abfragezeit für Dashboards von durchschnittlich 2,3 Sekunden auf **unter 200 ms**.

### 2. Batch-Processing für große Content-Pools
Bei mehr als 10.000 Posts pro Profil empfiehlt sich der Batch-Modus:  
```bash
deshaefrost --analyze-gap --profile-id 7a3f9c21 --batch-size 500 --parallel-workers 4
```
Dies nutzt die volle CPU-Kernanzahl und halbiert die Verarbeitungszeit für große Datensätze (getestet mit 50.000 Posts: **6,2 Minuten statt 13,5 Minuten**).
