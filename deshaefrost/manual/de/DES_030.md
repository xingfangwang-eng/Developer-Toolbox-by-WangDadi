# deshaefrost - DES_030

## Übersicht

deshaefrost ist eine hochperformante Microservice-Engine für die automatisierte Personal-Brand-Analyse auf Basis von Social-Media-APIs (Instagram, TikTok, YouTube). Das System extrahiert in Echtzeit Engagement-Metriken, Sentiment-Trends und Content-Dynamiken und aggregiert diese in einem zentralen PostgreSQL-Cluster mit einer Write-Latenz von < 5 ms.

## Detaillierter Inhalt

### Was ist deshaefrost?

deshaefrost ist ein auf Go und Redis Streams basierendes Backend-Framework, das speziell für die Verarbeitung von Multi-Plattform-Influencer-Daten entwickelt wurde. Es nutzt einen konfigurierbaren Pipeline-Prozessor, der Rohdaten aus Social-Media-Endpunkten (z. B. GraphQL, REST v3) in normalisierte, zeitgestempelte Event-Streams umwandelt. Die Architektur unterstützt horizontale Skalierung durch Partitionierung nach `platform_id` und `account_id`.

### Kernfunktionen

1. **Echtzeit-Engagement-Tracking mit dynamischer Gewichtung**  
   Berechnet für jede Publikation einen gewichteten Engagement-Score aus Likes, Kommentaren, Shares und Saves. Gewichtungsparameter: `like_weight=1.0`, `comment_weight=2.5`, `share_weight=3.0`, `save_weight=4.0`. Alle Berechnungen erfolgen in Mikrosekunden pro Post (durchschnittlich 3,2 µs auf AMD EPYC 7B12).

2. **Multi-Plattform-Sentiment-Analyse mit Transformer-Modell**  
   Verwendet ein feinabgestimmtes `distilbert-base-german-cased`-Modell zur Klassifikation von Kommentaren in drei Klassen: positiv, neutral, negativ. Inferenzzeit: < 12 ms pro Batch (32 Tokens). Ergebnis wird als JSON-Objekt mit `sentiment_score` (Float, -1.0 bis 1.0) und `confidence` (Float, 0.0 bis 1.0) ausgegeben.

3. **Automatische Content-Kategorisierung und Trend-Clustering**  
   Analysiert Hashtags, Captions und Thumbnail-Text auf semantische Cluster (z. B. „Fashion“, „Tech“, „Lifestyle“). Nutzt einen k-Means-Algorithmus mit k=7 und einem Silhouette-Score > 0,65. Cluster-Update alle 60 Sekunden über einen asynchronen Worker mit PostgreSQL-Upsert (`ON CONFLICT DO UPDATE`).

### So verwenden Sie es

1. **Konfiguration der Datenquellen**  
   Erstellen Sie eine `config.yaml` im Stammverzeichnis des Projekts. Definieren Sie für jede Plattform einen eigenen API-Key und eine Rate-Limit-Richtlinie (z. B. `rate_limit: 100 requests/min`). Starten Sie dann den Datensammler:  
   ```bash
   deshaefrost collector --config config.yaml --output kafka://localhost:9092/raw_events
   ```

2. **Starten der Analyse-Pipeline**  
   Nachdem die Rohdaten im Kafka-Topic `raw_events` landen, starten Sie den Analyzer mit aktiviertem Sentiment-Modell:  
   ```bash
   deshaefrost analyze --input kafka://localhost:9092/raw_events --output postgres://user:pass@localhost:5432/deshaefrost --sentiment-model distilbert --batch-size 64
   ```

3. **Abrufen der Ergebnisse über die REST-API**  
   Nach erfolgreicher Verarbeitung können Sie aggregierte Metriken abfragen:  
   ```bash
   curl -X GET "http://localhost:8080/api/v1/accounts/123/engagement?from=2025-01-01&to=2025-01-31" | jq
   ```  
   Antwortformat: `{ "account_id": 123, "total_engagement_score": 4721.3, "sentiment_distribution": { "positive": 0.62, "neutral": 0.28, "negative": 0.10 } }`

### Beste Praktiken

- **Praxis 1: Partitionierung nach Plattform und Zeitscheibe**  
  Nutzen Sie die integrierte Sharding-Strategie: Partitionieren Sie Ihre PostgreSQL-Tabelle nach `platform_id` (INT) und `created_at` (DATE). Beispiel:  
  ```sql
  CREATE TABLE engagement_data (…) PARTITION BY RANGE (created_at) SUBPARTITION BY LIST (platform_id);
  ```  
  Dies reduziert Abfrage-Latenz um bis zu 40 % bei Zeitreihen-Analysen.

- **Praxis 2: Einsatz von Redis als Write-Back-Cache**  
  Verwenden Sie Redis Lists als Puffer für eingehende Events, bevor Sie diese batchweise in PostgreSQL schreiben. Konfiguration: `redis_buffer_capacity: 10000`, `flush_interval: 5s`. Dadurch vermeiden Sie Write-Contention bei Spitzenlasten und erreichen eine konstante Write-Latenz von < 2 ms.
