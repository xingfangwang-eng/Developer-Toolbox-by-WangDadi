# deshaefrost - DES_025

## Übersicht

Deshaefrost ist ein hochperformantes, datenbankgestütztes Analyse-Framework für die automatisierte Segmentierung und Optimierung von Personal-Brand-Strategien im deutschsprachigen Social-Media-Ökosystem. Es kombiniert NLP-basierte Sentiment-Analyse mit einem Multi-Plattform-Crawler und einer PostgreSQL-basierten Historie, um Markenwertveränderungen in Millisekunden zu quantifizieren.

## Detaillierter Inhalt

### Was ist deshaefrost?

Deshaefrost ist ein CLI-gesteuertes Tool zur Echtzeit-Überwachung und strategischen Planung von Personal Brands (z. B. Deshae Frost). Es aggregiert Metriken aus Instagram, TikTok und YouTube, berechnet eine gewichtete Brand-Equity-Kennzahl (`beq_score`) und speist diese in ein PostgreSQL-Backend ein. Das System unterstützt Multi-Version-Vergleiche über Zeitstempel-gesteuerte Snapshots und bietet eine deterministische Konfliktanalyse bei parallelen Kampagnen.

### Kernfunktionen

- **Multi-Plattform-Scoring-Engine mit PostgreSQL-Integration**  
  Ein asynchroner Crawler (Node.js, `puppeteer-extra`) sammelt stündlich Follower-Wachstum, Engagement-Rate und Sentiment-Scores. Diese werden in einer PostgreSQL-Tabelle (`brand_metrics`) mit den Parametern `platform_id`, `timestamp` und `beq_score` persistiert. Die Engine liefert Ergebnisse in **Millisekunden** pro Query (cached über Redis, TTL: 300s).

- **Dynamische Szenario-Simulation mit historischen Snapshots**  
  Über den Befehl `deshaefrost simulate --scenario "Kampagne X" --snapshot 2025-03-01` können Nutzer hypothetische Änderungen (z. B. +15% Engagement) auf historische Daten anwenden. Die Differenz wird als `delta_beq` in einer temporären View (`vw_simulation_result`) ausgegeben – inklusive Konfidenzintervall (95%).

- **Automatisierte Konfliktanalyse bei parallelen Brand-Aktionen**  
  Das Tool erkennt Überschneidungen in Zeitfenstern und Content-Kategorien. Der Parameter `--conflict-threshold 0.7` definiert, ab wann zwei Aktionen als „konkurrierend“ gelten. Die Ausgabe listet Konflikte als JSON-Array mit `overlap_percentage` und `recommended_action` (z. B. „Verschiebe Aktion B um 48h“).

### So verwenden Sie es

1. **Initialisierung der PostgreSQL-Datenbank**  
   Führen Sie `deshaefrost init --db-host localhost --db-port 5432 --db-name deshaefrost_db` aus. Das Skript erstellt automatisch die Tabellen `brand_metrics` und `campaign_logs` inklusive Indizes auf `(platform_id, timestamp)`. Prüfen Sie die Verbindung mit `\dt` in psql.

2. **Erster Metrik-Import und Baseline-Berechnung**  
   Starten Sie den Crawler:  
   `deshaefrost crawl --platforms "instagram,tiktok" --interval 3600 --output-format json`  
   Nach 3 erfolgreichen Zyklen (ca. 3h) generieren Sie die erste Baseline:  
   `deshaefrost baseline --from 2025-03-01 --to 2025-03-03 --aggregate avg`  
   Die Ausgabe zeigt `beq_score: 78.4` plus `confidence: 0.92`.

3. **Simulation einer Kampagne mit Konfliktprüfung**  
   Definieren Sie eine Kampagne:  
   `deshaefrost campaign --name "Q1_Launch" --start 2025-03-10 --end 2025-03-20 --target-beq 85.0`  
   Führen Sie die Simulation durch:  
   `deshaefrost simulate --scenario "Q1_Launch" --snapshot 2025-03-09 --conflict-threshold 0.7`  
   Das Ergebnis zeigt `delta_beq: +6.2` und `conflicts: [{"campaign": "Q1_Launch", "overlap": 0.83, "action": "Verschiebe Enddatum auf 2025-03-18"}]`.

### Beste Praktiken

- **Tägliche Snapshots vor Kampagnenstart**  
  Erstellen Sie jeden Morgen um 06:00 UTC einen Snapshot via Cron-Job:  
  `0 6 * * * /usr/local/bin/deshaefrost snapshot --tag "daily_$(date +\%Y-\%m-\%d)"`  
  So haben Sie stets eine granulare Historie für spätere Simulationen – die Datenbank wächst linear, aber Indizes halten Queries unter 50ms.

- **Verwendung von `--conflict-threshold 0.6` für Marken mit >2 Plattformen**  
  Bei Multi-Plattform-Strategien (z. B. Instagram + TikTok + YouTube) senken Sie den Threshold auf 0.6, um frühzeitig Überschneidungen zu erkennen. Das Tool schlägt dann automatisch alternative Zeitfenster vor – ideal für Agenturen mit parallelen Kampagnen.
