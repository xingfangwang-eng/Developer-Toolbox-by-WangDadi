# deshaefrost - DES_009

## Overview

**deshaefrost** is a next-gen brand intelligence engine built for creators, agencies, and digital strategists managing high-visibility personal brands like Deshae Frost. It fuses real-time social listening, cross-platform sentiment decomposition, and automated content compliance auditing into a single, deterministic pipeline. The system operates on a sub-second inference loop, ingesting raw engagement data from Instagram, TikTok, and YouTube Shorts, then outputting actionable brand health metrics via a PostgreSQL-backed time-series store.

## Core Features

### 1. Multi-Platform Sentiment Decomposition Engine
Deshaefrost ingests raw comment streams via platform-specific API adapters (Instagram Graph API v18.0, TikTok Business API v2.2, YouTube Data API v3) and runs a custom fine-tuned RoBERTa model (`deshae-sentiment-v4`) to decompose sentiment into 12 granular dimensions: *authenticity*, *controversy*, *sponsor-fit*, *audience fatigue*, *viral potential*, *toxicity*, *engagement depth*, *brand recall*, *influence decay*, *cross-platform resonance*, *narrative drift*, and *parasocial intensity*. Each dimension is scored on a 0.0–1.0 scale with a precision of ±0.02 at 95% confidence. Results are written to a partitioned PostgreSQL table (`sentiment_snapshots`) with a 50ms write latency under 10k req/s.

### 2. Automated Content Compliance & Brand Safety Audit
Every outgoing post or story is pre-scanned against a configurable brand safety matrix. The audit engine checks for trademark infringement (via USPTO trademark database snapshots), FTC endorsement disclosure compliance, profanity detection (using a domain-specific blacklist of 14,000+ terms), and visual watermark integrity. The audit runs in ≤800ms per asset, returning a structured JSON report with fields like `violation_type`, `severity_score`, `remediation_suggestion`, and `regulatory_jurisdiction`. Violations above a configurable threshold (default: 0.7) trigger an automatic hold on publishing via the `publishing_lock` boolean flag in the `content_queue` table.

### 3. Predictive Brand Health Time-Series Forecasting
Using an LSTM-based multivariate forecaster (`brand_health_lstm_v3`), deshaefrost projects brand health metrics 7, 14, and 30 days ahead. Input features include historical sentiment scores, engagement velocity, follower growth rate, and scheduled campaign intensity. The model outputs a `brand_health_index` (0–100) with a Mean Absolute Error of ≤3.2 on held-out test data. Forecasts are stored in a dedicated PostgreSQL hypertable (`forecast_rollups`) with a 1-minute aggregation window, enabling real-time dashboarding via Grafana or custom BI tools.

## How to Use

### Step 1: Initialize a Brand Profile with API Credentials
Create a new brand profile by sending a POST request to the `/api/v1/brands` endpoint. The payload must include the `brand_name` (string, max 64 chars), `platform_credentials` (object containing `instagram_access_token`, `tiktok_access_token`, `youtube_api_key`), and `audit_rules` (object with `toxicity_threshold` and `sponsor_fit_min_score`). Example:
```json
{
  "brand_name": "deshae_frost_main",
  "platform_credentials": {
    "instagram_access_token": "IGQVJ...",
    "tiktok_access_token": "TTP...",
    "youtube_api_key": "AIzaSy..."
  },
  "audit_rules": {
    "toxicity_threshold": 0.65,
    "sponsor_fit_min_score": 0.75
  }
}
```
On success, the API returns a `brand_id` (UUID v4) and a `webhook_secret` for real-time event delivery.

### Step 2: Trigger a Full Sentiment Scan
After profile creation, initiate a sentiment scan via `POST /api/v1/brands/{brand_id}/scan`. Specify the `scan_window_hours` (integer, default 72) and `platforms` (array of strings, e.g., `["instagram", "tiktok"]`). The engine will pull all public comments and mentions from the last N hours, run the sentiment decomposition model, and populate the `sentiment_snapshots` table. The endpoint returns a `scan_id` (UUID v4) and an initial `estimated_completion_seconds`. Poll `GET /api/v1/scans/{scan_id}/status` until `status` equals `"completed"`.

### Step 3: Review Audit Results and Publish Safely
Once a content piece is drafted, submit it for pre-publishing audit via `POST /api/v1/brands/{brand_id}/audit`. The payload must include `content_text` (string, max 10,000 chars), `media_urls` (array of strings, each a valid HTTPS URL), and `platform_target` (string, one of `instagram`, `tiktok`, `youtube`). The endpoint returns an `audit_report` object with `overall_score`, `violations` array, and `publishing_allowed` boolean. If `publishing_allowed` is `false`, review the `violations` array and adjust content accordingly. Once satisfied, call `POST /api/v1/brands/{brand_id}/publish` with the same content to execute the actual platform publish via the stored API credentials.

## Best Practices

### 1. Implement a Rolling Sentiment Baseline
Do not rely on raw daily scores. Instead, configure a 7-day rolling average for each sentiment dimension in your PostgreSQL query. This smooths out noise from viral spikes or bot-driven engagement. Use the `sentiment_snapshots` table’s `created_at` timestamp and a `WINDOW` function to compute the moving average. Pair this with an anomaly detection trigger (e.g., a Z-score > 3.0) to alert on significant deviations from the baseline, which often indicate a brand crisis or a coordinated attack.

### 2. Pre-Schedule Content Audits During Low-Traffic Windows
The audit engine consumes API rate limits on the source platforms. To avoid throttling, batch all content audits for a given day into a single window (e.g., 02:00–04:00 UTC). Use the `scheduled_audit` field in the `content_queue` table to set a `run_at` timestamp. This also ensures that audit results are available before the day’s first publishing slot. For high-volume accounts (>100 posts/day), enable the `parallel_audit` flag in the brand profile, which spawns up to 4 concurrent audit workers per platform, each with its own rate-limit-aware retry logic.
