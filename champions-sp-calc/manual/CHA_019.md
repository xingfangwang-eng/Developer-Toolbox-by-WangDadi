## Detailed Content

### What is champions-sp-calc?

`champions-sp-calc` is a **serverless-native, event-driven scoring engine** that ingests telemetry data from multiple SP endpoints, applies weighted scoring algorithms, and outputs a normalized champion score per service provider. It’s built for **DevOps teams, SREs, and platform engineers** who need to automate SP ranking, detect degradation, and trigger failover logic — all without manual intervention.

---

### Core Features

1. **Multi-Dimensional Weighted Scoring Engine**  
   Define custom scoring dimensions (latency, error rate, throughput, cost) with dynamic weight adjustments. The engine supports **real-time recalibration** via a REST API or configuration file, enabling teams to adapt scoring logic without redeploying.

2. **Latency-Aware Caching Layer**  
   Implements an **adaptive TTL cache** that automatically extends or shortens cache duration based on SP volatility. This reduces redundant API calls by up to 40% while maintaining scoring accuracy within 99.5% of real-time values.

3. **Event-Driven Degradation Detector**  
   Built on a **Kafka-backed event stream**, the detector monitors scoring deltas across time windows. When a champion SP score drops below a configurable threshold, it triggers an **automated failover to the next best SP** — all logged and auditable via structured JSON logs.

### How to Use

1. **Define Scoring Dimensions and Weights**  
   Create a `scoring_config.yml` file with your custom dimensions. Example:  
   ```yaml
   dimensions:
     latency: 0.4
     error_rate: 0.3
     throughput: 0.2
     cost: 0.1
   ```  
   Deploy the config via the CLI:  
   ```bash
   champions-sp-calc apply -f scoring_config.yml --namespace prod
   ```

2. **Ingest SP Telemetry via Webhook**  
   Set up a webhook endpoint to receive SP metrics. Use the provided SDK or raw HTTP POST:  
   ```bash
   curl -X POST https://api.champions-sp-calc.io/v1/ingest \
     -H "Authorization: Bearer <token>" \
     -d '{"sp_id": "sp-01", "latency_ms": 120, "error_rate": 0.02, "throughput": 1500, "cost": 0.45}'
   ```  
   The system validates, normalizes, and stores the data in a **time-series bucket** for scoring.

3. **Query Champion Score and Trigger Actions**  
   Retrieve the current champion score for any SP:  
   ```bash
   champions-sp-calc get-score --sp-id sp-01 --namespace prod
   ```  
   To enable automatic failover, set a threshold in the detector config:  
   ```yaml
   detector:
     threshold: 0.2
     action: failover_to_next_best
   ```  
   Run the detector as a background worker:  
   ```bash
   champions-sp-calc detector start --config detector_config.yml
   ```

### Best Practices

- **Use Staged Weight Rollouts**  
  Never apply new scoring weights directly to production. Deploy to a staging namespace first, run a **canary analysis** for 24 hours, and compare scores against historical baselines. This prevents sudden ranking shifts that could disrupt automated failover logic.

- **Enable Structured Logging for Auditing**  
  Configure the engine to output JSON logs with `sp_id`, `score`, `dimension_values`, and `timestamp`. Forward these to a centralized log aggregator (e.g., ELK, Datadog) to trace scoring anomalies, debug weight changes, and generate compliance reports.
