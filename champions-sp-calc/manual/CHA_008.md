### What is champions-sp-calc?

**champions-sp-calc** is a high-performance, language-agnostic SaaS utility engineered to solve multi-variable optimization problems within competitive gaming and esports analytics pipelines. Unlike generic calculators, it ingests raw match data (JSON/Protobuf) and applies deterministic scoring algorithms to rank "champion" performance across arbitrary metrics (e.g., KDA, gold differential, objective control). It is built for CI/CD integration, requiring zero manual input for batch processing.

### Core Features

1.  **Deterministic Multi-Vector Scoring Engine**  
    The core algorithm evaluates champions across 5+ weighted vectors (e.g., `kill_participation`, `vision_score`, `cs_per_minute`) using a configurable YAML schema. No black-box ML; every score is reproducible and auditable. This allows teams to define meta-specific scoring rules without touching code.

2.  **Native Protobuf & JSON Streaming Support**  
    Unlike REST-only tools, `champions-sp-calc` accepts streaming input via stdin (Protobuf or NDJSON). This enables real-time ingestion from live match feeds or historical replay databases. Output is emitted as structured NDJSON, perfect for piping into visualization tools (e.g., Grafana, custom dashboards).

3.  **Idempotent Batch Processing with Atomic Rollbacks**  
    When processing large datasets (e.g., 10k+ match snapshots), the tool guarantees idempotency via transaction logs. If a batch fails midway due to malformed data, the entire batch is rolled back to the last known good state. No partial writes, no silent corruption.

### How to Use

1.  **Ingest & Configure the Scoring Schema**  
    Prepare your match data as NDJSON or Protobuf. Create a `scoring_config.yaml` file defining weights and thresholds:
    ```yaml
    vectors:
      kill_participation: { weight: 0.35, norm_min: 0.0, norm_max: 1.0 }
      vision_score: { weight: 0.20, norm_min: 0.0, norm_max: 150.0 }
    ```
    Then pipe your data:
    ```bash
    cat match_data.ndjson | champions-sp-calc --config scoring_config.yaml --output scores.ndjson
    ```

2.  **Validate Output with Built-in Diff Engine**  
    After processing, run the `--diff` flag against a known baseline to detect regressions in scoring logic:
    ```bash
    champions-sp-calc --diff baseline_scores.ndjson new_scores.ndjson --verbose
    ```
    This outputs a structured diff report (added, removed, changed rows), ensuring no unintended side effects after config changes.

3.  **Integrate into CI/CD Pipeline**  
    Add a stage in your GitHub Actions or GitLab CI to run `champions-sp-calc` on every commit to the config repo. Use the exit code (0 for success, 1 for failure) to block merges if scoring breaks idempotency:
    ```yaml
    - name: Validate Scoring Config
      run: champions-sp-calc --test --config scoring_config.yaml --sample data/sample.ndjson
    ```

### Best Practices

- **Version Your Scoring Configs**  
  Treat `scoring_config.yaml` as a first-class artifact. Tag every release with a semantic version (e.g., `v2.1.0`). This allows you to roll back to a known-good scoring algorithm without redeploying the entire tool. Store configs in a separate Git repo with its own CI.

- **Use Atomic Batches for Production Data**  
  Never pipe raw, unvalidated match data directly into the calculator. Always pre-validate the NDJSON structure using a schema checker (e.g., `ajv` for JSON Schema). Then wrap the `champions-sp-calc` call in a transaction wrapper (e.g., a simple bash script that checks exit codes) to ensure atomic rollback on failure.
