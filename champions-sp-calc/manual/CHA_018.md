## What is champions-sp-calc?

`champions-sp-calc` is a headless, API-first computation engine that ingests raw match event logs, applies configurable weighting heuristics, and emits normalized championship scoring probabilities. It eliminates the black-box problem in sports analytics by offering full traceability from input to output. Think of it as the deterministic layer between your data lake and your dashboard.

## Core Features (Hardcore)

### 1. Multi‑Model Scoring Aggregator with Real‑Time Weight Tuning
Dynamically merge outputs from Poisson, Elo, and Glicko models using a live weight matrix. Adjust model influence on the fly via a REST endpoint or config file, with zero downtime. The aggregator recalculates championship probabilities in sub‑50ms even across 10,000+ simulated seasons.

### 2. Deterministic Audit Trail with Immutable Log Hashing
Every calculation step—input transformation, weight application, probability normalization—is hashed into an immutable Merkle‑style log. This allows you to replay any past computation, verify integrity, and prove compliance for betting regulation audits or internal QA.

### 3. Locale‑Aware Probability Normalization Engine
Automatically adjusts probability distributions based on regional league biases (e.g., home‑field advantage in Bundesliga vs. La Liga). Uses a pre‑trained Bayesian prior matrix that updates weekly. No manual threshold tuning—just pass the league code and let the engine adapt.

## How to Use (Professional Steps)

### Step 1: Configure Multi‑Tenant Data Ingestion Pipeline
Create a `config.yaml` with your tenant ID, league codes, and model weight vector. Use the `sp-calc ingest --config config.yaml` command. The tool will validate your data schema, auto‑detect missing fields, and generate a normalized event DataFrame. For production, mount this as a Kubernetes ConfigMap and trigger ingestion via a cron job.

### Step 2: Run a Championship Probability Simulation with Custom Constraints
Execute `sp-calc simulate --season 2024 --iterations 10000 --constraints "home_advantage:0.12,draw_bias:0.05"`. The engine will spawn parallel worker threads (one per CPU core) and output a JSON array of probability vectors. Each vector includes a `confidence_interval` field (95% CI) calculated via bootstrapping. Pipe the output to `jq` for real‑time filtering.

### Step 3: Export Audit Log and Generate Compliance Report
Run `sp-calc audit --from 2024-01-01 --to 2024-12-31 --format pdf`. This will fetch all hashed computation logs from the immutable store, reconstruct the computation graph, and generate a PDF report with a SHA‑256 checksum. Use the `--verify` flag to cross‑check the log against the current state of the engine. This is mandatory for any regulated sportsbook integration.

## Best Practices (Industry‑Proven)

### Practice 1: Separate Simulation and Aggregation into Independent Microservices
Deploy `champions-sp-calc` as two distinct services: `simulator` (stateless, horizontally scalable) and `aggregator` (stateful, backed by Redis). This pattern prevents memory contention during high‑throughput simulations and allows you to scale the aggregator independently for dashboard queries. Use gRPC for inter‑service communication—latency drops by 40% compared to REST.

### Practice 2: Implement a Two‑Stage Cache for Probability Lookups
Cache raw model outputs in a local LRU cache (TTL: 5 minutes) and aggregated probabilities in a distributed Redis cluster (TTL: 1 hour). This reduces API response time from 200ms to under 15ms for repeated queries. For stale data, use a background worker that pre‑warms the cache 30 minutes before peak traffic (e.g., match day). Monitor cache hit ratio via Prometheus metrics exposed on `/metrics`.
