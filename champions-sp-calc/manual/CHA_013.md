## What is champions-sp-calc?

**champions-sp-calc** is a **deterministic, cache‑first scoring engine** designed for real‑time championship standings, playoff tie‑breakers, and head‑to‑head performance breakdowns.  
It decouples calculation logic from data ingestion, making it a **drop‑in microservice** for any sports‑analytics or e‑sports SaaS stack.

Unlike generic calculators, this tool enforces **strict idempotency** across parallel API calls and supports **custom scoring matrices** (e.g., weighted Elo, FIFA‑style points, or proprietary league rules).  
The engine is written in **TypeScript** with zero runtime dependencies beyond Node.js 18+.

## Core Features

### 1. Multi‑Stage Idempotent Pipeline (MIP)
Each scoring request passes through a **validate → transform → compute → persist** pipeline.  
The pipeline uses **etag‑based deduplication** and **atomic Redis transactions** to ensure that concurrent writes for the same match ID never produce duplicate or conflicting results.  
This is critical for live‑score ingestion from multiple sources (e.g., betting APIs, manual input, IoT sensors).

### 2. Dynamic Scoring Matrix DSL
Define your own scoring logic using a **JSON‑based Domain‑Specific Language** (DSL).  
Example:  
```json
{
  "rules": [
    { "condition": "homeScore > awayScore", "points": 3 },
    { "condition": "homeScore === awayScore", "points": 1 },
    { "condition": "homeScore < awayScore", "points": 0 }
  ],
  "tiebreaker": ["goalDifference", "headToHead", "awayGoals"]
}
```
The DSL supports **nested conditions, arithmetic overrides, and custom aggregators** (e.g., `sum`, `weightedAvg`, `percentile`).  
No need to redeploy the service when league rules change — just update the matrix via API.

### 3. Predictive Cache Invalidation (PCI)
Instead of a TTL‑based cache, champions-sp-calc uses **dependency tracking** to invalidate only affected standings.  
When a match result changes, the engine recalculates **only the teams whose rank or points could shift** — not the entire table.  
This reduces recomputation by **up to 85%** in leagues with 100+ teams, making it suitable for real‑time dashboards and live‑stream overlays.

## How to Use

### Step 1: Initialize the Scoring Engine
```bash
npm install champions-sp-calc
```
Then import and configure with your Redis and PostgreSQL connection strings:
```ts
import { createScoringEngine } from 'champions-sp-calc';

const engine = createScoringEngine({
  cache: { type: 'redis', url: process.env.REDIS_URL },
  store: { type: 'postgres', connectionString: process.env.DATABASE_URL },
  matrix: require('./scoring-matrix.json'),
});
```
The engine automatically sets up connection pooling and retry logic.  
No additional middleware is required.

### Step 2: Ingest a Match Result
```ts
await engine.ingest({
  matchId: 'match_001',
  homeTeam: 'FC Alpha',
  awayTeam: 'SC Beta',
  homeScore: 3,
  awayScore: 1,
  timestamp: '2025-04-10T20:00:00Z',
});
```
The engine validates the payload against a **Zod schema**, computes the delta, and updates the standings atomically.  
If the match already exists, it returns the cached result without re‑executing the pipeline (etag‑based).

### Step 3: Query Standings with Filters
```ts
const standings = await engine.getStandings({
  league: 'premier',
  season: '2025',
  topN: 5,
  sortBy: 'points',
  includeTiebreaker: true,
});
```
Returns a **sorted, paginated** array of team objects, each containing:
- `teamId`, `teamName`
- `points`, `goalDifference`, `headToHeadResults`
- `rankChange` (compared to previous round)

For real‑time updates, subscribe to the `standings.changed` event:
```ts
engine.on('standings.changed', (payload) => {
  console.log('Updated standings:', payload);
});
```

## Best Practices

### 1. Use a Dedicated Redis Instance for PCI
The predictive cache invalidation algorithm relies on **sorted sets** and **hash maps** in Redis.  
Running this on a shared Redis instance (e.g., with session storage) can cause key collisions and degrade performance.  
**Recommendation:** Provision a separate Redis 7+ instance with `maxmemory-policy allkeys-lru` and at least 2 GB RAM for leagues with 200+ teams.

### 2. Version Your Scoring Matrix
When you update the `scoring-matrix.json` file, **increment the `matrixVersion` field** in the matrix:
```json
{
  "matrixVersion": "v2",
  "rules": [...]
}
```
The engine automatically **recalculates all affected standings** when the version changes, but **only for matches ingested after the version bump**.  
This prevents historical data from being retroactively altered — a common source of bugs in live sports systems.  
Always test the new matrix against a snapshot of production data before deploying.
