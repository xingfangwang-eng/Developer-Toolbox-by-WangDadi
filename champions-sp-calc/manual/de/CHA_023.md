# champions-sp-calc - CHA_023

## Übersicht

`champions-sp-calc` ist eine **High‑Performance CLI & Web‑API** zur Berechnung von Skill‑Point‑Optima in kompetitiven Multiplayer‑Spielen. Das Tool erlaubt es Entwicklern, Data Scientists und Game‑Designern, SP‑Verteilungen auf Basis von Echtzeit‑Telemetrie, Balancing‑Constraints und Spieler‑Skill‑Profilen zu modellieren. Optimiert für **Low‑Latency** und **Batch‑Processing** unter Node.js 18+.

**Keywords:** `champions-sp-calc tool`, `champions-sp-calc solution`, `champions-sp-calc software`

---

## Hardcore‑Funktionen

### 1. **Constraint‑Aware Skill‑Graph Solver**  
Statt einfacher Summenberechnung nutzt `champions-sp-calc` einen **weighted DAG‑Solver** (Directed Acyclic Graph). Jeder Skill‑Point wird als Knoten mit mehrdimensionalen Kosten (Cooldown, Mana, Synergie‑Multiplier) modelliert. Der Solver findet die **Pareto‑optimale SP‑Verteilung** unter bis zu 64 Constraints – in unter 50 ms bei 200 Knoten.

### 2. **Real‑Time Meta‑Snapshots via WebSocket**  
Integriert eine **Pub/Sub‑Engine**, die SP‑Metriken aus Live‑Matches streamt. Über einen einfachen WebSocket‑Client können Teams **Balancing‑Shifts** (z. B. „Skill X wird in Patch 2.3 gebufft“) sofort in ihre lokalen Kalkulationen einfließen lassen. Kein manuelles Daten‑Pollen – das Tool aktualisiert den Graphen automatisch.

### 3. **Declarative Config‑as‑Code (YAML / JSON Schema)**  
Alle SP‑Modelle, Constraints und Output‑Formatter werden über eine **versionierte Konfigurationsdatei** definiert. Das Tool generiert daraus automatisch eine TypeScript‑Typdefinition und eine **Prisma‑ähnliche Query‑API**. Ideal für CI/CD‑Pipelines: Änderungen am Balancing werden als Pull‑Request getestet, bevor sie in den Game‑Client fließen.

## Detaillierte Anwendung

### Schritt 1: Modell‑Setup mit YAML  
Erstelle eine Datei `sp_model.yaml`:

```yaml
skills:
  - id: "fireball"
    base_cost: 3
    cooldown: 2.5
    synergy_with: ["ignite", "meteor"]
constraints:
  max_total_sp: 30
  min_sp_per_skill: 1
  max_sp_per_skill: 5
```

Führe dann den Validator aus:  
```bash
champions-sp-calc validate --config sp_model.yaml
```  
Das Tool prüft auf Zyklen im Graph, überlappende Constraints und ungültige Synergien. Bei Fehlern erhältst du eine **AST‑basierte Fehlermeldung** mit Zeilennummer.

### Schritt 2: Optimierung mit Telemetrie‑Daten  
Lade eine CSV mit Spieler‑Performance‑Daten (z. B. `player_stats.csv`):

```bash
champions-sp-calc optimize --config sp_model.yaml --input player_stats.csv --output optimized_sp.json
```  

Das Tool führt eine **iterative Gradient‑Descent‑Optimierung** durch, wobei jeder Iterationsschritt die Verteilung gegen die gemessene Win‑Rate kalibriert. Der Output enthält:  
- **Optimales SP‑Array** (z. B. `[3, 4, 2, ...]`)  
- **Konformitäts‑Score** (0–100)  
- **Sensitivitäts‑Heatmap** (welche SP‑Punkte den größten Einfluss haben)

### Schritt 3: Live‑Streaming via WebSocket  
Starte den WebSocket‑Server:

```bash
champions-sp-calc stream --port 8080 --config sp_model.yaml
```  

Verbinde dich mit einem Client:

```javascript
const ws = new WebSocket('ws://localhost:8080');
ws.onmessage = (event) => {
  const snapshot = JSON.parse(event.data);
  console.log(`Neue SP‑Verteilung: ${snapshot.sp_allocation}`);
};
```  

Der Server sendet alle 5 Sekunden ein **Delta‑Update** – nur die geänderten SP‑Werte. So bleibt die Bandbreite minimal, selbst bei 10.000 gleichzeitigen Clients.

## Branchen‑Best Practices

### 1. **Constraint‑Hierarchie für Live‑Balancing**  
Definiere Constraints in **drei Kategorien**: *hard* (z. B. maximale SP), *soft* (z. B. bevorzugte Synergie‑Kombinationen) und *dynamic* (z. B. temporäre Buffs). Nutze die `priority`‑Property in der YAML‑Config, um dem Solver mitzuteilen, welche Constraints er zuerst verletzen darf, wenn kein Pareto‑Optimum existiert. So vermeidest du „unlösbare“ Modelle in Produktion.

### 2. **Versionierte SP‑Modelle im Git‑Workflow**  
Checke jede `sp_model.yaml` in ein separates Git‑Repository ein. Verwende **Git Hooks**, um vor jedem Commit automatisch `champions-sp-calc validate` auszuführen. Kombiniere das mit einem **GitHub Action**, das bei Pull‑Requests die Optimierung mit historischen Telemetrie‑Daten wiederholt – so siehst du sofort, ob eine Balancing‑Änderung die SP‑Verteilung negativ beeinflusst.
