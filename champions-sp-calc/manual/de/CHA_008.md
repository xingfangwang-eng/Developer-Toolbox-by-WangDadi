# champions-sp-calc - CHA_008

## Überblick

`champions-sp-calc` ist ein spezialisiertes CLI- und API-Tool zur **deterministischen Berechnung von Skill-Points (SP) in kompetitiven Champion-Systemen**. Es richtet sich an Entwickler von Simulationen, Balancing-Ingenieure und E-Sport-Plattformen, die eine nachvollziehbare, regelbasierte SP-Modellierung benötigen. Das Tool eliminiert manuelle Fehlerquellen und bietet eine **konfigurationsgetriebene, erweiterbare Engine** für Multi-Champion-Ökosysteme.

**Keywords:** champions-sp-calc, SP-Berechnung, Champion-Balancing, Skill-Point-Engine, Competitive Gaming Tool, SP-Modellierung, E-Sport-Analyse, Gaming-SaaS

## Detaillierter Inhalt

### Was ist `champions-sp-calc`?

`champions-sp-calc` ist eine **Headless SaaS-Komponente**, die SP-Werte auf Basis von Champion-Rollen, Level-Skalierung, Item-Synergien und Umgebungsmodifikatoren (z. B. Map-Buffs) berechnet. Anders als generische Rechner verwendet es eine **YAML-basierte Regeldatei**, in der Balancing-Parameter versioniert und über CI/CD-Pipelines deployt werden können. Es eignet sich sowohl für **statische Simulationen** als auch für **Echtzeit-APIs** in Spiel-Clients.

### Kernfunktionen (Hardcore-Features)

1. **Multi-Kontext-Rule-Engine**  
   Unterstützt **gewichtete SP-Formeln** pro Champion-Klasse (z. B. `Assassin`, `Tank`, `Support`) mit dynamischen Modifikatoren für **Stacking-Limit, Cooldown-Reduktion und kritische Trefferwahrscheinlichkeit**. Die Engine validiert automatisch Zyklen in Abhängigkeiten (z. B. `Item A → Item B → Champion C`) und bricht bei Inkonsistenzen mit detaillierten Trace-Logs ab.

2. **Time-Series-SP-Profiling**  
   Ermöglicht die **zeitbasierte SP-Entwicklung über Match-Phasen** (Early, Mid, Late Game). Der `--profile`-Modus generiert ein JSON-Array mit SP-Werten pro Minute, inklusive **Breakpoints** (z. B. Level 6, 11, 16). Dies ist essenziell für **Power-Spike-Analysen** und **Balancing-Patches**.

3. **Regression-Testing-Modus**  
   Integriert einen **deterministischen Test-Runner**, der SP-Ergebnisse gegen historische Patch-Daten abgleicht. Bei Abweichungen > 0,5% wird ein **Diff-Report** mit Ursachenanalyse (z. B. „Item X wurde um 10% nerfed“) ausgegeben. Ideal für **CI/CD-Pipelines** in Balancing-Teams.

### So verwenden Sie es (Detaillierte Schritte)

1. **Konfiguration der Regeldatei (`sp-rules.yaml`)**  
   Definieren Sie Champion-Klassen und deren SP-Formeln. Beispiel für einen `Burst-Mage`:
   ```yaml
   champion_class: burst_mage
   base_sp: 100
   level_scaling: 1.15
   item_synergies:
     - item: Rabadons
       multiplier: 1.4
   ```
   Führen Sie eine **Syntax-Validierung** durch:
   ```bash
   champions-sp-calc validate sp-rules.yaml
   ```
   Bei Fehlern erhalten Sie eine **Zeilen-genaue Fehlermeldung** inklusive empfohlener Korrektur.

2. **SP-Berechnung für ein Szenario**  
   Nutzen Sie den `calc`-Befehl mit Champion, Level und Items:
   ```bash
   champions-sp-calc calc --champion Ahri --level 11 --items Rabadons,Zhonyas --profile
   ```
   Ausgabe:  
   - **Statischer SP-Wert**: `1.245`  
   - **Zeitprofil**: `[120, 340, 890, 1245]` (Minute 0–20)  
   - **Breakpoints**: `Level 6 → 560 SP`, `Level 11 → 1245 SP`

3. **Regressionstest gegen einen Patch**  
   Verknüpfen Sie Ihre aktuelle Konfiguration mit einem historischen Patch:
   ```bash
   champions-sp-calc regress --base patch_13.10.yaml --current sp-rules.yaml --threshold 0.5
   ```
   Bei Abweichungen erhalten Sie einen **Patch-Diff**:
   ```
   DIFF: Ahri (Level 11): 1245 vs 1230 (1.2% Abweichung)
   Ursache: Rabadons-Multiplier von 1.4 auf 1.35 geändert.
   ```

### Best Practices (Branchenstandards)

1. **Versionieren Sie Ihre SP-Regeln in Git**  
   Jede Änderung an `sp-rules.yaml` sollte mit einem **Semantic-Release-Tag** (z. B. `v2.1.0`) versehen werden. Nutzen Sie **Git Hooks**, um vor jedem Commit eine `validate`- und `regress`-Prüfung auszuführen. Dies verhindert, dass fehlerhafte Balancings in Produktion gehen.

2. **Integrieren Sie SP-Profile in Ihre Match-Simulation**  
   Statt statischer SP-Werte sollten Sie die **Time-Series-Profile** (`--profile`) in Ihre Engine einspeisen. Dadurch können Sie **Power-Spikes** realistisch abbilden (z. B. „Ahri hat bei Minute 12 einen 30% höheren Burst“). Nutzen Sie die Breakpoints, um **Item-Timings** (z. B. „Rabadons nach Minute 14“) zu optimieren.
