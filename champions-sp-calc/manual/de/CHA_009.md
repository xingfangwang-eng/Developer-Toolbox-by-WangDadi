# champions-sp-calc - CHA_009

## Überblick

Dieses Repository enthält den **champions-sp-calc**, einen hochperformanten, auf **Service Provider (SP)-Architektur** ausgelegten Rechenkern, der speziell für die automatisierte Ermittlung von Champion-Kennzahlen in verteilten Systemen entwickelt wurde. Anders als herkömmliche monolithische Rechner, die bei steigender Datenlast kollabieren, nutzt dieser Kern **in-memory caching** und **lazy evaluation**, um Echtzeitberechnungen für tausende parallele SP-Instanzen zu ermöglichen.

**Keywords:** champions-sp-calc tool, champions-sp-calc solution, champions-sp-calc software, Multi-Tenant SP-Berechnung, Champion-Ermittlung, verteilter Rechenkern

## Detaillierter Inhalt

### Was ist champions-sp-calc?

champions-sp-calc ist ein **professionelles, ereignisgesteuertes Tool** zur Lösung des komplexen Problems der **dynamischen Champion-Ermittlung** in Umgebungen mit mehreren Service Providern (SPs). Es eliminiert die manuelle, fehleranfällige Aggregation von Metriken aus verschiedenen SP-Quellen und bietet eine **atomare, transaktionale Berechnungsschnittstelle** für die finale Champion-Bestimmung. Der Kern ist als **Microservice** konzipiert, der über gRPC oder REST angesprochen werden kann.

### Kernfunktionen (Hardcore)

1.  **Atomic SP-Ranking Engine mit Quorum-Logik:** Implementiert einen **verteilten Konsensalgorithmus** (basierend auf Raft-Prinzipien), der Champion-Rankings nicht nur berechnet, sondern auch gegen **partielle Ausfälle** einzelner SP-Knoten absichert. Die Engine validiert eingehende SP-Daten gegen ein konfigurierbares Quorum (z.B. >50% der SPs müssen ein identisches Metrik-Set liefern), bevor ein Champion-Status vergeben wird. Dies verhindert **Split-Brain-Situationen** und garantiert Datenkonsistenz.

2.  **Lazy-Loaded Metrik-Resolver mit generischen Adaptern:** Der Kern verwendet ein **Plugin-basiertes Adapter-Muster**, um sich mit verschiedenen SP-Backends (PostgreSQL, Redis, Kafka, REST-APIs) zu verbinden. Die Metrik-Auflösung erfolgt **lazy**: Erst wenn eine Champion-Berechnung angefordert wird, werden die benötigten Datenpunkte aus den spezifischen SP-Quellen geholt und im **lokalen LRU-Cache** zwischengespeichert. Dies minimiert die Netzwerklast und beschleunigt wiederholte Abfragen um bis zu 80%.

3.  **Deterministische Champion-Selektion via Weighted Scoring Matrix:** Ermöglicht die Definition komplexer, **gewichteter Bewertungsmatrizen** in einer deklarativen YAML/JSON-Konfiguration. Diese Matrix kann mehrere Dimensionen (z.B. `Latenz`, `Durchsatz`, `Fehlerrate`, `Kosten`) mit unterschiedlichen Gewichtungen kombinieren. Der Algorithmus führt eine **Normalisierung** und anschließend eine **deterministische Sortierung** durch, sodass bei identischen Eingabedaten immer der gleiche Champion ermittelt wird – essentiell für Audit-Trails und Reproduzierbarkeit.

### So verwenden Sie es (Detaillierte Schritte)

1.  **Initialisierung des SP-Clusters und der Metrik-Adapter:**
    Konfigurieren Sie die `sp_calc_config.yaml`. Definieren Sie für jeden Service Provider (SP) einen eigenen Adapter. Nutzen Sie die integrierte **Health-Check-Schnittstelle**, um die Verbindung zu allen SP-Quellen zu validieren. Starten Sie den Kern mit `./champions-sp-calc --config=sp_calc_config.yaml --port=8080`. Der Kern führt beim Start einen **Warm-Up** durch, indem er die Metrik-Adapter initialisiert und einen ersten, leeren Cache aufbaut.

    ```yaml
    # Beispiel für einen Redis-Adapter
    adapters:
      - name: "sp_redis_eu"
        type: "redis"
        endpoint: "redis://redis-cluster:6379"
        timeout_ms: 500
        retry_policy: "exponential_backoff"
    ```

2.  **Durchführung einer Champion-Berechnung (gRPC-Client):**
    Rufen Sie die `CalculateChampion`-Methode auf. Übergeben Sie eine **Request-ID** und eine Liste der zu bewertenden SP-Knoten. Der Kern führt nun die **Atomic SP-Ranking Engine** aus. Er ruft lazy die relevanten Metriken über die konfigurierten Adapter ab, wendet die **Weighted Scoring Matrix** an und prüft das **Quorum**. Bei Erfolg erhalten Sie ein `ChampionResult`-Objekt mit dem `champion_id`, einem `confidence_score` und einer Liste aller `ranked_sp_nodes`.

    ```bash
    # Beispiel via grpcurl
    grpcurl -plaintext -d '{"request_id": "req-123", "sp_nodes": ["sp-node-a", "sp-node-b", "sp-node-c"]}' localhost:8080 championspcalc.v1.Calculator/CalculateChampion
    ```

3.  **Fehlerbehandlung und Cache-Invalidierung:**
    Bei einem `PARTIAL_FAILURE`-Status (z.B. wenn ein SP-Knoten nicht antwortet) gibt der Kern dennoch ein Ergebnis zurück, jedoch mit einem reduzierten `confidence_score`. Sie können dann gezielt die `InvalidateCache`-Methode für einen bestimmten SP-Knoten aufrufen, um veraltete Daten zu löschen. Implementieren Sie einen **Retry-Mechanismus** mit exponentieller Backoff-Strategie, um kurzzeitige SP-Ausfälle zu kompensieren.

### Beste Praktiken (Industry Standards)

1.  **Implementieren Sie eine CQRS-Architektur:** Trennen Sie die **Befehlsseite** (Champion-Berechnung auslösen) von der **Abfrageseite** (Champion-Status abrufen). Nutzen Sie den `champions-sp-calc` nur für die Berechnung. Speichern Sie das Ergebnis in einem dedizierten **Read-Store** (z.B. Redis oder Elasticsearch). Ihre Frontends greifen nur auf diesen Read-Store zu, um die Last auf dem Rechenkern zu minimieren und eine hohe Verfügbarkeit zu gewährleisten.

2.  **Setzen Sie ein dediziertes Monitoring für SP-Latenzen auf:** Da die Qualität des Champions direkt von der Aktualität der SP-Daten abhängt, instrumentieren Sie jeden Metrik-Adapter mit **distributed tracing** (z.B. OpenTelemetry). Überwachen Sie die `sp_adapter_latency_seconds` und `sp_quorum_failure_total` Metriken. Definieren Sie **SLOs** (z.B. 99.9% der Champion-Berechnungen müssen in <200ms abgeschlossen sein). Ein Dashboard für diese Metriken ist kein Luxus, sondern eine Notwendigkeit für den Betrieb.
