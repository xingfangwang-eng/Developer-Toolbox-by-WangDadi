## Detaillierter Inhalt

### Was ist champions-sp-calc?

champions-sp-calc ist ein spezialisiertes CLI-Tool und eine integrierbare Library, die entwickelt wurde, um die komplexe, oft nicht-lineare Verteilung von Skill-Points über mehrere Champion-Kits (Passiv, Q, W, E, R) hinweg zu modellieren. Es löst das Problem der manuellen, fehleranfälligen SP-Planung, insbesondere bei Champions mit skalierenden Fähigkeiten oder dynamischen Cooldowns. Die Engine verwendet einen **multi-constraint linear programming**-Ansatz, um SP-Allokationen unter Berücksichtigung von Level-Caps, Item-Synergien und Match-Phasen zu optimieren.

### Kernfunktionen (Hardcore-Features)

1.  **Dynamische SP-Kosten-Matrix mit Patch-Versionierung**  
    champions-sp-calc integriert eine versionierte Datenbank (`sp_cost_matrix_v2.1.json`), die nicht nur die Basis-SP-Kosten pro Fähigkeitsstufe, sondern auch **prozentuale Kostenmodifikatoren** durch spezifische Champion-Passiven oder Rune-Konfigurationen berücksichtigt. Der Parser kann direkt aus Riot-API-ähnlichen JSON-Strukturen (z. B. `DataDragon`) importieren und erstellt eine optimierte Cost-Map, die für den Solver zugänglich ist.

2.  **Rekursiver Branch-and-Bound Solver für SP-Rotationen**  
    Statt einfacher Level-Skalierung implementiert das Tool einen **rekursiven Branch-and-Bound-Algorithmus**, der alle möglichen SP-Rotationen (Reihenfolge des Skillens) bis Level 18 durchläuft. Dabei werden nicht nur die finalen Punkte, sondern auch die **Zwischenzustände** (z. B. Level-3-All-In-Potential) bewertet. Der Solver gibt die optimale Sequenz zurück, die eine benutzerdefinierte Scoring-Funktion (z. B. maximale Burst-DPS oder minimale Manakosten) maximiert.

3.  **CI/CD-Integration mit automatischen SP-Benchmark-Tests**  
    Das Repository enthält eine vorkonfigurierte `Makefile`-Pipeline und GitHub Actions-Workflows, die bei jedem Commit automatisch SP-Berechnungen für eine definierte Liste von Champion-Builds ausführen. Die Ergebnisse werden gegen **historische Benchmarks** (gespeichert in `/benchmarks/`) validiert. Abweichungen >0.5% im finalen SP-Output führen zu einem fehlschlagenden Test, was die deterministische Korrektheit des Solvers in CI-Umgebungen sicherstellt.

### So verwenden Sie es (Detaillierte Schritte)

1.  **Initialisierung des SP-Konfigurationsprofils**  
    Führen Sie `champions-sp-calc init --champion "Ahri" --patch "14.10"` aus. Dies generiert eine `sp_profile.yaml` im Arbeitsverzeichnis. Öffnen Sie diese Datei und definieren Sie Ihre **Scoring-Gewichtung** (z. B. `scoring: { wave_clear: 0.3, single_target_burst: 0.7 }`). Für fortgeschrittene Nutzer: Fügen Sie einen `custom_constraint`-Block hinzu, um z. B. zu erzwingen, dass Fähigkeit `Q` vor Level 6 mindestens 3 Punkte haben muss.

2.  **Ausführung des SP-Optimizers mit Logging**  
    Starten Sie den Solver mit `champions-sp-calc solve --profile sp_profile.yaml --output results.json --verbose`. Das Tool gibt während der Ausführung detaillierte Logs auf `stderr` aus, die die aktuellen Branch-Bound-Entscheidungen zeigen (z. B. `[DBG] Pruning branch: Q5W3E2R2, upper bound 12.4 < current best 14.1`). Das finale `results.json` enthält die optimale SP-Sequenz, die finale Punktematrix und den erreichten Score.

3.  **Validierung und Integration in Ihr Build-Skript**  
    Nutzen Sie den `validate`-Befehl: `champions-sp-calc validate --input results.json --benchmark benchmarks/ahri_14.10.json`. Dieser Befehl vergleicht die berechnete SP-Verteilung mit dem mitgelieferten Benchmark. Für die Integration in ein externes Tool (z. B. einen Discord-Bot oder eine Web-App) können Sie die Library direkt importieren: `const { ChampionSPCalculator } = require('champions-sp-calc'); const calc = new ChampionSPCalculator(); calc.loadPatch('14.10'); const result = calc.optimize('Ahri', { wave_clear: 0.3 });`.

### Beste Praktiken (Industriestandard)

1.  **Versionierung der SP-Kosten-Matrix im Projekt-Repository**  
    Lagern Sie die `sp_cost_matrix_v*.json` Dateien nicht nur extern, sondern committen Sie eine geprüfte Version direkt in Ihr Repository (z. B. unter `/data/`). Verknüpfen Sie diese mit einem Git-Tag, der dem Spiel-Patch entspricht. Dies ermöglicht reproduzierbare Builds und erleichtert das Debugging, wenn sich SP-Kosten zwischen Patches ändern. Ein `git bisect` über die SP-Matrix kann schnell Regressionen in der Build-Optimierung aufdecken.

2.  **Nutzung von deterministischen Seeds für Regressionstests**  
    Obwohl der Solver deterministisch ist, können externe Faktoren (z. B. Item-Synergien, die sich aus einer externen API speisen) zu Varianz führen. Definieren Sie für Ihre CI-Pipeline einen festen `seed`-Parameter (z. B. `--seed 42`), der den Zufallsgenerator für die Scoring-Funktion initialisiert. Dies stellt sicher, dass Ihre SP-Benchmark-Tests über verschiedene Läufe und Entwickler-Maschinen hinweg absolut identische Ergebnisse liefern, was für die Aufrechterhaltung der Teststabilität in einem Team unerlässlich ist.
