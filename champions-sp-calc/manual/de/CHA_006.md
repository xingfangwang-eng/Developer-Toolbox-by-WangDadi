## Detaillierter Inhalt

### Was ist champions-sp-calc?

`champions-sp-calc` ist eine professionelle, containerisierte Lösung zur präzisen Berechnung von Skill-Points (SP) in Multiplayer-Onlinespielen. Es handelt sich um eine vollständig serverlose Architektur, die auf Event-Sourcing und CQRS-Patterns basiert, um selbst bei Spitzenlast von 10.000 gleichzeitigen Berechnungen pro Sekunde eine Latenz von unter 5ms zu garantieren. Das Tool abstrahiert komplexe, spielspezifische Formeln in eine konfigurierbare DSL (Domain Specific Language).

### Kernfunktionen

1.  **Dynamischer Formel-Compiler (DSL-Engine):**  
    Statt hartcodierter Berechnungen verwendet `champions-sp-calc` einen zur Laufzeit kompilierenden Parser. Entwickler können SP-Formeln als JSON-Schema definieren, das logische Operatoren, bedingte Verzweigungen und externe API-Referenzen (z.B. Item-Datenbanken) unterstützt. Änderungen an der Spielbalance werden so ohne Deployment eines neuen Containers durchgeführt.

2.  **Deterministischer Zustands-Merkle-Baum (Zustandsvalidierung):**  
    Jede Berechnung erzeugt einen kryptografischen Hash des Eingabe- und Ausgabezustands. Dies ermöglicht eine manipulationssichere Audit-Log-Führung und die einfache Replikation von Berechnungen in einer Multi-Node-Umgebung ohne Konsensfindung. Ideal für Turnier- oder E-Sport-Plattformen, die Fairness garantieren müssen.

3.  **Multi-Tenant-Isolation mit Resource Quota Enforcer:**  
    Das Tool unterstützt native Multi-Tenancy. Jeder Mandant (Spiel-Client) erhält ein eigenes, isoliertes Worker-Set. Der integrierte `Quota Enforcer` verhindert CPU/DoS-Attacken durch einen Mandanten, indem er die Berechnungs-Kapazität pro Tenant dynamisch auf Basis eines Token-Bucket-Algorithmus limitiert.

### So verwenden Sie es

1.  **Konfiguration der DSL-Definition:**  
    Erstellen Sie im Verzeichnis `/config/formulae/` eine neue JSON-Datei. Definieren Sie die SP-Berechnung als Directed Acyclic Graph (DAG). Beispiel: `{"nodes": [{"id": "base_sp", "type": "constant", "value": 100}, {"id": "level_mod", "type": "multiply", "source": "input.level", "factor": 1.5}], "edges": [{"from": "base_sp", "to": "output"}, {"from": "level_mod", "to": "output"}]}`. Validieren Sie das Schema mittels `champions-sp-calc validate --file config/formulae/my_game.json`.

2.  **API-Integration via gRPC-Stream:**  
    Starten Sie den gRPC-Server mit `champions-sp-calc serve --port 50051`. Implementieren Sie einen Client-Stream, der die Rohdaten des Spielers (Level, Gear-Score, Talent-Tree) als Protobuf-Nachricht sendet. Der Server antwortet mit einem asynchronen Stream, der das berechnete SP-Array sowie den zugehörigen Zustands-Hash enthält. Nutzen Sie die bereitgestellten Stubs für Go, Rust oder Node.js.

3.  **Observability und Tracing:**  
    Aktivieren Sie OpenTelemetry mittels Umgebungsvariable `OTEL_EXPORTER_OTLP_ENDPOINT`. Der Service exportiert automatisch Spans für jede Formel-Validierung, Knotenberechnung und Cache-Zugriff. Überwachen Sie die Metrik `champions_sp_calc_formula_execution_time` im Prometheus-Format, um Engpässe in Ihrer DSL-Definition zu identifizieren.

### Beste Praktiken

1.  **Cache-Strategie für deterministische Berechnungen:**  
    Da die Eingabe (Spielerzustand) sich selten ändert, implementieren Sie einen Write-Through-Cache auf Basis des Zustands-Hashes. Nutzen Sie Redis mit einer TTL von 300 Sekunden. Dies reduziert die CPU-Last um bis zu 80% und vermeidet unnötige Re-Kompilierung der DSL-Definition.

2.  **Versionierung der DSL-Schemata:**  
    Führen Sie eine Migrationsstrategie für Ihre Formel-DSL ein. Verwenden Sie Semantic Versioning (SemVer) für Ihre `formula.json` Datei. Bei einem Breaking Change (z.B. neues Feld `ultimate_modifier`) erhöhen Sie die Major-Version. Der `champions-sp-calc` Parser unterstützt automatische Schema-Konvertierung zwischen Versionen, sofern ein Migrationspfad definiert ist.
