## Detaillierter Inhalt

### Was ist champions-sp-calc?

`champions-sp-calc` ist ein professionelles SaaS-Tool zur präzisen, dynamischen Berechnung von Champion-Spezialwerten (SP) in Echtzeit. Es ermöglicht Entwicklern, Balancing-Parameter, Synergie-Effekte und Skalierungsfaktoren über eine modulare API zu definieren und zu berechnen. Das Tool ist besonders geeignet für Game-Design-Teams, die komplexe Kampfsysteme mit multiplen Buffs, Debuffs und prozentualen Boni abbilden müssen. Die Architektur basiert auf einer ereignisgesteuerten Pipeline, die asynchrone Berechnungen mit Caching und Versionierung unterstützt.

### Kernfunktionen (Hardcore-Features)

1.  **Dynamische Formel-Engine mit AST-Transformation**
    Die Engine parst benutzerdefinierte SP-Formeln (z. B. `(base_damage * (1 + (crit_chance * crit_damage))) * armor_penalty`) in einen abstrakten Syntaxbaum (AST). Dieser AST wird vor der Ausführung optimiert (Konstantenfaltung, Dead-Code-Eliminierung) und dann in eine native JavaScript-Funktion kompiliert. Dadurch wird eine Ausführungsgeschwindigkeit von < 0.5 ms pro Berechnung bei 100.000 gleichzeitigen Requests erreicht.

2.  **Multi-Tenant Caching mit invalidation-on-write**
    Jeder Tenant (z. B. ein Spieleprojekt) erhält einen eigenen Redis-Cluster-Bereich. Berechnete SP-Werte werden mit einem zweistufigen Cache (L1: In-Memory, L2: Redis) zwischengespeichert. Bei einer Änderung der zugrunde liegenden Championdaten (Patch-Update) wird ein gestaffelter Invalidation-Mechanismus ausgelöst: Zuerst wird der L1-Cache des betroffenen Tenants invalidated, dann asynchron der Redis-Cache. Dies verhindert Stale-Reads bei gleichzeitiger Minimierung der Latenz.

3.  **Versionierte SP-Historie mit Rollback-Fähigkeit**
    Jede SP-Berechnung wird automatisch versioniert (SemVer-konform). Über die API kann der Entwickler eine bestimmte Version eines Champion-SP-Profils abrufen oder ein Rollback auf eine vorherige Version durchführen. Die History wird in einer PostgreSQL-Tabelle mit `jsonb`-Spalten gespeichert und ist über einen GraphQL-Endpunkt abfragbar. Dies ermöglicht ein detailliertes Audit-Trail und die Reproduktion von Balancing-Entscheidungen.

### So verwenden Sie es (Detaillierte Schritte)

1.  **API-Key generieren und Tenant konfigurieren**
    Rufen Sie den Admin-Endpunkt `POST /api/v1/tenants` auf, um einen neuen Tenant zu erstellen. Senden Sie dabei ein JSON-Objekt mit dem Namen des Projekts und der gewünschten Datenbank-ID. Sie erhalten einen `API-Key` und eine `Tenant-ID` zurück. Speichern Sie diese Umgebungsvariablen in Ihrer `.env`-Datei:
    ```bash
    CHAMPIONS_SP_CALC_API_KEY=your_api_key_here
    CHAMPIONS_SP_CALC_TENANT_ID=your_tenant_id_here
    ```

2.  **Champion-Daten definieren und SP-Formel hinterlegen**
    Verwenden Sie den Endpunkt `POST /api/v1/champions`, um einen neuen Champion anzulegen. Übergeben Sie die Basiswerte (z. B. `base_health`, `base_attack`, `crit_chance`) und die SP-Formel als String. Beispiel-Request:
    ```json
    {
      "name": "Aatrox",
      "base_stats": {
        "base_health": 580,
        "base_attack": 60,
        "crit_chance": 0.25
      },
      "sp_formula": "(base_attack * 1.5) + (base_health * 0.02) + (crit_chance * 50)"
    }
    ```
    Der Server validiert die Formel syntaktisch und speichert sie als AST.

3.  **SP-Wert abrufen und Ergebnis interpretieren**
    Um den aktuellen SP-Wert zu berechnen, rufen Sie `GET /api/v1/champions/{champion_id}/sp` auf. Der Response enthält den berechneten SP-Wert sowie die Version der Berechnung:
    ```json
    {
      "sp_value": 142.5,
      "version": "1.0.1",
      "computation_time_ms": 0.34
    }
    ```
    Sie können optional Query-Parameter wie `?patch=13.10` hinzufügen, um eine bestimmte Patch-Version zu laden.

### Beste Praktiken (Industry Best Practices)

1.  **Verwenden Sie statische Formeln für konstante Berechnungen**
    Wenn Ihre SP-Formel keine dynamischen Variablen enthält (z. B. reine Konstantenmultiplikation), definieren Sie sie als `static: true` im Champion-Objekt. Die Engine wird die Formel dann nur einmal beim Start des Services auswerten und das Ergebnis als Konstante im Cache ablegen. Dies reduziert die CPU-Last bei Massenberechnungen um bis zu 40 %.

2.  **Implementieren Sie einen lokalen Proxy für Batch-Requests**
    Statt einzelner API-Calls für jeden Champion sollten Sie Batch-Endpunkte wie `POST /api/v1/champions/batch-sp` verwenden. Senden Sie ein Array von Champion-IDs und optionalen Parametern. Die Engine verarbeitet diese Anfragen parallel (Worker-Threads) und gibt die Ergebnisse als geordnetes Array zurück. Dies senkt die Netzwerklatenz drastisch und ist für Echtzeit-Balancing-Tools unerlässlich.
