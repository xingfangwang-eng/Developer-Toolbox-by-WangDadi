## Detaillierter Inhalt

### Was ist champions-sp-calc?

champions-sp-calc ist ein professionelles, plattformunabhängiges Kommandozeilen-Tool (CLI) und API-Modul, das die manuelle, fehleranfällige Berechnung von Skill-Points in Multiplayer-Online-Battle-Arena-Spielen (MOBAs) und RPGs automatisiert. Es eliminiert Guesswork und liefert deterministische SP-Verteilungen basierend auf benutzerdefinierten Constraints, Meta-Daten und Echtzeit-Spielstatistiken. Das Tool ist in Rust geschrieben und bietet sowohl eine native Binärdatei als auch ein Python-Binding via PyO3.

### Kernfunktionen

1.  **Constraint-basierte SP-Optimierung mit SAT-Solver-Integration:** Statt einfacher Addierer verwendet `champions-sp-calc` einen integrierten SAT-Solver (MiniSat). Du definierst komplexe, logische Abhängigkeiten (z.B. "Skill A erfordert 5 Punkte in Baum X UND Skill B muss Level 3 sein, BEVOR Skill C freigeschaltet wird"). Das Tool findet die optimale SP-Verteilung, die alle Regeln erfüllt, und gibt eine "Proof of Optimality" aus.

2.  **Dynamisches Meta-Import-Modul (JSON/YAML):** Das Tool kann externe Meta-Daten in Echtzeit importieren. Lade eine `champion_meta.json` aus einer API (z.B. von U.GG oder OP.GG) und das Tool berechnet automatisch die effektivste SP-Verteilung für die aktuelle Patch-Version, inklusive Win-Rate-Gewichtung und Synergie-Bonifaktoren zwischen Skills.

3.  **Deterministischer Build-Vergleichsmodus (Diff Engine):** Führe zwei verschiedene SP-Builds (z.B. "Build_A.json" vs. "Build_B.json") parallel aus. Die Diff-Engine zeigt auf Millisekundenebene die exakten Unterschiede in Skill-DPS, Mana-Effizienz und Cooldown-Reduktion an. Das Ergebnis ist eine farbcodierte Tabelle im Terminal (via `colored`-Crate), die den "Worst-Case"- und "Best-Case"-SP-Nutzungsgrad darstellt.

### So verwenden Sie es

1.  **Initialisierung und Konfiguration (CLI):** Führe `champions-sp-calc init --game league --champion akali` aus. Das Tool generiert ein leeres `sp_config.toml` im aktuellen Verzeichnis. Öffne die Datei und definiere deine Constraints: `[constraints] max_sp = 18`, `[skills] q = 5`, `[dependencies] "q > 3" => "e > 1"`. Speichere die Datei.

2.  **Berechnung mit externem Meta-Feed:** Lade die aktuellen Meta-Daten: `curl -s https://api.example.com/meta/akali.json | champions-sp-calc solve --meta -`. Das Tool liest den Pipe-Input (`-`), parst das JSON und führt die SAT-Solver-Optimierung durch. Die Ausgabe (`stdout`) ist ein JSON-Objekt: `{"build": {"q":5, "w":1, "e":3, "r":2}, "sp_used":11, "sp_remaining":7, "proof": "SAT"}`.

3.  **Detaillierte Diff-Analyse:** Erstelle zwei Build-Dateien (`build_v1.json`, `build_v2.json`). Führe `champions-sp-calc diff build_v1.json build_v2.json --output table` aus. Das Tool gibt eine tabellarische Übersicht aus, die exakt zeigt, wie sich die SP-Verteilung auf die finalen Skill-Werte auswirkt. Nutze `--output csv` für die Weiterverarbeitung in Analyse-Scripts.

### Beste Praktiken

1.  **Constraint-Management mit Versionierung:** Lege alle `sp_config.toml`-Dateien in einem dedizierten Git-Repository ab. Nutze Branching, um verschiedene Build-Strategien (z.B. "Burst-Mage", "Bruiser") zu versionieren. Führe `champions-sp-calc validate` in der CI/CD-Pipeline aus, um sicherzustellen, dass nach einem Patch keine Constraints verletzt werden.

2.  **Meta-Daten-Caching für CI/CD:** Implementiere ein Caching-System für die externen Meta-Feeds. Da die Meta-Daten sich nur mit jedem Patch ändern, solltest du sie nicht bei jedem Build aus der API laden. Nutze `champions-sp-calc cache --ttl 3600` und speichere die JSON-Dateien lokal. In deiner CI/CD-Pipeline (z.B. GitHub Actions) kannst du den Cache-Key auf den Patch-Version-Hash setzen, um unnötige API-Calls zu vermeiden.
