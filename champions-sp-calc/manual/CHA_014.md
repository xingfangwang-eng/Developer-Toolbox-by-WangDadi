## What is champions-sp-calc?

**champions-sp-calc** is a lightweight, serverless microservice that calculates champion-specific skill power coefficients across 10+ languages. It’s built for modern CI/CD pipelines, edge deployments, and real-time game analytics. No bloated dependencies. No vendor lock-in. Just raw, deterministic math with i18n-aware output.

## 3 Hardcore Features

### 1. Polyglot Coefficient Engine
The core engine parses champion skill data in 10 languages (EN, DE, ES, JA, ZH, KO, PT, FR, RU, AR) using a unified JSON schema. It applies locale-specific normalization (e.g., decimal separators, unit scaling) without breaking precision. This means you can feed in a `de_DE` skill set and get `en_US` SP values back—fully localized, no manual mapping.

### 2. Real-Time Delta Diffing
Every calculation is logged as a delta diff between the base champion state and the modified skill tree. The tool outputs a structured diff (JSON Patch RFC 6902) so you can roll back, audit, or replay any SP change. This is critical for A/B testing balance patches or simulating “what-if” scenarios in production.

### 3. Zero-Dependency CLI + WebAssembly (WASM) Dual Mode
Run the same logic server-side via a 2MB static binary, or embed it client-side as a WASM module. No Node.js, no Python runtime, no Docker overhead. The WASM build is ~180KB gzipped and executes in under 50µs per calculation—perfect for edge functions (Cloudflare Workers, Lambda@Edge).

## How to Use

### Step 1: Initialize the Champion Context
```bash
./champions-sp-calc --init --champion "Aatrox" --locale "en_US" --skill-set "Q:1,W:2,E:3,R:4"
```
This loads the base champion profile from the embedded polyglot database and validates the skill tree against the locale-specific schema. If the locale is missing, the engine falls back to `en_US` and logs a warning to stderr.

### Step 2: Apply a Skill Power Modifier
```json
{
  "operation": "multiply",
  "field": "Q.base_damage",
  "value": 1.15,
  "locale": "de_DE"
}
```
Pipe this JSON into the tool via stdin:
```bash
echo '{"operation":"multiply","field":"Q.base_damage","value":1.15,"locale":"de_DE"}' | ./champions-sp-calc --apply
```
The engine returns a delta diff and the updated SP coefficient. The `--apply` flag ensures the change is ephemeral unless you commit with `--commit`.

### Step 3: Export the Final SP Table
```bash
./champions-sp-calc --export --format "csv" --output "sp_table_en.csv"
```
This generates a CSV with columns: `champion`, `skill`, `base_sp`, `modified_sp`, `delta`, `locale`. For JSON output, use `--format json`. The export respects the current locale context and includes all uncommitted changes.

## 2 Industry Best Practices

### Practice 1: Always Pin the Locale During Batch Processing
When running batch calculations (e.g., 10,000 champion variations), explicitly set `--locale` at the start of the pipeline. If the locale is omitted, the engine auto-detects from `LANG` env var, which can cause silent mismatches in multi-region deployments. Pin it once, and the engine will cache the locale-specific coefficient table for the entire session—reducing lookup overhead by ~40%.

### Practice 2: Use Delta Diffs for Version-Controlled Balance Patches
Instead of manually tweaking SP values in a spreadsheet, generate delta diffs for every patch iteration. Store these diffs in a Git repo alongside the champion base profiles. This gives you a full audit trail, rollback capability, and the ability to programmatically compare two patches (e.g., `git diff patch_v1.json patch_v2.json`). It’s the same pattern used by game studios for live balance tuning.
