## What is champions-sp-calc?

`champions-sp-calc` is a **high-performance, multi-tenant SaaS engine** purpose-built for real-time sports performance scoring and predictive analytics. Unlike off-the-shelf calculators, it decouples scoring logic from data ingestion, enabling teams to plug in custom metrics, weightings, and normalization curves without touching the core pipeline. Designed for **low-latency, high-throughput** environments—think live game feeds, fantasy league integrations, and athlete benchmarking dashboards.

## Core Features (Hardcore)

### 1. Declarative Scoring DSL with Hot-Reload
Define your scoring formula as a **YAML/JSON DSL**, not code. The engine compiles it into an AST, runs it through a vectorized execution layer (SIMD-optimized for x86_64 and ARM64), and supports **hot-reload** without downtime. Perfect for SaaS teams that need to iterate on scoring rules mid-season without a full deploy.

### 2. Multi-Tenant Data Sharding with Per-Tenant Caching
Each tenant gets an **isolated shard** (PostgreSQL + Redis cluster) with automatic cache invalidation based on scoring delta. The engine supports **tenant-level rate limiting** and **stale-while-revalidate** patterns, ensuring that a spike in one tenant’s traffic doesn’t degrade others. Built-in Prometheus metrics for shard health and cache hit ratio.

### 3. Real-Time WebSocket Stream with Backpressure-Aware Aggregation
`champions-sp-calc` exposes a **WebSocket endpoint** that streams live score updates. Under the hood, it uses a **backpressure-aware aggregation pipeline** (based on RxJava / Kotlin Flow), batching updates every 100ms or 100 events (whichever comes first). This prevents client overload and reduces bandwidth by 60% compared to naive per-event push.

## How to Use (Detailed Steps)

### Step 1: Define Your Scoring Schema
Create a `scoring-config.yaml` file in the `/config` directory of your tenant.  
Example snippet:

```yaml
tenant: "team_alpha"
metrics:
  - name: "speed_score"
    weight: 0.4
    normalizer: "z-score"
  - name: "accuracy_score"
    weight: 0.6
    normalizer: "min-max"
aggregation: "weighted_sum"
cache_ttl: 300
```

Run `champions-sp-calc validate --config ./scoring-config.yaml` to syntax-check and dry-run the scoring logic against sample data. The CLI will output a **JSON schema validation report** and a **performance estimate** (e.g., “Estimated throughput: 12,000 ops/sec per shard”).

### Step 2: Ingest Data via REST or WebSocket
For batch ingestion, POST to `/api/v1/ingest/batch`:

```bash
curl -X POST https://api.champions-sp-calc.com/v1/ingest/batch \
  -H "Authorization: Bearer <tenant-token>" \
  -H "Content-Type: application/json" \
  -d '{"events": [{"player_id": "p1", "speed": 8.2, "accuracy": 0.95}, ...]}'
```

For real-time ingestion, connect to the WebSocket endpoint:

```javascript
const ws = new WebSocket("wss://api.champions-sp-calc.com/v1/stream?tenant=team_alpha");
ws.onmessage = (event) => {
  const scoreUpdate = JSON.parse(event.data);
  console.log(`Live score: ${scoreUpdate.score}`);
};
```

The server will respond with a **batch ID** or **stream ID** for tracking and debugging.

### Step 3: Query Aggregated Scores
After ingestion, query the scoring endpoint with optional filters:

```bash
curl -X GET "https://api.champions-sp-calc.com/v1/scores?tenant=team_alpha&player_id=p1&time_range=last_24h" \
  -H "Authorization: Bearer <tenant-token>"
```

Response:

```json
{
  "tenant": "team_alpha",
  "player_id": "p1",
  "score": 87.3,
  "components": {
    "speed_score": 34.2,
    "accuracy_score": 53.1
  },
  "cached": true,
  "timestamp": "2025-04-06T14:30:00Z"
}
```

Use the `X-Cache-Status` header to verify whether the result came from Redis (HIT) or was computed fresh (MISS).

## Best Practices

### 1. Use Stale-While-Revalidate for High-Volume Dashboards
If your dashboard refreshes every 5 seconds, configure `cache_ttl: 60` with `stale_while_revalidate: true`. This means the cache serves the previous score immediately while the engine computes the new one in the background. This reduces P99 latency from 200ms to under 5ms for read-heavy workloads.

### 2. Pre-Warm the Scoring Engine for Known Peak Events
For scheduled events (e.g., a championship game), pre-load the scoring DSL and tenant shards using the `warmup` CLI command:

```bash
champions-sp-calc warmup --tenant team_alpha --event "finals_2025"
```

This forces the engine to compile the DSL, populate the Redis cache with baseline data, and spin up additional worker threads. It can reduce cold-start latency by 80%.
