### What is champions-sp-calc?

champions-sp-calc is a production-grade, API-first microservice that ingests raw champion stat matrices, applies weighted power scaling algorithms, and outputs precise specific power (SP) scores per champion per patch. It eliminates manual spreadsheet hell and provides deterministic SP calculations with sub-millisecond latency, even under heavy concurrent load.

### Core Features

1. **Deterministic SP Engine with Patch-Aware Versioning**  
   The core engine uses a locked, versioned algorithm (semver-based) that accounts for base stats, scaling ratios, and itemization modifiers. Each patch triggers an automatic recalculation cascade, ensuring your SP values never drift from live game data.

2. **Multi-Tenant Rate-Limited REST & gRPC Gateway**  
   Exposes both REST and gRPC endpoints with per-tenant rate limiting, request deduplication, and response caching. Supports batch SP queries (up to 1000 champions per request) with configurable precision levels (float32, float64, or fixed-point).

3. **Real-Time WebSocket Subscription for SP Delta Streams**  
   Subscribe to live SP delta streams when champion stats change mid-patch (e.g., hotfixes). Each delta includes a diff payload, confidence score, and rollback flag. Ideal for building real-time dashboards or competitive analysis tools.

### How to Use

1. **Initialize the SP Calculator Client**  
   Instantiate the client with your tenant API key and preferred transport (REST or gRPC). Use the `NewSPCalculator` factory function, passing in a `Config` struct that defines default precision, cache TTL, and patch fallback behavior. Example:  
   ```go
   calc := spcalc.NewSPCalculator(spcalc.Config{
       TenantID:   "esports-prod-01",
       Precision:  spcalc.Float64,
       CacheTTL:   5 * time.Minute,
       PatchFallback: spcalc.FallbackToPrevious,
   })
   ```

2. **Submit a Batch SP Calculation Request**  
   Build a `ChampionStatMatrix` slice with champion IDs, base stats, and optional item overrides. Call `calc.BatchCalculate(ctx, matrix)` — the engine will return a `map[string]SPResult` with SP scores, confidence intervals, and patch version used. For large batches, set `WithParallelism(8)` to enable concurrent evaluation.

3. **Subscribe to Real-Time SP Delta Streams**  
   Open a WebSocket connection via `calc.Subscribe(ctx, "champion-sp-deltas")`. Parse incoming `SPDelta` messages: each contains `ChampionID`, `PreviousSP`, `CurrentSP`, `DeltaPercent`, and `PatchID`. Use `DeltaPercent` to trigger alerts or update cached SP values in your frontend.

### Best Practices

- **Always pin your SP algorithm version in production.**  
  Never rely on the `latest` tag for the SP engine. Use explicit semver constraints (e.g., `^1.2.0`) in your `go.mod` or Docker image tag. This prevents silent SP drift when the algorithm is updated for a new patch cycle. Validate SP outputs against known reference values from the official game data API before deploying.

- **Implement client-side SP caching with TTL fallback.**  
  In high-throughput environments, cache SP results locally with a TTL of 2–5 minutes. Use the WebSocket delta stream to invalidate specific champion entries early. This reduces load on the SP engine and keeps your UI responsive during peak usage (e.g., tournament draft phases).
