Unlike static odds calculators, champions-sp-calc uses a dynamic weight engine that ingests live match data, historical performance vectors, and meta-shift signals. The engine applies Bayesian inference with adjustable decay factors, allowing developers to tune recency bias per tournament tier. Built for polyglot data pipelines.

### 3. GraphQL Subscription Endpoint for Live Odds Streaming
Exposes a real-time GraphQL subscription API that streams delta-updated probability shifts during live matches. Each subscription is scoped to a match ID or tournament phase. The server uses WebSocket multiplexing with backpressure handling, ensuring zero data loss even during peak traffic spikes. Ideal for building live dashboards or trading bots.

## How to Use (Highly Detailed, Production-Grade Steps)

### Step 1: Initialize the Engine with a Custom Meta Profile
```bash
champions-sp-calc init --region=KR --tier=Challenger --decay=0.85 --cache-ttl=30
```
This creates a localized engine instance calibrated for Korean Challenger meta with 85% recency decay and a 30-second cache TTL. The engine preloads champion vectors from the embedded dataset and validates against the latest patch fingerprint. Use `--profile=export` to dump the config as a JSON schema for CI/CD pipelines.

### Step 2: Subscribe to a Live Match Probability Stream
```graphql
subscription {
  matchProbability(matchId: "KR_2025_03_21_1234", intervalMs: 200) {
    timestamp
    champion
    winProb
    banProb
    pickProb
    metaShift
  }
}
```
Deploy this subscription in your frontend or bot backend. The `intervalMs` parameter controls push frequency—200ms is optimal for real-time dashboards. The `metaShift` field indicates a sudden probability anomaly (e.g., a surprise pick or patch bug), triggering your alert system.

### Step 3: Export a Snapshot for Offline Model Training
```bash
champions-sp-calc export --format=parquet --output=./training_data/ --match-range=last_500
```
This dumps raw probability snapshots as Apache Parquet files, structured for ML ingestion. Each row includes champion, opponent, map side, game phase, and computed probability vector. Use this to retrain your own prediction models or backtest strategies. The export respects the current locale and weight settings.

## Best Practices (Industry Standards for Champions-sp-calc)

### 1. Decouple the Weight Engine from the UI Layer
Run the PWE as a separate microservice behind a Redis pub/sub bus. This allows you to scale the probability computation horizontally without affecting frontend latency. Use the GraphQL subscription endpoint as the only interface between your dashboard and the engine. This pattern is battle-tested in high-frequency trading and esports analytics.

### 2. Always Validate Champion Names Against the Normalization Layer
Before injecting any champion data into the engine, pass it through the multi-locale normalization module. A single mismatched champion name (e.g., “LeBlanc” vs. “Leblanc”) can corrupt the entire probability vector for a match. Implement a pre-flight check in your CI/CD pipeline using the `--validate` flag. This prevents silent data drift across regional patches.
