# champions-sp-calc - CHA_030

## Overview

champions-sp-calc is a precision-engineered SaaS tool for calculating **Standardized Performance (SP)** metrics in competitive gaming and esports ecosystems. It eliminates manual data wrangling and subjective bias, delivering deterministic SP scores that align with tournament-grade validation frameworks. Designed for developers, analysts, and platform integrators who demand reproducible, audit-friendly results.

## Hardcore Features

### 1. Multi-Game SP Normalization Engine
champions-sp-calc applies a **game-agnostic normalization layer** that maps raw performance data (KDA, CS/min, objective control, vision score) onto a unified SP scale across 10+ titles (LoL, Dota 2, Valorant, CS2, Overwatch 2, Apex, etc.). The engine uses per-game statistical distribution curves to eliminate meta-specific inflation or deflation. You can pass a single JSON payload and get a normalized SP score comparable across entirely different games.

### 2. Real-Time Tilt & Variance Detection (SP-Var)
Beyond static SP calculation, champions-sp-calc runs a **rolling variance analysis** on a player’s last N matches. If the SP variance exceeds a configurable threshold (default: 0.35), the tool flags the player as “tilted” or “high-variance” — a critical signal for roster decisions, bet placement, or automated content generation. This is not a simple standard deviation; it’s a weighted moving variance with decay factor that prioritizes recent matches.

### 3. Deterministic SP Audit Trail with JSON-LD Export
Every SP calculation generates a **cryptographic hash chain** (SHA-256) of all input parameters and intermediate transformations. The output includes a full audit trail in JSON-LD format, ready for embedding into tournament rulebooks or blockchain-based verification systems. No black box. No hidden weighting. You can replay the exact same calculation years later and get the exact same SP score.

## How to Use

### Step 1: Prepare Your Performance Payload
Structure your match data as a JSON array of player objects. Each object must include `game_id`, `player_id`, `kills`, `deaths`, `assists`, `gold_per_min`, `cs_per_min`, `vision_score`, and `game_duration`. Optional fields: `damage_per_min`, `kill_participation`, `objective_control_ratio`. champions-sp-calc validates the schema on ingestion — any missing or malformed field triggers a clear error with the exact path.

```json
[
  {
    "game_id": "match_0421",
    "player_id": "faker_01",
    "kills": 12,
    "deaths": 3,
    "assists": 7,
    "gold_per_min": 450,
    "cs_per_min": 9.2,
    "vision_score": 85,
    "game_duration": 2100
  }
]
```

### Step 2: Run the SP Calculation with Custom Normalization Weights
Call the API endpoint `/v1/sp/calculate` with your payload. You can override default normalization weights by passing a `weights` object in the query string or request body. Example: `?weights={"kills":0.3,"deaths":-0.2,"vision_score":0.15}`. The engine will re-normalize on the fly and return both the raw SP and the normalized SP. The response includes a `sp_components` array showing each metric’s contribution.

### Step 3: Interpret the SP-Var Flag and Export the Audit Trail
In the API response, check the `sp_var` field. If `sp_var.status === "tilted"`, the player’s recent performance is unstable — consider this a red flag for high-stakes decisions. For permanent record-keeping, call `/v1/sp/audit/{calculation_id}` to retrieve the full JSON-LD audit trail. This object includes `input_hash`, `transformation_log`, `normalization_curve_id`, and `output_hash`. Save this to your own database or IPFS for verifiable SP history.

## Best Practices

### 1. Always Use Game-Specific Normalization Curves
Do not rely on the default “global” normalization curve for production use. champions-sp-calc ships with 10+ pre-built curves (one per game), but you should calibrate your own curve using at least 5,000 matches from your target region/rank. Load your custom curve via `/v1/curves/upload` and reference it in your calculation request. This dramatically improves SP accuracy for regional or tier-specific analysis.

### 2. Implement SP-Var as a Pre-Processing Gate for Automated Decisions
If you’re building an automated roster suggestion tool, bet placement system, or content generation pipeline, use the `sp_var` flag as a pre-processing gate. Do not feed high-variance players into your decision engine without manual review. A simple rule: if `sp_var.status === "tilted"`, route the player to a human moderator queue. This reduces false positives in automated systems by roughly 40% based on our internal benchmarks.
