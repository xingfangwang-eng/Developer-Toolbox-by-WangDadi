# champions-sp-calc - CHA_026

## Overview

**champions-sp-calc** is a battle-tested SaaS toolkit engineered for high-performance **skill point optimization** in competitive gaming and simulation environments. Designed for developers, modders, and esports analysts, it eliminates manual guesswork by delivering mathematically precise allocation models based on real-time meta data and constraint solving.


---

## What is champions-sp-calc?

champions-sp-calc is a **deterministic constraint solver** wrapped in a lightweight API and CLI interface. It ingests champion-level skill trees, cooldown matrices, and synergy graphs, then outputs optimal SP (skill point) distributions under user-defined constraints (e.g., max DPS, min cooldown, survivability thresholds). Unlike generic calculators, it supports **multi-objective Pareto optimization** and **real-time patch data injection**.

---

## Core Features

### 1. Multi-Objective Pareto Solver Engine
Instead of returning a single "optimal" build, champions-sp-calc generates a **Pareto frontier** of viable SP allocations. Users can toggle between objectives—max burst, sustain DPS, utility uptime—and visualize trade-offs in a 2D/3D plot. The solver uses a hybrid of **integer linear programming (ILP)** and **genetic algorithms** with local search heuristics.

### 2. Live Meta Injection via JSON Patch Streams
The tool accepts **external patch data** as a stream of JSON diffs. This allows you to hot-reload champion reworks, cooldown nerfs, or new synergy bonuses without rebuilding the model. Built-in schema validation ensures patches don’t break the constraint graph. Ideal for CI/CD pipelines that scrape patch notes from official sources.

### 3. Graph-Based Synergy & Anti-Synergy Detection
Each skill node is represented as a **weighted directed graph** with edges for prerequisites, cooldown chains, and hidden interactions. The engine detects **anti-synergy cycles** (e.g., two skills that cancel each other’s buffs) and flags them with a confidence score. This is critical for champions with complex passive-to-active skill dependencies.

## How to Use

### 1. Initialize the Solver with a Champion Schema
Start by defining your champion’s skill tree as a JSON schema. Each node must include `id`, `cost`, `cooldown`, `prerequisites`, and an optional `synergy_group`. Run the CLI with the schema flag:

```bash
champions-sp-calc --schema ./champions/akali_v4.json --mode pareto
```

The engine will parse the graph, detect cycles, and generate a **constraint satisfaction problem (CSP)** in memory. If the schema is invalid (e.g., circular prerequisites), it returns a human-readable graph trace with offending edges highlighted.

### 2. Run Multi-Objective Optimization with Custom Bounds
Set your optimization targets using a YAML config file. For example, to maximize DPS while keeping cooldown under 8 seconds and survivability above 70%:

```yaml
objectives:
  - maximize: dps
  - minimize: cooldown
constraints:
  survivability: { min: 0.7 }
  total_sp: { max: 60 }
```

Execute:

```bash
champions-sp-calc --config ./builds/assassin_burst.yaml --output pareto_frontier.json
```

The output is a JSON array of Pareto-optimal allocations, each with a `sp_distribution` map and a `tradeoff_score` for ranking.

### 3. Visualize and Export the Pareto Frontier
Use the built-in export module to generate an interactive HTML plot or a CSV for further analysis in R/Python:

```bash
champions-sp-calc --visualize pareto_frontier.json --format html --out frontier_plot.html
```

Each point on the frontier is clickable, showing the exact SP distribution and predicted performance metrics. For CI workflows, pipe the CSV to your dashboard:

```bash
champions-sp-calc --export csv < pareto_frontier.json | tee frontier.csv
```

## Best Practices

### 1. Always Validate Patch Data with the Built-In Schema Checker
Before injecting a live patch stream, run `champions-sp-calc validate --patch patch_v2.1.json` to catch structural errors (e.g., missing node IDs, non-integer costs). This prevents runtime constraint violations that can silently corrupt the solver state. In production, chain this as a pre-commit hook in your data pipeline.

### 2. Use Multi-Objective Mode for Meta-Agnostic Builds
Avoid single-objective optimization unless you’re targeting a specific matchup. The Pareto frontier approach gives you a **portfolio of viable builds** that can adapt to shifting metas. Store the frontier in a versioned database (e.g., SQLite) and serve it via a REST endpoint for real-time client queries. This pattern is used by top-tier esports analytics teams.
