# champions-sp-calc - CHA_023

## Overview

champions-sp-calc is a high-performance, multi-tenant skill point (SP) calculator built for theorycrafters, speedrunners, and competitive build optimizers. It handles complex SP allocation logic across multiple champion trees, factoring in diminishing returns, synergy caps, and patch-specific nerfs. This tool eliminates guesswork by delivering deterministic SP distributions based on real game data.

## Core Features

### 1. Multi-Tree Synergy Engine with Diminishing Returns Modeling
The tool computes cross-tree SP efficiency using a custom weighted graph algorithm. It accounts for hidden cap mechanics (e.g., soft caps on critical damage scaling) and outputs the optimal SP distribution per level bracket. Supports 12+ champion archetypes with auto-detection of tree dependencies.

### 2. Patch-Precise Simulation Mode
Each SP calculation is tied to the latest game patch data. The engine parses patch notes in real-time and adjusts node costs, cap values, and synergy thresholds. You can lock a specific patch version (e.g., 4.21c) for historical comparisons or regression testing.

### 3. CLI & API-First Architecture with Batch Processing
Designed for CI/CD pipelines and automation. Accepts JSON or YAML inputs for bulk champion builds, outputs structured results (JSON / CSV) with delta analysis between builds. Works offline with cached patch snapshots. Ideal for integration with Discord bots, spreadsheets, or web dashboards.

## How to Use

### 1. Define Your Champion Build Profile in YAML
Create a file `build_profile.yaml` with your current SP allocation and target champion ID. Example:

```yaml
champion_id: "soraka_023"
patch_lock: "4.21c"
current_sp:
  tree_a: 12
  tree_b: 8
  tree_c: 3
constraints:
  max_sp_per_tree: 18
  synergy_threshold: 0.75
```

Run the CLI: `champions-sp-calc --profile build_profile.yaml`

The engine returns a ranked list of optimal reallocations, including the expected SP gain per node shift and the associated risk (nerf probability) for each move.

### 2. Simulate a Full Rotation with Patch Rollback
Use the `--simulate` flag to test hypothetical scenarios. Example:

```bash
champions-sp-calc --simulate --patch 4.20b --target_tree tree_b --sp_shift +3
```

This outputs a delta report showing how shifting 3 SP from tree_a to tree_b changes your effective damage output, cooldown reduction, and synergy score. The report includes a confidence interval based on the variance in the patch data.

### 3. Batch Process Multiple Builds for Regression Testing
Prepare a JSON array of build profiles and run in batch mode:

```json
[
  {"champion_id": "ahri_017", "current_sp": {"tree_a": 10, "tree_b": 6}},
  {"champion_id": "nasus_022", "current_sp": {"tree_a": 14, "tree_c": 4}}
]
```

Execute: `champions-sp-calc --batch builds.json --output results.csv`

The tool processes all builds in parallel (up to 16 threads by default) and outputs a CSV with columns: `champion_id`, `current_efficiency`, `optimized_efficiency`, `delta`, `recommended_actions`.

## Best Practices

### 1. Always Lock a Patch Version Before Making Build Decisions
SP mechanics change between patches, sometimes silently. Locking a patch version prevents your calculations from being invalidated by an unannounced hotfix. Use `--patch_lock` in all production pipelines. For tournament prep, lock the patch used in the official client at least 48 hours prior.

### 2. Use Diminishing Returns Thresholds to Avoid Over-Allocation
Every tree has a soft cap beyond which each additional SP yields less than 0.5% improvement. The engine flags these nodes automatically. Set `synergy_threshold` to 0.8 or higher in your profile to force the optimizer to redistribute SP to underutilized trees. This is especially critical for hybrid builds where cross-tree synergy can be more valuable than raw SP density.
