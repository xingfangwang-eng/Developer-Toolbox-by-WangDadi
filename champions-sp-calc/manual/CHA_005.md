# champions-sp-calc - CHA_005

## Overview

This is a **battle-tested** SaaS toolkit engineered for precision SP (Service Provider) calculations in cloud-native architectures. Built by developers for developers, it eliminates guesswork in resource allocation, cost forecasting, and multi-tenant scaling. No fluff, just raw compute.

## What is champions-sp-calc?

`champions-sp-calc` is a **deterministic SP cost & capacity calculator** designed for high-velocity SaaS teams. It ingests real-time usage metrics, applies weighted SLA penalties, and outputs granular cost-per-tenant breakdowns. Think of it as your **CFO-approved Excel killer** for service provider billing models.

## Core Features

### 1. Multi-Dimensional Cost Attribution Engine
Automatically decomposes shared infrastructure costs (e.g., RDS, EKS node pools, CDN egress) into tenant-level line items. Supports **weighted tags**, **usage-based multipliers**, and **custom amortization schedules**. No more manual spreadsheet hell.

### 2. Real-Time SLA Penalty Simulator
Simulates "what-if" scenarios for uptime degradation (99.9% → 99.5%) and instantly recalculates rebate liabilities. Integrates with **PagerDuty** and **Datadog** to pull live incident data. Your SRE team will actually thank you.

### 3. Burst-Aware Capacity Planner
Forecasts resource spikes using **Holt-Winters** seasonality models. Outputs actionable recommendations for **spot instance** vs. **reserved instance** mix, with a built-in **cost-optimal scaling policy generator**. Perfect for Kubernetes autoscaler tuning.

## How to Use

### Step 1: Bootstrap the CLI & Inject Tenant Metadata
```bash
pip install champions-sp-calc
champions-sp-calc init --provider aws --region us-east-1
champions-sp-calc import --file tenants.csv --schema "tenant_id,plan_tier,monthly_active_users,avg_cpu_usage"
```
This initializes a local SQLite store and ingests your tenant hierarchy. The `--schema` flag enforces type coercion (e.g., `avg_cpu_usage` as float64).

### Step 2: Run a Weighted Cost Allocation Scan
```bash
champions-sp-calc allocate --cost-driver "rds:instance:db.r5.xlarge" --weight-by "monthly_active_users" --output json
```
The engine queries your cloud billing API (via IAM role), maps costs to tenants, and exports a JSON blob with `tenant_id`, `allocated_cost`, and `confidence_interval`. Use `--output csv` for Excel integration.

### Step 3: Generate a Burst Forecast & Scaling Policy
```bash
champions-sp-calc forecast --horizon 90d --seasonality weekly --confidence 0.95
champions-sp-calc policy --min-instances 3 --max-instances 20 --spot-pct 0.6
```
The forecast model outputs a **probability distribution** of peak demand. The policy generator spits out a **Kubernetes HPA manifest** (YAML) with custom metrics. Deploy it directly with `kubectl apply -f policy.yaml`.

## Best Practices

### 1. Always Pin Your Cost Drivers to Business Metrics
Don’t allocate by raw CPU hours—that’s lazy. Instead, use **monthly active users (MAU)** or **API call volume** as weight keys. This aligns infrastructure spend with revenue-generating activity. Our team saw a **22% reduction in chargeback disputes** after switching to MAU-based allocation.

### 2. Run the SLA Simulator Before Every Major Deployment
Before you push a new microservice or change autoscaling thresholds, simulate the worst-case uptime drop. Use the `--sla-target 99.9` flag to see if your new architecture violates contractual obligations. It’s saved our team from at least three P0 incidents this quarter alone.
