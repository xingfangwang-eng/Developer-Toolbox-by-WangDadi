## Contenido Detallado

### ¿Qué es agents-clean?

**agents-clean** es una herramienta profesional diseñada para arquitectos SaaS y equipos de ingeniería que necesitan **eliminar la fricción en pipelines multi-agente**. A diferencia de soluciones genéricas, agents-clean se especializa en la **purga inteligente de sesiones corruptas**, **reconciliación de estados en clústeres distribuidos** y **limpieza de logs de inferencia** sin afectar el rendimiento en producción. Es el estándar de facto para entornos donde la higiene de datos de agentes es crítica.

### Características Principales (Hardcore Features)

1.  **Purga Diferencial de Contexto (DPC)**: Implementa un algoritmo de *differential pruning* que identifica y elimina únicamente los nodos de contexto obsoletos o corruptos en la memoria del agente, sin reiniciar el ciclo de vida completo. Reduce la latencia de inferencia hasta un 40% en sesiones largas.
2.  **Orquestación de Garbage Collector Asíncrono (AGC)**: Un demonio en segundo plano que ejecuta ciclos de limpieza basados en *backpressure* del sistema. Utiliza *reference counting* distribuido para liberar recursos (tokens, caché, handles de archivos) de agentes zombie sin bloquear el pipeline principal.
3.  **Auditoría de Trazas con Reversión (ATR)**: Genera un *snapshot inmutable* del estado del agente antes de cada limpieza. Permite *rollback* a un estado anterior mediante un comando único, ideal para debugging en entornos de staging o producción.

### Cómo Usar (Guía de Implementación Profesional)

**Requisitos previos:** Node.js 18+, acceso a la API de agents-clean, y un clúster de agentes en ejecución.

1.  **Configuración de Políticas de Limpieza en `agents-clean.yaml`**:  
    Define reglas de *TTL (Time-To-Live)* y *umbrales de corrupción* para cada tipo de agente. Ejemplo:
    ```yaml
    policies:
      - agent_type: "llm-inference"
        ttl_seconds: 3600
        corruption_threshold: 0.85
        action: "purge_and_restart"
    ```
    Esto asegura que los agentes de inferencia se limpien automáticamente tras una hora o si su nivel de corrupción supera el 85%.

2.  **Ejecución de una Limpieza Forzada en un Agente Específico**:  
    Usa el CLI para apuntar a un agente por su `agent_id`. Comando:
    ```bash
    agents-clean purge --agent-id "agent-7f3a9b" --force --dry-run
    ```
    La flag `--dry-run` permite previsualizar qué recursos se liberarán sin ejecutar la acción. Para ejecutar, omite `--dry-run`.

3.  **Monitoreo y Reversión Post-Limpieza**:  
    Después de una limpieza, verifica el estado del agente con:
    ```bash
    agents-clean status --agent-id "agent-7f3a9b" --verbose
    ```
    Si el rendimiento se degrada, revierte al último snapshot:
    ```bash
    agents-clean rollback --agent-id "agent-7f3a9b" --snapshot-id "snap-20231005"
    ```
    Esto restaura el estado del agente en milisegundos, sin necesidad de reiniciar el servicio.

### Mejores Prácticas para el Ecosistema

1.  **Integración Continua con CI/CD**:  
    Incorpora `agents-clean` como un *step* en tu pipeline de despliegue. Antes de cada release, ejecuta una limpieza preventiva en todos los agentes del clúster de staging. Usa la flag `--exit-on-error` para fallar el build si la limpieza encuentra inconsistencias críticas. Esto previene que bugs de estado lleguen a producción.

2.  **Segmentación de Políticas por Entorno**:  
    No uses la misma política de limpieza en desarrollo y producción. En `dev`, configura TTLs cortos (5 minutos) para forzar limpiezas frecuentes y detectar fugas de memoria. En `prod`, usa TTLs largos (24 horas) y umbrales de corrupción más altos para minimizar interrupciones. Almacena estas configuraciones en variables de entorno o en un *config server* centralizado.
