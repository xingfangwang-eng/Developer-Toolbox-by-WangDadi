# agents-clean - AGE_016

## Resumen Técnico

`agents-clean` es una herramienta de limpieza y normalización de datos para entornos multi-agente, diseñada para arquitecturas SaaS que requieren orquestación de agentes con alta fidelidad de contexto. Este proyecto resuelve problemas de contaminación de memoria, colisiones de estado y fragmentación de historial en sistemas de agentes colaborativos. Ideal para equipos que gestionan pipelines de agentes en producción con requisitos de compliance y trazabilidad.

## Funcionalidades Clave

### 1. **Purga Contextual por Sesión de Agente**
Limpieza granular de datos de sesión sin afectar el estado global del sistema. `agents-clean` permite eliminar registros específicos de interacciones de agentes (por ID de sesión, timestamp o tipo de evento) manteniendo la integridad de los datos maestros. Ideal para entornos donde múltiples agentes comparten un mismo espacio de trabajo y se requiere aislar datos sensibles o irrelevantes.

### 2. **Reconciliación de Estados Asíncronos**
Resuelve conflictos de estado entre agentes que operan en paralelo. La herramienta detecta automáticamente divergencias en datos compartidos (como colas de tareas, tablas de rutas o cachés distribuidas) y aplica estrategias configurables de resolución: *last-write-wins*, *merge semántico* o *rollback por consenso*. Perfecto para sistemas con agentes concurrentes que actualizan el mismo recurso.

### 3. **Eliminación de Datos Huérfanos con Políticas de Retención**
Barrido automático de registros huérfanos generados por agentes caídos o tareas fallidas. `agents-clean` implementa políticas de retención configurables (por tiempo, por cantidad de eventos o por prioridad del agente) y elimina datos obsoletos sin interrumpir el flujo de trabajo. Incluye logs de auditoría para cumplir con normativas como GDPR o SOC 2.

## Cómo Usar

### 1. **Configurar un Job de Limpieza Programado**
Define un cron job que ejecute `agents-clean` con parámetros de sesión y políticas de retención. Ejemplo en un entorno Linux con systemd:

```bash
# Crear un archivo de servicio systemd para ejecución diaria
sudo cat > /etc/systemd/system/agents-clean.service <<EOF
[Unit]
Description=Agents Clean - Limpieza de datos de agentes
After=network.target

[Service]
Type=oneshot
ExecStart=/usr/local/bin/agents-clean --session-id "all" --retention-policy "7d" --log-level debug
User=agente
Group=agente

[Install]
WantedBy=multi-user.target
EOF

# Habilitar e iniciar el servicio
sudo systemctl daemon-reload
sudo systemctl enable agents-clean.service
sudo systemctl start agents-clean.service
```

### 2. **Ejecutar una Limpieza Manual con Filtros Avanzados**
Usa la CLI para eliminar datos específicos de una sesión de agente fallida. Ejemplo con filtro por ID de agente y rango de tiempo:

```bash
agents-clean --agent-id "agent-42" --start-time "2023-10-01T00:00:00Z" --end-time "2023-10-05T23:59:59Z" --dry-run
```

El flag `--dry-run` muestra los registros que se eliminarían sin ejecutar la acción. Para confirmar, quita el flag y agrega `--force`.

### 3. **Integrar con un Pipeline CI/CD**
Añade `agents-clean` como paso de post-deploy en tu pipeline de GitHub Actions. Ejemplo de workflow:

```yaml
name: Post-deploy cleanup
on:
  deployment_status:
    types: [success]

jobs:
  cleanup:
    runs-on: ubuntu-latest
    steps:
      - name: Run agents-clean
        run: |
          curl -sL https://agents-clean.example.com/install.sh | bash
          agents-clean --session-id "deploy-${{ github.sha }}" --retention-policy "30d" --audit-log
```

## Mejores Prácticas

### 1. **Define Políticas de Retención por Tipo de Agente**
No todas las sesiones de agentes son iguales. Para agentes de alto riesgo (por ejemplo, aquellos que manejan datos financieros), configura retenciones cortas (24h) y logs de auditoría. Para agentes de monitoreo, usa retenciones largas (90d) sin auditoría. Esto minimiza el overhead de almacenamiento y cumple con regulaciones sectoriales.

### 2. **Usa Dry-Run en Entornos de Producción**
Antes de ejecutar una limpieza masiva en producción, siempre usa `--dry-run` para revisar los registros afectados. Combínalo con `--log-level debug` para obtener un reporte detallado de las acciones que se ejecutarán. Esto evita borrados accidentales de datos críticos, especialmente en sistemas multi-tenant.
