### ¿Qué es agents-clean?

`agents-clean` es una herramienta profesional de orquestación y limpieza de datos para flujos de trabajo multi-agente. Resuelve problemas comunes de contaminación de contexto, duplicación de estados y fragmentación de memoria en sistemas basados en agentes autónomos. Ideal para equipos que trabajan con arquitecturas RAG, fine-tuning dinámico o pipelines de inferencia en tiempo real.

### Características Principales (Hardcore)

1.  **Purga Diferencial de Contexto con Hash de Sesión**  
    Implementa un algoritmo de deduplicación basado en *rolling hash* sobre el historial de interacciones del agente. Elimina automáticamente fragmentos redundantes sin perder coherencia semántica. Reduce el consumo de tokens en un 40% en cargas de trabajo con ventanas de contexto largas (32K+ tokens).

2.  **Saneamiento Asíncrono con Políticas de Retención Configurables**  
    Permite definir reglas de expiración por tipo de dato (prompts, respuestas, metadatos de herramientas) usando cron jobs internos. Soporta *soft delete* con recuperación por tiempo limitado y *hard purge* para cumplir con GDPR/CCPA. Ejecuta limpiezas en segundo plano sin bloquear el pipeline principal.

3.  **Validación de Integridad Estructural con Schemas Dinámicos**  
    Verifica automáticamente que cada mensaje en el historial del agente cumpla con un schema JSON definido por el usuario. Si un mensaje está corrupto (por ejemplo, falta un campo `role` o `content` es `null`), lo repara o lo descarta según la política configurada. Incluye un modo *strict* para entornos de compliance.

### Cómo Usar (Pasos Detallados)

1.  **Configurar el Pipeline de Limpieza**  
    Define tu política de saneamiento en un archivo `clean-config.yaml`.  
    ```yaml
    pipeline:
      dedup:
        enabled: true
        hash_window: 512  # tokens
      retention:
        max_age_hours: 72
        soft_delete: true
      schema_validation:
        strict: false
        repair: true
    ```
    Luego, ejecuta `agents-clean init --config clean-config.yaml` para cargar la configuración. El sistema validará automáticamente la sintaxis y te mostrará un resumen de las reglas activas.

2.  **Ejecutar una Limpieza Manual en un Pipeline Activo**  
    Conecta `agents-clean` a tu pipeline de agente usando el middleware integrado.  
    ```python
    from agents_clean import CleanMiddleware
    
    # Instancia el middleware con tu configuración
    cleaner = CleanMiddleware.from_config("clean-config.yaml")
    
    # Ejecuta limpieza sobre el historial actual
    cleaned_history = cleaner.sanitize(agent.history)
    
    # Reemplaza el historial del agente
    agent.history = cleaned_history
    ```
    Esto purga duplicados, repara mensajes inválidos y aplica políticas de retención en menos de 50ms para historiales de hasta 10K mensajes.

3.  **Automatizar con Webhooks y Triggers**  
    Configura un webhook para ejecutar limpieza cada vez que se complete una sesión de agente.  
    ```bash
    agents-clean webhook --port 8080 --endpoint /clean-session
    ```
    Luego, desde tu aplicación, envía un POST con el ID de sesión:  
    ```json
    POST /clean-session
    {
      "session_id": "abc123",
      "force_purge": false
    }
    ```
    El sistema responde con un resumen de la limpieza: tokens liberados, mensajes reparados y tiempo de ejecución.

### Mejores Prácticas (Industry Best Practices)

- **Implementa limpieza diferida en pipelines de alta concurrencia**  
  En entornos con picos de tráfico (ej: chatbots de soporte), programa la limpieza para ejecutarse en ventanas de baja actividad usando colas de mensajes (RabbitMQ, Redis). Esto evita latencia adicional durante las peticiones del usuario y asegura que la purga no compita por recursos con la inferencia del modelo.

- **Usa logs estructurados para auditoría**  
  Activa el modo `--audit-log` para que `agents-clean` registre cada acción de saneamiento en formato JSON estructurado. Esto es crítico para cumplir con regulaciones como SOC2 o HIPAA. Los logs incluyen: timestamp, sesión afectada, tipo de operación (dedup, repair, purge) y hash del estado antes/después. Puedes enviarlos a un sistema centralizado como ELK o Datadog.
