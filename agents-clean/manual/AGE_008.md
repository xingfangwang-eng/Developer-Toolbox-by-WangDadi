## Detailed Content

### What is agents-clean?

**agents-clean** is a production-grade, runtime-safe agent memory & state sanitizer designed for multi-agent orchestration systems. It automatically detects, isolates, and purges stale, corrupted, or hallucination-prone agent context without breaking inter-agent communication pipelines. Built for developers who need deterministic cleanup in non-deterministic LLM environments.

### Core Features

- **Zero-Trust Context Sweeper** – Automatically scans agent memory for hallucinated variables, orphaned references, and expired tokens. Uses a rule-based + heuristic engine to flag and remove contaminated state before it propagates to downstream agents.
- **Immutable Audit Trail** – Every cleanup operation is logged as a cryptographically signed event. Supports replay-based debugging for post-mortem analysis of agent state corruption chains.
- **Pluggable Cleanup Policies** – Define custom cleanup rules per agent role (e.g., `retriever`, `planner`, `executor`). Supports time-to-live (TTL), dependency graph pruning, and semantic similarity thresholds for context eviction.

### How to Use

1. **Inject the Cleanup Hook into Your Agent Runtime**  
   Install `agents-clean` via npm or pip, then wrap your agent’s `step()` or `execute()` method with the `@clean_context` decorator. Example:  
   ```python
   from agents_clean import clean_context
   
   @clean_context(policy="aggressive", ttl=300)
   async def my_agent_step(state):
       # your logic here
       return state
   ```
   This ensures every agent cycle triggers a state validation before the next LLM call.

2. **Define Role-Specific Cleanup Policies in YAML**  
   Create a `cleanup_policies.yaml` file in your project root. Specify rules per agent role:  
   ```yaml
   roles:
     retriever:
       max_context_size: 4096
       evict_similarity: 0.85
     executor:
       ttl_seconds: 120
       block_orphan_refs: true
   ```
   Run `agents-clean validate` to lint your config before deployment.

3. **Run a Dry-Clean Audit in Staging**  
   Use the CLI tool to simulate cleanup without side effects:  
   ```bash
   agents-clean audit --env staging --log-level debug
   ```
   Inspect the generated `audit_report.json` for flagged contexts, then apply the cleanup with `--apply`. This prevents accidental data loss in production.

### Best Practices

- **Always pair `agents-clean` with a state persistence layer (e.g., Redis, PostgreSQL).**  
  Cleanup should never be the source of truth—only the curator. Store agent state externally and let `agents-clean` manage eviction and compaction. This decouples cleanup logic from agent execution, allowing rollback and recovery.

- **Set aggressive TTLs for intermediate agent steps, but soft TTLs for final outputs.**  
  Agents in the middle of a pipeline (e.g., summarizers, translators) often carry volatile context. Use a TTL of 60–120 seconds for these. For final output agents (e.g., report generators), use a soft TTL with a similarity threshold to avoid truncating valid but verbose results.
