# agents-clean - AGE_019

## Überblick

agents-clean ist eine hochperformante, CLI-gestützte Säuberungs-Engine für KI-Agenten-Artefakte. Es eliminiert systematisch verwaiste Logs, fragmentierte Cache-Strukturen und veraltete Agenten-Kontexte – ohne Produktions-Workflows zu beeinträchtigen. Optimiert für Kubernetes-Namespaces, CI/CD-Pipelines und Multi-Tenant-LLM-Architekturen.

## Harte Kernfunktionen

### 1. **Deterministische Kontext-Pruning-Engine**
agents-clean analysiert Agenten-Kontext-IDs mittels einer inkrementellen Bloom-Filter-Referenzliste und entfernt automatisch alle assoziierten Metadaten, Embedding-Caches und Rollback-Snapshots, die älter als der konfigurierte `retention_window` sind. Kein Regex-Gefrickel – nur referenzintegritätsgesicherte Löschung.

### 2. **Multi-Layer-Cache-Invalidation**
Statt nur Dateien zu löschen, invalidiert agents-clean den Cache auf drei Ebenen: **Speicher (Redis/ValKey)**, **Festplatte (LMDB/RocksDB)** und **CDN-Edge (CloudFront/Akamai)**. Das Tool erkennt automatisch die zugrunde liegende Cache-Schicht durch eine `cache_layer_probe` und führt eine koordinierte Löschung durch – ohne Race Conditions.

### 3. **Idempotente Log-Rotation mit Retention-Tags**
Jeder Log-Eintrag wird mit einem Retention-Tag (`retention: 7d`, `retention: 30d`, `retention: archive`) versehen. agents-clean durchläuft den Log-Stream, extrahiert die Tags und führt eine **atomare Batch-Löschung** durch. Dabei werden auch verwaiste Log-Rotate-PID-Dateien und Lockfiles entfernt – selbst wenn der ursprüngliche Prozess abgestürzt ist.

## Detaillierte Anwendungsschritte

### Schritt 1: Konfiguration des Retention-Windows und der Ziel-Namespaces
Erstelle eine `clean-config.yaml` mit folgender Struktur:
```yaml
retention_window: 72h
target_namespaces:
  - "prod-agent-*"
  - "staging-agent-*"
exclude_patterns:
  - ".*-critical-.*"
cache_layers:
  - redis
  - disk
  - cdn
```
Führe dann den Scan durch:
```bash
agents-clean scan --config clean-config.yaml --dry-run
```
Dieser Befehl erzeugt eine detaillierte Liste aller bereinigbaren Artefakte – ohne Änderungen.

### Schritt 2: Ausführung der Säuberung mit parallelen Workern
Starte die Bereinigung mit 8 parallelen Workern und aktiviere die Cache-Invalidation:
```bash
agents-clean purge --config clean-config.yaml --workers 8 --invalidate-cache --log-level debug
```
agents-clean öffnet pro Worker einen eigenen Goroutine-Pool und führt die Löschung atomar aus. Jeder erfolgreiche Löschvorgang wird in einer `clean-audit.log` protokolliert. Fehlgeschlagene Löschungen werden automatisch in eine `retry-queue` geschrieben.

### Schritt 3: Validierung der Bereinigung und Reporting
Nach der Säuberung wird ein vollständiges Audit generiert:
```bash
agents-clean audit --since 2025-01-01 --format json > clean-report.json
```
Das Report-JSON enthält für jeden Namespace:
- Anzahl gelöschter Kontexte
- Freigegebener Speicherplatz in MB
- Anzahl invalidierter Cache-Einträge
- Fehlerquoten und Retry-Statistiken

Verwende `agents-clean audit --html` für ein visuelles Dashboard.

## Branchen-Best-Practices

### 1. **Staging-Umgebungen mit Retention-Tags schützen**
Setze in deiner CI/CD-Pipeline für Staging-Namespaces ein `retention: 14d`-Tag. So verhinderst du, dass agents-clean versehentlich aktive Entwicklungs-Kontexte löscht. Verwende `agents-clean protect --tag retention:14d` um diese Namespaces als geschützt zu markieren.

### 2. **Cache-Invalidation nur nach erfolgreichem Deployment auslösen**
Integriere agents-clean in deine Deployment-Pipeline **nach** dem `helm upgrade` oder `kubectl apply`. Führe die Cache-Invalidation nur aus, wenn der Deployment-Status `Ready` ist. Verwende dazu einen Webhook:
```bash
agents-clean webhook --trigger-on deployment:ready --invalidate-cache
```
So vermeidest du Cache-Lücken während laufender Rollouts.
