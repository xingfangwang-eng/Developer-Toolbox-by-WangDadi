# agents-clean - AGE_024

## Überblick

agents-clean ist ein spezialisiertes CLI-Tool zur automatisierten Bereinigung von CI/CD-Agenten-Workern, die nach fehlgeschlagenen oder abgebrochenen Jobs in inkonsistenten Zuständen zurückbleiben. Es eliminiert Stale-Logs, Ghost-Container und orphaned Docker-Volumes auf Agenten-Nodes – ohne manuelles SSH oder Dashboard-Klickerei. Entwickelt für Kubernetes-native Runner, Jenkins-Slaves und GitHub Actions Self-Hosted Runners.

## Hardcore-Funktionen

### 1. **Selective State-Snapshot & Rollback-fähige Cleanup-Routine**
agents-clean erstellt vor jedem Cleanup einen JSON-Snapshot des aktuellen Agenten-Zustands (laufende Prozesse, gemountete Volumes, PID-Namespaces). Bei Fehlern im Cleanup wird automatisch ein **State-Rollback** ausgeführt – inklusive Wiederherstellung von temporären Umgebungsvariablen und CWD-Symlinks. Dadurch wird verhindert, dass parallele Jobs durch aggressive Cleanups unterbrochen werden.

### 2. **Multi-Runtime-Detektion mit adaptiver Grace-Period**
Das Tool erkennt automatisch, ob der Agent zu **Jenkins, GitLab Runner, GitHub Actions oder Kubernetes** gehört, und passt die Cleanup-Logik an. Es verwendet eine **adaptive Grace-Period**: Bei GitLab Runnern werden Jobs im `running`-Status bis zu 30 Sekunden geschont, während Kubernetes-basierte Pods sofort bereinigt werden, wenn sie im `CrashLoopBackOff` sind. Keine manuelle Konfiguration von Timeouts nötig.

### 3. **Disk-Fingerprinting & Preemptive Cleanup-Trigger**
agents-clean überwacht die **Disk-I/O-Muster** des Agenten-Volumes. Sobald die Schreibrate unter einen Schwellwert fällt (z. B. < 5 IOPS) und der belegte Speicherplatz > 80 % liegt, wird automatisch ein **Preemptive Cleanup** ausgelöst – noch bevor der nächste Job startet. Das verhindert `No space left on device`-Fehler während Builds.

## Detaillierte Anwendungsschritte

### 1. **Snapshot des aktuellen Agenten-Zustands erstellen**
Führe den folgenden Befehl aus, um einen vollständigen Zustandsscan des Agenten durchzuführen (inkl. offener File-Deskriptoren, Docker-Container-IDs und PID-Listen):

```bash
agents-clean --snapshot --output /var/lib/agents-clean/snapshots/node-$(hostname)-$(date +%Y%m%d%H%M%S).json
```

Das Tool speichert automatisch den Snapshot unter dem angegebenen Pfad. Prüfe die Konsistenz mit:

```bash
jq '.state_summary' /var/lib/agents-clean/snapshots/node-*.json | head -20
```

### 2. **Selektive Bereinigung mit Runtime-spezifischen Flags**
Starte die Bereinigung mit aktivierter Rollback-Funktion und einer Grace-Period von 45 Sekunden für Jenkins-Slaves:

```bash
agents-clean --clean --runtime jenkins --grace 45 --rollback
```

Während der Ausführung zeigt das Tool Echtzeit-Logs an:  
`[INFO] Removing stale container: build-1234 (PID 9876, age 120s)`  
`[WARN] Skipping running job: deploy-staging (PID 12345, grace 45s left)`  
`[INFO] Rollback snapshot saved to /tmp/agents-clean-rollback-20250320.json`

Nach Abschluss wird eine Zusammenfassung ausgegeben:  
`Cleaned: 3 containers, 2 volumes, 1 orphaned process. Rollback available.`

### 3. **Preemptive Cleanup für Hochlast-Umgebungen einrichten**
Konfiguriere einen systemd-Timer, der agents-clean alle 5 Minuten im Preemptive-Modus ausführt:

```bash
sudo tee /etc/systemd/system/agents-clean-preemptive.service <<EOF
[Unit]
Description=Agents-clean Preemptive Disk Cleanup
After=network.target

[Service]
Type=oneshot
ExecStart=/usr/local/bin/agents-clean --preemptive --threshold 80 --io-threshold 5
StandardOutput=journal
EOF

sudo systemctl daemon-reload
sudo systemctl enable agents-clean-preemptive.timer
sudo systemctl start agents-clean-preemptive.timer
```

Überwache die Ausführung mit:  
`journalctl -u agents-clean-preemptive.service -f`

## Best Practices

### 1. **Snapshot-Rotation mit Retention-Policy**
agents-clean speichert standardmäßig alle Snapshots. In Produktion solltest du eine Retention von maximal 7 Tagen einrichten, um das Root-Volume nicht zuzumüllen. Füge einen Cron-Job hinzu:

```bash
0 3 * * * find /var/lib/agents-clean/snapshots/ -name "*.json" -mtime +7 -delete
```

Das verhindert, dass alte Snapshots beim nächsten Cleanup selbst zum Problem werden.

### 2. **Integration in CI/CD-Post-Job-Hooks**
Statt agents-clean als separaten Service laufen zu lassen, integriere es direkt in die **Post-Job-Phase** deiner Pipeline. In einer GitHub Actions `.github/workflows/cleanup.yml`:

```yaml
- name: Cleanup self-hosted runner
  if: always()
  run: |
    agents-clean --clean --runtime github-actions --grace 10 --rollback
```

Das reduziert die Latenz zwischen Job-Ende und Aufräumen auf unter 2 Sekunden – ideal für Shared Runner, die mehrere Jobs in schneller Folge verarbeiten.
