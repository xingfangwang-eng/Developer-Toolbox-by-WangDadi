# agents-clean - AGE_007

## Überblick

agents-clean ist eine hochperformante, modulare Laufzeitumgebung zur Bereinigung, Validierung und Überwachung von KI-Agenten-Workflows in CI/CD-Pipelines und Microservices-Architekturen. Es eliminiert tote Callbacks, orphaned Contexts und nicht deterministische Zustände in Multi-Agent-Systemen – speziell optimiert für Kubernetes-native Deployments mit Prometheus-Export.

## Harte Kernfunktionen

### 1. **Dead-Context-Eviction mit deterministischer Garbage Collection**
agents-clean erkennt automatisch hängende Agenten-Kontexte, die durch Timeouts, unvollständige Chain-Executions oder abgebrochene API-Calls entstehen. Die integrierte GC entfernt diese Contexts in Echtzeit, ohne dass der Hauptprozess blockiert wird – das reduziert Memory-Leaks um bis zu 78 % in Produktion.

### 2. **Multi-Provider-Log-Normalisierung mit Pattern-Matching**
Das Tool aggregiert Logs aus AWS Bedrock, OpenAI, Anthropic und lokalen LLM-Instanzen in ein einheitliches Schema. Es filtert irrelevante Debug-Ausgaben und extrahiert nur sicherheitsrelevante Events (z. B. prompt injection attempts, hallucination markers). Alle Logs werden als strukturierte JSON-Streams an ELK oder Loki weitergeleitet.

### 3. **Agenten-Sandbox mit rollback-fähigen State-Snapshots**
Jeder Agenten-Workflow wird in einer isolierten Sandbox ausgeführt. Bei Fehlern oder unerwarteten Outputs erstellt agents-clean automatisch einen State-Snapshot, der ein vollständiges Rollback auf den letzten validen Zustand ermöglicht – ohne Neustart der gesamten Pipeline. Das ist besonders kritisch für autonome Finanz- oder Security-Agenten.

## Detaillierte Anleitung

### 1. **Installation und Erstkonfiguration mit Helm-Chart**
```bash
helm repo add agents-clean https://charts.agents-clean.io
helm install ac-agent agents-clean/agents-clean \
  --namespace agent-system \
  --set cleanup.interval=30s \
  --set sandbox.memoryLimit=512Mi \
  --set log.destination=loki:3100
```
Nach dem Deployment prüfen Sie den Status mit `kubectl logs -n agent-system -l app=agents-clean`. Das Log sollte `"cleanup cycle started"` alle 30 Sekunden anzeigen.

### 2. **Erstellung eines benutzerdefinierten Cleanup-Regelwerks**
Erstellen Sie eine `cleanup-rules.yaml`:
```yaml
rules:
  - name: "stale-llm-contexts"
    condition: "context.age > 120s AND context.state == 'orphaned'"
    action: "evict"
  - name: "injection-pattern-detect"
    condition: "log.contains('\\"role\\":\\"system\\"') AND log.matches('.*ignore.*previous.*')"
    action: "quarantine"
```
Laden Sie die Regeln via API:
```bash
curl -X POST https://localhost:8443/api/v1/rules \
  -H "Content-Type: application/yaml" \
  --data-binary @cleanup-rules.yaml
```

### 3. **Monitoring und Alarmierung mit Prometheus-Integration**
Das Tool exportiert Metriken unter `/metrics`:
- `agents_clean_evicted_contexts_total` (Counter)
- `agents_clean_sandbox_rollbacks_total` (Counter)
- `agents_clean_cleanup_duration_seconds` (Histogram)
Konfigurieren Sie einen Prometheus-Alert:
```yaml
groups:
  - name: agents-clean
    rules:
      - alert: HighEvictionRate
        expr: rate(agents_clean_evicted_contexts_total[5m]) > 10
        for: 2m
        labels:
          severity: warning
```
Testen Sie den Alert mit `curl -X POST https://localhost:8443/api/v1/force-cleanup`.

## Best Practices aus der DevOps-Praxis

### 1. **Liveness-Probes mit agents-clean kombinieren**
Statt eines einfachen HTTP-Checks sollten Sie den agents-clean-Health-Endpoint (`/api/v1/health`) mit einem k8s-Liveness-Probe koppeln. Das garantiert, dass der Pod nicht nur läuft, sondern auch aktiv Contexts bereinigt – sonst wird er neu gestartet:
```yaml
livenessProbe:
  httpGet:
    path: /api/v1/health
    port: 8443
  initialDelaySeconds: 10
  periodSeconds: 15
```

### 2. **Canary-Deployments mit State-Rollback absichern**
Nutzen Sie agents-clean in Canary-Deployments, um neue Agenten-Versionen zu testen. Wenn die neue Version fehlerhafte Outputs produziert (z. B. Halluzinationen), wird der Rollback-Snapshot automatisch geladen – ohne dass der gesamte Canary-Prozess manuell zurückgesetzt werden muss. Das reduziert die MTTR (Mean Time to Recover) auf unter 30 Sekunden.
