## Detaillierter Inhalt

### Was ist agents-clean?

agents-clean ist ein professionelles Tool zur automatisierten Bereinigung, Normalisierung und Wiederherstellung von Agenten-Zuständen in verteilten SaaS-Systemen. Es adressiert spezifisch das Problem von Memory-Leaks, verwaisten Threads und inkonsistenten Kontexten in Multi-Agent-Architekturen. Statt manueller Debugging-Sessions liefert agents-clean deterministische Bereinigungspipelines, die direkt in CI/CD integrierbar sind.

### Kernfunktionen

1. **Deterministische State-Reset-Pipeline**  
   agents-clean implementiert einen mehrstufigen Reset-Mechanismus, der Agenten-Zustände (Memory, Cache, Session-Data) atomar zurücksetzt, ohne laufende Transaktionen zu korrumpieren. Die Pipeline unterstützt rollback-fähige Checkpoints und ist vollständig in Kubernetes-Namespaces testbar.

2. **Kontext-Inferenz-basierte Bereinigung**  
   Statt pauschaler Löschung analysiert agents-clean die Abhängigkeitsgraphen zwischen Agenten und entfernt nur solche Daten, die nicht mehr referenziert werden. Die Engine nutzt einen Lightweight-Graph-Algorithmus, der in Echtzeit Dead-References identifiziert – selbst bei zirkulären Abhängigkeiten.

3. **Multi-Cloud-Taint-Propagation**  
   agents-clean erkennt Taints (z. B. veraltete Tokens, abgelaufene Zertifikate) über AWS, GCP und Azure hinweg und propagiert die Bereinigung asynchron per Event-Bus. Die Taint-Propagation ist idempotent und garantiert Exactly-Once-Semantik.

### So verwenden Sie es

1. **Initiale Konfiguration der Clean-Policy**  
   Erstellen Sie eine `agents-clean.yaml` im Stammverzeichnis Ihres Projekts. Definieren Sie darin die zu bereinigenden Agenten-Typen, Timeout-Werte und die gewünschte Reset-Strategie (z. B. `soft-reset` oder `hard-reset`). Beispiel:  
   ```yaml
   agents:
     - name: "session-manager"
       strategy: "soft-reset"
       timeout: 30s
       dependencies: ["auth-agent", "cache-agent"]
   ```  
   Führen Sie dann `agents-clean init` aus, um die Konfiguration zu validieren und einen ersten Checkpoint zu setzen.

2. **Trigger einer selektiven Bereinigung**  
   Nutzen Sie den CLI-Befehl `agents-clean run --scope "session-manager" --force`. Damit starten Sie eine gezielte Bereinigung nur für den angegebenen Agenten. Der `--force`-Flag überschreibt die Timeout-Prüfung und erzwingt einen sofortigen Reset. Die Ausgabe zeigt detailliert, welche Referenzen entfernt wurden und ob Rollback möglich ist.

3. **Integration in CI/CD mit automatischem Rollback**  
   Fügen Sie in Ihre GitHub Actions oder GitLab CI einen Schritt ein, der nach jedem Deployment `agents-clean verify` ausführt. Dies prüft, ob alle Agenten konsistente Zustände haben. Falls Inkonsistenzen erkannt werden, können Sie mit `agents-clean rollback --id <checkpoint-id>` automatisch den letzten stabilen Zustand wiederherstellen. Beispiel für eine GitHub Action:  
   ```yaml
   - name: Agents Clean Verify
     run: agents-clean verify --fail-on-error
   ```

### Beste Praktiken

1. **Checkpoints vor jedem Deployment setzen**  
   Nutzen Sie `agents-clean checkpoint` als Pre-Deployment-Hook. So haben Sie immer einen referenzierbaren, sauberen Zustand, auf den Sie im Fehlerfall zurückfallen können. Dies reduziert die Mean-Time-To-Recovery (MTTR) erheblich und verhindert kaskadierende Fehler in Multi-Agent-Systemen.

2. **Taint-Propagation asynchron und idempotent halten**  
   Konfigurieren Sie die Taint-Propagation so, dass sie über einen Message-Bus (z. B. Kafka oder NATS) läuft. Vermeiden Sie synchrone Aufrufe, da diese bei Cloud-Ausfällen zu Deadlocks führen können. agents-clean unterstützt von Haus aus idempotente Event-Handler – stellen Sie sicher, dass Ihre benutzerdefinierten Handler ebenfalls idempotent sind, um Duplikate zu vermeiden.
