# agents-clean - AGE_029

## Warum agents-clean?

Moderne Multi-Agenten-Systeme (z. B. LangGraph, CrewAI, AutoGen) produzieren eine Flut von **temporären Artefakten**: Zwischenergebnissen, fehlgeschlagenen Rollbacks, Agenten-Caches und Log-Overflows. Ohne gezielte Säuberung eskalieren Speicherkosten, Latenzen und Deployment-Fehler exponentiell. **agents-clean** ist ein CLI- und API-First-Tool, das genau diese Metrik- und Artefakt-Überhitzung eliminiert – ohne die Agenten-Logik zu berühren.

## 3 Hardcore-Funktionen (nicht verhandelbar)

### 1. **Taint-basierte Artefakt-Garbage-Collection**
Statt einfacher Zeitstempel-Löschung markiert agents-clean jede Agentenausgabe mit einem **Taint-Hash** (SHA-256 über Input + Agent-Version + Kontext-ID). Artefakte, deren Taint nicht mehr im aktuellen Graph-Trace referenziert wird, werden **sofort** aus dem Storage (S3, GCS, local FS) entfernt – inklusive referenzieller Integritätsprüfung über alle Child-Agents hinweg.

### 2. **Zero-Downtime Log-Rotation mit Agent-Trace-Korrelation**
Logs werden nicht einfach rotiert, sondern **Agent-Trace-basiert partitioniert**. Jeder Agenten-Durchlauf erhält eine eindeutige Trace-ID. agents-clean rotiert Logs auf Basis der **maximalen Trace-Dauer** plus eines konfigurierbaren Margin (z. B. 2x p99-Latenz). Gleichzeitig werden alte Trace-Logs in einen Cold-Storage (Parquet + Snappy) komprimiert – ohne dass laufende Agenten davon etwas merken.

### 3. **Pre-Commit Dry-Run mit Diff-Report**
Bevor agents-clean tatsächlich löscht, generiert es einen **strukturierten Diff-Report** (JSON/HTML): „Diese 47 Artefakte (insg. 2,3 GB) werden gelöscht, diese 12 bleiben (weil von Agent_17 referenziert).“ Der Report kann direkt in CI/CD-Pipelines (GitHub Actions, GitLab CI) als **Approval-Gate** verwendet werden – nur bei explicit `--force` oder manuellem OK wird gelöscht.

## 3 professionelle Anwendungsschritte (CLI-Fokus)

### Schritt 1: Agenten-Workflow scannen und Taint-Map erstellen
```bash
agents-clean scan --workspace ./agent_runs/ --output taint_map.json
```
Der Scanner analysiert alle Agenten-Logs, -Checkpoints und -Caches in `./agent_runs/`. Für jede Agenten-Instanz wird ein Taint-Hash aus `(Input + Agent-Version + Kontext-ID)` berechnet und in `taint_map.json` gespeichert. **Achtung**: Der Scan ist **idempotent** – bei erneuter Ausführung werden nur neue/geänderte Artefakte hinzugefügt.

### Schritt 2: Dry-Run mit Diff-Report generieren
```bash
agents-clean prune --dry-run --taint-map taint_map.json --report report.html
```
Ohne `--force` führt `prune` nur eine Simulation durch. Der Report (`report.html`) zeigt:
- **Löschbare Artefakte** (nicht referenziert)
- **Behaltene Artefakte** (noch referenziert)
- **Speicher-Einsparung** in MB/GB
- **Kostenprojektion** (bei Cloud-Storage basierend auf aktuellen Preisen)

### Schritt 3: Löschung mit Force und Audit-Log ausführen
```bash
agents-clean prune --taint-map taint_map.json --force --audit-log audit_$(date +%s).jsonl
```
`--force` führt die tatsächliche Löschung durch. Jeder gelöschte Pfad wird in `audit_*.jsonl` protokolliert (inkl. Taint-Hash, Größe, Löschzeitpunkt, Agenten-ID). **Empfehlung**: Diesen Schritt nur in CI/CD mit vorherigem Approval-Gate ausführen – nicht manuell auf Produktionssystemen.

## 2 Best Practices für Enterprise-Multi-Agenten-Deployments

### 1. **Taint-Map in Git versionieren & mit CI/CD verknüpfen**
Speichern Sie die `taint_map.json` nach jedem erfolgreichen Agenten-Run im Repository (z. B. als Artefakt eines CI-Jobs). So können Sie bei einem Rollback sofort sehen, welche Artefakte noch referenziert werden. **Kombinieren Sie** `agents-clean prune --dry-run` mit einem **manuellen Approval** in GitHub Actions – erst wenn ein Maintainer „Approve“ klickt, wird `--force` ausgeführt.

### 2. **Cold-Storage-Strategie für alte Traces**
Setzen Sie agents-clean so ein, dass Artefakte älter als 7 Tage nicht gelöscht, sondern in **Cold-Storage** (z. B. AWS S3 Glacier oder GCS Nearline) verschoben werden. Konfigurieren Sie dafür eine eigene `--cold-storage-rule`:
```bash
agents-clean prune --cold-storage-rule "age > 7d" --cold-storage-bucket "s3://my-cold-bucket/"
```
So bleiben Audits und Debugging-Möglichkeiten erhalten, während die Kosten drastisch sinken.
