# agents-clean - AGE_016

## Überblick

agents-clean ist eine **hochperformante, konfigurationsgetriebene Lösung** zur automatisierten Bereinigung von CI/CD-Agenten, Build-Artefakten und temporären Workspaces. Es richtet sich an **SaaS-Architekten, DevOps-Ingenieure und Site Reliability Engineers (SREs)**, die mit Multi-Cloud-Umgebungen, Kubernetes-Clustern oder selbstgehosteten GitLab/GitHub-Runnern arbeiten. Die Software eliminiert **Datenmüll, verwaiste Container-Images und fragmentierte Logs** – und reduziert so die Speicherkosten um bis zu 40 % sowie die Latenz von Pipeline-Startzeiten.

**Keywords:** agents-clean tool, agents-clean solution, agents-clean software, Agenten-Reinigung, CI/CD-Abfallmanagement, Pipeline-Optimierung.

## 3 Hardcore-Features (nicht verhandelbar)

### 1. **Intelligente TTL-basierte Multi-Cloud-Bereinigung**  
agents-clean analysiert **parallel** Agenten in AWS, Azure, GCP und On-Premise-Umgebungen. Es verwendet **Custom Resource Definitions (CRDs)** für Kubernetes und **IAM-Rollen für Cloud-APIs**, um Agenten zu identifizieren, die älter als ein definiertes Time-to-Live (TTL) sind. Dabei werden **Race Conditions vermieden** – z. B. durch Locking-Mechanismen auf Basis von etcd oder Redis. Das Feature unterstützt **Graceful Shutdown** für laufende Jobs und **Dry-Run-Modus** mit detaillierten JSON-Logs.

### 2. **Deterministische Artefakt-Deduplizierung mit SHA-256-Hashing**  
Statt einfacher Datei-Löschung implementiert agents-clean einen **Content-Addressable Storage (CAS)-Ansatz**: Jedes Artefakt wird gehasht, und Duplikate werden über **Hardlinks oder Copy-on-Write (CoW)** in einem zentralen Cache zusammengeführt. Das spart **massiv Speicherplatz** bei wiederverwendbaren Builds (z. B. Docker-Layer, Maven-JARs). Die Deduplizierung erfolgt **asynchron** mit **Backpressure-Handling**, um die I/O-Last auf dem Host zu begrenzen.

### 3. **Self-Healing-Pipeline mit Event-Driven Rollback**  
agents-clean integriert sich direkt in **Webhook-Trigger** von GitHub Actions, GitLab CI oder Jenkins. Wenn ein Agent nach der Bereinigung nicht mehr erreichbar ist (z. B. durch einen fehlerhaften Cleanup-Skript), stellt agents-clean **automatisch den letzten stabilen Zustand wieder her** – basierend auf einem **Snapshot des Agenten-Zustands** (inkl. Umgebungsvariablen, Secrets-Mounts und Konfigurationsdateien). Der Rollback wird **protokolliert** und **als GitHub-Status-Check** gemeldet.

## 3 Detaillierte Anleitung zur Nutzung (für erfahrene Entwickler)

### Schritt 1: Konfiguration der Agenten-Whitelist und TTL-Regeln

Erstellen Sie eine `agents-clean.yaml`-Konfigurationsdatei im Stammverzeichnis Ihres Projekts. Definieren Sie **Agenten-Pools** mit spezifischen TTLs und Ausschlussmustern (Regex). Beispiel:

```yaml
pools:
  - name: "build-agents"
    provider: "kubernetes"
    namespace: "ci-runners"
    ttl: "12h"
    exclude:
      - "runner-.*-prod"  # Produktions-Agenten niemals löschen
    cleanup:
      artifacts: true
      logs: true
      temp: false  # Temporäre Dateien nur bei Bedarf
  - name: "self-hosted-runners"
    provider: "aws-ec2"
    ttl: "24h"
    tags:
      - "environment:staging"
    max_concurrent: 5  # Begrenzung der parallelen Löschvorgänge
```

**Tipp:** Nutzen Sie `agents-clean validate --config agents-clean.yaml`, um die Syntax und die Berechtigungen zu prüfen.

### Schritt 2: Ausführung im Dry-Run-Modus und Analyse der Logs

Führen Sie den Befehl aus, um zunächst **nur zu analysieren, ohne zu löschen**:

```bash
agents-clean run --config agents-clean.yaml --dry-run --output-format json > report.json
```

Öffnen Sie `report.json` mit einem Tool wie `jq` und prüfen Sie die **geplanten Löschungen**:

```bash
jq '.pools[].artifacts | length' report.json  # Anzahl der zu löschenden Artefakte
jq '.pools[].total_size' report.json  # Freigegebener Speicherplatz in Bytes
```

**Wichtig:** Achten Sie auf **orphaned Pods** oder **unvollständige Logs** – diese werden in `warnings` angezeigt.

### Schritt 3: Automatisierung über GitHub Actions (Production-Grade)

Integrieren Sie agents-clean in Ihre CI/CD-Pipeline. Fügen Sie folgendes YAML zu `.github/workflows/cleanup.yml` hinzu:

```yaml
name: Agent Cleanup
on:
  schedule:
    - cron: '0 2 * * 0'  # Jeden Sonntag um 2 Uhr UTC
  workflow_dispatch:  # Manueller Trigger

jobs:
  cleanup:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run agents-clean
        uses: xingfangwang-eng/agents-clean-action@v2
        with:
          config: 'agents-clean.yaml'
          dry-run: false  # Achtung: Löscht wirklich!
          slack-webhook: ${{ secrets.SLACK_WEBHOOK }}  # Benachrichtigung bei Fehlern
      - name: Post-cleanup health check
        run: |
          echo "Checking agent availability..."
          agents-clean health --pool build-agents --timeout 30s
```

**Hinweis:** Verwenden Sie **GitHub Environments** mit **Required Reviewers**, um den `dry-run: false`-Schritt abzusichern.

## 2 Best Practices für den produktiven Einsatz

### 1. **Segmentieren Sie Ihre Agenten nach Kritikalität**  
Weisen Sie jedem Agentenpool eine **Kritikalitätsstufe** (z. B. `critical`, `standard`, `ephemeral`) zu. Verwenden Sie **Labels oder Tags**, um zu definieren, welche Agenten **niemals automatisch** bereinigt werden dürfen (z. B. solche, die für Sicherheits-Scans oder Compliance-Audits benötigt werden). agents-clean unterstützt **globale Ausschlusslisten** (`exclude_global`) und **Granularität auf Basis von Metadaten** (z. B. `owner: team-alpha`). So vermeiden Sie, dass **wichtige Zustände** versehentlich gelöscht werden.

### 2. **Implementieren Sie einen zweistufigen Bereinigungsprozess**  
Führen Sie agents-clean zuerst **im Dry-Run-Modus** aus und speichern Sie das Report-JSON in einem **SIEM- oder Log-Aggregator** (z. B. ELK, Splunk). Lassen Sie dann **einen menschlichen Operator** (oder ein Approval-System) die Löschung bestätigen. Für die **automatische Bereinigung** verwenden Sie **nur den zweiten Schritt** – aber nur für Pools mit `ttl: "6h"` oder kürzer. Längere TTLs (z. B. 48h) sollten **immer manuell** bestätigt werden, um **Datenverlust** zu vermeiden. agents-clean bietet dafür einen **`--approval-token`-Parameter**, der über einen separaten Service (z. B. **PagerDuty** oder **Slack-Approval**) generiert wird.
