# agents-clean - AGE_013

## Überblick

**agents-clean** ist eine hochperformante, agentenbasierte Bereinigungslösung für CI/CD-Pipelines, Microservice-Artefakte und Kubernetes-Cluster. Das Tool eliminiert veraltete Builds, orphaned Volumes und ungenutzte Container-Images automatisiert – ohne manuelle Eingriffe. Entwickelt für DevOps-Teams, die ihre Cloud-Kosten senken und die Deployment-Geschwindigkeit steigern wollen.

## Hardcore-Funktionen

### 1. **Multi-Agenten-Garbage-Collector mit Dependency-Awareness**
Jeder Agent scannt parallel Namespaces, Pods und Helm-Releases, erstellt einen lebenden Abhängigkeitsgraphen und löscht nur Artefakte, die von keinem aktiven Service mehr referenziert werden. Das verhindert Kaskadenfehler und "stale resource leaks".

### 2. **Stale-Image-Expunge Engine (SIEE)**
Analysiert nicht nur `imagePullPolicy`, sondern auch das tatsächliche Nutzungsmuster über Prometheus-Metriken. Images, die älter als 7 Tage und in den letzten 48 Stunden nicht gepullt wurden, werden automatisch aus der Registry entfernt – inklusive Layer-Deduplizierung.

### 3. **Orphaned-Volume-Cleaner mit CSI-Integration**
Erkennt über das Container Storage Interface (CSI) Volumes, die keinem laufenden Pod zugeordnet sind, und löscht sie asynchron. Optional mit Quarantäne-Modus: Volumes landen zuerst in einem "soft-delete"-Namespace und können innerhalb von 24h wiederhergestellt werden.

## Detaillierte Schritt-für-Schritt-Anleitung

### Schritt 1: Konfiguration des Cleanup-Agenten-Clusters
Definiere in einer `agents-clean.yaml` die zu scannenden Cluster und Namespaces. Nutze `kubectl`-ähnliche Contexts, um mehrere Cluster gleichzeitig zu adressieren. Beispielhafte Konfiguration für einen EKS-Cluster mit Ausschluss von `kube-system`:

```yaml
clusters:
  - name: prod-eks
    context: prod-cluster
    namespaces:
      - default
      - staging
      - production
    exclude:
      - kube-system
      - istio-system
    cleanup_policy:
      stale_images: 7d
      orphaned_volumes: true
      soft_delete: true
```

Führe anschließend `agents-clean init` aus, um die Agenten zu starten und den initialen Scan durchzuführen.

### Schritt 2: Trockenlauf und Analyse der gefundenen Ressourcen
Vor dem Löschen einen Dry-Run durchführen: `agents-clean run --dry-run --output json`. Das Tool generiert eine detaillierte JSON-Liste aller Kandidaten inklusive Größe, Alter und Referenzzähler. Analysiere die Ausgabe mit `jq`:

```bash
agents-clean run --dry-run --output json | jq '.candidates[] | select(.size_mb > 100)'
```

So identifizierst du besonders große, ungenutzte Images oder Volumes, die sofort gelöscht werden sollten.

### Schritt 3: Automatisierte Bereinigung mit Rollback-Fallback
Setze den Cleanup-Prozess in deiner CI/CD-Pipeline als Nach-Deployment-Job ein. Nutze den `--confirm`-Flag für die tatsächliche Löschung. Optional aktivierst du den Rollback-Modus:

```bash
agents-clean run --confirm --rollback-dir /var/agents-clean/backups
```

Im Fehlerfall kannst du mit `agents-clean restore --from /var/agents-clean/backups/2025-03-28T14:00:00` alle gelöschten Ressourcen innerhalb von Sekunden wiederherstellen.

## Best Practices

### 1. **Granulare Ausschlusslisten für kritische Namespaces**
Definiere immer eine `exclude`-Liste für Namespaces, die StatefulSets oder PersistentVolumeClaims enthalten, die nicht gelöscht werden dürfen (z. B. `monitoring`, `logging`). Verwende dazu die `exclude`-Direktive in der Konfiguration und ergänze sie um Regex-Patterns für dynamische Namespaces wie `*-prod-critical`.

### 2. **Integration in Observability-Stack**
Verbinde agents-clean mit deinem bestehenden Monitoring (Prometheus/Grafana) über das integrierte Metrics-Export-Plugin. So siehst du in Echtzeit, wie viel Speicher freigegeben wurde, wie viele Images gelöscht wurden und ob es zu Fehlern kam. Setze Alarme auf `agents_clean_errors_total` > 0, um sofort auf Probleme reagieren zu können.
