## Detailed Content

### What is agents-clean?

agents-clean is a purpose-built utility for purging orphaned agent states, reconciling distributed task queues, and enforcing idempotent cleanup cycles across multi-agent systems. It’s not a generic scheduler—it’s a surgical instrument for environments where agent lifecycle drift is a liability.

### Core Features

1. **Orphaned State Garbage Collector**  
   Automatically detects and removes stale agent contexts from memory and persistent stores. Uses a configurable TTL + heartbeat threshold to avoid false positives, even in high-latency networks.

2. **Idempotent Cleanup Hooks**  
   Each cleanup operation is wrapped in a deterministic hash-lock, preventing double-execution across retries or parallel agents. No more “already cleaned” race conditions.

3. **Distributed Queue Reconciliation**  
   Scans for stuck or phantom tasks in Redis, RabbitMQ, or Kafka backends, then either re-queues or dead-letters them based on custom rules. Works without disrupting active consumers.

### How to Use

1. **Bootstrap with a Config File**  
   Drop a `agents-clean.yml` into your project root. Define your agent types, cleanup triggers, and backend connections. Example snippet:
   ```yaml
   agents:
     - name: worker-pool
       backend: redis
       ttl: 300
       cleanup_hook: "POST_CLEANUP"
   ```
   Run `agents-clean init` to validate the config and generate a lockfile.

2. **Execute a Dry-Run Audit**  
   Before any destructive operation, run `agents-clean audit --dry-run`. This outputs a detailed report of all orphaned states and stuck tasks—no side effects. Review the JSON output and confirm the scope:
   ```bash
   agents-clean audit --dry-run --format json > audit_report.json
   ```

3. **Run a Targeted Cleanup Cycle**  
   Once verified, execute a cleanup with explicit scope:
   ```bash
   agents-clean purge --scope worker-pool --force
   ```
   This triggers the garbage collector for the specified agent pool, respecting the TTL and hook rules. Use `--verbose` to stream real-time logs to stdout.

### Best Practices

- **Always run a dry-run in staging first.**  
   agents-clean is powerful, but a misconfigured TTL can wipe active agents. Mirror your production config in staging, run the audit, and compare the report against your expected state. This catches drift before it hits prod.

- **Pair cleanup hooks with a monitoring alert.**  
   Set up a webhook or log sink for `POST_CLEANUP` events. When agents-clean detects an unusually high number of orphans, trigger an alert. This often points to a deeper issue—like a failing agent or a misbehaving scheduler.
