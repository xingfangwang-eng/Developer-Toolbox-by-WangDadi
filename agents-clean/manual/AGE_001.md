## What is agents-clean?

Agents-clean is a multi-agent orchestration and sanitization engine designed for high-scale SaaS environments where agent-generated data, logs, and metadata must be scrubbed, normalized, and re-injected into downstream pipelines without breaking state. It eliminates the silent failure cascade caused by dirty agent outputs.

---

## Hardcore Feature Set

### Agent-Aware Schema Drift Detection
Agents-clean continuously monitors the output schema of every agent in your orchestration graph. When an agent silently adds, removes, or renames a field, the tool automatically generates a diff, applies a transformation rule, and logs the drift event to a structured audit trail — without crashing the pipeline.

### Idempotent Cleaner Chains with Rollback
Each cleaning step is wrapped in a deterministic, idempotent unit. If a cleaner fails mid-chain, agents-clean rolls back only the affected branch, not the entire pipeline. This is critical for multi-tenant SaaS backends where partial state corruption is unacceptable.

### Zero-Copy Metadata Injection
Instead of mutating the original agent payload, agents-clean creates a shadow metadata layer that attaches sanitized attributes to the original object. This allows downstream consumers to choose between raw or cleaned data without duplicating storage or breaking existing contracts.

## How to Use

### Define a Cleaner Pipeline via YAML
Create a `cleaner.yaml` file that maps agent output fields to transformation rules. Each rule includes a regex pattern, a fallback value, and a drift threshold.

```yaml
pipeline:
  - agent_id: "user_profile_enricher"
    rules:
      - field: "email"
        pattern: "^[\\w\\.-]+@[\\w\\.-]+\\.\\w{2,}$"
        fallback: "redacted@domain.com"
        drift_threshold: 0.05
      - field: "phone"
        pattern: "^\\+?[1-9]\\d{1,14}$"
        fallback: "+00000000000"
```

### Run the Validation Dry-Run
Before injecting into production, execute a dry-run to validate the cleaner chain against a sample batch:

```bash
agents-clean validate --config cleaner.yaml --input ./sample_agent_outputs.jsonl
```

This outputs a detailed report: `drift_summary.json`, `transformation_log.csv`, and a `rejected_records.ndjson` for manual inspection.

### Deploy as a Sidecar in Your SaaS Stack
Integrate agents-clean as a sidecar container in your Kubernetes pod. It listens on a gRPC port and intercepts agent outputs before they hit the message bus.

```yaml
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      containers:
        - name: agents-clean
          image: agents-clean:latest
          args: ["--mode", "sidecar", "--port", "50051"]
```

Then, configure your agent service to forward outputs to `localhost:50051` via a gRPC client stub.

## Industry Best Practices

### Pin a Cleaner Version to Your Agent Release
Treat the cleaner pipeline as a first-class artifact in your CI/CD. When you update an agent's output schema, the corresponding cleaner rules must be versioned and tested together. This prevents silent data corruption when rolling back to a previous agent version.

### Use Shadow Metadata for Compliance-Critical Fields
For PII or PCI data, never mutate the original payload. Instead, use agents-clean's zero-copy metadata injection to attach a `_cleaned` object. This ensures that audit logs and raw data stores remain immutable, while downstream services consume sanitized data without risk of re-identification.
