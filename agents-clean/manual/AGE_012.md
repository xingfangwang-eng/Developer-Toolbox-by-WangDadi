# agents-clean - AGE_012

## Overview

This is a high-performance, multilingual SaaS framework engineered for developers who build and maintain agent-based cleaning pipelines across distributed systems. `agents-clean` eliminates boilerplate, enforces idempotent task execution, and provides real-time observability into agent state transitions. Whether you're orchestrating CI/CD artifact scrubbing, GDPR-compliant data purges, or ephemeral environment teardowns, this tool is your deterministic backbone.

## Core Features

### 1. Declarative Agent Graph with Idempotent Rollback
Define cleaning workflows as a directed acyclic graph (DAG) of agents. Each agent exposes a `clean()` and `rollback()` handler. The runtime automatically snapshots state before execution and triggers rollback on failure—no orphaned resources, no partial cleans. Supports conditional branching and parallel fan-out across agents.

### 2. Context-Aware Resource Tracer (CART)
CART maintains a real-time reference map of all resources (files, containers, API keys, DB entries) created by agents. When a cleaning cycle triggers, CART performs a diff against the last known good state and only targets delta resources. This reduces execution time by up to 60% in large-scale systems and prevents accidental deletion of shared infrastructure.

### 3. Multi-Tenant Policy Engine with Audit Logging
Define per-tenant cleaning policies using YAML or JSON schemas. The engine enforces rate limits, blacklists, and mandatory approval gates for destructive operations. Every action is logged with a traceable AgentID, timestamp, and payload checksum—fully compatible with SOC 2 and GDPR audit trails.

## How to Use

### 1. Bootstrap a Cleaning Pipeline with a DAG Definition
Instantiate a `CleaningPipeline` and register agents with explicit dependencies. Each agent must implement the `Agent` interface with `clean()` and `rollback()` methods. Use `pipeline.add_agent(agent, depends_on=['agent_a', 'agent_b'])` to define execution order. The runtime validates the DAG for cycles and missing dependencies before first execution.

```python
from agents_clean import CleaningPipeline, Agent

class TempFileAgent(Agent):
    def clean(self, context):
        # remove temp files older than 24h
        pass
    def rollback(self, context):
        # restore from snapshot
        pass

pipeline = CleaningPipeline()
pipeline.add_agent(TempFileAgent(), depends_on=[])
pipeline.run()
```

### 2. Configure CART for Selective Resource Scrubbing
Initialize the Context-Aware Resource Tracer with a persistence backend (Redis, PostgreSQL, or in-memory). Attach it to your pipeline using `pipeline.attach_tracer(cart_instance)`. During execution, CART logs every resource creation event and computes the diff set for the next clean cycle. Use `cart.dry_run()` to preview deletions before committing.

```bash
agents-clean --tracer-backend redis --tracer-ttl 3600
```

### 3. Enforce Multi-Tenant Policies via YAML Schemas
Create a `policy.yaml` file that defines tenant-specific rules. Use `pipeline.load_policy('policy.yaml')` to apply constraints. The policy engine validates each agent action against the schema before execution. For destructive operations, set `require_approval: true` and integrate with Slack or PagerDuty for manual confirmation.

```yaml
tenants:
  - id: tenant_a
    blacklist: ["drop_database", "purge_logs"]
    rate_limit: 10
    require_approval: ["delete_s3_bucket"]
```

## Best Practices

- **Always pair CART with a persistent backend in production.** In-memory mode is fine for local testing, but Redis or PostgreSQL ensures state survives restarts and provides a source of truth for rollback decisions. Without persistence, a crash during a multi-agent clean can leave your system in an inconsistent state.
- **Use the DAG validation hook in your CI pipeline.** Run `pipeline.validate()` as a pre-merge check to catch cyclic dependencies or orphaned agents before they hit production. This dramatically reduces runtime failures and simplifies debugging across distributed teams.
