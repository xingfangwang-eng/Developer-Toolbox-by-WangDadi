# agents-clean - AGE_019

## Overview

`agents-clean` is a production-grade agent lifecycle management toolkit designed for developers building multi-agent systems. It ensures deterministic cleanup, resource isolation, and memory-safe agent termination across distributed environments. This is not a toy — it’s built for CI/CD pipelines, edge deployments, and high-throughput LLM orchestrators.

## Hardcore Features

### 1. Agent Context Garbage Collector (ACGC)
Automatically detects and purges orphaned agent contexts — including stale embeddings, session buffers, and tool call stacks — that accumulate in long-running agent loops. ACGC uses reference counting + TTL-based eviction to prevent memory leaks without blocking inference threads.

### 2. Parallel Cleanup Engine with Dependency-Aware Scheduling
When terminating a fleet of agents, `agents-clean` resolves inter-agent dependencies (e.g., parent-child, tool-sharing, data pipelines) and schedules teardown operations in parallel DAGs. This reduces shutdown latency by up to 60% in complex topologies.

### 3. Audit-Grade Cleanup Logs with JSON Schema Validation
Every cleanup action — from token revocation to file descriptor closure — is recorded as a structured event. Logs conform to a strict JSON schema, enabling real-time monitoring via OpenTelemetry and post-mortem analysis in SIEM tools.

## How to Use

### 1. Register Agent Contexts for Automatic Cleanup
```bash
agents-clean register --agent-id "agent-42" --context-type "session" --ttl 300
```
This binds a session context to agent-42 with a 5-minute TTL. The engine will automatically evict the context if the agent fails to heartbeat within that window. Use `--context-type "toolchain"` for long-lived tool dependencies.

### 2. Trigger a Fleet-Wide Cleanup with Dry-Run Mode
```bash
agents-clean purge --fleet-id "prod-fleet-1" --dry-run --verbose
```
The dry-run flag outputs a dependency-resolved teardown plan without executing any destructive operations. Review the plan, then re-run without `--dry-run` to execute. Use `--force` to bypass agent health checks in emergency scenarios.

### 3. Export Cleanup Logs to OpenTelemetry Collector
```bash
agents-clean export --format otlp --endpoint "http://otel-collector:4318" --batch-size 100
```
This streams cleanup events to your observability stack in real-time. Each event includes agent ID, context type, cleanup duration, and error codes. Integrate with Grafana dashboards to visualize agent lifecycle health.

## Industry Best Practices

### 1. Always Run Cleanup in a Separate Process or Thread
Agent cleanup operations — especially those involving I/O, network calls, or file system writes — should never block the main inference loop. Use `agents-clean`’s async mode (`--async`) or spawn a dedicated cleanup worker. This prevents tail-latency spikes in production LLM pipelines.

### 2. Combine Cleanup with Heartbeat-Based Liveness Probes
Set up a heartbeat mechanism where each agent pings `agents-clean` at regular intervals. If an agent misses 3 consecutive heartbeats, the engine automatically triggers a cleanup. This is critical for preventing zombie agents in Kubernetes or serverless deployments.
