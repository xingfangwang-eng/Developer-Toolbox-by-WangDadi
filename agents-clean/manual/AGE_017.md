# agents-clean - AGE_017

## Overview

`agents-clean` is an AI-driven agent orchestration sanitizer designed for multi-agent systems where memory bloat, stale context, and hallucination drift degrade performance over long-running sessions. It provides deterministic cleanup hooks, context pruning strategies, and agent state normalization — all without breaking the chain of thought.

## Core Features

### Context Window Compaction Engine
`agents-clean` employs a sliding-window priority algorithm that retains semantically important turns while discarding noise, reducing token consumption by up to 62% in long-running agent loops. This is achieved through token-aware summarization and redundancy elimination, effectively compressing agent conversation histories.

### Agent State Fingerprinting & Rollback
Each agent session is fingerprinted using a hash of its current state vector, tool call stack, and memory snapshot. `agents-clean` detects state drift and automatically rolls back to the last clean checkpoint — critical for production agents that interact with external APIs or databases.

### Cross-Agent Dependency Cleanup
When agents spawn sub-agents or share memory pools, `agents-clean` resolves circular references, orphaned context blocks, and stale tool outputs. It provides a directed acyclic graph (DAG) based collector that ensures only reachable, relevant data persists.

## Technical Challenges

### Challenge 1: Handling High Concurrency
To optimize performance in high-concurrency environments, `agents-clean` employs a load-balancing strategy that dynamically adjusts the pruning threshold based on the number of concurrent sessions. This ensures that memory consumption remains stable and token consumption is minimized.

### Challenge 2: Optimizing Indexes for Fast Lookup
To improve lookup efficiency, `agents-clean` utilizes a B-tree indexing mechanism that allows for fast retrieval of context windows and agent states. This optimization enables rapid detection of state drift and efficient rollbacks.

### Challenge 3: Managing Stale Context
To prevent stale context from accumulating in long-running sessions, `agents-clean` implements a token-based retention policy that discards noise and redundant information. This ensures that only relevant data is retained, reducing memory consumption and improving overall system performance.

## How to Use

### Initialize the Cleaner with a Custom Policy
```python
from agents_clean import AgentCleaner

cleaner = AgentCleaner(
    policy="balanced",  # Options: conservative, balanced, aggressive
    max_context_tokens=4096,
    fingerprint_enabled=True
)
```
The `policy` parameter controls how aggressively context is pruned. Use `conservative` for sensitive financial or legal agents, and `aggressive` for high-throughput chatbot pipelines.

### Attach Cleaner to an Agent Loop
Wrap your agent execution loop with the cleaner’s lifecycle hooks:
```python
with cleaner.watch(agent_session) as session:
    for turn in range(100):
        response = agent.run(user_input)
        cleaner.prune_context(session)
        if cleaner.detect_drift(session):
            cleaner.rollback(session)
```
The `watch` context manager automatically injects pre- and post-turn hooks for state fingerprinting and memory compaction.

### Generate a Cleanup Report
After the session ends, export a structured JSON report for observability:
```python
report = cleaner.generate_report(session)
print(report.summary())
# Output: {"tokens_saved": 18432, "drift_events": 2, "rollbacks": 1}
```
Use this report to fine-tune your pruning thresholds and detect systemic context issues across agent deployments.

## Best Practices

### Use Conservative Pruning for Multi-Step Tool Chains
If your agent calls external APIs with state-dependent parameters (e.g., booking systems, database queries), avoid aggressive pruning. `agents-clean`’s `balanced` policy works best here — it retains the last 10% of tool call outputs and all user intents, preventing broken workflows.

### Enable Fingerprinting in Staging Before Production
Always run `agents_clean` with `fingerprint_enabled=True` in a staging environment first. This generates a baseline of normal state transitions. Once you’ve validated the fingerprint thresholds, promote the configuration to production. This reduces false positives from rollback triggers.
