# agents-clean - AGE_021

## Overview

agents-clean is a production-grade orchestration toolkit designed to sanitize, deduplicate, and normalize multi-agent output streams in real time. Built for SaaS engineers managing high-throughput LLM pipelines, it eliminates hallucination cascades, token waste, and cross-agent data contamination without sacrificing latency. This is not a wrapper—it’s a deterministic post-processing layer that plugs directly into your agent graph.

## Core Features

### 1. **Semantic Deconfliction Engine**
Agents-clean runs a lightweight, offline embedding comparator that detects semantic overlap across agent outputs before they hit your downstream store. It uses cosine similarity thresholds you define per pipeline stage, then merges or discards duplicates based on recency and source authority. No more redundant context windows or bloated memory stores.

### 2. **Token-Aware Output Sanitizer**
Each agent output is passed through a regex-free, AST-based scrubber that strips PII, hallucinated identifiers, and malformed JSON in a single pass. The sanitizer respects nested structures and preserves valid schema fragments, making it ideal for tools generating structured logs, API responses, or markdown blocks. It runs in under 2ms per 4K tokens on commodity hardware.

### 3. **Cross-Agent Consistency Validator**
This feature enforces a shared schema contract across all agents in your pipeline. It validates field types, required keys, and value ranges against a user-defined YAML spec. If an agent returns a field that violates the contract (e.g., a string in an integer field), the validator auto-corrects or drops the record with a structured error log. This prevents downstream crashes in data lakes or analytics dashboards.

## How to Use

### 1. **Inject the Cleaner into Your Agent Graph**
Add `agents-clean` as a middleware node in your orchestration framework (LangGraph, CrewAI, or custom DAG). Configure the `sanitize` and `deduplicate` flags in the YAML config:

```yaml
cleaner:
  enabled: true
  dedup_threshold: 0.87
  schema_validation: strict
```

Then run your pipeline normally. The cleaner intercepts outputs at the aggregation point, not inside each agent’s loop, so you avoid per-agent overhead.

### 2. **Define a Schema Contract for Validation**
Create a `contract.yaml` file that mirrors your expected output structure. For example, if you’re processing agent-generated customer support tickets:

```yaml
fields:
  ticket_id:
    type: string
    required: true
    pattern: "^TKT-\d{6}$"
  priority:
    type: integer
    required: false
    range: [1, 5]
  summary:
    type: string
    max_length: 512
```

Pass this file to the validator via the `--contract` flag. Any agent output that fails validation is logged with the exact field and violation type.

### 3. **Run Batch or Streaming Cleanup**
For batch jobs, pipe your agent outputs into the cleaner:

```bash
cat agent_outputs.jsonl | agents-clean --config config.yaml --contract contract.yaml --output clean_outputs.jsonl
```

For real-time streams, use the WebSocket endpoint:

```python
import websockets
async with websockets.connect("ws://localhost:8765") as ws:
    await ws.send(json.dumps({"agent_output": {...}}))
    clean = await ws.recv()
```

The streaming mode adds less than 15ms latency per message on average.

## Best Practices

- **Validate early, deduplicate late.** Run schema validation immediately after each agent’s output is emitted, but defer deduplication until you have aggregated outputs from all agents in a stage. This preserves the semantic context needed for accurate overlap detection and avoids discarding valid data prematurely.

- **Use token-aware sanitization before logging.** Many teams log raw agent outputs for debugging, but this leaks PII and inflates storage costs. Apply the sanitizer before writing to any log sink or data warehouse. This also ensures compliance with GDPR and SOC2 without requiring a separate redaction service.
