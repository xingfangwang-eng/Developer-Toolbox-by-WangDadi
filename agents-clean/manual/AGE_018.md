# agents-clean - AGE_018

## Overview

`agents-clean` is a high-performance agent lifecycle management toolkit designed for production-grade multi-agent systems. It provides deterministic cleanup routines, memory leak detection, and cross-runtime resource deallocation for LLM-based agent orchestrations. Built for engineers who demand zero-downtime agent recycling and reproducible state resets.

## What is agents-clean?

`agents-clean` solves the critical problem of **agent state pollution** in long-running conversational AI pipelines. It automatically purges stale context windows, terminates orphaned sub-agents, and reinitializes tool registries without restarting the parent process. Designed for Kubernetes-native deployments and serverless agent farms.

## Core Features

- **Deterministic Agent Tear-Down with DAG Validation**: Executes a topological-sorted teardown of all dependent agents, ensuring no dangling references or unclosed DB connections. Supports custom `on_exit` hooks for legacy tool integrations.

- **Real-Time Memory Hotspot Analysis**: Uses eBPF-based probes to detect agent memory leaks in production. Automatically triggers GC compaction and agent cloning when heap fragmentation exceeds 15% of the allocated pool.

- **Zero-Cost Snapshot Rollback**: Atomically serializes agent state to a distributed KV store (Redis/RocksDB) every 50ms. Enables instant rollback to any previous checkpoint with zero data loss, even across pod restarts.

## How to Use

### 1. Initialize the Cleaner with a Custom Namespace

```bash
agents-clean init --namespace=prod-agent-v2 --policy=strict --gc-interval=500ms
```
This creates a scoped cleanup context. The `strict` policy enforces that any agent not explicitly registered in the DAG manifest will be force-terminated. Use `--gc-interval` to match your agent’s heartbeat frequency.

### 2. Register Agents and Dependencies

```python
from agents_clean import AgentRegistry, CleanScheduler

registry = AgentRegistry(namespace="prod-agent-v2")
registry.register(
    agent_id="translator-v3",
    dependencies=["tokenizer-v2", "cache-proxy"],
    lifecycle_hooks={"on_exit": "flush_buffers"}
)
CleanScheduler.schedule(registry, mode="parallel")
```
The scheduler validates the dependency graph before execution. If a circular dependency is detected (e.g., agent A depends on B, B depends on A), the scheduler raises a `DAGCycleError` and aborts the cleanup.

### 3. Trigger a Full Clean with Health Check

```bash
agents-clean run --namespace=prod-agent-v2 --force --health-check
```
The `--health-check` flag pings all registered agents before termination. If an agent is unresponsive for more than 3 seconds, the cleaner logs a `CRITICAL` event and attempts a SIGKILL via the orchestration layer. After cleanup, a summary JSON is written to `/var/log/agents-clean/`.

## Best Practices

### 1. Pair with Canary Deployments for Agent Versioning
Always run `agents-clean` on a canary agent pool before rolling out to production. Use the `--dry-run` flag to simulate cleanup and inspect the generated teardown plan. This prevents accidental termination of stateful agents that are mid-conversation.

### 2. Integrate with OpenTelemetry for Traceable Cleanups
Emit custom spans from `agents-clean` to your observability stack. Tag each cleanup event with the agent ID, namespace, and GC duration. This allows you to correlate agent memory leaks with specific LLM prompt patterns and optimize your context window strategy.
