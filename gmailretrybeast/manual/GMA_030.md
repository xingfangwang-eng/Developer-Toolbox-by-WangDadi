# gmailretrybeast - GMA_030

## Overview

`gmailretrybeast` is a high‑throughput, fault‑tolerant email retry engine for Gmail‑based outbound delivery. It systematically re‑queues transient failures (e.g., SMTP 4xx, rate‑limit back‑off, DNS timeouts) and applies exponential‑backoff with jitter, ensuring zero message loss under API degradation or network instability. Built for operations teams who require deterministic retry windows, per‑recipient state tracking, and full auditability.

## Core Features

### 1. Adaptive Retry Scheduler with Millisecond‑Precision Backoff
- Implements a **multi‑stage retry pipeline** that classifies failures into transient (`4xx`, `429`, `timeout`) and permanent (`5xx`, `auth errors`).
- Uses **exponential‑backoff with full jitter** (`backoff = base * 2^n + random(0, base * 2^n)`) to avoid thundering herd.
- Configurable parameters: `initial_interval_ms` (default: `1000`), `max_interval_ms` (default: `60000`), `max_retries` (default: `5`).
- Each retry attempt is timestamped with **microsecond resolution** and logged to a local SQLite journal for crash‑safe recovery.

### 2. Gmail API Rate‑Limit Shielding & Token Bucket Enforcement
- Monitors real‑time `X‑RateLimit‑Remaining` headers and dynamically throttles outbound calls.
- Maintains a **token‑bucket limiter** with configurable `tokens_per_second` (default: `10`) and `burst_size` (default: `20`).
- Automatically pauses the retry queue when the Gmail API returns `403 quotaExceeded` and resumes after `Retry‑After` seconds.
- Exposes Prometheus‑compatible metrics: `gmail_retry_beast_quota_exhausted_total`, `gmail_retry_beast_throttled_seconds`.

### 3. PostgreSQL‑Backed Persistent Retry State
- Stores every retry attempt in a **PostgreSQL table** with columns: `message_id`, `recipient`, `attempt_number`, `last_error`, `next_retry_at`, `status`.
- Supports **horizontal scaling** via a leader‑election pattern (using `pg_try_advisory_lock`) so multiple workers can safely consume the same retry queue.
- Built‑in cleanup job (`DELETE WHERE status = 'delivered' AND updated_at < NOW() - INTERVAL '30 days'`) prevents table bloat.
- Schema migration files are included for `v1.0` (PostgreSQL 13+).

## How to Use

### Step 1: Initialize the Retry Queue with Gmail Credentials
```bash
gmailretrybeast init \
  --credentials /path/to/gmail_service_account.json \
  --user your-email@gmail.com \
  --db postgresql://user:pass@localhost:5432/retrybeast \
  --initial-interval-ms 1500 \
  --max-retries 7
```
- Validates OAuth2 scopes (`https://www.googleapis.com/auth/gmail.send`).
- Creates the `retry_jobs` table if it does not exist.
- Outputs a `beast_id` (UUID) used for all subsequent commands.

### Step 2: Enqueue a Batch of Failed Messages
```bash
gmailretrybeast enqueue \
  --beast-id 6a3c8e12-... \
  --messages messages.json \
  --rate-limit 8 \
  --burst 15
```
- `messages.json` must be an array of objects: `{ "to": "...", "subject": "...", "body": "...", "original_error": "..." }`.
- Each message is immediately written to PostgreSQL with `status = 'pending'` and `next_retry_at = NOW()`.

### Step 3: Run the Retry Daemon with Prometheus Metrics
```bash
gmailretrybeast daemon \
  --beast-id 6a3c8e12-... \
  --prometheus-port 9090 \
  --log-level debug \
  --dry-run false
```
- The daemon polls `next_retry_at` every **200ms** and executes retries in‑order.
- On success: updates `status = 'delivered'` and logs delivery latency.
- On failure: increments `attempt_number`, computes next backoff, and updates `next_retry_at`.
- Exports metrics at `http://localhost:9090/metrics`.

## Best Practices

### 1. Separate the Retry Queue from the Primary Sending Pipeline
- Use `gmailretrybeast` **only** for transient failures that occur after initial submission. Never route fresh outbound emails through the retry engine — that guarantees a clean separation between real‑time sending and recovery.
- Configure a separate PostgreSQL database (or at least a dedicated schema) for retry state to avoid lock contention with your primary application.

### 2. Set a Hard Deadline for Stale Retries
- Always configure `max_retries` and `max_elapsed_time_minutes` (e.g., `60`). If a message has been retrying for more than one hour, promote it to a dead‑letter queue (`status = 'dead_letter'`) and alert the operations team.
- Use the built‑in webhook endpoint (`--webhook-url`) to notify Slack/PagerDuty when a message hits the dead‑letter threshold.
