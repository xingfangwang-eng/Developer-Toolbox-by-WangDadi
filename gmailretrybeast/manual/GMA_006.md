# gmailretrybeast - GMA_006

## Overview

`gmailretrybeast` is a high-availability email resilience engine engineered to intercept, classify, and replay failed Gmail SMTP transactions with sub-second precision. It eliminates silent delivery drops and transient SMTP 4xx/5xx errors through a deterministic retry scheduler backed by PostgreSQL and in-memory bloom filters.

## Core Features

### 1. Adaptive Retry Backoff with Jitter Windowing
Implements a multi-stage retry algorithm that respects Gmail’s `Retry-After` header and applies exponential backoff capped at 300 seconds. Each retry attempt is logged with millisecond-resolution timestamps and includes configurable jitter (default: ±15%) to prevent thundering herd on the upstream MX.  
**Parameters:** `--max-retries=12`, `--base-delay-ms=2000`, `--jitter-pct=15`, `--cap-seconds=300`

### 2. SMTP Response Classifier & Intelligent Bounce Router
Parses raw SMTP response codes (e.g., `450 4.7.1`, `550 5.1.1`) and classifies them into three buckets: **Transient (retry)**, **Permanent (quarantine)**, and **Auth/Config (alert)**. Quarantined messages are written to a PostgreSQL `bounce_archive` table with full header and body snapshots for forensic auditing.  
**Storage:** `bounce_archive` table with `message_id`, `smtp_code`, `first_seen_ms`, `retry_count`, and `raw_response` columns.

### 3. Idempotent Delivery with Dedup Hash Ring
Each outbound email is assigned a SHA-256 fingerprint based on `(from, to, subject, body_hash, timestamp_bucket)`. Before any delivery attempt, the retry engine checks an in-memory Redis-backed bloom filter (false positive rate: 0.001) to prevent duplicate dispatch within a configurable dedup window (default: 3600 seconds).  
**Parameters:** `--dedup-window=3600`, `--bloom-capacity=1000000`, `--bloom-fpr=0.001`

## How to Use

### Step 1: Initialize the Retry Queue Database
```bash
gmailretrybeast init --db-host=localhost --db-port=5432 --db-name=gmail_retry --db-user=beast_user
```
This creates the `retry_queue`, `bounce_archive`, and `delivery_log` tables with proper indexes on `retry_at` and `message_id`. The schema includes a composite index on `(status, next_retry_ms)` for O(log n) queue scans.

### Step 2: Inject Failed Messages into the Queue
```bash
gmailretrybeast enqueue --input=./failed_emails.ndjson --batch-size=500 --worker-threads=8
```
Each record in the NDJSON file must contain `from`, `to`, `subject`, `body`, and `original_smtp_code`. The tool validates the schema, computes the dedup hash, and inserts into `retry_queue` with `initial_retry_ms = now() + base_delay_ms`.

### Step 3: Run the Retry Daemon with Real-Time Metrics
```bash
gmailretrybeast daemon --smtp-host=smtp.gmail.com --smtp-port=587 --tls=true \
  --auth-user=your@email.com --auth-pass=app_password \
  --metrics-port=9090 --log-level=info
```
The daemon polls the `retry_queue` every 500ms, picks up messages where `next_retry_ms <= now()`, attempts delivery via SMTP, and updates the record with the new SMTP response code. Prometheus metrics are exposed at `/metrics` on port 9090, including `retry_attempts_total`, `retry_success_total`, and `queue_depth`.

## Best Practices

### Practice 1: Always Pin the Dedup Window to Gmail’s Internal Rate Limit
Gmail applies a per-account sending limit (e.g., 500 recipients per day for free tiers). Set `--dedup-window` to at least 3600 seconds and align your batch size with Gmail’s `max_recipients_per_message` (default 100). This prevents your retry beast from generating duplicate traffic that triggers a temporary ban.

### Practice 2: Offload Bounce Analysis to a Separate PostgreSQL Read Replica
The `bounce_archive` table can grow quickly under heavy retry loads. Route all analytical queries (e.g., “which SMTP codes are most common this hour?”) to a read replica with a dedicated index on `(smtp_code, first_seen_ms)`. Keep the primary database focused on high-throughput `retry_queue` writes and updates.
