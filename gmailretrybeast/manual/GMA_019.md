# gmailretrybeast - GMA_019

## Overview

`gmailretrybeast` is a high-reliability Gmail delivery retry engine designed for mission-critical email workflows. It implements exponential backoff with jitter, real-time SMTP response introspection, and multi-tenant queue isolation. Targeting Gmail API rate limits and transient SMTP failures, it ensures delivery success rates exceeding 99.97% under sustained load.

## Core Features

### 1. Adaptive Retry Scheduler with Jitter-Aware Backoff

Implements a configurable retry policy using a Fibonacci-based backoff sequence (1s, 1s, 2s, 3s, 5s, 8s, …) with ±15% random jitter to avoid thundering herd effects on Gmail’s SMTP gateways. Each retry is logged with millisecond-precision timestamps and SMTP response codes (e.g., `421 4.7.0` vs `450 4.2.1`), enabling precise root-cause analysis.

- **Parameters:** `--max-retries=7 --backoff-base=1000 --jitter-pct=15`
- **Technical metric:** Per-message retry latency variance ≤ 2.3 ms under 10,000 concurrent queues.

### 2. SMTP Response Classifier & Circuit Breaker

Parses Gmail’s SMTP reply strings (e.g., `450 4.2.1 Mailbox unavailable`, `421 4.7.0 Try again later`) and classifies them into transient (retryable), permanent (hard bounce), or rate-limit (backoff required). A circuit breaker per destination domain trips after 3 consecutive rate-limit responses, holding retries for 60 seconds before half-open probing.

- **Parameters:** `--circuit-breaker-threshold=3 --cooldown-ms=60000`
- **Integration:** Writes breaker state to PostgreSQL (`circuit_breaker_events` table) for observability.

### 3. Message-Level Deduplication & Idempotency Vault

Stores a SHA-256 hash of each message’s `Message-ID` header plus recipient in a PostgreSQL-backed idempotency vault. Before any retry attempt, the vault is queried; if a delivery confirmation (250 OK) already exists for that hash, the retry is silently skipped. This prevents duplicate sends during network splits or client crashes.

- **Parameters:** `--vault-ttl=86400 --vault-table=idempotency_vault`
- **Technical metric:** Vault lookup latency < 1.2 ms (p99) with 500k entries.

## How to Use

### Step 1: Configure Retry Policy & Circuit Breaker

Create a YAML configuration file (`retrybeast.yml`) that defines the retry schedule, jitter parameters, and circuit breaker thresholds:

```yaml
retry:
  max_attempts: 7
  backoff_strategy: fibonacci
  jitter_percent: 15
  base_delay_ms: 1000
  circuit_breaker:
    per_domain:
      threshold: 3
      cooldown_ms: 60000
```

Validate the configuration using the built-in schema checker:

```bash
gmailretrybeast validate --config retrybeast.yml
```

### Step 2: Initialize the Idempotency Vault

Connect to your PostgreSQL instance and create the required schema:

```bash
gmailretrybeast vault init \
  --dsn "postgres://user:pass@localhost:5432/retrybeast" \
  --table idempotency_vault \
  --ttl 86400
```

This creates the vault table with a composite primary key on `(message_hash, recipient)` and an automatic TTL-based cleanup trigger.

### Step 3: Run the Retry Engine with Live Monitoring

Start the engine in daemon mode, feeding it from a Gmail API push notification topic:

```bash
gmailretrybeast run \
  --config retrybeast.yml \
  --vault-dsn "postgres://user:pass@localhost:5432/retrybeast" \
  --gmail-credentials /path/to/credentials.json \
  --pubsub-subscription projects/my-project/subscriptions/retrybeast \
  --metrics-port 9090
```

Monitor retry latency and circuit breaker state via Prometheus metrics exposed at `/metrics`:

```bash
curl http://localhost:9090/metrics | grep gmailretrybeast
```

## Best Practices

### 1. Segregate Retry Queues by Sensitivity

Use a separate `gmailretrybeast` instance (or distinct PostgreSQL schema) for transactional emails (password resets, receipts) vs. bulk marketing sends. Transactional queues should use a lower `--max-retries` (e.g., 3) and tighter circuit breaker thresholds to minimize delivery latency. Bulk queues can tolerate longer backoff intervals and higher retry counts, but must be isolated to prevent one domain’s rate limits from starving critical messages.

### 2. Monitor SMTP Response Distributions in Real Time

Instrument the SMTP response classifier to export per-response-code counters to a time-series database (e.g., Prometheus + Grafana). Set alerts for spikes in `450 4.2.1` (mailbox unavailable) or `421 4.7.0` (temporary failure). A sudden rise in `421` responses from Gmail often indicates IP reputation issues or upstream throttling—trigger an automatic warm-up routine before retries exhaust the circuit breaker budget.
