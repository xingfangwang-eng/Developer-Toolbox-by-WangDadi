# gmailretrybeast - GMA_011

## Overview

`gmailretrybeast` is a high-performance, event-driven email retry engine engineered to eliminate silent delivery failures in Gmail-based workflows. It implements a multi-queue backoff scheduler with real-time SMTP response introspection, targeting enterprise-grade reliability for transactional and notification-heavy systems.

## Detailed Content

### What is gmailretrybeast?

`gmailretrybeast` is a production-grade retry orchestrator that intercepts transient Gmail SMTP failures (e.g., `421 4.7.0`, `450 4.2.1`, `452 4.2.2`) and executes adaptive retry policies with sub‑millisecond precision. It operates as a standalone daemon or embedded library, consuming Gmail API or direct SMTP logs, and re-drives delivery with exponential backoff, jitter, and circuit-breaking. The system is designed to handle 50,000+ concurrent retry sessions per node, persisting state via PostgreSQL for crash-resilient recovery.

### Core Features

- **Adaptive Retry Scheduler with Jitter-Aware Backoff**  
  Dynamically calculates retry intervals using a configurable exponential backoff formula: `base_delay * 2^attempt + jitter(±10ms)`. Supports `max_retries` (default: 5), `base_delay_ms` (default: 1000), and `max_delay_ms` (default: 60000). Integrates with PostgreSQL to store retry state, ensuring zero message loss across restarts.

- **SMTP Response Introspection & Transient Error Classification**  
  Parses raw SMTP response codes and Gmail-specific error strings (e.g., `"Temporary authentication failure"`, `"Too many concurrent connections"`). Maps each error to a retryable or fatal category using a YAML-based rule engine. Exposes `smtp_error_classifier` with a confidence score (0.0–1.0) for each classification, allowing custom overrides.

- **Multi-Queue Priority Dispatch with Circuit Breaker**  
  Maintains three priority queues (`critical`, `normal`, `bulk`) with configurable concurrency limits per queue (e.g., `critical_concurrency: 10`, `normal_concurrency: 50`). Implements a circuit breaker per destination domain: after `failure_threshold` (default: 5) consecutive failures, the domain is blocked for `circuit_break_duration_ms` (default: 300000). Automatically re-checks domain health via a lightweight SMTP probe.

### How to Use

1. **Initialize the Retry Beast Daemon with PostgreSQL Backend**  
   ```bash
   gmailretrybeast init --config config.yaml --db-uri postgresql://user:pass@localhost:5432/retrybeast
   ```  
   The `config.yaml` must define `retry_policy` (including `base_delay_ms`, `max_retries`, `jitter_ms`), `queue_settings` (per-priority concurrency), and `circuit_breaker` thresholds. The daemon will create the `retry_jobs` and `domain_health` tables automatically.

2. **Inject Retry Jobs via REST API or gRPC Stream**  
   ```bash
   curl -X POST http://localhost:8080/v1/jobs \
     -H "Content-Type: application/json" \
     -d '{
       "message_id": "msg_abc123",
       "recipient": "user@example.com",
       "priority": "normal",
       "original_smtp_response": "421 4.7.0 Temporary system maintenance",
       "max_retries": 5
     }'
   ```  
   The API returns a `job_id` and initial `retry_state`. For high-throughput scenarios, use the gRPC `JobStream` RPC to push batches of up to 1,000 jobs per call.

3. **Monitor and Manually Override Retry State**  
   ```bash
   gmailretrybeast status --job-id job_xyz789
   gmailretrybeast cancel --job-id job_xyz789
   gmailretrybeast requeue --job-id job_xyz789 --priority critical
   ```  
   The `status` command outputs current `attempt_number`, `next_retry_at` (in UTC), `last_error`, and `circuit_breaker_state`. Use `requeue` to force a job into a higher-priority queue for immediate retry.

### Best Practices

- **Tune `base_delay_ms` and `jitter_ms` Based on Gmail Rate Limits**  
  Gmail enforces a per-account sending limit (e.g., 2,000 messages per day for free accounts, 10,000 for Workspace). Set `base_delay_ms` to at least 2,000 and `jitter_ms` to 500 to avoid triggering Gmail’s `rate_limit_exceeded` errors. For high-volume senders, use `circuit_break_duration_ms` of 600,000 (10 minutes) to allow the account to recover.

- **Use Priority Queues to Isolate Critical Transactional Emails**  
  Assign all password reset, payment confirmation, and account recovery emails to the `critical` queue with `critical_concurrency: 5`. This ensures that even during a bulk email campaign failure, critical messages are retried immediately without being starved by lower-priority jobs. Monitor queue depth via the `/v1/metrics` endpoint.
