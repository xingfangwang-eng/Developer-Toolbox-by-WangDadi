# gmailretrybeast - GMA_003

## Overview

gmailretrybeast is a high-reliability, multi-threaded Gmail message delivery engine designed to systematically recover failed email transmissions. It implements an exponential backoff retry policy with granular per-message state tracking, ensuring that transient Gmail API quota violations, SMTP timeouts, and authentication token expiry do not result in permanent message loss. The system is optimized for high-volume transactional email environments where delivery guarantee is non-negotiable.

## Core Features

### 1. Adaptive Quota-Aware Retry Scheduler with Token Bucket Throttling

The retry beast does not blindly retry. It reads live Gmail API usage metrics (`quotaExceeded`, `rateLimitExceeded`, `dailyLimitExceeded`) and adjusts its retry cadence dynamically. Each retry attempt is governed by a **token bucket algorithm** with configurable `bucketCapacity` (default: 100) and `refillRate` (default: 10 tokens/sec). The scheduler calculates the exact **millisecond-level delay** between retries using the formula:

```
delay = min(maxDelay, baseDelay * (2 ^ attemptCount)) + jitter(0, 500ms)
```

This prevents cascading quota failures while maintaining maximum throughput.

- **Parameter**: `--max-retries` (integer, default 5)  
- **Parameter**: `--base-delay-ms` (integer, default 1000)  
- **Parameter**: `--max-delay-ms` (integer, default 60000)  
- **Parameter**: `--jitter-max-ms` (integer, default 500)  

### 2. Persistent State Machine with PostgreSQL Checkpoint Recovery

Every email in the retry pipeline is tracked via a **finite state machine** (`PENDING → RETRYING → EXPONENTIAL_BACKOFF → DEAD_LETTER`). The state is persisted in a **PostgreSQL** table (`retrybeast.email_states`) with columns: `message_id`, `recipient`, `retry_count`, `last_attempt_timestamp`, `next_attempt_timestamp`, `error_code`, and `error_message`. On process restart, the beast reconstructs its in-memory retry queue from the database, ensuring zero message loss even after a crash.

- **Parameter**: `--db-connection-string` (PostgreSQL URI)  
- **Parameter**: `--checkpoint-interval-sec` (integer, default 30)  

### 3. Multi-Threaded Concurrent Delivery with Backpressure Control

The engine spawns a configurable number of **worker goroutines** (default: 16) that pull messages from a priority queue sorted by `next_attempt_timestamp`. Each worker authenticates via **OAuth 2.0** with a service account (`--service-account-json`). The system implements **backpressure** via a buffered channel of size `--max-inflight-messages` (default: 256). If the Gmail API returns a `429 Too Many Requests`, the worker releases its token and goes into a **cool-down** period proportional to the `Retry-After` header value.

- **Parameter**: `--worker-count` (integer, default 16)  
- **Parameter**: `--max-inflight-messages` (integer, default 256)  
- **Parameter**: `--oauth-scopes` (string, default: `https://www.googleapis.com/auth/gmail.send`)  

## How to Use

### Step 1: Initialize the PostgreSQL Schema and Configure Authentication

```bash
gmailretrybeast init-db \
  --db-connection-string "postgres://user:pass@localhost:5432/retrybeast?sslmode=disable" \
  --create-tables
```

Then generate a Gmail service account JSON key via the Google Cloud Console and store it securely. The beast will read it at runtime:

```bash
export GOOGLE_APPLICATION_CREDENTIALS="/etc/gmailretrybeast/service-account.json"
```

### Step 2: Inject Messages into the Retry Queue via the API Endpoint

Start the beast in daemon mode with a REST API listener:

```bash
gmailretrybeast daemon \
  --api-listen ":8080" \
  --db-connection-string "postgres://user:pass@localhost:5432/retrybeast" \
  --worker-count 32 \
  --max-retries 7 \
  --base-delay-ms 2000
```

Inject a message with a `POST /v1/messages` request:

```json
{
  "to": "user@example.com",
  "subject": "Payment Confirmation",
  "body": "Your payment of $49.99 was successful.",
  "priority": 1,
  "max_retries": 5
}
```

The beast returns a `message_id` (UUID v4) immediately and begins processing asynchronously.

### Step 3: Monitor Dead Letter Queue and Replay Failed Messages

Query the `retrybeast.dead_letter` table for messages that exhausted all retries:

```sql
SELECT message_id, error_code, error_message, retry_count
FROM retrybeast.dead_letter
WHERE created_at > NOW() - INTERVAL '24 hours';
```

To replay a specific dead letter, use:

```bash
gmailretrybeast replay \
  --message-id "a1b2c3d4-e5f6-7890-abcd-ef1234567890" \
  --reset-retry-count
```

This resets the state machine back to `PENDING` and re-queues the message with a fresh retry budget.

## Best Practices

### 1. Implement a Circuit Breaker for Gmail API Project Quotas

Google Cloud imposes a **daily sending limit** per project (e.g., 2,000 messages/day for trial accounts). Configure the beast with `--daily-quota-limit` to automatically pause all retries when the limit is within 10% of exhaustion. This prevents your project from being soft-banned. Use the `gmailretrybeast stats` command to expose real-time quota consumption via Prometheus metrics (`retrybeast_quota_remaining`).

### 2. Separate Retry Queues by Message Criticality

Create two distinct database schemas (`retrybeast_critical` and `retrybeast_bulk`) with different `--max-retries` and `--base-delay-ms` values. Critical transactional emails (password resets, payment receipts) should have `max-retries=7` and `base-delay-ms=500`, while bulk marketing emails should have `max-retries=3` and `base-delay-ms=5000`. This ensures that high-priority messages never get stuck behind a flood of low-priority retries.
