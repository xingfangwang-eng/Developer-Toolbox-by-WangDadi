# gmailretrybeast - GMA_007

## Overview

gmailretrybeast is a high‑performance, resilient email delivery engine purpose‑built for Gmail‑based transactional flows. It intercepts transient SMTP failures, throttles retry backoff with sub‑millisecond precision, and maintains full delivery audit trails in PostgreSQL. Designed for systems that cannot tolerate silent message drops.

## Core Features

### 1. Adaptive Retry Scheduler with Jitter‑Aware Backoff
- Implements exponential backoff with configurable base interval (`--base-interval-ms=500`) and maximum cap (`--max-interval-ms=300000`).
- Applies full‑jitter algorithm to prevent thundering herd on Gmail’s SMTP servers.
- Tracks per‑recipient failure counters; after `N` consecutive failures (default `N=5`) the tool escalates to a dead‑letter queue (DLQ) stored in a dedicated PostgreSQL table (`dlq_messages`).
- Real‑time metrics exposed via `/metrics` endpoint (Prometheus format): `retry_attempts_total`, `retry_success_duration_ms`, `dlq_ingress_total`.

### 2. Gmail API‑Native Auth & Quota Guard
- Uses OAuth 2.0 with Gmail API scopes (`https://www.googleapis.com/auth/gmail.send`).
- Automatically detects daily sending quota via `users.settings.getSendAs` and pauses scheduling when remaining quota drops below a configurable threshold (`--quota-warning-threshold=0.15`).
- Supports multiple Gmail accounts in a single run via `--accounts-file=accounts.json`; each account runs its own independent retry loop.
- All API calls are rate‑limited to 250 QPS (Gmail API limit) with a token‑bucket implementation (`--bucket-capacity=250`, `--bucket-refill-rate=250`).

### 3. Full Delivery Audit Trail & Replay Capability
- Every send attempt (success or failure) is logged to a `delivery_audit` table with columns: `message_id`, `recipient`, `attempt_number`, `timestamp_ms`, `smtp_response_code`, `error_detail`.
- Supports replay of any failed message from the audit log using `--replay-from-audit --message-id=<UUID>`.
- Built‑in integrity check: `--verify-delivery` re‑queries Gmail’s `users.messages.get` after a successful send to confirm the message actually landed in the sent folder. Mismatches trigger an alert.

## How to Use

### Step 1: Configure Gmail API Credentials & Quota
```bash
# 1. Enable Gmail API in Google Cloud Console.
# 2. Download OAuth 2.0 client secrets as `credentials.json`.
# 3. Authenticate and generate a refresh token:
gmailretrybeast auth --credentials-file=credentials.json --output-token=token.json
# 4. (Optional) Pre‑fetch daily quota for all accounts:
gmailretrybeast quota-check --accounts-file=accounts.json
```

### Step 2: Prepare Message Payload & Retry Policy
```bash
# Messages can be provided as a JSON array (one object per email).
# Example: `messages.json`
# [
#   { "to": "user@example.com", "subject": "Invoice #123", "body": "..." },
#   { "to": "admin@example.com", "subject": "Alert", "body": "..." }
# ]
#
# Override default retry policy via CLI:
gmailretrybeast send \
  --messages-file=messages.json \
  --token-file=token.json \
  --base-interval-ms=1000 \
  --max-interval-ms=120000 \
  --max-retries=7 \
  --quota-warning-threshold=0.20
```

### Step 3: Monitor Delivery & Recover Failures
```bash
# Watch live retry progress (stdout updates every 2 seconds):
gmailretrybeast monitor --status-live

# After run completes, inspect delivery audit:
gmailretrybeast audit --limit=50 --status=failed

# Replay a specific failed message:
gmailretrybeast replay \
  --message-id="a1b2c3d4-e5f6-7890-abcd-ef1234567890" \
  --token-file=token.json \
  --base-interval-ms=500
```

## Best Practices

- **Always pre‑warm your Gmail quota before bulk sends.**  
  Run `gmailretrybeast quota-check --accounts-file=accounts.json` 24 hours in advance. If any account shows less than 20% remaining, rotate in a fresh account. This avoids mid‑run throttling that wastes retry attempts.

- **Set up a dead‑letter alerting pipeline.**  
  Configure your monitoring system to watch the `dlq_messages` table row count. When it exceeds a threshold (e.g., 100 rows in 10 minutes), trigger a PagerDuty alert. Use `--replay-from-dlq` after fixing the underlying issue (e.g., recipient address typo) to re‑inject those messages.
