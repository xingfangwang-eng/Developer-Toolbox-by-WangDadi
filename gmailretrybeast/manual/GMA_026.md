# gmailretrybeast - GMA_026

## Overview

`gmailretrybeast` is a high-performance email retry engine built specifically for Gmail API transient failures. It implements exponential backoff with jitter, per-thread credential rotation, and intelligent message queue draining to ensure zero message loss during SMTP outages or daily Gmail quota resets.

## Core Features

### 1. Adaptive Backoff Scheduler with Jitter Injection
- Implements truncated exponential backoff with a configurable `base_delay_ms` (default: 1000ms) and `max_delay_ms` (default: 300000ms).
- Uses `jitter_factor` (range: 0.0–0.3) to randomize retry intervals, preventing thundering herd on Gmail’s rate limiter.
- Tracks `retry_count` per message and terminates after `max_retries` (default: 7) with a detailed `failure_reason` logged to PostgreSQL.

### 2. Multi-Account Credential Pool & OAuth2 Token Refresh
- Manages a pool of up to 50 Gmail accounts via `credentials.json` array, each with independent `token_expiry_epoch`.
- Automatically refreshes expired OAuth2 tokens using `refresh_token` and `client_secret` before each retry attempt.
- Monitors `quota_usage_percent` per account and rotates to the next available account when exceeding 90% daily send quota.

### 3. Dead-Letter Queue (DLQ) with PostgreSQL Persistence
- Every failed message after exhausting retries is serialized into a `dlq_messages` table with columns: `message_id`, `raw_mime`, `failure_stacktrace`, `retry_attempts`, and `created_at` (UTC timestamp with millisecond precision).
- Supports manual replay via `SELECT * FROM dlq_messages WHERE retry_attempts < 3` and re-injection into the main retry pipeline.
- Integrates with `pg_notify` to trigger real-time alerts on DLQ threshold breaches (configurable via `dlq_alert_threshold`, default: 100).

## How to Use

### Step 1: Configure Credential Pool and Retry Parameters
```bash
# Create a JSON file with your Gmail API credentials
cat > /etc/gmailretrybeast/credentials.json <<EOF
[
  {
    "email": "sender1@gmail.com",
    "client_id": "YOUR_CLIENT_ID_1",
    "client_secret": "YOUR_CLIENT_SECRET_1",
    "refresh_token": "YOUR_REFRESH_TOKEN_1"
  },
  {
    "email": "sender2@gmail.com",
    "client_id": "YOUR_CLIENT_ID_2",
    "client_secret": "YOUR_CLIENT_SECRET_2",
    "refresh_token": "YOUR_REFRESH_TOKEN_2"
  }
]
EOF

# Set retry policy via environment variables
export BASE_DELAY_MS=2000
export MAX_DELAY_MS=120000
export MAX_RETRIES=10
export JITTER_FACTOR=0.25
```

### Step 2: Initialize the PostgreSQL Schema and Start the Beast
```bash
# Run schema migration
psql -h localhost -U admin -d gmailretrybeast -f schema/init.sql

# Launch the retry daemon with verbose logging
./gmailretrybeast --credentials /etc/gmailretrybeast/credentials.json \
                  --db-conn "postgresql://admin:password@localhost:5432/gmailretrybeast" \
                  --log-level debug \
                  --dlq-alert-threshold 50
```

### Step 3: Inject Messages and Monitor Retry Metrics
```bash
# Inject a single MIME message for retry processing
echo "From: sender1@gmail.com\nTo: recipient@example.com\nSubject: Test\n\nHello" | \
./gmailretrybeast inject --raw-mime-stdin --priority high

# Monitor real-time retry stats via the built-in HTTP endpoint
curl http://localhost:9090/metrics | grep gmailretrybeast
# Example output:
# gmailretrybeast_retry_count_total{status="success"} 1423
# gmailretrybeast_retry_count_total{status="failed"} 12
# gmailretrybeast_dlq_current_size 8
```

## Best Practices

### 1. Implement Gradual Credential Rotation During High-Volume Campaigns
When sending more than 500 emails per hour, never rely on a single Gmail account. Configure at least 5 accounts in the credential pool and set `quota_rotation_threshold` to 85%. This prevents hitting the 1,500 recipient/day limit per account and ensures uniform distribution of retry load across all accounts.

### 2. Schedule DLQ Replay During Off-Peak Hours Using Cron
Set up a cron job that replays DLQ messages between 02:00–04:00 UTC when Gmail API latency is lowest. Use the `--dlq-replay --batch-size 50` flag to process failed messages in controlled batches, and monitor the `dlq_replay_success_rate` metric. If the success rate drops below 95%, pause the replay and inspect the `failure_stacktrace` column for systemic issues (e.g., expired refresh tokens).
