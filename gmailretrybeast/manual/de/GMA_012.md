# gmailretrybeast - GMA_012

## Übersicht

Dieses Projekt realisiert ein hochperformantes, asynchrones E-Mail-Retry-System für die Gmail API. Es überwacht fehlgeschlagene Sendevorgänge, klassifiziert Fehlertypen und führt exponentielle Backoff-Strategien mit konfigurierbaren Schwellwerten aus. Zielsprache der Systemausgabe und Logs: Deutsch. Optimiert für hohe Durchsatzraten und minimale Latenz bei der Wiederholungsplanung.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist ein spezialisiertes Daemon-Tool zur automatischen Wiederholung von Gmail-API-Aufrufen, die aufgrund temporärer Fehler (z. B. `429 Too Many Requests`, `503 Service Unavailable`, `500 Internal Server Error`) fehlschlagen. Es implementiert einen mehrstufigen Retry-Mechanismus auf Basis von Jitter-basiertem exponentiellen Backoff (Basisintervall: 250 ms, maximales Intervall: 64 s) und persistiert den Zustand jeder fehlgeschlagenen Nachricht in einer PostgreSQL-Datenbank. Die Fehleranalyse erfolgt über einen eingebauten Klassifikator, der HTTP-Statuscodes, gRPC-Error-Details und Gmail‑spezifische Fehlermeldungen (z. B. `rateLimitExceeded`, `backendError`) auswertet.

### Kernfunktionen

1. **Dynamischer, konfigurierbarer Retry-Scheduler mit Prioritätswarteschlange**  
   – Parameter: `--max-retries` (Standard: 5), `--base-delay-ms` (Standard: 250), `--max-delay-ms` (Standard: 64000), `--jitter-factor` (Standard: 0,3).  
   – Jeder Retry-Vorgang wird in einer PostgreSQL-Tabelle (`retry_queue`) mit den Spalten `message_id`, `retry_count`, `next_attempt_at` (Timestamp mit μs‑Genauigkeit), `error_code` und `error_message` gespeichert.  
   – Der Scheduler nutzt einen Heap-basierten Timer (O(log n) pro Insert) und garantiert, dass kein Retry vor Ablauf seines geplanten Zeitfensters ausgeführt wird.

2. **Intelligente Fehlerklassifikation und Eskalationslogik**  
   – Analysiert die Gmail API‑Response auf spezifische Fehlerobjekte: `error.details[0].reason` (z. B. `rateLimitExceeded`, `userRateLimitExceeded`, `quotaExceeded`).  
   – Bei wiederholtem `rateLimitExceeded` (≥3 Fehler in 60 s) wird automatisch ein Cooldown von 120 s erzwungen, bevor weitere Retries gestartet werden.  
   – Parameter: `--rate-limit-threshold` (Standard: 3), `--rate-limit-window-ms` (Standard: 60000), `--cooldown-ms` (Standard: 120000).  
   – Alle Klassifikationsergebnisse werden in einer separaten Tabelle (`error_classification`) mit Metriken wie `first_seen_at`, `last_seen_at` und `occurrence_count` protokolliert.

3. **Transparente Zustandspersistenz und Wiederherstellbarkeit**  
   – Nach jedem fehlgeschlagenen API-Aufruf wird der aktuelle Zustand (inkl. Rohdaten der Gmail‑Response) als JSONB in PostgreSQL gespeichert.  
   – Bei einem Neustart des Daemons werden alle Einträge aus `retry_queue` mit `next_attempt_at` in der Zukunft geladen und der Scheduler setzt die Timer exakt fort (kein Verlust von geplanten Retries).  
   – Parameter: `--db-connection-string` (PostgreSQL URI), `--recover-on-start` (Standard: true).  
   – Die maximale Aufbewahrungsdauer abgeschlossener Retries wird über `--retention-hours` (Standard: 168) gesteuert; alte Einträge werden automatisch per Cron‑Job (alle 6 h) gelöscht.

### So verwenden Sie es

1. **Initialisierung der PostgreSQL-Datenbank**  
   Führen Sie das bereitgestellte SQL‑Skript `init_schema.sql` aus, um die Tabellen `retry_queue`, `error_classification` und `sent_messages` anzulegen. Das Skript erstellt zudem einen Index auf `retry_queue.next_attempt_at` (B‑Tree) und eine Funktion zur automatischen Berechnung des nächsten Backoff-Intervalls.  
   ```bash
   psql -h localhost -U gmail_retry -d gmailretrybeast -f init_schema.sql
   ```

2. **Konfiguration und Start des Daemons**  
   Erstellen Sie eine Konfigurationsdatei `config.yaml` mit folgenden Pflichtfeldern:  
   ```yaml
   gmail:
     credentials_path: "/etc/gmailretrybeast/credentials.json"
     token_path: "/var/lib/gmailretrybeast/token.json"
     user_id: "me"
   retry:
     max_retries: 5
     base_delay_ms: 250
     max_delay_ms: 64000
     jitter_factor: 0.3
   database:
     connection_string: "postgresql://gmail_retry:pass@localhost:5432/gmailretrybeast"
   ```
   Starten Sie den Daemon mit:  
   ```bash
   gmailretrybeast --config /etc/gmailretrybeast/config.yaml
   ```

3. **Überwachung und manuelle Eingriffe**  
   Der Daemon loggt jeden Retry-Vorgang im strukturierten Format (JSON‑Zeilen) nach `/var/log/gmailretrybeast/retry.log`. Verwenden Sie `journalctl -u gmailretrybeast -f` für Echtzeit-Logs.  
   Um einen bestimmten Eintrag manuell aus der Warteschlange zu entfernen:  
   ```sql
   DELETE FROM retry_queue WHERE message_id = '<MESSAGE_ID>';
   ```
   Um alle fehlgeschlagenen Sendungen eines Zeitraums neu zu starten:  
   ```sql
   UPDATE retry_queue SET retry_count = 0, next_attempt_at = NOW() WHERE last_attempt_at < NOW() - INTERVAL '1 hour';
   ```

### Beste Praktiken

- **Praxis 1: Rate-Limiting pro Benutzerkonto überwachen**  
  Konfigurieren Sie einen separaten Prometheus-Exporter, der die Metriken `gmailretrybeast_retry_queue_size`, `gmailretrybeast_rate_limit_hits_total` und `gmailretrybeast_backoff_duration_seconds` aus dem Daemon abgreift. Setzen Sie einen Alert (z. B. in Grafana), wenn die Warteschlangengröße innerhalb von 5 Minuten um mehr als 1000 Einträge ansteigt – das deutet auf ein massives API-Problem oder eine fehlerhafte Konfiguration hin.

- **Praxis 2: Staging-Umgebung mit simulierten Fehlern testen**  
  Nutzen Sie den integrierten `--dry-run`-Modus (Parameter: `--dry-run`), der alle Retry-Entscheidungen trifft, aber keine tatsächlichen API-Aufrufe ausführt. Kombinieren Sie dies mit einem lokalen Mock-Server, der gezielt die HTTP-Statuscodes 429, 500 und 503 zurückgibt. Validieren Sie, dass die Backoff-Intervalle exakt den konfigurierten Werten entsprechen und die PostgreSQL-Tabellen korrekt befüllt werden. Empfohlen: mindestens 10.000 simulierte Fehler pro Testlauf.
