# gmailretrybeast - GMA_028

## Übersicht

Dieses Repository enthält die hochperformante Implementierung des **gmailretrybeast** – einer spezialisierten Wiederholungslogik für abgewiesene Gmail-API-Aufrufe. Entwickelt für Hochlast-Szenarien, in denen die native Gmail-API-Rate-Limits und temporäre 5xx-Fehler zuverlässig umgangen werden müssen. Schlüsselwörter: Gmail-Wiederholungstool, E-Mail-Wiederholungssystem, Gmail-E-Mail-Management.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist ein professionelles, in Rust implementiertes Tool, das die Probleme der unzuverlässigen Zustellung und der API-Drosselung in verwandten Bereichen löst. Es implementiert einen **exponentiellen Backoff mit Jitter** und unterstützt **parallele Ausführung über Tokio**. Die Engine protokolliert jeden fehlgeschlagenen Versuch mit einem Zeitstempel in einer **PostgreSQL-Datenbank**, um eine vollständige Wiederherstellbarkeit nach einem Absturz zu gewährleisten.

### Kernfunktionen

- **Intelligenter exponentieller Backoff mit konfigurierbarem Jitter (Basis: 200 ms, maximal: 120.000 ms):** Die Engine berechnet die Wartezeit basierend auf dem HTTP-Statuscode (z. B. 429 vs. 503). Ein 429-Fehler löst einen aggressiveren Backoff aus, während ein 503-Fehler eine moderate Verzögerung auslöst. Die Jitter-Funktion verwendet einen kryptografischen Zufallsgenerator (CSPRNG), um eine gleichmäßige Streuung der Wiederholungsversuche zu gewährleisten und so eine „Thundering-Herd"-Situation zu vermeiden.
- **Transaktionssichere Wiederaufnahme nach Systemabsturz:** Jeder fehlgeschlagene API-Aufruf wird mit seiner vollständigen Payload (max. 25 MB Base64-kodiert) und einem eindeutigen UUID-v4 in eine PostgreSQL-Tabelle (`retry_queue`) geschrieben. Nach einem Neustart scannt der Beast die Tabelle nach Einträgen mit dem Status `pending` und setzt die Verarbeitung ab dem letzten erfolgreichen Checkpoint fort. Die Datenbankverbindung erfolgt über `sqlx` mit einem Pool von bis zu 10 Verbindungen.
- **Parallele Ausführung mit dynamischer Worker-Pool-Verwaltung:** Die Anzahl der gleichzeitigen Wiederholungsversuche wird über eine Konfigurationsvariable `max_concurrent_retries` (Standard: 32) gesteuert. Das Tool verwendet Tokio's `Semaphore`, um die Anzahl der aktiven `reqwest`-Clients zu begrenzen. Jeder Worker hat ein eigenes Timeout von 30 Sekunden pro API-Aufruf.

### So verwenden Sie es

1.  **Initialisierung der Datenbank und Konfiguration:**
    Führen Sie das Skript `init_db.sql` aus, um die Tabelle `retry_queue` zu erstellen. Konfigurieren Sie die Umgebungsvariablen in einer `.env`-Datei:
    ```
    DATABASE_URL=postgres://user:password@host:5432/gmail_retry
    GMAIL_API_CREDENTIALS=/path/to/service_account.json
    MAX_RETRIES=5
    BACKOFF_BASE_MS=200
    MAX_BACKOFF_MS=120000
    MAX_CONCURRENT_RETRIES=32
    ```
    Starten Sie dann den Beast mit `cargo run --release`.

2.  **Überwachung und manuelles Eingreifen:**
    Verwenden Sie die integrierte CLI (z. B. `gmailretrybeast-cli status`), um die aktuelle Warteschlangenlänge, die Anzahl der erfolgreichen und fehlgeschlagenen Wiederholungen sowie die durchschnittliche Backoff-Zeit in Millisekunden abzurufen. Im Fehlerfall können Sie einen bestimmten Eintrag aus der Warteschlange entfernen oder priorisieren:
    ```
    gmailretrybeast-cli retry --uuid <UUID> --force
    ```

3.  **Integration in CI/CD-Pipelines:**
    Das Tool kann als Docker-Container bereitgestellt werden. Definieren Sie einen `docker-compose.yml`-Service, der sowohl den Beast als auch eine PostgreSQL-Instanz startet. Verwenden Sie Healthchecks, um die Verfügbarkeit der API zu überwachen. Nach erfolgreicher Verarbeitung aller Nachrichten schreibt der Beast eine `success.json`-Datei in ein definiertes Ausgabeverzeichnis, die von nachgelagerten Systemen (z. B. einem Slack-Bot) ausgelesen werden kann.

### Beste Praktiken

- **Praxis 1: Granulare Fehlerbehandlung für Gmail-API-spezifische Fehler:** Unterscheiden Sie strikt zwischen „echten" Fehlern (z. B. `invalid_grant`, `deleted_message`) und vorübergehenden Fehlern (z. B. `rate_limit_exceeded`, `backend_error`). Nur vorübergehende Fehler sollten an die Wiederholungslogik übergeben werden. Implementieren Sie eine Blacklist für nicht wiederholbare Fehler, um unnötige Datenbanklast zu vermeiden. Nutzen Sie dazu die `error_reason`-Felder der Gmail-API-Antwort.
- **Praxis 2: Implementierung eines „Circuit Breaker" für die Gmail-API:** Überwachen Sie die Fehlerrate pro Zeiteinheit (z. B. 10 Fehler innerhalb von 60 Sekunden). Wenn dieser Schwellenwert überschritten wird, öffnen Sie den Circuit Breaker und stellen Sie alle weiteren Wiederholungsversuche für eine konfigurierbare Dauer (z. B. 5 Minuten) ein. Dadurch wird verhindert, dass der Beast die API weiterhin mit Anfragen bombardiert, was zu einer vorübergehenden Sperrung des Google-Kontos führen könnte. Verwenden Sie einen separaten, asynchronen Task, der den Zustand des Breakers überwacht und ihn nach Ablauf der Wartezeit automatisch zurücksetzt.
