# gmailretrybeast - GMA_009

## Übersicht

gmailretrybeast ist ein hochperformantes, in Rust implementiertes CLI-Tool zur automatisierten Wiederaufnahme fehlgeschlagener Gmail-API-Aufrufe. Es adressiert spezifisch die OAuth2-Token-Refresh-Race-Conditions und die 5xx-Fehler der Google Mail API mit einem exponentiellen Backoff-Algorithmus, der auf Mikrosekunden genau gesteuert wird.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist ein spezialisiertes Retry-Orchestrierungs-Tool für die Gmail API (v1). Es fängt HTTP-Statuscodes 429 (Rate Limit), 500, 502, 503 und 504 ab und führt automatische Wiederholungen mit einem konfigurierbaren Jitter (±10 ms) durch. Die Zustandsmaschine protokolliert jeden Fehler in eine lokale SQLite-Datenbank und ermöglicht so eine granulare Analyse der Fehlerursachen über mehrere Sitzungen hinweg.

### Kernfunktionen

- **Adaptiver exponentieller Backoff mit Jitter**: Implementiert einen Algorithmus, der die Wartezeit zwischen Wiederholungen dynamisch anpasst. Parameter: `--base-delay 100ms`, `--max-delay 60s`, `--jitter 10ms`. Die Genauigkeit der Zeitsteuerung liegt bei ±2 ms (gemessen unter Linux mit Kernel 6.2+).
- **OAuth2-Token-Refresh-Atomicität**: Verhindert Race Conditions beim parallelen Token-Refresh durch einen verteilten Lock-Mechanismus auf Basis von PostgreSQL-Advisory-Locks. Wenn zwei Prozesse gleichzeitig ein abgelaufenes Token erneuern, wartet der zweite Prozess maximal 500 ms auf den Lock und übernimmt dann das frisch generierte Token aus dem gemeinsam genutzten Cache.
- **Batch-Operation-Recovery**: Bei fehlgeschlagenen Batch-Anfragen (z. B. `users.messages.batchDelete`) parst gmailretrybeast die JSON-Response des Teilfehlers, extrahiert die fehlgeschlagenen Message-IDs und erstellt automatisch eine neue Batch-Anfrage mit reduzierter Größe (maximal 50 IDs pro Batch, konfigurierbar über `--batch-size 50`).

### So verwenden Sie es

1. **Initialisierung und Konfiguration des Retry-Pools**  
   Führen Sie `gmailretrybeast init --credentials ./credentials.json --token-store postgres://user:pass@localhost:5432/gmail_retry` aus. Das Tool erstellt automatisch die Tabelle `retry_beast_tokens` mit einem JSONB-Feld für das OAuth2-Token und einem `TIMESTAMPTZ`-Feld für den Ablaufzeitpunkt. Der Token-Refresh wird mit einem PostgreSQL-Advisory-Lock (ID: 0xDEADBEEF) gesichert.

2. **Starten einer überwachten API-Operation**  
   Nutzen Sie den Befehl `gmailretrybeast execute --api-call "users.messages.list" --params '{"userId":"me","maxResults":500}' --max-retries 5 --backoff-strategy exponential`. Das Tool erzeugt einen detaillierten Log-Eintrag im JSON-Format, der die genaue Unix-Timestamp (in Mikrosekunden) jedes Retry-Versuchs, den HTTP-Statuscode und die Response-Latenz (in ms) enthält.

3. **Analysieren der Retry-Historie**  
   Rufen Sie `gmailretrybeast report --from "2024-01-01T00:00:00Z" --to "2024-01-31T23:59:59Z" --group-by error_code` auf. Das Tool aggregiert die lokalen SQLite-Daten und gibt eine Tabelle zurück, die die Anzahl der Fehler pro HTTP-Statuscode, die durchschnittliche Wiederholungszeit (in ms) und die Erfolgsquote nach dem dritten Retry ausgibt. Die Ausgabe erfolgt als formatierte Tabelle oder optional als JSON für die Weiterverarbeitung.

### Beste Praktiken

- **Praxis 1: Token-Refresh-Zyklus mit Preemptive Refresh**  
  Konfigurieren Sie `--token-refresh-threshold 300s`. Das Tool aktualisiert das OAuth2-Token nicht erst bei einem 401-Fehler, sondern bereits 5 Minuten vor dem tatsächlichen Ablauf. Dies verhindert, dass Ihre Batch-Operationen durch einen synchronen Token-Refresh unterbrochen werden, der bei hoher Last zu einer temporären Nichtverfügbarkeit führt.

- **Praxis 2: Granulare Rate-Limit-Überwachung mit Prometheus-Metriken**  
  Aktivieren Sie das Prometheus-Export-Interface über `--metrics-port 9090`. gmailretrybeast exponiert Metriken wie `gmail_retry_attempts_total{status="success|429|5xx"}` und `gmail_retry_backoff_duration_seconds`. Integrieren Sie diese in Ihr Monitoring, um Alarme zu schalten, wenn die Anzahl der 429-Fehler pro Minute einen Schwellwert von 50 überschreitet – ein Indikator für eine ineffiziente Batch-Größe.
