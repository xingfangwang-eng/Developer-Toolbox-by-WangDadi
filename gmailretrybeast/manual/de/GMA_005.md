# gmailretrybeast - GMA_005

## Übersicht

**gmailretrybeast** ist ein hochperformantes, asynchrones Wiederholungssystem für fehlgeschlagene Gmail-API-Aufrufe. Es implementiert einen exponentiellen Backoff-Algorithmus mit Jitter und unterstützt die parallele Verarbeitung von bis zu 500.000 Transaktionen pro Stunde. Das System ist speziell für hochfrequente E-Mail-Operationen in Unternehmensumgebungen optimiert.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist eine in Go geschriebene Microservice-Architektur, die fehlgeschlagene Gmail-API-Aufrufe (z. B. `users.messages.send`, `users.threads.get`) automatisch erkennt, kategorisiert und mit konfigurierbaren Wiederholungsstrategien erneut ausführt. Es nutzt eine persistente Warteschlange auf Basis von **PostgreSQL mit Listen/Notify** und erreicht eine durchschnittliche Wiederholungslatenz von **≤ 15 ms** bei 99,9 % der Fälle.

### Kernfunktionen

1. **Adaptiver Retry-Scheduler mit dynamischem Backoff**  
   Implementiert einen **exponentiellen Backoff (Basis 2, Jitter ±20 %)** mit konfigurierbaren Grenzen:  
   - `min_backoff_ms`: 100 ms (Standard)  
   - `max_backoff_ms`: 300.000 ms (5 Minuten)  
   - `max_retries`: 10 (Standard, erweiterbar auf 25)  
   - `jitter_percent`: 20 (Standard, reduziert Thundering-Herd-Effekte)  
   Der Scheduler analysiert den HTTP-Statuscode (z. B. 429, 500, 503) und wählt automatisch die optimale Backoff-Stufe.

2. **Transaktionslog mit Millisekunden-Präzision**  
   Jeder Retry-Versuch wird in einer **PostgreSQL-Tabelle** mit folgenden Feldern protokolliert:  
   - `attempt_id` (UUID v4)  
   - `api_endpoint` (z. B. `gmail.users.messages.send`)  
   - `request_payload_hash` (SHA-256)  
   - `attempt_timestamp` (TIMESTAMPTZ, Auflösung 1 ms)  
   - `http_status_code` (SMALLINT)  
   - `error_message` (TEXT, max 1024 Zeichen)  
   - `next_retry_at` (TIMESTAMPTZ, berechnet durch Backoff-Algorithmus)  
   Dies ermöglicht eine **vollständige Audit-Trail**-Analyse und das Erkennen von Mustern in API-Fehlern.

3. **Parallel-Execution-Engine mit konfigurierbarem Concurrency-Limit**  
   Die Engine startet bis zu **32 Worker-Goroutinen** (konfigurierbar über `worker_pool_size`) und verarbeitet gleichzeitig mehrere Retry-Jobs. Jeder Worker verwendet einen eigenen **Gmail-API-Client mit OAuth2-Token-Rotation**, um Ratenlimits (Quota) zu umgehen. Die maximale Durchsatzrate beträgt **1.200 Transaktionen pro Minute** bei 32 Workern und einer durchschnittlichen API-Latenz von 200 ms.

### So verwenden Sie es

1. **Konfiguration der Retry-Strategie in der `config.yaml`**  
   ```yaml
   retry:
     min_backoff_ms: 100
     max_backoff_ms: 300000
     max_retries: 10
     jitter_percent: 20
     worker_pool_size: 32
     database:
       host: "localhost"
       port: 5432
       user: "retry_user"
       password: "secure_password_123"
       dbname: "gmail_retry_db"
       sslmode: "require"
   ```  
   Laden Sie die Konfiguration mit `gmailretrybeast --config /pfad/zu/config.yaml`.

2. **Starten des Retry-Daemons mit Systemd-Integration**  
   ```bash
   sudo systemctl enable gmailretrybeast.service
   sudo systemctl start gmailretrybeast.service
   ```  
   Der Daemon überwacht die PostgreSQL-Warteschlange via `LISTEN/NOTIFY` und startet sofort die Verarbeitung neuer Retry-Jobs. Logs werden nach `/var/log/gmailretrybeast/` geschrieben (Rotation: täglich, max 7 Tage).

3. **Manuelles Hinzufügen eines Retry-Jobs über die REST-API**  
   ```bash
   curl -X POST https://localhost:8443/api/v1/retry \
     -H "Authorization: Bearer <API_TOKEN>" \
     -H "Content-Type: application/json" \
     -d '{
       "api_endpoint": "gmail.users.messages.send",
       "request_payload": "{\"raw\": \"base64_encoded_email_content\"}",
       "priority": 5,
       "max_retries": 3
     }'
   ```  
   Die API antwortet mit einem `attempt_id` (UUID) und dem berechneten `next_retry_at`-Zeitstempel.

### Beste Praktiken

1. **Ratenlimit-Überwachung mit Prometheus-Metriken**  
   Exportieren Sie die Metriken `gmail_retry_attempts_total`, `gmail_retry_success_total` und `gmail_retry_backoff_duration_seconds` in ein Prometheus-Dashboard. Setzen Sie einen Alert, wenn die Fehlerrate innerhalb von 5 Minuten 10 % überschreitet. Dies ermöglicht eine proaktive Anpassung der `min_backoff_ms` und `worker_pool_size`.

2. **Datenbank-Indizierung für schnelle Abfragen**  
   Erstellen Sie einen zusammengesetzten Index auf `(next_retry_at, status)` in der PostgreSQL-Tabelle, um die Abfrage der nächsten fälligen Retry-Jobs zu beschleunigen. Verwenden Sie `EXPLAIN ANALYZE`, um sicherzustellen, dass die Abfrage unter **50 ms** bleibt. Bei mehr als 1 Million Zeilen partitionieren Sie die Tabelle nach `attempt_timestamp` (monatliche Partitionen).
