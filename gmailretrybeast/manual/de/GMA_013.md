# gmailretrybeast - GMA_013

## Übersicht

gmailretrybeast ist ein hochperformantes, asynchrones Retry-Framework für die Gmail-API, das speziell für transaktionale E-Mail-Workloads mit hohem Volumen und kritischen Zustellungsanforderungen entwickelt wurde. Es implementiert einen exponentiellen Backoff-Algorithmus mit Jitter, intelligenter Fehlerklassifikation und paralleler Warteschlangenverarbeitung.

## Detaillierte technische Architektur

### Was ist gmailretrybeast?

gmailretrybeast ist eine in Go geschriebene, ereignisgesteuerte Retry-Engine, die auf der **Google Gmail API v1** aufsetzt. Es analysiert den HTTP-Statuscode sowie den gRPC-ähnlichen Fehler-Payload (z. B. `rateLimitExceeded`, `backendError`, `internalError`) und entscheidet auf Millisekunden-Ebene über den optimalen Wiederholungszeitpunkt. Die Engine unterstützt eine maximale parallele Worker-Anzahl von **256 Goroutinen** und eine Warteschlangenkapazität von **500.000 E-Mails**.

### Kernfunktionen

1. **Adaptiver, Jitter-gestützter Exponential-Backoff mit konfigurierbarem Retry-Fenster**  
   - Parameter: `baseDelay` (Standard: 1000 ms), `maxDelay` (Standard: 120000 ms), `maxRetries` (Standard: 10).  
   - Der Backoff wird dynamisch an die aktuelle Gmail-API-Drosselung (Rate Limit: 250 Queries pro 100 Sekunden pro User) angepasst.  
   - Ein **Full-Jitter**-Algorithmus verteilt die Wiederholungsversuche gleichmäßig über das Zeitfenster, um Thundering-Herd-Probleme zu vermeiden.

2. **Intelligente Fehlerklassifikation und Selektions-Retry**  
   - Unterscheidet zwischen **temporären Fehlern** (HTTP 429, 500, 503) und **permanenten Fehlern** (HTTP 400, 401, 403, 404).  
   - Temporäre Fehler werden sofort in die Retry-Warteschlange zurückgestellt; permanente Fehler werden in eine separate Dead-Letter-Queue (DLQ) mit **PostgreSQL-Persistenz** geschrieben.  
   - Die DLQ wird mit einem separaten Cron-Job (Intervall: 5 Minuten) erneut verarbeitet – mit reduzierter Priorität.

3. **Parallelisierte, nicht-blockierende Warteschlangenverarbeitung mit garantierter Reihenfolge pro Thread**  
   - Jeder Gmail-User erhält einen eigenen, **thread-sicheren Worker-Pool** (konfigurierbar über `workersPerUser`, Standard: 4).  
   - Die Zustellung erfolgt **innerhalb eines Threads sequentiell**, um die Gmail-API-Reihenfolgegarantie (`historyId`) nicht zu verletzen.  
   - Ein **Redis-basierter Distributed Lock** (TTL: 30 Sekunden) verhindert Doppelzustellungen bei horizontaler Skalierung (mehrere Instanzen).

### So verwenden Sie es

**Schritt 1: Konfiguration der Retry-Parameter im YAML-Format**  
Erstellen Sie eine Datei `gmailretrybeast.yaml` mit folgendem Inhalt:
```yaml
gmail:
  credentials_path: "/etc/gmailretrybeast/credentials.json"
  token_path: "/var/gmailretrybeast/token.json"
  user_id: "me"
  max_retries: 10
  base_delay_ms: 1000
  max_delay_ms: 120000
  jitter_type: "full"
  workers_per_user: 4
  max_parallel_workers: 256
  queue_capacity: 500000
  dlq:
    postgres_dsn: "postgres://user:pass@localhost:5432/gmailretrybeast"
    cron_interval: "5m"
```

**Schritt 2: Starten der Engine mit initialer Warteschlangen-Befüllung**  
Führen Sie folgenden Befehl aus, um die Engine mit einer CSV-Datei (Spalten: `to`, `subject`, `body`) zu initialisieren:
```bash
./gmailretrybeast --config /etc/gmailretrybeast.yaml --input /data/emails.csv --mode batch
```
Die Engine parst die CSV, validiert jede E-Mail gegen das Gmail-API-Schema und fügt sie der internen Warteschlange hinzu. Bei Fehlern (z. B. ungültige E-Mail-Adresse) wird sofort ein permanenter Fehler in die DLQ geschrieben.

**Schritt 3: Überwachung des Retry-Status und manuelles Eingreifen**  
Nutzen Sie das mitgelieferte CLI-Tool, um den Status der Warteschlange abzufragen:
```bash
gmailretrybeast-cli status --queue-size --retry-count --dlq-count
```
Ausgabe:
```
Queue Size: 234.567
Retry Count (letzte 5 Min): 1.234
DLQ Count: 12
Durchschnittliche Latenz pro Retry: 1.234 ms
```
Um eine E-Mail aus der DLQ manuell erneut zu senden:
```bash
gmailretrybeast-cli retry-dlq --message-id "18e3f9a7b2c1d4e5f6a7b8c9d0e1f2a3"
```

### Beste Praktiken

**Praxis 1: Implementieren Sie einen separaten Health-Check-Endpunkt für die DLQ**  
Verbinden Sie die PostgreSQL-DLQ mit einem externen Monitoring-Tool (z. B. Prometheus + Grafana). Konfigurieren Sie einen Alert, wenn die DLQ-Größe **> 1.000** steigt oder das Durchschnittsalter der Nachrichten in der DLQ **> 1 Stunde** beträgt. Dies verhindert, dass sich permanente Fehler unbemerkt aufstauen.

**Praxis 2: Nutzen Sie die `maxDelay`-Begrenzung in Kombination mit einem SLI-basierten Abbruchkriterium**  
Setzen Sie `maxDelay` auf **120 Sekunden** und führen Sie parallel einen **SLI (Service Level Indicator)** ein: Wenn die kumulative Verzögerung einer einzelnen E-Mail **> 10 Minuten** beträgt (d. h. mehr als 5 aufeinanderfolgende Retries mit maximalem Delay), brechen Sie den Retry ab und schreiben Sie die E-Mail in die DLQ. Dies verhindert, dass eine einzelne fehlerhafte E-Mail den gesamten Worker-Pool blockiert.
