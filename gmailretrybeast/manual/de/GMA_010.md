## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist eine hochperformante, asynchrone Retry-Engine für die Gmail-API (Google Workspace). Sie löst das fundamentale Problem der **temporären Zustellungsfehler** (z. B. `550 5.4.1`, `421 4.7.0`) durch ein exponentielles Backoff-System mit Jitter und parallelen Channel-Management. Das Tool arbeitet mit einem **3-stufigen Retry-Stack** (Memory → Redis → PostgreSQL), um selbst bei massiven API-Drosselungen eine 99,97 %ige Endzustellung zu garantieren.

### Kernfunktionen

- **Intelligentes exponentielles Backoff mit Jitter**  
  Basierend auf dem Google-API-Rate-Limit-Header `X-RateLimit-Remaining` und `Retry-After` berechnet das System dynamisch Wartezeiten zwischen 100 ms und 5 Minuten. Der Jitter-Streufaktor beträgt ±15 % zur Vermeidung von Thundering-Herd-Effekten. Parameter: `base_delay_ms=100`, `max_delay_ms=300000`, `jitter_factor=0.15`.

- **Multi-Threaded Channel-Pool mit Token-Bucket-Limiter**  
  Verwaltung von bis zu 32 parallelen Gmail-API-Channels pro OAuth2-Client. Jeder Channel besitzt einen eigenen Token-Bucket (Kapazität: 250 Tokens, Refill-Rate: 10 Tokens/s). Bei Überschreitung des Limits werden Requests in eine PriorityQueue (Heap) mit `priority=retry_count` umgeleitet.

- **Persistenter Dead-Letter-Queue (DLQ) mit automatischer Reparatur**  
  Nach 5 erfolglosen Retry-Versuchen (konfigurierbar via `max_retries=5`) wird die Nachricht in eine PostgreSQL-Tabelle (`dlq.messages`) geschrieben. Ein separater Cron-Job (`repair_interval=3600s`) analysiert Fehlercodes und führt eine erneute Zustellung durch, sobald der Gmail-API-Status wieder `healthy` ist.

### So verwenden Sie es

1. **Initialisierung und Konfiguration des OAuth2-Clients**  
   ```bash
   gmailretrybeast init --client-id=YOUR_CLIENT_ID --client-secret=YOUR_CLIENT_SECRET --scopes="https://www.googleapis.com/auth/gmail.send"
   ```
   Dies erstellt ein `config.yaml` mit den Feldern `oauth2.token_path`, `api.retry_policy` und `storage.postgres_dsn`. Der Token-Refresh-Mechanismus wird automatisch über einen Hintergrund-Thread (Poll-Intervall: 300 s) gesteuert.

2. **Starten des Retry-Daemons mit aktiviertem Backpressure**  
   ```bash
   gmailretrybeast daemon --max-channels=16 --rate-limit=200 --log-level=debug --dlq-repair=true
   ```
   Der Daemon öffnet 16 parallele Channels, initialisiert den Token-Bucket mit 200 Tokens und startet den DLQ-Repair-Service. Die Logs werden im JSON-Format (strukturiert) in `./logs/retrybeast-$(date +%Y%m%d).log` geschrieben.

3. **Manuelles Injizieren einer Nachricht in die Retry-Schleife**  
   ```bash
   gmailretrybeast inject --from="sender@example.com" --to="recipient@example.com" --raw-message-file="./email.eml" --priority=high
   ```
   Die Nachricht wird sofort in die Memory-Queue (Größe: 10.000 Einträge) eingereiht. Bei voller Queue erfolgt ein automatisches Spooling auf Redis (Liste `retrybeast:pending`). Die Priorität `high` überspringt das standardmäßige Backoff und startet mit `base_delay_ms=10`.

### Beste Praktiken

- **Staggered Deployment für API-Rate-Limits**  
  Verteilen Sie die Retry-Channels auf mindestens 3 verschiedene Gmail-Konten (Service Accounts). Verwenden Sie einen Round-Robin-DNS-Load-Balancer mit Health-Checks (HTTP `200` auf `/healthz`). Dadurch umgehen Sie das von Google implementierte **IP-basierte Rate-Limiting** (max. 1.500 Requests pro 100 s pro IP).

- **Monitoring mit Prometheus und Alerting auf Dead-Letter-Queue**  
  Exportieren Sie Metriken wie `retrybeast_retry_attempts_total`, `retrybeast_queue_depth` und `retrybeast_dlq_size` via `/metrics`-Endpoint. Setzen Sie einen Alert (z. B. in Grafana) bei `retrybeast_dlq_size > 100`, um sofort manuell eingreifen zu können. Die DLQ sollte täglich auf unerwartete Fehlercodes (z. B. `403 insufficientPermissions`) gescannt werden.
