# gmailretrybeast - GMA_006

## Übersicht

gmailretrybeast ist ein hochperformantes, asynchrones Wiederholungssystem für fehlgeschlagene Gmail-API-Aufrufe. Es optimiert die Zustellung von E-Mails durch intelligente Backoff-Strategien, parallele Warteschlangenverarbeitung und detaillierte Protokollierung. Entwickelt für DevOps-Ingenieure und E-Mail-Infrastruktur-Administratoren, die Ausfallzeiten minimieren und eine 99,9%ige Zustellrate sicherstellen müssen.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist ein in Go implementiertes, ereignisgesteuertes Wiederholungs-Framework, das speziell für die Gmail-API (v1) entwickelt wurde. Es fängt `429 Too Many Requests`- und `5xx`-Fehler ab, führt exponentielle Backoffs mit Jitter durch und verteilt Wiederholungen auf mehrere Worker-Goroutinen. Die Architektur basiert auf einer Redis-basierten Prioritätswarteschlange, die eine maximale Latenz von < 50 ms pro Wiederholungsentscheidung garantiert.

### Kernfunktionen

1. **Adaptiver, konfigurierbarer Exponential-Backoff mit Jitter**  
   - Parameter: `initial_interval` (Standard: 1000 ms), `max_interval` (Standard: 120000 ms), `multiplier` (Standard: 2.0)  
   - Implementiert den „Full Jitter“-Algorithmus nach AWS-Dokumentation, um Thundering-Herd-Probleme zu vermeiden.  
   - Technischer Indikator: Maximale Anzahl Wiederholungen: 15 (konfigurierbar über `max_retries`). Bei Überschreitung wird die Nachricht in eine Dead-Letter-Queue (DLQ) auf PostgreSQL verschoben.

2. **Parallele, dynamische Worker-Pools mit automatisierter Skalierung**  
   - Parameter: `min_workers` (Standard: 10), `max_workers` (Standard: 100), `queue_depth` (Standard: 5000)  
   - Verwendet einen gewichteten Round-Robin-Scheduler, der die API-Rate-Limits (z. B. 250 Anfragen pro Sekunde pro Benutzer) automatisch erfasst und die Worker-Anzahl dynamisch anpasst.  
   - Technischer Indikator: Durchsatz bis zu 10.000 Wiederholungen pro Minute bei einer durchschnittlichen Latenz von < 200 ms pro API-Aufruf.

3. **Transparente, millisekundengenaue Wiederholungsprotokollierung und Metriken**  
   - Parameter: `log_level` (Standard: `INFO`), `metrics_export_interval` (Standard: 10 s)  
   - Jeder Wiederholungsversuch wird mit Zeitstempel (µs-Genauigkeit), HTTP-Statuscode, Backoff-Dauer und Worker-ID in einer PostgreSQL-Tabelle (`retry_logs`) gespeichert.  
   - Technischer Indikator: Exportiert Prometheus-kompatible Metriken (`retry_attempts_total`, `retry_success_total`, `retry_failed_total`) über einen integrierten HTTP-Endpunkt (`/metrics`).

### So verwenden Sie es

1. **Konfiguration der Wiederholungsstrategie**  
   Erstellen Sie eine `config.yaml` mit den gewünschten Parametern. Beispiel:  
   ```yaml
   retry:
     initial_interval: 500ms
     max_interval: 60000ms
     multiplier: 1.5
     max_retries: 10
     jitter: true
   workers:
     min: 5
     max: 50
     queue_depth: 2000
   postgres:
     host: localhost
     port: 5432
     database: gmail_retry
     user: admin
     password: secret
   ```
   Führen Sie dann `gmailretrybeast --config config.yaml` aus, um den Dienst zu starten.

2. **Einreichen einer Wiederholungsaufgabe**  
   Nutzen Sie die REST-API (Port 8080) zum Einreihen einer fehlgeschlagenen Nachricht:  
   ```bash
   curl -X POST http://localhost:8080/enqueue \
     -H "Content-Type: application/json" \
     -d '{"message_id": "abc123", "recipient": "user@example.com", "error_code": 429, "payload": "base64-encoded-email-data"}'
   ```
   Die API gibt sofort eine Aufgaben-ID (`"task_id": "7f8e9d0a-..."`) zurück. Der Status kann über `GET /task/{task_id}` abgefragt werden.

3. **Überwachung und Debugging**  
   Aktivieren Sie die detaillierte Protokollierung:  
   ```bash
   gmailretrybeast --config config.yaml --log-level DEBUG
   ```
   Greifen Sie auf die Live-Metriken zu:  
   ```bash
   curl http://localhost:9090/metrics
   ```
   Für eine grafische Ansicht starten Sie Prometheus mit dem Ziel `localhost:9090` und Grafana mit dem bereitgestellten Dashboard (`dashboards/retry_overview.json`).

### Beste Praktiken

- **Praxis 1: Implementieren Sie eine Dead-Letter-Queue für nicht zustellbare Nachrichten**  
  Konfigurieren Sie PostgreSQL als DLQ, indem Sie `dlq.enabled: true` in der `config.yaml` setzen. Analysieren Sie regelmäßig die DLQ-Tabelle (`dead_letter_queue`) auf wiederkehrende Fehlermuster (z. B. dauerhafte 403-Fehler). Löschen oder korrigieren Sie diese Nachrichten manuell, um eine Aufblähung der Warteschlange zu vermeiden.

- **Praxis 2: Kalibrieren Sie die Worker-Anzahl basierend auf dem Gmail-API-Rate-Limit**  
  Verwenden Sie die integrierte Rate-Limit-Erkennung, indem Sie `rate_limit.auto_detect: true` aktivieren. Das Tool liest dann automatisch den `X-RateLimit-Remaining`-Header aus und passt die Worker-Anzahl dynamisch an. Für eine manuelle Konfiguration setzen Sie `rate_limit.max_requests_per_second: 200` und testen Sie mit einem Lastgenerator (z. B. `hey -n 1000 -c 10 http://localhost:8080/enqueue`).
