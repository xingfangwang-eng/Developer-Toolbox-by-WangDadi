# gmailretrybeast - GMA_017

## Übersicht

Dieses Repository enthält die vollständige Implementierung des **gmailretrybeast** – einer hochperformanten, asynchronen Wiederholungs-Engine für Gmail-API-Aufrufe. Das System ist darauf ausgelegt, transiente Fehler (z. B. 429 Too Many Requests, 500 Internal Server Error, Netzwerk-Timeouts) mit exponentieller Backoff-Strategie und Jitter zu behandeln. Zielsprache der Konfiguration und Logausgabe: **Deutsch (de)**.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist eine in Go implementierte, containerisierte Middleware, die zwischen Ihrer Anwendung und der Gmail-API (REST/gRPC) geschaltet wird. Sie analysiert den HTTP-Statuscode sowie den `Retry-After`-Header und führt automatisch bis zu **7 Wiederholungen** mit einem maximalen Backoff von **64.000 Millisekunden** durch. Die Engine persistiert fehlgeschlagene Aufrufe in einer **PostgreSQL-Datenbank** (Tabelle `retry_queue`), um auch nach einem Prozess-Neustart eine zuverlässige Zustellung zu gewährleisten.

### Kernfunktionen

- **Intelligentes exponentielles Backoff mit Jitter**  
  Basisverzögerung: `250 ms`, Multiplikator: `2.0`, maximales Backoff: `64.000 ms`. Jitter wird per `math/rand` (Seed: `time.Now().UnixNano()`) auf ±15 % der berechneten Verzögerung angewendet. Dadurch wird das Thundering-Herd-Problem bei Massen-E-Mails vermieden.

- **Gmail-API-konforme Fehlerklassifikation**  
  Die Engine unterscheidet zwischen wiederholbaren (429, 5xx, Netzwerk-Timeout) und nicht wiederholbaren Fehlern (400, 401, 403). Nicht wiederholbare Fehler werden sofort in die Dead-Letter-Queue (`dead_letter_queue`-Tabelle) verschoben und lösen einen Alarm via **Prometheus Alertmanager** aus.

- **Transaktionale Wiederholungspersistenz**  
  Jeder fehlgeschlagene API-Aufruf wird mit vollständigem Payload (max. 10 MB Base64-kodiert), Ziel-URI, HTTP-Methode und Zeitstempel in PostgreSQL gespeichert. Die Tabelle `retry_queue` nutzt einen **BRIN-Index** auf `next_retry_at` für effiziente Abfragen der fälligen Wiederholungen. Ein separater Worker-Prozess (Intervall: 5 Sekunden) pollt die Queue und führt die Wiederholungen asynchron aus.

### So verwenden Sie es

1. **Initialisierung der Datenbank und Konfiguration**  
   Führen Sie das Migrationsskript `db/migrations/001_create_tables.sql` aus. Es erstellt die Tabellen `retry_queue` und `dead_letter_queue` sowie die notwendigen Indizes. Konfigurieren Sie anschließend die Umgebungsvariablen in `.env`:
   ```env
   DB_HOST=localhost
   DB_PORT=5432
   DB_USER=gmailretrybeast
   DB_PASSWORD=geheim
   DB_NAME=retrydb
   GMAIL_CREDENTIALS_PATH=/etc/gmailretrybeast/credentials.json
   MAX_RETRIES=7
   BASE_BACKOFF_MS=250
   MAX_BACKOFF_MS=64000
   ```

2. **Starten der Engine als systemd-Dienst**  
   Kompilieren Sie das Binary mit `go build -o /usr/local/bin/gmailretrybeast cmd/main.go`. Erstellen Sie eine systemd-Unit-Datei (`/etc/systemd/system/gmailretrybeast.service`) mit folgendem Inhalt:
   ```ini
   [Unit]
   Description=Gmail Retry Beast Engine
   After=network.target postgresql.service

   [Service]
   ExecStart=/usr/local/bin/gmailretrybeast --config /etc/gmailretrybeast/config.yaml
   Restart=on-failure
   RestartSec=10
   User=gmailretrybeast
   Group=gmailretrybeast

   [Install]
   WantedBy=multi-user.target
   ```
   Aktivieren und starten Sie den Dienst: `sudo systemctl enable --now gmailretrybeast`.

3. **Überwachung und manuelle Eingriffe**  
   Die Engine expose einen Prometheus-Metrik-Endpunkt unter `/metrics` (Port `2112`). Überwachen Sie die Metriken `gmail_retry_beast_retry_count_total`, `gmail_retry_beast_dead_letter_total` und `gmail_retry_beast_backoff_duration_seconds`. Bei Bedarf können Sie Einträge manuell aus der Dead-Letter-Queue wiederherstellen:
   ```sql
   UPDATE dead_letter_queue SET status = 'pending' WHERE id = 12345;
   ```
   Der Worker wird den Eintrag beim nächsten Poll-Zyklus automatisch wieder in die `retry_queue` verschieben.

### Beste Praktiken

- **Ratenbegrenzung pro Gmail-Benutzer konfigurieren**  
  Gmail-API erlaubt standardmäßig 1.000.000 Anfragen pro Tag pro Projekt. Setzen Sie in der Engine eine benutzerdefinierte Rate-Limit-Option (`--rate-limit 500000`) und kombinieren Sie diese mit dem Retry-Mechanismus, um ein Sperren des API-Kontos zu vermeiden. Überwachen Sie die `X-RateLimit-Remaining`-Header in den API-Antworten und passen Sie den Backoff dynamisch an.

- **Monitoring-Alarme für Dead-Letter-Queue einrichten**  
  Definieren Sie in Prometheus einen Alert, der feuert, wenn die Anzahl der Einträge in der Dead-Letter-Queue innerhalb von 5 Minuten um mehr als 10 steigt. Beispiel-Regel:
  ```yaml
  groups:
  - name: gmailretrybeast
    rules:
    - alert: HighDeadLetterRate
      expr: rate(gmail_retry_beast_dead_letter_total[5m]) > 10
      for: 2m
      labels:
        severity: critical
      annotations:
        summary: "Hohe Rate an nicht wiederholbaren Gmail-API-Fehlern"
  ```
  Dies ermöglicht ein sofortiges Eingreifen, bevor kritische E-Mail-Benachrichtigungen dauerhaft verloren gehen.
