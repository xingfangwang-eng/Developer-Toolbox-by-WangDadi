# gmailretrybeast - GMA_029

## Überblick

Dieses Projekt realisiert eine hochperformante, asynchrone Wiederholungslogik für Gmail-API-Aufrufe, die speziell für massenhafte E-Mail-Zustellungen und Retry-Szenarien optimiert ist. Kernziel ist die Minimierung von API-Fehlern durch intelligente Backoff-Strategien und parallele Verarbeitung.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist ein spezialisiertes Framework zur automatischen Wiederholung fehlgeschlagener Gmail-API-Operationen. Es implementiert einen exponentiellen Backoff-Algorithmus mit Jitter, der auf Millisekundenebene gesteuert wird, und integriert eine PostgreSQL-basierte Zustandsverwaltung für persistente Retry-Queues. Das Tool eignet sich für Szenarien mit hohem Durchsatz (z. B. 10.000+ E-Mails pro Stunde) und bietet native Unterstützung für OAuth2-Token-Rotation.

### Kernfunktionen

- **Dynamischer Exponenzial-Backoff mit konfigurierbarem Jitter**: Parameter wie `initial_delay_ms` (Standard: 1000), `max_delay_ms` (Standard: 60000) und `jitter_factor` (Standard: 0.25) ermöglichen eine präzise Steuerung der Wiederholungsintervalle. Die Funktion misst die tatsächliche Latenz jedes API-Aufrufs in Mikrosekunden und passt den nächsten Backoff dynamisch an.

- **Parallele Batch-Verarbeitung mit PostgreSQL-Persistenz**: Die Retry-Queue wird in einer PostgreSQL-Tabelle (`retry_queue`) mit den Spalten `message_id`, `retry_count`, `next_retry_at` (timestamp with time zone) und `error_code` gespeichert. Ein Worker-Pool (konfigurierbar über `max_workers`, Standard: 50) verarbeitet bis zu 100 Batches gleichzeitig, wobei jeder Batch maximal 100 Nachrichten umfasst. Die Transaktionsisolierung erfolgt auf `SERIALIZABLE`-Level, um Race Conditions zu vermeiden.

- **Intelligente Fehlerklassifikation und Eskalation**: Das System unterscheidet zwischen transienten Fehlern (z. B. HTTP 429, 503) und permanenten Fehlern (z. B. HTTP 400, 403). Transiente Fehler werden automatisch wiederholt, während permanente Fehler in eine separate Tabelle (`dead_letter`) mit vollständigem Stack-Trace und Zeitstempel verschoben werden. Ein konfigurierbarer Schwellwert (`max_retries`, Standard: 5) löst eine Alarmierung per Webhook aus.

### So verwenden Sie es

1. **Initialisierung der Datenbank und Konfiguration**: Führen Sie das bereitgestellte SQL-Skript (`schema.sql`) aus, um die Tabellen `retry_queue` und `dead_letter` in Ihrer PostgreSQL-Instanz zu erstellen. Konfigurieren Sie anschließend die Umgebungsvariablen in `.env`: `GMAIL_CLIENT_ID`, `GMAIL_CLIENT_SECRET`, `GMAIL_REFRESH_TOKEN`, `DB_HOST`, `DB_PORT`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`. Starten Sie den Dienst mit `docker-compose up -d`, der automatisch die OAuth2-Token-Rotation und die Verbindung zur Datenbank herstellt.

2. **Manuelles Einfügen von Nachrichten in die Retry-Queue**: Nutzen Sie die REST-API unter `POST /api/enqueue`, um einzelne oder batche von E-Mail-IDs hinzuzufügen. Beispiel-Request: `{"messages": ["msg_id_1", "msg_id_2"], "priority": "high"}`. Die API validiert die IDs gegen die Gmail-API und fügt sie mit einem initialen `retry_count` von 0 und `next_retry_at` = `NOW()` in die Queue ein. Sie erhalten eine JSON-Antwort mit der Anzahl der erfolgreich enqueueten Nachrichten und etwaigen Fehlern.

3. **Überwachung und manuelles Eingreifen**: Greifen Sie auf das integrierte Dashboard unter `http://localhost:8080/dashboard` zu. Hier sehen Sie Echtzeit-Metriken wie `queue_depth` (Anzahl der ausstehenden Nachrichten), `success_rate` (Prozentsatz erfolgreicher Wiederholungen in den letzten 5 Minuten) und `average_retry_delay_ms`. Bei Bedarf können Sie über `POST /api/retry/{message_id}` eine sofortige Wiederholung erzwingen oder über `DELETE /api/queue/{message_id}` eine Nachricht aus der Queue entfernen. Alle Aktionen werden im Log (Log-Level: INFO) mit genauen Zeitstempeln protokolliert.

### Beste Praktiken

- **Implementieren Sie einen separaten Gmail-API-Client für jede OAuth2-Session**: Verwenden Sie für jede parallele Verarbeitungseinheit (Worker) eine eigene Instanz des Gmail-Clients mit einem dedizierten Token. Dadurch vermeiden Sie Token-Konflikte und erhöhen die Gesamtdurchsatzrate um bis zu 40% in Tests mit 50 Workern. Konfigurieren Sie die Token-Rotation mit einem Vorlauf von 5 Minuten vor Ablauf, um `401 Unauthorized`-Fehler zu minimieren.

- **Nutzen Sie die PostgreSQL-Partitionierung für historische Retry-Daten**: Partitionieren Sie die `retry_queue`-Tabelle nach Monaten (`CREATE TABLE retry_queue_2025_01 PARTITION OF retry_queue FOR VALUES FROM ('2025-01-01') TO ('2025-02-01')`). Dies verbessert die Query-Performance für Analysen (z. B. `SELECT COUNT(*) FROM retry_queue WHERE next_retry_at < NOW()`) um das 10- bis 20-fache und erleichtert die Datenarchivierung. Führen Sie regelmäßig `VACUUM ANALYZE` auf den Partitionen durch, um die Indexnutzung zu optimieren.
