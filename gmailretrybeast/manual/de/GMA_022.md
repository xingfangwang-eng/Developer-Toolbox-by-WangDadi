## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist ein hochperformantes, in Rust implementiertes CLI-Tool zur automatisierten, exponentiellen Wiederholung fehlgeschlagener Gmail-API-Aufrufe. Es behandelt spezifisch die HTTP-Statuscodes 429 (Rate Limit) und 5xx (Serverfehler) mit einer granularen, millisekundengenauen Backoff-Strategie. Das Tool persistiert den Zustand der Wiederholungswarteschlange in einer lokalen SQLite-Datenbank, sodass auch nach einem abrupten Abbruch des Prozesses (z. B. System-Crash) kein Eintrag verloren geht.

### Kernfunktionen

1.  **Exponentielles Backoff mit Jitter (Millisekunden-Präzision)**
    - **Parameter:** `--initial-delay 1000` (Startwert in ms), `--max-delay 60000` (maximales Warteintervall), `--backoff-factor 2.0` (Multiplikator), `--jitter-ratio 0.1` (zufällige Streuung ±10%).
    - **Technik:** Implementiert einen vollständigen Fibonacci-Backoff-Algorithmus. Die Wartezeit zwischen zwei Versuchen wird nicht nur verdoppelt, sondern folgt der Fibonacci-Reihe (1, 2, 3, 5, 8, 13... Sekunden), um Thundering-Herd-Probleme bei parallelen API-Aufrufen zu vermeiden.

2.  **Intelligentes Header-Parsing & Retry-After-Respektierung**
    - **Parameter:** `--respect-retry-after true` (Standard: `true`), `--header-policy strict`.
    - **Technik:** Das Tool parst den `Retry-After`-Header (sowohl Sekunden als auch HTTP-Datum) aus der Gmail-API-Antwort. Der Wert aus dem Header überschreibt immer den internen Backoff-Zähler. Zusätzlich wird der `X-RateLimit-Remaining`-Header überwacht. Fällt dieser unter 5, wird eine defensive Pause von mindestens 2 Sekunden eingelegt, bevor der nächste Request gesendet wird.

3.  **Transaktionale Warteschlange mit PostgreSQL-Integration (Optional)**
    - **Parameter:** `--db sqlite:///tmp/retry.db` (Standard) oder `--db postgresql://user:pass@host:5432/retrydb`, `--batch-size 50`.
    - **Technik:** Alle fehlgeschlagenen API-Aufrufe (inklusive vollständiger Payload und Metadaten wie `historyId`) werden in einer ACID-konformen Datenbanktabelle gespeichert. Bei Verwendung von PostgreSQL wird eine `SKIP LOCKED`-Klausel genutzt, um mehrere parallele Instanzen von `gmailretrybeast` sicher zu betreiben, ohne dass es zu doppelten Verarbeitungen kommt.

### So verwenden Sie es

1.  **Initialisierung der Warteschlange aus einer CSV-Datei**
    ```bash
    gmailretrybeast enqueue --csv-file ./failed_messages.csv --csv-format "message_id,user_id,error_code" --batch-size 100
    ```
    *Technische Erklärung:* Das Tool validiert jede Zeile der CSV gegen ein JSON-Schema. Nur Einträge mit einem gültigen `message_id`-Format (Base64-URL-kodiert) und einem `error_code` zwischen 429 und 599 werden in die Queue aufgenommen. Ungültige Zeilen werden in eine separate `error.csv` ausgelagert.

2.  **Starten des Retry-Prozesses mit Live-Monitoring**
    ```bash
    gmailretrybeast process --concurrency 10 --max-retries 5 --timeout 30000 --verbose
    ```
    *Technische Erklärung:* Es werden 10 parallele Worker-Gorouts (Go-Routinen) gestartet, die unabhängig voneinander auf die PostgreSQL-Warteschlange zugreifen. Jeder Worker hat ein hartes Timeout von 30 Sekunden pro API-Call. Das `--verbose`-Flag aktiviert die strukturierte JSON-Logging-Ausgabe auf stdout, die direkt in Logstash oder Datadog eingespeist werden kann.

3.  **Dashboard-Export und Bereinigung**
    ```bash
    gmailretrybeast export --format json --output ./retry_report.json --include-metrics
    ```
    *Technische Erklärung:* Der Export-Befehl aggregiert alle Metriken aus der Datenbank (Anzahl erfolgreicher, fehlgeschlagener, ausstehender Versuche) und gibt ein maschinenlesbares JSON-Objekt aus. Nach erfolgreichem Export kann die Queue mit `gmailretrybeast purge --status success --older-than 7d` bereinigt werden, um die Datenbankgröße unter 500 MB zu halten.

### Beste Praktiken

- **Praxis 1: Defensives Rate-Limiting mit globalen Limits**
  Implementieren Sie ein globales, token-basiertes Rate-Limiting im Wrapper-Skript. Setzen Sie das Limit auf 80% des von Google angegebenen Limits (z. B. 250.000 Requests pro 100 Sekunden für die Gmail API). `gmailretrybeast` sollte als Kindprozess dieses Limiters laufen. Verwenden Sie den Parameter `--max-concurrent-requests 150`, um eine Überlastung des lokalen Netzwerkstacks zu vermeiden.

- **Praxis 2: Einsatz von Dead-Letter-Queues (DLQ) für finale Fehler**
  Konfigurieren Sie einen separaten, asynchronen Webhook-Endpunkt als DLQ. Setzen Sie den Parameter `--dlq-webhook https://hooks.slack.com/services/T00/B00/XXXXX`. Nach dem Erreichen der maximalen Anzahl von Wiederholungsversuchen (`--max-retries 5`) wird der gesamte Payload des fehlgeschlagenen API-Calls inklusive des finalen HTTP-Statuscodes und der Fehlermeldung als JSON-POST an den Webhook gesendet. Dies ermöglicht eine automatisierte Alarmierung in Tools wie PagerDuty.
