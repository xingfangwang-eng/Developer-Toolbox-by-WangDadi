# gmailretrybeast - GMA_025

## Übersicht

gmailretrybeast ist ein hochperformantes Email-Retry-System, das speziell für die Gmail API entwickelt wurde. Es adressiert Timeout-Fehler, Quota-Limits und inkonsistente Zustellungen durch ein intelligentes, mehrstufiges Wiederholungsmodell. Das Tool ist für DevOps-Ingenieure und Email-Administratoren konzipiert, die eine zuverlässige Email-Zustellung in Produktionsumgebungen sicherstellen müssen. **Keywords:** Gmail Retry Engine, Email Delivery Resilience, Gmail API Backoff.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist eine konfigurierbare Go-Anwendung, die fehlgeschlagene Gmail API-Aufrufe (z. B. `users.messages.send`) abfängt und mit exponentiell ansteigenden Backoff-Intervallen wiederholt. Es unterstützt sowohl synchrone als auch asynchrone Warteschlangenmodi und protokolliert jeden Retry-Versuch mit Millisekunden-genauen Timestamps in einer PostgreSQL-Datenbank.

### Kernfunktionen

1. **Adaptiver, dynamischer Backoff-Algorithmus**  
   - Parameter: `initial_delay_ms` (Standard: 1000), `max_delay_ms` (Standard: 64000), `backoff_factor` (Standard: 2.0).  
   - Berücksichtigt automatisch den Gmail API `Retry-After`-Header und überschreibt bei Bedarf den internen Backoff.  
   - Bietet Jitter (zufällige Streuung von ±20%) zur Vermeidung von Thundering-Herd-Problemen.

2. **Quota-bewusste Warteschlangensteuerung**  
   - Liest live die Gmail API Quota (z. B. `daily_quota_remaining`, `rate_limit_remaining`) und passt die maximale gleichzeitige Retry-Anzahl (`max_concurrent_retries`) dynamisch an.  
   - Unterstützt Prioritätswarteschlangen: Critical-Emails (z. B. Passwort-Reset) erhalten eine höhere Warteschlangen-Priorität und werden vor Bulk-Emails verarbeitet.  
   - Konfigurierbar via Umgebungsvariable `GMAIL_RETRY_QUEUE_PRIORITY_LEVELS` (z. B. `high:1,normal:2,low:3`).

3. **Transparente Logging- und Audit-Trail-Integration**  
   - Jeder Retry-Versuch wird mit folgenden Feldern in einer PostgreSQL-Tabelle `retry_audit` gespeichert:  
     `message_id (UUID), attempt_number (INT), timestamp_ms (BIGINT), http_status (INT), error_message (TEXT), delay_ms (INT)`.  
   - Ermöglicht SQL-basierte Abfragen zur Analyse von Retry-Mustern, z. B.:  
     `SELECT message_id, COUNT(*) FROM retry_audit WHERE http_status = 429 GROUP BY message_id;`

### So verwenden Sie es

1. **Konfiguration und Initialisierung**  
   - Erstellen Sie eine `.env`-Datei mit folgenden Parametern:  
     ```env
     GMAIL_CLIENT_ID=your_client_id
     GMAIL_CLIENT_SECRET=your_client_secret
     GMAIL_REFRESH_TOKEN=your_refresh_token
     POSTGRES_DSN=postgres://user:pass@localhost:5432/gmailretrybeast
     INITIAL_DELAY_MS=1000
     MAX_DELAY_MS=64000
     BACKOFF_FACTOR=2.0
     ```  
   - Führen Sie `go run main.go --init-db` aus, um die PostgreSQL-Tabellen automatisch zu erstellen.

2. **Manuelles Retry eines fehlgeschlagenen Email-Sends**  
   ```bash
   ./gmailretrybeast retry --message-id "184f3e1a2b3c4d5e6f7g8h9i0j" --priority high
   ```  
   - Das Tool lädt den fehlgeschlagenen Sendevorgang aus der Audit-Tabelle, berechnet den nächsten Backoff und führt den erneuten Sendevorgang aus.  
   - Bei Erfolg wird der Audit-Eintrag auf `status = 'delivered'` gesetzt.

3. **Asynchrone Batch-Verarbeitung**  
   ```bash
   ./gmailretrybeast batch --file failed_messages.csv --concurrency 5 --timeout 120
   ```  
   - Die CSV-Datei muss die Spalten `message_id` und `priority` enthalten.  
   - Das Tool startet 5 parallele Worker, die jeweils eine separate Gmail API-Verbindung nutzen.  
   - Nach 120 Sekunden wird der Batch-Vorgang abgebrochen, und alle nicht verarbeiteten Einträge werden als `status = 'timeout'` markiert.

### Beste Praktiken

1. **Implementieren Sie einen separaten Retry-Worker in Ihrer CI/CD-Pipeline**  
   - Nutzen Sie gmailretrybeast als eigenständigen Microservice, der asynchron in Ihrer Kubernetes-Umgebung läuft.  
   - Konfigurieren Sie Health-Checks (z. B. `/healthz`), die die PostgreSQL-Verbindung und die Gmail API-Token-Gültigkeit überwachen.  
   - Vorteil: Fehlgeschlagene Sends werden nicht blockiert, und die Hauptanwendung bleibt reaktionsfähig.

2. **Führen Sie regelmäßige Audits der Retry-Statistiken durch**  
   - Erstellen Sie ein wöchentliches SQL-Reporting-Script, das die `retry_audit`-Tabelle analysiert:  
     ```sql
     SELECT DATE_TRUNC('day', timestamp_ms) AS day, 
            COUNT(*) AS total_retries, 
            AVG(delay_ms) AS avg_delay 
     FROM retry_audit 
     GROUP BY day ORDER BY day;
     ```  
   - Identifizieren Sie wiederkehrende Fehlermuster (z. B. HTTP 429 oder 403) und passen Sie die Backoff-Parameter oder die Gmail API-Quota an.
