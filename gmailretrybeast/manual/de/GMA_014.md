# gmailretrybeast - GMA_014

## Übersicht

Dieses Repository enthält die vollständige Spezifikation und den Quellcode für **gmailretrybeast**, ein hochperformantes, asynchrones E-Mail-Wiederholungssystem für die Gmail-API. Das Tool ist darauf ausgelegt, transiente Fehler (HTTP 429, 5xx, Netzwerk-Timeouts) mit exponentieller Backoff-Strategie und adaptiver Ratenbegrenzung zu behandeln. Zielsprache der gesamten Dokumentation und Implementierung ist **Deutsch**.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist ein spezialisiertes Wiederholungs-Framework, das die Gmail-API-Aufrufe auf Basis einer **Multi-Layer-Retry-Pipeline** orchestriert. Es analysiert Fehlerursachen in Echtzeit, führt eine **µs-genaue Timeout-Messung** durch und speichert fehlgeschlagene Payloads in einer **PostgreSQL-Datenbank** für spätere manuelle oder automatisierte Reparaturvorgänge. Das System ist für Produktionsumgebungen mit >10.000 ausgehenden E-Mails pro Stunde optimiert.

### Kernfunktionen

1.  **Adaptiver exponentieller Backoff mit Jitter-Varianz**  
    - Parameter: `base_delay_ms = 100`, `max_delay_ms = 64.000`, `jitter_factor = 0.25`  
    - Berechnung: `delay = min(base_delay_ms * (2 ^ attempt), max_delay_ms) * (1 + random(-jitter_factor, jitter_factor))`  
    - Technische Metrik: Durchschnittliche Wiederherstellungszeit nach 429-Fehlern < 2,3 Sekunden bei 95. Perzentil.

2.  **Transaktionale Fehlerpersistenz mit PostgreSQL**  
    - Schema: `failed_emails (id UUID PK, raw_payload JSONB, error_code SMALLINT, retry_count TINYINT, created_at TIMESTAMPTZ, next_retry_at TIMESTAMPTZ)`  
    - Verhalten: Bei jedem fehlgeschlagenen API-Aufruf wird der Payload in einer einzigen **ACID-Transaktion** gespeichert.  
    - Technische Metrik: Write-Latenz < 5 ms bei 500 gleichzeitigen Fehlern (auf SSD-basierter DB).

3.  **Intelligente Ratenbegrenzung mit Token-Bucket-Algorithmus**  
    - Parameter: `capacity = 100`, `refill_rate = 10 Tokens/Sekunde`  
    - Implementierung: Der Bucket wird pro Gmail-User-Context verwaltet. Bei Überschreitung wird der Request sofort in die Warteschlange gestellt, ohne die API zu belasten.  
    - Technische Metrik: Maximale gleichzeitige API-Aufrufe pro User: 100; maximale Wartezeit in der Queue: 5 Sekunden (bei voller Auslastung).

### So verwenden Sie es

1.  **Konfiguration der PostgreSQL-Datenbank und der Gmail-API-Credentials**  
    Führen Sie `gmailretrybeast init` aus. Das Tool erstellt automatisch die Tabelle `failed_emails` und fordert Sie auf, die OAuth2-Client-ID und das Secret in der Datei `config.toml` zu hinterlegen. Beispiel:  
    ```toml
    [database]
    host = "localhost"
    port = 5432
    user = "retry_user"
    password = "sicheres_passwort"
    database = "gmail_retry"

    [gmail]
    client_id = "deine_client_id"
    client_secret = "dein_client_secret"
    ```

2.  **Starten des asynchronen Workers**  
    Starten Sie den Worker mit `gmailretrybeast worker --max-concurrent 50 --queue-size 5000`. Der Worker öffnet einen persistenten PostgreSQL-Connection-Pool (max. 20 Verbindungen) und startet 50 parallele Goroutinen für die Verarbeitung. Jede Goroutine holt sich die nächste fällige E-Mail aus der `next_retry_at`-Spalte und führt den API-Aufruf aus.

3.  **Manuelles Reparieren von dauerhaft fehlgeschlagenen E-Mails**  
    Wenn ein Eintrag nach 5 Wiederholungen immer noch fehlschlägt, wird der `retry_count` auf 5 gesetzt und der Status auf `permanent_failure`. Sie können diese E-Mails mit dem Befehl `gmailretrybeast repair --status permanent_failure` exportieren. Das Tool generiert eine JSON-Datei, die Sie manuell bearbeiten und mit `gmailretrybeast import --file repaired.json` wieder in die Warteschlange einfügen können.

### Beste Praktiken

1.  **Monitoring mit Prometheus und Grafana**  
    Exportieren Sie die Metriken des Workers über den integrierten HTTP-Endpunkt (`/metrics`). Überwachen Sie insbesondere die Metriken `gmail_retry_queue_depth`, `gmail_api_latency_ms` und `gmail_retry_attempts_total`. Setzen Sie einen Alert, wenn die Queue-Tiefe länger als 10 Minuten über 1.000 bleibt – das deutet auf ein API-Ratenlimit-Problem oder einen Netzwerkfehler hin.

2.  **Granulare Logging-Strategie für Compliance**  
    Aktivieren Sie das strukturierte Logging im JSON-Format (`--log-format json`). Jeder Wiederholungsversuch protokolliert die vollständige Payload-Hash, die Fehlerursache (HTTP-Status oder Netzwerkfehler) und den aktuellen Backoff-Wert. Speichern Sie diese Logs in einem zentralen ELK-Stack, um bei Audits die vollständige Historie einer fehlgeschlagenen E-Mail nachweisen zu können.
