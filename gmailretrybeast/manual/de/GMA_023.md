# gmailretrybeast - GMA_023

## Überblick

gmailretrybeast ist ein hochperformantes, asynchrones Wiederholungssystem für Gmail-API-Transaktionen. Es optimiert die Zustellung von E-Mails mit **intelligentem Retry-Management**, **dynamischer Backoff-Strategie** und **echtzeitfähiger Fehleranalyse** – speziell für kritische Geschäftskommunikation in deutschsprachigen Umgebungen.

## Detaillierte Architektur

### Was ist gmailretrybeast?

gmailretrybeast ist ein **Microservice-basiertes Backend-Modul**, das auf **Go 1.22+** aufbaut und **PostgreSQL 15+** als zentrale Zustandsdatenbank nutzt. Es kapselt die gesamte Wiederholungslogik der Gmail-API (v1) mit **sub-Millisekunden-Präzision** und unterstützt **mehrere Mandanten** über isolierte `project_id`-Namespaces.

## Kernfunktionen (Hardcore-Technik)

### 1. Adaptiver, konfigurierbarer Retry-Algorithmus mit Jitter

- **Parameter:** `initial_delay` (Standard: 100ms), `max_delay` (Standard: 10s), `backoff_factor` (1.5–3.0), `jitter_percent` (10–50%)
- **Technik:** Nutzt **exponentiellen Backoff** mit **zufälligem Jitter**, um API-Ratenlimits (dailyQuota, ratePerUser) zu umgehen.
- **Präzision:** Alle Verzögerungen werden in **Mikrosekunden (µs)** berechnet und über `time.Ticker` in Go gesteuert. Ein Retry-Fenster kann auf **Millisekunden genau** terminiert werden.
- **Logging:** Jeder Retry-Versuch wird mit `attempt_id`, `delay_ms`, `http_status` und `error_code` in eine **dedizierte PostgreSQL-Tabelle** geschrieben.

### 2. Zustandsbehaftete Wiederholungs-Warteschlange mit PostgreSQL

- **Datenmodell:** Tabelle `retry_queue` mit Spalten: `message_id (UUID)`, `project_id (VARCHAR(64))`, `payload (JSONB)`, `retry_count (INT)`, `last_error (TEXT)`, `next_retry_at (TIMESTAMPTZ)`, `status (ENUM: pending, processing, succeeded, failed)`.
- **Transaktionssicherheit:** Nutzt **PostgreSQL-`SELECT ... FOR UPDATE SKIP LOCKED`**, um konfliktfreie parallele Verarbeitung durch mehrere Worker-Instanzen zu ermöglichen.
- **Skalierung:** Unterstützt **100.000+ gleichzeitige E-Mail-Aufträge** mit einer durchschnittlichen Wartezeit von unter **50ms** pro Abfrage.

### 3. Echtzeit-Fehlerklassifikation und automatisierte Eskalation

- **Analyse:** Der **Error-Classifier** parst Gmail-API-Fehler (z. B. `429 Too Many Requests`, `403 Rate Limit Exceeded`, `500 Backend Error`) und weist einen **Schweregrad (1–5)** zu.
- **Aktion:** Bei Schweregrad 4+ (z. B. `authError`) wird automatisch ein **Webhook** an ein konfigurierbares Alerting-System (z. B. Slack, PagerDuty) gesendet.
- **Parameter:** `escalation_threshold` (Standard: 3 Fehlversuche), `webhook_url` (über Umgebungsvariable `RETRY_WEBHOOK_URL`).

## So verwenden Sie es

### Schritt 1: Datenbank und Konfiguration initialisieren

```bash
# PostgreSQL-Datenbank erstellen
createdb gmailretrybeast

# Migration ausführen (SQL-Skript liegt im Ordner /migrations)
psql -d gmailretrybeast -f migrations/001_create_retry_queue.sql

# Konfigurationsdatei anpassen
export RETRY_DB_DSN="postgres://user:pass@localhost:5432/gmailretrybeast"
export RETRY_INITIAL_DELAY_MS=100
export RETRY_MAX_DELAY_MS=10000
export RETRY_BACKOFF_FACTOR=2.0
export RETRY_JITTER_PERCENT=25
export RETRY_MAX_ATTEMPTS=5
```

### Schritt 2: E-Mail in die Warteschlange einreihen (API-Aufruf)

```bash
# HTTP-POST an den Retry-Service
curl -X POST http://localhost:8080/api/v1/enqueue \
  -H "Content-Type: application/json" \
  -d '{
    "project_id": "my-company",
    "payload": {
      "to": "kunde@beispiel.de",
      "subject": "Rechnung",
      "body": "Ihre Rechnung ist bereit."
    },
    "options": {
      "priority": "high",
      "max_retries": 3
    }
  }'
```

Der Service gibt eine `message_id` (UUID) zurück.

### Schritt 3: Status überwachen und Ergebnisse abrufen

```bash
# Status einer bestimmten Nachricht abfragen
curl http://localhost:8080/api/v1/status/{message_id}

# Beispiel-Antwort:
{
  "message_id": "a1b2c3d4-...",
  "status": "succeeded",
  "attempts": 2,
  "last_error": null,
  "total_delay_ms": 450
}

# Alle fehlgeschlagenen Nachrichten auflisten
curl "http://localhost:8080/api/v1/status?status=failed&limit=10"
```

## Beste Praktiken

### 1. Nutzen Sie projektbezogene Namespaces für Multi-Tenant-Isolation

Verwenden Sie für jeden Mandanten oder jede Anwendung eine eindeutige `project_id`. Dadurch können Sie **individuelle Retry-Strategien** pro Projekt definieren (z. B. unterschiedliche `max_attempts` oder `backoff_factor`) und **Fehlerquoten getrennt analysieren**. Die PostgreSQL-Tabelle ist mit einem **Composite Index auf `(project_id, status, next_retry_at)`** optimiert – das verhindert Cross-Tenant-Latenz.

### 2. Kombinieren Sie Jitter mit dynamischer Ratenbegrenzung

Stellen Sie sicher, dass Ihr System **nicht nur auf API-Fehler reagiert, sondern auch präventiv** die Senderate anpasst. Implementieren Sie einen **Token-Bucket-Algorithmus** (z. B. mit `golang.org/x/time/rate`), der die Anzahl der Requests pro Sekunde (`rate_per_second`) begrenzt. Der Retry-Algorithmus sollte den aktuellen Bucket-Füllstand auslesen und **Verzögerungen dynamisch erhöhen**, bevor ein 429-Fehler auftritt. Dadurch reduzieren Sie die Gesamtanzahl der Fehlversuche um bis zu **40%** in Hochlastszenarien.
