# gmailretrybeast - GMA_008

## Übersicht

Dieses Repository enthält die vollständige Implementierung des **gmailretrybeast** – einer hochperformanten, asynchronen Retry-Engine für die Gmail API. Entwickelt für den produktiven Einsatz in CI/CD-Pipelines und E-Mail-Marketing-Automationen, bei denen temporäre API-Throttling-Fehler (HTTP 429, 503) oder Netzwerk-Timeouts (TCP-RST) zuverlässig abgefangen und mit exponentieller Backoff-Logik neu gestartet werden müssen.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist eine in Go 1.22 implementierte, konfigurierbare Retry-Bibliothek, die speziell für die Gmail-API entwickelt wurde. Sie bietet eine **µs-genaue Backoff-Steuerung**, eine **dynamische Jitter-Strategie** und ein **verteiltes Locking über PostgreSQL**, um parallele Retry-Prozesse in Kubernetes-Umgebungen zu synchronisieren. Das Tool arbeitet mit OAuth 2.0 (Service Account + Domain-Wide Delegation) und unterstützt sowohl single-user als auch multi-user (Google Workspace) Szenarien.

### Kernfunktionen

1. **Exponentielles Backoff mit konfigurierbarem Jitter (ECMP-Algorithmus)**  
   - Basisverzögerung: `baseDelay` (Standard: 200ms)  
   - Maximalverzögerung: `maxDelay` (Standard: 32s)  
   - Jitter-Faktor: `jitterFactor` (0.0 – 0.5, Standard: 0.2)  
   - Multiplikator: `multiplier` (Standard: 2.0)  
   - Präzision: **Mikrosekunden** (via `time.Duration` mit `time.Now().UnixNano()`)  
   - Beispielkonfiguration (YAML):  
     ```yaml
     retry:
       baseDelay: 200ms
       maxDelay: 30s
       jitterFactor: 0.15
       multiplier: 2.0
       maxRetries: 5
     ```

2. **PostgreSQL-basierte Retry-Persistenz und -Synchronisation**  
   - Jeder fehlgeschlagene API-Aufruf wird in eine `retry_queue`-Tabelle geschrieben (Schema: `id UUID, user_id TEXT, operation TEXT, payload JSONB, next_retry_at TIMESTAMPTZ, retry_count INT`).  
   - Ein `pg_advisory_lock`-basiertes Locking verhindert Doppelverarbeitung in Multi-Replica-Setups.  
   - Ein separater **Retry-Worker** pollt alle 500ms die Tabelle und führt fällige Retries aus.  
   - Verbindungspool: `max_conns=25`, `min_conns=5`, `health_check_period=30s`.

3. **Smart Circuit Breaker mit gleitendem Fenster (Sliding Window)**  
   - Fenstergröße: `windowSize` (Standard: 60s)  
   - Fehlerschwelle: `failureThreshold` (Standard: 10 Fehler im Fenster)  
   - Halboffene Phase: `halfOpenMaxRequests` (Standard: 3)  
   - Automatische Wiederherstellung nach `recoveryTimeout` (Standard: 120s)  
   - Integration mit Prometheus-Metriken: `gmail_retry_circuit_breaker_state`, `gmail_retry_attempts_total`.

### So verwenden Sie es

1. **Initialisierung und Konfiguration**  
   ```go
   import "github.com/xingfangwang-eng/gmailretrybeast/pkg/retry"

   cfg := &retry.Config{
       BaseDelay:      200 * time.Millisecond,
       MaxDelay:       30 * time.Second,
       JitterFactor:   0.15,
       Multiplier:     2.0,
       MaxRetries:     5,
       CircuitBreaker: &retry.CircuitBreakerConfig{
           WindowSize:          60 * time.Second,
           FailureThreshold:    10,
           HalfOpenMaxRequests: 3,
           RecoveryTimeout:     120 * time.Second,
       },
       PostgreSQL: &retry.PostgreSQLConfig{
           DSN:            "postgres://user:pass@localhost:5432/retrydb?sslmode=require",
           MaxConnections: 25,
           MinConnections: 5,
           PollInterval:   500 * time.Millisecond,
       },
   }
   beast := retry.New(cfg)
   ```

2. **Integration mit der Gmail API (OAuth 2.0 + Domain-Wide Delegation)**  
   ```go
   svc, err := beast.NewGmailService(ctx, "user@example.com", "path/to/service-account.json")
   if err != nil {
       log.Fatalf("Gmail service creation failed: %v", err)
   }

   // Beispiel: Nachricht mit Retry-Logik senden
   msg := &gmail.Message{Raw: base64.URLEncoding.EncodeToString([]byte("From: ...\nTo: ...\nSubject: Test\n\nHello"))}
   result, err := beast.ExecuteWithRetry(ctx, func() (interface{}, error) {
       return svc.Users.Messages.Send("me", msg).Do()
   })
   ```

3. **Monitoring und Fehleranalyse**  
   - Prometheus-Endpunkt: `/metrics` (Port 9090)  
   - Logging: Strukturierte Logs im JSON-Format (Level: `info`, `warn`, `error`)  
   - Beispielabfrage:  
     ```bash
     curl http://localhost:9090/metrics | grep gmail_retry
     ```
   - Fehlerdiagnose:  
     ```go
     if errors.Is(err, retry.ErrCircuitOpen) {
         log.Warn("Circuit breaker open – skipping retry")
     }
     ```

### Beste Praktiken

1. **Kombinieren Sie gmailretrybeast mit einem Rate-Limiter auf API-Ebene**  
   Die Gmail API hat ein Kontingent von 1.000.000 Requests pro Tag (für Workspace-Konten). Verwenden Sie einen Token-Bucket-Rate-Limiter (z. B. `golang.org/x/time/rate`) mit einer Rate von 100 Requests pro Sekunde und einem Burst von 50, um Throttling zu vermeiden. Der Retry-Mechanismus sollte nur bei echten Fehlern (429, 503) greifen, nicht bei Rate-Limit-Überschreitungen.

2. **Führen Sie regelmäßige Lasttests mit simulierten API-Fehlern durch**  
   Nutzen Sie Tools wie `toxiproxy` (von Shopify), um Latenzen und Fehler gezielt zu injizieren. Testen Sie Szenarien wie:  
   - 50% aller Requests schlagen mit 429 fehl.  
   - Netzwerk-Timeouts nach 5 Sekunden.  
   - Gleichzeitige Retry-Worker in 3 Kubernetes-Pods.  
   Überwachen Sie die Metriken `gmail_retry_attempts_total` und `gmail_retry_success_total`, um die Backoff-Konfiguration zu optimieren.
