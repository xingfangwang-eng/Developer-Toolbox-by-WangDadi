# gmailretrybeast - GMA_002

## Übersicht

gmailretrybeast ist ein spezialisiertes Tool für die automatisierte Wiederaufnahme fehlgeschlagener Gmail-Zustellungen. Es optimiert das Retry-Handling auf Basis von SMTP-Statuscodes, DSN-Benachrichtigungen und konfigurierbaren Timeout-Schwellwerten. Ziel ist es, die Zustellrate bei temporären Fehlern (z. B. 4xx‑Codes) drastisch zu erhöhen.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist ein serverseitiger Daemon, der ausgehende Gmail-Nachrichten überwacht und bei fehlgeschlagenen Zustellversuchen automatisch einen intelligenten Retry-Mechanismus auslöst. Das Tool analysiert SMTP‑Antworten, extrahiert DSN‑Diagnosecodes und führt exponentielle Backoff‑Strategien mit Jitter aus. Es ist für den Einsatz in Hochverfügbarkeitsumgebungen konzipiert und unterstützt parallele Retry‑Pipelines.

### Kernfunktionen

1. **Adaptiver Retry‑Manager mit Backoff‑Algorithmus**  
   - Konfigurierbare exponentielle Backoff‑Parameter: `initial_delay_ms = 1000`, `max_delay_ms = 3600000`, `backoff_factor = 2.0`  
   - Jitter‑Modus: „full_jitter“ oder „equal_jitter“  
   - Automatische Erkennung von SMTP‑4xx‑Codes (z. B. 421, 450, 451, 452) und DSN‑Klassifikation (z. B. 4.2.1, 4.7.0)  
   - Maximale Retry‑Anzahl pro Nachricht: `max_retries = 10`  

2. **PostgreSQL‑gestützte Zustandsverwaltung**  
   - Persistenz der Retry‑Queue in PostgreSQL mit Tabellenstruktur: `retry_queue (id, message_id, sender, recipient, smtp_code, dsn_code, retry_count, next_attempt_at, status)`  
   - Index auf `next_attempt_at` und `status` für effiziente Abfragen  
   - Transaktionale Updates mit `SELECT ... FOR UPDATE SKIP LOCKED` zur Vermeidung von Doppelverarbeitung  
   - Automatische Bereinigung alter Einträge nach `cleanup_interval_hours = 72`  

3. **Parallel‑Pipeline mit konfigurierbarem Durchsatz**  
   - Worker‑Pool‑Architektur: `worker_count = 16`, `queue_batch_size = 50`  
   - Jeder Worker öffnet eine eigene SMTP‑Verbindung (via `smtplib` mit TLS‑Unterstützung)  
   - Metriken‑Export über Prometheus: `gmail_retry_attempts_total`, `gmail_retry_success_total`, `gmail_retry_duration_seconds`  
   - Timeout‑Konfiguration: `smtp_timeout_ms = 30000`, `dns_timeout_ms = 5000`  

### So verwenden Sie es

1. **Installation und Konfiguration**  
   ```bash
   git clone https://github.com/xingfangwang-eng/gmailretrybeast.git
   cd gmailretrybeast
   cp config.example.yaml config.yaml
   ```
   Bearbeiten Sie `config.yaml` mit Ihren Gmail‑SMTP‑Credentials (App‑Passwort erforderlich) und den PostgreSQL‑Verbindungsdaten. Setzen Sie `retry_strategy: exponential_backoff` und `jitter_mode: full_jitter`.

2. **Datenbank‑Schema initialisieren**  
   ```bash
   psql -U your_user -d your_db -f schema.sql
   ```
   Das Schema erstellt die Tabelle `retry_queue` sowie einen Index auf `(next_attempt_at, status)`. Optional können Sie `cleanup_trigger.sql` einspielen, um alte Einträge automatisch zu löschen.

3. **Daemon starten und überwachen**  
   ```bash
   python beast_daemon.py --config config.yaml
   ```
   Der Daemon läuft im Vordergrund. Loggen Sie die Ausgabe in eine Datei:  
   ```bash
   nohup python beast_daemon.py --config config.yaml > beast.log 2>&1 &
   ```
   Überwachen Sie die Metriken unter `http://localhost:9090/metrics` (Prometheus‑Endpoint). Prüfen Sie die Queue‑Größe mit:  
   ```sql
   SELECT status, COUNT(*) FROM retry_queue GROUP BY status;
   ```

### Beste Praktiken

- **Praxis 1: Retry‑Limits an die DSN‑Klassifikation koppeln**  
  Verwenden Sie unterschiedliche `max_retries` für verschiedene DSN‑Codes. Beispiel: `4.2.1` (Mailbox full) → 3 Retries, `4.7.0` (Temporary auth failure) → 8 Retries. Dies reduziert unnötige Wiederholungen bei harten Fehlern und erhöht die Erfolgsrate bei temporären Problemen.

- **Praxis 2: Monitoring‑Alarme auf Basis von Retry‑Metriken**  
  Richten Sie Prometheus‑Alarme ein:  
  - `gmail_retry_attempts_total` > 1000 in 5 Minuten → potenzielles SMTP‑Problem  
  - `gmail_retry_success_total` / `gmail_retry_attempts_total` < 0.5 → Backoff‑Strategie prüfen  
  - `gmail_retry_duration_seconds` > 60 (95. Perzentil) → Netzwerk‑ oder DNS‑Latenz
