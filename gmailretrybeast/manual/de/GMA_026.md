# gmailretrybeast - GMA_026

## Überblick

gmailretrybeast ist ein spezialisiertes Retry-Engine-System für Gmail-API-Transaktionen, das fehlgeschlagene E-Mail-Zustellungen mit exponentieller Backoff-Strategie und konfigurierbaren Wiederholungsfenstern automatisiert behandelt. Es optimiert die Zustellungsrate durch adaptive Timeout-Verwaltung und minimiert API-Ratenbegrenzungsfehler.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist eine hochperformante Retry-Orchestrierungs-Engine, die auf die Gmail-API zugeschnitten ist. Sie analysiert fehlgeschlagene Sendevorgänge in Echtzeit, klassifiziert Fehlertypen (z. B. `429 Too Many Requests`, `550 Mailbox Unavailable`) und führt intelligente Wiederholungen mit dynamischen Intervallen durch. Das System unterstützt parallele Retry-Pipelines mit bis zu 500 gleichzeitigen Worker-Instanzen und protokolliert jede Wiederholung mit Millisekunden-genauen Zeitstempeln in einer PostgreSQL-Datenbank.

### Kernfunktionen

1. **Adaptive Retry-Intervall-Berechnung mit Jitter**  
   Implementiert exponentielles Backoff (Basis 2) mit konfigurierbarem Jitter (Standard ±15 %). Parameter: `base_delay = 1000ms`, `max_delay = 120000ms`, `jitter_factor = 0.15`. Berechnung: `delay = min(base_delay * 2^(retry_count-1) * (1 + random(-jitter, jitter)), max_delay)`. Ermöglicht bis zu 10 Wiederholungen pro Transaktion, bevor ein endgültiger Fehlerstatus gesetzt wird.

2. **Fehlerklassifizierung und selektive Wiederholung**  
   Analysiert HTTP-Statuscodes und Gmail-API-Fehlermeldungen mittels eines regelbasierten Klassifikators (YAML-konfigurierbar). Beispielregel:  
   ```yaml
   - error_code: 429
     action: retry
     max_retries: 5
     cooldown: 30000ms
   - error_code: 550
     action: block
     reason: "Mailbox dauerhaft nicht erreichbar"
   ```  
   Nicht wiederholbare Fehler (z. B. `403 Forbidden` wegen fehlender Berechtigungen) werden sofort an die Fehlerwarteschlange übergeben.

3. **PostgreSQL-gestützte Transaktionsprotokollierung und Audit**  
   Jeder Retry-Vorgang wird mit folgenden Feldern in einer `retry_logs`-Tabelle gespeichert:  
   - `transaction_id` (UUID v4)  
   - `email_id` (String, 256 Zeichen)  
   - `retry_attempt` (Integer, 1-10)  
   - `delay_ms` (Integer, Millisekunden)  
   - `status` (Enum: SUCCESS, FAILED, PENDING)  
   - `created_at` (TIMESTAMP WITH TIME ZONE, Millisekunden-Präzision)  
   Ermöglicht granulare Abfragen wie: „Alle Transaktionen mit mehr als 3 Wiederholungen in den letzten 24 Stunden“.

### So verwenden Sie es

1. **Konfiguration der Retry-Parameter in `config.yml`**  
   ```yaml
   retry:
     max_attempts: 10
     base_delay_ms: 1000
     max_delay_ms: 120000
     jitter_percent: 15
     parallel_workers: 500
   database:
     host: localhost
     port: 5432
     dbname: gmail_retry
     user: admin
     password: ****
   ```  
   Passen Sie die Werte an Ihre API-Ratenlimits an (Gmail-API: 250 Queries pro Sekunde pro Benutzer).

2. **Starten der Retry-Engine über die CLI**  
   ```bash
   ./gmailretrybeast --config /etc/gmailretrybeast/config.yml --mode daemon
   ```  
   Der Daemon-Modus startet eine persistente Schleife, die alle 5 Sekunden die Warteschlange auf fehlgeschlagene Sendungen prüft.

3. **Überwachung der Retry-Statistiken via REST-API**  
   ```bash
   curl http://localhost:8080/api/v1/stats?time_range=24h
   ```  
   Antwort (JSON):  
   ```json
   {
     "total_retries": 1234,
     "successful_retries": 1100,
     "failed_retries": 134,
     "avg_delay_ms": 2345.67,
     "p95_delay_ms": 45000
   }
   ```  
   Nutzen Sie diese Daten zur Optimierung der Backoff-Parameter.

### Beste Praktiken

- **Implementieren Sie einen dedizierten PostgreSQL-Pool mit Connection-Pooling** (z. B. pgBouncer) für die Retry-Logs, um Engpässe bei hohen Durchsätzen (>1000 Transaktionen/Minute) zu vermeiden. Dimensionieren Sie die `max_connections` auf mindestens 2x der parallelen Worker-Instanzen.
- **Kombinieren Sie gmailretrybeast mit einem Message-Queue-System** (z. B. RabbitMQ oder Redis Streams) für asynchrone Zustellung. Lagern Sie fehlgeschlagene Sendungen in eine Dead-Letter-Queue aus, um eine manuelle Überprüfung zu ermöglichen, und konfigurieren Sie Alarme (z. B. via Prometheus) bei Überschreitung der maximalen Retry-Versuche.
