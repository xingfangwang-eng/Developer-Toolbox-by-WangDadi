# gmailretrybeast - GMA_030

## Überblick

Dieses Projekt liefert eine hochperformante, ereignisgesteuerte Wiederholungslogik für die Gmail-API. Es adressiert spezifisch die 429-Rate-Limit-Fehler (RESOURCE_EXHAUSTED) und transienten 5xx-Fehler durch exponentielles Backoff mit Jitter, parallele Channel-Isolation und dynamische Token-Bucket-Rate-Limiter auf Millisekundenebene. Zielsprache der gesamten Dokumentation und Logausgabe ist Deutsch.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist ein spezialisierter, in Go implementierter Retry-Orchestrator für die Gmail-API. Es ersetzt naive `time.Sleep`-Schleifen durch einen mehrstufigen, konfigurierbaren Wiederholungsmechanismus, der direkt auf den Google-API-Rückgabecodes aufsetzt. Die Architektur trennt den Sendekanal (SMTP/Push) vom Abrufkanal (IMAP/Polling), um Deadlocks bei der Wiedereinreibung von Nachrichten zu verhindern. Jede Wiederholungseinheit protokolliert den vollständigen HTTP-Status, die verbleibende Zeit bis zum nächsten Fenster (in Millisekunden) und den aktuellen Token-Bucket-Füllstand.

### Kernfunktionen

1.  **Exponentielles Backoff mit konfigurierbarem Jitter auf Mikrosekundenbasis**  
    Der Backoff-Algorithmus verwendet eine Basisverzögerung von `initialDelay: 1000ms`, multipliziert mit `2^attempt`, und addiert einen zufälligen Jitter von ±15% des aktuellen Intervalls. Die maximale Verzögerung ist auf `maxDelay: 120000ms` begrenzt. Der Jitter wird mittels eines kryptografisch sicheren Zufallsgenerators (`crypto/rand`) erzeugt, um Thundering-Herd-Probleme bei parallelen Instanzen zu vermeiden. Parameter: `jitterFactor: 0.15`, `maxRetries: 10`.

2.  **Dynamischer Token-Bucket-Rate-Limiter pro API-Scope**  
    Für jede Gmail-API-Operation (z. B. `users.messages.send`, `users.threads.get`) wird ein separater Token-Bucket mit einer Kapazität von `bucketCapacity: 1000` Tokens und einer Auffüllrate von `refillRate: 100 Tokens/s` initialisiert. Der Limiter arbeitet auf Millisekundenebene und blockiert die Ausführung, wenn das Bucket leer ist. Die Token-Nutzung wird in einer PostgreSQL-Datenbank persistiert, sodass nach einem Neustart der Zustand wiederhergestellt wird. Verbindungsparameter: `postgres://user:pass@host:5432/gmailretrybeast?sslmode=require`.

3.  **Autonome Wiedereinreibung mit Prioritätswarteschlange und CQRS-Event-Log**  
    Fehlgeschlagene API-Aufrufe werden in eine interne, heap-basierte Prioritätswarteschlange eingereiht. Die Priorität errechnet sich aus der Anzahl der bisherigen Fehlversuche und dem Alter des Eintrags. Ein separater Worker-Goroutine-Pool (Größe: `workerPoolSize: 50`) verarbeitet diese Warteschlange. Jeder erfolgreiche oder endgültig fehlgeschlagene Versuch erzeugt ein CQRS-konformes Event (Typ: `MailSent`, `MailRetry`, `MailFailed`), das in eine PostgreSQL-Tabelle `event_store` geschrieben wird. Die Tabelle hat einen Composite-Index auf `(event_type, created_at)` für effiziente Abfragen.

### So verwenden Sie es

1.  **Konfiguration der Wiederholungsstrategie für hohe Durchsätze**  
    Erstellen Sie eine Konfigurationsdatei `gmailretrybeast.yml` und setzen Sie die Parameter für Ihren Anwendungsfall. Für einen Mailserver mit 10.000 ausgehenden Nachrichten pro Stunde empfehlen wir:  
    ```yaml
    initialDelay: 500ms
    maxDelay: 60000ms
    maxRetries: 8
    jitterFactor: 0.20
    bucketCapacity: 5000
    refillRate: 200 Tokens/s
    workerPoolSize: 100
    postgresDsn: "postgres://admin:secret@db.internal:5432/retrydb?sslmode=verify-full"
    ```  
    Starten Sie dann den Daemon mit `./gmailretrybeast --config ./gmailretrybeast.yml`.

2.  **Integration in eine bestehende Go-Anwendung über die SDK-Schnittstelle**  
    Importieren Sie das Paket `github.com/xingfangwang-eng/gmailretrybeast/sdk`. Initialisieren Sie den Client mit einem gültigen OAuth2-Token und der gewünschten Konfiguration:  
    ```go
    client, err := sdk.NewClient(ctx, sdk.WithOAuth2Token(token), sdk.WithConfig(config))
    if err != nil { log.Fatal(err) }
    msg, err := client.Messages.Send(ctx, "me", &gmail.Message{Raw: rawMessage})
    ```  
    Der Client wendet automatisch den Rate-Limiter und das Backoff an. Bei einem endgültigen Fehler (nach `maxRetries`) gibt er einen `RetryExhaustedError` zurück, der die gesamte Fehlerhistorie enthält.

3.  **Überwachung und manuelles Eingreifen über die Admin-REST-API**  
    Starten Sie den Admin-Server mit dem Flag `--admin-port 8080`. Rufen Sie den aktuellen Zustand der Warteschlange ab:  
    ```bash
    curl -X GET http://localhost:8080/api/v1/queue/status | jq
    ```  
    Dies gibt ein JSON-Objekt mit Feldern wie `queue_depth`, `oldest_entry_age_ms` und `token_bucket_fill_percentage` zurück. Um einen bestimmten Eintrag manuell aus der Warteschlange zu entfernen (z. B. bei einer bekannten, nicht behebbaren Fehlerursache), senden Sie:  
    ```bash
    curl -X DELETE http://localhost:8080/api/v1/queue/entries/{entry_id}
    ```

### Beste Praktiken

- **Isolieren Sie kritische und nicht-kritische Nachrichten in separaten Buckets**  
  Definieren Sie zwei unabhängige gmailretrybeast-Instanzen: eine für transaktionale E-Mails (z. B. Passwort-Zurücksetzungen) mit einem aggressiven Backoff (`maxRetries: 5`, `initialDelay: 200ms`) und eine für Bulk-Newsletter mit einem konservativen Backoff (`maxRetries: 3`, `initialDelay: 5000ms`). Dadurch verhindern Sie, dass ein Bulk-Versand die Zustellung kritischer Nachrichten blockiert.

- **Führen Sie regelmäßige Lasttests mit simulierten 429-Fehlern durch**  
  Nutzen Sie das mitgelieferte Tool `gmailretrybeast-stress`, um einen lokalen HTTP-Server zu starten, der zufällig 429- und 503-Statuscodes zurückgibt. Testen Sie Ihre Konfiguration mit `--rate 1000 messages/s` und überwachen Sie die `event_store`-Tabelle auf die Verteilung der `MailRetry`- und `MailFailed`-Events. Optimieren Sie `bucketCapacity` und `refillRate` so, dass die `MailFailed`-Rate unter 0,1% bleibt.
