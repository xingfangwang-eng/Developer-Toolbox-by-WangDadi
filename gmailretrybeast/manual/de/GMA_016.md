# gmailretrybeast - GMA_016

## Überblick

gmailretrybeast ist eine hochperformante, asynchrone Wiederholungs-Engine für fehlgeschlagene Gmail-API-Aufrufe. Das System optimiert den E-Mail-Versand unter Last, indem es exponentielle Backoff-Strategien mit dynamischer Priorisierung kombiniert. Zielgruppe sind DevOps-Ingenieure und E-Mail-Infrastruktur-Teams, die täglich >100.000 Transaktions-E-Mails verarbeiten.

## Technische Kernfunktionen

### 1. Dynamischer exponentieller Backoff mit Jitter + Quota-Awareness

Das System implementiert einen **erweiterten exponentiellen Backoff** (Basis: 2^RetryCount * 100 ms), der zusätzlich einen **zufälligen Jitter** (±15% des aktuellen Werts) einstreut, um Thundering-Herd-Probleme zu vermeiden. Kritisch: Die Backoff-Dauer wird automatisch an die **Gmail-API-Rate-Limits** angepasst (Standard: 250 Anfragen pro Benutzer pro Sekunde). Ein eingebauter **Quota-Consumer** liest den `X-RateLimit-Remaining`-Header aus und pausiert die Warteschlange, sobald das verbleibende Kontingent unter 10% fällt.

- **Parameter:** `initial_delay_ms=100`, `max_delay_ms=60000`, `jitter_percent=15`
- **Technik:** Verwendung von `asyncio.sleep()` mit `random.uniform()` für präzise, nicht blockierende Verzögerungen im Millisekundenbereich.

### 2. Multi-Stage Retry mit konfigurierbaren Fehlertyp-Filtern

Nicht jeder Fehler erfordert einen Wiederholungsversuch. Das Tool unterscheidet zwischen **transienten Fehlern** (z. B. `429 Too Many Requests`, `500 Internal Server Error`, `503 Service Unavailable`) und **permanenten Fehlern** (z. B. `400 Bad Request` wegen ungültiger E-Mail-Adresse). Für jeden Fehlertyp kann eine eigene **maximale Wiederholungsanzahl** und ein **Backoff-Profil** definiert werden. Transiente Fehler erhalten bis zu 5 Versuche, permanente Fehler werden sofort in eine separate **Dead-Letter-Queue** (DLQ) verschoben.

- **Parameter:** `retry_on_http_codes=[429, 500, 503]`, `max_retries_transient=5`, `dead_letter_on=[400, 403, 404]`
- **Integration:** Die DLQ wird in einer **PostgreSQL-Tabelle** (`gmail_retry_beast.dlq`) persistiert, inklusive vollständigem Request-Body und Stack-Trace.

### 3. Asynchrone Batch-Verarbeitung mit parallelen Channels

Um die Gmail-API effizient auszulasten, gruppiert das System E-Mails in **konfigurierbare Batches** (Standard: 50 E-Mails pro Batch). Jeder Batch wird über einen eigenen **asyncio-Channel** verarbeitet, wobei maximal 5 Channels parallel laufen. Innerhalb eines Channels werden die API-Aufrufe strikt sequenziell ausgeführt, um Rate-Limits nicht zu überschreiten. Ein **globaler Semaphor** (Limit: 10) verhindert, dass zu viele Channels gleichzeitig aktiv sind.

- **Parameter:** `batch_size=50`, `max_concurrent_channels=5`, `global_semaphore_limit=10`
- **Leistungsindikator:** Durchsatz von bis zu 250 E-Mails pro Sekunde bei einem P99-Latenz von <200 ms pro API-Aufruf.

## Detaillierte Verwendung

### Schritt 1: Konfiguration der Wiederholungsstrategie

Erstellen Sie eine `retry_config.yaml` mit spezifischen Parametern für Ihre Umgebung. Definieren Sie Backoff-Profile für verschiedene Fehlertypen und legen Sie die DLQ-Verbindung zur PostgreSQL-Datenbank fest.

```yaml
retry:
  transient:
    max_retries: 5
    backoff_base_ms: 100
    jitter_percent: 15
  permanent:
    max_retries: 0
    dead_letter: true
database:
  host: localhost
  port: 5432
  dbname: gmail_retry
  table: dlq
```

### Schritt 2: Initialisieren der asynchronen Engine

Starten Sie die Engine mit einem Python-Skript, das die Konfiguration lädt und eine Warteschlange aus einer CSV-Datei oder einer Datenbank-Tabelle befüllt. Die Engine erstellt automatisch die benötigten Channels und startet die Verarbeitung.

```python
from gmailretrybeast import RetryEngine
engine = RetryEngine(config_path='retry_config.yaml')
engine.load_queue('failed_emails.csv')
await engine.run()
```

### Schritt 3: Überwachung und manuelles Eingreifen

Während der Laufzeit gibt die Engine alle 5 Sekunden einen **Status-Report** auf der Konsole aus: Anzahl verarbeiteter E-Mails, aktuelle Backoff-Dauer, Anzahl der Fehler pro Typ. Verwenden Sie das integrierte **CLI-Tool**, um die DLQ manuell zu inspizieren und fehlgeschlagene E-Mails erneut in die Warteschlange zu stellen.

```bash
gmailretrybeast-cli --list-dlq --limit 10
gmailretrybeast-cli --retry-dlq --id 12345
```

## Best Practices

### Praxis 1: Implementieren Sie eine separate Monitoring-Datenbank

Lagern Sie alle Metriken (Anzahl Wiederholungen, Fehlertypen, Latenz) in eine **separate PostgreSQL-Instanz** aus, die nicht für die DLQ genutzt wird. Verwenden Sie **Prometheus-Exporteure**, um die Daten in Echtzeit zu visualisieren. So vermeiden Sie, dass die DLQ-Schreibvorgänge die Leistung der Hauptdatenbank beeinträchtigen.

### Praxis 2: Nutzen Sie dynamische Rate-Limit-Anpassung

Statt feste Backoff-Werte zu verwenden, lesen Sie den `Retry-After`-Header der Gmail-API aus und überschreiben Sie den aktuellen Backoff-Wert dynamisch. Dies reduziert die Anzahl unnötiger Wiederholungen um bis zu 40% und senkt die API-Kosten signifikant.
