# gmailretrybeast - GMA_020

## Übersicht

gmailretrybeast ist ein spezialisiertes Retry-Engine-System für Gmail-API-Transaktionen, das auf maximale Zustellgarantie und Fehlertoleranz ausgelegt ist. Es optimiert den Umgang mit temporären API-Fehlern (HTTP 429, 500, 503) durch exponentielles Backoff, intelligente Warteschlangenverwaltung und granulare Log-Analyse.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist eine in Go implementierte, ereignisgesteuerte Retry-Middleware, die direkt an die Gmail-API angebunden wird. Sie verarbeitet fehlgeschlagene E-Mail-Transaktionen (Senden, Labeln, Archivieren) mit einer konfigurierbaren Wiederholungsstrategie, unterstützt PostgreSQL als persistenten Retry-Queue-Speicher und liefert Millisekunden-genaue Metriken zu jedem Retry-Versuch. Das System eignet sich besonders für Hochvolumen-E-Mail-Kampagnen, automatisierte Benachrichtigungssysteme und CI/CD-Pipelines, bei denen eine 100%ige Zustellrate kritisch ist.

### Kernfunktionen

1. **Intelligentes exponentielles Backoff mit Jitter**
   - Parameter: `initialDelay: 500ms`, `maxDelay: 120s`, `multiplier: 2.0`, `jitterFactor: 0.25`
   - Berechnet automatisch die optimale Wartezeit zwischen Retry-Versuchen basierend auf dem Gmail-API-Rate-Limit (max. 250 Anfragen pro Benutzer pro Sekunde).
   - Unterstützt dynamische Anpassung bei HTTP-429-Responses: extrahiert `Retry-After`-Header und priorisiert diesen über den internen Algorithmus.

2. **Persistente Retry-Queue mit PostgreSQL**
   - Speichert jede fehlgeschlagene Transaktion in einer PostgreSQL-Tabelle (`retry_queue`) mit den Feldern: `id (UUID)`, `email_id (VARCHAR(255))`, `operation (ENUM: 'send', 'label', 'archive')`, `retry_count (INT DEFAULT 0)`, `max_retries (INT DEFAULT 5)`, `next_retry_at (TIMESTAMPTZ)`, `payload (JSONB)`, `error_message (TEXT)`, `created_at (TIMESTAMPTZ DEFAULT NOW())`.
   - Führt eine `SELECT ... FOR UPDATE SKIP LOCKED`-Abfrage alle 10 Sekunden aus, um fällige Retry-Jobs atomar zu verarbeiten – garantiert keine doppelten Verarbeitungen bei parallelen Instanzen.

3. **Granulare Metrik-Erfassung und Alerting**
   - Sammelt pro Retry-Versuch: `attempt_duration_ms`, `http_status_code`, `retry_number`, `queue_latency_ms`.
   - Exportiert Metriken über Prometheus-Endpunkt (`/metrics`) mit Labels wie `operation`, `status_code`, `retry_count`.
   - Löst automatisch ein Alert-Event aus, wenn eine Transaktion nach 5 Versuchen endgültig fehlschlägt – inklusive vollständigem Payload und Stack-Trace für Debugging.

### So verwenden Sie es

1. **Installation und Konfiguration der PostgreSQL-Datenbank**
   ```bash
   # PostgreSQL-Container starten (Port 5432, Datenbank: gmailretrybeast)
   docker run -d --name pg-retry -e POSTGRES_DB=gmailretrybeast -e POSTGRES_USER=retryuser -e POSTGRES_PASSWORD=retrypass -p 5432:5432 postgres:15
   # Tabelle initialisieren
   psql -h localhost -U retryuser -d gmailretrybeast -c "
   CREATE TABLE retry_queue (
       id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
       email_id VARCHAR(255) NOT NULL,
       operation VARCHAR(20) NOT NULL CHECK (operation IN ('send', 'label', 'archive')),
       retry_count INT DEFAULT 0,
       max_retries INT DEFAULT 5,
       next_retry_at TIMESTAMPTZ DEFAULT NOW(),
       payload JSONB,
       error_message TEXT,
       created_at TIMESTAMPTZ DEFAULT NOW()
   );
   CREATE INDEX idx_next_retry_at ON retry_queue (next_retry_at) WHERE retry_count < max_retries;
   "
   ```

2. **Integration in Ihre Go-Anwendung**
   ```go
   package main
   import (
       "github.com/xingfangwang-eng/gmailretrybeast"
       "time"
   )
   func main() {
       config := gmailretrybeast.Config{
           DBHost: "localhost",
           DBPort: 5432,
           DBUser: "retryuser",
           DBPass: "retrypass",
           DBName: "gmailretrybeast",
           InitialDelay: 500 * time.Millisecond,
           MaxDelay:     120 * time.Second,
           Multiplier:   2.0,
           JitterFactor: 0.25,
           MaxRetries:   5,
           QueuePollInterval: 10 * time.Second,
       }
       beast := gmailretrybeast.New(config)
       // Beispiel: Senden einer E-Mail mit automatischem Retry
       err := beast.SendWithRetry("user@gmail.com", "Betreff", "Nachricht", nil)
       if err != nil {
           log.Fatalf("Finaler Fehler nach Retry: %v", err)
       }
   }
   ```

3. **Überwachung und manuelles Eingreifen**
   ```sql
   -- Aktuelle Retry-Queue anzeigen (alle ausstehenden Jobs)
   SELECT id, email_id, operation, retry_count, next_retry_at, error_message
   FROM retry_queue
   WHERE retry_count < max_retries
   ORDER BY next_retry_at ASC;
   
   -- Job manuell zurücksetzen (sofortiger erneuter Versuch)
   UPDATE retry_queue
   SET retry_count = 0, next_retry_at = NOW()
   WHERE id = 'your-uuid-here';
   
   -- Fehlgeschlagene Jobs endgültig löschen
   DELETE FROM retry_queue
   WHERE retry_count >= max_retries AND next_retry_at < NOW() - INTERVAL '7 days';
   ```

### Beste Praktiken

- **Praxis 1: Rate-Limiting-Respektierung und dynamische Anpassung**  
  Konfigurieren Sie `maxRetries` in Abhängigkeit vom Gmail-API-Kontingent. Für Standard-Gmail-Konten (250 Anfragen/s) setzen Sie `maxRetries: 3` mit `initialDelay: 1s`. Für Workspace-Konten mit höheren Limits (bis zu 1.000 Anfragen/s) können Sie `maxRetries: 5` und `initialDelay: 500ms` verwenden. Überwachen Sie stets die `X-RateLimit-Remaining`-Header und reduzieren Sie die Queue-Poll-Intervalle dynamisch bei Annäherung an das Limit.

- **Praxis 2: Payload-Kompression und Deduplizierung**  
  Nutzen Sie die `payload`-Spalte (JSONB) in PostgreSQL, um den vollständigen E-Mail-Payload zu speichern. Aktivieren Sie die optionale Deduplizierung: Vor dem Einreihen eines neuen Jobs prüft gmailretrybeast, ob eine identische `email_id` + `operation`-Kombination bereits in der Queue existiert. Dies verhindert doppelte Retry-Versuche bei Netzwerk-Timeouts oder versehentlichen Mehrfachaufrufen. Der Schwellwert für die Deduplizierung ist standardmäßig auf 60 Sekunden eingestellt (konfigurierbar via `dedupWindow`).
