# gmailretrybeast - GMA_001

## Übersicht

Dieses Projekt optimiert die Zustellung und Wiederholungslogik von E-Mails über die Gmail API. Fokus liegt auf robuster Fehlerbehandlung, automatischem Retry mit exponentiellem Backoff und detailliertem Logging für die Fehlersuche. Keywords: Gmail Retry Tool, Email Retry System, Gmail E-Mail Management.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist ein professionelles Tool, das speziell für die Wiederholung fehlgeschlagener Gmail-API-Aufrufe entwickelt wurde. Es behandelt systematisch Timeouts, Ratenbegrenzungen (HTTP 429) und temporäre Serverfehler (5xx). Das Tool implementiert einen konfigurierbaren, mehrstufigen Retry-Mechanismus mit Jitter und speichert den Zustand jeder Wiederholung in einer lokalen SQLite-Datenbank (optional PostgreSQL). Dadurch wird sichergestellt, dass keine E-Mail im Retry-Zyklus verloren geht und die API-Quote nicht überschritten wird.

### Kernfunktionen

1.  **Adaptives exponentielles Backoff mit Jitter**
    -   Automatische Berechnung der Wartezeit zwischen Wiederholungen: `min(300.000 ms, (2^RetryCount * 1.000 ms) + random(0 – 1.000 ms))`.
    -   Konfigurierbare maximale Anzahl von Wiederholungen (`maxRetries: 10`).
    -   Unterstützt benutzerdefinierte HTTP-Statuscodes, die einen Retry auslösen (z. B. `429`, `500`, `502`, `503`).

2.  **Zustandsbehaftetes Retry-Logging und -Persistenz**
    -   Jeder API-Aufruf wird mit einer eindeutigen `correlationId` (UUID v4) versehen.
    -   Details wie `attemptNumber`, `responseTimeMs`, `httpStatusCode` und `errorMessage` werden in einer `retry_logs`-Tabelle gespeichert.
    -   Bei Abbruch nach allen Versuchen wird der vollständige Fehlerbericht als JSON-Datei im Verzeichnis `./failed_deliveries/` abgelegt.

3.  **Intelligente Gmail-API-Quote-Verwaltung**
    -   Überwachung des verbleibenden API-Kontingents in Echtzeit mittels `X-Google-Usage`-Header.
    -   Dynamische Reduzierung der parallelen Anfragen (`maxConcurrency: 5`) bei Erreichen von 80 % der täglichen Quote.
    -   Implementierung eines internen Token-Bucket-Algorithmus mit einer Rate von `10 tokens/second` und einer maximalen Bucket-Größe von `50 tokens`.

### So verwenden Sie es

1.  **Konfiguration und Initialisierung**
    -   Klonen Sie das Repository und installieren Sie die Abhängigkeiten: `pip install -r requirements.txt`.
    -   Erstellen Sie eine `config.yaml` mit den folgenden Parametern:
        ```yaml
        gmail:
          credentials_path: "./credentials.json"
          token_path: "./token.json"
        retry:
          max_retries: 10
          base_delay_ms: 1000
          max_delay_ms: 300000
          jitter: true
          retry_on_status: [429, 500, 502, 503]
        logging:
          database: "sqlite:///retry_beast.db"
          failed_dir: "./failed_deliveries"
        ```
    -   Authentifizieren Sie sich mit einem Gmail-Dienstkonto oder OAuth 2.0.

2.  **Ausführen eines Retry-Zyklus für eine einzelne Nachricht**
    -   Starten Sie das Tool mit der Nachrichten-ID und dem gewünschten Action-Typ:
        ```bash
        python beast.py --message-id "18a3f5b7c9d1e2f3" --action "send"
        ```
    -   Das Tool führt die Aktion aus und wiederholt sie gemäß der Konfiguration. Der Fortschritt wird in Echtzeit in der Konsole angezeigt:
        ```text
        [2025-03-21 10:00:01] correlationId: a1b2c3d4-e5f6-7890-abcd-ef1234567890
        [2025-03-21 10:00:01] Attempt 1/10 – Status: 429 – Backoff für 2.341 ms...
        [2025-03-21 10:00:04] Attempt 2/10 – Status: 200 – Erfolg.
        ```

3.  **Batch-Verarbeitung und Berichterstellung**
    -   Verwenden Sie den Batch-Modus für mehrere Nachrichten aus einer CSV-Datei:
        ```bash
        python beast.py --batch-file "./outgoing_messages.csv" --action "send"
        ```
    -   Nach Abschluss generiert das Tool einen detaillierten Bericht (`report.json`):
        ```json
        {
          "total_messages": 150,
          "successful": 148,
          "failed_after_retries": 2,
          "average_response_time_ms": 345,
          "total_api_calls": 178,
          "retry_rate_percent": 12.0
        }
        ```

### Beste Praktiken

1.  **Ratenbegrenzung pro Benutzer einhalten**
    -   Konfigurieren Sie `maxConcurrency` und `tokensPerSecond` basierend auf dem Gmail-API-Kontingent Ihres Projekts (Standard: 1.000.000 Einheiten pro Tag, 250 Einheiten pro Benutzer pro Sekunde). Nutzen Sie den integrierten Token-Bucket, um plötzliche Spitzen zu glätten und eine Sperrung des Kontos zu vermeiden.

2.  **Fehlgeschlagene Zustellungen strukturiert analysieren**
    -   Aktivieren Sie die Persistenz in einer PostgreSQL-Datenbank, um langfristige Trends zu erkennen. Führen Sie regelmäßig Analysen der `retry_logs`-Tabelle durch, um häufige Fehlercodes (`errorMessage`) zu identifizieren und die `retry_on_status`-Liste entsprechend anzupassen. Archivieren Sie das `failed_deliveries`-Verzeichnis wöchentlich und überprüfen Sie die JSON-Logs auf wiederkehrende Muster (z. B. immer derselbe Empfänger oder Anhangstyp).
