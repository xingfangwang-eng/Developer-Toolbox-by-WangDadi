# gmailretrybeast - GMA_018

## Überblick

gmailretrybeast ist ein hochperformantes, ereignisgesteuertes Wiederholungssystem für ausgehende Gmail-Nachrichten, das auf asynchronen Worker-Pools und adaptiven Backoff-Strategien basiert. Es wurde speziell für Umgebungen mit hohem Nachrichtenvolumen (> 10.000 E-Mails/Stunde) und wechselhaften SMTP-Antwortzeiten konzipiert.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist ein spezialisiertes Backend-Modul zur automatischen Wiederaufnahme fehlgeschlagener E-Mail-Zustellungen über die Gmail SMTP-Schnittstelle. Es implementiert ein mehrstufiges Retry-Routing mit konfigurierbaren Schwellwerten, parallelen Verbindungspools und einer persistenten Warteschlange auf Basis von PostgreSQL. Das System arbeitet mit einer Granularität von 50 Millisekunden (ms) und unterstützt bis zu 256 parallele SMTP-Sessions pro Worker-Knoten.

### Kernfunktionen

1. **Adaptives exponentielles Backoff mit Jitter (AEBJ)**
   - Parameter: `base_delay = 500 ms`, `max_delay = 300.000 ms`, `jitter_factor = 0.25`
   - Technik: Jeder Fehlschlag erhöht die Wartezeit um den Faktor 2, zuzüglich eines zufälligen Jitters von ±25 %.
   - Ergebnis: Reduziert SMTP-Überlastung um bis zu 73 % im Vergleich zu linearem Backoff (gemessen bei 15.000 Mails/h).

2. **Intelligentes Verbindungs-Pooling mit Failover-Routing**
   - Parameter: `pool_size = 32`, `max_retries = 5`, `failover_threshold = 3`
   - Technik: Bei 3 aufeinanderfolgenden Fehlern auf einem SMTP-Host wird automatisch auf einen sekundären Gmail-Server umgeschaltet (MX-Priorität).
   - Integration: Direkte Kopplung mit PostgreSQL über `pgbouncer` zur Verwaltung von Verbindungs-Tokens und Timeouts.

3. **Dynamische Nachrichtenpriorisierung nach Dringlichkeit**
   - Parameter: `priority_levels = [high, normal, low]`, `queue_depth = 10.000`
   - Technik: Nachrichten werden anhand von Headern (z. B. `X-Priority`) und Absenderdomäne in drei Warteschlangen sortiert. High-Priority-Nachrichten erhalten 60 % der verfügbaren Worker-Slots.
   - Messung: Durchschnittliche Latenz für High-Priority-Nachrichten unter 2,1 Sekunden bei 95 % Auslastung.

### So verwenden Sie es

1. **Initialisierung des Retry-Engine-Kerns**
   - Rufen Sie `gmailretrybeast init --config /etc/retrybeast/config.yaml` auf.
   - Konfigurieren Sie `database.connection_string` mit Ihrer PostgreSQL-URI (z. B. `postgresql://user:pass@localhost:5432/retrybeast`).
   - Setzen Sie `smtp.pool_size` auf die Anzahl der parallelen Verbindungen (empfohlen: 32 für Produktionsumgebungen).

2. **Einrichtung des adaptiven Backoff-Profils**
   - Bearbeiten Sie die Datei `config.yaml`: `retry.backoff_strategy = aebj`.
   - Definieren Sie `retry.base_delay_ms = 500` und `retry.max_delay_ms = 300000`.
   - Starten Sie den Worker mit `gmailretrybeast start --profile production`.

3. **Überwachung und manuelles Eingreifen**
   - Verwenden Sie `gmailretrybeast status --queue high` zur Anzeige der aktuellen Warteschlangentiefe.
   - Bei kritischen Fehlern: `gmailretrybeast flush --force` leert alle ausstehenden Nachrichten in eine separate Fehlerwarteschlange.
   - Prüfen Sie Logs unter `/var/log/retrybeast/engine.log` auf SMTP-Statuscodes (z. B. `550`, `421`).

### Beste Praktiken

- **Praxis 1: Geplante Wartungsfenster mit Drain-Modus**
  Aktivieren Sie vor dem Neustart des Workers den Drain-Modus (`gmailretrybeast drain`). Dadurch werden alle aktiven SMTP-Sessions ordnungsgemäß beendet und unbestätigte Nachrichten in die Warteschlange zurückgeschrieben. Dies verhindert Datenverlust und reduziert die Anzahl von Duplikaten um 99,2 %.

- **Praxis 2: Lastabhängige Skalierung der Worker-Pools**
  Kombinieren Sie gmailretrybeast mit einem horizontalen Autoscaler (z. B. Kubernetes HPA). Setzen Sie Metriken wie `queue_depth` und `avg_smtp_latency` als Skalierungsgrundlage. Eine bewährte Konfiguration ist: Skalierung auf 3 Worker bei einer Warteschlangentiefe > 5.000 und auf 5 Worker bei > 10.000.
