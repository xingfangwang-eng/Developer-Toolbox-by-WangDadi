# gmailretrybeast - GMA_011

## Übersicht

Dieses Repository enthält die vollständige Spezifikation und Implementierung des **gmailretrybeast**. Das System ist ein hochperformantes, asynchrones Retry-Framework für die Gmail API, das speziell für die Bewältigung von transienten Fehlern (HTTP 429, 503, 500) und Quota-Limits entwickelt wurde. Kernziel ist die Maximierung der Zustellrate bei gleichzeitiger Minimierung von Latenz und API-Kosten.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist ein in Go implementiertes, zustandsbehaftetes Retry-Orchestrierungssystem. Es analysiert Gmail API-Fehlerantworten in Echtzeit, extrahiert `Retry-After`-Header und implementiert einen **exponentiellen Backoff mit Jitter** (Bereich: ±15 % des Basiswerts). Das System persistiert den Retry-Status in einer **PostgreSQL-Datenbank** (Tabellenschema `retry_beast.queue`) und ermöglicht so eine Wiederaufnahme nach Prozessneustarts mit einer maximalen Abweichung von **±50 ms** gegenüber dem geplanten Ausführungszeitpunkt.

### Kernfunktionen

- **Adaptives Quota-Management mit dynamischer Ratenbegrenzung**  
  Das System liest kontinuierlich den `X-Goog-RateLimit-Remaining`-Header aus und passt den maximalen Durchsatz pro Minute (konfigurierbar via `--max-rpm`, Standard: 250) automatisch an. Bei Unterschreitung eines Schwellwerts von 10 % des Kontingents wird ein **Cooldown-Fenster** von 30 Sekunden aktiviert. Die tatsächliche Sendefrequenz wird über einen **Token-Bucket-Algorithmus** mit einer Bucket-Größe von 50 und einer Auffüllrate von 5 Tokens pro Sekunde gesteuert.

- **Intelligente Fehlerklassifikation und Retry-Strategie**  
  Jede API-Antwort wird anhand eines **3-stufigen Klassifikators** (HTTP-Status, Fehlertext-Matching, `Retry-After`-Header) in eine von drei Retry-Kategorien eingeteilt:  
  - **Kategorie A** (transient): Maximal 7 Wiederholungen, Backoff: `2^n * 1000 ms` (± Jitter)  
  - **Kategorie B** (quota-bedingt): Maximal 3 Wiederholungen, Backoff: `2^n * 30000 ms` (± Jitter)  
  - **Kategorie C** (permanent): Sofortiger Abbruch, Fehler wird in die Dead-Letter-Queue (`retry_beast.dead_letter`) geschrieben.  
  Die Klassifikationslogik ist vollständig über eine YAML-Konfigurationsdatei anpassbar (`--config /etc/gmailretrybeast/classifier.yaml`).

- **Transaktionale Zustellgarantie mit Write-Ahead-Log (WAL)**  
  Bevor eine API-Anfrage gesendet wird, wird ein **Transaktions-Eintrag** in PostgreSQL erstellt (Status: `pending`). Nach erfolgreicher Zustellung wird der Status auf `delivered` gesetzt. Bei einem Absturz während der Anfrage erkennt das System beim nächsten Start alle Einträge mit Status `pending`, die älter als 10 Sekunden sind, und führt einen **geplanten Retry** durch. Die maximale Transaktionsdauer ist auf 120 Sekunden begrenzt (`--tx-timeout 120`). Die Persistenz erfolgt mit **SERIALIZABLE-Isolation**, um Race-Conditions auszuschließen.

### So verwenden Sie es

1. **Initialisierung der Datenbank und Konfiguration des API-Endpunkts**  
   Führen Sie das bereitgestellte SQL-Skript `schema.sql` aus, um die Tabellen `retry_beast.queue`, `retry_beast.dead_letter` und `retry_beast.metrics` zu erstellen. Konfigurieren Sie anschließend die Verbindungsparameter in der Datei `config.yaml` unter dem Abschnitt `database`. Setzen Sie den Gmail API-Endpunkt auf `https://gmail.googleapis.com/gmail/v1/users/me/messages/send` und hinterlegen Sie das Service-Account-JSON unter dem Pfad `--credentials /etc/gmailretrybeast/credentials.json`.

2. **Starten des Daemons mit definierten Schwellwerten**  
   Starten Sie den Dienst mit dem Befehl:  
   `gmailretrybeast --config /etc/gmailretrybeast/config.yaml --max-rpm 250 --tx-timeout 120 --log-level debug`.  
   Der Daemon beginnt sofort mit dem Abrufen von Nachrichten aus der Warteschlange. Überwachen Sie die Startausgabe auf Meldungen wie `"Queue monitor started: polling interval 1000 ms"`.

3. **Manuelles Einfügen einer Nachricht in die Warteschlange**  
   Verwenden Sie das mitgelieferte CLI-Tool `beastctl` zum Einfügen einer Nachricht:  
   `beastctl enqueue --to "empfaenger@beispiel.de" --subject "Test" --body "Hallo Welt" --priority high`.  
   Der Befehl gibt eine eindeutige Transaktions-ID (UUIDv4) zurück. Verfolgen Sie den Status mit:  
   `beastctl status --tx-id <UUID>`.  
   Bei Erfolg erscheint `Status: delivered`. Bei Fehlern wird der Fehlercode und die Anzahl der bisherigen Wiederholungen angezeigt.

### Beste Praktiken

- **Praxis 1: Implementieren Sie ein separates Quota-Monitoring-Dashboard**  
  Nutzen Sie die Metrik-Tabelle `retry_beast.metrics`, die automatisch für jede API-Anfrage ein `latency_ms`, `status_code` und `remaining_quota` speichert. Verbinden Sie diese Tabelle mit einem Grafana-Dashboard (z. B. via PostgreSQL-Datasource) und konfigurieren Sie Alarme für `remaining_quota < 20`. Dies ermöglicht eine proaktive Skalierung der `--max-rpm`-Einstellung, bevor es zu massiven Retry-Schleifen kommt.

- **Praxis 2: Nutzen Sie die Dead-Letter-Queue für manuelle Nachbearbeitung**  
  Konfigurieren Sie einen Cron-Job, der stündlich die Tabelle `retry_beast.dead_letter` auf Einträge mit `error_class = 'C'` prüft und eine Zusammenfassung per E-Mail an das Admin-Team sendet. Analysieren Sie die Fehlerursache (z. B. ungültige Empfängeradresse, gesperrter Absender) und korrigieren Sie die Daten. Führen Sie anschließend den Befehl `beastctl requeue --tx-id <UUID>` aus, um die Nachricht wieder in die Hauptwarteschlange zu verschieben.
