# gmailretrybeast - GMA_021

## Überblick

gmailretrybeast ist ein hochperformantes, asynchrones Wiederholungssystem für fehlgeschlagene Gmail-API-Aufrufe. Es minimiert Datenverluste durch intelligente Retry-Strategien mit exponentiellen Backoff-Algorithmen und bietet eine vollständige Transparenz über den Zustellungsstatus jeder einzelnen E-Mail. Das System ist für den Dauerbetrieb unter hoher Last (10.000+ E-Mails pro Minute) ausgelegt.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist ein in Go implementiertes Daemon-Tool, das fehlgeschlagene Gmail-Nachrichten (Sendevorgänge, Label-Änderungen, Thread-Updates) aus einer PostgreSQL-Warteschlange abruft und mit konfigurierbaren Wiederholungsintervallen (min. 100 ms, max. 300 s) erneut zustellt. Es verwendet OAuth 2.0 mit serverseitigen Tokens und protokolliert jeden Retry-Versuch in einer separaten Audit-Tabelle (`retry_audit_log`).

### Kernfunktionen

- **Adaptiver exponentieller Backoff mit Jitter**  
  Implementiert einen konfigurierbaren Backoff-Algorithmus: `delay = min(base * 2^attempt + random(0, jitter), max_delay)`. Standardwerte: `base = 1 s`, `jitter = 500 ms`, `max_delay = 300 s`. Dadurch werden API-Rate-Limits (Gmail: 250 QPS pro Benutzer) zuverlässig eingehalten.

- **Mehrstufige Fehlerklassifikation und Routing**  
  Fehler werden in drei Kategorien eingeteilt:  
  - **Transient** (HTTP 429, 500, 503) → sofortige Wiederholung nach Backoff  
  - **Auth-Fehler** (HTTP 401, 403) → Token-Refresh mit anschließendem Retry  
  - **Permanent** (HTTP 400, 404) → sofortige Archivierung in `dead_letter_queue`  
  Jede Kategorie hat eine eigene maximale Wiederholungsanzahl (Standard: transient 5, auth 3, permanent 0).

- **PostgreSQL-gestützte Persistenz mit Partitionierung**  
  Die Warteschlangentabelle `retry_queue` ist nach `created_at` monatlich partitioniert. Jeder Eintrag enthält: `message_id (UUID v4)`, `raw_payload (JSONB)`, `retry_count (INT)`, `next_retry_at (TIMESTAMPTZ)`, `last_error (TEXT)`. Ein Cron-Job (alle 5 Minuten) bereinigt Einträge älter als 30 Tage in der `dead_letter_queue`.

### So verwenden Sie es

1. **Initiale Konfiguration und Datenbank-Setup**  
   ```bash
   # PostgreSQL-Connection-String setzen
   export GMAIL_RETRY_DSN="postgres://user:pass@localhost:5432/gmail_retry?sslmode=require"
   # Migration ausführen (erzeugt Tabellen, Partitionen, Indizes)
   ./gmailretrybeast migrate --path ./migrations
   # Überprüfung: Tabelle "retry_queue" muss existieren
   psql $GMAIL_RETRY_DSN -c "\dt retry_queue"
   ```

2. **Worker starten mit benutzerdefinierten Parametern**  
   ```bash
   # Startet 4 parallele Worker, maximale Wiederholungen: 8, Backoff-Basis: 2s
   ./gmailretrybeast worker \
     --workers 4 \
     --max-retries 8 \
     --backoff-base 2000 \
     --jitter 1000 \
     --max-delay 300000 \
     --poll-interval 500
   # Log-Ausgabe: "Worker-1: Verarbeite Nachricht a1b2c3... (Versuch 3/8)"
   ```

3. **Überwachung und Dead-Letter-Management**  
   ```bash
   # Aktuelle Warteschlangengröße abfragen
   ./gmailretrybeast stats --queue-size
   # Ausgabe: "retry_queue: 1.234 Einträge, dead_letter_queue: 56 Einträge"
   # Tote Nachrichten manuell erneut einreihen
   ./gmailretrybeast requeue --from dead_letter_queue --limit 50
   # Erfolgsrate der letzten Stunde anzeigen
   ./gmailretrybeast stats --success-rate --period 1h
   ```

### Beste Praktiken

- **Monitoring mit Prometheus-Metriken integrieren**  
  Exportieren Sie die folgenden Metriken: `gmail_retry_attempts_total{status="success|failed"}`, `gmail_retry_queue_depth`, `gmail_retry_backoff_duration_seconds`. Richten Sie einen Alert ein, wenn die `dead_letter_queue` mehr als 1.000 Einträge enthält, um auf ein systematisches Problem hinzuweisen.

- **Staging-Umgebung mit Dummy-Gmail-API nutzen**  
  Verwenden Sie für Tests eine lokale Mock-API (z. B. mit `wiremock`), die konfigurierbare Fehlerraten (HTTP 429, 500) simuliert. Testen Sie das System mit einer Last von 500 Nachrichten pro Minute und überprüfen Sie, ob die maximale Wiederholungsanzahl eingehalten wird und keine Nachrichten verloren gehen.
