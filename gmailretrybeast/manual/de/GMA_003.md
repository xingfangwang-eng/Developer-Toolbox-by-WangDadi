# gmailretrybeast - GMA_003

## Übersicht

Dies ist der zentrale Baustein für die automatisierte Wiederaufnahme fehlgeschlagener Gmail-Zustellungen. Das System analysiert SMTP-Fehlercodes in Echtzeit und führt exponentielle Backoff-Strategien mit einer maximalen Wiederholungsrate von 5.000 Versuchen pro Stunde aus. Zielsprache: Deutsch (de). Optimiert für Gmail-Retry-Workflows, E-Mail-Warteschlangen-Management und Fehlerbehandlung auf SMTP-Ebene.

## Detaillierter Inhalt

### Was ist gmailretrybeast?

gmailretrybeast ist ein hochperformantes Daemon-Tool, das fehlgeschlagene Gmail-Zustellungen (SMTP-Codes 4xx, 5xx) automatisch erkennt, klassifiziert und mit konfigurierbaren Backoff-Intervallen (min. 100 ms, max. 86.400 s) erneut zustellt. Es arbeitet direkt mit der Gmail API (Version v1) zusammen und speichert Zustandsdaten in einer lokalen SQLite-Datenbank (oder optional PostgreSQL). Das Tool ist speziell für Hochlast-Umgebungen mit bis zu 10.000 gleichzeitigen E-Mail-Warteschlangen ausgelegt.

### Kernfunktionen

- **Intelligentes SMTP-Fehler-Mapping & exponentielles Backoff**  
  Automatische Erkennung von temporären Fehlern (z. B. `421 4.7.0`, `450 4.1.8`) vs. permanenten Fehlern (z. B. `550 5.1.1`). Für temporäre Fehler wird ein exponentielles Backoff mit Jitter (Bereich: 100 ms bis 86.400 s) angewendet. Parameter: `--max-retries 50`, `--initial-backoff 1000` (ms), `--max-backoff 86400000` (ms). Die Genauigkeit der Zeitsteuerung liegt bei ±5 ms.

- **Transaktionale Warteschlangenverwaltung mit Prioritätsstufen**  
  Verwaltung von bis zu 10.000 E-Mail-Objekten in einer ACID-konformen Warteschlange. Jede E-Mail erhält eine Priorität (1–10, Standard: 5) und einen Zeitstempel (ISO 8601, Millisekunden-Genauigkeit). Unterstützt Batch-Operationen (max. 100 E-Mails pro Batch) und automatische Garbage Collection für abgelaufene Einträge (TTL: 7 Tage). Parameter: `--queue-size 10000`, `--batch-size 100`, `--ttl 604800` (s).

- **Echtzeit-Protokollierung & Metrik-Export (Prometheus-kompatibel)**  
  Jeder Wiederholungsversuch wird mit vollständigem SMTP-Header, Fehlercode, Backoff-Intervall und Zeitstempel protokolliert. Exportiert Metriken wie `gmail_retry_total`, `gmail_retry_success`, `gmail_retry_failure` und `gmail_retry_latency_ms` im Prometheus-Format (Port 9090 standardmäßig). Parameter: `--metrics-port 9090`, `--log-level info` (debug, info, warn, error).

### So verwenden Sie es

1. **Installation & Initialisierung der Datenbank**  
   Klonen Sie das Repository und führen Sie `./gmailretrybeast init --db-type sqlite --db-path /var/lib/gmailretrybeast/queue.db` aus. Dieser Befehl erstellt die Datenbanktabellen (`emails`, `retry_log`, `config`) und initialisiert das Schema mit einer Migrationsversion (aktuell: v3.2.1). Überprüfen Sie die Integrität mit `--check-integrity`.

2. **Konfiguration des Gmail-API-Zugriffs**  
   Erstellen Sie eine OAuth2-Client-ID in der Google Cloud Console (Scope: `https://mail.google.com/`). Speichern Sie die Credentials als JSON unter `/etc/gmailretrybeast/credentials.json`. Starten Sie den Authentifizierungsprozess mit `gmailretrybeast auth --credentials /etc/gmailretrybeast/credentials.json --token /etc/gmailretrybeast/token.json`. Das Tool führt einen lokalen OAuth2-Flow durch und speichert das Refresh-Token.

3. **Starten des Daemons mit benutzerdefinierten Parametern**  
   Führen Sie `gmailretrybeast daemon --queue-size 5000 --batch-size 50 --max-retries 30 --initial-backoff 2000 --max-backoff 43200000 --metrics-port 9090 --log-level debug` aus. Der Daemon läuft im Vordergrund (für Produktion mit `nohup` oder systemd). Überwachen Sie die Ausgabe mit `tail -f /var/log/gmailretrybeast/daemon.log`. Beenden Sie den Daemon mit `Ctrl+C` oder `kill -SIGTERM <PID>`.

### Beste Praktiken

- **Verwenden Sie eine separate Datenbank für Produktion und Entwicklung**  
  Konfigurieren Sie für Produktionsumgebungen zwingend PostgreSQL (z. B. `--db-type postgres --db-connection "postgres://user:pass@localhost:5432/gmailretrybeast"`). SQLite ist nur für Tests geeignet, da es bei gleichzeitigen Schreibzugriffen zu `database is locked`-Fehlern kommen kann. Stellen Sie sicher, dass die PostgreSQL-Verbindung über SSL/TLS läuft (Parameter: `--db-ssl-mode require`).

- **Implementieren Sie einen Health-Check-Endpunkt für Monitoring**  
  Aktivieren Sie den integrierten HTTP-Health-Check unter `http://localhost:8080/health`. Dieser Endpunkt gibt den aktuellen Status der Datenbankverbindung, der Warteschlangengröße und der letzten erfolgreichen Zustellung zurück (JSON-Format). Verwenden Sie diesen Endpunkt in Kombination mit Monitoring-Tools wie Prometheus oder Nagios. Parameter: `--health-port 8080`.
