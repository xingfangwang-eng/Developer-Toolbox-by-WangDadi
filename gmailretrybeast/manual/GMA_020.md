# gmailretrybeast - GMA_020

## Overview

`gmailretrybeast` is a high-reliability email retry engine designed for Gmail API environments where transient failures, quota limits, and propagation delays are common. It implements exponential backoff with jitter, rate-limit-aware scheduling, and concurrency throttling to ensure delivery of critical outbound messages under adverse conditions. Built for CI/CD pipelines, monitoring alerts, and transactional email workflows.

## Core Features

### 1. Adaptive Throttle with Gmail API Quota Awareness
Dynamically adjusts retry intervals based on real-time Gmail API `RateLimitExceeded` and `UserRateLimitExceeded` responses. Uses a sliding-window token bucket (capacity: 250 tokens, refill rate: 10 tokens/sec) to prevent silent drops. Integrates with `google.golang.org/api/gmail/v1` to parse `retryAfter` headers and enforce per-user concurrency caps (default: 5 concurrent sends per `userId`).

### 2. Message-Level Retry Chain with Idempotency Verification
Each outbound email is assigned a deterministic `messageId` (SHA-256 of `To + Subject + Timestamp`). On retry, the engine queries Gmail's `users.messages.list` with `q= rfc822msgid:<id>` to confirm non-delivery before re-sending. Prevents duplicate sends even after partial API failures. Supports configurable max retries (default: 7) with a total timeout ceiling of 300 seconds.

### 3. Structured Error Classification & Escalation Pipeline
Categorizes failures into three tiers:
- **Transient**: `500`, `503`, `429` — auto-retry with backoff (`baseDelay=1s`, `maxDelay=64s`, `jitterFactor=0.25`).
- **Non-Retryable**: `400` (invalid attachment), `403` (scope missing), `404` (user not found) — immediately logged to PostgreSQL (`failures` table with `raw_response` JSONB column).
- **Escalation**: consecutive failures exceeding 3 attempts — triggers a webhook payload to a configurable endpoint (e.g., PagerDuty, Slack) with full trace context.

## How to Use

### Step 1: Initialize Engine with Gmail Credentials and Retry Policy
```go
import "github.com/xingfangwang-eng/gmailretrybeast/engine"

cfg := engine.Config{
    CredentialsFile: "/etc/gmail/credentials.json",
    TokenDir:        "/var/lib/gmailretrybeast/tokens",
    MaxRetries:      7,
    BaseDelayMs:     1000,
    MaxDelayMs:      64000,
    ConcurrencyLimit: 5,
    DBConnection:    "postgres://user:pass@localhost:5432/email_logs",
    WebhookURL:      "https://hooks.example.com/escalate",
}
beast, err := engine.New(cfg)
```

### Step 2: Enqueue a Message with Delivery Metadata
```go
msg := engine.Message{
    To:       "user@domain.com",
    Subject:  "Critical Alert — Disk Usage > 90%",
    Body:     "Plain text or HTML body here",
    Attachments: []engine.Attachment{
        {Filename: "report.csv", Data: csvBytes, MimeType: "text/csv"},
    },
    IdempotencyKey: "alert-20250321-001",
}

jobID, err := beast.Send(context.Background(), msg)
// Returns immediately; retries handled asynchronously.
```

### Step 3: Monitor Delivery Status via PostgreSQL and Prometheus Metrics
```sql
-- Check final disposition of a specific job
SELECT status, retry_count, last_error, completed_at
FROM email_jobs
WHERE idempotency_key = 'alert-20250321-001';
```
The engine exposes a Prometheus `/metrics` endpoint with counters `gmail_retry_total{status="success|failure|escalated"}` and histogram `gmail_retry_duration_seconds` for latency tracking.

## Best Practices

### 1. Always Set an Idempotency Key for Transactional Emails
Without a deterministic key, a partial Gmail API failure (e.g., timeout after sending but before response) can cause duplicate deliveries on retry. Compute the key from immutable fields (recipient + subject + timestamp truncated to minute granularity) to avoid double-sends without sacrificing retry coverage.

### 2. Pair with a Dead-Letter Queue for Escalated Failures
Configure the `WebhookURL` to push to a separate DLQ (e.g., Amazon SQS or Redis Stream). This decouples retry logic from manual intervention. Use the `failures` table to periodically replay escalated jobs after fixing the root cause (e.g., re-authenticating expired OAuth tokens).
