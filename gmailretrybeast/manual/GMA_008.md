# gmailretrybeast - GMA_008

## Overview

`gmailretrybeast` is a high-availability Gmail message retry and delivery assurance engine designed for mission-critical email workflows. It addresses the inherent unreliability of transient Gmail API failures, quota exhaustion, and unpredictable SMTP gateways by implementing a deterministic retry pipeline with exponential backoff, idempotency enforcement, and persistent state tracking. The tool is optimized for enterprise-grade email management where a single undelivered message can cascade into SLA breaches or data loss.

## Core Features

### 1. Adaptive Retry Scheduler with Jitter-Aware Backoff
Implements a multi-tier retry strategy that analyzes Gmail API response headers (`Retry-After`, `X-Goog-Quota-User`) and adjusts the retry interval dynamically. The scheduler supports configurable `max_retry_attempts` (default: 5), `initial_backoff_ms` (default: 1000), and `max_backoff_ms` (default: 60000). Each retry applies a **full-jitter** algorithm (range: `[0, current_backoff * 2)`) to avoid thundering herd issues. Failed messages are stored in a PostgreSQL-backed queue with `status`, `attempt_count`, and `last_error` columns, enabling crash-resilient recovery.

### 2. Idempotent Delivery with Deduplication Key Hashing
Ensures that identical messages are never sent more than once by generating a **SHA-256 deduplication hash** from the concatenation of `to`, `subject`, `body_hash`, and `timestamp_ms`. The hash is stored in a `dedup_keys` table with a TTL of 86,400 seconds (24 hours). If a retry attempt encounters a 409 Conflict or 429 Too Many Requests, the system checks the dedup store before re-queuing. This prevents duplicate invoices, alerts, or confirmation emails even under extreme retry pressure.

### 3. Real-Time Quota Exhaustion Predictor
Monitors Gmail API usage metrics via the `users.settings.delegates.list` and `users.messages.list` endpoints, combined with local `rate_limit_remaining` counters. The predictor calculates the **available quota window** (in seconds) before the next `quota_reset_at` timestamp and throttles retry issuance proportionally. If `remaining_quota < 10`, all non-critical retries are queued into a **cold storage bucket** (S3-compatible object store) for deferred processing. The predictor exposes a Prometheus gauge metric `gmail_quota_remaining_pct` for dashboard integration.

## How to Use

### Step 1: Configure the Retry Pipeline with Environment Variables
Set up the core parameters in your deployment environment. The tool reads `RETRYBEAST_DB_DSN` (PostgreSQL connection string), `RETRYBEAST_GMAIL_CREDENTIALS` (path to OAuth2 JSON), and `RETRYBEAST_MAX_RETRIES` (integer). Example:
```bash
export RETRYBEAST_DB_DSN="postgres://user:pass@localhost:5432/retrybeast?sslmode=disable"
export RETRYBEAST_GMAIL_CREDENTIALS="/etc/secrets/gmail_oauth.json"
export RETRYBEAST_MAX_RETRIES=7
export RETRYBEAST_QUOTA_THRESHOLD_PCT=15
```
Run `gmailretrybeast init` to validate the connection and create the `retry_queue` and `dedup_keys` tables.

### Step 2: Enqueue a Message with Custom Retry Policy
Use the CLI or HTTP API to submit a message with explicit retry overrides. For example, via the REST endpoint:
```bash
curl -X POST https://localhost:8080/enqueue \
  -H "Content-Type: application/json" \
  -d '{
    "to": "alert@example.com",
    "subject": "Critical: Disk Usage Exceeded",
    "body": "Server srv-045: /dev/sda1 at 95% capacity",
    "priority": "high",
    "retry_policy": {
      "initial_backoff_ms": 500,
      "max_backoff_ms": 30000,
      "max_attempts": 10,
      "jitter_type": "full"
    }
  }'
```
The API returns a `message_id` (UUID v4) and `dedup_key` (SHA-256 hex string). The message is immediately inserted into the PostgreSQL `retry_queue` with `status='pending'`.

### Step 3: Monitor and Force Retry from the Dashboard
Access the built-in web UI at `/dashboard` or expose metrics via the `/metrics` endpoint (Prometheus format). To manually trigger a retry for a stuck message, use:
```bash
gmailretrybeast retry --message-id "a1b2c3d4-e5f6-7890-abcd-ef1234567890" --force
```
This bypasses the backoff schedule and immediately resends the message, updating `attempt_count` and `last_retry_at` in the database. The dashboard shows real-time `total_enqueued`, `successful_deliveries`, and `dead_letter_count` (messages exceeding `max_retries`).

## Best Practices

### 1. Implement a Dead Letter Queue with Manual Review
Configure `RETRYBEAST_DEAD_LETTER_SINK` to point to an S3 bucket or a separate PostgreSQL table (`dead_letter_queue`). After a message exhausts all retry attempts, it should be moved to the dead letter queue rather than silently dropped. Schedule a weekly review of dead letters using `gmailretrybeast export-dead-letters --format csv` to identify systemic issues (e.g., invalid recipient addresses, permanent Gmail API bans). This practice prevents silent data loss and provides an audit trail for compliance.

### 2. Use Staged Retry Escalation for Critical Workflows
For transactional emails (e.g., password resets, payment confirmations), define a staged retry policy: first 3 attempts with 1-second backoff, next 3 attempts with 10-second backoff, and final 2 attempts with 60-second backoff. Couple this with a webhook notification (`RETRYBEAST_ESCALATION_WEBHOOK`) that fires when a message reaches attempt 6. This ensures that human operators are alerted before the message enters the dead letter queue, allowing them to manually intervene (e.g., regenerate OAuth tokens) before the SLA expires.
