# gmailretrybeast - GMA_017

## Overview

gmailretrybeast is a high‑performance, fault‑tolerant email retry engine engineered to eliminate silent delivery failures in Gmail‑based workflows. It implements a multi‑stage exponential backoff strategy with real‑time SMTP response analysis, ensuring that transient errors (e.g., 4xx, 5xx SMTP codes) are retried with sub‑second precision while respecting Gmail’s rate limits and quota policies.

## Core Features

### 1. Adaptive Retry Scheduler with Millisecond‑Granularity Backoff
- **Parameter:** `--retry-delay-base=500` (base delay in ms), `--max-retry-interval=300000` (max 5 min), `--jitter-factor=0.25`
- Dynamically adjusts retry intervals based on Gmail’s `Retry‑After` header and SMTP response codes. Uses a Fibonacci‑based backoff curve (not linear) to minimize throttling risk.
- Logs every retry attempt with exact timestamp, response code, and computed delay in a PostgreSQL `retry_log` table for post‑mortem analysis.

### 2. Real‑Time SMTP Response Classifier & Circuit Breaker
- **Parameter:** `--circuit-breaker-threshold=5` (failures within `--circuit-breaker-window=60` seconds), `--circuit-breaker-reset=120` (seconds)
- Classifies SMTP responses into transient (4xx), permanent (5xx), and rate‑limit (420/429) categories. If a specific Gmail endpoint returns 5 consecutive 4xx errors in 60 seconds, the circuit breaker opens, halting all retries to that endpoint for 120 seconds.
- Integrates with Gmail API’s `users.messages.send` and SMTP relay, providing a unified failure taxonomy.

### 3. Quota‑Aware Throttling Engine with Google Workspace Admin API Integration
- **Parameter:** `--quota-limit=100` (messages per `--quota-window=60` seconds), `--admin-api-refresh-interval=300` (seconds)
- Polls the Google Workspace Admin API (`gmail.users.settings.sendAs.list`) to fetch current daily sending quotas (e.g., 2000 recipients/day for standard accounts). If the remaining quota falls below `--quota-warning-threshold=10%`, the engine switches to a priority queue mode, only processing urgent emails until quota resets.
- Uses a token bucket algorithm with per‑user rate limiting to avoid Gmail’s `rateLimitExceeded` error.

## How to Use

### Step 1: Configure Retry Policy and Quota Limits
```bash
gmailretrybeast --retry-delay-base=500 --max-retry-interval=300000 --jitter-factor=0.25 \
                --quota-limit=100 --quota-window=60 --admin-api-refresh-interval=300 \
                --circuit-breaker-threshold=5 --circuit-breaker-window=60 --circuit-breaker-reset=120
```
This sets a 500ms base delay, a 5‑minute maximum retry interval, and a 25% jitter. The circuit breaker opens after 5 failures in 60 seconds and resets after 120 seconds. Quota is limited to 100 messages per 60 seconds, with admin API polling every 5 minutes.

### Step 2: Start the Retry Daemon with PostgreSQL Logging
```bash
gmailretrybeast --mode=daemon --postgres-conn="host=pg1 user=retry beast dbname=email_retry sslmode=require" \
                --retry-log-table=retry_log --smtp-endpoint=smtp.gmail.com:587 \
                --oauth-client-id=your_client_id --oauth-client-secret=your_client_secret
```
The daemon will connect to the PostgreSQL instance, create the `retry_log` table if missing, and begin processing the retry queue. All retry attempts (including successful ones) are logged with SMTP response code, latency, and computed delay.

### Step 3: Inspect Circuit Breaker Status and Quota Health
```bash
gmailretrybeast --status --circuit-breaker-summary --quota-remaining
```
Outputs a real‑time dashboard:
- **Circuit Breaker:** `[OPEN]` or `[CLOSED]` with reason (e.g., “5 consecutive 4xx from smtp.gmail.com within 60s”)
- **Quota:** “Remaining: 45/100 (45%) – Warning threshold: 10% – Priority queue active: NO”
- **Retry Queue:** “Depth: 12 – Oldest message age: 34s – Average retry latency: 1.2s”

## Best Practices

### 1. Always Combine with a Dead‑Letter Queue for Permanent Failures
Configure a separate PostgreSQL table (e.g., `dead_letter_queue`) to capture messages that hit `--max-retry-count=10`. This prevents infinite retries on unrecoverable errors (e.g., invalid recipient address). Use a scheduled job to review the dead‑letter queue daily and re‑inject corrected messages.

### 2. Monitor Circuit Breaker Transitions with Prometheus Metrics
Expose the circuit breaker state and quota usage as Prometheus metrics (`gmailretrybeast_circuit_breaker_state`, `gmailretrybeast_quota_remaining`). Set alerts for `circuit_breaker_state=1` (open) or `quota_remaining < 10` to proactively investigate Gmail endpoint health or quota exhaustion before it impacts business‑critical email delivery.
