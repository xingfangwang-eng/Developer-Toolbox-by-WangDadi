# gmailretrybeast - GMA_021

## Overview

gmailretrybeast is a high-throughput, fault-tolerant Gmail API retry engine designed for mission-critical email delivery pipelines. It systematically handles transient failures, rate-limit backoffs, and quota exhaustion from Google Workspace APIs, ensuring zero message loss under extreme load conditions.

## Core Features

### 1. Adaptive Exponential Backoff with Jitter (AEBJ)
- **Parameterized retry window**: `--base-delay=250ms` to `--max-delay=120s` with configurable multiplier (`--backoff-multiplier=2.0`).
- **Full-Jitter algorithm**: randomizes sleep intervals within `[0, delay * multiplier)` to prevent thundering herd on Gmail API endpoints.
- **Retry budget enforcement**: `--max-retries=10` and `--total-timeout=300s` per message, with per-second throughput cap (`--rps-limit=15`).
- **Granular error classification**: distinguishes `429 Too Many Requests`, `500 Internal Server Error`, and `503 Service Unavailable`; applies distinct retry policies per error code.

### 2. Stateful Queue Persistence with PostgreSQL
- **Transactional message store**: each email delivery attempt is recorded in a `delivery_attempts` table with columns: `message_id UUID`, `attempt_number INT`, `response_code SMALLINT`, `backoff_duration_ms INT`, `timestamp TIMESTAMPTZ`.
- **Dead-letter queue (DLQ)**: messages exceeding `--max-retries` are moved to `dlq_messages` table with full headers and error snapshots for forensic analysis.
- **Idempotency key deduplication**: uses `--idempotency-key=sha256(message_body)` to prevent duplicate sends across retries; enforced via unique constraint on `message_id`.
- **Connection pooling**: `--pg-pool-size=20` with `--pg-query-timeout=5s` for low-latency state reads/writes.

### 3. Real-time Metrics & Circuit Breaker
- **Prometheus metric exports**: `gmail_retry_attempts_total`, `gmail_retry_success_total`, `gmail_retry_backoff_milliseconds` (histogram with buckets: 50ms, 200ms, 1s, 5s, 30s, 120s).
- **Circuit breaker**: `--failure-threshold=50` within `--window-seconds=60` triggers open state; half-open after `--cooldown-seconds=30` with probe requests.
- **Health check endpoint**: `/healthz` returns `{"status":"ok","circuit_state":"closed","pending_messages":142,"dlq_count":3}`.

## How to Use

### Step 1: Initialize Database Schema
```bash
gmailretrybeast --init-db \
  --pg-host=db.internal \
  --pg-port=5432 \
  --pg-user=retry_user \
  --pg-password=<secret> \
  --pg-database=gmail_retry
```
This creates `delivery_attempts`, `dlq_messages`, and `idempotency_keys` tables with composite indexes on `(message_id, attempt_number)` and `(idempotency_key)`.

### Step 2: Configure Gmail API Credentials & Retry Policy
```bash
gmailretrybeast --credentials=/etc/gmail/service-account.json \
  --user-id=admin@company.com \
  --scope=https://www.googleapis.com/auth/gmail.send \
  --base-delay=500ms \
  --max-delay=60s \
  --backoff-multiplier=2.0 \
  --max-retries=8 \
  --rps-limit=10 \
  --idempotency-key=sha256
```
Validate connectivity with `--dry-run=true` flag; the tool will simulate 5 test sends without actually delivering.

### Step 3: Process Queue with Circuit Breaker & Monitoring
```bash
gmailretrybeast --process-queue \
  --batch-size=100 \
  --concurrency=4 \
  --circuit-breaker \
  --failure-threshold=50 \
  --window-seconds=60 \
  --cooldown-seconds=30 \
  --metrics-port=9090 \
  --dlq-path=/var/log/gmailretrybeast/dlq
```
Streams `delivery_attempts` table in `--batch-size` chunks, applies retry logic, and writes failed messages to DLQ. Exposes `/metrics` for Prometheus scraping and `/healthz` for load balancer health checks.

## Best Practices

### 1. Pre-warm Connection Pools Before Production Deployments
Before routing live email traffic, run a 5-minute warm-up phase with `--warmup-duration=300s` and `--warmup-rps=5`. This establishes database connections, authenticates Gmail API OAuth tokens, and populates connection pool caches. Measure `gmail_retry_backoff_milliseconds` histogram; if p99 latency exceeds 2s, increase `--pg-pool-size` or reduce `--concurrency`.

### 2. Implement Dead-Letter Queue Monitoring with Alert Thresholds
Set up a Grafana dashboard with alert rules: if `dlq_count` exceeds 1% of total processed messages in a 15-minute sliding window, trigger a PagerDuty notification. Configure automated reprocessing via `gmailretrybeast --reprocess-dlq --dlq-batch=50` during off-peak hours (e.g., 02:00–04:00 UTC) to avoid rate-limit collisions with live traffic.
