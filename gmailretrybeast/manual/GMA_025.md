# gmailretrybeast - GMA_025

## Overview

`gmailretrybeast` is a high-throughput, fault-tolerant email delivery engine designed specifically for Gmail’s API constraints. It implements a layered retry architecture with exponential backoff, per-thread rate limiting, and real-time SMTP error classification. The system is optimized for bulk email campaigns where Gmail’s 500/day sending limit and 2000/day recipient cap must be strictly enforced while maintaining a delivery success rate above 99.7%.

## Core Features

### 1. Adaptive Rate-Limiting & Token Bucket Throttling
- **Token Bucket Parameters:** `max_tokens=500`, `refill_rate=0.0058 tokens/sec` (calculated from Gmail’s 500 sends/day limit).
- **Burst Mode:** Allows up to 50 concurrent connections with a 300ms minimum gap between each, using `time.Sleep(time.Duration(rand.Intn(200)) * time.Millisecond)` to avoid API burst penalties.
- **Per-Thread Queue Depth:** Configurable via `--queue-depth=1000`, with automatic drain when Gmail returns `429 Too Many Requests`.

### 2. Smart Retry Engine with SMTP Error Classification
- **Error Category Mapping:** Maps Gmail-specific errors (e.g., `550 5.1.1`, `421 4.7.0`) to retry strategies: `Permanent (no retry)`, `Transient (retry with 30s-120s jittered backoff)`, `RateLimit (retry with 600s cooldown)`.
- **Retry Budget:** `max_retries=5` per email, with an exponential backoff formula: `delay = base_delay * (2^attempt) + random_jitter(0, 1000ms)`. Base delay is 60 seconds.
- **Dead Letter Queue (DLQ):** Emails that exhaust all retries are written to `dlq/gma_025_{timestamp}.jsonl` with full headers and error trace.

### 3. Real-Time Delivery Audit & PostgreSQL Integration
- **Event Sourcing:** Every send attempt is logged to a PostgreSQL table `delivery_events` with columns: `email_id UUID`, `attempt INT`, `status TEXT`, `error_code VARCHAR(10)`, `timestamp TIMESTAMPTZ`.
- **Latency Tracking:** Measures round-trip time per email in milliseconds, storing `min_latency`, `max_latency`, `avg_latency` per batch of 100 emails.
- **Dashboard Export:** Generates a CSV report at `reports/delivery_summary_{date}.csv` with fields: `total_sent`, `failed`, `retried`, `avg_retry_count`, `p99_latency_ms`.

## How to Use

### Step 1: Configure Gmail API Credentials & Rate Limits
```bash
# Create OAuth 2.0 client ID from Google Cloud Console
# Save credentials as 'credentials.json' in project root

# Initialize token with required scopes:
gmailretrybeast init --scopes "https://www.googleapis.com/auth/gmail.send" \
                     --client-id "YOUR_CLIENT_ID" \
                     --client-secret "YOUR_CLIENT_SECRET" \
                     --token-path "token.json"

# Verify rate limit alignment:
gmailretrybeast validate-limits --daily-send-limit 500 \
                                --daily-recipient-limit 2000 \
                                --thread-pool-size 10
```

### Step 2: Prepare Email Payload & Retry Policy
```json
{
  "batch": [
    {
      "to": "user@example.com",
      "subject": "Account Verification",
      "body": "<html>...</html>",
      "priority": "high"
    }
  ],
  "policy": {
    "max_retries": 5,
    "base_backoff_ms": 60000,
    "jitter_ms": 1000,
    "dlq_path": "dlq/gma_025_20250321.jsonl"
  }
}
```
```bash
# Load batch from file:
gmailretrybeast send --batch-file batch_001.json \
                     --policy-file retry_policy.json \
                     --output-report reports/delivery_001.csv
```

### Step 3: Monitor Delivery & Handle Dead Letters
```bash
# Real-time monitoring (updates every 5 seconds):
gmailretrybeast monitor --status-interval 5s \
                        --show-latency \
                        --show-queue-depth

# Process dead letters manually:
gmailretrybeast replay-dlq --dlq-file dlq/gma_025_20250321.jsonl \
                           --new-policy-file retry_policy_v2.json \
                           --skip-permanent-errors

# Generate PostgreSQL audit query:
gmailretrybeast audit --db-uri "postgres://user:pass@localhost:5432/gmail_retry" \
                      --query "SELECT email_id, status, error_code, timestamp FROM delivery_events WHERE status = 'failed' AND attempt >= 3 ORDER BY timestamp DESC LIMIT 100"
```

## Best Practices

### 1. Implement a Staged Retry Strategy with Circuit Breaker
- **Practice:** Use a two-stage retry: first stage with 3 retries at 60s intervals, second stage with 2 retries at 300s intervals. This prevents hammering Gmail’s API during transient outages while allowing recovery from longer throttling periods.
- **Rationale:** Gmail’s rate limit resets every 24 hours. A 300s cooldown in the second stage ensures you don’t exhaust your daily budget on a single failed batch.

### 2. Pre-Validate Email Addresses with SMTP RCPT Check
- **Practice:** Before sending, run a lightweight SMTP RCPT TO validation against Gmail’s MX servers using `net.LookupMX("gmail-smtp-in.l.google.com")` and a 10-second timeout. Reject emails with invalid or non-existent addresses immediately.
- **Rationale:** Gmail charges your daily send limit even for failed deliveries (e.g., `550 5.1.1`). Pre-validation reduces wasted quota by up to 40% in lists with high bounce rates.
