# gmailretrybeast - GMA_024

## Overview

gmailretrybeast is a high-frequency Gmail API retry engine designed for mission-critical email delivery pipelines. It systematically re-processes transient failures (rate limits, 429s, temporary authentication glitches) with exponential backoff, per-thread concurrency control, and real-time quota monitoring. The tool is optimized for environments where email delivery reliability directly impacts SLA compliance, such as transactional email systems, automated notification services, and bulk CRM mailers.

## Detailed Content

### What is gmailretrybeast?

gmailretrybeast is a stateless, configurable retry daemon that intercepts Gmail API send failures, classifies error codes into retryable vs. fatal categories, and re-drives failed send operations using a sliding-window quota-aware scheduler. It consumes a JSON-line failure log, applies user-defined retry policies (max attempts, backoff multiplier, jitter window), and writes success/failure outcomes to a PostgreSQL audit table. The tool supports both synchronous blocking mode (for low-volume, guaranteed delivery) and asynchronous batch mode (for high-throughput, best-effort recovery).

### Core Features

- **Quota-Aware Exponential Backoff with Jitter**  
  Implements a configurable retry schedule: initial delay `retry_initial_ms=1000`, backoff multiplier `retry_multiplier=2.0`, max delay `retry_max_ms=300000`, and jitter window `jitter_ms=500`. The engine respects Gmail's per-user send quota (`quota_max_per_second=10`, `quota_burst=20`) by dynamically throttling retry rate based on real-time `X-RateLimit-Remaining` headers. Each retry attempt logs the remaining quota and resets the delay if quota is exhausted.

- **Error Classification & Fatal Short-Circuit**  
  Parses Gmail API error responses using `error.code` and `error.details.reason`. Retryable codes: `429` (rate limit), `500` (backend error), `503` (service unavailable). Fatal codes: `400` (invalid grant), `401` (auth failure), `403` (forbidden), `404` (resource not found). Fatal errors are immediately written to a dead-letter queue (`dead_letter_path=/var/log/gmailretrybeast/dead_letter.ndjson`) with full payload and error context. The engine supports custom error mapping via a YAML config file.

- **PostgreSQL Audit Trail with Idempotency Key**  
  Every retry attempt is recorded in a PostgreSQL table `retry_audit` with columns: `idempotency_key (UUID)`, `attempt (INT)`, `status (ENUM: success, retryable_fail, fatal_fail)`, `http_status_code (INT)`, `response_body (TEXT)`, `quota_remaining (INT)`, `delay_ms (INT)`, `created_at (TIMESTAMPTZ)`. The `idempotency_key` is derived from the original message ID (`msg_id`) and a user-supplied namespace (`namespace`). On success, the engine marks the original failure log as resolved, preventing duplicate execution.

### How to Use

1. **Prepare Failure Log & Configuration**  
   Generate a JSON-lines file (`failures.ndjson`) with each line containing: `{"msg_id": "abc123", "namespace": "prod", "original_payload": {"to": "user@example.com", "subject": "Invoice", "body": "..."}, "error": {"code": 429, "reason": "rateLimitExceeded", "message": "Exceeded send rate"}`. Create a YAML config (`retrybeast.yaml`) with parameters: `input_path: /data/failures.ndjson`, `output_db: postgres://user:pass@localhost:5432/retrydb`, `table_name: retry_audit`, `max_attempts: 5`, `retry_initial_ms: 1000`, `retry_multiplier: 2.0`, `retry_max_ms: 300000`, `jitter_ms: 500`, `quota_max_per_second: 10`, `quota_burst: 20`, `dead_letter_path: /var/log/gmailretrybeast/dead_letter.ndjson`.

2. **Execute the Retry Engine in Dry-Run Mode**  
   Run `gmailretrybeast --config retrybeast.yaml --dry-run` to validate configuration and simulate retry scheduling without sending API calls. The engine prints the planned execution order, estimated total duration, and quota usage projection. Verify that the `retry_audit` table is created (if `auto_create_table: true`) and that the dead-letter path is writable.

3. **Run in Production with Monitoring**  
   Execute `gmailretrybeast --config retrybeast.yaml --log-level debug --prometheus-port 9090` to start the retry process. Monitor real-time metrics via Prometheus: `retry_attempts_total`, `retry_success_total`, `retry_fatal_fail_total`, `retry_quota_exhausted_total`, `retry_delay_seconds`. Observe the PostgreSQL audit table for idempotency violations (duplicate `idempotency_key` attempts). After completion, review the dead-letter queue for fatal failures and manually resolve by correcting auth tokens or permissions.

### Best Practices

- **Configure Idempotency Keys for All Send Operations**  
  Always generate a deterministic `idempotency_key` from the message ID and namespace before the initial Gmail API call. This ensures that if the retry engine re-processes a failure log, it never duplicates a successfully delivered email. Store the key in a separate column (`idempotency_key`) alongside the send attempt in your application database. For high-volume systems, use a hash of the entire payload (SHA-256) as the idempotency key to cover identical retry scenarios.

- **Set Conservative Quota Limits and Monitor Burst Behavior**  
  Never set `quota_max_per_second` above 80% of your Gmail API quota (typically 10 for standard accounts, 100 for Google Workspace). Use `quota_burst=20` to allow short spikes but monitor `retry_quota_exhausted_total` metric. If the metric rises above 0.1% of total retry attempts, reduce `quota_max_per_second` by 10% and increase `retry_max_ms` to 600000. For bulk campaigns, run the engine in batch mode (`--batch-size 50`) with a 5-second cooldown between batches to avoid triggering Gmail's abuse detection.
