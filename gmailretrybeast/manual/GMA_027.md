# gmailretrybeast - GMA_027

## Overview

`gmailretrybeast` is a high‑performance, event‑driven retry engine designed to recover Gmail API failures with sub‑second precision. It combines an adaptive backoff scheduler, a PostgreSQL‑backed state machine, and a real‑time monitoring dashboard to ensure zero message loss in mission‑critical email workflows.

## Detailed Content

### What is gmailretrybeast?

`gmailretrybeast` is a hardened retry orchestration layer that intercepts transient Gmail API errors (rate limits, 5xx, network timeouts) and replays them with configurable exponential backoff, jitter, and circuit‑breaker logic. It maintains a durable retry queue in PostgreSQL, tracks each attempt’s latency and HTTP status, and exposes Prometheus metrics for alerting. Designed for systems that send or modify thousands of Gmail messages per minute, it eliminates silent failures and manual reprocessing.

### Core Features

- **Adaptive Backoff Scheduler with Jitter**  
  Automatically calculates retry intervals using `baseDelay × 2^attempt + random(0, jitter_ms)`. Parameters: `base_delay_ms` (default 1000), `max_delay_ms` (default 300000), `jitter_ms` (default 500). Supports per‑error‑code thresholds (e.g., 429 → longer backoff, 503 → immediate retry).

- **PostgreSQL‑Backed Durable Retry State Machine**  
  Each email send attempt is recorded in a `retry_state` table with columns: `message_id`, `attempt_count`, `next_retry_at` (timestamptz), `last_error` (jsonb), and `status` (enum: `pending`, `in_progress`, `success`, `dead_lettered`). Enables crash‑recovery: on restart, the beast resumes all `pending` and `in_progress` items within 500 ms.

- **Real‑Time Retry Dashboard & Prometheus Exporter**  
  Exposes `/metrics` endpoint with gauges: `retry_queue_depth`, `retry_attempts_total{status="success|failure|dead_letter"}`, `retry_latency_seconds` (histogram buckets: 0.1, 0.5, 1, 5, 30, 120). A built‑in WebSocket‑powered UI shows live retry streams, including `message_id`, `attempt_number`, `delay_ms`, and `error_code`.

### How to Use

1. **Initialize the PostgreSQL Schema & Configuration**  
   ```sql
   CREATE TABLE retry_state (
     message_id UUID PRIMARY KEY,
     attempt_count INT DEFAULT 0,
     max_attempts INT DEFAULT 5,
     next_retry_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
     last_error JSONB,
     status RETRY_STATUS DEFAULT 'pending',
     created_at TIMESTAMPTZ DEFAULT NOW()
   );
   ```
   Then set environment variables: `GMAIL_RETRY_BEAST_DSN=postgres://user:pass@host:5432/retrydb`, `GMAIL_RETRY_BEAST_MAX_ATTEMPTS=7`, `GMAIL_RETRY_BEAST_BASE_DELAY_MS=2000`.

2. **Inject the Beast into Your Gmail Send Pipeline**  
   Wrap your `users.messages.send` call with the beast’s `RetrySend(ctx, message)` method. It returns a `RetryResult` containing `final_status`, `total_attempts`, and `total_delay_ms`. Example (Go):
   ```go
   result, err := beast.RetrySend(ctx, &gmail.Message{Raw: rawMsg})
   if err != nil {
       log.Fatalf("dead lettered after %d attempts: %v", result.TotalAttempts, err)
   }
   ```

3. **Monitor and Tune via the Dashboard**  
   Start the beast with `--dashboard=:8080`. Open `http://localhost:8080` to see live retry events. Use the `/config` endpoint (POST) to adjust `max_attempts` or `base_delay_ms` without restarting. For production, scrape `/metrics` into Prometheus and set alerts on `retry_queue_depth > 1000` or `retry_latency_seconds_p99 > 30`.

### Best Practices

- **Pair with a Dead‑Letter Queue (DLQ) for Irrecoverable Errors**  
  Configure `max_attempts` to 7–10 and route `dead_lettered` messages to a separate PostgreSQL table `dlq_messages`. Use a weekly cron job that replays DLQ items after manual inspection. This prevents infinite retries while preserving data for forensic analysis.

- **Use Circuit‑Breaker Pattern for Burst Rate Limits**  
  When the Gmail API returns 429 (rate limit exceeded) more than 5 times in a 30‑second window, the beast opens a circuit for 60 seconds, queuing all new sends into `pending` state. This avoids hammering the API and complies with Gmail’s per‑second quota. Log circuit state transitions to a dedicated `circuit_events` table for audit.
