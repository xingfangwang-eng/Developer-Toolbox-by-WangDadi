# gmailretrybeast - GMA_014

## Overview

gmailretrybeast is a high‑performance email delivery resilience engine engineered for Gmail SMTP relay environments. It systematically intercepts transient delivery failures (e.g., 4xx SMTP codes, rate‑limiting backoffs, TLS handshake drops) and orchestrates intelligent retry sequences with exponential backoff, jitter, and per‑recipient state tracking. The system is designed to operate under Google’s Gmail sending quotas while maximizing inbox placement rates.

## Core Features

### 1. Adaptive Retry Scheduler with Multi‑Stage Backoff
Implements a configurable retry chain that respects Gmail’s `450 4.7.0` and `421 4.7.0` transient error codes. The scheduler uses a **Fibonacci‑based backoff** (base interval: 30 seconds, max interval: 3600 seconds) combined with **±10% jitter** to avoid thundering‑herd effects. Each retry attempt records the SMTP response code, latency (in milliseconds), and the exact `X‑Gm‑MessageState` header from the Gmail response. The scheduler supports up to **7 retry stages** before moving the message to a dead‑letter queue.

### 2. Real‑Time Quota Guardian & Throttle Breaker
Monitors Gmail’s **per‑day sending quota** (default: 2,000 recipients for free accounts, 10,000 for Workspace) and **per‑second rate limit** (default: 5 recipients/second). The guardian tracks the `Retry‑After` header from Gmail’s `429 Too Many Requests` responses and dynamically adjusts the dispatch rate. It exposes a **Prometheus‑compatible metrics endpoint** (`/metrics`) with counters for `gmail_retry_attempts_total`, `gmail_quota_exhausted_total`, and `gmail_send_latency_seconds`. When the quota is within 10% of exhaustion, it automatically switches to a **deferred delivery mode**, queuing messages in PostgreSQL for later release.

### 3. Per‑Recipient State Machine & Transactional Logging
Maintains a **state machine** (states: `PENDING`, `IN_FLIGHT`, `SUCCESS`, `TRANSIENT_FAILURE`, `PERMANENT_FAILURE`) for each recipient in a **PostgreSQL database** (using `pgx` driver). The log table stores the full SMTP conversation dump (up to 4KB per attempt), the `Message‑ID` from Gmail, and a **retry‑count** column. A background worker (`reaper`) runs every 60 seconds to prune entries older than 30 days. The system exposes a **REST API** (`/api/v1/recipients/{id}/status`) to query the current state and retry history for any recipient.

## How to Use

### Step 1: Configure the Retry Profile and Gmail Quota
Create a `retrybeast.yaml` configuration file:
```yaml
retry:
  max_attempts: 7
  backoff_strategy: "fibonacci"
  base_interval_ms: 30000
  max_interval_ms: 3600000
  jitter_percent: 10
  dead_letter_threshold: 3
quota:
  daily_limit: 2000
  per_second_limit: 5
  quota_exhaustion_threshold_percent: 90
database:
  dsn: "postgres://user:pass@localhost:5432/gmail_retry?sslmode=disable"
  max_connections: 25
```
Validate the config using:
```bash
gmailretrybeast --validate-config ./retrybeast.yaml
```
The tool will parse the YAML, check the Gmail SMTP endpoint (`smtp.gmail.com:587`) connectivity, and report any misconfigurations (e.g., `backoff_strategy` not in `[linear, exponential, fibonacci]`).

### Step 2: Seed the Retry Queue with Failed Messages
Use the `enqueue` subcommand to inject messages that previously failed during a primary delivery attempt. The tool expects a CSV file with columns: `recipient_email`, `subject`, `body`, `original_error_code`, and `original_timestamp`:
```bash
gmailretrybeast enqueue --input ./failed_deliveries.csv --config ./retrybeast.yaml --batch-size 50
```
The enqueuer validates each email address against RFC 5321, normalizes the domain to lowercase, and stores the record in PostgreSQL with an initial state of `PENDING`. The `--batch-size` flag controls the number of rows inserted per transaction (default: 100). After enqueuing, run `gmailretrybeast status --queue-depth` to see the total count of pending messages.

### Step 3: Start the Retry Daemon and Monitor Metrics
Launch the retry engine in daemon mode:
```bash
gmailretrybeast daemon --config ./retrybeast.yaml --log-level debug
```
The daemon will:
- Open a connection pool to PostgreSQL (25 connections by default).
- Start the **Quota Guardian** background goroutine that pings Gmail SMTP every 30 seconds to refresh the `Retry‑After` state.
- Begin dispatching retry attempts from the `PENDING` queue at a rate controlled by `per_second_limit`.
- Expose a **Prometheus metrics endpoint** at `http://localhost:2112/metrics`.

To see real‑time retry progress, query the state machine:
```bash
curl -s http://localhost:2112/metrics | grep gmail_retry
```
Example output:
```
gmail_retry_attempts_total{state="success"} 142
gmail_retry_attempts_total{state="transient_failure"} 37
gmail_send_latency_seconds{quantile="0.95"} 1.23
```

## Best Practices

### 1. Pair with a Dedicated Dead‑Letter Queue and Manual Review Workflow
Configure the `dead_letter_threshold` to 3 or lower. When a recipient exceeds this threshold, the message is moved to a `DEAD_LETTER` table in PostgreSQL. Set up a **Slack webhook** (via the `--webhook-url` flag) to notify the operations team immediately. The dead‑letter queue should be reviewed every 12 hours: inspect the full SMTP dump for patterns (e.g., repeated `550 5.1.1` means the address is invalid; `450 4.7.0` with a specific `X‑Gm‑MessageState` may indicate IP reputation issues). Use the REST API to manually requeue or permanently discard entries.

### 2. Implement a Staged Rollout for Retry Window Adjustments
Before changing the `max_attempts` or `base_interval_ms` in production, run a **shadow mode** test. Use the `--dry-run` flag on the daemon: it will simulate the retry schedule without actually sending emails, but will log every decision and the expected SMTP response code. Compare the simulated success rate against historical data over a 24‑hour window. Only promote the new configuration after verifying that the simulated retry chain does not exceed the daily quota by more than 5%. This prevents accidental quota overruns during peak hours.
