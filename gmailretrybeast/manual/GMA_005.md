# gmailretrybeast - GMA_005

## Overview

`gmailretrybeast` is a high-reliability Gmail API retry engine designed for mission-critical email delivery systems. It eliminates transient failures in Gmail SMTP and REST API calls through exponential backoff, circuit-breaking, and parallel retry orchestration. Built for DevOps pipelines, transactional email services, and bulk notification systems that demand 99.99% delivery success rates.

## Core Features

### 1. Adaptive Retry Scheduler with Jitter-Aware Backoff
- **Parameters:** `base_delay_ms=200`, `max_delay_ms=30000`, `jitter_factor=0.25`, `retry_multiplier=2.0`
- Implements a **full-jitter exponential backoff** algorithm to avoid thundering herd problems on Gmail API rate limits.
- Dynamically adjusts retry intervals based on real-time HTTP 429 / 503 response headers (`Retry-After`).
- Supports **millisecond-precision scheduling** via internal priority queue, ensuring no retry is fired earlier than the calculated next attempt timestamp.

### 2. Multi-Strategy Failure Classification & Circuit Breaker
- **Parameters:** `failure_threshold=5`, `recovery_timeout_s=60`, `half_open_max_requests=3`
- Automatically classifies failures into **transient** (network timeouts, 5xx) vs. **permanent** (auth errors, 400 bad request). Only transient failures trigger retries.
- Integrated **circuit breaker** (state: CLOSED → OPEN → HALF-OPEN) prevents wasted API calls when Gmail endpoint is degraded.
- Logs each state transition with full stack trace and request fingerprint to `gmailretrybeast_events` PostgreSQL table.

### 3. Idempotency Key Generation & Deduplication Engine
- **Parameters:** `idempotency_key_length=32`, `key_ttl_seconds=86400`, `storage_backend=postgresql`
- For every outgoing Gmail API call, generates a **cryptographically random idempotency key** (SHA-256 based) stored in PostgreSQL with TTL.
- On retry, the engine reuses the same key, guaranteeing that Gmail processes the request exactly once—even if the response was lost.
- Built-in **garbage collector** runs every 3600 seconds to purge expired keys, maintaining < 1ms lookup latency.

## How to Use

### Step 1: Configure Retry Profile
Create a `retrybeast_config.json` file with your Gmail API credentials and retry parameters:
```json
{
  "gmail": {
    "client_id": "your-oauth-client-id",
    "client_secret": "your-oauth-client-secret",
    "refresh_token": "your-refresh-token",
    "scopes": ["https://www.googleapis.com/auth/gmail.send"]
  },
  "retry_policy": {
    "base_delay_ms": 200,
    "max_delay_ms": 30000,
    "jitter_factor": 0.25,
    "retry_multiplier": 2.0,
    "failure_threshold": 5,
    "recovery_timeout_s": 60,
    "half_open_max_requests": 3,
    "idempotency_key_ttl_seconds": 86400
  },
  "storage": {
    "postgresql_dsn": "postgresql://user:pass@localhost:5432/retrybeast?sslmode=require"
  }
}
```

### Step 2: Initialize Engine & Enqueue a Message
Use the CLI or SDK to enqueue an email with full payload:
```bash
gmailretrybeast enqueue \
  --config /etc/retrybeast_config.json \
  --to "user@example.com" \
  --subject "Critical Alert: Node Down" \
  --body "<html>...</html>" \
  --priority "high" \
  --max_retries 10
```
The engine immediately registers the message in the internal queue and assigns a unique `request_id` (UUIDv4). The idempotency key is generated and persisted to PostgreSQL before the first API call.

### Step 3: Monitor Retry Lifecycle via Dashboard
Start the built-in monitoring server:
```bash
gmailretrybeast monitor --port 9090 --config /etc/retrybeast_config.json
```
Access `http://localhost:9090/metrics` to view:
- **`gmail_retries_total{status="success"}`** – cumulative successful deliveries
- **`gmail_retries_total{status="failed_permanent"}`** – permanent failures after exhaustion
- **`gmail_circuit_breaker_state`** – current state (0=CLOSED, 1=OPEN, 2=HALF-OPEN)
- **`gmail_backoff_duration_ms`** – histogram of actual backoff delays applied

## Best Practices

### 1. Pair with Idempotent Mailbox Processing
Always configure your downstream email processor (e.g., a webhook receiver) to deduplicate based on `Message-Id` or a custom header injected by `gmailretrybeast`. Even with idempotency keys at the API layer, Gmail may deliver duplicate messages in rare network split scenarios. Use the engine's `x-retrybeast-id` header to filter duplicates at the application level.

### 2. Pre-Warm the Connection Pool for Bulk Campaigns
When sending >10,000 messages per hour, pre-initialize the engine's HTTP connection pool with `max_connections=50` and `keepalive_timeout_s=300`. This avoids connection establishment overhead during retry bursts. Additionally, run a `gmailretrybeast healthcheck` command before the campaign to validate OAuth token freshness and PostgreSQL connectivity—this reduces the risk of permanent failures due to expired credentials mid-campaign.
