# gmailretrybeast - GMA_022

## Overview

gmailretrybeast is a high-resilience email delivery engine engineered to eliminate silent Gmail API failures, transient SMTP back-offs, and rate-limit-induced message drops. It operates as a stateful retry orchestrator with exponential backoff, circuit-breaker patterns, and real-time Gmail quota introspection.

## Detailed Content

### What is gmailretrybeast?

gmailretrybeast is a production-grade, event-driven retry framework purpose-built for Gmail API `users.messages.send` and `users.threads.*` endpoints. It intercepts HTTP 429/503/500 responses, classifies error codes against Google's documented transient fault taxonomy, and executes retry sequences with jitter-aware backoff (base interval: 250ms, max interval: 64s). It maintains an internal PostgreSQL-backed state machine to track delivery attempts, idempotency keys, and per-thread retry budgets.

### Core Features

- **Adaptive Quota-Aware Retry Scheduler**: Monitors Gmail API `rateLimitExceeded` and `dailyLimitExceeded` errors in real-time. Dynamically adjusts retry concurrency (max 8 parallel workers) based on the `X-RateLimit-Remaining` header and the `quotaRemaining` field from the `gmail.users.getProfile` response. Uses a sliding-window token bucket with a configurable capacity (default: 250 tokens per 100 seconds) and refill rate (2.5 tokens/sec) to prevent cascading quota exhaustion.

- **Idempotent Message Replay with Payload Integrity Check**: Generates SHA-256 content hashes for each MIME message body. On retry, compares the stored hash against the original payload to detect and reject any data corruption. Persists the full `Message` object (raw base64url-encoded string) in a PostgreSQL `retry_queue` table with columns: `message_id`, `thread_id`, `payload_hash`, `retry_count`, `next_retry_at` (TIMESTAMPTZ with millisecond precision), and `error_class`. Supports batch replay of up to 500 messages per transaction using `COPY` statements for bulk insert.

- **Circuit Breaker with Gmail Backend Health Probe**: Implements a three-state circuit (CLOSED, OPEN, HALF_OPEN) driven by a 60-second rolling error rate. If the error rate exceeds 25% of total requests within the window, the circuit opens for 30 seconds. During HALF_OPEN, a single probe request is sent to `gmail.users.getProfile` (with `fields=userId`). If the probe succeeds, the circuit resets; otherwise, it reopens with a 2x backoff (max 300 seconds). All state transitions are logged to a `circuit_events` table with `event_type`, `timestamp` (UTC, microsecond resolution), and `reason` fields.

### How to Use

1. **Initialize the Retry Beast with Gmail API Credentials and PostgreSQL Connection**:
   ```bash
   gmailretrybeast init \
     --credentials-path /etc/gmail/service-account.json \
     --db-host postgres-cluster-1.internal \
     --db-port 5432 \
     --db-name gmail_retry \
     --db-user retry_beast \
     --db-password "$(vault read -field=password secret/gmailretrybeast/db)" \
     --quota-capacity 250 \
     --quota-refill-rate 2.5 \
     --max-workers 8
   ```
   This command validates the service account's `https://www.googleapis.com/auth/gmail.send` scope, tests the PostgreSQL connection with `SELECT 1`, and creates the `retry_queue` and `circuit_events` tables if they do not exist. It also initializes the token bucket with the specified capacity and refill rate.

2. **Enqueue a Batch of Failed Gmail Messages for Retry**:
   ```bash
   gmailretrybeast enqueue \
     --input-file /var/log/gmail_failures.ndjson \
     --batch-size 500 \
     --priority urgent \
     --max-retries 5 \
     --base-backoff-ms 250 \
     --max-backoff-ms 64000 \
     --jitter-factor 0.2
   ```
   The input NDJSON file must contain objects with `message_id` (string), `raw_mime` (base64url-encoded string), `thread_id` (string, optional), and `original_error` (string). The tool validates each payload's base64url encoding and SHA-256 hash, then inserts them into the `retry_queue` table using a single `COPY` statement. The `priority` column (enum: `urgent`, `normal`, `low`) influences the ordering in the scheduler's priority queue.

3. **Execute the Retry Loop with Real-Time Monitoring**:
   ```bash
   gmailretrybeast run \
     --daemon \
     --poll-interval-ms 1000 \
     --circuit-breaker-error-threshold 0.25 \
     --circuit-breaker-window-seconds 60 \
     --circuit-breaker-open-seconds 30 \
     --circuit-breaker-max-backoff-seconds 300 \
     --log-level debug \
     --metrics-port 9090
   ```
   The daemon enters an infinite loop, polling the `retry_queue` for messages where `next_retry_at <= NOW()` and `retry_count < max_retries`. It processes up to `max-workers` messages concurrently, respecting the token bucket. For each message, it calls `gmail.users.messages.send` with the original raw MIME. On success, it deletes the row from `retry_queue`. On failure, it increments `retry_count`, calculates the next retry time using exponential backoff with jitter (`base_backoff * (2^retry_count) + random(0, jitter_factor * base_backoff * (2^retry_count))`), and updates `next_retry_at`. The circuit breaker monitors the rolling error rate and opens/closes accordingly. Prometheus metrics are exposed on port 9090.

### Best Practices

- **Implement a Dead-Letter Queue (DLQ) for Permanently Failed Messages**: Configure a separate PostgreSQL table `retry_dlq` with identical schema to `retry_queue`, plus an additional `failure_reason` column. Set `max-retries` to 5. After all retries are exhausted, the `run` daemon should automatically move the message to the DLQ, appending the final error code (e.g., `error_class: "backendError", error_detail: "HTTP 503: Backend Error"`). Periodically review the DLQ with `SELECT * FROM retry_dlq ORDER BY next_retry_at DESC LIMIT 100` to identify systemic issues like revoked API scopes or misconfigured service accounts.

- **Use a Dedicated Gmail API Service Account with Domain-Wide Delegation**: To avoid conflicts with user-specific rate limits, create a service account in Google Cloud Console and delegate domain-wide authority for the `gmail.send` scope. Assign this service account to a dedicated organizational unit with a separate daily quota (default: 1,500,000,000 quota units/day). Monitor the `quotaRemaining` field from `gmail.users.getProfile` every 60 seconds using a background goroutine. If `quotaRemaining` drops below 10% of the daily limit, reduce `max-workers` to 1 and increase `base-backoff-ms` to 1000 to ensure critical messages are not lost.
