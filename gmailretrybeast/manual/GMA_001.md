# gmailretrybeast - GMA_001

## Overview

gmailretrybeast is a high-throughput, fault-tolerant email retry engine designed for Gmail API rate limit exhaustion and transient SMTP failures. It implements exponential backoff with jitter, supports multi-account rotation, and provides real-time delivery diagnostics via structured logging.

## Detailed Content

### What is gmailretrybeast?

gmailretrybeast is a production-grade retry orchestrator that intercepts Gmail API `429` and `5xx` errors, applies configurable retry policies per message priority, and persists failed deliveries to PostgreSQL for forensic analysis. It leverages the Gmail API `users.messages.send` endpoint with OAuth 2.0 scoped to `https://www.googleapis.com/auth/gmail.send`.

### Core Features

- **Adaptive Retry with Token Bucket Throttling**  
  Dynamically adjusts retry intervals based on real-time Gmail API quota consumption (`quotaRemaining` header). Supports configurable `initialIntervalMs` (default: 1000ms), `maxIntervalMs` (default: 60000ms), and `multiplier` (default: 2.0) with full-jitter randomization to avoid thundering herd.

- **Multi-Account Failover with Health Checks**  
  Manages a pool of Gmail accounts via `--accounts-file accounts.json` (supports up to 50 accounts). Automatically rotates to next healthy account when current one hits rate limit or token expiry. Runs periodic OAuth token refresh checks every `--health-check-interval 300s` via `oauth2.TokenSource`.

- **Delivery Confirmation with Idempotency Keys**  
  Assigns a UUID-based `idempotencyKey` per message to prevent duplicate sends on retry. Logs delivery status to PostgreSQL with schema: `deliveries(id UUID, status TEXT, retry_count INT, latency_ms INT, error_code TEXT)`. Supports `--pg-conn-string "postgres://user:pass@host:5432/gmailretrybeast"`.

### How to Use

1. **Initialize Account Pool and Validate Credentials**  
   Run `gmailretrybeast init --accounts-file accounts.json --scopes "https://www.googleapis.com/auth/gmail.send"`. This validates each account's OAuth 2.0 credentials, tests `users.getProfile` for connectivity, and writes a `accounts_verified.json` with `health_status: "ok"` or `"error"` per account.

2. **Submit Batch with Custom Retry Policy**  
   Execute `gmailretrybeast send --input messages.csv --batch-size 100 --retry-policy '{"maxRetries":5,"initialIntervalMs":2000,"maxIntervalMs":120000}' --idempotency-key-field "message_id"`. The tool parses CSV with columns `to`, `subject`, `body`, `message_id`, and streams sends with per-message retry counters stored in memory.

3. **Analyze Delivery Failures and Requeue**  
   After run, query `SELECT * FROM deliveries WHERE status = 'failed' AND retry_count < 5 ORDER BY created_at DESC LIMIT 100` via `gmailretrybeast report --db-query "SELECT id, error_code FROM deliveries WHERE status='failed'"`. Then requeue using `gmailretrybeast requeue --ids-file failed_ids.txt --new-retry-policy '{"maxRetries":3,"initialIntervalMs":5000}'`.

### Best Practices

- **Prefer Dedicated Gmail Service Accounts**  
  Use Google Workspace service accounts with domain-wide delegation instead of personal Gmail accounts. This avoids the 500-emails-per-day limit for personal accounts and provides a clean quota of 1,000,000 API requests per day per project. Configure via `--auth-type service-account --service-account-key key.json`.

- **Implement Circuit Breaker for Sustained Failures**  
  When consecutive failures exceed `--circuit-breaker-threshold 10` within a `--circuit-breaker-window 60s`, pause all sends to that account for `--cooldown-period 300s`. Monitor via `/metrics` endpoint exposing Prometheus gauge `gmail_retry_beast_account_status{account="user@gmail.com"}`.
