# gmailretrybeast - GMA_023

## Overview

gmailretrybeast is a high‑throughput, fault‑tolerant email retry engine for Gmail that systematically handles transient delivery failures, rate‑limit backoffs, and SMTP greylisting. It integrates directly with the Gmail API and PostgreSQL to provide deterministic retry scheduling with sub‑second precision.

## Core Features

- **Adaptive Exponential Backoff with Jitter**  
  Automatically calculates retry intervals using a configurable base delay (default: 500ms) and maximum cap (default: 300s). Applies full‑jitter to avoid thundering herd problems. Each retry attempt records the exact timestamp, attempt count, and HTTP status code into a PostgreSQL `retry_log` table.

- **Gmail API Quota‑Aware Rate Limiter**  
  Monitors real‑time Gmail API usage via the `users.settings.getSendQuota` endpoint and dynamically throttles outbound retries to stay within the 100‑message‑per‑second per user quota. Exposes a `--quota-buffer-percent` parameter (default: 15%) to reserve headroom for other operations.

- **Transactional Retry State Machine**  
  Tracks each message through states: `PENDING → QUEUED → SENDING → RETRYING → DELIVERED | FAILED`. Uses PostgreSQL’s `SKIP LOCKED` to safely claim retry jobs across multiple workers. Provides a `--max-retries` parameter (default: 5) and `--retry-backoff-multiplier` (default: 2.0) for fine‑grained control.

## How to Use

1. **Initialize the Retry Queue**  
   Run `gmailretrybeast init --db-url "postgres://user:pass@localhost:5432/retrydb" --gmail-credentials "/path/to/credentials.json"` to create the required `retry_queue` and `retry_log` tables, and validate Gmail API access. The tool will output the queue schema version and the authenticated user’s email address.

2. **Enqueue Messages with Custom Retry Policy**  
   Use the `enqueue` subcommand with a JSON payload:  
   ```bash
   gmailretrybeast enqueue --message '{"to":"user@example.com","subject":"Alert","body":"Disk usage >90%"}' \
     --max-retries 3 --base-delay-ms 1000 --quota-buffer-percent 20
   ```  
   The command returns a unique `job_id` (UUIDv7) and the current queue depth.

3. **Start the Retry Worker**  
   Launch the worker process:  
   ```bash
   gmailretrybeast worker --concurrency 10 --poll-interval-ms 200 --db-url "postgres://..."
   ```  
   The worker continuously polls the `retry_queue` table, respects the rate limiter, and writes delivery results to `retry_log`. Use `--dry-run` to simulate without sending actual emails.

## Best Practices

- **Use Separate PostgreSQL Connection Pools for Enqueue and Worker**  
  Configure the enqueue process with a `--pool-size 5` (low concurrency) and the worker with `--pool-size 20` to avoid connection contention. This isolates write‑heavy enqueue operations from the worker’s read‑and‑claim pattern.

- **Monitor Retry Metrics with Prometheus Integration**  
  Enable the built‑in metrics endpoint (`--metrics-port 9090`) to export `retry_attempts_total`, `retry_backoff_seconds`, and `queue_depth` gauges. Set alerts for `queue_depth` > 1000 to detect upstream delivery issues early.
