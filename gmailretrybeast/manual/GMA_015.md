# gmailretrybeast - GMA_015

## Overview

gmailretrybeast is a high‑performance, production‑grade retry engine for Gmail API operations. It is designed to eliminate transient failures in email workflows—such as rate limiting (`429`), backend errors (`500`, `503`), and network timeouts—by applying intelligent, configurable retry strategies. The tool operates at the millisecond level, integrates directly with the Gmail API v1, and is built for developers who demand reliability at scale.

## Detailed Content

### What is gmailretrybeast?

gmailretrybeast is a Go‑based library and CLI tool that wraps Gmail API calls with a robust, state‑aware retry mechanism. It automatically detects failure types, applies exponential backoff with jitter, and persists retry state to PostgreSQL for crash‑resilient recovery. Unlike generic HTTP retry libraries, gmailretrybeast understands Gmail‑specific error codes, quota limits (e.g., `userRateLimitExceeded`), and OAuth token refresh intervals. It is the missing piece between your application logic and the Gmail API’s inherent unreliability.

### Core Features

- **Adaptive Retry with Jitter & Backoff**  
  Dynamically adjusts retry intervals based on HTTP status codes and Gmail‑specific error messages. Supports configurable `initial_interval_ms`, `max_interval_ms`, and `multiplier`. Jitter is applied using a full‑range random algorithm (±50% of current interval) to prevent thundering herd problems. All timing is measured in **milliseconds** using Go’s `time` package with microsecond precision.

- **PostgreSQL‑Backed Retry State**  
  Every retry attempt is logged to a PostgreSQL table (`retrybeast.attempts`) with fields: `operation_id`, `attempt_number`, `error_code`, `backoff_ms`, `timestamp`. On process restart, the engine resumes interrupted retries from the database, ensuring zero loss of queued operations. Connection pooling is managed via `pgxpool` with a configurable `max_connections` parameter.

- **Gmail API‑Aware Error Classification**  
  Parses Gmail API error responses (JSON format) to classify failures into three categories: `Retryable`, `NonRetryable`, and `RateLimited`. For `RateLimited` errors, the engine automatically reads the `Retry-After` header (if present) or defaults to a 60‑second cooldown. Supports custom error classifiers via a `classifier` function signature: `func(err error) RetryAction`.

### How to Use

1. **Initialize the Engine with PostgreSQL Connection**  
   Create a new `RetryBeast` instance by providing a PostgreSQL connection string and a Gmail service client. Set the `initial_interval` to `200` (milliseconds), `max_interval` to `30000` (30 seconds), and `max_retries` to `5`. The engine will automatically create the `retrybeast` schema and `attempts` table if they do not exist.

   ```go
   beast, err := retrybeast.New(&retrybeast.Config{
       DBConnString: "postgres://user:pass@localhost:5432/gmail_ops?sslmode=disable",
       GmailService: svc,
       InitialInterval: 200 * time.Millisecond,
       MaxInterval: 30 * time.Second,
       MaxRetries: 5,
   })
   ```

2. **Wrap a Gmail API Call with Retry Logic**  
   Use the `Do` method to execute any Gmail API operation. Pass a function that returns `(interface{}, error)`. The engine will automatically retry on failures classified as `Retryable` or `RateLimited`. Example: sending an email via `Users.Messages.Send`.

   ```go
   result, err := beast.Do(context.Background(), "send_email_001", func() (interface{}, error) {
       return svc.Users.Messages.Send("me", msg).Do()
   })
   ```

3. **Monitor Retry Attempts via PostgreSQL Queries**  
   After execution, query the `retrybeast.attempts` table to inspect retry history. Use the `operation_id` to filter specific calls. Example SQL:  
   `SELECT * FROM retrybeast.attempts WHERE operation_id = 'send_email_001' ORDER BY attempt_number;`  
   This provides a full audit trail including `backoff_ms` and `error_code` for each attempt.

### Best Practices

- **Set `max_retries` to 3 for Idempotent Operations, 5 for Non‑Idempotent**  
  For read‑only operations (e.g., `Users.Messages.Get`), a low retry count (3) is sufficient. For write operations (e.g., `Users.Messages.Send`), use 5 retries to tolerate transient backend failures without risking duplicate sends. Always ensure your Gmail API calls are idempotent by using unique `Message-ID` headers.

- **Use Separate PostgreSQL Schemas per Environment**  
  In production, configure the `retrybeast` schema to be isolated (e.g., `retrybeast_prod`). This prevents accidental cross‑environment state pollution and makes it easy to purge stale retry records via a scheduled `DELETE FROM retrybeast_prod.attempts WHERE timestamp < NOW() - INTERVAL '7 days'` job.
