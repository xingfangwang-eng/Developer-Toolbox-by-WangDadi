# gmailretrybeast - GMA_016

## Overview

`gmailretrybeast` is a high‑throughput, fault‑tolerant retry engine engineered for mission‑critical Gmail API workflows. It eliminates silent delivery failures by implementing exponential backoff with jitter, real‑time quota monitoring, and multi‑layer fallback routing. Designed for developers and ops teams who demand sub‑second recovery and zero message loss in production email pipelines.

## Detailed Content

### What is gmailretrybeast?

`gmailretrybeast` is a purpose‑built retry orchestrator for the Gmail API. It intercepts transient failures (rate limits, 5xx errors, network timeouts) and automatically replays failed operations with configurable retry strategies. The beast tracks every attempt with millisecond precision, persists state to PostgreSQL, and exposes a live dashboard for observability. It is not a generic retry library—it is a hardened system tailored to the quirks of Gmail’s `users.messages.send` and `users.threads.get` endpoints.

### Core Features

- **Adaptive Quota‑Aware Backoff**  
  Reads real‑time Gmail API quota usage (`X‑Quota‑Remaining` headers) and dynamically adjusts retry intervals. When remaining quota drops below `100` requests per second, the beast automatically throttles to `250ms` minimum delay, preventing further 429s. Jitter is injected using `random.uniform(0, 0.5 * base_delay)` to avoid thundering herd.

- **Stateful Retry with PostgreSQL Persistence**  
  Every failed operation is recorded in a `retry_queue` table with fields: `operation_id`, `http_method`, `endpoint`, `request_body_hash`, `attempt_count`, `next_retry_at`, and `error_code`. A background worker polls for rows where `next_retry_at <= NOW()` and `attempt_count < max_retries` (default: `5`). This ensures zero data loss across process restarts.

- **Multi‑Stage Fallback Routing**  
  If a Gmail API call fails after exhausting all retries, the beast executes a chain of fallback actions: (1) write the failed payload to a dead‑letter S3 bucket, (2) send a structured alert to a configured Slack webhook, and (3) insert a compensating event into a secondary `dlq_events` table for manual review. Each fallback stage is independently configurable via environment variables.

### How to Use

1. **Initialize the Retry Queue and PostgreSQL Schema**  
   ```bash
   gmailretrybeast init --db-host localhost --db-port 5432 --db-name gmail_retry --db-user admin --db-password secret
   ```  
   This creates the `retry_queue` and `dlq_events` tables with proper indexes on `next_retry_at` and `attempt_count`. The command also sets up a `pg_cron` job to vacuum the tables every 6 hours.

2. **Configure the Retry Strategy and Quota Limits**  
   ```bash
   gmailretrybeast config set --max-retries 5 --base-delay-ms 2000 --max-delay-ms 60000 --quota-threshold 100 --slack-webhook https://hooks.slack.com/services/T00/B00/xxx
   ```  
   This sets exponential backoff: `delay = min(2000 * 2^(attempt-1), 60000)` plus jitter. The quota threshold ensures the beast pauses if Gmail reports fewer than `100` remaining requests per second.

3. **Start the Retry Worker Daemon**  
   ```bash
   gmailretrybeast run --workers 4 --poll-interval-ms 500 --log-level debug
   ```  
   Launches 4 concurrent goroutines that poll the `retry_queue` every `500ms`. Each worker picks up a batch of up to `50` operations, replays them against the Gmail API, and updates the row with the new `attempt_count` and `next_retry_at`. All retry decisions are logged with millisecond timestamps.

### Best Practices

- **Always Set a Realistic `max‑retries` and `base‑delay‑ms`**  
  Gmail’s daily sending limits for free accounts are `500` messages. For production, set `max‑retries` to `5` and `base‑delay‑ms` to `2000`. This gives a total retry window of roughly `62` seconds per message, which is safe for most transient errors. For paid Workspace accounts (unlimited sending), you can lower `base‑delay‑ms` to `500` and increase `max‑retries` to `10`.

- **Monitor the Dead‑Letter Queue and Alert Immediately**  
  Configure `gmailretrybeast` to stream `dlq_events` to a centralized logging system (e.g., Elasticsearch or Datadog). Set up an alert if the number of dead‑lettered operations exceeds `10` per hour. This often indicates a permanent misconfiguration (e.g., invalid OAuth scope) or a Gmail API outage that requires manual intervention.
