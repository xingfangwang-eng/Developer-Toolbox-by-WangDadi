# gmailretrybeast - GMA_029

## Overview

`gmailretrybeast` is a high-throughput, fault-tolerant email retry engine designed specifically for Gmail API v1. It handles transient failures, rate limits, and quota exhaustion with exponential backoff and intelligent queue management. This tool is purpose-built for mission-critical email workflows where delivery guarantees and latency predictability are non-negotiable.

## Core Features

### 1. Adaptive Retry with Jittered Exponential Backoff
Implements a custom retry strategy that dynamically adjusts intervals based on Gmail API response headers (`Retry-After`, `X-Goog-RateLimit-Reset`). Supports configurable base delay (default: `250ms`), maximum backoff cap (`120s`), and full jitter randomization. Each retry attempt logs the exact HTTP status code, response body snippet (truncated to 512 bytes), and elapsed wall-clock time in milliseconds.

### 2. Concurrency-Bounded Queue with Per-User Rate Limiting
Manages a thread-safe, bounded work queue (capacity: `10,000` tasks) using a priority heap sorted by next retry timestamp. Enforces per-user Gmail quota limits via a token bucket algorithm (default: `250` requests per user per 100 seconds, configurable). Supports dynamic throttling based on real-time `quotaRemaining` values parsed from API responses.

### 3. Persistent State with PostgreSQL Checkpointing
Stores all pending, in-flight, and completed retry tasks in a PostgreSQL database with a dedicated schema (`gmail_retry_beast`). Each task record includes: `message_id` (Gmail UUID), `retry_count` (integer, max 10), `last_error` (text, up to 2048 chars), `next_retry_at` (timestamptz, millisecond precision), and `user_email` (indexed). Supports crash recovery by reloading unacknowledged tasks on startup.

## How to Use

### Step 1: Initialize the Retry Queue and Database Schema
Execute the provided migration script (`init_schema.sql`) against your PostgreSQL instance. This creates the `tasks` table with a composite index on `(user_email, next_retry_at)`. Then, seed the queue with failed Gmail API calls by passing a JSON array of `{ "userId": "...", "rawMessage": "base64url", "originalError": "..." }` objects via the `POST /v1/queue/seed` endpoint.

```bash
curl -X POST http://localhost:8080/v1/queue/seed \
  -H "Content-Type: application/json" \
  -d '[{"userId": "user@example.com", "rawMessage": "RnJvbTogdXNlckBleGFtcGxlLmNvbQ==", "originalError": "429 Too Many Requests"}]'
```

### Step 2: Configure Retry Policy and Rate Limits
Set environment variables in your deployment file (e.g., `docker-compose.yml` or Kubernetes ConfigMap). Key parameters include `RETRY_BASE_DELAY_MS=250`, `RETRY_MAX_DELAY_SEC=120`, `RATE_LIMIT_PER_USER=250`, `RATE_LIMIT_WINDOW_SEC=100`, and `DB_CONNECTION_STRING=postgres://user:pass@host:5432/gmail_retry`. Optionally, enable verbose logging by setting `LOG_LEVEL=debug`.

### Step 3: Launch the Beast and Monitor Throughput
Start the service with `./gmailretrybeast --config config.yaml`. The tool will immediately begin processing the queue, respecting per-user rate limits and backoff schedules. Monitor real-time metrics via the built-in Prometheus endpoint (`/metrics`): track `retry_attempts_total`, `retry_success_total`, `retry_failed_total`, `queue_depth`, and `rate_limit_remaining`. Use the `GET /v1/queue/status?user_email=...` endpoint to inspect a specific user's pending tasks.

## Best Practices

### 1. Integrate with Gmail Push Notifications for Immediate Retry Triggering
Instead of polling the Gmail API for failed sends, set up a Google Cloud Pub/Sub subscription that pushes `watch` notifications to your `gmailretrybeast` instance. When a message delivery fails, the tool subscribes to the `gmail-delivery-failure` topic and instantly enqueues the task. This reduces detection latency from seconds to milliseconds and avoids unnecessary API calls.

### 2. Pre-Compute Retry Windows to Avoid Thundering Herd Problems
When recovering from a prolonged outage (e.g., Gmail API downtime), do not enqueue all failed tasks with the same `next_retry_at` timestamp. Instead, spread them across a configurable window (e.g., `RECOVERY_SPREAD_SEC=300`) using a hash of the `user_email` and `message_id` to deterministically assign staggered retry times. This prevents a burst of simultaneous API calls that could trigger secondary rate limits.
