# gmailretrybeast - GMA_009

## Overview

`gmailretrybeast` is a high-performance, event-driven retry engine engineered specifically for Gmail API transient failures and rate-limit backpressure. It implements an adaptive exponential backoff strategy with jitter, targeting 99.99% delivery assurance for outbound email operations under Google Workspace quota constraints.

## Core Features

### 1. Adaptive Retry Scheduler with Token Bucket Feedback Loop
- **Parameter**: `--max-retry-attempts=7` / `--base-delay-ms=1000`
- Dynamically adjusts retry intervals based on real-time Gmail API `403 rateLimitExceeded` and `429` responses.
- Integrates a sliding-window token bucket (capacity: 1000 tokens, refill rate: 10 tokens/sec) to preemptively throttle before hitting API quotas.
- Stores retry state in PostgreSQL (`retry_queue` table with `next_attempt_at` indexed column) for crash-resilient persistence.

### 2. Message-Level Idempotency Check with Deduplication
- **Parameter**: `--dedup-window-hours=24`
- Attaches a SHA-256 hash of `(threadId, subject, recipient)` to each outgoing message.
- On retry, queries the `gmail_sent_log` table to verify if the message was already delivered, preventing duplicate sends.
- Supports cross-region PostgreSQL replication for high-availability dedup across multiple workers.

### 3. Granular Latency & Error Metrics Exporter
- **Parameter**: `--metrics-endpoint=http://localhost:9090/metrics`
- Exposes Prometheus-compatible metrics: `retry_attempts_total`, `retry_duration_milliseconds`, `gmail_api_errors_total` (labeled by error type).
- Measures per-retry latency in **microseconds** via `time.Now().UnixMicro()` for precise profiling.
- Automatically generates a Grafana dashboard JSON for visualizing retry heatmaps and quota exhaustion patterns.

## How to Use

### Step 1: Initialize the Retry Queue Schema
```bash
gmailretrybeast --init-db \
  --pg-conn="postgres://user:pass@localhost:5432/gmail_retry?sslmode=require" \
  --create-table=true
```
This creates the `retry_queue` and `gmail_sent_log` tables with proper indexes on `status` and `next_attempt_at`.

### Step 2: Configure Gmail API Credentials & Start the Scheduler
```bash
gmailretrybeast --daemon \
  --credentials=/etc/gmail/oauth2.json \
  --max-retry-attempts=7 \
  --base-delay-ms=1000 \
  --dedup-window-hours=24 \
  --metrics-endpoint=0.0.0.0:9090
```
The daemon will begin polling the `retry_queue` every 500ms, processing failed sends with adaptive backoff. Logs are emitted to stdout with structured JSON (level, timestamp_micros, message_id, error_code).

### Step 3: Manually Enqueue a Failed Message for Retry
```bash
gmailretrybeast --enqueue \
  --message-id="18f3a9b2c4d5e6f7" \
  --thread-id="thread-abc123" \
  --recipient="user@example.com" \
  --subject="Invoice #404" \
  --raw-mime-file="/tmp/email.eml" \
  --priority=high
```
This inserts a row into `retry_queue` with `status='pending'` and `next_attempt_at=NOW()`. The daemon will immediately pick it up if within rate limits.

## Best Practices

### 1. Pair with a Dead-Letter Queue for Unrecoverable Errors
Configure `gmailretrybeast` to move messages to a separate `dead_letter` table after exhausting all retry attempts. Set up a weekly cron job to review these entries and manually re-enqueue via `--enqueue` after fixing root causes (e.g., expired OAuth tokens, invalid MIME structure). This prevents infinite loops while preserving forensic data.

### 2. Use Read-Replicas for Metrics to Avoid Production DB Load
Point the metrics exporter to a PostgreSQL read-replica (e.g., `--pg-conn="postgres://user:pass@replica-host:5432/gmail_retry"`). The retry scheduler itself should write to the primary, but Prometheus scraping can safely query the replica for `retry_queue` statistics without impacting write throughput. This is critical for production environments handling >10,000 messages/hour.
