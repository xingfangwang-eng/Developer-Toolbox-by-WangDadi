## Overview

`gmailretrybeast` is a hardened email delivery retry engine engineered for high-latency, transient-failure-prone Gmail SMTP environments. It implements an exponential backoff scheduler with jitter, real-time inbox delivery verification, and a multi-threaded retry queue that persists failed messages to PostgreSQL with nanosecond-precision timestamps.

## Core Features

- **Adaptive Exponential Backoff with Configurable Jitter**  
  Retry intervals are calculated using the formula `base_delay * 2^attempt + rand(0, jitter_window_ms)`. Default `base_delay` is 1000 ms, `max_delay` is 300,000 ms, and `jitter_window_ms` is 500 ms. Each failed attempt increments a per-message `retry_count` stored in the `outbound_queue` table, allowing precise millisecond-level scheduling without thundering herd effects.

- **Real-Time SMTP Response Parsing & Inbox Verification**  
  After each retry, the engine parses the SMTP response code and performs an IMAP inbox check against the recipient's Gmail folder using `imap.gmail.com:993`. It verifies delivery by matching the `Message-Id` header and logs the `verification_latency_ms` metric. If the message is not found within 30 seconds, the retry is re-queued with a `delivery_check_timeout` flag.

- **PostgreSQL-Backed Retry Queue with Dead Letter Channel**  
  Failed messages exceeding `max_retries` (default: 5) are moved to a `dead_letter_queue` table, which stores the full MIME payload, last SMTP error code, and `failure_timestamp` at microsecond precision. A separate `dead_letter_consumer` process can replay these messages via a dedicated channel using `LISTEN`/`NOTIFY` for zero-polling recovery.

## How to Use

1. **Initialize the Retry Database Schema**  
   Run `psql -U your_user -d gmail_retry -f schema/init.sql` to create the `outbound_queue`, `dead_letter_queue`, and `retry_metrics` tables. The `outbound_queue` includes columns `message_id UUID PRIMARY KEY`, `retry_count INT DEFAULT 0`, `next_retry_at TIMESTAMPTZ`, and `payload BYTEA`. Confirm the indexes are built on `(next_retry_at, retry_count)` for efficient polling.

2. **Configure the Retry Beast Daemon**  
   Edit `config/retrybeast.toml` to set `smtp_host = "smtp.gmail.com"`, `smtp_port = 587`, `max_retries = 5`, `base_delay_ms = 1000`, and `jitter_window_ms = 500`. Set `imap_verify = true` and provide your OAuth2 credentials for IMAP access. Then launch with `./gmailretrybeast --config config/retrybeast.toml --daemon`. The process will log each retry attempt with `attempt_number`, `delay_actual_ms`, and `smtp_response_code`.

3. **Monitor and Recover from Dead Letter Queue**  
   Query the dead letter queue using `SELECT * FROM dead_letter_queue WHERE failure_timestamp > NOW() - INTERVAL '24 hours';` To replay a specific message, run `./gmailretrybeast --replay <message_id> --force`. For bulk replay, use the `dead_letter_consumer` module: `./dead_letter_consumer --channel retry_recovery --batch-size 50 --interval-ms 5000`.

## Best Practices

- **Always Enable IMAP Delivery Verification**  
  Relying solely on SMTP response codes is insufficientŌĆöGmail may accept a message and then silently defer delivery. Set `imap_verify = true` and configure a dedicated Gmail account with IMAP access. Monitor the `verification_latency_ms` metric; if it consistently exceeds 10 seconds, consider increasing the IMAP connection pool size or switching to the Gmail API for faster inbox polling.

- **Tune Jitter to Avoid SMTP Throttling**  
  Gmail's SMTP servers enforce per-connection and per-IP rate limits. Use a `jitter_window_ms` of at least 500 ms and a `base_delay_ms` of 2000 ms for high-volume senders. If you observe frequent `450 4.7.0` or `421 4.7.0` responses, reduce the `max_retries` to 3 and increase the `base_delay_ms` to 5000 ms. Log all `throttle_hit_count` events to adjust your backoff parameters dynamically.
