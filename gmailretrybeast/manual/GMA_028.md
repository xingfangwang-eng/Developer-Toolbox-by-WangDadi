# gmailretrybeast - GMA_028

## Overview

gmailretrybeast is a high-performance, event-driven email retry orchestration engine designed to eliminate silent delivery failures in Gmail-based transactional workflows. It implements an adaptive backoff scheduler with millisecond-precision retry windows, ensuring that transient SMTP errors, rate-limit penalties (e.g., 421/452/550), and OAuth token expiration are handled without manual intervention. The system is optimized for bulk sending pipelines where even a 0.1% loss is unacceptable.

## Core Features

- **Adaptive Retry Scheduler with Jittered Exponential Backoff**  
  Implements a configurable retry policy where each failed delivery attempt triggers a delay calculated as `baseDelay * 2^attempt + randomJitter(max 250ms)`. The scheduler supports up to 15 retry attempts with per-attempt timeout control (default: 30s). Failed messages are persisted to PostgreSQL with full SMTP response headers for forensic analysis.

- **OAuth2 Token Lifecycle Manager with Automatic Refresh**  
  Monitors Gmail API `access_token` expiration in real-time using a background goroutine that checks token TTL every 60 seconds. When the remaining TTL drops below 300 seconds, it proactively refreshes the token via the configured OAuth2 endpoint, preventing 401 errors mid-batch. Supports multi-tenant credential stores with per-account `maxConcurrentSessions` parameter (default: 10).

- **SMTP Connection Pool with Health-Check and Graceful Drain**  
  Maintains a pool of up to `poolSize` (default: 25) persistent SMTP connections per Gmail account. Each connection is health-checked every `healthCheckInterval` (default: 120s) by sending a NOOP command. Connections that fail three consecutive health checks are automatically drained and replaced. The pool uses a least-recently-used (LRU) eviction strategy to minimize reconnection overhead.

## How to Use

### Step 1: Configure Retry Policy and Credential Store
Create a `config.yaml` file in the project root. Define your retry parameters and OAuth2 credentials:
```yaml
retry:
  maxAttempts: 12
  baseDelayMs: 1000
  jitterMs: 250
  perAttemptTimeoutSec: 30
oauth:
  clientId: "your-client-id"
  clientSecret: "your-client-secret"
  tokenEndpoint: "https://oauth2.googleapis.com/token"
  refreshThresholdSec: 300
smtp:
  poolSize: 25
  healthCheckIntervalSec: 120
  maxConsecutiveFailures: 3
```
Run `gmailretrybeast --config ./config.yaml` to load the configuration. The tool will validate all parameters and print a summary of the retry schedule.

### Step 2: Queue Messages for Delivery
Use the CLI to enqueue messages from a CSV file. Each row must contain `to`, `subject`, and `body` fields. Optionally include `cc`, `bcc`, and `attachments` (comma-separated file paths):
```bash
gmailretrybeast enqueue --file ./batch.csv --account "primary@example.com"
```
The tool will parse the CSV, validate email addresses against RFC 5322, and insert each message into the PostgreSQL queue with a unique `message_id` (UUIDv4). You can monitor the queue depth via `gmailretrybeast status --queue-depth`.

### Step 3: Execute Retry Loop with Real-Time Monitoring
Start the delivery engine with live metrics:
```bash
gmailretrybeast run --account "primary@example.com" --metrics-addr ":9090"
```
The engine will begin processing the queue. Each delivery attempt is logged to stdout with millisecond timestamps and SMTP response codes. Access Prometheus metrics at `http://localhost:9090/metrics` for `gmail_retry_attempts_total`, `gmail_retry_success_total`, and `gmail_retry_failure_total`. To inspect failed messages after all retries are exhausted, run:
```bash
gmailretrybeast dead-letter --account "primary@example.com" --format json
```

## Best Practices

- **Implement a Dead-Letter Queue with Human-in-the-Loop**  
  Configure `maxAttempts` to a value that allows at least 24 hours of retry (e.g., 12 attempts with base delay of 1s yields ~8 hours of retry window). After exhaustion, route failed messages to a separate PostgreSQL table (`dead_letter_queue`) with full SMTP response headers. Use a weekly review process to analyze patterns (e.g., recurring 550 errors from specific domains) and adjust your sender reputation or whitelist.

- **Use Connection Pooling with Per-Account Rate Limiting**  
  Gmail imposes a daily sending limit (e.g., 500 for free accounts, 2000 for Workspace). Set `poolSize` to no more than 5 for free accounts to avoid triggering rate-limit penalties. For Workspace accounts, calculate `poolSize` as `dailyLimit / 24 / 3600 * 1.2` (20% headroom). Monitor `gmail_retry_rate_limit_hits_total` metric to dynamically reduce pool size if the metric increases by more than 10% in a 15-minute window.
