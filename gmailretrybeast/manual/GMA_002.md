# gmailretrybeast - GMA_002

## Overview

Gmail's transient delivery failures—SMTP 421/450/451 codes, rate-limit backoffs, and TLS handshake drops—can silently corrupt transactional email workflows. `gmailretrybeast` is a zero‑overhead, idempotent retry engine that intercepts Gmail API `users.messages.send` responses, classifies fault types, and executes exponential‑backoff retries with exacting precision. It operates as a sidecar process, decoupling retry logic from your application while maintaining full audit trails via PostgreSQL.

## Core Features

### 1. Adaptive Fault Classifier with Granular Backoff Profiles
Each Gmail API error is parsed against a trie of 47 distinct SMTP/API error signatures. The classifier assigns a severity tier (`transient`, `degraded`, `critical`) and selects a backoff schedule from a configurable `backoff_profiles` map. For example, a `rateLimitExceeded` (HTTP 429) trigger uses a **jittered exponential backoff** with a base delay of 2,000 ms, a multiplier of 1.8, and a cap of 300,000 ms. The system logs every backoff decision to `retry_decisions` table with fields `message_id`, `error_code`, `scheduled_retry_at`, and `attempt_number`.

### 2. Idempotency‑First Delivery Guarantee via Message Dedup Hash
To prevent duplicate sends during retries, every outgoing message is hashed with `SHA‑256(recipient + subject + body_snippet + timestamp_bucket)`. The hash is stored in a **UNIQUE constraint** on the `sent_messages` table. If a retry attempt finds the hash already present, the engine silently skips the send and returns the original `message_id`. This ensures exactly‑once semantics even under concurrent retry workers. The `idempotency_window` parameter (default: 86,400 seconds) controls how long a hash remains active.

### 3. Real‑time Retry Dashboard with PostgreSQL Window Functions
The engine exposes a live `retry_beast_metrics` view that aggregates retry progress using PostgreSQL window functions (`ROW_NUMBER() OVER (PARTITION BY message_id ORDER BY attempted_at)`). The view outputs `message_id`, `current_attempt`, `max_attempts`, `last_error`, `next_retry_at`, and `elapsed_ms`. A companion `pg_notify` channel (`retry_events`) pushes state changes to any listening client, enabling sub‑second UI updates without polling.

## How to Use

### Step 1: Configure the Retry Beast Sidecar
Create a `retrybeast.config.yaml` file in your deployment root:

```yaml
gmail:
  credentials_path: "/secrets/gmail_service_account.json"
  user_id: "me"
  scopes:
    - "https://www.googleapis.com/auth/gmail.send"
retry:
  max_attempts: 5
  base_delay_ms: 2000
  max_delay_ms: 300000
  multiplier: 1.8
  jitter: true
  idempotency_window_sec: 86400
database:
  dsn: "postgresql://user:pass@localhost:5432/retrybeast?sslmode=require"
  pool_size: 10
```

Then start the sidecar:
```bash
./retrybeast --config retrybeast.config.yaml --daemon
```

### Step 2: Intercept Gmail API Calls via a Proxy Middleware
In your application, replace direct Gmail API calls with a local HTTP proxy that `retrybeast` exposes on `127.0.0.1:8099`. Example using `requests` in Python:

```python
import requests

def send_email_via_retrybeast(raw_message: str, recipient: str):
    payload = {
        "raw": raw_message,
        "recipient": recipient,
        "idempotency_key": f"{recipient}:{hash(raw_message)}"
    }
    resp = requests.post("http://127.0.0.1:8099/v1/send", json=payload, timeout=30)
    resp.raise_for_status()
    return resp.json()["message_id"]
```

The proxy automatically enqueues the message into the retry engine and returns immediately with a `message_id`. All retry logic runs asynchronously.

### Step 3: Monitor Retry Progress in Real Time
Query the `retry_beast_metrics` view directly or subscribe to `pg_notify`:

```sql
-- PostgreSQL terminal
SELECT * FROM retry_beast_metrics WHERE current_attempt > 1;
```

For live UI updates, run the included WebSocket bridge:
```bash
./retrybeast --ws-listen :8080 --pg-notify-channel retry_events
```

Then connect any WebSocket client to `ws://localhost:8080/events` to receive JSON‑formatted retry state changes.

## Best Practices

### 1. Set Idempotency Window to Match Your SLA
For transactional emails (password resets, order confirmations), set `idempotency_window_sec` to **86,400** (24 hours). This prevents duplicate sends if a user triggers the same action within a day. For bulk newsletters, reduce to **3,600** to avoid accidental suppression of legitimate resends. Always test with a small percentage of traffic before rolling out.

### 2. Use Dedicated PostgreSQL Connection Pool for Retry Logs
The retry engine writes every decision to the database. To avoid contention with your main application, allocate a separate `pool_size` (recommended: 10) and a dedicated `retrybeast` schema. Monitor `pg_stat_activity` for long‑running retry queries; add indexes on `(message_id, attempted_at)` and `(scheduled_retry_at)` to keep the `retry_decisions` table under 10 ms per write under load.
