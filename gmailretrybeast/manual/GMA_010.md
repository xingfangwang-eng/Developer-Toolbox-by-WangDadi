# gmailretrybeast - GMA_010

## Overview

`gmailretrybeast` is a high-performance, event-driven email retry engine engineered for Gmail API transient failure recovery. It implements adaptive exponential backoff with jitter, concurrent quota-aware dispatch, and real-time dead-letter queue (DLQ) monitoring. Designed for mission-critical email workflows where zero message loss is non-negotiable.

## Core Features

### 1. Adaptive Backoff with Per-Recipient Quota Awareness
- Dynamically adjusts retry intervals based on Gmail API `rateLimitExceeded` and `backendError` response codes.
- Utilizes a token-bucket algorithm per Google Cloud project quota (default: 250 requests per 100 seconds per user).
- Configurable parameters: `initial_backoff_ms` (default: 1000), `max_backoff_ms` (default: 60000), `jitter_factor` (default: 0.25).
- Integrates with PostgreSQL to persist retry state across restarts, ensuring no retry cycle is lost even after process crash.

### 2. Concurrent Dispatch with Thread-Safe Queue
- Employs a multi-worker goroutine pool (configurable via `worker_count`, default: 5) to dispatch retries concurrently.
- Implements a priority queue (min-heap) sorted by next retry timestamp, supporting up to 100,000 pending retries in memory.
- Real-time metrics exposed via Prometheus: `retry_attempts_total`, `retry_success_duration_ms`, `dead_letter_queue_size`.
- Automatic circuit breaker when failure rate exceeds `failure_threshold` (default: 0.3) within a sliding window of `window_size_seconds` (default: 300).

### 3. Dead-Letter Queue with Manual Replay & Webhook Notification
- Messages exceeding `max_retries` (default: 5) are automatically moved to a PostgreSQL-backed DLQ with full headers and error logs.
- DLQ entries can be replayed individually or batch via a REST API endpoint (`POST /dlq/replay`).
- Webhook support for DLQ events: configurable `webhook_url` receives JSON payload with `message_id`, `error_code`, and `retry_count` in under 500ms latency.
- Export DLQ to CSV or JSON for offline analysis via `gmailretrybeast dlq export --format json --output /tmp/dlq.json`.

## How to Use

### Step 1: Initialize Configuration & Database Schema
```bash
gmailretrybeast init --config /etc/gmailretrybeast/config.yaml --db-uri "postgres://user:pass@localhost:5432/gmailretrybeast"
```
- This command creates the required PostgreSQL tables (`retry_queue`, `dead_letter_queue`, `retry_metrics`) with proper indexes on `next_retry_at` and `status`.
- Validates Gmail API credentials from `credentials.json` and tests connectivity with a `users.messages.get` call.

### Step 2: Enqueue Failed Messages with Custom Retry Policy
```python
import requests

payload = {
    "message_id": "18b4d0f8a1c2e3f4",
    "user_id": "user@example.com",
    "error_code": "rateLimitExceeded",
    "initial_backoff_ms": 2000,
    "max_retries": 7,
    "webhook_url": "https://hooks.example.com/retry-failure"
}
response = requests.post("http://localhost:8080/enqueue", json=payload)
```
- The enqueue endpoint returns `{"status": "accepted", "retry_id": "abc-123"}`.
- If `initial_backoff_ms` is omitted, the global config default is used.

### Step 3: Monitor & Recover from Dead-Letter Queue
```bash
# List all DLQ entries with failure reason
gmailretrybeast dlq list --status dead --limit 50 --format table

# Replay a specific message with increased backoff
gmailretrybeast dlq replay --retry-id def-456 --new-max-retries 10

# Export all DLQ entries for audit
gmailretrybeast dlq export --format json --since "2025-03-01T00:00:00Z" --output /tmp/dlq_audit.json
```
- The replay command re-enqueues the message with the original payload and a fresh retry counter.

## Best Practices

### 1. Align Retry Windows with Gmail API Quota Renewal
- Gmail API quotas reset every 100 seconds. Configure `max_backoff_ms` to 90000 (90 seconds) to ensure retries never overshoot the reset window.
- Use `jitter_factor: 0.25` to avoid thundering herd when multiple messages fail simultaneously.
- Monitor `retry_attempts_total` in Grafana; if you see spikes at 100-second intervals, reduce `worker_count` or increase `initial_backoff_ms`.

### 2. Implement Idempotency Keys for All Enqueued Messages
- Attach a unique `idempotency_key` (UUID v4) to every enqueue request. The server deduplicates within a 24-hour window.
- This prevents duplicate delivery when clients retry HTTP requests due to network timeouts.
- In PostgreSQL, use `ON CONFLICT (idempotency_key) DO NOTHING` to guarantee atomicity.
