# gmailretrybeast - GMA_018

## Overview

`gmailretrybeast` is a high‑throughput, event‑driven retry engine purpose‑built for Gmail API transient failures. It implements a **circuit‑breaker with exponential backoff** (jitter‑aware) and a **dedicated PostgreSQL‑backed retry queue**, ensuring zero message loss during 429 / 5xx bursts. The system is designed to handle 10,000+ concurrent retry sessions with sub‑100ms scheduling precision.

## Core Features

### 1. Adaptive Backoff Scheduler with Jitter Window
- **Parameter:** `backoff_base_ms`, `jitter_max_ms`, `max_retry_attempts`
- **Behavior:** After each Gmail API failure, the beast computes `delay = min(backoff_base_ms * 2^attempt + random(0, jitter_max_ms), 300_000)`.  
- **Metric:** Schedules retries at **±15ms accuracy** even under 5,000 concurrent jobs.  
- **Storage:** Each retry state is persisted as a JSONB row in `retry_beast.jobs` (PostgreSQL), enabling crash‑safe recovery.

### 2. Gmail‑Specific Error Classifier & Circuit Breaker
- **Parameter:** `error_threshold_open`, `half_open_timeout_ms`, `error_classifier_rules`
- **Behavior:** Inline classification of `googleapiclient.errors.HttpError`:  
  - 429 → immediate backoff (no breaker open)  
  - 5xx → increment failure counter; if `failure_count > error_threshold_open`, open circuit for `half_open_timeout_ms`  
  - 403 (rateLimitExceeded) → same as 429, but with extended jitter  
- **Metric:** Breaker state transitions logged to `circuit_breaker_events` table with millisecond timestamps.

### 3. Multi‑Stage Retry Pipeline with Idempotency Keys
- **Parameter:** `idempotency_key_field`, `pipeline_stages = ["queue", "validate", "send", "callback"]`
- **Behavior:**  
  - Each message is assigned a UUID‑based idempotency key before entering the queue.  
  - The pipeline processes messages in strict order per key, preventing duplicate sends.  
  - On success, the key is stored in a Redis‑backed bloom filter (TTL: 24h) for instant dedup.  
- **Metric:** Pipeline throughput > 2,000 messages/minute with < 0.01% duplicate rate.

## How to Use

### Step 1: Initialize the Retry Queue & Circuit Breaker
```bash
# Create the PostgreSQL schema and configure the circuit breaker
gmailretrybeast init \
  --pg-conn "postgresql://user:pass@localhost:5432/retrybeast" \
  --backoff-base-ms 1000 \
  --jitter-max-ms 500 \
  --max-retry-attempts 5 \
  --error-threshold-open 3 \
  --half-open-timeout-ms 60000
```
This command creates tables `retry_beast.jobs`, `retry_beast.circuit_breaker_events`, and applies the default error classification rules.

### Step 2: Enqueue a Gmail Send Operation with Idempotency
```python
from gmailretrybeast import RetryBeastClient

client = RetryBeastClient(
    service_account_json="/path/to/sa.json",
    user_id="user@gmail.com",
    pipeline="queue,validate,send"
)

message_id = client.send_with_retry(
    to="recipient@example.com",
    subject="Critical Alert",
    body="...",
    idempotency_key="msg-2025-04-10-001"
)
# Returns immediately; the beast handles retries asynchronously.
```
The client automatically generates a UUID if no `idempotency_key` is provided.

### Step 3: Monitor & Force Drain the Retry Queue
```bash
# View pending retries grouped by error code
gmailretrybeast status --show-breaker

# Force immediate retry of all jobs in "backoff" state (bypasses remaining delay)
gmailretrybeast drain --state backoff --force

# Export failed jobs to JSON for manual inspection
gmailretrybeast export-failed --format json --output failed_jobs.json
```

## Best Practices

### 1. Always Set a Conservative `error_threshold_open`
For production Gmail API usage, start with `--error-threshold-open 5` and `--half-open-timeout-ms 120000`. This prevents the circuit breaker from tripping on short‑lived 5xx spikes while still protecting your quota. Monitor the `circuit_breaker_events` table to tune the threshold based on your actual error pattern.

### 2. Combine Idempotency Keys with a Bloom Filter for High‑Volume Sends
When sending > 10,000 messages per hour, the PostgreSQL‑backed dedup table can become a bottleneck. Enable the Redis bloom filter (`--bloom-filter-redis redis://localhost:6379/0`) to reduce lookup latency from ~50ms to <1ms. Always set a TTL equal to your maximum retry window (e.g., 24h) to avoid unbounded memory growth.
