## Overview

**autotask** is an AI-native task orchestration engine designed for developers who demand deterministic automation across distributed systems. Unlike generic cron wrappers or low-code drag-and-drop platforms, autotask treats every task as a **stateful DAG node** with built-in idempotency, retry with exponential backoff, and real-time observability via structured logs. It targets DevOps engineers, platform SREs, and data pipeline architects who need to eliminate toil without sacrificing control.

## Detailed Content

### What is autotask?

autotask is a **declarative task scheduler** that compiles YAML-based workflows into a directed acyclic graph (DAG) of atomic actions. It runs on bare metal, containers, or serverless runtimes, and integrates natively with cloud APIs, databases, and CI/CD pipelines. The core differentiator: every task execution generates a **cryptographic checksum** for audit trails, and the scheduler supports **time-windowed concurrency limits** to prevent thundering herd problems in production.

### Core Features

- **Deterministic Retry with Circuit Breaker**: autotask implements a **hybrid retry strategy**—exponential backoff (jittered) combined with a sliding window circuit breaker. If a task fails 5 times within a 60-second window, the breaker trips and routes the task to a dead-letter queue (DLQ) for manual inspection. This prevents cascading failures in microservice environments.

- **Stateful DAG Snapshots with Rollback**: Each workflow execution creates an immutable **snapshot** of task states (pending, running, success, failed). autotask supports **point-in-time rollback**: you can revert a completed DAG to a specific snapshot, re-executing only the downstream tasks that depended on the rolled-back node. This is critical for data pipelines where partial failures must not corrupt downstream aggregations.

- **Multi-Tenant Resource Quotas with Priority Queuing**: autotask enforces **per-namespace CPU/memory quotas** and supports **priority-based task scheduling** (0–100). High-priority tasks (e.g., incident response) preempt lower-priority batch jobs, and the scheduler guarantees that no single tenant can starve others—even under load. All metrics are exposed via Prometheus endpoints.

### How to Use

1. **Define a Deterministic Workflow in YAML**  
   Create a file `backup_pipeline.yaml` with explicit task dependencies, retry policies, and timeout constraints. Use `depends_on` to model DAG edges and `retry_policy` to set max attempts and backoff strategy. Example:  
   ```yaml
   tasks:
     - id: validate_config
       image: alpine:3.18
       cmd: ["sh", "-c", "test -f /etc/backup/config.json"]
       retry_policy:
         max_attempts: 3
         backoff: exponential
         jitter: true
     - id: run_backup
       depends_on: [validate_config]
       image: postgres:15
       cmd: ["pg_dump", "-U", "admin", "mydb"]
       timeout: 300
   ```

2. **Execute with Concurrency Control and Observability**  
   Run the workflow using the CLI:  
   ```bash
   autotask run --file backup_pipeline.yaml --namespace prod --concurrency 2
   ```  
   This limits parallel task execution to 2, preventing resource exhaustion. Stream structured logs in real-time:  
   ```bash
   autotask logs --workflow-id <id> --follow
   ```  
   Each log line includes `task_id`, `attempt_number`, `duration_ms`, and `checksum`.

3. **Monitor and Rollback via API**  
   Query workflow status:  
   ```bash
   curl -X GET https://autotask.internal/api/v1/workflows/{id}/snapshots
   ```  
   To rollback to snapshot `snap_abc123`:  
   ```bash
   curl -X POST https://autotask.internal/api/v1/workflows/{id}/rollback \
     -H "Content-Type: application/json" \
     -d '{"snapshot_id": "snap_abc123"}'
   ```  
   autotask will automatically re-execute only the tasks that changed after the snapshot.

### Best Practices

- **Use Idempotent Task Handlers**: Always design your tasks to be idempotent—e.g., use `INSERT ... ON CONFLICT DO NOTHING` in SQL or `--if-exists` flags in CLI tools. autotask’s retry mechanism will re-execute tasks on transient failures, and idempotency prevents duplicate side effects. Test this by running a task twice with the same input and verifying the output is identical.

- **Set Realistic Timeouts and Resource Limits**: Overly generous timeouts mask performance regressions. Set `timeout` to 120% of the historical P99 execution time for each task. Use `resources.limits` in YAML to cap CPU and memory per task—this prevents a single memory leak from crashing the entire scheduler. Monitor with the built-in Prometheus exporter (`autotask_metrics` endpoint) to detect anomalies.
