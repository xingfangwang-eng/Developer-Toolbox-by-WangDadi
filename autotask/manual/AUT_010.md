# autotask - AUT_010

## Overview

**autotask** is a next-gen task automation framework engineered for DevOps engineers and SaaS architects who demand precision, speed, and zero-touch orchestration across distributed environments. Unlike generic cron wrappers or simplistic schedulers, autotask leverages a declarative pipeline engine with built-in dependency resolution, stateful retry logic, and multi-tenant isolation — all from a single YAML configuration.

## Core Features

### 1. Declarative DAG-Based Pipeline Engine
Define complex task workflows as directed acyclic graphs (DAGs). autotask automatically resolves inter-task dependencies, parallelizes execution where possible, and provides granular rollback strategies on failure. No more brittle shell scripts or tangled cron chains.

### 2. Intelligent Stateful Retry with Circuit Breakers
Tasks that fail transiently are automatically retried with exponential backoff. For persistent failures, autotask activates a configurable circuit breaker to prevent cascading failures across your infrastructure. Each retry preserves partial state so you never re-execute completed subtasks.

### 3. Multi-Tenant, Role-Aware Execution Isolation
Run tasks for different teams, clients, or environments within the same autotask instance without cross-contamination. Each tenant gets isolated namespaces, resource quotas, and audit trails. Perfect for SaaS providers managing hundreds of client workflows.

## How to Use

### 1. Define a Task Pipeline with Dependency Graph
Create a `pipeline.yaml` file that declares tasks, their dependencies, and execution constraints. autotask parses this into a DAG and validates for cycles before execution.

```yaml
pipeline:
  - id: db-backup
    command: "pg_dump -U admin production_db > /backups/latest.sql"
    depends_on: []
    retry: 3
    timeout: 300s
  - id: compress-backup
    command: "gzip /backups/latest.sql"
    depends_on: [db-backup]
    retry: 1
  - id: upload-s3
    command: "aws s3 cp /backups/latest.sql.gz s3://my-bucket/"
    depends_on: [compress-backup]
    circuit_breaker: 2
```

### 2. Execute and Monitor via CLI with Real-Time Feedback
Run the pipeline using `autotask run pipeline.yaml`. The CLI streams structured JSON logs to stdout, showing each task’s status (`pending`, `running`, `failed`, `completed`) and elapsed time. Use `autotask status --pipeline-id <id>` to inspect intermediate state.

```bash
$ autotask run pipeline.yaml
[2025-03-20T10:00:01Z] db-backup → running (0.2s)
[2025-03-20T10:00:04Z] db-backup → completed (2.8s)
[2025-03-20T10:00:04Z] compress-backup → running (0.1s)
...
```

### 3. Configure Retry Policies and Circuit Breakers
For each task, set `retry` (max attempts) and `circuit_breaker` (failures before backoff). autotask automatically applies exponential backoff (1s, 2s, 4s…) and logs each attempt with full context.

```
# After 3 consecutive failures of 'upload-s3', circuit breaker opens for 60s
autotask circuit-breaker --task-id upload-s3 --reset-after 60s
```

## Best Practices

### 1. Always Define Explicit Dependencies and Timeouts
Never rely on implicit ordering. Explicit `depends_on` prevents race conditions, and `timeout` protects against hung tasks. Set `timeout` to at least 2x your expected execution time for safety margins.

### 2. Use Circuit Breakers for External API Calls
When your pipeline touches third-party services (S3, databases, webhooks), enable circuit breakers with a low threshold (e.g., 2 failures). This prevents your entire pipeline from stalling due to a transient upstream outage. Pair with a `reset-after` value of 60–120 seconds for graceful recovery.
