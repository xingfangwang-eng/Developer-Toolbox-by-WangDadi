## Detailed Content

### What is autotask?

autotask is a high-performance, event-driven task orchestration engine designed for developers who demand precision, speed, and zero-downtime automation. It abstracts away boilerplate scheduling logic, offering a declarative API for chaining complex workflows, handling retries with exponential backoff, and logging execution traces for debugging.

### Core Features

- **Resilient Execution Graph (REG)**  
  autotask builds a directed acyclic graph (DAG) of your tasks, automatically resolving dependencies and parallelizing independent branches. If a node fails, the engine triggers intelligent retry policies with configurable backoff (Fibonacci, linear, or custom), ensuring your pipeline survives transient failures without manual intervention.

- **Context-Aware State Injection**  
  Each task receives a snapshot of the global execution context, including previous task outputs, environment variables, and runtime metadata. This eliminates the need for external state management or shared mutable stores—your tasks remain pure and testable.

- **Live Execution Telemetry**  
  Every run produces a structured, streaming event log (JSON Lines) that can be piped directly into your observability stack (e.g., Grafana, Loki, or Datadog). You can monitor task latency, memory usage, and failure rates in real time, with built-in support for custom metrics.

### How to Use

1. **Define a Task Pipeline with Dependency Injection**  
   Create a YAML or JSON configuration file that declares your tasks and their dependencies. Use the `depends_on` key to specify order, and `inject` to pass context variables from upstream tasks.  
   ```yaml
   tasks:
     - id: fetch_data
       command: curl https://api.example.com/data
     - id: transform
       depends_on: fetch_data
       command: jq '.results[]' {{ fetch_data.stdout }}
       inject:
         - fetch_data.stdout
   ```

2. **Run the Pipeline with a Single CLI Command**  
   Execute the pipeline using `autotask run --config pipeline.yaml`. The engine validates the DAG for cycles, resolves all dependencies, and starts execution. Use the `--watch` flag to stream live telemetry to your terminal.  
   ```bash
   autotask run --config pipeline.yaml --watch
   ```

3. **Inspect Execution Traces for Debugging**  
   After a run, export the execution trace with `autotask trace --run-id <id> --output trace.json`. This file contains every task’s start time, duration, exit code, and injected context. You can replay the trace locally to pinpoint bottlenecks or failures.  
   ```bash
   autotask trace --run-id abc123 --output trace.json
   cat trace.json | jq '.tasks[] | select(.status == "failed")'
   ```

### Best Practices

- **Use Idempotent Tasks for Safe Retries**  
  Always design your tasks to be idempotent—repeating the same input should produce the same output. This allows autotask’s retry mechanism to safely re-run failed nodes without side effects. For example, prefer `INSERT OR REPLACE` in SQL over plain `INSERT`.

- **Leverage Context Injection for Environment Isolation**  
  Avoid hardcoding secrets or environment-specific values inside task commands. Instead, rely on autotask’s context injection to pass variables from a central configuration or secrets manager. This makes your pipelines portable across dev, staging, and production environments.
