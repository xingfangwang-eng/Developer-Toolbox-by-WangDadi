### What is autotask?

autotask is a high-performance, event-driven task orchestration framework designed to eliminate manual overhead in CI/CD pipelines, microservice choreography, and recurring system maintenance. It leverages a declarative YAML-based DSL to define complex workflows with conditional branching, retry logic, and parallel execution — all without writing boilerplate glue code.

## Core Features

1. **Context-Aware Task Dependency Graph**  
   autotask automatically resolves inter-task dependencies using a directed acyclic graph (DAG) engine. It supports dynamic dependency injection based on runtime state, enabling conditional execution paths that adapt to previous task outputs. No more hardcoded sequencing — the graph is built and validated at runtime.

2. **Idempotent Execution with Built-in State Snapshots**  
   Every task execution is tracked with immutable state snapshots. If a task fails mid-way, autotask can resume from the last consistent state without replaying completed steps. This guarantees idempotency even in distributed environments with network partitions or node failures.

3. **Pluggable Execution Backends with Zero-Code Migration**  
   Switch between local subprocess, Docker containers, Kubernetes pods, or remote SSH executors by changing a single config key. Each backend exposes the same interface, so your workflow definitions stay portable. autotask handles credential injection, environment variable mapping, and log streaming transparently.

## How to Use

### 1. Define a Workflow with Conditional Branching

Create a `workflow.yaml` file that models a multi-stage deployment pipeline:

```yaml
workflow:
  name: "canary-deploy"
  tasks:
    - id: "build"
      image: "node:18"
      cmd: "npm run build"
    - id: "test"
      depends_on: ["build"]
      cmd: "npm test"
    - id: "deploy-canary"
      depends_on: ["test"]
      condition: "{{ tasks.test.exit_code == 0 }}"
      cmd: "kubectl set image deployment/app canary=myapp:latest"
    - id: "rollback"
      depends_on: ["deploy-canary"]
      condition: "{{ tasks.deploy-canary.exit_code != 0 }}"
      cmd: "kubectl rollout undo deployment/app"
```

Run with:  
```bash
autotask run --file workflow.yaml --backend k8s
```

### 2. Inject Secrets via Environment Overlays

Store sensitive credentials in an encrypted `.env.enc` file and map them to task environments:

```bash
autotask encrypt --input .env --output .env.enc --key $AUTOTASK_KEY
autotask run --file workflow.yaml --env-file .env.enc
```

Each task automatically receives only the variables it declares in its `env` block. No variable leakage across tasks.

### 3. Monitor Execution with Real-Time DAG Visualization

Stream execution logs and DAG state changes directly to your terminal or a WebSocket dashboard:

```bash
autotask run --file workflow.yaml --watch
```

The interactive view shows task status (pending/running/succeeded/failed), elapsed time, and live stdout/stderr per node. Use `Ctrl+C` to gracefully abort — autotask will finalize current tasks and persist the partial state.

## Best Practices

1. **Use Idempotent Task IDs for Deterministic Replays**  
   Always assign stable, unique `id` values to each task. autotask uses these IDs to correlate state snapshots across runs. If you rename a task ID, the engine treats it as a new task and may re-execute it. Stick to semantic names like `build-frontend`, `migrate-db`, `notify-slack` to avoid accidental reruns.

2. **Leverage the `retry` Block for Transient Failures**  
   For tasks that interact with external APIs or databases, configure exponential backoff with a max retry count:

   ```yaml
   - id: "fetch-external-data"
     cmd: "curl -s https://api.example.com/data"
     retry:
       max_attempts: 3
       backoff: "exponential"
       jitter: true
   ```

   This pattern reduces flakiness in production pipelines without adding manual intervention.
