# autotask - AUT_006

## Resumen

autotask es un orquestador nativo de flujos de trabajo automatizados diseñado para desarrolladores que necesitan eliminar tareas repetitivas sin sacrificar control granular. No es un simple task scheduler; es un motor de ejecución determinista con capacidad de encadenamiento condicional, logging estructurado y recuperación ante fallos en tiempo real. Ideal para pipelines de CI/CD, sincronización de datos entre microservicios o automatización de mantenimiento en infraestructuras cloud.

## Contenido Detallado

### ¿Qué es autotask?

autotask es un framework ligero pero extensible que permite definir, ejecutar y monitorizar tareas automatizadas mediante archivos YAML con sintaxis declarativa y hooks en Python. A diferencia de herramientas como Cron o Airflow, autotask prioriza la simplicidad de despliegue (binario único sin dependencias externas) y la transparencia total del estado de ejecución a través de una API REST integrada. Está pensado para equipos que necesitan automatización sin el overhead de una plataforma completa.

### Características Principales (Hardcore)

- **Ejecución con DAGs Condicionales y Retry Inteligente**: Cada tarea puede definir un DAG (Directed Acyclic Graph) con dependencias entre pasos, condiciones de salida y reintentos exponenciales con backoff configurable. Si un paso falla, autotask no solo reintenta; decide si continuar con una ruta alternativa basada en reglas de lógica booleana definidas en el propio workflow.
- **Pipeline de Logging Estructurado con Exportación a Elasticsearch y Loki**: No se limita a stdout. autotask genera logs en formato JSON con metadatos de ejecución (timestamp, duración, código de salida, variables de entorno) y los envía directamente a sinks configurables como Elasticsearch, Loki o un archivo local rotado. Ideal para debugging forense y dashboards en Grafana.
- **Sandbox por Tarea con Aislamiento de Sistema de Archivos y Variables de Entorno**: Cada tarea se ejecuta en un contenedor ligero (namespace de Linux) con un sistema de archivos temporal y variables de entorno aisladas. Esto evita colisiones entre tareas concurrentes y permite ejecutar scripts no confiables sin riesgo de contaminación del entorno principal.

### Cómo Usar (Pasos Profesionales)

1. **Definición del Workflow con Dependencias Explícitas**: Crea un archivo `autotask.yaml` en la raíz de tu proyecto. Usa la clave `tasks` para listar cada tarea, y dentro de cada una, especifica `depends_on` para indicar precedencias. Por ejemplo, para una tarea de backup que debe ejecutarse solo si la tarea de validación de disco fue exitosa:
   ```yaml
   tasks:
     validate_disk:
       command: "df -h /data | grep -q '80%' && exit 1 || exit 0"
       retry: 2
     backup_data:
       command: "rsync -avz /data /backup"
       depends_on:
         - validate_disk
       condition: "validate_disk.exit_code == 0"
   ```
   Luego, ejecuta `autotask run --file autotask.yaml`. El motor evaluará las dependencias y condiciones en tiempo real.

2. **Configuración de Sinks de Logging con Autenticación**: Para enviar logs estructurados a Elasticsearch, edita el bloque `logging` en el archivo de configuración global (`~/.autotask/config.yaml`). Especifica el endpoint, índice y credenciales (usando variables de entorno para seguridad):
   ```yaml
   logging:
     sinks:
       - type: elasticsearch
         url: "${ES_URL}"
         index: "autotask-logs-{date}"
         auth:
           user: "${ES_USER}"
           password: "${ES_PASS}"
   ```
   Cada vez que una tarea se ejecute, autotask enviará un documento JSON con campos como `task_name`, `duration_ms`, `exit_code`, `error_message` y `env_vars`.

3. **Monitoreo en Tiempo Real con la API REST**: Inicia el servidor de monitoreo con `autotask serve --port 8080`. Luego, desde tu terminal o un dashboard personalizado, consulta el estado de todas las tareas activas:
   ```bash
   curl http://localhost:8080/api/v1/tasks | jq '.tasks[] | {name, status, last_run}'
   ```
   También puedes forzar la ejecución de una tarea específica mediante POST: `curl -X POST http://localhost:8080/api/v1/tasks/backup_data/run`. La respuesta incluirá el ID de ejecución y un enlace al log en Elasticsearch.

### Mejores Prácticas (Industry Insights)

- **Usa Variables de Entorno para Secretos y Configuración Dinámica**: Nunca hardcodees tokens, URLs o rutas en los archivos YAML. autotask soporta interpolación de variables de entorno con sintaxis `${VAR}`. Define un archivo `.env` o exporta las variables antes de ejecutar el motor. Esto facilita la portabilidad entre entornos (desarrollo, staging, producción) sin modificar los workflows.
- **Implementa un Sistema de Notificaciones con Webhooks**: Para tareas críticas (como backups o despliegues), configura un webhook en el bloque `notifications` de cada tarea. autotask puede enviar un POST a un endpoint de Slack, Discord o un servicio propio cuando una tarea falla o completa exitosamente. Ejemplo:
  ```yaml
  notifications:
    on_failure:
      url: "https://hooks.slack.com/services/T00/B00/xxx"
      method: POST
      body: '{"text": "Tarea {{task_name}} falló con código {{exit_code}}"}'
  ```
