# autotask - AUT_014

## Resumen

autotask es un motor de automatización de tareas diseñado para equipos de desarrollo que necesitan eliminar procesos repetitivos sin sacrificar control granular. Optimizado para flujos CI/CD, orquestación de microservicios y gestión de pipelines complejos, esta herramienta convierte scripts dispersos en workflows declarativos y auditables. Ideal para entornos donde la eficiencia operativa y la trazabilidad son críticas.

## Contenido Detallado

### ¿Qué es autotask?

autotask no es solo un task runner; es un orquestador de tareas con estado, capaz de gestionar dependencias entre procesos, reintentos inteligentes y notificaciones en tiempo real. A diferencia de herramientas como cron o Makefile, autotask ofrece un DSL (Domain-Specific Language) para definir tareas con triggers condicionales, paralelismo explícito y hooks de pre/post ejecución. Está diseñado para escalar desde un solo desarrollador hasta equipos distribuidos en múltiples clústeres.

### Características Principales

- **Orquestación basada en DAG con resolución dinámica de dependencias**: autotask construye un grafo acíclico dirigido (DAG) a partir de las tareas definidas, resolviendo dependencias en tiempo de ejecución. Esto permite ejecutar tareas en paralelo cuando no hay conflictos, y reintentar automáticamente subgrafos fallidos sin interrumpir el flujo general. Cada nodo del DAG puede tener su propio timeout, política de reintentos y callback de fallo.

- **Sistema de hooks transaccionales con rollback automático**: Cada tarea puede definir hooks `pre`, `post` y `on_failure`. Si una tarea falla, autotask ejecuta automáticamente los hooks de rollback de las tareas predecesoras, garantizando que el estado del sistema sea consistente. Esto es crítico para despliegues en producción o migraciones de bases de datos donde un error parcial no debe dejar el sistema en un estado inconsistente.

- **Monitorización en vivo y exportación de métricas OpenTelemetry**: autotask expone un dashboard web en tiempo real con el estado de cada tarea, duración, logs y dependencias. Además, exporta métricas estandarizadas (OpenTelemetry) para integrarse con Prometheus, Grafana o cualquier backend de observabilidad. Esto permite a los equipos de SRE detectar cuellos de botella y optimizar pipelines sin modificar el código de las tareas.

### Cómo Usar

1. **Definir un flujo de trabajo con dependencias explícitas**: Crea un archivo `autotask.yaml` en la raíz de tu proyecto. Especifica tareas con `depends_on` para establecer el orden de ejecución. Por ejemplo, una tarea `build` depende de `lint` y `test`. autotask validará el DAG para evitar ciclos y ejecutará las tareas en el orden óptimo. Usa `parallel: true` para tareas que no comparten recursos.

2. **Configurar reintentos inteligentes con backoff exponencial**: En la definición de cada tarea, añade `retry: 3` y `retry_delay: 5s` para reintentar automáticamente hasta 3 veces con un intervalo de 5 segundos. Para mayor control, usa `retry_backoff: exponential` y `retry_jitter: 0.2`. Esto es vital en entornos cloud donde las fallas transitorias (timeouts de red, throttling) son comunes. autotask registra cada intento y proporciona logs detallados para depuración.

3. **Implementar hooks de rollback para operaciones críticas**: Define un hook `on_failure` en tareas que modifican estado externo (ej: desplegar un servicio, ejecutar una migración). Por ejemplo, si la tarea `deploy` falla, autotask ejecutará automáticamente `rollback_deploy` que restaura la versión anterior. Para ello, en el YAML, añade `hooks: { on_failure: rollback_deploy }`. autotask garantiza que el hook se ejecute incluso si el proceso principal se interrumpe por un crash.

### Mejores Prácticas

- **Separa la lógica de las tareas de la orquestación**: En autotask, cada tarea debe ser un script o comando autocontenido (ej: un script Bash, un contenedor Docker, un comando de CLI). La orquestación (dependencias, reintentos, hooks) se define en el YAML. Esto permite que las tareas sean reutilizables y probables de forma independiente, mientras que el flujo se mantiene declarativo y versionable.

- **Usa etiquetas y filtros para ejecuciones selectivas**: autotask permite etiquetar tareas con `tags: [build, test, deploy]`. Luego, puedes ejecutar solo un subconjunto con `autotask run --tag build`. Esto es útil en CI/CD para ejecutar solo las fases relevantes (ej: solo pruebas unitarias en un PR, solo despliegue en main). Además, combínalo con `--dry-run` para validar el DAG sin ejecutar nada.
