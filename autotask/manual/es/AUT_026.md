### ¿Qué es autotask?

autotask es un motor de automatización de tareas declarativo, escrito en Go, que ejecuta workflows basados en DAG (Directed Acyclic Graph). A diferencia de herramientas tradicionales como cron o Jenkins, autotask ofrece resolución de dependencias en tiempo real, reinteligentes inteligentes con backoff exponencial y persistencia de estado en Redis o PostgreSQL. Ideal para equipos que necesitan orquestar tareas asíncronas sin infraestructura pesada.

### Características Principales

- **Orquestación de DAG con paralelismo adaptativo**: autotask analiza automáticamente las dependencias entre tareas y ejecuta en paralelo aquellas sin conflictos, ajustando el número de workers según la carga del sistema. Cada nodo del grafo puede tener condiciones de ejecución dinámicas (ej: `if previous_task.status == "success"`). Soporta reintentos configurables por tarea y timeouts individuales.

- **Plugin system con hooks post-ejecución**: Arquitectura basada en plugins para extender funcionalidades sin modificar el core. Los hooks se disparan después de cada tarea, permitiendo logging estructurado, notificaciones a Slack/Telegram o actualización de métricas en Prometheus. Los plugins se escriben en Lua o como binarios independientes, facilitando la integración con APIs externas.

- **Modo dry-run y simulación de costos**: Antes de ejecutar un workflow, autotask permite simular el flujo completo sin efectos secundarios. Calcula el tiempo estimado de ejecución, el número de tareas paralelas y el consumo de recursos (CPU, memoria, ancho de banda). Ideal para validar pipelines en entornos de staging antes de desplegar a producción.

### Cómo Usar

1. **Definir un workflow declarativo en YAML**: Crea un archivo `autotask.yml` con la estructura de tareas, dependencias y condiciones. Por ejemplo, para un pipeline de despliegue:  
   ```yaml
   tasks:
     - id: build
       image: node:18
       command: "npm run build"
       timeout: 120s
     - id: test
       depends_on: [build]
       image: node:18
       command: "npm test"
       retry:
         max_attempts: 3
         backoff: exponential
     - id: deploy
       depends_on: [test]
       condition: "test.status == 'success'"
       command: "kubectl apply -f deploy.yaml"
   ```
   Luego ejecuta `autotask run --file autotask.yml` para iniciar el workflow.

2. **Monitorear en tiempo real con la CLI interactiva**: Usa `autotask watch --workflow-id <id>` para ver el estado de cada tarea en vivo, incluyendo logs, métricas de CPU/memoria y progreso del DAG. La CLI soporta filtros por estado (`running`, `failed`, `completed`) y exportación a JSON para integración con dashboards externos.

3. **Programar ejecuciones recurrentes con triggers**: Configura triggers basados en cron o eventos de webhook. Por ejemplo, para ejecutar un backup cada 6 horas:  
   ```bash
   autotask schedule --cron "0 */6 * * *" --file backup.yml
   ```
   O integra con GitHub Actions:  
   ```yaml
   - name: Trigger autotask
     run: |
       curl -X POST https://api.autotask.io/v1/trigger \
         -H "Authorization: Bearer ${{ secrets.AUTOTASK_TOKEN }}" \
         -d '{"workflow_id": "backup-prod"}'
   ```

### Mejores Prácticas

- **Separar configuración de secretos**: Nunca incluyas tokens, contraseñas o claves API en los archivos YAML de workflow. Usa variables de entorno o un vault externo (Hashicorp Vault, AWS Secrets Manager) referenciadas mediante `${{ secrets.MY_SECRET }}`. autotask soporta inyección automática de secretos en tiempo de ejecución sin exponerlos en logs.

- **Implementar circuit breakers en tareas críticas**: Para tareas que dependen de servicios externos (APIs, bases de datos), configura un patrón de circuit breaker con umbral de fallos consecutivos. Por ejemplo, si una tarea falla 5 veces seguidas, autotask detiene automáticamente el workflow y envía una alerta. Esto evita cascadas de fallos en pipelines complejos.
