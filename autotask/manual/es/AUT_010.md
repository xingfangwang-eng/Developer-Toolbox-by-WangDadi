# autotask - AUT_010

## Resumen Técnico

autotask es un motor de automatización de tareas diseñado para entornos SaaS donde el control granular y la ejecución predecible son críticos. No es un simple wrapper de cron jobs; es un orquestador de flujos con tolerancia a fallos integrada, diseñado para equipos que necesitan desacoplar lógica de negocio de infraestructura temporal.

## Funcionalidades Clave (Hardcore)

### 1. Ejecución con Control de Estado mediante DAGs Dinámicos
autotask no ejecuta tareas lineales. Permite definir **grafos acíclicos dirigidos (DAGs)** donde cada nodo es una tarea con dependencias explícitas. El motor valida el grafo en tiempo real, detecta ciclos y ejecuta en paralelo cuando no hay colisiones. Ideal para pipelines de ETL, despliegues multi-servicio o procesamiento de eventos encadenados.

### 2. Retry Inteligente con Backoff Exponencial y Circuit Breaker
No te conformes con reintentos lineales. autotask implementa un **sistema de reintentos adaptativo**: si una tarea falla por un error transitorio (timeout, rate limit), el motor aplica backoff exponencial con jitter. Si el error persiste más de N intentos, activa un **circuit breaker** que pausa la tarea y notifica al webhook de alertas. Esto evita saturar APIs externas o bases de datos en caída.

### 3. Webhook de Estado y Logs Estructurados en JSON
Cada ejecución de tarea genera un **payload JSON** con metadatos: timestamp, duración, código de salida, traza de errores, y variables de entorno. autotask expone un endpoint de webhook que recibe estos eventos en tiempo real. Puedes encadenar este flujo con herramientas como Datadog, Grafana o tu propio sistema de logging. No más grepping de logs planos.

## Cómo Usar (Pasos Detallados)

### 1. Definir un DAG de Tareas con Dependencias Explícitas
Crea un archivo `autotask.yml` en la raíz de tu proyecto. Define las tareas como nodos con `depends_on` para especificar orden de ejecución. Ejemplo:
```yaml
tasks:
  - id: fetch_data
    command: "./scripts/fetch.sh"
    depends_on: []
  - id: transform_data
    command: "./scripts/transform.sh"
    depends_on: ["fetch_data"]
  - id: load_data
    command: "./scripts/load.sh"
    depends_on: ["transform_data"]
```
Ejecuta `autotask run` para que el motor valide el grafo y lo ejecute secuencialmente. Si `fetch_data` falla, `transform_data` y `load_data` nunca se ejecutan.

### 2. Configurar Retry con Backoff y Circuit Breaker
En el mismo `autotask.yml`, añade configuración de reintentos por tarea:
```yaml
tasks:
  - id: api_call
    command: "./scripts/call_api.sh"
    retry:
      max_attempts: 5
      backoff_seconds: 2
      multiplier: 2
      circuit_breaker:
        threshold: 3
        cooldown_seconds: 60
```
Esto significa: si falla, espera 2s, luego 4s, luego 8s... hasta 5 intentos. Si falla 3 veces seguidas, el circuito se abre durante 60s. Durante ese tiempo, la tarea no se ejecuta aunque se llame de nuevo.

### 3. Integrar Webhook de Estado en Tiempo Real
Para recibir notificaciones de cada ejecución, añade un bloque `webhook` en la raíz del archivo de configuración:
```yaml
webhook:
  url: "https://tu-servidor.com/events"
  headers:
    Authorization: "Bearer tu-token"
  events: ["task_started", "task_completed", "task_failed"]
```
autotask enviará un POST con el payload JSON a esa URL cada vez que ocurra un evento. Puedes usar esto para alimentar un dashboard de monitoreo o activar alertas en Slack/PagerDuty.

## Mejores Prácticas para Equipos SaaS

### 1. Separar Tareas por Dominio y No por Frecuencia
No agrupes tareas por "cada 5 minutos" o "cada hora". En su lugar, agrúpalas por **dominio funcional**: tareas de ingesta de datos, tareas de limpieza, tareas de notificación. Esto facilita la reutilización de DAGs y la depuración cuando algo falla. autotask permite importar DAGs desde otros archivos, así que puedes tener un DAG base para "ingesta" y otro para "procesamiento".

### 2. Usar Variables de Entorno para Configuración Dinámica
No hardcodees rutas, tokens o URLs en los comandos. autotask expone un sistema de variables de entorno que se inyectan en tiempo de ejecución. Define un bloque `env` en el archivo de configuración:
```yaml
env:
  DB_HOST: "localhost"
  DB_PORT: "5432"
  API_KEY: "${API_KEY}"  # desde entorno del sistema
```
Esto permite que el mismo DAG funcione en desarrollo, staging y producción sin cambios en el código de las tareas.
