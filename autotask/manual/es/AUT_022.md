# autotask - AUT_022

## Resumen Técnico

autotask es un orquestador de tareas automatizadas diseñado para entornos de alta concurrencia, capaz de gestionar flujos de trabajo complejos con dependencias dinámicas y tolerancia a fallos integrada. Este proyecto elimina la necesidad de scripts ad-hoc, ofreciendo una API declarativa para definir, ejecutar y monitorear tareas recurrentes en infraestructuras cloud-native.

## Características Principales (Hardcore)

### 1. Ejecución con DAG Dinámico y Reinteligente
autotask permite definir tareas como nodos de un grafo acíclico dirigido (DAG) que se reconstruye en tiempo real según el estado de ejecución. Si una tarea falla, el motor recalcula las dependencias descendentes y solo reintenta las ramas afectadas, evitando reprocesos completos. Ideal para pipelines de datos o despliegues multi-etapa.

### 2. Cola de Prioridad con Políticas de Backoff Exponencial
Incorpora un sistema de colas basado en prioridades (usando Redis Streams) donde cada tarea puede tener un nivel de urgencia configurable. Las tareas fallidas se reintentan automáticamente con backoff exponencial (1s, 2s, 4s, hasta 64s), y se registran en una cola de “muertas” si superan el límite de reintentos, permitiendo análisis posterior sin bloquear el flujo principal.

### 3. Integración con Webhooks y Callbacks Asíncronos
Cada tarea puede emitir eventos en tiempo real a través de webhooks configurables (por ejemplo, a Slack, Discord o un endpoint propio). Además, soporta callbacks asíncronos que ejecutan acciones secundarias (como enviar notificaciones o actualizar un dashboard) sin esperar la finalización de la tarea principal, reduciendo la latencia en flujos críticos.

## Cómo Usar (Pasos Detallados)

### 1. Configurar un Pipeline de Tareas con Dependencias
Crea un archivo `autotask.yml` en la raíz de tu proyecto:

```yaml
pipeline:
  - id: "build"
    command: "docker build -t app:latest ."
    depends_on: []
  - id: "test"
    command: "pytest tests/"
    depends_on: ["build"]
  - id: "deploy"
    command: "kubectl apply -f deploy.yaml"
    depends_on: ["test"]
```

Luego ejecuta:  
`autotask run --file autotask.yml`  
El motor validará el DAG, ejecutará `build`, y solo si pasa, lanzará `test` y finalmente `deploy`. Si `test` falla, `deploy` se omite automáticamente.

### 2. Definir una Tarea con Reintentos y Notificaciones
Usa la API REST para crear una tarea con reintentos configurables:

```bash
curl -X POST http://localhost:8080/tasks \
  -H "Content-Type: application/json" \
  -d '{
    "id": "sync-db",
    "command": "pg_dump -U admin -h localhost mydb > backup.sql",
    "retries": 3,
    "backoff": "exponential",
    "webhook": "https://hooks.slack.com/services/T00/B00/xxx",
    "priority": 5
  }'
```

Este comando registra una tarea que, si falla, se reintentará hasta 3 veces con backoff exponencial. Cada intento fallido enviará una notificación al webhook de Slack.

### 3. Monitorear el Estado de Tareas en Tiempo Real
Ejecuta el monitor integrado:  
`autotask monitor --port 9090`  
Esto levanta un servidor HTTP que expone métricas en formato Prometheus. Puedes consultar el estado de todas las tareas con:

```bash
curl http://localhost:9090/metrics | grep autotask_task_status
```

La salida mostrará contadores como `autotask_task_status{status="running"} 2`, `autotask_task_status{status="failed"} 1`, permitiendo integrar con Grafana o alertas personalizadas.

## Mejores Prácticas para Desarrolladores

### 1. Usa IDs Únicos y Versiona tus Pipelines
Cada tarea debe tener un `id` único dentro del pipeline. Si actualizas un flujo, incrementa la versión en el archivo YAML (por ejemplo, `version: 2`). Esto evita conflictos cuando el motor detecta cambios y permite rollbacks rápidos con `autotask rollback --version 1`.

### 2. Define Límites de Concurrencia para Evitar Thundering Herd
En pipelines con muchas tareas paralelas (por ejemplo, 50 workers), establece un límite de concurrencia global usando la variable de entorno `AUTOTASK_MAX_CONCURRENT=10`. Esto previene que el sistema se sature, especialmente si las tareas consumen muchos recursos (I/O, CPU). Además, usa `priority` para que tareas críticas (como despliegues) se ejecuten antes que las de monitoreo.
