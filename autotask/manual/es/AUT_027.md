# autotask - AUT_027

## Resumen

autotask es un orquestador de tareas automatizadas diseñado para desarrolladores que necesitan gestionar flujos de trabajo complejos sin depender de herramientas monolíticas. Este proyecto elimina la fricción entre la planificación y la ejecución, permitiendo definir pipelines declarativos con triggers basados en eventos del sistema, cronogramas personalizados o cambios en repositorios. Ideal para equipos que buscan reducir el tiempo muerto entre commits y despliegues, o para sysadmins que requieren automatizar tareas de mantenimiento en entornos multi-nube.

## Contenido Detallado

### ¿Qué es autotask?

autotask no es un simple task scheduler; es un motor de ejecución con estado que gestiona dependencias, reintentos inteligentes y notificaciones en tiempo real. A diferencia de herramientas como cron o Jenkins, autotask ofrece una API RESTful para integración directa con CI/CD, monitoreo y sistemas de alerta. Su arquitectura basada en plugins permite extender funcionalidades sin tocar el núcleo, ideal para entornos donde cada milisegundo cuenta.

### Características Principales

- **Pipeline Declarativo con DAGs Dinámicos**: Define flujos de trabajo como grafos acíclicos dirigidos (DAGs) que se reconfiguran en tiempo de ejecución según condiciones del sistema. Ejemplo: si un nodo falla, autotask redistribuye automáticamente las tareas a nodos redundantes sin intervención manual.

- **Trigger Multimodal con Filtros Semánticos**: Activa tareas basadas en eventos de Git (push, merge, tag), métricas de Prometheus (CPU > 85%), o incluso cambios en archivos específicos dentro de un bucket S3. Los filtros permiten expresiones regulares y lógica booleana para evitar falsos positivos.

- **Ejecución Aislada con Contenedores Efímeros**: Cada tarea se ejecuta en un contenedor Docker o Podman que se destruye al finalizar, garantizando cero contaminación del entorno. autotask inyecta variables de entorno y secretos desde HashiCorp Vault o AWS Secrets Manager sin exponerlos en logs.

### Cómo Usar

1. **Configurar un Pipeline con Dependencias Condicionales**  
   Crea un archivo `autotask.yml` en la raíz de tu proyecto. Define tareas con `depends_on` y condiciones `if` basadas en salidas de tareas anteriores. Por ejemplo:
   ```yaml
   tasks:
     - id: build
       image: node:18
       script: npm run build
     - id: test
       depends_on: [build]
       if: "{{ tasks.build.exit_code == 0 }}"
       script: npm test
   ```
   Ejecuta `autotask run --pipeline autotask.yml` para iniciar el flujo. autotask valida el DAG antes de ejecutar, evitando ciclos infinitos.

2. **Integrar con Webhooks para Despliegues Continuos**  
   Expón un endpoint POST en `http://localhost:8080/webhook/github`. Configura tu repositorio para enviar eventos `push`. autotask filtra por rama (ej: `main`) y lanza un pipeline de build + deploy a Kubernetes. Usa `autotask webhook --port 8080 --secret your_secret` para iniciar el servidor. Los logs se centralizan en stdout con formato JSON para ingestión en ELK.

3. **Automatizar Backups con Rotación y Notificaciones**  
   Define una tarea programada con `schedule: "0 3 * * *"` (diario a las 3 AM). autotask ejecuta un script que comprime archivos, los sube a S3 y elimina backups mayores a 30 días. Si falla, envía una alerta a Slack con el mensaje de error. Usa `autotask schedule --cron "0 3 * * *" --task backup` para registrar la tarea. El estado se persiste en SQLite local para auditoría.

### Mejores Prácticas

- **Divide tareas complejas en micro-pipelines**: En lugar de un solo script monolítico, segmenta en tareas independientes con `depends_on`. Esto permite reintentos parciales y facilita la depuración. Por ejemplo, separa la compilación, pruebas unitarias y despliegue en pipelines distintos que se encadenan solo si el anterior es exitoso.

- **Usa etiquetas para priorizar y filtrar logs**: Asigna etiquetas como `critico`, `mantenimiento` o `dev` a cada tarea. autotask permite consultar logs por etiqueta (`autotask logs --tag critico`) y establecer prioridades de ejecución. Esto es vital en entornos con alta concurrencia para evitar que tareas de baja prioridad bloqueen las críticas.
