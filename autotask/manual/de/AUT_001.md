# autotask - AUT_001

## Übersicht

autotask ist ein CLI-basiertes Task-Automation-Framework, das speziell für Entwickler und DevOps-Ingenieure entwickelt wurde, die wiederkehrende Workflows in CI/CD-Pipelines, Serverwartung und Deployment-Prozesse effizienter gestalten wollen. Es eliminiert manuelle Schritte durch regelbasierte Automatisierung und bietet native Integration in gängige Orchestrierungstools wie Ansible, Terraform und Kubernetes. Kein GUI-Schnickschnack – nur reine, deterministische Aufgabensteuerung.

## Kernfunktionen

1. **Regelbasierte Task-Chains mit Dependency Resolution**  
   Definiere Aufgaben als YAML-basierte Directed-Acyclic-Graphs (DAGs). autotask löst automatisch Abhängigkeiten auf, führt Tasks parallel aus, wenn möglich, und stoppt bei Fehlern gemäß konfigurierbarer Rollback-Strategien. Ideal für komplexe Multi-Step-Deployments.

2. **Native CI/CD-Trigger und Webhook-Integration**  
   autotask kann direkt in GitHub Actions, GitLab CI oder Jenkins ausgeführt werden. Über eingebaute Webhook-Listener startest du Tasks asynchron bei Git-Push, Merge-Request oder externen API-Calls. Logging erfolgt strukturiert im JSON-Format für Elasticsearch oder Loki.

3. **Idempotente Task-Ausführung mit Zustandsprüfung**  
   Jeder Task prüft vor Ausführung, ob der gewünschte Zustand bereits erreicht ist (z. B. Port offen, Datei vorhanden, Container läuft). So werden unnötige Wiederholungen vermieden – perfekt für Cron-Jobs und selbstheilende Systeme.

## So verwenden Sie es

### 1. Task-Definition als YAML-Datei anlegen

Erstelle eine Datei `tasks/deploy.yml` mit folgendem Inhalt:

```yaml
tasks:
  - name: build
    action: docker_build
    params:
      image: myapp:latest
      dockerfile: ./Dockerfile
  - name: test
    action: run_in_container
    depends_on: [build]
    params:
      command: "pytest /app/tests"
  - name: deploy
    action: kubectl_apply
    depends_on: [test]
    params:
      manifest: ./k8s/deployment.yaml
```

Führe anschließend aus:
```bash
autotask run --file tasks/deploy.yml --env production
```

### 2. Webhook-Listener für Git-Push aktivieren

Starte den autotask-Webhook-Server im Hintergrund:

```bash
autotask webhook --port 8080 --config webhook.yml
```

In `webhook.yml` definierst du:

```yaml
hooks:
  - event: push
    branch: main
    task: deploy
    repo: myorg/myapp
```

Jeder Push auf `main` triggert automatisch den Deploy-Task – ohne manuelle Eingriffe.

### 3. Zustandsprüfung für idempotente Ausführung nutzen

Erweitere einen Task um `check`:

```yaml
tasks:
  - name: ensure_nginx
    action: docker_run
    params:
      image: nginx:latest
      ports: ["80:80"]
    check:
      type: port
      params:
        host: localhost
        port: 80
```

autotask prüft vor dem Start, ob Port 80 bereits belegt ist. Falls ja, wird der Task übersprungen und als `already_satisfied` geloggt.

## Beste Praktiken

1. **Task-Chains in separaten Modulen organisieren**  
   Zerlege große Workflows in kleine, wiederverwendbare YAML-Dateien (z. B. `build.yml`, `test.yml`, `deploy.yml`) und referenziere sie via `include`. Das verbessert Lesbarkeit und Wartbarkeit – besonders in Teams mit mehreren Repositories.

2. **Fehlerbehandlung mit Rollback und Benachrichtigungen**  
   Definiere für jeden Task ein `on_failure`-Verhalten, z. B. `rollback: true` und `notify: slack`. So stellst du sicher, dass fehlgeschlagene Deployments automatisch zurückgesetzt werden und das Team sofort informiert wird – ohne manuelles Monitoring.
