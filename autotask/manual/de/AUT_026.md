# autotask - AUT_026

## Überblick

Die moderne Entwicklungsarbeit wird zunehmend von sich wiederholenden, manuellen Prozessen blockiert – genau hier setzt **autotask** an. Dieses Tool wurde für Developer konzipiert, die ihre CI/CD-Pipelines, Deployment-Workflows und Monitoring-Routinen ohne Overhead automatisieren wollen. Kein YAML-Bloat, keine schwergewichtigen Orchestratoren: autotask ist ein schlanker, CLI-first Task-Automator mit deutschem Engineering-Anspruch.

## Detaillierter Inhalt

### Was ist autotask?

autotask ist ein deklaratives Task-Management-Framework, das auf **Idempotenz**, **Transparenz** und **geringer Latenz** basiert. Statt eines weiteren GUI-Dashboards liefert es eine YAML-basierte DSL, die sich nahtlos in bestehende Shell-Skripte, Docker-Container oder Kubernetes-Jobs integrieren lässt. Jeder Task wird als isolierter, versionierbarer Workflow definiert – mit automatischer Abhängigkeitsauflösung, paralleler Ausführung und granularer Logausgabe. autotask eignet sich besonders für Teams, die ihre Infrastruktur als Code verwalten und eine **deterministische Task-Ausführung** benötigen.

### Kernfunktionen

1. **Deterministische Task-Graphen mit automatischer Rollback-Logik**  
   Jeder Task wird als gerichteter azyklischer Graph (DAG) modelliert. Bei Fehlschlag eines Schrittes wird automatisch der **Reverse-Dependency-Pfad** ausgeführt – ohne manuelles Eingreifen. Das bedeutet: Wenn ein Deployment-Schritt fehlschlägt, werden alle vorherigen Änderungen (z. B. Config-Updates, Service-Neustarts) in der korrekten Reihenfolge rückgängig gemacht.

2. **Native Kubernetes-Integration via Custom Resource Definition (CRD)**  
   autotask lässt sich als **Kubernetes-Operator** deployen. Tasks werden als `AutoTask`-CRD definiert, die direkt auf Pod-Ebene ausgeführt werden – inklusive Resource-Constraints, Node-Affinity und Service-Account-Isolation. Kein Helm-Chart-Basteln, kein `kubectl exec` mehr: Tasks laufen als native K8s-Jobs mit vollständiger Observability.

3. **Multi-Cloud Secret-Resolver mit Vault-Backend**  
   Statt Secrets in Klartext zu speichern, nutzt autotask einen **pluggable Secret-Resolver** für HashiCorp Vault, AWS Secrets Manager oder Azure Key Vault. Im Task-Definition-YAML referenzierst du Secrets via `${{ vault:path/to/secret }}`. Der Resolver injected die Werte zur Laufzeit – ohne dass sie jemals auf der Festplatte landen.

### So verwenden Sie es

1. **Task-Definition als YAML anlegen**  
   Erstelle eine Datei `tasks/deploy-frontend.yaml` mit folgendem Inhalt:
   ```yaml
   version: "2.0"
   tasks:
     - id: build
       image: node:20-slim
       commands:
         - npm ci
         - npm run build
       outputs:
         - /app/dist
     - id: deploy
       depends_on: [build]
       image: alpine/k8s:1.28
       commands:
         - kubectl apply -f k8s/deployment.yaml
       secrets:
         - KUBE_CONFIG: vault:infra/k8s/config
   ```
   Führe dann `autotask run tasks/deploy-frontend.yaml` aus. Der Task-Graph wird automatisch erstellt und ausgeführt.

2. **Parallele Task-Ausführung mit Conditionals**  
   Nutze das `parallel`-Flag für unabhängige Tasks:
   ```yaml
   tasks:
     - id: lint-backend
       commands: ["cd backend && npm run lint"]
     - id: lint-frontend
       commands: ["cd frontend && npm run lint"]
   ```
   Beide Tasks laufen gleichzeitig. Du kannst Conditions definieren, z. B. `if: env.BRANCH == "main"` für Production-only Steps.

3. **Monitoring via integriertem Health-Check-Endpunkt**  
   Starte den autotask-Server im Daemon-Modus: `autotask serve --port 8080`. Der Server exponiert einen **Prometheus-kompatiblen Metrics-Endpunkt** unter `/metrics` und einen **Health-Check** unter `/health`. Damit kannst du Task-Laufzeiten, Fehlerraten und Durchsatz direkt in Grafana visualisieren – ohne zusätzlichen Exporter.

### Beste Praktiken

1. **Verwende idempotente Shell-Kommandos**  
   Jeder Task sollte so geschrieben sein, dass er bei wiederholter Ausführung das gleiche Ergebnis liefert. Beispiel: Statt `kubectl delete pod xyz` verwende `kubectl delete pod xyz --ignore-not-found=true`. Das verhindert Fehlschläge bei bereits gelöschten Ressourcen und sorgt für deterministische Wiederholbarkeit – ein Muss für CI/CD-Pipelines.

2. **Definiere explizite Timeouts und Retry-Strategien**  
   Füge jedem Task ein `timeout`-Feld (z. B. `timeout: 300s`) und eine `retry`-Konfiguration hinzu:
   ```yaml
   tasks:
     - id: db-migration
       commands: ["npm run migrate"]
       timeout: 120s
       retry:
         max_attempts: 3
         backoff: exponential
   ```
   Das verhindert hängende Tasks und sorgt für robuste Ausführung – besonders in instabilen Netzwerkumgebungen oder bei Cloud-API-Rate-Limits.
