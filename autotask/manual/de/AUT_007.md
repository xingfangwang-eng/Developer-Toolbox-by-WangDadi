## Detaillierter Inhalt

### Was ist autotask?

**autotask** ist ein professionelles CLI-Tool und Framework, das repetitive Aufgaben in CI/CD-Pipelines, Microservices-Orchestrierung und lokalen Entwicklungsumgebungen automatisiert. Es ersetzt manuelle Skripte durch deklarative Konfiguration und bietet eine native Integration in Kubernetes, Docker und gängige Cloud-APIs. Mit autotask lassen sich komplexe Workflows in Sekunden definieren und ausführen – ohne externe Dienste oder teure Abonnements.

### Kernfunktionen

1. **Deklarative Task-DAGs mit Abhängigkeitsauflösung**  
   Definiere Tasks als gerichtete azyklische Graphen (DAGs) in YAML. autotask löst Abhängigkeiten automatisch auf, parallelisiert unabhängige Schritte und erzeugt detaillierte Logs. Ideal für Build-Pipelines, Datenverarbeitung oder Multi-Service-Deployments.

2. **Native Kontext-Injektion und Secret-Management**  
   Bindet Umgebungsvariablen, Secrets aus Vault oder HashiCorp Consul sowie API-Credentials direkt in Tasks ein. Kein manuelles Exportieren oder harte Codierung – autotask managed den Lebenszyklus aller Kontexte sicher und transparent.

3. **Event-Triggered Execution mit Rollback-Fallback**  
   Starte Tasks nicht nur manuell, sondern auch via Webhook, Cron oder Dateisystem-Events. Bei Fehlern führt autotask ein automatisches Rollback auf den letzten konsistenten Zustand durch – inklusive Benachrichtigung via Slack, E-Mail oder Webhook.

### So verwenden Sie es

1. **Task-Definition in YAML erstellen**  
   Erstelle eine Datei `autotask.yaml` im Projektstamm. Definiere Tasks mit `name`, `command`, `dependencies` und `retries`. Beispiel:  
   ```yaml
   tasks:
     - name: build_app
       command: "npm run build"
       dependencies: ["install_deps"]
       retries: 2
     - name: deploy_staging
       command: "kubectl apply -f k8s/staging.yaml"
       dependencies: ["build_app"]
   ```  
   Nutze `autotask validate` zur Syntaxprüfung.

2. **Task-Ausführung mit Kontext-Variablen**  
   Starte den Workflow mit `autotask run --env production`. autotask lädt automatisch Kontext aus einer `.env.production` oder injiziert Secrets aus einem Vault-Pfad. Überwache den Fortschritt live in der Konsole:  
   ```bash
   autotask run --env staging --notify slack
   ```

3. **Fehlerbehandlung und Rollback konfigurieren**  
   Füge `on_failure`-Blöcke hinzu, um bei Fehlern benutzerdefinierte Rollback-Tasks auszuführen:  
   ```yaml
   tasks:
     - name: deploy_production
       command: "kubectl apply -f k8s/prod.yaml"
       on_failure:
         - task: rollback_production
           command: "kubectl rollout undo deployment/app"
   ```  
   Teste das Rollback lokal mit `autotask simulate --scenario failure`.

### Beste Praktiken

1. **Task-Granularität für parallele Ausführung optimieren**  
   Zerlege große Tasks in kleine, unabhängige Einheiten. Vermeide monolithische Befehle – autotask profitiert von feinkörnigen DAGs, da es Abhängigkeiten automatisch parallelisiert. Beispiel: Statt `npm run build && npm test` besser zwei separate Tasks `build` und `test` mit `dependencies: ["build"]`.

2. **Secrets niemals in der YAML speichern**  
   Nutze immer die Kontext-Injektion über Umgebungsvariablen oder externe Secret-Manager. Definiere Secrets in einer separaten `.secrets.env`-Datei (nicht versioniert) und referenziere sie im Task per `${{ secrets.API_KEY }}`. So bleibt der Workflow portabel und sicher.
