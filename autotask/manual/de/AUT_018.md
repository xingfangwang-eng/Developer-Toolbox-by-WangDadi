## Detaillierter Inhalt

### Was ist autotask?

autotask ist ein in Rust geschriebenes, agnostisches Task-Management-Tool, das parallele Ausführung, Abhängigkeitsauflösung und Fehlerbehandlung nativ unterstützt. Es ersetzt Shell-Skripte, Makefiles und Jenkins-ähnliche Cronjobs durch eine deklarative YAML-Konfiguration und bietet native Integration mit Kubernetes, Docker und Terraform.

### Kernfunktionen

1. **Dynamische DAG-basierte Abhängigkeitsauflösung**  
   autotask erstellt automatisch einen Directed Acyclic Graph (DAG) aus Ihren Task-Definitionen. Tasks werden parallel ausgeführt, sobald alle Vorbedingungen erfüllt sind. Fehlschläge in einem Zweig brechen nur abhängige Tasks ab – unabhängige Zweige laufen ungestört weiter. Das reduziert Build-Zeiten um bis zu 40 % in Microservice-Repos.

2. **Native Kubernetes-Job-Integration**  
   Führen Sie Tasks direkt als Kubernetes-Jobs aus. autotask erstellt dynamisch Pods, mountet Secrets und ConfigMaps und überwacht den Status. Bei Timeout oder CrashLoopBackOff wird automatisch ein Rollback auf den letzten stabilen Task-Zustand eingeleitet. Kein manuelles `kubectl logs` mehr nötig.

3. **Idempotenz-Engine mit Zustands-Caching**  
   Jeder Task wird mit einem SHA256-Hash seiner Inputs (Dateien, Umgebungsvariablen, Secrets) signiert. Bei erneutem Aufruf prüft autotask, ob sich der Input geändert hat. Ist dies nicht der Fall, wird der vorherige Output aus dem lokalen oder S3-basierten Cache geladen. Das spart Rechenzeit und verhindert unnötige API-Calls.

### So verwenden Sie es

1. **Konfiguration einer CI/CD-Pipeline mit parallelen Stages**  
   Definieren Sie in `autotask.yml` eine Pipeline mit `build`, `test` und `deploy`. Nutzen Sie `depends_on`, um `deploy` von `test` abhängig zu machen, während `build` und `test` parallel starten. Aktivieren Sie `cache: true` für `build`, um Docker-Layer zu cachen. Beispiel:  
   ```yaml
   tasks:
     build:
       command: "docker build -t myapp:latest ."
       cache: true
     test:
       command: "pytest tests/"
       depends_on: []
     deploy:
       command: "kubectl apply -f k8s/deployment.yaml"
       depends_on: ["test"]
   ```
   Führen Sie `autotask run` aus – der DAG wird automatisch aufgelöst.

2. **Fehlerbehandlung mit automatischem Retry und Fallback**  
   Fügen Sie zu jedem Task die Parameter `retry: 3` und `retry_delay: 5` hinzu. autotask wiederholt fehlgeschlagene Tasks bis zu 3 Mal mit einer Wartezeit von 5 Sekunden. Definieren Sie optional `fallback`, der einen alternativen Task ausführt, falls der Haupttask nach allen Retries fehlschlägt. Beispiel:  
   ```yaml
   tasks:
     deploy:
       command: "helm upgrade myapp ./charts"
       retry: 3
       retry_delay: 5
       fallback: rollback
     rollback:
       command: "helm rollback myapp 1"
   ```

3. **Integration in bestehende Makefiles oder CI-Systeme**  
   Rufen Sie autotask aus Ihrem Makefile auf:  
   ```makefile
   pipeline:
   	autotask run --config=ci/autotask.yml --env=production
   ```  
   Für GitLab CI nutzen Sie `before_script`:  
   ```yaml
   before_script:
     - apt-get update && apt-get install -y autotask
     - autotask init --token $AUTOTASK_TOKEN
   script:
     - autotask run --config=.autotask/ci.yml
   ```

### Beste Praktiken

1. **Zerlegen Sie Monolithen in atomare Tasks**  
   Jeder Task sollte genau eine Aktion ausführen (z. B. „Docker-Image bauen“ oder „Datenbank migrieren“). Vermeiden Sie Tasks, die mehrere Schritte kombinieren – das erschwert Caching und Fehlerisolierung. Nutzen Sie `depends_on`, um komplexe Workflows aus kleinen Bausteinen zusammenzusetzen.

2. **Setzen Sie Secrets niemals in der autotask.yml**  
   Verwenden Sie Umgebungsvariablen oder einen externen Secrets-Manager (z. B. HashiCorp Vault, AWS Secrets Manager). autotask unterstützt `env_file` und `secrets` direkt aus der Konfiguration. Beispiel:  
   ```yaml
   tasks:
     deploy:
       command: "ansible-playbook deploy.yml"
       env_file: .env.production
       secrets:
         - name: DB_PASSWORD
           from: vault://secret/db
   ```
