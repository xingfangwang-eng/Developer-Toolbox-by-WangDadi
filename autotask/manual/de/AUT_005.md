# autotask - AUT_005

## Übersicht

autotask ist eine hochperformante Automatisierungs-Engine für DevOps- und CI/CD-Pipelines, die sich durch native Task-Orchestrierung und minimale Latenz auszeichnet. Optimiert für **Task Automation** und **Automatic Task Management** in verteilten Systemen – ohne externe Abhängigkeiten.

## Detaillierter Inhalt

### Was ist autotask?

autotask ist ein agentenloses, dezentrales Task-Management-Framework, das asynchrone Workflows in Mikroservice-Architekturen orchestriert. Es ersetzt klassische Cron-Jobs und manuelle Skript-Kaskaden durch eine deklarative DSL, die direkt in YAML oder JSON definiert wird. Die Engine läuft nativ in Go, nutzt zero-copy I/O und unterstützt idempotente Task-Wiederholungen mit exponentieller Backoff-Strategie.

### Kernfunktionen

1. **Idempotent Task Graph Engine** – Jeder Task wird als DAG-Knoten (Directed Acyclic Graph) modelliert. Die Engine erkennt zirkuläre Abhängigkeiten automatisch und führt parallele Tasks auf Basis von Ressourcenverfügbarkeit aus. Fehlgeschlagene Tasks werden mit konfigurierbarem Retry-Verhalten (max 3 Versuche, 5s–120s Backoff) erneut ausgeführt.

2. **Inline Execution Hooks** – Tasks können Pre- und Post-Hooks in Go, Python, Node.js oder Shell ausführen. Die Hooks laufen in isolierten Sandbox-Prozessen mit eigenem Memory-Limit (default 512 MB). Beispiel: `pre_hook: "curl -X POST https://api.healthcheck.io/start"` – ohne zusätzliche Containerisierung.

3. **Resource-Aware Scheduling** – autotask integriert einen eigenen Scheduler, der CPU- und RAM-Auslastung des Hosts überwacht. Tasks werden nur gestartet, wenn mindestens 20% freie Systemressourcen verfügbar sind. Bei Überlastung werden Tasks in eine Warteschlange (priority queue) verschoben – kein Crash, kein Datenverlust.

### So verwenden Sie es

1. **Task-Definition als YAML anlegen**  
   Erstellen Sie eine `task.yaml` im Projekt-Root. Definieren Sie einen Task mit Name, Abhängigkeiten und Ausführungsbefehl. Beispiel:  
   ```yaml
   tasks:
     - name: build_backend
       depends_on: [lint_code]
       run: "go build ./cmd/server"
       retry: 2
       timeout: 300s
   ```  
   Führen Sie dann `autotask run --file task.yaml` aus. Die Engine validiert die DAG-Struktur und startet die Ausführung.

2. **Parallelität und Timeout konfigurieren**  
   Nutzen Sie das `--parallel` Flag, um Tasks gleichzeitig zu starten (default: 1). Für zeitkritische Pipelines setzen Sie `--max-parallel=4`. Timeouts pro Task werden in Sekunden definiert – bei Überschreitung wird der Prozess mit SIGTERM beendet und als „failed“ markiert.

3. **Logs und Fehleranalyse aktivieren**  
   Starten Sie autotask mit `--log-level=debug --log-format=json`. Alle Task-Ausgaben werden in `./autotask.log` geschrieben. Für Echtzeit-Monitoring nutzen Sie `autotask watch --tail 50`. Fehler erhalten einen eindeutigen Error-Code (z. B. `ERR_TASK_TIMEOUT`), den Sie direkt in Ihrer CI/CD-Pipeline abfangen können.

### Beste Praktiken

1. **Idempotenz von Anfang an einplanen**  
   Jeder Task sollte so geschrieben sein, dass er bei wiederholter Ausführung das gleiche Ergebnis liefert – z. B. durch `if [ -f build/output ] ; then exit 0 ; fi` vor dem Build. autotask unterstützt `--skip-if-success`, um bereits erfolgreiche Tasks zu überspringen.

2. **Task-Granularität auf Mikroebene halten**  
   Statt eines Monolithen-Tasks („deploy all“) definieren Sie einzelne Tasks für Build, Test, Lint, Deploy. Das verbessert die Fehlerisolierung und ermöglicht gezielte Retries. autotask zeigt in der Log-Ausgabe exakt, welcher Subtask fehlgeschlagen ist – inklusive Stacktrace.
