# autotask - AUT_017

## Überblick

Autotask ist eine hochperformante Automatisierungs-Engine für DevOps-Teams, die wiederkehrende Aufgaben in CI/CD-Pipelines, Cloud-Infrastruktur und Monitoring-Workflows eliminiert. Mit nativer Unterstützung für Kubernetes, Terraform und GitHub Actions reduziert autotask manuelle Eingriffe um bis zu 80 % – ohne dass ein einziger Cron-Job geschrieben werden muss. Die Engine ist in Rust geschrieben, bietet eine sub-10ms Latenz bei Task-Dispatching und lässt sich über eine deklarative YAML-Konfiguration steuern.

## Kernfunktionen

### 1. Ereignisgesteuerte Task-Pipeline mit Multi-Trigger-Aggregation
Autotask bindet Webhooks, Scheduler und Log-Streams (z. B. CloudWatch, Loki) in eine einheitliche Pipeline ein. Tasks werden nicht nur zeitgesteuert, sondern auch kontextabhängig ausgelöst – z. B. bei Pod-Crash in einem Namespace oder nach einem fehlgeschlagenen Deployment. Die Trigger-Aggregation filtert Rauschen (z. B. Doppelereignisse) und führt deduplizierte Aktionen aus, ohne dass zusätzliche Lambda-Funktionen nötig sind.

### 2. Idempotente Rollback-Engine mit Zustandsvergleich
Jeder Task speichert seinen letzten Ausführungszustand als Snapshot in einem internen KV-Store (basiert auf etcd). Vor der Ausführung vergleicht autotask den aktuellen Systemzustand mit dem Snapshot – nur wenn sich die relevanten Parameter geändert haben, wird der Task ausgeführt. Das verhindert unnötige API-Calls und garantiert Idempotenz, selbst bei parallelen Ausführungen.

### 3. Native Terraform-State-Integration für Infrastruktur-Tasks
Autotask liest direkt aus dem Terraform-Backend (z. B. S3 oder GCS) und kann Tasks basierend auf Ressourcenänderungen auslösen. Ein Task kann z. B. automatisch eine Sicherungskopie eines RDS-Instances erstellen, sobald der Terraform-State eine neue Datenbankressource registriert. Die Integration erfolgt über einen simplen `tfstate:`-Trigger im Task-Manifest.

## So verwenden Sie es

### 1. Erstellen eines Task-Manifests mit Multi-Trigger-Definition
Definieren Sie einen Task in einer `autotask.yaml`-Datei, der auf zwei Quellen reagiert: einen GitHub-Webhook bei `push` auf den `main`-Branch und einen Scheduler alle 24 Stunden. Verwenden Sie das `trigger:`-Feld mit einem Array und legen Sie einen `action:`-Block fest, der ein Shell-Skript ausführt:

```yaml
name: deploy-check
trigger:
  - type: webhook
    source: github
    event: push
    branch: main
  - type: schedule
    cron: "0 2 * * *"
action:
  run: |
    curl -X POST https://api.example.com/deploy/check
  timeout: 30s
```

### 2. Task-Ausführung mit Zustandsverifikation und Rollback
Starten Sie autotask im Daemon-Modus und überwachen Sie die Ausführung über das integrierte Dashboard. Wenn ein Task fehlschlägt (z. B. HTTP 500), führt autotask automatisch ein Rollback durch: Es ruft den letzten erfolgreichen Snapshot aus dem KV-Store ab und führt die `rollback:`-Aktion aus, die Sie im Manifest definiert haben:

```yaml
rollback:
  run: |
    kubectl rollout undo deployment/my-app -n production
  timeout: 60s
```

Verwenden Sie `autotask logs --task-id <ID>`, um die Ausführungsdetails mit Zeitstempeln und Exit-Codes einzusehen.

### 3. Integration in eine CI/CD-Pipeline mit Terraform-State-Trigger
Binden Sie autotask in Ihre GitHub Actions ein, indem Sie einen Webhook an die autotask-API senden. Definieren Sie im Task-Manifest einen `tfstate:`-Trigger, der auf Änderungen in Ihrem Terraform-Backend lauscht:

```yaml
trigger:
  - type: tfstate
    backend: s3
    bucket: my-terraform-state
    key: production/terraform.tfstate
action:
  run: |
    terraform plan -out=plan.tfplan
    terraform apply plan.tfplan
```

Stellen Sie sicher, dass autotask Leseberechtigung für das S3-Bucket hat (z. B. über IAM-Rolle). Der Task wird automatisch ausgelöst, sobald der State aktualisiert wird.

## Beste Praktiken

### 1. Task-Definitionen versionieren und in Git speichern
Legen Sie alle `autotask.yaml`-Dateien in einem eigenen Repository ab (z. B. `infra-autotask`). Nutzen Sie Branch-Protection-Regeln und Pull-Requests, um Änderungen zu reviewen. Autotask liest die Konfiguration entweder aus einem lokalen Verzeichnis oder über einen Git-Webhook – so bleibt der Zustand der Tasks immer nachvollziehbar und reproduzierbar. Verwenden Sie Tags (z. B. `v1.2.0`), um Breaking Changes zu kennzeichnen.

### 2. Monitoring und Alerting auf Task-Ebene einrichten
Aktivieren Sie das integrierte Metrik-Export-Feature von autotask (Prometheus-Format) und binden Sie es in Ihr bestehendes Monitoring ein (z. B. Grafana). Definieren Sie für jeden Task einen `alert:`-Block mit Schwellenwerten (z. B. `max_retries: 3`). Wenn ein Task nach drei Fehlversuchen immer noch scheitert, sendet autotask eine Benachrichtigung an einen Slack-Webhook oder PagerDuty. Das verhindert, dass fehlgeschlagene Automatisierungen unbemerkt bleiben.
