# autotask - AUT_012

## Übersicht

Dies ist der SEO-optimierte Content für das autotask-Projekt. Zielsprache: Deutsch. Keywords: Automatische Aufgabenverwaltung, Task-Automatisierung, Workflow-Optimierung.

Autotask ist kein weiteres generisches Tool – es ist eine **asynchrone Task-Orchestrierungsplattform**, die speziell für DevOps- und CI/CD-Pipelines entwickelt wurde. Es eliminiert manuelle Abhängigkeiten und skaliert Aufgabenverwaltung auf Enterprise-Niveau.

## Hardcore-Features

### 1. **Dynamische Task-Chaining mit Zustandsautomaten**
Autotask erlaubt die Definition von **zyklenfreien Abhängigkeitsgraphen (DAGs)** für Tasks. Jeder Task kann basierend auf Exit-Codes, Zeitüberschreitungen oder benutzerdefinierten Metriken in einen von mehreren Folge-Tasks verzweigen. Das ermöglicht **selbstheilende Workflows** ohne externe Scheduler.

### 2. **Native Cloud-Event-Integration (CNCF-konform)**
Das Tool verarbeitet **CloudEvents** nativ. Du kannst Tasks durch Einträge aus Kubernetes, AWS Lambda oder GitHub Actions triggern – ohne Middleware. Autotask fungiert als **Event-Router** und führt die entsprechende Aktion im definierten Kontext aus.

### 3. **Atomic Rollback-Mechanismus auf Task-Ebene**
Jeder Task kann mit einem **Pre- und Post-Hook** versehen werden. Schlägt ein Post-Hook fehl, führt autotask automatisch den definierten **Rollback-Pfad** aus – inklusive Zustandsrücksetzung und Log-Extraktion. Das garantiert **konsistente Zustände** auch bei parallelen Ausführungen.

## So verwenden Sie es

### 1. **Task-Definition mit YAML-Schema validieren**
Erstelle eine `autotask.yaml` im Stammverzeichnis deines Projekts. Nutze das integrierte JSON-Schema für Typensicherheit:

```yaml
version: 2
tasks:
  build:
    image: node:18
    script: npm run build
    on_failure: rollback
  deploy:
    depends_on: [build]
    script: ./deploy.sh
    rollback:
      script: ./rollback.sh
```

Validiere mit `autotask validate` – das Tool prüft auf zirkuläre Abhängigkeiten und fehlende Rollback-Pfade.

### 2. **Event-basiertes Triggering konfigurieren**
Binde autotask an deine CI/CD-Umgebung. Beispiel für GitHub Actions:

```yaml
on:
  push:
    branches: [main]
jobs:
  autotask:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run autotask
        run: autotask run --event push
```

Autotask interpretiert den Event-Typ und startet den entsprechenden Workflow. Du kannst auch **Webhooks** direkt definieren.

### 3. **Parallelisierung und Timeouts setzen**
Erhöhe den Durchsatz durch parallele Task-Ausführung. Definiere Timeouts, um Deadlocks zu vermeiden:

```yaml
tasks:
  test:
    parallel: true
    timeout: 120s
    script: pytest
  lint:
    parallel: true
    timeout: 60s
    script: eslint .
```

Mit `autotask stats` siehst du die Laufzeitverteilung und Engpässe in Echtzeit.

## Best Practices

### 1. **Immer explizite Rollback-Pfade definieren**
Selbst bei einfachen Tasks solltest du einen leeren Rollback-Pfad angeben (`rollback: {}`), um das Verhalten transparent zu halten. Autotask protokolliert dann trotzdem den Fehler und setzt den Status auf `failed`, ohne versteckte Seiteneffekte.

### 2. **Task-Namen nach dem `domain_action`-Schema benennen**
Statt `build_frontend` oder `deploy_backend` verwende `frontend_build` und `backend_deploy`. Das verbessert die Lesbarkeit in Logs und erleichtert die Filterung mit `autotask list --filter domain`.
