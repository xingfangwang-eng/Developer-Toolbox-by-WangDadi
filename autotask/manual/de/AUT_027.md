# autotask - AUT_027

## Überblick

autotask ist eine schlanke, aber extrem leistungsfähige CLI-gestützte Automatisierungsengine, die speziell für Developer entwickelt wurde, die wiederkehrende Aufgaben in CI/CD-Workflows, Deployment-Routinen und Multi-Cloud-Umgebungen eliminieren wollen. Kein GUI-Overhead, keine Abhängigkeit von Drittanbietern – nur reine, deklarative Task-Automation via YAML oder TypeScript.

## Harte Kernfunktionen

### 1. **Polyglotte Task-Chains mit verzögerter Ausführung**
Statt nur Shell-Befehle zu wrappen, unterstützt autotask native Task-Blöcke in **Bash, Python, Node.js und Go** innerhalb einer einzigen Pipeline. Tasks können zudem über `await` oder `defer` gesteuert werden – ideal für asynchrone Cloud-API-Aufrufe oder zeitgesteuerte Backups.

### 2. **Kontextuelles Rollback & State-Tracking**
Jeder Task hinterlässt einen **deterministischen Zustands-Hash**. Tritt ein Fehler auf, wird automatisch ein kontextuelles Rollback ausgelöst – basierend auf dem letzten validen Zustand. Kein manuelles Reset-Skript mehr nötig. Perfekt für Kubernetes-Operator-Pipelines oder Terraform-Post-Processing.

### 3. **Multi-Tenant Task-Scheduling mit RBAC**
autotask erlaubt die Definition von **Task-Pools** mit granularer Rechteverwaltung (Read/Execute/Admin). Tasks können für verschiedene Teams oder Projekte isoliert werden – mit eigenem Logging, eigener Queue und optionalem Slack/Webhook-Trigger. Ideal für Plattform-Teams, die interne Automatisierung für mehrere Produktteams bereitstellen.

## So verwenden Sie es (Professionelle Anleitung)

### 1. **Task-Definition mit Abhängigkeitsgraph**
Erstellen Sie eine `autotask.yml` im Projektroot. Definieren Sie Tasks als DAG (Directed Acyclic Graph) mit expliziten `depends_on`-Referenzen:

```yaml
tasks:
  build:
    run: "npm run build"
  test:
    run: "npm test"
    depends_on: [build]
  deploy:
    run: "kubectl apply -f deploy.yaml"
    depends_on: [test]
    env:
      KUBECONFIG: "./kubeconfig-staging"
```

Führen Sie dann aus: `autotask run deploy` – autotask löst automatisch alle Abhängigkeiten in der richtigen Reihenfolge aus.

### 2. **Parallelisierung & Timeout-Steuerung**
Nutzen Sie `parallel`-Blöcke für unabhängige Tasks. Setzen Sie `timeout` pro Task, um hängende Prozesse zu vermeiden:

```yaml
tasks:
  lint:
    run: "eslint src/"
  format:
    run: "prettier --check src/"
  security:
    run: "trivy fs ."
    timeout: 120

parallel: [lint, format, security]
```

Mit `autotask run --parallel` starten Sie alle unabhängigen Tasks gleichzeitig – Logging erfolgt farbcodiert pro Task.

### 3. **Secrets & Environment-Injection**
Definieren Sie Secrets in einer `.autotask.secrets.env` (nie committed). Nutzen Sie dann `${{ secret.API_KEY }}` in Tasks:

```yaml
tasks:
  deploy:
    run: "curl -X POST -H 'Authorization: Bearer ${{ secret.API_KEY }}' https://api.example.com/deploy"
```

autotask lädt Secrets automatisch aus lokalen Dateien oder Vault-Backends (z. B. HashiCorp Vault) und maskiert sie im Log-Output.

## Best Practices für den produktiven Einsatz

### 1. **Idempotenz durch State-Checks**
Definieren Sie immer einen `check`-Block pro Task, der prüft, ob die Aufgabe bereits ausgeführt wurde. Beispiel:

```yaml
tasks:
  migrate:
    run: "node migrate.js"
    check: "test -f .migrated-$(date +%Y%m%d)"
```

So vermeiden Sie doppelte Ausführungen – essenziell für Cron-Jobs oder wiederholte Deployments.

### 2. **Task-Templates für Multi-Cloud-Umgebungen**
Nutzen Sie YAML-Anchor-Techniken, um wiederverwendbare Task-Blöcke für AWS, GCP und Azure zu definieren:

```yaml
x-cloud: &cloud
  env:
    CLOUD_PROVIDER: "aws"
    REGION: "eu-central-1"

deploy-aws:
  <<: *cloud
  run: "aws ecs update-service ..."

deploy-gcp:
  <<: *cloud
  env:
    CLOUD_PROVIDER: "gcp"
  run: "gcloud run deploy ..."
```

Das reduziert Redundanz und erhöht die Wartbarkeit Ihrer Pipeline-Definitionen.
