# autotask - AUT_003

## Übersicht

Mit autotask eliminieren wir manuelle Routineprozesse in CI/CD-Pipelines und DevOps-Workflows. Kein weiteres „vergessenes Skript“, kein manuelles Nachtragen von Tasks – das Tool orchestriert Aufgaben automatisch basierend auf Ereignissen, Zeitplänen oder API-Triggers. Ideal für Teams, die ihre Infrastruktur als Code leben und jede Aufgabe nachvollziehbar versionieren wollen.

## Detaillierter Inhalt

### Was ist autotask?

autotask ist eine deklarative Task-Engine, die in jede bestehende Automatisierungsumgebung integriert wird – ob als Sidecar in Kubernetes, als CLI-Tool für lokale Entwicklung oder als zentraler Scheduler in der Cloud. Statt sich durch Dutzende Cronjobs, Hooks und manuelle Scripts zu kämpfen, definierst du Tasks in YAML, koppelst sie an Events und lässt autotask den Rest erledigen.  
Die Engine unterstützt idempotente Ausführung, parallele Dependencies und automatische Retry-Logik – genau das, was in produktiven Systemen zählt.

### Kernfunktionen

- **Event-gesteuerte Task-Orchestrierung mit DAG-Unterstützung**  
  Tasks werden nicht linear abgearbeitet, sondern als gerichteter azyklischer Graph (DAG) definiert. Das System erkennt Abhängigkeiten automatisch und führt Tasks parallel aus, sobald alle Vorbedingungen erfüllt sind. Perfekt für komplexe Deployment-Pipelines oder Datenverarbeitungs-Workflows.

- **Integrierte Observability über OpenTelemetry und strukturierte Logs**  
  Jeder Task liefert standardisierte Metriken und Traces – ohne zusätzlichen Agenten. Du siehst auf einen Blick, wo ein Task hängt, wie lange er läuft und welche Ressourcen er verbraucht. Das erleichtert das Debugging und die Kapazitätsplanung erheblich.

- **Multi-Provider-Trigger-Engine**  
  autotask kann auf Git-Webhooks, Kafka-Nachrichten, S3-Event-Benachrichtigungen, SQS-Queues oder einfache HTTP-Callbacks reagieren. Du definierst einen Trigger einmal und koppelst ihn an beliebig viele Tasks – das reduziert die Anzahl der Cronjobs und Webhook-Endpoints drastisch.

### So verwenden Sie es

1. **Task-Definition in YAML mit idempotenten Handlern**  
   Erstelle eine Datei `tasks/backup.yaml`:
   ```yaml
   task: database-backup
   schedule: "0 2 * * *"
   handler:
     type: shell
     command: pg_dump -U admin -d production | gzip > /backups/$(date +%Y%m%d).sql.gz
   retry:
     max_attempts: 3
     backoff: exponential
   ```
   Danach registrierst du den Task mit `autotask register tasks/backup.yaml`. Der Task wird sofort in den Scheduler aufgenommen und läuft nach dem definierten Zeitplan – inklusive automatischer Wiederholung bei Fehlern.

2. **Event-Trigger mit Filterlogik konfigurieren**  
   Lege eine Datei `triggers/ci-pipeline.yaml` an:
   ```yaml
   trigger:
     source: github
     event: push
     filter:
       branch: main
       paths: ["src/**", "Dockerfile"]
   tasks:
     - build-image
     - run-tests
     - deploy-staging
   ```
   Mit `autotask trigger activate triggers/ci-pipeline.yaml` wird der Trigger aktiv. Jeder Push auf `main` löst dann automatisch die drei Tasks aus – in der richtigen Reihenfolge und parallel, wo möglich.

3. **Task-Status über CLI und API abfragen**  
   Nutze `autotask status --task database-backup --output json` für eine detaillierte Ausgabe mit Laufzeit, Exit-Code und Log-Auszügen. Oder integriere die REST-API in dein Monitoring-Dashboard – ein `GET /api/v1/tasks/database-backup/history` liefert die letzten 100 Ausführungen mit Metadaten.

### Beste Praktiken

- **Tasks als Code versionieren und in CI/CD integrieren**  
  Lege alle Task-Definitionen und Trigger-Konfigurationen in einem eigenen Repository ab. Jede Änderung durchläuft einen Review-Prozess und wird automatisch in der Staging-Umgebung getestet, bevor sie in Produktion deployed wird. So vermeidest du „tote“ Tasks, die niemand mehr pflegt.

- **Observability von Anfang an mitdenken**  
  Konfiguriere für jeden Task mindestens einen Alert bei Fehlschlag und einen Log-Export nach z. B. Loki oder Elasticsearch. Nutze die eingebauten Metriken (Dauer, Fehlerrate, Retry-Count) für Dashboards in Grafana. Das verhindert, dass ein Task stumm scheitert und erst Tage später auffällt.
