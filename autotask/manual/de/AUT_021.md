# autotask - AUT_021

## Überblick

autotask ist ein hochperformantes CLI-Tool und eine Rust-basierte Automatisierungsengine, die repetitive DevOps-Workflows, Cron-Jobs und CI/CD-Pipelines durch deklarative YAML-Konfigurationen ersetzt. Es integriert native Task-Graphen, parallele Ausführung und kontextbewusste Fehlerbehandlung – ohne externe Abhängigkeiten.

## Hardcore-Funktionen

### 1. **DAG-basierte Task-Orchestrierung mit dynamischer Abhängigkeitsauflösung**
autotask parst YAML-Definitionen in einen gerichteten azyklischen Graphen (DAG). Tasks werden parallel ausgeführt, sobald ihre Vorgänger erfolgreich abgeschlossen sind. Fehlgeschlagene Tasks lösen automatisch Rollbacks oder alternative Pfade aus – definierbar über `on_failure: [retry, abort, fallback]`.

### 2. **Native Shell- und Docker-Integration mit Kontext-Caching**
Jeder Task kann direkt Shell-Befehle, Docker-Container oder Kubernetes-Jobs ausführen. autotask cached Ausgabe- und Statuskontexte in einem lokalen KV-Store (SQLite). Nachfolgende Tasks greifen über `{{ .task_name.output }}` darauf zu – ideal für Pipeline-übergreifende Datenflüsse.

### 3. **Self-Healing Scheduler mit Timeout und Graceful Shutdown**
Der integrierte Scheduler überwacht Task-Laufzeiten und terminiert hängende Prozesse via `SIGTERM` nach konfigurierbarem Timeout. Bei Abbruch durch `Ctrl+C` oder System-Exit führt autotask einen Graceful Shutdown durch – inklusive persistenter Status-Snapshots für Wiederaufnahme.

## Detaillierte Anleitung zur Nutzung

### 1. **Task-Graph definieren und ausführen**
Erstelle eine Datei `tasks.yaml` mit folgendem Inhalt:
```yaml
tasks:
  - name: build
    command: "docker build -t myapp ."
    timeout: 120s
  - name: test
    depends_on: [build]
    command: "docker run myapp pytest"
  - name: deploy
    depends_on: [test]
    command: "kubectl apply -f deploy.yaml"
```
Führe den Graphen aus: `autotask run tasks.yaml`. Der Scheduler startet `build`, nach Erfolg parallel `test`, und nach Erfolg `deploy`. Bei Fehlschlag von `test` wird `deploy` automatisch übersprungen.

### 2. **Kontextweitergabe zwischen Tasks nutzen**
Erweitere `tasks.yaml` um Kontextübergabe:
```yaml
tasks:
  - name: version
    command: "echo v1.2.3"
  - name: build
    depends_on: [version]
    command: "docker build -t myapp:{{ .version.output }} ."
```
autotask ersetzt `{{ .version.output }}` dynamisch durch den Output des Tasks `version`. So entstehen flexible, datengetriebene Pipelines ohne manuelle Variablen.

### 3. **Fehlerbehandlung und Wiederaufnahme konfigurieren**
Füge Recovery-Logik hinzu:
```yaml
tasks:
  - name: risky_deploy
    command: "helm upgrade myapp ./chart"
    on_failure: retry
    retry_count: 3
    retry_delay: 5s
  - name: notify
    depends_on: [risky_deploy]
    command: "curl -X POST https://hooks.slack.com/... -d 'Deploy done'"
```
Bei Fehlschlag von `risky_deploy` wiederholt autotask den Task bis zu 3 Mal mit 5s Pause. Erst nach endgültigem Fehlschlag wird `notify` übersprungen. Der Status wird in `~/.autotask/state.db` persistiert – bei Neustart wird dort weitergemacht, wo es abbrach.

## Best Practices für den produktiven Einsatz

### 1. **Immer Timeouts setzen und Ressourcenlimits definieren**
Ohne explizites `timeout` kann ein hängender Task die gesamte Pipeline blockieren. Setze für jeden Task ein realistisches Timeout (z. B. `300s` für Builds). Bei Docker-Tasks zusätzlich `memory_limit` und `cpu_limit` definieren, um Ressourcenkonflikte im Cluster zu vermeiden.

### 2. **Status-Snapshots für Disaster Recovery nutzen**
Aktiviere die persistente State-Datei mit `--state-file /pfad/zu/state.db`. Nach einem System-Crash oder manuellen Abbruch startest du die Pipeline einfach neu: `autotask run tasks.yaml --resume`. autotask setzt automatisch an der letzten fehlgeschlagenen oder nicht ausgeführten Task-Position fort – ohne Datenverlust oder manuelle Reparatur.
