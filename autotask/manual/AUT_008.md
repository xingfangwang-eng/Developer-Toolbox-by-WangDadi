# autotask - AUT_008

## Overview

autotask is a production-grade task automation framework engineered for developers who need to eliminate repetitive workflows without sacrificing control. It combines declarative YAML pipelines with runtime hooks, enabling seamless integration into CI/CD chains, DevOps stacks, and local development environments. This document is fully optimized for English-speaking developers and SEO performance.

## What is autotask?

autotask is a lightweight, dependency-free task runner that parses automation manifests into executable DAGs (Directed Acyclic Graphs). It supports parallel execution, conditional branching, and real-time state persistence — designed to replace brittle shell scripts and bloated orchestration tools.

## Core Features

- **Dynamic Task Graph with Conditional Rollbacks**: autotask builds an execution graph at runtime. If any node fails, the engine can trigger a user-defined rollback chain — not just a simple exit code. This makes it ideal for multi-step deployments or data pipelines where partial failures must be handled gracefully.

- **Context-Aware Variable Injection**: Each task can inherit and override variables from parent scopes, environment files, or secret backends (e.g., HashiCorp Vault). Variables are interpolated lazily, meaning you can define dynamic values like `$(date +%s)` or `${ secrets.API_KEY }` without pre-processing.

- **Idempotent Execution with State Snapshots**: autotask caches task outputs and checksums. On re-run, it skips unchanged tasks unless explicitly forced. This drastically reduces execution time in iterative development or CI retries, and snapshots can be exported for audit trails.

## How to Use

### 1. Define a Task Manifest

Create a `taskfile.yaml` in your project root. Each task is a node with a command, dependencies, and optional hooks.

```yaml
version: "2.0"
tasks:
  build:
    command: "npm run build"
    depends_on: ["install"]
    env:
      NODE_ENV: "production"
    on_failure:
      - command: "echo 'Build failed, cleaning up...'"
      - command: "rm -rf ./dist"
```

Run it with: `autotask run build`

### 2. Chain Tasks with Parallel Execution

Use the `parallel` directive to run independent tasks concurrently. autotask resolves the DAG and spawns each group in a separate OS thread.

```yaml
tasks:
  lint:
    command: "eslint src/"
  test:
    command: "jest --ci"
  deploy:
    command: "deploy.sh"
    depends_on: ["lint", "test"]
    parallel: true
```

Execute: `autotask run deploy` — lint and test run simultaneously, then deploy triggers only after both succeed.

### 3. Inject Secrets and Environment Overrides

Define a `.env.local` file and reference it in your manifest. autotask merges environment variables at runtime, with explicit `env` blocks taking precedence.

```bash
# .env.local
DB_HOST=localhost
DB_PASS=supersecret
```

```yaml
tasks:
  migrate:
    command: "npx prisma migrate deploy"
    env_file: ".env.local"
    env:
      DB_HOST: "prod-db.internal"
```

Run: `autotask run migrate --env-file .env.prod` — this overrides only the specified file, keeping other variables intact.

## Best Practices

- **Use Task Hooks for Observability**: Attach `on_start` and `on_complete` hooks to log timestamps, emit metrics, or trigger webhooks. This turns autotask into a lightweight observability layer without adding a monitoring stack. For example, `on_complete: ["curl -X POST https://your-monitor.com/event -d 'status=ok'"]`.

- **Version Your Taskfiles with Semantic Tags**: Treat `taskfile.yaml` as code. Tag releases with semantic versions and pin autotask to a specific schema version (e.g., `version: "2.0"`). This prevents breaking changes when upgrading the tool and ensures reproducibility across environments.
