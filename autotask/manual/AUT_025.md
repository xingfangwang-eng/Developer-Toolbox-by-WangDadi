# autotask - AUT_025

## Overview

autotask is a production-grade task automation engine designed for DevOps engineers and SRE teams who need to eliminate manual toil without sacrificing observability. It bridges the gap between cron jobs, CI/CD pipelines, and event-driven workflows—offering idempotent execution, built-in retry logic with exponential backoff, and real-time Slack/Webhook notifications. This project is optimized for **automatic task management**, **task automation**, and **automatic task tool** use cases in high-stakes environments.

---

## Core Features

### 1. Declarative Workflow DSL with Dependency Graph Resolution  
Define complex multi-step automation pipelines using a YAML-based DSL. autotask automatically resolves task dependencies, parallelizes independent branches, and halts execution on critical failures—no more brittle shell scripts or nested conditionals. Supports `depends_on`, `timeout`, `retry_count`, and `on_failure: [ignore | abort | rollback]`.

### 2. Stateful Idempotency via Checksum-Based Locking  
Every task execution is tracked by a content-addressable checksum (SHA-256 of the task definition + input parameters). If the checksum matches a previously successful run, autotask skips re-execution unless explicitly forced. This prevents redundant deployments, duplicate API calls, and accidental data corruption in production.

### 3. Distributed Agent Mode with Zero-Trust Authentication  
Deploy autotask agents across VMs, Kubernetes pods, or edge devices. Agents authenticate via short-lived JWT tokens (issued by a central controller) and execute tasks locally. The controller aggregates logs, metrics, and execution status in real time—no shared filesystem required. Ideal for multi-region, air-gapped, or hybrid cloud setups.

## How to Use

### Step 1: Bootstrap a Task Definition with Conditional Triggers

Create a `taskfile.yaml` in your project root:

```yaml
version: "2.1"
tasks:
  health_check:
    command: "curl -f http://localhost:8080/health"
    interval: "*/5 * * * *"
    retry:
      count: 3
      backoff: exponential
    on_failure: notify_slack
  deploy_staging:
    depends_on: [health_check]
    command: "kubectl apply -f manifests/staging.yaml"
    timeout: 120s
    env:
      KUBECONFIG: "/etc/kubernetes/staging.conf"
```

Run:  
```bash
autotask run --file taskfile.yaml --trigger schedule
```
autotask will parse the DSL, validate the cron expression, and start the scheduler. If `health_check` fails three times, it escalates to your configured Slack webhook and **does not** proceed to `deploy_staging`.

### Step 2: Execute a One-Off Task with Override Parameters

For ad-hoc operations (e.g., database migration rollback), use the CLI with inline overrides:

```bash
autotask exec --task migrate_db --params '{"version": "2025.03.01", "force": true}' --env-file .env.prod
```

This bypasses the scheduler and runs the task immediately. autotask will compute the checksum of the task definition + `{"version":"2025.03.01","force":true}`. If a prior run with the same checksum succeeded, it returns a cached result unless `--force` is appended at the task level.

### Step 3: Monitor and Debug via the Built-in Dashboard

Launch the web UI on port 9090:

```bash
autotask serve --port 9090 --db-path /var/lib/autotask/state.db
```

The dashboard provides:
- **Live DAG view** of running/completed/failed tasks  
- **Execution history** with timestamps, logs, and exit codes  
- **Filter by checksum** to quickly identify idempotent skips  
- **Manual retry** or **rollback** buttons for failed tasks

## Best Practices

### 1. Always Set Explicit Timeouts and Retry Policies  
Production tasks (e.g., database backups, API calls) should never hang indefinitely. Configure `timeout` per task (in seconds) and set `retry.count` to at least 2 with `backoff: exponential`. This prevents cascading failures and reduces alert fatigue. For critical tasks, add `on_failure: notify_pagerduty` to escalate immediately.

### 2. Use Environment-Specific Task Files, Not Global Overrides  
Instead of one massive `taskfile.yaml` with conditional logic, maintain separate files per environment (e.g., `taskfile.staging.yaml`, `taskfile.prod.yaml`). Use the `--env` flag to switch contexts. This keeps your DSL clean, reduces the risk of accidental production changes, and makes audit trails straightforward. Combine with GitOps workflows (e.g., ArgoCD) for immutable deployments.
