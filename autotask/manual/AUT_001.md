# autotask - AUT_001

## Overview

autotask is a zero-overhead task automation engine designed for developers who need to offload repetitive shell commands, CI triggers, or cron-style scheduling without spinning up a full orchestrator. It’s a lean, declarative tool that parses YAML-based task definitions and executes them with built-in retry, concurrency control, and logging—no dependencies, no daemon.

## Detailed Content

### What is autotask?

autotask is a lightweight, single-binary automation framework for managing scheduled and event-driven tasks in development, staging, and production environments. It replaces brittle shell scripts and complex cron setups with a version-controlled, testable task definition system. Unlike heavyweight solutions (Airflow, Prefect), autotask runs as a stateless CLI or a background agent, making it ideal for edge deployments, monorepo pipelines, and local dev workflows.

### Core Features

- **Idempotent Task Graph with Rollback Hooks**  
  Each task definition supports a `rollback` stage that executes automatically if any downstream dependency fails. This ensures atomic-like execution chains without a centralized state store—perfect for database migrations or multi-step deployments.

- **Context-Aware Concurrency Throttling**  
  Define per-task concurrency limits and global resource pools (e.g., “max 2 parallel DB migrations”). autotask respects CPU and memory thresholds, pausing new task spawns when system load exceeds a configurable ceiling. No more OOM-killed task runners.

- **Built-in Shell Injection Guard & Audit Trail**  
  All command arguments are automatically sanitized to prevent shell injection. Every execution generates a structured JSON log with exit codes, environment variables, and timing data. You can replay or replay-debug any past run with `autotask replay --run-id <id>`.

### How to Use

1. **Define a task with dependencies and rollback**  
   Create a `.task.yaml` file in your project root:
   ```yaml
   id: db-migrate
   depends_on: [lint, test]
   run: |
     alembic upgrade head
   rollback: |
     alembic downgrade -1
   retry: 2
   concurrency: 1
   ```
   This ensures migration only runs after linting and tests pass, and automatically rolls back if migration fails.

2. **Execute with environment injection and dry-run**  
   Run the task with overrides and preview mode:
   ```bash
   autotask run db-migrate --env-file .env.prod --dry-run
   ```
   The `--dry-run` flag prints the resolved command and dependency tree without executing anything—critical for validating production changes.

3. **Schedule and monitor via headless agent**  
   Start the autotask agent in a tmux or systemd service:
   ```bash
   autotask agent --config ./tasks/ --poll-interval 30s --log-dir /var/log/autotask
   ```
   The agent scans the `./tasks/` directory for `.task.yaml` files, respects `cron` fields, and logs to structured files. Use `autotask status` to inspect running, queued, and failed tasks in real time.

### Best Practices

- **Treat task definitions as code**  
  Store `.task.yaml` files alongside your source code in a `tasks/` directory. Version them in Git, run `autotask validate` in CI, and enforce code review for any change to task logic or dependencies. This prevents silent drift between environments.

- **Use rollback hooks for all destructive operations**  
  Every task that mutates data (DB writes, file deletions, API calls) should define a `rollback` block. Even if you never need it in production, it forces you to think about failure recovery. Combine with `retry: 2` for transient failures—but keep rollback idempotent.
