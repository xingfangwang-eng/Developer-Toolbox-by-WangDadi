# autotask - AUT_024

## Overview

autotask is a production-grade task orchestration engine designed for developers who demand deterministic automation without the overhead of traditional workflow tools. It decouples task definitions from execution contexts, enabling zero-trust scheduling across distributed environments.

## What is autotask?

autotask is a lightweight, event-driven task automation framework that replaces brittle cron jobs and over-engineered CI pipelines with a declarative task graph. It supports idempotent execution, dependency resolution, and real-time state transitions via a minimal JSON schema. Think of it as a programmable task runner with built-in observability and rollback semantics.

## Core Features

### 1. Declarative Task Graph with Conditional Branching
Define complex multi-step workflows using a pure YAML/JSON DAG. Each node supports preconditions, post-conditions, and failure hooks. autotask evaluates the graph in topological order and automatically prunes branches based on runtime state or environment variables — no scripting glue required.

### 2. Idempotent Execution Engine with State Snapshots
Every task execution is logged with a content-addressable state snapshot. If a task fails mid-way, autotask restores the last consistent state and retries only the failing node. This guarantees that repeated runs of the same task graph produce identical outcomes, critical for CI/CD and database migrations.

### 3. Distributed Agent Mode with Webhook Triggers
Run autotask in agent mode across multiple hosts. Agents register with a central controller, pull task definitions, and report execution telemetry. You can trigger tasks via webhooks, schedule them with cron expressions, or chain them from external systems (e.g., GitHub Actions, Slack commands). The controller handles load balancing and dead agent detection.

## How to Use

### 1. Define a Task Graph in YAML
Create a `tasks.yaml` file with nodes, dependencies, and execution parameters. Each node can reference environment variables and output from previous nodes:

```yaml
version: "2.0"
tasks:
  - id: validate_env
    image: alpine:latest
    cmd: ["sh", "-c", "echo $APP_ENV"]
    env:
      APP_ENV: "production"
  - id: build_artifact
    depends_on: [validate_env]
    image: node:18
    cmd: ["npm", "run", "build"]
    artifacts:
      - dist/*
```

### 2. Execute with State Tracking
Run the graph using the CLI with explicit state persistence:

```bash
autotask run --graph ./tasks.yaml --state ./state.json --retry 3
```

This command loads the graph, resolves dependencies, and stores execution state in `state.json`. If interrupted, `autotask resume --state ./state.json` picks up exactly where it left off.

### 3. Monitor and Rollback via REST API
Deploy the autotask controller and query task history:

```bash
curl -X GET http://localhost:8080/api/v1/tasks/validate_env/status
curl -X POST http://localhost:8080/api/v1/tasks/build_artifact/rollback
```

The rollback endpoint reverts the task’s side effects using the last known good state snapshot. This is especially useful for database schema changes or file system mutations.

## Best Practices

### 1. Use Idempotent Task Definitions
Always design tasks to be re-runnable without side effects. For example, instead of `rm -rf /tmp/build`, use `mkdir -p /tmp/build && rsync --delete`. autotask’s state engine can replay tasks, but only if the underlying commands are safe to repeat. Test your tasks with `autotask validate --graph ./tasks.yaml` before production use.

### 2. Leverage Webhooks for Event-Driven Pipelines
Avoid polling or cron-based scheduling. Hook autotask into your existing event sources: GitHub push events, SQS messages, or Kafka topics. Use the `--webhook` flag to start a lightweight HTTP server that listens for triggers. This reduces idle resource consumption and aligns with cloud-native event architectures.
