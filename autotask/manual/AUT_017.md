# autotask - AUT_017

## Overview

autotask is a lightweight yet production-grade automation engine designed for developers who need to offload repetitive CLI workflows, cron jobs, and multi-step orchestration without bloat. It’s optimized for low-latency task scheduling and deterministic execution across distributed runners.

## Detailed Content

### What is autotask?

autotask is a declarative task runner that transforms YAML-defined pipelines into executable workflows. It supports dependency resolution, retry logic with exponential backoff, and real-time state propagation. Built for CI/CD, DevOps, and data pipeline engineers who want to avoid vendor lock-in and run automation natively.

### Core Features

- **Deterministic Dependency Graph Execution**  
  autotask parses task dependencies into a Directed Acyclic Graph (DAG) and executes nodes in topological order. No more race conditions or manual sequencing — the engine guarantees that each task runs only after its upstream dependencies succeed, with automatic rollback on failure.

- **Idempotent Task Caching with Content Hashing**  
  Each task is cached based on a SHA-256 hash of its input parameters and environment. If nothing changes, the task is skipped — not re-run. This reduces pipeline runtime by up to 70% in iterative development workflows.

- **Distributed Runner with Native Webhook Triggers**  
  Deploy autotask agents across multiple nodes using a lightweight gRPC protocol. Tasks can be triggered via HTTP webhooks, file system watchers, or time-based cron expressions. The engine automatically handles task queueing, load balancing, and failure recovery.

### How to Use

1. **Define a task pipeline in YAML**  
   Create a file named `autotask.yml` in your project root. Declare tasks with `name`, `command`, `dependencies`, and optional `retry` and `timeout` fields. Use `${{ }}` syntax for variable interpolation from environment or secrets.

   ```yaml
   tasks:
     - name: build
       command: "npm run build"
       timeout: 120
     - name: test
       depends_on: [build]
       command: "npm test"
       retry: 3
   ```

2. **Run the pipeline with granular control**  
   Execute `autotask run` to trigger the entire DAG. Use `--task test` to run a single task and its dependencies, or `--dry-run` to preview the execution plan without side effects. The CLI outputs a structured JSON log for downstream processing.

3. **Monitor and replay failed tasks**  
   After execution, run `autotask status` to view task states (pending, running, failed, cached). Use `autotask replay --task test` to re-run only the failed tasks with the same inputs. The replay preserves the original execution context, including environment variables and timestamps.

### Best Practices

- **Use content-based caching to accelerate iterative development**  
  Always define task dependencies explicitly and avoid using wildcards in file globs. This ensures autotask’s caching engine can accurately detect changes. For CI/CD pipelines, enable `cache: true` on tasks that perform linting, compilation, or testing — this reduces feedback loop time significantly.

- **Design tasks as idempotent units with clear failure boundaries**  
  Each task should be self-contained and stateless where possible. If a task writes to a shared resource (e.g., a database or S3 bucket), include a deterministic identifier (like a commit hash) in the output path. This prevents state corruption during retries and simplifies rollback logic.
