# autotask - AUT_006

## Overview

autotask is a **zero-overhead task automation framework** engineered for developers who demand deterministic execution, declarative workflows, and deep system integration. It eliminates boilerplate orchestration logic by treating tasks as first-class immutable objects with built-in dependency resolution, retry policies, and audit trails. Designed for CI/CD pipelines, serverless backends, and edge computing environments, autotask compresses complex multi-step processes into a single, version-controlled YAML or JSON manifest.

## Hardcore Features

### 1. **Declarative Dependency Graph with Parallel Execution**
Define tasks as nodes in a directed acyclic graph (DAG). autotask automatically resolves inter-task dependencies, executes independent branches concurrently, and halts on failure with granular error propagation. No more hand-rolling topological sorts or race-condition-prone scripts.

### 2. **Idempotent State Locks & Atomic Rollbacks**
Every task execution is wrapped in a transactional state lock. If a task fails mid-execution, autotask triggers an atomic rollback—reverting files, environment variables, and API side effects to the last known good state. This guarantees that partial failures never leave your system in an inconsistent state.

### 3. **Pluggable Runtime Adapters (Docker, Lambda, SSH, Webhook)**
Run tasks anywhere: inside a Docker container, as an AWS Lambda invocation, over SSH on a remote bare-metal server, or via an HTTP webhook callback. Each adapter handles environment injection, secret mounting, and output collection transparently. Switch runtimes without rewriting a single line of task logic.

## How to Use

### 1. **Declare a Task Graph in `autotask.yaml`**
```yaml
tasks:
  - id: fetch_data
    runtime: docker
    image: python:3.11-slim
    command: ["python", "fetch.py"]
    env:
      API_KEY: "${SECRETS.API_KEY}"
  - id: transform
    depends_on: [fetch_data]
    runtime: lambda
    function_arn: arn:aws:lambda:us-east-1:123456789012:function:transform
  - id: load
    depends_on: [transform]
    runtime: ssh
    host: db-server.internal
    user: deploy
    command: ["/usr/local/bin/load.sh", "--input", "/tmp/transformed.json"]
    retry: 3
    retry_delay: 10
```
Run with: `autotask run --manifest autotask.yaml`

### 2. **Inspect Execution History & Replay**
```bash
autotask history --task-id fetch_data --limit 10
autotask replay --task-id fetch_data --revision 3
```
Each run produces a signed execution log with timestamps, exit codes, and output hashes. Replaying a specific revision re-executes exactly the same command in the same environment, enabling deterministic debugging.

### 3. **Integrate with CI/CD via Webhook Trigger**
```bash
autotask webhook --listen :8080 --manifest autotask.yaml
```
Post a JSON payload to `http://localhost:8080/trigger` to kick off the entire graph. autotask returns a `run_id` immediately, and you can poll `GET /status/{run_id}` for real-time progress. Perfect for GitHub Actions, GitLab CI, or any event-driven pipeline.

## Best Practices

- **Use `depends_on` to model fan-out/fan-in patterns.** For example, a data pipeline might have three independent extraction tasks (fan-out) that all feed into a single aggregation task (fan-in). autotask handles the concurrency and synchronization automatically, so you never need to write a `Promise.all` or a `wait-for` step.
- **Always set `retry` and `retry_delay` for network-bound tasks.** Transient failures in API calls, database connections, or file transfers are inevitable. A sensible default is `retry: 3` with exponential backoff (e.g., `retry_delay: 5` seconds). autotask’s retry logic respects the task’s idempotency contract, so repeated attempts won’t corrupt state.
