# autotask - AUT_016

## 概要

autotask は、反復的なタスクを自動化し、開発ワークフローを最適化するために設計された、軽量かつ高パフォーマンスなタスク管理ツールです。キーワード：自動タスクツール、タスク自動化、自動タスク管理。

## 詳細

### autotask とは？

autotask は、CI/CD パイプライン、バッチ処理、定期的なメンテナンス作業など、開発現場で頻繁に発生するタスクを自動化するためのプロフェッショナルツールです。YAML ベースの設定ファイルでタスクを定義し、依存関係や並列実行を柔軟に制御できます。特に、複数のプロジェクトや環境を跨いだタスク管理に強みを持ち、チーム全体の生産性を向上させます。

### コア機能

- **コンテキスト依存の動的タスクスケジューリング**  
  タスクの実行条件を、ファイル変更、API 応答、環境変数、時刻など複数のトリガーから動的に判定。例えば、「特定のブランチにプッシュされた場合のみ、かつ前回のビルドが成功している場合にのみデプロイする」といった複合条件を YAML 一行で記述可能。

- **並列実行エンジンとリソース制御**  
  タスク間の依存関係を自動解析し、独立したタスクは最大同時実行数（例：`parallelism: 4`）の範囲内で並列実行。メモリや CPU 使用率を監視し、リソースが逼迫した場合は自動でキューイングするインテリジェントなスロットリング機能を搭載。

- **エラーハンドリングと自動リトライ戦略**  
  タスク失敗時に、指数バックオフ（例：`retry: 3, backoff: exponential`）や特定の終了コードのみリトライする条件付きリトライを設定可能。さらに、失敗したタスクのログを Slack やメールに自動通知し、後続タスクの実行をブロックする「フェイルファスト」モードも標準サポート。

### 使い方

1. **タスク定義ファイルの作成と構造化**  
   プロジェクトルートに `autotask.yml` を作成し、以下のようにタスクを定義します。各タスクには `name`、`command`、`depends_on`（依存タスク）、`trigger`（実行条件）を指定。例えば、ビルド→テスト→デプロイのパイプラインを、依存関係を明示しながら記述します。  
   ```yaml
   tasks:
     - name: build
       command: "npm run build"
       trigger:
         event: push
         branch: main
     - name: test
       command: "npm test"
       depends_on: [build]
     - name: deploy
       command: "deploy.sh"
       depends_on: [test]
       retry: 2
   ```

2. **タスクの実行とモニタリング**  
   CLI から `autotask run` を実行すると、autotask は定義ファイルを解析し、依存関係グラフを構築。タスクの実行状況はリアルタイムでターミナルに表示され、各タスクの開始・終了時刻、終了コード、経過時間が確認できます。`autotask status` で現在のキュー状況と実行履歴を照会可能。

3. **高度な条件分岐と動的パラメータの注入**  
   タスク内で `{{ env.VAR_NAME }}` や `{{ secrets.API_KEY }}` のように環境変数やシークレットを参照可能。さらに、`when` 句を使って「前のタスクが成功した場合のみ」「特定のファイルが変更された場合のみ」といった条件分岐を記述。例えば、デプロイタスクを「テストが成功し、かつ `CHANGELOG.md` が更新された場合のみ」実行するよう設定できます。  
   ```yaml
   - name: deploy
     command: "deploy.sh {{ env.DEPLOY_TARGET }}"
     depends_on: [test]
     when:
       condition: "{{ tasks.test.exit_code == 0 }}"
       file_changed: "CHANGELOG.md"
   ```

### ベストプラクティス

- **タスクの粒度を適切に保つ**  
  1 つのタスクに複数の処理を詰め込まず、原則「1 タスク 1 責務」に分割してください。例えば、「ビルドとテストを 1 タスクにまとめる」のではなく、`build` と `test` を別タスクにし、依存関係で結合します。これにより、エラー発生時の切り分けが容易になり、並列実行の恩恵も最大化できます。

- **シークレット管理は外部サービスに委譲する**  
  タスク定義ファイルに直接 API キーやパスワードを記述するのは避け、autotask の `secrets` 機能（例：Vault や AWS Secrets Manager との統合）を利用してください。YAML ファイル内では `{{ secrets.DB_PASSWORD }}` のように参照し、実際の値は実行時に安全に注入します。これにより、ソースコード管理上のセキュリティリスクを大幅に低減できます。
