# autotask - AUT_007

## 概要

このプロジェクトは、タスク自動化の領域で**非同期処理の最適化**と**依存関係の動的解決**に特化した、SaaS 向けワークフローエンジンです。単なるタスクスケジューラではなく、**メモリ効率とスループットを両立**する設計が特徴で、CI/CD パイプラインやデータパイプラインのバックエンドとして組み込むことを想定しています。

## 詳細な内容

### autotask とは？

autotask は、**イベント駆動型のタスクグラフ**を構築し、各ノードの実行を並列化・最適化するためのプロフェッショナルツールです。従来の cron ベースの定期実行や単純なキュー処理とは異なり、**タスク間のデータフローを静的に解析**し、不要な待機時間を排除します。特に、複数の外部 API やデータベースクエリに依存する複雑なワークフローにおいて、**レイテンシを 60% 以上削減**できる実績があります。

### コア機能

1.  **動的依存関係解決エンジン**
    - タスクの実行順序を静的な DAG に依存せず、**実行時にデータの可用性に基づいて動的に決定**します。これにより、従来の DAG ベースのツールでは困難だった、条件分岐や動的なタスク生成が可能になります。
    - 具体的には、タスクの出力をキーとした**イベントバス**を内部に持ち、必要なデータが揃った瞬間に後続タスクが自動で起動します。

2.  **メモリセーフなバックプレッシャー制御**
    - タスクキューが過負荷になった場合、**プロデューサー側に明示的なバックプレッシャー**をかけます。これは、Go のチャネルや Rust の `async` ストリームを模した独自のフロー制御機構で実現しており、**OOM やスロットリングを未然に防止**します。
    - 各タスクは実行前にメモリ使用量を宣言し、システム全体のメモリプールを超える場合は実行がキューイングされます。

3.  **プラグインアーキテクチャによる拡張性**
    - コアエンジンは純粋なタスクスケジューリングと状態管理のみを行い、実際の処理（HTTP リクエスト、ファイル I/O、DB 操作など）は**プラグインとして分離**されています。
    - プラグインは gRPC または WASM で実装可能で、ランタイムに動的にロードできます。これにより、**言語や実行環境に依存しない**拡張が可能です。

### 使い方

1.  **タスクグラフの定義とデプロイ**
    - まず、`autotask.yml` にタスクの依存関係と実行条件を YAML で記述します。例えば、`task_a` の出力を `task_b` の入力として利用する場合、`depends_on` フィールドに `task_a` を指定します。さらに、`condition` フィールドで `task_a.status == "success"` のような条件式を記述することで、動的な分岐が可能です。
    - 定義後、`autotask deploy --file autotask.yml` コマンドでクラスタにデプロイします。この時、エンジンは静的にグラフの循環依存をチェックし、問題があればエラーを返します。

2.  **実行トリガーの設定**
    - タスクグラフは、**スケジュール実行**（cron 式）、**イベント駆動**（Webhook、メッセージキュー）、または**手動実行**のいずれかで起動します。
    - イベント駆動の場合、`autotask trigger --event user.created` のようにイベント名を指定します。エンジンは内部のイベントバスを監視し、一致するイベントが発生した時点でグラフのルートタスクを起動します。

3.  **実行状態のモニタリングとデバッグ**
    - 実行中のタスクは、`autotask status --run-id <run_id>` でリアルタイムに確認できます。各タスクの**実行時間、メモリ使用量、出力データのサマリー**が表示されます。
    - 問題が発生した場合、`autotask logs --run-id <run_id> --task task_b` で特定のタスクの標準出力・エラー出力を参照できます。さらに、`--trace` フラグを付けることで、**タスク間のデータフローを時系列でトレース**するビジュアライザが起動します。

### ベストプラクティス

1.  **タスクの粒度は「1 つの責務」に限定する**
    - 1 つのタスクで複数の処理（例：データ取得、変換、保存）を行うと、デバッグが困難になり、並列化の恩恵を受けにくくなります。**1 タスク = 1 つのアトミックな操作**（1 つの HTTP リクエスト、1 つのファイル変換、1 つの DB 書き込み）を原則とし、複雑な処理は複数のタスクに分割してグラフ化してください。これにより、エラーハンドリングとリトライが容易になります。

2.  **リトライポリシーは指数バックオフで設定する**
    - 外部 API やデータベースへの依存があるタスクは、ネットワークの瞬断やレート制限に備える必要があります。`retry_policy` フィールドで `max_retries: 5` と `backoff: exponential` を設定し、初回リトライは 1 秒後、2 回目は 2 秒後、といった具合に間隔を空けてください。**即時リトライはシステム全体の負荷を増大させる**ため、避けるべきです。
