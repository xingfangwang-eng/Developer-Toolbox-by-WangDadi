# autotask - AUT_027

## 概要

autotask は、反復的なタスク実行を自動化し、開発者の手動作業をゼロに近づける次世代型タスクオーケストレーションツールです。ワークフローの定義、トリガー管理、実行履歴の可視化を一元化し、複数プロジェクトにまたがるタスク管理の複雑性を根本から解消します。

## 詳細な内容

### autotask とは？

autotask は、タスクの自動実行・管理に特化したSaaSツールです。従来のジョブスケジューラーのように単純な定期実行だけでなく、イベント駆動型のトリガー、条件分岐、エラーハンドリングを組み合わせた高度なワークフローを、コード不要で設計・運用できます。特に、CI/CDパイプラインの後処理や、定型的なデータメンテナンス、複数サービス間の連携タスクにおいて、その真価を発揮します。

### コア機能

- **マルチステートワークフローエンジン**: タスクの実行状態（pending, running, failed, retrying, completed）を自動管理し、失敗時には指数バックオフリトライを適用。ワークフロー内の各ステップは依存関係グラフで定義でき、並列実行や条件付き分岐もネイティブサポート。
- **イベント駆動トリガーマトリックス**: GitHub Webhook、Slackコマンド、HTTPリクエスト、スケジュール（cron式）、さらには他SaaSのWebhookを統合。トリガー条件をAND/ORで組み合わせた複合トリガーも設定可能。
- **実行トレースと自己修復ログ**: 全タスク実行の完全なトレースログを自動収集。失敗箇所を特定するだけでなく、同じエラーパターンを検知した場合に自動でワークフローを一時停止し、管理者にアラートを送信する自己修復機能を実装。

### 使い方

1. **ワークフローの定義とトリガー設定**
   `autotask workflow create --name "nightly_db_cleanup"` で新しいワークフローを作成。続けて `autotask trigger add --workflow nightly_db_cleanup --type cron --expression "0 2 * * *"` で毎日午前2時に実行するcronトリガーを設定。さらに、`autotask step add --workflow nightly_db_cleanup --name "archive_old_records" --action "postgres:archive" --params '{"table":"logs","older_than_days":30}'` で具体的なアクションを追加。各ステップには `--retry 3 --retry-delay 30` でリトライポリシーを付与。

2. **条件分岐とエラーハンドリングの実装**
   `autotask step add --workflow nightly_db_cleanup --name "check_disk_space" --action "system:disk_usage" --params '{"threshold":80}'` でディスク使用率をチェックするステップを挿入。次に、`autotask condition add --workflow nightly_db_cleanup --if "check_disk_space.percent < 80" --then "archive_old_records" --else "send_slack_alert"` で条件分岐を定義。`send_slack_alert` ステップは `autotask step add --workflow nightly_db_cleanup --name "send_slack_alert" --action "slack:send_message" --params '{"channel":"#ops-alerts","text":"ディスク容量が不足しています"}'` で作成。

3. **実行モニタリングと手動介入**
   `autotask run list --workflow nightly_db_cleanup --status failed` で失敗した実行を一覧表示。特定の実行IDに対して `autotask run inspect --id <run_id>` で詳細なトレースログを確認。失敗したステップを修正した後、`autotask run retry --id <run_id> --step archive_old_records` で特定のステップのみ再実行可能。さらに、`autotask run cancel --id <run_id>` で実行中のワークフローを即時停止。

### ベストプラクティス

- **タスクの冪等性を確保せよ**: ワークフロー内の各ステップは、何度実行しても同じ結果になるように設計する。特にデータベース操作やファイル処理では、UPSERTやべき等キーを活用し、リトライによる重複実行を防ぐ。autotaskのリトライ機能は強力だが、それを前提とした設計が安定運用の鍵。
- **トリガーにセマフォロックを導入せよ**: 同一ワークフローが短時間に多重起動されるのを防ぐため、トリガー設定時に `--concurrency 1` フラグを必ず付与する。また、外部APIを呼び出すステップにはレート制限を考慮し、`--rate-limit 10/min` を設定することで、プロバイダー側の制限に抵触するリスクを回避する。
