# autotask - AUT_013

## 概要

autotask は、反復作業の完全自動化とタスク依存関係の動的最適化を実現する、次世代型タスクオーケストレーターです。単なるスケジューラーではなく、実行コンテキストをリアルタイムで解析し、ワークフローを自己修復・適応させるインテリジェントな自動化基盤を提供します。特に CI/CD パイプラインやクラウドリソースのプロビジョニング、データパイプラインの運用において、その真価を発揮します。

## 詳細な内容

### とは autotask?

autotask は、複雑なタスクチェーンを宣言的に記述し、実行時の状態変化に応じて分岐・再試行・ロールバックを自動判定するプロフェッショナルツールです。従来の cron ベースのジョブ管理や単純なタスクランナーとは異なり、タスク間の依存グラフを DAG（有向非巡回グラフ）として管理し、並列実行の最適化と障害耐性を内蔵しています。これにより、運用チームは手動の監視や障害対応から解放され、本質的な開発に集中できます。

### コア機能

- **コンテキストアウェアな動的リトライ機構**  
  タスク失敗時に、エラーの種類（ネットワークタイムアウト、リソース不足、認証エラーなど）を自動解析し、適切なバックオフ戦略とリトライ回数を動的に決定。単なる指数バックオフではなく、過去の実行履歴とシステム負荷を考慮した適応型リトライを実装しています。

- **宣言的ワークフロー定義と依存解決エンジン**  
  YAML ベースの DSL でタスクの依存関係を宣言するだけで、実行エンジンが最適な並列実行計画を自動生成。循環依存の検出、クリティカルパスの抽出、リソース競合の回避を実行前に行い、デッドロックのリスクを排除します。

- **実行時メトリクスによる自己修復パイプライン**  
  各タスクの実行時間、メモリ使用量、I/O レイテンシをリアルタイム収集し、事前定義されたしきい値を超えた場合に自動でタスク分割やリソース増強をトリガー。例えば、データ処理タスクがメモリ不足で失敗しそうな場合、自動的にチャンクサイズを調整して再実行します。

### 使い方

1. **タスクグラフの宣言的定義**  
   `autotask.yml` にタスクと依存関係を記述します。各タスクには `id`、`command`、`depends_on` を指定し、必要に応じて `retry` ポリシーや `timeout` を設定します。依存関係は配列で複数指定可能で、すべての依存タスクが成功した後に実行されます。

   ```yaml
   tasks:
     - id: fetch_data
       command: "python fetch.py"
       retry:
         max_attempts: 3
         strategy: exponential
     - id: process_data
       command: "python process.py"
       depends_on:
         - fetch_data
       timeout: 300
   ```

2. **実行とリアルタイムモニタリング**  
   `autotask run` でワークフローを開始。実行中はターミナルに DAG の進捗がリアルタイム表示され、各タスクのログは `--log-level debug` で詳細出力可能。`Ctrl+C` で安全停止し、再開時には `--resume` フラグで中断位置から再開できます。

3. **障害発生時の自動リカバリと通知**  
   タスクが失敗すると、autotask は定義されたリトライポリシーに従って自動再実行。すべてのリトライが失敗した場合、依存する後続タスクはスキップされ、ワークフロー全体が `failed` 状態になります。この時、Slack やメールへの通知を `notifiers` セクションで設定可能です。

   ```yaml
   notifiers:
     slack:
       webhook_url: "https://hooks.slack.com/services/xxx"
       on: [failure, success]
   ```

### ベストプラクティス

- **タスクの粒度は「1 つの責務」に絞る**  
  1 つのタスクが複数の処理（データ取得、変換、保存）をまとめて行うと、障害の影響範囲が大きくなり、リトライの効率が低下します。autotask の DAG エンジンを活かすため、タスクは「1 つのコマンドで完結する最小単位」に分割し、依存関係で連結しましょう。これにより、部分的な再実行が容易になり、全体のスループットが向上します。

- **リトライ戦略は「エラーの種類ごとにカスタマイズ」する**  
  すべてのタスクに同じリトライポリシーを適用するのは非効率です。ネットワークエラーは短い間隔でリトライしても問題ありませんが、リソース不足の場合は長めのバックオフが必要です。autotask の `retry` ブロックでは、`conditions` でエラーパターンを指定できるため、`ConnectionError` と `MemoryError` で異なるリトライ戦略を設定しましょう。
