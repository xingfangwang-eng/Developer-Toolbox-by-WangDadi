# autotask - AUT_022

## 概要

autotask は、反復作業の自動化とタスク管理の効率化に特化した、プロフェッショナル向けのオープンソース CLI ツールです。バックエンド開発者や DevOps エンジニアが日常的に行う定型作業を、YAML ベースの設定ファイルで定義し、シェルスクリプトや API 呼び出しと統合することで、手動操作を排除します。本ツールは、複数のプロジェクトやリモートサーバーにまたがるタスクの一元管理を実現し、チーム全体の生産性を劇的に向上させます。

## コア機能

### 1. 動的タスクチェーンと条件分岐エンジン
タスクの実行順序を静的に定義するだけでなく、前のタスクの終了コードや出力内容に基づいて、次のタスクを動的に選択・分岐させることが可能です。例えば、デプロイスクリプトの成功時のみテストスイートを実行し、失敗時にはロールバックタスクを自動発火するといった、複雑なワークフローを YAML 一行で記述できます。

### 2. マルチランタイムプラグインアーキテクチャ
シェルコマンド、Python スクリプト、Node.js モジュール、Docker コンテナなど、異なるランタイムで書かれたタスクを統一的に管理・実行できます。各プラグインは独立したサンドボックスで動作し、環境変数やシークレットの受け渡しも安全に行えます。これにより、既存のスクリプト資産を無駄にせず、段階的に自動化を拡張できます。

### 3. 分散実行と集中ログ管理
SSH や gRPC を介して、複数のリモートマシンやクラウドインスタンスにタスクを分散実行できます。各ノードの実行結果は、リアルタイムで中央のログサーバーに集約され、タイムスタンプ付きで構造化保存されます。エラー発生時には、該当ログへの直接リンクが生成されるため、デバッグ時間を大幅に短縮できます。

## 使い方

### 1. タスク定義ファイルの作成とバリデーション
プロジェクトルートに `autotask.yaml` を作成し、以下のようにタスクを記述します。`autotask validate` コマンドで、YAML の構文チェックと依存関係の整合性検証を実行します。これにより、実行前に循環参照や未定義タスクの呼び出しを検出できます。

```yaml
tasks:
  build:
    type: shell
    command: "npm run build"
    env:
      NODE_ENV: "production"
  test:
    type: shell
    command: "npm test"
    depends_on:
      - build
    condition: "{{ tasks.build.exit_code == 0 }}"
```

### 2. シークレット管理と環境変数の注入
機密情報は平文で YAML に記述せず、`autotask secrets set` コマンドで暗号化ストアに登録します。タスク実行時には、`{{ secrets.DEPLOY_KEY }}` の形式で参照可能です。また、`.env` ファイルや CI/CD 環境変数との連携もサポートしており、`autotask run --env-file .env.prod` のように指定します。

### 3. 分散タスクの実行と進捗監視
`autotask run --targets node1,node2` のように対象ノードを指定して実行します。実行中は、`autotask status` で各ノードの進捗状況（実行中/成功/失敗）をリアルタイムに確認できます。また、`autotask logs --tail 50` で最新のログをストリーミング表示し、問題発生時には即座に介入できます。

## ベストプラクティス

### 1. タスクの粒度は「単一責任の原則」に従う
1 つのタスクには 1 つの責務だけを持たせ、複数のコマンドを 1 つのタスクに詰め込まないでください。例えば、「ビルドとテスト」を 1 つのタスクにするのではなく、`build` と `test` に分割し、依存関係で結びます。これにより、失敗時の原因特定が容易になり、タスクの再利用性も向上します。また、`autotask graph` コマンドでタスク間の依存関係を可視化し、チーム全体でワークフローを共有しましょう。

### 2. 冪等性を保証するための事前チェックを実装する
タスクの先頭に、実行条件が既に満たされているかを確認するガード節を必ず入れます。例えば、デプロイタスクであれば「対象サーバーに既に最新バージョンがデプロイされているか」をチェックし、スキップ可能な場合は `exit 0` で早期終了します。これにより、同じタスクを複数回実行しても副作用が発生せず、CI/CD パイプラインでの再実行が安全になります。
