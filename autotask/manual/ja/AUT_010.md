# autotask - AUT_010

## 概要

autotask は、反復作業を完全自動化する次世代型タスクオーケストレーションツールです。手動で行っていた定形処理を自動化し、エンジニアの認知負荷を劇的に軽減します。特にCI/CDパイプラインやバッチ処理、監視応答といった開発現場のルーティンワークに最適化されています。

## 詳細な内容

### autotask とは？

autotask は、タスク定義ファイル（YAML/JSON）に基づき、複数のアクションをシーケンシャルまたはパラレルに実行する軽量タスクランナーです。依存関係の解決、エラーハンドリング、リトライロジックを内蔵し、複雑なワークフローをシンプルに記述できます。スクリプト言語に依存せず、シェルコマンド、HTTPリクエスト、Dockerコンテナ実行など多様なアクションを統一的に扱います。

### コア機能

- **依存グラフベースのタスクスケジューリング**  
  タスク間の依存関係をDAG（有向非巡回グラフ）で定義。あるタスクが完了するまで後続タスクをブロックし、並列実行可能なタスクは自動で並列化。無駄な待ち時間を排除し、全体のスループットを最大化します。

- **プラグイン可能なアクションエンジン**  
  標準で `shell`、`http`、`docker`、`file` アクションを提供。カスタムプラグインをGoで実装すれば、任意の外部サービスや内部APIと連携可能。アクション実行前後のフック（pre/post hook）も定義でき、ログ出力や通知処理を柔軟に挿入できます。

- **冪等性保証と自動リトライ機構**  
  各タスクに冪等性フラグを設定可能。同一タスクが複数回実行されても副作用が発生しない設計。加えて、指数バックオフ付き自動リトライとサーキットブレーカーを内蔵し、ネットワーク不安定や一時的な障害からの復旧を自動化します。

### 使い方

1. **タスク定義ファイルの作成**  
   プロジェクトルートに `autotask.yaml` を作成し、タスク名、アクション種別、依存関係を記述します。以下は、コードのビルド→テスト→デプロイの3段階を定義する例です。

   ```yaml
   tasks:
     build:
       action: shell
       command: "go build -o ./bin/app ./cmd/"
     test:
       action: shell
       command: "go test ./..."
       depends_on: [build]
     deploy:
       action: http
       url: "https://api.example.com/deploy"
       method: POST
       body: '{"version": "{{ .Version }}"}'
       depends_on: [test]
       retry:
         attempts: 3
         backoff: exponential
   ```

2. **タスクの実行と進捗確認**  
   ターミナルで `autotask run` を実行。各タスクの実行状態がリアルタイムで出力され、失敗したタスクは赤色でハイライトされます。`autotask status` で現在の実行中タスク一覧と残り時間予測を確認可能。`--dry-run` フラグを付けると、実行せずに依存グラフと実行順序をプレビューできます。

3. **変数とシークレットの注入**  
   タスク定義内で `{{ .VarName }}` 形式のテンプレート変数を使用可能。実行時に `--var Version=1.2.3` で値を渡すか、`.env` ファイルから読み込みます。機密情報は `--secret` フラグで渡し、ログ出力時には自動マスクされます。これにより、環境ごとに異なるパラメータを安全に切り替えられます。

### ベストプラクティス

- **タスクの粒度は「1アクション＝1責務」に分割する**  
  1つのタスクに複数のシェルコマンドやHTTPリクエストを詰め込まず、単一の責務（ビルド、テスト、通知など）ごとにタスクを分割します。これにより、失敗時の再実行範囲が最小化され、依存グラフの可視性が向上します。特にCI/CDでは、後続タスクへの影響を局所化するために厳守すべき設計原則です。

- **冪等性を前提としたタスク設計を徹底する**  
  すべてのタスクは「何度実行しても同じ結果になる」ように実装します。例えば、ファイル生成タスクでは `--force` オプションで上書きを許容し、DBマイグレーションでは `IF NOT EXISTS` 句を使用します。冪等性を担保することで、リトライや部分再実行が安全に行え、障害復旧時の運用負荷が大幅に低下します。
