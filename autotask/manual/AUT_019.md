# autotask - AUT_019

## Overview

autotask is a zero-overhead task automation engine engineered for developers who demand precision, speed, and scalability. It eliminates manual toil by intelligently orchestrating workflows across distributed systems, from CI/CD pipelines to cloud-native event triggers. Built with a polyglot core (Go + Rust), it delivers sub-100ms execution latency while maintaining full idempotency guarantees.

## Core Features

### 1. **Declarative Workflow Graph with DAG Validation**
Define complex task dependencies using a YAML/JSON-based directed acyclic graph (DAG) schema. autotask automatically validates cycles, detects orphan nodes, and optimizes execution order using topological sorting. Supports conditional branching, parallel fan-out, and retry policies with exponential backoff (configurable via `retry_strategy`).

### 2. **Real-Time State Machine with Observable Hooks**
Every task instance is backed by a finite state machine (FSM) with states: `pending`, `running`, `failed`, `completed`, `cancelled`. Expose granular lifecycle events via WebSocket or gRPC streams. Integrate with Prometheus or Datadog using built-in metrics exporters for SLI/SLO tracking.

### 3. **Multi-Tenant Resource Isolation with Quota Enforcement**
Run thousands of concurrent task pipelines without contention. autotask implements per-namespace resource quotas (CPU, memory, I/O) via cgroup v2 and Linux namespaces. Tasks exceeding limits are automatically paused or killed, with audit logs streamed to stdout or a centralized log aggregator (e.g., Loki, ELK).

## How to Use

### Step 1: Bootstrap a Task Pipeline via CLI
```bash
autotask init --name "deploy-staging" \
  --file ./workflow.yaml \
  --namespace prod \
  --max-retries 3
```
This creates a new pipeline definition. The `--file` flag parses a YAML with task definitions (e.g., `build`, `test`, `deploy`). The namespace ensures isolation; tasks under `prod` cannot interfere with `dev`.

### Step 2: Define Tasks with Conditional Logic
In your `workflow.yaml`:
```yaml
tasks:
  - id: build
    image: node:18-alpine
    command: ["npm", "run", "build"]
    depends_on: []
    on_failure: retry
    retry_count: 2
  - id: test
    image: node:18-alpine
    command: ["npm", "test"]
    depends_on: [build]
    conditions:
      - if: "{{ tasks.build.exit_code == 0 }}"
        then: run
        else: skip
  - id: deploy
    image: alpine
    command: ["sh", "-c", "kubectl apply -f ./k8s/"]
    depends_on: [test]
    timeout: 300
```
Use Go templates (`{{ }}`) for dynamic condition evaluation. The engine evaluates `exit_code` or `stdout` from upstream tasks.

### Step 3: Trigger Execution and Monitor via Dashboard
```bash
autotask run --pipeline deploy-staging --async
```
The `--async` flag returns a pipeline ID immediately. Stream logs:
```bash
autotask logs --pipeline-id <ID> --follow
```
Or open the built-in TUI dashboard (`autotask dashboard`) for real-time DAG visualization, task status, and resource consumption graphs.

## Best Practices

### 1. **Idempotency via Deterministic Task IDs**
Always assign a deterministic `task_id` based on input hash (e.g., `sha256(git_commit + env)`). This ensures that rerunning a completed task with identical parameters returns cached results, avoiding redundant work and side effects. Pair with `--idempotency-key` flag for external triggers.

### 2. **Graceful Degradation with Circuit Breaker Pattern**
For tasks hitting external APIs (e.g., AWS Lambda, Slack), wrap them with a circuit breaker. autotask supports a `circuit_breaker` block:
```yaml
circuit_breaker:
  failure_threshold: 5
  reset_timeout: 60
  half_open_max_requests: 3
```
This prevents cascading failures when downstream services are degraded. Combine with `on_failure: fallback` to route to a backup task (e.g., a mock endpoint).
