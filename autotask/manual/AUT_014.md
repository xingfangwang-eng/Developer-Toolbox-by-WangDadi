### What is autotask?

autotask is a professional-grade, open-source task automation framework that solves the problem of brittle, hard-coded background job logic. It decouples task definitions from their execution, allowing you to define, schedule, and monitor complex workflows via a declarative YAML or JSON schema. Think of it as a lightweight, developer-first alternative to heavyweight enterprise schedulers, built for modern CI/CD pipelines, data processing, and microservice choreography.

### Core Features

1.  **Atomic Task Chaining with Conditional Rollbacks:** Unlike simple linear pipelines, autotask supports directed acyclic graphs (DAGs) for task dependencies. If any task in a chain fails, the engine automatically executes a predefined rollback sequence for all successfully completed predecessors, ensuring data integrity without manual intervention.
2.  **Distributed Lock-Free Execution via Idempotency Keys:** autotask eliminates race conditions in distributed environments. Each task execution is assigned a unique idempotency key. The engine guarantees exactly-once execution semantics, even under network partitions or worker crashes, making it safe for critical financial or database operations.
3.  **Real-Time State Stream with WebSocket Subscriptions:** Instead of polling for status, autotask streams every state transition (queued, running, failed, completed, skipped) to a configurable WebSocket endpoint. This allows you to build live dashboards or trigger downstream microservices with zero latency, directly from the task execution flow.

### How to Use

1.  **Define a Stateful Workflow with Retry Policies:**
    Create a `workflow.yaml` file. Define a task that sends a confirmation email after a user signs up. Crucially, attach a retry policy with exponential backoff and a circuit breaker to handle temporary SMTP failures.
    ```yaml
    workflow: user_onboarding
    tasks:
      - id: send_welcome_email
        action: email.send
        input:
          to: "{{trigger.user.email}}"
          template: welcome
        retry:
          max_attempts: 3
          backoff: exponential
          max_delay_seconds: 120
        on_failure: notify_admin_task
    ```
    Then, register this workflow using the CLI: `autotask register --file workflow.yaml`.

2.  **Trigger the Workflow via API with Custom Context:**
    Use the REST API to trigger the workflow. Pass a dynamic JSON payload as the context. This allows you to inject user-specific data at runtime without modifying the workflow definition.
    ```bash
    curl -X POST https://your-autotask-instance/api/v1/workflows/user_onboarding/trigger \
      -H "Content-Type: application/json" \
      -d '{"user": {"email": "dev@example.com", "plan": "pro"}}'
    ```
    The engine will instantiate a new execution, bind the context, and queue the first task.

3.  **Monitor Execution via WebSocket for Real-Time Feedback:**
    Subscribe to the execution stream to get live updates. This is essential for building a responsive UI or for logging in a separate monitoring system.
    ```javascript
    const ws = new WebSocket('wss://your-autotask-instance/api/v1/stream');
    ws.onmessage = (event) => {
      const execution = JSON.parse(event.data);
      if (execution.task_id === 'send_welcome_email' && execution.status === 'completed') {
        console.log('Welcome email sent successfully.');
        // Trigger a follow-up action in your app
      }
    };
    ```

### Best Practices

1.  **Design for Idempotency from Day One:** Always ensure your task handlers (e.g., API calls, database writes) are idempotent. autotask's retry mechanism relies on this. Use the provided `execution_id` as a unique key in your downstream systems to prevent duplicate side effects. This is non-negotiable for production-grade reliability.
2.  **Use Workflow Versioning for Zero-Downtime Updates:** When modifying a workflow definition, always bump its version (e.g., `user_onboarding:v2`). This allows running instances of the old version to complete gracefully while new triggers use the updated logic. This prevents breaking in-flight tasks and maintains a clean audit trail.
