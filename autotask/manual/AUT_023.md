# autotask - AUT_023

## Overview

autotask is a developer-first, event-driven task orchestration engine designed for microservices, CI/CD pipelines, and edge computing environments. It eliminates boilerplate scheduling logic and provides real-time observability into distributed task execution.

## What is autotask?

autotask is a lightweight, language-agnostic runtime that decouples task definitions from execution infrastructure. It supports idempotent retries, dynamic dependency injection, and pluggable backends (Redis, PostgreSQL, NATS). Built for teams that need to automate complex multi-step workflows without writing a single cron expression.

## Core Features

- **Declarative DAG Pipelines** – Define task graphs using YAML or JSON. autotask resolves topological dependencies at runtime, parallelizes independent tasks, and handles fan-out/fan-in patterns automatically. No more nested callback hell.

- **Intelligent Retry with Exponential Backoff + Jitter** – Failed tasks are retried with configurable backoff strategies. The engine injects random jitter to avoid thundering herd problems in distributed systems. Supports dead-letter queues and manual re-queue via CLI.

- **Real-Time State Streaming via WebSocket** – Every task transition (pending → running → success/failure) is pushed to subscribed clients. Integrates with Prometheus metrics and structured logging (JSON, OpenTelemetry). No polling required.

## How to Use

1. **Define a Task Graph**  
   Create a `tasks.yaml` file with your workflow. Each task specifies its command, environment variables, dependencies, and retry policy. Example:  
   ```yaml
   tasks:
     - id: build
       command: "make build"
       depends_on: []
     - id: test
       command: "make test"
       depends_on: [build]
   ```
   Validate the graph with `autotask validate tasks.yaml`.

2. **Execute the Workflow**  
   Run `autotask run tasks.yaml --backend redis --watch`. The engine parses the DAG, spawns tasks in order, and streams logs to stdout. Use `--parallel 4` to control concurrency. Monitor execution via the built-in Web dashboard at `http://localhost:8080`.

3. **Handle Failures & Audit**  
   After execution, inspect results with `autotask history --last 10`. For failed tasks, rerun with `autotask retry <task-id> --force`. All state is persisted in the configured backend—no data loss on crash.

## Best Practices

- **Use Idempotent Task Handlers** – Design each task to be safely re-runnable. autotask relies on idempotency for retries. For example, use `CREATE TABLE IF NOT EXISTS` in SQL tasks or checkpoints in data pipelines. This prevents duplicate side effects.

- **Isolate Secrets via Environment Injection** – Never hardcode API keys or tokens in task definitions. Use `autotask secret set <key> --value <secret>` to store encrypted secrets. Reference them in tasks as `$SECRET_<KEY>`. The engine decrypts them at execution time, leaving no trace in logs.
