## Detailed Content

### What is autotask?

autotask is not just another cron wrapper or todo list manager. It is a declarative task orchestration layer that runs locally or in CI/CD pipelines, allowing you to define, schedule, and chain tasks using a minimal YAML DSL. Built for polyglot environments, it supports native hooks for Python, Node.js, Shell, and Docker containers, and outputs structured logs for observability.

### Core Features

- **Dependency-Aware Task Graph**  
  autotask automatically resolves task dependencies using a directed acyclic graph (DAG). You can define parallel execution branches, conditional triggers, and retry policies without writing glue code. Failures are isolated and logged with full stack traces.

- **Idempotent Execution with State Snapshots**  
  Every task run produces a deterministic state snapshot. If a task fails mid-way, autotask can resume from the last successful checkpoint—no more manual cleanup or rerunning entire pipelines.

- **Built-in Rate Limiting & Resource Quotas**  
  Prevent runaway automation with per-task CPU/memory limits and concurrency throttling. Ideal for multi-tenant setups or cloud-hosted runners where resource contention is a real concern.

### How to Use

1. **Define a Task Pipeline with Conditional Branching**  
   Create a `autotask.yaml` file in your project root. Use the `depends_on` field to specify upstream tasks, and add `when` clauses for conditional execution (e.g., only run cleanup if a build task succeeded). Run `autotask run --pipeline nightly` to execute the entire graph.

2. **Inject Secrets via Environment Mapping**  
   Instead of hardcoding API keys or database credentials, use the `env_map` directive to pull values from your local `.env` file or a secure vault. autotask will validate required variables at parse time and fail fast if any are missing—saving you from runtime surprises.

3. **Monitor Execution with Structured Logs & Alerts**  
   Enable `log_format: json` in your config to pipe machine-readable output to tools like Logstash or Datadog. Use the built-in `notify` block to send Slack or email alerts only on task failure or threshold breaches. No third-party plugin needed.

### Best Practices

- **Design for Idempotency from Day One**  
  Every task in autotask should be safe to run multiple times. Use idempotent operations (e.g., `PUT` instead of `POST`, upserts instead of inserts) and leverage state snapshots to avoid side effects. This makes debugging and rollback trivial.

- **Keep Pipeline Definitions Under Version Control**  
  Treat your `autotask.yaml` files like code. Store them in your repository, review changes via pull requests, and tag releases. This ensures your automation logic is auditable, reproducible, and testable across environments.
