## What is autotask?

**autotask** is a lightweight, event-driven task orchestration framework built for developers who need to automate complex workflows without sacrificing control. It is not a glorified cron job—it is a stateful, dependency-aware execution engine that supports conditional branching, retry policies, and real-time observability out of the box.

---

## Core Features

### 1. **Stateful Workflow Graph with Conditional Gates**
autotask models each automation as a directed acyclic graph (DAG) of tasks, where each node can hold its own state. You can define conditional gates (e.g., `if task_A.status == "failed"`) to dynamically skip, retry, or reroute execution paths—without writing boilerplate.

### 2. **Pluggable Execution Backends**
Swap between local threads, Redis-backed queues, or Kubernetes Job executors via a single config entry. autotask abstracts the execution layer, letting you scale from a single dev machine to a production cluster without rewriting your task definitions.

### 3. **Built-in Observability via Structured Logs & Metrics**
Every task execution emits structured JSON logs and Prometheus-compatible metrics (latency, retry count, state transitions). No need to instrument manually—just point your logging backend or monitoring stack at the autotask output stream.

## How to Use

### 1. Define a Task with Dependencies and Retry Policy
Create a `task.yaml` file and declare a task that depends on a previous one. Specify a retry policy with exponential backoff.

```yaml
id: "data_ingestion"
depends_on: ["auth_check"]
retry:
  max_attempts: 3
  backoff: "exponential"
  initial_delay_ms: 1000
run:
  command: "python ingest.py --source s3://bucket/raw"
  timeout_sec: 300
```

Run it:

```bash
autotask run --file task.yaml
```

### 2. Trigger a Conditional Execution Based on Previous Task Status
Use the `gate` field to control flow. This example runs a cleanup task only if the ingestion task succeeded.

```yaml
id: "cleanup"
depends_on: ["data_ingestion"]
gate:
  condition: "task.data_ingestion.status == 'completed'"
run:
  command: "python cleanup.py"
```

autotask evaluates the gate expression at runtime and skips the node if the condition fails.

### 3. Monitor Execution in Real Time via the CLI Dashboard
Start the built-in streaming dashboard to watch task state transitions live.

```bash
autotask dashboard --stream
```

You’ll see a live table with columns: `task_id`, `status`, `attempt`, `duration_ms`, `error`. Press `Ctrl+C` to stop.

## Best Practices for Production Use

### 1. **Always Set a `timeout_sec` and `max_attempts` on Critical Tasks**
Without explicit timeouts, a hanging task can block downstream tasks indefinitely. Combine this with a retry policy that uses exponential backoff to avoid hammering external services during transient failures.

### 2. **Use Structured Logs Instead of Console Output for Debugging**
autotask emits JSON logs by default. Pipe them into a log aggregator (e.g., Loki, ELK) and set up alerts on `status == "failed"` or `duration_ms > p95`. This gives you actionable insights without grepping raw output.
