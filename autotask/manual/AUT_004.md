## Detailed Content

### What is autotask?

**autotask** is a lightweight, event-driven task orchestration engine designed to eliminate manual ticketing and cron-job spaghetti. It acts as a state machine for your DevOps pipeline, handling complex dependency chains, retries with exponential backoff, and real-time status broadcasting. Think of it as a headless, API-first alternative to heavy-weight workflow managers, optimized for microservice environments where latency and idempotency are non-negotiable.

### Core Features

1.  **Declarative DAG-based Execution Graph**  
    Unlike linear step runners, autotask parses a YAML/JSON DAG (Directed Acyclic Graph) to resolve complex inter-task dependencies. It automatically parallelizes independent branches and implements a topological sort for execution. This enables true "fan-out/fan-in" patterns without manual thread management.

2.  **Idempotent Retry with Circuit Breaker Integration**  
    Each task is wrapped in an idempotency key. If a task fails, autotask doesn't just retry; it uses a configurable circuit breaker (stateful failure threshold) to prevent cascading failures. The engine stores the last successful checkpoint, allowing for granular resume—not a full restart—after a crash.

3.  **Real-time State Webhook & Dead Letter Queue (DLQ)**  
    Every state transition (pending, running, failed, completed) triggers a configurable webhook. For tasks that exhaust all retries, autotask automatically routes the payload and metadata to a dedicated DLQ (e.g., SQS, RabbitMQ). This ensures zero data loss and allows for manual forensic analysis without blocking the main pipeline.

### How to Use

1.  **Define a Task DAG with Strict Dependency Hashing**  
    Create a `autotask.yml` file. Use the `depends_on` field to define exact task IDs. To avoid race conditions, autotask uses a SHA-256 hash of the parent task’s output as a dependency validation token. Example:
    ```yaml
    tasks:
      - id: "build_artifact"
        image: "node:18"
        command: "npm run build"
      - id: "deploy_staging"
        depends_on: ["build_artifact"]
        image: "aws-cli"
        command: "aws s3 sync ./dist s3://staging-bucket"
    ```
    *Pro tip: Always use explicit `id` values; autotask does not infer dependencies from names.*

2.  **Execute with Strict Mode & Atomic Logging**  
    Trigger execution via the CLI: `autotask run --file autotask.yml --strict`. The `--strict` flag ensures the engine validates the entire DAG for cycles before executing a single task. Logs are streamed atomically to `stdout` and a local SQLite database (`autotask.db`) for post-mortem debugging. Use `autotask logs --task-id <id>` to stream a specific task’s output without blocking the main process.

3.  **Configure Circuit Breaker & DLQ Routing**  
    Add a global config block to `autotask.yml`:
    ```yaml
    engine:
      retry_policy:
        max_attempts: 3
        backoff: "exponential" # base delay: 2s
      circuit_breaker:
        failure_threshold: 5
        timeout: 30s
      dead_letter_queue:
        type: "sqs"
        url: "https://sqs.us-east-1.amazonaws.com/1234567890/autotask-dlq"
    ```
    When a task fails 3 times, it is marked `failed` and the full payload (including environment variables) is pushed to the DLQ. The main DAG continues executing, but any task depending on the failed one enters a `blocked` state.

### Best Practices

1.  **Treat Every Task as a Stateless Microservice**  
    Do not rely on local file system state between tasks. Use external volumes (Docker volumes or S3) for artifact sharing. autotask does not guarantee that two tasks run on the same host. By enforcing statelessness, you unlock horizontal scaling and idempotent retries without side effects.

2.  **Implement a "Canary" Task for Pre-Production Validation**  
    Before running a full deployment DAG, insert a canary task that executes a synthetic health check against the target environment. Use the `depends_on` field to make all subsequent deployment tasks depend on this canary. If the canary fails, the circuit breaker stops the entire pipeline, preventing a bad release from hitting production.
