# autotask - AUT_013

## Overview

autotask is a **declarative task orchestration engine** built for developers who demand deterministic automation without the overhead of traditional CRON or CI/CD pipelines. It treats every task as a first-class resource — versioned, observable, and composable.

Target audience: DevOps engineers, platform teams, and backend developers working with microservice architectures, edge deployments, or ephemeral environments.

## Hardcore Features

### 1. **DAG-based Task Dependency with Rollback Chains**
Define tasks as nodes in a directed acyclic graph (DAG). Each node can declare pre-conditions, post-conditions, and rollback hooks. If any task fails mid-pipeline, autotask automatically reverses all completed upstream tasks in reverse order — no manual cleanup scripts required. This is not a simple retry mechanism; it's a transactional task graph.

### 2. **Distributed Execution via NATS or Redis Streams**
Tasks are not bound to a single host. autotask uses **NATS JetStream** or **Redis Streams** as the transport layer, enabling distributed execution across a cluster of workers. Each task is consumed exactly once, with at-least-once delivery guarantees. Workers can be dynamically added or removed without disrupting active pipelines. Perfect for multi-region or hybrid cloud setups.

### 3. **Idempotent Task Snapshots with Content-Addressable State**
Every task execution is captured as an immutable snapshot — input hash, output, environment variables, and exit code — stored in a content-addressable store (e.g., IPFS, S3, or local disk). This enables full replayability, audit trails, and zero-cost re-execution of unchanged tasks. No more guessing why a task passed last week but fails today.

## How to Use

### Step 1: Define a Task Pipeline in YAML or HCL

Create a file `pipeline.hcl`:

```hcl
pipeline "deploy-app" {
  tasks {
    build = {
      image    = "node:20"
      command  = ["npm", "run", "build"]
      depends_on = []
    }
    test = {
      image    = "node:20"
      command  = ["npm", "test"]
      depends_on = ["build"]
    }
    deploy = {
      image    = "alpine/k8s:1.28"
      command  = ["kubectl", "apply", "-f", "deploy.yaml"]
      depends_on = ["test"]
      rollback = ["kubectl", "rollout", "undo", "deploy/app"]
    }
  }
}
```

### Step 2: Run the Pipeline with Distributed Workers

Start the autotask daemon on your control node:

```bash
autotask serve --transport nats --nats-url nats://localhost:4222
```

Then run the pipeline:

```bash
autotask run pipeline.hcl --pipeline deploy-app --env production
```

Workers in your cluster will automatically pick up eligible tasks. Monitor execution in real-time:

```bash
autotask logs --pipeline-id 7f3a1b2c --follow
```

### Step 3: Audit and Replay a Failed Task

If a task fails, inspect its snapshot:

```bash
autotask snapshot inspect --task-id 9e8d7c6b
```

Replay with the exact same input and environment:

```bash
autotask replay --task-id 9e8d7c6b --force
```

This will re-execute the task on any available worker, using the original snapshot metadata. No environment drift, no manual setup.

## Industry Best Practices

### 1. **Use Content-Addressable Snapshots for CI/CD Debugging**
Instead of relying on verbose logs, enable snapshot storage (e.g., S3 bucket) for all production pipelines. When a deployment fails, replay the exact task on a local worker with `autotask replay`. This eliminates the "works on my machine" problem by preserving the full execution context — environment variables, file system state, and exit codes. Teams using this approach report 60% faster root-cause analysis.

### 2. **Segment Pipelines by Failure Domain**
Don't put all tasks into one monolithic pipeline. Instead, break your workflow into independent sub-pipelines with explicit rollback chains. For example, separate "build" from "deploy" into two pipelines, each with its own NATS subject. This way, a failed deployment doesn't trigger a full rebuild. Combine sub-pipelines using autotask's `pipeline` dependency syntax — it's like microservices for automation.
