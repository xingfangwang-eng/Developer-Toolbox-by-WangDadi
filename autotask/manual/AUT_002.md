# autotask - AUT_002

## Overview

`autotask` is a purpose-built task automation engine designed for developers, DevOps engineers, and SaaS operators who need to eliminate repetitive workflows without writing fragile shell scripts or maintaining brittle cron jobs. It’s not just another task scheduler—it’s a declarative, idempotent automation layer that integrates directly with your CI/CD pipeline, cloud APIs, and monitoring stack.

## What is autotask?

`autotask` is a lightweight, YAML-driven automation runtime that executes conditional, multi-step task sequences with built-in error recovery, state persistence, and observability hooks. It treats automation as code—version-controlled, testable, and auditable.

## Core Features

### 1. Idempotent Task Execution with State Snapshots
Every task run is snapshot-aware: `autotask` stores the execution state in a local or remote backend (SQLite, Redis, S3). If a task fails mid-sequence, the next run resumes from the last successful checkpoint—no duplicate side effects, no manual cleanup.

### 2. Declarative Dependency Graph (DAG) Engine
Define task dependencies as a directed acyclic graph in plain YAML. `autotask` resolves execution order, parallel branches, and conditional gates automatically. No more nested `if-else` spaghetti—just a clean, visualizable workflow.

### 3. Native Webhook & Event-Driven Triggers
Skip polling. `autotask` exposes a built-in HTTP server that listens for webhooks from GitHub, GitLab, Slack, or any REST endpoint. Tasks can be triggered by events (e.g., `on: push`, `on: alert`) with payload extraction and dynamic variable injection.

## How to Use

### 1. Define a Task Workflow in YAML
Create a `.autotask/workflows/backup-db.yaml` file:

```yaml
version: "2.0"
workflow:
  id: db-backup
  triggers:
    - cron: "0 3 * * *"
  steps:
    - name: dump-database
      action: exec
      command: "pg_dump -U admin mydb > /tmp/mydb.sql"
      retry: 2
    - name: upload-to-s3
      action: s3-upload
      source: "/tmp/mydb.sql"
      bucket: "my-backups"
      key: "{{ .Date }}/mydb.sql"
      depends_on: ["dump-database"]
    - name: notify-slack
      action: webhook
      url: "https://hooks.slack.com/services/T.../B.../xxx"
      payload: '{ "text": "Backup completed for {{ .Date }}" }'
      depends_on: ["upload-to-s3"]
```

### 2. Run the Workflow Locally or in CI
Execute with a single command:

```bash
autotask run --workflow db-backup --env-file .env.prod
```

Use `--watch` to tail logs in real time, or `--dry-run` to validate the DAG without side effects.

### 3. Trigger via Webhook for On-Demand Execution
Send a POST request to the built-in webhook endpoint:

```bash
curl -X POST http://localhost:8080/webhook/db-backup \
  -H "Content-Type: application/json" \
  -d '{"branch": "main", "force": true}'
```

`autotask` validates the payload against a schema defined in the workflow, injects variables, and executes the DAG.

## Best Practices

### 1. Use Idempotent Actions with Retry Policies
Always design actions to be safely re-run. For example, use `INSERT ... ON CONFLICT DO NOTHING` instead of raw `INSERT`. Pair this with `autotask`’s built-in retry and backoff (exponential or fixed) to handle transient failures without manual intervention.

### 2. Isolate Workflow State per Environment
Leverage the `--state-backend` flag to point each environment (dev, staging, prod) to a separate state store. This prevents cross-contamination during testing and ensures that a failed production task doesn’t corrupt staging state. Use environment variables for secrets (e.g., `AUTOTASK_STATE_DSN=redis://...`).
