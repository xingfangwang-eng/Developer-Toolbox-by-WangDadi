## Detailed Content

### What is autotask?

autotask is a high-performance, self-hosted task orchestration engine designed for developers and DevOps teams who need to automate complex, multi-step workflows without relying on cloud-dependent services. It integrates natively with CI/CD pipelines, cron schedulers, and webhook triggers, providing a lightweight yet robust alternative to enterprise automation suites.

### Core Features

- **Idempotent Task Graph Engine**  
  autotask executes tasks as a directed acyclic graph (DAG) where each node is idempotent. This means even if a step fails mid-execution, retries resume from the last successful checkpoint—no duplicate side effects, no wasted compute.

- **Declarative YAML-Based Workflow DSL**  
  Define workflows in pure YAML with built-in support for conditionals, parallel branches, and dynamic variable injection. No GUI drag-and-drop; just version-controlled, Git-friendly configuration files that are as easy to review as code.

- **Zero-Dependency Agentless Execution**  
  Unlike tools that require heavy agents or daemons, autotask uses SSH, Docker exec, or local subprocess calls directly. This reduces attack surface and deployment complexity, making it ideal for secure or air-gapped environments.

### How to Use

1. **Initialize a Workflow File**  
   Create `workflow.yml` in your project root. Define the `tasks` block with each step’s command, environment variables, and retry policy. Example:  
   ```yaml
   tasks:
     - id: build
       run: "docker build -t app:latest ."
       retry: 2
     - id: test
       run: "pytest tests/"
       depends_on: [build]
   ```

2. **Trigger Execution via CLI**  
   Run `autotask run workflow.yml` to execute the DAG. Use flags like `--env=production` to inject environment-specific variables or `--dry-run` to validate syntax without execution. The engine prints real-time logs with task-level timestamps and exit codes.

3. **Monitor and Debug with Graph Output**  
   After execution, autotask generates a `graph.json` file containing the execution tree, including each node’s duration, status, and output hash. Visualize it with any graphviz-compatible tool, or pipe it into `autotask inspect graph.json` for a terminal-based interactive view.

### Best Practices

- **Keep tasks idempotent and stateless**  
  Each task should produce the same output given the same input, regardless of how many times it runs. Use checksums or lock files to prevent redundant operations. This ensures your pipeline is resilient to partial failures and manual retries.

- **Version-control your workflow definitions**  
  Store `workflow.yml` and any associated `.autotask/` config files in your Git repository. Tag releases with the workflow version to enable rollback. This makes auditing and collaboration seamless, especially in teams with multiple contributors.
