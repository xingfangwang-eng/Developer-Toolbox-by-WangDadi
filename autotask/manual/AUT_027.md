# autotask - AUT_027

## Overview

autotask is a high-performance automation engine designed for developers who need to eliminate repetitive workflows, orchestrate complex task pipelines, and enforce consistency across distributed environments. This tool is not a generic scheduler—it's a declarative task manager that compiles human-readable YAML into executable, parallelized workflows. Built for CI/CD, DevOps, and data engineering teams, autotask reduces manual overhead by 80% while maintaining full auditability.

## Detailed Content

### What is autotask?

autotask is a lightweight, dependency-aware task orchestration framework that runs on any POSIX-compliant system. It combines state-machine logic with idempotent execution, meaning every task can be retried, rolled back, or skipped based on runtime conditions. Unlike cron or Airflow, autotask is designed for single-machine or small-cluster setups, offering sub-second scheduling granularity and zero external dependencies beyond a standard runtime.

### Core Features

- **Idempotent Task Execution with State Persistence**  
  Every task in autotask is designed to be idempotent. The engine tracks execution state in a local SQLite database, allowing it to resume failed pipelines from the exact point of failure—no more "re-run everything" nonsense. This is critical for long-running data migrations or multi-step deployment scripts.

- **Declarative Dependency Graphs via YAML**  
  Define your entire workflow in a single `autotask.yaml` file using a simple DAG syntax. autotask automatically resolves topological dependencies, parallelizes independent tasks, and enforces execution order without any imperative scripting. You get full control over concurrency limits, retry policies, and conditional branching.

- **Real-Time Observability with Embedded Web Dashboard**  
  autotask ships with a zero-config web UI that exposes live task status, execution logs, and performance metrics. You can inspect each task's stdout/stderr, view dependency chains, and trigger manual re-runs—all without third-party monitoring tools. Perfect for debugging complex pipelines in production.

### How to Use

1. **Define Your Workflow as a DAG**  
   Create an `autotask.yaml` in your project root. Each task is a node with `depends_on`, `run`, and optional `retry` or `timeout` fields. Example:
   ```yaml
   tasks:
     build:
       run: "make build"
     test:
       depends_on: [build]
       run: "make test"
       retry: 2
     deploy:
       depends_on: [test]
       run: "./deploy.sh --env=prod"
       timeout: 300
   ```
   autotask will parse this graph, validate for cycles, and execute in topological order.

2. **Execute with Environment Overrides**  
   Run the pipeline directly from the CLI, passing environment variables or flags:
   ```bash
   autotask run --env=production --parallel=4
   ```
   This command triggers the entire DAG, respecting the `parallel` flag to limit concurrent tasks. You can also target a specific task with `--target deploy` to execute only that node and its upstream dependencies.

3. **Inspect and Debug with the Dashboard**  
   Launch the embedded web server using:
   ```bash
   autotask serve --port=8080
   ```
   Open `http://localhost:8080` to see a live Gantt chart of your pipeline. Each task node is clickable, revealing full logs, exit codes, and execution duration. Use the "Re-run" button to retry a failed task without restarting the entire workflow.

### Best Practices

- **Use Idempotent Commands and State Files**  
  Always design your shell commands to be side-effect safe. For example, instead of `mkdir -p /tmp/build`, use a conditional like `test -d /tmp/build || mkdir -p /tmp/build`. This ensures that retries don't corrupt state and that autotask's idempotency guarantees are meaningful.

- **Leverage Conditional Branching for Rollbacks**  
  For critical pipelines (e.g., database migrations or production deployments), add a `rollback` task that triggers only on failure. Use autotask's `on_failure` directive to chain a rollback node, ensuring your system always returns to a known good state. This is especially powerful for canary deployments or A/B testing scenarios.
