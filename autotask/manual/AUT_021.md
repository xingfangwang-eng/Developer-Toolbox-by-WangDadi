### What is autotask?

autotask is a lightweight, event-driven task automation framework that lets you define, chain, and execute complex workflows using declarative YAML or a native Python DSL. It's engineered for low-latency execution, idempotent task handling, and seamless integration with existing monitoring stacks (Prometheus, Grafana). Unlike cron or basic schedulers, autotask supports conditional branching, retry strategies with exponential backoff, and distributed execution across worker nodes.

### Core Features

- **Idempotent Task Graph Engine** — Every task in autotask is designed to be idempotent. The engine tracks execution state via a pluggable backend (Redis, PostgreSQL, or SQLite), ensuring that retries never produce duplicate side effects. You can safely restart failed pipelines mid-flight without corrupting state.
- **Dynamic Dependency Injection & Context Propagation** — Tasks can inject runtime parameters, secrets, and outputs from upstream tasks automatically. Use `${{ steps.previous_task.output }}` syntax to chain results, or override with environment variables. No more hardcoded configs or brittle environment files.
- **Built-in Circuit Breaker & Rate Limiter** — Prevent cascading failures in distributed task runs. autotask integrates a circuit breaker pattern per task node, with configurable failure thresholds and cooldown periods. Rate limiters throttle API calls or external service interactions to avoid provider bans.

### How to Use

1. **Define a Multi-Stage Workflow with Conditional Branching**  
   Create a `workflow.yaml` file. Use `on_failure` and `on_success` hooks to branch logic based on exit codes. Example:  
   ```yaml
   tasks:
     - id: validate_config
       image: alpine:latest
       command: ["sh", "-c", "test -f /etc/app/config.yml"]
     - id: deploy_staging
       depends_on: [validate_config]
       on_success: ["notify_slack"]
       on_failure: ["rollback"]
       command: ["ansible-playbook", "deploy.yml"]
   ```
   Run with `autotask run workflow.yaml --env staging`.

2. **Execute Parallel Fan-Out Tasks with Aggregation**  
   Use the `fan_out` directive to spawn multiple identical tasks across different inputs, then aggregate results with a reducer task.  
   ```yaml
   tasks:
     - id: scan_ports
       fan_out:
         targets: ["10.0.1.1", "10.0.1.2", "10.0.1.3"]
         parallel: 3
       command: ["nmap", "-p", "80,443", "$TARGET"]
     - id: aggregate_results
       depends_on: [scan_ports]
       command: ["python", "merge_scan.py"]
   ```
   This pattern is ideal for security audits or batch data processing.

3. **Integrate External Secrets & Vault Backends**  
   autotask supports HashiCorp Vault and AWS Secrets Manager natively. Reference secrets in your workflow without exposing them in logs:  
   ```yaml
   tasks:
     - id: deploy_prod
       secrets:
         DB_PASSWORD: vault://secret/db#password
         API_KEY: aws://arn:aws:secretsmanager:us-east-1:123456789:secret:api-key
       command: ["deploy.sh"]
   ```
   The engine resolves secrets at runtime and masks them in output streams.

### Best Practices

- **Use Idempotency Keys for Critical Transactions** — When autotask retries a failed task, ensure the underlying operation is safe to repeat. For database migrations, use migration version checks; for API calls, include an idempotency token in the request header. autotask can generate these tokens via the `$IDEMPOTENCY_KEY` built-in variable.
- **Monitor Task Latency with Built-in Metrics Export** — Enable Prometheus metrics export by setting `metrics.enabled: true` in the autotask config. Expose task duration, retry counts, and circuit breaker state. This is critical for production pipelines where a single slow task can block downstream jobs. Alert on P99 latency exceeding your SLA.
