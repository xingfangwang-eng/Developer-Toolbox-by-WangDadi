# autotask - AUT_005

## Overview

autotask is a production-grade automation engine engineered for engineers who need to eliminate repetitive workflows without compromising control. It combines declarative task orchestration, contextual state persistence, and multi-agent dispatch into a single, lightweight binary. No DSL overhead. No cloud dependency. Just a CLI that scales from a single dev machine to a distributed runner cluster.

## What is autotask?

autotask is a task-centric automation framework that treats every operation as a stateful, observable unit. It decouples task definitions from execution logic, allowing you to define complex pipelines in YAML or JSON, then run them locally, via CI/CD, or through a headless daemon. Designed for DevOps engineers, SREs, and platform builders who need deterministic, auditable automation without vendor lock-in.

## Core Features

### 1. Context-Aware Task Graph Engine
Tasks are not just scripts—they are nodes in a directed acyclic graph (DAG) with built-in dependency resolution, conditional branching, and retry policies. Each node retains its execution context (env vars, artifacts, logs) and can be resumed from the last checkpoint. This eliminates the need for external state management or workflow engines.

### 2. Multi-Agent Dispatch with Dynamic Routing
Define task handlers as local scripts, Docker containers, or remote gRPC agents. autotask auto-routes tasks to the appropriate executor based on resource tags, latency targets, or load metrics. Agents can be added or removed at runtime without restarting the orchestrator.

### 3. Immutable Audit Trail with Replay
Every task execution generates a cryptographically signed log that captures input, output, environment snapshot, and timing. You can replay any past task exactly as it ran—down to the same container hash and network conditions. This turns debugging into a deterministic repeatable process.

## How to Use

### 1. Define a multi-step pipeline with conditional gates
Create a file `pipeline.yaml`:
```yaml
tasks:
  - id: build
    image: node:20
    script: npm run build
    outputs:
      - dist/
  - id: test
    depends_on: [build]
    condition: "{{ tasks.build.exit_code == 0 }}"
    script: npm test
  - id: deploy
    depends_on: [test]
    condition: "{{ env.BRANCH == 'main' }}"
    executor: ssh://deploy@prod-server
    script: ./deploy.sh
```
Run with `autotask run pipeline.yaml`. The engine skips `deploy` if the branch is not `main` or if tests fail.

### 2. Attach a remote agent for latency-sensitive tasks
On a worker machine:
```bash
autotask agent --name worker-east --tags region=east,arch=arm64 --listen :9090
```
Back on the orchestrator:
```bash
autotask run --match-tags region=east --match-arch arm64 my-task.yaml
```
The orchestrator automatically routes the task to the correct agent. No manual IP configuration or SSH key management.

### 3. Replay a failed task with exact environment
After a failure:
```bash
autotask inspect --task-id abc123 --show-env
autotask replay --task-id abc123 --override-env DEBUG=true
```
This re-runs the task with the same container, same input, same network simulation—but with `DEBUG=true` injected. All logs are compared side-by-side against the original run.

## Best Practices

### 1. Use idempotent task definitions with deterministic IDs
Always assign explicit `id` fields to tasks, and avoid relying on relative timestamps or random values inside task scripts. autotask uses task IDs for deduplication and replay. If a task is deterministic, you can safely rerun it without side effects. This is critical for production pipelines where partial failures must be retried without corrupting state.

### 2. Pin agent versions and use tags for canary rollouts
When deploying new agent builds, tag them with a version label (e.g., `agent-v2.1.0`) and route only a subset of tasks to that tag. This allows you to validate new executor logic in production without affecting all workflows. Combine with the `--match-tags` flag to gradually shift traffic. Rollback is a single tag change.
