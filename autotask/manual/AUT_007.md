## Overview

autotask is an enterprise-grade automatic task orchestration engine designed for DevOps engineers, SREs, and platform teams who need to eliminate manual toil without sacrificing control. It bridges the gap between cron jobs, CI/CD pipelines, and event-driven automation by providing a declarative, state-aware task graph that executes across distributed environments. Whether you're managing cloud infrastructure, data pipelines, or incident response runbooks, autotask ensures every scheduled or triggered action is idempotent, observable, and auditable—straight out of the box.

## What is autotask?

autotask is a lightweight yet powerful task automation framework that lets you define, schedule, and monitor complex workflows using a single YAML-based DSL. It natively supports dependency resolution, retry policies with exponential backoff, concurrency limits, and real-time status streaming via WebSocket. Unlike traditional task schedulers (like cron or Jenkins), autotask treats every task as a state machine with explicit success, failure, and timeout transitions—making it ideal for mission-critical automation in production environments.

## Core Features

### 1. Declarative Task Graph with Parallel Execution
Define tasks as nodes in a directed acyclic graph (DAG). autotask automatically resolves dependencies and executes independent tasks in parallel, maximizing throughput. Each node can specify custom retry strategies, timeouts, and environment variables—no boilerplate code required.

### 2. Event-Driven Triggers with Webhook & Cron
Combine time-based scheduling with event-driven triggers. Use cron expressions for periodic jobs, or expose a webhook endpoint to kick off tasks on-demand (e.g., from GitHub, Slack, or monitoring alerts). autotask deduplicates incoming events and ensures at-least-once execution semantics.

### 3. Built-in Observability & Alerting
Every task emits structured logs, metrics, and status changes to a configurable sink (e.g., stdout, Loki, Datadog, or a custom webhook). autotask provides a real-time dashboard showing task latency, failure rates, and execution history. You can set up alert rules to notify your team via Slack, PagerDuty, or email when a task fails or exceeds its SLA.

## How to Use

### Step 1: Define Your Task Graph in YAML
Create a `taskfile.yaml` in your project root. Each task has a unique name, a command or script to run, and optional dependencies. Use `depends_on` to create a DAG. Example snippet:

```yaml
tasks:
  build:
    command: "make build"
    timeout: 120s
  test:
    command: "make test"
    depends_on: [build]
    retry: 3
    retry_delay: 10s
  deploy:
    command: "./deploy.sh"
    depends_on: [test]
    env:
      ENV: "production"
```

Run `autotask validate taskfile.yaml` to check for cycles and syntax errors before execution.

### Step 2: Schedule or Trigger Execution
Use the CLI to run the task graph immediately:
```bash
autotask run taskfile.yaml --target deploy
```
Or schedule it via cron syntax:
```bash
autotask schedule taskfile.yaml --cron "0 2 * * *" --timezone UTC
```
To trigger via webhook, start the autotask server:
```bash
autotask serve --port 8080
```
Then send a POST request to `/trigger/taskfile.yaml` with an optional JSON payload. autotask will execute the graph and return a run ID for tracking.

### Step 3: Monitor and Debug in Real-Time
Use the built-in dashboard or CLI to inspect running and completed tasks:
```bash
autotask status --run-id <run_id>
```
Stream logs live with:
```bash
autotask logs --run-id <run_id> --follow
```
For debugging, enable verbose output:
```bash
autotask run taskfile.yaml --verbose --dry-run
```
This prints the resolved DAG and environment variables without executing anything.

## Best Practices

### Use Idempotent Commands in Every Task
Design each task to be safe to run multiple times. For example, use `terraform apply -auto-approve` only after a `terraform plan`, or wrap database migrations in `IF NOT EXISTS` checks. autotask’s retry mechanism will re-execute failed tasks—if your command isn’t idempotent, you risk corrupting state. Always test with `--dry-run` first.

### Separate Configuration from Secrets
Never hardcode API keys or passwords in your `taskfile.yaml`. Instead, use environment variables passed via `--env-file` or a secrets manager like HashiCorp Vault. autotask supports injecting secrets at runtime without exposing them in logs or the dashboard. For example:
```bash
autotask run taskfile.yaml --env-file .env.prod --mask-secrets
```
This keeps your automation secure and audit-friendly.
