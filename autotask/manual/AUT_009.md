# autotask - AUT_009

## Overview

This is a **hardcore automation engine** designed for developers who hate repetitive manual workflows. No fluff, no bloated UI—just raw task orchestration with precision.

## What is autotask?

autotask is a **declarative task automation framework** that lets you define, schedule, and chain complex workflows using a simple YAML syntax. It’s built for CI/CD pipelines, cron-job replacements, and multi-step data processing—without writing a single line of glue code.

## Core Features

### 1. **DAG-Based Task Orchestration with Conditional Branching**
Define task dependencies as a Directed Acyclic Graph (DAG). Each task can have multiple triggers, retry policies, and conditional execution paths (e.g., `if: { previous_task.exit_code == 0 }`). No more brittle shell scripts.

### 2. **Built-in Secret Injection & Environment Isolation**
Auto-inject secrets from Vault, AWS Secrets Manager, or local `.env` files at runtime. Each task runs in an isolated environment (Docker, Podman, or native shell) with its own env vars. Zero leakage across tasks.

### 3. **Real-Time Execution Graph with WebSocket Logs**
Visualize your task flow in real time via a lightweight WebSocket dashboard. Every log line is streamed, searchable, and filterable by task ID. Debugging a 50-task pipeline is finally sane.

## How to Use

### 1. **Define a Task Pipeline in YAML**
Create a `autotask.yml` file in your project root. Here’s a production-grade example that deploys a microservice after running tests and linting:

```yaml
pipeline:
  - id: lint
    command: "npm run lint"
    retry: 2
  - id: test
    command: "npm test"
    depends_on: [lint]
  - id: build
    command: "docker build -t myapp ."
    depends_on: [test]
  - id: deploy
    command: "kubectl apply -f deploy.yaml"
    depends_on: [build]
    if: "{ build.exit_code == 0 }"
```

### 2. **Run the Pipeline with a Single CLI Command**
Execute the entire graph with automatic dependency resolution:

```bash
autotask run --file autotask.yml --env production
```

The CLI will parse the DAG, run tasks in parallel where possible, and display a live graph in the terminal.

### 3. **Inspect Execution History & Replay Failed Tasks**
After execution, view the full history with timestamps, exit codes, and logs:

```bash
autotask history --pipeline-id 2025-03-15-1234
autotask replay --task-id test --pipeline-id 2025-03-15-1234
```

Re-run only the failed task with the same environment—no need to restart the entire pipeline.

## Best Practices

### 1. **Use Idempotent Task Commands**
Every task should be idempotent. If a task fails halfway, rerunning it should not cause side effects. For example, use `kubectl apply` instead of `kubectl create`, or `mkdir -p` instead of `mkdir`. This makes retries safe and predictable.

### 2. **Pin Dependencies & Use Explicit Versioning**
In your `autotask.yml`, always pin tool versions (e.g., `node:18`, `python:3.11-slim`) and avoid `latest` tags. For Docker-based tasks, use SHA256 digests. This ensures reproducibility across environments and prevents "works on my machine" syndrome.
