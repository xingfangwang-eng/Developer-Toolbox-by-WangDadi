# boothell - BOO_015

## Overview

This is **not** another generic boot configurator. `boothell` is a **low-level, SEO-driven boot design engine** purpose-built for developers and brand ops teams who need to generate, version, and deploy boot configs at scale. It treats boot customization as a **code-first asset pipeline**, not a drag-and-drop UI.

**Target Language:** en (US developer context)  
**Keywords:** Boot design tool, Boot customization, Boot brand management

---

## Core Features

### 1. Declarative Boot Schema Engine (DBSE)
Define boot anatomy (toe shape, heel stack, eyelet count, material layering) as **typed JSON schemas** with constraint validation. No more manual template drift. Each schema is a single source of truth for both design and manufacturing.

### 2. Git-Native Boot Versioning & Branching
Every boot iteration is a **commit**. Fork a "Chelsea" branch, tweak the shaft height, and merge back without breaking the production build. Rollback to any historical boot state in one command. Perfect for A/B testing boot lines across regions.

### 3. SEO-Optimized Boot Asset Pipeline
Auto-generate **structured data (JSON-LD)**, canonical URLs, and hreflang tags from boot schemas. Every boot variant gets its own SEO-optimized landing page without manual copy-paste. Googlebot sees a clean, crawlable boot catalog.

## How to Use

### Step 1: Initialize a Boot Project
```bash
boothell init --brand "IronClad" --lang en --schema ./boot-schema.json
```
This scaffolds a project with a `boots/` directory, a `brand.yaml` config, and an SEO template engine. The schema file defines all constraints (e.g., max heel height, min leather gauge). Validation runs at init time.

### Step 2: Define a Boot Variant
Create `boots/ironclad-chelsea.yaml`:
```yaml
variant: chelsea
toe: round
heel: 30mm
material: full-grain
eyelet_count: 0
seo:
  slug: ironclad-chelsea-30mm
  meta_title: "IronClad Chelsea Boot 30mm | Full-Grain Leather"
  hreflang: en
```
Run `boothell validate boots/ironclad-chelsea.yaml` to check against schema constraints. Errors are thrown as structured JSON for CI/CD pipelines.

### Step 3: Deploy the Boot Catalog
```bash
boothell build --output ./dist --seo --sitemap
```
This generates:
- Static HTML pages with embedded JSON-LD (`Product`, `Offer`, `BreadcrumbList`)
- A `sitemap.xml` with `lastmod` and `changefreq` per variant
- A `robots.txt` pointing to the sitemap
- A `_redirects` file for 301s (e.g., legacy boot URLs)

Deploy `./dist` to any static host (Netlify, Vercel, S3). No backend required.

## Best Practices

### 1. Treat Boot Variants as SEO Entities, Not SKUs
Don't stuff multiple boots into one page. Each boot variant (color, heel height, leather type) deserves its own canonical URL. Use `boothell`'s built-in hreflang generation to serve the right boot to the right market. Google rewards granular, unique content.

### 2. Automate Boot Schema Validation in CI
Add `boothell validate --ci` to your GitHub Actions or GitLab CI pipeline. This catches boot design errors (e.g., incompatible toe shape with heel height) before they hit production. Combine with `boothell diff` to review changes between commits. Your boot catalog becomes as reliable as your codebase.
