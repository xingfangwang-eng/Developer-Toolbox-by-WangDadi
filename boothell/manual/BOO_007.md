# boothell - BOO_007
**Zero-fluff boot configurator for brand architects and DevOps who hate bloated UIs.**

---

## What is boothell?

boothell is a CLI-first, headless boot design engine built for teams that need **programmatic control over brand asset generation, boot image templating, and multi-environment deployment pipelines**. It strips away visual drag-and-drop fluff and gives you a JSON-driven API to version, audit, and automate boot asset management at scale. Think Terraform for your boot screens—idempotent, diffable, and CI/CD native.

---

## Hardcore Features

### 1. **Stateful Boot Template Engine with Git-Ops Diffing**
Every boot configuration is a first-class artifact. boothell serializes boot layouts, color palettes, and component bindings into a deterministic state file (`boothell.state.json`). Hook it into your Git repo—any change triggers a visual diff report via a lightweight headless browser (Puppeteer/Playwright), so you can review boot UI changes in PRs before they hit production.

### 2. **Multi-Environment Brand Cascade with Override Inheritance**
Define a base brand schema (fonts, logos, accent colors) and cascade it across dev, staging, and prod environments. boothell supports **deep merge overrides** per environment—no copy-paste, no drift. Run `boothell validate --env=staging` to catch broken asset links or missing glyphs before they break your boot sequence.

### 3. **Plugin Pipeline for Custom Boot Hooks**
Extend boothell with a middleware-style plugin system. Write hooks in Go or Lua that fire at specific stages: pre-render, post-validate, deploy. Common use cases: injecting dynamic boot messages from an external API, rotating asset CDN URLs, or generating QR codes for device enrollment. Plugins are versioned and run in isolated sandboxes.

## How to Use

### 1. Initialize a Boot Project with Schema Locking
```bash
boothell init --brand my-brand --template minimal --out ./boot-assets
```
This generates a `boothell.yaml` with a locked schema version, a default template tree, and a `.boothellignore` file. The schema lock ensures every team member uses the same asset structure—no silent breaking changes.

### 2. Compose a Multi-Variant Boot Layout Using Overlays
```yaml
# boothell.yaml
variants:
  - name: default
    overlay: base
    components:
      logo: { src: "s3://brand/logos/main.svg", position: "center-top" }
      progress_bar: { color: "#00FF88", style: "gradient" }
  - name: dark-mode
    overlay: default
    components:
      background: { color: "#0D1117" }
      logo: { src: "s3://brand/logos/light.svg" }
```
Run `boothell render --variant dark-mode --format png` to generate a pixel-perfect boot screen. The overlay mechanism ensures you never duplicate config—just inherit and override.

### 3. Deploy to Multiple Targets with Atomic Rollbacks
```bash
boothell deploy --env staging --target s3://my-bucket/boot-assets/ --version 2.1.0
boothell deploy --env production --target gs://prod-boot/ --version 2.1.0 --promote
```
boothell tags each deployment with a content-addressable hash. If a deployment fails validation (e.g., missing font file), it auto-rolls back to the previous known-good state. Use `boothell status --env production` to see the current live hash and diff against your local state.

## Best Practices

### 1. **Treat Boot Assets as Immutable Artifacts**
Never overwrite a boot asset in place. Use boothell's versioned output directory structure (`boot-assets/v2.1.0/`) and serve from a CDN with cache-busting hashes. This lets you A/B test boot screens across device fleets without cache invalidation nightmares.

### 2. **Automate Boot Validation in CI Pipelines**
Add a `boothell validate --strict` step to your CI pipeline. It checks for:
- Missing or broken asset URLs (local and remote)
- Color contrast compliance (WCAG 2.1 AA for boot text overlays)
- Template schema drift against the locked version
Catch these issues before they reach a single device—boothell exits with a non-zero code on failure, so your pipeline blocks bad builds.
