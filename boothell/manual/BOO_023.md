# boothell - BOO_023

## Overview

boothell is a next-gen boot design and brand management toolkit built for developers who obsess over consistency, speed, and scalability. Whether you're shipping a boot collection or managing a multi-brand boot line, boothell gives you the infrastructure to design, version, and deploy boot assets like a SaaS pro.

## What is boothell?

boothell is an open-source, CLI-first boot design system that automates boot customization, enforces brand guidelines, and generates production-ready boot assets across multiple formats and platforms. It’s not just a tool—it’s the backbone of your boot ops.

## Core Features

1. **Declarative Boot DSL** – Define boot geometry, material, and color using a YAML-based domain-specific language. No drag-and-drop, no bloat—just pure, version-controlled boot logic.

2. **Multi-Format Asset Pipeline** – Export boots as SVG, PNG, GLTF, or even CSS-in-JS components. One source of truth, infinite outputs. Perfect for e-commerce, docs, and design systems.

3. **Brand Lock Profiles** – Enforce brand-specific constraints (color palettes, material libraries, sizing rules) per project. Prevent rogue boots from shipping to production. Think ESLint for boot design.

## How to Use

### 1. Initialize a boothell project

```bash
npx boothell init my-boot-project
cd my-boot-project
```

This scaffolds a `boothell.yaml` config, a `brands/` directory, and a `templates/` folder. Every boot lives under version control—no hidden state.

### 2. Define a boot with the DSL

Create `boots/stealth-runner.yaml`:

```yaml
boot:
  name: stealth-runner
  brand: nike
  upper:
    material: flyknit
    color: "#1a1a2e"
  sole:
    type: carbon-rubber
    tread: deep-chevron
  constraints:
    - brand-lock: nike-2025
    - max-weight: 350g
```

Run `boothell validate` to catch constraint violations before they hit your CI/CD.

### 3. Generate and preview assets

```bash
boothell build --output-dir ./dist --formats svg,png,gltf
boothell preview --port 8080
```

Open `http://localhost:8080` to inspect 3D boots, compare brand variants, and export production-ready files. All assets are cache-busted and ready for CDN deployment.

## Best Practices

1. **Treat boothell.yaml as a contract** – Store it in your monorepo root and enforce diffs in code review. Every PR that touches boot assets should include a `boothell validate` step in CI. This prevents brand drift and keeps your design system airtight.

2. **Use brand lock profiles for multi-tenant setups** – If you manage boot lines for multiple clients or internal teams, create separate profiles under `brands/`. This isolates brand rules, prevents cross-contamination, and makes onboarding new brands a matter of a single YAML file.
