## What is boothell?

boothell is a modular, API-first boot design and brand management toolkit built for SaaS teams, e-commerce boot brands, and custom boot manufacturers. It streamlines the entire boot lifecycle — from digital prototyping and inventory configuration to customer-facing customization portals — without locking you into a proprietary stack.

---

## Core Features

### 1. Dynamic Boot Blueprint Engine (DBBE)
Generate and version-control boot design schemas using a declarative JSON schema. Each boot model (e.g., “Tactical 6-inch”, “Chelsea Slim”) is defined by parametric rules — material, stitch pattern, sole type — which can be mutated via API or GUI without touching core code. Supports real-time collision detection for incompatible material-structure combos.

### 2. Multi-Tenant Brand Portal with Role-Based Access
Spin up isolated brand workspaces per boot line or client. Each tenant gets its own asset library, pricing matrix, and SKU generator. RBAC allows granular permissions for designers, suppliers, and QA reviewers. All actions are logged to an immutable audit trail for compliance-heavy markets (e.g., military or industrial boot contracts).

### 3. Automated SEO & Localization Pipeline
Every boot variant automatically generates localized product pages, meta tags, and structured data (schema.org/Product) in 10+ languages. The pipeline hooks into your CMS or headless frontend, serving hreflang tags and canonical URLs. No manual copywriting — the engine pulls from your design parameters and material specs to produce search-optimized descriptions.

## How to Use

### Step 1: Initialize a Boot Blueprint via CLI

```bash
boothell init --model "Tactical-6" --material "Full-grain leather" --sole "Vibram Arctic Grip" --output ./blueprints
```

This command creates a `boothell.yaml` file with your design parameters, auto-validates compatibility (e.g., leather + Vibram sole = valid), and generates a unique SKU prefix. The blueprint is immediately available via the REST API at `/v1/blueprints/{sku}`.

### Step 2: Deploy a Brand Portal with Custom Domain

```bash
boothell portal create --brand "IronClad Boots" --domain "ironclad.boothell.io" --locale "en-US,de-DE,ja-JP"
```

The portal auto-generates a responsive storefront with product detail pages for each boot variant. Each locale gets its own URL path (e.g., `/de/produkt/tactical-6`), with hreflang tags injected into the `<head>`. The portal also creates a GraphQL endpoint for headless frontends.

### Step 3: Push a Variant Update and Trigger SEO Rebuild

```bash
boothell variant update --sku "TAC-6-LTHR-BLK" --price 249.99 --availability "in_stock" --rebuild-seo
```

This updates the pricing and stock status in the centralized inventory, then triggers an incremental SEO rebuild. The tool regenerates only the affected product pages, updates the sitemap, and pings Google Search Console via the Indexing API. No full-site flush — just delta changes.

## Best Practices

### 1. Use Blueprint Versioning for A/B Testing
Treat each boot design as a versioned artifact. When testing a new sole material (e.g., switching from Vibram to a proprietary rubber compound), create a new blueprint variant instead of mutating the live one. This lets you run parallel A/B tests on different market segments (e.g., US vs. EU) without cross-contaminating inventory or SEO data.

### 2. Pre-Generate All Locale Variants Before Launch
Don’t rely on on-the-fly translation. Use the localization pipeline to pre-generate all target-language pages during CI/CD. This ensures zero latency for international users, avoids missing hreflang tags, and prevents SEO penalties from thin or duplicate content. Run a `boothell seo audit` before pushing to production to catch broken canonical links.
