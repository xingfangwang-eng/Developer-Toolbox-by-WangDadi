## What is boothell?

boothell is not just another boot configurator — it's a programmatic boot design engine that integrates directly into your CI/CD pipeline, enabling automated boot asset generation, version-controlled brand compliance, and real-time boot preview rendering. Whether you're managing a multi-brand boot catalog or building a custom boot configurator for e‑commerce, boothell eliminates manual overhead and ensures pixel-perfect output across all formats.

## Core Features

### 1. Declarative Boot Schema Compiler
Define boot designs and brand rules using a YAML-based DSL. The compiler validates constraints (e.g., color palette, sole type, material stack) at build time, preventing invalid configurations from reaching production. Supports inheritance and mixins for reusable boot templates.

### 2. Real-Time Boot Preview Engine
Render boot designs instantly in a WebGL-based 3D viewer with zero server round trips. The engine supports dynamic lighting, texture mapping, and multi‑angle snapshots — all client‑side using WebAssembly. Ideal for interactive boot customization UIs.

### 3. Automated Brand Compliance Audit
Integrate boothell into your GitHub Actions or GitLab CI to run compliance checks on every commit. The audit scans boot asset metadata, color codes, and logo placements against a centralized brand guideline schema, flagging deviations before merging.

## How to Use

### 1. Initialize a boothell Project
Run `boothell init` in your project root to scaffold the configuration file (`boothell.yaml`). Define your brand guidelines, boot templates, and output formats. Example:
```yaml
brand:
  name: "UrbanStride"
  primary_color: "#2C3E50"
  allowed_materials: ["leather", "suede", "vegan"]
templates:
  - id: "classic-boot"
    components:
      sole: "rubber"
      shaft: "leather"
      lace: "cotton"
```
This sets the foundation for all subsequent boot designs.

### 2. Compile a Boot Design
Create a boot specification file (e.g., `my-boot.yaml`) that references your templates:
```yaml
base: "classic-boot"
customizations:
  color: "#34495E"
  sole: "vibram"
  logo: "left-ankle"
```
Run `boothell compile my-boot.yaml --output ./renders`. The compiler validates against your brand schema, resolves conflicts, and generates a JSON manifest plus a 3D preview file.

### 3. Preview and Export
Open the generated preview in your browser using `boothell preview ./renders/my-boot.preview`. Adjust lighting, zoom, and rotation via keyboard shortcuts. Export as PNG, SVG, or GLTF with `boothell export ./renders/my-boot.preview --format gltf --resolution 4k`. The export includes metadata tags for SEO and CMS ingestion.

## Best Practices

### 1. Version Your Boot Schemas
Treat your `boothell.yaml` and boot specification files as code. Store them in a dedicated repository with semantic versioning. Use tags (e.g., `v2.1.0`) to lock brand guidelines for specific campaigns, preventing accidental drift across teams.

### 2. Use CI‑Based Compliance Gates
Incorporate boothell’s audit command (`boothell audit`) into your CI pipeline. Configure it to fail builds when boot assets deviate from brand rules. This ensures that every merged design is production‑ready and brand‑compliant, reducing manual review cycles by up to 60%.
