# boothell - BOO_002

## 概要

boothell は、ブートデザインのワークフローを根本から変える、プロフェッショナル向けのブランド管理ツールです。単なるカスタマイズツールではなく、デザインシステム全体をコードベースで一元管理し、ブランドの一貫性を保ちながら、高速なブート生成を実現します。キーワード：ブートデザインツール、ブートカスタマイズ、ブートブランド管理。

## 詳細な内容

### boothell とは？

boothell は、ブートローダー、OS イメージ、ファームウェアのデザインとブランディングを、コードレベルで統制・自動化するための CLI ファーストなツールです。デザイナーとエンジニアの間にある「ブート画面の微調整」という摩擦を解消し、ブランドアセットのバージョン管理、動的テーマ生成、マルチプラットフォームへの一括適用を可能にします。

### コア機能

1.  **動的テーマエンジン (Dynamic Theme Engine)**
    単一の YAML 設定ファイルから、ブートロゴ、プログレスバーの色、フォント、アニメーションのタイミングまでを動的に生成。CSS 変数や SVG スプライトを内部的に最適化し、ビルド時にターゲットプラットフォーム (UEFI, GRUB, カスタムカーネル) に適したアセットを自動出力します。デザイントークンの変更は、全てのブートフェーズに即座に反映されます。

2.  **ブランドアセットパイプライン (Brand Asset Pipeline)**
    SVG、PNG、WebP のロゴやアイコンを入力として、解像度別、色深度別に自動リサイズ・最適化。さらに、ブートローダーが要求する特定のフォーマット（例：BMP 24bit、RAW データ）への変換を、ロスレスかつ高速に行います。アセットのハッシュ化によるキャッシュバスティングも標準搭載。

3.  **マルチターゲットデプロイメント (Multi-Target Deployment)**
    設定ファイル内で `targets` セクションを定義するだけで、同一のブランド設定を、x86_64 UEFI、ARM64 UEFI、Raspberry Pi のファームウェア、さらにはコンテナイメージのブート画面に、一括で適用。各ターゲットの仕様（解像度、色空間、ファイルシステム）に応じた最適なアセットを自動選択し、デプロイします。

### 使い方

1.  **プロジェクトの初期化とブランド設定の定義**
    ```bash
    boothell init --brand "YourBrand" --primary-color "#FF6600"
    ```
    このコマンドで、`boothell.yaml` と `assets/` ディレクトリが生成されます。次に、`boothell.yaml` を編集し、ブランド名、プライマリ/セカンダリカラー、フォントファミリー、ロゴのパスを定義します。`boothell validate` で設定ファイルのスキーマチェックとアセットの存在確認を行い、エラーを事前に検出します。

2.  **カスタムブートアニメーションの作成とテスト**
    `assets/animation/` に SVG フレームを配置し、`boothell.yaml` の `animation` セクションでフレームレートとループ設定を記述します。`boothell preview --target uefi` を実行すると、QEMU 上でブートシーケンスがエミュレートされ、アニメーションの表示とパフォーマンスをリアルタイムで確認できます。アニメーションの各フレームは、内部的にスプライトシートに最適化され、メモリフットプリントを最小化します。

3.  **プロダクションビルドとターゲットへのデプロイ**
    ```bash
    boothell build --target uefi --output-dir ./dist/uefi
    ```
    このコマンドは、定義されたブランド設定とアセットを元に、UEFI ブートローダー用の `bootx64.efi` と、それに埋め込まれるロゴやアニメーションデータを含む `.img` ファイルを生成します。生成されたファイルは、`--output-dir` に出力され、そのままターゲットの EFI システムパーティションにコピーするだけで利用可能です。`--target arm64-uefi` のようにターゲットを変更すれば、自動的にアーキテクチャに適したバイナリが生成されます。

### ベストプラクティス

1.  **デザイントークンを一元管理し、コードレビューの対象とする**
    `boothell.yaml` を Git で管理し、プルリクエストを通じてブランド変更をレビューするプロセスを確立します。これにより、「ロゴの色がちょっと違う」といった、従来は見逃されがちだったブランドの一貫性問題を、コードレビューの段階で検出・修正できます。デザイナーは YAML の変更を提案し、エンジニアはビルドへの影響を確認する、という明確な役割分担が生まれます。

2.  **ビルドパイプラインに boothell を統合し、CI/CD で自動テストする**
    GitHub Actions や GitLab CI などの CI/CD パイプラインに `boothell build` と `boothell validate` を組み込みます。これにより、ブランチにプッシュされるたびに、ブートアセットの整合性チェックとビルドが自動実行されます。ビルドが失敗した場合や、生成されたアセットのハッシュ値が変化した場合にアラートを出すことで、ブランドの劣化を未然に防ぎます。さらに、`boothell diff` コマンドを使って、2つのビルド間で生成されたアセットの差分を視覚的に確認することも可能です。
