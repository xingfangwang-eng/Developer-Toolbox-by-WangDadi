# boothell - BOO_021

## Overview

This is a no-bullshit, production-grade toolkit for **boot design, customization, and brand management**. Built for developers who hate bloated CMSs and need a lean, API-first solution to manage boot configs, generate dynamic assets, and enforce brand consistency at scale.

**Target audience:** US-based SaaS architects, DevOps engineers, and frontend-heavy teams shipping boot-related products or services.

## What is boothell?

boothell is a **headless boot configuration engine** that lets you define, version, and serve boot design tokens (colors, materials, stitching patterns, lace types) via a RESTful API. It replaces spreadsheets, static JSON files, and fragile YAML pipelines with a single source of truth. Think of it as a **design token server for boots**, not a UI builder.

## Core Features

1. **Declarative Boot Schema Engine**  
   Define boot models as JSON Schema (draft-07). boothell automatically generates validation rules, default values, and a REST endpoint for each model. No more hand-rolling CRUD for every boot variant.

2. **Real-Time Brand Locking with GitOps**  
   Enforce brand guidelines (e.g., “only these 3 leather finishes”, “heel height must be between 2-4 inches”) via a `brand_policy.yaml` file in your repo. boothell watches for changes and rejects any API call that violates policy. Merge request = brand update.

3. **Multi-Format Asset Generation Pipeline**  
   On config change, boothell can trigger a serverless function to generate SVG previews, 3D model placeholders, or PDF spec sheets. All via webhooks. No manual export, no stale assets.

## How to Use

### 1. Define a Boot Model with JSON Schema

Create a file `models/combat_boot.json`:

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "properties": {
    "leather_type": { "type": "string", "enum": ["full-grain", "suede", "nubuck"] },
    "heel_height": { "type": "number", "minimum": 1.5, "maximum": 4.0 },
    "lace_system": { "type": "string", "enum": ["speed", "traditional", "boa"] }
  },
  "required": ["leather_type", "heel_height"]
}
```

Then register it via the CLI:

```bash
boothell models register combat_boot --file models/combat_boot.json
```

boothell automatically creates a live endpoint at `/api/v1/models/combat_boot`.

### 2. Enforce Brand Policy with a YAML Lockfile

In your repo root, add `brand_policy.yaml`:

```yaml
models:
  combat_boot:
    allowed_leather: ["full-grain"]
    heel_height_range: [2.0, 3.5]
    lace_system: ["speed", "traditional"]
```

Push to `main`. boothell’s webhook listens for `refs/heads/main` and applies the policy. Any `POST` or `PATCH` to the model endpoint will now be validated against this policy in real time.

### 3. Wire Up Asset Generation via Webhook

In your boothell dashboard, add a webhook target:

```
POST https://us-east-1.aws.webhooks.io/boothell-render
```

Payload example:

```json
{
  "event": "config.created",
  "model": "combat_boot",
  "data": {
    "leather_type": "full-grain",
    "heel_height": 3.0,
    "lace_system": "speed"
  }
}
```

Your serverless function receives this, generates an SVG preview, and posts it back to boothell’s asset endpoint. The preview is then available at `/api/v1/models/combat_boot/assets/preview.svg`.

## Best Practices

- **Treat boothell as a design token server, not a config store.**  
  Keep business logic (pricing, inventory) in your own services. boothell is for *design constraint management* only. This keeps your API surface clean and your CI/CD pipeline fast.

- **Version your brand policies like code.**  
  Use semantic versioning for `brand_policy.yaml`. Tag releases in Git. Rollback a policy change with `git revert` and a re-push. This gives you audit trail and disaster recovery without any extra tooling.
