## What is boothell?

boothell is a production-grade, multi-language SaaS engine built for the boot industry — from independent designers to global footwear brands. It abstracts the entire boot design lifecycle into modular, API-first components, enabling real-time 3D customization, automated pattern generation, and centralized brand asset governance. Think of it as the **Figma for boots**, but with manufacturing-grade constraints and export pipelines.

## Core Features

### 1. Parametric Boot Geometry Engine (PBGE)
A constraint-based 3D geometry kernel that allows designers to define boot parameters — shaft height, toe shape, heel angle, material stretch — as tunable variables. Changes propagate instantly across the model, with automatic collision detection and manufacturing tolerance checks. Supports STEP, OBJ, and GLB export with embedded BOM metadata.

### 2. Multi-Tenant Brand Asset Vault
A version-controlled, role-based repository for boot design assets: lasts, patterns, material swatches, and stitching maps. Each brand gets an isolated namespace with immutable audit trails, granular permission scopes, and automated compliance checks against regional sizing standards (EU, US, JP, MX). Integrates with PLM tools via GraphQL API.

### 3. Real-Time Customization Engine with AR Preview
A WebGL-based configurator that renders boot models in-browser with live material swaps, colorways, and embossing. Outputs a unique SKU hash and generates downloadable spec sheets (PDF, JSON) with cost estimates. Optional AR mode (WebXR) projects the boot onto the user's foot via smartphone camera for fit validation.

## How to Use

### Step 1: Define Boot Geometry Constraints
Initialize a new project via CLI or dashboard. Use the PBGE schema to set design boundaries:
```bash
boothell init --project "MX-2025" --last "cowboy_eu42" --shaft-min 30cm --shaft-max 45cm
```
The engine validates against the selected last's digital twin and generates a constraint graph. All parameters are stored as JSON in the project manifest.

### Step 2: Configure Brand Asset Pipeline
Upload your brand's material library (PBR textures, leather grades, sole molds) into the vault. Assign metadata tags (e.g., `water_resistant`, `vegan`, `certified_organic`) and set lifecycle statuses (`prototype`, `production`, `archived`). Use the built-in diff tool to compare material versions:
```graphql
mutation {
  uploadMaterial(input: {brandId: "heritage_works", file: "croc_emboss_v3.zip", tags: ["luxury", "limited"]}) {
    assetId
    checksum
  }
}
```

### Step 3: Generate and Export Custom SKU
Launch the customization UI — either embedded in your e-commerce site or as a standalone page. Users adjust parameters (color, stitching, hardware finish). Each configuration generates a unique SKU hash and a production-ready spec sheet. Export via API:
```bash
curl -X POST https://api.boothell.io/v1/skus/generate \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"projectId": "MX-2025", "config": {"colorway": "coyote_brown", "sole": "vibram_arctic", "embossing": "monogram_v2"}}'
```
Returns a JSON payload with cost breakdown, estimated lead time, and a downloadable PDF spec sheet.

## Best Practices

### 1. Use Parametric Templates for Seasonal Drops
Instead of designing each boot from scratch, create parametric templates that define the design space (e.g., "winter collection 2025"). Lock non-negotiable parameters (e.g., sole type for traction) and expose only style-relevant variables (color, lining material). This reduces iteration time by 60% and ensures brand consistency across SKUs.

### 2. Enforce Asset Lifecycle Policies in the Vault
Set up automated retention rules in the Brand Asset Vault: mark materials as `deprecated` after 12 months of inactivity, and archive them after 24. Use the built-in compliance scanner to flag assets that violate regional standards (e.g., REACH for EU, CPSC for US). This prevents production errors and legal exposure when scaling to new markets.
