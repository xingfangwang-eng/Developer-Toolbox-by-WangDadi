# boothell - BOO_028

## Overview

This is a no-bullshit, production-grade boot design and brand management toolkit built for developers who hate wasting time on repetitive UI scaffolding and brand inconsistency. `boothell` automates boot-level customization, enforces brand rules at the component layer, and ships a CLI that speaks your stack. No drag-and-drop fluff. No bloated dashboards. Just raw, composable control over your design system.

Target language: **English (US)** – built for the global dev community, with a hard focus on performance, modularity, and CI/CD integration.

---

## What is boothell?

`boothell` is a CLI-first, API-driven boot design tool that lets you define, version, and deploy boot-level design tokens, component blueprints, and brand compliance rules across multiple projects. Think of it as a design system compiler with a built-in linter and a dead-simple plugin architecture. It’s not a toy. It’s the missing piece between your design tokens and your production codebase.

---

## Core Features

### 1. Token-Aware Boot Compilation
Define boot-level design tokens (spacing, typography, color palettes, breakpoints) in a single YAML file. `boothell` compiles them into framework-specific output (Tailwind, Bootstrap, Chakra, custom CSS-in-JS) with zero runtime overhead. Supports inheritance, overrides, and conditional exports per environment.

### 2. Brand Rule Enforcement Engine
Inject brand compliance rules directly into your component pipeline. The engine scans every boot variant, checks contrast ratios, spacing consistency, and naming conventions, and fails the build if a rule is violated. No more “design drift” across teams. Works with any component library.

### 3. Plugin-Based CI/CD Integration
Extend `boothell` with custom plugins for Figma token export, Storybook sync, or even Slack notifications on brand violations. The plugin API is written in TypeScript, fully typed, and supports lifecycle hooks (pre-build, post-build, error). Drop it into your GitHub Actions or GitLab CI pipeline in under 10 lines.

## How to Use

### 1. Initialize a Boot Project
```bash
npx boothell init --framework tailwind --output ./boot-tokens
```
This scaffolds a `boothell.config.ts` file, a `tokens/` directory with a default YAML schema, and a `.boothellignore` for token exclusions. The CLI auto-detects your package manager and framework version. No prompts. No bloat.

### 2. Define Tokens and Brand Rules
Edit `tokens/brand.yaml`:
```yaml
brand:
  primary: "#1A73E8"
  spacing: [4, 8, 16, 24, 32]
  typography:
    heading: { font: "Inter", weight: 700, scale: "1.25" }
rules:
  contrast: { min: 4.5 }
  naming: { pattern: "^boot-", enforce: true }
```
Run `boothell validate` to check for conflicts, missing tokens, or rule violations. The output is JSON – pipe it into your linter or CI.

### 3. Compile and Export
```bash
boothell build --env production --format css-variables --output ./dist/boot.css
```
This generates a single `boot.css` file with all tokens as CSS custom properties, plus a `boot-tokens.json` for programmatic access. The build step also runs the rule engine and exits with code 1 on failure. Integrate it into your `package.json` build script:
```json
"build:design": "boothell build && next build"
```

## Best Practices

### 1. Version Your Tokens Like Code
Treat your `tokens/` directory as a first-class artifact. Commit it. Tag it. Use semantic versioning for breaking changes. `boothell` supports a `--version` flag that writes a `boothell.lock` file – include it in your repo to ensure deterministic builds across environments.

### 2. Automate Brand Audits in CI
Add a `boothell audit` step to your CI pipeline that runs on every PR. It checks for token drift, missing brand rules, and deprecated values. Fail the build if the audit fails. This prevents design debt from accumulating and keeps your boot system consistent across micro-frontends.
