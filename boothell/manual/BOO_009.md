## Detailed Content

### What is boothell?

Boothell is not just another boot configurator. It's a CLI-first, API-driven platform that lets you define, version, and deploy boot-level branding and customization logic as code. Think of it as the missing layer between your bootloader and your brand identity system — enabling deterministic builds, audit trails, and multi-environment orchestration without fragile shell scripts or hardcoded assets.

### Core Features

1. **Deterministic Boot Asset Pipeline**  
   Boothell compiles boot splash screens, firmware logos, and UEFI branding assets into a single, hash-verified binary blob. Every build produces an identical output for identical inputs — no surprises, no cache poisoning, no "works on my machine."

2. **Multi-Tenant Boot Configuration Engine**  
   Manage boot configurations for hundreds of devices or SKUs from a single YAML/JSON schema. Boothell supports inheritance, overrides, and conditional rules (e.g., "use dark theme for production, light theme for staging") without duplicating config files.

3. **Real-Time Boot Brand Validation**  
   Integrated linting and validation rules check for resolution mismatches, color space violations, and firmware compatibility issues before deployment. Boothell rejects invalid assets at commit time, not at boot time.

### How to Use

1. **Initialize a Boothell Project**  
   ```bash
   boothell init --template minimal --output ./my-boot-project
   ```  
   This creates a structured project directory with a `boothell.yaml` manifest, an `assets/` folder, and a default `brand-rules.json`. The manifest defines your target firmware versions, branding constraints, and output formats (e.g., `.bmp`, `.raw`, `.efi`).

2. **Define Brand Assets with Validation Rules**  
   ```yaml
   # boothell.yaml
   version: "2.0"
   branding:
     logo:
       path: "./assets/logo.png"
       min_width: 800
       color_space: "sRGB"
       allowed_formats: ["png", "bmp"]
     splash:
       path: "./assets/splash.png"
       max_size_kb: 256
       overlay_text: "BOOTHELL v2.0"
   ```  
   Run `boothell validate` to check all assets against the rules. Boothell outputs a detailed report with line-level errors and suggestions (e.g., "Logo width 640px < min 800px").

3. **Build and Deploy Boot Artifacts**  
   ```bash
   boothell build --env production --output ./dist
   boothell deploy --target uefi --device-id "SKU-2025" --artifact ./dist/boot-blob.bin
   ```  
   The `build` command generates a single binary blob and a checksum file. `deploy` pushes the artifact to a remote boot server or directly to a device's EFI partition via SSH or a REST API endpoint.

### Best Practices

1. **Version Your Boot Branding Like Code**  
   Treat `boothell.yaml` and all asset files as first-class citizens in your Git repository. Use semantic versioning for branding releases and tag each build with the corresponding Git commit hash. This allows you to roll back boot branding independently of firmware updates.

2. **Integrate Boothell into CI/CD Pipelines**  
   Add `boothell validate` and `boothell build` as stages in your GitHub Actions or GitLab CI. This catches branding regressions early and ensures that only validated boot blobs reach production devices. For multi-tenant setups, use environment-specific overrides to test branding changes in staging before rolling out globally.
