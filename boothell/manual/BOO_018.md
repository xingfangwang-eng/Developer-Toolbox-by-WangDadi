# boothell - BOO_018

## Overview

This is a hardened, production-grade toolchain for **boot design, customization, and brand-level boot management**. Built for developers who treat boot configuration as code, boothell eliminates guesswork and manual overhead in multi-environment boot setups.

## What is boothell?

boothell is a CLI-first, API-extensible framework that automates boot asset generation, version-controlled branding injection, and cross-platform boot validation. It’s not a GUI toy—it’s a DevOps-native boot engine for teams shipping firmware, embedded systems, or custom Linux/BSD boots at scale.

## Core Features

### 1. Declarative Boot Manifest (bootfile.yaml)
Define your entire boot chain—kernel params, initramfs hooks, splash assets, and bootloader flags—in a single YAML manifest. boothell compiles this into platform-specific boot images (GRUB, systemd-boot, UEFI, or bare‑metal) with zero drift.

### 2. Brand Injection Pipeline with Checksum Locking
Inject logos, boot splash sequences, and OEM metadata directly into the boot binary. Each injection is verified via SHA‑256 checksum locking, ensuring that your brand assets are tamper‑proof across CI/CD stages.

### 3. Multi‑Boot Validation & Rollback Orchestration
boothell can simulate boot sequences for 12+ architectures (x86_64, ARM64, RISC‑V) in a sandboxed QEMU environment. It compares boot logs against a golden reference and triggers automatic rollback to the last known‑good boot manifest if anomalies are detected.

## How to Use

### 1. Initialize a Boot Manifest
```bash
boothell init --target uefi --kernel vmlinuz-6.8 --initrd initrd.img
```
This generates a `bootfile.yaml` with auto‑detected boot parameters. Edit it to add custom kernel cmdline args, splash screen paths, or brand variables.

### 2. Inject Brand Assets & Lock Checksums
```bash
boothell brand inject --splash ./assets/splash.bmp --oem-id "MYCORP_2025" --lock
```
boothell hashes the injected assets, appends the checksum to the boot header, and outputs a `boot.signed` binary. Use `boothell brand verify` in your CI pipeline to ensure no asset has been silently swapped.

### 3. Run Multi‑Arch Validation with Rollback
```bash
boothell validate --arch arm64,riscv64 --golden-ref ./bootlogs/golden.log --rollback
```
boothell boots each target architecture in QEMU, captures boot logs, diffs them against the golden reference, and automatically reverts to the previous manifest if any boot stage deviates by more than 0.5% timing variance.

## Best Practices

### 1. Version‑Control Your bootfile.yaml
Store `bootfile.yaml` in the same repo as your kernel source. Tag boot manifests with release versions so you can trace any boot regression to a specific asset or parameter change. Never edit boot images manually—always regenerate from the manifest.

### 2. Always Run Validation Before Flashing
Even for a single‑architecture deployment, run `boothell validate --arch $(uname -m)` in your pre‑flash CI stage. This catches bootloader misconfigurations (e.g., missing `initrd` path, incorrect root partition UUID) before they brick a device.
