# boothell - BOO_005

## Overview

boothell is a no-BS bootstrapping toolkit built for engineers who hate wasting time on boilerplate, branding, and template hell. It’s not a UI toy. It’s a CLI-first, multi-language boot factory that generates production-ready boot configurations, brand skeletons, and design tokens in seconds. If you're tired of rewriting the same `_boot.scss`, `tailwind.config.js`, or `brand.json` across projects, boothell is your new co-pilot.

## Core Features

### 1. Brand-Aware Boot Generator
boothell doesn’t just spit out generic CSS. It parses your brand identity (colors, typography, spacing) from a single `brand.yaml` and generates a fully typed, tree-shakable boot layer for **React, Vue, Svelte, and static HTML**. No more manual theming across stacks.

### 2. Multi-Language Token Engine
Supports **10+ languages** for design token output — Tailwind, Chakra, MUI, Bootstrap, PrimeNG, and even raw CSS custom properties. boothell auto-translates your design system into the framework’s native syntax. One source of truth, zero copy-paste.

### 3. CLI-Born CI/CD Integration
boothell ships with a `boothell build` command that outputs a `boot.json` manifest. Hook it into your CI pipeline, and every deploy auto-generates a fresh boot config based on your latest brand tokens. No manual versioning. No stale styles.

## How to Use

### 1. Initialize a Boot Project
```bash
npx boothell init --lang en --brand ./brand.yaml
```
This scaffolds a `boothell/` directory with a `boot.config.ts`, a `brand.yaml` template, and a `locales/` folder for multi-language support. The `--lang en` flag sets the default output language to English, but you can override it per component.

### 2. Generate Framework-Specific Boot Tokens
```bash
boothell generate --target tailwind --output ./theme/tokens.css
```
This reads your `brand.yaml`, maps colors to Tailwind’s `colors` object, spacing to `spacing`, and typography to `fontFamily`. The output is a fully typed `tokens.css` file that you can import directly into your `tailwind.config.js`. For React, use `--target chakra` to get a `theme.ts` file with `extendTheme()` ready to go.

### 3. Validate and Diff Boot Configs
```bash
boothell validate --manifest ./boot.json --baseline ./boot.baseline.json
```
This compares your current boot manifest against a baseline. It flags breaking changes (e.g., color palette shifts, spacing scale changes) and outputs a human-readable diff. Use it in CI to prevent accidental brand drift across environments.

## Best Practices

### 1. Treat `brand.yaml` as a Single Source of Truth
Never hardcode brand values in multiple places. Store everything in `brand.yaml`, and let boothell propagate changes across all your projects. This ensures consistency across your landing page, admin panel, and mobile app.

### 2. Version Your Boot Manifests
Include `boot.json` in your version control. When you update brand tokens, commit the new manifest alongside the code changes. This makes rollbacks trivial and gives your team a clear audit trail of brand evolution.
