# boothell - BOO_017

## Overview

boothell is a high-performance, opinionated boot design and brand management toolkit engineered for SaaS teams that obsess over consistency. It strips away the boilerplate from boot-level UI customization and enforces a strict, scalable design token architecture. Whether you're shipping a white-label dashboard or a multi-tenant brand system, boothell gives you the guardrails to move fast without breaking the visual contract.

## What is boothell?

boothell is a CLI-driven, design-token-first boot customization engine. It automates the generation of brand-compliant Bootstrap variants, enforces semantic color mapping, and bakes in responsive breakpoint logic. Think of it as a design system compiler for boot frameworks—purpose-built for teams managing multiple brand identities under a single codebase.

## Core Features

1. **Semantic Token Mapping Engine**  
   boothell parses your brand JSON (colors, typography, spacing) and auto-generates a full set of CSS custom properties and SCSS maps. It enforces a strict naming convention (e.g., `--boot-primary-500`, `--boot-surface-elevated`) that eliminates guesswork across your component library.

2. **Multi-Tenant Boot Compiler**  
   Ship one codebase, serve infinite brands. boothell compiles separate `bootstrap.css` variants per tenant, each with its own theme, breakpoints, and component overrides. No runtime bloat, no CSS collisions—just pure, compiled output per deployment.

3. **Boot Compliance Linter**  
   Integrates directly into your CI pipeline. boothell scans your compiled CSS and JSX for violations against your defined token schema. It flags hardcoded colors, mismatched spacing, and orphaned breakpoints before they hit production. Think ESLint, but for boot-level design integrity.

## How to Use

1. **Initialize a boothell project**  
   Run `npx boothell init --brand ./brand-config.json` in your project root. This generates a `.boothellrc` configuration file and a `_boot-tokens.scss` partial. The CLI will validate your brand config against the required schema—if your JSON is missing a `primary-500` or `font-family-base`, it will fail immediately with a descriptive error.

2. **Compile a tenant-specific build**  
   Execute `npx boothell build --tenant acme-corp --output ./dist/acme` to generate a fully compiled Bootstrap CSS variant. boothell will resolve all token overrides, apply responsive breakpoint aliases (e.g., `--bp-tablet: 768px`), and output a minified, sourcemapped file. The compiler also generates a `_variables.scss` for downstream component libraries.

3. **Run the compliance audit**  
   Integrate `npx boothell lint --dir ./dist` into your pre-commit hook or CI step. The linter will scan every `.css` and `.jsx` file in the target directory, flagging any rule violations. For example, if a component uses `color: #333` instead of `var(--boot-text-primary)`, it will output the file path, line number, and suggested token replacement.

## Best Practices

1. **Treat your brand config as a single source of truth.**  
   Store your `brand-config.json` in a version-controlled monorepo and use boothell's CI linter to enforce it across all micro-frontends. This prevents drift between your design system and the actual compiled output. For teams managing 5+ brands, this alone reduces visual regression bugs by ~40%.

2. **Use boothell's multi-tenant compiler for staging and production parity.**  
   Never rely on runtime theme switching for production. Instead, compile separate CSS bundles per tenant during your build step. This eliminates the risk of a broken theme toggle affecting all users and allows you to cache each brand's CSS independently via CDN. boothell's output is deterministic—same input, same output, every time.
