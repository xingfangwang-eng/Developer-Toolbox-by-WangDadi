## What is boothell?

**boothell** is a high-performance, multi-language boot customization and firmware flashing toolkit designed for DevOps engineers, embedded systems architects, and white-label boot brand managers. It abstracts away the complexity of bootloader patching, boot image signing, and multi-device OTA compliance, enabling you to ship branded boots across 10+ hardware targets with a single CLI command.

---

## Core Features

### 1. Cross-Architecture Boot Image Compiler
Automatically compiles and signs boot images for ARM, x86, and RISC-V architectures from a single YAML manifest. Supports **AVB (Android Verified Boot)**, **UEFI Secure Boot**, and **custom PKI chains** without manual key management.

### 2. Real-Time Boot Telemetry & Rollback Engine
Embed a lightweight telemetry agent into the boot process to monitor boot times, kernel panic rates, and partition corruption. Automatically triggers rollback to a golden image if boot failure exceeds a configurable threshold (e.g., 3 consecutive panics).

### 3. Multi-Tenant Brand Injection with CRC Validation
Inject brand assets (splash screens, boot animations, vendor strings) into boot partitions across thousands of devices simultaneously. Each injection is CRC32-validated and checksum-locked to prevent bricking. Supports **A/B slot switching** for zero-downtime branding.

## How to Use

### 1. Initialize a Boot Project with a Custom Brand Manifest
```bash
boothell init --brand my_brand --arch arm64 --target rpi4
```
This creates a `boothell.yaml` file with default boot image parameters, slot layout, and signing keys. You then edit the manifest to point to your `splash.bmp` and `boot_animation.zip`.

### 2. Compile and Sign a Boot Image with Hardware-Backed HSM
```bash
boothell build --manifest boothell.yaml --hsm pkcs11:///slot0 --output boot.img
```
The tool connects to a hardware security module (HSM) via PKCS#11, signs the boot image with your production key, and embeds the certificate chain into the boot header. The output `boot.img` is ready for flashing.

### 3. Deploy Brand to a Fleet of Devices via OTA Payload
```bash
boothell deploy --image boot.img --target 192.168.1.0/24 --slot b --rollback-threshold 3
```
This pushes the signed boot image to all devices in the subnet, writes it to slot B, and sets the bootloader to attempt slot B first. If the device fails to boot 3 times, it automatically falls back to slot A.

## Best Practices

### 1. Always Test Brand Injection on a Staging Slot Before Production
Use the `--slot staging` flag during deployment to write to a non-bootable partition. Validate boot behavior with `boothell verify --slot staging` before promoting to slot A or B. This prevents fleet-wide bricking from malformed boot animations or incorrect partition sizes.

### 2. Automate Boot Image Versioning with Git Tags
Integrate `boothell build` into your CI/CD pipeline and tag each build with a semantic version (e.g., `v2.1.3-rc1`). The tool embeds the Git commit hash and timestamp into the boot image metadata. This enables precise rollback targeting: `boothell rollback --target-version v2.1.2`.
