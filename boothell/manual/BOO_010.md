### What is boothell?

**boothell** is a command-line-first boot configuration and brand management framework engineered for developers and DevOps teams who need to automate, version-control, and scale boot asset pipelines.  
It eliminates manual boot design fragmentation by treating every boot parameter — from color palettes to boot-loader metadata — as code.

---

### Core Features

1. **Declarative Boot Manifest (DBM)**  
   Define your entire boot identity in a single YAML file. boothell compiles it into platform-specific boot assets (GRUB, systemd-boot, UEFI stubs) with zero runtime overhead.

2. **Multi-Tenant Brand Injection Engine**  
   Inject per-environment boot branding (dev, staging, prod) without touching your kernel or initramfs. Supports dynamic logo, boot-splash, and boot-entry text substitution via template variables.

3. **Boot Asset Diff & Audit Trail**  
   Every boot configuration change is tracked as a git-compatible diff. boothell generates a cryptographically signed audit log for compliance-heavy environments (PCI-DSS, FedRAMP).

---

### How to Use

1. **Initialize a Boot Manifest**  
   Run `boothell init --manifest boothell.yaml` to scaffold a minimal boot configuration. The manifest includes fields like `boot.entry.label`, `boot.splash.image`, and `boot.timeout`. This replaces manual editing of `/etc/default/grub` or `loader.conf`.

2. **Compile & Validate Across Targets**  
   Execute `boothell build --target uefi --env production`. The tool validates your manifest against a schema, resolves template variables, and outputs a hardened boot configuration. Use `boothell validate --strict` to catch boot-time regressions before deployment.

3. **Deploy with Rollback Support**  
   Run `boothell deploy --rollback 3` to push the boot configuration to the target system. The command automatically backs up the previous state and retains up to 3 rollback snapshots. To revert: `boothell rollback --snapshot 2`.

---

### Best Practices

- **Version your boot manifest in the same repo as your infrastructure-as-code**. This ensures boot configuration changes are reviewed, tested, and deployed alongside your stack. Use `boothell diff` in CI to flag unintended boot changes.  
- **Use environment-specific overlays** instead of duplicating manifests. Define a base `boothell.yaml` and override only the fields that differ per environment (e.g., boot splash for staging vs. production). boothell merges these automatically during `build`.
