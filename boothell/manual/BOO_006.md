### What is boothell?

boothell is not just another boot configurator. It's a **design-to-deployment pipeline** for boot brands that need to manage complex SKU hierarchies, real-time material constraints, and multi-language product sheets. Think of it as the missing layer between your CAD files and your Shopify storefront. It handles the dirty work: variant explosion, rule-based pricing engines, and automated SEO metadata injection for every boot silhouette you ship.

### Core Features

- **Constraint-Driven Variant Engine**  
  Define material compatibility matrices (e.g., "Horween leather cannot pair with Vibram 430 sole") and boothell auto-generates only valid SKUs. No more manual pruning of impossible combinations. This reduces catalog bloat and prevents customer checkout errors.

- **Dynamic SEO Metadata Injection**  
  Every boot variant gets a unique, algorithmically-generated title, meta description, and alt text based on its material, color, and last shape. boothell hooks directly into your CMS, ensuring each page competes for long-tail keywords (e.g., "hand-stitched horsehide engineer boot with brass hardware").

- **Multi-Locale Boot Terminology Normalizer**  
  Boot names vary wildly across markets (e.g., "chukka" vs. "desert boot" vs. "paddock boot"). boothell maps your internal taxonomy to local search intent in 10+ languages, auto-generating localized product names and descriptions without breaking your backend schema.

### How to Use

1. **Initialize the Boot Schema**  
   Create a `boothell.yaml` file in your project root. Define your base boot structure (toe shape, heel height, sole type, leather grade) as a JSON Schema. Then run `boothell init --schema ./boothell.yaml`. This generates a SQLite database with all constraint rules and a `variants/` folder containing empty product stubs.

2. **Run the Variant Explosion**  
   Execute `boothell expand --input ./materials.csv --output ./variants`. The engine cross-references your material CSV (columns: `leather_id, sole_id, hardware_finish, lining_material`) against the constraint matrix. Only valid combinations survive. Each variant is output as a Markdown frontmatter file, pre-populated with SEO fields. Example output: `boothell-variant-001.md` with `title: "Black CXL Service Boot - Brass Eyes - Vibram 430"`.

3. **Deploy with SEO Injection**  
   Use `boothell deploy --source ./variants --target ./public --language en`. This command generates static HTML pages for each variant, injects canonical URLs, and writes a `sitemap.xml` with priority scores based on variant popularity (calculated from your analytics CSV). The tool also auto-generates `hreflang` tags if you pass `--language ja` for a Japanese market deployment.

### Best Practices

- **Use boothell as a CI/CD Gate**  
  Integrate `boothell validate` into your pre-commit hook. Every time a designer updates the materials CSV, the hook checks for constraint violations (e.g., "exotic leather + rubber sole" flagged as incompatible). This prevents broken variants from reaching staging. Pair it with `boothell diff` to compare two variant sets and spot regressions.

- **Leverage the SEO Preview Mode**  
  Before deploying, run `boothell preview --variant boothell-variant-001.md`. This opens a local HTML preview showing exactly what Google will index: title tag, meta description, and image alt text. Tweak your keyword density in the `boothell.yaml` `seo_keywords` field and re-run preview to see real-time changes. This avoids the "deploy and pray" loop.
