## What is boothell?

boothell is a CLI-first, API-driven toolkit that lets you generate, version, and deploy boot configurations at scale. It eliminates manual design bottlenecks by treating boot specs as code — think Terraform for boots, but way cooler.

## Core Features

### 1. BootSpec DSL (Domain-Specific Language)
Write boot designs in YAML or JSON. Define materials, stitching patterns, sole types, and branding elements programmatically. boothell compiles these specs into production-ready CAD files and BOM sheets.

### 2. Multi-Tenant Brand Sync
Manage boot branding across 50+ SKUs simultaneously. Sync logos, color palettes, and sizing matrices across all product variants from a single config file. Supports Git-based version control for rollbacks.

### 3. Real-Time Fit Simulation Engine
Input foot measurements and get instant 3D visualizations of how the boot will fit. Adjust heel height, toe box volume, and arch support dynamically. Outputs sizing recommendations and pattern adjustments.

## How to Use

### Step 1: Initialize a boot project
```bash
boothell init --project my-custom-boots --template rugged-hiker
```
This creates a `boothell.yaml` file with default parameters. Edit it to set sole material (`vibram`, `crepe`, `rubber`), upper leather type (`full-grain`, `suede`, `nubuck`), and lacing style (`speed-hook`, `eyelet`, `mixed`).

### Step 2: Apply branding and sizing rules
```bash
boothell brand --apply --config brand-config.yaml
boothell size --generate --range 7-13 --half-sizes true
```
The brand config injects your logo SVG, embossing depth, and color hex codes. The size generator creates a full size run with proportional scaling for toe shape and shaft height.

### Step 3: Export and deploy
```bash
boothell export --format dxf --output ./production-files
boothell deploy --env production --registry s3://my-boot-designs
```
This outputs DXF files for CNC cutting, a CSV BOM for procurement, and a JSON manifest for your 3D printer farm. The deploy command pushes everything to your cloud registry with version tags.

## Best Practices

### Practice 1: Use GitOps for boot design versioning
Treat every `boothell.yaml` change as a Git commit. This gives you a full audit trail, enables code reviews for design changes, and allows instant rollback if a batch fails QC. Pair with GitHub Actions to auto-generate sample boots on every PR.

### Practice 2: Pre-validate specs with `boothell check`
Before exporting to production, run `boothell check --strict` to validate material compatibility, sizing consistency, and brand asset integrity. This catches issues like logo resolution too low for embossing or sole thickness exceeding manufacturer limits.
