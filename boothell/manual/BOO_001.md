# boothell - BOO_001

## Overview

boothell is a no-BS, developer-first boot design and brand management tool built for teams who treat their boot assets like code. Whether you're iterating on boot silhouettes, managing a multi-brand boot catalog, or automating boot spec generation for manufacturing, boothell gives you version control, visual diffing, and CLI-first workflows out of the box.

---

## What is boothell?

boothell is a modular platform that combines boot design automation, brand asset governance, and SEO-optimized boot content generation. It's designed for footwear designers, brand managers, and e-commerce teams who need to scale boot production without sacrificing consistency or performance.

---

## 3 Hardcore Features

### 1. Boot Spec Diff Engine
Compare two boot design versions (e.g., `boot_v2.3.0.json` vs `boot_v2.4.0.json`) with semantic diffing. Detect changes in toe shape, heel height, material stack, and stitching pattern. Output a clean, machine-readable diff that can be fed into CI/CD pipelines or automated QA.

### 2. Brand Asset Locking with Git Hooks
Lock brand-specific boot parameters (logo placement, color palette, sole texture) into a `.boothellrc` config file. Enforce these rules via pre-commit hooks — no more rogue boot designs slipping into production. Works with any Git-based workflow.

### 3. SEO-Optimized Boot Content Generator
Generate structured boot product descriptions, alt text, and schema markup directly from boot spec files. Supports multilingual output (EN, DE, ES, JA) and integrates with headless CMS platforms via REST API or Webhook.

---

## How to Use

### 1. Initialize a boothell project
```bash
boothell init --brand "Timberland" --model "6-Inch Premium"
```
This creates a `bootspec/` directory with a default `boot.json` template and a `.boothellrc` config file. The template includes required fields: `model_id`, `material_stack`, `last_code`, `stitch_pattern`, and `brand_guidelines`.

### 2. Run a visual diff between two boot versions
```bash
boothell diff bootspec/v2.3.0.json bootspec/v2.4.0.json --output diff.html
```
This generates an interactive HTML diff report. Each changed field is highlighted with a color-coded legend (green for additions, red for removals, yellow for modifications). You can also export as JSON for pipeline integration.

### 3. Generate SEO-ready boot content
```bash
boothell generate --spec bootspec/v2.4.0.json --lang en --output ./dist/boot-content.md
```
This outputs a Markdown file with:
- H1 title (e.g., "Timberland 6-Inch Premium Waterproof Boot")
- Meta description (auto-generated from spec)
- Schema.org `Product` JSON-LD snippet
- Alt text for boot images (e.g., "Side view of Timberland 6-Inch Premium in wheat nubuck")
- Bullet-point feature list optimized for voice search

## Best Practices

### 1. Treat boot specs like API contracts
Define boot specs in strict JSON Schema and validate them before any design or content pipeline. Use boothell's built-in schema validator (`boothell validate`) to catch missing fields, type mismatches, or brand guideline violations. This prevents downstream errors in manufacturing, SEO, and e-commerce feeds.

### 2. Automate boot content generation in CI/CD
Integrate `boothell generate` into your GitHub Actions or GitLab CI pipeline. Trigger it on every merge to `main` to regenerate SEO content automatically. Combine with a diff check to only regenerate when boot specs change — saves build time and keeps your product pages fresh.
