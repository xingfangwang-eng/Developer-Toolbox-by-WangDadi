## What is boothell?

boothell is a high-performance, API-first boot design and brand management platform built for developers who need to generate, version, and deploy boot visual assets at scale. Unlike static design tools, boothell treats every boot as a programmable entity — enabling real-time customization, multi-locale rendering, and automated brand compliance checks across your entire product catalog.

---

## Core Features

### 1. Declarative Boot Schema Engine
Define boot geometry, materials, and color palettes as JSON schemas. The engine validates constraints (e.g., heel height vs. sole thickness) and auto-generates optimized SVG/PNG assets with sub-10ms latency. Supports inheritance and overrides for brand families.

### 2. Multi-Tenant Brand Locking via GitOps
Each brand gets a dedicated branch in the boothell config repo. Changes to brand guidelines (logos, stitching patterns, sole gradients) are PR-reviewed and merged into production. The system enforces strict validation rules — any boot asset that violates brand constraints is flagged before deployment.

### 3. Real-Time Collaboration with Conflict Resolution
Multiple designers or engineers can edit the same boot spec simultaneously. The built-in CRDT-based engine auto-merges changes and highlights conflicts (e.g., two users changing the same lace color). Conflicts are resolved via a visual diff panel or programmatic merge strategy.

## How to Use

### 1. Define a Boot Blueprint
Create a `boot.blueprint.yaml` file in your project root:

```yaml
version: "2.0"
boot:
  type: "chelsea"
  heel:
    height_mm: 40
    material: "rubber"
  upper:
    leather: "full-grain"
    color: "#2C3E50"
  sole:
    pattern: "herringbone"
    thickness_mm: 12
```

Run `boothell validate boot.blueprint.yaml` to check structural integrity and brand compliance.

### 2. Generate a Multi-Variant Asset Pipeline
Use the CLI to batch-generate assets for different locales or marketing channels:

```bash
boothell render \
  --blueprint boot.blueprint.yaml \
  --variants "us,eu,asia" \
  --format "svg,png,webp" \
  --output ./dist/boot-assets/
```

Each variant auto-applies locale-specific sizing, label placement, and resolution.

### 3. Deploy with Version Control and Rollback
Push the generated assets to your CDN with a manifest:

```bash
boothell deploy \
  --source ./dist/boot-assets/ \
  --manifest ./manifest.json \
  --tag "v2.1.0"
```

To rollback, simply redeploy a previous manifest tag. The system keeps a full audit trail of every asset version.

## Best Practices

### 1. Use Semantic Versioning for Boot Specs
Treat your boot blueprints like software dependencies. Major versions for breaking design changes (e.g., new sole geometry), minor for additive features (new color options), and patches for fixes (material texture tweaks). This ensures backward compatibility and clean rollback paths.

### 2. Automate Brand Compliance with CI/CD
Integrate boothell into your CI pipeline. Before any boot asset is merged to production, run:

```bash
boothell audit --branch main --strict
```

This checks all boot specs against the latest brand guidelines and fails the build if any violation is found. This is especially critical when multiple teams contribute to the same boot catalog.
