## Detailed Content

### What is capsule-factory-saas?

capsule-factory-saas is a modular, cloud-native platform designed to orchestrate the entire capsule production lifecycle—from raw gelatin batching and filling to encapsulation, polishing, and final packaging. It replaces fragmented manual logs with a unified data model, enabling granular batch tracking, yield optimization, and compliance with GMP (Good Manufacturing Practice) standards. Built for DevOps-minded production managers and QA engineers who demand zero-downtime deployments and real-time anomaly detection.

### Core Features

- **Granular Batch DNA & Traceability Engine**  
  Every capsule batch is assigned a unique digital fingerprint, capturing raw material lot numbers, environmental conditions (temperature, humidity), machine parameters, and operator IDs. The engine generates a tamper-proof audit trail compliant with FDA 21 CFR Part 11 and EU GMP Annex 11. Supports downstream recall simulation and root-cause analysis via graph-based lineage queries.

- **Adaptive Yield Optimization with ML Anomaly Detection**  
  The platform ingests real-time sensor data from encapsulation machines and applies a lightweight LSTM model to predict yield deviations before they occur. It automatically adjusts fill weight setpoints and dwell times, reducing scrap rates by up to 18% in controlled trials. Alerts are pushed to Slack, PagerDuty, or custom webhooks.

- **Multi-Tenant Production Scheduler with Finite Capacity Constraints**  
  A constraint-based scheduler that respects machine downtime, tooling changeover windows, and raw material availability. It generates shift-level plans with Gantt visualization, supports drag-and-drop rescheduling, and automatically recalculates estimated time of completion (ETC) for each order. Integrates with ERP systems via RESTful APIs.

### How to Use

1. **Initialize a New Production Campaign via the CLI**  
   Run `capsule-factory-saas campaign create --batch-size 500000 --capsule-type HPMC --gelatin-lot GEL-2025-03` to bootstrap a campaign. The system validates raw material inventory, reserves machine slots, and generates a batch ID. Use `--dry-run` to simulate without committing resources. Logs are streamed to stdout and a local `campaign_<id>.json` file.

2. **Configure Real-Time Machine Telemetry Integration**  
   Edit the `config/telemetry.yaml` file to define your MQTT broker endpoints and machine-specific topics. Example:  
   ```yaml
   machines:
     encapsulator_01:
       mqtt_topic: "factory/encapsulator/01/telemetry"
       fields: ["fill_weight_g", "temperature_c", "humidity_pct", "speed_rpm"]
   ```  
   Run `capsule-factory-saas telemetry start --config config/telemetry.yaml` to begin streaming. Data is automatically normalized and stored in TimescaleDB for downsampling queries.

3. **Generate a Batch Performance Report with Custom Thresholds**  
   Use the built-in reporting module:  
   ```bash
   capsule-factory-saas report generate --batch-id BATCH-2025-03-17-001 --metrics yield,defect_rate,downtime --threshold-yield 0.95 --output-format html
   ```  
   The report includes a heatmap of defect types by machine station, a trend line of fill weight variance, and a compliance summary. Output is saved to `reports/BATCH-2025-03-17-001_report.html` and can be emailed directly via SMTP configuration.

### Best Practices

- **Instrument Every Machine with a Redundant Telemetry Path**  
  Always configure both primary (MQTT) and secondary (HTTP POST fallback) data ingestion channels. In high-throughput environments, network jitter can cause packet loss. The platform’s telemetry pipeline includes a local buffer that retries failed writes with exponential backoff. Test your fallback path weekly using `capsule-factory-saas telemetry test-fallback`.

- **Implement a Pre-Production Dry-Run for Every New Capsule Type**  
  Before ramping to full production, run a dry campaign with a small batch (e.g., 10,000 capsules) using `--dry-run` and `--skip-machine-allocation`. This validates your material BOM, machine recipe parameters, and quality thresholds without consuming physical resources. Analyze the generated `dry_run_summary.json` for any constraint violations or yield warnings.
