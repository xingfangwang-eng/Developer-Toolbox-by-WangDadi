# capsule-factory-saas - CAP_002

## Overview

This is a hardcore, production-grade SaaS platform engineered for capsule manufacturers who demand real-time control over their entire production lifecycle. Built for DevOps-minded factory operators and QA engineers, **capsule-factory-saas** eliminates silos between formulation, encapsulation, and packaging. It’s not just another ERP wrapper—it’s a purpose-built, API-first system that treats every capsule batch as a deployable artifact.

## What is capsule-factory-saas?

capsule-factory-saas is a multi-tenant, event-driven platform that orchestrates capsule production from raw material blending to final quality sign-off. It exposes granular, idempotent REST endpoints for batch tracking, environmental sensor ingestion, and equipment telemetry. Designed for high-throughput factories running 24/7, it supports rollback of batch parameters, real-time OEE dashboards, and automated compliance logging (ISO 9001, GMP).

## Core Features

### 1. Batch-as-a-Deployment (BaaD) Pipeline
Every capsule batch is treated as a versioned, immutable deployment. The system automatically snapshots ingredient lots, machine settings, and environmental conditions (temperature, humidity, pressure) at each encapsulation phase. Rollbacks are atomic: you can revert a batch to any previous stage while preserving audit trails—no manual reconciliation, no spreadsheet hell.

### 2. Real-Time Sensor Fusion & Anomaly Detection
Ingest data from PLCs, IoT humidity sensors, and weight-checkers via MQTT or OPC-UA. The platform fuses these streams into a unified time-series model, triggering alerts when deviations exceed configurable SPC (Statistical Process Control) limits. Pre-built Grafana dashboards expose live OEE, yield rates, and defect clustering—no third-party BI tool required.

### 3. Declarative Recipe Engine with CI/CD for Formulations
Define capsule recipes as YAML manifests with version-controlled parameters (fill weight, shell thickness, drying time). Push recipe changes through a staging environment before promoting to production. Each change triggers automated regression tests against historical batch data, flagging potential yield drops or quality regressions before a single capsule is produced.

## How to Use

### 1. Onboard a Production Line via API
Use the `/v1/lines` endpoint to register a new encapsulation line with its sensor topology and machine specs. The API expects a JSON payload containing line ID, PLC endpoint, and sensor mapping. The response returns a unique line token and a WebSocket URL for real-time telemetry. This step is idempotent—rerunning it updates the line configuration without duplicating data.

**Example:**
```bash
curl -X POST https://api.capsule-factory-saas.io/v1/lines \
  -H "Authorization: Bearer <your-token>" \
  -H "Content-Type: application/json" \
  -d '{
    "line_id": "L-202",
    "plc_endpoint": "opc.tcp://192.168.1.50:4840",
    "sensors": ["humidity", "fill_weight", "shell_thickness"]
  }'
```

### 2. Deploy a Recipe Version to a Line
Push a new recipe version using the `/v1/recipes/{recipe_id}/deploy` endpoint. The request must include the target line ID, the recipe version tag, and a dry-run flag. The system validates the recipe against the line’s current machine capabilities—if the drying time exceeds the oven’s max cycle, the deployment is rejected with a detailed error log. Use `dry_run: true` to validate without executing.

**Example:**
```bash
curl -X POST https://api.capsule-factory-saas.io/v1/recipes/vitamin-d3-v2/deploy \
  -H "Authorization: Bearer <your-token>" \
  -H "Content-Type: application/json" \
  -d '{
    "line_id": "L-202",
    "version": "2.1.0",
    "dry_run": true
  }'
```

### 3. Trigger a Batch Rollback with Atomic Snapshot
If a batch shows deviation (e.g., fill weight variance > 3%), initiate a rollback to a previous stage using `/v1/batches/{batch_id}/rollback`. The request must specify the target stage ID and a reason string. The system restores the exact sensor readings, machine parameters, and ingredient lots from that stage, then re-queues the batch for rework. All actions are logged to an immutable audit trail.

**Example:**
```bash
curl -X POST https://api.capsule-factory-saas.io/v1/batches/B-1045/rollback \
  -H "Authorization: Bearer <your-token>" \
  -H "Content-Type: application/json" \
  -d '{
    "target_stage": "encapsulation-v1",
    "reason": "Fill weight deviation detected at packaging stage"
  }'
```

## Best Practices

### 1. Implement Canary Deployments for Recipe Changes
Never push a new recipe version directly to all lines. Instead, deploy it to a single “canary” line running low-volume SKUs. Monitor yield, defect rate, and OEE for 24 hours. Only promote to full production after the canary line shows no regression. This mirrors blue-green deployment patterns in software and drastically reduces scrap costs.

### 2. Use Idempotent Ingredient Lot Tracking
When registering ingredient lots via the `/v1/ingredients` endpoint, always include a unique lot ID and a checksum of the supplier’s COA (Certificate of Analysis). The system will reject duplicate submissions silently. This prevents double-counting of raw materials and ensures traceability during audits—critical for GMP compliance.
