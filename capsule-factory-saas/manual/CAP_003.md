# capsule-factory-saas - CAP_003

## Overview

This is a production-grade, multi-tenant SaaS platform engineered for capsule manufacturers who need real-time control over formulation, encapsulation, and batch traceability. Built for DevOps-minded pharma engineers and plant operators, `capsule-factory-saas` eliminates manual spreadsheet loops and legacy ERP gaps with a modular, API-first stack optimized for GMP compliance and high-throughput lines.

## Core Features

### 1. Real-Time Viscosity & Fill Weight PID Loop
Each encapsulation head is instrumented with a closed-loop PID controller that adjusts fill weight in sub-second intervals based on live viscosity readings from inline rheometers. The system logs every deviation to a time-series database (InfluxDB) and triggers automated reject gate actions when tolerances exceed ±0.5%. No manual calibration windows—zero drift.

### 2. Multi-Tenant Batch Genealogy Engine
Every capsule batch is assigned a cryptographically signed lineage hash that links raw material lots, machine parameters, environmental conditions, and QC test results into a single Merkle tree. This engine enables instant root-cause analysis across tenants without compromising data isolation—ideal for contract manufacturers who run multiple clients on the same floor.

### 3. Predictive Shell Integrity Monitoring
Using a lightweight ONNX model trained on historical cracking patterns, the system predicts shell failure probability 90 seconds before occurrence. Alerts are pushed via WebSocket to the operator HMI and automatically slow the line speed to prevent catastrophic jams. The model retrains nightly on edge-case data from all tenants under differential privacy.

## How to Use

### 1. Deploy the Tenant-Aware Pipeline
```bash
# Clone the monorepo and bootstrap the infrastructure
git clone https://github.com/xingfangwang-eng/capsule-factory-saas.git
cd capsule-factory-saas/infra
terraform init && terraform apply -auto-approve \
  -var="tenant_id=acme_pharma" \
  -var="region=us-east-1" \
  -var="k8s_namespace=capsule-prod"
```
This spins up a dedicated Kubernetes namespace, PostgreSQL with Row-Level Security, and a NATS JetStream cluster for telemetry ingestion. The tenant ID is embedded into every resource tag and audit log.

### 2. Configure the Encapsulation Head Profile
```yaml
# configs/head_profile_01.yaml
head_id: "H-023"
formulation:
  gel_type: "HPMC"
  target_viscosity_cp: 450
  tolerance_pct: 0.5
pid:
  kp: 1.8
  ki: 0.04
  kd: 0.12
  sample_rate_ms: 50
alerts:
  webhook_url: "https://hooks.slack.com/services/T.../B.../xxx"
  pagerduty_routing_key: "your_pd_key"
```
Apply the profile via the CLI:  
`capsule-cli apply-profile --file configs/head_profile_01.yaml`  
The system validates the YAML schema, pushes the PID constants to the edge PLC, and starts streaming telemetry within 3 seconds.

### 3. Run a Batch and Trigger Automated QC Hold
```bash
capsule-cli start-batch \
  --tenant acme_pharma \
  --recipe "Gelatin_A_12" \
  --quantity 500000 \
  --lot "LOT-20250321" \
  --qc-hold-on-deviance
```
The engine spawns a batch context, assigns a unique batch ID (`B-20250321-001`), and begins ingesting sensor data. If any head exceeds the viscosity tolerance for more than 2 consecutive samples, the batch is automatically placed in `HOLD` status, a Jira ticket is created via webhook, and the operator HMI displays a red overlay with the exact head ID and timestamp.

## Best Practices

### 1. Pin Your ONNX Runtime Version in CI/CD
Always lock the `onnxruntime` version to a specific minor release (e.g., `1.17.x`) in your Dockerfile and GitHub Actions matrix. The shell integrity model depends on ops that changed between 1.16 and 1.17—unpinned versions cause silent inference failures in production. Use Dependabot with a manual approval gate for runtime upgrades.

### 2. Implement Tenant-Level Rate Limiting on the Telemetry API
Each encapsulation head emits ~20 messages per second. Without per-tenant rate limiting, a misconfigured head in one tenant can flood the NATS JetStream and degrade latency for all others. Use `envoy.ext.filters.http.rate_limit` with a `tenant_id` header and a sliding window of 1000 requests/sec per tenant. Monitor the `nats_server_dropped_messages` metric as a canary.
