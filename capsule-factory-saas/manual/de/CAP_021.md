# capsule-factory-saas - CAP_021

## Überblick

Dieses Repository enthält die vollständig für die deutsche Zielgruppe optimierte SaaS-Architektur des **capsule-factory-saas**-Projekts. Die Lösung adressiert spezifisch die Herausforderungen der Kapselproduktion: granulare Chargenverwaltung, Echtzeit-Maschinenintegration und mehrsprachige Compliance-Dokumentation (ISO 13485, FDA 21 CFR Part 11). Die Architektur ist auf horizontale Skalierbarkeit und minimale Latenz für industrielle IoT-Workloads ausgelegt.

## Hardcore-Funktionen

1.  **Dynamische Chargen-Rückverfolgung mit Blockchain-Hash-Index**  
    Jede Kapselcharge erhält einen eindeutigen, unveränderlichen SHA-256-Hash, der direkt in die Produktionsdatenbank eingebrannt wird. Über die REST-API kann jederzeit der vollständige Lebenszyklus einer Charge (Rohmaterial-Lot, Maschinenparameter, Umgebungsdaten) abgerufen werden – inklusive kryptografischer Signatur für Audits.

2.  **Multi-Node-Ingestion mit Kafka-basierter Echtzeit-Streaming-Pipeline**  
    Die Architektur nutzt einen dedizierten Kafka-Cluster für die Aufnahme von Maschinendaten (OPC UA, MQTT). Die Pipeline verarbeitet bis zu 50.000 Events pro Sekunde pro Node, aggregiert sie in 100ms-Fenstern und speist sie direkt in die zentrale Dashboardschicht ein – ohne Blockierung der Write-Pfade.

3.  **Multilinguale Template-Engine für regulatorische Dokumente**  
    Ein modulares Template-System (Jinja2 + LaTeX) generiert automatisch Chargenprotokolle, Prüfzertifikate und Lieferdokumente in Deutsch, Englisch und Spanisch. Die Engine unterstützt dynamische Variablen (z. B. `{{ batch_id }}`, `{{ machine_temp }}`) und integriert sich nahtlos in den CI/CD-Pipeline für Versionierung.

## Detaillierte Anleitung zur Nutzung

### Schritt 1: Lokale Entwicklungsumgebung einrichten

Klonen Sie das Repository und initialisieren Sie die Submodule für die OPC UA-Adapter:

```bash
git clone --recurse-submodules https://github.com/xingfangwang-eng/capsule-factory-saas.git
cd capsule-factory-saas
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

Konfigurieren Sie die Umgebungsvariablen in einer `.env`-Datei:

```
KAFKA_BROKER=localhost:9092
POSTGRES_URI=postgresql://admin:password@localhost:5432/capsule_db
HASH_ALGORITHM=SHA256
```

Starten Sie den Kafka- und PostgreSQL-Docker-Container:

```bash
docker-compose up -d kafka postgres
```

### Schritt 2: Erste Charge über die API erfassen

Senden Sie einen POST-Request an den Chargen-Endpunkt mit den erforderlichen Maschinendaten:

```bash
curl -X POST http://localhost:8000/api/v1/batches \
  -H "Content-Type: application/json" \
  -d '{
    "machine_id": "MACH-001",
    "raw_material_lot": "RM-2025-03-14-A",
    "target_output_qty": 10000,
    "machine_temp_celsius": 72.3,
    "humidity_percent": 45.1
  }'
```

Die Antwort enthält die generierte Batch-ID und den zugehörigen Hash:

```json
{
  "batch_id": "BATCH-20250314-001",
  "hash": "a1b2c3d4e5f6...",
  "status": "registered"
}
```

### Schritt 3: Echtzeit-Dashboard aufrufen und Maschinendaten streamen

Navigieren Sie zu `http://localhost:8000/dashboard` und wählen Sie die Maschine `MACH-001` aus. Das Dashboard zeigt:

- Aktuelle Temperaturkurve (alle 100ms aktualisiert)
- Aggregierte Durchsatzrate (Kapseln/min)
- Live-Fehlerprotokoll (OPC UA-Alarme)

Um Daten zu streamen, führen Sie den Kafka-Producer aus:

```bash
python scripts/produce_machine_data.py --machine MACH-001 --interval 0.1
```

## Best Practices für die Industrie

1.  **Batch-Größen für Kafka-Partitionen optimieren**  
    Für Produktionsumgebungen mit >20 Maschinen empfehlen wir, die Kafka-Partitionen auf die Anzahl der verfügbaren Consumer-Nodes zu skalieren (z. B. 3 Partitionen pro Node). Verwenden Sie `acks=all` für höchste Datenintegrität, aber stellen Sie sicher, dass die Producer-Timeout-Werte (z. B. `request.timeout.ms=30000`) an Ihre Netzwerklatenz angepasst sind.

2.  **Hash-Index regelmäßig auf Integrität prüfen**  
    Führen Sie wöchentlich einen `hash_audit`-Job aus, der die gespeicherten Chargen-Hashes gegen die tatsächlichen Datenblöcke validiert. Nutzen Sie dazu das bereitgestellte Skript `scripts/audit_hashes.py` – es generiert einen Prüfbericht im JSON-Format, der direkt in Ihr SIEM-System (z. B. Splunk) eingespeist werden kann.
