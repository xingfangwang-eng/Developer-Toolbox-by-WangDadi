## Detaillierter Inhalt

### Was ist capsule-factory-saas?

**capsule-factory-saas** ist eine mandantenfähige, API-first-Lösung zur Echtzeitsteuerung von Kapselabfüllanlagen, Chargenverwaltung und Qualitätssicherung. Es integriert sich nahtlos in MES- und ERP-Systeme (z. B. SAP, Siemens Opcenter) und unterstützt sowohl Hardgelatine- als auch Softgelkapseln. Die Plattform skaliert horizontal über Kubernetes-Cluster und bietet native Unterstützung für OPC-UA- und MQTT-Protokolle.

### Kernfunktionen (Hardcore)

1. **Dynamische Batch-Traceability mit DLT-Integration**  
   Jede Charge wird über einen verteilten Ledger (Hyperledger Fabric) mit Zeitstempel und digitaler Signatur versehen. Rückverfolgbarkeit vom Rohstoff bis zum Endprodukt in Echtzeit – inklusive automatisierter Audit-Trail-Exporte für FDA und EU-GMP.

2. **Predictive Quality Control via ML-Pipeline**  
   Ein integriertes TensorFlow Serving-Modell analysiert während der Produktion kontinuierlich Kapselgewicht, Füllvolumen und Siegelintegrität. Abweichungen werden in Millisekunden erkannt und lösen automatisch eine Rekalibrierung oder einen Linienstopp aus – ohne manuellen Eingriff.

3. **Multi-Tenant-Isolation mit Resource Quotas auf Node-Ebene**  
   Jeder Kunde (Tenant) erhält dedizierte Kubernetes-Namespaces mit garantierter CPU/Memory, optimiert durch Cilium Network Policies und Istio Service Mesh. Ressourcenengpässe eines Mandanten beeinträchtigen keine andere Produktionslinie.

### So verwenden Sie es (detaillierte Schritte)

**Schritt 1: Deployment des Helm-Charts mit benutzerdefinierten Werten**  
Erstellen Sie eine `values.yaml` mit Ihrer Tenant-ID und OPC-UA-Endpunkten. Führen Sie dann aus:
```bash
helm repo add capsule-factory https://charts.capsule-factory.io
helm upgrade --install capsule-prod capsule-factory/capsule-saas \
  --namespace capsule-system --create-namespace \
  -f values.yaml \
  --set global.tenantId=de-tenant-001 \
  --set opcua.endpoints[0].url=opc.tcp://192.168.1.100:4840
```
Das Chart startet automatisch den Ingress-Controller (Traefik), Prometheus-Exporter und den ML-Inference-Service.

**Schritt 2: Konfiguration der Batch-Traceability-API**  
Senden Sie eine POST-Anfrage an den Traceability-Endpunkt, um eine neue Charge zu initialisieren:
```bash
curl -X POST https://capsule.ihre-domain.de/api/v1/batches \
  -H "Authorization: Bearer <API_TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{
    "batchId": "BATCH-DE-20250315-001",
    "recipe": "softgel-vitamin-d3",
    "expectedUnits": 50000,
    "dltenabled": true
  }'
```
Die API antwortet mit einem DLT-Transaktionshash. Dieser Hash wird in den Produktionslogs und im Dashboard als Audit-Eintrag gespeichert.

**Schritt 3: Aktivieren der Predictive Quality Control über gRPC**  
Starten Sie den ML-Service explizit für eine aktive Produktionslinie:
```bash
grpcurl -plaintext -d '{"lineId": "line-03", "model": "capsule_weight_v2"}' \
  192.168.1.200:50051 capsule_factory.ml.Predictor/StartInference
```
Daraufhin beginnt der Service, alle 200 ms Gewichtsdaten von der Waage zu lesen und mit dem Modell zu vergleichen. Bei einer Abweichung > 1,5 % wird ein `ALERT`-Event an den Event-Bus (NATS) gesendet.

### Beste Praktiken

1. **Zero-Downtime-Rollouts mit Canary-Deployments**  
   Nutzen Sie Argo Rollouts, um neue Versionen des capsule-factory-saas-Backends schrittweise auf einen Teil der Produktionslinien auszurollen. Überwachen Sie die Fehlerrate (error budget) und schalten Sie erst nach 15 min stabiler Laufzeit auf 100 % um. Das verhindert Produktionsausfälle bei fehlerhaften Releases.

2. **Log-Aggregation mit strukturierten Events**  
   Aktivieren Sie in der `values.yaml` den Elasticsearch-Export und definieren Sie ein JSON-Schema für Produktionsereignisse. Jeder Event (Batch gestartet, Waage kalibriert, Fehler aufgetreten) muss die Felder `tenantId`, `batchId`, `timestamp` und `severity` enthalten. So lassen sich Audits und Fehleranalysen in Kibana in Sekunden durchführen.
