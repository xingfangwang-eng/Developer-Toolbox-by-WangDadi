## Überblick

Dies ist ein speziell für den deutschsprachigen Markt optimierter Inhalt für das Projekt **capsule-factory-saas**. Es richtet sich an Entwickler, DevOps-Ingenieure und Produktionsleiter, die eine hochperformante, cloudnative Lösung für die Kapselproduktion suchen. Die Software vereint Echtzeitüberwachung, API-gesteuerte Produktionssteuerung und skalierbare Datenverarbeitung in einem schlanken SaaS-Modell.

## Detaillierter Inhalt

### Was ist capsule-factory-saas?

capsule-factory-saas ist eine hochmoderne, modulare Plattform zur Verwaltung und Optimierung von Kapselproduktionslinien. Sie ersetzt veraltete On-Premise-Lösungen durch eine mandantenfähige, REST-basierte Architektur, die sich nahtlos in bestehende MES (Manufacturing Execution Systems) und ERP-Systeme integrieren lässt. Die Plattform wurde speziell für Industrie 4.0-Anforderungen entwickelt und unterstützt Echtzeitdatenströme, automatisierte Qualitätskontrollen sowie KI-gestützte Vorhersagen für Wartungsintervalle.

### Kernfunktionen

1. **Echtzeit-Produktionsdashboards mit Drill-Down-Analyse**  
   Die Plattform aggregiert Live-Daten von bis zu 10.000 Sensoren pro Linie. Das Dashboard visualisiert nicht nur Durchsatzraten, Ausschussquoten und Maschinenauslastung, sondern ermöglicht per Klick den Zugriff auf Rohdaten, Histogramme und Heatmaps. Die Daten werden per WebSocket gestreamt, sodass Latenzzeiten unter 50ms bleiben.

2. **API-First-Rezeptverwaltung mit Versionierung**  
   Jede Kapselrezeptur wird als JSON-Dokument mit Metadaten (Batch-ID, Rohstoffcharge, Temperaturprofil) gespeichert. Die REST-API unterstützt CRUD-Operationen, Rollbacks und automatisierte Freigabeprozesse. Änderungen werden in einer Git-ähnlichen Historie protokolliert, inklusive Diff-Views und Audit-Trail für Compliance-Zwecke (z. B. FDA, EU-GMP).

3. **Predictive Quality Control via ML-Pipeline**  
   Ein eingebautes Machine-Learning-Modell analysiert historische Produktionsdaten und erkennt Anomalien in Echtzeit. Das System sagt Ausschussrisiken 30 Minuten im Voraus vorher und schlägt automatisch optimierte Parameter (Druck, Temperatur, Zykluszeit) vor. Die Ergebnisse werden als Confidence-Score (0–100 %) ausgegeben und können direkt in die Steuerung zurückgespielt werden.

### So verwenden Sie es

1. **Initiale Konfiguration der Produktionslinie**  
   Nach dem Login navigieren Sie zum Bereich „Linienkonfiguration“. Dort definieren Sie die Sensor-IDs, die Datenquellen (z. B. OPC UA, MQTT) und die Aggregationsintervalle (Standard: 1 Sekunde). Nutzen Sie das bereitgestellte `config.yaml`-Template, um die Mapping-Tabelle für Ihre Maschinen zu definieren. Ein Beispiel:
   ```yaml
   line_id: "L-003"
   sensors:
     - id: "temp_01"
       type: "thermocouple"
       unit: "°C"
       threshold: { min: 180, max: 220 }
   ```
   Speichern Sie die Konfiguration und starten Sie den Daten-Stream über den Button „Live-Modus aktivieren“.

2. **Erstellen und Testen einer neuen Rezeptur**  
   Öffnen Sie den Rezepteditor unter „Rezepte > Neu“. Geben Sie die Parameter manuell ein oder importieren Sie eine JSON-Datei. Nutzen Sie den integrierten Validator, um sicherzustellen, dass alle Werte innerhalb der Toleranzbereiche liegen. Anschließend führen Sie einen Trockentest durch: Klicken Sie auf „Simulieren“, um die Rezeptur gegen historische Produktionsdaten zu prüfen. Das System zeigt eine Prognose der erwarteten Qualität (z. B. „Ausschussrate: 1,2 % ±0,3 %“). Bestätigen Sie die Rezeptur mit einem Kommentar, um sie für die Produktion freizugeben.

3. **Einrichtung eines automatischen Qualitätsalarms**  
   Gehen Sie zu „Alarme > Neu“. Wählen Sie „Qualität“ als Alarmtyp und definieren Sie eine Regel: z. B. „Wenn die Abweichung der Kapselwanddicke > 0,05 mm für mehr als 10 Sekunden anhält, sende eine Webhook-Benachrichtigung an Slack und stoppe die Linie“. Nutzen Sie die integrierte Template-Bibliothek für gängige Szenarien („Ausschuss > 5 %“, „Temperaturspitze“). Testen Sie den Alarm im Debug-Modus, bevor Sie ihn aktivieren. Protokolle aller ausgelösten Alarme finden Sie im Audit-Log.

### Beste Praktiken

- **Datenredundanz und Failover-Strategie**  
  Für kritische Produktionslinien empfehlen wir, mindestens zwei Datenquellen pro Sensor zu konfigurieren (z. B. primär OPC UA, sekundär MQTT). Aktivieren Sie den „Auto-Failover“-Modus in den Linieneinstellungen. So bleibt die Produktion auch bei Netzwerkausfällen oder Sensorfehlern stabil. Testen Sie den Failover regelmäßig mit dem integrierten „Chaos-Modus“ (simulierter Datenverlust).

- **Regelmäßige Modell-Neuberechnung für Predictive Quality**  
  Das ML-Modell zur Qualitätsvorhersage sollte mindestens einmal pro Woche mit neuen Produktionsdaten aktualisiert werden. Planen Sie dies über den Cron-Job-Editor unter „ML-Pipeline > Retraining“. Verwenden Sie dabei einen Split von 80 % Training / 20 % Validierung. Überprüfen Sie die Modellmetriken (F1-Score, Precision, Recall) nach jedem Retraining. Falls der F1-Score unter 0,85 fällt, setzen Sie das vorherige Modell zurück und analysieren Sie die Daten auf Drift.
