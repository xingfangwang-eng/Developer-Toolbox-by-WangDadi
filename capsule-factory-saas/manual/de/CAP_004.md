## Detaillierter Inhalt

### Was ist capsule-factory-saas?

capsule-factory-saas ist eine modulare Cloud-Plattform, die speziell für die **Echtzeit-Überwachung und Steuerung von Kapselproduktionslinien** entwickelt wurde. Anders als traditionelle MES-Systeme (Manufacturing Execution Systems) setzt dieses Tool auf eine **Event-Sourcing-Architektur** und eine **GraphQL-Schnittstelle**, um Batch-Daten, Maschinenparameter und Qualitätskennzahlen (z. B. Füllgewicht, Härtegrad, Auflösungszeit) mit submillisekundiger Latenz zu verarbeiten.

### Kernfunktionen (Hardcore-Features)

1.  **Multi-Tenant Batch-Orchestrierung mit automatischer Chargen-Rückverfolgung (Track & Trace)**  
    Jede Kapselcharge wird über eine eindeutige, kryptografisch signierte Batch-ID (SHA-256) mit allen Produktionsparametern verknüpft. Das System erlaubt die parallele Verwaltung von bis zu 1.000 aktiven Chargen pro Tenant, inklusive automatischer Generierung von Dossier-Exporten (PDF, XML) für regulatorische Audits (GMP-konform).

2.  **Maschinen-Agnostisches Plug-in-Protokoll (MAPP) für Echtzeit-Datenerfassung**  
    Statt proprietärer Treiber nutzt capsule-factory-saas ein offenes, auf MQTT und OPC UA basierendes Plug-in-System. Entwickler können eigene Adapter für beliebige Kapselmaschinen (z. B. Zanasi, IMA, MG2) in Go oder Rust schreiben. Die Daten werden automatisch normalisiert und in einer Time-Series-Datenbank (InfluxDB-kompatibel) gespeichert.

3.  **Predictive Quality Engine (PQE) mit ML-basierter Echtzeit-Anomalieerkennung**  
    Die Engine analysiert live alle eingehenden Sensorwerte (Temperatur, Feuchtigkeit, Druck, Drehmoment) und erkennt Abweichungen, die zu Ausschuss oder Chargenfehlern führen könnten. Sie liefert direkt in der UI eine **Wahrscheinlichkeitsbewertung (0–100%)** und schlägt automatisch Korrekturmaßnahmen vor (z. B. „Füllgewicht um 2,3 mg reduzieren“). Das Modell kann über eine REST-API mit eigenen Trainingsdaten erweitert werden.

### So verwenden Sie es (Detaillierte Anleitung)

1.  **Initiale Einrichtung eines Multi-Tenant-Clusters mit Helm-Chart**  
    Stellen Sie sicher, dass Sie einen Kubernetes-Cluster (v1.28+) mit mindestens 16 GB RAM und 8 Kernen haben. Führen Sie das offizielle Helm-Chart aus:  
    ```bash
    helm repo add capsule-factory https://charts.capsule-factory-saas.io
    helm install my-factory capsule-factory/capsule-factory \
      --set global.tenantId=acme-corp \
      --set global.licenseKey=YOUR_LICENSE \
      --set persistence.storageClass=gp3 \
      --set mqtt.broker.endpoint=tcp://mqtt-broker:1883
    ```  
    Nach erfolgreichem Deployment prüfen Sie die Pod-Health über `kubectl get pods -n capsule-factory`. Die API ist unter `https://api.acme-corp.capsule-factory-saas.io/graphql` erreichbar.

2.  **Integration einer neuen Kapselmaschine über das MAPP-Plugin**  
    Erstellen Sie ein neues Plugin-Verzeichnis im `plugins/`-Ordner des Repositories. Beispiel für eine virtuelle Maschine:  
    ```rust
    // plugins/zanasi_90/mod.rs
    use capsule_factory_sdk::prelude::*;

    #[derive(Default)]
    struct Zanasi90Plugin;

    impl MachinePlugin for Zanasi90Plugin {
        fn name(&self) -> &str { "Zanasi 90" }
        fn protocol(&self) -> Protocol { Protocol::OPCUA { endpoint: "opc.tcp://192.168.1.100:4840" } }
        fn collect(&self) -> Vec<SensorReading> {
            vec![
                SensorReading::new("temperature", 23.5, "celsius"),
                SensorReading::new("fill_weight_mg", 500.2, "mg"),
            ]
        }
    }

    register_plugin!(Zanasi90Plugin);
    ```  
    Kompilieren Sie das Plugin mit `cargo build --release` und legen Sie die `.so`-Datei in das Verzeichnis `/opt/capsule-factory/plugins`. Starten Sie den `machine-agent`-Pod neu, um die Maschine zu registrieren.

3.  **Konfiguration einer Predictive-Quality-Regel mit benutzerdefinierten Schwellwerten**  
    Öffnen Sie die Admin-Konsole unter `/admin/pqe`. Definieren Sie eine neue Regel:  
    ```json
    {
      "ruleName": "fill_weight_deviation",
      "metric": "fill_weight_mg",
      "operator": "outside_range",
      "threshold": [495.0, 505.0],
      "consecutiveFailures": 3,
      "action": "PAUSE_LINE",
      "mlModelId": "v2.3.1"
    }
    ```  
    Aktivieren Sie die Regel und beobachten Sie das Dashboard. Bei Überschreitung wird die Produktionslinie automatisch pausiert und ein Ticket im integrierten Incident-Management erstellt. Die PQE loggt jeden Schritt in der `audit_trail`-Tabelle.

### Beste Praktiken (Industry Best Practices)

1.  **Implementieren Sie ein striktes Schema-on-Read für Maschinendaten**  
    Vermeiden Sie es, Sensorwerte unverändert in die Datenbank zu schreiben. Nutzen Sie stattdessen die integrierte Transformations-Pipeline (ETL), um alle Daten in ein einheitliches Metrik-Schema zu konvertieren (z. B. `{timestamp, tenant, machine_id, metric_name, value, unit}`). Das reduziert die Komplexität bei späteren Analysen und erleichtert den Austausch von Maschinenherstellern.

2.  **Führen Sie wöchentliche Regressionstests mit simulierten Batch-Läufen durch**  
    capsule-factory-saas bietet ein CLI-Tool (`capsule-factory-cli simulate`), das synthetische Sensorströme generiert. Nutzen Sie dieses, um Ihre PQE-Modelle und Alarmierungsregeln unter Last zu testen. Empfohlen wird ein Test mit mindestens 10.000 simulierten Chargen pro Woche, um Modell-Drift frühzeitig zu erkennen und die Genauigkeit über 99,5 % zu halten.
