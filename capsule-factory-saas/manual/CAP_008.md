# capsule-factory-saas - CAP_008

## Overview

**capsule-factory-saas** is a production-grade, multi-tenant SaaS platform engineered for precision capsule manufacturing environments. It streamlines batch tracking, recipe management, and real-time equipment orchestration—purpose-built for pharma, nutraceutical, and cosmetic factories operating under strict GMP and FDA guidelines. This document delivers SEO-optimized technical content targeting English-speaking DevOps, production engineers, and SaaS architects.

## Core Features

### 1. Dynamic Recipe Versioning with Rollback
Each capsule formulation (fill weight, shell composition, drying time) is version-controlled via Git-like semantic tagging. The system auto-generates a production manifest per batch, linking every capsule unit to its exact recipe hash. Rollback to any prior recipe version triggers a dry-run simulation, preventing costly material waste.

### 2. Multi-Tenant Equipment Scheduler with Real-Time OEE
Leveraging a distributed task queue (e.g., Celery + Redis), the scheduler assigns capsule filling, sealing, and polishing machines across tenants. OEE (Overall Equipment Effectiveness) is calculated per shift, factoring in planned downtime, micro-stops, and speed loss. Alerts fire when OEE dips below a configurable threshold, pushing directly to Slack or PagerDuty.

### 3. Traceability Graph for Lot Genealogy
Every capsule batch is mapped as a directed acyclic graph (DAG) showing parent-child relationships from raw material intake to finished goods. Using a graph database (Neo4j), the system enables instant root-cause analysis: a single query reveals which supplier lot caused a hardness deviation across 12,000 units.

## How to Use

### 1. Deploy the Tenant Isolation Layer
```bash
# Clone repo and bootstrap a new tenant with isolated DB schema
git clone https://github.com/xingfangwang-eng/capsule-factory-saas.git
cd capsule-factory-saas
./scripts/provision_tenant.sh --tenant-id "acme-pharma" --region "us-east-1" --db-pool-size 10
```
This script creates a dedicated PostgreSQL schema, spins up a separate Redis namespace, and configures tenant-specific API rate limits. Validate with:
```bash
curl -X GET https://api.capsule-factory.example.com/v1/tenants/acme-pharma/health \
  -H "Authorization: Bearer ${TENANT_API_KEY}"
```

### 2. Define a Capsule Recipe via YAML
Create a `recipe.yaml` file:
```yaml
recipe_id: "RCP-2025-03-14-v2"
capsule_type: "hard_gelatin"
fill_weight_mg: 500
shell_thickness_mm: 0.8
drying_curve:
  - temp_c: 25
    humidity_pct: 35
    duration_min: 120
  - temp_c: 40
    humidity_pct: 20
    duration_min: 60
```
Push it to the API:
```bash
curl -X POST https://api.capsule-factory.example.com/v1/recipes \
  -H "Content-Type: application/yaml" \
  --data-binary @recipe.yaml
```
The system validates against your tenant’s equipment capabilities and returns a recipe ID.

### 3. Initiate a Production Run with Live Monitoring
```bash
# Start batch BATCH-0314 with recipe RCP-2025-03-14-v2 on line L3
capsule-cli run --batch-id "BATCH-0314" \
                --recipe-id "RCP-2025-03-14-v2" \
                --line-id "L3" \
                --target-units 100000
```
Open the WebSocket stream for real-time metrics:
```bash
wscat -c wss://api.capsule-factory.example.com/v1/batches/BATCH-0314/events \
  -H "Authorization: Bearer ${TENANT_API_KEY}"
```
You’ll see JSON events like `{"event": "fill_head_1_cycle_complete", "units": 250, "timestamp": "2025-03-14T10:32:17Z"}`.

## Best Practices

### 1. Implement Idempotency Keys for Batch Operations
When retrying a failed batch initiation (e.g., due to network blip), always include an `Idempotency-Key` header. This prevents duplicate production runs that could double material consumption. Use UUID v4 keys scoped to the tenant ID and batch ID.

### 2. Set Up Predictive Maintenance via ML on Sensor Data
Collect vibration and temperature telemetry from capsule polishing machines. Train a lightweight LSTM model (e.g., using TensorFlow Lite) on historical failure patterns. Deploy inference as a sidecar container in your Kubernetes cluster. When the model predicts a bearing failure within 48 hours, automatically reserve a maintenance slot and re-route production to a redundant line.
