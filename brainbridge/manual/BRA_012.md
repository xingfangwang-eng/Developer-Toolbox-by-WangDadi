# brainbridge - BRA_012

## Overview

`brainbridge` is not another note-taking app or mind mapper. It’s a **thought-connection engine** designed for developers, researchers, and systems thinkers who need to externalize, trace, and traverse complex mental models across distributed contexts. Think of it as a **semantic adjacency matrix** for your brain—built for speed, queryability, and long-term recall.

**Keywords:** Brain bridge tool, Thought connection software, Brain thinking tool

---

## Hardcore Features

### 1. Bidirectional Thought Linking with Weighted Edges
Unlike flat tags or hierarchical folders, `brainbridge` lets you create **directional, weighted connections** between any two nodes (ideas, code snippets, files, or external URIs). Each link carries a *relevance score* (0.0–1.0) and a *relation type* (e.g., `causes`, `contradicts`, `extends`, `depends_on`). This enables graph traversal queries like:  
> *“Show me all nodes that indirectly depend on `concept_X` with path depth ≤ 3 and edge weight ≥ 0.7.”*

### 2. Context-Aware Sliding Window Sync
`brainbridge` supports **multi-session state reconciliation** via a CRDT-backed sync layer. When you work across devices, the tool automatically merges local graphs using a sliding window of the last 200 edits. This avoids merge conflicts while preserving temporal context—perfect for teams collaborating on evolving architectures or research hypotheses.

### 3. CLI-First with REPL and JSON-RPC
No bloat. `brainbridge` ships as a **single binary** with a REPL interface, a JSON-RPC endpoint, and a minimal HTTP API. You can pipe thoughts from `stdin`, query the graph with a custom DSL (`:find nodes where type == "hypothesis" and weight > 0.8`), and export the entire graph as DOT, JSON, or Obsidian-compatible Markdown. No GUI required.

## How to Use

### 1. Initialize a Thought Graph and Seed Nodes
```bash
brainbridge init --path ~/projects/brainbridge-workspace
echo "Quantum decoherence as a model for attention collapse" | brainbridge add --type hypothesis --tag physics,cognition
```
This creates a local graph store (SQLite-backed, with optional Postgres for multi-user). Each node gets a SHA256 hash as its ID, deduplicating identical content across sessions.

### 2. Create Weighted, Typed Edges Between Nodes
```bash
brainbridge link \
  --from <node_id_A> \
  --to <node_id_B> \
  --type extends \
  --weight 0.92 \
  --context "See also: Tegmark's decoherence paper"
```
You can batch-link via CSV or JSON. The `--context` field stores a human-readable note that is indexed for full-text search. Edges are stored as signed, timestamped entries—immutable by default, with optional soft-delete.

### 3. Run a Recursive Graph Query and Export
```bash
brainbridge query \
  "start node <node_id_A> traverse depth 4 filter edge.weight > 0.6 return path, node.content limit 20" \
  --format dot | dot -Tpng > thought_map.png
```
This generates a directed graph visualization. For programmatic access, use `--format json` and pipe into `jq` or your own analysis pipeline. The query DSL supports `union`, `intersect`, and `where` clauses on node metadata.

## Best Practices

### 1. Use Edge Typing to Model Epistemic Dependencies
In research or architecture docs, avoid generic "related to" links. Define a taxonomy of edge types (`supports`, `refutes`, `specializes`, `parallels`) and enforce it via a schema file. This turns your graph into a **formal argument map**, making it trivial to spot logical gaps or contradictory assumptions during code review or paper writing.

### 2. Integrate `brainbridge` into Your CI/CD Pipeline
Treat your thought graph as version-controlled documentation. Run `brainbridge validate` in CI to check for orphaned nodes (no incoming/outgoing edges) or broken external references. This ensures that your team’s shared mental model stays coherent across sprints, especially when onboarding new members or deprecating legacy concepts.
