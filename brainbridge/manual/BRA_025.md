# brainbridge - BRA_025

## Overview

**brainbridge** is a next-generation cognitive scaffolding tool engineered for developers, architects, and knowledge workers who need to externalize, structure, and traverse complex thought graphs at scale. Unlike conventional mind-mapping or note-taking software, brainbridge treats every idea, code snippet, or architectural decision as a first-class node in a bidirectional, queryable semantic network. It bridges the gap between raw ideation and structured execution — think of it as a low-latency neural cache for your engineering brain.

## Core Features

### 1. **Bi‑Directional Thought Graph Engine**
Every node (idea, task, code block) automatically generates implicit reverse links. When you link `Authentication Microservice` to `JWT Rotation Strategy`, brainbridge instantly creates a backlink from `JWT Rotation Strategy` to `Authentication Microservice`. This enables zero‑friction traversal: you can start from any leaf node and walk the entire dependency tree without manual tagging.

### 2. **Semantic Context Injection via LLM Hooks**
brainbridge integrates with local or remote LLMs to perform real‑time context enrichment. When you paste a stack trace or a PR description, the tool automatically extracts entities, relationships, and potential solutions — then injects them as new nodes linked to the original context. This turns passive note‑taking into an active, evolving knowledge base.

### 3. **Schema‑Aware Node Typing**
Define custom node schemas (e.g., `RFC`, `Bug`, `Architecture Decision Record`) with typed fields, validation rules, and automatic aggregation views. brainbridge dynamically generates aggregate dashboards — for example, a “Decision Debt” view that surfaces all ADRs with unresolved trade‑offs. This is not a generic Kanban; it’s a structured ontology engine for engineering teams.

## How to Use

### Step 1: Bootstrap a Thought Graph from Existing Codebase
Run `brainbridge scan --path ./src` to auto‑parse your project’s module structure, function signatures, and dependency imports. brainbridge will generate a base graph where each module is a node, each import is a directed edge, and each exported function is a typed sub‑node. You can then annotate nodes with architectural intent (e.g., “this module should be extracted to a microservice”) — the graph becomes a living architecture diagram.

### Step 2: Perform a Contextual Deep‑Link
While reviewing a bug report, highlight the error message and press `Ctrl+Shift+L`. brainbridge will query its local LLM to extract entities (e.g., `NullPointerException`, `UserService.login()`) and automatically create a new bug node linked to the relevant source code nodes. You can then add a `Reproduction Steps` field (schema‑enforced) and link it to the `UserService` node. This creates a traceable path from symptom to root cause.

### Step 3: Run a Cross‑Graph Query
Use the built‑in query language (similar to Cypher) to surface hidden patterns. For example:  
`MATCH (n:Bug)-[:AFFECTS]->(m:Module) WHERE n.severity = 'critical' RETURN m.name, count(n) AS bugCount ORDER BY bugCount DESC`  
brainbridge will render the result as an interactive subgraph — you can click any module node to expand its linked bugs, ADRs, and tests. This turns your knowledge base into a queryable, executable asset.

## Best Practices

### 1. **Treat Every RFC as a Node, Not a Document**
Instead of writing RFCs in Google Docs and linking them from brainbridge, create a dedicated `RFC` node type with fields like `Status`, `Decision`, `Trade‑offs`, and `Dependencies`. Then link RFC nodes to the relevant code modules and bug nodes. This makes RFCs queryable and keeps them in the same graph as the implementation — no more hunting for “the final version.”

### 2. **Establish a “Link‑First” Culture**
Before writing a long explanation in a node’s description, force yourself to create at least two explicit links to other nodes. This ensures that every piece of information is contextualized and discoverable. Over time, the graph becomes a self‑documenting map of your system’s rationale — onboarding new engineers becomes a graph traversal exercise, not a document reading marathon.
