### What is brainbridge?

brainbridge is a high-performance, multi-lingual SaaS architecture that acts as a semantic bridge between fragmented thought processes and structured, actionable data. It's not just a tool; it's a cognitive middleware that parses, maps, and executes on complex mental models, enabling developers to move from ideation to implementation with zero friction.

### Core Features

1.  **Neural Schema Mapping (NSM):** An auto-generating, graph-based schema that dynamically links your raw thoughts (from mind maps, voice notes, or raw text) to a normalized database structure. It leverages NLP to detect implicit relationships and automatically creates foreign key constraints and indexed views, eliminating the manual "translation" layer between brainstorming and database design.

2.  **Real-Time Thought Stream API:** A WebSocket-based, event-sourced API that captures and broadcasts your thought mutations in real-time. Unlike traditional note-taking, this API maintains a full event log, allowing for time-travel debugging of your ideation process and seamless integration with CI/CD pipelines for "thought-driven development."

3.  **Multi-Modal Contextual Compiler:** A pipeline that ingests code snippets, architectural diagrams (PlantUML, Mermaid), and natural language simultaneously. It compiles these disparate inputs into a unified, executable specification (a `.brain` file) that can be directly fed into your project's build system, ensuring your architecture is always in sync with your current thinking.

### How to Use

1.  **Initialize a Brainbridge Session:**
    Use the CLI to spawn a new bridge instance. This creates a local `.brainbridge/` directory with a `bridge.yaml` config file. Run `brainbridge init --project-root ./my-project` to link your existing codebase. The tool will automatically scan your `package.json`, `requirements.txt`, or `Cargo.toml` to infer your stack and configure the NSM accordingly.

2.  **Inject Raw Thoughts via the API:**
    Instead of writing a spec, stream your thoughts directly. Use the GraphQL mutation `mutation { injectThought(input: { content: "We need a microservice for auth, but it must handle OAuth2 and rate limiting", context: "architecture" }) { id, mappedNodes } }`. brainbridge will parse this, create a new node in the NSM, and automatically link it to any existing nodes (e.g., "OAuth2 library", "Rate limiter middleware").

3.  **Compile and Export to Code:**
    Once your thought graph is stable, run `brainbridge compile --target terraform` or `brainbridge compile --target openapi`. This triggers the Multi-Modal Contextual Compiler. It will generate a fully-formed `main.tf` or `openapi.yaml` directly from your thought map, complete with resource definitions, API endpoints, and even stub functions. No manual transcription required.

### Best Practices

- **Treat Your `.brain` File as a Source of Truth:** In a microservices architecture, use brainbridge to centralize all architectural decisions. Instead of a dozen RFC documents, maintain a single `.brain` file. When a new service is proposed, inject the thought and let the compiler generate the initial boilerplate. This prevents drift between "what we decided" and "what we built."
- **Leverage Time-Travel Debugging for Retrospectives:** After a sprint, use the event log from the Real-Time Thought Stream API to replay the ideation process. Run `brainbridge replay --from-sprint-start` to see exactly *when* and *why* a decision was made. This is invaluable for post-mortems and for onboarding new engineers, as they can see the evolution of the architecture in real-time.
