# brainbridge - BRA_002

## Overview

brainbridge is not another note-taking app or mind map toy. It’s a **cross-contextual thought orchestration engine** designed for developers, system architects, and knowledge workers who need to bridge fragmented mental models across codebases, documentation, and runtime environments. Built with a polyglot architecture (Rust core + WASM bindings for JS/Python/Ruby), brainbridge treats your ideas as **distributed graph nodes** with real-time sync, conflict resolution, and semantic compression.

## Core Features

### 1. **Semantic Chunking & Auto-Linking Engine**
brainbridge automatically segments your raw thoughts into **context-aware chunks** using a lightweight transformer (distilled from Sentence-BERT). Each chunk is fingerprinted and linked to related nodes via cosine similarity, dependency graphs, or manual “thought anchors.” No more manual tagging—the tool infers relationships between your brainstorming session, a PR comment, and a Slack thread.

### 2. **Bidirectional IDE & Terminal Integration**
Unlike passive tools, brainbridge runs as a **daemon process** (via `brainbridge daemon`) that exposes a gRPC endpoint. Plugins for VS Code, Neovim, and JetBrains allow you to `/bridge` any selection of code or log output directly into your thought graph. The terminal plugin (`bb`) can capture stdout/stderr streams and link them to your current cognitive context—perfect for debugging sessions or architectural RFC drafting.

### 3. **Conflict-Aware Merge for Thought Graphs**
When multiple users (or your own past/present selves) modify the same thought node, brainbridge applies a **three-way merge** inspired by Git’s recursive strategy but adapted for semantic content. It detects contradictions (e.g., “use Redis” vs. “use Memcached”) and flags them as **cognitive conflicts** with a diff view. You resolve via CLI or web UI, and the tool logs the resolution rationale for audit.

## How to Use

### 1. **Initialize a Thought Graph from a Codebase**
```bash
cd /path/to/your/monorepo
brainbridge init --scope ./src --name "monorepo-arch-v2"
brainbridge scan --depth 3 --include-logs
```
This creates a `.brainbridge/` directory with a SQLite-backed graph. The scan command indexes all source files, commit messages, and any `// BRIDGE:` comments you’ve left in code. Output: a visual graph of how modules, functions, and your architectural notes interconnect.

### 2. **Bridge a Real-Time Debugging Session**
```bash
brainbridge daemon &  # Start the background process
tail -f /var/log/app.log | bb stream --context "debug-session-20250321"
# In your editor, select a stack trace line, then run :BridgeSelection
```
The daemon captures each log line, deduplicates timestamps, and links the stack trace to your existing “error-handling” thought node. You can later query: `brainbridge query "show me all nodes related to 'connection timeout' since last deploy"`.

### 3. **Merge Two Divergent Thought Branches**
```bash
brainbridge branch list  # Shows active branches: "main", "feature/async-refactor"
brainbridge merge "feature/async-refactor" --into "main" --strategy semantic
```
If a conflict arises (e.g., one branch says “use asyncio” and the other “use trio”), brainbridge presents a **side-by-side semantic diff** with confidence scores. You accept, reject, or edit via `brainbridge merge --edit`. The resolved node gets a `resolved_by: <user>` and `resolution_timestamp` metadata.

## Best Practices

### 1. **Adopt a “One Graph Per Domain” Policy**
Don’t dump every thought into a single graph. Use `brainbridge init --domain "backend-architecture"` and `--domain "frontend-state-management"` separately. This keeps query latency low and prevents cognitive noise. Cross-link domains via `brainbridge link --from domain-a --to domain-b --label "shared-auth-context"`.

### 2. **Use Semantic Tags for Temporal Context**
Add `--tag "sprint-24"` or `--tag "pre-refactor"` when bridging sessions. Later, run `brainbridge query --tag "sprint-24" --output json | jq '.nodes[] | select(.type=="decision")'` to extract all architectural decisions made during that sprint. This becomes your **living ADR** (Architecture Decision Record) without extra tooling.
