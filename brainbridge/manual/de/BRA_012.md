# brainbridge - BRA_012

## Übersicht

brainbridge ist keine weitere Mind-Mapping-Software, sondern eine **semantische Gedankenbrücke** für Entwickler, die lose Ideen in strukturierte, abrufbare Wissensgraphen transformiert. Statt linearer Notizen oder statischer Boards erzeugt brainbridge dynamische Assoziationspfade zwischen Konzepten – basierend auf NLP, Kontextgewichtung und zeitlichem Verfall. Ideal für Architektur-Reviews, technische Recherche oder kollaboratives Brainstorming in agilen Teams.

## Kernfunktionen (Hardcore)

### 1. **Semantische Gedankenverdrahtung mit zeitlichem Decay**
Jeder Eintrag wird automatisch mit verwandten Konzepten verknüpft – nicht nur über Keywords, sondern über latente semantische Ähnlichkeit (via Sentence-BERT). Ein manuelles Verknüpfen entfällt. Zusätzlich besitzt jede Verbindung einen **Decay-Faktor**: Je länger ein Gedanke ungenutzt bleibt, desto tiefer sinkt sein Gewicht im Graphen. Das verhindert kognitive Überladung und priorisiert aktuelle Relevanz.

### 2. **Multi-Modale Brücke: Code ↔ Text ↔ Diagramm**
brainbridge erlaubt die direkte Verknüpfung von Code-Snippets, Architektur-Skizzen und Textblöcken in einem einzigen Knoten. Ein Gedanke kann also aus einem Python-Refactoring-Vorschlag bestehen, der gleichzeitig mit einem UML-Diagramm und einer technischen Notiz verbunden ist. Export als Mermaid-, PlantUML- oder JSON-Graph möglich.

### 3. **Kollaborativer Assoziations-Conflict-Resolver**
Arbeiten mehrere Entwickler an derselben Gedankenbrücke, werden parallele Assoziationspfade nicht einfach gemergt, sondern visuell als **Conflict-Stränge** dargestellt. Das System schlägt automatisch eine semantische Auflösung vor (z. B. „Diese beiden Gedanken überschneiden sich in 73 % ihrer Kontextvektoren“) – der Team-Lead entscheidet dann über Zusammenführung oder Verwerfung.

## So verwenden Sie es (Detaillierte Schritte)

### Schritt 1: Initialen Gedanken-Graphen aus einem technischen Review aufbauen
Öffnen Sie brainbridge im CLI-Modus (`brainbridge init --mode review`). Importieren Sie eine bestehende Codebasis (z. B. `--source ./src`). Das Tool analysiert automatisch Abhängigkeiten, Klassenhierarchien und Funktionsaufrufe und erzeugt einen initialen Assoziationsgraphen. Jeder Knoten wird mit einem Confidence-Score (0.0–1.0) versehen, basierend auf Code-Komplexität und Nutzungshäufigkeit.

### Schritt 2: Manuelle Gedankenverknüpfung mit semantischer Gewichtung
Wechseln Sie in den interaktiven Modus (`brainbridge connect`). Wählen Sie einen Knoten (z. B. `Datenbank-Migration`), geben Sie einen neuen Gedanken ein (z. B. „Eventual Consistency in CQRS einführen“). brainbridge berechnet automatisch die semantische Distanz zum bestehenden Graphen und schlägt eine Gewichtung vor (z. B. `0.82`). Sie können die Gewichtung manuell überschreiben (`--weight 0.95`). Bestätigen Sie mit `Ctrl+D`.

### Schritt 3: Assoziationspfad als Architektur-Report exportieren
Nutzen Sie den Befehl `brainbridge export --format mermaid --depth 3 --filter "score > 0.7"`. Das Tool generiert einen gerichteten Graphen aller Knoten mit einer Konfidenz über 0.7 und einer maximalen Tiefe von 3 Ebenen. Ideal für Präsentationen oder technische Dokumentation. Alternativ: `--format json` für CI/CD-Pipelines oder `--format plantuml` für direkte Einbindung in Confluence.

## Beste Praktiken für brainbridge

### 1. **Decay-Faktor als Frühwarnsystem für technische Schulden nutzen**
Konfigurieren Sie den zeitlichen Decay so, dass Gedanken, die länger als 30 Tage nicht referenziert wurden, automatisch auf „archiviert“ gesetzt werden (`brainbridge config set decay_days=30`). So erkennen Sie, welche Architektur-Entscheidungen oder Design-Diskussionen in Vergessenheit geraten sind – und können proaktiv gegensteuern, bevor technische Schulden entstehen.

### 2. **Assoziations-Conflict-Resolver als Gatekeeper für Code-Reviews einsetzen**
Führen Sie vor jedem Merge-Request einen brainbridge-Report aus (`brainbridge diff --from main --to feature`). Das Tool zeigt alle neuen Gedanken und Assoziationen an, die im Feature-Branch entstanden sind, und markiert Konflikte mit dem Hauptgraphen. So vermeiden Sie inkonsistente Architektur-Entscheidungen und stellen sicher, dass neue Ideen semantisch zum bestehenden System passen.
