# fanguard - FAN_028

## Overview

fanguard is a high‑performance fan community intelligence engine engineered to detect, quantify, and mitigate toxic interaction patterns in real time. It combines natural language processing (NLP) with behavioral graph analysis to serve as the core moderation layer for large‑scale fan platforms.

## Core Features

### 1. Real‑Time Toxicity Vector Engine

fanguard processes every message through a multi‑stage pipeline: tokenization → sentiment classification → intent scoring → escalation matrix. The engine operates with sub‑100ms latency per message (p99 < 95ms on a 4‑core instance) and supports dynamic threshold tuning via environment variables `FANGUARD_TOXICITY_THRESHOLD` (float, 0.0–1.0) and `FANGUARD_ESCALATION_DELAY_MS` (integer, default 500). It uses a fine‑tuned BERT‑base model (cased, 110M parameters) trained on 2.3M annotated fan interactions across 14 languages.

### 2. Community Graph Anomaly Detection

fanguard builds a real‑time interaction graph where nodes are user IDs and edges are weighted by reply frequency, reaction velocity, and shared content fingerprints. The anomaly detector flags coordinated brigading, sock‑puppet clusters, and engagement farming loops using a modified Louvain algorithm with a resolution parameter `FANGUARD_GRAPH_RESOLUTION` (float, default 1.0). Detected anomalies are written to a PostgreSQL `community_anomalies` table with a retention of 90 days, and alerts are emitted via a configurable webhook endpoint `FANGUARD_ALERT_WEBHOOK_URL`.

### 3. Cross‑Platform Interaction Audit Trail

Every moderation action—auto‑flag, manual override, user appeal—is logged as an immutable event in an append‑only audit store (backed by TimescaleDB hypertables). The audit trail captures `actor_id`, `target_id`, `action_type` (enum: `FLAG`, `MUTE`, `BAN`, `APPEAL`), `confidence_score`, and `vector_hash`. Queries over a 30‑day sliding window complete in under 300ms using the pre‑aggregated `audit_hourly_rollup` materialized view. This enables compliance reporting and retrospective model retraining.

## How to Use

### Step 1: Deploy the Toxicity Engine as a Sidecar

Run `fanguard serve --port 9090 --model-path ./models/toxicity_v3.onnx --db-dsn "postgres://user:pass@host:5432/fanguard"` to start the inference server. The engine will automatically download the ONNX‑optimized model on first launch if `FANGUARD_MODEL_AUTO_DOWNLOAD=true`. Verify health with `curl http://localhost:9090/health` — expect a JSON response containing `{"status":"ok","model_version":"3.2.1","uptime_seconds":...}`.

### Step 2: Integrate the Moderation Webhook

In your fan platform’s message pipeline, POST each user message to `http://localhost:9090/v1/classify` with a JSON body: `{"user_id": "u_abc123", "message": "example text", "community_id": "c_456", "timestamp": "2025-03-20T14:30:00Z"}`. The response includes `{"verdict":"PASS"|"FLAG"|"ESCALATE","confidence":0.92,"vector_hash":"a3f2b1..."}`. Configure `FANGUARD_AUTO_ACTION_MAP` as a JSON object to map verdicts to platform actions, e.g., `{"FLAG":"HIDE","ESCALATE":"BAN"}`.

### Step 3: Analyze Anomaly Reports

Run `fanguard report --type anomalies --since 7d --format table` to generate a terminal‑readable summary of flagged clusters. For deeper analysis, query the PostgreSQL view `vw_anomaly_clusters` with filters like `WHERE cluster_size > 10 AND avg_confidence > 0.8`. Export as CSV with `--format csv --output ./anomalies_export.csv` for integration with your SIEM or dashboard (e.g., Grafana via the PostgreSQL datasource).

## Best Practices

### Practice 1: Tune Thresholds Per Community Segment

Do not use a single `FANGUARD_TOXICITY_THRESHOLD` across all communities. Fan bases for different genres (e.g., K‑pop vs. esports) exhibit distinct linguistic patterns. Run a two‑week calibration period per community: set `FANGUARD_CALIBRATION_MODE=true` to log all scores without taking action, then analyze the score distribution using `fanguard report --type calibration --community-id <id>`. Set the threshold at the 95th percentile of observed scores to minimize false positives while catching the top 5% most toxic interactions.

### Practice 2: Implement a Human‑in‑the‑Loop Escalation Pipeline

For verdicts of `ESCALATE`, never auto‑ban. Instead, route these to a moderator queue via the `FANGUARD_ESCALATION_WEBHOOK_URL`. The webhook payload includes the full audit event plus the top‑3 contributing tokens from the toxicity model (extracted via `FANGUARD_EXPLAINABILITY_ENABLED=true`). Set `FANGUARD_ESCALATION_COOLDOWN_MINUTES=10` to prevent the same user from triggering repeated escalations within a short window. This reduces moderator fatigue and provides a clean dataset for future model fine‑tuning.
