# fanguard - FAN_015

## Overview

fanguard is a high-performance fan community protection and interaction platform engineered for real-time moderation, behavioral analytics, and cross-platform community health monitoring. It is purpose-built for large-scale fan ecosystems requiring sub-second threat detection and automated response workflows.

## Core Features

### 1. Real-Time Toxicity & Spam Detection Engine
Deploys a multi-layer neural inference pipeline (TensorFlow Lite + ONNX runtime) that classifies user-generated content across 12 toxicity categories (hate speech, harassment, spam, impersonation, etc.) with a measured inference latency of under 50 milliseconds per request. Supports custom keyword blacklists and regex-based pattern matching with hot-reload capability. All classifications are logged with a confidence score (0.00–1.00) and stored in a PostgreSQL time-series table partitioned by community ID and timestamp.

- **Parameter:** `--toxicity-threshold 0.85` (default; adjustable per community)
- **Metric:** 99.2% precision at 95% recall on internal benchmark dataset (FAN_015v3)

### 2. Fan Community Sentiment & Engagement Heatmap
Aggregates real-time sentiment scores (positive/neutral/negative) from chat, comments, and reaction data across Discord, Telegram, and custom WebSocket feeds. Outputs a per-minute heatmap JSON payload with fields: `community_id`, `timestamp_unix_ms`, `engagement_score` (0–1000), `sentiment_delta` (float, -1.0 to 1.0), and `top_emotion_labels`. Data is retained for 90 days with automatic rollup to hourly aggregates after 7 days.

- **Parameter:** `--heatmap-window 3600` (seconds for rolling window aggregation)
- **Metric:** 95th percentile ingestion latency < 200 ms per event

### 3. Automated Moderation Workflow Engine
Allows creation of multi-step moderation rules using a YAML-based DSL. Each rule can chain conditions (e.g., `user_reputation < 0.3 AND message_toxicity > 0.9`) with actions such as `mute`, `ban`, `flag_for_review`, or `send_webhook`. Supports dry-run mode for testing. All actions are recorded in an immutable audit log with SHA-256 hashing.

- **Parameter:** `--moderation-dry-run false` (set to `true` to simulate without enforcement)
- **Metric:** Rule evaluation throughput > 10,000 events/second per node

## How to Use

### Step 1: Initialize the fanguard Instance
```bash
fanguard init --community-id "fanbase_alpha" \
  --db-connection "postgresql://user:pass@localhost:5432/fanguard" \
  --toxicity-threshold 0.85 \
  --heatmap-window 3600
```
This creates the required PostgreSQL schema (auto-migration enabled), downloads the default toxicity model, and starts the background health-check goroutine.

### Step 2: Configure Moderation Rules
Create a file `rules.yaml`:
```yaml
rules:
  - name: "auto_mute_high_toxicity"
    conditions:
      - field: "message_toxicity"
        operator: ">"
        value: 0.9
      - field: "user_reputation"
        operator: "<"
        value: 0.3
    actions:
      - type: "mute"
        duration_seconds: 3600
      - type: "send_webhook"
        url: "https://hooks.example.com/mod-alert"
```
Apply with:
```bash
fanguard apply-rules --rules-file ./rules.yaml --community-id "fanbase_alpha"
```

### Step 3: Start the Real-Time Monitor
```bash
fanguard monitor --community-id "fanbase_alpha" \
  --feed-type "discord" \
  --feed-endpoint "wss://gateway.discord.gg/?v=10" \
  --output-format "json" \
  --log-level "info"
```
The monitor will print structured JSON events to stdout. Pipe to `jq` for filtering or redirect to a log aggregator.

## Best Practices

### 1. Tiered Moderation with Escalation
Define three tiers: automatic (threshold > 0.95), semi-automatic (0.85–0.95, requires human review within 5 minutes), and manual (flagged for senior moderator). Use the `--moderation-dry-run true` flag for 48 hours to calibrate thresholds before enabling enforcement. This reduces false positives by up to 40% in initial deployment.

### 2. Periodic Model Retraining with Community Feedback
Export flagged messages and moderator decisions weekly using `fanguard export-feedback --community-id "fanbase_alpha" --output ./feedback.csv`. Retrain the toxicity model using the provided `fanguard train --dataset ./feedback.csv --epochs 5` command. Schedule this as a cron job every Sunday at 02:00 UTC to keep precision above 98% as community language evolves.
