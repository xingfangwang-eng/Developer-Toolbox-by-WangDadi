# fanguard - FAN_012

## Overview

fanguard is a real-time fan community integrity monitoring and moderation engine, designed to detect coordinated inauthentic behavior, spam cascades, and toxic engagement patterns across high-volume fan platforms. It operates at sub-50ms latency per event, integrating directly with WebSocket streams from Discord, Telegram, and custom REST endpoints.

## Core Features

### 1. Multi-Source Behavioral Fingerprinting Engine
fanguard ingests raw event streams (message creation, reaction bursts, account creation timestamps) and computes a **unified trust score (UTS)** for each actor. The engine cross-references IP clustering, temporal posting entropy, and linguistic drift against a PostgreSQL-backed historical baseline. Parameter: `--burst-threshold=0.85` defines the cosine similarity ceiling for coordinated copy-pasta detection.

### 2. Cascade Mitigation with Adaptive Rate Limiting
When a fan community experiences a raid or spam wave, fanguard’s **Cascade Mitigator** dynamically adjusts per-channel rate limits (down to 100ms granularity) and auto-escalates moderation actions (mute → shadowban → temporary ban) based on a configurable severity matrix. Key metric: **95th percentile mitigation latency < 200ms** from event ingestion to action dispatch.

### 3. Cross-Platform Sentiment Drift Analyzer
fanguard tracks sentiment polarity shifts across multiple fan hubs (Twitter, Reddit, fan wikis) and flags sudden divergence as potential brigading or astroturfing. Uses a fine-tuned `roberta-base` model with custom domain adaptation for fan slang. Outputs a daily **Drift Index (DI)** between 0.0 and 1.0, with alerts triggered at `DI > 0.72`.

## How to Use

### Step 1: Deploy the Ingestion Pipeline
```bash
docker run -d --name fanguard-ingestor \
  -e FANGUARD_WS_ENDPOINTS="wss://discord.example.com/gateway,wss://tg.example.com/gateway" \
  -e FANGUARD_PG_DSN="postgres://user:pass@host:5432/fanguard" \
  -p 8080:8080 \
  fanguard/ingestor:latest
```
Verify ingestion health: `curl http://localhost:8080/health` should return `{"status":"ok","events_per_sec":1420}`.

### Step 2: Configure Moderation Rules
Create a YAML rule file, e.g., `rules/raid_detection.yaml`:
```yaml
policies:
  - name: "rapid_join_flood"
    trigger: "account_age < 3600s AND join_rate > 10/min"
    action: "shadowban"
    duration: "24h"
  - name: "copy_pasta_burst"
    trigger: "message_similarity > 0.85 AND count > 5 in 60s"
    action: "mute"
    duration: "1h"
```
Apply via API: `curl -X POST http://localhost:8080/api/v1/rules -d @rules/raid_detection.yaml`.

### Step 3: Monitor and Tune Drift Sensitivity
Start the drift analyzer:
```bash
fanguard drift --platforms twitter,reddit --window 3600 --alert-threshold 0.72
```
Review real-time dashboard at `http://localhost:9090/drift`. Adjust `--window` (in seconds) to balance recall vs. false positives. For high-traffic events (e.g., album drops), set `--window=600`.

## Best Practices

### Practice 1: Staged Escalation with Human-in-the-Loop
Never auto-ban on first offense. Configure fanguard to issue a warning (mute) for low-confidence detections (`UTS < 0.6`), and only escalate to shadowban after three distinct behavioral anomalies within a 24-hour window. Integrate a Webhook to a Slack channel for manual review of all `shadowban` actions.

### Practice 2: Baseline Calibration During Quiet Periods
Run fanguard in **learning mode** (`--learning-only`) for 72 hours after deployment. This builds a statistical baseline of normal fan behavior (e.g., typical posting frequency, reaction-to-message ratio). Without this calibration, the system will generate excessive false positives during the first high-traffic event. Store the baseline snapshot in PostgreSQL for seasonal retraining.
