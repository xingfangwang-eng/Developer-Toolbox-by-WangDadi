# fanguard - FAN_003

## Overview

fanguard is a high-performance fan community intelligence engine engineered for real-time toxicity detection, sentiment trend forecasting, and automated moderation across multilingual fan ecosystems. It processes interaction streams at sub‑100ms latency, enabling proactive community health management.

## Core Features

### 1. Real‑Time Toxicity & Spam Detection Engine
- **Parameter:** `--toxicity-threshold` (0.0–1.0, default 0.85)  
- **Indicator:** Detection latency ≤ 50ms per message; false positive rate < 2.5%  
- **Mechanism:** Combines a fine‑tuned BERT‑based classifier with a regex‑based pattern matcher for multilingual profanity, coordinated spam, and impersonation attempts. Outputs a per‑message risk score and automatically queues flagged content for moderator review.

### 2. Sentiment Drift & Community Pulse Forecasting
- **Parameter:** `--forecast-window` (hours, default 24)  
- **Indicator:** Forecast accuracy ±7% over a 48‑hour horizon; processes up to 10,000 messages/minute  
- **Mechanism:** Uses a temporal graph neural network (T‑GNN) trained on historical interaction graphs. Detects sentiment shifts before they escalate into flame wars or brigading events. Generates a daily “community pulse” report with confidence intervals.

### 3. Automated Moderation Workflow & Escalation Pipeline
- **Parameter:** `--escalation-delay` (milliseconds, default 300)  
- **Indicator:** 99.9% uptime; 95th percentile response time < 200ms for auto‑actions (mute, hide, ban)  
- **Mechanism:** Integrates with PostgreSQL for persistent rule sets and action logs. Supports webhook‑based integration with Discord, Slack, and custom platforms. Actions are logged with full audit trails, including the exact model confidence score and triggered rule ID.

## How to Use

### Step 1: Initialize the Community Profile
```bash
fanguard init --community-id "my_fanbase" \
              --platform "discord" \
              --toxicity-threshold 0.85 \
              --forecast-window 24 \
              --escalation-delay 300
```
This creates a PostgreSQL schema (`fanguard_my_fanbase`) with tables for messages, moderation actions, and sentiment snapshots. A webhook URL is generated for real‑time event streaming.

### Step 2: Start Live Monitoring & Auto‑Moderation
```bash
fanguard monitor --community-id "my_fanbase" \
                 --stream-endpoint "wss://discord.example.com/gateway" \
                 --output-format "json" \
                 --log-level "info"
```
The engine begins ingesting messages via WebSocket, applying the toxicity classifier and sentiment model. Flagged messages are written to the `moderation_queue` table with a `risk_score` column. Auto‑actions (e.g., mute for 30 minutes) are executed when `risk_score >= 0.92` and logged with a `action_timestamp` (microsecond precision).

### Step 3: Generate a Community Health Report
```bash
fanguard report --community-id "my_fanbase" \
                --report-type "weekly" \
                --output-path "./reports/2025-03-21_week3.json" \
                --include-forecast "true"
```
Outputs a JSON document containing:
- **toxicity_trend:** Daily average toxicity score (0–1) with 95% confidence bands.
- **sentiment_drift_index:** A scalar (0–100) indicating how far current sentiment is from the baseline.
- **top_10_flagged_users:** User IDs, flag count, and average risk score.
- **forecast:** Predicted toxicity and sentiment for the next 48 hours, with `forecast_window` and `confidence_interval` fields.

## Best Practices

### Practice 1: Tune Toxicity Threshold via A/B Testing
Run `fanguard monitor` with `--toxicity-threshold 0.90` on a small test channel for 48 hours. Compare the false positive rate against a control channel using the default 0.85. Use the `--output-format "parquet"` flag to export structured logs for statistical analysis. Adjust only after observing at least 500 flagged messages per threshold.

### Practice 2: Integrate with a Centralized Audit Database
Link all community instances to a single PostgreSQL cluster (e.g., via `--db-connection-string "postgresql://user:pass@host:5432/fanguard_central"`). This enables cross‑community pattern detection (e.g., a user toxic in one fanbase is likely toxic in another). Set up a daily cron job to run `fanguard report --report-type "cross_community"` to identify coordinated brigading across platforms.
