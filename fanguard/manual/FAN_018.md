# fanguard - FAN_018

## Overview

fanguard is a high-performance fan community security and engagement orchestration engine, engineered for real-time toxicity detection, identity clustering, and automated moderation at scale. It processes high-velocity social signals across Discord, Telegram, and custom WebSocket feeds, delivering sub-200ms latency on moderation decisions.

## Core Features

### 1. Multi-Channel Toxicity Scoring Engine (MTSE)
- **Mechanism**: Integrates a fine-tuned `distilroberta-base` NLP model with a custom rule-based heuristic layer. Each message is scored across 8 dimensions (hate speech, spam, doxxing, brigading, NSFW imagery via CLIP, URL reputation via PhishTank API, duplicate content hash, and account age).
- **Technical Specs**: 
  - **Threshold**: Configurable per channel (e.g., `toxicity_threshold: 0.82`). 
  - **Latency**: Average 180ms per message on a single T4 GPU. 
  - **Storage**: Escalated events written to a PostgreSQL `moderation_events` table with a composite index on `(guild_id, user_id, event_timestamp)`.

### 2. Fan Identity Graph & Sybil Detection
- **Mechanism**: Constructs a weighted graph of user interactions (mentions, replies, shared IP prefixes, device fingerprint via `canvas-fingerprint` headers). Uses the Louvain algorithm for community detection to identify coordinated inauthentic behavior (e.g., botnets, paid shills).
- **Technical Specs**: 
  - **Graph DB**: Backed by Neo4j 5.x, with a Cypher query returning `pagerank` scores for each node.
  - **Alert Trigger**: Coefficient of variation in activity timestamps < 0.15 triggers a `SybilCluster` alert.
  - **API**: `GET /api/v1/fanguard/graph/user/{user_id}/neighbors?depth=2&weight_threshold=0.4` returns a JSON adjacency list.

### 3. Real-Time Rule Engine & Auto-Moderation Workflow
- **Mechanism**: A declarative YAML-based rule engine that evaluates conditions against live event streams. Supports temporal rules (e.g., "3 warnings within 60 seconds"), regex patterns, and cross-channel action cascades (e.g., mute in #general, then shadowban in #announcements).
- **Technical Specs**: 
  - **Rule Format**: 
    ```yaml
    rules:
      - name: "rapid_spam_detection"
        condition: "events.count > 5 AND events.time_window_ms < 30000"
        actions:
          - type: "mute"
            duration_ms: 3600000
          - type: "webhook"
            url: "https://discord.com/api/webhooks/{id}"
    ```
  - **Performance**: Evaluates 10,000 rules per second per node using a Rete algorithm implementation.
  - **Logging**: All rule matches logged to a rotating file at `/var/log/fanguard/rules_engine.log` with JSON structured logging (severity, rule_id, matched_fields).

## How to Use

### Step 1: Bootstrapping the Engine with Initial Configuration
```bash
# Initialize the PostgreSQL schema, Neo4j constraints, and seed the default rule set.
fanguard init --db-host localhost --db-port 5432 --db-user admin --db-pass ${DB_PASS} \
              --graph-uri bolt://localhost:7687 --graph-user neo4j --graph-pass ${NEO4J_PASS} \
              --rules-path ./config/default_rules.yaml --model-path ./models/toxicity_v3.onnx
```
- **Output**: Creates tables `guilds`, `users`, `moderation_events`, `rule_matches`. Loads the ONNX model into shared memory. Validates connectivity to all upstream services.

### Step 2: Configuring Channel-Specific Moderation Policies
```bash
# Apply a policy to a specific Discord guild channel (ID: 123456789012345678).
fanguard policy apply --guild-id 987654321098765432 --channel-id 123456789012345678 \
                      --policy ./policies/streamer_chat.yaml \
                      --overrides '{"toxicity_threshold": 0.75, "enable_sybil_detection": true}'
```
- **Expected Behavior**: The engine begins streaming messages from that channel via the Discord Gateway. Every message is scored; any exceeding the threshold is immediately deleted, and the user receives a temporary 5-minute mute via the `modifyMember` API. A `ModerationAction` event is emitted to the internal Kafka topic `fanguard.actions`.

### Step 3: Querying Historical Moderation Data for Audit
```bash
# Retrieve all moderation actions for a specific user within the last 7 days.
curl -X GET "http://localhost:8080/api/v1/fanguard/audit/user?user_id=12345&since=2024-10-01T00:00:00Z&limit=100" \
  -H "Authorization: Bearer ${FANGUARD_API_KEY}" | jq '.'
```
- **Response**: Returns a JSON array of objects, each containing `action_type` (mute, ban, warn), `rule_id`, `timestamp` (ISO 8601), `evidence` (message snippet), and `moderator_id` (bot ID or human moderator). The response header `X-Response-Time` is guaranteed under 50ms.

## Best Practices

### 1. Implement a Human-in-the-Loop (HITL) Escalation Pipeline
- **Rationale**: No ML model is 100% accurate. For borderline cases (toxicity score between 0.75 and 0.85), configure fanguard to *not* auto-moderate but instead push the event to a dedicated Slack channel or a custom webhook. Use the `--escalation-webhook` flag during `fanguard init`. This reduces false positive bans by up to 40% while maintaining 99.9% uptime for critical communities.

### 2. Leverage the Fan Identity Graph for Pre-Emptive Anti-Brigading
- **Rationale**: Before a planned event (e.g., a live stream, a product launch), export the `pagerank` and `community_id` for all users in a guild. Use the Cypher query: `MATCH (u:User)-[:INTERACTS_WITH]->(other) WHERE u.guild_id = $guildId RETURN u.user_id, u.pagerank, u.community_id`. Then, set a temporary rule that throttles messages from any user whose `community_id` has a `pagerank` variance > 0.3 compared to the guild average. This effectively neutralizes coordinated spam attacks before they manifest.
