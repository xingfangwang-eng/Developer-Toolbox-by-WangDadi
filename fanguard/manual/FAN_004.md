# fanguard - FAN_004

## Overview

fanguard is an enterprise-grade fan community protection and interaction orchestration engine, designed to detect malicious engagement patterns, enforce community health policies in real-time, and provide granular analytics for fan-driven platforms. It operates at sub-50ms latency per decision node, integrating natively with PostgreSQL for persistent audit trails and Redis for ephemeral state management.

## Core Features

### 1. Real-Time Fan Sentiment & Toxicity Scoring Engine
Leverages a multi-model NLP pipeline (fine-tuned BERT + custom regex cascades) to classify fan interactions across 14 toxicity dimensions (e.g., harassment, spam, coordinated brigading). Each interaction is assigned a `toxicity_score` (float, 0.0–1.0) and a `confidence_interval` (95% CI). The engine supports hot-reloading of policy sets via a gRPC endpoint, enabling live updates without service disruption. All decisions are logged to a `fan_action_log` table in PostgreSQL with a retention policy of 90 days.

### 2. Dynamic Fan Cohort Segmentation & Policy Binding
Automatically segments fans into cohorts based on behavioral signals (e.g., `interaction_frequency`, `account_age_days`, `cross-platform_identity_hash`). Each cohort can be bound to a distinct `policy_profile` (JSON schema validated) that defines rate limits (`max_posts_per_minute`, `max_reports_per_hour`), content filters, and escalation rules. Cohort reassignment triggers a `fan_cohort_event` in the audit stream, ensuring full traceability. Supports up to 1,000 concurrent cohorts with a reconciliation latency of <200ms.

### 3. Multi-Version Fan Interaction Histogram & Anomaly Detection
Generates time-series histograms of fan interactions (e.g., `posts_per_hour`, `reactions_per_post`, `unique_reporters_per_incident`) with configurable bucket widths (1 min, 5 min, 1 hour). Anomaly detection uses a Holt-Winters seasonal model with automatic retraining every 24 hours. Deviations exceeding `3.0 * sigma` trigger a `fan_alert` webhook to a configurable endpoint (e.g., Slack, PagerDuty). Histogram data is stored in a dedicated `fan_histogram` table with a composite index on `(fan_id, timestamp)`.

## How to Use

### Step 1: Initialize the Fan Policy Engine
```bash
fanguard init --policy-file ./policies/default_policy.json --db-uri postgresql://user:pass@localhost:5432/fanguard --cache-uri redis://localhost:6379/0
```
This command validates the policy file against the `policy_profile` JSON schema, bootstraps the PostgreSQL schema (creating `fan_action_log`, `fan_cohort_event`, `fan_histogram` tables), and establishes the Redis connection pool. Use `--dry-run` to validate without side effects.

### Step 2: Ingest Fan Interactions & Trigger Scoring
```python
from fanguard import FanInteraction, ToxicityScorer

interaction = FanInteraction(
    fan_id="fan_78901",
    content="This is an example spam message with multiple links",
    source_platform="discord",
    timestamp=1710000000
)

scorer = ToxicityScorer(endpoint="grpc://localhost:50051")
result = scorer.score(interaction)
print(result.toxicity_score, result.policy_action)  # e.g., 0.87, "FLAG"
```
The `score` method returns a `ScoringResult` object containing `toxicity_score`, `policy_action` (one of `ALLOW`, `FLAG`, `BLOCK`, `ESCALATE`), and `applied_policy_profile`. All results are automatically written to `fan_action_log`.

### Step 3: Query Cohort Assignment & Histogram
```bash
fanguard cohort get --fan-id fan_78901 --format json
# Output: {"cohort_id": "cohort_high_risk", "policy_profile": "pp_aggressive", "assigned_at": 1710000000}

fanguard histogram query --fan-id fan_78901 --bucket-width 5m --start 1709900000 --end 1710000000 --output csv
# Output: timestamp,posts_per_hour,reactions_per_post
# 1709900100,12,3.4
# 1709900400,15,2.8
```
Use `--anomaly-threshold 3.0` to automatically flag anomalous buckets in the CSV output.

## Best Practices

### Best Practice 1: Gradual Policy Rollout with Shadow Mode
Before enforcing a new policy profile on all fans, deploy it in `shadow` mode. This executes the full scoring and segmentation logic but only logs the `policy_action` without enforcing it (i.e., all actions are `ALLOW`). Monitor the `fan_action_log` for false positives over a 48-hour window. A false positive rate exceeding 2% should trigger a policy revision. This prevents disruptive mass-flagging of legitimate fans.

### Best Practice 2: Database Indexing for Histogram Queries
For high-frequency fan interaction histograms (e.g., >1M events/hour), ensure the `fan_histogram` table has a composite index on `(fan_id, timestamp)` and a partial index on `(anomaly_flag)` where `anomaly_flag = TRUE`. This reduces query latency for anomaly review dashboards from ~3s to <50ms. Additionally, set `autovacuum` to run hourly on this table to prevent bloat from continuous inserts.
