# fanguard - FAN_024

## Overview

fanguard is a high‑performance fan community intelligence engine engineered to detect, analyze, and neutralize toxic interaction patterns in real time. It combines NLP‑driven sentiment profiling with behavioral anomaly detection to protect fan ecosystems from coordinated spam, harassment, and brand‑dilution attacks. Designed for communities exceeding 500K active members, fanguard operates at sub‑100ms latency per event.

## Core Features

### 1. Real‑Time Toxicity Propagation Mapping (TPM)
fanguard’s TPM engine ingests every message, reaction, and share event via WebSocket streams. It constructs a directed graph of user interactions and applies a custom **PoisonSpread** algorithm to identify cascading toxicity chains before they reach critical mass.  
- **Parameters:** `propagation_threshold` (default: 0.75), `decay_factor` (0.3–0.9), `max_depth` (3–10 hops).  
- **Performance:** 95th percentile detection latency < 80ms on Redis Streams + Apache Kafka pipeline.  
- **Output:** Real‑time heatmap of infected user clusters, exported as GeoJSON for moderation dashboards.

### 2. Multi‑Modal Sentiment Integrity Scoring (SIS)
Beyond simple keyword filtering, fanguard fuses text, emoji density, and reaction velocity into a single **Integrity Score** (0–100). It uses a fine‑tuned DistilBERT model (trained on 2.1M fan‑community messages) to classify intent: genuine enthusiasm, coordinated astroturfing, or malicious trolling.  
- **Parameters:** `sentiment_window_size` (50–500 events), `emoji_entropy_weight` (0.1–0.4), `reaction_velocity_zscore` (threshold: 3.0).  
- **Storage:** PostgreSQL with TimescaleDB hypertables for time‑series score history; queryable via SQL for custom audits.  
- **Accuracy:** 94.2% F1‑score on internal benchmark (FAN‑024‑BENCH‑v3).

### 3. Automated Community Quarantine & Remediation (CQR)
When a user or thread crosses the configurable **Quarantine Threshold** (`quarantine_score` ≥ 85), fanguard automatically:  
1. Silently isolates the offending content into a sandboxed channel (visible only to moderators).  
2. Triggers a dynamic rate‑limit escalation on the user’s account (e.g., 1 message per 300 seconds).  
3. Generates a forensic report containing full interaction history, sentiment drift graph, and recommended action (warn, temp‑ban, perm‑ban).  
- **Parameters:** `quarantine_cooldown` (1–24 hours), `escalation_steps` (3 levels), `report_format` (JSON / PDF).  
- **Integration:** REST API endpoints for Slack, Discord, and custom webhooks.

## How to Use

### Step 1: Deploy the Ingestion Gateway
```bash
docker run -d --name fanguard-ingest \
  -e FANGUARD_BROKER=kafka:9092 \
  -e FANGUARD_TOPIC=fan_events \
  -e FANGUARD_REDIS=redis://10.0.0.1:6379 \
  ghcr.io/xingfangwang-eng/fanguard:latest ingest
```
Configure your platform’s webhook to POST events to `http://<gateway>:8080/events`. Each payload must include `user_id`, `message_text`, `reactions[]`, and `timestamp` (ISO 8601).

### Step 2: Tune Detection Parameters
Edit `config/production.yaml`:
```yaml
toxicity:
  propagation_threshold: 0.75
  decay_factor: 0.6
  max_depth: 5
sentiment:
  model_path: "/models/distilbert-fanguard-v2"
  sentiment_window_size: 200
  emoji_entropy_weight: 0.25
quarantine:
  quarantine_threshold: 85
  quarantine_cooldown: 6h
  escalation_steps: 3
```
Validate config with `fanguard validate --config config/production.yaml` — it checks for schema correctness and model file integrity.

### Step 3: Start the Moderator Dashboard & Monitor Alerts
```bash
docker run -d -p 3000:3000 \
  -e FANGUARD_DB=postgresql://user:pass@pg-host:5432/fanguard \
  -e FANGUARD_REDIS=redis://10.0.0.1:6379 \
  ghcr.io/xingfangwang-eng/fanguard:latest dashboard
```
Open `http://localhost:3000`. The dashboard auto‑populates a live feed of flagged events. Click any event to view the forensic report (sentiment graph, user history, recommended action). Use the “Quarantine” button to manually trigger CQR or override the auto‑quarantine.

## Best Practices

### Practice 1: Calibrate Quarantine Thresholds with A/B Testing
Before enforcing auto‑quarantine on your entire community, run a 7‑day shadow mode (`quarantine_threshold: 100`) that logs all would‑be quarantines without action. Analyze false‑positive rate against known toxic content. Adjust `propagation_threshold` and `emoji_entropy_weight` iteratively until FPR drops below 2%. Only then enable auto‑quarantine at `quarantine_threshold: 85`.

### Practice 2: Integrate fanguard Alerts with Your Existing Incident Response Pipeline
fanguard’s webhook output (`POST /webhook/alert`) sends a structured JSON payload to any endpoint. Configure it to push alerts to PagerDuty, Opsgenie, or a dedicated Slack channel. Include the `quarantine_score` and `user_id` in the alert title. This ensures your moderation team can acknowledge, escalate, or dismiss incidents within 5 minutes of detection, maintaining SLA for community safety.
