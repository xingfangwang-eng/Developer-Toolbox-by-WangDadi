# fanguard - FAN_022

## Overview

fanguard is a high-performance fan community protection and moderation engine, designed for real-time abuse detection, identity verification, and cross-platform interaction analytics. It operates at sub-100ms latency for flagging toxic content and supports multi-tenant fan group management with PostgreSQL-backed event logs.

## Core Features

- **Real-Time Toxicity Radar**  
  Deploys a fine-tuned NLP model (BERT-based) to scan fan posts, comments, and DMs for harassment, spam, or impersonation. Configurable sensitivity thresholds (`toxicity_threshold: 0.72`), with per-channel whitelist/blacklist regex rules. Outputs structured JSON events to a Kafka pipeline for downstream moderation.

- **Fan Identity Trust Scoring**  
  Aggregates signals from OAuth providers (Twitter, Discord, Telegram) and on-chain wallet activity (EVM-compatible). Computes a `trust_score` (0.0–1.0) using weighted factors: account age (days), verified badge, report history, and cross-platform name match. Scores update every 15 minutes via cron job, stored in PostgreSQL with GIN-indexed JSONB columns.

- **Cross-Platform Interaction Bridge**  
  Enables fan communities to sync events (mentions, reposts, polls) between Discord, Telegram, and custom WebSocket endpoints. Uses a configurable `event_deduplication_window_ms: 500` to prevent duplicate processing. Supports `fan_group_id` scoping and role-based access via JWT tokens with 1-hour expiry.

## How to Use

1. **Initialize the Moderation Pipeline**  
   - Clone the repository and run `docker-compose up -d` to spin up PostgreSQL, Redis, and the fanguard worker.  
   - Set environment variables: `FANGUARD_TOXICITY_THRESHOLD=0.72`, `FANGUARD_KAFKA_BROKERS=kafka:9092`, `FANGUARD_DB_DSN=postgresql://user:pass@pg:5432/fanguard`.  
   - Execute `python cli.py init --fan-group-id=fan_022` to create the schema and seed default moderation rules.

2. **Configure Trust Scoring**  
   - Edit `config/trust.yaml` to define weight parameters:  
     ```yaml
     weights:
       account_age_days: 0.3
       verified_badge: 0.4
       report_history: -0.2
       cross_platform_match: 0.1
     ```  
   - Run `fanguard trust --recalculate --fan-group-id=fan_022` to compute scores for all registered users.  
   - Verify output via `SELECT user_id, trust_score FROM fan_trust WHERE fan_group_id='fan_022' AND trust_score < 0.5;`.

3. **Enable Cross-Platform Sync**  
   - In `config/bridge.yaml`, map external channels:  
     ```yaml
     bridges:
       - platform: discord
         channel_id: "123456789"
         fan_group_id: "fan_022"
       - platform: telegram
         chat_id: "-1001234567890"
         fan_group_id: "fan_022"
     ```  
   - Start the bridge service: `fanguard bridge --run --dedup-window-ms=500`.  
   - Monitor logs for `event_bridged` entries with `latency_ms` under 200.

## Best Practices

- **Tune Toxicity Threshold per Fan Group**  
  For high-traffic fan communities (e.g., K-pop, esports), set `toxicity_threshold` to 0.65 to catch subtle harassment early. For niche, low-volume groups, raise to 0.85 to reduce false positives. Always A/B test with a 10% shadow traffic sample before deploying.

- **Audit Trust Score Drift Weekly**  
  Schedule a cron job every Sunday at 02:00 UTC to run `fanguard trust --recalculate --fan-group-id=fan_022` and export drift reports. Compare current scores against a 7-day moving average. Flag any user whose score dropped >0.15 for manual review—this catches coordinated sock-puppet attacks.
