# fanguard - FAN_011

## Übersicht

Dieses Repository enthält die produktionsreife Implementierung des **fanguard**-Frameworks, einer hochperformanten, in Rust entwickelten Plattform zur Echtzeit-Identifikation und -Steuerung von Fan-Interaktionsmustern. Ziel ist die latenzoptimierte Analyse von Community-Signalen (≤ 5 ms pro Event) und die autonome Durchsetzung vordefinierter Moderations- und Engagement-Richtlinien.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine **ereignisgesteuerte, modulare Middleware** für Fan-Community-Plattformen. Es koppelt einen **In-Memory-Event-Stream** (Apache Arrow-basiert) an eine **PostgreSQL-Datenbank** für persistente Zustandsverwaltung. Die Kernaufgabe: Erkennung von anomalen Verhaltensmustern (z. B. Spam-Wellen, Brigading, Bot-ähnliche Aktivitäten) in < 50 Millisekunden und Ausführung von regelbasierten oder ML-gestützten Gegenmaßnahmen.

### Kernfunktionen

1. **Echtzeit-Anomalie-Detektion mit adaptiven Schwellwerten**  
   Implementiert einen **Exponential Moving Average (EMA)** über die `fanguard::metrics::TimeWindow`-Struktur (konfigurierbar: `window_size_ms: 30_000`, `threshold_sigma: 3.5`). Jede eingehende Aktion (`ActionType::Like`, `ActionType::Comment`, `ActionType::Follow`) wird gegen den aktuellen EMA geprüft. Überschreitet die Abweichung den Sigma-Faktor, wird ein `AlertEvent` erzeugt und über einen **Tokio-basierten, asynchronen Channel** an die Rule Engine gesendet. Parameter: `min_events_for_baseline: 100`, `decay_factor: 0.95`.

2. **Multi-Instanz-Konsens-Engine für verteilte Fan-Communities**  
   Nutzt **Raft-basiertes Consensus** (via `openraft`-Crate) zur Synchronisation von Moderationsentscheidungen über mehrere fanguard-Instanzen hinweg. Jede Instanz verwaltet einen lokalen `DecisionLog` (PostgreSQL-Tabelle: `fanguard_decisions` mit Spalten `decision_id UUID PRIMARY KEY, action_hash BYTEA, verdict TEXT, timestamp TIMESTAMPTZ`). Entscheidungen werden erst nach Bestätigung durch eine konfigurierbare Mehrheit (`consensus_quorum: 3` von `total_nodes: 5`) exekutiert. Latenz für einen vollständigen Konsens: < 200 ms bei 3 Knoten.

3. **Prädiktives Engagement-Routing mit dynamischer Lastverteilung**  
   Ein **Weighted-Round-Robin-Scheduler** (Algorithmus: `fanguard::routing::WeightedFairQueue`) verteilt eingehende Fan-Interaktionen auf bis zu **16 Worker-Threads**. Die Gewichtung basiert auf der aktuellen CPU-Auslastung (`/proc/stat`-Polling alle 500 ms) und der Queue-Tiefe (max `queue_depth: 10_000` Events). Ein **Feedback-Loop** passt die Gewichtung alle 2 Sekunden an (`adjustment_interval_s: 2`). Ziel: Maximierung des Durchsatzes bei gleichzeitiger Garantie einer maximalen Event-Verarbeitungszeit von **10 ms (p99)**.

### So verwenden Sie es

1. **Initialisierung und Konfiguration der Datenbankverbindung**  
   ```bash
   # Erstellen Sie die PostgreSQL-Datenbank und den Benutzer
   psql -h localhost -U postgres -c "CREATE DATABASE fanguard_db;"
   psql -h localhost -U postgres -c "CREATE USER fanguard_user WITH PASSWORD 'secure_password_123';"
   psql -h localhost -U postgres -c "GRANT ALL PRIVILEGES ON DATABASE fanguard_db TO fanguard_user;"

   # Führen Sie das Migrationsskript aus (im Repository unter /migrations/001_init.sql)
   psql -h localhost -U fanguard_user -d fanguard_db -f migrations/001_init.sql
   ```
   *Technische Anmerkung:* Die Migration erstellt die Tabellen `fanguard_events`, `fanguard_decisions` und `fanguard_metrics` mit den entsprechenden Indizes (z. B. `btree_index` auf `event_timestamp`).

2. **Starten des fanguard-Daemons mit benutzerdefinierten Parametern**  
   ```bash
   # Setzen Sie Umgebungsvariablen für die Produktion
   export FANGUARD_DB_CONNECTION_STRING="postgres://fanguard_user:secure_password_123@localhost:5432/fanguard_db"
   export FANGUARD_EMA_WINDOW_MS=30000
   export FANGUARD_EMA_THRESHOLD_SIGMA=3.5
   export FANGUARD_CONSENSUS_QUORUM=3
   export FANGUARD_WORKER_THREADS=8
   export FANGUARD_MAX_QUEUE_DEPTH=10000
   export RUST_LOG=info

   # Starten Sie den Daemon im Hintergrund
   ./target/release/fanguard --config /etc/fanguard/config.toml &
   ```
   *Hinweis:* Der Daemon lädt die Konfiguration aus der angegebenen `config.toml`. Umgebungsvariablen überschreiben die Datei-Parameter.

3. **Überwachen des Systemzustands und der Alert-Queue**  
   ```bash
   # Rufen Sie den Health-Endpunkt ab (HTTP GET)
   curl -X GET http://localhost:8080/health
   # Erwartete Antwort: {"status":"ok","uptime_secs":12345,"events_processed":67890,"alerts_fired":15}

   # Lesen Sie die letzten 10 Alerts aus der Datenbank
   psql -h localhost -U fanguard_user -d fanguard_db -c \
     "SELECT decision_id, action_hash, verdict, timestamp FROM fanguard_decisions ORDER BY timestamp DESC LIMIT 10;"
   ```
   *Interpretation:* Eine `verdict`-Spalte mit `"BLOCK"` oder `"FLAG_FOR_REVIEW"` zeigt die getroffene Maßnahme. Die `timestamp`-Spalte ist im UTC-Format.

### Beste Praktiken

- **Praxis 1: Gestaffelte Schwellwerte für unterschiedliche Community-Größen**  
  Verwenden Sie für Communities mit < 10.000 aktiven Nutzern einen `threshold_sigma` von 4.0, um False Positives zu minimieren. Für große Communities (> 100.000 Nutzer) senken Sie den Wert auf 3.0, da die statistische Basis robuster ist. Passen Sie den `decay_factor` abhängig von der durchschnittlichen Interaktionsfrequenz an: Für hochfrequente Umgebungen (z. B. Live-Streaming) setzen Sie `decay_factor: 0.98`, für Foren mit geringer Frequenz `decay_factor: 0.90`.

- **Praxis 2: Asynchrone Entscheidungslogik zur Vermeidung von Blockaden**  
  Konfigurieren Sie die Rule Engine so, dass sie **nicht blockierend** arbeitet. Nutzen Sie den integrierten `fanguard::engine::AsyncRule`-Trait, um I/O-lastige Operationen (z. B. Abfragen einer externen Blacklist-API) in einen separaten Tokio-Task auszulagern. Setzen Sie ein Timeout von `max_rule_execution_ms: 500`. Für den Fall, dass ein Timeout auftritt, definieren Sie eine Fallback-Regel (z. B. `verdict: "FLAG_FOR_REVIEW"`), um den Event-Stream nicht zu unterbrechen.
