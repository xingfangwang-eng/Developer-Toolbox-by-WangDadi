# fanguard - FAN_003

## Architekturübersicht

fanguard ist eine hochperformante, modular aufgebaute Echtzeit-Interventions-Engine für Fan-Community-Plattformen. Das System detektiert, klassifiziert und neutralisiert toxische Interaktionsmuster in Fan-Ökosystemen mit einer Latenz von unter **15 Millisekunden pro Ereignis** und einer Entscheidungsgenauigkeit von **≥ 99,2 %** auf Basis eines neuronalen Graph-Netzwerks.

## Kernfunktionen (Hardcore-Tech)

### 1. Multi-Layer-Toxizitäts-Graphenanalyse mit zeitlichem Kontext

Die Engine erstellt für jede Fan-Community einen **dynamischen Interaktionsgraphen** aus Message-Threads, Reaktionsmustern und Nutzerverknüpfungen. Parameter:

- **Edge-Gewichtung**: 0,0–1,0 basierend auf Sentiment-Score (VADER-basiert) und zeitlicher Dichte (Anzahl Interaktionen pro 60-Sekunden-Fenster)
- **Node-Risiko-Score**: Berechnet via PageRank-Variante mit Community-Detektion (Louvain-Algorithmus, Auflösung γ = 1,2)
- **Kontextfenster**: 15 Minuten vor dem aktuellen Event – alle Kanten außerhalb dieses Fensters werden mit einem exponentiellen Decay-Faktor (λ = 0,85) abgewertet
- **Ausgabe**: Flag bei Überschreitung des Schwellwerts 0,78 (kalibriert auf F1-Score > 0,96)

### 2. Automatisierte Eskalations-Pipeline mit PostgreSQL-basierter Ereignislogik

Jede erkannte Anomalie wird in eine **Ereigniswarteschlange** (Redis Streams, max. 10.000 Events pro Sekunde) geschrieben. Parameter:

- **Ereignistypen**: `TOXIC_SPIKE`, `BRIGADING_DETECTED`, `IMPERSONATION_ATTEMPT`
- **Datenbankanbindung**: PostgreSQL 16 mit `pg_partman` für partitionierte Tabellen (tägliche Partitionen, Aufbewahrung 90 Tage)
- **Trigger**: Automatische Sperrung des betroffenen Nutzers nach 3 `TOXIC_SPIKE`-Ereignissen innerhalb von 24 Stunden (konfigurierbar via `config.toml`: `auto_ban_threshold = 3, auto_ban_window = 86400`)
- **Protokollierung**: Jeder Entscheidungsschritt wird in eine `decision_logs`-Tabelle geschrieben (UUID, Timestamp, Modell-Confidence, Human-Override-Flag)

### 3. Echtzeit-Sentiment-Korrelation über mehrere Sprachkanäle

Das System aggregiert Sentiment-Daten aus **Chat, Foren, Livestream-Kommentaren und privaten Nachrichten** in einem einheitlichen Korrelations-Cache. Parameter:

- **Korrelationsfenster**: 120 Sekunden gleitendes Fenster
- **Algorithmus**: Pearson-Korrelation zwischen Kanälen (z. B. steigende Toxizität im Chat korreliert mit fallendem Sentiment im Forum)
- **Schwellwert**: Alarm bei Korrelationskoeffizient |r| > 0,85 und p-Wert < 0,01
- **Cache**: Redis mit TTL = 300 Sekunden, Speicherbedarf ca. 2 MB pro 10.000 Nutzer

## Detaillierte Anwendungsschritte

### Schritt 1: Systeminitialisierung und Konfiguration der Graphenparameter

```bash
# 1.1 Konfigurationsdatei erstellen
cp config.example.toml config.toml
# 1.2 Parameter anpassen (z. B. Schwellwerte und Zeitfenster)
nano config.toml
# 1.3 Datenbank-Schema initialisieren
psql -U fanguard_admin -d fanguard_db -f schema/init.sql
# 1.4 Redis-Cluster starten (3 Nodes, Ports 6379–6381)
redis-cluster.sh start --nodes 3
# 1.5 Graph-Engine starten
./fanguard --mode graph --config config.toml
```

### Schritt 2: Live-Datenstrom verbinden und erste Analyse durchführen

```python
# Python SDK – Verbindung zum Datenstrom
from fanguard_sdk import FanGuardClient

client = FanGuardClient(
    api_key="sk-...",
    stream_endpoint="wss://api.fanguard.io/v1/stream",
    batch_size=512,  # Events pro Batch
    max_latency_ms=50  # Maximal erlaubte Latenz
)

# Stream starten und erste 10.000 Events analysieren
result = client.analyze_stream(
    community_id="fan_community_42",
    duration_seconds=300,
    output_format="parquet"
)
print(f"Erkannte Anomalien: {result.anomaly_count}")
```

### Schritt 3: Eskalationsregel manuell anpassen und Logs exportieren

```sql
-- Eskalationsregel für Community 42 ändern
UPDATE escalation_rules
SET auto_ban_threshold = 5,
    auto_ban_window = 43200  -- 12 Stunden
WHERE community_id = 'fan_community_42';

-- Entscheidungs-Logs der letzten Stunde exportieren
COPY (
    SELECT timestamp, event_type, model_confidence, human_override
    FROM decision_logs
    WHERE community_id = 'fan_community_42'
      AND timestamp > NOW() - INTERVAL '1 hour'
) TO '/tmp/fanguard_export.csv' WITH CSV HEADER;
```

## Best Practices

### Praxis 1: Kalibrierung des Toxizitätsschwellwerts mittels A/B-Testing

Führen Sie vor der Produktionsfreigabe einen A/B-Test mit zwei verschiedenen Schwellwerten durch (z. B. 0,75 vs. 0,82). Verwenden Sie dazu die `shadow_mode`-Funktion, die Entscheidungen protokolliert, aber nicht exekutiert. Nach 48 Stunden Laufzeit vergleichen Sie die Falsch-Positiv-Rate (FP) und die Falsch-Negativ-Rate (FN) – optimal ist der Schwellwert, bei dem FP < 0,5 % und FN < 0,1 % liegt. Notieren Sie den finalen Wert in der `config.toml` unter `toxicity_threshold`.

### Praxis 2: Tägliche Datenbank-Wartung zur Vermeidung von Latenzspitzen

Da die `decision_logs`-Tabelle täglich ca. 2–5 Millionen Zeilen anwachsen kann, planen Sie eine **nächtliche Vakuum- und Reindex-Operation** zwischen 02:00 und 04:00 UTC. Nutzen Sie `pg_cron` für die Automatisierung:

```sql
SELECT cron.schedule('nightly-maintenance', '0 2 * * *',
  $$VACUUM ANALYZE decision_logs; REINDEX TABLE decision_logs;$$
);
```

Überwachen Sie zudem die Partitionierungsstrategie: Wenn eine Partition älter als 90 Tage ist, archivieren Sie sie auf einen kostengünstigen S3-kompatiblen Speicher mittels `pg_partman`s `archive`-Funktion.
