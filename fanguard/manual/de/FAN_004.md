# fanguard - FAN_004

## Übersicht

Dieses Repository enthält die produktionsreife Implementierung von **fanguard**, einer Hochleistungsplattform zur Verwaltung und Absicherung von Fan-Communitys. Die Architektur ist auf submillisekundengenaue Echtzeit-Interaktionen ausgelegt und integriert sich nahtlos in bestehende Fan-Engagement-Ökosysteme. Zielsprache der Benutzeroberfläche ist Deutsch (de).

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine modulare, ereignisgesteuerte Plattform, die speziell für die Analyse, Moderation und Optimierung von Fan-Interaktionen entwickelt wurde. Sie nutzt einen **Multi-Layer-Trust-Score** (MLTS), der aus über 200 Metriken besteht, um toxisches Verhalten in Millisekunden zu erkennen und zu unterbinden. Die Plattform unterstützt **PostgreSQL 16** als primäre Datenbank für Transaktionsdaten und **Redis Stack** für Echtzeit-Caching und Pub/Sub-Kommunikation.

### Kernfunktionen

1.  **Echtzeit-Toxizitätsdetektion mit adaptiven Schwellwerten**
    *   **Parameter:** `toxicity_threshold` (float, 0.0–1.0, Standard: 0.85), `context_window_ms` (int, 500–5000, Standard: 1500), `model_version` (string, z. B. `v3.2.1`).
    *   **Beschreibung:** Verwendet ein feinabgestimmtes Transformer-Modell (DistilBERT), das auf deutschsprachigen Fan-Foren trainiert wurde. Die Analyse erfolgt in Echtzeit mit einer Latenz von < 50 ms pro Nachricht. Der Schwellwert passt sich dynamisch an die Historie der Community an (z. B. erhöht bei Vandalismuswellen).

2.  **Multi-Community-Richtlinien-Engine (MCRE)**
    *   **Parameter:** `policy_id` (UUID), `ruleset` (JSON, bis zu 10 MB), `enforcement_level` (enum: `log`, `warn`, `mute`, `ban`).
    *   **Beschreibung:** Ermöglicht die granulare Definition von Moderationsregeln pro Fan-Gruppe. Die Engine wertet bis zu 10.000 Regeln pro Sekunde aus und protokolliert jede Aktion in einer unveränderlichen Audit-Tabelle (`audit_logs`). Unterstützt Bedingungen wie `if user.trust_score < 0.3 AND message.sentiment < -0.7`.

3.  **Community-Heatmap und Engpass-Analyse**
    *   **Parameter:** `time_granularity` (enum: `1m`, `5m`, `1h`), `metric` (enum: `post_frequency`, `report_count`, `sentiment_volatility`).
    *   **Beschreibung:** Generiert automatisch Heatmaps der Aktivität über alle Kanäle hinweg. Identifiziert Engpässe (z. B. Überlastung eines Moderationsteams) und schlägt automatisch die Skalierung von Worker-Pools vor. Die Daten werden in **TimescaleDB** (PostgreSQL-Erweiterung) gespeichert und aggregiert.

### So verwenden Sie es

1.  **Initialisierung der Datenbank und des ML-Modells**
    ```bash
    # 1. PostgreSQL-Datenbank erstellen und Schema migrieren
    psql -U admin -d fanguard_db -f schema/v4.2/init_schema.sql
    
    # 2. ML-Modell herunterladen und in den Modell-Store laden
    python scripts/download_model.py --model-name "fanguard-toxicity-de-v3" --output-dir ./models/
    
    # 3. Konfigurationsdatei anpassen (config/production.yaml)
    #    - database.host: postgres-cluster.internal
    #    - redis.host: redis-cluster.internal
    #    - ml.model_path: ./models/fanguard-toxicity-de-v3.onnx
    ```
    *   **Erwartetes Ergebnis:** Die Plattform startet und verbindet sich mit der Datenbank. Der Health-Check-Endpunkt `GET /health` gibt `{"status": "ok", "model_loaded": true}` zurück.

2.  **Erstellen einer Community und Definieren von Moderationsregeln**
    ```bash
    # 1. Neue Community anlegen
    curl -X POST http://localhost:8080/api/v1/communities \
      -H "Content-Type: application/json" \
      -d '{"name": "FC Bayern Fanclub", "language": "de", "default_toxicity_threshold": 0.8}'
    
    # 2. Regel hinzufügen: Automatischer Mute bei wiederholten Beleidigungen
    curl -X POST http://localhost:8080/api/v1/policies \
      -H "Content-Type: application/json" \
      -d '{"community_id": "<ID>", "ruleset": {"conditions": [{"metric": "report_count", "operator": ">", "value": 3, "within_seconds": 300}], "action": "mute", "duration_minutes": 60}}'
    ```
    *   **Erwartetes Ergebnis:** Die Community ist aktiv. Sobald ein Benutzer 3 Meldungen innerhalb von 5 Minuten erhält, wird er automatisch für 60 Minuten stummgeschaltet.

3.  **Überwachung der Live-Heatmap und Skalierung**
    ```bash
    # 1. Heatmap für die letzte Stunde abrufen (Metrik: report_count)
    curl -X GET "http://localhost:8080/api/v1/analytics/heatmap?community_id=<ID>&time_granularity=5m&metric=report_count"
    
    # 2. Automatische Skalierung eines Worker-Pools auslösen (falls Heatmap einen Engpass anzeigt)
    #    Die Plattform erkennt dies selbstständig, kann aber manuell angestoßen werden:
    curl -X POST http://localhost:8080/api/v1/admin/scale-workers \
      -H "Content-Type: application/json" \
      -d '{"pool_name": "moderation_workers", "target_count": 15}'
    ```
    *   **Erwartetes Ergebnis:** Die Heatmap zeigt ein Balkendiagramm der Meldungen. Bei Überschreitung eines Schwellwerts (z. B. > 100 Meldungen/5min) wird automatisch ein Alert an den Operations-Manager gesendet.

### Beste Praktiken

1.  **Granulare Vertrauensstufen statt Binärer Sperren**
    *   **Empfehlung:** Nutzen Sie den Multi-Layer-Trust-Score (MLTS) als primäres Steuerungsinstrument. Anstatt einen Benutzer sofort zu sperren, senken Sie seinen Score schrittweise. Ein Score unter 0.2 kann automatisch zu einem 24h-Mute führen, unter 0.1 zu einer temporären Sperre. Dies reduziert False Positives und gibt der Community die Chance zur Besserung. Konfigurieren Sie die `score_decay_rate` (Standard: 0.01 pro Tag), um inaktive Benutzer nicht zu bestrafen.

2.  **Datengetriebene Regeloptimierung mit A/B-Testing**
    *   **Empfehlung:** Aktivieren Sie den `policy_ab_testing`-Modus in der Konfiguration (`config/production.yaml` → `experimentation.enabled: true`). Definieren Sie zwei Versionen einer Regel (z. B. unterschiedliche `toxicity_threshold`-Werte) und lassen Sie die Engine automatisch die Version mit der höheren Benutzerzufriedenheit (gemessen an `sentiment_volatility` und `report_count`) auswählen. Die Ergebnisse werden in einer separaten Tabelle (`ab_test_results`) gespeichert und können mit `SELECT * FROM ab_test_results ORDER BY conversion_rate DESC;` ausgewertet werden.
