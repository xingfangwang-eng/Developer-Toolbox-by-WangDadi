# fanguard - FAN_029

## Übersicht

fanguard ist eine hochperformante Echtzeit-Orchestrierungsplattform für Fan-Community-Management, ausgelegt auf sub-50ms Latenz bei 10.000 gleichzeitigen Nutzern. Das System aggregiert und analysiert Fan-Interaktionsdaten aus Discord, Telegram und proprietären Foren, um automatisierte Governance- und Engagement-Strategien zu ermöglichen.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine auf Rust und Tokio basierende Middleware, die eine **Multi-Queue-Event-Engine** mit PostgreSQL und Redis Streams kombiniert. Es ermöglicht Community-Managern, granulare Filterregeln (z. B. `spam_score > 0.85` oder `sentiment < -0.3`) auf eingehende Nachrichten anzuwenden und Aktionen wie temporäre Mutes oder automatisierte Antworten auszulösen. Die Plattform speichert alle Metadaten in einer partitionierten PostgreSQL-Datenbank (Sharding über `fan_group_id`), während der Zustand der aktiven Sitzungen im Redis-Cluster gehalten wird.

### Kernfunktionen

- **Echtzeit-Multi-Stream-Sentiment-Analyse mit dynamischer Gewichtung**  
  Das System verarbeitet bis zu 25.000 Nachrichten pro Sekunde (Nachrichten-Throughput) und analysiert den Sentiment-Score in Echtzeit (maximale Latenz: 45 ms). Für jede Nachricht wird ein `sentiment_score` (Float, -1.0 bis 1.0) berechnet, der in eine gewichtete `action_priority` (Integer, 0–100) einfließt. Die Gewichtung basiert auf dem `fan_trust_level` (Integer, 1–10) des Absenders und der `channel_priority` (Enum: `low`, `medium`, `high`, `critical`). Parameter: `sentiment_threshold: -0.2`, `spam_threshold: 0.8`.

- **Automatisiertes Rule-Engine mit SQL-basierten Bedingungen**  
  Administratoren definieren Regeln in einer YAML-Datei, die zur Laufzeit in PostgreSQL-ähnliche WHERE-Klauseln übersetzt werden. Beispielregel: `WHERE sentiment_score < -0.5 AND fan_trust_level < 5 AND message_length > 200`. Die Engine evaluiert diese Bedingungen für jede eingehende Nachricht in unter 2 ms (99. Perzentil). Jede Regel hat einen `cooldown_seconds` (Integer, Standard: 300) und eine `max_actions_per_minute` (Integer, Standard: 10). Bei Überschreitung wird die Aktion in eine Dead-Letter-Queue (Redis-Liste `fanguard:dlq`) verschoben.

- **Multi-Node-Konfigurationssynchronisation via etcd**  
  Bei einem Cluster mit 3+ Knoten wird die gesamte Konfiguration (Regeln, Whitelists, Blacklists) über etcd (Key-Value-Store) synchronisiert. Die Write-Latenz für Konfigurationsänderungen beträgt maximal 200 ms. Jeder Knoten hostet einen gRPC-Health-Check-Endpunkt (`/fanguard.v1.Health/Check`), der alle 5 Sekunden den Status `SERVING` oder `NOT_SERVING` meldet. Bei einem Knotenausfall werden die aktiven Verbindungen automatisch auf die verbleibenden Knoten umgeleitet (durchschnittliche Failover-Zeit: 1,2 Sekunden).

### So verwenden Sie es

1. **Initiale Konfiguration und Datenbank-Setup**  
   Führen Sie das Skript `scripts/setup_db.sh` aus, das automatisch die PostgreSQL-Partitionierung für `fan_group_id` (Modulo-8-Sharding) und die Redis-Streams (`fanguard:inbound`, `fanguard:outbound`, `fanguard:dlq`) erstellt. Setzen Sie die Umgebungsvariablen `FANGUARD_DB_URL` (z. B. `postgres://user:pass@host:5432/fanguard`) und `FANGUARD_REDIS_URL` (z. B. `redis://host:6379/0`). Überprüfen Sie die Verbindung mit `cargo run --bin healthcheck`.

2. **Regeldefinition und Deployment**  
   Erstellen Sie eine Datei `rules/fan_protection.yaml` mit folgendem Inhalt:  
   ```yaml
   rules:
     - name: "toxicity_auto_mute"
       condition: "sentiment_score < -0.6 AND fan_trust_level < 4"
       action: "mute"
       action_params:
         duration_seconds: 3600
         reason: "Auto-mute due to high toxicity"
       cooldown_seconds: 600
       max_actions_per_minute: 5
   ```  
   Führen Sie `fanguard-cli apply rules/fan_protection.yaml` aus. Das CLI validiert die YAML gegen ein JSON-Schema und lädt die Regeln in den etcd-Cluster. Überprüfen Sie den Status mit `fanguard-cli status`.

3. **Monitoring und Fehlerbehebung**  
   Nutzen Sie das integrierte Prometheus-Metrik-Endpunkt (`/metrics`), der unter anderem `fanguard_messages_processed_total`, `fanguard_rule_evaluation_duration_seconds` und `fanguard_dlq_size` bereitstellt. Bei einer hohen DLQ-Größe (z. B. > 1000) führen Sie `fanguard-cli dlq replay` aus, um die Nachrichten erneut zu verarbeiten. Verwenden Sie `fanguard-cli logs --tail 100`, um die letzten 100 Log-Einträge (strukturiertes Logging im JSON-Format) anzuzeigen.

### Beste Praktiken

- **Granulare Trust-Level-Definition für präzise Filterung**  
  Definieren Sie mindestens 5 Trust-Level (1 = Neuankömmling, 5 = Veteran, 10 = Moderator). Verknüpfen Sie den Trust-Level mit der Anzahl der positiven Interaktionen (z. B. `interaction_count > 100` für Level 3). Dies verhindert, dass neue Benutzer fälschlicherweise von aggressiven Regeln getroffen werden, während wiederkehrende Störer (niedriger Trust-Level, hohe Spam-Scores) sofort isoliert werden.

- **Stufenweise Regelaktivierung mit A/B-Testing-Ansatz**  
  Aktivieren Sie neue Regeln zunächst in einem separaten `staging`-Namespace (etcd-Prefix `/fanguard/staging/`). Leiten Sie 10 % des Traffics dorthin um, indem Sie den `fan_group_id`-Hash auf den staging-Namespace mappen. Überwachen Sie die Metriken (`false_positive_rate`, `user_complaint_rate`) für 48 Stunden, bevor Sie die Regel in den Produktions-Namespace (`/fanguard/production/`) übernehmen. Dies reduziert das Risiko von Massen-Bans aufgrund fehlerhafter Regeln erheblich.
