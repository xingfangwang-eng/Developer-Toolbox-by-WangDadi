# fanguard - FAN_027

## Übersicht

fanguard ist eine hochperformante, modulare Plattform zur Echtzeit-Überwachung und Steuerung von Fan-Community-Interaktionen. Das System analysiert Metriken wie Sentiment-Index, Engagement-Rate und Toxizitätsschwellwerte in Millisekunden und ermöglicht eine automatisierte, regelbasierte Moderation ohne manuelle Eingriffe. Die Architektur basiert auf einer asynchronen Event-Queue (RabbitMQ) und einer PostgreSQL-Datenbank mit partitionierten Tabellen für historische Analysen.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine skalierbare, ereignisgesteuerte Backend-Engine, die speziell für die Verwaltung großer, dezentraler Fangruppen entwickelt wurde. Es aggregiert Daten aus mehreren Quellen (Discord, Telegram, Eigenentwicklung) und wendet ML-gestützte Filter (TensorFlow Lite) zur Erkennung von Spam, Hate Speech und Koordinationsversuchen an. Die Latenz zwischen Ereigniseingang und Entscheidungsausgabe beträgt < 50 ms.

### Kernfunktionen

1. **Dynamische Toxizitätsanalyse mit Schwellwert-Adaption**  
   Das System berechnet einen kontinuierlichen Toxizitäts-Score (0.0 – 1.0) für jeden Nutzer basierend auf Frequenz, Wortwahl und Interaktionsmuster. Der Schwellwert (`toxicity_threshold`) kann pro Community dynamisch via API angepasst werden. Bei Überschreitung wird automatisch eine Aktion ausgelöst (z. B. temporäre Stummschaltung, Log-Eintrag in `moderation_log`). Die Berechnung erfolgt in Echtzeit mit einer maximalen Verzögerung von 30 ms pro Nachricht.

2. **Multi-Plattform-Event-Aggregation mit deduplizierter Queue**  
   fanguard integriert nativ Webhooks von Discord (`channel_id`, `guild_id`) und Telegram (`chat_id`, `message_thread_id`). Alle eingehenden Events werden in einer zentralen, deduplizierten Queue (RabbitMQ, `fan_events` Exchange) gespeichert. Die Deduplizierung basiert auf einem SHA-256-Hash aus `source`, `user_id` und `timestamp_ms`. Dies verhindert Doppelverarbeitung und reduziert die Last auf der PostgreSQL-Datenbank um bis zu 40 %.

3. **Automatisiertes Eskalationsmanagement mit Zeitfenster-Logik**  
   Das System definiert Eskalationsstufen (Level 1–5). Jede Stufe hat ein eigenes Zeitfenster (`escalation_window_ms`) und eine Aktion (z. B. Level 3: automatische Meldung an Admin via E-Mail). Die Logik wird in einer YAML-Konfigurationsdatei (`escalation_rules.yaml`) definiert und kann ohne Neustart neu geladen werden. Bei Überschreitung von Level 5 wird ein vollständiger Dump der letzten 500 Nachrichten des Nutzers in ein separates `forensic_log`-Table geschrieben.

### So verwenden Sie es

1. **Konfiguration der Datenquellen**  
   Erstellen Sie eine Datei `sources.yaml` mit folgendem Format:
   ```yaml
   sources:
     - type: discord
       guild_id: "123456789"
       channel_id: "987654321"
       bot_token: "env:DISCORD_BOT_TOKEN"
     - type: telegram
       chat_id: "-1001234567890"
       bot_token: "env:TELEGRAM_BOT_TOKEN"
   ```
   Starten Sie den Event-Listener mit `fanguard listen --config sources.yaml`. Das System öffnet persistente WebSocket-Verbindungen und beginnt mit der Aggregation.

2. **Festlegen der Toxizitätsschwelle**  
   Setzen Sie den Schwellwert für eine Community via API:
   ```bash
   curl -X POST https://localhost:8443/api/v1/community/toxicity \
     -H "Content-Type: application/json" \
     -d '{"community_id": "fan_027", "toxicity_threshold": 0.75, "action": "mute"}'
   ```
   Der Endpunkt antwortet mit einem JSON-Objekt, das die aktuelle `threshold_version` und die Anzahl der betroffenen Nutzer (`affected_users`) enthält. Änderungen werden innerhalb von 100 ms aktiv.

3. **Überwachung der Eskalationslogik**  
   Aktivieren Sie das Dashboard-Modul mit:
   ```bash
   fanguard dashboard --port 9090 --db postgresql://user:pass@localhost:5432/fanguard
   ```
   Das Dashboard zeigt einen Echtzeit-Feed aller Eskalationen an. Sie können manuell eingreifen, indem Sie auf eine Eskalation klicken und eine Aktion auswählen (z. B. „Nutzer freigeben“). Alle manuellen Aktionen werden in der Tabelle `manual_interventions` mit Zeitstempel und Admin-ID protokolliert.

### Beste Praktiken

- **Segmentierung der Communities nach Risikoprofil**  
  Definieren Sie für jede Community separate `toxicity_threshold`-Werte basierend auf historischen Daten. Verwenden Sie dazu das Tool `fanguard analyze --community fan_027 --days 30`, das automatisch einen optimalen Schwellwert vorschlägt (basierend auf dem 95. Perzentil der Toxizitäts-Scores). Dies reduziert Fehlalarme um bis zu 60 %.

- **Regelmäßige Überprüfung der Eskalationsregeln**  
  Führen Sie alle 7 Tage einen `fanguard audit --rules escalation_rules.yaml` durch. Das Tool simuliert 1000 zufällige Events und prüft, ob die Eskalationslogik korrekt greift. Bei Abweichungen werden Warnungen mit detaillierten Logs ausgegeben. Integrieren Sie diesen Audit in Ihre CI/CD-Pipeline (z. B. als GitHub Action).
