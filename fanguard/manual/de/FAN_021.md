# fanguard - FAN_021

## Übersicht

fanguard ist eine hochperformante Analyse- und Steuerungsplattform für Fan-Ökosysteme. Sie erkennt toxische Interaktionsmuster in Echtzeit, segmentiert Nutzerkohorten nach Engagement-Tiefe und ermöglicht eine granulare Steuerung von Community-Richtlinien. Das System verarbeitet bis zu 50.000 Events pro Sekunde bei einer Latenz von unter 200 Millisekunden.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist ein auf Event-Sourcing und regelbasierten Entscheidungsbäumen aufbauendes Backend-System zur Überwachung, Analyse und Steuerung von Fan-Interaktionen. Es aggregiert Daten aus Chat-Verläufen, Reaktionsmustern und Freigabeverhalten in einer PostgreSQL-Datenbank mit TimescaleDB-Erweiterung. Die Ausgabe erfolgt als JSON-Dashboard mit Metriken wie „Toxizitätsindex (0–100)“, „Engagement-Kohorte (A–D)“ und „Aktionswahrscheinlichkeit in Prozent“.

### Kernfunktionen

- **Echtzeit-Toxizitätsdetektion mit gewichteten Schlüsselwortmatrizen**  
  Jeder eingehende Text wird gegen eine konfigurierbare Matrix aus 10.000+ Schlüsselwörtern und Phrasen geprüft. Die Gewichtung erfolgt über einen Vektor aus `[keyword_id, weight, context_score]`. Ergebnisse werden in Millisekunden berechnet und als `toxicity_score` (Float, 0.0–100.0) ausgegeben.

- **Dynamische Kohortenbildung auf Basis von Interaktions-Tiefe**  
  Nutzer werden anhand von Parametern wie `message_frequency_7d`, `reaction_ratio` und `report_count` in vier Kohorten eingeteilt. Die Segmentierung erfolgt automatisch alle 6 Stunden über eine PostgreSQL-gestützte Batch-Verarbeitung. Ausgabe ist eine JSON-Struktur mit `cohort_id`, `member_count` und `avg_toxicity`.

- **Regelbasierte Aktionsauslösung mit konfigurierbaren Schwellwerten**  
  Administratoren definieren Regeln im Format `IF toxicity_score > 80 AND cohort = 'C' THEN action = 'shadow_mute'`. Jede Regel wird über einen asynchronen Worker ausgeführt, der Änderungen in unter 50 Millisekunden an die Steuerungsebene weiterleitet. Alle Aktionen werden in einer `action_log`-Tabelle mit Timestamp und User-ID protokolliert.

### So verwenden Sie es

1. **Konfiguration der Toxizitätsmatrix**  
   Erstellen Sie eine JSON-Datei mit dem Schema `[{"keyword": "Beleidigung", "weight": 0.7, "context_score": 0.4}, ...]`. Laden Sie diese über den Endpunkt `/api/v1/matrix/upload` hoch. Die Matrix wird sofort in den Arbeitsspeicher geladen und steht für alle folgenden Anfragen zur Verfügung. Validierung erfolgt gegen eine Whitelist von 500 Basis-Schlüsselwörtern.

2. **Ausführen einer Kohortenanalyse**  
   Senden Sie einen `POST`-Request an `/api/v1/cohort/analyze` mit dem Body `{"time_range_days": 7, "min_interactions": 3}`. Das System aggregiert alle Interaktionen der letzten 7 Tage, berechnet für jeden Nutzer die drei oben genannten Parameter und speichert die Kohortenzuordnung in der Tabelle `cohort_assignments`. Die Antwort enthält ein Array aller Kohorten mit Metadaten.

3. **Aktivieren einer automatischen Regel**  
   Definieren Sie eine Regel über `/api/v1/rules/create` mit folgendem JSON:  
   ```json
   {
     "name": "Auto-Mute bei hoher Toxizität",
     "condition": {"toxicity_score": {"gt": 85}, "cohort": {"in": ["C", "D"]}},
     "action": {"type": "shadow_mute", "duration_hours": 24}
   }
   ```  
   Die Regel wird im Worker-Pool registriert und ab der nächsten Minute auf alle eingehenden Events angewendet. Logs sind unter `/api/v1/rules/{rule_id}/logs` einsehbar.

### Beste Praktiken

- **Praxis 1: Schwellwerte iterativ anpassen**  
  Starten Sie mit konservativen Toxizitäts-Schwellwerten (z. B. `> 70` für Mute) und senken Sie diese schrittweise über 2–3 Wochen. Nutzen Sie die `action_log`-Tabelle, um Fehlalarme zu identifizieren und die Gewichtungsmatrix anzupassen. Ein Verhältnis von maximal 5 % Fehlalarmen pro 1.000 Aktionen gilt als akzeptabel.

- **Praxis 2: Kohorten regelmäßig neu berechnen**  
  Setzen Sie die Batch-Verarbeitung der Kohorten auf einen festen Intervall von 6 Stunden. Vermeiden Sie Echtzeit-Berechnungen, da diese bei hoher Last zu Inkonsistenzen führen können. Nutzen Sie einen separaten PostgreSQL-Read-Replica für Analyseabfragen, um die Schreiblast auf dem Primärknoten zu reduzieren.
