## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine hochperformante, modulare Plattform zur Analyse und Steuerung von Fan-Communitys. Es kombiniert **Echtzeit-Stream-Processing** (Apache Kafka, Latenz < 50ms) mit **prädiktiver Verhaltensmodellierung** (TensorFlow Serving, Batch-Intervall: 10 Sekunden) und einer **PostgreSQL-basierten Audit-Trail-Engine** (Transaktionsvolumen: > 10.000 TPS). Ziel ist die automatisierte Erkennung von Toxizität, Spam und Koordinationsangriffen in Fan-Foren, Chat-Kanälen und Social-Media-Feeds.

### Kernfunktionen

1. **Echtzeit-Toxizitäts-Detektor mit Multi-Modell-Ensemble**  
   - **Parameter:** `--model-ensemble=bert-base,roberta-base,distilbert`  
   - **Technische Metrik:** Inferenzzeit pro Batch (32 Tokens) < 15ms auf NVIDIA T4 GPU.  
   - **Funktion:** Parallelisierung über drei vortrainierte NLP-Modelle. Jedes Modell gibt einen Toxizitäts-Score (0.0 – 1.0) aus. Ein gewichteter Mittelwert (`weights=[0.4,0.35,0.25]`) wird mit einem dynamischen Schwellwert (konfigurierbar via `--threshold-adjustment=adaptive_zscore`) verglichen. Bei Überschreitung wird automatisch eine `moderation_action` (Mute, Warnung, temporärer Bann) getriggert und in PostgreSQL geloggt.

2. **Community-Interaktions-Graph-Analyzer (CIGA)**  
   - **Parameter:** `--graph-window=300s`, `--edge-weight=interaction_frequency`  
   - **Technische Metrik:** Verarbeitung von > 50.000 Knoten (User) und > 200.000 Kanten (Interaktionen) pro Minute mit Apache Flink.  
   - **Funktion:** Baut dynamisch einen gerichteten, gewichteten Graphen aus Mentions, Replies und Shares auf. Nutzt den **Louvain-Algorithmus** (Implementierung: Neo4j Graph Data Science Library) zur Community-Erkennung. Erkennt automatisch **Koordinations-Cluster** (z. B. koordinierte Downvote-Wellen) anhand von `modularity_score` > 0,6 und `edge_density` > 0,4. Ergebnisse werden als JSON-Stream an die Admin-Oberfläche gesendet.

3. **Prädiktives Fan-Risiko-Dashboard (PFRD)**  
   - **Parameter:** `--prediction-horizon=24h`, `--feature-set=engagement_decay,toxicity_trend,account_age`  
   - **Technische Metrik:** AUC-ROC > 0,92 auf einem validierten Datensatz von 1,2 Millionen Fan-Profilen.  
   - **Funktion:** Ein **XGBoost-Regressor** (max_depth=8, learning_rate=0,05, n_estimators=500) wird stündlich auf den aggregierten Features neu trainiert. Gibt für jeden Nutzer einen `risk_score` (0–100) aus. Nutzer mit `risk_score > 75` werden in eine Watchlist aufgenommen. Das Dashboard zeigt einen **Heatmap-Trend** (letzte 7 Tage) und einen **Gefahrenindex** (Farbskala: Grün < 30, Gelb 30–60, Rot > 60). API-Endpunkt: `POST /api/v1/predict-risk` mit JSON-Input `{"user_id": "string", "features": {...}}`.

### So verwenden Sie es

1. **Initialisierung der Datenpipeline**  
   - Starten Sie den Kafka-Stream-Processor: `docker-compose up -d kafka flink postgres`  
   - Konfigurieren Sie die Eingangsquellen in `config/sources.yaml`:  
     ```yaml
     sources:
       - type: discord_webhook
         webhook_url: "https://discord.com/api/webhooks/..."
         batch_size: 100
       - type: twitter_stream
         bearer_token: "your_token"
     ```  
   - Überprüfen Sie die Datenrate via Kafka-Console-Consumer: `kafka-console-consumer --bootstrap-server localhost:9092 --topic raw_fan_events --from-beginning`

2. **Modell-Ensemble bereitstellen und testen**  
   - Laden Sie die vortrainierten Modelle in das `models/`-Verzeichnis.  
   - Führen Sie den Toxizitäts-Detektor im Dry-Run-Modus aus:  
     ```bash
     python toxicity_detector.py --dry-run --input test_data/sample_chats.json --output test_results/
     ```  
   - Analysieren Sie die Ausgabe: Jede Zeile enthält `user_id`, `message`, `toxicity_score` und `action` (z. B. `"action": "none"` oder `"action": "mute"`).  
   - Passen Sie den Schwellwert an: `--threshold=0.7` (empfohlen für geringe False-Positive-Rate).

3. **Dashboard konfigurieren und Watchlist aktivieren**  
   - Starten Sie das PFRD-Dashboard: `python risk_dashboard.py --port 8501 --db-host localhost`  
   - Öffnen Sie `http://localhost:8501` und navigieren Sie zu „Watchlist“.  
   - Setzen Sie die automatische Aktion bei `risk_score > 80`: `--auto-action=temp_ban --ban-duration=3600`  
   - Exportieren Sie die Watchlist als CSV: Klicken Sie auf „Export“ (generiert `watchlist_YYYY-MM-DD_HH-MM-SS.csv` mit Spalten: `user_id`, `risk_score`, `last_active`, `top_feature_contribution`).

### Beste Praktiken

- **Batch-Processing vs. Echtzeit:** Für Communitys mit > 100.000 aktiven Nutzern empfehlen wir, die Toxizitätserkennung auf **Mikro-Batches** (Fenster von 5 Sekunden) umzustellen, um die CPU-Auslastung unter 70 % zu halten. Verwenden Sie dazu `--batch-window=5s` im Detektor-Modul. Testen Sie die Latenz mit `curl -X POST http://localhost:8000/health` – die Antwortzeit sollte < 200ms betragen.  
- **Datenhygiene und Audit:** Führen Sie wöchentlich einen **Konsistenz-Check** zwischen PostgreSQL und dem Kafka-Topic `moderation_actions` durch. Skript: `python audit_trail_check.py --db-conn "postgresql://user:pass@localhost:5432/fanguard" --topic moderation_actions`. Bei Abweichungen > 0,5 % lösen Sie eine vollständige Replay-Pipeline aus (`--replay-mode=full`). Dies garantiert die Nachvollziehbarkeit aller Moderationsentscheidungen.
