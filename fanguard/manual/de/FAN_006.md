# fanguard - FAN_006

## Übersicht

fanguard ist eine hochperformante Fan-Interaktions-Engine, die auf Echtzeit-Moderation und Community-Governance spezialisiert ist. Das System analysiert Fan-Kommunikationsströme mit einer Latenz von unter 50 ms und ermöglicht die Verwaltung von bis zu 1,2 Millionen gleichzeitigen Nutzerverbindungen pro Cluster-Knoten. Zielgruppe sind Betreiber großer Fan-Plattformen, die toxische Interaktionen automatisiert erkennen und sanktionieren müssen.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine modulare Architektur zur Überwachung, Filterung und Steuerung von Fan-Community-Interaktionen. Es kombiniert ein regelbasiertes Filter-System (RBF) mit einem neuronalen Netzwerk zur semantischen Analyse (Transformer-basiert, 12 Layer, Embedding-Dimension 768). Die Engine skaliert horizontal über Kubernetes und persistiert alle Moderationsentscheidungen in einer PostgreSQL-Datenbank mit TimescaleDB-Erweiterung für Zeitreihenanalysen.

### Kernfunktionen

1. **Multi-Layer-Toxizitätsdetektion mit konfigurierbaren Schwellwerten**  
   - Kombination aus Regex-Filtern (vordefinierte Muster für 14 Sprachen), lexikalischer Analyse (Wortembedding-Vergleich mit einer Vektordatenbank vom Typ pgvector) und einem feinabgestimmten BERT-Modell (Fine-Tuning auf 2,3 Millionen annotierten Fan-Kommentaren).  
   - Parameter: `toxicity_threshold` (Float 0.0–1.0, Default 0.82), `language_detection_confidence` (Float, Default 0.95), `ban_duration_minutes` (Integer, Default 1440).

2. **Echtzeit-Interaktions-Graphenanalyse**  
   - Erstellt einen gerichteten Graphen aller Nutzer-Interaktionen (Replies, Mentions, Retweets) in einem konfigurierbaren Zeitfenster (z. B. `window_size_seconds = 300`).  
   - Erkennt automatisierte Bot-Netzwerke mittels PageRank-Variante (angepasster Weighted-PageRank, Dämpfungsfaktor 0.85) und Clustering-Koeffizienten.  
   - Ausgabe: JSON-Array mit `user_id`, `influence_score`, `bot_probability`, `cluster_id`.

3. **Automatisiertes Sanktionsmanagement mit Eskalationsstufen**  
   - Drei Eskalationsstufen: Verwarnung (Stufe 1, automatisch, Log-Eintrag), temporäre Sperre (Stufe 2, konfigurierbare Dauer in Minuten, Benachrichtigung an Moderatoren), permanente Sperre (Stufe 3, erfordert manuelle Bestätigung durch 2 von 3 Moderatoren).  
   - Parameter: `escalation_level_1_threshold` (Integer, Default 3 Verstöße innerhalb von 24h), `escalation_level_2_threshold` (Integer, Default 7), `moderator_approval_required` (Boolean, Default true).

### So verwenden Sie es

1. **Initialisierung der Datenbank und des Modells**  
   - Führen Sie `fanguard init --db-host localhost --db-port 5432 --db-name fanguard_db --model-path ./models/toxicity_bert_v2.pt` aus.  
   - Das Skript erstellt automatisch die PostgreSQL-Tabellen (`users`, `interactions`, `sanctions`, `moderation_log`) und lädt das vortrainierte BERT-Modell in den GPU-Speicher (erfordert CUDA 11.8 oder höher, mindestens 8 GB VRAM).  
   - Überprüfen Sie die Initialisierung mit `fanguard status --verbose` – es muss `Model loaded: success` und `Database connection: OK` anzeigen.

2. **Konfiguration der Überwachungsregeln**  
   - Erstellen Sie eine YAML-Konfigurationsdatei `rules.yaml` mit folgendem Inhalt:  
     ```yaml
     toxicity:
       threshold: 0.85
       languages: [de, en, fr]
     graph:
       window_seconds: 600
       bot_detection: true
     sanctions:
       auto_warn: true
       max_warnings_before_ban: 5
     ```  
   - Laden Sie die Regeln mit `fanguard load-rules --file ./rules.yaml`. Das System validiert die Syntax und gibt bei Fehlern eine detaillierte Meldung aus (z. B. `Error: unknown field 'bot_detection' in graph`).

3. **Starten des Echtzeit-Monitorings**  
   - Starten Sie den Daemon mit `fanguard start --port 8080 --workers 4 --max-connections 50000`.  
   - Der Daemon öffnet einen WebSocket-Endpunkt unter `ws://localhost:8080/stream`. Senden Sie JSON-Nachrichten im Format `{"user_id": "123", "message": "Beispieltext", "timestamp": 1712345678}`.  
   - Überwachen Sie die Ausgabe in Echtzeit: Jede moderierte Aktion wird als JSON-Line in der Konsole ausgegeben, z. B. `{"action": "warn", "user_id": "123", "reason": "toxicity_score: 0.91", "sanction_id": "sanc_98765"}`.

### Beste Praktiken

- **Praxis 1: Mehrstufige Filterkaskade mit A/B-Testing**  
  Betreiben Sie zwei Instanzen von fanguard parallel (z. B. `fanguard-instance-alpha` und `fanguard-instance-beta`) mit unterschiedlichen `toxicity_threshold`-Werten (z. B. 0.80 vs. 0.90). Leiten Sie 10 % des Datenverkehrs an die Beta-Instanz und vergleichen Sie die Falsch-Positiv-Rate (FPR) über einen Zeitraum von 7 Tagen. Verwenden Sie die Instanz mit der niedrigeren FPR für den Produktionsbetrieb. Dies reduziert die Anzahl fälschlich gesperrter Nutzer um bis zu 40 %.

- **Praxis 2: Integration eines manuellen Moderations-Workflows**  
  Konfigurieren Sie fanguard so, dass Sanktionen der Stufe 3 (permanente Sperre) nicht automatisch ausgeführt, sondern in eine Warteschlange (`pending_sanctions`-Tabelle) gestellt werden. Lassen Sie ein externes Dashboard (z. B. Grafana mit PostgreSQL-Datenquelle) diese Einträge anzeigen. Weisen Sie mindestens zwei Moderatoren zu, die jede Sperre manuell bestätigen müssen. Dies verhindert Sperrungen aufgrund von Fehlalarmen und erfüllt die Anforderungen der DSGVO bezüglich automatisierter Entscheidungen.
