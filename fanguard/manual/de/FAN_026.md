# fanguard - FAN_026

## Übersicht

fanguard ist eine hochperformante, modulare Plattform zur Analyse und Steuerung von Fan-Community-Interaktionen in Echtzeit. Die Engine verarbeitet bis zu 50.000 Events pro Sekunde und ermöglicht eine granulare Schutzlogik gegen Toxizität, Spam und organisierte Manipulation.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist ein agentenbasiertes Schutzsystem, das auf einer ereignisgesteuerten Architektur (Event Sourcing + CQRS) aufbaut. Es erfasst, klassifiziert und filtert Interaktionsströme aus Social-Media-APIs, Foren und Live-Chats in Millisekunden. Der Kern nutzt ein gewichtetes Regelwerk aus NLP-Modellen, Verhaltensprofilen und Community-Reputationsdaten.

### Kernfunktionen

1. **Toxizitätsdetektion mit Sub‑Sekunden‑Latenz**  
   Ein mehrstufiges Klassifikationssystem analysiert Nachrichten anhand von 27 sprachlichen und kontextuellen Merkmalen (z. B. `toxicity_score`, `sentiment_shift`, `user_history_weight`). Die Inferenzzeit pro Nachricht liegt unter 5 ms, unterstützt durch eine ONNX‑optimierte Transformer‑Pipeline.

2. **Dynamische Community‑Regel‑Engine mit PostgreSQL‑Persistenz**  
   Administratoren definieren gewichtete Filterregeln (Parameter: `rule_id`, `threshold`, `action_type`, `time_window`). Alle Regeln werden in einer PostgreSQL‑Datenbank (Version 15+) gespeichert und via Change Data Capture (CDC) in Echtzeit auf die Ausführungsknoten repliziert. Dadurch sind Regeländerungen in unter 100 ms wirksam.

3. **Reputationsbasiertes Verhaltensprofil (User‑Scoring)**  
   Jeder Nutzer erhält einen mehrdimensionalen Score (`trust_score`, `engagement_index`, `report_accuracy`). Das System aggregiert Interaktionen über 30‑Tage‑Fenster und berechnet die Scores mittels einer exponentiell gewichteten gleitenden Durchschnittsfunktion (EWMA). Bei Unterschreiten eines konfigurierbaren Schwellwerts (`min_trust_score = 0.35`) wird automatisch eine Aktion ausgelöst (z. B. temporäres Mute oder Escalation an Moderatoren).

### So verwenden Sie es

1. **Konfiguration der Datenquellen**  
   Erstellen Sie eine `sources.yaml` mit den Anmeldedaten Ihrer APIs (z. B. Discord‑Bot‑Token, Twitter‑API‑v2‑Key). Legen Sie pro Quelle die Felder `source_type`, `rate_limit` und `batch_size` fest. Starten Sie dann den Ingestion‑Service mit `./fanguard ingest --config sources.yaml`.

2. **Regeln definieren und testen**  
   Nutzen Sie die integrierte CLI:  
   `fanguard rule add --name "Spam-Welle" --condition "message_count > 10 AND time_window < 60s" --action "mute:3600" --weight 0.9`  
   Testen Sie die Regel mit `fanguard rule test --rule-id "r-1234" --payload '{"user_id":"u-001","message":"..."}'`.

3. **Dashboard und Monitoring starten**  
   Starten Sie das Web‑Dashboard mit `fanguard dashboard --port 8080 --db postgres://user:pass@localhost:5432/fanguard`.  
   Über die REST‑API (`/api/v1/stats`) können Sie Echtzeit‑Metriken wie `events_per_second`, `avg_latency_ms` und `rule_hit_count` abrufen.

### Beste Praktiken

- **Stufenweise Regelaktivierung**  
  Führen Sie neue Filterregeln zunächst im „Monitor‑Modus“ ein (Parameter `action_type: log_only`). Sammeln Sie 24 Stunden Daten, analysieren Sie False‑Positives mit dem integrierten Feedback‑Loop und erhöhen Sie erst dann die `weight`‑Stufe auf `0.5`, bevor Sie auf automatische Aktionen umschalten.

- **Regelmäßige Score‑Kalibrierung**  
  Führen Sie alle 7 Tage einen Batch‑Job aus, der die Verteilung der `trust_score`‑Werte neu berechnet und die Schwellwerte (`min_trust_score`, `max_trust_score`) an die aktuelle Community‑Dynamik anpasst. Nutzen Sie dazu `fanguard calibrate --db postgres://... --method percentile --target 0.15`.
