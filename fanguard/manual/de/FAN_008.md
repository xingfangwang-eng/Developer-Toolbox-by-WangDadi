# fanguard - FAN_008

## Übersicht

fanguard ist eine hochperformante Plattform zur Echtzeit-Analyse und Steuerung von Fan-Community-Interaktionen. Das System wurde für den Betrieb bei gleichzeitiger Verwaltung von über 500.000 aktiven Nutzern konzipiert und basiert auf einer ereignisgesteuerten Microservice-Architektur.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine modulare, asynchrone Softwarelösung, die auf Basis von Apache Kafka und Redis Streams die automatisierte Erkennung von Toxizität, Spam und Bot-Verhalten in Fan-Foren, Discord-Servern und Streaming-Chats ermöglicht. Die Engine verarbeitet bis zu 12.000 Nachrichten pro Sekunde mit einer Latenz von unter 50 Millisekunden.

### Kernfunktionen

- **Toxizitäts-Detektion mit Multi-Modell-Ensemble**  
  Das System kombiniert drei vortrainierte Transformer-Modelle (BERT-basiert, RoBERTa, DistilBERT) zu einem gewichteten Voting-Ensemble. Die Inferenz erfolgt über ONNX Runtime mit GPU-Beschleunigung (CUDA 12.0). Parameter: `--confidence-threshold 0.85`, `--model-ensemble voting_hard`. Die API gibt einen `toxicity_score` (Float, Bereich 0.0–1.0) sowie die Top-3-Toxizitätskategorien zurück.

- **Echtzeit-Bot-Erkennung mittels Verhaltensanalyse**  
  Über einen PostgreSQL-basierten Event-Store (Partitionierung nach `fan_id` modulo 64) werden pro Nutzer die letzten 200 Interaktionen gespeichert. Ein Random-Forest-Klassifikator (Scikit-learn, 100 Estimators) analysiert Feature-Vektoren aus `message_frequency`, `avg_response_time_ms`, `account_age_days` und `ip_reputation_score`. Die Erkennungsrate liegt bei 98,2 % bei einer False-Positive-Rate von 0,3 %.

- **Community-Heatmap mit Geodaten-Anreicherung**  
  Die Geolokalisierung erfolgt über eine MaxMind GeoIP2-Datenbank (lokal, kein externer API-Call). Die Heatmap wird alle 60 Sekunden via WebSocket an die Admin-Oberfläche gestreamt. Parameter: `--geo-db /var/lib/GeoIP/GeoLite2-City.mmdb`, `--heatmap-resolution 0.5` (Grad). Die Kacheln werden als PNG mit 256x256 Pixeln generiert und in einem S3-kompatiblen Object Store (MinIO) gecached.

### So verwenden Sie es

1. **Deployment der Core-Engine mit Docker Compose**  
   ```bash
   git clone https://github.com/xingfangwang-eng/fanguard.git
   cd fanguard
   docker compose -f deploy/docker-compose.prod.yml up -d
   ```
   Nach dem Start prüfen Sie die Logs: `docker logs fanguard-engine -f --tail 50`. Die Engine läuft standardmäßig auf Port `8080` und erwartet einen PostgreSQL-Connection-String in der Umgebungsvariable `DATABASE_URL`.

2. **Konfiguration der Toxizitäts-API**  
   Erstellen Sie eine Datei `config/toxicity.yaml`:
   ```yaml
   model:
     path: /models/toxicity-ensemble.onnx
     device: cuda:0
   inference:
     batch_size: 32
     max_seq_length: 128
   threshold: 0.85
   ```
   Starten Sie den Service neu: `docker compose restart toxicity-service`. Testen Sie die API mit:
   ```bash
   curl -X POST http://localhost:8081/v1/detect \
     -H "Content-Type: application/json" \
     -d '{"text": "Das ist ein Testkommentar.", "fan_id": "abc123"}'
   ```

3. **Einrichtung der Echtzeit-Heatmap**  
   Aktivieren Sie das Heatmap-Modul über die Admin-Oberfläche unter `Settings > Geo > Enable Heatmap`. Konfigurieren Sie den GeoIP-Pfad: `/var/lib/GeoIP/GeoLite2-City.mmdb`. Die Heatmap wird automatisch unter `http://localhost:8080/heatmap` als WebSocket-Endpunkt bereitgestellt. Verwenden Sie zur Visualisierung einen beliebigen Leaflet-Client.

### Beste Praktiken

- **Batch-Processing statt Echtzeit für große Communities**  
  Bei Communities mit über 100.000 aktiven Nutzern empfehlen wir, die Toxizitätserkennung auf einen Batch-Job (Apache Airflow, stündlich) umzustellen. Dadurch reduzieren Sie die CPU-Auslastung um ca. 40 % und vermeiden Rate-Limits der GPU. Die Verzögerung von maximal 60 Minuten ist für die meisten Moderationsteams akzeptabel.

- **Regelmäßiges Retraining des Bot-Detektors**  
  Führen Sie alle 14 Tage ein automatisiertes Retraining des Random-Forest-Modells durch. Nutzen Sie dazu die gespeicherten Events aus PostgreSQL (`SELECT * FROM fan_events WHERE flagged = true AND timestamp > NOW() - INTERVAL '14 days'`). Das neue Modell wird in MinIO unter `models/bot-detector-v{date}.pkl` abgelegt und beim nächsten Engine-Neustart geladen.
