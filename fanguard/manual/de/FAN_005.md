# fanguard - FAN_005

## Übersicht

fanguard ist eine hochperformante Plattform zur Echtzeit-Erkennung und -Steuerung von Fan-Community-Interaktionen. Das System analysiert Kommunikationsmuster, erkennt Eskalationsrisiken und orchestriert automatisierte Moderationsworkflows – mit einer Latenz von unter 50 Millisekunden pro Ereignis.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine Microservice-basierte Architektur, die auf Apache Kafka und PostgreSQL aufbaut. Sie verarbeitet bis zu 10.000 Events pro Sekunde, klassifiziert diese mittels eines vortrainierten NLP-Modells (BERT-basiert) und leitet Aktionen an Discord-, Telegram- oder Slack-Webhooks weiter. Die Zustandsverwaltung erfolgt über Redis mit einer TTL von 24 Stunden für temporäre Sitzungsdaten.

### Kernfunktionen

1. **Echtzeit-Eskalationsdetektion mit Sentiment-Score**  
   - Parameter: `sentiment_threshold` (Float, Standard: 0.75), `escalation_window` (Integer, Sekunden, Standard: 300)  
   - Technische Umsetzung: Ein Transformer-Modell (Fine-Tuned auf 500.000 deutschsprachigen Fan-Kommentaren) bewertet jedes eingehende Ereignis mit einem Sentiment-Score zwischen 0.0 und 1.0. Überschreitet der gleitende Durchschnitt innerhalb des `escalation_window` den `sentiment_threshold`, wird automatisch ein Moderations-Ticket mit Priorität `high` erstellt.  
   - Persistierung: PostgreSQL-Tabelle `escalation_logs` mit Index auf `(community_id, created_at)`.

2. **Multi-Plattform-Event-Aggregation mit deduplizierter Korrelation**  
   - Parameter: `dedup_hash_algorithm` (String, Standard: `SHA-256`), `event_ttl` (Integer, Sekunden, Standard: 3600)  
   - Technische Umsetzung: Ein zentraler Event-Bus (Kafka-Topic `fan_events`) konsumiert Nachrichten von Discord-Webhooks, Telegram-Bots und Slack-APIs. Jede Nachricht wird mit einem `SHA-256`-Hash aus `(user_id, message_text, timestamp_rounded_to_10s)` dedupliziert. Redis speichert den Hash mit einer TTL von `event_ttl`.  
   - Performance: 99,9 % Deduplizierung bei 5.000 Events pro Sekunde, gemessen unter Last mit Locust.

3. **Automatisierte Moderations-Workflows mit Zustandsmaschine**  
   - Parameter: `workflow_states` (Array, z. B. `["pending", "reviewed", "escalated", "resolved"]`), `auto_resolve_timeout` (Integer, Minuten, Standard: 30)  
   - Technische Umsetzung: Ein Finite-State-Machine (FSM) in Go steuert den Lebenszyklus jedes Moderations-Tickets. Bei Statuswechsel wird ein `webhook_url` aus der Community-Konfiguration (gespeichert in PostgreSQL-Tabelle `community_webhooks`) aufgerufen. Nach `auto_resolve_timeout` Minuten im Status `pending` wird das Ticket automatisch auf `resolved` gesetzt, sofern keine manuelle Intervention erfolgt.  
   - Audit: Jeder Statuswechsel wird in der Tabelle `workflow_audit` mit `actor_id`, `old_state`, `new_state` und `timestamp` protokolliert.

### So verwenden Sie es

1. **Konfiguration des Sentiment-Schwellenwerts pro Community**  
   - Rufen Sie die fanguard-API mit einem `POST`-Request auf:  
     `POST /api/v1/communities/{community_id}/config`  
     Body: `{"sentiment_threshold": 0.85, "escalation_window": 600}`  
   - Bestätigen Sie die Änderung mit einem `200 OK` und überprüfen Sie die neue Konfiguration via `GET /api/v1/communities/{community_id}/config`.  
   - Testen Sie die Detektion, indem Sie einen simulierten Event mit `sentiment=0.9` an den Kafka-Topic `fan_events` senden.

2. **Integration eines neuen Discord-Webhooks**  
   - Erstellen Sie in Ihrem Discord-Server einen Webhook und kopieren Sie die URL.  
   - Registrieren Sie den Webhook in fanguard:  
     `POST /api/v1/webhooks`  
     Body: `{"platform": "discord", "url": "https://discord.com/api/webhooks/...", "community_id": "c123"}`  
   - Validieren Sie die Verbindung: fanguard sendet einen Test-Ping an die URL und erwartet einen `200 OK`-Response. Bei Fehlschlag wird ein `400 Bad Request` mit Fehlercode `WEBHOOK_UNREACHABLE` zurückgegeben.

3. **Manuelles Eingreifen in einen eskalierten Workflow**  
   - Listen Sie alle offenen Tickets mit Priorität `high` auf:  
     `GET /api/v1/tickets?priority=high&status=pending`  
   - Wählen Sie ein Ticket aus (z. B. `ticket_id: "t-789"`) und setzen Sie es auf `reviewed`:  
     `PATCH /api/v1/tickets/t-789`  
     Body: `{"status": "reviewed", "actor_id": "mod-001"}`  
   - Überprüfen Sie die Audit-Logs für dieses Ticket:  
     `GET /api/v1/tickets/t-789/audit`  
     Erwartetes Ergebnis: Ein Eintrag mit `old_state: "pending"`, `new_state: "reviewed"`, `timestamp` im ISO-8601-Format.

### Beste Praktiken

1. **Segmentierung der Communities nach Risikoprofil**  
   Ordnen Sie Communities mit hohem Konfliktpotenzial (z. B. Esport-Fanclubs) einem separaten Kafka-Partition-Key zu. Nutzen Sie dazu das Feld `community_risk_score` (Integer, 1–100) in der PostgreSQL-Tabelle `community_metadata`. Dies verhindert, dass ein einzelner Community-Burst die gesamte Event-Pipeline blockiert. Empfohlen: Partition-Key = `community_id % 10` für 10 Partitionen.

2. **Regelmäßige Modell-Nachschulung mit Feedback-Daten**  
   Exportieren Sie wöchentlich alle Tickets, die manuell von Moderatoren auf `escalated` gesetzt wurden, als CSV (Spalten: `message_text`, `sentiment_score`, `final_action`). Verwenden Sie diese Daten, um das Sentiment-Modell mit einem inkrementellen Training (z. B. Hugging Face `Trainer` mit `learning_rate=2e-5`) nachzuschärfen. Ziel: Reduzierung der False-Positive-Rate unter 5 %.
