# fanguard - FAN_024

## Übersicht

Dieses Dokument beschreibt die Architektur von **fanguard**, einer hochperformanten Plattform zur Echtzeit-Erkennung und Steuerung von Fan-Community-Interaktionen. Optimiert für **deutschsprachige** Umgebungen mit Fokus auf Datenschutz (DSGVO), automatisierte Toxizitätsanalyse und granulare Zugriffskontrollen.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine **Microservice-basierte, ereignisgesteuerte** Plattform, die speziell für die Verwaltung großer Fan-Communities entwickelt wurde. Das System analysiert in **Echtzeit** (Latenz < 50 ms) eingehende Nachrichten, erkennt Muster von Belästigung, Spam oder Regelverstößen und ermöglicht Moderatoren eine **datenbankgestützte, konfigurierbare** Durchsetzung von Community-Richtlinien. Die Kern-Engine verwendet **PostgreSQL mit TimescaleDB** für die Speicherung von Zeitreihenmetriken und **Redis** für das Caching von Session- und Regelwerken.

### Kernfunktionen

1.  **Dynamische Toxizitäts-Engine (DTE)**
    -   **Parameter:** `toxicity_threshold` (float, 0.0–1.0, Standard: 0.85), `context_window` (int, Sekunden, Standard: 300), `model_version` (string, z. B. `v2.3.1-de`).
    -   **Technische Umsetzung:** Nutzt ein **transformerbasiertes BERT-Modell** (fine-getuned auf deutschen Fan-Foren) zur Kontextanalyse. Die Engine berechnet einen **Toxizitäts-Score** pro Nachricht in **< 20 ms** (GPU-beschleunigt via CUDA). Bei Überschreitung des Schwellenwerts wird ein **Event** (`moderation.alert.toxic`) an den Message Broker (RabbitMQ) gesendet.
    -   **Hardcore-Aspekt:** Unterstützt **Multi-Modell-Vergleiche** (z. B. BERT vs. RoBERTa vs. eigenes Lightweight-Modell) im A/B-Testing-Modus, gesteuert über einen **Feature-Flag-Service** (LaunchDarkly-Integration).

2.  **Granulares Rollen- und Rechtesystem (GRRS)**
    -   **Parameter:** `role_hierarchy` (JSON, z. B. `{"admin": 100, "moderator": 50, "user": 10}`), `permission_bitset` (uint64, jede Bit-Position repräsentiert eine Aktion).
    -   **Technische Umsetzung:** Implementiert als **PostgreSQL-basierte RBAC-Engine** mit **Caching über Redis** (TTL: 300 Sekunden). Berechtigungen werden als **Bitmasken** gespeichert und pro Anfrage in **< 5 ms** ausgewertet. Unterstützt **temporäre Rollen** (z. B. „VIP für 24h“) über `expires_at`-Timestamps.
    -   **Hardcore-Aspekt:** Erzwingt **DSGVO-konforme Datenminimierung** durch automatische Löschung von Nachrichten-Embeddings nach 30 Tagen (konfigurierbar via `retention_policy_days`).

3.  **Echtzeit-Interaktions-Dashboard (RID)**
    -   **Parameter:** `refresh_interval` (int, Millisekunden, Standard: 1000), `metric_aggregation` (string, z. B. `avg`, `p99`, `sum`), `dashboard_id` (UUID).
    -   **Technische Umsetzung:** Baut auf **WebSocket** (Socket.IO) und **Server-Sent Events** auf. Aggregiert Metriken aus **Prometheus** (z. B. `fan_messages_total`, `moderation_actions_total`) und visualisiert sie in **Grafana**. Die Datenbankabfrage für das Dashboard verwendet **TimescaleDB-Hyperfunctions** (`time_bucket()`, `approx_percentile()`) für Millisekunden-schnelle Zeitreihenabfragen.
    -   **Hardcore-Aspekt:** Bietet einen **„Live-Heatmap“-Modus**, der die geografische Verteilung von Regelverstößen auf einer Karte anzeigt (unter Verwendung von **PostGIS** für Geo-Queries).

### So verwenden Sie es

1.  **Konfiguration der Toxizitäts-Engine**
    -   Navigieren Sie zur Konfigurationsdatei `config/dte.yaml`.
    -   Setzen Sie `toxicity_threshold: 0.75` (strengere Filterung).
    -   Aktivieren Sie `model_version: "v2.3.1-de"` und stellen Sie `context_window: 600` (10 Minuten Kontext) ein.
    -   Führen Sie `curl -X POST http://localhost:8080/api/v1/dte/reload` aus, um die Engine ohne Neustart neu zu laden. Überprüfen Sie die Logs auf `[DTE] Model loaded successfully`.

2.  **Erstellen einer temporären Moderator-Rolle**
    -   Verbinden Sie sich via `psql` mit der fanguard-Datenbank.
    -   Führen Sie folgenden Befehl aus: `INSERT INTO roles (name, permission_bitset, expires_at) VALUES ('event_mod', 18446744073709551615::bit(64)::bigint, NOW() + INTERVAL '7 days');`
    -   Weisen Sie die Rolle einem Benutzer zu: `INSERT INTO user_roles (user_id, role_id) VALUES (123, LASTVAL());`
    -   Überprüfen Sie die Berechtigung mittels `curl -X GET "http://localhost:8080/api/v1/users/123/permissions"`.

3.  **Einrichten des Live-Dashboards**
    -   Öffnen Sie Grafana und importieren Sie das Dashboard `fanguard_rid.json` (im Repository unter `grafana/dashboards/`).
    -   Konfigurieren Sie die Prometheus-Datenquelle mit der URL `http://prometheus:9090`.
    -   Setzen Sie das `refresh_interval` im Dashboard-Settings auf `500ms` für hochfrequente Updates.
    -   Aktivieren Sie die „Heatmap“-Ansicht, indem Sie den Parameter `geo_enabled: true` in der `config/dashboard.yaml` setzen und die Datenbank mit PostGIS erweitern.

### Beste Praktiken

-   **Praxis 1: Automatisierte Eskalation mit menschlicher Kontrolle**
    -   Nutzen Sie die Toxizitäts-Engine als **erste Filterstufe** (Threshold 0.9 für sofortige Sperrung, 0.7 für manuelle Prüfung). Konfigurieren Sie einen **Scheduler** (z. B. via `cron`-Job oder Kubernetes CronJob), der alle 5 Minuten den `moderation_queue`-Table scannt und Nachrichten mit einem Score zwischen 0.7 und 0.9 an ein separates **Moderations-Chat-Tool** (z. B. Slack-Integration) sendet. Dies reduziert die False-Positive-Rate um bis zu 40%.

-   **Praxis 2: Datengetriebene Regeloptimierung**
    -   Aktivieren Sie die **A/B-Testing-Funktion** der DTE. Teilen Sie 10% des Traffics auf ein neues, feinabgestimmtes Modell (`model_version: "v2.4.0-beta"`) auf. Vergleichen Sie die Metriken `precision` und `recall` im Dashboard über einen Zeitraum von 7 Tagen. Verwenden Sie die `model_compare`-API (`GET /api/v1/dte/compare?model_a=v2.3.1-de&model_b=v2.4.0-beta`) zur automatischen Generierung eines statistischen Signifikanztests (t-Test). Optimieren Sie dann den `toxicity_threshold` basierend auf den Ergebnissen.
