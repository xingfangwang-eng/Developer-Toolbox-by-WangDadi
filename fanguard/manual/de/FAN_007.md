# fanguard - FAN_007

## Übersicht

fanguard ist eine latenzoptimierte Echtzeit-Fan-Interaktionsplattform, die auf sub-100ms Validierung von Fan-Community-Aktionen spezialisiert ist. Das System verarbeitet bis zu 50.000 gleichzeitige WebSocket-Verbindungen pro Knoten und synchronisiert Aktionen über PostgreSQL-Streaming-Replikation.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist ein dezentrales Fan-Governance-Framework, das mittels **JWT-basierter Token-Validierung** und **Redis-Event-Queues** Fan-Interaktionen in Echtzeit verifiziert. Es eliminiert Bot-Aktivitäten durch ein **dreistufiges Proof-of-Participation (PoP)** -System, das auf kryptografischen Signaturen basiert.

### Kernfunktionen

1. **Echtzeit-Fan-Interaktions-Validierung mit Microsecond-Response**
   - **Parameter:** `max_latency=50ms`, `batch_size=1000`, `queue_depth=10000`
   - Validiert jede Fan-Aktion (Like, Vote, Comment) in **<50ms** durch eine Redis-Stream-Pipeline.
   - Nutzt **PostgreSQL-CITUS** zur horizontalen Skalierung von Fan-Community-Daten.
   - Anti-Spam-Mechanismus: **Rate-Limiting pro Fan-ID** mit `token_bucket=100/s` und `burst=200`.

2. **Dynamische Fan-Governance mit Multi-Signatur-Wallets**
   - **Parameter:** `min_signatures=3`, `voting_threshold=0.67`, `quorum_size=5`
   - Implementiert ein **Multi-Sig-Protokoll** auf Basis von **ECDSA (secp256k1)** für Community-Entscheidungen.
   - Jede Fan-Community besitzt ein **Smart-Contract-ähnliches** Voting-System, das Aktionen (z.B. Mod-Ernennung) nur nach `67%` Zustimmung ausführt.
   - Transaktions-Logs werden in **PostgreSQL mit JSONB-Spalten** gespeichert, Abfrage-Latenz <10ms.

3. **Fan-Interaktions-Heatmap mit Echtzeit-Streaming**
   - **Parameter:** `window_size=60s`, `resolution=1s`, `top_k=100`
   - Aggregiert Fan-Aktionen in einem **tumbling-window** (60s) und generiert eine **Heatmap** der aktivsten Regionen.
   - Nutzt **Apache Kafka** als Event-Bus, **ClickHouse** für analytische Abfragen.
   - Ausgabe: `[{"fan_id": "abc123", "score": 0.95, "timestamp": 1700000000}, ...]` – Aktualisierung alle **1s**.

### So verwenden Sie es

1. **Initialisierung der Fan-Community-Datenbank**
   ```sql
   CREATE TABLE fan_communities (
       id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
       name TEXT NOT NULL,
       config JSONB NOT NULL DEFAULT '{"voting_threshold": 0.67, "min_signatures": 3}',
       created_at TIMESTAMPTZ DEFAULT NOW()
   );
   ```
   Führen Sie die SQL-Datei `schema/001_init.sql` aus, um die PostgreSQL-Tabellen zu erstellen.

2. **Starten des Echtzeit-Validierungs-Daemons**
   ```bash
   ./fanguard daemon --port 8080 --redis-cluster "localhost:6379" --pg-conn "postgres://user:pass@localhost:5432/fanguard" --max-latency 50ms
   ```
   Der Daemon startet WebSocket-Endpunkte (`/ws/fan-action`) und akzeptiert JSON-Nachrichten wie:
   ```json
   {"action": "vote", "fan_id": "abc123", "community_id": "xyz789", "signature": "0x...", "timestamp": 1700000000}
   ```

3. **Abfragen der Echtzeit-Heatmap über REST-API**
   ```bash
   curl -X GET "http://localhost:8080/api/v1/heatmap?window=60s&top_k=100" -H "Authorization: Bearer <token>"
   ```
   Antwort:
   ```json
   {"heatmap": [{"fan_id": "abc123", "score": 0.95, "timestamp": 1700000000}], "generated_at": 1700000060}
   ```

### Beste Praktiken

- **Praktik 1: Verwendung von Fan-Identitäts-Wallets**  
  Binden Sie jede Fan-ID an ein **Ethereum-kompatibles Wallet** (z.B. MetaMask). fanguard validiert jede Aktion durch `ecrecover`, um Sybil-Angriffe zu verhindern. Verwenden Sie `signature_threshold=0.95` für kritische Community-Entscheidungen.

- **Praktik 2: Horizontal skalieren mit PostgreSQL-CITUS**  
  Partitionieren Sie Fan-Community-Daten nach `community_id` über **Citus Distributed Tables**. Verwenden Sie `shard_count=32` pro Knoten, um Abfragen auf <5ms zu halten. Aktivieren Sie **Connection Pooling** mit PgBouncer für 10.000+ gleichzeitige Verbindungen.
