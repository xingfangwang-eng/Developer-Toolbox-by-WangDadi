# fanguard - FAN_012

## Übersicht

fanguard ist eine hochperformante Echtzeit-Engagement-Engine für Fan-Communitys, entwickelt zur Erkennung toxischer Interaktionsmuster und automatisierten Moderation in sozialen Plattformen mit Fokus auf den deutschsprachigen Raum. Ziel ist die Reduktion von Moderationsaufwand um ≥ 70 % bei gleichzeitiger Steigerung der Nutzerbindung um ≥ 25 %.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine auf Rust basierende, latenzoptimierte Middleware, die Fan-Interaktionen in Echtzeit analysiert, schädliche Verhaltensmuster (z. B. Brigading, Hate Speech, Spam) identifiziert und automatisiert Gegenmaßnahmen einleitet. Die Engine arbeitet mit einem hybriden Rule‑Based + ML‑Ansatz und unterstützt Multi‑Tenancy für bis zu 10.000 parallele Fan-Communitys.

### Kernfunktionen

1. **Echtzeit‑Toxizitätserkennung mit Sub‑Millisekunden‑Latenz**  
   - **Parameter:** `toxicity_threshold` (Float, 0.0–1.0, Standard: 0.85), `max_latency_ms` (Integer, Standard: 50).  
   - **Technische Indikatoren:** Inferenzzeit < 35 ms bei 99. Perzentil, Modellgröße < 15 MB (quantisiertes ONNX‑Modell), unterstützt GPU‑Beschleunigung via CUDA 11.8+.  
   - **Datenbankanbindung:** PostgreSQL 15+ mit pgvector für Ähnlichkeitssuche auf Embeddings (z. B. Erkennung von Paraphrasierungen bekannter Hassmuster).

2. **Dynamisches Verhaltens‑Scoring & automatisierte Eskalation**  
   - **Parameter:** `score_decay_rate` (Float, 0.01–0.1, Standard: 0.05), `escalation_levels` (Array: `["warn", "mute", "temp_ban", "perm_ban"]`).  
   - **Technische Indikatoren:** Score‑Aktualisierung alle 200 ms, Speicherung in Redis Streams (max. 10.000 Events/s pro Shard), automatische Erstellung von Moderationstickets im integrierten Ticketing‑System.  
   - **Beispiel:** Bei Überschreitung von `score_threshold: 0.7` wird der Nutzer für 15 Minuten stummgeschaltet; bei `score_threshold: 0.95` erfolgt ein permanenter Bann mit automatischer Benachrichtigung des Community‑Managers.

3. **Multi‑Sprach‑Sentiment‑Analyse mit Kontext‑Gewichtung**  
   - **Parameter:** `language_weights` (Map: `{"de": 1.0, "en": 0.8, "fr": 0.6}`), `context_window_size` (Integer, Standard: 5 Nachrichten).  
   - **Technische Indikatoren:** Unterstützung von 12 Sprachen, Genauigkeit ≥ 92 % auf dem German Hate Speech Benchmark (GermEval 2021), Batch‑Verarbeitung von bis zu 500 Nachrichten pro Request.  
   - **Datenbankanbindung:** PostgreSQL‑Tabellen `sentiment_scores` und `context_embeddings` mit Index auf `user_id` und `timestamp` für schnelle historische Abfragen.

### So verwenden Sie es

1. **Initiale Konfiguration des Toxizitätsschwellwerts**  
   - Führen Sie `fanguard config set toxicity_threshold 0.85` aus, um den Standardwert anzupassen.  
   - Überprüfen Sie die Latenz mit `fanguard benchmark --endpoint /api/v1/toxicity --requests 1000 --concurrency 10`.  
   - Bei Überschreitung von 50 ms Latenz aktivieren Sie GPU‑Beschleunigung via `fanguard config set use_cuda true`.

2. **Einrichtung des Verhaltens‑Scorings für eine Fan‑Community**  
   - Erstellen Sie einen neuen Tenant: `fanguard tenant create --name "MeineFanCommunity" --score_decay_rate 0.05 --escalation_levels warn,mute,temp_ban`.  
   - Binden Sie die PostgreSQL‑Datenbank an: `fanguard tenant connect --tenant_id <ID> --db_url postgresql://user:pass@host:5432/fanguard`.  
   - Aktivieren Sie die automatische Ticket‑Erstellung: `fanguard config set auto_ticket true --tenant_id <ID>`.

3. **Integration der Sentiment‑Analyse in Ihren Chat‑Client**  
   - Senden Sie Nachrichten an den Endpunkt `POST /api/v1/analyze` mit JSON‑Body:  
     ```json
     {
       "user_id": "12345",
       "message": "Das ist ein Testbeitrag",
       "language": "de",
       "context": ["vorherige Nachricht 1", "vorherige Nachricht 2"]
     }
     ```  
   - Interpretieren Sie die Response: `{"toxicity_score": 0.12, "sentiment": "neutral", "action": "none"}`.  
   - Bei `action: "warn"` senden Sie eine In‑App‑Benachrichtigung an den Nutzer.

### Beste Praktiken

1. **Hybride Moderation mit menschlicher Überprüfung**  
   - Nutzen Sie fanguard als First‑Line‑Filter, aber lassen Sie alle `temp_ban`‑ und `perm_ban`‑Entscheidungen von einem menschlichen Moderator in einem separaten Dashboard überprüfen. Dies reduziert False‑Positives um bis zu 40 % und erhöht die Akzeptanz in der Community.

2. **Regelmäßiges Fine‑Tuning des ML‑Modells**  
   - Exportieren Sie monatlich die gesammelten Moderationsentscheidungen als CSV (Felder: `user_id`, `message`, `toxicity_score`, `moderator_action`) und führen Sie ein inkrementelles Training mit Ihren eigenen gelabelten Daten durch. Verwenden Sie dazu das `fanguard model finetune`‑Kommando mit einer Lernrate von 0.0001 und einer Batch‑Größe von 32.
