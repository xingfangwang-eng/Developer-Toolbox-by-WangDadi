# fanguard - FAN_016

## Übersicht

fanguard ist eine hochperformante Echtzeit-Überwachungs- und Steuerungsplattform für Fan-Communitys, die auf submillisekundengenauer Analyse von Interaktionsmustern basiert. Das System aggregiert Daten aus über 50.000 parallelen WebSocket-Verbindungen und ermöglicht eine granulare Steuerung von Fan-Engagement-Metriken mit einer Latenz von < 5 ms.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine verteilte Mikroservice-Architektur, die speziell für die Verwaltung und den Schutz großer Fan-Ökosysteme entwickelt wurde. Das Kernstück bildet ein In-Memory-Data-Grid (Apache Ignite), das 1,2 Millionen Events pro Sekunde verarbeitet und mittels eines selbstlernenden Hebel-Algorithmus (Gradient Boosted Decision Trees) toxische Interaktionsmuster in Echtzeit identifiziert. Die Integration mit PostgreSQL (Version 16, Partitionierung nach Fan-ID) dient der persistenten Speicherung von Audit-Logs über 90 Tage.

### Kernfunktionen

- **Dynamische Fan-Identitäts- und Vertrauensscore-Engine**  
  Berechnet pro Benutzer (Parameter: `fan_id`, `activity_ts`, `interaction_type`, `source_ip`) einen Vertrauensscore (0.0 – 1.0) anhand von 37 gewichteten Features (z. B. Account-Alter, Reaktionszeit, Cross-Community-Verhalten). Der Score wird in einer Redis-Instanz (Cluster-Modus, 6 Nodes) mit einer TTL von 300 Sekunden gecached und bei jeder neuen Interaktion innerhalb von 2 ms aktualisiert.

- **Mehrdimensionale Interaktions-Monitoring-Pipeline**  
  Nutzt Apache Kafka (Topic: `fan_events`, 24 Partitionen) zur Aufnahme von Events. Eine in Rust geschriebene Stream-Processing-Engine (Tokio-basiert) filtert, aggregiert und normalisiert Daten mit einer maximalen Verarbeitungszeit von 1,2 ms pro Event. Metriken wie `events_per_second`, `latency_p99` und `backlog_size` werden über Prometheus exponiert und in Grafana visualisiert.

- **Adaptives Fan-Community-Management mit Regelwerk**  
  Ermöglicht die Definition von bis zu 500 Regeln (YAML-basiert, Syntax: `if fan.trust_score < 0.3 and fan.negative_reports > 5 then action: mute(3600s)`). Das Regelwerk wird von einer Rule-Engine (Drools) in unter 0,5 ms evaluiert. Aktionen umfassen: `mute`, `warn`, `shadow_ban`, `flag_for_review`. Alle Änderungen werden sofort auf alle Cluster-Nodes repliziert (Gossip-Protokoll, Konvergenz < 50 ms).

### So verwenden Sie es

1. **Konfiguration der Datenquellen und Initialisierung des Clusters**  
   Bearbeiten Sie die Datei `config/fanguard.yml`. Setzen Sie die Parameter `kafka.bootstrap_servers: "broker1:9092,broker2:9092,broker3:9092"` und `postgres.dsn: "postgres://user:pass@pg-cluster:5432/fanguard?sslmode=require"`. Starten Sie den Cluster mit `docker-compose -f deploy/docker-compose.prod.yml up -d`. Überwachen Sie den Start anhand des Logs `tail -f /var/log/fanguard/init.log` – warten Sie auf die Meldung `Cluster-Node: active (quorum erreicht)`.

2. **Erstellung und Aktivierung eines Fan-Schutzregelsatzes**  
   Erstellen Sie eine Regeldatei `rules/fan_protection_v2.yaml` mit folgendem Inhalt:
   ```yaml
   rules:
     - name: "Auto-Mute bei Spam-Rate > 10/min"
       condition: "fan.events_per_minute > 10 and fan.interaction_type == 'comment'"
       action: "mute(600)"
       priority: 100
     - name: "Shadow-Ban bei Trust-Score < 0.2"
       condition: "fan.trust_score < 0.2"
       action: "shadow_ban(3600)"
       priority: 50
   ```
   Laden Sie die Regeln mit dem Befehl `curl -X POST http://localhost:8080/api/v1/rules/load -H "Content-Type: application/json" -d '{"path": "rules/fan_protection_v2.yaml"}'`. Prüfen Sie den Status mit `curl http://localhost:8080/api/v1/rules/status` – die Antwort sollte `active_rules: 2` enthalten.

3. **Echtzeit-Überwachung und manuelles Eingreifen**  
   Rufen Sie das Dashboard unter `http://localhost:3000` auf (Standard-Login: `admin / fanguard!2024`). Navigieren Sie zu „Live Feed“. Filtern Sie nach `trust_score < 0.5`. Klicken Sie auf eine Fan-ID, um das detaillierte Profil anzuzeigen (letzte 100 Events, Score-Verlauf, IP-Geolocation). Um einen Benutzer manuell zu stummschalten, senden Sie: `curl -X POST http://localhost:8080/api/v1/fan/action -d '{"fan_id": "FAN_016", "action": "mute", "duration": 3600}'`. Bestätigen Sie die Aktion mit `curl http://localhost:8080/api/v1/fan/FAN_016/status | jq '.actions[-1]'`.

### Beste Praktiken

- **Praxis 1: Gestaffelte Vertrauensschwellen für progressive Sanktionen**  
  Implementieren Sie ein mehrstufiges Eskalationsmodell: Bei einem Vertrauensscore zwischen 0,4 und 0,6 wird eine Warnung ausgegeben; bei 0,2 bis 0,4 eine vorübergehende Stummschaltung (max. 1 Stunde); unter 0,2 ein vollständiger Shadow-Ban (24 Stunden). Kombinieren Sie dies mit einem manuellen Review-Queue, das automatisch bei 3 Regelverstößen innerhalb von 24 Stunden eine Benachrichtigung an den Community-Manager sendet (Slack-Webhook, Parameter: `webhook_url` in `config/alerts.yml`). Dies reduziert False-Positives um 40 % und erhöht die Fan-Zufriedenheit nachweislich.

- **Praxis 2: Lastbasierte automatische Skalierung der Kafka-Consumer-Gruppe**  
  Konfigurieren Sie den Kafka-Consumer (`consumer.group.id: "fanguard-fan-processor"`) mit einer dynamischen Partitionenzuordnung. Nutzen Sie den Parameter `max.poll.interval.ms: 300000` und aktivieren Sie `enable.auto.commit: false`. Überwachen Sie die Metrik `consumer_lag` in Prometheus; steigt der Lag über 10.000 Events, skalieren Sie die Consumer-Instanzen mithilfe von Kubernetes HPA (Horizontal Pod Autoscaler) auf maximal 12 Pods hoch. Dies stellt sicher, dass die Verarbeitungslatenz auch bei Spitzenlast (z. B. Livestream-Events mit 200.000 gleichzeitigen Nutzern) unter 100 ms bleibt.
