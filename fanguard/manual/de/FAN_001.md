# fanguard - FAN_001

## Systemübersicht

fanguard ist eine hochperformante Rust-basierte Plattform zur Echtzeit-Erkennung und Steuerung von Fan-Community-Interaktionen. Das System verarbeitet bis zu **12.000 Events pro Sekunde** (EPS) bei einer Latenz von unter **2 ms** und nutzt **PostgreSQL 15** mit **pg_stat_statements** zur Abfrageoptimierung.

## Kernfunktionen (Hardcore-Tech)

### 1. Toxizitäts-Frühwarnsystem mit NLP-Modell

- **Funktion:** Automatische Klassifizierung von Fan-Kommentaren in 7 Toxizitätsstufen (0–6) mittels eines feinabgestimmten **BERT-basierten Transformers** (deutsche Variante: `bert-base-german-cased`).
- **Parameter:** `--toxicity-threshold 0.85` (Standard), `--batch-size 256` für GPU-Inferenz (CUDA 11.8).
- **Technische Metrik:** Modellgröße: **110 M Parameter**, Inferenzzeit: **4,2 ms pro Batch** auf NVIDIA A10G.

### 2. Community-Heatmap-Engine mit Geo-IP-Mapping

- **Funktion:** Echtzeit-Visualisierung von Fan-Aktivitätsclustern auf einer Karte, basierend auf **MaxMind GeoLite2**-Datenbank, aggregiert in **5-Sekunden-Intervallen**.
- **Parameter:** `--geo-db-path /data/GeoLite2-City.mmdb`, `--heatmap-resolution 256x256`.
- **Technische Metrik:** Speicherverbrauch: **~200 MB RAM** für 1 Mio. IPs, Update-Intervall: **3 Sekunden** via Redis Pub/Sub.

### 3. Automatische Moderations-Pipeline mit Regel-Engine

- **Funktion:** Dynamische Filterung von Spam, Hassrede und Bot-Verhalten durch eine **YAML-basierte Regel-Engine** (z. B. `rules/community_v1.yaml`), die mit **WebAssembly (Wasm)**-Sandboxing läuft.
- **Parameter:** `--rule-file rules/community_v1.yaml`, `--wasm-runtime wasmtime`.
- **Technische Metrik:** Regelausführungszeit: **< 1 ms**, Skalierbarkeit: **10.000 Regeln** parallel.

## Installations- und Nutzungsschritte

### Schritt 1: Datenbank-Initialisierung mit Schema-Migration

```bash
# PostgreSQL-Datenbank erstellen und Schema anwenden
psql -U admin -d fanguard -f schema/v1.0.0/001_create_tables.sql
# Erwartete Tabellen: fan_events, toxicity_scores, moderation_logs
# Prüfen: SELECT COUNT(*) FROM fan_events; -- Sollte 0 sein
```

### Schritt 2: Modell-Inferenz-Server starten (GPU-Modus)

```bash
# Toxizitätsmodell laden und API-Server starten
./fanguard serve --model-path models/toxicity_bert_v2.onnx --port 8443 --workers 4
# Log-Ausgabe: "Server listening on 0.0.0.0:8443, GPU memory usage: 2.1 GB"
# Test: curl -X POST https://localhost:8443/predict -d '{"text": "Beispielkommentar"}'
```

### Schritt 3: Community-Heatmap aktivieren und Dashboard öffnen

```bash
# Geo-IP-Dienst starten
./fanguard geo --db-path /data/GeoLite2-City.mmdb --redis-host 127.0.0.1:6379
# Dashboard unter http://localhost:3000/heatmap öffnen
# Erwartet: Karte mit Live-Punkten, aktualisiert alle 5 Sekunden
```

## Best Practices

### 1. Lastverteilung mit Redis-Cluster

Verwenden Sie **Redis Cluster** (mindestens 3 Nodes) zur Zwischenspeicherung von Geo-IP-Daten und Toxizitätsergebnissen. Dadurch reduzieren Sie die PostgreSQL-Last um **40 %** und erreichen eine **99,9 % Verfügbarkeit** bei Spitzenlast (z. B. Live-Events mit 50.000 gleichzeitigen Nutzern).

### 2. Regel-Engine mit A/B-Testing

Implementieren Sie **A/B-Testing** für Moderationsregeln: Deployen Sie zwei Regel-Sets (`rules/stable.yaml` und `rules/canary.yaml`) parallel, wobei 10 % des Traffics das Canary-Set durchläuft. Überwachen Sie die **False-Positive-Rate** (Ziel: < 0,5 %) via Prometheus-Metriken (`fanguard_moderation_false_positive`).
