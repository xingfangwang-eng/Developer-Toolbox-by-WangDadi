# fanguard - FAN_002

## Übersicht

Dieses Dokument beschreibt die technische Architektur von **fanguard**, einem skalierbaren Tool zur Fan-Community-Moderation und Interaktionsanalyse. Zielsprache der Implementierung ist **Deutsch (de)**. Das System verarbeitet Echtzeitdaten aus Fan-Foren, Discord-Servern und Streaming-Plattformen, um toxisches Verhalten zu erkennen und Community-Managern handfeste Metriken zu liefern.

## Kernfunktionen (Technische Spezifikation)

### 1. Echtzeit-Toxizitätserkennung mit Multi-Modell-Ensemble

- **Parameter:** `--toxicity-threshold 0.85`, `--model-ensemble "bert-base-german-cased;roberta-hate-speech;xlm-roberta"`  
- **Technische Indikatoren:** Inferenz-Latenz < 50 ms pro Nachricht, Batch-Verarbeitung von bis zu 10.000 Nachrichten/Minute, PostgreSQL-Upsert mit `ON CONFLICT DO UPDATE` für Historienverfolgung.  
- **Funktionsweise:** Ein Ensemble aus drei vortrainierten NLP-Modellen bewertet jede Nachricht parallel. Die Ergebnisse werden via Weighted Voting fusioniert. Bei Überschreitung des Schwellwerts wird automatisch ein Moderations-Ticket erstellt (inkl. Kontext der letzten 20 Nachrichten).  
- **Hardcore-Detail:** Nutzt ONNX-Runtime für GPU-beschleunigte Inferenz. Fallback auf CPU mit dynamischer Batch-Größe bei GPU-Überlastung.

### 2. Fan-Interaktions-Graph mit Zyklusanalyse

- **Parameter:** `--graph-depth 3`, `--cycle-detection "johnson"`, `--edge-weight "interaction_frequency"`  
- **Technische Indikatoren:** Aufbau eines gerichteten Graphen mit bis zu 1 Mio. Knoten, Speicherung in Neo4j, Abfrage-Zeit für Zyklen < 200 ms.  
- **Funktionsweise:** Jeder Fan wird als Knoten modelliert, jede Erwähnung, Antwort oder Reaktion als Kante. Mittels Johnsons Algorithmus werden alle elementaren Zyklen identifiziert – das deckt organisierte Trolling-Netzwerke auf.  
- **Hardcore-Detail:** Integration mit Apache Kafka für Streaming-Updates. Zyklen werden als JSON-Snapshot in S3 exportiert, inkl. Metadaten wie `cycle_length` und `avg_toxicity_score`.

### 3. Automatische Community-Segmentierung via DBSCAN-Clustering

- **Parameter:** `--eps 0.5`, `--min-samples 10`, `--feature-vector "engagement_score;sentiment_avg;toxicity_ratio;post_frequency"`  
- **Technische Indikatoren:** Clustering von bis zu 500.000 Nutzerprofilen in < 30 Sekunden, Speicherung der Cluster-ID in PostgreSQL mit Index auf `(cluster_id, community_id)`.  
- **Funktionsweise:** Jeder Nutzer wird durch einen 4-dimensionalen Feature-Vektor repräsentiert. DBSCAN gruppiert sie in Cluster – z. B. „toxische Power-User“, „passive Lurker“ oder „engagierte Moderatoren“.  
- **Hardcore-Detail:** Verwendet `scikit-learn` mit `n_jobs=-1` für Parallelisierung. Ergebnisse werden als `cluster_centroids` in eine separate Tabelle geschrieben und täglich neu berechnet.

## So verwenden Sie es (Detaillierte Schritte)

### Schritt 1: Datenquellen konfigurieren

Erstellen Sie eine `sources.yaml` im Projektstamm:

```yaml
sources:
  - type: discord
    token: "YOUR_BOT_TOKEN"
    channel_ids: ["123456789", "987654321"]
    batch_size: 100
  - type: forum
    url: "https://forum.example.com"
    api_key: "YOUR_API_KEY"
    poll_interval: 30
```

Führen Sie dann den Befehl aus:

```bash
python fanguard ingest --config sources.yaml --db postgresql://user:pass@localhost:5432/fanguard
```

Das System startet sofort die Datensammlung und schreibt Rohdaten in die Tabelle `raw_messages`. Überwachen Sie den Fortschritt mit:

```bash
tail -f logs/ingest.log | grep "INFO:ingest:Processed"
```

### Schritt 2: Toxizitätsmodell trainieren oder laden

Laden Sie ein vortrainiertes Modell oder starten Sie ein Feintuning mit eigenen Daten:

```bash
python fanguard train --model bert-base-german-cased --data ./data/labeled_messages.csv --epochs 3 --batch-size 16 --output ./models/toxicity_v1.onnx
```

Alternativ können Sie ein fertiges Modell aus dem Modell-Repository laden:

```bash
python fanguard load-model --from-s3 s3://fanguard-models/toxicity_ensemble_v2/ --local-path ./models/
```

Prüfen Sie die Modell-Metadaten:

```bash
python fanguard info --model ./models/toxicity_v1.onnx
```

### Schritt 3: Interaktionsgraph erstellen und analysieren

Bauen Sie den Graphen aus den gesammelten Daten:

```bash
python fanguard build-graph --db postgresql://user:pass@localhost:5432/fanguard --neo4j bolt://localhost:7687 --depth 3
```

Führen Sie die Zyklusanalyse durch:

```bash
python fanguard detect-cycles --graph-name "fan_interactions" --algorithm johnson --output ./reports/cycles_2025-04-01.json
```

Die Ausgabe enthält eine Liste aller Zyklen mit Details wie:

```json
{
  "cycle_id": "c_001",
  "nodes": ["user_123", "user_456", "user_789"],
  "length": 3,
  "avg_toxicity": 0.92,
  "first_detected": "2025-03-28T14:22:10Z"
}
```

## Best Practices

### 1. Datenqualität vor Modellqualität

Stellen Sie sicher, dass die gelabelten Trainingsdaten mindestens 10.000 Beispiele pro Sprache enthalten. Nutzen Sie `fanguard validate-labels --data ./data/labeled_messages.csv`, um Inkonsistenzen (z. B. widersprüchliche Labels) automatisch zu erkennen. Ein hoher Anteil an False Positives deutet meist auf ein unausgewogenes Label-Set hin – führen Sie dann eine Stratified Split (80/20) durch.

### 2. Monitoring der Inferenz-Pipeline

Richten Sie Prometheus-Metriken ein für:

- `fanguard_inference_latency_ms` (Histogramm, Buckets: 10, 50, 100, 200, 500)
- `fanguard_toxicity_hits_total` (Counter, gelabelt mit `model_name`)
- `fanguard_queue_depth` (Gauge, aktuelle Nachrichten in der Warteschlange)

Ein Alarm bei `fanguard_queue_depth > 5000` für mehr als 5 Minuten deutet auf einen Engpass hin – skalieren Sie dann die Worker-Pods horizontal.
