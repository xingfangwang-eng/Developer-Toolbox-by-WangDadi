# fanguard - FAN_023

## Übersicht

fanguard ist eine hochperformante, modulare Plattform zur Echtzeit-Überwachung und Steuerung von Fan-Community-Interaktionen. Das System analysiert Datenströme aus sozialen Netzwerken, Foren und Messengern mit einer Latenz von unter 50 Millisekunden und ermöglicht eine granulare Berechtigungssteuerung auf Benutzerebene. Es richtet sich an professionelle Community-Manager, die toxische Inhalte automatisiert erkennen, Eskalationsketten unterbrechen und gleichzeitig die Authentizität der Fanbasis validieren möchten.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist ein dezentrales, ereignisgesteuertes Framework, das auf einer PostgreSQL-Datenbank mit TimescaleDB-Erweiterung aufbaut. Es verarbeitet bis zu 10.000 Events pro Sekunde pro Cluster-Knoten, identifiziert mittels eines trainierten XGBoost-Modells (Feature-Vektor: 128 Dimensionen) anomales Nutzerverhalten und führt präventive Maßnahmen wie temporäre Kontosperrungen oder Nachrichtenzensur in Echtzeit aus. Das Tool ist vollständig containerisierbar (Docker/Kubernetes) und bietet eine REST-API mit OAuth 2.0-Authentifizierung.

### Kernfunktionen

1. **Multi-Stream-Sentiment-Analyse mit adaptiver Schwellwertlogik**  
   fanguard aggregiert kontinuierlich Text-, Bild- und Metadaten aus bis zu 50 parallelen Quellen (Discord, Telegram, Reddit, eigene Foren). Ein LSTM-Netzwerk (trainiert auf 2,3 Millionen deutschsprachigen Fan-Posts) klassifiziert jede Nachricht in 8 Kategorien (z. B. `TOXIC`, `SPAM`, `SUPPORT`, `OFFTOPIC`) mit einer Genauigkeit von 94,7 %. Der Schwellwert für automatische Moderationseingriffe ist dynamisch konfigurierbar (`--threshold 0.85`, Wertebereich 0.0–1.0) und passt sich über einen gleitenden Durchschnitt von 1000 Events an die aktuelle Community-Stimmung an.

2. **Pseudonym-Authentizitäts-Score mit Graph-basierter Netzwerkanalyse**  
   Das System erstellt für jeden Nutzer einen gewichteten Graphen aus 15 Merkmalen (z. B. Account-Alter, Frequenz der Interaktionen, IP-Wechsel, Verhältnis von positiven zu negativen Reaktionen). Ein PageRank-ähnlicher Algorithmus berechnet daraus einen `fan_score` (0–100, normalverteilt um 50). Nutzer mit einem Score unter 20 werden automatisch als „Bot-Verdacht“ markiert und erhalten eine Captcha-Herausforderung. Der Parameter `--min-fan-score 25` definiert die untere Grenze für uneingeschränkte Teilnahme.

3. **Echtzeit-Eskalationskette mit konfigurierbaren Aktionen**  
   Überschreitet ein Event einen benutzerdefinierten Risikoindex (z. B. `--risk-threshold 0.9`), löst fanguard eine mehrstufige Aktion aus: Stufe 1: Nachricht wird zurückgehalten und in eine Quarantäne-Tabelle (`quarantine_queue`) verschoben. Stufe 2: Eine Webhook-Benachrichtigung wird an Discord oder Slack gesendet (`--webhook-url https://hooks.slack.com/...`). Stufe 3: Nach 30 Sekunden ohne manuelle Freigabe wird der Nutzer für 15 Minuten stummgeschaltet (`--mute-duration 900`). Alle Aktionen werden mit einem UNIX-Timestamp (Millisekunden) geloggt.

### So verwenden Sie es

1. **Installation und initiale Datenbankeinrichtung**  
   Klonen Sie das Repository und führen Sie `docker-compose up -d` aus. Das System initialisiert automatisch eine PostgreSQL-Datenbank mit der Schema-Definition `schema_v2.1.sql`. Überprüfen Sie die Verbindung mit:  
   `psql -h localhost -U fanguard_admin -d fanguard_db -c "SELECT COUNT(*) FROM fan_events;"`  
   Erwarten Sie eine Ausgabe von `0`, falls keine Testdaten geladen wurden.

2. **Konfiguration der Datenquellen und des Modells**  
   Erstellen Sie eine Datei `sources.yaml` im Verzeichnis `/etc/fanguard/`. Beispiel für einen Discord-Kanal:  
   ```yaml
   sources:
     - type: discord
       token: "YOUR_BOT_TOKEN"
       channel_ids: ["123456789", "987654321"]
       rate_limit: 50 # Nachrichten pro Sekunde
   ```  
   Starten Sie den Datenstrom mit:  
   `fanguard ingest --config /etc/fanguard/sources.yaml --model /models/sentiment_v3.xgb`  
   Das Modell lädt in etwa 2,1 Sekunden (gemessen auf einer NVIDIA T4 GPU).

3. **Ausführen einer manuellen Nutzeranalyse**  
   Analysieren Sie einen spezifischen Nutzer mit der CLI:  
   `fanguard analyze --user-id "discord|123456789" --output-format json --verbose`  
   Die Ausgabe enthält den `fan_score`, die letzten 50 Events mit Zeitstempel sowie eine Liste der ausgelösten Regeln. Beispiel für einen verdächtigen Nutzer:  
   ```json
   {
     "user_id": "discord|123456789",
     "fan_score": 12.3,
     "risk_flags": ["IP_ANOMALY", "HIGH_FREQUENCY"],
     "last_event": 1700000000123
   }
   ```

### Beste Praktiken

1. **Schwellwerte staffeln nach Community-Größe**  
   Für kleine Communities (< 1.000 Mitglieder) empfehlen wir einen `--risk-threshold` von 0,95, um Fehlalarme zu minimieren. Ab 10.000 Mitgliedern senken Sie den Wert schrittweise auf 0,85. Testen Sie jede Änderung zuerst in einer Staging-Umgebung mit replizierten Daten (`fanguard replay --from 2024-01-01 --to 2024-01-31`), bevor Sie in Produktion gehen.

2. **Regelmäßiges Retraining des Sentiment-Modells**  
   Planen Sie alle 14 Tage ein automatisiertes Retraining des XGBoost-Modells ein. Nutzen Sie dazu die integrierte Pipeline:  
   `fanguard train --data-dir /data/labeled_posts --model-output /models/sentiment_v4.xgb --epochs 50`  
   Validieren Sie das neue Modell gegen den aktuellen Datensatz (`--validation-split 0.2`). Ein Abfall der Genauigkeit um mehr als 1,5 % sollte ein Rollback auf die letzte stabile Version auslösen.
