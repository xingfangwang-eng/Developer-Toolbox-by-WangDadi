# fanguard - FAN_020

## Übersicht

fanguard ist eine hochperformante Fan-Community-Governance-Engine, die sich auf Echtzeit-Moderation, automatisierte Interaktionsanalyse und datenschutzkonformes Fan-Community-Management spezialisiert. Zielsprache: Deutsch. Keywords: Fan-Schutz-Tool, Fan-Community-Management, Fan-Interaktionsplattform.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine **serverseitige Rust-basierte Middleware**, die Fan-Community-Plattformen in die Lage versetzt, toxische Interaktionen in **< 50 ms Latenz** zu erkennen und zu blockieren – ohne dass personenbezogene Daten den Server verlassen. Es integriert sich über **gRPC-Streams** in bestehende Chat-, Foren- oder Ticketing-Systeme und bietet eine **PostgreSQL-gestützte Audittrail-Datenbank** mit **automatischer Retention-Policy nach DSGVO**.

### Kernfunktionen

1. **Echtzeit-Toxizitätsfilter mit Multi-Modell-Ensemble**  
   Nutzt ein **3-stufiges Ensemble aus transformer-basierten NLP-Modellen (z. B. BERT-de, hateBERT, RoBERTa)**, die auf deutschsprachigen Fan-Foren vortrainiert sind. Jeder eingehende Text wird in **< 20 ms** bewertet – parallel auf **CPU/GPU (CUDA 11.8, FP16-Inferenz)**. Parameter: `confidence_threshold: f64 = 0.85`, `batch_size: u32 = 64`. Ergebnisse werden als **Protobuf‑Message** an den Client gestreamt.

2. **Fan-Interaktions-Graphenanalyse mit zeitlichem Verlauf**  
   Baut einen **gerichteten, gewichteten Graphen** aus allen Interaktionen (Likes, Replies, Shares) innerhalb eines **konfigurierbaren Zeitfensters (default: 7 Tage)** auf. Speicherung in **Neo4j** – Abfrage über **Cypher** (z. B. `MATCH (f:Fan)-[r:REPLIED_TO]->(t:Fan) WHERE r.timestamp > timestamp() - duration('P7D') RETURN f.id, count(r)`). Output: **JSON‑Struktur** mit Metriken wie `centrality_score`, `reciprocity_index`, `influence_rank`. Parameter: `graph_window_hours: u32 = 168`, `min_edge_weight: u32 = 3`.

3. **Automatisiertes Sanktions-Workflow-Engine mit Eskalationsmatrix**  
   Definiert **regelbasierte Sanktionsstufen** (Verwarnung → temporäre Sperre → permanente Sperre) basierend auf **kumulierten Toxizitäts-Scores** pro Fan. Jede Aktion wird mit einem **reason_code** (z. B. `TOXICITY_HIGH`, `SPAM_DETECTED`, `REPEAT_OFFENDER`) und einem **SHA‑256‑Hash des Originaltexts** in der PostgreSQL‑Audittabelle `fan_actions` geloggt. Parameter: `escalation_thresholds: [u32; 3] = [3, 7, 15]`, `action_timeout_seconds: u32 = 86400`. Workflow wird via **YAML‑Konfigurationsdatei** (`workflow.yaml`) gesteuert.

### So verwenden Sie es

1. **Deployment der fanguard-Engine als Docker-Container**  
   ```bash
   docker run -d --name fanguard-engine \
     -p 50051:50051 \
     -e FANGUARD_DB_URL=postgres://user:pass@host:5432/fanguard \
     -e FANGUARD_NEO4J_URI=bolt://host:7687 \
     -e FANGUARD_MODEL_PATH=/models/hatebert-v1.onnx \
     -v /local/models:/models:ro \
     xingfangwang/fanguard:latest
   ```  
   Überprüfen Sie den Status: `grpcurl -plaintext localhost:50051 list`

2. **Integration in Ihre Fan-Plattform (Python-Client-Beispiel)**  
   ```python
   import grpc
   from fanguard_pb2 import ToxicityRequest, ToxicityResponse
   from fanguard_pb2_grpc import FanguardStub

   channel = grpc.insecure_channel('localhost:50051')
   stub = FanguardStub(channel)
   req = ToxicityRequest(fan_id="fan_12345", text="Dein Beitrag ist echt unterirdisch.")
   response = stub.AnalyzeToxicity(req, timeout=0.5)
   print(f"Toxizitäts-Score: {response.toxicity_score}, Handlung: {response.recommended_action}")
   ```  
   **Wichtig**: Setzen Sie `timeout` auf maximal **500 ms**, um die Echtzeitfähigkeit zu gewährleisten.

3. **Konfiguration der Eskalationsmatrix**  
   Erstellen Sie eine Datei `workflow.yaml` im Projektstamm:  
   ```yaml
   rules:
     - condition: "toxicity_score > 0.85"
       action: "WARN"
       reason_code: "TOXICITY_HIGH"
     - condition: "toxicity_score > 0.95 OR (toxicity_score > 0.85 AND repeat_offender == true)"
       action: "TEMPORARY_BAN"
       duration_hours: 24
       reason_code: "REPEAT_OFFENDER"
     - condition: "violations_in_30_days >= 5"
       action: "PERMANENT_BAN"
       reason_code: "PERSISTENT_ABUSE"
   ```  
   Laden Sie die Konfiguration: `grpcurl -plaintext -d '{"config_yaml": "$(cat workflow.yaml)"}' localhost:50051 fanguard.WorkflowService/UpdateConfig`

### Beste Praktiken

1. **Datenminimierung und Pseudonymisierung vor der Analyse**  
   Übermitteln Sie niemals Klarnamen oder E-Mail-Adressen an fanguard. Verwenden Sie stattdessen **pseudonymisierte Fan-IDs (UUID v4)**, die nur in Ihrer eigenen Datenbank aufgelöst werden. Kombinieren Sie dies mit einer **automatischen Löschroutine** (z. B. Retention von 90 Tagen) in der PostgreSQL‑Audittabelle, um DSGVO‑Konformität zu gewährleisten.

2. **Modell-Update-Strategie mit A/B-Testing**  
   Führen Sie neue NLP‑Modelle zunächst in einer **Shadow‑Mode‑Instanz** aus (d. h. sie analysieren den Traffic, blockieren aber nicht). Vergleichen Sie die **False-Positive-Rate** und **Recall** über mindestens 48 Stunden. Erst wenn die Metrik `f1_score > 0.92` erreicht, schalten Sie das Modell in den produktiven Modus. Nutzen Sie **MLflow** zur Nachverfolgung der Modellversionen.
