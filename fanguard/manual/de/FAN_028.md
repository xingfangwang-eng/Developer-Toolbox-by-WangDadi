# fanguard - FAN_028

## Übersicht

fanguard ist eine hochperformante, auf Echtzeitanalyse spezialisierte Plattform zur automatisierten Fan-Community-Governance und Interaktionssteuerung. Das System verarbeitet bis zu 50.000 Events pro Sekunde und ermöglicht eine granulare Steuerung von Community-Richtlinien auf Basis von NLP-gestützten Sentiment-Analysen.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist ein modulares Backend-System zur Verwaltung, Filterung und Orchestrierung von Fan-Interaktionen in digitalen Ökosystemen. Es kombiniert eine regelbasierte Engine mit einem adaptiven Machine-Learning-Modell, das auf über 2 Millionen annotierten deutschsprachigen Fan-Beiträgen trainiert wurde. Die Architektur unterstützt Multi-Tenant-Betrieb mit mandantenspezifischen Policy-Sets und einer Latenz von unter 8 Millisekunden pro Entscheidung.

### Kernfunktionen

1. **Dynamische Sentiment-Filterung mit metrikgesteuerter Schwellwertlogik**  
   Implementiert eine mehrstufige Filter-Pipeline: Rohdaten → Tokenisierung (mit spaCy) → Feature-Extraktion (TF-IDF + Word Embeddings) → Klassifikation (Random Forest + LightGBM-Ensemble). Die Entscheidungsgrenzen werden automatisch anhand von 12 konfigurierbaren Metriken angepasst, darunter `toxicity_score` (0.0–1.0), `engagement_velocity` (Anzahl Interaktionen pro Minute) und `community_health_index` (gleitender Durchschnitt über 10.000 Events). Alle Filterentscheidungen werden in einer PostgreSQL-Datenbank mit JSONB-Schema persistiert, inklusive Timestamp und Confidence-Score.

2. **Echtzeit-Interaktionssteuerung mit konfigurierbaren Rate-Limitern**  
   Bietet eine verteilte Rate-Limiting-Engine auf Basis von Redis-Clustern (6 Shards). Erlaubt die Definition von Limits pro Benutzer, IP-Range oder Content-Kategorie (z. B. `max_posts_per_hour: 50`, `max_reactions_per_minute: 100`). Die Limits werden in Millisekunden granular überprüft und bei Überschreitung automatisch ein `429 Too Many Requests` mit einem Retry-After-Header in Millisekunden zurückgegeben. Ein integriertes Dashboard visualisiert die Auslastung in Echtzeit mit einer Aktualisierungsrate von 200 ms.

3. **Automatisiertes Policy-Management mit Versionierung und Rollback**  
   Ermöglicht die Definition von Policies als YAML-basierte Konfigurationsdateien, die in einer Git-artigen Versionsverwaltung (eigene Implementierung auf Basis von PostgreSQL und pg_audit) gespeichert werden. Jede Policy enthält Parameter wie `policy_id`, `version`, `rules` (Array von JSON-Objekten mit `condition`, `action`, `priority`) und `valid_from`. Ein Rollback auf eine beliebige frühere Version ist in unter 50 ms möglich. Policies können über eine REST-API (OpenAPI 3.0) deployed und getestet werden.

### So verwenden Sie es

1. **Initiale Konfiguration des Policy-Engines**  
   Erstellen Sie eine YAML-Datei `fan_policy_v1.yaml` mit folgendem Inhalt:  
   ```yaml
   policy_id: "fan_community_001"
   version: 1
   rules:
     - condition: "toxicity_score > 0.85"
       action: "block_content"
       priority: 1
     - condition: "engagement_velocity > 1000"
       action: "rate_limit_user"
       priority: 2
   ```  
   Führen Sie dann den Befehl aus: `fanguard-cli policy deploy --file fan_policy_v1.yaml --environment production`. Das System validiert die Syntax, prüft auf Konflikte (z. B. doppelte Prioritäten) und aktiviert die Policy innerhalb von 3 Sekunden.

2. **Echtzeit-Überwachung der Interaktionsmetriken**  
   Nutzen Sie das integrierte Prometheus-Export-Endpunkt unter `http://localhost:9090/metrics`. Rufen Sie dort gezielt Metriken ab, z. B.:  
   `curl -s http://localhost:9090/metrics | grep fanguard_interaction_rate`  
   Die Ausgabe zeigt Daten wie `fanguard_interaction_rate{policy_id="fan_community_001", action="block"} 1423` (Anzahl blockierter Inhalte seit Start). Für eine detaillierte Analyse verwenden Sie Grafana mit einem vorkonfigurierten Dashboard, das alle 200 ms aktualisiert wird.

3. **Manuelles Eingreifen bei kritischen Vorfällen**  
   Bei einem Alarm (z. B. `community_health_index < 0.3`) können Sie über die Admin-API sofort eine temporäre Policy aktivieren:  
   ```bash
   curl -X POST http://localhost:8080/api/v1/policies/emergency \
     -H "Content-Type: application/json" \
     -d '{"action": "pause_all_interactions", "duration_ms": 300000}'
   ```  
   Der Befehl setzt eine 5-minütige Pause für alle Interaktionen durch. Das System protokolliert diesen Eingriff mit User-ID, Timestamp und Grund in der Audit-Tabelle.

### Beste Praktiken

- **Praxis 1: Gestaffelte Policy-Aktivierung mit Canary-Deployment**  
  Führen Sie neue Policies immer zuerst in einer separaten Staging-Umgebung mit 5 % des Traffics aus. Beobachten Sie die Metriken `false_positive_rate` und `user_satisfaction_score` über mindestens 24 Stunden. Erst bei einer Fehlerrate unter 0,1 % und einem Satisfaction-Score über 0,75 deployen Sie auf die gesamte Produktion. Dies verhindert Massen-Fehlentscheidungen und schützt die Community-Erfahrung.

- **Praxis 2: Regelmäßige Modell-Nachschulung mit annotierten Daten**  
  Sammeln Sie wöchentlich 10.000 zufällig ausgewählte, manuell annotierte Fan-Beiträge (Toxizität, Relevanz, Engagement). Führen Sie ein Retraining des Sentiment-Modells mit diesen neuen Daten durch, idealerweise nachts zwischen 2:00 und 4:00 Uhr (geringste Last). Nutzen Sie den Befehl `fanguard-cli model retrain --dataset weekly_feb_2025.csv --output model_v2.1.pt`. Dokumentieren Sie die Performance-Veränderung (z. B. F1-Score von 0,89 auf 0,91) im internen Wiki.
