# fanguard - FAN_030

## Übersicht

fanguard ist eine hochperformante Echtzeit-Analyse-Engine für Fan-Community-Ökosysteme. Das System erfasst, filtert und klassifiziert Interaktionsdaten aus bis zu 50.000 parallelen Quellkanälen mit einer Latenz von unter 12 Millisekunden pro Transaktion. Es dient als zentrales Steuerungsinstrument für Community-Manager, um toxische Muster, Spam-Attacken und Koordinationsversuche in Fan-Netzwerken automatisiert zu erkennen und zu unterbinden.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine auf PostgreSQL und Redis Streams basierende Multi-Layer-Schutzarchitektur, die Fan-Interaktionsplattformen vor Manipulation, Missbrauch und organisierter Desinformation schützt. Das Tool arbeitet mit einem hybriden Regelwerk aus heuristischen Filtern und gewichteten Wahrscheinlichkeitsmodellen, die auf historischen Community-Daten trainiert wurden. Die gesamte Verarbeitung erfolgt in-memory mit asynchroner Persistierung, um eine Durchsatzrate von 120.000 Events pro Sekunde zu gewährleisten.

### Kernfunktionen

1. **Multi-Phasen-Kollisionserkennung mit Zeitstempel-Clustering**  
   Erkennt koordinierte Aktionen (z. B. gleichzeitige Like-Wellen oder identische Kommentartexte) durch Analyse von Zeitstempel-Clustern im Mikrosekundenbereich. Parameter: `collision_window_ms` (Standard: 250 ms), `min_cluster_size` (Standard: 5). Ausgabe: JSON-Array mit Cluster-ID, Startzeit, Endzeit und beteiligten Nutzer-IDs.

2. **Dynamisches Toxizitäts-Profil mit gewichteten Feature-Vektoren**  
   Bewertet jedes eingehende Event anhand eines 12-dimensionalen Feature-Vektors (z. B. Wortfrequenz, Emoji-Dichte, Account-Alter, IP-Reputation). Der finale Toxizitäts-Score wird über eine sigmoide Aktivierungsfunktion berechnet: `score = 1 / (1 + exp(-(w·x + b)))`. Schwellwert: `toxicity_threshold` (Standard: 0.78). Bei Überschreitung erfolgt automatische Quarantäne.

3. **Graphenbasierte Einfluss-Netzwerkanalyse mit Community-Detektion**  
   Baut aus Interaktionsdaten einen gerichteten Graphen auf (Knoten: Nutzer, Kanten: Mentions/Retweets). Wendet den Louvain-Algorithmus zur Erkennung von Sub-Communities an. Parameter: `resolution` (Standard: 1.0), `min_community_size` (Standard: 10). Ergebnis: Liste der Communities mit Zentralitätsmetriken (Betweenness, Closeness, PageRank).

### So verwenden Sie es

1. **Konfiguration der Datenquellen und Filter-Pipelines**  
   Öffnen Sie die Datei `config/pipelines.yaml` und definieren Sie mindestens eine Quelle mit Angabe von `source_type` (z. B. `discord_webhook`, `twitter_stream`), `rate_limit_per_sec` (Standard: 1000) und `filter_rules` (z. B. `language: de`, `min_account_age_days: 30`). Führen Sie anschließend `fanguard init --pipeline <name>` aus, um die Pipeline zu instanziieren.

2. **Starten der Echtzeit-Überwachung mit Metrik-Export**  
   Starten Sie den Daemon mit `fanguard run --export-metrics --metrics-port 9090`. Das System öffnet einen Prometheus-Endpunkt unter `/metrics`, der z. B. `fanguard_events_total`, `fanguard_toxicity_score_avg` und `fanguard_collision_cluster_count` bereitstellt. Nutzen Sie `curl http://localhost:9090/metrics` zur Abfrage.

3. **Auswertung von Reports und manuellen Eingriffen**  
   Generieren Sie einen detaillierten Bericht über die letzten 24 Stunden mit `fanguard report --time-range 24h --format json --output report.json`. Der Report enthält aggregierte Statistiken (z. B. `total_events`, `quarantined_events`, `top_toxic_users`). Greifen Sie manuell ein, indem Sie einen Nutzer mit `fanguard quarantine --user-id <id> --reason "Spam-Verdacht"` auf die Blacklist setzen.

### Beste Praktiken

- **Schrittweise Schwellwert-Kalibrierung in einer Staging-Umgebung**  
  Betreiben Sie fanguard zunächst in einer isolierten Testumgebung mit replizierten Datenströmen. Starten Sie mit `toxicity_threshold = 0.9` und senken Sie den Wert in 0.05-Schritten, bis die False-Positive-Rate unter 2 % fällt. Dokumentieren Sie jede Änderung im `config/calibration_log.yaml`. Dies verhindert Fehlalarme im Produktivbetrieb.

- **Regelmäßige Modell-Neuberechnung mit historischen Daten**  
  Trainieren Sie das Toxizitäts-Modell wöchentlich mit einem Datensatz aus den letzten 7 Tagen. Verwenden Sie den Befehl `fanguard train --data-source postgresql://user:pass@localhost:5432/fanguard --output-model ./models/latest.pkl`. Überprüfen Sie die Modellgüte anhand der Metrik `AUC-ROC` (Zielwert: > 0,92). Ein veraltetes Modell führt zu sinkender Erkennungsrate bei neuen Angriffsmustern.
