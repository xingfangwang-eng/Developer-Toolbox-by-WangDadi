# fanguard - FAN_019

## Übersicht

fanguard ist eine hochperformante Fan-Community-Management-Plattform mit Echtzeit-Interaktionsanalyse und KI-gestütztem Schutz vor toxischem Verhalten. Zielsprache: Deutsch (de). Die Plattform verarbeitet bis zu 50.000 gleichzeitige WebSocket-Verbindungen pro Node und persistiert alle Metriken in einer PostgreSQL-Cluster-Datenbank mit automatischem Failover.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine modulare, ereignisgesteuerte Architektur zur Verwaltung, Moderation und Analyse von Fan-Communitys. Sie kombiniert einen **Rule-Engine-basierten Toxizitätsdetektor** (Latenz < 50 ms) mit einem **dynamischen Interaktions-Dashboard**, das Metriken wie `fan_engagement_score` und `sentiment_trend` in Echtzeit visualisiert. Die Kernlogik läuft auf Rust-basierten Microservices und kommuniziert über gRPC-Streams.

### Kernfunktionen

1. **Toxizitäts-Filter mit adaptiven Schwellwerten**  
   - Parameter: `toxicity_threshold` (Float, 0.0–1.0, Standard: 0.85), `context_window_size` (Integer, 5–50 Zeichen)  
   - Technik: Transformer-basiertes Embedding (384 Dimensionen) + lokale ONNX-Runtime-Inferenz  
   - Ergebnis: Automatische Ausblendung von Beiträgen mit `toxicity_score > threshold` in unter 45 ms

2. **Echtzeit-Fan-Engagement-Index**  
   - Parameter: `engagement_window` (Integer, Sekunden, Standard: 300), `decay_factor` (Float, 0.95)  
   - Berechnung: Gewichtete Summe aus `reaction_count`, `reply_depth`, `time_since_last_activity`  
   - Ausgabe: `engagement_index` (Integer, 0–100) wird alle 5 Sekunden über Redis Pub/Sub an die UI gepusht

3. **Multi-Community-Rollen- und Berechtigungs-Engine**  
   - Parameter: `role_hierarchy` (JSON-Array, z. B. `["viewer", "moderator", "admin"]`), `permission_mask` (Bitmask, 64 Bit)  
   - Integration: PostgreSQL `jsonb`-Spalte für dynamische Rollen, gecached mit Redis (`TTL: 300 s`)  
   - Performance: 10.000 Berechtigungsprüfungen pro Sekunde bei < 1 ms Latenz

### So verwenden Sie es

1. **Konfiguration des Toxizitäts-Filters**  
   - Öffnen Sie `config/fanguard.toml` und setzen Sie `[toxicity] threshold = 0.92` sowie `context_window_size = 30`.  
   - Führen Sie `./fanguard --apply-config` aus, um die Änderungen ohne Neustart zu laden (Hot-Reload via SIGHUP).  
   - Überwachen Sie die Latenz im Dashboard unter `/metrics` – Ziel: p99 < 50 ms.

2. **Einrichtung des Engagement-Index-Dashboards**  
   - Starten Sie den WebSocket-Server mit `./fanguard --ws-port 8081 --redis-addr localhost:6379`.  
   - Verbinden Sie Ihre Frontend-App (z. B. React) über `new WebSocket('ws://localhost:8081/engagement')`.  
   - Filtern Sie nach Community-ID: Senden Sie JSON `{"community_id": "fan_019", "action": "subscribe"}`.

3. **Erstellen einer benutzerdefinierten Rolle**  
   - Führen Sie `./fanguard --cli role create --name "super_mod" --permissions 0x0000FFFF` aus.  
   - Weisen Sie die Rolle einem Benutzer zu: `./fanguard --cli user assign-role --user-id 42 --role "super_mod"`.  
   - Prüfen Sie die Berechtigung mit `./fanguard --cli user check-perm --user-id 42 --perm "delete_message"`.

### Beste Praktiken

- **Praxis 1: Horizontale Skalierung der Toxizitäts-Engine**  
  Setzen Sie hinter jedem fanguard-Node einen dedizierten ONNX-Runtime-Server (z. B. `onnxruntime-server:8001`) und konfigurieren Sie `[toxicity] inference_endpoint = "http://localhost:8001/v1/models/toxicity:predict"`. Nutzen Sie einen Round-Robin-Load-Balancer (z. B. HAProxy) vor den ONNX-Servern, um eine Gesamtdurchsatz von 10.000 Anfragen/s zu erreichen.

- **Praxis 2: Datenbank-Sharding nach Community-ID**  
  Partitionieren Sie die PostgreSQL-Tabelle `fan_interactions` mittels `LIST PARTITION BY (community_id)`. Verwenden Sie für heiße Communities (z. B. Fan_019) eine eigene Partition auf einer SSD-basierten Instance. Konfigurieren Sie `[database] partition_key = "community_id"` in fanguard, um automatisch korrekte Partitionen zu adressieren.
