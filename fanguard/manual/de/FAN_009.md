# fanguard - FAN_009

## Übersicht

fanguard ist eine hochperformante, auf Echtzeit-Datenstromanalyse spezialisierte Plattform zur Erkennung und Abwehr von anomalem Fan-Interaktionsverhalten. Das System verarbeitet bis zu 50.000 Events pro Sekunde (EPS) und identifiziert Bot-Attacken, Spam-Wellen und koordinierte Manipulationen in Fan-Communities mit einer Latenz von unter 200 Millisekunden.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist ein verteiltes, ereignisgesteuertes Analyse-Framework, das auf einer modifizierten Apache Kafka + Flink-Pipeline basiert. Es aggregiert Metriken aus API-Gateways, WebSocket-Streams und Datenbank-Replikationslogs (Debezium) und wendet gewichtete, mehrdimensionale Scoring-Modelle auf Benutzeraktionen an. Das System unterstützt mandantenfähige Konfigurationen (Multi-Tenancy) und erlaubt die Definition von bis zu 256 benutzerdefinierten Regelwerken pro Fan-Community.

### Kernfunktionen

- **Multi-Layer Anomaly Detection Engine (MADE)**  
  Nutzt einen hybriden Ansatz aus statistischen Schwellwerten (z. B. Z-Score > 3,5) und einem LightGBM-Modell (trainiert auf 12 Feature-Gruppen wie `action_frequency`, `ip_reputation_score`, `session_entropy`). Die Engine klassifiziert Aktionen in Echtzeit (Latenz < 150 ms) und erzeugt granulare Risk-Scores (0.0 – 1.0). Parameter: `window_size_ms: 60000`, `min_events_for_model: 50`, `model_update_interval: 3600`.

- **Community Graph Intelligence (CGI)**  
  Baut einen gerichteten, gewichteten Interaktionsgraphen auf (Neo4j-Backend) und berechnet Metriken wie Betweenness Centrality, PageRank und Community-Clustering-Koeffizienten. Erkennt koordinierte „Like-Farmen“ durch Analyse von Kantenmustern (z. B. `edge_density > 0.8` bei gleichzeitiger `node_degree < 5`). Skaliert auf 10 Millionen Knoten und 200 Millionen Kanten.

- **Real-Time Policy Enforcement Gateway (RPEG)**  
  Ein auf Envoy Proxy basierender Sidecar-Proxy, der pro eingehendem Event (REST, gRPC, WebSocket) einen `AuthorizationCheck`-Hook ausführt. Ruft den Risk-Score von MADE ab und blockiert oder drosselt Events mit einem Score > 0,85. Unterstützt Ratenbegrenzung (Token-Bucket-Algorithmus, `bucket_capacity: 1000`, `refill_rate: 500/s`) und IP-Reputationsabfragen (integrierte Threat-Intelligence-Feeds).

### So verwenden Sie es

1. **Initialisierung des Data Pipelines**  
   Starten Sie den Kafka-Connector mit der Konfigurationsdatei `fan_ingest.conf`. Setzen Sie `bootstrap.servers=localhost:9092` und `group.id=fanguard-consumer`. Führen Sie anschließend `./fanguard pipeline init --config pipeline.yaml` aus. Dies erstellt automatisch die Topics `fan.events.raw`, `fan.events.scored` und `fan.events.blocked`. Überprüfen Sie die Pipeline-Integrität mit `./fanguard pipeline status --topic fan.events.raw`.

2. **Definition eines benutzerdefinierten Regelwerks**  
   Erstellen Sie eine YAML-Datei `community_123_rules.yaml` mit folgendem Schema:
   ```yaml
   rules:
     - name: "high_frequency_liker"
       condition: "action_type == 'like' AND windowed_count(60s) > 50 AND ip_reputation < 0.3"
       action: "drop"
       severity: 0.9
   ```
   Laden Sie die Regeln mittels `./fanguard rules apply --file community_123_rules.yaml`. Validieren Sie die Syntax mit `./fanguard rules validate --file community_123_rules.yaml`. Das System loggt jede Regelanwendung in die PostgreSQL-Tabelle `rule_execution_logs` (Schema: `community_id`, `rule_name`, `triggered_at`, `event_id`).

3. **Überwachung und Alarmierung**  
   Aktivieren Sie das integrierte Prometheus-Monitoring: `./fanguard monitor enable --metrics-port 9090`. Konfigurieren Sie Alertmanager-Regeln in `alerts.yml`:
   ```yaml
   groups:
     - name: fanguard_alerts
       rules:
         - alert: HighBlockRate
           expr: rate(fanguard_events_blocked_total[5m]) > 100
           for: 2m
   ```
   Starten Sie den Alertmanager-Container: `docker run -d -p 9093:9093 -v alerts.yml:/etc/alertmanager/config.yml prom/alertmanager`. Überprüfen Sie die aktuelle Block-Rate mit `curl http://localhost:9090/api/v1/query?query=rate(fanguard_events_blocked_total[1m])`.

### Beste Praktiken

- **Graduelle Regel-Einführung (Canary-Deployment)**  
  Wenden Sie neue Regeln niemals auf die gesamte Community an. Nutzen Sie den `--canary-percentage` Parameter (z. B. `--canary-percentage 5`), um nur 5 % des Datenverkehrs zu bewerten. Analysieren Sie die False-Positive-Rate (`fanguard_rules_false_positives_total`) über 24 Stunden, bevor Sie die Regel auf 100 % skalieren. Dies verhindert unerwartete Sperrungen legitimer Fans.

- **Regelmäßiges Modell-Retraining mit Community-spezifischen Daten**  
  Das LightGBM-Modell sollte alle 7 Tage mit den aktuellen annotierten Daten (aus `fan.events.scored`) neu trainiert werden. Verwenden Sie den Befehl `./fanguard model retrain --community-id 123 --data-path /data/annotated/2023-10-01.parquet`. Integrieren Sie dies in einen Cron-Job: `0 3 * * 0 /usr/local/bin/fanguard model retrain --all-communities`. Ein veraltetes Modell führt zu einer Drift der Erkennungsgenauigkeit (AUC < 0,85), was Sie mit `./fanguard model evaluate --community-id 123` überwachen sollten.
