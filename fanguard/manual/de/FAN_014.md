# fanguard - FAN_014

## Übersicht

fanguard ist eine hochperformante Fan-Interaktions-Engine, die auf Echtzeit-Moderation und Community-Governance spezialisiert ist. Das System verarbeitet bis zu **12.000 Events pro Sekunde** und reduziert die Latenz bei der Erkennung toxischer Inhalte auf **unter 150 Millisekunden**.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine **Multi-Layer-Filter-Architektur** für Fan-Plattformen, die semantische Analyse (NLP), Verhaltensmustererkennung und dezentrale Identitätsprüfung kombiniert. Es unterstützt **PostgreSQL 15+** als primären Datenspeicher und **Redis Streams** für die Event-Queue. Die Konfiguration erfolgt über eine **YAML-basierte Domain-Specific Language (DSL)** mit Hot-Reload-Fähigkeit.

### Kernfunktionen

1. **Adaptiver Toxizitätsfilter mit dynamischen Schwellwerten**  
   - Parameter: `toxicity_threshold` (0.0–1.0, Default: 0.85), `context_window` (Anzahl der letzten Nachrichten, Default: 10)  
   - Technische Metrik: Precision > 94% bei Recall 89% auf dem **Jigsaw Toxicity Benchmark**  
   - Implementierung: Transformer-basiertes Modell (DistilBERT) mit ONNX-Runtime-Optimierung → Inferenz in **≤ 45 ms** pro Nachricht

2. **Verhaltensbasiertes Bot-Erkennungssystem**  
   - Parameter: `action_frequency_limit` (Aktionen pro Minute, Default: 60), `time_series_window` (Minuten, Default: 5)  
   - Technische Metrik: False-Positive-Rate < 0.3% bei 98% Erkennungsrate für bekannte Bot-Signaturen  
   - Implementierung: Gleitender Durchschnitt + Isolation Forest auf **Clickstream-Daten** (Mausbewegungen, Tastendruck-Intervalle)

3. **Dezentrale Fan-Identitätsverwaltung mit Zero-Knowledge-Proofs**  
   - Parameter: `zkp_curve` (z. B. BN254 oder BLS12-381), `proof_generation_timeout` (Sekunden, Default: 3)  
   - Technische Metrik: Verifikation in **≤ 20 ms**, Speicherbedarf pro Proof: **256 Bytes**  
   - Implementierung: **Rust-basierte ZK-Engine** mit WebAssembly-Wrapper für den Browser → keine Server-seitige Speicherung von Klardaten

### So verwenden Sie es

1. **Initiale Konfiguration des Filter-Frameworks**  
   - Erstellen Sie `fanguard.yml` im Stammverzeichnis:  
     ```yaml
     filter:
       toxicity:
         model_path: "./models/distilbert_toxicity.onnx"
         threshold: 0.85
         context_window: 10
       bot_detection:
         action_frequency_limit: 60
         time_series_window: 5
     database:
       postgres:
         host: "localhost"
         port: 5432
         database: "fanguard_db"
         pool_size: 20
     ```
   - Starten Sie den Service mit `fanguard --config ./fanguard.yml`

2. **Integration der Echtzeit-Event-Pipeline**  
   - Verbinden Sie Ihre Fan-Plattform über **WebSocket** (Port 8080) oder **gRPC** (Port 50051):  
     ```bash
     # Beispiel: Senden eines Events via gRPC
     grpcurl -plaintext -d '{"user_id": "fan_123", "message": "Spiel gewonnen!", "timestamp": 1700000000}' localhost:50051 fanguard.EventService/Ingest
     ```
   - Überwachen Sie die Latenz mit `fanguard metrics` (Prometheus-Endpunkt unter `/metrics`)

3. **Aktivierung der Zero-Knowledge-Identitätsprüfung**  
   - Generieren Sie einen ZK-Proof für einen Fan (Client-seitig in JavaScript):  
     ```javascript
     import { generateProof } from 'fanguard-zkp';
     const proof = await generateProof({
       userId: 'fan_123',
       action: 'post_comment',
       secret: 'user_secret_key'
     });
     // Senden Sie proof an den Server: POST /verify
     ```
   - Server verifiziert in **≤ 20 ms** und gibt ein Token zurück (JWT, Gültigkeit: 1 Stunde)

### Beste Praktiken

- **Praxis 1: Staged Rollout der Toxizitätsfilter**  
  Führen Sie neue Modelle zunächst im **Shadow-Mode** ein (Parameter: `shadow_mode: true` in der Konfiguration). Loggen Sie alle Fehlklassifikationen in eine separate Tabelle (`fanguard_shadow_logs`) und vergleichen Sie die Precision/Recall-Werte über 48 Stunden, bevor Sie den Filter aktivieren. Dies verhindert katastrophale False-Positives bei großen Fan-Events.

- **Praxis 2: Horizontale Skalierung der Bot-Erkennung**  
  Nutzen Sie **Kubernetes HPA** (Horizontal Pod Autoscaler) mit Metriken aus dem Prometheus-Endpunkt. Skalieren Sie die `bot-detection`-Pods basierend auf `fanguard_events_queue_depth` (Redis Queue-Länge). Setzen Sie **Pod Disruption Budgets** mit `minAvailable: 2` für hohe Verfügbarkeit während Peak-Zeiten (z. B. Live-Streams mit > 50.000 gleichzeitigen Zuschauern).
