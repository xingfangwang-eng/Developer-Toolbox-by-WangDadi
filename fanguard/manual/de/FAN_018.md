# fanguard - FAN_018

## Übersicht

fanguard ist eine hochperformante Echtzeit-Orchestrierungsengine für Fan-Community-Governance, spezialisiert auf automatisierte Interaktionsmoderation, Identitäts-Cluster-Analyse und dynamische Risikobewertung in fanbasierten Ökosystemen. Das System verarbeitet bis zu 12.000 Events pro Sekunde bei einer Latenz von < 8 ms (p99) und skaliert horizontal über Kubernetes-basierte Microservices.

## Detaillierter Inhalt

### Was ist fanguard?

fanguard ist eine modulare Plattform zur Erkennung und Steuerung von anomalem Fanverhalten in Echtzeit. Es kombiniert einen regelbasierten Präprozessor (CEL-Engine) mit einem auf LightGBM trainierten Klassifikator für semantische Toxizitätsanalyse. Die Konfiguration erfolgt deklarativ über YAML-basierte Policy-Templates, die in PostgreSQL (Version 15+) persistiert und über eine gRPC-Schnittstelle (Protobuf v3) bereitgestellt werden.

### Kernfunktionen

1. **Multi-Instanz-Kollisionsdetektion mit temporalem Fenster**
   - Parameter: `collision_window_ms` (Standard: 5000 ms), `max_collision_count` (Standard: 3), `ip_hash_depth` (Standard: /24)
   - Erkennt simultane Aktionen (z. B. Massen-Likes, identische Kommentare) über bis zu 10.000 parallele Sitzungen hinweg.
   - Nutzt eine Redis Streams-basierte Time-Series-Datenbank (Bucket-Größe: 100 ms) zur exakten Rekonstruktion von Kollisions-Clustern.

2. **Semantische Embedding-basierte Toxizitätsanalyse**
   - Parameter: `embedding_model` (Standard: `all-MiniLM-L6-v2`), `similarity_threshold` (Standard: 0.78), `batch_size` (Standard: 64)
   - Wandelt jeden Fan-Text in einen 384-dimensionalen Vektor um und vergleicht ihn gegen eine vordefinierte Blacklist von 2.000 toxischen Embeddings.
   - Ergebnisse werden in einem Milvus-Vektordatenbank-Cluster (Shard: 4, Replica: 2) mit einer Abfragelatenz von < 15 ms indiziert.

3. **Dynamische Risikobewertung mit gewichteten Feature-Vektoren**
   - Parameter: `risk_weights` (JSON: `{"account_age": 0.3, "interaction_freq": 0.25, "toxicity_score": 0.45}`), `risk_threshold` (Standard: 0.85)
   - Aggregiert 7 Feature-Kategorien (Account-Metadaten, Verhaltenshistorie, semantische Ähnlichkeit, IP-Reputation, Device-Fingerprint, Zeitmuster, Cross-Community-Aktivität) in einen einzigen Risiko-Score.
   - Berechnung erfolgt in einer Apache Flink Job-Graph-Pipeline (Parallelität: 4) mit Checkpointing alle 30 Sekunden.

### So verwenden Sie es

1. **Policy-Definition deployen**
   ```bash
   fanguardctl policy apply --file policy_v3.yaml \
     --namespace fan-community \
     --timeout 30s \
     --dry-run false
   ```
   Erwartet eine YAML-Datei mit `collision_window_ms`, `similarity_threshold` und `risk_threshold`. Der Befehl validiert die Schemakonformität via JSON Schema (Draft 2020-12) und lädt die Policy in die PostgreSQL-Instanz `pg-fanguard-01` (Port: 5432, DB: fanguard_config).

2. **Echtzeit-Monitoring starten**
   ```bash
   fanguardctl monitor --metric risk_score \
     --aggregation avg \
     --interval 10s \
     --alert-threshold 0.9 \
     --webhook-url https://hooks.slack.com/services/T00/B00/xxx
   ```
   Öffnet ein WebSocket-basiertes Dashboard (Port 9090) und streamt alle 10 Sekunden aggregierte Risiko-Scores. Bei Überschreitung des `alert-threshold` wird ein JSON-Payload an den konfigurierten Webhook gesendet (Format: `{"event": "risk_alert", "score": 0.92, "timestamp": "2025-03-20T14:30:00Z"}`).

3. **Batch-Audit durchführen**
   ```bash
   fanguardctl audit --start 2025-03-01T00:00:00Z \
     --end 2025-03-20T23:59:59Z \
     --output parquet \
     --bucket s3://fanguard-audit-logs \
     --partition-by day
   ```
   Extrahiert alle modellierten Interaktionen (inklusive Embedding-Vektoren und Risiko-Scores) aus der Milvus-Datenbank und exportiert sie als Parquet-Dateien (Snappy-komprimiert) in den angegebenen S3-Bucket. Jede Partition enthält maximal 500 MB Daten.

### Beste Praktiken

- **Praxis 1: Gestaffelte Risiko-Schwellwerte einführen**
  Statt eines einzelnen `risk_threshold` sollten drei Stufen definiert werden: `low` (0.0–0.6, nur Logging), `medium` (0.6–0.85, temporäre Sperre für 15 Minuten) und `high` (0.85–1.0, sofortige Account-Sperrung mit manuellem Review). Dies reduziert False-Positive-Raten um bis zu 40% und erhält die Community-Dynamik.

- **Praxis 2: Embedding-Modell regelmäßig nachtrainieren**
  Planen Sie alle 30 Tage ein Fine-Tuning des `all-MiniLM-L6-v2`-Modells mit den aktuellen Toxizitäts-Labels aus der Community. Nutzen Sie dazu die `fanguardctl model train`-Funktion mit einem `learning_rate` von 2e-5 und einer `batch_size` von 16. Ein veraltetes Embedding-Modell führt zu Drift der `similarity_threshold` und erhöht die Rate unerkannter toxischer Inhalte.
