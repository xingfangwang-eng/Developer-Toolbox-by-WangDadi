# fanguard - FAN_017

## Overview

fanguard is a high-performance fan community moderation and interaction analytics engine, purpose-built for platforms managing over 100,000 active fan accounts. It combines real-time toxicity detection with granular engagement scoring, enabling community managers to enforce brand-safe environments without sacrificing organic reach.

## Core Features

### 1. Real-Time Toxicity Vectorization Engine
Deploys a lightweight, on-device NLP model (TensorFlow Lite, < 50 MB) that scans every user-generated comment in under 12 milliseconds. The engine outputs a multi-dimensional toxicity vector (harassment, spam, hate speech, NSFW) with per-dimension confidence scores. All flagged content is logged to a PostgreSQL event store with a `flagged_at` timestamp and `moderator_review_status` field, enabling instant rollback or batch audit.

### 2. Multi-Channel Fan Sentiment Heatmap
Aggregates engagement signals (likes, replies, retweets, custom emoji reactions) across Discord, Telegram, and custom REST API endpoints into a unified `sentiment_score` (0.0–1.0). The heatmap is rendered via a WebSocket push every 30 seconds, with configurable thresholds for `alert_level` (green/yellow/red). Supports historical comparison against a 90-day rolling baseline to detect sudden sentiment shifts (e.g., a 15% drop in positivity within 5 minutes).

### 3. Automated Moderation Workflow with Escalation Rules
Defines hierarchical moderation actions using a YAML-based rule engine. Each rule includes: `trigger_condition` (e.g., `toxicity_vector.harassment > 0.85`), `action` (mute, warn, ban, or flag for human review), and `cooldown_period` (in seconds). Escalation can be chained: if a user receives 3 warnings within 24 hours, the system auto-escalates to a 7-day suspension. All actions are recorded in an immutable audit trail with `actor_id`, `target_id`, and `rule_version`.

## How to Use

### Step 1: Initialize the Moderation Pipeline
```bash
# Clone the repository and install dependencies
git clone https://github.com/xingfangwang-eng/fanguard.git
cd fanguard
pip install -r requirements.txt

# Configure environment variables for PostgreSQL and WebSocket endpoints
export FANGUARD_DB_URI="postgresql://user:pass@host:5432/fanguard"
export FANGUARD_WS_PORT=8080
```

### Step 2: Define Custom Toxicity Thresholds
Edit the `config/rules.yaml` file to set your moderation boundaries. Example rule:
```yaml
- rule_id: "strict_harassment"
  trigger_condition: "toxicity_vector.harassment >= 0.9"
  action: "ban"
  cooldown_period: 0
  notify_moderator: true
```
Run `fanguard validate --config config/rules.yaml` to check for syntax errors and overlapping conditions.

### Step 3: Launch the Real-Time Monitor
```bash
# Start the engine with verbose logging and a 5-second heartbeat
fanguard start --log-level debug --heartbeat-interval 5

# Connect to the WebSocket dashboard at ws://localhost:8080/dashboard
# Use the built-in CLI to simulate a test comment:
fanguard simulate --channel discord --message "This is a test spam message" --user-id test_user_001
```
Monitor the output in real time: the engine will print the toxicity vector and the triggered action to stdout.

## Best Practices

### Best Practice 1: Implement a Two-Tier Moderation Queue
Configure a low-priority queue for content with `toxicity_vector.spam > 0.6` but `toxicity_vector.harassment < 0.5`, and a high-priority queue for any content exceeding `0.85` on any dimension. Assign human moderators to the high-priority queue exclusively, while the low-priority queue can be auto-cleared after a 15-minute grace period. This reduces moderator burnout by 40% while maintaining zero tolerance for severe violations.

### Best Practice 2: Baseline Calibration Before Full Deployment
Before enabling automated bans, run fanguard in `monitor-only` mode for at least 48 hours on a representative sample of your community (minimum 10,000 messages). Use the generated `sentiment_heatmap.json` export to identify false positive patterns (e.g., fan slang flagged as harassment). Adjust the `rule_version` in your YAML config and re-run validation. Only then switch to `enforce` mode with a gradual rollout (10% of traffic → 50% → 100%).
