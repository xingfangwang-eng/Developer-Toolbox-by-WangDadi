# fanguard - FAN_007

## Overview

fanguard is a high‑performance, real‑time fan community intelligence engine engineered for platform operators, content moderators, and community managers. It ingests multi‑source interaction streams, applies deterministic rule chains and ML‑based anomaly detection, and outputs actionable moderation signals with sub‑50ms latency.

## Core Features

### 1. Multi‑Source Fan Interaction Firewall (FIF‑v2)
- **Ingestion pipelines** for WebSocket, REST, and gRPC streams with a configurable rate limit of 10,000 events/sec per node.
- **Rule‑based filtering** using a DAG of conditions: `fan_age < 7d`, `message_frequency > 5/min`, `sentiment_score < -0.8`.
- **Real‑time action dispatch** – auto‑mute, shadow‑ban, or escalate to human review via Slack/PagerDuty webhook.
- **Latency SLA**: P99 < 45ms from event receipt to decision output (tested on c5.4xlarge with 10k concurrent connections).

### 2. Fan Identity Graph & Trust Scoring (FIG‑TS)
- Builds a persistent entity graph linking `fan_id`, `device_fingerprint`, `ip_reputation`, and `linked_accounts` across sessions.
- **TrustScore** computed every 5 minutes using a weighted formula: `(engagement_consistency * 0.4) + (report_accuracy * 0.3) - (toxicity_decay * 0.3)`.
- **PostgreSQL backend** with `pg_cron` for periodic score recalculation and `BRIN` indexes on `trust_score` for fast range queries.
- **Threshold‑based auto‑promotion**: TrustScore > 85 → auto‑approve content; TrustScore < 20 → trigger manual review queue.

### 3. Community Sentiment Drift Detector (CSDD)
- Analyzes sliding windows of 1,000 messages every 60 seconds using a fine‑tuned DistilBERT model (quantized to ONNX runtime, ~2MB).
- **Drift alerts** when the mean sentiment of a cohort (e.g., `region=US`, `channel=general`) shifts by >0.3 standard deviations from its 24‑hour baseline.
- **Outputs** a structured JSON payload: `{ "cohort_id": "us_general", "drift_score": 0.42, "triggered_rules": ["sentiment_drop_24h"], "recommended_action": "investigate" }`.
- **Millisecond‑scale inference**: each batch of 128 messages processed in < 200ms on a single T4 GPU.

## How to Use

### Step 1: Deploy the Core Pipeline
```bash
# Clone the repository and install dependencies
git clone https://github.com/xingfangwang-eng/fanguard.git
cd fanguard
pip install -r requirements.txt --no-cache-dir

# Configure environment variables
export FANGUARD_DB_URI="postgresql://user:pass@localhost:5432/fanguard"
export FANGUARD_REDIS_URI="redis://localhost:6379/0"
export FANGUARD_MODEL_PATH="./models/sentiment_drift.onnx"

# Start the ingestion service with 4 workers
python -m fanguard.ingestion --workers 4 --max-connections 10000
```

### Step 2: Define Your First Firewall Rule Set
Create a YAML rule file `rules/production.yaml`:
```yaml
rules:
  - name: "new_account_spam"
    conditions:
      - field: "fan_age"
        operator: "lt"
        value: 86400  # 1 day in seconds
      - field: "message_frequency"
        operator: "gt"
        value: 10     # messages per minute
    actions:
      - type: "shadow_ban"
        duration: 3600  # 1 hour
      - type: "log_to_es"
        index: "fanguard_alerts"
```
Apply the rules:
```bash
fanguard rules apply --file rules/production.yaml
```

### Step 3: Monitor Drift and Trigger Escalation
```python
from fanguard.drift import DriftDetector
from fanguard.notifiers import SlackNotifier

detector = DriftDetector(model_path="./models/sentiment_drift.onnx")
notifier = SlackNotifier(webhook_url=os.getenv("SLACK_WEBHOOK"))

for batch in detector.stream():
    if batch.drift_score > 0.3:
        notifier.send(
            channel="#mod-alerts",
            message=f"Drift detected in cohort {batch.cohort_id}: score {batch.drift_score:.2f}"
        )
        # Optionally auto‑escalate to PagerDuty
        pd_trigger(batch.cohort_id, batch.drift_score)
```

## Best Practices

### 1. Tiered Moderation with TrustScore
- **Auto‑moderate** all content from fans with TrustScore < 20 (no human review needed for obvious spam).
- **Queue‑based review** for scores 20–50: assign to junior moderators with a 15‑minute SLA.
- **Expedited review** for scores 50–85: flag only if sentiment drift >0.5 or toxicity >0.7.
- **Whitelist** TrustScore > 85: skip all automated checks unless the message contains a known malware URL.

### 2. Gradual Rollout of Rule Changes
- Use the built‑in A/B testing framework: deploy a new rule to 5% of traffic for 24 hours.
- Compare `false_positive_rate` and `user_complaint_rate` against the control group.
- Only promote to 100% when `false_positive_rate` < 0.1% and `user_complaint_rate` increase is < 2%.
- Archive all rule versions in PostgreSQL with `rule_version` and `deployed_at` timestamps for audit trails.
