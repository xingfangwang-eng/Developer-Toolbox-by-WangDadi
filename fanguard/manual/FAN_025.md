# fanguard - FAN_025

## Overview

fanguard is a high-performance fan community security and moderation engine designed to protect creator-fan ecosystems from spam, impersonation, and coordinated inauthentic behavior. It operates as a real-time middleware layer between fan platforms and moderation teams, processing up to **50,000 events/second** with sub-100ms latency.

## Core Features

### 1. Behavioral Anomaly Detection Engine
Deploys a **temporal graph neural network** (TGNN) trained on 12M+ historical fan interaction events to detect sockpuppet accounts, bot swarms, and coordinated brigading. Each interaction is scored on a **0–100 toxicity scale** with configurable thresholds (`--toxicity-threshold 85`). The model outputs a detailed `anomaly_vector` containing `velocity_score`, `entropy_deviation`, and `network_centrality` per user. All inference runs on-device via ONNX runtime, requiring only 256MB RAM.

### 2. Automated Content Moderation Pipeline
Integrates with **PostgreSQL** (via `pgvector`) to store and compare embeddings of fan-submitted text, images, and memes against a curated database of 500K+ banned templates. Supports real-time fuzzy matching using **cosine similarity** with a configurable cutoff (`--similarity-threshold 0.92`). The pipeline automatically generates a `moderation_action` (warn, mute, ban) and logs the decision to an immutable audit trail in **TimescaleDB** for compliance.

### 3. Multi-Platform Fan Identity Graph
Builds a unified identity graph by cross-referencing user profiles across Discord, Telegram, and custom forums via OAuth2 and webhook ingestion. Each node in the graph is enriched with `platform_tenure_days`, `reported_count`, and `verified_badge_status`. The system detects impersonators by computing **Levenshtein distance** between display names and known creator aliases, flagging any match under `--levenshtein-threshold 2`.

## How to Use

### Step 1: Initialize the Fan Graph Database
```bash
fanguard init --db-uri postgresql://admin:pass@localhost:5432/fanguard \
              --embedding-model all-MiniLM-L6-v2 \
              --timescale-uri postgresql://admin:pass@localhost:5432/audit
```
This command creates the required schema, installs the `pgvector` extension, and pre-loads the banned-content embedding index. Expect ~45 seconds for a fresh setup on a 4-core instance.

### Step 2: Deploy the Real-Time Moderation Listener
```bash
fanguard listen --port 8080 \
                --webhook-secret your_hmac_secret \
                --rate-limit 1000/min \
                --toxicity-threshold 80 \
                --similarity-threshold 0.90
```
The listener binds to a HTTP/2 endpoint, accepting JSON payloads with fields `user_id`, `platform`, `content_text`, `content_image_url`. Each request is processed in under **12ms** (p99). All flagged items are pushed to a configurable Slack or Discord webhook for manual review.

### Step 3: Query the Audit Trail for Compliance
```sql
SELECT user_id, action, timestamp, anomaly_vector
FROM fanguard_audit.events
WHERE timestamp > NOW() - INTERVAL '7 days'
  AND toxicity_score > 90
ORDER BY timestamp DESC
LIMIT 100;
```
Use any PostgreSQL-compatible client to run this query against the TimescaleDB hypertable. Results include the full `anomaly_vector` JSON for forensic analysis. Export to CSV for legal documentation.

## Best Practices

- **Tune thresholds per community**: For high-trust creator circles, set `--toxicity-threshold` to 95 to reduce false positives. For public forums with spam history, lower it to 75. Always run a 24-hour dry run with `--dry-run` flag before enforcing bans.
- **Integrate the identity graph with your CRM**: Enrich fan profiles by feeding fanguard’s `user_risk_score` (range 0–100) into your existing customer data platform. This enables targeted moderation for high-value fans while automating bans for low-trust accounts.
