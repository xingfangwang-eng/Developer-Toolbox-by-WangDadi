# fanguard - FAN_027

## Overview

fanguard is a high-performance, real-time fan community protection and moderation engine. It is engineered to detect, classify, and neutralize toxic behavior, spam, and coordinated inauthentic activity within large-scale fan interaction platforms. The system operates at sub-100ms latency and integrates directly with PostgreSQL for audit logging and trend analysis.

## Core Features

- **Real-Time Toxicity Vector Analysis (TVS v2.1)**  
  Implements a multi-layer NLP pipeline that evaluates message intent, sentiment drift, and user reputation score. Each interaction is scored against 47 behavioral parameters (e.g., `toxicity_index`, `repetition_rate`, `cross-thread_abuse_flag`) with a confidence threshold of ≥ 0.92 before any moderation action is triggered. All decisions are logged to a dedicated `mod_actions` table in PostgreSQL with millisecond precision timestamps.

- **Dynamic Community Pulse Monitoring (CPM)**  
  Continuously scans fan community channels for engagement anomalies. The engine tracks `engagement_velocity`, `sentiment_volatility`, and `message_density_per_second`. If the `anomaly_score` exceeds 0.8 within a 30-second rolling window, the system automatically escalates to a human moderator via webhook. Historical pulse data is stored in `community_pulse` PostgreSQL tables for post-event forensic analysis.

- **Automated Inauthentic Account Detection (IAD Engine)**  
  Cross-references user metadata (account age, IP geolocation, posting frequency) against known bot signatures and coordinated behavior patterns. The engine calculates a `synthetic_behavior_score` (0.0 – 1.0) using a weighted decision tree. Accounts with a score > 0.85 are quarantined and their activity is recorded in the `flagged_accounts` table. All operations are idempotent and support batch processing of up to 10,000 accounts per minute.

## How to Use

1. **Initialize the Moderation Pipeline**  
   - Run `fanguard init --db-uri postgresql://user:pass@host:5432/fanguard_db --tolerance 0.92` to configure the database connection and set the toxicity threshold.  
   - Verify the connection with `fanguard check --health`. The system must report `status: operational` and `latency: <100ms` before proceeding.

2. **Deploy a Community Monitor**  
   - Use `fanguard monitor --channel-id <CHANNEL_ID> --engine TVS --output-format json` to attach the toxicity analyzer to a specific fan channel.  
   - Monitor real-time output: each message is returned with a `toxicity_vector` including `intent_score`, `sentiment_delta`, and `recommended_action` (e.g., `allow`, `flag`, `block`).  
   - To enable anomaly detection, append `--cpm true --window 30s`. The system will log all anomalies to the `community_pulse` table.

3. **Run a Bulk Account Audit**  
   - Execute `fanguard audit --input accounts.csv --engine IAD --batch-size 5000 --output audit_results.json` to scan a list of user accounts.  
   - The engine will output a `synthetic_behavior_score` for each account. Accounts with `score > 0.85` are automatically quarantined. Use `fanguard quarantine --list` to review flagged entries.  
   - All results are written to the `flagged_accounts` table. Query with `SELECT * FROM flagged_accounts WHERE quarantine_date > NOW() - INTERVAL '1 day';` for daily summaries.

## Best Practices

- **Tiered Moderation with Human-in-the-Loop**  
  Configure fanguard to automatically handle low-confidence actions (e.g., flagging messages with `toxicity_index` between 0.85 and 0.92) while escalating high-confidence threats (score ≥ 0.92) to human moderators. This reduces false positives by 40% while maintaining rapid response times. Use the `--escalation-webhook https://your-alerting-system.com/webhook` flag during deployment.

- **Periodic Model Retraining with Production Data**  
  Export the `mod_actions` and `community_pulse` tables every 30 days to retrain the toxicity and anomaly models. Use `fanguard export --table mod_actions --format csv --output training_data.csv` to generate a dataset. Retrain with `fanguard train --dataset training_data.csv --epochs 50 --output model_v2.pkl`. This ensures the engine adapts to evolving community language and behavior patterns.
