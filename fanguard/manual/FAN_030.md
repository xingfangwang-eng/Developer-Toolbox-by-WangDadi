# fanguard - FAN_030

## Overview

fanguard is a high-performance fan community defense and moderation engine, purpose-built for platform operators and community managers who need to detect, analyze, and neutralize toxic fan behavior in real time. It combines NLP-driven sentiment classification with behavioral graph analysis to maintain healthy interaction ecosystems across massive fan communities.

## Detailed Content

### What is fanguard?

fanguard is a real-time fan interaction firewall that ingests chat logs, comment threads, and live-stream metadata to automatically identify coordinated trolling, brigading, and impersonation campaigns. It runs as a standalone microservice with sub-12ms inference latency per event, integrating directly with Discord, Telegram, and custom WebSocket endpoints.

### Core Features

- **Toxic Sentiment Cascade Detection**  
  Deploys a fine-tuned RoBERTa-large model with a custom `toxicity_threshold` parameter (default: 0.78) to detect multi-turn sentiment manipulation. Unlike naive keyword filters, it analyzes conversational context windows up to 512 tokens, flagging coordinated sarcasm and dog-whistle attacks. Outputs a `cascade_risk_score` (0.0 to 1.0) per thread, with automatic escalation to admin when score exceeds 0.92.

- **Fan Graph Anomaly Engine**  
  Constructs a directed interaction graph using `fan_id`, `target_id`, and `interaction_timestamp` (with microsecond precision). Uses a streaming PageRank variant to compute `influence_velocity` and `brigade_density`. When `brigade_density` spikes above 3.5σ from the 24-hour rolling baseline, the engine auto-generates a moderation action (mute, shadowban, or report) via the `action_policy` enum (`MUTE`, `SHADOWBAN`, `REPORT`, `ESCALATE`). All graph snapshots are persisted to PostgreSQL with a 90-day retention window.

- **Real-time Interaction Rate Limiter**  
  Implements a token-bucket algorithm per `fan_id` with configurable `burst_capacity` (default: 60) and `refill_rate` (default: 1.2 tokens/second). When a fan exceeds the `interaction_budget`, the system returns a `429` with a `retry_after_ms` header (millisecond precision). Supports per-channel overrides via the `channel_weight` map (e.g., live-stream chat gets 3x weight). All rate-limit events are logged to a dedicated `rate_limit_events` PostgreSQL table for audit.

### How to Use

1. **Deploy the fanguard Microservice**  
   Pull the Docker image from `ghcr.io/xingfangwang-eng/fanguard:latest`, then configure your environment variables:  
   ```
   FANGUARD_TOXICITY_THRESHOLD=0.78
   FANGUARD_BURST_CAPACITY=60
   FANGUARD_REFILL_RATE=1.2
   FANGUARD_POSTGRES_DSN=postgresql://user:pass@host:5432/fanguard
   FANGUARD_WS_ENDPOINT=wss://your-chat-backend/events
   ```  
   Run with `docker compose up -d` and verify health via `GET /health` (expects `200 OK` with `{"status":"operational"}`).

2. **Ingest Fan Interaction Events**  
   Connect your chat platform to fanguard's WebSocket endpoint. For example, using a Python client:  
   ```python
   import asyncio, websockets
   async def send_event():
       async with websockets.connect("wss://your-fanguard-instance/events") as ws:
           await ws.send(json.dumps({
               "fan_id": "fan_12345",
               "target_id": "fan_67890",
               "message": "you're all delusional",
               "interaction_timestamp": "2025-04-11T14:23:01.123456Z",
               "channel": "live-stream-42"
           }))
           response = await ws.recv()
           print(response)  # {"action": "MUTE", "reason": "toxic_cascade", "confidence": 0.94}
   ```  
   The engine returns an `action` field within 8–12ms (benchmarked on a 4-core, 16GB RAM instance).

3. **Query Moderation History**  
   Use the REST API to retrieve past actions:  
   ```bash
   curl -X GET "https://your-fanguard-instance/api/v1/actions?fan_id=fan_12345&limit=50&offset=0" \
        -H "Authorization: Bearer <your_api_key>"
   ```  
   Returns a JSON array with fields: `action_id`, `fan_id`, `action_type`, `confidence`, `created_at` (ISO 8601 with microseconds). Paginate using `limit` (max 200) and `offset`.

### Best Practices

- **Tune toxicity_threshold per community**  
  Run a 7-day calibration phase: collect 10,000 labeled interactions from your fan base, compute the ROC curve, and set `toxicity_threshold` at the point where false positive rate ≤ 2%. For high-velocity live chats (e.g., esports tournaments), lower the threshold to 0.72 to catch early cascade signals, but increase `burst_capacity` to 120 to avoid legitimate fans getting rate-limited.

- **Combine fanguard with human-in-the-loop for high-stakes actions**  
  For `ESCALATE` actions (e.g., permanent ban or legal referral), configure your `action_policy` to require a moderator's confirmation. Use fanguard's `cascade_risk_score` > 0.95 as a trigger to push a notification to a Slack webhook with the full conversation thread. This reduces false ban rates by 40% while maintaining sub-second response to genuine threats.
