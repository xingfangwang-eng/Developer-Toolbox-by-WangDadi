# fanguard - FAN_019

## Overview

fanguard is a high-performance fan community protection and interaction orchestration engine. It provides real-time, granular moderation and engagement analytics across decentralized fan platforms, targeting sub-50ms latency for threat detection and community health scoring.

## Core Features

### 1. Real-Time Fan Sentiment Anomaly Detection & Auto-Mitigation
Leverages a custom NLP pipeline (transformer-based, fine-tuned on fan community jargon) to detect toxic sentiment, coordinated spam, or brigading attempts within 30ms of message submission. The system automatically escalates to configurable actions: shadow-ban, rate-limit, or soft-block, based on a `toxicity_score` (0.0–1.0) and `source_credibility_index` (derived from account age, interaction history, and cross-platform ID verification). All events are logged with nanosecond-precision timestamps in a PostgreSQL + TimescaleDB cluster for post-mortem analysis.

### 2. Cross-Platform Fan Community Graph Mapping
Builds a dynamic, real-time interaction graph across multiple platforms (Discord, Telegram, custom forums) using a federated identity resolver. The `interaction_frequency` metric (edges per minute) and `community_cohesion_coefficient` (0–100) are calculated via a distributed graph processing engine (Apache Flink). This enables detection of echo chambers, influencer-driven cascades, and fan migration patterns, with data exportable as JSON or GraphML for external analysis.

### 3. Automated Fan Engagement Campaign Orchestrator
Allows creation of multi-step, conditional engagement workflows (e.g., “If user reacts to post A with emoji B, then unlock hidden content C”). Supports A/B testing of engagement triggers with statistical significance reporting (p-value < 0.05). Each step is parameterized with `timeout_ms` (default 60000ms), `max_participants`, and `reward_delivery_webhook`. All campaign metrics are streamed to a dedicated Kafka topic for real-time dashboarding.

## How to Use

### Step 1: Initialize the Fan Community Connector
Deploy the fanguard agent using the provided Docker image. Configure the `platforms.yaml` file with your API tokens and webhook endpoints for each target platform (e.g., Discord, Telegram). Run `fanguardctl init --config ./platforms.yaml` to test connectivity. The agent will validate each endpoint with a ping request and log the latency (must be <200ms for production use).

### Step 2: Configure Sentiment Thresholds & Auto-Mitigation Rules
Define your moderation policy in `policy.json`. Example:
```json
{
  "toxicity_threshold": 0.85,
  "action_on_breach": "shadow_ban_24h",
  "rate_limit_per_user_per_minute": 5,
  "webhook_on_escalation": "https://your-server.com/webhook"
}
```
Apply the policy using `fanguardctl apply-policy --policy policy.json`. The system will immediately begin evaluating all incoming fan messages against this rule set, logging every decision with a unique `decision_id`.

### Step 3: Launch an Engagement Campaign
Create a campaign definition file `campaign.yaml`:
```yaml
campaign_name: "Fan Art Contest"
trigger:
  type: reaction
  platform: discord
  target_message_id: "123456789"
  emoji: "🎨"
action:
  type: unlock_content
  payload_url: "https://cdn.example.com/secret_art.png"
  delivery_method: direct_message
limits:
  timeout_ms: 300000
  max_participants: 100
```
Run `fanguardctl start-campaign --campaign campaign.yaml`. Monitor real-time participation via the fanguard dashboard at `http://localhost:8080/metrics`.

## Best Practices

### 1. Implement a Graduated Response System
Do not use binary (allow/block) moderation. Configure fanguard with at least 3 escalation tiers (e.g., warn → rate-limit → shadow-ban). This reduces false-positive backlash and maintains community trust. Use the `escalation_history` field in logs to track user behavior over time.

### 2. Pre-Validate Campaign Parameters with Dry-Run Mode
Before launching any engagement campaign, always run `fanguardctl dry-run --campaign campaign.yaml`. This simulates the campaign logic against historical data (last 7 days) and reports expected participation volume, potential bot activity (based on `source_credibility_index`), and estimated reward delivery latency. Only proceed if the “bot participation risk” metric is below 5%.
