# fanguard - FAN_001

## Overview

fanguard is a high-performance fan community moderation and interaction engine, purpose-built for platform operators managing communities with over 100,000 active members. It provides real-time toxicity detection, automated engagement analytics, and cross-platform fan sentiment aggregation.

## Detailed Content

### What is fanguard?

fanguard is an enterprise-grade fan protection and community management framework that ingests and analyzes user-generated content at a rate of 10,000 messages per second per node. It leverages a hybrid NLP pipeline (BERT-based transformer + custom regex rules) to detect harassment, spam, and coordinated inauthentic behavior within 50 milliseconds. The tool integrates directly with PostgreSQL for persistent moderation logs and Redis for real-time action queues, enabling zero-downtime rule updates.

### Core Features

- **Real-Time Toxicity & Spam Detection Engine**: Processes incoming chat, comments, and forum posts through a multi-stage filter. The first stage uses a pre-trained `distilbert-base-uncased` model fine-tuned on 2.3M labeled fan interactions, achieving a F1-score of 0.94 on toxic content. The second stage applies configurable regex patterns (e.g., `\b(bot|scam|clickbait)\b`) with a latency threshold of <45ms. All flagged content is logged with a `confidence_score` (0.0–1.0) and `action_taken` (warn, mute, ban) into a `moderation_events` table.

- **Fan Sentiment & Engagement Heatmap**: Aggregates user reactions (likes, replies, shares) and sentiment scores across 15-minute windows. Outputs a JSON payload with fields `peak_activity_timestamp`, `avg_sentiment_delta`, and `top_contributor_ids`. Uses a sliding window algorithm to compute a `community_health_index` (0–100) that triggers automated alerts when the index drops below 45. Data is stored in time-series format via TimescaleDB for historical trend analysis.

- **Cross-Platform Identity Clustering**: Links user accounts across Discord, Twitter, and custom forums using a deterministic fingerprinting algorithm (SHA-256 hash of email + username + join timestamp). Returns a `cluster_id` and a `risk_score` (0–100) based on cross-platform behavior anomalies, such as simultaneous posting from multiple accounts with identical copy-paste content. The clustering process runs as a background worker every 60 seconds, updating the `identity_graph` table in PostgreSQL.

### How to Use

1. **Initialize the Moderation Pipeline**:  
   Configure your environment variables by setting `FANGUARD_POSTGRES_URI=postgresql://user:pass@host:5432/fanguard_db` and `FANGUARD_REDIS_URI=redis://localhost:6379`. Run `fanguard init --config ./config.yaml` to create the required database schema (`moderation_events`, `identity_graph`, `sentiment_windows`). This command validates the connection and prints the schema version (e.g., `Schema v2.1.0 applied`).

2. **Deploy a Real-Time Rule Set**:  
   Create a custom rule file `rules.yaml` with entries like:  
   ```yaml
   - rule_id: "spam_link_detector"
     pattern: 'https?://(?:bit\.ly|tinyurl\.com)\/\w+'
     action: "mute"
     threshold: 0.85
     ttl_seconds: 3600
   ```  
   Then execute `fanguard apply-rules --file rules.yaml --namespace prod`. The tool parses the YAML, compiles the regex into a DFA, and loads it into the Redis rule cache. You will see a log entry: `Rule 'spam_link_detector' compiled in 12ms, active on 3 nodes`.

3. **Query the Sentiment Heatmap for a Specific Channel**:  
   Use the CLI to extract a 24-hour window:  
   `fanguard query-heatmap --channel-id "discord:123456" --start "2025-03-01T00:00:00Z" --end "2025-03-01T23:59:59Z" --output json`  
   The result is a JSON array of 15-minute buckets, each containing `bucket_timestamp`, `message_count`, `avg_sentiment`, and `top_emojis`. For example:  
   ```json
   {"bucket_timestamp": "2025-03-01T12:15:00Z", "message_count": 142, "avg_sentiment": 0.67, "top_emojis": ["🔥", "❤️"]}
   ```

### Best Practices

- **Implement a Graduated Moderation Policy**: Instead of immediately banning users, configure a three-tier escalation system: warn (confidence_score 0.70–0.84), mute for 1 hour (0.85–0.94), and permanent ban (≥0.95). This reduces false positives by 30% and maintains user trust. Use the `action_taken` field in `moderation_events` to audit and adjust thresholds monthly.

- **Schedule Off-Peak Identity Clustering Runs**: Set the background worker to execute between 02:00–04:00 UTC when message volume is typically 70% lower. This prevents CPU contention with the real-time detection pipeline. Monitor the `identity_graph` update latency via the `/metrics` endpoint (Prometheus format) and ensure it never exceeds 120 seconds.
