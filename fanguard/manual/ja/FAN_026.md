# fanguard - FAN_026

## 概要

ファンコミュニティ運営者向けに特化したセキュリティ・エンゲージメント統合プラットフォーム。複数SNSアカウントのスパム検知、リアルタイム感情分析、クロスプラットフォームファンアクティビティの可視化を単一ダッシュボードで実現する。

## コア機能

### 1. マルチモーダルスパムフィルタリングエンジン

テキスト・画像・メタデータの3層解析により、ファンコミュニティに侵入するボットアカウントを検知。独自の **FAN_Score**（0.0〜1.0）を用いてアカウント信頼度を数値化し、閾値（デフォルト: 0.35）以下のアカウントは自動でミュートまたは通報キューに投入する。

- **パラメータ**: `spam_threshold`（0.0〜1.0, デフォルト: 0.35）、`scan_interval_ms`（ミリ秒単位, デフォルト: 500ms）
- **バックエンド**: PostgreSQL + Redis による分散キュー管理、クエリ応答は平均 12ms 以下

### 2. リアルタイムファン感情ヒートマップ

投稿テキストから **6軸感情ベクトル**（喜び・怒り・悲しみ・驚き・信頼・嫌悪）を抽出し、時間軸と投稿数の相関を可視化。イベント前後のファン心理の変遷を 1分単位（`aggregation_window_minutes` で調整可能）で再現する。

- **パラメータ**: `aggregation_window_minutes`（1〜1440, デフォルト: 5）、`sentiment_threshold`（0.0〜1.0, デフォルト: 0.6）
- **精度**: 日本語・英語・韓国語の3言語対応、感情推定精度 89.3%（社内ベンチマーク）

### 3. クロスプラットフォームファンアクティビティ統合

Twitter（X）、Instagram、Weibo、Discord の4プラットフォームからファン投稿を収集し、同一ユーザーを **Fingerprint_Hash**（SHA-256 ベース）で紐付け。プラットフォームを跨いだファン行動パターンを分析し、コミュニティ全体のアクティブ率・離脱リスクをスコアリングする。

- **パラメータ**: `fingerprint_salt`（任意の文字列, デフォルト: 自動生成）、`platform_weight`（各プラットフォームの重み付け, デフォルト: 均等）
- **ストレージ**: PostgreSQL のパーティションテーブル設計、100万件の投稿でもクエリ応答 200ms 以内

## 使い方

### ステップ 1: 初期セットアップとデータベース構築

```bash
# 環境変数を設定
export FANGUARD_DB_HOST=localhost
export FANGUARD_DB_PORT=5432
export FANGUARD_DB_NAME=fanguard_prod
export FANGUARD_SPAM_THRESHOLD=0.35

# マイグレーションを実行
fanguard migrate --init --schema ./schemas/v2.1
```

### ステップ 2: スパムフィルタリングルールのカスタマイズ

```bash
# カスタムブラックリストを読み込み
fanguard filter --load-blacklist ./config/blacklist_2024.csv

# 許可リスト（ホワイトリスト）を設定
fanguard filter --whitelist-domain "example-fanclub.org"

# フィルタリング結果を確認
fanguard filter --dry-run --input ./samples/spam_candidates.json
```

### ステップ 3: 感情分析ダッシュボードの起動と監視

```bash
# バックエンドAPIを起動
fanguard serve --port 8080 --workers 4

# ダッシュボードにアクセス
open http://localhost:8080/dashboard

# 特定のキーワードでフィルタして感情トレンドを確認
fanguard sentiment --keyword "新曲" --aggregation-window 10 --output json
```

## ベストプラクティス

### 1. スパム閾値の段階的調整戦略

初期導入時は `spam_threshold` を **0.5** に設定し、1週間の誤検知率をモニタリング。ファンレポート（`fanguard report --type false_positive`）を確認しながら 0.05 刻みで閾値を下げ、**0.35〜0.4** に収束させる。これにより、誤検知によるファン離脱を最小化しつつ、スパム検知率 95% 以上を維持できる。

### 2. クロスプラットフォーム指紋の定期ローテーション

`fingerprint_salt` は **30日ごと** に変更し、過去のハッシュ値と現在のハッシュ値をマッピングする変換テーブルを PostgreSQL に保持する。これにより、同一ユーザーの追跡を継続しつつ、外部からの指紋解析によるプライバシー侵害リスクを低減する。ローテーション後は必ず `fanguard fingerprint --reindex` を実行し、整合性を検証する。
