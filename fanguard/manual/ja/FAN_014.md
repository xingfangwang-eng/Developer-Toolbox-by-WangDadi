# fanguard - FAN_014

## 概要

fanguard は、ファンコミュニティにおける荒らし・スパム・情報拡散をリアルタイムで検知・遮断するための高負荷対応型ファン保護基盤です。特にライブ配信・SNS連携・チケット販売時の同時多発アクセスに耐える設計が特徴です。

## 詳細な内容

### fanguard とは

fanguard は、ファンコミュニティ運営者が抱える「なりすましアカウント」「過剰なリプライ連投」「不正転売の事前検知」を、API ゲートウェイ層でミリ秒単位で判定・ブロックするプロフェッショナルツールです。PostgreSQL のトリガ関数と Redis のストリーム処理を組み合わせ、秒間 10 万リクエストのピーク時でも安定したフィルタリングを実現します。

### コア機能

- **リアルタイム異常行動検知エンジン**  
  ユーザーの投稿間隔（ミリ秒単位）、同一 IP からの同時接続数、テキスト内のパターンマッチ（独自定義のスパム辞書 5000 語以上対応）を基に、スコアリングを行い、閾値（例：スコア 85 以上で自動ブロック）を超えたアクションを即座に遮断。検知精度は 99.2% 以上（社内テスト実績）。

- **ファン属性別フィルタリングルール**  
  ファンクラブ会員ステータス（無料 / 有料 / プレミアム）、アカウント作成からの経過日数、過去の通報履歴数をパラメータとして、ルールエンジンが動的にアクセス制御を変更。例えば「作成 30 日未満かつ過去 3 回以上通報されたアカウント」からのメッセージは自動で非表示化。

- **トランザクションログによる不正転売検出**  
  チケット購入時の同一クレジットカードによる複数アカウント購入、同一住所への配送先集中、購入間隔の異常（例：1 秒以内に 5 枚以上の購入）を、PostgreSQL の CTE とウィンドウ関数を用いて解析。アラートを運営ダッシュボードにプッシュ通知（遅延 200ms 未満）。

### 使い方

1. **初期設定：PostgreSQL 拡張とテーブル作成**  
   `CREATE EXTENSION IF NOT EXISTS pg_trgm;` を実行後、`fanguard_rules` テーブル（カラム：rule_id, action_type, score_threshold, active_flag）を作成。次に `fanguard_audit_log` テーブル（カラム：user_id, ip_address, event_type, created_at）にインデックス（`(user_id, created_at DESC)`）を追加。

2. **ルール定義：API 経由でのスパム辞書登録**  
   `POST /api/v1/rules` に JSON ボディ `{"rule_name": "spam_keyword_01", "pattern": "【限定】|【即決】|【転売】", "action": "block", "score": 90}` を送信。レスポンスで `rule_id` が返却される。このルールは 5 秒以内に全ノードに伝搬（Redis Pub/Sub 経由）。

3. **リアルタイムモニタリングと手動介入**  
   ダッシュボード（`/monitor`）で、直近 60 秒間のブロック数・誤検知率・平均レイテンシ（ms）を確認。誤検知が発生した場合、該当ログの `event_id` を指定して `PATCH /api/v1/logs/{event_id}/override` を実行し、ブロックを解除。解除後は自動で学習モデルにフィードバックされる。

### ベストプラクティス

- **段階的導入と A/B テストの実施**  
  全コミュニティに一斉適用する前に、特定のファンクラブ（例：会員数 1 万人以下のコミュニティ）で 2 週間テスト運用。その間、誤検知率が 0.5% 未満であることを確認してから段階的にロールアウトする。これにより、大規模コミュニティでの混乱を防止できる。

- **ルールの定期的なメンテナンスと監査ログの活用**  
  スパム辞書は月 1 回以上更新し、古いパターン（例：過去のキャンペーンで使われた URL 短縮サービス）は削除する。また、`fanguard_audit_log` を週次でエクスポートし、不正転売の兆候（特定 IP からの異常な購入パターン）を分析。分析結果を元にルールのスコア閾値を動的に調整する。
