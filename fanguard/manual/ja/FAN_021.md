## 詳細な内容

### fanguard とは

fanguard は、SNS・掲示板・ライブ配信チャットなど複数プラットフォームに跨るファン発言を統合監視し、**ミリ秒単位の異常検知**と**自動アクション実行**を実現するエンタープライズグレードのガードエンジンです。内部では**PostgreSQL + Redis**によるイベントストリーム処理と、**BERTベースの日本語特化型NLPモデル**を搭載し、文脈に応じた誹謗中傷の判定を行います。

### コア機能

1. **マルチプラットフォーム統合フィルタリングエンジン**  
   - Twitter / YouTube Live / Discord / 独自フォーラムなど、各プラットフォームのAPI（WebSocket・REST）に直接接続。  
   - 設定可能な**レートリミット（例：1秒あたり100リクエスト）**を超えたスパム投稿を自動ブロック。  
   - **PostgreSQLの全文検索（pg_trgm）**を用いた過去ログとの類似度マッチングにより、同一ユーザーの使い回し荒らしを検出。

2. **リアルタイム感情スコアリング & アクション自動実行**  
   - 各投稿に対し**0.0（安全）～1.0（危険）の感情スコア**を算出。閾値（例：0.85以上）超過時、即座に投稿非表示・ユーザー警告・モデレーター通知をトリガー。  
   - **カスタムルールエンジン**搭載：特定キーワード（例：「死ね」「消えろ」）+ スコア条件を組み合わせた複合ルールをGUI上で定義可能。  
   - 全アクションは**監査ログとしてPostgreSQLに記録**され、後日レポート出力（CSV / JSON）に対応。

3. **ファンコミュニティ健全度ダッシュボード**  
   - 全プラットフォームの投稿数・警告数・ブロック数を**秒単位で集計**し、グラフ表示。  
   - **異常値アラート**：通常の投稿数の**3σ（標準偏差の3倍）**を超えた急増を検知し、Slack / Discord / メールへプッシュ通知。  
   - 期間指定（例：過去24時間 / 7日間）で「最も危険な発言TOP50」を自動生成。

### 使い方

1. **初期セットアップ：プラットフォーム接続と基本閾値設定**  
   - 管理画面「接続設定」から、監視対象プラットフォームのAPIキー / WebSocket URLを登録。  
   - 各プラットフォームごとに**レートリミット（例：100 req/s）**と**感情スコアの警告閾値（例：0.75）**、**ブロック閾値（例：0.90）**を入力。  
   - 「テスト接続」ボタンで疎通確認後、**保存**をクリック。

2. **カスタムルールの作成と優先度設定**  
   - 「ルール管理」タブで新規ルール作成。  
   - 条件例：`キーワード「死ね」を含む AND 感情スコア >= 0.8` → アクション「投稿非表示 + ユーザー警告メッセージ送信」。  
   - ルールには**優先度（1～100）**を設定可能。数値が小さいほど先に評価される。  
   - 複数ルールがマッチした場合、**最も高い優先度のアクション**を実行。

3. **ダッシュボード監視とアラート設定**  
   - トップページの「ライブビュー」で全プラットフォームのリアルタイム投稿ストリームを確認。  
   - 「アラート設定」で、**3σ異常検知**の通知先（Slack Webhook URL / Discord Webhook URL / メールアドレス）を登録。  
   - 通知内容はカスタマイズ可能（例：`${platform} で異常検知: 投稿数 ${count}（通常比 ${ratio}倍）`）。  
   - 設定後「保存して有効化」をクリック。

### ベストプラクティス

- **段階的な閾値調整**  
  初期は感情スコアのブロック閾値を**0.95**に設定し、誤検知が少ないことを確認した上で徐々に**0.85**まで下げる。これにより、過剰ブロックによるファンの離脱を防止しつつ、実害のある投稿のみを確実に遮断できる。

- **定期的なルール監査とログ分析**  
  週に一度、**ブロックされた投稿のサンプル（100件程度）**をランダム抽出し、人間のモデレーターが目視確認。誤検知が5%を超える場合はルールの優先度やキーワードリストを見直す。また、**監査ログをPostgreSQLに保存**しているため、SQLクエリで「特定ユーザーの警告回数」や「時間帯別ブロック数」を分析し、ルール改善に活用する。
