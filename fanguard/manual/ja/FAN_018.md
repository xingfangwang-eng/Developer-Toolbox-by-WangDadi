# fanguard - FAN_018

## 概要

fanguardは、大規模ファンコミュニティにおける悪質アカウントの自動検知とリアルタイムモデレーションを実現する、エンタープライズ向けファン保護基盤です。ファン同士の健全な相互作用を担保しつつ、プラットフォーム全体のトラストスコアを動的最適化します。

## 詳細な内容

### fanguardとは何か

fanguardは、ファンコミュニティ運用者向けに設計されたAI駆動型インシデント対応システムです。従来のキーワードフィルタでは対応困難な文脈依存型スパム、なりすましアカウント、組織的な荒らし行動を、マルチモーダル分析とグラフベースの行動連鎖追跡により検出・排除します。全処理はミリ秒単位で完了し、PostgreSQL上の時系列イベントログと連携した証拠保全が可能です。

### コア機能

- **マルチモーダル異常行動検出エンジン**  
  テキスト・画像・メタデータの3モーダルを同時解析。各モーダルから抽出した特徴ベクトルをアテンション機構で統合し、閾値`θ_anomaly = 0.78`を超えたアカウントを自動隔離。隔離後は`POST /api/v1/accounts/{id}/quarantine`で手動レビュー可能。

- **グラフベースの共謀クラスタ検出**  
  フォロー・メンション・同一IPからの同時ログインをエッジとする有向グラフを構築。Louvain法によるコミュニティ検出と、ページランクスコア`PR_threshold = 0.15`を組み合わせ、ボットネットワークを特定。検出結果は`/api/v1/clusters/export?format=csv`でCSV出力。

- **リアルタイムトラストスコアリングAPI**  
  各アカウントに動的スコア`S(t) = α * engagement_rate + β * report_ratio - γ * action_frequency`を計算。`α=0.4, β=0.35, γ=0.25`のデフォルト値は環境変数`TRUST_ALPHA, TRUST_BETA, TRUST_GAMMA`で上書き可能。スコアが`S(t) < 30`を3回連続で下回ったアカウントは自動ミュート。

### 使い方

1. **初期設定: データソースの接続**  
   `config/database.yml`内の`fan_community_db`セクションに、PostgreSQL 14以上の接続情報（ホスト、ポート5432、データベース名`fanguard_prod`）を記述。その後`./fanguard init --schema`を実行し、`fan_events`テーブルと`account_graph`テーブルを自動生成。

2. **モデレーションポリシーの定義**  
   `policies/`ディレクトリ下にYAMLファイルを作成。例として`strict_mode.yaml`内で`anomaly_threshold: 0.78, cluster_min_size: 5, trust_score_floor: 20`を設定。ポリシー適用は`./fanguard apply --policy strict_mode.yaml --scope all`。

3. **モニタリングダッシュボードの起動**  
   `./fanguard dashboard --port 8080 --metrics-interval 500`でWeb UIを起動。リアルタイムのイベントストリーム、隔離アカウント一覧、トラストスコア分布ヒストグラムを確認可能。`/api/v1/metrics/prometheus`エンドポイントでPrometheus連携もサポート。

### ベストプラクティス

- **段階的ポリシーロールアウト**  
  全アカウントへの一括適用は避け、まず全ユーザーの5%を対象に`canary`スコープでポリシーをテスト。`./fanguard apply --policy moderate_mode.yaml --scope canary`で実行し、誤検知率が0.5%未満であることを確認してから全範囲に拡大する。

- **監査ログの長期保存戦略**  
  隔離・ミュート・BANの全操作は`fan_events`テーブルに記録される。`pg_partman`を用いて月次パーティショニングを設定し、`retention_period = 365`日経過したパーティションはAmazon S3に自動アーカイブ。アーカイブ後も`/api/v1/audit/search?from=2024-01-01&to=2024-12-31`で検索可能。
