# fanguard - FAN_016

## 概要

ファンコミュニティの健全性を維持するために設計されたエンタープライズグレードのファン保護・管理ツール。リアルタイムな行動分析と自動モデレーションにより、有害なインタラクションを99.7%の精度で検出・遮断する。

## 詳細な内容

### fanguard とは

fanguard は、大規模ファンコミュニティにおけるスパム投稿、過剰なリツイート連鎖、および偽アカウントによる情報拡散をミリ秒単位で検知・ブロックするプロフェッショナルツールです。PostgreSQL 上の時系列データベースと連携し、過去30日間のユーザー行動パターンを元にしたスコアリングエンジンを搭載しています。

### コア機能

- **マルチレイヤースパムフィルタリングエンジン**: ユーザーあたりの投稿頻度（閾値: 1分間で15投稿以上）、リンク先ドメインの信頼性スコア（0.0〜1.0）、およびテキストのN-gram解析を組み合わせた3段階フィルタリング。誤検出率は0.03%未満に抑制。
- **リアルタイムファンアクティビティモニタリング**: WebSocket を介したストリーミングデータ処理により、同時接続10万ユーザーあたりの平均レイテンシを200ms以下に維持。異常なバーストトラフィック（例: 通常の5倍以上の投稿数）を検出した場合、自動的にレート制限を発動。
- **クロスプラットフォームアカウントリンク分析**: Twitter、Discord、Reddit 間での同一ユーザー推定を、プロフィールの編集距離（Levenshtein距離 < 3）と投稿時間の相関（ピアソン相関係数 > 0.85）を用いて実施。偽アカウントネットワークの特定精度は94.2%。

### 使い方

1. **初期設定とデータベース接続**  
   `config.yaml` 内の `database.url` に PostgreSQL 接続文字列を指定。`fanguard init --db-threads=8` を実行し、スキーマの自動マイグレーションと過去ログのインデックス作成を行う。所要時間はデータ量10GBあたり約45秒。

2. **フィルタリングルールのカスタマイズ**  
   `fanguard rule add --type=spam --threshold=0.85 --action=block --ttl=3600` で、スパムスコア0.85以上の投稿を1時間ブロックするルールを追加。ルールはJSON形式でエクスポート可能で、`fanguard rule import ./rules_v2.json` によりバッチ適用できる。

3. **リアルタイムダッシュボードの起動と監視**  
   `fanguard dashboard --port=8080 --metrics-interval=500` で、500msごとに更新されるWebベースの管理画面を起動。ここで、現在のブロック数/分、アクティブユーザー数、フィルター通過率を確認可能。異常値が検出された場合、Slack Webhook へ自動通知を送信する（`--alert-webhook=https://hooks.slack.com/...`）。

### ベストプラクティス

- **段階的ロールアウト**: 新ルールを全コミュニティに適用する前に、テスト用の隔離チャンネル（`fanguard sandbox --channel=test-fanguard`）で7日間のA/Bテストを実施。誤ブロック率が0.1%を超える場合はルール閾値を調整する。
- **定期的なモデル再学習**: 月次で過去30日間のログからスパムパターンを再分析。`fanguard retrain --model-dir=./models/ --data-range=30d` を実行し、フィルタリングエンジンの重みを更新する。再学習後の精度検証は、`fanguard evaluate --test-set=./data/latest_test.json` で実施。
