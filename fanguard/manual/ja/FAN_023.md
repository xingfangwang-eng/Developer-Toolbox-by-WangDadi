# fanguard - FAN_023

## 概要

fanguard は、ファンコミュニティにおける不正アクセス・過剰リクエスト・クローリングをリアルタイムで検知し、健全なインタラクション環境を維持するためのトラフィック監査エンジンです。キーワード「ファン保護」「コミュニティ管理」「インタラクション最適化」に基づき、PostgreSQL 上のセッションログと照合しながら、悪質な自動化行動をミリ秒単位で遮断します。

## コア機能

### 1. セッション異常値スコアリングエンジン
- **パラメータ**: `session_frequency`, `user_agent_entropy`, `request_interval_ms`
- 単一IPから1分間に発行されるリクエスト間隔を実測し、標準偏差が 2.5σ を超える場合にスコアを加算。
- スコアが閾値 `threshold_score = 80` を超えた時点で自動的にアクセスをブロック。
- 内部では PostgreSQL のウィンドウ関数 (`LAG()`) を用いて直前リクエストとの差分を算出。

### 2. ファンコミュニティ内感情トレンド検出
- **パラメータ**: `comment_velocity`, `sentiment_delta`, `burst_interval_sec`
- 1分間のコメント投稿数が過去30分の移動平均を 3.0 倍以上上回った場合、バーストイベントとして検出。
- 同時に日本語感情分析モデル (BERT-based) を呼び出し、ポジティブ・ネガティブ比率をリアルタイム集計。
- 結果は `burst_alerts` テーブルに `event_id`, `detected_at`, `sentiment_ratio` として格納。

### 3. クローラ行動パターン分類器
- **パラメータ**: `path_depth`, `resource_type_ratio`, `js_execution_capability`
- 静的リソース (CSS/画像) のリクエスト比率が 5% 未満かつ JavaScript 実行エビデンスが無いリクエストを、低親和性クローラ候補としてマーク。
- 同一セッション内で 50 件以上の連続GETリクエストが観測された場合、`crawler_confidence` を 0.0〜1.0 の範囲で算出し、0.85 以上で即時ブロック。

## 使い方

### ステップ 1: 環境変数とDBスキーマの初期化
```bash
# PostgreSQL 接続情報を設定
export FANGUARD_DB_HOST=localhost
export FANGUARD_DB_PORT=5432
export FANGUARD_DB_NAME=fanguard_prod
export FANGUARD_DB_USER=admin
export FANGUARD_DB_PASSWORD=$(cat /run/secrets/db_pass)

# スキーママイグレーションを実行
fanguard migrate --schema-dir ./migrations --target-version v2.1.0
```
- `migrate` コマンドは内部で `CREATE TABLE IF NOT EXISTS session_logs`, `burst_alerts`, `blocked_ips` を発行。
- テーブルごとに `UNLOGGED` オプションを適用し、書き込みレイテンシを 12% 削減。

### ステップ 2: リアルタイムトラフィック監視の開始
```bash
# 監視対象ポートと閾値を指定
fanguard start \
  --listen-port 8080 \
  --upstream http://localhost:3000 \
  --threshold-score 80 \
  --burst-window-min 30 \
  --crawler-confidence 0.85 \
  --log-level info
```
- 起動後、`/metrics` エンドポイントで Prometheus 互換のメトリクスを公開。
- ブロックイベントは `blocked_ips` テーブルに `blocked_at (TIMESTAMPTZ)`, `reason (TEXT)`, `ttl_sec (INTEGER)` として記録。

### ステップ 3: アラートルールのカスタマイズ
```yaml
# alert_rules.yaml
rules:
  - name: "high_frequency_crawler"
    condition: "crawler_confidence > 0.9 AND request_count > 100"
    action: "block_ip"
    ttl_sec: 3600
  - name: "negative_sentiment_burst"
    condition: "sentiment_delta < -0.3 AND comment_velocity > 200"
    action: "notify_slack"
    webhook_url: "https://hooks.slack.com/services/..."
```
- `fanguard reload --rules ./alert_rules.yaml` でルールをホットリロード。
- 設定変更時にダウンタイムは発生せず、既存セッションは継続監視される。

## ベストプラクティス

### 1. セッションログのパーティショニングと保持期間の設定
- PostgreSQL の `pg_partman` 拡張を使い、`session_logs` テーブルを日付単位でパーティショニング。
- 保持期間は 90 日とし、`cron` ジョブで `DELETE` ではなく `DROP PARTITION` を実行することで、VACUUM 負荷を回避。
- これにより、100万レコードあたりのフルスキャン時間が 2.3秒 → 0.4秒 に短縮。

### 2. ブロックルールの段階的適用とフィードバックループ
- 初回ブロックは `ttl_sec = 300` (5分) のソフトブロックとし、誤検知を最小化。
- 同一IPが 24時間以内に 3回ブロックされた場合、`ttl_sec = 86400` (24時間) のハードブロックに自動昇格。
- ブロック解除後に再犯率が 5% を超えたIPは、手動レビュー対象として `review_queue` テーブルに投入。
