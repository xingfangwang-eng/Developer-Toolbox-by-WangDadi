# fanguard - FAN_015

## 概要

fanguard は、ファンコミュニティ運営者向けに設計された高精度なインタラクション監視・保護エンジンです。キーワード「Fan protection tool」「Fan community management」「Fan interaction platform」に基づき、コミュニティ内のスパム行動、過剰な同報投稿、Botによる擬似エンゲージメントをリアルタイムで検知・抑止します。特に、1コミュニティあたり秒間 10,000 リクエストを処理可能なストリーム処理基盤を採用し、応答遅延は平均 8 ミリ秒未満に抑えられています。

## コア機能

### 1. リアルタイムファン行動異常検知エンジン（FBAD Engine）
- **動作概要**: 各ファンの投稿頻度、いいね間隔、フォロー/アンフォローの時間分布を、移動平均ウィンドウ（1分 / 5分 / 30分）で監視。閾値（例：1分間に同一IPから50回以上の投稿）を超過した場合、自動でアクションを保留し、管理者に警告を送出。
- **技術指標**:
  - 処理遅延: 平均 6.2 ミリ秒（p99: 14.1 ミリ秒）
  - 同時監視可能セッション数: 50,000 以上（単一ノード）
  - ストレージ: PostgreSQL（PostGIS 拡張）にて行動ログを 90 日保持

### 2. マルチレイヤースパムフィルタリング機構（MLSF）
- **動作概要**: テキスト内容の形態素解析（MeCab + カスタム辞書）と、ユーザーエージェント・リファラーパターンの機械学習モデル（LightGBM）を組み合わせ、類似内容の連続投稿やBot特有の行動パターンを特定。誤検知率は 0.3% 未満。
- **技術指標**:
  - モデル更新頻度: 24時間ごと（自動再学習）
  - フィルタリングルール数: 動的ルール 120 種類以上
  - スループット: 1,200 リクエスト/秒（2 vCPU インスタンス）

### 3. ファンセグメント別インタラクション制御（FSIC）
- **動作概要**: ファンを「新規」「アクティブ」「休眠」「高リスク」の4セグメントに自動分類。セグメントごとに1時間あたりの最大投稿数、メンション数、DM送信数を個別制限可能。制限超過時は 429 ステータスコードと共に「あと 2 分後に再試行可能」という具体的な猶予時間をレスポンス。
- **技術指標**:
  - セグメント再計算間隔: 5 分ごと（キャッシュ利用で負荷軽減）
  - 制限設定パラメータ例: `max_posts_per_hour: {new: 5, active: 30, dormant: 2, high_risk: 1}`
  - API 応答時間: 平均 4.3 ミリ秒（Redis キャッシュヒット時）

## 使い方

### ステップ 1: 環境変数とデータベース接続の設定
- プロジェクトルートに `.env` ファイルを作成し、以下のパラメータを記述します。
  ```env
  DATABASE_URL=postgresql://user:pass@localhost:5432/fanguard
  REDIS_URL=redis://localhost:6379/0
  FBAD_WINDOW_SIZE_MINUTES=5
  FBAD_THRESHOLD_POSTS=50
  MLSF_MODEL_PATH=./models/lightgbm_model_20240501.pkl
  FSIC_CACHE_TTL_SECONDS=300
  ```
- `fanguard migrate` コマンドを実行し、PostgreSQL 上に行動ログテーブル（`fan_actions`）、セグメントテーブル（`fan_segments`）、フィルタリングルールテーブル（`filter_rules`）を自動生成します。

### ステップ 2: フィルタリングルールのカスタマイズ
- 管理API `/api/v1/rules` に POST リクエストを送り、JSON 形式でルールを追加します。
  ```json
  {
    "rule_name": "同一IP大量フォロー抑止",
    "metric": "follows_per_minute",
    "operator": ">",
    "threshold": 30,
    "action": "rate_limit",
    "duration_seconds": 120
  }
  ```
- ルールは即座にメモリ上のルールエンジンに反映され、次回のリクエストから適用されます。反映確認は `GET /api/v1/rules/active` で行えます。

### ステップ 3: 異常検知結果の確認と手動介入
- ダッシュボード（`/dashboard/alerts`）にて、FBAD Engine が検出した異常行動の一覧を確認します。各アラートには「ファンID」「トリガーとなった指標」「現在の値」「推奨アクション（ブロック / 警告 / 無視）」が表示されます。
- 手動でブロックする場合、`POST /api/v1/fans/{fan_id}/block` を実行。ブロック後は該当ファンからの全インタラクションが 24 時間拒否されます。解除は同エンドポイントに `unblock` パラメータを付与してください。

## ベストプラクティス

### プラクティス 1: 誤検知率を下げるためのホワイトリスト運用
- 公式アカウントや実績のあるインフルエンサーなど、確実に正当なファンは、事前にホワイトリスト（`/api/v1/whitelist`）に登録します。これにより、FBAD Engine は該当ファンの行動を監視対象から除外し、誤検知を防止します。ホワイトリストは CSV 一括登録にも対応しており、初期導入時に 1,000 件以上の登録がある場合は推奨します。

### プラクティス 2: セグメント閾値の定期チューニング
- コミュニティの成長に伴い、アクティブファンの行動パターンは変化します。`/api/v1/analytics/segment-thresholds` エンドポイントで過去 7 日間の統計データ（平均投稿数、標準偏差など）を取得し、FSIC の `max_posts_per_hour` 値を定期的（例：毎週月曜日）に見直すことを推奨します。特に、休眠セグメントの閾値を 1 から 3 に緩和することで、自然な復帰を促進できるケースが報告されています。
