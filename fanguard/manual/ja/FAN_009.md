# fanguard - FAN_009

## 概要

fanguardは、ファンコミュニティの健全性維持と悪質なファン行動の検知・防御に特化したプロフェッショナル向けツールです。ファン同士の相互作用をリアルタイムで解析し、トキシックな言動の抑制とエンゲージメントの最適化を実現します。

## コア機能

### 1. ファン行動異常検知エンジン（FADE）
- **リアルタイムテキスト解析**: ファン発言の感情極性（ポジティブ/ネガティブ）を`ミリ秒単位`でスコアリング。閾値（デフォルト: `0.75`）を超えるネガティブスパイクを検出し、自動モデレーションキューに投入。
- **行動パターン認識**: 同一IPからの短時間連続投稿（`3秒以内に5回以上`）をスパムとしてフラグ。`PostgreSQL`上の行動履歴テーブルと照合し、過去30日間の違反回数から永続ブロック判定を実施。
- **ファンダム健全性指数（FHI）**: コミュニティ全体の健全性を`0.0〜1.0`で数値化。`FHI < 0.3`の場合、管理者に`Slack Webhook`経由で即時アラート送信。

### 2. ファンセグメント別インタラクション最適化（FSIO）
- **属性ベースの動的ルーティング**: ファンを`新規参入者`・`コアファン`・`インフルエンサー`の3セグメントに自動分類。各セグメントに最適化されたコンテンツ（例：新規参入者にはFAQチャットボット、コアファンには限定コンテンツアクセス）を`API Gateway`経由で配信。
- **エンゲージメントスコアリング**: ファンのアクション（いいね、リツイート、コメント）に重み付け（`いいね=1pt`、`コメント=3pt`、`シェア=5pt`）を行い、`Redis`上でリアルタイム集計。スコアが`100pt`を超えたファンをVIPとしてマーク。
- **クールダウン機構**: 同一ファンからの連続インタラクションに`30秒`のクールダウンタイマーを設定。過剰なAPIコールを抑制し、サーバー負荷を`最大40%`削減。

### 3. マルチプラットフォームファンデータ連携（MPFD）
- **クロスプラットフォームID解決**: Twitter、Discord、公式フォーラムのアカウントを`SHA-256`ハッシュで紐付け。同一ファンの行動を横断的に追跡し、`PostgreSQL`の`fan_crosswalk`テーブルに保存。
- **データパイプライン**: `Apache Kafka`を介して各プラットフォームからファンイベントをストリーム処理。`1日あたり100万イベント`の取り込みを想定し、`バッチ処理間隔=5分`で集計。
- **エクスポート機能**: ファン分析結果を`CSV`・`JSON`形式でエクスポート。`gzip`圧縮を適用し、`10万レコード`のエクスポートを`1.2秒`で完了。

## 使い方

### ステップ1: 初期設定とファンチャンネル登録
1. `fanguard config init`を実行し、`config.yaml`に`platforms:`セクションでTwitter APIキー、Discord Botトークン、フォーラムRSSフィードURLを設定。
2. `fanguard channel add --type twitter --query @your_fanbase`で監視対象チャンネルを登録。`--rate-limit 100`でAPIレート制限を明示。
3. `fanguard health-check`で全チャンネルの接続状態を確認。`Status: OK`が返るまで`--retry 3`でリトライ。

### ステップ2: 異常検知ルールのカスタマイズ
1. `fanguard rule create --name toxic_language --threshold 0.85`でカスタムルールを作成。`--action warn`で警告のみ、`--action block`で即時ブロックを指定。
2. `fanguard rule test --input sample_fan_comment.txt`でルールの精度を検証。`Precision: 0.92`、`Recall: 0.88`のメトリクスを確認。
3. `fanguard rule deploy --environment production`でルールを本番適用。`Rollback`用に前バージョンが自動保存される。

### ステップ3: ファンセグメント分析の実行
1. `fanguard segment analyze --from 2025-01-01 --to 2025-01-31`で過去1ヶ月のファン行動を分析。結果は`segment_report_202501.json`に出力。
2. `fanguard segment visualize --type pie`でセグメント比率を円グラフで表示。`Core: 23%`、`Newcomer: 58%`、`Influencer: 19%`の分布を確認。
3. `fanguard segment export --format csv --output fan_segments.csv`で分析結果をエクスポート。`column: fan_id, segment, engagement_score`のスキーマで出力。

## ベストプラクティス

### 1. 段階的モデレーションの導入
最初から厳格なブロックルールを適用せず、`warn`→`mute`→`block`の3段階でペナルティを段階的に強化する。`FADE`の異常検知スコアが`0.6〜0.8`の場合は警告、`0.8〜0.95`の場合はミュート、`0.95以上`の場合のみブロックとすることで、誤検知によるファン離脱を最小化する。実運用では、`誤検知率 < 0.5%`を目標に設定する。

### 2. ファンセグメント別のエンゲージメント戦略
`FSIO`で分類されたセグメントごとに異なるコミュニケーション戦略を立てる。新規参入者には`週1回`のウェルカムメールと初心者向けFAQ、コアファンには`月1回`の限定コンテンツアクセス、インフルエンサーには`隔週`のフィードバックミーティングを自動スケジュールする。`エンゲージメントスコア`の推移を`週次`でモニタリングし、`スコア上昇率 < 5%`のセグメントには戦略の見直しを検討する。
