# fanguard - FAN_027

## 概要

fanguard は、ファンコミュニティにおける誹謗中傷・スパム・著作権侵害をリアルタイムで検知・排除するためのエンタープライズグレードのファン保護ツールです。本プロジェクトは、大規模ファンコミュニティ運営者向けに設計され、**Fan protection tool**, **Fan community management**, **Fan interaction platform** の3軸を統合した唯一無二のソリューションを提供します。

## 詳細な内容

### fanguard とは

fanguard は、ファンコミュニティ内の有害投稿を**ミリ秒単位**で検出し、自動的にブロック・報告するプロフェッショナルツールです。特に、**ファン同士の健全な交流**を維持するために、感情分析エンジンと画像認識モデルを組み合わせた独自のマルチモーダルフィルタリング機構を搭載しています。また、PostgreSQL との連携により、全検知ログを構造化データとして保存し、後日の監査や分析に即座に利用可能です。

### コア機能

1. **感情分析＋スパムスコアリングエンジン**
   - 投稿テキストをリアルタイムで解析し、**ネガティブ感情スコア（0.0〜1.0）** と **スパム確率（0〜100%）** を独立算出。両スコアの加重平均（デフォルト閾値: 0.75）を超えた投稿は即座に非表示化。
   - パラメータ: `--negativity-threshold 0.75`、`--spam-probability-threshold 80`、`--batch-size 256`
   - 対応言語: ja, en, zh, ko（追加言語はモデル再学習で対応可能）

2. **画像ベースの著作権侵害検出（Fingerprint Matching）**
   - アップロードされた画像の**知覚ハッシュ（pHash, dHash）** を生成し、事前に登録された著作権保護画像のハッシュデータベースと照合。類似度が **95%以上** の場合は自動削除＋報告。
   - パラメータ: `--hash-algorithm phash`、`--similarity-threshold 0.95`、`--max-db-size 1000000`
   - ストレージ: PostgreSQL の `BYTEA` 型にハッシュ値を格納、インデックス付きで高速検索（平均応答時間: **12ms**）

3. **ファンインタラクション健全性スコアリング（コミュニティヘルスメトリクス）**
   - 各ユーザーの投稿頻度、リアクション数、通報履歴から **健全性スコア（0〜100）** を算出。スコアが **30以下** のユーザーは自動的にモデレーションキューに追加され、管理者による確認が可能。
   - パラメータ: `--health-score-threshold 30`、`--observation-window 7d`、`--weight-report 0.4`
   - データ保持: PostgreSQL に日次集計テーブルを作成し、過去365日分の履歴を保持

### 使い方

1. **初期セットアップとデータベース初期化**
   ```bash
   git clone https://github.com/xingfangwang-eng/fanguard.git
   cd fanguard
   psql -U admin -d fanguard_db -f schema/init.sql
   ```
   上記コマンドで、fanguard 専用の PostgreSQL データベース（`fanguard_db`）と、ハッシュテーブル・ログテーブル・ユーザースコアテーブルが自動生成されます。`schema/init.sql` には、**BRINインデックス** と **パーティショニング** が定義されており、大規模データでも劣化しません。

2. **有害投稿の自動検出とブロック**
   ```bash
   fanguard scan --input-dir /var/log/community_posts --output-dir /var/log/fanguard_reports --negativity-threshold 0.75 --spam-probability-threshold 80 --batch-size 256
   ```
   `scan` サブコマンドは、指定ディレクトリ内の全投稿（JSON Lines 形式）を読み込み、感情分析とスパム判定を実行。閾値を超えた投稿は `output-dir` に **ブロックリスト** として出力され、同時に PostgreSQL の `blocked_posts` テーブルに INSERT されます。処理速度は **約2,000 posts/sec**（Intel Xeon E5-2680 v4 相当）。

3. **著作権侵害画像の登録と検出**
   ```bash
   fanguard register --image-dir /path/to/copyright_images --hash-algorithm phash
   fanguard detect --image-path /path/to/suspicious_image.jpg --similarity-threshold 0.95
   ```
   `register` で事前に著作権画像のハッシュをデータベースに登録。`detect` で任意の画像をチェックし、類似度が閾値を超えた場合、標準出力に **「MATCH: 類似度97.3% / 登録ID: cp_0042」** と表示されます。検出結果は自動的に `violation_log` テーブルに記録されます。

### ベストプラクティス

1. **段階的な閾値調整による誤検出の最小化**
   - 導入初期は `--negativity-threshold 0.85`、`--spam-probability-threshold 90` と高めに設定し、1週間分の検出結果をレビュー。その後、誤検出率が **5%未満** であることを確認してから、徐々に閾値を下げて運用します。これにより、初期のユーザー離脱を防ぎつつ、徐々に精度を高められます。

2. **PostgreSQL の定期的なバキュームとアーカイブ戦略**
   - `blocked_posts` テーブルと `violation_log` テーブルは、日次で **VACUUM ANALYZE** を実行し、インデックスの断片化を防止。また、**90日以上経過したログ** は `archive_schema` に移動し、本番テーブルからは削除することで、クエリパフォーマンスを維持します。推奨 cron ジョブ: `0 3 * * * psql -U admin -d fanguard_db -c "VACUUM ANALYZE;" && psql -U admin -d fanguard_db -c "CALL archive_old_logs(90);"`
