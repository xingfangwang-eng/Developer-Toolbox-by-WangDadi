# fanguard - FAN_005

## 概要

fanguard は、ファンコミュニティにおける悪質なスパム・荒らし行為をリアルタイム検知し、管理者の負担を極限まで削減するプロフェッショナル向けファン保護基盤です。キーワード「Fan protection tool」「Fan community management」「Fan interaction platform」に基づき、プラットフォーム横断でのトラフィク解析・行動スコアリングを実現します。

## 詳細な内容

### fanguard とは

fanguard は、Discord、Twitter、独自フォーラムなど複数プラットフォームから収集したユーザー行動ログを統合し、**ミリ秒単位の異常検知エンジン**と**PostgreSQLベースの永続化分析基盤**により、ファンコミュニティの健全性を自動維持する次世代型管理ツールです。従来のキーワードフィルタでは捉えられない、**文脈依存型スパム**や**同一IPによる多重アカウント攻撃**を高精度でブロックします。

### コア機能

- **マルチプラットフォーム異常行動スコアリングエンジン**  
  各プラットフォームのAPIから取得した投稿頻度（件/秒）、同一IPセッション数、テキスト類似度（Levenshtein距離 < 0.3）を統合し、0〜100のスコアをリアルタイム算出。閾値（デフォルト: 75）超過時に自動BANまたはモデレーター通知を発報。

- **PostgreSQL連携による長期トレンド分析ダッシュボード**  
  過去365日分の行動ログを時系列テーブルに格納。週次・月次での異常行動パターンの可視化（例: 特定ハッシュタグに紐づくスパム急増検知）を提供。クエリ応答は平均200ms以内。

- **カスタムルールエンジン（YAMLベース）**  
  管理者が任意のルールセットを定義可能。例: `rate_limit: { max_posts_per_minute: 5, action: mute }`。ルールの適用はサーバー再起動不要でホットリロード対応。

### 使い方

1. **初期設定とデータベースマイグレーション**  
   ```bash
   git clone https://github.com/xingfangwang-eng/fanguard.git
   cd fanguard
   cp config.example.yml config.yml
   # PostgreSQL接続情報（host, port, dbname, user, password）を編集
   psql -U postgres -d fanguard -f schema.sql
   ```
   これにより、`user_actions`、`ip_sessions`、`rule_definitions`の3テーブルが作成され、リアルタイム集計用のマテリアライズドビューが自動構築されます。

2. **異常検知エンジンの起動とスコアリング検証**  
   ```bash
   ./fanguard --mode daemon --port 8080 --log-level debug
   # テスト用にダミーイベントを送信
   curl -X POST http://localhost:8080/api/v1/event \
     -H "Content-Type: application/json" \
     -d '{"platform":"discord","user_id":"123","ip":"192.168.1.1","message":"spam text"}'
   ```
   レスポンス内の `score` フィールド（例: 82）が閾値を超えた場合、即座に `action` フィールドで `ban` が返ります。

3. **カスタムルールの適用とホットリロード**  
   ```yaml
   # rules/anti_raid.yml
   rules:
     - name: "同一IP多重アカウント検知"
       condition: "ip_session_count > 3 AND time_window_sec < 60"
       action: "block"
       notification: true
   ```
   ファイル保存後、`kill -HUP $(pgrep fanguard)` でルールを再読み込み。設定反映のダウンタイムはゼロです。

### ベストプラクティス

- **初期スコア閾値は70から開始し、1週間の観測後に調整**  
  過剰検知を防ぐため、まずは`threshold: 70`で運用。異常行動の誤検知率が5%を超える場合は閾値を75以上に引き上げ、逆に検知漏れが多い場合は65に下げてチューニングします。

- **PostgreSQLのパーティショニングを活用し、クエリ性能を担保**  
  `user_actions`テーブルは日付単位でパーティション分割（例: `user_actions_2025_03`）。これにより、月次レポート生成時のフルテーブルスキャンを回避し、応答時間を常に200ms以下に維持します。
