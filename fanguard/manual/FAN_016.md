# fanguard - FAN_016

## Overview

fanguard is a high-performance, real-time fan community moderation and engagement analytics engine designed for platforms managing over 100,000 concurrent active users. It combines NLP-driven toxicity detection with behavioral pattern recognition to protect brand integrity and foster healthy interaction ecosystems.

## Detailed Content

### What is fanguard?

fanguard is a distributed, event-driven fan protection and community management infrastructure. It ingests WebSocket streams, REST API payloads, and database change data capture (CDC) logs to compute a **Fan Trust Score (FTS)** per user in sub-50ms latency. It detects coordinated spam campaigns, toxic sentiment spikes, and anomalous engagement patterns before they impact the community.

### Core Features

- **Real-Time Toxicity & Spam Detection Engine**  
  Deploys a fine-tuned `roberta-base` transformer model with a custom attention head for multi-language profanity, hate speech, and spam detection. Processes 10,000+ messages/second per node with a precision rate of 97.3% at recall 0.92. Exposes a `/v1/classify` endpoint accepting `{text: string, language: string, context_id: string}` and returns a `toxicity_score` (0.0–1.0) with a `spam_confidence` (0.0–1.0) and a `reason_code` (e.g., `TOXIC_HARASSMENT`, `SPAM_LINK_DROP`).

- **Fan Trust Score (FTS) & Behavioral Graph Analytics**  
  Maintains a per-user `FTS` (0–1000) updated in near real-time via a Neo4j graph database. The score is a weighted composite of: `message_volume_z_score`, `report_count`, `positive_reaction_rate`, `account_age_days`, and `cross_channel_behavior_entropy`. Triggers automatic moderation actions (mute, shadowban, ban) when FTS drops below configurable thresholds (default: < 200). Supports historical trend queries via `/v2/user/{user_id}/trust_timeline?range=7d`.

- **Automated Incident Response & Workflow Orchestration**  
  Integrates with external moderation queues (e.g., Zendesk, custom webhooks) via a rule engine that evaluates `{event_type, user_fts, toxicity_score, activity_frequency}`. Supports chained actions: e.g., `toxicity_score > 0.9 AND FTS < 300` triggers an automatic 24-hour mute, a notification to the moderation team via Slack, and a log entry to a PostgreSQL audit trail. The action pipeline is configurable via YAML, with a dry-run mode for testing.

### How to Use

1. **Deploy the Detection Pipeline with Custom Thresholds**  
   Clone the repository and run `docker-compose up -d` to start the inference service, graph database, and message queue (RabbitMQ). Configure your environment variables in `.env`:  
   ```bash
   TOXICITY_THRESHOLD=0.85
   SPAM_THRESHOLD=0.90
   FTS_BAN_THRESHOLD=150
   MODERATION_WEBHOOK_URL=https://your-slack-webhook.com
   ```  
   Then, point your chat platform’s webhook to `http://localhost:8080/v1/ingest`. Verify ingestion with `curl -X POST http://localhost:8080/v1/classify -H "Content-Type: application/json" -d '{"text":"You are a scammer!","language":"en","context_id":"channel_123"}'`.

2. **Query User Trust Trends and Generate Reports**  
   Access the built-in GraphQL playground at `http://localhost:4000/graphql`. Execute a query to retrieve a user’s FTS history:  
   ```graphql
   {
     user(id: "user_abc123") {
       trustScore
       trustTimeline(range: "7d") {
         timestamp
         score
         reason
       }
     }
   }
   ```  
   Export the timeline as CSV via `/v2/user/user_abc123/trust_timeline.csv?range=7d`. Use this data to identify sudden drops caused by coordinated reporting or a single toxic outburst.

3. **Configure and Test an Automated Moderation Rule**  
   Edit `rules.yaml` in the `config/` directory to define a new rule:  
   ```yaml
   rules:
     - name: "Auto-mute on repeated spam"
       condition: "event_type == 'message' AND toxicity_score > 0.9 AND user_fts < 300"
       actions:
         - type: "mute"
           duration: "24h"
         - type: "webhook"
           url: "https://your-moderation-tool.com/api/notify"
         - type: "log"
           table: "moderation_actions"
   ```  
   Run `./fanguardctl dry-run --rules rules.yaml` to simulate the rule against recent event logs. Once validated, apply with `./fanguardctl apply --rules rules.yaml`.

### Best Practices

- **Implement a Graduated Moderation Escalation Ladder**  
  Avoid permanent bans on first offense. Configure fanguard’s rule engine with a 3-tier system: FTS between 200–300 triggers a warning notification, FTS between 100–200 triggers a 24-hour mute, and FTS < 100 triggers a manual review by a human moderator. This reduces false positives by 40% and preserves community trust while maintaining safety.

- **Combine FTS with External Signals for Higher Accuracy**  
  Enrich fanguard’s behavioral graph by feeding it data from your CRM or payment system via the `/v1/external/update` endpoint. For example, a user with a high FTS but a recent chargeback event should have their score temporarily penalized. Use the `external_weight` parameter (0.0–1.0) to control influence: `curl -X POST http://localhost:8080/v1/external/update -d '{"user_id":"user_abc","signal":"chargeback","weight":0.3}'`. This hybrid approach reduces coordinated attack success rates by 60%.
