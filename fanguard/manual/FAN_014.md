# fanguard - FAN_014

## Overview

fanguard is a high‑performance fan community management and protection platform, engineered for real‑time moderation, engagement analytics, and automated threat detection in large‑scale fan ecosystems. It operates at sub‑100ms latency for moderation decisions and scales to 10M+ concurrent user sessions.

## Core Features

### 1. Real‑Time Sentiment & Toxicity Analysis Engine
- **Parameter:** `toxicity_threshold` (0.0–1.0, default 0.85), `sentiment_window_ms` (100–5000 ms)
- **Technical detail:** Leverages a fine‑tuned transformer model (distilBERT‑based) with on‑device inference via ONNX Runtime. Achieves <50ms inference per message on a single NVIDIA T4 GPU. Supports multi‑language toxicity detection (EN, ZH, JP, KO) with per‑language confidence scores.
- **Output:** Returns `toxicity_score`, `sentiment_label` (positive/neutral/negative), and `action_recommendation` (warn/ban/allow) in JSON format.

### 2. Fan Community Interaction Graph & Anomaly Detection
- **Parameter:** `interaction_graph_depth` (1–5 hops), `anomaly_z_score_threshold` (2.0–5.0)
- **Technical detail:** Builds a directed interaction graph using Neo4j (v5.x) with edge weights derived from reply frequency, mention count, and shared content hashes. Runs a custom PageRank variant every 60 seconds to detect sudden influence spikes or coordinated bot activity. Alerts when a node’s influence score exceeds 3 standard deviations from its 24‑hour rolling mean.
- **Output:** Generates a `suspicious_node_list` with `node_id`, `influence_delta`, and `confidence_interval`.

### 3. Automated Moderation Workflow with PostgreSQL Audit Trail
- **Parameter:** `moderation_queue_priority` (0–100), `auto_ban_threshold` (e.g., 3 violations in 5 minutes)
- **Technical detail:** Integrates with PostgreSQL (v15+) via connection pooling (pgbouncer) for storing all moderation actions, user reports, and appeal logs. Supports custom SQL‑based rules (e.g., `SELECT COUNT(*) FROM reports WHERE reported_user_id = $1 AND created_at > NOW() - INTERVAL '5 minutes'`). Executes ban/unban operations through the platform’s REST API with idempotency keys to prevent duplicate actions.
- **Output:** Returns `action_id`, `execution_status`, and `affected_user_count`.

## How to Use

### Step 1: Configure the Sentiment Analysis Pipeline
```bash
# Set environment variables for the toxicity model
export FANGUARD_TOXICITY_THRESHOLD=0.85
export FANGUARD_SENTIMENT_WINDOW_MS=500
export FANGUARD_MODEL_PATH=/models/distilbert-toxicity.onnx

# Validate configuration
fanguard-cli --check-config --module sentiment
```
Expected output: `[INFO] Sentiment pipeline validated. Model loaded in 213ms.`

### Step 2: Deploy the Interaction Graph Builder
```python
from fanguard.graph import InteractionGraph

graph = InteractionGraph(
    neo4j_uri="bolt://localhost:7687",
    neo4j_user="fanguard",
    neo4j_password="<your_password>",
    depth=3,
    anomaly_z_score=3.5
)

# Process last 24 hours of fan interactions
result = graph.analyze_window(hours=24)
print(f"Detected {len(result['suspicious_nodes'])} anomalous nodes.")
```
- **Note:** Ensure Neo4j has at least 8GB heap memory for graphs with >1M nodes.

### Step 3: Set Up Automated Moderation with PostgreSQL
```sql
-- Create the moderation audit table
CREATE TABLE fanguard.moderation_actions (
    action_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id VARCHAR(64) NOT NULL,
    action_type VARCHAR(16) CHECK (action_type IN ('warn', 'ban', 'unban', 'appeal')),
    created_at TIMESTAMPTZ DEFAULT NOW(),
    rule_triggered TEXT,
    idempotency_key VARCHAR(128) UNIQUE
);

-- Enable automatic ban after 3 reports in 5 minutes
CREATE OR REPLACE FUNCTION auto_ban_if_needed()
RETURNS TRIGGER AS $$
BEGIN
    IF (SELECT COUNT(*) FROM fanguard.moderation_actions
        WHERE user_id = NEW.user_id
        AND action_type = 'warn'
        AND created_at > NOW() - INTERVAL '5 minutes') >= 3 THEN
        INSERT INTO fanguard.moderation_actions (user_id, action_type, rule_triggered)
        VALUES (NEW.user_id, 'ban', 'auto_ban_threshold_exceeded');
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```
Then run the fanguard daemon: `fanguard-cli --daemon --pg-conn "postgresql://user:pass@localhost:5432/fanguard"`

## Best Practices

### 1. Tune Toxicity Thresholds Per Community Segment
- **Why:** Fan communities vary widely in acceptable language (e.g., gaming vs. K‑pop). Use A/B testing with a 10% holdout group to find the optimal `toxicity_threshold` that minimizes false positives while catching 95% of flagged content. Log every threshold change with a version tag in PostgreSQL for auditability.

### 2. Implement a Three‑Tier Moderation Escalation
- **How:** Configure the moderation workflow to escalate automatically:
  - **Tier 1 (Automated):** Low‑confidence toxicity warnings → send to user via DM.
  - **Tier 2 (Semi‑automated):** Medium‑confidence or repeated violations → queue for human review via a Slack webhook.
  - **Tier 3 (Manual):** High‑confidence toxicity or anomaly alerts → immediate ban with an appeal link.
- **Benefit:** Reduces human moderator workload by 70% while maintaining 99.9% accuracy on critical cases.
