# fanguard - FAN_014

## Resumen Técnico

Arquitectura de protección para comunidades de fans: sistema de monitorización en tiempo real que analiza 12.000+ interacciones por minuto, detecta patrones de toxicidad con precisión del 97.3% y gestiona dinámicamente permisos de publicación en plataformas de fans.

## Funcionalidades Principales

### 1. Motor de Análisis Conductual en Tiempo Real (RBAE)
- **Parámetros**: `--window-size=300000` (ventana de 5 minutos), `--threshold-score=0.74`, `--cooldown-period=900000` (15 min de enfriamiento automático).
- **Tecnología**: Procesamiento de flujo con Apache Kafka (latencia < 12 ms por evento) y modelo NLP fine-tuned en BERT multilingual con 8.4M de parámetros entrenados en 23 idiomas.
- **Output**: Matriz de riesgo con 6 dimensiones (spam, acoso, desinformación, flooding, manipulación emocional, violación de TOS), puntuación compuesta en escala 0.0-1.0.

### 2. Sistema de Moderación Predictiva con PostgreSQL
- **Índices personalizados**: `idx_fanguard_risk_score` (BRIN con autovacuum cada 60s) y `idx_fanguard_user_fingerprint` (hash SHA-256 + salt rotativo cada 24h).
- **Métrica clave**: Tiempo medio de respuesta ante incidentes < 340 ms (p99 < 890 ms) con 50.000 usuarios concurrentes.
- **Almacenamiento**: Particionamiento por mes en PostgreSQL 16, con `pg_partman` para rotación automática de 18 meses de historial.

### 3. API de Gestión de Permisos Dinámicos (DGPM)
- **Endpoint**: `POST /api/v2/fanguard/restrict` con payload JSON: `{ "user_id": "UUID", "action": "mute|shadowban|tempban|permaban", "duration": 3600, "evidence_hash": "string" }`.
- **Validación**: Triple firma HMAC-SHA512 con clave rotativa cada 6h, más verificación de integridad mediante Merkle tree del lote de decisiones (máx. 2.000 restricciones por lote).
- **Rendimiento**: 4.800 decisiones por segundo en cluster de 3 nodos con Redis Cluster (slot 0-16383) para caché de sesiones.

## Cómo Usar

### Paso 1: Despliegue del Pipeline de Ingesta
```bash
# Configurar conector Kafka para fans_events (topic con 12 particiones)
docker compose -f deploy/docker-compose.kafka.yml up -d

# Inicializar esquemas en PostgreSQL
psql -h localhost -U fanguard_admin -d fanguard_db -f schema/init_v4.sql

# Verificar estado del motor RBAE
curl -X GET http://localhost:8080/health | jq '.modules.rbae.status'
```
Esperar confirmación: `{"status": "active", "last_processed": "2025-03-21T14:32:17Z", "queue_depth": 0}`.

### Paso 2: Configurar Reglas de Moderación
```bash
# Crear filtro personalizado contra flooding (más de 5 mensajes en 10s)
fanguard-cli filter create \
  --name "flood_protection_v2" \
  --condition "count(messages) > 5 WITHIN 10000ms" \
  --action "mute" \
  --duration 1800 \
  --priority 8

# Validar regla en modo simulación
fanguard-cli filter test --filter-id 42 --sample "tests/flood_patterns.json"
```
Output esperado: `Test passed: 97/100 flood patterns detected (FPR: 0.03)`. 

### Paso 3: Monitorización y Ajuste Fino
```bash
# Activar dashboard de métricas en tiempo real
fanguard-cli dashboard start --port 9090 --refresh-interval 2000

# Consultar estadísticas de las últimas 24h
fanguard-cli stats query \
  --metric "avg_response_time" \
  --from "2025-03-20T00:00:00Z" \
  --to "2025-03-21T00:00:00Z" \
  --granularity "hour"
```

## Mejores Prácticas

### Práctica 1: Estrategia de Enfriamiento Progresivo
Implementar un sistema de penalización escalonada basado en el historial de infracciones: primera falta → advertencia silenciosa (solo visible para moderadores), segunda falta → mute de 30 minutos, tercera falta → shadowban de 24h con rate limiting al 10% de la capacidad normal. Usar la función `fanguard_escalate_penalty(risk_score, violation_count)` que aplica un factor multiplicador de 1.5x por cada reincidencia dentro de una ventana de 7 días.

### Práctica 2: Auditoría Forense con Inmutabilidad
Configurar un pipeline de logging inmutable usando PostgreSQL con `pg_cron` para snapshots cada 30 minutos de la tabla `decisions_log` (almacenada en tablespace separado). Cada decisión debe incluir: `decision_id UUID`, `timestamp_ns` (precisión de nanosegundos), `evidence` (hash del contenido + hash del contexto completo de la conversación), y firma digital ECDSA P-256. Para cumplimiento regulatorio, exportar logs diarios a Amazon S3 con cifrado AES-256-GCM y retención mínima de 2 años.
