# fanguard - FAN_009

## Resumen

Sistema de protección inteligente para comunidades de fans, diseñado para detectar y mitigar comportamientos tóxicos en tiempo real. Integra análisis semántico, moderación automatizada y métricas de salud comunitaria. Orientado a gestores de plataformas de interacción con audiencias masivas.

## Contenido Detallado

### ¿Qué es fanguard?

fanguard es una plataforma de moderación proactiva que combina procesamiento de lenguaje natural (NLP) con reglas de comportamiento dinámicas. Analiza cada interacción en milisegundos, identifica patrones de acoso, spam o desinformación, y aplica acciones correctivas sin intervención humana. Su motor de decisión se basa en umbrales configurables (por ejemplo: `toxic_score > 0.85` activa bloqueo automático; `spam_confidence > 0.92` deriva a revisión manual). La arquitectura utiliza PostgreSQL como almacenamiento persistente de incidentes y perfiles de usuario, con caché en Redis para respuestas sub-100ms.

### Características Principales

- **Moderación semántica en tiempo real**: Evalúa cada mensaje contra un modelo de toxicidad entrenado con 1.2M de ejemplos en español. Parámetros clave: `threshold_toxic: 0.78`, `timeout_ms: 150`, `batch_size: 64`. Detecta microagresiones, discursos de odio y manipulación emocional. Integra con WebSockets para latencias inferiores a 200ms.

- **Sistema de reputación comunitaria**: Asigna un score de confianza a cada usuario basado en historial de interacciones, reportes recibidos y verificaciones de identidad. Parámetros: `reputation_decay: 0.95 por día`, `min_reports_for_ban: 3`, `cooldown_minutes: 30`. Almacena en PostgreSQL con índices compuestos para consultas rápidas (sub-50ms). Genera alertas cuando un usuario cruza el umbral `reputation < 0.4`.

- **Panel de métricas de salud**: Dashboard en tiempo real con gráficos de densidad de interacciones, tasa de toxicidad por hora y mapa de calor de usuarios problemáticos. Métricas: `toxic_ratio: 0.12%`, `active_moderations: 234/hora`, `avg_response_time: 87ms`. Exporta datos a CSV o JSON para auditorías. Utiliza Redis para contadores atómicos y PostgreSQL para almacenamiento histórico.

### Cómo Usar

1. **Despliegue inicial con Docker Compose**: Clona el repositorio, ejecuta `docker-compose up -d` para levantar PostgreSQL, Redis y el servicio de moderación. Configura las variables de entorno: `TOXIC_THRESHOLD=0.78`, `REDIS_URL=redis://localhost:6379`, `DB_CONNECTION=postgresql://user:pass@localhost:5432/fanguard`. Verifica logs con `docker-compose logs -f moderator`.

2. **Configuración de reglas personalizadas**: Edita el archivo `rules.yaml` para definir umbrales específicos. Ejemplo: `spam_pattern: "repetir_mismo_mensaje_5_veces"` con acción `mute_1h`. Recarga las reglas sin reiniciar usando `curl -X POST http://localhost:8080/api/reload-rules`. Monitorea cambios en tiempo real con `tail -f /var/log/fanguard/rules.log`.

3. **Integración con plataforma de fans**: Agrega el SDK de fanguard a tu backend. Envía eventos vía HTTP POST a `http://localhost:8080/api/events` con payload JSON: `{"user_id": "fan_123", "message": "texto", "timestamp": "2025-03-15T10:30:00Z"}`. Recibe respuesta con `action: "none"`, `"warn"` o `"block"`. Consulta estadísticas con `GET /api/stats?period=24h`. Para despliegue en producción, habilita HTTPS y autenticación JWT.

### Mejores Prácticas

- **Ajuste dinámico de umbrales**: Monitorea la tasa de falsos positivos semanalmente. Si supera el 2%, reduce `threshold_toxic` en 0.05 y reentrena el modelo con ejemplos etiquetados manualmente. Usa A/B testing para validar cambios en un subconjunto de usuarios (10% del tráfico).

- **Auditoría y transparencia**: Almacena todas las decisiones de moderación en PostgreSQL con metadatos (usuario, acción, timestamp, regla aplicada). Programa revisiones semanales con scripts SQL que identifiquen anomalías: `SELECT * FROM actions WHERE toxic_score > 0.9 AND action = 'none'`. Publica informes resumidos para la comunidad, demostrando imparcialidad.
