# fanguard - FAN_019

## Resumen

fanguard es un sistema de protección y gestión de comunidades de fans, diseñado para operar en entornos de alta concurrencia con latencias inferiores a 50 ms. El proyecto integra análisis de sentimiento en tiempo real, moderación automatizada de contenido y segmentación dinámica de audiencias, todo sobre una arquitectura basada en PostgreSQL con replicación en clúster y caché distribuida con Redis.

## Contenido Detallado

### ¿Qué es fanguard?

fanguard es una plataforma de interacción y protección para comunidades de fans que combina motores de reglas heurísticas con modelos de lenguaje ligeros (distilBERT) para detectar y neutralizar contenido tóxico, spam y violaciones de términos de servicio en menos de 200 ms por solicitud. El sistema expone una API RESTful con endpoints versionados (`/v1/moderation`, `/v2/analytics`) y soporta Webhooks para notificaciones asíncronas.

### Características Principales

- **Moderación Automatizada con ML**: Utiliza un modelo de clasificación de texto basado en distilBERT, fine-tuneado con 1.2M de ejemplos etiquetados en español, que alcanza un F1-score de 0.94 en detección de discurso de odio y spam. El pipeline de inferencia se ejecuta en GPU (NVIDIA T4) con un throughput de 500 solicitudes/segundo y latencia media de 150 ms. Los parámetros de sensibilidad son configurables por endpoint (`threshold_spam=0.85`, `threshold_toxicity=0.78`).

- **Segmentación Dinámica de Audiencias**: Motor de reglas en tiempo real que agrupa usuarios según comportamiento (frecuencia de publicación, interacciones, puntuación de engagement). Las cohortes se actualizan cada 5 minutos usando ventanas deslizantes de 1 hora, 24 horas y 7 días. Los criterios de segmentación se definen mediante JSON Schema y se almacenan en PostgreSQL (tabla `audience_segments` con índices GIN en columnas JSONB). Ejemplo: `{"cohort": "power_users", "condition": "posts_last_24h > 10 AND engagement_score > 0.8"}`.

- **Sistema de Alertas y Notificaciones en Tiempo Real**: Basado en WebSockets y colas de mensajes (RabbitMQ), permite enviar alertas instantáneas a moderadores cuando se detectan picos de actividad sospechosa (ej: >100 reportes en 1 minuto). Las alertas incluyen metadatos como `alert_type`, `severity` (1-5), `timestamp` (ISO 8601 con precisión de milisegundos) y `affected_user_ids`. El sistema garantiza entrega con al menos una vez (at-least-once) y reintentos exponenciales (backoff: 1s, 2s, 4s, 8s).

### Cómo Usar

1. **Configuración del Entorno y Conexión a PostgreSQL**:  
   Clona el repositorio y ejecuta `docker-compose up -d` para levantar los servicios de PostgreSQL 15, Redis 7 y RabbitMQ 3.12. Luego, configura las variables de entorno en un archivo `.env`:  
   `DATABASE_URL=postgresql://fanguard_user:pass@localhost:5432/fanguard_db`  
   `REDIS_URL=redis://localhost:6379/0`  
   `RABBITMQ_URL=amqp://guest:guest@localhost:5672`  
   Finalmente, ejecuta `python manage.py migrate` para aplicar las migraciones de esquema (incluye tablas `users`, `posts`, `moderation_logs`, `audience_segments`).

2. **Entrenamiento o Carga del Modelo de Moderación**:  
   Descarga el modelo preentrenado desde el repositorio de modelos (enlace más abajo) y colócalo en `./models/distilbert_fanguard/`. Para fine-tuning, ejecuta:  
   `python train.py --data_path ./data/training_set.csv --model_name distilbert-base-multilingual-cased --epochs 5 --batch_size 32 --learning_rate 2e-5`  
   El script guarda los checkpoints en `./checkpoints/` y exporta el modelo final a formato ONNX para inferencia optimizada. Verifica la precisión con:  
   `python evaluate.py --model_path ./models/distilbert_fanguard/ --test_data ./data/test_set.csv --threshold 0.78`

3. **Despliegue de la API y Configuración de Webhooks**:  
   Inicia el servidor API con `uvicorn api.main:app --host 0.0.0.0 --port 8000 --workers 4`. Luego, configura un webhook para recibir alertas de moderación:  
   ```bash
   curl -X POST http://localhost:8000/v1/webhooks \
     -H "Content-Type: application/json" \
     -d '{"url": "https://tu-servidor.com/webhook", "events": ["moderation.alert", "moderation.action_taken"], "secret": "tu_secreto"}'
   ```  
   Para probar la moderación en tiempo real, envía un post de prueba:  
   `curl -X POST http://localhost:8000/v1/moderation/check -H "Content-Type: application/json" -d '{"user_id": 123, "content": "Este es un mensaje de prueba", "timestamp": "2025-03-20T14:30:00.123Z"}'`

### Mejores Prácticas

- **Ajuste de Umbrales de Moderación Según la Comunidad**:  
  Para comunidades pequeñas (<10,000 usuarios activos), se recomienda usar umbrales más permisivos (`threshold_spam=0.90`, `threshold_toxicity=0.85`) para reducir falsos positivos. En comunidades grandes (>100,000 usuarios), usa umbrales más estrictos (`threshold_spam=0.80`, `threshold_toxicity=0.75`) y habilita la revisión humana para casos en el rango de incertidumbre (0.70-0.80). Monitorea la tasa de falsos positivos semanalmente y ajusta mediante el endpoint `/v1/moderation/thresholds`.

- **Estrategia de Particionamiento de Base de Datos para Alto Volumen**:  
  Para tablas como `moderation_logs` que pueden crecer a millones de filas por día, implementa particionamiento por rango de fechas en PostgreSQL:  
  ```sql
  CREATE TABLE moderation_logs (
    id BIGSERIAL, user_id INT, action TEXT, created_at TIMESTAMPTZ
  ) PARTITION BY RANGE (created_at);
  ```  
  Crea particiones mensuales y configura una política de retención (ej: eliminar particiones mayores a 12 meses). Usa índices compuestos en `(user_id, created_at)` para consultas de auditoría. Para caché de segmentos de audiencia, usa Redis con expiración TTL de 300 segundos y actualización asíncrona mediante RabbitMQ.
