# fanguard - FAN_005

## Resumen

Plataforma de gestión y protección de comunidades de fans con análisis en tiempo real de interacciones, detección de anomalías conductuales y orquestación automatizada de respuestas. Diseñada para administradores de comunidades con más de 10.000 miembros activos.

## Contenido Detallado

### ¿Qué es fanguard?

fanguard es un sistema de protección de ecosistemas de fans que integra motores de reglas dinámicas, análisis de sentimiento multilingüe y control de acceso granular. Opera con una latencia de procesamiento inferior a 50 ms por evento y soporta hasta 1.200 transacciones por segundo en clústeres de PostgreSQL con particionamiento por hash de ID de comunidad.

### Características Principales

- **Monitorización de Conducta con Umbrales Adaptativos**: Motor de reglas basado en ventanas deslizantes de 5 minutos que detecta picos de actividad sospechosa (ej: más de 30 mensajes por minuto desde una misma IP). Almacena metadatos en tablas particionadas de PostgreSQL con índices BRIN para consultas rápidas de series temporales.

- **Filtrado Semántico de Contenido en Tiempo Real**: Pipeline de NLP con modelo BERT fine-tuned para español que clasifica mensajes en 8 categorías de riesgo (spam, discurso de odio, desinformación). Procesa hasta 500 tokens por solicitud con un umbral de confianza configurable (por defecto 0.85) y escribe en colas Redis para persistencia asíncrona.

- **Orquestación de Respuestas Automatizadas con Backoff Exponencial**: Sistema de acciones reactivas que ejecuta desde advertencias silenciosas hasta bloqueos temporales. Cada acción se registra con timestamp Unix en microsegundos y se replica en tablas de auditoría con retención configurable (mínimo 90 días). Soporta webhooks con reintentos con backoff exponencial (3 intentos, base 2 segundos).

### Cómo Usar

1. **Configuración Inicial del Entorno**  
   Clona el repositorio y ejecuta `docker-compose up -d` para levantar los servicios de PostgreSQL 14, Redis 7 y el worker de Celery. Luego, desde la raíz del proyecto, ejecuta `python manage.py migrate --database=shard_0` para inicializar las tablas principales. Verifica con `curl -X GET http://localhost:8000/health` que el endpoint devuelva `{"status": "ok", "version": "2.1.3"}`.

2. **Definición de Reglas de Protección**  
   Accede a la API REST en `/api/v1/rules` con un POST que incluya el JSON:  
   ```json
   {
     "community_id": "fan_005",
     "rule_type": "velocity",
     "params": {
       "max_events": 25,
       "window_seconds": 300,
       "action": "warn"
     }
   }
   ```  
   El sistema validará el esquema contra el archivo YAML de reglas en `config/rules_schema.yaml` y devolverá un ID de regla único (UUID v4) con un campo `created_at` en formato ISO 8601.

3. **Activación de Monitoreo en Vivo**  
   Inicia el worker de consumo con `celery -A fanguard worker -l info -Q events`. Luego, envía un evento de prueba usando el SDK:  
   ```python
   from fanguard.client import FanguardClient
   client = FanguardClient(api_key="sk-...", base_url="http://localhost:8000")
   response = client.emit_event(community_id="fan_005", user_id="user_123", event_type="message", payload={"text": "prueba", "timestamp": 1710000000})
   ```  
   Revisa los logs del worker para confirmar que el evento se procesó y se almacenó en la tabla `events_2024_q1`.

### Mejores Prácticas

- **Particionamiento Temporal de Datos**: Para comunidades con más de 500.000 eventos diarios, configura el particionamiento por mes en PostgreSQL usando herencia de tablas. Esto reduce el tiempo de escaneo en consultas de auditoría hasta un 40% y permite eliminar particiones antiguas sin bloquear escrituras.

- **Estrategia de Reintentos con Circuit Breaker**: Al integrar webhooks externos, implementa un patrón de circuit breaker con umbral de 5 fallos consecutivos en 60 segundos. Usa Redis para mantener el estado del circuito y evitar saturar servicios externos durante picos de actividad maliciosa.
