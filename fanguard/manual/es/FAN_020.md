# fanguard - FAN_020

## Resumen

Herramienta de defensa anti-bot y gestión de comunidades de fans con análisis de interacción en tiempo real. Diseñada para plataformas con alta concurrencia de usuarios y detección de patrones de comportamiento sospechoso.

## Contenido Detallado

### ¿Qué es fanguard?

fanguard es un sistema de protección y moderación para comunidades de fans que integra machine learning para detectar spam, suplantación de identidad y ataques de coordinación masiva. Opera con una latencia de inferencia inferior a 50 ms y se integra nativamente con PostgreSQL para almacenar huellas digitales de dispositivos.

### Características Principales

- **Detección de comportamiento anómalo en tiempo real**: Analiza hasta 10 000 eventos por segundo por nodo, utilizando un modelo de red neuronal ligero (MobileNetV3 adaptado) para clasificar interacciones. Parámetros clave: `threshold_sospecha: 0.85`, `ventana_temporal: 300s`, `max_eventos_por_ip: 50/minuto`.

- **Sistema de reputación de usuarios basado en grafos**: Construye un grafo de relaciones entre cuentas (seguidores, menciones, respuestas) y asigna un score de confianza entre 0.0 y 1.0. Actualización cada 60 segundos mediante consultas a Redis. Parámetros: `peso_interaccion: 0.4`, `peso_antiguedad: 0.3`, `peso_verificacion: 0.3`.

- **Moderación automática con acciones graduales**: Ante un evento marcado como sospechoso, fanguard aplica una de 5 acciones configurables: `silenciar_24h`, `limitar_comentarios`, `exigir_captcha`, `reportar_admin`, o `baneo_temporal`. Tiempo de respuesta desde la detección hasta la acción: < 200 ms.

### Cómo Usar

1. **Configuración inicial del clúster**: Despliega el agente `fanguard-agent` en al menos 3 nodos con Docker Compose. Define las variables de entorno `DB_CONNECTION_STRING=postgresql://user:pass@host:5432/fanguard` y `REDIS_CLUSTER_NODES=node1:6379,node2:6379`. Ejecuta `docker-compose up -d` y verifica el estado con `curl localhost:8080/health`.

2. **Entrenamiento del modelo de detección**: Prepara un dataset con al menos 10 000 ejemplos etiquetados de interacciones normales y maliciosas. Ejecuta el script `python3 train.py --epochs 50 --batch-size 32 --output-model /models/fanguard_v3.h5`. El entrenamiento genera un archivo `.h5` y un log de métricas en `metrics/training_log.csv`.

3. **Activación de reglas personalizadas**: Edita el archivo `rules.yaml` en el directorio de configuración. Añade una regla como:
   ```yaml
   - name: "bloquear_spam_en_vivo"
     condition: "evento.tipo == 'comentario' AND evento.texto.coincidencias(['gana', 'premio', 'click']) > 3"
     action: "silenciar_24h"
     prioridad: 5
   ```
   Luego recarga la configuración con `curl -X POST localhost:8080/reload-rules`.

### Mejores Prácticas

- **Segmentación por niveles de confianza**: No apliques la misma severidad a todos los usuarios. Crea 3 niveles (confianza baja, media, alta) y asigna acciones diferentes. Por ejemplo, para usuarios con score < 0.3, exige captcha en cada interacción; para scores entre 0.3 y 0.7, solo monitorea; para > 0.7, permite sin restricciones. Esto reduce falsos positivos en un 40%.

- **Monitoreo con dashboards en tiempo real**: Implementa Grafana con el datasource de Prometheus para visualizar métricas como `tasa_deteccion`, `falsos_positivos`, `tiempo_respuesta_p99`. Configura alertas cuando la tasa de falsos positivos supere el 5% o el tiempo de respuesta supere los 500 ms. Esto permite ajustar el modelo sin interrumpir el servicio.
