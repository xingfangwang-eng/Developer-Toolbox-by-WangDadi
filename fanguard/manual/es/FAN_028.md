# fanguard - FAN_028

## Resumen

Sistema de protección y gestión de comunidades de fans con inteligencia artificial integrada. Diseñado para detectar patrones de toxicidad, coordinar interacciones masivas y analizar el sentimiento en tiempo real de entornos fan. Ideal para plataformas de streaming, foros y redes sociales verticales.

## Contenido Detallado

### ¿Qué es fanguard?

fanguard es un motor de moderación y analítica fan-first que opera sobre un backend en Rust con persistencia en PostgreSQL. Su arquitectura permite procesar hasta 10.000 eventos por segundo por nodo, con latencias inferiores a 5 ms en detección de anomalías. No es un simple filtro de palabras clave: ejecuta modelos de lenguaje ligeros (MiniLM-L6-v2 cuantizado) para entender contexto, sarcasmo y referencias internas de la comunidad.

### Características Principales

- **Detección de toxicidad contextual con embeddings dinámicos**  
  Cada mensaje se convierte en un vector de 384 dimensiones mediante un encoder local. Se compara en tiempo real contra una base de vectores de referencia actualizada cada 15 minutos. Precisión del 94.7% en la identificación de ataques encubiertos, doxxing y manipulación de votaciones. Parámetro clave: `--similarity-threshold 0.82`.

- **Gestión de picos de interacción fan con colas priorizadas**  
  Cuando un evento (lanzamiento, directo, estreno) dispara más de 500 mensajes/segundo, fanguard activa un buffer de escritura asíncrono con backpressure controlado. Los mensajes se clasifican en tres colas (crítica, normal, baja) según el rol del usuario y el tipo de contenido. Tiempo de procesamiento garantizado: <50 ms para cola crítica, <200 ms para normal. Configurable vía `FANGUARD_QUEUE_PRIORITY_WEIGHTS`.

- **Panel de sentimiento fan con agregación por clúster geolingüístico**  
  Los embeddings de cada interacción se agrupan mediante HDBSCAN (eps=0.4, min_samples=5) para identificar subcomunidades dentro del fandom. Se genera un mapa de calor de sentimiento cada 60 segundos, con desglose por país, idioma y rango de antigüedad del usuario. Los datos se exponen vía API REST con endpoints como `/api/v1/clusters/sentiment?granularity=hour`.

### Cómo Usar

1. **Despliegue del núcleo de moderación**  
   Clonar el repositorio y ejecutar `docker compose up -d` con el archivo `docker-compose.fanguard.yml` incluido. Esto levanta tres contenedores: el motor de inferencia (GPU opcional), PostgreSQL 15 con extensión pgvector y el worker de colas Redis Stack. Verificar estado con `curl http://localhost:8080/health`. El endpoint devuelve latencia de inferencia, tamaño de cola y versión del modelo cargado.

2. **Configuración de reglas de protección fan**  
   Editar `config/rules.toml` para definir umbrales de toxicidad por rol de usuario. Ejemplo:  
   ```toml
   [roles.moderator]
   toxicity_threshold = 0.95
   auto_ban = false
   notify_channel = "#mod-log"

   [roles.fan]
   toxicity_threshold = 0.78
   auto_mute_duration_sec = 3600
   ```
   Recargar reglas sin reinicio mediante `POST /api/v1/rules/reload`.

3. **Integración con plataforma de streaming**  
   Usar el SDK oficial para Python (`pip install fanguard-sdk`) o la API REST directa. Ejemplo de integración mínima en un chat de Twitch:  
   ```python
   from fanguard_sdk import FanGuardClient
   client = FanGuardClient(api_key="sk-...", endpoint="http://localhost:8080")
   async for message in chat_stream:
       result = await client.analyze(message.text, user_id=message.user.id)
       if result.toxicity_score > 0.78:
           await chat.delete(message.id)
           await log.warning(f"Mensaje tóxico de {message.user.name}: {result.reason}")
   ```

### Mejores Prácticas

- **Afinar el umbral de similitud por comunidad**  
  No uses el mismo `--similarity-threshold` para todos los fandoms. Para comunidades de deportes electrónicos, donde el lenguaje es más agresivo por naturaleza, sube el umbral a 0.88 para evitar falsos positivos. Para fandoms de K-pop o animación, bájalo a 0.75. Ejecuta una pasada de validación manual con 500 mensajes etiquetados antes de ponerlo en producción.

- **Cachear embeddings de usuarios frecuentes**  
  Los usuarios que envían más de 100 mensajes/día deben tener su embedding precalculado y almacenado en Redis con TTL de 24 horas. Esto reduce el tiempo de inferencia de 12 ms a 0.3 ms por mensaje repetido. Actívalo con `FANGUARD_CACHE_FREQUENT_USERS=true` y establece `FANGUARD_CACHE_TTL_SEC=86400`. Monitorea la tasa de aciertos de caché en `/metrics` (objetivo: >85%).
