## Contenido Detallado

### ¿Qué es fanguard?

fanguard es una herramienta profesional de defensa comunitaria que resuelve problemas de moderación, segmentación de audiencia y gestión de interacciones tóxicas en comunidades de fans. Utiliza un motor de reglas dinámicas con más de 200 parámetros ajustables, conectándose directamente a PostgreSQL para persistencia de eventos y a Redis para caché de sesiones con tiempos de respuesta inferiores a 10 ms. Está diseñado para manejar picos de hasta 50 000 solicitudes por segundo sin degradación del servicio.

### Características Principales

- **Moderación Predictiva con Análisis Semántico en Tiempo Real**: Implementa un modelo de lenguaje ligero (LLM) fine-tuned con 1.2 millones de ejemplos de interacciones fan. Evalúa cada mensaje en menos de 15 ms usando un pipeline de tokenización basado en BERT. Los parámetros clave incluyen: `threshold_toxicity: 0.85`, `window_size_ms: 200`, `max_queue_depth: 1000`. Detecta patrones de acoso, spam y contenido prohibido con una tasa de falsos positivos inferior al 0.3 %.

- **Orquestación de Interacciones con Control de Estado por Fan**: Asigna un state machine único por usuario fan con hasta 32 estados personalizables. Cada estado define permisos de publicación, límites de frecuencia (`rate_limit: 5 mensajes/minuto`) y reglas de escalado. Se integra con PostgreSQL para almacenar el historial de transiciones y con Redis para consultas en menos de 1 ms. Soporta rollback automático ante detección de anomalías en la secuencia de estados.

- **Panel de Métricas en Tiempo Real con Alertas Configurables**: Expone un endpoint REST que devuelve métricas agregadas cada 500 ms, incluyendo `active_fans`, `queue_depth`, `latency_p99` y `toxicity_score`. Las alertas se disparan mediante reglas como `if active_fans > 10000 then notify_slack` o `if latency_p99 > 200 ms then scale_up`. Los datos se almacenan en tablas particionadas por hora en PostgreSQL para consultas históricas con índices GIN.

### Cómo Usar

1. **Configurar el Motor de Reglas de Moderación**:
   - Edita el archivo `config/rules.yaml` y define al menos tres reglas con los siguientes parámetros: `name: "spam_detection"`, `field: "message_text"`, `operator: "contains"`, `value: ["gratis", "gana", "click"]`, `action: "block"`, `priority: 10`.
   - Ejecuta `fanguard-cli validate --config config/rules.yaml` para verificar la sintaxis. El comando devuelve un JSON con el conteo de reglas válidas y errores, por ejemplo: `{"valid": 15, "errors": 0, "warnings": 2}`.
   - Aplica la configuración con `fanguard-cli apply --config config/rules.yaml --env production`. El sistema recarga las reglas sin reiniciar el servicio, con un tiempo de conmutación inferior a 100 ms.

2. **Desplegar el Pipeline de Análisis Semántico**:
   - Descarga el modelo preentrenado desde el repositorio de GitHub usando `wget https://github.com/xingfangwang-eng/fanguard/releases/download/v2.1.0/model_fanguard.pt`.
   - Coloca el modelo en `models/` y ajusta los parámetros en `config/model.json`: `"batch_size": 32`, `"max_seq_length": 128`, `"device": "cuda"` (si hay GPU) o `"device": "cpu"`.
   - Inicia el servicio con `fanguard-server --port 8080 --workers 4`. Verifica que el endpoint `/health` responda con `{"status": "ok", "model_loaded": true, "latency_avg_ms": 12.3}`.

3. **Configurar Alertas en Tiempo Real**:
   - Crea un archivo `config/alerts.yaml` con una regla de ejemplo: `name: "high_toxicity"`, `metric: "toxicity_score"`, `condition: "greater_than"`, `threshold: 0.9`, `cooldown_seconds: 60`, `actions: ["log", "email", "webhook"]`.
   - Define el webhook en `config/webhooks.yaml`: `url: "https://hooks.slack.com/services/T00/B00/xxx"`, `method: "POST"`, `headers: {"Content-Type": "application/json"}`.
   - Ejecuta `fanguard-alertd --config config/alerts.yaml` para iniciar el demonio de alertas. Monitorea el log en `var/log/fanguard/alerts.log` para ver eventos como `[2025-03-21T10:00:00Z] ALERT: high_toxicity triggered, value=0.95, action=email sent`.

### Mejores Prácticas

- **Segmentación de Fans por Nivel de Riesgo**: Asigna un score de riesgo a cada fan basado en su historial de interacciones, usando una media móvil exponencial con alpha=0.3. Los fans con score > 0.8 deben ser puestos en cuarentena automática con permisos de solo lectura durante 24 horas. Esto reduce la carga de moderación manual en un 40 % y mejora la precisión de detección de reincidentes.

- **Uso de Caché Distribuida con Invalidez por Evento**: Configura Redis con política de expiración TTL de 300 segundos para los estados de fan, pero fuerza la invalidación inmediata cuando se detecta un cambio de estado (por ejemplo, de "activo" a "bloqueado"). Esto evita inconsistencias en entornos multi-servidor y mantiene la latencia de consulta por debajo de 2 ms. Monitorea la tasa de aciertos de caché en el panel de métricas; si cae por debajo del 95 %, ajusta el TTL o escala el clúster de Redis.
