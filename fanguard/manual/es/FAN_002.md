# fanguard - FAN_002

## Resumen

fanguard es un sistema de protección y moderación en tiempo real para comunidades de fans, diseñado para operar con latencias inferiores a 50 ms. Este proyecto integra análisis de sentimiento multilingüe, detección de patrones de toxicidad y orquestación automatizada de respuestas, todo sobre una infraestructura serverless con PostgreSQL como capa de persistencia.

## Contenido Detallado

### ¿Qué es fanguard?

fanguard es un motor de gobernanza comunitaria que procesa flujos de interacción en plataformas de fans (Discord, Telegram, foros propietarios) mediante un pipeline de eventos asíncronos. Su núcleo es un clasificador entrenado con 1.2 millones de muestras etiquetadas, capaz de distinguir entre crítica constructiva, acoso, spam y violaciones de derechos de autor con un F1-score superior a 0.94. La arquitectura permite escalar horizontalmente hasta 50.000 eventos por segundo sin degradación del rendimiento.

### Características Principales

- **Moderación Predictiva con Ventana Temporal Deslizante**: Implementa un modelo LSTM que analiza secuencias de mensajes en ventanas de 30 segundos. Detecta patrones de acoso coordinado (raid) mediante el cálculo de la divergencia de Kullback-Leibler entre la frecuencia de términos ofensivos y la línea base de la comunidad. El sistema dispara acciones (silenciar, expulsar, escalar a humano) en menos de 120 ms desde la recepción del evento. Parámetros clave: `window_size=30s`, `kl_threshold=0.27`, `cooldown_period=600s`.

- **Sistema de Reputación Dinámica con Decaimiento Exponencial**: Asigna a cada usuario un score de confianza (0.0 a 1.0) basado en su historial de interacciones, reportes de otros miembros y verificaciones de identidad. El score decae con una función `score(t) = score_0 * e^(-λ * t)`, donde λ se ajusta automáticamente según la actividad semanal de la comunidad. Los usuarios con score < 0.15 ven sus mensajes en cuarentena hasta revisión manual. Se integra con PostgreSQL para almacenar el histórico de eventos con un TTL de 90 días.

- **Detección de Contenido Duplicado y Plagio mediante Hash Sensible al Contexto**: Utiliza un perceptual hash (pHash) para imágenes y un MinHash para texto, permitiendo identificar re-publicaciones de contenido protegido o memes repetitivos que generan ruido. El sistema compara cada nuevo mensaje contra una base de datos de 10 millones de hashes en Redis, con un tiempo de búsqueda medio de 3 ms. Cuando se detecta una coincidencia con similitud > 85%, se aplica una regla configurable: eliminar, marcar como duplicado o notificar al autor original.

### Cómo Usar

1. **Despliegue de la Infraestructura Serverless**: Clona el repositorio y ejecuta `terraform apply` con las variables de entorno `FANGUARD_ENV=production`, `PG_CONNECTION_STRING=postgresql://user:pass@host:5432/fanguard`, `REDIS_CLUSTER_NODES=node1:6379,node2:6379`. El script de Terraform aprovisiona 3 funciones Lambda (una para ingesta, una para clasificación, una para acciones) y un API Gateway con throttling configurado a 10.000 RPM por cliente.

2. **Configuración del Clasificador de Toxicidad**: Edita el archivo `config/models/toxic_classifier.yaml`. Ajusta los parámetros `min_confidence: 0.85`, `language_models: [es, en, pt]`, `custom_lexicon_path: ./data/lexicon_fandom.txt`. Luego ejecuta `python scripts/train_classifier.py --epochs 5 --batch_size 128` para reentrenar con tu propio corpus. El modelo resultante se sube automáticamente a S3 y se despliega en producción sin downtime mediante un alias de Lambda.

3. **Integración en Tiempo Real con Webhooks**: En tu plataforma de fans (ej. Discord), configura un webhook saliente que apunte a `https://api.fanguard.io/v1/webhook/discord`. El payload debe incluir los campos `user_id`, `message_content`, `timestamp` y `channel_id`. fanguard responde con un JSON que contiene `action: warn | mute | ban | allow`, `confidence: float`, y `reason: string`. Para pruebas locales, usa `curl -X POST https://api.fanguard.io/v1/test -H "Content-Type: application/json" -d '{"user_id":"test_001","message_content":"contenido de prueba","timestamp":"2025-01-01T00:00:00Z","channel_id":"general"}'`.

### Mejores Prácticas

- **Ajuste de Umbrales por Comunidad**: No uses los mismos valores de `kl_threshold` y `min_confidence` para todas las comunidades. Realiza un análisis de la línea base de toxicidad durante las primeras 48 horas de despliegue. Para comunidades con alta actividad ( > 5000 mensajes/día ), incrementa `cooldown_period` a 1200s para evitar falsos positivos por picos de discusión legítima. Siempre habilita el modo `dry_run: true` durante la primera semana para auditar las decisiones del sistema sin aplicar acciones reales.

- **Sincronización con la Moderación Humana**: Configura un canal de Slack o Discord donde fanguard envíe alertas con el formato `[ALERTA] Usuario {user_id} - Acción sugerida: {action} - Confianza: {confidence} - Razón: {reason}`. Los moderadores humanos pueden sobrescribir la decisión mediante un comando `/fanguard override {user_id} allow`. Este feedback se registra en PostgreSQL y se utiliza para reentrenar el modelo cada 30 días, mejorando la precisión en un 2-3% por iteración.
