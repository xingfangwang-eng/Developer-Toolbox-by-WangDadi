## Resumen

fanguard es un sistema de protección y gestión de comunidades de fans con análisis en tiempo real, diseñado para plataformas de interacción masiva. Este proyecto integra motores de reglas antifraude, moderación automatizada de contenido y métricas de engagement con precisión de milisegundos.

## Contenido Detallado

### ¿Qué es fanguard?

fanguard es una herramienta profesional de backend que resuelve problemas críticos en la administración de comunidades de fans: detección de bots, moderación de contenido tóxico y análisis de patrones de interacción. Está construida sobre una arquitectura de microservicios con PostgreSQL como capa de persistencia principal y Redis para caché de sesiones en tiempo real.

### Características Principales

- **Motor de detección de bots con scoring en tiempo real**: Analiza 47 parámetros por solicitud (frecuencia de publicación, patrones de clics, tiempo entre acciones, geolocalización, hash de dispositivo) y asigna un score de 0 a 100. Los umbrales se configuran mediante archivos YAML. El tiempo de respuesta es inferior a 50 ms bajo carga de 10,000 peticiones concurrentes.

- **Moderación de contenido con pipeline de NLP**: Implementa un pipeline de 3 etapas: filtrado por reglas regex (palabras prohibidas), clasificación por modelo BERT fine-tuned (precisión > 94% en toxicidad) y verificación contra lista negra de URLs maliciosas. El procesamiento por mensaje no supera los 200 ms.

- **Dashboard de métricas de comunidad con exportación a CSV/JSON**: Expone endpoints REST que devuelven KPIs como tasa de retención semanal, ratio de interacciones positivas/negativas, y distribución horaria de actividad. Los datos se almacenan en tablas particionadas por mes en PostgreSQL, permitiendo consultas históricas con latencia menor a 2 segundos para rangos de hasta 12 meses.

### Cómo Usar

1. **Configurar el motor de detección de bots**:  
   Edita el archivo `config/bot_detection.yaml`. Define el umbral de score (`threshold: 75`) y los pesos de cada parámetro. Luego ejecuta `fanguard-cli --init-bot-engine`. El sistema cargará las reglas y comenzará a puntuar todas las solicitudes entrantes. Puedes verificar el estado con `curl http://localhost:8080/health/bot-engine`.

2. **Desplegar el pipeline de moderación de contenido**:  
   Descarga el modelo preentrenado desde `https://models.fanguard.io/bert-toxicity-v3.onnx` y colócalo en `models/`. Ejecuta `fanguard-cli --start-moderation-pipeline`. El pipeline se conectará automáticamente al stream de mensajes (Kafka topic `fanguard.messages`). Para probar, envía un mensaje de prueba: `curl -X POST http://localhost:8080/api/v1/moderate -H "Content-Type: application/json" -d '{"content":"texto de prueba"}'`. Recibirás un JSON con `toxicity_score` y `action_taken`.

3. **Consultar métricas de comunidad**:  
   Una vez que el sistema ha estado activo al menos 24 horas, ejecuta `fanguard-cli --export-metrics --format csv --output ./metrics.csv`. Esto generará un archivo CSV con columnas como `date`, `total_interactions`, `bot_score_avg`, `toxicity_rate`. Para consultas en vivo, usa `curl http://localhost:8080/api/v1/metrics?range=last_7_days`. La respuesta incluirá un array de objetos JSON con timestamps en formato ISO 8601.

### Mejores Prácticas

- **Ajuste fino de umbrales según la comunidad**: No uses el mismo umbral de bot score para todas las comunidades. Para comunidades pequeñas (< 10,000 miembros), establece `threshold: 85` para evitar falsos positivos. Para comunidades masivas (> 100,000 miembros), usa `threshold: 60` y combínalo con revisión manual de casos borderline. Monitorea la tasa de falsos positivos semanalmente y ajusta los pesos de los parámetros en consecuencia.

- **Integración con sistemas de ticketing existentes**: fanguard expone webhooks para cada acción de moderación (bot detectado, contenido tóxico eliminado). Configura estos webhooks para que envíen eventos a tu sistema de ticketing (ej. Jira, Zendesk) con prioridad basada en el score. Esto permite que los moderadores humanos revisen solo los casos más ambiguos (score entre 60 y 80) y automaticen completamente los casos de alto riesgo (score > 90).
