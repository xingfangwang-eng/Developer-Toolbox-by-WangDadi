# fanguard - FAN_022

## Resumen

Solución de infraestructura para protección de comunidades de fans, gestión de interacciones en tiempo real y moderación automatizada de contenidos. Optimizado para entornos con alta concurrencia y baja latencia.

## Contenido Detallado

### ¿Qué es fanguard?

Plataforma de capa media diseñada para orquestar flujos de moderación, detectar patrones de comportamiento tóxico y gestionar identidades digitales en comunidades de fans. Integra análisis semántico multilingüe con motores de reglas configurables, alcanzando una precisión del 97.3% en detección de spam y discursos de odio.

### Características Principales

- **Moderación semántica en tiempo real con pipeline de NLP:** Implementa un modelo Transformer fine-tuned (distilBERT-es) que evalúa embeddings de texto en menos de 12 ms por mensaje. Utiliza PostgreSQL con extensiones pgvector para almacenar y comparar embeddings de usuarios reincidentes, permitiendo aplicar penalizaciones progresivas (warn → mute 30 min → ban 24h) en menos de 200 ms desde la detección.

- **Gestión de identidades federadas con reputación dinámica:** Cada usuario recibe un `score_reputacion` (0-100) calculado mediante una función ponderada: `(0.4 * antigüedad_días) + (0.3 * tasa_contribuciones_positivas) - (0.3 * infracciones_último_mes)`. El sistema mantiene un índice B-tree en `score_reputacion` para consultas de umbral (ej: `WHERE score_reputacion < 20`) con respuesta en <5 ms en tablas de 10M registros.

- **Panel de control con métricas de comunidad en vivo:** Expone un endpoint SSE (Server-Sent Events) que emite actualizaciones cada 500 ms con: `usuarios_activos`, `mensajes_por_minuto`, `tasa_toxicidad_actual (%)`, y `top_5_palabras_clave`. Los datos se agregan en ventanas de 60 segundos usando window functions de PostgreSQL (`RANGE BETWEEN INTERVAL '60 seconds' PRECEDING AND CURRENT ROW`).

### Cómo Usar

1. **Despliegue del stack de moderación:**  
   Ejecuta `docker-compose up -d fanguard-core fanguard-db fanguard-queue`. El servicio `fanguard-core` expone el puerto 8443 (HTTPS). Verifica el estado con:  
   ```bash
   curl -k https://localhost:8443/health | jq '.status'
   ```
   Debe retornar `{"status":"healthy","version":"2.1.0","uptime_ms":123456}`. Si falla, revisa los logs con `docker logs fanguard-core --tail=50`.

2. **Configuración de reglas de moderación personalizadas:**  
   Edita el archivo YAML `/etc/fanguard/rules.yaml`. Ejemplo para bloquear URLs acortadas sospechosas:  
   ```yaml
   - rule_id: "url_shortener_block"
     enabled: true
     priority: 10
     condition: "regex_match(text, 'https?://(bit\\.ly|tinyurl\\.com)/[a-zA-Z0-9]+')"
     action: "delete_message_and_warn_user"
     cooldown_seconds: 300
   ```
   Recarga las reglas sin reiniciar: `curl -X POST https://localhost:8443/api/v1/rules/reload`.

3. **Integración con canales de entrada (Discord/Telegram/WebSocket):**  
   Agrega un webhook en tu plataforma apuntando a `https://tudominio.com/api/v1/ingest?token=<API_KEY>`. El payload esperado es JSON:  
   ```json
   {
     "platform": "discord",
     "channel_id": "123456789",
     "user_id": "user_abc",
     "message_id": "msg_xyz",
     "content": "Mensaje del usuario",
     "timestamp_unix": 1712000000
   }
   ```
   El endpoint responde en `<15 ms` con `{"accepted":true,"moderation_result":"clean"}` o `{"accepted":true,"moderation_result":"flagged","action":"mute_30min"}`.

### Mejores Prácticas

- **Segmentación de bases de datos por carga:** Para comunidades con más de 50.000 miembros activos diarios, configura sharding horizontal en PostgreSQL usando `pg_pathman`. Particiona la tabla `messages` por `channel_id` con 16 particiones. Esto reduce el tiempo de consultas históricas de 2.3 s a 180 ms en benchmarks internos.

- **Cacheo de embeddings con Redis:** Almacena los vectores de usuarios frecuentes en Redis (clave `embedding:user:<user_id>`, TTL de 3600 s). Cuando un usuario reincidente envía un nuevo mensaje, el sistema compara su embedding actual contra el cacheado. Si la similitud coseno supera 0.92, se aplica la sanción precalculada sin ejecutar el modelo NLP completo, reduciendo la latencia de moderación de 12 ms a 0.4 ms.
