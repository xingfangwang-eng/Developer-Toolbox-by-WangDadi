# fanguard - FAN_017

## Resumen Técnico

Sistema de protección y gestión de comunidades de fans con análisis de interacción en tiempo real. Arquitectura basada en microservicios con procesamiento de eventos a **<5ms de latencia** y soporte para **10.000+ conexiones concurrentes** por nodo.

## Funcionalidades Principales

### 1. Motor de Detección de Toxicidad Multilingüe
- Análisis semántico con modelos NLP fine-tuned (BERT-base-es) para español, inglés y portugués.
- Parámetros: umbral de toxicidad configurable (0.0-1.0), ventana de tiempo de análisis (1s-300s), ratio de falsos positivos <2.3%.
- Almacenamiento de embeddings en **PostgreSQL con pgvector** para búsqueda de patrones históricos.

### 2. Sistema de Moderación Automática con Reglas Dinámicas
- Reglas en YAML con triggers por: frecuencia de mensajes (>X/60s), palabras clave, patrones de URL, edad de cuenta (<7 días).
- Ejecución asíncrona mediante **Redis Streams** con persistencia en **PostgreSQL** y cola de reintentos con backoff exponencial.
- Métricas: tiempo medio de acción <200ms, throughput de 5000 eventos/segundo por instancia.

### 3. Panel de Analítica de Comunidad en Tiempo Real
- Dashboard con WebSocket para métricas: engagement rate, ratio de nuevos miembros, top 10 influencers, detección de picos de actividad.
- Consultas agregadas en **ClickHouse** con ventanas de 15min, 1h, 24h y 7d.
- Exportación a CSV/JSON con filtros por rango de fechas y segmentos de usuarios.

## Guía de Implementación

### Paso 1: Despliegue de Infraestructura Base
```bash
# Clonar repositorio
git clone https://github.com/xingfangwang-eng/fanguard
cd fanguard

# Inicializar stack con Docker Compose
docker compose up -d postgres redis clickhouse

# Verificar estado de servicios
docker compose ps
```
Asegurar que PostgreSQL escucha en puerto 5432, Redis en 6379, y ClickHouse en 8123.

### Paso 2: Configuración del Motor de Toxicidad
```yaml
# config/toxicidad.yaml
model:
  path: models/bert-base-es-fanguard-v2
  threshold: 0.75
  batch_size: 32
storage:
  connection: postgresql://fanguard:pass@localhost:5432/fanguard
  vector_dim: 768
```
Ejecutar migración de esquema:
```bash
python scripts/migrate_toxicity.py
```

### Paso 3: Integración con Plataforma de Fans
```python
from fanguard.client import FanguardClient

client = FanguardClient(api_key="sk-xxxx")
# Configurar webhook para eventos de mensajes
client.register_webhook(
    url="https://tuapp.com/webhook/fanguard",
    events=["message.created", "user.joined"],
    secret="whsec_xxxx"
)
# Probar con mensaje de ejemplo
result = client.analyze_message("contenido del mensaje aquí")
print(result.toxicity_score, result.risk_level)
```

## Mejores Prácticas Recomendadas

### 1. Segmentación de Cargas de Trabajo
Separar el procesamiento de toxicidad (alta CPU/GPU) de la analítica (alta I/O). Usar **Kubernetes** con nodos dedicados para NLP (GPU T4 o superior) y nodos de almacenamiento con SSDs NVMe para bases de datos. Escalar horizontalmente el motor de toxicidad con HPA basado en uso de GPU.

### 2. Estrategia de Retención y Backup
Configurar **pg_dump** diario para PostgreSQL con retención de 30 días, y **ClickHouse TTL** para datos brutos de eventos (>90 días). Implementar replicación síncrona para PostgreSQL (3 nodos) y asíncrona para ClickHouse. Probar restauración completa cada mes con tiempo objetivo <4 horas.
