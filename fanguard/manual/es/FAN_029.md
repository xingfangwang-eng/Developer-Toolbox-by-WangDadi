# fanguard - FAN_029

## Resumen

fanguard es un sistema de orquestación de protección y moderación para comunidades de fans, diseñado para operar en entornos de alta concurrencia (10k+ peticiones/min). Implementa un motor de reglas en tiempo real con latencia inferior a 50 ms, capaz de detectar y mitigar comportamientos tóxicos, spam organizado y violaciones de derechos de autor en plataformas de interacción fan. El proyecto se despliega como una arquitectura de microservicios con persistencia en PostgreSQL y caché distribuida en Redis Cluster.

## Contenido Detallado

### ¿Qué es fanguard?

fanguard es una plataforma de gobernanza de comunidades fan que integra tres módulos críticos: un analizador semántico multilingüe (es, en, ja, ko) para detección de discurso de odio, un sistema de reputación basado en grafos de interacción (score de confianza del fan, actualizado cada 15 segundos), y un motor de cumplimiento normativo (GDPR, COPPA, Ley de Propiedad Intelectual) que genera informes de auditoría en formato JSON/CSV. Todo el tráfico se procesa mediante un pipeline de eventos en Apache Kafka, garantizando un throughput de 500 MB/s.

### Características Principales

- **Motor de Reglas Dinámicas con Compilación AOT**: Implementa un DSL propio (FanGuard Rule Language, FGRL) que permite definir reglas de moderación con sintaxis declarativa. Las reglas se compilan a código máquina mediante LLVM, reduciendo la latencia de evaluación a 12 µs por regla. Soporta hasta 10,000 reglas concurrentes con un consumo de memoria de 2.1 GB. Parámetros clave: `threshold_toxicity: 0.85`, `window_size_ms: 5000`, `batch_size: 256`.

- **Sistema de Reputación Basado en Grafos Temporales**: Cada interacción fan-fan o fan-contenido se modela como un edge en un grafo dirigido con pesos temporales (decaimiento exponencial, half-life de 7 días). El algoritmo PageRank modificado (FanRank) calcula la influencia de cada usuario en tiempo real, con actualización incremental cada 100 ms. El score se normaliza en el rango [0, 100] y se persiste en PostgreSQL con índices GIN para consultas de vecindad. Parámetros: `alpha: 0.85`, `tolerance: 1e-6`, `max_iterations: 50`.

- **Orquestador de Cumplimiento Normativo Automatizado**: Escanea cada publicación en busca de datos personales (DNI, correos, teléfonos) usando expresiones regulares compiladas (RE2) y un modelo NER fine-tuned con BERT (accuracy 97.2%). Si detecta una infracción, genera automáticamente un ticket de eliminación con sello de tiempo y hash SHA-256 del contenido. El tiempo medio de respuesta es de 230 ms, y el sistema puede manejar 5,000 tickets/hora sin degradación. Parámetros: `ner_model_path: "/models/bert-ner-es.onnx"`, `regex_patterns_path: "/config/pii_patterns.json"`, `ticket_ttl: 86400s`.

### Cómo Usar

1. **Despliegue del Pipeline de Moderación**: Clona el repositorio y ejecuta `docker-compose up -d` para iniciar los servicios base (Kafka, Redis, PostgreSQL). Luego, configura las reglas iniciales en el archivo `rules.fgrl` usando la sintaxis FGRL. Ejemplo: `rule "bloquear_spam" when content.similarity > 0.9 and user.reputation < 20 then action = "delete"`. Finalmente, inicia el worker de procesamiento con `./fanguard-worker --config config.yaml --threads 16`.

2. **Integración con la API REST**: La API expone endpoints para ingesta de eventos (`POST /api/v1/events`), consulta de reputación (`GET /api/v1/users/{id}/reputation`), y gestión de reglas (`PUT /api/v1/rules`). Cada petición debe incluir el header `Authorization: Bearer <token>` y el payload en JSON con campos obligatorios: `user_id`, `content`, `timestamp` (formato ISO 8601). El tiempo de respuesta SLA es de 200 ms para el 99% de las solicitudes.

3. **Configuración del Dashboard de Monitoreo**: Accede a `http://localhost:3000` para ver métricas en tiempo real: latencia de reglas (p50, p99), tasa de aciertos de detección (TP/FP), y volumen de eventos por minuto. Para habilitar alertas, edita `alertmanager.yml` y define umbrales: `if latency_p99 > 500ms for 5m then notify_slack`. Las alertas se envían a un canal de Slack configurado mediante webhook.

### Mejores Prácticas

- **Segmentación de Reglas por Tipo de Comunidad**: Organiza las reglas en namespaces según el contexto (ej: `gaming`, `kpop`, `sports`). Cada namespace tiene su propio archivo `rules.fgrl` y se carga en un worker dedicado. Esto evita la interferencia entre reglas de diferentes comunidades y permite escalar horizontalmente. Por ejemplo, para una comunidad de gaming, activa la regla `anti-toxicity` con `threshold_toxicity: 0.9`, mientras que para una de kpop, usa `anti-sasaeng` con `threshold_toxicity: 0.95`.

- **Estrategia de Backfill de Reputación**: Al incorporar una comunidad existente con millones de usuarios, no calcules la reputación desde cero. Usa el script `fanguard-backfill --source postgres://old_db --target postgres://new_db --batch 10000`, que procesa el historial de interacciones en lotes de 10,000 registros. Establece un decay rate inicial de 0.5 para acelerar la convergencia, y después de 7 días cambia a 0.1. Monitorea el error cuadrático medio (RMSE) entre la reputación calculada y la esperada, manteniéndolo por debajo de 0.05.
