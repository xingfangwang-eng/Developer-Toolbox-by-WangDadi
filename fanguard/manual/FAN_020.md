# fanguard - FAN_020

## Overview

fanguard is a high-performance fan community governance engine designed for real-time toxicity detection, identity clustering, and engagement amplification. It processes unstructured fan interactions across Discord, Telegram, and custom WebSocket streams, delivering sub-100ms moderation decisions and actionable community heatmaps.

## Core Features

### 1. Real-Time Toxicity Scoring with Multi-Modal Signal Fusion
fanguard evaluates each message across three parallel pipelines: linguistic sentiment (BERT-based fine-tuned on fan-specific slang), emoji-toxicity mapping (custom Unicode range analyzer), and linked-media fingerprinting (image hash cross-referencing against a banned-content vector DB). The final score is computed via a weighted Kalman filter, outputting a `toxicity_score` between 0.0 and 1.0 with a latency of ≤85ms per message at 10k RPM.

- **Parameter**: `--toxicity-threshold 0.72` triggers automatic mute or flagging.
- **Integration**: PostgreSQL with TimescaleDB hypertables stores per-user score trajectories for trend analysis.

### 2. Identity Graph & Bot Cluster Detection
Using a combination of WebSocket handshake metadata (IP, User-Agent entropy, TLS fingerprint) and behavioral pattern mining (message interval entropy, cross-channel copy-paste ratio), fanguard constructs a real-time identity graph. It automatically merges or splits user nodes when the Jaccard similarity between two profiles exceeds 0.89. Bot clusters are identified via DBSCAN on the feature vector `[message_rate, link_ratio, emoji_diversity, account_age_days]`.

- **Parameter**: `--cluster-eps 0.42` and `--min-samples 5` for bot cluster isolation.
- **Output**: JSON stream of `cluster_id`, `suspicion_score`, and `evidence_log`.

### 3. Engagement Amplification Engine with A/B Test Orchestration
fanguard doesn’t just police—it boosts. It identifies high-value fan creators (based on reply-to-ratio and emotional contagion metrics) and automatically schedules pinned messages, sticky threads, or cross-channel reposts. Each campaign runs as an A/B test with configurable parameters: `amplification_factor` (1.0–5.0), `cooldown_minutes`, and `target_channel_regex`. Results are logged to a Prometheus counter vector for dashboarding.

- **Parameter**: `--amplification-factor 2.5 --cooldown 30` ensures no spammy over-boosting.
- **Data Pipe**: Exports to Grafana via OpenTelemetry traces.

## How to Use

### Step 1: Initialize the Fan Stream Pipeline
Configure the input source by creating a `fan_config.yaml` with your Discord bot token, Telegram API ID, and WebSocket endpoint. Run the bootstrap command to validate connectivity and ingest the last 500 messages for baseline profiling.
```bash
fanguard init --config ./fan_config.yaml --baseline-history 500
```
The system will output a `profile_snapshot.json` containing initial toxicity distribution and identity graph seed nodes.

### Step 2: Deploy the Moderation & Amplification Daemon
Launch the main daemon with custom thresholds. This spins up three goroutines: one for real-time scoring, one for cluster detection (triggered every 60 seconds), and one for campaign scheduling.
```bash
fanguard run --toxicity-threshold 0.72 --cluster-eps 0.42 --min-samples 5 --amplification-factor 2.5 --cooldown 30
```
Monitor the live stream via `fanguard stats --watch`; you’ll see per-second metrics like `msg_processed`, `toxicity_flagged`, and `campaigns_active`.

### Step 3: Analyze & Export Historical Reports
After 24 hours of runtime, generate a forensic report covering bot cluster evolution, top toxic users, and campaign ROI (engagement lift vs. baseline).
```bash
fanguard report --from "2025-03-01T00:00:00Z" --to "2025-03-02T00:00:00Z" --output-format json --export-path ./reports/
```
The report includes a `cluster_migration_graph.gml` file for Gephi visualization and a `campaign_performance.csv` with lift percentages per target channel.

## Best Practices

### 1. Calibrate Toxicity Thresholds per Community Culture
Fan communities vary wildly in slang and ironic aggression. Run a 48-hour dry-run with `--toxicity-threshold 0.5` and analyze the false-positive rate against human moderation logs. Adjust the threshold upward (e.g., 0.78) for communities with high sarcasm density, or downward (e.g., 0.6) for family-friendly spaces. Use the `fanguard threshold --suggest` command to auto-tune based on your training data.

### 2. Combine Bot Cluster Detection with Manual Whitelisting
Automated bot detection may flag power users running scripts (e.g., giveaway bots or analytics scrapers). Maintain a `whitelist.yaml` with known benign IP ranges and TLS fingerprints. fanguard will skip DBSCAN clustering for these entries, reducing false positives. Periodically review cluster centroids with `fanguard cluster --inspect` to detect new botnet variants before they scale.
