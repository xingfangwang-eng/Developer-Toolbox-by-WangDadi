# fanguard - FAN_008

## Overview

fanguard is a high-performance fan community protection and anti-toxicity engine designed for real-time moderation, user sentiment analysis, and cross-platform fan interaction management. It processes over 10,000 concurrent user sessions with sub-50ms latency per moderation decision.

## Core Features

### 1. Real-Time Toxicity Detection & Auto-Moderation Engine
- **Parameter:** `--toxicity-threshold` (0.0–1.0, default 0.85)  
- **Technical detail:** Integrates a fine-tuned BERT-based NLP model (distilbert-base-uncased-fan-v1) for multilingual hate speech detection. Achieves 97.3% F1-score on internal fan community benchmarks.  
- **Capability:** Automatically flags, quarantines, or deletes messages in under 200ms via Redis-backed rule cache. Supports custom whitelist/blacklist regex patterns per community.

### 2. Fan Sentiment & Engagement Heatmap  
- **Parameter:** `--heatmap-window` (in hours, default 24)  
- **Technical detail:** Aggregates real-time sentiment scores from chat logs, reaction emojis, and post upvote/downvote ratios using a sliding window algorithm. Outputs a JSON heatmap with per-minute granularity.  
- **Capability:** Exposes REST API endpoints (`/api/v1/heatmap`) returning timestamped sentiment vectors, usable for dashboards or automated alerts when toxicity spikes > 2 standard deviations above baseline.

### 3. Cross-Platform Fan Identity Linking & Ban Sync  
- **Parameter:** `--platform-bridge` (e.g., discord, telegram, twitter)  
- **Technical detail:** Uses SHA-256 hashed user identifiers plus OAuth2 token exchange to correlate fan profiles across platforms. Stores linked identities in PostgreSQL with unique composite keys (platform + user_id).  
- **Capability:** Enables instant cross-platform ban propagation: a user banned on Discord is automatically muted/kicked on Telegram and Twitter within 500ms via webhook-driven event bus.

## How to Use

1. **Initialize the moderation engine with custom thresholds:**  
   Run `fanguard --init --toxicity-threshold 0.9 --heatmap-window 48 --platform-bridge discord,telegram`  
   This configures a 48-hour sentiment heatmap and sets toxicity flagging to require 90% confidence. The engine will start listening on port 8080 for incoming webhooks.

2. **Integrate your fan community platform via webhook:**  
   For Discord: add the fanguard bot to your server, then run `fanguard --add-webhook discord --token YOUR_DISCORD_BOT_TOKEN`  
   The tool automatically registers an event listener for `MESSAGE_CREATE` and `MEMBER_JOIN` events. Test connectivity with `fanguard --ping discord`.

3. **Query real-time sentiment heatmap and export logs:**  
   Call `curl http://localhost:8080/api/v1/heatmap?window=24&format=json` to retrieve last 24 hours of sentiment data. To export full moderation logs to CSV, use `fanguard --export-logs --since 2025-01-01 --until 2025-01-31 --output /var/logs/fanguard_january.csv`

## Best Practices

- **Tune toxicity thresholds per community maturity:** For newly launched fan communities, set `--toxicity-threshold` to 0.95 to avoid over-moderation; for established communities with known toxic patterns, lower to 0.8 and enable auto-quarantine for first-time offenders.
- **Implement a tiered ban escalation policy:** Use the cross-platform ban sync feature only after a user has been flagged by at least two distinct moderation triggers (e.g., toxicity score > 0.9 + 3 reports within 1 hour) to reduce false positives. Log all sync actions to a separate PostgreSQL table for audit.
