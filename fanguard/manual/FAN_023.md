# fanguard - FAN_023

## Overview

fanguard is a high-performance fan ecosystem protection and moderation engine, designed for real-time community health monitoring, anti-toxicity filtering, and engagement analytics. It operates at sub‑100ms latency per moderation decision, integrating directly with Discord, Telegram, and custom WebSocket backends.

## Core Features

### 1. Real-Time Toxicity & Spam Sentinel (TSS)
Deploys a multi‑layer NLP pipeline (BERT‑based classifier + regex rule engine + reputation scoring) to detect and quarantine harmful messages before they reach the community feed.  
- **Latency:** <80ms per message (GPU‑accelerated inference)  
- **Configurable threshold:** `--toxicity-cutoff 0.85` (float 0.0–1.0)  
- **Action modes:** `mute`, `delete`, `shadow-ban`, or `escalate-to-admin`  
- **Storage:** All flagged messages logged to PostgreSQL with full context (user_id, channel_id, timestamp, confidence score) for audit trails.

### 2. Fan Engagement Heatmap & Anomaly Detector (FEHAD)
Aggregates real‑time interaction data (likes, replies, reactions, join/leave events) and builds a temporal heatmap to identify engagement spikes, bot raids, or coordinated harassment.  
- **Granularity:** Per‑minute buckets, configurable window size (`--window-minutes 15`)  
- **Alert triggers:** Z‑score > 3.0 on any metric triggers a Webhook notification  
- **Output:** JSON stream to Kafka topic `fanguard.engagement` for downstream dashboards (Grafana, DataDog)  
- **Data retention:** 90 days in TimescaleDB hypertables for historical trend analysis.

### 3. Automated Moderation Rule Engine (AMRE)
Allows operators to define complex, multi‑condition moderation rules using a YAML DSL without writing code.  
- **Rule syntax example:**
  ```yaml
  - name: "Link spam from new accounts"
    conditions:
      - field: account_age
        operator: lt
        value: 3600  # seconds
      - field: message_links
        operator: gt
        value: 2
    actions:
      - type: delete
      - type: warn_user
        message: "New accounts may not post external links."
  ```
- **Rule evaluation:** Compiled to a DAG, evaluated in <5ms per rule set  
- **Versioning:** Every rule change is stored with a commit hash in the embedded SQLite database.

## How to Use

1. **Initialize the fanguard instance and connect to your community platform.**  
   Run `fanguard init --platform discord --token YOUR_BOT_TOKEN` to generate a configuration file (`fanguard.yml`). This command automatically discovers available channels and prompts you to select moderation zones (e.g., `#general`, `#support`). The tool then performs a connectivity test and reports API latency.

2. **Configure your first moderation rule set.**  
   Create a file `rules/anti-spam.yml` with the YAML structure shown above. Then execute `fanguard load-rules --file rules/anti-spam.yml --apply`. The engine validates the rule syntax, compiles it into an internal DAG, and runs a dry‑match against the last 100 messages in the selected channels. A summary table is printed showing how many messages would have been flagged.

3. **Launch the daemon and monitor live metrics.**  
   Start the background process with `fanguard run --daemon --log-level info`. Open a second terminal and run `fanguard stats --live` to see a real‑time table of messages processed, flagged, and actioned per second. To view the engagement heatmap, run `fanguard heatmap --last 60m` which outputs a JSON array suitable for plotting with `jq` and `gnuplot`.

## Best Practices

- **Use gradual enforcement levels.** Start with `shadow-ban` (messages visible only to the sender) for the first week after deploying a new rule. Review the false‑positive rate in the `fanguard audit` report. Only switch to `delete` or `mute` after the precision exceeds 95% on your community’s historical data.  
- **Combine FEHAD with AMRE for raid detection.** Create an AMRE rule that triggers when FEHAD reports a Z‑score > 4.0 on join events. The rule can automatically enable rate‑limiting (`--max-joins-per-minute 10`) and notify the admin channel. This reduces manual monitoring overhead by 70% in high‑traffic communities.
