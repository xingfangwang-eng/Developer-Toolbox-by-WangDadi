# fanguard - FAN_009

## Overview

fanguard is a high-performance fan community governance and interaction analytics engine, purpose-built for platform operators and community managers who need to detect toxicity, measure engagement fidelity, and automate moderation at scale. Keywords: Fan protection tool, Fan community management, Fan interaction platform.

## Detailed Content

### What is fanguard?

fanguard is a real-time, rule-engine-driven middleware that sits between your fan-facing frontend and your moderation backend. It ingests user-generated content (UGC) streams, applies multi-layered sentiment and behavior heuristics, and outputs actionable moderation signals—all within sub-50ms latency per event. Unlike generic spam filters, fanguard is tuned specifically for the high-velocity, emotionally charged environment of fan communities (e.g., K-pop stan Twitter, esports Discord servers, live-stream chat rooms).

### Core Features

- **Real-Time Toxicity Vectorization Engine**  
  Instead of simple keyword blacklists, fanguard uses a lightweight, on-device BERT distilled model (`fanguard-tiny-v2`) to embed every message into a 128-dimensional toxicity vector. The engine compares each vector against a dynamic community baseline (updated every 60 seconds via sliding window). Any vector exceeding a configurable Mahalanobis distance threshold (default: 3.5σ) triggers an instant flag. Latency: <15ms per inference on a single NVIDIA T4 GPU.

- **Interaction Fidelity Score (IFS) Calculation**  
  fanguard calculates a per-user `IFS` metric using a weighted formula: `IFS = (0.4 * ReplyRate) + (0.3 * AvgResponseTime) + (0.2 * ContentNoveltyScore) + (0.1 * CrossThreadActivity)`. `ContentNoveltyScore` is derived from a cosine similarity comparison against the user's own last 50 posts (stored in a PostgreSQL vector extension, pgvector). A user with `IFS < 0.25` is flagged as a potential spam bot or engagement farmer. The entire calculation is executed as a single SQL window function over a 24-hour partition.

- **Automated Moderation Pipeline with Escalation Matrix**  
  fanguard defines three severity tiers—`WARN`, `MUTE`, `BAN`—each with a configurable action set. The pipeline reads from a Redis Stream (`fanguard:moderation:ingest`), applies the toxicity vectorizer and IFS calculator in parallel, then routes the result through a YAML-defined escalation matrix. Example config parameter: `escalation_matrix.thresholds.mute: {toxicity_vector_distance: 4.0, ifs: 0.15}`. The entire pipeline is idempotent and supports exactly-once processing via a PostgreSQL outbox pattern.

### How to Use

1. **Deploy the fanguard Sidecar Container**  
   Add the `fanguard-sidecar:2.1.0` image to your Kubernetes pod spec. Mount a ConfigMap with your community-specific `moderation_rules.yaml`. Set environment variable `FANGUARD_LOG_LEVEL=DEBUG` and `FANGUARD_PG_DSN=postgresql://user:pass@pg-host:5432/fanguard`. The sidecar will auto-discover your application's UGC endpoint via a health check on port `:9090`.

2. **Configure the Escalation Matrix via YAML**  
   Create a file named `moderation_rules.yaml` in the mounted volume. Define three actions under `actions:`:  
   ```yaml
   actions:
     warn:
       action: "log_only"
       threshold_toxicity: 3.5
     mute:
       action: "api_call"
       endpoint: "http://your-mod-bot:8080/mute"
       payload_template: '{"user_id": "{{.UserID}}", "duration_seconds": 3600}'
       threshold_ifs: 0.20
     ban:
       action: "webhook"
       url: "https://hooks.slack.com/services/xxx"
       threshold_toxicity: 5.0
   ```
   Validate the YAML with `fanguardctl validate --file moderation_rules.yaml`.

3. **Run a Historical Backtest on Your Last 7 Days of Data**  
   Use the CLI tool to simulate moderation:  
   ```bash
   fanguardctl backtest \
     --pg-dsn $FANGUARD_PG_DSN \
     --start "$(date -d '7 days ago' -Iseconds)" \
     --end "$(date -Iseconds)" \
     --rules moderation_rules.yaml \
     --output /tmp/backtest_results.json
   ```
   The output includes a confusion matrix (True Positive, False Positive, True Negative, False Negative) and a latency histogram (p50, p95, p99) for every inference step.

### Best Practices

- **Tune the Mahalanobis Distance Threshold per Community**  
  Run a 48-hour dry-run with `fanguardctl dry-run --mode monitor` on a representative sample (minimum 10,000 messages). Use the generated `toxicity_distance_distribution.csv` to find the natural 95th percentile of your community's behavior. Set your `WARN` threshold to that value. This prevents over-moderation in naturally passionate fan spaces while still catching genuine abuse.

- **Implement a Human-in-the-Loop (HITL) Review Queue for Borderline Cases**  
  Configure fanguard to push all events where the toxicity vector distance is between 3.0 and 4.0 (the "gray zone") into a dedicated PostgreSQL table `moderation_review_queue`. Use a simple dashboard (e.g., Streamlit or Retool) to allow moderators to review these cases with a one-click "override" button. This reduces false positive bans by an average of 40% in production deployments.
