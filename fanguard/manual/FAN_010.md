# fanguard - FAN_010

## Overview

fanguard is a high‑performance fan community intelligence engine engineered for real‑time sentiment monitoring, automated content moderation, and cross‑platform engagement analytics. It ingests data from chat, forums, and social feeds, then applies a custom rule engine and ML classifiers to detect toxicity, spam, and brand risk within sub‑second latency.

## Core Features

### 1. Multi‑Source Fan Activity Pipeline with Sub‑Second Ingestion
- **Parameter:** `--source-config` (YAML/JSON) supports WebSocket, REST, and Kafka connectors.
- **Technical Specs:** Ingestion latency ≤ 50 ms per event under 10,000 concurrent connections; PostgreSQL + Redis stream buffer for crash‑safe persistence.
- **Behavior:** Automatically deduplicates cross‑posted messages using SHA‑256 fingerprinting and a sliding window of 60 seconds.

### 2. Adaptive Toxicity Classifier with Custom Rule Engine
- **Parameter:** `--toxicity-threshold` (float, range 0.0–1.0, default 0.85) and `--rule-file` (YAML with regex + keyword patterns).
- **Technical Specs:** Uses a fine‑tuned BERT‑base model (ONNX runtime, inference time < 15 ms per message) plus a user‑defined rule layer for fan‑specific slang and inside jokes.
- **Behavior:** Returns a `ToxicityReport` with fields `score`, `matched_rules`, `explanation`, and `action` (allow / flag / block). Supports hot‑reload of rules without restart.

### 3. Cross‑Platform Engagement Heatmap & Anomaly Detection
- **Parameter:** `--heatmap-window` (int, minutes, default 1440) and `--anomaly-zscore` (float, default 3.0).
- **Technical Specs:** Aggregates engagement metrics (likes, replies, shares, mentions) per platform (Discord, Twitter, Reddit) into a normalized heatmap stored in TimescaleDB hypertables.
- **Behavior:** Detects statistically significant spikes (z‑score > threshold) and triggers a webhook payload with `event_type`, `platform`, `metric`, `timestamp`, and `severity`.

## How to Use

### Step 1: Initialize the Configuration & Data Source
```bash
fanguard init --source-config ./sources.yaml --rule-file ./custom_rules.yaml
```
This command validates the YAML schemas, tests connectivity to all defined sources (e.g., Discord bot token, Twitter API v2), and creates the required PostgreSQL schema (`fanguard_meta`, `fanguard_events`, `fanguard_toxicity`).

### Step 2: Launch the Ingestion Engine with Real‑Time Monitoring
```bash
fanguard run --toxicity-threshold 0.90 --heatmap-window 720 --anomaly-zscore 3.5
```
The engine starts streaming events. Use the built‑in dashboard at `http://localhost:8080/metrics` to view live throughput (events/sec), classifier latency (p50/p99), and heatmap updates. Press `Ctrl+C` to gracefully flush buffers and write a final snapshot to TimescaleDB.

### Step 3: Export a Compliance Report & Trigger a Remediation Workflow
```bash
fanguard export --since 2025-04-01T00:00:00Z --until 2025-04-07T23:59:59Z --format json --output ./weekly_report.json
```
The report includes a `Summary` block with total events, flagged items, and top‑10 toxic users (by `score`). Optionally combine with:
```bash
fanguard action --user-id <uid> --action block --reason "Automated toxicity threshold exceeded"
```
This sends a platform‑specific block command (Discord timeout, Twitter mute) and logs the action to `fanguard_actions` table for audit.

## Best Practices

### 1. Implement a Two‑Stage Moderation Queue
- **Practice:** Set `--toxicity-threshold` at 0.80 for automatic flagging (human review queue) and 0.95 for automatic block. This reduces false positives while catching extreme toxicity instantly.
- **Why:** Fan communities often use ironic or in‑group language that triggers classifiers incorrectly. A human review layer improves accuracy and user trust.

### 2. Schedule Regular Rule Hot‑Reloads Based on Community Trends
- **Practice:** Use `fanguard reload --rule-file ./updated_rules.yaml` every 12 hours, driven by a cron job that pulls the latest slang and emoji patterns from a shared wiki or moderation log.
- **Why:** Fan culture evolves rapidly. Static rules become stale within weeks, leading to missed violations or over‑moderation. Hot‑reload avoids downtime and keeps the engine adaptive.
