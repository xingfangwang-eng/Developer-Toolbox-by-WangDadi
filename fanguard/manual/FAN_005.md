# fanguard - FAN_005

## Overview

fanguard is a high-performance fan community protection and engagement platform, engineered to detect malicious interactions, automate moderation workflows, and deliver real-time sentiment analysis across large-scale fan ecosystems. Optimized for communities exceeding 100,000 active members, it operates with sub-100ms latency on event processing and integrates natively with PostgreSQL for persistent audit trails.

## Core Features

- **Real-Time Toxicity & Spam Detection Engine**  
  Deploys a dual-layer transformer model (fine-tuned BERT + custom rule-based filters) to classify user-generated content in under 50ms per request. Supports configurable sensitivity thresholds (`--toxicity-threshold 0.85`, `--spam-score-limit 3.2`) and outputs a structured JSON alert with `risk_score`, `flagged_tokens`, and `action_recommendation` (e.g., `mute`, `ban`, `flag_for_review`). All events are logged to PostgreSQL with millisecond timestamps for compliance auditing.

- **Dynamic Fan Interaction Heatmap & Engagement Analytics**  
  Aggregates real-time interaction data (likes, shares, comments, reaction velocity) into a time-series heatmap with configurable granularity (`--bin-size 30s`, `--window 24h`). Exposes a REST API endpoint `/api/v1/heatmap?fan_group_id={id}&metric=velocity` returning a GeoJSON-compatible payload. Supports automatic anomaly detection when engagement deviates beyond 2.5 standard deviations from the rolling 7-day baseline.

- **Automated Moderation Workflow Engine with Escalation Rules**  
  Allows administrators to define multi-step moderation pipelines using a YAML-based DSL (`fanguard-pipeline.yaml`). Example: `step: if risk_score > 0.9 then action: auto_ban + notify_via_webhook(url: "https://hooks.slack.com/...")`. Each action is recorded in a `moderation_actions` table with `action_id`, `actor_user_id`, `rule_triggered`, and `execution_latency_ms`. Supports parallel execution of up to 10,000 rules per second via a Goroutine-backed scheduler.

## How to Use

### Step 1: Initialize the Database Schema & Configuration  
Execute the provided migration script to create the required PostgreSQL tables (`fan_users`, `moderation_actions`, `interaction_events`). Then, create a `fanguard.yaml` configuration file with your community ID, API keys, and moderation thresholds. Run `fanguard init --config ./fanguard.yaml` to validate connectivity and schema integrity.

### Step 2: Deploy the Real-Time Moderation Listener  
Start the event ingestion daemon with `fanguard listen --port 8080 --workers 12`. This binds to a WebSocket endpoint (`ws://localhost:8080/events`) and begins processing incoming fan messages. Use the provided CLI tool `fanguard-cli inject --file test_events.json` to simulate a batch of 10,000 events for stress testing. Monitor throughput via the built-in Prometheus metrics endpoint (`/metrics`).

### Step 3: Analyze Heatmap Data & Adjust Moderation Rules  
Access the generated heatmap via `fanguard heatmap --fan-group-id FG-2024-001 --output heatmap.html`. Review flagged anomalies in the `flagged_events` table. To update a moderation rule without downtime, modify the pipeline YAML and run `fanguard reload --pipeline ./fanguard-pipeline.yaml`. The engine hot-reloads the rule set within 200ms, preserving all in-flight event processing.

## Best Practices

- **Implement Tiered Moderation Actions**  
  Avoid binary (ban/no-ban) decisions. Configure a three-tier threshold system: `risk_score 0.6–0.8` → auto-mute for 1 hour + log; `0.8–0.95` → auto-flag for human review via webhook; `>0.95` → instant ban + permanent record. This reduces false positives by up to 40% in high-volume communities.

- **Integrate with External Identity Providers for Cross-Platform Banning**  
  Use the `--external-idp-url` parameter to connect fanguard with a central identity management system (e.g., Auth0, custom OAuth2). When a user is banned, fanguard can propagate the ban across connected platforms (Discord, Telegram, custom forums) via the webhook relay system. Ensure the IDP token has a TTL of no more than 15 minutes to limit exposure.
