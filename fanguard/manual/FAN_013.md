# fanguard - FAN_013

## Overview

fanguard is a real-time fan community integrity engine designed to detect, flag, and neutralize coordinated inauthentic behavior (CIB) across high-velocity social platforms. It processes 50,000+ interactions per second with sub-200ms latency, enabling platform operators to maintain organic engagement without manual moderation overhead.

## Core Features

### 1. Multi-Vector Bot Detection with Temporal Graph Analysis
fanguard constructs a directed temporal graph of user interactions (likes, retweets, mentions) and applies a custom **Temporal Random Walk (TRW)** algorithm. It identifies bot swarms by detecting abnormally high edge density (ρ > 0.85) within sliding windows of 300 seconds. Each flagged node is scored using a **Suspicion Index (SI)** , calculated as `SI = (f_out / f_in) * (1 - entropy(user_agent_fingerprint))`. The system supports PostgreSQL-backed persistent storage of graph snapshots for forensic rollback up to 90 days.

### 2. Real-Time Content Authenticity Scoring via Stylometric Fingerprinting
Every post is processed through a **Character-Level LSTM** model trained on 12 million authentic fan posts. The model outputs a **Stylometric Authenticity Score (SAS)** between 0.0 and 1.0, with a threshold of 0.78 to trigger a manual review. The pipeline includes a **tokenizer** that normalizes Unicode, emoji sequences, and casing variants. Benchmarks show a **F1 score of 0.94** on synthetic bot-generated content. The engine runs on a dedicated GPU cluster with a batch size of 512 and a maximum sequence length of 1024 tokens.

### 3. Automated Community Sentiment Drift Alerting
fanguard tracks a **Community Sentiment Vector (CSV)** across 7 dimensions: positivity, negativity, toxicity, sarcasm, urgency, confusion, and brand affinity. Using a **Holt-Winters exponential smoothing model** (α=0.3, β=0.1, γ=0.05), it detects statistically significant drift (p < 0.01) within a 15-minute window. Alerts are pushed via Webhook to Discord, Slack, or PagerDuty, including a **Drift Magnitude (DM)** metric calculated as the Euclidean distance between the current CSV and the 24-hour rolling baseline.

## How to Use

### Step 1: Deploy the Detection Pipeline with Custom Thresholds
```bash
fanguard deploy --mode production \
  --suspicion-threshold 0.65 \
  --stylometric-threshold 0.78 \
  --drift-p-value 0.01 \
  --postgres-uri postgresql://user:pass@host:5432/fanguard \
  --gpu-count 4
```
This command initializes the TRW graph engine, loads the LSTM model weights, and connects to your PostgreSQL instance. The system will begin ingesting data from the configured stream (Kafka or WebSocket). Monitor the `/health` endpoint for `"status": "ready"` before proceeding.

### Step 2: Configure Alerting Channels and Escalation Rules
```bash
fanguard alert add --channel slack \
  --webhook-url https://hooks.slack.com/services/T00/B00/xxx \
  --escalation-level 2 \
  --min-drift-magnitude 0.45 \
  --cooldown-seconds 600
```
Define multiple channels with distinct thresholds. For example, set a low threshold (DM > 0.3) for Discord and a high threshold (DM > 0.6) for PagerDuty with a 30-minute cooldown. Use `fanguard alert list` to verify all active rules. The system supports up to 10 simultaneous channels.

### Step 3: Run a Forensic Back-Analysis on Historical Data
```bash
fanguard analyze --start "2025-01-01T00:00:00Z" \
  --end "2025-01-07T23:59:59Z" \
  --graph-snapshot-id snap_2025_01_07 \
  --output-format csv \
  --export-path ./reports/
```
This command replays the temporal graph for the specified period, recalculating SI and SAS scores. The output CSV includes columns: `user_id`, `avg_si`, `max_si`, `avg_sas`, `flagged_post_count`, and `community_drift_score`. Use this data to train custom classifiers or to generate compliance reports.

## Best Practices

### 1. Tune the Suspicion Index Threshold Using a Holdout Validation Set
Before deploying to production, run a backtest on 72 hours of labeled historical data. Use `fanguard validate --dataset ./labeled_data.json` to generate a precision-recall curve. Set your SI threshold at the point where precision >= 0.95 and recall >= 0.80. This ensures you minimize false positives while catching the majority of coordinated attacks. Re-validate after any model update or after a major platform algorithm change.

### 2. Implement a Two-Stage Review Workflow for Flagged Content
Configure fanguard to automatically quarantine posts with an SAS below 0.60. Then, route posts with SAS between 0.60 and 0.78 to a human review queue. Use the `--quarantine-action` flag: `fanguard deploy --quarantine-action hide`. This reduces moderator workload by 40% while maintaining a safety net for borderline cases. Combine this with the Drift Alerting system to trigger an immediate human review during high-velocity events (e.g., product launches or live streams).
