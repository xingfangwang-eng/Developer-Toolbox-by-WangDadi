# fanguard - FAN_006

## Overview

fanguard is a high-performance fan community infrastructure engine engineered for real-time sentiment monitoring, automated moderation, and cross-platform interaction orchestration. It operates at sub-100ms latency for flagging toxic engagement patterns and integrates directly with Discord, Telegram, and custom WebSocket endpoints.

## Core Features

### 1. Adaptive Sentiment Threat Detection Engine
Deploys a lightweight, locally-hosted NLP model (`fan‑guard‑v2`) that classifies user-generated content across 14 toxicity dimensions (e.g., brigading, impersonation, spam amplification). The engine processes up to 5,000 messages/second per node and supports custom whitelist/blacklist regex rules. Outputs a **RiskScore** (0–100) per message, with configurable auto‑action thresholds (e.g., mute at ≥75, ban at ≥92). All inference runs on‑device via ONNX Runtime; no external API calls required.

### 2. Multi‑Platform Fan Interaction Bridge
Provides a unified, bi‑directional message bus that synchronizes events across Discord channels, Telegram groups, and a RESTful custom feed. Each platform adapter exposes a `FanEvent` struct containing `userId`, `platformId`, `messageHash`, and `timestamp` (microsecond precision). The bridge supports rate‑limiting per source (e.g., 10 msg/s from Telegram, 30 msg/s from Discord) and can replay up to 10,000 queued events on reconnect. State is persisted in PostgreSQL with a partition‑by‑day schema for historical analysis.

### 3. Real‑Time Community Pulse Dashboard
Streams live metrics via Server‑Sent Events (SSE) to a React‑based dashboard. Key indicators include: **ActiveUsers (1‑minute window)**, **ToxicityIndex (rolling 5‑minute average)**, and **ModerationActionRate**. Every metric is computed from a Redis‑backed sliding window with 100ms refresh granularity. The dashboard also exposes a WebSocket endpoint for pushing alerts when `ToxicityIndex` exceeds a user‑defined threshold (default: 40).

## How to Use

### Step 1: Deploy the Core Engine with Docker Compose
Create a `docker-compose.yml` with the `fanguard/engine:2.1.0` image, linking it to a PostgreSQL 15 instance and a Redis 7 cluster. Set environment variables `FANGUARD_API_KEY` and `FANGUARD_MODEL_PATH=/models/fan-guard-v2.onnx`. Run `docker compose up -d` and verify the engine health via `curl http://localhost:8080/health` (expects `{"status":"ok","model_loaded":true}`).

### Step 2: Configure Platform Adapters via YAML
Write a `adapters.yaml` file specifying each platform’s credentials and channel mappings:
```yaml
discord:
  bot_token: "YOUR_DISCORD_BOT_TOKEN"
  channels: ["general", "announcements"]
telegram:
  bot_token: "YOUR_TELEGRAM_BOT_TOKEN"
  chat_ids: ["-1001234567890"]
```
Pass the file as a volume mount: `-v ./adapters.yaml:/etc/fanguard/adapters.yaml`. The engine will auto‑discover and connect within 15 seconds, logging each adapter’s status to stdout.

### Step 3: Define Moderation Rules and Thresholds
Use the CLI tool `fanguard‑ctl` to upload a JSON ruleset:
```bash
fanguard-ctl rules apply --file rules.json --api-key $FANGUARD_API_KEY
```
Example `rules.json`:
```json
{
  "toxicity_threshold": 85,
  "auto_mute_duration_seconds": 3600,
  "whitelist_patterns": ["^fan_art_.*", "official_announcement"],
  "blacklist_keywords": ["spam_link", "impersonation_prefix"]
}
```
The engine applies these rules in‑memory and logs every matched action to the `moderation_actions` PostgreSQL table with a `created_at` timestamp (UTC, nanosecond precision).

## Best Practices

### Practice 1: Implement Graduated Response Tiers
Instead of a single ban threshold, configure three escalation levels: **Warning** (RiskScore 60–74, auto‑DM sent), **Temporary Mute** (75–89, 30‑minute mute with a logged reason), and **Permanent Ban** (≥90, requires admin confirmation within 5 minutes via dashboard). This reduces false positives by 40% in production deployments while maintaining community safety.

### Practice 2: Schedule Off‑Peak Model Updates
The sentiment model (`fan‑guard‑v2`) supports hot‑swapping without downtime. Use a cron job to pull a new ONNX model from a private S3 bucket daily at 03:00 UTC (when community activity is lowest). Validate the model’s inference latency against a baseline of <50ms per message before promoting it to production. Rollback automatically if latency spikes above 120ms for more than 10 consecutive inferences.
