# fanguard - FAN_002

## Overview

fanguard is a high-performance fan community management and protection engine designed for real-time abuse detection, identity verification, and engagement analytics. It processes fan interaction streams at sub‑50ms latency per event, enabling platform operators to maintain healthy digital ecosystems without sacrificing user experience.

## Core Features

### 1. Real‑Time Fan Toxicity & Spam Filter (FanGuard Shield)
- **Parameter:** `toxicity_threshold` (float, 0.0–1.0, default 0.85) — adjusts the sensitivity of the NLP-based classifier.
- **Technical detail:** Uses a distilled RoBERTa model (ONNX Runtime, <10MB) running on CPU with a single‑digit millisecond inference time per message. Supports 15 languages out‑of‑the‑box.
- Filters spam, hate speech, and coordinated bot campaigns by cross‑referencing message content with a PostgreSQL‑backed pattern database (updated every 60 seconds via CDC pipeline).

### 2. Fan Identity & Trust Scoring Engine (FanTrust)
- **Parameter:** `trust_window_hours` (int, default 168) — lookback window for calculating a user’s trust score based on activity history.
- **Technical detail:** Scans account age, interaction frequency, report ratio, and linked social profiles (OAuth verification). Generates a `trust_score` (0–100) stored in a Redis‑backed cache with a TTL of 24h. Supports batch re‑scoring of up to 10,000 users per minute via a sidekiq‑compatible worker queue.
- Enables automatic moderation actions (mute, shadowban, or escalate) based on configurable trust thresholds.

### 3. Community Engagement Heatmap & Anomaly Detection (FanPulse)
- **Parameter:** `heatmap_resolution_minutes` (int, default 15) — granularity of the engagement time‑series buckets.
- **Technical detail:** Aggregates likes, comments, shares, and new follows into a time‑series database (TimescaleDB). Uses a Holt‑Winters forecasting model to detect anomalies (e.g., sudden spam influx or bot‑driven follow surges) with a latency of <2 seconds from event ingestion.
- Exposes a REST API endpoint `/api/v1/heatmap?period=7d` for frontend dashboards and alerting integrations (Slack, PagerDuty).

## How to Use

### Step 1: Configure the Toxicity Filter Threshold
```bash
export FANGUARD_TOXICITY_THRESHOLD=0.90
export FANGUARD_TRUST_WINDOW_HOURS=336
```
Edit `config/fanguard.yml` to set the `toxicity_threshold` and `trust_window_hours` parameters. Run `fanguardctl validate` to test the configuration against a sample dataset (provided in `./samples/`).

### Step 2: Ingest a Fan Event Stream
```python
from fanguard import FanGuardClient
client = FanGuardClient(api_key="sk-xxx", base_url="https://api.fanguard.io/v1")
event = {
    "user_id": "fan_8923",
    "message": "Check out this free merch link!",
    "platform": "discord",
    "timestamp": "2025-03-21T14:30:00Z"
}
result = client.analyze_event(event)
print(result)  # {'toxicity_score': 0.97, 'trust_score': 12, 'action': 'shadowban'}
```
The engine returns an `action` field (`allow`, `shadowban`, `mute`, `block`) based on the combined toxicity and trust scores.

### Step 3: Query the Engagement Heatmap
```bash
curl -X GET "https://api.fanguard.io/v1/heatmap?period=7d&resolution=30" \
  -H "Authorization: Bearer sk-xxx" | jq '.data[] | {timestamp, engagement_count}'
```
Outputs a JSON array of time‑bucketed engagement counts. Use this to overlay anomaly markers on your community dashboard.

## Best Practices

### Practice 1: Gradually Roll Out Trust Scoring
Start with a `trust_threshold` of 40 (out of 100) and only apply automatic actions to users with scores below 20. Monitor false‑positive rates for the first 72 hours. Tune the `trust_window_hours` parameter based on your community’s typical user lifecycle — for high‑churn platforms, reduce to 48 hours.

### Practice 2: Combine Toxicity Filter with Manual Review Queue
Do not rely solely on the automatic filter. Configure fanguard to push borderline events (toxicity score between 0.75 and 0.90) to a PostgreSQL‑backed review queue. Use the `/api/v1/review/pending` endpoint to build a moderator dashboard with one‑click approve/reject actions. This reduces over‑moderation while maintaining safety.
