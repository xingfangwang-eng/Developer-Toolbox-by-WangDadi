# fanguard - FAN_011

## Overview

fanguard is a high-performance fan community management and interaction platform engineered for real-time moderation, behavioral analytics, and automated engagement tracking. It targets the intersection of fan protection, community governance, and scalable interaction infrastructure.

## Core Features

### 1. Real-Time Toxicity & Spam Filter (FanGuard Shield)
- Implements a multi-layered NLP pipeline (TF-IDF + BERT-based classifier) with sub-50ms inference latency per message.
- Supports configurable sensitivity thresholds (`--toxicity-threshold=0.85`, `--spam-score-limit=0.7`) and auto-escalation to human moderators via webhook.
- Logs all flagged interactions to PostgreSQL with millisecond-precision timestamps for audit trails.

### 2. Fan Sentiment & Engagement Heatmap
- Aggregates interaction data (comments, reactions, shares) into a dynamic heatmap engine using Redis-backed time-series storage.
- Exposes a REST API endpoint `/api/v1/heatmap?interval=1h&fan_id=<id>` returning JSON with fields: `engagement_score`, `sentiment_magnitude`, `peak_timestamp_utc`.
- Supports cross-referencing with external CRM data via configurable webhook integration.

### 3. Automated Community Rule Enforcement Engine
- Parses custom rule sets defined in YAML (e.g., `rule: "ban_user_if_violation_count > 3 within 24h"`) and executes actions (mute, warn, ban) with atomic rollback capability.
- Uses a distributed lock manager (etcd) to prevent race conditions in high-concurrency environments (10k+ concurrent users).
- Generates compliance reports in PDF/CSV via `--export-format=csv --report-period=weekly`.

## How to Use

### Step 1: Initialize the FanGuard Shield
```bash
# Set environment variables for PostgreSQL and NLP model path
export FANGUARD_DB_URI="postgresql://user:pass@localhost:5432/fanguard"
export FANGUARD_MODEL_PATH="/models/bert-toxicity-v2"

# Start the shield service with custom thresholds
fanguard shield start --toxicity-threshold=0.80 --spam-score-limit=0.65 --log-level=debug
```
Verify service health: `curl http://localhost:8080/health` expects `{"status":"ok","uptime_seconds":1234}`.

### Step 2: Configure and Run Engagement Heatmap
```yaml
# heatmap_config.yaml
interval: 1h
fan_id: "fan_001"
data_sources:
  - type: "postgres"
    table: "fan_interactions"
  - type: "webhook"
    url: "https://crm.example.com/events"
```
Execute: `fanguard heatmap --config ./heatmap_config.yaml --output-format=json`  
Output sample: `{"fan_id":"fan_001","engagement_score":87.3,"peak_timestamp_utc":"2025-03-21T14:30:00Z"}`

### Step 3: Deploy Custom Rule Enforcement
```yaml
# rules.yaml
rules:
  - name: "spam_ban"
    condition: "violation_count > 3 AND time_window_hours = 24"
    action: "ban"
    rollback: true
```
Apply: `fanguard rules apply --rules-file ./rules.yaml --dry-run=false`  
Monitor enforcement logs: `tail -f /var/log/fanguard/enforcement.log` (shows `action_taken`, `fan_id`, `timestamp`).

## Best Practices

### 1. Use Staged Rollout for Rule Changes
- Always test new enforcement rules in a staging environment with a small fan subset (e.g., 5% of total users) before global deployment. Use the `--dry-run=true` flag to preview actions without committing bans or mutes. This prevents false positives from damaging community trust.

### 2. Combine Sentiment Heatmap with Manual Review Cycles
- Schedule weekly manual reviews of high-engagement spikes (e.g., `engagement_score > 90`) flagged by the heatmap. Pair this with automated alerts via Slack/Teams webhooks for critical thresholds. This hybrid approach balances algorithmic efficiency with human judgment, especially during viral events.
