# deployguard - DEP_022

## Overview

DeployGuard is a high-performance, zero-overhead static analysis engine purpose-built for solo developers and small teams who need production-grade code security auditing without the enterprise tax. It integrates directly into CI/CD pipelines, scanning for 200+ vulnerability patterns across Python, JavaScript, Rust, and Go—all with sub-200ms per-file analysis latency.

## Core Features

### 1. Real-Time Dependency Graph & Supply Chain Attack Detection
DeployGuard constructs a full dependency resolution graph at commit time, cross-referencing each package against a curated database of 1.2M+ known malicious package versions. It flags typosquatting, dependency confusion, and manifest tampering with a detection accuracy of 99.7% (validated against OSS-Fuzz datasets). The engine supports `npm`, `pip`, `cargo`, and `go mod` lockfiles, and emits a structured JSON report with exact CVE identifiers and remediation commit hashes.

### 2. Multi-Language Taint Analysis with Call-Graph Tracing
Built on top of a custom AST walker, DeployGuard performs inter-procedural taint propagation across function boundaries. It tracks unsanitized user input from entry points (e.g., HTTP handlers, file reads) to sinks (e.g., `exec()`, `eval()`, SQL queries). The analyzer supports 15+ sink categories and outputs a ranked severity score (0.0–10.0) based on exploitability and reachability. Average analysis time per 10k LOC is under 1.2 seconds on a standard GitHub Actions runner.

### 3. Pre-Commit Hook & CI Gate with Configurable Thresholds
DeployGuard ships with a native Git pre-commit hook that enforces a minimum security score before allowing commits. In CI mode (GitHub Actions, GitLab CI, Jenkins), it can block PR merges if any critical-severity issue is detected. Configuration is managed via a YAML file (`deployguard.yml`) with parameters such as `min_score: 8.5`, `fail_on_critical: true`, and `exclude_patterns: ["tests/*", "vendor/*"]`. The engine also supports incremental scanning—only re-analyzing changed files—reducing scan time by up to 70% on subsequent runs.

## How to Use

### Step 1: Initialize DeployGuard in Your Repository
Run `deployguard init` from the project root. This generates a `.deployguard/` directory containing the default configuration (`deployguard.yml`) and a local rule cache. Optionally, pass `--language=python,rust` to restrict analysis to specific languages. The init command also installs the Git pre-commit hook automatically.

### Step 2: Run a Full Security Audit
Execute `deployguard scan --path ./src --output report.json --threshold 8.0`. The scanner will parse all source files, build the dependency graph, and run taint analysis. Results are written to `report.json` with the following structure:
```json
{
  "score": 9.2,
  "issues": [
    {
      "file": "src/handler.py",
      "line": 47,
      "severity": "critical",
      "cve": "CVE-2024-1234",
      "suggestion": "Use `subprocess.run()` with shell=False"
    }
  ]
}
```
Use `--format=sarif` for GitHub Code Scanning integration.

### Step 3: Enforce Security Gates in CI
Add the following step to your GitHub Actions workflow:
```yaml
- name: Run DeployGuard
  uses: deployguard/action@v2
  with:
    min-score: '8.5'
    fail-on-critical: 'true'
    exclude: 'tests/*'
```
On failure, the action outputs a summary table to the job log and posts a PR comment with a breakdown of all blocking issues. To bypass the gate for a specific commit, include `[deployguard-skip]` in the commit message.

## Best Practices

### Practice 1: Pair DeployGuard with a Secret Scanner for Full Coverage
DeployGuard focuses on code-level vulnerabilities (injection, XSS, path traversal), but does not scan for hardcoded secrets. Integrate a complementary tool like `trufflehog` or `git-secrets` in the same CI pipeline. Configure DeployGuard to ignore `.env` files via `deployguard.yml` to avoid false positives, and let the secret scanner handle credential leaks.

### Practice 2: Use Incremental Scanning in Large Monorepos
For repositories exceeding 100k LOC, enable incremental mode by setting `scan_mode: incremental` in `deployguard.yml`. This leverages a local SQLite cache (`~/.deployguard/cache.db`) to store file hashes and previous analysis results. Only files with changed hashes are re-scanned, cutting total CI time from 45 seconds to under 12 seconds on average. Reset the cache with `deployguard cache --clear` after a major dependency update.
