# deployguard - DEP_009

## Overview

DeployGuard is a hardened static analysis engine designed for solo developers and small teams who ship code without a dedicated security review pipeline. It integrates directly into CI/CD workflows and local pre-commit hooks, scanning for credential leaks, misconfigured IAM permissions, and dependency vulnerabilities before deployment reaches production.

## Detailed Content

### What is DeployGuard?

DeployGuard is not just a linter—it is a deployment gate. It parses AST-level structures across Python, JavaScript, Go, and YAML to detect secrets (API keys, tokens, private keys) using entropy-based pattern matching and regex heuristics, validates cloud infrastructure templates (Terraform, CloudFormation) against OWASP Top 10 misconfigurations, and cross-references your `requirements.txt` / `package-lock.json` against the National Vulnerability Database (NVD) with a resolution of **≤ 24 hours from CVE publication**.

### Core Features

- **Secret Leak Detection with Contextual Whitelisting**  
  Scans every committed file for high-entropy strings (e.g., `ghp_xxxxxxxxxxxxxxxxxxxx`), AWS access keys (`AKIA...`), and private SSH keys. The engine supports custom regex patterns and a `deployguard.ignore` file for false-positive suppression. Each detection includes **line number, commit hash, and risk severity score (0–100)** based on entropy value and surrounding context.

- **Infrastructure-as-Code (IaC) Policy Enforcement**  
  Parses Terraform `*.tf` and CloudFormation `*.yaml` files, flagging S3 buckets with `public-read` ACLs, overly permissive security group rules (e.g., `0.0.0.0/0` on port 22), and unencrypted EBS volumes. Policies are defined in a `deployguard.policy.yaml` file with **threshold parameters like `max_open_ports: 2`** and `require_encryption: true`. Results are output as JSON or SARIF for ingestion into GitHub Security Tab.

- **Real-Time Dependency Vulnerability Mapping**  
  Queries a local SQLite cache (updated via `deployguard update-db`) that mirrors the NVD API. For each dependency, it reports **CVE ID, CVSS v3 score, affected version range, and a remediation suggestion** (e.g., `upgrade requests from 2.25.0 to 2.31.0`). The scan completes in **under 500ms for a 200-dependency project**.

### How to Use

1. **Initialize the Project Configuration**  
   Run `deployguard init --project my-app --languages py,js,go --iac terraform` in your repository root. This generates a `deployguard.yaml` file with default severity thresholds (`severity_threshold: 70` for secrets, `cvss_threshold: 7.0` for vulnerabilities) and a `.deployguard-ignore` file. You can then edit the YAML to add custom secret patterns or override policy rules.

2. **Execute a Pre-Commit Scan**  
   Add `deployguard scan --path . --output json --fail-on critical` to your `.git/hooks/pre-commit` script. The tool will scan only staged files (using `git diff --cached --name-only`), report findings with **file path, line number, and suggested fix**, and exit with code `1` if any critical-severity issue is found. For example, a detected AWS secret will output: `[CRITICAL] secret/aws_access_key: found in src/config.py:42 - revoke key immediately`.

3. **Integrate into CI Pipeline (GitHub Actions Example)**  
   Create a `.github/workflows/deployguard.yml` file with the following step:  
   ```yaml
   - name: Run DeployGuard
     run: |
       deployguard scan --path . --output sarif --severity-threshold 50 --cvss-threshold 5.0
       deployguard report --format markdown >> $GITHUB_STEP_SUMMARY
   ```  
   This will fail the build if any secret with entropy > 50 or any dependency with CVSS ≥ 5.0 is detected, and append a formatted report to the Actions summary page.

### Best Practices

- **Tune Severity Thresholds Per Environment**  
  For development branches, set `severity_threshold: 30` and `cvss_threshold: 4.0` to catch low-hanging fruit early. For `main` or `release` branches, raise the bar to `severity_threshold: 70` and `cvss_threshold: 7.0` to block only critical issues. This prevents developer burnout while maintaining production security.

- **Combine with Secret Rotation Automation**  
  After DeployGuard flags a leaked credential, do not simply delete the commit—trigger a secret rotation script via webhook. For example, if an AWS key is detected, call the AWS Secrets Manager API to rotate the key immediately. Pair DeployGuard’s `--output json` with a custom script that parses the findings and executes `aws secretsmanager rotate-secret --secret-id <arn>`.
