# deployguard - DEP_014

## Overview

DeployGuard is a pre-commit static analysis engine purpose-built for solo developers and small teams who ship code without a dedicated security reviewer. It enforces code security audit rules directly in your local CI pipeline, catching misconfigurations, hardcoded secrets, and dependency vulnerabilities before they reach production.

## Core Features

### 1. `--audit-level` with Differential Secret Scanning

DeployGuard’s secret scanner operates at three audit levels (`strict`, `standard`, `permissive`). In `strict` mode, it performs a multi-pass entropy analysis on every changed line, flagging any string with a Shannon entropy score above `4.2` and a length of 16+ characters. The scanner cross-references results against a local PostgreSQL cache of previously verified false positives (e.g., test API keys, documentation examples) to reduce noise by up to 40%.

### 2. `--dep-check` with Real-Time CVE Matching

When you run `deployguard check --dep-check`, the tool parses your `package-lock.json`, `requirements.txt`, or `Gemfile.lock` and compares each resolved version against a locally mirrored NVD (National Vulnerability Database) feed. It flags any dependency with a CVSS v3 score ≥ 7.0 and a known exploit in the wild. The entire scan completes in under 200 milliseconds for a typical Node.js project with 500+ dependencies.

### 3. `--policy-file` for Custom Rule Enforcement

Define your own security policies in a YAML file (e.g., `.deployguard.yml`). You can specify rules like "block any commit that contains a `DATABASE_URL` environment variable assignment" or "reject code that uses `eval()` in JavaScript files." DeployGuard evaluates these rules as AST-level patterns, not simple regex, so it won't trigger false positives on variable names like `evaluate`.

## How to Use

### Step 1: Initialize DeployGuard in Your Repository

Run `deployguard init --repo-type node --audit-level standard` from the root of your project. This command generates a `.deployguard.yml` configuration file, sets up a pre-commit hook (via `.git/hooks/pre-commit`), and downloads the latest NVD dependency cache. The hook will automatically run on every `git commit`.

### Step 2: Run a Manual Audit with Custom Policy

Execute `deployguard check --policy-file .deployguard.yml --output-format json --fail-on critical`. This scans all staged files, applies your custom rules, and outputs results as a JSON array. If any critical-severity issue is found (e.g., a hardcoded AWS access key with a Shannon entropy > 4.5), the command exits with code 1, blocking the commit.

### Step 3: Review and Suppress False Positives

After a scan, run `deployguard suppress --hash <sha256-of-flagged-line> --reason "test credential"` to add the flagged line to your local false-positive database. The suppression is stored in `.deployguard/cache.db` (SQLite). To share this across your team, push the cache file to a shared S3 bucket or include it in your repo under `.deployguard/`.

## Best Practices

### Practice 1: Combine `--audit-level strict` with Pre-Commit Hooks

For solo developers working on open-source projects, always enable `strict` mode in your pre-commit hook. This ensures that even a single hardcoded token (e.g., a GitHub personal access token) is caught before you accidentally push it to a public repository. Pair this with a `.gitignore` entry for `.deployguard/cache.db` to avoid committing your suppression list.

### Practice 2: Schedule Weekly Dependency Refresh

Run `deployguard update --cve-db` every Monday morning via a cron job or GitHub Actions scheduled workflow. This refreshes your local NVD cache with the latest CVEs. Without this, DeployGuard’s `--dep-check` may miss newly disclosed vulnerabilities that were published after your last scan.
