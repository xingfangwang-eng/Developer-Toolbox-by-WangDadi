# deployguard - DEP_004

## Overview

DeployGuard is a hardened static analysis engine purpose-built for solo developers and small teams who need enterprise-grade code security auditing without the overhead of a full CI pipeline. It analyzes source code for injection vulnerabilities, hardcoded secrets, and misconfigured dependency trees, outputting actionable, severity-ranked reports. This release (DEP_004) introduces multi-language bytecode-level inspection and zero-config baseline enforcement.

## Core Features

- **Secret Leakage Detection with Entropy Scoring**  
  Scans for API keys, tokens, and private keys using a hybrid regex + Shannon entropy model. Each candidate is assigned a confidence score (0.0–1.0), with a configurable threshold (default: `0.85`). Supports `.env`, `.yaml`, `.json`, and source files across Python, JavaScript, Go, and Rust. False positives are reduced by cross-referencing against a built-in allowlist of 200+ known test keys and placeholder values.

- **Dependency Graph Integrity Check**  
  Resolves the full transitive dependency tree (via `package-lock.json`, `go.sum`, `Cargo.lock`, or `requirements.txt`) and compares each package hash against a local cache of verified SHA-256 digests. Any mismatch or missing hash triggers a `HIGH` severity alert. The check completes in under 500ms for projects with up to 1,000 direct dependencies, using a streaming parser that avoids loading the entire lock file into memory.

- **Taint Analysis for SQL Injection & Command Injection**  
  Performs intra-procedural taint tracking on Python and JavaScript codebases. It traces data flow from user input sources (e.g., `request.get()`, `input()`, `sys.argv`) to sinks like `exec()`, `subprocess.call()`, or raw SQL queries. Each taint path is recorded with the exact line number and variable name. Analysis time scales linearly with AST node count, averaging 2.1ms per 1,000 nodes on a modern CPU.

## How to Use

1. **Initialize the baseline configuration**  
   Run `deployguard init --lang=python --threshold=0.85 --output-dir=.deployguard` in your project root. This generates a `.deployguard/config.yaml` file with default severity rules and an allowlist for known test keys. You can edit this file to add custom patterns or adjust the entropy threshold per language.

2. **Execute a full security audit**  
   Use `deployguard scan --path=./src --format=json --report=report.json`. The scanner walks all source files, skipping binary and vendor directories by default. It outputs a JSON report containing each finding with: file path, line number, vulnerability type, severity (`LOW`, `MEDIUM`, `HIGH`), and a confidence score. For taint analysis results, the report includes the full data flow trace as a list of `{ source_line, sink_line, variable }` tuples.

3. **Generate a human-readable summary with remediation hints**  
   Run `deployguard report --input=report.json --output=summary.md --include-remediation`. This creates a Markdown file that groups findings by severity and provides a one-liner remediation suggestion for each (e.g., "Use parameterized queries instead of string concatenation at line 142 in `app.py`"). The summary also includes a histogram of vulnerability counts per module and a "clean bill of health" stamp if zero `HIGH` findings exist.

## Best Practices

- **Integrate DeployGuard into your pre-commit hook**  
  Add `deployguard scan --path=. --format=json --report=/tmp/dg-report.json && deployguard report --input=/tmp/dg-report.json --output=/dev/stdout --fail-on=HIGH` as a pre-commit hook. This prevents any commit that introduces a `HIGH` severity vulnerability, enforcing a security gate at the earliest possible point in the development workflow. For solo developers, this catches mistakes before they reach a remote repository.

- **Run a weekly baseline drift check against your dependency allowlist**  
  Schedule `deployguard audit --baseline=.deployguard/baseline.json --diff=1w` in your CI (or as a cron job) to detect when a dependency hash changes without a version bump. This catches supply-chain attacks where a package is silently updated with malicious code. Store the baseline JSON file in version control and review any diff before merging.
