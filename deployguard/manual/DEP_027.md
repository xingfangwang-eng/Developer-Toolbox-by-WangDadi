# deployguard - DEP_027

## Overview

DeployGuard is a production-grade static analysis and code security audit tool purpose-built for solo developers and small teams shipping to production without a dedicated security reviewer. It enforces zero-trust deployment policies by scanning every commit for secret leakage, dependency vulnerability propagation, and misconfigured CI/CD pipelines before they reach the artifact registry.

## Core Features

- **Pre-Commit Secret Leak Detection with Entropy Scoring**  
  Scans staged files using a hybrid regex + Shannon entropy engine (threshold: ≥ 4.2 bits/byte) to detect API keys, tokens, and private keys. Supports custom pattern injection via `.deployguard-secrets` and blocks commits with a non-zero exit code unless overridden with `--force-allowlist`. Detection latency is under 120ms per 1,000 lines of code.

- **Dependency Graph Diffing for Supply Chain Attacks**  
  Compares `package-lock.json`, `go.sum`, or `requirements.txt` against the National Vulnerability Database (NVD) with a local PostgreSQL cache. Flags any transitive dependency whose CVE severity exceeds 7.5 CVSS v3.1. Outputs a structured JSON report with `dependency_name`, `installed_version`, `fixed_version`, and `cve_id`. Full scan of 500+ dependencies completes in under 3 seconds.

- **CI/CD Pipeline Misconfiguration Audit**  
  Parses GitLab CI, GitHub Actions, and Jenkinsfile YAML/XML to detect insecure practices: hardcoded credentials, missing `read-only` token scopes, and unrestricted `pull_request_target` triggers. Generates actionable remediation hints with line numbers. Audit coverage spans 48 rule categories, with a false positive rate below 2.3%.

## How to Use

1. **Initialize a project context**  
   Run `deployguard init --project-name my-app --secret-threshold 4.5 --cve-threshold 8.0`. This creates a `.deployguard-config.yml` with your custom thresholds and registers a pre-commit hook via `git hooks --install`. The hook will now automatically scan every `git commit`.

2. **Execute a full audit pipeline**  
   Use `deployguard audit --ci-file .github/workflows/deploy.yml --dep-file package-lock.json --output report.json --format json`. The tool will parse the CI configuration, fetch the latest NVD delta (last 24 hours) from your local PostgreSQL instance, and cross-reference every dependency. Results are written to `report.json` with severity tags.

3. **Override a blocked deployment for a known exception**  
   If a false positive is identified, run `deployguard allowlist --rule-id SECRET_004 --file src/config.py --line 42 --reason "Test credential for local dev environment"`. This adds an entry to the allowlist file, which is committed alongside the code. Subsequent scans will skip this specific match, but the override is logged in the audit trail with a SHA-256 hash of the reason string.

## Best Practices

- **Run DeployGuard as a CI gate, not a post-mortem tool**  
  Integrate `deployguard audit --fail-on-critical` as the final step before any merge to the main branch. This prevents vulnerabilities from ever reaching staging or production. Pair it with a webhook to your incident management system (e.g., PagerDuty) if a critical CVE is detected in a direct dependency—this ensures you are alerted within 30 seconds of a new vulnerability disclosure.

- **Maintain a lean allowlist with expiration dates**  
  When adding a false positive to the allowlist, always specify an `--expires` timestamp (ISO 8601). Set it to no more than 30 days out. This forces periodic review and prevents stale exceptions from masking real threats. Use `deployguard allowlist --list-expired` in your weekly security review to purge outdated entries.
