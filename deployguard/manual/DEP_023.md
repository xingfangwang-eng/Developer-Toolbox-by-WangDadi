# deployguard - DEP_023

## Overview

DeployGuard is a hardened static analysis engine purpose-built for solo developers and small teams who ship code without a dedicated security layer. It fuses **control-flow integrity checks**, **secrets regex scanning**, and **supply-chain dependency risk scoring** into a single, offline-first CLI tool that runs in under 1200 ms per commit.

## Core Features

### 1. Control-Flow Integrity (CFI) Audit Engine
- Enforces **call-graph whitelisting** via a `.guardflow.yml` config file.  
- Detects indirect jumps, ROP gadget patterns, and abnormal stack unwinding in compiled binaries (PE, ELF, Mach-O).  
- Emits a **CFI score (0–100)** per function, with a configurable threshold (`--cfi-threshold=85`).  
- Supports **multi-arch analysis** (x86_64, ARM64, RISC-V) with per-arch signature databases.

### 2. Deep Secrets & Credential Leak Scanner
- Scans source code and commit history with **regex-based and entropy-based detectors** (AWS keys, GitHub tokens, JWT, RSA private keys).  
- Reports false-positive probability (`--fp-prob <0.05`) and highlights the exact line + surrounding context.  
- Integrates with **PostgreSQL** to store historical scan results (`scan_id`, `sha256`, `risk_level`, `timestamp`).  
- Outputs a **risk heatmap** in JSON and HTML formats.

### 3. Supply-Chain Dependency Risk Profiler
- Resolves dependency trees from `package.json`, `Cargo.toml`, `go.mod`, and `requirements.txt`.  
- Assigns a **supply-chain risk score (0–1)** based on:  
  - Package age, maintainer count, known CVEs, deprecation status, and transitive dependency depth.  
- Flags packages with `--risk-threshold=0.7` and auto-generates an SBOM in CycloneDX format.  
- Caches resolved metadata locally (SQLite) to avoid redundant network calls.

## How to Use

### Step 1: Initialize a New Guard Profile
```bash
deployguard init --project-name "my-app" --lang python --cfi-threshold 85
```
This creates a `.deployguard/` directory with a default `config.yaml` and an `.ignore` file. The CLI auto-detects your runtime and sets the appropriate analysis pipeline.

### Step 2: Run a Full Audit Against the Current Commit
```bash
deployguard scan --path ./src --output-format json --secrets-depth 3 --supply-chain-depth 2
```
- `--secrets-depth`: Number of Git history commits to scan (default: 1).  
- `--supply-chain-depth`: Levels of transitive dependencies to resolve (default: 1).  
- Output is written to `deployguard_report.json`. You can also pipe to `--output-format html` for a visual dashboard.

### Step 3: Review and Suppress Known False Positives
```bash
deployguard suppress --scan-id "DEP_023_20250321" --fp-id "FP-42" --reason "test fixture key"
```
This marks a specific false positive in the PostgreSQL history. Future scans will skip this pattern unless `--include-suppressed` is passed. Use `deployguard dashboard` to open an interactive web UI that visualizes risk trends over time.

## Best Practices

1. **Run `deployguard scan` as a pre-commit hook.**  
   Add it to your `.git/hooks/pre-commit` with `--fail-on-risk=0.8`. This prevents any commit with a supply-chain risk score above 0.8 or a CFI score below 80 from being committed. Pair it with `--secrets-depth=5` to catch leaked credentials before they hit the remote.

2. **Use the PostgreSQL integration for team-wide audit trails.**  
   Set `database.url` in `config.yaml` to your PostgreSQL instance. This enables cross-commit trend analysis, false-positive suppression sharing, and SBOM versioning. Schedule a weekly `deployguard scan --full-history` via cron to catch regressions in old dependencies.
