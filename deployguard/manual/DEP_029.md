# deployguard - DEP_029

## Overview

DeployGuard is a production-grade static analysis engine purpose-built for solo developers and small teams who ship code without a dedicated security audit pipeline. It combines AST-level vulnerability scanning, dependency graph traversal, and runtime policy enforcement into a single CLI binary with zero external runtime dependencies.

## Detailed Content

### What is DeployGuard?

DeployGuard is not a linter. It is a pre-deployment gate that enforces a hardened security posture by analyzing your codebase against a customizable rule matrix. It detects hardcoded secrets, unsanitized SQL concatenation, insecure deserialization patterns, and supply-chain risks in your dependency tree — all before a single commit reaches your staging environment.

### Core Features

- **Context-Aware Secret Leak Detection** — Scans for 50+ credential patterns (AWS keys, JWT tokens, database connection strings) using entropy analysis plus regex anchors. Integrates with `.gitignore` and custom blocklists. Reports false-positive probability per finding. Supports `--min-entropy 4.5` threshold tuning.

- **Dependency Graph Integrity Checker** — Parses `package-lock.json`, `go.sum`, `Cargo.lock`, and `requirements.txt` to compute a transitive closure. Compares each dependency hash against the Sonatype OSS Index API (offline fallback via local SHA-256 cache). Flags packages with known CVEs, deprecated licenses, or mismatched integrity hashes. Outputs a risk score between 0.0 and 1.0 with per-package breakdown.

- **Runtime Policy Engine with Pre-Commit Hook** — Define policies in YAML (e.g., `block: true` for any file containing `eval()` in Node.js or `exec()` in Python). DeployGuard runs as a Git pre-commit hook, rejecting commits that violate policy. Supports `--fail-on critical` to abort the pipeline if any critical-severity finding is detected. Execution time is measured in milliseconds per 1000 LOC.

### How to Use

1. **Initialize a project with a baseline configuration**  
   Run `deployguard init --project-name myapp --language python --severity-threshold high`. This generates a `.deployguard.yml` file in the project root containing default rules for Python, including banned function calls (`os.system`, `subprocess.Popen` with `shell=True`), and a secret scan pattern set. The command also creates a `.deployguard-ignore` file where you can whitelist known false positives.

2. **Execute a full scan with JSON output for CI integration**  
   Run `deployguard scan --path ./src --format json --output report.json --fail-on critical`. The engine traverses `./src` recursively, parses all `.py` files into an AST, and cross-references every string literal against the secret dictionary. Simultaneously, it resolves `requirements.txt` against the local hash cache. The JSON report includes `file`, `line`, `column`, `rule_id`, `severity`, `message`, and `recommended_fix`. If any critical finding exists, the process exits with code 1 — suitable for blocking a GitHub Actions or GitLab CI pipeline step.

3. **Register a pre-commit hook for local enforcement**  
   Run `deployguard install-hook --hook-type pre-commit`. This writes a `.git/hooks/pre-commit` script that invokes `deployguard scan --path . --fail-on high --quiet`. If the scan fails, the commit is aborted and the developer sees a colorized diff of the offending lines. To bypass (e.g., for a known false positive), use `git commit --no-verify`. The hook execution adds less than 200ms overhead per 10,000 LOC.

### Best Practices

- **Run `deployguard scan` as the final step in your CI pipeline, after all tests pass.** This ensures that security gates do not block legitimate code changes during development, but still catch issues before deployment. Use `--fail-on high` for CI and `--fail-on critical` for production deployments.

- **Maintain a living `.deployguard-ignore` file with documented justifications.** For each ignored finding, add a comment explaining why it is a false positive (e.g., `# This is a test fixture for SQL injection`). Periodically run `deployguard audit-ignores` to detect patterns that are no longer present in the codebase and prune stale entries.
