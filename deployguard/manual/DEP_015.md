# deployguard - DEP_015

## Overview

DeployGuard is a lightweight, offline-capable static analysis engine engineered for solo developers and small teams who need production-grade code security audits without the overhead of CI/CD pipeline integration. It combines taint tracking, dependency graph traversal, and runtime behavior simulation to surface vulnerabilities before they reach staging.

## Core Features

### 1. Recursive Taint Propagation with Call‑Chain Visualization
DeployGuard performs intra‑ and inter‑procedural taint analysis across JavaScript, Python, and Go codebases. It traces user‑controlled input through closures, async/await chains, and nested callbacks, flagging unsanitized sinks (e.g., `exec()`, `eval()`, raw SQL queries). The engine outputs a JSON‑formatted call‑chain graph with per‑node confidence scores (0.0 – 1.0) and highlights the exact source line and column for each tainted path.

- **Parameter**: `--taint-depth` (default: 5, max: 20) controls recursion depth.
- **Metric**: Average analysis time per 10k LOC: **120 ms** (single‑threaded, M1 MacBook Pro).

### 2. Dependency Graph Integrity Checker
DeployGuard parses `package-lock.json`, `yarn.lock`, `go.sum`, and `requirements.txt` to build a full dependency tree. It cross‑references each package version against a locally cached vulnerability database (updated via `deployguard db sync --source nvd`), then reports direct and transitive vulnerabilities with CVSS 3.1 scores. The tool also detects version‑range collisions and suggests pinned resolutions.

- **Parameter**: `--severity-threshold` (low|medium|high|critical) filters output.
- **Metric**: Database sync completes in under **3 seconds** (20k+ CVE entries).

### 3. Runtime Behavior Simulation Sandbox
For interpreted languages, DeployGuard can execute a subset of the target code inside a restricted WASM sandbox to observe actual runtime behavior—variable assignments, type coercions, and dynamic property access. This simulation catches obfuscated injection points that static analysis alone would miss (e.g., dynamically constructed `require()` paths or `__proto__` pollution).

- **Parameter**: `--sandbox-timeout` (default: 5000 ms) limits execution per file.
- **Metric**: Sandbox overhead: **~15 ms** per 1,000 AST nodes.

## How to Use

### Step 1: Initialize the Project Audit
```bash
deployguard init --project-dir ./my-app --language python --taint-depth 8
```
This creates a `.deployguard/config.yaml` file with default rulesets for Python (Flask/Django sinks). The tool scans the directory structure and generates a preliminary AST index. If the project exceeds 50,000 LOC, it automatically enables chunked processing (256 KB per chunk) to keep memory under 200 MB.

### Step 2: Run a Full Static Scan with Sandbox Simulation
```bash
deployguard scan --config .deployguard/config.yaml --sandbox-enable --severity-threshold high
```
DeployGuard begins recursive taint propagation (depth 8) and simultaneously loads the dependency graph. After the static phase, it selects the top 20 most suspicious code paths and executes them inside the WASM sandbox. The output is a `report.json` containing:
- `vulnerabilities[]` – each with `source`, `sink`, `confidence`, `simulated` (boolean), and `runtime_evidence` (if sandbox matched an injection).

### Step 3: Review and Export the Audit Report
```bash
deployguard report --format html --output ./reports/audit-$(date +%Y%m%d).html
```
The HTML report includes an interactive call‑chain graph (SVG with clickable nodes), a dependency tree with CVSS badges, and a sandbox execution log. Use `--format sarif` for CI integration or `--format json` for automated ingestion. The report also lists all ignored rules (via `--skip-rule` in config) for transparency.

## Best Practices

### 1. Schedule Incremental Scans During Development
Instead of a full scan on every commit, configure DeployGuard to run only on changed files using `deployguard scan --incremental --diff HEAD~1`. This reduces scan time for a 100k LOC project from ~1.2 seconds to under **250 ms**. Pair this with a pre‑commit hook to block commits that introduce new high‑severity taint paths.

### 2. Combine Sandbox Simulation with Fuzz‑Generated Inputs
For maximum coverage, run DeployGuard with `--sandbox-enable` and feed it a set of fuzz‑generated payloads (e.g., via `deployguard fuzz --input-dir ./fuzz_corpus`). The sandbox will execute the target code with each payload and record any unexpected type changes or function calls. This hybrid approach catches logic‑based vulnerabilities that static taint analysis alone cannot detect—such as prototype pollution via crafted JSON keys.
