# deployguard - DEP_007

## Overview

DeployGuard is a pre-deployment static analysis and code security audit engine purpose-built for solo developers and small teams. It scans your codebase for misconfigurations, credential leaks, and dependency vulnerabilities before they reach production.

## Detailed Content

### What is DeployGuard?

DeployGuard is a lightweight, CLI-first security gate that integrates directly into your CI/CD pipeline. Unlike generic linters, it focuses exclusively on deployment-level risk detection: hardcoded API keys, insecure Dockerfile configurations, and misconfigured IAM policies. It outputs actionable, severity-ranked reports in JSON or SARIF format, enabling automated rollback triggers.

### Core Features

- **Pre-Commit Credential Sniffer**: Scans staged files for regex-matched secrets (AWS keys, GitHub tokens, database URLs) with configurable false-positive suppression via `.guardignore`. Supports custom regex patterns and entropy-based detection. Reports include line number, surrounding context, and a risk score (0–100).
- **Dependency Graph Exploit Analyzer**: Cross-references your `package-lock.json`, `requirements.txt`, or `go.sum` against the NVD (National Vulnerability Database) API in real-time. Flags transitive dependencies with known CVEs, and calculates a blast radius score based on the dependency tree depth. Cache is stored locally with a configurable TTL (default: 24 hours).
- **Dockerfile & Compose Policy Enforcer**: Validates Dockerfiles against a custom rule set (e.g., no `latest` tags, no `ADD` from untrusted URLs, no exposed root user). For Docker Compose, checks network mode isolation and volume mount permissions. Outputs a pass/fail per rule with a detailed diff suggestion.

### How to Use

1. **Initialize a new project scan**:  
   Run `deployguard init --project-name my-app --scan-depth full --output-format json`. This generates a `.deployguard.yml` config file in the project root, sets up the default rule set, and performs an initial full scan. Results are written to `deployguard-report.json` with timestamps in UTC.

2. **Integrate into your pre-commit hook**:  
   Add `deployguard pre-commit --fail-on high --timeout 15s` to your `.git/hooks/pre-commit` file. This runs the credential sniffer and dependency analyzer only on staged files, and exits with code 1 if any high-severity issue is found. Use `--skip-dependency` to speed up the hook when only config files changed.

3. **Automate rollback in CI pipeline**:  
   In your GitHub Actions or GitLab CI, add a step after the build: `deployguard ci --compare-to main --severity-threshold critical --block-deploy`. This compares the current branch’s scan results against the `main` branch baseline, and if new critical issues are introduced, the pipeline fails and triggers a rollback via webhook (configured in `.deployguard.yml` under `rollback.webhook_url`).

### Best Practices

- **Always run DeployGuard before merging to main**: Even if you use branch protection rules, a pre-merge scan catches issues that might slip through code review. Set `--fail-on medium` for merge pipelines to enforce a higher baseline.
- **Maintain a baseline report for each major release**: Run `deployguard baseline --tag v1.0.0` after a stable release. This snapshot allows `deployguard ci --compare-to` to accurately detect regressions in subsequent commits, reducing false alarms from pre-existing vulnerabilities.
