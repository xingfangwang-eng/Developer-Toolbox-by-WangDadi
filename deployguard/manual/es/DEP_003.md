# deployguard - DEP_003

## Resumen

DeployGuard es una herramienta de auditoría de seguridad de código diseñada específicamente para desarrolladores independientes (solo devs) que necesitan análisis estático profundo sin depender de infraestructura cloud compleja. Escanea vulnerabilidades, fugas de secretos y desviaciones de conformidad en tiempo real, con soporte nativo para más de 15 lenguajes y frameworks.

## Contenido Detallado

### ¿Qué es DeployGuard?

DeployGuard es un analizador estático de código fuente que ejecuta reglas de seguridad personalizables en pipelines CI/CD locales o remotos. Detecta inyecciones SQL, hardcodeo de credenciales, configuraciones inseguras de Docker y dependencias con CVEs conocidas, todo en menos de 200 ms por archivo. Integra con PostgreSQL para almacenar historial de auditorías y generar reportes comparativos entre releases.

### Características Principales

- **Escaneo de Secretos en Tiempo Real con Regex y ML**  
  Detecta tokens de AWS, GitHub, Stripe, y más de 50 proveedores mediante expresiones regulares predefinidas y un modelo ligero de machine learning (entrenado con 10,000+ muestras). Emite alertas en milisegundos con el contexto exacto de línea y archivo.

- **Análisis de Dependencias con Base de Vulnerabilidades Local**  
  Compara tu `package.json`, `requirements.txt` o `go.mod` contra una base de datos SQLite actualizada diariamente (más de 200,000 CVEs indexadas). Genera un score de riesgo ponderado (CRITICAL: 9.0–10.0, HIGH: 7.0–8.9, MEDIUM: 4.0–6.9, LOW: 0.1–3.9) y sugiere versiones parcheadas automáticamente.

- **Validación de Conformidad con Políticas Personalizadas**  
  Define reglas YAML/JSON para estándares como OWASP Top 10, PCI DSS o GDPR. DeployGuard verifica cada commit contra estas políticas y bloquea el merge si se supera el umbral configurable (por defecto: 5 violaciones HIGH). El resultado se exporta como reporte PDF con hash SHA-256 para auditoría legal.

### Cómo Usar

1. **Configurar el Archivo de Reglas Personalizadas**  
   Crea un archivo `deployguard-rules.yaml` en la raíz de tu repositorio. Define, por ejemplo, una regla para prohibir el uso de `eval()` en JavaScript:
   ```yaml
   rules:
     - id: JS001
       severity: HIGH
       pattern: "eval\\(.*\\)"
       languages: [javascript]
       message: "Uso de eval() detectado. Riesgo de inyección de código."
   ```
   Luego ejecuta: `deployguard scan --config deployguard-rules.yaml --path ./src`

2. **Ejecutar Auditoría Completa con Salida a PostgreSQL**  
   Conecta tu base de datos PostgreSQL configurando variables de entorno:
   ```bash
   export DEPGUARD_DB_HOST=localhost
   export DEPGUARD_DB_PORT=5432
   export DEPGUARD_DB_NAME=audit_logs
   export DEPGUARD_DB_USER=admin
   export DEPGUARD_DB_PASSWORD=secret
   ```
   Corre: `deployguard audit --output postgres --format json --threshold HIGH`. Esto inserta cada hallazgo en la tabla `audit_results` con campos como `commit_hash`, `file_path`, `line_number`, `severity`, `timestamp` (con precisión de microsegundos).

3. **Integrar en Pipeline CI/CD (GitHub Actions)**  
   Agrega este paso en tu workflow YAML:
   ```yaml
   - name: DeployGuard Security Scan
     run: |
       deployguard scan --config .deployguard/rules.yaml --path . \
         --fail-on HIGH --report-format sarif \
         --upload-to github-issues
     env:
       DEPGUARD_API_KEY: ${{ secrets.DEPGUARD_API_KEY }}
   ```
   El flag `--fail-on HIGH` hará que el pipeline falle si se encuentra alguna vulnerabilidad de severidad alta o crítica, y `--upload-to github-issues` crea automáticamente un issue en tu repositorio con los detalles.

### Mejores Prácticas

- **Establece un Umbral de Tolerancia por Entorno**  
  En desarrollo local permite violaciones MEDIUM, pero en staging y producción bloquea cualquier hallazgo HIGH o CRITICAL. Usa perfiles de configuración separados (`deployguard-dev.yaml`, `deployguard-prod.yaml`) y cámbialos según el branch en tu pipeline.

- **Programa Auditorías Programadas Semanales**  
  Ejecuta `deployguard full-audit` cada domingo a las 3:00 AM (cron job) para escanear todo el repositorio, incluyendo ramas antiguas y tags. Almacena los resultados en PostgreSQL y configura alertas por email si la cantidad de vulnerabilidades nuevas supera un 10% respecto a la semana anterior.
