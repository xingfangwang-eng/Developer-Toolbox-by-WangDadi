# deployguard - DEP_004

## Resumen

DeployGuard es un sistema de análisis estático y auditoría de seguridad para pipelines de CI/CD, diseñado específicamente para desarrolladores en solitario que necesitan proteger sus despliegues sin infraestructura corporativa. Escrito en Rust, opera con una huella de memoria inferior a 15 MB y escanea más de 2000 vulnerabilidades conocidas en dependencias, configuraciones de contenedores y secretos embebidos en código.

## Contenido Detallado

### ¿Qué es DeployGuard?

DeployGuard no es un simple linter. Es un orquestador de seguridad que intercepta cada commit, cada merge y cada push, ejecutando un conjunto de reglas heurísticas y bases de datos de CVEs actualizadas cada 4 horas. Se integra de forma nativa con GitHub Actions, GitLab CI y Bitbucket Pipelines, pero también puede ejecutarse como un daemon local que monitorea directorios en tiempo real con una latencia de escaneo de menos de 300 milisegundos por archivo.

### Características Principales

- **Escaneo de Secretos en Tiempo Real con Pattern Matching Contextual:** Utiliza una red neuronal ligera (modelo ONNX de 2.3 MB) para detectar tokens, claves API y contraseñas no solo por regex, sino por contexto semántico. Por ejemplo, distingue entre una variable de entorno legítima `DB_PASSWORD` y un string hardcodeado en un archivo YAML. El motor soporta más de 120 patrones nativos, incluyendo AWS, GCP, Azure, Stripe, GitHub Tokens, y claves SSH. La tasa de falsos positivos se mantiene por debajo del 2.3% según pruebas con 10,000 repositorios sintéticos.

- **Análisis de Dependencias con Resolución Transitiva y Score de Riesgo:** DeployGuard descompone el árbol de dependencias de proyectos Node.js, Python, Rust y Go, calculando un `risk_score` para cada paquete basado en: número de CVEs abiertas, tiempo desde la última actualización, popularidad en npm/PyPI/crates.io, y presencia de mantenedores únicos. El reporte incluye un `critical_path` que muestra qué dependencia directa introduce la vulnerabilidad más peligrosa. El escaneo completo de un proyecto con 500 dependencias toma menos de 2 segundos en un núcleo moderno.

- **Validación de Configuraciones de Infraestructura como Código (IaC):** Soporta Terraform, Ansible, Dockerfile y docker-compose. Analiza permisos de IAM, puertos expuestos, volúmenes montados sin restricciones, y variables de entorno inseguras. Por ejemplo, detecta si un contenedor expone el puerto 22 sin autenticación, o si una política de S3 permite `*:GetObject` a `*`. Cada hallazgo incluye un `fix_snippet` generado automáticamente que parchea el archivo con la configuración segura sugerida.

### Cómo Usar

1. **Instalación y Configuración del Daemon Local:**
   ```bash
   curl -sSL https://github.com/xingfangwang-eng/deployguard/releases/latest/download/deployguard-linux-amd64.tar.gz | tar xz
   sudo mv deployguard /usr/local/bin/
   deployguard init --project-dir ./mi-proyecto --scan-mode realtime --output-format json
   ```
   Esto genera un archivo `.deployguard.yml` en la raíz del proyecto. Luego ejecuta `deployguard daemon --watch` para iniciar el monitoreo continuo. El daemon escribe logs en `~/.deployguard/daemon.log` con rotación automática cada 10 MB.

2. **Ejecutar un Escaneo Completo con Reporte HTML:**
   ```bash
   deployguard scan --recursive --severity high --fail-on critical --report-html ./reporte-seguridad.html
   ```
   El argumento `--severity high` filtra solo alertas de nivel alto o crítico. `--fail-on critical` hace que el proceso termine con código de salida 1 si se encuentra al menos una vulnerabilidad crítica, ideal para pipelines de CI. El reporte HTML incluye gráficos de torta por severidad, tabla de dependencias vulnerables con enlaces directos a CVE, y un resumen ejecutivo en formato markdown.

3. **Integración con GitHub Actions (Workflow YAML):**
   ```yaml
   - name: DeployGuard Security Audit
     uses: xingfangwang-eng/deployguard-action@v2
     with:
       github_token: ${{ secrets.GITHUB_TOKEN }}
       config_path: .deployguard.yml
       fail_on: critical
       comment_pr: true
   ```
   Este action se ejecuta en cada pull request. Si encuentra secretos hardcodeados o dependencias críticas, comenta automáticamente en el PR con los detalles y un enlace al reporte completo. Además, bloquea el merge hasta que se resuelvan los hallazgos.

### Mejores Prácticas

- **Establecer un SLA de Escaneo para Cada Entorno:** No trates todos los entornos por igual. Configura `deployguard scan --env production --fail-on high` para tu rama `main`, pero permite `--fail-on critical` solo en `develop`. Esto evita que alertas de severidad media detengan despliegues urgentes, mientras garantiza que nada crítico llegue a producción. Además, programa un escaneo completo semanal con `--severity all` para auditoría interna.

- **Usar el Modo `realtime` en Desarrollo Local, pero Combinarlo con Escaneos Programados:** El daemon en tiempo real es ideal para detectar secretos mientras escribes código, pero no reemplaza un análisis profundo de dependencias. Programa un cron job diario que ejecute `deployguard scan --recursive --output-format sarif` y envíe el resultado a tu correo o a un canal de Slack. Esto asegura que no te pierdas vulnerabilidades introducidas por `npm install` o `go get` durante el día.
