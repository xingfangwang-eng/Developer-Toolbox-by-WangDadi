# deployguard - DEP_019

## Resumen

DeployGuard es un sistema de auditoría de seguridad de código estático y dinámico, diseñado específicamente para desarrolladores en solitario que necesitan garantizar la integridad de sus despliegues sin depender de infraestructuras complejas. Analiza en tiempo real las dependencias, las configuraciones de entorno y las vulnerabilidades de inyección, emitiendo alertas con una latencia inferior a 50 milisegundos por archivo analizado.

## Contenido Detallado

### ¿Qué es DeployGuard?

DeployGuard no es un simple linter. Es un orquestador de seguridad que intercepta el pipeline de CI/CD, escanea cada commit en busca de secretos expuestos (como tokens OAuth o claves SSH), valida la integridad de los binarios contra un hash SHA-256 almacenado en PostgreSQL, y bloquea automáticamente el despliegue si detecta una discrepancia. Funciona sin conexión a internet, con una base de reglas actualizable mediante parches diferenciales de 2 KB.

### Características Principales

- **Escaneo de secretos en tiempo real con detección de falsos positivos**: Emplea un motor de expresiones regulares adaptativo que, tras analizar 10.000 patrones de credenciales conocidas (AWS, GitHub, Stripe), reduce los falsos positivos al 0.3% mediante un modelo de Markov de segundo orden. Parámetro clave: `--confidence-threshold 0.97`. Almacena los resultados en una tabla particionada de PostgreSQL con índices BRIN para consultas sub-milisegundo.

- **Verificación de integridad de artefactos mediante prueba de conocimiento cero**: Cada binario o paquete npm se firma con una clave efímera ECDSA; DeployGuard verifica la firma sin revelar la clave privada, usando un protocolo de compromiso de Pedersen. El tiempo de verificación es de 12 ms para archivos de hasta 500 MB. Soporta rollback automático si la verificación falla, restaurando la versión anterior desde un snapshot almacenado en S3 con cifrado AES-256.

- **Análisis de dependencias con resolución de conflictos transactivos**: Escanea el árbol de dependencias de `package.json`, `requirements.txt` o `go.mod` y detecta versiones con CVE conocidas (base de datos actualizada cada 6 horas). Si encuentra un conflicto, genera un parche de actualización automática que se aplica en un entorno sandbox de Docker, validando que no se rompan pruebas unitarias. Parámetro: `--max-depth 5` para limitar la recursión. Los resultados se exportan a un informe JSON con metadatos de tiempo de análisis (promedio: 2.3 segundos para 1.000 dependencias).

### Cómo Usar

1. **Inicializar el guardián en tu repositorio local**  
   Ejecuta `deployguard init --project-dir ./mi-app --db-uri postgresql://user:pass@localhost:5432/deployguard --secret-scan-level aggressive`. Esto crea un archivo `.deployguard.yml` con la configuración base, conecta a PostgreSQL y genera la clave ECDSA para firmar artefactos. Luego, ejecuta `deployguard scan --path ./src --format json` para realizar un escaneo inicial de secretos; el resultado se guarda en `./.deployguard/scan_report.json`.

2. **Integrar con tu pipeline de CI/CD (GitHub Actions o GitLab CI)**  
   Añade el siguiente paso en tu archivo YAML:  
   ```yaml
   - name: DeployGuard Audit
     run: deployguard audit --commit-hash ${{ github.sha }} --fail-on-critical --rollback-on-failure
   ```  
   Esto ejecuta una auditoría completa: verifica la firma del binario, escanea secretos en el diff del commit y comprueba dependencias. Si se encuentra una vulnerabilidad crítica (score CVSS > 9.0), el pipeline falla y se revierte automáticamente al último commit estable, usando el snapshot almacenado en `./.deployguard/snapshots/`.

3. **Generar un informe de cumplimiento para auditoría externa**  
   Usa `deployguard report --type compliance --output pdf --include-evidence --from 2025-01-01 --to 2025-03-15`. Este comando extrae todos los eventos de seguridad registrados en PostgreSQL (tabla `audit_log`), los agrupa por severidad y genera un PDF con gráficos de barras (tiempo de respuesta, número de falsos positivos, etc.) y enlaces a los commits específicos. El informe incluye un resumen ejecutivo con recomendaciones técnicas.

### Mejores Prácticas

- **Configurar umbrales de alerta personalizados por entorno**: En producción, establece `--confidence-threshold 0.99` para reducir al mínimo los falsos positivos, pero en desarrollo usa `0.85` para capturar incluso patrones ambiguos. Combínalo con `--exclude-paths ./tests/` para evitar ruido en pruebas unitarias. Esto evita que un falso positivo bloquee un despliegue crítico, mientras que en desarrollo maximizas la cobertura.

- **Automatizar la rotación de claves de firma cada 30 días**: Programa un cron job que ejecute `deployguard rotate-keys --key-id $(deployguard list-keys --active | jq -r '.id') --backup-old-keys`. Las claves antiguas se archivan en PostgreSQL con cifrado, y el nuevo par se distribuye a todos los nodos mediante un webhook. Esto mitiga el riesgo de que una clave comprometida afecte a artefactos pasados, ya que DeployGuard valida la cadena de confianza hasta la clave raíz.
