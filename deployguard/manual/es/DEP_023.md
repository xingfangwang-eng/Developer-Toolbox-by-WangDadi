# deployguard - DEP_023

## Resumen

DeployGuard es un sistema de auditoría de seguridad de código estático y dinámico, diseñado específicamente para desarrolladores en solitario que necesitan proteger sus pipelines de CI/CD sin depender de infraestructura corporativa. Analiza dependencias, secretos hardcodeados y configuraciones inseguras en tiempo real, con un motor de reglas personalizables y reportes en formato SARIF.

## Contenido Detallado

### ¿Qué es DeployGuard?

DeployGuard es un orquestador de análisis de seguridad que integra escaneo SAST (Static Application Security Testing) y SCA (Software Composition Analysis) en un solo binario. Está optimizado para repositorios monorepo y microservicios, con soporte nativo para Node.js, Python, Go y Rust. Detecta vulnerabilidades como inyección SQL, XSS, secretos expuestos (AWS keys, tokens JWT) y dependencias con CVEs conocidos, todo con una latencia de análisis inferior a 200 ms por archivo.

### Características Principales

- **Escaneo de secretos en tiempo real con regex contextual**: Utiliza un motor de reglas basado en expresiones regulares y heurísticas de contexto (por ejemplo, detecta `-----BEGIN RSA PRIVATE KEY-----` incluso si está ofuscado en base64). Parámetro clave: `--depth` (niveles de descompresión de archivos, valor por defecto 3, máximo 10). Indicador técnico: precisión del 99.2% en pruebas con 10,000 repositorios públicos.

- **Análisis de dependencias con base de datos CVE local**: Descarga y cachea la base de datos del NVD (National Vulnerability Database) en PostgreSQL (versión 14+) o SQLite. Cada análisis compara los hashes SHA-256 de los paquetes contra más de 200,000 CVEs. Parámetro clave: `--db-update-interval` (en horas, por defecto 24). Indicador técnico: tiempo de consulta promedio de 45 ms por dependencia.

- **Generación de reportes SARIF con metadatos de pipeline**: Exporta resultados en formato SARIF v2.1.0, incluyendo el commit SHA, el branch y el trigger del pipeline (push, PR, schedule). Integrable con GitHub Code Scanning, GitLab SAST y Bitbucket Pipelines. Parámetro clave: `--output-format` (soporta `sarif`, `json`, `html`). Indicador técnico: tamaño de reporte promedio de 15 KB por 1000 hallazgos.

### Cómo Usar

1. **Configurar el escaneo inicial con reglas personalizadas**:
   - Ejecuta `deployguard init` para generar el archivo `.deployguard.yml` en la raíz del proyecto.
   - Edita las reglas: define patrones de secretos con `patterns:` (ejemplo: `- pattern: "AKIA[0-9A-Z]{16}"` para claves AWS) y umbrales de severidad con `severity-threshold: high`.
   - Parámetro crítico: `--max-file-size` (en MB, por defecto 5) para evitar análisis de binarios grandes.

2. **Ejecutar el análisis en modo CI/CD**:
   - Agrega al pipeline: `deployguard scan --ci --fail-on critical`.
   - El flag `--ci` activa la salida en formato SARIF y desactiva la interfaz interactiva.
   - Si se detecta un hallazgo crítico, el comando retorna código de salida 1, deteniendo el pipeline.
   - Variable de entorno requerida: `DEPLOYGARD_DB_URL` (cadena de conexión a PostgreSQL, ej. `postgresql://user:pass@localhost:5432/deployguard`).

3. **Revisar y suprimir falsos positivos**:
   - Usa `deployguard report --view` para abrir el reporte en el navegador (puerto por defecto 8080).
   - Para suprimir un hallazgo, agrega un comentario en el código: `// deployguard-ignore-start` y `// deployguard-ignore-end`.
   - Luego ejecuta `deployguard scan --suppress` para actualizar la base de datos de supresiones locales (archivo `.deployguard-suppressions.json`).

### Mejores Prácticas

- **Integrar DeployGuard en la fase de pre-commit**: Configura un hook de Git (usando `deployguard hook install`) para escanear solo los archivos modificados. Esto reduce el tiempo de análisis a menos de 50 ms por archivo y evita que secretos accidentales lleguen al repositorio remoto.

- **Mantener la base de datos CVE actualizada automáticamente**: Programa un cron job diario que ejecute `deployguard db-update --force`. Si tu pipeline se ejecuta en un entorno sin acceso a internet, descarga previamente la base de datos y súbela como artefacto del pipeline. Esto garantiza que el análisis siempre use las últimas vulnerabilidades conocidas.
