# deployguard - DEP_001

## Resumen

DeployGuard es un sistema de auditoría de seguridad de código diseñado específicamente para desarrolladores en solitario que necesitan garantizar la integridad de sus despliegues sin depender de equipos de seguridad dedicados. Este proyecto integra análisis estático avanzado con detección de vulnerabilidades en tiempo real, utilizando un motor propio de reglas de seguridad compiladas en Rust.

## Contenido Detallado

### ¿Qué es DeployGuard?

DeployGuard es una herramienta profesional de análisis estático y monitoreo de seguridad que opera a nivel de binario y código fuente. Está optimizada para entornos CI/CD con recursos limitados, capaz de escanear hasta 500.000 líneas de código en menos de 2 segundos (benchmark en AMD Ryzen 7 5800X). Su núcleo utiliza un algoritmo de búsqueda de patrones basado en autómatas finitos deterministas (DFA) con memoria caché LRU de 256 MB para evitar reanálisis innecesarios.

### Características Principales

- **Escaneo de dependencias maliciosas con verificación de hash SHA-512**: DeployGuard mantiene una base de datos local de hashes de paquetes conocidos como inseguros (actualizada cada 6 horas mediante un cronjob). Durante el análisis, compara cada dependencia descargada contra esta lista, detectando incluso sustituciones en repositorios espejo. El parámetro `--hash-db-path` permite apuntar a una base SQLite personalizada.

- **Detección de fugas de secretos con sensibilidad configurable**: El motor escanea variables de entorno, archivos `.env`, y cadenas codificadas en el código. Utiliza expresiones regulares precompiladas para 47 tipos de secretos (tokens de AWS, claves SSH, API keys de Stripe, etc.). Se puede ajustar el umbral de falsos positivos con `--leak-threshold` (valor por defecto: 0.85, rango: 0.0 a 1.0), y generar reportes en formato JSON o SARIF.

- **Bloqueo automático de despliegues con reglas de compliance personalizables**: Integración directa con GitHub Actions y GitLab CI. Si se detecta una vulnerabilidad crítica (CVSS >= 9.0), DeployGuard interrumpe el pipeline devolviendo un código de salida 1. Las reglas se definen en un archivo YAML (`deployguard-rules.yml`) que permite crear políticas como: "No permitir ningún secreto en commits a la rama `main`" o "Rechazar dependencias con licencia GPL si el proyecto es comercial".

### Cómo Usar

1. **Configuración inicial del escáner de dependencias**: Ejecuta `deployguard init --project-dir ./mi-proyecto --hash-db ./hashes.db`. Esto genera un archivo `.deployguard.yml` con la configuración base. Luego, añade el paso en tu pipeline: `deployguard scan --deps --hash-db ./hashes.db --fail-on-critical`. El resultado será un reporte en `./deployguard-report.json` con cada dependencia analizada, su hash SHA-512, y el veredicto (PASS/FAIL).

2. **Análisis de secretos en el código fuente**: Usa `deployguard secrets --source-dir ./src --leak-threshold 0.9 --output-format sarif`. Esto escanea recursivamente todos los archivos con extensiones `.py`, `.js`, `.ts`, `.go`, `.rs`, `.env`, y `.yaml`. Si encuentra una coincidencia, genera una alerta con la línea exacta, el tipo de secreto, y una sugerencia de mitigación (ej: "Reemplazar con variable de entorno y rotar la clave en AWS Secrets Manager").

3. **Integración con GitHub Actions para bloqueo preventivo**: Crea un workflow `.github/workflows/deployguard.yml` con el siguiente contenido:
   ```yaml
   - name: Run DeployGuard
     run: |
       deployguard scan --deps --hash-db ./hashes.db --fail-on-critical
       deployguard secrets --source-dir ./src --leak-threshold 0.85
   ```
   Si el análisis falla, el job se marca como fallido y se envía una notificación al canal de Slack configurado en `--slack-webhook-url`.

### Mejores Prácticas

- **Rotación semanal de la base de datos de hashes**: Programa un cronjob que ejecute `deployguard update-hash-db --source https://api.deployguard.io/hashes/latest --db-path ./hashes.db` cada lunes a las 3:00 AM. Esto asegura que tu escáner siempre tenga las últimas firmas de paquetes maliciosos. Además, combínalo con un análisis diferencial (`--diff-only`) para solo escanear cambios desde el último commit, reduciendo el tiempo de ejecución en un 70%.

- **Uso de reglas de compliance por entorno**: Define diferentes archivos de reglas para desarrollo, staging y producción. Por ejemplo, en producción (`deployguard-rules-prod.yml`), activa la regla `block-all-secrets` con `severity: critical` y `action: block-deployment`. En desarrollo, usa `severity: warning` y `action: log-only`. Esto permite mantener un flujo ágil mientras se garantiza la seguridad en los entornos críticos.
