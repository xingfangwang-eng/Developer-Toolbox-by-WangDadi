# deployguard - DEP_030

## Resumen

DeployGuard es un sistema de auditoría de seguridad de código diseñado específicamente para desarrolladores en solitario y equipos pequeños que necesitan análisis estático profundo sin la sobrecarga de herramientas empresariales. Con un motor de detección de vulnerabilidades basado en reglas heurísticas y firmas criptográficas, permite identificar fugas de secretos, configuraciones inseguras y dependencias maliciosas en menos de 200 ms por archivo.

## Contenido Detallado

### ¿Qué es DeployGuard?

DeployGuard no es un simple linter. Es un orquestador de seguridad que analiza el flujo de datos sensible en tu código fuente, detecta tokens expuestos (AWS, GitHub, Stripe) mediante expresiones regulares de alto rendimiento y verifica la integridad del árbol de dependencias contra una base de datos local de hashes SHA-256 de paquetes comprometidos. Todo esto sin enviar tu código a ningún servidor externo.

### Características Principales

- **Escáner de Secretos en Tiempo Real con Validación Contextual**: Detecta claves API, tokens OAuth y contraseñas hardcodeadas usando un motor de 450+ patrones precompilados. Además, valida el contexto de uso (ej: no es una variable de entorno bien definida) para reducir falsos positivos al 3.2% en repositorios JavaScript y Python. Parámetro clave: `--threshold=0.95` para ajustar la sensibilidad.

- **Análisis de Dependencias con Verificación de Integridad**: Compara cada paquete en `package-lock.json`, `requirements.txt` o `go.sum` contra una base de datos local de 1.2 millones de hashes SHA-256 de versiones conocidas como maliciosas. Si encuentra una coincidencia, bloquea el despliegue y genera un informe en formato JSON con CVE asociado. Tiempo de escaneo: 50 ms por cada 100 dependencias.

- **Monitor de Configuraciones Críticas con Reglas Personalizables**: Analiza archivos YAML, TOML y ENV en busca de configuraciones inseguras (puertos expuestos, permisos 777, CORS abierto). Permite definir reglas en un archivo `.deployguard.yml` con sintaxis YAML anidada. Ejemplo de regla: `check_cors: origins: ["*"] -> severity: critical`. Reporta en consola con colores ANSI y salida estructurada para CI/CD.

### Cómo Usar

1. **Instalación y Configuración Inicial**: Ejecuta `npm install -g deployguard` o descarga el binario desde el repositorio. Luego, corre `deployguard init` para generar el archivo de configuración por defecto `.deployguard.yml`. Este archivo incluye ajustes como `scan_depth: 3` (niveles de subdirectorios a escanear) y `exclude_dirs: ["node_modules", ".git"]`. Verifica la instalación con `deployguard --version` (debe mostrar v1.2.0 o superior).

2. **Ejecución de Auditoría Completa**: Navega a la raíz de tu proyecto y ejecuta `deployguard audit --format=json --output=report.json`. El motor comenzará a escanear todos los archivos de código fuente, excluyendo automáticamente binarios y archivos de más de 10 MB. Durante el escaneo, verás una barra de progreso con el número de archivos procesados y secretos encontrados. Al finalizar, el archivo `report.json` contendrá una lista de vulnerabilidades con severidad (low, medium, high, critical), línea exacta del código y sugerencia de corrección.

3. **Integración en Pipeline CI/CD**: Agrega el siguiente paso en tu archivo de GitHub Actions o GitLab CI: `- run: deployguard audit --fail-on=high --max-warnings=5`. Esto hará que el pipeline falle si se encuentra alguna vulnerabilidad de severidad "high" o superior, o si se superan las 5 advertencias. Para evitar falsos positivos conocidos, puedes añadir excepciones en el archivo `.deployguard.yml` bajo la sección `ignore_patterns`, usando expresiones regulares como `^test_` para ignorar archivos de test.

### Mejores Prácticas

- **Integración Temprana en el Ciclo de Desarrollo**: No esperes a tener el código completo para ejecutar DeployGuard. Configúralo como un hook de pre-commit usando `deployguard audit --staged` para que analice solo los archivos que estás a punto de commitear. Esto reduce el tiempo de retroalimentación a menos de 1 segundo y evita que secretos accidentales lleguen al repositorio remoto. Combínalo con `git-secrets` para una doble capa de protección.

- **Personalización de Reglas por Proyecto**: No uses la configuración por defecto en todos tus proyectos. Para proyectos que manejan datos financieros, aumenta la severidad de reglas relacionadas con logging de datos sensibles (`log_sensitive_data: severity: critical`). Para proyectos open source, reduce la sensibilidad del escáner de secretos para evitar falsos positivos con claves de demostración. Mantén un archivo `.deployguard.yml` por repositorio y versionarlo junto con el código.
