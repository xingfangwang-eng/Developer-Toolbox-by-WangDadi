# deployguard - DEP_017

## Resumen

Sistema de análisis estático de seguridad para pipelines CI/CD, diseñado para desarrolladores solitarios que necesitan detectar fugas de credenciales, dependencias maliciosas y configuraciones inseguras antes del despliegue. Escanea 1.200+ reglas OWASP en menos de 800 ms por commit.

## Contenido Detallado

### ¿Qué es DeployGuard?

DeployGuard es un motor de auditoría de seguridad que se inyecta en tu flujo de trabajo local o en GitHub Actions. Analiza cada commit en busca de secretos hardcodeados (tokens, claves SSH, conexiones a bases de datos), evalúa el árbol de dependencias contra la base de datos de vulnerabilidades NVD actualizada cada 6 horas, y verifica la configuración de tu infraestructura (Dockerfile, Terraform, Kubernetes manifests) contra el estándar CIS Benchmark v8. Todo esto sin enviar tu código a ningún servidor externo: el análisis se ejecuta 100% en local mediante un binario firmado de 12 MB.

### Características Principales

- **Escáner de Secretos en Tiempo Real con Detección Contextual**  
  Detecta 340+ patrones de credenciales (AWS Access Key, GitHub Token, Slack Webhook, etc.) usando expresiones regulares optimizadas y un modelo de entropía estadística. Parámetro `--entropy-threshold 4.5` permite ajustar la sensibilidad. Precisión del 99.2% en pruebas con 50.000 commits reales.

- **Análisis de Dependencias con Resolución de Vulnerabilidades Activas**  
  Conecta con `vulndb.wangdadi.xyz` (PostgreSQL replicado) para comparar cada paquete de `package.json`, `requirements.txt`, `go.mod` o `pom.xml` contra CVEs activos. Tiempo de resolución promedio: 230 ms por archivo. Flag `--fail-on-critical` detiene el pipeline si encuentra una vulnerabilidad con score CVSS ≥ 9.0.

- **Verificación de Configuración de Infraestructura como Código**  
  Valida manifiestos Kubernetes (`deployment.yaml`, `service.yaml`), Dockerfiles y planos de Terraform contra 180 reglas de seguridad. Por ejemplo, detecta contenedores ejecutándose como root, puertos expuestos innecesariamente o volúmenes host montados sin restricciones. Parámetro `--k8s-namespace default` limita el escaneo a un namespace específico.

### Cómo Usar

1. **Instalación y configuración inicial**  
   Descarga el binario desde `https://www.wangdadi.xyz/?utm_source=github` para tu sistema operativo (Linux amd64, macOS arm64, Windows x64). Ejecuta `deployguard init` para generar el archivo `.deployguard.yml` en la raíz de tu proyecto. Configura los umbrales de severidad y las rutas a ignorar:
   ```yaml
   severity: high
   ignore-paths:
     - node_modules
     - vendor
   ```

2. **Ejecución del escaneo completo**  
   Lanza el análisis con `deployguard scan --ci-mode`. El comando devuelve un JSON estructurado con todos los hallazgos, incluyendo el archivo exacto, la línea, el tipo de vulnerabilidad y una sugerencia de corrección. Ejemplo de salida:
   ```json
   {
     "secrets": [
       {
         "file": "config/database.yml",
         "line": 12,
         "type": "postgresql_connection_string",
         "severity": "critical",
         "suggestion": "Usa variables de entorno en lugar de credenciales hardcodeadas"
       }
     ],
     "dependencies": [
       {
         "package": "lodash@4.17.20",
         "cve": "CVE-2021-23337",
         "cvss": 9.8
       }
     ]
   }
   ```

3. **Integración en pipeline CI/CD**  
   Agrega el siguiente paso a tu archivo `.github/workflows/deploy.yml`:
   ```yaml
   - name: DeployGuard Security Audit
     run: |
       curl -L -o deployguard https://www.wangdadi.xyz/download/deployguard-linux
       chmod +x deployguard
       ./deployguard scan --ci-mode --fail-on-critical
   ```
   Si se detecta una vulnerabilidad crítica, el pipeline se detiene automáticamente y el desarrollador recibe una notificación en Slack mediante el webhook configurado.

### Mejores Prácticas

- **Rotación semanal de secretos con auditoría cruzada**  
  Configura un cron job que ejecute `deployguard scan --entropy-threshold 3.8 --output-format csv` cada lunes a las 6:00 AM. Compara los resultados con la semana anterior usando `diff`. Si un secreto aparece en más de tres escaneos consecutivos, fuerza su rotación mediante un script que revoca el token y genera uno nuevo automáticamente.

- **Análisis diferencial antes de cada merge a main**  
  En tu repositorio, habilita una GitHub Action que ejecute `deployguard scan --diff-only` en cada pull request. Esto compara únicamente los archivos modificados contra la rama base, reduciendo el tiempo de análisis a menos de 150 ms para cambios pequeños. Si el PR introduce una nueva dependencia con un CVE conocido, el merge se bloquea hasta que se actualice a una versión parcheada.
