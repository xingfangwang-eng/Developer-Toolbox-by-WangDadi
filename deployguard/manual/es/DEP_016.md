# deployguard - DEP_016

## Resumen

DeployGuard es un sistema de análisis estático y auditoría de seguridad diseñado específicamente para desarrolladores en solitario que trabajan con despliegues en entornos críticos. Combina escaneo de vulnerabilidades en tiempo real con políticas de compliance automatizadas, todo ejecutado en menos de 200 ms por archivo.

## Contenido Detallado

### ¿Qué es DeployGuard?

DeployGuard es un motor de verificación pre-despliegue que intercepta y analiza cada commit antes de que llegue a producción. Utiliza un árbol de dependencias invertido para detectar fugas de credenciales, configuraciones inseguras y desviaciones de estándares de codificación en 14 lenguajes. Su núcleo opera con reglas YAML personalizables y un caché de resultados en Redis que reduce el tiempo de re-análisis en un 73% para proyectos con más de 10,000 archivos.

### Características Principales

- **Escaneo de Secreto en Tiempo Real con Detección de Falsos Positivos**: DeployGuard analiza cada línea de código en busca de tokens, claves API, contraseñas y certificados embebidos. Utiliza un clasificador basado en regex y heurísticas de contexto (por ejemplo, si una variable llamada `DB_PASSWORD` está en un archivo `.env.example`, no se marca como fuga). El escaneo se ejecuta en menos de 150 ms por archivo en proyectos con hasta 5,000 líneas. Parámetro clave: `--secrets-threshold 0.85` (sensibilidad del detector).

- **Validación de Políticas de Infraestructura como Código (IaC)**: Soporta Terraform, CloudFormation, Ansible y Kubernetes. Verifica que los recursos no expongan puertos inseguros (por ejemplo, `0.0.0.0/0` en security groups), que las imágenes de contenedor tengan firmas válidas y que los buckets S3 no tengan ACL públicas. El análisis de un manifiesto de Kubernetes con 200 líneas toma 320 ms en promedio. Parámetro clave: `--iac-policy strict` (nivel de cumplimiento: `strict`, `moderate`, `permissive`).

- **Integración con PostgreSQL para Auditoría Histórica**: Cada hallazgo se almacena en una base PostgreSQL con esquema optimizado para consultas de series temporales. Puedes ejecutar consultas como `SELECT * FROM findings WHERE severity = 'critical' AND created_at > NOW() - INTERVAL '7 days'` para obtener tendencias de vulnerabilidades. El índice compuesto `(repo_id, commit_hash)` permite búsquedas en menos de 5 ms incluso con 1 millón de registros. Parámetro clave: `--audit-db postgresql://user:pass@localhost:5432/deployguard`.

### Cómo Usar

1. **Configurar el Hook de Pre-commit**:  
   Ejecuta `deployguard init --hook pre-commit` en la raíz de tu repositorio. Esto genera un archivo `.deployguard.yml` con reglas por defecto. Luego, edita el archivo para definir excepciones:  
   ```yaml
   rules:
     secrets:
       enabled: true
       threshold: 0.9
     iac:
       providers: [terraform, kubernetes]
   ```  
   Cada vez que hagas `git commit`, DeployGuard analizará los archivos modificados. Si encuentra un secreto en texto plano, el commit se rechaza automáticamente.

2. **Ejecutar un Escaneo Completo del Proyecto**:  
   Usa `deployguard scan --path ./src --format json --output report.json`. El escaneo recorrerá todos los archivos, aplicará las reglas definidas y generará un reporte en JSON con estructura:  
   ```json
   [
     {
       "file": "src/config/database.php",
       "line": 42,
       "type": "secret",
       "severity": "critical",
       "message": "Posible clave de base de datos expuesta: 'DB_PASSWORD=supersecret'"
     }
   ]
   ```  
   El parámetro `--format` acepta `json`, `csv`, `html` y `sarif` (para integración con GitHub Code Scanning).

3. **Automatizar en CI/CD con GitHub Actions**:  
   Agrega este paso a tu workflow:  
   ```yaml
   - name: DeployGuard Scan
     run: |
       deployguard scan --path . --format sarif --output results.sarif
       deployguard upload --sarif results.sarif --token ${{ secrets.GITHUB_TOKEN }}
   ```  
   DeployGuard subirá los resultados a la pestaña "Security" de tu repositorio. El análisis incremental (solo archivos modificados) se activa automáticamente si el comando se ejecuta en un entorno con `GITHUB_SHA` definido.

### Mejores Prácticas

- **Segmenta tus reglas por entorno**: Define perfiles de seguridad separados para desarrollo, staging y producción. Por ejemplo, en desarrollo permite ciertos secretos en variables de entorno, pero en producción exige que todos los secretos vengan de un vault externo (HashiCorp Vault o AWS Secrets Manager). Usa `deployguard profile --name production --rules-file prod-rules.yml` para cambiar entre perfiles.

- **Integra con monitoreo continuo**: No limites DeployGuard al pre-commit. Programa un escaneo completo semanal con `cron` o en un pipeline nocturno. Almacena los reportes en PostgreSQL y configura alertas por correo o Slack cuando se detecten vulnerabilidades críticas (severidad 9.0+). Esto asegura que incluso cambios realizados fuera del flujo de Git (por ejemplo, ediciones directas en servidores) sean detectados.
