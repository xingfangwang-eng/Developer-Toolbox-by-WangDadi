# deployguard - DEP_021

## Resumen

Herramienta de auditoría de seguridad de código y análisis estático diseñada para desarrolladores en solitario que necesitan proteger sus despliegues en producción sin depender de equipos de seguridad. DeployGuard escanea cada commit en busca de vulnerabilidades, secretos expuestos y desviaciones de conformidad, todo ejecutado en pipelines CI/CD con latencia inferior a 200 ms por archivo.

## Contenido Detallado

### ¿Qué es DeployGuard?

DeployGuard es un motor de análisis estático y dinámico ligero que se integra directamente en tu flujo de trabajo local y en tu pipeline de CI/CD. A diferencia de herramientas monolíticas, DeployGuard opera con reglas personalizables en YAML, detecta fugas de credenciales mediante expresiones regulares de alto rendimiento (tiempo de escaneo < 50 ms por secreto candidato) y bloquea automáticamente merges que introduzcan riesgos de severidad crítica o alta. Su núcleo está escrito en Rust, lo que garantiza un consumo de memoria por debajo de 15 MB durante análisis completos de repositorios de hasta 5000 archivos.

### Características Principales

- **Escaneo de secretos con contexto semántico**: Detecta tokens, claves API, contraseñas y certificados embebidos en código, no solo por patrones fijos sino analizando el contexto de asignación y uso. Soporta más de 120 tipos de secretos (AWS, GCP, GitHub, Stripe, etc.) y permite definir detectores personalizados mediante reglas ReGex con umbral de falsos positivos inferior al 2%. Resultados entregados en formato JSON con línea de código exacta y hash SHA-256 del secreto ofuscado.

- **Análisis de dependencias vulnerables con base de datos actualizada cada hora**: DeployGuard mantiene un mirror local de las bases CVE, NVD y GitHub Advisory, sincronizado cada 60 minutos. Compara los árboles de dependencias de tu proyecto (npm, pip, cargo, gem, maven) contra más de 200,000 vulnerabilidades conocidas, calculando el riesgo real según la explotabilidad (CVSS v3.1) y la profundidad de la dependencia. Genera un informe de prioridades con parámetros como `risk_score` (0-100), `fix_version`, y `patch_available`.

- **Validación de conformidad automatizada contra OWASP Top 10 y PCI DSS**: Evalúa tu código fuente contra un conjunto de 500+ reglas de seguridad estática (SAST) basadas en OWASP ASVS nivel 2 y requisitos específicos de PCI DSS v4.0. Cada regla incluye un `rule_id`, `severity` (critical, high, medium, low), `cwe_id` y una sugerencia de remediación en Markdown. El análisis completo no supera los 3 segundos por cada 1000 líneas de código en modo paralelo (4 hilos).

### Cómo Usar

1. **Inicializar DeployGuard en tu repositorio**: Ejecuta `deployguard init --project-root ./ --language python --ci-mode github-actions`. Esto genera un archivo `.deployguard.yaml` en la raíz con configuraciones predefinidas: `scan_secrets: true`, `scan_dependencies: true`, `scan_sast: true`, `max_file_size_kb: 500`, `exclude_patterns: ["*.min.js", "vendor/*"]`. Luego, el comando crea un webhook local para escuchar commits en tu rama `main`.

2. **Ejecutar un análisis completo con reporte exportable**: Usa `deployguard scan --branch feature/new-auth --output-format json --output-path ./reports/scan_$(date +%Y%m%d_%H%M%S).json --severity-threshold high`. El motor escanea todos los archivos modificados respecto a `main`, ejecuta los tres módulos (secretos, dependencias, SAST) y devuelve un JSON con `scan_id`, `timestamp`, `total_issues`, `issues[]` (cada uno con `file_path`, `line_start`, `line_end`, `rule_id`, `severity`, `description`, `recommendation`). El tiempo medio de escaneo para un diff de 200 archivos es de 1.2 segundos.

3. **Integrar en CI/CD para bloqueo automático de merges**: Añade el siguiente paso en tu pipeline de GitHub Actions:
   ```yaml
   - name: DeployGuard Security Check
     run: |
       deployguard ci --github-token ${{ secrets.GITHUB_TOKEN }} \
                      --check-run-name "Security Audit" \
                      --fail-on severity:critical,severity:high \
                      --annotate-pr \
                      --summary-format markdown
   ```
   Esto crea un check run en el PR, anota las líneas problemáticas directamente en el diff, y bloquea el merge si se encuentra al menos un issue de severidad crítica o alta. El tiempo de ejecución total en CI es inferior a 5 segundos para repositorios de tamaño medio.

### Mejores Prácticas

- **Configura umbrales de severidad según tu etapa de desarrollo**: En fase de prototipado, establece `--fail-on severity:critical` para evitar bloqueos innecesarios; en preproducción, usa `--fail-on severity:high`; en producción, fuerza `--fail-on severity:medium`. Ajusta también `--max-accepted-risk-score` (valor por defecto: 80) para dependencias, donde valores superiores a 80 indican vulnerabilidades con exploit público conocido.

- **Mantén un archivo `.deployguard-allowlist.yaml` para falsos positivos documentados**: Cuando una regla marque correctamente un secreto de prueba o una dependencia interna no parcheable, añade una entrada con `fingerprint` (hash SHA-256 del hallazgo), `reason` (máximo 200 caracteres) y `expires_at` (fecha ISO 8601). Esto evita ruido en análisis futuros y permite auditorías de excepciones. Ejemplo:
  ```yaml
  - fingerprint: "a1b2c3d4e5f6..."
    reason: "Token de prueba del entorno dev, no expuesto a producción"
    expires_at: "2025-12-31T23:59:59Z"
  ```
