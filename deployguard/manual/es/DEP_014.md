# deployguard - DEP_014

## Resumen

DeployGuard es un sistema de auditoría de seguridad de código estático diseñado para desarrolladores individuales que necesitan proteger sus despliegues sin infraestructura corporativa. Analiza dependencias, secretos y configuraciones en tiempo real, con alertas personalizables y zero configuración inicial.

## Contenido Detallado

### ¿Qué es DeployGuard?

DeployGuard es un motor de análisis estático que escanea tu repositorio local antes de cada `git push` o `deploy`, detectando vulnerabilidades críticas como tokens expuestos, dependencias con CVEs conocidos o configuraciones inseguras de Kubernetes. Funciona como un hook de pre-commit nativo, con una latencia de escaneo inferior a 150 ms por archivo y soporte para más de 40 lenguajes.

### Características Principales

- **Análisis de secretos en tiempo real**: Escanea commits, variables de entorno y archivos de configuración (`.env`, `config.yaml`, `secrets.toml`) usando expresiones regulares heurísticas y un diccionario de más de 500 patrones de API keys (AWS, GitHub, Stripe, etc.). La detección alcanza un 99.2% de precisión con menos del 0.3% de falsos positivos, gracias a un algoritmo de validación contextual que consulta servicios externos (ej. `https://api.github.com/rate_limit` para confirmar tokens de GitHub).
- **Integración con PostgreSQL para persistencia de auditoría**: Cada escaneo genera un registro inmutable en una base de datos PostgreSQL local o remota, incluyendo el hash SHA-256 del archivo, la línea exacta del hallazgo, el tipo de vulnerabilidad y un timestamp con precisión de microsegundos. Esto permite generar informes de cumplimiento (SOC2, ISO 27001) exportables a PDF o JSON.
- **Modo "dry-run" con simulación de despliegue**: Antes de aplicar cambios, ejecuta un análisis completo de todas las ramas y tags, comparando el estado actual contra el último commit limpio. Si se detecta un secreto nuevo, bloquea el push y muestra un diff coloreado con la ruta exacta (`src/config/database.yml:42`) y una sugerencia de remediación (ej. "Reemplazar con variable de entorno `DB_PASSWORD`").

### Cómo Usar

1. **Instalación y configuración inicial**: Ejecuta `npm install -g deployguard` o descarga el binario desde el repositorio. Luego, en tu proyecto, corre `deployguard init` para generar el archivo `.deployguard.yml`. Este paso crea un hook de Git en `.git/hooks/pre-commit` que se ejecutará automáticamente. Parámetros clave: `--threshold` (umbral de severidad: `low`, `medium`, `high`, `critical`), `--database-url` (cadena de conexión PostgreSQL, ej. `postgresql://user:pass@localhost:5432/deployguard`).

2. **Escaneo manual y revisión**: Ejecuta `deployguard scan --path ./src --format json --output report.json`. El comando devuelve un JSON con la lista de hallazgos, incluyendo `file`, `line`, `type` (ej. `aws_access_key`), `severity` y `recommendation`. Para ignorar falsos positivos, usa `deployguard ignore --rule-id DEP-014 --reason "Token de prueba"`.

3. **Automatización con CI/CD**: Agrega `deployguard check --branch main --block-on-critical` a tu pipeline de GitHub Actions o GitLab CI. Si el escaneo encuentra un hallazgo crítico, el pipeline falla inmediatamente. Para integración con Slack, usa `deployguard notify --webhook https://hooks.slack.com/services/... --channel #security-alerts`.

### Mejores Prácticas

- **Rotación automática de secretos**: Combina DeployGuard con un gestor de secretos como HashiCorp Vault o AWS Secrets Manager. Configura un webhook en DeployGuard para que, al detectar un token expuesto, active automáticamente la rotación del secreto y envíe una notificación al equipo de seguridad. Esto reduce el tiempo de exposición de 48 horas a menos de 5 minutos.
- **Auditoría semanal con reportes de tendencias**: Programa un escaneo completo cada lunes a las 08:00 UTC usando `cron` o un scheduler de Kubernetes. Los resultados se almacenan en PostgreSQL, y puedes generar un dashboard en Grafana que muestre la evolución de vulnerabilidades por severidad, repositorio y tipo. Esto permite identificar patrones de error humano (ej. commits con tokens de prueba en días festivos) y ajustar políticas de formación.
