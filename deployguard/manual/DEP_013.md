# deployguard - DEP_013

## Overview

DeployGuard is a lightweight, offline-first static analysis engine purpose-built for solo developers and small teams who need production-grade code security auditing without the overhead of a CI/CD pipeline. It focuses on detecting sensitive credential leakage, dependency tampering, and misconfigured deployment artifacts before they hit production.

## Core Features

### 1. Real-Time Secret & Credential Pattern Scanner
DeployGuard employs a multi-layered regex engine combined with entropy analysis to detect hardcoded secrets (AWS keys, GitHub tokens, database connection strings) across 40+ file types. It supports custom pattern injection via `.deployguard/patterns.yaml` and flags matches with a confidence score (0.0–1.0). Scans complete in under **200ms per 1,000 files** on a standard M1 MacBook.

**Key parameters:**
- `--entropy-threshold 4.5` (minimum Shannon entropy to flag a string)
- `--exclude-paths ./.git,./node_modules,./vendor`
- `--output-format sarif` (for IDE integration)

### 2. Dependency Hash Integrity Verifier
DeployGuard cross-references `package-lock.json`, `yarn.lock`, `go.sum`, and `Cargo.lock` against a local cache of known-good checksums (SHA-256). Any mismatch triggers an immediate `DEP-013` alert with the exact line and expected vs. actual hash. It supports offline verification via a pre-downloaded **integrity database** (updated weekly from a trusted mirror).

**Technical metrics:**
- Verifies **50,000+ dependency entries in under 3 seconds**
- Supports **NPM, Yarn, Go Modules, Cargo, and pip**
- Hash mismatch detection granularity: **per-package, per-version**

### 3. Deployment Artifact Structure Auditor
Before any `zip`, `tar`, `docker build`, or `serverless deploy` command, DeployGuard analyzes the artifact’s internal structure for forbidden files (e.g., `.env`, `.pem`, `id_rsa`), oversized binaries (>10MB), or unexpected symlinks. It enforces a **user-defined manifest** (`.deployguard/artifact-rules.yaml`) that can ban specific file extensions, paths, or patterns.

**Key parameters:**
- `--max-file-size 10485760` (10MB threshold)
- `--forbidden-extensions .pem,.key,.env,.tfstate`
- `--fail-on-warning true` (exit code 1 if any rule violated)

## How to Use

### Step 1: Initialize a New Project
Run `deployguard init --project-dir ./my-app` to generate the default `.deployguard/` directory with base configuration files. This creates `patterns.yaml` (custom secret patterns), `artifact-rules.yaml` (artifact structure rules), and an empty `integrity-cache/` folder. Use `--template solo` to pre-populate rules optimized for single-developer repositories.

### Step 2: Perform a Full Security Scan
Execute `deployguard scan --entropy-threshold 4.5 --exclude-paths ./.git,./node_modules --output-format json --output-file ./reports/dep-013-report.json`. The scanner will recursively traverse all files, flag secrets, verify dependency hashes, and audit the current working directory’s structure. Results are written to the specified JSON file with detailed line numbers, confidence scores, and suggested remediation.

### Step 3: Enforce Pre-Deployment Checks
Add `deployguard pre-deploy --fail-on-warning true --check-secrets --check-deps --check-artifacts` as a pre-hook in your `package.json` (`"predeploy": "deployguard pre-deploy"`) or as a Git pre-push hook. This ensures no deployment proceeds if any critical rule is violated. Use `--silent` to suppress verbose output in CI environments.

## Best Practices

### Practice 1: Run Scans Before Every Commit
Integrate DeployGuard into your pre-commit hook (`deployguard scan --fail-on-warning true --check-secrets --check-deps`). This catches credential leaks and dependency tampering at the earliest possible stage, preventing them from ever reaching a remote repository. Solo developers should configure `--entropy-threshold 4.0` for stricter detection on smaller codebases.

### Practice 2: Maintain a Whitelist for False Positives
Use `.deployguard/patterns.yaml` to define explicit whitelist entries for known test credentials or internal API keys. For example:
```yaml
whitelist:
  - pattern: "AKIAIOSFODNN7EXAMPLE"
    reason: "Test AWS key for integration tests"
  - pattern: "ghp_xxxxxxxxxxxxxxxxxxxx"
    reason: "GitHub token for automated CI"
```
This reduces noise and ensures that only truly risky secrets trigger alerts, making the tool practical for daily use without desensitization.
