# deployguard - DEP_021

## Overview

DeployGuard is a production-grade static analysis engine purpose-built for solo developers and small teams who need code security auditing without the overhead of enterprise-grade CI/CD pipelines. It targets JavaScript, TypeScript, Python, and Go codebases, scanning for 200+ vulnerability patterns including hardcoded secrets, dependency confusion, and misconfigured cloud SDKs.

## Core Features

### 1. **Real-Time Secret Leak Detection with Contextual Whitelisting**
DeployGuard scans every commit and staged change for 50+ regex-based secret patterns (AWS keys, GitHub tokens, database connection strings, private SSH keys). It supports `.deployguard-ignore` files for false-positive suppression and provides a `--severity` flag (error, warning, info) to filter results. Detection latency is under 200ms per 10,000 lines of code, with PostgreSQL-backed storage for historical leak tracking.

### 2. **Dependency Graph Integrity Checker**
The tool parses `package-lock.json`, `yarn.lock`, `go.sum`, and `requirements.txt` to detect dependency confusion attacks (typosquatting, repo-jacking). It cross-references package names against a local SQLite cache of the npm/PyPI registry and flags any package with a creation date newer than 30 days or a maintainer count below 3. The `--dep-depth` parameter controls recursive scanning up to 5 levels.

### 3. **Cloud SDK Misconfiguration Scanner**
DeployGuard inspects environment variables, `.env` files, and inline code for cloud SDK configuration blocks (AWS SDK, GCP client, Azure SDK). It validates region endpoints, IAM role ARNs, and bucket lifecycle policies against a built-in ruleset of 120+ best-practice patterns. The `--cloud-provider` flag accepts `aws`, `gcp`, `azure`, or `all`. Results include a risk score (0–100) and a remediation snippet in the target language.

## How to Use

### Step 1: Initialize the Scan Configuration
Run `deployguard init --project-name my-app --lang ts --cloud-provider aws` to generate a `.deployguard.yml` file. This file defines scan depth (`max-depth: 5`), secret severity threshold (`secret-severity: error`), and dependency cache TTL (`dep-cache-ttl: 3600`). Use `--output-format json` for machine-readable results.

### Step 2: Execute a Full Security Audit
Execute `deployguard scan --path ./src --dep-depth 3 --output-report ./reports/audit.html`. The tool will:
- Parse all source files and lockfiles in `./src`
- Compare dependency trees against the local SQLite cache (auto-updated every 3600 seconds)
- Generate an HTML report with CVE references, line numbers, and suggested fixes

### Step 3: Integrate with Pre-Commit Hook
Add to your project’s `.git/hooks/pre-commit`:
```bash
#!/bin/sh
deployguard scan --staged --fail-on error --output-report /dev/null
if [ $? -ne 0 ]; then
    echo "DeployGuard: Security issues found. Commit blocked."
    exit 1
fi
```
This ensures no commit with high-severity secrets or critical dependency vulnerabilities reaches the remote repository.

## Best Practices

### Practice 1: Enforce Secret Rotation via Historical Leak Tracking
Use DeployGuard’s PostgreSQL integration to store every detected secret hash. Set up a cron job that runs `deployguard scan --full-history --compare-last-week` weekly. If the same secret hash appears in two consecutive scans, trigger an automated Slack alert and force a key rotation via your cloud provider’s API.

### Practice 2: Combine with CI/CD for Pre-Deployment Validation
In your GitHub Actions workflow, add a step that runs `deployguard scan --path ${{ github.workspace }} --fail-on warning --output-format sarif`. Upload the SARIF file to GitHub Code Scanning. This ensures that any pull request introducing a misconfigured S3 bucket or a package with a known CVE (e.g., CVE-2024-XXXX) is blocked before merging.
