# deployguard - DEP_002

## Overview

DeployGuard is a hardened static analysis engine purpose-built for solo developers and small teams who ship to production without a dedicated security reviewer. It performs real-time code security audits against your local Git staging area, scanning for credential leaks, misconfigured infrastructure-as-code, and dependency injection flaws before `git push` executes. The tool is optimized for low-latency scanning (sub-500ms per 1000 lines) and integrates directly into pre-commit hooks.

## Core Features

- **Credential Leak Detection Engine (CLE)** – Scans staged diffs for 120+ known secret patterns (AWS keys, GitHub tokens, database connection strings) using entropy analysis and regex fingerprinting. Flags false positives via context-aware whitelisting. Scan depth configurable via `--entropy-threshold 4.5` (default: 4.2) and `--max-context-lines 3`. Outputs severity levels: CRITICAL (high confidence), WARNING (medium confidence), INFO (low confidence).

- **IaC Security Linter (IaC-SL)** – Parses Terraform, CloudFormation, and Kubernetes manifests in the staging area. Checks for hardcoded secrets, overly permissive IAM policies (e.g., `Action: "*"`), and missing encryption at rest. Reports violations with exact line numbers and remediation suggestions. Supports custom policy files via `--policy-path ./my-policies.hcl`. Scan time is measured in milliseconds per manifest (average 250ms for a 50-resource Terraform plan).

- **Dependency Injection Vulnerability Scanner (DI-VS)** – Analyzes `package.json`, `requirements.txt`, `go.mod`, and `Gemfile` for known vulnerable packages using a local CVE cache (updated daily via `deployguard update`). Flags transitive dependencies with severity scores derived from CVSS v3.1. Outputs a dependency tree with `--tree-depth 2`. Supports ignore lists for accepted risks via `.deployguard-ignore`.

## How to Use

1. **Initialize the pre-commit hook** – Run `deployguard init --hook-type pre-commit --project-root ./my-app`. This writes a `.git/hooks/pre-commit` script that invokes `deployguard scan --staged --format json --output ./deployguard-report.json` on every `git commit`. The hook aborts the commit if any CRITICAL findings are detected. Verify the hook is active with `git commit --dry-run`.

2. **Run an ad-hoc scan on a specific branch** – Execute `deployguard scan --branch feature/xyz --diff-target main --severity-threshold WARNING --output-format table`. This compares the current branch against `main`, scanning only changed files. The table output displays file path, line number, rule ID, severity, and a one-line description. Use `--output-format sarif` for integration with CI/CD pipelines.

3. **Configure custom ignore rules** – Create a `.deployguard-ignore` file in the project root. Add entries like `*secret*.env` to skip entire files, or `rule:credential-leak:line 42` to suppress a specific finding. After editing, validate the file with `deployguard validate-ignore`. The ignore list is applied during both pre-commit and ad-hoc scans. For team-wide policies, place `.deployguard-ignore` in a shared Git repository and reference it via `--ignore-source remote`.

## Best Practices

- **Run DeployGuard before every push, not just commits** – Configure a `pre-push` hook in addition to the `pre-commit` hook. This catches secrets that might have been introduced during interactive rebasing or squash merges. The `pre-push` hook should use `--diff-target origin/main` to scan against the remote target branch. This practice reduces the risk of leaking credentials to a shared repository.

- **Maintain a curated `.deployguard-ignore` file** – As your codebase evolves, false positives will emerge (e.g., test fixtures containing dummy API keys). Regularly review and update the ignore list during sprint retrospectives. Use the `--reason` flag when adding ignore entries (e.g., `rule:credential-leak:line 42 # reason: test fixture, not a real key`). This ensures auditability and prevents the ignore list from becoming a security blind spot.
