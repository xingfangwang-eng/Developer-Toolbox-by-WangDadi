# deployguard - DEP_008

## Overview

DeployGuard is a high-performance, zero-config static analysis engine purpose-built for solo developers and small teams who ship code without a dedicated security review pipeline. It integrates directly into CI/CD workflows and provides real-time vulnerability detection, dependency audit trails, and runtime policy enforcement—all without sending your source code to a third-party cloud.

## Core Features

### 1. Real-Time Secret Leakage Detection with Entropy Scoring
DeployGuard scans every commit and staged change for hardcoded secrets (API keys, tokens, private keys) using a combination of regex patterns and Shannon entropy analysis. It flags high-entropy strings above a configurable threshold (default: 4.2 bits per character) and correlates them against known service patterns (AWS, GitHub, Stripe, etc.). Results include precise file path, line number, surrounding context, and a risk score (0–100). False positives are reduced by a context-aware whitelist engine that learns from your repository’s commit history.

- **Parameters:** `--entropy-threshold 4.2`, `--max-file-size 5MB`, `--context-lines 3`
- **Performance:** Scans 10,000 files in under 2.3 seconds on a 4-core machine; memory usage capped at 256MB.

### 2. Supply Chain Dependency Graph with Version Drift Analysis
DeployGuard parses `package.json`, `requirements.txt`, `go.mod`, and `Cargo.toml` to build a full dependency tree, then cross-references each package against the National Vulnerability Database (NVD) and GitHub Advisory Database. It detects version drift (e.g., a pinned minor version that is 3+ releases behind) and computes a "blast radius" metric—the number of files directly or indirectly importing a vulnerable module. Alerts are generated only when the blast radius exceeds a user-defined threshold (default: 5 files).

- **Parameters:** `--blast-radius-threshold 5`, `--advisory-refresh-interval 3600` (seconds)
- **Integration:** Outputs a machine-readable JSON report for ingestion into PostgreSQL or Datadog; supports incremental scans (delta-only) for sub-second CI feedback.

### 3. Immutable Audit Trail with Cryptographic Anchoring
Every scan result is hashed (SHA-256) and appended to a local, append-only ledger stored in SQLite. The hash chain is anchored to a configurable external timestamp authority (e.g., RFC 3161 TSA server) every 24 hours. This creates a tamper-evident log suitable for compliance requirements (SOC 2, ISO 27001). The ledger can be exported as a signed, self-contained archive for external auditors.

- **Parameters:** `--tsa-url https://timestamp.digicert.com`, `--ledger-path ./deployguard.db`
- **Technical Detail:** Hash chain uses a Merkle tree structure; verification of a single scan result requires O(log n) operations. Maximum ledger size before auto-archiving: 1 million entries (~2.1 GB).

## How to Use

### Step 1: Initialize the Project Repository
Navigate to your project root and run `deployguard init --project-name my-app --entropy-threshold 4.5 --blast-radius-threshold 3`. This creates a `.deployguard/` directory containing the configuration file (`config.yaml`), the SQLite ledger (`guardian.db`), and a `.gitignore`-ready exclusion list for false positives. The command also performs an initial full scan of the working directory and outputs a summary table with counts of secrets, vulnerabilities, and drift issues.

### Step 2: Configure CI/CD Integration
Add the following to your CI pipeline (example for GitHub Actions): `deployguard scan --ci-mode --fail-on-critical --output-format sarif`. The `--ci-mode` flag suppresses interactive output and returns a non-zero exit code if any critical-severity issue is found. The SARIF output can be uploaded directly to GitHub Code Scanning or any SARIF-compatible platform. For faster feedback, use `--incremental` to scan only files changed since the last scan (requires the ledger to exist).

### Step 3: Review and Remediate with Interactive Triage
Run `deployguard triage --open` to launch a terminal-based interactive viewer. Issues are grouped by severity (critical, high, medium, low) and can be filtered by type (secret, vulnerability, drift). For each issue, you can:
- Mark as "false positive" (adds to local whitelist with a reason string)
- Open the file at the exact line number in your default editor
- Generate a one-line fix suggestion (e.g., `git config --global secrets.allowed <pattern>`)
- Export a PDF report for stakeholder review

## Best Practices

### Practice 1: Schedule a Weekly Full Re-Scan with Fresh Advisories
Relying solely on incremental scans can miss newly published CVEs that affect already-scanned dependencies. Set up a cron job or scheduled CI trigger to run `deployguard scan --full --refresh-advisories` every Sunday at 02:00 UTC. Combine this with a Slack webhook notification (`--notify-webhook https://hooks.slack.com/...`) so the team sees a summary of any new critical findings before the workweek begins.

### Practice 2: Enforce a Pre-Commit Hook for Secret Scanning
Prevent secrets from ever reaching the remote repository by installing a Git pre-commit hook: `deployguard install-hook --hook-type pre-commit --fail-on-secret`. The hook runs `deployguard scan --staged` and blocks the commit if any secret is detected. To avoid false positives blocking legitimate commits (e.g., test fixtures with fake API keys), maintain a `.deployguard/allowlist.txt` file with regex patterns that the hook respects. This pattern reduces secret leaks by an average of 97% in production teams using this workflow.
