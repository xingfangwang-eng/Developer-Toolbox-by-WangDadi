# deployguard - DEP_016

## Overview

DeployGuard is a hardened static analysis engine purpose-built for solo developers and small teams who ship code without a dedicated security auditor. It fuses real-time code scanning with deployment gating, ensuring that no vulnerable dependency, misconfigured secret, or policy violation reaches production. The tool is optimized for low-latency CI/CD pipelines and operates entirely offline for air-gapped environments.

## Detailed Content

### What is DeployGuard?

DeployGuard is a self-contained, rule-driven static analysis tool that intercepts code at the pre-commit and pre-merge stages. It evaluates every change against a configurable security policy baseline, flagging issues with a severity score (0–100) and a confidence percentage. Unlike generic linters, DeployGuard correlates static analysis results with known CVE databases (NVD, OSV) and runtime behavior patterns to reduce false positives to under 2%. It is written in Rust for sub-50ms per-file processing and integrates natively with GitHub Actions, GitLab CI, and Jenkins.

### Core Features

- **Zero-Runtime Dependency Graph Audit**  
  DeployGuard parses `package.json`, `Cargo.toml`, `requirements.txt`, and `Gemfile` to build a full transitive dependency tree. It cross-references every package version against a local, versioned CVE index (updated daily) and halts the build if any dependency exceeds a configurable risk threshold (default: CVSS ≥ 7.0). The audit completes in under 200ms for trees with up to 5000 nodes.

- **Secret Leak Detection with Contextual Scoring**  
  Using a hybrid regex + entropy engine, DeployGuard scans for 80+ secret patterns (AWS keys, GitHub tokens, database connection strings, private SSH keys). Each potential leak is assigned a confidence score (0.0–1.0) based on surrounding code context, file type, and proximity to known false-positive patterns. High-confidence leaks (≥0.95) trigger an immediate pre-commit hook rejection; medium-confidence leaks (0.70–0.94) are logged and require manual override via a signed commit message flag.

- **Policy-as-Code (PaC) Enforcement Engine**  
  Users define deployment gates in a YAML manifest (`deployguard.yml`) with rules like `block_on_high_cve: true`, `require_signed_commits: true`, and `max_line_length: 120`. The engine evaluates each rule against the diff, producing a structured JSON report with file:line references. Rules can be scoped per branch (e.g., `main` enforces stricter rules than `develop`). The engine supports custom Lua scripts for advanced logic, such as banning `eval()` in production JavaScript or requiring `# safety` comments on every `unsafe` block in Rust.

### How to Use

1. **Initialize the Policy Baseline**  
   Run `deployguard init --project-dir . --policy-template strict` to generate a `deployguard.yml` file with default rules for your detected language stack. The tool auto-detects the package manager and sets appropriate thresholds (e.g., `max_dep_cves: 2` for Node.js, `max_secret_confidence: 0.90` for Python). Verify the generated file with `deployguard validate --policy deployguard.yml`.

2. **Integrate into Pre-Commit Hook**  
   Add the following to your `.git/hooks/pre-commit` (or use `deployguard install-hook`):  
   ```bash
   #!/bin/sh
   deployguard scan --path . --diff-mode staged --output-format table --fail-on high
   ```  
   This scans only staged files, outputs a color-coded table, and returns exit code 1 if any high-severity issue is found. For CI pipelines, use `--output-format json --fail-on critical` and parse the JSON to block merge requests.

3. **Run a Full Repository Audit**  
   Execute `deployguard audit --path . --cve-db ./cve_cache --report report.html` to perform a comprehensive scan of the entire repo, including all branches and tags. The HTML report includes an executive summary (total issues, risk score), a per-file breakdown, and a link to each CVE entry. Use `--export-sarif report.sarif` to generate a SARIF file compatible with GitHub Advanced Security.

### Best Practices

- **Layer DeployGuard with Runtime Monitoring**  
  DeployGuard excels at static pre-deployment checks, but it cannot detect runtime injection or misconfigurations that only appear under load. Integrate it with a runtime security agent (e.g., Falco, Datadog ASM) to catch anomalies post-deployment. For example, if DeployGuard blocks a high-CVE dependency during build, the runtime agent can still alert if the same code path is exploited in production.

- **Version-Lock the CVE Index in CI**  
  Always pin the CVE database version in your CI configuration to ensure deterministic builds. Use `deployguard update-cve --version 2025-01-15` and store the index in a versioned artifact (e.g., S3 bucket or Git LFS). This prevents a new CVE release from suddenly breaking your pipeline mid-sprint. Schedule weekly updates via cron and review the diff before applying.
