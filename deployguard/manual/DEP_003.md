# deployguard - DEP_003

## Overview

DeployGuard is a deterministic, zero-configuration static analysis engine tailored for solo developers and small teams who ship code without a dedicated security layer. It performs real-time code security audits at the commit and pre-deployment stage, with a focus on detecting secret leakage, misconfigured CI/CD pipelines, and dependency injection vulnerabilities in JavaScript, TypeScript, and Python codebases. The engine operates entirely offline, with audit results generated in under 120 milliseconds for projects up to 50,000 lines of code.

## Detailed Content

### What is DeployGuard?

DeployGuard is not a general-purpose linter or a CI plugin. It is a context-aware security gate that sits between your local environment and your deployment target (e.g., a self-hosted Kubernetes cluster or a VPS). It hooks into the `pre-commit` and `pre-push` hooks via a lightweight Rust-based binary (under 3 MB) and executes a set of deterministic rules against the diff, not the full repository. This allows for sub-100ms scans even on large monorepos. The tool outputs a structured JSON report that can be consumed by deployment orchestrators like Kamal, Capistrano, or even a simple bash pipeline.

### Core Features

- **Smart Secret Pattern Scanner (SSPS)**  
  Scans staged diffs against a database of 200+ entropy-based and regex-driven patterns (e.g., AWS IAM keys, GitHub personal access tokens, PostgreSQL connection strings). Unlike `gitleaks`, DeployGuard uses a two-pass algorithm: first, a fast linear scan for high-entropy strings, then a secondary context check to reduce false positives by 94%. Supports custom pattern files in YAML format with a `pattern_id`, `regex`, and `severity` field (e.g., `severity: critical` blocks deployment).

- **CI/CD Pipeline Integrity Checker (PIC)**  
  Validates your deployment pipeline YAML files (`.github/workflows/*.yml`, `.gitlab-ci.yml`, `deploy.yml` for Kamal) against a set of 35 security rules. It checks for hardcoded runner tokens, unrestricted `pull_request_target` triggers, and missing `secrets` scoping. Outputs a `pipeline_risk_score` (0–100) with a threshold parameter `--pipeline-threshold=70` that can fail the deployment if exceeded.

- **Dependency Injection Attack Surface Analyzer (DIASA)**  
  For Node.js and Python projects, DeployGuard performs a static call-graph analysis to detect unsafe `eval()`, `exec()`, or `child_process.spawn()` calls that accept user-controlled input from query parameters, environment variables, or file uploads. It reports the exact file path, line number, and the propagation chain (e.g., `line 42: user_input -> sanitize() -> eval()`). The analysis is capped at 5000 AST nodes per file to guarantee sub-second performance.

### How to Use

1. **Install the binary and configure the pre-commit hook**  
   Run `curl -sSL https://deployguard.io/install.sh | bash` to download the `deployguard` binary to `/usr/local/bin`. Then execute `deployguard init --hook pre-commit` to generate a `.deployguard.yml` configuration file in your project root. The hook will automatically run `deployguard scan --diff --output=json` before each commit. If any `critical` or `high` severity issue is found, the commit is blocked with a detailed error message.

2. **Run a full repository scan with custom pattern file**  
   Use `deployguard scan --path ./src --patterns ./custom-patterns.yml --output report.json --pipeline-threshold=70`. This scans all files under `./src` (excluding `node_modules` and `.git` by default) using both built-in patterns and your custom rules. The `--pipeline-threshold` flag sets the maximum acceptable pipeline risk score; if the score exceeds 70, the scan exits with code 1. The report includes a `scan_duration_ms` field (e.g., `"scan_duration_ms": 87`) and a `total_issues` count.

3. **Integrate with a deployment orchestrator (e.g., Kamal)**  
   Add the following to your `config/deploy.yml` before the `builder` section:  
   ```yaml
   hooks:
     pre-deploy: "deployguard scan --diff --pipeline-threshold=70 --output=json"
   ```  
   This ensures that before Kamal builds the Docker image, DeployGuard runs a diff scan against the last commit. If the scan fails, Kamal aborts the deployment with a non-zero exit code. You can also pipe the JSON report to a logging service like `jq` for further processing.

### Best Practices

- **Use diff-only scanning in CI/CD pipelines**  
  Running a full repository scan on every push is wasteful and slow. DeployGuard's `--diff` flag only analyzes the changed lines between the current commit and the previous one. This reduces scan time to under 50 milliseconds for typical commits. Combine this with a scheduled full scan (e.g., via a weekly GitHub Actions workflow) to catch issues in legacy code.

- **Define severity thresholds per environment**  
  In your `.deployguard.yml`, set `severity_threshold: high` for production deployments and `severity_threshold: medium` for staging. This prevents blocking a staging deploy for a low-severity warning (e.g., a deprecated function call) while still catching critical secret leaks. Use the `--severity` flag to override at runtime: `deployguard scan --diff --severity=critical`.
