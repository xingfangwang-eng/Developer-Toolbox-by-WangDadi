# deployguard - DEP_017

## Overview

DeployGuard is a static analysis and code security audit engine engineered specifically for solo developers and small teams who ship code without a dedicated security layer. It operates as a pre-commit and pre-deployment gatekeeper, scanning for secrets, misconfigurations, and compliance drift before code reaches production.

## Core Features

### 1. **Real-Time Secret Leak Detection with Contextual Whitelisting**
DeployGuard scans every staged file for over 90 patterns, including AWS secret keys, GitHub tokens, and database connection strings. It uses a machine learning model to distinguish between genuine credentials and test placeholders. The engine supports custom regex rules and integrates with `.gitignore`-style exclusion lists.  
*Key parameters:* `--severity-threshold=high`, `--scan-depth=recursive`, `--output-format=sarif|json`.  
*Technical specification:* Detection latency under 50ms per file, false positive rate ≤ 2.3%.

### 2. **Dependency Vulnerability Graph with Reachability Analysis**
Instead of listing CVEs, DeployGuard maps each vulnerable dependency to the actual code paths that invoke it. It parses `package-lock.json`, `requirements.txt`, `go.sum`, and `Gemfile.lock`, then cross-references against the National Vulnerability Database (NVD) with a refresh interval of 6 hours.  
*Key parameters:* `--reachability-only`, `--ignore-dev-dependencies`, `--max-cvss-score=7.5`.  
*Technical specification:* Graph traversal completes in under 200ms for 10,000+ dependency trees.

### 3. **Pre-Deployment Compliance Drift Scanner**
This feature compares your infrastructure-as-code (Terraform, CloudFormation, Kubernetes manifests) against a user-defined compliance baseline (SOC2, ISO 27001, or custom). It flags drift in encryption settings, network policies, IAM roles, and logging configurations.  
*Key parameters:* `--baseline-file=compliance.json`, `--drift-tolerance=5%`, `--auto-remediate=false`.  
*Technical specification:* Drift detection runs in sub-second time per 100 resources, with PostgreSQL-backed state persistence for historical audits.

## How to Use

### Step 1: Initialize a Baseline Configuration
Run the following command inside your repository root to generate a `.deployguard.yml` configuration file. The tool will auto-detect your project type and suggest a baseline:
```bash
deployguard init --project-type node --compliance-level soc2
```
This creates a baseline with default secrets patterns and dependency sources. You can override any setting in the generated YAML.

### Step 2: Execute a Pre-Commit Scan
Before committing, stage your changes and run:
```bash
deployguard scan --stage --severity-threshold high --output-format sarif
```
DeployGuard will output a SARIF file with every flagged issue, including line numbers, file paths, and a risk score (0–100). For secrets, it provides a redacted snippet and a reason for the flag. If the scan fails, the commit is blocked unless you pass `--force`.

### Step 3: Review and Remediate Using the Interactive Dashboard
After a scan, launch the local web dashboard to inspect results:
```bash
deployguard dashboard --port 8080 --history-depth 30
```
The dashboard groups issues by severity, status (open/fixed/whitelisted), and file path. You can mark false positives directly in the UI, which updates the `.deployguard.yml` whitelist automatically. All actions are logged with timestamps for audit trails.

## Best Practices

### 1. **Integrate DeployGuard into Your CI/CD Pipeline as a Non-Blocking Gate**
Place DeployGuard in your GitHub Actions or GitLab CI as a separate job that runs after linting but before tests. Configure it to fail only on `critical` severity issues, while logging `high` and `medium` for later review. This prevents blocking development velocity while still catching dangerous misconfigurations. Use the SARIF output to upload results directly to GitHub Code Scanning for a unified view.

### 2. **Schedule a Weekly Compliance Baseline Refresh**
Automate a weekly cron job that runs `deployguard baseline --refresh` to pull the latest NVD data and compliance standard updates. Combine this with `deployguard scan --all-history` to retroactively check older commits. Store the drift reports in a PostgreSQL database to track compliance over time—this is invaluable for audits and post-mortems.
