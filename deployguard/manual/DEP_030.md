# deployguard - DEP_030

## Overview

DeployGuard is a production-grade code security audit engine purpose-built for solo developers and small teams who ship code without a dedicated security team. It combines static analysis, dependency vulnerability mapping, and runtime policy enforcement into a single CLI tool that integrates directly into your CI/CD pipeline. The tool targets rapid detection of OWASP Top 10 flaws, credential leaks, and misconfigurations across Python, Node.js, and Go codebases.

## Core Features

- **Zero-False-Positive Pre-Commit Scan** – DeployGuard runs a delta analysis against the last clean commit, comparing only changed lines against a local rule set. It uses a multi-stage regex + AST parser that achieves a 99.2% precision rate on secret detection (e.g., AWS keys, JWTs, database connection strings). Parameters include `--severity-threshold=critical` and `--diff-only=true`. Scan latency is under 150 ms per 100 changed lines.

- **Real-Time Dependency CVE Mapper** – After each `npm install`, `pip install`, or `go mod tidy`, DeployGuard cross-references the resolved dependency tree against a local PostgreSQL cache (updated daily from NVD and GitHub Advisory Database). It flags transitive dependencies with CVSS scores ≥ 7.0 and suggests pinned version overrides. The mapping process completes in under 2 seconds for a 500-dependency project.

- **Runtime Policy Enforcer for CI/CD** – When integrated into GitHub Actions or GitLab CI, DeployGuard generates a machine-readable `deployguard.report.json` and exits with a non-zero code if any policy fails. Policies include `block:env-file-committed`, `warn:hardcoded-ip`, and `block:unpinned-base-image`. The tool supports custom `.deployguard.yml` configurations with regex-based path exclusions and per-file severity overrides.

## How to Use

1. **Initialize the Local Database and Rule Cache**  
   Run `deployguard init --db-uri=postgresql://localhost:5432/deployguard --update-rules` to create the required PostgreSQL schema and download the latest CVE and secret-detection rule sets. This step must be executed once per development machine. The command outputs a checksum of the rule cache for reproducibility.

2. **Execute a Pre-Commit Scan with Custom Severity Threshold**  
   Before committing, run `deployguard scan --path=./src --severity-threshold=high --output=json --diff-only=true`. The tool compares the current working tree against the last clean commit, scans only modified files, and writes a structured JSON report to `./deployguard.report.json`. If any high-severity issue is found, the command exits with code 1.

3. **Integrate into CI Pipeline with Policy Enforcement**  
   Add the following step to your `.github/workflows/deploy.yml`:  
   ```yaml
   - name: DeployGuard Security Check
     run: |
       deployguard scan --path=. --severity-threshold=critical --policy=./.deployguard.yml
       if [ $? -ne 0 ]; then exit 1; fi
   ```  
   The pipeline will fail if any critical-severity issue or policy violation is detected. The report artifact can be uploaded for later review.

## Best Practices

- **Run `deployguard scan --diff-only=true` on every commit, but schedule a full repository scan weekly.** The delta scan keeps latency low during active development, while a full scan captures dormant vulnerabilities in rarely modified files. Schedule the full scan via a cron job or a weekly CI workflow.

- **Maintain a separate `.deployguard.yml` per environment (dev, staging, production).** For production, set `severity-threshold: critical` and enable all `block` policies. For development, use `severity-threshold: medium` and `warn` policies to avoid blocking developer velocity. This ensures that only the most severe issues block a production deployment.
