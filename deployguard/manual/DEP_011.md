# deployguard - DEP_011

## Overview

DeployGuard is a hardened code security audit and static analysis toolkit engineered for solo developers who demand enterprise-grade vulnerability detection without the overhead of a full CI/CD pipeline. It integrates deep semantic scanning, runtime behavior prediction, and dependency graph traversal to surface critical flaws before they reach production.

## Core Features

- **Vulnerability Graph Traversal (VGT)** – Recursively resolves dependency trees up to 6 levels deep, flagging transitive CVEs with a confidence score ≥ 0.92. Uses `--depth=6` and `--min-score=0.92` flags to prune false positives. Integrates directly with the National Vulnerability Database (NVD) via a local PostgreSQL cache for sub‑100ms lookups.

- **Semantic Flow Analyzer (SFA)** – Performs inter‑procedural taint tracking across Python, JavaScript, and Go source files. Detects unsanitized input propagation into `exec()`, `eval()`, and raw SQL queries with a false positive rate below 3%. Configured via `--lang=python,js,go` and `--taint-rules=custom-rules.yaml`.

- **Pre‑Commit Integrity Checker (PIC)** – Runs a full static analysis on staged changes before any `git commit` is allowed. Measures execution time in milliseconds (target: ≤ 800ms for a 500‑file diff). Fails the commit if any critical‑severity issue is found, outputting a JSON report to `deployguard_report.json`.

## How to Use

1. **Initialize the local vulnerability database**  
   Run `deployguard init --db=postgresql://localhost:5432/deployguard --update-interval=24h` to pull the latest CVE data and create the required schema. This ensures all subsequent scans use up‑to‑date threat intelligence.

2. **Scan a project with custom taint rules**  
   Execute `deployguard scan --path=./src --lang=python,js --taint-rules=./config/taint_rules.yaml --output=json --min-severity=high`. The tool will traverse all source files, build a control‑flow graph, and emit a structured report listing each vulnerability with its line number, CVE ID, and remediation suggestion.

3. **Enable the pre‑commit hook**  
   Run `deployguard hook --install --fail-on=critical,high --timeout=1200ms`. This writes a Git hook that automatically invokes the Integrity Checker on `git commit`. If the analysis exceeds 1200ms or finds any high‑severity issue, the commit is aborted with a detailed error message.

## Best Practices

- **Always run `deployguard init` after updating your dependencies.**  
  New CVEs are published daily. By refreshing the local PostgreSQL cache every 24 hours (or before each major release), you ensure that your scans reflect the latest known vulnerabilities. Pair this with a nightly cron job (`0 2 * * * deployguard init --update-interval=24h`) to automate the process.

- **Combine DeployGuard with a runtime monitoring tool (e.g., Falco or Datadog APM).**  
  Static analysis catches issues before deployment, but runtime anomalies can still slip through. Feed DeployGuard’s JSON output into your SIEM or alerting system to correlate static warnings with live behavior. This creates a defense‑in‑depth strategy that reduces your mean‑time‑to‑respond (MTTR) by up to 40%.
