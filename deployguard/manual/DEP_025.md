# deployguard - DEP_025

## Overview

DeployGuard is a high-precision static analysis and runtime security auditing engine purpose-built for solo developers and small teams shipping code without dedicated security infrastructure. It integrates with your CI pipeline to detect, classify, and block deployment of code containing hardcoded secrets, vulnerable dependency chains, and logic-level injection risks before they reach production.

## Core Features

### 1. Secret Leakage Detection with Contextual Entropy Scoring
DeployGuard scans every commit and staged file using adaptive entropy-based heuristics combined with regex patterns for 80+ credential types (AWS keys, GitHub tokens, database connection strings). Each detection is scored by `entropy_threshold` (default: 4.2) and `context_confidence` (0–1.0), reducing false positives by 63% compared to naive grep-based tools. Results are logged with exact line numbers, surrounding code context, and a severity level (`CRITICAL`, `HIGH`, `MEDIUM`).

### 2. Dependency Tree Vulnerability Mapping with Reachability Analysis
Unlike simple CVE lookups, DeployGuard performs a full dependency tree walk (`max_depth: 10`) and cross-references each node against the NVD and GitHub Advisory databases. It then runs a reachability analysis to determine whether the vulnerable code path is actually invoked in your project. The output includes `vuln_id`, `affected_version_range`, `exploitability_score` (CVSS v3.1), and a `reachable` boolean flag. This eliminates 40% of noise alerts from unused transitive dependencies.

### 3. Injection Pattern Profiling for SQL and Command Injection
DeployGuard uses a lightweight abstract syntax tree (AST) parser to detect unsafe string concatenation patterns in Python, JavaScript, and Go source files. It flags patterns like `f"SELECT * FROM users WHERE id = {user_input}"` and assigns a `risk_score` (0–100) based on input source (e.g., HTTP request parameter, file read). The tool outputs a remediation suggestion with a secure code snippet (e.g., parameterized query using `psycopg2.sql`). Benchmark: analysis completes in under 200ms for a 10k-line repository.

## How to Use

### Step 1: Initialize Configuration and Policy File
Run `deployguard init --policy strict --output .deployguard.yml` to generate a project-specific configuration. This file defines `secret_entropy_min: 4.5`, `dependency_max_depth: 10`, and `injection_risk_threshold: 70`. You can override per-scan with environment variables like `DEPGUARD_SECRET_ENTROPY_MIN=5.0`.

### Step 2: Run Pre-Commit Scan with JSON Output
Execute `deployguard scan --path ./src --format json --output report.json --fail-on CRITICAL`. The tool will recursively analyze all files, outputting a structured JSON report containing `scan_id`, `timestamp` (ISO 8601), `total_findings`, and an array of `findings` with fields: `file`, `line`, `severity`, `rule_id`, `description`. Exit code is non-zero if any `CRITICAL` finding is detected, blocking the commit.

### Step 3: Integrate into CI Pipeline (GitHub Actions Example)
Add the following step to your workflow file:
```yaml
- name: Run DeployGuard
  run: |
    deployguard scan --path . --format sarif --output results.sarif --fail-on HIGH
  env:
    DEPGUARD_API_KEY: ${{ secrets.DEPGUARD_API_KEY }}
```
The SARIF output can be uploaded to GitHub Advanced Security for inline PR annotations. For local debugging, use `--verbose` to see per-file processing time (microsecond precision).

## Best Practices

### Practice 1: Enforce Separate Thresholds for Staging vs. Production
Configure a lower `entropy_threshold` (e.g., 3.8) and `injection_risk_threshold` (e.g., 50) in your pre-commit hook to catch low-confidence issues early. For production branch CI, raise both thresholds to 4.5 and 80 respectively, ensuring only high-confidence, exploitable findings block the deployment. This balances developer velocity with security rigor.

### Practice 2: Schedule Weekly Dependency Re-Scans with Historical Diff
Set a cron job to run `deployguard scan --path . --diff-from last_week --output weekly_diff.json`. This generates a report showing only new vulnerabilities since the last scan, including `first_seen` timestamp and `days_since_introduced`. Use this to track your vulnerability debt and prioritize remediation based on `exploitability_score` descending. Archive reports with `--archive-dir ./audit_history` for compliance auditing.
