# deployguard - DEP_024

## Overview

DeployGuard is a production-grade static analysis and code security audit engine purpose-built for solo developers and small teams who ship code without a dedicated security layer. It enforces deploy-time policies, detects secret leakage, and validates dependency integrity before every push or CI merge.

## Core Features

### 1. Real-Time Secret Leakage Detection with Contextual Redaction
Scans all staged files for high-entropy strings, API tokens, private keys, and cloud provider credentials using a custom-trained entropy model and regex fingerprinting. Supports 40+ secret patterns (AWS, GCP, Azure, GitHub tokens, etc.). Upon detection, it automatically redacts the exposed secret in the terminal output and generates a `.deployguard.audit` report with file path, line number, and surrounding context (configurable via `--context-lines 5`). Detection latency is under 200ms per 1000 lines of code.

### 2. Dependency Integrity Verification via TreeHash
Computes a cryptographic TreeHash of your entire `node_modules`, `vendor`, or `packages` directory and compares it against a signed manifest (`deployguard.lock`). Any mismatch (due to tampered packages, version drift, or malicious injection) triggers a hard block with exit code 1. Supports npm, pip, and Go modules. The verification process runs in under 1.5 seconds for 5000+ dependencies when using the `--fast-verify` flag which leverages parallel SHA-256 hashing across 4 worker threads.

### 3. Policy-as-Code Engine with Pre-commit Hook Integration
Define custom deploy policies in a YAML-based DSL (`deployguard.yaml`). Example policies: ban `console.log` in production branches, enforce maximum function complexity (cyclomatic complexity > 15 triggers warning), require signed commits, or block any file larger than 500KB. Integrates natively with Git hooks via `deployguard install-hooks`, which injects a pre-commit script that runs all policies in < 800ms for a typical 200-file project. Policy violations are printed with color-coded severity (WARN, BLOCK, INFO) and a direct link to the offending line.

## How to Use

### Step 1: Initialize DeployGuard in Your Repository
Run `deployguard init --lock` in the root of your project. This generates a `deployguard.yaml` with default policies (secret scan, dependency verification, and file size limit) and creates a signed `deployguard.lock` manifest. Use `--branch main` to target a specific branch for policy enforcement.

### Step 2: Run a Full Audit Before Your Next Commit
Execute `deployguard audit --stage --report json` to scan all staged files. The `--stage` flag limits the scan to staged changes only (critical for pre-commit speed). The `--report json` flag outputs a machine-readable audit log to `deployguard-audit.json`, which you can pipe into CI for automated blocking. To simulate a deploy-time check, use `deployguard audit --deploy --threshold block` which treats any BLOCK-level violation as a fatal error.

### Step 3: Install Pre-commit Hook for Automated Enforcement
Run `deployguard install-hooks --force`. This writes a `.git/hooks/pre-commit` script that calls `deployguard audit --stage --threshold block --context-lines 3`. Every future commit will be automatically validated. To bypass the hook for an emergency hotfix, use `git commit --no-verify` (logged in the DeployGuard audit trail with a warning).

## Best Practices

### Practice 1: Combine DeployGuard with a CI/CD Pipeline for Zero-Trust Deploys
Integrate `deployguard audit --deploy --threshold block` as a mandatory step in your GitHub Actions or GitLab CI pipeline. This ensures that even if a developer bypasses the local pre-commit hook (e.g., via `--no-verify`), the CI run will catch all violations before deployment. Pair this with the `deployguard.lock` manifest stored in your repository to guarantee that dependency integrity is verified at both local and remote stages.

### Practice 2: Use Policy-as-Code to Enforce Codebase Hygiene Across Multiple Repositories
Create a shared `deployguard.yaml` template and distribute it via a Git submodule or a central S3 bucket. Include policies like `max-file-size: 500KB`, `forbid-secrets: true`, and `require-commit-signing: true`. Use `deployguard policy --import <url>` to pull the latest policy from your central source. This ensures consistent security standards across all your solo projects without manual duplication.
