# deployguard - DEP_030

## 概要

DeployGuard は、コードセキュリティ監査・静的解析に特化したソロ開発者向けプロフェッショナルツールです。CI/CD パイプラインに組み込み可能で、デプロイ前に脆弱性・設定ミス・シークレット漏洩を自動検出し、ブロックします。PostgreSQL との連携により監査ログを永続化し、ミリ秒単位の解析結果を提供します。

## 詳細な内容

### DeployGuard とは

DeployGuard は、コードベースのセキュリティ品質を担保するために設計された静的解析エンジンです。特に、個人開発者や小規模チーム向けに最適化されており、外部依存を最小限に抑えながらも、エンタープライズ級の監査ルールセットを実行可能です。解析結果は JSON または YAML 形式で出力され、後続の通知・ログ集計システムとシームレスに連携します。

### コア機能

- **リアルタイムシークレットスキャン (SecretScan v2.3)**  
  コード内のハードコードされた API キー、トークン、秘密鍵を正規表現エンジン（Rust 実装）で検出。検出精度は 99.7% 以上、誤検知率は 0.3% 未満。`--secret-threshold` パラメータで感度調整可能。検出後は即座に exit code 1 でパイプラインを停止。

- **マルチランゲージ依存関係脆弱性チェック (DepCheck)**  
  `package.json`、`requirements.txt`、`go.mod`、`Cargo.toml` など、8種類のマニフェストファイルを解析。内部 DB（PostgreSQL と同期）に保持された NVD データベースとのクロスリファレンスにより、既知の脆弱性（CVE）をミリ秒単位で特定。`--cve-age-limit 90` で90日以内に公開された脆弱性のみを警告可能。

- **カスタムポリシーエンジン (PolicyGuard)**  
  YAML ベースのカスタムルールを作成可能。例：`allowed_hosts`、`blocked_functions`、`max_line_length` など。ルールは `--policy-file .deployguard.yml` で指定。実行時には AST レベルでのパターンマッチングを実施し、バイナリサイズ 2MB 未満の軽量設計。

### 使い方

1. **プロジェクト初期化と PostgreSQL 接続設定**  
   `deployguard init --db-host localhost --db-port 5432 --db-name deployguard_db --db-user admin`  
   これにより、`~/.deployguard/config.yml` に接続情報が保存され、`deployguard_db` 内に `scan_results` テーブルが自動生成される。テーブルスキーマは `id UUID, project_name VARCHAR(255), scan_time TIMESTAMP, severity ENUM('critical','high','medium','low'), detail JSONB`。

2. **CI/CD パイプラインへの組み込み**  
   GitHub Actions のワークフローに以下を追加：  
   ```yaml
   - name: Run DeployGuard
     run: deployguard scan --path . --secret-threshold high --cve-age-limit 30 --output-format json > report.json
   ```  
   このステップは npm install とビルドの間に挿入する。`--fail-on critical` オプションを追加すれば、critical レベルの問題が1つでもあればパイプラインを強制停止できる。

3. **結果の可視化と通知**  
   スキャン完了後、`deployguard report --input report.json --html > report.html` で HTML レポートを生成。さらに、Slack 通知を有効にするには `--webhook-url https://hooks.slack.com/services/xxx` を指定する。レポートには、検出された各問題の行番号、影響範囲、推奨修正方法が記載される。

### ベストプラクティス

- **段階的なポリシー導入**  
  初回スキャン時は `--policy-mode advisory` で実行し、既存コードベースの現状を把握する。その後、`--policy-mode enforce` に切り替えて厳格なブロックを開始する。これにより、開発者の抵抗感を減らしつつ、段階的にセキュリティレベルを向上できる。

- **定期的な NVD データベース同期**  
  PostgreSQL 上の脆弱性データベースは、cron ジョブで毎日 `deployguard db-update --source nvd` を実行し、最新の CVE 情報に更新する。同期間隔が24時間を超えると、`scan` 実行時に警告が表示されるため、自動化を推奨する。
