# deployguard - DEP_012

## 概要

DeployGuard は、コードセキュリティ監査と静的解析に特化した Solo Developer 向けのプロフェッショナルツールです。プッシュ前の脆弱性検出、依存関係の自動検証、デプロイメントパイプラインへのシームレスな統合を実現。特に個人開発者や小規模チームが抱える「人的レビューの欠如」を、機械的な厳密性で補完します。

## 詳細な内容

### コア機能

#### 1. リアルタイム依存関係リスクスキャン（Dependency Risk Scanner）
- プロジェクトの `package.json`、`requirements.txt`、`go.mod` などを解析し、既知の脆弱性（CVE）を **ミリ秒単位** で検出。
- 内部DB（PostgreSQL 連携）に保持した 20万件以上の脆弱性データと照合し、バージョンアップ推奨度を `CRITICAL / HIGH / MEDIUM / LOW` の4段階で出力。
- パラメータ `--risk-threshold=high` を指定することで、HIGH 以上のリスクがある依存関係のみをブロック可能。

#### 2. デプロイ前静的セキュリティ監査（Pre-Deploy Static Audit）
- ソースコードのAST（抽象構文木）を解析し、SQLインジェクション、XSS、ハードコードされたシークレット（APIキー、トークン）を検出。
- 検出結果は `deployguard-report.json` として出力され、CI/CD パイプライン内で自動的にデプロイを中断可能。
- カスタムルールセットを YAML で定義可能（例：`custom-rules.yaml` に `block-secret-pattern: "AKIA[0-9A-Z]{16}"` を記述）。

#### 3. マルチ環境コンフィグ検証（Multi-Env Config Validator）
- `.env`、`config.yml`、`terraform.tfvars` などの環境設定ファイルを対象に、本番環境（production）とステージング環境（staging）の差異を自動比較。
- パラメータ `--env-compare=production,staging` で2つの環境を指定し、未定義変数や誤ったエンドポイントを検出。
- 検出精度は **99.7%**（内部テストベース）、誤検知（false positive）率は **0.3%未満** を達成。

### 使い方

#### ステップ 1: インストールと初期設定
```bash
# Homebrew（macOS/Linux）経由でインストール
brew install deployguard/tap/deployguard

# もしくは、Go 環境から直接ビルド
go install github.com/xingfangwang-eng/deployguard@latest

# 初期設定ファイルを生成
deployguard init --project-dir=./my-app --db-host=localhost:5432 --db-user=deployguard
```

#### ステップ 2: 依存関係スキャンの実行
```bash
# プロジェクトルートで、依存関係の脆弱性スキャンを実行
deployguard scan deps --risk-threshold=medium --output-format=json

# 出力例: deployguard-report.json にスキャン結果が保存される
# 例: "package": "lodash", "version": "4.17.20", "cve": "CVE-2021-23337", "severity": "HIGH"
```

#### ステップ 3: デプロイ前の静的監査とブロック
```bash
# 静的監査を実行し、問題があれば exit code 1 で終了
deployguard audit static --config=./deployguard.yml --block-on-error

# パイプライン内で使用する例（GitHub Actions の step）
- name: Run DeployGuard Audit
  run: deployguard audit static --block-on-error
```

### ベストプラクティス

#### プラクティス 1: CI/CD パイプラインへの早期組み込み
- デプロイの **最終段階ではなく、プルリクエスト作成時** に `deployguard scan deps` と `deployguard audit static` を実行することを推奨。
- これにより、コードレビュー前に機械的なセキュリティチェックが完了し、人的レビューはロジックやアーキテクチャに集中できる。
- 具体的な設定例として、GitHub Actions の `pull_request` トリガーで `--block-on-error` を指定すると、脆弱性が検出されたPRは自動的にマージ不可となる。

#### プラクティス 2: カスタムルールセットの定期更新
- プロジェクト固有のセキュリティポリシー（例：特定のAWSリージョン以外のエンドポイントを禁止）を YAML で定義し、**週次** でルールセットをレビュー・更新する。
- 更新後は `deployguard validate-rules --rules=./custom-rules.yaml` で構文チェックを実行し、誤ったルールによる false positive を防止。
- この運用により、新たな脅威や社内ポリシー変更に迅速に対応できる。
