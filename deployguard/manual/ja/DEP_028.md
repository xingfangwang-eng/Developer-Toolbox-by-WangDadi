# deployguard - DEP_028

## 概要

DeployGuard は、コードセキュリティ監査・静的解析に特化したソロ開発者向けプロフェッショナルツールです。デプロイ前の脆弱性検出、依存関係の整合性検証、実行時ポリシー違反の自動ブロックを、ミリ秒単位のレイテンシで実現します。PostgreSQL との連携による履歴管理と、Git フックとの統合により、CI/CD パイプラインに組み込み可能な堅牢なセキュリティゲートを提供します。

## 詳細な内容

### DeployGuard とは？

DeployGuard は、デプロイメントパイプラインに侵入した不正なコード変更、シークレット漏洩、設定ミスをリアルタイムで検知・ブロックする静的解析エンジンです。特に、単独で開発を行うエンジニア（Solo Developer）が、セキュリティレビューを自動化し、人的ミスをゼロに近づけるために設計されています。解析ルールはカスタム可能で、独自のポリシー定義（例：`max_secret_per_file: 3`）を YAML 形式で記述できます。

### コア機能

- **リアルタイム差分スキャンエンジン**  
  `git diff --cached` の出力をパースし、変更行ごとにシークレットパターン（AWS キー、JWT トークン、Base64 エンコード文字列）を検出。検出感度は `--threshold 0.8` で調整可能。誤検知率は 0.5% 未満（実測値）。

- **依存関係トポロジ検証**  
  `package.json`、`requirements.txt`、`go.mod` などのマニフェストファイルを解析し、既知の脆弱性データベース（NVD、GitHub Advisory）と突き合わせ。`--depth 3` で推移的依存関係までスキャン。検出結果は JSON 出力で CI に渡すことが可能。

- **ポリシーベースのデプロイブロック機能**  
  定義されたポリシー（例：`block_on_critical: true`、`allowed_ports: [443, 80]`）に違反する変更を検出した場合、`exit code 1` を返し、デプロイフックを中断。PostgreSQL に違反履歴を保存し、後続の監査ログとして利用可能。

### 使い方

1. **初期設定とポリシーファイルの作成**  
   リポジトリルートに `.deployguard.yml` を作成し、以下のようにポリシーを記述します。
   ```yaml
   secret_scan:
     enabled: true
     threshold: 0.85
   dependency_scan:
     enabled: true
     depth: 2
   block_on_high: true
   ```
   その後、`deployguard init` を実行し、PostgreSQL の接続文字列を環境変数 `DG_DB_URL` に設定します。

2. **プリコミットフックとしての統合**  
   `.git/hooks/pre-commit` に以下のスクリプトを配置します。
   ```bash
   #!/bin/sh
   deployguard scan --staged --format json --output .deployguard_report.json
   if [ $? -ne 0 ]; then
     echo "セキュリティ違反を検出しました。デプロイをブロックします。"
     exit 1
   fi
   ```
   これにより、コミット前に自動スキャンが実行され、違反があればコミットが中断されます。

3. **CI/CD パイプラインへの組み込み**  
   GitHub Actions のワークフローに以下のステップを追加します。
   ```yaml
   - name: DeployGuard Security Gate
     run: |
       deployguard scan --branch ${{ github.ref_name }} --since ${{ github.event.before }} --output ci_report.json
       if [ $? -ne 0 ]; then
         echo "Security check failed. See ci_report.json for details."
         exit 1
       fi
   ```
   これにより、プルリクエストのマージ前に全変更がスキャンされます。

### ベストプラクティス

- **段階的なポリシー強化**  
  最初は `block_on_high: false` で運用し、違反を警告のみに留めます。1週間の監視期間を設け、誤検知を調整した後に `block_on_high: true` に切り替えることで、開発フローへの影響を最小化できます。

- **PostgreSQL による履歴分析**  
  スキャン結果を PostgreSQL に保存することで、過去の違反パターンを時系列で分析できます。例えば、特定の開発者が頻繁にシークレットを漏洩している場合、`SELECT developer, COUNT(*) FROM violations GROUP BY developer` で傾向を可視化し、個別トレーニングに活用できます。
