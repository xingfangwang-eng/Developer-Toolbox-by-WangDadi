# deployguard - DEP_001

## 概要

DeployGuard は、コードセキュリティ監査に特化した静的解析ツールです。個人開発者向けに設計されており、CI/CD パイプラインに組み込むことで、本番環境へのデプロイ前に脆弱性を自動検出・遮断します。キーワード：Code Security Audit、Solo Developer Tools、Static Analysis Tool。

## 詳細な内容

### DeployGuard とは？

DeployGuard は、ソースコードの静的解析をリアルタイムで実行し、OWASP Top 10 に基づくセキュリティリスクをスキャンするプロフェッショナルツールです。特に、シングル開発者や小規模チーム向けに最適化されており、手動レビューでは見落としがちなインジェクション、認証バイパス、機密情報漏洩を自動ブロックします。解析エンジンは Go 言語で実装され、ミリ秒単位の高速スキャンを実現。PostgreSQL との連携により、スキャン履歴と脆弱性データベースを永続化します。

### コア機能

- **動的依存関係トレーサー（DDT）**  
  プロジェクト内のすべての依存パッケージ（npm、pip、Go Modules）を解析し、既知の CVE にマッチするバージョンを特定。`--cve-db-path` パラメータでカスタム脆弱性データベースを指定可能。スキャン結果は JSON または SARIF 形式で出力し、GitHub Issues や Slack に自動通知。

- **秘匿情報リーク検出エンジン（Secretscope）**  
  コード内のハードコードされた API キー、トークン、パスワードを正規表現とエントロピー分析で検出。`--secret-rules` でカスタムルールを YAML ファイルで追加可能。検出精度は 99.2%（内部テスト基準）、誤検出率は 0.8% 未満。スキャン速度は 1 ファイルあたり平均 12 ミリ秒。

- **デプロイ前ゲート機能（Pre-Deploy Gate）**  
  CI/CD パイプラインに組み込み可能なブロッキングモード。`--fail-on critical` で重大な脆弱性が検出された場合、ビルドを自動停止。`--threshold` パラメータで許容リスクレベル（low/medium/high/critical）を設定可能。GitHub Actions、GitLab CI、Jenkins との統合を標準サポート。

### 使い方

1. **初期セットアップとデータベース初期化**  
   `deployguard init --db postgresql://user:pass@localhost:5432/deployguard --cve-db-path ./cve_database.json` を実行。これにより、PostgreSQL にスキーマが作成され、CVE データベースがローカルにキャッシュされる。初回は約 3 秒で完了。

2. **プロジェクトスキャンと結果出力**  
   `deployguard scan --path ./src --format sarif --output ./report.sarif --secret-rules ./custom_rules.yaml` を実行。`--path` でスキャン対象ディレクトリを指定し、`--format` で出力形式を選択。結果は SARIF ファイルとして保存され、VS Code の SARIF Viewer で可視化可能。

3. **CI/CD パイプラインへの統合**  
   GitHub Actions のワークフローに以下のステップを追加：  
   ```yaml
   - name: Run DeployGuard
     run: deployguard scan --path . --fail-on critical --threshold high
   ```  
   これにより、プルリクエスト作成時に自動スキャンが実行され、critical または high の脆弱性が検出された場合、マージがブロックされる。`--threshold` のデフォルト値は medium。

### ベストプラクティス

- **スキャンルールの定期更新**  
  脆弱性データベースとカスタムルールは毎週更新することを推奨。`deployguard update --cve-db` コマンドで最新の CVE リストを自動取得可能。特に、npm や pip の依存関係が多いプロジェクトでは、`--auto-update` フラグを有効にすることで、スキャン前に自動更新が走る。

- **段階的な閾値設定**  
  初回導入時は `--threshold low` で全脆弱性を可視化し、その後チームの許容範囲に応じて `--threshold medium` または `--threshold high` に引き上げる。これにより、ノイズを減らしつつ重大なリスクのみをブロックできる。`--fail-on` は常に `critical` に設定し、致命的な脆弱性を絶対にデプロイさせない。
