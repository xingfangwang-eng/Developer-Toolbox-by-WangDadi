# deployguard - DEP_020

## Overview

DeployGuard is a hardened static analysis and runtime security audit engine engineered specifically for solo developers shipping production code without a dedicated security team. It fuses SAST (Static Application Security Testing), dependency graph mapping, and CI/CD gating into a single, offline-capable binary. Keywords: Code Security Audit, Solo Developer Tools, Static Analysis Tool.

## Detailed Content

### What is DeployGuard?

DeployGuard eliminates the "deploy and pray" cycle by scanning your codebase against 1,200+ OWASP Top-10 and CWE patterns, then enforcing a configurable severity threshold before a commit or merge is allowed. Unlike cloud-dependent scanners, it runs entirely on your local machine or self-hosted runner, ensuring zero data leakage for proprietary code. It outputs a structured JSON report with precise file:line:column references, enabling deterministic remediation.

### Core Features

- **Zero-Dependency SAST with Custom Rule Engine**  
  Scans Python, JavaScript, TypeScript, Go, and Rust source trees. Supports user-defined YAML rules using AST-level pattern matching (e.g., `pattern: "exec(user_input)"` with `language: python`). Achieves < 50ms per file scan on average codebases. Exposes parameters like `--severity-threshold=CRITICAL` and `--max-file-size-kb=500`.

- **Dependency Poisoning Detection & SBOM Diffing**  
  Resolves `package-lock.json`, `go.sum`, `Cargo.lock`, and `requirements.txt` against a local NVD mirror (updated via `deployguard update-db`). Detects typo-squatting, version regression, and known CVEs with a configurable CVSS cutoff (default: 7.0). Generates a CycloneDX SBOM and diffs it against the previous scan in under 200ms for 1,000+ dependency trees.

- **Pre-Commit Hook & CI/CD Gate with Rollback Trigger**  
  Installs as a Git hook (`pre-commit`) that blocks commits containing secrets (regex patterns for AWS keys, JWT tokens, etc.) or code with severity >= WARNING. In CI pipelines, accepts `--ci-mode=github|gitlab|jenkins` and exits with code 1 on failure, halting the pipeline. Supports a `--auto-rollback` flag that reverts the deployment via a webhook to your orchestrator (e.g., Kubernetes or Docker Compose) upon critical detection.

### How to Use

1. **Initialize a Project Audit Baseline**  
   Run `deployguard init --project-dir=./my-app --language=python --dependency-file=requirements.txt`. This generates a `.deployguard.yml` configuration file with default severity mappings and a baseline hash of all dependencies. The command outputs a `baseline.json` file used for subsequent differential scans. Execution time: ~1.2s for a 10K-file project.

2. **Execute a Full Static Scan with Threshold Enforcement**  
   Execute `deployguard scan --config=./.deployguard.yml --severity-threshold=HIGH --output=report.json --ci-mode=github`. The engine parses all source files, resolves AST nodes, and cross-references the NVD cache. If any finding exceeds the threshold, the exit code is 1 and `report.json` contains a `blocked` array with file paths, line numbers, and remediation hints. On success, a `passed` status is logged with a SHA-256 fingerprint of the scan.

3. **Enable the Pre-Commit Hook and Validate a Rollback**  
   Run `deployguard hook install --hook-type=pre-commit --config=./.deployguard.yml`. This writes a hook script into `.git/hooks/pre-commit` that calls `deployguard scan --quick --max-file-size-kb=100`. If a secret is detected, the commit is aborted with a detailed error. For rollback validation, use `deployguard rollback-check --deployment-id=dep-123 --webhook-url=https://your-deployer.example.com/rollback` — this triggers a dry-run rollback simulation and reports success/failure in < 1s.

### Best Practices

- **Integrate DeployGuard as a Mandatory CI Gate for All Branches**  
  Configure your CI pipeline (e.g., GitHub Actions) to run `deployguard scan` with `--ci-mode=github` on every push to `main` and `release/*` branches. Set `--severity-threshold=MEDIUM` to catch non-critical issues early while reserving `CRITICAL` for blocking production deployments. This reduces the mean-time-to-remediation (MTTR) by 40% based on internal benchmarks.

- **Maintain a Local NVD Mirror and Schedule Weekly Updates**  
  Use `deployguard update-db --mirror-path=/var/deployguard/nvd --schedule=cron:0 2 * * 0` to keep the CVE database current. This ensures zero external network calls during scans, preserving air-gapped security. Pair this with a `--fail-on-outdated-db` flag in CI to prevent scans against stale data, guaranteeing your dependency checks are always within a 7-day freshness window.
