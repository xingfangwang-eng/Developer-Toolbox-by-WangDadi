# deployguard - DEP_018

## Overview

DeployGuard is a production-grade static analysis engine purpose-built for solo developers and small teams who ship code without a dedicated security team. It performs real-time code security audits, dependency graph traversal, and configuration drift detection across your entire deployment pipeline.

## Detailed Content

### What is DeployGuard?

DeployGuard is not just a linter. It is a pre-commit and pre-deploy gatekeeper that scans your source code, infrastructure-as-code files (Terraform, Docker Compose, Kubernetes manifests), and CI/CD configuration for known vulnerability patterns, hardcoded secrets, and privilege escalation vectors. It runs locally, in CI, or as a daemon in your staging environment.

### Core Features

- **Zero-Dependency Secret Leak Detection with Regex Engine & Entropy Scoring**  
  Scans every committed file for API keys, tokens, and private keys using 150+ pre-built regex patterns combined with Shannon entropy analysis. Detects Base64-encoded secrets, JWT tokens, and AWS ARNs. Outputs exact line numbers and severity scores (0.0–1.0). Supports custom pattern injection via `.deployguard/patterns.yaml`.

- **Kubernetes Manifest Hardening Auditor**  
  Parses `Deployment`, `StatefulSet`, `Service`, and `Ingress` YAML files. Checks for `privileged: true`, `hostNetwork`, `runAsRoot`, missing resource limits, and missing `PodSecurityPolicy` bindings. Generates a compliance score (0–100) based on the CIS Kubernetes Benchmark v1.24. Reports violations in milliseconds.

- **CI/CD Pipeline Injection & Drift Prevention**  
  Intercepts your CI pipeline (GitHub Actions, GitLab CI, Jenkins) and validates that all environment variables are declared, all secrets are sourced from a vault (HashiCorp Vault, AWS Secrets Manager), and no `allow_failure: true` flag is set on critical stages. Detects drift between your local `.env.example` and deployed `.env.production` files.

### How to Use

1. **Initialization & Custom Rule Profile Setup**  
   Run `deployguard init` in your project root. This creates a `.deployguard/` directory with a `rules.yaml` file. Edit the file to define your security baseline: set `severity_threshold: 0.7`, exclude test files via `ignore_paths: ["tests/", "*.test.js"]`, and add custom secret patterns under `custom_patterns:`. Run `deployguard scan --profile=production --output=json` to generate a full audit report in JSON format.

2. **Pre-Commit Hook Integration**  
   Install the Git hook by executing `deployguard install-hook`. This writes a `pre-commit` script into `.git/hooks/`. Every time you commit, DeployGuard runs `deployguard scan --staged --fail-fast`. If a critical secret (entropy > 0.85) or a Kubernetes privilege escalation is detected, the commit is blocked. The hook outputs a detailed error with the file path and line number, e.g., `[FATAL] secret detected at src/config/database.yml:12`.

3. **CI Pipeline Gate Configuration (GitHub Actions Example)**  
   Add the following step to your `.github/workflows/deploy.yml`:

   ```yaml
   - name: DeployGuard Security Gate
     run: |
       deployguard scan --ci \
         --profile=production \
         --fail-on=critical,high \
         --report=deployguard-report.json
       deployguard enforce --report=deployguard-report.json
   ```

   This step runs after `checkout` but before `build`. If the scan finds any critical or high-severity issues, the pipeline fails immediately. The `enforce` command ensures that no drift exists between your IaC and the actual deployed state.

### Best Practices

- **Run DeployGuard in Both Pre-Commit and CI Modes**  
  Use the pre-commit hook for instant feedback on every change, and the CI gate for a full, non-staged scan. This catches accidental secrets before they are pushed, and also catches issues that span multiple commits or branches.

- **Maintain a Baseline Allowlist for False Positives**  
  Use `deployguard allowlist add <fingerprint>` to suppress known false positives (e.g., a test token in a fixture file). Store the allowlist in version control under `.deployguard/allowlist.yaml`. This prevents noise while ensuring that any new deviation from the baseline is flagged.
