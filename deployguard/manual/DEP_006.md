# deployguard - DEP_006

## Overview

DeployGuard is a **pre-commit static analysis engine** purpose-built for solo developers and small teams who ship code without a dedicated security reviewer. It enforces **code security audit** rules at the file-system level before any commit reaches the remote, catching secrets, dangerous patterns, and misconfigurations in under **500ms per file** (I/O-bound) and **1200ms per scan** (analysis-bound) on a standard M1 MacBook. The tool integrates natively with **Git hooks**, **GitHub Actions**, and **local CI runners** to provide continuous protection without adding friction to the development loop.

## Core Features

### 1. Real-Time Secret & Credential Leak Detection (with Pattern-Aware Contextual Analysis)

DeployGuard scans every staged file against a **compiled rule set of 240+ regex patterns** covering API keys, database connection strings, private keys, and OAuth tokens. It goes beyond simple regex matching by employing a **two-pass context engine**: the first pass flags probable secrets (e.g., `AWS_SECRET_ACCESS_KEY = "..."`), the second pass checks if the line is inside a test fixture, documentation, or a `.env.example` file. Only secrets in **production code** (source files, configs, infrastructure-as-code) trigger a hard block. False-positive rate: **< 1.5%** on a corpus of 1,000+ open-source repositories.

**Technical parameters:**
- `--threshold 0.85` (confidence score, 0.0–1.0)
- `--block-on-critical` (fails the commit if a secret is detected in non-exempt paths)
- `--exempt-path ./tests/` (allows known test credentials)
- Outputs a JSON report with line number, matched pattern ID, and risk level (low/medium/critical).

### 2. AST-Driven Dangerous Pattern & Anti-Pattern Detection

Instead of naive string matching, DeployGuard parses the abstract syntax tree (AST) of supported languages (Python, JavaScript/TypeScript, Go, Rust, Shell) and flags **known dangerous constructs**—like `eval()` with user input, SQL injection via string concatenation, `exec()` calls in production code, or hardcoded `root` passwords. The engine uses a **rule weight system**: each pattern is assigned a severity score (1–100), and a commit is blocked if the cumulative score exceeds the configurable `--max-risk 50`.

**Technical parameters:**
- Supported languages: `python`, `javascript`, `typescript`, `go`, `rust`, `bash`
- `--max-risk 50` (default; block if total risk score exceeds 50)
- `--rule-file .deployguard/rules.yaml` (custom rules with severity, description, and remediation hint)
- Scan time per file: **≤ 200ms** for files < 5,000 lines

### 3. Pre-Commit Dependency & Configuration Drift Check

DeployGuard compares the **staged dependency files** (`package.json`, `requirements.txt`, `go.mod`, `Cargo.toml`) against a local **known-vulnerability database** (updated daily from NVD, GitHub Advisory DB, and OSV). If a dependency with a known CVE (CVSS ≥ 7.0) is being added or upgraded, the commit is blocked with a detailed report. Additionally, it checks for **configuration drift** in Dockerfiles, Terraform plans, and `.env` templates—flagging any new `ALLOWED_HOSTS: ['*']`, `DEBUG=True`, or `--privileged` flags in Docker.

**Technical parameters:**
- `--vuln-db-update-interval 24h` (auto-updates local SQLite database)
- `--cvss-threshold 7.0` (only block on high/critical CVEs)
- `--config-drift` (enables config drift detection; default: on)
- Output: `deployguard-report.json` with CVE IDs, affected versions, and patch suggestions.

## How to Use

### Step 1: Initialize DeployGuard in Your Repository

Run the one-time initialization command from the root of your project. This creates a `.deployguard/` directory with a default configuration file, installs the Git pre-commit hook, and downloads the initial vulnerability database.

```bash
deployguard init --language python --hook pre-commit --db-path ~/.deployguard/cve.db
```

The `--language` flag sets the primary language for AST scanning. The `--hook pre-commit` flag installs a script in `.git/hooks/pre-commit` that runs DeployGuard on every `git commit`. The `--db-path` specifies where the 50 MB SQLite vulnerability database is stored (shared across all projects).

### Step 2: Configure Custom Rules & Exemptions

Edit `.deployguard/rules.yaml` to add project-specific patterns. For example, to block any commit that introduces a hardcoded `postgres://` connection string outside of `config/`:

```yaml
- pattern: "postgres://[^:]+:[^@]+@"
  severity: 90
  description: "Hardcoded PostgreSQL connection string detected"
  remediation: "Use environment variable POSTGRES_URL instead"
  exempt_paths:
    - "config/examples/"
```

Then run a dry-run to validate your rules without blocking a commit:

```bash
deployguard scan --dry-run --format json > dry-run-report.json
```

### Step 3: Run the Full Scan & Enforce Blocking

When you're ready to enforce, simply run `git commit`. The pre-commit hook will invoke DeployGuard with the following command (equivalent to what the hook runs):

```bash
deployguard scan --staged --config .deployguard/rules.yaml --vuln-db ~/.deployguard/cve.db --block-on-critical --max-risk 50 --output report.json
```

If any critical secret, high-risk AST pattern, or vulnerable dependency is detected, the commit is aborted and the report is printed to stderr. You can bypass the check with `git commit --no-verify` (discouraged in production).

## Best Practices

### 1. Integrate DeployGuard as a Gate in CI/CD, Not Just Locally

While the pre-commit hook is excellent for local protection, **always run DeployGuard again in your CI pipeline** (e.g., GitHub Actions, GitLab CI, Jenkins). This catches cases where a developer bypassed the hook, or where the local environment has an outdated vulnerability database. Configure the CI job to fail the build if DeployGuard produces any critical or high-severity findings. This creates a **defense-in-depth** strategy: local prevention + CI enforcement.

### 2. Maintain a Living Rule Set with Project-Specific Exemptions

Out-of-the-box rules are generic and may produce false positives for your domain (e.g., test fixtures with fake credentials, internal documentation with example API keys). **Curate a `.deployguard/rules.yaml` file per repository** and treat it as part of your codebase (store it in version control). Regularly review the `deployguard-report.json` from CI runs to identify patterns that need exemption or refinement. A well-maintained rule set reduces alert fatigue and ensures that real threats are never drowned out by noise.
