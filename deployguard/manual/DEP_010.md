# deployguard - DEP_010

## Overview

DeployGuard is a static analysis and code security audit engine purpose-built for solo developers and small teams who ship fast but cannot afford a dedicated security review pipeline. It integrates directly into the CI/CD loop, scanning source trees for injection vulnerabilities, credential leaks, and dependency poisoning **before** the build artifact is ever staged. Keywords: Code Security Audit, Solo Developer Tools, Static Analysis Tool.

## Core Features

### 1. Real-time Credential Leak Detection with Entropy Scoring
DeployGuard parses every commit’s diff in under **50 milliseconds** (per 1,000 lines) and applies a sliding-window Shannon entropy algorithm to detect high-entropy strings that match known credential patterns (AWS keys, GitHub tokens, database connection strings). It flags any string with an entropy score > 3.8 and a pattern match against the `credential-regex` dictionary (default: 450+ patterns). The tool supports a `--threshold` parameter (float, default 3.8) to tune sensitivity per project.

### 2. Multi-Engine Dependency Poisoning Check (MEPC)
Instead of relying on a single advisory database, DeployGuard queries **three** independent sources in parallel: OSV.dev, GitHub Advisory Database (via GraphQL API), and a local SBoM (Software Bill of Materials) cache stored in PostgreSQL. Each query has a configurable timeout (`--dep-timeout` in milliseconds, default 5000). If any engine reports a CVE with a CVSS score ≥ 7.0, the pipeline is immediately halted and a JSON report is written to `./deployguard/report.json`. The tool also compares the resolved dependency tree against a baseline snapshot (`baseline.lock`) to catch unexpected version drift.

### 3. Custom Rule Engine for Static Code Injection (SCIE)
DeployGuard ships with a **tiered rule engine** that allows users to define custom AST-level patterns in a YAML file (`deployguard-rules.yaml`). Each rule specifies: `severity` (critical, high, medium), `language` (e.g., Python, JavaScript, Go), `pattern` (using tree-sitter query syntax), and `action` (block, warn, log). The engine processes files at a rate of **2,000 AST nodes per second** (measured on a 4-core CPU). Example: a rule can block any `eval()` call in JavaScript if the argument is a user-controlled variable (detected via taint propagation).

## How to Use

### Step 1: Initialize the Baseline Snapshot
Run the following command in your repository root to generate a dependency baseline and credential entropy baseline. This creates `deployguard/baseline.lock` and `deployguard/entropy-baseline.json`.
```bash
deployguard init --project-name my-app --db-uri postgresql://user:pass@localhost:5432/deployguard
```
The `--db-uri` parameter connects to a PostgreSQL instance where DeployGuard stores historical scan data and CVSS mappings. The init command takes approximately **2.3 seconds** for a repository with 1,200 dependencies.

### Step 2: Execute a Pre-Deployment Scan
Before every deployment (e.g., as a GitHub Action or pre-commit hook), run:
```bash
deployguard scan --threshold 3.8 --dep-timeout 5000 --output-format json --fail-on critical
```
This command performs the full scan: credential leak detection, dependency poisoning check (MEPC), and static code injection analysis (SCIE). If any critical-severity rule is triggered, the exit code is `1`. The JSON report is written to `./deployguard/report.json` with timestamps in **ISO 8601 (millisecond precision)**.

### Step 3: Review and Promote the Artifact
If the scan passes, promote the build artifact by running:
```bash
deployguard promote --commit-hash $(git rev-parse HEAD) --status passed
```
This updates the PostgreSQL database with the commit hash, scan timestamp, and a `passed` status. The tool also writes a signed attestation (`deployguard/attestation.sig`) using Ed25519. The attestation can be verified later with `deployguard verify --attestation deployguard/attestation.sig`.

## Best Practices

### Best Practice 1: Integrate with a Pre-Commit Hook for Zero-Tolerance Leaks
Configure DeployGuard to run as a pre-commit hook with `--fail-on high`. This prevents any credential leak (even a test key) from reaching the remote repository. Use the `--threshold 3.5` setting for teams that handle sensitive data. In our benchmarks, this reduces credential exposure incidents by **94%** in solo developer workflows.

### Best Practice 2: Maintain a Centralized PostgreSQL Instance for Multi-Project Audits
Instead of running DeployGuard with ephemeral SQLite databases, deploy a shared PostgreSQL instance (e.g., on a $5/month VPS) and point all your repositories to it via the `--db-uri` parameter. This enables cross-project vulnerability correlation—e.g., if one project’s dependency has a CVE, you can instantly identify all other projects sharing that dependency. Set a `pg_cron` job to clean up scans older than 90 days to keep the database lean.
