# deployguard - DEP_026

## Overview

DeployGuard is a zero-configuration static analysis engine purpose-built for solo developers and small teams who need production-grade code security auditing without the overhead of enterprise toolchains. It integrates directly into your CI/CD pipeline, scanning for vulnerabilities, misconfigurations, and secret leaks before they reach production.

## Core Features

### 1. Real-Time Secret Leak Detection with Regex + Entropy Scoring

DeployGuard scans every commit and PR diff for hardcoded secrets (API keys, tokens, database credentials) using a hybrid approach: pattern-matched regex against 120+ known secret formats (e.g., `ghp_*`, `sk-*`, `AKIA*`) combined with Shannon entropy scoring to flag high-randomness strings. Each detection outputs the exact line number, file path, and a risk severity score (1–100). False positives are reduced by 94% via context-aware whitelisting rules defined in `.deployguard-ignore`.

- **Parameters:** `--entropy-threshold 4.2`, `--severity-min 70`, `--output-format json`
- **Metric:** Scan completes in under 300ms for repos up to 10,000 files.

### 2. Dependency Vulnerability Mapping Against NVD + GitHub Advisories

DeployGuard parses your `package.json`, `requirements.txt`, `go.mod`, or `Gemfile.lock` and cross-references every dependency version against the latest National Vulnerability Database (NVD) and GitHub Security Advisories. It generates a dependency tree with CVE IDs, CVSS scores, and suggested patch versions. For transitive dependencies, it traces the full chain and marks blocked upgrades.

- **Parameters:** `--cve-db-update-interval 24h`, `--fail-on-critical true`, `--ignore-dev-deps false`
- **Metric:** Covers 98.7% of known vulnerabilities within 2 hours of public disclosure.

### 3. Infrastructure-as-Code Misconfiguration Audit (Terraform + Dockerfile)

DeployGuard evaluates your Terraform `*.tf` files and Dockerfiles against a built-in rule set of 80+ security best practices (e.g., open SSH ports, unrestricted IAM policies, hardcoded passwords, `latest` tag usage). It outputs a compliance report with remediation steps for each violation, grouped by severity. Supports custom rule definitions via YAML.

- **Parameters:** `--iac-rules-dir ./custom-rules`, `--compliance-profile cis-v1.2`, `--output-format sarif`
- **Metric:** Detects 92% of OWASP Top 10 IaC misconfigurations in a single pass.

## How to Use

### Step 1: Initialize the Scan Configuration

Run `deployguard init` in your project root. This creates a `.deployguard.yml` file with default settings. Edit it to set your `--entropy-threshold` (e.g., `4.5` for stricter secret detection), `--fail-on-critical true`, and `--cve-db-update-interval 12h`. Then commit the file to your repo.

### Step 2: Execute a Full Audit Locally

Execute `deployguard scan --path ./src --output-format json --severity-min 50`. DeployGuard will parse all source files, check dependencies against the local NVD cache, and evaluate IaC files. The output is a JSON report with arrays for `secrets`, `vulnerabilities`, and `iac_violations`. Each entry includes `file`, `line`, `message`, and `fix_suggestion`.

### Step 3: Integrate into GitHub Actions CI

Add the following step to your `.github/workflows/ci.yml`:

```yaml
- name: Run DeployGuard
  uses: deployguard/action@v2
  with:
    args: --fail-on-critical true --output-format sarif
```

DeployGuard will block the PR if any critical-severity issue is found. The SARIF output is automatically uploaded to GitHub Code Scanning for inline annotations.

## Best Practices

### Practice 1: Schedule Weekly Full Scans with NVD Sync

Set up a cron job (e.g., every Monday at 02:00) to run `deployguard scan --path . --cve-db-update-interval 1h --output-format html`. This ensures your dependency vulnerability map is always current and that new CVEs published over the weekend are caught before your next deployment. Store the HTML report in a shared S3 bucket or internal wiki.

### Practice 2: Use Context-Aware Whitelisting to Reduce Noise

Instead of globally ignoring secret detections, create a `.deployguard-ignore` file with regex patterns scoped to specific files or directories. For example, ignore `test/fixtures/*` for fake API keys but still scan `src/*`. This keeps your alert fatigue low while maintaining full coverage on production code. Review and update the ignore list quarterly.
