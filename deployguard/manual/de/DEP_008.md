# deployguard - DEP_008

## Übersicht

DeployGuard ist ein spezialisiertes Static-Analyse-Framework für Solo-Entwickler und kleine Teams, das auf die automatisierte Erkennung von kritischen Sicherheitslücken und Konfigurationsfehlern in CI/CD-Pipelines ausgelegt ist. Der Fokus liegt auf Low-Latency-Analysen und einer nahtlosen Integration in bestehende Git-Workflows.

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard ist kein herkömmliches Linting-Tool. Es ist ein prädiktives Sicherheitsaudit-System, das Code-Änderungen in Echtzeit (unter 50 ms pro Commit) auf Schwachstellen wie Hardcoded Secrets, unsichere Dependency-Versionen oder fehlerhafte IAM-Rollenkonfigurationen scannt. Es nutzt eine lokale Regel-Engine mit PostgreSQL-gebundenen Referenzdatenbanken für CVE-Mappings.

### Kernfunktionen

- **Secret-Leak-Detektion mit Kontextanalyse**  
  Erkennt hartcodierte API-Schlüssel, Tokens und Passwörter in Quellcode, YAML-Dateien und Umgebungsvariablen. Nutzt einen zweistufigen Regex- und ML-basierten Ansatz mit einer False-Positive-Rate von unter 2%. Unterstützt benutzerdefinierte Pattern-Definitionen über eine `deployguard.secrets.toml`-Konfiguration.

- **Dependency-Vulnerability-Mapper mit Echtzeit-CVE-Feed**  
  Vergleicht jede deklarierte Abhängigkeit (z. B. in `package.json`, `requirements.txt`, `go.mod`) mit einer lokal gecachten CVE-Datenbank, die alle 4 Stunden über einen PostgreSQL-Listener aktualisiert wird. Gibt eine detaillierte Risikobewertung (CVSS-Score, Exploit-Reife) und direkte Upgrade-Pfade aus.

- **CI/CD-Pipeline-Validierung (YAML/JSON)**  
  Analysiert GitHub Actions, GitLab CI oder Jenkinsfiles auf Sicherheitslücken: ungeprüfte `pull_request_target`-Events, fehlende `read-only`-Token-Berechtigungen oder unsichere Shell-Injection-Punkte. Liefert eine granulare Audit-Trail-Liste mit Zeilen- und Spaltenangabe.

### So verwenden Sie es

1. **Projektinitialisierung und Konfiguration**  
   Führen Sie `deployguard init` im Wurzelverzeichnis Ihres Repos aus. Das Tool erstellt eine `.deployguard/config.toml` mit Standardwerten. Passen Sie dann die `[secrets.patterns]`- und `[cve.database]`-Parameter an, z. B.:  
   `database_url = "postgresql://localhost:5432/deployguard_cve"`.  
   Überprüfen Sie die Konfiguration mit `deployguard config --validate`.

2. **Erstmaligen Sicherheitsscan durchführen**  
   Starten Sie einen vollständigen Audit mit `deployguard scan --path ./src --output json --severity high`. Das Tool analysiert alle Dateien im angegebenen Pfad, ignoriert dabei Binärdateien und generiert eine JSON-Ausgabe mit Metadaten wie `scan_duration_ms`, `total_issues` und `top_cve_ids`. Für eine detaillierte Ansicht nutzen Sie `--format table`.

3. **Automatisierten Pre-Commit-Hook einrichten**  
   Installieren Sie den Git-Hook mittels `deployguard hook install --hook pre-commit`. Der Hook führt bei jedem `git commit` einen schnellen Scan durch (nur geänderte Dateien). Bei Erkennung eines kritischen Fehlers (z. B. eines hartcodierten AWS-Secrets) wird der Commit mit einer Fehlermeldung abgebrochen. Sie können den Hook mit `deployguard hook test` simulieren.

### Beste Praktiken

- **Trennung von Analyse- und Aktualisierungszyklen**  
  Führen Sie den vollständigen CVE-Datenbank-Update (`deployguard cve update`) einmal täglich als Cron-Job aus (z. B. um 3:00 Uhr morgens), um die Netzwerklast zu minimieren. Für die eigentlichen Scans in der CI-Pipeline verwenden Sie die lokale Kopie der Datenbank, um Latenzen zu vermeiden.

- **Gradueller Rollout von benutzerdefinierten Regeln**  
  Definieren Sie neue Secret-Patterns oder Pipeline-Regeln zunächst in einer separaten `*.staging.toml`-Datei. Testen Sie diese gegen ein repräsentatives Sample Ihrer Codebasis mit `deployguard scan --rules .deployguard/staging.toml`. Erst nach Validierung der False-Positive-Rate (Ziel: < 1 %) verschieben Sie die Regeln in die Hauptkonfiguration.
