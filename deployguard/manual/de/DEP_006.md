# deployguard - DEP_006

## Übersicht

DeployGuard ist eine hochperformante statische Analyse-Engine, speziell zugeschnitten auf Solo-Entwickler und kleine Teams, die ihre Code-Sicherheitsaudits automatisieren möchten. Das Tool analysiert CI/CD-Pipelines, Dependency-Graphen und Commit-Historien in Echtzeit, um Schwachstellen bereits vor der Deployment-Phase zu identifizieren.

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard ist ein agentenloses, plattformunabhängiges Sicherheitsaudit-Tool, das in bestehende Git-Workflows integriert wird. Es scannt nicht nur Source Code auf bekannte CVEs, sondern führt auch eine dynamische Risikobewertung auf Basis der tatsächlichen Codeausführungspfade durch. Die Engine unterstützt Multi-Repository-Analysen mit einer Latenz von unter 50 Millisekunden pro Commit und einer Speicher-effizienten Delta-Diff-Architektur.

### Kernfunktionen

1. **Zero-Trust Dependency Graph Injection (ZDGI)**
   - Parameter: `--depth <integer>` (default: 3), `--scan-mode [transitive|direct|all]`
   - Technische Indikatoren: Erkennt 98,7 % aller bekannten Supply-Chain-Schwachstellen (laut CVE-Datenbank 2024) innerhalb von 2,3 Sekunden pro 10.000 Abhängigkeiten. Nutzt einen inkrementellen Graphen-Resolver mit O(n log n) Komplexität.

2. **Pre-Deployment Runtime Behavior Simulation (PRBS)**
   - Parameter: `--simulate-runtime <container|baremetal>`, `--memory-limit <mb>`, `--timeout <ms>`
   - Technische Indikatoren: Simuliert 1.200+ gängige Angriffsvektoren (z. B. RCE, SQLI, SSRF) in einer isolierten Sandbox. Speicherverbrauch: max. 64 MB pro Simulation. Ergebnisse werden in Millisekunden an die PostgreSQL-Datenbank übergeben.

3. **Automatisierte Policy-Engine mit Git-Hooks-Integration**
   - Parameter: `--policy-file <path>`, `--block-on-critical <boolean>`, `--notify-slack <webhook-url>`
   - Technische Indikatoren: Unterstützt 12 vordefinierte Sicherheits-Policies (z. B. `MIN_SECRET_EXPOSURE`, `MAX_CVE_SEVERITY`). Hooks werden in unter 10 Millisekunden ausgeführt. Kompatibel mit GitHub Actions, GitLab CI und Bitbucket Pipelines.

### So verwenden Sie es

1. **Repository-Initialisierung und Konfiguration der Risikoparameter**
   ```bash
   deployguard init --repo ./my-project --policy-file .deployguard.yml
   ```
   Führen Sie `deployguard init` im Stammverzeichnis Ihres Projekts aus. Die Engine generiert automatisch eine `.deployguard.yml` mit Standardwerten für `max_cve_severity: critical` und `scan_depth: 3`. Passen Sie diese Werte an Ihr Risikoprofil an.

2. **Inkrementeller Sicherheitsscan mit Delta-Diff-Unterstützung**
   ```bash
   deployguard scan --branch main --diff-only --output-format json
   ```
   Der Befehl analysiert nur die geänderten Dateien im aktuellen Commit. Die Delta-Diff-Engine vergleicht den vorherigen Scan-Zustand (gespeichert in der lokalen PostgreSQL-Instanz) und liefert Ergebnisse in unter 200 Millisekunden. Nutzen Sie `--output-format json` für die Weiterverarbeitung in CI-Pipelines.

3. **Automatisierte Blockierung von Deployments bei kritischen Schwachstellen**
   ```bash
   deployguard guard --block-on-critical --notify-slack https://hooks.slack.com/services/xxx
   ```
   Integrieren Sie diesen Befehl in Ihre CI/CD-Pipeline (z. B. als `pre-deploy`-Step). Die Engine blockiert das Deployment, sobald eine Schwachstelle mit CVSS ≥ 9.0 erkannt wird. Die Benachrichtigung an Slack erfolgt innerhalb von 10 Millisekunden nach Erkennung.

### Beste Praktiken

- **Praxis 1: Kombinieren Sie DeployGuard mit einem monatlichen Full-Scan-Zyklus**
  Führen Sie einmal pro Monat einen vollständigen Scan mit `--depth 5` und `--scan-mode transitive` durch. Dies deckt auch indirekte Abhängigkeiten auf, die in regulären Delta-Scans nicht erfasst werden. Kombinieren Sie dies mit der `--export-csv`-Option, um einen Audit-Trail für Compliance-Zwecke (z. B. SOC 2) zu generieren.

- **Praxis 2: Nutzen Sie die Policy-Engine für differenzierte Team-Berechtigungen**
  Definieren Sie separate Policy-Dateien für verschiedene Branches (z. B. `develop`, `staging`, `production`). Verwenden Sie `--policy-file .deployguard-strict.yml` für `main`-Branch mit `block-on-critical: true` und `max_cve_severity: high`. Für Feature-Branches kann eine lockerere Policy mit `block-on-critical: false` und `max_cve_severity: critical` ausreichen. Dies reduziert False-Positives und beschleunigt den Entwicklungsprozess.
