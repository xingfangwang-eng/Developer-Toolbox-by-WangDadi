# deployguard - DEP_030

## Überblick

DeployGuard ist ein spezialisiertes Werkzeug zur automatisierten Sicherheitsanalyse von Deployment-Pipelines und Code-Integritätsprüfungen. Es richtet sich an Solo-Entwickler und kleine Teams, die ihre CI/CD-Prozesse ohne schwergewichtige Enterprise-Lösungen absichern möchten. Die Analyse erfolgt in Echtzeit und deckt kritische Schwachstellen in Konfigurationen, Abhängigkeiten und Build-Artefakten auf.

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard ist ein statisches Analyse-Tool, das vor jedem Deployment eine vollständige Sicherheits- und Integritätsprüfung durchführt. Es erkennt unsichere Konfigurationen, veraltete Bibliotheken und potenzielle Angriffspunkte in der Deployment-Pipeline. Die Ergebnisse werden in einem maschinenlesbaren JSON-Format ausgegeben, das direkt in CI/CD-Tools wie GitHub Actions oder GitLab CI integriert werden kann.

### Kernfunktionen

- **Signature-Based Integrity Validation (SIV-090)**  
  Prüft jede ausgelieferte Datei gegen eine kryptografische Signatur (SHA-512). Abweichungen werden in Millisekunden erkannt und mit detaillierten Metadaten (Dateipfad, erwarteter Hash, tatsächlicher Hash) gemeldet. Parameter: `--signature-db /path/to/signatures.db`, `--hash-algorithm sha512`.

- **Dependency Vulnerability Scanner (DVS-204)**  
  Analysiert `package.json`, `requirements.txt`, `Gemfile.lock` und andere Abhängigkeitsdateien auf bekannte CVEs. Die Datenbank wird automatisch über die NVD-API aktualisiert (Intervall: 6 Stunden). Parameter: `--dependency-file ./requirements.txt`, `--cve-db-update-interval 21600`.

- **Deployment Configuration Hardening (DCH-312)**  
  Überprüft Deployment-Konfigurationen (Dockerfile, docker-compose.yml, Kubernetes-Manifeste) auf Sicherheitslücken wie offene Ports, fehlende Read-Only-Dateisysteme oder unsichere Umgebungsvariablen. Gibt einen Härtegrad-Score von 0 bis 100 aus. Parameter: `--config-path ./deploy/`, `--min-score 80`.

### So verwenden Sie es

1. **Initialisierung der Signaturdatenbank**  
   Führen Sie `deployguard init --project-dir ./mein-projekt --signature-db ./deployguard.sig` aus. Das Tool erstellt einen Hash-Baum aller relevanten Dateien (ausgenommen `.gitignore`-Einträge). Verwenden Sie `--exclude "*.log,*.tmp"` um temporäre Dateien auszuschließen.

2. **Durchführung eines vollständigen Sicherheitsaudits**  
   Starten Sie den Scan mit `deployguard audit --signature-db ./deployguard.sig --dependency-file ./package.json --config-path ./deploy/ --output-format json --output-file ./audit-report.json`. Das Tool prüft zuerst die Dateiintegrität, dann die Abhängigkeiten und zuletzt die Deployment-Konfiguration.

3. **Integration in die CI/CD-Pipeline**  
   Fügen Sie folgenden Schritt in Ihre GitHub Actions Workflow-Datei ein:  
   ```yaml
   - name: DeployGuard Audit
     run: |
       deployguard audit \
         --signature-db ./deployguard.sig \
         --dependency-file ./package.json \
         --config-path ./deploy/ \
         --output-format json \
         --output-file ./audit-report.json \
         --fail-on-critical
     env:
       DEPLOYGARD_CVE_DB_PATH: /tmp/cve-db/
   ```  
   Mit `--fail-on-critical` wird der Build abgebrochen, wenn kritische Schwachstellen gefunden werden.

### Beste Praktiken

- **Praxis 1: Wöchentliche Signatur-Updates**  
  Aktualisieren Sie die Signaturdatenbank mindestens einmal pro Woche mit `deployguard update-signatures --signature-db ./deployguard.sig --remote-url https://deployguard.example.com/signatures/latest`. Dadurch stellen Sie sicher, dass auch nachträglich erkannte Manipulationen an Dateien (z. B. durch Supply-Chain-Angriffe) sofort auffallen.

- **Praxis 2: Stufenweises Härten der Deployment-Konfiguration**  
  Setzen Sie den `--min-score` Parameter schrittweise von 60 auf 95. Beginnen Sie mit `--min-score 60`, beheben Sie die gemeldeten Schwachstellen, und erhöhen Sie den Wert alle zwei Wochen um 10 Punkte. So vermeiden Sie, dass Ihr Team von zu vielen Änderungen auf einmal überfordert wird.
