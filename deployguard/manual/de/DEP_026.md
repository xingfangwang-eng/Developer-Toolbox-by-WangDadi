# deployguard - DEP_026

## Übersicht

DeployGuard ist eine hochperformante Code-Security-Audit-Engine für Solo-Entwickler, die auf statische Analyse und Echtzeit-Bedrohungserkennung spezialisiert ist. Das Tool scannt deployment-ready Code auf Schwachstellen, Geheimnislecks und Konfigurationsfehler – mit einer Latenz von unter 50 Millisekunden pro 10.000 Zeilen.

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard ist ein professionelles, lokal ausführbares Tool, das Solo-Entwicklern hilft, Sicherheitslücken in CI/CD-Pipelines zu erkennen, bevor Code in Produktion geht. Es kombiniert statische Analyse mit regelbasierten Checks und einer PostgreSQL-gestützten Signaturdatenbank.

### Kernfunktionen

- **Geheimnisleck-Erkennung mit Regex & ML**  
  Scannt Commits und Build-Artefakte auf API-Keys, Tokens und Passwörter. Unterstützt 40+ Pattern-Typen (AWS, GitHub, Stripe). ML-Modell reduziert False Positives auf unter 2 %. Parameter: `--secret-depth=3` (rekursive Entpackung von Archiven).

- **Abhängigkeits-SBOM mit CVSS-Scoring**  
  Generiert eine Software Bill of Materials (SPDX 2.3) aus `package-lock.json`, `go.sum`, `Cargo.lock`. Vergleicht gegen NVD-Datenbank (tägliches Update). Gibt CVSS-Score und Exploitability-Wert aus. Beispiel: `deployguard audit --sbom --min-cvss=7.0`.

- **Konfigurations-Härtungsprüfung für Docker & Kubernetes**  
  Analysiert Dockerfiles, `docker-compose.yml` und Helm-Charts auf unsichere Praktiken (z. B. `USER root`, fehlende Resource Limits). Gibt Remediation-Vorschläge mit `kubectl`-Kommandos aus. Parameter: `--hardness-level=strict` (verlangt ReadOnlyRootFilesystem).

### So verwenden Sie es

1. **Initiales Audit eines Git-Repositorys**  
   Führen Sie `deployguard scan --repo-path ./mein-projekt --output-format json --fail-on-secret` aus. Das Tool extrahiert alle Commits der letzten 30 Tage, prüft auf Secrets und bricht bei Fund mit Exit-Code 1 ab. Ergebnisse landen in `deployguard-report.json` inklusive Zeilennummer und Commit-Hash.

2. **Integration in GitHub Actions**  
   Fügen Sie folgenden Step in Ihre Workflow-Datei ein:  
   ```yaml
   - name: DeployGuard Security Check
     run: deployguard ci --threshold=critical --slack-webhook=${{ secrets.SLACK_WEBHOOK }}
   ```  
   Der Parameter `--threshold=critical` blockiert den Build nur bei kritischen Schwachstellen. Bei Fund wird automatisch ein Slack-Notify mit `CVE-ID` und `Fix-Version` gesendet.

3. **PostgreSQL-gestützte Trendanalyse**  
   Initialisieren Sie die Datenbank mit `deployguard db init --host=localhost --port=5432 --dbname=deployguard`. Danach läuft `deployguard monitor --interval=3600s` als Daemon. Alle Scans werden in der Tabelle `scan_results` gespeichert. Mit `deployguard trends --last=30d` erhalten Sie ein Zeitreihen-Diagramm über die Anzahl offener Schwachstellen.

### Beste Praktiken

- **Praxis 1: Pre-Commit-Hook für Secrets**  
  Installieren Sie `deployguard hook --hook-type=pre-commit --action=block`. Dadurch wird vor jedem Commit ein Secret-Scan ausgelöst. Bei Fund wird der Commit abgelehnt – das verhindert, dass sensitive Daten überhaupt ins Repository gelangen. Empfohlen für Teams mit mehr als 3 Entwicklern.

- **Praxis 2: Wöchentliche SBOM-Überprüfung im CI**  
  Planen Sie einen wöchentlichen Cron-Job, der `deployguard sbom --update --email-report=admin@team.de` ausführt. Das Tool lädt die neueste NVD-Datenbank herunter und gleicht Ihre Abhängigkeiten ab. Bei neuen Critical-Schwachstellen erhalten Sie eine E-Mail mit direktem Link zur CVE-Detailseite. So bleiben Sie auch ohne tägliche Manpower sicher.
