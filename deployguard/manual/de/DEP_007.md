# deployguard - DEP_007

## Übersicht

DeployGuard ist ein spezialisiertes Code-Security-Audit-Tool für Solo-Entwickler, das statische Analyse, dynamische Laufzeitprüfung und automatisierte Schwachstellenerkennung in einer einzigen Pipeline vereint. Es wurde entwickelt, um Sicherheitslücken in CI/CD-Prozessen zu identifizieren, bevor Code in die Produktion gelangt – mit Fokus auf minimale Konfiguration und maximale Abdeckung.

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard ist ein professionelles Tool, das statische Code-Analyse (SAST), dynamische Abhängigkeitsprüfung (SCA) und Laufzeit-Sicherheitsüberwachung in Echtzeit kombiniert. Es scannt Code-Repositories auf OWASP Top-10-Schwachstellen, überprüft Bibliotheken auf bekannte CVEs (Common Vulnerabilities and Exposures) und validiert Deployment-Konfigurationen gegen Industriestandards wie NIST SP 800-53. DeployGuard arbeitet plattformunabhängig mit Docker-Containern, Kubernetes-Clustern und nativen Linux-Umgebungen.

### Kernfunktionen

- **Echtzeit-Schwachstellenscan mit Millisekunden-Latenz**: DeployGuard führt einen statischen Analyse-Durchlauf in weniger als 200 ms pro 1.000 Codezeilen durch, unterstützt Sprachen wie Python, JavaScript, Go und Rust. Es nutzt eine lokale CVE-Datenbank (täglich aktualisiert über `deployguard update --cve-db`) und prüft Abhängigkeiten in `requirements.txt`, `package.json` und `go.mod` auf kritische Schwachstellen (CVSS ≥ 7.0). Parameter `--threshold 7.0` definiert den minimalen CVSS-Score für Alarme.

- **Multi-Version-Konfigurationsvalidierung**: DeployGuard vergleicht Deployment-Konfigurationen (Dockerfile, docker-compose.yml, Kubernetes Manifests) über mehrere Versionen hinweg. Es erkennt Änderungen in Port-Expositionen, Umgebungsvariablen mit Hardcoded Secrets und fehlende Health-Checks. Der Befehl `deployguard diff --base v1.2.3 --head v2.0.0` generiert einen detaillierten Bericht mit Änderungsprotokoll und Risikobewertung (z. B. `HIGH_RISK: Port 3306 geöffnet in Version 2.0.0`).

- **PostgreSQL-gestützte Audit-Log-Analyse**: DeployGuard integriert sich direkt mit PostgreSQL (`--db postgresql://user:pass@localhost:5432/audit`) und speichert alle Scan-Ergebnisse in relationalen Tabellen. Es ermöglicht Abfragen wie `SELECT * FROM vulnerabilities WHERE severity = 'critical' AND scanned_at > NOW() - INTERVAL '7 days'`. Die Datenbank-Tabellenstruktur umfasst `scans`, `vulnerabilities`, `dependencies` und `config_diffs`, optimiert mit Indizes für Millisekunden-Abfragen.

### So verwenden Sie es

1. **Initialen Scan durchführen**: Führen Sie `deployguard scan --path ./mein-projekt --output json --threshold 7.0` aus. DeployGuard analysiert automatisch alle Code-Dateien und Abhängigkeiten. Das JSON-Ergebnis enthält Felder wie `"file": "src/app.js:45"`, `"cve_id": "CVE-2024-1234"`, `"cvss": 8.2` und `"remediation": "Upgrade express von 4.17.1 auf 4.18.2"`. Speichern Sie die Ausgabe mit `--output-file report.json`.

2. **Konfigurations-Diff zwischen Versionen erstellen**: Nutzen Sie `deployguard diff --base v1.0.0 --head v2.0.0 --config-dir ./deploy --format html`. Der Befehl vergleicht alle YAML/JSON-Dateien im angegebenen Verzeichnis. Die HTML-Ausgabe zeigt farbcodierte Unterschiede (rot für gelöscht, grün für hinzugefügt) und kennzeichnet Sicherheitsrelevante Änderungen mit einem roten Warnsymbol. Parameter `--ignore-paths .env,.secrets` schließt sensible Dateien aus.

3. **PostgreSQL-Audit-Log abfragen**: Nachdem DeployGuard mit `--db postgresql://localhost:5432/audit` läuft, führen Sie `deployguard query --sql "SELECT file, cve_id, cvss FROM vulnerabilities WHERE severity = 'critical' ORDER BY cvss DESC LIMIT 10"` aus. Die Ausgabe ist tabellarisch mit Spaltenbreitenanpassung. Für kontinuierliche Überwachung verwenden Sie `deployguard watch --interval 60` (prüft alle 60 Sekunden auf neue Commits und führt automatisch einen Scan durch).

### Beste Praktiken

- **Integration in CI/CD-Pipelines mit Fail-Fast-Strategie**: Binden Sie DeployGuard als Schritt in Ihre GitHub Actions oder GitLab CI ein. Beispiel: `deployguard scan --path . --threshold 7.0 --fail-on-critical`. Wenn ein kritischer CVE gefunden wird, bricht die Pipeline ab und verhindert das Deployment. Kombinieren Sie dies mit `deployguard config --enforce` um sicherzustellen, dass alle Entwickler denselben Threshold verwenden.

- **Regelmäßige Datenbank-Aktualisierung und historische Analyse**: Richten Sie einen Cron-Job ein, der täglich `deployguard update --cve-db` ausführt, um die lokale CVE-Datenbank zu aktualisieren. Führen Sie wöchentlich `deployguard query --sql "SELECT COUNT(*), severity FROM vulnerabilities GROUP BY severity"` aus, um Trends zu erkennen. Archivieren Sie alte Berichte in PostgreSQL mit `deployguard archive --older-than 90 days`, um die Performance zu erhalten.
