# deployguard - DEP_001

## Übersicht

DeployGuard ist ein auf statische Code-Analyse spezialisiertes Sicherheits-Audit-Tool für Solo-Entwickler, das Schwachstellen in CI/CD-Pipelines und Deployment-Konfigurationen proaktiv identifiziert. Der Fokus liegt auf der Absicherung von Container-Orchestrierung, Secrets-Management und Infrastruktur-als-Code (IaC).

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard ist ein professionelles Tool, das Sicherheitslücken in Deployment-Workflows erkennt, bevor diese in Produktion gehen. Es analysiert Dockerfiles, Kubernetes-Manifeste, Terraform-Pläne und GitHub Actions Workflows auf kritische Fehlkonfigurationen. Mit einer Erkennungsrate von über 98% für OWASP Top 10 CI/CD-Risiken und einer durchschnittlichen Scan-Dauer von unter 300 Millisekunden pro Manifest ist es das schärfste Schwert im Arsenal eines jeden Solo-Entwicklers.

### Kernfunktionen

- **Dynamische Secrets-Exfiltration-Analyse (DSEA)**  
  Erkennt hartcodierte API-Keys, Tokens und Zertifikate in Echtzeit während des Pre-Commit-Hooks. Unterstützt 40+ Pattern-Typen (AWS ARN, GitHub PAT, Stripe API Key) mit einer False-Positive-Rate von unter 0,5%. Parameter: `--secret-depth 3` (rekursive Suche in Base64-kodierten Blobs) und `--entropy-threshold 4.2` (Shannon-Entropie-Schwellwert).

- **IaC-Constraint-Validierung (ICV)**  
  Prüft Terraform-Pläne und Kubernetes-Deployments gegen benutzerdefinierte Sicherheits-Policies. Ermöglicht die Definition von Regeln wie `container.privileged == false` oder `ingress.tls.enabled == true` in YAML-basierten Policy-Dateien. Unterstützt Multi-Cluster-Simulationen mit bis zu 10.000 Nodes in der Testumgebung.

- **Git-Action-Exploit-Prävention (GAEP)**  
  Analysiert GitHub Actions Workflows auf Command-Injection, unsichere `pull_request_target`-Trigger und Hardcoded Runner-Token. Generiert automatisch sichere Alternativvorschläge mit `--fix-mode dry-run` oder `--fix-mode auto`. Bietet Integration mit PostgreSQL zur zentralen Logging-Speicherung aller Audit-Ergebnisse.

### So verwenden Sie es

1. **Initialen Scan durchführen**  
   ```bash
   deployguard scan --path ./deployments --format json --output report.json --secret-depth 3 --entropy-threshold 4.2
   ```  
   Der Befehl analysiert alle YAML-, JSON- und HCL-Dateien im angegebenen Pfad. Die Ausgabe erfolgt als JSON-Report mit detaillierten Schwachstellen-Beschreibungen, CVSS-Scores und Fix-Vorschlägen.

2. **Policy-basierte Validierung aktivieren**  
   ```bash
   deployguard validate --policy-file ./security-policies.yaml --terraform-plan ./plan.out --kubeconfig ~/.kube/config
   ```  
   Hier wird ein Terraform-Plan gegen eine benutzerdefinierte Policy-Datei geprüft. Die Policy-Datei enthält Regeln wie:  
   ```yaml
   rules:
     - id: no-privileged-containers
       match: spec.containers[*].securityContext.privileged
       condition: "== false"
       severity: critical
   ```

3. **Git-Hook automatisch installieren**  
   ```bash
   deployguard hook --install --repo . --hook-type pre-commit --db-connection postgresql://user:pass@localhost:5432/deployguard
   ```  
   Installiert einen Pre-Commit-Hook, der bei jedem `git commit` automatisch einen Secret-Scan durchführt. Ergebnisse werden in einer PostgreSQL-Datenbank persistiert, um Trends und wiederkehrende Probleme zu analysieren.

### Beste Praktiken

- **Shift-Left-Security mit Git-Hooks**  
  Integrieren Sie DeployGuard als Pre-Commit-Hook in Ihr Repository. Dadurch werden Secrets und IaC-Fehlkonfigurationen abgefangen, bevor sie überhaupt in den Git-History landen. Kombinieren Sie dies mit einem CI/CD-Trigger (`deployguard scan --ci-mode`), der bei jedem Pull-Request automatisch einen vollständigen Audit durchführt.

- **Policy-as-Code für Multi-Environment-Deployments**  
  Definieren Sie separate Sicherheits-Policies für Development, Staging und Production. Nutzen Sie die `--environment`-Flag, um unterschiedliche Regelwerke pro Umgebung zu erzwingen. Beispiel: In Production darf `imagePullPolicy` nicht auf `Always` gesetzt sein, während in Development `privileged: true` für Debugging-Zwecke erlaubt ist.
