# deployguard - DEP_023

## Übersicht

DeployGuard ist eine statische Analyse-Engine für Solo-Entwickler, die Code-Sicherheitslücken vor dem Deployment aufdeckt – mit Fokus auf Python, JavaScript und Rust. Zielsprache: Deutsch.

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard ist ein professionelles Tool, das Schwachstellen in CI/CD-Pipelines, Hardcoded Secrets und unsichere Abhängigkeitsbäume erkennt, bevor sie in die Produktion gelangen. Es arbeitet mit einer Latenz von unter 50 ms pro Commit und integriert sich direkt in Ihre lokale Entwicklungsumgebung.

### Kernfunktionen

- **Secret-Leak-Detektion mit Regex-Engine**: Erkennt API-Keys, Token und Passwörter in über 20 gängigen Formaten (z. B. `sk-…`, `ghp_…`, `AKIA…`). Konfigurierbare Regeln über `deployguard.secrets.yaml`. Ausgabe als JSON mit Zeilen- und Dateiangabe.
- **Dependency-Trust-Score**: Analysiert die transitive Abhängigkeitskette (max. 5 Ebenen) und bewertet jedes Paket nach CVSS-Score, Repository-Sterne und letztem Update-Datum. Schwellwert einstellbar über `--min-trust 80`. Ergebnisse werden in einer lokalen SQLite-Datenbank (`~/.deployguard/audit.db`) gespeichert.
- **Pre-Commit-Hook-Integration**: Automatische Prüfung bei jedem `git commit`. Blockiert den Commit, wenn kritische Schwachstellen (CVSS ≥ 7.0) gefunden werden. Konfiguration über `.pre-commit-config.yaml`. Ausgabe als farbige Terminal-Liste mit Exit-Code 1 bei Fehlschlag.

### So verwenden Sie es

1. **Installation und Initialisierung**  
   Führen Sie `pip install deployguard --upgrade` aus. Initialisieren Sie das Projekt mit `deployguard init --lang python --ci github`. Dadurch wird eine `.deployguard.yaml` mit Standardregeln erstellt. Überprüfen Sie die Konfiguration mit `deployguard config --validate`.

2. **Erstmaliger Scan und Analyse**  
   Starten Sie einen vollständigen Scan mit `deployguard scan --path ./src --severity high --output json --report ./reports/scan.json`. Das Tool durchsucht rekursiv alle `.py`, `.js`, `.ts`, `.rs` und `.yaml` Dateien. Die Ausgabe enthält eine Liste von Fundstellen mit Dateipfad, Zeilennummer, gefundenem Pattern und CVSS-ähnlichem Risikoscore (0–10).

3. **Integration in den Pre-Commit-Hook**  
   Fügen Sie folgende Zeilen in Ihre `.pre-commit-config.yaml` ein:  
   ```yaml
   - repo: https://github.com/xingfangwang-eng/deployguard
     rev: v1.2.3
     hooks:
       - id: deployguard
         args: ["--min-severity", "medium", "--block-on-error"]
   ```  
   Testen Sie den Hook mit `pre-commit run --all-files`. Bei einem kritischen Fund wird der Commit mit Exit-Code 1 abgebrochen und eine detaillierte Meldung ausgegeben.

### Beste Praktiken

- **Praxis 1: Regelmäßige Aktualisierung der Regelbasis**  
  Führen Sie wöchentlich `deployguard update --rules` aus, um die neuesten Secret-Patterns und CVSS-Datenbanken zu laden. Alte Regeln übersehen oft moderne Angriffsmuster (z. B. JWT-Token in `.env`-Dateien). Nutzen Sie `deployguard diff --from last_week` um Änderungen nachzuvollziehen.

- **Praxis 2: Kombinieren Sie DeployGuard mit einem CI/CD-Proxy**  
  Setzen Sie DeployGuard als Gatekeeper vor Ihrem Deployment-Trigger ein. Konfigurieren Sie in GitHub Actions:  
  ```yaml
  - name: DeployGuard Check
    run: deployguard scan --path . --severity critical --fail-on-error
  ```  
  Dadurch wird verhindert, dass unsicherer Code überhaupt in die Build-Pipeline gelangt. Archivieren Sie alle Scan-Reports als Build-Artefakte für spätere Audits.
