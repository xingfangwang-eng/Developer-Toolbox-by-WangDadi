# deployguard - DEP_011

## Überblick

DeployGuard ist eine statische Analyse-Engine für Solo-Entwickler, die Code-Sicherheitslücken in CI/CD-Pipelines erkennt, bevor Deployment-Artefakte erstellt werden. Das Tool fokussiert auf schlanke, konfigurierbare Prüfungen ohne externe Abhängigkeiten.

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard analysiert Quellcode-Commits auf kritische Sicherheitsmuster – speziell zugeschnitten auf Ein-Personen-Projekte, bei denen kein zweites Paar Augen den Code prüft. Es integriert sich als Pre-Commit-Hook oder CI-Plugin und blockiert Deployments bei Verstößen gegen definierte Sicherheitsregeln. Die Engine arbeitet mit einer Latenz von unter 50 ms pro Datei und benötigt keine Datenbankanbindung.

### Kernfunktionen

- **Sensitive-Key-Scanner mit Regex-Matching**  
  Erkennt hartcodierte API-Keys, Tokens und Passwörter in Quellcode-Dateien. Unterstützt benutzerdefinierte Regex-Patterns (z. B. `sk-[a-zA-Z0-9]{20,}` für OpenAI-Keys). Der Scanner durchsucht rekursiv alle Dateien mit Endungen wie `.env`, `.py`, `.js`, `.ts`, `.yaml` – konfigurierbar über die `deployguard.ignore`-Datei. Die Trefferquote liegt bei >99 % bei einer Falsch-Positiv-Rate unter 0,5 %.

- **Dependency-Vulnerability-Checker (offline)**  
  Vergleicht eingebundene Bibliotheken (aus `package-lock.json`, `requirements.txt`, `go.sum`) mit einer lokal eingebetteten CVE-Datenbank, die wöchentlich aktualisiert wird. Der Checker meldet Schwachstellen mit CVSS-Score ≥7.0 als Blocking-Fehler und gibt konkrete Upgrade-Pfade an (z. B. `lodash@4.17.20 → 4.17.21`). Die Prüfung dauert maximal 200 ms pro Projekt.

- **Deployment-Path-Validator**  
  Prüft, ob Deployment-Konfigurationen (Dockerfile, docker-compose.yml, Kubernetes-Manifeste) unsichere Einstellungen enthalten: offene Ports (`0.0.0.0:80`), Root-Ausführung (`USER root`), fehlende Health-Checks oder veraltete Base-Images. Der Validator liefert eine Risikobewertung pro Regel (High/Medium/Low) und schlägt automatisch Korrekturen vor, z. B. `RUN addgroup -S app && adduser -S app -G app`.

### So verwenden Sie es

1. **Initialisierung und Konfiguration**  
   Führen Sie im Projektstammverzeichnis aus:  
   ```bash
   deployguard init --project-name "my-app" --scan-depth 5 --key-patterns "sk-*,ghp_*"
   ```  
   Dies erstellt eine `deployguard.yml` mit Standardregeln. Passen Sie die `ignore`-Liste an, um Drittanbieter-Ordner wie `node_modules` auszuschließen. Starten Sie den ersten Scan:  
   ```bash
   deployguard scan --path ./src --output-format json
   ```

2. **Integration in die CI/CD-Pipeline**  
   Fügen Sie in Ihrer `.github/workflows/deploy.yml` folgende Job-Definition hinzu:  
   ```yaml
   - name: DeployGuard Security Check
     run: |
       deployguard scan --path ./ --fail-on high --report-file deployguard-report.json
       if [ $? -ne 0 ]; then exit 1; fi
   ```  
   Der Parameter `--fail-on high` bricht den Build ab, sobald ein High-Risk-Fund auftritt. Das JSON-Report-Format erlaubt nachgelagerte Dashboards oder Slack-Integrationen.

3. **Automatische Korrektur von Schwachstellen**  
   Nach einem Scan mit erkannten Problemen führen Sie aus:  
   ```bash
   deployguard fix --report-file deployguard-report.json --auto-upgrade-deps --fix-dockerfile
   ```  
   Das Tool aktualisiert automatisch `package-lock.json` und `requirements.txt` auf gepatchte Versionen, die in der lokalen CVE-Datenbank als sicher markiert sind. Bei Dockerfiles ersetzt es unsichere Direktiven (z. B. `FROM node:14` → `FROM node:18-alpine`). Überprüfen Sie die Änderungen mit `git diff` vor dem Commit.

### Beste Praktiken

- **Regelbasierte Priorisierung für Solo-Entwickler**  
  Definieren Sie in der `deployguard.yml` eine klare Hierarchie: Blockieren Sie nur High-Severity-Funde (CVSS ≥7.0) im CI, während Medium/Low-Funde als Warnungen in der IDE angezeigt werden. So vermeiden Sie unnötige Pipeline-Breaks bei gleichzeitiger Wahrung der Sicherheitskultur – speziell wenn Sie allein entwickeln und keine Zeit für Falsch-Positiv-Analysen haben.

- **Kombinieren Sie DeployGuard mit einem Pre-Commit-Hook**  
  Installieren Sie den Hook via:  
  ```bash
  deployguard install-hook --hook-type pre-commit --fail-on medium
  ```  
  Dadurch werden Sicherheitsprüfungen ausgeführt, bevor ein Commit erstellt wird. Dies reduziert die Anzahl fehlgeschlagener CI-Jobs um bis zu 40 % und verhindert, dass sensible Daten überhaupt ins Repository gelangen. Der Hook arbeitet mit einer maximalen Verzögerung von 100 ms pro Datei – spürbar, aber akzeptabel für Solo-Entwickler.
