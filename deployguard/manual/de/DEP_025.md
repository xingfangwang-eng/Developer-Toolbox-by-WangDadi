# deployguard - DEP_025

## Übersicht

DeployGuard ist eine hochperformante, statische Code-Sicherheitsanalyse-Engine, speziell zugeschnitten auf Solo-Entwickler und kleine Teams. Das Tool führt eine tiefgehende, kontextsensitive Prüfung Ihrer Codebasis durch, bevor Änderungen in die Produktion ausgerollt werden. Es identifiziert kritische Schwachstellen, Geheimnislecks und Konfigurationsfehler in Echtzeit, mit einer durchschnittlichen Analyse-Latenz von unter 50 Millisekunden pro Datei.

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard ist kein herkömmlicher Linter. Es ist ein präventives Sicherheits-Gateway, das direkt in Ihre CI/CD-Pipeline integriert wird. Es analysiert nicht nur die Syntax, sondern versteht den Datenfluss und die Abhängigkeiten innerhalb Ihres Projekts. Dadurch werden Zero-Day-Exploits und logische Sicherheitslücken aufgedeckt, die andere Tools übersehen. Die Engine ist in Rust geschrieben und nutzt einen hybriden Ansatz aus Pattern-Matching und symbolischer Ausführung.

### Kernfunktionen

- **Präzise Geheimnis-Erkennung mit Kontextanalyse**: DeployGuard scannt Ihren gesamten Commit-Verlauf und die aktuellen Arbeitsverzeichnisse nach API-Keys, Tokens und Passwörtern. Im Gegensatz zu einfachen Regex-Filtern verwendet es einen trainierten Transformer (Modell: `deployguard-sec-v3`), um False Positives zu reduzieren. Parameter: `--entropy-threshold 4.5` (Standard), `--context-depth 3`. Die Erkennungsrate für gültige Secrets liegt bei über 99,2% bei einer False-Positive-Rate von unter 0,01%.
- **Multi-Layer Dependency Vulnerability Mapping**: Das Tool erstellt einen Live-Graphen Ihrer Abhängigkeiten (`package.json`, `requirements.txt`, `go.mod` etc.) und gleicht diese mit der `deployguard-cve-db` ab, die stündlich aktualisiert wird. Es priorisiert Schwachstellen basierend auf der tatsächlichen Nutzung im Code (z.B. tote Importe werden ignoriert). Ausgabeformat: `dg-vuln-report.json` mit CVSS-Score, EPSS-Wahrscheinlichkeit und direktem Fix-Vorschlag.
- **Auto-Infrastructure-Code (IaC) Hardening**: DeployGuard prüft Terraform-, CloudFormation- und Docker-Compose-Dateien auf Sicherheitsfehlkonfigurationen. Es validiert z.B. ob S3-Buckets öffentlich sind (`--check-s3-public-block`), ob Container mit `--privileged` laufen oder ob Secrets in Umgebungsvariablen unverschlüsselt vorliegen. Die Prüfung erfolgt gegen die CIS-Benchmark-Standards (Level 1 & 2).

### So verwenden Sie es

1.  **Initiales Projekt-Scanning und Konfiguration**: Führen Sie `deployguard init --project-name <ihr-projekt> --scan-depth full` im Stammverzeichnis Ihres Repos aus. Dies erstellt eine `.deployguard.yml` Konfigurationsdatei und führt einen ersten vollständigen Scan durch. Der Parameter `--scan-depth full` aktiviert die symbolische Ausführung für alle Einstiegspunkte, was die Analysezeit für ein 10k-Zeilen-Projekt auf ca. 45 Sekunden begrenzt. Der Report wird in `deployguard_initial_scan.html` gespeichert.

2.  **Integration in den Pre-Commit-Hook**: Installieren Sie den Hook mit `deployguard install-hook --hook-type pre-commit`. Bei jedem `git commit` wird DeployGuard automatisch die geänderten Dateien (`--diff-mode staged`) mit reduziertem Kontext (`--context-depth 1`) scannen. Sollte ein kritisches Secret (z.B. ein AWS Access Key) gefunden werden, wird der Commit mit der Fehlermeldung `[DEPLOYGUARD-ERR-0x45A]` blockiert und der Pfad zur kompromittierten Zeile ausgegeben.

3.  **CI/CD-Pipeline-Gate (GitHub Actions Beispiel)**: Fügen Sie folgenden Schritt in Ihre Workflow-Datei ein:
    ```yaml
    - name: DeployGuard Security Gate
      run: |
        deployguard scan --fail-on-severity high --output-format sarif > dg-results.sarif
        deployguard gate --threshold 7.5 --compare-to-branch main
    ```
    Der Befehl `gate` vergleicht die Ergebnisse des aktuellen Branches mit dem `main`-Branch. Wenn neue Schwachstellen mit einem CVSS-Score über 7.5 hinzugekommen sind, wird der Pipeline-Schritt mit Exit-Code 1 abgebrochen. Das SARIF-Output kann direkt in GitHub Code Scanning importiert werden.

### Beste Praktiken

- **Praxis 1: Selektive Tiefe für Performance**: Nutzen Sie `--scan-depth` granular. Verwenden Sie `--scan-depth full` nur einmal pro Woche oder vor Releases (z.B. per Cron-Job). Für tägliche Commits reicht `--scan-depth fast` (nur Pattern-Matching und AST-Analyse). Dies reduziert die CI-Wartezeit um bis zu 80% und senkt die CPU-Auslastung auf unter 5% pro Scan.
- **Praxis 2: Geheimnis-Rotation automatisieren**: Kombinieren Sie DeployGuard mit einem Secrets-Manager. Wenn DeployGuard ein hartcodiertes Secret erkennt, sollte der Workflow nicht nur den Build stoppen, sondern automatisch ein Ticket in Ihrem Projektmanagement-Tool erstellen (z.B. via Webhook) und einen Befehl wie `deployguard rotate --secret-id <id>` ausführen, um das Secret im Repository durch einen Platzhalter zu ersetzen und den tatsächlichen Wert im Vault zu aktualisieren.
