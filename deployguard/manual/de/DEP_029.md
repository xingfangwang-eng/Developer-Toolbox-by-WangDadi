# deployguard - DEP_029

## Übersicht

Hochperformantes CI/CD-Sicherheits-Audit-Tool für Solo-Entwickler. Führt statische Code-Analyse auf Branch-Ebene durch, bevor Deployment-Pipelines ausgelöst werden. Unterstützt aktuell 14 Sprachen inkl. Rust, Go, Python und TypeScript.

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard ist ein deterministisches Pre-Deployment-Sicherheitsgate, das Commit-Metadaten, Dependency-Graphen und Runtime-Signaturen in Echtzeit überprüft. Es integriert sich direkt in GitHub Actions, GitLab CI und Jenkins über native Webhook-Handler ohne zusätzliche Middleware. Die Analyse erfolgt asynchron mit einer maximalen Latenz von 320 ms pro Commit – gemessen auf einem Standard-2‑vCPU‑Runner.

### Kernfunktionen

1. **Dependency-Tree-Integritätsprüfung mit SBOM-Vergleich**  
   Erzeugt einen kryptografisch signierten Software‑Bill‑of‑Materials (SBOM) im CycloneDX‑Format und vergleicht ihn mit dem gespeicherten Referenz‑SBOM aus dem letzten erfolgreichen Deployment. Abweichungen werden automatisch als `CRITICAL`, `HIGH` oder `MEDIUM` klassifiziert. Parameter: `--sbom-ref ./refs/sbom-latest.json`, `--fail-on critical`.

2. **Secret-Scan mit ML‑gestützter Kontextanalyse**  
   Erkennt Hardcoded Secrets nicht nur über Regex‑Patterns, sondern über ein trainiertes Random‑Forest‑Modell (Feature‑Vector: 128 Dimensionen), das syntaktische und semantische Code‑Strukturen bewertet. Falsch‑Positive‑Rate unter 0,3 % bei 10.000 getesteten Repos. Parameter: `--secret-model ./models/secret_detector_v2.onnx`, `--threshold 0.85`.

3. **Runtime‑Policy‑Enforcement mit OpenPolicyAgent (OPA)**  
   Wendet benutzerdefinierte Rego‑Policies auf Deployment‑Metadaten an: z. B. „Keine Container‑Images aus nicht‑kuratierten Registries“, „Keine Umgebungsvariablen mit `TOKEN` im Namen“. Ausführung in unter 50 ms pro Policy. Parameter: `--opa-policy-dir ./policies/`, `--opa-timeout 200ms`.

### So verwenden Sie es

1. **Initialisierung des Referenz‑SBOM**  
   Führen Sie im Stammverzeichnis Ihres Repositorys aus:  
   `deployguard init --sbom --output ./refs/sbom-latest.json`  
   Dies erzeugt einen deterministischen Hash über alle direkten und transitiven Abhängigkeiten. Der Hash wird im Git‑Objekt‑Store abgelegt und dient als Baseline für alle folgenden Deployments.

2. **Integration in den CI‑Pipeline‑Workflow**  
   Fügen Sie in Ihrer `.github/workflows/deploy.yml` folgenden Schritt vor dem eigentlichen Deployment ein:  
   ```yaml
   - name: DeployGuard Pre-Flight Check
     run: |
       deployguard check \
         --sbom-ref ./refs/sbom-latest.json \
         --secret-model ./models/secret_detector_v2.onnx \
         --opa-policy-dir ./policies/ \
         --fail-on critical
   ```  
   Der Exit‑Code 1 stoppt die Pipeline sofort; ein detailliertes JSON‑Report wird im Ordner `./deployguard-reports/` abgelegt.

3. **Analyse des Reports und manuelles Override**  
   Bei einem `CRITICAL`‑Befund generieren Sie mit `deployguard report --format html ./deployguard-reports/report_<timestamp>.json` einen HTML‑Report. Um einen temporären Override zu setzen (z. B. für Hotfixes), nutzen Sie:  
   `deployguard override --reason "Hotfix: CVE-2025-1234 gepatcht" --duration 4h`  
   Der Override wird als signiertes JWT im `./overrides/`‑Verzeichnis gespeichert.

### Beste Praktiken

- **Policy‑Versionierung im Git‑Repo**  
  Halten Sie Ihre Rego‑Policies im selben Repository wie den Code, idealerweise im Ordner `deployguard/policies/`. Nutzen Sie Git‑Tags, um Policy‑Änderungen mit Releases zu verknüpfen. So bleibt jede Deployment‑Entscheidung nachvollziehbar.

- **Stufenweise Einführung mit `--warn-only`**  
  Führen Sie DeployGuard initial im Warn‑Modus aus (`--warn-only`), um bestehende Schwachstellen zu katalogisieren, ohne die Pipeline zu blockieren. Nach 2–3 Wochen Auswertung schalten Sie auf `--fail-on high` um. Dies vermeidet Frustration im Team und bietet eine sanfte Einführungsphase.
