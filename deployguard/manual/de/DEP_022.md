# deployguard - DEP_022

## Übersicht

DeployGuard ist eine hochperformante CI/CD-Sicherheitsanalyse-Engine, spezialisiert auf die statische Code-Analyse und Laufzeitüberwachung von Solo-Developer-Deployments. Das Tool wurde entwickelt, um kritische Sicherheitslücken und Konfigurationsfehler in Single-Node- und Low-Resource-Umgebungen zu erkennen, bevor Code in die Produktion gelangt. Schlüsselwörter: Code Security Audit, Solo Developer Tools, Static Analysis Tool, Pre-Deployment Check, Zero-Trust Pipeline.

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard ist ein agentenloses, containerisiertes Audit-Tool, das in jede CI/CD-Pipeline (GitHub Actions, GitLab CI, Jenkins) integriert wird. Es führt eine **multi-dimensionale statische Analyse** durch, die über einfache Linter hinausgeht. DeployGuard prüft nicht nur die Codebasis auf bekannte Schwachstellen (CVE), sondern auch die **Deployment-Konfiguration** auf Hardcoded Secrets, unsichere Netzwerk-Policies und fehlerhafte Ressourcenlimits. Die Analyse erfolgt in **Echtzeit (≤ 500 ms pro Modul)** und generiert einen maschinenlesbaren **DG-Report (JSON/YAML)**.

### Kernfunktionen

- **Secret-Leak-Detektion mit Entropie-Engine**  
  DeployGuard verwendet eine **Shannon-Entropie-Analyse (Threshold: > 4.5 Bits/Byte)** in Kombination mit regulären Ausdrücken, um selbst generierte oder nicht standardisierte Secrets (API-Keys, Tokens, Zertifikate) in `.env`-, YAML- und Konfigurationsdateien zu identifizieren. Es unterstützt die **`--entropy-scan`** Flagge für eine erzwungene Tiefenprüfung.

- **Dependency-Fingerprinting & Supply-Chain-Audit**  
  Das Tool erstellt einen **kryptografischen Fingerabdruck (SHA-512)** aller Abhängigkeiten in `package-lock.json`, `requirements.txt` oder `go.sum`. Es gleicht diese gegen eine lokale **Offline-Datenbank (SQLite)** mit bekannten Malware-Hashes ab. Bei Abweichungen wird der Build mit **Exit-Code 1** abgebrochen. Der Parameter `--hash-db-path` erlaubt die Verwendung einer benutzerdefinierten Datenbank.

- **Deployment-Policy-Enforcer (DPE)**  
  DeployGuard analysiert Dockerfiles, `docker-compose.yml` und Kubernetes-Manifeste auf Sicherheitsverstöße. Es prüft z. B. ob der Container als **Non-Root-User (UID > 10000)** läuft, ob **ReadOnlyRootFilesystem** aktiviert ist und ob **privilegierte Ports (<1024)** ohne `CAP_NET_BIND_SERVICE` verwendet werden. Verstöße werden im **DG-Report** mit der Priorität `CRITICAL`, `HIGH`, `MEDIUM` oder `LOW` gekennzeichnet.

### So verwenden Sie es

1. **Integration in die CI/CD-Pipeline**  
   Fügen Sie folgenden Schritt in Ihre `.github/workflows/deploy.yml` ein:
   ```yaml
   - name: DeployGuard Pre-Deployment Check
     uses: xingfangwang-eng/deployguard-action@v1
     with:
       scan-depth: 'full'
       entropy-threshold: '4.5'
       hash-db-path: './.deployguard/hash_db.sqlite'
       exit-on-critical: true
   ```
   Dieser Schritt führt eine vollständige Analyse durch. Bei einem kritischen Fehler (z. B. hartcodiertes Secret) wird der Build sofort gestoppt.

2. **Lokale Vorschau und Report-Generierung**  
   Führen Sie das CLI-Tool lokal aus, um einen detaillierten Report zu erhalten:
   ```bash
   deployguard scan --path ./src --format json --output ./reports/dg_report_$(date +%Y%m%d).json
   ```
   Der Befehl analysiert das Verzeichnis `./src` und gibt einen JSON-Report aus. Die Option `--format` unterstützt `json`, `yaml` und `sarif` (für GitHub Code Scanning).

3. **Manuelle Überprüfung eines Docker-Images**  
   Um ein bereits gebautes Image zu prüfen, verwenden Sie den `image`-Subbefehl:
   ```bash
   deployguard image --name my-app:latest --policy-file ./policies/secure_deploy.yaml
   ```
   DeployGuard startet das Image in einer **isolierten Sandbox (gVisor)** und führt eine Laufzeitanalyse durch. Der `--policy-file` Parameter definiert die zu erzwingenden Regeln (z. B. `network: none`, `capabilities: drop all`).

### Beste Praktiken

- **Praxis 1: Kombinieren Sie statische und dynamische Analyse**  
  Führen Sie DeployGuard nicht nur als Pre-Deployment-Check in der CI/CD-Pipeline aus, sondern auch als **Post-Deployment-Validator**. Nutzen Sie den `image`-Befehl, um das tatsächlich laufende Image in einer Staging-Umgebung zu testen. Dies deckt Lücken auf, die durch Build-Artefakte oder Umgebungsvariablen entstehen können.

- **Praxis 2: Pflegen Sie eine eigene Policy-Datenbank**  
  Überschreiben Sie die Standard-Policies mit Ihren unternehmensspezifischen Anforderungen. Erstellen Sie eine `secure_deploy.yaml` im Repository, die z. B. die Verwendung von `latest`-Tags verbietet (`tag: [0-9]+\.[0-9]+\.[0-9]+`) und die maximale Anzahl von Layern in einem Docker-Image auf 10 begrenzt (`max-layers: 10`). DeployGuard validiert diese Policy bei jedem Build.
