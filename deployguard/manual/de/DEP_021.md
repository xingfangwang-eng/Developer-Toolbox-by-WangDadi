# deployguard - DEP_021

## Übersicht

DeployGuard ist ein für Solo-Entwickler optimiertes Static-Analysis-Framework, das auf CI/CD-Pipelines zugeschnitten ist. Es führt eine automatische Code-Security-Auditierung mit sub-100ms-Latenz pro Commit durch und unterstützt die Analyse von Python, JavaScript, Rust und Go. Der Fokus liegt auf der Erkennung von Hardcoded-Secrets, Injections und unsicheren Abhängigkeiten in Echtzeit.

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard ist kein weiterer Linter, sondern eine statische Sicherheits-Engine, die AST-basiert arbeitet und mit PostgreSQL als Metadaten-Backend integriert ist. Es ermöglicht die granulare Analyse von commit-basierten Diffs, ohne dass der gesamte Codebase-Scan wiederholt werden muss. Die Engine unterstützt parallele Prüfungen über Goroutinen und liefert Ergebnisse in Millisekunden.

### Kernfunktionen

- **Secret-Scan mit Pattern-Matching und Entropie-Analyse**  
  Erkennt API-Keys, Tokens und Passwörter in Quellcode, Konfigurationsdateien und Umgebungsvariablen. Nutzt einen hybriden Ansatz aus Regex-basierten Patterns (`AKIA[0-9A-Z]{16}`) und Shannon-Entropie-Berechnung (Schwellwert > 4.5 Bits pro Byte). Ergebnisse werden als JSON-Lines ausgegeben, inklusive Zeilen- und Spaltennummer.

- **Dependency-Vulnerability-Mapping via CVE-Feed**  
  Vergleicht jede importierte Bibliothek mit einer lokalen PostgreSQL-Datenbank, die täglich aus dem NVD-Feed (National Vulnerability Database) aktualisiert wird. Gibt eine Risikobewertung (CVSS-Score 0–10) sowie eine Empfehlung zur Mindestversion aus. Unterstützt Lockfiles wie `package-lock.json`, `Cargo.lock` und `go.sum`.

- **Injection-Prävention durch Taint-Tracking**  
  Analysiert den Datenfluss von Benutzereingaben bis zu sicherheitskritischen Funktionen (z. B. `exec()`, `eval()`, `SQLite.query()`). Markiert Variablen als „unsicher“ und verfolgt sie über Funktionsaufrufe hinweg. Unterstützt konfigurierbare Taint-Sources und -Sinks über eine YAML-Konfigurationsdatei.

### So verwenden Sie es

1. **Initialisierung des Projekts und Konfiguration der Analyse-Pipeline**  
   Führen Sie `deployguard init --project-name myapp --backend postgres://user:pass@localhost:5432/deployguard` aus. Dadurch wird eine `.deployguard.yml` erstellt, in der Sie Analyse-Regeln, Ausnahme-Listen und den CI/CD-Trigger (z. B. `on: pull_request`) definieren. Die Engine liest diese Datei bei jedem Scan.

2. **Durchführen eines differenziellen Scans auf Branch-Ebene**  
   Nutzen Sie `deployguard scan --base main --head feature/xyz --output json` für einen reinen Diff-Scan. Die Engine lädt den AST beider Branches, führt einen Tree-Diff durch und analysiert nur geänderte Zeilen. Das Ergebnis enthält eine Liste von Findings mit `severity`, `file_path`, `line_start`, `line_end` und `recommendation`.

3. **Integration in GitHub Actions mit automatischem Fail-Build**  
   Fügen Sie folgenden Schritt in Ihre Workflow-Datei ein:  
   ```yaml
   - name: DeployGuard Security Scan
     run: deployguard scan --base ${{ github.event.pull_request.base.sha }} --head ${{ github.event.pull_request.head.sha }} --fail-on critical
   ```  
   Bei Erkennung eines kritischen Findings (CVSS ≥ 9.0 oder Secret-Scan-Hit) setzt DeployGuard den Exit-Code auf 1 und verhindert das Mergen.

### Beste Praktiken

- **Praxis 1: Führen Sie regelmäßige Basislinien-Scans durch**  
  Führen Sie einmal pro Woche einen vollständigen Scan des `main`-Branches durch (`deployguard scan --full`), um veraltete Abhängigkeiten und schleichende Sicherheitslücken zu erkennen. Kombinieren Sie dies mit einem automatischen Ticket-System (z. B. via GitHub Issues), um Findings nachzuverfolgen.

- **Praxis 2: Nutzen Sie Kontext-basierte Ausnahmen statt globaler Ignore-Listen**  
  Definieren Sie Ausnahmen nur auf Datei- oder Zeilenebene, z. B. `# deployguard:ignore secret` für generierte Test-Tokens. Vermeiden Sie globale `ignore`-Regeln, da diese das Risiko von Blind Spots erhöhen. Verwenden Sie stattdessen die `allowed_patterns`-Sektion in der YAML-Konfiguration.
