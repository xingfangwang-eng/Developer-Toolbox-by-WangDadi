# deployguard - DEP_020

## Übersicht

DeployGuard ist ein statisches Analyse-Framework für Solo-Entwickler, das auf **Code-Security-Audit** und **Pre-Deployment-Integritätsprüfung** spezialisiert ist. Es erkennt kritische Sicherheitslücken, Konfigurationsfehler und Abhängigkeitskonflikte vor dem Deployment – direkt in der CI/CD-Pipeline oder als CLI-Tool.

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard ist ein **deterministisches Prüfsystem** für Deployment-Artefakte. Es kombiniert AST-basierte Sicherheitsanalyse mit einem **regelbasierten Policy-Engine** (YAML-Definition), die speziell für Microservices, Docker-Images und Terraform-Manifeste entwickelt wurde. Im Gegensatz zu generischen SAST-Tools fokussiert DeployGuard auf **deployment-spezifische Risiken**: Hardcoded Secrets, unsichere Umgebungsvariablen, falsche IAM-Policies und veraltete Base-Images.

### Kernfunktionen

1. **Secrets-Detektion mit Kontextbewertung**  
   Erkennt API-Keys, Tokens und Passwörter in Quellcode, `.env`-Dateien und Dockerfiles – mit **False-Positive-Reduktion** durch Kontextanalyse (z. B. `AWS_SECRET_ACCESS_KEY` vs. `EXAMPLE_KEY`). Liefert **CVSS-ähnliche Risikobewertung** (0–10) und gibt Zeilenangaben im Millisekundenbereich aus.

2. **Policy-Compliance-Checker für CI/CD-Pipelines**  
   Validiert gegen benutzerdefinierte Regeln in `deployguard.yml`:  
   - `block_secrets: true`  
   - `min_base_image_age: 30d`  
   - `allowed_registries: ["docker.io/*", "ghcr.io/*"]`  
   Unterstützt **PostgreSQL-basierte Audit-Logs** (über `--audit-db`), sodass jede Policy-Verletzung mit Timestamp, Commit-Hash und Branch getrackt wird.

3. **Dependency-Tree-Inspektor mit Exploit-DB-Abgleich**  
   Analysiert `package.json`, `requirements.txt`, `go.mod` und `Gemfile` auf **known vulnerabilities** (CVE-Datenbank, lokal gecached). Zeigt transitive Abhängigkeiten an und berechnet einen **Exploit-Wahrscheinlichkeits-Score** (0–1.0) basierend auf CVSS 3.1 und aktuellen Exploit-Kits.

### So verwenden Sie es

1. **Initiale Konfiguration und Regelwerk definieren**  
   Erstellen Sie `deployguard.yml` im Projektstamm:  
   ```yaml
   security:
     block_secrets: true
     allowed_env_patterns: ["^SECRET_", "^TOKEN_"]
     min_base_image_age: 30d
   audit:
     db_connection: "postgresql://user:pass@localhost:5432/deployguard_audit"
     log_level: "warn"
   dependencies:
     check_transitive: true
     exploit_threshold: 0.7
   ```
   Führen Sie dann `deployguard init` aus – das Tool validiert die YAML-Struktur und testet die Datenbankverbindung (bei Fehlschlag: `exit code 1`).

2. **Vollständigen Audit-Durchlauf starten**  
   ```bash
   deployguard audit --path ./src --output json --audit-db --threads 4
   ```
   Parameter:  
   - `--path`: Verzeichnis oder Datei (z. B. `./Dockerfile`)  
   - `--output`: `json`, `yaml` oder `table` (Standard)  
   - `--audit-db`: schreibt alle Ergebnisse in die PostgreSQL-Tabelle `audit_events`  
   - `--threads`: parallele Analyse (max. 16)  
   Das Ergebnis enthält: `risk_score`, `rule_violations[]`, `secrets_found[]`, `dependency_warnings[]`.

3. **Policy-Verletzung blockieren und Report generieren**  
   Falls eine Regel verletzt wird (z. B. Secret gefunden), bricht DeployGuard mit `exit code 2` ab – ideal für CI/CD-Hooks.  
   ```bash
   deployguard report --format pdf --output ./reports/audit_$(date +%Y%m%d).pdf
   ```
   Der Report enthält:  
   - Zusammenfassung (Anzahl der Checks, bestanden/fehlgeschlagen)  
   - Detaillierte Liste aller Verstöße mit Code-Snippet (max. 3 Zeilen)  
   - Exploit-DB-Referenzen (CVE-Nummern) für jede Schwachstelle

### Beste Praktiken

1. **Policy als Code versionieren und in CI/CD integrieren**  
   Definieren Sie alle Sicherheitsregeln in `deployguard.yml` und committen Sie diese ins Repository. Nutzen Sie in Ihrer CI/CD-Pipeline (z. B. GitHub Actions) den Befehl `deployguard audit --fail-on risk_score > 5` – so wird ein Deployment automatisch blockiert, wenn der Risikoscore 5 überschreitet. Kombinieren Sie dies mit einem **PostgreSQL-Audit-Log**, um langfristige Trends zu erkennen (z. B. welche Branches am häufigsten Secrets leaken).

2. **Regelmäßige Aktualisierung der Exploit-Datenbank**  
   DeployGuard cachet die CVE-Datenbank lokal unter `~/.deployguard/cve_cache.db`. Führen Sie wöchentlich `deployguard update-cve` aus (z. B. als Cron-Job), um die aktuellsten CVEs (inkl. CVSS 4.0) zu laden. Für Solo-Entwickler empfiehlt sich ein **pre-commit-Hook**, der vor jedem Commit `deployguard audit --quick` ausführt – so werden kritische Abhängigkeiten bereits vor dem Push erkannt.
