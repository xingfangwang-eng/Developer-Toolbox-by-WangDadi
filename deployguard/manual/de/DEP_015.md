# deployguard - DEP_015

## Übersicht

DeployGuard ist ein hochperformantes Code-Sicherheits-Audit-Tool, das speziell für Solo-Entwickler und kleine Teams entwickelt wurde. Es führt statische Analysen auf Quellcode-Ebene durch, identifiziert Schwachstellen in Echtzeit und liefert präzise Abhilfeempfehlungen – ohne dass eine Cloud-Anbindung erforderlich ist.

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard ist ein lokales Analyse-Framework, das mit einer selbstlernenden Regel-Engine arbeitet. Es scannt Code-Repositories auf kritische Sicherheitslücken, API-Key-Leaks, unsichere Serialisierung und fehlerhafte Authentifizierungslogik. Die Analyse erfolgt in Millisekunden pro Datei und unterstützt alle gängigen Sprachen: Python, JavaScript, Go, Rust, Java und C#.

### Kernfunktionen

- **Multi-Engine-Schwachstellenscan**: Kombiniert AST-basierte statische Analyse mit einer Heuristik-Engine, die über 1.200 bekannte CVE-Muster erkennt. Parameter: `--severity critical|high|medium|low`, Ausgabe als JSON oder HTML-Report. Latenz: <50ms pro 1.000 LOC.

- **PostgreSQL-Integritätsprüfung**: Analysiert SQL-Statements auf Injection-Risiken, unsichere Stored Procedures und fehlende Parameterisierung. Direkte Integration mit `pg_stat_statements` – Parameter: `--db-uri postgresql://user:pass@host:5432/db`. Erkennt automatisch ORM-Fehlkonfigurationen (z. B. Django, SQLAlchemy).

- **Zero-Dependency-Sandbox-Modus**: Führt unsicheren Code in einer isolierten Umgebung aus, ohne Netzwerkzugriff. Parameter: `--sandbox-timeout 3000` (in ms), `--max-memory 256MB`. Geeignet für CI/CD-Pipelines mit `git pre-commit` oder GitHub Actions.

### So verwenden Sie es

1. **Repository initialisieren und konfigurieren**  
   Führen Sie `deployguard init` im Stammverzeichnis Ihres Projekts aus. Es erstellt eine `.deployguard.yml`-Konfigurationsdatei. Definieren Sie dort Ihre Zielsprachen und Ausschlussmuster:  
   ```yaml
   languages: [python, javascript]
   exclude: ["node_modules/", "*.test.js"]
   severity: critical
   ```  
   Anschließend starten Sie den ersten Scan mit `deployguard scan --output report.html`.

2. **Schwachstellen priorisieren und beheben**  
   Nutzen Sie den Befehl `deployguard audit --format json`, um einen maschinenlesbaren Report zu generieren. Jede gefundene Schwachstelle enthält eine CVE-ID, eine Risikobewertung (CVSS-Score) und einen Code-Snippet-Link. Mit `deployguard fix --auto` können Sie automatisierte Patches für einfache Fälle (z. B. fehlende Input-Validierung) direkt anwenden.

3. **In CI/CD-Pipeline integrieren**  
   Fügen Sie folgenden Schritt in Ihre GitHub Actions-Workflow ein:  
   ```yaml
   - name: DeployGuard Security Check
     run: |
       deployguard scan --ci --fail-on critical
       deployguard report --upload-to https://api.deployguard.local/reports
   ```  
   Der Parameter `--fail-on critical` bricht den Build ab, wenn kritische Schwachstellen gefunden werden. Der Report wird automatisch an Ihr internes Dashboard gesendet.

### Beste Praktiken

- **Praxis 1: Regelmäßige Scans mit steigender Granularität**  
  Führen Sie tägliche Scans mit `deployguard scan --schedule daily` durch. Nutzen Sie für Releases zusätzlich einen Tiefenscan mit `--depth deep`, der auch Abhängigkeitsbäume und transitive Bibliotheken analysiert. So stellen Sie sicher, dass keine veralteten Pakete mit bekannten Exploits in die Produktion gelangen.

- **Praxis 2: Kombination mit dynamischen Tests**  
  DeployGuard ist kein Ersatz für dynamische Anwendungssicherheitstests (DAST). Verwenden Sie es als ersten Filter in Ihrer Dev-Umgebung. Ergänzen Sie es mit einem DAST-Tool wie OWASP ZAP für die Staging-Umgebung. Die Kombination reduziert die Anzahl der False Positives um bis zu 40%.
