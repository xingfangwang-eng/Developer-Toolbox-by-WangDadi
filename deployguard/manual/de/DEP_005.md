# deployguard - DEP_005

## Übersicht

DeployGuard ist eine hochperformante statische Analyse-Engine für Code-Sicherheitsaudits, speziell optimiert für Solo-Entwickler und kleine Dev-Teams. Das Tool scannt Deployment-Pipelines, CI/CD-Konfigurationen und Quellcode auf Schwachstellen, Geheimnisleaks und Konfigurationsfehler – mit einer durchschnittlichen Scan-Latenz von **≤ 300 ms pro 10.000 LOC** und vollständiger **PostgreSQL-Integration** für Audit-Trail-Persistenz.

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard ist kein weiterer Linter. Es ist eine **deterministische Sicherheits-Orchestrierungsschicht**, die vor jedem Deployment automatisch folgende Prüfungen durchführt: Hardcoded-Secret-Erkennung (z. B. AWS-Keys, JWT-Tokens), unsichere Docker-Image-Referenzen (`latest`-Tag-Verbot), sowie Terraform-State-Leak-Erkennung. Die Engine unterstützt **Multi-Branch-Diff-Analyse** und gibt präzise Risikobewertungen mit CVSS-Scores aus.

### Kernfunktionen

- **Secret-Leak-Detection mit Fuzzy-Matching**  
  Erkennt hartcodierte Credentials in YAML, JSON, ENV und sogar Base64-verschlüsselten Strings. Parameter: `--confidence-threshold 0.92` (min. 92 % Trefferwahrscheinlichkeit), `--scan-depth recursive` (rekursives Scannen aller Unterverzeichnisse). Ausgabe als **JSON-Report mit Zeitstempel in Mikrosekunden-Genauigkeit**.

- **CI/CD-Pipeline-Validierung**  
  Prüft GitHub Actions, GitLab CI und Jenkinsfiles auf unsichere `run:`-Befehle, fehlende `if:`-Bedingungen und nicht signierte Artefakte. Parameter: `--pipeline-format auto`, `--block-on-critical true`. Unterstützt **Multi-Engine-Compare**: Gleicht die Pipeline-Definition mit der tatsächlich ausgeführten Version ab.

- **Dependency-Tree-Sicherheitsaudit**  
  Analysiert transitive Abhängigkeiten auf bekannte CVEs (Datenbankabgleich mit **NVD-API, alle 6 Stunden aktualisiert**). Parameter: `--depth 5` (maximale Rekursionstiefe), `--fail-on-severity high`. Generiert einen **Heatmap-Report** (SVG/HTML) mit farbcodierten Risikostufen.

### So verwenden Sie es

1. **Initialisierung des Audit-Projekts**  
   Führen Sie `deployguard init --project-name my-app --output-dir ./reports` aus. Das Tool erstellt eine `.deployguard.yml`-Konfigurationsdatei mit Standardregeln für **Python, Node.js und Go**. Anschließend wird eine **PostgreSQL-Datenbankverbindung** unter `db:postgresql://localhost:5432/deployguard_audit` eingerichtet – alle Ergebnisse werden dort persistent gespeichert.

2. **Durchführen eines Tiefenscans**  
   Starten Sie den Scan mit `deployguard scan --path ./src --pipeline-file .github/workflows/deploy.yml --secret-scan --dependency-audit`. Das Tool analysiert **alle 12.847 Dateien** in Ihrem Repository, erkennt **3 harte Secrets** (darunter ein Stripe-API-Key in einer `.env.example`) und gibt eine **kritische Warnung** für eine unsignierte Docker-Image-Referenz `ubuntu:latest` aus.

3. **Exportieren des Audit-Reports**  
   Nutzen Sie `deployguard report --format pdf --include-cvss --output ./reports/audit_2025-04-10.pdf`. Der Report enthält **detaillierte Risikomatrizen**, **Code-Snippets mit Markierung der Sicherheitslücken** und **Empfehlungen zur Behebung** – alles mit **exakten Zeilenangaben** (z. B. `src/config.py:47`).

### Beste Praktiken

- **Praxis 1: Pre-Commit-Hook-Integration**  
  Binden Sie DeployGuard als **Pre-Commit-Hook** ein (`deployguard scan --pre-commit`). Dadurch werden Secrets und Sicherheitslücken bereits vor dem ersten Commit erkannt – nicht erst in der CI/CD-Pipeline. Das reduziert die **Mean-Time-to-Detect (MTTD)** auf unter 2 Sekunden pro Commit.

- **Praxis 2: Regelmäßige Datenbankbereinigung**  
  Konfigurieren Sie ein **automatisches Cleanup** der PostgreSQL-Audit-Tabelle nach 90 Tagen (`deployguard db --retention-days 90`). So bleibt die Datenbank performant und Sie vermeiden **Storage-Bloat**. Kombinieren Sie dies mit einem **wöchentlichen Export** als JSON-Backup.
