# deployguard - DEP_014

## Übersicht

Dieses Dokument beschreibt die Architektur und Implementierung von DeployGuard, einem statischen Analyse-Framework für Code-Sicherheitsaudits, optimiert für Solo-Entwickler und kleine DevOps-Teams. Der Fokus liegt auf der Erkennung von Konfigurationsfehlern, Hardcoded Secrets und unerlaubten Deployment-Pfaden in CI/CD-Pipelines.

## Detaillierter Inhalt

### Was ist DeployGuard?

DeployGuard ist ein auf Rust-basiertes CLI-Tool, das Deployment-Manifeste (YAML, JSON, Dockerfile) auf 30+ vordefinierte Sicherheitsregeln prüft. Es integriert sich direkt in den Pre-Commit-Hook und den CI-Build-Prozess, um kritische Schwachstellen zu blockieren, bevor Code in die Produktion gelangt. Die Analyse erfolgt in Echtzeit mit einer Latenz von unter 50 Millisekunden pro Datei.

### Kernfunktionen

1. **Multi-Engine Secret Scanning mit Entropy-Filter**
   - Erkennt Hardcoded Secrets (API-Keys, Tokens, Passwörter) in 15+ Sprachen mittels Regex-Mustern und Shannon-Entropie (Schwellwert: > 3.8 Bits/Byte).
   - Parameter: `--entropy-threshold 3.8`, `--max-depth 5` (Verzeichnistiefe).
   - Unterstützt benutzerdefinierte Pattern-Dateien im HOCON-Format.

2. **Dependency-Tree Audit mit CVSS-Korrelation**
   - Analysiert `package.json`, `Cargo.toml`, `requirements.txt` und gleicht Abhängigkeiten gegen die NVD-Datenbank ab (offline Cache, aktualisiert alle 6 Stunden).
   - Parameter: `--cvss-min 7.0`, `--ignore-dev-deps true`.
   - Gibt einen detaillierten Bericht mit CVE-ID, CVSS-Score und Fix-Version aus.

3. **CI/CD Pipeline Compliance Checker**
   - Prüft GitHub Actions, GitLab CI und Jenkinsfiles auf unsichere Konfigurationen (z.B. `pull_request` mit `write-all` Permissions, Hardcoded Runner Tokens).
   - Parameter: `--pipeline-type github-actions`, `--block-on-warning true`.
     Liefert eine granulare Risikobewertung (Low/Medium/High/Critical) mit automatischen Korrekturvorschlägen.

### So verwenden Sie es

1. **Initiale Konfiguration und Scan-Ausführung**
   ```bash
   deployguard init --project-name "mein-projekt" --ruleset "default+severe"
   deployguard scan --path ./deployments/ --format json --output report.json
   ```
   Dies erstellt eine `.deployguard.yml` Konfigurationsdatei und führt einen vollständigen Scan aller YAML-Dateien im angegebenen Pfad durch. Der JSON-Report enthält Zeilennummern, Dateipfade und den genauen Regelverstoß.

2. **Integration in den Pre-Commit-Hook**
   ```bash
   # In .git/hooks/pre-commit:
   deployguard scan --path . --format compact --exit-code 1
   ```
   Der Hook blockiert den Commit, falls kritische oder hohe Schwachstellen gefunden werden. Der `--exit-code 1` Parameter stellt sicher, dass der Commit-Prozess mit einem Fehler abbricht.

3. **Automatisierte CI-Integration mit Slack-Benachrichtigung**
   ```yaml
   # In .github/workflows/deployguard.yml:
   - name: DeployGuard Security Audit
     run: deployguard scan --path . --format sarif --output results.sarif
   - name: Upload SARIF to GitHub Security Tab
     uses: github/codeql-action/upload-sarif@v3
     with:
       sarif_file: results.sarif
   ```
   Das SARIF-Format ermöglicht die direkte Anzeige der Ergebnisse im GitHub Security Tab. Bei kritischen Fehlern kann ein Slack-Webhook über `--webhook-url` konfiguriert werden.

### Beste Praktiken

- **Praxis 1: Stufenweise Einführung mit Baseline-Erstellung**  
  Führen Sie den ersten Scan mit `--baseline baseline.json` durch, um bestehende Warnungen zu erfassen und nur neue Verstöße zu melden. Dies verhindert Alarmmüdigkeit und ermöglicht eine schrittweise Bereinigung des Codebestands. Aktualisieren Sie die Baseline nach jedem abgeschlossenen Fix-Zyklus.

- **Praxis 2: Regel-Set an die Einsatzumgebung anpassen**  
  Nutzen Sie `--ruleset custom` und definieren Sie eigene Regeln in `.deployguard/rules/`. Für eine Kubernetes-Umgebung empfehlen wir das Aktivieren der Regel `k8s-privileged-container` (ID: K8S-012) und das Deaktivieren von `docker-latest-tag` (ID: DKR-003), falls Sie ein striktes Tagging-System verwenden. Führen Sie regelmäßige Audits mit `--ruleset "custom+community"` durch, um von neuen Community-Regeln zu profitieren.
