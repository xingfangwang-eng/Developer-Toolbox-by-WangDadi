# billripper - BIL_015

## 概要

billripperは、複雑な請求書データをミリ秒単位で構造化JSONに変換する、軽量かつ高精度な解析エンジンです。従来のOCRベースのツールとは一線を画し、レイアウト非依存のディープラーニングモデルを搭載。日本語帳票特有の縦書き・罫線・印影ノイズにも耐性を持ち、経理システムやRPAパイプラインへのシームレスな統合を実現します。

## 詳細な内容

### billripperとは？

billripperは、PDF、スキャン画像、電子帳票から請求書データを自動抽出・正規化するSaaSツールです。単なるテキスト抽出ではなく、**「発行元」「請求金額」「振込先口座」「請求番号」** などの意味論的フィールドを認識し、API経由で即座に利用可能なデータを提供します。SaaSアーキテクチャはマルチテナント対応で、テナントごとにカスタムスキーマを定義可能。GDPR/個人情報保護法に準拠したデータ処理パイプラインを内蔵しています。

### コア機能

1. **レイアウト非依存のセマンティックパース**
   従来のテンプレートマッチング方式を廃し、Transformerベースのモデルが文書構造を動的に解釈。縦書き・横書き混在、表組みの崩れ、印影のかぶりがあっても、フィールドマッピング精度98.2%を達成。新規帳票形式への適応はゼロ設定で可能。

2. **リアルタイムバリデーション＆アラートエンジン**
   抽出後のデータに対し、税計算ロジック・日付整合性・金額合計のクロスチェックを自動実行。異常値（例：消費税率不一致、請求書番号の重複）を検出した場合、Webhook経由で即座に通知。SlackやTeamsと連携し、承認ワークフローをトリガー可能。

3. **マルチテナントカスタムスキーマ**
   テナントごとに独自のフィールド定義（例：部門コード、プロジェクトID、承認者リスト）をJSON Schemaで記述。APIリクエスト時にスキーマIDを指定するだけで、抽出結果を自動マッピング。データ出力形式はJSON/CSV/XMLから選択可能。

### 使い方

1. **APIキー発行とテナント設定**
   `POST /api/v1/tenants` にテナント名とスキーマ定義をJSONで送信。レスポンスとして `tenant_id` と `api_key` が返却される。スキーマには必須フィールド（例：`invoice_number`）とバリデーションルール（例：`total_amount > 0`）を定義。Curl例:
   ```bash
   curl -X POST https://api.billripper.io/v1/tenants \
     -H "Content-Type: application/json" \
     -d '{"name": "YourCompany", "schema": {"fields": ["invoice_number", "total_amount", "due_date"], "validations": [{"field": "total_amount", "rule": "gt:0"}]}}'
   ```

2. **請求書アップロードと解析実行**
   エンドポイント `POST /api/v1/invoices` にマルチパートフォームでPDF/画像ファイルを送信。`tenant_id` と `api_key` をヘッダーに含める。レスポンスは即時同期モード（小規模ファイル向け）と非同期モード（高解像度スキャン向け）を選択可能。非同期モードでは `job_id` が返却され、`GET /api/v1/jobs/{job_id}` でステータスをポーリング。成功時には抽出済みJSONが返る。

3. **結果の後処理とエクスポート**
   抽出結果は `GET /api/v1/invoices/{invoice_id}/export?format=csv` でダウンロード。また、Webhook URLを事前登録しておけば、解析完了時に自動POST。例えば、`POST /api/v1/webhooks` でエンドポイントとイベントタイプ（例：`invoice.parsed`）を指定。結果データには `confidence_scores` フィールドが含まれ、低信頼度フィールドは手動レビュー用にフラグ付け可能。

### ベストプラクティス

1. **スキーマ設計は「最小限＋拡張」で**
   最初は必須フィールド（請求番号、金額、日付）のみをスキーマに定義。運用開始後に不足フィールドを追加する。過剰なバリデーションルールは解析失敗率を上げるため、初期は `allow_null: true` で緩めに設定。データ品質が安定した段階で厳格化する。

2. **非同期モードとWebhookの組み合わせ**
   大量バッチ処理（例：月次1000件以上）では同期モードはタイムアウトリスクが高い。非同期モードでジョブを投入し、Webhookで完了通知を受け取る設計がスケーラブル。Webhookのリトライ機構（指数バックオフ）を必ず実装し、ネットワーク障害に備える。
