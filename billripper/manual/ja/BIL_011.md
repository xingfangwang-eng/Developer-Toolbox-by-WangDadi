# billripper - BIL_011

## 概要

billripper は、インボイス・請求書データを機械可読な構造に瞬時に変換する、次世代型請求書解析エンジンです。従来のOCRツールとは一線を画し、非構造化PDFやスキャン画像からキーバリューペアを高精度で抽出。経理・監査・ERP連携の前処理を劇的に効率化します。

## billripper とは？

billripper は、請求書・領収書・発注書に特化したドメイン特化型パーサーです。単なるテキスト抽出に留まらず、日付、金額、税区分、取引先情報などを自動分類。SaaSプロダクトに組み込むためのRESTful APIと、CLIツールの両方を提供します。特に、日本語帳票特有の縦書き・罫線混在・複数フォーマットに強く、学習不要で即座に構造化データを生成します。

## コア機能（3つのハードコアポイント）

### 1. 適応型テンプレートレスパーシング
固定テンプレートや学習データを一切必要とせず、レイアウト解析と意味論的推論により、あらゆる請求書フォーマットからフィールドを動的に特定。従来のルールベースエンジンでは対応困難だった、項目名の表記揺れ（例：「小計」「小計金額」「subtotal」）や配置パターンを自動補正します。

### 2. マルチエンティティ・トランザクションリンク
1枚の請求書に複数の取引先やプロジェクトコードが混在するケース（合算請求）を、内部相関グラフを用いて自動分離。各明細行を正しいエンティティに紐付け、会計システムへの投入前に整合性チェックを実施します。

### 3. タイムスタンプ付き監査トレイル
全解析プロセスをJSON Lines形式で逐次記録。どのフィールドをどのアルゴリズムで抽出したか、Confidenceスコア（0.0〜1.0）とともに保持。後から手動補正があった場合も、オリジナルデータと変更履歴を完全追跡可能にし、内部監査やSOX対応を強力に支援します。

## 使い方（詳細3ステップ）

### ステップ1: バッチインポートと前処理設定
`billripper import` コマンドで、対象PDFまたは画像ファイル群をディレクトリごと投入。`--preprocess` フラグで歪み補正やノイズ除去レベル（low/medium/high）を指定可能。出力先はデフォルトで `./ripped/` に設定され、各ファイルにユニークなジョブIDが自動割り当てられます。

```bash
billripper import ./invoices/ --preprocess medium --output ./ripped/
```

### ステップ2: カスタムフィールドマッピングの定義
抽出したいフィールドをYAMLで定義。標準フィールド（`amount`, `date`, `tax`）に加え、プロジェクトコードや発注番号など独自フィールドを追加可能。`--map` オプションでマッピングファイルを指定しない場合、完全自動モードで動作します。

```yaml
fields:
  - name: project_code
    alias: ["プロジェクトコード", "PJ No", "Project#"]
    type: string
    required: false
  - name: invoice_number
    alias: ["請求書番号", "Invoice No"]
    type: string
    required: true
```

### ステップ3: 解析実行と結果検証
`billripper parse` で解析を実行。`--validate` フラグを付与すると、抽出データの整合性チェック（金額の合計一致、日付フォーマット検証など）を自動実行。結果は構造化JSONとして出力され、`--format csv` でCSV出力も可能。エラーが発生したレコードは `./ripped/errors/` に分離保存されます。

```bash
billripper parse --job-id <JOB_ID> --validate --format json
```

## ベストプラクティス（業界推奨）

### 1. 定期的なConfidenceスコア監視と閾値調整
本番運用では、全解析結果のConfidenceスコアを週次で集計し、0.8未満のレコードを重点的にレビューすることを推奨。新規フォーマットが混入した場合、スコア分布が急変するため、早期に閾値調整やマッピング更新を行うことで、精度低下を未然に防げます。特に、月次・四半期決算期はフォーマット変更が発生しやすいため、事前にテストセットを更新しておくと安全です。

### 2. エラーレコードのフィードバックループ構築
`./ripped/errors/` に格納された解析失敗レコードは、単に破棄せず、手動補正後に再投入するフローを設計してください。billripper は、過去の補正パターンを内部ヒューリスティックに学習（オプトイン機能）するため、補正データを一定量蓄積することで、同系統フォーマットのエラー率が劇的に低下します。理想的には、週1回のバッチ処理でエラーレコードを再解析するサイクルを組むと、長期的な運用コストが最小化します。
