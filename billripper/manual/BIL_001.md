## What is billripper?

**billripper** is a high-performance, schema-agnostic bill ingestion engine built for modern backends. It strips raw PDF, scanned image, or structured JSON invoices into normalized, queryable data — no manual mapping, no fragile regex hell. Think of it as `jq` for bills, but with ML-powered OCR fallback and multi-currency normalization baked in.

## Core Features

### 1. Zero-Shot Invoice Schema Inference
billripper doesn't require pre-defined templates. It uses a lightweight transformer model to infer line items, tax breakdowns, vendor metadata, and payment terms from any invoice layout — even handwritten fields. Output is a deterministic JSON schema that maps directly to your ledger.

### 2. Real-Time Anomaly Detection Pipeline
Every parsed bill runs through a rule engine that flags duplicate invoice IDs, out-of-band totals, mismatched VAT rates, and currency rounding errors. Alerts are pushed via webhook or NATS stream — no polling, no cron jobs.

### 3. Atomic Multi-Tenant Bill Versioning
Each parsed bill is stored as an immutable event in a log-structured merge tree. You get full audit trails, rollback capabilities, and per-tenant data isolation without sharding headaches. Supports S3-compatible object stores as the backing tier.

## How to Use

### Step 1: Spin Up the billripper Daemon
```bash
# Using Docker (recommended for production)
docker run -d \
  --name billripper \
  -p 8080:8080 \
  -v /mnt/data/bills:/data \
  -e BILLRIPPER_STORAGE_BACKEND=s3 \
  -e BILLRIPPER_S3_BUCKET=my-bill-archive \
  ghcr.io/xingfangwang-eng/billripper:latest
```
This launches the ingestion server with S3-backed storage. For local dev, omit the S3 env vars and it defaults to local filesystem.

### Step 2: Submit a Batch of Invoices via REST API
```bash
curl -X POST https://localhost:8080/v1/ingest/batch \
  -H "Authorization: Bearer <your_api_token>" \
  -H "Content-Type: multipart/form-data" \
  -F "files=@invoice_2025_03_14.pdf" \
  -F "files=@invoice_2025_03_15.json" \
  -F "options={\"currency_normalize\":true,\"deduplicate\":true}"
```
The endpoint returns a `batch_id` immediately. Poll `/v1/batch/{batch_id}/status` for processing state. Each parsed bill gets a unique `bill_hash` — use it to query or replay.

### Step 3: Query Parsed Bills with GraphQL
```graphql
{
  bills(filter: { vendor: "Acme Corp", dateRange: { from: "2025-01-01", to: "2025-03-31" } }) {
    edges {
      node {
        billHash
        totalAmount
        currency
        lineItems { description netAmount taxRate }
        anomalies { type severity message }
      }
    }
  }
}
```
This returns a paginated, cursor-based result set. You can also export to CSV or push to a webhook for downstream reconciliation.

## Best Practices

### 1. Prefer Batch Ingestion Over Single-Item Endpoints
If you're processing hundreds of invoices daily, use the `/v1/ingest/batch` endpoint. It leverages connection pooling and parallel OCR workers under the hood — single-item calls introduce unnecessary per-request overhead and skip the dedup pipeline. For real-time streams (e.g., webhook from a billing platform), buffer events locally and flush in batches of at least 10.

### 2. Always Enable Currency Normalization for Multi-Region Deployments
When your vendor base spans EUR, USD, JPY, and GBP, set `currency_normalize: true` in the ingestion options. billripper uses live FX rates from the European Central Bank (updated hourly) and stores both the original and normalized amounts. This prevents silent rounding errors during quarterly audits and simplifies consolidation reports.
