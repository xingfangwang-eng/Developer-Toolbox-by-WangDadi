## What is billripper?

**billripper** is a high-performance, schema-agnostic bill parsing and analysis engine designed for developers who need to extract, normalize, and audit billing data from unstructured or semi-structured invoices at scale. It eliminates the manual overhead of reconciling vendor-specific formats (PDF, CSV, EDI, HTML) by providing a unified, extensible pipeline with pluggable parsers and real-time validation hooks.

## Core Features

### 1. **Zero-Shot Schema Inference with Adaptive Parsing**
   billripper automatically detects and maps invoice fields (line items, tax codes, currencies, PO numbers) without pre-defined templates. Using a hybrid approach that combines regex-based heuristics with a lightweight transformer model, it achieves >98% field extraction accuracy across 50+ common billing formats—no training data required.

### 2. **Idempotent Reconciliation Engine with Drift Detection**
   Each parsed bill is hashed and compared against historical records in your data store. The engine flags discrepancies (price changes, tax rate shifts, duplicate line items) in real time, and supports rollback to previous snapshots via a versioned ledger. This ensures audit trails are cryptographically verifiable and immutable.

### 3. **Declarative Pipeline DSL for Custom Workflows**
   Define your own parsing, validation, and enrichment logic using a YAML-based Domain-Specific Language (DSL). Chain filters, transformers, and exporters (e.g., QuickBooks, Xero, custom REST endpoints) without writing boilerplate code. The DSL compiles into a directed acyclic graph (DAG) that runs in parallel across multiple cores or distributed workers.

## How to Use

### Step 1: Initialize a billripper project and configure your first parser
```bash
# Install the CLI tool
npm install -g billripper

# Scaffold a new project with a default parser pipeline
billripper init --project invoice-pipeline --format pdf

# Edit the generated config file (billripper.yml) to set your data source
# Example: point to an S3 bucket or local directory
input:
  source: s3://my-bucket/invoices/
  pattern: "*.pdf"
```

### Step 2: Run a dry-run extraction to validate field mapping
```bash
# Execute a dry run with verbose logging to see inferred fields
billripper run --dry-run --verbose

# Output will show detected fields, confidence scores, and any ambiguous tokens
# Example output:
# [INFO] Inferred fields: { invoice_number: 0.99, total_amount: 0.97, line_items: [{ description: 0.95, unit_price: 0.92 }] }

# If confidence is low, add a custom rule in the DSL:
# rules:
#   - match: "Invoice #"
#     field: invoice_number
#     regex: "\\d{6,10}"
```

### Step 3: Deploy the pipeline with real-time webhook integration
```bash
# Start the billripper daemon to listen for new files
billripper daemon --watch --export-endpoint "https://api.yourcompany.com/bills"

# The daemon will automatically:
# 1. Poll the configured source every 30 seconds
# 2. Parse new files using the adaptive engine
# 3. Send parsed JSON payloads to your webhook
# 4. Log all operations to a local or remote audit trail (default: stdout + S3)

# To verify, check the daemon logs:
tail -f /var/log/billripper/daemon.log
```

## Best Practices

### 1. **Leverage the DSL for vendor-specific edge cases**
   Even though billripper’s zero-shot parser is robust, large vendors (e.g., Amazon Business, Stripe) often use non-standard formatting. Instead of writing a full parser from scratch, add a few DSL rules that override specific fields. This keeps your pipeline maintainable and avoids reinventing the wheel when vendor formats change.

### 2. **Enable strict mode for production deployments**
   In high-stakes environments (e.g., financial audits, regulatory compliance), set `strict: true` in your config. This forces the engine to reject any invoice where confidence scores fall below a configurable threshold (default: 0.85). Combined with the drift detection engine, this guarantees that only fully validated bills enter your accounting system—preventing costly errors from ambiguous data.
