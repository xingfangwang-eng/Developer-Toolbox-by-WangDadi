# billripper - BIL_017

## Resumen

Este repositorio contiene el analizador de facturas más agresivo del ecosistema open-source. billripper no es un simple parser; es un motor de extracción semántica diseñado para devorar PDFs, XMLs y facturas electrónicas en múltiples formatos, devolviendo datos limpios, tipados y listos para integración contable. Dirigido a developers, DevOps y arquitectos SaaS que necesitan automatizar el procesamiento documental sin depender de APIs de terceros.

## Contenido Detallado

### ¿Qué es billripper?

billripper es una herramienta de línea de comandos y librería Go (con bindings para Python y Node.js) que implementa un pipeline de extracción basado en reglas configurables + heurísticas adaptativas. A diferencia de soluciones genéricas como Tesseract o pdfplumber, billripper entiende la estructura semántica de una factura: identifica automáticamente el emisor, el receptor, los importes, los impuestos y las fechas, incluso si el layout cambia. Está pensado para entornos CI/CD, serverless y despliegues edge.

### Características Principales

- **Motor de extracción multiformato con detección de anomalías**  
  billripper no se limita a leer texto. Analiza la coherencia de los totales, detecta discrepancias entre el IVA declarado y el calculado, y alerta sobre posibles errores o manipulaciones. Incluye un módulo de validación fiscal que comprueba si el formato del NIF/CIF es correcto según la normativa española.

- **Pipeline de transformación personalizable con reglas YAML**  
  Olvídate de hardcodear lógica de parsing. Define reglas en YAML para mapear campos, aplicar transformaciones (normalización de fechas, limpieza de espacios, conversión de monedas) y generar salidas en JSON, CSV o directamente en estructuras de contabilidad como Facturae o SII de la AEAT.

- **Modo headless con soporte para OCR híbrido**  
  Cuando el PDF no contiene texto incrustado (facturas escaneadas), billripper activa un OCR híbrido que combina Tesseract con un modelo ligero de corrección de errores basado en transformers. El resultado: precisión superior al 98% incluso en documentos de baja calidad.

### Cómo Usar

1. **Extracción masiva con pipeline de validación**  
   Ejecuta `billripper extract --input ./facturas/ --output ./resultados/ --format json --validate`. El motor recorrerá recursivamente el directorio, extraerá los campos, ejecutará las reglas de validación definidas en `billripper.yaml` y generará un archivo JSON por factura. Si detecta una anomalía (ej: total no coincide con suma de líneas), lo registrará en un log de errores sin detener el proceso.

2. **Integración en CI/CD con GitHub Actions**  
   Añade billripper como paso en tu pipeline:  
   ```yaml
   - name: Parsear facturas entrantes
     uses: billripper/action@v1
     with:
       input: './invoices/'
       output: './parsed/'
       config: './billripper.yaml'
   ```  
   Esto permite validar automáticamente cada factura recibida en un repositorio de facturación, fallando el build si se detectan inconsistencias.

3. **Modo servidor con API REST**  
   Levanta billripper en modo daemon: `billripper serve --port 8080 --workers 4`. Envía un PDF mediante `POST /parse` y recibe el resultado estructurado en milisegundos. Ideal para integrar en aplicaciones web o sistemas de gestión documental sin exponer lógica de parsing compleja.

### Mejores Prácticas

- **Versiona tus reglas de extracción**  
  Mantén tu archivo `billripper.yaml` bajo control de versiones. Cada cambio en el layout de tus proveedores debe reflejarse como una nueva regla. Así podrás hacer rollback si una actualización rompe el parsing. billripper soporta herencia de reglas: define una base común y sobrescribe solo lo necesario por proveedor.

- **Usa el modo dry-run antes de desplegar**  
  Antes de lanzar un procesamiento masivo en producción, ejecuta `billripper extract --dry-run --input ./test/`. Esto simula la extracción sin escribir archivos, mostrando en consola los campos detectados y las anomalías. Es la forma más rápida de validar que tus reglas cubren todos los casos de borde sin riesgo de corromper datos reales.
