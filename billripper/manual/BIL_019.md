# billripper - BIL_019

## Overview

billripper is a high-performance, multi-format invoice data extraction and reconciliation engine designed for developers and finance ops teams who need to rip through hundreds of PDF, scanned image, or e-invoice files without breaking a sweat. It’s not just another parser—it’s a deterministic pipeline that transforms messy billing artifacts into structured, queryable JSON.

## Hardcore Features

### 1. Zero-Shot Schema Inference Engine
Unlike traditional template-based parsers, billripper employs a hybrid **OCR + NLP + regex-free schema inference** layer. It dynamically detects invoice layouts (from Japanese *請求書* to German *Rechnung*), identifies line items, tax breakdowns, and vendor metadata without any pre-training or manual field mapping. You feed it raw files; it returns a normalized schema.

### 2. Multi-Stream Reconciliation Diff
This isn’t just extraction—it’s forensic-level matching. billripper can ingest two or more invoice streams (e.g., vendor PDF vs. internal PO system CSV) and generate a **structural diff** at the line-item level. It flags discrepancies in unit prices, quantities, tax rates, and even currency conversions, outputting a machine-readable diff report that can be piped directly into your ERP webhook.

### 3. Native CLI & gRPC Streaming Pipeline
Forget bloated UIs. billripper ships as a **single binary** with a zero-dependency CLI. It supports real-time streaming via gRPC for high-throughput batch processing. You can pipe invoices through stdin, get structured results on stdout, or stream them into a Kafka topic. No database required—just a Unix pipe.

## How to Use

### Step 1: Batch Extract with Schema Normalization
```bash
billripper extract --input ./invoices/ --output ./extracted/ --format json --normalize
```
This command scans all PDFs and images in the `invoices` directory, runs the zero-shot inference engine, and outputs a unified JSON array. Each object contains `vendor`, `invoice_number`, `line_items[]`, `tax_summary`, and `confidence_score`. Use `--normalize` to enforce ISO 4217 currency codes and ISO 8601 date formats.

### Step 2: Run a Two-Way Reconciliation Diff
```bash
billripper reconcile --source ./extracted/vendors.json --target ./purchase_orders.csv --output ./diff_report.json --tolerance 0.02
```
This compares the extracted vendor data against your internal PO CSV. The `--tolerance 0.02` flag allows a 2% variance in unit prices (e.g., due to rounding). The output is a structured diff with `type: "MISMATCH"`, `field: "unit_price"`, `expected: 12.50`, `actual: 12.75`, and `confidence: 0.89`.

### Step 3: Stream Live Invoices via gRPC
```python
import grpc
import billripper_pb2 as pb2

channel = grpc.insecure_channel('localhost:50051')
stub = pb2.InvoiceParserStub(channel)
response = stub.Parse(pb2.ParseRequest(raw_pdf=open('invoice.pdf', 'rb').read()))
print(response.normalized_invoice)
```
Spin up the gRPC server with `billripper server --port 50051`. This is ideal for CI/CD pipelines or serverless functions where you need per-invoice latency under 200ms.

## Industry Best Practices

### 1. Always Pre-Filter with `--confidence` Thresholds in Production
When running billripper in an automated AP workflow, set a minimum confidence score (e.g., `--min-confidence 0.85`) during the extract phase. Any invoice falling below this threshold should be flagged for manual review, not automatically posted. This prevents silent corruption of your financial data from low-quality scans or unusual layouts.

### 2. Use the Reconciliation Diff as a Pre-Commit Hook
Integrate `billripper reconcile` into your GitOps or CI pipeline. Before merging any changes to your vendor master data or PO terms, run a reconciliation against a historical batch of invoices. This catches regressions in your procurement logic and ensures that any schema changes in your ERP don’t silently break invoice matching.
