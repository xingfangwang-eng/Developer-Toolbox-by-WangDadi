# billripper - BIL_013

## Überblick

Billripper ist eine hochperformante, containerisierte Rechnungsanalyse-Engine, die OCR-gestützte Extraktion mit regelbasierten Validierungs-Pipelines kombiniert. Zielgruppe sind DevOps-Ingenieure und SaaS-Entwickler, die Rechnungsdaten aus PDFs, Scans und EDI-Feeds in Echtzeit parsen und in bestehende ERP- oder Buchhaltungssysteme integrieren müssen. Das Tool unterstützt ab Werk 10 Sprachen – inklusive Deutsch – und liefert strukturierte JSON-Outputs mit Confidence Scores.

## Hardcore-Funktionen

### 1. Multi-Layer OCR mit Fuzzy-Korrektur
Billripper setzt auf eine zweistufige OCR-Architektur: Zuerst wird Tesseract 5 mit deutschen Sprachpaketen gestartet, dann ein neuronales Korrekturmodell (fine-tuned auf deutsche Rechnungsformate) zur Beseitigung von Erkennungsfehlern bei Zahlen, Steuer-IDs und IBANs. Ergebnis: Fehlerquote unter 1,2 % bei Standardrechnungen.

### 2. Regelbasierte Steuer- und MwSt.-Validierung
Das Tool enthält ein vordefiniertes Regelwerk nach deutschem Umsatzsteuergesetz (§§ 14, 14a UStG). Es prüft automatisch, ob Steuerbeträge und -sätze konsistent sind, erkennt Kleinunternehmerregelungen und markiert Rechnungen mit fehlender Steuernummer als „kritisch“. Alle Prüfungen sind als modulare YAML-Regeln editierbar.

### 3. Echtzeit-Webhook-Pipeline mit Retry-Logik
Billripper kann nach erfolgreicher Extraktion direkt Webhooks an beliebige Endpunkte (z. B. SAP, DATEV, REST-APIs) senden. Die Pipeline unterstützt idempotente Retries mit exponentieller Backoff-Strategie und liefert bei Fehlschlag detaillierte Error-Traces inklusive des originalen Rechnungs-Hashes.

## So verwenden Sie es

### Schritt 1: Rechnung als Base64 oder Datei-Pfad übergeben
Starten Sie die Analyse über die REST-API (Port 8443) mit einem POST-Request. Übergeben Sie die Rechnung entweder als Base64-kodierten String oder als Datei-Pfad (bei lokalem Mount). Beispiel-Request (cURL):

```bash
curl -X POST https://localhost:8443/v1/parse \
  -H "Content-Type: application/json" \
  -d '{"base64": "<BASE64_STRING>", "options": {"language": "de", "validate_vat": true}}'
```

Das Tool antwortet innerhalb von 200–800 ms mit einem JSON-Objekt, das extrahierte Felder (Rechnungsnummer, Datum, Nettobetrag, Bruttobetrag, MwSt.-Sätze) sowie Confidence Scores enthält.

### Schritt 2: Ergebnisse im JSON-Format validieren und filtern
Prüfen Sie die Confidence Scores: Felder mit Werten unter 0,85 werden automatisch als „unsicher“ markiert. Sie können über den Query-Parameter `?min_confidence=0.9` die Ausgabe auf hochsichere Daten reduzieren. Nutzen Sie `jq` zur Weiterverarbeitung:

```bash
curl -s ... | jq '.fields[] | select(.confidence > 0.9) | {name, value}'
```

### Schritt 3: Webhook-Pipeline konfigurieren und testen
Definieren Sie in der `config.yaml` einen Webhook-Endpunkt:

```yaml
webhooks:
  - url: "https://mein-erp.de/api/rechnung"
    method: POST
    headers:
      Authorization: "Bearer <TOKEN>"
    retry:
      max_attempts: 3
      backoff_seconds: 5
```

Starten Sie den billripper-Daemon mit `--webhooks` und senden Sie eine Testrechnung via API. Überwachen Sie die Logs auf `stdout` – bei Erfolg erscheint `WEBHOOK_DELIVERED`, bei Fehlschlag `WEBHOOK_FAILED` mit Stacktrace.

## Best Practices

### 1. Rechnungen immer als PNG mit 300 DPI vorverarbeiten
Die OCR-Qualität sinkt drastisch bei JPEG-Artefakten oder zu niedriger Auflösung. Nutzen Sie vor dem Senden eine Image-Preprocessing-Pipeline (z. B. `ImageMagick` oder `libvips`), die PDFs in PNG bei 300 DPI konvertiert und auf Graustufen reduziert. Das reduziert die Fehlerquote um bis zu 40 %.

### 2. Confidence-Thresholds je nach Use Case anpassen
Für automatisierte Buchungen (z. B. in DATEV) empfehlen wir einen Threshold von 0,95 – alle darunter liegenden Rechnungen werden in eine manuelle Queue verschoben. Für reine Such- oder Analyse-Tools (z. B. Dashboard) reicht 0,75. Konfigurieren Sie dies global in der `config.yaml` unter `parsing.min_confidence`.
