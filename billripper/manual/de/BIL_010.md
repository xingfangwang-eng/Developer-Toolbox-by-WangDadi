# billripper - BIL_010

## Überblick

billripper ist eine **hochperformante Rechnungs-Parsing-Engine** für Entwickler, die aus PDFs, Scans und E‑Mails strukturierte Daten extrahiert – ohne Cloud-Abhängigkeit. Das Tool richtet sich an DevOps‑Teams, SaaS‑Gründer und Fintech‑Entwickler, die Rechnungsdaten in Echtzeit in ihre Systeme integrieren müssen. Unterstützt werden OCR, mehrsprachige Layouts (DE/EN/FR) und Batch‑Processing.

## Hardcore-Funktionen

### 1. **Zero‑Shot Layout‑Erkennung mit Transformer‑Backend**
billripper analysiert Rechnungslayouts ohne manuelle Konfiguration. Ein vortrainiertes Modell (basierend auf LayoutLMv3) erkennt Felder wie Rechnungsnummer, Steuersätze und Zahlungsfristen – selbst bei gescannten, schiefen oder handschriftlich ergänzten Belegen. Die Inferenz läuft lokal auf CPU/GPU, keine API-Kosten.

### 2. **Batch‑Pipeline für 500+ Rechnungen/Minute**
Über eine CLI oder REST‑API verarbeitest du ganze Ordner mit Rechnungen parallel. Die Engine nutzt Multithreading und einen internen Cache für wiederkehrende Layouts. Ausgabe erfolgt als JSON, CSV oder direkt in eine PostgreSQL‑Datenbank – ideal für monatliche Abrechnungen oder Audit‑Workflows.

### 3. **Fehlerkorrektur‑Engine mit Regex‑Playbook**
Nach der Extraktion prüft ein Regel‑System die Daten auf Plausibilität: Summen stimmen? Steuer‑ID gültig? Datum nicht in der Zukunft? Fehler werden markiert und können über ein YAML‑Playbook automatisch korrigiert oder zur manuellen Prüfung exportiert werden.

## So verwendest du billripper

### Schritt 1: Installation und initiale Konfiguration
```bash
pip install billripper
billripper init --lang de --output-format json
```
Erstelle eine `config.yml` im Projektverzeichnis. Lege dort Fallback‑OCR‑Engine (Tesseract oder EasyOCR) und den Pfad für Batch‑Verarbeitung fest. Beispiel:
```yaml
ocr_engine: tesseract
batch_dir: ./invoices/
output_dir: ./parsed/
validation:
  check_tax_id: true
  check_sum: true
```

### Schritt 2: Batch‑Verarbeitung starten und Logs überwachen
```bash
billripper run --watch --verbose
```
Das Tool überwacht `batch_dir` auf neue PDFs. Jede Rechnung wird innerhalb von 200 ms (CPU) analysiert. Logs zeigen pro Datei die extrahierten Felder und eventuelle Validierungswarnungen. Mit `--dry-run` testest du ohne Ausgabe.

### Schritt 3: Ergebnisse in dein System integrieren
Die ausgegebenen JSON‑Dateien enthalten:
```json
{
  "invoice_id": "RE-2024-1234",
  "total_net": 1190.00,
  "tax_rate": 19.0,
  "due_date": "2025-02-28",
  "confidence": 0.97
}
```
Importiere sie via `curl` in deine API oder lade sie mit einem SQL‑Skript in deine Datenbank. Für CI/CD‑Pipelines gibt es einen nativen GitHub‑Action‑Wrapper.

## Best Practices

### 1. **Trenne OCR von der Extraktion (bei hohem Volumen)**
Führe OCR nur einmal durch und speichere die Rohtexte als Cache. Bei Layout‑Updates musst du nicht neu scannen. Nutze `billripper cache --rebuild` nur nach Modell‑Updates.

### 2. **Definiere ein Fallback‑Playbook für Grenzfälle**
Nicht alle Rechnungen sind perfekt. Lege in `config.yml` ein `fallback_playbook` an, das bei niedrigem Confidence‑Score automatisch auf manuelle Prüfung umschaltet. Beispiel: Bei `confidence < 0.8` verschiebe die Datei in einen `review/`‑Ordner und sende eine Slack‑Benachrichtigung.
