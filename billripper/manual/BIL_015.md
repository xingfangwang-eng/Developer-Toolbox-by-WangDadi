# billripper - BIL_015

## Overview

billripper is not just another bill parser—it's a high-performance, multi-format document ripper designed for developers who need to extract, normalize, and analyze invoice data at scale. Built for SaaS platforms, fintech pipelines, and enterprise accounting systems, it strips metadata, line items, and tax structures from PDFs, images, and raw text with sub-second latency. No bloat. No UI. Just raw, schema-agnostic extraction.

## What is billripper?

billripper is a CLI-first, modular bill analysis engine that ingests heterogeneous billing documents and outputs structured JSON or CSV. It leverages a hybrid approach: regex-based pattern matching for legacy formats, plus a lightweight transformer model for semi-structured layouts. Think of it as `jq` for invoices—except it handles rotated PDFs, scanned receipts, and multi-currency tax breakdowns out of the box.

## Core Features

### 1. Adaptive Schema Inference & Normalization
billripper automatically detects vendor-specific layouts (e.g., Stripe, QuickBooks, Xero, custom ERP exports) and maps fields to a unified schema. No manual mapping. It uses a combination of positional heuristics and a small BERT-based classifier to handle ambiguous fields like "Total Due" vs. "Amount Payable." The output adheres to a strict JSON schema you can validate with `ajv` or `zod`.

### 2. Multi-Format Stream Processing
Process 10,000+ documents per minute via streaming pipeline. Supports PDF (text layer + OCR fallback via Tesseract), PNG/JPG (with deskew and binarization), CSV, and raw text. Each document is processed in isolated workers—no memory leaks, no cross-contamination. For batch jobs, billripper outputs a single NDJSON stream, ready for ingestion into BigQuery, Snowflake, or your own S3 bucket.

### 3. Tax & Currency Override Engine
Not all bills are created equal. billripper includes a declarative tax rule engine (YAML-based) that lets you override VAT, GST, or sales tax logic per vendor or jurisdiction. It also handles multi-currency conversions via a built-in exchange rate provider (or your own API). Perfect for SaaS companies dealing with international subscriptions where tax codes change quarterly.

## How to Use

### 1. Extract and Normalize a Batch of Invoices
```bash
# Point billripper to a directory of PDFs and output structured JSON
billripper extract ./invoices/ --format json --schema unified --output ./normalized/
```
This command scans `./invoices/` for all `.pdf` and `.png` files, applies OCR if needed, and writes one `.json` per document. Use `--schema unified` to enforce the billripper standard schema. For debugging, add `--verbose` to see per-document confidence scores.

### 2. Override Tax Logic for a Specific Vendor
```yaml
# Create a custom tax rule file: ./rules/vendor_acme.yaml
vendor: "Acme Corp"
tax_rules:
  - country: "DE"
    type: "VAT"
    rate: 0.19
    apply_to: ["subtotal", "shipping"]
  - country: "US"
    type: "sales_tax"
    rate: 0.0825
    apply_to: ["subtotal"]
```
Then run:
```bash
billripper extract ./acme_invoices/ --tax-rules ./rules/vendor_acme.yaml
```
The engine will override any detected tax fields for Acme Corp documents. This is critical when vendors embed tax in line items vs. separate fields.

### 3. Stream Invoices Directly into a Data Pipeline
```bash
# Pipe NDJSON output directly into a Kafka topic or a local file
billripper extract ./batch/ --format ndjson --stream | kafkacat -P -b localhost:9092 -t invoices
```
For production, use `--stream` to avoid buffering all results in memory. Each line is a validated document. Combine with `jq` for real-time filtering:  
```bash
billripper extract ./batch/ --format ndjson --stream | jq 'select(.total > 1000)'
```

## Best Practices

### 1. Always Pre-validate Your Input Documents
Before running a large batch, use `billripper validate ./input/` to check for corrupt PDFs, zero-byte files, or unsupported encodings. This saves hours of debugging. The validator outputs a report with file names, error types, and suggested fixes (e.g., "re-export as PDF/A-1b").

### 2. Use a Cached Exchange Rate Provider for Multi-Currency Pipelines
If your pipeline processes invoices in USD, EUR, and JPY, never fetch live rates per document. Instead, configure a daily cache:  
```yaml
# config.yaml
currency:
  provider: "openexchangerates"
  cache_ttl: 86400
  base_currency: "USD"
```
This reduces API calls by 99% and ensures consistent conversion across all documents in a single batch. billripper will log a warning if the cache expires mid-batch.
