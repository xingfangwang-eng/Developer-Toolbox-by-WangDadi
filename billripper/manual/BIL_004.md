## Detailed Content

### What is billripper?

**billripper** is a high-performance, developer-first bill ingestion and analysis engine. It’s built for teams that need to rip through thousands of invoices, receipts, or billing PDFs per minute — extracting structured data, detecting anomalies, and feeding clean output directly into your ERP or analytics pipeline. No fluff. No manual data entry.

### Core Features

1. **Multi-Format Ripping Engine**  
   Parses PDF, scanned TIFF, HTML invoices, and raw email attachments — with zero dependency on OCR for digital-native files. Uses a hybrid regex + ML classifier to extract line items, tax breakdowns, and vendor metadata at sub-100ms latency per document.

2. **Schema-Agnostic Output with Auto-Mapping**  
   Outputs JSON, CSV, or direct SQL inserts. The engine auto-detects common billing schemas (X12, EDIFACT, custom CSV headers) and maps fields to your existing data model — no manual template config required.

3. **Real-Time Anomaly Detection & Flagging**  
   Built-in statistical outlier detection flags duplicate invoices, price discrepancies >5% from historical averages, and missing mandatory fields (e.g., VAT ID, PO number). Each flag includes a confidence score and raw diff trace.

### How to Use

1. **Ingest a Batch of Invoices via CLI**  
   Run `billripper rip --input ./invoices/ --format pdf --output ./parsed/ --schema custom`  
   The tool auto-detects file types, spins up worker threads (configurable via `--workers`), and writes structured JSON per file. Use `--dry-run` to validate before bulk processing.

2. **Validate and Override Auto-Mapping**  
   After ingestion, run `billripper validate --input ./parsed/ --rules ./custom_rules.yaml`  
   This step checks field completeness, runs your custom business rules (e.g., “tax must be > 0”), and generates a diff report. Override mappings with `--map vendor_name:supplier` if the auto-detect misaligns.

3. **Push Clean Data to Your ERP**  
   Execute `billripper push --input ./parsed/validated/ --target postgres --connection "postgresql://..."`  
   The tool bulk-inserts all validated records, logs failures to `./push_errors.log`, and prints a summary of inserted vs. skipped rows. Use `--batch-size 500` for large datasets.

### Best Practices

- **Always run `validate` before `push` in production pipelines.**  
  Even with 99% accuracy on auto-mapping, a single misaligned field (e.g., date format) can corrupt your GL. Validate against a staging table first, or use `--dry-run` to catch schema drift.

- **Version your custom rules and schemas in git.**  
  Store your `custom_rules.yaml` and output schema definitions in the same repo as your ingestion scripts. This ensures repeatable parsing across environments and makes rollbacks trivial when vendor formats change.
