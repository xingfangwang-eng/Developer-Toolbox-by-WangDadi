# cleancsvai - CLE_005

## Overview

cleancsvai is not your grandfather's CSV cleaner. It's a **zero-config, AI-driven pipeline** that detects encoding anomalies, infers column types with statistical rigor, and normalizes malformed records in real time. Designed for data engineers, ML pipeline builders, and analysts who need production-grade CSV hygiene without manual mapping.

## Three Hardcore Differentiators

### 1. **Adaptive Schema Inference with Anomaly Scoring**
Unlike static parsers that fail on mixed-type columns or missing headers, cleancsvai runs a **two-pass entropy analyzer**. It assigns a confidence score (0.0–1.0) to every inferred data type—int, float, date, categorical, or raw text—and highlights columns where entropy exceeds a configurable threshold. You get a **schema heatmap** before a single row is cleaned.

### 2. **Multi-Encoding Detection & Transcoding Engine**
Raw CSVs from the wild often arrive in UTF-8 with BOM, ISO-8859-1, or even Shift-JIS. cleancsvai's **encoding probe** tries up to 12 codecs in parallel, uses byte-order mark detection, and falls back to a character frequency model. It then **transcodes everything to normalized UTF-8** (or your target encoding) with a full diff log of replaced characters.

### 3. **Row-Level Repair with Contextual Imputation**
When a row has a null or obviously wrong value (e.g., `"NaN"` in an integer column), cleancsvai doesn't just drop it. It applies **nearest-neighbor imputation** based on the column's distribution and adjacent rows. You can also define custom repair rules (e.g., "replace 'N/A' with column median") via a simple YAML config. Every repair is logged with the original and new value, plus the imputation method used.

## Three-Step Operational Workflow

### Step 1: **Feed & Profile**
```bash
cleancsvai profile ./dirty_data.csv --output-format json
```
The tool instantly generates a **data quality report**: column count, row count, null ratio, unique ratio, detected encoding, and a **schema confidence matrix**. If the CSV has inconsistent quoting or line breaks, it will warn you and suggest a pre-normalization pass.

### Step 2: **Define Repair Policies**
Create a `cleancsvai.yaml` in your project root:
```yaml
columns:
  - name: "age"
    type: "int"
    repair: "median_impute"
  - name: "email"
    type: "string"
    repair: "drop_if_invalid"
  - name: "price"
    type: "float"
    repair: "clip_outliers"  # removes values beyond 3 sigma
global:
  encoding: "utf-8"
  drop_duplicates: true
```
No YAML? No problem. Use the **interactive CLI wizard** (`cleancsvai wizard`) that asks you one column at a time.

### Step 3: **Execute & Audit**
```bash
cleancsvai clean ./dirty_data.csv --config cleancsvai.yaml --output ./clean_data.csv
```
The engine processes the file in **streaming mode** (no memory blowup for 10GB CSVs). It writes the cleaned output and generates an **audit trail** (`clean_audit.json`) containing:
- Total rows processed, repaired, dropped
- Per-column repair count
- Encoding conversion log
- Schema drift detection (if any column's inferred type changed mid-file)

## Two Industry Best Practices

### **1. Always Run a Pre-Clean Profile in CI/CD Pipelines**
Before merging any data ingestion code, run `cleancsvai profile` as a **CI check**. If the schema confidence drops below 0.8 or the null ratio exceeds 20%, fail the build. This catches upstream schema drift before it corrupts your ML training sets or dashboards.

### **2. Use the Audit Log for Data Lineage Documentation**
When you share cleaned CSVs with downstream teams, include the `clean_audit.json` file. It serves as a **provenance document**—exactly what changed, why, and by which rule. This is critical for SOC 2 compliance audits and for reproducing results months later.
