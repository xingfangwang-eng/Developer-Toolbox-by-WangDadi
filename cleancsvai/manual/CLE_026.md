# cleancsvai - CLE_026

## Overview

This is **not** your grandpa's CSV cleaner. cleancsvai is a production-grade data wrangling tool designed for developers and data engineers who deal with messy, inconsistent CSV files on a daily basis. It leverages LLM-based pattern recognition to infer column types, fix encoding issues, and deduplicate records without writing a single regex.

## Hardcore Features

### 1. Schema-Agnostic Anomaly Detection
cleancsvai doesn't just strip whitespace—it **learns** your data's statistical distribution. It automatically flags outliers (e.g., a `-999` in a column where 95% of values are positive integers) and suggests corrective actions. Under the hood, it runs a Kolmogorov-Smirnov test per column to detect deviations from expected distributions.

### 2. Multi-Encoding Auto-Detection with Fallback Chains
No more `UnicodeDecodeError` nightmares. cleancsvai probes `utf-8`, `latin-1`, `cp1252`, and `ISO-8859-1` in parallel, then picks the encoding that minimizes decoding failures. If ambiguity remains, it outputs a confidence matrix so you can make the final call.

### 3. Column Type Inference with Confidence Thresholds
Instead of guessing "int" or "float", cleancsvai assigns a **type probability vector** per column. For example, a column might be `[0.95 string, 0.04 datetime, 0.01 integer]`. You can set a confidence cutoff (default: 0.85) to auto-cast or flag ambiguous columns for human review.

## Detailed Usage Steps

### Step 1: Initialize the Cleaning Pipeline with Custom Rules
```bash
cleancsvai init --input raw_sales.csv --output cleaned_sales.csv --rules rules.yaml
```
The `rules.yaml` file can define column-specific transformations (e.g., `date_columns: [purchase_date, ship_date]` and `remove_duplicates: true`). If omitted, cleancsvai auto-generates a rules template based on first 100 rows.

### Step 2: Run Semantic Deduplication with Fuzzy Matching
```bash
cleancsvai dedup --input cleaned_sales.csv --threshold 0.92 --algorithm levenshtein
```
This step groups near-duplicate rows using Levenshtein distance on string columns, then merges them by keeping the row with the most non-null fields. The output includes a `_dedup_score` column so you can audit the merging logic.

### Step 3: Export a Type-Safe Parquet File with Column Annotations
```bash
cleancsvai export --input deduped_sales.csv --format parquet --schema sales_schema.yaml
```
cleancsvai writes a Parquet file with **embedded column metadata** (nullability, min/max, unique counts). The schema YAML can be version-controlled and reused across ETL runs, ensuring consistency between staging and production.

## Industry Best Practices

### 1. Always Run a Dry-Run Before Production
Use the `--dry-run` flag to generate a cleaning report without modifying the original file. This report includes row counts before/after, top 10 anomalies, and a distribution chart for each column. Review it to catch unintended data loss (e.g., accidental removal of valid nulls).

### 2. Version Your Cleaning Rules
Treat `rules.yaml` like code. Store it in your Git repo alongside your data pipeline. When you update the cleaning logic, tag the commit (e.g., `v1.2.3-cleaning-rules`). This makes it trivial to reprocess historical data with the exact same transformations—critical for audit trails and regulatory compliance.
