# cleancsvai - CLE_016

## Overview

cleancsvai is a production-grade CSV preprocessing toolkit designed for data engineers, ML pipeline architects, and DevOps practitioners who deal with dirty, malformed, or inconsistent tabular data at scale. Unlike generic spreadsheet cleaners, cleancsvai leverages contextual AI inference to detect semantic anomalies (e.g., a "USA" in a date column) and applies deterministic repair strategies without hallucinating values. This tool is optimized for batch processing, CI/CD data validation gates, and real-time streaming sanitization.

## Core Features

### 1. Schema-Agnostic Anomaly Detection with Contextual Repair
cleancsvai doesn't rely on rigid column type definitions. It uses a hybrid engine that first runs a statistical profile (null ratio, entropy, outlier Z-score) on each column, then feeds flagged cells into a lightweight LLM to infer the intended data type based on surrounding row context. For example, if a column is 95% numeric but contains "N/A" or "TBD", the tool replaces those with `NaN` or a configurable sentinel value—never with a fabricated number. This prevents silent data corruption common in naive imputation libraries.

### 2. Multi-Format Parsing with Frictionless Type Coercion
Supports CSV, TSV, PSV (pipe-separated), and fixed-width files. cleancsvai automatically detects delimiters, encoding (UTF-8, ISO-8859-1, Shift-JIS), and line endings (LF, CRLF). It applies a cascading type coercion strategy: first tries `pandas` inferred types, then falls back to `dateutil` for ambiguous date strings, then uses a regex-based validator for custom patterns (e.g., phone numbers, ZIP codes). If coercion fails for an entire column, it flags the column as `object` and logs a warning—never silently drops rows.

### 3. Deterministic Deduplication with Fuzzy Matching Thresholds
Beyond exact duplicate removal, cleancsvai implements a Jaccard similarity-based fuzzy deduplication engine. You can set a similarity threshold (default 0.85) to merge near-identical rows (e.g., "John Smith" vs "Jon Smith"). The tool preserves the row with the highest data completeness (least nulls) and writes merged records to a separate audit log. This is critical for CRM dedup or inventory reconciliation where exact matches are rare but semantic duplicates are costly.

## How to Use

### Step 1: Validate and Profile Your Source File
Before any cleaning, run `cleancsvai profile --input raw_sales.csv`. This generates a JSON report containing per-column statistics: null count, unique count, min/max length, outlier boundaries (IQR method), and a sample of 10 anomalous rows. Review this report to decide which cleaning strategies to apply. For production pipelines, integrate this step as a CI gate: if null ratio exceeds 15% in a critical column, fail the build and alert the team.

### Step 2: Apply Contextual Cleaning with a Config File
Create a YAML config (e.g., `clean_rules.yaml`) specifying column-level rules:
```yaml
columns:
  - name: "email"
    strategy: "regex_validate"
    pattern: "^[\\w.-]+@[\\w.-]+\\.\\w{2,}$"
    action: "drop_row"
  - name: "price"
    strategy: "coerce_numeric"
    fallback: "nan"
  - name: "date"
    strategy: "coerce_date"
    format: "%Y-%m-%d"
    invalid_action: "flag"
```
Then run: `cleancsvai clean --input raw_sales.csv --rules clean_rules.yaml --output clean_sales.csv`. The tool applies rules in order, logs every transformation to `cleancsvai_audit.log`, and writes a summary of changes (rows dropped, cells modified) to stdout.

### Step 3: Audit and Rollback with Versioned Outputs
cleancsvai automatically creates a timestamped backup of the original file in `./backups/`. After cleaning, run `cleancsvai diff --original raw_sales.csv --cleaned clean_sales.csv` to see a side-by-side diff of changed cells, highlighted by color (green for repairs, red for removals). If the diff reveals unexpected changes, restore the backup and adjust your rules. For regulatory compliance (GDPR, HIPAA), the audit log serves as a complete provenance trail.

## Best Practices

### Practice 1: Always Pre-Profile Before Cleaning
Never run a cleaning pipeline blind. The profile step costs negligible time (sub-second for 100k rows) but prevents catastrophic data loss. In a recent deployment at a fintech startup, profiling revealed that 12% of "transaction_date" values were actually timestamps in Unix epoch format—cleancsvai's contextual repair handled this automatically, but without profiling the team would have dropped those rows silently.

### Practice 2: Use Fuzzy Dedup Sparingly and Validate with a Sample
Fuzzy deduplication is powerful but can merge genuinely distinct records (e.g., "John Smith" and "John Smith Jr."). Always run `cleancsvai dedup --input clean_sales.csv --threshold 0.85 --dry-run` first to see a preview of merged pairs. If the preview shows false positives, raise the threshold to 0.92 or switch to exact matching for that field. For high-stakes dedup (e.g., patient records), never use fuzzy matching without manual review.
