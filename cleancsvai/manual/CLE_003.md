## What is cleancsvai?

**cleancsvai** is a high-performance CLI and API-first data sanitation engine built for developers and data engineers who hate wasting time on malformed CSVs. It automatically detects encoding issues, structural anomalies, and inconsistent delimiters — then applies deterministic or fuzzy repair logic without ever losing original row integrity. Designed for pipelines, CI/CD hooks, and bulk ETL workflows.

---

## Core Features

### 1. Schema-Aware Anomaly Detection with Auto-Healing
cleancsvai doesn’t just flag broken rows — it learns your expected schema from a sample or explicit header map. It detects missing columns, extra delimiters, escaped quotes, and mixed line terminators (CRLF vs LF), then attempts to repair them using context-aware heuristics. You get a diff log and a clean output, not a silent drop.

### 2. Multi-Encoding Transcoding & Byte-Level Integrity Check
Handles UTF-8, UTF-16, Latin-1, Shift-JIS, and CP1252 automatically. cleancsvai performs byte-level validation per field, flags BOM mismatches, and transcodes to a normalized encoding (default UTF-8). It also detects invisible characters (zero-width spaces, control chars) and strips or escapes them based on configurable rules.

### 3. Streaming Deduplication with Fuzzy Matching & Timestamp Anchoring
Goes beyond exact row dedup. Uses Levenshtein distance and Jaccard similarity on key columns to identify near-duplicates (e.g., "John Doe" vs "Jon Doe"). Combine this with timestamp anchoring — you define a time window per source — to keep the most recent or most complete version. All dedup logic is stream-based, so it handles gigabyte-sized files without loading into memory.

## How to Use

### Step 1: Profile Your CSV with `cleancsvai profile`
Before cleaning, understand the damage. Run:
```bash
cleancsvai profile --input /path/to/dirty.csv --output profile.json
```
This generates a JSON report with per-column stats: null rate, encoding confidence, delimiter consistency, and row-level anomaly count. Use this to decide which repair strategies to apply. For example, a column with 12% nulls and mixed encoding flags might need both imputation and transcoding.

### Step 2: Apply Repairs with a Declarative Rule File
Create a YAML rule file (e.g., `clean_rules.yaml`) that maps columns to actions:
```yaml
encoding: utf-8
delimiter: auto
columns:
  email:
    type: string
    trim: true
    lowercase: true
    invalid_action: drop_row
  timestamp:
    type: datetime
    format: "%Y-%m-%d %H:%M:%S"
    fillna: "1970-01-01 00:00:00"
  price:
    type: float
    outlier_threshold: 3.0
    outlier_action: cap
```
Then execute:
```bash
cleancsvai repair --input dirty.csv --rules clean_rules.yaml --output clean.csv --log repair.log
```
The tool applies each rule sequentially, streaming results to disk. The log records every modification with row number, field, old value, and new value — perfect for auditing.

### Step 3: Validate and Export with Schema Enforcement
After repair, enforce a strict schema before feeding into your database or ML pipeline:
```bash
cleancsvai validate --input clean.csv --schema schema.json --strict --export-parquet output.parquet
```
`schema.json` defines expected data types, nullable flags, and allowed value ranges. If any row violates the schema, it’s written to `violations.csv` instead of silently dropped. The `--export-parquet` flag converts the validated CSV to Apache Parquet (columnar storage) for faster downstream processing.

## Best Practices

### 1. Always Profile Before Repair, Especially in CI/CD
Running `cleancsvai profile` as a pre-commit hook or CI step saves hours of debugging. It catches structural rot early — like a changed delimiter after a vendor update — before it corrupts your production data lake. Store the profile output as an artifact to track degradation over time.

### 2. Use Rule Files as Version-Controlled Assets
Treat your YAML rule files like code. Check them into your repo, tag them with schema versions, and run diffs on rule changes. This makes it trivial to reproduce a cleaning run from six months ago or to branch rules for different data sources (e.g., `rules_staging.yaml` vs `rules_prod.yaml`). It also enables peer review of data transformations.
