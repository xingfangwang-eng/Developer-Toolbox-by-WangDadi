# cleancsvai - CLE_014

## Why cleancsvai Exists

Every data engineer has faced the nightmare of a 2GB CSV with malformed rows, mixed encodings, and phantom delimiters. cleancsvai is a deterministic, streaming CSV cleaner that doesn't guess — it validates against your schema, repairs structural corruption, and outputs a normalized dataset without loading the entire file into memory. Built for CI/CD pipelines, ETL preprocessing, and one-shot disaster recovery.

## Three Hardcore Features

### 1. Schema-Aware Row Reconstruction
Instead of blindly dropping malformed rows, cleancsvai parses your target schema (types, nullable columns, expected delimiter count) and uses a **Levenshtein-based column aligner** to re-split misaligned rows. If a row has 14 columns instead of 12, it doesn't fail silently — it attempts to merge or split cells based on nearby pattern entropy. Logs every reconstruction decision to a diff report.

### 2. Encoding Funnel with Byte-Level Fallback
Automatically detects encoding via charset-normalizer, then runs a **three-stage funnel**: UTF-8 → Latin-1 → Windows-1252. If a byte sequence is undecodable, it applies a configurable substitution strategy (replace with `\uFFFD`, drop the byte, or hex-escape it). No silent data loss. Every substitution is timestamped and grouped by byte offset.

### 3. Streaming Dedup with Fuzzy Matching
Deduplication isn't just exact match. cleancsvai supports **sliding-window fuzzy dedup** using MinHash + LSH. You set a similarity threshold (e.g., 0.92) and a window size (e.g., 5,000 rows). The engine keeps a rolling signature buffer and flags near-duplicates without holding the entire dataset in RAM. Outputs a separate CSV of dropped duplicates with similarity scores.

## Three Detailed Workflows

### Workflow 1: Schema Validation & Repair
```bash
cleancsvai validate --input dirty_sales.csv --schema sales_schema.json --repair-mode aggressive
```
1. The engine reads the first 500 rows to infer the delimiter, quote character, and column count. If the inference is ambiguous (e.g., mixed commas and semicolons), it prompts a manual override via `--delimiter`.
2. For each row that deviates from the schema's expected column count, the **column aligner** fires: it computes the edit distance between the malformed row and the expected pattern, then attempts to split or merge cells using the nearest valid row as a template.
3. All repaired rows are written to a separate `_repaired.csv` file. Rows that cannot be repaired (e.g., 5 columns vs 12 with no plausible split) are written to `_dropped.csv` with a reason code. The main output only contains clean rows.

### Workflow 2: Encoding Normalization Pipeline
```bash
cleancsvai encode --input mixed_encodings.csv --output normalized.csv --target-encoding utf-8 --fallback hex-escape
```
1. The engine first runs `chardet`-level detection on the first 10 KB, but also performs a **byte-level scan** for null bytes and invalid UTF-8 sequences. If it finds a null byte, it halts and warns (null bytes are often a sign of binary data injection).
2. During conversion, each byte sequence that fails UTF-8 decoding is caught by the fallback handler. With `--fallback hex-escape`, the engine replaces the invalid byte with `\x{hex}` notation, preserving the original byte value for forensic analysis.
3. The output file includes a sidecar `.encoding_log.json` that lists every byte offset where a substitution occurred, the original byte, the replacement, and the encoding context at that point.

### Workflow 3: Fuzzy Dedup with Rolling Window
```bash
cleancsvai dedup --input customer_export.csv --output deduped.csv --threshold 0.95 --window 10000
```
1. The engine computes a MinHash signature for each row (using 128 permutations). Rows are streamed into a **rolling window** of 10,000 rows. For each new row, the engine compares its signature against all signatures in the window using Jaccard similarity.
2. If a match above 0.95 is found, the engine checks a secondary **exact-match filter** on a subset of critical fields (e.g., email, phone). This prevents false positives from accidentally similar text fields.
3. Duplicates are written to `_duplicates.csv` with a `similarity_score` column and the row index of the kept original. The deduped output retains the first occurrence of each duplicate group.

## Two Industry Best Practices

### Practice 1: Always Pin a Schema for Production Pipelines
Never run cleancsvai in "auto-infer" mode for scheduled ETL jobs. Instead, generate a schema file from a known-good sample using `cleancsvai infer --sample clean_sample.csv --output schema.json`, then validate that schema against the production data source. This prevents silent schema drift — if the data source adds a column or changes delimiter, the pipeline fails fast with a clear error rather than producing corrupted output.

### Practice 2: Use Encoding Logs as a Data Quality Signal
Don't treat encoding substitutions as noise. If your encoding log shows more than 0.1% byte substitutions across a dataset, it's a strong indicator that the upstream system is misconfigured. Set up a monitoring alert that triggers when the substitution rate exceeds that threshold. This catches encoding rot before it spreads to downstream databases or ML training pipelines.
