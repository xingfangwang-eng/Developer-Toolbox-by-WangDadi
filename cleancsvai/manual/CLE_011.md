# cleancsvai - CLE_011

If you’ve ever burned a weekend regexing a 2GB CSV that refuses to play nice with pandas, you already know the pain. **cleancsvai** is a CLI-first, AI-augmented data cleaning toolkit built for engineers who need to ship clean datasets without the spreadsheet drag.

We don’t guess column types. We don’t hallucinate rows. We apply deterministic transformation pipelines with optional ML-powered anomaly detection, all from a single YAML config.

---

## Three Features That Actually Matter

### 1. Schema-Aware Funnel Validation

Instead of treating every row equally, cleancsvai lets you define **column-level constraints** (type, regex, enum, range, uniqueness) and then runs a two-pass validation: first pass catches structural violations, second pass uses a lightweight transformer model to flag semantic outliers (e.g., a `price` column with a string like "free" in an otherwise numeric field). The result is a `.errors.parquet` file you can feed directly into your ETL debugging pipeline.

### 2. Zero-Copy Streaming Deduplication

Most tools load the entire dataset into memory to find duplicates. cleancsvai uses a **bloom filter + rolling hash** approach to detect and remove near-duplicate rows (Levenshtein distance ≤ 2 on key columns) in a single pass. You can process 10M-row CSVs on a 4GB RAM laptop without swapping. The dedup engine also outputs a `duplicate_clusters.json` so you can inspect why rows were flagged.

### 3. Declarative Transformation DSL (YAML-native)

Define your cleaning logic as data. No Python scripts, no notebooks. You write a `cleancsvai.yaml` that describes column remapping, null imputation strategies (mean, median, ffill, or model-based), outlier capping, and date normalization. The engine compiles this into an execution graph, parallelizes independent operations, and logs every mutation with a deterministic hash for full auditability.

```yaml
pipeline:
  - column: "price"
    impute: "median"
    cap_outliers: { method: "iqr", multiplier: 1.5 }
  - column: "email"
    validate: { pattern: "^[\\w.-]+@[\\w.-]+\\.\\w{2,}$" }
    on_failure: "drop_row"
```

## Three Steps to Production-Ready Data

### Step 1: Profile Before You Touch

Don’t clean blind. Run `cleancsvai profile ./raw_data.csv` to generate a comprehensive data profile: column cardinality, missing ratio, skewness, value frequency distribution, and a sample of 100 rows with detected anomalies highlighted. The output is a self-contained HTML report that even non-technical stakeholders can read. This step takes 2 seconds for a 500K-row file.

### Step 2: Write a Targeted Pipeline

Create a `cleancsvai.yaml` that addresses only the issues you found in the profile. For example, if the profile shows 12% missing in `zip_code` and 3% in `city`, you might impute `zip_code` from `city` using a pre-trained geocoding model (bundled with the tool). If `age` has negative values, cap them at 0. If `phone_number` has inconsistent formatting, normalize to E.164. Run `cleancsvai apply --pipeline ./cleancsvai.yaml --input ./raw_data.csv --output ./clean_data.csv`. The engine prints a diff summary: rows removed, cells imputed, outliers capped.

### Step 3: Validate & Lock

After cleaning, run `cleancsvai validate ./clean_data.csv --against ./expected_schema.yaml`. This checks every column against your expected schema (nullability, type, range, uniqueness). If validation passes, the tool generates a **checksum manifest** (`sha256sums.txt`) that you can pin to your CI/CD pipeline. Now your clean dataset is both auditable and reproducible.

## Two Industry Best Practices

### Practice 1: Always Clean at the Pipeline Level, Not the Notebook Level

Data scientists love Jupyter, but notebooks are terrible for reproducibility. Instead, wrap your cleaning logic in a `cleancsvai.yaml` and version it alongside your data schema. This way, when you re-run the pipeline six months later on a new batch of data, you get the exact same transformations. No more "I think I ran cell 17 but maybe not."

### Practice 2: Use Profile-Driven Cleaning, Not Intuition

Don't guess which columns need work. Run `cleancsvai profile` first. The profile will show you exactly where the data is broken—high null rates, unexpected types, outliers, duplicates. Then write a minimal pipeline that only addresses those issues. This prevents over-cleaning (removing valid data) and under-cleaning (leaving garbage in). It also makes your pipeline self-documenting: the YAML file becomes the source of truth for what was done and why.
