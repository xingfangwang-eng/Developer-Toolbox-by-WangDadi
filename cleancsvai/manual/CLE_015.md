# cleancsvai - CLE_015

## Why cleancsvai Exists

Every data scientist and engineer has faced the same nightmare: a CSV file with inconsistent delimiters, rogue line breaks, mixed encoding, and phantom null values that break your pipeline. cleancsvai is not just another CSV parser—it’s a **deterministic repair engine** built for high‑stakes data ingestion. Whether you’re migrating legacy datasets or preprocessing for ML, cleancsvai guarantees a normalized output without silent data loss.

## Hardcore Features That Matter

### 1. **Schema‑Aware Anomaly Injection & Repair**
cleancsvai doesn’t just clean—it learns your expected schema from a sample row, then **injects controlled anomalies** (e.g., swapped date formats, truncated floats, stray Unicode characters) to validate your downstream pipeline. After repair, it outputs a diff report with exact row/column coordinates. You’ll never wonder “did it change something critical?” again.

### 2. **Zero‑Copy Streaming with Multi‑Encoding Auto‑Detection**
For files >2GB, cleancsvai streams rows directly from disk using **memory‑mapped I/O** and a custom FSM parser. It automatically detects and reconciles mixed encodings (UTF‑8, Latin‑1, Shift‑JIS, and 6 others) in the same file—without loading the entire dataset into RAM. This means you can clean a 10‑GB CSV on a 4‑GB laptop.

### 3. **In‑Place Null Token Normalization with Provenance Tags**
Instead of dropping or imputing values, cleancsvai normalizes every null representation (`NA`, `N/A`, `NULL`, `null`, `None`, `NaN`, `""`, `-`, `?`, etc.) into a single `None` token *and* appends a provenance column per field. You can trace exactly which original token was replaced, enabling audit‑grade data lineage.

## Step‑by‑Step: Professional Workflow

### Step 1: Schema Profiling & Anomaly Injection
```bash
cleancsvai profile --input dirty_sales.csv --sample 1000 --output schema.json
```
This command generates a schema profile containing detected column types, null patterns, and value distributions. Then, inject controlled anomalies to test your pipeline:
```bash
cleancsvai inject --schema schema.json --rate 0.05 --output injected.csv
```
You now have a CSV with 5% realistic anomalies (e.g., date format flips, missing quotes). Run your ETL against `injected.csv` to verify it handles edge cases—before touching real data.

### Step 2: Streaming Repair with Multi‑Encoding Fallback
```bash
cleancsvai repair --input injected.csv --schema schema.json --encoding auto --fallback latin1 --output clean.csv --log repair.log
```
The engine streams the file, applies per‑field null normalization, fixes broken UTF‑8 sequences via the fallback encoder, and logs every repaired row with a unique repair ID. The `repair.log` includes original vs. normalized values for audit. Use `--stream` to process files larger than RAM.

### Step 3: Validation & Provenance Export
```bash
cleancsvai validate --input clean.csv --schema schema.json --provenance --output validated.csv
```
This step runs a final schema check (type consistency, null rate, row count). With `--provenance`, it appends a `__cleancsvai_provenance` column to each row, containing a JSON object of original null tokens and their positions. Export the report:
```bash
cleancsvai report --log repair.log --format html --output audit.html
```
You now have a human‑readable audit trail for compliance.

## Best Practices from the Trenches

### 1. **Always Inject Anomalies Before Production Ingestion**
Don’t trust that your pipeline “handles everything.” Use cleancsvai’s `inject` command to create a dirty variant of your production dataset. Run your ingestion pipeline against it and compare with the clean output. This catches silent failures (e.g., a parser that drops rows with stray quotes) before they corrupt your analytics.

### 2. **Pair Null Normalization with Provenance for GDPR/Audit**
If you’re cleaning customer data, never drop or impute nulls silently. Use cleancsvai’s `--provenance` flag to preserve the original null token. Store the provenance column in a separate table. When an auditor asks “why did this field become None?”, you can point to the exact original token and row.
