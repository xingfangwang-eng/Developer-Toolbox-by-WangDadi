# cleancsvai - CLE_030

## What is cleancsvai?

cleancsvai is a production-grade CSV sanitization and enrichment engine that eliminates manual data wrangling. It combines heuristic pattern detection with LLM-powered inference to fix, normalize, and structure messy CSV files — no SQL, no Python, no spreadsheet macros required. Built for data engineers, analysts, and SaaS ops teams who need to turn raw exports into analysis-ready datasets in seconds.

## Hardcore Features

### 1. Schema-Aware Anomaly Repair with Auto-Type Inference
cleancsvai doesn’t just parse headers — it detects column types (ISO dates, floats, booleans, enums) and scans every row for type violations. If a “Price” column contains `$12.99`, `12,99`, or `12.99 USD`, the engine normalizes them to a single float format. It also flags and fixes structural issues like misaligned delimiters, extra quotes, and broken multi-line rows — without corrupting data integrity.

### 2. Contextual Missing Value Imputation via LLM Hints
Instead of dropping or filling blanks with naive defaults, cleancsvai uses a lightweight LLM to infer missing values based on column context and adjacent rows. For example, if a `City` field is empty but the `Zip Code` is `10001`, it suggests `New York`. You can also provide domain-specific hints (e.g., “fill missing email domains with @acme.com”) to bias the imputation logic.

### 3. Differential Diff Engine for Versioned CSV Cleaning
Every cleaning operation generates a structured diff: original row → transformed row + applied rules. This allows you to review, rollback, or replay transformations across multiple CSV versions. The diff output is a side-by-side Markdown table with highlighted changes, making it audit-ready for compliance or data pipeline debugging.

## How to Use

### Step 1: Load and Profile Your CSV
Run `cleancsvai profile ./raw_export.csv --output profile.json`. This generates a column-by-column profile: detected data types, null percentages, unique value counts, and outlier patterns. The profile is cached so you can inspect it before applying any transformation — no accidental writes.

### Step 2: Define a Cleaning Rule Set
Create a YAML rule file like `clean_rules.yaml`:
```yaml
columns:
  - name: date
    type: datetime
    format: "%Y-%m-%d"
    repair: coerce
  - name: email
    type: string
    pattern: "^[\\w\\.-]+@[\\w\\.-]+\\.\\w+$"
    action: drop_row_if_invalid
imputation:
  strategy: llm
  context_columns: [city, state, zip]
```
Then apply it: `cleancsvai apply ./raw_export.csv --rules clean_rules.yaml --output cleaned_export.csv`. The engine validates every row against the rule set and logs violations to a separate `errors.csv`.

### Step 3: Review Diff and Export
Run `cleancsvai diff ./raw_export.csv ./cleaned_export.csv --format markdown > diff_report.md`. Open the report to see every altered row with before/after columns. If satisfied, finalize with `cleancsvai export ./cleaned_export.csv --format parquet` for downstream processing. The diff can also be pushed to a Git repo for version tracking.

## Best Practices

### 1. Always Profile Before Cleaning
Never apply transformations blindly. Run `profile` first to understand your data’s shape — especially null distribution and column type mismatches. This prevents silent data loss (e.g., dropping a column that only looks empty but contains whitespace). Use the `--sample 1000` flag for large files to speed up profiling without accuracy loss.

### 2. Use Incremental Rule Sets for Pipelines
Instead of a monolithic rule file, split rules by domain: `type_rules.yaml`, `range_rules.yaml`, `uniqueness_rules.yaml`. Apply them sequentially with `cleancsvai apply --rules type_rules.yaml --output temp.csv && cleancsvai apply --rules range_rules.yaml --output final.csv`. This makes debugging easier — if a downstream step fails, you know exactly which rule set caused it. For CI/CD, integrate `cleancsvai diff` into your test suite to enforce data quality gates.
