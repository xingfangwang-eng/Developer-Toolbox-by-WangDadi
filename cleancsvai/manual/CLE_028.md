# cleancsvai - CLE_028

## Overview

`cleancsvai` is a lightweight, CLI-first data cleaning utility engineered for developers who hate wrestling with malformed CSVs. It doesn't just strip whitespace—it performs schema inference, fuzzy column matching, and outlier detection on the fly. Whether you're preprocessing training data for ML pipelines or sanitizing legacy exports, this tool eliminates the friction between raw input and production-ready datasets.

**Keywords:** CSV file cleaning, Data cleaning tool, CSV data analysis

---

## 3 Hardcore Features That Set It Apart

### 1. **Adaptive Schema Inference with Auto-Type Casting**
Unlike naive parsers that choke on mixed-type columns or missing headers, cleancsvai scans the first N rows (configurable) to infer column types—`int`, `float`, `datetime`, `string`, or `categorical`. It then auto-casts values, flags type inconsistencies, and optionally coerces or drops rows that break the inferred schema. No more `pd.read_csv()` guessing games.

### 2. **Fuzzy Header Normalization & Column Mapping**
Input CSVs often have headers like `"First Name"`, `"first_name"`, or `"fname"`. cleancsvai applies Levenshtein-based fuzzy matching against a user-defined schema map, then renames and reorders columns accordingly. This is a lifesaver when merging datasets from different departments or third-party APIs with inconsistent naming conventions.

### 3. **Statistical Outlier Detection & Anomaly Flagging**
Beyond simple null checks, cleancsvai runs a configurable IQR (Interquartile Range) or Z-score analysis on numeric columns. It generates a summary report of statistical anomalies—like a `price` column containing `$0.00` or `999999`—and lets you decide whether to clip, impute, or exclude them. This turns a mundane cleanup task into a data quality audit.

## 3 Detailed Operational Steps (Professional Workflow)

### Step 1: Schema Definition & Initial Scan
Create a YAML schema file (`schema.yaml`) that defines expected column names, types, and constraints. Then run the initial scan:

```bash
cleancsvai scan --input ./raw_data.csv --schema schema.yaml --output ./scan_report.json
```

The tool will output a JSON report showing column-wise statistics: null counts, type mismatches, value ranges, and the top 5 most frequent values per column. Review this report to decide whether to auto-fix or manually override.

### Step 2: Apply Transformations with Rule-Based Filtering
Define transformation rules in a TOML config file. For example, to strip whitespace, lowercase strings, and remove rows where `age` is negative or > 120:

```toml
[transforms]
trim_whitespace = true
lowercase_columns = ["email", "city"]
drop_rows = ["age < 0", "age > 120"]
fillna = { strategy = "median", columns = ["salary"] }
```

Execute the cleanup:

```bash
cleancsvai clean --input ./raw_data.csv --config ./rules.toml --output ./cleaned_data.csv
```

The tool prints a diff summary: how many rows dropped, how many cells imputed, and how many types casted.

### Step 3: Validate & Export with Integrity Checks
Run a post-cleanup validation to ensure the output meets your production constraints:

```bash
cleancsvai validate --input ./cleaned_data.csv --schema schema.yaml --checks unique_keys,foreign_keys,value_ranges
```

If all checks pass, the tool returns exit code 0 and writes a validation certificate (`validation_pass.json`). For CI/CD pipelines, integrate this step as a gate before data ingestion.

## 2 Industry Best Practices

### 1. **Always Run a “Dry Clean” Before Production**
Use the `--dry-run` flag on large datasets to preview what will be dropped or modified. This is especially critical when dealing with financial or healthcare records where data loss is unacceptable. The dry run generates a detailed log of every change, which can be audited by a second team member.

### 2. **Version Your Schema Definitions**
Store your schema YAML and transformation TOML files in the same repository as your data pipelines. Tag each schema version (e.g., `schema_v2.1.yaml`) and reference it in your CI/CD configuration. This ensures reproducibility: a dataset cleaned today will produce identical results six months later, assuming the same schema version is used.
