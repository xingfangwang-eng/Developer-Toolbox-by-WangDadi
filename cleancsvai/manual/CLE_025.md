# cleancsvai - CLE_025

## What cleancsvai Actually Does

cleancsvai is a deterministic CSV cleaning engine that doesn't just strip whitespace or deduplicate rows—it **infers column semantics**, **detects structural drift**, and **auto-repairs malformed records** without hallucinating data. Built for developers who ingest unreliable CSV dumps from legacy systems, third-party APIs, or scraped datasets, cleancsvai treats every row as a candidate for schema violation and resolves it using configurable rules, not guesswork.

## Hardcore Features That Set It Apart

### 1. **Schema-Aware Row Repair with Typed Coercion**
cleancsvai doesn't flatten everything to strings. It reads your declared schema (or infers it from the first N rows) and applies type-specific repair: it coerces `"N/A"`, `"null"`, and `"—"` to actual `None` or `NaN`, converts malformed dates using a fallback chain (ISO → US → EU → Unix timestamp), and raises a structured warning for each transformation so you can audit the repair trail.

### 2. **Fuzzy Column Alignment for Misaligned Headers**
When columns shift mid-file (e.g., a missing value pushes the next field left), cleancsvai uses Levenshtein-based header matching to detect the shift and re-aligns rows without dropping data. It outputs a `shift_log.json` showing exactly where alignment corrections were applied—critical for ETL pipelines where silent corruption is unacceptable.

### 3. **Constraint-Aware Deduplication with Conflict Resolution**
Standard dedup removes exact duplicates. cleancsvai goes further: you define a primary key (or composite key), and if two rows match on that key but differ in other fields, it applies a resolution strategy—latest timestamp wins, most complete row wins, or a custom merge function. The conflict log is written to a separate file so you never lose the original variants.

## How to Use cleancsvai Like a Pro

### Step 1: Define a Schema with Type Hints and Fallbacks
Instead of letting the tool guess, provide a JSON schema file that maps each column to a type (`int`, `float`, `date`, `string`, `email`, `url`) and a fallback value for coercion failures. Example schema entry:
```json
{
  "column": "revenue",
  "type": "float",
  "fallback": 0.0,
  "null_values": ["N/A", "n/a", "null", ""]
}
```
Run with `--schema schema.json` to enforce this across the entire file. Any row that fails coercion after fallback is flagged in a separate `coercion_errors.csv`.

### Step 2: Run a Dry-Run with Full Structural Audit
Before modifying your source file, execute a dry-run using `--dry-run --audit audit.json`. This produces a detailed report: number of rows repaired, columns realigned, duplicates resolved, and coercion failures. Inspect `audit.json` to see exact row indices and transformations—this is your safety net before committing changes.

### Step 3: Apply Repairs and Export a Cleaned File with Metadata
Once you're satisfied with the audit, run without `--dry-run` and specify `--output cleaned.csv --metadata repair_manifest.json`. The manifest includes:
- Original row count vs. cleaned row count
- Number of alignment corrections per column
- Dedup resolution strategy applied per conflict
- Coercion fallback usage per column type

This manifest becomes part of your data lineage for compliance or debugging downstream.

## Industry Best Practices for CSV Sanitization

### Practice 1: **Never Clean In-Place; Always Preserve Raw Snapshots**
Many teams overwrite the original CSV, then lose the ability to trace back when a downstream model fails. Instead, use cleancsvai's `--source-backup` flag to create a timestamped copy of the raw file before any transformation. Store this alongside the cleaned output and the repair manifest. This three-file pattern (`raw_20250315.csv`, `cleaned_20250315.csv`, `repair_manifest_20250315.json`) gives you full reproducibility.

### Practice 2: **Run Schema Validation as a Pre-Flight Check in CI/CD Pipelines**
Integrate `cleancsvai validate --schema schema.json --input incoming.csv` into your CI pipeline before allowing any CSV into your data warehouse. If validation fails (e.g., type mismatches or column count drift), the pipeline should reject the file and notify the upstream source. This prevents garbage-in/garbage-out and forces data producers to conform to your schema early, not after hours of downstream debugging.
