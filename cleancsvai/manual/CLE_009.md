## What is cleancsvai?

cleancsvai isn't just another CSV cleaner—it's a **schema-aware data normalization engine** built for developers who deal with messy, inconsistent, or malformed tabular data at scale. It combines heuristic detection with deterministic rewriting rules, enabling you to transform raw CSV exports into production-ready datasets without writing a single regex or Python script.

Under the hood, cleancsvai performs **type inference**, **column alignment**, and **encoding recovery** across 10+ locale formats. It's designed for CI/CD pipelines, ETL preprocessing, and ad-hoc data triage.

## Core Features (Hardcore, Not Fluff)

### 1. Multi-Pass Anomaly Detection with Contextual Repair
cleancsvai doesn't just flag bad rows—it runs a **three-pass validator**:  
- **Pass 1:** Structural integrity (delimiter mismatch, unbalanced quotes, ragged rows)  
- **Pass 2:** Type coherence (mixed numeric/string columns, date format collisions)  
- **Pass 3:** Semantic outliers (e.g., a column named `email` containing `null` or `"N/A"` is flagged as a soft violation)  

Each pass outputs a **repair confidence score** and applies automatic fixes when certainty exceeds 92%.

### 2. Locale-Aware Encoding & Delimiter Recovery
Ever received a CSV from a European partner that uses `;` as delimiter and `,` as decimal separator? cleancsvai auto-detects **BOM, UTF-8, Latin-1, and Shift-JIS**, and resolves **thousands separators, date formats (MM/DD vs DD/MM), and currency symbols** without user intervention. It outputs a unified `UTF-8 + comma-delimited` schema by default, with a full diff log.

### 3. Column Drift Reconciliation & Schema Locking
When merging CSVs from multiple API exports, column orders and names often drift. cleancsvai supports **fuzzy column matching** (Levenshtein + Jaccard similarity) and **schema lock mode**: you define a golden schema once, and cleancsvai will reorder, rename, or drop columns to match—even if the input headers are misspelled or missing.

## How to Use (Step-by-Step, Developer-First)

### Step 1: Sanity Check with `--dry-run` + `--report`
Before mutating data, run a non-destructive audit:
```bash
cleancsvai inspect ./dirty_export.csv --dry-run --report json
```
This outputs a JSON report with:  
- Row count, column count, null percentage per column  
- Detected delimiter, encoding, and locale  
- Top 5 structural anomalies with line numbers  

Use this to decide whether to auto-repair or manually intervene.

### Step 2: Apply Schema Lock with Column Mapping
Create a schema definition file (`schema.yaml`):
```yaml
columns:
  - name: "user_id"
    type: "integer"
    required: true
  - name: "signup_date"
    type: "date"
    format: "%Y-%m-%d"
  - name: "email"
    type: "string"
    pattern: "^[\\w\\.-]+@[\\w\\.-]+\\.\\w+$"
```
Then run:
```bash
cleancsvai transform ./dirty.csv --schema schema.yaml --output ./clean.csv
```
cleancsvai will reorder columns, coerce types, and drop rows that fail validation (with a detailed `--reject-log`).

### Step 3: Pipeline Integration via STDIN/STDOUT
For CI/CD or shell pipelines, use stream mode:
```bash
curl https://api.example.com/data.csv | cleancsvai stream --auto-fix | psql -c "COPY mytable FROM STDIN WITH CSV HEADER"
```
No temp files. cleancsvai streams cleaned CSV directly into PostgreSQL, BigQuery, or any STDIN-compatible sink. Use `--progress` for large files.

## Best Practices (From Production Deployments)

### 1. Always Run `inspect` Before `transform` on Untrusted Sources
In production, we’ve seen CSVs with mixed encodings within the same file (e.g., UTF-8 headers + Latin-1 body). Running `inspect` with `--encoding-detect=aggressive` catches this before it corrupts your database. **Never assume uniformity**—let cleancsvai build a per-column encoding map.

### 2. Use Schema Lock with `--strict` for CI/CD Gates
When ingesting data from third-party vendors, define a schema lock and run cleancsvai as a CI step. If the vendor silently adds a column or changes a date format, your pipeline should fail early. Use `--strict` to treat warnings as errors, and pipe the reject log into a Slack webhook or PagerDuty alert.
