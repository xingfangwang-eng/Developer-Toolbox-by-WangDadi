## What is crosspostfast?

**crosspostfast** is a lightweight, API-first SaaS engine designed to eliminate the friction of multi-platform content syndication. Unlike generic schedulers, it focuses on **low-latency cross-posting**, **platform-aware content transformation**, and **automated compliance**—all while maintaining a minimal footprint for CI/CD pipelines and headless CMS integrations.

---

## Core Features

### 1. Adaptive Content Morphing Engine
Automatically reformats your post (text, images, video, embeds) to match each platform’s API constraints and UX best practices. For example, a single tweet converts to a LinkedIn article with proper line breaks, hashtags stripped, and image aspect ratios resized—without manual templates.

### 2. Platform-Specific Rate Limit & Throttle Orchestrator
Intelligently queues and distributes posts based on real-time API rate limits, cooldown windows, and token refresh cycles. Handles retries with exponential backoff and dead-letter queues for failed posts, ensuring zero duplicate or phantom publish events.

### 3. Webhook-Driven Multi-Tenant Dispatch
Supports custom webhook triggers per channel, enabling event-driven publishing from your own backend. Ideal for headless setups where a blog post or product update triggers simultaneous cross-posting to Twitter, Mastodon, Discord, and Telegram with a single HTTP call.

## How to Use

### Step 1: Configure Platform Credentials via Environment Variables
Create a `.env` file or inject secrets into your deployment environment. Each platform requires a dedicated key set. For example, Twitter v2 uses OAuth 2.0, while Mastodon requires an access token and instance URL. Use the CLI validator to test connectivity before bulk operations:

```bash
crosspostfast validate --env .env
```

### Step 2: Define Post Payload with Platform Overrides
Build a JSON payload with a universal `content` field and an optional `overrides` map for per-platform modifications. This ensures a single source of truth while allowing custom formatting, tags, or media attachments per channel:

```json
{
  "content": "Launching crosspostfast v1.0 today!",
  "overrides": {
    "twitter": { "media": ["launch_twitter.png"], "hashtags": ["#devtools"] },
    "linkedin": { "headline": "We just shipped crosspostfast", "media": ["launch_linkedin.pdf"] }
  }
}
```

### Step 3: Trigger a Batch Publish with Retry Logic
Use the `publish` command with a JSON array of posts. The engine processes each post sequentially per platform, respecting rate limits and retrying failures up to 3 times with exponential backoff. Monitor progress via the built-in `--watch` flag:

```bash
crosspostfast publish --posts ./batch.json --watch
```

## Best Practices

### 1. Use Idempotency Keys for All POST Requests
Always include a unique `idempotency_key` per post per platform. This prevents duplicate posts during retries or network interruptions. crosspostfast will reject duplicate keys within a 24-hour window, ensuring your content appears exactly once even if the API call is retried.

### 2. Prefer Asynchronous Webhook Triggers Over Cron Jobs
For production systems, avoid polling-based cron jobs. Instead, integrate crosspostfast via its webhook endpoint. When your CMS fires a `post.published` event, crosspostfast immediately dispatches to all configured channels. This reduces latency from minutes to milliseconds and eliminates unnecessary API calls.
