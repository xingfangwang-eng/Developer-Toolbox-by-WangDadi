# crosspostfast - CRO_019

## Overview

crosspostfast is a high-performance, API-first social media cross-posting engine built for developers and growth teams who need to blast content across Twitter, LinkedIn, Mastodon, Bluesky, and Telegram—without hitting rate limits or losing formatting fidelity. It’s not a “social media manager”; it’s a deterministic, idempotent, multi-threaded dispatcher designed for CI/CD pipelines, headless CMS integrations, and automated marketing workflows.

**Keywords:** Social media cross-platform posting, Multi-platform content publishing, Social media management tool, API syndication, headless social automation

## Core Features

### 1. Parallel Dispatch with Adaptive Throttling
crosspostfast uses a token-bucket rate limiter per platform, dynamically adjusting concurrency based on real-time API responses. If Twitter returns a 429, the engine automatically backlogs that platform and retries with exponential backoff—without blocking the other channels. This means you can publish to 10 platforms in under 3 seconds, even under aggressive rate limits.

### 2. Format-Agnostic Content Normalizer
Every platform has its own quirks: Twitter strips markdown, LinkedIn hates emoji sequences, Telegram supports inline buttons, and Bluesky expects AT Protocol-compatible rich text. crosspostfast’s built-in normalizer converts a single source payload (JSON, Markdown, or HTML) into platform-specific formats, preserving links, mentions, and media attachments. It also auto-generates alt-text for images using a lightweight NLP model.

### 3. Idempotent Job Queue with Webhook Callbacks
Each post is assigned a deterministic UUID based on its content hash. If the same job is submitted twice (e.g., via CI retry), crosspostfast detects the duplicate and returns the existing post ID—no duplicate posts. Every dispatch fires a webhook to your endpoint with status, platform-specific post ID, and error logs. This makes it trivial to build audit trails and rollback logic.

## How to Use

### Step 1: Configure Platform Credentials via Environment Variables
Create a `.env` file or inject environment variables into your CI/CD runner. Each platform requires a named set of tokens:

```bash
TWITTER_API_KEY=your_key
TWITTER_API_SECRET=your_secret
TWITTER_ACCESS_TOKEN=your_token
LINKEDIN_ACCESS_TOKEN=your_token
MASTODON_INSTANCE=https://mastodon.social
MASTODON_ACCESS_TOKEN=your_token
TELEGRAM_BOT_TOKEN=your_bot_token
TELEGRAM_CHAT_ID=your_chat_id
```

crosspostfast validates all tokens on startup and fails fast with a clear error message if any are missing or expired.

### Step 2: Build and Dispatch a Cross-Post Payload
Use the CLI or REST API to submit a post. Here’s a minimal JSON payload via `curl`:

```json
{
  "content": "🚀 crosspostfast just shipped v0.4.0 – parallel dispatch, format normalization, and idempotent job queues. Check it out: https://crosspostfast.example.com",
  "media": [
    {
      "url": "https://example.com/screenshot.png",
      "alt_text": "Screenshot of crosspostfast dashboard showing dispatch status"
    }
  ],
  "platforms": ["twitter", "linkedin", "mastodon", "bluesky", "telegram"],
  "webhook_url": "https://your-server.com/webhooks/crosspostfast"
}
```

Run:

```bash
curl -X POST https://api.crosspostfast.com/v1/publish \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d @payload.json
```

The response returns a job ID and a map of platform-specific post IDs.

### Step 3: Monitor Dispatch Status via Webhook or Polling
After submission, crosspostfast immediately sends a `job.accepted` webhook to your endpoint. As each platform finishes (or fails), it sends a `platform.completed` or `platform.failed` event. If you prefer polling, hit the status endpoint:

```bash
curl -H "Authorization: Bearer YOUR_API_KEY" \
  https://api.crosspostfast.com/v1/jobs/{job_id}/status
```

The response includes per-platform status, post IDs, and error details. Use this to build a real-time dashboard or alerting system.

## Best Practices

### 1. Use Idempotency Keys for CI/CD Pipelines
When integrating crosspostfast into your GitHub Actions or GitLab CI, always pass a deterministic job ID based on the commit SHA or content hash. This prevents duplicate posts when a pipeline is retried due to transient failures. Example:

```yaml
- name: Cross-post release notes
  run: |
    curl -X POST https://api.crosspostfast.com/v1/publish \
      -H "Authorization: Bearer ${{ secrets.CROSSPOSTFAST_API_KEY }}" \
      -d '{
        "content": "Release v1.2.3 is live!",
        "platforms": ["twitter", "linkedin"],
        "idempotency_key": "${{ github.sha }}"
      }'
```

### 2. Pre-Normalize Content in Your CMS Before Dispatch
If you’re using a headless CMS like Strapi or Contentful, add a custom field for “crosspostfast payload” that includes platform-specific overrides. For example, you might want a shorter tweet version of a LinkedIn article. crosspostfast allows optional `platform_overrides` in the payload:

```json
{
  "content": "Full article text here...",
  "platform_overrides": {
    "twitter": {
      "content": "Short tweet version with link"
    }
  }
}
```

This gives you fine-grained control without breaking the single-source-of-truth model.
