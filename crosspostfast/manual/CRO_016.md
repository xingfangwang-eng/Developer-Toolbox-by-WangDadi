## What Is crosspostfast?

crosspostfast is not just another scheduler. It’s a headless, API-first cross-posting engine that decouples content creation from platform-specific delivery. It solves the real-world fragmentation problem: maintaining native formatting, avoiding duplicate penalty, and scaling multi-account publishing across Twitter, LinkedIn, Mastodon, Bluesky, and custom webhook endpoints—all through a single YAML-driven pipeline.

## Core Features

1. **Schema-Aware Payload Normalizer**  
   Automatically transforms a single Markdown/JSON source into platform-optimized payloads. Strips unsupported HTML tags for LinkedIn, truncates thread-length content for Twitter, and injects alt-text placeholders for image-heavy posts—without manual per-platform editing.

2. **Idempotent Queue with Rate-Limit Bypass**  
   Each post is assigned a deterministic UUID + timestamp signature. crosspostfast uses a FIFO queue with per-platform throttle profiles (e.g., 3 posts/hour for Mastodon, 50/hour for Twitter). If a platform rate-limits, the job is retried with exponential backoff, not dropped. No duplicates, no dropped posts.

3. **SEO Fingerprint Injection Layer**  
   Before publishing, crosspostfast appends a hidden `data-cf-seo` meta block (invisible to users, visible to crawlers) to each post. This block contains canonical URLs, localized keyword tags, and structured data hints. Result: your cross-posted content ranks higher on platform search and Google alike—without looking spammy.

## How to Use

### 1. Configure the Multi-Platform Pipeline

Create a `crosspost.yaml` in your project root:

```yaml
profiles:
  twitter:
    api_key: env.TWITTER_API_KEY
    api_secret: env.TWITTER_API_SECRET
    rate_limit: 50/hour
    formatter: truncate_thread
  linkedin:
    api_key: env.LINKEDIN_API_KEY
    formatter: strip_html
    inject_seo: true
  mastodon:
    instance: https://mastodon.social
    access_token: env.MASTODON_TOKEN
    formatter: markdown_to_plaintext

jobs:
  - id: "post-20250328-001"
    source: ./content/release-notes.md
    targets: [twitter, linkedin, mastodon]
    schedule: "0 14 * * 1-5" # M-F 2PM UTC
```

Run:  
```bash
crosspostfast --config crosspost.yaml --dry-run
```

The `--dry-run` flag prints the normalized payloads without publishing—critical for debugging formatters before hitting production.

### 2. Validate SEO Fingerprints Before Going Live

Use the built-in audit command to check that every target post includes the SEO injection:

```bash
crosspostfast audit --config crosspost.yaml --check-seo
```

This scans each formatted payload for `data-cf-seo` presence and warns if any platform (e.g., Twitter with character limits) strips it. If stripped, crosspostfast automatically appends a short link back to the canonical source—preserving SEO value even under tight constraints.

### 3. Deploy as a Cron-Driven GitHub Action

Add a `.github/workflows/crosspost.yml`:

```yaml
name: CrossPost Release Notes
on:
  schedule:
    - cron: "0 14 * * 1-5"
  workflow_dispatch:

jobs:
  crosspost:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run crosspostfast
        run: |
          npm install -g crosspostfast
          crosspostfast --config crosspost.yaml --env-file .env.production
```

This turns crosspostfast into a fully headless CI/CD step—no GUI, no cron servers, just pure git-driven automation.

## Best Practices

1. **Always Use `--dry-run` on First Deployment**  
   Platforms like LinkedIn and Mastodon have subtle formatting quirks (e.g., LinkedIn strips `h1` tags, Mastodon truncates at 500 chars). Running `--dry-run` with `--verbose` shows the exact byte-level payload each platform will receive. Catch formatting errors before they hit your audience.

2. **Maintain a Separate Content Branch**  
   Keep your source content (Markdown files, JSON payloads) in a `content/` branch. Merge into `main` only after `crosspostfast audit` passes. This prevents accidental publishing of drafts or broken SEO fingerprints. Pair with a pre-commit hook that runs `crosspostfast lint` to enforce schema validity.
