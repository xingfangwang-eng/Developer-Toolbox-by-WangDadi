## Detailed Content

### What is crosspostfast?

**crosspostfast** is a developer-first, multi-tenant platform that eliminates the friction of distributing content across social channels. It’s built for content teams, growth hackers, and indie makers who need to push posts from a single API to Twitter, LinkedIn, Mastodon, Bluesky, and more—without manual copy-paste or third-party middleware.

### Core Features

1. **Idempotent Multi-Platform Dispatch Engine**  
   Each post is assigned a deterministic hash-based ID. The engine ensures exactly-once delivery across all target platforms, even if a network timeout occurs mid-flight. No duplicates. No partial failures.

2. **Template-Aware Content Sharding**  
   Define platform-specific post variants using a simple YAML schema. The system automatically truncates, reformats, or remaps content (e.g., stripping markdown for Twitter, adding image alt text for LinkedIn) based on per-platform rules. No more “threads” or “link in bio” hacks.

3. **Real-Time Webhook-Driven Sync**  
   Every successful post triggers a configurable webhook payload containing platform-specific metadata (post ID, URL, engagement stats). You can pipe this into your own analytics pipeline, Slack channel, or CI/CD workflow. Zero polling required.

### How to Use

1. **Initialize Your Workspace and API Client**  
   Run `crosspostfast init` to generate a `.crosspost.yml` configuration file. Inside, define your target platforms and their credentials (OAuth tokens, API keys). The tool validates token scopes at startup—no silent 401s later.

2. **Define Platform-Specific Post Variants**  
   Create a `posts/` directory with one Markdown file per post. Use frontmatter to specify platform-specific overrides:
   ```yaml
   ---
   platforms:
     twitter:
       text: "Launching crosspostfast today 🚀"
       image: "assets/og-twitter.png"
     linkedin:
       text: "We’re excited to announce crosspostfast..."
       image: "assets/og-linkedin.png"
   ---
   ```
   Run `crosspostfast preview` to see exactly how each platform will render your post before publishing.

3. **Dispatch and Monitor via CLI**  
   Execute `crosspostfast publish --dry-run` to simulate the full pipeline (validation, sharding, webhook triggers) without actually posting. Once satisfied, run `crosspostfast publish` to go live. Use `crosspostfast status` to view delivery logs, retry counts, and platform-specific error codes in real time.

### Best Practices

1. **Use Git-Based Version Control for Post Configurations**  
   Treat your `.crosspost.yml` and `posts/` directory as code. Commit them to a private repo. This allows you to rollback posts, audit changes, and collaborate with your team via pull requests. It also makes CI/CD integration trivial—trigger a deploy on every merge to `main`.

2. **Set Up Webhook-Driven Analytics Pipelines**  
   Don’t rely on platform dashboards. Configure a webhook endpoint (e.g., a Vercel function, AWS Lambda, or your own backend) to receive post delivery events. Store them in a time-series database like ClickHouse or TimescaleDB. This gives you cross-platform engagement metrics, error rates, and latency data—all in one place.
