## What is crosspostfast?

**crosspostfast** is a zero-latency, API-first social media orchestration engine designed to eliminate the repetitive grind of manual cross-posting. It abstracts away platform-specific rate limits, formatting quirks, and authentication flows, allowing you to publish once and deploy everywhere—from Twitter/X to LinkedIn, Mastodon, Bluesky, and beyond—with deterministic scheduling and real-time analytics.

## Core Features

### 1. **Declarative Multi-Platform Schema Engine**
Define a single JSON/YAML payload that maps content (text, media, threads) to each platform’s native API spec. The engine automatically handles character limits, media resizing, and thread splitting (e.g., X threads vs. LinkedIn carousels). No more conditional logic per platform.

### 2. **Atomic Scheduling with Webhook-Aware Retry Logic**
Schedule posts down to the second with atomic commit semantics. If a platform is down, crosspostfast uses exponential backoff + idempotency keys to retry without duplicates. Each post lifecycle emits a webhook payload (status, latency, error codes) for your CI/CD pipelines or monitoring dashboards.

### 3. **Real-Time Cross-Platform Analytics Aggregator**
Aggregate engagement metrics (likes, shares, comments, reach) across all platforms into a unified time-series stream. Exposed via a gRPC endpoint or a RESTful `/analytics` endpoint with support for custom time windows and cohort analysis. Perfect for A/B testing content strategies at scale.

## How to Use

### Step 1: Bootstrap a Cross-Platform Post Manifest
Create a `crosspost.yml` file in your project root. Define your content, media URLs, and target platforms. Each platform block accepts optional overrides (e.g., custom alt text for Instagram, poll options for X). Validate the manifest locally using the CLI: `crosspostfast validate manifest.yml`.

### Step 2: Deploy via CLI or API with Idempotency Tokens
Run `crosspostfast publish manifest.yml --schedule 2025-06-15T14:00:00Z` to queue the post. The tool generates a unique `x-idempotency-key` per post. If you re-run the same command, it will skip duplicates—safe for cron jobs and retries. For programmatic usage, POST to `/v1/publish` with the manifest JSON and your API key.

### Step 3: Monitor Post Lifecycle via Webhook Callbacks
Set up a webhook URL in your dashboard or via the `crosspostfast config set webhook https://your-api.com/callback` command. Each post enters states: `queued → publishing → published | failed`. The webhook payload includes `post_id`, `platform_responses`, and `latency_ms`. Use this to trigger downstream actions (e.g., Slack alerts, database updates) in your stack.

## Best Practices

### Practice 1: **Use Environment-Specific Manifests for Staging vs. Production**
Maintain separate `crosspost.staging.yml` and `crosspost.prod.yml` files. Leverage the `--env` flag to switch contexts. This prevents accidentally pushing draft content to production audiences. Pair with a pre-commit hook that validates the manifest against a schema registry.

### Practice 2: **Implement Dead Letter Queues for Failed Posts**
Configure a DLQ endpoint (e.g., an SQS queue or a Redis stream) via the `crosspostfast config set dead-letter-queue` command. Failed posts are automatically routed here with full error context. Regularly inspect the DLQ to identify platform-specific issues (e.g., rate limits, token expiration) and adjust your publishing cadence accordingly.
