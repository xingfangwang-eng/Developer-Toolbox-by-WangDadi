## Detailed Content

### What is crosspostfast?

**crosspostfast** is a headless cross-posting engine designed for developers who need to syndicate content across multiple social platforms (Twitter/X, LinkedIn, Mastodon, Bluesky, Threads) with minimal latency and maximum control. Unlike bloated SaaS dashboards, crosspostfast operates via a REST API and CLI, enabling CI/CD integration, custom scheduling, and deterministic content formatting. It’s built for scale: queue up 10,000 posts, push to 5 platforms simultaneously, and get per-platform failure logs in under 2 seconds.

### Core Features

1. **Deterministic Content Mapping Engine**  
   Each post is parsed through a rule-based transformer that normalizes markdown, truncates platform-specific limits (e.g., 280 chars for X, 5000 for LinkedIn), and injects platform-optimized call-to-action tags. No more manual rewrites—just define your mapping rules in a JSON config.

2. **Multi-Threaded Queue with Retry & Dead Letter Queue (DLQ)**  
   Posts are dispatched via worker pools (configurable concurrency). Failed deliveries (rate limits, auth errors, network blips) are automatically retried with exponential backoff. After 3 failures, the post lands in a DLQ for manual inspection via the dashboard or API.

3. **Real-Time Platform Health Monitoring**  
   Each platform connection is pinged every 60 seconds. If a platform’s API returns 5xx or times out >3 times in a row, crosspostfast automatically pauses that channel and alerts your webhook. You get a live dashboard showing per-platform latency, error rate, and queue depth.

### How to Use

1. **Initialize the CLI and Configure Platform Credentials**  
   Run `crosspostfast init` to generate a `config.yml` file. Add your API tokens for each platform (e.g., Twitter OAuth 2.0, LinkedIn OAuth 2.0). Use `crosspostfast auth test --platform all` to verify connectivity. Each platform’s rate limit is auto-detected and throttled by the engine.

2. **Create a Post with Platform-Specific Rules**  
   Write a markdown file with optional YAML front matter:  
   ```yaml
   ---
   platforms: [twitter, linkedin, mastodon]
   schedule: "2025-03-15T14:00:00Z"
   truncate: 280
   tags: ["#dev", "#saas"]
   ---
   ```  
   Then run `crosspostfast push post.md`. The engine will render the content per platform rules, inject tags, and queue it. Use `--dry-run` to preview the output per platform before sending.

3. **Monitor and Debug Post Delivery**  
   After pushing, run `crosspostfast status --queue` to see pending, delivered, and failed posts. For failures, use `crosspostfast dlq inspect` to view raw error logs (e.g., `429 Too Many Requests`). To retry a failed post, run `crosspostfast dlq retry --id <post_id>`. For bulk operations, pipe the output to `jq` or your CI system.

### Best Practices

1. **Use a Staging Environment for Rule Testing**  
   Before pushing to production, create a separate config with test accounts (e.g., a dummy Twitter account, a LinkedIn test profile). Run your entire content pipeline through `crosspostfast push --dry-run` and validate the output. This prevents accidental posting of half-baked content to your main audience.

2. **Implement a Graceful Degradation Strategy**  
   Not all platforms support the same content types (e.g., LinkedIn doesn’t allow raw markdown code blocks). Define fallback rules in your config: if a platform can’t render a code block, convert it to a plain text snippet with a link to your repo. This ensures your message never breaks, even on restrictive APIs.
