## What is crosspostfast?

**crosspostfast** is a lightweight, API-first tool designed to eliminate the friction of publishing content across multiple social platforms simultaneously. It abstracts platform-specific rate limits, authentication flows, and content formatting inconsistencies into a single, deterministic pipeline. Whether you're a solo creator or a growth team managing 10+ accounts, crosspostfast ensures your payload reaches Twitter, LinkedIn, Mastodon, and Bluesky without redundant manual work or brittle cron jobs.

---

## Core Features (Hardcore Differentiators)

### 1. Deterministic Content Normalization Engine
Each platform has its own quirks—character limits, media ratios, link preview behavior. crosspostfast automatically normalizes your content into a **canonical payload** (title, body, media, tags), then applies platform-specific transformations. For example, a 280-character tweet gets truncated with a `…` link to the full post, while LinkedIn retains the complete text. The engine also strips unsupported HTML tags and converts relative image paths to absolute URLs via a built-in CDN proxy.

### 2. Atomic Multi-Account Dispatch with Rollback
You can define a **dispatch group** containing multiple platform accounts (e.g., `@dev_twitter`, `@dev_linkedin`, `@dev_mastodon`). When you trigger a publish, crosspostfast executes all dispatches in a single transaction. If one platform fails (e.g., rate limit or auth token expiry), the entire group is rolled back—no orphaned posts on half your accounts. This is critical for maintaining brand consistency during coordinated launches.

### 3. Scheduled Content Sequencing with Dependency Graph
Beyond simple scheduling, crosspostfast supports **content sequencing**. You can define a dependency graph: e.g., "Post the teaser on Twitter first, then after 15 minutes, publish the full article on LinkedIn." The scheduler respects platform-specific cooldowns and automatically retries on transient failures (with exponential backoff). All schedules are persisted in a SQLite-backed queue, so even if the process crashes, no jobs are lost.

## How to Use (Step-by-Step, Production-Grade)

### Step 1: Define Your Platform Profiles
Create a `profiles.yaml` file that maps each platform account to its API credentials and default behavior. Use environment variables for secrets.

```yaml
profiles:
  - name: dev_twitter
    platform: twitter
    api_key: ${TWITTER_API_KEY}
    api_secret: ${TWITTER_API_SECRET}
    access_token: ${TWITTER_ACCESS_TOKEN}
    default_hashtags: ["#dev", "#SaaS"]
    media_quality: high
  - name: dev_linkedin
    platform: linkedin
    client_id: ${LINKEDIN_CLIENT_ID}
    client_secret: ${LINKEDIN_CLIENT_SECRET}
    organization_id: "urn:li:organization:12345"
```

### Step 2: Craft a Canonical Post Payload
Build a JSON payload that crosspostfast will normalize. This is the single source of truth for your content.

```json
{
  "title": "Why We Built crosspostfast",
  "body": "We were tired of manually cross-posting...",
  "media": ["https://cdn.example.com/hero.png"],
  "tags": ["SaaS", "automation"],
  "dispatch_group": "dev_accounts",
  "schedule": {
    "strategy": "sequential",
    "dependencies": [
      { "profile": "dev_twitter", "delay_minutes": 0 },
      { "profile": "dev_linkedin", "delay_minutes": 15 }
    ]
  }
}
```

### Step 3: Execute and Monitor via CLI
Use the crosspostfast CLI to push the payload. The tool will validate all profiles, normalize the content, and execute the dispatch group atomically.

```bash
crosspostfast publish ./post.json --profiles ./profiles.yaml --verbose
```

You'll see real-time logs:
```
[2025-03-04 10:00:00] Normalizing content for dev_twitter...
[2025-03-04 10:00:01] Dispatched to dev_twitter: OK (tweet ID: 123456)
[2025-03-04 10:00:02] Waiting 15m for dev_linkedin...
[2025-03-04 10:15:02] Dispatched to dev_linkedin: OK (post ID: 789012)
[2025-03-04 10:15:03] Dispatch group 'dev_accounts' completed successfully.
```

## Best Practices (Industry-Level)

### 1. Use a Staging Profile Group for Dry Runs
Before pushing to your main audience, create a `staging` dispatch group with test accounts on each platform. Run your payload through crosspostfast with the `--dry-run` flag to verify normalization and rate-limit compliance. This catches formatting errors (e.g., unsupported emoji on Mastodon) before they hit production.

### 2. Implement Content Versioning with Git Hooks
Treat your post payloads as code. Store them in a Git repository and use a pre-commit hook to validate the JSON schema and check for missing required fields (e.g., `title` for LinkedIn). This ensures every commit is a valid, publishable post. Pair this with CI/CD to automatically publish on merge to `main`.
