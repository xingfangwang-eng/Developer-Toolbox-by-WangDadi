### What is crosspostfast?

crosspostfast is a high-performance, headless social media automation engine that eliminates the manual grind of multi-platform publishing. It’s built for SaaS teams, indie hackers, and content ops who need to fire off posts to Twitter, LinkedIn, Reddit, and Mastodon—without ever leaving their terminal or CI/CD pipeline. Think of it as a lightweight, API-first orchestration layer that respects rate limits, handles auth rotation, and logs every response for debugging.

### Core Features

- **Atomic Multi-Platform Dispatch with Idempotency Guards**  
  Each post is hashed and deduplicated across all target platforms. If a network fails mid-push (e.g., 429 or OAuth token expiry), crosspostfast automatically retries with exponential backoff and logs the exact payload for manual intervention. No double-posting, no silent drops.

- **Template-Driven Content Normalization**  
  Define one canonical post body using Markdown + liquid-style variables. crosspostfast auto-strips unsupported HTML, truncates to platform-specific character limits (e.g., 280 for Twitter, 3000 for LinkedIn), and injects platform-optimized hashtags or mentions. This ensures your message survives the translation layer intact.

- **Real-Time Webhook Callbacks for Pipeline Integration**  
  Every successful or failed publish triggers a configurable webhook (JSON payload) to your own backend. Use this to update your CMS status, alert your Slack channel, or trigger a fallback sequence. No polling, no guesswork.

### How to Use

1. **Initialize the Configuration File**  
   Create a `crosspostfast.yaml` in your project root. Define your platform credentials as environment variables (e.g., `TWITTER_API_KEY`, `LINKEDIN_ACCESS_TOKEN`). Use the `platforms` block to list targets and their rate-limit profiles. Example:  
   ```yaml
   platforms:
     twitter:
       enabled: true
       max_retries: 3
       backoff_seconds: 10
     linkedin:
       enabled: true
       max_length: 3000
   ```

2. **Compose and Validate a Post**  
   Run `crosspostfast draft --file ./posts/launch.md`. The CLI will parse your Markdown, validate it against each platform’s constraints, and output a diff report showing how the content will be truncated or modified. Fix any flagged issues (e.g., unsupported emoji on Mastodon) before publishing.

3. **Publish and Monitor via Webhooks**  
   Execute `crosspostfast publish --draft-id 42 --webhook https://your-api.com/callback`. The tool fires posts asynchronously. Your webhook endpoint receives a `POST` payload with `{ "platform": "twitter", "status": "success", "post_id": "123", "raw_response": {...} }`. Set up a simple Node.js or Go handler to log these and alert on failures.

### Best Practices

- **Use a Staging Channel for Dry Runs**  
   Before blasting a high-stakes announcement, run `crosspostfast publish --dry-run`. This validates credentials, checks rate limits, and simulates the post without actually sending. Pair this with a dedicated test account on each platform to catch formatting regressions.

- **Implement a Graceful Degradation Layer**  
   If a platform’s API is down, crosspostfast should not block the entire pipeline. Configure a fallback sequence: e.g., if Twitter fails, queue the post to a local SQLite file and retry every 15 minutes. Use the webhook to notify your ops team via PagerDuty or Slack.
