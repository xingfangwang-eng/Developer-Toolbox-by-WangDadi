# crosspostfast - CRO_021

## Overview

**crosspostfast** is a high-performance, headless SaaS engine designed for developers and content ops teams who need to blast content across multiple social platforms (Twitter/X, LinkedIn, Mastodon, Bluesky, Threads, Telegram, Discord) with sub-second latency and full API control. Think of it as a programmable webhook-to-multipost pipeline, not a GUI dashboard. Built for CI/CD workflows, headless CMS integrations, and serverless functions.

## Core Features

### 1. **Declarative Multi-Platform Schema Mapping**
Define a single YAML/JSON schema per post type (thread, poll, media carousel, link preview) and map fields to platform-specific APIs automatically. No per-platform adapter code. Supports conditional field injection (e.g., `alt_text` for Twitter images, `poll_expiry` for LinkedIn). Schema versioning enforced via semver.

### 2. **Atomic Cross-Platform Queue with Idempotency Tokens**
Each post is assigned a deterministic `idempotency_key` (SHA-256 of content hash + platform list). The queue guarantees exactly-once delivery per platform, even under retries or partial failures. Failed pushes to one platform never block others—think distributed saga pattern. Real-time status stream via Server-Sent Events (SSE) or WebSocket.

### 3. **Dynamic Rate-Limiting & Backpressure Orchestration**
Automatically adapts to per-platform rate limits (Twitter’s 300/3h, LinkedIn’s 100/day) using a token bucket with sliding window. Exposes a `/throttle` endpoint for clients to query current capacity. Integrates with Redis-based distributed counters for multi-node deployments. No manual throttling configs needed.

## How to Use

### 1. **Quickstart: Deploy a Serverless Webhook Endpoint**
```bash
# 1. Install CLI
npm install -g crosspostfast-cli

# 2. Init project
crosspostfast init --platforms twitter,linkedin,mastodon

# 3. Deploy to Vercel/AWS Lambda
crosspostfast deploy --provider vercel --region us-east-1

# 4. Send a test post
curl -X POST https://your-endpoint.vercel.app/api/post \
  -H "Content-Type: application/json" \
  -d '{
    "content": "Hello from crosspostfast!",
    "media_url": "https://example.com/image.png",
    "platforms": ["twitter", "linkedin"]
  }'
```

### 2. **Integrate with Headless CMS (Sanity/Contentful)**
Add crosspostfast as a webhook target in your CMS’s "Publish" event. The webhook payload must include `content`, `platforms`, and optional `metadata` (e.g., `tags`, `scheduled_at`). crosspostfast automatically parses the CMS’s schema (via GraphQL introspection) and maps fields to platform-specific formats. Example Sanity webhook config:
```json
{
  "name": "crosspostfast",
  "url": "https://your-crosspostfast-instance.com/api/post",
  "secret": "your_webhook_secret",
  "triggers": ["document.publish"]
}
```

### 3. **Advanced: Custom Post Scheduler with Cron**
Use crosspostfast’s `/schedule` endpoint with ISO 8601 timestamps. The scheduler supports timezone-aware delivery and can batch posts for optimal engagement windows. Example using `node-cron`:
```javascript
const cron = require('node-cron');
cron.schedule('0 8 * * 1-5', async () => {
  await fetch('https://your-crosspostfast-instance.com/api/schedule', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      content: "Monday morning thread",
      platforms: ["twitter", "linkedin"],
      scheduled_at: "2025-04-07T08:00:00Z"
    })
  });
});
```

## Best Practices

### 1. **Use Idempotency Keys for CI/CD Pipelines**
When integrating crosspostfast into a GitHub Actions or GitLab CI workflow, always generate a deterministic `idempotency_key` from the commit SHA. This prevents duplicate posts if the pipeline is re-run (e.g., after a failed deploy). Example:
```yaml
- name: Post release notes
  run: |
    curl -X POST https://your-crosspostfast-instance.com/api/post \
      -H "Content-Type: application/json" \
      -d '{
        "idempotency_key": "${ github.sha }",
        "content": "v2.1.0 released!",
        "platforms": ["twitter", "mastodon"]
      }'
```

### 2. **Monitor Platform-Specific Error Codes via Webhook Callbacks**
Set up a callback URL in crosspostfast’s config to receive per-platform delivery status (HTTP 200, 429, 403). Use this to trigger alerts (e.g., Slack, PagerDuty) or auto-retry with exponential backoff. For 429 (rate limited), crosspostfast already handles retries internally, but you can track the `retry_count` field in the callback payload to detect abusive patterns.
