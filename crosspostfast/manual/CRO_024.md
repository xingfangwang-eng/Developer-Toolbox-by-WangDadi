## What is crosspostfast?

crosspostfast is a lightweight, API-first orchestrator that eliminates the friction of manual cross-posting. It handles platform-specific content normalization (e.g., Twitter/X character limits vs. LinkedIn article formats), preserves rich media attachments, and provides a deterministic scheduling engine. Designed for scale, it hooks directly into your CI/CD pipeline or CMS webhook.

## Core Features

- **Deterministic Scheduler with Idempotency Guards**  
  Every post carries a unique content hash. If the same hash is detected within a configurable window (default: 24h), the engine silently skips the duplicate—preventing accidental spam during retries or pipeline restarts.

- **Platform-Aware Content Transpilation**  
  Automatically adapts rich text (Markdown, HTML) to each target platform’s native format. Converts `#hashtags` to LinkedIn article tags, strips unsupported HTML tags for Twitter, and re-encodes video thumbnails to fit Instagram’s aspect ratio—all through a single declarative config.

- **Granular Rate-Limit Backpressure**  
  Instead of failing silently, crosspostfast queues posts per platform and respects each API’s rate-limit headers (e.g., `x-rate-limit-remaining`). It uses exponential backoff with jitter, and exposes real-time queue depth via a Prometheus endpoint. No more 429 surprises.

## How to Use

1. **Initialize the Config with Platform Credentials**  
   Create a `crosspost.yaml` file in your project root. Define each platform’s API key, secret, and optional sandbox mode. Example snippet:  
   ```yaml
   platforms:
     twitter:
       api_key: ${TWITTER_API_KEY}
       api_secret: ${TWITTER_API_SECRET}
       sandbox: false
     linkedin:
       access_token: ${LINKEDIN_TOKEN}
   ```  
   Run `crosspostfast init` to validate connectivity and store encrypted tokens in the local keystore.

2. **Draft a Multi-Platform Post with Contextual Overrides**  
   Use the CLI or POST to `/api/post` with a JSON payload. Include a `body` field for the universal content, then optional `overrides` per platform to tweak copy or media.  
   ```json
   {
     "body": "We just shipped v2.1 with zero-downtime migrations.",
     "media": ["https://cdn.example.com/screenshot.png"],
     "overrides": {
       "twitter": { "body": "Shipped v2.1 🚀 Zero-downtime migrations are here." },
       "linkedin": { "body": "Proud to announce crosspostfast v2.1 with true zero-downtime DB migrations." }
     }
   }  
   ```  
   The engine transpiles the body for each target, applies overrides, and validates character limits before enqueuing.

3. **Execute with Dry-Run Mode and Audit Logging**  
   Run `crosspostfast push --dry-run` to see exactly what will be posted without hitting live APIs. Once satisfied, execute with `crosspostfast push --audit`. This writes a JSON audit trail to `./audit/` containing response codes, posted timestamps, and retry counts per platform. Use this for compliance or debugging.

## Best Practices

- **Always run a `--dry-run` before a scheduled burst.**  
  A single malformed media URL can cascade into partial failures. The dry-run validates all media reachability and platform-specific constraints (e.g., Twitter’s 4-image limit) before you commit to the API queue.

- **Leverage the idempotency hash for retries in CI/CD.**  
  If your deployment pipeline auto-posts release notes, ensure the pipeline job ID is part of the content hash. This way, even if the pipeline retries the same step, crosspostfast will skip the duplicate post—keeping your feeds clean.
