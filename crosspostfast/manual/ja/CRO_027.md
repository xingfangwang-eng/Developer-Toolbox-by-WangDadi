# crosspostfast - CRO_027

## 概要

crosspostfast は、SNSクロスプラットフォーム投稿におけるレイテンシとフォーマット崩れを根本解決する、ミドルウェア型高速配信エンジンです。マルチプラットフォームコンテンツ公開におけるAPIレートリミットとメディア変換のボトルネックを、非同期キューと動的フォーマットアダプターで排除します。

## 詳細な内容

### crosspostfast とは

crosspostfast は、単一のペイロードからTwitter、Facebook、Instagram、LinkedIn、TikTok、Pinterest、YouTube Shortsの7プラットフォームへ、平均 **380ミリ秒** で同時投稿を完了するプロフェッショナルツールです。PostgreSQLとの連携により、投稿履歴とメディアキャッシュを管理し、重複投稿やレートリミット超過を自動回避します。

### コア機能

1. **動的フォーマットアダプター（DFA）**  
   各プラットフォームのAPI仕様に基づき、画像はWebP/AVIF/JPEG/PNGの自動変換（`--format-priority` パラメーター指定可）、動画はH.264/H.265/VP9のコーデック選択、解像度は **アスペクト比維持のまま最大4K（3840x2160）までダウンスケール**。メディアサイズ超過時は、分割アップロードとチャンクストリーミングを自動実行。

2. **レートリミット予測エンジン（RLPE）**  
   各プラットフォームのAPI制限（例: Twitterは15分間で300ツイート）をリアルタイムでモニタリングし、`--rate-limit-window`（デフォルト900秒）内の残り容量を予測。超過リスクを検出した場合、**自動でキューイング** し、最適なタイミングで再送信。PostgreSQLの `rate_limit_logs` テーブルに全履歴を保存し、過去7日間のパターンから次回の制限時間を **±5秒精度** で予測。

3. **クロスプラットフォームハッシュ検証（CPHV）**  
   同一コンテンツの多重投稿を防止するため、投稿内容（テキスト+メディアSHA-256）から **64ビットハッシュ** を生成。`crosspostfast_hashes` テーブルで重複チェックを行い、同一ハッシュが存在する場合は `--force` フラグが明示されない限りスキップ。ハッシュ衝突率は **10億件あたり0.0001%未満** を実現。

### 使い方

1. **初期設定と認証情報の投入**  
   `config.yaml` に各プラットフォームのAPIキーとアクセストークンを記述。PostgreSQL接続文字列（例: `postgresql://user:pass@localhost:5432/crosspostfast`）を `--db-dsn` で指定。初回起動時に `schema.sql` が自動実行され、`rate_limit_logs`、`post_hashes`、`media_cache` の3テーブルが作成される。

2. **ペイロードの作成と投稿実行**  
   以下のCLIコマンドで投稿を実行：  
   ```bash
   crosspostfast --payload payload.json --platforms twitter,facebook,instagram --rate-limit-window 900 --format-priority webp,avif --force false
   ```  
   `payload.json` には `text`、`media`（URLまたはBase64）、`metadata`（altテキスト、ハッシュタグ）をJSON形式で記述。投稿後、各プラットフォームの投稿IDとレスポンスタイム（ミリ秒単位）が `post_hashes` テーブルに記録される。

3. **結果の検証とエラーハンドリング**  
   `--verbose` フラグを追加すると、各プラットフォームのAPIレスポンス（HTTPステータスコード、エラーメッセージ、レートリミット残量）が標準出力に表示。エラーが発生した場合、自動で3回リトライ（`--retry-count` で変更可能）。リトライ失敗時は `error_logs` テーブルにスタックトレースとタイムスタンプを保存し、Slack Webhookへの通知も可能。

### ベストプラクティス

- **プラクティス 1: メディアキャッシュの事前生成**  
  同一メディアを複数の投稿で使い回す場合、`--precache` フラグで事前に全プラットフォーム向けの変換済みメディアを生成し、PostgreSQLの `media_cache` テーブルに保存。これにより、初回投稿時の変換時間を **最大70%削減** し、APIレートリミットの消費を最小化。

- **プラクティス 2: レートリミットウィンドウの動的最適化**  
  各プラットフォームのアクティブ時間帯（例: 日本時間の12:00-14:00はInstagramのAPI制限が厳しい）を学習させるため、`--adaptive-window` フラグを使用。過去30日間の投稿パターンから、最適な `rate-limit-window` を自動調整し、深夜帯にバースト投稿をスケジュール。これにより、1日あたりの投稿可能数を **平均40%増加** させる。
