# crosspostfast - CRO_003

## Overview

crosspostfast is a high-performance cross-platform publishing engine engineered to eliminate multi-platform content distribution bottlenecks. It achieves sub-200ms API handshake latency across 12+ social media networks, including Twitter, LinkedIn, Instagram, and TikTok, through a concurrent connection pool architecture. The system is built on a PostgreSQL-backed job queue with ACID compliance, ensuring zero data loss during scheduled bursts of up to 10,000 posts per minute.

## Detailed Content

### What is crosspostfast?

crosspostfast is a production-grade tool that solves the core challenge of multi-platform content synchronization without sacrificing post fidelity. It handles platform-specific media format conversions (e.g., Instagram's 1080x1080 square crop vs. Twitter's 16:9 aspect ratio) in real-time, using FFmpeg-based transcoding pipelines. The tool eliminates manual copy-paste errors and API rate-limit penalties by implementing intelligent throttling with configurable `max_requests_per_second` (default: 50) and `retry_backoff_factor` (exponential: 1.5x, up to 10 retries).

### Core Features

- **Adaptive Media Transcoding Engine**: Automatically resizes, compresses, and reformats images and videos per platform spec. Parameters include `target_width`, `target_height`, `max_file_size_mb` (default: 8), and `codec_profile` (e.g., `h264_high` for LinkedIn, `vp9` for YouTube). Outputs are cached in a Redis-backed blob store with 30-minute TTL for rapid republishing.

- **Atomic Multi-Platform Scheduler**: Uses a distributed cron-like scheduler with `schedule_cron_expression` (standard cron syntax) and `timezone_offset_utc` (integer, e.g., `-5` for EST). Each post is wrapped in a database transaction with `post_id` (UUIDv4), `platform_endpoints` (JSON array), and `status` (enum: `pending`, `published`, `failed`). Failed posts auto-retry with `retry_interval_seconds` (minimum: 60) and log full stack traces to `error_logs` table.

- **Real-Time Analytics Aggregator**: Collects platform-specific engagement metrics via webhook callbacks. Tracks `impressions`, `clicks`, `shares`, and `conversion_rate` across all platforms, stored in a time-series PostgreSQL table partitioned by `post_date` (daily). Provides a `GET /analytics/:post_id` endpoint returning JSON with `metrics_by_platform` and `aggregated_totals`, refreshed every 60 seconds via a background worker.

### How to Use

1. **Initialize Configuration and Connection Pool**  
   Run `crosspostfast init --config config.yaml` to generate a base configuration file. Edit the YAML to set `postgresql_dsn: "postgres://user:pass@host:5432/crosspostfast?sslmode=require"`, `redis_url: "redis://localhost:6379/0"`, and `platform_credentials` (e.g., `twitter_api_key`, `linkedin_access_token`). Execute `crosspostfast connect --pool-size 20 --timeout 5000` to establish persistent HTTP/2 connections to all platforms. Verify with `crosspostfast status --verbose` which prints `latency_ms` per platform.

2. **Create and Schedule a Cross-Platform Post**  
   Use `crosspostfast schedule --post-id "uuid" --content "Hello world" --media ./image.png --platforms twitter,linkedin,instagram --cron "0 9 * * 1-5" --timezone -5`. The tool validates media dimensions against platform limits (e.g., Instagram requires `min_width >= 320`). On success, it returns `scheduled_post_id` and `next_run_at` (ISO 8601). To preview, run `crosspostfast preview --post-id <id>` which generates a simulated output for each platform.

3. **Monitor and Debug Failed Publications**  
   Check failed jobs with `crosspostfast failures --limit 50 --since "2023-10-01T00:00:00Z"`. Output includes `post_id`, `platform`, `error_code` (e.g., `429` for rate limit), `error_message`, and `retry_count`. Force retry a specific post via `crosspostfast retry --post-id <id> --platform twitter --force`. For deep inspection, dump the full error log: `crosspostfast logs --post-id <id> | jq '.errors[] | {platform, timestamp, stack_trace}'`.

### Best Practices

- **Implement Idempotency Keys to Prevent Duplicates**  
  Always generate a unique `idempotency_key` (e.g., SHA-256 of content + timestamp) for each post. Pass this in the `X-Idempotency-Key` header when calling platform APIs. crosspostfast stores these keys in a PostgreSQL unique index, ensuring that network retries don't result in duplicate posts. This is critical for platforms like LinkedIn which have eventual consistency and may accept the same post twice without this safeguard.

- **Use Staged Rollouts for New Platform Connections**  
  When adding a new platform (e.g., Mastodon or Threads), do not enable it in production immediately. Use the `--dry-run` flag in `crosspostfast schedule` to test formatting and API behavior. Monitor the `latency_p99` metric from `crosspostfast metrics`; if it exceeds 3000ms, adjust `max_requests_per_second` downward. Once stable, promote the platform to active with `crosspostfast enable --platform mastodon --throttle 10`. This prevents cascading failures across your entire publishing pipeline.
