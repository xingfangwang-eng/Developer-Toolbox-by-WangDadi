# crosspostfast - CRO_029

## Überblick

crosspostfast optimiert die Multi-Platform-Content-Pipeline für Social-Media-Manager, die täglich zwischen 5 und 20+ Kanälen jonglieren. Statt manueller Copy-Paste-Fehlersuche und fragmentierten Scheduling-Tools liefert dieses System eine einheitliche, API-first Engine für Cross-Posting mit automatischer Formatbereinigung, kanalspezifischer Metadaten-Injektion und Echtzeit-Post-Health-Checks. Die Zielsetzung: Redundanz eliminieren, Konsistenz erzwingen und die Time-to-Publish auf unter 90 Sekunden pro Kampagnen-Batch drücken.

## Hardcore-Funktionen

### 1. Adaptive Content-Normalisierung mit kanalspezifischen Constraints
Jeder Social-Media-Kanal hat eigene Limits – Zeichenzahl, Bildformate, Link-Vorschau, Hashtag-Länge, Emoji-Kompatibilität. crosspostfast parst den eingegebenen Content (Markdown, HTML, Plain Text) und generiert automatisch eine kanalkonforme Version. Beispiel: Twitter/X erhält eine auf 280 Zeichen getrimmte Kurzform mit optimiertem Link-Shortener, während LinkedIn die vollständige, strukturierte Version mit Absätzen und Emoji-Ersatz für Accessibility bekommt. Die Engine erkennt zudem doppelte Postings über mehrere Kanäle und verhindert Cross-Post-Duplikate.

### 2. Multi-Account Scheduler mit Batch-Processing und Post-Queue-Orchestrierung
Statt einzelner Posts erlaubt crosspostfast die Definition von Kampagnen-Blöcken: Ein Blog-Artikel wird gleichzeitig auf LinkedIn, Mastodon, Bluesky, Threads und Facebook geteilt – mit individuellen Timestamps, Timezone-Anpassung und automatischer Retry-Logik bei API-Fehlern. Die Queue-Orchestrierung priorisiert Posts basierend auf Engagement-Historie und Kanal-Auslastung. Zusätzlich gibt es eine „Post-Health“-API, die vor dem Senden prüft, ob Links tot sind, Bilder fehlen oder Hashtags gegen Channel-Richtlinien verstoßen.

### 3. Echtzeit-Cross-Platform-Analytics mit Unified Tracking
Nach dem Posten aggregiert crosspostfast Engagement-Daten (Likes, Shares, Comments, Impressions) aus allen angeschlossenen Kanälen in einem einheitlichen Dashboard. Die Analytics-Engine dedupliziert Interaktionen, die über mehrere Kanäle eingehen (z. B. gleicher Kommentar auf LinkedIn und Twitter), und berechnet kanalübergreifende KPIs wie „Cross-Platform-Reach“ oder „Engagement-Rate pro Kampagne“. API-Export in CSV, JSON oder Google Sheets für Reporting-Pipelines.

## Detaillierte Anleitung – 3 professionelle Workflows

### Workflow 1: Batch-Cross-Posting einer Content-Kampagne

1. **Kampagnen-Definition in der YAML-Konfiguration**  
   Erstelle eine Datei `campaign_blog_post.yaml` mit folgendem Schema:
   ```yaml
   campaign:
     id: "blog_2024_12_01"
     content:
       title: "10 SaaS Growth Hacks for Q1"
       body: "Hier der vollständige Blog-Text in Markdown..."
       image: "./images/hero.png"
       link: "https://example.com/blog/saas-growth"
     channels:
       - name: "linkedin"
         format: "full"
         schedule: "2024-12-02T09:00:00+01:00"
       - name: "twitter"
         format: "short"
         schedule: "2024-12-02T09:05:00+01:00"
       - name: "mastodon"
         format: "medium"
         schedule: "2024-12-02T10:00:00+01:00"
   ```
2. **API-Request mit Validierung**  
   Sende den YAML-Content per POST an den `/api/v1/campaigns` Endpoint:
   ```bash
   curl -X POST https://api.crosspostfast.com/v1/campaigns \
     -H "Authorization: Bearer YOUR_API_KEY" \
     -H "Content-Type: application/yaml" \
     --data-binary @campaign_blog_post.yaml
   ```
   Die API führt sofort eine Pre-Flight-Validierung durch: Prüft Bild-Formate (JPEG/PNG/WEBP), Link-Erreichbarkeit, Zeichenlimits und Kanal-Authentifizierung. Bei Fehlern erhältst du ein detailliertes JSON-Array mit channel-spezifischen Error-Codes.

3. **Queue-Überwachung und Post-Health-Report**  
   Nach erfolgreicher Validierung wird die Kampagne in die Queue gestellt. Über den Endpoint `/api/v1/campaigns/{id}/status` kannst du den Fortschritt live verfolgen:
   ```json
   {
     "campaign_id": "blog_2024_12_01",
     "status": "in_progress",
     "progress": {
       "linkedin": "scheduled",
       "twitter": "posted",
       "mastodon": "failed_retry_1"
     },
     "post_health": {
       "linkedin": {"link_ok": true, "image_ok": true},
       "twitter": {"link_ok": true, "image_ok": false, "error": "Image exceeds 5MB limit"}
     }
   }
   ```
   Im Fehlerfall kannst du per PATCH `/api/v1/campaigns/{id}/retry` einen manuellen Retry mit korrigiertem Image auslösen.

### Workflow 2: Automatisierte Hashtag-Optimierung für LinkedIn & Twitter

1. **Hashtag-Strategie-Definition**  
   Lege in der Konfigurationsdatei `hashtag_strategy.json` kanalspezifische Regeln fest:
   ```json
   {
     "linkedin": {
       "max_hashtags": 5,
       "preferred_categories": ["SaaS", "Tech", "Growth"],
       "exclude": ["#ad", "#sponsored"]
     },
     "twitter": {
       "max_hashtags": 3,
       "preferred_categories": ["Startup", "Marketing"],
       "exclude": ["#followme"]
     }
   }
   ```
2. **API-Integration in den Publishing-Workflow**  
   Wenn du einen Post erstellst, übergibst du den Rohtext und die Zielkanäle. crosspostfast ruft intern die Hashtag-Engine auf, die aus einer vordefinierten Datenbank (oder via NLP aus dem Content) relevante Hashtags extrahiert und nach Channel-Regeln filtert. Beispiel:
   ```bash
   curl -X POST https://api.crosspostfast.com/v1/posts/optimize-hashtags \
     -H "Authorization: Bearer YOUR_API_KEY" \
     -H "Content-Type: application/json" \
     -d '{
       "text": "Unser neues Feature reduziert Ladezeiten um 40%.",
       "channels": ["linkedin", "twitter"]
     }'
   ```
   Response:
   ```json
   {
     "linkedin": {
       "hashtags": ["#SaaS", "#Performance", "#TechInnovation"],
       "text_updated": "Unser neues Feature reduziert Ladezeiten um 40%. #SaaS #Performance #TechInnovation"
     },
     "twitter": {
       "hashtags": ["#Startup", "#Speed"],
       "text_updated": "Unser neues Feature reduziert Ladezeiten um 40%. #Startup #Speed"
     }
   }
   ```
3. **Manuelle Überprüfung und manuelles Posting**  
   Die optimierten Texte werden als Vorschlag zurückgegeben – du kannst sie vor dem Posten manuell anpassen. Erst nach Bestätigung (z. B. per PATCH mit `approved: true`) wird der Post in die Queue gestellt. Dies verhindert automatisierte Fehler bei unpassenden Hashtags.
