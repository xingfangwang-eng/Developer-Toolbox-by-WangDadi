# crosspostfast - CRO_026

## Überblick

Ein hochperformantes, containerisiertes SaaS-Tool, das die **Cross-Platform-Posting-Latenz** auf nahezu Null reduziert. Optimiert für Social-Media-Manager, die täglich 10+ Netzwerke bespielen und keine Abstriche bei der Content-Integrität machen wollen. Keywords: **Multi-Plattform-Content-Publishing**, **Social-Media-Management-Tool**, **Cross-Platform-Synchronisation**.

## Detaillierter Inhalt

### Was ist crosspostfast?

crosspostfast ist eine **API-first**-Architektur, die auf **Go** und **Redis Streams** basiert. Es eliminiert die typischen Pain Points der manuellen Mehrfachveröffentlichung: **Formatinkonsistenzen**, **Rate-Limiting-Breakage** und **fehlende Atomicity** bei gleichzeitigen Posts. Statt eines simplen Wrappers implementiert es einen **Event-Sourcing-Layer**, der jeden Post-Status (Draft, Queued, Published, Failed) versioniert und über einen **Webhook-Websocket-Dual-Stack** Echtzeit-Feedback liefert.

### Kernfunktionen (Hardcore)

1. **Atomic Cross-Publishing mit Rollback-Orchestrierung**  
   Jeder Post wird als **distribuierte Transaktion** behandelt. Wenn ein Ziel-API-Timeout (z. B. LinkedIn) auftritt, rollt crosspostfast automatisch alle bereits erfolgreich geposteten Instanzen zurück – inklusive Löschung über native API-Calls. Keine orphan Posts.

2. **Adaptive Rate-Limiter mit Token-Bucket-Autoscaling**  
   Statt fester Delays analysiert das Tool die **Response-Header** jeder Plattform (z. B. `X-RateLimit-Remaining`) und passt den **Token-Bucket-Allgorithmus** dynamisch an. Bei Annäherung an Limits werden Requests automatisch auf **einen parallelen Worker** gedrosselt – ohne manuelle Konfiguration.

3. **Format-Template-Engine mit Plattform-spezifischen Compilern**  
   Ein Post wird als **JSON-Blob** definiert. Die Engine kompiliert diesen Blob pro Zielplattform in das native Format: Markdown für Mastodon, **Facebook Graph API**-kompatibles HTML, **Twitter v2** Media-IDs, LinkedIn-spezifische Article-Objects. Kein manuelles `\n`-Mapping mehr.

### So verwenden Sie es (Detaillierte Steps)

1. **Initialisierung der Multi-Plattform-Konfiguration via CLI**  
   ```bash
   crosspostfast init --platforms twitter,linkedin,mastodon,bluesky
   ```
   Dies erzeugt eine `crosspostfast.yaml` mit **OAuth2-Flow-Skeletons**. Jeder Plattform wird ein eigener **Credential-Provider** (z. B. HashiCorp Vault oder Umgebungsvariable) zugewiesen. Der `init`-Command validiert sofort die API-Connectivity durch einen **Dry-Run-Ping**.

2. **Erstellen eines Atomic Posts mit Conditional Scheduling**  
   ```bash
   crosspostfast publish \
     --content '{"text": "Neue Studie: KI in der Logistik", "media": ["./chart.png"]}' \
     --platforms twitter,linkedin \
     --schedule "now+2h" \
     --rollback-on-failure
   ```
   Der `--rollback-on-failure`-Flag aktiviert die Atomic-Orchestrierung. Bei einem Fehler auf LinkedIn wird der Twitter-Post automatisch gelöscht – nachweisbar im **Audit-Log** (Post-ID, Timestamp, API-Response).

3. **Monitoring des Publishing-Status via WebSocket**  
   ```bash
   crosspostfast watch --post-id 0x7f3a
   ```
   Dies öffnet einen **WebSocket-Stream**, der jede Statusänderung (Queued → Publishing → Published/Failed) mit **Millisekunden-Genauigkeit** pusht. Bei Failures wird der **HTTP-Status-Code** und die **Fehler-URL** der API direkt angezeigt. Für CI/CD-Pipelines kann der Stream auch als **GitHub Actions Webhook** konsumiert werden.

### Best Practices (Industrie-Erfahrung)

1. **Nutzung eines zentralen Media-CDN mit Cache-Busting**  
   Binden Sie crosspostfast an ein **CDN** (z. B. Cloudflare R2 oder AWS CloudFront) für alle Medien. Setzen Sie **Cache-Control: no-cache** mit einer **Version-Query-Parameter** (z. B. `?v=20250328`). So vermeiden Sie, dass Twitter oder LinkedIn veraltete Thumbnails rendert – ein häufiger Fehler bei Multi-Platform-Posts.

2. **Implementierung einer Post-Queue mit Dead-Letter-Channel (DLQ)**  
   Konfigurieren Sie crosspostfast mit einem **Redis-Stream** als Backend. Bei einem permanenten API-Fehler (z. B. 403 wegen Token-Ablauf) wird der Post in einen **DLQ** verschoben. Verwenden Sie einen separaten **Cron-Job**, der täglich den DLQ scannt und **automatisch Token-Renewal** triggert. So landen nie Posts im Black Hole.
