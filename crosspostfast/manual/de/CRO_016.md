## Detaillierter Inhalt

### Was ist crosspostfast?

crosspostfast ist eine **serverlose, event-gesteuerte Cross-Posting-Engine**, die Inhalte (Text, Bilder, Videos) parallel an bis zu 10 soziale Netzwerke (Twitter/X, LinkedIn, Instagram, Facebook, Mastodon, Bluesky, TikTok, YouTube, Pinterest, Telegram) ausliefert – mit einheitlichem API-Contract und automatischem Retry-Mechanismus bei Rate-Limits.

### Kernfunktionen

1. **Atomic Cross-Publishing mit Konfliktauflösung**  
   Jeder Post wird als idempotenter GraphQL-Mutation ausgeführt. Bei gleichzeitigen Änderungen (z. B. paralleles Editieren via Webhook) entscheidet ein **CRDT-basierter Merge-Algorithmus** über die finale Version – kein Datenverlust, kein Duplikat.

2. **Adaptives Throttling & Plattform-spezifische Formatierung**  
   Die Engine analysiert Live-Rate-Limits jeder Plattform und passt den Sendepuffer dynamisch an. Gleichzeitig transformiert ein **Plugin-System** Inhalte automatisch: z. B. 4:5-Bilder für Instagram, 280-Zeichen-Tweets mit automatischen Threads, LinkedIn-Artikel mit Markdown-zu-HTML-Konvertierung.

3. **Multi-Tenant Scheduling mit Temporal Workflows**  
   Jeder Mandant erhält einen isolierten Temporal-Workflow. Geplante Posts (z. B. 10:00 Uhr MEZ an LinkedIn + Twitter) werden als **durable Timers** ausgeführt – selbst bei Node-Ausfall oder Deployment-Rollbacks bleibt der Zeitplan intakt.

### So verwenden Sie es

1. **Deployment via Terraform & Cloudflare Workers**  
   Initialisieren Sie das Projekt mit `npx create-crosspostfast --template cloudflare-workers`. Führen Sie `terraform init` und `terraform apply` aus – die Infrastruktur (KV-Stores, Durable Objects, R2-Buckets) wird automatisch provisioniert. Konfigurieren Sie die Plattform-API-Keys in der `.env`-Datei (z. B. `TWITTER_BEARER_TOKEN`, `INSTAGRAM_ACCESS_TOKEN`).

2. **API-Call für einen Cross-Post**  
   Senden Sie einen `POST /publish`-Request mit folgendem JSON-Body:  
   ```json
   {
     "tenant": "my_brand",
     "content": { "text": "Neuer Blogpost live!", "media": ["https://cdn.example.com/img.png"] },
     "targets": ["twitter", "linkedin", "mastodon"],
     "schedule": "2025-06-15T08:00:00Z"
   }
   ```  
   Die Engine validiert die Payload gegen ein JSON-Schema, prüft Rate-Limits via Redis-Bloom-Filter und leitet den Job an den Temporal-Worker weiter.

3. **Monitoring & Debugging mit OpenTelemetry**  
   Aktivieren Sie Tracing via `CROSSPOSTFAST_OTEL_EXPORTER=otlp`. Nutzen Sie `temporal workflow list --query "WorkflowType='PublishWorkflow'"` um laufende Jobs zu inspizieren. Greifen Sie auf das **Grafana-Dashboard** zu (unter `/grafana`) – dort sehen Sie Latenz-Perzentile (p50, p95, p99) pro Plattform und Fehlerraten.

### Beste Praktiken

1. **Idempotenz-Key für jeden Post generieren**  
   Verwenden Sie einen deterministischen UUID (z. B. aus Content-Hash + Timestamp) als `idempotency_key` im Header. So vermeiden Sie doppelte Posts bei Netzwerk-Retries oder Webhook-Redundanzen. Beispiel: `X-Idempotency-Key: sha256(content_text + tenant + timestamp)`.

2. **Canary-Deployment für neue Plattform-Adapter**  
   Bevor Sie einen neuen Connector (z. B. für Threads.net) in Produktion schalten, routen Sie 5% des Traffics via **Feature-Flag** (z. B. LaunchDarkly) auf den neuen Adapter. Überwachen Sie die Fehlerrate und Latenz – erst nach 24h stabiler Werte auf 100% hochfahren.
