# crosspostfast - CRO_030

## Übersicht

`crosspostfast` ist kein weiterer Social-Media-Manager, sondern eine **Headless Cross-Posting API & CLI-Engine** für Entwickler, DevOps und Content-Operations. Optimiert für die automatisierte Multi-Plattform-Publikation mit granularer Steuerung über Metadaten, Throttling und Retry-Logik. Keywords: Social media cross-platform posting, Multi-platform content publishing, Social media management tool.

## Detaillierter Inhalt

### Was ist crosspostfast?

crosspostfast löst das zentrale Problem fragmentierter Publishing-Workflows: Es abstrahiert die unterschiedlichen API-Verträge von Plattformen wie LinkedIn, X, Mastodon, Bluesky und WordPress hinter einer **einheitlichen GraphQL-Schnittstelle**. Statt für jede Plattform einen eigenen Wrapper zu schreiben, definierst du einmal dein Content-Schema – `crosspostfast` kümmert sich um Authentifizierung, Raten-Limits, Content-Normalisierung und Fehlerbehandlung. Ideal für CI/CD-Pipelines, statische Site-Generatoren oder Multi-Channel-Marketing-Automation.

### Kernfunktionen (Hardcore-Features)

1.  **Atomic Cross-Posting mit Rollback auf Plattform-Ebene**: Jeder Post ist eine atomare Transaktion. Schlägt die Publikation auf einer Plattform fehl (z. B. 429 Too Many Requests oder 403 Forbidden), wird der Post auf den erfolgreichen Plattformen **automatisch zurückgesetzt** (soft delete via API). Keine inkonsistenten Zustände mehr zwischen deinen Kanälen. Konfigurierbar über den `rollbackStrategy`-Parameter im `crosspostfast.yaml`.

2.  **Adaptive Rate-Limiting & Queue Shedding**: Die Engine analysiert die `Retry-After`-Header und API-Drosselungsmuster jeder Plattform in Echtzeit. Statt eines starren globalen Limits, passt `crosspostfast` die Queue-Drain-Rate dynamisch an – mit **exponentiellem Backoff und Jitter**. Bei Überlastung wirft das System automatisch niedrig priorisierte Posts aus der Queue (Shedding), ohne die gesamte Pipeline zu blockieren.

3.  **Multi-Format Content Normalization Engine**: Bilder, Videos, Cards, Polls – jedes Netzwerk hat sein eigenes MIME-Type- und Attachment-Schema. `crosspostfast` normalisiert deine Assets automatisch: Konvertiert HEIC zu JPEG, skaliert auf Ziel-Maße, generiert Thumbnails und erstellt fallback-Text für Plattformen ohne Rich-Media-Support. Alles konfigurierbar über `mediaTransformRules` in der `config.toml`.

### So verwenden Sie es (Detaillierte Steps)

1.  **Initialisierung der Pipeline via CLI und Konfigurationsdatei**:  
    Lege ein `crosspostfast.yaml` im Root deines Projekts an. Definiere deine Plattformen, API-Keys (via Umgebungsvariablen) und Content-Regeln. Führe `crosspostfast init --config ./crosspostfast.yaml` aus. Das Tool validiert sofort alle API-Credentials, testet die Verbindung und generiert einen `pipeline.lock`-Status-File. Fehlschläge werden mit spezifischen HTTP-Statuscodes und Plattform-IDs ausgegeben – kein generischer "Auth failed".

    ```yaml
    platforms:
      - name: linkedin
        type: v2
        api_key: ${LINKEDIN_API_KEY}
        account_id: urn:li:organization:123456
        retry: { max_attempts: 3, backoff: exponential }
      - name: mastodon
        type: v1
        instance: https://mastodon.social
        access_token: ${MASTODON_TOKEN}
    content:
      media_rules:
        - max_width: 1200
        - format: jpeg
          quality: 85
    ```

2.  **Erstellung und Dispatch eines Multi-Plattform-Posts über die API**:  
    Sende einen POST-Request an den `crosspostfast`-Daemon (läuft z. B. als Sidecar in deinem K8s-Cluster). Der Body enthält das zentrale Content-Objekt plus plattformspezifische Overrides. Der Daemon parst das JSON, wendet die Normalisierungsregeln an und dispatched die Posts parallel mit adaptivem Throttling. Die Response enthält eine `batch_id` und für jede Plattform den `status` (`posted`, `queued`, `failed`) sowie den `platform_post_id`.

    ```json
    POST /v1/publish
    {
      "content": {
        "text": "Unser neuer Release v3.2 ist da! #devops #release",
        "media": ["https://cdn.example.com/screenshot.png"]
      },
      "overrides": {
        "linkedin": { "visibility": "PUBLIC", "article_url": "https://blog.example.com/v3.2" },
        "mastodon": { "sensitive": false, "language": "de" }
      },
      "strategy": {
        "rollback": true,
        "priority": "high"
      }
    }
    ```

3.  **Monitoring und Fehlerbehandlung mit structured Logging**:  
    Nutze `crosspostfast logs --batch-id <id> --format json` um detaillierte, strukturierte Logs im NDJSON-Format zu erhalten. Jede Zeile enthält `timestamp`, `platform`, `event` (z. B. `publish_start`, `rate_limited`, `rollback_initiated`), `error_code` und `duration_ms`. Für kritische Fehler (z. B. permanent invalid token) schreibt das Tool einen `error.fatal`-Event und stoppt die Pipeline für diese Plattform. Du kannst diese Logs direkt in dein ELK/Loki-Stack pipen.

### Beste Praktiken (Industry-Pro-Tipps)

1.  **Idempotency Keys für deduplizierte Publikation**:  
    Jeder POST-Request an `crosspostfast` sollte einen eindeutigen `idempotency_key` (z. B. UUID) im Header mitliefern. Die Engine speichert diesen Key für 24 Stunden. Falls ein Request durch einen Netzwerkfehler (z. B. Timeout) doppelt gesendet wird, erkennt `crosspostfast` den Key und gibt den gespeicherten Response zurück – ohne einen zweiten Post auf den Plattformen zu erzeugen. Das ist besonders kritisch bei automatischen Retries in CI/CD-Pipelines.

2.  **Staging-Instanz mit Mock-Plattformen betreiben**:  
    Starte `crosspostfast` mit `--mode staging` und einer lokalen `mock-platforms.yaml`. Diese Instanz simuliert alle Plattform-APIs mit realistischen Latenzen (200-800ms) und zufälligen Fehlern (z. B. 10% 429, 5% 500). So testest du deine Content-Rules, Rollback-Logik und Queue-Shedding ohne echte API-Credentials. Führe Lasttests mit `crosspostfast stress --requests 1000 --concurrency 10` gegen die Mock-Instanz durch, bevor du in Produktion gehst.
