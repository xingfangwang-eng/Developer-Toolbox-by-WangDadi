## Detaillierter Inhalt

### Was ist crosspostfast?

crosspostfast ist eine hochperformante, containerisierte SaaS-Lösung für automatisiertes Cross-Posting auf Social-Media-Plattformen (LinkedIn, X, Mastodon, Bluesky). Es eliminiert manuelle Redundanzen durch eine asynchrone Event-Architektur (RabbitMQ + Redis), unterstützt native API-Ratenlimits und bietet ein modulares Plugin-System für benutzerdefinierte Content-Transformer (z. B. Markdown-to-HTML, Bildkomprimierung via Sharp). Ideal für Teams, die mehrere Kanäle gleichzeitig bespielen, ohne gegen Plattform-Richtlinien zu verstoßen.

### Kernfunktionen

1. **Intelligentes Quota-Management mit Auto-Queueing**  
   Jede Plattform hat individuelle API-Limits (z. B. 300 Tweets/3h auf X). crosspostfast trackt diese Limits in Echtzeit und stellt Beiträge automatisch in eine prioritätsbasierte Warteschlange – inklusive exponentielles Backoff bei 429-Responses. Keine manuelle Drosselung mehr nötig.

2. **Content-Transformer Pipeline (Plugin-basiert)**  
   Vor dem Posting durchläuft jeder Content eine konfigurierbare Pipeline:  
   - **Format-Mapper** (z. B. automatische Kürzung auf 280 Zeichen für X, Link-Expansion für LinkedIn).  
   - **Medien-Optimierer** (automatische Konvertierung von PNG zu WebP, Exif-Entfernung).  
   - **Hash-Tag-Injektor** (KI-gestützt, basierend auf NLP-Analyse des Texts).  
   Jeder Transformer ist als isoliertes Docker-Image implementiert – Austausch ohne Downtime.

3. **Multi-Account-Scheduling mit zeitgesteuerter Georedundanz**  
   Plane Beiträge für bis zu 50 Accounts pro Plattform gleichzeitig. Nutze Cron-ähnliche Ausdrücke mit Zeitzonen-Unterstützung (z. B. `0 9 * * 1-5 Europe/Berlin`). Die Ausführung erfolgt über einen verteilten Worker-Pool (Kubernetes CronJob), der Last automatisch auf verschiedene Rechenzentren verteilt – minimiert Latenz und Single-Point-of-Failure.

### So verwenden Sie es

1. **API-Key und Plattform-Integration konfigurieren**  
   Erstelle einen API-Key über das Admin-Dashboard (`/admin/api-keys`). Für jede Plattform hinterlegst du OAuth2-Tokens (z. B. über den integrierten OAuth-Flow). Beispiel für X:  
   ```bash
   curl -X POST https://api.crosspostfast.io/v1/platforms/twitter/token \
     -H "Authorization: Bearer <API_KEY>" \
     -d '{"consumer_key":"...", "consumer_secret":"...", "access_token":"...", "access_token_secret":"..."}'
   ```  
   crosspostfast validiert die Tokens sofort und speichert sie verschlüsselt (AES-256-GCM) in einer separaten Vault-Instanz.

2. **Content-Pipeline definieren**  
   Lege über die REST-API eine Pipeline an:  
   ```json
   POST /v1/pipelines
   {
     "name": "blog-to-social",
     "steps": [
       {"type": "truncate", "max_length": 280, "platform": "twitter"},
       {"type": "image_optimizer", "format": "webp", "quality": 85},
       {"type": "tag_injector", "model": "bert-base-multilingual-cased", "max_tags": 3}
     ],
     "fallback": "skip" // bei Fehler: Beitrag überspringen oder in Dead-Letter-Queue verschieben
   }
   ```  
   Die Pipeline wird asynchron im Hintergrund ausgeführt – Status via Webhook (z. B. `POST /webhooks/pipeline-status`) abrufbar.

3. **Beitrag planen und ausliefern**  
   Sende einen POST-Request mit dem Content und der Zielplattform:  
   ```bash
   curl -X POST https://api.crosspostfast.io/v1/posts \
     -H "Authorization: Bearer <API_KEY>" \
     -d '{
       "content": "Heute: Neue Funktionen in crosspostfast v2.4 – automatische Hash-Tag-Generierung!",
       "platforms": ["twitter", "linkedin", "mastodon"],
       "pipeline_id": "pipeline_abc123",
       "schedule": "2025-03-15T09:00:00+01:00"
     }'
   ```  
   Das System bestätigt sofort mit einer `post_id` und liefert den Status über WebSocket-Events (z. B. `post.scheduled`, `post.published`, `post.failed`). Fehlerlogs sind über das Dashboard oder via `GET /v1/posts/{id}/logs` einsehbar.

### Beste Praktiken

1. **Rate-Limiting pro Account und Plattform granular konfigurieren**  
   Deutsche Social-Media-Manager stoßen oft an harte Limits (z. B. LinkedIn: 100 API-Calls/Tag). Nutze in crosspostfast die Option `rate_limit_per_account` in der Plattform-Konfiguration. Beispiel:  
   ```json
   "rate_limit": {"max_requests": 80, "window": 86400, "unit": "seconds"}
   ```  
   Setze den Wert 10–20 % unter dem offiziellen Limit, um Puffer für Retries zu haben. Kombiniere dies mit `retry_after`-Headern aus der API-Antwort – crosspostfast parst diese automatisch und setzt den nächsten Versuch entsprechend zurück.

2. **Pipeline-Tests in einer Staging-Umgebung durchführen**  
   Bevor du eine Pipeline in Produktion schaltest, simuliere sie in einer isolierten Umgebung (z. B. `POST /v1/pipelines/test`). crosspostfast führt die Pipeline aus, ohne tatsächlich zu posten – stattdessen wird ein JSON-Report generiert:  
   ```json
   {
     "status": "simulated",
     "output": {"twitter": "Heute: Neue Funktionen...", "linkedin": "Heute: Neue Funktionen..."},
     "warnings": ["Image too large for X (max 5MB)"],
     "errors": []
   }
   ```  
   So vermeidest du ungewollte Formatierungsfehler oder Medienfehler in der Live-Umgebung. Insbesondere bei deutschen Texten mit Sonderzeichen (Umlaute, ß) ist eine Vorabprüfung essenziell.
