# crosspostfast - CRO_009

## Übersicht

**crosspostfast** ist eine modulare, event-getriebene Cross-Posting-Engine, die speziell für Content-Ökosysteme mit hohem Durchsatz entwickelt wurde. Statt auf starre API-Wrapper setzt das Tool auf eine **plugin-basierte Adapter-Architektur**, die es erlaubt, Plattformen wie LinkedIn, X, Mastodon und Bluesky parallel über asynchrone Worker zu bespielen. Die Engine unterstützt nativ **idempotente Zustellung**, **Rate-Limiting mit Token-Bucket** und **automatische Content-Normalisierung** (HTML→Markdown→Plattform-Markup).

Das Tool ist nicht nur ein Publisher, sondern ein **Orchestrator für verteilte Content-Pipelines** – inklusive Preflight-Validierung, Metadaten-Injection und Post-Mortem-Logs.

## Kernfunktionen

### 1. Multi-Adapter mit dynamischer Plattform-Detektion
Jede Zielplattform wird durch einen unabhängigen **Adapter-Plugin** repräsentiert, der zur Laufzeit über eine YAML-Konfiguration geladen wird. Der Adapter übernimmt automatisch die **Content-Transformation** (z. B. Bildskalierung, Link-Expansion, Hashtag-Filter) und stellt sicher, dass alle Plattform-spezifischen Limits (Zeichenanzahl, Medienanzahl, Thumbnail-Formate) eingehalten werden.

**Beispielkonfiguration:**
```yaml
platforms:
  linkedin:
    adapter: linkedin_v2
    max_chars: 3000
    media_limit: 9
    image_format: jpg
  mastodon:
    adapter: mastodon_v4
    max_chars: 500
    media_limit: 4
    image_format: png
```

### 2. Asynchrone Queue-basierte Zustellung mit Dead-Letter-Handling
crosspostfast nutzt eine **Redis-basierte Job-Queue** (BullMQ oder Sidekiq-kompatibel), um Posts asynchron zu verarbeiten. Jeder Post durchläuft eine Pipeline: `Preflight → Transform → Dispatch → Log`. Fehlgeschlagene Zustellungen werden in einer separaten **Dead-Letter-Queue** (DLQ) gespeichert, mit vollständigem Payload und Fehler-Code. Ein integrierter **Retry-Mechanismus** (exponentielles Backoff) sorgt für Resilienz.

**Pipeline-Log (Auszug):**
```
[2025-03-20T10:00:00Z] Job #a1b2c3: Preflight passed (chars: 420, media: 2)
[2025-03-20T10:00:01Z] Job #a1b2c3: Transform → Mastodon (html→plaintext)
[2025-03-20T10:00:02Z] Job #a1b2c3: Dispatch → Mastodon (HTTP 200)
[2025-03-20T10:00:03Z] Job #a1b2c3: Post-Mortem logged (latency: 2.3s)
```

### 3. Idempotenz-Token + Zustandsmaschine für Transactional Consistency
Jeder Post erhält beim Erstellen einen **UUID-basierten Idempotenz-Token**, der in einer persistenten Tabelle (PostgreSQL) gespeichert wird. Der Zustand eines Posts wird durch eine **Zustandsmaschine** modelliert: `DRAFT → QUEUED → DISPATCHED → CONFIRMED` (oder `FAILED`). Bei erneuter Zustellung (z. B. nach Timeout) wird der Token geprüft – doppelte Zustellungen werden verhindert.

**Zustandsübergänge:**
```
DRAFT → (submit) → QUEUED
QUEUED → (dispatch) → DISPATCHED
DISPATCHED → (webhook confirm) → CONFIRMED
DISPATCHED → (timeout) → FAILED
FAILED → (retry) → QUEUED
```

## So verwenden Sie es

### Schritt 1: Initialisierung und Plattform-Konfiguration
Erstellen Sie eine `config/crosspostfast.yml` mit Ihren API-Keys und Plattform-Adaptern. Verwenden Sie **Umgebungsvariablen** für Secrets (z. B. `CROSSPOST_MASTODON_TOKEN`). Starten Sie dann den Queue-Worker:

```bash
crosspostfast worker --config ./config/crosspostfast.yml --queue redis://localhost:6379
```

Der Worker lädt alle Adapter, validiert die Konfiguration (z. B. ob API-Endpunkte erreichbar sind) und startet die asynchrone Verarbeitung.

### Schritt 2: Post-Erstellung mit Preflight-Prüfung
Erstellen Sie einen Post im JSON-Format und senden Sie ihn an den REST-Endpunkt:

```bash
curl -X POST https://api.crosspostfast.io/v1/posts \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "idempotency_key": "unique-uuid-1234",
    "content": "Hallo Welt! Dies ist ein Test-Post für Mastodon und LinkedIn.",
    "media": ["https://example.com/image.jpg"],
    "platforms": ["mastodon", "linkedin"]
  }'
```

Der Endpoint führt eine **Preflight-Prüfung** durch: Zeichenanzahl, Medien-URLs, Plattform-Kompatibilität. Bei Erfolg wird der Post in die Queue eingereiht und erhält den Status `QUEUED`.

### Schritt 3: Monitoring und Post-Mortem-Analyse
Überwachen Sie die Zustellung über den integrierten **WebSocket-Stream** oder über das **Admin-Dashboard** (optional). Jeder Post durchläuft eine Post-Mortem-Phase, bei der Latenz, Fehler-Codes und Plattform-Antworten geloggt werden.

```bash
crosspostfast dashboard --port 8080
```

Im Dashboard sehen Sie:
- Aktuelle Queue-Länge
- Erfolgs-/Fehlerraten pro Plattform
- Dead-Letter-Queue mit Payload-Dump
- Retry-Statistiken

## Best Practices

### 1. Nutzen Sie die Preflight-Validierung für Content-Normalisierung
Validieren Sie nicht nur Zeichenlimits, sondern normalisieren Sie den Content **vor** der Zustellung. Beispiel: Entfernen Sie unsichere HTML-Tags, konvertieren Sie relative URLs in absolute, und fügen Sie plattformspezifische Hashtags (z. B. `#tech` für LinkedIn, `#technology` für Mastodon) hinzu. crosspostfast unterstützt dafür **Custom Hooks** (JavaScript oder Rust), die in der Pipeline ausgeführt werden.

**Empfehlung:** Definieren Sie für jede Plattform einen eigenen Normalizer, der auf die spezifischen Content-Richtlinien abgestimmt ist. So vermeiden Sie Rate-Limits oder Shadow-Bans.

### 2. Implementieren Sie ein dediziertes Monitoring für Dead-Letter-Queues
Fehlgeschlagene Posts landen in der DLQ. Richten Sie einen **Alert** (z. B. über Slack oder PagerDuty) ein, der bei einer DLQ-Größe > 10 ausgelöst wird. Analysieren Sie die Fehler-Codes: HTTP 429 (Rate-Limit) erfordert eine Anpassung des Token-Bucket, HTTP 403 (Permission) deutet auf abgelaufene Tokens hin. crosspostfast bietet dafür ein **CLI-Tool**:

```bash
crosspostfast dlq --inspect --limit 50
```

So behalten Sie die Kontrolle über Ihre Content-Pipeline und vermeiden stille Fehler.
