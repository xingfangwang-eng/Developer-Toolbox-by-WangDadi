## Detaillierter Inhalt

### Was ist crosspostfast?

crosspostfast ist keine gewöhnliche Social-Media-Verwaltungslösung. Es ist ein **asynchrones, ereignisgesteuertes Multi-Platform-Publishing-System**, das speziell für die gleichzeitige Verteilung von Inhalten auf Twitter, LinkedIn, Mastodon und Bluesky optimiert wurde. Es löst das zentrale Problem der **API-Rate-Limit-Kollision** und der **Content-Normalisierung** zwischen unterschiedlichen Plattform-Schemata.

### Kernfunktionen (Hardcore-Features)

1.  **Intelligentes API-Rate-Limit-Queuing (IRLQ)**  
    Statt einfachem Round-Robin implementiert crosspostfast ein gewichtetes Warteschlangensystem, das die aktuelle Drosselung jeder Plattform-API in Echtzeit analysiert. Es verteilt Posting-Aufträge dynamisch auf Basis von `X-RateLimit-Remaining`-Headern und priorisiert Plattformen mit höheren Kontingenten. Dies reduziert die Gesamt-Posting-Latenz um bis zu 40 % im Vergleich zu synchronen Batch-Postern.

2.  **Content-Normalisierungs-Engine mit Plattform-Schema-Mapping**  
    Jede Plattform hat eigene Content-Constraints: Zeichenlimits, Media-Typen, Link-Formate (z. B. Twitter vs. LinkedIn). crosspostfast wandelt eingehende Rohdaten (Markdown + JSON-Metadaten) automatisch in plattformspezifische Payloads um – inklusive automatischer Bildkompression, Link-Shortening und Hashtag-Extraktion. Dies geschieht über ein konfigurierbares Plugin-System, das in Rust geschrieben ist, um minimale Overhead zu gewährleisten.

3.  **Atomic Rollback & Consistency Check**  
    Sollte ein Posting auf einer Plattform fehlschlagen (z. B. aufgrund eines API-Timeouts oder einer 403-Antwort), führt crosspostfast keinen partiellen Commit durch. Stattdessen wird ein **Atomic Rollback** ausgelöst, der alle erfolgreich geposteten Inhalte auf den anderen Plattformen innerhalb von 5 Sekunden automatisch löscht oder deaktiviert. Dies verhindert inkonsistente Content-Fragmente und bewahrt die Markenintegrität.

### So verwenden Sie es (Detaillierte Schritte)

1.  **API-Key-Konfiguration und Plattform-Registrierung**  
    Rufen Sie die `config.yml` im Stammverzeichnis auf. Definieren Sie für jede Zielplattform einen eigenen API-Key und ein Rate-Limit-Profil. Beispiel für Twitter (X):
    ```yaml
    platforms:
      twitter:
        api_key: "YOUR_TWITTER_BEARER_TOKEN"
        rate_limit_profile: "default"  # Optionen: "default", "aggressive", "conservative"
        max_tweets_per_hour: 300
    ```
    Führen Sie dann `crosspostfast init` aus, um die Plattform-Registrierung zu validieren. Das Tool prüft automatisch, ob die API-Keys gültig sind und ob die Rate-Limit-Profile mit den aktuellen API-Dokumentationen übereinstimmen.

2.  **Content-Template-Erstellung mit Conditional Logic**  
    Erstellen Sie eine `content.tmpl`-Datei im Ordner `templates/`. Nutzen Sie Go-Template-Syntax, um plattformspezifische Blöcke zu definieren:
    ```markdown
    {{ if eq .Platform "linkedin" }}
    **Professioneller Beitrag:** {{ .Title }}  
    {{ .Body }}  
    #Tech #Innovation  
    {{ else if eq .Platform "mastodon" }}
    {{ .Title }}  
    {{ .Body }}  
    #Tech #OpenSource  
    {{ end }}
    ```
    crosspostfast rendert diese Vorlage zur Laufzeit und fügt automatisch plattformspezifische Hashtags oder Formatierungen ein. Validieren Sie das Template mit `crosspostfast validate --template templates/content.tmpl`.

3.  **Batch-Posting mit Atomic Rollback aktivieren**  
    Starten Sie einen Batch-Release mit folgendem Befehl:
    ```bash
    crosspostfast publish --batch data/posts.json --rollback true --timeout 10s
    ```
    Das Tool liest die JSON-Datei mit mehreren Post-Objekten (Titel, Body, Media-URLs) und verteilt sie asynchron. Sollte ein Posting auf einer Plattform fehlschlagen, wird automatisch der Rollback-Mechanismus ausgelöst. Überwachen Sie den Status live mit `crosspostfast status --job-id <ID>`.

### Beste Praktiken für den produktiven Einsatz

1.  **Definieren Sie separate Rate-Limit-Profile für Kampagnen vs. Routine-Posting**  
    Für Produkt-Launches oder virale Kampagnen sollten Sie das `aggressive`-Profil nutzen, das die API-Limits bis zu 95 % ausreizt. Für tägliches Content-Scheduling reicht das `conservative`-Profil, das Puffer für unerwartete Retry-Versuche lässt. Wechseln Sie zwischen Profilen über die `--profile`-Flag, ohne die Konfigurationsdatei ändern zu müssen.

2.  **Nutzen Sie die Atomic-Rollback-Funktion nur in kritischen Workflows**  
    Der Rollback-Mechanismus ist ressourcenintensiv (löscht API-Requests auf allen Plattformen). Verwenden Sie ihn nur für Compliance-relevante Inhalte (z. B. rechtliche Ankündigungen oder Korrekturen). Für Standard-Posts deaktivieren Sie den Rollback (`--rollback false`), um unnötige API-Calls zu vermeiden.
