## What is agents?

`agents` is a lightweight, modular, and language-agnostic framework for designing, deploying, and orchestrating autonomous AI agents. It abstracts away the complexity of multi-agent communication, state persistence, and dynamic task scheduling, making it suitable for building customer support bots, code review agents, or distributed research assistants.

---

## Core Features

### Declarative Agent Graph (DAG) Engine
Define agent workflows as directed acyclic graphs using YAML or JSON. This allows for parallel execution, retry logic, and dead-letter queues to be handled out of the box, reducing complexity and increasing productivity.

### Runtime State Snapshots & Rollback
Every agent execution produces a deterministic state snapshot. This enables pausing, inspecting, and rolling back any agent to a previous state, making it easier to debug complex multi-step reasoning chains in production environments where hallucinations or tool failures occur.

### Polyglot Agent Adapters
Agents can be written in Python, TypeScript, Go, or Rust. The framework provides a unified RPC layer (gRPC + WebSocket) for seamless communication between agents in different languages, eliminating the need for microservice hell and allowing for plug-and-play integration.

## How to Use

### Initialize a Multi-Agent Pipeline with Dynamic Routing
Create a project scaffold using `agents init --name "research-pipeline" --lang python`, which generates a `graph.yaml` file. Define your agents in this file, specifying dependencies and execution logic.
```yaml
nodes:
  - id: "scraper"
    type: "python-agent"
    entry: "agents/scraper.py"
    retry: 3
  - id: "summarizer"
    type: "typescript-agent"
    entry: "agents/summarizer.ts"
    depends_on: ["scraper"]
edges:
  - from: "scraper"
    to: "summarizer"
    condition: "output.status == 'success'"
```
Then, run the pipeline using `agents run --graph graph.yaml --input "https://example.com/report"`, which will execute the agents in the specified order and route data as necessary.

### Inject Custom Tools with Sandboxed Execution
Use the `tool` decorator to inject custom tools that can call external APIs, databases, or shell commands. For example:
```python
# agents/scraper.py
from agents import tool, Agent

@tool(sandbox=True, timeout=10)
def fetch_url(url: str) -> str:
    import requests
    return requests.get(url).text

agent = Agent(tools=[fetch_url])
result = agent.run("Fetch and parse the page")
```
This ensures that custom tools do not leak side effects into the host system, making it suitable for untrusted code or third-party integrations.

### Deploy as a Stateful WebSocket Service
Start a gRPC + WebSocket server using `agents serve --port 8080 --state-dir /var/agents/state`, which allows clients to connect and send tasks. Clients can then receive deterministic state snapshots:
```javascript
const ws = new WebSocket("ws://localhost:8080/agents");
ws.send(JSON.stringify({
  action: "run",
  agent_id: "research-pipeline",
  input: "https://news.ycombinator.com"
}));
ws.onmessage = (event) => {
  const snapshot = JSON.parse(event.data);
  console.log(snapshot.state); // deterministic state
};
```
Each session is isolated, enabling pausing, resuming, or rolling back any agent session via the admin API.

## Best Practices

### Use State Snapshots for A/B Testing Agent Configs
In production, never deploy a new agent configuration without a rollback plan. Use `agents snapshot --id <session_id>` to capture the state before a config change, allowing you to quickly roll back if necessary.
This is especially useful when iterating on prompt engineering or tool selection.

### Isolate Agent Memory Per Tenant
When building a multi-tenant SaaS product, set `--memory-scope tenant` when starting the service. This ensures that agent memory (conversation history, cached tool results) is never shared across tenants, simplifying compliance with GDPR or CCPA and allowing for easy purging of a tenant's entire agent state.
