## Detaillierter Inhalt

### Was ist agents?

**agents** ist ein professionelles, hochperformantes Framework zur Orchestrierung und Verwaltung autonomer KI-Agenten. Es ermöglicht Entwicklern, komplexe, mehrstufige Workflows zu definieren, Agenten mit spezifischen Rollen und Tools auszustatten und deren Interaktion in Echtzeit zu überwachen. Anders als herkömmliche Chatbots oder einfache Automatisierungslösungen bietet agents eine **ereignisgesteuerte, zustandsbehaftete Laufzeitumgebung** für Produktionsumgebungen.

### Kernfunktionen

1.  **Multi-Agenten-Orchestrierung mit dynamischem Task-Routing:**  
    agents verteilt Aufgaben intelligent basierend auf Agenten-Kompetenzen und Systemauslastung. Das Routing erfolgt über ein konfigurierbares, regelbasiertes System (z. B. `priority`, `round-robin`, `least-loaded`), das sich nahtlos in Kubernetes-Umgebungen integrieren lässt. Jeder Agent kann eigene LLMs, Tools (APIs, Datenbanken, Dateisysteme) und Memory-Pools besitzen.

2.  **Stateful Execution Graph mit Checkpointing:**  
    Der Kern von agents ist ein **gerichteter azyklischer Graph (DAG)**, der den gesamten Workflow eines Agenten abbildet. Jeder Knoten (Task) kann einen eigenen Zustand halten. Bei Fehlern oder Timeouts wird der Graph automatisch auf dem letzten erfolgreichen Checkpoint neu gestartet – ohne Datenverlust und mit vollständiger Audit-Trail-Protokollierung.

3.  **Native LLM-Caching & Prompt-Template-Engine:**  
    agents integriert ein **zweistufiges Caching** (In-Memory + Redis) für LLM-Antworten, um Kosten und Latenz zu senken. Die integrierte Template-Engine (Go-Template-basiert) erlaubt die dynamische Generierung von Prompts aus Kontextvariablen, ohne dass der Code neu kompiliert werden muss. Unterstützt werden alle gängigen LLM-Provider (OpenAI, Anthropic, Ollama).

### So verwenden Sie es

1.  **Agenten-Definition per YAML-Konfiguration:**  
    Erstellen Sie eine Datei `agent.yaml` mit folgender Struktur:
    ```yaml
    agent:
      name: "data-analyzer"
      model: "gpt-4-turbo"
      tools:
        - name: "sql_executor"
          type: "database"
          connection: "postgres://user:pass@host:5432/db"
        - name: "web_search"
          type: "api"
          endpoint: "https://api.serper.dev/search"
      memory:
        type: "vector"
        provider: "chromadb"
        collection: "project_insights"
    ```
    Laden Sie die Konfiguration mit `agents load --file agent.yaml`.

2.  **Workflow-Graph definieren und ausführen:**  
    Nutzen Sie die agents CLI, um einen DAG zu erstellen:
    ```bash
    agents workflow create \
      --name "customer_onboarding" \
      --steps "verify_email, enrich_profile, assign_tier, send_welcome" \
      --edges "verify_email->enrich_profile, enrich_profile->assign_tier, assign_tier->send_welcome" \
      --checkpoint-interval 2
    ```
    Starten Sie den Workflow mit `agents run --workflow customer_onboarding --input '{"email":"user@example.com"}'`.

3.  **Echtzeit-Monitoring und Fehlerbehandlung:**  
    Aktivieren Sie das integrierte Dashboard:
    ```bash
    agents dashboard --port 9090
    ```
    Hier sehen Sie Live-Logs, Graph-Status (Knoten farblich markiert: grün = Erfolg, rot = Fehler, gelb = Wiederholung) und können manuelle Eingriffe vornehmen (z. B. einen fehlgeschlagenen Task neu starten oder den Workflow pausieren). Nutzen Sie `agents logs --workflow customer_onboarding --tail 50` für detaillierte Debug-Informationen.

### Beste Praktiken

1.  **Agenten-Spezialisierung statt Generalisten:**  
    Definieren Sie Agenten mit **engen, klar abgegrenzten Verantwortlichkeiten** (z. B. „SQL-Abfrage-Agent“, „E-Mail-Validierungs-Agent“). Vermeiden Sie monolithische Agenten, die alles können. Das erleichtert das Debugging, verbessert die Wiederverwendbarkeit und ermöglicht parallele Ausführung. Nutzen Sie das `tools`-Feld, um jeden Agenten auf eine spezifische Domäne zu beschränken.

2.  **Idempotente Task-Implementierung:**  
    Jeder Task in Ihrem Workflow sollte **idempotent** sein – das bedeutet, dass die mehrmalige Ausführung desselben Tasks mit denselben Eingabedaten immer das gleiche Ergebnis liefert. Dies ist entscheidend für das Checkpointing und die Fehlerbehandlung. Verwenden Sie eindeutige Task-IDs (z. B. UUIDs) und prüfen Sie vor der Ausführung, ob der Task bereits erfolgreich abgeschlossen wurde. agents unterstützt dies nativ über den `checkpoint`-Mechanismus.
