# agents - AGE_026

## Überblick

`agents` ist kein weiteres Low-Code-Spielzeug. Es ist eine **entwicklerzentrierte, modulare Plattform** zur Orchestrierung intelligenter Agenten-Systeme. Statt starrer Pipelines definierst du **dezentrale Entscheidungsgraphen**, in denen spezialisierte Agents parallel verhandeln, iterieren und Ergebnisse validieren. Perfekt für DevOps-Automatisierung, Echtzeit-Datenanalyse und komplexe NLP-Ketten ohne Vendor-Lock-in.

## 3 Hardcore-Funktionen (die den Unterschied machen)

### 1. **Dynamische Agenten-Rekursion & Selbstheilung**
Jeder Agent kann bei Fehlschlag oder Timeout automatisch einen **Child-Agent** spawnen, der die Aufgabe mit alternativen Parametern wiederholt. Kein manuelles Retry-Handling – das System heilt sich selbst, während du den Log-Stream in Echtzeit monitorst.

### 2. **Multi-Modaler Kontext-Graph (MMCG)**
Agents teilen keinen flachen Prompt-Speicher, sondern einen **gerichteten, versionierten Graphen** aus Text, Bildern, Tabellen und API-Responses. Änderungen an einem Knoten triggern automatisch abhängige Agents – ideal für CI/CD-Pipelines, die auf externe Sensor-Daten reagieren.

### 3. **Lokales, quantisiertes Embedding-Caching**
Statt jedes Mal die Cloud zu fragen, speichert `agents` Embeddings **lokal in einer HNSW-Datenbank** (z. B. LanceDB). Bei wiederholten Queries (z. B. „Finde alle fehlerhaften Logs“) sinkt die Latenz von 2s auf < 50ms – ohne GPU-Beschleunigung.

## 3 Operations-Schritte (für erfahrene Entwickler)

### Schritt 1: **Agenten-Topologie als YAML definieren**
Erstelle eine `agents.yaml` mit verschachtelten Agenten, die über `depends_on` und `fallback_agent` verknüpft sind. Nutze `type: recursive` für selbstheilende Workflows. Beispiel:
```yaml
agents:
  - name: log_parser
    type: llm
    model: mistral-7b-instruct
    fallback_agent: log_parser_fallback
  - name: log_parser_fallback
    type: rule_based
    rules: [regex: "ERROR.*", action: "escalate"]
```

### Schritt 2: **Kontext-Graph mit externen Quellen füttern**
Starte den Graph-Server mit `agents graph serve --port 9090`. Nutze die REST-API, um Knoten aus Kafka, Prometheus oder GitHub Issues zu pushen:
```bash
curl -X POST http://localhost:9090/nodes \
  -H "Content-Type: application/json" \
  -d '{"id": "issue-42", "type": "github", "payload": {"title": "Timeout", "severity": "critical"}}'
```
Agents, die auf `type: github` hören, werden sofort getriggert.

### Schritt 3: **Quantisiertes Embedding-Caching aktivieren**
Setze die Umgebungsvariable `EMBEDDING_CACHE_BACKEND=lancedb` und definiere `EMBEDDING_CACHE_PATH=./cache`. Starte den Agenten-Orchestrator neu. Jetzt werden alle Embeddings lokal gecached – perfekt für Offline-Umgebungen oder große Batch-Jobs.

## 2 Best Practices (aus der deutschen DevOps-Community)

### Praxis 1: **Strikte Trennung von Business-Logik und Agenten-Kommunikation**
Nutze den **Message Broker** (z. B. NATS oder RabbitMQ) als Middleware zwischen Agents. So bleiben Agenten entkoppelt und du kannst einzelne Agents ohne Downtime neu starten. Beispiel: Ein Agent parst Logs, ein anderer aggregiert Metriken – beide kommunizieren nur über Topics, nie direkt.

### Praxis 2: **Embedding-Caching mit TTL für dynamische Daten**
Bei Daten, die sich häufig ändern (z. B. Stock-Preise, Wetter), setze `EMBEDDING_TTL=300` (5 Minuten). So vermeidest du veraltete Embeddings, ohne die Performance zu opfern. Für statische Daten (z. B. Dokumentation) setze `TTL=0` für unbegrenztes Caching.
