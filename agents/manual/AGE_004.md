# agents - AGE_004

## Overview

`agents` is a high-performance, event-driven AI agent orchestration framework designed for developers building production-grade autonomous systems. It abstracts away the boilerplate of multi-agent communication, state persistence, and tool integration, letting you focus on logic, not plumbing. Whether you're deploying a fleet of customer-facing bots or a complex internal automation mesh, `agents` gives you the hooks to instrument, scale, and debug with surgical precision.

## Core Hardcore Features

### Declarative Agent Graph (DAG) with Dynamic Routing
Define your agent topology as a directed acyclic graph (DAG) using YAML or a fluent Python API. The runtime supports conditional branching, fan-out/fan-in, and dynamic agent spawning based on runtime context—no hardcoded `if/else` chains. Each node can emit typed events that trigger downstream agents, enabling complex workflows like recursive research, multi-step validation, or human-in-the-loop escalation without state management headaches.

In a real-world scenario, imagine you're building a customer service chatbot that needs to escalate issues to human representatives based on the customer's feedback and product complexity. You can define this workflow as a DAG, where each node represents an AI-powered agent, and the edges represent conditional branching and event triggering. This allows you to create a robust and scalable conversational flow without getting bogged down in state management.

### Pluggable Memory & Context Windows
Agents are stateless by default, but `agents` ships with a tiered memory backend (Redis, PostgreSQL, or local SQLite). You can configure sliding context windows (e.g., last 50 turns) or persistent episodic memory. The framework automatically serializes agent state, tool call history, and intermediate results into a structured log, enabling full replay and audit trails. This is critical for compliance-heavy environments like fintech or healthcare.

For instance, consider a medical research pipeline where AI agents need to analyze genomic data and generate hypotheses. By using `agents` with a sliding context window of 50 turns, you can create an episodic memory that captures the agent's decision-making process and contextual information, enabling researchers to re-run the pipeline in isolation or analyze the results retrospectively.

### Tool Execution Sandbox with Rate Limiting & Circuit Breakers
Every tool call (API, database, file I/O) runs inside a sandboxed executor with configurable timeouts, retry policies, and circuit breakers. The runtime tracks latency and error rates per tool, and can dynamically throttle or disable misbehaving tools without crashing the agent. This is a game-changer for production deployments where third-party APIs are flaky or rate-limited.

In a real-world scenario, imagine you're integrating with an external API that has rate limits and occasional downtime. By using `agents` with rate limiting and circuit breakers, you can dynamically adjust the tool's invocation frequency based on its performance, ensuring that your AI agents don't get stuck in an infinite loop or crash due to excessive retries.

## How to Use

### Define the Agent Graph & Tool Registry
Create a `agents.yaml` file in your project root. This defines your agent topology and the tools they can invoke.
```yaml
agents:
  - name: researcher
    model: gpt-4-turbo
    tools: [web_search, document_parser]
    next: [validator]
  - name: validator
    model: gpt-4o-mini
    tools: [fact_check_api]
    next: [reporter]
  - name: reporter
    model: claude-3-haiku
    tools: [markdown_renderer]
    next: null

tools:
  web_search:
    type: http
    endpoint: "https://api.search.com/v1"
    timeout: 10s
    rate_limit: 10/min
    circuit_breaker:
      threshold: 5
      recovery_time: 30s
  document_parser:
    type: local
    command: "python parsers/doc_parser.py"
    timeout: 30s
```

Then, in your Python entry point, load the graph and register the tool implementations.
```python
from agents import AgentGraph, ToolRegistry

graph = AgentGraph.from_yaml("agents.yaml")
registry = ToolRegistry()
registry.register("web_search", WebSearchTool(api_key="sk-..."))
registry.register("document_parser", DocumentParserTool())
registry.register("fact_check_api", FactCheckTool())
registry.register("markdown_renderer", MarkdownRendererTool())
```

### Inject Context & Run the Pipeline
Pass a context object (dictionary or Pydantic model) that includes the initial user query, any system prompts, and session metadata. The runtime will propagate this context through the DAG, allowing downstream agents to access upstream results.
```python
context = {
    "user_query": "What are the latest climate models for 2025?",
    "session_id": "sess_abc123",
    "metadata": {"source": "slack_bot"}
}

result = graph.run(context, tool_registry=registry)
print(result.final_output)  # Markdown report from the reporter agent
```

### Monitor & Debug with the Built-in Dashboard
`agents` includes a lightweight web dashboard (`agents dashboard`) that shows real-time agent state, tool call latency, and error logs. You can inspect the full event stream for any session, re-run a specific agent node in isolation, or export the trace as a JSON file for offline analysis.
```bash
# Start the dashboard on port 8080
agents dashboard --port 8080 --db postgresql://user:pass@localhost:5432/agents

# Then open http://localhost:8080 to see live agent traces
```

The dashboard also exposes a REST API for programmatic monitoring (e.g., integrating with Datadog or Grafana).

## Best Practices

### Design for Idempotency at the Tool Level
Since `agents` supports retries and circuit breakers, every tool implementation should be idempotent—calling it multiple times with the same input should produce the same side effects (or none). For example, a database insert tool should use `INSERT ... ON CONFLICT DO NOTHING` or generate a unique idempotency key. This prevents duplicate charges, duplicate records, or corrupted state when a tool times out and gets re-invoked.

### Use Ephemeral Context for High-Volume, Low-Latency Pipelines
If you're running agents in a real-time customer-facing setting (e.g., a chatbot that must respond in <2s), avoid persistent memory backends like PostgreSQL for every turn. Instead, use Redis with a TTL of 5 minutes, and only flush critical events to a durable store asynchronously. This keeps the hot path fast while still maintaining an audit trail. Configure your agent graph with `memory_backend: redis` and set `context_window: 50` to limit memory consumption.
