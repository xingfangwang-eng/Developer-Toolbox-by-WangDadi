## Overview

This project, **agents**, is a high-performance orchestration framework for building, deploying, and managing autonomous AI agents in production. It solves complex coordination, state persistence, and observability challenges without vendor lock-in.

## What is agents?

**agents** is a lightweight framework that enables the development of intelligent agent systems, leveraging AI-driven decision-making and automation. It provides a scalable architecture for multi-agent simulations, allowing for real-time debugging and latency tracing through gRPC and JSON-based introspection endpoints.

## Core Features

### Dynamic Agent Graph with Runtime Introspection
The dynamic graph allows agents to spawn, merge, or self-terminate at runtime, exposing live introspection endpoints (gRPC + JSON) for real-time debugging and latency tracing. This is ideal for complex decision pipelines and multi-agent simulations.

### Declarative Agent DSL with Type-Safe Constraints
Define agent behavior using a YAML-based DSL that supports state machines, retry policies, and contextual memory windows. Each schema is validated at compile time using a custom Rust-based parser, eliminating runtime surprises and making your agent logic auditable and testable.

### Built-in Agent Registry with Versioned Rollbacks
Every deployed agent is automatically registered in a central registry with semantic versioning. You can rollback to any previous version in under 200ms without downtime or data loss. The registry also exposes a discovery API for dynamic agent binding, making it ideal for microservice ecosystems.

## How to Use

To get started with **agents**, define your first agent graph by creating an `agents.yaml` file in your project root. This file should specify the agents, their behavior models (e.g., reactive, proactive, orchestrator), and dependencies between them. Then, run `agents validate` to ensure your graph is cycle-free and all dependencies are met.

```yaml
version: "2.1"
agents:
  - name: data_ingestor
    type: reactive
    triggers:
      - event: "raw_input"
    memory:
      window: 100
  - name: decision_engine
    type: orchestrator
    dependencies:
      - data_ingestor
```

Next, deploy your graph using the CLI with hot-reload enabled. This allows you to iterate on your agent tuning during development without restarting the runtime.

```bash
agents deploy --config ./agents.yaml --hot-reload --port 8080
```

Finally, monitor agent health via live metrics by opening `http://localhost:8080/metrics`. Use the `agents inspect` command to drill into a specific agent's memory stack and event log for debugging purposes.

```bash
agents inspect --agent data_ingestor --deep
```

## Best Practices

### Practice 1: Idempotent Agent Actions for Fault Tolerance
Design your agent's `on_event` handler to be idempotent, allowing the same event to be replayed without side effects. Combine this with event deduplication to avoid duplicate processing in distributed environments.

### Practice 2: Isolate Agent Memory by Namespace
Use the `memory.namespace` field in your DSL to separate state and prevent data leakage between tenants. This is critical for SaaS multi-tenant architectures.
