# agents - AGE_023

## Overview

This is a high-performance, SEO-optimized content page for the **agents** project — a next-gen AI agent orchestration framework designed for developers who demand low-latency, multi-model coordination, and deterministic execution. Built for production-scale intelligent agent systems, it abstracts away the complexity of managing agent lifecycles, memory, and tool integration.

## Core Features

### 1. **Deterministic Agent Graph Execution Engine**
Unlike black-box LLM chains, **agents** uses a directed acyclic graph (DAG) execution model where each node is a self-contained agent with typed inputs/outputs. This guarantees reproducible behavior across runs, critical for CI/CD pipelines and A/B testing. Supports conditional branching, parallel agent spawning, and dynamic tool injection at runtime.

### 2. **Zero-Copy Cross-Agent Memory Bus**
Agents share a distributed, lock-free memory bus using shared-memory IPC (Linux `mmap` + atomic operations). This eliminates serialization overhead for inter-agent context passing — achieving sub-microsecond latency for state handoffs. Perfect for real-time multi-agent simulations or streaming data pipelines.

### 3. **Plug-and-Play Tool Registry with Schema Validation**
Every tool (API, CLI, database, or custom binary) is declared via a JSON Schema contract. The registry auto-generates OpenAPI-compatible endpoints and validates inputs/outputs at the agent boundary. No more runtime type errors or silent failures — agent failures are caught at registration time.

## How to Use

### 1. **Define Agent Graph in YAML**
Create a `agent_graph.yaml` file with your agent topology. Each agent node must specify its `model`, `tools`, and `memory_policy`. Example:
```yaml
agents:
  - id: data_parser
    model: gpt-4-turbo
    tools: [csv_reader, json_validator]
    memory_policy: ephemeral
  - id: report_generator
    model: claude-3-opus
    tools: [markdown_renderer]
    memory_policy: persistent
edges:
  - from: data_parser
    to: report_generator
    condition: "output.status == 'valid'"
```
Run `agents deploy agent_graph.yaml` to compile the graph into a runnable binary.

### 2. **Launch Agent Runtime with Custom Scheduler**
Start the agent runtime with a custom scheduling policy (e.g., FIFO, priority queue, or deadline-aware):
```bash
agents run --scheduler deadline --max-concurrency 16 --log-level debug
```
Each agent runs in its own OS thread. Monitor execution via the built-in WebSocket dashboard at `http://localhost:8080/agents/dashboard`.

### 3. **Instrument with Telemetry and Debugging**
Attach a Jaeger-compatible tracer to inspect each agent’s execution path:
```go
import "github.com/xingfangwang-eng/agents/trace"

tracer := trace.New("my-pipeline")
defer tracer.Close()
// Agent calls are automatically traced via context propagation
```
Use `agents inspect <agent_id>` to dump the full execution trace as a flamegraph.

## Best Practices

### 1. **Use Ephemeral Memory for Stateless Agents**
For agents that perform one-shot transformations (e.g., data cleaning, translation), set `memory_policy: ephemeral`. This avoids unnecessary disk I/O and reduces memory pressure. Reserve `persistent` memory only for agents that maintain long-running conversations or accumulate state across graph cycles.

### 2. **Pin Critical Agents to Dedicated CPU Cores**
In production, use `taskset` or cgroups to pin latency-sensitive agents (e.g., real-time decision agents) to isolated CPU cores. This prevents context-switching noise from other agents or system processes. Example:
```bash
taskset -c 0-3 agents run --agent-id critical_agent
```
Monitor core utilization with `perf stat` to validate isolation.
