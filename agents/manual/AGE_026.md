# agents - AGE_026

## Overview

`agents` is a production-grade AI agent orchestration framework designed for developers who need to build, deploy, and manage autonomous agent systems at scale. It eliminates the complexity of multi-agent coordination, memory persistence, and tool integration, letting you focus on agent logic rather than infrastructure.

## Hardcore Features

### 1. **Dynamic Agent Topology Engine**
Unlike static pipelines, `agents` supports runtime-reconfigurable agent graphs. You can spawn, merge, and kill agent nodes on the fly based on task complexity, system load, or even agent self-assessment. This means your swarm can adapt to unexpected workloads without manual intervention.

### 2. **Contextual Memory Sharding**
Standard RAG or sliding-window memory fails in long-running agents. `agents` implements hierarchical memory sharding with automatic compaction and semantic deduplication. Each agent maintains a working memory (short-term), a project memory (mid-term), and a global store (long-term). The engine decides when to promote or archive context based on relevance scoring, not just token limits.

### 3. **Zero-Config Multi-Model Routing**
You don't lock into one LLM. The built-in router profiles each incoming task (reasoning, code generation, summarization, tool calling) and dispatches it to the optimal model—GPT-4 for complex reasoning, Claude for safety-critical tasks, local Llama for latency-sensitive operations. No API wrappers, no fallback logic to write.

## Detailed Usage

### How to Deploy a Multi-Agent Research System

1. **Define agent roles and capabilities in a single YAML manifest**  
   Create a file `agents.yaml` where you declare a `Researcher` agent (with web search + PDF parsing tools) and a `Synthesizer` agent (with summarization + report generation tools). Each agent gets its own system prompt, temperature, and allowed tool list. The engine validates the graph for circular dependencies before boot.

2. **Launch the swarm with persistent state**  
   Run `agents run --manifest agents.yaml --store postgresql://localhost/agents_db`. This spins up both agents, connects them via a shared message bus, and persists all conversation history and intermediate results into PostgreSQL. You can kill the process and resume later without losing context.

3. **Inject a research objective and monitor execution**  
   Use the CLI: `agents inject --objective "Analyze the impact of GPT-5 on cloud cost optimization"`. The engine splits the objective into sub-tasks, routes them to appropriate agents, and streams logs to stdout. You can observe real-time agent decisions, tool calls, and memory state transitions via the built-in dashboard (localhost:8080).

## Industry Best Practices

### 1. **Use the Circuit Breaker Pattern for Tool Calls**
When your agent invokes external APIs (e.g., web search, database queries), wrap them with `agents.toolkit.circuit_breaker`. This prevents cascading failures: if an API returns 5xx errors three times in a row, the agent automatically falls back to a cached result or a secondary tool. Set `failure_threshold=3` and `recovery_timeout=30s` in the tool definition.

### 2. **Implement Human-in-the-Loop Checkpoints for High-Stakes Decisions**
For production deployments handling financial or legal data, configure agent checkpoints using `agents.hooks.approval_gate`. Insert a `@approval_required` decorator on critical tool functions (e.g., `send_invoice`, `modify_contract`). The agent pauses execution, pushes a notification to Slack/Teams, and waits for a human to approve or reject the action. This keeps autonomy high while maintaining compliance.
