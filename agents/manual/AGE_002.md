# agents - AGE_002

## Overview


**Intelligent Agent Orchestration**

`agents` is a polyglot-compatible runtime for deploying and managing intelligent agent systems. It abstracts away complexities of state persistence, task scheduling, and inter-agent negotiation, allowing you to focus on defining agent logic in Python, JavaScript, or Go.

## Core Features

### 1. **Declarative Agent Graph with Dynamic Routing**

Define agent workflows using a YAML-based directed acyclic graph (DAG). Each node is an agent with its own Large Language Model (LLM) backend, tools, and memory. The runtime automatically resolves dependencies, parallelizes independent agents, and supports dynamic rerouting based on intermediate outputs.

### 2. **Live Agent Telemetry with Structured Logging**

Every agent emits structured JSON logs at each lifecycle step: `init`, `tool_call`, `llm_invoke`, `decision`, and `terminate`. These logs are streamed to a local WebSocket dashboard or exported to OpenTelemetry-compatible backends (e.g., Jaeger, Grafana).

### 3. **Pluggable Agent Memory Backends**

Swap memory backends without changing agent code. Built-in adapters for Redis (ephemeral), PostgreSQL (persistent), and SQLite (embedded). Each backend supports vector search via pgvector or Redis Stack, enabling agents to recall past interactions across sessions.

## How to Use

### 1: Define Your Agent Graph in YAML**

Create a file `agent_graph.yaml` that declares agent roles, tools, and edges. For example:

```yaml
agents:
  scraper:
    llm: gpt-4o
    tools: [fetch_url, extract_links]
    memory: ephemeral
  classifier:
    llm: claude-3-haiku
    tools: [tag_content]
    memory: persistent
edges:
  - from: scraper
    to: classifier
    condition: "output.urls | length > 0"
```

Run `agents run --graph agent_graph.yaml` to instantiate the graph.

### 2: Inject Context and Trigger Execution**

Pass initial context via environment variables or a JSON file:

```bash
agents run --graph agent_graph.yaml --context '{"seed_url": "https://example.com"}'
```

The runtime injects this context into the root agent (first in topological order). Each agent inherits and can mutate a shared context object.

### 3: Inspect and Replay Agent Sessions**

After execution, export the full session trace as a JSON file:

```bash
agents export --session-id <uuid> --format trace.json
```

Load this trace into the built-in replay UI (`agents replay trace.json`) to step through each agent's reasoning chain, examine tool call arguments, and compare token usage across runs.

## Best Practices

### 1: **Isolate Agent Tools by Capability**

Instead of giving every agent a monolithic toolset, create granular tool classes (e.g., `FileSystemTool`, `DatabaseTool`, `HTTPTool`) and assign them to agents via YAML. This prevents agents from accidentally invoking destructive operations.

### 2: **Leverage Ephemeral Memory for Short-Lived Agents**

For agents that only exist within a single graph execution, use `memory: ephemeral` (backed by Redis with TTL). For long-running swarms, use `memory: persistent` (PostgreSQL with vector search).
