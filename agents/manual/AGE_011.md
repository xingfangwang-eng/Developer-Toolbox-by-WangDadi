## Detailed Content

### What is agents?

`agents` is a lightweight, production-grade framework for orchestrating intelligent agent systems. It abstracts away boilerplate like state management, inter-agent communication, and error recovery, letting you focus on agent logic and domain-specific behaviors. Think of it as the missing runtime for agentic architectures.

### Core Features

1. **Dynamic Agent Topology**  
   Agents are not hardcoded into pipelines. Instead, `agents` uses a declarative YAML/JSON schema to define agent roles, dependencies, and routing rules. At runtime, the engine resolves the DAG (Directed Acyclic Graph) and spawns agents on demand—perfect for systems that need to scale horizontally without recompilation.

2. **Built-in Contextual Memory with Vector Fallback**  
   Each agent maintains a short-term context window (conversation history, task state). When the window exceeds a configurable threshold, the system automatically chunks and indexes older data into a local vector store (e.g., FAISS or Chroma). This prevents token overflow in LLM calls while preserving long-term recall—no external DB required.

3. **Self-Healing Execution Engine**  
   If an agent fails (timeout, API error, malformed output), the engine doesn't just crash. It triggers a retry policy with exponential backoff, and if the failure persists, it routes the task to a fallback agent or replans the workflow using a recovery prompt. This is critical for production deployments where uptime matters.

### How to Use

1. **Define Your Agent Graph**  
   Create a `agents.yaml` file that declares each agent’s role, input schema, output schema, and dependencies. For example:
   ```yaml
   agents:
     - name: researcher
       role: "Search and summarize web content"
       input_type: "query"
       output_type: "summary"
       depends_on: []
     - name: writer
       role: "Generate blog post from summary"
       input_type: "summary"
       output_type: "markdown"
       depends_on: ["researcher"]
   ```
   Then run `agents run --config agents.yaml` to validate the graph and start execution.

2. **Implement Agent Handlers**  
   Each agent requires a Python function (or async coroutine) decorated with `@agent.handler`. The handler receives a `Context` object containing the input payload, previous agent outputs, and a `memory` client for vector queries. Example:
   ```python
   from agents import agent, Context

   @agent.handler
   async def researcher_handler(ctx: Context):
       query = ctx.input["query"]
       # Call your search API or LLM here
       result = await search_web(query)
       ctx.memory.store(query, result)  # auto-indexed
       return {"summary": result}
   ```
   The engine will automatically inject the correct dependencies from the graph.

3. **Monitor and Recover**  
   Use the built-in CLI dashboard: `agents monitor`. It shows real-time agent status, latency histograms, and failure logs. If an agent enters a `FAILED` state, you can manually invoke a recovery with `agents recover --agent researcher --strategy replan`. The engine will re-run the failed agent with a modified prompt that includes the error context.

### Best Practices

- **Use Idempotent Handlers**  
  Always design your agent handlers to be idempotent. Because the self-healing engine may retry failed tasks, your handler should produce the same output for the same input (even if it calls external APIs). Use `ctx.memory` to cache intermediate results and avoid duplicate side effects.

- **Set Explicit Timeout Budgets**  
  Each agent can declare a `timeout_seconds` in the YAML config. For example, a web-scraping agent might have 30s, while a local LLM call might have 120s. This prevents a single slow agent from blocking the entire DAG. The engine will kill the agent after the timeout and trigger the fallback path.
