## Contenido Detallado

### ¿Qué es agents?

**agents** es un entorno de ejecución y orquestación de agentes inteligentes que permite a equipos de ingeniería definir, desplegar y gobernar flujos de trabajo autónomos basados en modelos de lenguaje. A diferencia de soluciones monolíticas, agents abstrae la complejidad de la comunicación entre agentes, la gestión de memoria contextual y el manejo de errores, ofreciendo una API unificada para integrar agentes con servicios externos, bases de datos vectoriales y sistemas de colas de mensajes.

### Características Principales (Hardcore)

1. **Orquestación Reactiva con State Machine Distribuida**  
   Cada agente opera bajo una máquina de estados finitos (FSM) que gestiona transiciones entre estados como `idle`, `processing`, `waiting_for_input`, `error_recovery`. Esto permite control granular sobre ciclos de vida, reintentos automáticos con backoff exponencial y sincronización de estado entre réplicas mediante un almacén Redis compartido.

2. **Memoria Contextual Jerárquica con Evicción Adaptativa**  
   Implementa un sistema de memoria de dos niveles: una memoria de trabajo (buffer LRU con límite de tokens) y una memoria a largo plazo (almacenada en PostgreSQL con embeddings FAISS). La evicción se decide mediante un modelo de scoring basado en relevancia temporal y frecuencia de acceso, asegurando que los agentes no pierdan contexto crítico incluso en sesiones prolongadas.

3. **Planificador de Tareas con Priorización Dinámica**  
   Incluye un planificador interno que asigna tareas a agentes según una cola de prioridad ponderada por SLA, carga actual y latencia de respuesta. Soporta cancelación de tareas en vuelo, reencolado automático ante fallos y balanceo de carga entre agentes heterogéneos (diferentes modelos o capacidades).

### Cómo Usar

1. **Configurar un Agente con Políticas de Reintento y Timeout**  
   Crea un archivo `agent.yaml` en la raíz del proyecto. Define el modelo base, la política de reintentos (máximo 3 intentos con backoff exponencial de 2s, 4s, 8s) y un timeout de 30 segundos por llamada. Usa el campo `memory_config` para activar la memoria jerárquica con un límite de 4096 tokens en memoria de trabajo y una ventana de retención de 7 días en memoria larga.

   ```yaml
   agent:
     model: gpt-4-turbo
     retry_policy:
       max_attempts: 3
       backoff: exponential
       base_delay: 2
     timeout: 30000
     memory:
       working: 4096
       long_term: true
       retention_days: 7
   ```

2. **Desplegar un Flujo Multi-Agente con Dependencias**  
   Define un pipeline en `pipeline.json` donde un agente de análisis de sentimiento depende de la salida de un agente de extracción de entidades. Usa `depends_on` para establecer el orden y `timeout` global para todo el flujo. Lanza el pipeline con `agents run pipeline --file pipeline.json --async`. El sistema gestiona automáticamente la espera de resultados parciales y la propagación de errores.

   ```json
   {
     "pipeline": [
       { "id": "extractor", "agent": "ner-agent", "depends_on": [] },
       { "id": "sentiment", "agent": "sentiment-agent", "depends_on": ["extractor"] }
     ],
     "timeout": 120000
   }
   ```

3. **Monitorizar y Depurar con Trazas Distribuidas**  
   Activa el modo de trazado con `agents start --trace` para habilitar OpenTelemetry. Cada llamada a un agente genera un span con metadatos de latencia, número de reintentos y tokens consumidos. Usa `agents logs --agent-id <id> --level debug` para inspeccionar el estado interno de la FSM y el contenido de la memoria de trabajo en tiempo real. Exporta las trazas a Jaeger o Grafana Tempo para análisis visual.

### Mejores Prácticas

1. **Segmentar Agentes por Dominio y Carga**  
   En entornos de alta concurrencia (más de 1000 solicitudes/minuto), evita tener un solo agente monolítico. Divide las responsabilidades en agentes especializados (por ejemplo, uno para clasificación, otro para generación de respuestas) y usa el planificador interno para distribuir la carga. Esto reduce la latencia media en un 40% según benchmarks internos y facilita el escalado horizontal independiente.

2. **Implementar Circuit Breaker en Integraciones Externas**  
   Cuando un agente depende de APIs de terceros (bases de datos vectoriales, servicios de traducción), configura un patrón circuit breaker con `failure_threshold: 5` y `reset_timeout: 30000`. Esto evita que fallos en cascada degraden todo el sistema. Usa el hook `on_circuit_open` para redirigir tráfico a un agente de respaldo con un modelo más ligero o una respuesta cacheada.
