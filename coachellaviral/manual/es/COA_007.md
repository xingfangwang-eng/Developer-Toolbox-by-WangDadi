# coachellaviral - COA_007

## Resumen

Motor de análisis viral en tiempo real para festivales musicales, optimizado para detectar picos de engagement en redes sociales durante el Coachella Valley Music and Arts Festival. Procesa 12.000+ menciones por minuto con latencia inferior a 200 ms, integrando datos de TikTok, Instagram y Twitter/X mediante APIs nativas.

## Contenido Detallado

### ¿Qué es coachellaviral?

Plataforma de inteligencia de contenidos que identifica patrones de viralidad en eventos masivos. No es un simple monitor de hashtags: utiliza un modelo de predicción basado en LSTM (Long Short-Term Memory) entrenado con 4,7 millones de publicaciones históricas de Coachella (2017-2024) para anticipar qué momentos escénicos, outfits o interacciones generarán picos de reproducción. Su pipeline de datos consume feeds en streaming desde WebSocket de las plataformas, normaliza metadatos (timestamp, geolocalización, métricas de engagement) y aplica un algoritmo de clustering por similitud semántica (cosine similarity + TF-IDF ponderado) para agrupar tendencias emergentes.

### Características Principales

- **Predicción de Viralidad con Modelo Temporal Híbrido**: Implementa una red neuronal recurrente (LSTM con 3 capas ocultas de 256 unidades) que procesa secuencias de 15 minutos de métricas (tasa de compartición, velocidad de comentarios, ratio de likes por visualización). El modelo se reentrena cada 6 horas con datos frescos mediante un job programado en Airflow. Parámetros clave: `sequence_length=900` (segundos), `forecast_horizon=1800` (30 minutos adelante), `confidence_threshold=0.82`. La salida es un score de viralidad escalado 0-100, almacenado en una columna `viral_score` de PostgreSQL con índice GiST para consultas geoespaciales.

- **Detección de Micro-Momentos en Video en Tiempo Real**: Utiliza un pipeline de procesamiento de video con FFmpeg y OpenCV que extrae keyframes cada 2 segundos de transmisiones en vivo (RTMP). Cada frame se pasa por un modelo YOLOv8 fine-tuned con 50.000 imágenes etiquetadas de escenarios, artistas y accesorios (guitarras, micrófonos, vestimenta fluorescente). Los metadatos resultantes (bounding boxes, etiquetas de objeto, coordenadas de centroide) se envían a una cola RabbitMQ para su agregación. Tiempo medio de procesamiento por frame: 34 ms en GPU NVIDIA T4.

- **Análisis de Sentimiento Multilingüe con Contexto de Festival**: Motor de NLP basado en BERT (modelo `bert-base-multilingual-cased`) fine-tuned con un corpus de 120.000 tweets y comentarios de Coachella etiquetados con 7 emociones (euforia, decepción, sorpresa, aburrimiento, nostalgia, ansiedad, admiración). Se ejecuta en lotes de 512 tokens con solapamiento de 128 tokens para preservar contexto de menciones a artistas. Precisión medida: 91.3% F1-score en validación cruzada. Los resultados se almacenan en una tabla TimescaleDB con particionamiento por hora para consultas rápidas de series temporales.

### Cómo Usar

1. **Configurar el pipeline de ingesta de datos**:
   - Clonar el repositorio y ejecutar `docker-compose up -d` para levantar los servicios (PostgreSQL 15, Redis 7, RabbitMQ 3.12, MinIO para almacenamiento de keyframes).
   - Editar `config/ingestion.yaml`: definir credenciales de API de Twitter (Bearer Token v2), TikTok (Access Key + Secret Key) e Instagram (Graph API Token). Especificar `keywords_filter: ["Coachella", "#Coachella2025", "Empire Polo Club", "Indio"]`. Ajustar `batch_size=1000` y `flush_interval_seconds=30` para controlar la frecuencia de escritura a la base de datos.
   - Iniciar el consumidor de streaming: `python src/stream_consumer.py --platforms twitter,tiktok,instagram --output-queue viral_events`. Verificar logs en `./logs/stream_consumer_*.log` para confirmar conexiones activas.

2. **Entrenar o cargar el modelo de predicción**:
   - Para usar el modelo preentrenado: descargar `lstm_viral_model_v3.h5` desde el bucket de MinIO (`mc cp minio/models/lstm_viral_model_v3.h5 ./models/`).
   - Para reentrenar con datos locales: ejecutar `python src/train_model.py --data-source postgresql --table raw_events --epochs 50 --batch-size 64 --validation-split 0.2`. El script exporta automáticamente el modelo a `models/lstm_viral_model_v4.h5` y registra métricas en MLflow (servicio en `localhost:5000`).
   - Activar predicción continua: `python src/predictor.py --model-path ./models/lstm_viral_model_v3.h5 --redis-host localhost --redis-port 6379 --channel viral_scores`. Las predicciones se publican cada 5 minutos en el canal de Redis.

3. **Desplegar el dashboard de monitoreo**:
   - Construir la imagen Docker del frontend: `docker build -t coachellaviral-dashboard -f dashboard/Dockerfile dashboard/`.
   - Ejecutar el contenedor con montaje de volumen para configuración: `docker run -d -p 8501:8501 -v $(pwd)/config/dashboard.yaml:/app/config.yaml coachellaviral-dashboard`.
   - Acceder a `http://localhost:8501`. En la barra lateral, seleccionar "Vista en Vivo" para ver heatmaps de viralidad por escenario (datos actualizados cada 10 segundos desde WebSocket). Usar el selector de rango temporal para comparar tendencias de ediciones anteriores (datos históricos en `timescaledb.historical_events`).

### Mejores Prácticas

- **Optimización de costos de API**: Configurar rate limiting en `config/ingestion.yaml` con `max_requests_per_minute=450` para Twitter y `max_requests_per_hour=3600` para TikTok. Implementar un buffer en Redis (lista `rate_limit_buffer`) que almacene temporalmente menciones excedentes y las reenvíe en ventanas de baja demanda (entre 2:00 AM y 6:00 AM UTC). Esto reduce costos de suscripción empresarial hasta un 40% sin perder cobertura en horas pico (6:00 PM - 2:00 AM UTC, cuando ocurren los sets principales).

- **Estrategia de almacenamiento en frío**: Los keyframes de video (formato WebP, calidad 85, resolución 640x360) deben migrarse a almacenamiento S3 (o MinIO) después de 48 horas mediante un script cron (`0 3 * * * python src/cold_storage_migrator.py --age-hours 48 --bucket coachella-keyframes-archive`). En PostgreSQL, mantener solo los metadatos (timestamp, bounding boxes, etiquetas) con índices en `(event_timestamp, artist_name)`. Para consultas históricas (más de 30 días), usar vistas materializadas que agreguen conteos por hora, reduciendo el tiempo de respuesta de 12 segundos a 80 ms.
