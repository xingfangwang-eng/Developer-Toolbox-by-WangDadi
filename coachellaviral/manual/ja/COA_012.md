# coachellaviral - COA_012

## 概要

本プロジェクトは、Coachella 音楽フェスティバルを中心としたバイラルマーケティング戦略を支援する、高精度データ解析エンジンである。フェスティバル期間中のソーシャルメディアトレンド、アーティストエンゲージメント、会場内のリアルタイム人流データを統合し、マーケティング施策のROIを最大化する。

## 詳細な内容

### coachellaviral とは

フェスティバルマーケティングに特化したプロフェッショナルツールであり、複数のデータソース（Twitter API v2、Instagram Graph API、Google Trends、会場センサーデータ）をミリ秒単位でクローリング・集約。フェスティバル運営者やブランドマーケターが、バイラルコンテンツの発生を予測し、最適なタイミングでプロモーションを投入するための意思決定を支援する。

### コア機能

#### 1. リアルタイムバイラルスコアリングエンジン

- **パラメータ**: `viral_score`（0.0〜100.0）、`engagement_rate`（インプレッションあたりのインタラクション率）、`velocity`（1時間あたりの投稿増加速度）
- **技術仕様**: PostgreSQL + Redis によるハイブリッドキャッシュ層を介し、5秒以内にスコアを更新。時系列DB（TimescaleDB）で過去7日間のスコア推移を保持。
- **出力**: 各アーティストやハッシュタグのバイラルスコアをダッシュボードに可視化。閾値（デフォルト75.0）を超えた場合、Slack Webhook で即時通知。

#### 2. 会場内ヒートマップ生成モジュール

- **パラメータ**: `lat` / `lng`（会場内グリッド座標）、`density`（1m²あたりの滞在人数）、`timestamp`（UNIX エポックミリ秒）
- **技術仕様**: 会場に設置された Bluetooth Low Energy (BLE) ビーコンとスマートフォンアプリの位置情報を連携。PostGIS で空間インデックスを構築し、1分単位でヒートマップを更新。
- **出力**: ステージ間の移動パターンや滞留時間を可視化。特定エリアの混雑が閾値（例: 4人/m²）を超えた場合、運営スタッフに警報を発報。

#### 3. インフルエンサーエンゲージメント予測モデル

- **パラメータ**: `influencer_id`（Instagram/Twitter のアカウントID）、`post_frequency`（日間投稿数）、`sentiment_score`（-1.0〜1.0）
- **技術仕様**: TensorFlow Lite による軽量ニューラルネットワークモデルをサーバーサイドで実行。学習データは過去3年分の Coachella 関連投稿（約500万件）を使用。推論時間は平均120ミリ秒。
- **出力**: 各インフルエンサーが次にバイラル投稿を行う確率（%表示）と、最適な投稿時間帯（UTC ベース）を提案。

### 使い方

#### ステップ 1: データソースの接続設定

```bash
# 環境変数を設定
export COACHELLA_API_KEY="your_api_key_here"
export TWITTER_BEARER_TOKEN="your_bearer_token"
export INSTAGRAM_ACCESS_TOKEN="your_instagram_token"
export POSTGRESQL_DSN="postgresql://user:pass@localhost:5432/coachellaviral"

# データパイプラインを起動
./coachellaviral pipeline start --interval 300s
```

- `--interval` はクローリング間隔（秒単位）。デフォルトは300秒（5分）。
- 接続テストは `./coachellaviral pipeline test` で実行可能。全データソースの死活確認を行い、結果をJSONで出力。

#### ステップ 2: バイラルスコアの閾値設定とアラート作成

```bash
# 閾値を75.0に設定し、Slack Webhook URL を登録
./coachellaviral alert create \
  --metric viral_score \
  --threshold 75.0 \
  --comparator greater_than \
  --webhook_url "https://hooks.slack.com/services/T00/B00/xxxxx" \
  --cooldown 600

# 現在のアラート一覧を確認
./coachellaviral alert list --format table
```

- `--cooldown` は再通知までのクールダウン時間（秒）。600秒（10分）に設定することで、過剰な通知を防止。
- アラートは `viral_score` の他に `engagement_rate` や `density` にも設定可能。

#### ステップ 3: ヒートマップのエクスポートと可視化

```bash
# 直近1時間のヒートマップデータを GeoJSON 形式でエクスポート
./coachellaviral heatmap export \
  --start_time $(date -u -d '1 hour ago' +%s)000 \
  --end_time $(date -u +%s)000 \
  --format geojson \
  --output ./output/heatmap_$(date +%Y%m%d_%H%M%S).geojson

# ダッシュボードサーバーを起動（ポート8080）
./coachellaviral dashboard start --port 8080 --host 0.0.0.0
```

- エクスポートされた GeoJSON は QGIS や Leaflet.js でそのまま読み込み可能。
- ダッシュボードでは、バイラルスコア、ヒートマップ、インフルエンサー予測がリアルタイムで更新される。

### ベストプラクティス

#### 1. データソースの冗長化とフェイルオーバー設計

Twitter API や Instagram API はレート制限や障害が発生しやすい。coachellaviral では、主要なデータソースに対して少なくとも2つの代替ソース（例: Twitter の代わりに Mastodon パブリックタイムライン、Instagram の代わりに TikTok API）を設定し、自動フェイルオーバーを有効にすることを推奨する。設定は `config/pipeline.yml` の `fallback_sources` セクションで定義可能。

#### 2. プライバシーバイデザインに基づくデータ保持ポリシー

会場内の位置情報データ（BLE ビーコン由来）は、GDPR および CCPA に準拠するため、個人を特定できない形に匿名化した上で、最大72時間のみ保持する。保持期間を超えたデータは、cron ジョブで自動削除される。設定は `config/retention.yml` で管理し、監査ログは PostgreSQL の `audit_logs` テーブルに保存される。
