# coachellaviral - COA_029

## Übersicht

Dieses Repository enthält die vollständige Architektur und Implementierung des **coachellaviral**-Frameworks – einer hochperformanten, ereignisgesteuerten Plattform zur Erkennung, Analyse und Verstärkung viraler Content-Muster im Kontext des Coachella Music Festivals. Die Engine verarbeitet Echtzeitdaten aus sozialen Netzwerken, Audio-Fingerprints und geolokalisierte Metriken, um Mikro-Trends mit einer Latenz von unter 50 Millisekunden zu identifizieren. ## Detaillierter Inhalt

### Was ist coachellaviral?

coachellaviral ist ein **verteiltes, KI-gestütztes Observability- und Orchestrierungssystem**, das speziell für die hyperdynamische Umgebung von Musikfestivals entwickelt wurde. Es aggregiert Multi-Modale Datenströme (Video-Frames, Audio-Spektrogramme, Hashtag-Graphen, Ticket- und WLAN-Hotspot-Dichte) und führt eine **prädiktive Virality-Scoring-Engine** aus, die auf einem Transformer-basierten Modell mit 12 Aufmerksamkeitsköpfen und einer Eingabedimension von 768 Embedding-Vektoren basiert. Die Architektur nutzt Apache Kafka für die Stream-Verarbeitung und PostgreSQL mit der TimescaleDB-Erweiterung für die persistente Speicherung von Zeitreihen.

### Kernfunktionen

1. **Multi-Modale Echtzeit-Virality-Detektion mit Temporal Fusion Transformer (TFT)**
   - **Parameter:** `window_size_ms = 5000`, `prediction_horizon = 3` (3 Intervalle à 5 Sekunden), `attention_head_count = 12`.
   - **Technische Metrik:** Die Engine verarbeitet bis zu 20.000 Events pro Sekunde (EPS) pro Node. Die Inferenzzeit des TFT-Modells beträgt < 45 ms auf einer NVIDIA A100 GPU.
   - **Funktion:** Fusioniert visuelle Features (ResNet-50-Embeddings), Audio-Features (MFCCs mit 40 Koeffizienten) und soziale Graphen (PageRank-basierte Influencer-Gewichtung) zu einem einzigen, zeitgewichteten Virality-Score. Der Score wird in einer PostgreSQL-Tabelle mit der Struktur `(event_id UUID, timestamp TIMESTAMPTZ, venue_zone VARCHAR(10), virality_score FLOAT8, confidence_interval FLOAT8[])` persistiert.

2. **Geolokalisierte Heatmap-Generierung mit räumlich-zeitlicher Clusterung (DBSCAN + ST-DBSCAN)**
   - **Parameter:** `eps_spatial = 50` (Meter), `eps_temporal = 300` (Sekunden), `min_samples = 5`.
   - **Technische Metrik:** Die Cluster-Engine reduziert Rauschen um 92 % im Vergleich zu reinen Schwellwertverfahren. Die Kachelgenerierung für die Heatmap (WebMercator-Projektion, Zoom-Level 15-18) erfolgt in < 200 ms pro Cluster.
   - **Funktion:** Aggregiert GPS-Koordinaten aus Instagram-Posts, Twitter-Geotags und WLAN-AP-Logs. Identifiziert „Viral Hotspots“ (z. B. einen bestimmten Bereich der Sahara-Tent-Bühne) und löst automatisch eine Caching-Präfeching-Strategie für das Content-Delivery-Netzwerk aus.

3. **Automatisiertes Viral-Content-Republishing mit semantischer Duplikaterkennung (SimHash + MinHash LSH)**
   - **Parameter:** `hash_bits = 64`, `bands = 4`, `rows_per_band = 16`, `jaccard_threshold = 0.85`.
   - **Technische Metrik:** Die Duplikaterkennung erreicht eine Precision von 0.97 und eine Recall-Rate von 0.94 bei einem Durchsatz von 5.000 Posts pro Sekunde.
   - **Funktion:** Erkennt visuell und textuell ähnliche Inhalte (z. B. denselben Livestream-Clip, der von 100 verschiedenen Accounts gepostet wird). Der „Viral Amplifier“-Service erstellt dann automatisch einen gebündelten, attributierten Repost mit einem einzigartigen Tracking-Parameter (`?utm_source=coachellaviral&campaign_id={cluster_id}`) und verteilt ihn über einen Facebook- und TikTok-Webhook.

### So verwenden Sie es

1. **Initialisieren der Datenpipeline und Starten des Stream-Processors**
   ```bash
   # 1. Starten des Kafka-Clusters (3 Broker, Replication Factor 3)
   docker-compose -f infrastructure/kafka/docker-compose.yml up -d
   
   # 2. Erstellen der erforderlichen Topics
   kafka-topics.sh --bootstrap-server localhost:9092 --create --topic raw.social.feeds --partitions 12 --replication-factor 3
   kafka-topics.sh --bootstrap-server localhost:9092 --create --topic processed.virality.scores --partitions 6 --replication-factor 3
   
   # 3. Starten des TFT-Inferenz-Services (GPU-Modus)
   python3 -m coachellaviral.services.inference_service \
       --model-path /models/tft_virality_v2.onnx \
       --kafka-bootstrap localhost:9092 \
       --input-topic raw.social.feeds \
       --output-topic processed.virality.scores \
       --batch-size 64 \
       --device cuda:0
   ```

2. **Konfigurieren und Ausführen der räumlich-zeitlichen Clusterung**
   ```python
   # Beispielskript zur Konfiguration der ST-DBSCAN-Engine
   from coachellaviral.analytics.spatial_clusterer import SpatialTemporalClusterer

   clusterer = SpatialTemporalClusterer(
       eps_spatial=50.0,        # Meter
       eps_temporal=300.0,      # Sekunden
       min_samples=5,
       metric='haversine'       # Für GPS-Koordinaten erforderlich
   )

   # Stream von Kafka-Topic lesen und Cluster berechnen
   for batch in kafka_consumer.poll(timeout_ms=1000):
       geo_events = [extract_geojson(msg) for msg in batch.value()]
       clusters = clusterer.fit_predict(geo_events)
       persist_clusters_to_postgres(clusters, connection_string="postgresql://user:pass@timescaledb:5432/coachellaviral")
   ```

3. **Aktivieren des automatischen Viral-Republishing mit Duplikaterkennung**
   ```bash
   # Aktivieren des Republishing-Services mit benutzerdefinierten Webhook-Endpunkten
   curl -X POST http://localhost:8080/api/v1/amplifier/start \
       -H "Content-Type: application/json" \
       -d '{
           "platforms": ["facebook", "tiktok"],
           "webhook_url_facebook": "https://graph.facebook.com/v18.0/me/feed",
           "webhook_url_tiktok": "https://open-api.tiktok.com/share/video/upload/",
           "duplicate_threshold": 0.85,
           "campaign_tag": "coachella2024_amplifier"
       }'
   
   # Überwachen der Service-Logs
   tail -f /var/log/coachellaviral/amplifier.log | grep -E "PUBLISHED|DUPLICATE_SKIPPED|ERROR"
   ```
