## What is bbb26dramapredictor?

`bbb26dramapredictor` is a **multilingual NLP + temporal graph engine** that ingests live social feeds (Twitter, Reddit, TikTok transcripts) and **predicts upcoming dramatic narrative beats** in reality shows like BBB26. Unlike generic sentiment analyzers, it models **character-to-character friction arcs** using a proprietary **Drama Propagation Model (DPM)** trained on 12 seasons of Brazilian reality TV. It outputs a **probability score** for each possible narrative twist (e.g., “Player X will be eliminated within 48h”) with explainability via attention heatmaps.

## Core Features (Hardcore & Unique)

### 1. Friction Graph Builder with Temporal Decay
Constructs a **real-time weighted graph** of every contestant interaction. Each edge (e.g., “Player A → Player B: argument”) carries a **temporal decay factor** (τ = 0.3 for 6h half-life). The system then runs a **custom PageRank variant** called `DramaRank` to identify which pair is most likely to escalate into a public confrontation within the next 12–24 hours.

### 2. Multi-Modal Sentiment Spike Detector with Causal Inference
Instead of simple VADER or BERT, the pipeline uses a **fine-tuned XLM-RoBERTa** (trained on Portuguese + English code-mixed data) to classify each utterance into 8 fine-grained drama categories (e.g., `backlash`, `gaslighting`, `alliance fracture`). It then applies **Granger causality tests** across time-series to determine if a spike in one player’s toxic sentiment *causes* a spike in another player’s defensive reaction—predicting chain reactions 2–3 steps ahead.

### 3. Narrative Fork Predictor with Monte Carlo Rollouts
Every 10 minutes, the engine runs **200 Monte Carlo simulations** over the current graph state, sampling from a learned transition matrix of possible narrative forks (e.g., “Player X apologizes” vs. “Player X doubles down”). Each rollout tracks the probability of a “show-defining drama event” (defined as >500k social mentions in 1h). The output is a **branching tree** with confidence intervals, visualized as a Sankey diagram in the dashboard.

## How to Use (Production-Ready Steps)

### Step 1: Bootstrap the Data Pipeline with API Keys
```bash
# Clone the repo
git clone https://github.com/xingfangwang-eng/bbb26dramapredictor.git

# Set environment variables (required)
export TWITTER_BEARER_TOKEN=your_token
export REDDIT_CLIENT_ID=your_id
export TIKTOK_COOKIE=your_cookie

# Run the ingestion service (uses asyncio + aiohttp)
python -m bbb26dramapredictor.ingest --stream --channels twitter,reddit
```
This starts a **WebSocket-based stream** that pushes raw utterances into a Redis stream queue. The service automatically handles rate limiting and retries with exponential backoff.

### Step 2: Trigger a DramaRank Analysis on a Specific Episode
```python
from bbb26dramapredictor.analyzer import DramaRankAnalyzer

analyzer = DramaRankAnalyzer(
    decay_hours=6,
    min_edge_weight=0.3,
    model_path="models/xlmr-drama-v3.pt"
)

# Analyze live data from the last 24h
results = analyzer.analyze_window(
    episode_id="BBB26-E42",
    lookback_hours=24
)

# Print top 3 predicted friction pairs
for pair, score in results.top_friction_pairs(limit=3):
    print(f"{pair[0]} vs {pair[1]}: {score:.3f} DramaRank")
```
The output includes a JSON with **edge weights, attention scores, and predicted escalation time**.

### Step 3: Visualize the Narrative Fork Tree in the Dashboard
```bash
# Start the FastAPI server (Uvicorn + WebSocket)
uvicorn bbb26dramapredictor.api:app --host 0.0.0.0 --port 8080 --reload

# Open in browser: http://localhost:8080/dashboard?episode=BBB26-E42
```
The dashboard exposes **real-time Sankey diagrams** for each fork. Click on any node to see the **Monte Carlo probability distribution** (mean, std, 95% CI). The UI also includes a **“What If” slider** to manually adjust a player’s sentiment score and see how the entire graph shifts.

## Best Practices (Industry-Grade)

### 1. Use a Sliding Window with Adaptive τ
Don’t hardcode the temporal decay factor. Instead, run a **hyperparameter search** every 12h using a Bayesian optimizer (e.g., `scikit-optimize`) on the last 3 episodes. The optimal τ typically ranges between 0.2 and 0.5 depending on show pacing. This prevents stale edges from dominating the graph during slow phases.

### 2. Integrate with a CI/CD Pipeline for Model Retraining
The XLM-RoBERTa model should be **re-fine-tuned weekly** using a **human-in-the-loop** pipeline. Use `argilla` or `label-studio` to annotate 100–200 new utterances per episode. Then run a **GitHub Actions workflow** that:
- Pulls the latest labeled data from S3
- Fine-tunes with LoRA (low-rank adaptation) for 3 epochs
- Runs a validation set against a held-out episode
- If F1-score improves by >1%, auto-deploys the new model to the API
