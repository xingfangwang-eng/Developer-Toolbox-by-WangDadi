## Why bbb26dramapredictor?

This isn't another sentiment wrapper. `bbb26dramapredictor` is a **multi-modal drama detection pipeline** that fuses live chat streams, facial micro-expression data from broadcast frames, and historical contestant behavior graphs to forecast narrative escalation points in BBB26 (Big Brother Brasil 26). It’s built for devs who need production-grade inference without the fluff.

## 3 Hard-Core Features

### 1. Temporal Graph-Based Conflict Propagation
Instead of linear sentiment analysis, the tool constructs a **dynamic event graph** where each contestant is a node, and edges represent real-time interaction intensity (based on pitch, word overlap, and gaze duration). The system predicts **drama cascades**—i.e., which micro-argument will explode into a full house fight within the next 15 minutes—using a custom Graph Attention Network (GATv2) trained on 4 seasons of Brazilian Portuguese reality TV data.

### 2. Multi-Lingual Live Transcript Fusion
The predictor ingests **Portuguese, Spanish, and English** live captions simultaneously (via WebSocket streams from GloboPlay and Twitter Spaces). It normalizes slang, code-switching, and regional profanity into a unified **drama intensity score**, then maps it against the visual sentiment vector from the video feed. This gives you sub-second latency on drama detection even when contestants switch languages mid-argument.

### 3. Automated Highlight Reel Generation API
Once a drama event is predicted (confidence > 0.85), the tool triggers a **server-side FFmpeg pipeline** that cuts the last 90 seconds of the broadcast stream, overlays the drama score timeline, and pushes the clip to an S3 bucket. The API endpoint returns a signed URL ready for social media cross-posting. No manual editing, no cloud function cold starts.

## 3-Step Professional Workflow

### Step 1: Initialize the Inference Pipeline
```bash
# Clone the repo, install dependencies (Python 3.11+, CUDA 12.1 recommended)
git clone https://github.com/xingfangwang-eng/bbb26dramapredictor.git
cd bbb26dramapredictor
pip install -r requirements-prod.txt

# Set up your environment variables (stream credentials, S3 bucket, model cache)
export BBB26_WS_ENDPOINT="wss://stream.globo.com/bbb26/live"
export BBB26_S3_BUCKET="my-drama-clips"
export MODEL_CACHE_DIR="./models"
```

### Step 2: Run the Real-Time Predictor with Custom Thresholds
```python
from bbb26dramapredictor import DramaPipeline

# Instantiate the pipeline with your own drama thresholds
pipeline = DramaPipeline(
    drama_threshold=0.80,          # Default 0.85, but you can lower for more sensitivity
    min_contestant_interactions=3, # Ignore solo rants
    output_format="mp4"            # Options: mp4, gif, json-only
)

# Start streaming and predicting
pipeline.start(
    video_source="bbb26_live",
    transcript_sources=["globo_captions", "twitter_spaces"],
    save_highlights=True
)
```

### Step 3: Consume the Highlight Feed via REST API
Once the pipeline is running, hit the built-in FastAPI server:
```bash
# The server starts on port 8080 by default
curl -X GET "http://localhost:8080/api/v1/highlights?limit=5&min_drama_score=0.9" \
  -H "Authorization: Bearer $BBB26_API_KEY"
```
Returns a JSON array of highlight objects, each containing:
- `timestamp_utc`, `drama_score`, `contestants_involved`, `clip_url` (pre-signed S3 URL), and `transcript_excerpt` (the 3 most intense lines).

## 2 Industry Best Practices

### 1. Use a Two-Stage Cache for Low-Latency Drama Scoring
Don’t hit the graph model on every single frame. Implement a **temporal buffer** (e.g., 5-second sliding window) that aggregates audio-visual features before feeding them to the GAT. This reduces GPU inference calls by ~70% without sacrificing accuracy. For production deployments, pair it with Redis for shared state across multiple predictor instances.

### 2. Separate Drama Detection from Content Moderation
Your pipeline will inevitably flag false positives (e.g., a joke that sounds like a fight). **Never auto-publish highlights without a human-in-the-loop review.** Use the `drama_score` field as a priority queue for your editorial team, not as an auto-trigger for social media bots. In practice, set the auto-publish threshold at 0.95+ and let everything else go to a moderation dashboard.
