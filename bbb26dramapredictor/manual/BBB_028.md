# bbb26dramapredictor - BBB_028

This is a production-grade tool designed for developers, data analysts, and reality show researchers who need to quantify, predict, and visualize narrative volatility in Big Brother Brasil 26 (BBB26). Unlike generic sentiment trackers, this engine ingests multi-modal streams (Twitter, Reddit, live feed transcripts) and outputs structured drama scores with causal attribution.

## Core Features

### 1. Multi-Modal Drama Vectorization
We fuse NLP embeddings from Portuguese-language social media with audio prosody features extracted from live feed clips. The resulting **Drama Vector** encodes both *what* is being said and *how* it is being said (pitch, cadence, emotional arousal). This bypasses the limitations of keyword-based filters that miss sarcasm or micro-conflicts.

### 2. Causal Conflict Graph (CCG)
Instead of flat sentiment trends, we generate a directed acyclic graph of interactions: `[Participant A] -> [Action] -> [Participant B] -> [Escalation Score]`. Each edge is weighted by a proprietary **Friction Coefficient** derived from reply velocity, emoji polarity, and moderator intervention patterns. You can query the graph to find the *root cause* of a drama spike within the last 15 minutes.

### 3. Predictive Heatmap API
Our LSTM-based model, trained on 8 seasons of BBB data, outputs a 6-hour forecast of drama intensity per participant. The heatmap is exposed as a REST endpoint (`/api/v2/heatmap`) with confidence intervals. This allows you to schedule alerts or trigger automated content moderation before conflicts peak.

## How to Use

### Step 1: Configure Stream Ingestion & API Key
Clone the repo and set up your `.env` file with Twitter API v2 credentials and Reddit OAuth. Run `python3 ingest.py --source twitter --stream bbb26` to start a live stream. The engine will automatically deduplicate retweets and filter out bot accounts using a pre-trained BotScore model. Monitor the logs for `[INGEST]` tags to confirm data flow.

### Step 2: Query the Drama Vector Store
Once data is streaming, use the CLI tool to retrieve a participant’s drama vector:  
`./bbb26dramapredictor query --participant "Alane" --timeframe 15m`  
This returns a JSON object with `drama_score` (0-100), `top_contributing_edges` (from the CCG), and `forecast` (next 6 hours). For programmatic access, hit the GraphQL endpoint at `localhost:4000/graphql` with a query like:  
```graphql
{
  participant(name: "Alane") {
    dramaScore
    causalGraph {
      edges { source target frictionCoefficient }
    }
  }
}
```

### Step 3: Evaluate Model Drift & Retrain
Run `python3 evaluate.py --test-set data/latest_episode.json` to compare predicted drama scores against human-annotated labels (we release a weekly test set in `data/`). If the Mean Absolute Error exceeds 12.5, trigger a retraining job: `python3 train.py --epochs 50 --learning-rate 0.001`. The new model artifact is saved to `models/` and automatically loaded on the next API restart.

## Best Practices

### 1. Decouple Ingestion from Inference
Run the ingestion pipeline on a separate worker (e.g., using Celery or a dedicated Kubernetes pod). If the Twitter stream rate-limits, the inference engine should still serve cached CCG data. We recommend a Redis-backed buffer with a TTL of 30 minutes for real-time queries.

### 2. Version Your Drama Vectors
When you update the embedding model (e.g., from BERTimbau to a fine-tuned XLM-R), tag each vector with a `model_version` field. This allows you to roll back predictions if a new model introduces drift. Store versioned vectors in a time-series database like InfluxDB for longitudinal analysis.
