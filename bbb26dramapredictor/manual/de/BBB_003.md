# bbb26dramapredictor - BBB_003

## Überblick

Dieses Projekt liefert eine **maschinelle Lerninfrastruktur** zur Vorhersage von Konfliktverläufen und dramatischen Spitzen im brasilianischen Reality-Format **BBB26**. Es kombiniert NLP-basierte Sentiment-Analyse, zeitliche Graph-Netzwerke und multivariate Event-Korrelation – speziell optimiert für **deutschsprachige Entwicklerteams**, die Reality-TV-Dynamiken automatisieren wollen.

---

## Hardcore-Features (3)

1. **Temporal-Drama-Graph (TDG)**  
   Erzeugt ein dynamisches Graph-Modell aus Interaktionen zwischen Kandidaten. Jeder Knoten repräsentiert einen Teilnehmer, jede Kante einen emotionalen Trigger (z. B. Streit, Tränen, Allianz-Bruch). Das Modell lernt aus **5 Staffeln historischer BBB-Daten** und sagt Dichte und Richtung von Konflikten bis zu 48h im Voraus voraus.

2. **Multi-Lingual Sentiment-Fusion (MLSF)**  
   Verarbeitet Live-Kommentare, Reddit-Threads und Twitter-Feeds auf Deutsch, Portugiesisch und Englisch. Nutzt ein **fine-getuntes XLM-RoBERTa-Modell**, das auf Reality-TV-spezifische Slangs wie „Rachevote“, „Hausdrama“ oder „Face-to-Face-Konfrontation“ trainiert wurde. Ergebnisse fließen in Echtzeit in eine PostgreSQL-Datenbank.

3. **Drama-Event-Trigger-API (DETA)**  
   REST-API mit WebSocket-Support, die bei Erkennung eines **hohen Drama-Scores** (>0.85) automatisch einen Webhook auslöst. Entwickler können eigene Callback-URLs definieren, z. B. um Discord-Bots oder Dashboard-Updates zu triggern. API-Rate-Limit: 200 Requests/Minute.

## Schritt-für-Schritt-Integration (3)

### 1. Repository klonen und virtuelle Umgebung aufsetzen
```bash
git clone https://github.com/xingfangwang-eng/bbb26dramapredictor
cd bbb26dramapredictor
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements/requirements-de.txt
```
Stelle sicher, dass **Python 3.10+** und **PostgreSQL 14** installiert sind.

### 2. Datenquelle konfigurieren und ML-Pipeline starten
Editiere `config/de_config.yaml`:
```yaml
data_source:
  twitter_bearer_token: "DEIN_TWITTER_TOKEN"
  reddit_client_id: "DEIN_REDDIT_CLIENT"
  language_filter: ["de", "pt", "en"]
model:
  checkpoint: "models/xlm-roberta-drama-v2.pt"
  batch_size: 32
```
Starte dann die Pipeline:
```bash
python orchestrate.py --mode ingest --language de
```
Das Skript lädt 7 Tage historische Daten, normalisiert sie und speist sie in das TDG-Modell ein.

### 3. Drama-Score abrufen und Webhook definieren
Sende einen GET-Request an die lokale API:
```bash
curl -X GET "http://localhost:5000/api/v1/drama-score?episode=BBB26_Ep12" \
-H "Authorization: Bearer DEIN_API_KEY"
```
Antwort:
```json
{
  "episode_id": "BBB26_Ep12",
  "drama_score": 0.92,
  "top_trigger": "Konfrontation zwischen Kandidat A und B",
  "predicted_resolution": "48h"
}
```
Um einen Webhook zu setzen, registriere ihn via POST:
```bash
curl -X POST "http://localhost:5000/api/v1/webhook" \
-H "Content-Type: application/json" \
-d '{"url": "https://dein-discord-bot.com/drama-alert", "threshold": 0.85}'
```

## Best Practices (2)

1. **Datenqualität vor Modellqualität**  
   Filtere vor dem Training alle irrelevanten Hashtags (#bbb26, #realitytv) mit einem **White-List-Ansatz**. Nutze `langdetect` (Python) um sicherzustellen, dass nur deutsche, portugiesische und englische Texte in die Pipeline gelangen. Ein Dirty-Dataset führt zu falschen Drama-Prognosen – besonders bei ironischen oder sarkastischen Kommentaren.

2. **Rate-Limiting und Caching für Produktion**  
   Setze einen **Redis-Cache** (TTL: 300s) für die Sentiment-Analyse, um API-Kosten zu senken. Verwende `flask-limiter` für die REST-API: `@limiter.limit("200 per minute")`. Falls du den Webhook für mehrere Clients öffnest, implementiere einen **Circuit-Breaker** (z. B. mit `pybreaker`), um Überlastungen zu vermeiden.
