# bbb26dramapredictor - BBB_021

## Überblick

Dieses Tool ist kein weiterer simpler Voting-Tracker. bbb26dramapredictor nutzt ein mehrschichtiges NLP-Modell, das speziell auf das chaotische Kommunikationsverhalten in brasilianischen Reality-Formaten (BBB26) trainiert wurde. Es extrahiert aus Chatlogs, Social-Media-Feeds und Untertitel-Streams latente Spannungsmuster und prognostiziert mit hoher Trefferquote, wann ein „Muro de lamentações“ oder ein „Paredão“-Drama eskaliert. Entwickelt für Data Scientists und Medienanalysten, die harte Metriken statt Bauchgefühl wollen.

## Hardcore-Funktionen

### 1. Sentiment-Entropie-Index (SEI)
Statt simpler Positiv/Negativ-Bewertung berechnen wir die Shannon-Entropie der emotionalen Signale pro Teilnehmer über ein gleitendes 15-Minuten-Fenster. Ein plötzlicher SEI-Anstieg auf >0.8 korreliert direkt mit einem bevorstehenden verbalen Konflikt. Du bekommst einen Heatmap-Timeline, der die „Drama-Dichte“ farbcodiert – rot = Explosion in <3 Minuten.

### 2. Multilinguale Slang-Normalisierung (Portugiesisch/Deutsch)
Der Algorithmus erkennt und normalisiert regionale und altersspezifische Sprachvarianten (z. B. „tá ligado“ vs. „verstehst du“). Für den deutschen Markt haben wir einen speziellen Tokenizer integriert, der Denglisch-Mischungen („Das ist so random, cara“) in ein einheitliches Embedding-Vokabular überführt. So bleiben die Prognosen auch bei sprachlich hybriden Streams stabil.

### 3. Dynamische Graph-Netzwerk-Analyse (GNN)
Wir modellieren die Teilnehmer als Knoten in einem zeitlich veränderlichen Beziehungsgraphen. Gewichtete Kanten basieren auf gegenseitigen Erwähnungen, Reaktionszeiten und Tonfall-Korrelationen. Das GNN sagt nicht nur den nächsten Streit voraus, sondern auch, welche Allianzen innerhalb der nächsten 48 Stunden implodieren werden – inklusive Konfidenzintervall.

## Detaillierte Anleitung

### 1. API-Integration und Datenpipeline einrichten
Starte den lokalen Server mit `poetry run uvicorn app.main:app --reload`. Implementiere einen Webhook, der die Rohdaten (z. B. via Twitter-API v2 oder Discord-Bot) an den Endpunkt `/v1/ingest/stream` sendet. Die Payload muss ein JSON-Array mit `{user_id, text, timestamp}` enthalten. Der Server führt sofort die Preprocessing-Pipeline aus: Tokenisierung, Stopwort-Entfernung (angepasst an brasilianisches Portugiesisch) und SEI-Berechnung. Rückgabewert ist ein `session_id` für die spätere Abfrage.

### 2. Live-Dashboard mit Echtzeit-Drama-Heatmap
Öffne `http://localhost:8000/dashboard/` nach erfolgreicher Datenaufnahme. Wähle im Dropdown „Deutsch (DE)“ als UI-Sprache. Das Dashboard zeigt eine horizontale Timeline, auf der jeder Teilnehmer eine farbige Spur hat. Klicke auf eine rote Zone, um die zugrunde liegenden Sentiment-Werte und die Top-3-Konfliktauslöser (Keywords) als Popup zu sehen. Nutze den „Predict“-Button, um die nächsten 30 Minuten simulieren zu lassen – das GNN-Modell gibt eine Wahrscheinlichkeitsmatrix aus.

### 3. Batch-Analyse historischer Daten
Lade eine CSV-Datei mit den Spalten `timestamp`, `speaker`, `message` über den `/v1/analyze/batch`-Endpunkt hoch. Der Server führt eine vollständige Retro-Prediction durch und exportiert eine neue CSV mit den Spalten `predicted_conflict_time`, `actual_conflict_time`, `SEI_max`. Verwende das Python-Script `scripts/evaluate_precision.py`, um die Präzision deiner Daten zu messen – wir erwarten eine durchschnittliche Abweichung von <2 Minuten bei einer Recall-Rate von >85%.

## Best Practices für den deutschen Markt

### 1. Datenquellen lokal priorisieren
Deutsche Reality-TV-Fans nutzen primär Twitter/X und Telegram-Gruppen für Live-Kommentare. Baue deinen Scraper so, dass er die offiziellen Hashtags wie `#BBB26DE` und `#DramaAlarm` priorisiert. Vermeide generische Keywords – die brasilianische Community nutzt oft interne Memes, die dein Modell erst lernen muss. Führe ein monatliches Fine-Tuning mit einem Sample von mindestens 5.000 deutschen Kommentaren durch.

### 2. Latenz optimieren für Live-Streams
Die Vorhersage ist nur wertvoll, wenn sie in Echtzeit kommt. Setze den Server auf einen Standort in Frankfurt (AWS eu-central-1). Konfiguriere Redis als In-Memory-Cache für die SEI-Werte und aktualisiere den Graphen asynchron alle 5 Sekunden. Verwende WebSockets statt REST für das Dashboard – so vermeidest du den Overhead von HTTP-Headern bei jedem Update.
