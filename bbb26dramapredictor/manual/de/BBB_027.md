# bbb26dramapredictor - BBB_027

## Überblick

Dieses Repository enthält den vollständigen Stack des **bbb26dramapredictor** – einer auf maschinellem Lernen basierenden Analyse-Pipeline, die speziell für die Vorhersage von Dramabögen, Voting-Ausgängen und sozialen Dynamiken in der 26. Staffel von *Big Brother Brasil* entwickelt wurde. Das Tool aggregiert Echtzeit-Daten aus Social-Media-Feeds, Livestream-Transkripten und historischen Staffelverläufen, um mit hoher Präzision narrative Wendungen zu prognostizieren. Zielgruppe sind Datenjournalisten, Reality-TV-Analysten und Medienstrategen, die narrative Muster algorithmisch erfassen und monetarisieren möchten.

## 3 Hardcore-Features

1. **Multimodale Drama-Korrelation (MDK)**  
   Kombiniert NLP-Sentiment-Analyse aus Twitter/X-Posts (pt-BR) mit Audio-Emotion-Detection aus Livestream-Ausschnitten. Das Modell erkennt Diskrepanzen zwischen verbalem Inhalt und paraverbalen Signalen (Tonlage, Pausen) und markiert diese als potenzielle Konfliktindikatoren mit einer Konfidenzschwelle von ≥ 0,87.

2. **Narrative Fork Prediction (NFP)**  
   Ein auf Transformer-Architekturen basierendes Modell, das staffelübergreifende Plot-Strukturen (z. B. „Allianzzerfall“, „Underdog-Comeback“) lernt und zu jedem Entscheidungszeitpunkt (Voting, Leader-Aufgabe) 3–5 wahrscheinlichste narrative Verzweigungen inklusive Wahrscheinlichkeitsverteilung ausgibt. Die Ausgabe erfolgt als gerichteter Graph im JSON-Format.

3. **Echtzeit-Voting-Odds-Engine**  
   Aggregiert Daten aus 4 verschiedenen Voting-Plattformen (Gshow, UOL, Twitter Polls, Discord) und gleicht sie mit historischen Abstimmungsmustern ab. Die Engine liefert dynamisch aktualisierte Eliminationswahrscheinlichkeiten pro Kandidat, inklusive eines Confidence-Intervalls basierend auf Stichprobengröße und zeitlichem Verfall älterer Daten.

## 3 Detaillierte Anwendungsschritte

### 1. Pipeline initialisieren und Datenquellen konfigurieren

```bash
# Repository klonen und virtuelle Umgebung einrichten
git clone https://github.com/xingfangwang-eng/bbb26dramapredictor.git
cd bbb26dramapredictor
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# API-Keys für Twitter/X (v2), Gshow und Discord in config.yaml eintragen
# Beispiel: twitter_bearer_token, gshow_cookie, discord_bot_token
```

Die Konfiguration erfolgt über eine zentrale `config.yaml`, die Datenquellen, Modellparameter und Caching-Intervalle definiert. Für den Produktivbetrieb empfehle ich die Verwendung von Umgebungsvariablen via `dotenv`.

### 2. Sentiment- und Plot-Twist-Analyse ausführen

```python
from predictor import DramaPipeline

pipeline = DramaPipeline(config_path="config.yaml")
pipeline.run_full_analysis(
    episode="bbb26_ep23",
    include_audio=True,
    n_future_forks=4
)
```

Dieser Befehl startet eine vollständige Analyse der aktuellen Episode: Social-Media-Scraping, Audio-Emotion-Detection (via `librosa` + vortrainiertem `wav2vec2`-Modell) und die NFP-Modellinferenz. Die Ausgabe erfolgt als strukturiertes JSON-Dokument mit Metadaten zu jedem erkannten Dramapunkt.

### 3. Ergebnisse visualisieren und exportieren

```python
from visualization import plot_drama_timeline, export_voting_odds

plot_drama_timeline("output/bbb26_ep23_drama.json", interactive=True)
export_voting_odds("output/bbb26_ep23_odds.json", format="csv")
```

Die Visualisierung erzeugt ein interaktives HTML-Dashboard (via Plotly), das zeitliche Cluster von Dramaspitzen, Sentiment-Verläufe und narrative Forks darstellt. Die Voting-Odds können als CSV für Tabellenkalkulationen oder als JSON für API-Endpunkte exportiert werden.

## 2 Best Practices für Medienanalysten

1. **Kalibrierung der Sentiment-Baseline pro Staffel**  
   Führen Sie vor der ersten Nutzung eine Kalibrierung mit mindestens 3 Episoden der aktuellen Staffel durch, um domänenspezifische Slang-Ausdrücke (z. B. „planta“, “falso”, “planta baixa”) im Sentiment-Lexikon zu verankern. Ohne diese Anpassung sinkt die Präzision der Drama-Detektion um bis zu 22 %, wie interne Tests gezeigt haben.

2. **Datenquellen gewichten und zeitliche Decay-Faktoren setzen**  
   Nicht alle Voting-Plattformen haben die gleiche Repräsentativität. Setzen Sie in der `config.yaml` unter `voting_weights` eine höhere Gewichtung für Gshow (offiziell) und eine niedrigere für Twitter Polls (anfällig für Bot-Aktivität). Der zeitliche Decay sollte für Livestream-Daten auf 15 Minuten gesetzt werden, für historische Staffeldaten auf 7 Tage – sonst verfälschen veraltete Trends die Prognose.
