# bbb26dramapredictor - BBB_026

**bbb26dramapredictor** is a high-frequency sentiment & narrative drift analyzer built specifically for the Brazilian reality show **Big Brother Brasil 26 (BBB26)**. It ingests live social chatter, edit timing, and contestant interaction graphs to predict **drama spikes** before they hit the broadcast.

Unlike generic sentiment dashboards, this tool is tuned for **telenovela-style reality arcs**, with bespoke NLP models trained on past BBB seasons (BBB22–BBB25) and real-time Twitter/X + YouTube chat streams.

---

## Core Features

### 1. Narrative Tension Score (NTS) with Temporal Decay
A proprietary metric that quantifies **conflict probability** between contestants based on:
- Co-occurrence of emotionally charged Portuguese keywords (e.g., *"traição"*, *"falsa"*, *"emboscada"*)
- Rate of change in reply-to-mention ratios on Twitter
- Weighted decay function that penalizes stale drama (older than 4 hours gets 60% less weight)

Outputs a **0–100 score** updated every 90 seconds.

### 2. Edição Inference Engine (EIE)
Reverse-engineers the **Globo editing team** by analyzing:
- Time gaps between live feeds and broadcast segments
- Camera angle shifts (detected via frame diff in official clips)
- Soundtrack mood changes (using audio embedding similarity)

The EIE predicts which **unedited moments** will be highlighted in the next episode, giving you a 3–6 hour lead on major plotlines.

### 3. Voting Behavior Simulator (VBS)
Monte Carlo simulation based on:
- Historical voting patterns from BBB25 (elimination rounds)
- Current hashtag frequency per contestant (#Fica vs #Elimina)
- Poll data from UOL and Gshow (scraped every 15 minutes)

Returns **elimination probability** per contestant with confidence intervals. Perfect for betting or fan engagement strategies.

## How to Use (3-Step Workflow)

### Step 1: Bootstrap the Data Pipeline
```bash
git clone https://github.com/xingfangwang-eng/bbb26dramapredictor.git
cd bbb26dramapredictor
cp .env.example .env
# Fill in your Twitter API Bearer Token and YouTube Data API key
docker-compose up -d
```
This spins up:
- `kafka` for stream ingestion (Twitter + YouTube chat)
- `postgres` for time-series storage
- `fastapi` for REST endpoints

### Step 2: Query the Drama Forecast API
```python
import requests

url = "http://localhost:8000/api/v1/nts/current"
params = {"window_minutes": 30, "min_confidence": 0.75}
response = requests.get(url, params=params)

# Returns a ranked list of drama hotspots
for event in response.json()["events"]:
    print(f"{event['contestants']} | NTS: {event['score']} | Predicted trigger: {event['trigger_phrase']}")
```

### Step 3: Simulate Elimination Outcome
```bash
curl -X POST http://localhost:8000/api/v1/vbs/simulate \
  -H "Content-Type: application/json" \
  -d '{"num_simulations": 10000, "include_historical_bias": true}'
```
Response includes:
- `elimination_probabilities`: dict of contestant → probability
- `confidence_interval_95`: lower/upper bounds
- `most_likely_eviction_order`: ordered list

## Industry Best Practices

### 1. **Tune the Decay Window Per Phase**
During **Prova do Líder** (Leader's Test) weeks, drama cycles are shorter (2–3 hours). Set `NTS_DECAY_HOURS=2` in your `.env`. For **Paredão** (elimination) weeks, extend to 6 hours to capture slow-burn conspiracies. This prevents false positives from transient spam.

### 2. **Cross-Validate with YouTube Chat Emotes**
Twitter data alone is noisy. In the EIE module, enable `YOUTUBE_EMOTE_WEIGHT=0.3` to factor in repeated emotes like `Kappa` (sarcasm) or `WutFace` (shock). This dramatically improves NTS accuracy for non-Twitter users (common in Brazil's older demographic).
