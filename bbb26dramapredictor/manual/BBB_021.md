# bbb26dramapredictor - BBB_021

## Overview

bbb26dramapredictor is a production-grade, multilingual analytics engine that ingests live social chatter, broadcast transcripts, and audience reaction metrics to forecast emotional flashpoints in BBB26. It’s built for data-savvy reality show analysts, social listening teams, and media strategists who need to stay ahead of the narrative curve—not just watch it unfold.

## Three Hardcore Features

### 1. Sentiment Cascading Engine (SCE)
An NLP pipeline that tokenizes real-time tweets, Reddit threads, and YouTube comments in Portuguese (with fallback for Spanish and English). It detects micro-shifts in sentiment polarity (e.g., "neutral → negative" within a 3-minute window) and triggers alerts when a drama threshold is crossed. The engine uses a fine-tuned BERT model on a custom corpus of 200k BBB26-labeled posts.

### 2. Drama Signature Generator
Each episode is reduced to a unique "drama signature"—a vector of 12 dimensions (e.g., conflict intensity, alliance volatility, public empathy score). These signatures are compared across seasons using cosine similarity to surface recurring patterns (e.g., "this fight resembles the Davi vs. Isabelle arc from BBB24"). Output is served as a JSON blob for downstream dashboards or Slack bots.

### 3. Temporal Anomaly Detector
A lightweight ML model (XGBoost + rolling window statistics) that identifies statistical outliers in engagement velocity. If a 5-minute spike in negative mentions exceeds 2.5 standard deviations from the rolling mean, the system auto-generates a "drama event" with a confidence score. This is ideal for triggering automated clips or alerting editorial teams before the hashtag trends.

## How to Use (Professional Workflow)

### Step 1: Configure Data Sources
Edit the `config/drama_sources.yaml` file to bind your API keys for Twitter/X (v2), YouTube Data API, and a WebSocket bridge to a live chat platform (e.g., Twitch or Discord). Each source requires a rate-limit bucket and a language filter (default: `pt-BR`). Run `python setup.py --verify-sources` to test connectivity and validate token quotas.

### Step 2: Run a Historical Calibration
Before live mode, calibrate the model using the `--backtest` flag:  
```bash
python main.py --mode backtest --season bbb25 --output ./calibration_report.json
```
This replays the last 7 days of the previous season, generating drama signatures and comparing them against known events (e.g., eliminations, fights). The report includes precision/recall metrics for the anomaly detector. Tune the `drama_threshold` in `config/model_params.yaml` until you hit >85% recall on labeled events.

### Step 3: Deploy the Live Dashboard
Start the streaming pipeline with:
```bash
docker-compose up -d
```
This launches three containers: a Kafka ingestion layer, the prediction worker (GPU-accelerated if available), and a FastAPI server exposing a `/events` endpoint. The dashboard (built with Streamlit) auto-refreshes every 15 seconds, showing a heatmap of current drama zones and a leaderboard of "most volatile" participants. Use the `--webhook` flag to push predictions to your team’s Slack or Discord channel.

## Two Industry Best Practices

### 1. Layer in Cultural Context
Don’t rely solely on sentiment polarity. Brazilian reality TV often uses irony, sarcasm, and regional slang (e.g., "casa do caralho" or "mimimi") that generic sentiment models misinterpret. Fine-tune your tokenizer on a curated corpus of BBB-specific memes and catchphrases. For multi-language support, use a per-language sentiment model (e.g., `bert-base-portuguese-cased` for PT-BR, `distilbert-base-uncased` for EN) rather than a single multilingual checkpoint.

### 2. Implement a Human-in-the-Loop Feedback Loop
The anomaly detector will generate false positives during low-activity hours (e.g., 3 AM GMT-3). Set up a lightweight moderation queue: push high-confidence events (>90% confidence) directly to alerts, but route medium-confidence events (60–90%) to a human reviewer via a webhook. Collect those reviews as labeled data, then retrain the model weekly with `python main.py --mode retrain`. This reduces alert fatigue and improves precision over time.
