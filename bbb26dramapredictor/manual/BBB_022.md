# bbb26dramapredictor - BBB_022

This is a specialized SaaS toolkit engineered to parse, quantify, and predict conflict escalation patterns inside the **Big Brother Brasil 26** ecosystem. Unlike generic sentiment scrapers, `bbb26dramapredictor` uses multi-modal NLP + behavioral graph analysis to surface actionable drama trajectories before they trend.

## Core Features

### 1. Multi-Modal Drama Signature Extraction
Ingests live chat, edited clips (via whisper transcript), and social media cross-references to build a **drama fingerprint** per contestant. The engine identifies micro-conflict triggers (verbal tone shifts, gaze avoidance, alliance fractures) and assigns a real-time **drama probability score (0–100)**.

### 2. Temporal Conflict Cascade Engine
Trains on past BBB seasons to model how small disagreements escalate into full-blown house splits. The cascade engine outputs a **5-hour drama trajectory map** with confidence intervals, allowing you to see which current conflict is likely to blow up next.

### 3. Contextual Sentiment Drift Detector
Instead of static polarity, this module tracks **sentiment drift** relative to the house's current power dynamics. It flags when a contestant’s public perception diverges from their in-game behavior by more than 2 standard deviations—a strong indicator of impending confrontation or strategic betrayal.

## How to Use

### Step 1: Bootstrap the Drama Pipeline
Clone the repo and run the bootstrap script to set up the live ingestion stack. This spins up a Redis-backed stream that connects to the BBB26 live feed API (via unofficial WebSocket proxy) and the Twitter/X filtered stream for real-time mentions.

```bash
git clone https://github.com/xingfangwang-eng/bbb26dramapredictor
cd bbb26dramapredictor
./bootstrap.sh --live-feed-endpoint wss://bbb26-feeds.example.com --twitter-bearer-token YOUR_TOKEN
```

### Step 2: Calibrate the Conflict Thresholds
Edit the configuration file `drama_config.yaml` to set your baseline thresholds. For a standard season, start with:

```yaml
drama:
  escalation_threshold: 72    # trigger cascade analysis at 72% drama probability
  sentiment_drift_window: 15  # minutes
  cascade_depth: 3            # how many steps of escalation to model
```

Then run the calibration suite to validate against last season's known conflicts:

```bash
python calibrate.py --season bbb25 --output-calibration-report
```

### Step 3: Deploy the Forecasting Dashboard
Launch the real-time dashboard using the built-in Streamlit app. This exposes the drama trajectory map, sentiment drift overlays, and a **Conflict Heatmap** showing which dyads are most likely to clash in the next 4 hours.

```bash
streamlit run dashboard/app.py --server.port 8501 -- --live-mode
```

Navigate to `http://localhost:8501` and toggle the **"Enable Cascade Alerts"** switch to receive push notifications when a conflict crosses your threshold.

## Best Practices

### 1. Use Dynamic Thresholding Per Phase
Don't use a static drama threshold across the entire season. Early in the game (weeks 1–3), conflicts are often manufactured; set a higher threshold (e.g., 80) to avoid false positives. During the final 4 weeks, lower it to 60 because real alliances break down. The `phase_aware_threshold` flag in the config handles this automatically when you supply the season start date.

### 2. Cross-Validate with Production Logs
The cascade engine is probabilistic—it amplifies false signals if you only feed it social media noise. Always cross-validate your drama trajectory map with the actual production logs (if you have access) or with the official BBB26 recap clips. A mismatch between predicted and actual escalation means your sentiment drift window is too narrow; widen it by 5 minutes and re-run calibration.
