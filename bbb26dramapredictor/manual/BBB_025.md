## What is bbb26dramapredictor?

bbb26dramapredictor is a production-grade sentiment and narrative forecasting engine built specifically for **Big Brother Brasil 26** (and extensible to other reality shows). It ingests live social feeds, closed captioning, and audience reaction metrics to predict breakout drama moments, alliance fractures, and viewer sentiment shifts before they hit the mainstream. Think of it as a real-time **drama CI/CD pipeline** for unscripted entertainment.

## Core Features

### 1. Multi-Modal Narrative Conflict Detection (M2NCD)
Combines NLP sentiment analysis from Twitter/X, Reddit, and live chat with audio tone analysis from broadcast clips. The model flags **narrative inflection points** (e.g., a 0.75+ sentiment delta between two contestants within a 5-minute window) and classifies them as *alliance shakeup*, *romantic tension*, or *game move*. Output is a timestamped JSON event stream ready for ingestion into your own dashboard or alerting system.

### 2. Drama Probability Heatmap API
Exposes a RESTful endpoint (`/api/v1/drama/heatmap`) that returns a 15-minute rolling probability score (0–1) for each contestant pair. The score is derived from a **stacked ensemble** of LSTM-based time series models and a graph neural network (GNN) trained on historical BBB seasons. The heatmap is color-coded (green = calm, red = imminent blowup) and includes confidence intervals.

### 3. Automated Clip Curation & Export
When a drama event crosses a configurable threshold (default: 0.85), the tool automatically captures the preceding 60 seconds of broadcast audio + transcript, overlays the predicted sentiment curve, and exports a `.mp4` clip with burned-in timestamps. This is designed for OBS integration or direct upload to platforms like YouTube Shorts or TikTok. No manual editing required.

## How to Use

### Step 1: Configure the Event Pipeline
First, clone the repo and run the bootstrap script to set up your environment variables. You’ll need to provide Twitter API v2 credentials (with `tweet.read` and `users.read` scopes) and a WebSocket URL for the live broadcast feed. The tool uses **Redis Streams** as the event backbone—ensure your Redis instance is running locally or via a cloud provider (ElastiCache recommended for production).

```bash
git clone https://github.com/xingfangwang-eng/bbb26dramapredictor.git
cd bbb26dramapredictor
cp .env.example .env
# Edit .env: TWITTER_BEARER_TOKEN, REDIS_URL, BROADCAST_WS_URL
docker-compose up -d
```

### Step 2: Launch the Real-Time Observer
Start the observer service, which continuously polls the Twitter filtered stream and the broadcast WebSocket. It writes raw events to a Redis stream named `drama:raw`. You can monitor its health via the built-in Prometheus endpoint at `localhost:9090/metrics`. The observer uses **backpressure-aware batching** to handle spikes (e.g., eviction night).

```bash
python observer.py --stream twitter,broadcast --batch-size 100 --flush-interval 5
```

### Step 3: Query the Drama Heatmap & Trigger Alerts
Once the observer is running, query the heatmap API to get live probabilities. For example, to check the drama probability between contestants *Isabella* and *Lucas* over the last 15 minutes:

```bash
curl -X GET "http://localhost:5000/api/v1/drama/heatmap?pair=isabella,lucas&window=15m" | jq
```

To set up an automated clip export, configure a webhook in your monitoring system (e.g., Grafana Alertmanager) to POST to `/api/v1/clip/export` when the heatmap score exceeds 0.85. The clip will be saved to `./exports/` with a filename like `drama_2025-03-21_22-15-00.mp4`.

## Best Practices

### 1. Tune the Drama Threshold Per Season
Default thresholds are calibrated from BBB24 data, but each season has unique dynamics. Run the `calibrate.py` script during the first week of the new season using a labeled dataset of actual drama events (e.g., from fan wikis or recap sites). This adjusts the GNN edge weights and sentiment baseline. **Do not skip this step**—it can improve precision by up to 30%.

### 2. Use a Separate Redis Instance for State Management
The drama predictor writes heavily to Redis (streams, sorted sets for rolling windows, and hash maps for contestant profiles). To avoid contention with your main application cache, spin up a dedicated Redis instance (or a separate logical database). Set `maxmemory-policy allkeys-lru` to prevent OOM during peak hours like eviction nights. We recommend a `cache.r6g.large` on AWS or equivalent.
