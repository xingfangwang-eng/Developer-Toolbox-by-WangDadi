# bbb26dramapredictor - BBB_009

## Overview

bbb26dramapredictor is a production-grade NLP pipeline that ingests live social chatter, episode transcripts, and audience reaction data to forecast drama escalation points during Big Brother Brasil 26. Unlike generic sentiment analyzers, this tool is fine-tuned on Brazilian Portuguese reality TV discourse, detecting sarcasm, fan faction shifts, and producer manipulation signals.

## Core Features

### 1. **Tension Wavefront Detection via Temporal Graph Neural Networks**
   - Models contestant interactions as a dynamic graph where edge weights shift based on live Twitter/X volume, tone, and co-occurrence patterns. The model predicts "drama peaks" 15–30 minutes before they trend, using a custom LSTM-GNN hybrid trained on 12 seasons of BBB data.
   - Outputs a real-time `drama_score` (0–100) per scene, with explainable heatmaps showing which contestants or topics are driving the surge.

### 2. **Multimodal Sentiment Fuser (Text + Audio + Emoji)**
   - Beyond text, the pipeline ingests audio pitch/variance from live feeds (via wav2vec 2.0) and emoji clusters from WhatsApp/Telegram groups. The fuser uses a cross-attention transformer to weigh modalities adaptively—e.g., if text is sarcastic but audio is aggressive, the model flags a "stealth conflict."
   - Supports real-time streaming via Kafka, with a latency under 2 seconds from data ingestion to prediction output.

### 3. **Contestant Archetype & Narrative Arc Predictor**
   - Each contestant is profiled into one of 8 archetypes (e.g., "Moleque", "Estrategista", "Mãe do Grupo") using a BERT-based classifier on their confessionals. The tool then predicts narrative arcs (e.g., "Vilão redemption" or "Casal fake breakup") by matching current behavior against historical pattern sequences via a Markov chain model.
   - Generates a daily "Drama Forecast" report with probability distributions for the next 24 hours.

## How to Use

### Step 1: Configure Data Sources & Model Parameters
   - Edit `config.yaml` to set your Twitter API bearer token, YouTube Data API key (for live feed transcripts), and Telegram bot token (for group scraping).  
   - Tune the `DETECTION_THRESHOLD` (default: 0.65) to balance precision vs. recall—lower values catch more subtle drama but increase false positives.  
   - Run `python bbb26_dramapredictor.py --init` to validate your credentials and download the latest model weights from our S3 bucket.

### Step 2: Launch Real-Time Prediction Pipeline
   - Execute `docker-compose up -d` to spin up the stack: Kafka broker, Spark streaming job, and the Flask API server.  
   - The API exposes three endpoints:  
     - `GET /api/v1/drama/current` → returns live `drama_score` per scene.  
     - `GET /api/v1/drama/forecast?hours=6` → returns probabilistic forecast.  
     - `POST /api/v1/contestants/archetype` → accepts a list of contestant names and returns their predicted archetype with confidence scores.  
   - Monitor logs via `docker logs -f bbb26-drama-predictor-spark` to ensure no lag in the streaming pipeline.

### Step 3: Analyze Outputs & Trigger Alerts
   - The tool writes results to a PostgreSQL database (`drama_events` table) with columns: `timestamp`, `scene_description`, `drama_score`, `top_contributors`, `archetype_shifts`.  
   - Use the bundled Grafana dashboard (`dashboards/bbb26_drama_dashboard.json`) to visualize trends—set up alert rules (e.g., if `drama_score > 80` for 3 consecutive minutes, send a Slack webhook).  
   - For advanced users, export the raw feature vectors via `GET /api/v1/debug/features` to build custom downstream models (e.g., betting odds prediction).

## Best Practices

### 1. **Domain-Adaptive Fine-Tuning for Each BBB Season**
   - The pre-trained model is generic, but each season introduces new slang, memes, and contestant dynamics. **Do not skip the fine-tuning step** on the first 48 hours of the season. Run `python bbb26_dramapredictor.py --fine-tune --season 26` with at least 10K labeled tweets (using our labeling tool `labeling_ui/`). This reduces false drama alerts by ~40%.

### 2. **Rate-Limit Your Data Sources to Avoid API Bans**
   - Twitter/X and Telegram have strict rate limits. Implement exponential backoff and a token bucket queue (built into the `data_sources/twitter_stream.py` module). For high-traffic events (e.g., elimination nights), spin up a secondary Twitter account with a separate bearer token and use a round-robin load balancer. **Never exceed 450 requests per 15-minute window** per token.
