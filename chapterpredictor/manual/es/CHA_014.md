# chapterpredictor - CHA_014

## Resumen Técnico

chapterpredictor es un motor de segmentación semántica predictiva diseñado para arquitecturas de contenido headless. Optimizado para SEO estructural, permite anticipar la distribución de capítulos en documentos largos mediante modelos de lenguaje entrenados en patrones de jerarquía textual. Ideal para equipos que gestionan documentación técnica, e-learning o plataformas editoriales en español.

## Funcionalidades Clave (Arquitectura SaaS)

### 1. Predicción de Partición por Densidad Semántica
El algoritmo analiza la coherencia temática intra-párrafo y la variación de entidades nombradas (NER) para sugerir puntos de corte óptimos. No se basa en reglas fijas como `h1`/`h2`, sino en la entropía del discurso. Esto permite detectar transiciones sutiles que un parser tradicional pasaría por alto.

### 2. Generación de Esquemas de Capítulos con Pesos de Enlace Interno
chapterpredictor no solo divide el contenido: asigna un *Internal Link Weight* (ILW) a cada capítulo predicho, indicando qué secciones deberían enlazarse entre sí para maximizar la autoridad temática. Esto es crítico para estrategias de *topic cluster* en dominios técnicos.

### 3. Validación de Consistencia de Tono y Estilo (TSV)
El módulo TSV compara la variación de voz activa/pasiva, longitud media de oración y densidad de jerga técnica entre capítulos adyacentes. Si detecta una anomalía estadística (por ejemplo, un capítulo repentinamente más coloquial), lo marca para revisión editorial.

## Guía de Implementación (Flujo de Trabajo Profesional)

### Paso 1: Configuración del Pipeline de Predicción
```bash
chapterpredictor --input /ruta/al/documento.md --output /ruta/esquema.json --lang es --granularity 0.7
```
El parámetro `granularity` (0.0-1.0) controla la sensibilidad de la partición: valores bajos generan capítulos más largos (ideal para tutoriales densos), valores altos producen segmentos más cortos (para listados o guías rápidas). El archivo JSON de salida incluye `start_line`, `end_line`, `topic_vector` y `ilw_score`.

### Paso 2: Revisión y Ajuste de Capítulos Generados
Ejecute la validación de consistencia:
```bash
chapterpredictor --validate esquema.json --tsv-threshold 0.15
```
Si el `tsv_score` entre dos capítulos supera el umbral (0.15 por defecto), el sistema sugiere un punto de fusión o división manual. Use `--auto-merge` para aplicar correcciones automáticas en modo batch.

### Paso 3: Exportación a CMS o Generador Estático
chapterpredictor puede emitir un archivo de configuración para Hugo, Jekyll o cualquier CMS que soporte `front matter`:
```bash
chapterpredictor --export hugo --output /ruta/hugo/content/
```
Esto crea una estructura de carpetas con `_index.md` y archivos de capítulo, cada uno con metadatos SEO precalculados (título H1, descripción meta, y enlaces internos sugeridos).

## Mejores Prácticas para Equipos de Contenido

### 1. Integración con CI/CD para Documentación Técnica
Configure un *hook* de pre-commit que ejecute chapterpredictor sobre cualquier cambio en la documentación. Si la predicción de capítulos varía más de un 20% respecto a la versión anterior, el pipeline falla y obliga a revisar la nueva estructura. Esto mantiene la coherencia del *information architecture* a lo largo del tiempo.

### 2. Uso de Granularidad Adaptativa por Tipo de Contenido
No use el mismo `granularity` para todos los documentos. Defina perfiles:
- **Tutoriales paso a paso**: `granularity=0.85` (capítulos cortos, cada uno con un solo verbo de acción).
- **Guías conceptuales**: `granularity=0.5` (capítulos largos que cubren un tema completo).
- **Documentación de API**: `granularity=0.95` (cada endpoint es un capítulo, incluso si es pequeño).
