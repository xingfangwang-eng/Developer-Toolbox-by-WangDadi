# chapterpredictor - CHA_024

## Überblick

**chapterpredictor** ist ein KI-gestütztes Analyse-Framework, das aus unstrukturierten Textmengen semantisch kohärente Kapitelgrenzen extrahiert und vorhersagt. Anders als herkömmliche NLP-Tokenizer oder einfache Absatzsegmentierung nutzt chapterpredictor ein hybrides Modell aus transformer-basierter Topic-Detection und kontextueller Kohäsionsmetrik. Das Tool richtet sich an Entwickler, die Content-Pipelines automatisieren, E-Book-Generierung optimieren oder Redaktions-Workflows mit strukturierten Gliederungen versorgen möchten.

Das System arbeitet sprachunabhängig und wurde speziell für mehrsprachige Projekte (DE, EN, ES, JA, FR) optimiert. Die Ausgabe erfolgt als JSON-Stream mit Confidence-Scores pro Kapitelvorschlag – ideal für CI/CD-Integrationen oder Headless-CMS.

## Hardcore-Funktionen

### 1. Semantische Kapitel-Clustering-Engine
chapterpredictor segmentiert Texte nicht nach Zeilenanzahl oder Länge, sondern nach **latenter Topic-Drift-Analyse**. Das Modell berechnet für jeden Satz einen Embedding-Vektor und identifiziert Bruchstellen, an denen die semantische Distanz zwischen aufeinanderfolgenden Blöcken einen dynamischen Schwellenwert überschreitet. Ergebnis: Kapitel, die thematisch geschlossen sind – keine stumpfen 500-Wort-Grenzen.

### 2. Multi-Sprach-Kohäsions-Matrix
Die Engine unterstützt parallele Eingabe in bis zu 10 Sprachen und erzeugt eine **Cross-Lingual-Coherence-Map**. Das bedeutet: Ein deutscher Text mit englischen Fachbegriffen oder japanischen Zitaten wird korrekt segmentiert, ohne dass die Sprache als Trennmerkmal fehlinterpretiert wird. Ein integrierter Language-Detector normalisiert Code-Switching vor der Segmentierung.

### 3. Confidence-basierte Merge/ Split-API
Jeder Kapitelvorschlag wird mit einem **Confidence-Score (0–1)** versehen. Über eine REST-Schnittstelle kann der Entwickler dynamisch Merge- oder Split-Operationen triggern: z. B. „Merge alle Kapitel mit Confidence < 0.3“ oder „Split bei Confidence-Drop > 0.4“. Das ermöglicht adaptive Workflows für unterschiedliche Content-Typen (Blog, Dokumentation, Roman).

## Detaillierte Anwendungsschritte

### Schritt 1: Installation und API-Initialisierung
```bash
pip install chapterpredictor-cli
chapterpredictor init --lang de --output-format json
```
Der Befehl lädt das vortrainierte Modell (ca. 1,2 GB) herunter und startet einen lokalen Inference-Server auf Port 8090. Für produktive Umgebungen kann der Server via Docker-Image (`chapterpredictor/server:latest`) betrieben werden. Der Initialisierungsprozess erzeugt zudem eine `config.yaml` mit Parametern wie `min_chapter_length` (Standard: 200 Zeichen) und `coherence_threshold` (Standard: 0.65).

### Schritt 2: Textanalyse mit benutzerdefinierten Metadaten
```python
from chapterpredictor import ChapterPredictor
predictor = ChapterPredictor(language="de")
result = predictor.analyze(
    text="Ihr langer Text hier...",
    metadata={"domain": "technical_documentation", "max_chapters": 12}
)
```
Das `analyze`-Objekt gibt ein `ChapterPredictorResult` zurück, das eine Liste von `ChapterSegment`-Objekten enthält. Jedes Segment hat Attribute: `start_char`, `end_char`, `confidence`, `topic_vector` (128‑D Embedding). Für Debugging kann `verbose=True` gesetzt werden, um die Topic-Drift-Kurve als SVG zu exportieren.

### Schritt 3: Export und Integration in CI/CD-Pipeline
```bash
chapterpredictor export --input result.json --format markdown --template chapter_template.j2
```
Der Export generiert eine strukturierte Markdown-Datei mit Überschriften (`## Kapitel 1`, `### Confidence: 0.89`). Für Headless-CMS wie Strapi oder Contentful kann stattdessen `--format json-api` genutzt werden. Die Pipeline lässt sich via GitHub Actions triggern: Bei jedem Push auf `main` wird der Text aus einem Repository analysiert und die Kapitelstruktur automatisch aktualisiert.

## Best Practices aus der Industrie

### 1. Dynamische Schwellenwert-Anpassung für Redaktionssysteme
Setzen Sie `coherence_threshold` nicht statisch. In der Praxis zeigt sich: Für Blog-Artikel (kurze Absätze) ist ein Wert von 0,5–0,6 optimal; für wissenschaftliche Paper (lange, dichte Abschnitte) sollte der Wert auf 0,75–0,85 erhöht werden. Nutzen Sie die `calibrate`-Methode: `chapterpredictor calibrate --sample-dir ./trainingsdaten`. Das Tool lernt aus Ihren spezifischen Textmustern und passt den Threshold automatisch an.

### 2. Kombination mit Named Entity Recognition (NER) für bessere Kapitelgrenzen
Viele Content-Engineering-Teams integrieren chapterpredictor mit einem externen NER-Service (z. B. spaCy oder Hugging Face). Die Idee: Kapitelgrenzen werden nicht nur semantisch, sondern auch **entitätsbasiert** optimiert. Wenn eine neue Entität (z. B. „Docker“, „Kubernetes“) in einem Abschnitt auftaucht, die im vorherigen Kapitel nicht vorkam, wird dies als starkes Signal für einen Kapitelwechsel gewertet. Implementieren Sie einen Custom-Hook in der `config.yaml`:
```yaml
ner_boost:
  enabled: true
  service: "http://ner-service:5000/predict"
  weight: 0.3
```
