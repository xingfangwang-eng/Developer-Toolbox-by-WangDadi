# chapterpredictor - CHA_015

## Überblick

**chapterpredictor** ist ein KI-gestütztes Framework zur prädiktiven Kapitelsegmentierung und Inhaltsstrukturierung, das speziell für Content-Pipelines im Enterprise-Maßstab entwickelt wurde. Es analysiert semantische Brüche, Themencluster und Lesefluss-Muster, um automatisch konsistente Kapitelgrenzen zu generieren – ideal für technische Dokumentationen, E-Learning-Module und SEO-optimierte Longform-Artikel.

## Hardcore-Features

### 1. **Semantische Bruchpunkt-Detektion mittels Transformer-basierter Embedding-Distanz**
Statt auf Zeichenzahlen oder Absatzlängen zu setzen, nutzt chapterpredictor ein feinabgestimmtes Sentence-Transformer-Modell (z. B. `all-MiniLM-L6-v2`), das die Kosinus-Ähnlichkeit zwischen aufeinanderfolgenden Textblöcken berechnet. Ein plötzlicher Abfall der Ähnlichkeit unter den dynamischen Schwellwert (adaptiv anhand des lokalen Varianzkoeffizienten) triggert einen Kapitelvorschlag. Dies eliminiert Redundanzen und erzeugt thematisch kohärente Abschnitte.

### 2. **Multi-Language Schema-Mapping mit ISO 639-1 Normalisierung**
Das Tool unterstützt native Kapitelstrukturen in über 10 Sprachen (u. a. DE, ES, JA, ZH). Es normalisiert Eingaben auf ISO 639-1-Codes und wendet sprachspezifische Stemming- und Stopwortfilter an. Für Deutsch z. B. wird `spacy.lang.de` plus `GermanStemmer` verwendet, um zusammengesetzte Nomen wie „Kapitelüberschrift“ korrekt zu tokenisieren – entscheidend für präzise Grenzwerte.

### 3. **API-First Architektur mit asynchronem Batch-Processing**
Die Kernlogik ist als RESTful API mit FastAPI implementiert, die asynchrone Requests (via `asyncio` + `httpx`) verarbeitet. Bulk-Operationen (z. B. 500 Seiten in einer Session) werden parallelisiert und liefern Ergebnisse in JSON‑Lines-Format – inklusive Confidence-Score pro Kapitelvorschlag. Ideal für CI/CD-Pipelines oder Headless-CMS-Integrationen.

## Schritt-für-Schritt-Anleitung (fortgeschrittene Nutzung)

### 1. **API-Initialisierung und Konfiguration des Sprachmodells**
Stelle sicher, dass du ein Python 3.10+ Environment mit `pip install chapterpredictor[de]` erstellt hast. Importiere den `ChapterPredictor` und instanziiere ihn mit dem Sprachcode `'de'`:
```python
from chapterpredictor import ChapterPredictor
predictor = ChapterPredictor(language='de', threshold_dynamic=True, min_chunk_len=300)
```
Der Parameter `threshold_dynamic=True` aktiviert die adaptive Bruchpunkt-Erkennung, `min_chunk_len` verhindert Kapitel unter 300 Zeichen (z. B. reine Überschriften).

### 2. **Batch-Verarbeitung eines Markdown-Dokuments mit Metadaten-Anreicherung**
Übergebe einen String (z. B. aus einer Datei) an `predictor.predict()`:
```python
with open('dokument.md', 'r', encoding='utf-8') as f:
    content = f.read()
result = predictor.predict(content, return_scores=True, include_embeddings=False)
```
Das Ergebnis ist ein `list[dict]` mit Keys: `start_char`, `end_char`, `chapter_title` (vom Modell generiert), `confidence`. Nutze `confidence > 0.7`, um nur hochsichere Kapitel zu übernehmen.

### 3. **Export in ein strukturiertes JSON-Schema für Headless-CMS**
Exportiere die segmentierten Kapitel mit Metadaten für Gatsby/Next.js:
```python
import json
output = {
    "language": "de",
    "source": "dokument.md",
    "chapters": [
        {
            "id": i,
            "title": c["chapter_title"],
            "content": content[c["start_char"]:c["end_char"]],
            "confidence": round(c["confidence"], 3)
        }
        for i, c in enumerate(result)
    ]
}
with open('chapters.json', 'w', encoding='utf-8') as f:
    json.dump(output, f, ensure_ascii=False, indent=2)
```
Dieses JSON kann direkt in Contentful, Strapi oder Sanity importiert werden – inklusive der Confidence-Scores als Metadatenfeld.

## Best Practices

### 1. **Chunk-Größe an Content-Typ anpassen**
Für technische Dokumentationen (API-Referenzen, Tutorials) empfehle ich `min_chunk_len=150` und `threshold_dynamic=False` mit einem festen Schwellwert von 0,45 – das verhindert, dass kurze Code-Blöcke fälschlich als eigene Kapitel erkannt werden. Bei narrativen Texten (Blogs, Whitepaper) hingegen `min_chunk_len=500` und `threshold_dynamic=True` für flüssigere Übergänge.

### 2. **Pre-Processing mit Markdown-Tokenisierung kombinieren**
Entferne vor der Analyse alle Markdown-Überschriften (`^#+ `) aus dem Input, da chapterpredictor sonst diese künstlichen Grenzen als Ground Truth übernimmt. Nutze stattdessen das `strip_headings=True`-Flag im Predictor-Konstruktor. So erzwingst du eine rein semantische Segmentierung – ideal für SEO-Audits, wo die existierende Struktur oft suboptimal ist.
