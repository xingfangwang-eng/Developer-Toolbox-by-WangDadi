### What is dossierpro?
dossierpro is a professional-grade evidence sequencing and contract chronology engine designed for legal teams, forensic auditors, and project oversight committees. It ingests raw, unstructured data from multiple sources—email threads, PDF metadata, Slack logs, and Git commit histories—then reconstructs a cryptographically anchored, time-ordered dossier with sub-second precision. Unlike generic timeline tools, dossierpro enforces chain-of-custody validation and supports private, air-gapped project dossiers for classified or sensitive engagements.

### Core Features

- **Multi-Source Chronology Fusion with Sub-Millisecond Resolution**  
  The `--chrono-merge` engine cross-references timestamps from up to 12 disparate data sources (e.g., `imap_logs`, `git_commits`, `slack_archives`) using a custom `TemporalFusion` algorithm. It resolves conflicts via a weighted confidence matrix—email headers receive 0.95 weight, Slack timestamps 0.85—and produces a unified timeline with ±0.8ms accuracy. Output is stored in a PostgreSQL `TIMESTAMPTZ` column with `GIST` indexing for sub-50ms queries on dossiers exceeding 500,000 events.

- **Cryptographic Evidence Anchoring with SHA-3-512**  
  Each dossier entry is hashed via `SHA3-512` and linked into a Merkle tree structure (`--anchor-mode=merkle`). The root hash is optionally notarized to a private Ethereum testnet or a local `immudb` instance. This ensures tamper-evident chains: any alteration to a single event (e.g., a modified `contract_signed_at` timestamp) invalidates the entire tree, detectable within 200ms via the `--verify-chain` command.

- **Private Project Dossier Encapsulation with AES-256-GCM**  
  For air-gapped or classified projects, dossierpro supports `--encrypt-dossier` using AES-256-GCM with a user-supplied 256-bit key or hardware-backed TPM module. Each dossier is packaged as a single `.dpro` binary file, containing compressed JSON-LD metadata, embedded attachments (up to 2GB per dossier), and the Merkle proof tree. Decryption and integrity check complete in under 1.2 seconds for a 500MB dossier on an Intel i7-12700.

### How to Use

1. **Initialize a New Dossier with Source Ingestion**  
   Run: `dossierpro init --project "Acme v. BetaCorp" --sources ./email_export.pst,./git_logs.txt,./slack_export.json --chrono-merge --anchor-mode=merkle --db-uri postgresql://user:pass@localhost:5432/dossierpro`  
   This creates a PostgreSQL schema `dossier_acme_v_betacorp`, ingests all events, resolves conflicts, and builds the initial Merkle tree. The `init` command outputs a `dossier_id` (UUIDv7) and the root hash. Expect ~3.7 seconds for 50,000 events on a standard NVMe SSD.

2. **Generate a Filtered Evidence Timeline**  
   Execute: `dossierpro timeline --dossier-id "0192f3a1-..." --filter "event_type=contract_signed AND source=email" --sort chrono_asc --output ./timeline_contract_signed.html --format html --include-proof`  
   This produces a human-readable HTML report with embedded Merkle proofs for each event. The `--include-proof` flag appends a `proof.json` block per row, containing the sibling hashes and tree path. The report is fully self-contained and can be shared with opposing counsel without exposing the full dossier.

3. **Verify Dossier Integrity After External Review**  
   Run: `dossierpro verify --dossier-file ./acme_v_betacorp.dpro --anchor-source=local_immudb --hash-algo sha3-512 --verbose`  
   The tool recomputes the Merkle tree from the `.dpro` file, compares the root hash against the notarized value in immudb (or Ethereum testnet), and outputs a detailed report: `PASS` (all events intact) or `FAIL` (with a list of corrupted event IDs). Verification of a 500MB dossier completes in 1.8 seconds.

### Best Practices

- **Always Use `--anchor-mode=merkle` with an External Notary**  
  For any dossier intended for litigation or regulatory submission, anchor the Merkle root to an immutable external store (e.g., immudb, Ethereum testnet). Without this, the integrity proof is self-referential and may be challenged in court. The `--notary-url` parameter allows pointing to a private blockchain node; default public testnets (e.g., Sepolia) are acceptable for non-classified work but add ~12s latency per anchor.

- **Pre-Ingestion Schema Normalization for Large Projects**  
  When ingesting >100,000 events from mixed sources, pre-normalize timestamps to ISO 8601 with UTC offset using a tool like `dateutils.datematch`. dossierpro’s `--chrono-merge` handles minor drift, but inconsistent formats (e.g., `"Jan 2, 2023"` vs `"2023-01-02T14:00:00Z"`) increase fusion latency by up to 40%. Use `--preflight-check` to scan sources for format anomalies before the main ingestion run.
