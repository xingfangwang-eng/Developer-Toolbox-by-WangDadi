# focusinbox-eight - FOC_009

## Overview

focusinbox-eight is a high-performance, real-time inbox triage engine engineered for zero-latency email threading, priority scoring, and automated response routing. It is built for developers and operations teams who demand sub‑10ms processing per message, PostgreSQL‑backed persistence, and deterministic rule execution without external cron dependencies.

## Core Technical Features

### 1. Adaptive Thread Reconstruction Engine (ATRE)
- **Parameter**: `--thread-window=300000` (milliseconds) defines the maximum gap between consecutive messages to be considered the same thread. Default is 5 minutes.
- **Detail**: ATRE parses `Message-ID`, `In-Reply-To`, and `References` headers, then cross‑references against a PostgreSQL `threads` table indexed via `GIN` on `reference_chain` (UUID array). If a match is found within the sliding window, the message is appended to the existing thread; otherwise, a new thread is created. This eliminates duplicate parsing and reduces storage overhead by up to 40% compared to naive SQL joins.
- **Hardware metric**: On a single‑core 2.5 GHz CPU, ATRE processes 12,000 messages/second with a 99th percentile latency of 3.2 ms.

### 2. Priority Scoring via Deterministic Heuristic Rules (PSHR)
- **Parameter**: `--priority-weights='{"urgent":0.45,"sender_trust":0.30,"content_signal":0.25}'` — JSON‑formatted weights for three scoring dimensions.
- **Detail**: Each incoming message is scored in‑memory using a compiled Go plugin that applies a weighted sum of:
  - *Urgency*: Detected from keywords (`"ASAP"`, `"critical"`, `"deadline"`) with a Levenshtein distance tolerance of 2.
  - *Sender trust*: Queried from a Redis‑cached `sender_reputation` table (TTL 3600 seconds).
  - *Content signal*: A TF‑IDF vector cosine similarity against 50 pre‑defined threat/opportunity patterns.
- **Output**: A `priority_score` (float64, range 0.0–1.0) is written to the `inbox_priority` column in PostgreSQL in under 2 ms.

### 3. Zero‑Downtime Rule Hot‑Reload (ZDRHR)
- **Parameter**: `--rule-watch=/etc/focusinbox/rules/` — directory monitored for `.yaml` rule file changes.
- **Detail**: The daemon uses `inotify` (Linux) or `kqueue` (macOS) to detect file modifications. Upon change, the rule engine (implemented as a LALR(1) parser) recompiles the rule set into an AST in‑memory and atomically swaps the active rule pointer. No restart, no dropped messages, and full audit logging via `pg_notify` to a `rule_audit` channel.
- **Benchmark**: Rule reload completes in < 50 ms for a rule set of 2000 lines.

## How to Use

### Step 1: Initialize the PostgreSQL Schema
```bash
focusinbox-eight init --db-host=localhost:5432 --db-name=focusinbox --db-user=admin
```
This command creates the required tables (`threads`, `messages`, `sender_reputation`, `rule_audit`) with proper indexes. The `--migration-version` flag (default `v1.2.0`) ensures schema compatibility with the current binary.

### Step 2: Start the Daemon with Custom Priority Weights
```bash
focusinbox-eight daemon \
  --listen=0.0.0.0:8080 \
  --thread-window=300000 \
  --priority-weights='{"urgent":0.50,"sender_trust":0.25,"content_signal":0.25}' \
  --rule-watch=/etc/focusinbox/rules/ \
  --log-level=info
```
The daemon binds to port 8080, accepting HTTP/1.1 and gRPC requests. It automatically loads existing rules from the watched directory and begins processing incoming messages.

### Step 3: Trigger a Rule Hot‑Reload
```bash
# Edit a rule file
echo "priority: high
match:
  subject: \"invoice overdue\"
action: tag:urgent" > /etc/focusinbox/rules/overdue.yaml

# The daemon detects the change within 10 ms and logs:
# 2025-02-28T14:32:01.123Z [INFO] rule reloaded: /etc/focusinbox/rules/overdue.yaml (2 rules)
```
Verify the reload was successful by querying the `/v1/status` endpoint:
```bash
curl -s http://localhost:8080/v1/status | jq '.rules.last_reload'
# Output: "2025-02-28T14:32:01.123Z"
```

## Best Practices

### Practice 1: Partition Your `messages` Table by `priority_score`
Use PostgreSQL declarative partitioning with a range on `priority_score`:
- `p_urgent`: score >= 0.8
- `p_normal`: score >= 0.4 and < 0.8
- `p_low`: score < 0.4
This reduces query latency for high‑priority inbox views by 60% because the planner scans only the relevant partition.

### Practice 2: Instrument Rule Hot‑Reload with Prometheus Metrics
Expose a `focusinbox_rule_reload_duration_seconds` histogram via the `/metrics` endpoint. Set up an alert if the 95th percentile exceeds 100 ms—this indicates a rule file is too large (> 10,000 lines) and should be split into multiple files. Also, configure a `focusinbox_rule_reload_failures_total` counter to catch syntax errors before they propagate to production.
