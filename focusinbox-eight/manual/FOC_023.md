# focusinbox-eight - FOC_023

## Overview

focusinbox-eight is a high‑performance, real‑time inbox triage and prioritization engine designed for productivity engineers who manage hundreds of daily notifications across fragmented channels. It implements an adaptive Bayesian filtering pipeline that learns from user interaction patterns, delivering sub‑50ms classification latency per message and a false‑positive rate below 1.2% after 48 hours of training.

## Core Features

### 1. Dynamic Attention‑Weighted Priority Scoring (DAWPS)

Each incoming message is assigned a real‑time attention score based on a weighted combination of sender reputation, historical response latency, message thread depth, and calendar proximity. The scoring engine runs as a PostgreSQL stored procedure, updating user‑specific weights every 30 seconds via a background worker. The `focusinbox‑eight‑score` parameter accepts a JSON payload with fields such as `sender_id`, `thread_depth`, `time_since_last_reply`, and `calendar_overlap_minutes`. The output is a float between 0.0 and 1.0, where messages above `0.75` are automatically pinned to the top of the inbox queue.

### 2. Multi‑Channel De‑duplication with Semantic Fingerprinting

Messages from Slack, email, and Jira are merged into a single unified feed. A locally‑deployed Sentence‑Transformer model (MiniLM‑L6‑v2) generates a 384‑dimensional embedding for each message body. Duplicates are detected when cosine similarity exceeds `0.92`, and the system retains only the highest‑priority copy. The deduplication engine runs at a throughput of 1,200 messages per second on a single CPU core, with a memory footprint of ~64 MB for the embedding cache.

### 3. Predictive Snooze with Context‑Aware Resurfacing

Instead of manual snooze, focusinbox‑eight predicts the optimal time to resurface a deferred message. The model uses a gradient‑boosted decision tree trained on historical reopen patterns, taking into account the user’s current calendar load (retrieved via Microsoft Graph API) and the message’s urgency score. The `snooze_predict` endpoint returns a Unix timestamp with millisecond precision. If the user’s next available slot is within 30 minutes, the system automatically schedules resurfacing at that time; otherwise, it falls back to the next business hour.

## How to Use

### Step 1: Initialize the Focus Engine

```bash
focusinbox-eight init --db-uri postgresql://user:pass@localhost:5432/focusdb \
                      --embedding-model miniLM-L6-v2 \
                      --score-threshold 0.75
```

This command creates the required tables (`inbox_messages`, `user_weights`, `snooze_predictions`) and downloads the embedding model into a local cache. The initialization completes in under 3 seconds on a machine with an SSD.

### Step 2: Ingest Messages from Multiple Channels

```python
from focusinbox_eight import Ingestor

ingestor = Ingestor()
ingestor.add_channel("slack", token="xoxb-...", channel_ids=["C1234", "C5678"])
ingestor.add_channel("email", imap_server="imap.gmail.com", email="user@example.com")
ingestor.add_channel("jira", base_url="https://your-domain.atlassian.net", api_token="...")

ingestor.start_polling(interval_ms=5000)  # Polls every 5 seconds
```

The ingestor buffers up to 10,000 messages locally before flushing to PostgreSQL. Each message is automatically fingerprinted and scored within 50 ms of ingestion.

### Step 3: Run the Predictive Snooze Scheduler

```bash
focusinbox-eight snooze --schedule-interval 300 \
                        --calendar-api microsoft \
                        --fallback-hour 9 \
                        --fallback-minute 0
```

This starts a background daemon that re‑evaluates all snoozed messages every 5 minutes. It queries the user’s calendar for the next 24 hours and adjusts resurface times accordingly. Logs are written to `stdout` at INFO level, with WARN level alerts for any messages that cannot be resurfaced within the predicted window.

## Best Practices

### 1. Calibrate the Priority Threshold Weekly

Run the built‑in calibration tool every Monday morning to adjust the `score_threshold` based on the previous week’s false‑positive rate. Use the command `focusinbox-eight calibrate --window 7d --target-fpr 0.01`. This re‑trains the Bayesian filter on the last 7 days of user interactions, ensuring the model adapts to changing communication patterns without manual tuning.

### 2. Enable Cross‑Device Sync via PostgreSQL Logical Replication

For teams using focusinbox‑eight across multiple workstations, configure PostgreSQL logical replication to keep the priority scores and snooze states synchronized. Set up a publication on the primary database and a subscription on each secondary instance. This ensures that a message snoozed on your laptop is immediately reflected on your desktop, with a replication lag of less than 200 ms under typical load.
