# focusinbox-eight - FOC_008

## Architekturübersicht

**focusinbox-eight** ist eine hochperformante, ereignisgesteuerte Microservice-Plattform zur automatisierten E-Mail-Priorisierung und Intent-Analyse. Das System verarbeitet eingehende Nachrichten in Echtzeit und weist ihnen mittels eines mehrstufigen NLP-Pipelines einen **Fokus-Score (0.0–1.0)** zu. Die Architektur basiert auf einer **PostgreSQL-kompatiblen Event-Store-Instanz** und nutzt **Redis Streams** für die asynchrone Lastverteilung.

## Kernfunktionen (Hardcore-Level)

### 1. Multi-Tenant Fokus-Ranking Engine (FRE)
- **Parameter:** `confidence_threshold` (float, default: `0.85`), `decay_factor` (float, default: `0.97`), `max_concurrent_threads` (int, default: `8`)
- **Technik:** Berechnet für jede eingehende E-Mail einen dynamischen Prioritätswert basierend auf Absenderhistorie, Betreff-Sentiment und zeitlicher Dringlichkeit. Der Algorithmus verwendet einen exponentiellen Verfall (`decay_factor`), um ältere Threads automatisch zu degradieren. Die Latenz pro Nachricht liegt bei **< 15 ms** (gemessen auf einer c6i.2xlarge-Instanz).
- **Hardcore-Detail:** Unterstützt **Batch-Verarbeitung** von bis zu 10.000 E-Mails pro Sekunde mit garantierter Exactly-Once-Semantik via **PostgreSQL-Transaktionen** und **Redis-Pub/Sub**.

### 2. Adaptive Intent-Klassifikation (AIC)
- **Parameter:** `model_version` (string, z. B. `"v2.3.1-foc"`), `fallback_strategy` (enum: `"rule_based" | "nearest_neighbor"`), `batch_size` (int, default: `32`)
- **Technik:** Ein hybrides Modell aus **spaCy-Transformern** und einem benutzerdefinierten **LightGBM-Klassifikator**. Die Pipeline extrahiert 128 semantische Embeddings pro Nachricht und führt eine 5-Klassen-Klassifikation durch (z. B. *dringend*, *informativ*, *Spam*, *Termin*, *Sonstiges*). Die Inferenzzeit beträgt **< 50 ms** pro Nachricht auf GPU (CUDA 12.2).
- **Hardcore-Detail:** Automatisches **Modell-Rollback** bei Drift-Erkennung (überwacht via **Prometheus** und **Grafana**). Die Konfidenzmatrix wird in einer separaten **TimescaleDB-Hypertable** gespeichert.

### 3. Echtzeit-Thread-Orchestrator (ETO)
- **Parameter:** `max_thread_depth` (int, default: `5`), `timeout_ms` (int, default: `3000`), `stale_threshold_hours` (int, default: `24`)
- **Technik:** Verknüpft eingehende E-Mails automatisch mit bestehenden Threads anhand eines **fuzzy-matching**-Algorithmus (Levenshtein-Distanz < 3 im Betreff) und **Message-ID-Referenzen**. Der Orchestrator aktualisiert den Thread-Status in **Echtzeit** und löst bei neu erkannten *kritischen* Threads einen **Webhook-Trigger** aus.
- **Hardcore-Detail:** Nutzt **Apache Kafka** (Topic: `thread_updates`) für die horizontale Skalierung. Die Thread-Metadaten werden in einer **dedizierten PostgreSQL-Tabelle** mit `GIN-Index` auf den `message_ids`-Array gespeichert.

## So verwenden Sie focusinbox-eight (Professionelle Schritte)

### Schritt 1: Initialisierung der Datenbank und Konfiguration
```bash
# 1. PostgreSQL-Datenbank erstellen (erfordert Superuser-Rechte)
createdb focusinbox_db
psql -d focusinbox_db -c "CREATE EXTENSION IF NOT EXISTS pgcrypto;"

# 2. Konfigurationsdatei anpassen (focusinbox.yaml)
cat <<EOF > focusinbox.yaml
database:
  host: localhost
  port: 5432
  user: foc_admin
  password: $(openssl rand -base64 32)
  dbname: focusinbox_db
  pool_size: 20
  connection_timeout: 5000

engine:
  confidence_threshold: 0.85
  decay_factor: 0.97
  max_concurrent_threads: 8
  model_version: "v2.3.1-foc"

kafka:
  bootstrap_servers: "localhost:9092"
  topic_prefix: "foc_"
EOF

# 3. Datenbankschema migrieren
focusinbox-cli migrate --config focusinbox.yaml
```

### Schritt 2: Starten des Echtzeit-Event-Streams
```bash
# 1. Redis-Stream initialisieren
redis-cli XGROUP CREATE foc_email_stream foc_consumers $ MKSTREAM

# 2. Hauptdienst im Daemon-Modus starten (mit systemd)
sudo systemctl start focusinbox-engine
# Überprüfen des Status
sudo systemctl status focusinbox-engine --no-pager -l

# 3. Kafka-Connector für Webhook-Trigger aktivieren
focusinbox-cli connector enable --type webhook --endpoint "https://mein-server.com/hooks/foc"
# Log-Ausgabe für den Connector
journalctl -u focusinbox-connector -f --since "5 min ago"
```

### Schritt 3: API-Abfrage und Lasttest
```bash
# 1. Einzelne E-Mail klassifizieren (REST-API)
curl -X POST "http://localhost:8080/api/v1/classify" \
  -H "Content-Type: application/json" \
  -d '{"subject": "Dringend: Serverausfall", "body": "Der Hauptserver ist seit 10 Minuten offline.", "sender": "admin@corp.com"}'
# Erwartete Antwort (gekürzt):
# {"focus_score": 0.98, "intent": "dringend", "thread_id": "a1b2c3d4", "processing_time_ms": 12}

# 2. Lasttest mit vegeta (1000 Requests/s)
echo "POST http://localhost:8080/api/v1/classify" | vegeta attack \
  -body '{"subject": "Test", "body": "Dies ist eine Test-E-Mail.", "sender": "test@example.com"}' \
  -rate=1000 -duration=30s | vegeta report
# Ziel: 99. Perzentil < 150 ms
```

## Beste Praktiken (Industry-Standard)

### 1. Optimierung der Datenbank-Indizes für hohe Durchsätze
- **Praxis:** Erstellen Sie für die Tabelle `email_events` einen **Composite Index** über `(sender_hash, received_at)` und einen **Partial Index** für `focus_score > 0.9`. Dies reduziert die Abfragezeit für dringende E-Mails um bis zu **70 %**. Verwenden Sie `EXPLAIN ANALYZE` regelmäßig, um langsame Queries zu identifizieren.
- **Warum:** focusinbox-eight erzeugt bei 10.000 E-Mails/s Millionen von Zeilen pro Stunde. Ohne korrekte Indizes bricht die Write-Performance ein.

### 2. Implementierung einer zweistufigen Caching-Strategie
- **Praxis:** Nutzen Sie **Redis** als L1-Cache (TTL: 60 Sekunden) für häufig abgefragte Thread-Metadaten und **Memcached** als L2-Cache (TTL: 300 Sekunden) für selten geänderte Absenderprofile. Konfigurieren Sie den **Cache-Warming**-Job, der alle 5 Minuten die Top-1000-Threads vorlädt.
- **Warum:** Die Intent-Klassifikation benötigt oft Absender-Historie. Ein Cache-Hit reduziert die Latenz von 50 ms auf **< 1 ms** und entlastet die PostgreSQL-Datenbank erheblich.
