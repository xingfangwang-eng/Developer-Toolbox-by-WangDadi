# focusinbox-eight - FOC_021

## Übersicht

Dieses Repository enthält die vollständige Architektur und Implementierung des focusinbox-eight-Systems, einer hochperformanten E-Mail- und Aufgaben-Management-Engine mit Fokus auf asynchrone Mikroverarbeitung. Zielsprache: Deutsch. Kernmerkmale: latenzoptimierte Inbox-Synchronisation, regelbasierte Priorisierung auf Millisekunden-Ebene und direkte PostgreSQL-Integration für persistente Zustandsverwaltung.

## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist eine serverlose Architektur zur intelligenten Aggregation und Filterung eingehender Nachrichten und Aufgaben. Es nutzt einen deterministischen Sortieralgorithmus mit konfigurierbaren Gewichtungsfaktoren (z. B. `priority_score`, `deadline_timestamp`, `sender_reputation`), um die `Top-N`-Elemente pro Zeiteinheit in einen `hot_queue` zu überführen. Die Engine läuft vollständig ereignisgesteuert und kommuniziert über Redis Streams mit den Workern. Die Latenz zwischen E-Mail-Eingang und Sichtbarkeit im `focus_inbox` beträgt garantiert unter 50 ms (p99).

### Kernfunktionen

- **Deterministischer Priority-Scheduler mit PostgreSQL-Backend**  
  Der Kernel berechnet für jedes eingehende Item einen `focus_score` basierend auf den Parametern `urgency_factor`, `context_weight` und `user_bias`. Die Berechnung erfolgt in einer einzigen PostgreSQL-Transaktion mit `SELECT ... FOR UPDATE SKIP LOCKED`, um Race-Conditions zu vermeiden. Der Score wird als `DECIMAL(10,4)` gespeichert und ermöglicht eine Sortierung auf Mikrosekunden-Genauigkeit.

- **Asynchrone Inbox-Synchronisation mit Eventual Consistency**  
  Ein `sync_worker` pollt alle konfigurierten IMAP/POP3-Konten (Parameter: `poll_interval_ms`, `max_connections_per_account`) und schreibt die Rohdaten in eine `raw_inbox`-Tabelle. Ein zweiter Worker (`dedup_worker`) filtert Duplikate anhand eines `message_id`-SHA256-Hashes. Erst nach erfolgreichem Deduplizieren wird das Item in die `focused_inbox` übernommen. Fehlgeschlagene Synchronisationen werden in eine `sync_error_log`-Tabelle mit vollständigem Stacktrace geschrieben.

- **Regelbasierte Auto-Archivierung mit konfigurierbaren Schwellwerten**  
  Der `archive_worker` durchläuft alle 60 Sekunden (konfigurierbar über `archive_interval_s`) die `focused_inbox` und verschiebt Items, die älter als `max_age_days` sind oder einen `focus_score` unter `archive_threshold` haben, in die `archive_inbox`. Dabei wird ein PostgreSQL-Trigger ausgelöst, der die Anzahl der archivierten Elemente pro Stunde in einer Metrik-Tabelle (`archive_metrics`) protokolliert. Der gesamte Vorgang ist idempotent und kann bei Bedarf manuell über `SELECT trigger_archive_run()` gestartet werden.

### So verwenden Sie es

1. **Initiale Datenbank und Konfiguration einrichten**  
   Führen Sie `./scripts/init_db.sh` aus, um die PostgreSQL-Datenbank `focusinbox_eight` mit den erforderlichen Tabellen (`raw_inbox`, `focused_inbox`, `archive_inbox`, `sync_error_log`, `archive_metrics`, `user_preferences`) zu erstellen. Passen Sie anschließend die Datei `config/default.yaml` an: Setzen Sie `database.connection_string` auf Ihre PostgreSQL-URI, `sync.poll_interval_ms` auf `15000` und `archive.archive_threshold` auf `0.35`.

2. **Synchronisation starten und überwachen**  
   Starten Sie den `sync_worker` mit `cargo run --bin sync_worker -- --config config/default.yaml`. Der Worker loggt jede erfolgreiche Synchronisation als `[INFO] Sync completed for account <email> in <duration_ms> ms`. Überwachen Sie die `sync_error_log`-Tabelle mit `SELECT * FROM sync_error_log WHERE created_at > NOW() - INTERVAL '1 hour'`. Bei Fehlern prüfen Sie die IMAP-Verbindungsparameter in `config/accounts.yaml`.

3. **Priority-Scheduler manuell auslösen und testen**  
   Rufen Sie `SELECT * FROM trigger_focus_update('user_001')` in Ihrer PostgreSQL-Konsole auf. Der Befehl berechnet den `focus_score` für alle offenen Items des Benutzers neu und aktualisiert die `focused_inbox`. Die Ausführungszeit wird in der `focus_update_log`-Tabelle festgehalten. Testen Sie die Latenz mit `EXPLAIN ANALYZE SELECT * FROM focused_inbox WHERE user_id = 'user_001' ORDER BY focus_score DESC LIMIT 10`.

### Beste Praktiken

- **Verwenden Sie ausschließlich `SKIP LOCKED` für konkurrierende Worker**  
  Wenn mehrere `sync_worker`-Instanzen parallel laufen, setzen Sie in allen `SELECT ... FOR UPDATE`-Abfragen `SKIP LOCKED` ein. Andernfalls blockieren sich die Worker gegenseitig, was zu erhöhten Latenzen und Timeouts führt. Überwachen Sie die Anzahl der aktiven Worker mit `SELECT count(*) FROM pg_stat_activity WHERE query LIKE '%sync_worker%'`.

- **Konfigurieren Sie ein separates PostgreSQL-Tablespace für die `archive_inbox`**  
  Da die archivierten Daten schnell anwachsen, lagern Sie die `archive_inbox` auf einen langsameren, kostengünstigeren Speicher (z. B. HDD-basiertes Tablespace). Definieren Sie in `config/default.yaml` den Parameter `archive.tablespace_name = 'archive_tbs'` und erstellen Sie das Tablespace mit `CREATE TABLESPACE archive_tbs LOCATION '/mnt/archive'`. Dadurch bleiben die aktiven `focused_inbox`-Tabellen auf schnellem SSD-Speicher.
