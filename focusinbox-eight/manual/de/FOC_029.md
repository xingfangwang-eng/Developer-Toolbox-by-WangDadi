# focusinbox-eight - FOC_029

## Übersicht

Dieses Repository enthält die vollständige Implementierung des focusinbox-eight Systems – einer hochperformanten, asynchronen Echtzeit-Postfach-Aggregations-Engine für die automatisierte Priorisierung eingehender Nachrichten. Zielsprache: DE. Optimiert für Workflows mit sub-100ms Latenz und mandantenfähiger PostgreSQL-Integration.

## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist eine serverlose, ereignisgesteuerte Architektur zur intelligenten Kategorisierung und zeitgesteuerten Zustellung von E-Mail- und Messaging-Volumen. Es analysiert eingehende Payloads mittels eines gewichteten Entscheidungsbaums (max. 128 Knoten pro Regel) und priorisiert Aufgaben basierend auf Absender-Hash, Betreff-Sentiment und zeitlichem Dringlichkeitsfaktor (Δt < 300 Sekunden). Die Engine läuft vollständig in einer isolierten Node.js 20 LTS Runtime mit nativen Worker-Threads für parallele Verarbeitung.

### Kernfunktionen

- **Multi-Queue-Priorisierung mit dynamischer Gewichtung**  
  Implementiert einen gewichteten, zeitgesteuerten Scheduling-Algorithmus (`weightedRoundRobin`). Parameter: `queueDepth` (max. 10.000 Einträge), `priorityDecay` (exponentieller Faktor 0,92 pro Minute). Die Engine berechnet für jede eingehende Nachricht einen `urgencyScore` (0.0–1.0) aus `senderReputation`, `subjectSentiment` und `responseDeadline`. Ergebnisse werden in Millisekunden in einer Redis-Stream-Instanz (Cluster-Modus, 6 Shards) persistiert.

- **Echtzeit-Sentiment-Analyse mit benutzerdefinierten Lexika**  
  Nutzt ein vortrainiertes, deutsches Sentiment-Modell (FastText-basiert, 300 Dimensionen) zur Klassifizierung von Nachrichteninhalten. Unterstützt benutzerdefinierte Lexika (JSON-Format, max. 5000 Einträge) für domänenspezifische Begriffe. Die Analyse erfolgt inline im Request-Handler mit einer maximalen Latenz von 50ms pro Nachricht. Ergebnisse werden als `sentimentVector` (Array mit 3 Float-Werten: negativ/neutral/positiv) zurückgegeben.

- **Automatisierte Eskalationskette mit PostgreSQL-Integration**  
  Bei Überschreitung eines konfigurierbaren `escalationThreshold` (Standard: 0,85) wird automatisch ein `escalationTicket` in einer PostgreSQL-Datenbank (Version 15) erstellt. Das Ticket enthält: `ticketId` (UUID v4), `sourceQueue`, `urgencyScore`, `escalationTimestamp` (ISO 8601, Millisekunden-Genauigkeit) und `assignedHandler` (via Round-Robin aus einer Handler-Pool-Tabelle). Die Transaktion wird mittels `pg-promise` mit `REPEATABLE READ` Isolation ausgeführt.

### So verwenden Sie es

1. **Konfiguration der Queue-Regeln**  
   Erstellen Sie eine JSON-Datei `rules.json` mit folgendem Schema:  
   ```json
   {
     "queues": [
       { "name": "support", "priorityWeight": 0.7, "maxDepth": 5000, "decayFactor": 0.95 },
       { "name": "billing", "priorityWeight": 0.9, "maxDepth": 2000, "decayFactor": 0.98 }
     ],
     "defaultQueue": "general",
     "escalationThreshold": 0.85
   }
   ```  
   Starten Sie den Service mit `npm run start -- --config ./rules.json`. Der Service lädt die Regeln asynchron in den Shared-Memory-Cache (max. 256 MB).

2. **Senden einer Nachricht an die Engine**  
   Nutzen Sie den HTTP-Endpoint `POST /api/v1/message` mit folgendem Payload:  
   ```json
   {
     "sender": "max.mustermann@example.de",
     "subject": "Dringend: Rechnungskorrektur erforderlich",
     "body": "Bitte korrigieren Sie die Rechnung Nr. 2024-0456.",
     "deadline": "2024-12-15T18:00:00Z"
   }
   ```  
   Die Engine berechnet den `urgencyScore` (z. B. 0,92) und leitet die Nachricht entweder direkt in die `billing`-Queue oder erstellt ein Eskalationsticket, falls der Schwellenwert überschritten wird. Die Antwort enthält `messageId`, `assignedQueue` und `estimatedProcessingTime` (in ms).

3. **Abrufen der priorisierten Queue**  
   Rufen Sie `GET /api/v1/queue/{queueName}` auf, um die nächsten 50 Einträge einer Queue abzurufen. Der Endpoint liefert ein Array von `queueEntry`-Objekten, sortiert nach `urgencyScore` (absteigend). Jeder Eintrag enthält `messageId`, `senderHash` (SHA-256), `subjectPreview` (max. 100 Zeichen) und `priorityDecayRemaining` (Anzahl der verbleibenden Minuten bis zur vollständigen Dekadierung). Nutzen Sie den `X-Queue-Token`-Header für Authentifizierung (JWT-basiert, 60 Minuten Gültigkeit).

### Beste Praktiken

- **Praxis 1: Nutzung von Multi-Region-Deployment für Ausfallsicherheit**  
  Betreiben Sie mindestens zwei Instanzen von focusinbox-eight in geografisch getrennten Rechenzentren (z. B. Frankfurt und Irland). Verwenden Sie einen globalen Load Balancer (z. B. AWS Global Accelerator) mit Health-Checks alle 5 Sekunden. Konfigurieren Sie die Redis-Streams im Active-Passive-Modus, um Datenverlust bei Regionen-Failover zu vermeiden. Die maximale RTO (Recovery Time Objective) beträgt 30 Sekunden.

- **Praxis 2: Regelmäßige Überprüfung der Sentiment-Lexika**  
  Führen Sie monatlich ein automatisiertes Audit Ihrer benutzerdefinierten Lexika durch. Nutzen Sie das mitgelieferte Skript `npm run audit-lexicon`, das die Abdeckung (Coverage) und die Konsistenz der Sentiment-Werte prüft. Ein Coverage-Wert unter 0,75 (gemessen an einem repräsentativen Sample von 10.000 Nachrichten) sollte zu einer sofortigen Überarbeitung des Lexikons führen. Protokollieren Sie alle Änderungen in einer `lexicon_changelog`-Tabelle (PostgreSQL) für Compliance-Zwecke.
