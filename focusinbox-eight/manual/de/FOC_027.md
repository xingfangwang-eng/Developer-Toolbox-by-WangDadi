# focusinbox-eight - FOC_027

## Übersicht

Dieses Repository implementiert die **FOC_027**-Spezifikation für die hochperformante E-Mail-Intent-Klassifikation. focusinbox-eight ist eine asynchrone Microservice-Architektur, die eingehende Nachrichten in Echtzeit analysiert und in bis zu 27 konfigurierbare Kategorien (z. B. `INVOICE`, `SUPPORT_TICKET`, `NEWSLETTER`) einordnet. Die Latenz pro Nachricht liegt bei < 50 ms (p99) bei paralleler Verarbeitung von 10.000+ Threads.

## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist ein **Event-Driven Classifier Backend** für intelligente Posteingangssteuerung. Es kombiniert ein vortrainiertes Transformer-Modell (DistilBERT‑finetuned) mit einem regelbasierten Overlay zur Erkennung von Metadaten (z. B. `X-Priority-Header`, `DKIM-Signatur-Pattern`). Die Ausgabe erfolgt als strukturiertes JSON‑Objekt mit Confidence-Score und Feature-Vektor für nachgelagerte Automatisierungen.

### Kernfunktionen

- **Multimodale Intent-Extraktion**  
  Parst sowohl den Betreff (`subject`) als auch den Body (`body_text`) mittels eines zweistufigen Pipelines: (1) Tokenisierung über `sentencepiece` (Vokabular: 32.000 Subwords), (2) Inferenz auf einer NVIDIA T4 GPU (FP16) mit Batch-Größe 64. Der Confidence-Schwellwert für die Klassifikation ist standardmäßig auf `0.78` gesetzt und kann über die Umgebungsvariable `FOC_CONFIDENCE_THRESHOLD` überschrieben werden.

- **PostgreSQL-gestützte Feedback-Schleife**  
  Jede Klassifikation wird als `classification_event` in einer PostgreSQL‑Tabelle persistiert (Schema: `{id UUID, timestamp TIMESTAMPTZ, raw_email_hash BYTEA, predicted_label VARCHAR(27), confidence REAL, user_feedback_label VARCHAR(27)}`). Ein asynchroner Cron-Job (alle 60 Sekunden) aggregiert die `user_feedback_label` und führt ein periodisches Fine-Tuning des Modells durch (Batch-Learning mit SGD, Lernrate `1e-5`).

- **Multi-Tenant-Rate-Limiter mit Redis-Backing**  
  Implementiert einen Token-Bucket-Algorithmus pro `tenant_id` (max. 500 Requests/Minute). Die Bucket-Größe und Auffüllrate werden in Redis (Key: `foc:rate_limit:{tenant_id}`) mit einer TTL von 60 Sekunden gespeichert. Bei Überschreitung wird ein `429 Too Many Requests` mit dem Header `X-RateLimit-Reset: <Unix-Timestamp>` zurückgegeben.

### So verwenden Sie es

1. **Deployment der Infrastruktur**  
   Führen Sie `docker-compose up -d` aus – dies startet den `classifier-service` (Port 8080), `redis` (Port 6379) und `postgres` (Port 5432). Setzen Sie die Umgebungsvariablen `POSTGRES_DSN=postgresql://user:pass@localhost:5432/focusinbox` und `REDIS_URL=redis://localhost:6379/0`.

2. **Ersten Klassifikationsrequest senden**  
   Nutzen Sie `curl` mit einem gültigen JSON-Body:  
   ```bash
   curl -X POST http://localhost:8080/v1/classify \
     -H "Content-Type: application/json" \
     -H "X-Tenant-ID: acme_corp" \
     -d '{
       "email_id": "msg_001",
       "subject": "Rechnung 2024-08-15",
       "body_text": "Sehr geehrte Damen und Herren, anbei die Rechnung für August.",
       "headers": {"x-priority": "1"}
     }'
   ```
   Die Antwort enthält `{"label": "INVOICE", "confidence": 0.942, "processing_time_ms": 12}`.

3. **Feedback zur Verbesserung des Modells geben**  
   Senden Sie ein PUT an den `feedback`-Endpunkt, um die automatische Klassifikation zu korrigieren:  
   ```bash
   curl -X PUT http://localhost:8080/v1/feedback \
     -H "Content-Type: application/json" \
     -d '{"email_id": "msg_001", "corrected_label": "SUPPORT_TICKET"}'
   ```
   Das System speichert das Feedback und berücksichtigt es beim nächsten Batch-Training (alle 60 Sekunden).

### Beste Praktiken

- **Batch-Inferenz für hohe Durchsätze**  
  Senden Sie nie einzelne E-Mails, sondern sammeln Sie mindestens 32 Nachrichten in einem Array (`{"emails": [...]}`) und nutzen Sie den `/v1/classify/batch`-Endpunkt. Dadurch wird die GPU-Auslastung maximiert (Tensor Cores arbeiten effizienter) und die durchschnittliche Latenz pro E-Mail sinkt auf unter 15 ms.

- **Kontinuierliches Monitoring der Confidence-Verteilung**  
  Richten Sie ein Dashboard (z. B. mit Prometheus + Grafana) ein, das den `confidence`-Mittelwert pro `predicted_label` sowie die `user_feedback_label`-Abweichung („Drift“) anzeigt. Ein plötzlicher Abfall der durchschnittlichen Confidence unter `0.65` deutet auf einen Data-Drift hin – dann sollte das Modell neu trainiert werden, bevor die Klassifikationsqualität leidet.
