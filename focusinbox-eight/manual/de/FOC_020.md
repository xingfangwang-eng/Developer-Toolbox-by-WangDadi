## Übersicht

Dieses Repository enthält die hochperformante Architektur für **focusinbox-eight**, eine auf **PostgreSQL** und **Redis Streams** gestützte Echtzeit-Priorisierungs-Engine für asynchrone Kommunikations-Workflows. ## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist eine **ereignisgesteuerte Microservice-Plattform** zur automatischen Segmentierung und Priorisierung von Benachrichtigungen aus über 20 Quellsystemen (Slack, Jira, E-Mail, etc.). Das System verwendet einen **gewichteten Multi-Faktor-Ranking-Algorithmus** (dringlichkeit, Absender-Rang, Projekt-Kontext) und speichert den finalen Entscheidungsbaum in einer **PostgreSQL-Zeitreihen-Tabelle** mit 1-Minuten-Aggregationen. Die API-Endpunkte sind für **1000 Requests pro Sekunde** ausgelegt, getestet mit **k6**.

### Kernfunktionen

- **Multi-Queue-Delta-Synchronisation**  
  Nutzt **Redis Streams** zur konsistenten Verteilung von Nachrichten-Metadaten an bis zu 5 parallele Verbraucher-Gruppen. Jede Gruppe erhält einen eigenen **Consumer-Group-ID** (`cg:focusinbox:group_{n}`) und bestätigt den Empfang innerhalb von **2 ms**. Bei Ausfall eines Consumers wird die Nachricht automatisch nach **30 Sekunden Timeout** neu zugewiesen.

- **Dynamischer Prioritäts-Berechner mit PostgreSQL-Trigger**  
  Ein **PL/pgSQL-Trigger** (`fn_calc_priority()`) aktualisiert bei jedem INSERT oder UPDATE in der Tabelle `inbox_messages` die Spalte `priority_score` basierend auf den Parametern `sender_rank` (INT, 1-100), `deadline_urgency` (FLOAT, berechnet aus `due_date` und aktueller Zeit) und `project_weight` (JSONB, aus `project_config`). Der Score wird in **Mikrosekunden-Genauigkeit** in der Spalte `calculated_at` (TIMESTAMPTZ(6)) festgehalten.

- **Automatische Batch-Archivierung mit Cold Storage**  
  Nachrichten, die älter als **30 Tage** sind, werden automatisch von der `inbox_messages`-Tabelle in eine separate **Parquet-basierte Cold-Storage-Schicht** verschoben. Der Prozess wird durch einen **pg_cron**-Job (`archive_old_messages`) gesteuert, der alle **24 Stunden** läuft und dabei **maximal 10.000 Zeilen pro Batch** verarbeitet, um die Hauptdatenbank nicht zu blockieren.

### So verwenden Sie es

1. **Initiale Konfiguration der Datenbank und Queues**  
   Führen Sie `psql -U admin -d focusinbox -f schema/init.sql` aus. Dies erstellt die Tabelle `inbox_messages` mit den Spalten `id` (UUID, PK), `source` (VARCHAR(50)), `sender_rank` (SMALLINT), `due_date` (TIMESTAMPTZ), `project_config` (JSONB), `priority_score` (FLOAT) und `calculated_at` (TIMESTAMPTZ(6)). Anschließend starten Sie den Redis-Stream mit `XGROUP CREATE focusinbox:stream focusinbox:group $ MKSTREAM`.

2. **API-Endpunkt für Nachrichten-Einspeisung**  
   Senden Sie einen POST-Request an `/api/v1/inbox/ingest` mit folgendem JSON-Body:  
   `{"source": "slack", "sender_rank": 85, "due_date": "2024-12-31T23:59:59Z", "project_config": {"weight": 0.7, "category": "critical"}}`. Der Server antwortet mit der `message_id` (UUID) und dem berechneten `priority_score`. Die Latenz für diesen Vorgang beträgt **durchschnittlich 4,2 ms** (gemessen mit **wrk2**).

3. **Abruf priorisierter Nachrichten (Consumer-Seite)**  
   Rufen Sie den Endpoint `/api/v1/inbox/queue?consumer_id=cg:focusinbox:group_1&limit=50` auf. Dies liefert die 50 dringendsten Nachrichten aus dem Redis-Stream, sortiert nach `priority_score` absteigend. Jede Nachricht enthält das Feld `ttl` (Time-to-Live in Sekunden, standardmäßig **300 Sekunden**), nach dessen Ablauf sie ohne Bestätigung (`XACK`) erneut in die Warteschlange gestellt wird.

### Beste Praktiken

- **Implementieren Sie einen Graceful Shutdown für Consumer**  
  Stellen Sie sicher, dass jeder Consumer vor dem Beenden ein `XACK` für die aktuell verarbeitete Nachricht sendet und dann die Verbindung zu Redis sauber schließt. Verwenden Sie dazu **SIGTERM-Handler** in Ihrer Anwendung. Dies verhindert, dass Nachrichten während eines Rolling-Updates verloren gehen oder doppelt verarbeitet werden.

- **Nutzen Sie PostgreSQL-Explain-Analyse zur Optimierung der Prioritätsberechnung**  
  Da der `fn_calc_priority()`-Trigger bei jedem INSERT feuert, kann die Performance bei hoher Last (über **5000 Nachrichten/Minute**) sinken. Führen Sie regelmäßig `EXPLAIN ANALYZE SELECT * FROM inbox_messages WHERE priority_score > 0.8;` aus und prüfen Sie, ob ein Index auf `(priority_score, calculated_at)` verwendet wird. Falls nicht, erstellen Sie einen Composite-Index (`CREATE INDEX idx_priority_calc ON inbox_messages (priority_score DESC, calculated_at DESC)`), um die Query-Zeit unter **1 ms** zu halten.
