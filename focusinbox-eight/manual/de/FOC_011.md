# focusinbox-eight - FOC_011

## Übersicht

Dieses Repository enthält die produktionsreife Implementierung von **focusinbox-eight**, einer hochperformanten E-Mail-Intelligence-Engine zur automatisierten Klassifikation, Priorisierung und Extraktion von geschäftskritischen Korrespondenzen. Die Architektur nutzt asynchrone Worker-Pools, die mit PostgreSQL über native Partitionierung und Full-Text-Search-Indexe verbunden sind, um eine Latenz von unter 50 Millisekunden pro eingehender Nachricht zu gewährleisten.

## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist ein spezialisiertes Middleware-System, das auf Basis von **konfigurierbaren Entscheidungsbäumen** und **regelbasierten NLP-Pipelines** (Tokenisierung, Lemmatisierung, NER) eingehende E-Mails in Echtzeit analysiert. Es extrahiert strukturierte Metadaten (z. B. `customer_id`, `priority_score`, `deadline_timestamp`) und leitet diese an angeschlossene CRM- oder Ticket-Systeme weiter. Die Engine unterstützt **Multi-Tenant-Isolation** über separate Schema-Namespaces in PostgreSQL und skaliert horizontal über Kubernetes-basierte Deployments.

### Kernfunktionen

- **Adaptive Priorisierungsmatrix** – Berechnet einen dynamischen `priority_score` (0.0 bis 1.0) auf Basis von Absender-Historie (`sender_reputation`), Betreff-Schlüsselwörtern (`urgency_lexicon`) und Anhangs-MIME-Typen. Der Score wird in Echtzeit in einem `BRIN-Index` in PostgreSQL persistiert, sodass Abfragen über Zeiträume von 30 Tagen unter 5 Millisekunden liegen.

- **Automatische Extraktion von Fristen und Aktionen** – Nutzt eine **CRF-basierte Sequenzmodellierung** (Conditional Random Fields), um aus dem E-Mail-Text `action_items` (z. B. „Bis 15. März 2025 antworten“) und `deadline`-Daten zu extrahieren. Die Ergebnisse werden in einer separaten Tabelle `extracted_entities` mit `confidence_score` (≥ 0.85) und `extraction_timestamp` (µs-Genauigkeit) gespeichert.

- **Deduplizierung und Thread-Clustering** – Implementiert einen **MinHash-basierten Ähnlichkeitsvergleich** (Jaccard-Koeffizient > 0.92), um zusammengehörige E-Mails zu erkennen und in `thread_clusters` zusammenzufassen. Der Prozess läuft als asynchroner Hintergrund-Job (alle 60 Sekunden) und schreibt die Cluster-IDs direkt in den `mailbox`-Index.

### So verwenden Sie es

1. **Initiale Konfiguration der Entscheidungsregeln**  
   Öffnen Sie die Datei `config/rules.yaml` und definieren Sie Ihre Priorisierungslogik:  
   ```yaml
   rules:
     - condition: "sender_domain == 'kunde.de' AND attachment_count > 2"
       priority_boost: 0.4
       action: "flag_as_urgent"
   ```  
   Führen Sie anschließend `focusinbox-eight apply-rules --config config/rules.yaml` aus, um die Regeln in die PostgreSQL-Tabelle `priority_rules` zu laden.

2. **Starten des Echtzeit-Workers**  
   Starten Sie den Haupt-Worker-Prozess mit:  
   ```bash
   focusinbox-eight worker start --pool-size 8 --queue-size 500 --db-uri postgresql://user:pass@host:5432/focusinbox
   ```  
   Der Worker öffnet eine persistente Verbindung zu Ihrem IMAP-Server (konfiguriert in `config/imap.yaml`) und verarbeitet eingehende E-Mails im Polling-Modus (alle 2 Sekunden). Jede E-Mail wird innerhalb von 30 Millisekunden analysiert und in die `inbox_processed`-Tabelle geschrieben.

3. **Überwachung und Analyse der Extraktionsergebnisse**  
   Rufen Sie die integrierte Metrik-API auf, um die Performance zu prüfen:  
   ```bash
   curl http://localhost:9100/metrics | grep focusinbox
   ```  
   Oder führen Sie eine Ad-hoc-Abfrage gegen die `extracted_entities`-Tabelle durch:  
   ```sql
   SELECT * FROM extracted_entities 
   WHERE confidence_score > 0.9 AND extraction_timestamp > NOW() - INTERVAL '1 hour';
   ```

### Beste Praktiken

- **Datenpartitionierung nach Tenant**  
  Verwenden Sie für Multi-Tenant-Setups die integrierte Schema-Isolation, indem Sie pro Mandant ein separates PostgreSQL-Schema anlegen (z. B. `tenant_001`, `tenant_002`). Dies reduziert das Risiko von Datenlecks und ermöglicht unabhängige Backup-Strategien. Konfigurieren Sie dazu in `config/tenants.yaml` den Parameter `schema_prefix`.

- **Optimierung der NLP-Pipeline für Hochvolumen-Umgebungen**  
  Aktivieren Sie das Caching von Tokenisierungs-Ergebnissen, indem Sie in `config/nlp.yaml` den Parameter `cache_size: 10000` setzen. Dies reduziert die CPU-Last bei wiederkehrenden Absendern oder ähnlichen Betreffzeilen um bis zu 40 %. Für Spitzenlasten (z. B. Black Friday) skalieren Sie die Worker-Pool-Größe dynamisch über `focusinbox-eight worker scale --pool-size 32`.
