# focusinbox-eight - FOC_022

## Übersicht

focusinbox-eight ist eine hochperformante Echtzeit-Postfach-Analyse-Engine, die speziell für die automatisierte Priorisierung eingehender Korrespondenz in stark frequentierten Inbox-Umgebungen entwickelt wurde. Das System nutzt ein mehrschichtiges KI-Modell zur semantischen Gewichtung und ermöglicht eine submillisekundenschnelle Klassifizierung von bis zu 10.000 E-Mails pro Stunde.

## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist eine containerisierte Microservice-Architektur, die auf einer PostgreSQL-Datenbank mit TimescaleDB-Erweiterung aufbaut und mittels eines neuronalen Netzwerks (Transformer-basiert) eingehende Nachrichten in Echtzeit analysiert. Das System extrahiert nicht nur Schlüsselwörter, sondern berechnet einen dynamischen Fokus-Score (0.0 - 1.0) basierend auf Absenderhistorie, Betreffzeilen-Semantik und Anhang-Metadaten. Die Latenz für die vollständige Verarbeitung einer E-Mail liegt bei durchschnittlich 12 Millisekunden (ms) auf einer Standard-Cloud-Instanz (4 vCPUs, 16 GB RAM).

### Kernfunktionen

- **Dynamische Fokus-Gewichtung mit semantischer Clusteranalyse:** Das Kernmodul `weight-engine` analysiert jede eingehende Nachricht mittels eines feinabgestimmten BERT-Modells (deutsches Sprachmodell) und berechnet einen `focus_score`. Dieser Score wird mit einem konfigurierbaren Schwellenwert (`threshold: 0.65`) verglichen. Bei Überschreitung wird die E-Mail automatisch in die `high_priority_queue` verschoben. Die Analyse umfasst die Extraktion von `sender_domain`, `subject_tokens` und `attachment_hash` zur Deduplizierung und zur Erkennung von Korrespondenzketten.

- **Asynchrone Batch-Verarbeitung mit PostgreSQL-Integration:** Das System nutzt `pg_notify` für die Echtzeit-Benachrichtigung bei neuen Einträgen in der Tabelle `incoming_mails`. Ein Hintergrund-Worker (`batch_processor`) aggregiert alle 500 Millisekunden (ms) eingehende Nachrichten in Batches (max. 200 E-Mails pro Batch) und führt eine Bulk-Upsert-Operation in die Tabelle `processed_mails` durch. Die Indexierung erfolgt über einen Composite-Index auf `(sender_id, received_at)`, um Abfragen für das Dashboard unter 50 Millisekunden (ms) zu halten.

- **RESTful API mit Rate-Limiting und Webhook-Unterstützung:** Die API-Endpunkte (`/api/v1/process`, `/api/v1/status`) sind mit einem Token-Bucket-Rate-Limiter (100 Requests pro Minute pro API-Key) gesichert. Bei erfolgreicher Verarbeitung einer hochpriorisierten E-Mail wird ein konfigurierbarer Webhook an eine externe URL (z.B. Slack oder Teams) gesendet. Der Payload enthält `email_id`, `focus_score`, `summary` (max. 200 Zeichen) und `action_required` (boolean).

### So verwenden Sie es

1. **Konfiguration der Datenbankverbindung und des Modellpfads:**  
   Setzen Sie die Umgebungsvariablen `DB_CONNECTION_STRING` (Format: `postgresql://user:pass@host:5432/focusinbox`) und `MODEL_PATH` (absoluter Pfad zum trainierten BERT-Modell im ONNX-Format). Führen Sie dann `docker-compose up -d` aus, um die PostgreSQL-Instanz, den Redis-Cache und die Hauptanwendung zu starten. Überprüfen Sie die Logs mit `docker-compose logs -f app`, um sicherzustellen, dass die Verbindung zur Datenbank erfolgreich war und das Modell geladen wurde (Log-Eintrag: `Model loaded successfully. Vocab size: 30522`).

2. **API-Key generieren und ersten Testaufruf durchführen:**  
   Generieren Sie einen API-Key über den Admin-Endpunkt (`POST /api/v1/admin/keys`) mit einem gültigen JWT-Token. Nutzen Sie dann `curl` für einen Test:  
   `curl -X POST http://localhost:8080/api/v1/process -H "X-API-Key: YOUR_API_KEY" -H "Content-Type: application/json" -d '{"sender":"max@beispiel.de", "subject":"Dringende Anfrage zum Projekt", "body":"Bitte um Rückmeldung bis morgen."}'`  
   Die Antwort sollte einen `focus_score` von > 0.7 enthalten, da der Betreff das Wort "Dringend" enthält, welches im Modell hoch gewichtet ist.

3. **Dashboard und Webhook-Konfiguration:**  
   Öffnen Sie das mitgelieferte Grafana-Dashboard (unter `http://localhost:3000`, Standard-Login: `admin/admin`). Importieren Sie das JSON-Modell aus dem Ordner `./grafana/dashboards/`. Konfigurieren Sie im Reiter "Alerts" einen Webhook, der bei einer `high_priority_queue`-Länge von über 50 E-Mails eine Benachrichtigung an einen Slack-Channel sendet. Die Webhook-URL wird in der Datei `config.yml` unter `notifications.slack_webhook_url` gesetzt.

### Beste Praktiken

- **Batch-Größe und Schwellenwert optimieren:** Für Umgebungen mit hohem E-Mail-Aufkommen (> 5.000 E-Mails/Stunde) empfehlen wir, die Batch-Größe auf 500 zu erhöhen (`batch_size: 500`) und das Intervall auf 1.000 ms (`batch_interval_ms: 1000`). Dies reduziert die Anzahl der Datenbank-Transaktionen und verbessert den Durchsatz. Der `focus_score`-Schwellenwert sollte je nach gewünschter Sensitivität zwischen 0.5 (sehr sensitiv, viele Falsch-Positive) und 0.8 (sehr präzise, weniger Treffer) angepasst werden.

- **Regelmäßiges Fine-Tuning des Modells:** Das mitgelieferte Basis-Modell ist auf allgemeine Geschäftskorrespondenz trainiert. Für optimale Ergebnisse in Ihrer spezifischen Domäne (z.B. Support-Tickets, Bewerbungen) sollten Sie das Modell mit Ihren eigenen, gelabelten Daten (mindestens 1.000 Beispiele pro Kategorie) nachtrainieren. Nutzen Sie dazu das Skript `./scripts/fine_tune.py`, das eine Transfer-Learning-Pipeline mit einer Lernrate von 2e-5 und einer Batch-Größe von 16 implementiert. Das trainierte Modell wird automatisch im ONNX-Format exportiert und kann ohne Neustart der Anwendung über die Admin-API (`POST /api/v1/admin/reload_model`) geladen werden.
