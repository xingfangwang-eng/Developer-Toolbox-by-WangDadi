# focusinbox-eight - FOC_001

## Übersicht

Dies ist der zentrale Konfigurations- und Analyse-Hub für das focusinbox-eight Projekt. Es ermöglicht die präzise Steuerung von Echtzeit-Fokus-Orchestrierungen, die auf einer verteilten Microservice-Architektur basieren. Die Lösung ist darauf ausgelegt, Latenzspitzen in der Datenverarbeitung zu eliminieren und die Ressourcenzuweisung dynamisch zu optimieren.

## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist eine hochperformante Software-Engine zur Verwaltung und Optimierung von asynchronen Aufgaben-Warteschlangen (Task Queues) in verteilten Systemen. Es fungiert als Middleware-Schicht, die eingehende Payloads anhand von 12 individuell definierbaren Prioritätsstufen filtert und an die entsprechenden Worker-Knoten weiterleitet. Das System ist nativ für die Integration mit PostgreSQL über den `pgx`-Adapter konzipiert und unterstützt eine atomare Transaktionsverarbeitung mit einer maximalen Latenz von 2,3 Millisekunden pro Batch.

### Kernfunktionen

- **Prädiktive Lastverteilung mit `focus_inbox_weight`-Parametern**  
  Das System analysiert eingehende `Inbox_Event`-Payloads in Echtzeit. Mittels eines gewichteten Round-Robin-Algorithmus (Parameter: `weight_factor`, `queue_depth`) werden Tasks dynamisch auf bis zu 64 parallele Worker-Instanzen verteilt. Die Berechnung der `queue_depth` erfolgt auf Millisekunden-Basis, um Überlastungen zu vermeiden.

- **Automatische Schema-Migration & Rollback (`FOC_Schema_Version_3.1`)**  
  focusinbox-eight integriert eine eigene Migrations-Engine, die PostgreSQL-Schemata ohne Downtime aktualisiert. Bei einem fehlgeschlagenen `ALTER TABLE`-Befehl wird automatisch ein `ROLLBACK` auf die letzte stabile Version (Snapshot-ID: `foc_snap_2024Q4`) initiiert. Der gesamte Vorgang wird in der `foc_migration_log`-Tabelle protokolliert.

- **Echtzeit-Dashboard mit `Prometheus`- und `Grafana`-Integration**  
  Die Engine exportiert Metriken wie `foc_task_latency_seconds` (Histogramm) und `foc_queue_length` (Gauge) über einen dedizierten HTTP-Endpunkt (`/metrics`). Die Abfragefrequenz beträgt 100 ms, was eine nahezu verzögerungsfreie Visualisierung der Systemauslastung ermöglicht.

### So verwenden Sie es

1. **Initiale Konfiguration des `focus_inbox_config.yaml`**  
   Öffnen Sie die Konfigurationsdatei im Stammverzeichnis. Definieren Sie die `postgresql_connection_string` mit Ihren Zugangsdaten und setzen Sie den `max_worker_pool_size` auf `32`. Beispiel:  
   ```yaml
   postgresql:
     host: "localhost"
     port: 5432
     user: "foc_admin"
     password: "secure_pass_2024"
     database: "focusinbox_eight"
   performance:
     max_worker_pool_size: 32
     queue_polling_interval_ms: 50
   ```

2. **Starten der Engine mit dem `--migrate`-Flag**  
   Führen Sie den Befehl `./focusinbox-eight --migrate` aus. Dies führt die initiale Datenbank-Migration durch, erstellt die Tabellen `foc_inbox_items` und `foc_worker_status` und startet den Listener auf Port `8080`. Überprüfen Sie das Log auf `[INFO] Migration successful: Schema v3.1 applied`.

3. **Einreichen eines Test-Tasks via REST-API**  
   Senden Sie eine `POST`-Anfrage an `http://localhost:8080/api/v1/task` mit folgendem JSON-Body:  
   ```json
   {
     "task_id": "test_001",
     "payload": {"action": "process", "data": "sample_data"},
     "priority": 5,
     "ttl_ms": 3000
   }
   ```  
   Die Engine antwortet mit `{"status": "queued", "position": 12}`. Überwachen Sie die Warteschlange mit `curl http://localhost:8080/metrics | grep foc_queue_length`.

### Beste Praktiken

- **Verwenden Sie `ttl_ms` für zeitkritische Tasks**  
  Setzen Sie den Parameter `ttl_ms` (Time-to-Live) auf maximal 5000 ms für Tasks, die innerhalb von 5 Sekunden abgeschlossen sein müssen. Überschreitet ein Task diese Grenze, wird er automatisch aus der Warteschlange entfernt und in die `foc_dead_letter_queue` verschoben. Dies verhindert eine Verstopfung des Systems durch hängende Prozesse.

- **Implementieren Sie eine horizontale Skalierung über `worker_pool_size`**  
  Passen Sie den `max_worker_pool_size` dynamisch an die CPU-Kerne an. Eine bewährte Formel ist: `worker_pool_size = Anzahl_CPU_Kerne * 2`. Beispiel: Auf einem 16-Kern-Server setzen Sie den Wert auf `32`. Dies maximiert den Durchsatz ohne Overhead durch Kontextwechsel.
