# focusinbox-eight - FOC_030

## Systemarchitektur

focusinbox-eight ist eine hochperformante, ereignisgesteuerte E-Mail-Priorisierungs-Engine, die auf einer Microservices-Architektur mit **Go 1.22** und **PostgreSQL 16** aufbaut. Die Latenz für die Klassifizierung einer eingehenden Nachricht liegt unter **12 ms** (p99). Das System verarbeitet **50.000 Transaktionen pro Sekunde** auf einer einzelnen `c6i.4xlarge`-Instanz.

## Technische Kernfunktionen

### 1. **Dynamische Intent-Klassifizierung mit Multi-Modell-Ensemble**

Das Herzstück ist ein **BERT-basiertes Ensemble** aus drei Modellen: `distilbert-base-german-cased`, `xlm-roberta-base` und einem eigens trainierten **LightGBM-Classifier** für Metadaten (Absender-Rang, Reply-Chain-Länge, Anhangs-MIME-Typ).  
Die Inferenz erfolgt **asynchron über RabbitMQ**; die Modelle laufen auf **GPU-beschleunigten Pods (NVIDIA T4)**.  
Parameter: `--ensemble-threshold 0.87`, `--fallback-model fasttext-german`.  
Technische Metrik: **AUC-ROC > 0.94** auf dem internen `focusinbox_de_v3`-Testset.

### 2. **Autonome Smart-Folder-Generierung mit Temporal Workflows**

Statt statischer Ordner erzeugt focusinbox-eight **dynamische Smart-Folder** auf Basis von **Temporal.io Workflows**. Jeder Workflow analysiert den **temporalen Verlauf** (z. B. „alle E-Mails von Freitag 14–16 Uhr mit Anhang > 5 MB“).  
Die Ordnerstruktur wird als **JSONB in PostgreSQL** persistiert und über **WebSocket-Push** an den Client gesendet.  
Parameter: `--max-folders 12`, `--auto-archive-after 90d`.  
Technische Metrik: **Ordner-Neugenerierung in < 200 ms** bei 10.000 E-Mails.

### 3. **Zero-Downtime-Index-Synchronisation mit Change Data Capture (CDC)**

Alle E-Mail-Änderungen (Lesen, Löschen, Verschieben) werden über **Debezium + Kafka Connect** als **CDC-Stream** erfasst. Der **Inverted Index** (Tantivy-basiert) wird inkrementell aktualisiert – **kein Vollindex nötig**.  
Parameter: `--cdc-batch-size 5000`, `--index-refresh-interval 30s`.  
Technische Metrik: **Index-Lag < 1 Sekunde** bei 1.000 Änderungen pro Sekunde.

## Detaillierte Einrichtung und Betrieb

### Schritt 1: Datenbank-Schema und CDC-Pipeline initialisieren

```bash
# PostgreSQL 16 mit Logical Replication aktivieren
psql -U focusinbox -d focusinbox_db -c "CREATE PUBLICATION focusinbox_pub FOR ALL TABLES;"

# Debezium Connector via Kafka Connect bereitstellen
curl -X POST http://localhost:8083/connectors \
  -H "Content-Type: application/json" \
  -d '{
    "name": "focusinbox-cdc-connector",
        "config": {
            "connector.class": "io.debezium.connector.postgresql.PostgresConnector",
            "database.hostname": "postgres-primary",
            "database.port": "5432",
            "database.user": "focusinbox",
            "database.password": "${DB_PASSWORD}",
            "database.dbname": "focusinbox_db",
            "plugin.name": "pgoutput",
            "slot.name": "focusinbox_slot",
            "publication.name": "focusinbox_pub",
            "transforms": "unwrap,reroute",
            "transforms.unwrap.type": "io.debezium.transforms.ExtractNewRecordState",
            "transforms.reroute.type": "org.apache.kafka.connect.transforms.RegexRouter",
            "transforms.reroute.regex": "focusinbox_db.public.(.*)",
            "transforms.reroute.replacement": "focusinbox.cdc.$1"
        }
  }'
```

### Schritt 2: Ensemble-Modell-Server starten (GPU-beschleunigt)

```bash
# Triton Inference Server mit Model-Repository starten
docker run --gpus all -d --name focusinbox-triton \
  -p 8000:8000 -p 8001:8001 -p 8002:8002 \
  -v /data/models:/models \
  nvcr.io/nvidia/tritonserver:24.01-py3 \
  tritonserver --model-repository=/models \
    --model-control-mode=explicit \
    --load-model=distilbert-german \
    --load-model=xlm-roberta-base \
    --load-model=lightgbm-metadata \
    --ensemble-timeout-microseconds=5000 \
    --backend-config=tensorflow,version=2

# Modelle laden und Ensemble-Konfiguration prüfen
curl -X POST http://localhost:8000/v2/repository/index
```

### Schritt 3: focusinbox-eight Worker-Node starten (mit Temporal Worker)

```bash
# Temporal Worker für Smart-Folder-Workflows starten
./focusinbox-worker --temporal-host=temporal-frontend:7233 \
  --temporal-namespace=focusinbox-prod \
  --pg-conn-string="postgres://focusinbox:${DB_PASSWORD}@pgpool:5432/focusinbox_db?sslmode=require" \
  --kafka-brokers=kafka-1:9092,kafka-2:9092 \
  --ensemble-endpoint=triton:8001 \
  --ensemble-threshold=0.87 \
  --max-folders=12 \
  --auto-archive-after=90d \
  --index-path=/data/tantivy-index \
  --cdc-topic=focusinbox.cdc.emails \
  --log-level=debug

# Worker-Health-Check via gRPC
grpcurl -plaintext localhost:50051 list focusinbox.v1.WorkerService
```

## Best Practices aus dem Produktionsbetrieb

### Praxis 1: **CDC-Lag mit Prometheus-Alerting überwachen**

Richten Sie einen **Alertmanager** ein, der feuert, wenn der Kafka-Lag für den `focusinbox.cdc.emails`-Topic **> 5 Sekunden** beträgt.  
Verwenden Sie dazu den **Kafka Exporter** und folgende PromQL:

```promql
(kafka_consumer_lag{group="focusinbox-cdc-group", topic="focusinbox.cdc.emails"} > 5000)
```

Setzen Sie zudem einen **Dead-Letter-Queue (DLQ)** für fehlgeschlagene CDC-Ereignisse auf. Dies verhindert einen Backpressure-Stau auf der Index-Ebene.

### Praxis 2: **Ensemble-Modell-A/B-Testing mit Canary-Deployment**

Führen Sie neue Modellversionen immer als **Canary** ein: Starten Sie einen zweiten Worker mit `--ensemble-version=canary` und leiten Sie **5 % des Traffics** über einen **Traefik-Router** mit gewichteten Services.  
Überwachen Sie die **AUC-ROC** und die **p99-Latenz** in Grafana. Erst nach **24 h stabiler Performance** (Abfall < 0,5 %) wird die Canary-Version zum neuen Stable-Promoted.
