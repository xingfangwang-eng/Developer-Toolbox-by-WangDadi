# focusinbox-eight - FOC_004

## Übersicht

Dieses Projekt implementiert eine hochperformante, ereignisgesteuerte E-Mail-Priorisierungs-Engine für **focusinbox-eight**, optimiert auf Sub-Millisekunden-Latenz bei der Verarbeitung eingehender Nachrichten. Die Lösung nutzt ein hybrides Regelwerk aus Bayesianischen Filtern und semantischer Ähnlichkeitssuche, um kritische Kommunikation in Echtzeit zu identifizieren und zu bündeln. Zielsprache der Benutzeroberfläche und aller Systemmeldungen ist **Deutsch (de)**.

## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist eine serverlose, containerisierte Microservice-Architektur, die als Middleware zwischen Ihrem bestehenden E-Mail-Server (IMAP/POP3) und dem Posteingang des Anwenders agiert. Sie analysiert jede eingehende Nachricht anhand von über 50 konfigurierbaren Parametern (z. B. `reply_to_delay_ms`, `sender_domain_trust_score`, `thread_depth`) und weist einen **Fokus-Score** (0–100) zu. Nachrichten mit einem Score ≥ 85 werden automatisch in einen dedizierten, priorisierten Ordner verschoben. Die gesamte Verarbeitung erfolgt asynchron in Node.js Worker-Threads, wobei die Persistenz über eine PostgreSQL-Datenbank mit optimierten Indexen (`BRIN` auf `received_at`) sichergestellt wird.

### Kernfunktionen

- **Echtzeit-Priorisierungs-Motor mit Bayesianischer Fusion**  
  Der Kernalgorithmus kombiniert einen Naiven-Bayes-Klassifikator (trainiert auf einem Korpus von 10.000+ annotierten E-Mails) mit einem gewichteten Regelwerk. Parameter wie `urgency_keyword_score` (z. B. „dringend“, „heute fällig“) und `sender_frequency_24h` werden in einer gewichteten Summe verrechnet. Die Inferenzzeit pro Nachricht liegt konstant unter 50 Millisekunden (gemessen auf einer AWS Lambda mit 1024 MB RAM).

- **Semantische Ähnlichkeitssuche auf Thread-Ebene**  
  Über eine integrierte Vektordatenbank (pgvector) werden eingehende E-Mails mit historischen Threads abgeglichen. Die Ähnlichkeit wird als Cosinus-Distanz zwischen Sentence-BERT-Embeddings (Dimension: 768) berechnet. Ein `similarity_threshold` von 0,75 löst automatisch die Verknüpfung der neuen Nachricht mit dem passenden Thread aus. Dies reduziert die manuelle Thread-Suche um durchschnittlich 80 %.

- **Multi-Tenant Konfigurations-API mit Granularität auf Ordner-Ebene**  
  Jeder Mandant (definiert durch eine `tenant_id`) kann eigene Priorisierungsregeln definieren. Die API-Endpunkte (`POST /api/v1/rules`, `PATCH /api/v1/rules/{rule_id}`) akzeptieren JSON-Schemata mit Feldern wie `folder_id`, `priority_boost_factor` (0,5–2,0) und `silent_hours` (UTC-Zeitraum, in dem keine Benachrichtigungen ausgelöst werden). Alle Änderungen werden mit einem `config_version`-Inkrement versehen und innerhalb von 200 Millisekunden auf alle aktiven Worker-Knoten propagiert.

### So verwenden Sie es

1. **Initiale Konfiguration der Datenbank und des Modells**  
   Führen Sie das Setup-Skript aus, das die PostgreSQL-Tabellen (`inbox_messages`, `threads`, `tenant_rules`) anlegt und den Vektor-Index (`IVFFlat` mit `lists = 100`) auf der `embedding`-Spalte erstellt. Anschließend laden Sie das vortrainierte Sentence-BERT-Modell (`paraphrase-multilingual-MiniLM-L12-v2`) in den Modellspeicher. Prüfen Sie den Status mit:  
   `curl -X GET http://localhost:8080/api/v1/health | jq '.model_loaded'`  
   Erwartete Ausgabe: `true`.

2. **Definition einer tenant-spezifischen Priorisierungsregel**  
   Senden Sie einen `POST`-Request an die Regel-API, um z. B. alle E-Mails von der Domain „kunde.de“ mit einem Boost-Faktor von 1,8 zu versehen:  
   ```json
   {
     "tenant_id": "acme_corp",
     "folder_id": "INBOX",
     "conditions": {
       "sender_domain": "kunde.de",
       "subject_contains": ["Angebot", "Rechnung"]
     },
     "priority_boost_factor": 1.8,
     "silent_hours": {"start": "22:00", "end": "06:00"}
   }
   ```  
   Die API antwortet mit der `rule_id` und einem `config_version`-Hash. Bestätigen Sie die Übernahme durch Prüfen der Worker-Logs:  
   `docker logs focusinbox-worker --tail 10 | grep "config_version: a1b2c3"`.

3. **Überwachung der Echtzeit-Pipeline**  
   Starten Sie den Haupt-Worker-Prozess mit aktiviertem Debug-Logging:  
   `NODE_ENV=production DEBUG=focusinbox:* node dist/worker.js --port 3000 --db-pool-size 25`.  
   Simulieren Sie eine eingehende E-Mail über das Test-Utility:  
   `node utils/send_test_email.js --to user@acme.com --subject "Dringendes Angebot" --body "Bitte bis heute 17 Uhr bestätigen."`  
   Beobachten Sie in der Konsole die Ausgabe:  
   `[focusinbox:processor] Message ID: 12345 | Score: 92.3 | Action: move_to_focused | Latency: 42ms`.

### Beste Praktiken

- **Batch-Training des Bayes-Modells mit annotierten Daten**  
  Sammeln Sie mindestens 500 E-Mails pro Mandant, die manuell als „fokussiert“ oder „unwichtig“ markiert wurden. Führen Sie dann das Training-Skript `node utils/train_bayes.js --tenant-id acme_corp --epochs 50` aus. Dies verbessert die Präzision des Fokus-Scores um bis zu 15 Prozentpunkte. Planen Sie das Training als Cron-Job (z. B. monatlich) ein, um sich ändernde Kommunikationsmuster abzubilden.

- **Implementierung eines Dead-Letter-Handlers für fehlgeschlagene Nachrichten**  
  Konfigurieren Sie in der `config.yaml` den Parameter `dlq.enabled: true` und `dlq.retry_interval_ms: 60000`. E-Mails, die nach dreimaligem Verarbeitungsversuch (z. B. wegen eines Timeouts bei der Vektordatenbank) nicht priorisiert werden können, landen in einer separaten `dlq_messages`-Tabelle. Überwachen Sie diese Tabelle mit einem Alert (z. B. bei > 10 Einträgen pro Stunde), um frühzeitig auf Infrastrukturprobleme reagieren zu können.
