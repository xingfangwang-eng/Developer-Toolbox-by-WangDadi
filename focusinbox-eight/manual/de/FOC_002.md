# focusinbox-eight - FOC_002

## Übersicht

**focusinbox-eight** ist eine hochperformante, auf Mikroservices basierende E-Mail-Intelligence-Engine, die speziell für die automatisierte Inhaltsklassifikation und Priorisierung im deutschsprachigen Geschäftsumfeld entwickelt wurde. Durch die Kombination von NLP-basierter Intent-Erkennung und einer konfigurierbaren Regel-Engine ermöglicht das Tool eine präzise Filterung und Weiterleitung von eingehenden Nachrichten mit einer Latenz von unter 50 ms pro Transaktion.

## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist ein plattformunabhängiges Backend-System, das als Middleware zwischen dem E-Mail-Provider (IMAP/POP3) und dem CRM-System operiert. Es analysiert jede eingehende Nachricht in Echtzeit auf ihre semantische Relevanz, extrahiert Schlüsselparameter wie `customer_id`, `priority_score` und `language_code`, und leitet sie an die entsprechende Workflow-Pipeline weiter. Die Engine unterstützt sowohl asynchrone Batch-Verarbeitung als auch synchrone Webhook-Integrationen.

### Kernfunktionen

1. **Dynamisches Intent-Mapping mit PostgreSQL-gestützter Regelhierarchie**  
   - Parameter: `intent_id`, `rule_weight`, `timeout_ms` (Standard: 200 ms)  
   - Technische Basis: PostgreSQL 15+ mit GIN-Index für Volltextsuche und JSONB-Speicherung von Regelbäumen.  
   - Jede eingehende Nachricht wird gegen bis zu 10.000 aktive Regeln in unter 15 ms gematcht.

2. **Realzeit-Konfliktanalyse mit Multi-Threading und Lock-freier Queue**  
   - Parameter: `conflict_threshold` (float, 0.0–1.0), `queue_depth` (int, Standard: 1024)  
   - Technische Basis: Go-Routinen mit atomaren Zählern und einer ringbasierten Lock-freien Queue (SPSC).  
   - Ermöglicht parallele Verarbeitung von bis zu 500 Nachrichten pro Sekunde bei einer CPU-Auslastung unter 60 %.

3. **Semantische Priorisierung mittels Transformer-basierter Embedding-Vektoren**  
   - Parameter: `embedding_dim` (int, Standard: 768), `similarity_metric` (cosine/euclidean)  
   - Technische Basis: ONNX-Runtime mit quantisierten Modellen (FP16), integriert in eine C++-Native-Bridge.  
   - Berechnung der semantischen Distanz zwischen Nachricht und vordefinierten Kategorien erfolgt in unter 30 ms pro Einheit.

### So verwenden Sie es

1. **Initialisierung der Regelbasis**  
   - Führen Sie `focusinbox-eight init --db-host=localhost --db-port=5432 --db-name=focusinbox` aus.  
   - Das Tool erstellt automatisch die Tabellen `rules`, `intents` und `audit_log` mit vordefinierten Indizes.  
   - Überprüfen Sie die Verbindung mit `focusinbox-eight status --check-db`.

2. **Konfiguration der Intent-Pipeline**  
   - Erstellen Sie eine YAML-Datei `pipeline.yaml` mit folgendem Schema:  
     ```yaml
     intents:
       - name: "support_request"
         weight: 0.9
         rules:
           - { field: "subject", match: "Problem|Fehler|Störung", operator: "regex" }
           - { field: "from", match: "@kunde\.de$", operator: "regex" }
     ```  
   - Laden Sie die Konfiguration: `focusinbox-eight apply --config pipeline.yaml --force-reload`.

3. **Start des Echtzeit-Monitors**  
   - Aktivieren Sie den IMAP-Listener: `focusinbox-eight watch --imap-server=imap.example.com --imap-user=admin --poll-interval=5000`.  
   - Das Tool öffnet eine persistenten WebSocket auf Port 8080, über den Sie Metriken wie `throughput`, `avg_latency_ms` und `queue_utilization` abrufen können.  
   - Testen Sie die Pipeline mit einer Test-E-Mail: `focusinbox-eight test --from=test@kunde.de --subject="Fehler im Modul X"`.

### Beste Praktiken

- **Optimierung der Regelbasis durch A/B-Testing**  
  Nutzen Sie die integrierte `--dry-run`-Option, um neue Regeln gegen historische Daten zu testen, bevor Sie sie in der Produktion aktivieren. Führen Sie für jede Regelversion einen separaten Audit-Log-Export durch (`focusinbox-eight export --format=csv --since=7d`), um die Trefferquote (`precision`) und die Fehlalarmrate (`false_positive_rate`) zu vergleichen.

- **Skalierung durch horizontale Partitionierung**  
  Bei einem Nachrichtenaufkommen über 10.000 E-Mails pro Stunde empfiehlt es sich, mehrere Instanzen von focusinbox-eight mit einer gemeinsamen PostgreSQL-Datenbank zu betreiben. Nutzen Sie dazu den Parameter `--shard-key=customer_id` und setzen Sie einen Load-Balancer (z. B. HAProxy) vor die WebSocket-Endpunkte. Überwachen Sie die `queue_depth` jeder Instanz und skalieren Sie dynamisch mittels Kubernetes HPA.
