# focusinbox-eight - FOC_006

## Übersicht

Dieses Repository enthält die hochperformante Implementierung des focusinbox-eight Moduls, einer asynchronen Echtzeit-Postfach-Orchestrierungs-Engine. Kernstück ist die latenzoptimierte Verarbeitung von eingehenden Nachrichten-Streams mit garantierter Zustellreihenfolge (FIFO) und automatischer Deduplizierung auf Basis von `message_id` und `timestamp_ns`. Zielsprache: Deutsch (de).

## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist eine spezialisierte Middleware-Schicht zur Aggregation und Priorisierung von Nachrichten aus verteilten Quellen (z. B. Webhooks, Message Queues, Datenbank-Change-Streams). Das System arbeitet mit einem internen `PriorityQueue`-Algorithmus (O(log n) Insertion) und einem konfigurierbaren `BatchProcessor`, der Nachrichten in Mikro-Batches (Standard: 256 KB oder 10.000 Datensätze) zusammenfasst, bevor sie an die Ziel-API (`/api/v2/inbox/commit`) übergeben werden. Die maximale Verarbeitungslatenz pro Batch beträgt garantiert unter 50 ms (gemessen im P99-Perzentil).

### Kernfunktionen

- **Deterministische Nachrichten-Deduplizierung & FIFO-Garantie**  
  Nutzt einen internen `LRU-Cache` (max. 500.000 Einträge) mit `ttl`-Parameter (Standard: 3600 Sekunden), um Dubletten anhand der Felder `message_id` (UUID v7) und `origin_timestamp` (Unix-Nanosekunden) zu erkennen. Die `SequenceNumber`-Verwaltung stellt sicher, dass Nachrichten innerhalb einer Partition (`partition_key`) strikt in der Reihenfolge ihres Eintreffens verarbeitet werden. Konfigurierbar über `config.yaml` unter `dedup.strategy: "strict_fifo"`.

- **Dynamische Lastverteilung mit Backpressure-Steuerung**  
  Implementiert einen adaptiven `RateLimiter`, der die eingehende Nachrichtenrate (`msg/sec`) anhand des aktuellen `ConsumerLag` in der PostgreSQL-Datenbank (`pg_stat_replication`) regelt. Der `backpressure_threshold` (Standard: 10.000 Nachrichten) löst automatisch eine Drosselung aus. Der `Retry-Mechanismus` verwendet exponentielles Backoff (Basis: 200 ms, Faktor: 2, max. 5 Versuche) mit einem separaten `DeadLetterQueue`-Channel für fehlgeschlagene Nachrichten.

- **Multi-Format-Transformation & Schema-Validierung**  
  Unterstützt die automatische Konvertierung von eingehenden Payloads (JSON, Avro, Protobuf) in ein einheitliches internes `InboxMessage`-Objekt. Die Validierung erfolgt gegen ein JSON Schema (Version 2020-12), das unter `schemas/inbox_message_v3.json` abgelegt ist. Fehlerhafte Nachrichten werden mit einem detaillierten `ValidationError`-Code (z. B. `ERR_SCHEMA_MISMATCH_102`) und der exakten Zeilennummer im Batch geloggt.

### So verwenden Sie es

1. **Initialisierung der Datenbankverbindung und Cache-Konfiguration**  
   Führen Sie `focusinbox-eight --init-db --db-host=127.0.0.1 --db-port=5432 --db-name=focusinbox_prod --cache-ttl=7200` aus. Dies erstellt die erforderlichen PostgreSQL-Tabellen (`inbox_queue`, `dead_letter_log`, `processing_metrics`) und initialisiert den internen LRU-Cache mit der angegebenen TTL. Überprüfen Sie den Erfolg mit `SELECT count(*) FROM processing_metrics WHERE status = 'ready';`.

2. **Starten des Echtzeit-Listener-Daemons**  
   Starten Sie den Hauptprozess mit `nohup ./focusinbox-eight --mode=daemon --listen-port=9090 --workers=4 --batch-size=5000 --batch-interval=100 2>&1 &`. Der Daemon öffnet einen TCP-Socket auf Port 9090, akzeptiert eingehende Nachrichten und verarbeitet sie in Batches von maximal 5000 Stück oder alle 100 Millisekunden (je nachdem, was zuerst eintritt). Der Log-Level `--verbose=debug` zeigt jede Batch-Kommittierung und den aktuellen `ConsumerLag` an.

3. **Manuelles Einspielen eines Nachrichten-Batches über die REST-API**  
   Senden Sie einen Batch mit `curl -X POST http://localhost:9090/api/v2/inbox/commit -H "Content-Type: application/json" -d '{"messages": [{"message_id": "01912345-6789-7abc-def0-123456789abc", "partition_key": "user_42", "payload": {"action": "update", "data": {"id": 123}}, "origin_timestamp": 1712345678000000000}], "batch_id": "batch_001"}'`. Die API antwortet mit einem `BatchCommitResponse`-Objekt, das die Felder `accepted_count`, `duplicate_count`, `error_count` und `processing_latency_ms` enthält. Bei Fehlern (z. B. `error_count > 0`) prüfen Sie das Log unter `/var/log/focusinbox-eight/error.log`.

### Beste Praktiken

- **Praxis 1: Partitionierung nach Geschäftstransaktionen**  
  Wählen Sie den `partition_key` so, dass er eine logische Geschäftseinheit repräsentiert (z. B. `user_id`, `order_id` oder `session_id`). Vermeiden Sie zu feine Partitionierung (z. B. `message_id`), da dies die Effizienz des Batch-Prozessors reduziert. Eine Faustregel: Jede Partition sollte mindestens 100 Nachrichten pro Sekunde erhalten, um die FIFO-Garantie ohne Overhead zu gewährleisten. Überwachen Sie die Partition-Größe mit der Metrik `partition_depth` im Prometheus-Endpoint (`/metrics`).

- **Praxis 2: Implementierung eines idempotenten Consumers**  
  Konfigurieren Sie den Ziel-Service (der die committed Batches empfängt) so, dass er idempotent gegenüber doppelten `batch_id`-Werten ist. Nutzen Sie dazu das Feld `batch_id` als Primärschlüssel oder Unique-Constraint in der Zieldatenbank. Kombinieren Sie dies mit dem `dedup.strategy: "strict_fifo"` auf der focusinbox-eight-Seite, um eine exactly-once-Semantik zu erreichen. Testen Sie dies mit einem Chaos-Monkey, der zufällig `POST`-Requests dupliziert.
