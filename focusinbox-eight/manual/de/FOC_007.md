# focusinbox-eight - FOC_007

## Übersicht

focusinbox-eight ist eine hochperformante Echtzeit-Postfach-Synchronisations-Engine für Multi-Tenant-Umgebungen. Das System wurde für die gleichzeitige Verarbeitung von bis zu 50.000 IMAP- und Microsoft Graph-API-Verbindungen optimiert. Kern der Architektur ist ein auf Rust-basierender Worker-Pool, der mittels `tokio`-Asynchronität und `rayon`-Parallelität eine Latenz von unter 12 ms pro Postfach-Refresh erreicht. Die gesamte Konfiguration erfolgt über eine deklarative YAML-Schnittstelle (`focusinbox-eight.yml`), die automatisch durch einen integrierten Schema-Validator (`zod`-basiert) auf Korrektheit geprüft wird.

## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist ein Postfach-Aggregator der nächsten Generation. Es konsolidiert eingehende E-Mails aus beliebig vielen Konten, klassifiziert sie mittels eines lokal laufenden Transformer-Modells (distilbert-base-german-cased) in Echtzeit und speichert die Ergebnisse in einer PostgreSQL-Datenbank mit TimescaleDB-Erweiterung für zeitseriengestützte Analysen. Die Kommunikation zwischen den Microservices erfolgt ausschließlich über gRPC mit TLS 1.3 und mutual Authentication. Das System ist für eine Ausfallsicherheit von 99,99 % ausgelegt und unterstützt horizontale Skalierung mittels Kubernetes HPA (Horizontal Pod Autoscaler) auf Basis von CPU- und Memory-Metriken.

### Kernfunktionen

1. **Multi-Protokoll-Adapter mit automatischer Failover-Erkennung**  
   Der Adapter unterstützt IMAP (RFC 3501), Microsoft Graph API (v1.0) und Exchange Web Services (EWS). Jeder Adapter wird als separater Tokio-Task gestartet. Bei Verbindungsabbrüchen erfolgt automatisch ein exponentieller Backoff (initial 1s, max 300s) mit Jitter. Die Verbindungsstatistiken werden in einer dedizierten InfluxDB-Datenbank mit einer Granularität von 100 ms erfasst. Parameter: `imap.connection_pool_size` (Standard: 32), `graph_api.retry_policy.max_attempts` (Standard: 5).

2. **Echtzeit-Intent-Klassifikation mit lokalem ONNX-Runtime-Modell**  
   Jede eingehende E-Mail wird nach dem Parsen (mittels `mailparse`-Crate) in einen Vektor der Dimension 768 umgewandelt und durch ein quantisiertes ONNX-Modell (INT8) gejagt. Die Inferenzzeit beträgt durchschnittlich 4,2 ms auf einer Intel Xeon Platinum 8375C. Die Klassifikationsergebnisse werden in einer `jsonb`-Spalte der Tabelle `emails` gespeichert. Parameter: `model.quantization` (Werte: `int8`, `fp16`, `fp32`), `inference.batch_size` (Standard: 16).

3. **Granulare Rechteverwaltung mittels RBAC mit PostgreSQL Row-Level Security**  
   Jeder Tenant erhält eine eigene PostgreSQL-Schema-Isolation. Die Zugriffsrechte werden über eine zentrale Policy-Tabelle gesteuert, die mittels `CREATE POLICY` auf Zeilenebene durchgesetzt wird. Die Authentifizierung erfolgt über JWT-Tokens mit einer maximalen Gültigkeit von 15 Minuten. Parameter: `rbac.policy_table` (Standard: `iam.policies`), `jwt.issuer` (Standard: `focusinbox-eight-auth`).

### So verwenden Sie es

1. **Initialisierung der Datenbank und des Adapters**  
   Führen Sie `cargo run --bin focusinbox-eight-init -- --config focusinbox-eight.yml` aus. Dies erstellt automatisch die benötigten PostgreSQL-Tabellen (`tenants`, `mailboxes`, `emails`) und startet den IMAP-Adapter mit den in der Konfiguration angegebenen Credentials. Die Ausgabe enthält eine UUID für den initialen Tenant.

2. **Konfiguration der Intent-Klassifikation**  
   Bearbeiten Sie die Datei `focusinbox-eight.yml` und setzen Sie unter `classifier.model_path` den Pfad zu Ihrem quantisierten ONNX-Modell. Führen Sie dann `curl -X POST https://localhost:8443/api/v1/classifier/reload` aus, um das Modell ohne Neustart zu laden. Überprüfen Sie den Status mit `curl https://localhost:8443/api/v1/classifier/status`. Die Antwort enthält den Parameter `inference_latency_p99`.

3. **Einrichtung der RBAC-Policies**  
   Verwenden Sie das CLI-Tool `focusinbox-eight-rbac` mit dem Befehl `./focusinbox-eight-rbac policy create --tenant-id <UUID> --role admin --action read`. Die Policy wird sofort wirksam. Testen Sie den Zugriff mit `curl -H "Authorization: Bearer <JWT>" https://localhost:8443/api/v1/emails`. Bei fehlenden Rechten erhalten Sie einen HTTP 403 mit dem Fehlercode `ERR_RBAC_ACCESS_DENIED`.

### Beste Praktiken

1. **Optimierung der Verbindungspool-Größe**  
   Setzen Sie `imap.connection_pool_size` auf das 1,5-fache der Anzahl Ihrer gleichzeitig aktiven Postfächer. Beispiel: Bei 200 Postfächern → `connection_pool_size: 300`. Dies verhindert TCP-Überlastung und reduziert die Latenz um bis zu 40 %. Überwachen Sie die Metrik `imap.connection_pool_utilization` im Prometheus-Endpoint (`/metrics`).

2. **Regelmäßige Modell-Nachtrainings**  
   Führen Sie monatlich ein Fine-Tuning des Klassifikationsmodells auf Ihren spezifischen E-Mail-Datensatz durch. Nutzen Sie dazu das bereitgestellte Skript `scripts/fine_tune.py` mit `--epochs 3 --learning_rate 2e-5`. Das neu trainierte Modell wird automatisch in das ONNX-Format exportiert und kann ohne Ausfallzeit via API geladen werden. Die Genauigkeit steigt erfahrungsgemäß um 12–18 %.
