# focusinbox-eight - FOC_023

## Übersicht

Dieses Repository implementiert die **FOC_023**-Architektur für eine hochperformante, ereignisgesteuerte E-Mail-Priorisierungs-Engine. Ziel ist die Reduzierung der durchschnittlichen Reaktionslatenz auf kritische eingehende Nachrichten um mindestens 40 % durch den Einsatz von Multi-Layer-Filtering und kontextueller Gewichtung. Die Lösung ist vollständig containerisiert und für den Betrieb in einer Kubernetes-Umgebung mit PostgreSQL 15+ und Redis Stack optimiert.

## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist eine serverlose, KI-gestützte Microservice-Plattform zur automatischen Kategorisierung und Priorisierung von E-Mail-Volumen in Echtzeit. Das System analysiert eingehende Nachrichten anhand von über 200 parametrisierbaren Merkmalen (z. B. `sender_reputation_score`, `thread_depth`, `attachment_mime_type`) und berechnet einen dynamischen **Fokus-Index** (0.0 - 1.0). Dieser Index bestimmt die endgültige Sortierung im Posteingang des Nutzers. Die Entscheidungslogik ist vollständig deklarativ in YAML-Dateien konfigurierbar und benötigt keine Neukompilierung.

### Kernfunktionen

- **Multi-Layer-Klassifikation mit kontextueller Gewichtung**  
  Das System führt eine gestaffelte Analyse durch: Zuerst eine Regel-basierte Vorverarbeitung (Spam-Erkennung, DLP-Checks), gefolgt von einer ML-basierten semantischen Analyse mittels eines auf `DistilBERT` feinabgestimmten Modells. Die Gewichtung der Layer ist über den Parameter `classification.weight.rule_based` (Standard: 0.4) und `classification.weight.semantic` (Standard: 0.6) einstellbar. Die finale Priorisierung erfolgt in **Echtzeit unter 50 Millisekunden** pro Nachricht bei einer Last von bis zu 1000 Nachrichten pro Sekunde.

- **Dynamische Eskalations-Engine mit PostgreSQL-Integration**  
  Basierend auf dem berechneten Fokus-Index und benutzerdefinierten Schwellwerten (`escalation.threshold.urgent`: 0.85, `escalation.threshold.attention`: 0.65) kann die Engine automatisch Aktionen auslösen. Diese werden als asynchrone Jobs in einer PostgreSQL-Tabelle `escalation_queue` persistiert. Die Tabelle nutzt partielle Indizes (`WHERE status = 'pending'`) und `LISTEN/NOTIFY` für eine sub-10ms Benachrichtigung der Worker-Pods. Ein integrierter Circuit-Breaker verhindert Eskalationsstürme bei Systemüberlastung.

- **Granulare Audit-Trail-Erstellung mit Datenlebenszyklus-Management**  
  Jede durchgeführte Klassifikation und Eskalation wird als strukturierter Log-Eintrag in einer separaten PostgreSQL-Tabelle `audit_trail` gespeichert. Die Einträge enthalten den vollständigen Entscheidungsbaum (inkl. `feature_vector`, `intermediate_scores`, `applied_rules`). Ein konfigurierbarer TTL-Mechanismus (`audit.retention_days`: 90) löscht alte Einträge automatisch mittels eines `pg_cron`-Jobs, um die Datenbankgröße unter 500 GB zu halten.

### So verwenden Sie es

1. **Initiale Konfiguration und Datenbank-Seeding**  
   Führen Sie `make db-init` aus. Dies erstellt die erforderlichen PostgreSQL-Datenbanken (`focusinbox_core`, `focusinbox_audit`) und führt die Migrationsskripte aus (`migrations/001_create_escalation_queue.sql`, `migrations/002_create_audit_trail.sql`). Anschließend seeden Sie die initialen Regelsätze mit `./scripts/seed_rules.py --config configs/rules/default.yaml`. Der Befehl validiert die YAML-Syntax und schreibt die Regeln in die Tabelle `classification_rules`.

2. **Starten des Klassifikations-Microservice**  
   Starten Sie den Service mittels `docker-compose up -d classifier`. Der Service lauscht auf Port `8080` und akzeptiert POST-Anfragen an den Endpunkt `/api/v1/classify`. Ein gültiger Payload muss das Feld `email_body` (String) und optional `metadata` (JSON-Objekt mit `sender`, `subject`, `thread_id`) enthalten. Beispiel:  
   ```bash
   curl -X POST http://localhost:8080/api/v1/classify \
     -H "Content-Type: application/json" \
     -d '{"email_body": "Serverausfall im Rechenzentrum B", "metadata": {"sender": "admin@firma.de", "subject": "KRITIS: Ausfall"}}'
   ```  
   Die Antwort enthält den `focus_index` und die Liste der ausgelösten `actions`.

3. **Überwachen und Anpassen der Eskalationsschwellwerte**  
   Die aktuellen Schwellwerte werden im laufenden Betrieb über den Endpunkt `/api/v1/config/escalation` abgefragt (`GET`). Zum Ändern senden Sie einen `PATCH`-Request mit den neuen Werten. Beispiel:  
   ```bash
   curl -X PATCH http://localhost:8080/api/v1/config/escalation \
     -H "Content-Type: application/json" \
     -d '{"threshold_urgent": 0.90, "threshold_attention": 0.70}'
   ```  
   Die Änderung wird sofort wirksam und im `audit_trail` protokolliert.

### Beste Praktiken

- **Praxis 1: Iterative Kalibrierung der Feature-Gewichte**  
  Vermeiden Sie es, die Standardgewichte blind zu übernehmen. Nutzen Sie den bereitgestellten `evaluate.py`-Skript, um einen historischen Datensatz (z. B. die letzten 30 Tage) gegen Ihre manuell priorisierten E-Mails zu validieren. Passen Sie die Gewichte in der `configs/weights.yaml` an, bis die Precision-Recall-Kurve für Ihre kritischste Klasse (z. B. `urgent`) ein F1-Score > 0.92 erreicht. Eine Überanpassung auf einzelne Absender vermeiden Sie durch die Aktivierung des Regularisierungsparameters `feature.regularization.l1_lambda`.

- **Praxis 2: Geplante Rotation der ML-Modelle**  
  Trainieren Sie Ihr semantisches Modell nicht auf einem statischen Datensatz. Implementieren Sie einen CI/CD-Pipeline-Job, der wöchentlich ein neues Modell auf Basis der letzten 7 Tage an Audit-Trail-Daten (mit korrigierten Labels) trainiert. Nutzen Sie den Parameter `model.rollout.canary_percentage` (z. B. 5 %), um das neue Modell zunächst nur auf einem kleinen Traffic-Segment zu testen. Steigen Sie erst auf 100 % Traffic, wenn die Fehlerrate (Anzahl der `rerun_requests`) unter 0.1 % fällt.
