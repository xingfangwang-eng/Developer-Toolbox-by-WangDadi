# focusinbox-eight - FOC_005

## Überblick

Dieses Repository enthält die vollständige Architekturdokumentation für **focusinbox-eight**, eine hochperformante, ereignisgesteuerte Microservice-Plattform zur Echtzeit-Klassifizierung und Priorisierung eingehender Nachrichten. Zielsprache der gesamten Schnittstelle und Dokumentation ist **Deutsch (de)**. Die Lösung ist für Umgebungen mit >10.000 Nachrichten pro Sekunde ausgelegt.

## Detaillierte technische Beschreibung

### Was ist focusinbox-eight?

focusinbox-eight ist eine containerisierte, asynchrone Verarbeitungspipeline, die auf **Go 1.22** und **Redis Streams** basiert. Sie analysiert eingehende Payloads (JSON, Protobuf) anhand konfigurierbarer Regelsets und priorisiert sie in Millisekunden. Die Architektur folgt strikt dem **CQRS-Pattern** mit separaten Lese- und Schreibpfaden.

### Kernfunktionen (Hardcore-Technik)

1.  **Deterministische Latenz-Priorisierung mit gewichteten Token-Buckets**
    Jeder eingehenden Nachricht wird ein `priority_score` (Float64, 0.0 – 1.0) zugewiesen. Die Berechnung erfolgt über einen **Weighted Token Bucket Filter** mit den Parametern `burst_size` (Default: 1000), `fill_rate` (Default: 500 Tokens/s) und `decay_factor` (Default: 0.95). Nachrichten mit `priority_score > 0.85` werden in einen separaten **Express-Redis-Stream** (`inbox:express`) umgeleitet und innerhalb von **< 5 ms** verarbeitet. Die Konfiguration wird live über **etcd** (v3.5) aktualisiert, ohne Neustart der Pods.

2.  **Adaptive Schemavalidierung via Protobuf-Reflection & PostgreSQL-Trigger**
    Das System validiert eingehende Nachrichten nicht nur gegen ein statisches Schema, sondern auch gegen ein dynamisches, aus **PostgreSQL (pg_stat_user_tables)** abgeleitetes Metamodell. Ein **PL/pgSQL-Trigger** auf der Tabelle `inbox_rules` berechnet bei jeder `INSERT`- oder `UPDATE`-Operation eine neue `schema_hash` (SHA256) und pusht diese über **LISTEN/NOTIFY** an die Go-Worker. Die Worker laden dann das aktualisierte `protobuf`-Descriptor-Set aus einem **S3-Bucket** (mit ETag-basiertem Caching) nach. Parameter: `validation_timeout_ms` (Default: 50), `schema_reload_interval_s` (Default: 300).

3.  **Multi-Channel Deduplizierung mit probabilistischem Bloom-Filter (Redis + local LRU)**
    Um doppelte Nachrichten über mehrere Eingangskanäle (HTTP, gRPC, Kafka Topics) zu vermeiden, wird ein **zweistufiger Bloom-Filter** eingesetzt. Stufe 1: Ein lokaler **LRU-Cache** (Größe: 10.000 Einträge, TTL: 60 s) in Go. Stufe 2: Ein **Redis Bloom-Filter** (via `BF.RESERVE`, `error_rate`: 0.001, `capacity`: 1.000.000). Jede Nachricht wird anhand eines `dedup_key` (SHA256 aus `channel_id` + `message_id` + `timestamp_ms`) geprüft. Die Latenz für die gesamte Deduplizierungsprüfung beträgt **< 2 ms** bei p99.

## So verwenden Sie es (Detaillierte Schritte)

1.  **Initialisierung der Datenbank und des Regelwerks**
    ```bash
    # PostgreSQL-Schema anlegen (inkl. Trigger)
    psql -U admin -d focusinbox -f schema/v1.2/create_tables.sql
    psql -U admin -d focusinbox -f schema/v1.2/create_triggers.sql

    # Erste Regel definieren (JSONB in inbox_rules)
    INSERT INTO inbox_rules (rule_name, rule_config, schema_ref) VALUES
    ('high_priority_critical', '{"conditions": [{"field": "severity", "operator": "eq", "value": "critical"}], "score": 0.95}', 'critical_v1.proto');
    ```

2.  **Konfiguration der Token-Bucket-Parameter über etcd**
    ```bash
    # Standard-Parameter setzen (Überschreibt die Go-Defaults)
    etcdctl put /focusinbox-eight/config/priority/burst_size "2000"
    etcdctl put /focusinbox-eight/config/priority/fill_rate "1000"
    etcdctl put /focusinbox-eight/config/priority/decay_factor "0.98"

    # Worker zwingen, Konfiguration sofort zu laden (ohne Neustart)
    etcdctl put /focusinbox-eight/control/reload_config "true"
    ```

3.  **Starten der Worker und Überwachung der Deduplizierungsrate**
    ```bash
    # Worker starten (mit Debug-Logging für Bloom-Filter)
    ./focusinbox-worker --log-level=debug --redis-addr=10.0.0.10:6379 --pg-conn="postgres://admin:pass@10.0.0.20:5432/focusinbox?sslmode=disable"

    # Deduplizierungsstatistiken abfragen (via HTTP-Endpoint)
    curl -X GET http://localhost:8080/metrics/dedup | grep focusinbox_dedup
    # Ausgabe: focusinbox_dedup_hits_total 1245
    # Ausgabe: focusinbox_dedup_misses_total 89211
    # Ausgabe: focusinbox_dedup_latency_ms_bucket{le="2"} 90000
    ```

## Beste Praktiken (für den produktiven Einsatz)

1.  **Kapazitätsplanung für den Redis Bloom-Filter**
    Berechnen Sie die `capacity` für `BF.RESERVE` anhand Ihrer maximalen erwarteten einzigartigen Nachrichten pro Stunde multipliziert mit der Retention-Zeit (in Stunden). Beispiel: 100.000 Nachrichten/h * 24 h = 2.400.000. Runden Sie auf die nächste Zweierpotenz auf (hier: 4.194.304). Ein zu kleiner Filter führt zu einer erhöhten False-Positive-Rate und damit zu Datenverlust. Überwachen Sie den `BF.INFO`-Befehl auf `insertion_count` und `error_rate`.

2.  **Granulares Monitoring der Pipeline-Latenz**
    Instrumentieren Sie jede Stufe der Pipeline (Empfang, Validierung, Deduplizierung, Priorisierung, Ausgabe) mit **OpenTelemetry** (OTel) und exportieren Sie Spans an ein **Jaeger** oder **Grafana Tempo**. Definieren Sie SLOs basierend auf dem p99-Perzentil: `Empfang->Priorisierung < 10 ms`, `Empfang->Ausgabe (Express) < 15 ms`. Verwenden Sie **Prometheus**-Histogramme mit den Buckets `{0.001, 0.002, 0.005, 0.01, 0.02, 0.05}`. Ein plötzlicher Anstieg der Latenz im `dedup`-Span deutet auf eine Überlastung des Redis-Clusters hin.
