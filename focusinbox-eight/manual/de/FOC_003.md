# focusinbox-eight - FOC_003

## Übersicht

**focusinbox-eight** ist eine hochperformante, ereignisgesteuerte E-Mail-Priorisierungs-Engine, die auf Basis von Multi-Faktor-Attribut-Gewichtung und Echtzeit-Latenzanalyse arbeitet. Das System analysiert eingehende Nachrichten in **Sub-Millisekunden (< 0.8 ms pro Transaktion)** und ordnet sie dynamisch in 8 konfigurierbare Fokus-Container (Inbox-Buckets) ein. Die Lösung ist für **PostgreSQL 15+ mit pgvector-Erweiterung** optimiert und unterstützt die gleichzeitige Verarbeitung von **10.000+ IMAP-Streams** pro Knoten.

## Kernfunktionen

1. **Dynamische Bucket-Allokation via Weighted-Score-Engine**  
   Jeder eingehenden E-Mail wird ein **FOC-Score (0.0 – 1.0)** zugewiesen, basierend auf Parametern wie `sender_trust_level`, `subject_entropy`, `attachment_mime_type` und `thread_depth`. Der Algorithmus berechnet die finale Priorität in **Echtzeit über eine C-Rust-Brücke**, die in den PostgreSQL-Trigger integriert ist. Die Konfiguration erfolgt über eine YAML-Datei (`focusinbox.config.yaml`) mit Unterstützung für **Hot-Reload via SIGHUP**.

2. **Semantische Cluster-Analyse mit pgvector**  
   Der Inhalt jeder E-Mail wird durch ein **lokal installiertes Sentence-Transformer-Modell (all-MiniLM-L6-v2)** in einen 384-dimensionalen Embedding-Vektor umgewandelt. Diese Vektoren werden in einer **pgvector-Indextabelle (IVFFlat, Liste: 100)** gespeichert. Die **Bucket-Zuweisung** erfolgt dann durch einen **K-Means-Clustering-Algorithmus (k=8)**, der die **Cosine-Ähnlichkeit** zwischen dem Embedding und den 8 Cluster-Zentroiden misst. Dies ermöglicht eine **themenbasierte Sortierung** ohne manuelle Regeln.

3. **Proaktive Threat-Erkennung und Quarantäne**  
   Eine **zweistufige Pipeline** analysiert Anhänge und Links: Stufe 1 verwendet einen **statischen Pattern-Matcher (YARA-Regeln, 500+ Regeln)** für bekannte Malware-Signaturen. Stufe 2 führt eine **Sandbox-Ausführung** in einem **isolierten Linux-Container (cgroups v2, Speicherlimit: 256 MB)** durch, der die Datei in **Echtzeit (Time-to-Decision < 2 Sekunden)** auf verdächtiges Verhalten prüft. Ergebnisse werden in einer **separaten Quarantäne-Tabelle** mit `risk_score` (0–100) gespeichert.

## So verwenden Sie es

1. **Initiale Konfiguration und Datenbank-Setup**  
   Führen Sie `psql -U postgres -d focusinbox_db -f schema.sql` aus, um die erforderlichen Tabellen (`inbox_items`, `bucket_config`, `embedding_cache`, `quarantine_log`) und die `pgvector`-Erweiterung zu erstellen. Passen Sie anschließend die Datei `/etc/focusinbox/focusinbox.config.yaml` an: Setzen Sie `database.connection_string = "postgresql://user:pass@localhost:5432/focusinbox_db"` und definieren Sie unter `buckets` die 8 Container mit ihren Gewichtungsfaktoren (`weight_sender_trust: 0.4`, `weight_attachment_risk: 0.3`, etc.).

2. **Ersten IMAP-Stream aktivieren und testen**  
   Starten Sie den Daemon: `systemctl start focusinbox-eight`. Überwachen Sie die Logs mit `journalctl -u focusinbox-eight -f`. Verwenden Sie das CLI-Tool `focusctl` für einen manuellen Test: `focusctl test-stream --imap-server imap.gmail.com --username test@example.com --app-password "xxxx"`. Der Befehl gibt nach **ca. 3 Sekunden** eine Liste der 10 neuesten E-Mails mit ihrem berechneten `FOC-Score` und dem zugewiesenen Bucket-Namen aus.

3. **Dynamische Bucket-Grenzen anpassen**  
   Um die Schwellenwerte für die Bucket-Zuweisung zu ändern, bearbeiten Sie die Tabelle `bucket_config` direkt in PostgreSQL:  
   `UPDATE bucket_config SET upper_bound = 0.85 WHERE bucket_name = 'critical';`  
   Die Änderung wird **innerhalb von 5 Sekunden** durch den internen Watchdog-Prozess übernommen. Überprüfen Sie die neue Verteilung: `focusctl stats --show-distribution`. Die Ausgabe zeigt die **Anzahl der E-Mails pro Bucket** sowie die **durchschnittliche Latenz** der letzten 1000 Transaktionen.

## Beste Praktiken

- **Batch-Verarbeitung für hohe Volumen**  
  Bei mehr als 50.000 E-Mails pro Stunde sollten Sie die **Batch-Größe** in `focusinbox.config.yaml` von `batch_size: 100` auf `batch_size: 500` erhöhen. Kombinieren Sie dies mit der Aktivierung der **PostgreSQL-Connection-Pooling**-Funktion (`pgbouncer`), um die Anzahl der gleichzeitigen Datenbankverbindungen auf **max. 50** zu begrenzen. Dies reduziert die CPU-Auslastung des Embedding-Servers um bis zu **40%** und verhindert Timeouts bei der Vektorindizierung.

- **Regelmäßige Modell-Neuberechnung**  
  Planen Sie einen **wöchentlichen Cron-Job** (`crontab -e; 0 3 * * 1 /usr/local/bin/focusctl retrain-clusters`), der die 8 Cluster-Zentroiden auf Basis der letzten **100.000 E-Mail-Embeddings** neu berechnet. Dies stellt sicher, dass sich die Themen-Cluster an saisonale Änderungen anpassen (z. B. erhöhte Anzahl von Support-Anfragen nach Produktlaunches). Der Prozess erstellt eine **Sicherungskopie** der alten Zentroiden in der Tabelle `cluster_backup`, sodass ein Rollback innerhalb von **30 Sekunden** möglich ist.
