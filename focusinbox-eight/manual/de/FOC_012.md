# focusinbox-eight - FOC_012

## Übersicht

focusinbox-eight ist eine hochperformante Echtzeit-Postfach-Priorisierungs-Engine, speziell optimiert für die deutschsprachige Geschäftskommunikation. Das System analysiert eingehende Nachrichten mittels eines mehrstufigen NLP-Pipelines, der auf einem eigens trainierten BERT-Modell (fine-getuned auf 2,3 Mio. deutschsprachigen Support-Tickets) basiert. Die Latenz pro Nachricht beträgt < 15 ms bei einer Genauigkeit von 99,2 % bei der Klassifizierung von Dringlichkeit und Kategorie.

## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist eine serverseitige Middleware, die als intelligenter Filter zwischen Ihrem SMTP-Server und Ihrem CRM-System (z. B. HubSpot, Salesforce) agiert. Es extrahiert nicht nur Metadaten, sondern führt eine semantische Analyse des Nachrichteninhalts durch, um automatisch Arbeitspakete zu erstellen, Termine zu extrahieren und Eskalationsstufen festzulegen. Die gesamte Verarbeitung erfolgt in einer PostgreSQL-Datenbank mit Partitionierung nach `customer_id` und `timestamp`, um Abfragen im Millisekundenbereich zu garantieren.

### Kernfunktionen

- **Dynamische Postfach-Priorisierung mittels gewichteter Feature-Matrix**  
  Jede eingehende E-Mail wird anhand von 47 Parametern bewertet, darunter `urgency_score` (0–100, berechnet aus Sentiment, Absenderhistorie und Betreffstruktur), `response_time_risk` (basierend auf SLA-Kalender) und `topic_cluster_id` (generiert durch Latent Dirichlet Allocation). Das System priorisiert automatisch Nachrichten, die eine `response_time_risk` > 0,85 aufweisen, und verschiebt sie in eine separate Warteschlange mit garantierter Bearbeitung innerhalb von 5 Minuten.

- **Multi-Version-Datenvergleich mit Delta-Analyse**  
  focusinbox-eight speichert jede Nachricht als Version in einer eigenen Tabelle (`email_versions`). Über die REST-API kann ein `diff` zwischen zwei Versionen abgerufen werden (z. B. `GET /api/v2/diff?version_a=123&version_b=456`). Die Ausgabe enthält nicht nur Textänderungen, sondern auch eine semantische Abweichungsanalyse (z. B. „Neue Dringlichkeit: von 'niedrig' auf 'hoch' geändert“) und eine Aufschlüsselung der Änderungen in Millisekunden.

- **Automatische Termin- und Aufgaben-Extraktion mit Kalender-Integration**  
  Der integrierte Parser (basierend auf spaCy und einem benutzerdefinierten Regelwerk für deutsche Datumsformate) erkennt Terminvorschläge, Fristen und Meeting-Anfragen. Diese werden als JSON-Objekte mit `start_datetime` (ISO 8601), `duration_minutes` und `participants` extrahiert und direkt per Webhook an Google Calendar, Outlook oder Nextcloud gesendet. Fehlertoleranz: < 0,5 % bei Datumserkennung in ganzen Sätzen.

### So verwenden Sie es

1. **Konfiguration der Eingangs-Pipeline**  
   Definieren Sie in der Datei `config/pipeline.yaml` Ihre SMTP-Verbindungsparameter (`smtp_host`, `smtp_port`, `use_tls: true`) und die PostgreSQL-Verbindungszeichenfolge (`postgresql://user:pass@localhost:5432/focusinbox`). Starten Sie dann den Daemon: `focusinbox-eight daemon --config config/pipeline.yaml`. Das System beginnt sofort mit dem Polling aller konfigurierten Postfächer im Intervall von 10 Sekunden.

2. **Anpassung der Priorisierungsregeln**  
   Bearbeiten Sie die Datei `rules/priority_rules.json`, um gewichtete Bedingungen zu definieren. Beispiel:  
   ```json
   {
     "rules": [
       {
         "condition": "urgency_score > 80 AND response_time_risk > 0.9",
         "action": "move_to_queue('critical')",
         "weight": 10
       }
     ]
   }
   ```  
   Führen Sie anschließend `focusinbox-eight reload --rules` aus, um die Regeln ohne Neustart zu aktivieren.

3. **Aufruf der Delta-Analyse-API**  
   Senden Sie eine GET-Anfrage mit zwei Versions-IDs:  
   ```bash
   curl -X GET "https://localhost:8080/api/v2/diff?version_a=1001&version_b=1002" \
   -H "Authorization: Bearer YOUR_API_KEY" \
   -H "Accept: application/json"
   ```  
   Die Antwort enthält ein Array von Änderungsobjekten, z. B.:  
   ```json
   [
     {
       "field": "urgency_score",
       "old_value": 45,
       "new_value": 92,
       "change_type": "increase",
       "timestamp_ms": 1623456789123
     }
   ]
   ```

### Beste Praktiken

- **Implementieren Sie eine zweistufige Eskalationskaskade**  
  Nutzen Sie die `response_time_risk`-Metrik, um automatisch eine erste Eskalation an den Teamleiter (per Webhook an Slack/Teams) auszulösen, wenn der Wert > 0,9 steigt. Eine zweite Eskalation an den Abteilungsleiter erfolgt nach 30 Minuten ohne Reaktion. Dies reduziert die durchschnittliche Reaktionszeit um 47 % (gemessen in einer Pilotstudie mit 12 Unternehmen).

- **Kombinieren Sie die Terminextraktion mit einem automatischen Kalender-Blocking**  
  Richten Sie einen Webhook in `config/webhooks.yaml` ein, der bei jeder erfolgreichen Terminextraktion einen POST an Ihren Kalender-API-Endpunkt sendet. Verwenden Sie dabei den Parameter `auto_confirm: false`, um eine manuelle Bestätigung zu erzwingen. So vermeiden Sie Doppelbuchungen und stellen sicher, dass nur bestätigte Termine in den Kalender eingetragen werden.
