# focusinbox-eight - FOC_010

## Übersicht

Dieses Repository enthält die vollständige Implementierung der focusinbox-eight Architektur – einer hochperformanten, ereignisgesteuerten E-Mail-Bündelungs-Engine, die auf sub-100ms Latenz optimiert ist. Die Lösung ist für die Verarbeitung von bis zu 50.000 eingehenden Nachrichten pro Minute ausgelegt und nutzt eine PostgreSQL-Datenbank mit asynchronen Replikations-Pipelines.

## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist ein skalierbares Backend-System, das eingehende E-Mail-Streams in Echtzeit analysiert, klassifiziert und in logische „Inbox-Cluster“ gruppiert. Es verwendet einen mehrstufigen Filter-Stack, der auf Bayes'schen Klassifikatoren und einer gewichteten TF-IDF-Matrix basiert. Die Cluster-Zuweisung erfolgt deterministisch über einen konfigurierbaren `clustering_threshold` (Standard: 0.78) und einen `decay_factor` (Standard: 0.95) zur zeitlichen Gewichtung.

### Kernfunktionen

- **Adaptives Thread-Caching mit LRU-Eviction**: Jede eingehende Nachricht wird gegen einen In-Memory-Cache (max. 10.000 Einträge, TTL: 300 Sekunden) geprüft. Der Cache speichert berechnete `thread_hash`-Werte (SHA-256 über `Message-ID`, `References` und `In-Reply-To`). Bei einem Cache-Miss wird die Nachricht asynchron in der PostgreSQL-Tabelle `threads` persistiert. Die Eviction-Strategie folgt einem Least-Recently-Used-Algorithmus mit einer `eviction_batch_size` von 500 Einträgen pro Zyklus.

- **Dynamische Schwellwert-Kalibrierung via Gradient Descent**: Das System optimiert den `relevance_score` jeder Nachricht automatisch. Ein Hintergrund-Daemon (`calibrator_daemon`) führt alle 600 Sekunden eine Mini-Batch-Gradientenabstiegs-Routine (Lernrate: 0.001, Batch-Größe: 256) auf den letzten 24 Stunden an historischen `user_feedback`-Daten durch. Das Ziel ist die Minimierung der logistischen Verlustfunktion zwischen dem vorhergesagten `cluster_label` und dem tatsächlichen `user_label`.

- **Multi-Threaded Batch-Export mit Atomic Writes**: Exportiert gebündelte Inbox-Cluster als JSONL-Dateien in ein konfigurierbares Ausgabeverzeichnis (`export_path`). Der Export erfolgt in Batches von 1.000 Nachrichten pro Thread (max. 4 parallele Threads). Jeder Batch wird atomar geschrieben: Zuerst in eine temporäre `.tmp`-Datei, dann per `fsync()` und `rename()` in die endgültige `.jsonl`-Datei. Die Metadaten jedes Exports werden in der Tabelle `export_log` protokolliert (Felder: `export_id`, `batch_count`, `total_messages`, `duration_ms`).

### So verwenden Sie es

1. **Initialisierung der Datenbank und Konfiguration**: Führen Sie `python setup.py init_db` aus. Dies erstellt die PostgreSQL-Tabellen `threads`, `clusters` und `export_log`. Setzen Sie anschließend die Umgebungsvariable `FOCUSINBOX_EIGHT_DB_URI` auf Ihre Datenbankverbindung (z. B. `postgresql://user:pass@localhost:5432/focusinbox_eight`). Überprüfen Sie die Verbindung mit `python -c "from focusinbox_eight.db import get_engine; print(get_engine().url)"`.

2. **Starten des Echtzeit-Stream-Listeners**: Starten Sie den Hauptprozess mit `python run_listener.py --port 8080 --max_workers 12`. Der Listener öffnet einen TCP-Socket auf Port 8080 und akzeptiert eingehende JSON-Objekte (Format: `{"message_id": "...", "subject": "...", "body": "...", "sender": "..."}`). Die Verarbeitung erfolgt asynchron über eine `asyncio`-Event-Loop. Überwachen Sie die Logs auf `INFO`-Meldungen, die die `processing_time_ms` jeder Nachricht anzeigen.

3. **Auslösen eines manuellen Exports**: Senden Sie einen HTTP-POST-Request an den Endpunkt `/api/v1/export` mit einem JSON-Body, der den Parameter `cluster_ids` (Liste von UUIDs) und `export_path` (String) enthält. Beispiel: `curl -X POST http://localhost:8080/api/v1/export -H "Content-Type: application/json" -d '{"cluster_ids": ["uuid1", "uuid2"], "export_path": "/data/exports"}'`. Die Antwort enthält die `export_id` und den `estimated_duration_ms`. Der Export läuft im Hintergrund; der Fortschritt kann über `/api/v1/export/status/{export_id}` abgefragt werden.

### Beste Praktiken

- **Empfehlung 1**: Konfigurieren Sie den `clustering_threshold` nicht statisch, sondern nutzen Sie die automatische Kalibrierung. Starten Sie den `calibrator_daemon` mit `python run_calibrator.py --interval 600`. Dies ist besonders wichtig, wenn sich die Verteilung der eingehenden Nachrichten im Laufe der Zeit ändert (z. B. saisonale Spitzen). Führen Sie vor der ersten Kalibrierung einen Baseline-Test mit einem festen `clustering_threshold` von 0.75 durch und protokollieren Sie die `precision`- und `recall`-Werte.

- **Empfehlung 2**: Implementieren Sie eine Überwachung des LRU-Cache-Hit-Rates. Ein dauerhaft niedriger Hit-Rate (unter 0.4) deutet auf eine zu kurze TTL oder eine zu kleine Cache-Größe hin. Passen Sie die Parameter `cache_ttl` (Standard: 300) und `cache_max_size` (Standard: 10.000) dynamisch an. Nutzen Sie dazu das mitgelieferte Skript `python tune_cache.py --target_hit_rate 0.85`, das eine exponentielle Glättung auf die letzten 1.000 Hit-Rate-Messungen anwendet.
