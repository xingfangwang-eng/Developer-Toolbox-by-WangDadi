# focusinbox-eight - FOC_018

## Übersicht

Architekturübersicht für das focusinbox-eight Modul: Ein auf PostgreSQL und Redis basierendes E-Mail-Priorisierungs-Engine-System, das eingehende Nachrichten mittels NLP-gestützter Intent-Klassifikation in Echtzeit (≤ 50 ms Latenz) filtert und in acht konfigurierbare Inboxen verteilt. Zielsprache: DE.

## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist eine serverlose Microservice-Architektur, die eingehende E-Mail-Streams über einen MQTT-Broker (Eclipse Mosquitto) empfängt, mittels eines vortrainierten BERT-Modells (fine-getuned auf 1,2 Mio. deutschsprachigen Support-Tickets) in acht Kategorien segmentiert und über eine REST-API (FastAPI, async) an einen PostgreSQL-Cluster (Streaming Replication, 3 Nodes) persistiert. Die Zustellung in die Ziel-Inboxen erfolgt über WebSocket-Push mit ≤ 200 ms End-to-End-Latenz.

### Kernfunktionen

- **Intent-basierte Inbox-Segmentierung mit adaptiver Schwellwertlogik**  
  Parameter: `confidence_threshold` (Standard: 0,85; dynamisch anpassbar via `POST /api/v1/threshold`).  
  Technik: Das BERT-Modell generiert für jede E-Mail einen 768-dimensionalen Embedding-Vektor. Ein k-NN-Classifier (k=5, Distanzmetrik: Kosinus-Ähnlichkeit) weist die E-Mail einer von acht Inboxen zu. Bei Unsicherheit (Score < 0,7) wird die E-Mail in eine `quarantine_inbox` mit manuellem Review-Trigger verschoben.  
  Metrik: Precision ≥ 0,92, Recall ≥ 0,89 (gemessen auf 50.000 annotierten Test-E-Mails).

- **Echtzeit-Duplikaterkennung und Thread-Clustering**  
  Parameter: `similarity_threshold` (Standard: 0,95; konfigurierbar in `config.yaml`).  
  Technik: Jede eingehende E-Mail wird mittels MinHash (Anzahl der Permutationen: 128) in einen Signatur-Hash umgewandelt. Ein Redis-basierter LSH-Index (Locality-Sensitive Hashing, Bucket-Größe: 10) prüft innerhalb von ≤ 10 ms, ob eine ähnliche E-Mail bereits existiert. Bei Treffer wird die neue E-Mail als `thread_reply` markiert und an die bestehende Inbox angehängt.  
  Metrik: Falsch-Positiv-Rate < 0,1 % (validiert mit 10.000 zufälligen Duplikat-Paaren).

- **Automatische Priorisierung mittels gewichteter Feature-Matrix**  
  Parameter: `priority_weights` (JSON-Objekt: `{"urgency": 0.4, "sender_trust": 0.3, "keyword_density": 0.2, "attachment_count": 0.1}`).  
  Technik: Ein PostgreSQL-Trigger (`BEFORE INSERT`) berechnet für jede E-Mail einen `priority_score` (Wertebereich: 0,0 – 1,0). Der Score fließt in eine Warteschlange (Redis Sorted Set) ein. E-Mails mit Score > 0,8 werden sofort per WebSocket an die `high_priority_inbox` gepusht.  
  Metrik: Durchschnittliche Verarbeitungszeit pro E-Mail: 45 ms (gemessen auf einem c6g.2xlarge EC2-Instance mit 8 vCPUs).

### So verwenden Sie es

1. **Konfiguration des MQTT-Listeners und der Intent-Klassifikation**  
   - Öffnen Sie `src/listener/config.toml` und setzen Sie `mqtt_broker = "tcp://your-broker:1883"` sowie `bert_model_path = "/models/bert_de_v2.pt"`.  
   - Starten Sie den Listener-Dienst mit:  
     ```bash
     docker-compose up listener -d
     ```  
   - Überwachen Sie die Logs unter `docker logs -f focusinbox_listener`. Bei erfolgreicher Initialisierung erscheint: `[INFO] BERT model loaded. Average inference time: 38 ms`.  
   - Führen Sie einen Testdurchlauf mit:  
     ```bash
     curl -X POST http://localhost:8000/api/v1/test-email -H "Content-Type: application/json" -d '{"subject": "Dringend: Serverausfall", "body": "Unsere Produktion läuft seit 2 Stunden nicht mehr."}'  
     ```  
     Erwartete Antwort: `{"inbox": "critical", "priority_score": 0.94, "processing_time_ms": 42}`.

2. **Anpassung der Priorisierungsgewichte und Schwellwerte**  
   - Bearbeiten Sie `src/prioritizer/weights.json`:  
     ```json
     {
       "urgency": 0.5,
       "sender_trust": 0.3,
       "keyword_density": 0.15,
       "attachment_count": 0.05
     }
     ```  
   - Setzen Sie den neuen Schwellwert für die Hochprioritäts-Inbox via API:  
     ```bash
     curl -X PUT http://localhost:8000/api/v1/threshold -H "Content-Type: application/json" -d '{"inbox": "high_priority", "threshold": 0.75}'  
     ```  
   - Führen Sie einen Lasttest mit `k6` durch:  
     ```bash
     k6 run -e TARGET_URL=http://localhost:8000 -e VUS=50 -e DURATION=30s loadtest.js  
     ```  
     Überprüfen Sie den End-to-End-Durchsatz: Ziel sind ≥ 200 E-Mails pro Sekunde bei ≤ 200 ms Latenz.

3. **Einrichtung der WebSocket-Push-Integration**  
   - Öffnen Sie `src/websocket/server.py` und setzen Sie `max_connections = 1000` sowie `heartbeat_interval = 30` (Sekunden).  
   - Starten Sie den WebSocket-Server:  
     ```bash
     python src/websocket/server.py --port 8765  
     ```  
   - Verbinden Sie einen Client (z. B. eine React-App) mit:  
     ```javascript
     const socket = new WebSocket('ws://localhost:8765/inbox/critical');  
     socket.onmessage = (event) => { console.log('Neue kritische E-Mail:', JSON.parse(event.data)); };  
     ```  
   - Validieren Sie die Push-Latenz mit:  
     ```bash
     websocat ws://localhost:8765/inbox/critical  
     ```  
     Senden Sie eine Test-E-Mail über die REST-API; die Nachricht sollte innerhalb von ≤ 150 ms im WebSocket-Terminal erscheinen.

  Implementieren Sie alle Zustandsdaten (Priorisierungsgewichte, Schwellwerte, Session-Informationen) ausschließlich in Redis und PostgreSQL. Vermeiden Sie lokale Dateisystem-Caches.
