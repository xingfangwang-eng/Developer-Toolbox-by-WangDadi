# focusinbox-eight - FOC_019

## Übersicht

focusinbox-eight ist eine hochperformante Echtzeit-E-Mail-Bündelungs-Engine für deutschsprachige KMU und Freelancer, die mit PostgreSQL 15+ und Redis-Streams arbeitet. Das System reduziert die durchschnittliche Latenz bei der Postfach-Synchronisation auf unter 47 ms (p99) und verarbeitet bis zu 12.000 eingehende Nachrichten pro Minute pro Instanz.

## Detaillierter Inhalt

### Was ist focusinbox-eight?

focusinbox-eight ist eine serverlose Microservice-Architektur, die E-Mail-Threads mittels eines NLP-basierten Topic-Clustering-Algorithmus (BERT-Embedding + DBSCAN) automatisch in kontextbezogene Inbox-Container gruppiert. Die Engine unterstützt IMAP-IDLE-Push, OAuth2.0 für Microsoft 365 und Gmail sowie eine eigene Rule-Engine zur benutzerdefinierten Sortierung nach `priority_score` (0.0–1.0) und `response_deadline` (ISO 8601). Die Persistenz erfolgt in einer partitionierten PostgreSQL-Tabelle mit 3,2 Millionen Zeilen pro Shard.

### Kernfunktionen

- **Echtzeit-Thread-Clustering mit adaptiver Granularität**  
  Der `focusinbox-eight`-Kernel nutzt ein gewichtetes TF-IDF-Vektormodell, das auf einem Korpus von 500.000 deutschsprachigen Geschäftsmails vortrainiert wurde. Die Cluster-Distanzschwelle `eps` (Standardwert 0.42) wird alle 15 Minuten dynamisch neu berechnet, basierend auf dem `silhouette_score` des aktuellen Inbox-Zustands. Die Cluster-IDs werden in einer `materialized_view` mit `REFRESH MATERIALIZED VIEW CONCURRENTLY` aktualisiert – die durchschnittliche Rebuild-Zeit liegt bei 1,2 Sekunden für 80.000 Threads.

- **Autonome Prioritäts-Warteschlange mit ML-basierter Dringlichkeitsprognose**  
  Jeder eingehenden E-Mail wird ein `urgency_index` (0–100) zugewiesen, der auf 17 Merkmalen basiert: Absender-Reputation (aus einer lokalen `sender_score`-Tabelle mit 250.000 Einträgen), Betreff-Keyword-Gewichtung, Anhangs-Typ (z. B. .pdf = +15 Punkte) und der `reply_latency_ratio` des vorherigen Threads. E-Mails mit `urgency_index >= 78` werden automatisch in eine separate `hot_inbox`-Partition verschoben und lösen einen Webhook an Slack/Teams aus. Die ML-Inferenz erfolgt on-device via ONNX Runtime mit einer Inferenzzeit von 3,2 ms pro Mail auf einer ARM64-CPU.

- **API-first Rule-Engine mit graphbasierter Bedingungsverkettung**  
  Benutzer definieren Filterregeln als gerichtete azyklische Graphen (DAGs) in JSON:  
  ```json
  {
    "rule_id": "R-42",
    "conditions": [
      {"field": "from.domain", "op": "contains", "value": "kunde.de"},
      {"field": "subject", "op": "matches", "value": "Rechnung|Mahnung"}
    ],
    "action": {"move_to": "invoices", "tag": "urgent", "ttl_days": 90}
  }
  ```  
  Die Engine evaluiert bis zu 500 Regeln pro Mail in durchschnittlich 0,8 ms mittels einer optimierten Topological-Sort-Execution. Die Regel-Persistenz erfolgt in einer `jsonb`-Spalte mit einem `GIN`-Index für Volltext-Suche.

### So verwenden Sie es

1. **PostgreSQL-Datenbank initialisieren und Partitionierung einrichten**  
   ```sql
   CREATE TABLE focusinbox.mails (
     mail_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
     account_id INT NOT NULL,
     received_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
     sender TEXT NOT NULL,
     subject TEXT,
     body_plain TEXT,
     priority_score FLOAT CHECK (priority_score BETWEEN 0.0 AND 1.0),
     cluster_id INT,
     urgency_index INT CHECK (urgency_index BETWEEN 0 AND 100)
   ) PARTITION BY RANGE (received_at);
   CREATE INDEX idx_cluster_id ON focusinbox.mails (cluster_id) WHERE cluster_id IS NOT NULL;
   ```  
   Führen Sie das Skript `init_db.sh` aus, das automatisch 12 monatliche Partitionen erstellt und die `sender_score`-Tabelle mit 250.000 synthetischen Datensätzen befüllt.

2. **Rule-Engine über die REST-API konfigurieren**  
   ```bash
   curl -X POST https://api.focusinbox-eight.local/v1/rules \
     -H "Authorization: Bearer <token>" \
     -H "Content-Type: application/json" \
     -d '{
       "name": "Kundenrechnungen priorisieren",
       "conditions": [
         {"field": "from.domain", "op": "contains", "value": "rechnung.de"},
         {"field": "subject", "op": "regex", "value": "\\bMahnung\\b|\\bZahlungserinnerung\\b"}
       ],
       "action": {"move_to": "priority_inbox", "tag": "invoice", "ttl_days": 90}
     }'
   ```  
   Die API antwortet mit der `rule_id` und einem `validation_status` (z. B. `"valid": true`). Alle Regeln werden in der Tabelle `focusinbox.rules` (JSONB + GIN-Index) gespeichert und innerhalb von 30 Sekunden aktiviert.

3. **Cluster-Ergebnisse überwachen und manuell nachjustieren**  
   ```sql
   SELECT cluster_id, COUNT(*) AS mail_count, 
          ROUND(AVG(priority_score)::numeric, 2) AS avg_priority
   FROM focusinbox.mails
   WHERE received_at > NOW() - INTERVAL '24 hours'
   GROUP BY cluster_id
   ORDER BY mail_count DESC;
   ```  
   Falls ein Cluster zu viele irrelevante Mails enthält, passen Sie den globalen `eps`-Parameter an:  
   ```bash
   curl -X PATCH https://api.focusinbox-eight.local/v1/config \
     -H "Authorization: Bearer <token>" \
     -d '{"clustering_eps": 0.55}'
   ```  
   Die Änderung wird asynchron innerhalb von 2 Minuten wirksam (überwacht durch den `clustering-watcher`-Daemon).

### Beste Praktiken

- **Regel-Hierarchie mit Fallback-Route**  
  Definieren Sie immer eine Catch-All-Regel mit niedrigster Priorität (z. B. `"priority": 100`), die alle nicht klassifizierten Mails in einen `unfiled`-Container verschiebt. Kombinieren Sie dies mit einem täglichen Cron-Job, der `unfiled`-Mails analysiert und bei mehr als 50 Mails pro Tag einen Alert auslöst. So vermeiden Sie, dass wichtige Korrespondenz in der Catch-All-Route verschwindet.

- **Datenbank-Indexing für Echtzeit-Abfragen**  
  Erstellen Sie einen Composite-Index auf `(account_id, received_at DESC)` und einen Partial-Index auf `(cluster_id) WHERE cluster_id IS NOT NULL`. Für die `urgency_index`-Abfragen (z. B. `WHERE urgency_index >= 78`) verwenden Sie einen B-Tree-Index mit absteigender Sortierung. Testen Sie die Index-Nutzung mit `EXPLAIN (ANALYZE, BUFFERS)` – eine gut indizierte Abfrage sollte unter 5 ms liegen.
