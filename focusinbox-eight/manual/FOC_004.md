# focusinbox-eight - FOC_004

## Overview

**focusinbox-eight** is a high-performance, real-time email triage and task extraction engine designed for knowledge workers who demand sub-100ms response times from their inbox automation. It leverages a hybrid retrieval-augmented generation (RAG) pipeline with an embedded vector store (FAISS + ONNX runtime) to classify, prioritize, and extract actionable items from raw email threads without sending data to external APIs. Keywords: focusinbox-eight tool, focusinbox-eight solution, focusinbox-eight software.

## Detailed Content

### What is focusinbox-eight?

focusinbox-eight is a self-hosted, privacy-first inbox intelligence layer that ingests IMAP streams, applies multi-label classification via a fine-tuned DistilBERT model (trained on 500k+ enterprise emails), and outputs structured JSON task objects. It supports incremental indexing with sub-millisecond deduplication checks using a Bloom filter (false positive rate < 0.1%). The engine is built on Rust with a Python inference bridge, ensuring zero-copy memory handling for high-throughput environments (10k+ emails/day on a single 4-core instance).

### Core Features

- **Real-time Priority Scoring with Temporal Decay**: Each email is assigned a `priority_score` (float 0.0–1.0) using a custom algorithm that combines sender reputation (weighted by `reply_latency` history), thread depth, and a logarithmic decay factor (`score = base_score * e^(-λ * hours_since_received)`). The decay constant λ is configurable per project (default: 0.05). Scores are recalculated on every inbox poll (configurable interval in milliseconds, default: 30,000 ms).
- **Zero-shot Task Extraction with Schema Validation**: Extracted tasks are validated against a user-defined JSON Schema (e.g., `{ "due_date": "string (ISO 8601)", "assignee": "string", "action_verb": "enum[review, approve, implement, report]" }`). The extraction pipeline uses a combination of spaCy NER (custom-trained on project management corpora) and a lightweight transformer model (MiniLM-L6-v2) for verb-phrase chunking. Non-conforming extractions are flagged with a `confidence` field (0–100) and queued for manual review.
- **PostgreSQL-backed Thread Reconstruction with Conflict Resolution**: All email threads are stored in a PostgreSQL database with a custom `thread_tree` data structure (adjacency list with materialized path). When multiple replies arrive simultaneously, the engine uses a Lamport timestamp-based conflict resolution to merge `thread_id` assignments. The `reconstructed_thread` endpoint returns a sorted array of messages with `sequence_number` (integer) and `parent_hash` (SHA-256 of parent message body), enabling deterministic playback of conversation history.

### How to Use

1. **Initialize the IMAP Watcher with Custom Polling Strategy**  
   Create a configuration file `focusinbox.toml` with the following parameters:
   ```toml
   [imap]
   host = "imap.example.com"
   port = 993
   polling_interval_ms = 15000
   batch_size = 50
   dedup_bloom_filter_capacity = 1000000
   dedup_bloom_false_positive_rate = 0.001
   ```
   Run `focusinbox-eight watch --config focusinbox.toml`. The watcher will open a persistent TLS connection and poll for new messages every 15 seconds. Each batch of up to 50 messages is processed through the Bloom filter; duplicates are discarded before any NLP pipeline is invoked.

2. **Define a Project-Specific Task Schema and Trigger Extraction**  
   Create a JSON schema file `project_schema.json`:
   ```json
   {
     "$schema": "http://json-schema.org/draft-07/schema#",
     "type": "object",
     "properties": {
       "due_date": { "type": "string", "format": "date-time" },
       "assignee": { "type": "string" },
       "priority": { "type": "integer", "minimum": 1, "maximum": 5 }
     },
     "required": ["due_date", "assignee"]
   }
   ```
   Execute: `focusinbox-eight extract --schema project_schema.json --output-format jsonl`. The engine will parse all unread emails in the last 24 hours, run the zero-shot extractor, and write valid task objects to `extracted_tasks.jsonl`. Each line includes a `source_email_id` (UUID v4) and `validation_errors` array (if any).

3. **Run a Thread Reconstruction Query with Temporal Filters**  
   Use the CLI to reconstruct a specific thread by `thread_id`:
   ```bash
   focusinbox-eight thread --thread-id abc123 --sort-by sequence_number --order asc --include-attachments false
   ```
   The output is a JSON array:
   ```json
   [
     {
       "sequence_number": 1,
       "message_id": "<original@example.com>",
       "parent_hash": null,
       "subject": "Q4 Budget Review",
       "received_at": "2025-03-21T10:00:00Z",
       "priority_score": 0.87
     },
     {
       "sequence_number": 2,
       "parent_hash": "a1b2c3d4e5f6...",
       "subject": "Re: Q4 Budget Review",
       "received_at": "2025-03-21T10:05:30Z",
       "priority_score": 0.92
     }
   ]
   ```

### Best Practices

- **Use a Separate PostgreSQL Instance for Production Workloads**  
  For inboxes processing more than 5,000 emails per day, deploy a dedicated PostgreSQL 15+ instance with the `pg_stat_statements` extension enabled. Configure `shared_buffers` to at least 25% of available RAM and set `work_mem` to 64MB to avoid disk spilling during thread reconstruction. This ensures `reconstructed_thread` queries complete in under 200ms for threads with up to 500 messages.

- **Implement a Two-Tier Priority Decay Strategy for Team Inboxes**  
  Set a short decay half-life (e.g., λ = 0.1, half-life ≈ 7 hours) for emails from known external domains, and a longer half-life (λ = 0.02, half-life ≈ 35 hours) for internal senders. This prevents urgent external client emails from being buried by internal chatter. Configure this via the `priority.decay_map` parameter in the config file:
  ```toml
  [priority]
  decay_map = { "external" = 0.1, "internal" = 0.02 }
  default_decay = 0.05
  ```
