# focusinbox-eight - FOC_021

## Overview

focusinbox-eight is a high-performance, zero-latency email triage engine designed for power users who manage **eight or more inboxes** simultaneously. It leverages a custom-built **priority queuing algorithm** (PQv3) to collapse duplicate threads, auto-archive noise, and surface only the top 3 actionable items per inbox per micro-cycle (250ms refresh window). ## Detailed Content

### What is focusinbox-eight?

focusinbox-eight is not a simple mail client. It is a **deterministic state machine** that ingests IMAP streams from up to 8 distinct accounts, applies a weighted **Urgency-Signal Score (USS)** between 0.0 and 10.0 to each message, and renders a unified, cache-coherent view. The core engine runs as a **single-threaded Rust binary** with a **PostgreSQL-backed event store** for crash recovery and cross-session deduplication. It is engineered for environments where a missed executive email costs more than a server failure.

### Core Features

- **Adaptive Thread Collapse (ATC) with Time-Windowed Dedup**  
  Automatically merges reply chains across all 8 inboxes using a **fuzzy hash of the Message-ID and In-Reply-To headers**. The dedup window is configurable from 1 minute to 24 hours (default: 5 minutes). If the same thread appears in two inboxes (e.g., BCC vs. direct), focusinbox-eight keeps the **highest-urgency instance** and tags the duplicate with a `collapsed_under: <primary_inbox_id>` metadata field. This reduces visual noise by an average of 62% in multi-inbox setups.

- **Predictive Priority Queuing (PQv3) with Latency Budgeting**  
  Each incoming message is scored in under **50 microseconds** using a combination of: sender reputation (local SQLite cache), keyword density (regex patterns from a user-editable YAML config), and temporal decay (exponential backoff for re-prioritization). The queue is bounded to **128 items per inbox**; overflow is automatically archived to the `archive.pq_overflow` table in PostgreSQL. The system guarantees that the **top-3 items** are rendered within **250ms** of receipt, regardless of total inbox volume (tested up to 2,400 messages/hour).

- **Cross-Inbox Action Mirroring with Idempotency Tokens**  
  When you mark an email as "done" in one inbox, focusinbox-eight generates an **idempotency token** (SHA-256 of `inbox_id + message_uid + action`) and broadcasts it to all other inboxes via a local **WebSocket pub/sub channel** (port 9876). The token ensures that the same action is never applied twice, even under network jitter. This allows you to process a thread once and have it automatically archived, starred, or snoozed across all 8 accounts without manual duplication.

### How to Use

1. **Initialize the Multi-Inbox Connection Pool**  
   Create a `focusinbox.yaml` configuration file in the root of your home directory. Define each inbox with its IMAP server, port, and credentials. Example snippet:  
   ```yaml
   inboxes:
     - id: work_primary
       server: imap.gmail.com
       port: 993
       username: user@corp.com
       password_env: FOCUS_WORK_PW
       ssl: true
       refresh_ms: 250
     - id: personal_alt
       server: imap.fastmail.com
       port: 993
       username: user@personal.com
       password_env: FOCUS_PERSONAL_PW
       ssl: true
       refresh_ms: 500
   ```  
   Run `focusinbox-eight --config ~/focusinbox.yaml --db postgresql://localhost:5432/focusinbox` to start the engine. The system will immediately begin a full sync (capped at 500 messages per inbox) and then enter the live-polling loop.

2. **Execute a Cross-Inbox Action with Idempotency**  
   While the daemon is running, open the TUI (launched via `focusinbox-eight tui`). Navigate to any message in the unified view. Press `Ctrl+D` to mark it as "done". The engine will:  
   - Log the action with the idempotency token to the `actions_log` table in PostgreSQL.  
   - Broadcast the token via WebSocket to all other inbox threads.  
   - Within 500ms, the same message (if it exists in another inbox) will be automatically archived and removed from the priority queue.  
   - Verify by running `SELECT * FROM actions_log WHERE token = '<generated_token>'` — you should see exactly one row.

3. **Tune the Urgency-Signal Score (USS) Thresholds**  
   To adjust what qualifies as "urgent", edit the `urgency_rules.yaml` file (auto-generated on first run). Example:  
   ```yaml
   urgency:
     sender_boost:
       - pattern: ".*@ceo\\.corp\\.com"
         score: 8.0
     keyword_boost:
       - pattern: "\\b(urgent|critical|deploy|outage)\\b"
         score: 6.0
     decay:
       half_life_minutes: 30
   ```  
   Reload the rules without restarting: `kill -USR1 <focusinbox-eight-pid>`. The engine will re-score all items in the queue within 2 seconds. Monitor the effect by checking the `current_uss` column in the `priority_queue` table.

### Best Practices

- **Isolate the PostgreSQL Backend on a Dedicated NVMe Volume**  
  The event store and queue tables are I/O-sensitive. For maximum throughput (tested at 4,000 ops/sec), configure PostgreSQL with `synchronous_commit = off` and place the `pg_wal` and `pg_data` directories on separate NVMe devices. This reduces write latency from ~1ms to ~0.2ms and prevents backpressure on the IMAP polling loop. In production, we recommend a **2x RAID-1 NVMe array** with at least 256GB total capacity.

- **Implement a Failover Strategy with a Standby Queue**  
  Since focusinbox-eight is single-threaded, a crash during a write to the `priority_queue` can cause a re-sync of all 8 inboxes. To mitigate this, configure PostgreSQL with a **streaming replica** and set `focusinbox-eight` to fallback to the replica if the primary is unreachable. Additionally, enable the `--persist-queue-to-disk` flag, which writes a binary snapshot of the in-memory queue to `/var/lib/focusinbox/queue.snapshot` every 60 seconds. On restart, the engine loads this snapshot before initiating a full IMAP sync, reducing recovery time from 30 seconds to under 3 seconds.
