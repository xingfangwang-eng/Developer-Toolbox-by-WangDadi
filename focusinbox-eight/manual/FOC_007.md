# focusinbox-eight - FOC_007

## Overview

This is the architectural blueprint for the **focusinbox-eight** project, a high-performance email triage engine designed for zero-inbox practitioners. It ingests raw MIME streams, applies real-time NLP-based intent classification, and surfaces actionable items with sub-100ms latency. Keywords: focusinbox-eight tool, focusinbox-eight solution, focusinbox-eight software.

## Detailed Content

### What is focusinbox-eight?

focusinbox-eight is a **streaming-first** email processing pipeline that eliminates inbox clutter by assigning a `priority_score` (0.0–1.0) and a `context_tag` (e.g., `billing`, `code_review`, `calendar_invite`) to every incoming message. It runs as a standalone daemon or as a sidecar container, consuming IMAP IDLE notifications and writing structured results to a PostgreSQL `inbox_items` table. The system is built for engineers who need to maintain focus without missing critical SLA-bound communications.

### Core Features

- **Real‑time Priority Scoring with Token‑Aware Heuristics**  
  Each email is parsed into a `ParsedMessage` struct containing `from_domain`, `subject_tokens`, `body_ngrams`, and `attachment_mime_types`. A weighted scoring engine (configurable via `priority_weights.json`) assigns a `priority_score` based on sender reputation (looked up in a Redis `senders_trust` cache), keyword density, and thread depth. The entire pipeline—from IMAP fetch to PostgreSQL insert—completes in **≤ 85 ms** (p95) on a standard t3.medium instance.

- **Multi‑Account Aggregation with Deduplication**  
  Supports simultaneous connections to up to 8 IMAP accounts via a `connection_pool` of size 8. Each account’s `mailbox_watchlist` (configurable in `accounts.yaml`) is polled every **30 seconds**. A Bloom filter (size: 1,024 bits, false‑positive rate: 0.01) prevents duplicate processing of messages that have already been pushed to the `inbox_items` table. The `message_id_hash` (SHA‑256 of the `Message‑ID` header) is used as the dedup key.

- **Actionable Item Extraction via Regex + LLM Fallback**  
  The `extractor` module scans body text for deadlines (e.g., `due by 2025-04-15`), action verbs (`review`, `approve`, `merge`), and Jira/GitHub issue references (e.g., `PROJ‑123`). If the regex confidence is below **0.6**, the system invokes a local LLM (e.g., Llama 3.2 1B) with a `max_tokens=128` prompt to extract a `next_action` string. The LLM call is rate‑limited to **10 requests per minute** per account to avoid API throttling.

### How to Use

1. **Configure Account Credentials and Watchlists**  
   Edit `config/accounts.yaml` and provide IMAP server, port (993), username, and app‑specific password. Set `poll_interval_seconds: 30` and `max_emails_per_poll: 50`. Define a `priority_rules` block under each account to override global weights—for example, `from: "@critical-corp.com"` maps to `score_multiplier: 2.5`.

   ```bash
   # Validate the YAML schema before starting the daemon
   focusinbox-eight validate --config config/accounts.yaml
   ```

2. **Start the Daemon and Monitor the Stream**  
   Run `focusinbox-eight daemon --config config/accounts.yaml --db postgresql://user:pass@localhost:5432/focusinbox`. The daemon will log each processed message with its `priority_score` and `context_tag`. Open a second terminal and tail the live stream:

   ```bash
   focusinbox-eight stream --db postgresql://user:pass@localhost:5432/focusinbox --min-score 0.7
   ```

   This command prints a JSON line for every message with `priority_score >= 0.7` in real time.

3. **Query and Export Prioritized Items**  
   Use the `query` subcommand to filter by `from_domain`, `context_tag`, or `date_range`. For example, to export all high‑priority billing emails from the last 7 days:

   ```bash
   focusinbox-eight query --context-tag billing --min-score 0.8 --since "7 days ago" --format csv > urgent_billing.csv
   ```

   The CSV includes columns: `message_id`, `from`, `subject`, `priority_score`, `next_action`, `received_at`.

### Best Practices

- **Tune Priority Weights with A/B Testing**  
  Run two instances of the daemon against the same mailbox—one with your current `priority_weights.json` and one with experimental weights. Compare the `false_positive_rate` (low‑priority emails scored above 0.7) over a 48‑hour window. Use the `metrics` endpoint (`/metrics` in Prometheus format) to export `focusinbox_priority_distribution` and `focusinbox_action_extraction_latency` histograms.

- **Integrate with a Notification Router**  
  Instead of pushing all high‑priority emails to your inbox, pipe the `stream` output into a webhook (e.g., via `focusinbox-eight stream | jq -r '. | select(.priority_score > 0.9) | .next_action' | notify-slack`). This decouples the triage logic from your primary email client and reduces context‑switching overhead. For critical alerts, set a secondary threshold (`score > 0.95`) to trigger an SMS via Twilio.
