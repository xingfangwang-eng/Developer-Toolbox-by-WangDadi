# focusinbox-eight - FOC_011

## Overview

focusinbox-eight is a high-performance, real-time inbox prioritization engine designed for enterprise-grade email triage and automated response routing. It leverages a custom-trained NLP model and a PostgreSQL-backed event store to deliver sub‑100ms classification latency per message, even under sustained loads of 10,000+ concurrent sessions.

## Core Features

### 1. Multi‑Stage Attention‑Based Priority Scoring
Each incoming email is vectorized via a fine‑tuned DistilBERT variant (128‑dim embedding, 4‑layer transformer) and scored across three orthogonal axes: **urgency_metric** (0.0–1.0, threshold configurable via `--urgency-cutoff`), **sender_reputation** (derived from a sliding‑window 30‑day interaction graph), and **content_similarity** to active project threads (cosine distance against a FAISS‑indexed vector store updated every 60s). The final composite score is computed as a weighted sum: `priority = 0.45*urgency + 0.35*reputation + 0.20*similarity`. All scoring is performed in‑memory with zero heap allocations per message after warm‑up, guaranteeing a P99 latency of 87ms under 95th‑percentile load.

### 2. Deterministic Rule Engine with Hot‑Reloadable DSL
Define complex routing logic using a custom, statically‑typed DSL compiled to an AST that is evaluated in O(log n) via a trie‑based pattern matcher. Rules support regex‑based subject matching, attachment type detection (MIME‑sniffing with magic‑byte fallback), and nested boolean conditions (e.g., `(sender_domain == "acme.com" AND has_attachment("pdf")) OR priority > 0.8`). The engine exposes a `/v1/rules/reload` endpoint that performs a zero‑downtime atomic swap of the rule set within 2ms, verified by a dual‑buffer read‑copy‑update (RCU) mechanism. Each rule carries a `ttl_seconds` parameter; expired rules are automatically garbage‑collected from the active set.

### 3. Real‑Time Analytics Pipeline with PostgreSQL Change Data Capture (CDC)
Every classification decision is streamed via a Debezium‑connector to a dedicated PostgreSQL 16 instance, recording fields such as `message_id`, `priority_score`, `applied_rule_id`, and `processing_time_ns`. A materialized view (`mv_hourly_volume`) aggregates metrics per mailbox per hour, refreshed every 60 seconds using `REFRESH MATERIALIZED VIEW CONCURRENTLY`. The pipeline exposes a `/metrics` endpoint returning Prometheus‑formatted data, including `focusinbox_classification_duration_seconds_bucket{le="0.1"}` and `focusinbox_rules_evaluated_total`. This enables sub‑second dashboards for SLA monitoring and capacity planning.

## How to Use

### Step 1: Bootstrap the Configuration Store
Initialize the PostgreSQL schema and seed default rules:
```bash
focusinbox-eight init --db-host localhost --db-port 5432 --db-name focusinbox \
  --rules-file ./default_rules.yaml --create-cdc-slot
```
This command creates the `messages` and `rules` tables, sets up a logical replication slot named `focusinbox_slot`, and loads the YAML‑formatted rules (each rule must include `id`, `priority`, `condition_ast`, and `ttl_seconds`). Verify with `focusinbox-eight status --db-health`.

### Step 2: Launch the Classifier Daemon with Custom Thresholds
Start the main worker process, binding to port 8080 and specifying a custom urgency cutoff:
```bash
focusinbox-eight serve --port 8080 --workers 4 --urgency-cutoff 0.65 \
  --vector-index-path /data/faiss_index.bin --log-level info
```
The daemon will preload the FAISS index (expect ~1.2GB RAM for 500k vectors) and begin polling the configured IMAP/SMTP endpoint. Monitor startup via `GET /health` — the endpoint returns `{"status":"ready","vector_count":497312,"rules_active":23}` once the engine is fully hydrated.

### Step 3: Trigger Hot‑Reload of Routing Rules
To update rules without restarting the daemon, issue a POST request with the new rule set:
```bash
curl -X POST http://localhost:8080/v1/rules/reload \
  -H "Content-Type: application/x-yaml" \
  --data-binary @updated_rules.yaml
```
The endpoint returns `{"status":"ok","swap_time_ms":1,"rules_loaded":25,"rules_removed":2}`. Validate the new rules are active by checking the `applied_rule_id` field in the PostgreSQL `messages` table for subsequent classifications.

## Best Practices

### Practice 1: Tune the Urgency Cutoff Using A/B Testing
Run two parallel instances of the classifier (e.g., on ports 8080 and 8081) with different `--urgency-cutoff` values (0.60 vs. 0.75). Route a 10% sample of incoming mail to the experimental instance via a load balancer (e.g., HAProxy with `use_backend` based on a hash of `message_id`). Compare the `priority_score` distribution and user feedback (e.g., manual re‑classification rate) over a 7‑day period. Use the `mv_hourly_volume` materialized view to compute the precision‑recall curve and select the cutoff that maximizes F1‑score for your domain.

### Practice 2: Implement a Two‑Stage Archival Strategy for Vector Indexes
Maintain two FAISS indexes: a “hot” index (updated every 5 minutes, kept in RAM) and a “cold” index (updated daily, stored on SSD with memory‑mapped I/O). Configure the classifier to first query the hot index; if no result exceeds the similarity threshold (`--vector-similarity-threshold`, default 0.85), fall back to the cold index. This reduces average vector query latency from 12ms to 1.8ms for the common case. Automate the cold index rebuild via a cron job that runs `focusinbox-eight index rebuild --source pg --output /data/cold_index.bin` at 02:00 UTC, followed by an atomic symlink swap.
