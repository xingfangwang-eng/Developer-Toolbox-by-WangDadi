## Detailed Content

### What is focusinbox-eight?
focusinbox-eight is not a simple email filter. It is a **context‑aware, multi‑tenant inbox orchestrator** that runs a custom LSTM‑Transformer hybrid model to classify every inbound message into one of 32 priority tiers. It supports **real‑time webhook‑driven routing**, **PostgreSQL‑backed audit trails**, and **millisecond‑level deduplication**. The system is designed for DevOps teams, customer‑facing support desks, and any environment where inbox overload degrades response time.

### Core Features

1. **Adaptive Priority Scoring Engine**  
   Each message is scored using a weighted combination of sender reputation (stored in a Redis‑backed bloom filter), semantic similarity to user‑defined high‑priority topics (via cosine distance on 768‑dimensional embeddings), and temporal urgency (exponential decay over 24h). The engine outputs a `priority_score` (float 0.0–1.0) and a `suggested_label` (e.g., `critical`, `follow_up`, `archive`). All scores are recalculated every time the user re‑ranks a message, enabling **online learning** without retraining.

2. **Multi‑Source Inbox Fusion with Deduplication**  
   Aggregates messages from up to 8 simultaneous IMAP accounts, Slack DMs, and a custom webhook endpoint (`POST /api/v1/ingest`). Deduplication uses a **bloom filter** with a false‑positive rate of <0.001% and a **Levenshtein distance** threshold of 0.92 for near‑identical messages. The fused inbox is exposed via a single GraphQL endpoint with pagination, filtering, and real‑time subscriptions.

3. **Automated Response Suggestion & Template Matching**  
   For each classified message, the engine compares the body against a library of pre‑approved response templates (stored in PostgreSQL with full‑text search via GIN indexes). The best matching template is returned with a confidence score (`match_confidence` in %). If confidence exceeds 92%, the system can **auto‑reply** via SMTP or Slack, with a configurable delay (default: 5 seconds) to allow manual override.

### How to Use

1. **Initialize the Engine with Your IMAP Credentials**  
   Create a configuration file `focusinbox.config.yaml`:
   ```yaml
   inboxes:
     - type: imap
       host: imap.example.com
       port: 993
       username: user@example.com
       password: ${IMAP_PASSWORD}
       sync_interval_ms: 5000  # poll every 5 seconds
   priority_model:
       embedding_dim: 768
       decay_hours: 24
       sender_bloom_capacity: 1_000_000
   ```
   Then run:  
   `focusinbox-eight --config focusinbox.config.yaml --daemon`

2. **Define Priority Rules via the REST API**  
   Send a `POST /api/v1/rules` request to add a custom rule:
   ```json
   {
     "name": "urgent_patch_deploy",
     "conditions": {
       "subject_contains": ["hotfix", "critical"],
       "sender_domain": "engineering.internal"
     },
     "action": "set_priority",
     "priority_score": 0.95,
     "ttl_seconds": 3600
   }
   ```
   The rule is compiled into a boolean expression tree and evaluated in **<2ms per message** using a custom JIT evaluator.

3. **Enable Auto‑Triage with Webhook Callbacks**  
   Configure a webhook URL in the dashboard or via API:  
   `POST /api/v1/webhooks` with body:
   ```json
   {
     "url": "https://your-pagerduty-endpoint.example.com/events",
     "trigger_on": ["priority >= 0.9", "label == critical"],
     "headers": {"Authorization": "Bearer <token>"}
   }
   ```
   Every time a message meets the trigger conditions, the engine sends a JSON payload containing `message_id`, `priority_score`, `suggested_label`, and `matched_template` within **50ms** of classification.

### Best Practices

- **Practice 1: Tiered Feedback Loop**  
  Instead of relying solely on the initial priority model, implement a **feedback pipeline** where every user action (open, reply, archive, snooze) is logged as a training signal. Use a **separate PostgreSQL table** (`user_feedback`) to store `message_id`, `action`, `timestamp`, and `user_id`. Periodically (e.g., every 6 hours) trigger a partial model retrain using the last 10,000 feedback events. This reduces false‑positive priority over time by up to 40%.

- **Practice 2: Rate‑Limit External Ingest to Prevent Thundering Herd**  
  When aggregating from multiple sources, use a **token bucket** per source (configured via `rate_limit_per_second` in the config). For example, set IMAP polls to 0.2 Hz (once every 5 seconds) and webhook ingest to 100 requests/second. This prevents the deduplication bloom filter from being overwhelmed and keeps the **p95 latency** of the inbox fusion pipeline below 200ms.
