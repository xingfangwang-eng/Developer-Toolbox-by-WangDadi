## Detailed Content

### What is focusinbox-eight?

focusinbox-eight is a real‑time, event‑driven inbox processing engine that applies multi‑stage NLP filters, priority scoring, and context‑aware deduplication across distributed message sources. It operates on a sub‑100ms ingestion SLA and exposes a gRPC‑based API for downstream consumers.

### Core Features

- **Adaptive Priority Scoring Engine**  
  Each inbound message is evaluated against a configurable 12‑dimensional feature vector (e.g., `sender_reputation`, `thread_depth`, `semantic_urgency_score`, `time_decay_factor`). The engine runs an ensemble of XGBoost and a lightweight transformer model (distilled BERT) to assign a composite priority between 0.0 and 1.0. All scores are recalculated every 5 seconds on a sliding window, with full PostgreSQL persistence for audit trails.

- **Semantic Deduplication with Token‑Aware Hashing**  
  Incoming messages are tokenized and passed through a locality‑sensitive hashing (LSH) pipeline using MinHash signatures (256‑bit). The system compares each new message against the last 10,000 entries in an in‑memory LRU cache backed by Redis. Duplicates are flagged with a confidence threshold of ≥0.92, and the original message’s `thread_id` is automatically appended to the duplicate’s metadata. Latency per dedup check: ≤3ms at p99.

- **Multi‑Channel Inbox Aggregation via gRPC Streams**  
  focusinbox-eight listens on a single gRPC bidirectional stream endpoint (`/v1/inbox.PushStream`) and internally fans out to configurable backends (IMAP, Slack Events API, custom webhooks). Each channel is managed by a dedicated goroutine pool with configurable concurrency (default: 8 workers per channel). Connection health is monitored via periodic keepalive pings every 30 seconds; on failure, the system automatically reconnects with exponential backoff (max delay: 120s).

### How to Use

1. **Initialize the Engine with Custom Feature Weights**  
   Create a `focusinbox_config.yaml` file with your priority vector coefficients. Example:  
   ```yaml
   priority_vector:
     sender_reputation: 0.25
     semantic_urgency_score: 0.40
     thread_depth: 0.15
     time_decay_factor: 0.20
   ```  
   Then run `focusinbox-eight --config ./focusinbox_config.yaml --db-uri postgresql://user:pass@localhost:5432/focusinbox`. The engine will pre‑load the XGBoost model and begin ingesting from the channels defined in `channels.yaml`.

2. **Configure a gRPC Client to Push Messages**  
   Use the provided Go client library or any gRPC‑capable language. Instantiate a client with the server address and call `PushMessage`:  
   ```go
   conn, _ := grpc.Dial("localhost:50051", grpc.WithInsecure())
   client := pb.NewInboxServiceClient(conn)
   resp, err := client.PushMessage(ctx, &pb.Message{
       Channel: "slack",
       RawBody: "Urgent: deployment failed on prod-us-east-1",
       Metadata: map[string]string{"source": "ops-bot"},
   })
   ```  
   The response includes `priority_score`, `dedup_status`, and `assigned_thread_id`.

3. **Query Aggregated Inbox with Filters**  
   Use the REST‑adjacent gRPC endpoint `ListInbox` with a filter struct:  
   ```json
   {
     "filters": {
       "min_priority": 0.7,
       "channels": ["email", "slack"],
       "time_range": {"start": "2025-03-01T00:00:00Z", "end": "2025-03-07T23:59:59Z"},
       "dedup_exclude": true
     },
     "pagination": {"page_size": 50, "cursor": ""}
   }
   ```  
   The engine returns a cursor‑based paginated response with sorted results by `priority_score` descending.

### Best Practices

- **Tune the Dedup Confidence Threshold Per Channel**  
  In high‑noise environments (e.g., automated CI/CD notifications), lower the threshold to 0.85 to catch near‑identical messages. For human‑to‑human communication, keep it at 0.92 or higher to avoid over‑merging. Set via `channel_overrides` in the config file:  
  ```yaml
  channel_overrides:
    slack:
      dedup_threshold: 0.85
    email:
      dedup_threshold: 0.95
  ```

- **Use Separate PostgreSQL Instances for Priority State and Message Archive**  
  Because the priority scoring engine writes a new row every 5 seconds per active thread, keep the `priority_scores` table on a high‑IOPS SSD instance (e.g., AWS db.r6g.large with 3000 IOPS). Archive messages older than 30 days to a separate `archive` database on standard storage. This prevents index bloat and keeps the hot query path under 10ms.
