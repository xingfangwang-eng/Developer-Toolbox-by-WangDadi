# focusinbox-eight - FOC_012

## Overview

focusinbox-eight is a high-performance email triage engine architected for zero-latency inbox processing. It leverages a dual-layer priority pipeline (hot/warm storage) to achieve sub-10ms classification latency per message, operating entirely within a PostgreSQL-backed event store with CDC (Change Data Capture) triggers.

## Core Features

### 1. Dynamic Priority Scoring Engine with Temporal Decay
- **Parameters**: `priority_score = (urgency_coeff * 0.6) + (sender_reputation * 0.3) + (thread_depth * 0.1) * exp(-age_hours / decay_constant)`
- **Technical detail**: Each inbound email is parsed through a custom NLP tokenizer (trained on 50k+ enterprise threads) to extract `urgency_coeff` (0.0–1.0) from lexical cues like "ASAP" or "blocker". The `decay_constant` defaults to 24 hours but is configurable per folder via `config.decay_constant = 48`. PostgreSQL `RETURNING` clause writes the score into a `priority_queue` table with a GIN index on `(score, folder_id)` for real-time sorting.

### 2. Multi-version Thread Diffing & Merge Conflict Resolution
- **Parameters**: `diff_threshold = 0.15` (Jaccard similarity), `merge_strategy = "latest_timestamp_with_authority"`
- **Technical detail**: When two replies to the same thread arrive within 500ms, focusinbox-eight compares their bodies using a rolling hash (Rabin-Karp variant) and computes a similarity score. If `score < diff_threshold`, it forks the thread into two branches, storing each in a `thread_branches` table linked by `thread_id`. The system then applies a three-way merge (base email + branch A + branch B) using a custom diff algorithm that respects `reply_headers` and `in-reply-to` fields. The merged result is written to `thread_final` with a `merge_conflict` flag set to `FALSE` unless overlapping edits are detected.

### 3. Predictive Snooze with Calendar Integration
- **Parameters**: `snooze_duration = (event_end_timestamp - now()) * 0.8`, `confidence_threshold = 0.7`
- **Technical detail**: For emails containing calendar invites or meeting references, the engine extracts `event_start` and `event_end` via regex on `text/calendar` MIME parts. It then computes an optimal snooze time as 80% of the event duration (to avoid post-meeting overflow). The snooze is stored as a `pg_cron` job that executes `UPDATE priority_queue SET status = 'active' WHERE email_id = $1` at the computed timestamp. If the user’s calendar has conflicting events (checked via `calendar.conflicts` API), the confidence drops below 0.7, and the system falls back to a manual snooze prompt.

## How to Use

### Step 1: Configure the PostgreSQL Event Store
Initialize the database schema with the required triggers and indexes. Run the following command to set up the `priority_queue` table and the `email_ingest` trigger that automatically computes priority scores on insert:
```bash
focusinbox-eight init --db-host localhost --db-port 5432 --db-name focusinbox --schema-file ./schemas/priority_queue.sql
```
This creates a `BEFORE INSERT` trigger that calls the `compute_priority_score()` function, which reads `sender_reputation` from a cached `sender_stats` table (refreshed every 15 minutes via `pg_stat_statements`). Verify the schema with `\d priority_queue` in psql.

### Step 2: Ingest Emails via the SMTP Proxy
Point your MTA (e.g., Postfix or Sendmail) to the focusinbox-eight SMTP proxy on port 2525. The proxy parses each email using the `mail-parser` library and writes it to the `priority_queue` table with a `folder_id` derived from the `To` header’s domain. For example, emails to `team@company.com` are routed to `folder_id = 12`. The proxy logs each ingestion with millisecond precision:
```bash
focusinbox-eight proxy --smtp-port 2525 --db-uri postgresql://user:pass@localhost:5432/focusinbox
```
Monitor throughput with `watch -n 1 'psql -c "SELECT count(*) FROM priority_queue WHERE inserted_at > now() - interval '1 minute';"'`.

### Step 3: Trigger Thread Diffing on Duplicate Detection
When two emails share the same `Message-ID` (via `References` header), the system automatically invokes the thread diffing engine. Run this command to manually force a merge check on a specific thread:
```bash
focusinbox-eight diff --thread-id 42 --output-format json --merge-strategy latest_timestamp_with_authority
```
This returns a JSON object with `diff_score`, `branch_ids`, and a `merge_conflict` boolean. If `merge_conflict = true`, the system writes the conflicting branches to `thread_branches` and logs a warning with the conflicting line numbers. To resolve, use `focusinbox-eight resolve --thread-id 42 --branch-id 5 --accept branch_a`.

## Best Practices

### Practice 1: Optimize PostgreSQL Indexing for Real-Time Queries
For sub-50ms priority queue reads, create a composite index on `(folder_id, score DESC, inserted_at)`. Use `EXPLAIN ANALYZE` to verify that queries with `WHERE folder_id = $1 ORDER BY score DESC LIMIT 20` use an index-only scan. Additionally, set `autovacuum_vacuum_scale_factor = 0.01` on the `priority_queue` table to prevent bloat from high-frequency inserts (thousands per second). Monitor dead tuple ratio with `pg_stat_user_tables` and schedule a manual `VACUUM` if it exceeds 20%.

### Practice 2: Tune the Decay Constant Based on Workflow Patterns
If your team handles time-sensitive emails (e.g., support tickets with SLAs), set `decay_constant = 6` hours to rapidly deprioritize older messages. For project management threads where responses may take days, use `decay_constant = 72`. Test the impact by running `focusinbox-eight simulate --folder-id 5 --decay-constant 6 --duration 24h` to see how priority scores shift over time. Adjust the `urgency_coeff` weight from 0.6 to 0.8 if your NLP model over-weights neutral language.
