# focusinbox-eight - FOC_019

## Resumen

`focusinbox-eight` es una herramienta de gestión de bandeja de entrada inteligente, diseñada para priorizar correos electrónicos mediante clasificación semántica en tiempo real. Integra un motor de reglas dinámicas con capacidad de aprendizaje incremental, permitiendo filtrar, etiquetar y reenviar mensajes según patrones de comportamiento del usuario. El sistema opera con una latencia inferior a 50 ms por mensaje procesado, utilizando embeddings generados por un transformer ligero (distilBERT) y almacenamiento en PostgreSQL con índices GIN para búsqueda por similitud vectorial.

## Contenido Detallado

### ¿Qué es focusinbox-eight?

`focusinbox-eight` es una plataforma de automatización de bandeja de entrada que ejecuta pipelines de procesamiento de correo electrónico en tiempo real. Cada mensaje entrante es descompuesto en tokens, embeddings y metadatos (remitente, dominio, asunto, cuerpo), y luego evaluado contra una base de reglas probabilísticas. El sistema soporta hasta 10.000 reglas concurrentes con un tiempo de evaluación promedio de 12 ms por regla. La persistencia se realiza en PostgreSQL 15+ con particionamiento por fecha y tablas de vectores (`pgvector`). La arquitectura es completamente serverless, desplegada sobre AWS Lambda con invocaciones asíncronas mediante SQS.

### Características Principales

- **Clasificación semántica con embeddings contextuales**: Cada correo es convertido en un vector de 384 dimensiones usando `all-MiniLM-L6-v2`. La similitud coseno entre el embedding del mensaje y los embeddings de las reglas define la prioridad. El umbral de coincidencia se ajusta dinámicamente mediante un algoritmo de retropropagación simple basado en la tasa de falsos positivos (target < 2%). Los vectores se indexan con `IVFFlat` en PostgreSQL, permitiendo consultas ANN con recall > 95% en menos de 20 ms.

- **Pipeline de reglas con ejecución diferida y encadenamiento**: Las reglas se definen en YAML y soportan operadores lógicos (`AND`, `OR`, `NOT`), expresiones regulares (PCRE2), y acciones como `tag`, `archive`, `forward`, `reply_template`. Cada regla puede invocar otras reglas mediante un sistema de dependencias acíclicas. El pipeline se ejecuta en un worker concurrente (máx. 256 hilos) con colas de prioridad. La latencia total del pipeline no supera los 200 ms para cadenas de hasta 10 reglas.

- **Dashboard de métricas en tiempo real con exportación a Prometheus**: Cada invocación del pipeline expone métricas como `inbox_processing_duration_ms`, `rules_matched_total`, `embedding_latency_ms`. Estas métricas se recolectan mediante un sidecar que las envía a un endpoint de Prometheus cada 15 segundos. El dashboard (Grafana) muestra histogramas de latencia por regla, tasa de acierto por etiqueta, y volumen de correos procesados por hora. Los datos históricos se almacenan en PostgreSQL con retención configurable (por defecto 90 días).

### Cómo Usar

1. **Configurar el archivo de reglas en YAML**: Crear un archivo `rules.yaml` en el directorio `config/` del proyecto. Cada regla debe incluir `name`, `priority` (entero de 0 a 100), `condition` (objeto con `field`, `operator`, `value`), y `action` (objeto con `type` y `params`). Ejemplo:
   ```yaml
   - name: "urgent_client"
     priority: 90
     condition:
       field: "subject"
       operator: "regex"
       value: "^(URGENT|ASAP|CRITICAL)"
     action:
       type: "tag"
       params:
         tag: "alta_prioridad"
         notify: true
   ```
   Validar la sintaxis con `focusinbox-eight validate --rules config/rules.yaml`.

2. **Conectar la bandeja de entrada mediante IMAP/OAuth2**: Ejecutar el comando de conexión:
   ```bash
   focusinbox-eight connect --provider gmail --credentials client_secret.json
   ```
   El sistema solicitará autorización OAuth2 y almacenará el token de acceso cifrado en la base de datos (columna `credentials.token` en tabla `providers`). La conexión IMAP se mantiene viva con un heartbeat cada 30 segundos. Para proveedores Exchange, usar `--provider exchange --endpoint https://outlook.office365.com/EWS/Exchange.asmx`.

3. **Iniciar el pipeline de procesamiento continuo**:
   ```bash
   focusinbox-eight run --rules config/rules.yaml --max-concurrency 50 --log-level debug
   ```
   El proceso se ejecuta en primer plano; para modo demonio, añadir `--daemon`. El sistema monitorea la bandeja cada 10 segundos (configurable con `--poll-interval`). Cada lote de correos (máx. 100) se procesa en paralelo. Los resultados se escriben en la tabla `processed_emails` con campos `email_id`, `matched_rules`, `actions_taken`, `processing_time_ms`. Para detener, enviar SIGINT (Ctrl+C).

### Mejores Prácticas

- **Utilizar reglas con condiciones específicas y evitar patrones demasiado amplios**: Cada regla debe tener al menos dos condiciones (por ejemplo, dominio del remitente + palabra clave en el asunto) para reducir falsos positivos. Se recomienda probar las reglas en un entorno de staging con un subconjunto de correos (usando `--dry-run`) antes de aplicar al entorno productivo. Mantener un ratio de reglas por usuario inferior a 50 para garantizar latencias sub-100 ms.

- **Implementar un sistema de retroalimentación para mejorar la clasificación**: Configurar un endpoint webhook que reciba confirmaciones de usuario (por ejemplo, "marcar como correcto" o "mover a otra carpeta"). Estos eventos se almacenan en la tabla `feedback` y se utilizan para recalibrar los umbrales de similitud coseno cada 24 horas mediante un job programado (cron). La calibración ajusta los pesos de los embeddings utilizando un optimizador Adam con tasa de aprendizaje 0.001, reduciendo la tasa de falsos negativos hasta un 15% en 30 días.
