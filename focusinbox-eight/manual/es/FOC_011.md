# focusinbox-eight - FOC_011

## Resumen Técnico

focusinbox-eight es un sistema de orquestación de bandejas de entrada inteligentes con análisis semántico en tiempo real, diseñado para la priorización automatizada de notificaciones y la sincronización bidireccional con motores de reglas personalizadas. Utiliza un modelo de atención por lotes (batch attention) con ventanas de contexto de 2048 tokens para clasificar correos electrónicos, tickets y mensajes Slack en microsegundos.

## Funcionalidades Principales

### 1. Filtrado Predictivo con Umbral de Confianza Configurable
Implementa un clasificador basado en `XGBoost` entrenado con más de 500.000 muestras etiquetadas (precisión del 94,7%). El sistema expone el parámetro `confidence_threshold` (rango: 0.65–0.99) para ajustar la agresividad del filtrado. Cada decisión se registra con un `decision_id` único y un timestamp en formato ISO 8601 con precisión de milisegundos.

### 2. Motor de Reglas Dinámicas con PostgreSQL y Event Sourcing
Las reglas de priorización se almacenan en tablas PostgreSQL con esquema `focusinbox.rules`, donde cada regla incluye campos como `rule_name`, `condition_expression` (sintaxis JSON-Logic), `action_type` (mover, etiquetar, silenciar) y `ttl` (time-to-live en segundos). El motor ejecuta un `event_store` que captura cada cambio de estado en la bandeja de entrada, permitiendo revertir acciones en menos de 50 ms mediante `ROLLBACK TO SAVEPOINT`.

### 3. Sincronización Asíncrona con Colas Redis y Webhooks Firmados
Cada conexión entrante (IMAP, Graph API, Slack RTM) se gestiona mediante un pool de workers Redis Pub/Sub. El sistema expone un endpoint `POST /webhook` que valida la firma HMAC-SHA256 con un secreto rotable (`webhook_secret`). Las colas de reintento implementan backoff exponencial con jitter (base: 1s, máximo: 300s, factor: 2). El rendimiento sostenido es de 12.000 eventos por segundo en hardware estándar (4 vCPU, 8 GB RAM).

## Cómo Usar

### Paso 1: Configurar el Filtro Predictivo
```bash
curl -X POST https://api.focusinbox-eight.io/v1/filters \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "confidence_threshold": 0.85,
    "source_protocol": "IMAP",
    "folder": "INBOX",
    "max_batch_size": 500,
    "expiration_seconds": 3600
  }'
```
Este comando crea un filtro persistente que solo deja pasar correos con una confianza superior al 85%. El parámetro `expiration_seconds` define la vida útil del filtro en segundos; si se omite, el filtro es permanente.

### Paso 2: Definir una Regla de Priorización
```sql
INSERT INTO focusinbox.rules (rule_name, condition_expression, action_type, ttl)
VALUES (
  'urgent_client',
  '{"and": [{"==": [{"var": "sender.domain"}, "cliente.com"]}, {">=": [{"var": "priority"}, 3]}]}',
  'MOVE_TO_FOLDER',
  86400
);
```
La regla anterior mueve automáticamente cualquier mensaje del dominio `cliente.com` con prioridad >= 3 a la carpeta `URGENTE`. El campo `ttl` (86400 segundos = 24 horas) elimina la regla automáticamente después de un día.

### Paso 3: Verificar el Estado de Sincronización
```bash
curl -s https://api.focusinbox-eight.io/v1/status/connections \
  -H "Authorization: Bearer <token>" | jq '.connections[] | select(.protocol=="IMAP") | {id, last_sync, pending_events, error_count}'
```
Este comando devuelve, para cada conexión IMAP activa, el `id` de la conexión, el timestamp del último síncrono (`last_sync` en formato `2023-10-05T14:30:00.123Z`), el número de eventos pendientes en la cola (`pending_events`) y el contador de errores desde el último reinicio (`error_count`). Si `error_count > 0`, se recomienda revisar los logs con `GET /v1/logs/errors`.

## Mejores Prácticas

### Práctica 1: Segmentación de Reglas por Entorno
Para equipos que gestionan múltiples buzones (ventas, soporte, desarrollo), cree reglas con `ttl` corto (ej. 3600 segundos) durante las primeras horas de prueba. Una vez validado el comportamiento, migre a reglas permanentes (`ttl = NULL`). Utilice siempre el campo `rule_name` con un prefijo del equipo (ej. `ventas_prioridad_alta`) para facilitar la auditoría en la tabla `focusinbox.rule_execution_log`.

### Práctica 2: Monitoreo Proactivo con Alertas en Webhooks
Configure un webhook de salida en `POST /webhook/alerts` que reciba eventos de tipo `FILTER_OVERRIDE` (cuando un usuario manualmente mueve un correo filtrado) y `RULE_EXPIRATION`. Esto permite detectar patrones de falsos positivos en tiempo real. Ajuste el `confidence_threshold` dinámicamente según la tasa de `FILTER_OVERRIDE` semanal; si supera el 5%, reduzca el umbral en 0.05.
