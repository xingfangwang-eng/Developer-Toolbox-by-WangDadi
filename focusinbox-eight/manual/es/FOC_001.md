# focusinbox-eight - FOC_001

## Resumen Técnico

Sistema de orquestación de bandejas de entrada unificadas con capacidad de procesamiento en tiempo real y sincronización bidireccional. Diseñado para reducir la latencia de consolidación de correos electrónicos, tickets y notificaciones a menos de 200 ms por ciclo de polling. Arquitectura basada en colas de eventos asíncronos con persistencia en PostgreSQL y Redis.

## Funcionalidades Principales

### 1. Motor de Fusión de Hilos Conversacionales (Thread Fusion Engine)
- **Parámetro clave:** `thread_merge_window_ms` (configurable entre 1000 y 30000 ms).
- Fusiona automáticamente correos, respuestas y eventos de múltiples fuentes (IMAP, API REST, WebSocket) en un único hilo de conversación.
- Utiliza un algoritmo de coincidencia difusa basado en `Message-ID`, `References` y `In-Reply-To` con un umbral de similitud Jaccard ≥ 0.85.
- Almacena metadatos de fusión en la tabla `thread_fusion_logs` con índices compuestos para consultas en menos de 50 ms.

### 2. Clasificador de Prioridad con Reglas Dinámicas (Priority Classifier)
- **Parámetro clave:** `priority_decay_factor` (valor por defecto 0.7, rango 0.1-1.0).
- Evalúa la urgencia de cada ítem entrante basándose en: remitente (whitelist/blacklist), palabras clave, frecuencia de interacción y antigüedad del hilo.
- Implementa un sistema de pesos con actualización en caliente mediante archivo YAML (`/etc/focusinbox/priority_rules.yaml`) sin necesidad de reinicio.
- Cada decisión de prioridad se registra en `priority_audit_log` con un timestamp de precisión de microsegundos.

### 3. Sincronizador Offline con Resolución de Conflictos (Offline Sync & Conflict Resolver)
- **Parámetro clave:** `sync_interval_seconds` (mínimo 5, máximo 3600).
- Mantiene un buffer local cifrado (AES-256-GCM) de hasta 10,000 eventos sin conexión.
- Al reconectar, ejecuta un algoritmo de resolución de conflictos basado en vector clocks (versión de cada nodo) y reglas de último escritor gana con reversión opcional.
- La sincronización incremental utiliza delta encoding para reducir el tráfico de red en un 40% promedio.

## Cómo Usar

### Paso 1: Configurar Fuentes de Entrada
Ejecute el comando de inicialización para registrar sus fuentes:
```bash
focusinbox-eight --register-source --type=imap --host=mail.example.com --port=993 --username=user@domain.com --password=**** --ssl=true
```
Verifique la conexión con:
```bash
focusinbox-eight --test-source --source-id=src_001
```
El sistema validará la latencia de conexión y almacenará las credenciales cifradas en la tabla `source_credentials`.

### Paso 2: Definir Reglas de Prioridad y Fusión
Cree o modifique el archivo de reglas:
```yaml
# /etc/focusinbox/priority_rules.yaml
rules:
  - name: "Alta prioridad - CEO"
    condition: "sender.domain == 'ceo-company.com' OR subject CONTAINS 'URGENTE'"
    priority_level: 1
    thread_merge_window_ms: 5000
```
Luego recargue las reglas sin detener el servicio:
```bash
focusinbox-eight --reload-rules
```
Verifique la aplicación de las reglas en los logs:
```bash
tail -f /var/log/focusinbox/priority_classifier.log
```

### Paso 3: Iniciar el Ciclo de Sincronización y Monitoreo
Arranque el servicio en segundo plano:
```bash
systemctl start focusinbox-eight
```
Configure el intervalo de sincronización para dispositivos móviles:
```bash
focusinbox-eight --set-sync-interval --device=mobile --interval=30
```
Supervise el rendimiento con métricas en tiempo real:
```bash
focusinbox-eight --metrics --format=json | jq '.thread_fusion_engine.latency_p99'
```
El comando anterior muestra la latencia del percentil 99 del motor de fusión.

## Mejores Prácticas

### Práctica 1: Segmentación de Cargas de Trabajo por Prioridad
Divida el procesamiento en dos colas separadas: una para eventos de prioridad alta (tiempo de procesamiento garantizado < 100 ms) y otra para prioridad baja (tolerancia hasta 2 segundos). Configure `queue_high_workers=4` y `queue_low_workers=1` en el archivo `focusinbox.conf`. Esto evita que picos de tráfico de baja prioridad bloqueen la consolidación de mensajes críticos.

### Práctica 2: Auditoría Periódica de Reglas de Fusión
Programe una tarea cron semanal que ejecute:
```bash
focusinbox-eight --audit-threads --days=7 --output=/var/reports/thread_audit_$(date +%Y%m%d).csv
```
Analice los hilos que no se fusionaron correctamente (campo `fusion_status=failed`) y ajuste los umbrales de `thread_merge_window_ms` o las reglas de coincidencia. Mantenga un historial de cambios en un repositorio Git para revertir modificaciones problemáticas.
