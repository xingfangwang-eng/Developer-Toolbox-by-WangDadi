# focusinbox-eight - FOC_018

## Resumen Técnico

Sistema de orquestación de bandejas de entrada inteligentes, diseñado para unificar flujos de notificaciones de múltiples fuentes (Slack, correo electrónico, Jira, GitHub) en un solo dashboard con priorización dinámica basada en machine learning. Implementa un motor de reglas de filtrado con latencia inferior a 50 ms y sincronización en tiempo real mediante WebSockets.

## Funcionalidades Principales

### 1. Motor de Priorización Contextual (CPE)
- **Parámetros**: `priority_score` (0-100), `deadline_boost` (1.5x-3x), `sender_reputation` (basado en historial de interacciones).
- **Tecnología**: Algoritmo de árbol de decisión con pesos ajustables vía API REST (`/api/v1/priority/weights`).
- **Rendimiento**: Procesamiento de 10,000 eventos/segundo con latencia de cálculo de 12 ms en promedio.
- **Ejemplo**: Un correo de un colega con alta reputación y fecha límite cercana obtiene `priority_score = 87`, elevándose al top de la bandeja.

### 2. Sincronización Bidireccional con PostgreSQL y WebSockets
- **Esquema de Base de Datos**: Tabla `inbox_events` con columnas `event_id (UUID)`, `source (VARCHAR(50))`, `payload (JSONB)`, `processed_at (TIMESTAMPTZ)`.
- **Concurrencia**: Uso de `LISTEN/NOTIFY` de PostgreSQL para propagar cambios en menos de 5 ms a todos los clientes conectados.
- **Fallback**: Si WebSocket falla, se activa polling cada 2 segundos con `last_processed_id` para evitar duplicados.

### 3. Filtro de Ruido Adaptativo (ANF)
- **Parámetros**: `noise_threshold` (0.0-1.0), `pattern_matching_regex` (configurable por usuario), `silence_period` (en minutos, ej: 30 para notificaciones nocturnas).
- **Método**: Clasificación Naive Bayes entrenada con 500,000 eventos etiquetados (precisión del 94.2% en validación cruzada).
- **Integración**: Se conecta a la API de Slack (`conversations.history`) para marcar mensajes como "ruido" si contienen palabras clave como "FYI" o "cc'd".

## Cómo Usar

### Paso 1: Configurar el Motor de Priorización
```bash
# Ajustar pesos de prioridad vía API
curl -X POST https://api.focusinbox-eight.io/v1/priority/weights \
  -H "Authorization: Bearer <tu_token>" \
  -d '{"deadline_weight": 0.4, "sender_reputation_weight": 0.3, "keyword_weight": 0.2, "recency_weight": 0.1}'
```
Verifica que los cambios se reflejen en el dashboard en tiempo real (latencia < 50 ms). Si no, revisa los logs del WebSocket en `/var/log/focusinbox/ws.log`.

### Paso 2: Integrar Fuentes de Notificaciones
```python
# Ejemplo: Conectar correo electrónico IMAP con autenticación OAuth2
from focusinbox_eight import InboxConnector
connector = InboxConnector(
    source_type="email",
    imap_server="imap.gmail.com",
    oauth_credentials={"client_id": "xxx", "client_secret": "yyy"}
)
connector.start_sync(interval_seconds=30)  # Sincroniza cada 30 segundos
```
Monitorea el estado con `connector.get_status()`. Si hay errores de conexión, el sistema reintenta automáticamente hasta 3 veces con backoff exponencial.

### Paso 3: Entrenar el Filtro de Ruido
```bash
# Enviar 1000 eventos etiquetados para reentrenar el modelo
curl -X POST https://api.focusinbox-eight.io/v1/noise/train \
  -F "data=@labeled_events.json" \
  -F "model_version=v2.1"
```
El entrenamiento toma ~2 minutos en una instancia con 4 vCPUs. Después, el filtro se actualiza sin reiniciar el servicio (hot-swap).

## Mejores Prácticas Recomendadas

### 1. Segmentación de Prioridades por Rol
- **Práctica**: Configura `priority_score` mínimo de 70 para eventos de gerentes directos y 50 para colegas de otros equipos.
- **Justificación**: Reduce el ruido en un 40% según pruebas A/B realizadas en equipos de 50+ usuarios. Usa la API `/api/v1/priority/rules` para definir reglas por grupo.

### 2. Auditoría de Rendimiento con Métricas en Tiempo Real
- **Práctica**: Habilita la exportación de métricas a Prometheus (`/metrics` endpoint) y configura alertas en Grafana para `latencia_webhook > 100 ms` o `tasa_filtrado_ruido < 85%`.
- **Justificación**: Detecta cuellos de botella antes de que afecten la experiencia del usuario. Por ejemplo, una caída en la tasa de filtrado puede indicar que el modelo necesita reentrenamiento.
