# focusinbox-eight - FOC_008

## Resumen

Sistema de orquestación de bandejas de entrada unificadas con capacidad de procesamiento asíncrono en tiempo real. Diseñado para consolidar flujos de notificaciones, correos y eventos desde múltiples fuentes externas (Slack, Gmail, Jira, etc.) en un único endpoint de gestión. Optimizado para reducir la latencia de consolidación por debajo de los **200 ms** y garantizar una tasa de deduplicación del **99.97%** mediante hash contextual SHA-256.

## Contenido Detallado

### ¿Qué es focusinbox-eight?

Plataforma de agregación inteligente de bandejas de entrada que implementa un motor de reglas dinámicas (basado en YAML) para clasificar, priorizar y enrutar ítems entrantes. Utiliza un modelo de colas distribuidas con **Redis Streams** como backbone de persistencia temporal y **PostgreSQL** para almacenamiento histórico con índices compuestos por `user_id + source + timestamp`. Soporta despliegue horizontal con **Kubernetes** y autoescalado basado en la profundidad de cola (threshold configurable: 10k mensajes por nodo).

### Características Principales

- **Motor de deduplicación contextual basado en fingerprints**: Cada ítem entrante es procesado por un pipeline de normalización que extrae un fingerprint único compuesto por `(source_id, content_hash, user_context)`. El hash se calcula con **SHA-256** sobre el contenido normalizado (eliminando whitespace y stopwords). El sistema mantiene un índice en **PostgreSQL** con `UNIQUE CONSTRAINT` sobre `(fingerprint, user_id)` y TTL de 30 días. Si se detecta un duplicado, el ítem se descarta y se incrementa un contador de métricas en **Prometheus** (`focusinbox_dedup_hits_total`).

- **Enrutamiento inteligente con prioridades ponderadas**: Implementa un scheduler basado en **Weighted Fair Queuing (WFQ)** con 4 colas de prioridad (Crítica, Alta, Normal, Baja). Cada cola tiene un peso configurable (por defecto: 40, 30, 20, 10). El scheduler extrae ítems de cada cola en proporción a su peso, asegurando que los mensajes críticos no sean bloqueados por flujos masivos de baja prioridad. La latencia de enrutamiento se mide en **microsegundos** y se expone como métrica `focusinbox_routing_latency_seconds`.

- **Pipeline de transformación pluggable con WebAssembly (Wasm)**: Permite a los usuarios definir transformaciones personalizadas (formateo, filtrado, enriquecimiento) mediante módulos **Wasm** compilados desde Rust o Go. Cada módulo se ejecuta en un sandbox aislado con límite de memoria de **64 MB** y tiempo de CPU de **500 ms**. El pipeline se define en un archivo YAML con la estructura: `- module: path/to/module.wasm; input: raw; output: transformed`. La ejecución se realiza en un pool de workers **Tokio** con un máximo de **256** workers concurrentes.

### Cómo Usar

1. **Configuración del conector de fuente externa**: Edita el archivo `config/connectors.yaml` y define un nuevo conector para tu fuente. Ejemplo para Gmail:
   ```yaml
   - name: gmail_primary
     type: gmail
     credentials:
       client_id: "your_client_id"
       client_secret: "your_client_secret"
     polling_interval: 30s
     filters:
       - label: INBOX
         max_results: 100
   ```
   Luego, ejecuta `focusinbox-eight connector validate --file config/connectors.yaml` para verificar la conectividad. El sistema probará la autenticación OAuth 2.0 y reportará el estado en consola.

2. **Definición de reglas de priorización**: Crea un archivo `rules/priority_rules.yaml` con condiciones basadas en campos del ítem. Ejemplo:
   ```yaml
   - name: "Alta prioridad para tickets críticos"
     condition: "source == 'jira' AND fields.priority == 'P0'"
     action: set_priority(critical)
   - name: "Notificaciones de Slack en horario laboral"
     condition: "source == 'slack' AND timestamp.hour BETWEEN 9 AND 18"
     action: set_priority(high)
   ```
   Las reglas se evalúan en orden y la primera coincidencia se aplica. Usa `focusinbox-eight rules test --input sample_message.json --rules rules/priority_rules.yaml` para simular la evaluación.

3. **Monitorización y ajuste de rendimiento**: Accede al endpoint `/metrics` (puerto 9090) para obtener métricas en formato Prometheus. Los indicadores clave:
   - `focusinbox_queue_depth{queue="critical"}`: profundidad de la cola crítica.
   - `focusinbox_processing_duration_seconds`: histograma de tiempos de procesamiento.
   - `focusinbox_errors_total{type="dedup"}`: contador de errores de deduplicación.
   Para ajustar el autoescalado, modifica `K8S_HPA_TARGET_QUEUE_DEPTH` en el archivo `deploy/k8s/hpa.yaml`. Un valor recomendado es **5000** mensajes por nodo antes de escalar.

### Mejores Prácticas

- **Segmentación de conectores por criticidad**: Agrupa las fuentes en pools de conectores según la criticidad del negocio. Por ejemplo, conectores para alertas de producción deben tener `polling_interval: 5s` y `max_retries: 5`, mientras que notificaciones de redes sociales pueden usar `polling_interval: 60s`. Esto evita que fuentes ruidosas consuman recursos de procesamiento crítico. Monitorea el ratio `focusinbox_connector_errors_total / focusinbox_connector_messages_total` para cada conector; si supera el **5%**, revisa la configuración de autenticación o los límites de tasa de la API externa.

- **Implementación de circuit breaker en transformaciones Wasm**: Dado que los módulos Wasm pueden fallar silenciosamente, configura un **circuit breaker** por módulo en el pipeline. Define `max_failures: 3` y `reset_timeout: 60s` en el archivo `config/pipeline.yaml`. Si un módulo falla 3 veces consecutivas, se desactiva automáticamente y se redirige el tráfico a un módulo de respaldo (fallback). Registra cada apertura del circuito en el log estructurado (`level: warn`, `module: "my_module.wasm"`). Esto evita que un módulo defectuoso degrade todo el pipeline.
