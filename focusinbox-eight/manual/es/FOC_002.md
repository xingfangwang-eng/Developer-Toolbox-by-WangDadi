# focusinbox-eight - FOC_002

## Resumen

Sistema de orquestación de bandejas de entrada inteligentes, diseñado para consolidar, priorizar y ejecutar acciones automáticas sobre flujos de notificaciones y tareas entrantes desde múltiples fuentes (email, Slack, Jira, GitHub). Optimizado para latencias inferiores a 200 ms por ciclo de procesamiento y sincronización bidireccional con PostgreSQL 15+.

## Contenido Detallado

### ¿Qué es focusinbox-eight?

focusinbox-eight es un motor de clasificación y enrutamiento contextual que opera sobre streams de eventos en tiempo real. Utiliza una arquitectura basada en pipelines asíncronos con colas de prioridad configurables por umbral de urgencia (`priority_threshold: 0.85`) y ventanas de tiempo de expiración (`ttl_window_ms: 300000`). Permite definir reglas de acción basadas en patrones semánticos extraídos mediante modelos ligeros de NLP (FastText + embeddings de 300 dimensiones) que se ejecutan localmente sin dependencia de API externas.

### Características Principales

- **Fusión de fuentes con deduplicación temporal**: Agrega eventos de Slack, Gmail IMAP, webhooks de Jira y GitHub Actions en un único stream unificado. Implementa un algoritmo de deduplicación basado en hash SHA-256 del contenido y una ventana de coincidencia de `dedup_window_ms: 60000`. Los eventos duplicados se marcan con el campo `duplicate_of: <uuid>` y se descartan automáticamente del pipeline de priorización.
- **Clasificación por urgencia y carga cognitiva**: Cada elemento entrante recibe una puntuación compuesta (`urgency_score`) calculada como `(0.4 * time_sensitivity) + (0.3 * sender_authority) + (0.3 * semantic_relevance)`. El campo `cognitive_load` se estima a partir de la longitud del contenido, número de archivos adjuntos y menciones a otros usuarios. Los elementos con `urgency_score > 0.75` se elevan automáticamente al nivel `priority: critical` y se notifican por push vía WebSocket a los clientes conectados.
- **Ejecución de acciones contextuales con rollback**: Soporta la definición de reglas ECA (Event-Condition-Action) que se disparan tras la clasificación. Por ejemplo, ante un tag `bug` en Jira con `priority: P0`, se puede ejecutar un script remoto vía SSH que cree un incidente en PagerDuty y envíe un mensaje a un canal de Slack específico. Cada acción se ejecuta dentro de una transacción PostgreSQL (`BEGIN...COMMIT/ROLLBACK`) que garantiza atomicidad. El campo `action_result` almacena el código de salida (`exit_code: 0`) y el log de ejecución (`exec_log`).

### Cómo Usar

1. **Configuración del pipeline de ingesta**: Edita el archivo `config/pipelines.yaml` y define las fuentes. Por ejemplo, para conectar con Gmail IMAP, añade:  
   ```yaml
   sources:
     - type: imap
       host: imap.gmail.com
       port: 993
       ssl: true
       mailbox: INBOX
       poll_interval_ms: 30000
       credentials:
         user: "tu_correo@gmail.com"
         password: "app_password_generado"
   ```  
   Luego ejecuta `focusinbox-eight --validate-pipelines` para verificar la conexión y que el campo `status` sea `"connected"`.

2. **Definición de reglas de priorización**: Crea un archivo `rules/urgency_rules.json` con umbrales personalizados. Por ejemplo:  
   ```json
   [
     {
       "rule_id": "urgent_slack_mention",
       "source": "slack",
       "condition": "{message}.contains('@here') || {message}.contains('@channel')",
       "action": "set_priority('critical')",
       "ttl_ms": 60000
     }
   ]
   ```  
   Recarga las reglas sin reiniciar el servicio con `curl -X POST http://localhost:8080/api/rules/reload`.

3. **Monitoreo y depuración en tiempo real**: Accede al dashboard web embebido en `http://localhost:8080/dashboard`. Usa el panel de logs en vivo (`/logs/stream`) con filtros por `level: error` o `source: jira`. Para depurar una acción fallida, consulta la tabla `action_audit` en PostgreSQL:  
   ```sql
   SELECT * FROM action_audit WHERE exit_code != 0 ORDER BY created_at DESC LIMIT 10;
   ```

### Mejores Prácticas

- **Segmentación por dominios de carga**: Para equipos con alta rotación de notificaciones (más de 500 eventos/hora), configura múltiples pipelines independientes por dominio (soporte, desarrollo, finanzas) y asigna a cada uno su propia cola de prioridad. Esto evita que eventos de baja urgencia (como cambios de estado en Jira) bloqueen la clasificación de incidentes críticos. Usa el parámetro `pipeline_concurrency: 4` en `config/system.yaml` para paralelizar el procesamiento.
- **Retención y auditoría con ventanas móviles**: Configura la política de retención de eventos en PostgreSQL mediante `pg_cron` para eliminar registros con más de 90 días. Pero conserva siempre los eventos que hayan disparado acciones con `exit_code != 0` (fallos) durante al menos 365 días para facilitar la auditoría. Esto se logra con una regla en `config/retention.yaml`:  
  ```yaml
  retention:
    default_ttl_days: 90
    exceptions:
      - condition: "action_result.exit_code != 0"
        ttl_days: 365
  ```
