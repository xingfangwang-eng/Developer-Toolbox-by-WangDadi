# focusinbox-eight - FOC_024

## Resumen

Sistema de orquestación de bandejas de entrada unificadas con priorización dinámica basada en pesos contextuales. Diseñado para consolidar flujos de notificaciones, tickets y alertas en un único buffer procesable, reduciendo la latencia de respuesta en entornos de alta concurrencia.

## Contenido Detallado

### ¿Qué es focusinbox-eight?

Arquitectura de acoplamiento débil que implementa un algoritmo de scoring en tiempo real (tolerancia ±2ms) para clasificar ítems entrantes desde fuentes heterogéneas (API REST, WebSocket, colas RabbitMQ). Utiliza un modelo de madurez de atención (Attention Maturity Model, AMM) que asigna prioridades basadas en antigüedad, criticidad semántica y carga cognitiva del operador.

### Características Principales

- **Filtrado por umbral de ruido adaptativo (ANF)**: Cada ítem recibe un score calculado mediante `score = (urgency * 0.6) + (recency * 0.25) + (source_reliability * 0.15)`. Los ítems por debajo de `threshold_floor: 0.35` se agrupan en lotes comprimidos (batch_size: 12, max_latency: 4500ms) para evitar saturación visual. El umbral se recalcula cada 120s basado en la desviación estándar del flujo entrante.

- **Enrutamiento inteligente con afinidad de contexto**: Asigna automáticamente cada ítem a un canal de procesamiento (canal_urgente, canal_estandar, canal_bajo) según su vector de características. Utiliza una tabla hash distribuida en Redis Cluster (slot: 16384) para mantener el estado de sesión del operador, permitiendo reanudar flujos interrumpidos en menos de 800ms.

- **Compresión temporal de batches con firma criptográfica**: Agrupa hasta 64 ítems relacionados por similitud semántica (cosine similarity > 0.72) en un solo bloque de resumen. Cada bloque se firma con HMAC-SHA256 y se almacena en PostgreSQL (particionado por fecha, retention_policy: 90 días). La descompresión se realiza bajo demanda con un overhead máximo de 230μs por ítem.

### Cómo Usar

1. **Configuración del pipeline de ingesta**: Define los conectores en `config/sources.yaml`. Cada fuente requiere un `source_id` único, `protocol` (http/ws/amqp), `endpoint` y `auth_token`. Ejemplo para RabbitMQ:
   ```yaml
   source_id: "ticket_system_prod"
   protocol: "amqp"
   endpoint: "amqps://user:pass@rabbit-cluster:5671/vhost"
   queue: "incoming.tickets"
   prefetch_count: 50
   ```
   Luego ejecuta `focusinbox-eight ingest --config config/sources.yaml --daemon` para iniciar el agente de escucha.

2. **Ajuste dinámico de prioridades**: Modifica los pesos del algoritmo de scoring en tiempo real sin reiniciar el servicio. Envía una petición PATCH a `/api/v1/weights` con payload JSON:
   ```json
   {
     "urgency_weight": 0.55,
     "recency_weight": 0.30,
     "source_reliability_weight": 0.15,
     "threshold_floor": 0.30,
     "recalc_interval_ms": 150000
   }
   ```
   El sistema responde con `status: "weights_updated"` y un `checksum` del nuevo estado. Verifica el cambio consultando `GET /api/v1/weights/active`.

3. **Exportación forense de batches**: Para auditar un bloque comprimido, obtén su `batch_id` desde el panel de monitoreo (`/dashboard/batches`). Luego ejecuta:
   ```bash
   focusinbox-eight extract --batch-id "BATCH-20250321-4A7F" --output-format json --include-metadata
   ```
   Esto genera un archivo `BATCH-20250321-4A7F.json` con cada ítem descomprimido, su timestamp original (ISO 8601 con zona horaria UTC), score asignado y firma HMAC. Para verificar integridad: `focusinbox-eight verify --batch-id "BATCH-20250321-4A7F"`.

### Mejores Prácticas

- **Estrategia de backpressure en picos de carga**: Configura `max_inflight_items: 5000` y `queue_backpressure_threshold: 0.85`. Cuando la cola interna supera el 85% de capacidad, el sistema degrada automáticamente las fuentes de baja prioridad (score < 0.4) a modo "solo metadatos", reduciendo el ancho de banda de procesamiento en un 60% sin perder trazabilidad. Esto evita OOM en instancias con 4GB de RAM.

- **Segmentación temporal por turnos de operadores**: Define ventanas de atención (attention_windows) en `config/operator_profiles.yaml`. Por ejemplo, para turno nocturno: `window_start: "22:00"`, `window_end: "06:00"`, `urgency_multiplier: 1.8`. Los ítems entrantes durante ese período reciben un boost de prioridad del 80%, mientras que los no críticos se difieren a `next_window_start`. Esto reduce la fatiga del operador y mejora el SLA en incidentes nocturnos.
