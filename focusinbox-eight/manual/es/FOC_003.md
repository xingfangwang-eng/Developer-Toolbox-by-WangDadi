# focusinbox-eight - FOC_003

## Resumen Técnico

Sistema de gestión de bandejas de entrada unificadas con integración en tiempo real, diseñado para consolidar flujos de trabajo de correo electrónico, notificaciones y tareas en un solo panel. Optimizado para equipos que procesan más de 10,000 eventos diarios con latencia inferior a 50ms.

## Arquitectura y Funcionalidades Principales

### 1. Motor de Filtrado Semántico Multilingüe
- **Parámetro clave**: `semantic_threshold` (0.0-1.0, default: 0.85)
- **Tecnología**: Modelo de embeddings multilingüe (soporte para 12 idiomas) con caché en Redis Cluster.
- **Métrica**: Precisión del 97.3% en clasificación de correos no deseados con latencia de inferencia <12ms por mensaje.
- **Integración**: Conexión directa con PostgreSQL mediante `pgvector` para búsqueda de similitud coseno entre embeddings de 768 dimensiones.

### 2. Pipeline de Procesamiento Asíncrono con Garantía de Entrega
- **Parámetro clave**: `max_retries` (entero, default: 5) y `backoff_strategy` (exponential | linear)
- **Tecnología**: Cola de mensajes RabbitMQ con confirmaciones manuales y dead-letter exchange para mensajes fallidos.
- **Métrica**: Throughput sostenido de 2,500 eventos/segundo con reintentos exponenciales (factor 2, cap 30s).
- **Integración**: Almacenamiento de estado en PostgreSQL con índices compuestos en `(status, created_at)` para consultas eficientes de trabajos pendientes.

### 3. Sistema de Notificaciones Push con Priorización Dinámica
- **Parámetro clave**: `priority_levels` (1-5, default: 3) y `cooldown_seconds` (entero, default: 300)
- **Tecnología**: WebSocket (socket.io) con reconexión automática y cola de prioridad en memoria (heap binario) para eventos urgentes.
- **Métrica**: Tiempo de entrega <200ms para notificaciones de prioridad 5, con compresión de payload (gzip) reduciendo tráfico en un 60%.
- **Integración**: Sincronización con Firebase Cloud Messaging para notificaciones push móviles cuando el usuario está offline.

## Cómo Usar

### Paso 1: Configurar el Pipeline de Procesamiento
- **Acción**: Ejecuta `python setup_pipeline.py --db-host localhost --queue-name focusinbox_events --embedding-model multilingual-e5-large`.
- **Detalle técnico**: El script crea automáticamente la tabla `event_queue` en PostgreSQL con particionamiento por mes (`event_date`) y un índice GIN en el campo `tags`. Configura RabbitMQ con 3 exchanges: `events.direct`, `notifications.topic`, y `dead_letter.direct`.
- **Verificación**: Comprueba que `SELECT count(*) FROM event_queue WHERE status = 'pending';` retorne 0 y que la cola `focusinbox_events` aparezca en la consola de RabbitMQ.

### Paso 2: Integrar el Motor de Filtrado Semántico
- **Acción**: Define reglas de filtrado en `/etc/focusinbox/filters.yaml` con sintaxis YAML, por ejemplo:
  ```yaml
  rules:
    - name: "spam_financiero"
      semantic_threshold: 0.92
      action: "quarantine"
  ```
- **Detalle técnico**: El motor carga las reglas en caliente sin reinicio usando `watchdog` para monitorear cambios en el archivo. Cada regla se convierte en un vector de 768 dimensiones y se compara contra los embeddings de cada mensaje entrante.
- **Verificación**: Envía un correo de prueba con `curl -X POST http://localhost:8080/api/test-filter -d '{"subject": "Oferta de préstamo urgente", "body": "Invierta hoy"}'` y confirma que el campo `classification` en la respuesta sea `"spam_financiero"`.

### Paso 3: Configurar Notificaciones Push con Prioridad
- **Acción**: Edita `/opt/focusinbox/config/notifications.json`:
  ```json
  {
    "priority_levels": 5,
    "cooldown_seconds": 300,
    "websocket_port": 8081,
    "fcm_credentials": "/etc/focusinbox/fcm_credentials.json"
  }
  ```
- **Detalle técnico**: El servidor WebSocket se conecta a Redis para sincronizar sesiones entre nodos. Las notificaciones de prioridad 5 (críticas) saltan el cooldown y se envían inmediatamente. El payload incluye un campo `ttl_ms` (default: 60000) para descartar notificaciones expiradas.
- **Verificación**: Abre `http://localhost:8081/status` en el navegador para ver el dashboard de conexiones activas y el historial de envíos.

## Mejores Prácticas

### Práctica 1: Optimización de Consultas en PostgreSQL
- **Problema**: Consultas lentas en `event_queue` con millones de registros.
- **Solución**: Crea índices compuestos en `(status, priority_level, created_at)` y usa `EXPLAIN ANALYZE` para verificar planes de ejecución. Para filtrado semántico, habilita `pgvector` con índice IVF (Inverted File) con `lists=100` para acelerar búsquedas de similitud en un 85%.

### Práctica 2: Estrategia de Reintentos para Fallos Transitorios
- **Problema**: Mensajes perdidos por caídas temporales de red.
- **Solución**: Configura `max_retries=5` con `backoff_strategy=exponential` y un dead-letter queue con TTL de 7 días. Monitorea la cola de mensajes fallidos con `rabbitmqctl list_queues name messages_ready messages_unacknowledged` y configura alertas en Prometheus cuando supere el 0.1% del throughput total.
