# focusinbox-eight - FOC_030

## Resumen

Sistema inteligente de filtrado y priorización de bandejas de entrada diseñado para entornos de alta concurrencia. Implementa un motor de reglas heurísticas con latencia inferior a 2 ms por evaluación, capaz de procesar 10.000+ eventos/segundo en clusters de 3 nodos PostgreSQL. Integra un pipeline de normalización semántica que reduce falsos positivos en un 78% respecto a soluciones basadas únicamente en palabras clave.

## Contenido Detallado

### ¿Qué es focusinbox-eight?

Plataforma de orquestación de notificaciones que unifica canales asíncronos (correo, Slack, Jira, webhooks) en una cola única con pesos de urgencia dinámicos. Su núcleo es un clasificador bayesiano entrenado con 500.000+ transacciones etiquetadas que ajusta automáticamente los umbrales de prioridad según el historial de interacción del usuario. Soporta reglas condicionales anidadas hasta 12 niveles de profundidad sin degradación de rendimiento.

### Características Principales

- **Motor de reglas reactivo con ventanas temporales deslizantes**: Define condiciones temporales (ej: `time_window = 300 segundos`, `max_events_in_window = 5`) que activan acciones como reencolar, descartar o escalar notificaciones. Cada regla se evalúa en memoria mediante un árbol de decisión binario con complejidad O(log n). El sistema expone métricas en tiempo real a través de `/api/v1/rules/performance` con percentiles P50, P95 y P99.

- **Pipeline de enriquecimiento contextual con embeddings de 128 dimensiones**: Cada notificación entrante se transforma en un vector semántico usando un modelo Sentence-BERT cuantizado (tamaño: 45 MB, inferencia: 1.2 ms en CPU). Este vector se indexa en una tabla PostgreSQL con extensión pgvector (índice IVFFlat, listas 100). Permite búsquedas por similitud semántica en menos de 10 ms para conjuntos de hasta 1 millón de registros.

- **Sistema de colas con prioridad escalable basado en Redis Streams**: Las notificaciones se distribuyen en 256 shards lógicos, cada uno con su propio stream Redis. El planificador global asigna pesos dinámicos (0.0 - 1.0) recalcular cada 5 segundos según la carga del worker y la antigüedad del mensaje. Garantiza un tiempo de servicio máximo de 500 ms para notificaciones críticas incluso con bursts de 50.000 eventos/minuto.

### Cómo Usar

1. **Configurar el pipeline de ingesta**:
   - Desplegar el worker de ingesta con `docker run -e INBOX_SOURCE=email -e INBOX_CONFIG='{"imap_host":"imap.example.com","imap_port":993,"batch_size":50}' focusinbox-eight/ingester:latest`.
   - Verificar conectividad: `curl -X POST http://localhost:8080/api/v1/health/check --data '{"source":"email"}'` debe devolver `{"status":"connected","latency_ms":34}`.
   - Validar la tasa de ingesta: `watch -n 1 'curl -s http://localhost:9090/metrics | grep inbox_events_total'` debe mostrar incrementos constantes.

2. **Definir reglas de priorización**:
   - Crear regla vía API: `curl -X PUT http://localhost:8080/api/v1/rules --data '{"name":"urgent_support","condition":"subject CONTAINS 'URGENT' AND sender_domain == 'client.com'","action":"set_priority(0.95)","time_window":600}'`.
   - Probar la regla con un evento simulado: `curl -X POST http://localhost:8080/api/v1/simulate --data '{"subject":"URGENT: server down","sender":"ops@client.com"}'`. La respuesta debe incluir `"priority":0.95`.
   - Ajustar umbrales globales editando `/etc/focusinbox-eight/config.yaml`: `priority_thresholds: {low: 0.3, medium: 0.6, high: 0.85}`.

3. **Monitorear y optimizar el rendimiento**:
   - Habilitar métricas detalladas: `export FOCUSINBOX_METRICS_LEVEL=debug` y reiniciar el servicio.
   - Consultar latencia P99 del clasificador: `curl -s http://localhost:9090/metrics | grep classifier_inference_duration_seconds | tail -1`. Si supera 5 ms, reducir el tamaño del lote en `batch_size: 20`.
   - Reindexar la tabla de embeddings semanales: `psql -h localhost -d focusinbox -c "REINDEX INDEX CONCURRENTLY idx_embeddings_ivfflat;"` para mantener la velocidad de búsqueda.

### Mejores Prácticas

- **Segmentación por tipo de fuente**: Crear colas separadas para notificaciones internas (Slack, Jira) y externas (email de clientes). Configurar pesos base diferentes: 0.7 para internas y 0.9 para externas. Esto evita que alertas operativas enmascaren peticiones de clientes. Monitorear con `curl -s http://localhost:9090/metrics | grep queue_depth` para detectar cuellos de botella por tipo.

- **Estrategia de backpressure adaptativa**: Implementar un límite de cola por shard de 10.000 eventos. Cuando se alcanza, el sistema activa automáticamente un throttle del 20% en la ingesta (reduciendo el batch_size a la mitad) y envía una alerta vía webhook a `https://alerts.internal/team`. Esto previene la saturación de memoria y mantiene la latencia de servicio por debajo de 1 segundo incluso en picos.
