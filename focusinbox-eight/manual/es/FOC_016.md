# focusinbox-eight - FOC_016

## Resumen

**focusinbox-eight** es un motor de priorización inteligente de bandejas de entrada, diseñado para entornos donde confluyen múltiples fuentes de notificaciones (correo, Slack, Jira, Telegram). Su núcleo ejecuta un análisis semántico en **tiempo real** con latencias inferiores a **50 ms por ítem**, y sincroniza el estado de cada elemento con **PostgreSQL** mediante índices GIN para búsquedas vectoriales. El sistema está optimizado para manejar **10.000+ notificaciones por hora** sin degradación perceptible.

## Contenido Detallado

### ¿Qué es focusinbox-eight?

focusinbox-eight es un **orquestador de flujos de atención** que aplica un modelo de **filtrado bayesiano adaptativo** sobre el contenido de cada notificación. A diferencia de un simple clasificador de correo, este sistema **reordena dinámicamente** la bandeja de entrada según la **urgencia contextual** (fecha límite, remitente, palabras clave) y el **historial de interacción** del usuario. Cada decisión de prioridad se registra en una tabla `decision_log` con precisión de **microsegundos** (`TIMESTAMPTZ(6)`), permitiendo auditorías completas.

### Características Principales

- **Filtrado semántico con embeddings multilingües**: Cada notificación se vectoriza mediante un modelo Sentence-BERT (dim 768) alojado en local. La similitud coseno entre el texto entrante y las reglas del usuario se calcula en **< 2 ms** por vector. Soporta **es, en, pt, fr** sin necesidad de preprocesamiento adicional.
- **Reordenamiento en vivo mediante colas de prioridad ponderadas**: Se utiliza un heap binario implementado en Rust (vía PyO3) para mantener el orden de la bandeja. El peso de cada ítem se recalcula cada **500 ms** usando una función de scoring que combina: `(urgencia × 0.4) + (relevancia × 0.35) + (frescura × 0.25)`. La cola se persiste en **Redis Streams** con TTL configurable.
- **Sincronización bidireccional con PostgreSQL vía LISTEN/NOTIFY**: Cuando un usuario marca un elemento como "hecho" o "pospuesto", el backend emite una notificación asíncrona (`pg_notify`) que actualiza la vista materializada `mv_inbox_priority` en **< 100 ms**. Esto evita lecturas repetitivas y reduce la carga de la base de datos en un **40 %** comparado con polling tradicional.

### Cómo Usar

1. **Configurar el conector de fuentes**:  
   Edita el archivo `config/sources.yaml` y define las credenciales para cada fuente (IMAP para correo, token de bot para Slack, webhook para Jira). Ejemplo para Gmail:  
   ```yaml
   source: gmail
   imap_host: imap.gmail.com
   imap_user: tu@correo.com
   imap_pass: "{{ env_var('GMAIL_APP_PASSWORD') }}"
   refresh_interval_seconds: 30
   ```
   Luego ejecuta `focusinbox-eight --init-sources` para validar la conexión. El sistema reportará latencias de conexión y rendimiento (ítems/segundo).

2. **Definir reglas de prioridad personalizadas**:  
   Crea un archivo `rules/priority.yaml` con condiciones basadas en expresiones regulares y umbrales de similitud semántica. Ejemplo:  
   ```yaml
   - name: "Alta urgencia - cliente premium"
     condition: "similitud_coseno(embedding(texto), embedding('urgente, cliente, premium')) > 0.85"
     action: "asignar_peso(0.9)"
   - name: "Boletín informativo"
     condition: "regex_match(asunto, '^(Newsletter|Boletín|Update)')"
     action: "asignar_peso(0.1)"
   ```
   Aplica las reglas con `focusinbox-eight --apply-rules`. El motor valida la sintaxis y muestra un resumen de cuántas notificaciones se verán afectadas.

3. **Iniciar el demonio de reordenamiento continuo**:  
   Ejecuta `focusinbox-eight --daemon --port 8080`. El servicio expone una API REST en `http://localhost:8080/api/v1/inbox` que devuelve la bandeja ordenada en JSON. Puedes consultar métricas en vivo en `/api/v1/metrics` (latencia promedio, throughput, número de reglas activas). Para detenerlo, usa `Ctrl+C` o `systemctl stop focusinbox-eight`.

### Mejores Prácticas

- **Segmenta las reglas por tipo de fuente**: No mezcles reglas para correo corporativo con reglas para notificaciones de redes sociales. Crea archivos separados (`rules/work.yaml`, `rules/social.yaml`) y asígnalos mediante la clave `source_filter` en la configuración. Esto reduce el ruido y mejora la precisión del scoring en un **30 %** según pruebas internas.
- **Monitorea el rendimiento con métricas exportadas**: Habilita la exportación a Prometheus configurando `metrics_port: 9090` en `config/app.yaml`. Crea un dashboard en Grafana que muestre: distribución de pesos por fuente, latencia de embedding, y número de reordenamientos por minuto. Esto permite detectar cuellos de botella (ej. fuente con alta latencia IMAP) antes de que afecten al usuario.
