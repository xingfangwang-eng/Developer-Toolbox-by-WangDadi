# focusinbox-eight - FOC_025

## Resumen

Sistema de orquestación de bandejas de entrada unificadas con capacidad de procesamiento en tiempo real, diseñado para consolidar flujos de trabajo de múltiples fuentes (correo, tickets, mensajería) en un único buffer de priorización. Optimizado para entornos con latencia máxima de 120 ms y sincronización bidireccional con PostgreSQL 15+.

## Contenido Detallado

### ¿Qué es focusinbox-eight?

Plataforma de agregación cognitiva que implementa un algoritmo de `weighted-inbox-scoring` para clasificar entradas según 14 parámetros configurables (urgencia, remitente, contenido semántico, SLA residual). Utiliza un motor de reglas en C++ embebido (vía FFI) para evaluar 8.000 transacciones/segundo sin bloqueo de E/S.

### Características Principales

- **Fusión de flujos con deduplicación determinista**: Emplea un `hash-ring` de 256 bits sobre campos `message_id + source_channel + timestamp_ns` para eliminar duplicados en menos de 5 μs. Integra con Exchange Web Services, Gmail API v1 y Slack Events API mediante workers asíncronos con backpressure controlada por `semáforos de tokens` (máx. 1.024 conexiones concurrentes).

- **Clasificación heurística con feedback loop**: El módulo `priority-engine` asigna un score entre 0.0 y 1.0 usando reglas definidas en YAML (ej: `urgent_patterns: ["ASAP", "crítico", "SLA_breach"]`). Cada 30 minutos, un proceso de `retroalimentación bayesiana` ajusta los pesos según la tasa de acierto en acciones del usuario (abrir, archivar, escalar). Datos persistidos en tablas `inbox_scores` con índices compuestos por `user_id + score DESC`.

- **Exportación de contexto en crudo**: Genera archivos JSON con estructura `{event_id, raw_payload, parsed_entities, confidence}`. Cada exportación incluye un sello de tiempo con precisión de nanosegundos (`clock_gettime(CLOCK_MONOTONIC_RAW)`) y checksum SHA-256 para auditoría. Soporta volcado incremental a S3 o buckets locales mediante el comando `focusinbox-eight export --format=json --bucket=gs://inbox-archive`.

### Cómo Usar

1. **Configuración inicial del motor de reglas**  
   Ejecutar `focusinbox-eight init --config-path=./rules.yaml --db-uri=postgresql://user:pass@localhost:5432/focusinbox`.  
   Editar el archivo `rules.yaml` para definir al menos 3 criterios de scoring:  
   ```yaml
   rules:
     - name: "urgent_sla"
       field: "subject"
       pattern: "ASAP|urgente"
       weight: 0.8
     - name: "vip_sender"
       field: "from"
       list: ["ceo@corp.com", "director@corp.com"]
       weight: 1.0
   ```  
   Validar con `focusinbox-eight validate --rules=rules.yaml`.

2. **Arranque del servicio en modo demonio**  
   Lanzar con `focusinbox-eight daemon --port=9090 --workers=8 --cache-ttl=300s`.  
   Verificar estado: `curl -X GET http://localhost:9090/health` → debe devolver `{"status":"ok","uptime_seconds":1234,"queue_depth":0}`.  
   Para monitoreo en vivo: `focusinbox-eight stats --interval=1s` (muestra throughput, latencia p99, errores de conexión).

3. **Exportación programada de datos**  
   Crear un cron job que ejecute cada 6 horas:  
   `focusinbox-eight export --since=6h --format=jsonl --output=/mnt/backup/inbox_$(date +%Y%m%d_%H%M%S).jsonl --compress=gzip`.  
   Verificar integridad: `focusinbox-eight verify --file=/mnt/backup/inbox_*.jsonl.gz --checksum=sha256`.

### Mejores Prácticas

- **Particionamiento de tablas por usuario**: Para evitar contención en escrituras concurrentes, crear tablas mensuales `inbox_entries_2025_01` con partición por rango de `user_id`. Usar `pg_partman` para auto-gestión. Esto reduce el tiempo de consultas `SELECT ... WHERE user_id = X AND score > 0.5` de ~800 ms a ~12 ms.

- **Circuit breaker en integraciones externas**: Configurar timeout máximo de 3 segundos por llamada a APIs (Gmail, Slack). Si se superan 5 fallos consecutivos en 60 segundos, el worker debe pausar automáticamente y registrar un evento en `alert_log`. Implementar con `resilience4j` o similar.
