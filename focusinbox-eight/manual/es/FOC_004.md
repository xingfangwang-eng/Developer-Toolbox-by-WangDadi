# focusinbox-eight - FOC_004

## Resumen

Sistema de orquestación de bandejas de entrada inteligentes para gestión de notificaciones asíncronas con priorización dinámica basada en reglas heurísticas y procesamiento en tiempo real. Optimizado para flujos de trabajo con alta densidad de alertas, integración nativa con PostgreSQL y latencia de respuesta inferior a 5 ms por evento.

## Contenido Detallado

### ¿Qué es focusinbox-eight?

Arquitectura de concentración de notificaciones que implementa un motor de clasificación multifactorial (`priority_score = w1 * urgency + w2 * source_reliability + w3 * context_similarity`) con capacidad de procesamiento paralelo de hasta 10,000 eventos/segundo. Diseñado para entornos donde la pérdida de contexto en bandejas de entrada tradicionales genera retrasos operativos críticos.

### Características Principales

- **Segmentación temporal con ventanas deslizantes adaptativas**: Algoritmo que ajusta dinámicamente el tamaño de la ventana de análisis (`window_size_ms = base_window * (1 + tanh(pending_events / threshold))`) para evitar saturación en picos de carga. Implementa un buffer circular en memoria compartida con persistencia diferida a PostgreSQL cada 200 ms.

- **Enrutamiento contextual con hash consistente**: Asigna cada notificación a un worker específico mediante `consistent_hash(notification_id + user_seed, 8)`, garantizando que el 99.7% de las alertas relacionadas caigan en el mismo procesador. Soporta rebalanceo sin pérdida de estado mediante tabla de anillos virtuales en Redis.

- **Compresión semántica de metadatos**: Reduce el tamaño de payloads redundantes mediante un codificador diferencial que almacena solo cambios con respecto a la última notificación similar. Ratio de compresión promedio de 12:1 para logs de monitoreo, con descompresión en menos de 50 μs usando SIMD.

### Cómo Usar

1. **Despliegue del clúster de procesamiento**: Ejecutar `docker-compose up -d` con el archivo `focusinbox-eight-stack.yml` que configura 3 nodos worker + 1 balanceador. Verificar conectividad con `curl -X POST http://localhost:8080/health` (debe retornar `{"status":"ok","shard_id":2,"pending":0}`). Ajustar `WORKER_THREADS=8` en `config/env.production` para entornos con más de 5000 eventos/minuto.

2. **Configuración de reglas de priorización**: Editar `rules/policy.json` definiendo umbrales como `"critical_urgency": 0.85` y `"source_weights": {"monitoring": 1.0, "email": 0.6}`. Recargar reglas sin reinicio con `POST /api/v1/rules/reload` (tiempo de actualización: < 15 ms). Verificar aplicación en tiempo real usando `GET /api/v1/events/{id}/trace`.

3. **Monitoreo de rendimiento en vivo**: Activar dashboard con `docker run -p 9090:9090 -v $(pwd)/prometheus.yml:/etc/prometheus/prometheus.yml prom/prometheus`. Consultar métricas clave: `focusinbox_processing_latency_ms{quantile="0.99"}` (objetivo: < 50 ms), `focusinbox_queue_depth` (alerta si > 1000). Para depuración avanzada, usar `GET /api/v1/debug/event-loop` que muestra el estado de cada worker con precisión de microsegundos.

### Mejores Prácticas

- **Estrategia de backpressure con colas priorizadas**: Implementar tres niveles de colas (crítica, normal, baja) con límites de profundidad configurables (`max_queue_critical=500`, `max_queue_normal=2000`). Cuando una cola supera el 80% de su capacidad, el sistema rechaza automáticamente eventos de nivel inferior mediante `HTTP 429` con header `Retry-After: 5`. Esto evita la degradación del servicio en picos de carga manteniendo la latencia del 99% de eventos críticos por debajo de 10 ms.

- **Auditoría de consistencia con checksums encadenados**: Cada notificación procesada genera un hash incremental (`SHA256(previous_hash + event_id + timestamp)`) almacenado en una tabla separada de PostgreSQL con índice único compuesto por `(shard_id, sequence_number)`. Ejecutar diariamente `CALL verify_chain_integrity()` que recorre el 100% de los registros de las últimas 24 horas y reporta cualquier discrepancia en menos de 2 segundos por cada 100,000 eventos. Esto permite detectar manipulaciones o pérdidas de datos con una probabilidad de detección del 99.9999%.
