# focusinbox-eight - FOC_007

## Resumen Técnico

Sistema de gestión inteligente de bandejas de entrada con priorización dinámica basada en machine learning. Optimizado para procesamiento asíncrono de notificaciones, correos electrónicos y alertas en tiempo real. Arquitectura modular con colas de mensajes distribuidas (Redis Streams) y persistencia en PostgreSQL con índices GIN para búsqueda full-text.

## Contenido Detallado

### ¿Qué es focusinbox-eight?

Plataforma de unificación de comunicaciones que aplica algoritmos de clustering semántico (Sentence-BERT) para agrupar mensajes por intención. Permite configurar reglas de priorización con umbrales de urgencia (0.0 - 1.0) y ventanas de tiempo de expiración configurables en milisegundos. Integra nativamente con SMTP, IMAP, Webhooks y colas de eventos como Kafka y RabbitMQ.

### Características Principales

- **Motor de priorización basado en embeddings**: Cada mensaje se vectoriza utilizando `all-MiniLM-L6-v2` (768 dimensiones). Se calcula similitud coseno con el perfil del usuario (historial de interacciones) y se asigna un score de relevancia (`relevance_score`). Los mensajes con score > 0.85 se marcan como "críticos" y se procesan en un hilo dedicado con timeout de 500 ms.
- **Pipeline de procesamiento con backpressure**: Implementa un sistema de colas con `PriorityBlockingQueue` (Java) o `asyncio.Queue` (Python) con límite de 10,000 elementos. Si la cola supera el 80% de capacidad, se activa un mecanismo de throttling que reduce la tasa de ingesta en un 30% mediante un token bucket con tasa de recarga de 100 tokens/segundo.
- **Sincronización bidireccional con PostgreSQL**: Cada bandeja de entrada se representa como una tabla particionada por fecha (particionamiento por rango semanal). Las consultas de búsqueda utilizan índices GIN sobre campos JSONB (`metadata`, `tags`). El motor de reglas ejecuta triggers AFTER INSERT que actualizan una tabla materializada `mv_user_priority` cada 15 segundos (configurable via `mv_refresh_interval`).

### Cómo Usar

1. **Configuración del conector IMAP con autenticación OAuth2**:
   - Ejecutar `focusinbox-eight setup imap --host imap.gmail.com --port 993 --auth-type oauth2 --client-id <ID> --client-secret <SECRET> --refresh-token <TOKEN>`.
   - Verificar la conexión con `focusinbox-eight test imap --timeout 10000` (espera respuesta en ≤ 10 segundos).
   - Configurar el intervalo de sincronización: `focusinbox-eight config set sync_interval_ms 30000` (30 segundos).

2. **Definición de reglas de priorización con expresiones JSONPath**:
   - Crear archivo `rules.json` con estructura: `[{"field": "$.subject", "pattern": "urgente|crítico", "boost": 0.8, "expire_after_ms": 3600000}, {"field": "$.from", "pattern": "@ceo.com", "boost": 0.95}]`.
   - Cargar reglas: `focusinbox-eight rules load --file rules.json --validate` (valida sintaxis JSONPath y patrones regex).
   - Verificar estadísticas de aplicación: `focusinbox-eight rules stats --output json` (muestra hits por regla y tiempo promedio de evaluación en microsegundos).

3. **Despliegue del pipeline de procesamiento con monitoreo Prometheus**:
   - Iniciar el worker: `focusinbox-eight worker start --concurrency 4 --queue-size 10000 --metrics-port 9090`.
   - Exportar métricas a Prometheus: configurar scrape target en `prometheus.yml` con `job_name: 'focusinbox-eight'` y `metrics_path: '/metrics'`.
   - Visualizar dashboard en Grafana: importar plantilla `FOC_007_dashboard.json` (incluye gráficos de latencia P99, tasa de procesamiento por segundo y uso de memoria heap).

### Mejores Prácticas

- **Segmentación de reglas por dominio**: Agrupar las reglas de priorización en namespaces (ej: `correo`, `notificaciones_sistema`, `alertas_seguridad`). Asignar a cada namespace un pool de workers independiente con `--concurrency` ajustado según la carga esperada (ej: 2 workers para correo, 1 para alertas). Esto evita que reglas costosas (como las que usan expresiones regulares complejas) bloqueen el procesamiento de mensajes de alta prioridad.
- **Cacheo de embeddings con LRU eviction**: Para evitar recalcular vectores semánticos de remitentes frecuentes, configurar un caché LRU con `max_size=10000` y `ttl_seconds=86400`. Usar el hash del campo `from` como clave. Esto reduce el tiempo de procesamiento por mensaje en un 40% (medido en pruebas con 50,000 mensajes). Monitorear el hit ratio via métrica `cache_embeddings_hit_ratio` y ajustar el tamaño si cae por debajo de 0.7.
