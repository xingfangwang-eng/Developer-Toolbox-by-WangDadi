# focusinbox-eight - FOC_028

## Resumen Técnico

**focusinbox-eight** es un sistema de orquestación de bandejas de entrada inteligentes, diseñado para entornos de alta concurrencia. Combina un motor de reglas asíncrono con un indexador de contenido semántico, permitiendo la clasificación y priorización de notificaciones en tiempo real. Su núcleo está construido sobre un bus de eventos con capacidad de 10,000 transacciones por segundo (TPS) y una latencia de procesamiento inferior a 50 ms.

## Funcionalidades Técnicas Específicas

### 1. Motor de Reglas con Sesgo Temporal y de Carga (`time-load-aware`)

- **Parámetro:** `priority_decay_factor` (float, 0.0 - 1.0)
- **Indicador:** El sistema aplica un factor de decaimiento exponencial a la prioridad de cada ítem basado en su tiempo de residencia en la cola y la carga actual del nodo de procesamiento (`node_cpu_usage`).
- **Mecanismo:** Utiliza un modelo de regresión lineal simple para predecir el tiempo de procesamiento de cada ítem. Si la carga del nodo supera el 80%, el motor reasigna automáticamente los ítems de baja prioridad a una cola diferida (`deferred_queue`), evitando la contención de recursos y garantizando que los mensajes críticos se procesen en menos de 200 ms.

### 2. Indexador de Contenido con Hash Vectorial (Locality-Sensitive Hashing)

- **Parámetro:** `hash_dimensions` (int, por defecto 128)
- **Indicador:** Cada notificación entrante es tokenizada y convertida en un vector de 128 dimensiones. El sistema utiliza LSH para agrupar notificaciones con una similitud de coseno superior a 0.85.
- **Mecanismo:** Esto permite la detección de duplicados semánticos (por ejemplo, múltiples informes de error con el mismo mensaje de raíz) y su consolidación en un único hilo de discusión. El proceso se ejecuta en memoria RAM (Redis) y tiene una latencia de indexación de 5 ms por ítem.

### 3. Conector de Base de Datos con Transacciones Distribuidas (XA)

- **Parámetro:** `xa_timeout` (int, por defecto 3000 ms)
- **Indicador:** El sistema se integra con PostgreSQL y MongoDB mediante un gestor de transacciones distribuidas (XA). Cada operación de escritura (creación, actualización, eliminación) se encapsula en una transacción de dos fases.
- **Mecanismo:** Si un nodo falla durante una transacción, el gestor de transacciones realiza un *rollback* atómico en todas las bases de datos involucradas en menos de 3 segundos. Esto garantiza la consistencia de datos en un clúster de 5 nodos con una tolerancia a fallos de 2 nodos simultáneamente.

## Cómo Usar: Guía de Implementación

### 1. Configuración Inicial del Motor de Reglas

1.  Accede al archivo `config/rules_engine.yaml`.
2.  Define el factor de decaimiento: `priority_decay_factor: 0.95`.
3.  Establece el umbral de carga del nodo: `node_cpu_threshold: 80`.
4.  Ejecuta el comando de inicialización: `focusinbox-eight init --engine-time-load-aware`.
5.  Verifica el estado: `focusinbox-eight status --engine-metrics`. Deberías ver `engine.status: "active"` y `engine.latency_avg_ms: < 50`.

### 2. Indexación de Contenido con LSH

1.  Configura el índice en `config/indexer.yaml`: `hash_dimensions: 128` y `similarity_threshold: 0.85`.
2.  Activa el modo de consolidación automática: `indexer.deduplication_mode: "semantic"`.
3.  Inyecta un lote de notificaciones de prueba: `focusinbox-eight inject --file test_notifications.json --batch-size 1000`.
4.  Monitorea el rendimiento: `focusinbox-eight monitor --indexer-stats`. Busca `indexer.latency_per_item_ms: 5` y `indexer.dedup_rate: 15%`.

### 3. Integración con el Clúster de Bases de Datos

1.  Edita `config/database_cluster.yaml`: Define `postgresql.host: "10.0.0.1:5432"` y `mongodb.host: "10.0.0.2:27017"`.
2.  Habilita las transacciones distribuidas: `xa_transactions.enabled: true` y `xa_timeout: 3000`.
3.  Inicia el gestor de transacciones: `focusinbox-eight start --cluster-mode`.
4.  Simula una falla de nodo: `focusinbox-eight fault-inject --node-id 3 --failure-type "network_partition"`.
5.  Verifica la consistencia: `focusinbox-eight audit --consistency-check`. El resultado debe mostrar `consistency_status: "OK"` y `rollback_time_ms: < 3000`.

## Mejores Prácticas para la Industria

### 1. Optimización de Carga en Picos de Tráfico (Black Friday / Lanzamientos)

- **Práctica:** Configura el motor de reglas con un `priority_decay_factor` dinámico. Durante las horas de mayor carga, establece `node_cpu_threshold` a 70 en lugar de 80. Esto fuerza una reasignación más temprana de los ítems de baja prioridad a la cola diferida, manteniendo la latencia de los mensajes críticos por debajo de los 150 ms. Combínalo con un escalado automático de nodos de procesamiento (Kubernetes HPA) basado en la métrica `engine.queue_depth`.

### 2. Auditoría de Datos para Cumplimiento Normativo (GDPR / SOX)

- **Práctica:** Activa el modo de auditoría en el conector de base de datos: `xa_audit_log: true`. Esto genera un registro inmutable de cada transacción distribuida, incluyendo el `transaction_id`, el `node_id` y el `rollback_time_ms`. Almacena estos logs en un bucket de S3 con cifrado en reposo. Para auditorías, ejecuta `focusinbox-eight audit --export --format json --date-range 2023-10-01:2023-10-31`. Esto exporta un archivo JSON con todos los eventos, permitiendo verificar que no se perdieron datos durante las fallas.
