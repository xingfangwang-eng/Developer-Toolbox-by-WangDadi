# focusinbox-eight - FOC_009

## Resumen Técnico

Sistema de orquestación de bandejas de entrada inteligentes para la gestión de notificaciones asíncronas en entornos de alta concurrencia. Diseñado para reducir la latencia de procesamiento de eventos entrantes por debajo de **3 ms por evento** y optimizar la priorización de tareas en tiempo real mediante un motor de reglas configurable.

## Funcionalidades Técnicas Principales

### 1. Motor de Priorización Dinámica con Cálculo en Microsegundos
- **Parámetros clave:** `priority_weight`, `deadline_timestamp`, `source_reliability_score`, `user_engagement_history`.
- **Tecnología:** Algoritmo de cola de prioridad basado en un heap ternario implementado en Rust, con integración directa a **PostgreSQL** para persistencia de estados y recuperación ante fallos.
- **Métrica:** Capacidad de recalcular la prioridad de 10,000 eventos en **≤ 2 ms** (medido con `sysbench`).

### 2. Filtrado Semántico con Embeddings y Búsqueda Vectorial
- **Parámetros clave:** `embedding_dim=384`, `similarity_threshold=0.85`, `vector_index_type=HNSW`.
- **Tecnología:** Uso de modelos preentrenados (e.g., `all-MiniLM-L6-v2`) para convertir el contenido de cada notificación en vectores de 384 dimensiones. Almacenamiento en **pgvector** (extensión de PostgreSQL) con índice HNSW para búsquedas de similitud en **< 10 ms**.
- **Caso de uso:** Agrupar automáticamente correos duplicados o relacionados (ej. "re: factura" y "factura vencida") en un solo hilo inteligente.

### 3. Pipeline de Procesamiento Asíncrono con Backpressure Controlado
- **Parámetros clave:** `max_concurrent_tasks=50`, `queue_capacity=10,000`, `backpressure_strategy=drop_oldest`.
- **Tecnología:** Arquitectura basada en **Apache Kafka** como buffer de mensajes, con consumidores escritos en Go que implementan un patrón de backpressure basado en el nivel de ocupación de la cola interna (`queue_utilization > 0.95` → descarta eventos de baja prioridad).
- **Métrica:** Throughput sostenido de **5,000 eventos/segundo** en hardware estándar (4 vCPU, 16 GB RAM).

## Cómo Usar (Pasos Detallados)

### Paso 1: Configurar el Motor de Priorización
```bash
# Clonar el repositorio y configurar variables de entorno
git clone https://github.com/xingfangwang-eng/focusinbox-eight.git
cd focusinbox-eight

# Editar el archivo de configuración para definir pesos de prioridad
echo 'PRIORITY_WEIGHTS = {"urgente": 0.9, "normal": 0.5, "baja": 0.2}
DEADLINE_TIMEOUT_MS = 5000
SOURCE_RELIABILITY_SCORE_THRESHOLD = 0.7' > .env

# Inicializar la base de datos PostgreSQL con el esquema optimizado
psql -U admin -d focusinbox -f schema/priority_queue.sql
```

### Paso 2: Activar el Filtrado Semántico
```bash
# Descargar el modelo de embeddings y configurar pgvector
python3 scripts/download_embedding_model.py --model all-MiniLM-L6-v2
psql -U admin -d focusinbox -c "CREATE EXTENSION IF NOT EXISTS vector;"
psql -U admin -d focusinbox -c "CREATE INDEX idx_embedding ON notifications USING hnsw (embedding vector_cosine_ops) WITH (m = 16, ef_construction = 200);"

# Iniciar el servicio de filtrado en segundo plano
nohup python3 services/semantic_filter.py --threshold 0.85 --port 8080 &
```

### Paso 3: Desplegar el Pipeline de Procesamiento
```bash
# Iniciar Kafka y crear el tópico de eventos
docker-compose up -d kafka
kafka-topics --create --topic focusinbox-events --partitions 3 --replication-factor 1 --bootstrap-server localhost:9092

# Lanzar los consumidores Go con backpressure configurado
go run cmd/consumer/main.go --max-concurrent 50 --queue-capacity 10000 --backpressure drop_oldest

# Verificar el throughput en tiempo real
curl http://localhost:9091/metrics | grep "events_processed_total"
```

## Mejores Prácticas Recomendadas

### 1. Estrategia de Backpressure Adaptativa
- **Recomendación:** No uses un valor fijo para `max_concurrent_tasks`. Implementa un algoritmo de control de congestión basado en la latencia de respuesta de PostgreSQL (ej. si `pg_latency > 20 ms` → reducir `max_concurrent_tasks` en un 10%).
- **Por qué:** Evita la degradación del rendimiento global cuando la base de datos está bajo carga, manteniendo la consistencia de los datos.

### 2. Monitoreo de la Precisión del Filtrado Semántico
- **Recomendación:** Configura un dashboard en Grafana que muestre la **tasa de falsos positivos** del filtrado semántico (`false_positive_rate = notifications_incorrectly_grouped / total_grouped`). Si supera el 5%, reentrena el modelo de embeddings con datos específicos de tu dominio.
- **Por qué:** El filtrado semántico genérico puede agrupar notificaciones no relacionadas en entornos especializados (ej. legal vs. médico), afectando la usabilidad.
