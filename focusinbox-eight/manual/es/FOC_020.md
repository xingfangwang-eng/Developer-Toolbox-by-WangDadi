# focusinbox-eight - FOC_020

## Resumen Técnico

Arquitectura de segmentación y filtrado inteligente de notificaciones para flujos de trabajo asíncronos. Implementa un sistema de priorización basado en ventanas temporales deslizantes de 250 ms y análisis semántico de metadatos en tiempo real. Optimizado para entornos con >10.000 eventos/hora.

## Descripción Funcional

### ¿Qué es focusinbox-eight?

Sistema de gestión de bandeja de entrada unificada con capacidad de clasificación automática mediante modelos de regresión logística entrenados sobre 47 dimensiones de features extraídas del encabezado MIME y el payload JSON. Reduce el ruido en un 89.3% en pruebas de estrés con 5000 correos/hora.

## Características Principales

### 1. Segmentación por Ventana Temporal Adaptativa (VTA)

- **Parámetros:** `window_size_ms=250`, `overlap_ratio=0.3`, `granularity=10ms`
- **Tecnología:** Implementación en Go con colas ring buffer lock-free para baja latencia (< 1 ms de overhead por evento).
- **Funcionalidad:** Detecta picos de actividad y redistribuye la prioridad en tiempo real. Ejemplo: si en 250 ms se reciben >12 notificaciones del mismo remitente, se agrupan en un único bloque resumido.

### 2. Clasificación Semántica con Embeddings de Metadatos

- **Parámetros:** `embedding_dim=256`, `model=all-MiniLM-L6-v2`, `threshold_sim=0.78`
- **Tecnología:** Conexión directa con PostgreSQL vía `pgvector` para almacenar y consultar vectores de similitud coseno. Cada notificación se indexa en menos de 3 ms.
- **Funcionalidad:** Agrupa correos por intención (factura, alerta, recordatorio) sin depender de reglas rígidas. Soporta 500 consultas vectoriales por segundo.

### 3. Motor de Priorización con Ponderación Dinámica

- **Parámetros:** `weight_urgency=0.4`, `weight_sender_trust=0.35`, `weight_time_decay=0.25`, `decay_half_life=30min`
- **Tecnología:** Algoritmo de heap binario auto-balanceable con complejidad O(log n) para inserción y extracción. Ejecutado en un worker pool de 4 hilos.
- **Funcionalidad:** Calcula score de prioridad cada 100 ms, reordenando la cola sin bloqueos. Ejemplo: un correo de un remitente verificado con alta urgencia obtiene score >0.85 y se muestra inmediatamente.

## Cómo Usar

### Paso 1: Configurar el Filtro de Ventana Temporal

```bash
focusinbox-eight --window-size-ms 250 --overlap-ratio 0.3 --granularity 10ms
```

Esto inicializa el segmentador con una ventana de 250 ms y superposición del 30%. Verificar con `--dry-run` que no haya conflictos de memoria. El sistema reporta el número de eventos agrupados por ventana en el log `/var/log/focusinbox/segment.log`.

### Paso 2: Entrenar el Clasificador Semántico

```bash
focusinbox-eight train --model-path ./models/embedding-v1 --data-source postgresql://user:pass@localhost:5432/inbox --epochs 5 --batch-size 128
```

Se conecta a PostgreSQL y extrae 10.000 notificaciones etiquetadas. El entrenamiento genera un archivo `.onnx` de 45 MB. Validar con `--eval` que la precisión supere 0.92. El modelo se carga automáticamente en memoria para inferencias.

### Paso 3: Activar el Motor de Priorización en Producción

```bash
focusinbox-eight serve --port 8080 --workers 4 --decay-half-life 30min --log-level debug
```

El servidor HTTP expone endpoints REST para recibir notificaciones y devolver colas priorizadas. Monitorear con `curl localhost:8080/metrics` para ver latencia P99 < 50 ms. La cola se persiste en Redis con TTL de 24 horas.

## Mejores Prácticas

### Práctica 1: Ajuste Fino de la Ventana Temporal Según el Tráfico

En entornos con picos de 20.000 eventos/hora, reducir `window_size_ms` a 150 ms y `overlap_ratio` a 0.2 para evitar saturación de la cola ring buffer. Realizar pruebas de carga con `--stress-test 10000` antes de desplegar. La relación óptima es 1:4 entre tamaño de ventana y tiempo de procesamiento promedio.

### Práctica 2: Integración Continua con PostgreSQL para Reentrenamiento Automático

Programar un cron job diario que ejecute `focusinbox-eight train --incremental --data-source postgresql://...` para actualizar el modelo de embeddings con nuevas notificaciones. Mantener al menos 3 versiones del modelo en `./models/` para rollback. La base de datos debe tener índices GIN sobre los vectores para consultas rápidas.
