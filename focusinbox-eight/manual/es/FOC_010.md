# focusinbox-eight - FOC_010

## Resumen

Sistema de procesamiento y enrutamiento inteligente de bandejas de entrada unificadas, diseñado para consolidar flujos de notificaciones, alertas y mensajes desde múltiples canales en un único punto de control con priorización dinámica y persistencia en PostgreSQL.

## Funcionalidades Técnicas Principales

### 1. Motor de Fusión y Desduplicación de Eventos en Tiempo Real

- **Parámetros clave:** `dedup_window_ms=500`, `max_batch_size=256`, `channel_weight_map` (JSONB en PostgreSQL).
- Implementa un algoritmo de ventana deslizante en memoria (Redis Streams + Lua scripting) que agrupa notificaciones idénticas o similares (según hash SHA-256 del payload) dentro de una ventana de 500 milisegundos.
- Reduce el volumen de alertas redundantes hasta en un 97% antes de la inserción en la tabla `inbox_events`, la cual cuenta con índices compuestos en `(source, hash, received_at)` para consultas de desduplicación inversa.
- Soporta pesos por canal (`critical=10`, `info=1`) para priorizar eventos en colas de procesamiento asíncrono.

### 2. Enrutador Semántico de Acciones con Clasificación por NLP Ligero

- **Parámetros clave:** `classifier_threshold=0.85`, `action_routing_table` (YAML externo recargable en caliente vía `SIGHUP`).
- Cada evento entrante es analizado por un modelo de clasificación TF-IDF + SVM (entrenado con 10,000 muestras etiquetadas) que asigna una etiqueta de acción (`responder`, `escalar`, `archivar`, `notificar_slack`).
- La tabla `routing_logs` registra cada decisión con precisión de microsegundos (`routed_at TIMESTAMPTZ(6)`), permitiendo auditoría completa.
- Integración con webhooks externos mediante colas RabbitMQ con confirmación manual (`manual_ack`) para garantizar entrega al menos una vez.

### 3. Panel de Control Analítico con Métricas de Latencia P99

- **Parámetros clave:** `latency_buckets_ms=[1, 5, 10, 50, 100, 500]`, `retention_days=90`.
- Exposición de métricas en formato OpenMetrics (`/metrics`) con histogramas de latencia de procesamiento desde la recepción hasta el enrutamiento final.
- Consultas agregadas en PostgreSQL usando window functions (`percentile_cont(0.99) WITHIN GROUP (ORDER BY processing_ms)`) para calcular P99 en tiempo real.
- Gráficos generados con Chart.js en el frontend, actualizados cada 10 segundos vía WebSocket (biblioteca `ws` nativa).

## Cómo Usar

### Paso 1: Configurar la Conexión a la Base de Datos y el Pool de Conexiones

Editar el archivo `config/database.yml` y definir los parámetros de conexión a PostgreSQL, incluyendo `pool_size=20`, `statement_timeout=5000ms` y `sslmode=require`. Ejecutar:

```bash
focusinbox-eight --db-init --schema-version=v2.1.0
```

Esto creará las tablas `inbox_events`, `routing_logs` y `channel_weights` con las restricciones CHECK correspondientes.

### Paso 2: Definir Reglas de Enrutamiento en el Archivo YAML

Crear `config/routing/rules.yaml` con la siguiente estructura:

```yaml
rules:
  - source: "slack"
    action: "notificar_slack"
    threshold: 0.85
    ttl_seconds: 3600
  - source: "email"
    action: "responder"
    model_path: "/models/classifier_v3.pkl"
```

Recargar reglas sin reiniciar el servicio enviando la señal `SIGHUP`:

```bash
kill -HUP $(pgrep focusinbox-eight)
```

### Paso 3: Iniciar el Servicio con Registro de Latencia Detallada

Ejecutar el binario con flags de depuración activados:

```bash
focusinbox-eight --log-level=debug --latency-log=/var/log/focusinbox/latency.ndjson
```

Verificar en tiempo real las métricas de P99 accediendo a `http://localhost:8080/metrics` y buscando la línea `focusinbox_latency_seconds{quantile="0.99"}`.

## Mejores Prácticas Recomendadas

### Práctica 1: Particionamiento Temporal de la Tabla `inbox_events`

Para mantener el rendimiento de las consultas de desduplicación, implementar un particionamiento por rango en la columna `received_at` usando herencia de tablas en PostgreSQL (por ejemplo, particiones mensuales). Esto reduce el escaneo de índices en un 60% y permite purgar datos antiguos sin bloqueos (`DROP TABLE IF EXISTS inbox_events_2023_01`).

### Práctica 2: Monitorización Proactiva de la Cola de RabbitMQ

Configurar alertas en Prometheus para cuando la profundidad de la cola `focusinbox.routing` supere los 10,000 mensajes no confirmados. Combinar con un umbral de latencia P99 > 200ms para activar un escalado horizontal automático del consumidor mediante Kubernetes HPA (Horizontal Pod Autoscaler), usando la métrica personalizada `rabbitmq_queue_messages_ready`.
