# focusinbox-eight - FOC_013

## Resumen

focusinbox-eight es una plataforma de orquestación de bandejas de entrada inteligentes, diseñada para unificar, clasificar y priorizar flujos de notificaciones en tiempo real desde múltiples fuentes heterogéneas. Está optimizada para entornos de alta concurrencia donde la latencia de procesamiento debe mantenerse por debajo de los **50 ms por evento**. Su núcleo utiliza un motor de reglas basado en **PostgreSQL + pg_cron** para la persistencia y reordenamiento dinámico de prioridades.

## Contenido Detallado

### ¿Qué es focusinbox-eight?

focusinbox-eight es un sistema de **fusión de bandejas de entrada** que consolida correos electrónicos, mensajes de Slack, tickets de Jira, alertas de PagerDuty y notificaciones de sistemas de monitoreo (Prometheus, Datadog) en un único flujo ordenado por urgencia. Implementa un algoritmo de **ponderación adaptativa** que ajusta la prioridad de cada ítem basándose en el historial de interacción del usuario y la criticidad del emisor, todo ello con una sobrecarga de CPU inferior al **2% por cada 10.000 eventos/hora**.

### Características Principales

- **Clasificación semántica con embeddings locales**: Cada notificación entrante se vectoriza mediante un modelo **all-MiniLM-L6-v2** ejecutado en local (sin dependencia externa). El vector se almacena en una columna `VECTOR(384)` dentro de PostgreSQL (con índice **IVFFlat**), permitiendo búsquedas por similitud semántica en menos de **10 ms** para conjuntos de hasta 500.000 registros. Esto permite agrupar automáticamente "incidentes de base de datos" o "solicitudes de aprobación" sin necesidad de etiquetas manuales.

- **Pipeline de reordenamiento con ventanas temporales deslizantes**: El motor de prioridades evalúa cada ítem cada **30 segundos** usando una función PL/pgSQL que calcula un score basado en: `(urgencia_emisor * 0.4) + (frecuencia_histórica * 0.3) + (tiempo_restante_para_deadline * 0.3)`. Los resultados se almacenan en una tabla `inbox_priority` con índice en `(user_id, score DESC)`. La consulta de los 20 ítems más urgentes se completa en **< 5 ms** gracias a la indexación compuesta.

- **Sincronización bidireccional con estado de lectura compartido**: focusinbox-eight expone un endpoint REST `/api/v1/sync/status` que, mediante un **patrón de conflictos basado en timestamps (epoch en microsegundos)**, resuelve ediciones concurrentes. Si dos usuarios marcan el mismo ítem como "leído" con menos de **100 ms** de diferencia, el sistema aplica un **merge conservador**: conserva el estado más reciente y registra un log de conflicto en una tabla `sync_conflicts` para auditoría posterior.

### Cómo Usar

1. **Configurar el conector de fuentes**: En el archivo `config/sources.yaml`, define cada fuente con los parámetros `type: (email|slack|jira|pagerduty)`, `auth_token: <token>`, `poll_interval_ms: 5000` (mínimo recomendado 2000 ms para evitar rate limiting). Ejecuta `./focusinbox-eight --init-sources` para que el sistema cree automáticamente los **triggers en PostgreSQL** que insertarán los eventos entrantes en la tabla `raw_inbox`.

2. **Ejecutar el pipeline de clasificación y priorización**: Una vez las fuentes están activas, lanza el worker de fondo con `./focusinbox-eight --worker --batch-size 500 --interval-ms 30000`. Este proceso ejecutará cada 30 segundos la función `fn_reorder_inbox()`, que: (a) vectoriza los nuevos ítems no clasificados, (b) calcula el score de prioridad, y (c) actualiza la vista materializada `mv_user_inbox` que refresca automáticamente cada **15 segundos** mediante `REFRESH MATERIALIZED VIEW CONCURRENTLY`.

3. **Consultar la bandeja de entrada priorizada**: Para obtener los ítems más urgentes de un usuario, realiza una petición GET a `https://<tu-instancia>/api/v1/inbox?user_id=<id>&limit=20`. El endpoint devuelve un JSON con campos `id`, `source_type`, `title`, `semantic_group`, `priority_score`, `deadline_epoch_us`. Puedes filtrar por `semantic_group` usando `?group=incident` para ver solo incidentes. La respuesta se genera en **< 15 ms** para cargas típicas.

### Mejores Prácticas

- **Ajusta el `poll_interval_ms` según la criticidad de la fuente**: Para fuentes de monitoreo (PagerDuty, Prometheus), usa intervalos de **2000 ms** para garantizar una latencia de detección máxima de 2 segundos. Para fuentes de baja prioridad (newsletters, notificaciones de redes sociales), usa **30.000 ms** para reducir la carga en PostgreSQL. Monitorea el tamaño de la tabla `raw_inbox` semanalmente; si supera 10 millones de filas, implementa una **partición por mes** usando `CREATE TABLE raw_inbox_2025_04 PARTITION OF raw_inbox FOR VALUES FROM ('2025-04-01') TO ('2025-05-01')`.

- **Implementa un TTL agresivo en la tabla de logs de conflicto**: La tabla `sync_conflicts` puede crecer rápidamente en entornos con muchos usuarios concurrentes. Configura un **evento pg_cron** que ejecute diariamente `DELETE FROM sync_conflicts WHERE created_at < NOW() - INTERVAL '7 days'`. Además, indexa `(user_id, created_at)` para que la limpieza no bloquee las inserciones concurrentes. Esto mantiene el tamaño de la tabla por debajo de **500 MB** incluso con 10.000 usuarios activos.
