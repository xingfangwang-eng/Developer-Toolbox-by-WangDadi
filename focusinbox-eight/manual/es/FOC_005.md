# focusinbox-eight - FOC_005

## Resumen

Sistema de priorización y consolidación de bandejas de entrada multi-cuenta con análisis de latencia en tiempo real para equipos distribuidos. Optimiza la fusión de flujos de notificaciones (Slack, Jira, correo) bajo un modelo de colas ponderadas con umbrales configurables en microsegundos.

## Contenido Detallado

### ¿Qué es focusinbox-eight?

Arquitectura de procesamiento asíncrono que agrega, filtra y ordena eventos entrantes desde múltiples fuentes usando un motor de reglas basado en prioridades dinámicas. Diseñado para reducir el ruido informativo en entornos de alta frecuencia (>10,000 eventos/hora), con persistencia en PostgreSQL y caché distribuida en Redis Cluster.

### Características Principales

- **Motor de fusión por colas de prioridad ponderada**: Cada fuente (Slack, Jira, correo SMTP/IMAP) se asigna a una cola independiente con parámetros `weight_source` (0.1–1.0) y `latency_tolerance_ms` (50–5000 ms). El planificador evalúa cada 100 ms el estado de todas las colas y extrae eventos según el ratio `urgency_score = (weight_source * 1000) / (current_latency_ms + 1)`. Soporta hasta 256 colas concurrentes con un throughput medido de 2,800 eventos/segundo en pruebas con hardware estándar (4 vCPU, 8 GB RAM).

- **Filtrado contextual con expresiones regulares precompiladas**: Permite definir reglas de inclusión/exclusión usando patrones `regex_match` contra los campos `subject`, `body` o `sender`. Las reglas se almacenan en una tabla `filter_rules` (PostgreSQL) con índices GIN para búsqueda rápida. Cada regla incluye `priority_boost` (entero, -10 a +10) y `max_events_per_minute` (entero, 1–1000). Se aplican en pipeline de 5 etapas con un tiempo de procesamiento promedio de 2.3 ms por evento.

- **Dashboard de telemetría en tiempo real con WebSocket**: Expone métricas vía endpoint `/api/v1/stream` con datos serializados en JSON binario (BSON). Incluye `queue_depth`, `avg_latency_ms`, `events_dropped` y `top_sources`. La interfaz se actualiza cada 200 ms usando Socket.IO con compresión deflate. Almacena histórico en tablas `metrics_hourly` y `metrics_daily` para análisis retrospectivo con consultas SQL parametrizadas.

### Cómo Usar

1. **Configuración inicial de fuentes y pesos**:  
   Editar el archivo `config/sources.yaml` con la estructura:  
   ```yaml
   sources:
     slack:
       type: slack_rtm
       weight_source: 0.8
       latency_tolerance_ms: 500
       filter_rules:
         - regex_match: "^(urgente|crítico)"
           priority_boost: 5
           max_events_per_minute: 50
     jira:
       type: jira_webhook
       weight_source: 0.6
       latency_tolerance_ms: 2000
   ```  
   Ejecutar `focusinbox-eight --validate-config` para verificar sintaxis y conectividad. El comando devuelve un informe detallado con tiempos de conexión a cada fuente.

2. **Inicio del procesador de colas**:  
   Lanzar el demonio con `focusinbox-eight --daemon --queue-workers 8 --redis-cluster "redis://10.0.0.1:6379,redis://10.0.0.2:6379"`. El parámetro `--queue-workers` define el número de goroutines (recomendado: número de núcleos * 2). Monitorear en `/var/log/focusinbox-eight/queue.log` los eventos `QUEUE_DISPATCH` con timestamp, cola origen y `urgency_score`. Si se detecta un pico de latencia > `latency_tolerance_ms`, se registra una alerta en `alerts.log` con nivel WARN.

3. **Consulta del estado y ajuste dinámico**:  
   Usar el endpoint REST `GET /api/v1/status` para obtener JSON con campos `total_events_processed`, `current_queue_depth` y `avg_latency_ms`. Para modificar pesos en caliente: `POST /api/v1/adjust` con body `{"source": "slack", "weight_source": 0.9}`. El cambio se aplica en menos de 50 ms y queda registrado en `audit.log` con timestamp y dirección IP del solicitante.

### Mejores Prácticas

- **Segmentar fuentes críticas en colas separadas**: Asignar `weight_source` alto (0.9–1.0) a canales de incidentes (PagerDuty, OpsGenie) y bajo (0.1–0.3) a notificaciones de CI/CD. Esto evita que eventos masivos de baja prioridad retrasen alertas de producción. Validar con `focusinbox-eight --simulate-load --events 5000 --sources 10` para ajustar umbrales antes del despliegue.

- **Implementar rotación de logs y métricas con retención configurable**: Usar `--log-retention-days 30` y `--metrics-retention-hours 168` para evitar saturación de disco. Los logs rotan automáticamente cuando superan 100 MB mediante compresión gzip. Las métricas agregadas se almacenan en tablas particionadas por fecha (`metrics_daily_2025_03`) con índices en `source` y `timestamp`, permitiendo consultas de más de 6 meses en menos de 200 ms.
