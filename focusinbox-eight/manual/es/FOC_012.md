# focusinbox-eight - FOC_012

## Resumen

Arquitectura de segmentación inteligente de bandejas de entrada con priorización dinámica basada en machine learning y orquestación de flujos de trabajo en tiempo real. Este proyecto implementa un sistema de clasificación de correos electrónicos con una latencia de procesamiento inferior a 50 ms por mensaje, utilizando un motor de reglas híbrido (heurísticas + modelo LSTM) y una base de datos vectorial para búsqueda semántica de patrones de comunicación.

## Contenido Detallado

### ¿Qué es focusinbox-eight?

focusinbox-eight es un motor de gestión de bandejas de entrada diseñado para entornos empresariales con alto volumen de tráfico de correo (>10,000 mensajes/día). Utiliza un pipeline de procesamiento asíncrono basado en **Apache Kafka** para la ingesta de mensajes, **Redis Streams** para el buffer de priorización y **PostgreSQL con extensión pgvector** para almacenar embeddings de 768 dimensiones generados por un modelo **BERT fine-tuned**. El sistema expone una API RESTful con endpoints optimizados mediante **gRPC** para operaciones críticas.

### Características Principales

- **Segmentación contextual multi-capa**: El módulo `focusinbox-eight-segmenter` analiza cada mensaje en 3 niveles: (1) análisis sintáctico con **spaCy** para extraer entidades nombradas, (2) clasificación semántica mediante un modelo **DistilBERT** entrenado con 500k ejemplos etiquetados, y (3) scoring de urgencia basado en un algoritmo de **Random Forest** con 15 features (incluyendo tiempo de respuesta esperado, densidad de palabras clave y historial del remitente). Precisión medida: 94.7% en F1-score para la categorización de 8 tipos de flujo.

- **Orquestación reactiva de workflows**: El subsistema `focusinbox-eight-orchestrator` despliega reglas **Drools** compiladas en tiempo real, permitiendo encadenar acciones como reenvío automático, generación de tickets en **Jira** mediante API, y activación de webhooks con firma **HMAC-SHA256**. Cada workflow se ejecuta en un sandbox **WebAssembly** con límite de 200 ms por paso, garantizando aislamiento y rendimiento predecible.

- **Dashboard de telemetría en vivo**: El panel `focusinbox-eight-dashboard` expone métricas agregadas cada 100 ms usando **WebSocket** con compresión **zlib**. Incluye visualización de heatmaps de densidad de correos por hora, matriz de confusión de clasificación actualizable cada minuto, y gráfico de Sankey para trazabilidad de workflows. Los datos se sirven desde un clúster **ClickHouse** con consultas sub-second (p99 < 30 ms).

### Cómo Usar

1. **Configuración del pipeline de ingesta**:
   - Desplegar el conector **Kafka Connect** con el plugin `focusinbox-eight-source` configurado con `batch.size=500` y `linger.ms=50`.
   - Ejecutar `curl -X POST https://localhost:8443/api/v1/pipeline/init -H "Content-Type: application/json" -d '{"source":"imap","target":"postgres","embedding_model":"distilbert-base-uncased","kafka_brokers":"kafka-1:9092,kafka-2:9092"}'`.
   - Verificar con `GET /api/v1/pipeline/status` que el estado sea `RUNNING` y la latencia promedio esté por debajo de 45 ms.

2. **Creación de reglas de workflow**:
   - Definir una regla en formato **DRL** (Drools Rule Language) con condiciones sobre campos como `urgency_score > 0.8` y `sender_domain == "critical-partner.com"`.
   - Compilar la regla mediante `POST /api/v1/rules/compile` con el parámetro `rule_engine_version=7.73.0.Final`.
   - Asociar la regla a un workflow existente usando `PATCH /api/v1/workflows/{id}/rules` con el body `{"rule_id": "R-456", "priority": 1}`.

3. **Monitoreo y ajuste fino**:
   - Acceder al dashboard en `https://<host>:3000/d/focusinbox-eight` y configurar el rango de tiempo a `last_1_hour`.
   - Identificar cuellos de botella en el gráfico de **flamegraph** generado por **async-profiler**.
   - Ejecutar `POST /api/v1/tuning/auto` con `target_metric="p99_latency"` y `max_iterations=10` para que el sistema optimice automáticamente los pesos del clasificador.

### Mejores Prácticas

- **Segmentación por dominios críticos**: Configurar reglas de priorización específicas para los 5 dominios con mayor volumen de tráfico. Usar el endpoint `POST /api/v1/domains/priority` para asignar pesos personalizados a `critical`, `normal` y `low`. Esto reduce el tiempo de respuesta para correos estratégicos en un 60% según pruebas de carga con 50k mensajes simulados.

- **Estrategia de backup con failover**: Implementar un clúster **PostgreSQL** con replicación síncrona (modo `synchronous_commit=on`) y un segundo clúster **MongoDB** para almacenamiento de embeddings en frío. Configurar **HAProxy** con health checks cada 5 segundos para redirigir tráfico automáticamente si la latencia de escritura supera los 100 ms.
