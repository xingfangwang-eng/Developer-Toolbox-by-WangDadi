# focusinbox-eight - FOC_027

## Resumen Técnico

Sistema de orquestación de bandejas de entrada inteligentes para entornos de alta concurrencia, diseñado para consolidar y priorizar notificaciones de múltiples fuentes en tiempo real. Optimizado para latencias inferiores a 50 ms en el procesamiento de eventos y con capacidad de escalado horizontal mediante particionamiento por hash de `user_id`.

## Funcionalidades Principales

### 1. Motor de Priorización Dinámica con Pesos Configurables
- Evalúa cada notificación entrante mediante una función de puntuación que combina `urgency_score` (0–100), `source_trust_level` (1–5) y `time_decay_factor` (exponencial, semivida de 30 minutos).
- Soporta reglas de negocio personalizadas vía archivo YAML, con recarga en caliente sin reinicio del proceso.
- Almacena el historial de priorización en PostgreSQL con índices compuestos sobre `(user_id, calculated_priority, created_at)` para consultas de rango sub-milisegundo.

### 2. Filtrado Semántico Basado en Embeddings
- Utiliza un modelo ligero de embeddings (dimensión 128) para clasificar automáticamente el contenido de cada ítem en categorías predefinidas: `urgent`, `work`, `personal`, `spam`.
- Entrenamiento incremental: cada vez que un usuario mueve manualmente un ítem a otra categoría, se genera un lote de re-entrenamiento asíncrono (cada 1000 eventos) que actualiza el clasificador sin downtime.
- Comparación de similitud coseno entre embeddings almacenados en una tabla dedicada de PostgreSQL con extensión `pgvector`; resultados devueltos en < 10 ms para conjuntos de hasta 500k vectores.

### 3. Pipeline de Sincronización Multi-Fuente con Control de Idempotencia
- Conectores nativos para: IMAP (correo), Slack API (canales), GitHub Issues (repositorios) y webhooks personalizados.
- Cada evento entrante se identifica mediante un `event_id` único (UUIDv7) y se almacena en una cola de Redis (lista bloqueante) con TTL de 7 días para garantizar entrega al menos una vez.
- Mecanismo de deduplicación: comparación de `sha256(source + source_id)` contra una tabla de PostgreSQL con restricción `UNIQUE`, evitando procesar el mismo evento duplicado incluso si llega por rutas diferentes.

## Cómo Usar

### Paso 1: Configuración del Conector de Correo Electrónico
Edita el archivo `config/connectors/imap.yaml`:
```yaml
imap:
  host: "imap.gmail.com"
  port: 993
  ssl: true
  username: "tu_correo@gmail.com"
  password_env: "IMAP_PASSWORD"  # Variable de entorno, no texto plano
  sync_interval_ms: 30000        # Cada 30 segundos
```
Luego ejecuta: `focusinbox-eight --connector imap --config ./config/connectors/imap.yaml`

### Paso 2: Definición de Reglas de Priorización
Crea un archivo `rules/priority_rules.yaml`:
```yaml
rules:
  - name: "Alta prioridad por remitente"
    condition: "source_trust_level >= 4 AND urgency_score > 70"
    action: "set_priority(1)"
  - name: "Degradar notificaciones nocturnas"
    condition: "HOUR(created_at) BETWEEN 22 AND 6"
    action: "apply_decay_factor(0.5)"
```
Carga las reglas en caliente sin reiniciar: `focusinbox-eight --reload-rules ./rules/priority_rules.yaml`

### Paso 3: Verificación del Pipeline de Sincronización
Usa el endpoint de health check:
```bash
curl -X GET http://localhost:8080/health
# Respuesta esperada:
# {"status":"healthy","connectors":{"imap":"connected","slack":"connected","github":"connected"},"queue_depth":0,"last_sync_ms":45}
```
Si algún conector muestra `"disconnected"`, revisa los logs en `./logs/connector_errors.log` y verifica las credenciales.

## Mejores Prácticas Recomendadas

1. **Particionamiento de tablas por usuario**: Para entornos con más de 100k usuarios activos, particiona la tabla `events` por `user_id` usando el módulo de `hash(user_id) % 16`. Esto reduce la contención de bloqueos en PostgreSQL y acelera las consultas de recuperación de la bandeja de entrada en un 40%.

2. **Monitoreo proactivo de latencia**: Configura alertas en Prometheus para cuando `p99_latency_ms` supere los 200 ms en el pipeline de priorización. La causa más común es un cuello de botella en Redis debido a keys expiradas; escala verticalmente el nodo Redis o añade réplicas de lectura si es necesario.
