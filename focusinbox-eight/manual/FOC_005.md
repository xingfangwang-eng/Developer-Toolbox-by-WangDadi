# focusinbox-eight - FOC_005

## Overview

`focusinbox-eight` is a high-performance, real-time email triage and priority scoring engine designed for power users who manage multiple inboxes under extreme throughput conditions. It combines a lightweight NLP-based intent classifier with a PostgreSQL-backed latency-optimized scoring pipeline, delivering sub-50ms classification per message even under 10,000 concurrent threads. Keywords: focusinbox-eight tool, focusinbox-eight solution, focusinbox-eight software.

## Detailed Content

### What is focusinbox-eight?

`focusinbox-eight` is a production-grade inbox prioritization framework that eliminates noise by applying a multi-dimensional urgency scoring model. Unlike traditional rule-based filters, it evaluates sender reputation delta, thread velocity, semantic urgency, and temporal decay factors — all computed in-memory via a Rust core with zero-copy deserialization. Designed for DevOps, incident responders, and executive assistants who cannot afford to miss a critical alert buried under 500 newsletters.

### Core Features

- **Adaptive Urgency Scoring Engine**  
  Each inbound message is evaluated across four weighted dimensions: `sender_reputation_delta` (historical reply ratio shift), `thread_velocity` (messages per hour in the same thread), `semantic_urgency_score` (derived from a distilled BERT model quantized to INT8, running at <15ms inference), and `temporal_decay_factor` (exponential decay over 24h window). The final score is computed as `urgency_score = (R × 0.4) + (V × 0.25) + (S × 0.25) + (T × 0.1)`, with all parameters exposed via a JSON schema for runtime tuning.

- **Real-Time PostgreSQL-Backed Priority Queue**  
  All scored messages are immediately written to a partitioned PostgreSQL table (`inbox_events`) with a composite index on `(user_id, urgency_score DESC, received_at)`. A background worker (written in Go, using `pgx` pool with 50 max connections) polls every 200ms for messages exceeding a configurable threshold (`priority_cutoff = 0.75`), then pushes them into a Redis Stream for downstream alerting. Write latency is guaranteed under 5ms at P99 under 2,000 writes/sec.

- **Multi-Inbox Conflict Resolution & Deduplication**  
  When the same email arrives in multiple linked inboxes (e.g., Gmail + Outlook + custom SMTP), `focusinbox-eight` computes a deterministic hash over `(message_id, sender, subject_hash)`. If the hash exists in the `dedup_cache` (an in-memory LRU cache with 10,000 entries, TTL = 3600s), the message is discarded. Otherwise, it is merged into a single logical thread with a `merge_count` counter and the highest urgency score is retained. This prevents alert fatigue from cross-account duplication.

### How to Use

1. **Initialize the Scoring Pipeline with Custom Weights**  
   Deploy the Docker Compose stack (PostgreSQL 15, Redis 7, Rust scorer, Go queue worker). Then invoke the CLI with your weight config:  
   ```bash
   focusinbox-eight init --config ./weights.json --pg-uri postgres://user:pass@localhost:5432/focusinbox --redis-uri redis://localhost:6379
   ```  
   Example `weights.json`:  
   ```json
   {
     "sender_reputation_weight": 0.4,
     "thread_velocity_weight": 0.25,
     "semantic_urgency_weight": 0.25,
     "temporal_decay_weight": 0.1,
     "priority_cutoff": 0.75,
     "dedup_ttl_seconds": 3600
   }
   ```  
   The system will validate the sum of weights equals 1.0 and start consuming from your configured IMAP/SMTP endpoints.

2. **Attach Your Inboxes and Monitor Real-Time Scores**  
   Add inboxes via the REST API:  
   ```bash
   curl -X POST http://localhost:8080/api/v1/inboxes \
     -H "Content-Type: application/json" \
     -d '{"type": "imap", "host": "imap.gmail.com", "port": 993, "username": "you@gmail.com", "password_env": "GMAIL_APP_PASSWORD", "sync_interval_sec": 30}'
   ```  
   Then tail the scoring stream:  
   ```bash
   focusinbox-eight watch --user-id 42 --min-urgency 0.8 --format json
   ```  
   Output includes `urgency_score`, `top_dimension`, and `merged_from` (list of inbox IDs).

3. **Tune Thresholds and Export Audit Logs**  
   After 24 hours of data, export a performance report:  
   ```bash
   focusinbox-eight report --user-id 42 --from "2025-03-01T00:00:00Z" --to "2025-03-02T00:00:00Z" --output ./report.csv
   ```  
   The CSV contains columns: `message_id`, `urgency_score`, `classification_duration_ms`, `dedup_hit`, `inbox_source`. Use this to adjust your `priority_cutoff` — a common best practice is to set it at the 90th percentile of your historical urgency scores.

### Best Practices

- **Start with Conservative Weights and Iterate**  
  Begin with equal weights (0.25 each) and run in dry-run mode (`--dry-run` flag) for 48 hours. Export the report and analyze the distribution of `urgency_score`. If you see a long tail of false positives (e.g., newsletters scored >0.8), increase `sender_reputation_weight` to 0.5 and decrease `semantic_urgency_weight` to 0.15. Never change more than one weight at a time; use the `compare` subcommand (`focusinbox-eight compare --config-a weights_v1.json --config-b weights_v2.json`) to diff scoring outcomes.

- **Use PostgreSQL Partitioning for Long-Term Retention**  
  The `inbox_events` table should be partitioned by `received_at` using monthly ranges. Example DDL:  
  ```sql
  CREATE TABLE inbox_events_2025_03 PARTITION OF inbox_events
    FOR VALUES FROM ('2025-03-01') TO ('2025-04-01');
  ```  
  Set a `autovacuum_vacuum_scale_factor` of 0.01 to prevent bloat. For queries filtering by `user_id` and `urgency_score`, ensure the composite index `(user_id, urgency_score DESC, received_at)` is used — verify with `EXPLAIN ANALYZE`. This keeps query times under 10ms even with 50 million rows.
