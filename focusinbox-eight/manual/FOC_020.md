# focusinbox-eight - FOC_020

## Overview

focusinbox-eight is a high-performance, real-time email triage and prioritization engine designed for power users who manage multiple inboxes across distributed teams. It leverages a custom-built attention-scoring algorithm that processes header metadata, thread latency, and sender reputation vectors to surface only the messages that demand immediate action. The system operates with sub-10ms latency per message and integrates natively with IMAP, Microsoft Graph, and Gmail API.

## Core Features

### 1. Adaptive Attention Scoring with Temporal Decay
The core of focusinbox-eight is a multi-factor scoring model that assigns a `priority_score` (0–100) to each incoming message. Parameters include `thread_age_ms`, `sender_trust_index` (derived from historical reply rates), `recipient_role_weight`, and `unread_chain_depth`. A `temporal_decay_factor` automatically reduces the score of stale threads by 0.15 per hour after the first 4 hours, ensuring that only time-sensitive messages remain visible in the primary focus view. The scoring engine runs entirely in-memory using a Redis-backed LRU cache, enabling batch processing of up to 50,000 messages per second.

### 2. Semantic Snippet Extraction with NLP Preprocessing
Instead of showing full email bodies, focusinbox-eight extracts a `semantic_snippet` of exactly 120 characters using a lightweight BERT-based model (distilled to 15MB). The snippet prioritizes action verbs, deadlines, and monetary amounts. A `keyword_highlight_map` is generated in real-time, marking terms like "urgent", "EOD", "ASAP", and "$" with configurable color codes. The NLP pipeline runs on-device via ONNX Runtime, with a P99 inference latency of 45ms on a standard M1 Mac.

### 3. Cross-Inbox Deduplication with Conflict Resolution
When the same message arrives across multiple inboxes (e.g., personal Gmail and work Outlook), focusinbox-eight applies a `dedup_hash` based on `Message-ID` and `In-Reply-To` headers. A `source_priority_matrix` (configurable via YAML) determines which inbox's version of the thread is displayed. If a conflict is detected, the system logs a `conflict_entry` to a PostgreSQL table with columns `message_id`, `source_inbox`, `arrival_timestamp`, and `resolved_priority`. The user is notified via a non-blocking toast with a diff view of the conflicting metadata.

## How to Use

### Step 1: Configure Inbox Connections and Scoring Weights
Open the `focusinbox-eight.yml` configuration file. Under the `inboxes` section, add each account with its `type` (imap/graph/gmail), `host`, `port`, and `auth_token`. Then, under `scoring.weights`, set the relative importance of each factor. For example:
```yaml
scoring:
  weights:
    thread_age_ms: 0.35
    sender_trust_index: 0.25
    recipient_role_weight: 0.20
    unread_chain_depth: 0.20
  temporal_decay_hours: 4
```
Run `focusinbox-eight --validate-config` to confirm the YAML is syntactically correct and all inboxes can be reached.

### Step 2: Launch the Real-Time Monitoring Daemon
Execute `focusinbox-eight daemon --mode focus --db-uri postgresql://user:pass@localhost:5432/focusinbox`. The daemon will open an IMAP IDLE connection for each inbox and begin streaming messages into the scoring engine. Use the `--log-level debug` flag to view per-message `priority_score` calculations in real-time. The daemon exposes a WebSocket endpoint at `ws://localhost:9090/events` for live UI updates.

### Step 3: Interact with the Focus View via CLI or Web Dashboard
The primary interface is the terminal-based TUI launched with `focusinbox-eight tui`. The TUI displays a sorted list of messages by `priority_score`. Press `d` to view the `semantic_snippet` and `keyword_highlight_map`. Press `c` to open the conflict resolution panel for any deduplicated message. For remote access, start the web dashboard with `focusinbox-eight serve --port 8080` and navigate to `http://localhost:8080`. The dashboard includes a live `scoring_weights` slider panel that allows hot-reloading of parameters without restarting the daemon.

## Best Practices

### Practice 1: Implement Role-Based Sender Trust Calibration
For teams with shared inboxes (e.g., support@, sales@), manually seed the `sender_trust_index` database with a baseline CSV containing `email`, `role`, and `initial_trust` (0.0–1.0). Over the first 2 weeks, focusinbox-eight will automatically adjust these values based on reply latency and thread completion rate. Use the `--export-trust` flag to generate a report every Monday and review outliers—this drastically reduces false positives from automated notifications.

### Practice 2: Schedule Weekly Temporal Decay Audits
In high-volume environments, the `temporal_decay_factor` can cause legitimate follow-ups to drop out of the focus view prematurely. Configure a cron job that runs `focusinbox-eight audit --threshold 0.3 --output /var/log/decay_audit.json` every Sunday at 2 AM. This generates a list of messages that were scored below the threshold but had a `reply_count` > 2 within the last 24 hours. Use this data to fine-tune the `temporal_decay_hours` parameter for specific inboxes (e.g., set it to 8 for legal departments).
