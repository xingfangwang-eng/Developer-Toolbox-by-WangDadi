# focusinbox-eight - FOC_025

## Overview

focusinbox-eight is a high‑performance, latency‑optimized email threading and triage engine designed for power users who manage 500+ messages daily across multiple inboxes. It leverages a custom token‑aware attention mechanism to classify, cluster, and prioritize emails with sub‑5ms inference time per message on commodity hardware.

## Core Features

### 1. Real‑Time Temporal Thread Reconstruction with Conflict Detection
- **Parameter:** `--window-ms 300000` (5‑minute sliding window)  
- **Technical detail:** Uses a DAG‑based merge algorithm that resolves email threads by comparing `Message-ID`, `In-Reply-To`, and `References` headers against a local SQLite cache. Detects and flags fork conflicts (e.g., two replies to the same parent with differing subjects) and outputs a `conflict_score` between 0.0 and 1.0.  
- **Performance:** Handles 10,000 concurrent threads with <2ms overhead per new message.

### 2. Adaptive Priority Scoring via Bayesian Personalized Ranking (BPR)
- **Parameter:** `--bpr-learning-rate 0.01`, `--bpr-epochs 50`  
- **Technical detail:** Trains a per‑user model on implicit feedback (open, reply, archive, delete). Features include sender domain reputation (from a pre‑compiled `reputation.db`), recency (decay factor λ = 0.85), and attachment type. The model outputs a `priority` integer (1–5) and a `confidence` float (0–1).  
- **Storage:** PostgreSQL with `pgvector` extension for storing user embeddings; supports incremental retraining every 24 hours.

### 3. Automated Snippet Generation with Token‑Aware Summarization
- **Parameter:** `--max-snippet-tokens 128`, `--model-path ./models/t5-small-email`  
- **Technical detail:** Runs a distilled T5 model (ONNX‑optimized, 40MB) that extracts the top 3 action items from each email body. Uses `spaCy` for sentence segmentation and a custom `heuristic_filter` to remove signatures, disclaimers, and quoted text. Outputs a `snippet` string with a `relevance` score (0–100).  
- **Latency:** Average 12ms per snippet on CPU (Intel i7‑12700).

## How to Use

### Step 1: Initialize the Local Data Store and Import Mailboxes
```bash
focusinbox-eight init --store sqlite:///inbox.db --import-imap "imaps://user@domain.com:993/INBOX"
```
- The tool will crawl the last 90 days of email, building the thread DAG and populating the `reputation.db` with sender domain data.  
- **Verify:** Check `inbox.db` for `threads` table with `conflict_score` column.

### Step 2: Train the BPR Priority Model
```bash
focusinbox-eight train --bpr-epochs 50 --bpr-learning-rate 0.01 --output ./models/user_weights.pkl
```
- Uses all user interactions (opens, replies, archivals) from the local store.  
- **Monitor:** Training logs output `avg_auc` per epoch; target AUC >0.85 for production readiness.

### Step 3: Run the Real‑Time Triage Daemon
```bash
focusinbox-eight daemon --port 8080 --window-ms 300000 --max-snippet-tokens 128
```
- Listens for new emails via IMAP IDLE or webhook (POST `/incoming`).  
- **Output:** JSON stream on port 8080 with fields: `thread_id`, `priority`, `conflict_score`, `snippet`, `relevance`.  
- **Test:** Send a test email with `curl -X POST -d '{"subject":"Urgent: Q3 Review","body":"Please confirm by Friday"}' http://localhost:8080/incoming`.

## Best Practices

### Practice 1: Integrate with a CRM via Webhook for Automatic Task Creation
- Configure focusinbox‑eight to POST high‑priority emails (priority ≥ 4, confidence ≥ 0.9) to your CRM’s API endpoint. Use the `snippet` field as the task description. This reduces manual triage time by ~40% for sales and support teams.

### Practice 2: Schedule Weekly Model Retraining with A/B Comparison
- Run `focusinbox-eight train` every Monday at 02:00 UTC, but keep the previous model’s weights. Compare `avg_auc` on the following Sunday’s interactions. Only deploy the new model if AUC improves by >0.02. This prevents regressions due to seasonal email patterns.
