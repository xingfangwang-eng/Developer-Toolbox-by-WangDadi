# focusinbox-eight - FOC_017

## Overview

focusinbox-eight is a high-performance, multi-tenant inbox processing engine designed for real-time email triage, semantic classification, and automated response orchestration. It ingests raw mail streams, applies a custom NLP pipeline with sub-millisecond latency per message, and exposes a deterministic API for downstream workflow integration.

## Detailed Content

### What is focusinbox-eight?

focusinbox-eight is a production-grade email routing and classification system that eliminates inbox noise by assigning each incoming message a priority score (`p_score`, range 0.0–1.0) and a contextual tag (`c_tag`, e.g., `urgent_invoice`, `support_tier2`, `auto_reply_candidate`). It operates as a standalone daemon with a gRPC interface and can be embedded into existing ticketing or CRM pipelines via its native PostgreSQL connector.

### Core Features

- **Semantic Priority Scoring Engine**  
  Each message is analyzed through a three-stage pipeline: tokenization, feature extraction (using a distilled BERT model quantized to INT8), and heuristic weighting. The final `p_score` is computed in under 800 microseconds on a single CPU core. The engine supports configurable decay curves for repeated senders, ensuring that spam or low-value threads are automatically deprioritized.

- **Contextual Tagging with Multi-Label Support**  
  The system assigns up to 5 `c_tag` values per message using a hierarchical taxonomy of 128 predefined labels. Tags are stored as a `TEXT[]` array in PostgreSQL, enabling efficient GIN-indexed queries. The tagging model is fine-tuned on 2.7 million real-world support tickets and achieves a macro F1 score of 0.94 on holdout data.

- **Deterministic Auto-Reply Generation**  
  For messages classified as `auto_reply_candidate`, the engine generates a response template using a lightweight transformer (6 layers, 512 hidden units) that runs on-device. The reply is constrained to a maximum of 512 tokens and includes a `confidence_score` threshold (default: 0.85) below which the message is escalated to a human agent. All generated replies are logged to the `focusinbox_eight.reply_log` table with a `latency_ms` column for performance monitoring.

### How to Use

1. **Initialize the Database Schema**  
   Run the included migration script against your PostgreSQL instance (version 14+ required). This creates the `focusinbox_eight.inbox` table with columns `message_id UUID`, `raw_body TEXT`, `p_score FLOAT4`, `c_tags TEXT[]`, and `processed_at TIMESTAMPTZ`. Use the `SET search_path TO focusinbox_eight;` command to isolate the schema.

   ```sql
   psql -U admin -d inboxdb -f migrations/001_create_inbox.sql
   ```

2. **Start the Ingest Daemon**  
   Launch the daemon with a configuration file specifying the IMAP mailbox and PostgreSQL connection string. Example invocation:

   ```bash
   ./focusinbox-eight --config /etc/focusinbox/config.toml
   ```

   The `config.toml` must contain `[imap]` and `[database]` sections. The daemon polls the IMAP folder every 30 seconds by default (configurable via `poll_interval_secs`). All processed messages are immediately written to the `inbox` table.

3. **Query for High-Priority Items and Trigger Replies**  
   Use the exposed gRPC method `GetActionableMessages` with a `min_p_score` parameter (e.g., 0.7) and an optional `c_tag` filter. The response includes each message’s `auto_reply_candidate` flag. If the flag is `true` and `confidence_score >= 0.85`, call the `GenerateReply` method to obtain the response text. Example using `grpcurl`:

   ```bash
   grpcurl -plaintext -d '{"min_p_score": 0.7, "c_tag": "urgent_invoice"}' \
     localhost:50051 focusinboxeight.InboxService/GetActionableMessages
   ```

### Best Practices

- **Tune the Priority Decay Curve per Tenant**  
  For multi-tenant deployments, adjust the `decay_rate` parameter in the per-tenant configuration. A rate of `0.95` (default) works well for general support; for sales-heavy inboxes, use `0.99` to avoid deprioritizing follow-up sequences. Monitor the `latency_ms` histogram in the daemon’s Prometheus metrics endpoint (`/metrics`) to ensure the decay calculation does not exceed 50 microseconds per message.

- **Archive Processed Messages with a Retention Policy**  
  Implement a daily cron job that moves rows older than 90 days from `focusinbox_eight.inbox` to a partitioned archive table (`inbox_archive`). Use PostgreSQL’s `pg_cron` extension with a query like `DELETE FROM focusinbox_eight.inbox WHERE processed_at < NOW() - INTERVAL '90 days'`. This keeps the active table under 10 million rows, ensuring sub-10ms queries on the GIN-indexed `c_tags` column.
