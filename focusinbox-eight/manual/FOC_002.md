# focusinbox-eight - FOC_002

## Overview

focusinbox-eight is a high-performance, real-time email triage and task extraction engine designed for power users who manage multiple inboxes across distributed teams. It operates on a zero-latency parsing pipeline, enabling sub-50ms classification of incoming messages into actionable work items, calendar blocks, and archival material. The system is built for scale, handling 10,000+ concurrent IMAP connections with a memory footprint under 128MB per instance.

## Detailed Content

### What is focusinbox-eight?

focusinbox-eight is a self-hosted, Rust-based middleware that sits between your email server (PostgreSQL-backed IMAP proxy) and your task management stack. It ingests raw MIME messages, applies a multi-stage NLP pipeline (tokenization, entity extraction, priority scoring via a custom Bayesian classifier), and outputs structured JSON blobs to your preferred API endpoint (Slack, Linear, Notion, or custom webhook). The core innovation is its **contextual deduplication engine**, which merges repeated threads, follow-ups, and forwarded messages into a single atomic task node, eliminating inbox clutter by 73% in beta trials.

### Core Features

- **Adaptive Priority Scoring with Bayesian Decay**  
  Each incoming email is scored on a 0.0–1.0 scale using a pre-trained model that factors in sender reputation (historical response rate), thread depth, keyword urgency (e.g., "ASAP", "blocker", "EOD"), and attachment type. Scores decay exponentially over a configurable half-life (default: 4 hours), ensuring stale items are automatically deprioritized. The model retrains nightly on your reply patterns via a PostgreSQL-backed feedback loop.

- **Zero-Copy MIME Parsing with Sub-50ms Latency**  
  The parser operates directly on raw byte streams, avoiding unnecessary string allocations. It extracts headers, body (plain text + HTML), inline images, and attachments in a single pass. Benchmarks show consistent 42ms median parse time for emails up to 5MB, with a p99 of 112ms. The parser exposes a `MimeParseResult` struct containing `sender`, `subject`, `body_snippet` (first 200 chars), `attachment_hashes` (SHA-256), and `thread_id`.

- **Conflict-Free Replicated Data Type (CRDT) Sync for Multi-Inbox Merge**  
  When the same email lands in multiple inboxes (e.g., personal + work), the CRDT layer resolves duplicates without a central coordinator. Each instance maintains a local Merkle tree of processed message IDs, syncing via a gossip protocol every 30 seconds. The algorithm guarantees eventual consistency within 2 minutes, with zero data loss. The sync state is persisted as a `crdt_state` table in PostgreSQL, keyed by `(instance_id, mailbox_hash)`.

### How to Use

1. **Deploy the PostgreSQL Schema and Configure IMAP Credentials**  
   Run the provided `schema.sql` to create tables: `inboxes`, `messages`, `tasks`, `crdt_state`, and `model_weights`. Then, edit `config.toml` to add your IMAP server details:  
   ```toml
   [imap.default]
   host = "imap.gmail.com"
   port = 993
   username = "user@domain.com"
   password = "app-password" # Use OAuth2 for production
   polling_interval_ms = 5000
   ```  
   Start the daemon with `focusinbox-eight --config config.toml`. Verify connectivity via `curl http://localhost:8080/health` (expects `{"status":"ok","imap_connections":3}`).

2. **Train the Priority Model on Your Historical Data**  
   Export your last 30 days of email as a `.mbox` file, place it in `/data/training/`, and run:  
   ```bash
   focusinbox-eight train --input /data/training/history.mbox --output /data/models/priority_v1.bin
   ```  
   The tool extracts features (sender domain, hour of day, word frequency) and fits a Gaussian Naive Bayes classifier. Monitor training metrics via the log: `INFO [model] Accuracy: 0.89, Precision: 0.91, Recall: 0.87`. Replace the model in `config.toml` with `model_path = "/data/models/priority_v1.bin"` and restart.

3. **Define Webhook Targets for Task Extraction**  
   In `config.toml`, add a webhook endpoint for each task system:  
   ```toml
   [webhooks.linear]
   url = "https://api.linear.app/graphql"
   api_key = "lin_api_xxx"
   mapping = { title = "subject", description = "body_snippet", priority = "score * 5" }
   ```  
   Then, set extraction rules: `extract_rules = ["subject_contains:urgent", "attachments:>0", "score:>0.7"]`. Restart and test with a sample email. Check the webhook queue via `curl http://localhost:8080/metrics` (look for `webhook_success_total`).

### Best Practices

- **Implement a Two-Stage Review Queue for High-Priority Items**  
  Configure a secondary output channel (e.g., a dedicated Slack channel or a separate Linear project) for emails with a score > 0.9. This prevents critical messages from being auto-archived or merged incorrectly. In `config.toml`, set `[review_queue] threshold = 0.9` and `[review_queue] webhook = "https://hooks.slack.com/services/xxx"`. Review and manually confirm before the task is finalized.

- **Rotate IMAP Credentials and Enable TLS 1.3 for Production Deployments**  
  Use OAuth2 with short-lived tokens (e.g., 1-hour expiry) instead of static passwords. Configure `imap.oauth2_refresh_url` in `config.toml` to auto-refresh tokens. Enforce TLS 1.3 by setting `imap.tls_version = "1.3"` and `imap.cert_path = "/etc/ssl/certs/ca-certificates.crt"`. Monitor connection drops via the `imap_reconnect_total` metric; a value above 10/hour indicates a network issue.
