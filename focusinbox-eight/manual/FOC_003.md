## Overview

## Detailed Content

### What is focusinbox-eight?

focusinbox-eight is a distributed, event-driven system that ingests raw email streams via IMAP IDLE or webhook, applies a multi-stage NLP pipeline (tokenization → intent classification → priority scoring), and outputs structured task objects into a PostgreSQL-backed queue. It achieves sub-200ms processing per email at 10,000+ concurrent connections, with automatic deduplication and attachment-to-text conversion using Tesseract OCR.

### Core Features

- **Adaptive Priority Engine with Temporal Decay Scoring**  
  Each incoming email is parsed through a custom `PriorityScorer` module that evaluates: `sender_reputation_score` (0.0–1.0, based on historical reply rate), `urgency_keyword_density` (regex-matched from a 500-term dictionary), and `thread_depth` (integer; emails in threads >5 get +0.3 weight). The final score decays exponentially over 48 hours using a configurable `lambda` parameter (default: 0.02). Results are stored in a `priority_scores` column with millisecond-precision timestamps.

- **Multi-Model Intent Classification Pipeline**  
  A three-model ensemble runs in parallel: (1) a fine-tuned BERT-base for 12 predefined intents (e.g., `meeting_request`, `invoice_approval`), (2) a lightweight FastText fallback for low-confidence predictions (<0.75), and (3) a regex-based rule engine for deterministic patterns (e.g., `[A-Z]{2,4}-\d{3,6}` for ticket IDs). The final intent is selected via a weighted voting mechanism (`bert_weight=0.6`, `fasttext_weight=0.3`, `regex_weight=0.1`). All model inference runs on a dedicated GPU node with CUDA 11.8.

- **PostgreSQL-Backed Task Queue with Exactly-Once Delivery**  
  Extracted tasks are inserted into a `focus_tasks` table with fields: `task_id` (UUID v7), `email_message_id` (indexed), `extracted_due_date` (timestamptz), `assigned_owner` (nullable, resolved from email signature via NER), and `status` enum (`pending`, `processing`, `completed`, `failed`). The queue uses PostgreSQL’s `SKIP LOCKED` for worker contention, with a configurable `max_retries=3` and exponential backoff (`base_delay=1s`). A background `VACUUM` job runs every 5 minutes to reclaim dead tuples.

### How to Use

1. **Deploy the IMAP Listener with OAuth2 Credentials**  
   Start the `focusinbox-eight` daemon with the following environment variables:  
   `IMAP_HOST=imap.gmail.com`, `IMAP_PORT=993`, `OAUTH2_CLIENT_ID=<your_client_id>`, `OAUTH2_CLIENT_SECRET=<your_secret>`, `OAUTH2_REFRESH_TOKEN=<token>`.  
   Run `./focusinbox-eight --config /etc/focusinbox/config.yaml`. The daemon will establish an IDLE connection and log raw email headers to `stdout` with millisecond timestamps.

2. **Configure the Priority Scorer Thresholds**  
   Edit `/etc/focusinbox/priority.yaml` to set `min_priority_threshold: 0.65` (emails below this are archived without task extraction), `temporal_decay_lambda: 0.02`, and `sender_reputation_db_path: /var/lib/focusinbox/reputation.db`.  
   Reload the config via `kill -HUP $(pidof focusinbox-eight)` — the system will hot-reload without dropping connections.

3. **Monitor the Task Queue via PostgreSQL**  
   Connect to the database: `psql -h localhost -U focusadmin -d focusinbox`.  
   Query pending tasks: `SELECT task_id, extracted_due_date, assigned_owner FROM focus_tasks WHERE status = 'pending' ORDER BY priority_score DESC LIMIT 50;`.  
   Manually retry a failed task: `UPDATE focus_tasks SET status = 'pending', retry_count = 0 WHERE task_id = '<uuid>' AND status = 'failed';`.  
   Use the built-in `GET_TASK_STATS()` function to view throughput metrics: `SELECT * FROM get_task_stats(interval '1 hour');`.

### Best Practices

- **Practice 1: Tune Priority Decay Based on Email Volume**  
  For inboxes receiving >500 emails/day, set `temporal_decay_lambda` between 0.03 and 0.05 to prevent stale emails from floating to the top. Use the `priority_scores` histogram endpoint (`GET /metrics/priority-distribution`) to visualize score distribution and adjust weekly. A lambda too low (e.g., 0.01) causes 24-hour-old emails to retain 0.8 of original score, crowding out urgent new arrivals.

- **Practice 2: Implement a Two-Phase Intent Classification Fallback**  
  If BERT inference latency exceeds 500ms for more than 5% of emails in a 10-minute window, enable FastText-only mode via the `--fasttext-only` flag. Monitor accuracy via the `intent_confidence_log` table — if FastText confidence drops below 0.6 for more than 100 consecutive emails, trigger an alert to retrain the BERT model on recent email samples. This ensures sub-100ms classification during traffic spikes.
