# focusinbox-eight - FOC_030

## Overview

`focusinbox-eight` is a high-performance, latency-optimized inbox processing engine designed to filter, prioritize, and batch transactional email streams in real time. Built for systems that handle over 10,000 inbound messages per second, it reduces cognitive overhead by applying rule-based triage with sub‑millisecond decision latency. The tool is engineered for DevOps engineers and platform architects who need to eliminate noise from automated notification pipelines.

## Core Features

### 1. Adaptive Priority Scoring Engine
Uses a tunable scoring model based on `sender_reputation`, `urgency_header`, `thread_depth`, and `keyword_density` to assign a `priority_score` between 0.0 and 1.0. The engine recalculates scores every 500 ms using an in‑memory LRU cache with a 95th‑percentile lookup time of 120 µs. Supports custom weight vectors via a JSON configuration file (`priority_weights.json`), enabling per‑tenant calibration without code changes.

### 2. Multi‑Stage Batching with Backpressure Control
Implements a three‑stage pipeline: **Ingestion** → **Deduplication** → **Dispatch**. The deduplication stage uses a Bloom filter with a false‑positive rate of 0.001% and a TTL of 30 seconds. Backpressure is managed via a configurable `max_batch_size` (default: 1024) and `batch_window_ms` (default: 250). When the dispatch queue exceeds 10,000 items, the system automatically throttles ingestion using a token‑bucket algorithm with a rate of 8,000 ops/s.

### 3. PostgreSQL‑Backed Audit Trail with Full‑Text Search
Every processed message is persisted to a `focusinbox_audit` table with columns for `message_id`, `ingestion_timestamp`, `priority_score`, and `action_taken`. The table is indexed with a GIN index on `tsvector` for fast content‑based queries. Retention is configurable via a `retention_days` parameter (default: 90). Queries for “all high‑priority messages from a specific domain in the last hour” complete in under 200 ms on a dataset of 5 million rows.

## How to Use

### Step 1: Configure the Priority Weight Map
Create a `priority_weights.json` file in the working directory:
```json
{
  "sender_reputation": 0.4,
  "urgency_header": 0.3,
  "thread_depth": 0.2,
  "keyword_density": 0.1
}
```
Run the validation command:
```bash
focusinbox-eight validate-weights --file priority_weights.json
```
The tool will output a summary of the expected score distribution based on a sample of 10,000 recent messages.

### Step 2: Start the Ingestion Pipeline
Launch the daemon with explicit PostgreSQL connection parameters:
```bash
focusinbox-eight daemon \
  --pg-host 10.0.0.5 \
  --pg-port 5432 \
  --pg-db focusinbox_prod \
  --pg-user inbox_worker \
  --max-batch-size 1024 \
  --batch-window-ms 250 \
  --bloom-fpr 0.001
```
Monitor the startup logs for `Pipeline ready` confirmation. The daemon will begin pulling from the configured message queue (e.g., RabbitMQ or Amazon SQS).

### Step 3: Query the Audit Trail
After processing, retrieve top‑priority messages using the built‑in CLI:
```bash
focusinbox-eight query \
  --since "2025-03-01T00:00:00Z" \
  --until "2025-03-01T23:59:59Z" \
  --min-score 0.8 \
  --limit 50
```
The output is a JSON array with each message’s `message_id`, `sender_domain`, `priority_score`, and `action`. Append `--format table` for a human‑readable terminal view.

## Best Practices

### 1. Pre‑Warm the Bloom Filter During Deployment
When rolling out a new version, seed the Bloom filter with the last 24 hours of message IDs from the PostgreSQL audit table. This prevents false positives during the first 30 seconds of operation, which can cause duplicate dispatches. Use the `seed-bloom` subcommand:
```bash
focusinbox-eight seed-bloom --pg-db focusinbox_prod --hours 24
```

### 2. Set Up a Dead‑Letter Queue for Score Anomalies
Configure a separate queue for messages where `priority_score` is exactly 0.5 (the default midpoint). This often indicates a configuration error or a missing weight mapping. Route these to a monitoring channel in your alerting system (e.g., PagerDuty or Opsgenie) to catch scoring misconfigurations before they impact production routing.
