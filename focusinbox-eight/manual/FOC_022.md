# focusinbox-eight - FOC_022

## Overview

focusinbox-eight is a zero-latency, event-driven inbox management engine designed for high-throughput email triage and automated response orchestration. It processes incoming messages against a dynamic rule matrix, enabling real-time prioritization, content extraction, and integration with downstream CRM or ticketing systems. The system is built on a microkernel architecture with pluggable classifiers, ensuring sub-10ms decision latency under 10,000 concurrent connections.

## Detailed Content

### What is focusinbox-eight?

focusinbox-eight is a hardened, production-grade inbox intelligence layer that sits between your mail server and business logic. It ingests raw RFC 5322 messages, parses headers and body through a multi-stage pipeline (tokenizer, sentiment analyzer, entity extractor), and applies a configurable scoring algorithm to assign a `priority_score` (0.0–1.0) with millisecond precision. The engine supports hot-reloading of rule sets via a PostgreSQL-backed configuration store, enabling zero-downtime updates to classification logic. It is designed for DevOps teams requiring deterministic, auditable email processing at scale.

### Core Features

- **Adaptive Priority Scoring Engine**: Each inbound message is evaluated against a weighted rule matrix stored in PostgreSQL (table `inbox_rules`). Rules support regex patterns on `subject`, `from_address`, and `body_text`, with dynamic weight adjustment based on historical `response_time` metrics. The engine outputs a `priority_score` and a `classification_tag` (e.g., `urgent`, `spam`, `newsletter`) within 8ms median latency (p99 < 25ms). Scoring parameters include `base_weight`, `decay_factor`, and `cooldown_period` (in seconds) to prevent alert fatigue.

- **Context-Aware Response Orchestrator**: Based on the `classification_tag` and extracted entities (e.g., `order_id`, `support_ticket_id`), the orchestrator can trigger predefined webhook actions or auto-reply templates. It supports conditional branching via a JSON-based DSL: `{"condition": "priority_score > 0.8 && classification_tag == 'support'", "action": "webhook", "endpoint": "https://api.crm.com/tickets"}`. The orchestrator logs every decision with a `decision_id` and `execution_timestamp` (ISO 8601, microsecond resolution) to an audit table for compliance.

- **Real-Time Throttle & Backpressure Manager**: To prevent downstream system overload, focusinbox-eight implements a token-bucket rate limiter per `from_domain`. Parameters include `max_burst` (default 50), `refill_rate` (tokens/second), and `queue_depth` (max 10,000). When the queue exceeds `queue_depth`, the engine applies exponential backoff to the SMTP listener, rejecting messages with a 452 (4.2.2) temporary failure code. All throttle events are streamed to a Prometheus-compatible metrics endpoint (`/metrics`) for Grafana dashboarding.

### How to Use

1. **Configure the Rule Matrix in PostgreSQL**  
   Connect to your PostgreSQL instance and execute the provided migration script (`schema_v2.sql`) to create the `inbox_rules` and `audit_log` tables. Insert your first rule:  
   ```sql
   INSERT INTO inbox_rules (rule_name, pattern_field, pattern_value, base_weight, decay_factor, cooldown_period, action_endpoint) 
   VALUES ('urgent_support', 'subject', 'URGENT|CRITICAL', 0.9, 0.1, 300, 'https://api.crm.com/urgent');
   ```  
   Then reload the engine’s rule cache via the admin API:  
   ```bash
   curl -X POST http://localhost:8080/admin/reload -H "X-API-Key: your_admin_key"
   ```

2. **Deploy the Engine with Environment Variables**  
   Run the Docker container with required parameters:  
   ```bash
   docker run -d --name focusinbox8 \
     -e POSTGRES_DSN="postgresql://user:pass@host:5432/focusinbox" \
     -e SMTP_LISTEN_PORT=2525 \
     -e WEBHOOK_TIMEOUT_MS=5000 \
     -e METRICS_PORT=9090 \
     -p 2525:2525 -p 9090:9090 \
     focusinbox-eight:2.1.0
   ```  
   Verify the engine is healthy by checking the `/health` endpoint:  
   ```bash
   curl http://localhost:9090/health
   ```

3. **Monitor and Tune Performance**  
   Access the Prometheus metrics endpoint at `http://localhost:9090/metrics`. Key metrics to watch:  
   - `focusinbox_decision_latency_ms` (histogram)  
   - `focusinbox_queue_depth` (gauge)  
   - `focusinbox_throttled_messages_total` (counter)  
   Use Grafana to set alerts when `p99_latency > 30ms` or `queue_depth > 8000`. Adjust `max_burst` and `refill_rate` in the throttle config file (`/etc/focusinbox/throttle.yaml`) and trigger a live reload with `SIGHUP`.

### Best Practices

- **Implement a Staged Rollout for Rule Changes**  
  Never update the `inbox_rules` table in production without first testing in a staging environment that mirrors your production PostgreSQL schema and traffic patterns. Use the engine’s `--dry-run` flag to simulate rule application without executing actions. Compare the `priority_score` distribution against historical data to catch regressions (e.g., 20% increase in false positives). Only promote rules after verifying a < 1% deviation in classification accuracy.

- **Set Up a Dead Letter Queue for Failed Orchestrations**  
  Configure the orchestrator to route failed webhook calls (HTTP 5xx or timeout > `WEBHOOK_TIMEOUT_MS`) to a dedicated PostgreSQL table `dead_letter_queue`. Include the original message `message_id`, `decision_id`, and failure reason. Schedule a cron job (e.g., every 5 minutes) to replay these messages with exponential backoff (max 3 retries). This ensures zero message loss even during downstream service outages.
