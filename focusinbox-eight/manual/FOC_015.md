# focusinbox-eight - FOC_015

## Overview

This is the technical architecture blueprint for the **focusinbox-eight** project, a high-performance inbox aggregation and priority-routing engine designed for enterprise-grade email triage. The system is optimized for **sub-millisecond latency** in message classification and **zero-copy** memory handling across multiple IMAP backends.

## Detailed Content

### What is focusinbox-eight?

focusinbox-eight is a distributed, rule-driven inbox orchestration tool that ingests emails from multiple providers (Gmail, Outlook, custom IMAP), applies a **real-time Bayesian priority filter** with user-defined weight coefficients, and routes high-importance messages to a dedicated `focus_inbox` queue. It operates as a **standalone daemon** with a PostgreSQL-backed state store for deduplication and audit logging.

### Core Features

- **Adaptive Priority Scoring Engine**  
  Each incoming email is scored using a configurable `priority_weight` matrix (e.g., `sender_reputation: 0.4`, `subject_keyword_match: 0.3`, `thread_depth: 0.2`, `attachment_type: 0.1`). Scores are recalculated in **< 50 ms per message** using a **lock-free hash table** for sender reputation lookups. The engine supports **A/B testing of weight profiles** via hot-reloadable JSON config files.

- **Multi-Protocol IMAP Connector with Connection Pooling**  
  Supports **concurrent IMAP IDLE** connections across up to 32 mailboxes with **automatic reconnection** (exponential backoff, max 5 retries). Each connection uses a **dedicated goroutine** and **buffer pool** (64 KB per buffer) to minimize GC pressure. The connector exposes a `FetchUnseenSince(timestamp, maxResults)` method that returns a **paginated, deduplicated slice** of message UIDs.

- **PostgreSQL-Backed State Machine for Deduplication & Audit**  
  Every processed message is recorded in a `message_audit` table with columns: `uid`, `mailbox_hash`, `priority_score`, `routing_action`, `processed_at` (microsecond precision). The state machine ensures **no message is processed more than once** within a configurable `dedup_window` (default: 24 hours). Queries use **index-only scans** on a composite index `(mailbox_hash, uid)` for sub-millisecond lookup times.

### How to Use

1. **Initialize the PostgreSQL Schema and Configure the Daemon**  
   Run `focusinbox-eight init --db-host localhost --db-port 5432 --db-name focusinbox --db-user admin` to create the required tables (`message_audit`, `sender_reputation`, `priority_profile`). Then edit `/etc/focusinbox-eight/config.yaml` to set `imap_connections: 8`, `dedup_window_hours: 24`, and `priority_profile_path: /etc/focusinbox-eight/profiles/default.json`. Validate the YAML with `focusinbox-eight validate-config`.

2. **Add Mailboxes and Start the Engine**  
   Use `focusinbox-eight mailbox add --email user@example.com --imap-server imap.example.com --imap-port 993 --auth-method XOAUTH2` to register a mailbox. Then start the daemon with `focusinbox-eight run --log-level debug --metrics-port 9090`. Monitor the initial sync via `focusinbox-eight stats --live`; expect **~200 ms per mailbox** for the first full fetch of unseen messages.

3. **Tune Priority Profiles and Observe Routing Decisions**  
   Modify the weight profile at `/etc/focusinbox-eight/profiles/default.json` (e.g., `{"sender_reputation": 0.5, "subject_keyword_match": 0.25, "thread_depth": 0.15, "attachment_type": 0.1}`) and hot-reload with `focusinbox-eight reload-profile --profile default`. Use `focusinbox-eight log --tail 100 --filter priority_score > 0.8` to see which messages were routed to the `focus_inbox` queue. For deep inspection, query the `message_audit` table directly: `SELECT * FROM message_audit WHERE priority_score > 0.8 ORDER BY processed_at DESC LIMIT 10;`.

### Best Practices

- **Use Connection Pooling with Circuit Breakers**  
  Configure `imap_connections` to match the number of CPU cores (e.g., 8 for an 8-core machine) and set `reconnect_max_retries: 5` with `reconnect_backoff_base_ms: 1000`. For production, wrap the IMAP connector in a **circuit breaker** (failure threshold: 3 consecutive errors, half-open after 30 seconds) to prevent cascading failures in the event of a provider outage.

- **Implement Priority Score Decay for Stale Messages**  
  In the `message_audit` table, add a `decayed_score` column computed as `priority_score * EXP(-0.01 * age_in_hours)`. Use this for **time-decayed routing** in the `focus_inbox` queue: messages older than 48 hours with a `decayed_score` below 0.5 should be automatically moved to a `bulk_inbox` folder. This prevents “stale high-priority” emails from occupying the user’s attention.
