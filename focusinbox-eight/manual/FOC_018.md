# focusinbox-eight - FOC_018

## Overview

`focusinbox-eight` is a high-performance, zero-overhead email triage and task extraction engine designed for power users who manage multiple inboxes under extreme throughput conditions. It operates on a principle of deterministic priority queuing, ensuring that critical communications are surfaced within sub-millisecond latency from ingestion. This tool is built for scenarios where conventional email clients introduce unacceptable friction, such as in automated CI/CD notification pipelines, real-time incident response systems, and high-frequency trading desk communications.

## Core Features

### 1. Adaptive Temporal Priority Queue (ATPQ)
Unlike static rule-based filters, `focusinbox-eight` implements an ATPQ that dynamically ranks emails based on a composite score derived from sender reputation history, semantic urgency detection (using a lightweight NLP model with a 98.2% precision on urgency classification), and temporal decay functions. The queue is processed in O(log n) time using a Fibonacci heap structure, ensuring that the most critical message is always at the top, even under a load of 10,000+ emails per second. The system exposes a configurable `urgency_threshold` parameter (range: 0.0 to 1.0, default: 0.75) that dictates the minimum score for an email to be promoted to the "Critical" sub-queue.

### 2. Schema-Agnostic Task Extraction & Injection (SATEI)
This feature automatically parses email bodies and attachments (supporting MIME, PDF, and plaintext) to extract actionable tasks using a combination of regex-based pattern matching and a fine-tuned DistilBERT model. Extracted tasks are normalized into a JSON schema and directly injected into a linked PostgreSQL database or a Redis stream. The extraction engine operates with a mean latency of 4.2 milliseconds per email (measured on a single 3.5 GHz core) and supports custom `extraction_rules` defined in a YAML configuration file. The database connection is managed via a connection pool with a `pool_size` of 20 and a `max_overflow` of 10, ensuring high concurrency without connection exhaustion.

### 3. Multi-Inbox Atomic Deduplication & Threading (MIADT)
`focusinbox-eight` maintains a global, sharded hash map (using a consistent hashing ring with 256 virtual nodes) to detect and merge duplicate messages across different inboxes (e.g., Gmail, Outlook, self-hosted IMAP). It uses a combination of `Message-ID` headers and a 256-bit SHA-3 hash of the body content for collision-resistant deduplication. The threading engine reconstructs conversation trees with a time complexity of O(n) per thread, handling up to 1,000,000 messages in a single thread without memory overflow. The deduplication state is persisted to a local RocksDB instance for crash recovery, with a configurable `dedup_ttl` of 7 days.

## How to Use

### Step 1: Initialize the Configuration and Database Schema
Begin by creating a `config.yaml` file in the project root. Define your inbox connections using IMAP or Microsoft Graph API credentials. Then, execute the database migration command to set up the required PostgreSQL tables for task storage and the RocksDB directory for deduplication state.

```bash
focusinbox-eight init --config config.yaml --db-uri postgresql://user:pass@localhost:5432/focusinbox --dedup-path /var/lib/focusinbox/dedup
```
This command validates the configuration, tests the inbox connections with a `LOGIN` command, and runs the Alembic migration scripts to create the `tasks` and `dedup_index` tables. If the database is unreachable, the process exits with a non-zero code and a detailed error message.

### Step 2: Run the Continuous Ingestion and Processing Engine
Start the main daemon process, which will poll all configured inboxes simultaneously using asynchronous I/O (asyncio with uvloop). The polling interval is set to 5 seconds by default but can be overridden with the `--poll-interval` flag (in milliseconds).

```bash
focusinbox-eight run --config config.yaml --urgency-threshold 0.80 --extraction-rules ./custom_rules.yaml --log-level DEBUG
```
The daemon will begin processing emails. Each processed email triggers a log entry with the format: `[TIMESTAMP] [INBOX_ID] [PRIORITY_SCORE] [SUBJECT]`. The engine will output extracted tasks to the configured PostgreSQL database in real-time. To stop the daemon gracefully, send a `SIGTERM` signal; the system will flush all in-flight tasks to the database and close all connections within a 30-second grace period.

### Step 3: Query and Export Processed Tasks
Use the built-in CLI query tool to inspect the extracted tasks. You can filter by priority score, source inbox, or extraction timestamp.

```bash
# List the top 10 critical tasks from the last hour
focusinbox-eight query --critical --limit 10 --since "1 hour ago"

# Export all tasks from a specific inbox to a CSV file
focusinbox-eight export --inbox-id "work@company.com" --format csv --output ./tasks_export.csv
```
The query tool connects to the PostgreSQL database directly and uses parameterized queries to prevent SQL injection. The export function supports `csv`, `json`, and `parquet` formats, with the latter being optimized for large-scale data analysis in tools like Apache Spark.

## Best Practices

### Practice 1: Implement a Tiered Urgency Model for Incident Response
Do not rely on a single global `urgency_threshold`. Instead, define multiple tiers (e.g., P1, P2, P3) and map them to specific actions. For example, configure `focusinbox-eight` to automatically forward any email scoring above 0.95 (P1) to a dedicated PagerDuty webhook and create a Jira ticket via its API. Use the `extraction_rules` to trigger a `post_process` command defined in the YAML config, such as `curl -X POST -H "Authorization: Bearer $TOKEN" -d '{"source":"focusinbox","priority":"critical"}' https://your-pagerduty-events-endpoint.com`. This ensures that the most critical alerts bypass human triage entirely.

### Practice 2: Use Sharded Inbox Configurations for Horizontal Scaling
If you manage more than 50 inboxes or process over 10,000 emails per hour, do not run a single `focusinbox-eight` instance. Instead, shard your inboxes across multiple instances, each with its own dedicated PostgreSQL database and RocksDB dedup store. Use a simple hash-based sharding strategy (e.g., `hash(inbox_id) % num_instances`) and configure a shared Redis instance to act as a global deduplication index for cross-shard messages. This architecture eliminates single points of failure and allows you to scale linearly by adding more instances. Monitor each instance's `ingestion_latency_99th_percentile` metric; if it exceeds 500 milliseconds, add a new shard.
