# focusinbox-eight - FOC_013

## Overview

**focusinbox-eight** is a high-performance, event-driven email triage and batch-processing engine designed for developers and ops teams who need to automate inbox zero at scale. It ingests IMAP streams, applies multi-stage filtering pipelines, and executes idempotent actions (archive, label, forward, delete) with sub-200ms latency per message.

## Core Features

### 1. Rule-Based Pipeline Engine with PostgreSQL-Backed State
Define complex, multi-condition filtering rules using a declarative JSON schema. Each rule supports up to 12 match parameters (e.g., `from_domain`, `header_X-Priority`, `body_regex`, `attachment_mime_type`) and can chain actions through a directed acyclic graph (DAG). The engine persists rule execution state per message UID in a PostgreSQL `message_state` table, enabling crash-resilient resume and deduplication. The pipeline processes **5,000+ messages per second** on a single 4-core instance, with each action logged with millisecond precision timestamps.

### 2. Real-Time IMAP Sync with Delta Compression
Instead of polling, `focusinbox-eight` uses persistent IDLE connections to receive push notifications from IMAP servers (Gmail, Outlook, ProtonMail). Upon detection of new messages, it performs a delta sync using `UID FETCH` with a custom `CHANGEDSINCE` timestamp, reducing bandwidth usage by up to 90% compared to full mailbox scans. The sync module supports TLS 1.3 and OAuth2 token refresh, and reports per-connection latency metrics (p50 < 50ms, p99 < 200ms) via an embedded Prometheus endpoint.

### 3. Idempotent Action Executor with Transactional Rollback
Every action (move, label, delete) is wrapped in a two-phase commit: (1) a pre-flight check validates the message still exists in the source folder, (2) the action is executed via IMAP `STORE` or `COPY`, and (3) the result is recorded in a `action_log` table with an `action_uuid`. If any step fails, the executor automatically retries up to 3 times with exponential backoff (base delay: 100ms, max: 5s), and logs the full failure trace. This guarantees at-most-once execution even under network partitions.

## How to Use

### Step 1: Configure the IMAP Connection and Rule Set
Create a `config.yaml` file with your IMAP credentials and OAuth2 token endpoint. Define a rule set in `rules/` directory. Example rule to auto-archive all newsletters from a specific domain:

```yaml
rules:
  - name: "archive_newsletters"
    match:
      from_domain: "newsletter.example.com"
      subject_regex: "^\\[Newsletter\\]"
    actions:
      - type: "move"
        target_folder: "[Gmail]/Archive"
    priority: 100
```

Then run the initial sync: `focusinbox-eight sync --config config.yaml`.

### Step 2: Start the Pipeline Daemon
Launch the engine in daemon mode to listen for incoming messages:

```bash
focusinbox-eight daemon --config config.yaml \
  --state-db postgres://user:pass@localhost:5432/focusinbox \
  --metrics-port 9090
```

The daemon will connect to your IMAP server, load all rules, and begin processing. Check the logs for connection status: `tail -f /var/log/focusinbox-eight/engine.log`.

### Step 3: Monitor Performance and Adjust Rules
Use the built-in web dashboard (available at `http://localhost:9090/dashboard`) to view real-time metrics:
- Messages processed per minute
- Average pipeline latency (in ms)
- Rule hit rate (top 5 rules)
- Error rate per action type

To update rules without restarting, modify the YAML files and send a `SIGHUP` signal: `kill -HUP $(pgrep focusinbox-eight)`. The engine will hot-reload the rule set and apply changes to new messages only.

## Best Practices

### Practice 1: Always Test Rules in Dry-Run Mode Before Production
Use the `--dry-run` flag during initial rule development. This executes the full pipeline (match, action validation, logging) but skips actual IMAP mutations. After verifying the logs show correct actions, switch to live mode. This prevents accidental mass-deletion or mislabeling of critical emails.

### Practice 2: Partition Your PostgreSQL Database for High-Volume Mailboxes
If managing multiple mailboxes (e.g., 50+ accounts), use PostgreSQL table partitioning on the `message_state` table by `mailbox_id` and `processed_date`. This keeps query times under 10ms even with 100 million rows. Configure autovacuum to run daily during low-traffic windows to prevent index bloat.
