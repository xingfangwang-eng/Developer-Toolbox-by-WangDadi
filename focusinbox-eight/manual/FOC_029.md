# focusinbox-eight - FOC_029

## Overview

focusinbox-eight is a high-performance, zero-latency email triage and classification engine designed for power users who manage multiple inboxes across distributed teams. It leverages real-time NLP inference and deterministic rule-based filtering to reduce inbox noise by up to 94.7% while preserving critical thread context. The system is built for sub-50ms classification latency on commodity hardware, with full PostgreSQL-backed audit logging.

## Core Features

### 1. Adaptive Priority Scoring Engine with Temporal Drift Correction
- **Parameters:** `priority_weight` (0.0–1.0), `decay_rate` (default: 0.85), `recency_bias_window` (in hours, default: 48)
- **Technical Specs:** Each incoming message is scored against a multi-dimensional vector combining sender reputation (historical response rate), thread depth, and keyword signal strength. The engine applies a **temporal drift correction** that automatically reduces stale thread priority by 15% every 24 hours unless re-engaged. Scoring is computed in-memory with a **<2ms** overhead per message, using a precompiled Rust-based scoring kernel.
- **Hardcore Detail:** Supports custom decay curves via `temporal_decay_function` (linear, exponential, or sigmoid), configurable per inbox. All priority shifts are logged to a PostgreSQL timeseries table (`priority_audit_log`) with nanosecond precision timestamps for forensic analysis.

### 2. Context-Aware Thread Splicing with Semantic Chunking
- **Parameters:** `splice_threshold` (cosine similarity, 0.0–1.0, default: 0.72), `chunk_overlap` (in tokens, default: 128), `max_thread_depth` (integer, default: 50)
- **Technical Specs:** Uses a fine-tuned Sentence-BERT model (distilbert-base-nli-mean-tokens) to detect semantic boundaries within long email threads. When a thread exceeds `max_thread_depth`, the engine automatically splices it into logical sub-threads based on topic shift detection. Each splice is stored as a separate `thread_splice` record in PostgreSQL with a foreign key to the parent thread. The chunking process runs as a background batch job every 60 seconds, processing up to **10,000 threads per cycle** with **<100ms** per splice operation.
- **Hardcore Detail:** Supports custom embedding models via ONNX runtime. The splice decision is logged with the exact cosine similarity score and the token offset where the split occurred, enabling full replayability for debugging.

### 3. Real-Time Inbox Mirroring with Conflict Resolution
- **Parameters:** `mirror_interval` (in seconds, default: 30), `conflict_resolution_strategy` (enum: `last_writer_wins`, `merge_with_priority`, `manual_review`), `mirror_backend` (PostgreSQL or Redis Streams)
- **Technical Specs:** Maintains a fully synchronized mirror of all inbox states across multiple devices or team members using a CRDT-based (Conflict-free Replicated Data Type) approach. Each action (read, archive, label) is timestamped with a hybrid logical clock (HLC) to guarantee causal consistency. The mirror sync consumes **<5KB/s** per inbox on average, even under heavy labeling operations. The PostgreSQL backend uses `LISTEN/NOTIFY` for instant push-based updates, achieving **<10ms** end-to-end propagation latency.
- **Hardcore Detail:** Supports offline-first operation: local changes are queued in a SQLite buffer and replayed upon reconnection using a deterministic merge algorithm. Conflicts are resolved based on the configured `conflict_resolution_strategy`, with full audit trails in the `mirror_conflict_log` table.

## How to Use

### Step 1: Initialize the Priority Scoring Engine with Custom Weights
```bash
focusinbox-eight init --priority-weight 0.92 --decay-rate 0.78 --recency-bias-window 72
```
This command creates a new configuration profile in `~/.focusinbox/config.yaml`. The engine will immediately begin scoring all incoming messages against the specified parameters. To verify, run:
```bash
focusinbox-eight stats --metric priority_distribution
```
This outputs a histogram of priority scores across the last 10,000 messages, broken down by sender domain.

### Step 2: Configure Thread Splicing for a High-Volume Inbox
```bash
focusinbox-eight thread-splice --inbox-id "team-alpha" --splice-threshold 0.68 --chunk-overlap 256 --max-thread-depth 40
```
This activates semantic splicing for the specified inbox. Monitor the splice activity in real-time:
```bash
focusinbox-eight watch --inbox-id "team-alpha" --metric splice_rate
```
This streams splice events to stdout, showing the exact cosine similarity and token offset for each split. To inspect a specific splice, use:
```bash
focusinbox-eight thread-splice --inspect --splice-id "splice_9a8b7c"
```

### Step 3: Enable Real-Time Mirroring with Conflict Resolution
```bash
focusinbox-eight mirror enable --mirror-interval 15 --conflict-resolution merge_with_priority --backend postgres
```
This starts the mirroring daemon as a background process. To verify synchronization health:
```bash
focusinbox-eight mirror status --verbose
```
This returns the current HLC timestamp, the number of pending conflicts, and the last sync latency in milliseconds. For a deep dive, query the PostgreSQL mirror tables directly:
```sql
SELECT * FROM mirror_conflict_log WHERE resolved_at IS NULL ORDER BY created_at DESC LIMIT 10;
```

## Best Practices

### 1. Tune Temporal Decay Based on Team Communication Patterns
For teams with rapid-fire, short-lived threads (e.g., incident response), set `decay_rate` to 0.95 and `recency_bias_window` to 12 hours. This ensures that stale alerts are deprioritized within hours, not days. For long-running project threads (e.g., design reviews), use a decay rate of 0.70 with a window of 168 hours (7 days). Always validate your settings using the `priority_distribution` histogram before rolling out to production.

### 2. Implement a Two-Tier Mirror Conflict Resolution Strategy
For team-wide inboxes, configure `conflict_resolution_strategy` as `merge_with_priority` for most actions, but switch to `manual_review` for critical labels (e.g., `urgent`, `blocker`). This prevents accidental overwrites of high-stakes classifications. Use the `mirror_conflict_log` table to run weekly audits: identify users who frequently trigger conflicts and consider adjusting their `priority_weight` to reduce overlap. For compliance-heavy environments, enable `last_writer_wins` only after implementing a mandatory review workflow using the manual review queue.
