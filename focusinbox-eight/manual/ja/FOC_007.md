# focusinbox-eight - FOC_007

## 概要

focusinbox-eight は、高頻度・高精度な受信トレイ処理を実現するための、超低遅延メールフィルタリングエンジンです。キーワード「focusinbox-eight tool」「focusinbox-eight solution」「focusinbox-eight software」に特化し、**ミリ秒単位のレイテンシ**と**PostgreSQLとの密結合**を前提とした設計が特徴です。

## コア機能

### 1. リアルタイム・マルチスレッド受信トレイフィルタリングエンジン

- **パラメータ**: `--thread-pool-size`（デフォルト: 8）、`--batch-flush-interval-ms`（デフォルト: 50ms）
- 内部で **libevent** ベースのイベントループを活用し、**1,000通/秒**の受信メールに対して **99.9%** のフィルタリングを **50ms以内** に完了。
- フィルタリング結果は即座に **PostgreSQL（テーブル: `inbox_filter_log`）** に書き込まれ、**トランザクション分離レベル Serializable** で整合性を担保。

### 2. 動的ルールエンジン with ネストクエリ解析

- **パラメータ**: `--rule-cache-ttl-sec`（デフォルト: 300）、`--max-nest-depth`（デフォルト: 5）
- ユーザーが定義した **JSONベースのルールセット**（例: `{"match": {"from": "example.org", "subject_regex": "^重要"}}`）を **AST（抽象構文木）** として保持し、**O(log n)** でのマッチングを実現。
- ルール変更は **Hot Reload** に対応し、**ダウンタイムゼロ** で適用可能。

### 3. 履歴集約・異常検知ダッシュボード連携

- **パラメータ**: `--aggregation-window-min`（デフォルト: 15）、`--anomaly-threshold-zscore`（デフォルト: 3.0）
- 過去 **7日間** のフィルタリング結果を **PostgreSQLのウィンドウ関数** で集約し、**Zスコア** に基づく異常値検知を実装。
- 検知結果は **Webhook** 経由で任意のダッシュボード（Grafana等）に **JSON** 形式で即時通知。

## 使い方

### ステップ1: 環境変数とPostgreSQL接続設定

```bash
export DB_HOST=localhost
export DB_PORT=5432
export DB_USER=focusinbox
export DB_PASSWORD=secure_password
export DB_NAME=focusinbox_eight
```

- 必須: `postgresql.conf` で `max_connections` を **200以上** に設定。
- 推奨: `shared_buffers` を **4GB以上** に割り当て。

### ステップ2: フィルタリングルールの定義

`rules.json` を作成し、以下の形式でルールを記述:

```json
[
  {
    "id": "rule-001",
    "priority": 1,
    "match": {
      "from_domain": ["phishing-site.com", "malware.org"],
      "subject_regex": "^緊急|^重要|【警告】"
    },
    "action": "quarantine",
    "ttl_sec": 86400
  }
]
```

- ルールは **優先度順** にソートされ、最初にマッチしたルールが適用される。
- 各ルールには **TTL** を設定可能で、期限切れ後は自動無効化。

### ステップ3: エンジンの起動とパフォーマンス監視

```bash
./focusinbox-eight --config ./rules.json --thread-pool-size 16 --batch-flush-interval-ms 30
```

- 起動後、標準出力に **毎秒の処理件数** と **平均レイテンシ（ms）** が出力される。
- 異常検知が発生した場合、**syslog** に `[ANOMALY]` タグ付きで出力されるため、`journalctl -u focusinbox-eight -f` でリアルタイム監視可能。

## ベストプラクティス

### 1. ルールの定期メンテナンスとキャッシュ戦略

- ルールセットは **1日1回** の定期更新を推奨。更新時は `--rule-cache-ttl-sec` を **3600秒（1時間）** に設定し、キャッシュの無効化タイミングをずらすことで **スパイク負荷** を回避。
- ルールの総数が **10,000件** を超える場合は、**インデックス付きのPostgreSQLテーブル** にルールを格納し、エンジン起動時に `COPY` で一括ロードする設計が望ましい。

### 2. 異常検知の閾値チューニング

- 初期値として `--anomaly-threshold-zscore 3.0` を推奨するが、誤検知が多い場合は **2.5** に緩和し、逆に検知漏れが多い場合は **3.5** に引き上げる。
- 異常検知の結果は、**PostgreSQLの `anomaly_log` テーブル** に **UNLOGGED** テーブルとして保存し、**24時間** で自動パージする運用が効率的。
