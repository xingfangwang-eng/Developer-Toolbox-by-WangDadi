# focusinbox-eight - FOC_012

## 概要

focusinbox-eight は、受信トレイ管理を極限まで効率化するための次世代型メールフィルタリングエンジンです。特に、大量のメールを処理するプロフェッショナル向けに設計されており、キーワード「focusinbox-eight tool」「focusinbox-eight solution」「focusinbox-eight software」に基づき、メールの自動分類、優先度スコアリング、およびコンテキスト依存のアーカイブを実現します。

## 詳細な内容

### focusinbox-eight とは

focusinbox-eight は、独自の**マルチレイヤーフィルターアーキテクチャ**を採用し、メールの送信元、本文のトピックモデリング、添付ファイルのメタデータ解析を組み合わせて、受信トレイ内のノイズを排除します。PostgreSQL との連携により、全メールの履歴をミリ秒単位でクエリ可能なインデックスとして保存し、ユーザーの行動パターンに基づいた動的なルール生成を実現します。

### コア機能

- **動的優先度スコアリングエンジン**: メール受信時に、`priority_score` パラメータ（0.0～1.0、小数点第3位まで算出）をリアルタイムで計算。過去7日間の返信率、送信者のドメイン信頼性、メール本文の緊急度（キーワード密度分析）を考慮し、`threshold=0.85` を超えるメールのみを「フォーカスビュー」に表示。
- **コンテキスト依存アーカイブ機能**: `archive_context` パラメータに基づき、メールをプロジェクト別、クライアント別、またはトピッククラスター別に自動分類。PostgreSQL の `jsonb` カラムにメタデータを格納し、`tsvector` による全文検索をミリ秒単位で実行可能。
- **マルチスレッドバッチ処理**: 1秒間に最大500通のメールを処理可能な非同期ワーカーを搭載。`batch_size=100`、`timeout_ms=3000` の設定で、添付ファイルのウイルススキャンとOCR処理を並列実行し、結果を`processed_at` タイムスタンプとともにデータベースにコミット。

### 使い方

1. **初期セットアップとデータベース接続**:
   - PostgreSQL 16.0 以上を用意し、`focusinbox_db` データベースを作成。
   - 環境変数 `DATABASE_URL=postgresql://user:pass@localhost:5432/focusinbox_db` を設定。
   - 次のコマンドでマイグレーションを実行: `focusinbox-eight migrate --schema=public --index-type=gin`。これにより、`emails` テーブルに `priority_score` と `archive_context` 用のGINインデックスが自動生成されます。

2. **フィルタールールのカスタマイズ**:
   - `config/rules.yaml` を編集し、`rules[].condition` に正規表現またはキーワードリストを指定（例: `pattern: "緊急|至急|urgent"`）。
   - `rules[].action` に `priority_boost: 0.2` または `archive_to: "project_x"` を設定。
   - 設定反映後、`focusinbox-eight reload --config=./config/rules.yaml` を実行。変更はホットリロードされ、`reload_interval_ms=500` で適用されます。

3. **バッチ処理とモニタリング**:
   - メールサーバー（IMAP対応）の接続情報を `config/imap.yaml` に記述（例: `host: imap.gmail.com, port: 993, ssl: true`）。
   - `focusinbox-eight process --batch-size=200 --timeout=5000` を実行。処理中は `logs/process.log` に各メールの `message_id`、`processing_time_ms`、`result`（`filtered`、`archived`、`failed`）が記録されます。
   - 進捗確認は `focusinbox-eight status --format=json` で、`total_processed`、`avg_latency_ms`、`error_rate` をリアルタイム監視。

### ベストプラクティス

- **段階的なルール適用**: 初回は `threshold=0.9` に設定し、誤フィルタリングを最小限に抑えてから、1週間後に `threshold=0.8` に下げる。これにより、ユーザーのフィードバックループを構築し、`priority_score` の精度を向上させる。
- **定期的なインデックスメンテナンス**: PostgreSQL の `VACUUM ANALYZE` を週次で実行し、`emails` テーブルの `priority_score` インデックスを再構築。これにより、クエリ応答時間を `avg_query_time < 50ms` に維持できる。
