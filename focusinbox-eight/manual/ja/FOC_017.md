# focusinbox-eight - FOC_017

## 概要

focusinbox-eight は、メール受信トレイのノイズをリアルタイムでフィルタリングし、重要度スコアに基づいて自動分類・優先順位付けを行う次世代型メール管理エンジンです。特に大規模なB2Bコミュニケーション環境において、1日あたり10,000件以上の着信メールを処理し、ユーザーが真に注力すべきメッセージのみを抽出します。

## 詳細な内容

### focusinbox-eight とは？

focusinbox-eight は、機械学習ベースの意図推定エンジンと、PostgreSQL による高スループットなイベントストアを組み合わせた、エンタープライズ向けメールインテリジェンスプラットフォームです。従来のルールベースのフィルタとは異なり、送信者の過去の行動パターン、返信確率、スレッドの緊急度を動的に学習し、ユーザーごとに最適化された受信トレイを構築します。

## コア機能

- **リアルタイム優先度スコアリングエンジン**  
  各メールに対し、`priority_score`（0.0〜1.0）をミリ秒単位で算出。`urgency_factor`（送信時刻からの経過時間とスレッド深度に基づく）、`sender_reputation_index`（過去の返信率と返信速度から計算）を掛け合わせ、閾値 `0.75` 以上のメールのみを「フォーカスビュー」に表示。PostgreSQL の BRIN インデックスを活用し、1,000万件以上の履歴から高速にスコアを再計算。

- **コンテキスト依存の自動ラベルリング**  
  メール本文と添付ファイルのテキストを形態素解析し、`context_tags`（例：`["urgent_contract", "legal_review", "client_escalation"]`）を自動生成。各タグには信頼度 `tag_confidence`（0.0〜1.0）が付与され、`0.9` 以上のタグはユーザー承認なしで自動分類される。タグ定義は YAML ベースの設定ファイルでカスタマイズ可能。

- **マルチテナント対応のスレッド再構築エンジン**  
  `thread_id` を `message_id` と `references` ヘッダから動的に生成し、同一スレッド内のメールをグループ化。`thread_depth` が 5 を超えるスレッドは自動的に「高緊急スレッド」としてフラグ付けされ、`escalation_threshold`（デフォルト 30 分）以内に応答がない場合、Slack や Teams への通知をトリガーする。

## 使い方

1. **初期設定とデータソースの接続**  
   `config/focusinbox.yml` に IMAP/SMTP サーバー情報を記述。`imap_host`、`imap_port`（993 推奨）、`auth_type`（`OAUTH2` または `PLAIN`）を指定。`sync_interval`（デフォルト 300 秒）でポーリング間隔を設定し、`max_batch_size`（デフォルト 500）で1回の同期で取得するメール数を制限。接続確認は `focusinbox validate-connection` コマンドで実行。

2. **優先度スコアリングの閾値調整**  
   `focusinbox threshold set --priority 0.80 --urgent-only` を実行し、フォーカスビューに表示するメールの基準を調整。`--dry-run` フラグを付けると、実際のフィルタリングを行わずに影響を受けるメール数をプレビュー可能。`focusinbox stats --distribution` で現在のスコア分布をヒストグラム表示し、最適な閾値を決定。

3. **自動ラベルリングのカスタムルール追加**  
   `rules/custom_tags.yaml` に正規表現パターンとタグ名を定義。例：  
   ```yaml
   - pattern: "緊急|至急|ASAP"
     tag: "urgent"
     confidence: 0.95
   - pattern: "契約書|署名|サイン"
     tag: "contract"
     confidence: 0.90
   ```  
   その後 `focusinbox rules reload` でルールをホットロード。`focusinbox rules test --message-id <ID>` で特定メールに対するタグ適用結果を検証。

## ベストプラクティス

- **段階的な閾値最適化**  
  導入初週は `priority_score` の閾値を `0.60` に設定し、フォーカスビューに表示されるメール数を観察。週次で `focusinbox stats --false-positive-rate` を確認し、誤って除外されたメールの割合が 5% を超える場合は閾値を 0.05 ずつ引き下げる。理想的には `0.75` 〜 `0.85` の範囲に収束させる。

- **スレッド再構築の定期メンテナンス**  
  月に一度、`focusinbox threads rebuild --start-date 2024-01-01` を実行し、過去のスレッド構造を再計算。特に `references` ヘッダが欠損しているメールが多い環境では、`--enable-fuzzy-match` フラグを有効にし、件名と送信者アドレスの類似度（Levenshtein距離 0.85 以上）で補完する。
