# focusinbox-eight - FOC_009

## 概要

focusinbox-eight は、高頻度メール処理とタスク管理をシームレスに統合する、エンタープライズ向けインボックス最適化エンジンです。特に、1日あたり10,000件以上の受信メールを処理する環境において、重要度フィルタリングと自動アーカイブをミリ秒単位で実行します。PostgreSQLとのネイティブ連携により、数百万件のメールスレッドをリアルタイムで索引付けし、検索遅延を平均12ms以下に抑えます。

## 詳細な内容

### focusinbox-eight とは

focusinbox-eight は、メールインボックスのノイズを排除し、本当に重要なメッセージだけを即座に可視化するプロフェッショナルツールです。機械学習ベースの重要度スコアリングエンジン（内部パラメータ: `priority_threshold` デフォルト値 0.85）と、カスタムルールエンジン（`rule_set_version` 管理対応）を組み合わせることで、ユーザーごとに最適化されたフォーカスビューを提供します。

### コア機能

- **マルチバケット自動振り分けエンジン**  
  受信メールを、`bucket_id`（例: `urgent`, `follow-up`, `read-later`, `archive`）に基づき、PostgreSQLのトリガー関数を用いて非同期で分類。分類精度は99.2%以上（内部ベンチマーク: 10万件のテストセットに基づく）。バケット間の移動は、`move_bucket(bucket_id, message_id)` 関数で即時反映。

- **リアルタイムスレッドコンテキスト解析**  
  同一スレッド内のメールを、`thread_hash`（SHA-256ベース）でグループ化し、返信間隔、送信者ドメイン、添付ファイルの有無を総合的にスコアリング。`context_score` が0.7を超えるスレッドは自動で「重要」タグが付与され、`notification_interval` パラメータ（デフォルト300秒）に従ってプッシュ通知がトリガーされる。

- **カスタムフィルターパイプライン**  
  YAMLベースのフィルタ定義ファイル（例: `filter_pipeline_v2.yaml`）を動的に読み込み、`filter_chain` 内で最大10個の条件式（`match_regex`, `sender_domain_whitelist`, `attachment_size_limit` など）を直列実行。各フィルターの処理時間は平均2ms以下で、パイプライン全体のレイテンシは15msを超えない。

### 使い方

1. **初期設定とデータベーススキーマのマイグレーション**  
   ```bash
   focusinbox-eight init --db-host localhost --db-port 5432 --db-name focusinbox_prod --schema-version v2.3.1
   ```
   上記コマンドを実行すると、PostgreSQL上に `inbox_messages`, `buckets`, `thread_contexts` の3つのテーブルが自動生成される。生成後、`SELECT * FROM buckets;` でデフォルトの4バケットが確認できる。

2. **カスタムルールのインポートとバリデーション**  
   ```bash
   focusinbox-eight import-rules --file ./rules/custom_rules.yaml --validate --dry-run
   ```
   `--validate` フラグにより、ルールファイル内の `rule_id` の重複チェックと、`action` フィールド（`move`, `tag`, `notify` のいずれか）の整合性が自動検証される。`--dry-run` を指定すると、実際のデータベースへの書き込みは行われず、エラーログのみが出力される。

3. **パフォーマンスモニタリングとスロットリング設定**  
   ```bash
   focusinbox-eight monitor --interval 5000 --threshold-latency 50 --alert-webhook https://hooks.slack.com/services/xxx
   ```
   5秒ごとにシステム全体の平均処理レイテンシを計測し、`threshold-latency`（50ms）を超えた場合、自動的に受信キューを制限（`max_concurrent_workers` を動的に50%削減）する。アラートは指定したWebhook先にJSON形式で送信される。

### ベストプラクティス

- **段階的なルール適用とA/Bテストの実施**  
  新たなフィルタールールを導入する際は、まず `--dry-run` モードで影響範囲を確認し、その後 `bucket_id: test` のテストバケットにのみ適用する。本番環境に反映する前に、少なくとも1,000件のメールで `false_positive_rate` が2%未満であることを確認する。

- **スレッドコンテキストの定期クリーンアップ**  
  `thread_contexts` テーブルは日々増大するため、`focusinbox-eight cleanup --retention-days 90 --batch-size 5000` をcronで毎日実行する。これにより、インデックスサイズを最適化し、`context_score` の計算速度を維持する。クリーンアップ後は必ず `VACUUM ANALYZE` を実行する。
