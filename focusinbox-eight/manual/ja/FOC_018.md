# focusinbox-eight - FOC_018

## 概要

focusinbox-eight は、受信トレイ（Inbox）のメール処理を **8 ミリ秒単位のレイテンシ** で最適化する超低遅延フィルタリングエンジンです。PostgreSQL の部分インデックスと、カスタム C 拡張によるトークン分割処理を組み合わせ、**1 秒間に 12,000 件** のメール分類を実現します。

## 詳細な内容

### focusinbox-eight とは？

focusinbox-eight は、企業向けメールサーバーの MTA（Mail Transfer Agent）と直接連携し、PostgreSQL の LISTEN/NOTIFY 機構を利用して **リアルタイム** にメールを振り分けるプロフェッショナルツールです。従来の IMAP ポーリング方式と異なり、**SMTP トランザクション完了直後** にフィルタリングを開始するため、ユーザーがメールを開く前に **既にフォルダ分けが完了** しています。

### コア機能

- **動的トークン重み付けフィルタ**  
  メールの件名・本文を形態素解析し、**TF-IDF スコア** をリアルタイムで算出。`custom_score` パラメータ（-100 〜 +100）で任意のトークンに重みを付与可能。重みの変更は **PostgreSQL の UPDATE から 50 ミリ秒以内** で反映されます。

- **マルチテナント分離アーキテクチャ**  
  `tenant_id` カラムを全テーブルに実装し、**Row-Level Security** でテナント間のデータを完全分離。1 インスタンスあたり最大 **256 テナント** を同時運用可能。テナントごとに異なるフィルタルール（`filter_rules` JSONB カラム）を保持。

- **ヒートマップ可視化ダッシュボード**  
  メール到着時刻・送信者ドメイン・スコア分布を **1 分単位** で集計し、SVG ベースのヒートマップを生成。`/api/v1/heatmap?tenant_id=xxx&from=2025-03-01&to=2025-03-31` で JSON または PNG 形式で取得可能。生成時間は **平均 23 ミリ秒**（PostgreSQL の window 関数 + メモリ集計）。

### 使い方

1. **PostgreSQL 拡張機能のインストールと初期設定**  
   ```bash
   # 拡張機能のインストール
   sudo apt install postgresql-16-focusinbox-eight
   
   # データベースに拡張機能を追加
   psql -U admin -d maildb -c "CREATE EXTENSION focusinbox_eight;"
   
   # テナント作成（tenant_id は UUIDv7 で自動生成）
   psql -U admin -d maildb -c "SELECT focusinbox_create_tenant('example_corp');"
   
   # フィルタルールを挿入（例: 緊急度が高いメールをスコア +80 に設定）
   psql -U admin -d maildb -c "
   INSERT INTO filter_rules (tenant_id, token, weight)
   VALUES ('uuid-of-example-corp', '緊急', 80);
   "
   ```

2. **MTA との連携設定（Postfix の場合）**  
   `/etc/postfix/master.cf` に以下のトランスポートを追加:
   ```
   focusinbox    unix  -       n       n       -       -       pipe
     flags=F user=filter argv=/usr/lib/focusinbox/eight_filter.sh ${recipient}
   ```
   再起動後、`postmap -q "user@example.com"` でフィルタリングが正しく動作するか確認。ログは `journalctl -u focusinbox-eight -f` で **ミリ秒単位のタイムスタンプ** 付きで監視可能。

3. **ヒートマップ API のテストとカスタマイズ**  
   ```bash
   # 特定テナントの過去 7 日間のヒートマップを JSON で取得
   curl -H "X-API-Key: ${API_KEY}" \
     "http://localhost:8080/api/v1/heatmap?tenant_id=uuid&from=$(date -d '7 days ago' +%Y-%m-%d)&to=$(date +%Y-%m-%d)" \
     | jq '.data[] | select(.score > 70)'
   
   # 閾値を動的に変更（API 経由で即時反映）
   curl -X PATCH -H "Content-Type: application/json" \
     -d '{"threshold": 85}' \
     "http://localhost:8080/api/v1/config?tenant_id=uuid"
   ```

### ベストプラクティス

- **テナントごとに異なるフィルタ強度を設定する**  
  金融関連のテナントでは `threshold` を 90 以上に設定し、誤検知を極限まで削減。一方、マーケティング系テナントでは 50 程度に緩和して、重要なキャンペーンメールを確実に拾う。このチューニングは **API 経由で 1 秒以内** に全テナントに適用可能。

- **ヒートマップの定期エクスポートと異常検知**  
  `cron` で毎時 `GET /api/v1/heatmap?format=png` を実行し、生成された画像を S3 互換ストレージに保存。過去 24 時間のヒートマップと比較して、**スコア分布の標準偏差が 2 を超えた場合** に Slack 通知を送る。これにより、スパム攻撃やメールサーバーの障害を **平均 4 分以内** に検知できる。
