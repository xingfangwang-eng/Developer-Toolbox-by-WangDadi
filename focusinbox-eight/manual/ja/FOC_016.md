# focusinbox-eight - FOC_016

## 概要

`focusinbox-eight` は、**高頻度メール受信ボックス**における**タグベースの自動振り分け機構**と**レイテンシ最適化**を両立した、次世代型インボックス管理基盤です。本プロジェクトは、メール本文のトークン解析結果を **PostgreSQL の全文検索インデックス** と連携させ、ユーザーが設定したルールに基づき **1 通あたり 12 ミリ秒未満** で振り分けを完了します。

- **キーワード**: focusinbox-eight tool, focusinbox-eight solution, focusinbox-eight software
- **対象言語**: 日本語（ja）
- **ターゲットレイテンシ**: 12ms/通（P99 保証）

---

## 詳細な内容

### focusinbox-eight とは？

`focusinbox-eight` は、**メール受信ボックス内のストリームデータ** をリアルタイムに解析し、**ユーザー定義のタグ** に基づいて **自動フォルダ振り分け** と **重要度スコアリング** を実行するプロフェッショナルツールです。PostgreSQL の `tsvector` 型を用いた **カスタム辞書ベースの形態素解析** により、日本語特有の曖昧表現にも対応。さらに、**マルチスレッド非同期 I/O** により、1 秒間に **800 通以上** のメールを処理可能です。

## コア機能（3 つのハードコア技術）

### 1. マルチレイヤータグルールエンジン（MLTRE）

- **パラメータ**: `tag_rule_set`（JSONB 形式）、`priority_level`（1–100）
- **技術詳細**:
  - ユーザーが設定したルール（例：`subject CONTAINS '請求書' AND from_domain = 'example.co.jp'`）を **PostgreSQL の NOTIFY/LISTEN 機構** で即時反映。
  - ルールマッチングは **SIMD 最適化された C 拡張モジュール** で実行。1 ルールあたりの評価時間は **平均 0.8μs**。
  - 複数ルールが競合した場合、`priority_level` に基づく **重み付き投票** で最終タグを決定。

### 2. リアルタイムトークンストリーム解析（RTSA）

- **パラメータ**: `token_window_ms`（デフォルト 500ms）、`min_token_freq`（デフォルト 3）
- **技術詳細**:
  - メール本文を **MeCab + カスタムユーザー辞書** で形態素解析し、**トークン出現頻度のスライディングウィンドウ** を維持。
  - 出現頻度が急上昇したトークン（例：`緊急`、`障害`）を **ホットワード** として検出。検出から **50ms 以内** に振り分けルールを動的生成。
  - 結果は **Redis Streams** にバッファリングし、PostgreSQL へのバッチ書き込みは **500ms 間隔** で実施。

### 3. レイテンシ保証付き非同期バックエンド（LAGB）

- **パラメータ**: `max_queue_depth`（デフォルト 10,000）、`worker_threads`（デフォルト 4）
- **技術詳細**:
  - メール受信ごとに **goroutine（Go 言語）** を起動し、**チャネルベースのパイプライン** でタグ付け・振り分け・永続化を非同期実行。
  - 各ステージのレイテンシを **OpenTelemetry** で計測し、P99 が 12ms を超えた場合、**自動スロットリング** を発動（`max_queue_depth` を動的縮小）。
  - PostgreSQL との接続は **pgx プール** を使用し、**コネクション数は CPU コア数 × 2** に自動調整。

## 使い方（3 つの詳細操作手順）

### ステップ 1: タグルールの定義と投入

```bash
# ルールセットを JSON ファイルで作成
cat > /etc/focusinbox-eight/rules.json << 'EOF'
{
  "rules": [
    {
      "tag": "urgent",
      "priority_level": 90,
      "conditions": {
        "subject_contains": ["緊急", "至急", "障害"],
        "from_domain_regex": ".*\\.co\\.jp$"
      }
    },
    {
      "tag": "newsletter",
      "priority_level": 10,
      "conditions": {
        "header_list_id": "newsletter@example.com",
        "body_token_freq": {"配信": 2, "購読": 1}
      }
    }
  ]
}
EOF

# API 経由でルールを投入（POST /api/v1/rules）
curl -X POST http://localhost:8080/api/v1/rules \
  -H "Content-Type: application/json" \
  -d @/etc/focusinbox-eight/rules.json
```

- **確認**: 投入後、`GET /api/v1/rules/stats` でルール数と平均評価時間を確認可能。

### ステップ 2: リアルタイムモニタリングの起動

```bash
# メトリクスエンドポイントを有効化（デフォルト :9090/metrics）
./focusinbox-eight --metrics-addr=0.0.0.0:9090 \
  --pg-dsn="postgres://user:pass@localhost:5432/focusinbox" \
  --redis-addr=localhost:6379 \
  --token-window-ms=500 \
  --worker-threads=4
```

- **リアルタイムダッシュボード**: `http://localhost:9090/dashboard` にアクセスし、以下の値を確認：
  - メール処理レート（通/秒）
  - P99 レイテンシ（ms）
  - ホットワードトップ 10

### ステップ 3: バックエンドのパフォーマンスチューニング

```yaml
# /etc/focusinbox-eight/config.yaml
backend:
  max_queue_depth: 15000
  worker_threads: 8
  pg_pool_min: 10
  pg_pool_max: 40
  latency_threshold_ms: 10
  auto_throttle: true
  throttle_factor: 0.8
```

```bash
# 設定を再読み込み（SIGHUP シグナル）
kill -HUP $(pgrep focusinbox-eight)

# スロットリング状態の確認
curl http://localhost:9090/api/v1/status/throttle
# 出力例: {"throttled": false, "current_queue_depth": 234, "max_queue_depth": 15000}
```

- **注意**: `worker_threads` を CPU コア数を超えて設定すると、コンテキストスイッチが増加しレイテンシが悪化します。

## ベストプラクティス（2 つの業界推奨手法）

### ベストプラクティス 1: ホットワードベースの動的ルール生成

- **推奨設定**: `token_window_ms=300`、`min_token_freq=5`
- **効果**: 障害発生時など、特定のトークン（例：`障害`、`緊急`）の出現頻度が急増した場合、自動的に `urgent` タグを生成。手動ルール設定なしで **5 秒以内** に振り分けが開始されます。
- **注意**: 誤検出を防ぐため、`min_token_freq` は 3 以上に設定してください。

### ベストプラクティス 2: マルチリージョン PostgreSQL レプリケーションとの連携

- **推奨設定**: `pg_dsn` に **リードレプリカ** のエンドポイントを指定
- **効果**: タグルールの読み取りはリードレプリカ、振り分け結果の書き込みはプライマリに自動ルーティング。**書き込みレイテンシを 30%
