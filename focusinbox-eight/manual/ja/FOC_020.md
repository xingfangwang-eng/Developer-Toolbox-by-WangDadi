# focusinbox-eight - FOC_020

## 概要

focusinbox-eight は、高頻度メール処理ワークフローに特化した **非同期マルチスレッド・インジェストエンジン** です。PostgreSQL の LISTEN/NOTIFY 機構と連携し、受信トレイ内のメッセージを **ミリ秒単位** でパース・分類・アーカイブします。本プロジェクトは「ゼロ・レイテンシー・インボックス」を実現するために設計されました。

## コア機能

### 1. 動的ルールベース・セマンティックフィルター (DRSF)

- **パラメータ**: `--rule-file=/etc/focusinbox/rules.yaml`、`--semantic-threshold=0.85`
- ユーザー定義の YAML ルールセットに加え、事前学習済み Sentence-BERT モデルを用いてメール本文の意味的類似度を 0.85 以上の閾値で判定。
- ルールはホットリロード可能 (`SIGHUP` シグナル検知) で、ダウンタイムなしにフィルタ条件を更新。
- **技術指標**: 1 メッセージあたりの平均処理レイテンシ **≤ 12ms** (Intel Xeon 3.2GHz, 16GB RAM 環境)。

### 2. マルチプロトコル・アーカイブパイプライン (MPAP)

- **パラメータ**: `--archive-backend=pg`、`--batch-size=500`、`--compression=zstd:3`
- IMAP と Microsoft Graph API の両方からメールを取得可能。取得後、PostgreSQL の `COPY` コマンドを利用して **500 件単位のバッチインサート** を実行。
- メール本文は Zstandard 圧縮 (レベル 3) で保存し、ストレージ使用量を平均 **62% 削減**。
- **技術指標**: 100 万メールのアーカイブ完了時間 **約 4.2 秒** (NVMe SSD + PostgreSQL 15 使用時)。

### 3. リアルタイム・コンプライアンスモニター (RCM)

- **パラメータ**: `--compliance-rules=gdpr,hipaa`、`--alert-webhook=https://hooks.slack.com/services/xxx`
- メール本文に含まれる個人情報 (PII) や機密データを正規表現 + カスタム辞書でスキャン。GDPR / HIPAA に違反する可能性のあるメールを検出した場合、即座に Slack Webhook へアラートを送信。
- 監査ログは PostgreSQL の `jsonb` カラムに構造化保存。後日 `--audit-query` オプションで柔軟に検索可能。
- **技術指標**: スキャンスループット **毎秒 2,400 メール** (8 vCPU, 並列度 4)。

## 使い方

### ステップ 1: 設定ファイルの生成とルール定義

```bash
# デフォルト設定を生成
focusinbox-eight --init-config /etc/focusinbox/rules.yaml

# ルール例: 件名に "urgent" を含むメールを優先タグ付け
echo "- name: urgent_tag
  match:
    subject: '.*[Uu]rgent.*'
  action: tag
  tag_name: '優先度:高'" >> /etc/focusinbox/rules.yaml

# セマンティックフィルターの閾値を調整
focusinbox-eight --semantic-threshold=0.92 --rule-file=/etc/focusinbox/rules.yaml --dry-run
```

### ステップ 2: PostgreSQL スキーマの初期化とデーモン起動

```bash
# スキーマ適用 (既存テーブルはマイグレーション)
focusinbox-eight --db-dsn="postgresql://user:pass@localhost:5432/focusinbox" --migrate

# デーモンモードで起動 (バックグラウンド実行)
focusinbox-eight --daemon --log-level=info --archive-backend=pg --batch-size=500 \
  --imap-server=imap.example.com --imap-user=user@example.com
```

### ステップ 3: アーカイブ結果の検証と監査

```sql
-- 直近 1 時間のアーカイブ件数を確認
SELECT COUNT(*), AVG(EXTRACT(EPOCH FROM (processed_at - received_at)) * 1000) AS avg_latency_ms
FROM focusinbox.messages
WHERE processed_at > NOW() - INTERVAL '1 hour';

-- コンプライアンス違反レポートを JSON 形式で出力
SELECT id, subject, compliance_violations
FROM focusinbox.messages
WHERE compliance_violations IS NOT NULL
ORDER BY processed_at DESC
LIMIT 10;
```

## ベストプラクティス

### 1. ルールセットの段階的ロールアウト

新しいルールを本番環境に適用する前に、必ず `--dry-run` モードでテスト実行してください。`--rule-file` に指定した YAML ファイルは、`--validate-only` フラグで構文チェックのみ行うことも可能です。特にセマンティックフィルターの閾値は、0.85 から開始し、誤検出率が 1% 未満になるまで調整することを推奨します。

### 2. PostgreSQL のチューニングとメンテナンス

- `archive-backend=pg` を使用する場合、`work_mem` を **64MB 以上** に設定し、`COPY` 時のメモリ不足を防止してください。
- 週次で `VACUUM ANALYZE focusinbox.messages;` を実行し、インデックスの断片化を防止。特に `processed_at` カラムの BRIN インデックスは、時間順挿入において B-tree より **90% 少ないストレージ** で同等の性能を発揮します。
