# focusinbox-eight - FOC_010

## 概要

focusinbox-eight は、メールインボックスを高精度かつ高頻度で監視し、特定条件下で即座に外部システムへ通知・連携を行うためのリアルタイムインボックス監視エンジンです。特に、受信メール内の特定キーワード、送信者ドメイン、添付ファイルの有無などを複合条件として判定し、ミリ秒単位のトリガー制御を実現します。

## 詳細な内容

### focusinbox-eight とは

focusinbox-eight は、PostgreSQL 上のイベントストアと連携し、IMAP または Microsoft Graph API 経由でインボックスをポーリングします。ポーリング間隔はユーザー定義可能で、最短 100 ミリ秒の高頻度監視が可能です。受信メールのメタデータを解析し、設定されたルールに基づいて Slack、Webhook、カスタム API エンドポイントへのペイロード送信を実行します。

### コア機能

- **マルチ条件トリガーエンジン**  
  送信者アドレス、件名の正規表現、添付ファイルの MIME タイプ、メールサイズ（バイト単位）を組み合わせた複合条件を定義可能。AND/OR/NOT 演算子を用いた条件ツリーを JSON で記述し、リアルタイムに評価。評価結果は 50 ミリ秒以内に返却。

- **イベント永続化とリプレイ機能**  
  すべての受信イベントは PostgreSQL の `inbox_events` テーブルに保存。トリガー条件に合致しなかったメールも含めて全履歴を保持。障害発生時には、指定したタイムスタンプ範囲のイベントを再実行可能。

- **レート制御付き外部連携アダプター**  
  Slack Incoming Webhook、SendGrid、カスタム HTTP エンドポイントへの送信時に、トークンバケット方式によるレート制限を適用。1 秒あたりの最大リクエスト数（例: 10 req/sec）を設定可能で、バースト時もシステム負荷を一定に保つ。

### 使い方

1. **初期セットアップとデータベースマイグレーション**  
   プロジェクトルートで `./migrate up` を実行し、PostgreSQL に必要なテーブル（`inbox_events`、`trigger_rules`、`webhook_endpoints`）を作成。`config.yaml` に IMAP サーバー情報（host、port、use_tls）、ポーリング間隔（poll_interval_ms）、データベース接続文字列を設定。

2. **トリガールールの登録**  
   `POST /api/v1/rules` エンドポイントに以下の JSON を送信し、条件を定義。  
   ```json
   {
     "name": "urgent_support",
     "conditions": {
       "and": [
         {"field": "from_domain", "operator": "equals", "value": "support.example.com"},
         {"field": "subject", "operator": "regex", "value": "緊急|障害|ダウン"},
         {"field": "has_attachment", "operator": "equals", "value": true}
       ]
     },
     "action": {
       "type": "webhook",
       "endpoint_id": "wh_001",
       "payload_template": "{{.Subject}} - {{.From}}"
     }
   }
   ```

3. **監視の開始とログ確認**  
   `./focusinbox-eight start` を実行すると、設定されたポーリング間隔で監視が開始。標準出力および `logs/` ディレクトリに JSON 形式のログが出力される。各イベントの処理時間（processing_time_ms）とトリガー判定結果（triggered: true/false）が記録される。

### ベストプラクティス

- **ポーリング間隔の適正化**  
  監視対象のメールボックスが高頻度で受信する場合、ポーリング間隔は 500 ミリ秒以上に設定し、IMAP サーバーへの負荷を軽減。低頻度の場合は 100 ミリ秒でも問題ないが、サーバーのレート制限に抵触しないよう事前に確認すること。

- **トリガールールのバージョン管理**  
  ルール定義はすべて `trigger_rules` テーブルで管理され、変更履歴は自動で記録されない。そのため、ルール変更前には必ず `GET /api/v1/rules/{id}` で現在の定義をエクスポートし、バックアップとして保存することを推奨。CI/CD パイプラインでルール定義を管理する場合は、YAML ファイルで一元管理し、マイグレーションとして適用する。
