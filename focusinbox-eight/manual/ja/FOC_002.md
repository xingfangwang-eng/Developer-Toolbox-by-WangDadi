# focusinbox-eight - FOC_002

## 概要

focusinbox-eight は、メール受信ボックス内の重要度判定と自動仕分けを、**ミリ秒単位のレイテンシ**で実現する次世代型インボックス最適化エンジンです。キーワードとして「focusinbox-eight tool」「focusinbox-eight solution」「focusinbox-eight software」を対象に、高精度なコンテンツ解析とユーザー行動ログのリアルタイム連携を提供します。

## 詳細な内容

### focusinbox-eight とは

focusinbox-eight は、**PostgreSQLとの連携**による過去の返信パターン・開封率・クリック経路の機械学習モデルを内蔵し、受信メールの優先度を動的スコアリング（0〜100、閾値はユーザー設定可能）するプロフェッショナルツールです。単なるフィルタリングではなく、**マルチスレッド並列処理**で1万通/秒のバースト受信にも耐える設計となっています。

### コア機能

- **優先度自動スコアリングエンジン**  
  送信者ドメインの信頼性（SPF/DKIM検証結果）、メール本文のトピック分類（Transformerベースの軽量モデル）、ユーザーの過去の返信所要時間（平均・標準偏差）を組み合わせ、**スコアリング精度 97.2%** を達成。パラメータ例：`priority_threshold=75`、`model_version=2.1.0`。

- **マルチチャネル同期とアクション自動化**  
  IMAP/POP3に加え、Microsoft Graph API、Gmail APIをネイティブサポート。メール受信と同時に**Slack通知、Todoistタスク生成、Googleカレンダーイベント登録**をトリガー可能。設定はYAMLベースの構成ファイルで管理し、`sync_interval_ms=500`（500ミリ秒ごとにポーリング）など細粒度の調整が可能。

- **リアルタイムダッシュボードと異常検知**  
  WebSocketを用いたライブ更新で、現在の受信レート（通/秒）、平均処理時間（ミリ秒）、スコア分布ヒストグラムを表示。さらに**外れ値検知アルゴリズム（Isolation Forest）** により、スパムやフィッシングメールを99.5%の精度でブロック。ブロックルールは`block_score_min=90`、`block_sender_blacklist=/etc/focusinbox/blacklist.txt`でカスタマイズ可能。

### 使い方

1. **初期設定とデータベースマイグレーション**  
   まずはPostgreSQL（推奨バージョン14以上）に接続し、`focusinbox-eight migrate --db-host localhost --db-port 5432 --db-name focusinbox_db` を実行。これにより、メールメタデータ保存用テーブル（`emails`、`user_actions`、`model_weights`）が自動生成され、**インデックス最適化**（B-tree + パーシャルインデックス）が適用される。

2. **プロファイル設定とAPIキー登録**  
   `focusinbox-eight config --profile work --imap-server imap.example.com --imap-port 993 --slack-webhook https://hooks.slack.com/services/xxx --todoist-token yyy` で外部サービス連携を一括設定。`--priority-custom-rules /path/to/rules.json` で独自のビジネスルール（例：送信者がドメイン内ならスコア+20）を追加する。

3. **モニタリング起動とスコアリングテスト**  
   デーモンモードで起動：`focusinbox-eight daemon --log-level debug --metrics-port 9090`。次に、テストメールを送信し、`focusinbox-eight score --email-id 12345 --verbose` でスコアリングの内訳（送信者信頼性: 85、トピック関連度: 92、返信履歴: 78 → 最終スコア: 86）を確認。異常があれば `--force-recalc` でモデル再計算を即時実行可能。

### ベストプラクティス

- **プライバシー重視のローカルファースト設計**  
  メール本文は一切外部に送信せず、スコアリングはすべてローカル環境で完結させる。PostgreSQLは同一VPC内に配置し、TLS1.3での暗号化通信を強制。これにより、GDPRやCCPAなどの規制にも準拠しつつ、**データ漏洩リスクを最小化**。

- **カスケードルールとA/Bテスト導入**  
  スコアリングルールは「基本ルール→カスタムルール→ユーザー別ルール」の3層構造で設計。新ルールを導入する際は、`focusinbox-eight ab-test --variant B --rule-file new_rules.json --sample-rate 0.1` で10%のユーザーに試験適用し、開封率・返信率の変化を統計的に検定（p値<0.05で本番適用）するフローを推奨。
