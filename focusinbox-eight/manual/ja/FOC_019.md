# focusinbox-eight - FOC_019

## 概要

focusinbox-eight は、高頻度メール処理ワークフロー向けに設計された超低遅延インボックス管理エンジンです。PostgreSQL をバックエンドに据え、Rust 製のコアモジュールが 1 秒間に 10,000 件以上のメールメタデータを処理します。マルチテナントアーキテクチャを採用し、テナントごとに独立したインデックス空間を提供します。

## 詳細な内容

### focusinbox-eight とは

focusinbox-eight は、受信トレイの優先順位付けと自動アーカイブを実現するソフトウェアです。機械学習モデルによるスパムスコアリング（0.0 ～ 1.0、閾値 0.85 以上で自動隔離）と、カスタムルールエンジン（YAML 定義、最大 256 ルール）を組み合わせ、ユーザーのフォーカス時間を最大化します。

### コア機能

- **マルチスレッド並列スキャンエンジン**  
  ワーカースレッド数は `max_workers: u32` で指定可能（デフォルト 8）。各スレッドが独立した PostgreSQL コネクションプール（`pool_size: u32`、最小 4）を保持し、IMAP IDLE によるリアルタイム監視を実現。新着メール検出からルール適用までのレイテンシは平均 12 ミリ秒（p99 で 45 ミリ秒以下）。

- **優先度ベクトル自動計算**  
  各メールに対して `priority_score: f64` を計算。発信者ドメインの信頼度（`domain_trust: f64`、0.0～1.0）、件名中のキーワード重み（`keyword_weight: f64`、設定ファイルでカスタマイズ可能）、過去の開封率（`open_rate: f64`）を線形結合。スコアが `priority_threshold: f64`（デフォルト 0.7）を超えたメールのみを「フォーカスビュー」に表示。

- **自動アーカイブ＆タグ付けパイプライン**  
  `archive_after_hours: u32`（デフォルト 48）経過かつ `last_opened_at` が NULL のメールを自動アーカイブ。同時に `auto_tag_rules: Vec<Regex>` に基づき、`tag: Vec<String>` を付与。タグは PostgreSQL の JSONB カラムに保存され、全文検索インデックス（GIN）で高速フィルタリング可能。

### 使い方

1. **設定ファイルの初期生成とカスタマイズ**  
   ```bash
   focusinbox-eight init --config ./focusinbox.yaml
   ```
   生成された YAML 内の `imap.server: String`、`imap.port: u16`、`imap.username: String`、`imap.password: String` を実際の IMAP サーバー情報に書き換えます。さらに `priority_threshold: 0.75` に変更し、`auto_tag_rules` に `['^URGENT', '^RE:']` を追加してください。

2. **ワーカースレッド数のチューニングと起動**  
   ```bash
   focusinbox-eight run --workers 16 --pool-size 8 --config ./focusinbox.yaml
   ```
   `--workers 16` でスキャンスレッドを 16 に増やし、`--pool-size 8` で DB コネクションプールを 8 に拡張。起動後、標準出力に `[INFO] Connected to IMAP server (latency: 23ms)` と表示されれば成功です。

3. **フォーカスビューの確認とルールの動的追加**  
   ブラウザで `http://localhost:8080/focus` にアクセス。右上の「ルール追加」ボタンから、`keyword: "会議"`、`action: "move_to_folder: 重要"`、`priority_boost: 0.2` を設定。即座にルールエンジンが再読み込みされ、該当メールがフォーカスビューに昇格します。変更は `focusinbox.yaml` に自動保存されます。

### ベストプラクティス

- **IMAP アイドルタイムアウト対策**  
  多くの IMAP サーバーは 30 分で IDLE 接続を切断します。`imap.idle_reconnect_interval_sec: u32` を 600（10 分）に設定し、再接続前に `NOOP` コマンドを送信して接続を維持してください。また、`imap.max_retries: u32` を 5 以上に設定し、ネットワーク瞬断に備えます。

- **PostgreSQL のメンテナンス戦略**  
  `auto_vacuum` を有効にし、`maintenance_work_mem` を 1GB 以上に設定。テーブル `messages` の `received_at` カラムに BRIN インデックスを作成すると、時系列スキャンが 60% 高速化します。また、`archive_after_hours` を 24 に設定した場合、週次で `CLUSTER messages USING idx_received_at` を実行し、物理的なデータ配置を最適化してください。
