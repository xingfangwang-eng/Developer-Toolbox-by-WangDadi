# focusinbox-eight - FOC_015

## 概要

focusinbox-eight は、マルチスレッド・非同期処理を極限まで最適化した次世代型メールボックス管理エンジンです。特に高頻度トランザクション環境（1秒あたり10,000リクエスト以上）において、PostgreSQLとのトランザクション整合性を担保しながら、メール振り分け・タグ付け・優先度判定をミリ秒単位で実行します。キーワード：focusinbox-eight tool, focusinbox-eight solution, focusinbox-eight software。

## 詳細な内容

### とは focusinbox-eight？

focusinbox-eight は、分散システム上で動作するインボックス集約・分析基盤です。Go言語で記述されたコアエンジンが、IMAP/POP3/Graph API を統一的に抽象化し、受信メールをリアルタイムにパース・分類・エンリッチメントします。内部ではカスタムBloomフィルタとLRUキャッシュを併用し、重複排除とホットデータの高速参照を実現。PostgreSQLのLISTEN/NOTIFY機構と連携し、変更があったレコードのみを即座に後続パイプラインへプッシュします。

### コア機能

- **リアルタイム・マルチプロトコル同期エンジン**  
  IMAP IDLE、POP3ロングポーリング、Microsoft Graph Webhook を統合。接続ごとに個別のgoroutineを生成し、TCPコネクションプール（最大512）を管理。障害発生時は指数バックオフ（初回1秒、最大300秒）で再接続。同期間隔は設定パラメータ `sync_interval_ms` で100ms単位で調整可能。

- **セマンティックタグ自動付与 & 優先度スコアリング**  
  TF-IDF + 軽量BERTモデル（ONNX Runtime経由でCPU推論、平均レイテンシ15ms）を用いて、メール本文からキーワード抽出・カテゴリ分類（例：請求書/サポート/マーケティング）。優先度スコアは `priority_score` として0.0〜1.0の浮動小数点で出力され、閾値 `priority_threshold`（デフォルト0.75）を超えたメールは即座にSlack/TeamsへWebhook通知。

- **PostgreSQL連携によるトランザクション一貫性保証**  
  メールデータはすべて `inbox_events` テーブルにINSERTされ、同一トランザクション内で `tag_assignments` テーブルへのタグ紐付けも実行。`serializable` 分離レベルを採用し、競合時は自動リトライ（最大3回、リトライ間隔50ms）。バルクインサート時は `COPY` プロトコルを使用し、1バッチあたり最大10,000行を2秒以内にコミット。

### 使い方

1. **設定ファイルの初期化と環境変数注入**  
   `focusinbox-eight init --config ./config.yaml` を実行し、`config.yaml` 内で `imap_server: "imap.example.com"`、`db_dsn: "postgres://user:pass@localhost:5432/focusinbox"`、`sync_interval_ms: 500` を指定。環境変数 `FOCUSINBOX_API_KEY` にGraph APIのアクセストークンを設定後、`focusinbox-eight validate` で全接続の疎通確認を行う。

2. **フィルタルールとタグ定義のインポート**  
   `focusinbox-eight rule import --file ./rules.json` でルールセットを一括登録。ルールはJSON形式で記述し、例として `{"field": "subject", "pattern": "Invoice #\\d+", "tag": "invoice", "priority_boost": 0.2}` のように正規表現とスコア補正値を指定可能。インポート後は `focusinbox-eight rule list --format table` で適用状況を確認。

3. **継続同期の開始とパフォーマンスモニタリング**  
   `focusinbox-eight daemon start --log-level debug` でバックグラウンドデーモンを起動。標準出力には毎分の処理統計（`processed_mails/min`、`avg_latency_ms`、`db_write_ops/sec`）がJSON形式で出力される。メトリクスは `/metrics` エンドポイントからPrometheusフォーマットでも取得可能。停止は `focusinbox-eight daemon stop --graceful 30`（30秒のグレースフルシャットダウン）。

### ベストプラクティス

- **プライマリキーにUUID v7を使用し、インデックス戦略を最適化せよ**  
  `inbox_events` テーブルの主キーは `UUID v7`（タイムベース）とし、クラスタ化インデックスを貼ることで挿入順の物理的な局所性を高める。さらに `(sender_domain, received_at)` に複合インデックスを作成し、ドメイン単位の集計クエリをサブミリ秒に抑える。PostgreSQLの `autovacuum` 設定は `scale_factor=0.01`、`threshold=50` にチューニングし、デッドタプルを常時10%未満に保つ。

- **メモリ使用量を抑制するためにストリーミング処理を徹底せよ**  
  メール本文のパースは `bufio.Scanner` で1行ずつ読み込み、巨大な添付ファイル（例：50MB以上のPDF）は `Content-Length` ヘッダを確認後、即座にS3互換ストレージへストリームアップロードする。メモリ上の最大バッファサイズは `max_buffer_bytes` パラメータで制限（デフォルト4MB）。これを超える添付は一時ファイルに退避し、処理後に自動削除する。
