# focusinbox-eight - FOC_011

## 概要

focusinbox-eight は、受信トレイの優先順位付けとタスク集中管理を極限まで最適化するためのエンタープライズグレードツールです。本プロジェクトは、メール・チャット・プロジェクト管理ツールからの通知を統合し、機械学習ベースの重要度スコアリングにより、ユーザーが真に注力すべきタスクのみを抽出します。レイテンシは平均 120ms 以下、PostgreSQL によるトランザクション管理を前提としています。

## 詳細な内容

### コア機能

- **重要度スコアリングエンジン**  
  過去 90 日間の応答履歴、送信者属性（直属上司・重要顧客など）、およびメール本文内のキーフレーズ（例: `deadline:YYYY-MM-DD`、`urgent`）を解析し、0.0 ～ 1.0 のスコアをリアルタイム算出。スコア閾値は `config.score_threshold` パラメータで調整可能。PostgreSQL の `tsvector` を用いた全文検索と、Redis 上の一時キャッシュにより、1 万件の受信トレイを 800ms で再スコアリング可能。

- **マルチチャネル統合フィルター**  
  IMAP（メール）、Slack API（メッセージ）、Jira Webhook（チケット更新）の 3 系統を同時接続。各チャネルのメッセージは `channel_type`（`email` / `slack` / `jira`）と `source_id` で一意に識別され、同一スレッドの重複排除を自動実行。重複排除ロジックは `dedup_window_ms`（デフォルト 300,000ms）内の類似メッセージをマージする。

- **集中モードタイマー**  
  ユーザーが「集中」を宣言すると、指定時間（`focus_duration_min`、デフォルト 25 分）だけ新規通知のポップアップをブロック。ブロック中に到着したメッセージはキューイングされ、解除後にスコア順で一覧表示。この間、システムは `focus_block_stats` テーブルにブロック開始・終了時刻と処理メッセージ数を記録し、後日の集中度分析に利用する。

### 使い方

1. **初期セットアップとチャネル認証**  
   `config.yaml` に IMAP サーバー情報（`imap_host`、`imap_port=993`、`imap_ssl=true`）、Slack Bot Token（`slack_bot_token`）、Jira API キー（`jira_api_key`）を記述。`focusinbox-eight init` コマンドを実行すると、各チャネルの接続テストが行われ、成功したチャネルのみが `active_channels` リストに追加される。認証失敗時は `auth_error.log` にスタックトレースが出力される。

2. **重要度スコアのカスタマイズ**  
   `scoring_rules.json` を編集し、送信者ドメインごとの重み（`domain_weight`）、キーワードブースト係数（`keyword_boost`）、スコア減衰曲線の半減期（`decay_half_life_days`）を設定。例えば `"domain_weight": {"example.com": 1.5, "competitor.com": 0.3}` と指定すると、example.com からのメールはスコアが 1.5 倍に増幅される。変更後は `focusinbox-eight reload-rules` で即時反映。

3. **集中モードの起動とレポート生成**  
   CLI から `focusinbox-eight focus --duration 30 --tag "sprint-review"` と入力すると、30 分間の集中セッションが開始。終了後、自動的に `focus_report_YYYY-MM-DD_HH-MM-SS.json` が生成され、ブロックされたメッセージ数、処理された重要メッセージ数、スコア分布のヒストグラムが含まれる。このレポートは `POST /api/v1/reports` エンドポイント経由で外部 BI ツールに送信可能。

### ベストプラクティス

- **スコアリングルールの段階的チューニング**  
  初週はデフォルト設定で運用し、`focusinbox-eight stats --period 7d` でスコア分布を確認。誤って低スコアになったメッセージ（偽陰性）が 5% を超える場合は、`keyword_boost` を 0.1 刻みで増加させる。逆に高スコアのノイズが多い場合は `domain_weight` の閾値を 0.8 以上に制限する。このチューニングサイクルを 2 週間ごとに繰り返すことで、精度 95% 以上を維持できる。

- **PostgreSQL のパーティショニング戦略**  
  `messages` テーブルは `received_at` で月単位のパーティションに分割する。`CREATE TABLE messages_2025_01 PARTITION OF messages FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');` のように設定し、古いパーティションは `pg_repack` で定期的に圧縮。これにより、1000 万件超のメッセージでもスコアリングクエリの応答時間を 200ms 未満に抑えられる。また、`focus_block_stats` テーブルには `user_id` と `started_at` に複合インデックスを必ず作成する。
