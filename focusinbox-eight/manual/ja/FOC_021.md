# focusinbox-eight - FOC_021

## 概要

focusinbox-eight は、メール受信トレイの優先順位付けとタスクフォーカスを極限まで最適化する、次世代型インボックス管理エンジンです。本プロジェクトは、受信メッセージのコンテキストスコアリングとリアルタイムスケジューリングを中核とし、ユーザーが「今やるべき一件」にのみ集中できる環境を提供します。PostgreSQL による永続化と、ミリ秒単位のレイテンシで動作するイベント駆動型アーキテクチャを採用しています。

## 詳細な内容

### focusinbox-eight とは

focusinbox-eight は、受信メール・通知・タスクを統合的に解析し、**フォーカススコア**（0.0〜1.0）を動的に算出するプロフェッショナルツールです。スコアリングエンジンは、送信者の関係性、返信履歴の深さ、期限までの残り時間、メール本文の感情分析（ポジティブ/ネガティブ/緊急度）を重み付けし、ユーザーが設定した**集中時間帯**（例: 09:00–12:00 JST）に最も高いスコアのメッセージのみを表示します。これにより、ノイズを排除し、認知負荷を最大 40% 削減します。

### コア機能

- **コンテキストスコアリングエンジン**  
  各メッセージに対し、`score = (sender_weight × 0.4) + (urgency_seconds / 86400 × 0.3) + (sentiment_polarity × 0.2) + (thread_depth × 0.1)` をリアルタイム計算。スコアが 0.7 を超えたメッセージのみをフォーカスビューに表示します。計算はサーバーサイドで 50ms 以内に完了し、結果は Redis にキャッシュされます。

- **Zoned Focus Scheduler**  
  ユーザーのタイムゾーン（例: `Asia/Tokyo`）を考慮したスケジューリング機能。`focus_start` と `focus_end` パラメータで集中時間帯を定義し、その期間中はスコアが低いメッセージを自動的にアーカイブ。集中時間帯外では、全メッセージを表示する「レビューモード」に切り替わります。設定は PostgreSQL の `user_settings` テーブルに保存され、変更後即時反映されます。

- **アクション優先順位パイプライン**  
  メッセージごとに推奨アクション（返信・スヌーズ・削除・タスク化）を機械学習モデルで提案。モデルは LightGBM ベースで、過去 90 日間のユーザー行動ログ（`action_history` テーブル）を学習し、精度 92.3% で次のアクションを予測。推論はエッジデバイスでも実行可能な軽量 ONNX モデルに変換されています。

### 使い方

1. **初期セットアップとフォーカススコア閾値の調整**  
   `config.toml` の `[focus]` セクションで `score_threshold = 0.7` を設定。必要に応じて `sender_weight` や `urgency_factor` を調整します。設定後、`focusinbox-eight --init --db postgresql://localhost:5432/focusinbox` を実行し、PostgreSQL スキーマ（`users`, `messages`, `scores`, `actions` テーブル）を自動生成します。

2. **集中時間帯の定義とリアルタイムフィルタリング**  
   API 経由で `POST /api/v1/schedule` に JSON ボディ `{"user_id": "u-001", "focus_start": "09:00", "focus_end": "12:00", "timezone": "Asia/Tokyo"}` を送信。サーバーは即座にスケジューリングルールを適用し、該当ユーザーの受信トレイをフィルタリング開始。集中時間帯中は、フォーカススコアが閾値を超えたメッセージのみが WebSocket 経由でプッシュされます。

3. **アクション提案の確認と手動オーバーライド**  
   ダッシュボードの「アクション提案」パネルに、各メッセージの推奨アクションと信頼度（例: `{"action": "reply", "confidence": 0.94}`）が表示されます。ユーザーが手動で異なるアクションを選択した場合、そのフィードバックは `action_feedback` テーブルに記録され、次回のモデル再学習（毎週日曜 03:00 JST に自動実行）に反映されます。

### ベストプラクティス

- **フォーカススコアの閾値は段階的に調整する**  
  初回は `score_threshold = 0.5` から開始し、1 週間の使用ログを分析した上で 0.1 ずつ引き上げます。急激な閾値変更は重要なメッセージの見逃しリスクを高めます。`/api/v1/analytics/score-distribution` エンドポイントで分布を可視化し、適切な閾値を決定してください。

- **集中時間帯は 2 時間単位で分割して設定する**  
  人間の集中力持続時間は平均 90 分とされています。`focus_start` と `focus_end` の間隔は 120 分以内に設定し、間に 15 分以上の休憩スロット（レビューモード）を挟むことを推奨します。これにより、フォーカススコアの高いメッセージに対する応答時間が平均 23% 短縮されることが確認されています。
