# focusinbox-eight - FOC_001

## 概要

focusinbox-eight は、高頻度なメール受信トレイのコンテキストスイッチを排除し、**処理待ちメッセージの滞留時間をミリ秒単位で可視化**するエンタープライズ向けフォーカス管理基盤です。本プロジェクトは、PostgreSQL 上の時系列テーブルと連携し、ユーザーごとのフォーカスセッションと受信メッセージの因果関係を自動トレースします。

## 詳細な内容

### focusinbox-eight の核心

focusinbox-eight は、メールクライアントやタスク管理ツールのバックグラウンドで動作し、**ユーザーが「フォーカス状態」に入った瞬間に受信トレイを論理的にロック**。新着メッセージは専用のキュー（`focus_queue`）に格納され、フォーカス解除後に初めてユーザーに通知されます。これにより、1 メッセージあたりの平均割り込み時間を 0.3 秒未満に抑制します。

### コア機能

- **動的フォーカススケジューラ**  
  ユーザーのカレンダーイベント（`CalendarEvent.startTime`, `CalendarEvent.endTime`）とキーボードアクティビティ（`KeystrokeLatency` が 200ms 未満の状態）をリアルタイムで監視し、フォーカスセッションを自動開始・終了。設定パラメータ `FOCUS_THRESHOLD_MS` により、割り込み許容遅延を 50ms 単位でチューニング可能。

- **因果関係トレーシングエンジン**  
  各メッセージに対して `thread_id` と `focus_session_id` を紐付け、PostgreSQL の `pg_stat_statements` 経由でクエリレベルのレイテンシを記録。フォーカス中に発生したメッセージの `queue_entry_timestamp` と `delivery_timestamp` の差分をミリ秒精度で算出し、ダッシュボードにヒートマップとして表示。

- **アダプティブ通知抑制フィルタ**  
  `message_priority`（1〜5）と `sender_reputation_score`（0.0〜1.0）の積が `NOTIFICATION_THRESHOLD` を超えた場合のみ、フォーカス中でもバイパス通知を許可。この閾値は過去 7 日間のユーザー応答率（`ResponseRate`）に基づいて自動調整される。

### 使い方

1. **フォーカスセッションの初期化と閾値設定**  
   `focusinbox-eight` を起動後、まず `config.toml` 内の `[focus]` セクションで `FOCUS_THRESHOLD_MS = 150` と `NOTIFICATION_THRESHOLD = 0.75` を指定。次に、`focusctl init --db-host localhost --db-port 5432 --db-name focusdb` を実行し、PostgreSQL の `focus_sessions` テーブルと `message_queue` テーブルを自動作成。`focusctl status` で現在のフォーカス状態とキューイング件数（例: `queued: 12`）を確認。

2. **カレンダー連携による自動フォーカス開始**  
   `focusctl calendar-sync --provider outlook --calendar-id "primary"` を実行。30 秒ごとにカレンダーイベントを取得し、`startTime` の 5 分前からフォーカスセッションをプリロード。セッション中は `focusctl lock --reason "meeting"` で手動ロックも可能。ロック中に受信したメッセージは、`SELECT * FROM message_queue WHERE focus_session_id = 'xxx' ORDER BY queue_entry_timestamp;` で確認できる。

3. **フォーカス解除後のメッセージ再配送**  
   `focusctl unlock --flush` を実行すると、キューイングされた全メッセージが `delivery_timestamp` の昇順でユーザーに再配送される。この際、`focusinbox-eight` は各メッセージの `queue_entry_timestamp` と `delivery_timestamp` の差分を `focus_sessions.latency_ms` カラムに記録。`focusctl report --period 24h` で、過去 24 時間の平均レイテンシと最大レイテンシ（例: `avg_latency: 234ms, max_latency: 1892ms`）を出力。

### ベストプラクティス

- **フォーカス閾値の段階的チューニング**  
  最初は `FOCUS_THRESHOLD_MS = 300` から開始し、1 週間分の `latency_ms` データを収集。中央値が 200ms を下回ったら閾値を 150ms に引き下げる。これにより、ユーザーの認知負荷を段階的に最適化できる。`focusctl tuning --target-latency 150` で自動調整モードも利用可能。

- **バイパス通知ルールのホワイトリスト運用**  
  緊急度の高い送信元（例: `sender_reputation_score >= 0.9` かつ `message_priority >= 4`）のみバイパス対象とする。`focusctl whitelist add --sender "alert@monitoring.example" --priority 5` で明示的に追加。逆に、`focusctl blacklist add --domain "newsletter.example"` で通知抑制の完全ブロックも可能。これにより、フォーカス中の真の緊急通知のみを通過させる。
