# focusinbox-eight - FOC_028

## Overview

focusinbox-eight is a high-performance, real-time email triage and prioritization engine designed for knowledge workers processing 500+ daily inbound messages. It leverages a lightweight NLP pipeline running locally to classify, rank, and surface the eight most actionable items from your inbox within 120ms of a new message arrival.

## Core Features

### 1. Dynamic Priority Scoring with Temporal Decay
The core engine assigns a `priority_score` (0.0 – 1.0) to each inbound message using a weighted formula: `(sender_reputation * 0.4) + (thread_depth * 0.3) + (keyword_density * 0.2) - (age_hours * decay_factor)`. The `decay_factor` is configurable in microseconds (µs) via the `DECAY_RATE_MS` environment variable. Messages from a user-defined `high_priority_domains` list (e.g., `["@clientcorp.com", "@ceo.com"]`) automatically receive a `sender_reputation` floor of 0.85. This ensures that a 48-hour-old email from a CEO is still surfaced before a 2-minute-old internal newsletter.

### 2. Multi-Source Inbox Aggregation with Deduplication
focusinbox-eight connects to up to 5 IMAP/POP3 accounts simultaneously, using a non-blocking I/O model (libuv-based). It performs SHA-256 hashing on the `Message-ID` header combined with the `Received` timestamp (nanosecond precision) to create a unique `fingerprint`. This fingerprint is stored in a local SQLite3 database (`inbox_cache.db`) to prevent duplicate processing. The aggregation pipeline runs on a configurable cron interval (default: 30 seconds) and emits a `batch_complete` event containing an array of `ProcessedMessage` objects, each with a `latency_us` field.

### 3. Contextual Snippet Extraction & Actionable Entity Linking
Upon processing, the engine extracts the first 200 characters of the message body and runs it through a regex-based entity extractor. It identifies patterns for `ticket_id` (e.g., `JIRA-1234`, `ZEN-567`), `date_time` (ISO 8601), and `monetary_value` (e.g., `$1,234.56`). These entities are linked to the `focusinbox-eight` local REST API endpoint (`/api/v1/actions/link`) which can trigger external webhooks. For example, a message containing a `ticket_id` will automatically generate a clickable link to the ticketing system’s detail page, rendered in the output JSON as `linked_actions: ["https://jira.company.com/browse/JIRA-1234"]`.

## How to Use

### Step 1: Initialize the Inbox Profile
Run the following command to create a new profile configuration. You will be prompted to enter SMTP/IMAP credentials, priority domain lists, and the decay rate in microseconds.
```bash
focusinbox-eight --init-profile --profile-name "work_prod" --imap-host imap.example.com --imap-port 993 --decay-rate-us 50000
```
This creates a `profile_work_prod.json` file in `~/.focusinbox/`. The engine will immediately validate the IMAP connection and return a `connection_status: ok` or a detailed error code (e.g., `ERR_AUTH_FAILED`).

### Step 2: Trigger a Manual Inbox Scan
To force a scan outside the scheduled interval, use the `--scan-now` flag. This bypasses the deduplication cache for the current run.
```bash
focusinbox-eight --profile "work_prod" --scan-now --output-format json --max-results 8
```
The engine will output a JSON array of the top 8 scored messages. Each object contains `fingerprint`, `priority_score`, `snippet`, `linked_actions`, and `latency_us`. A non-zero `latency_us` indicates the time spent on the IMAP fetch and processing for that specific message.

### Step 3: Integrate with the Local Dashboard
After scanning, start the built-in HTTP server to view the results in a web UI.
```bash
focusinbox-eight --serve --port 8080 --profile "work_prod"
```
Navigate to `http://localhost:8080/dashboard`. The dashboard auto-refreshes every 10 seconds via Server-Sent Events (SSE). It displays a real-time list of the top 8 items, including a visual `priority_bar` (width proportional to `priority_score`). Clicking a `linked_actions` button will open the external link in a new tab. The server logs all API calls to `access.log` with millisecond timestamps.

## Best Practices

### 1. Optimize the Decay Rate for Your Workflow
For a standard 9-to-5 workflow, set `DECAY_RATE_MS` to `100000` (100 seconds). This ensures that an email from 9:00 AM is nearly invisible by 5:00 PM, preventing inbox clutter from the morning. For a 24/7 on-call team, reduce this to `5000` (5 seconds) to keep critical alerts at the top for a maximum of 2 hours. Monitor the `latency_us` field in the JSON output to ensure your IMAP server can handle the increased scan frequency without timeout errors (target: < 5000µs per message).

### 2. Use Domain Whitelists for Critical Communication Channels
Create a separate profile for each major client or internal team. In the `profile.json`, set the `high_priority_domains` array to include only the relevant domains. For example, for a profile named "client_acme", set `["@acme.com", "@acme-support.com"]`. This forces the `sender_reputation` floor to 0.85 for all emails from these domains, effectively creating a "VIP lane" that bypasses the temporal decay. Combine this with a short `DECAY_RATE_MS` (e.g., `10000`) to ensure that only the most recent VIP emails are surfaced, while older ones are automatically pruned.
