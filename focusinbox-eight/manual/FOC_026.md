# focusinbox-eight - FOC_026

## Overview

focusinbox-eight is a high-performance, real-time email triage and context-switching engine designed for knowledge workers managing multiple inboxes across fragmented communication channels. It leverages a local-first, encrypted indexing layer to provide sub-100ms search across 50,000+ messages, with zero-latency focus mode switching between projects, clients, or time-bounded deep work sessions.

## Detailed Content

### What is focusinbox-eight?

focusinbox-eight is an open-core productivity infrastructure tool that ingests, clusters, and semantically indexes emails from IMAP, Microsoft Graph, and custom webhook sources. It dynamically assigns a "focus score" to each message based on sender priority, thread velocity, and your current context window (defined by a project ID or time block). The core engine runs as a sidecar process, exposing a gRPC API for integration with editors, task managers, and automation pipelines.

### Core Features

- **Semantic Context Pinning with Vector Embedding Cache**  
  Automatically generates 768-dimensional embeddings for each email body using a local ONNX-optimized MiniLM model. When you activate a "Focus Session" (e.g., `--project=alpha`), the engine computes cosine similarity between your session descriptor and all indexed messages in under 50ms, surfacing only the top 15 most relevant threads. The embedding cache is persisted to a local SQLite database with WAL mode, enabling instant recall across reboots.

- **Zero-Copy Inbox Switching via Memory-Mapped State Files**  
  Each focus profile (e.g., `client-x`, `deep-work`, `incident-response`) is serialized as a memory-mapped binary state file at `~/.focusinbox/state/*.fstate`. Switching between profiles requires no database queries; the engine simply remaps the pointer to the new file, achieving sub-microsecond context loads. The state includes a bloom filter of 10,000 recently seen Message-IDs to prevent duplicate notifications during rapid switches.

- **Predictive Thread Snoozing with Temporal Graph Analysis**  
  Constructs a directed temporal graph of email threads, tracking response latency per sender-recipient pair. Using a Holt-Winters forecasting model (trained on the last 90 days of thread activity), focusinbox-eight predicts the optimal time to re-surface a snoozed email. Parameters include `alpha=0.3`, `beta=0.1`, `gamma=0.6`, with a seasonal period of 168 hours. The engine auto-snoozes messages with a predicted response probability below 0.25, reducing inbox noise by ~40%.

### How to Use

1. **Initialize a Focus Profile with Custom Embedding Threshold**  
   Run `focusinbox init --profile=deep-work --embed-threshold=0.82 --max-results=20`. This creates a new profile with a cosine similarity cutoff of 0.82, ensuring only highly relevant messages surface. The engine will immediately begin indexing your primary IMAP inbox (`imap://user@domain.com/INBOX`) and building the local vector store at `~/.focusinbox/index/`. Verify ingestion with `focusinbox status --profile=deep-work`; expect a `messages_indexed` count and a `latency_p50` metric in milliseconds.

2. **Execute a Real-Time Context Switch with Priority Override**  
   To switch to a client-focused session, execute:  
   `focusinbox switch --profile=client-x --priority-override=urgent --time-window=2h`.  
   This re-pins the context to the `client-x` state file, applies a priority boost to emails from your top-5 senders (configured in `~/.focusinbox/priorities.yaml`), and filters results to only messages received in the last 2 hours. The gRPC endpoint at `localhost:9090` will now return a `FocusSnapshot` protobuf with `thread_ids`, `scores`, and `suggested_actions`.

3. **Trigger Predictive Snoozing with Custom Forecast Horizon**  
   Snooze a low-priority thread using:  
   `focusinbox snooze --thread-id=abc123 --horizon=48h --model=holt-winters`.  
   The engine calculates the best wake-up time within the 48-hour window. To inspect the forecast, run `focusinbox forecast --thread-id=abc123 --show-params`. Output includes `predicted_response_time`, `confidence_interval_95`, and `seasonal_components`. The thread will be re-injected into your active focus queue at the predicted time, with a `snooze_reason` field set to "predicted_optimal."

### Best Practices

- **Pair with Time-Boxed Task Managers for Cognitive Load Reduction**  
  Integrate focusinbox-eight with a local task tracker (e.g., Taskwarrior or a simple Markdown log). Configure a post-switch hook that automatically logs the `focus_session_start` timestamp and the `profile_name` to a structured log file. This creates a verifiable audit trail of your context shifts, enabling weekly reviews of "focus fragmentation" metrics (e.g., number of switches per hour, average dwell time per profile). Aim for a fragmentation score below 0.3 switches per hour.

- **Use the Bloom Filter to Eliminate Cross-Profile Notification Duplication**  
  When running multiple focusinbox-eight instances (e.g., one for work, one for personal projects), share a common bloom filter file via a symlink at `~/.focusinbox/global/bloom.bin`. This prevents the same email from triggering notifications in both profiles. Periodically rebuild the filter with `focusinbox bloom --rebuild --capacity=100000 --false-positive-rate=0.01`. Monitor the false positive rate weekly; if it exceeds 5%, increase the capacity parameter.
