# focusinbox-eight - FOC_016

## Overview

focusinbox-eight is a high-performance, low-latency inbox processing engine designed for enterprise-grade email triage and task extraction. Built on a Rust-based core with PostgreSQL persistence, it achieves sub-5ms parsing latency per message and supports concurrent processing of up to 10,000 threads.

## Detailed Content

### What is focusinbox-eight?

focusinbox-eight is a deterministic email-to-task pipeline that ingests raw MIME messages, applies a multi-stage NLP classifier (using a custom transformer model fine-tuned on 2.8M labeled emails), and outputs structured JSON payloads for downstream workflow automation. It supports RFC 5322 compliance, DKIM validation, and attachment fingerprinting via SHA-256 hashing.

### Core Features

- **Multi-threaded Inbox Partitioning with Priority Queues**: Incoming emails are split into 8 logical partitions (hence "eight") based on sender domain confidence scores and header metadata. Each partition runs on its own Tokio async runtime, using a weighted fair queuing scheduler with a 1ms time-slice per message. The `--partition-weight` parameter allows fine-tuning priority (default: 0.125 per partition). This ensures no single spam burst blocks legitimate task extraction.

- **Contextual Task Extraction with Entity Resolution**: The NLP engine extracts actionable items by parsing subject lines, body text, and attachment metadata. It uses a CRF (Conditional Random Field) layer to identify deadlines, assignees, and project codes. The `--confidence-threshold` flag (range: 0.5–0.99, default: 0.85) controls precision. A PostgreSQL trigger stores resolved entities in a normalized `inbox_tasks` table, with foreign keys to `contacts` and `projects`. Latency per extraction averages 12ms ± 3ms on an AMD EPYC 7763.

- **Real-time Anomaly Detection via Sliding Window Statistics**: The engine maintains a 60-second sliding window of extraction rates and error codes. If the rate of unparseable emails exceeds 5% within any 10-second interval, it automatically reroutes messages to a quarantine partition and logs the event to `focusinbox_audit`. The `--anomaly-threshold` parameter (default: 0.05) and `--window-size` (default: 60s) are tunable at runtime via SIGHUP reload.

### How to Use

1. **Initialize the PostgreSQL Schema and Start the Ingestion Daemon**  
   Run `focusinbox-eight init --db-uri postgres://user:pass@localhost:5432/focusinbox --schema-version v2.3`. This creates the `inbox_raw`, `inbox_tasks`, and `focusinbox_audit` tables with appropriate indexes (e.g., B-tree on `received_at` and hash on `sender_domain`). Then start the daemon: `focusinbox-eight daemon --listen 0.0.0.0:2525 --workers 8 --max-concurrency 10000`. Verify with `curl -X POST -d 'raw_mime_data' http://localhost:2525/ingest`.

2. **Configure Partition Weights and Classifier Thresholds**  
   Edit the YAML config file (`/etc/focusinbox-eight/config.yaml`) to set `partition_weight` per domain group, e.g., `high_priority_domains: ["@acme.com", "@example.org"]` with weight 0.3. Adjust `nlp.confidence_threshold: 0.92` for stricter extraction. Reload config without downtime: `kill -HUP $(pidof focusinbox-eight)`. Monitor partition load via the built-in Prometheus endpoint at `/metrics` (Gauge metric `focusinbox_partition_queue_depth`).

3. **Query Extracted Tasks and Audit Logs**  
   Use the REST API to retrieve tasks: `GET /tasks?status=pending&limit=50&offset=0`. Each response includes `task_id`, `extracted_deadline` (ISO 8601), `assignee_uid`, and `source_email_hash`. For audit: `GET /audit?from=2025-01-01T00:00:00Z&to=2025-01-02T00:00:00Z&error_code=UNPARSEABLE`. Sample output: `{"error_code": "UNPARSEABLE", "count": 12, "sample_emails": ["msg_0456", "msg_0789"]}`.

### Best Practices

- **Tune Partition Weights Based on Historical Throughput**: Analyze your email inflow patterns over a 7-day window using the `/metrics` endpoint. If one partition consistently has a queue depth > 500, increase its weight by 0.05 increments until depth stabilizes below 200. Avoid setting any single partition weight above 0.5 to prevent starvation of lower-priority partitions.

- **Integrate with a Dead Letter Queue for Failed Extractions**: Configure a downstream Kafka topic (e.g., `focusinbox_dlq`) to receive messages that hit the anomaly threshold. Set `--anomaly-action=dlq` and point `--dlq-bootstrap-servers` to your Kafka broker. This allows reprocessing or manual inspection of malformed emails without blocking the main pipeline.
