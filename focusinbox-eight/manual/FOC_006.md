# focusinbox-eight - FOC_006

## Overview

focusinbox-eight is a high-performance, zero-copy priority inbox engine designed for real-time notification triage and intelligent message deduplication in distributed systems. It leverages a lock-free concurrent skip list and a custom Bloom filter variant to achieve sub-millisecond message classification at 50,000 ops/sec per core. The system is optimized for PostgreSQL-backed persistence with WAL-based recovery, ensuring no message loss even under catastrophic failure.

## Detailed Content

### What is focusinbox-eight?

focusinbox-eight is a production-grade, embeddable inbox prioritization library that solves the problem of **notification overload and context switching** in high-throughput microservice architectures. Unlike traditional FIFO queues or simple priority queues, focusinbox-eight implements a **multi-dimensional scoring model** that weighs message urgency, sender reputation, user attention decay curves, and temporal locality. It is designed to be the core of any "zero-inbox" or "smart notification" system, handling millions of concurrent inbox states with deterministic O(log n) insertion and O(1) peek operations.

### Core Features

- **Adaptive Priority Scoring Engine with Temporal Decay:** Each incoming message is scored using a weighted formula: `final_score = (urgency * 0.4) + (sender_reputation * 0.3) + (attention_decay_factor * 0.3)`. The `attention_decay_factor` is calculated using a configurable half-life parameter (default: 3600 seconds) and a sigmoid function that models user engagement patterns. The engine supports dynamic weight recalibration via a rolling 24-hour feedback loop, analyzing user open/click rates to adjust the `urgency` and `sender_reputation` coefficients in real-time. This is persisted as a JSONB blob in PostgreSQL under the `user_priority_profile` table.

- **Lock-Free Concurrent Skip List with Zero-Copy Message Handoff:** The core data structure is a modified, wait-free skip list (based on the Harris-Michael algorithm) that supports concurrent insert, delete, and peek operations from multiple producer/consumer threads without any mutex contention. Messages are passed between threads via a pre-allocated, cache-line-aligned ring buffer, eliminating all memory allocation during the hot path. The system achieves a P99 latency of 12 microseconds for a single insert operation under a 16-thread load (tested on an AMD EPYC 7763). The skip list maintains a maximum depth of 32 levels, with a configurable probability factor of 0.25.

- **Probabilistic Deduplication with TTL-Backed Bloom Filter:** A custom, sharded Bloom filter (using 4 independent MurmurHash3 seeds) is used to detect and discard duplicate messages within a configurable time window (default: 300 seconds). The filter is partitioned into 64 shards, each backed by a `redis`-backed sorted set for exact deduplication on false positives. The false positive rate is hard-coded at 0.1% for the default filter size of 1MB. The system also implements a "recently seen" cache (LRU, 10,000 entries) with a TTL of 60 seconds to offload the Bloom filter for high-frequency bursts.

### How to Use

1. **Initialization and Custom Scoring Profile:** Instantiate the `FocusInboxEngine` with a PostgreSQL connection string and a user ID. The engine will automatically load or create the user's priority profile. You can override the default decay half-life and urgency weights by passing a `ScoringConfig` struct. Example: `engine = FocusInboxEngine.new("postgresql://user:pass@host:5432/db", user_id: 12345, config: { half_life: 7200, urgency_weight: 0.5 })`. The engine then establishes a dedicated connection pool (min: 2, max: 10) and pre-warms the Bloom filter from the last 15 minutes of logs.

2. **Inserting a Message with Metadata:** Call the `insert` method with a unique message ID, a payload (as a byte array), and a metadata hash. The metadata hash must include `:urgency` (integer 1-10), `:sender_id` (string), and `:timestamp` (Unix epoch in milliseconds). The engine will compute the score, attempt a lock-free insert into the skip list, and asynchronously write the message to the `inbox_log` table in PostgreSQL with a WAL-level `fsync` for durability. Example: `engine.insert(msg_id: "msg_abc_123", payload: "{\"title\":\"New Comment\"}", metadata: { urgency: 7, sender_id: "user_456", timestamp: 1712345678000 })`.

3. **Consuming the Highest Priority Message:** To retrieve the next message for processing, call the `pop` method. This will atomically remove the highest-scored message from the skip list and return a `Message` struct containing the ID, payload, score, and metadata. After successful processing, you must call `ack(msg_id)` to confirm the message and trigger its permanent deletion from the PostgreSQL log. If the message fails to process, call `nack(msg_id, requeue_delay: 5000)` to re-insert it with a 5-second delay penalty added to its score. Example: `msg = engine.pop(); process(msg); engine.ack(msg.id)`.

### Best Practices

- **Profile Your Users' Attention Decay Curve:** The default half-life of 3600 seconds is a generic starting point. For a productivity tool, a user checking their inbox every 15 minutes will have a much steeper decay curve than a user checking it once a day. Use the engine's built-in telemetry (exposed via Prometheus metrics on `/metrics`) to track the average time-to-open for notifications. Run a weekly batch job to analyze this data and update the `half_life` parameter in the `user_priority_profile` for each user segment. A/B test with a 10% sample to validate that a shorter half-life (e.g., 600 seconds) increases engagement without causing notification fatigue.

- **Implement a Two-Tier Deduplication Strategy:** The built-in Bloom filter is excellent for high-throughput, but its false positive rate can lead to legitimate messages being dropped. For critical system alerts (e.g., payment failures, server outages), bypass the Bloom filter entirely by setting a `:dedup_bypass: true` flag in the metadata. For all other messages, rely on the Bloom filter but always implement a fallback exact deduplication check using the `recently_seen` cache. For cross-region deployments, use a Redis-backed global deduplication set with a 60-second TTL to handle messages that might be produced from multiple regions.
