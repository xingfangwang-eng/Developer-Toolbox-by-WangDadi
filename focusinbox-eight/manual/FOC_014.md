# focusinbox-eight - FOC_014

## Overview

focusinbox-eight is a high-performance, multi-tenant email triage and intelligent notification aggregation engine. It is designed to parse, classify, and route inbound messages from disparate sources (SMTP, API webhooks, IMAP) into a unified, zero-latency processing pipeline. The system is optimized for throughput exceeding 10,000 messages per second per node, with sub-100-millisecond classification latency.

## Detailed Content

### What is focusinbox-eight?

focusinbox-eight is a distributed, event-driven framework that solves the problem of email and notification overload in enterprise SaaS environments. It leverages a custom-trained NLP classifier (based on a distilled BERT variant) to extract intent, priority, and action items from unstructured text payloads. Unlike generic email clients, focusinbox-eight provides deterministic routing rules, deduplication across channels, and a plugin architecture for custom transformers (e.g., for Salesforce case creation, Jira ticket linking, or PagerDuty alert enrichment).

### Core Features

- **Real-Time Content Vectorization & Priority Scoring**: Each inbound message is passed through a tokenizer with a maximum sequence length of 512 tokens. A quantized ONNX runtime model computes a priority score (0.0–1.0) and an intent vector (128 dimensions) in under 50ms. The system supports configurable thresholds; messages with a score >0.85 are automatically escalated to a dedicated high-priority queue backed by Redis Streams with a TTL of 24 hours.

- **Deterministic Multi-Stage Routing Engine**: The router uses a trie-based rule matcher that evaluates up to 10,000 routing rules in O(log n) time. Rules can match on header fields (e.g., `X-Organization-ID`, `From` domain), body regex patterns, or the computed intent vector via cosine similarity to a stored centroid. Each rule specifies a target pipeline (e.g., `slack-webhook`, `s3-archive`, `postgres-event-log`). Failed matches are sent to a dead-letter queue (DLQ) with automatic retry logic (3 retries, exponential backoff, max interval 300s).

- **Immutable Audit Trail with PostgreSQL & S3** : Every processed message is stored as a JSONB row in a partitioned PostgreSQL table (`events_YYYY_MM`). The table includes a `processing_metadata` column tracking the full decision path: matched rule ID, priority score, transformer invocations, and latency breakdown (in microseconds). Raw payloads larger than 64KB are offloaded to S3 with a pre-signed URL stored in the database. The system enforces a 90-day retention policy with automatic partition cleanup via `pg_cron`.

### How to Use

1. **Deploy the Core Pipeline with Docker Compose**:
   - Clone the repository and navigate to `deploy/docker/`.
   - Set required environment variables in `.env`: `POSTGRES_URI=postgresql://user:pass@host:5432/focusinbox`, `REDIS_URI=redis://host:6379/0`, `S3_BUCKET=my-audit-bucket`.
   - Run `docker-compose up -d --scale worker=4`. This spins up the API gateway (port 8080), 4 worker instances (each consuming from the Redis stream), and a PostgreSQL 15 instance with the `pgvector` extension for centroid storage.
   - Verify health: `curl http://localhost:8080/health` should return `{"status":"ok","workers_connected":4,"queue_depth":0}`.

2. **Configure Routing Rules via the Admin API**:
   - Use the `/v1/rules` endpoint to add a rule. Example payload:
     ```json
     {
       "priority": 10,
       "match": {
         "header": {"from_domain": "critical-client.com"},
         "body_pattern": "(?i)urgent|outage|down"
       },
       "action": {
         "type": "webhook",
         "target": "https://hooks.slack.com/services/T00/B00/xxx",
         "transform": ["jira-ticket-creator"]
       }
     }
     ```
   - The rule is compiled into the trie instantly. Use `/v1/rules/test` with a sample message body to simulate routing and view the matched rule ID and decision path.

3. **Monitor Throughput and Latency via Prometheus Metrics**:
   - The `/metrics` endpoint exposes counters for `focusinbox_messages_processed_total`, `focusinbox_classification_latency_ms` (histogram with buckets [5, 10, 25, 50, 100, 200, 500]), and `focusinbox_queue_depth`.
   - Set up a Grafana dashboard using the provided JSON template in `monitoring/grafana-dashboard.json`. Key panels: P99 classification latency, rule match distribution, and DLQ size (alert if >100).

### Best Practices

- **Use Staged Rollouts for Rule Changes**: Always deploy new routing rules to a staging environment first. Use the `/v1/rules/dry-run` endpoint to evaluate the rule against a replay of the last 24 hours of production traffic (stored in the `events` table). This prevents accidental misrouting of critical alerts. For production, enable the `canary` flag on the rule to route only 5% of matching traffic for 15 minutes before full activation.

- **Optimize for Cold Start and Burst Traffic**: The worker pool uses a dynamic scaling policy based on the Redis stream's pending entry count. Set `WORKER_MIN=2` and `WORKER_MAX=20` in the Docker Compose file. For environments with sporadic high-volume bursts (e.g., after a product launch), pre-warm the NLP model by running a health check that sends 100 synthetic messages to the `/v1/ingest` endpoint. This ensures the ONNX runtime is cached and the tokenizer is initialized before real traffic arrives.
