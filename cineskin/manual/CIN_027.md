## What is cineskin?

**cineskin** is a high-performance, modular skin-design engine for cinematic video effects. It decouples visual styling from rendering pipelines, allowing real-time, GPU-accelerated skin mapping across any video frame sequence. Think of it as a programmable texture layer system for post-production — designed for compositors who need deterministic, repeatable results without bloated GUI overhead.

---

## Core Features

### 1. **Real-Time LUT-Baked Skin Mapping**
cineskin compiles custom color lookup tables (LUTs) directly into skin texture layers. This means you can apply complex, multi-stop skin tones or stylized dermabrasion effects without touching the base video codec. The engine supports 16-bit float precision and runs inference on CUDA/Metal shaders for sub-frame latency.

### 2. **Node-Graph Procedural Skin Compositing**
Instead of linear layer stacking, cineskin uses a directed acyclic graph (DAG) for skin effects. Each node can be a procedural mask, a displacement map, or a temporal filter. You can chain, fork, and merge effects in real time — all without re-rendering the entire sequence. This is critical for iterative color grading and skin retouching in 4K/8K pipelines.

### 3. **Metadata-Driven Effect Persistence**
Every skin effect applied via cineskin is embedded as sidecar metadata in OpenTimelineIO (OTIO) format. This means your skin design decisions are fully reproducible across different NLEs (DaVinci Resolve, Premiere Pro, Final Cut Pro) and render farms. No more “lost settings” when switching between tools.

## How to Use

### Step 1: Initialize a Skin Context with Custom LUT Calibration
```bash
cineskin init --project my_film --lut /path/to/custom_lut.cube --precision float16
```
This creates a `cineskin_context.json` file that stores your base LUT, resolution, and frame range. The engine will pre-cache the LUT into GPU memory. For multi-cam setups, use `--sync-tc` to timecode-lock all skin layers.

### Step 2: Build a Procedural Skin Graph via CLI or Python API
```python
from cineskin import SkinGraph, MaskNode, DisplaceNode

graph = SkinGraph()
mask = MaskNode(type="chroma_key", threshold=0.15, softness=0.02)
displace = DisplaceNode(map="skin_texture_4k.exr", strength=0.3)
graph.add_node(mask)
graph.add_node(displace)
graph.connect(mask, displace)
graph.compile(device="cuda:0")
```
This graph applies a chroma-key mask (with soft edges) and a displacement map to the skin region. You can export the graph as a JSON pipeline for version control.

### Step 3: Render and Export with Sidecar Metadata
```bash
cineskin render --input source_clip.mov --graph skin_pipeline.json --output rendered_skin.mov --metadata otio
```
The output will include an `.otio` file alongside the video. You can import this into any OTIO-compatible NLE to reconstruct the exact skin effect stack. For batch processing, use `--batch /path/to/clips.txt`.

## Best Practices

### 1. **Always Pre-Calibrate LUTs on Reference Frames**
Before applying any skin effect across a sequence, run a calibration pass on a single reference frame. Use `cineskin calibrate --frame 100 --lut-ref /path/to/reference.cube`. This ensures that your skin mapping is consistent across different lighting conditions and camera sensors. Avoid “one LUT fits all” — it will break skin tones in high-contrast scenes.

### 2. **Version-Controlled Skin Graphs with CI/CD Pipelines**
Treat your `skin_pipeline.json` as code. Store it in a Git repo, and use a CI/CD pipeline (e.g., GitHub Actions) to validate and render skin effects on every commit. This is especially useful for studios with multiple compositors: you can enforce style guides, detect graph conflicts, and automate regression testing on a set of reference frames.
