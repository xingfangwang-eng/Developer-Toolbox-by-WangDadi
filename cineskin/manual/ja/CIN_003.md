# cineskin - CIN_003

## 概要

cineskin は、映画ポストプロダクションにおけるスキンデザインと映像エフェクトを、コードベースで精密に制御するための次世代ツールです。従来のGUI依存のワークフローを打破し、スクリプト駆動による再現性と高速なイテレーションを実現します。本プロジェクトは、映像制作の現場で求められる「アーティストの意図を損なわない自動化」をコアバリューとしています。

## 詳細な内容

### cineskin とは？

cineskin は、映像のルックデヴ（Look Development）プロセスをプログラム可能にするSaaSツールです。特に、キャラクターやプロップの「スキン（外皮）」に特化したテクスチャマッピング、シェーディング、そして時間軸に沿ったエフェクト合成を、高レベルなAPIとCLIで提供します。LUTやマテリアル定義をJSON/YAMLで管理し、CI/CDパイプラインに組み込むことで、チーム全体のアセット管理を効率化します。

### コア機能

1.  **動的シェーダーグラフのスクリプト制御**
    cineskin は、ノードベースのシェーダーエディタを内部的に持ちながら、そのすべてのパラメーターをPython/Rubyスクリプトから操作可能です。例えば、キャラクターの皮膚の血色を、ストーリーの進行に応じてリアルタイムに変化させる「生体反応シェーダー」を、コードで定義・適用できます。これにより、従来手動でキーフレームを打っていた複雑な表情変化を、アルゴリズムで自動生成できます。

2.  **マルチレイヤー・タイムワープエフェクトエンジン**
    映像の特定のフレーム範囲にのみ、ノイズ、グレイン、被写界深度などのエフェクトを「レイヤー」として重ねられます。各レイヤーには独立した時間軸（タイムリマップ）を設定可能で、例えば「フラッシュバックシーンだけにフィルムグレインを強く乗せる」といった処理を、数行の設定ファイルで実現します。レンダリングはGPUアクセラレーションに対応し、4K/8K素材でもリアルタイムプレビューが可能です。

3.  **アセットバリデーション＆自動コンフリクト解決**
    チーム開発で発生しがちな、テクスチャの解像度不一致やマテリアル参照の欠損を、コミット前に検出・自動修正します。cineskin は、プロジェクト全体の依存関係グラフを解析し、例えば「キャラAの肌テクスチャが2Kから4Kに変更された場合、関連する全マテリアルのシェーダーパラメーターを自動調整する」といった高度なバリデーションルールを設定できます。これにより、ビルドエラーを未然に防ぎます。

### 使い方

1.  **プロジェクトの初期化とマテリアル定義**
    まず、`cineskin init` コマンドでプロジェクトルートを作成します。次に、`cineskin material create --name "character_skin" --template pbr_skin` でPBRベースのスキンマテリアルを生成します。生成された `materials/character_skin.yaml` を編集し、ベースカラー、ラフネス、スペキュラなどのパラメーターを数値で指定します。サブサーフェススキャタリング（SSS）の強度も、`sss_radius: [0.5, 0.3, 0.1]` のようにRGBチャンネルごとに設定可能です。

2.  **エフェクトレイヤーの適用とタイムライン設定**
    `cineskin effect add --layer "film_grain" --type "noise"` でエフェクトレイヤーを追加します。次に、`cineskin effect configure --layer "film_grain" --params '{"intensity": 0.15, "seed": 42}'` でパラメーターを調整します。さらに、`cineskin effect timeline --layer "film_grain" --start 100 --end 200 --ease-in 10` で、フレーム100から200の間にのみエフェクトが適用されるように設定します。イージング関数を指定することで、エフェクトの出現・消失を滑らかにできます。

3.  **バリデーションルールの設定と自動ビルド**
    プロジェクトルートに `.cineskin/validation.yml` を作成します。ここで、例えば「すべてのテクスチャは4K以上であること」というルールを定義します。
    ```yaml
    rules:
      - type: texture_resolution
        min: [4096, 4096]
        action: warn
      - type: material_reference
        check_all: true
        action: fix
    ```
    その後、`cineskin build --validate` を実行すると、ルールに違反するアセットが自動的に修正され、問題なければ `build/` ディレクトリに最終的なルックデヴデータが出力されます。

### ベストプラクティス

-   **マテリアルバージョニングとブランチ戦略**
    キャラクターのスキンは、シーンや感情によって複数のバリエーション（例：汗をかいた状態、傷がある状態）が必要です。cineskin のマテリアル定義はテキストファイルなので、GitなどのVCSで管理し、ブランチごとに異なるルックを開発しましょう。メインブランチには「ベースルック」、フィーチャーブランチには「特殊効果用」の設定を持たせることで、マージ時のコンフリクトを防ぎつつ、アーティスト間の並行作業を促進します。

-   **CI/CDパイプラインへの統合**
    映像制作においても、CI/CDの考え方は有効です。GitHub ActionsやGitLab CIで、`cineskin build --validate` をプルリクエストごとに実行するように設定しましょう。これにより、テクスチャの欠損や解像度の不一致を、レビュー前に自動検出できます。さらに、`cineskin render --preview` でサムネイルを生成し、デプロイ先のストレージにアップロードすることで、視覚的なレビューもコードレビューと同時に行えます。
