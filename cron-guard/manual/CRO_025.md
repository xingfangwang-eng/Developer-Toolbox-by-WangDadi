# cron-guard - CRO_025

## Overview

Cron Guard is a production-grade cron job monitoring and orchestration tool designed for DevOps engineers and SREs who demand zero-tolerance for missed schedules. It eliminates silent failures in batch processing pipelines by providing real-time heartbeat tracking, anomaly detection, and automated recovery workflows. Whether you manage 50 or 50,000 scheduled tasks, Cron Guard ensures your critical jobs run on time, every time.

## Core Features

### 1. **Predictive Anomaly Detection with Adaptive Thresholds**
Cron Guard doesn't just alert on failures—it learns your job execution patterns over time. Using a rolling window of historical run times, it dynamically adjusts alert thresholds to account for natural latency spikes (e.g., database replication delays or network congestion). When a job deviates beyond 3σ from its expected duration, Cron Guard triggers a preemptive escalation before the next scheduled run, preventing cascading failures.

### 2. **Distributed Job Dependency Graph with Fault-Tolerant DAG**
Model complex workflows with Cron Guard's built-in directed acyclic graph (DAG) engine. Define parent-child dependencies across jobs, and the system automatically pauses downstream tasks when an upstream failure occurs. The engine supports conditional branching (e.g., skip job C if job A succeeds but job B fails) and auto-retry with exponential backoff. All dependency states are persisted in a distributed key-value store, ensuring recovery even after full cluster restarts.

### 3. **Multi-Protocol Heartbeat with Silent Failure Detection**
Cron Guard ingests heartbeats via HTTP, gRPC, and NATS streaming, allowing it to monitor jobs running on bare metal, Kubernetes, or serverless functions. If a heartbeat is missing for more than 2x the job's configured interval, the system initiates a "silent failure" protocol: it attempts a health check via the job's registered endpoint, then escalates to PagerDuty, Slack, or email. The system also supports "dead man's switch" patterns for long-running batch jobs.

## How to Use

### Step 1: Configure a Job with Custom Anomaly Thresholds
Define a job in `cron-guard.yml` with explicit runtime baselines:
```yaml
jobs:
  - id: data-export
    schedule: "0 */2 * * *"
    timeout: 300s
    anomaly_threshold:
      base_duration: 120s
      max_deviation: 30%
    heartbeat:
      protocol: http
      endpoint: "https://api.example.com/health"
    retry:
      max_attempts: 3
      backoff: exponential
```
The system will automatically compare each run against the `base_duration` and alert if execution time exceeds 156s (120s + 30%). For jobs without explicit thresholds, Cron Guard learns the baseline after 10 runs.

### Step 2: Wire Up Dependencies Using the DAG Engine
Create a multi-job pipeline in a separate `workflow.yaml`:
```yaml
workflows:
  - id: nightly-etl
    jobs:
      - extract
      - transform
      - load
    dependencies:
      - from: extract
        to: transform
        condition: success
      - from: transform
        to: load
        condition: success
    failure_strategy: pause_downstream
```
Deploy this via the CLI: `cron-guard apply -f workflow.yaml`. The engine validates the DAG for cycles and ensures all referenced jobs exist. If `extract` fails, `transform` and `load` are automatically paused, and a notification is sent to the configured channel.

### Step 3: Set Up Multi-Channel Escalation with Dead Man's Switch
Configure escalation policies in `alerting.yaml`:
```yaml
alerting:
  channels:
    - type: pagerduty
      routing_key: "your_key"
      severity: critical
    - type: slack
      webhook: "https://hooks.slack.com/services/T00/B00/xxx"
  dead_mans_switch:
    enabled: true
    interval: 300s
    endpoint: "https://heartbeat.cron-guard.io/ping"
```
The dead man's switch sends a heartbeat every 5 minutes. If Cron Guard itself fails to receive this heartbeat for 15 minutes, it triggers an alert to all configured channels. This ensures you're notified even if the monitoring system goes down.

## Best Practices

### Practice 1: **Use Heartbeat Intervals Shorter Than Job Schedules**
For critical jobs running every hour, set the heartbeat interval to 15 minutes. This gives you a 45-minute window to detect and remediate failures before the next scheduled run. Combine this with the `anomaly_threshold` to catch slow executions early. For example, a data pipeline that normally takes 10 minutes but starts taking 40 minutes can be flagged before it misses its next deadline.

### Practice 2: **Implement a Tiered Alerting Strategy**
Avoid alert fatigue by tiering your notifications:
- **Tier 1 (Warning)**: Job runs 20% over expected duration → log to Elasticsearch and send low-priority Slack message.
- **Tier 2 (Critical)**: Job fails or misses heartbeat → page on-call engineer via PagerDuty.
- **Tier 3 (Emergency)**: Dead man's switch triggers → escalate to engineering manager and ops team.
This ensures that minor deviations don't wake up engineers at 3 AM, while catastrophic failures get immediate human attention.
