# cron-guard - CRO_014

## Overview

Cron Guard is a battle-tested monitoring and alerting toolkit designed for engineers who manage thousands of scheduled tasks across distributed environments. It eliminates silent failures, reduces noise, and ensures your cron jobs run exactly as intended—every time.

## Hardcore Features

### 1. Predictive Anomaly Detection with Multi‑Window Baseline
Cron Guard doesn’t just check “did the job run?”—it learns the historical execution patterns (duration, exit code, output size) across multiple time windows (last 7 days, 30 days, seasonal). When a job finishes 2.3σ faster or slower than its baseline, the system flags it as a potential regression or resource contention before it causes downstream failures.

### 2. Hierarchical Alert Routing with Escalation DAGs
Instead of flat alerting, you define a Directed Acyclic Graph (DAG) of escalation paths. For example: a job failure first notifies the on‑call SRE via PagerDuty; if not acknowledged within 5 minutes, it escalates to the team lead and automatically opens a GitHub issue with a pre‑filled runbook. No more alert fatigue—only the right people at the right time.

### 3. Atomic Job Locking with Distributed Lease Management
Running the same cron job on multiple nodes? Cron Guard provides a distributed lease mechanism (backed by etcd or Redis) that guarantees exactly‑once execution—even after node crashes or network partitions. The lease duration is adaptive: if a job historically takes 2 minutes, the lease expires after 3 minutes, preventing deadlocks without manual tuning.

## How to Use

### 1. Bootstrap Your First Cron Guard Agent
```bash
# Install the agent (Linux/macOS)
curl -sSL https://get.cron-guard.io | bash

# Initialize with your cluster token
cron-guard init --token YOUR_CLUSTER_TOKEN --endpoint https://api.cron-guard.io

# Register a job
cron-guard register --name "daily-db-backup" --schedule "0 2 * * *" \
  --command "/usr/local/bin/pg_dump -U postgres mydb > /backups/db.sql"
```
The agent automatically collects exit codes, duration, stdout/stderr, and sends heartbeat signals to the control plane.

### 2. Define Alerting Rules with YAML
Create a `rules.yaml` file in your repo:
```yaml
rules:
  - name: "backup-duration-anomaly"
    type: duration
    job: "daily-db-backup"
    baseline: "30d"
    threshold: 2.0  # standard deviations
    actions:
      - type: pagerduty
        service_key: "YOUR_PD_KEY"
      - type: github_issue
        repo: "myorg/infra"
        labels: ["cron-failure", "auto-generated"]
```
Deploy with:
```bash
cron-guard apply -f rules.yaml
```
Cron Guard validates the YAML and creates the alerting pipeline immediately.

### 3. Inspect Execution History via CLI
```bash
# Show the last 50 runs for a specific job
cron-guard logs --job "daily-db-backup" --tail 50

# Export a run’s full context as JSON
cron-guard inspect --run-id "run-abc123" --format json | jq .
```
Each run includes a trace ID that links to your existing APM (e.g., Datadog, Honeycomb) for deep debugging.

## Best Practices

### 1. Always Set a Graceful Kill Timeout
When a cron job hangs, the default SIGTERM may not be enough. In Cron Guard, configure `stop_timeout` (e.g., 30 seconds) and a `kill_signal` (e.g., SIGQUIT) so your job can flush buffers before being force‑killed. This prevents data corruption in stateful jobs like database dumps or file transfers.

### 2. Use Run‑Book Automation for Recurring Failures
Instead of manually SSHing into a box, attach a run‑book script to your alerting rules. For example, if “disk‑cleanup” fails due to low space, Cron Guard can automatically execute a predefined cleanup script, then re‑run the job. This reduces MTTR by 80% and keeps your SLO intact.
