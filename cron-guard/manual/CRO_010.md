## What is Cron Guard?

Cron Guard is a battle-tested, zero-overhead cron job monitoring and alerting tool engineered for modern DevOps pipelines. It eliminates silent failures, detects drift in scheduled task execution, and provides real-time observability into your cron infrastructure — without agent bloat or complex setup.

Unlike generic uptime monitors, Cron Guard is purpose-built for the unique failure modes of scheduled tasks: missed runs, delayed execution, unexpected exits, and environment-specific inconsistencies.

---

## Core Features

### 1. Execution Drift Detection with Adaptive Baselines
Cron Guard doesn’t just check if a job ran — it measures *when* and *how long* it ran relative to historical patterns. Using a rolling window algorithm, it builds an adaptive baseline for each cron task and flags any execution that deviates beyond a configurable threshold (e.g., 2σ from the mean). This catches silent regressions like resource starvation or database lock contention before they become outages.

### 2. Declarative Cron Manifest Validation (Schema + Runtime)
Define your expected cron schedule in a YAML manifest with optional runtime constraints (max duration, expected exit code, environment variables). Cron Guard validates the manifest against the actual system cron table at deploy time and continuously at runtime — preventing configuration drift caused by manual edits, deployment scripts, or container restarts. It also supports multi-node manifest sync for distributed cron clusters.

### 3. Sidecar-Free, Log-Based Monitoring via Syslog/Journald
Unlike traditional agents that require a sidecar or HTTP endpoint, Cron Guard integrates directly with the system’s native logging infrastructure (`syslog`, `journald`, or `Windows Event Log`). It parses cron execution logs in real time, extracting job IDs, exit codes, and timestamps without modifying your existing cron jobs or adding network overhead. This makes it ideal for air-gapped environments and legacy systems.

## How to Use

### Step 1: Deploy the Cron Guard Daemon as a Systemd Service
Download the binary and install it as a systemd service on your target node. Run the following to bootstrap:

```bash
sudo curl -L -o /usr/local/bin/cron-guard https://github.com/xingfangwang-eng/cron-guard/releases/latest/download/cron-guard-linux-amd64
sudo chmod +x /usr/local/bin/cron-guard
sudo cron-guard init --config /etc/cron-guard/config.yaml
sudo systemctl enable --now cron-guard
```

The `init` command generates a default configuration file that binds to your system’s log socket (e.g., `/run/systemd/journal/socket`). No further setup is required for basic monitoring.

### Step 2: Define Your Cron Manifest with Runtime Constraints
Create a manifest file `cron-manifest.yaml` that declares each cron job’s expected schedule, max runtime, and exit code:

```yaml
jobs:
  - name: "db-backup"
    schedule: "0 3 * * *"
    max_duration_sec: 600
    expected_exit_code: 0
    env:
      - "DB_HOST=localhost"
  - name: "cache-purge"
    schedule: "*/30 * * * *"
    max_duration_sec: 30
    expected_exit_code: 0
```

Apply the manifest:

```bash
cron-guard apply --manifest cron-manifest.yaml --validate
```

The `--validate` flag checks for schedule conflicts and missing cron entries before writing the manifest to the daemon’s database.

### Step 3: Configure Alert Routing and Thresholds
Set up alert destinations in `config.yaml`:

```yaml
alerting:
  channels:
    - type: slack
      webhook_url: "https://hooks.slack.com/services/..."
    - type: pagerduty
      routing_key: "your-pd-key"
  thresholds:
    missed_runs: 1
    max_latency_sec: 120
    drift_percent: 20
```

Reload the daemon to apply changes:

```bash
sudo systemctl reload cron-guard
```

Now, if `cache-purge` runs 30 seconds late or `db-backup` exceeds 10 minutes, you’ll receive an alert within 5 seconds.

## Best Practices

### 1. Always Pin Your Cron Guard Version in Production
Cron Guard is under active development. Use a specific release tag (e.g., `v2.3.1`) in your deployment scripts and CI/CD pipelines. Never rely on `latest` — it can introduce breaking changes or unexpected log format shifts. We recommend pinning via SHA256 checksum verification in your infrastructure-as-code (e.g., Terraform or Ansible).

### 2. Combine Cron Guard with a Centralized Log Aggregator
While Cron Guard provides real-time local alerts, pair it with a log shipper (e.g., Vector, Fluentd, or Logstash) to forward parsed cron events to a centralized observability platform (Elasticsearch, Datadog, or Grafana Loki). This gives you historical trend analysis, multi-node dashboards, and the ability to correlate cron failures with other system metrics. Use the `--export-json` flag to output structured logs.
