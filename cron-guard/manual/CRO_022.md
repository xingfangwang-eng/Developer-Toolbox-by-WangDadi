# cron-guard - CRO_022

## Overview

Cron Guard is a high‑performance, distributed cron job orchestration and observability engine designed for mission‑critical scheduled task execution. It provides sub‑second scheduling precision, real‑time anomaly detection, and seamless integration with modern infrastructure stacks. Built for DevOps, SREs, and platform engineers, Cron Guard eliminates silent failures and ensures SLA compliance for cron‑based workloads.

## Detailed Content

### What is Cron Guard?

Cron Guard is an open‑source, event‑driven cron supervisor that monitors, audits, and self‑heals scheduled jobs across heterogeneous environments. It replaces traditional cron daemons with a resilient, state‑aware architecture that supports multi‑node failover, historical execution analysis, and dynamic schedule modification without downtime.

### Core Features

- **Sub‑Millisecond Scheduling with Skew Compensation**  
  Employs a hybrid clock synchronization algorithm (NTP + hardware timestamping) to achieve ≤500µs scheduling jitter across distributed nodes. The `--cron-precision` parameter accepts nanosecond granularity (e.g., `500000ns`), and the scheduler automatically compensates for clock drift using a built‑in Kalman filter. Supports PostgreSQL‑backed schedule persistence with ACID compliance for audit trails.

- **Predictive Anomaly Detection via Execution Time Series**  
  Uses a Holt‑Winters forecasting model on historical execution durations (stored in a time‑series bucket with 1‑second resolution) to predict normal completion windows. If a job exceeds its predicted P99 latency by more than 2σ, Cron Guard triggers an alert via `--alert-webhook` and optionally kills the process with `SIGKILL` after a configurable `--max-grace-period` (default: 30s). The model adapts to weekly and daily seasonality automatically.

- **Zero‑Downtime Schedule Mutation with Versioned Rollback**  
  Supports dynamic schedule changes via a REST API endpoint `PATCH /api/v1/jobs/:id/schedule` without restarting the cron daemon. Each mutation is recorded as a versioned delta (e.g., `v1→v2`), and the system maintains a 90‑day history of schedule changes. A `rollback --to-version` CLI command instantly reverts to any previous schedule state, with atomic PostgreSQL transaction support ensuring consistency across all worker nodes.

### How to Use

1. **Initialize a Cron Guard Cluster with Persistent Backend**  
   Deploy the control plane using the provided Docker Compose stack, which includes a PostgreSQL 15 instance with `cron_guard` schema. Run `cron-guard init --db-uri postgresql://user:pass@host:5432/cron_guard?sslmode=require --node-id worker-01`. This creates the `job_executions` and `schedule_versions` tables, sets up a materialized view for real‑time metrics, and starts the scheduler daemon with a default `--heartbeat-interval=5s`.

2. **Register a Job with Execution Constraints**  
   Use the YAML configuration file (e.g., `jobs/db_backup.yaml`) to define a job:  
   ```yaml
   name: hourly-db-backup
   schedule: "0 * * * *"
   command: "/usr/local/bin/pg_dump -U admin mydb > /backups/db_$(date +%Y%m%d%H%M).sql"
   timeout: 300s
   retry: 3
   retry-delay: 10s
   alert-webhook: "https://hooks.slack.com/services/T00/B00/xxx"
   ```  
   Apply with `cron-guard apply -f jobs/db_backup.yaml`. The system validates the cron expression against a custom parser that supports non‑standard macros like `@hourly` and `@every 30m`.

3. **Monitor Execution and Trigger Rollback**  
   Query real‑time status via `cron-guard status --job-id hourly-db-backup --format json`. If a job fails consecutively beyond `--max-failures=5`, Cron Guard automatically pauses the schedule and sends a `POST` to the `--alert-webhook`. To roll back the schedule to a previous version, run `cron-guard rollback --job-id hourly-db-backup --to-version 3`. The system verifies the version exists in the `schedule_versions` table, locks the job, and atomically updates the active schedule.

### Best Practices

- **Always Configure a Graceful Shutdown Hook**  
  Set `--shutdown-timeout=60s` on the daemon to allow running jobs to complete before termination. Cron Guard sends `SIGTERM` to the job process, waits for the timeout, then issues `SIGKILL`. This prevents orphaned processes and ensures clean state transitions in the execution log.

- **Implement a Two‑Tier Alerting Strategy**  
  Use `--alert-webhook` for immediate notifications (e.g., PagerDuty, Slack) and `--audit-log` for offline analysis. Configure `--audit-log` to write to a separate PostgreSQL table or a syslog endpoint. This decouples real‑time response from long‑term trend analysis, enabling you to tune the predictive model without alert fatigue.
