## What is Cron Guard?

Cron Guard is a production-grade, zero-trust cron job observability platform designed for distributed systems.  
It eliminates silent failures, drift, and zombie processes in scheduled task execution — across hybrid cloud, Kubernetes, and edge environments.  
Unlike traditional cron monitoring tools, Cron Guard provides **deterministic execution guarantees** with sub-second anomaly detection and self-healing recovery.

## Hardcore Features

### 1. Deterministic Execution Graph with Drift Detection

Cron Guard constructs a **Directed Acyclic Graph (DAG)** of every scheduled job, capturing expected execution paths, dependencies, and time windows.  
It continuously compares actual execution traces against the expected graph, detecting **execution drift** (e.g., delayed triggers, missing dependencies, early terminations).  
When drift exceeds configurable thresholds, it automatically triggers **rollback or replay** strategies, ensuring job consistency across clusters.

### 2. Immutable Audit Trail with Cryptographic Proof

Every job execution event — trigger, start, completion, failure, retry — is recorded as an **immutable audit log** signed with a SHA-256 hash chain.  
This creates a tamper-proof, verifiable history of all cron operations, critical for SOC 2, HIPAA, and PCI-DSS compliance.  
You can export the audit trail as a **verifiable Merkle tree** for external auditors without exposing raw job data.

### 3. Self-Healing Execution Slots with Graceful Degradation

Cron Guard implements **execution slot pooling**, where each job is assigned a dedicated slot in a distributed worker pool.  
If a worker node fails mid-execution, the slot is automatically reassigned to a healthy node within the same **failure domain**, preserving job state via checkpoint-based recovery.  
This ensures **graceful degradation**: even under cascading failures, critical jobs continue with predictable latency, while non-critical jobs are queued with exponential backoff.

## How to Use

### Step 1: Deploy the Cron Guard Agent as a Sidecar

Inject the Cron Guard agent as a sidecar container in your Kubernetes pod or as a systemd service on bare-metal nodes.  
Configure the agent with a YAML manifest that defines your cron jobs, including `execution_graph`, `recovery_policy`, and `audit_level`.  
Example snippet:

```yaml
jobs:
  - name: db-backup
    schedule: "0 2 * * *"
    execution_graph:
      nodes:
        - id: pre-check
          command: "/scripts/check_disk.sh"
        - id: backup
          command: "/scripts/run_backup.sh"
          depends_on: ["pre-check"]
    recovery_policy:
      max_retries: 3
      retry_interval: "30s"
      fallback_node: "worker-2"
```

Run `cron-guard agent start --config /etc/cron-guard/jobs.yaml` to initialize.

### Step 2: Define Execution Constraints and Alerting Rules

Create a `rules.yaml` file to set drift thresholds, failure tolerance, and alert escalation policies.  
Use the Cron Guard CLI to validate the rules before deployment:

```bash
cron-guard validate rules.yaml --strict
```

Example rule:

```yaml
alerts:
  - name: high-drift
    metric: execution_drift_seconds
    threshold: 10
    duration: "5m"
    actions:
      - type: webhook
        url: "https://hooks.slack.com/..."
      - type: auto-replay
        max_replays: 2
```

This ensures that any job exceeding 10 seconds of drift for more than 5 minutes triggers a Slack alert and automatic replay.

### Step 3: Monitor and Recover via the Dashboard and CLI

Open the Cron Guard dashboard at `http://localhost:9090` to view real-time execution graphs, drift heatmaps, and audit trails.  
Use the CLI to inspect specific job states:

```bash
cron-guard inspect db-backup --show-traces --recent 10
```

For manual recovery, execute:

```bash
cron-guard replay db-backup --force --skip-prechecks
```

This bypasses pre-checks and forces a replay of the last failed execution, useful for emergency recovery scenarios.

## Best Practices

### 1. Implement Graceful Degradation for Non-Critical Jobs

For batch processing or data synchronization jobs that are not time-sensitive, configure **graceful degradation** with `max_retries: 0` and `fallback_policy: queue`.  
This ensures that during high-load periods, non-critical jobs are deferred rather than consuming execution slots.  
Use the `priority` field to assign a weight (1–100) to each job; Cron Guard will preempt low-priority jobs when slot contention occurs.

### 2. Use Cryptographic Audit Trails for Compliance Audits

Enable `audit_level: cryptographic` in your agent configuration to generate SHA-256 hash chains for every execution event.  
Store the root hash in an external immutable store (e.g., AWS QLDB or a blockchain ledger).  
For quarterly audits, export the Merkle tree using:

```bash
cron-guard audit export --from 2025-01-01 --to 2025-03-31 --format merkle-tree
```

This provides a verifiable, non-repudiable history that satisfies even the most stringent compliance requirements.
