# cron-guard - CRO_029

**A Production-Grade Cron Job Monitoring & Alerting Engine for Modern SaaS Infrastructures**

Cron Guard is not just another cron job tracker. It’s a lightweight, self-healing scheduler designed for DevOps engineers who demand zero-tolerance for silent failures. Built for high-frequency job execution environments, it provides real-time observability, anomaly detection, and automated recovery — all without a heavy agent footprint.

---

## 🔧 Core Features (Hardcore Edition)

### 1. **Predictive Execution Drift Detection**
Cron Guard doesn’t just check if a job ran — it analyzes historical execution patterns to detect *when* a job is running outside its expected window. Using a rolling window of 1000+ execution cycles, it flags jobs that drift by more than 2 standard deviations from the mean. This catches silent slowdowns before they escalate into cascading failures.

### 2. **Dead Man’s Switch with Multi-Channel Fallback**
Each cron job can be assigned a “heartbeat” contract. If a job fails to check in within its TTL (time-to-live), Cron Guard escalates alerts through a priority chain: Slack → PagerDuty → SMS → Telegram. If no acknowledgment is received within 5 minutes, it can trigger a pre-configured rollback or restart script — fully idempotent and isolated by namespace.

### 3. **Idempotent Execution Guard (IEG)**
Prevents race conditions and double-execution in distributed environments. Cron Guard maintains a distributed lock via Redis or PostgreSQL advisory locks. If a job is already running, the new invocation is either queued, dropped, or logged with full context — configurable per job. This is critical for jobs interacting with rate-limited APIs or shared resources.

## 🚀 How to Use

### Step 1: Define a Job Contract in YAML
Create a `cron_guard.yml` file in your project root. This defines the job name, schedule, execution timeout, and failure recovery behavior.

```yaml
jobs:
  - name: sync_inventory
    schedule: "*/5 * * * *"
    command: "python3 /opt/scripts/sync_inventory.py"
    timeout: 120
    max_retries: 3
    retry_delay: 30
    heartbeat_ttl: 300
    alert_chain: ["slack", "pagerduty"]
    lock_enabled: true
```

### Step 2: Initialize the Agent
Run the Cron Guard agent in daemon mode. It will parse the YAML, register each job with the backend, and begin monitoring.

```bash
cron-guard agent --config ./cron_guard.yml --backend redis://localhost:6379
```

The agent will spawn a lightweight HTTP health endpoint at `:9090/health` and a Prometheus metrics endpoint at `:9090/metrics`.

### Step 3: Simulate a Failure & Observe Auto-Recovery
Kill the process of `sync_inventory` manually. Cron Guard will detect the missing heartbeat within 5 seconds, escalate via Slack, and automatically re-execute the job after the retry delay. Check the logs:

```bash
cron-guard logs --job sync_inventory --tail 20
```

You’ll see a structured JSON log output showing the failure detection, alert dispatch, and recovery execution with timestamps and execution IDs.

## 🧠 Best Practices

### 1. **Always Set a `heartbeat_ttl` for Critical Jobs**
If your job writes to a database or triggers an external API, set a `heartbeat_ttl` that is *shorter* than your total expected execution time. This ensures that even if the job hangs (not crashes), Cron Guard will detect the stall and escalate before your SLA is breached.

### 2. **Use `lock_enabled` for Jobs That Mutate Shared State**
Without a distributed lock, two overlapping cron executions can corrupt data or trigger duplicate payments. Always enable `lock_enabled` for jobs that write to the same file, database row, or external API. Pair this with a `lock_ttl` (default: 60s) to prevent deadlocks.
