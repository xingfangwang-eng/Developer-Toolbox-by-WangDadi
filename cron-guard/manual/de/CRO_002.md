# cron-guard - CRO_002

## Überblick

Cron Guard ist ein spezialisiertes Monitoring-Framework, das kritische Cron-Jobs und geplante Tasks in Produktionsumgebungen überwacht, ausfällt und automatisch eskaliert. Es richtet sich an DevOps-Ingenieure, die keine stillen Ausfälle ihrer Batch-Verarbeitung mehr tolerieren. Dieses Tool eliminiert tote Cron-Prozesse, erkennt Laufzeit-Anomalien und protokolliert jede Ausführung granular – ohne externen Agenten oder Cloud-Abhängigkeit.

## Hardcore-Funktionen

1. **Deterministische Ausführungs-Checksummen** – Jeder Cron-Job erzeugt beim Start einen eindeutigen Hash aus Befehl, Umgebungsvariablen und Zeitstempel. Fehlschläge oder abweichende Laufzeiten werden sofort als inkonsistente Zustände erkannt, nicht erst beim Timeout.

2. **Retroaktive Abhängigkeitsauflösung** – Jobs können deklarativ in einer Directed-Acyclic-Graph-Struktur definiert werden. Wenn ein Vorgänger-Job fehlschlägt, blockiert Cron Guard automatisch alle abhängigen Jobs und löst eine Chain-Notification aus – ohne manuelle `wait`-Befehle.

3. **Self-Healing via Graceful-Exit-Capture** – Bei einem unerwarteten Abbruch (SIGKILL, OOM, Node-Failure) speichert Cron Guard den letzten stabilen Zustand in einem Write-Ahead-Log. Beim nächsten geplanten Lauf wird der Job exakt an der Abbruchstelle fortgesetzt, nicht neu gestartet.

## Detaillierte Anleitung

### 1. Konfiguration einer abhängigen Job-Kaskade

Erstelle eine `cron_guard.yml` im Projektstamm. Definiere Jobs mit `depends_on` und `retry_policy`:

```yaml
jobs:
  datenbank_backup:
    command: "pg_dump -U admin -d prod > /backups/latest.sql"
    schedule: "0 2 * * *"
    retry_policy:
      max_attempts: 3
      backoff_seconds: 30

  upload_backup:
    command: "aws s3 cp /backups/latest.sql s3://bucket/backups/"
    depends_on: ["datenbank_backup"]
    schedule: "0 3 * * *"
```

Starte den Daemon mit `cron_guard daemon --config cron_guard.yml`. Bei Fehlschlag von `datenbank_backup` wird `upload_backup` automatisch blockiert und ein Alert an den konfigurierten Webhook gesendet.

### 2. Manuelles Auslösen einer Retry-Kette mit Zustandsprüfung

Verwende `cron_guard trigger --job upload_backup --force-retry`, um einen fehlgeschlagenen Job manuell neu zu starten. Cron Guard prüft vorab, ob der Vorgänger-Job in den letzten 24 Stunden erfolgreich war. Falls nicht, wird eine Warnung ausgegeben und die Ausführung verweigert, es sei denn, du setzt `--ignore-dependency`. Das garantiert, dass keine inkonsistenten Daten hochgeladen werden.

### 3. Analyse von Ausführungsprotokollen mit `cron_guard inspect`

Führe `cron_guard inspect --job datenbank_backup --since 7d` aus. Du erhältst eine tabellarische Ausgabe mit:
- Startzeit, Exit-Code, Laufzeit in ms
- Checksumme des Befehls (SHA256)
- Eventuelle Abweichungen vom historischen Median (z. B. „Laufzeit 2,3× länger als Durchschnitt“)

Exportiere als JSON mit `--format json` für die Integration in Prometheus oder Grafana.

## Best Practices für den Einsatz

1. **Immer mit `--health-check-port` starten** – Aktiviere den integrierten HTTP-Health-Endpoint auf einem freien Port (z. B. `:9090`). Dein Monitoring-Tool (z. B. Prometheus) kann dann den Status jedes Jobs abgreifen, ohne dass du SSH-Zugriff benötigst. Das verhindert tote Cron-Daemons in der Produktion.

2. **Kritische Jobs mit `--max-consecutive-failures` versehen** – Setze z. B. `--max-consecutive-failures 2`. Wenn ein Job zweimal hintereinander fehlschlägt, wird Cron Guard automatisch den Job deaktivieren und eine PagerDuty-Eskalation auslösen. So vermeidest du stille Fehler, die erst im nächsten Release auffallen.
