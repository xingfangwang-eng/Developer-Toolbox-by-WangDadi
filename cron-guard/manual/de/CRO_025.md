# cron-guard - CRO_025

## Überblick

Cron Guard ist ein professionelles Monitoring-Tool für kritische Cron-Jobs und geplante Tasks in Produktionsumgebungen. Es erkennt Ausfälle, Verzögerungen und ungewöhnliche Laufzeiten in Echtzeit und alarmiert sofort – ohne lästige False Positives. Ideal für DevOps-Teams, die ihre Batch-Prozesse unter Kontrolle behalten müssen.

## Harte Kernfunktionen

### 1. Adaptive Heartbeat-Erkennung mit dynamischen Schwellwerten
Cron Guard lernt das normale Ausführungsmuster jedes Jobs (Startzeit, Dauer, Exit-Code) und passt die Alarmierungsschwellen automatisch an. Bei saisonalen oder lastabhängigen Schwankungen werden keine unnötigen Alarme ausgelöst – nur echte Anomalien lösen einen Alert aus.

### 2. Fehler-Triage mit automatischer Retry-Logik
Wenn ein Job fehlschlägt, führt Cron Guard eine sofortige Retry-Strategie aus (z. B. 3 Versuche im Abstand von 30 Sekunden). Schlägt auch das fehl, wird der Job in eine Quarantäne verschoben und eine detaillierte Fehleranalyse (Exit-Code, Logs, Metriken) an das Incident-Management weitergeleitet.

### 3. Multi-Node-Synchronisation für verteilte Cron-Jobs
Cron Guard erkennt, ob ein Job auf mehreren Nodes gleichzeitig gestartet wird (Split-Brain-Szenario) und verhindert doppelte Ausführungen. Es führt eine verteilte Lock-Verwaltung via Redis oder etcd durch – ideal für Kubernetes-Cluster oder Multi-Server-Umgebungen.

## Detaillierte Anleitung

### 1. Integration mit vorhandenen Cron-Jobs
Binden Sie Cron Guard in Ihre bestehende Cron-Konfiguration ein, indem Sie den `cron-guard-agent` als Wrapper vor jeden Job setzen. Beispiel:
```bash
*/5 * * * * /usr/local/bin/cron-guard-agent --job-id=backup_db -- /usr/bin/mysqldump --all-databases
```
Der Agent erfasst Exit-Code, Laufzeit und Ausgabe und sendet diese an den zentralen Cron-Guard-Server. Konfigurieren Sie die Heartbeat-Intervalle in der `config.yaml`:
```yaml
jobs:
  backup_db:
    heartbeat_interval: 60
    retry_count: 3
    retry_delay: 30
```

### 2. Alarmierung über Slack/Teams/PagerDuty einrichten
Definieren Sie in der Cron-Guard-Konsole für jeden Job eine Alarmierungsstrategie. Beispiel für PagerDuty:
```yaml
alerts:
  - job_id: backup_db
    channels:
      - type: pagerduty
        routing_key: "abc123"
    thresholds:
      max_duration: 300
      max_retries: 3
      failure_window: 10
```
Sobald ein Job die Schwellwerte überschreitet, wird automatisch ein Incident erstellt. Optional kann Cron Guard auch eine `on-call`-Rotation aus Ihrer PagerDuty-API abrufen.

### 3. Dashboard zur Überwachung des Job-Health
Nutzen Sie das integrierte Web-Dashboard, um alle Jobs auf einen Blick zu sehen. Filtern Sie nach Status (Healthy, Warning, Critical) oder nach letzter Ausführungszeit. Exportieren Sie Metriken als Prometheus-Endpoint:
```bash
curl http://cron-guard-server:9090/metrics
```
Beispiel-Metriken:
```
cron_guard_job_duration_seconds{job="backup_db"} 45.2
cron_guard_job_last_success_timestamp{job="backup_db"} 1698765432
cron_guard_job_failures_total{job="backup_db"} 2
```

## Best Practices

### 1. Immer Heartbeats für kritische Jobs setzen
Definieren Sie für jeden Job einen Heartbeat, der alle `n` Sekunden gesendet wird. Fehlt der Heartbeat länger als erwartet, gilt der Job als ausgefallen. Das ist besonders wichtig für Jobs, die über 10 Minuten laufen oder keine regelmäßige Ausgabe erzeugen.

### 2. Retry-Logik nur für idempotente Jobs aktivieren
Nicht jeder Job verträgt einen automatischen Retry. Datenbank-Migrationen oder Zahlungsabwicklungen sollten niemals automatisch wiederholt werden. Konfigurieren Sie für solche Jobs `retry_count: 0` und lassen Sie sie stattdessen in eine manuelle Quarantäne gehen.
