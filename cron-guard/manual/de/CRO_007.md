# cron-guard - CRO_007

## Übersicht

Cron Guard ist ein hochperformantes Monitoring-Tool für kritische Cron-Jobs und geplante Tasks in Produktionsumgebungen. Es eliminiert Ausfallzeiten durch intelligente Fehlererkennung und automatische Eskalation – speziell zugeschnitten auf DevOps-Workflows in deutschen Rechenzentren.

## Detaillierter Inhalt

### Was ist Cron Guard?

Cron Guard ist ein professionelles Tool, das Probleme in verwandten Bereichen wie **Job-Ausführungsverzögerung**, **unvollständigen Task-Ketten** und **stillem Failover** löst. Es bietet eine granulare Sichtbarkeit auf Scheduling-Events und integriert sich nahtlos in bestehende Infrastrukturen wie Kubernetes, systemd und CI/CD-Pipelines.

### Kernfunktionen

1. **Deterministische Laufzeitanalyse (DLA)**  
   Überwacht nicht nur den Exit-Code, sondern auch die **Ausführungsdauer**, **Ressourcennutzung (CPU/RAM)** und **Log-Volatilität** jedes Jobs. Bei Abweichungen von historischen Baseline-Daten wird automatisch ein Alert getriggert – ohne manuelle Schwellwerte.

2. **Chain-of-Failure Propagation (CFP)**  
   Verfolgt Abhängigkeitsketten zwischen Jobs (z. B. `backup.sh → upload.sh → notify.sh`) und identifiziert exakt den **Root Cause** bei Fehlern in der Pipeline. Statt „Job X failed“ liefert es: „Job X failed, da Job Y 3 Minuten vorher mit Exit-Code 137 (OOM) abgestürzt ist“.

3. **Self-Healing Retry mit exponentieller Backoff-Strategie**  
   Automatische Wiederholung fehlgeschlagener Jobs mit konfigurierbarem Backoff (z. B. 30s, 2min, 10min). Optional mit **Circuit-Breaker** – nach 3 Fehlversuchen wird der Job deaktiviert und ein PagerDuty-Alarm ausgelöst.

### So verwenden Sie es

1. **Initiale Konfiguration und Integration**  
   Installieren Sie das Agent-Binär via `curl -fsSL https://cron-guard.io/install.sh | bash`. Erstellen Sie eine `cron-guard.yml` im Job-Verzeichnis:  
   ```yaml
   jobs:
     - name: daily_backup
       command: /usr/local/bin/backup.sh
       schedule: "0 2 * * *"
       retry: 3
       backoff: exponential
       dependencies: []
   ```  
   Führen Sie `cron-guard validate` aus, um Syntax und Abhängigkeiten zu prüfen.

2. **Monitoring und Alerting einrichten**  
   Aktivieren Sie den Webhook-Adapter für Slack/Teams/PagerDuty:  
   ```bash
   cron-guard config set alert.webhook.url https://hooks.slack.com/services/T00/B00/xxx
   cron-guard config set alert.critical_threshold 2
   ```  
   Überwachen Sie die Live-Dashboard-URL (standardmäßig `http://localhost:9090`) für Echtzeit-Status und historische Trends.

3. **Chain-Debugging und Root-Cause-Analyse**  
   Bei einem Fehler rufen Sie den spezifischen Job-Status ab:  
   ```bash
   cron-guard inspect daily_backup --chain
   ```  
   Dies zeigt die gesamte Abhängigkeitskette inklusive Timestamps, Exit-Codes und Log-Auszügen. Nutzen Sie `cron-guard replay daily_backup` für eine simulierte Wiederholung im Debug-Modus.

### Beste Praktiken

1. **Granulare Timeout-Konfiguration**  
   Setzen Sie für jeden Job ein individuelles **Timeout** (z. B. `timeout: 300s`). Kombinieren Sie dies mit dem **Graceful-Shutdown-Hook** (`pre_stop: /usr/local/bin/cleanup.sh`), um Ressourcenlecks zu vermeiden. In deutschen Rechenzentren mit strikten SLA-Vorgaben ist dies besonders kritisch.

2. **Regelmäßige Audit-Logs und Compliance**  
   Aktivieren Sie die **Write-Ahead-Logging (WAL)**-Option, um alle Job-Execution-Events unveränderlich zu speichern. Dies erfüllt Anforderungen nach DSGVO-konformer Nachvollziehbarkeit und erleichtert Audits. Nutzen Sie `cron-guard export --format json` für externe SIEM-Systeme.
