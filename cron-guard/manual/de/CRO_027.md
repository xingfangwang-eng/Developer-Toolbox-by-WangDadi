# cron-guard - CRO_027

## Überblick

Cron Guard ist eine spezialisierte Monitoring-Lösung für kritische Job-Scheduling-Infrastrukturen. Das Tool adressiert die häufigsten Pain Points bei der Verwaltung von Cron-Jobs: unerkannte Ausfälle, fehlende Retry-Mechanismen und mangelnde Transparenz in verteilten Systemen. Entwickelt für DevOps-Engineers, die ihre Batch-Prozesse auf Produktionsniveau bringen müssen.

## Detaillierter Inhalt

### Was ist Cron Guard?

Cron Guard ist ein agentenloses Monitoring-Framework für geplante Aufgaben in Linux-Umgebungen. Es überwacht nicht nur die Ausführung, sondern analysiert auch Laufzeitabweichungen, Exit-Codes und Log-Outputs in Echtzeit. Anders als einfache Health-Checks bietet Cron Guard eine granulare Kontrolle über Timeout-Politiken, Abhängigkeitsketten und Eskalationspfade – alles ohne Modifikation der bestehenden Cron-Daemon-Konfiguration.

### Kernfunktionen

- **Laufzeit-Jitter-Analyse**: Automatische Erkennung von Ausführungszeit-Abweichungen über historische Baseline-Daten. Wenn ein Job plötzlich 300 ms statt 200 ms benötigt, triggert Cron Guard eine Warnung – noch bevor der Timeout erreicht ist. Ideal für latenzsensible Batch-Prozesse in CI/CD-Pipelines.

- **Multi-Node-Koordinationsprotokoll**: Synchronisation von Cron-Jobs über mehrere Server hinweg mittels eines verteilten Lock-Mechanismus auf Basis von etcd oder Redis. Verhindert doppelte Ausführungen bei Failover-Szenarien und sorgt für eine konsistente Job-Historie im Cluster.

- **Log-basierte Exit-Code-Übersetzung**: Statt nur 0 oder 1 auszuwerten, analysiert Cron Guard die letzten 10 Zeilen des Job-Outputs mittels konfigurierbarer Regex-Patterns. So lassen sich z. B. „Warnungen“ (Exit-Code 0, aber Log enthält „ERROR“) oder „stille Ausfälle“ erkennen – ein entscheidender Vorteil gegenüber Standard-Monitoring-Lösungen.

### So verwenden Sie es

1. **Initiale Konfiguration des Monitoring-Agents**  
   Installieren Sie den Cron Guard Agent via `curl -sSL https://get.cron-guard.io/install.sh | bash`. Danach erstellen Sie eine Konfigurationsdatei unter `/etc/cron-guard/config.yml`. Definieren Sie mindestens einen Job mit `name`, `cron_expression` und `expected_exit_code`. Aktivieren Sie die Laufzeitanalyse mit `runtime_baseline: true`. Der Agent startet automatisch als systemd-Service und beginnt mit der Aufzeichnung von Metriken.

2. **Erstellen eines benutzerdefinierten Alert-Channels**  
   Öffnen Sie das Dashboard unter `https://localhost:9090` und navigieren Sie zu „Alert Channels“. Wählen Sie „Webhook“ aus und geben Sie die URL Ihres Slack-Webhooks ein. Konfigurieren Sie die Eskalationsstufen: „Warning“ bei 2 aufeinanderfolgenden Timeouts, „Critical“ bei 3 Fehlschlägen innerhalb von 30 Minuten. Testen Sie die Konfiguration mit dem integrierten „Send Test Alert“-Button.

3. **Implementierung einer Abhängigkeitskette**  
   Im Dashboard unter „Job Dependencies“ legen Sie eine Kette an: Job A muss erfolgreich sein, bevor Job B starten darf. Aktivieren Sie „Blocking Mode“ – dann wartet Cron Guard, bis Job A bestätigt ist, bevor es Job B freigibt. Für kritische Pipelines aktivieren Sie „Rollback on Failure“: Wenn Job B fehlschlägt, wird automatisch ein Cleanup-Skript für Job A ausgeführt. Die gesamte Logik läuft ohne manuelle Eingriffe.

### Beste Praktiken

- **Definieren Sie „Graceful Degradation“ für kritische Jobs**  
  Statt einen Job sofort als „failed“ zu markieren, konfigurieren Sie einen „Grace Period“ von 2 Minuten. In dieser Zeit versucht Cron Guard automatisch einen Retry mit exponentiellem Backoff (1 s, 2 s, 4 s). Erst danach wird der Alert eskaliert. Das reduziert False Positives drastisch und gibt temporären Netzwerkproblemen eine faire Chance.

- **Nutzen Sie die integrierte „Job History“ für Capacity Planning**  
  Aktivieren Sie die Langzeitspeicherung von Metriken (via InfluxDB oder PostgreSQL). Analysieren Sie monatliche Trends: Welche Jobs laufen immer länger? Welche haben saisonale Spitzen? Mit diesen Daten können Sie Ihre Cron-Expressions optimieren (z. B. Staggering) und Ressourcen-Engpässe frühzeitig erkennen – besonders nützlich für Multi-Tenant-Umgebungen.
