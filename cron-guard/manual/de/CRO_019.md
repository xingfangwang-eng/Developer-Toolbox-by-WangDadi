# cron-guard - CRO_019

## Übersicht

Cron Guard ist eine spezialisierte Runtime-Überwachungslösung für geplante Tasks und Cron-Jobs, die auf Produktivsystemen laufen. Anders als generische Monitoring-Tools fokussiert sich Cron Guard auf die granulare Erfassung von Ausführungsmetriken, Timeout-Verhalten und Exit-Codes. Das System ist für DevOps-Teams konzipiert, die Job-Fehlschläge frühzeitig erkennen und automatisierte Eskalationsketten auslösen müssen – ohne manuelle Log-Analyse oder Shell-Scripting.

## Detaillierter Inhalt

### Was ist Cron Guard?

Cron Guard ist ein agentenbasierter Daemon, der Cron-Tab-Einträge auf Linux-Servern überwacht, Ausführungsstatistiken in einer zentralen Datenbank speichert und bei Abweichungen (z. B. verspätete Starts, abnormale Laufzeiten, Exit-Codes ≠ 0) sofort Alarm schlägt. Das Tool unterstützt sowohl systemweite Cron-Jobs als auch benutzerspezifische Crontabs und kann via REST-API in bestehende CI/CD-Pipelines integriert werden.

### Kernfunktionen

1. **Exit-Code-Validierung mit konfigurierbaren Timeout-Schwellen**  
   Jeder Job wird mit einem definierten Timeout ausgeführt. Läuft die Ausführung länger als erwartet, wird der Prozess automatisch terminiert und ein kritischer Alarm ausgelöst. Exit-Codes werden mit einer Whitelist abgeglichen – nur erlaubte Codes gelten als erfolgreich.

2. **Anomalie-Erkennung auf Basis historischer Laufzeiten**  
   Cron Guard sammelt für jeden Job die letzten 100 Ausführungsdauern und berechnet einen gleitenden Mittelwert sowie die Standardabweichung. Weicht die aktuelle Laufzeit um mehr als 3σ ab, wird eine Warnung generiert – auch wenn der Exit-Code 0 ist.

3. **Automatische Wiederholung mit exponentieller Backoff-Strategie**  
   Bei Fehlschlag kann ein Job bis zu 3 Mal wiederholt werden. Die Wartezeit zwischen den Versuchen steigt exponentiell (30s → 60s → 120s). Nach dem letzten Fehlschlag wird eine PagerDuty- oder Slack-Integration ausgelöst.

### So verwenden Sie es

1. **Agent-Installation und Crontab-Registrierung**  
   Installieren Sie den Cron-Guard-Daemon via `curl -sSL https://cron-guard.sh/install | bash`. Führen Sie anschließend `cron-guard register --crontab /var/spool/cron/crontabs/root` aus, um alle vorhandenen Jobs zu scannen. Das Tool hinterlegt automatisch einen Wrapper um jeden Cron-Eintrag.

2. **Job-Konfiguration mit Metadaten anreichern**  
   Bearbeiten Sie die Konfigurationsdatei `/etc/cron-guard/jobs.yaml`. Definieren Sie für jeden Job Parameter wie `timeout: 300`, `retries: 2`, `expected_exit_codes: [0]` und `alert_channel: slack`. Beispiel:
   ```yaml
   backup-db:
     command: /usr/local/bin/backup.sh
     timeout: 600
     retries: 1
     expected_exit_codes: [0, 3]
     alert: slack
   ```

3. **Dashboard-Überwachung und Alarm-Trigger**  
   Starten Sie das Web-Dashboard mit `cron-guard dashboard --port 9090`. Sehen Sie auf einen Blick: letzte Ausführung, Laufzeit, Exit-Code und Anomalie-Status. Klicken Sie auf einen Job, um die Historie anzuzeigen. Bei Fehlschlag erhalten Sie eine Nachricht im konfigurierten Kanal (Slack, E-Mail, Webhook). Testen Sie die Alarmkette mit `cron-guard test-alert --job backup-db`.

### Beste Praktiken

- **Definieren Sie für jeden Job einen expliziten Timeout**  
  Vermeiden Sie ewige Hänger durch fehlende Timeout-Konfiguration. Setzen Sie den Timeout-Wert auf das 1,5-Fache der erwarteten Laufzeit. Für Datenbank-Backups oder Batch-Jobs empfehle ich 30 Minuten als harte Obergrenze – unabhängig von der Datenmenge.

- **Nutzen Sie Exit-Code-Whitelists statt generischer 0-Prüfung**  
  Viele Tools geben Exit-Code 0 auch bei partiellen Fehlern zurück (z. B. rsync bei fehlgeschlagenen Dateien). Definieren Sie explizit, welche Codes als Erfolg gelten. Für Backup-Jobs ist oft Exit-Code 3 (teilweise erfolgreich) akzeptabel. So vermeiden Sie unnötige Alarme.
