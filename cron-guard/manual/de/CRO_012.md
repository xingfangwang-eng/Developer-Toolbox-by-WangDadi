# cron-guard - CRO_012

## Überblick

Cron Guard ist ein hochperformantes Monitoring-Tool für mission-critical Cron-Jobs und geplante Tasks in verteilten Systemen. Es eliminiert Silent Failures, erkennt Latenzanomalien und liefert granulare Einblicke in Scheduling-Engpässe – ohne Agenten, ohne Overhead.

## Hardcore-Funktionen

### 1. **Predictive Execution Drift Detection**
Cron Guard analysiert historische Ausführungszeiträume und erkennt frühzeitig Abweichungen vom erwarteten Startfenster. Bei Überschreitung eines konfigurierbaren Toleranzwerts (z. B. ±3σ) wird automatisch ein `SLA_BREACH`-Event ausgelöst, ohne dass ein Job tatsächlich fehlschlägt.

### 2. **Multi-Node Heartbeat Synchronization**
Für verteilte Cron-Instanzen (z. B. Kubernetes CronJobs, Airflow, systemd-timer) synchronisiert Cron Guard Heartbeat-Signale über einen dedizierten, asynchronen Pub/Sub-Kanal. Dadurch lassen sich Split-Brain-Szenarien und doppelte Ausführungen in Clustern zuverlässig erkennen und loggen.

### 3. **Granular Failure Attribution via Exit Code Mapping**
Cron Guard mapped Exit-Codes aus Shell-Skripten, Python-Prozessen und Docker-Containern auf semantische Fehlerkategorien (z. B. `OOM_KILL`, `CONNECTION_TIMEOUT`, `CONFIG_PARSE_ERROR`). Diese werden in einem strukturierten JSON-Log mit Timestamp und Host-ID persistiert – bereit für ELK, Grafana oder OpenSearch.

## Detaillierte Anleitung (3 Schritte)

### Schritt 1: Installation & Initialisierung des Daemon-Prozesses
```bash
curl -sSL https://cron-guard.io/install.sh | bash
cron-guard init --config /etc/cron-guard/config.yaml
```
Der Daemon startet als `systemd`-Service und überwacht standardmäßig `/var/log/cron` sowie `journalctl -u cron`.  
**Tipp:** Nutzen Sie `--watch-recursive` für Unterverzeichnisse.

### Schritt 2: Definition eines Monitoring-Profils für einen kritischen Job
Erstellen Sie eine Profil-Datei unter `/etc/cron-guard/profiles/db-backup.yaml`:
```yaml
job:
  name: "postgres-full-backup"
  schedule: "0 2 * * *"
  expected_duration: "30m"
  max_retries: 3
  alert_channels:
    - type: slack
      webhook: "https://hooks.slack.com/services/T.../B.../xxx"
    - type: webhook
      url: "https://monitor.example.com/cron-guard/alerts"
  heartbeat_interval: "15s"
```
Laden Sie das Profil mit:
```bash
cron-guard profile apply db-backup.yaml
```

### Schritt 3: Analyse & Alerting im Live-Dashboard
Öffnen Sie das Web-Dashboard unter `http://localhost:8484`:
- **Execution Timeline**: Zeigt jeden Lauf als vertikalen Balken (Farbe = Status).
- **Anomaly Score**: Prozentuale Abweichung vom gleitenden Durchschnitt.
- **Alert History**: Gefiltert nach `critical`, `warning`, `info`.  
Exportieren Sie Logs direkt als CSV oder JSON über die REST-API (`GET /api/v1/jobs/{job_id}/history`).

## Best Practices für produktiven Einsatz

### 1. **Heartbeat-Pflicht für alle Jobs > 10 Minuten Laufzeit**
Aktivieren Sie `heartbeat_interval` für Jobs, die länger als 10 Minuten laufen. Dadurch erkennen Sie Deadlocks oder Hänger bereits nach 2 versäumten Heartbeats – noch bevor der Exit-Code eintrifft.  
**Empfehlung:** Setzen Sie `heartbeat_interval` auf 25 % der erwarteten Laufzeit (z. B. 5 Minuten bei 20 Minuten Laufzeit).

### 2. **Separate Alert-Kanäle für kritische vs. informative Events**
Nutzen Sie zwei unabhängige Webhook-Endpunkte: einen für `critical` (z. B. PagerDuty, Slack mit `@here`) und einen für `warning`/`info` (z. B. E-Mail oder Discord). Vermeiden Sie Alarm Fatigue durch deduplizierte Events – Cron Guard unterstützt nativ `cooldown_period` in Sekunden pro Alert-Typ.
