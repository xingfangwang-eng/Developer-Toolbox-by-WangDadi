# cron-guard - CRO_013

## Übersicht

Cron Guard ist ein spezialisiertes Monitoring-Framework für **kritische Cron-Jobs und geplante Tasks** in Produktionsumgebungen. Es eliminiert die häufigsten Ausfallursachen wie Timeout-Silent-Failures, fehlerhafte Exit-Codes oder unerwartete Laufzeitüberschreitungen. Dieses Tool richtet sich an DevOps-Ingenieure und SREs, die ihre Job-Zuverlässigkeit auf Enterprise-Niveau heben möchten.

## Detaillierter Inhalt

### Was ist Cron Guard?

Cron Guard ist ein **ereignisgesteuerter Job-Wächter**, der weit über einfache Heartbeat-Checks hinausgeht. Es analysiert Exit-Codes, Log-Volumina und Laufzeitmuster in Echtzeit und alarmiert granular bei Abweichungen. Im Gegensatz zu generischen Monitoring-Lösungen bietet es eine native Integration in **systemd, cron, Jenkins und Kubernetes CronJobs** und erkennt Silent-Failures, die herkömmliche Tools übersehen.

### Kernfunktionen

- **Exit-Code-Validierung mit Log-Korrelation**  
  Jeder Job wird nicht nur auf Exit-Code ≠ 0 geprüft, sondern auf semantische Exit-Codes (z. B. 64 = Konfigurationsfehler, 75 = temporärer Fehler). Die Logs werden automatisch nach Stacktraces oder spezifischen Mustern durchsucht und mit dem Exit-Code verknüpft – für eine sofortige Root-Cause-Analyse.

- **Laufzeit-Anomalie-Detektion mittels ML-Mustererkennung**  
  Cron Guard lernt die historische Laufzeit jedes Jobs und erkennt Anomalien wie plötzliche Verkürzungen (Hinweis auf fehlgeschlagene Subprozesse) oder Verlängerungen (Hinweis auf Deadlocks). Abweichungen ab 3σ werden als kritisch eingestuft und lösen automatisierte Eskalationsketten aus.

- **Multi-Environment-Synchronisation mit Failover-Routing**  
  Bei Ausfall eines Cron-Servers oder einer Zone leitet Cron Guard Jobs automatisch auf einen sekundären Worker um – inklusive State-Management, um doppelte Ausführungen zu vermeiden. Dies erfolgt über einen konsistenten Hash-Ring und ist ideal für Geo-Redundanz-Setups.

### So verwenden Sie es

1. **Initiale Konfiguration mit YAML-Definition**  
   Erstellen Sie eine `cron_guard.yml` im Wurzelverzeichnis Ihres Projekts. Definieren Sie jeden Job mit `name`, `schedule` (Cron-Expression), `command`, `timeout` (in Sekunden) und `retry_policy` (z. B. `exponential_backoff`). Beispiel:
   ```yaml
   jobs:
     backup_db:
       schedule: "0 3 * * *"
       command: "/usr/local/bin/backup.sh --full"
       timeout: 1200
       retry_policy:
         max_retries: 3
         backoff_factor: 2.0
   ```
   Validieren Sie die Konfiguration mit `cron-guard validate`.

2. **Agent-Start mit Daemon-Modus**  
   Starten Sie den Cron Guard Agent als systemd-Service:  
   ```bash
   sudo cron-guard agent start --config /etc/cron_guard.yml --daemon
   ```  
   Der Agent registriert sich automatisch im zentralen Dashboard (Port 9090) und beginnt mit dem Monitoring. Überprüfen Sie den Status mit `cron-guard status --json`.

3. **Alarm-Integration und automatisierte Remediation**  
   Konfigurieren Sie Alarme in der `alerts`-Sektion der YAML-Datei. Binden Sie Webhooks für Slack, PagerDuty oder E-Mail ein. Für kritische Jobs können Sie eine automatisierte Remediation definieren, z. B.:
   ```yaml
   alerts:
     on_failure:
       webhook: "https://hooks.slack.com/services/..."
       remediation:
         action: "restart"
         max_attempts: 2
   ```
   Testen Sie die Alarmkette mit `cron-guard test-alert --job backup_db`.

### Beste Praktiken

- **Idempotenz als Grundlage vor dem Monitoring**  
  Stellen Sie sicher, dass jeder Cron-Job idempotent ist – d. h., mehrfache Ausführungen führen zum gleichen Ergebnis. Cron Guard erkennt doppelte Ausführungen und markiert sie als Warnung, aber nur idempotente Jobs können sicher auf Failover-Systeme umgeleitet werden.

- **Granulare Timeout-Definitionen pro Job**  
  Vermeiden Sie generische Timeouts. Definieren Sie für jeden Job einen spezifischen Timeout-Wert basierend auf historischen Laufzeiten. Nutzen Sie die integrierte `--dry-run`-Option, um Laufzeiten zu erfassen:  
  ```bash
  cron-guard dry-run --job backup_db --iterations 10
  ```  
  So erhalten Sie präzise Basiswerte für die Anomalieerkennung.
