# cron-guard - CRO_011

## Überblick

Cron Guard ist ein spezialisiertes Monitoring-Framework für Cron-Jobs und geplante Aufgaben in Linux-Umgebungen. Es erkennt frühzeitig Ausfälle, Verzögerungen und fehlerhafte Ausführungen, bevor diese zu Dateninkonsistenzen oder Produktionsausfällen führen. Das Tool eignet sich für DevOps-Teams, die eine zuverlässige Automatisierung sicherstellen müssen, ohne auf komplexe Drittanbieter-Plattformen angewiesen zu sein.

## Detaillierter Inhalt

### Was ist Cron Guard?

Cron Guard ist ein Open-Source-Systemüberwachungsdienst, der speziell für die Ausführung von Cron-Jobs in verteilten und containerisierten Umgebungen entwickelt wurde. Es protokolliert nicht nur Exit-Codes, sondern analysiert auch Laufzeiten, Ressourcenverbrauch und Fehlermuster. Das Tool lässt sich nahtlos in bestehende CI/CD-Pipelines integrieren und benachrichtigt Teams proaktiv über Slack, E-Mail oder Webhooks.

### Kernfunktionen

1. **Zeitfenster-basierte Ausführungserkennung** – Chron Guard verwendet eine adaptive Baseline-Logik, um die erwartete Laufzeit jedes Jobs zu berechnen. Abweichungen von mehr als 20 % werden als kritische Alarme markiert, nicht nur als Warnung. Dies verhindert falsch positive Meldungen, die bei starren Timeout-Werten auftreten.

2. **Exit-Code-Validierung mit Kontextanalyse** – Standard-Cron meldet nur Exit-Codes. Cron Guard interpretiert Shell-Signale (SIGTERM, SIGKILL) und erkennt, ob ein Job aufgrund von Ressourcenmangel (OOM) oder einer Timeout-Überschreitung abgebrochen wurde. Diese Kontextinformationen werden in strukturierten Logs gespeichert und sind über eine REST-API abrufbar.

3. **Job-Abhängigkeitsgraph mit automatischer Wiederholungslogik** – Definiere Abhängigkeiten zwischen Jobs (z. B. „Job B startet erst, wenn Job A erfolgreich beendet wurde“). Bei Fehlern führt Cron Guard automatisch eine exponentielle Backoff-Wiederholung durch, ohne dass eine manuelle Intervention erforderlich ist. Dies reduziert die Fehlerquote in komplexen Workflows um bis zu 70 %.

### So verwenden Sie es

1. **Initiale Konfiguration des Cron-Guard-Agenten**  
   Installiere den Agenten auf allen Nodes, die Cron-Jobs ausführen. Verwende dazu das offizielle Debian-Paket:
   ```bash
   sudo apt-get install cron-guard-agent
   ```
   Anschließend konfigurierst du die `config.yml` mit den Endpunkten deiner Benachrichtigungskanäle (z. B. Slack-Webhook, SMTP-Server). Starte den Dienst mit:
   ```bash
   sudo systemctl enable --now cron-guard-agent
   ```

2. **Einrichtung eines Überwachungsjobs**  
   Erstelle eine Job-Definition im Format YAML. Beispiel für einen Datenbank-Backup-Job:
   ```yaml
   jobs:
     db_backup:
       command: "/usr/local/bin/backup.sh"
       schedule: "0 3 * * *"
       timeout: 3600
       retries: 3
       dependencies: []
   ```
   Führe dann den Befehl aus, um den Job zu registrieren:
   ```bash
   cron-guard-cli register --file /etc/cron-guard/jobs/db_backup.yaml
   ```

3. **Überwachung und Alarmierung aktivieren**  
   Rufe das Dashboard auf, um Live-Statistiken zu sehen:
   ```bash
   cron-guard-dashboard --port 8080
   ```
   Wenn ein Job fehlschlägt, erhältst du eine detaillierte Nachricht im Format:
   ```
   [CRITICAL] Job 'db_backup' fehlgeschlagen um 03:15 UTC. Exit-Code: 137 (SIGKILL). Laufzeit: 45 min (erwartet: 30 min). Empfehlung: Speicherlimit prüfen.
   ```

### Beste Praktiken

- **Strukturiere Cron-Jobs nach Domänen** – Lege für jede Microservice-Domäne einen eigenen Cron-Guard-Namespace an. Verwende Tags wie `database`, `cache` oder `notification`, um Alarme nach Team zu filtern. Das reduziert Rauschen und verbessert die Reaktionszeit bei kritischen Ausfällen.

- **Implementiere einen Health-Check-Endpunkt für jeden Job** – Lasse deine Cron-Jobs am Ende ihrer Ausführung einen HTTP-Endpunkt aufrufen (z. B. `https://your-api.com/health/cron/backup`). Cron Guard kann diesen Endpunkt zusätzlich überwachen und bei fehlenden Heartbeats innerhalb von 5 Minuten eine Eskalation auslösen.
