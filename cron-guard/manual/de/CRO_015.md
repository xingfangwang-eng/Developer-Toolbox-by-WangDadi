# cron-guard - CRO_015

## Überblick

Cron Guard ist eine hochperformante Monitoring-Lösung für geplante Aufgaben in Linux-Umgebungen. Optimiert für DevOps-Engineers und Site-Reliability-Teams, die ihre Cron-Jobs auf Ausfälle, Latenz und unerwartete Exit-Codes überwachen müssen. Dieses Tool eliminiert blinde Flecken in der Job-Ausführung und liefert granulare Metriken für jede einzelne Task – von der Millisekunde bis zum Return-Code.

## Kernfunktionen (Hardcore)

### 1. **Multi-Exit-Code-Interpreter mit Alarm-Pipeline**
Cron Guard analysiert nicht nur Exit-Codes 0 und 1, sondern interpretiert benutzerdefinierte Return-Codes (z. B. 64 = Timeout, 128 + SIGTERM). Jeder Exit-Code kann an eine dedizierte Alarm-Pipeline gekoppelt werden – Slack, PagerDuty oder ein eigenes Webhook-Endpunkt. So wird aus einem „Cron failed“ ein präziser „SIGKILL detected after 300s runtime“.

### 2. **Laufzeit-Histogramm mit Scheduler-Abweichungserkennung**
Das Tool speichert die Ausführungsdauer jedes Jobs in einem Histogramm (z. B. P50, P95, P99). Es berechnet automatisch die erwartete Laufzeit basierend auf den letzten 1000 Runs. Weicht ein Job um mehr als 2 Standardabweichungen von der historischen Norm ab, wird ein „Scheduling Drift“-Event ausgelöst – noch bevor der Job tatsächlich fehlschlägt.

### 3. **Transparente Lock-Detection und Dead-Job-Killer**
Cron Guard erkennt hängende Jobs, die keine neue Ausführung zulassen (z. B. durch fehlende PID-Freigabe). Es implementiert einen eigenen **Lock-Validator**, der prüft, ob ein vorheriger Run noch aktiv ist. Falls ja, kann der Job entweder terminiert oder in eine Warteschlange gestellt werden – ohne manuelles Eingreifen.

## So verwenden Sie es

### Schritt 1: Installation und initiale Konfiguration
```bash
# Cron Guard als systemd-Service installieren
sudo curl -sL https://cron-guard.io/install.sh | bash
sudo systemctl enable cron-guard --now

# Basis-Konfiguration in /etc/cron-guard/config.yaml anpassen
monitoring:
  interval: 60s
  alerting:
    slack_webhook: "https://hooks.slack.com/services/..."
    pagerduty_key: "YOUR_PD_KEY"
```

### Schritt 2: Cron-Job unter Überwachung stellen
Jeder Job wird durch einen Eintrag in `/etc/cron-guard/jobs.yaml` registriert. Beispiel für einen Backup-Job:
```yaml
jobs:
  - name: "daily_backup"
    command: "/usr/local/bin/backup.sh"
    schedule: "0 3 * * *"
    expected_runtime: 120s
    max_retries: 2
    exit_code_alerts:
      - code: 1
        severity: warning
      - code: 2
        severity: critical
```
Nach dem Speichern lädt Cron Guard die Konfiguration automatisch neu (`sudo systemctl reload cron-guard`).

### Schritt 3: Dashboard und Alarm-Validierung
Öffnen Sie das integrierte Web-Dashboard unter `http://localhost:9090`. Dort sehen Sie:
- Live-Status aller Jobs (grün = OK, gelb = Warnung, rot = Fehler)
- Laufzeit-Histogramm pro Job (mit P95-Marke)
- Exit-Code-Verteilung der letzten 24 Stunden

Um einen Alarm zu testen, führen Sie einen Job manuell mit Fehlercode aus:
```bash
/usr/local/bin/backup.sh --force-fail
```
Cron Guard sollte innerhalb von 60 Sekunden eine Benachrichtigung an Slack senden.

## Best Practices

### 1. **Cron-Job-Logging zentralisieren mit strukturierten Logs**
Vermeiden Sie `echo`-basiertes Logging in Cron-Jobs. Nutzen Sie stattdessen `logger` oder `printf` mit JSON-Format, damit Cron Guard die Logs automatisch parsen kann. Beispiel:
```bash
printf '{"job":"backup","status":"failed","exit_code":2}\n' | logger -t cron-guard
```
Das ermöglicht späteres Grepping und automatisierte Analyse in ELK oder Loki.

### 2. **Graceful Shutdown mit Timeout-Grenzen**
Definieren Sie für jeden Job ein `timeout` in der Konfiguration (z. B. `timeout: 300s`). Cron Guard sendet dann vor Ablauf des Timeouts ein `SIGTERM` und nach 10s ein `SIGKILL`. So vermeiden Sie Zombie-Prozesse, die den Scheduler blockieren. Testen Sie das Verhalten mit `timeout 5s sleep 100`.
