## Detaillierter Inhalt

### Was ist Cron Guard?

Cron Guard ist ein professionelles, in Rust implementiertes CLI-Tool, das speziell für die Überwachung, Protokollierung und Fehlerbehandlung von Cron-Jobs entwickelt wurde. Es eliminiert typische Probleme wie stille Ausfälle, fehlende Benachrichtigungen oder inkonsistente Ausführungszeiten, indem es eine granulare Kontrolle über Job-Status, Timeouts und Wiederholungslogik bietet.

### Kernfunktionen

1.  **Automatische Timeout-Erkennung mit Eskalations-Pipeline**  
    Jeder Cron-Job wird mit einem konfigurierbaren Timeout versehen. Überschreitet ein Job diese Grenze, wird nicht nur ein Alarm ausgelöst, sondern das Tool führt automatisch eine vordefinierte Eskalationskaskade aus: von E-Mail-Benachrichtigung über Slack-Webhook bis hin zu PagerDuty-Integration. So bleibt kein hängender Job unbemerkt.

2.  **Granulare Wiederholungslogik mit exponentieller Backoff-Strategie**  
    Statt einfacher Wiederholungen implementiert Cron Guard eine intelligente Retry-Routine. Fehlgeschlagene Jobs werden nach einem exponentiellen Backoff-Muster erneut gestartet (z. B. 1s, 2s, 4s, 8s). Nach maximal 5 Fehlversuchen wird der Job als „kritisch“ markiert und eine separate Alarmierung ausgelöst.

3.  **Metrik-Export für Prometheus & Grafana**  
    Jeder ausgeführte Job exportiert automatisch Metriken wie `cron_job_duration_seconds`, `cron_job_status` (0/1) und `cron_job_retry_count`. Diese können direkt von Prometheus gescrapt werden, sodass Echtzeit-Dashboards in Grafana erstellt werden können – inklusive Historie über die letzten 30 Tage.

### So verwenden Sie es

1.  **Konfiguration einer neuen Überwachung**  
    Erstellen Sie eine YAML-Datei (`cron_guard.yml`) mit folgendem Inhalt:
    ```yaml
    jobs:
      backup_db:
        command: "pg_dump -U admin production_db > /backups/latest.sql"
        timeout: 300s
        retry:
          max_attempts: 3
          backoff: exponential
        notifications:
          - type: email
            to: "ops@company.de"
          - type: slack
            webhook: "https://hooks.slack.com/services/..."
    ```
    Validieren Sie die Konfiguration mit `cron-guard validate cron_guard.yml`.

2.  **Starten des Daemon-Modus**  
    Führen Sie `cron-guard daemon --config cron_guard.yml` aus. Das Tool läuft als Hintergrundprozess, überwacht die definierten Jobs und protokolliert jede Ausführung in `/var/log/cron_guard/`. Nutzen Sie `--log-level debug` für detaillierte Ausgaben bei der Fehlersuche.

3.  **Integration in bestehende Cron-Umgebungen**  
    Ersetzen Sie in Ihrer Crontab den ursprünglichen Befehl durch den Cron Guard-Wrapper:
    ```
    */5 * * * * /usr/local/bin/cron-guard run backup_db
    ```
    Der Wrapper startet den Job, misst die Laufzeit und sendet Metriken an den konfigurierten Endpunkt. Überwachen Sie den Status mit `cron-guard status backup_db`.

### Beste Praktiken

1.  **Definieren Sie Timeouts basierend auf historischen Daten**  
    Analysieren Sie die durchschnittliche Laufzeit Ihrer Jobs über die letzten 7 Tage (z. B. mit `cron-guard metrics --since 7d`). Setzen Sie das Timeout auf das 1,5-fache des Maximalwerts. So vermeiden Sie false positives bei saisonalen Lastspitzen.

2.  **Implementieren Sie eine gestaffelte Alarmierung**  
    Nutzen Sie die Eskalations-Pipeline: Bei einem ersten Fehlschlag reicht eine E-Mail an das Team. Nach dem dritten Fehlschlag innerhalb einer Stunde sollte eine SMS oder ein PagerDuty-Alarm ausgelöst werden. Dies verhindert Alarmmüdigkeit und stellt sicher, dass kritische Jobs sofort eskaliert werden.
