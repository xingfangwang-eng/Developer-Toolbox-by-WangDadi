# cron-guard - CRO_016

## Überblick

Cron Guard ist ein spezialisiertes Monitoring-Framework für unternehmenskritische Cron-Jobs und geplante Tasks. Es eliminiert tote Winkel in der Job-Überwachung, indem es Laufzeitabweichungen, fehlgeschlagene Ausführungen und Ressourcenengpässe in Echtzeit erkennt. Optimiert für DevOps-Teams, die auf Linux-Cron, systemd-Timer und Kubernetes-CronJobs setzen.

## Harte Kernfunktionen

### 1. **Anomalie-basierte Laufzeitüberwachung mit adaptiven Schwellwerten**
Cron Guard analysiert historische Ausführungsmuster und definiert dynamisch Toleranzbereiche für Job-Dauern. Statt statischer Timeouts werden Abweichungen vom Median der letzten 100 Läufe (konfigurierbar) als kritisch eingestuft. Dies verhindert Fehlalarme bei saisonalen Lastspitzen und erkennt gleichzeitig selbst minimale Performance-Verschlechterungen.

### 2. **Multi-Exit-Code-Validierung mit Stack-Trace-Injection**
Jeder Job-Abschluss wird nicht nur auf Exit-Code 0 geprüft. Cron Guard injiziert automatisch einen Wrapper, der STDERR/STDOUT parst, Exit-Codes gegen eine benutzerdefinierte Mapping-Tabelle validiert und bei nicht-konformen Zuständen (z. B. Exit-Code 127 bei fehlender Abhängigkeit) einen detaillierten Stack-Trace in das zentrale Logging-System schreibt.

### 3. **Dependency-Graph-basierte Ausfallkaskaden-Erkennung**
Verknüpfen Sie Jobs über eine YAML-basierte Abhängigkeitsmatrix. Cron Guard überwacht nicht nur einzelne Jobs, sondern die gesamte Kaskade: Fällt ein vorgelagerter Job aus, werden abhängige Jobs automatisch in den „Pending“-Status versetzt und eine Alarmierung mit dem genauen Root-Cause-Job ausgelöst. Dies verhindert stille Fehlerfortpflanzungen in CI/CD-Pipelines.

## Detaillierte Anleitung zur Einrichtung

### Schritt 1: Agenten-Installation und Konfiguration
```bash
# Repository klonen und Agent installieren
git clone https://github.com/xingfangwang-eng/cron-guard.git
cd cron-guard/agent
make install

# Basis-Konfiguration anpassen
sudo nano /etc/cron-guard/agent.yml
# Wichtige Parameter:
# monitor_interval: 30s
# log_destination: syslog
# alert_channel: slack
# slack_webhook: https://hooks.slack.com/services/... 
```

### Schritt 2: Job-Registrierung mit erweiterten Metadaten
Erstellen Sie eine Job-Definitionsdatei für jeden zu überwachenden Cron-Job:
```yaml
# /etc/cron-guard/jobs/db-backup.yml
name: "db_backup_prod"
cron_expression: "0 2 * * *"
command: "/usr/local/bin/pg_dump -U admin prod_db > /backup/latest.sql"
max_runtime_seconds: 1800
exit_code_mapping:
  0: "success"
  1: "warning"
  2: "critical"
dependencies:
  - "health_check_db"
  - "mount_backup_volume"
alert_on:
  - "failure"
  - "timeout"
  - "dependency_fail"
```
Registrieren Sie den Job:
```bash
cron-guard register /etc/cron-guard/jobs/db-backup.yml
```

### Schritt 3: Dashboard-Konfiguration und Alarm-Trigger
Richten Sie das Web-Dashboard ein und definieren Sie Alarm-Regeln:
```bash
# Dashboard starten
cron-guard dashboard --port 8080

# Alarm-Regel für kritische Fehler erstellen
cron-guard alert create \
  --name "critical_job_failure" \
  --condition "exit_code == 2 OR runtime > max_runtime" \
  --action "slack:channel=#ops-alerts" \
  --action "pagerduty:severity=critical" \
  --cooldown 300
```
Das Dashboard zeigt live: Job-Status, Histogramme der Laufzeiten, Abhängigkeitsketten und einen Heatmap-Kalender für Ausfallmuster.

## Best Practices für den produktiven Einsatz

### 1. **Staging-Umgebung mit simulierten Fehlern validieren**
Bevor Sie Cron Guard in der Produktion einsetzen, richten Sie eine Staging-Umgebung ein, in der Sie gezielt Job-Fehler provozieren (z. B. durch Löschen von Abhängigkeiten oder Setzen von künstlichen Timeouts). Testen Sie, ob die Alarmierungs-Kaskaden korrekt ausgelöst werden und die Dependency-Graphen wie erwartet reagieren. Dies deckt Konfigurationslücken auf, bevor echte Ausfälle eintreten.

### 2. **Retentions- und Komprimierungsstrategie für Logs definieren**
Cron Guard erzeugt detaillierte Logs pro Job-Ausführung, die bei hoher Job-Frequenz schnell Speicher fressen. Implementieren Sie eine Rotation: Logs älter als 30 Tage automatisch löschen, aber eine wöchentliche Zusammenfassung als JSON in einem S3-Bucket archivieren. Nutzen Sie die integrierte Log-Komprimierung (gzip), um Speicherkosten zu senken. Dies verhindert, dass das Monitoring selbst zum Performance-Engpass wird.
