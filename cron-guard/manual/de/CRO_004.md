# cron-guard - CRO_004

## Übersicht

Cron Guard ist ein spezialisiertes Monitoring-Tool für **Cron-Jobs** und **geplante Aufgaben** in Linux/Unix-Umgebungen. Es erkennt Laufzeitabweichungen, Ausfälle und Konfigurationsfehler in Echtzeit und alarmiert DevOps-Teams über definierte Kanäle (Slack, E-Mail, Webhook). Optimiert für **kritische Scheduling-Workflows** mit hohen Verfügbarkeitsanforderungen.

## Detaillierter Inhalt

### Was ist Cron Guard?

Cron Guard ist ein **agentenloses** Tool, das Crontab-Einträge analysiert, Job-Logs überwacht und Ausführungsstatistiken historisiert. Es unterstützt **multi-node**-Umgebungen und bietet eine zentrale Dashboard-Ansicht für alle geplanten Aufgaben. Die Konfiguration erfolgt über YAML-Dateien, die **CI/CD-Pipelines** integrierbar sind.

### Kernfunktionen

1. **Laufzeitvarianz-Detektion**  
   Erstellt dynamische Baseline-Profile für jeden Job und alarmiert, wenn die Ausführungsdauer von der Norm abweicht (z. B. +20% über 7-Tage-Mittelwert). Verhindert **schleichende Performance-Degradation**.

2. **Exit-Code-Validierung mit Retry-Logik**  
   Interpretiert nicht nur `exit 0`, sondern auch **benutzerdefinierte Exit-Codes** (z. B. `exit 75` für temporäre Fehler). Automatischer Retry mit exponentiellem Backoff (max. 3 Versuche) und Eskalation an PagerDuty bei finalem Fehlschlag.

3. **Cron-Expressions-Prüfung auf Konflikte**  
   Analysiert alle Crontab-Einträge auf **Overlaps** und **Resource-Contention** (z. B. zwei Jobs, die gleichzeitig dieselbe Datenbank abfragen). Generiert einen Konflikt-Report mit Optimierungsvorschlägen.

### So verwenden Sie es

1. **Installation und Initialisierung**  
   ```bash
   wget https://github.com/xingfangwang-eng/cron-guard/releases/latest/download/cron-guard-linux-amd64.tar.gz
   tar -xzf cron-guard-linux-amd64.tar.gz
   sudo ./cron-guard init --config /etc/cron-guard/config.yaml
   ```
   Der `init`-Befehl erstellt einen systemd-Service und kopiert die Beispielkonfiguration nach `/etc/cron-guard/config.yaml`. Anschließend muss die Konfiguration mit den eigenen Crontab-Pfaden und Alarmkanälen befüllt werden.

2. **Überwachung eines bestimmten Jobs aktivieren**  
   ```yaml
   jobs:
     - name: database-backup
       command: "/usr/local/bin/backup.sh"
       schedule: "0 2 * * *"
       max_duration: 300s
       retry_on_exit_codes: [75, 126]
       alert_channels: ["slack-ops", "email-admin"]
   ```
   Nach dem Editieren der Konfiguration:  
   ```bash
   sudo cron-guard reload
   ```
   Der Befehl validiert die YAML-Syntax und startet den Watchdog neu. Bei Fehlern in der Konfiguration wird eine detaillierte Fehlermeldung mit Zeilennummer ausgegeben.

3. **Dashboard und Alarmierung testen**  
   ```bash
   sudo cron-guard test-alert --channel slack-ops --message "Cron Guard Test"
   ```
   Dies sendet einen Test-Webhook an den konfigurierten Slack-Channel. Im Dashboard (`http://localhost:9090`) erscheint der Job-Status als `RUNNING`, `SUCCESS` oder `FAILED`. Für historische Daten:  
   ```bash
   sudo cron-guard export --format json --since 24h > /tmp/cron_guard_report.json
   ```

### Beste Praktiken

1. **Heartbeat-Check für kritische Jobs**  
   Fügen Sie einen **synthetischen Cron-Job** ein, der nur einen HTTP-Health-Check auf Ihre Applikation ausführt. Wenn dieser Job fehlschlägt, liegt ein Netzwerk- oder Applikationsproblem vor – **bevor** Ihre Geschäftslogik-Jobs betroffen sind.

2. **Log-Rotation und Retention**  
   Konfigurieren Sie Cron Guard so, dass Logs älter als 30 Tage automatisch gelöscht werden:  
   ```yaml
   storage:
     retention_days: 30
     compression: true
   ```  
   Andernfalls belegt das Tool bei 100+ Jobs schnell mehrere GB Speicher. Verwenden Sie zudem **separate Volume-Mounts** für Logs, um eine Überlastung des Systempartitions zu vermeiden.
