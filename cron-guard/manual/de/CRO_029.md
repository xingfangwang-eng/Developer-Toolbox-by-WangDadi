# cron-guard - CRO_029

## Übersicht

Cron Guard ist ein hochperformantes Monitoring-Tool für **geplante Tasks (Cron Jobs)**, das speziell für DevOps-Ingenieure und Backend-Entwickler entwickelt wurde. Es erkennt Laufzeitabweichungen, fehlgeschlagene Ausführungen und Speicherlecks in Linux-Cron-Umgebungen in Echtzeit.

**Keywords:** cron job monitoring, scheduled tasks, job execution tracking, Linux crontab, Fehlerfrüherkennung

## Detailierte Funktionsbeschreibung

### Was ist Cron Guard?

Cron Guard ist eine **agentenbasierte Lösung**, die sich nahtlos in bestehende Cron-Infrastrukturen integriert. Es protokolliert jede Ausführung eines Cron-Jobs, überwacht Exit-Codes, Ausführungsdauer und Systemressourcen (CPU, RAM, I/O). Das Tool ist ideal für Multi-Server-Umgebungen, in denen eine zentrale Übersicht über alle geplanten Tasks erforderlich ist.

### 3 Hardcore-Funktionen

1. **Anomaliebasierte Alerting-Engine**  
   Statt nur auf Exit-Codes ≠ 0 zu reagieren, lernt Cron Guard das normale Verhalten jedes Jobs (z. B. Dauer ± 5 %, Speicherverbrauch). Bei Abweichungen (z. B. Job läuft 3× länger als üblich, obwohl Exit-Code 0) wird ein kritischer Alert ausgelöst – inklusive Stack-Trace und Log-Snippet.

2. **Cron-Job-Dependency-Graph**  
   Ermöglicht die Definition von Abhängigkeiten zwischen Jobs (z. B. Job B startet erst, wenn Job A erfolgreich beendet ist). Bei Fehlern in der Kette wird automatisch ein **Rollback-Trigger** für nachgelagerte Jobs gesendet. Visualisierung als gerichteter Graph in der Web-UI.

3. **Self-Healing via Retry-Strategie mit exponentieller Backoff**  
   Bei fehlgeschlagenen Jobs kann Cron Guard automatisch einen erneuten Versuch starten, wobei die Wartezeit exponentiell ansteigt (z. B. 1s → 2s → 4s → 8s). Der Admin kann die maximale Anzahl von Retries und die Backoff-Faktoren pro Job konfigurieren.

### 3 detaillierte Anwendungsschritte

#### Schritt 1: Installation und Agent-Registrierung  
```bash
# Auf jedem Zielserver (Linux, systemd-basiert)
curl -sSL https://cron-guard.io/install.sh | bash
cron-guard agent register --token YOUR_API_TOKEN
```
Der Agent erstellt automatisch einen systemd-Dienst (`cron-guard-agent.service`), der alle Cron-Jobs überwacht. Nach der Registrierung erscheint der Server in der Web-UI.

#### Schritt 2: Konfiguration einer benutzerdefinierten Alert-Regel  
In der Web-UI unter `Alerts → Create Rule`:
- **Trigger:** `job_name` = `cleanup_logs` AND `duration` > `30s`
- **Action:** `Webhook POST` an Slack-Channel `#ops-alerts` mit Payload: `{"text": "Job cleanup_logs läuft länger als 30s – bitte prüfen!"}`
- **Cooldown:** 5 Minuten (verhindert Alert-Flooding)

#### Schritt 3: Einrichtung eines Dependency-Graphen  
In der UI unter `Jobs → Dependencies`:
- Job `db_backup` → Job `sync_to_s3` → Job `cleanup_temp`
- Falls `db_backup` fehlschlägt, wird `sync_to_s3` **nicht gestartet** und ein `ROLLBACK`-Event an `cleanup_temp` gesendet.  
- Log-Eintrag: `[CRON-GUARD] Dependency chain broken: db_backup → sync_to_s3 (skipped) | cleanup_temp (rollback executed)`

## 2 Best Practices aus der Industrie

1. **Immer ein Timeout pro Job definieren**  
   Setzen Sie für jeden Cron-Job ein maximales Timeout (z. B. 5 Minuten). Cron Guard kann Jobs, die länger laufen, automatisch **killen** und einen Alert auslösen. So verhindern Sie Zombie-Prozesse und unnötige Last auf dem Server.

2. **Log-Level anpassen für Produktion**  
   Verwenden Sie in kritischen Jobs (z. B. Datenbank-Backups) den Log-Level `debug`, um im Fehlerfall sofort die Ursache zu erkennen. In der Produktion setzen Sie das globale Log-Level auf `info`, um die Performance zu optimieren. Cron Guard erlaubt die dynamische Änderung des Log-Levels pro Job ohne Neustart.
