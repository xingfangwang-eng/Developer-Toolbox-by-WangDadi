# cron-guard - CRO_003

## Übersicht

Cron Guard ist ein hochperformantes Überwachungs- und Orchestrierungs-Framework für unternehmenskritische Cron-Jobs und zeitgesteuerte Workflows. Entwickelt für DevOps-Teams, die eine lückenlose Ausführung, granulare Fehleranalyse und automatisierte Eskalation bei Job-Fehlern benötigen. Unterstützt native Integrationen mit Kubernetes CronJobs, systemd-Timer und legacy-cron-Daemons.

## Detaillierter Inhalt

### Was ist Cron Guard?

Cron Guard ist ein dezentrales Monitoring-System, das Scheduled Tasks in Echtzeit überwacht, Ausführungsmetriken mit einer Auflösung von **100 Millisekunden** erfasst und fehlgeschlagene Jobs automatisch mit konfigurierbaren Backoff-Strategien (linear, exponentiell, Fibonacci) neu startet. Es persistiert alle Ausführungslogs in einer **PostgreSQL-Datenbank** mit automatischem Partitioning nach Zeitstempel und Job-ID. Die Architektur basiert auf einem ereignisgesteuerten Pub/Sub-Modell (NATS oder Redis Streams), das eine horizontale Skalierung auf tausende parallele Job-Instanzen ermöglicht.

### Kernfunktionen

- **Präzise Ausführungszeit-Messung und SLA-Überwachung**: Jeder Job wird mit einem `execution_timeout`-Parameter (in Millisekunden) versehen. Überschreitet die tatsächliche Laufzeit diesen Schwellwert, wird ein `SLA_BREACH`-Event ausgelöst. Die Metrik `p99_latency` wird über ein gleitendes Fenster von 5 Minuten berechnet und in Prometheus exportiert. Konfigurierbare Aktionen: E-Mail-Benachrichtigung, Webhook-Trigger (z.B. an PagerDuty) oder automatisches Skalieren von Worker-Pods in Kubernetes.

- **Deterministische Abhängigkeitsauflösung und DAG-Orchestrierung**: Jobs können in einem Directed Acyclic Graph (DAG) definiert werden. Parameter `depends_on` akzeptiert eine Liste von Job-IDs. Der Scheduler berechnet den kritischen Pfad und startet Child-Jobs erst, wenn alle Parent-Jobs erfolgreich abgeschlossen sind. Bei Fehlern in einem Parent-Job werden alle abhängigen Child-Jobs auf `SKIPPED` gesetzt und eine `DEPENDENCY_FAILURE`-Nachricht an den konfigurierten `error_channel` (z.B. Slack-Webhook) gesendet.

- **Multi-Environment-Konfiguration und rollback-fähige Job-Definitionen**: Job-Konfigurationen werden als YAML-Manifeste versioniert (z.B. in Git). Der Parameter `config_version` (SemVer) ermöglicht ein sofortiges Rollback auf eine vorherige Konfiguration via API-Aufruf: `POST /api/v1/jobs/{job_id}/rollback?version=1.2.3`. Alle Änderungen werden in einer `config_audit_log`-Tabelle in PostgreSQL mit einem vollständigen Diff (JSONB) protokolliert.

### So verwenden Sie es

1. **Job-Registrierung und erstmalige Ausführung**: Definieren Sie einen Job in einer YAML-Datei (z.B. `daily_backup.yml`) mit den Parametern `name`, `schedule` (Cron-Expression), `command`, `timeout_ms` und `retry_policy`. Übermitteln Sie die Konfiguration an den Cron Guard Agenten via `cron-guard apply -f daily_backup.yml`. Der Agent validiert die Cron-Expression, registriert den Job in der PostgreSQL-Datenbank und startet den Scheduler. Prüfen Sie den Status mit `cron-guard status --job-id daily_backup`.

2. **Integration mit Kubernetes CronJobs**: Fügen Sie das Label `cron-guard/enabled: "true"` zu Ihrem CronJob-Manifest hinzu. Der Cron Guard Operator (separates Helm-Chart) injected automatisch einen Sidecar-Container in den Pod, der die Ausführung überwacht. Der Sidecar sendet Heartbeats alle 10 Sekunden an den zentralen Collector. Bei Ausbleiben von 3 Heartbeats wird ein `POD_DEAD`-Alarm ausgelöst. Konfigurieren Sie den `heartbeat_interval` im `cron-guard-operator-config` ConfigMap.

3. **Erstellen eines benutzerdefinierten Eskalations-Workflows**: Definieren Sie eine Eskalationskette in einer JSON-Datei: `{"job_id": "critical_etl", "escalation_steps": [{"delay_ms": 300000, "action": "slack", "channel": "#alerts"}, {"delay_ms": 600000, "action": "pagerduty", "severity": "critical"}]}`. Senden Sie diese via `cron-guard escalate -f escalation.json`. Der Cron Guard Event Processor wertet die Schritte aus und führt sie zeitgesteuert aus. Bei erfolgreicher Job-Wiederherstellung wird die Eskalation automatisch abgebrochen.

### Beste Praktiken

- **Implementieren Sie eine verteilte Job-Lock-Strategie mit PostgreSQL Advisory Locks**: Verwenden Sie `SELECT pg_advisory_xact_lock(job_id_hash)` in Ihren Job-Skripten, um Race Conditions bei parallelen Ausführungen auf mehreren Nodes zu vermeiden. Kombinieren Sie dies mit einem `lock_timeout` von 5 Sekunden, um Deadlocks zu verhindern. Protokollieren Sie alle Lock-Erwerbe und -Freigaben in einer separaten `job_locks`-Tabelle für das Debugging.

- **Nutzen Sie die integrierte Job-Dependency-Grafik für Canary-Deployments**: Definieren Sie einen DAG, bei dem ein Canary-Job (`canary_deploy`) vor dem Haupt-Job (`production_deploy`) läuft. Der Canary-Job führt einen Smoke-Test aus (z.B. HTTP-Healthcheck auf eine neue Version). Nur wenn der Canary-Job innerhalb von 30 Sekunden mit Exit-Code 0 abschließt, wird der Produktions-Job freigegeben. Verwenden Sie den Parameter `dag_retry_delay` (z.B. 60 Sekunden), um bei Fehlern eine automatische Wiederholung des Canary-Jobs zu ermöglichen.
