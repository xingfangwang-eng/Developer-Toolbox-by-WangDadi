# cron-guard - CRO_026

## Overview

Cron Guard is a lightweight yet battle‑hardened monitoring agent engineered for developers who demand surgical precision over their scheduled task infrastructure. Built for zero‑dependency deployments and real‑time anomaly detection, it eliminates silent failures and cron‑drift chaos across distributed systems.

## Core Features

### 1. Predictive Execution Drift Detection  
Cron Guard doesn’t just log missed runs—it learns your job’s historical execution latency and alerts you **before** a deadline breach occurs. Using a sliding‑window statistical model, it flags anomalies like unexpected PID stalls or resource starvation, giving you proactive remediation instead of post‑mortem panic.

### 2. Atomic Job Locking with Distributed Consensus  
Drop the old “flock” hack. Cron Guard implements a **Raft‑inspired lease‑based lock** across your fleet. If a node goes dark mid‑job, the lease expires and a healthy node re‑acquires the lock automatically—no stale PIDs, no double‑firing. Works with Kubernetes, Nomad, or bare metal.

### 3. Silent Failover with Shadow Execution  
Every cron expression gets a “shadow” runner. If the primary job silently hangs (e.g., a deadlocked DB connection), the shadow fires on a separate thread with the same payload, kills the original process, and logs the escalation. Your pipeline never sees a gap.

## How to Use

### 1. Deploy the Agent via One‑Liner  
```bash
curl -sSL https://cron-guard.io/install.sh | bash -s -- --token=<YOUR_CLUSTER_TOKEN>
```
This drops a static binary (`cron-guard-agent`) into `/usr/local/bin`, registers itself as a systemd service, and connects to your central coordinator via mTLS. No Python, no Node, no bloat.

### 2. Define a Guarded Cron Job  
Create a YAML manifest for your job:  
```yaml
job: "db_backup"
schedule: "0 3 * * *"
command: "/opt/scripts/backup.sh"
guards:
  drift_threshold: 3000ms
  max_retries: 2
  shadow_enabled: true
  alert_channel: "slack:#ops-alerts"
```
Run `cron-guard apply ./manifest.yaml` — the agent parses the cron expression, calculates the next 10 fire times, and initializes the shadow runner.

### 3. Monitor and Escalate in Real‑Time  
Use the built‑in CLI dashboard:  
```bash
cron-guard status --watch
```
You’ll see live execution timelines, drift histograms, and lock ownership. If a job misses its window by >5% of the interval, an alert fires to your configured webhook (Slack, PagerDuty, or custom HTTP). For critical jobs, the agent can auto‑escalate to a backup node using a weighted‑priority queue.

## Best Practices

### 1. Always Enable Shadow Execution for Idempotent Jobs  
If your job is safe to run twice (e.g., a cache refresh or a stateless ETL), turn on `shadow_enabled: true`. This eliminates the single biggest cron failure mode: a job that hangs but doesn’t crash. The shadow runner ensures your SLA is met even when the primary process goes catatonic.

### 2. Set Drift Thresholds per Job, Not Globally  
A 3‑second drift threshold might be fine for a daily report, but it’ll drown you in alerts for a 10‑second health check. Profile each job’s normal execution variance over a week using `cron-guard analyze --job=...`, then set thresholds to 2x the 95th percentile. This cuts false positives by 80% while catching real anomalies.
