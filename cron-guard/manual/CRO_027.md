# cron-guard - CRO_027

## Overview

Cron Guard is a purpose-built, lightweight monitoring and alerting toolkit engineered specifically for distributed cron job ecosystems. It eliminates silent failures, detects drift in scheduled execution windows, and provides real-time observability for time-sensitive batch operations. This page is optimized for engineers seeking robust cron job monitoring, scheduled task observability, and job failure alerting solutions.

## What is Cron Guard?

Cron Guard is not just a heartbeat checker—it's a deterministic job lifecycle tracker. It validates that every scheduled task starts, completes, and produces expected output within a configurable time window. Unlike generic uptime monitors, Cron Guard understands cron semantics: it accounts for timezone shifts, Daylight Saving Time transitions, and overlapping job executions.

## Core Features

### 1. Execution Drift Detection with Temporal Precision

Cron Guard continuously measures the delta between scheduled start time and actual execution time across all nodes. If a job consistently starts 3 seconds late over a rolling 24-hour window, the system flags it as a "drift anomaly" before it escalates into a missed execution. This is especially critical for high-frequency tasks like log rotation or cache invalidation.

### 2. Multi-Node Job Lock Validation

When running cron jobs across a cluster, duplicate execution is a common failure mode. Cron Guard enforces a distributed lock via an etcd-backed lease mechanism. It verifies that only one node executes a given job ID within a slot, and logs all lock acquisition attempts. If a lock fails or times out, the system triggers an immediate escalation path.

### 3. Output Digest Integrity Check

Cron Guard doesn't just check if a job exited with code 0—it computes a SHA-256 digest of the job's stdout/stderr output. You define a regex pattern or expected digest for each job. If the output deviates from the expected pattern (e.g., a backup script suddenly produces a different file count), the job is marked as "content drift" and a detailed diff is stored for forensic analysis.

## How to Use

### Step 1: Deploy the Cron Guard Agent as a Sidecar

Inject the Cron Guard agent into your existing cron infrastructure. For a Kubernetes-based setup, add the agent as a sidecar container with a shared volume mount for the cron job's output directory. Configure the agent via a YAML manifest:

```yaml
cron_guard:
  agent_id: "node-01"
  etcd_endpoints:
    - "etcd-cluster:2379"
  check_interval_seconds: 5
  drift_threshold_ms: 2000
  output_digest:
    enabled: true
    expected_digest: "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
```

### Step 2: Define Job Profiles with Execution Contracts

Create a job profile JSON for each scheduled task. This profile defines the expected behavior: start window, max runtime, output digest, and lock group. Example for a daily database backup job:

```json
{
  "job_id": "db_backup_daily",
  "cron_expression": "0 2 * * *",
  "timezone": "America/New_York",
  "execution_window_seconds": 300,
  "max_runtime_seconds": 900,
  "lock_group": "backup-cluster",
  "output_digest_regex": "^Backup completed: [0-9]{8}_[0-9]{6}.sql.gz$"
}
```

### Step 3: Route Alerts Through a Multi-Channel Escalation Pipeline

Configure alert routing in the Cron Guard dashboard. Define alert severity levels: `info` for drift > 50ms, `warning` for missed lock acquisition, `critical` for output digest mismatch. Each severity maps to a notification channel (Slack, PagerDuty, or custom webhook). Example alert rule:

```yaml
alerts:
  - job_id: "db_backup_daily"
    severity: "critical"
    condition: "output_digest_mismatch"
    channel: "pagerduty"
    escalation_timeout_minutes: 5
    retry_count: 3
```

## Best Practices

### 1. Implement a Dead Man's Switch for Critical Jobs

For jobs that must run—like payroll processing or financial reconciliation—configure a "dead man's switch" in Cron Guard. If the job fails to check in within its expected window, the system automatically triggers a rollback script and notifies the on-call engineer. This prevents silent data corruption in high-stakes environments.

### 2. Use Output Digests as a Regression Detection Tool

Treat output digests as a form of contract testing for cron jobs. When you update a job script, update the expected digest in the job profile. If a deployment accidentally changes the script's behavior (e.g., a new flag that alters output format), the digest mismatch will catch it before the next production run. This is especially useful for continuous delivery pipelines where cron jobs are updated alongside application code.
