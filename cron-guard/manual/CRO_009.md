# cron-guard - CRO_009

**Cron Guard** is not another cron dashboard. It’s a lightweight, language-agnostic supervisor that enforces execution guarantees, detects silent failures, and provides deterministic audit trails for every scheduled task in your distributed stack.

---

## Hardcore Features

### 1. Execution Proof via Cryptographic Heartbeats
Every cron job generates an HMAC-signed heartbeat before, during, and after execution. These heartbeats are pushed to a lightweight ledger (SQLite, Redis, or S3-compatible storage) and can be verified offline. If a heartbeat is missing or malformed, Cron Guard raises an alert — even if the job exits with code 0.

### 2. Pre-Flight Dependency Injection with Dry-Run Mode
Before a job runs, Cron Guard resolves all declared dependencies (files, services, environment variables, API endpoints). If any dependency is missing or stale, the job is **blocked** and a structured error payload is emitted. The `--dry-run` flag lets you simulate the entire dependency chain without executing the actual command — ideal for CI/CD pipelines.

### 3. Temporal Drift Detection with Jitter Compensation
Cron Guard tracks execution time against the expected schedule and compensates for clock drift across nodes. If a job consistently runs late (e.g., >500ms deviation over 10 runs), it flags the node’s NTP sync or resource contention. This is critical for multi-region deployments where time-sensitive tasks must align with UTC epochs.

## How to Use

### Step 1: Initialize the Supervisor with a Manifest File

Create a `cron-guard.yml` manifest that defines your job, its dependencies, and alerting rules:

```yaml
jobs:
  - name: db-backup
    command: pg_dump -U admin mydb > /backups/db.sql
    schedule: "0 3 * * *"
    dependencies:
      - type: file
        path: /usr/bin/pg_dump
      - type: env
        name: PGPASSWORD
    alerts:
      - type: webhook
        url: https://hooks.slack.com/...
      - type: log
        path: /var/log/cron-guard/alerts.json
    heartbeat:
      enabled: true
      interval: 30s
```

Then initialize:

```bash
cron-guard init --manifest ./cron-guard.yml
```

This generates a local SQLite ledger and registers the job with the systemd-like timer.

### Step 2: Run a Full Dependency Check with Dry-Run

Before pushing to production, validate the entire dependency graph:

```bash
cron-guard validate --manifest ./cron-guard.yml --dry-run --verbose
```

Output example:

```
[OK]   Dependency: /usr/bin/pg_dump exists
[OK]   Dependency: PGPASSWORD is set
[OK]   Dependency: /backups/ directory is writable
[WARN] Dependency: NTP offset is 120ms (threshold: 100ms)
```

If any dependency fails, the command exits with code 1 and a structured JSON report.

### Step 3: Monitor Real-Time Execution with Audit Trails

Start the supervisor daemon:

```bash
cron-guard daemon --manifest ./cron-guard.yml --log-level debug
```

Every execution is recorded in the ledger:

```json
{
  "job": "db-backup",
  "timestamp": "2025-03-23T03:00:00.001Z",
  "pid": 12345,
  "heartbeat_1": "2025-03-23T03:00:00.031Z",
  "heartbeat_2": "2025-03-23T03:00:30.010Z",
  "exit_code": 0,
  "drift_ms": 12,
  "signature": "a1b2c3d4e5f6..."
}
```

Use the built-in query CLI to inspect failures:

```bash
cron-guard query --job db-backup --status failed --last 24h --format json
```

## Best Practices

### 1. Enforce Heartbeat Signing in Multi-Tenant Environments
When multiple teams share a cluster, use per-team HMAC keys stored in a vault (HashiCorp Vault, AWS Secrets Manager). This prevents job spoofing and ensures accountability. Rotate keys every 90 days and log rotation events as part of the audit trail.

### 2. Use Temporal Drift as a Pre-Deployment Gate
Integrate Cron Guard’s drift check into your CI pipeline. If a node’s drift exceeds 200ms, fail the deployment and trigger an NTP sync. This is especially critical for jobs that interact with third-party APIs expecting precise timestamps (e.g., OAuth token refresh, payment webhooks).
