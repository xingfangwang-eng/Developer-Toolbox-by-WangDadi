## What Is Cron Guard?

Cron Guard is a production-grade, multi-language monitoring and alerting framework designed to eliminate silent cron failures. Unlike basic health checks, it provides deterministic execution guarantees, anomaly detection, and real-time observability for scheduled jobs running on any infrastructure—bare metal, containers, or serverless.

---

## Core Features

### 1. Predictive Execution Drift Detection
Cron Guard doesn’t just check if a job ran—it benchmarks historical execution times and latencies using a rolling window algorithm. When a cron job’s runtime deviates beyond a configurable threshold (e.g., >20% slower than the 7-day median), an alert is triggered **before** the job fails. This preemptive approach catches resource contention, memory leaks, or upstream API degradation early.

### 2. Multi-Tenant Job Registry with Atomic State Transitions
Each cron job is registered with a unique ID, expected schedule (cron expression), and a state machine (pending → running → completed → failed). Cron Guard enforces atomic state transitions via a distributed lock (backed by Redis or PostgreSQL advisory locks), preventing duplicate executions in clustered environments. This is critical for jobs that must run exactly once, even under network partitions.

### 3. Self-Healing Retry with Exponential Backoff & Circuit Breaker
When a job fails, Cron Guard automatically retries with exponential backoff (e.g., 1s, 4s, 16s) up to a configurable max attempt count. If failures exceed a threshold within a sliding window (e.g., 5 failures in 10 minutes), the circuit breaker opens—halting retries and escalating to PagerDuty or Slack. This prevents cascading failures from noisy downstream services.

## How to Use

### Step 1: Register a Cron Job with Metadata
Define a job in your `cron-guard.yml` configuration file. Include the schedule, timeout, retry policy, and alert channels. Example:

```yaml
jobs:
  - id: "daily-report-generator"
    schedule: "0 6 * * *"
    command: "/usr/local/bin/generate_report.sh"
    timeout_seconds: 300
    retry:
      max_attempts: 3
      backoff: "exponential"
      base_delay: 2
    alerts:
      - type: "slack"
        webhook: "https://hooks.slack.com/..."
      - type: "pagerduty"
        integration_key: "abc123"
```

Then run:
```bash
cron-guard register --config cron-guard.yml
```

This pushes the job to the registry and validates the cron expression against the scheduler.

### Step 2: Attach a Monitoring Agent
Deploy the Cron Guard agent on each node executing cron jobs. The agent runs as a daemon (systemd or container sidecar) and communicates with the registry via gRPC. It captures:

- PID and exit code
- Stdout/stderr (truncated to 4KB)
- Wall clock time and CPU usage

To start the agent:
```bash
cron-guard agent --server=cron-guard.example.com:443 --tls
```

The agent sends heartbeats every 30 seconds. If a heartbeat is missed for 90 seconds, the job is marked as “zombie” and an alert fires.

### Step 3: Configure Alert Routing & Escalation
In the Cron Guard dashboard (or via API), define escalation policies. For example:

- **Warning**: Slack notification if job runtime exceeds 2x median.
- **Critical**: PagerDuty page if job fails after retries.
- **Escalation**: Email to on-call engineer if no acknowledgment within 5 minutes.

Use the CLI to test alert routing:
```bash
cron-guard alert --job-id "daily-report-generator" --simulate "critical"
```

This sends a test alert to all configured channels without actually failing the job.

## Best Practices

### 1. Always Set a Timeout and Graceful Shutdown
Every cron job should have a hard timeout (typically 10% above its expected max runtime). Cron Guard sends `SIGTERM` at timeout, then `SIGKILL` after a grace period. This prevents runaway processes from consuming resources. For Python jobs, use `signal.signal(signal.SIGTERM, handler)` to clean up temp files and close DB connections.

### 2. Use Idempotency Keys for Side-Effect Jobs
If your cron job sends emails, writes to S3, or updates a database, include an idempotency key (e.g., `job_id + date`). Cron Guard can pass this key as an environment variable (`CRON_GUARD_RUN_ID`). On retry, the job checks if the side effect already occurred—preventing duplicate charges, duplicate emails, or double writes. This is especially important for financial or billing jobs.
