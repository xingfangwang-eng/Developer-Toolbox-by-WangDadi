# cron-guard - CRO_024

## Overview

Cron Guard is a production-grade cron job monitoring and alerting tool designed for DevOps engineers, SREs, and backend developers who need bulletproof visibility into scheduled task execution. Unlike generic uptime monitors, Cron Guard focuses exclusively on cron job lifecycle—tracking execution start, duration, exit codes, and missed runs with millisecond precision. This tool eliminates silent failures in batch processing pipelines, ETL workflows, and system maintenance scripts.

## Core Features

### 1. Execution Drift Detection with Adaptive Baselines
Cron Guard doesn't just check if a job ran—it analyzes how it ran. The tool builds a statistical baseline of execution duration, exit code patterns, and resource consumption over time. When a job finishes 3 standard deviations faster or slower than its historical average, or when exit code patterns shift (e.g., from 0 to 127), it fires a preemptive alert before the failure cascades downstream. This catches subtle performance degradation or silent corruption before they become outages.

### 2. Multi-Node Heartbeat Verification with Quorum-Based Alerting
For distributed cron deployments across Kubernetes pods, EC2 instances, or bare-metal servers, Cron Guard implements a quorum-based heartbeat system. Each node sends a signed heartbeat to the monitoring endpoint. If fewer than N/2+1 nodes report within the expected window, the system triggers a "split-brain" alert instead of a false positive from a single missing heartbeat. This eliminates noise from transient network partitions or rolling deployments while maintaining high sensitivity to actual failures.

### 3. Cron Expression Fuzzing and Conflict Analysis
Cron Guard includes a built-in validator that tests cron expressions against real-world edge cases: leap years, DST transitions, month boundaries, and even Gregorian calendar anomalies. It also detects overlapping schedules (e.g., two jobs that both fire at 03:00 AM and compete for the same lock file or database connection), flagging potential race conditions before they cause data corruption or deadlocks.

## How to Use

### Step 1: Deploy the Monitoring Agent via Docker Compose
Start by deploying the Cron Guard agent as a sidecar container alongside your cron jobs. Create a `docker-compose.yml` file:

```yaml
version: '3.8'
services:
  cron-guard:
    image: cronguard/agent:latest
    environment:
      - CG_ENDPOINT=https://your-instance.cronguard.io/ingest
      - CG_API_KEY=${CG_API_KEY}
      - CG_HEARTBEAT_INTERVAL=30
      - CG_QUORUM_THRESHOLD=3
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock:ro
    network_mode: host
```

Run `docker compose up -d` to start the agent. It will automatically discover running cron jobs by parsing `/etc/crontab`, Docker labels, and Kubernetes CronJob manifests.

### Step 2: Configure Custom Alert Channels with Webhook Templates
Define alert routing in a `cron-guard.yaml` configuration file:

```yaml
alerts:
  channels:
    - id: pagerduty-prod
      type: pagerduty
      routing_key: ${PD_ROUTING_KEY}
      severity: critical
      conditions:
        - metric: execution_drift
          operator: gt
          threshold: 2.5
          unit: stddev
    - id: slack-devops
      type: slack
      webhook: ${SLACK_WEBHOOK}
      severity: warning
      conditions:
        - metric: missed_job
          operator: eq
          threshold: 1
```

Apply the configuration with `cron-guard apply -f cron-guard.yaml`. The tool validates the webhook URLs and tests connectivity before persisting the config.

### Step 3: Run a Forensic Audit with the CLI
When a job fails, use the CLI to drill into the root cause:

```bash
cron-guard audit --job-id "data-pipeline-123" --time-range "2024-11-01T00:00:00Z/2024-11-07T23:59:59Z" --format json
```

This returns a structured report including:
- Execution timeline with sub-millisecond start/end timestamps
- Exit code histogram across all nodes
- Resource utilization spikes (CPU, memory, I/O) correlated with failure events
- Adjacent job interference analysis (e.g., "Job 'cleanup-db' started 2 seconds before failure, holding a read lock on table 'transactions'")

## Industry Best Practices

### 1. Implement a "Graceful Degradation" Fallback for Missed Jobs
Never rely on a single cron trigger for critical tasks like database backups or log rotation. Configure Cron Guard to trigger a fallback execution from a secondary node if the primary fails to report within 2 minutes. This is especially important for ETL pipelines where a missed window means stale data for the next 24 hours. Use the `fallback_execution` flag in your job definition:

```yaml
jobs:
  - name: nightly-backup
    cron: "0 2 * * *"
    fallback:
      enabled: true
      delay: 120
      node_selector: "role=secondary"
```

### 2. Audit Cron Expressions Quarterly for DST and Calendar Drift
Daylight Saving Time changes and leap seconds cause more cron failures than most teams realize. Schedule a quarterly audit using Cron Guard's expression fuzzer: `cron-guard fuzz --expression "0 3 * * *" --years 5`. This generates a report showing every execution time shift, including ambiguous times (e.g., 2:30 AM that occurs twice during fall-back). Adjust your schedules to use UTC for all production jobs to eliminate timezone-related drift entirely.
