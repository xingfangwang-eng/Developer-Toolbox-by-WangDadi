# cron-guard - CRO_013

## Overview

Cron Guard is a battle-tested, zero-dependency toolkit designed for DevOps engineers and backend developers who need bulletproof cron job execution, real-time failure detection, and intelligent retry logic—without the overhead of a full orchestration platform.

## Core Features

### 1. **Predictive Execution Guardrails**
Cron Guard pre-validates your cron expressions against actual system time, daylight saving shifts, and leap seconds. It flags ambiguous schedules (e.g., jobs that would run twice or not at all during DST transitions) before they hit production.

### 2. **Self-Healing Job Runner with Exponential Backoff**
When a job fails, Cron Guard doesn't just log it—it automatically retries with configurable exponential backoff, capped at a max interval. If the failure persists beyond a threshold, it escalates via webhook, Slack, or PagerDuty, and optionally triggers a fallback script.

### 3. **Execution Drift Detector**
Cron Guard tracks wall-clock vs. actual execution time. If a job consistently starts late (e.g., due to resource contention or scheduler lag), it emits a drift alert and suggests an adjusted cron expression to compensate.

## How to Use

### 1. **Initialize and Validate Your Cron Schedule**
```bash
cron-guard init --config ./cron-guard.yml
cron-guard validate --file ./crontab
```
This parses your crontab, resolves all expressions against the current timezone, and outputs a human-readable execution timeline. It also warns about overlapping jobs or expressions that will never trigger.

### 2. **Configure a Job with Retry and Escalation**
```yaml
jobs:
  - name: db-backup
    schedule: "0 3 * * *"
    command: "./backup.sh"
    retry:
      max_attempts: 3
      backoff: exponential
      initial_delay: 30s
    escalate:
      after: 2 failures
      webhook: https://hooks.slack.com/services/xxx
```
Run `cron-guard run --job db-backup` to execute it under the guard's supervision. The process is forked, and stdout/stderr are captured and rotated automatically.

### 3. **Monitor Execution Drift in Real-Time**
```bash
cron-guard monitor --watch
```
This opens an interactive dashboard in your terminal. It shows each job's scheduled vs. actual execution time, cumulative drift, and last exit code. You can filter by status: `healthy`, `drifted`, `failed`.

## Best Practices

### **Always Pre-Validate Before Deploying to Production**
Use `cron-guard validate` in your CI/CD pipeline. This catches silent failures like jobs that would run on a non-existent date (e.g., February 30) or expressions that are syntactically valid but logically dead. It's your first line of defense.

### **Set a Drift Threshold and Alert**
For time-sensitive jobs (e.g., billing runs, data exports), set a drift threshold of 30 seconds in your config. If a job consistently drifts beyond that, Cron Guard will alert you before it causes a downstream failure. This is especially critical in multi-tenant or low-resource environments.
