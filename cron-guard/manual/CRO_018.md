# cron-guard - CRO_018

## Overview

Cron Guard is a high-precision distributed cron job scheduler and execution guard designed for mission-critical production environments. It eliminates silent failures, drift in execution timing, and cascading dependency collapses across microservices architectures. Keywords: cron job, scheduled tasks, job monitoring

## Detailed Content

### What is Cron Guard?

Cron Guard provides deterministic execution guarantees for periodic tasks through a combination of leader-election-based orchestration, real-time execution trace logging, and a self-healing retry engine with exponential backoff. Unlike traditional crontab wrappers, it decouples job definitions from runtime state, enabling cross-host synchronization and audit-grade execution records.

### Core Features

- **Sub‑millisecond Execution Drift Detection & Correction**  
  Each job is assigned a precise UTC timestamp at scheduling time. Cron Guard continuously monitors actual start time against the scheduled timestamp with ±0.5 ms precision via a custom high‑resolution timer (using `CLOCK_MONOTONIC_RAW` on Linux). If drift exceeds 1 ms, the scheduler automatically issues a phase‑adjust command to the next run cycle, preventing accumulation of timing errors.

- **Multi‑Phase Dependency Graph with Circuit Breaker Integration**  
  Jobs are defined as DAG nodes with explicit `depends_on`, `timeout` (in milliseconds), and `max_retries` parameters. Cron Guard evaluates upstream completion status before triggering downstream tasks. If an upstream job fails consecutively more than `circuit_breaker_threshold` (default: 3), the entire subgraph is paused and a `CircuitBreakerOpen` event is emitted to PostgreSQL via a dedicated `cron_guard_events` table, allowing downstream systems to react before cascading failures occur.

- **Execution Trace Logging with PostgreSQL & Prometheus Export**  
  Every job run produces a structured log entry containing `job_id`, `scheduled_ts`, `actual_start_ts`, `duration_ms`, `exit_code`, and `stdout_hash`. These are written synchronously to a PostgreSQL `cron_guard_executions` table with an index on `(job_id, scheduled_ts)`. Simultaneously, a Prometheus counter metric `cron_guard_job_duration_seconds` is updated with labels for `job_name`, `status`, and `host`. This enables real‑time dashboards and historical forensics.

### How to Use

1. **Define a Job with Dependency and Retry Policy**  
   Create a YAML configuration file `guard_jobs.yaml`:
   ```yaml
   jobs:
     - name: "data_pipeline_export"
       command: "/usr/local/bin/export_data.sh --output /data/export"
       schedule: "0 */2 * * *"
       timeout_ms: 300000
       max_retries: 2
       retry_delay_ms: 5000
       depends_on:
         - "db_connection_check"
       circuit_breaker_threshold: 3
   ```
   Validate the file using `cron-guard validate guard_jobs.yaml`. The validator checks for cyclic dependencies, invalid cron expressions, and timeout values exceeding the global `max_timeout_ms` (default: 600000).

2. **Bootstrap the Guard Agent and Register Jobs**  
   Start the agent with a PostgreSQL connection string:
   ```bash
   cron-guard agent start --postgres-uri "postgres://user:pass@host:5432/cron_guard?sslmode=require" --prometheus-listen ":9090"
   ```
   Then register all jobs from the configuration:
   ```bash
   cron-guard jobs sync --file guard_jobs.yaml
   ```
   This inserts or updates job definitions in the `cron_guard_jobs` table. The agent will immediately begin scheduling based on the next occurrence of each cron expression.

3. **Monitor Execution and Handle Failures**  
   Query the execution history for a specific job:
   ```sql
   SELECT scheduled_ts, actual_start_ts, duration_ms, exit_code
   FROM cron_guard_executions
   WHERE job_id = 'data_pipeline_export'
   ORDER BY scheduled_ts DESC
   LIMIT 10;
   ```
   To manually trigger a retry for a failed job run, use:
   ```bash
   cron-guard jobs retry --job-id data_pipeline_export --run-id <run_id>
   ```
   This forces an immediate execution regardless of the schedule, respecting the existing retry policy.

### Best Practices

- **Always Set `timeout_ms` and `max_retries` Explicitly**  
  Without explicit timeouts, a stuck job can block downstream dependencies indefinitely. Set `timeout_ms` to 1.5× the expected maximum execution time. Combine with `max_retries: 2` and `retry_delay_ms: 10000` to handle transient failures (e.g., network blips) without exhausting resources.

- **Use a Dedicated PostgreSQL Instance for Event Storage**  
  Do not share the `cron_guard_events` and `cron_guard_executions` tables with application databases. Isolate them on a separate PostgreSQL instance or at least a separate schema with its own connection pool. This prevents cron‑related writes from competing with business transactions and ensures the event store remains available during application‑level outages.
