# cron-guard - CRO_014

## 概要

Cron Guard は、cronジョブの実行漏れや異常終了をリアルタイムに検知し、システム全体のスケジュールタスクを統合監視するためのプロフェッショナルツールです。特に、分散環境やマイクロサービスアーキテクチャにおけるジョブチェーンの依存関係管理に強みを持ち、障害発生時の自動リカバリとアラート通知を提供します。本ツールは、SRE（Site Reliability Engineering）チームやDevOpsエンジニアが抱える「cronジョブのサイレント障害」という根本的な課題を解決します。

## 詳細な内容

### とは Cron Guard？

Cron Guard は、従来のcronジョブ監視ツールとは一線を画す、**ジョブ実行の可観測性（Observability）を最大限に高めるプラットフォーム**です。単なる死活監視ではなく、各ジョブの実行時間、終了コード、標準出力/エラー出力、リソース消費量（CPU、メモリ、I/O）を取得し、それらを時系列データとして保存。異常検知アルゴリズムにより、閾値ベースの単純なアラートではなく、過去の実行パターンから逸脱する「異常な挙動」を自動的に特定します。さらに、ジョブ間の依存関係をグラフ構造で管理し、あるジョブが失敗した場合、その影響範囲を即座に可視化します。これにより、障害の根本原因分析（RCA）にかかる時間を大幅に短縮します。

### コア機能

1. **依存関係グラフによるチェーン障害の自動検知と影響範囲可視化**  
   Cron Guard は、各ジョブの実行順序と依存関係をDAG（有向非巡回グラフ）として定義します。あるジョブがタイムアウトまたは異常終了した場合、そのジョブに依存するすべての後続ジョブの実行を自動的にブロックし、影響範囲をグラフ上にハイライト表示します。さらに、ブロックされたジョブが再実行可能になる条件（例：親ジョブのリトライ成功）を設定でき、手動介入なしに復旧プロセスを開始できます。

2. **マルチクラウド/オンプレミス対応のエージェントレス監視とメトリクス収集**  
   Cron Guard は、SSH、WinRM、またはKubernetes APIを介してリモートホストのcronテーブルを動的に取得し、ジョブの実行状態をポーリングします。エージェントのインストールが不要なため、既存のインフラストラクチャへの影響ゼロで導入可能です。収集したメトリクス（実行時間、終了コード、リソース使用率）は、Prometheus互換のフォーマットでエクスポートされ、Grafanaなどの可視化ツールとシームレスに連携します。

3. **異常検知エンジンによる動的閾値アラートと履歴分析**  
   過去30日間の実行データを基に、ジョブごとの「正常な実行時間帯」と「標準偏差」を自動学習します。例えば、通常5分で完了するジョブが突如15分かかった場合、固定閾値ではなく、統計的に異常と判断してアラートを発報します。さらに、ジョブの標準出力/エラー出力を全文保存し、障害発生時にSlackやPagerDutyへ該当ログの抜粋を自動送信。エンジニアが即座に原因を特定できるよう支援します。

### 使い方

1. **ジョブ定義ファイルの作成と依存関係の記述**  
   プロジェクトルートに `cron-guard.yml` を作成し、以下のようにジョブ名、cron式、実行コマンド、依存関係をYAML形式で記述します。
   ```yaml
   jobs:
     data_pipeline:
       schedule: "0 */2 * * *"
       command: "/usr/local/bin/run_pipeline.sh"
       depends_on: []
       timeout: 300
     report_generator:
       schedule: "30 */2 * * *"
       command: "/usr/local/bin/generate_report.sh"
       depends_on: [data_pipeline]
       timeout: 600
   ```
   依存関係を `depends_on` フィールドにリスト形式で指定することで、Cron Guard はジョブの実行順序を自動的に制御します。上記の例では、`report_generator` は `data_pipeline` の正常完了を待ってから実行されます。

2. **監視エージェントの起動とリモートホストへの接続設定**  
   Cron Guard の監視エージェントを起動し、SSHキーまたはパスワード認証でリモートホストに接続します。以下のコマンドで、ホスト一覧を指定して監視を開始します。
   ```bash
   cron-guard agent start --hosts "192.168.1.10,192.168.1.11" --config ./cron-guard.yml
   ```
   エージェントは、各ホストのcronテーブルと実際のジョブ実行プロセスを監視し、収集したデータをローカルのSQLiteデータベースに保存します。また、`--export-prometheus` フラグを付与することで、メトリクスをPrometheusのプッシュゲートウェイに送信可能です。

3. **アラートルールの設定と通知チャネルの統合**  
   Web UI（デフォルトポート8080）にアクセスし、ジョブごとにアラートルールを定義します。例えば、`data_pipeline` が3回連続でタイムアウトした場合にクリティカルアラートを発報するルールを設定します。通知チャネルは、Slack、Microsoft Teams、PagerDuty、メールに対応。Slackの場合、以下のようにWebhook URLを登録するだけで完了します。
   ```
   設定 > 通知チャネル > Slack > Webhook URL: https://hooks.slack.com/services/xxx/yyy/zzz
   ```
   アラートが発報されると、Cron Guard はジョブの最新の標準エラー出力と実行時間の推移グラフを自動添付してSlackに投稿します。これにより、エンジニアはSlack上で一次判断が可能です。

### ベストプラクティス

1. **ジョブの冪等性を確保し、リトライ戦略を事前に定義する**  
   Cron Guard の依存関係管理を最大限活用するには、各ジョブが冪等（何度実行しても同じ結果になる）であることが前提です。特に、データベースへの書き込みや外部API呼び出しを行うジョブは、重複実行による副作用を防ぐために、ジョブ内で一意のトランザクションIDを発行し、既存レコードの有無をチェックするロジックを組み込みましょう。また、Cron Guard の設定で `retry_count` と `retry_interval` を指定し、一時的なネットワーク障害やリソース競合に対処するリトライ戦略を事前に定義しておくことを推奨します。例えば、`retry_count: 3`、`retry_interval: 60` と設定すれば、失敗後60秒間隔で最大3回再試行します。

2. **実行時間のベースラインを週単位で分析し、季節変動に対応する**  
   Cron Guard の異常検知エンジンは、デフォルトで過去30日間のデータを学習しますが、週末や月末に処理量が増えるジョブ（例：月次レポート生成）では、誤検知が発生しやすくなります。これを回避するには、ジョブ定義に `seasonality: weekly` フラグを追加し、異常検知アルゴリズムを曜日単位で調整します。具体的には、`cron-guard.yml` の該当ジョブに以下のように記述します。

   ```yaml
   jobs:
     monthly_report:
       schedule: "0 0 1 * *"
       command: "/usr/local/bin/generate_monthly.sh"
       seasonality: weekly
       timeout: 1800
   ```
