# cron-guard - CRO_008

## 概要

Cron Guard は、cronジョブの実行管理、監視、障害検知に特化したプロフェッショナルツールです。スケジュールタスクの失敗や遅延をリアルタイムで検出し、システム全体の信頼性を向上させます。本ツールは、分散環境におけるジョブスケジューリングの可視化と制御を実現します。

## 詳細な内容

### Cron Guard とは？

Cron Guard は、従来のcronジョブ管理における「実行結果の不透明性」と「障害対応の遅れ」という2つの根本的問題を解決するために設計されました。ジョブの実行履歴、リトライロジック、アラート通知を一元管理し、DevOps チームが迅速に問題を特定・対応できる環境を提供します。特に、複数サーバーにまたがるジョブの依存関係管理や、実行時間の統計分析に強みを持ちます。

### コア機能

- **ジョブ実行履歴の完全トレース**  
  各ジョブの開始・終了時刻、終了コード、標準出力/エラー出力、実行ノード情報を自動収集。過去30日間の全ログをElasticsearchに保存し、任意の期間・条件で検索可能。たとえば、`exit_code != 0` かつ `node == web-03` のような複合条件クエリをサポート。

- **適応型リトライとバックオフ戦略**  
  ジョブ失敗時に、指数バックオフ（Exponential Backoff）をベースにしたリトライを自動実行。初回リトライは5秒後、2回目は25秒後、3回目は125秒後といった具合に、リトライ間隔を動的に調整。最大リトライ回数はジョブ単位で設定可能で、リトライ上限到達時にはSlackやPagerDutyへ即座にエスカレーション通知を送信。

- **依存関係グラフの可視化とDAG実行制御**  
  ジョブ間の依存関係をDAG（有向非巡回グラフ）として定義可能。たとえば、`data_import` が成功した後にのみ `data_transform` を実行し、さらに `data_export` を実行するといったチェーン制御をGUI上で設定。依存関係にループが発生した場合、自動で検出して警告を表示。

### 使い方

1. **ジョブ定義ファイルの作成と登録**  
   まず、YAML形式でジョブ定義ファイルを作成します。以下は、データベースバックアップジョブの例です。  
   ```yaml
   job:
     name: db_backup
     schedule: "0 2 * * *"
     command: "/usr/local/bin/pg_dump -U admin -F c mydb > /backup/mydb_$(date +%Y%m%d).dump"
     retry:
       max_attempts: 3
       backoff: exponential
     notify:
       on_failure: slack
       channel: "#ops-alerts"
   ```
   このファイルをCron Guardの管理コンソールからアップロードし、`cron_guard_job_register db_backup.yaml` コマンドで登録します。登録後、自動でcronエントリが生成され、監視が開始されます。

2. **アラートルールの設定とテスト**  
   ジョブが失敗した場合の通知ルールを設定します。管理コンソールの「アラート設定」画面で、以下の条件を指定します。  
   - トリガー条件：ジョブの終了コードが0以外、または実行時間が許容最大時間（例：300秒）を超過  
   - 通知先：Slack Webhook URL または PagerDuty サービスキー  
   - エスカレーション：初回通知から5分以内に確認がない場合、2次通知を送信  
   設定後、テストボタンで擬似的な失敗イベントを生成し、通知が正しく届くか確認します。

3. **実行履歴の分析とレポート生成**  
   週次・月次のジョブ実行レポートを自動生成します。管理コンソールの「レポート」タブで、期間を指定して「レポート生成」ボタンをクリック。レポートには以下の項目が含まれます。  
   - 全ジョブの成功率（%）  
   - 平均実行時間と標準偏差  
   - 最も失敗が多いジョブのトップ5  
   - リトライが発生したジョブの一覧とリトライ回数  
   生成されたレポートはCSVまたはPDFでエクスポート可能で、チーム内で共有できます。

### ベストプラクティス

- **ジョブの実行時間に基づくアラート閾値の動的最適化**  
  ジョブの実行時間は日々変動するため、静的な閾値を設定すると誤検知が増えます。Cron Guardの「学習モード」を最初の2週間有効にし、各ジョブの実行時間の平均と標準偏差を自動計算。その後、アラート閾値を「平均 + 3σ」に設定すると、異常な遅延だけを正確に検出できます。この方法により、誤検知率を80%以上削減した事例があります。

- **依存関係チェーンにおけるクリティカルパスの監視**  
  複数のジョブが直列に連なる場合、全体のスループットは最も遅いジョブに依存します。Cron Guardの「クリティカルパス分析」機能を使い、DAG内で最も長い実行時間を持つジョブを特定。そのジョブに対しては、リトライ回数を増やす（例：5回）か、並列実行可能なジョブと分割することを推奨します。実際の運用では、クリティカルパス上のジョブのリトライ回数を2倍にすることで、全体の成功率が99.5%から99.95%に向上したケースがあります。
