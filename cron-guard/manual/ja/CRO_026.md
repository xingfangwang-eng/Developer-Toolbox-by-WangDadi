# cron-guard - CRO_026

## 概要

Cron Guard は、cron ジョブの実行監視・障害検知に特化したプロフェッショナルツールです。スケジュールタスクの未実行、遅延、異常終了をリアルタイムで検出し、即座にアラートを発報します。本ツールは、ミッションクリティカルなバッチ処理基盤の安定運用を支援します。

## 詳細な内容

### Cron Guard とは

Cron Guard は、分散環境下で動作する cron ジョブの実行状態を一元管理し、障害発生時に迅速な対応を可能にする監基盤です。従来のシェルスクリプトベースの監視では捉えきれなかった、ジョブの「未実行」や「実行遅延」を、独自のハートビート機構とタイムウィンドウ分析により検出します。また、ジョブのログを構造化データとして保存し、事後分析やパフォーマンスチューニングに活用できます。

### コア機能

- **ハートビート型死活監視とタイムウィンドウ分析**  
  各 cron ジョブは、実行開始時に一意のハートビートシグナルを送信します。Cron Guard は、ジョブごとに設定された期待実行間隔と最大許容遅延（例：5分）に基づき、シグナルが届かない場合に即座にアラートを発報します。これにより、ジョブの完全な未実行や、想定外の遅延を検出します。

- **ジョブ実行履歴の構造化ログとメトリクス可視化**  
  全ジョブの実行結果、開始・終了時刻、終了コード、標準出力・エラー出力を構造化ログとして保存。Prometheus 形式のメトリクスエンドポイントを提供し、Grafana 等でジョブの実行時間推移、成功率、遅延分布を可視化できます。

- **依存ジョブチェーン管理と条件付きトリガー**  
  ジョブ間に依存関係を定義可能。例えば「ジョブAの正常終了を確認後、ジョブBを実行する」といったワークフローを、YAMLベースの設定ファイルで宣言的に記述できます。ジョブBは、ジョブAの終了コードや出力内容に応じて実行条件を動的に変更できます。

### 使い方

1. **エージェントのインストールとジョブ登録**  
   監視対象サーバーに Cron Guard エージェントをインストールします。エージェントは、`cron_guard register --job-name <ジョブ名> --schedule "*/5 * * * *" --timeout 300` のように、監視対象ジョブの名前、cron 式、タイムアウト値を登録します。登録後、エージェントはジョブの実行を監視し、ハートビートを管理サーバーに送信し始めます。

2. **アラートルールと通知先の設定**  
   管理サーバーの設定ファイル（`/etc/cron_guard/config.yml`）で、アラートルールを定義します。例えば、ジョブが2回連続でタイムアウトした場合に、Slack と PagerDuty に通知するルールを記述します。通知先は、Webhook URL やメールアドレスを指定可能です。設定後、`cron_guard reload` で設定を再読み込みします。

3. **ダッシュボードでの状態確認と障害対応**  
   Web UI ダッシュボードにアクセスし、全ジョブの実行状態をリアルタイムで確認します。障害が発生したジョブをクリックすると、直近の実行ログとエラー内容が表示されます。また、ログから類似の過去障害を検索し、再発防止策を迅速に特定できます。

### ベストプラクティス

- **ジョブのタイムアウト値は、実行時間のP99に基づいて設定する**  
  タイムアウト値を短く設定しすぎると、正常なジョブが誤検知される原因になります。過去1週間の実行時間のP99（99パーセンタイル）を計測し、その値に10～20%のマージンを加えた値をタイムアウト値として設定することを推奨します。これにより、異常な遅延のみを検出し、誤報を最小限に抑えられます。

- **依存ジョブチェーンには、明示的なタイムアウトとリトライポリシーを設定する**  
  依存ジョブチェーンを組む際は、各ジョブに個別のタイムアウトとリトライ回数を設定してください。例えば、「ジョブAが5分以内に正常終了しない場合、ジョブBをスキップし、ジョブAを最大2回リトライする」といったポリシーを定義します。これにより、チェーン全体の停止を防ぎ、部分的な障害からの復旧を自動化できます。
