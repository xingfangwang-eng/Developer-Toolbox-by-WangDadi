# cron-guard - CRO_028

## 概要

Cron Guard は、cron ジョブの実行監視・障害検知に特化したプロフェッショナルツールです。スケジュールタスクの未実行、タイムアウト、異常終了をリアルタイムで検出し、即座に通知します。複数サーバー・複数環境にまたがるジョブ管理を一元化し、運用負荷を大幅に軽減します。

## 詳細な内容

### Cron Guard とは

Cron Guard は、cron ジョブの死活監視と実行履歴の可視化を提供する SaaS 型プラットフォームです。ジョブのスケジュール定義、実行ログの収集、アラート発報、および障害解析を一貫して行います。従来の `cron` と `syslog` に依存した運用から脱却し、ジョブの「実行されなかった」を見逃さない仕組みを実現します。

### コア機能

1. **ジョブ実行のヘルスチェックとタイムアウト検知**  
   各ジョブに期待される最大実行時間を設定可能。設定時間を超過したジョブは自動的にタイムアウトとしてマークされ、即座に Slack / PagerDuty / メールへアラートが送信されます。ジョブの実行間隔が `*/5 * * * *` であれば、5分以内に次の実行が開始されない場合も異常として検出します。

2. **ジョブチェーンと依存関係管理**  
   ジョブ A が成功した後でなければジョブ B を実行しない、といった依存関係を DAG 形式で定義可能。実行順序の制御と、依存先ジョブの失敗時の後続ジョブ自動停止・通知を実装。複雑なバッチ処理パイプラインも単一のダッシュボードで管理できます。

3. **実行履歴の全文検索と統計分析**  
   全ジョブの実行ログを構造化データとして保存。実行時間の中央値・P95・P99、失敗率、直近の実行結果を時系列で可視化します。特定のエラーメッセージを含むログを瞬時に検索可能で、障害原因の特定と再発防止に貢献します。

### 使い方

1. **ジョブの登録と監視設定**  
   - 管理画面の「ジョブ追加」から、ジョブ名、cron 式（例: `0 3 * * *`）、期待実行時間（秒）を入力。  
   - ジョブの実行開始時に、Cron Guard が提供する HTTP エンドポイント（`https://your-instance.cron-guard.io/job/{job_id}/start`）を叩くよう、cron コマンドに `curl` を埋め込む。  
   - 実行終了時には同じエンドポイントに `?status=success` または `?status=failure` を付与してリクエスト。  
   - これにより、ジョブの開始・終了・成否がリアルタイムで反映される。

2. **アラートルールのカスタマイズ**  
   - アラート条件として「ジョブがタイムアウトした」「ジョブが失敗した」「ジョブが予定時刻から10分以上遅延した」を選択。  
   - 通知先を Slack の Webhook URL、PagerDuty のサービスキー、または SMTP で指定。  
   - エスカレーションポリシーを設定可能：1回目の失敗は Slack のみ、2回連続失敗は PagerDuty へ通知、という段階的なアラート運用が可能。

3. **ダッシュボードでのモニタリングと障害対応**  
   - ダッシュボードには全ジョブのステータス（正常/警告/異常）が一覧表示。  
   - 異常ジョブをクリックすると、直近10回の実行履歴と各実行のログ出力が確認できる。  
   - 「再実行」ボタンで該当ジョブを即座に再起動可能。再実行時には、元の実行IDと紐づけてログを保存するため、事後分析が容易。

### ベストプラクティス

1. **ジョブの分割とタイムアウト設定の適正化**  
   1つの cron ジョブに複数の処理を詰め込まず、処理単位でジョブを分割する。各ジョブの期待実行時間は、過去の実績から P95 の1.5倍程度に設定する。これにより、突発的な遅延を許容しつつ、明らかな異常を早期に検出できる。

2. **依存関係の明示と障害波及の防止**  
   ジョブチェーンを定義する際は、後続ジョブが依存するジョブの失敗を自動検知し、チェーン全体を停止する設定を推奨する。これにより、無駄なリソース消費やデータ不整合を防ぐ。また、チェーンの各段階で再試行回数（例: 3回）と再試行間隔（例: 5分）を設定し、一時的な障害には自動復旧させる。
