# cron-guard - CRO_010

## 概要

Cron Guard は、cron ジョブやスケジュールタスクの信頼性を徹底的に向上させるプロフェッショナル向け監視ツールです。ジョブの欠落、遅延、異常終了をリアルタイムで検知し、運用チームの負荷を最小限に抑えます。特に、マイクロサービスアーキテクチャや分散環境でのジョブ管理において、その真価を発揮します。

## 詳細な内容

### Cron Guard とは？

Cron Guard は、単なるジョブスケジューラーではありません。**ジョブの実行状態をエンドツーエンドで可視化し、異常を自動検知・通知する監視基盤**です。従来の cron デーモンが持つ「ジョブが本当に実行されたか」「正常に完了したか」という不確実性を排除し、SLA を厳守するための強力な武器となります。

### コア機能

- **ジョブ実行の整合性チェックと自動リカバリー**  
  各ジョブの予定実行時刻と実際の実行時刻を秒単位で比較。遅延が閾値を超えた場合、ジョブを自動的に再実行し、失敗時にはスタックトレースと共に管理者へ即時通知します。これにより、手動での確認作業を大幅に削減します。

- **分散環境対応のジョブ依存関係管理**  
  複数サーバーやコンテナ間で実行されるジョブの依存関係をグラフ構造で定義。あるジョブの完了をトリガーに、後続ジョブを確実に起動します。依存関係の循環や欠落を事前に検出し、実行前に警告を発します。

- **ジョブ履歴の全文検索とメトリクスダッシュボード**  
  過去の全ジョブ実行ログを Elasticsearch に保存。実行時間、終了コード、標準出力/エラー出力を対象に、全文検索が可能です。また、ダッシュボードではジョブの成功率、平均実行時間、リソース消費量を可視化し、ボトルネックの特定を支援します。

### 使い方

1. **ジョブ定義ファイルの作成**  
   プロジェクトルートに `cron-guard.yml` を作成し、監視対象のジョブを YAML 形式で定義します。各ジョブには、実行コマンド、スケジュール（cron 式）、タイムアウト値、リトライ回数を指定します。依存関係がある場合は、`depends_on` フィールドで先行ジョブを列挙します。

   ```yaml
   jobs:
     backup_db:
       command: "pg_dump -U admin mydb > /tmp/backup.sql"
       schedule: "0 2 * * *"
       timeout: 600
       retries: 3
       depends_on: []
     sync_to_s3:
       command: "aws s3 cp /tmp/backup.sql s3://my-bucket/"
       schedule: "0 3 * * *"
       timeout: 300
       retries: 2
       depends_on:
         - backup_db
   ```

2. **監視エージェントの起動とジョブの登録**  
   `cron-guard agent start` コマンドを実行し、エージェントを起動します。エージェントは `cron-guard.yml` を読み込み、各ジョブのスケジュールを内部のスケジューラーに登録します。登録後、`cron-guard jobs list` で全ジョブの状態（pending/running/success/failed）を確認できます。

   ```bash
   $ cron-guard agent start --config ./cron-guard.yml
   [INFO] Agent started. 2 jobs registered.
   $ cron-guard jobs list
   Job Name      | Status   | Next Run
   backup_db     | pending  | 2024-03-15 02:00:00
   sync_to_s3    | pending  | 2024-03-15 03:00:00
   ```

3. **アラートルールの設定とテスト**  
   ジョブの失敗や遅延を検知するためのアラートルールを設定します。`cron-guard alert add` コマンドで、通知先（Slack、メール、PagerDuty）とトリガー条件（例：3回連続失敗、遅延が60秒超過）を指定します。設定後、`cron-guard alert test` でテスト通知を送信し、設定が正しいことを確認します。

   ```bash
   $ cron-guard alert add --job backup_db --channel slack --webhook https://hooks.slack.com/services/xxx --on-failure 3
   [INFO] Alert rule created for job 'backup_db'.
   $ cron-guard alert test --job backup_db
   [INFO] Test notification sent to Slack.
   ```

### ベストプラクティス

- **ジョブのべき等性を設計に組み込む**  
  リトライや依存関係の再実行が発生するため、各ジョブは何度実行しても同じ結果を保証するべき等性を持つべきです。例えば、データベースのバックアップジョブは、既存のバックアップファイルを上書きするのではなく、タイムスタンプ付きのファイル名で保存するように設計します。これにより、予期せぬ重複実行がデータを破壊するリスクを回避できます。

- **ジョブ実行時間のベースラインを定期的に更新する**  
  システムの負荷変動やデータ量の増加に伴い、ジョブの実行時間は変化します。Cron Guard のダッシュボードで過去30日間の平均実行時間を確認し、月に一度、タイムアウト値やアラートの閾値を更新することを推奨します。この習慣により、誤検知を減らし、真の異常に集中できます。
