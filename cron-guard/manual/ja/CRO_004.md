# cron-guard - CRO_004

## 概要

Cron Guard は、マイクロサービス環境における cron ジョブの実行管理・監視・障害復旧を一元化するエンタープライズグレードのツールです。数千のスケジュールタスクを抱えるシステムでも、レイテンシ 10ms 未満のジョブディスパッチと、秒単位の障害検知を実現します。キーワード：cron job 監視、スケジュールタスク管理、ジョブオーケストレーション。

## 詳細な内容

### とは Cron Guard？

Cron Guard は、従来の crontab や systemd タイマーが抱える「実行漏れの検知困難」「分散環境での同期不全」「障害時のロストジョブ」といった根本課題を解決するために設計されたプロフェッショナルツールです。各ジョブの実行履歴をイミュータブルログとして保存し、任意の時点へのロールバックと再実行をサポートします。さらに、Prometheus メトリクスとネイティブ統合し、Grafana ダッシュボードでリアルタイムのジョブ健全性を可視化します。

### コア機能

- **ジョブ依存関係グラフとクリティカルパス最適化**  
  ジョブ間に定義された依存関係を DAG（有向非巡回グラフ）として自動構築。実行順序の整合性を保証し、並列実行可能なジョブは自動スケールアウト。クリティカルパス分析により、全体のジョブチェーン完了時間を最大 40% 短縮します。

- **フェイルオーバーとリージョン間レプリケーション**  
  アクティブ-アクティブ構成のワーカーノードが、ジョブ実行中にノード障害が発生した場合でも、残存ノードが自動的にジョブを引き継ぎます。さらに、リージョン間のレプリケーションにより、AWS 東京リージョン障害時でも 30 秒以内にバージニアリージョンへフェイルオーバー可能。

- **ジョブ実行結果の差分検証と自動ロールバック**  
  ジョブ実行前後のシステム状態（ファイルハッシュ、DB レコード数、API レスポンス）をスナップショット比較。事前定義された閾値を超える差分が検出された場合、ジョブを自動的にロールバックし、管理者に Slack / PagerDuty 経由でアラートを送信します。

### 使い方

1. **ジョブ定義ファイルの作成とスキーマバリデーション**  
   `cron-guard.yml` にジョブを YAML 形式で記述します。各ジョブには `name`、`schedule`（cron 式）、`command`（実行コマンドまたは Docker イメージ）、`retry`（最大リトライ回数と間隔）、`dependencies`（依存ジョブ名の配列）を必須項目として設定。  
   ```
   jobs:
     - name: data_pipeline_etl
       schedule: "0 3 * * *"
       command: "docker run --rm etl-image:latest"
       retry:
         max_attempts: 3
         interval: 30s
       dependencies: ["db_migration_check"]
   ```
   設定後、`cron-guard validate cron-guard.yml` を実行し、スキーマエラーや循環依存がないことを確認します。

2. **ワーカーノードのクラスタリングとジョブディスパッチ**  
   3 台以上のワーカーノードを起動し、`cron-guard cluster init --nodes 10.0.1.10,10.0.1.11,10.0.1.12` でクラスタを形成。各ノードは Raft 合意アルゴリズムによりリーダーを選出し、リーダーのみがジョブスケジューリングを担当します。ジョブ実行時は、リーダーが各ワーカーの負荷（CPU、メモリ、ジョブキュー長）を考慮して最適なノードにディスパッチ。  
   ```
   cron-guard cluster status
   # 出力例: Leader: 10.0.1.10, Followers: 10.0.1.11, 10.0.1.12, Health: all green
   ```

3. **ジョブ実行履歴のクエリと障害復旧**  
   特定ジョブの実行履歴を確認するには `cron-guard history --job data_pipeline_etl --since 2025-03-01` を実行。各エントリには実行開始時刻、終了時刻、終了コード、標準出力/エラーのログ、実行ノードが記録されています。障害が発生したジョブを再実行する場合、`cron-guard rerun --job data_pipeline_etl --run-id 20250301103000` で指定の run ID のジョブを再実行。再実行時は元の依存関係も自動解決され、関連ジョブも必要に応じて再実行されます。

### ベストプラクティス

- **ジョブの冪等性を徹底する**  
  Cron Guard は障害復旧時に同一ジョブを複数回実行する可能性があります。各ジョブは必ず冪等性を保証する設計にしてください。具体的には、ジョブ内で処理開始前に「処理済みフラグ」をチェックし、既に実行済みの場合はスキップするロジックを組み込むことを推奨します。また、ジョブの出力先（ファイル、DB レコード）には一意な ID を付与し、重複挿入を防止してください。

- **ジョブの実行時間にバッファを持たせる**  
  ジョブのスケジュール間隔は、ジョブの平均実行時間の 1.5 倍以上に設定してください。例えば、ジョブが通常 10 分で完了する場合、スケジュール間隔は 15 分以上にします。これにより、ジョブの実行遅延が発生しても次のスケジュール時刻と衝突せず、キューイングによるスタックを防げます。Cron Guard はジョブの実行時間を自動計測し、設定値との乖離が大きい場合はアラートを発する機能も提供しています。
