# cron-guard - CRO_020

## 概要

Cron Guard は、cronジョブの実行失敗やスケジュールのズレをリアルタイムで検知し、障害発生時に即座にアラートを発報するプロフェッショナル向けジョブ監基盤です。特に、複数サーバーにまたがる分散ジョブや、依存関係が複雑なタスクチェーンを持つ環境において、監視の抜け漏れをゼロに近づける設計が特徴です。本ツールは、cronジョブ、スケジュールタスク、ジョブ監視といったキーワードに最適化されたSEOコンテンツとしても構成されています。

## 詳細な内容

### とは Cron Guard?

Cron Guard は、従来のcron監視ツールが抱える「ジョブの実行有無のみを確認する」という限界を突破するために開発されました。具体的には、ジョブの実行時間、出力内容、終了コード、リソース消費量を統合的にログに記録し、異常が検出された場合にはSlackやメール、PagerDutyなどへ即時通知を行います。また、ジョブの実行履歴を可視化するダッシュボードを備えており、過去のトレンドと比較した異常検知も可能です。これにより、深夜のバッチ処理や定期メンテナンスタスクの「気づかないうちに失敗していた」という事態を根本的に排除します。

### コア機能

- **マルチノード分散ジョブの依存関係グラフ監視**  
  複数サーバーで実行されるジョブ間の依存関係をグラフ構造で管理し、特定のジョブが失敗した場合、その影響を受ける後続ジョブを自動的に一時停止または再スケジュールします。依存関係の変更はYAMLファイルで宣言的に記述でき、Git管理と連携可能です。

- **実行時間のパーセンタイルベース異常検知**  
  各ジョブの過去の実行時間データをパーセンタイル（P50, P90, P99）で学習し、現在の実行時間がP99を超えた場合にのみアラートを発報します。これにより、単発的な遅延やネットワークの揺らぎによる誤報を抑制し、本当に深刻なパフォーマンス劣化だけを検出します。

- **ジョブの出力ログに対する正規表現ベースの異常パターンマッチング**  
  ジョブが出力する標準出力・標準エラー出力に対して、ユーザー定義の正規表現パターン（例: `ERROR`, `FATAL`, `timeout`）をリアルタイムでマッチングし、一致した場合に即座にアラートを発報します。出力ログの内容そのものを監視することで、終了コードが0でも内部で問題が発生しているケースを捕捉できます。

### 使い方

1. **ジョブの登録と依存関係の定義**  
   まず、監視対象のcronジョブを`cron-guard.yml`に記述します。各ジョブにはユニークなID、実行コマンド、期待される実行時間（秒）、そして依存するジョブのIDリストを指定します。  
   ```yaml
   jobs:
     - id: data_pipeline_extract
       command: "/usr/local/bin/extract_data.sh"
       expected_duration: 120
       depends_on: []
     - id: data_pipeline_transform
       command: "/usr/local/bin/transform_data.sh"
       expected_duration: 300
       depends_on: ["data_pipeline_extract"]
   ```
   依存関係はDAG（有向非巡回グラフ）として解釈され、Cron Guardは起動時にグラフの整合性を検証します。循環依存が検出された場合、登録は拒否されます。

2. **アラートルールの設定と通知チャンネルの統合**  
   `cron-guard.yml`に`alerts`セクションを追加し、通知先とトリガー条件を定義します。例えば、ジョブがP99を超えた場合にSlackへ通知する設定は以下の通りです。  
   ```yaml
   alerts:
     - type: slack
       webhook_url: "https://hooks.slack.com/services/T00/B00/xxxxx"
       triggers:
         - condition: duration_exceeds_p99
           job_id: "data_pipeline_transform"
     - type: pagerduty
       integration_key: "your_pd_key"
       triggers:
         - condition: job_failed
           job_id: "*"
   ```
   通知は複数チャンネルに同時に送信でき、条件ごとに異なるチャンネルを指定することも可能です。

3. **ダッシュボードでの可視化と履歴分析**  
   Cron Guardは内蔵のWeb UI（ポート8080）を提供します。ダッシュボードでは、各ジョブの直近24時間の実行時間の推移、失敗率、依存グラフの状態をリアルタイムで確認できます。また、過去30日間のデータを基にしたパーセンタイル値のグラフも表示され、ジョブのパフォーマンスが徐々に劣化しているトレンドを視覚的に把握できます。ダッシュボードのURLは`http://<host>:8080/dashboard`です。

### ベストプラクティス

- **ジョブの実行時間にバッファを持たせる**  
  ジョブの`expected_duration`には、通常の実行時間に加えて20〜30%のバッファを設定することを推奨します。これにより、一時的な負荷上昇やネットワーク遅延による誤報を防ぎつつ、深刻な遅延のみを検出できます。例えば、通常120秒で完了するジョブには150秒を設定します。

- **依存関係の変更は必ずPRベースで管理する**  
  `cron-guard.yml`をGitリポジトリで管理し、依存関係の追加・変更は必ずPull Requestを通じて行います。Cron Guardは起動時にYAMLのバリデーションを実行するため、PRのCIパイプラインに`cron-guard validate`コマンドを組み込むことで、依存関係の循環やジョブIDの重複を事前に検出できます。これにより、本番環境での設定ミスによる障害を未然に防げます。
