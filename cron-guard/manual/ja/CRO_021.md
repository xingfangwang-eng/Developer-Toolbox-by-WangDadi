# cron-guard - CRO_021

## 概要

Cron Guard は、cron ジョブの実行漏れ、遅延、異常終了をリアルタイムで検知し、システム全体の信頼性を最大化するためのプロフェッショナルツールです。本プロジェクトは、cron ジョブ管理における「監視の死角」を排除し、スケジュールタスクの健全性を継続的に保証します。

## 詳細な内容

### Cron Guard とは？

Cron Guard は、従来の cron ジョブ管理が抱える「サイレント障害」（エラーなくジョブがスキップされる、または異常終了しても通知されない）を根本から解決するために設計されました。マルチノード環境やコンテナ化されたワークロードにおいても、ジョブの実行履歴、所要時間、終了コードを一元管理し、異常を即座に検知・通知します。

### コア機能

- **ジッター検知と自動リカバリ**  
  ジョブの実行間隔が設定値から±10%以上ずれた場合（例: 毎時0分に実行すべきジョブが5分以上遅延）、即座にアラートを発報。さらに、指定されたリトライポリシーに従って自動再実行を行い、データ欠損を防止します。

- **実行履歴のタイムライン可視化**  
  各ジョブの実行開始時刻、終了時刻、終了コード、標準出力・エラー出力をタイムライン形式で表示。過去30日間の履歴を保持し、特定のジョブのパフォーマンストレンドを分析可能。異常ジョブのパターン学習機能も搭載。

- **マルチノード対応の分散ロック管理**  
  複数サーバーで同一ジョブが重複実行されるのを防ぐため、Redis または PostgreSQL をバックエンドとした分散ロック機構を実装。ノード障害時には自動でロックを解放し、フェイルオーバーを実現。

### 使い方

1. **ジョブ定義ファイルの作成**  
   プロジェクトルートに `cron-guard.yml` を作成し、監視対象のジョブを YAML 形式で定義します。各ジョブには `name`、`schedule`（cron 式）、`command`、`timeout`（秒）、`retry`（最大リトライ回数）を指定。例:  
   ```yaml
   jobs:
     - name: daily_data_sync
       schedule: "0 3 * * *"
       command: "/usr/local/bin/sync_data.sh"
       timeout: 300
       retry: 2
   ```

2. **エージェントのデプロイと起動**  
   各監視対象ノードに `cron-guard-agent` バイナリを配置し、`cron-guard-agent --config ./cron-guard.yml` で起動。エージェントは指定されたスケジュールに従ってジョブを実行し、結果を中央の `cron-guard-server` に HTTP POST で送信します。送信先サーバーは環境変数 `CRON_GUARD_SERVER_URL` で指定。

3. **ダッシュボードでの監視とアラート設定**  
   `cron-guard-server` が提供する Web UI（ポート 8080）にアクセスし、全ジョブの実行状態を確認。各ジョブの「アラート設定」タブから、Slack、メール、PagerDuty への通知チャネルを選択。しきい値（例: 遅延許容時間、エラー率）をカスタマイズし、保存をクリックで即時反映。

### ベストプラクティス

- **ジョブの実行時間をベースライン化する**  
  新規ジョブを追加する際は、まず `dry-run` モードで 3 回以上実行し、平均実行時間を計測。その後、`timeout` 値を平均値の 1.5 倍に設定することで、異常な長時間実行を早期に検知できます。

- **重要度に応じた通知エスカレーションを設計する**  
  クリティカルなジョブ（例: 決済処理、DB バックアップ）には、最初の失敗で即座に Slack 通知 + メール、2 回目の失敗で PagerDuty へのインシデント発行、3 回目の失敗で SMS 通知というエスカレーションポリシーを設定。Cron Guard の `alert.escalation` フィールドで段階的に定義可能です。
