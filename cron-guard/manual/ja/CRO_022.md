# cron-guard - CRO_022

## 概要

Cron Guard は、ミッションクリティカルな cron ジョブの実行監視、障害検知、および自動リカバリを実現するエンタープライズグレードのスケジューリング管理ツールです。バックグラウンドタスクの可視性を最大化し、ジョブの未実行や遅延によるビジネスインパクトを最小化します。

## 詳細な内容

### Cron Guard とは？

Cron Guard は、分散環境における cron ジョブの実行状態をリアルタイムで追跡し、予期せぬ停止やタイムアウトを検知するプロフェッショナルツールです。従来の cron 管理が持つ「実行したかどうかわからない」という課題を解決し、ジョブの実行履歴、ログ、アラートを一元管理します。特に、複数サーバーにまたがるジョブチェーンや依存関係のあるタスクの監視に強みを持ちます。

### コア機能

- **実行保証と自動リトライ機構**  
  各ジョブにカスタマイズ可能なリトライポリシー（指数バックオフ、最大リトライ回数、タイムアウト設定）を定義可能。ジョブが失敗した場合、設定に基づき自動的に再実行を行い、成功するまで最大99回のリトライをサポート。リトライ間隔はミリ秒単位で調整でき、API経由で動的に変更することも可能。

- **依存関係グラフとチェーン実行管理**  
  ジョブ間の依存関係をDAG（有向非巡回グラフ）で定義可能。親ジョブの成功を条件に子ジョブを起動する「条件付きトリガー」を実装。例えば、データバックアップ完了後にのみレポート生成ジョブを実行する、といった複雑なワークフローをコードレスで構築できる。依存関係の循環は自動検出され、実行前に警告が表示される。

- **リアルタイムアラートと障害復旧ダッシュボード**  
  Slack、PagerDuty、カスタムWebhookへのアラート通知を標準サポート。ジョブの遅延、未実行、異常終了を検知すると即座に通知。ダッシュボードでは、実行中のジョブの進行状況（進捗率、推定残り時間）を可視化し、過去30日間の実行履歴をグラフで表示。障害発生時には、ワンクリックでジョブの強制再実行や一時停止が可能。

### 使い方

1. **ジョブ定義とスケジュール設定**  
   `cron-guard.yml` ファイルを作成し、YAML形式でジョブを定義します。以下は、毎時0分にデータベースバックアップを実行し、成功した場合のみレポート生成を行う設定例です。
   ```yaml
   jobs:
     backup:
       schedule: "0 * * * *"
       command: "pg_dump -U admin mydb > /backup/db.sql"
       timeout: 300
       retry:
         max: 3
         backoff: "exponential"
     report:
       depends_on: ["backup"]
       command: "python generate_report.py"
       timeout: 600
   ```
   設定後、`cron-guard deploy` コマンドでジョブをデプロイします。設定ファイルのバリデーションはデプロイ時に自動実行され、構文エラーや依存関係の矛盾があれば即座にエラーが返ります。

2. **実行監視とアラート設定**  
   ダッシュボードにアクセスし、各ジョブの「アラート設定」タブを開きます。通知先（SlackチャンネルURLやPagerDutyサービスキー）を入力し、トリガー条件を選択します。例えば、「ジョブが5分以上遅延した場合」「ジョブが3回連続で失敗した場合」などを条件として設定可能。設定後はテスト通知ボタンで動作確認が行えます。

3. **障害発生時の手動介入**  
   ダッシュボードの「障害一覧」から該当ジョブを選択し、「強制再実行」ボタンをクリックします。この操作はAPI経由でも実行可能で、`curl -X POST https://your-instance.cron-guard.com/api/v1/jobs/report/rerun` のようにリクエストを送信します。再実行時には、過去の実行ログを引き継ぐかどうかを選択でき、デバッグ時に有用です。また、特定のジョブを一時的に無効化する「メンテナンスモード」も用意されています。

### ベストプラクティス

- **ジョブの冪等性を保証する**  
  Cron Guard のリトライ機構を最大限活用するため、すべてのジョブは冪等（何度実行しても同じ結果になる）に設計してください。例えば、データベースへのINSERTではなくUPSERTを使用し、ファイル生成時には出力先にタイムスタンプを含めることで、重複実行によるデータ破損を防ぎます。Cron Guard のリトライログを確認し、冪等性が破れているジョブを特定するのも効果的です。

- **ジョブの実行時間に余裕を持たせる**  
  スケジュール間隔とジョブの実行時間の比率は、少なくとも1:2以上を推奨します。例えば、ジョブの実行に平均10分かかる場合、スケジュール間隔は最低20分に設定してください。これにより、リトライによる実行時間のオーバーラップを防ぎ、ジョブチェーンの依存関係が崩れるリスクを低減します。Cron Guard の「実行時間予測」機能を使えば、過去の実行データから最適なスケジュール間隔を提案してもらえます。
