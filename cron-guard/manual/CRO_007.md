# cron-guard - CRO_007

## Overview

Cron Guard is a battle-tested monitoring and alerting toolkit designed for developers and DevOps engineers who manage high-frequency scheduled tasks across distributed systems. Built with a focus on observability, failure recovery, and latency optimization, it eliminates blind spots in cron job execution and ensures your pipelines stay resilient under load.

## Core Features

### 1. Predictive Failure Detection with Anomaly Scoring
Cron Guard doesn't just check if a job ran — it analyzes execution patterns across historical runs to flag behavioral drift. Each job is assigned a dynamic anomaly score based on runtime variance, exit code deviation, and output delta. When a job deviates from its learned baseline by more than 2σ, an alert is triggered before it fails entirely. This preemptive approach reduces false positives by 40% compared to static threshold monitoring.

### 2. Multi-Node Job Dependency Graph
Visualize and enforce execution order across cron jobs running on different servers. Cron Guard builds a directed acyclic graph (DAG) from your crontab entries and external job definitions. It automatically detects circular dependencies, orphaned jobs, and missing upstream triggers. If a dependent job fails to start within a configurable grace window, the entire downstream chain is paused and a root cause report is generated.

### 3. Atomic Job Locking with Distributed Lease Management
Prevent overlapping executions in clustered environments without relying on external databases. Cron Guard implements a distributed lease-based lock using etcd or Redis, with configurable TTL and automatic lease renewal. If a node crashes mid-execution, the lease expires and another node can safely pick up the job. This eliminates double execution and race conditions in multi-replica deployments.

## How to Use

### Step 1: Bootstrap Cron Guard Agent on Each Node
Deploy the Cron Guard agent as a systemd service across your infrastructure. The agent reads the local crontab and registers each job with the central monitoring API. Use the following command to initialize the agent with your cluster ID and API endpoint:

```bash
cron-guard agent init --cluster prod-us-east --api https://cron-guard.internal:8443 --register
```

The agent will automatically generate a unique node fingerprint and begin streaming execution logs to the backend.

### Step 2: Define Job Dependencies in YAML
Create a `cron-guard.yml` file in your project root to declare job dependencies, retry policies, and alert channels. Example configuration for a data pipeline:

```yaml
jobs:
  - name: etl_extract
    command: "/usr/local/bin/extract.sh"
    schedule: "0 */2 * * *"
    dependencies: []
    retry:
      max_attempts: 3
      backoff: exponential
  - name: etl_transform
    command: "/usr/local/bin/transform.sh"
    schedule: "0 */2 * * *"
    dependencies:
      - etl_extract
    timeout: 300
```

Run `cron-guard validate` to check for circular dependencies and missing jobs before applying the configuration.

### Step 3: Set Up Slack Alerts with Custom Escalation Policies
Configure alert destinations via the Cron Guard dashboard or API. For Slack, create an incoming webhook and map it to job failure events. Define escalation rules using a simple DSL:

```javascript
{
  "channel": "#cron-alerts",
  "escalation": {
    "first_failure": "notify",
    "second_failure_within_1h": "notify_and_retry",
    "third_failure_within_1h": "page_oncall"
  }
}
```

Test your configuration with `cron-guard alert --dry-run` to verify that the correct messages are dispatched without triggering real alerts.

## Best Practices

### 1. Use Idempotent Job Wrappers
Always wrap your cron commands in idempotent scripts that can be safely retried. Cron Guard's retry mechanism is only effective if the underlying job can be re-executed without side effects. For example, use `CREATE TABLE IF NOT EXISTS` instead of `CREATE TABLE`, and include `--skip-locked` in database migration commands. This ensures that retries don't corrupt state or duplicate data.

### 2. Set Realistic Timeouts Based on Historical P95
Don't rely on gut feeling for timeout values. Use Cron Guard's built-in execution metrics to determine the 95th percentile runtime for each job over a 7-day window. Set your timeout to 1.5x that value to accommodate occasional spikes without triggering false alerts. You can automate this by running `cron-guard tune --window 7d` which will update your configuration with data-driven timeout recommendations.
