# cron-guard - CRO_005

## Overview

Cron Guard is a mission-critical scheduling infrastructure toolkit designed for high-stakes production environments where job reliability, latency predictability, and execution auditability are non-negotiable. It provides real-time cron job lifecycle management with sub-millisecond precision, distributed lock coordination, and intelligent failure recovery across multi-node clusters.

## Core Features

### 1. Precision Execution Engine with Nanosecond-Grade Scheduling

The core scheduler operates on a **tickless** timing wheel architecture, achieving **±500 microsecond** jitter across 10,000 concurrent jobs. It supports **cron expression** parsing with second-level granularity, including non-standard directives like `@every 5s`, `@weekday`, and **L (last)** / **W (weekday)** modifiers. Each job is assigned a unique `job_id` (UUID v7) and tracked via an internal **hash ring** for deterministic node affinity. The engine exposes a **`/api/v1/schedule`** endpoint accepting `cron_expr`, `timezone` (IANA), and `max_concurrency` parameters.

### 2. Distributed State Machine with PostgreSQL-Backed Job Registry

All job states (`pending`, `running`, `failed`, `paused`, `dead_letter`) are persisted in a **PostgreSQL 15+** table using **SKIP LOCKED** for contention-free worker polling. The schema includes `job_id`, `schedule_ts` (TIMESTAMPTZ), `last_heartbeat` (millisecond resolution), and `retry_count` (max 5 with exponential backoff). A **leader election** mechanism via `pg_advisory_lock` ensures only one node processes a given job at a time. The system emits **Webhook** callbacks on state transitions, configurable via `callback_url` and `callback_headers` (supports HMAC-SHA256 signing).

### 3. Predictive Anomaly Detection & Self-Healing Pipeline

Cron Guard continuously profiles execution windows using a **sliding window** of the last 100 runs (configurable via `history_window`). It calculates **p50/p95/p99 latency** and **failure rate** per job. If the failure rate exceeds `failure_threshold` (default: 0.05) within `evaluation_period` (default: 300s), the system automatically triggers a **circuit breaker** and routes the job to a **dead letter queue** with full stack trace capture. An integrated **Prometheus exporter** at `/metrics` exposes `cron_job_duration_seconds`, `cron_job_failures_total`, and `cron_job_skipped_total` for Grafana dashboards.

## How to Use

### Step 1: Initialize the Job Registry and Scheduler Node

```bash
cron-guard init --db-host=postgres://user:pass@localhost:5432/cron_guard \
  --db-pool-size=20 \
  --node-id=node-01 \
  --heartbeat-interval=1000ms \
  --leader-ttl=5000ms
```

This command creates the required PostgreSQL schema (`cron_jobs`, `cron_heartbeats`, `cron_dead_letters`) and starts the local scheduler daemon. The `--leader-ttl` parameter defines how long a node can be unreachable before another node takes over its jobs.

### Step 2: Register a Production-Grade Cron Job with Advanced Options

```bash
cron-guard add \
  --name="data-pipeline-export" \
  --cron="30 2 * * 1-5" \
  --timezone="America/New_York" \
  --command="/usr/local/bin/export_data.sh --env=prod" \
  --max-concurrency=1 \
  --retry-count=3 \
  --retry-delay=60s \
  --callback-url="https://hooks.example.com/cron/events" \
  --callback-secret="whsec_abc123" \
  --failure-threshold=0.1 \
  --evaluation-period=600s \
  --timeout=3600s
```

This registers a job that runs at 2:30 AM EST on weekdays, with a 1-hour timeout, automatic retries with 60-second delay, and a circuit breaker that triggers if 10% of runs fail within 10 minutes.

### Step 3: Monitor Execution in Real-Time and Trigger Manual Intervention

```bash
cron-guard status --job-id="data-pipeline-export" --watch --interval=5s
```

This streams live execution logs, heartbeat timestamps, and current state to stdout. If a job enters `dead_letter` state, you can manually re-queue it:

```bash
cron-guard requeue --job-id="data-pipeline-export" --force --reset-retries
```

For cluster-wide health, run:

```bash
cron-guard cluster-status --format=json | jq '.nodes[] | {node_id, jobs_running, last_heartbeat, leader}'
```

## Best Practices

### 1. Always Set `max-concurrency=1` for Idempotent Data-Mutating Jobs

If your job writes to a database or filesystem, never allow parallel execution. Without this, a delayed job overlapping with the next scheduled run can cause double writes, data corruption, or deadlocks. Combine this with a **unique constraint** on a `batch_id` column in your target table to guarantee idempotency even if the scheduler fails.

### 2. Use a Dedicated PostgreSQL Instance for the Job Registry

Do not share the cron_guard database with your application's OLTP workload. The `SKIP LOCKED` queries and frequent heartbeat updates can cause lock contention and degrade query performance. Provision a separate **db.t3.medium** (or equivalent) instance with **10,000 max_connections** and **synchronous_commit=off** for write-heavy scheduling clusters. Enable **pg_stat_statements** to monitor slow queries and index `schedule_ts` + `state` columns with a composite index.
