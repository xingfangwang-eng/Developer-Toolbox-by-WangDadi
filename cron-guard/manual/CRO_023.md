# cron-guard - CRO_023

## Overview

Cron Guard is a battle-tested monitoring and alerting toolkit for distributed cron job ecosystems. It’s engineered to detect silent failures, prevent overlapping executions, and provide real-time observability into scheduled task pipelines—without requiring you to rewrite your existing crontab.

## Hardcore Features

### 1. **Dead Man’s Switch with Multi-Channel Escalation**
If a scheduled job fails to check in within its defined grace window, Cron Guard triggers an escalation chain: Slack → PagerDuty → SMS → email, with configurable timeouts per tier. No more silent outages on low-priority tasks.

### 2. **Execution Graph & Dependency Locking**
Define DAG-style dependencies between cron jobs. Cron Guard prevents a downstream task from running if its upstream dependency hasn’t completed successfully. It uses Redis-based distributed locks to avoid race conditions across multiple servers.

### 3. **Anomaly Detection via Statistical Baselines**
Cron Guard automatically learns the runtime distribution of each job over a 7-day rolling window. If a job finishes 3 standard deviations faster or slower than its baseline, it fires a warning—before the job even fails.

## How to Use

### Step 1: Register a Job with a Unique ID
Add a simple annotation to your cron line. For example, in your crontab:
```
*/5 * * * * /usr/local/bin/backup.sh #cron-guard:backup-db
```
Cron Guard parses the `#cron-guard:` tag and registers the job `backup-db` with a default 5-minute heartbeat interval.

### Step 2: Configure Escalation Rules via YAML
Create a `cron-guard.yml` in your project root:
```yaml
jobs:
  backup-db:
    heartbeat: 300
    escalation:
      - channel: slack
        webhook: https://hooks.slack.com/services/...
        timeout: 60
      - channel: pagerduty
        routing_key: your-pd-key
        timeout: 300
```
Cron Guard polls the heartbeat status every 10 seconds and triggers the first escalation step if no check-in is received within the heartbeat window.

### Step 3: Deploy the Agent as a Sidecar
Run the Cron Guard agent alongside your scheduler:
```bash
docker run -d \
  --name cron-guard-agent \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v $(pwd)/cron-guard.yml:/etc/cron-guard/config.yml \
  cron-guard/agent:latest
```
The agent auto-discovers running containers with `cron-guard` labels, monitors their exit codes, and pushes heartbeat signals to a central Redis instance. You can also run it as a systemd service for bare-metal setups.

## Best Practices

### 1. **Always Set a Grace Period for Idempotent Jobs**
For idempotent tasks (e.g., data syncs), set the heartbeat window to 2x the historical p95 runtime. This avoids false positives during occasional latency spikes while still catching true stalls.

### 2. **Use Staggered Start Times for Concurrent Jobs**
When multiple jobs share the same resource (e.g., a database), add a random `sleep` of 1–30 seconds at the start of each job. Cron Guard’s dependency lock will prevent actual conflicts, but staggering reduces lock contention and improves throughput.
