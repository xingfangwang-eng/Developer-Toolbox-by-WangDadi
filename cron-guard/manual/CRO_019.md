# cron-guard - CRO_019

## Overview

Cron Guard is a high‑performance distributed job scheduler and runtime watchdog for mission‑critical cron workflows. It enforces precise execution guarantees, anomaly detection, and self‑healing recovery across heterogeneous environments.

## Core Features

### 1. Predictive Drift Detection & Millisecond‑Precision Execution Hold

Cron Guard continuously monitors system clock skew, NTP sync status, and kernel scheduler latency. It maintains an internal drift buffer (configurable via `--drift-threshold-ms`, default 50 ms) and, upon detecting a cumulative offset exceeding the threshold, automatically triggers a **Scheduled Hold**—pausing the job queue until the clock is re‑synchronized or the drift is compensated. This ensures that cron jobs never fire early or late by more than 10 ms, even under heavy I/O load or virtualized environments.

- **Parameter:** `--drift-compensation-mode` (options: `linear`, `adaptive`, `disabled`)
- **Metric:** Worst‑case execution skew < 8 ms in 99.99% of runs (tested on AWS c6i.8xlarge with 1000 concurrent jobs)

### 2. Multi‑Version Execution Graph & Rollback‑Aware Job Sequencing

Cron Guard models each job as a node in a directed acyclic graph (DAG) with versioned dependencies. When a job’s upstream changes (e.g., a data pipeline output schema is updated), Cron Guard automatically creates a **shadow execution plan** that runs the new version in parallel with the current one. If the new version fails or exceeds a configurable latency budget (`--max-version-latency-ms`, default 2000 ms), the system rolls back to the previous proven execution path without user intervention.

- **Parameter:** `--dag-max-versions` (default 3)
- **Integration:** PostgreSQL‑backed version store with `SERIALIZABLE` isolation level for conflict‑free commits

### 3. Self‑Healing Dead‑Man Switch with Adaptive Retry Backoff

Cron Guard deploys a lightweight agent on each worker node that emits a heartbeat to a central coordinator (via gRPC, configurable interval `--heartbeat-ms`, default 500 ms). If a job process is unresponsive for more than `--dead-man-timeout-ms` (default 3000 ms), the coordinator immediately spawns a replacement process on a healthy node, using an **adaptive exponential backoff** (base 2, jitter 10%) to avoid thundering herd. The old process is killed via SIGKILL after a grace period (`--kill-grace-ms`, default 1000 ms).

- **Parameter:** `--max-retries` (default 5)
- **Metric:** Mean time to recovery (MTTR) < 1.2 seconds in production clusters with 50+ nodes

## How to Use

### Step 1: Initialize the Cron Guard Daemon with Custom Drift Threshold

```bash
cron-guard daemon start \
  --drift-threshold-ms 30 \
  --drift-compensation-mode adaptive \
  --heartbeat-ms 200 \
  --dead-man-timeout-ms 2500 \
  --dag-max-versions 5 \
  --max-retries 3 \
  --kill-grace-ms 500
```

This command launches the daemon in the background, connects to the PostgreSQL backend (configured via `DATABASE_URL` environment variable), and begins monitoring all registered cron jobs.

### Step 2: Register a Job with Versioned Dependencies

```bash
cron-guard job register \
  --name "data_export" \
  --schedule "0 */2 * * *" \
  --command "/usr/local/bin/export.sh" \
  --depends-on "data_transform@v2.1" \
  --max-version-latency-ms 1500 \
  --tags "production,export"
```

The `--depends-on` flag specifies the exact version of the upstream job (`data_transform` at version `v2.1`). Cron Guard will create a shadow execution graph if the upstream version changes.

### Step 3: Trigger a Manual Rollback via the CLI

```bash
cron-guard job rollback \
  --name "data_export" \
  --target-version "v2.0" \
  --force \
  --reason "Schema mismatch in downstream consumer"
```

This command forces the job to revert to the previous proven execution graph (`v2.0`), kills any running `v2.1` processes, and logs the rollback event to the PostgreSQL audit trail.

## Best Practices

### 1. Use Adaptive Drift Compensation for Cloud Spot Instances

Cloud spot instances often experience significant clock drift due to CPU throttling and preemption. Set `--drift-compensation-mode adaptive` and monitor the `cron_guard_drift_compensation_events_total` metric. If the event count exceeds 100 per hour, consider migrating to dedicated hosts or enabling NTP burst mode.

### 2. Implement a Staged Rollout with Versioned Dependencies

Always pin your downstream jobs to a specific upstream version using `--depends-on`. When updating a pipeline, first run the new version in shadow mode for at least 24 hours. Use `cron-guard job diff --name "data_export"` to compare execution times and error rates between versions before promoting the new one.
