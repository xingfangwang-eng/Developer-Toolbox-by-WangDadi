# cron-guard - CRO_015

## Overview

Cron Guard is an enterprise-grade distributed cron job scheduler and real-time monitoring engine, purpose-built for mission-critical scheduled task orchestration across heterogeneous environments. It provides sub-second execution granularity, intelligent failure recovery, and full audit traceability for DevOps pipelines, ETL workflows, and microservice health checks.

## Detailed Content

### What is Cron Guard?

Cron Guard eliminates blind spots in traditional cron job management by offering a declarative YAML-based job definition system, a centralized scheduler with nanosecond-level timestamp precision, and a self-healing runtime that automatically re-runs failed jobs with exponential backoff (base delay: 500ms, max delay: 64s). It integrates natively with PostgreSQL for persistent job state storage and Prometheus for real-time metric export.

### Core Features

- **Sub‑second Precision Scheduling with Drift Correction**  
  Cron Guard uses a hybrid clock‑sync algorithm (NTP + hardware timestamp counters) to execute jobs at a resolution of 1 millisecond. It automatically detects system clock drift > 10ms and recalibrates the next tick using a PID controller. Jobs can be defined with `cron_expression` (standard 5‑field) or `interval_ms` (integer, min 100). Example: `interval_ms: 250` triggers every 250ms with jitter < 2ms.

- **Distributed Job State Machine with PostgreSQL Backed Quorum**  
  All job lifecycle transitions (PENDING → RUNNING → SUCCESS/FAILED/TIMEOUT) are persisted via `pg_notify` channels and a two‑phase commit protocol. Each job carries a `retry_policy` block: `max_retries: 5`, `backoff_base_ms: 500`, `backoff_multiplier: 2.0`. The scheduler uses an advisory lock (`pg_try_advisory_xact_lock`) to guarantee exactly‑once execution across multiple Cron Guard nodes. Job history is retained for 90 days by default, indexed on `job_name` and `execution_ts`.

- **Real‑time Anomaly Detection and Slack/PagerDuty Integration**  
  Cron Guard exposes a `/metrics` endpoint (Prometheus format) with gauges for `cron_guard_last_success_timestamp_seconds`, `cron_guard_consecutive_failures_total`, and `cron_guard_execution_latency_milliseconds`. When `consecutive_failures_total` exceeds the `alert_threshold` (default: 3), it fires a webhook to Slack or PagerDuty with full job context (last error log, runtime args, node ID). The webhook payload includes a `remediation_suggestion` field generated by a built‑in rule engine (e.g., "Increase `timeout_seconds` from 30 to 60").

### How to Use

1. **Define a High‑Frequency Monitoring Job with Custom Timeout**  
   Create a YAML file `check_health.yml`:
   ```yaml
   job_name: health_ping
   cron_expression: "*/5 * * * *"   # every 5 minutes
   timeout_seconds: 30
   retry_policy:
     max_retries: 3
     backoff_base_ms: 1000
   command: "curl -f http://localhost:8080/health"
   alert_threshold: 2
   webhook_url: "https://hooks.slack.com/services/T00/B00/xxxx"
   ```
   Deploy via `cron-guard apply -f check_health.yml`. The scheduler will immediately register the job, compute the next execution timestamp (accurate to 1ms), and begin the state machine.

2. **Scale to a Multi‑Node Cluster with Database‑Backed Coordination**  
   On each node, set environment variables:
   ```bash
   export CRON_GUARD_NODE_ID="node-$(hostname)"
   export CRON_GUARD_DB_DSN="postgres://user:pass@pg-host:5432/cron_guard?sslmode=require"
   export CRON_GUARD_CLUSTER_MODE="true"
   ```
   Run `cron-guard daemon`. Each node connects to the same PostgreSQL instance and competes for advisory locks on job IDs. Only the lock holder executes the job. If a node crashes, its locks are released within 10 seconds (configurable via `lock_timeout_seconds`), and another node immediately picks up the next tick.

3. **Query Job Execution History and Export Metrics**  
   Use the built‑in CLI to inspect the last 50 runs:
   ```bash
   cron-guard history --job-name health_ping --limit 50 --output json
   ```
   This returns an array of objects with `execution_ts`, `state`, `duration_ms`, `exit_code`, and `error_log`. For Prometheus integration, add a scrape target:
   ```yaml
   - job_name: 'cron_guard'
     scrape_interval: 15s
     static_configs:
       - targets: ['localhost:9091']
   ```
   Grafana dashboards can then visualize `cron_guard_execution_latency_milliseconds` percentiles (p50, p95, p99).

### Best Practices

- **Always Define a `timeout_seconds` Per Job**  
  Without a timeout, a hung job can consume a node’s worker slot indefinitely, causing backpressure. Set `timeout_seconds` to 2× the historical p95 execution time of the job. For database‑backed jobs, also set `statement_timeout` inside the command (e.g., `PGOPTIONS='-c statement_timeout=30s'`). This prevents runaway queries from blocking the scheduler.

- **Use Separate PostgreSQL Schemas for Different Environments**  
  Production, staging, and development should each have their own schema (e.g., `cron_guard_prod`, `cron_guard_staging`). Set `CRON_GUARD_DB_SCHEMA` accordingly. This isolates job definitions and history, preventing accidental cross‑environment interference. Additionally, enable `pg_stat_statements` on the PostgreSQL instance to monitor query performance and index usage for the `cron_guard_job_executions` table.
