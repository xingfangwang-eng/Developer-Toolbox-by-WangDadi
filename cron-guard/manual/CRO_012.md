## What Is Cron Guard?

Cron Guard is a lightweight, high-performance monitoring and alerting tool designed specifically for **scheduled tasks** and **cron job** environments. It provides real-time visibility into job execution, failure detection, and performance drift—without bloat or unnecessary dependencies.

Unlike generic system monitoring tools, Cron Guard is purpose-built for the **ephemeral, stateless, and often silent** nature of cron workflows. It integrates directly with your crontab, systemd timers, or CI/CD schedulers, capturing exit codes, execution duration, and output logs.

## Core Features

### 1. Real-Time Job Drift Detection with Adaptive Baselines

Cron Guard doesn't just monitor whether a job runs—it tracks how long it *should* take. Using rolling window statistics, it establishes a dynamic baseline for each scheduled task. If a job completes 40% faster or 60% slower than its historical norm, an alert is fired immediately. This catches silent failures like database connection timeouts, resource contention, or degraded API endpoints before they escalate.

### 2. Atomic Execution Logging with Structured Output Capture

Every execution is logged atomically, meaning no partial writes or corrupted logs. Output is captured in structured JSON format, including `stdout`, `stderr`, exit code, PID, and environment variables (sanitized). This enables precise post-mortem analysis and integration with log aggregation systems like Elasticsearch, Loki, or Datadog.

### 3. Push-Based Alert Routing with Multi-Channel Fallback

Cron Guard supports push-based alerts via webhooks, Slack, PagerDuty, and email. If the primary channel fails, it automatically falls back to a secondary channel within 10 seconds. This ensures zero missed alerts, even during infrastructure disruptions. Each alert includes a direct link to the execution log and a one-click "re-run" command.

## How to Use

### Step 1: Install and Initialize the Cron Guard Agent

```bash
curl -sSL https://install.cron-guard.io | bash
cron-guard init --token YOUR_API_TOKEN --config /etc/cron-guard/config.yml
```

This installs the agent binary and creates a default configuration file. The `init` command registers the agent with your Cron Guard dashboard, generates a unique agent ID, and starts the background health-check daemon.

### Step 2: Wrap Your Cron Job with the Monitoring Wrapper

```bash
# Instead of:
0 3 * * * /usr/local/bin/backup.sh

# Use:
0 3 * * * cron-guard run --job-id backup-daily -- /usr/local/bin/backup.sh
```

The `cron-guard run` command wraps your existing binary or script. It captures all output, measures execution time, and sends a structured event to the monitoring backend. You can also pass `--timeout 300` to automatically kill jobs that exceed 5 minutes.

### Step 3: Configure Alert Thresholds and Notification Channels

Edit your config file to define alert rules:

```yaml
alerts:
  - job: backup-daily
    on_failure: true
    on_timeout: true
    on_drift: 30%  # Alert if execution time drifts by 30% or more
    channels:
      - slack
      - pagerduty
      - email

channels:
  slack:
    webhook_url: "https://hooks.slack.com/services/T00/B00/xxxx"
  pagerduty:
    integration_key: "YOUR_PD_KEY"
  email:
    smtp_host: "smtp.example.com"
    smtp_user: "alerts@example.com"
    smtp_pass: "****"
```

Run `cron-guard reload` to apply the changes without restarting the daemon.

## Best Practices

### 1. Always Set a Timeout for Every Job

Many cron jobs fail silently because they hang indefinitely. Always define a `--timeout` value in seconds. For example, a database backup that normally takes 2 minutes should have a timeout of 300 seconds. This prevents zombie processes from consuming system resources and masking failures.

### 2. Use Job Labels for Granular Alert Routing

Label your jobs by environment, team, or criticality (e.g., `env=production`, `team=billing`, `severity=critical`). Then create alert rules based on labels rather than individual job IDs. This scales cleanly as your infrastructure grows and allows you to route production alerts to the on-call team while sending staging alerts to a dedicated Slack channel.
