# cron-guard - CRO_027

## Resumen Técnico

Cron Guard es un **orquestador de tareas programadas** diseñado para entornos de producción de alta criticidad. A diferencia de herramientas tradicionales como cron o systemd timers, Cron Guard incorpora un motor de **monitoreo proactivo** que detecta fallos de ejecución, desviaciones de tiempo y fugas de memoria en segundo plano. Ideal para equipos DevOps que necesitan garantizar SLA estrictos en pipelines de datos, backups y procesos batch.

## Funcionalidades Clave (Hardcore)

### 1. **Validación de Dependencias en Cadena (DAG-based Scheduling)**
Cron Guard permite definir **grafos acíclicos dirigidos (DAG)** entre tareas. Si una tarea upstream falla, todas las downstream se detienen automáticamente, evitando corrupción de datos o ejecuciones huérfanas. Soporta reintentos inteligentes con backoff exponencial y notificaciones vía webhook (Slack, PagerDuty).

### 2. **Análisis de Latencias y Desviación Temporal (Jitter Detection)**
El motor incorpora un **algoritmo de detección de jitter** que compara el tiempo real de ejecución contra el programado. Si una tarea se desvía más de un umbral configurable (ej. ±500ms), Cron Guard genera una alerta y registra métricas detalladas en Prometheus. Ideal para entornos donde la sincronización es crítica (ej. trading algorítmico, replicación de bases de datos).

### 3. **Sandboxing de Ejecución con Aislamiento de Recursos**
Cada tarea se ejecuta en un **contenedor efímero** con límites de CPU, RAM y E/S definidos por el usuario. Cron Guard monitorea el uso de recursos en tiempo real y **mata procesos zombie** que excedan los límites, evitando que una tarea maliciosa o con fugas afecte al sistema anfitrión. Soporta Docker, Podman y contenedores nativos de Linux.

## Guía de Uso (Pasos Detallados)

### 1. **Definir un Pipeline de Tareas con Dependencias**
Crea un archivo `cron-guard.yml` en tu repositorio. Usa la sintaxis YAML para definir tareas y sus dependencias:

```yaml
tasks:
  - name: "backup-db"
    schedule: "0 2 * * *"
    command: "pg_dump -U admin production_db > /backups/latest.sql"
    dependencies: []  # Sin dependencias
    resources:
      cpu: 0.5
      memory: 256Mi
  - name: "compress-backup"
    schedule: "0 3 * * *"
    command: "gzip /backups/latest.sql"
    dependencies: ["backup-db"]  # Espera a que backup-db termine con éxito
    resources:
      cpu: 0.2
      memory: 128Mi
```

Ejecuta `cron-guard validate` para verificar la sintaxis y detectar ciclos en el DAG.

### 2. **Configurar Alertas por Desviación Temporal**
Edita el archivo de configuración global (`/etc/cron-guard/config.toml`) para habilitar la detección de jitter:

```toml
[monitoring]
jitter_threshold_ms = 500  # Alerta si la tarea se desvía más de 500ms
alert_endpoints = ["https://hooks.slack.com/services/TXXXX/BXXXX/XXXXX"]
metrics_export = "prometheus"  # Expone métricas en /metrics para Grafana
```

Reinicia el servicio con `systemctl restart cron-guard`. Ahora, cualquier tarea que se ejecute fuera del tiempo programado generará una alerta inmediata.

### 3. **Ejecutar Tareas en Sandbox con Límites de Recursos**
Para lanzar una tarea con aislamiento, usa el flag `--sandbox`:

```bash
cron-guard run --sandbox --task "backup-db" --limits "cpu=0.5,memory=256Mi"
```

Cron Guard creará un contenedor temporal, ejecutará el comando y lo destruirá al finalizar. Si el proceso consume más de 256Mi de RAM, será automáticamente terminado y se registrará un evento de "OOM kill". Puedes ver los logs del sandbox con `cron-guard logs --task "backup-db" --sandbox`.

## Mejores Prácticas (Best Practices)

### 1. **Usa Etiquetas (Tags) para Segmentar Tareas por Prioridad**
En entornos con cientos de tareas, asigna etiquetas como `critical`, `batch` o `maintenance`. Luego configura reglas de alerta diferenciadas: las tareas `critical` deben notificar a PagerDuty, mientras que las `batch` solo registran logs. Ejemplo:

```yaml
tasks:
  - name: "sync-payments"
    tags: ["critical", "finance"]
    alert_on_failure: true
```

### 2. **Implementa un Ciclo de Vida de Retención de Logs**
Cron Guard puede rotar logs automáticamente. Configura en `config.toml`:

```toml
[logs]
retention_days = 30
compression = "gzip"
storage_path = "/var/log/cron-guard"
```

Esto evita que los logs ocupen espacio en disco y facilita auditorías. Combínalo con un exportador a S3 o GCS para almacenamiento a largo plazo.
