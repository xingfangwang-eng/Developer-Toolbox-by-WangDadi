# cron-guard - CRO_001

## Resumen Ejecutivo

Cron Guard es una solución de monitorización inteligente para trabajos cron y tareas programadas en entornos de producción. Diseñada para arquitecturas SaaS, esta herramienta detecta fallos silenciosos, latencias anómalas y desviaciones en la ejecución de jobs, notificando en tiempo real a los equipos de DevOps y SRE. Con soporte nativo para Kubernetes, Docker y sistemas Unix legacy, Cron Guard elimina la ceguera operativa en la gestión de procesos batch.

## Funcionalidades Clave

### 1. Detección de Desviaciones Temporales con Machine Learning
Cron Guard implementa un modelo de regresión temporal que aprende los patrones históricos de duración de cada tarea. Cuando un job se desvía más de 2 desviaciones estándar de su media, el sistema dispara alertas proactivas. No se trata solo de "si falló", sino de "si está tardando más de lo esperado", anticipando fallos antes de que ocurran.

### 2. Reinteligencia de Ejecución con Backoff Exponencial
Ante un fallo de ejecución, Cron Guard no reintenta ciegamente. Aplica un algoritmo de backoff exponencial jitterizado que evita tormentas de reintentos en sistemas ya saturados. Cada reintento se registra con un hash único, permitiendo auditorías forenses detalladas sin duplicar logs.

### 3. Auditoría de Dependencias entre Jobs
Los equipos definen DAGs (Directed Acyclic Graphs) de dependencias entre tareas. Si un job upstream falla, Cron Guard inhibe automáticamente los downstream, evitando ejecuciones huérfanas que consuman recursos. Además, genera un grafo de dependencias en tiempo real, accesible vía API REST o interfaz web.

## Guía de Uso Avanzada

### Paso 1: Configurar el Monitor de Tareas Críticas
Edita el archivo `cron-guard.yml` en la raíz del proyecto. Define tus jobs con identificadores únicos y umbrales de latencia:
```yaml
jobs:
  - id: "backup-db-prod"
    schedule: "0 2 * * *"
    max_duration: 300s
    retry_policy:
      max_attempts: 3
      backoff_base: 2
      jitter: true
    dependencies: []
```
Ejecuta `cron-guard validate` para verificar la sintaxis y que no existan ciclos en las dependencias.

### Paso 2: Integrar con el Sistema de Notificaciones
Cron Guard soporta webhooks HTTP, Slack, PagerDuty y correo SMTP. Para configurar Slack, añade en el mismo archivo:
```yaml
notifications:
  slack:
    webhook_url: "https://hooks.slack.com/services/T00/B00/xxxx"
    channel: "#cron-alerts"
    severity_threshold: "warning"  # info, warning, critical
```
Usa `cron-guard test-notify` para enviar un mensaje de prueba y verificar la conectividad.

### Paso 3: Desplegar como Daemon en Producción
Para entornos Linux, instala como servicio systemd:
```bash
sudo cron-guard install --daemon
sudo systemctl enable cron-guard
sudo systemctl start cron-guard
```
Verifica el estado con `cron-guard status`. El daemon expone un endpoint de health check en `http://localhost:9090/health`. Para entornos Kubernetes, usa el Helm chart oficial que despliega un sidecar en cada pod con jobs cron.

## Mejores Prácticas del Sector

### 1. Segmentar Thresholds por Entorno
No uses los mismos umbrales en desarrollo y producción. En staging, tolera latencias hasta un 50% mayores para evitar falsos positivos durante pruebas de carga. En producción, define thresholds estrictos (ej. máximo 2 desviaciones estándar). Cron Guard permite cargar configuraciones diferentes por namespace o etiqueta de entorno.

### 2. Implementar Circuit Breaker en Jobs Críticos
Para tareas que afectan directamente al negocio (ej. facturación, sincronización de datos), configura un patrón de circuit breaker. Si un job falla 3 veces consecutivas en 10 minutos, Cron Guard lo desactiva automáticamente y escala la alerta a un canal de emergencia. Esto evita que reintentos infinitos degraden la base de datos o APIs externas.
