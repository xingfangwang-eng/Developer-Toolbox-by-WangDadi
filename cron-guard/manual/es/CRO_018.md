# cron-guard - CRO_018

## Resumen

Cron Guard es una solución ligera y robusta diseñada para el monitoreo en tiempo real de trabajos cron y tareas programadas en entornos de producción. Ideal para equipos DevOps que necesitan visibilidad inmediata sobre fallos silenciosos, desviaciones en tiempos de ejecución y dependencias entre tareas. Este contenido está optimizado para profesionales que buscan mejorar la fiabilidad de sus pipelines de automatización.

## Contenido Detallado

### ¿Qué es Cron Guard?

Cron Guard no es solo otro monitor de logs. Es una herramienta proactiva que detecta anomalías en la ejecución de trabajos programados, notifica en tiempo real y ofrece dashboards de rendimiento. Diseñada para entornos multi-cluster y arquitecturas distribuidas, elimina la incertidumbre sobre si una tarea nocturna se ejecutó correctamente o si un script crítico falló sin dejar rastro.

### Características Principales

- **Detección de Desviaciones Temporales**: Cron Guard compara la duración real de cada ejecución con una línea base histórica. Si un trabajo que normalmente tarda 2 segundos comienza a ejecutarse en 10, el sistema genera una alerta automática, permitiendo identificar degradaciones antes de que se conviertan en fallos.
- **Alertas Contextuales con Enriquecimiento de Logs**: No solo notifica que un trabajo falló; adjunta el fragmento del log relevante, el código de salida y las métricas del sistema en el momento del error. Estas alertas se envían a Slack, PagerDuty o webhooks personalizados con formato JSON estructurado.
- **Grafo de Dependencias entre Tareas**: Permite definir relaciones de precedencia entre cron jobs. Si el trabajo A falla, Cron Guard automáticamente bloquea la ejecución del trabajo B (que depende de A) y notifica al equipo, evitando cascadas de errores y datos corruptos.

### Cómo Usar

1. **Configuración Inicial con Archivo YAML**: Crea un archivo `cron-guard.yml` en la raíz de tu proyecto. Define cada trabajo con su nombre, comando, programación cron y umbral de tiempo máximo. Ejemplo:
   ```yaml
   jobs:
     - name: backup_database
       command: pg_dump -U admin mydb > /backups/latest.sql
       schedule: "0 2 * * *"
       max_duration: 300
       dependencies: []
   ```
   Luego ejecuta `cron-guard init` para registrar los trabajos en el agente de monitoreo.

2. **Integración con Webhooks para Notificaciones**: En el mismo archivo, agrega un bloque `notifications` para definir canales. Por ejemplo, para enviar alertas a un canal de Slack:
   ```yaml
   notifications:
     slack:
       webhook_url: "https://hooks.slack.com/services/T00/B00/xxxx"
       channel: "#alerts-cron"
       events: ["failure", "timeout", "dependency_blocked"]
   ```
   Cron Guard enviará un mensaje formateado con el nombre del trabajo, el error y un enlace directo al dashboard.

3. **Revisión de Historial y Métricas en Tiempo Real**: Accede al panel web embebido en `http://localhost:9090`. Allí podrás ver:
   - El estado actual de todos los trabajos (running, success, failed).
   - Un histograma de duración por trabajo en las últimas 24 horas.
   - El grafo de dependencias con nodos coloreados según su estado.
   Para exportar datos a tu sistema de logging centralizado, usa el endpoint API: `GET /api/v1/jobs/history?last=48h`.

### Mejores Prácticas

- **Implementa un Umbral de Tiempo Mínimo y Máximo**: No solo configures un `max_duration` para detectar trabajos lentos. Define también un `min_duration` para identificar trabajos que se ejecutan demasiado rápido (lo que puede indicar que no hicieron nada o que fallaron al inicio). Esto es clave en tareas de ETL o procesamiento de lotes.
- **Usa Etiquetas para Organizar por Entorno o Prioridad**: Añade etiquetas como `env:production`, `critical:true` o `team:data` a tus trabajos en el YAML. Luego filtra en el dashboard por etiquetas para enfocarte en lo que importa. Además, configura reglas de alerta diferenciadas: trabajos críticos pueden notificar a PagerDuty, mientras que los no críticos solo a Slack.
