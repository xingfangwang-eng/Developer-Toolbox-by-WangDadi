# cron-guard - CRO_005

## Resumen

Cron Guard es un sistema de monitoreo y orquestación de tareas programadas diseñado para eliminar la incertidumbre en la ejecución de cron jobs en entornos de producción. Con él, puedes detectar fallos silenciosos, retrasos en la ejecución y desviaciones en los tiempos de respuesta, todo desde una única interfaz centralizada.

## Contenido Detallado

### ¿Qué es Cron Guard?

Cron Guard es una herramienta profesional que resuelve el problema crónico de la falta de visibilidad en los trabajos programados. En lugar de depender de logs dispersos o de scripts artesanales de monitoreo, Cron Guard ofrece un sistema de alertas inteligente, un historial de ejecuciones con trazabilidad completa y una integración directa con tu stack de infraestructura (Docker, Kubernetes, sistemas Linux bare-metal). Está pensado para equipos de DevOps, SREs y desarrolladores que necesitan garantizar la fiabilidad de sus pipelines de datos, backups y procesos batch.

### Características Principales

- **Detección de Anomalías en Tiempo Real con Machine Learning:** Cron Guard no solo verifica si un job se ejecutó o no. Analiza patrones históricos de duración, hora de inicio y consumo de recursos. Si un trabajo que normalmente tarda 30 segundos empieza a ejecutarse en 45 segundos, el sistema genera una alerta de "deriva de rendimiento" antes de que se convierta en un fallo crítico.
- **Reinteligencia de Ejecución (Retry con Backoff Exponencial):** Cuando un job falla, Cron Guard no se limita a notificarlo. Ejecuta automáticamente un reintento con una estrategia de backoff exponencial configurable (por ejemplo, 1s, 2s, 4s, 8s). Además, registra cada intento en un log inmutable, permitiendo auditorías forenses detalladas.
- **Webhook de Estado con Payload Enriquecido:** Cada cambio de estado (éxito, fallo, timeout, retry) dispara un webhook con un payload JSON que incluye no solo el estado, sino también el PID del proceso, el tiempo de ejecución, el código de salida y cualquier mensaje de error capturado en stderr. Esto permite integrar Cron Guard con Slack, PagerDuty, Opsgenie o cualquier sistema de alertas personalizado.

### Cómo Usar

1. **Configurar un Job con Validación de Salida:**  
   En lugar de simplemente ejecutar un comando, define una regla de validación para el código de salida. Por ejemplo, si tu script de backup devuelve `0` solo si todo está bien, pero también devuelve `1` si hay archivos corruptos, puedes configurar Cron Guard para que solo considere el job como "exitoso" si el código de salida es exactamente `0`. Cualquier otro código (como `2` o `3`) se tratará como un fallo controlado, activando la alerta correspondiente.

   ```bash
   cron-guard run --name "backup-db" --command "/usr/local/bin/backup.sh" --exit-code-success 0 --exit-code-warning 1 --timeout 300
   ```

2. **Establecer un Límite de Tiempo (Timeout) con Acción Automática:**  
   Define un timeout máximo para cada job. Si el proceso supera ese límite, Cron Guard lo termina forzosamente (SIGTERM seguido de SIGKILL si es necesario) y registra el evento como un "timeout". Además, puedes configurar una acción posterior, como reiniciar el job o enviar una notificación de emergencia.

   ```bash
   cron-guard run --name "data-pipeline" --command "python3 etl.py" --timeout 600 --on-timeout "restart" --on-timeout-delay 30
   ```

3. **Auditar el Historial de Ejecuciones con Filtros:**  
   Accede al historial completo de ejecuciones con filtros avanzados: por nombre de job, rango de fechas, estado (éxito, fallo, timeout, retry) y tiempo de ejecución. Esto es esencial para análisis post-mortem y para generar informes de fiabilidad mensuales.

   ```bash
   cron-guard history --name "backup-db" --since "2023-10-01" --until "2023-10-31" --status "failure" --format json
   ```

### Mejores Prácticas

- **Implementa un Sistema de Heartbeat para Jobs Críticos:**  
  Para procesos que deben ejecutarse cada minuto (como health checks de servicios), configura un "heartbeat" en Cron Guard. Si el job no se ejecuta dentro de un intervalo de 90 segundos (por ejemplo), el sistema asume que el proceso ha muerto y dispara una alerta inmediata. Esto es más fiable que depender de un simple cron job, ya que cubre casos de fallo silencioso.

- **Usa Etiquetas (Tags) para Organizar y Priorizar Jobs:**  
  Asigna etiquetas como `produccion`, `desarrollo`, `critico`, `batch` a tus jobs. Luego, configura reglas de alerta por etiqueta. Por ejemplo, los jobs etiquetados como `critico` pueden enviar notificaciones a Slack y PagerDuty, mientras que los de `desarrollo` solo envían un correo electrónico. Esto evita la fatiga de alertas y asegura que los problemas importantes reciban atención inmediata.
