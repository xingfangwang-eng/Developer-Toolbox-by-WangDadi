# cron-guard - CRO_007

## Resumen Técnico

Sistema de monitoreo inteligente para tareas programadas (cron jobs) diseñado para entornos de producción de alto rendimiento. Cron Guard detecta fallos silenciosos, latencias anómalas y desviaciones en la ejecución de tareas batch, ofreciendo alertas en tiempo real y auto-remediación sin intervención manual. Ideal para equipos DevOps que gestionan cientos de jobs críticos en infraestructuras distribuidas.

## Funcionalidades Clave (Hardcore)

### 1. Detección de Fallos Silenciosos con Huerfano Watch
Cron Guard implementa un algoritmo de **huérfano watch** que identifica jobs que no reportan su estado de finalización dentro de la ventana esperada. A diferencia de herramientas básicas, no solo monitorea el código de salida, sino que analiza logs de syslog y journald en busca de procesos zombies o forks no controlados. Esto previene acumulaciones de procesos huérfanos que pueden saturar la memoria del servidor.

### 2. Auto-Remediación con Rollback de Tareas
Cuando un cron job falla más de 3 veces consecutivas dentro de una ventana de 5 minutos, Cron Guard ejecuta automáticamente un **rollback de estado** basado en snapshots previos del sistema de archivos o bases de datos. Esto es vital para jobs que modifican datos críticos (como migraciones ETL) y evita corrupciones en cascada.

### 3. Análisis Predictivo de Latencias con ML Ligero
Utiliza un modelo de regresión lineal simple (entrenado localmente con datos históricos de ejecución) para predecir la duración esperada de cada job. Si la latencia real supera en un 150% la predicción, se dispara una alerta de **anomalía temporal**. Esto permite detectar cuellos de botella en recursos (CPU, I/O) antes de que afecten a otros servicios.

## Cómo Usar (Pasos Profesionales)

### 1. Configurar el Agente de Monitoreo con Autenticación TLS
Ejecuta el siguiente comando para desplegar el agente en tu servidor Linux (Ubuntu 22.04+):
```bash
curl -sSL https://cron-guard.io/agent/install.sh | bash -s -- --api-key=TU_API_KEY --tls-cert=/etc/ssl/certs/cron-guard.pem
```
Esto registra el agente en tu panel de control y habilita conexiones cifradas. Verifica el estado con:
```bash
systemctl status cron-guard-agent
```

### 2. Definir Reglas de Monitoreo con Expresiones CRON Mejoradas
En el archivo `/etc/cron-guard/rules.yaml`, define un job crítico con alertas condicionales:
```yaml
job: "backup-db"
cron: "0 2 * * *"
expected_duration: 300  # en segundos
max_retries: 2
on_failure: "rollback"
on_latency: "alert_slack"
```
Cron Guard interpreta `expected_duration` y compara con métricas reales. Si la latencia excede el límite, envía una notificación a Slack con el contexto del error.

### 3. Activar Auto-Remediación y Verificar Historial
Para jobs que requieren rollback automático, añade la directiva `rollback_snapshot` en la misma regla:
```yaml
rollback_snapshot: "/var/backups/db-snapshots/"
```
Luego, revisa el historial de ejecuciones en la CLI:
```bash
cron-guard-cli history --job=backup-db --since=24h
```
Esto muestra timestamps, duración real, código de salida y si se ejecutó rollback. Ideal para auditorías.

## Mejores Prácticas Recomendadas

### 1. Segmentación de Jobs por Prioridad en Entornos de Producción
No monitorees todos los jobs con la misma configuración. Clasifica los jobs en **críticos** (afectan datos de usuario), **importantes** (procesos internos) y **baja prioridad** (logs rotativos). Para los críticos, activa alertas por SMS y rollback automático; para los de baja prioridad, solo logs. Esto evita ruido en las notificaciones y optimiza recursos del agente.

### 2. Integración con Sistemas de Logs Centralizados (ELK o Loki)
Cron Guard expone métricas en formato Prometheus (`/metrics` endpoint). Configura un scraper de Prometheus para recolectar datos como `cron_guard_job_duration_seconds` y `cron_guard_failures_total`. Luego, visualiza tendencias en Grafana. Esto permite detectar patrones de fallos estacionales (ej: picos de latencia los lunes por backups masivos) y ajustar las reglas de alerta dinámicamente.
