# cron-guard - CRO_006

## Resumen Técnico

Monitorización inteligente de tareas programadas con detección proactiva de fallos, latencia anómala y ejecuciones fantasma. Diseñado para entornos de producción con alta densidad de cron jobs.

## Funcionalidades Duras

### 1. Detección de "Ejecuciones Huérfanas"
Identifica procesos que quedan en estado zombie tras un fallo del scheduler o un kill forzado. Cron Guard analiza el PID real del proceso hijo y lo compara con el árbol de procesos del sistema, notificando si un job no libera recursos después de su timeout configurado.

### 2. Hash de Salida Diferencial (Diff-Hash)
Cada ejecución genera un hash SHA-256 de la salida estándar y de error. Si dos ejecuciones consecutivas del mismo job producen hashes idénticos, se considera una ejecución "silenciosa" (sin cambios reales). Esto permite identificar jobs que se ejecutan pero no modifican estado, ayudando a optimizar frecuencias.

### 3. Ventana de Tolerancia Adaptativa (ATW)
En lugar de usar timeouts fijos, Cron Guard aprende el patrón de duración histórica de cada tarea. Si una ejecución supera en más de 3 desviaciones estándar su media histórica, se dispara una alerta de latencia crítica. Esto evita falsos positivos en jobs con picos naturales (ej: backups nocturnos).

## Pasos de Implementación

### 1. Configurar el Daemon de Escucha con Política de Retry Exponencial
```bash
# Instalar el agente en el nodo de monitoreo
curl -sSL https://cron-guard.io/install.sh | bash -s -- --daemon-mode

# Configurar el archivo /etc/cron-guard/config.toml
[cron_jobs."mi_backup"]
command = "/usr/local/bin/backup.sh"
schedule = "0 3 * * *"
timeout_seconds = 3600
retry_policy = "exponential_backoff"
max_retries = 3

# Verificar que el daemon está activo
systemctl status cron-guard-daemon
```

### 2. Integrar Notificaciones con Webhook Personalizado
```yaml
# En el archivo de configuración, sección [notifications]
[notifications.webhook]
url = "https://hooks.slack.com/services/T00/B00/xxxxx"
headers = { "X-Custom-Header" = "cron-guard" }
payload_template = """
{
  "text": "⚠️ *Cron Guard Alert*\\nJob: {{job_name}}\\nStatus: {{status}}\\nDuration: {{duration}}s\\nLog: {{log_url}}"
}
"""
events = ["failure", "timeout", "orphan_detected"]
```

### 3. Activar el Modo "Dry-Run" para Pruebas de Regresión
```bash
# Simular ejecución sin modificar el scheduler real
cron-guard dry-run --job-id "mi_backup" --mock-exit-code 1

# Analizar el impacto de cambiar la frecuencia
cron-guard simulate --job-id "mi_backup" --new-schedule "*/30 * * * *" --history-file /var/log/cron-guard/history.db

# Revisar el reporte de impacto
cat /tmp/cron-guard-simulation-report.html
```

## Mejores Prácticas para Entornos de Producción

### 1. Segmentar Jobs por Criticidad con Etiquetas (Tags)
Asigna etiquetas como `critical`, `batch`, `maintenance` a cada job. Configura canales de notificación distintos según la etiqueta:
- **critical**: Slack directo + SMS vía Twilio
- **batch**: Solo correo diario resumen
- **maintenance**: Sin notificaciones, solo log

Esto evita saturar a los equipos con alertas de tareas no críticas.

### 2. Implementar "Canary Deployments" para Cambios en Schedulers
Antes de modificar la frecuencia de un job en producción, despliega el cambio primero en un nodo canary (5% del tráfico). Cron Guard comparará las métricas de éxito/fallo entre el nodo canary y el resto. Si la tasa de fallo supera el 2% en el canary, se revierte automáticamente el cambio y se notifica al equipo de SRE.
