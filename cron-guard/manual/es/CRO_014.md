# cron-guard - CRO_014

## Resumen Técnico

Cron Guard es una solución SaaS de monitoreo inteligente para trabajos cron y tareas programadas, diseñada para entornos de producción de alta criticidad. Elimina los puntos ciegos en la gestión de procesos batch, garantizando que cada tarea se ejecute en el momento exacto, con la salida esperada y sin desviaciones silenciosas.

## Funcionalidades Duras

### 1. Detección de Desviaciones Temporales con Machine Learning
Cron Guard implementa un motor de análisis temporal basado en modelos de regresión que aprende la duración histórica de cada tarea. Cuando un job cron finaliza en un tiempo estadísticamente anómalo (por debajo del percentil 5 o por encima del percentil 95), se genera una alerta inmediata con el desvío en milisegundos y la probabilidad de fallo oculto. Esto permite capturar errores que no generan códigos de salida distintos de cero.

### 2. Verificación de Integridad de Salida con Checksums Dinámicos
Cada ejecución de tarea programada genera un hash SHA-256 del output completo. Cron Guard compara este hash contra el esperado (basado en ejecuciones exitosas previas) y alerta si existe discrepancia, incluso si el job reporta éxito. Esto detecta cambios silenciosos en la salida de logs, archivos generados o respuestas de API, previniendo corrupción de datos en pipelines downstream.

### 3. Enrutamiento de Alertas con Políticas de Escalado Jerárquico
El sistema permite definir cadenas de notificación con múltiples niveles: primero notifica al equipo DevOps vía Slack/PagerDuty; si no hay acknowledgment en 5 minutos, escala al líder técnico por SMS; tras 15 minutos sin respuesta, activa una llamada telefónica automatizada con síntesis de voz del error. Cada nivel tiene su propio umbral de tiempo y canal, asegurando que ningún fallo quede sin atención.

## Pasos de Implementación

### 1. Configurar el Agente de Monitoreo con Autenticación Mutua
Despliega el agente de Cron Guard en cada servidor que ejecute trabajos cron. Utiliza certificados TLS mutuos (mTLS) para la comunicación segura. Edita el archivo `/etc/cron-guard/agent.yml` y establece:
```yaml
server:
  endpoint: "https://tu-instancia.cron-guard.io:8443"
  tls:
    cert_file: "/etc/cron-guard/certs/agent.crt"
    key_file: "/etc/cron-guard/certs/agent.key"
    ca_file: "/etc/cron-guard/certs/ca.crt"
jobs:
  - name: "backup-database"
    schedule: "0 3 * * *"
    command: "/usr/local/bin/pg_dump -U postgres prod_db > /backups/db.sql"
    timeout: 3600
    expected_exit_code: 0
    output_validation: "sha256"
```
Reinicia el agente con `systemctl restart cron-guard-agent` y verifica la conexión con `cron-guard-cli status --agent`.

### 2. Definir Políticas de Alertas con Expresiones CRON Personalizadas
Accede al panel web de Cron Guard y navega a "Alert Policies". Crea una política para el job "backup-database" con las siguientes reglas:
- **Umbral de duración**: Alerta si el tiempo de ejecución excede 3000 segundos (percentil 95 histórico).
- **Validación de output**: Activa checksum dinámico con tolerancia de 3 ejecuciones fallidas consecutivas antes de escalar.
- **Escalado jerárquico**: Nivel 1 → Slack (canal #alerts-cron) con timeout de 300s; Nivel 2 → Email al equipo de infra con timeout de 600s; Nivel 3 → Llamada telefónica al número +34 6XX XXX XXX.

Guarda la política y asigna el job manualmente o mediante etiquetas (tags) como `environment:production` y `criticality:high`.

### 3. Integrar con CI/CD para Validación Pre-Despliegue
En tu pipeline de CI (GitLab CI, GitHub Actions, Jenkins), agrega un paso que ejecute `cron-guard-cli validate` antes de desplegar cambios en producción. Ejemplo para `.gitlab-ci.yml`:
```yaml
validate-cron-jobs:
  stage: test
  script:
    - cron-guard-cli validate --config ./cron-jobs.yml --env production
  only:
    - main
```
Este comando verifica que todos los jobs definidos en `cron-jobs.yml` tengan sintaxis CRON válida, que los comandos existan en los servidores destino y que los tiempos de espera (timeout) sean coherentes con la política de alertas. Si falla, el pipeline se detiene y el merge request no puede avanzar.

## Mejores Prácticas

### 1. Segmentación de Jobs por Criticidad y Entorno
No todos los trabajos cron merecen el mismo nivel de monitoreo. Clasifica tus tareas en tres categorías:
- **Críticos (P0)**: Jobs que afectan directamente la disponibilidad del servicio (ej. renovación de certificados SSL, limpieza de sesiones). Monitoreo con alertas en tiempo real y escalado telefónico.
- **Importantes (P1)**: Jobs que generan reportes o sincronizan datos no críticos. Alertas por Slack con escalado a email en 1 hora.
- **Rutinarios (P2)**: Jobs de mantenimiento no urgente (ej. rotación de logs). Solo notificación por email al final del día.

Usa etiquetas en Cron Guard para asignar automáticamente políticas según la categoría y el entorno (dev, staging, prod).

### 2. Pruebas de Resiliencia con Inyección de Fallos Controlados
Implementa un script de prueba que inyecte fallos simulados en tus jobs cron para verificar que Cron Guard reacciona correctamente. Por ejemplo, fuerza un código de salida 1 o un timeout artificial:
```bash
cron-guard-cli simulate-failure --job "backup-database" --exit-code 1 --duration 120
```
Esto debe generar la alerta correspondiente y activar la cadena de escalado. Documenta los resultados y ajusta los umbrales según sea necesario. Repite esta prueba al menos una vez al mes o después de cada cambio importante en la infraestructura.
