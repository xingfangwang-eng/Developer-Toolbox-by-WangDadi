# cron-guard - CRO_025

## Resumen

Cron Guard es una herramienta de monitoreo inteligente para trabajos programados (cron jobs) diseñada para entornos de producción de alto rendimiento. Elimina falsos positivos, detecta fallos silenciosos y optimiza la ejecución de tareas batch sin intervención manual. Ideal para equipos DevOps que gestionan decenas de miles de tareas diarias.

## Contenido Detallado

### ¿Qué es Cron Guard?

Cron Guard no es un simple monitor de logs. Es un orquestador proactivo que intercepta cada ejecución de cron, evalúa su estado en tiempo real y reacciona automáticamente ante anomalías. Gracias a su motor de reglas contextuales, distingue entre un retraso normal por carga del servidor y un fallo crítico que requiere escalado inmediato. Integra alertas multicanal (Slack, PagerDuty, Webhook) y ofrece dashboards con métricas de latencia, éxito y tendencias de ejecución.

### Características Principales

1. **Detección de Fallos Silenciosos con Heurística Adaptativa**  
   Cron Guard analiza la salida estándar (stdout) y el código de retorno de cada trabajo. Si un cron finaliza con código 0 pero no produce la salida esperada (por ejemplo, un backup vacío o una sincronización sin datos), el sistema lo marca como "fallo silencioso" y ejecuta una acción correctiva predefinida (reintento, notificación o rollback).

2. **Reinteligencia de Ejecución con Backoff Exponencial**  
   Cuando un cron falla, Cron Guard no reintenta inmediatamente. Implementa un algoritmo de backoff exponencial (1s, 2s, 4s, 8s...) hasta un máximo de 5 reintentos. Si el fallo persiste, desactiva el cron y envía una alerta crítica. Esto evita tormentas de reintentos que saturan el sistema.

3. **Validación de Dependencias entre Tareas**  
   Cron Guard permite definir grafos de dependencia entre crons. Por ejemplo, `backup_db` debe ejecutarse antes de `sync_to_s3`. Si la primera falla, la segunda se pone en cola automáticamente hasta que la primera se resuelva. Esto elimina errores en cascada y garantiza consistencia en pipelines de datos.

### Cómo Usar

1. **Configurar el Archivo de Reglas (`cron-guard.yml`)**  
   Crea un archivo YAML en la raíz de tu proyecto con la definición de cada cron. Especifica el comando, el schedule en cron expression, el timeout máximo (en segundos), y las acciones post-fallo. Ejemplo:  
   ```yaml
   crons:
     - name: "backup_db"
       command: "pg_dump -U user db > /backups/db.sql"
       schedule: "0 2 * * *"
       timeout: 300
       on_failure: "retry:3; alert:slack"
       dependencies: []
     - name: "sync_to_s3"
       command: "aws s3 sync /backups s3://my-bucket"
       schedule: "0 3 * * *"
       timeout: 600
       on_failure: "retry:2; alert:pagerduty"
       dependencies: ["backup_db"]
   ```

2. **Ejecutar el Daemon de Monitoreo**  
   Lanza Cron Guard como un servicio systemd o directamente en terminal:  
   ```bash
   cron-guard --config cron-guard.yml --daemon
   ```  
   El daemon escuchará en el puerto 8080 (por defecto) y expondrá un endpoint `/health` para checks de uptime. Cada vez que un cron se ejecute, Cron Guard intercepta su PID, monitorea su consumo de CPU/RAM y registra métricas en una base de datos SQLite local.

3. **Interpretar el Dashboard en Tiempo Real**  
   Accede a `http://localhost:8080/dashboard` para ver un panel con:  
   - **Estado actual**: verde (éxito), amarillo (retraso > 20% del timeout), rojo (fallo).  
   - **Historial de ejecuciones**: gráfico de líneas con latencia media y percentil 99.  
   - **Alertas activas**: lista de fallos no resueltos con timestamp y acción tomada.  
   - **Botón de reinicio manual**: permite forzar la ejecución de un cron sin esperar su schedule.

### Mejores Prácticas

1. **Implementar Pruebas de Humo en Entornos Staging**  
   Antes de desplegar un nuevo cron en producción, ejecútalo en un entorno staging con Cron Guard activado. Configura el modo `dry-run` para que el daemon solo registre métricas sin ejecutar acciones correctivas. Así validas que el comando, las dependencias y los timeouts son correctos sin riesgo de impacto en producción.

2. **Usar Etiquetas de Contexto para Alertas Diferenciadas**  
   En el archivo de reglas, añade una etiqueta `env: production` o `env: staging` a cada cron. Luego, configura reglas de alerta distintas: para producción, envía notificaciones a PagerDuty con prioridad alta; para staging, solo a Slack en un canal de logs. Esto evita ruido innecesario y asegura que los fallos críticos reciban atención inmediata.
