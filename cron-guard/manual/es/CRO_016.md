### ¿Qué es Cron Guard?

Cron Guard es una herramienta profesional de monitoreo y gestión de tareas programadas (cron jobs) que resuelve problemas críticos en entornos de producción: fallos silenciosos, desincronización de horarios y sobrecarga de recursos. A diferencia de soluciones básicas, ofrece visibilidad en tiempo real y alertas proactivas.

### Características Principales (Hardcore)

1. **Validación de Dependencias en Tiempo Real**  
   Cron Guard ejecuta un chequeo de estado de cada tarea antes de lanzarla, verificando que las dependencias (servicios, archivos, bases de datos) estén disponibles. Si una dependencia falla, la tarea se pausa automáticamente y se registra un log estructurado en formato JSON con timestamp y contexto.

2. **Reinteligente con Backoff Exponencial**  
   Cuando un cron job falla, Cron Guard no se limita a reintentar de forma lineal. Implementa un algoritmo de backoff exponencial (1s, 2s, 4s, 8s...) hasta un máximo de 5 reintentos, con un límite de tiempo configurable. Esto evita saturar el sistema y permite recuperar tareas críticas sin intervención manual.

3. **Dashboard de Latencia y Throughput**  
   Proporciona un panel en vivo con métricas de latencia (percentiles P50, P95, P99) y throughput (tareas completadas por minuto) para cada cron job. Los datos se almacenan en una base de datos de series temporales (InfluxDB) y se visualizan con gráficos interactivos, permitiendo identificar cuellos de botella en milisegundos.

### Cómo Usar

1. **Configuración del Archivo de Reglas**  
   Crea un archivo `cron-guard.yml` en la raíz de tu proyecto. Define cada tarea con su expresión cron, comando a ejecutar y dependencias. Ejemplo:  
   ```yaml
   tasks:
     - name: backup-database
       schedule: "0 3 * * *"
       command: "pg_dump -U admin mydb > /backups/db.sql"
       dependencies:
         - service: postgres
           port: 5432
   ```  
   Luego, ejecuta `cron-guard validate` para verificar la sintaxis y las dependencias sin lanzar las tareas.

2. **Monitoreo en Segundo Plano**  
   Inicia Cron Guard como un daemon: `cron-guard start --config cron-guard.yml`. Esto lanza un proceso en segundo plano que ejecuta las tareas según su programación. Para ver logs en tiempo real, usa `cron-guard logs --follow`. Cada tarea registra su salida (stdout y stderr) en archivos separados bajo `/var/log/cron-guard/`.

3. **Alertas y Notificaciones**  
   Configura alertas en el mismo archivo YAML para recibir notificaciones vía webhook o correo electrónico cuando una tarea falle más de 2 veces seguidas. Ejemplo:  
   ```yaml
   alerts:
     - type: slack
       webhook: "https://hooks.slack.com/services/T.../B.../xxx"
       threshold: 2
   ```  
   Luego, prueba la alerta con `cron-guard test-alert --task backup-database`.

### Mejores Prácticas para el Ecosistema

1. **Segmentación por Entornos**  
   No mezcles tareas de desarrollo, staging y producción en el mismo archivo de configuración. Usa archivos separados (ej. `cron-guard-prod.yml`) y variables de entorno para rutas y credenciales. Esto evita que un error en staging afecte a producción.

2. **Monitoreo de Tareas Críticas con Umbrales Dinámicos**  
   Para tareas que procesan datos variables (como limpieza de logs), define umbrales de tiempo basados en el historial. Por ejemplo, si una tarea normalmente tarda 30 segundos, pero un día tarda 5 minutos, Cron Guard puede disparar una alerta de anomalía. Configúralo con:  
   ```yaml
   thresholds:
     latency: 2x # alerta si la latencia supera el doble del promedio histórico
   ```
