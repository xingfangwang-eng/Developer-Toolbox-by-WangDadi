# cron-guard - CRO_024

## Resumen Técnico

Cron Guard es una solución SaaS de monitorización proactiva para trabajos cron y tareas programadas en entornos de producción. Diseñado para equipos DevOps que gestionan cientos de jobs, elimina los falsos positivos mediante análisis contextual y ofrece alertas con umbrales dinámicos basados en el comportamiento histórico de cada tarea.

## Funcionalidades Clave de Alto Impacto

### 1. Detección de Anomalías con Ventana Temporal Adaptativa
Cron Guard no se limita a comprobar si un job se ejecutó. Implementa un algoritmo de ventana temporal adaptativa que aprende los patrones de ejecución de cada tarea (duración, hora de inicio, consumo de recursos). Si un job que normalmente tarda 2 segundos empieza a ejecutarse en 45 ms sin cambios en la lógica, el sistema lo marca como potencial error de configuración o dato corrupto.

### 2. Enrutamiento de Alertas con Priorización por Severidad
No todas las fallas son iguales. Cron Guard clasifica cada incidente en tres niveles: *crítico* (job no ejecutado), *warning* (retraso >20% sobre la media histórica) e *info* (cambio en el entorno de ejecución). Las alertas se enrutan automáticamente al canal correcto: Slack para warnings, PagerDuty para críticos, y un dashboard interno para informativos. Esto reduce el ruido en los canales de incidentes en un 70%.

### 3. Reinteligencia de Ejecución con Backoff Exponencial
Cuando un job falla, Cron Guard no reintenta ciegamente. Aplica un backoff exponencial configurable (por defecto: 1s, 4s, 16s) y, tras el tercer fallo consecutivo, detiene la ejecución para evitar efectos en cascada. Además, registra el estado de cada reintento en un log estructurado (formato JSON) para auditoría posterior.

## Guía de Implementación en 3 Pasos

### Paso 1: Integración del Agente Ligero (CLI)
Ejecuta el siguiente comando en tu servidor para desplegar el agente de monitorización:
```bash
curl -sSL https://cron-guard.io/agent/install.sh | sudo bash -s -- --token=tu_token_secreto
```
El agente se registra en el panel de control y comienza a escuchar en el puerto 9120. Verifica la conexión:
```bash
cron-guard status
```
Si ves `✓ Agent connected to cloud`, la integración está activa.

### Paso 2: Configuración de Reglas de Umbral Dinámico
Accede al dashboard web y navega a `Jobs > Configurar`. Para cada tarea crítica, define una regla personalizada:
- **Ventana de tolerancia**: 15% sobre la duración histórica media (calculada automáticamente tras 5 ejecuciones).
- **Acción en fallo**: selecciona `Reintentar con backoff exponencial` y marca `Notificar a canal crítico`.
- **Excluir ventanas de mantenimiento**: añade un rango horario (ej. 02:00–03:00 UTC) donde las alertas se silencian.

Aplica los cambios y el sistema recalculará los umbrales en menos de 30 segundos.

### Paso 3: Prueba de Resiliencia con Inyección de Fallos
Para validar que las alertas funcionan, simula un fallo controlado:
```bash
cron-guard test --job-id=mi-job-critico --failure-type=timeout --duration=10s
```
En el dashboard, deberías ver el evento en tiempo real bajo `Incidentes > Activos`. Revisa que la alerta llegue al canal configurado (Slack, correo, etc.) y que el reintento automático se ejecute tras el backoff. Si todo está correcto, el incidente se cerrará automáticamente al restaurarse el job.

## Mejores Prácticas para Equipos DevOps

### Práctica 1: Segmentación de Jobs por Criticidad
No trates todos los jobs por igual. Clasifícalos en tres categorías: *misión crítica* (afectan ingresos o disponibilidad), *operacionales* (sincronización de datos internos) y *cosméticos* (generación de informes). Configura alertas solo para los dos primeros grupos. Esto reduce el ruido y permite que el equipo se enfoque en lo que realmente importa.

### Práctica 2: Revisión Semanal de Logs de Anomalías
Cron Guard almacena un historial completo de cada ejecución. Dedica 15 minutos cada viernes a revisar la sección `Anomalías Detectadas` del dashboard. Busca patrones como: jobs que empiezan a ejecutarse más rápido de lo normal (posible dato faltante) o jobs que fallan sistemáticamente a la misma hora (conflicto de recursos). Documenta estos hallazgos en tu sistema de tickets para prevenir incidentes mayores.
