# cron-guard - CRO_017

## Resumen Técnico

Cron Guard es una herramienta de monitorización y orquestación de tareas programadas diseñada para entornos de producción de alta disponibilidad. Elimina la incertidumbre en la ejecución de cron jobs mediante detección temprana de fallos, reinteligentes automáticos y alertas contextuales en tiempo real. Ideal para equipos DevOps que gestionan pipelines complejos con dependencias entre tareas.

## Funcionalidades Clave

### 1. **Detección de Desviaciones en Tiempo Real con Baseline Adaptativo**
Cron Guard aprende automáticamente los patrones de ejecución de cada tarea (duración, consumo de recursos, frecuencia de éxito/fallo) y genera un *baseline* dinámico. Cuando una ejecución se desvía más de 2 desviaciones estándar del comportamiento histórico, se dispara una alerta con el delta exacto. Esto permite detectar problemas incipientes antes de que se conviertan en caídas completas.

### 2. **Reinteligente Jerárquico con Backoff Exponencial**
No todos los fallos son iguales. Cron Guard implementa un sistema de reintentos inteligente que distingue entre errores transitorios (timeouts de red, picos de carga) y errores permanentes (código roto, dependencias caídas). Para los primeros, aplica un backoff exponencial con jitter (hasta 3 reintentos en intervalos de 1s, 4s, 16s). Para los segundos, aborta inmediatamente y escala la alerta al canal de PagerDuty o Slack correspondiente.

### 3. **SLA Tracking con Garantía de Ejecución**
Cada tarea programada puede asociarse a un SLA (por ejemplo, "debe ejecutarse al menos una vez cada 5 minutos con éxito"). Cron Guard monitorea el cumplimiento en ventanas deslizantes de 1 hora, 24 horas y 7 días. Si se incumple el SLA, se activa un flujo de remediación automático: reinicio del contenedor, reencolamiento de la tarea o notificación al responsable del servicio.

## Guía de Uso Avanzada

### 1. **Configuración de Monitoreo con Alertas Contextuales**
```bash
cron-guard monitor --task-id "backup-db" \
  --alert-channel slack \
  --alert-webhook "https://hooks.slack.com/services/T.../B.../..." \
  --baseline-window 72h \
  --alert-threshold 2.5
```
Este comando configura el monitoreo de la tarea `backup-db` con un baseline basado en las últimas 72 horas de ejecución. Las alertas se enviarán a Slack solo cuando la duración de la ejecución supere 2.5 desviaciones estándar del promedio histórico. El parámetro `--alert-context` incluye en el mensaje de alerta el último log de error, el stack trace y el estado de las dependencias.

### 2. **Definición de Dependencias entre Tareas**
```yaml
# cron-guard-deps.yaml
tasks:
  - id: "etl-process"
    depends_on:
      - id: "data-ingestion"
        wait_for: "success"
        timeout: 300s
      - id: "schema-validation"
        wait_for: "completion"
        timeout: 60s
    retry_policy:
      max_attempts: 3
      backoff: "exponential"
      initial_interval: 2s
```
Cron Guard interpreta este archivo YAML para construir un DAG (Directed Acyclic Graph) de ejecución. Si `data-ingestion` falla, `etl-process` no se ejecuta y se registra una alerta de dependencia bloqueada. El parámetro `wait_for: "completion"` permite que `etl-process` arranque incluso si `schema-validation` termina con error (útil para tareas de logging o monitoreo).

### 3. **Recuperación Automática con Rollback**
```bash
cron-guard recover --task-id "deploy-app" \
  --rollback-strategy "previous-version" \
  --rollback-trigger "failure-rate > 10%" \
  --health-check-endpoint "https://api.example.com/health"
```
Este comando configura una tarea de despliegue con recuperación automática. Si después de una ejecución el health check endpoint devuelve un código distinto de 200 durante más de 30 segundos, o si la tasa de fallos en las siguientes 5 ejecuciones supera el 10%, Cron Guard ejecuta automáticamente un rollback a la versión anterior. El historial de versiones se mantiene en un almacenamiento externo (S3, GCS, etc.) configurado en el archivo `cron-guard.yaml`.

## Mejores Prácticas Recomendadas

### 1. **Segmentación de Tareas por Prioridad y SLA**
No todas las tareas merecen el mismo nivel de monitoreo. Clasifica tus cron jobs en tres categorías: **Críticas** (SLA < 5 minutos, alertas inmediatas a PagerDuty), **Importantes** (SLA < 1 hora, alertas a Slack con tolerancia de 2 reintentos) y **Rutinarias** (SLA < 24 horas, logs centralizados sin alertas). Cron Guard permite asignar etiquetas (`--tag critical`, `--tag routine`) y configurar políticas por etiqueta en un solo archivo de configuración.

### 2. **Pruebas de Resiliencia con Modo Simulación**
Antes de desplegar cambios en producción, utiliza el modo `--dry-run` de Cron Guard para validar dependencias, reintentos y alertas sin ejecutar realmente las tareas. Ejemplo:
```bash
cron-guard validate --config cron-guard-deps.yaml --dry-run --report output.json
```
Este modo simula 100 ejecuciones con fallos aleatorios (configurables con `--failure-rate 0.2`) y genera un reporte JSON con métricas de cobertura de reintentos, tiempos de resolución y caminos críticos. Así puedes identificar cuellos de botella y ajustar políticas antes de que afecten a usuarios reales.
