## What is Cron Guard?

Cron Guard is not just another cron job monitor. It’s a lightweight, self-hostable agent that tracks execution timestamps, exit codes, and runtime anomalies for every scheduled task in your infrastructure. Whether you’re running cron jobs on bare metal, Kubernetes, or serverless environments, Cron Guard provides real-time visibility into job health—without adding latency or requiring a database.

---

## Core Features

### 1. **Execution Drift Detection**
Cron Guard automatically flags jobs that deviate from their expected execution window. If a job scheduled to run every 5 minutes starts at +2 seconds or +10 seconds, it logs the drift and triggers an alert. This is critical for detecting resource contention or upstream service degradation before it cascades.

### 2. **Zero-Dependency Heartbeat Protocol**
Instead of relying on external databases or message queues, Cron Guard uses a lightweight HTTP heartbeat mechanism. Each job sends a simple `GET` request to a configurable endpoint with a unique job ID and timestamp. The agent validates the heartbeat against a local state file—no Redis, no PostgreSQL, no overhead.

### 3. **Graceful Degradation with Slack & PagerDuty Fallback**
When a job fails or misses its window, Cron Guard immediately fires alerts via Slack, email, or PagerDuty. If the primary notification channel is down, it automatically retries with a secondary channel after a configurable timeout (default: 30 seconds). This ensures you never miss a failed job, even during partial outages.

## How to Use

### Step 1: Deploy the Cron Guard Agent
Download the latest binary from the [GitHub Releases page](https://github.com/xingfangwang-eng/cron-guard/releases) and run it as a systemd service or a Docker container. The agent requires no external dependencies—just a writable directory for the state file.

```bash
# Example: Run as a Docker container
docker run -d \
  --name cron-guard \
  -v /var/lib/cron-guard:/data \
  -e CRON_GUARD_PORT=8080 \
  xingfangwang/cron-guard:latest
```

### Step 2: Configure Job Monitoring Rules
Create a YAML configuration file that defines each job’s schedule, tolerance window, and notification channels. Below is a production-ready example for a daily database backup job:

```yaml
jobs:
  - name: "daily-db-backup"
    schedule: "0 2 * * *"            # Every day at 2:00 AM
    tolerance: 30                     # Allow up to 30 seconds drift
    timeout: 300                      # Job must complete within 5 minutes
    notify:
      - type: slack
        webhook: "https://hooks.slack.com/services/T00/B00/xxxx"
      - type: pagerduty
        routing_key: "your-pagerduty-key"
```

### Step 3: Integrate Heartbeat into Your Cron Jobs
Modify each cron job to send a heartbeat before and after execution. Use the provided CLI tool, `cron-guard-heartbeat`, which ships with the agent:

```bash
# In your crontab:
0 2 * * * /usr/local/bin/cron-guard-heartbeat start daily-db-backup && \
  /usr/local/bin/backup-script.sh && \
  /usr/local/bin/cron-guard-heartbeat end daily-db-backup
```

If the job fails or exceeds the timeout, Cron Guard automatically triggers the configured alerts.

## Best Practices

### 1. **Always Set a Tolerance Window for Non-Critical Jobs**
For jobs that don’t need millisecond precision (e.g., daily reports, cache refreshes), set a tolerance of 60–120 seconds. This reduces false positives during network hiccups or cloud provider maintenance windows.

### 2. **Use Multiple Notification Channels with Different Severity Levels**
Configure Slack for warning-level alerts (e.g., drift > 10 seconds) and PagerDuty for critical alerts (e.g., job timeout or missing heartbeat). This prevents alert fatigue while ensuring urgent issues are escalated immediately.
