# cron-guard - CRO_021

## Overview

Cron Guard is a production-grade cron job scheduler and execution monitor designed for distributed systems running on Linux kernels ≥ 4.15. It provides sub-millisecond precision scheduling, real-time execution tracing via eBPF, and automatic failure recovery with configurable backoff strategies. The tool integrates natively with systemd timers and Kubernetes CronJobs, offering a unified observability layer across hybrid infrastructure.

## Core Features

### 1. **Deterministic Scheduling with Nanosecond Drift Compensation**
- **Parameter**: `--drift-threshold=<nanoseconds>` (default: 500,000 ns)
- Implements a hybrid clock synchronization algorithm combining CLOCK_MONOTONIC_RAW and NTP-derived offsets.
- Automatically recalculates next-run timestamps after each execution to prevent cumulative drift beyond 1 ms over 30 days.
- Exposes `/metrics` endpoint with `cron_guard_schedule_drift_ns` gauge for Prometheus scraping.

### 2. **Execution Provenance via eBPF Tracing**
- **Parameter**: `--ebpf-buffer-size=<bytes>` (default: 8,192 bytes per CPU)
- Attaches kprobes to `do_exit` and `schedule` syscalls to capture exact process start/stop times, CPU affinity, and OOM killer interactions.
- Generates a structured JSON log per execution with fields: `pid`, `exit_code`, `wall_time_ms`, `cpu_time_ms`, `memory_peak_kb`.
- Supports PostgreSQL ingestion via COPY protocol (`--pg-conn-string`), enabling historical querying like "find all jobs that exceeded 95th percentile wall time in the last 7 days".

### 3. **Adaptive Retry with Exponential Backoff and Circuit Breaker**
- **Parameter**: `--max-retries=<int>` (default: 3), `--circuit-breaker-failure-count=<int>` (default: 5 within 60s window)
- Retry intervals follow the formula: `base_delay * 2^(attempt-1) + jitter(0, base_delay)`, where `base_delay` defaults to 1,000 ms.
- After the circuit breaker trips, all subsequent executions are held in a pending queue until the cooldown period (`--cb-cooldown-sec=<int>`, default: 120) expires.
- Emits structured error events to a configurable Webhook endpoint (`--webhook-url`) with full execution context including stdout/stderr truncated to 4 KB each.

## How to Use

### Step 1: Define a Job with Execution Constraints
```bash
cron-guard job create \
  --name="db-backup" \
  --schedule="0 3 * * *" \
  --command="/usr/local/bin/pg_dump -U admin -Fc mydb > /backups/db_$(date +\%Y\%m\%d).dump" \
  --timeout=3600 \
  --drift-threshold=200000 \
  --max-retries=2 \
  --circuit-breaker-failure-count=3
```
This creates a job that runs daily at 3 AM, with a 1-hour hard timeout, nanosecond drift compensation set to 200 µs, and a circuit breaker that trips after 3 consecutive failures within 60 seconds.

### Step 2: Attach a PostgreSQL Sink for Execution History
```bash
cron-guard config set \
  --pg-conn-string="postgres://user:pass@localhost:5432/cron_guard?sslmode=require" \
  --pg-schema="execution_logs" \
  --pg-batch-size=500 \
  --pg-flush-interval=10
```
All execution records are buffered in memory and flushed every 10 seconds or after 500 events, whichever comes first. The schema is auto-created with a composite primary key on `(job_name, execution_time)` and a BRIN index on `execution_time` for efficient range scans.

### Step 3: Start the Daemon and Verify Metrics Endpoint
```bash
cron-guard daemon start \
  --listen-addr="0.0.0.0:9090" \
  --ebpf-buffer-size=16384 \
  --log-level=info \
  --log-format=json
```
Verify with:
```bash
curl -s http://localhost:9090/metrics | grep cron_guard
```
Expected output includes `cron_guard_schedule_drift_ns`, `cron_guard_executions_total`, `cron_guard_retries_total`, and `cron_guard_circuit_breaker_trips_total`. Use `cron-guard status --job=db-backup` to inspect the last 10 executions with full provenance.

## Best Practices

### 1. **Use eBPF Tracing for Capacity Planning**
- Enable eBPF tracing on all critical jobs and pipe the JSON logs into a time-series database (e.g., TimescaleDB). Create a continuous aggregate that computes P50, P95, and P99 wall times per job per hour. When P95 exceeds 80% of the configured timeout, trigger an alert. This proactive approach prevents silent SLA violations in high-throughput environments.

### 2. **Implement Circuit Breaker Tiers for Dependent Job Chains**
- For job DAGs (e.g., ETL pipeline: `extract → transform → load`), set `--circuit-breaker-failure-count=2` on upstream jobs and `--circuit-breaker-failure-count=4` on downstream jobs. This ensures that a transient failure in extraction doesn't cascade into unnecessary retries in transformation. Combine with `--webhook-url` pointing to a PagerDuty endpoint for immediate incident response.
