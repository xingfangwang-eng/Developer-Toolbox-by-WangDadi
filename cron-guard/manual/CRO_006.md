# cron-guard - CRO_006

## Overview

Cron Guard is a production-grade cron job monitoring and alerting toolkit designed for DevOps engineers, SREs, and backend developers who need bulletproof scheduled task observability. It eliminates silent failures, drift in execution windows, and log blindness — turning cron into a first-class citizen of your monitoring stack.


---

## What Makes Cron Guard Different?

Unlike basic health checks or generic uptime monitors, Cron Guard provides deep introspection into every cron execution lifecycle. It’s built for high-frequency, distributed, and latency-sensitive environments.

### 3 Hardcore Features

1. **Execution Drift Detection & Slack Correction**  
   Cron Guard doesn’t just track if a job ran — it measures *when* it ran relative to its schedule. If a job consistently fires 12 seconds late over 10 consecutive runs, it triggers a drift alert and optionally auto-adjusts the crontab entry via a fallback scheduler hook.

2. **Log-Loss Prevention with Checkpoint Recovery**  
   Each cron execution writes a checkpoint to a local or remote store (SQLite, Redis, or S3). If the job crashes mid-execution or the log stream drops, Cron Guard replays the last known checkpoint and resumes from the exact failure point — no duplicate runs, no missed intervals.

3. **Multi-Node Consensus for Distributed Cron**  
   In clustered deployments, Cron Guard uses a lightweight Raft-based consensus layer to ensure a cron job runs exactly once across all nodes. It prevents double execution in active-active setups and handles leader election transparently without external dependencies like ZooKeeper.

## How to Use Cron Guard

### Step 1: Instrument Your Existing Crontab

Wrap any cron command with the `cron-guard` binary. This injects execution metadata (PID, start timestamp, expected schedule) into a local journal file.

```bash
# Before
*/5 * * * * /usr/local/bin/backup.sh

# After
*/5 * * * * /usr/bin/cron-guard run --job-id backup_job_01 --cmd "/usr/local/bin/backup.sh"
```

Cron Guard will capture stdout/stderr, compute execution latency, and write a structured log to `/var/log/cron-guard/backup_job_01.ndjson`.

### Step 2: Configure Alerting Thresholds

Create a YAML config file to define what constitutes a failure for each job. This goes beyond exit codes — you can alert on runtime variance, log volume spikes, or missing heartbeats.

```yaml
# /etc/cron-guard/jobs.d/backup_job_01.yaml
alerts:
  - type: exit_code
    value: 0
  - type: runtime_variance
    max_stddev_ms: 2000
  - type: log_volume_drop
    min_bytes: 1024
    window: 5m
notifications:
  - channel: slack
    webhook: https://hooks.slack.com/services/xxx
```

Apply the config and restart the daemon. Cron Guard will now evaluate every run against these rules.

### Step 3: Deploy the Health Dashboard

Run the embedded HTTP server to expose real-time job status, execution history, and failure summaries. This is useful for on-call engineers who need a quick glance at cron health.

```bash
cron-guard serve --port 9090 --config /etc/cron-guard/global.yaml
```

The dashboard provides:
- A timeline of recent runs per job
- Heatmap of execution latency over the last 24 hours
- One-click retry for failed jobs (via API endpoint)

## Best Practices for Production Cron

### 1. Use Idempotent Job Wrappers

Always wrap your cron jobs with an idempotency layer. Cron Guard can detect re-runs, but it cannot undo side effects. For example, if a batch import job fails mid-way, the next run should skip already-processed records. Use a database transaction log or a dedup key to protect against partial writes.

### 2. Set a Graceful Shutdown Timeout

Cron jobs often get killed by the OS or orchestration layer when they exceed their expected runtime. Configure a `shutdown_timeout` in your Cron Guard config to send a SIGTERM first, then SIGKILL after a grace period. This allows the job to flush buffers, close connections, and write a final checkpoint — preventing data corruption and false negatives in monitoring.
