# cron-guard - CRO_016

## Overview

Cron Guard is a distributed job orchestration and execution monitoring engine designed for mission-critical scheduled task pipelines. It provides millisecond-precision execution tracking, adaptive retry with exponential backoff, and real-time failure escalation across heterogeneous environments. Keywords: cron job, scheduled tasks, job monitoring, distributed scheduler, execution audit.

## Core Features

### 1. Predictive Execution Skew Detection with Temporal Drift Correction

Cron Guard continuously analyzes the delta between scheduled trigger time and actual execution start across all registered workers. When a deviation exceeds a configurable threshold (default: `drift_threshold_ms=500`), the system automatically recalculates the next invocation window using a weighted moving average of historical drift. This ensures that jobs dependent on strict timing—such as financial market data ingestion or CDN cache invalidation—remain synchronized within ±50 ms of the intended schedule.

- **Parameters:** `drift_threshold_ms`, `history_window_count=20`, `correction_mode=linear|adaptive`
- **Storage:** Drift metrics persisted to PostgreSQL via `cron_guard.execution_skew` table, indexed by `job_id` and `worker_id`

### 2. Multi-Phase Execution with Stateful Checkpointing

Each job execution is split into configurable phases (e.g., `fetch`, `transform`, `notify`). Cron Guard writes a checkpoint to its embedded key-value store (backed by RocksDB) at the start and completion of each phase. If a worker crashes mid-execution, the scheduler reassigns the job to a healthy worker and resumes from the last successful checkpoint—not from scratch. This reduces redundant work by up to 78% in long-running batch pipelines.

- **Parameters:** `checkpoint_interval_ms=200`, `max_retries_per_phase=3`, `phase_timeout_s=300`
- **Integration:** Native gRPC hooks for custom checkpoint logic; fallback to HTTP callback if gRPC unavailable

### 3. Anomaly-Driven Auto-Escalation with Runbook Injection

Cron Guard maintains a rule engine that evaluates execution outcomes against historical baselines. When a job fails more than `failure_rate_threshold=0.15` over a sliding window of `window_size_hours=24`, the system automatically creates a PagerDuty incident or posts a Slack alert with a pre-generated runbook link. The runbook includes the exact error stack, the last three successful execution logs, and a diff of environment variables since the last success.

- **Parameters:** `failure_rate_threshold`, `window_size_hours`, `escalation_channels=[pagerduty, slack, email]`
- **Storage:** All anomaly events logged to `cron_guard.anomaly_events` with `event_type`, `severity`, and `resolved_at` timestamps

## How to Use

### Step 1: Deploy the Cron Guard Agent on Each Worker Node

```bash
cron-guard agent install --worker-id=worker-01 \
  --coordinator-endpoint=https://coordinator.cron-guard.io:9443 \
  --checkpoint-dir=/var/lib/cron-guard/checkpoints \
  --log-level=info
```

The agent registers itself with the coordinator, opens a bidirectional gRPC stream for heartbeat and checkpoint sync, and begins listening for job assignments. Verify registration with:

```bash
cron-guard agent status --worker-id=worker-01
```

Expected output: `Status: connected | Drift: +12 ms | Checkpoints: 0`

### Step 2: Define a Job with Multi-Phase Checkpointing

Create a YAML job definition (`my_etl_job.yaml`):

```yaml
job:
  id: etl-user-import
  schedule: "*/5 * * * *"
  phases:
    - name: fetch_raw
      command: "python3 /opt/scripts/fetch_users.py"
      timeout_s: 120
    - name: transform
      command: "python3 /opt/scripts/transform_users.py"
      timeout_s: 300
    - name: load
      command: "python3 /opt/scripts/load_users.py"
      timeout_s: 180
  retry:
    max_retries: 3
    backoff: exponential
    backoff_base_ms: 1000
  escalation:
    failure_rate_threshold: 0.15
    window_size_hours: 24
    channels: ["slack"]
```

Submit the job:

```bash
cron-guard job submit --file=my_etl_job.yaml --coordinator-endpoint=...
```

### Step 3: Monitor Execution and Inspect Checkpoint Recovery

Tail the execution log for a specific job run:

```bash
cron-guard logs follow --job-id=etl-user-import --run-id=run-20250321-1445
```

If a worker crashes during the `transform` phase, you will see output like:

```
[WARN] Worker worker-01 disconnected at phase 'transform' (checkpoint: phase_start)
[INFO] Reassigning job etl-user-import to worker-03
[INFO] Resuming from checkpoint 'transform' (phase_start) — skipping 'fetch_raw'
```

To inspect the checkpoint store directly:

```bash
cron-guard checkpoint inspect --job-id=etl-user-import --run-id=run-20250321-1445
```

## Best Practices

### 1. Align Checkpoint Granularity with Job Criticality

For jobs where re-execution cost is high (e.g., database migrations, ML model training), set `checkpoint_interval_ms` to 50–100 ms and define at least three phases. For lightweight jobs (e.g., health pings, cache clears), reduce checkpoint frequency to 500 ms or disable checkpointing entirely with `checkpoint_enabled=false`. Over-checkpointing on trivial jobs introduces unnecessary I/O overhead and can degrade scheduler throughput by up to 12%.

### 2. Use Separate Worker Pools for Drift-Sensitive and Drift-Tolerant Jobs

Create two worker groups: one with `drift_threshold_ms=100` for time-critical jobs (e.g., financial trades, live stream encoding), and another with `drift_threshold_ms=2000` for batch analytics. Assign jobs to pools via the `worker_pool` label in the job definition. This prevents drift-sensitive jobs from being starved by long-running batch processes and keeps your P99 execution skew under 80 ms for critical pipelines.
