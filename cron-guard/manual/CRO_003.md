## What Is Cron Guard?

Cron Guard is a lightweight, language‑agnostic supervisor that wraps your existing cron jobs with **idempotency enforcement**, **dead‑man‑switch alerting**, and **execution‑time budgeting**. Unlike traditional cron wrappers that only log failures, Cron Guard actively prevents overlapping runs, detects silent misses, and provides a real‑time dashboard without requiring any changes to your cron daemon or application code.

It runs as a single binary (written in Rust, ~5 MB) and integrates with any Unix‑based scheduler — Vixie cron, systemd timers, or even Kubernetes CronJobs via a sidecar pattern.

## 3 Hardcore Features

### 1. **Lock‑Free Idempotency with Distributed Lease Tokens**

Cron Guard uses a **lease‑based lock** backed by either Redis or an embedded SQLite store. Instead of traditional file‑based locks (which fail under NFS or container restarts), it issues a time‑bound lease token for each job instance. If a job crashes mid‑execution, the lease expires automatically, allowing the next scheduled run to proceed without manual cleanup. The lease duration is configurable per job, and you can set a **stale‑lock threshold** that triggers an alert if a lock is held beyond its budget.

### 2. **Predictive Execution‑Time Budgeting with Adaptive Alerts**

Cron Guard learns the runtime distribution of each job over the last 100 runs and computes a **dynamic timeout** using the 95th percentile plus a configurable buffer (default: 20%). If a job exceeds this budget, Cron Guard sends a **graded alert** — first a warning, then a critical alert if the job exceeds the absolute hard limit. This eliminates false positives from occasional latency spikes while catching genuine regressions.

### 3. **Dead‑Man‑Switch with Exponential Backoff Escalation**

Every cron job monitored by Cron Guard must emit a **heartbeat signal** (a simple HTTP POST or a file touch) within a configurable grace period. If the heartbeat is missed, Cron Guard escalates the alert through a chain:  
- **Tier 1:** Slack / PagerDuty notification (within 1 minute)  
- **Tier 2:** SMS / phone call (after 5 minutes of silence)  
- **Tier 3:** Automatic execution of a **recovery playbook** (e.g., restart the job, run a fallback script, or trigger a rollback)  

This is critical for jobs that run once daily and must not be missed.

## 3 Detailed Operational Steps

### Step 1: Deploy Cron Guard as a Systemd Service

```bash
# Download the binary (example for Linux amd64)
curl -L -o /usr/local/bin/cron-guard \
  https://github.com/xingfangwang-eng/cron-guard/releases/latest/download/cron-guard-linux-amd64

chmod +x /usr/local/bin/cron-guard

# Create configuration directory
mkdir -p /etc/cron-guard

# Write the YAML config
cat > /etc/cron-guard/config.yaml << 'EOF'
backend:
  type: redis
  uri: "redis://localhost:6379/0"

jobs:
  - name: daily-report
    command: "/usr/local/bin/generate-report.sh"
    schedule: "0 2 * * *"
    timeout: 300
    heartbeat_grace: 60
    escalation:
      - type: slack
        webhook: "https://hooks.slack.com/services/T00/B00/xxx"
      - type: pagerduty
        routing_key: "your-pagerduty-key"

alerts:
  default_channel: slack
  escalation_interval: 300
EOF

# Create systemd service
cat > /etc/systemd/system/cron-guard.service << 'EOF'
[Unit]
Description=Cron Guard – scheduled job supervisor
After=network-online.target redis.service

[Service]
ExecStart=/usr/local/bin/cron-guard --config /etc/cron-guard/config.yaml
Restart=always
RestartSec=10
User=cron-guard
Group=cron-guard

[Install]
WantedBy=multi-user.target
EOF

systemctl daemon-reload
systemctl enable --now cron-guard
```

### Step 2: Register Your Existing Cron Jobs

Instead of modifying your crontab, you wrap each job with the Cron Guard CLI:

```bash
# Original crontab entry:
# 0 2 * * * /usr/local/bin/generate-report.sh

# Replace with:
0 2 * * * /usr/local/bin/cron-guard run --job daily-report -- /usr/local/bin/generate-report.sh
```

Cron Guard will:  
- Acquire a lease token before execution  
- Monitor runtime and emit heartbeat signals  
- Release the lease on successful completion (or force‑release after timeout)  
- Log every event to a structured JSON file at `/var/log/cron-guard/jobs/daily-report.log`

### Step 3: Set Up Real‑Time Monitoring Dashboard

Cron Guard exposes a Prometheus metrics endpoint on port 9090 by default. To visualize:

```yaml
# prometheus.yml scrape config
scrape_configs:
  - job_name: 'cron-guard'
    static_configs:
      - targets: ['localhost:9090']
```

Then import the **Cron Guard Dashboard** (ID: 12345) into Grafana. It shows:  
- Job execution status (success, failure, timeout, overlap)  
- Runtime distribution histograms  
- Lease token health (expired, active, stale)  
- Heartbeat miss rate per job  

You can also set up alert rules directly in Prometheus, e.g.,:

```yaml
groups:
  - name: cron-guard-alerts
    rules:
      - alert: CronJobHeartbeatMissed
        expr: rate(cron_guard_heartbeat_misses_total[5m]) > 0
        for: 1m
        annotations:
          summary: "Heartbeat missed for job {{ $labels.job }}"
```

## 2 Industry Best Practices

### 1. **Always Use a Separate User for Cron Guard**

Create a dedicated system user (`cron-guard`) with minimal permissions. The user should only have:  
- Read access to the job scripts (via a dedicated group, e.g., `cron-jobs`)  
- Write access to the log directory (`/var/log/cron-guard/`)  
- No shell access (`/usr/sbin/nologin`)  

This prevents privilege escalation if a monitored job is compromised. Also, run the Cron Guard daemon itself under a read‑only root filesystem (e.g., using `ReadOnlyPaths=/` in systemd).

### 2. **Implement a Stale‑Lock Cleanup Cron Job**

Even with lease expiry, a network partition or Redis outage can leave stale leases. Run a separate, minimal cron job (every 5 minutes) that:  
- Queries the Redis store for leases older than 2× the max timeout  
- Logs and alerts on any stale leases  
- Force‑removes them  

This is your safety net. Example:

```bash
*/5 * * * * /usr/local/bin/cron-guard cleanup-stale --max-age 600
```
