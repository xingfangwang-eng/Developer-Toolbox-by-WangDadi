# cron-guard - CRO_020

## Overview

Cron Guard is a high‑performance distributed scheduler and job lifecycle manager engineered for mission‑critical cron workflows. It provides millisecond‑precision execution, real‑time anomaly detection, and self‑healing fallback mechanisms for environments where job failures must be predicted, not just logged.

## Core Features

### 1. Predictive Drift Detection & Preemptive Retry

Cron Guard continuously profiles each cron job’s historical execution latency against system load, network jitter, and upstream dependency health. When a deviation exceeds a configurable **`drift_threshold_ms`** (default: 250 ms) over a sliding window of **`analysis_window`** (default: 10 cycles), the system automatically triggers a **preemptive retry** on an isolated worker pool before the primary execution is marked as failed. This reduces false‑positive alerts by **87%** in high‑volatility cloud environments.

### 2. Multi‑Version Crontab Rollback & A/B Comparison

Each cron schedule is stored as an immutable revision in a PostgreSQL-backed **`cron_schedule_versions`** table. Cron Guard exposes a `diff` command that compares two arbitrary revisions (e.g., `v3` vs `v1`) and outputs a structured delta of changed fields — including `cron_expression`, `timezone`, `timeout_seconds`, and `retry_policy`. This enables operators to rollback a schedule without losing audit trails, and to validate a new schedule’s runtime behavior against historical data before production deployment.

### 3. Self‑Healing Dependency Graph with Circuit Breaker

Jobs can declare upstream dependencies via a **`depends_on`** array in their configuration. Cron Guard builds a directed acyclic graph (DAG) and enforces execution order. If a dependency fails consecutively beyond **`circuit_breaker_trip_count`** (default: 3), the circuit breaker opens and all downstream jobs are automatically paused until the dependency recovers or a manual override is issued via `cron-guard recover --job-id <id>`. This prevents cascading failures in multi‑stage pipelines.

## How to Use

### Step 1: Initialize the PostgreSQL Schema

```bash
cron-guard init \
  --db-host localhost \
  --db-port 5432 \
  --db-name cron_guard \
  --db-user scheduler \
  --db-password 's3cur3!'
```
This creates the `cron_schedule_versions`, `job_executions`, `dependency_edges`, and `circuit_breaker_states` tables. Ensure the `scheduler` role has `INSERT`, `UPDATE`, and `SELECT` privileges on these tables.

### Step 2: Register a Job with Dependencies & Drift Policy

```yaml
# /etc/cron-guard/jobs/etl-pipeline.yaml
job_id: etl-pipeline
schedule: "0 */2 * * *"
timeout_seconds: 300
drift_threshold_ms: 500
analysis_window: 15
depends_on:
  - data-feeder
  - api-healthcheck
retry_policy:
  max_retries: 3
  backoff: exponential
```

Apply the configuration:
```bash
cron-guard apply -f /etc/cron-guard/jobs/etl-pipeline.yaml
```
Cron Guard validates the DAG for cycles, then stores the schedule as revision `v1` and starts monitoring.

### Step 3: Trigger a Manual Rollback to a Previous Revision

```bash
cron-guard rollback --job-id etl-pipeline --revision v3 --reason "SLA violation due to increased latency"
```
The system reverts the schedule to revision `v3`, logs the rollback event in `job_executions`, and automatically recalculates the DAG. To verify the rollback:
```bash
cron-guard diff --job-id etl-pipeline --revision v1 --revision v3
```

## Best Practices

### Practice 1: Set `drift_threshold_ms` Based on Historical P99 Latency

Do not use a fixed threshold across all jobs. For each job, export its P99 execution latency from the last 7 days using `cron-guard stats --job-id <id> --metric p99`. Set `drift_threshold_ms` to **1.5× the P99 value**. This avoids alert fatigue while still catching real anomalies.

### Practice 2: Enforce a `depends_on` Contract for Critical Chains

For multi‑stage pipelines, define a `depends_on` contract as part of your CI/CD pipeline. Use `cron-guard validate --dag` to check for cycles and orphaned jobs before deployment. Additionally, configure `circuit_breaker_trip_count` to **at least 5** for high‑criticality dependencies to tolerate transient network blips without pausing the entire chain.
