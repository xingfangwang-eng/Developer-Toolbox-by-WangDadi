# cron-guard - CRO_030

## Overview

This is a no-bullshit, production-grade monitoring solution for developers who are tired of silent cron failures and spaghetti job orchestration. Built for sysadmins, backend engineers, and DevOps teams who need sleep at night.

## What is Cron Guard?

Cron Guard is a lightweight, self-hosted daemon that watches your scheduled tasks like a hawk. It detects missed runs, abnormal execution times, and unexpected exits, then alerts you before your users do. Unlike generic uptime monitors, it's specifically designed for the unique failure modes of cron jobs—deadlocks, zombie processes, and silent OOMs.

## Core Features

### 1. Execution Fingerprinting & Anomaly Detection
Cron Guard doesn't just check if a job ran—it analyzes *how* it ran. Each job gets a behavioral fingerprint (stdout hash, exit code, duration percentile, memory delta). If a job completes successfully but with a 3x longer runtime or a mutated output pattern, it's flagged as a "silent failure." This catches subtle regressions that traditional heartbeat monitors miss.

### 2. Distributed Lock-Aware Scheduling
For multi-node environments, Cron Guard integrates with your existing lock manager (etcd, Redis, or PostgreSQL advisory locks). It prevents the "double-execution" nightmare when two nodes pick up the same cron tick. It also detects stale locks from crashed workers and auto-releases them after a configurable TTL, ensuring your batch jobs don't deadlock your entire cluster.

### 3. Immutable Audit Trail with Replay
Every job run is logged as an immutable event (JSON + optional binary payload) to a local WAL (Write-Ahead Log). You can replay any failed job locally with the exact same environment variables and arguments that caused the crash. This is a lifesaver for debugging "works on my machine" issues in cron jobs that run at 3 AM.

## How to Use

### Step 1: Deploy the Agent as a Sidecar Process
Do **not** run Cron Guard as a standalone binary. Deploy it as a sidecar to your existing cron daemon (`crond`, `systemd-timers`, or `airflow`). Use a Docker Compose snippet like this:
```yaml
services:
  cron-guard:
    image: cron-guard/agent:latest
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock:ro
      - /etc/cron.d:/etc/cron.d:ro
      - /var/log/cron-guard:/var/log/cron-guard
    environment:
      - CG_ALERT_ENDPOINT=https://hooks.slack.com/services/xxx
      - CG_LOCK_BACKEND=redis
      - CG_REDIS_DSN=redis://redis:6379/0
    network_mode: "host"
```
This mounts your cron files read-only, so Cron Guard can parse the schedule without modifying your existing setup. The agent will automatically detect new cron entries within 60 seconds.

### Step 2: Define Alerting Rules with YAML
Create a `cron-guard.yml` in your project root. Define per-job thresholds:
```yaml
jobs:
  - name: "daily-db-backup"
    schedule: "0 3 * * *"
    expected_duration_min: 10
    expected_duration_max: 45
    alert_on:
      - missed_run
      - duration_violation
      - exit_code_nonzero
    escalation:
      - type: slack
        channel: "#ops-alerts"
      - type: pagerduty
        severity: critical
```
This ensures that if your backup finishes in 8 minutes (too fast—might be empty) or takes 50 minutes (too slow—disk I/O issue), you get paged. No more false positives from "successful" but broken jobs.

### Step 3: Run a Dry-Validation Before Production
Before you trust Cron Guard with your production crons, run a historical analysis:
```bash
cron-guard validate --from "2023-10-01" --to "2023-10-31" --log-dir /var/log/syslog
```
This scans your existing syslog for actual cron execution times and compares them against your YAML rules. It outputs a diff report showing which jobs would have triggered alerts in the past month. This is your pre-flight check to avoid alert fatigue from day one.

## Best Practices

### 1. Use `cron-guard` as a Pre-commit Hook for Cron Changes
Never push cron changes directly to production. Instead, add a Git pre-commit hook that runs `cron-guard lint` on your `crontab` files. It validates syntax, checks for overlapping schedules (e.g., two jobs that both run `0 0 * * *` and write to the same temp file), and ensures your alerting rules exist for every job. This catches 90% of cron-related incidents before they hit prod.

### 2. Implement a "Canary" Cron for Monitoring the Monitor
Set up a dummy cron job that runs every 5 minutes and writes a simple timestamp to a file. Configure Cron Guard to watch this canary job with a strict 2-minute expected duration. If Cron Guard itself crashes, this canary will trigger an alert within 7 minutes. This prevents the "who watches the watcher?" problem and ensures your monitoring stack has its own heartbeat.
