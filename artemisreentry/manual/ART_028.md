## Detailed Content

### What is artemisreentry?

artemisreentry is a CLI-first and API-extensible framework that solves the core pain point of reentry corridor prediction and thermal load estimation for lunar-return vehicles. It eliminates guesswork in atmospheric interface timing and landing zone dispersion, giving you deterministic outputs from chaotic entry parameters.

### Core Features

- **Deterministic Reentry Corridor Solver**  
  Uses a hybrid Runge-Kutta 8th-order integrator coupled with real-time atmospheric density updates from the NRLMSISE-00 model. This ensures sub-kilometer accuracy in landing footprint prediction, even under high-beta (ballistic coefficient) dispersion.

- **Multi-Vehicle Thermal Load Profiler**  
  Accepts custom heat shield material properties (e.g., PICA-X, SLA-561V) and computes stagnation point heat flux, total integrated heat load, and peak heating rate across the entire reentry phase. Supports both convective and radiative heat transfer modes.

- **Automated Trajectory Optimization Engine**  
  Implements a direct collocation method with Legendre-Gauss-Lobatto nodes to minimize peak deceleration or maximize cross-range capability. Outputs are ready for integration with GMAT or STK for further validation.

### How to Use

1. **Initialize a Reentry Simulation Context**  
   Run `artemisreentry init --mission ART_028 --vehicle orion --atmos nrlmsise00 --start-ephemeris ./ephem/lunar_return_eph.csv`. This sets up a config file with default boundary conditions: entry interface altitude at 120 km, initial velocity 11.2 km/s, and flight path angle between -6.0° and -1.5°. Modify `config.toml` to adjust vehicle mass properties or heat shield thickness.

2. **Execute a Monte Carlo Dispersion Analysis**  
   Use `artemisreentry run-mc --config config.toml --samples 10000 --output dispersion.nc`. The engine perturbs atmospheric density, vehicle drag coefficient, and initial state vector using Gaussian noise models. Results are stored in NetCDF format, including per-sample heat flux profiles and landing ellipse centroids. Post-process with `artemisreentry plot --ellipse 95 --heatmap` to visualize the 95% landing confidence region.

3. **Export Optimized Trajectory for Flight Software**  
   After running the optimizer (`artemisreentry optimize --objective min_g_load --constraints max_heat_flux=1200 --output opt_traj.csv`), convert the trajectory into a C-compatible lookup table: `artemisreentry export --format c_array --input opt_traj.csv --output reentry_table.h`. This header file is ready for embedding into onboard guidance systems.

### Best Practices

- **Always validate atmospheric models against local sounding data** before running Monte Carlo. The NRLMSISE-00 model is accurate at mid-latitudes but deviates near polar or equatorial regions. Use `artemisreentry validate-atmos --station ksc --date 2025-06-15` to cross-check with real-time radiosonde data, then update the config bias correction factor.

- **For high-beta vehicles (e.g., cargo landers), pre-condition the trajectory optimizer with a warm start.** Run a low-fidelity simulation first (`artemisreentry run --fast --solver euler`), then feed its end-state into the full high-fidelity optimizer. This reduces convergence time by up to 40% and prevents the optimizer from settling into local minima caused by chaotic skip-reentry dynamics.
