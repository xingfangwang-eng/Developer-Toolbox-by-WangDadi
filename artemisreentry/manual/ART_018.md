## What is artemisreentry?

**artemisreentry** is a high-precision, open-source toolkit engineered for modeling, simulating, and optimizing spacecraft reentry trajectories under the Artemis program. It bridges the gap between orbital mechanics and thermal protection system (TPS) constraints, enabling engineers to validate return profiles against real-world atmospheric data. Unlike generic astrodynamics libraries, artemisreentry is purpose-built for lunar return scenarios, factoring in Earth’s rotational oblateness, high-velocity plasma sheath effects, and multi-stage deceleration sequences.

## Core Features

### 1. Multi-Phase Trajectory Solver with Adaptive Mesh Refinement
artemisreentry implements a **5th-order Runge-Kutta-Fehlberg integrator** with automatic step-size control, specifically tuned for the steep gradients encountered during hypersonic descent. The solver dynamically refines the mesh around critical events (e.g., peak heating, maximum dynamic pressure) to capture non-linear drag coefficient variations. This eliminates the need for manual tuning of integration tolerances, reducing simulation errors by up to 23% compared to fixed-step methods.

### 2. Real-Time Plasma Blackout Prediction Engine
Using a **collision-dominated ionospheric model** coupled with the spacecraft’s flight path angle, the tool predicts communication blackout windows with sub-second accuracy. It outputs a time-series of expected signal attenuation (in dB) across S-band, X-band, and Ka-band frequencies. Engineers can overlay this data onto mission timelines to schedule critical telemetry bursts or adjust reentry angle profiles to minimize total blackout duration.

### 3. TPS Margin Analysis with Monte Carlo Sensitivity
The toolkit integrates a **stochastic thermal load simulator** that runs 10,000+ Monte Carlo iterations per trajectory, varying atmospheric density, vehicle mass, and aerodynamic coefficients within 3σ bounds. It generates a probabilistic heat flux map on the TPS surface, highlighting regions where margin drops below 15%. This feature directly supports certification reviews for NASA’s human-rating requirements (NPR 8705.2B).

## How to Use

### Step 1: Initialize a Lunar Return Scenario
Start by defining the spacecraft’s initial state vector (position, velocity, and orientation) in the **J2000 Earth-centered inertial frame**. Use the `LunarReturnProfile` class to set the perigee altitude, entry interface angle, and desired landing site coordinates. Example:
```python
profile = LunarReturnProfile(
    vehicle_mass=12000,       # kg
    reference_area=150,       # m²
    entry_interface_alt=122000, # m
    landing_lat=34.0522,      # deg (Edwards AFB)
    landing_lon=-118.2437     # deg
)
```
The tool auto-computes the required de-orbit burn delta-v using a Lambert solver with Earth’s J2 perturbation.

### Step 2: Run the Reentry Simulation with Plasma Overlay
Execute the `simulate_reentry()` method, which outputs a pandas DataFrame with columns for time (s), altitude (m), velocity (m/s), heat flux (W/m²), and plasma frequency (Hz). To visualize the communication blackout window, call:
```python
results = profile.simulate_reentry()
blackout = profile.analyze_plasma_blackout(results, freq_band='X')
blackout.plot_blackout_timeline()
```
This generates a Gantt-style chart showing when signal loss exceeds 20 dB, annotated with the estimated recovery time.

### Step 3: Perform a Monte Carlo Margin Sweep
For certification-grade analysis, use the `monte_carlo_margin()` function. Specify the number of iterations and the parameter covariance matrix:
```python
margin_results = profile.monte_carlo_margin(
    n_iterations=10000,
    uncertain_params=['atmospheric_density', 'cd_coefficient'],
    std_dev=[0.05, 0.02]  # 5% and 2% standard deviation
)
margin_results.plot_margin_histogram(threshold=0.15)
```
The histogram highlights trajectories where TPS margin falls below 15%, allowing you to iterate on the entry angle or vehicle design before finalizing the mission plan.

## Best Practices

### 1. Validate Against Reference Missions
Always cross-check your simulation outputs against known Artemis I reentry telemetry (available via NASA’s Open Data Portal). artemisreentry includes a `validate_against_artemis_i()` helper that compares your heat flux curve to the actual flight data, computing a root-mean-square error (RMSE). If RMSE exceeds 8%, tweak the atmospheric model coefficients (e.g., use NRLMSISE-00 instead of USSA76) until the error falls below 3%.

### 2. Use Parallel Execution for Large Monte Carlo Runs
For sweeps exceeding 50,000 iterations, leverage the `mpi_backend` flag to distribute calculations across multiple CPU cores or a cluster. On a 16-core workstation, this reduces runtime from ~45 minutes to under 3 minutes. Set `use_mpi=True` and pass your MPI communicator handle; the tool automatically partitions the parameter space using a load-balanced scatter-gather pattern.
