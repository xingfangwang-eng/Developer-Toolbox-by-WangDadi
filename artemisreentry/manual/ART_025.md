# artemisreentry - ART_025

## Overview

`artemisreentry` is a high-precision, low-latency reentry analysis engine built for aerospace engineers and mission planners who need to model, simulate, and validate spacecraft return trajectories from cislunar orbits. It bridges the gap between theoretical orbital mechanics and real-time telemetry validation, offering a CLI-first and API-native interface optimized for CI/CD pipelines in space mission software stacks.

## Core Features

### 1. Multi-Phase Reentry Trajectory Solver with Adaptive Step Integration
The engine runs a hybrid Runge-Kutta (RK45 + implicit Adams-Moulton) solver that dynamically adjusts step size based on atmospheric density gradients and heat flux thresholds. It supports real-time perturbation injection (e.g., unexpected thruster misfire or drag coefficient drift) and outputs state vectors at configurable telemetry intervals.

### 2. Real-Time Ablation & Thermal Protection Margin Calculator
Instead of relying on post-process CFD, `artemisreentry` embeds a reduced-order thermal model that estimates TPS (Thermal Protection System) erosion and back-wall temperature in lockstep with trajectory propagation. This allows engineers to flag margin violations during the simulation itself, not after.

### 3. Automated Multi-Mission Regression Suite with Delta-V Sensitivity
The tool includes a built-in regression harness that compares current mission parameters (entry angle, velocity, mass) against a historical database of Artemis-style reentries. It automatically generates a sensitivity matrix showing which parameters most affect peak deceleration and landing ellipse uncertainty.

## How to Use

### Step 1: Define Mission Profile and Boundary Conditions
Create a YAML configuration file specifying initial state vectors (ECI frame), vehicle mass properties (including jettison events), and atmospheric model selection (NRLMSISE-00 or custom tabular). Example snippet:
```yaml
mission:
  name: "ART-025"
  epoch: "2025-09-15T14:32:00Z"
  state_vector: [ -6524.1, 2841.3, 0.0, -1.2, -7.8, 0.0 ]
  vehicle:
    mass_initial_kg: 8900
    tps_type: "PICA"
    jettison_events:
      - time_s: 120.0
        mass_loss_kg: 320
```

### Step 2: Run a Staged Simulation with Event Triggers
Execute the simulation with event-based logging:
```bash
artemisreentry run --config art025.yaml --events thermal_peak,decel_max,chute_deploy
```
The engine will output a JSON stream with timestamps for each event, including the exact second when TPS back-wall temperature exceeds 95% of its rated limit.

### Step 3: Analyze Landing Ellipse and Sensitivity
Use the built-in Monte Carlo mode to perturb key parameters (entry angle ±0.5°, velocity ±10 m/s) across 1000 runs:
```bash
artemisreentry montecarlo --config art025.yaml --iterations 1000 --output ellipse.geojson
```
This generates a GeoJSON polygon representing the 95% confidence landing ellipse, along with a CSV summary of which parameters contributed most to dispersion.

## Best Practices

### 1. Validate Against High-Fidelity Reference Data Before Flight
Always run a validation pass using known telemetry from a similar reentry (e.g., Orion EFT-1). Use the `--compare` flag to overlay your `artemisreentry` output against a reference trajectory CSV. Discrepancies in peak heat flux greater than 3% indicate a need to recalibrate the solver’s atmospheric density model.

### 2. Integrate Into Your CI/CD Pipeline with Pre-Commit Checks
Add a pre-commit hook that runs a lightweight sanity check on any mission config changes:
```bash
artemisreentry validate --config art025.yaml --quick
```
This catches invalid state vectors, missing jettison events, or out-of-range entry angles before they reach the simulation stage. For production, run a full Monte Carlo suite nightly and compare results against a baseline commit hash.
