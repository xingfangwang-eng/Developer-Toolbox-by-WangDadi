# artemisreentry - ART_017

## Übersicht

Dieses Repository enthält die präzise Steuerungslogik für atmosphärische Wiedereintritte von Raumfahrzeugen. Es richtet sich an Entwickler und Missionsingenieure, die eine reproduzierbare, simulationsgesteuerte Rückkehrberechnung benötigen. Der Fokus liegt auf numerischer Stabilität, Echtzeit-Telemetrie-Kopplung und validierten Grenzwerten für Hitzefluss und strukturelle Lasten.

## Detaillierter Inhalt

### Was ist artemisreentry?

artemisreentry ist ein modulares Framework zur Berechnung und Optimierung von Wiedereintrittstrajektorien. Es kombiniert skriptgesteuerte Atmosphärenmodelle (NRLMSISE-00, US Standard 1976) mit einem eigenen ODE-Solver für sechs Freiheitsgrade. Die API ist für die Integration in CI/CD-gesteuerte Missionssimulationen ausgelegt und unterstützt sowohl Batch- als auch Echtzeit-Modi.

### Kernfunktionen

1. **Dynamische Hitzeflusslimitierung mit adaptiver Bankrotation**  
   Das System überwacht kontinuierlich den konvektiven und radiativen Wärmestrom an der Hitzeschildoberfläche. Sinkt der Abstand zum kritischen Grenzwert unter 5 %, wird automatisch eine Rollmanöver-Sequenz eingeleitet, die den Anstellwinkel modifiziert und die thermische Last gleichmäßig verteilt. Die Logik verwendet einen PID-Regler mit Feedforward-Faktor aus der aktuellen Machzahl.

2. **Echtzeit-Telemetrie-Kopplung via ZeroMQ**  
   Über einen asynchronen Pub/Sub-Kanal werden Trajektoriendaten (Position, Geschwindigkeit, Beschleunigung, Wärmestrom) mit einer Frequenz von 100 Hz an externe Systeme gesendet. Empfänger können beispielsweise Bodensensoren oder Bordcomputer sein. Die Serialisierung erfolgt über Protocol Buffers, was eine Latenz unter 1 ms ermöglicht.

3. **Deterministisches Fallback-Modell bei Atmosphärenausfall**  
   Sollte das primäre Atmosphärenmodell keine gültigen Daten liefern (z. B. durch Sensorausfall oder Modellinkonsistenz), schaltet die Engine nahtlos auf ein vorberechnetes Lookup-Table um. Dieses basiert auf einer kubischen Spline-Interpolation historischer Wiedereintrittsdaten und garantiert einen stabilen, wenn auch suboptimalen Pfad.

### So verwenden Sie es

1. **Konfiguration der Missionsparameter**  
   Erstellen Sie eine YAML-Datei mit den Startwerten:  
   ```yaml
   mission:
     altitude: 120000     # Meter
     velocity: 7800       # m/s
     flight_path_angle: -1.5  # Grad
     mass: 8500           # kg
     reference_area: 12.6 # m²
   ```
   Übergeben Sie diese an den `ReentrySimulator`:
   ```python
   from artemisreentry import ReentrySimulator
   sim = ReentrySimulator.from_yaml('mission_config.yaml')
   ```

2. **Ausführung der Trajektoriensimulation**  
   Starten Sie die Simulation mit gewünschter Schrittweite (z. B. 0.1 s) und optionalem Telemetrie-Stream:
   ```python
   results = sim.run(dt=0.1, telemetry_port=5555)
   ```
   Während der Laufzeit können Sie über den ZeroMQ-Socket Daten abgreifen:
   ```bash
   python telemetry_listener.py --port 5555
   ```

3. **Post-Processing und Validierung**  
   Nutzen Sie die integrierte Exportfunktion für CSV oder HDF5:
   ```python
   results.to_hdf5('reentry_data.h5')
   ```
   Führen Sie anschließend die integrierte Validierung durch:
   ```python
   validator = ReentryValidator(results)
   validator.check_heat_flux_limit(limit=1200)  # kW/m²
   validator.check_deceleration_limit(limit=8)  # g
   ```
   Bei Überschreitung liefert der Validator detaillierte Logs mit Zeitstempel und Abweichung.

### Beste Praktiken

1. **Simulation vor dem Flug mit Monte-Carlo-Variationen absichern**  
   Variieren Sie mindestens die Parameter Anfangsgeschwindigkeit (±2 %), Atmosphärendichte (±5 %) und aerodynamische Beiwerte (±10 %). Nutzen Sie dazu die `MonteCarloRunner`-Klasse, die automatisch 1000 Läufe parallelisiert und die statistische Verteilung der maximalen Hitzefluss- und Beschleunigungswerte ausgibt. So erkennen Sie kritische Randbedingungen vor dem echten Einsatz.

2. **Telemetrie-Daten in Echtzeit mit einem externen Dashboard visualisieren**  
   Richten Sie einen separaten Prozess ein, der über ZeroMQ die Daten des simulierten Wiedereintritts empfängt und in Grafana anzeigt. Nutzen Sie die bereitgestellte `GrafanaPublisher`-Klasse, die Datenpunkte direkt an InfluxDB sendet. So behalten Sie während der Simulation jederzeit den Überblick über kritische Größen und können im Fehlerfall manuell eingreifen.
