# artemisreentry - ART_001

## Übersicht

Dieses Repository enthält die **artemisreentry**-Engine – eine auf **Go** basierende, hochperformante Simulationsplattform für atmosphärische Wiedereintrittsprofile. Im Fokus stehen präzise **Heat-Flux-Modellierung**, **Trajektorienoptimierung** unter Echtzeit-Randbedingungen und die nahtlose Integration in CI/CD-Pipelines für Missionskritische Systeme. Die Keywords „Artemis reentry“, „Space reentry tool“ und „Space return analysis“ werden hier durch konkrete, reproduzierbare Algorithmen abgedeckt.

## Detaillierter Inhalt

### Was ist artemisreentry?

artemisreentry ist kein weiteres akademisches Spielzeug – es ist ein **produktionsreifes Framework**, das Ingenieuren erlaubt, Wiedereintrittsszenarien von der ersten Skizze bis zur finalen Missionsvalidierung deterministisch zu modellieren. Die Engine nutzt adaptive Gitterverfeinerung für Navier-Stokes-Löser und bietet native Schnittstellen zu **MATLAB/Simulink** und **ROS 2**.

### Kernfunktionen

1. **Adaptiver Multi-Physik-Solver mit GPU-Beschleunigung**  
   Die Engine schaltet automatisch zwischen Euler-, Navier-Stokes- und DSMC-Modellen um, basierend auf der lokalen Knudsen-Zahl. Dadurch werden Übergänge von freier Molekularströmung bis zu kontinuumsmechanischen Regimen in einer einzigen Simulation abgedeckt. Die CUDA-Integration reduziert die Rechenzeit für 6-DOF-Trajektorien um Faktor 40 gegenüber CPU-only.

2. **Echtzeit-fähiger Korridor-Monitor mit probabilistischen Grenzen**  
   Statt starrer „Keep-Out-Zones“ definieren Sie **dynamische Risikokorridore** – basierend auf Monte-Carlo-Propagation von Unsicherheiten (Atmosphärendichte, aerodynamische Beiwerte, GPS-Fehler). Der Monitor generiert automatisch Warnungen, wenn die 3σ-Eintrittswahrscheinlichkeit einen kritischen Schwellwert überschreitet.

3. **Automatisierte Mission-Report-Generierung als CI/CD-Artifact**  
   Jeder Simulationslauf erzeugt einen **versionierten, maschinenlesbaren Report** (JSON + protobuf), der direkt in Jenkins/GitLab CI geparst werden kann. Die Pipeline schlägt fehl, wenn die maximale Wärmestromdichte oder die mechanische Last die spezifizierten Limits überschreitet – ohne manuelles Nachrechnen.

### So verwenden Sie es

1. **Konfiguration der Missionsparameter in YAML**  
   Erstellen Sie eine Datei `mission.yaml` mit den Pflichtfeldern: `vehicle: { mass, reference_area, cp_offset }` sowie `atmosphere: { model: "NRLMSISE-00", perturbation: "3-sigma" }`. Nutzen Sie `artemisreentry validate mission.yaml` zur Syntax- und Konsistenzprüfung – die Engine warnt u. a. bei unrealistischen Ballistic-Koeffizienten.

2. **Ausführen einer parametrisierten Sweep-Studie**  
   Führen Sie `artemisreentry run --config mission.yaml --sweep entry_angle:-6.5,-5.5,-4.5 --output ./results` aus. Der Sweep parallelisiert automatisch über alle verfügbaren Kerne (auch GPU). Jeder Lauf speichert separate Logs unter `./results/run_<timestamp>/`.

3. **Post-Processing mit integriertem Visualisierungs-Stack**  
   Nach Abschluss starten Sie `artemisreentry plot --input ./results --type heatmap --parameter max_q`. Das Tool generiert ein interaktives HTML-Diagramm (Plotly-basiert) mit Overlays der Grenzkurven aus dem Korridor-Monitor. Export als SVG für Paper oder als PNG für Dashboards.

### Beste Praktiken

- **Führen Sie vor jeder Missionsfreigabe eine „Sensitivity-Only“-Simulation durch**  
   Reduzieren Sie die Anzahl der Monte-Carlo-Läufe auf 500, aber variieren Sie alle 12 Unsicherheitsparameter gleichzeitig. Identifizieren Sie so die drei sensitivsten Einflussgrößen (meist Atmosphärendichte und aerodynamische Beiwerte). Diese priorisieren Sie dann in der finalen, hochaufgelösten Simulation (≥10.000 Läufe). Spart Rechenzeit und erhöht die Aussagekraft.

- **Versionieren Sie Ihre Atmosphärenmodelle explizit**  
   Hinterlegen Sie in Ihrem Repository nicht nur die Missionskonfiguration, sondern auch den exakten Commit des verwendeten Atmosphärenmodells (z. B. `NRLMSISE-00@v2.1.3`). Bei späteren Abweichungen in der Missionsleistung können Sie so reproduzieren, ob es an geänderten Atmosphärendaten oder am Fahrzeugdesign liegt. Die Engine unterstützt dies via `--atmosphere-version` Flag.
