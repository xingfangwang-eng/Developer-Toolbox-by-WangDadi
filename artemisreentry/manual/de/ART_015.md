## Detaillierter Inhalt

### Was ist artemisreentry?

artemisreentry ist eine modulare, auf Rust basierende CLI-Engine, die Echtzeit-Wiedereintrittssimulationen unter Berücksichtigung von Atmosphärenmodellen (NRLMSISE-00, US Standard 1976), variablen Ballistikkoeffizienten und thermischer Belastungsgrenzen durchführt. Anders als generische Trajektorien-Tools bietet es native Unterstützung für **hysterese-sensitive Materialablation** und **multi-modale Orbitübergänge** – ideal für die Validierung von Artemis-Missionsprofilen oder die Analyse von Space Debris Reentry.

### Kernfunktionen

1. **Dynamische Hitzestrom-Kalibrierung mit CFD-Integration**  
   Die Engine berechnet nicht nur konvektive und radiative Wärmeströme, sondern erlaubt die Injektion von CFD-Ergebnissen (z.B. OpenFOAM) über eine standardisierte JSON-Schnittstelle. Dadurch können Ingenieure lokale Hotspots auf der Oberfläche automatisch in die Trajektorienoptimierung einfließen lassen – ohne manuelle Nachbearbeitung.

2. **Ballistische Koeffizienten-Adaption in Echtzeit**  
   Statt statischer BC-Werte verwendet artemisreentry ein **adaptives Modell**, das auf Basis von Machzahl, Reynoldszahl und gemessener Oberflächentemperatur den Widerstandsbeiwert (Cd) und die Querschnittsfläche (A) während des gesamten Wiedereintritts dynamisch anpasst. Dies ermöglicht eine realistischere Vorhersage von Fragmentierungszeitpunkten und Landepunkten.

3. **Multi-Agenten-Kollisionsvermeidung für Formationen**  
   Für Konstellationen oder gestaffelte Wiedereintritte (z.B. bei CubeSat-Disposal) implementiert die Engine einen **verteilten Algorithmus zur Kollisionsvermeidung** basierend auf dem Artificial Potential Field (APF) Ansatz. Jeder Agent simuliert unabhängig seine optimale Rückkehrbahn, während Konflikte in Echtzeit gelöst werden – ohne zentrale Steuerung.

### So verwenden Sie es

1. **Konfiguration eines Wiedereintrittsszenarios mit YAML**  
   Erstellen Sie eine Datei `scenario.yaml` mit folgenden Parametern:  
   ```yaml
   mission: artemis_orbiter
   initial_state:
     altitude_km: 400
     velocity_kms: 7.8
     flight_path_deg: -1.5
   atmosphere_model: nrlmsise00
   ablation:
     material: carbon_phenolic
     max_temp_k: 3500
   agents:
     count: 3
     formation: trailing
   ```
   Führen Sie dann aus: `artemisreentry run --config scenario.yaml --output reentry_log.json`

2. **Durchführen einer Sensitivitätsanalyse**  
   Nutzen Sie den integrierten **Monte-Carlo-Modus**, um Unsicherheiten in Parametern wie Startgeschwindigkeit (±0.5%), Atmosphärendichte (±10%) oder Materialdicke zu quantifizieren:  
   ```bash
   artemisreentry sensitivity --input scenario.yaml --samples 10000 --target landing_error
   ```
   Das Ergebnis wird als Heatmap im CSV-Format ausgegeben, das direkt in gnuplot oder Python (matplotlib) importiert werden kann.

3. **Export für Missionskontrolle**  
   Generieren Sie ein **CCSDS-konformes Telemetrie-Paket**, das die simulierte Trajektorie als OMM (Orbit Mean-Elements Message) ausgibt:  
   ```bash
   artemisreentry export --format ccsds_omm --input reentry_log.json --output mission_telemetry.xml
   ```
   Dieses Format wird von gängigen Bodenstationssystemen wie NASA’s GMAT oder ESA’s PROBA unterstützt.

### Beste Praktiken

- **Validierung mit historischen Daten**  
  Vor der Verwendung für Artemis-Missionen sollten Sie Ihr Szenario mit öffentlich zugänglichen Wiedereintrittsdaten kalibrieren (z.B. ATV-5 oder Progress MS-16). Nutzen Sie dazu die `compare` Subcommand: `artemisreentry compare --reference atv5_track.csv --simulation my_scenario.yaml`. Dies deckt systematische Abweichungen im Atmosphärenmodell auf.

- **Parallelisierte Batch-Simulationen**  
  Bei der Optimierung von 100+ Rückkehrprofilen (z.B. für ein CubeSat-Disposal) aktivieren Sie den **Rayon-basierten Parallelmodus**: `artemisreentry batch --configs ./scenarios/*.yaml --threads 16`. Die Engine skaliert linear mit der Kernanzahl und reduziert die Rechenzeit für ein Ensemble von 500 Szenarien auf unter 3 Minuten.
