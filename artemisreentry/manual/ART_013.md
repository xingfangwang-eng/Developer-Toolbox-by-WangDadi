# artemisreentry - ART_013

## Overview

Artemis Reentry is a precision-engineered orbital return toolkit designed for aerospace engineers and mission planners who need deterministic reentry corridor analysis. Unlike generic trajectory simulators, this tool focuses on the unique thermal and aerodynamic constraints of return vehicles from lunar and cislunar trajectories—specifically optimized for Artemis-class missions. It bridges the gap between high-fidelity CFD simulations and rapid trade-space exploration, enabling real-time parametric sweeps that would otherwise require hours on a HPC cluster.

## Core Features

### 1. Multi-Phase Ablation Profile Engine (MAPE)
MAPE dynamically models TPS (Thermal Protection System) erosion across distinct reentry phases—exoatmospheric, rarefied, and continuum flow. It ingests real-time material property tables (PICA, SLA-561V, or custom composites) and couples them with trajectory-derived heat flux to predict recession rates within ±2% margin. This eliminates the need for post-mission reconstruction of TPS performance; you can iterate on shield thickness and material layup directly from the command line.

### 2. Cislunar-to-Surface Trajectory Morphing (CST-Morph)
CST-Morph enables continuous transition from a NRHO (Near-Rectilinear Halo Orbit) departure state to a targeted landing site by solving a two-point boundary value problem with embedded atmospheric drag coefficients. It automatically adjusts bank angle modulation to satisfy both peak deceleration (≤4.5 g) and landing footprint constraints. The solver outputs a time-indexed state vector compatible with NASA’s SPICE toolkit, allowing direct integration with existing mission planning pipelines.

### 3. Real-Time Monte Carlo Uncertainty Propagation (RT-MCUP)
RT-MCUP runs a 10,000-member ensemble in under 3 seconds per simulation step, perturbing initial state vectors, atmospheric density models (NRLMSISE-00 vs. JB2008), and vehicle aerodynamic coefficients. It generates probabilistic heat load and landing accuracy distributions, flagging any trajectory that exceeds a user-defined 3σ threshold. This is critical for validating abort scenarios and ensuring crew safety margins are met across all plausible off-nominal conditions.

## How to Use

### Step 1: Configure Reentry Corridor Parameters
Initialize the simulation by defining the entry interface state (altitude 122 km, velocity 11.0 km/s) and vehicle geometry. Use the included `artemis_config.yaml` template to set TPS material, reference area (19.6 m² for Orion-like capsules), and trim angle of attack. Validate the configuration by running `artemisreentry --validate-config`—this checks for thermodynamic consistency (e.g., TPS thickness vs. expected stagnation heat flux) and warns of any non-physical parameter combinations.

```bash
artemisreentry --config artemis_config.yaml --validate-config
```

### Step 2: Execute Parametric Sweep with CST-Morph
Launch a sweep across entry flight path angles (−6.5° to −5.0° in 0.1° increments) to identify the feasible corridor. The tool outputs a `corridor_sweep.json` file containing per-trajectory peak heat rate, total heat load, and landing ellipse semi-major axis. Use the built-in filter `artemisreentry --filter "heat_rate < 80 W/cm² AND ellipse < 10 km"` to isolate viable trajectories. For advanced users, pipe the output into `jq` to extract specific telemetry:

```bash
artemisreentry --sweep --angle_range -6.5 -5.0 --step 0.1 | jq '.trajectories[] | select(.peak_heat_rate < 80) | {angle, ellipse}'
```

### Step 3: Run Monte Carlo Uncertainty Analysis
Activate RT-MCUP by setting `monte_carlo.enabled: true` in the config file. Define perturbation scales for atmospheric density (±15%), drag coefficient (±5%), and initial velocity (±50 m/s). Execute with `artemisreentry --mc --runs 10000`. The output `mc_results.h5` contains an HDF5 dataset with columns for each perturbed variable and the resultant landing deviation. Generate a summary report with:

```bash
artemisreentry --report mc_results.h5 --percentiles 5 50 95
```

This prints the 5th, 50th, and 95th percentile values for peak g-load and cross-range error, along with a histogram plot saved as `mc_distribution.png`.

## Best Practices

### 1. Validate TPS Margins Against Off-Nominal Abort Trajectories
Always run RT-MCUP with a 20% increase in atmospheric density variability when assessing abort scenarios from lunar return. Standard density models (e.g., JB2008) underestimate diurnal bulge effects at high latitudes. Override the default model in `artemis_config.yaml` by setting `atmosphere.model: "NRLMSISE-00"` and `atmosphere.scale_factor: 1.2`. This ensures your TPS design still meets recession limits even under worst-case thermospheric expansion.

### 2. Cross-Check CST-Morph Outputs with SPICE Kernels for Long-Duration Missions
When planning multi-day coast phases prior to reentry (e.g., from a Distant Retrograde Orbit), validate the CST-Morph state vectors against SPICE’s `spkezr` routine. Small errors in Earth rotation model or precession can accumulate into kilometer-level landing errors. Use the included `artemis_to_spice.py` converter to generate a `.bpc` binary PCK file, then run a difference check: `artemisreentry --spice-validate artemis_output.bpc spice_kernels/naif0012.tls`. This catches ephemeris mismatches before they affect mission-critical decisions.
