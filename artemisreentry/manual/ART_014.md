## Core Features (Hardcore)

### 1. Real-Time Conjugate Heat Transfer (CHT) Overlay
Unlike traditional tools that decouple aerothermal loads from structural response, artemisreentry implements a **direct CHT solver** that runs on GPU-accelerated kernels. This allows for sub-second iteration on TPS material thickness vs. peak heat flux, giving you a live, converged solution without the multi-hour CFD wait times.

### 2. Deterministic Ballistic Coefficient Inversion Engine
Stop guessing your drag coefficient. Our engine uses a **Monte Carlo Markov Chain (MCMC)** inversion algorithm on live telemetry streams (or historical logs) to reverse-engineer the exact ballistic coefficient of your reentry vehicle. This is critical for correcting trajectory drift during blackout phases.

### 3. Multi-Phase Constraint Solver with Pareto Front Optimization
artemisreentry handles the chaotic transition from hypersonic to supersonic regimes natively. It uses a **NSGA-III** genetic algorithm to solve for the Pareto-optimal trade-off between peak g-load, landing ellipse error, and maximum stagnation temperature, outputting a ranked list of viable entry corridors.

## How to Use

### Step 1: Initialize a Reentry Profile with Custom Atmospheric Models
Start by instantiating a `ReentryVehicle` object. Instead of relying on standard 1976 US Standard Atmosphere, you can inject a custom `AtmosphericProfile` object derived from high-altitude balloon data or Mars-GRAM models. Use the `profile.set_turbulence(model='k-omega')` method to define boundary layer transition points.

```python
from artemisreentry import ReentryVehicle, AtmosphericProfile

profile = AtmosphericProfile.from_csv('mars_gram_2025.csv')
vehicle = ReentryVehicle(mass=1200, drag_area=3.2, profile=profile, tps_material='PICA-X')
vehicle.solve_initial_conditions(altitude=120_000, velocity=7.8, flight_path_angle=-1.5)
```

### Step 2: Run a Constrained Monte Carlo Sensitivity Sweep
Define your uncertain parameters (e.g., atmospheric density perturbation, initial angle of attack). Use the `sensitivity_analysis` method with a `Latin Hypercube` sampling strategy for maximum coverage with minimal runs.

```python
from artemisreentry.sensitivity import LatinHypercubeSampler

sampler = LatinHypercubeSampler(parameters=['density_factor', 'aoa_offset'], bounds=[(0.95, 1.05), (-2.0, 2.0)])
results = vehicle.run_sweep(sampler, n_samples=5000, output_metrics=['peak_g', 'landing_ellipse_km'])
results.plot_correlation_matrix()
```

### Step 3: Export a Certified Reentry Flight Card
Once you have a viable trajectory, export it as a **NASA-STD-8719.24 compliant Flight Card**. This generates a signed JSON file containing the time-stamped state vectors, control surface commands, and TPS margin calculations.

```bash
artemisreentry export-flight-card \
  --vehicle-id "Orion-014" \
  --trajectory-file "trajectory_opt.npy" \
  --output "flight_card_orion_014.json" \
  --signature "sha256"
```

## Best Practices

### 1. Always Validate Against Post-Flight Telemetry
Don't trust a simulation that hasn't been calibrated against real-world data. Use the **MCMC inversion engine** to back-fit your model parameters against the telemetry from your last flight. This creates a "digital twin" that gets more accurate with every mission. Store these calibrated models in a version-controlled registry.

### 2. Use Multi-Fidelity Surrogates for Design Space Exploration
When running parameter sweeps for TPS sizing, start with a low-fidelity (e.g., Newtonian impact theory) solver for the first 10,000 iterations. Then, use the results to train a Gaussian Process surrogate model. Only deploy the high-fidelity CHT solver on the top 5% of candidates. This reduces total compute time by 80% while maintaining final accuracy within 0.5%.
