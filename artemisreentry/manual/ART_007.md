# artemisreentry - ART_007

## Overview

artemisreentry is a precision-engineered toolkit for modeling, simulating, and optimizing spacecraft reentry trajectories—specifically designed for the Artemis program and next-gen lunar return missions. Unlike generic orbital mechanics libraries, this tool focuses on hypersonic aerothermodynamics, real-time telemetry stitching, and multi-phase trajectory validation.

## Core Features

- **Multi-Phase Reentry Trajectory Solver**  
  Automatically transitions between exoatmospheric coast, atmospheric interface, and terminal descent phases using a hybrid Runge-Kutta + Monte Carlo dropout solver. Supports variable drag coefficient (Cd) models for lifting-body and capsule geometries.

- **Telemetry Gap Interpolation Engine**  
  Handles sparse or noisy telemetry data from TDRSS and lunar relay satellites. Uses a custom Kalman smoother with quaternion-based attitude reconstruction to fill gaps in velocity, heat flux, and G-load profiles without introducing phase lag.

- **Real-Time Ablation Margin Analyzer**  
  Computes TPS (Thermal Protection System) ablation rates against user-defined safety margins. Integrates with NASA’s FIAT and CMA codes via API, outputting a continuous ablation risk index across the reentry corridor.

## How to Use

1. **Initialize a Reentry Profile**  
   Load a mission config (YAML or JSON) specifying entry interface altitude, velocity vector, vehicle mass, and reference area. Call `ReentryProfile.from_config("artemis_001.yaml")` to instantiate. The solver auto-detects the atmospheric model (NRLMSISE-00 or US Standard 1976) based on the altitude range.

2. **Run a Multi-Core Simulation Sweep**  
   Use `sweep_simulate(profile, perturbations={"drag_coeff": [0.8, 1.2], "bank_angle": [-45, 45]})` to execute 10,000+ trajectory variants in parallel via OpenMP or CUDA. Each run outputs a time-series DataFrame with altitude, Mach number, stagnation temperature, and heat flux.

3. **Export a Compliance Report**  
   Generate a PDF or JSON report with `generate_compliance_report(results, thresholds={"max_g_load": 5.5, "max_heat_flux": 120})`. The report highlights any phase where thresholds are breached, along with recommended bank-angle corrections and TPS margin adjustments.

## Best Practices

- **Always Validate Against Flight Data**  
  Before trusting simulation outputs for critical mission planning, run a validation sweep using known reentry data (e.g., Apollo 4 or Orion EFT-1). Use the built-in `compare_to_flight()` method to overlay simulated vs. actual telemetry and compute RMSE for each phase.

- **Use Sensitivity Analysis for TPS Sizing**  
  When designing or certifying thermal protection, run a Monte Carlo sweep over atmospheric density (±10%) and entry angle (±2°). The ablation margin analyzer will flag worst-case scenarios, allowing you to over-design the TPS only where statistically necessary—saving mass without sacrificing safety.
