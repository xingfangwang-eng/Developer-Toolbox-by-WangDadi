# artemisreentry - ART_029

## Overview

**artemisreentry** is a precision-engineered toolkit for modeling, simulating, and analyzing atmospheric reentry trajectories of space vehicles. Built for aerospace engineers, mission planners, and orbital debris researchers, it provides deterministic and probabilistic reentry profiles with sub-kilometer accuracy. The tool is optimized for low-latency computation and integrates directly with GMAT, STK, and custom telemetry pipelines.

## Core Features

### 1. Multi-Phase Reentry Solver with Real-Time Aerothermal Coupling
Unlike traditional single-phase models, artemisreentry uses a **segmented trajectory solver** that dynamically switches between free molecular, transitional, and continuum flow regimes. It couples aerodynamic heating, radiative flux, and ablation rates in real time, allowing engineers to predict thermal protection system (TPS) failure points before they occur. The solver supports both **Eulerian and Lagrangian meshless methods** for irregular vehicle geometries.

### 2. Probabilistic Dispersion Engine with Monte Carlo Sensitivity Analysis
The tool includes a built-in **stochastic dispersion engine** that runs Monte Carlo simulations over atmospheric density, wind shear, and ballistic coefficient variations. Each simulation outputs a 3σ footprint ellipse, debris fragmentation probability, and impact time windows. The engine automatically identifies the most sensitive parameters using Sobol indices, enabling engineers to prioritize uncertainty reduction efforts.

### 3. Real-Time Telemetry Integration via gRPC Streaming
artemisreentry exposes a **gRPC streaming API** that ingests live telemetry from ground stations or onboard flight computers. It performs real-time state estimation using an Unscented Kalman Filter (UKF) and replans the trajectory on the fly. The system can trigger automated alerts when the predicted impact zone shifts beyond predefined safety thresholds, making it suitable for both mission control and autonomous reentry vehicles.

## How to Use

### 1. Define Vehicle and Mission Parameters in YAML
Begin by creating a YAML configuration file that specifies the vehicle's mass, geometry, TPS material properties, and initial orbital elements. For example:

```yaml
vehicle:
  mass: 4500  # kg
  nose_radius: 0.5  # m
  tps_material: "PICA-X"
  ballistic_coefficient: 120  # kg/m²
mission:
  initial_altitude: 120000  # m
  initial_velocity: 7800  # m/s
  flight_path_angle: -1.2  # degrees
```

Validate the file using the built-in schema checker:  
`artemisreentry validate config.yaml`

### 2. Execute High-Fidelity Trajectory Simulation
Run the simulation with the following command, specifying the desired output granularity and solver type:

```bash
artemisreentry simulate config.yaml --solver=eulerian --output=trajectory.csv --step=0.01
```

The tool will output a CSV file with timestamped altitude, velocity, heat flux, and deceleration profiles. To visualize the trajectory in 3D, pipe the output into the integrated viewer:

```bash
artemisreentry simulate config.yaml | artemisreentry view --render
```

### 3. Perform Probabilistic Impact Footprint Analysis
To generate a probabilistic impact zone, run the Monte Carlo engine with 10,000 iterations:

```bash
artemisreentry montecarlo config.yaml --iterations=10000 --output=footprint.json
```

The resulting JSON file contains a GeoJSON-compatible polygon of the 95% confidence ellipse, along with a breakdown of fragmentation scenarios. Use the `--plot` flag to generate an interactive HTML map of the dispersion footprint.

## Best Practices

### 1. Always Validate Atmospheric Models Against Local Sounding Data
The default atmospheric model (NRLMSISE-00) is accurate for global averages, but local density variations can shift the impact footprint by several kilometers. Before running Monte Carlo simulations, **ingest real-time radiosonde or satellite occultation data** using the `--atmosphere custom` flag. This is especially critical for missions targeting recovery zones over land or near populated areas.

### 2. Use Incremental Solver Refinement for Time-Critical Decisions
When operating under tight mission timelines (e.g., during a deorbit burn), start with the **continuum-flow solver** for rapid convergence, then switch to the **transitional solver** only when the vehicle passes below 90 km altitude. This reduces simulation time by up to 40% while maintaining sub-100-meter accuracy at the point of impact. Configure this behavior in the YAML file under `solver.strategy: adaptive`.
