# artemisreentry - ART_019

## Overview

**artemisreentry** is a precision-engineered toolkit for modeling, simulating, and optimizing spacecraft reentry trajectories under the Artemis program framework. It bridges the gap between orbital mechanics and real-time atmospheric reentry analysis, enabling developers and mission engineers to run high-fidelity return scenarios without spinning up a full STK or GMAT stack. Built for speed, modularity, and reproducibility, it’s the go-to CLI and API layer for reentry logic in modern space ops pipelines.

---

## Core Features

### 1. Multi‑Phase Reentry Profile Engine  
Define reentry in distinct phases—deorbit burn, coast, atmospheric interface, descent, and landing—each with its own configurable drag model, heat flux constraints, and guidance law. The engine auto‑solves for g‑load limits and skip‑skip boundaries, outputting state vectors at configurable cadence.

### 2. Real‑Time Aerothermal Margin Calculator  
Integrates with NASA’s 1962/1976 Standard Atmosphere models (plus user‑defined custom profiles) to compute stagnation point heating, convective and radiative flux, and TPS margin in real time. No more guessing whether your heatshield holds—it outputs pass/fail with safety factor overlays.

### 3. Closed‑Loop Guidance Simulator with Monte Carlo Injection  
Inject dispersions across initial state, atmospheric density, drag coefficient, and wind profiles. The guidance module runs a closed‑loop predictor‑corrector (based on Apollo‑era algorithms updated for Orion‑class vehicles) and logs every iteration. Perfect for sensitivity analysis and robustness testing before flight.

## How to Use

### 1. Configure a Reentry Mission Profile  
Create a `mission.toml` file with initial orbital elements, vehicle mass, reference area, and desired landing site coordinates. Use the `--profile` flag to load it:  
```bash
artemisreentry run --profile mission.toml --output telemetry.json
```  
The engine validates all constraints (e.g., entry flight path angle within ±0.5°) and warns on any out‑of‑bound parameters before simulation starts.

### 2. Run a Monte Carlo Dispersion Sweep  
Launch a 10,000‑run batch to evaluate robustness under uncertainty:  
```bash
artemisreentry mc-sweep --profile mission.toml --runs 10000 --seed 42 --output-dir ./mc_results
```  
Each run logs the full state history, max heat rate, peak deceleration, and landing ellipse. Use `--parallel` to distribute across cores or a Dask cluster.

### 3. Visualize Trajectory and Margins  
Generate an interactive HTML report with 3D trajectory, ground track, and aerothermal margin charts:  
```bash
artemisreentry visualize --input telemetry.json --output report.html --style dark
```  
Hover over any point to see time‑stamped state vectors, dynamic pressure, and TPS margin. Export as PNG/PDF for documentation or peer review.

## Best Practices

- **Always pair with a validated atmosphere model** — Use the built‑in 1976 Standard Atmosphere for nominal runs, but switch to a custom profile (e.g., from MSIS‑2.0 or NRLMSISE‑00) when simulating off‑nominal or seasonal conditions. The default model can mask density variations that shift landing ellipses by kilometers.

- **Run a sensitivity scan on entry flight path angle (EFPA)** — A 0.1° change in EFPA can alter peak heat flux by 15–20%. Use `artemisreentry mc-sweep` with a narrow EFPA distribution (±0.3°) to map the safe corridor before locking in your trajectory. This alone can save weeks of manual iteration.
