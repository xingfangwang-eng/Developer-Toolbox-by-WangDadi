# artemisreentry - ART_002

## Overview

**Artemis Reentry** is a next-gen simulation engine purpose-built for modeling atmospheric reentry trajectories of spacecraft returning from lunar or deep-space missions. Unlike generic physics solvers, this tool ingests real-time telemetry formats (CCSDS, XML-TLM, raw CSV) and outputs high-fidelity thermal, aerodynamic, and structural stress predictions optimized for Orion-class vehicles and future lander architectures.

## Core Features

### 1. Multi-Phase Trajectory Solver with Adaptive Mesh Refinement
The solver dynamically switches between hypersonic, supersonic, and subsonic regimes without manual intervention. It uses a hybrid Runge-Kutta + implicit Euler integrator that self-tunes step size based on atmospheric density gradients and vehicle ballistic coefficient. This eliminates the "stiff solver" failures common in standard reentry models when transitioning through the ionospheric blackout zone.

### 2. Real-Time Ablative Thermal Protection System (TPS) Degradation Mapping
Instead of relying on static heat flux tables, Artemis Reentry couples a finite-difference thermal conduction model with a charring ablator submodule. It outputs time-stamped 3D heat maps of the heatshield surface, showing recession rates, bondline temperatures, and residual strength margins. This is critical for validating PICA-X or SLA-561V performance under off-nominal entry angles.

### 3. Closed-Loop Guidance, Navigation, and Control (GNC) Emulator
The tool includes a built-in GNC block that simulates bank-angle modulation, drag-based energy management, and parachute deployment sequencing. You can inject custom guidance laws (e.g., Apollo-derived "skip entry" or Mars EDL-inspired "guided lifting entry") and observe how the vehicle's control authority degrades as dynamic pressure spikes. This enables pre-flight verification of flight software without tying up a hardware-in-the-loop rig.

## How to Use

### Step 1: Configure the Mission Profile and Vehicle Model
Start by defining the entry interface (EI) state vector: altitude (typically 120 km), velocity (11.2 km/s for lunar return), flight-path angle (-6.0° to -5.5° for nominal), and vehicle mass properties. Use the `config/mission.yaml` file to set the atmospheric model (NRLMSISE-00 or GRAM-2016) and TPS material properties. Example snippet:

```yaml
atmosphere:
  model: "NRLMSISE-00"
  f107: 150
  ap: 15
vehicle:
  mass_kg: 8500
  reference_area_m2: 16.5
  ballistic_coefficient: 320
  tps_material: "PICA-X"
  initial_tps_thickness_mm: 45
```

Validate the input by running `artemisreentry validate --profile mars_return` to catch unit mismatches or out-of-range parameters before the solver kicks in.

### Step 2: Run the Multi-Phase Simulation with Live Monitoring
Execute the simulation using `artemisreentry run --profile lunar_return --output ./results/`. The solver will stream real-time metrics to stdout: current Mach number, stagnation point heat flux (W/cm²), dynamic pressure (kPa), and TPS recession rate (mm/s). You can attach a WebSocket listener to visualize the trajectory in a browser via the built-in dashboard (`artemisreentry dashboard --port 8080`). If the solver detects an imminent structural failure (e.g., bondline temperature exceeding 300°C), it triggers a soft abort and saves a crash dump for forensic analysis.

### Step 3: Post-Process and Export Engineering Reports
After the simulation completes, generate a comprehensive report using `artemisreentry report --format pdf --output ./report.pdf`. The report includes:
- Time-history plots of heat flux, deceleration (g-load), and TPS erosion.
- A "missed landing zone" probability map if the parachute deployment sequence failed.
- A comparison table against NASA's DRA 5.0 reference trajectories.
- A raw HDF5 archive of all state vectors for custom analysis in MATLAB or Python.

For team collaboration, export the results to a shared S3 bucket or Azure Blob Storage with `artemisreentry export --cloud aws --bucket artemis-reentry-runs`.

## Best Practices

### 1. Always Run a Sensitivity Sweep on Flight-Path Angle
The entry flight-path angle (EFPA) is the single most sensitive parameter in reentry dynamics. A deviation of just 0.1° can shift peak heating by 15% or cause the vehicle to skip out of the atmosphere entirely. Use the built-in `artemisreentry sweep --parameter efpa --range -6.5:-5.0 --steps 20` to generate a Pareto front of safe entry corridors. This is standard practice at NASA JSC for mission assurance reviews.

### 2. Validate TPS Margins Against Off-Nominal Atmospheric Conditions
Don't rely solely on standard atmosphere models. Run at least three scenarios using perturbed density profiles (e.g., +20% density at 70 km altitude to simulate a winter polar vortex). Use `artemisreentry run --atmosphere perturbed --density_perturbation 1.2` to stress-test the TPS. If the bondline temperature exceeds the material's safe operating limit (e.g., 250°C for PICA-X), you need to either increase TPS thickness or adjust the entry profile. This is how SpaceX validated Crew Dragon's heatshield for the Demo-2 mission.
