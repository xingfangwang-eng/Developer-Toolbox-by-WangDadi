## What is artemisreentry?

**artemisreentry** is a domain-specific, lightweight yet extensible framework for modeling, simulating, and validating atmospheric reentry trajectories for lunar and deep-space return missions. It bridges the gap between high-fidelity physics engines and rapid prototyping for mission planners, aerospace engineers, and space software developers.

Unlike generic trajectory tools, artemisreentry is **optimized for the Artemis program’s unique return corridors**, offering pre-built ballistic coefficient profiles, real-time heat flux estimation, and reusable entry state validation.

## Core Features

### 1. Real-Time Skip-Entry Phase Detection
Automatically identifies and segments the trajectory into **skip, glide, and terminal descent phases** based on dynamic pressure and aerodynamic heating thresholds. This enables granular performance tuning for each flight regime without manual intervention.

### 2. Integrated Thermal Protection Margin Calculator
Leverages a custom **TPS (Thermal Protection System) erosion model** calibrated against NASA’s Orion MPCV data. It outputs real-time ablation depth and safety margins for user-defined materials (e.g., Avcoat, PICA-X), allowing engineers to validate TPS design against worst-case reentry conditions.

### 3. Multi-Constraint Trajectory Optimizer
Uses a **direct collocation method with NLP (Nonlinear Programming)** to satisfy simultaneous constraints: peak heat flux ≤ 120 W/cm², peak g-load ≤ 4.5 g, and landing ellipse semi-major axis ≤ 20 km. The optimizer outputs a deterministic control profile (bank angle and angle of attack) readable by flight software.

## How to Use

### Step 1: Define Entry State and Vehicle Parameters
Create a JSON configuration file specifying initial conditions (e.g., altitude 122 km, velocity 11.2 km/s, flight path angle -1.5°) and vehicle properties (mass, reference area, aerodynamic coefficients). Use the `artemisreentry validate-config` CLI command to check for missing or out-of-range parameters before simulation.

### Step 2: Run a Constrained Reentry Simulation
Execute `artemisreentry simulate --config entry_config.json --output trajectory.csv`. The engine will automatically perform a **multi-phase integration** using a variable-step Runge-Kutta (RK8) solver with event detection for skip-entry boundaries. The output includes time-stamped state vectors, heat flux, and dynamic pressure at 0.1-second resolution.

### Step 3: Analyze Thermal and Trajectory Metrics
Use the built-in post-processing module: `artemisreentry analyze --input trajectory.csv --plot all`. This generates a **unified dashboard** with overlaid plots of altitude vs. velocity, heat flux vs. time, and g-load vs. Mach number. For deeper inspection, export the dataset as a NumPy `.npy` file for custom Python scripts or MATLAB integration.

## Best Practices

### 1. Use Monte Carlo Dispersion Runs for Robustness
Do not rely on a single deterministic simulation. Use the `artemisreentry montecarlo --config entry_config.json --runs 1000` command to sample initial state uncertainties (e.g., ±3σ in velocity and flight path angle). Analyze the 95th percentile of peak heat flux and landing ellipse to ensure the design meets safety margins under realistic dispersion.

### 2. Validate Against Reference Missions Before Customization
Before tuning parameters for your mission, run the built-in **Orion EFT-1 benchmark** using `artemisreentry benchmark orion_eft1`. This verifies that your local installation reproduces known trajectory data within 1% error. Only then should you modify vehicle mass or aerodynamic coefficients—this prevents model drift and ensures traceability in peer-reviewed analysis.
