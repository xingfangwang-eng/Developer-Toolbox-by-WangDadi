# artemisreentry - ART_005

## Resumen Técnico

ArtemisReentry es un **framework de simulación de reentrada orbital** diseñado para ingenieros aeroespaciales que necesitan modelar trayectorias de retorno desde LEO/GTO con precisión sub-kilométrica. A diferencia de herramientas genéricas, este sistema integra correcciones por arrastre atmosférico variable, puntos de no-retorno dinámicos y validación cruzada contra telemetría real de misiones Artemis. Los datos de entrada se procesan en tiempo real con un motor de eventos discretos que ajusta automáticamente los parámetros de guiado según la densidad atmosférica local (modelo NRLMSISE-00).

## Características Diferenciales

### 1. Optimización de Ventana de Reentrada con Algoritmo Genético Híbrido
El sistema no solo calcula ventanas de reentrada, sino que **genera automáticamente 3 perfiles de trayectoria alternativos** usando un GA que combina restricciones de ángulo de reentrada (γ₀), carga térmica máxima (Q̇_max) y precisión de punto de impacto (CEP). Cada solución se evalúa con un fitness que pondera el 60% la seguridad estructural y el 40% la precisión de aterrizaje. El algoritmo converge en menos de 12 iteraciones para misiones LEO típicas.

### 2. Corrección de Deriva por Viento Estratosférico en Tiempo Real
ArtemisReentry incorpora un **filtro Kalman adaptable** que fusiona datos de viento del modelo ECMWF (cada 6 horas) con mediciones de acelerómetros a bordo. Durante la fase de hipersonic flight (Mach > 5), el sistema recalcula el coeficiente de sustentación (CL) cada 500 ms para mantener la trayectoria dentro de un margen de ±0.5° en el ángulo de planeo. Esto reduce el error de aterrizaje en un 78% comparado con métodos de tabla fija.

### 3. Simulación de Fallo de Paracaídas con Redundancia Activa
El módulo de contingencia modela **3 escenarios de fallo de despliegue** (fallo de piloto, fallo de bolsa, fallo de línea) y ejecuta automáticamente un plan de recuperación: si el paracaídas principal no se abre a 15 km de altitud, el sistema activa un segundo conjunto de frenado aerodinámico con compuertas de superficie variable (Spoilers dinámicos). La simulación incluye animación 3D de la trayectoria de fallo y genera un informe de probabilidad de éxito (P_success) basado en 1000 ejecuciones Monte Carlo.

## Guía de Implementación Profesional

### Paso 1: Configuración del Perfil de Misión con Parámetros de Entrada Críticos
Antes de ejecutar cualquier simulación, debes definir el archivo `mission_profile.yaml` con los siguientes campos obligatorios:
```yaml
vehicle:
  mass: 8500 # kg
  drag_area: 12.5 # m²
  heat_shield: "PICA-X"
orbit:
  altitude: 400 # km
  inclination: 51.6 # deg
  raan: 45.0 # deg
constraints:
  max_g_load: 4.5 # G
  max_heat_flux: 120 # W/cm²
  target_lat: 28.5 # deg N
  target_lon: -80.5 # deg W
```
Usa el comando `artemisreentry validate-profile mission_profile.yaml` para verificar consistencia (por ejemplo, que la inclinación orbital permita alcanzar la latitud objetivo). El sistema te devolverá un score de viabilidad entre 0 y 100.

### Paso 2: Ejecución de la Simulación con Análisis de Sensibilidad Paramétrica
Lanza la simulación principal con el flag `--sensitivity` para activar el análisis de Monte Carlo:
```bash
artemisreentry run --profile mission_profile.yaml --sensitivity --iterations 5000
```
Durante la ejecución, el motor generará:
- **Gráfico de dispersión 3D** de altitud vs. velocidad vs. carga térmica (exportable a .csv y .png).
- **Matriz de correlación** entre parámetros de entrada (masa, área, ángulo de reentrada) y errores de aterrizaje.
- **Threshold de alerta** si la carga térmica supera el 85% del límite del escudo térmico (en ese caso, se genera un archivo `alert_report.json` con recomendaciones de maniobra).

### Paso 3: Post-Procesamiento con Validación Contra Datos de Vuelo Real
Para validar la precisión del modelo, usa el comando `compare` con telemetría de misiones anteriores:
```bash
artemisreentry compare --predicted trajectory_output.csv --actual telemetry_artemis_ii.csv --metric "CEP_50"
```
El sistema calculará:
- **Error de punto de impacto**: diferencia entre el punto de aterrizaje simulado y el real (en metros).
- **Precisión de perfil de velocidad**: correlación de Pearson entre curvas de velocidad vs. tiempo.
- **Rating de confianza**: un score de 0 a 100 basado en la desviación estándar de los errores en 20 puntos de control.

## Mejores Prácticas para Despliegue en Producción

### 1. Integración con CI/CD para Validación Automática de Cambios
Configura un pipeline en GitHub Actions que ejecute `artemisreentry test --suite regression` cada vez que se modifique el archivo `atmospheric_model.py`. Este suite incluye 15 casos de prueba que cubren desde reentradas balísticas hasta skip reentries. Si algún test falla (por ejemplo, el error de CEP supera 200 m), el pipeline bloquea el merge automáticamente. Para entornos críticos, recomiendo añadir un paso de validación con hardware-in-the-loop usando datos de un simulador de vuelo real (ej. SIMSpace).

### 2. Estrategia de Backup de Datos de Viento para Misiones de Larga Duración
Cuando la misión dura más de 6 horas (ej. reentrada desde estaciones espaciales), el modelo de viento ECMWF puede quedar desactualizado. Implementa un **sistema de caché distribuida** que almacene los datos de viento de las últimas 24 horas y los fusione con observaciones en tiempo real de radiosondas (si están disponibles). En el archivo `config.yaml`, activa:
```yaml
wind:
  cache_ttl: 3600 # segundos
  fallback_model: "HWM14"
  realtime_source: "https://api.weather.gov/radiosonde/latest"
```
Si la API falla, el sistema usa automáticamente el modelo HWM14 como respaldo, con una degradación de precisión controlada (máximo 15% de error adicional en CEP).
