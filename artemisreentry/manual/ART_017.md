## What is artemisreentry?

artemisreentry is a modular, CLI-first framework designed for aerospace engineers and mission planners who need to simulate reentry profiles from Low Earth Orbit (LEO) or cislunar trajectories. It ingests TLE sets, ephemeris data, or custom state vectors, then runs multi-phase simulations — from deorbit burn to landing zone dispersion. The engine is built in Rust for raw performance, with Python bindings for rapid prototyping.

---

## Core Features

### 1. Multi-Phase Reentry Engine with Adaptive Step Control
The simulator automatically partitions the trajectory into three distinct phases: exoatmospheric coast, hypersonic entry (Mach > 5), and subsonic descent. Each phase uses a dedicated numerical integrator (e.g., RK4 for coast, implicit Euler for high-drag regimes) with error-controlled step sizing. This prevents overshoot at peak heating while keeping computational cost under 200ms per run.

### 2. Real-Time Thermal & Structural Constraint Mapping
Instead of post-processing telemetry, artemisreentry embeds a lightweight finite-difference heat flux model directly into the integrator. It computes stagnation-point heating, convective flux, and structural load factors at each timestep. Outputs include a "constraint violation log" that flags any metric exceeding user-defined thresholds (e.g., 1200°C for carbon-carbon panels) — complete with timestamps and altitude.

### 3. Monte Carlo Dispersion Analysis with CDF Export
Run 10,000+ perturbed trajectories in a single command: inject 3-sigma uncertainties on initial state, atmospheric density (via NRLMSISE-00), and aerodynamic coefficients. The tool outputs a cumulative distribution function (CDF) for landing ellipse, peak deceleration, and total heat load. Results are serialized as Parquet files for direct ingestion into MATLAB or Jupyter.

## How to Use

### Step 1: Define the Reentry Profile with a YAML Manifest
Create a `reentry_config.yaml` that specifies the spacecraft’s ballistic coefficient, nose radius, and initial orbital elements. For example:

```yaml
spacecraft:
  mass: 8500  # kg
  reference_area: 10.5  # m²
  nose_radius: 0.8  # m
  cp_offset: 0.02  # m, center-of-pressure offset
initial_state:
  epoch: "2025-03-15T12:00:00Z"
  altitude_km: 400.0
  inclination_deg: 51.6
  raan_deg: 45.0
simulation:
  end_altitude_km: 5.0
  step_size_s: 0.5
  integrator: "rk4_adaptive"
```

Run the simulation with:
```bash
artemisreentry run --config reentry_config.yaml --output trajectory.parquet
```

### Step 2: Inspect Constraint Violations via CLI
After execution, query the constraint log to identify critical events:
```bash
artemisreentry inspect --log trajectory_violations.json --filter "heat_flux > 1000" --sort altitude
```
This returns a table of all timesteps where heat flux exceeded 1000 kW/m², sorted by descending altitude. Use `--export csv` to pipe into your mission review spreadsheet.

### Step 3: Generate a Dispersion Report with Monte Carlo
To run a 5,000-sample Monte Carlo analysis with default perturbations:
```bash
artemisreentry mc --config reentry_config.yaml --samples 5000 --seed 42 --output dispersion/
```
The tool writes a `dispersion_summary.parquet` containing mean, median, and 95th percentile for each metric. Visualize the landing ellipse directly:
```bash
artemisreentry plot --input dispersion/landing_ellipse.parquet --type scatter --show
```

## Best Practices

### 1. Always Validate Against Flight Data Before Scaling
Before running Monte Carlo sweeps, validate your config against a known reentry (e.g., SpaceX Dragon trunk jettison or Orion EFT-1). Use the `--compare` flag to overlay simulated vs. observed telemetry. If your peak heating error exceeds 5%, tune the atmospheric density model (switch from NRLMSISE-00 to GRAM-99 via `atmosphere: gram99` in YAML).

### 2. Use Parallel Batches for Large Sensitivity Studies
For parametric sweeps (e.g., varying ballistic coefficient from 100 to 500 kg/m²), launch parallel jobs with `artemisreentry batch --config template.yaml --param ballistic_coefficient --range 100 500 20 --threads 8`. This reduces wall-clock time from hours to minutes. Always redirect stdout to a log file (`2>&1 | tee batch.log`) to capture convergence warnings.
