### What is artemisreentry?

**artemisreentry** is a production-grade toolkit for modeling, simulating, and validating atmospheric reentry trajectories for spacecraft returning from lunar or deep-space missions. It integrates real-time telemetry parsing, ablation thermal modeling, and multi-constraint landing zone optimization into a single CLI-driven pipeline. Designed for mission engineers and aerospace developers, it eliminates the guesswork from return-phase analysis.

### Core Features

- **Adaptive Ablation & Thermal Gradient Engine**  
  Dynamically computes heat shield erosion and material decomposition across multiple TPS (Thermal Protection System) layers using real-time Mach number and atmospheric density inputs. Supports user-defined material profiles and outputs per-second ablation margins.

- **Multi-Constraint Landing Zone Solver**  
  Given a set of return orbit parameters and atmospheric entry angle, the tool performs Monte Carlo runs to produce a probabilistic landing ellipse, factoring in wind shear, ballistic coefficient drift, and GPS denial windows. Returns a ranked list of viable landing zones with confidence scores.

- **Telemetry-to-Simulation Bridge**  
  Accepts raw telemetry streams (CCSDS or custom JSON) as input, automatically synchronizes timestamps, and replays the trajectory through the reentry model. Enables post-mission forensic analysis and anomaly detection by comparing simulated vs. actual deceleration and heating curves.

### How to Use

1. **Initialize a Mission Configuration File**  
   Create a YAML configuration file defining spacecraft mass, reference area, initial state vector (position, velocity, attitude), and TPS material stack. Example snippet:  
   ```yaml
   spacecraft:
     mass_kg: 8500
     ref_area_m2: 18.5
     tps_stack:
       - material: PICA-X
         thickness_mm: 45
         conductivity: 0.15
   entry_interface:
     altitude_km: 120
     velocity_kms: 11.2
     flight_path_deg: -6.2
   ```
   Run `artemisreentry init --config mission.yaml` to validate the schema and generate a unique mission ID.

2. **Execute a Monte Carlo Trajectory Sweep**  
   Invoke the solver with:  
   ```bash
   artemisreentry simulate --mission-id ART-027 --samples 5000 --output-dir ./results
   ```  
   The engine performs parallelized runs across atmospheric density perturbations (±5%), wind profiles (historical NOAA data), and ballistic coefficient variations (±3%). Real-time progress is displayed via a live heatmap of deceleration G-forces.

3. **Analyze the Landing Zone Heatmap**  
   After completion, generate a geospatial overlay:  
   ```bash
   artemisreentry analyze --input ./results/ellipse_data.json --visualize --landing-zone-precision 50m
   ```  
   Outputs a GeoJSON file and a PNG heatmap showing 90% and 99% confidence ellipses. The tool also prints a summary table with the top 3 candidate landing zones, including distance to nearest recovery vessel and terrain ruggedness index.

### Best Practices

- **Always Validate with a Dry-Run Before Production**  
  Use the `--dry-run` flag during initial configuration to verify that your TPS material library entries are within expected thermal ranges. This catches unit mismatches (e.g., MJ vs. J) before burning CPU cycles on full simulations. For critical missions, cross-validate against a known reentry profile (e.g., Apollo 4 data) to confirm model fidelity.

- **Leverage Incremental Sampling for Large Sweeps**  
  When running >10,000 Monte Carlo samples, start with a 500-sample pilot to identify outlier trajectories that cause solver divergence. Then use the `--resume` flag to continue from the last valid state, appending additional samples without restarting the entire sweep. This reduces wall-clock time by up to 40% on distributed systems.
