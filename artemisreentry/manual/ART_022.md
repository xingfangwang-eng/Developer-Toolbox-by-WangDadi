# artemisreentry - ART_022

## Overview

`artemisreentry` is a precision-engineered toolkit for modeling, simulating, and analyzing spacecraft reentry trajectories, with a specific focus on the Artemis mission profile. It bridges the gap between raw orbital mechanics data and actionable engineering insights, enabling developers and mission analysts to validate reentry corridors, thermal loads, and landing dispersion in real time.

**Keywords:** Artemis reentry, Space reentry tool, Space return analysis

---

## Core Features

### 1. **Multi-Phase Atmospheric Interface Engine**
Unlike generic reentry simulators, `artemisreentry` decouples the atmospheric interface into three distinct phases: exoatmospheric coast, hypersonic entry, and subsonic descent. Each phase uses its own adaptive timestep solver (Runge-Kutta-Fehlberg for coast, implicit Euler for dense atmosphere) to prevent numerical drift during high-dynamic regime changes. You can inject custom drag coefficient tables (`.cdb` format) or use the built-in US Standard 1976 atmosphere model.

### 2. **Monte Carlo Dispersion Profiler with Real-Time Covariance Mapping**
Run 10,000+ trajectory variants in under 30 seconds using GPU-accelerated Monte Carlo sampling. The profiler automatically generates a 3-sigma landing ellipse and overlays it on OpenStreetMap tiles or local GIS shapefiles. Each run logs per-second state vectors (position, velocity, heat flux) to a Parquet file, making post-mission analysis reproducible and queryable via DuckDB or Pandas.

### 3. **Thermal Protection System (TPS) Margin Calculator**
This isn’t just a peak-heating estimator. The TPS calculator models transient heat conduction through multi-layer insulation (MLI) and ablative materials (PICA, SLA-561V). It outputs a time-to-failure margin for each tile or blanket segment, factoring in re-radiation, catalytic heating, and surface roughness. You can export the thermal profile as a `.vtk` file for visualization in ParaView.

## How to Use

### Step 1: Define the Reentry Corridor
Create a `config.yaml` file with your initial state vector (ECI frame), ballistic coefficient, and desired landing site coordinates. Example snippet:

```yaml
mission: artemis_reentry_022
initial_state:
  epoch: "2025-03-15T12:00:00Z"
  position_km: [6578.0, 0.0, 0.0]
  velocity_kms: [0.0, 7.8, 0.0]
atmosphere: us1976
corridor:
  entry_angle_deg: -6.2
  cross_range_km: 120
```

Run `artemisreentry validate config.yaml` to check for constraint violations (e.g., entry angle exceeds structural limits).

### Step 2: Execute the Simulation
Launch a deterministic or probabilistic run:

```bash
artemisreentry run --config config.yaml --mode monte_carlo --samples 5000 --output ./results/
```

The CLI streams real-time delta-V corrections and heat flux peaks to stdout. Use the `--progress bar` flag to see estimated time to completion. For deterministic runs, append `--deterministic` to disable sampling.

### Step 3: Analyze the Output
Load the generated `trajectory.parquet` into your analysis pipeline:

```python
import duckdb
conn = duckdb.connect()
df = conn.execute("SELECT time_s, altitude_km, heat_flux_kwm2 FROM 'results/trajectory.parquet' WHERE altitude_km < 100").fetchdf()
```

Or visualize the landing ellipse directly:

```bash
artemisreentry plot --input results/monte_carlo_ellipse.geojson --mapbox-token <your_token>
```

This overlays the 1/2/3-sigma ellipses on a satellite map, with waypoints for expected parachute deployment and splashdown.

## Best Practices

### 1. **Always Run a Sensitivity Sweep Before Final Trajectory Lock**
Don’t rely on a single nominal run. Use the built-in `sweep` subcommand to vary entry angle (±0.5°), ballistic coefficient (±10%), and atmospheric density (using a scale factor of 0.8–1.2). The output will rank each parameter’s impact on landing dispersion, helping you identify which variables need tighter control during the actual mission.

### 2. **Version-Control Your Config and Output Parquet Files**
Reentry analysis is iterative. Store every `config.yaml` and its corresponding `trajectory.parquet` in a Git LFS repository. Tag each run with the mission phase (e.g., `v1.2.0-preliminary`). This makes it trivial to roll back to a previous corridor if a later design change introduces unacceptable thermal margins. Pair with DVC to track data lineage across team members.
