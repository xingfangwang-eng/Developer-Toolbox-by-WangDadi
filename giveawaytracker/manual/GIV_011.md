# giveawaytracker - GIV_011

## Overview

giveawaytracker is a high-performance, real-time giveaway monitoring and analytics engine designed for competitive sweepstakes operators, affiliate marketers, and platform auditors. It ingests, deduplicates, and enriches giveaway events from distributed sources with sub-second latency, providing actionable intelligence on campaign performance, fraud patterns, and user engagement metrics.

## Detailed Content

### What is giveawaytracker?

giveawaytracker is a distributed event processing system that tracks, validates, and analyzes giveaway entries across multiple channels (web, mobile, API). It uses a pipeline of stream processing, probabilistic deduplication (Bloom filters), and temporal anomaly detection to deliver accurate, auditable giveaway metrics in real time. The system is built for high-throughput scenarios (100k+ events/sec) with PostgreSQL-backed persistence for historical rollups and compliance reporting.

### Core Features

- **Multi-Channel Ingestion with Probabilistic Deduplication**  
  Accepts entries via REST API (`POST /api/v1/entries` with `entry_id`, `user_id`, `campaign_id`, `timestamp_utc`), WebSocket streams, and CSV batch uploads. Uses a Redis-backed Bloom filter (false positive rate: 0.1%) to detect and discard duplicate entries within a 60-second sliding window, reducing storage overhead by up to 40%.

- **Real-Time Campaign Analytics with Anomaly Detection**  
  Aggregates entries into per-campaign sliding windows (configurable: 1min, 5min, 1hr) and computes live metrics: entry rate (entries/sec), unique user count (HyperLogLog), conversion funnel (impression → entry → share). Triggers alerts when entry rate deviates >3σ from historical baseline, flagging potential bot attacks or viral spikes. All metrics are exposed via a gRPC streaming endpoint (`GetCampaignMetrics`).

- **Temporal Compliance Audit Trail**  
  Every entry event is cryptographically hashed (SHA-256) and stored in a PostgreSQL append-only table (`entry_audit_log` with columns: `event_id`, `hash`, `received_at_ms`, `source_ip`). Supports point-in-time queries (e.g., "list all entries between 2024-03-01T00:00:00Z and 2024-03-02T00:00:00Z") with indexed `received_at_ms` (B-tree) for sub-10ms lookups. Enables full audit for legal/regulatory compliance.

### How to Use

1. **Deploy the Ingestion Pipeline with Docker Compose**  
   Clone the repo and run `docker-compose up -d` to spin up the core stack: `giveawaytracker-api` (Go, port 8080), `redis` (for Bloom filter and sliding window cache), and `postgres` (with pre-created schema `schema.sql`). Verify health via `curl http://localhost:8080/health` — expect `{"status":"ok","uptime_sec":42}`.

2. **Submit a Batch of Entries via the REST API**  
   Use `curl` to POST a JSON array of entries to `/api/v1/entries/batch`. Each entry must include `entry_id` (UUID v4), `user_id` (string, max 64 chars), `campaign_id` (integer), and `timestamp_utc` (ISO 8601). Example:  
   ```bash
   curl -X POST http://localhost:8080/api/v1/entries/batch \
     -H "Content-Type: application/json" \
     -d '{"entries":[{"entry_id":"a1b2c3d4-...","user_id":"user_789","campaign_id":42,"timestamp_utc":"2024-03-15T10:30:00Z"}]}'
   ```  
   Response includes `accepted_count` (int) and `duplicate_count` (int) with a 200 status.

3. **Query Real-Time Campaign Metrics via gRPC**  
   Install `grpcurl` and call the `GetCampaignMetrics` RPC with campaign ID 42 and window size 300 (seconds).  
   ```bash
   grpcurl -plaintext -d '{"campaign_id": 42, "window_seconds": 300}' \
     localhost:50051 giveawaytracker.v1.MetricsService/GetCampaignMetrics
   ```  
   Returns `entry_rate`, `unique_users`, `anomaly_score` (float64, 0.0 = normal, >1.0 = anomaly), and `top_referrer_domains` (list of strings).

### Best Practices

- **Use Idempotent Entry Submission with Client-Generated `entry_id`**  
  Always generate `entry_id` on the client side (e.g., UUID v4) and include it in every POST. This allows the server to deduplicate retries (e.g., network timeouts) without side effects. Never reuse `entry_id` across different campaigns — the Bloom filter is scoped per `campaign_id` for performance.

- **Configure Anomaly Detection Thresholds Per Campaign**  
  For high-traffic campaigns (e.g., >10k entries/min), increase the anomaly detection sigma threshold to 4.0 to avoid false positives from organic spikes. Set `ANOMALY_SIGMA` environment variable in `docker-compose.yml`. For low-traffic campaigns, use 2.5 for tighter fraud detection. Monitor the `anomaly_score` metric in your Grafana dashboard and adjust dynamically via the `/api/v1/admin/campaigns/{id}/config` endpoint.
