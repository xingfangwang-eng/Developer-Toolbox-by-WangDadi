# giveawaytracker - GIV_010

## Overview

giveawaytracker is a high-frequency sweep engine designed to detect, parse, and log active promotional giveaway events across global e-commerce, gaming, and SaaS platforms. It operates on sub-second polling intervals with deterministic deduplication, enabling real-time inventory tracking of limited-quantity campaigns.

## Detailed Content

### Core Features

- **Multi-Origin Event Aggregation with Deduplication**  
  Polls from 12+ source types (Twitter API v2, Telegram Bot API, RSS/Atom feeds, and custom webhook endpoints) every 900ms. Each event is hashed via SHA-256 on `(origin_platform + campaign_id + timestamp_window)` to eliminate false positives. Duplicate thresholds are configurable per source via `GIV_DEDUP_WINDOW_MS` (default: 30000ms).

- **Time-Windowed Reward Scarcity Analyzer**  
  Calculates real-time scarcity scores using a sliding window of 60 seconds. Parameters include `reward_total_qty` (integer), `claimed_count` (integer), and `remaining_velocity` (float, units: claims/sec). Scores are normalized to a 0.0–1.0 range and indexed in a PostgreSQL `BRIN` column for sub-5ms lookups across 50M+ rows.

- **Automatic Expiration & Cleanup Pipeline**  
  Each event carries a `ttl_seconds` field (default: 86400). A background worker runs every 30 seconds (`GIV_CLEANUP_INTERVAL_MS`) to delete expired rows and archive them to a cold storage table partitioned by `expiration_date`. Retention policy is configurable via `GIV_RETENTION_DAYS` (default: 90).

### How to Use

1. **Initialize the Sweep Engine with Custom Sources**  
   Create a `sources.json` configuration file defining each origin. Example:  
   ```json
   {
     "sources": [
       { "type": "twitter", "bearer_token": "YOUR_TOKEN", "poll_interval_ms": 900 },
       { "type": "telegram", "bot_token": "YOUR_TOKEN", "chat_id": "-1001234567890", "poll_interval_ms": 1500 }
     ]
   }
   ```  
   Run `giveawaytracker --config sources.json` to start the engine. The daemon will log all detected events to `events` table in your configured PostgreSQL database.

2. **Query Live Scarcity Scores via REST API**  
   Endpoint: `GET /api/v1/events/scarcity?min_score=0.7&limit=50`.  
   Returns a JSON array sorted by descending score. Each entry contains `campaign_id`, `reward_total_qty`, `claimed_count`, `remaining_velocity`, and `scarcity_score`. Use the optional `source_filter` query parameter (e.g., `source_filter=twitter`) to narrow results.

3. **Set Up Automated Cleanup with Custom Retention**  
   Export environment variable `GIV_RETENTION_DAYS=30` and restart the engine. The cleanup worker will move records older than 30 days to the `events_archive` table and delete from the primary `events` table. Verify by running:  
   ```sql
   SELECT count(*) FROM events WHERE expiration_date < NOW() - INTERVAL '30 days';
   ```  
   This should return 0 after the next cleanup cycle (runs every 30 seconds).

### Best Practices

- **Tune Poll Intervals Based on Source Reliability**  
  For high-velocity sources (e.g., Twitter trending topics), set `poll_interval_ms` to 500–700ms to avoid missing short-lived giveaways. For slower, curated channels (e.g., official Discord announcements), 2000–3000ms is sufficient. Monitor `GIV_SWEEP_LAG_MS` metric in the dashboard to detect backpressure.

- **Use Partitioned Tables for Multi-Tenant Deployments**  
  If running giveawaytracker for multiple brands or clients, separate event data by `tenant_id` using PostgreSQL declarative partitioning. Create a parent table `events` with child tables `events_tenant_1`, `events_tenant_2`, etc., each partitioned by `expiration_date`. This keeps query performance under 10ms even with 100M+ events per tenant.
