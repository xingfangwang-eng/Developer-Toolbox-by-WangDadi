# giveawaytracker - GIV_006

## Overview

giveawaytracker is a high-throughput, event-driven analytics engine purpose-built for monitoring, deduplicating, and aggregating giveaway participation streams across Discord, Twitter, Gleam, and custom webhooks. It ingests real-time entry logs, computes fraud probability scores via heuristic + ML hybrid models, and exposes a RESTful API for dashboarding and payout reconciliation. The system is designed to handle 10,000+ concurrent entries per second with sub-50ms latency on deduplication lookups.

## Core Features

- **Multi-Platform Entry Ingestion & Real-Time Deduplication**  
  Each incoming entry is parsed through a configurable adapter layer (Discord `messageCreate`, Twitter `tweet.create`, Gleam webhook). A Redis-backed Bloom filter (with 99.9% FP tolerance) checks for duplicate user IDs, IPs, and wallet addresses within a sliding 24-hour window. Duplicates are flagged and excluded from the final winner pool with a `dedup_reason` field logged to PostgreSQL for audit trails.

- **Probabilistic Fraud Scoring Engine**  
  Every entry is scored on a 0.0–1.0 scale using a LightGBM model trained on 500k+ labeled giveaway events. Features include: account age (seconds), follower/engagement ratio, time delta between entry and giveaway start (milliseconds), and device fingerprint entropy. Scores above a configurable threshold (default 0.85) trigger automatic suspension and push a `fraud_alert` event to a RabbitMQ queue for manual review.

- **Winner Selection & Payout Reconciliation API**  
  After the giveaway closes, the system runs a cryptographically seeded Fisher-Yates shuffle on the filtered entry pool. The `GET /giveaways/:id/winners` endpoint returns a list of winners with `selection_round`, `random_seed_hex`, and `payout_status`. A background job then matches each winner against an on-chain payout ledger (EVM-compatible via Web3.py), updating the `tx_hash` and `confirmed_at` columns in PostgreSQL. Failed payouts are retried up to 3 times with exponential backoff.

## How to Use

1. **Configure Ingest Adapters & Environment Variables**  
   Clone the repository and set the following in `.env`:  
   ```
   DISCORD_BOT_TOKEN=your_token_here
   TWITTER_BEARER_TOKEN=your_token_here
   REDIS_URL=redis://localhost:6379/0
   DATABASE_URL=postgresql://user:pass@localhost:5432/giveawaytracker
   FRAUD_MODEL_PATH=./models/lightgbm_fraud_v2.pkl
   WEB3_PROVIDER_URI=https://mainnet.infura.io/v3/your_project_id
   ```  
   Run `docker-compose up -d` to spin up Redis, PostgreSQL, RabbitMQ, and the main worker service.

2. **Create a Giveaway Event & Start Ingestion**  
   Use the CLI tool:  
   ```bash
   python manage.py create_giveaway \
       --platform discord \
       --channel_id 123456789 \
       --start_ts 2025-04-01T00:00:00Z \
       --end_ts 2025-04-03T00:00:00Z \
       --max_entries 5000 \
       --fraud_threshold 0.85
   ```  
   This registers the event in PostgreSQL and starts the ingest worker. Monitor live entry counts via `GET /giveaways/:id/stats` which returns `total_entries`, `unique_entries`, `fraud_flagged`, and `duplicates_removed` in JSON.

3. **Run Winner Selection & Trigger Payouts**  
   After the event ends, execute:  
   ```bash
   python manage.py select_winners --giveaway_id GIV_006 --winners_count 10
   ```  
   The system outputs a JSON array of winners with their `user_id`, `wallet_address`, and `selection_round`. Then run:  
   ```bash
   python manage.py payout --giveaway_id GIV_006 --token_contract 0x... --amount 100
   ```  
   This sends ERC-20 tokens via the configured Web3 provider and logs each transaction. Check payout status with `GET /giveaways/:id/payouts`.

## Best Practices

- **Always Pre-Warm the Bloom Filter**  
  Before a high-traffic giveaway (e.g., 50k+ entries in 1 hour), run the `ingest_warmup` script which loads historical user IDs and IPs from PostgreSQL into the Redis Bloom filter. This prevents false positives during the live event and reduces dedup latency from ~200ms to under 10ms.

- **Set Up Webhook Alerts for Fraud Spikes**  
  Configure a Slack or Discord webhook in `config/alerts.yaml` to receive notifications when the `fraud_flagged` rate exceeds 5% of total entries within a 5-minute rolling window. This allows your moderation team to pause the giveaway manually and review the flagged entries before the event ends, avoiding a payout to bots.
