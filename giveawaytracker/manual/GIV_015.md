# giveawaytracker - GIV_015

## Overview

giveawaytracker is a high-throughput, event-driven giveaway management and analytics engine designed for real-time campaign orchestration. It processes giveaway entries, verifies eligibility rules, and aggregates conversion metrics across distributed platforms with sub-100ms latency.

## Core Features

### 1. Multi-Platform Entry Deduplication & Validation Engine
- **Parameters**: `platform_type` (Twitter, Discord, Gleam), `user_fingerprint` (SHA-256 hash), `entry_timestamp` (Unix epoch in milliseconds)
- **Technical Specs**: Uses a Redis-backed sliding window counter (TTL configurable, default 3600s) to enforce per-user entry caps. Integrates with PostgreSQL via a write-ahead log (WAL) for persistent audit trails. Supports 10,000+ concurrent entry validations per second with atomic increment operations.

### 2. Real-Time Giveaway Funnel Analytics with Cohort Segmentation
- **Parameters**: `campaign_id` (UUID v4), `conversion_stage` (view, entry, share, purchase), `cohort_date` (ISO 8601)
- **Technical Specs**: Streams raw event data through Apache Kafka (partition count = 3 * broker nodes) into a ClickHouse materialized view. Calculates funnel conversion rates (e.g., entry-to-share ratio) with a 99.9th percentile latency under 50ms. Supports dynamic cohort slicing by `user_region` and `referrer_domain`.

### 3. Automated Winner Selection with Verifiable Randomness
- **Parameters**: `prize_pool_id` (BIGSERIAL), `winner_count` (integer, max 1000), `random_seed` (hex string, 64 chars)
- **Technical Specs**: Implements a commit-reveal scheme using a SHA-3 (Keccak-256) hash chain. The selection algorithm runs as a PostgreSQL stored procedure (`select_winners(prize_pool_id, winner_count, random_seed)`) to ensure on-chain verifiability. Outputs a JSON array of winner IDs with a proof hash, auditable within 2 seconds.

## How to Use

1. **Initialize a Campaign via REST API**:  
   Send a `POST /api/v1/campaigns` request with a JSON body containing `campaign_name`, `start_time` (ISO 8601), `end_time`, and `max_entries_per_user`. The API returns a `campaign_id` (UUID v4) and a `webhook_url` for event callbacks. Validate the response status `201 Created` and store the `campaign_id` for subsequent operations.

2. **Stream User Entries with Batch Processing**:  
   Use the `POST /api/v1/entries/batch` endpoint to submit up to 500 entries per request. Each entry must include `platform_type`, `user_fingerprint`, and `entry_timestamp`. The engine returns a `batch_id` and an array of `entry_status` objects (accepted, duplicate, or invalid). Monitor the `X-RateLimit-Remaining` header to avoid throttling (default: 10,000 requests/min).

3. **Trigger Winner Selection and Retrieve Results**:  
   After the campaign ends, call `POST /api/v1/campaigns/{campaign_id}/draw` with a `winner_count` and a `random_seed` (generated from a cryptographically secure source). The system executes the stored procedure and responds with a `draw_id` and `winner_ids` array. Verify the integrity by computing the SHA-3 hash of `winner_ids` concatenated with `random_seed` and comparing it to the `proof_hash` in the response.

## Best Practices

- **Implement Idempotency for Entry Submissions**: Always include a unique `idempotency_key` (UUID v4) in each entry request. The engine will reject duplicates with a `409 Conflict` status, preventing accidental double-counting during network retries. This is critical for high-volume campaigns where entry timing is competitive (e.g., limited-quantity giveaways).

- **Precompute Funnel Metrics Off-Peak**: Schedule a cron job to run `SELECT * FROM materialized_funnel WHERE campaign_id = ?` during low-traffic hours (e.g., 2:00 AM UTC). The materialized view refreshes every 15 minutes by default, but for campaigns with >100,000 entries, trigger an explicit `REFRESH MATERIALIZED VIEW CONCURRENTLY` to avoid query timeouts during live dashboards.
