# giveawaytracker - GIV_024

## Overview

giveawaytracker is a real-time, event-driven giveaway tracking and management system designed for high-throughput, multi-platform sweepstakes, raffles, and promotional events. It provides millisecond-level latency for entry validation, duplicate detection, and winner selection, integrating directly with PostgreSQL for persistent audit trails and campaign analytics.

## Core Features

- **Multi-Platform Entry Aggregation & Deduplication Engine**  
  Ingests entries from Twitter, Discord, Telegram, and custom webhooks via a unified API. The deduplication engine uses a combination of SHA-256 hashing of user identity fields (`user_id`, `platform`, `campaign_id`) and a Redis-backed Bloom filter to detect and reject duplicate entries within **< 5ms** per request, preventing ballot stuffing in real time.

- **Weighted & Tiered Winner Selection with Entropy Validation**  
  Supports configurable weight parameters (`entry_weight`, `tier_multiplier`, `bonus_actions`) for each participant. The selection algorithm uses a cryptographically secure pseudo-random number generator (CSPRNG) seeded by a combination of the block timestamp and a campaign-specific salt. The system outputs a list of `winner_id`, `prize_tier`, and a `selection_proof_hash` for verifiable fairness.

- **Campaign Lifecycle Automation & PostgreSQL Audit Log**  
  Automates campaign state transitions: `scheduled -> live -> drawing -> completed -> archived`. Each state change triggers a transactional write to a `campaign_audit_log` table in PostgreSQL, recording `campaign_id`, `previous_state`, `new_state`, `triggered_by`, and a `unix_timestamp_ms` (millisecond precision). The system also calculates real-time metrics like `total_entries`, `unique_participants`, and `entries_per_second` stored in a materialized view refreshed every 10 seconds.

## How to Use

1. **Initialize a Campaign with Custom Parameters**  
   Use the `/api/v1/campaigns` endpoint with a POST request. Required parameters: `campaign_name` (string), `start_time` (ISO 8601 with timezone), `end_time` (ISO 8601), `prize_tiers` (JSON array of objects: `{ "tier_name": "Grand Prize", "quantity": 1, "weight": 2.0 }`), and `allowed_platforms` (array of strings: `["twitter", "discord"]`). The API returns a `campaign_id` (UUID v4) and a `campaign_secret` (32-byte hex string) used for signing entries.

2. **Submit Entries via Signed Webhook Payloads**  
   Configure your platform bots to POST to `/api/v1/entries`. The payload must include: `campaign_id`, `user_id` (platform-specific unique ID), `platform` (string), `entry_timestamp` (unix timestamp in milliseconds), and an `hmac_signature` (HMAC-SHA256 of the payload body using the `campaign_secret`). The system validates the signature and deduplicates within 5ms, returning a `201 Created` with `entry_id` or a `409 Conflict` if duplicate.

3. **Execute a Verifiable Winner Draw**  
   Trigger a draw via `/api/v1/campaigns/{campaign_id}/draw` with a POST request. The endpoint accepts an optional `seed` parameter (64-bit integer) for reproducible draws in testing. The system locks the campaign state to `drawing`, runs the weighted selection algorithm, and returns a JSON response with `winners` array (each containing `winner_id`, `prize_tier`, `selection_proof_hash`), and a `draw_metadata` object including `seed_used`, `total_entries_at_draw`, and `draw_timestamp_ms`.

## Best Practices

- **Use a Dedicated PostgreSQL Instance with Connection Pooling**  
  For campaigns expecting >100,000 entries, configure a PostgreSQL instance with `max_connections = 200` and use PgBouncer for connection pooling. Enable `track_io_timing` and set `work_mem = 64MB` to optimize the materialized view refresh for real-time metrics. This prevents lock contention during high-frequency entry writes and audit log inserts.

- **Implement a Webhook Retry with Exponential Backoff for Winner Notifications**  
  After a draw, configure a retry queue (using Redis Streams) for sending winner notifications via your chosen platform APIs. Set the initial retry delay to 1 second, with a maximum backoff of 60 seconds and a retry limit of 5. Log all delivery attempts (including HTTP status codes and response bodies) to the `campaign_audit_log` table for compliance and debugging.
