# giveawaytracker - GIV_007

## Overview

giveawaytracker is a high‑performance, real‑time giveaway event orchestration and analytics engine. It ingests multi‑source promotional feeds, deduplicates entries via cryptographic fingerprinting, and computes fraud‑risk scores within sub‑100ms latency. Designed for enterprise‑scale campaigns (100k+ concurrent participants), it exposes a gRPC API for programmatic control and ships with a built‑in PostgreSQL materialized view layer for instant leaderboard generation.

## Detailed Content

### What is giveawaytracker?

giveawaytracker solves the core challenges of running large‑scale giveaway campaigns: entry validation across heterogeneous platforms (Twitter, Discord, Telegram), real‑time winner selection with verifiable randomness (drift‑resistant SHA‑3 based lottery), and comprehensive audit trails for compliance. It replaces brittle spreadsheet‑based workflows and third‑party lottery services with a self‑hosted, horizontally scalable architecture.

### Core Features

- **Multi‑Source Entry Ingestion with Cryptographic Deduplication**  
  Parses webhook payloads from Twitter API v2, Discord Gateway, and Telegram Bot API concurrently. Each entry is fingerprinted using SHA‑3‑256 over normalized user identity fields (e.g., `{platform}:{user_id}`). Duplicate entries are rejected within 50ms and logged to a dedicated `dedup_events` table in PostgreSQL with a unique index on `(event_id, fingerprint)`. Supports retry‑with‑backoff for transient webhook failures.

- **Verifiable Random Winner Selection with Drift Resistance**  
  Implements a commit‑reveal scheme: the admin generates a salted SHA‑3‑512 hash of the seed, publishes it on‑chain (Ethereum Sepolia testnet), then reveals the seed after the entry window closes. The final winner index is computed via `SHA3(seed || block_timestamp || entry_count) mod total_entries`. All steps are logged to a `lottery_proofs` table, enabling external verification. Selection latency is under 200ms for 50k entries.

- **Real‑Time Fraud Risk Scoring & Auto‑Disqualification**  
  Each entry is scored on a 0‑100 scale using a lightweight rule engine (e.g., >5 entries from same IP/24 subnet → +30 points; account age <7 days on platform → +25 points; email domain on known burner list → +40 points). Entries exceeding a configurable threshold (default: 75) are automatically moved to a `quarantined_entries` view and trigger a Slack webhook alert. Scoring is done in‑memory with a per‑event LRU cache, achieving <10ms per entry.

### How to Use

1. **Deploy the Ingestion Pipeline with Multi‑Platform Webhooks**  
   - Run `docker compose up -d` to spin up the core stack (PostgreSQL 16, Redis 7 for caching, and the `giveawaytracker-ingestion` service).  
   - Configure webhook endpoints in your Twitter App (set URL to `https://<your-host>/webhook/twitter`), Discord Bot (use `Interactions Endpoint URL` pointing to `https://<your-host>/webhook/discord`), and Telegram Bot (`setWebhook` to `https://<your-host>/webhook/telegram`).  
   - Verify ingestion by sending a test tweet with a campaign hashtag (e.g., `#GIV007`) – the entry should appear in `SELECT * FROM entries WHERE event_id = 'GIV007'` within 2 seconds.

2. **Configure Winner Selection with On‑Chain Commit**  
   - Generate a seed: `openssl rand -hex 32`. Compute its SHA‑3‑512 hash: `echo -n "<seed>" | openssl dgst -sha3-512`.  
   - Publish the hash to Ethereum Sepolia via the included `scripts/commit_seed.py` script (requires an Infura endpoint and a funded wallet).  
   - After the entry window closes, run `giveawaytracker select --event-id GIV007 --seed <seed>` – the tool fetches the block timestamp from Sepolia, computes the winner, and inserts the proof into the `lottery_proofs` table.  
   - Verify the winner by running `giveawaytracker verify --event-id GIV007` – it re‑computes the selection and compares against the stored proof.

3. **Monitor Fraud Risk and Auto‑Disqualify**  
   - Set the risk threshold: `giveawaytracker config set --event-id GIV007 --fraud-threshold 75`.  
   - Start the real‑time scoring daemon: `giveawaytracker daemon --event-id GIV007 --scoring-interval 5s`.  
   - When an entry crosses the threshold, it appears in the `quarantined_entries` table, and a Slack alert is sent to the channel configured in `config/slack.yml`.  
   - Review quarantined entries: `giveawaytracker list --event-id GIV007 --status quarantined`. Approve or dismiss individually: `giveawaytracker review --entry-id <uuid> --action approve`.

### Best Practices

- **Always Use a Commit‑Reveal Scheme for High‑Stakes Giveaways**  
  For campaigns with prizes exceeding $10k in value, never rely on a single random number generator (RNG) call. The commit‑reveal pattern with an on‑chain anchor eliminates any possibility of seed manipulation by the admin. Store the commit hash in a smart contract event log for permanent auditability – this also provides a tamper‑proof record for legal disputes.

- **Pre‑Compute Risk Scores Offline and Cache Aggressively**  
  For events with >10k entries, avoid scoring each entry on‑the‑fly during the ingestion phase. Instead, run a nightly batch job (`giveawaytracker batch-score --event-id GIV007`) that computes scores for all entries and writes them to a `scored_entries` materialized view. Use Redis to cache the top‑100 riskiest entries for real‑time dashboard display. This reduces the ingestion latency from 50ms to under 5ms per entry.
