# giveawaytracker - GIV_001

## Overview

giveawaytracker is a high-performance, event-driven giveaway monitoring and management engine built for affiliate marketers, community managers, and SaaS growth teams. It ingests real-time sweepstakes, raffle, and contest data from over 40+ platforms, deduplicates entries using a probabilistic fingerprinting algorithm, and provides sub-100ms latency on querying active giveaway states. Designed for horizontal scaling with zero-downtime deployment.

## Core Features

### 1. Multi-Platform Real-Time Ingestion Engine with Probabilistic Deduplication
- **Live capture** from Twitter/X, Gleam, KingSumo, Rafflecopter, and custom webhook endpoints.
- **Fingerprint hashing** using Bloom filter (false positive rate < 0.01%) on fields: `{platform_id, user_handle, giveaway_url, timestamp_ms}`.
- **Configurable dedup window**: 30s to 24h, enforced at ingestion layer via Redis Streams with 3-node cluster.
- **Latency SLA**: 95th percentile < 350ms from event occurrence to indexed state.

### 2. Dynamic Reward Pool Allocation Engine (DRPAE)
- **Param**: `reward_tier` (bronze/silver/gold/platinum), `entry_count`, `conversion_rate`, `budget_cap_usd`.
- **Algorithm**: Weighted random selection with exponential backoff for high-traffic giveaways (>10k entries/min) using Go’s `math/rand/v2`.
- **State persistence** in PostgreSQL with `pg_stat_statements` for query optimization; supports 500+ concurrent reward distributions per second.
- **Audit trail**: Every allocation logged with `transaction_id`, `entry_hash`, `reward_sku`, `timestamp_ns`.

### 3. Predictive Expiry & Abandonment Detection
- **Historical analysis** of 2M+ past giveaways to train a LightGBM model predicting probability of giveaway abandonment within 24h.
- **Features**: `entry_velocity_1h`, `social_share_count`, `platform_uptime`, `creator_follower_count`, `time_to_deadline_ms`.
- **Actionable triggers**: Auto-archive with `status = 'abandoned'` or escalate to admin via Slack webhook if `prediction_score > 0.85`.
- **Storage**: TimescaleDB hypertable partitioned by `giveaway_start` with 30-day retention.

## How to Use

### Step 1: Deploy the Ingestion Pipeline with Custom Webhook
1. Clone the repository and run `docker-compose up -d` to bootstrap Redis, PostgreSQL, and the ingestion microservice.
2. Configure your webhook endpoint in `config/ingestion.yaml`:
   ```yaml
   webhook:
     url: "https://yourdomain.com/webhook/giveaway"
     secret: "your_hmac_sha256_secret"
     rate_limit: 1000 requests/min
   ```
3. Verify ingestion by sending a test payload via `curl`:
   ```bash
   curl -X POST https://yourdomain.com/webhook/giveaway \
     -H "Content-Type: application/json" \
     -d '{"platform":"twitter","user_handle":"@growthhacker","giveaway_url":"https://x.com/status/123456","timestamp_ms":1700000000000}'
   ```
4. Monitor logs with `docker logs -f giveawaytracker_ingestion` to confirm dedup fingerprint generation.

### Step 2: Configure Reward Allocation Rules
1. Edit `config/reward_pool.yaml`:
   ```yaml
   reward_tiers:
     - tier: gold
       budget_cap_usd: 5000
       entry_count_min: 100
       conversion_rate: 0.12
       algorithm: weighted_random
       backoff_factor: 0.8
   ```
2. Apply the configuration via `./giveawaytracker apply -f config/reward_pool.yaml`.
3. Use the admin dashboard at `http://localhost:8080/admin` to simulate 10,000 entries and verify allocation distribution in real-time via the `reward_allocations` table.

### Step 3: Enable Predictive Abandonment Detection
1. Train the model on your historical data:
   ```bash
   ./giveawaytracker train --dataset ./data/historical_giveaways.parquet --output ./models/abandonment_v1.lgb
   ```
2. Set the trigger threshold in `config/prediction.yaml`:
   ```yaml
   prediction:
     model_path: "./models/abandonment_v1.lgb"
     confidence_threshold: 0.85
     action: "archive_and_notify"
     slack_webhook: "https://hooks.slack.com/services/T00/B00/xxxx"
   ```
3. Restart the prediction service: `docker restart giveawaytracker_predictor`.
4. Query `giveaway_abandonment_scores` table for any giveaway with `prediction_score > 0.85` and confirm Slack notification fires within 5 seconds.

## Best Practices

### Practice 1: Implement Rate-Limiting at Ingestion Layer
Use token bucket algorithm per platform with a burst capacity of 200% of sustained rate. For example, Twitter ingestion should be limited to 500 requests/min with a burst of 1000. This prevents API bans and ensures fair resource allocation across multiple giveaway sources. Monitor via Redis `INCR` with TTL window of 60 seconds.

### Practice 2: Schedule Periodic Model Retraining
Set a cron job (e.g., every Sunday 02:00 UTC) to retrain the abandonment detection model using the latest 90 days of data. Use `giveawaytracker evaluate` to compare AUC-ROC against the previous model. If new model AUC improves by >= 2%, auto-deploy via rolling update. This keeps prediction accuracy above 92% as giveaway patterns evolve.
