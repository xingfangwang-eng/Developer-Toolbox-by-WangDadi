# giveawaytracker - GIV_017

## Overview

giveawaytracker is a real-time, multi-source giveaway aggregation and analytics engine. It ingests event feeds from Discord bots, Twitter/X API v2, Telegram channels, and RSS endpoints, deduplicates them via locality-sensitive hashing (LSH), and surfaces actionable opportunities with precise temporal decay scoring. This system is built for power users who manage 50+ concurrent giveaways across platforms and need millisecond-level latency on entry window closures.

## Core Features

- **Cross-Platform LSH Deduplication Engine**: Each giveaway entry is fingerprinted using a 128-bit MinHash signature computed from title, URL, and host platform metadata. The system maintains a sliding window of 10,000 fingerprints in Redis, enabling O(1) duplicate detection with a Jaccard similarity threshold of 0.85. False positive rate is below 0.3% across 500k+ test entries.

- **Temporal Decay Scoring & Auto-Expiry**: Every giveaway is assigned a dynamic decay score `S(t) = base_value * e^(-λ * Δt)` where `λ` is tuned per platform (0.05 for Twitter, 0.12 for Discord). Entries with score below 0.15 are automatically archived to PostgreSQL with a TTL of 30 days. The system triggers webhook callbacks (JSON payload with `event_id`, `remaining_ms`, `score`) exactly 60 seconds before expiry.

- **Multi-Variant Entry Condition Parser**: Parses raw giveaway rules into a structured condition tree (AND/OR operators, regex patterns for "follow", "retweet", "tag friend"). Supports nested conditions up to depth 5. Each parsed condition is stored as a JSONB column in PostgreSQL, enabling full-text search over `conditions['actions']` with GIN indexes. Average parse time per rule is 1.2 ms on a 4-core instance.

## How to Use

1. **Initialize the tracker with platform credentials and filter rules**  
   Create a `config.yaml` file in the root directory with your API tokens and a `filters` block specifying keywords (e.g., `["giveaway", "RT to win", "airdrop"]`), minimum prize value in USD (`min_prize: 50`), and excluded domains (`exclude: ["spam-site.io"]`). Run `giveawaytracker --config config.yaml` to start the ingestion daemon. The system will validate each token with a test request (HTTP 200 expected) and print connection status per platform.

2. **Start the real-time dashboard with decay score overlays**  
   Launch the built-in web dashboard via `giveawaytracker serve --port 8080`. The dashboard renders a live table with columns: `event_id`, `source_platform`, `title`, `score` (0.0–1.0), `expires_in_ms`. Click any row to expand the parsed condition tree. Use the `--score-threshold 0.7` flag to filter out low-value entries. The dashboard refreshes every 2 seconds via WebSocket push (no polling overhead).

3. **Export high-confidence entries to a webhook or file**  
   To push filtered results to an external system (e.g., a Telegram bot or Slack channel), set `webhook_url` in `config.yaml`. The webhook payload is a JSON array of `{event_id, title, url, score, expiry_epoch_ms}`. Alternatively, run `giveawaytracker export --format csv --score-min 0.85 --output ./top_giveaways.csv` to generate a timestamped CSV file. The export uses a cursor-based pagination over the PostgreSQL `giveaways` table with `ORDER BY score DESC LIMIT 100`.

## Best Practices

- **Tune decay lambda per platform weekly**: Discord giveaways tend to close faster (48h average) than Twitter giveaways (72h). Adjust `lambda` in the config file based on historical data from the last 7 days. Use the built-in `giveawaytracker stats --platform discord` command to view median expiry time and adjust accordingly. A lambda that is too aggressive (e.g., >0.15) will prematurely archive still-valid entries.

- **Combine LSH deduplication with manual blacklist**: The LSH engine catches near-duplicates (e.g., same URL with different tracking parameters), but some spam accounts post identical text with different URLs. Maintain a `blacklist.txt` file (one regex per line) in the config directory. The system will check each entry's `title` and `source_url` against the blacklist before scoring. Use `giveawaytracker test --blacklist` to validate your patterns against recent entries.
