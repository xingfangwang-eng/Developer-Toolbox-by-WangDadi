# giveawaytracker - GIV_020

## Overview

giveawaytracker is a high-performance, event-driven giveaway monitoring and management system engineered for real-time sweepstakes, contest, and promotional event tracking. It processes concurrent giveaway entries across multiple platforms with sub-millisecond latency, and integrates with PostgreSQL for persistent state management and historical analytics.

## Detailed Content

### What is giveawaytracker?

giveawaytracker is a distributed tracking engine designed to ingest, normalize, and reconcile giveaway events from heterogeneous sources (e.g., Twitter, Discord, Telegram, custom APIs). It employs a multi-threaded pipeline with a configurable deduplication window (default: 250ms) and a CRDT-based state store to guarantee exactly-once processing of participant actions—even under network partitions or partial failures.

### Core Features

- **Multi-Source Event Ingestion & Deduplication Pipeline**  
  Parses raw webhook payloads from 6+ platform connectors (Twitter API v2, Discord Gateway, Telegram Bot API) using a pluggable adapter architecture. Each event is hashed (SHA-256) and deduplicated against a Redis-backed bloom filter with a false-positive rate < 0.1%. Supports parallel ingestion at 10,000 events/second per node.

- **Real-Time Leaderboard & Prize Allocation Engine**  
  Computes weighted random selection with configurable entropy sources (e.g., timestamp + user ID + nonce). Exposes a gRPC streaming endpoint that pushes leaderboard deltas every 50ms. Prize pool distribution supports tiered rarity (common, rare, legendary) with probability normalization enforced at the database level via PostgreSQL CHECK constraints.

- **Incremental State Snapshots & Audit Trail**  
  Every giveaway lifecycle transition (created → active → drawn → settled) is recorded as an immutable event in a PostgreSQL event-sourcing table. Supports point-in-time replay to reconstruct any giveaway’s state at a given UTC millisecond. Full audit log exportable as NDJSON for compliance review.

### How to Use

1. **Configure Connector Adapters & Webhook Endpoints**  
   In `config/connectors.yaml`, define each platform’s authentication and endpoint:  
   ```yaml
   twitter:
     bearer_token: "YOUR_BEARER_TOKEN"
     webhook_path: "/webhook/twitter"
     rate_limit: 5000 requests/hour
   discord:
     bot_token: "YOUR_BOT_TOKEN"
     gateway_intents: ["GUILD_MESSAGES", "MESSAGE_CONTENT"]
   ```  
   Run `./giveawaytracker init-connectors` to register webhooks with each platform and verify connectivity via health check endpoints (`/health/connectors`).

2. **Define Giveaway Rules & Prize Pool**  
   Create a giveaway manifest in JSON:  
   ```json
   {
     "id": "summer-sweeps-2025",
     "start_utc": "2025-06-01T00:00:00Z",
     "end_utc": "2025-06-30T23:59:59Z",
     "max_entries_per_user": 5,
     "prizes": [
       { "tier": "legendary", "count": 1, "weight": 0.01 },
       { "tier": "rare", "count": 10, "weight": 0.09 },
       { "tier": "common", "count": 500, "weight": 0.90 }
     ],
     "selection_algorithm": "weighted_shuffle",
     "seed": "env://GIVEAWAY_SEED"
   }
   ```  
   Post it to `POST /api/v1/giveaways` — the server validates rules against schema (OpenAPI 3.1) and returns a `giveaway_id` immediately.

3. **Monitor & Trigger the Draw**  
   Stream live events via WebSocket at `ws://host:8080/ws/events?giveaway_id=summer-sweeps-2025`. When the giveaway ends, call `POST /api/v1/giveaways/{id}/draw` with optional `dry_run: true` to preview winners without finalizing. After the draw, winners are published to a configurable callback URL (e.g., `https://your-callback.com/winners`) with HMAC-SHA256 signed payloads.

### Best Practices

- **Use Horizontal Scaling for High-Volume Campaigns**  
  Deploy behind an L7 load balancer (e.g., NGINX or HAProxy) with sticky sessions based on `giveaway_id`. Each node maintains its own Redis shard for deduplication; use Redis Cluster to auto-shard across nodes. Monitor `giveawaytracker_events_ingested_total` and `giveawaytracker_dedup_rate` Prometheus metrics to tune the bloom filter capacity.

- **Always Enable Audit Trail for Compliance**  
  For regulated industries (e.g., gambling, crypto promotions), set `audit.log_all_events: true` in `config/audit.yaml`. This forces every API call and state mutation to be logged with a monotonic sequence number. Export logs nightly to an immutable S3 bucket with bucket versioning enabled. Pair with periodic `SELECT * FROM audit_events WHERE giveaway_id = ? ORDER BY seq_no` queries to prove fair draw execution.
