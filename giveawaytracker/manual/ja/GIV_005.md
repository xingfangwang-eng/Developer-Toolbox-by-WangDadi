# giveawaytracker - GIV_005

## 概要

giveawaytracker は、ギブアウェイ（プレゼント企画）の参加者管理・抽選・配信結果をリアルタイムでトラッキングするための高パフォーマンスイベント追跡システムです。本プロジェクトは、ソーシャルメディアプラットフォーム上のキャンペーンを自動クローリングし、重複参加の排除と抽選バイアスの除去を実現します。

## 詳細な内容

### giveawaytracker とは

giveawaytracker は、大規模ギブアウェイキャンペーンにおける参加者エンゲージメントの計測、抽選ロジックの透明化、配信結果の事後分析を一元管理するために設計されたプロフェッショナルツールです。特に、複数プラットフォーム（Twitter、Instagram、Discord）にまたがるキャンペーンの同時監視と、ミリ秒単位のタイムスタンプ同期を特徴とします。

## コア機能

### 機能 1: マルチプラットフォーム参加者デデュプリケーションエンジン

- **概要**: 同一ユーザーが異なるSNSアカウントやメールアドレスを使用して重複参加した場合、デバイスフィンガープリント（IPアドレス、User-Agentハッシュ、ブラウザフィンガープリント）とアカウントリンケージ（OAuthプロバイダ間の紐付け）を用いて自動排除します。
- **技術仕様**:
  - 重複検出アルゴリズム: Bloom Filter と HyperLogLog のハイブリッド実装（Redis 上で動作）
  - 処理速度: 10万レコードあたり平均 120 ミリ秒
  - データベース: PostgreSQL 15 以上、部分インデックス（`WHERE status = 'active'`）を使用
- **パラメータ例**:
  - `--dedup-window 3600`（重複判定の時間窓を秒単位で指定、デフォルト3600秒）
  - `--fingerprint-depth high`（high: 全フィンガープリント比較、low: IPのみ）

### 機能 2: 重み付き確率抽選エンジン with 監査ログ

- **概要**: 参加者のアクション履歴（リツイート、いいね、タグ付け数）に基づいて抽選確率を動的に重み付けし、抽選結果と乱数生成過程を完全に監査可能な形で保存します。
- **技術仕様**:
  - 乱数生成: `crypto/rand`（Go言語）による暗号論的擬似乱数、シード値はエポックミリ秒＋ノンスで生成
  - 重み計算: 対数正規分布に基づくスコアリング（`μ=0, σ=1.5`）
  - 抽選結果の監査: 全抽選イベントをJSON Lines形式でS3互換ストレージに保存（例: `audit/2025-03-17/run_001.jsonl`）
- **パラメータ例**:
  - `--weight-multiplier 1.2`（アクションごとの重み倍率）
  - `--audit-log-retention 90d`（監査ログの保存期間）

### 機能 3: リアルタイムダッシュボード with WebSocket プッシュ

- **概要**: 参加者数、抽選進捗、配信ステータスを100ミリ秒以内のレイテンシでブラウザにプッシュ。バックエンドはGoのgoroutineチャネルとRedis Pub/Subで構成。
- **技術仕様**:
  - フロントエンド: React 18 + Vite + Recharts（グラフライブラリ）
  - バックエンド: Go 1.22、`gorilla/websocket`、`redis/go-redis/v9`
  - 更新頻度: 100ミリ秒ごとに差分データをプッシュ（全件再送は10秒ごと）
  - パラメータ例: `--ws-ping-interval 30s`（WebSocketのハートビート間隔）

## 使い方

### ステップ 1: 環境構築と初期設定

```bash
# PostgreSQL にデータベースとユーザーを作成
psql -U postgres -c "CREATE DATABASE giveawaytracker;"
psql -U postgres -c "CREATE USER giv_user WITH PASSWORD 'your_strong_password';"
psql -U postgres -c "GRANT ALL PRIVILEGES ON DATABASE giveawaytracker TO giv_user;"

# マイグレーション実行（テーブル自動作成）
./giveawaytracker migrate --dsn "postgres://giv_user:your_strong_password@localhost:5432/giveawaytracker?sslmode=disable"

# Redis の起動確認
redis-cli ping  # PONG が返ればOK
```

### ステップ 2: キャンペーン定義とクローリング開始

```yaml
# campaign.yaml の例
campaign:
  name: "Spring Giveaway 2025"
  platforms:
    - twitter:
        query: "#SpringGiveaway2025 -filter:retweets"
        since: "2025-03-01T00:00:00Z"
        until: "2025-03-31T23:59:59Z"
    - discord:
        guild_id: "123456789012345678"
        channel_id: "987654321098765432"
  rules:
    dedup_window_sec: 3600
    weight_multiplier: 1.5
    max_entries_per_user: 1

# クローリング開始
./giveawaytracker crawl --config campaign.yaml --workers 4
```

### ステップ 3: 抽選実行と結果配信

```bash
# 抽選実行（10名当選、重み付き）
./giveawaytracker draw --campaign-id 1 --winners 10 --weighted --output json > draw_result.json

# 結果をWebhookで配信（Discord or Slack）
./giveawaytracker notify --webhook-url "https://hooks.slack.com/services/xxx/yyy/zzz" --payload draw_result.json

# 監査ログの確認
cat audit/2025-03-17/run_001.jsonl | jq '.seed, .winner_ids'
```

## ベストプラクティス

### プラクティス 1: 抽選の透明性を担保するために、抽選前にシード値を公開せよ

抽選開始前に、乱数生成に使用するシード値（例: `2025-03-17T12:00:00Z` のエポックミリ秒＋固定ノンス）をキャンペーンページやSNSで事前公開することで、抽選結果の改ざん防止と参加者の信頼獲得に繋がります。giveawaytracker の `--publish-seed` フラグを使用すると、抽選実行時に自動でシード値をハッシュ化して公開できます。

### プラクティス 2: 高負荷時のスケーリングにはRedis ClusterとPostgreSQLのパーティショニングを併用せよ

100万件以上の参加者が見込まれるキャンペーンでは、Redis Clusterを用いた分散デデュプリケーションと、PostgreSQLのテーブルパーティショニング（日付単位）を推奨します。giveawaytracker の `--redis-cluster-nodes` オプションで複数ノードを指定でき、`--partition-interval daily` で自動パーティショニングが有効になります。
