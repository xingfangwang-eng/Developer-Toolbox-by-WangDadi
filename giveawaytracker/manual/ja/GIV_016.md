# giveawaytracker - GIV_016

## 概要

giveawaytracker は、大規模キャンペーンにおけるプレゼント抽選・配布イベントの全行程をリアルタイム追跡・管理するためのエンタープライズ向けトラッキング基盤です。キーワード: プレゼント追跡ツール、抽選管理システム、イベントトラッキング

## 詳細な内容

### giveawaytracker とは

giveawaytracker は、SNSキャンペーンやECサイトでのプレゼント企画において、応募受付から当選者確定、発送完了までを一貫して管理するプロフェッショナルツールです。PostgreSQL との連携により、ミリ秒単位の応募タイムスタンプ記録と重複排除を実現。API 応答速度は平均 12ms 以下を維持します。

### コア機能

- **マルチチャネル重複排除エンジン**: Twitter、Instagram、LINE、自社サイトからの応募をリアルタイムに統合。`dedup_window_ms` パラメータで重複判定の時間窓（デフォルト 3000ms）を設定可能。同一ユーザーが異なるチャネルから応募した場合、`user_fingerprint_hash`（SHA-256）を用いて自動的に排除し、公平性を担保します。

- **動的抽選重み付けアルゴリズム**: 抽選時、`weight_pool` に指定された条件（フォロワー数、過去参加回数、アカウント年数）に基づき、各応募者に動的重みを付与。`batch_size` パラメータ（最大 100,000 件/バッチ）で大量応募を分割処理し、抽選結果を 500ms 以内に返却。`random_seed` を固定することで再現可能な抽選結果を出力可能。

- **発送ステータス自動同期パイプライン**: 発送代行サービス（ヤマト運輸 B2 連携 API、日本郵便 Web コレクト）と直接接続。`shipping_sync_interval`（デフォルト 60 秒）で自動ポーリング。`tracking_number` が発行された時点で当選者にプッシュ通知（Firebase Cloud Messaging 経由）を送信。発送遅延が発生した場合、`delay_threshold_hours`（デフォルト 48 時間）を超えると管理者に Slack Webhook でアラート。

### 使い方

1. **キャンペーン初期設定と応募受付チャネル統合**  
   `giveawaytracker init --campaign-name "2025春の大感謝祭" --channels twitter,instagram,line,web` を実行。  
   各チャネルの API キーを `config/channels.yaml` に記述し、`giveawaytracker sync --channel twitter --mode historical` で過去 30 日間の応募データを一括取り込み。  
   `--dedup-window 5000` で重複排除の時間窓を 5 秒に設定。  
   取り込み後、`giveawaytracker stats --campaign-id <ID>` で重複率、チャネル別応募数、ユニークユーザー数を確認。

2. **抽選実行と当選者確定**  
   `giveawaytracker draw --campaign-id <ID> --winners 100 --weight-pool followers,participation --batch-size 50000` で抽選を実行。  
   抽選後、`giveawaytracker winners export --format csv --output ./winners_2025spring.csv` で当選者リストをエクスポート。  
   `giveawaytracker verify --campaign-id <ID> --winner-id <ID>` で個別の抽選結果を検証（`random_seed` と応募タイムスタンプから再計算）。

3. **発送管理とステータス追跡**  
   `giveawaytracker shipping setup --carrier yamato --api-key <KEY>` で発送代行サービスと連携。  
   `giveawaytracker shipping dispatch --campaign-id <ID> --batch-size 200` で当選者 200 件ずつ発送指示を送信。  
   `giveawaytracker monitor --campaign-id <ID> --interval 30` で 30 秒ごとに発送ステータスをポーリング。  
   発送完了後、`giveawaytracker report --campaign-id <ID> --type full` で応募数、当選者数、発送完了率、平均発送リードタイム（ミリ秒単位）を含む包括レポートを生成。

### ベストプラクティス

- **プラクティス 1: 応募期間中のリアルタイム監視と動的重み調整**  
  キャンペーン開始後、`giveawaytracker monitor --campaign-id <ID> --interval 10` で 10 秒間隔で応募状況を監視。不正応募（同一 IP からの短時間連続応募）が検出された場合、`--block-ip` オプションで即時ブロック。また、特定チャネルからの応募が予想を超えた場合、`giveawaytracker draw --reweight --channel twitter --weight 0.5` で動的に重みを調整し、チャネル間の公平性を維持。

- **プラクティス 2: 抽選結果の監査証跡と再現性確保**  
  抽選実行前に必ず `giveawaytracker snapshot --campaign-id <ID> --output ./snapshot_<timestamp>.json` で応募データ全体のスナップショットを取得。抽選後、`giveawaytracker audit --campaign-id <ID> --snapshot ./snapshot_<timestamp>.json` でスナップショットと抽選結果の整合性を検証。`random_seed` はキャンペーン ID と実行時刻のハッシュ値（SHA-256）から自動生成し、監査ログに記録。これにより、後日第三者による抽選結果の検証が可能。
