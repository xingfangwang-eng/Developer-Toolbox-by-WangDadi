# giveawaytracker - GIV_020

## 概要

giveawaytracker は、大規模プレゼントキャンペーンのエントリー管理、抽選実行、当選者確定までを自動化するプロフェッショナル向けトラッキングシステムです。本ツールは、Twitter、Discord、Webフォームからのエントリーデータを統合し、重複排除、抽選バイアス補正、リアルタイム配信を実現します。

## 詳細な内容

### giveawaytracker とは

giveawaytracker は、複数プラットフォームに分散するプレゼント応募データを一元管理し、**抽選の公平性担保**と**当選者確定までのリードタイム短縮**を目的としたエンタープライズグレードのイベントトラッキングツールです。PostgreSQL による堅牢なデータ永続化と、Redis を用いたミリ秒単位のエントリー重複チェックを標準搭載しています。

### コア機能

#### 1. マルチプラットフォームエントリー統合エンジン

Twitter、Discord、カスタムWebフォームからの応募データを **1秒以内に正規化** し、一意のエントリーID（`entry_uid`）を発行。`platform`、`user_id`、`entry_timestamp` をキーにした複合ユニーク制約により、同一ユーザーの重複応募を自動排除します。

- **パラメータ**: `entry_uid`, `platform`, `user_id`, `entry_timestamp`, `raw_payload`
- **技術指標**: 重複検出レイテンシ < 50ms（Redis SETNX 利用）、PostgreSQL へのバルクインサート 10,000 rows/sec

#### 2. 確率的公平抽選エンジン（PFLE）

エントリー数が10万件を超えるキャンペーンでも、**抽選バイアスを±0.1%以内**に抑える乱数生成アルゴリズムを実装。`random_seed` をキャンペーン開始時に決定論的に生成し、全エントリーに対して **Fisher-Yates シャッフル** を適用します。抽選結果は `lottery_result` テーブルにハッシュチェーンで記録され、後から検証可能です。

- **パラメータ**: `campaign_id`, `random_seed`, `winner_count`, `lottery_timestamp`
- **技術指標**: 10万エントリーの抽選完了時間 < 2秒、ハッシュチェーン検証時間 < 100ms

#### 3. リアルタイム当選者確定・通知パイプライン

抽選後、当選者情報を `winner_notification` キューに投入し、**非同期ワーカー**が各プラットフォームのDM送信APIを呼び出します。送信失敗時は **3回のリトライ（バックオフ係数2.0）** を実行し、最終失敗時は `notification_status` を `failed` に更新して管理者にアラートを発報します。

- **パラメータ**: `winner_id`, `notification_channel`, `retry_count`, `backoff_factor`, `alert_threshold`
- **技術指標**: 初回通知送信レイテンシ < 500ms、リトライ間隔 1s / 2s / 4s、アラート発報閾値 3 failures

```bash
    "winner_count": 100,
    "random_seed": "a1b2c3d4e5f6"
  }'

# プラットフォーム別エントリーWebhook を登録（Twitter, Discord, Web）
# 各プラットフォームのエントリーデータは自動的に正規化され、entry_uid が発行される
```

#### ステップ 2: 抽選実行と結果検証

```bash
# 抽選実行（PFLE エンジンを呼び出し、lottery_result テーブルに結果を書き込む）
```

### ベストプラクティス

#### ベストプラクティス 1: エントリー重複排除の事前フィルタリング

キャンペーン開始前に、**同一ユーザーの複数プラットフォーム応募を許容するかどうか**をポリシーとして定義し、`entry_uid` の生成ロジックに組み込んでください。例えば、`user_email` が同一のエントリーは `platform` に関わらず重複と見なす場合、`entry_uid` のハッシュキーに `user_email` を含めることで、PostgreSQL のユニーク制約違反を事前に防止できます。これにより、抽選時のバイアスを根本から排除できます。

#### ベストプラクティス 2: 抽選シードの決定論的生成と監査ログ

`random_seed` はキャンペーン開始前にオフラインで生成し、暗号学的ハッシュ関数（SHA-256）でコミットしてください。抽選実行時にシードを公開することで、**抽選結果の検証可能性**を担保します。また、`lottery_result` テーブルへの書き込みはトランザクション分離レベル `SERIALIZABLE` で行い、同時実行による競合を防止します。監査ログは最低でも **365日間** 保持することを推奨します。
