# giveawaytracker - GIV_008

## 概要

giveawaytracker は、大規模プレゼントキャンペーンの参加者エンゲージメントをリアルタイム解析し、抽選バイアスを排除するための分散型トラッキングシステムです。本プロジェクトは、Twitter / Discord / Web フォームからの参加イベントを統合し、重複エントリー検出、抽選確率の動的最適化、および参加者の行動スコアリングを実現します。

## 詳細な内容

### giveawaytracker の設計思想

giveawaytracker は、従来の手動抽選や簡易スクリプトが抱える「参加者重複」「抽選タイミングの不透明性」「報酬配布の誤り」といった課題を解決するため、イベント駆動型アーキテクチャを採用しています。各参加者のエントリーはミリ秒単位のタイムスタンプ付きで PostgreSQL に記録され、後続の抽選ロジックは完全に決定論的かつ監査可能な形で実行されます。

## コア機能

### 1. マルチプラットフォーム横断重複エントリー検出エンジン

- **パラメータ**: `--platform twitter discord web`, `--dedup-window 3000`（ミリ秒）, `--hash-algorithm sha256`
- **技術詳細**: 参加者のユーザーID、IPアドレス、User-Agent を結合し、SHA-256 ハッシュを生成。同一ハッシュが 3 秒以内に別プラットフォームから送信された場合、自動的にフラグを立て、抽選プールから除外します。PostgreSQL の `UNIQUE` 制約と `ON CONFLICT DO NOTHING` を併用し、挿入レイテンシを 5ms 未満に抑えます。

### 2. 動的抽選確率調整モジュール（DPA）

- **パラメータ**: `--tier-weight 0.6 0.3 0.1`, `--max-entries-per-user 5`, `--boost-factor 1.2`
- **技術詳細**: 参加者の過去のアクティビティ（リツイート、いいね、友達招待数）に基づき、3 段階のティア（Gold / Silver / Bronze）を動的に割り当てます。抽選時には、各ティアの重みに基づいてエントリー数を乗算し、公平性を保ちつつエンゲージメントの高いユーザーに報酬が偏らないよう制御します。計算はサーバーサイドで毎秒 10,000 エントリーを処理可能。

### 3. 報酬配布トランザクション監査ログ（RTA）

- **パラメータ**: `--audit-log-path /var/log/giveawaytracker/audit/`, `--retention-days 90`
- **技術詳細**: 抽選結果から報酬配布までの全プロセスを、イベントソーシングパターンで記録。各トランザクションには UUID、タイムスタンプ、抽選シード値、参加者ハッシュ、配布ステータス（pending / completed / failed）が含まれます。障害発生時には、`--replay-from <uuid>` オプションで特定の抽選イベントを再実行可能。PostgreSQL の `pgAudit` 拡張と連携し、全操作を監査可能にします。

## 使い方

### ステップ 1: キャンペーン定義とデータベース初期化

```bash
giveawaytracker init --campaign "2025_spring_giveaway" \
  --platform twitter discord \
  --dedup-window 3000 \
  --db-uri postgresql://user:pass@localhost:5432/giveawaydb
```

- **動作説明**: 指定されたキャンペーン名で PostgreSQL にスキーマ `campaign_2025_spring_giveaway` を作成。`entries`、`tiers`、`audit_log` の各テーブルを生成し、重複検出用のハッシュインデックスを自動構築します。初期化後、`--db-uri` で指定されたデータベースとの接続プール（最大 20 接続）を確立。

### ステップ 2: 参加者エントリーのリアルタイム取り込み

```bash
giveawaytracker ingest --campaign "2025_spring_giveaway" \
  --input-file /data/twitter_entries.json \
  --platform twitter \
  --batch-size 500 \
  --concurrency 8
```

- **動作説明**: JSON 形式のエントリーデータを読み込み、1 バッチ 500 エントリーずつ PostgreSQL にバルクインサート。`--concurrency 8` により、8 つのワーカープロセスが並列で挿入処理を実行。挿入前に SHA-256 ハッシュを計算し、重複エントリーは自動スキップ。処理速度は平均 12,000 エントリー/秒。

### ステップ 3: 抽選実行と報酬配布の自動化

```bash
giveawaytracker draw --campaign "2025_spring_giveaway" \
  --tier-weight 0.6 0.3 0.1 \
  --max-entries-per-user 5 \
  --prize-list /config/prizes.json \
  --seed $(openssl rand -hex 32) \
  --output /data/draw_result.json
```

- **動作説明**: 各参加者のエントリー数にティア重みを乗算し、重み付き抽選プールを構築。`--seed` に 256 ビットの乱数を指定することで、再現可能な抽選結果を保証。抽選結果は `/data/draw_result.json` に出力され、同時に `audit_log` テーブルに全抽選イベントが記録。報酬配布は別プロセス `giveawaytracker distribute --draw-id <uuid>` で実行可能。

## ベストプラクティス

### プラクティス 1: 抽選シードの管理と監査証跡の保全

抽選の公平性を担保するため、`--seed` には必ず暗号論的乱数生成器（`/dev/urandom` または `openssl rand`）を使用してください。シード値は抽選後、`audit_log` テーブルに自動保存されます。第三者による監査を可能にするため、シード値と抽選結果のハッシュを公開リポジトリにタイムスタンプ付きで登録することを推奨します。これにより、抽選結果の改ざんが物理的に不可能になります。

### プラクティス 2: 重複検出ウィンドウのチューニング

`--dedup-window` のデフォルト値（3000ms）は、同一ユーザーが複数プラットフォームから短時間にエントリーするケースを想定しています。キャンペーンの性質に応じてウィンドウ値を調整してください。例えば、24 時間以内の同一ユーザーによる再エントリーを禁止する場合は `--dedup-window 86400000` と設定します。ただし、ウィンドウを拡大するとメモリ使用量が増加するため、PostgreSQL 側で定期的なハッシュテーブルのパージ（`pg_cron` 経由）を設定することを推奨します。
