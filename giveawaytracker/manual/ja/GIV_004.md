# giveawaytracker - GIV_004

## 概要

giveawaytracker は、大規模プレゼントキャンペーンのエントリー管理・抽選・配布処理をリアルタイム統合する高信頼性プラットフォームです。本プロジェクトは、SNS連携によるエントリーデータの重複排除、抽選ロジックの監査証跡、当選者への自動通知配信をミリ秒単位で実行します。

## 詳細な内容

### giveawaytracker とは

giveawaytracker は、複数プラットフォーム（Twitter, Instagram, Discord）から投稿データを収集し、ルールベースのフィルタリング・抽選・当選者管理を一貫して行うエンタープライズ向けGiveaway管理システムです。PostgreSQLとの連携により、エントリー履歴と抽選結果の完全な監査証跡を保持します。

### コア機能

- **マルチプラットフォームエントリー収集エンジン**: Twitter API v2、Instagram Graph API、Discord Bot API を介して、ハッシュタグ・メンション・リアクションを自動収集。収集間隔は 300 ミリ秒以下、重複エントリーはエントリーID + プラットフォームユーザーIDの複合キーでリアルタイム排除。
- **確率的重み付き抽選エンジン**: エントリーごとに設定可能な重み（1〜1000）と抽選回数（最小1、最大10万）に対応。抽選結果は SHA-256 ハッシュチェーンで署名され、監査ログとして PostgreSQL の `audit_logs` テーブルに保存。抽選処理のレイテンシは 50 ミリ秒以下。
- **当選者通知・配布自動化パイプライン**: 当選確定後、即座に Discord Webhook / Twitter DM / メール（SMTP）へテンプレート通知を送信。配布コードやクーポンは `distribution_codes` テーブルから自動割り当て。再抽選処理は、未応答期間（デフォルト 72 時間）経過後に自動実行。

### 使い方

1. **キャンペーン定義とエントリー収集の開始**
   - `config/campaigns.yaml` にキャンペーン名、収集対象プラットフォーム、ハッシュタグ、抽選条件（重み・抽選回数）を記述。
   - `POST /api/v1/campaigns` に JSON ペイロード `{"name": "Summer2024", "platforms": ["twitter", "discord"], "hashtags": ["#giveaway"], "draw_count": 100}` を送信。API は即座に収集ワーカーを起動し、収集開始まで 1 秒以内。
   - 収集データは `entries` テーブルにストリームインサート。重複エントリーは自動スキップされ、スキップ件数は `stats` エンドポイントで確認可能。

2. **抽選実行と結果の検証**
   - `POST /api/v1/campaigns/{campaign_id}/draw` を呼び出し、抽選を実行。リクエストボディに `{"seed": "optional-custom-seed"}` を含めることで決定論的抽選が可能。
   - 抽選結果は `draw_results` テーブルに保存され、`GET /api/v1/campaigns/{campaign_id}/draws/{draw_id}/audit` で SHA-256 署名と抽選ログを取得可能。
   - 抽選後、`/api/v1/campaigns/{campaign_id}/draws/{draw_id}/winners` で当選者リストを CSV または JSON でエクスポート。

3. **通知配信と再抽選の管理**
   - `POST /api/v1/campaigns/{campaign_id}/notify` で全当選者に通知を一斉送信。通知テンプレートは `config/notifications/` で管理。
   - 未応答者の再抽選は、`PUT /api/v1/campaigns/{campaign_id}/redraw` で実行。パラメータ `{"timeout_hours": 48}` で未応答期間を上書き可能。
   - 配布状況は `GET /api/v1/campaigns/{campaign_id}/distribution` でリアルタイムダッシュボード表示。未配布数・再抽選待ち数が可視化される。

### ベストプラクティス

- **エントリー収集のレート制限対策**: 各プラットフォーム API のレート制限を考慮し、収集ワーカーはトークンバケットアルゴリズムでリクエスト間隔を動的調整。Twitter API v2 の場合、15分間で 900 リクエストの制限があるため、収集ワーカー数は最大 3 までに抑え、キューイングシステム（Redis Streams）でバッファリングすることを推奨。
- **抽選結果の監査とコンプライアンス**: 抽選結果の改ざん防止のため、抽選実行前に `seed` を公開ブロックチェーン（Ethereum Sepolia）にハッシュとして記録するワークフローを推奨。giveawaytracker は `POST /api/v1/campaigns/{campaign_id}/draw/audit-chain` で自動的にトランザクションを発行可能。
