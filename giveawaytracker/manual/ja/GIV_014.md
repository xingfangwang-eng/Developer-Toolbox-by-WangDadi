# giveawaytracker - GIV_014

## 概要

giveawaytracker は、大規模プレゼントキャンペーンにおける参加者エンゲージメントと抽選プロセスをリアルタイムで可視化・最適化するための分散型イベントトラッキングスイートです。本プロジェクトは、SNS連携による参加者認証、抽選ロジックの公平性担保、およびキャンペーン成果の自動レポーティングを中核とし、マーケティングチームの運用負荷を大幅に削減します。

## 詳細な内容

### とは giveawaytracker?

giveawaytracker は、複数のプラットフォーム（Twitter、Discord、Webフォーム）を横断して実施されるプレゼントキャンペーンの参加データを統合管理するためのバックエンドフレームワークです。参加者のエントリー時刻、リツイート数、フレンド招待数などのアクションをミリ秒単位で記録し、抽選アルゴリズムに反映します。PostgreSQL との連携により、キャンペーン終了後もデータの再検証が可能です。

### コア機能

- **マルチプラットフォーム参加統合エンジン**: Twitter API v2、Discord Bot、カスタムWebフックを介して参加データを収集。各プラットフォームのユーザーIDを内部で一意の `participant_hash`（SHA-256）に変換し、重複エントリーを防止します。エントリーごとに `entry_timestamp`（UNIXミリ秒）と `action_weight`（例: リツイート=1.5、招待=2.0）を付与し、抽選確率に動的影響を与えます。
- **公平抽選ロジック（FEL-2024）**: ブロックチェーン由来のコミットメント方式を採用。抽選実行時にシード値 `seed_hex`（64文字の16進数）と参加者リストのハッシュチェーンを組み合わせ、決定論的かつ検証可能な抽選結果を生成します。抽選結果は `lottery_result` テーブルに保存され、外部監査ツールからも確認可能です。
- **リアルタイムダッシュボードと自動レポート生成**: WebSocket を用いて、現在の参加者数、エントリー率、残り時間を100ミリ秒間隔で更新。キャンペーン終了後、PostgreSQL 上の集計ビュー `campaign_summary` から自動でPDFレポートを生成し、指定のS3バケットに保存します。レポートにはクリックスルー率（CTR）、エンゲージメントスコア、地域別参加分布が含まれます。

### 使い方

1. **キャンペーン設定と参加ルールの定義**
   - `POST /api/v1/campaigns` エンドポイントに対し、`campaign_name`（文字列）、`platforms`（配列：["twitter", "discord"]）、`entry_deadline`（ISO 8601形式）、`max_participants`（整数）をJSONで送信。
   - ルールエンジン `RuleSet` を定義し、各アクションの `action_weight` と `required_actions`（例：フォロー必須）を設定。設定は `rules_config` テーブルに保存され、キャンペーン開始後も変更不可とする。

2. **参加者データの収集と検証**
   - Twitter連携の場合、`/api/v1/entries/twitter` に `tweet_id` と `user_id` を送信。サーバー側でTwitter APIを用いてリツイートの有無を確認後、`participant_hash` を生成し `entries` テーブルに挿入。
   - 重複エントリー検出は `participant_hash` と `campaign_id` の複合ユニーク制約で実施。重複時はHTTP 409を返却。

3. **抽選実行と結果公開**
   - キャンペーン終了後、`/api/v1/lotteries/run` に `campaign_id` を送信。サーバーは全参加者の `entry_timestamp` と `action_weight` を取得し、`FEL-2024` アルゴリズムで当選者を選出。
   - 結果は `lottery_result` テーブルに `winner_hash`、`seed_hex`、`execution_timestamp` と共に保存。公開用エンドポイント `/api/v1/lotteries/{campaign_id}/result` で誰でも検証可能。

### ベストプラクティス

- **抽選シードの管理**: 抽選シード `seed_hex` は、キャンペーン開始時に公開し、参加者全員がそのハッシュを確認できるようにする。抽選後、シードと参加者リストのハッシュチェーンを公開することで、結果の改ざんが不可能であることを証明できる。運用時は、シード生成に `/dev/urandom` を使用し、外部からの推測を防ぐ。
- **キャンペーン終了後のデータアーカイブ**: キャンペーン終了後、`entries` テーブルと `lottery_result` テーブルは即座に読み取り専用に変更し、`archived_campaigns` スキーマに移動する。これにより、運用中のキャンペーンデータベースのパフォーマンスを維持しつつ、後日の監査や分析に備える。アーカイブデータは年次でS3 Glacierに移行し、コストを最適化する。
