# giveawaytracker - GIV_027

## 概要

giveawaytracker は、大規模キャンペーンのプレゼント抽選・配布プロセスを完全自動化するプロフェッショナルツールです。キーワード: プレゼント抽選追跡ツール、景品管理システム、イベント配布トラッキング。全処理をミリ秒単位で記録し、PostgreSQL との連携によりデータ整合性を担保します。

## 詳細な内容

### giveawaytracker とは

giveawaytracker は、複数チャネル（Twitter、Discord、Web フォーム）からの応募をリアルタイムで統合し、抽選ロジックから当選者通知、在庫引当、配送ステータス管理までを一貫して実行するエンタープライズグレードの配布管理基盤です。応募重複排除、抽選バイアス補正、レート制限（Rate Limiting）を標準実装しています。

### コア機能

- **マルチチャネル応募統合 & リアルタイム重複排除**: Twitter API v2、Discord Bot、カスタム Webhook から流入する応募データを `applicant_id` と `channel_source` の複合キーでユニーク管理。同一ユーザーが複数チャネルから応募した場合、`dedup_window_ms`（デフォルト 300,000ms）内で自動マージし、不正多重応募を防止。
- **確率的抽選エンジン & 当選バイアス制御**: 重み付け抽選アルゴリズム（`weight_field` パラメータで任意のユーザー属性を指定可能）を実装。抽選結果は `lottery_timestamp`（Unix ミリ秒）と `seed_hash`（SHA-256）により完全トレーサブル。抽選バイアス補正として、過去の当選履歴に基づく `recent_win_penalty` 係数を動的に適用。
- **在庫引当 & 配送ステータス自動同期**: PostgreSQL の `inventory_reservation` テーブルを使用し、当選決定と同時に在庫を `RESERVED` 状態にロック。`reservation_timeout_ms`（デフォルト 86,400,000ms = 24時間）を超過した未確定予約は自動解放。配送業者 API（ヤマト運輸、佐川急便）と連携し、`tracking_number` と `delivery_status` を 5 分間隔でポーリング更新。

### 使い方

1. **初期設定 & チャネル登録**: プロジェクトルートの `config/channels.yaml` に `channel_id`、`api_endpoint`、`rate_limit_per_minute`（例: 60）を記述。`giveawaytracker init --channels channels.yaml` を実行すると、PostgreSQL に `channels` テーブルが作成され、各チャネルの Webhook エンドポイントが自動生成される。生成されたエンドポイント URL を各プラットフォームの Developer Portal に登録する。

2. **キャンペーン作成 & 抽選パラメータ設定**: `giveawaytracker campaign create --name "2025春の大感謝祭" --start 2025-04-01T00:00:00Z --end 2025-04-07T23:59:59Z --max_winners 100 --weight_field "follower_count"` を実行。`weight_field` に基づき、フォロワー数が多いユーザーほど当選確率が高くなる重み付け抽選が有効になる。抽選は `--end` 時刻を過ぎた直後のバッチジョブ（cron 式 `0 0 * * *`）で自動実行される。

3. **結果確認 & 配送管理**: `giveawaytracker results export --campaign_id GIV_027 --format csv --output ./results.csv` で当選者一覧をエクスポート。各当選者の `winner_id` に対して `giveawaytracker shipping assign --winner_id WIN_001 --carrier yamato --product_id PRD_123` を実行すると、在庫予約と配送ラベル生成が一括処理される。配送ステータスは `giveawaytracker shipping status --tracking_number 1234567890` でリアルタイム確認可能。

### ベストプラクティス

- **抽選前に負荷テストを実施**: 本番キャンペーン開始前に、`giveawaytracker benchmark --concurrent_users 10000 --duration_sec 300` を実行し、PostgreSQL のコネクションプールサイズ（推奨: `max_connections=200`）と API のレート制限値を調整する。特に Twitter API は `GET /2/tweets/search/stream` のレート制限が 50 リクエスト/15分と厳しいため、`rate_limit_per_minute` を適切に設定すること。
- **在庫予約のタイムアウト値を事業要件に合わせて調整**: 当選者への連絡期限が 48 時間の場合は、`reservation_timeout_ms` を `172,800,000ms` に設定。タイムアウトが短すぎると、当選者が連絡前に予約が解放され在庫が無駄になる。長すぎると、在庫が長期ロックされ他のキャンペーンに支障が出る。過去のデータから `avg_confirmation_time_ms` を計測し、その 1.5 倍を推奨値とする。
