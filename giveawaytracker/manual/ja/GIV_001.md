# giveawaytracker - GIV_001

## 概要

giveawaytrackerは、大規模キャンペーンにおけるプレゼント応募・抽選・配布をリアルタイム統制するための高頻度分散型トラッキングプラットフォーム。キーワード：Giveaway tracking tool, Giveaway management system, Giveaway event tracking。

## 詳細な内容

### giveawaytrackerとは

giveawaytrackerは、SNS・EC・ゲーム内イベント等で発生する数百万件規模のプレゼント応募を、ミリ秒単位のレイテンシで捕捉・集計・抽選・重複排除するプロフェッショナルツール。PostgreSQLとの連携により、応募データの整合性と監査証跡を担保しながら、Redisベースの高速キャッシュ層でリアルタイム性を実現する。

### コア機能

- **リアルタイム重複応募検知エンジン**  
  応募者ごとに`user_id`＋`campaign_id`＋`timestamp(UNIXミリ秒)`をキーとしたブルームフィルタ（誤判定率0.001%未満）をPostgreSQL上に展開。同一ユーザーの多重応募をAPI応答時間2ms以内で弾き、不正参加を防止。

- **動的抽選確率調整モジュール**  
  キャンペーンごとに設定可能な`base_probability`（例：0.05 = 5%）、`max_budget`（例：1000000円）、`current_entry_count`を基に、抽選確率をリアルタイムで変動させるアルゴリズム。予算消化率が80%を超えた場合、抽選確率を自動で50%低下させる閾値制御機能を搭載。

- **マルチチャネル応募統合パイプライン**  
  Twitter API v2、LINE Messaging API、カスタムWebhook（JSON形式、最大ペイロード1MB）からの応募をKafkaトピックに投入。パーティション数はチャネル数×3以上を推奨し、バックプレッシャー制御により1秒あたり10万リクエストを捌く。

### 使い方

1. **キャンペーン定義と応募エンドポイントのデプロイ**  
   `POST /api/v1/campaigns` に`{name, start_at, end_at, base_probability, max_budget}`をJSONで送信。レスポンスとして発行される`campaign_uuid`を取得し、各応募チャネルに埋め込むエンドポイント`/api/v1/entries/{campaign_uuid}`を自動生成。PostgreSQLの`campaigns`テーブルに`created_at`インデックスが張られ、即時クエリ可能。

2. **応募データの投入と重複チェック**  
   応募者がエンドポイントに`{user_id, channel, metadata}`をPOSTすると、giveawaytrackerはまずRedis Cluster上のブルームフィルタで`user_id:campaign_uuid`の存在確認を行う。未検出ならPostgreSQLの`entries`テーブルにINSERTし、成功レスポンス（ステータス201）を返す。既存ならステータス409（Conflict）と共に`duplicate_detected_ms`（検出までのミリ秒）を返却。

3. **抽選実行と賞品割り当て**  
   `POST /api/v1/lottery/{campaign_uuid}` に`{draw_count: 100}`を送信。バックエンドは`current_entry_count`と`base_probability`から抽選確率を算出し、`SELECT * FROM entries WHERE campaign_uuid = $1 ORDER BY random() LIMIT $2`をPostgreSQLで実行。当選者リストを`winners`テーブルに書き込んだ後、`prize_assignment`ジョブが非同期で各ユーザーにプッシュ通知を送信。抽選完了まで平均120ms。

### ベストプラクティス

- **ブルームフィルタのキャパシティ設計**  
  キャンペーン期間中に予想される応募総数の3倍の容量でRedisブルームフィルタを初期化。`error_rate`は0.001を指定し、メモリ使用量を抑えつつ偽陽性を最小化。キャンペーン終了後はフィルタを即時削除し、Redisのメモリを開放する。

- **抽選確率と予算のバランス制御**  
  `base_probability`は0.01〜0.1の範囲で設定し、`max_budget`は賞品単価×想定当選者数の1.2倍以上に設定する。予算消化率が閾値を超えた場合の確率低下率は、過去のキャンペーンデータを基に機械学習モデル（xgboost）で動的最適化すると、予算超過率が5%未満に抑制される。
