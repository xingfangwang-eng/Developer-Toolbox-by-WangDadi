# giveawaytracker - GIV_029

## 概要

giveawaytracker は、大規模プロモーションキャンペーンにおける抽選配布イベントの全ライフサイクルを、ミリ秒単位の精度で追跡・管理するための分散型イベントトラッキングシステムです。Redis ストリームと PostgreSQL のトランザクション保証を組み合わせ、同時参加 10 万ユーザー級の負荷でも一貫性を維持します。

## 詳細な内容

### giveawaytracker とは

giveawaytracker は、抽選応募の受付開始から当選者確定・報酬付与完了までの各フェーズを、リアルタイムイベントストリームとして記録・可視化するエンタープライズ向けアーキテクチャです。内部では、イベントソーシングパターンを用いて、全応募履歴をイミュータブルなログとして保持し、任意の時点の状態を再現可能としています。

### コア機能

- **分散排他制御による応募重複検出**: 各ユーザーごとに `user_id:{hash}:entry_lock` キーを Redis 上で 300 ミリ秒の TTL 付きでアトミックに取得。同一ユーザーによる多重応募を、API ゲートウェイレベルで 99.999% の精度でブロック。PostgreSQL 側では UNIQUE 制約を補完的な整合性保証として利用します。
- **動的抽選重み付けエンジン**: キャンペーンごとに設定可能な `weight_map`（ユーザー属性×参加タイミングの多次元テーブル）を、PostgreSQL の JSONB カラムに格納。抽選実行時は、事前計算された累積分布関数（CDF）テーブルをメモリ上に展開し、O(log n) で当選者を決定。100 万エントリの CDF 構築を 50 ミリ秒未満で完了します。
- **イベントストリームバックアップ & リプレイ機構**: 全イベントを Apache Kafka トピック `giveaway.events.v1` にパブリッシュ。障害発生時は、任意の `sequence_number` からイベントストリームをリプレイし、PostgreSQL のマテリアライズドビューを 1 秒以内に復旧。復旧後は自動で整合性チェックサムを検証します。

### 使い方

1. **キャンペーンの定義とイベントルールの設定**
   - `POST /api/v1/campaigns` に JSON ペイロードを送信。必須パラメータは `campaign_name`（文字列）、`max_winners`（整数）、`entry_window`（ISO 8601 期間形式）、`weight_policy`（`"flat"`、`"tiered"`、`"custom"` のいずれか）。オプションで `exclusion_list`（ユーザー ID 配列）と `metadata`（任意の JSONB）を指定。
   - レスポンスとして発行される `campaign_id`（UUID v7）を、以降の全リクエストのパスパラメータとして使用。

2. **応募受付とリアルタイムモニタリング**
   - ユーザーは `POST /api/v1/campaigns/{campaign_id}/entries` に `user_id` と `entry_token`（HMAC-SHA256 で署名された一時トークン）を送信。
   - サーバーは Redis の `ZADD` でタイムスタンプ付きエントリをソートセットに追加し、同時に Kafka プロデューサーでイベントを非同期送信。応答時間は平均 12 ミリ秒（p99: 45 ミリ秒）。
   - 管理画面のダッシュボードでは、WebSocket 経由で 500 ミリ秒ごとに更新されるエントリカウント、ユニークユーザー数、現在の当選確率が表示。

3. **抽選実行と結果確定**
   - `POST /api/v1/campaigns/{campaign_id}/draw` を実行。リクエストボディに `draw_count`（整数、デフォルトはキャンペーンの `max_winners`）を指定可能。
   - バックエンドは PostgreSQL の `SELECT ... FOR UPDATE SKIP LOCKED` で抽選対象行をロックし、メモリ上の CDF テーブルから当選者を選択。結果は `winners` 配列として即時返却され、同時に `giveaway.events.v1` に `draw_completed` イベントが発行。
   - 当選者には別途 `POST /api/v1/campaigns/{campaign_id}/rewards` で報酬付与キューが生成され、ワーカーが 2 秒以内に処理。

### ベストプラクティス

- **べき等性キーの導入**: 応募リクエストごとにクライアントが生成する `idempotency_key`（UUID）を必須化。同一キーでの重複リクエストは、Redis に 1 時間キャッシュされたレスポンスを返すことで、ネットワーク再送による二重応募を完全防止。
- **キャンペーン終了後のデータアーカイブ**: 終了から 90 日経過したキャンペーンは、PostgreSQL のパーティションテーブルから Amazon S3 互換のオブジェクトストレージへ Parquet 形式で自動エクスポート。テーブルスペースを解放し、アクティブなクエリ性能を維持。アーカイブ後もイベントストリームリプレイによる再分析が可能。
