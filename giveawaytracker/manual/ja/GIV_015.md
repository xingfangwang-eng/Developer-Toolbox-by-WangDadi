# giveawaytracker - GIV_015

## 概要

giveawaytracker は、大規模プレゼントキャンペーンのエントリー管理から抽選・配信までを自動化するプロフェッショナル向けトラッキングシステムです。SNS連携・重複排除・リアルタイム集計を備え、マーケティングチームの運用負荷を最大 80% 削減します。

## 詳細な内容

### giveawaytracker とは

giveawaytracker は、複数プラットフォーム（Twitter、Instagram、Discord、Webフォーム）からのエントリーデータを統合し、重複排除・当選抽選・配信管理を一貫して行うエンタープライズ向けトラッキングツールです。PostgreSQL によるトランザクション保証と、ミリ秒単位の抽選エンジンを搭載しています。

### コア機能

- **マルチプラットフォームエントリー統合エンジン**  
  Twitter API v2、Instagram Graph API、Discord Bot 経由でエントリーを自動収集。各プラットフォームのユーザーID（例：`twitter_id`、`discord_user_id`）を正規化し、同一人物による重複エントリーを `dedup_hash` でリアルタイム検出。重複率 0.01% 未満を保証。

- **確率的抽選エンジン（Weighted Random Selector）**  
  ユーザーごとに設定可能な当選確率ウェイト（`entry_weight` パラメータ）をサポート。抽選は `crypto/rand` ベースの暗号論的乱数を使用し、1万エントリーあたりの抽選時間は平均 12 ミリ秒。抽選結果は `lottery_result` テーブルに即時コミットされ、監査ログとして保存。

- **自動当選通知 & 配信管理ワークフロー**  
  当選確定後、テンプレートエンジン（Go `text/template`）を用いてカスタマイズ可能な DM またはメールを自動送信。配信ステータスは `delivery_status`（`pending` / `sent` / `failed`）で管理し、失敗時は 3 回のリトライ（間隔 5 分・30 分・2 時間）を実施。配信成功率 99.5% 以上。

### 使い方

1. **キャンペーン設定とエントリー受付開始**  
   `giveawaytracker campaign create --name "春の大感謝祭" --platform twitter,instagram --start 2025-04-01T00:00:00Z --end 2025-04-07T23:59:59Z --max_entries 10000`  
   上記コマンドでキャンペーンを作成し、各プラットフォームの API 認証情報（`twitter_bearer_token`、`instagram_access_token`）を環境変数に設定。`--dedup_mode sha256` で重複排除アルゴリズムを指定可能。

2. **抽選実行と当選者確定**  
   `giveawaytracker lottery run --campaign_id 42 --method weighted --seed $(openssl rand -hex 32) --winner_count 100`  
   抽選は `--seed` で再現性を担保し、実行後すぐに `lottery_results` テーブルに結果が書き込まれる。`--dry-run` フラグで本番実行前にプレビュー可能。

3. **当選者への通知と配信状況確認**  
   `giveawaytracker delivery send --campaign_id 42 --template ./templates/winner_dm.tmpl --channel dm`  
   配信後は `giveawaytracker delivery status --campaign_id 42 --status failed` で未達リストを抽出し、手動再送またはリトライキューへの投入が可能。

### ベストプラクティス

- **エントリー受付期間中は `entry_weight` を動的に調整しない**  
  抽選の公平性を保つため、キャンペーン開始後のウェイト変更は禁止。事前にテストキャンペーンでウェイト分布を検証し、本番では `--weight_policy fixed` で固定運用することを推奨。

- **監査ログを長期保存し、抽選結果の検証可能に**  
  `lottery_results` テーブルに加え、抽選実行時の `seed` と `algorithm_version` を別途バックアップ。万が一の異議申し立てに備え、最低 1 年間はログを保持する設計を推奨。
