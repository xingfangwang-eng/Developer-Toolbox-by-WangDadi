# giveawaytracker - GIV_025

## 概要

giveawaytracker は、大規模キャンペーンにおける Giveaway イベントのリアルタイム追跡・管理体制をゼロから構築するためのプロフェッショナルツールです。本プロジェクトは、**Giveaway tracking tool**、**Giveaway management system**、**Giveaway event tracking** の3つのキーワードを軸に、エンタープライズグレードの配信・抽選・分析を実現します。

## 詳細な内容

### giveawaytracker とは

giveawaytracker は、抽選イベントのエントリー受付、当選者抽選、配信ステータス管理、参加者の行動分析を統合的に処理する、マイクロサービスアーキテクチャに基づくバックエンドシステムです。特に、**ミリ秒単位のエントリー受付**と**PostgreSQL との連携によるトランザクション管理**を前提として設計されています。

### コア機能

- **リアルタイムエントリーパイプライン**  
  参加者のエントリーリクエストを、Kafka を介した非同期キューで受信。バーストトラフィック時でも、PostgreSQL の `INSERT` 処理をバッチ化し、1秒間に最大 50,000 件のエントリーを永続化可能。エントリー時刻は **ミリ秒単位のタイムスタンプ** で記録され、抽選の公平性を担保。

- **確率重み付き抽選エンジン**  
  各エントリーに対して、ユーザー属性や過去の参加履歴に基づく**動的重み付け**を実施。抽選アルゴリズムには **Fisher-Yates シャッフル** を採用し、当選確率の偏りを排除。抽選結果は即座に PostgreSQL の `winners` テーブルに書き込まれ、同時にキャッシュ層（Redis）に保存されるため、**100 ミリ秒未満**での結果取得が可能。

- **配信ステータス追跡とリマインダー自動化**  
  当選者に対して、メールまたは SMS による配信ステータスを追跡。`delivery_status` カラム（`pending`, `sent`, `claimed`, `expired`）を管理し、未応答の当選者には **24 時間ごとに自動リマインダー** を送信。リマインダーの送信間隔は環境変数 `REMINDER_INTERVAL_HOURS` で調整可能。

### 使い方

1. **キャンペーン定義とエントリー受付開始**  
   `POST /api/v1/campaigns` エンドポイントに、キャンペーン名、開始時刻、終了時刻、最大エントリー数を指定。例：  
   ```json
   {
     "name": "2025春の大感謝祭",
     "start_at": "2025-04-01T00:00:00Z",
     "end_at": "2025-04-07T23:59:59Z",
     "max_entries": 100000
   }
   ```  
   応答として `campaign_id` が返却され、この ID を元にエントリー受付が開始されます。エントリーは `POST /api/v1/entries` にて、ユーザーIDと任意のメタデータを送信することで完了。

2. **抽選実行と結果確認**  
   `POST /api/v1/campaigns/{campaign_id}/draw` を呼び出すと、抽選エンジンが起動。パラメータとして `number_of_winners`（当選者数）を指定。抽選後、`GET /api/v1/campaigns/{campaign_id}/winners` で当選者リストを取得可能。各当選者には固有の `claim_token` が発行され、このトークンを使って当選者は賞品を請求できます。

3. **配信ステータス更新とリマインダー設定**  
   当選者の配信ステータスは `PATCH /api/v1/winners/{winner_id}` で更新。`status` フィールドに `sent` や `claimed` を指定。リマインダーの自動化は、バックグラウンドジョブ（Celery または Sidekiq）が定期的に `delivery_status = 'sent'` かつ `updated_at` が `REMINDER_INTERVAL_HOURS` 時間以上経過したレコードを抽出し、リマインダーを送信します。

### ベストプラクティス

- **エントリー受付のレート制限**  
  エントリーAPIには、ユーザーごとに **1秒間に10リクエスト** のレート制限を実装することを推奨。これにより、不正な大量エントリーやシステム負荷を軽減。制限を超えた場合、HTTP 429（Too Many Requests）を返却。

- **抽選結果の監査ログ**  
  抽選エンジンが実行されるたびに、`audit_logs` テーブルに抽選時刻、使用した乱数シード、当選者IDのリストを記録。これにより、後日抽選の公平性を検証可能。ログの保持期間は `AUDIT_LOG_RETENTION_DAYS` で設定。
