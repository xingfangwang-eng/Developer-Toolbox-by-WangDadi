# giveawaytracker - GIV_012

## 概要

giveawaytracker は、大規模キャンペーンにおけるプレゼント応募の重複排除・当選確率のリアルタイム補正・抽選プロセスの完全監査証跡を実現する、エンタープライズグレードの抽選管理基盤です。SNS連携による自動エントリー収集から、PostgreSQL上のトランザクション抽選、当選者通知までのパイプラインを単一のCLIで制御します。

## 詳細な内容

### giveawaytracker とは

giveawaytracker は、複数プラットフォーム（Twitter、Instagram、Discord）から非同期に流入する応募データをミリ秒単位で統合し、抽選ロジックに適用するための高スループット分散システムです。内部ではApache Kafkaをメッセージキューとして採用し、応募の重複チェックにはBloom Filter（偽陽性率0.01%未満）を利用することで、秒間10万件のエントリー処理を実現します。

### コア機能

- **リアルタイム重複排除エンジン**  
  ユーザーID、メールアドレス、IPアドレス、デバイスフィンガープリントの4要素を用いたマルチカラム重複チェックを実施。重複と判定された応募は、`duplicate_reason` フィールドに原因コード（例: `DUP_IP_SAME_HOUR`）を記録し、当選確率計算から自動除外します。

- **動的当選確率補正モジュール**  
  抽選プール内の応募者ごとに、過去の当選履歴・応募時刻・プラットフォーム重み（例: Discord: 1.0、Twitter: 1.2）を反映したスコアリングを実施。`--weight-factor` パラメータで補正強度を0.0～2.0の範囲で調整可能。内部ではPostgreSQLのウィンドウ関数を用いたバッチスコアリングを100ms以内に完了します。

- **完全監査証跡生成機能**  
  抽選実行のたびに、全応募レコードのスナップショット、抽選ロジックのバージョン（`--logic-hash`）、乱数生成シード（`/dev/urandom` 由来）をJSON Lines形式で出力。出力先はS3互換ストレージを指定可能で、後日第三者による検証を可能にします。

### 使い方

1. **キャンペーン初期化と応募データの取り込み**  
   ```bash
   giveawaytracker init --campaign-id CAMPAIGN_202503 --platforms twitter,instagram,discord \
       --dedup-window 3600 --bloom-false-positive 0.0001
   ```
   上記コマンドで、Bloom Filterの偽陽性率を0.01%に設定し、重複チェックの時間窓を3600秒（1時間）に指定。その後、各プラットフォームのAPIキーを環境変数 `GIVEAWAY_TRACKER_API_KEYS` に設定し、`giveawaytracker ingest --batch-size 5000` でストリーミング取り込みを開始します。

2. **抽選実行とスコアリングパラメータの調整**  
   ```bash
   giveawaytracker draw --campaign-id CAMPAIGN_202503 --winners 10 \
       --weight-factor 1.5 --exclude-duplicates true --output-format json
   ```
   `--weight-factor 1.5` により、Twitter経由の応募を通常の1.5倍の当選確率に補正。抽選結果は標準出力にJSON形式で出力され、同時にPostgreSQLの `draw_results` テーブルにトランザクションとして記録されます。

3. **監査証跡のエクスポートと検証**  
   ```bash
   giveawaytracker export-audit --campaign-id CAMPAIGN_202503 --from 2025-03-01T00:00:00Z \
       --to 2025-03-07T23:59:59Z --destination s3://audit-bucket/campaigns/
   ```
   指定期間内の全抽選イベントをJSON Lines形式でS3にエクスポート。出力ファイルには、各抽選で使用された乱数シード、スコアリング関数のハッシュ値、応募者ごとの重複判定結果が含まれます。検証用ツール `giveawaytracker verify --audit-file s3://...` を用いて、抽選の公正性を第三者に証明できます。

### ベストプラクティス

- **重複チェックの時間窓はキャンペーン期間に応じて最適化する**  
  短期キャンペーン（24時間以内）では `--dedup-window 86400`（24時間）に設定し、同一ユーザーの複数回応募を完全に排除。長期キャンペーン（1週間以上）では `--dedup-window 3600` に短縮し、正当な再応募（例: 別プラットフォームからの参加）を許可する設計にすることで、参加者のエンゲージメントを最大化します。

- **抽選前のドライランモードでスコアリング分布を事前確認する**  
  本番抽選の前に `giveawaytracker dry-run --campaign-id CAMPAIGN_202503 --iterations 1000` を実行し、スコアリングの偏りや重複排除の漏れがないかを検証。特に `--weight-factor` の値が極端な場合（2.0に近い場合）、特定プラットフォームの応募者に不当な優遇が発生していないかをヒストグラムで確認します。
