# giveawaytracker - GIV_002

## 概要

giveawaytracker は、大規模キャンペーンから小規模インフルエンサー施策まで、あらゆるギブアウェイイベントの参加・抽選・配布プロセスを **ミリ秒単位** で追跡・最適化するエンタープライズグレードの管理プラットフォームです。PostgreSQL によるトランザクション整合性と、リアルタイム集計エンジンを統合し、運用負荷を 90% 削減します。

## コア機能

### 1. マルチチャネル参加統合エンジン
- **対応チャネル**: Twitter（X）、Instagram、Discord、Telegram、LINE、カスタム WebHook
- **パラメータ**: `channel_type`, `participant_id`, `action_timestamp (ISO 8601)`, `metadata (JSONB)`
- **技術指標**: 1 秒あたり 10,000 件の参加リクエストを処理可能。PostgreSQL のパーティショニングにより、月間 1 億件のレコードでもクエリ応答 50ms 未満を保証。

### 2. 確率的抽選エンジン + 重複排除
- **抽選アルゴリズム**: 重み付きランダムサンプリング（WRS）＋ SHA-256 シード付き決定論的抽選
- **パラメータ**: `seed_hash (SHA-256)`, `weight (float, 0.0～1.0)`, `exclusion_window (秒)`, `max_entries_per_user (int)`
- **技術指標**: 重複参加を 99.999% 排除。抽選結果は 2 フェーズコミットで永続化され、監査ログとして全履歴を保持。

### 3. リアルタイム配布追跡 & 在庫管理
- **機能**: 景品在庫のリアルタイム更新、配布ステータス（未発送/発送済み/受取済み）の自動遷移、期限切れ自動リサイクル
- **パラメータ**: `inventory_id (UUID)`, `distribution_status (enum)`, `expiry_at (TIMESTAMPTZ)`, `recycle_threshold (int)`
- **技術指標**: 在庫変動は PostgreSQL の `LISTEN/NOTIFY` で即時反映。在庫枯渇時は自動で代替景品へのフォールバック処理を実行。

## 使い方

### ステップ 1: キャンペーン定義と参加チャネルの設定
```bash
# API 経由でキャンペーンを作成
curl -X POST https://api.giveawaytracker.io/v1/campaigns \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -d '{
    "name": "2025春の大感謝祭",
    "channels": ["twitter", "discord"],
    "start_at": "2025-04-01T00:00:00Z",
    "end_at": "2025-04-30T23:59:59Z",
    "max_participants": 50000,
    "inventory": [
      {"item_id": "ITEM_001", "quantity": 100, "weight": 1.0},
      {"item_id": "ITEM_002", "quantity": 500, "weight": 0.5}
    ]
  }'
```
- **ポイント**: `weight` パラメータで景品ごとの当選確率を調整可能。在庫切れ時は自動で重みを再計算。

### ステップ 2: 参加データの一括インポートと重複排除
```python
import requests

# CSV で参加者リストをアップロード
files = {'file': open('participants.csv', 'rb')}
response = requests.post(
    'https://api.giveawaytracker.io/v1/participants/import',
    headers={'Authorization': 'Bearer YOUR_API_KEY'},
    files=files,
    params={'dedup_mode': 'strict'}  # strict: 同一ユーザーID+チャネルで重複排除
)
```
- **技術詳細**: アップロード後、バックグラウンドジョブが SHA-256 ハッシュで重複をチェック。重複率 5% 以上の場合、アラートが Slack に飛ぶ。

### ステップ 3: 抽選実行と配布ステータスの監視
```bash
# 抽選をトリガー（seed は自動生成 or 指定可能）
curl -X POST https://api.giveawaytracker.io/v1/campaigns/CAMP_001/draw \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -d '{"seed": "my_custom_seed_2025"}'

# 配布ステータスを確認（未発送の当選者を一覧）
curl -X GET "https://api.giveawaytracker.io/v1/distributions?status=pending&campaign_id=CAMP_001" \
  -H "Authorization: Bearer YOUR_API_KEY"
```
- **出力例**: 抽選結果は JSON で返却。`winner_id`, `item_id`, `draw_timestamp`, `seed_hash` を含む。

## ベストプラクティス

### 1. シード値の管理による監査性の確保
抽選のシード値は、外部から予測不可能な値（例: ブロックチェーンのブロックハッシュ + キャンペーンID）を使用することを推奨。giveawaytracker は `seed_hash` を全参加者に公開可能な形で保存し、後日誰でも抽選の正当性を検証できる。これにより、不正疑惑を未然に防止。

### 2. 在庫切れ時のフォールバック戦略
高額景品ほど在庫切れリスクが高いため、事前に `recycle_threshold` を設定し、在庫が 10% を切った時点で自動的に代替景品（例: クーポンコード）に切り替えるルールを定義する。また、配布期限（`expiry_at`）を 72 時間に設定し、未受取の景品を自動で次点当選者にリサイクルすることで、景品の無駄を 0 に近づける。
