# giveawaytracker - GIV_023

## 概要

giveawaytracker は、大規模プレゼントキャンペーンの抽選・配布・トラッキングを統合管理するエンタープライズ向けプラットフォームです。本ツールは、同時 10 万件以上のエントリーをリアルタイム処理し、配布期限や重複応募を自動検知・排除します。キーワード：Giveaway tracking tool, Giveaway management system, Giveaway event tracking。

## 詳細な内容

### とは giveawaytracker?

giveawaytracker は、抽選ロジックの完全監査可能な実行と、参加者の行動履歴に基づく不正検知エンジンを備えた、高信頼性プレゼント管理システムです。PostgreSQL との連携により、全エントリー・配布ログをミリ秒単位で記録し、後日完全なトレーサビリティを提供します。

### コア機能

- **重複・多重応募排除エンジン**：参加者の IP アドレス、User-Agent、ブラウザフィンガープリントを組み合わせた独自のハッシュ照合を実施。同一ハッシュが 30 秒以内に 3 回以上検出された場合、自動で応募をブロックし、管理者にアラートを発報します。
- **時間制限付き抽選プール管理**：各キャンペーンに `start_at` / `end_at` パラメータを設定可能。抽選プールは PostgreSQL の `TIMESTAMPTZ` 型で厳密に管理され、期間外のエントリーは自動破棄。さらに、抽選実行時に `pg_try_advisory_lock()` を用いた排他制御を実装し、同時抽選による競合を防止します。
- **配布ステータスリアルタイム同期**：当選者に対して `claim_deadline`（デフォルト 72 時間）を設定。期限切れ未クレーム分は自動的に再抽選プールへ戻し、`reassigned_at` カラムに再割り当て時刻をミリ秒単位で記録。全履歴は JSONB 型で保存され、後日監査クエリを即座に実行可能。

### 使い方

1. **キャンペーン作成と抽選パラメータ設定**  
   `POST /api/v1/campaigns` に以下の JSON を送信します。  
   ```json
   {
     "campaign_name": "2026_Spring_Giveaway",
     "max_winners": 500,
     "start_at": "2026-04-01T00:00:00Z",
     "end_at": "2026-04-07T23:59:59Z",
     "claim_deadline_hours": 72,
     "duplicate_check_window_seconds": 30,
     "duplicate_max_attempts": 3
   }
   ```  
   応答として返却される `campaign_id` を後続のエントリー受付に使用します。抽選プールは自動的に `TIMESTAMPTZ` 範囲制約付きテーブルに格納されます。

2. **エントリー受付と重複チェックの実行**  
   `POST /api/v1/entries` に参加者情報を送信します。  
   ```json
   {
     "campaign_id": "abc-123",
     "participant_id": "user_456",
     "ip_address": "203.0.113.1",
     "user_agent": "Mozilla/5.0 ...",
     "fingerprint_hash": "a1b2c3d4e5f6..."
   }
   ```  
   サーバーは 30 秒以内の同一ハッシュ出現回数をカウントし、3 回を超えた場合 `409 Conflict` を返します。正常な場合は `entry_id` と `created_at`（ミリ秒精度）が返却されます。

3. **抽選実行と再抽選処理の自動化**  
   管理者が `POST /api/v1/campaigns/{campaign_id}/draw` を呼び出すと、`pg_try_advisory_lock()` で排他制御されたトランザクション内で抽選が実行されます。当選者には `winner_flag = true` が設定され、`claim_deadline` が自動計算・保存されます。期限切れ後、`/api/v1/campaigns/{campaign_id}/reassign` を呼ぶと、未クレーム分を再抽選プールへ戻し、新たな当選者を決定。全履歴は `entry_logs` テーブルの JSONB カラムに追記されます。

### ベストプラクティス

- **抽選前の負荷テストを必ず実施する**：同時エントリー数が 10 万件を超える場合、PostgreSQL の `max_connections` と `shared_buffers` のチューニングが必要です。特に `pg_try_advisory_lock()` を使用する抽選処理は、事前にステージング環境で 5 万件同時アクセスをシミュレートし、応答時間が 200 ミリ秒以内であることを確認してください。
- **重複排除の閾値はキャンペーン特性に合わせて調整する**：短期間で高頻度の応募が見込まれるキャンペーンでは、`duplicate_check_window_seconds` を 10 秒に短縮し、`duplicate_max_attempts` を 2 に設定することで、ボット対策を強化します。逆に、長期開催のキャンペーンでは 60 秒・5 回まで緩和し、正当な参加者の利便性を損なわないようにします。
