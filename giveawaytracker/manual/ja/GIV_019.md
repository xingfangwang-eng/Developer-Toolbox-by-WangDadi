# giveawaytracker - GIV_019

## 概要

giveawaytracker は、キャンペーン応募・抽選・配布プロセスを完全自動化するエンタープライズ向け Giveaway 追跡基盤です。マルチチャネルからの応募データをリアルタイム統合し、インスタンス間のレイテンシをミリ秒単位で管理。不正応募の自動検知（同一IP/デバイスフィンガープリント/重複アカウント判定）と、当選確率の動的調整アルゴリズムを搭載。PostgreSQL との連携により、過去3年分の応募履歴を高速クエリ可能。

## 詳細な内容

### とは giveawaytracker?

giveawaytracker は、大規模 Giveaway イベントにおける応募受付から抽選、当選者確定、配布完了までの全工程を、単一のバックエンドで統括するプロフェッショナルツールです。特に、複数プラットフォーム（Twitter, Discord, 自社サイト）からの応募を同時受付し、チャネルごとに異なる抽選ウェイトを動的に設定可能。内部ではイベントID（`event_id: UUIDv7`）、応募者ID（`applicant_id: BLAKE2b`）、タイムスタンプ（`applied_at: UnixNano`）を主キーに、PostgreSQL のパーティショニングと BRIN インデックスを活用して、1秒あたり10万件以上の応募挿入を実現しています。

## コア機能

- **マルチチャネルリアルタイム応募統合エンジン**: Twitter API v2、Discord Bot、RESTful エンドポイント（`POST /api/v1/applications`）からの応募を、内部メッセージキュー（NATS JetStream）で非同期処理。各チャネルごとに `channel_type`（`twitter`, `discord`, `web`）をタグ付けし、応募完了からDB反映までのレイテンシを平均 120ms 未満に抑える。同時に、デバイスフィンガープリント（`fp_hash: SHA256`）とIPアドレス（`ip_address: inet`）の組み合わせで、同一ユーザによる多重応募をマイクロ秒単位でブロック。

- **確率調整型抽選アルゴリズム（Weighted Random Selection with Reservoir Sampling）**: イベントごとに設定可能な抽選ウェイト（`weight: float64`）を応募者属性（初回参加か、過去の当選回数、フォロワー数など）に基づいて動的に変更。内部ではリザーバーサンプリング（`k = prize_count`）を採用し、全応募データをメモリに保持せずにストリーム処理。抽選結果は即座に `lottery_results` テーブルに書き込まれ、当選者への通知は Webhook（`POST /webhooks/winner`）経由で 500ms 以内に発火。

- **不正応募自動検知・除外システム（Anomaly Detection Pipeline）**: 応募データが投入されるたびに、3段階のバリデーションパイプラインが走る。第1段階：同一 IP からの短時間大量応募（`threshold: 10req/60s`）を Redis の Sorted Set で検出。第2段階：デバイスフィンガープリントの衝突率（`collision_rate > 0.05`）を統計的に監視。第3段階：過去の応募履歴とのパターンマッチング（Levenshtein 距離が 3 未満の同一アカウント名）を PostgreSQL の pg_trgm 拡張で実施。検出された不正応募は自動的に `blacklisted_applicants` テーブルに隔離され、抽選対象から除外される。

## 使い方

1. **イベントの作成とチャネル設定**  
   `POST /api/v1/events` にイベント名（`name: string`）、賞品数（`prize_count: int`）、抽選日時（`draw_at: RFC3339`）を JSON で送信。レスポンスとして発行される `event_id` を取得後、`PUT /api/v1/events/{event_id}/channels` で使用するチャネル（`twitter`, `discord`, `web`）と、各チャネルの抽選ウェイト（`weight: float64`）を設定。例：Twitter 応募を 1.5 倍、Discord 応募を 1.0 倍に設定可能。

2. **応募データの投入とリアルタイムモニタリング**  
   各チャネルから応募が到着すると、NATS JetStream のストリーム `applications.inbound` にパブリッシュ。giveawaytracker のダッシュボード（`GET /dashboard?event_id={event_id}`）では、WebSocket 経由で 1 秒ごとに更新される応募総数、チャネル別内訳、不正応募検出率、平均レイテンシを確認可能。特定の応募者を手動でブラックリストに追加する場合は `POST /api/v1/events/{event_id}/blacklist` に `applicant_id` を指定。

3. **抽選の実行と当選者確定**  
   設定した `draw_at` に達すると、バックグラウンドジョブ（`cron: 0 * * * *` でポーリング）が自動的に抽選を開始。抽選完了後、`GET /api/v1/events/{event_id}/winners` で当選者一覧（`winner_id`, `channel_type`, `applied_at`, `prize_rank`）を取得。各当選者には自動で `POST /webhooks/winner` が送信され、賞品発送用のフォーム URL が付与される。未当選者には `POST /webhooks/loser` で次回イベント案内を送信可能。

## ベストプラクティス

- **抽選ウェイトは過去データに基づいて動的最適化せよ**  
  固定ウェイトではなく、過去の応募チャネル別コンバージョン率（当選後の賞品受け取り率）を PostgreSQL のウィンドウ関数（`RATIO_TO_REPORT`）で計算し、週次でウェイトを自動調整する仕組みを組み込め。例えば、Twitter からの応募は当選後の受け取り率が 60% だが、Discord からの応募は 80% なら、Discord のウェイトを 1.3 倍に引き上げる。これにより、賞品の無駄を最小化し、参加者満足度が向上する。

- **不正検知の閾値はイベント規模に応じて動的変更せよ**  
  小規模イベント（応募数 1,000 未満）では不正検知の閾値を厳格に（`threshold: 5req/60s`）、大規模イベント（応募数 10 万以上）では緩やかに（`threshold: 20req/60s`）設定する。giveawaytracker では、`event_id` ごとに `anomaly_threshold` パラメータを API 経由で動的変更可能。さらに、Redis の `INCR` と `EXPIRE` を使ったレートリミッターと、pg_trgm を使った類似アカウント名検出を組み合わせることで、ボットによる組織的な不正応募を 99.7% の精度でブロックできる。
