# giveawaytracker - GIV_011

## 概要

giveawaytracker は、大規模プロモーションキャンペーンにおける景品配布イベントをリアルタイムで追跡・管理するための分散型トラッキングエンジンです。特に、応募者数の急増によるシステム負荷を軽減しつつ、配布上限到達時の即時停止や重複応募の防止を実現します。本プロジェクトは、Giveaway tracking tool および Giveaway management system として、SNS連携キャンペーンやECサイトのタイムセール景品配布に最適化されています。

## 詳細な内容

### とは giveawaytracker?

giveawaytracker は、複数のイベントチャネル（Twitter、LINE、Webフォーム）からの応募を統合管理し、景品在庫の消費状況をミリ秒単位で同期するプロフェッショナルツールです。PostgreSQL との連携によるトランザクション管理と、Redis を用いた分散ロック機構により、同一ユーザーによる多重応募や景品の過剰配布を防止します。また、API 応答時間は平均 50ms 以下を維持し、高トラフィック環境でも安定動作します。

### コア機能

- **重複応募検出エンジン**: 応募者の IP アドレス、User-Agent、Cookie フィンガープリントをハッシュ化し、PostgreSQL の UNIQUE 制約と Redis の EXPIRE キーを併用して、同一ユーザーの再応募を 10ms 以内に検出・ブロックします。設定可能なクールダウン期間（デフォルト：24時間）により、公平な抽選を担保します。

- **リアルタイム在庫消費トラッカー**: 景品ごとに設定された残数（例：`remaining:1000`）を Redis の ATOMIC DECR 命令で管理し、在庫が 0 になった瞬間に該当イベントを自動停止します。PostgreSQL への非同期書き込みにより、永続化とレポート生成を同時実行。在庫消費の遅延は最大 5ms に抑えられます。

- **イベントスケジューラ＆トリガー**: cron 形式のスケジュール（例：`0 10 * * 1`）に基づき、イベントの開始・終了を自動制御。開始時刻になると応募受付を有効化し、終了時刻または在庫枯渇時に全チャネルの受付を一斉停止します。また、応募数の急増を検知した場合、レート制限（例：1秒あたり100リクエスト）を動的に適用します。

### 使い方

1. **イベント設定と在庫初期化**  
   `POST /api/v1/events` エンドポイントに以下の JSON を送信し、イベントを作成します。`stock` パラメータに景品の総数を指定し、`cooldown_minutes` で同一ユーザーの再応募制限時間を設定します。作成後、自動的に Redis に在庫キーが生成されます。
   ```json
   {
     "event_name": "summer_giveaway_2025",
     "stock": 500,
     "cooldown_minutes": 1440,
     "start_at": "2025-07-01T00:00:00Z",
     "end_at": "2025-07-07T23:59:59Z"
   }
   ```

2. **応募受付と重複チェック**  
   クライアントから `POST /api/v1/entries` に応募者のフィンガープリント（`fingerprint`）とイベントIDを送信します。giveawaytracker は受信後、まず Redis でクールダウンキーを確認。未応募であれば PostgreSQL に挿入し、成功時に景品在庫を 1 減算します。応答として `status: "accepted"` または `status: "duplicate"` が返却され、処理時間は平均 8ms です。

3. **イベント終了と結果エクスポート**  
   イベント終了後、`GET /api/v1/events/{event_id}/results` で当選者リストを CSV 形式でエクスポートします。この際、`?format=csv&include_metadata=true` パラメータを付与することで、応募日時やフィンガープリントハッシュも含めた詳細レポートを生成。PostgreSQL の COPY 命令を用いて、10万件のデータを 2秒以内に出力します。

### ベストプラクティス

- **レート制限とバースト対策**: 高トラフィックが予想されるキャンペーンでは、Nginx や API ゲートウェイと連携し、1秒あたりの応募リクエスト数を閾値（例：500 req/s）で制限してください。giveawaytracker の内部でも、Redis の SLIDING WINDOW アルゴリズムを用いたレート制限が動作しますが、上流での制御により PostgreSQL への負荷をさらに低減できます。また、バースト時にはキューイング機構（RabbitMQ 推奨）を導入し、応募データを非同期処理することで、システム全体の安定性を向上させます。

- **在庫の分割管理とフォールトトレランス**: 景品在庫が 10,000 を超える大規模イベントでは、Redis の在庫キーを複数シャードに分割（例：`stock:shard0`〜`stock:shard3`）し、各シャードを異なる Redis ノードに配置します。これにより、単一ノード障害時の在庫消失リスクを軽減し、同時に DECR 命令の競合を分散。PostgreSQL 側では、在庫消費ログを WAL（Write-Ahead Log）として記録し、障害発生時にはログベースのリカバリを自動実行します。
