# giveawaytracker - GIV_024

## 概要

giveawaytracker は、大規模キャンペーン向けに設計された分散型プレゼント抽選・配布管理基盤です。リアルタイム抽選エンジン、参加者行動分析、マルチチャネル配布連携を統合し、エンタープライズグレードのスケーラビリティを実現します。

## 詳細な内容

### giveawaytracker とは

giveawaytracker は、SNSキャンペーン、ライブ配信連動型プレゼント、アプリ内イベントなど、複数チャネルにまたがるプレゼント企画の参加・抽選・当選通知・配布処理を一元管理するプロフェッショナルツールです。1秒あたり最大50,000リクエストの同時参加登録処理が可能で、抽選結果の決定から当選者へのプッシュ通知までを平均200ミリ秒未満で完了します。

### コア機能

- **分散抽選エンジン**: Redis Streams と PostgreSQL のパーティショニングを組み合わせた独自の抽選アルゴリズムを実装。抽選参加者数を `participant_count`、抽選枠数を `slot_limit`、抽選重み付け係数を `weight_factor` としてパラメータ化し、公平性担保のための乱数生成には `Xoroshiro128+` アルゴリズムを使用。抽選結果は即座に `lottery_results` テーブルに書き込まれ、トランザクション分離レベルは `SERIALIZABLE` に設定。

- **参加者行動リアルタイム分析**: 参加者のクリックストリーム、リファラルチェーン、コンバージョンパスをミリ秒単位で追跡。`user_behavior_pipeline` は Apache Kafka トピックを介してデータを収集し、`click_funnel` パラメータでファネル分析を実行。異常参加（bot判定）は `anomaly_score` が 0.85 を超えた時点で自動ブロック。

- **マルチチャネル配布連携**: Twitter、LINE、Slack、カスタム Webhook など10以上のチャネルに対応。配布テンプレートは `distribution_template` に基づき、`rate_limit_per_minute` と `concurrency_limit` をチャネルごとに設定可能。当選者へのコード発行は `voucher_code` を UUIDv7 形式で生成し、有効期限 `expires_at` と使用回数上限 `max_redemptions` を厳密に管理。

### 使い方

1. **キャンペーン設定と抽選パラメータの最適化**  
   `POST /campaigns` エンドポイントにて、`campaign_name`、`start_at`、`end_at`、`max_participants`、`winning_count` を必須パラメータとして指定。抽選重み付けが必要な場合は `weight_strategy` に `uniform`（均等）または `tiered`（階層別）を選択。`tiered` の場合、参加者の属性に応じた `tier_weights` をJSON配列で渡す。設定後、`POST /campaigns/:id/dry-run` で100万件の模擬参加データを用いた負荷テストを実行し、抽選処理時間が `target_latency_ms: 150` を超える場合は `batch_size` を調整。

2. **参加受付とリアルタイム抽選の実行**  
   参加者は `POST /entries` に `user_id`、`campaign_id`、`entry_metadata`（任意のJSON）を送信。バックエンドでは `entry_queue` に投入後、`lottery_scheduler` が `cron_expression` に従って抽選をトリガー。抽選実行時は `lottery_batch_id` を自動生成し、結果は `lottery_results` テーブルに `result_status: 'won' | 'lost'` として記録。当選者のみ `notification_queue` にプッシュされ、`webhook_url` が設定されている場合は即座にコールバックが発火。

3. **配布処理とレポートエクスポート**  
   当選者へのコード配布は `POST /distributions` で実行。`distribution_channel` に `twitter_dm` や `line_push` を指定し、`message_template` に `{{user_name}}様、当選コード: {{voucher_code}}` のように変数を含める。配布完了後、`GET /campaigns/:id/report?format=csv` で参加者数、当選率、配布成功率、平均応答時間（ミリ秒）を含む詳細レポートをエクスポート可能。`report_interval` パラメータで日次・週次・月次の集計を選択。

### ベストプラクティス

- **抽選の公平性担保と監査ログの完全保存**  
  抽選ロジックは決定論的乱数シード `seed` をキャンペーン作成時に指定可能。監査目的のため、全抽選トランザクションは `audit_logs` テーブルに `participant_hash`、`random_value`、`decision_timestamp`（ナノ秒精度）を保存。抽選結果に異議があった場合、`GET /audit/:campaign_id/verify` で再現可能な検証を提供。

- **負荷分散と障害耐性の設計**  
  参加者が急増した場合、`auto_scaling_threshold` を `1000 req/s` に設定し、Kubernetes HPA が自動でポッド数を `min_replicas: 3` から `max_replicas: 50` までスケールアウト。PostgreSQL は `pg_bouncer` によるコネクションプーリングと `read_replica` による読み取り負荷分散を前提とする。障害発生時は `circuit_breaker` が `failure_threshold: 5` で作動し、`fallback_queue` にリクエストを退避。
