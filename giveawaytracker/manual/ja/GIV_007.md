# giveawaytracker - GIV_007

## 概要

giveawaytrackerは、大規模プロモーション施策におけるGiveawayイベントの参加トラッキング、抽選ロジックの分散実行、及び当選者管理を統合するエンタープライズ向けGiveaway管理システムです。特に高トラフィック下での抽選整合性と参加者重複排除に特化した設計となっています。

## 詳細な内容

### とは giveawaytracker?

giveawaytrackerは、Giveawayイベントの開始から当選者確定までの全行程をリアルタイムで追跡・管理するためのプロフェッショナルツールです。Redis Streamsを利用したイベントソーシングアーキテクチャを採用しており、参加リクエストの重複排除（同一IP/デバイスフィンガープリント）をミリ秒単位で実施。PostgreSQLとの連携により、抽選結果の永続化と監査証跡の完全保証を実現します。

### コア機能

- **マルチウェイト抽選エンジン**: 参加者の属性（新規ユーザー/リピーター/地域/デバイス種別）に基づき、重み付き抽選を実行。抽選アルゴリズムはFisher-Yatesシャッフルを拡張した独自実装で、N=10万以上の参加者プールでも50ms以内に抽選完了。
- **分散デバイスフィンガープリント照合**: 参加リクエスト時に、UA、WebGLフィンガープリント、Canvasフィンガープリントの3要素をハッシュ化し、Bloom Filterによる高速重複チェックを実行。誤検出率は0.01%未満にチューニング。
- **当選者自動通知パイプライン**: 抽選完了後、即座にSQS/SNS経由でメール/SMSプッシュ通知を発火。当選者情報は暗号化（AES-256-GCM）してPostgreSQLに保存し、配信ステータスを追跡。

### 使い方

1. **Giveawayイベントの作成と抽選条件設定**
   - `POST /api/v1/giveaway` にリクエスト。必須パラメータ：`event_name`（文字列、最大64文字）、`prize_pool`（JSON配列、各賞品に `prize_id`、`quantity`、`weight` を指定）、`start_at` / `end_at`（ISO 8601形式のタイムスタンプ）、`dedup_window_ms`（重複排除の時間窓、デフォルト300000ms）。
   - 成功時、レスポンスに `event_id`（UUID v7）が返却。即座にRedisにイベントキャッシュが構築される。

2. **参加リクエストの受付と重複排除**
   - `POST /api/v1/entry` に `event_id`、`user_id`（任意の一意識別子）、`fingerprint`（Base64エンコードされたデバイスフィンガープリント）を送信。
   - サーバー側で `BloomFilter.Exists(fingerprint)` を実行。重複と判定された場合、HTTP 409 Conflictと共に `dedup_reason` フィールドを返却。
   - 正常時、`entry_id` と `position`（参加者プール内のインデックス）が返却。

3. **抽選実行と当選者確定**
   - `POST /api/v1/draw` に `event_id`、`draw_count`（抽選人数）を指定。内部で `SELECT * FROM entries WHERE event_id = $1 ORDER BY random() LIMIT $2` を実行し、PostgreSQLの `pg_shuffle` 拡張を利用して乱数生成。
   - 抽選結果は `winners` 配列としてレスポンス。同時に `INSERT INTO winners` で永続化し、通知ジョブをSQSにエンキュー。

### ベストプラクティス

- **抽選前の参加者プールのスナップショット化**: 高負荷イベントでは、抽選実行直前に `SELECT ... FOR UPDATE` で参加者テーブルをロックし、スナップショットアイソレーションレベルを設定。これにより、抽選中の新規参加リクエストによる結果不整合を防止。
- **通知配信のリトライ戦略**: 当選者通知はSQSの可視性タイムアウトを30秒に設定し、DLQ（デッドレターキュー）への移動は3回のリトライ失敗後。通知失敗時は、`winners` テーブルの `notification_status` カラムを `failed` に更新し、管理者ダッシュボードで再送トリガーを提供。
