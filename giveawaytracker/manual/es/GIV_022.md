# giveawaytracker - GIV_022

## Resumen

giveawaytracker es un sistema de monitoreo y análisis en tiempo real para campañas de sorteos y concursos en redes sociales. Diseñado para equipos de marketing y growth hackers, permite rastrear la participación, detectar fraudes y optimizar la distribución de premios con una precisión de milisegundos.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es una plataforma de ingeniería inversa de eventos sociales que captura, normaliza y analiza interacciones (likes, retweets, comentarios, menciones) desde APIs públicas de Twitter, Instagram y TikTok. Utiliza un motor de reglas configurable para validar participantes, calcular probabilidades de ganancia y ejecutar sorteos automatizados con integración directa a bases de datos PostgreSQL y colas de mensajería Redis.

### Características Principales

- **Sistema de detección de fraude por huella digital**: Analiza patrones de comportamiento (frecuencia de publicación, antigüedad de cuenta, ratio seguidores/seguidos) usando un modelo de regresión logística entrenado con 50.000+ muestras. Asigna un score de autenticidad (0.0 a 1.0) y bloquea automáticamente cuentas con score < 0.35 en menos de 200 ms.
- **Motor de sorteo probabilístico con sesgo controlado**: Permite configurar pesos personalizados por tipo de interacción (ej: retweet = 2.5 puntos, comentario = 1.8 puntos). El algoritmo de selección usa el método de Monte Carlo con 10.000 iteraciones por ronda, garantizando una distribución estadísticamente uniforme incluso con 1.000.000+ participantes.
- **API de webhooks en tiempo real con latencia < 50 ms**: Cada evento de participación (nuevo seguidor, like, retweet) se procesa mediante un pipeline de Apache Kafka, se almacena en PostgreSQL con índices compuestos (user_id + campaign_id + timestamp) y se notifica vía webhooks firmados con HMAC-SHA256. Soporta 10.000 peticiones concurrentes sin degradación.

### Cómo Usar

1. **Configurar campaña y reglas de elegibilidad**:
   - Ejecuta `giveawaytracker campaign create --name "SorteoVerano2025" --platform twitter --hashtag "#GanaYa" --min-followers 100 --max-age-days 30 --auth-threshold 0.4`
   - El sistema crea automáticamente una tabla en PostgreSQL (`campaigns.{id}_participants`) con columnas para user_id, interaction_type, timestamp, authenticity_score, y weight.
   - Verifica la configuración con `giveawaytracker campaign validate --id <campaign_id>` que ejecuta 100 consultas de prueba contra la API de Twitter.

2. **Iniciar monitoreo en vivo y depuración**:
   - Lanza el worker de escucha: `giveawaytracker listener start --campaign-id <id> --concurrency 50 --batch-size 100`
   - El worker consume eventos desde el tópico de Kafka `giveawaytracker.raw.events` y los transforma usando un pipeline de 3 etapas: normalización de datos (ISO 8601 para timestamps), cálculo de peso (según reglas), y actualización de la tabla de participantes.
   - Para depurar, usa `giveawaytracker listener status --campaign-id <id>` que muestra métricas en vivo: eventos/segundo, tasa de rechazo por fraude, y latencia promedio del pipeline.

3. **Ejecutar sorteo y exportar resultados**:
   - Una vez finalizada la campaña, ejecuta `giveawaytracker draw --campaign-id <id> --winners 10 --method weighted --seed 0xABC123`
   - El algoritmo genera 10 ganadores usando el método Monte Carlo con 10.000 iteraciones, almacena el resultado en `campaigns.{id}.draws` (incluyendo el seed y la lista de ganadores con sus pesos), y envía notificaciones vía webhook.
   - Exporta los datos completos a CSV con `giveawaytracker export --campaign-id <id> --format csv --output ./resultados.csv`, incluyendo columnas de autenticidad y peso.

### Mejores Prácticas

- **Segmentación de campañas por región horaria**: Configura ventanas de participación (ej: solo de 9:00 a 21:00 UTC) para evitar picos de bots nocturnos. Usa el parámetro `--time-window` en el comando `campaign create` y combínalo con un rate limiter de 5 solicitudes por minuto por IP para reducir falsos positivos en un 40%.
- **Pruebas A/B con diferentes umbrales de autenticidad**: Antes de una campaña masiva, ejecuta 3 campañas de prueba con umbrales 0.3, 0.4 y 0.5. Analiza la tasa de retención de participantes legítimos vs. bloqueados usando el comando `giveawaytracker campaign analyze --id <id> --metric retention`. El umbral óptimo suele estar entre 0.35 y 0.45 para campañas B2C.
