# giveawaytracker - GIV_017

## Resumen

Sistema de rastreo y gestión de sorteos en tiempo real, diseñado para plataformas de marketing digital y comunidades online. Procesa eventos de giveaway con latencia inferior a 200 ms, integra detección de fraude por huella digital y sincroniza datos con bases de datos PostgreSQL para análisis histórico de campañas.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es un motor de orquestación de sorteos que automatiza la captura, validación y asignación de premios en entornos multicanal (Discord, Twitter, Telegram). Utiliza un sistema de colas basado en Redis para manejar picos de 10.000 solicitudes concurrentes por segundo, con persistencia en PostgreSQL 15+ y replicación en clúster. Cada transacción de participación genera un hash SHA-256 único que se almacena en una tabla particionada por fecha (`giveaway_events_2025_03`), permitiendo consultas de agregación en menos de 50 ms.

### Características Principales

- **Asignación de premios con ponderación dinámica**: El módulo `weighted_rng` combina el peso del participante (calculado a partir de un sistema de puntos acumulados por referidos, antigüedad de cuenta y tasa de clics) con una semilla criptográfica generada desde `/dev/urandom`. El resultado se normaliza mediante una función de distribución exponencial inversa para evitar que cuentas con alta actividad monopolicen los sorteos. Parámetros configurables: `seed_entropy_bits` (mínimo 256), `weight_decay_factor` (por defecto 0.85), `max_weight_cap` (límite superior de 1000 puntos).

- **Detección de fraude multicapa**: Cada solicitud de participación pasa por un pipeline de 3 filtros: (1) verificación de IP mediante geolocalización MaxMind GeoIP2 con tolerancia de 3 países por evento, (2) análisis de huella digital del navegador (canvas fingerprinting + WebGL) con umbral de colisión < 0.001%, (3) validación de cuenta contra la API de la red social (OAuth 2.0) con tiempo de expiración de token configurable (`token_ttl_ms`: 300000). El resultado se registra en una tabla `fraud_logs` con índice compuesto por `participant_id` y `giveaway_id`.

- **Sincronización de estado en tiempo real**: Un WebSocket bidireccional (socket.io con compresión permessage-deflate) transmite actualizaciones de estado del sorteo (participantes totales, tiempo restante, premios asignados) a todos los clientes conectados. El servidor utiliza un buffer de eventos con capacidad de 10.000 mensajes antes de aplicar backpressure. El tiempo de propagación máximo es de 150 ms medido desde el servidor hasta el cliente más lejano (latencia de red incluida, percentil 95).

### Cómo Usar

1. **Configuración del entorno de desarrollo**: Clona el repositorio y ejecuta `docker compose up -d` para levantar los servicios de PostgreSQL 15, Redis 7.2 y el broker de mensajes RabbitMQ 3.12. Luego, modifica el archivo `.env.example` a `.env` y establece las variables `DB_HOST=localhost`, `DB_PORT=5432`, `REDIS_HOST=localhost`, `REDIS_PORT=6379`. Finalmente, ejecuta `npm run migrate:up` para aplicar las migraciones de esquema (`knex.js` con motor de migraciones).

2. **Creación de un giveaway programado**: Usa el endpoint `POST /api/v1/giveaways` con el siguiente payload JSON: `{"title": "Sorteo de verano", "prize_pool": 5000, "currency": "USD", "start_time": "2025-06-01T00:00:00Z", "end_time": "2025-06-30T23:59:59Z", "max_participants": 10000, "weight_config": {"seed_entropy_bits": 256, "weight_decay_factor": 0.85}}`. La respuesta incluye un `giveaway_id` UUID v4 que se usará para todas las operaciones posteriores.

3. **Validación de participación en vivo**: Para participar, envía un `POST /api/v1/giveaways/{id}/participate` con los headers `Authorization: Bearer <token_oauth>` y el body `{"platform": "discord", "user_id": "1234567890"}`. El servidor devuelve un código de estado `201` si la participación es válida, `409` si el usuario ya participó, `429` si superó el límite de 5 participaciones por hora (configurable en `rate_limit_per_hour`). Puedes monitorear el progreso en tiempo real suscribiéndote al WebSocket en `ws://localhost:3000/ws?giveaway_id={id}`.

### Mejores Prácticas

- **Particionamiento de tablas por tiempo**: Para eventos con más de 100.000 participantes, particiona la tabla `participants` por mes (`PARTITION BY RANGE (created_at)`) y crea índices compuestos en `(giveaway_id, participant_id)`. Esto reduce el tiempo de consulta de asignación de premios de 2 segundos a menos de 100 ms en tablas de 10 millones de registros.

- **Uso de colas asíncronas para notificaciones**: No sincronices el envío de correos electrónicos o mensajes de Discord con la asignación de premios. En su lugar, encola cada notificación en una cola RabbitMQ con prioridad (alta para ganadores, baja para no ganadores). El consumidor procesa 500 mensajes por segundo con un backoff exponencial en caso de fallo de la API externa (máximo 3 reintentos con intervalo de 1, 4 y 16 segundos).
