# giveawaytracker - GIV_016

## Resumen

giveawaytracker es un sistema de monitorización y gestión de sorteos en tiempo real, diseñado para rastrear, analizar y optimizar campañas de giveaway en múltiples plataformas. Proporciona métricas precisas de participación, detección de fraudes y automatización de notificaciones.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es una plataforma de análisis de eventos promocionales que integra scraping distribuido, procesamiento de datos en streaming y un motor de reglas para validar participantes. Utiliza un backend en Node.js con conexión a PostgreSQL para almacenar eventos, y Redis para caché de sesiones de usuario. Su arquitectura permite manejar hasta 10.000 solicitudes por segundo con una latencia media de 50 ms.

### Características Principales

- **Detección de fraude en tiempo real**: Analiza patrones de comportamiento como IP duplicadas, tiempos de registro inferiores a 200 ms entre intentos, y correlación de cuentas de redes sociales. Usa un modelo de machine learning (Random Forest) entrenado con 500.000 registros históricos, alcanzando una precisión del 98.7% en la identificación de bots.
- **Sincronización multiplataforma**: Conecta con APIs de Twitter, Instagram y Discord mediante OAuth 2.0. Cada conexión se autentica con tokens refresh cada 60 minutos, y los datos de participación se actualizan cada 15 segundos mediante WebSockets. Incluye un buffer de colas en RabbitMQ para evitar pérdida de eventos durante picos de tráfico.
- **Dashboard de métricas en vivo**: Renderiza gráficos con Chart.js y actualiza datos cada 500 ms mediante Server-Sent Events (SSE). Muestra KPIs como tasa de conversión por hora, distribución geográfica de participantes (usando geoIP con base de datos MaxMind), y tiempo medio de respuesta del sistema (objetivo: < 100 ms).

### Cómo Usar

1. **Configurar el rastreador de eventos**: Ejecuta `npm run tracker -- --platform=twitter --keywords="sorteo,regalo" --max-retries=3 --timeout=5000` para iniciar el scraping de menciones en Twitter. El sistema usará proxies rotativos (lista en `config/proxies.json`) para evitar rate limiting. Los datos se almacenan en la tabla `events` de PostgreSQL con campos como `event_id`, `platform`, `user_handle`, `timestamp` y `metadata` (JSONB).
2. **Validar participantes manualmente**: Usa el endpoint POST `/api/validate` con el body `{ "event_id": "GIV-20231015", "user_id": "12345" }`. El sistema ejecutará 3 reglas: verificación de cuenta antigua (>30 días), ausencia de reportes de spam, y coincidencia de IP con la región del evento. Devuelve un objeto JSON con `status` ("approved"/"rejected") y `confidence_score` (0-100).
3. **Generar informe de campaña**: Ejecuta `npm run report -- --event-id=GIV-20231015 --format=pdf --output=./reports/`. El script consulta la vista materializada `campaign_stats` (actualizada cada hora) y genera un PDF con tablas de participación por hora, gráfico de embudo de conversión y lista de ganadores. Usa Puppeteer para renderizar el HTML y wkhtmltopdf para la conversión.

### Mejores Prácticas

- **Escalar con colas de mensajería**: Para campañas con más de 50.000 participantes, implementa un worker de RabbitMQ que procese lotes de 1.000 registros por minuto. Configura el prefetch count a 100 para evitar sobrecargar la base de datos. Monitorea la cola con `rabbitmqadmin list queues name messages` y ajusta el número de workers dinámicamente con PM2.
- **Optimizar la detección de fraudes**: Entrena el modelo de ML con datos sintéticos generados por `scripts/generate_fake_participants.py` que simula bots con patrones de clic cada 100 ms. Usa el parámetro `--fraud-threshold=0.85` en el tracker para ajustar la sensibilidad. Almacena las predicciones en una tabla separada `fraud_predictions` con índices en `event_id` y `user_id` para consultas rápidas.
