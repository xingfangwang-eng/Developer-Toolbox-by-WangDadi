# giveawaytracker - GIV_019

## Resumen

giveawaytracker es un sistema de rastreo y gestión de sorteos en tiempo real, diseñado para auditar la distribución de premios, detectar patrones de fraude y optimizar campañas promocionales. Su núcleo procesa eventos de sorteo con una latencia inferior a **50 ms** y soporta hasta **10,000 transacciones concurrentes** por segundo. Se integra con **PostgreSQL 15+** y **Redis Streams** para garantizar consistencia y alta disponibilidad.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es un motor de análisis de eventos promocionales que captura, valida y correlaciona datos de participaciones, ganadores y recompensas. Utiliza un modelo de datos normalizado con **índices compuestos** sobre `participant_id`, `campaign_id` y `timestamp` para ejecutar consultas de agregación en menos de **100 ms** sobre tablas de hasta **50 millones de registros**. Además, implementa un sistema de alertas basado en **reglas heurísticas** (ej. umbrales de participación por IP, frecuencia de ganancia por usuario) que se evalúan en **cada inserción de evento**.

### Características Principales

- **Validación de integridad con doble firma temporal**: Cada evento de sorteo incluye un `server_timestamp` (precisión de **microsegundos**) y un `client_nonce` cifrado con HMAC-SHA256. El sistema rechaza automáticamente eventos con desviación mayor a **5 segundos** o nonce duplicado, previniendo ataques de repetición y manipulación de tiempo.

- **Dashboard de correlación multicampaña**: Agrupa datos de hasta **50 campañas simultáneas** en un único visor. Permite filtrar por `prize_pool_id`, `region` (ISO 3166-1 alpha-2) y `device_fingerprint`. Las métricas clave (tasa de conversión, valor medio del premio, tiempo medio de reclamo) se actualizan en **tiempo real** mediante WebSocket con **compresión Brotli**.

- **Exportación de auditoría con sello de integridad**: Genera archivos CSV/Parquet con un hash Merkle de todos los eventos exportados. Cada lote incluye un `audit_signature` (ECDSA P-384) que puede verificarse externamente. Soporta exportaciones incrementales usando `last_exported_id` (secuencia monotónica) y `export_window_minutes` (predeterminado: 1440).

### Cómo Usar

1. **Configurar el pipeline de ingesta de eventos**:
   - Ejecuta `giveawaytracker ingest --endpoint https://api.tusorteos.com/webhook --batch-size 500 --flush-interval 2000` para habilitar el modo receptor de webhooks.
   - Define las reglas de validación en `config/validation_rules.json` usando el esquema JSON `event_schema_v2`. Ejemplo: `{"max_entries_per_ip": 3, "window_seconds": 3600, "action_on_violation": "reject"}`.
   - Verifica la conectividad con la base de datos: `giveawaytracker health --check postgres --timeout 3000`. Debe responder con `{"status": "ok", "latency_ms": 12}`.

2. **Lanzar una campaña de prueba con datos sintéticos**:
   - Usa `giveawaytracker simulate --campaign-id test_001 --participants 10000 --prizes 50 --distribution uniform --rate 100` para generar tráfico simulado a **100 eventos/segundo**.
   - Monitorea la cola de procesamiento: `giveawaytracker stats --queue --format table`. Busca que `backlog` sea menor que `100` y `avg_processing_time` inferior a **30 ms**.
   - Ejecuta la auditoría de integridad: `giveawaytracker audit --campaign test_001 --verify-hashes`. El comando debe devolver `"integrity_check": "passed"` sin entradas en `mismatched_events`.

3. **Realizar una consulta analítica compleja**:
   - Conéctate al shell interactivo: `giveawaytracker query --campaign-id promo_q4_2024 --aggregate "COUNT(DISTINCT participant_id)" --group-by "region, prize_tier" --order-by "region ASC, prize_tier DESC"`.
   - El resultado se muestra en formato tabla con paginación automática (página de **100 filas**). Usa `--export-json` para obtener la salida en bruto.
   - Para análisis histórico, especifica `--range-start "2024-10-01T00:00:00Z" --range-end "2024-10-31T23:59:59Z"`. El motor usará **particionamiento por mes** en la tabla `events` para acelerar la consulta.

### Mejores Prácticas

- **Segmentación de campañas por prioridad**: Asigna un `priority_level` (1-5) a cada campaña en `campaigns.metadata`. El scheduler del sistema procesa eventos con prioridad **1** (alta) en un hilo dedicado con **CPU pinning**, garantizando latencias menores a **10 ms** para sorteos críticos. Las campañas de baja prioridad (4-5) se procesan en lotes con **backpressure** controlada.

- **Rotación y archivado de datos históricos**: Configura `giveawaytracker archive --age-days 90 --destination s3://bucket-archivo/` para mover eventos antiguos a almacenamiento frío. Mantén en la base activa solo los últimos **90 días** para optimizar consultas. El archivado preserva los índices y el esquema, permitiendo consultas distribuidas mediante **Foreign Data Wrappers** de PostgreSQL sobre los archivos Parquet.
