# giveawaytracker - GIV_014

## Resumen

Sistema de rastreo y gestión de sorteos en tiempo real, diseñado para plataformas de alto tráfico. Integra detección automática de patrones de fraude, análisis de engagement y exportación de métricas auditables.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es una plataforma de ingeniería inversa de eventos promocionales que captura, normaliza y correlaciona datos de sorteos desde múltiples fuentes (Twitter, Discord, Telegram). Implementa un pipeline de procesamiento asíncrono con latencias inferiores a 200ms por evento, capaz de manejar 10.000+ concurrentes sin degradación.

### Características Principales

- **Detección de Fraude Probabilístico**: Motor basado en `scikit-learn` que analiza más de 30 features por usuario (edad de cuenta, ratio de participación, patrones de IP, huella digital del navegador). Genera un score de riesgo en tiempo real con precisión del 94.7% en pruebas A/B con 500.000 muestras. Parámetros configurables: `FRAUD_THRESHOLD` (float, default 0.85), `ANOMALY_WINDOW` (int, segundos, default 3600).

- **Sincronización Multiplataforma con Colas de Mensajes**: Utiliza Apache Kafka como backbone para ingerir eventos desde APIs REST, WebSockets y webhooks. Cada sorteo se descompone en micro-lotes de 100 registros con confirmación de entrega (acks=all) y replicación en 3 nodos. Métrica clave: throughput sostenido de 2.500 eventos/segundo con una latencia P99 de 45ms.

- **Panel de Analytics con PostgreSQL y Grafana**: Almacena metadatos de sorteos en tablas particionadas por fecha (`giveaway_events_2025_02`). Consultas optimizadas con índices compuestos en `(platform, event_id, created_at)`. Dashboard exportable como JSON con 12 widgets predefinidos: tasa de conversión, distribución geográfica (GeoIP), heatmap de actividad horaria. Tiempo de carga del panel: <800ms para conjuntos de 1M de registros.

### Cómo Usar

1. **Configuración del Pipeline de Ingesta**  
   Despliega el agente `giveawaytracker-agent` con variables de entorno: `KAFKA_BROKERS=broker1:9092,broker2:9092`, `REDIS_URL=redis://cache:6379/0`. Ejecuta `docker-compose up -d` para iniciar los microservicios de captura (WebSocket listener en puerto 8080, REST API en 8081). Verifica la conexión con `curl -X POST http://localhost:8081/health` que debe retornar `{"status":"ok","uptime":12345}`.

3. **Exportación de Datos Auditables**  
   Utiliza la API REST `GET /api/v1/export?format=csv&rule_id=<UUID>&start=2025-01-01&end=2025-02-01`. El endpoint devuelve un archivo CSV con columnas: `user_id, platform, entry_time, fraud_score, ip_hash`. Para exportaciones mayores a 100.000 registros, se genera un enlace de descarga asíncrono (S3 presigned URL válida por 15 minutos). Monitoriza el progreso con `GET /api/v1/export/status?job_id=<UUID>`.

### Mejores Prácticas

- **Segmentación por Zona Horaria**: Programa la ejecución de reglas de captura en horarios de alta actividad (por ejemplo, 10:00-14:00 UTC para LATAM). Usa el parámetro `ACTIVE_HOURS` en el archivo de reglas para evitar saturar el pipeline con datos de baja calidad durante la madrugada. Esto reduce el costo de procesamiento en un 30%.

- **Rotación de Claves de API**: Implementa un ciclo de rotación de tokens cada 72 horas para las conexiones con plataformas externas (Twitter API v2, Discord Bot Token). Almacena las claves en HashiCorp Vault y renueva automáticamente mediante un cron job que ejecuta `giveawaytracker-cli rotate-keys --vault-path secret/giveaway`. Esto previene bloqueos por rate limiting.
