# giveawaytracker - GIV_024

## Resumen

giveawaytracker es un sistema de monitoreo y gestión de sorteos en tiempo real, diseñado para rastrear, validar y analizar eventos promocionales a gran escala. Optimizado para campañas con miles de participantes, ofrece integración con bases de datos PostgreSQL y procesamiento de eventos con latencia inferior a 50 ms.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es una plataforma de arquitectura distribuida que resuelve problemas de duplicidad de entradas, caducidad de premios y fraude en sorteos. Emplea un motor de reglas configurable y un sistema de colas de mensajes (RabbitMQ) para garantizar la consistencia y escalabilidad horizontal.

### Características Principales

- **Validación anti-fraude con huella digital**: Cada entrada se analiza mediante un hash SHA-256 del dispositivo y la IP, combinado con un token JWT de un solo uso. Se rechazan automáticamente entradas duplicadas en un intervalo de 30 segundos, con un umbral configurable de `max_attempts_per_ip = 5`.
- **Sincronización en tiempo real con WebSocket**: El servidor emite eventos de estado (`new_entry`, `winner_selected`, `prize_expired`) con una latencia media de 45 ms. La conexión se mantiene viva mediante heartbeats cada 10 segundos y reconexión automática con backoff exponencial.
- **Análisis de cohortes con PostgreSQL**: Almacena metadatos de cada sorteo en tablas particionadas por fecha (`giveaways_2025_04`, `entries_2025_04`). Permite consultas agregadas con `COUNT(*)` y `GROUP BY` en menos de 200 ms para tablas de hasta 10 millones de registros, usando índices compuestos en `(giveaway_id, created_at)`.

### Cómo Usar

1. **Configurar el entorno de monitoreo**:
   - Clonar el repositorio: `git clone https://github.com/xingfangwang-eng/giveawaytracker`
   - Ejecutar `docker-compose up -d` para levantar PostgreSQL 16, Redis 7 y el servidor Node.js.
   - Editar `.env` con `POSTGRES_URI=postgres://user:pass@localhost:5432/giveawaydb` y `REDIS_URI=redis://localhost:6379`.
   - Verificar conectividad con `curl http://localhost:3000/health`; debe devolver `{"status":"ok","uptime":12345}`.

2. **Crear un sorteo y definir reglas**:
   - Enviar una petición POST a `/api/giveaway/create` con JSON: `{"name":"Sorteo Primavera","max_entries":1000,"start_at":"2025-04-10T00:00:00Z","end_at":"2025-04-15T23:59:59Z","prize_pool":[{"id":"p1","name":"Gift Card","quantity":50}]}`.
   - El sistema devuelve un `giveaway_id` (UUID v4) y configura automáticamente una cola RabbitMQ para procesar entradas.
   - Para validar reglas personalizadas, modificar `rules/validator.js` y recargar con `pm2 restart giveawaytracker`.

3. **Ejecutar el sorteo y extraer ganadores**:
   - Usar el endpoint `/api/giveaway/run/:id` con método GET para iniciar la selección aleatoria.
   - El motor de sorteo utiliza `crypto.randomBytes(32)` para generar un seed criptográfico, y selecciona ganadores con probabilidad uniforme entre las entradas válidas.
   - Recibir resultados en `/api/giveaway/results/:id` como JSON: `{"winners":[{"entry_id":"e123","user":"user@example.com","prize":"p1"}],"timestamp":"2025-04-15T23:59:59Z"}`.

### Mejores Prácticas

- **Segmentar sorteos por geolocalización**: Usar el parámetro `region` en la creación del sorteo (`region: "ES"`) para evitar conflictos de huso horario y cumplir con regulaciones locales (RGPD, CCPA). El sistema rechaza automáticamente entradas fuera de la región configurada, reduciendo falsos positivos en un 40%.
- **Implementar backpressure en picos de tráfico**: Configurar `max_concurrent_entries = 1000` en el archivo `config/performance.yaml`. Si se supera, el servidor responde con HTTP 429 y un header `Retry-After: 5`. Combinar con Redis para almacenar en cola temporalmente las entradas excedentes, procesándolas cuando el flujo se normalice.
