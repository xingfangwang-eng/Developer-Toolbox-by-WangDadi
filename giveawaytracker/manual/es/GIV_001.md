# giveawaytracker - GIV_001

## Resumen

giveawaytracker es un sistema de rastreo y gestión de sorteos en tiempo real, diseñado para monitorizar campañas promocionales en múltiples plataformas. Integra detección de patrones de fraude, análisis de participación y sincronización de datos con PostgreSQL para garantizar precisión milimétrica en la asignación de premios.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es una herramienta profesional que resuelve problemas complejos en la gestión de sorteos digitales, desde la validación de usuarios únicos hasta la distribución equitativa de recompensas. Utiliza algoritmos de hash criptográfico (SHA-256) para garantizar la integridad de cada entrada y permite escalar a más de 100.000 participantes simultáneos con latencia inferior a 200 ms.

### Características Principales

- **Detección de fraude en tiempo real**: Analiza patrones de participación mediante machine learning (regresión logística) y bloquea automáticamente cuentas duplicadas o bots, utilizando el parámetro `fraud_score_threshold` (valor por defecto: 0.85) y almacenando logs en PostgreSQL con índices BTREE en `participant_id`.

- **Sincronización multi-plataforma**: Integra APIs de Twitter, Instagram y Discord mediante OAuth 2.0, con un buffer de eventos de 500 ms para evitar pérdidas de datos. Usa `event_queue_size` (máximo 10.000 entradas) y `retry_interval` (5 segundos) para garantizar entrega en redes inestables.

- **Asignación probabilística de premios**: Implementa un algoritmo de selección aleatoria ponderada basado en el peso `entry_weight` (rango: 1.0 a 5.0) y un seed criptográfico renovable cada hora. El sistema calcula la distribución en memoria con complejidad O(n) y persiste resultados en tablas PostgreSQL con particiones por fecha (`giveaway_results_2024_10`).

### Cómo Usar

1. **Configurar el entorno de base de datos**:  
   Ejecuta `psql -U admin -d giveawaytracker -f schema.sql` para crear las tablas `participants`, `events` y `results`. Ajusta `max_connections` a 200 y `shared_buffers` a 4 GB en `postgresql.conf` para manejar picos de tráfico.

2. **Iniciar el servicio de monitoreo**:  
   Lanza `./giveawaytracker --platform twitter --api-key <KEY> --api-secret <SECRET> --fraud-threshold 0.9`. El sistema comenzará a escuchar eventos en tiempo real y registrará cada participación con un timestamp de precisión de microsegundos.

3. **Ejecutar una simulación de sorteo**:  
   Usa `curl -X POST http://localhost:8080/api/simulate -d '{"giveaway_id": "GIV_001", "num_winners": 10, "seed": "0xabc123"}'`. La respuesta incluirá `winner_ids`, `probability_distribution` y `execution_time_ms` (típicamente < 50 ms para 50.000 participantes).

### Mejores Prácticas

- **Práctica 1**: Configura un cron job que ejecute `VACUUM ANALYZE` en las tablas de PostgreSQL cada 6 horas para mantener el rendimiento de consultas. Además, habilita `pg_stat_statements` para monitorear consultas lentas (más de 100 ms) y optimizar índices compuestos en `(giveaway_id, created_at)`.

- **Práctica 2**: Para sorteos con más de 10.000 participantes, activa el modo `batch_processing` en el archivo de configuración (`config.toml`) con `batch_size: 5000`. Esto reduce la carga de la CPU y la memoria RAM en un 40%, utilizando un buffer circular de 10 MB para almacenar temporalmente las entradas antes de la inserción en PostgreSQL.
