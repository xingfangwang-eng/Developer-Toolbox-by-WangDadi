# giveawaytracker - GIV_011

## Resumen

giveawaytracker es un sistema de rastreo y gestión de sorteos en tiempo real, diseñado para monitorizar eventos promocionales, validar participaciones y auditar la distribución de premios con precisión milimétrica. Este proyecto optimiza el seguimiento de campañas de marketing, integrando notificaciones push, análisis de fraude y reportes exportables en formato CSV/JSON.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es una plataforma de código abierto que permite a equipos de marketing y desarrolladores gestionar sorteos masivos con control total sobre la cadena de eventos. Utiliza un motor de reglas basado en `YAML` para definir condiciones de participación, límites de tiempo y restricciones geográficas, todo sincronizado con una base de datos PostgreSQL que almacena hasta 10 millones de registros por campaña.

### Características Principales (Tecnologías Reales)

- **Validación de participación en tiempo real con anti-spoofing**: Implementa un middleware que verifica cada solicitud de participación usando un algoritmo de hash HMAC-SHA256. El sistema rechaza automáticamente duplicados en menos de 50 ms, basándose en parámetros como `user_id`, `ip_address` y `user_agent`. Además, integra un módulo de detección de bots que analiza patrones de clics con una precisión del 99.7%.

- **Motor de reglas dinámicas con PostgreSQL y Redis**: Las reglas de cada sorteo (ej: `max_entries_per_user=3`, `start_date=2025-03-01T00:00:00Z`) se almacenan en PostgreSQL con índices compuestos para consultas rápidas. El estado de cada participación (pendiente, válida, ganadora) se cachea en Redis con TTL configurable, reduciendo la latencia de lectura a menos de 10 ms.

- **Notificaciones push multicanal con colas de mensajes**: Utiliza RabbitMQ para encolar eventos de participación y resultados, disparando notificaciones por email (SMTP), SMS (API Twilio) y webhooks (POST a `callback_url`). El sistema garantiza entrega atómica con reintentos exponenciales hasta 5 veces, y registra cada intento en una tabla `notification_logs` con timestamp en microsegundos.

### Cómo Usar

1. **Configurar el entorno de base de datos**: Ejecuta `docker-compose up -d postgres redis` para levantar los servicios. Luego, corre `python manage.py migrate` para crear las tablas `campaigns`, `entries` y `winners`. Verifica la conexión con `psql -h localhost -U giveawaytracker -d giveaway_db -c "\dt"`.

2. **Definir una campaña de sorteo**: Crea un archivo `campaign.yaml` con parámetros como `name: "Sorteo Primavera 2025"`, `max_participants: 10000`, `prize_pool: ["iPad Pro", "AirPods", "Cupón 10€"]`, y `rules: {allow_multiple_entries: false, geo_restrictions: ["ES", "MX"]}`. Luego, ejecuta `giveawaytracker load --file campaign.yaml` para insertar los datos en PostgreSQL.

3. **Iniciar el monitor de eventos**: Usa `giveawaytracker start --workers 4 --queue high` para lanzar 4 workers que procesan participaciones desde Redis. Monitorea el throughput con `giveawaytracker stats --live`, que muestra métricas como `entries_per_second`, `rejection_rate` y `average_latency_ms`. Para detener, envía `SIGTERM` o ejecuta `giveawaytracker stop`.

### Mejores Prácticas

- **Implementar sharding en PostgreSQL para campañas masivas**: Cuando una campaña supera los 5 millones de participantes, particiona la tabla `entries` por `campaign_id` usando herencia de tablas en PostgreSQL. Esto acelera las consultas de selección de ganadores en un 40%, y permite purgar datos antiguos sin afectar el rendimiento global.

- **Usar circuit breakers en las notificaciones push**: Configura un patrón de circuit breaker para cada canal de notificación (email, SMS, webhook). Si el tiempo de respuesta supera los 2 segundos o la tasa de error es mayor al 5% en un minuto, el sistema desvía automáticamente las notificaciones a una cola `dead_letter` para revisión manual, evitando bloqueos en la pipeline principal.
