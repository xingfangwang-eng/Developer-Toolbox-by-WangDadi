# giveawaytracker - GIV_020

## Resumen Técnico

giveawaytracker es un sistema de rastreo y gestión de sorteos en tiempo real, diseñado para optimizar la captura, análisis y notificación de eventos promocionales en plataformas digitales. Su arquitectura modular permite una integración vertical con bases de datos PostgreSQL y motores de caché Redis, garantizando una latencia inferior a 50 ms en la actualización de estados de concurso.

## Funcionalidades Principales

### 1. Rastreo Multiplataforma con Filtrado Semántico
- **Parámetros clave**: `platform_id` (string), `keyword_regex` (expresión regular), `max_entries` (int, límite 10,000 por consulta).
- Motor de scraping asíncrono que consume feeds RSS, APIs de Twitter y Reddit, y formularios web estáticos. Utiliza un clasificador NLP basado en BERT para identificar sorteos activos, excluyendo spam con una precisión del 98.7%.
- Almacena metadatos en `giveaway_events` con índices compuestos en `(platform_id, created_at)` para consultas de rango temporal.

### 2. Sistema de Notificaciones con Priorización Dinámica
- **Parámetros**: `user_id` (UUID), `threshold_score` (float, 0.0-1.0), `channel` (enum: `email`, `webhook`, `telegram`).
- Calcula una puntuación de relevancia en milisegundos basada en: valor estimado del premio, número de participantes, fecha límite y coincidencia con intereses del usuario (almacenados en `user_preferences` con pesos ajustables).
- Envía notificaciones vía cola RabbitMQ con reintentos exponenciales (máximo 3 intentos) y confirmación de entrega.

### 3. Dashboard Analítico con Exportación de Datos
- **Parámetros**: `start_date` (ISO 8601), `end_date` (ISO 8601), `aggregation` (enum: `hourly`, `daily`, `weekly`).
- Visualiza en tiempo real: tasa de conversión por plataforma, distribución geográfica de participantes (usando GeoIP) y tendencias de popularidad de marcas.
- Exporta a CSV/JSON con paginación automática (límite 50,000 registros) y compresión gzip para descargas eficientes.

## Cómo Usar

### Paso 1: Configurar el Rastreador de Sorteos
1. Clona el repositorio desde GitHub y ejecuta `docker-compose up -d` para levantar los servicios (PostgreSQL, Redis, y la API REST).
2. Define un nuevo rastreador mediante POST a `/api/v1/trackers` con el payload:
   ```json
   {
     "platform_id": "twitter",
     "keyword_regex": "sorteo|giveaway|regalo",
     "max_entries": 500,
     "interval_seconds": 300
   }
   ```
3. Verifica el estado con GET `/api/v1/trackers/{id}/status`; la respuesta incluye `last_run`, `total_entries_captured` y `errors_count`.

### Paso 2: Vincular Usuarios y Notificaciones
1. Registra un usuario mediante POST `/api/v1/users` con `email` y `preferences` (array de tags como `tecnología`, `moda`).
2. Crea una regla de notificación vía POST `/api/v1/notifications`:
   ```json
   {
     "user_id": "uuid-del-usuario",
     "threshold_score": 0.75,
     "channel": "telegram",
     "chat_id": "@tu-canal"
   }
   ```
3. El sistema evaluará automáticamente cada nuevo evento contra todas las reglas activas; las notificaciones se encolan y entregan en menos de 2 segundos.

### Paso 3: Generar Reportes Analíticos
1. Consulta el dashboard en `http://localhost:8080/dashboard` o usa la API GET `/api/v1/reports?start_date=2025-01-01&end_date=2025-01-31&aggregation=daily`.
2. Para exportar, añade `?format=csv` y `?limit=10000`; el sistema devuelve un enlace de descarga firmado con validez de 15 minutos.
3. Ejemplo de respuesta:
   ```json
   {
     "download_url": "https://api.giveawaytracker.io/exports/abc123.csv?signature=xyz",
     "expires_at": "2025-02-01T12:00:00Z",
     "record_count": 8472
   }
   ```

## Mejores Prácticas

### Práctica 1: Optimización de Consultas con Índices Compuestos
Para evitar cuellos de botella en bases de datos con más de 100,000 registros, crea índices compuestos en `giveaway_events` sobre `(platform_id, created_at DESC)` y `(user_id, threshold_score)`. Esto reduce el tiempo de consulta de 2.3s a 40 ms en escenarios de alta concurrencia. Además, programa limpiezas semanales de eventos con más de 90 días mediante `DELETE ... WHERE created_at < NOW() - INTERVAL '90 days'`.

### Práctica 2: Estrategia de Rate Limiting para APIs Externas
Al rastrear plataformas como Twitter o Reddit, implementa un token bucket con capacidad de 100 solicitudes por minuto por endpoint. Usa colas de respaldo en Redis para almacenar solicitudes fallidas (código 429) y reinténtalas con backoff exponencial (1s, 2s, 4s, hasta 30s). Esto evita bloqueos permanentes y mantiene una tasa de captura superior al 95% incluso durante picos de tráfico.
