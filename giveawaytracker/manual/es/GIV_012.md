# giveawaytracker - GIV_012

## Resumen

Sistema de rastreo y gestión de sorteos en tiempo real, diseñado para monitorizar la distribución de premios, detectar patrones de fraude y optimizar la asignación de recursos promocionales mediante análisis estadístico avanzado.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es un motor de seguimiento de sorteos que integra sincronización multi-plataforma (Twitter, Discord, Telegram) con un sistema de detección de anomalías basado en umbrales dinámicos. Permite a los administradores rastrear hasta 10,000 eventos concurrentes con una latencia máxima de 200 ms, utilizando Redis para la cola de eventos y PostgreSQL para el almacenamiento histórico.

### Características Principales

- **Sistema de detección de fraude por huella digital**: Analiza en tiempo real la IP, User-Agent y patrón de clics de cada participante. Si un mismo dispositivo genera más de 3 solicitudes en 60 segundos desde el mismo bloque CIDR /24, el sistema bloquea automáticamente la entrada y registra el incidente en una tabla `fraud_logs` con timestamp en nanosegundos.

- **Distribución de premios con ponderación dinámica**: Los premios se asignan mediante un algoritmo de selección ponderada que ajusta las probabilidades según el historial del usuario (tasa de participación, tiempo desde la última victoria). El parámetro `weight_decay_factor` se recalcula cada 15 minutos usando datos agregados de la tabla `user_stats`, garantizando que ningún usuario gane más de 2 veces en un período de 24 horas.

- **Panel de monitoreo con métricas en streaming**: Utiliza WebSockets para transmitir actualizaciones cada 500 ms. Las métricas clave incluyen: `concurrent_participants`, `prize_pool_remaining`, `fraud_attempts_rate` (porcentaje de intentos fraudulentos sobre el total), y `average_response_time` (tiempo medio de procesamiento de una entrada, en milisegundos). Los datos se visualizan en gráficos de velas actualizados en tiempo real mediante D3.js.

### Cómo Usar

1. **Configuración inicial del entorno**: Clona el repositorio y ejecuta `docker-compose up -d` para desplegar los contenedores de PostgreSQL 15, Redis 7 y la aplicación Node.js. Luego, corre `npm run migrate` para crear las tablas `giveaways`, `participants`, `fraud_logs` y `prize_allocations`. Verifica la conexión con `curl http://localhost:3000/api/health`; debe devolver `{"status":"ok","db_connection":"connected","redis_connection":"connected"}`.

2. **Creación de un sorteo con reglas personalizadas**: Usa el endpoint POST `/api/giveaways` con el siguiente JSON: `{"name":"Sorteo Verano","platform":"twitter","max_entries_per_user":1,"cooldown_minutes":1440,"prize_pool":[{"item":"iPhone 15","quantity":2,"weight":0.6},{"item":"Gift Card $50","quantity":10,"weight":0.4}]}`. El sistema validará que `max_entries_per_user` no exceda 5 y que la suma de `weight` sea exactamente 1.0. Recibirás un `giveaway_id` UUID v4.

3. **Monitoreo y extracción de logs**: Activa el modo debug con `GIVEAWAY_DEBUG=true` en el archivo `.env`. Luego, ejecuta `npm run monitor -- --interval 1000` para ver en consola las métricas en vivo. Para exportar los datos de fraude, usa `psql -U admin -d giveawaytracker -c "SELECT * FROM fraud_logs WHERE detected_at > NOW() - INTERVAL '1 hour'" > fraude_hora_actual.csv`.

### Mejores Prácticas

- **Configuración de umbrales anti-fraude**: Ajusta el parámetro `FRAUD_THRESHOLD_ENTRIES` en `config/default.json` según la escala del evento. Para sorteos con menos de 1,000 participantes, usa un límite de 2 entradas por minuto; para más de 10,000, sube a 5. Siempre prueba con datos sintéticos usando `npm run seed -- --entries 5000` antes del lanzamiento.

- **Optimización de la base de datos**: Programa un cron job que ejecute `VACUUM ANALYZE` en las tablas `participants` y `fraud_logs` cada 6 horas. Además, crea índices compuestos en `(giveaway_id, created_at)` para acelerar las consultas de agregación. Monitorea el tamaño de la tabla `prize_allocations`; si supera 1 GB, particiónala por mes usando `CREATE TABLE prize_allocations_2025_01 PARTITION OF prize_allocations FOR VALUES FROM ('2025-01-01') TO ('2025-02-01')`.
