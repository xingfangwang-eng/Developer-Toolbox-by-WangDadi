# giveawaytracker - GIV_021

## Resumen

giveawaytracker es un sistema de rastreo y gestión de sorteos en tiempo real, diseñado para analizar la participación, detectar patrones de fraude y optimizar la distribución de premios. Integra un motor de eventos asíncrono con PostgreSQL y Redis para garantizar una latencia inferior a 50 ms por transacción.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es una plataforma modular que captura, procesa y visualiza eventos de sorteos masivos. Utiliza un pipeline de datos basado en Apache Kafka para ingerir hasta 10,000 solicitudes por segundo, con un sistema de reglas heurísticas que identifica automáticamente comportamientos anómalos (como múltiples entradas desde una misma IP o patrones temporales sospechosos). Además, ofrece un dashboard interactivo con métricas en tiempo real, exportables a CSV/JSON mediante una API RESTful con autenticación JWT.

### Características Principales

- **Motor de detección de fraude con heurística adaptativa**: Analiza hasta 50 parámetros por entrada (hash de dispositivo, User-Agent, geolocalización, tiempo entre clics) y aplica un modelo de puntuación de riesgo (0-100) en menos de 200 ms. Los umbrales se ajustan dinámicamente según el historial del sorteo, reduciendo falsos positivos en un 30% comparado con sistemas estáticos.

- **Orquestador de distribución de premios con colas prioritarias**: Utiliza Redis Streams para encolar las recompensas y un scheduler interno que respeta reglas de límite por usuario, región y tipo de premio. Soporta hasta 500 lotes por minuto con confirmación de entrega (ACK) y reintentos automáticos en caso de fallo de la API externa (timeout configurable: 3 segundos).

- **Panel de análisis temporal con granularidad de milisegundos**: Almacena cada evento de participación en una tabla particionada de PostgreSQL (por fecha y hora UTC), permitiendo consultas agregadas en ventanas de 1 minuto, 1 hora o 1 día. Incluye un endpoint `/analytics/heatmap` que devuelve datos JSON con coordenadas de actividad geográfica, procesados mediante la librería `maxmind-geolite2`.

### Cómo Usar

1. **Configuración inicial del entorno**: Clona el repositorio y ejecuta `docker-compose up -d` para levantar los servicios de PostgreSQL 15, Redis 7 y el worker de Kafka. Luego, edita el archivo `config.yaml` para definir las variables de entorno: `DB_HOST=localhost`, `KAFKA_BOOTSTRAP_SERVERS=127.0.0.1:9092`, y `REDIS_PASSWORD=your_secure_pass`. Verifica la conexión con `curl http://localhost:8080/health`; debe devolver `{"status":"ok","uptime":1234}`.

2. **Creación de un sorteo y registro de participantes**: Usa el comando CLI `giveawaytracker create --name "Sorteo Navidad" --max-entries 1000 --start 2025-01-01T00:00:00Z --end 2025-01-07T23:59:59Z`. Para registrar participantes, envía una petición POST a `/api/v1/entry` con el body JSON: `{"user_id":"u123","device_fingerprint":"abc123","ip":"192.168.1.1","timestamp":"2025-01-02T10:30:00.123Z"}`. El sistema devuelve un `entry_id` único y un campo `risk_score` calculado en tiempo real.

3. **Ejecución del sorteo y extracción de ganadores**: Una vez finalizado el plazo, ejecuta `giveawaytracker draw --id sorteo_navidad --method weighted` para que el motor seleccione ganadores usando un algoritmo de ponderación basado en el historial de participación y el riesgo. Los resultados se guardan en la tabla `winners` con los campos `user_id`, `prize_type`, `timestamp_drawn` y un hash de verificación SHA-256. Para exportar, usa `giveawaytracker export --id sorteo_navidad --format csv --output ./ganadores.csv`.

### Mejores Prácticas

- **Segmentación de carga con sharding geográfico**: Si manejas sorteos globales, configura particiones en PostgreSQL por región (ej. `entries_na`, `entries_eu`) y un balanceador de Redis por zona. Esto reduce la latencia de escritura en un 40% y evita cuellos de botella en picos de tráfico (recomendado para >50,000 entradas por hora).

- **Auditoría de reglas con versionado semántico**: Cada regla de detección de fraude debe tener un campo `version` (ej. `fraud_rule_v2.1`) y almacenarse en una tabla `rules_history`. Al actualizar, el sistema compara automáticamente los resultados contra la versión anterior y genera un reporte de regresión. Esto permite revertir cambios en menos de 5 minutos sin afectar sorteos en curso.
