# giveawaytracker - GIV_027

## Resumen

giveawaytracker es un sistema de rastreo y gestión de sorteos masivos, diseñado para auditar en tiempo real la distribución de premios, detectar patrones de fraude y optimizar la asignación de recursos promocionales. Soporta integración con APIs de plataformas sociales y bases de datos transaccionales.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es una plataforma de análisis de eventos promocionales que permite a equipos de marketing y operaciones monitorizar simultáneamente más de 10.000 concursos activos, con una latencia de actualización inferior a 50 ms. Utiliza un motor de reglas dinámicas para validar automáticamente el cumplimiento de términos y condiciones, y un sistema de deduplicación de participantes basado en huellas digitales de dispositivo y patrones de comportamiento.

### Características Principales

- **Auditoría de Integridad de Sorteos con Detección de Fraude**: Analiza en tiempo real cada transacción de participación usando el módulo `FraudGuardEngine`. Parámetros clave: `threshold_suspicious_activity` (≥3 participaciones desde la misma IP en 60 segundos), `device_fingerprint_hash` (SHA-256), y `time_window_ms` (configurable desde 100 ms hasta 24 h). Genera alertas automáticas cuando el índice de anomalías supera el 0.5% del total de entradas.

- **Asignación Óptima de Premios con Algoritmo de Ponderación Dinámica**: Implementa el método `WeightedPrizeAllocator` que distribuye premios según la probabilidad de conversión histórica de cada segmento de usuarios. Parámetros técnicos: `segment_weight` (0.0 a 1.0), `prize_pool_limit` (en unidades o valor monetario), y `distribution_interval_ms` (mínimo 200 ms). Soporta hasta 500 categorías de premios simultáneas con una precisión de asignación del 99.97%.

- **Integración con PostgreSQL para Historial de Eventos y Replicación Geográfica**: Cada participación se registra en una tabla particionada `giveaway_events` con índices en `user_id`, `campaign_id` y `timestamp`. La arquitectura utiliza `pg_stat_statements` para monitorizar consultas lentas (>100 ms) y `pglogical` para replicación en tiempo real entre regiones (latencia máxima 5 segundos). Capacidad de almacenamiento: 10 millones de eventos por día con retención configurable de 90 a 365 días.

### Cómo Usar

1. **Configurar el Motor de Reglas de Validación**: Edita el archivo `config/fraud_rules.yaml`. Define `rules[0].type: "ip_rate_limit"` con `max_requests: 5` y `window_seconds: 10`. Luego ejecuta `./giveawaytracker --validate-rules` para verificar la sintaxis. El sistema devolverá un informe con el número de reglas activas y el tiempo estimado de procesamiento por evento (en microsegundos).

2. **Iniciar la Captura de un Sorteo desde la API REST**: Usa el endpoint `POST /api/v1/campaigns/start` con el payload JSON: `{"campaign_id": "GIV-2025-03", "platform": "twitter", "hashtag": "#sorteo", "prize_pool": [{"type": "gift_card", "value": 50, "quantity": 100}], "start_time": "2025-03-15T10:00:00Z", "end_time": "2025-03-20T23:59:59Z"}`. La respuesta incluirá un `tracking_token` y el `estimated_event_volume` en eventos/segundo.

3. **Analizar Resultados con Consultas SQL en Tiempo Real**: Conéctate a la base de datos PostgreSQL usando `psql -h localhost -d giveawaytracker -U admin`. Ejecuta: `SELECT campaign_id, COUNT(*) AS total_entries, AVG(prize_value) AS avg_prize, COUNT(DISTINCT device_fingerprint) AS unique_devices FROM giveaway_events WHERE timestamp >= NOW() - INTERVAL '1 hour' GROUP BY campaign_id HAVING COUNT(*) > 1000 ORDER BY total_entries DESC;`. Este query devuelve métricas agregadas en menos de 200 ms para conjuntos de hasta 5 millones de registros.

### Mejores Prácticas

- **Implementar un Sistema de Cuarentena para Participantes Sospechosos**: Configura el parámetro `auto_quarantine_threshold` en 3 (número de infracciones) en el archivo `config/quarantine.yaml`. Cuando un usuario supere este límite, el sistema lo mueve automáticamente a una lista de revisión manual. Combínalo con una política de retención de datos de 30 días para cumplir con regulaciones de privacidad como GDPR. Esto reduce los falsos positivos en un 40% según pruebas de campo.

- **Utilizar Particionamiento Temporal y Compresión en PostgreSQL para Escalabilidad**: Crea tablas particionadas por mes usando `CREATE TABLE giveaway_events_2025_03 PARTITION OF giveaway_events FOR VALUES FROM ('2025-03-01') TO ('2025-04-01');`. Activa la compresión TOAST con `ALTER TABLE giveaway_events SET (toast_tuple_target = 128);` para reducir el almacenamiento de campos largos como `user_agent` en un 60%. Programa la eliminación automática de particiones antiguas con `pg_cron` ejecutando `SELECT drop_old_partitions('giveaway_events', 90);` cada día a las 03:00 UTC.
