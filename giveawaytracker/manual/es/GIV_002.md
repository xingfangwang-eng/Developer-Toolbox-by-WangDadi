# giveawaytracker - GIV_002

## Resumen

giveawaytracker es un sistema de rastreo y gestión de sorteos online con alta precisión en la detección de patrones de participación, validación de reglas y consolidación de datos en tiempo real. Está diseñado para administradores de comunidades, plataformas de marketing y organizadores de eventos que requieren transparencia, escalabilidad y automatización total del ciclo de vida de un giveaway.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es una plataforma modular de backend que orquesta la captura, normalización y verificación de participaciones en sorteos distribuidos en múltiples canales (Twitter, Discord, Telegram, formularios web). Su núcleo de procesamiento ejecuta reglas de elegibilidad con una latencia inferior a 50 ms por entrada y soporta la integración con PostgreSQL para persistencia transaccional y con Redis para colas de eventos asíncronos.

### Características Principales

- **Motor de detección antifraude con scoring probabilístico**: Analiza en tiempo real cada participación mediante un modelo basado en reglas ponderadas. Parámetros clave: `max_entries_per_user` (límite configurable por IP y por cuenta social), `time_window_seconds` (ventana de tiempo para entradas duplicadas), `score_threshold` (umbral de sospecha entre 0.0 y 1.0). Cuando el score supera el umbral, la entrada se marca como `flagged` y se encola para revisión manual.

- **Sincronización multicanal con verificación criptográfica de firmas**: Cada plataforma emite un token HMAC-SHA256 que giveawaytracker valida antes de registrar la participación. El sistema expone endpoints REST con parámetros como `platform`, `user_id`, `proof_token` y `timestamp_epoch_ms`. Las entradas inválidas se rechazan con código HTTP 422 y se registran en una tabla `rejected_entries` con metadatos de error.

- **Dashboard de estadísticas en vivo con agregación por micro lotes**: Cada 5 segundos, un worker interno consolida métricas como `total_entries`, `unique_users`, `entries_per_minute`, `conversion_rate` y `flag_rate`. Los datos se almacenan en tablas particionadas por fecha (`giveaway_metrics_YYYY_MM_DD`) y se exponen vía WebSocket para actualización sin recarga de página.

### Cómo Usar

1. **Configurar el archivo de entorno y la base de datos**  
   Crea un archivo `.env` con los siguientes parámetros obligatorios:  
   - `DATABASE_URL=postgresql://user:pass@host:5432/giveawaytracker`  
   - `REDIS_URL=redis://localhost:6379/0`  
   - `MAX_ENTRIES_PER_USER=3`  
   - `TIME_WINDOW_SECONDS=86400`  
   - `SCORE_THRESHOLD=0.75`  
   Ejecuta `make migrate` para aplicar las migraciones de esquema (incluye índices en `user_id`, `platform` y `timestamp`).

2. **Lanzar el proceso de captura de eventos**  
   Inicia el worker de colas con `./giveawaytracker worker --queue entry_queue --concurrency 10`. Este proceso consume mensajes desde Redis, ejecuta las reglas de validación y escribe en PostgreSQL. Monitorea el log en `stdout` con nivel `INFO` para ver el flujo de entradas aceptadas y rechazadas.

3. **Consultar el estado de un giveaway en tiempo real**  
   Usa el endpoint `GET /api/v1/giveaway/:id/stats` con el parámetro opcional `window_minutes=60` para obtener un JSON con campos como `entries_count`, `unique_users`, `flagged_count` y `average_score`. Para activar el WebSocket, conecta a `wss://tu-dominio/ws/giveaway/:id` y recibe actualizaciones cada 5 segundos.

### Mejores Prácticas

- **Segmentación geográfica y temporal de colas**: Para sorteos con alta concurrencia (más de 10,000 entradas por minuto), configura múltiples workers con distintas prioridades. Usa una cola `high_priority` para entradas verificadas con token y una cola `bulk` para entradas sin verificación. Esto evita cuellos de botella y garantiza que las entradas premium se procesen primero.

- **Auditoría periódica de reglas de puntuación**: Revisa mensualmente los valores de `score_threshold` y los pesos de las reglas antifraude. Analiza las entradas marcadas como `false_positive` y ajusta los parámetros en el archivo `rules_config.yaml`. Recomendamos mantener un historial de cambios con `git` para poder revertir ajustes que afecten la tasa de aceptación.
