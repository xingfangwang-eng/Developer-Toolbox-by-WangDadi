# giveawaytracker - GIV_005

## Resumen

Sistema de rastreo y gestión de sorteos en tiempo real, optimizado para campañas multicanal con detección de fraudes y análisis predictivo de engagement. Arquitectura basada en microservicios con PostgreSQL como capa persistente y WebSocket para latencias inferiores a 50 ms.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es una plataforma de orquestación de sorteos que unifica la captura, validación y distribución de premios en redes sociales, sitios web y aplicaciones móviles. Emplea un motor de reglas dinámicas (DRM) para adaptarse a términos y condiciones variables, y un sistema de colas asíncronas (RabbitMQ) para manejar picos de hasta 10,000 participantes concurrentes sin degradación del servicio.

### Características Principales

- **Detección de fraude multicapa con huella digital**: Analiza 27 parámetros por entrada (User-Agent, resolución de pantalla, latencia de red, patrón de clics, tiempo entre acciones) para calcular un `fraud_score` (0.0 - 1.0). Los participantes con `fraud_score > 0.85` se marcan automáticamente como sospechosos y se envían a revisión manual mediante webhook. La validación se ejecuta en menos de 200 ms por entrada gracias a un caché distribuido con Redis.

- **Sorteo ponderado por engagement histórico**: Asigna pesos dinámicos a cada participante basados en su `engagement_index`, calculado a partir de la frecuencia de interacciones previas, el valor de la comunidad (`community_value`) y la tasa de conversión de campañas anteriores. El algoritmo de selección (`weighted_random_selection`) utiliza la distribución de alias para garantizar que sorteos con más de 50,000 entradas se resuelvan en menos de 100 ms.

- **Panel de control en tiempo real con predicción de resultados**: Visualiza métricas como `entries_per_second`, `conversion_rate` (participantes que completan todos los pasos), `estimated_winner_probability` (porcentaje de probabilidad de ganar para cada participante basado en la tendencia actual) y `time_to_reach_goal` (tiempo estimado para alcanzar el número objetivo de entradas). Los datos se refrescan cada 500 ms mediante WebSocket, y las predicciones se generan con un modelo de regresión lineal simple entrenado sobre los últimos 15 minutos de actividad.

### Cómo Usar

1. **Configuración del motor de reglas para una campaña**: Crea un archivo `rules.yaml` en la raíz del proyecto. Define los criterios de elegibilidad, por ejemplo: `min_account_age: 30` (días), `required_interactions: 3` (compartir, like, comentar), `excluded_countries: ["CU", "IR", "KP"]`. Luego, ejecuta `giveawaytracker validate-rules --file rules.yaml` para verificar la sintaxis y simular la aplicación de las reglas sobre un conjunto de 100 participantes de prueba.

2. **Lanzamiento de un sorteo con integración de Twitter**: Utiliza el comando `giveawaytracker launch --platform twitter --campaign-id "summer2024" --max-entries 5000 --duration 48h`. El sistema se conecta a la API de Twitter v2, filtra tweets con la palabra clave configurada y almacena cada entrada en PostgreSQL. Durante la ejecución, puedes monitorear el progreso con `giveawaytracker status --campaign-id "summer2024" --format json`, que devuelve el número actual de entradas, la tasa de crecimiento y el tiempo restante.

3. **Selección de ganadores y distribución de premios**: Una vez finalizado el sorteo, ejecuta `giveawaytracker draw --campaign-id "summer2024" --winners 10 --method weighted --seed 42`. El sistema aplica el algoritmo de selección ponderada, genera un hash de cada ganador (`winner_hash`) para verificación pública, y envía notificaciones por correo electrónico y DM de Twitter. Para auditar, usa `giveawaytracker audit --campaign-id "summer2024" --winner-id 3` para obtener el `fraud_score`, el `engagement_index` y el timestamp exacto de selección (precisión de milisegundos).

### Mejores Prácticas

- **Segmentación de colas para alta concurrencia**: Configura colas separadas en RabbitMQ por tipo de acción (validación de entrada, cálculo de fraude, selección de ganador). Asigna prioridades: `high` para la validación (debe completarse en < 50 ms), `medium` para el cálculo de fraude y `low` para la notificación. Esto evita que un pico de entradas bloquee la selección final del ganador.

- **Estrategia de caché para reglas dinámicas**: Almacena las reglas de cada campaña en Redis con una TTL de 5 minutos. Cuando un participante envía una entrada, el sistema primero consulta el caché; si no existe, carga las reglas desde PostgreSQL y las cachea. Esto reduce la latencia de validación en un 40% y disminuye la carga en la base de datos durante campañas con más de 100,000 entradas.
