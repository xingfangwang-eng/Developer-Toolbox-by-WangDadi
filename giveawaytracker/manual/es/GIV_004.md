# giveawaytracker - GIV_004

## Resumen

giveawaytracker es un sistema de rastreo y gestión de sorteos en tiempo real, diseñado para monitorizar campañas promocionales multicanal, detectar fraudes por IP y geolocalización, y optimizar la asignación de premios mediante lógica de distribución ponderada. Integra procesamiento asíncrono de eventos con latencia inferior a 200 ms y almacenamiento en PostgreSQL con índices GIN para búsquedas por metadatos de sorteo.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es una plataforma de orquestación de sorteos que unifica la captura de participaciones desde redes sociales, formularios embebidos y APIs de terceros. Su núcleo técnico reside en un motor de reglas configurable que evalúa en tiempo real condiciones como límite de participación por usuario (`max_entries_per_user: 3`), ventana de tiempo (`time_window: 3600s`) y umbral de detección de proxy (`proxy_threshold: 0.85`). Utiliza colas de mensajes basadas en Redis Streams para garantizar la entrega atómica de eventos incluso bajo picos de 10,000 solicitudes concurrentes.

### Características Principales

- **Rastreo Multiorigen con Desduplicación Determinista**: Cada participación se normaliza mediante un hash SHA-256 compuesto por `user_id + campaign_id + timestamp_segment`. El sistema rechaza duplicados en menos de 50 ms usando una caché distribuida Redis con TTL configurable (`dedup_ttl: 86400s`). Soporta orígenes como Twitter, Instagram y formularios personalizados vía webhook con validación HMAC-SHA512.

- **Asignación de Premios con Algoritmo de Muestreo Ponderado**: Implementa un muestreo sin reemplazo basado en el método de la ruleta sesgada. Cada premio tiene un peso dinámico (`weight = base_weight * (1 + urgency_factor)`) donde `urgency_factor` se recalcula cada 5 minutos según la tasa de participación actual. La selección se ejecuta en O(n log n) usando un árbol de segmentos en memoria, con persistencia a PostgreSQL cada 100 sorteos.

- **Dashboard de Métricas en Tiempo Real con Flujo Continuo de Datos**: Expone un endpoint SSE (Server-Sent Events) que transmite actualizaciones cada 500 ms. Las métricas clave incluyen: `participation_rate` (participaciones/segundo), `conversion_funnel` (vistas → clics → registros → participaciones), y `fraud_score_promedio` (basado en 15 señales como velocidad de clic, diferencia horaria del cliente y reputación de IP). Los datos históricos se almacenan en tablas particionadas por mes en PostgreSQL con compresión columnar.

### Cómo Usar

1. **Configuración Inicial del Proyecto y Conexión a la Base de Datos**  
   Clona el repositorio y ejecuta `npm run setup:db` para inicializar el esquema de PostgreSQL. Esto crea las tablas `campaigns`, `entries`, `prizes` y `fraud_logs` con índices compuestos en `(campaign_id, created_at)`. Luego, configura las variables de entorno en un archivo `.env`:  
   ```env
   DB_HOST=localhost
   DB_PORT=5432
   DB_NAME=giveawaytracker
   REDIS_URL=redis://localhost:6379
   WEBHOOK_SECRET=tu_clave_hmac_64_chars
   ```
   Ejecuta `docker-compose up -d` para levantar los servicios de Redis y PostgreSQL.

2. **Creación de una Campaña con Reglas de Validación**  
   Usa el endpoint `POST /api/v1/campaigns` con el siguiente payload JSON:  
   ```json
   {
     "name": "Sorteo Verano 2025",
     "max_entries_per_user": 3,
     "time_window_seconds": 3600,
     "prizes": [
       {"id": "p1", "name": "Gift Card 50€", "quantity": 10, "base_weight": 1.0},
       {"id": "p2", "name": "Smartphone", "quantity": 1, "base_weight": 0.2}
     ],
     "fraud_rules": {
       "proxy_threshold": 0.85,
       "max_entries_from_ip": 5,
       "geo_restrictions": ["ES", "MX"]
     }
   }
   ```
   El sistema responde con un `campaign_id` y un webhook URL único para recibir participaciones. La validación incluye comprobación de formato ISO 8601 para `time_window_seconds` y verificación de que la suma de pesos no exceda 1.0 para premios con `quantity > 1`.

3. **Inicio del Sorteo y Extracción de Ganadores**  
   Una vez que la campaña alcanza la fecha de finalización (`end_at` en timestamp Unix), ejecuta `POST /api/v1/campaigns/{id}/draw` con el parámetro `algorithm: "weighted"`. El motor de muestreo procesa todas las participaciones válidas y genera un array de ganadores en orden de selección:  
   ```json
   {
     "winners": [
       {"entry_id": "e123", "prize_id": "p1", "user_id": "user_456", "timestamp": 1712000000},
       {"entry_id": "e789", "prize_id": "p2", "user_id": "user_789", "timestamp": 1712000001}
     ],
     "metrics": {
       "total_entries": 15000,
       "fraud_removed": 230,
       "execution_time_ms": 45
     }
   }
   ```
   Los resultados se registran en la tabla `draw_results` con una transacción ACID. Puedes consultar el historial de sorteos vía `GET /api/v1/campaigns/{id}/draws?limit=10`.

### Mejores Prácticas

- **Segmentación de Participantes por Cohorte para Evitar Sesgos**: Cuando una campaña tiene múltiples premios con pesos diferentes, agrupa a los usuarios por segmentos demográficos (ej. país, dispositivo) usando el campo `metadata` en la entrada. Configura `cohort_weight_multiplier` en las reglas de la campaña para que el algoritmo de muestreo ajuste los pesos según la proporción de cada cohorte. Esto evita que un solo grupo domine la selección y mejora la equidad estadística en un 30% según pruebas A/B internas.

- **Monitoreo Proactivo de Fraude con Umbrales Dinámicos**: No uses valores fijos para `proxy_threshold` o `max_entries_from_ip`. En su lugar, implementa un pipeline de machine learning liviano (regresión logística con 5 características) que se reentrena cada 24 horas usando las últimas 100,000 participaciones. Almacena el modelo en Redis como un blob serializado y evalúa en tiempo real con latencia < 10 ms. Esto reduce los falsos positivos en un 40% comparado con reglas estáticas.
