# giveawaytracker - GIV_013

## Resumen

Sistema de rastreo y gestión de sorteos en tiempo real con análisis de fraude, integración multi-plataforma y generación automática de informes. Optimizado para campañas de alto volumen (hasta 10,000 eventos/segundo) con latencia de consulta <50ms.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es un motor de tracking descentralizado que unifica la captura, validación y visualización de participaciones en sorteos desde redes sociales, formularios web y APIs externas. Utiliza un pipeline de procesamiento basado en Apache Kafka y almacenamiento en PostgreSQL con particionamiento por hash de usuario, garantizando consistencia eventual y tolerancia a fallos en clusters de 3+ nodos.

### Características Principales

- **Detección de fraude en tiempo real**: Algoritmo de scoring heurístico que analiza patrones de participación (IP, User-Agent, geolocalización, tiempo entre clics) y bloquea automáticamente cuentas con probabilidad de fraude >85%. Parámetros configurables: `fraud_threshold` (0-100), `ip_blacklist_expiry` (segundos), `max_entries_per_ip` (entero).

- **Sincronización multi-plataforma con cola de eventos**: Integración nativa con Twitter, Instagram, Telegram y formularios embebidos mediante webhooks. Cada evento se encola en un topic de Kafka con partición por `platform_id` y `campaign_id`. Métrica de rendimiento: throughput sostenido de 5,000 eventos/segundo por partición, con reintentos automáticos (hasta 3, backoff exponencial).

- **Generación de informes dinámicos con exportación a PDF/CSV**: Dashboard interactivo con filtros por fecha, plataforma, estado de participación (válida, duplicada, fraudulenta). Exportación a PDF con plantillas personalizables (cabecera, logo, colores) y a CSV con encoding UTF-8 BOM para compatibilidad con Excel. Tiempo de generación: <2 segundos para 100,000 registros.

### Cómo Usar

1. **Configurar el pipeline de ingesta de eventos**  
   Editar el archivo `config/kafka_connector.yaml` y definir los parámetros:  
   ```yaml
   bootstrap_servers: "localhost:9092,node2:9092,node3:9092"
   topic: "giveaway_events"
   partition_count: 6
   replication_factor: 3
   consumer_group: "giveaway_tracker_group"
   ```  
   Luego ejecutar `docker-compose up -d zookeeper kafka` para iniciar el clúster. Verificar conectividad con `kafka-topics --describe --topic giveaway_events --bootstrap-server localhost:9092`.

2. **Integrar webhooks de plataformas externas**  
   En el panel de administración, ir a `Settings > Webhooks` y añadir las URLs de callback:  
   - Twitter: `https://tu-dominio.com/webhook/twitter?token=<API_KEY>`  
   - Instagram: `https://tu-dominio.com/webhook/instagram?token=<API_KEY>`  
   - Telegram: `https://tu-dominio.com/webhook/telegram?chat_id=<CHAT_ID>`  
   Cada webhook debe responder con un código 200 en menos de 500ms; de lo contrario, el sistema reintenta hasta 3 veces con un intervalo de 1 segundo.

3. **Ejecutar el análisis de fraude y generar informe**  
   Llamar al endpoint REST:  
   ```bash
   curl -X POST "https://api.giveawaytracker.com/v1/analyze" \
        -H "Authorization: Bearer <TOKEN>" \
        -d '{"campaign_id": "CAM-2025-03", "fraud_threshold": 80, "export_format": "pdf"}'
   ```  
   El sistema devolverá un JSON con el resumen de participaciones (totales, válidas, fraudulentas) y un enlace de descarga del informe PDF. El archivo se almacena en `./exports/reports/` con nombre `report_CAM-2025-03_YYYYMMDD_HHMMSS.pdf`.

### Mejores Prácticas

- **Segmentar campañas por geolocalización**: Configurar filtros de IP por país usando la base de datos GeoLite2 (actualizada semanalmente). Esto reduce falsos positivos de fraude en un 30% y mejora la precisión del scoring heurístico. Implementar un cache Redis con TTL de 24 horas para las consultas de geolocalización.

- **Monitorear el throughput de Kafka con Prometheus**: Exportar métricas como `kafka_consumer_lag`, `kafka_topic_partition_count` y `kafka_request_time_ms` a un dashboard de Grafana. Configurar alertas cuando el lag supere 10,000 mensajes o cuando el tiempo de procesamiento exceda 200ms. Esto permite escalar particiones horizontalmente antes de que el sistema se degrade.
