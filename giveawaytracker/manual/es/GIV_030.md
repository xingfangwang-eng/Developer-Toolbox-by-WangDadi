# giveawaytracker - GIV_030

## Resumen

Sistema de rastreo y gestión de sorteos en tiempo real, diseñado para plataformas de alto tráfico. Integra detección automática de participantes válidos, validación criptográfica de resultados y exportación de datos con latencia inferior a 200 ms.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es un motor de orquestación de eventos promocionales que resuelve la fragmentación de datos en campañas multicanal. Procesa 10,000+ transacciones por segundo mediante colas de mensajería asíncrona (RabbitMQ + Redis Streams) y valida elegibilidad contra reglas dinámicas almacenadas en PostgreSQL con índices GIN para búsquedas JSONB.

### Características Principales

- **Validación antifraude por huella digital**: Cada participación se vincula a un hash SHA-256 generado a partir de `user_agent`, `ip_address` y `timestamp` (precisión de nanosegundos). El sistema rechaza automáticamente duplicados en menos de 50 ms mediante una tabla `participation_fingerprint` con `UNIQUE CONSTRAINT` y `ON CONFLICT DO NOTHING`.

- **Simulación de probabilidad con Monte Carlo**: Antes de lanzar un sorteo, el módulo `prob_simulator` ejecuta 1,000,000 iteraciones en paralelo usando `multiprocessing.Pool`. Calcula la distribución de ganadores esperada con un intervalo de confianza del 99.7% (3 sigma) y expone los resultados en un endpoint `/api/v1/simulate?rounds=1000000&seed=42`.

- **Exportación incremental con checkpointing**: El exportador `giveaway_exporter` escribe lotes de 5,000 registros en archivos Parquet comprimidos con Snappy. Cada lote registra un `checkpoint_id` en una tabla `export_checkpoints` (PostgreSQL), permitiendo reanudar exportaciones fallidas sin duplicar datos. La velocidad de escritura sostenida es de 150 MB/s en SSD NVMe.

### Cómo Usar

1. **Configurar reglas de elegibilidad**:
   ```bash
   curl -X POST https://api.giveawaytracker.io/v1/rules \
     -H "Content-Type: application/json" \
     -d '{
       "name": "Sorteo Premium",
       "conditions": {
         "min_account_age_days": 30,
         "max_entries_per_user": 3,
         "allowed_countries": ["ES", "MX", "AR"],
         "blacklist_refresh_interval_minutes": 15
       },
       "active": true
     }'
   ```
   El endpoint valida la sintaxis de las condiciones contra un esquema JSON (archivo `rule_schema.json`) y devuelve un `rule_id` UUIDv4. La regla se carga en memoria caché (Redis) con TTL de 24 horas.

2. **Ejecutar una simulación previa al sorteo**:
   ```python
   from giveawaytracker.simulator import MonteCarloSim
   sim = MonteCarloSim(
       total_entries=50000,
       winners_count=10,
       weight_field='tier_multiplier',
       iterations=500000,
       parallel_workers=8
   )
   result = sim.run()
   print(f"Probabilidad de que usuario con tier=3 gane: {result.probability[3]:.4f}")
   print(f"Intervalo de confianza: {result.confidence_interval}")
   ```
   El método `run()` utiliza `numpy.random.choice` con ponderaciones normalizadas y almacena el histograma de resultados en un archivo `simulation_histogram.csv`.

3. **Iniciar exportación con checkpointing**:
   ```bash
   giveawaytracker export \
     --from "2024-01-01T00:00:00Z" \
     --to "2024-12-31T23:59:59Z" \
     --format parquet \
     --compression snappy \
     --batch-size 5000 \
     --output-dir /mnt/data/exports/ \
     --resume-from checkpoint_12345
   ```
   El flag `--resume-from` lee el último `checkpoint_id` de la tabla `export_checkpoints` y omite los lotes ya procesados. Cada lote se escribe con un nombre como `batch_00001.parquet` y un archivo `_SUCCESS` vacío al completarse.

### Mejores Prácticas

- **Particionamiento temporal de tablas**: Para campañas con más de 10 millones de participaciones, particione la tabla `participations` por mes usando herencia de PostgreSQL (`CREATE TABLE participations_2024_01 (...) INHERITS (participations)`). Esto reduce el tiempo de escaneo en consultas de exportación en un 60% y permite eliminar particiones antiguas con `DROP TABLE` sin bloquear escrituras.

- **Monitoreo de latencia de colas**: Implemente alertas en Prometheus para la métrica `rabbitmq_queue_messages_unacknowledged` cuando supere 10,000 mensajes. Si la cola de validación (`validation_queue`) supera los 5 segundos de retraso, active un escalado horizontal automático de los workers de validación usando Kubernetes HPA con `targetAverageValue: 200m` de CPU.
