# giveawaytracker - GIV_009

## Resumen

Sistema de rastreo y análisis en tiempo real para campañas de sorteos y giveaways, diseñado para detectar patrones de fraude, optimizar la distribución de premios y generar informes de rendimiento con precisión de milisegundos.

## Contenido Detallado

### ¿Qué es giveawaytracker?

giveawaytracker es una plataforma de ingeniería inversa de eventos promocionales que captura, normaliza y correlaciona datos de participación en sorteos de múltiples fuentes (Twitter, Discord, Telegram, formularios web). Utiliza un motor de reglas dinámicas basado en PostgreSQL con extensiones de ventanas temporales (window functions) para calcular la elegibilidad de usuarios en tiempo real, detectar cuentas duplicadas por fingerprinting de navegador y hash de IP + User-Agent, y asignar pesos de probabilidad según la actividad histórica del participante.

### Características Principales

- **Motor de detección de fraude por análisis de grafos**: Implementa un algoritmo de detección de comunidades (Louvain modularity) sobre la red de referidos y direcciones IP. Cada nodo (usuario) se evalúa con un score de riesgo (`risk_score`) calculado a partir de la densidad de conexiones, la antigüedad de la cuenta y la tasa de participación en sorteos previos. El sistema bloquea automáticamente a usuarios con `risk_score > 0.85` y genera alertas en `alert_log` con timestamps en microsegundos.

- **Sistema de asignación ponderada de premios con colas de prioridad**: Cada giveaway tiene un `prize_pool` con `n` slots. El backend asigna probabilidades dinámicas usando una función de distribución exponencial inversa basada en el `engagement_score` del usuario (calculado a partir de `tweets_retweeted`, `discord_messages`, `time_spent_on_site`). Los premios se distribuyen mediante un scheduler asíncrono que respeta cuotas por región (`region_quota`) y franjas horarias (`time_window_start`, `time_window_end`), con un límite de 10.000 asignaciones por minuto.

- **Pipeline de normalización de datos con validación por checksum**: Cada evento de participación se ingiere a través de un endpoint REST que exige un payload con `user_id`, `campaign_id`, `timestamp` y un `hmac_signature` generado con SHA-256. El sistema valúa la integridad del dato contra una tabla `event_checksums` en PostgreSQL, y si el checksum no coincide, el evento se rechaza y se registra en `rejected_events` con código de error `ERR_CHECKSUM_MISMATCH`. La latencia de procesamiento es inferior a 50 ms por evento.

### Cómo Usar

1. **Configurar el pipeline de ingesta de datos**: Despliega el endpoint `POST /api/v1/events` con autenticación mediante API key. Define en el archivo `config.toml` los parámetros `max_concurrent_connections = 500`, `batch_size = 1000`, `flush_interval_ms = 200`. Usa el comando `giveawaytracker ingest --source twitter --campaign-id CAM-2024-09 --output-dir /data/raw` para iniciar la captura en vivo. Verifica la integridad de los datos con `giveawaytracker validate --checksum-db postgresql://user:pass@host:5432/giveawaytracker`.

2. **Ejecutar el análisis de detección de fraude**: Lanza el módulo de análisis con `giveawaytracker detect --risk-threshold 0.85 --window-hours 72 --output-format json`. El sistema generará un archivo `fraud_report_<campaign_id>.json` que contiene una lista de usuarios sospechosos con sus `risk_score`, `community_id` y `evidence` (ej: "IP 192.168.1.1 compartida con 12 cuentas"). Para visualizar el grafo, usa `giveawaytracker graph --campaign-id CAM-2024-09 --output graphviz`.

3. **Configurar la asignación de premios con colas de prioridad**: Define las reglas en `prize_rules.yaml`:
   ```yaml
   prize_pool:
     - name: "Gold Pass"
       quantity: 50
       region_quota: {"US": 20, "EU": 20, "APAC": 10}
       time_window: {"start": "2024-09-01T00:00:00Z", "end": "2024-09-30T23:59:59Z"}
       distribution: "exponential_decay"
       decay_factor: 0.7
   ```
   Ejecuta `giveawaytracker assign --rules-file prize_rules.yaml --dry-run` para simular la asignación. Para ejecutar en producción, omite `--dry-run`. El sistema generará un log en `assignments.log` con cada asignación, incluyendo `user_id`, `prize_id`, `timestamp` y `probability_assigned`.

### Mejores Prácticas

- **Segmentación de campañas por ventanas de tiempo y geografía**: Para evitar sesgos en la distribución de premios, define `time_window` y `region_quota` en cada campaña. Usa `giveawaytracker stats --campaign-id CAM-2024-09 --group-by region` para monitorizar la equidad en tiempo real. Si detectas que una región supera el 120% de su cuota, ajusta el `decay_factor` en `prize_rules.yaml` para reducir la probabilidad de asignación en esa región.

- **Auditoría continua con checksums y alertas**: Configura un cron job que ejecute `giveawaytracker validate --checksum-db postgresql://... --compare-with /data/raw/events.json` cada 5 minutos. Si el número de eventos rechazados supera el 1% del total, envía una alerta a Slack o PagerDuty. Además, almacena los `hmac_signature` en una tabla separada (`event_signatures`) para auditoría forense posterior. Esto garantiza que ningún evento sea modificado después de la ingesta.
