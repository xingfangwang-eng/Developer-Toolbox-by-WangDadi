# giveawaytracker - GIV_005

## Overview

giveawaytracker is a high-performance giveaway event monitoring and analytics engine designed for real-time sweepstakes, raffle, and contest tracking. It ingests multi-source event streams, deduplicates entries with sub‑millisecond latency, and provides actionable fraud detection heuristics. The system is built to handle 10k+ concurrent events per second while maintaining a 99.9% uptime SLA.

## Detailed Content

### What is giveawaytracker?

giveawaytracker is a distributed, event‑driven platform that captures, normalizes, and analyzes giveaway participation data across social media, email campaigns, and embedded widgets. It leverages a time‑series database (PostgreSQL with TimescaleDB extension) for historical trend analysis and a Redis‑backed rate‑limiter to enforce per‑user entry caps. The tool exposes a RESTful API and a WebSocket stream for live dashboard updates, all secured via OAuth 2.0 bearer tokens.

### Core Features

- **Real‑Time Entry Deduplication & Fraud Scoring**  
  Uses a Bloom filter (with configurable false‑positive rate, default 0.01) to detect duplicate entries within 50‑millisecond windows. Each entry is assigned a fraud score (0‑100) based on IP reputation, device fingerprint, and temporal pattern analysis. The scoring model is trained on historical giveaway data and updated weekly via an automated pipeline.

- **Multi‑Channel Event Aggregation with Custom Filters**  
  Accepts JSON payloads from Twitter API v2, Instagram Graph API, and custom webhooks. Supports parameterized filters such as `source="twitter"`, `hashtag="#sweepstakes"`, and `entry_time BETWEEN NOW() - INTERVAL '7 days' AND NOW()`. Aggregated events are stored in partitioned PostgreSQL tables for efficient querying across millions of rows.

- **Automated Winner Selection & Audit Trail**  
  Implements a cryptographically seeded random selection algorithm (using `pgcrypto`’s `gen_random_uuid()` as the seed) to ensure verifiable fairness. Each selection round logs the seed, timestamp, participant IDs, and hash of the final winner list to an append‑only table. The audit trail can be exported as a signed JSON file for third‑party verification.

### How to Use

1. **Deploy the ingestion endpoint**  
   Send a POST request to `/api/v1/events` with the following JSON schema:  
   ```json
   {
     "source": "twitter",
     "campaign_id": "camp_001",
     "participant": {
       "user_id": "12345",
       "handle": "@user",
       "entry_time": "2025-03-21T14:30:00Z"
     },
     "metadata": {
       "ip": "192.168.1.1",
       "user_agent": "Mozilla/5.0"
     }
   }
   ```  
   The endpoint returns a `202 Accepted` with a `dedup_hash` header. Use the dedup hash to query the status via `GET /api/v1/events/{dedup_hash}`.

2. **Configure fraud detection rules**  
   Edit the `fraud_rules.json` file in the `config/` directory. Example rule:  
   ```json
   {
     "rule_id": "ip_rate_limit",
     "type": "rate_limit",
     "params": {
       "window_seconds": 60,
       "max_entries_per_ip": 5
     },
     "action": "flag"
   }
   ```  
   Reload the rules without restarting the service by sending `SIGHUP` to the process or calling `POST /api/v1/admin/reload-rules`.

3. **Run a winner selection**  
   Use the CLI tool:  
   ```bash
   giveawaytracker draw --campaign-id camp_001 --winners 3 --seed $(uuidgen)
   ```  
   The output prints the winner IDs and the audit hash. Verify the draw integrity by running:  
   ```bash
   giveawaytracker verify --audit-hash <hash> --campaign-id camp_001
   ```

### Best Practices

- **Batch event ingestion during high‑traffic periods**  
  Instead of sending one event per HTTP request, batch up to 1,000 events in a single POST to `/api/v1/events/batch`. This reduces network overhead and improves throughput by up to 40%. Ensure each event in the batch has a unique `client_event_id` to enable idempotent processing.

- **Set up a monitoring dashboard with real‑time alerts**  
  Use the WebSocket endpoint `wss://your-instance:8080/ws` to stream key metrics: event ingestion rate, fraud flag count, and duplicate hit ratio. Configure alerts (e.g., via PagerDuty webhook) when the fraud flag rate exceeds 5% of total events within a 5‑minute sliding window, indicating a potential bot attack.
