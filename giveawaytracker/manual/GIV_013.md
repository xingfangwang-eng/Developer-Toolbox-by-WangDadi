# giveawaytracker - GIV_013

## Overview

This is the technical documentation for **giveawaytracker**, a high-performance giveaway event monitoring and analytics engine. Designed for real-time scraping, deduplication, and cross-platform aggregation, it provides deterministic tracking of promotional campaigns across social media, e-commerce, and community platforms.

## Detailed Content

### What is giveawaytracker?

giveawaytracker is a distributed event tracking system that ingests, normalizes, and indexes giveaway entries from multiple sources. It leverages a custom rule engine for dynamic field extraction and a Redis-backed deduplication layer to ensure each participant is counted exactly once per campaign, regardless of platform.

### Core Features

- **Real-Time Cross-Platform Ingestion with Deduplication**  
  Uses a multi-threaded HTTP scraper (configurable via `MAX_WORKERS` and `RATE_LIMIT_MS`) to poll 10+ platforms simultaneously. Each entry is hashed via `SHA256(platform + user_id + campaign_id)` and checked against a Redis set with a TTL of 3600 seconds. Duplicates are logged to `duplicates_{campaign_id}.log` with millisecond timestamps.

- **Deterministic Winner Selection with Verifiable Randomness**  
  Implements a Fisher-Yates shuffle on the deduplicated entry list, seeded by a SHA-256 hash of the campaign's `start_timestamp` and a server-provided `nonce`. The selection process outputs a `winner_proof.json` containing the seed, shuffled indices, and final winner IDs. This allows any third party to independently verify the outcome.

- **PostgreSQL-Backed Aggregation & Historical Analytics**  
  All normalized entries are stored in a PostgreSQL database with a schema optimized for time-series queries. The `entries` table is indexed on `(campaign_id, platform, created_at)`. A nightly cron job (`pg_cron`) executes a materialized view refresh that computes per-campaign metrics (e.g., unique entrants, entry velocity per hour, top referral domains) and stores them in `campaign_stats`.

### How to Use

1. **Initialize the Scraper Configuration**  
   Create a `config.json` file in the project root with the following structure:
   ```json
   {
     "targets": [
       {"platform": "twitter", "query": "giveaway OR contest", "interval_ms": 30000},
       {"platform": "reddit", "subreddit": "giveaways", "interval_ms": 60000}
     ],
     "redis": {"host": "127.0.0.1", "port": 6379, "db": 0},
     "postgresql": {"dsn": "postgresql://user:pass@localhost:5432/giveawaytracker"}
   }
   ```
   Run `python scraper.py --config config.json` to start the ingestion pipeline. The scraper will log each entry to `stdout` in JSON format with a `received_at` timestamp in ISO 8601.

2. **Trigger a Winner Selection**  
   After the campaign ends, execute:  
   `python select_winner.py --campaign_id GIV_013 --nonce $(openssl rand -hex 16)`  
   The script will output the winner ID and generate `winner_proof.json` in the `./proofs/` directory. Validate the proof by running:  
   `python verify_winner.py --proof_path ./proofs/winner_proof.json`

3. **Query Historical Analytics**  
   Connect to the PostgreSQL database and run the following to get hourly entry velocity:  
   ```sql
   SELECT date_trunc('hour', created_at) AS hour, COUNT(*) AS entries
   FROM entries
   WHERE campaign_id = 'GIV_013'
   GROUP BY hour
   ORDER BY hour;
   ```
   For a pre-aggregated view, query the `campaign_stats` materialized view:  
   `SELECT * FROM campaign_stats WHERE campaign_id = 'GIV_013';`

### Best Practices

- **Rate-Limit Your Scrapers to Avoid IP Bans**  
  Set `RATE_LIMIT_MS` to at least 1000 for high-traffic platforms like Twitter and Reddit. Use rotating proxies (defined in `proxies.txt`) and randomize user-agent strings per request. Monitor the `scraper_errors.log` file for HTTP 429 responses and adjust intervals dynamically.

- **Always Verify Winner Selection with a Third Party**  
  Before announcing a winner, share the `winner_proof.json` file with an independent auditor or post it publicly. The verification script `verify_winner.py` can be run by anyone with the proof file, ensuring trustless transparency. Never rely on a single source of randomness—always include the server nonce.
