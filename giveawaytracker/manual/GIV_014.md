# giveawaytracker - GIV_014

## Overview

`giveawaytracker` is a high-performance, event-driven giveaway monitoring and analytics engine designed for real-time sweepstakes, raffle, and promotional event aggregation. It provides deterministic tracking of giveaway lifecycles, fraud detection via entropy scoring, and automated compliance logging across multiple jurisdictions.

## Detailed Content

### What is giveawaytracker?

giveawaytracker is a distributed system that ingests giveaway metadata from 50+ platforms (Twitter, Gleam, Rafflecopter, KingSumo) via webhook and REST API, deduplicates entries using a Bloom filter with a 0.01% false-positive rate, and computes per-user win probability using a weighted Monte Carlo simulation. It supports sub-100ms event ingestion and PostgreSQL-backed state persistence with automatic sharding by giveaway ID.

### Core Features

- **Multi-Platform Webhook Aggregator with Entropy-Based Fraud Detection**  
  Ingests giveaway entries via configurable webhook endpoints (OAuth2, HMAC-SHA256 signed) and runs a real-time entropy check using a Markov chain model on entry timestamps and user-agent fingerprints. Suspicious entries (e.g., burst patterns >3σ from mean) are flagged and quarantined to a separate `fraud_hold` table in PostgreSQL, with automatic re-evaluation every 60 seconds.

- **Deterministic Giveaway Lifecycle Engine with Temporal Consistency**  
  Manages giveaway state transitions (Scheduled → Open → Drawing → Closed → Archived) using a finite state machine (FSM) backed by Redis streams. Each transition is logged with a monotonic clock timestamp (nanosecond precision) to guarantee causal ordering. Supports timezone-aware scheduling via IANA TZ database integration, with automatic DST correction.

- **Weighted Winner Selection with Verifiable Randomness**  
  Uses a Fisher-Yates shuffle seeded by a SHA-256 hash of the giveaway ID + block timestamp from a public blockchain oracle (Ethereum Ropsten testnet). Winners are selected according to user-defined weights (e.g., referral multipliers, loyalty tiers). The selection process is fully deterministic and reproducible: given the same seed and participant list, the output is identical across any compliant implementation.

### How to Use

1. **Deploy the Ingestion Pipeline**  
   Configure your webhook endpoint in `config/webhooks.yaml`:  
   ```yaml
   webhooks:
     - platform: twitter
       url: "https://api.giveawaytracker.io/v1/ingest/twitter"
       auth:
         type: hmac
         secret: "${TWITTER_HMAC_SECRET}"
       rate_limit: 1000 req/min
   ```  
   Run `./giveawaytracker ingest --config config/webhooks.yaml` to start the listener. The system will automatically create partitioned tables in PostgreSQL: `giveaways_{YYYY_MM}`, `entries_{YYYY_MM}`.

2. **Run a Deterministic Winner Selection**  
   After the giveaway closes, execute:  
   ```bash
   ./giveawaytracker draw --giveaway-id GIV-2025-014 \
     --seed-source ethereum \
     --rpc-endpoint "https://ropsten.infura.io/v3/YOUR_KEY" \
     --weight-multiplier 2.5 \
     --output winners_014.json
   ```  
   This fetches the latest block hash from Ethereum Ropsten, seeds the RNG, performs a weighted Fisher-Yates shuffle on all entries from `entries_2025_01` where `giveaway_id = 'GIV-2025-014'` and `status = 'verified'`, and writes the top 10 winners to a JSON file with a Merkle proof for auditability.

3. **Query Historical Giveaway Performance**  
   Use the built-in analytics CLI to generate a time-series report:  
   ```bash
   ./giveawaytracker analytics --from 2025-01-01 --to 2025-01-31 \
     --metric entries_per_minute \
     --group-by platform \
     --format csv | psql -h localhost -d giveawaytracker -c "\copy analytics_csv FROM STDIN CSV HEADER"
   ```  
   The output is a CSV with columns: `timestamp`, `platform`, `entries_per_minute`. This can be directly imported into PostgreSQL or fed into Grafana for real-time dashboards.

### Best Practices

- **Always Use a Blockchain Oracle for Seed Generation**  
  Avoid using `Math.random()` or system clock seeds for winner selection. Instead, integrate with a public blockchain (Ethereum, Solana) to obtain a verifiable, immutable seed. This ensures the draw is auditable and legally defensible in regulated markets (e.g., EU GDPR, California CCPA). Configure `seed_source: ethereum` in your giveaway config and set a minimum block confirmation of 12 to prevent reorg attacks.

- **Implement Graceful Degradation for Webhook Failures**  
  Your webhook ingestion pipeline must handle platform outages. Use a dead-letter queue (DLQ) backed by Redis with a TTL of 7 days. If a webhook returns 5xx, retry with exponential backoff (base: 1s, max: 300s, jitter: 0.1). After 5 failures, write the payload to `webhook_dlq` and alert via Prometheus + PagerDuty. This prevents data loss during Twitter API rate limits or Gleam server hiccups.
