# giveawaytracker - GIV_002

## Overview

giveawaytracker is a high‑performance, event‑driven giveaway monitoring and analytics engine designed for real‑time tracking, automated compliance validation, and cross‑platform aggregation. It ingests giveaway events from multiple sources, deduplicates entries at sub‑millisecond latency, and exposes a RESTful API for downstream reporting and automation.

## Detailed Content

### What is giveawaytracker?

giveawaytracker is a distributed giveaway management system that solves three core problems: **entry fraud detection**, **multi‑platform event synchronization**, and **real‑time leaderboard computation**. It uses a hybrid storage layer (PostgreSQL for persistent state + Redis Streams for ephemeral event queues) to guarantee exactly‑once processing even under 10,000 concurrent entries per second.

### Core Features

- **Adaptive Entry Deduplication Engine**  
  Combines IP fingerprinting, device fingerprint (`navigator.webdriver` checks, canvas fingerprint hash), and time‑windowed bloom filters (Redis Bloom, `BF.RESERVE` with `error_rate=0.001`) to reject duplicate entries within 50ms. Supports configurable sliding windows (e.g., 1 entry per IP per 24h).

- **Multi‑Platform Event Aggregator**  
  Unified webhook receiver that normalizes payloads from Twitter/X, Discord, Telegram, and custom REST endpoints. Each platform adapter validates signatures (HMAC‑SHA256 for webhooks, OAuth 2.0 Bearer tokens for API calls) and transforms fields into a canonical schema (`platform`, `user_id`, `entry_timestamp`, `referrer`). All events are persisted in a partitioned PostgreSQL table (`giveaway_events` with daily range partitioning).

- **Real‑Time Winner Selection with Weighted Randomization**  
  Uses a cryptographically secure PRNG (`crypto.randomBytes` in Node.js, or `secrets.randbelow` in Python) to select winners from a materialized view of eligible entries. Supports custom weight functions (e.g., “followers > 1000 → 2x weight”) computed via a PostgreSQL `RANK()` window function. Results are written to a `winners` table with a unique constraint on `(giveaway_id, winner_rank)` to prevent double‑awarding.

### How to Use

1. **Deploy the Webhook Receiver & Configure Platform Adapters**  
   - Clone the repository and run `docker-compose up -d` to spin up PostgreSQL, Redis, and the API server.  
   - Edit `config/platforms.yaml`: for each platform (e.g., Twitter), set `webhook_secret`, `api_key`, and `entry_filter` (e.g., `must_contain: “#giveaway”`).  
   - Verify connectivity: `curl -X POST http://localhost:8080/webhook/twitter -H "X-Signature: <hmac>" -d '{"tweet_id":"123","user_id":"abc"}'` should return `{"status":"accepted","event_id":"evt_...}`.

2. **Define Giveaway Rules via the Admin API**  
   - POST to `/api/giveaways` with a JSON payload:  
     ```json
     {
       "name": "Summer Launch Giveaway",
       "starts_at": "2025-06-01T00:00:00Z",
       "ends_at": "2025-06-07T23:59:59Z",
       "max_entries_per_user": 1,
       "weight_function": "CASE WHEN followers >= 1000 THEN 2 ELSE 1 END",
       "winner_count": 5
     }
     ```  
   - The system will create a partitioned table `giveaway_events_<id>` and start accepting entries.

3. **Trigger Winner Selection & Export Results**  
   - After the giveaway ends, call `POST /api/giveaways/<id>/draw` with optional `seed` parameter for deterministic replay.  
   - Retrieve winner list: `GET /api/giveaways/<id>/winners` returns a CSV‑ready JSON array:  
     ```json
     [
       {"rank":1,"user_id":"abc","platform":"twitter","entry_id":"evt_001"},
       {"rank":2,"user_id":"def","platform":"discord","entry_id":"evt_002"}
     ]
     ```  
   - Export to Google Sheets or Slack via the built‑in webhook notifier (configure `notifiers.slack.webhook_url` in `config.yaml`).

### Best Practices

- **Always enable deduplication with a sliding window of at least 24 hours**  
  Without it, a single user can flood entries via multiple accounts or proxies. Use the `dedup.window_seconds` parameter (default: 86400) and combine IP + device fingerprint for robust detection. Test with `BF.EXISTS` in Redis to verify false‑positive rate stays below 0.1%.

- **Partition PostgreSQL tables by giveaway start date**  
  When running hundreds of concurrent giveaways, a monolithic `giveaway_events` table will degrade query performance. Use daily range partitioning (`PARTITION BY RANGE (entry_timestamp)`) and attach a new partition for each giveaway. This allows `VACUUM` and `ANALYZE` to run per‑partition, keeping `SELECT` for winner selection under 200ms even with 1M+ entries.
