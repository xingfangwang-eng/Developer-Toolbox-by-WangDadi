# giveawaytracker - GIV_022

## Overview

giveawaytracker is a high-performance, event-driven giveaway monitoring and management engine designed for real-time sweepstakes aggregation, cross-platform prize validation, and automated compliance auditing. It ingests giveaway feeds from 40+ sources (Twitter, Gleam, Rafflecopter, Instagram, Discord) and normalizes them into a unified schema with sub-200ms latency.

## Detailed Content

### What is giveawaytracker?

giveawaytracker is a distributed tracking system that solves giveaway fraud detection, entry deduplication, and deadline synchronization across time zones. It uses an in-memory time-series database (Redis Streams + PostgreSQL) to track 500,000+ concurrent giveaways with millisecond-accurate expiration timestamps. The system supports multi-tenant analytics, WebSocket push for live winner announcements, and automated CAPTCHA resolver integration for entry validation.

### Core Features

- **Multi-Platform Entry Deduplication Engine**  
  Uses a combination of SHA-256 hashing on user-agent + IP + social profile ID to detect duplicate entries across platforms. Stores entry fingerprints in a Redis Bloom filter with 99.99% accuracy and a false-positive rate below 0.001%. Emits alerts via webhook when a single user enters >3 identical giveaways within a 60-second window.

- **Expiration-Aware Deadline Scheduler**  
  Implements a priority queue (based on min-heap) with PostgreSQL `TIMESTAMPTZ` columns for giveaway end times. The scheduler triggers `NOTIFY/LISTEN` events 5 minutes before expiration, and auto-archives giveaways with a 30-second grace period. Supports timezone offset normalization via IANA timezone database (v2024a), ensuring no missed deadlines across DST transitions.

- **Fraud Detection via Behavioral Scoring**  
  Assigns a risk score (0–100) per entry based on 12 features: entry velocity (entries/minute), account age (days), referral chain depth, and CAPTCHA solve time. Uses a pre-trained XGBoost model (ONNX runtime) that runs inference in <15ms per entry. Entries with score >85 are flagged, quarantined, and logged to a separate `fraud_audit` table with full JSONB context.

### How to Use

1. **Initialize the Tracker with Custom Rules**  
   Deploy via Docker Compose with `docker-compose up -d`. Configure `config.yaml` with your API keys (Twitter Bearer Token, Gleam Webhook Secret, Discord Bot Token). Set `dedup_window: 60` (seconds) and `fraud_threshold: 85`. Run `giveawaytracker init --db-url postgres://user:pass@host:5432/giveaways` to create the schema and apply migrations. Verify with `giveawaytracker status`—expected output: `Tracker ID: giv-022 | Sources: 4 | Queued: 0`.

2. **Ingest a Live Giveaway Feed**  
   Use the CLI to add a source: `giveawaytracker source add --type twitter --query "giveaway RT to win" --interval 30`. This creates a background worker that polls Twitter API v2 every 30 seconds. Each tweet is parsed via regex for giveaway keywords, then inserted into the `giveaways` table with a unique `gid` (UUID v7) and `expires_at` (ISO 8601). Monitor ingestion via `giveawaytracker logs --tail 20`—you should see `[INFO] Ingested tweet 1234567890 | Score: 72`.

3. **Trigger a Winner Selection with Audit Trail**  
   Run `giveawaytracker draw --gid GIV-2024-03-15-001 --method random --seed $(date +%s)`. The system fetches all validated entries, shuffles using Fisher-Yates with the seed, and selects the winner. Output includes: `Winner: @user123 | Entry ID: e-98765 | Timestamp: 2024-03-15T14:30:00Z`. The draw is logged to `winner_log` with a JSONB column containing full entry metadata. To export audit: `giveawaytracker audit --gid GIV-2024-03-15-001 --format json > audit_001.json`.

### Best Practices

- **Always Set a Minimum Entry Interval**  
  Configure `min_entry_interval: 10` (seconds) in `config.yaml` to prevent bot-driven rapid entries. This reduces false positives in the fraud model by 40% and cuts database write load by 25%. For high-traffic giveaways (e.g., 10k+ entries), enable batch writes with `batch_size: 100` and flush every 2 seconds.

- **Use Webhook-Based Monitoring Over Polling**  
  For real-time updates (e.g., winner announcements, fraud alerts), prefer webhooks over HTTP polling. Set up a receiver endpoint at `POST /webhook/giveaway` that accepts JSON payloads with schema `{ "type": "winner_selected"|"fraud_detected", "gid": "string", "data": {} }`. This reduces latency from 2–5 seconds (polling) to <50ms and lowers API costs by 90%.
