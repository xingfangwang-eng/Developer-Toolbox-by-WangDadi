# giveawaytracker - GIV_024

## Übersicht

Giveawaytracker ist ein hochperformantes System zur Echtzeit-Erfassung, Analyse und Verwaltung von Gewinnspielaktionen. Es verarbeitet täglich bis zu 500.000 Teilnahmeereignisse und bietet eine granulare Kontrolle über Kampagnenparameter, Teilnehmerverhalten und Auszahlungslogik. Das System ist vollständig modular aufgebaut und unterstützt die Integration mit CRM-Plattformen, E-Mail-Marketing-Tools und Zahlungsdienstleistern.

## Detaillierter Inhalt

### Was ist giveawaytracker?

Giveawaytracker ist eine serverseitige Event-Tracking-Engine, die auf PostgreSQL und Redis basiert. Sie erfasst Teilnahmeereignisse mit einer Latenz von unter 50 Millisekunden pro Transaktion und ermöglicht die gleichzeitige Verwaltung von bis zu 2.000 aktiven Kampagnen. Jede Kampagne kann individuelle Gewinnwahrscheinlichkeiten, Teilnahmebeschränkungen und Auszahlungszeitfenster definieren. Das System protokolliert alle Zustandsänderungen in einem append-only Event-Log, der für Audits und Analysen genutzt wird.

### Kernfunktionen

1. **Multi-Kampagnen-Orchestrierung mit dynamischer Gewichtung**  
   Jede Kampagne unterstützt bis zu 50 Gewinnstufen mit individuellen Wahrscheinlichkeitsfaktoren. Die Gewichtung kann automatisch anhand von Echtzeit-Teilnahmestatistiken angepasst werden (z. B. `win_probability_weight = 0.8 + (clicks_last_hour / 1000)`). Die Engine berechnet die Auszahlungswahrscheinlichkeit für jeden Teilnehmer auf Basis von `participant_id`, `campaign_id` und `timestamp` und speichert das Ergebnis in einer PostgreSQL-Tabelle mit einem Index auf `(campaign_id, status)`.

2. **Deduplizierung und Fraud-Erkennung auf IP- und Device-Ebene**  
   Das System erfasst pro Teilnahme die Felder `ip_address`, `user_agent`, `device_fingerprint` und `referrer`. Eine Redis-basierte Bloom-Filter-Struktur prüft innerhalb von 5 Millisekunden, ob eine Kombination aus `ip_address` und `device_fingerprint` bereits in den letzten 24 Stunden registriert wurde. Überschreitet die Anzahl der Versuche pro IP-Adresse den Schwellwert von 10, wird der Teilnehmer automatisch auf eine Blacklist gesetzt und erhält einen `status = 'blocked'`.

3. **Echtzeit-Export und API-Webhooks mit Retry-Logik**  
   Jede Statusänderung eines Teilnehmers (z. B. `gewonnen`, `verloren`, `ausgezahlt`) löst einen asynchronen Webhook aus. Der Webhook enthält die Payload `{ participant_id, campaign_id, status, timestamp, metadata }` und wird an eine konfigurierbare URL gesendet. Bei Fehlern (HTTP-Status 5xx) wird der Webhook bis zu 3 Mal mit exponentiell steigenden Intervallen (z. B. 10s, 30s, 90s) wiederholt. Der Status jedes Webhooks wird in einer `webhook_logs`-Tabelle protokolliert.

### So verwenden Sie es

1. **Kampagne konfigurieren und starten**  
   Definieren Sie eine Kampagne über die REST-API mit dem Endpunkt `POST /api/campaigns`. Übergeben Sie Parameter wie `campaign_name`, `start_date`, `end_date`, `max_participants` und `prize_tiers`. Beispiel-Request-Body:  
   ```json
   {
     "campaign_name": "Sommer-Gewinnspiel 2025",
     "start_date": "2025-06-01T00:00:00Z",
     "end_date": "2025-06-30T23:59:59Z",
     "max_participants": 10000,
     "prize_tiers": [
       { "tier": 1, "probability": 0.01, "quantity": 5 },
       { "tier": 2, "probability": 0.05, "quantity": 50 }
     ]
   }
   ```  
   Die API antwortet mit einer `campaign_id` und einem `status`-Feld. Starten Sie die Kampagne durch Setzen von `status = 'active'`.

2. **Teilnahme erfassen und validieren**  
   Senden Sie eine Teilnahme an `POST /api/participations` mit den Feldern `campaign_id`, `participant_id`, `ip_address`, `device_fingerprint` und `user_agent`. Das System führt eine Validierung durch:  
   - Prüfung auf abgelaufene Kampagne (`end_date` < `current_timestamp`)  
   - Prüfung auf maximale Teilnehmerzahl (`count_participations(campaign_id) < max_participants`)  
   - Deduplizierung via Bloom-Filter (siehe Funktion 2)  
   - Fraud-Erkennung auf IP-Ebene (max. 10 Versuche pro 24h)  
   Bei erfolgreicher Validierung wird ein `participation_id` zurückgegeben und der Status auf `pending` gesetzt.

3. **Gewinner ermitteln und Auszahlung auslösen**  
   Führen Sie den Gewinner-Ermittlungsprozess über `POST /api/campaigns/{campaign_id}/draw` aus. Der Endpunkt akzeptiert einen optionalen Parameter `draw_strategy` (z. B. `'random_weighted'` oder `'first_come_first_serve'`). Das System wählt für jede Gewinnstufe die entsprechende Anzahl an Teilnehmern aus, aktualisiert deren Status auf `won` und löst automatisch einen Webhook an die konfigurierte URL aus. Die Ergebnisse werden in der Tabelle `draw_results` mit den Feldern `campaign_id`, `tier`, `participant_id` und `drawn_at` gespeichert.

### Beste Praktiken

1. **Batch-Verarbeitung für große Teilnahmemengen**  
   Bei Kampagnen mit mehr als 10.000 Teilnahmen pro Stunde sollten Sie die Teilnahmeerfassung asynchron über eine Message Queue (z. B. RabbitMQ oder Kafka) abwickeln. Der Haupt-API-Endpunkt sollte nur die Validierung und das Einreihen in die Queue übernehmen, während ein Worker-Prozess die Deduplizierung, Fraud-Prüfung und Speicherung in PostgreSQL durchführt. Dies reduziert die Latenz pro Request auf unter 20 Millisekunden und verhindert Timeouts bei hohem Durchsatz.

2. **Regelmäßige Index-Optimierung und Datenbereinigung**  
   Überwachen Sie die Performance der Datenbankabfragen mittels `EXPLAIN ANALYZE`. Stellen Sie sicher, dass die Tabellen `participations` und `draw_results` einen zusammengesetzten Index auf `(campaign_id, status, created_at)` besitzen. Führen Sie wöchentlich ein `VACUUM ANALYZE` auf PostgreSQL durch, um die Abfragegeschwindigkeit zu erhalten. Archivieren Sie abgeschlossene Kampagnen älter als 30 Tage in eine separate History-Tabelle, um die aktive Datenbank schlank zu halten.
