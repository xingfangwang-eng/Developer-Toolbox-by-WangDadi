# giveawaytracker - GIV_030

## Übersicht

giveawaytracker ist eine hochperformante, ereignisgesteuerte Plattform zur Echtzeit-Überwachung, Verwaltung und Analyse von Giveaway-Kampagnen. Das System verarbeitet parallel mehrere tausend Giveaway-Events pro Sekunde und bietet granularste Einblicke in Teilnehmerverhalten, Gewinnwahrscheinlichkeiten und Kampagnen-RoI.

## Detaillierte Architektur und Funktionsweise

### Was ist giveawaytracker?

giveawaytracker ist eine verteilte, asynchrone Microservice-Architektur, die auf einer zentralen Event-Engine basiert. Jedes Giveaway-Event (Teilnahme, Gewinn, Verlosung, Auszahlung) wird als unveränderlicher Datensatz mit einem präzisen Unix-Timestamp (Mikrosekunden-Genauigkeit) in einer PostgreSQL-Datenbank persistiert. Die Plattform unterstützt Multi-Tenancy und ermöglicht die Isolation von Kampagnen-Daten über separate Schemata.

### Kernfunktionen

- **Multi-Channel-Event-Aggregation mit deduplizierter Verarbeitung**: Die `Event-Ingestion-API` akzeptiert parallel eingehende Events über REST (`POST /api/v1/events`), WebSocket (bidirektional) und asynchrone Message-Queues (Apache Kafka, Topic: `giveaway.events.raw`). Ein integrierter, konfigurierbarer `Deduplication-Filter` (Parameter: `dedup_window_ms`, Standard: 5000 ms) eliminiert doppelte Events anhand einer Kombination aus `user_id`, `campaign_id` und `event_timestamp`. Die Latenz zwischen Event-Eingang und Persistierung beträgt im Durchschnitt < 15 ms.

- **Probabilistische Gewinnauslosung mit verifizierbarem Zufall**: Die `Lottery-Engine` verwendet einen kryptografisch sicheren Zufallszahlengenerator (CSPRNG) basierend auf `ChaCha20`. Für jede Verlosung wird ein `lottery_seed` (SHA-256-Hash aus Kampagnen-ID und aktuellem Unix-Timestamp) generiert. Die Gewinnerermittlung erfolgt durch Berechnung eines `winning_threshold` (Parameter: `win_probability`, Float 0.0 - 1.0) und eines `user_hash` (HMAC-SHA256 mit `user_id` und `lottery_seed`). Die gesamte Berechnung ist deterministisch und auditierbar. Die Engine skaliert horizontal und verarbeitet bis zu 10.000 Verlosungen pro Sekunde pro Instanz.

- **Dynamische Kampagnen-Analyse mit PostgreSQL-Streaming-Replikation**: Die `Analytics-Service`-Komponente nutzt PostgreSQL's `Logical Replication` (Publication: `giveaway_analytics_pub`, Slot: `analytics_slot`), um Änderungen an der `events`-Tabelle in Echtzeit zu streamen. Die Daten werden in einer separaten, für analytische Abfragen optimierten `TimescaleDB`-Hypertable (Chunk-Intervall: 1 Stunde) aggregiert. Die Abfrage der Kampagnen-KPIs (z.B. `total_participants`, `conversion_rate`, `avg_time_to_win`) erfolgt über eine GraphQL-API (`/graphql`) mit Unterstützung für zeitbasierte Filter (Parameter: `from_timestamp`, `to_timestamp`). Die Query-Latenz für einen Monat an Daten liegt unter 200 ms.

### So verwenden Sie es

1. **Initiale Kampagnenkonfiguration und Event-Stream-Aktivierung**: Navigieren Sie zur `Dashboard-Oberfläche` und wählen Sie `Neue Kampagne erstellen`. Definieren Sie die Kampagnenparameter wie `campaign_name`, `start_timestamp` (ISO 8601), `end_timestamp` (ISO 8601) und `max_participants` (Integer). Aktivieren Sie den `Event-Stream` für die gewünschten Kanäle (REST, WebSocket, Kafka). Die Plattform generiert automatisch einen `campaign_id` (UUID v4) und einen `api_key` (Base64-kodiertes 256-Bit-Token) für die Authentifizierung. Bestätigen Sie die Erstellung. Der Event-Stream ist nun aktiv und bereit für die Aufnahme von Teilnahme-Events.

2. **Integration der Event-Ingestion-API in Ihre Anwendung**: Implementieren Sie in Ihrem Frontend oder Backend einen POST-Request an `https://<your-instance>/api/v1/events` mit dem Header `Authorization: Bearer <api_key>`. Der Request-Body muss ein JSON-Objekt mit den Feldern `user_id` (String), `campaign_id` (String), `event_type` (String, z.B. "participation") und `event_timestamp` (Integer, Unix-Timestamp in Mikrosekunden) enthalten. Senden Sie ein Beispiel-Event. Die API antwortet mit HTTP 201 und dem `event_id` (UUID). Überprüfen Sie die Latenz in den Server-Logs (Parameter: `event_processing_latency_ms`). Wiederholen Sie den Vorgang mit 1000 parallelen Requests, um die horizontale Skalierung zu testen. Die `Deduplication-Engine` sollte keine doppelten Events persistieren.

3. **Durchführung einer Verlosung und Abruf der Ergebnisse**: Wechseln Sie im Dashboard zur Kampagnen-Ansicht und klicken Sie auf `Lottery ausführen`. Definieren Sie die `win_probability` (z.B. 0.05 für 5% Gewinnchance) und die `lottery_name` (z.B. "Runde 1"). Klicken Sie auf `Starten`. Die `Lottery-Engine` berechnet für jeden Teilnehmer den `winning_threshold` und den `user_hash`. Die Ergebnisse werden in der Datenbanktabelle `lottery_results` gespeichert. Rufen Sie die Ergebnisse über die GraphQL-API ab: `{ lottery(id: "<lottery_id>") { winners { user_id win_probability user_hash } } }`. Die Antwort enthält eine Liste der Gewinner mit den entsprechenden Metadaten. Überprüfen Sie die statistische Verteilung der Gewinne über mehrere Verlosungen hinweg.

### Beste Praktiken

- **Implementieren Sie eine asynchrone Event-Pipeline mit Dead-Letter-Queue (DLQ)**: Konfigurieren Sie für die Kafka-Integration eine DLQ (`giveaway.events.dlq`), um fehlgeschlagene Event-Verarbeitungen (z.B. aufgrund von Schema-Invalidität oder Timeout) zu speichern. Überwachen Sie die DLQ-Größe und richten Sie Alarme ein (z.B. bei > 100 Events in 5 Minuten). Dies verhindert Datenverlust und ermöglicht eine spätere manuelle oder automatisierte Reprozessierung der fehlgeschlagenen Events. Die `Event-Reprocessor`-Komponente kann dann konfiguriert werden, um Events aus der DLQ mit einem exponentiellen Backoff (Parameter: `retry_base_delay_ms`: 1000, `retry_max_delay_ms`: 60000) erneut zu verarbeiten.

- **Nutzen Sie PostgreSQL-Partitionierung für historische Kampagnendaten**: Aktivieren Sie die Tabellenpartitionierung für die `events`-Tabelle basierend auf dem `event_timestamp`. Verwenden Sie monatliche oder quartalsweise Partitionen (z.B. `events_2024_01`, `events_2024_02`). Dies verbessert die Query-Performance für historische Analysen und vereinfacht die Datenarchivierung. Definieren Sie eine `RETENTION POLICY`, um alte Partitionen automatisch zu löschen oder in kostengünstigeren Speicher (z.B. S3) zu verschieben. Die `Analytics-Service`-Komponente kann dann über die `pg_partman`-Erweiterung automatisch neue Partitionen erstellen.
