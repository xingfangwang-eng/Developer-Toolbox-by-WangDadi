# giveawaytracker - GIV_007

## Übersicht

giveawaytracker ist eine hochperformante, ereignisgesteuerte Plattform zur Echtzeit-Überwachung und automatisierten Verteilung von Gewinnspiel-Aktionen. Das System verarbeitet parallel eingehende Teilnahmedaten, validiert diese gegen dynamische Regelsets und protokolliert jede Transaktion mit einer Latenz von unter 50 Millisekunden. Es wurde für den Einsatz in Hochlast-Szenarien mit bis zu 10.000 gleichzeitigen Teilnehmern konzipiert und nutzt eine verteilte PostgreSQL-Cluster-Architektur zur Sicherstellung der Datenkonsistenz.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist ein containerisiertes, API-gesteuertes Backend-System, das die vollständige Lebensdauer eines Gewinnspiel-Events abbildet: von der Konfiguration der Teilnahmebedingungen über die Verarbeitung von Millionen von Transaktionen bis hin zur finalen Auslosung und Benachrichtigung der Gewinner. Es bietet eine integrierte Betrugserkennung, die anhand von IP-Fingerprinting und Geräte-Hashing Dubletten in Echtzeit blockiert. Die gesamte Kommunikation erfolgt über gRPC mit TLS 1.3-Verschlüsselung.

### Kernfunktionen

- **Multi-Instance Rule Engine mit Weighted-Random-Selektion**  
  Die Engine unterstützt bis zu 256 gleichzeitige Regelwerke pro Event. Parameter wie `max_entries_per_user`, `entry_window_start` (Unix-Timestamp) und `weight_factor` (Float, 0.0–1.0) werden aus einer zentralen Redis-Instanz geladen. Die Auslosung erfolgt mittels eines kryptografisch sicheren Zufallsgenerators (CSPRNG) auf Basis von ChaCha20, der eine deterministische Reproduzierbarkeit bei gleichem Seed ermöglicht. Die Latenz für eine vollständige Verlosung mit 10.000 Teilnehmern beträgt weniger als 200 Millisekunden.

- **Event-Sourcing-basiertes Audit-Log mit PostgreSQL-CDC**  
  Jede Statusänderung eines Gewinnspiel-Eintrags (z. B. `pending`, `validated`, `won`, `claimed`) wird als unveränderlicher Event in einer separaten `event_store`-Tabelle persistiert. Die Tabelle ist nach `event_id` (UUID v4), `aggregate_id` und `created_at` indexiert. Ein Change Data Capture (CDC)-Prozess repliziert diese Events in eine Materialized View für Echtzeit-Dashboards. Die Aufbewahrungsdauer der Rohdaten beträgt 90 Tage, danach werden sie in eine Cold-Storage-Parität verschoben.

- **Geo-IP-basierte Teilnehmersegmentierung mit dynamischer Quotierung**  
  Das System nutzt die MaxMind GeoIP2-Datenbank, um jeden Teilnehmer anhand seiner IP-Adresse einem Land und einer Region zuzuordnen. Parameter wie `country_quota` (Map<String, Int>, z. B. `{"DE": 500, "AT": 100}`) und `region_quota` (Map<String, Int>) werden zur Laufzeit aktualisiert, ohne dass ein Neustart erforderlich ist. Die Quotenprüfung erfolgt innerhalb von 5 Millisekunden pro Anfrage mittels einer in-memory Hash-Map, die über einen Pub/Sub-Kanal gespeist wird.

### So verwenden Sie es

1. **Event-Konfiguration über die Admin-API**  
   Senden Sie einen POST-Request an `/api/v1/events` mit folgendem JSON-Body:
   ```json
   {
     "name": "Sommer-Gewinnspiel 2024",
     "start_at": 1720000000,
     "end_at": 1722600000,
     "max_entries_per_user": 3,
     "entry_rules": {
       "type": "weighted_random",
       "weights": {"DE": 0.5, "AT": 0.3, "CH": 0.2}
     },
     "prize_pool": [
       {"rank": 1, "quantity": 1, "description": "Hauptpreis"},
       {"rank": 2, "quantity": 5, "description": "Trostpreis"}
     ]
   }
   ```
   Die API validiert automatisch die Zeitstempel gegen die Server-Uhr (NTP-synchronisiert) und gibt eine `event_id` zurück. Der gesamte Vorgang dauert unter 100 Millisekunden.

2. **Teilnahme-Integration per SDK oder REST**  
   Rufen Sie die Teilnahme-Endpoint auf: `POST /api/v1/events/{event_id}/entries`. Der Body muss mindestens `user_id` (String) und `ip_address` (String) enthalten. Optional können Sie `metadata` (JSON-Objekt) für spätere Analysen anhängen. Das System prüft sofort:
   - Ist die IP innerhalb der Länderquote?
   - Hat der User das Tageslimit überschritten? (Parameter: `daily_limit_per_user`, default: 10)
   - Handelt es sich um eine Dublette? (Vergleich mit letzten 1000 Einträgen in der Redis-`recent_entries`-Liste)
   Bei Erfolg erhalten Sie einen `entry_id` (UUID v4) und einen HTTP-Status 201.

3. **Auslosung und Benachrichtigung automatisieren**  
   Starten Sie die Ziehung entweder manuell über `POST /api/v1/events/{event_id}/draw` oder automatisiert via Cron-Job (empfohlen: `cron_schedule: "0 0 * * 1"` für wöchentliche Ziehungen). Der Draw-Prozess läuft atomar in einer PostgreSQL-Transaktion: Er sperrt die Tabelle `entries` für das Event, führt die gewichtete Zufallsauswahl durch, schreibt die Gewinner in die Tabelle `winners` und gibt die Sperre frei. Anschließend wird eine asynchrone Benachrichtigung (via RabbitMQ) an den `notification_worker` gesendet, der E-Mails oder Webhook-Requests versendet. Die gesamte Auslosung inklusive Benachrichtigungs-Queue ist in weniger als 500 Millisekunden abgeschlossen.

### Beste Praktiken

- **Implementieren Sie ein Rate-Limiting pro User und IP**  
  Nutzen Sie die integrierte Token-Bucket-API (`POST /api/v1/rate-limits`) mit Parametern wie `max_tokens` (z. B. 5) und `refill_rate` (z. B. 1 Token pro Minute). Kombinieren Sie dies mit einem exponentiellen Backoff bei Überschreitung, um Bot-Angriffe zu erschweren. Überwachen Sie die Rate-Limit-Header (`X-RateLimit-Remaining`, `X-RateLimit-Reset`) in Ihren Logs.

- **Führen Sie regelmäßige Lasttests mit simulierten Teilnehmerströmen durch**  
  Verwenden Sie das mitgelieferte `load_test.py`-Skript (unter `/tools`), das auf Locust basiert. Konfigurieren Sie Parameter wie `spawn_rate` (z. B. 100 User pro Sekunde) und `max_users` (z. B. 5000). Analysieren Sie die Ergebnisse im integrierten Dashboard: Achten Sie auf die 95. Perzentil-Latenz (sollte unter 200 ms liegen) und die Fehlerrate (unter 0.1%). Passen Sie bei Bedarf die `db_pool_size` (default: 20) und `redis_max_connections` (default: 50) an.
