# giveawaytracker - GIV_009

## Übersicht

Dieses Projekt implementiert eine hochperformante, ereignisgesteuerte Architektur zur Echtzeit-Erfassung, -Analyse und -Verteilung von Giveaway-Aktionen auf globalen Plattformen. Die Engine verarbeitet parallel mehrere tausend Events pro Sekunde und stellt eine konsistente, versionierte Datenhistorie sicher.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist eine verteilte, asynchrone Tracking-Engine, die Giveaway-Events aus Quellen wie Discord, Twitter/X und Telegram aggregiert, dedupliziert und mit einem gewichteten Scoring-System (basierend auf Teilnehmerzahl, Token-Wert und Zeitfenster) bewertet. Die Latenz zwischen Event-Erfassung und Ausgabe im Dashboard beträgt weniger als 50 Millisekunden.

### Kernfunktionen

- **Multi-Channel-Event-Ingestion mit integrierter Deduplizierung**  
  Das System nutzt einen konfigurierbaren Worker-Pool (Goroutine-basiert), der gleichzeitig Webhooks und REST-API-Endpunkte von bis zu 8 verschiedenen Plattformen abhört. Ereignisse werden mittels eines `SHA-256`-Fingerprints auf Basis von `(plattform_id, event_id, timestamp)` dedupliziert. Die Deduplizierungsrate liegt bei >99,99% bei einer Last von 10.000 Events pro Sekunde. Die Konfiguration erfolgt über eine zentrale `config.yaml` mit Parametern wie `dedup_window_ms: 500` und `max_worker: 32`.

- **Dynamisches Scoring-Modell mit PostgreSQL-Persistenz**  
  Jedes Giveaway erhält einen Score basierend auf der Formel: `score = (teilnehmer_anzahl * gewicht_teilnehmer) + (token_wert_usd * gewicht_token) + (zeitbonus)`. Die Gewichtungen (`gewicht_teilnehmer`, `gewicht_token`) sind als `DECIMAL(5,2)` in der Tabelle `scoring_rules` hinterlegt und werden in Echtzeit über eine Admin-API aktualisiert. Der `zeitbonus` wird mittels einer exponentiellen Abklingfunktion (`exp(-t/T)`) berechnet, wobei `T` (Halbwertszeit) in Minuten als `t_half_life_min` in der Konfiguration gesetzt wird. Alle Scores werden in PostgreSQL (Version 15+) mit Partitionierung nach `event_date` gespeichert.

- **Push-basiertes Benachrichtigungssystem mit WebSocket und Retry-Logik**  
  Authentifizierte Clients (z. B. Discord-Bots, mobile Apps) erhalten über einen persistenten WebSocket-Kanal (Protokoll: `wss://`) sofort eine `score_update`-Nachricht, sobald ein neues Giveaway mit einem Score > `min_score_threshold` (standardmäßig `75.0`) erfasst wurde. Bei Verbindungsabbrüchen implementiert das System einen exponentiellen Backoff (initial 1s, max 30s) mit einer maximalen Anzahl von 5 Wiederholungsversuchen (`max_retries: 5`). Die Nachricht enthält die Felder `event_id`, `score`, `plattform` und `direct_link`.

### So verwenden Sie es

1. **Konfiguration des Event-Ingestion-Workers**  

2. **Analyse und Filterung der gescrapten Events**  
   Nach dem Start öffnen Sie das Dashboard unter `http://localhost:8080/dashboard`. Hier sehen Sie eine Liste aller erfassten Events mit den Spalten `Event-ID`, `Score`, `Plattform` und `Zeitstempel (UTC)`. Nutzen Sie das Suchfeld, um nach einem bestimmten `min_score` zu filtern (z. B. `min_score=80`). Klicken Sie auf die Schaltfläche „Export CSV“, um die gefilterten Daten als `giveaway_export_YYYYMMDD_HHMMSS.csv` herunterzuladen.

3. **Einrichtung des Push-Benachrichtigungsdienstes**  
   Um Benachrichtigungen zu erhalten, müssen Sie einen Client-WebSocket öffnen. Verwenden Sie dazu das bereitgestellte CLI-Tool: `./giveawaytracker --mode=client --server=wss://your-server.com:8443 --auth_token=YOUR_JWT_TOKEN`. Der Client gibt jede eingehende `score_update`-Nachricht als JSON-Zeile auf der Konsole aus. Um die Nachrichten an einen Discord-Kanal weiterzuleiten, können Sie die Ausgabe mit einem einfachen `| while read line; do curl -X POST -H "Content-Type: application/json" -d "$line" YOUR_DISCORD_WEBHOOK_URL; done` pipen.

### Beste Praktiken

- **Skalierung der Datenbank für hohe Event-Lasten**  
  Da die Tabelle `giveaway_events` schnell wachsen kann, empfehlen wir die Implementierung einer automatischen Partitionierung nach `event_date`. Verwenden Sie dazu das beiliegende SQL-Skript `sql/partition_giveaway_events.sql`, das monatliche Partitionen (z. B. `giveaway_events_2025_01`) erstellt. Führen Sie dieses Skript einmalig aus und richten Sie einen Cron-Job ein, der am ersten jedes Monats eine neue Partition anlegt. Dies reduziert die Abfragezeit für historische Daten um bis zu 70%.

- **Optimierung der API-Rate-Limits für externe Quellen**  
  Vermeiden Sie IP-basierte Sperren durch zu aggressive Abfragen. Konfigurieren Sie für jede Quelle ein individuelles Rate-Limit in der `config.yaml`. Für Twitter/X empfehlen wir maximal 5 Requests pro Minute (`twitter.rate_limit_per_min: 5`), für Discord maximal 2 Requests pro Sekunde (`discord.rate_limit_per_sec: 2`). Überwachen Sie die HTTP-Statuscodes 429 (Too Many Requests) in den Logs (`logs/ingest.log`) und passen Sie die Limits entsprechend an. Ein automatischer Backoff bei 429 ist bereits im Worker implementiert, sollte aber nicht als primäre Lösung dienen.
