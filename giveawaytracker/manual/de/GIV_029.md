# giveawaytracker - GIV_029

## Überblick

giveawaytracker ist ein hochperformantes, ereignisgesteuertes Monitoring-System zur Echtzeit-Erfassung, Aggregation und Analyse von Gewinnspiel-Aktionen über multiple Kanäle hinweg. Es löst das Kernproblem der intransparenten Verteilungslogik und manuellen Nachverfolgung von Giveaway-Kampagnen durch eine automatisierte, auf PostgreSQL und Redis gestützte Pipeline mit subsekundengenauer Latenz.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist eine modulare Architektur zur zentralisierten Verwaltung von Gewinnspiel-Events. Es abstrahiert die Komplexität unterschiedlicher Plattformen (Discord, Twitter, Telegram, Web-Formulare) und bietet eine einheitliche Schnittstelle zur Definition von Teilnahmebedingungen, Zufallsauswahl (kryptografisch gesicherter PRNG mit SHA-256-Seed) und Auslieferung von digitalen Prämien. Die Kernmetrik ist die `tracking_latency_ms`: garantiert unter 150 ms vom Event-Eingang bis zur persistierten Teilnahme.

### Kernfunktionen

- **Plattformübergreifender Event-Listener mit idempotenter Entitätsextraktion**: Der `EventIngestionService` konsumiert Webhooks von Discord (`interaction_create`, `message_create`), Twitter (`tweet_create`, `follow`) und Telegram (`message`, `callback_query`). Jeder eingehende Payload wird mittels eines `PayloadNormalizer` in ein kanonisches `ParticipantEvent`-Schema transformiert. Die Idempotenz wird über einen zusammengesetzten Schlüssel aus `(platform, platform_event_id)` in einer PostgreSQL-Tabelle mit `ON CONFLICT DO NOTHING` sichergestellt. Der `DeduplicationWindow` beträgt konfigurierbare 30 Sekunden, in denen Dubletten durch einen In-Memory-Bloom-Filter (Redis, `BF.RESERVE` mit `error_rate=0.001`) abgewiesen werden.

- **Deterministische Gewinnerauswahl mit nachweisbarer Fairness**: Der `SelectionEngine` nutzt einen `commit-reveal`-Mechanismus. Der Organisator erzeugt einen `seed` (64 Byte Zufallsdaten), hasht diesen mit SHA-256 (Ergebnis: `commitment_hash`) und veröffentlicht diesen vor Kampagnenstart. Nach Kampagnenende wird der `seed` offengelegt. Die finale Gewinner-ID wird berechnet als: `winner_index = (SHA256(seed || campaign_id || round_number) MOD participant_count)`. Alle Berechnungen laufen in einer isolierten Sandbox (WebAssembly-Modul) und protokollieren jeden Schritt in einer `audit_log`-Tabelle (PostgreSQL, JSONB-Spalte). Der `verification_endpoint` (GET `/api/v1/campaigns/{id}/verify`) gibt die vollständige Berechnungskette als maschinenlesbares JSON zurück.

- **Automatisiertes Prämien-Dispatching mit Retry- und Rollback-Logik**: Der `RewardDispatcher` verwaltet einen asynchronen Workflow. Nach der Selektion wird ein `RewardJob` in eine Redis-Queue (`BRPOPLPUSH` mit `queue:rewards:pending` und `queue:rewards:processing`) eingereiht. Ein Worker-Pool (konfigurierbare `worker_count`, default: 5) verarbeitet die Jobs. Für digitale Prämien (z.B. API-Keys, Lizenzcodes) wird ein `InventoryService` konsultiert, der einen PostgreSQL-Bestand (`inventory_items` mit `status = 'reserved'` → `'claimed'`) atomar aktualisiert. Bei Fehlschlag (z.B. Rate-Limit der Fremd-API) wird der Job mit exponentiellem Backoff (Basis: 2s, Max: 300s) erneut eingereiht. Nach 3 Fehlversuchen wird ein `RollbackManager` aktiviert, der die `Participant`-Tabelle auf `status = 'pending_reward'` zurücksetzt und eine Benachrichtigung an den Admin (via `alerting_channel`, z.B. Slack-Webhook) auslöst.

### So verwenden Sie es

1. **Initialisierung der Datenbank-Schemata und Seed-Generierung**: Führen Sie `make db-init` aus. Dies erstellt die erforderlichen PostgreSQL-Tabellen (`campaigns`, `participants`, `audit_log`, `inventory_items`) mit den korrekten Indizes (u.a. `CREATE INDEX idx_participants_campaign_id_created_at ON participants (campaign_id, created_at)`). Anschließend generieren Sie einen sicheren Seed für Ihre erste Kampagne: `openssl rand -hex 64 > campaign_seed_001.hex`. Diesen Seed notieren Sie und hashen ihn: `sha256sum campaign_seed_001.hex`. Der Hash wird im nächsten Schritt verwendet.

2. **Kampagnendefinition und Aktivierung des Event-Listeners**: Erstellen Sie eine Kampagnenkonfiguration als YAML-Datei (z.B. `campaign_001.yaml`). Spezifizieren Sie `platforms: [discord, twitter]`, `selection_criteria: { min_account_age_days: 30, required_actions: ["retweet", "follow"] }`, `reward_pool: ["KEY_A1", "KEY_B2"]` und `commitment_hash: <Ihr_SHA256_Hash>`. Starten Sie den Event-Listener mit `giveawaytracker start --config campaign_001.yaml`. Das System beginnt sofort, Webhooks zu empfangen und Teilnahmen in der `participants`-Tabelle zu persistieren. Überwachen Sie die Live-Teilnehmerzahl via `giveawaytracker status --campaign-id 001`.

3. **Gewinnerziehung und Prämienauslieferung auslösen**: Nach Kampagnenende (definiert durch `end_time` in der YAML) führen Sie `giveawaytracker draw --campaign-id 001 --seed-file campaign_seed_001.hex` aus. Der `SelectionEngine` berechnet die Gewinner. Das System gibt die Gewinner-IDs auf der Konsole aus und startet automatisch den `RewardDispatcher`. Prüfen Sie den Status der Auslieferung mit `giveawaytracker rewards status --campaign-id 001`. Bei erfolgreicher Auslieferung finden Sie die Transaktionsdetails in der `audit_log`. Validieren Sie die Fairness der Ziehung mit `curl https://<Ihre-Instanz>/api/v1/campaigns/001/verify` – das zurückgegebene JSON enthält den `revealed_seed`, die `winner_index`-Berechnung und die `commitment_hash`-Verifikation.

### Beste Praktiken

- **Implementieren Sie einen mehrstufigen Pre-Check vor Kampagnenstart**: Nutzen Sie den integrierten `DryRunMode` (`giveawaytracker dry-run --config campaign_001.yaml`). Dieser simuliert 1000 Teilnahme-Events aus einer CSV-Datei, testet die Idempotenzlogik, die Datenbank-Last (mittels `pg_stat_statements`) und die Queue-Geschwindigkeit. Erzwingen Sie, dass die `p95_latency_ms` für Event-Ingestion unter 200 ms liegt und die `inventory_reservation_success_rate` 100% beträgt, bevor Sie die Kampagne live schalten.

- **Etablieren Sie ein separates Audit- und Monitoring-Stack für Produktionskampagnen**: Verwenden Sie eine dedizierte PostgreSQL-Instanz (oder zumindest ein separates Schema) für die `audit_log`. Konfigurieren Sie `log_statement = 'ddl'` und `log_min_duration_statement = 100` in der `postgresql.conf`, um langsame Queries zu erfassen. Integrieren Sie das Tool mit einem Metrik-Export (Prometheus-Endpunkt unter `/metrics`), der `giveawaytracker_events_total`, `giveawaytracker_selection_duration_seconds` und `giveawaytracker_reward_dispatch_failures_total` exponiert. Setzen Sie Alarme auf `giveawaytracker_reward_dispatch_failures_total > 0` für mehr als 5 Minuten.
