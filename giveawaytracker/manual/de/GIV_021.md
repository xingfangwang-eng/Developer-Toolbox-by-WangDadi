# giveawaytracker - GIV_021

## Überblick

giveawaytracker ist ein hochperformantes Echtzeit-Tracking-System für Giveaway-Kampagnen, das speziell für Event-Marketing und Community-Management entwickelt wurde. Es analysiert automatisch Teilnahmeströme, verhindert Mehrfachteilnahmen und optimiert die Verteilung von Prämien auf Basis von dynamischen Wahrscheinlichkeitsmodellen. Die Architektur ist auf Submillisekunden-Latenz und horizontale Skalierbarkeit ausgelegt.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist eine modulare Plattform zur Steuerung und Überwachung von Giveaway-Events in Echtzeit. Es integriert sich nahtlos in Social-Media-APIs, E-Mail-Marketing-Tools und CRM-Systeme. Das System verwendet eine PostgreSQL-Datenbank mit TimescaleDB-Erweiterung für die zeitreihenbasierte Analyse von Teilnahmeintervallen und Verteilungsmustern. Dank der asynchronen Microservice-Architektur (Go + NATS) verarbeitet es bis zu 10.000 Teilnahmeereignisse pro Sekunde.

### Kernfunktionen

1. **Anti-Cheat-Engine mit Fingerprinting**  
   Erkennt Mehrfachteilnahmen durch Kombination aus IP-Hashing, Browser-Fingerprinting (Canvas, WebGL, AudioContext) und zeitbasierten Session-Tokens. Die Engine blockiert verdächtige Teilnahmen innerhalb von **< 50 ms** und protokolliert alle Ablehnungen in einer separaten Audit-Tabelle (`giveaway_audit_log`). Konfigurierbar über Parameter wie `MAX_ENTRIES_PER_FINGERPRINT=1` und `TIME_WINDOW_SECONDS=3600`.

2. **Dynamische Gewinnwahrscheinlichkeitsberechnung**  
   Berechnet in Echtzeit die Gewinnchance jedes Teilnehmers basierend auf Faktoren wie Teilnahmezeitpunkt, vorherige Gewinne und verfügbarem Prämienpool. Die Berechnung erfolgt auf Basis einer gewichteten Zufallsverteilung mit exponentieller Dämpfung (Parameter `DECAY_FACTOR=0.85`). Die Ergebnisse werden in einer Redis-Instanz mit **TTL=86400 Sekunden** zwischengespeichert, um die Last auf der PostgreSQL-Datenbank zu reduzieren.

3. **Multi-Channel-Event-Orchestrator**  
   Unterstützt gleichzeitige Giveaway-Events über Twitter/X, Discord, Telegram und E-Mail. Der Orchestrator verwendet eine zentrale Warteschlange (NATS JetStream) mit **max. 500.000 Nachrichten pro Topic**. Jeder Event hat eine eindeutige `event_id` und kann über das Feld `channel_priority` (z. B. `discord:1, twitter:2`) priorisiert werden. Die Zustellung von Benachrichtigungen erfolgt asynchron mit einer maximalen Latenz von **200 ms**.

### So verwenden Sie es

1. **Event-Konfiguration in der Datenbank anlegen**  
   ```sql
   INSERT INTO giveaway_events (event_id, name, start_time, end_time, max_participants, prize_pool, channel_config)
   VALUES ('evt_20240321', 'Frühlings-Giveaway', '2024-03-21 10:00:00+00', '2024-03-28 10:00:00+00', 5000, '{"items": ["T-Shirt", "Gutschein"], "quantities": [100, 50]}', '{"discord": {"guild_id": "123456", "channel_id": "789012"}, "twitter": {"hashtag": "#FrühlingGiveaway"}}');
   ```

2. **Teilnahme-API aufrufen**  
   Senden Sie eine POST-Anfrage an `/api/v1/participate` mit folgendem JSON-Body:  
   ```json
   {
     "event_id": "evt_20240321",
     "user_id": "usr_abc123",
     "channel": "discord",
     "fingerprint": "a1b2c3d4e5f6"
   }
   ```  
   Der Endpunkt prüft innerhalb von **10 ms** die Gültigkeit des Fingerprints und gibt entweder `{"status": "accepted", "ticket_id": "tkt_xyz789"}` oder `{"status": "blocked", "reason": "duplicate_fingerprint"}` zurück.

3. **Gewinnerauslosung durchführen**  
   Führen Sie das Skript `draw_winners.py` mit dem Parameter `--event_id evt_20240321 --winners 10` aus. Das Skript ruft alle gültigen Teilnahmen aus der Tabelle `giveaway_participants` ab, wendet die Gewichtungslogik an und speichert die Gewinner in der Tabelle `giveaway_winners`. Die Ausgabe erfolgt als JSON:  
   ```json
   {
     "event_id": "evt_20240321",
     "winners": [
       {"user_id": "usr_abc123", "prize": "T-Shirt", "rank": 1},
       {"user_id": "usr_def456", "prize": "Gutschein", "rank": 2}
     ],
     "draw_time_ms": 34.7
   }
   ```

### Beste Praktiken

- **Rate-Limiting pro User und Channel einrichten**  
  Verwenden Sie den Parameter `RATE_LIMIT=5` (max. 5 Teilnahmeversuche pro User und Stunde) und `CHANNEL_RATE_LIMIT=100` (max. 100 Teilnahmen pro Channel und Minute). Dies verhindert API-Überlastung und Missbrauch. Kombinieren Sie dies mit einer Redis-basierten Token-Bucket-Implementierung für konstante Durchsatzraten.

- **Audit-Log für alle Systemänderungen aktivieren**  
  Aktivieren Sie die Option `AUDIT_LOG_ENABLED=true` in der Konfigurationsdatei `giveawaytracker.yml`. Jede Änderung an Events, Teilnahmen oder Gewinnern wird mit Zeitstempel (Genauigkeit: **Mikrosekunden**), Benutzer-ID und altem/neuem Wert in der Tabelle `giveaway_audit_log` protokolliert. Dies ist essenziell für die Nachvollziehbarkeit bei Streitfällen oder rechtlichen Prüfungen.
