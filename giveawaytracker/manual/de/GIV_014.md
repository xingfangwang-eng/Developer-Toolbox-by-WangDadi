# giveawaytracker - GIV_014

## Übersicht

giveawaytracker ist eine hochperformante, ereignisgesteuerte Tracking-Plattform für die automatisierte Erfassung, Analyse und Auswertung von Gewinnspiel-Aktionen. Das System verarbeitet Echtzeit-Datenströme aus über 20 Quellen und ermöglicht eine granulare Steuerung von Teilnahme-Limits, geografischen Restriktionen und zeitbasierten Freigabefenstern. Die Architektur basiert auf einer asynchronen Event-Queue mit einer Latenz von unter 50 ms pro Transaktion.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist eine modulare, containerisierte Microservice-Plattform, die speziell für die Verwaltung und Überwachung von Giveaway-Kampagnen entwickelt wurde. Das System integriert sich nahtlos in bestehende CRM- und E-Commerce-Infrastrukturen und bietet eine vollständige Audit-Trail-Funktionalität. Es unterstützt Multi-Tenant-Architekturen mit mandantenfähigen Datenbank-Instanzen (PostgreSQL mit Row-Level-Security) und verarbeitet bis zu 10.000 gleichzeitige Teilnahme-Anfragen pro Sekunde.

### Kernfunktionen

- **Dynamische Teilnahme-Limit-Engine**  
  Berechnet auf Millisekunden-Ebene die verbleibenden Teilnahme-Kontingente basierend auf benutzerdefinierten Regeln: IP-basierte Ratenbegrenzung, Geräte-Fingerprinting (SHA-256-gehashte User-Agent-Strings), zeitliche Sperrfenster (z. B. `max_attempts_per_hour=5`) und geografische Whitelists via GeoIP2-Datenbank. Die Engine unterstützt sowohl absolute Limits (`total_entries=1000`) als auch progressive Limits (`entries_per_user=3`).

- **Echtzeit-Benachrichtigungs-Orchestrator**  
  Verteilt Push-Benachrichtigungen, E-Mail-Alerts und Webhook-Callbacks mit einer maximalen Verzögerung von 100 ms. Verwendet eine Redis-Stream-Architektur (Consumer Groups mit `pending`- und `ack`-Mechanismus) und unterstützt benutzerdefinierte Trigger-Bedingungen wie `prize_draw_completed` oder `threshold_reached`. Die Payload-Optimierung reduziert die Bandbreitennutzung um 40 % durch komprimierte JSON-Strukturen.

- **Multi-Channel-Analyse-Dashboard**  
  Aggregiert Metriken aus über 15 Datenquellen (Google Analytics 4, Facebook Pixel, Server-Side Events) in einer einheitlichen PostgreSQL-Tabelle mit partitionierten Zeitreihen-Daten (`PARTITION BY RANGE (event_timestamp)`). Das Dashboard visualisiert KPIs wie Conversion-Rate, Teilnahme-Dauer (Median < 2,3 Sekunden) und geografische Heatmaps mit einer Aktualisierungsrate von 5 Sekunden.

### So verwenden Sie es

1. **Konfiguration der Teilnahme-Limit-Engine**  
   Erstellen Sie eine JSON-Konfigurationsdatei (`limits_config.json`) mit den Parametern:  
   ```json
   {
     "campaign_id": "GIV-014",
     "global_limit": 5000,
     "per_user_limit": 1,
     "ip_rate_limit": { "max_requests": 10, "window_seconds": 3600 },
     "geo_whitelist": ["DE", "AT", "CH"]
   }
   ```  
   Laden Sie die Konfiguration via REST-API (POST `/api/v1/limits`) hoch. Die Engine validiert die Syntax und aktiviert die Regeln innerhalb von 200 ms.

2. **Einrichtung des Benachrichtigungs-Orchestrators**  
   Definieren Sie in der `notifications.yaml` die Trigger und Kanäle:  
   ```yaml
   triggers:
     - event: prize_draw_completed
       actions:
         - type: email
           template: "winner_notification_de"
           recipients: ["{{user_email}}"]
         - type: webhook
           url: "https://meine-crm.com/webhook"
           headers: { "X-API-Key": "sk-..." }
   ```  
   Führen Sie `giveawaytracker orchestrator --config notifications.yaml` aus. Der Orchestrator startet einen asynchronen Worker-Prozess, der auf Redis-Streams lauscht.

3. **Dashboard-Konfiguration und Metrik-Export**  
   Verbinden Sie das Dashboard mit Ihrer PostgreSQL-Datenbank:  
   ```bash
   giveawaytracker dashboard --db-host localhost --db-port 5432 --db-name giveawaytracker --db-user admin
   ```  
   Konfigurieren Sie den Export-Zyklus (z. B. `--export-interval 300` für alle 5 Minuten). Das Dashboard generiert automatisch aggregierte Views (`materialized_views`) für schnelle Abfragen. Nutzen Sie `EXPLAIN ANALYZE` zur Optimierung der Abfrage-Pläne.

### Beste Praktiken

- **Echtzeit-Deduplizierung mit Bloom-Filtern**  
  Implementieren Sie einen Redis-basierten Bloom-Filter (Fehlerrate < 0,1 %) zur Erkennung von doppelten Teilnahmen. Setzen Sie den `key` als Kombination aus `user_id` und `campaign_id`. Dies reduziert die Datenbank-Last um 70 % und verhindert fehlerhafte Auszahlungen.

- **Horizontale Skalierung der Event-Queue**  
  Partitionieren Sie die Redis-Streams nach `campaign_id` (z. B. `stream:campaign:123`). Verwenden Sie Consumer Groups mit mindestens 3 Replikaten für Hochverfügbarkeit. Überwachen Sie die Queue-Länge mit `XLEN` und skalieren Sie Worker-Pods automatisch via Kubernetes HPA (Horizontal Pod Autoscaler) bei einer Queue-Länge > 1000.
