# giveawaytracker - GIV_011

## Übersicht

giveawaytracker ist ein hochperformantes, ereignisgesteuertes Monitoring- und Analyse-Framework zur Echtzeit-Erfassung, -Normalisierung und -Aggregation von Giveaway-Aktionen auf globalen Plattformen (Twitter, Discord, Telegram, Web3-Protokolle). Das System verarbeitet bis zu **12.000 Events pro Sekunde** und ermöglicht eine granular konfigurierbare, mehrdimensionale Auswertung von Gewinnspiel- und Airdrop-Kampagnen.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist eine modulare, quelloffene Backend-Architektur, die auf **Node.js (v20 LTS)** mit **PostgreSQL (15+)** und **Redis (7+)** aufbaut. Es löst das Problem der intransparenten und fragmentierten Giveaway-Landschaft, indem es **Proof-of-Participation (PoP)** Metadaten automatisiert extrahiert, auf Plausibilität prüft und in einer normalisierten SQL-Datenbank persistiert. Das Tool unterstützt sowohl **RESTful API** als auch **WebSocket-Streams** für Live-Updates.

### Kernfunktionen

- **Multi-Protokoll Event Normalizer (MPEN):** Ein asynchroner Parser, der rohe JSON-Payloads von Twitter API v2, Discord Gateway (Intents: `GUILD_MESSAGES`, `MESSAGE_CONTENT`) und Telegram Bot API in ein einheitliches Schema transformiert. Parameter: `raw_payload` (JSONB), `source_platform` (ENUM: 'twitter', 'discord', 'telegram', 'web3'), `parsed_meta` (JSONB). Das System validiert automatisch Hash-Tags gegen eine konfigurierbare Blacklist (regex-basiert, max. 500 Regeln) und entfernt Duplikate mittels eines **SHA256-Fingerprints** über `(user_id, platform, event_timestamp)`.

- **Stochastischer Gewinnwahrscheinlichkeitsrechner (SGP):** Ein serverseitiger Algorithmus, der auf Basis von **Poisson-Verteilung** und **historischen Teilnehmerzahlen** (letzte 90 Tage) eine dynamische Gewinnchance für jeden Eintrag kalkuliert. Parameter: `participant_count` (INT), `prize_pool_size` (INT), `historical_retention_rate` (FLOAT, 0.0–1.0). Die Berechnung erfolgt in **Sub-Millisekunden** (durchschnittlich 0.47 ms pro Event) und wird als `win_probability` (FLOAT, 6 Dezimalstellen) in der Tabelle `giveaway_entries` gespeichert.

- **Geo-distribuierter Audit-Trail (GDAT):** Jede Transaktion (Entry-Erstellung, Gewinnermittlung, Auszahlung) wird mit einem **digitalen Fingerabdruck** aus `user_agent`, `IP-Adresse` (maxmind GeoLite2), `timestamp` (ISO 8601, Millisekunden-Genauigkeit) und einem `nonce` (kryptografisch sicherer Zufallswert) in einer append-only Audit-Tabelle protokolliert. Parameter: `audit_type` (ENUM: 'entry', 'win', 'payout', 'rejection'), `entry_id` (UUID v4), `geo_data` (JSONB mit `city`, `country`, `latitude`, `longitude`). Die Tabelle ist **partitioniert** (monatlich) und unterstützt Abfragen mit `EXPLAIN ANALYZE` unter 50 ms für 10 Millionen Zeilen.

### So verwenden Sie es

1. **Konfiguration der Datenquellen:** Bearbeiten Sie die Datei `config/sources.yaml`. Definieren Sie für jede Plattform einen `endpoint`, `api_key` (verschlüsselt in Vault) und `rate_limit` (Anfragen pro Minute). Beispiel für Twitter: `source: twitter, endpoint: https://api.twitter.com/2/tweets/search/stream, rate_limit: 450`. Führen Sie anschließend `npm run source:validate` aus, um die Konnektivität und Token-Gültigkeit zu prüfen (HTTP-Status 200 erforderlich).

2. **Starten der Event-Processing-Pipeline:** Initialisieren Sie die Datenbank mit `npm run db:migrate` (erstellt Tabellen, Indizes und Partitionen). Starten Sie dann den Worker-Prozess: `node dist/workers/event-ingestor.js --concurrency=8 --batch-size=500`. Der Worker liest aus der Redis-Queue `giveaway:raw` (Blocking-Pop), normalisiert die Events und schreibt sie in die PostgreSQL-Tabelle `giveaway_entries`. Überwachen Sie die Latenz mit `npm run monitor:latency` – der Zielwert liegt unter **200 ms** vom Event-Eingang bis zur Persistenz.

3. **Ausführen einer Gewinnermittlung:** Nutzen Sie die CLI: `node dist/cli/draw.js --campaign-id=GIV-011-2024 --method=weighted_random --seed=$(date +%s)`. Das Skript führt eine **deterministische Zufallsziehung** durch (basierend auf SHA256(seed + entry_id)) und schreibt die Ergebnisse in die Tabelle `giveaway_winners`. Validieren Sie die Fairness mit `npm run audit:draw --draw-id=<uuid>` – das Tool generiert einen **Merkle-Baum** aller Teilnehmer und deren Gewinnwahrscheinlichkeiten, der als JSON exportiert werden kann.

### Beste Praktiken

- **Datenintegrität durch idempotente Verarbeitung:** Implementieren Sie immer einen **Unique Constraint** auf `(user_id, platform, giveaway_id)` in Ihrer Datenbank. Verwenden Sie `ON CONFLICT DO NOTHING` in Ihren INSERT-Statements, um Double-Entries zu vermeiden. Kombinieren Sie dies mit einem **Redis-Lock** (TTL: 10 Sekunden) pro User-Plattform-Paar, um Race-Conditions bei parallelen Webhook-Requests zu verhindern.

- **Skalierung der Audit-Tabelle:** Partitionieren Sie Ihre Audit-Logs nach **Monat** (`CREATE TABLE giveaway_audit_2024_12 PARTITION OF giveaway_audit FOR VALUES FROM ('2024-12-01') TO ('2025-01-01')`). Setzen Sie einen **automatischen Cleanup-Job** (Cron: täglich um 03:00 UTC), der Partitionen älter als 12 Monate in eine kostengünstige S3-kompatible Cold-Storage verschiebt (z. B. mittels `pg_partman` und `pg_dump`). Dies reduziert die aktive Tabellengröße und verbessert die `SELECT`-Performance auf Audit-Daten um bis zu 70%.
