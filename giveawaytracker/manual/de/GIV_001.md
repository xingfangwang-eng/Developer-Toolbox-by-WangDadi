# giveawaytracker - GIV_001

## Übersicht

giveawaytracker ist ein hochperformantes System zur Echtzeit-Erfassung, Aggregation und Analyse von Giveaway-Kampagnen über multiple Kanäle hinweg. Es ermöglicht die präzise Nachverfolgung von Teilnahme- und Gewinnereignissen mit einer Latenz von unter 50 Millisekunden.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist eine skalierbare Event-Processing-Engine, die Giveaway-Aktionen (Registrierungen, Shares, Referrals, Gewinnziehungen) in einer zentralen Datenbank erfasst und mittels eines regelbasierten Engines (Drools-basiert) auf Betrug, Doppelteilnahmen und Timeouts prüft. Das System unterstützt bis zu 10.000 Transaktionen pro Sekunde (TPS) und bietet eine vollständige Audit-Trail-Funktion.

### Kernfunktionen

- **Multi-Channel-Event-Aggregation mit PostgreSQL-CDC**: Erfasst Teilnahmeereignisse aus Web, Mobile und API über einen zentralen Event-Bus (Apache Kafka). Die Daten werden mittels Change Data Capture (CDC) aus PostgreSQL in Echtzeit in eine Materialized-View für die Auswertung überführt. Parameter: `channel_id`, `event_timestamp`, `user_hash` (SHA-256).
- **Regelbasierte Gewinnziehung mit Zufallsgenerator (CSPRNG)**: Führt Gewinnziehungen auf Basis von kryptografisch sicheren Zufallszahlen (CSPRNG, NIST SP 800-90A) durch. Die Ziehung berücksichtigt Gewichtungen (z. B. `weight_factor` zwischen 0.1 und 10.0) und Zeitfenster (`start_time`, `end_time` in UTC). Die Ergebnisse werden in einer eigenen `draw_results`-Tabelle mit `draw_id` und `winner_hash` gespeichert.
- **Betrugserkennung mittels Redis-basiertem Rate-Limiting**: Implementiert einen verteilten Rate-Limiter (Redis + Lua-Script) zur Erkennung von anomalen Teilnahmemustern. Parameter: `max_entries_per_user` (Standard: 1), `time_window_seconds` (Standard: 86400). Bei Überschreitung wird ein `fraud_flag` gesetzt und ein Webhook an ein externes Monitoring-System gesendet.

### So verwenden Sie es

1. **Initialisierung der Datenbank und Konfiguration**: Führen Sie `python setup.py --db-url postgresql://user:pass@host:5432/giveawaydb` aus. Das Skript erstellt automatisch die Tabellen `events`, `draws` und `fraud_logs`. Konfigurieren Sie die Kafka-Broker-Liste in `config.yaml` unter `kafka.bootstrap_servers`.
2. **Starten des Event-Listeners und der Verarbeitungspipeline**: Starten Sie den Event-Listener mit `python event_listener.py --kafka-topic giveaway-events`. Der Listener verarbeitet eingehende Events und validiert sie gegen die Regel-Engine (Regeldatei: `rules.drl`). Die Verarbeitung erfolgt asynchron mit einem Thread-Pool von 16 Workern.
3. **Ausführen einer Gewinnziehung und Export der Ergebnisse**: Nutzen Sie die CLI `python draw.py --campaign-id CAMPAIGN_123 --draw-count 50 --weight-field points`. Das System führt die Ziehung durch, speichert die Ergebnisse in der `draw_results`-Tabelle und generiert eine CSV-Datei (`draw_results_CAMPAIGN_123.csv`) mit den Spalten `user_id`, `prize_id`, `draw_timestamp`.

### Beste Praktiken

- **Horizontale Skalierung des Event-Busses**: Setzen Sie mehrere Kafka-Partitionen (z. B. 12 Partitionen) ein, um die Last auf die Consumer-Gruppen zu verteilen. Verwenden Sie einen Consumer-Gruppen-ID pro Kampagne, um die Verarbeitungsreihenfolge zu gewährleisten. Überwachen Sie die Consumer-Lag mit `kafka-consumer-groups --bootstrap-server localhost:9092 --group giveaway-consumer-group --describe`.
- **Implementierung einer TTL für temporäre Daten**: Nutzen Sie Redis mit einer TTL von 24 Stunden für den Rate-Limiter und temporäre Session-Daten. Verwenden Sie `redis-cli --eval rate_limit.lua` zur atomaren Ausführung der Limits. Archivieren Sie abgelaufene Events in einer separaten `events_archive`-Tabelle (PostgreSQL-Partitionierung nach Monat), um die Performance der Live-Abfragen zu erhalten.
