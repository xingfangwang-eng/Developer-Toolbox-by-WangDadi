# giveawaytracker - GIV_010

## Übersicht

giveawaytracker ist eine hochperformante, ereignisgesteuerte Plattform zur Echtzeit-Überwachung und -Analyse von Giveaway-Kampagnen über mehrere Kanäle hinweg. Das System verarbeitet bis zu 10.000 Transaktionen pro Sekunde und aggregiert Daten aus Discord, Twitter und Telegram in einer einzigen, normalisierten Datenstruktur.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist eine modulare Architektur, die auf Node.js und PostgreSQL mit TimescaleDB-Erweiterung aufbaut. Es ermöglicht die automatische Erkennung von Gewinnern, die Überprüfung von Teilnahmebedingungen und die Berechnung von Gewinnwahrscheinlichkeiten in Echtzeit. Das System unterstützt Multi-Tenancy und bietet eine granulare Zugriffskontrolle auf Organisationsebene.

### Kernfunktionen

- **Probabilistische Gewinnerermittlung mit PostgreSQL-Window-Funktionen**  
  Das System nutzt `ROW_NUMBER()` über partitionierte Teilnehmer-Tabellen, um unter Berücksichtigung von Gewichtungen (z. B. Retweet-Bonus `weight_factor = 2.5`) einen deterministischen, aber fair verteilten Gewinner zu ermitteln. Die Laufzeitkomplexität liegt bei O(n log n) pro Kampagne.

- **Echtzeit-Metrik-Streaming via WebSocket und Redis Pub/Sub**  
  Jede Teilnahme löst ein Event aus, das über einen Redis-Stream (`XADD giveaway:events * participant_id user_id campaign_id`) an die WebSocket-Server weitergeleitet wird. Die Latenz zwischen Teilnahme und Dashboard-Update beträgt weniger als 50 Millisekunden.

- **Automatische Betrugserkennung mittels Graph-basierter Analyse**  
  Ein Neo4j-Graph speichert Beziehungen zwischen Benutzern (z. B. gemeinsame IP-Adressen, ähnliche Wallet-Adressen). Ein Cypher-Query (`MATCH (u:User)-[:SHARES_IP]-(v:User) RETURN u, v`) identifiziert Sybil-Angriffe und schließt betrügerische Teilnehmer vor der Ziehung aus.

### So verwenden Sie es

1. **Kampagnen-Konfiguration über YAML-Manifest**  
   Erstellen Sie eine Datei `campaign_manifest.yaml` mit Parametern wie `max_participants: 5000`, `draw_time: "2025-06-15T18:00:00Z"` und `eligibility_rules: {min_account_age_days: 30, verified_email: true}`. Das System validiert das Manifest gegen ein JSON-Schema und initialisiert die Kampagne in der Datenbank.

2. **Teilnehmer-Import via Bulk-API-Endpunkt**  
   Senden Sie einen `POST /api/v1/participants/bulk`-Request mit einem JSON-Array von maximal 10.000 Teilnehmern pro Batch. Jeder Eintrag muss die Felder `user_id`, `channel` (z. B. `"discord"`), `entry_timestamp` (ISO 8601) und optionale Metadaten wie `referral_code` enthalten. Der Endpunkt antwortet mit einem `batch_id` und einem `status`-Objekt, das die Anzahl der erfolgreich importierten und abgelehnten Datensätze angibt.

3. **Gewinnerziehung und Benachrichtigung via idempotentem Workflow**  
   Führen Sie `GET /api/v1/campaigns/{campaign_id}/draw` aus. Das System sperrt die Kampagne (`SELECT ... FOR UPDATE`), führt die probabilistische Ziehung durch und sendet die Ergebnisse an eine konfigurierte Callback-URL (z. B. ein Discord-Webhook). Bei Netzwerkfehlern wird der Callback bis zu 3 Mal mit exponentiellem Backoff wiederholt.

### Beste Praktiken

- **Verwenden Sie partitionierte Tabellen für Kampagnen mit >100.000 Teilnehmern**  
  Teilen Sie die `participants`-Tabelle nach `campaign_id` und `entry_date` auf. Dies reduziert die Index-Scan-Zeit bei der Gewinnerziehung um bis zu 80 % und verhindert Lock-Contention bei parallelen Kampagnen.

- **Implementieren Sie eine Ratenbegrenzung pro IP und pro User-ID**  
  Setzen Sie einen Token-Bucket-Algorithmus mit einer Rate von 10 Requests pro Sekunde pro IP und 1 Request pro Sekunde pro User-ID auf dem API-Gateway. Dies verhindert DoS-Angriffe und stellt sicher, dass die Betrugserkennung nicht durch künstlich erzeugte Teilnahmen überlastet wird.
