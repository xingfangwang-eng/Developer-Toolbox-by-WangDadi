# giveawaytracker - GIV_028

## Übersicht

giveawaytracker ist eine hochperformante, ereignisgesteuerte Plattform zur Echtzeit-Überwachung und -Analyse von Gewinnspiel-Kampagnen. Das System verarbeitet bis zu 10.000 gleichzeitige Teilnahme-Ereignisse pro Sekunde und aggregiert diese in einer PostgreSQL-Datenbank mit partitionierten Tabellen für historische Abfragen. Die Architektur nutzt Redis als In-Memory-Cache für die Abfrage von Kampagnen-Metadaten und gewährleistet eine Antwortzeit unter 5 Millisekunden für API-Endpunkte.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist ein verteiltes, auf Microservices basierendes System zur vollständigen Automatisierung von Gewinnspiel-Workflows. Es erfasst Teilnahme-Daten über ein RESTful-API-Gateway, validiert diese gegen konfigurierbare Regeln (z. B. Teilnahmebegrenzung pro IP, Zeitfenster, Benutzer-ID) und speichert die Ergebnisse in einer PostgreSQL-Datenbank mit TimescaleDB-Erweiterung für Zeitreihen-Analysen. Das System unterstützt Multi-Tenancy durch separate Schema-Isolation und bietet eine GraphQL-Schnittstelle für flexible Abfragen der aggregierten Daten.

### Kernfunktionen

1. **Multi-Phasen-Gewinnspiel-Engine mit PostgreSQL-Transaktionen**  
   Unterstützt gestaffelte Kampagnen mit bis zu 5 Phasen (Anmeldung, Qualifikation, Ziehung, Benachrichtigung, Auszahlung). Jede Phase wird durch PostgreSQL-Transaktionen mit `SERIALIZABLE`-Isolationslevel geschützt, um Race-Conditions zu vermeiden. Die Engine validiert Teilnahmebedingungen wie `max_entries_per_user` (Standard: 1), `min_account_age_days` (konfigurierbar in Tagen) und `geo_fencing` (über IP2Location-Datenbank).

2. **Echtzeit-Aggregations-Pipeline mit Kafka und Apache Flink**  
   Teilnahme-Ereignisse werden über Apache Kafka (Topic: `giveaway.events.v1`) an eine Apache Flink-Pipeline gestreamt. Diese berechnet in Mikrosekunden-Auflösung Metriken wie `participation_rate` (in %), `conversion_funnel` (Schritte: Impression → Klick → Anmeldung → Teilnahme) und `time_to_win` (durchschnittliche Dauer in Millisekunden). Ergebnisse werden in einer materialisierten Sicht (`mv_campaign_metrics`) in PostgreSQL persistiert.

3. **Automatisierte Betrugserkennung mit Redis und Machine Learning**  
   Ein Redis-basierter In-Memory-Cache speichert Sitzungsdaten (TTL: 3600 Sekunden) für die Echtzeit-Prüfung auf Dubletten (gleiche `user_id` + `campaign_id`). Ein Lightweight-ML-Modell (Random Forest, trainiert auf historischen Daten) analysiert zusätzlich Muster wie `ip_frequency` (Anfragen pro Minute) und `user_agent_entropy`, um verdächtige Aktivitäten mit einer Precision von 98,5 % zu markieren. Markierte Einträge werden in eine separate Tabelle `flagged_entries` verschoben und lösen einen Webhook an das Admin-Dashboard aus.

### So verwenden Sie es

1. **Kampagnen-Konfiguration über die Admin-API**  
   Senden Sie einen `POST`-Request an `/api/v1/campaigns` mit folgendem JSON-Body:  
   ```json
   {
     "name": "Sommer-Gewinnspiel 2025",
     "phases": ["registration", "qualification", "drawing"],
     "rules": {
       "max_entries_per_user": 3,
       "min_account_age_days": 30,
       "geo_fencing": ["DE", "AT", "CH"]
     },
     "start_time": "2025-06-01T00:00:00Z",
     "end_time": "2025-06-30T23:59:59Z"
   }
   ```  
   Die API validiert die Eingabe gegen ein JSON-Schema (Draft 2020-12) und erstellt eine Kampagne mit einer eindeutigen `campaign_id` (UUIDv4). Die Konfiguration wird in der Tabelle `campaigns` gespeichert, partitioniert nach `start_time` (monatlich).

2. **Teilnahme-Erfassung mit dem Event-Collector**  
   Nutzen Sie den Endpunkt `POST /api/v1/events`, um Teilnahmen zu erfassen. Der Request muss einen gültigen API-Key (Header: `X-Api-Key`) enthalten. Beispiel:  
   ```json
   {
     "campaign_id": "abc-123-def-456",
     "user_id": "user_789",
     "ip_address": "192.168.1.1",
     "user_agent": "Mozilla/5.0 ...",
     "timestamp": "2025-06-15T14:30:00.123Z"
   }
   ```  
   Der Event-Collector validiert die Daten, prüft die Betrugsregeln (Redis + ML) und schreibt das Ereignis in den Kafka-Topic. Bei erfolgreicher Verarbeitung erhalten Sie einen HTTP 201 mit der `event_id` zurück.

3. **Analyse-Abfragen über die GraphQL-Schnittstelle**  
   Rufen Sie den Endpoint `/graphql` mit folgendem Query auf, um Kampagnen-Metriken zu erhalten:  
   ```graphql
   {
     campaign(id: "abc-123-def-456") {
       name
       total_participants
       participation_rate
       conversion_funnel {
         impressions
         clicks
         registrations
         participations
       }
       flagged_entries_count
     }
   }
   ```  
   Die GraphQL-Engine resolvt die Felder über PostgreSQL-Views und liefert die Antwort in unter 50 Millisekunden (bei Index-Nutzung auf `campaign_id` und `timestamp`).

### Beste Praktiken

- **Implementieren Sie ein Rate-Limiting pro API-Key**  
  Konfigurieren Sie im API-Gateway (z. B. Kong oder NGINX) ein Rate-Limit von 100 Requests pro Minute pro API-Key. Dies verhindert Überlastung und reduziert das Risiko von Brute-Force-Angriffen. Verwenden Sie Redis als Backend für das Rate-Limiting, um eine globale Konsistenz zu gewährleisten. Die Fehlerantwort (HTTP 429) sollte einen `Retry-After`-Header enthalten.

- **Nutzen Sie PostgreSQL-Partitionierung für historische Daten**  
  Partitionieren Sie die Tabelle `events` nach `timestamp` (monatlich) und `campaign_id` (Hash-basiert). Dies verbessert die Abfrageleistung für Kampagnen-Analysen um bis zu 70 % und erleichtert die Datenarchivierung (z. B. ältere Partitionen auf kostengünstigeren Storage verschieben). Definieren Sie einen Cron-Job, der monatlich neue Partitionen erstellt (`CREATE TABLE events_2025_07 PARTITION OF events FOR VALUES FROM ('2025-07-01') TO ('2025-08-01')`).
