# giveawaytracker - GIV_015

## Übersicht

giveawaytracker ist eine hochperformante, ereignisgesteuerte Tracking-Engine für die automatisierte Erfassung, Validierung und Analyse von Gewinnspiel-Aktionen. Das System verarbeitet parallel eingehende Teilnahmedaten aus Webhooks, API-Calls und manuellen CSV-Imports und normalisiert sie in Echtzeit auf eine einheitliche Datenstruktur. Ziel ist die Eliminierung von Dubletten, die lückenlose Audit-Trail-Dokumentation und die Bereitstellung von Live-Dashboards mit subsekundengenauer Latenz.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist eine modulare, auf PostgreSQL und Redis basierende Backend-Architektur, die speziell für die Verwaltung von Gewinnspiel-Events mit hohem Teilnahmevolumen (>100.000 Einträge pro Stunde) entwickelt wurde. Das System führt eine mehrstufige Deduplizierungslogik (Fingerprint-Hashing + Fuzzy-Matching) direkt auf Datenbankebene aus und stellt über eine GraphQL-Schnittstelle granulare Filter- und Aggregationsmöglichkeiten bereit. Jede Transaktion wird mit einem UUIDv7 und einem ISO-8601-Zeitstempel (Mikrosekunden-Genauigkeit) versehen und in einer append-only Tabelle persistiert.

### Kernfunktionen

- **Multi-Layer-Deduplizierung mit Bloom-Filter und Levenshtein-Distanz**  
  Eingehende Teilnahmen werden zunächst über einen Redis-basierten Bloom-Filter (Fehlerrate < 0,1 %) auf exakte Duplikate (E-Mail, Wallet-Adresse) geprüft. Anschließend führt ein PostgreSQL-Trigger eine Levenshtein-Distanz-Berechnung (Threshold: ≤ 3 Zeichen) auf Namens- und Adressfeldern durch, um typografische Variationen zu erkennen. Doppelte Einträge werden mit einem `duplicate_of`-Foreign-Key markiert, nicht gelöscht.

- **Echtzeit-Aggregations-Pipeline mit Kafka-Streams**  
  Jeder validierte Teilnahme-Eintrag wird als Avro-Nachricht an einen Kafka-Topic gesendet. Eine Kafka-Streams-Anwendung (State Store: RocksDB) berechnet sekündlich gleitende Fenster von 5, 15 und 60 Minuten für Metriken wie `entries_per_second`, `unique_participants` und `conversion_rate`. Diese Werte werden als JSON-Dokumente in eine PostgreSQL-Materialized View (`mv_realtime_metrics`) geschrieben und über Server-Sent-Events an das Frontend gepusht.

- **Automatisiertes Reward-Dispatching mit idempotenter Transaktionslogik**  
  Nach Abschluss eines Giveaways (definiert durch `end_time` im Event-Schema) startet ein PostgreSQL-pg_cron-Job die `dispatch_rewards()`-Funktion. Diese liest alle validierten, nicht-dublizierten Teilnahmen, wendet eine gewichtete Zufallsauswahl (Mersenne-Twister, seed = `event_id + round_start_timestamp`) an und erstellt für jeden Gewinner einen Eintrag in der Tabelle `reward_claims` mit Status `pending`. Jeder Dispatch-Vorgang wird durch eine `dispatch_id` (UUIDv7) und einen `CHECK`-Constraint (`status IN ('pending', 'claimed', 'expired')`) geschützt.

### So verwenden Sie es

1. **Initialisierung der Event-Konfiguration**  
   Führen Sie `INSERT INTO giveaway_events (event_name, start_time, end_time, max_winners, dedup_config) VALUES (...)` mit einem JSONB-Dokument in `dedup_config` aus, das Felder wie `"fuzzy_match_fields": ["full_name", "email"], "levenshtein_threshold": 3, "bloom_filter_capacity": 1000000` enthält. Verwenden Sie `SELECT * FROM validate_event_config('{...}')`, um die Konfiguration vorab zu prüfen.

2. **Bulk-Import von Teilnahmedaten**  
   Nutzen Sie den Endpunkt `POST /api/v1/entries/bulk` mit einem JSON-Array von Teilnahme-Objekten. Jedes Objekt muss `participant_id`, `source` (z. B. `"webhook"`, `"csv"`) und `metadata` (JSONB) enthalten. Das System gibt pro Eintrag ein Ergebnis-Objekt mit `entry_id`, `status` (`accepted`/`duplicate`/`invalid`) und `processing_time_ms` zurück.

3. **Monitoring und Export der Ergebnisse**  
   Rufen Sie `SELECT * FROM mv_realtime_metrics WHERE event_id = '...'` auf, um die aktuellen Live-Metriken abzurufen. Für einen vollständigen Audit-Export nutzen Sie `COPY (SELECT * FROM entries WHERE event_id = '...' AND status = 'accepted') TO '/tmp/export.csv' WITH (FORMAT CSV, HEADER, DELIMITER ';')`.

### Beste Praktiken

- **Praxis 1: Partitionierung nach Event-ID und Zeit**  
  Definieren Sie die Tabelle `entries` als partitionierte Tabelle mit `PARTITION BY RANGE (created_at)` und erstellen Sie monatliche Subpartitionen. Kombinieren Sie dies mit einem Index auf `(event_id, created_at)`, um Abfragen auf aktuelle Events zu beschleunigen. Verwenden Sie `pg_partman` für die automatische Verwaltung der Partitionen.

- **Praxis 2: Idempotenz-Key für externe Webhooks**  
  Implementieren Sie auf allen Write-Endpunkten einen `Idempotency-Key`-Header (UUIDv7). Speichern Sie diesen Key in einer separaten Tabelle `idempotency_keys` mit einem `UNIQUE`-Constraint und einer TTL von 24 Stunden (via Redis EXPIRE). So stellen Sie sicher, dass Netzwerk-Retransmissionen keine doppelten Einträge erzeugen.
