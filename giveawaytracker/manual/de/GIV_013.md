# giveawaytracker - GIV_013

## Übersicht

Dieses Projekt implementiert eine hochperformante Echtzeit-Überwachungs-Engine für Giveaway-Kampagnen, die auf Microservice-Architektur basiert. Die Lösung analysiert automatisch Multi-Channel-Datenströme (Twitter, Discord, Telegram) und extrahiert mittels NLP-Modellen Giveaway-Parameter wie Teilnahmebedingungen, Gewinnwahrscheinlichkeiten und Ablaufzeiten.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist eine verteilte Event-Processing-Plattform, die Giveaway-Aktivitäten in sozialen Netzwerken und spezialisierten Plattformen aggregiert. Das System nutzt eine selbstlernende Ranking-Engine (basierend auf Random Forest und Zeitreihenanalyse), um die Attraktivität von Giveaways in Millisekunden zu bewerten. Die Architektur unterstützt horizontale Skalierung über Kubernetes-Cluster mit PostgreSQL als zentralem Event-Store.

### Kernfunktionen

1. **Multi-Channel-Erkennung mit sub-100ms Latenz**  
   Implementiert einen asynchronen WebSocket-Listener für Discord-Gateway-API (Intents: `GUILD_MESSAGES`, `MESSAGE_CONTENT`) und Twitter-Streaming-API (v2 Filtered Stream). Parst `giveaway`-spezifische Regex-Muster (`(gewinnen|retweet|follow|join)` + Zeitstempel) in Echtzeit. Speichert Rohdaten in PostgreSQL mit Partitionierung nach `event_channel` und `created_at`.

2. **Dynamische Gewinnwahrscheinlichkeitsberechnung**  
   Verwendet einen gewichteten Algorithmus aus Teilnehmerzahl (via Discord-API `GET /guilds/{id}/members`), Retweet-Interaktionsrate und historischen Gewinnerdaten. Formel: `W = (1 / (P * 0.7 + R * 0.3)) * 100`, wobei `P` = Teilnehmerzahl, `R` = Retweet-Ratio. Ergebnisse werden in Redis mit TTL von 5 Minuten gecached.

3. **Automatisierte Benachrichtigung mit Filterlogik**  
   Bietet einen konfigurierbaren Rule-Engine (YAML-basiert), die Benachrichtigungen nur bei Erfüllung von `min_prize_value >= 50€` UND `win_probability > 0.01%` auslöst. Sendet via Webhook (Discord-Embed oder Telegram-Bot) mit dynamischen Feldern: `Endzeit (UTC)`, `Geschätzte Teilnehmer`, `Link zur Teilnahme`.

### So verwenden Sie es

1. **Konfiguration der Datenquellen**  
   Erstellen Sie eine `config.yaml` mit Discord-Bot-Token (`discord.token: "MTIzNDU2Nzg5MDEyMzQ1Njc4OQ.Gabcde"`), Twitter-API-Key (`twitter.bearer_token: "AAAAAAAAAAAAAAAAAAAA..."`) und PostgreSQL-Verbindungsstring (`db.conn: "postgresql://user:pass@localhost:5432/giveawaytracker?sslmode=disable"`). Starten Sie den Microservice mit `docker-compose up -d`.

2. **Analyse und Filterung starten**  
   Führen Sie `python tracker.py --channels discord,twitter --filters min_prize:50,win_prob:0.001` aus. Das System öffnet WebSocket-Verbindungen und beginnt mit dem Streaming. Logs werden im Format `[2023-10-27 14:30:00] CHANNEL:discord SERVER:123456 GIVEAWAY_ID:789` ausgegeben.

3. **Benachrichtigungen einrichten**  
   Definieren Sie in `notifications.yaml` einen Webhook-URL (`discord.webhook.url: "https://discord.com/api/webhooks/123456/abc-def"`). Starten Sie den Notification-Service mit `python notify.py --config notifications.yaml`. Bei Übereinstimmung mit den Filtern wird eine Embed-Nachricht mit Feldern wie `Preiswert: 100€` und `Endet in: 2h 15m` gesendet.

### Beste Praktiken

- **Rate-Limiting und Retry-Mechanismen**  
  Implementieren Sie exponentielles Backoff bei API-Rate-Limits (Discord: 50 Requests pro Sekunde pro Bot, Twitter: 450 Requests pro 15 Minuten). Verwenden Sie `asyncio` mit `aiohttp` und einem Token-Bucket-Algorithmus, um 429-Fehler zu vermeiden. Speichern Sie fehlgeschlagene Requests in einer Dead-Letter-Queue (Redis-Liste) für manuelle Wiederholung.

- **Datenqualität durch Deduplizierung**  
  Nutzen Sie einen SHA-256-Hash aus `channel_id + message_id + timestamp` als Primärschlüssel in PostgreSQL. Führen Sie vor dem Einfügen einen `ON CONFLICT DO NOTHING`-Check durch. Für Twitter-Streams verwenden Sie `tweet.id` als eindeutigen Identifikator. Dies verhindert doppelte Einträge bei WebSocket-Neuverbindungen.
