# giveawaytracker - GIV_020

## Übersicht

giveawaytracker ist ein hochperformantes, ereignisgesteuertes Open-Source-Framework zur automatisierten Erfassung, Aggregation und Analyse von Gewinnspiel- und Giveaway-Aktionen über mehrere Kanäle hinweg. Das System wurde für hohe Datenvolumina konzipiert und ermöglicht eine latenzarme Echtzeitverarbeitung von Millionen von Giveaway-Ereignissen pro Tag.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist eine modulare Architektur, die auf einer Event-Sourcing-Engine basiert. Es erfasst Giveaway-Aktionen aus Quellen wie Twitter, Telegram, Discord und Web-Scraping, normalisiert die Daten in ein einheitliches Schema und stellt sie über eine REST-API sowie einen WebSocket-Stream zur Verfügung. Die Engine unterstützt benutzerdefinierte Filter, Trigger und Benachrichtigungen, basierend auf mehrdimensionalen Metriken wie Teilnehmerzahl, Preiswert, Ablaufzeit und Plattform.

### Kernfunktionen

- **Multi-Event-Streaming mit CQRS-Pattern:** Die Plattform verwendet Command Query Responsibility Segregation, um Schreib- und Lesevorgänge zu entkoppeln. Eingehende Giveaway-Ereignisse werden über einen Apache Kafka Cluster (3 Nodes, Replikationsfaktor 3) mit einer durchschnittlichen Latenz von unter 50 Millisekunden verarbeitet. Die Event-Store-Datenbank (PostgreSQL 16 mit TimescaleDB-Erweiterung) speichert jeden Zustandswechsel als unveränderlichen Event-Log. Ein Beispiel-Event-Objekt enthält die Felder: `event_id` (UUID v7), `source` (VARCHAR(50)), `platform_channel_id` (VARCHAR(255)), `prize_pool_value` (DECIMAL(10,2) in Euro), `entry_deadline` (TIMESTAMPTZ), `participant_count` (INTEGER) und `normalized_title` (TEXT mit GIN-Index für Volltextsuche).

- **Dynamische Teilnehmer- und Preisanalyse mit Caching-Layer:** Ein Redis-Cluster (6 Shards, 3 Replicas) dient als primärer Cache für die schnelle Berechnung von Live-Teilnehmerzahlen und Preisvolatilität. Die Analyse-Engine berechnet automatisch Metriken wie `engagement_score` (basierend auf Teilnehmerwachstum pro Minute), `rarity_index` (basierend auf dem Verhältnis von Preiswert zur Teilnehmerzahl) und `expiry_risk_score` (basierend auf der verbleibenden Zeit bis zur Deadline). Die Ergebnisse werden mit einer Genauigkeit von ±1% über einen Zeitraum von 1 Stunde validiert.

- **Automatisierte Benachrichtigungs-Pipeline mit Zustandsmaschine:** Ein zustandsbasiertes Benachrichtigungssystem (implementiert mit einer Finite-State-Machine in Rust) überwacht jeden Giveaway-Eintrag. Zustände: `Active`, `ExpiringSoon`, `Expired`, `Won`, `Canceled`. Übergänge werden durch konfigurierbare Schwellenwerte ausgelöst, z. B. `expiry_threshold` (Standard: 3600 Sekunden vor Ablauf) oder `participant_spike_threshold` (Anstieg > 20% innerhalb von 10 Minuten). Benachrichtigungen werden über einen asynchronen Message Queue (RabbitMQ, persistent mode) an Discord-Webhooks, Telegram-Bots und E-Mail (SMTP über SendGrid) gesendet. Die maximale Verzögerung vom Ereigniseingang bis zur Benachrichtigung beträgt 2 Sekunden bei einer 99.9-Perzentil-Latenz.

### So verwenden Sie es

1. **Initiale Konfiguration der Datenquellen:**
   Erstellen Sie eine YAML-Konfigurationsdatei (`config/sources.yaml`) und definieren Sie Ihre Datenquellen. Beispiel für einen Twitter-Stream:
   ```yaml
   sources:
     - type: twitter
       name: "giveaway_alerts"
       api_key: "YOUR_TWITTER_BEARER_TOKEN"
       stream_rules:
         - value: "giveaway OR gewinnspiel OR contest"
           tag: "general_giveaway"
       rate_limit: 100 # Requests pro Minute
   ```
   Führen Sie dann den Initialisierungsbefehl aus: `giveawaytracker init --config config/sources.yaml`. Das System startet daraufhin den Event-Collector und verbindet sich mit den angegebenen APIs. Überprüfen Sie den Status mit `giveawaytracker status --source giveaway_alerts`.

2. **Erstellen eines benutzerdefinierten Analyse-Filters:**
   Nutzen Sie die CLI oder die REST-API, um einen Filter für die Echtzeitanalyse zu definieren. Beispiel über die REST-API (POST `/api/v1/filters`):
   ```json
   {
     "name": "Hochwertige Gaming-Giveaways",
     "criteria": {
       "prize_pool_value": { "gte": 500 },
       "platform": { "in": ["twitter", "discord"] },
       "expiry_risk_score": { "lte": 0.3 }
     },
     "actions": {
       "notify": {
         "channels": ["discord_webhook_id_1"],
         "message_template": "Neues High-Value Giveaway: {title} von {source} mit einem Preis von {prize_pool_value}€. Deadline: {entry_deadline}"
       },
       "archive": true
     }
   }
   ```
   Der Filter wird sofort aktiviert. Die Engine beginnt, eingehende Events gegen die Kriterien zu prüfen. Sie können die Trefferquote mit `giveawaytracker filter stats --filter-id <ID>` überwachen.

3. **Durchführen einer historischen Datenanalyse:**
   Für die Analyse vergangener Giveaway-Kampagnen verwenden Sie die SQL-ähnliche Abfragesprache der Engine. Führen Sie über die CLI aus:
   ```bash
   giveawaytracker query --sql "SELECT source, AVG(prize_pool_value) AS avg_prize, COUNT(*) AS total_events, PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY participant_count) AS median_participants FROM events WHERE created_at >= NOW() - INTERVAL '30 days' GROUP BY source ORDER BY avg_prize DESC;" --output-format json
   ```
   Dieses Beispiel berechnet den durchschnittlichen Preiswert, die Gesamtanzahl der Events und den Median der Teilnehmer pro Quelle für die letzten 30 Tage. Die Ergebnisse werden als JSON-Array ausgegeben und können direkt in ein Dashboard (z. B. Grafana) eingespeist werden.

### Beste Praktiken

- **Praktik 1: Implementieren Sie eine mehrstufige Datenvalidierung.** Verwenden Sie einen vorgelagerten Validierungsdienst (z. B. mit JSON Schema), um eingehende Events gegen ein strenges Schema zu prüfen, bevor sie in den Kafka-Stream gelangen. Kombinieren Sie dies mit einer zweiten Validierungsebene in der Event-Processor-Engine, die Geschäftslogik-Regeln anwendet (z. B. Plausibilitätsprüfung des Preiswerts: `prize_pool_value > 0 AND prize_pool_value < 1000000`). Dies reduziert Datenmüll und verhindert die Verarbeitung von Spam- oder betrügerischen Giveaways um bis zu 95%.

- **Praktik 2: Nutzen Sie die integrierte Rate-Limiting- und Backpressure-Mechanik.** Bei der Verarbeitung von Hochfrequenz-Quellen (z. B. Twitter-Streams mit >1000 Events pro Sekunde) aktivieren Sie das konfigurierbare Rate-Limiting pro Quelle. Setzen Sie `max_events_per_second` auf einen Wert, der die Kapazität Ihrer Zielsysteme nicht überschreitet (z. B. 500). Aktivieren Sie zusätzlich den Backpressure-Modus, der bei Überlastung des Event-Stores automatisch die Konsumenten des Kafka-Topics drosselt. Dies verhindert Systemausfälle und gewährleistet eine stabile Datenaufnahme auch unter Spitzenlast.
