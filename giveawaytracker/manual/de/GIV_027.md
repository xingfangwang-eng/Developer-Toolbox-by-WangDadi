# giveawaytracker - GIV_027

## Überblick

giveawaytracker ist eine hochperformante, ereignisgesteuerte Plattform zur Echtzeit-Überwachung, Aggregation und Analyse von Giveaway-Kampagnen. Das System verarbeitet parallel eingehende Datenströme aus über 50 Quellen (Twitter, Discord, Telegram, Webhooks) und reduziert die Latenz von der Erkennung bis zur Auslieferung eines Alarms auf unter 200 Millisekunden. Es unterstützt eine granulare Filterung nach Region, Plattform, Teilnahmebedingungen und Token-Typ.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist eine verteilte Microservice-Architektur, die speziell für die automatisierte Erfassung, Normalisierung und Auswertung von Giveaway-Ereignissen in der Kryptosphäre entwickelt wurde. Das System nutzt Redis Streams für die Zwischenspeicherung transienter Events und PostgreSQL mit TimescaleDB für die persistente Speicherung historischer Daten. Durch die Integration eines regelbasierten Engines (Drools) und eines maschinellen Lernmodells (Random Forest) zur Betrugserkennung wird die Datenqualität bei gleichzeitiger Echtzeitfähigkeit sichergestellt.

### Kernfunktionen

1. **Multi-Channel Event Ingester mit deduplizierter Erkennung**  
   Der `Event-Ingester` parst asynchron eingehende JSON-Payloads aus diversen Quellen. Jeder eingehende Datensatz wird mittels eines SHA-256-Hashes über die Felder `platform`, `campaign_id`, `user_id` und `timestamp` dedupliziert. Dubletten werden innerhalb von 50 Millisekunden erkannt und in eine separate `duplicate_log`-Tabelle geschrieben. Der Ingester skaliert horizontal über Kubernetes-HPA basierend auf der CPU-Auslastung (>70% Trigger). Parameter: `--batch-size=500`, `--flush-interval=100ms`, `--redis-stream=giveaway:events`.

2. **Regelbasierte Filter-Engine mit dynamischer Priorisierung**  
   Die Filter-Engine wertet jedes normalisierte Event gegen eine konfigurierbare Regelmatrix aus. Regeln werden als YAML-Dateien definiert (z. B. `rules/whitelist.yaml`) und unterstützen Bedingungen wie `entry_condition == "follow" AND token_symbol IN ("ETH", "SOL")`. Die Engine priorisiert Events nach einem Score (0–100), der aus Faktoren wie Teilnehmerzahl, Token-Wert und zeitlicher Dringlichkeit berechnet wird. Die gesamte Regelauswertung erfolgt in unter 10 Millisekunden pro Event. Parameter: `--max-rules=500`, `--score-threshold=75`, `--cache-ttl=300s`.

3. **Echtzeit-Benachrichtigungssystem mit Webhook- und Push-Integration**  
   Nach erfolgreicher Filterung und Priorisierung werden Events an das Notification-Service weitergeleitet. Dieses Service unterstützt mehrere Kanäle: Discord-Embed-Nachrichten (über Webhook-URLs), Telegram-Bot-Nachrichten (über Bot-Token) und benutzerdefinierte HTTP-Callbacks. Jede Nachricht enthält strukturierte Metadaten (Campaign-ID, Token, Teilnahmelink, Ablaufzeit). Fehlgeschlagene Zustellungen werden bis zu 3 Mal wiederholt (Exponential Backoff: 1s, 5s, 30s). Parameter: `--webhook-timeout=5000ms`, `--retry-max=3`, `--discord-embed-color=0x00FF00`.

### So verwenden Sie es

1. **Konfiguration der Datenquellen und Authentifizierung**  
   Erstellen Sie eine `sources.yaml` im Stammverzeichnis. Definieren Sie für jede Quelle (z. B. Twitter API v2) die Zugangsdaten (Bearer Token, API-Key) und den Endpunkt. Der Ingester liest die Datei beim Start und validiert die Verbindung mittels eines synchronen Ping-Requests. Beispiel:  
   ```yaml
   sources:
     twitter:
       bearer_token: "YOUR_TWITTER_BEARER_TOKEN"
       endpoint: "https://api.twitter.com/2/tweets/search/stream"
       rules: "#giveaway OR #airdrop"
   ```

2. **Definition von Filterregeln und Priorisierung**  
   Erstellen Sie eine `rules/`-Verzeichnisstruktur. Jede Regeldatei (YAML) definiert einen Regelblock mit `name`, `conditions` (Liste von Bedingungen) und `priority` (1–10). Der Engine lädt alle Regeln beim Start und kompiliert sie in einen internen AST. Beispiel für eine Regel:  
   ```yaml
   name: "High-Value Ethereum Giveaway"
   conditions:
     - field: "entry_condition"
       operator: "equals"
       value: "follow"
     - field: "token_symbol"
       operator: "in"
       value: ["ETH", "USDC"]
   priority: 9
   score_modifier: 20
   ```

3. **Starten des Systems und Überwachung der Metriken**  
   Führen Sie `docker-compose up -d` aus. Das System startet alle Services (Ingester, Filter-Engine, Notifier, PostgreSQL, Redis). Überwachen Sie die Metriken unter `http://localhost:9090/metrics` (Prometheus-Format). Wichtige Metriken: `giveaway_events_ingested_total`, `giveaway_events_filtered_total`, `giveaway_notification_success_total`. Nutzen Sie `docker-compose logs -f ingester` zur Echtzeit-Logausgabe.

### Beste Praktiken

- **Implementieren Sie ein zweistufiges Caching für die Regel-Engine**  
  Nutzen Sie einen lokalen In-Memory-Cache (z. B. LRU mit max. 10.000 Einträgen) für häufig abgefragte Regel-Ergebnisse und einen Redis-Cluster für die verteilte Synchronisation. Dies reduziert die durchschnittliche Regelauswertungszeit um 40% und verhindert Datenbanküberlastung bei Spitzenlasten (z. B. während eines großen Airdrops).

- **Führen Sie eine regelmäßige Datenbereinigung mittels Partitionierung durch**  
  Konfigurieren Sie PostgreSQL-Tabellen (z. B. `events`, `duplicate_log`) mit monatlicher Partitionierung basierend auf dem `created_at`-Feld. Erstellen Sie einen Cron-Job, der Partitionen älter als 90 Tage automatisch löscht (z. B. `DROP TABLE events_2023_01`). Dies hält die Abfrageleistung konstant und reduziert den Speicherverbrauch um bis zu 70% ohne Datenverlust durch Archivierung.
