# giveawaytracker - GIV_026

## Übersicht

giveawaytracker ist ein hochperformantes, event-getriebenes Tracking-System zur Echtzeit-Erfassung und -Analyse von Giveaway-Kampagnen. Es verarbeitet parallel eingehende Teilnahmesignale, validiert diese gegen benutzerdefinierte Regelsets und aggregiert Metriken wie Teilnahmerate, Conversion-Funnel und Reward-Distribution. Die Architektur basiert auf einer asynchronen Message-Queue (NATS) und einer PostgreSQL-Datenbank mit optimierten Indizes für sub-100ms Query-Latenzen.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist ein modulares Framework zur Verwaltung und Überwachung von Giveaway-Events auf Unternehmensebene. Es ermöglicht die Definition komplexer Teilnahmebedingungen (z.B. Multi-Channel-Requirement, Time-Window-Constraints), die Echtzeit-Deduplizierung von Teilnehmern mittels Bloom-Filtern und die automatische Auslosung mit nachweisbarer Fairness (kryptografisches Commit-Reveal-Schema). Die Plattform skaliert horizontal auf 10.000+ Events pro Sekunde und bietet eine RESTful API mit OpenAPI 3.1-Spezifikation.

### Kernfunktionen

- **Multi-Chain-Giveaway-Engine**: Unterstützt parallele Kampagnenlogiken auf Basis von benutzerdefinierten Bedingungsbäumen (z.B. `{channel: "twitter", min_followers: 100} AND {channel: "discord", role: "member"}`). Jede Bedingung wird als unabhängiger Node in einer Directed-Acyclic-Graph (DAG) ausgeführt, mit einer maximalen Verarbeitungstiefe von 7 Ebenen. Die Engine nutzt einen optimierten Rule-Compiler, der Bedingungen in PostgreSQL-Trigger-Funktionen übersetzt (Latenz < 50ms pro Node).

- **Echtzeit-Deduplizierung & Anti-Bot-Filter**: Implementiert einen mehrstufigen Filterstack: (1) Redis-basierter Bloom-Filter mit 0.1% False-Positive-Rate für IP- und Wallet-Adressen, (2) heuristische Bot-Erkennung via Machine-Learning-Modell (LightGBM, trainiert auf 2M+ historischen Teilnahmen), (3) Time-Window-Constraint (max. 3 Teilnahmen pro 24h pro IP). Der Filterstack läuft asynchron in einer Worker-Farm (Go-Routinen, 8 Kerne), mit einer durchschnittlichen Verarbeitungszeit von 12ms pro Teilnahme.

- **Kryptografische Auslosung & Audit-Trail**: Jede Auslosung verwendet ein Commit-Reveal-Schema: (1) Admin generiert einen 256-Bit-Zufallswert (Commit), (2) Teilnehmer-IDs werden gehasht (SHA-256) und mit dem Commit kombiniert, (3) Reveal-Phase mit öffentlichem Seed (z.B. Bitcoin-Blockhash). Die Gewinnerliste wird als Merkle-Tree in PostgreSQL gespeichert (inkl. Proof-of-Inclusion). Der gesamte Vorgang ist in < 2 Sekunden für 10.000 Teilnehmer abgeschlossen.

### So verwenden Sie es

1. **Kampagnen-Konfiguration via API**  
   Senden Sie einen POST-Request an `/api/v1/campaigns` mit einem JSON-Body, der die Kampagnenparameter definiert: `{ "name": "Q4_Giveaway", "channels": ["twitter", "discord"], "rules": { "twitter_follow": {"min_followers": 100}, "discord_join": {"guild_id": "123456"} }, "reward_pool": {"token": "USDC", "amount": 5000}, "max_participants": 5000, "start_time": "2024-12-01T00:00:00Z", "end_time": "2024-12-31T23:59:59Z" }`. Die API validiert die Regeln gegen ein Schema (JSON Schema Draft 7) und erstellt eine Kampagnen-ID (UUIDv4) mit einem initialen Status `draft`.

2. **Teilnahme-Erfassung & Validierung**  
   Implementieren Sie einen Webhook-Endpunkt (POST `/api/v1/participations`) der die Teilnahmesignale empfängt: `{ "campaign_id": "abc-123", "user_id": "0xUserWallet", "channel": "twitter", "proof": {"tweet_id": "123456789"} }`. Der Endpunkt führt synchron die Bloom-Filter-Prüfung durch (Redis, < 1ms) und asynchron die vollständige Regelvalidierung (Worker-Queue). Bei Erfolg wird eine Teilnahme-ID (UUIDv4) zurückgegeben, bei Fehlschlag ein 422-Status mit Fehlercode (z.B. `DUPLICATE_USER`, `RULE_VIOLATION`).

3. **Auslosung & Ergebnis-Abruf**  
   Rufen Sie den Auslosungs-Endpoint auf (POST `/api/v1/campaigns/{id}/draw`) mit einem Body: `{ "seed": "0xBitcoinBlockHash" }`. Der Server generiert den Commit, führt die Auslosung aus und speichert die Gewinnerliste. Der Status wechselt zu `completed`. Abruf der Ergebnisse via GET `/api/v1/campaigns/{id}/winners` – die Antwort enthält eine Liste von Gewinner-IDs und einen Merkle-Proof für jede. Die Proofs können clientseitig verifiziert werden (npm-Paket `giveawaytracker-verify`).

### Beste Praktiken

- **Praxis 1: Staging-Umgebung mit synthetischen Daten**  
  Nutzen Sie die integrierte `sandbox`-Mode-Flag (`GIVEAWAY_SANDBOX=true`), die alle Transaktionen in einer separaten PostgreSQL-Datenbank (Suffix `_sandbox`) speichert und keine echten Token ausgibt. Generieren Sie synthetische Teilnahmesignale mit dem CLI-Tool `gt-generator` (z.B. `gt-generator --campaign-id abc-123 --count 1000 --rate 50/s`). Validieren Sie die Auslosungsergebnisse manuell mit dem `gt-verify`-Tool, das die Merkle-Proofs gegen den öffentlichen Seed prüft.

- **Praxis 2: Monitoring & Alerting auf Basis von Metriken**  
  Konfigurieren Sie Prometheus-Endpunkte (standardmäßig unter `/metrics`), die folgende Metriken exponieren: `giveaway_participation_latency_ms` (Histogram, Buckets: [5, 10, 25, 50, 100]), `giveaway_rule_evaluation_errors_total` (Counter, Labels: rule_name), `giveaway_draw_duration_seconds` (Gauge). Setzen Sie Alarme in Grafana auf: (1) Latenz > 100ms für mehr als 5% der Requests, (2) Fehlerrate > 1% über 5 Minuten, (3) Auslosungsdauer > 5 Sekunden. Nutzen Sie die integrierte Slack-Integration via Webhook für Benachrichtigungen.
