# giveawaytracker - GIV_005

## Übersicht

giveawaytracker ist eine hochperformante, ereignisgesteuerte Plattform zur Echtzeit-Überwachung, Verwaltung und Analyse von Gewinnspiel-Aktionen. Das System verarbeitet parallel mehrere tausend Giveaway-Events pro Sekunde, aggregiert Teilnahmedaten aus über 20 Quellen und bietet eine granulare Betrugserkennung auf Basis von Zeitstempel- und IP-Fingerprinting.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist eine verteilte, microservices-basierte Architektur zur vollautomatischen Erfassung und Auswertung von Gewinnspiel-Interaktionen. Das Tool nutzt eine Kombination aus Redis-Streams für die Event-Queue, PostgreSQL mit TimescaleDB für die Zeitreihenanalyse und einer RESTful API mit OAuth 2.0-Authentifizierung. Es skaliert horizontal auf bis zu 10 Nodes und verarbeitet Latenzen im einstelligen Millisekundenbereich.

### Kernfunktionen

- **Multi-Channel Event Aggregation & Echtzeit-Deduplizierung**: Das System aggregiert Giveaway-Eingaben aus Discord, Twitter, Telegram, E-Mail und Web-Forms. Jeder eingehende Payload (`user_id`, `channel_id`, `timestamp_ms`, `fingerprint_hash`) wird in einer Redis-Stream-Pipeline mit einer maximalen Verzögerung von **50 ms** verarbeitet. Die Deduplizierung erfolgt über einen Bloom-Filter mit einer False-Positive-Rate von unter 0,1 % und einem `window_size` von 60 Sekunden.

- **Dynamische Gewinnspiel-Logik Engine (Rule Engine)**: Die Engine unterstützt komplexe Bedingungsbäume mit Parametern wie `min_participants`, `max_entries_per_user`, `allowed_regions`, `time_window_start` (Unix-Timestamp) und `weighted_random_selection`. Gewinner werden mittels eines kryptografisch sicheren Zufallsgenerators (CSPRNG) ermittelt. Die Engine loggt jede Entscheidung in eine PostgreSQL-Tabelle `decision_log` mit den Spalten `rule_id`, `user_id`, `decision`, `execution_time_ms`.

- **Automatisierte Benachrichtigungs-Pipeline (Multi-Channel Dispatch)**: Nach der Gewinnerermittlung wird ein asynchroner Job über RabbitMQ gestartet. Der Job sendet personalisierte Nachrichten via Discord-Webhook (Rate-Limit: 30 Nachrichten pro Sekunde), Telegram-Bot-API und E-Mail (SMTP mit TLS 1.3). Jeder Versand wird in der Tabelle `notification_queue` mit dem Status `pending`, `sent` oder `failed` und einem `retry_count` von maximal 3 protokolliert.

### So verwenden Sie es

1. **Integration der Event-API**: Senden Sie einen POST-Request an den Endpunkt `/api/v1/events` mit dem JSON-Body `{"user_id": "UUID", "channel": "discord", "timestamp_ms": 1712345678000, "fingerprint_hash": "sha256hex"}`. Die API authentifiziert via `Authorization: Bearer <JWT-Token>`. Bei Erfolg erhalten Sie einen `201 Created` mit der `event_id` und einer Bestätigungslatenz von unter **10 ms**.

2. **Konfiguration einer Gewinnspiel-Regel**: Erstellen Sie eine neue Regel via `POST /api/v1/rules`. Übergeben Sie Parameter wie `{"rule_name": "Summer Sale Giveaway", "min_participants": 100, "max_entries_per_user": 1, "time_window_start": 1712345678, "time_window_end": 1712432078, "selection_method": "weighted_random", "weight_field": "loyalty_points"}`. Das System validiert die Regel und speichert sie in der `rules`-Tabelle. Die Aktivierung erfolgt sofort nach dem Speichern.

3. **Auslösen der Gewinner-Ermittlung und Benachrichtigung**: Senden Sie einen `POST /api/v1/draw` mit der `rule_id`. Das System durchläuft die `decision_log`, filtert gültige Teilnehmer und führt die Ziehung aus. Die Ergebnisse werden in der `draw_results`-Tabelle gespeichert. Parallel startet die Benachrichtigungs-Pipeline. Sie können den Status über `GET /api/v1/draw/{draw_id}/status` abfragen, der die Felder `total_participants`, `winners_count`, `notifications_sent` und `execution_time_ms` enthält.

### Beste Praktiken

- **Implementieren Sie einen exponentiellen Backoff für API-Rate-Limits**: Da externe APIs wie Discord oder Telegram eigene Rate-Limits haben, sollte Ihre Benachrichtigungs-Pipeline einen exponentiellen Backoff-Mechanismus implementieren. Verwenden Sie einen `retry_delay` von initial 1 Sekunde, der sich bei jedem Fehlschlag verdoppelt (maximal 5 Versuche). Loggen Sie jeden Fehlschlag mit dem HTTP-Statuscode und der `retry_after`-Header-Information.

- **Führen Sie Lasttests mit simulierten Teilnehmern durch**: Nutzen Sie ein Tool wie `k6` oder `Locust`, um Ihre Event-API mit 10.000 gleichzeitigen Verbindungen zu testen. Überwachen Sie die `event_queue`-Länge in Redis und die CPU-Auslastung der Worker-Nodes. Stellen Sie sicher, dass die `timestamp_ms`-Differenz zwischen Event-Eingang und Verarbeitung nie über **200 ms** steigt. Skalieren Sie horizontal, sobald die Queue-Länge 1.000 überschreitet.
