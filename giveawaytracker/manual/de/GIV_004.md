# giveawaytracker - GIV_004

## Übersicht

Dieses Projekt realisiert eine hochperformante, ereignisgesteuerte Gewinnspiel-Tracking-Plattform. Die Architektur setzt auf asynchrone Verarbeitung von Millionen von Teilnahmeereignissen (bis zu 50.000 Events pro Sekunde) und ermöglicht die Echtzeit-Aggregation von Gewinnspieldaten über mehrere Kanäle hinweg. Das System ist für den Einsatz in hochverfügbaren, containerisierten Umgebungen optimiert und nutzt eine Kombination aus In-Memory-Datenhaltung (Redis) und persistentem Logging (PostgreSQL mit TimescaleDB-Erweiterung).

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist ein verteiltes, auf Mikrodiensten basierendes Backend-System zur lückenlosen Erfassung, Validierung und Analyse von Gewinnspiel-Teilnahmen. Es löst das Problem der Dateninkonsistenz bei parallelen Teilnahmeversuchen und bietet eine granulare Betrugserkennung auf Basis von Zeitstempeln (Nanosekunden-Genauigkeit) und Nutzerverhaltensmustern. Das System ist mandantenfähig und unterstützt dynamische Gewinnspiel-Konfigurationen ohne Neustart der Dienste.

### Kernfunktionen

- **Echtzeit-Deduplizierung und Betrugserkennung mittels Bloom-Filter und HyperLogLog:** Jede eingehende Teilnahme wird innerhalb von < 5 ms gegen einen verteilten Bloom-Filter in Redis geprüft (Parameter: `bloom_filter.capacity=10000000`, `bloom_filter.error_rate=0.0001`). Parallel dazu wird ein HyperLogLog-Datensatz verwendet, um die ungefähre Anzahl eindeutiger Teilnehmer pro Kampagne zu schätzen, ohne die exakten Nutzer-IDs speichern zu müssen. Bei Überschreitung eines konfigurierbaren Schwellenwerts (z.B. `fraud_rate_limit=100 Teilnahmen/Minute/IP`) wird die Teilnahme automatisch abgewiesen und in eine Dead-Letter-Queue (DLQ) auf Kafka verschoben.

- **Multivariate Gewinnspiel-Logik-Engine mit AST-Interpretation:** Die Gewinnauswahl erfolgt nicht durch einfache Zufallsziehung, sondern durch eine konfigurierbare Engine, die einen abstrakten Syntaxbaum (AST) interpretiert. Parameter wie `selection_algorithm=weighted_random`, `weight_field=user_loyalty_score` und `time_decay_factor=0.95` ermöglichen komplexe Auswahlstrategien. Die Engine unterstützt Bedingungen wie `if (user.region == "DE" AND user.registration_date < "2023-01-01") then increase_weight(1.5)`. Die Berechnung erfolgt asynchron und skaliert horizontal über mehrere Worker-Nodes.

- **Temporale Datenaggregation mit TimescaleDB für Historie und Trendanalyse:** Alle validierten Teilnahmeereignisse werden in einer PostgreSQL-Datenbank mit TimescaleDB-Erweiterung persistiert. Die Hypertabelle `participations` ist nach dem Feld `created_at` partitioniert (Chunk-Intervall: 1 Tag). Abfragen wie „Anzahl der Teilnahmen pro Stunde für Kampagne X in den letzten 30 Tagen“ werden durch kontinuierliche Aggregationen (Continuous Aggregates) vorgehalten und in < 100 ms beantwortet. Parameter: `chunk_time_interval=INTERVAL '1 day'`, `compression_policy=compress_chunks older than 30 days`.

### So verwenden Sie es

1.  **Initiale Konfiguration der Datenquellen und des Event-Busses:**
    Definieren Sie in der Konfigurationsdatei `config/giv_tracker.yml` die Verbindungsparameter für den Kafka-Cluster (z.B. `bootstrap_servers=broker1:9092,broker2:9092`, `topic=giveaway_events`, `consumer_group=giv_tracker_group`). Legen Sie die Redis-Verbindung für den Bloom-Filter fest (`redis.host=redis-cluster`, `redis.port=6379`, `redis.password=your_secure_password`). Starten Sie anschließend den `Event-Ingestion-Service` mit `docker-compose up -d ingestion-service`.

2.  **Definition eines Gewinnspiel-Kampagnen-Schemas via REST-API:**
    Senden Sie einen `POST`-Request an den `Campaign-Manager`-Service (`/api/v1/campaigns`) mit einem JSON-Body, der die Logik-Parameter enthält:
    ```json
    {
      "campaign_id": "CAM-2024-001",
      "selection_algorithm": "weighted_random",
      "weight_field": "user_tier",
      "rules": [
        {"condition": "user_tier == 'gold'", "weight_multiplier": 2.5},
        {"condition": "user_tier == 'silver'", "weight_multiplier": 1.0}
      ],
      "fraud_detection": {
        "ip_rate_limit": 50,
        "device_id_deduplication": true
      }
    }
    ```
    Die API antwortet mit dem Status `201 Created` und der `campaign_uid`. Dieser UID wird für alle weiteren Operationen verwendet.

3.  **Überwachung der Echtzeit-Aggregationen und manuelle Auslösung der Gewinnziehung:**
    Greifen Sie auf das Metrik-Dashboard des `Analytics-Service` zu (Standard-Port `9090`). Hier sehen Sie Metriken wie `total_valid_participations`, `fraud_attempts_blocked` und `current_bloom_filter_load`. Um die Gewinnziehung für eine Kampagne auszulösen, senden Sie einen `POST`-Request an `/api/v1/campaigns/{campaign_uid}/draw`. Der Service führt daraufhin die AST-basierte Logik aus, wählt die Gewinner aus und schreibt die Ergebnisse in die Tabelle `winners`. Der Status der Ziehung ist unter `/api/v1/campaigns/{campaign_uid}/draw/status` einsehbar.

### Beste Praktiken

- **Optimierung der Bloom-Filter-Kapazität:** Die Kapazität des Bloom-Filters sollte auf Basis der erwarteten maximalen Teilnehmerzahl einer Kampagne zuzüglich eines Sicherheitsaufschlags von 20% konfiguriert werden. Eine zu geringe Kapazität (`bloom_filter.capacity`) führt zu einer erhöhten Falsch-Positiv-Rate und damit zu fälschlicherweise abgewiesenen Teilnahmen. Verwenden Sie die Formel `capacity = expected_participants * 1.2` und überwachen Sie die Metrik `bloom_filter.false_positive_rate` im Dashboard. Eine Rate über 0,1% erfordert eine Neukonfiguration und einen Reset des Filters.

- **Strategische Partitionierung des Kafka-Topics:** Das Topic `giveaway_events` sollte auf Basis der `campaign_id` partitioniert werden (z.B. `partitions=12` für eine moderate Last). Dies stellt sicher, dass alle Ereignisse einer Kampagne von demselben Consumer verarbeitet werden, was die Notwendigkeit für verteilte Transaktionen eliminiert. Die Anzahl der Partitionen sollte ein Vielfaches der Anzahl der Consumer-Instanzen sein, um eine gleichmäßige Lastverteilung zu gewährleisten. Überwachen Sie die `consumer_lag`-Metrik, um Engpässe frühzeitig zu erkennen.
