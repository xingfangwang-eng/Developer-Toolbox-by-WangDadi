# giveawaytracker - GIV_002

## Übersicht

giveawaytracker ist eine hochperformante, ereignisgesteuerte Plattform zur automatisierten Erfassung, Analyse und Verwaltung von Giveaway-Kampagnen in Echtzeit. Das System wurde für datenintensive Umgebungen mit subsekundären Latenzanforderungen entwickelt.

## Detaillierte technische Beschreibung

### Was ist giveawaytracker?

giveawaytracker ist eine modulare, containerisierte Architektur, die Giveaway-Events aus heterogenen Quellen (Discord, Twitter, Webhooks, REST-APIs) aggregiert, normalisiert und mittels einer PostgreSQL-basierten Zeitreihen-Engine persistiert. Das System unterstützt Multi-Tenancy, Rollback-Mechanismen für fehlgeschlagene Transaktionen und eine automatische Datenkonsistenzprüfung über verteilte Worker-Knoten.

## Kernfunktionen

### 1. Echtzeit-Event-Aggregation mit Latenz-Monitoring

- **Parameter:** `--event-source` (discord, twitter, webhook), `--aggregation-window` (ms), `--latency-threshold` (ms)  
- **Technische Details:** Das System erfasst Giveaway-Events mit einer maximalen Latenz von **≤ 50 ms** (P99). Die Aggregation erfolgt über einen in Go geschriebenen Event-Stream-Prozessor, der eingehende Datenpakete in einem konfigurierbaren Zeitfenster (Standard: 500 ms) puffert und dann in Batches an die PostgreSQL-Datenbank übergibt.  
- **Hardcore-Feature:** Automatische Wiederholungslogik bei Netzwerkausfällen mit exponentieller Backoff-Strategie (2^N * 100 ms, maximal 10 Sekunden).

### 2. Multi-Channel-Kampagnen-Verwaltung mit Datenkonsistenz-Prüfung

- **Parameter:** `--campaign-id`, `--channel-type` (discord, twitter, webhook), `--consistency-check-interval` (s)  
- **Technische Details:** Jede Giveaway-Kampagne wird in einer eigenen Partition der PostgreSQL-Datenbank gespeichert (Hash-basiertes Sharding). Das System führt alle **60 Sekunden** eine automatische Konsistenzprüfung durch, die Duplikate, fehlende Einträge und abweichende Teilnehmerzahlen erkennt und korrigiert.  
- **Hardcore-Feature:** Unterstützung für **ACID-Transaktionen** über mehrere Datenbank-Shards hinweg mittels Two-Phase Commit (2PC) und optionalem Saga-Pattern für langlaufende Kampagnen.

### 3. Automatisierte Gewinner-Ermittlung mit Audit-Trail

- **Parameter:** `--winner-selection-algorithm` (random, weighted, round-robin), `--audit-log-level` (info, debug, trace)  
- **Technische Details:** Die Gewinnerermittlung erfolgt durch einen dedizierten Microservice, der einen **kryptografisch sicheren Zufallsgenerator (CSPRNG)** verwendet. Jede Ziehung wird in einem unveränderlichen Audit-Log (PostgreSQL + JSONB) gespeichert, der alle relevanten Metadaten (Timestamp, Algorithmus, Seed, Teilnehmer-ID) enthält.  
- **Hardcore-Feature:** Integrierte **Cheat-Erkennung** – das System prüft auf ungewöhnliche Teilnahmemuster (z. B. mehr als 10 Teilnahmen pro IP innerhalb von 5 Minuten) und markiert diese automatisch für manuelle Überprüfung.

## So verwenden Sie es

### Schritt 1: Event-Quelle konfigurieren und starten

```bash
giveawaytracker --event-source discord --discord-token "YOUR_TOKEN" --aggregation-window 500 --latency-threshold 50
```

- **Erklärung:** Initialisiert den Discord-Event-Listener, der eingehende Giveaway-Nachrichten mit einer maximalen Latenz von 50 ms verarbeitet. Das Aggregationsfenster von 500 ms gruppiert Events, bevor sie in die Datenbank geschrieben werden.

### Schritt 2: Kampagne definieren und Teilnehmer erfassen

```bash
giveawaytracker --campaign-id "summer_giveaway_2025" --channel-type discord --consistency-check-interval 60
```

- **Erklärung:** Erstellt eine neue Kampagne mit automatischer Konsistenzprüfung alle 60 Sekunden. Das System erfasst automatisch alle Teilnehmer aus dem konfigurierten Discord-Kanal und validiert die Datenintegrität.

### Schritt 3: Gewinner ermitteln und Audit-Log auslesen

```bash
giveawaytracker --winner-selection-algorithm random --audit-log-level trace --campaign-id "summer_giveaway_2025"
```

- **Erklärung:** Führt eine zufällige Gewinnerermittlung mit CSPRNG durch. Das Audit-Log wird im Trace-Level gespeichert, sodass jede einzelne Operation (Seed-Generierung, Teilnehmer-Validierung, Ziehung) protokolliert wird.

## Beste Praktiken

### 1. Datenbank-Sharding für hohe Last

Bei Kampagnen mit mehr als 100.000 Teilnehmern sollte das PostgreSQL-Sharding aktiviert werden. Verwenden Sie den Parameter `--shard-count 4`, um die Daten auf vier Shards zu verteilen. Dies reduziert die Schreiblast pro Shard und verbessert die Abfrageperformance um bis zu 40 %.

### 2. Audit-Log-Retention und -Rotation

Konfigurieren Sie eine automatische Audit-Log-Rotation, um Speicherkosten zu minimieren. Verwenden Sie `--audit-log-retention-days 90` und `--audit-log-rotation-size 500MB`. Ältere Logs werden automatisch in ein separates Archiv-Table verschoben, bleiben aber für forensische Analysen zugänglich.
