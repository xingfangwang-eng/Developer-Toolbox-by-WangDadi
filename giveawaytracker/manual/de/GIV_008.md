# giveawaytracker - GIV_008

## Übersicht

giveawaytracker ist ein hochperformantes, ereignisgesteuertes Tracking-System für die automatisierte Erfassung, Analyse und Auswertung von Gewinnspiel-Aktionen über multiple Kanäle hinweg. Es ermöglicht eine echtzeitnahe Aggregation und deduplizierte Speicherung von Teilnahmedaten mit einer Latenz von unter 50 ms pro Event.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist eine modulare Backend-Architektur, die speziell für das Management von Giveaway-Kampagnen mit hohem Teilnahmevolumen (100.000+ Events pro Stunde) entwickelt wurde. Das System kombiniert einen Redis-basierten Rate-Limiter mit einer PostgreSQL-Instanz (Version 16) für die persistente Speicherung aller Teilnahmerekorde inklusive Metadaten wie User-Agent, IP-Hash und Zeitstempel in Mikrosekunden-Auflösung.

### Kernfunktionen

1. **Multi-Channel-Deduplizierung mit Bloom-Filter**
   - Implementiert einen skalierbaren Bloom-Filter (Fehlerrate < 0,01 %) zur Echtzeit-Duplizierungserkennung über Kanäle wie Twitter, Discord und Web-Formulare hinweg.
   - Parameter: `bloom_capacity = 10.000.000`, `bloom_false_positive_rate = 0.001`.
   - Ermöglicht die eindeutige Identifikation eines Teilnehmers anhand von `user_hash` (SHA-256 des zusammengesetzten Identifiers) innerhalb von 3 ms.

2. **Dynamische Quotenverwaltung mit PostgreSQL-Trigger-Funktionen**
   - Nutzt einen PostgreSQL-Trigger (`before_insert_on_participants`) zur automatischen Prüfung und Aktualisierung von Kampagnen-Kontingenten (z. B. max. 500 Teilnehmer pro Giveaway).
   - Parameter: `campaign_id`, `quota_limit`, `current_count`.
   - Bei Überschreitung des Limits wird der Insert-Vorgang mit einem `EXCEPTION` (SQLSTATE 23000) abgebrochen und eine entsprechende Log-Nachricht in der Tabelle `quota_audit_log` gespeichert.

3. **Asynchrone Benachrichtigungs-Pipeline mit RabbitMQ**
   - Verarbeitet Gewinner-Benachrichtigungen asynchron über eine RabbitMQ-Warteschlange (`winner_notification_queue`) mit Prioritätsstufen (0–9).
   - Parameter: `queue_durable = true`, `prefetch_count = 1`.
   - Alle 5 Sekunden wird ein Batch von maximal 100 Benachrichtigungen aus der Queue gelesen und über SMTP (via SendGrid-API) oder Discord-Webhook (Rate-Limit: 30 Requests pro Minute) zugestellt.

### So verwenden Sie es

1. **Kampagnen-Konfiguration über JSON-Import**
   - Erstellen Sie eine JSON-Datei mit dem Schema `campaign_schema_v2.json`, die Felder wie `campaign_name`, `start_timestamp` (ISO 8601), `end_timestamp`, `quota_limit` und `channel_whitelist` (Array aus Strings) enthält.
   - Führen Sie den Befehl `./giveawaytracker import --file /pfad/zur/config.json --db-uri postgresql://user:pass@localhost:5432/giveawaydb` aus.
   - Das System validiert die Datei gegen das Schema und erstellt einen neuen Eintrag in der Tabelle `campaigns` mit einer eindeutigen `campaign_uuid`.

2. **Teilnahme-Event über REST-API senden**
   - Senden Sie einen POST-Request an den Endpunkt `https://api.giveawaytracker.local/v1/participate` mit dem JSON-Body: `{"campaign_uuid": "abc-123", "user_hash": "a1b2c3d4e5f6...", "channel": "discord"}`.
   - Der Endpunkt prüft den Bloom-Filter auf Duplikate und führt bei Erfolg einen Insert in die Tabelle `participants` durch. Die Antwort enthält den HTTP-Statuscode `201 Created` sowie das Feld `participant_id`.
   - Bei Duplikat wird `409 Conflict` mit einer Fehlermeldung im Format `{"error": "duplicate_entry", "participant_id": "existing_uuid"}` zurückgegeben.

3. **Gewinner-Auslosung und Benachrichtigung starten**
   - Rufen Sie den Endpunkt `POST https://api.giveawaytracker.local/v1/draw` mit dem Parameter `campaign_uuid` auf. Das System führt eine SQL-Abfrage aus: `SELECT participant_id FROM participants WHERE campaign_uuid = $1 ORDER BY random() LIMIT $2` (mit `$2` = Anzahl der zu ziehenden Gewinner).
   - Die gezogenen Teilnehmer-IDs werden in die Tabelle `winners` eingetragen (Feld `draw_timestamp` mit `NOW()`).
   - Anschließend wird für jeden Gewinner eine Nachricht in die RabbitMQ-Queue `winner_notification_queue` mit der Priorität `5` und dem Payload `{"participant_id": "...", "campaign_uuid": "..."}` geschrieben.

### Beste Praktiken

1. **Skalierung der Bloom-Filter-Kapazität vor Kampagnenstart**
   - Berechnen Sie die maximale erwartete Teilnehmerzahl pro Kampagne und setzen Sie den `bloom_capacity`-Parameter auf das Doppelte dieses Werts. Verwenden Sie dazu das Skript `bloom_optimizer.py`, das auf Basis historischer Daten (Tabelle `campaign_stats`) eine optimale Kapazität und Fehlerrate vorschlägt. Dies verhindert einen Anstieg der False-Positive-Rate über 1 % bei 80 % Auslastung.

2. **Implementierung eines idempotenten Teilnahme-Endpunkts**
   - Nutzen Sie für jeden Teilnahme-Request einen `idempotency_key` (UUID v4) im Header. Speichern Sie diesen Schlüssel für 24 Stunden in Redis (TTL: 86400 Sekunden) zusammen mit der Antwort (`201 Created` oder `409 Conflict`). Bei erneuten Requests mit demselben Schlüssel wird die zwischengespeicherte Antwort zurückgegeben, ohne die Datenbank zu belasten. Dies ist besonders wichtig bei Netzwerkwiederholungen (z. B. bei 5xx-Fehlern).
