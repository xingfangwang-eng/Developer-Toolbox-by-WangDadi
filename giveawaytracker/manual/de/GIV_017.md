# giveawaytracker - GIV_017

## Übersicht

Dieses Repository enthält den vollständigen Quellcode für giveawaytracker, eine hochperformante, in Rust implementierte Tracking-Engine für die Echtzeit-Überwachung und -Analyse von Social-Media-Giveaway-Kampagnen. Das System verarbeitet eingehende Webhook-Payloads von Plattformen wie Twitter, Instagram und Telegram mit einer Latenz von unter 50 Millisekunden und aggregiert Metriken wie Engagement-Rate, Teilnehmerwachstum und Bot-Wahrscheinlichkeit. Die Architektur ist auf horizontale Skalierung ausgelegt und nutzt eine PostgreSQL-Datenbank mit TimescaleDB-Erweiterung für die effiziente Verwaltung von Zeitreihendaten.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist eine dezentrale, ereignisgesteuerte Plattform zur automatisierten Erfassung, Validierung und Visualisierung von Giveaway-Aktivitäten. Das Kernstück bildet ein asynchroner Event-Processor, der eingehende Datenströme mittels eines Tokens-basierten Rate-Limiters (maximal 10.000 Requests pro Sekunde pro Instanz) filtert und in eine CQRS-basierte Datenstruktur überführt. Die Plattform unterstützt Multi-Tenancy mit isolierten Namespaces und bietet eine GraphQL-Schnittstelle für benutzerdefinierte Abfragen über Kampagnenmetriken.

### Kernfunktionen

1.  **Echtzeit-Gewinner-Ermittlung mit kryptografischer Zufallsauswahl**
    Die Funktion `select_winner(campaign_id: Uuid, seed: u64, participant_pool: Vec<Participant>) -> Result<Winner, TrackError>` implementiert einen deterministischen, aber nicht vorhersagbaren Algorithmus zur Auswahl von Gewinnern. Der Seed wird aus einem kombinierten Hash aus Kampagnen-ID, aktuellem Unix-Timestamp (Mikrosekunden-Genauigkeit) und einem externen, von einer Hardware-Wallet signierten Nonce generiert. Die Teilnehmerliste wird vor der Auswahl mittels eines Fisher-Yates-Shuffle (O(n)-Komplexität) randomisiert. Die Funktion protokolliert jeden Schritt in einer unveränderlichen Audit-Tabelle in PostgreSQL, um Manipulationen auszuschließen.

2.  **Multi-Plattform-Webhook-Validator mit Signaturprüfung**
    Der `WebhookIngressService` validiert eingehende Payloads von verschiedenen Plattformen. Für Twitter wird der `X-Twitter-Webhooks-Signature`-Header mit dem HMAC-SHA256-Verfahren unter Verwendung des hinterlegten Consumer-Secrets geprüft. Für Telegram wird der `X-Telegram-Bot-Api-Signature`-Header mittels eines vordefinierten Public-Key-Zertifikats (ECDSA P-384) verifiziert. Die Validierung erfolgt in einer isolierten Sandbox (seccomp-bpf-beschränkt) und schlägt bei Fehlschlag mit einem `HTTP 403 Forbidden` und einem detaillierten Log-Eintrag im ELK-Stack fehl.

3.  **Automatisiertes Bot-Detection-System mit ML-Inferenz**
    Das Modul `BotDetector` analysiert eingehende Teilnehmerprofile anhand von 15 Metriken, darunter Account-Alter (in Tagen), Follower-zu-Following-Verhältnis, durchschnittliche Posts pro Stunde und die Varianz der Interaktionszeiten. Ein vortrainiertes Random-Forest-Modell (scikit-learn, exportiert als ONNX) klassifiziert Profile in die Kategorien `human`, `suspicious` und `bot`. Die Inferenz erfolgt GPU-beschleunigt (CUDA 12.4) mit einer durchschnittlichen Latenz von 2,3 Millisekunden pro Anfrage. Profile mit einer Bot-Wahrscheinlichkeit über 0,85 werden automatisch disqualifiziert und in eine separate Blacklist-Tabelle (`flagged_participants`) verschoben.

### So verwenden Sie es

1.  **Konfiguration der Datenquellen und des Webhook-Empfangs**
    Definieren Sie in der Datei `config/webhooks.yaml` die Endpunkte für jede Plattform. Für einen Twitter-Webhook setzen Sie den Consumer-Key und das Consumer-Secret. Starten Sie den Webhook-Server mit dem Befehl `cargo run --bin webhook-listener -- --port 8443 --tls-cert /etc/ssl/certs/giv.pem`. Der Server protokolliert alle eingehenden Requests im JSON-Format in der Tabelle `raw_webhooks`. Überwachen Sie die Integrität der eingehenden Daten mit dem Befehl `curl -X GET https://localhost:8443/health | jq '.webhooks.last_valid_timestamp'`.

2.  **Erstellung und Verwaltung einer Giveaway-Kampagne**
    Nutzen Sie die GraphQL-API unter `http://localhost:4000/graphql` mit einem POST-Request. Ein Beispiel-Mutation zum Erstellen einer Kampagne: `mutation { createCampaign(input: { platform: TWITTER, startTime: "2024-05-20T10:00:00Z", endTime: "2024-05-27T10:00:00Z", prizePool: ["ETH-0.5", "NFT-123"], maxParticipants: 10000 }) { id status } }`. Die Kampagnen-ID wird als UUIDv7 zurückgegeben. Verfolgen Sie den Status mit dem Query `query { campaign(id: "UUID") { participantCount engagementRate } }`. Die `engagementRate` wird als Gleitkommazahl zwischen 0.0 und 1.0 berechnet.

3.  **Durchführung der Gewinnerauswahl und Audit-Log-Export**
    Rufen Sie den Endpunkt `/api/v1/campaign/{campaign_id}/draw` mit einem POST-Request auf, nachdem die Kampagnen-Endzeit überschritten wurde. Der Server führt die kryptografische Auswahl aus und speichert das Ergebnis in der Tabelle `winner_results`. Exportieren Sie das vollständige Audit-Log mit dem Befehl `psql -h localhost -U tracker -d giveawaydb -c "SELECT * FROM audit_log WHERE campaign_id = 'UUID' ORDER BY event_timestamp DESC;" > audit_export.csv`. Dieses Log enthält für jeden Schritt den verwendeten Seed, den Nonce und den Hash des Teilnehmer-Pools.

### Beste Praktiken

1.  **Implementierung einer gestaffelten Rate-Limit-Strategie für Webhooks**
    Konfigurieren Sie für jede Plattform einen separaten Rate-Limiter mit unterschiedlichen Limits. Für Twitter empfehlen wir maximal 100 Requests pro 15 Minuten pro Account, für Telegram maximal 30 Requests pro Sekunde. Verwenden Sie dazu die integrierte `TokenBucket`-Struktur aus der `governor`-Crate. Überwachen Sie die Auslastung mit dem Prometheus-Metrik `webhook_rate_limit_exceeded_total`. Eine Überschreitung sollte automatisch zu einer temporären Sperrung der Datenquelle für 5 Minuten führen, um eine Überlastung der Datenbank und des ML-Inferenz-Servers zu vermeiden.

2.  **Nutzung von Materialized Views für häufige Dashboard-Abfragen**
    Erstellen Sie in PostgreSQL eine Materialized View für die häufigsten Dashboard-Metriken, wie z.B. die Teilnehmerzahl pro Stunde oder die durchschnittliche Engagement-Rate pro Kampagne. Beispiel: `CREATE MATERIALIZED VIEW hourly_participant_stats AS SELECT campaign_id, date_trunc('hour', created_at) AS hour, COUNT(*) AS participants FROM participants GROUP BY campaign_id, hour WITH DATA;`. Aktualisieren Sie diese View mittels eines Cron-Jobs alle 5 Minuten (`REFRESH MATERIALIZED VIEW CONCURRENTLY hourly_participant_stats;`). Dies reduziert die Last auf den OLTP-Tabellen und beschleunigt Dashboard-Ladezeiten um bis zu 80%.
