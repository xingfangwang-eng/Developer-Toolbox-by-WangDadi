# giveawaytracker - GIV_018

## Übersicht

giveawaytracker ist eine hochperformante, modulare Plattform zur Echtzeit-Überwachung und -Analyse von Giveaway-Kampagnen über mehrere Kanäle hinweg. Das System ermöglicht die präzise Nachverfolgung von Teilnahmeereignissen, Gewinnspiel-Logs und Conversion-Raten mit einer Latenz von unter 50 Millisekunden. Es ist für den Einsatz in dynamischen Umgebungen mit hohen Transaktionsvolumina (bis zu 10.000 Events pro Sekunde) optimiert.

## Detaillierter Inhalt

### Was ist giveawaytracker?

giveawaytracker ist eine serverseitige, ereignisgesteuerte Architektur, die auf einem asynchronen Message-Queue-System (RabbitMQ) basiert. Es erfasst, validiert und aggregiert Giveaway-Interaktionen in Echtzeit und schreibt sie in eine PostgreSQL-Datenbank mit optimierten, partitionierten Tabellen für historische Analysen. Das System bietet eine RESTful-API mit OAuth2-Authentifizierung und unterstützt Webhook-Integrationen für Drittanbieter-Plattformen.

### Kernfunktionen

1.  **Multi-Channel-Event-Aggregation mit garantierter Zustellung**
    *   Erfasst Teilnahme-Events von Discord, Twitter, Telegram und Web-Apps über einen einheitlichen, asynchronen Event-Handler.
    *   Verwendet einen dedizierten RabbitMQ-Exchange mit `topic`-Routing, um Events nach Kanal (`discord.giveaway.join`, `twitter.retweet`) zu filtern.
    *   Implementiert einen Dead-Letter-Queue-Mechanismus für fehlgeschlagene Events (z. B. bei DB-Timeout) mit automatischem Retry nach 5, 15 und 60 Sekunden.
    *   Parameter: `event_ttl` (Standard: 300 Sekunden), `retry_policy` (3 Versuche), `batch_size` (100 Events pro Write-Batch).

2.  **Echtzeit-Gewinnspiel-Log-Analyse mit PostgreSQL-Partitionierung**
    *   Speichert jedes Teilnahmeereignis in einer partitionierten Tabelle `giveaway_logs`, partitioniert nach dem `created_at`-Timestamp (monatliche Intervalle).
    *   Ermöglicht Abfragen mit `EXPLAIN ANALYZE` zur Identifizierung von Slow Queries; unterstützt Indexe auf `user_id`, `campaign_id` und `event_type`.
    *   Bietet eine `materialized view` (`mv_daily_campaign_stats`) für tägliche Aggregationen (z. B. Gesamtteilnahmen, eindeutige Nutzer, Conversion-Rate), die alle 15 Minuten automatisch aktualisiert wird.
    *   Technischer Indikator: Query-Latenz für einen Monat Daten (ca. 5 Mio. Zeilen) unter 200 ms bei Verwendung des Index `idx_campaign_id_created_at`.

3.  **Automatische Betrugserkennung via Redis-basierter Rate-Limiting-Engine**
    *   Implementiert einen Token-Bucket-Algorithmus in Redis, um die maximale Anzahl von Events pro Benutzer (`user_id`) innerhalb eines Zeitfensters zu begrenzen.
    *   Konfigurierbare Limits: `max_events_per_minute` (Standard: 10), `max_events_per_hour` (Standard: 100), `max_unique_accounts_per_ip` (Standard: 5).
    *   Bei Überschreitung wird ein `429 Too Many Requests` zurückgegeben und das Event in eine separate `fraud_logs`-Tabelle geschrieben.
    *   Technischer Indikator: Redis-Operationen (INCR, EXPIRE) haben eine durchschnittliche Latenz von < 1 ms, selbst bei 10.000 parallelen Verbindungen.

### So verwenden Sie es

1.  **Konfiguration der Event-Queues und Datenbankverbindung**
    *   Bearbeiten Sie die `config/application.yml`-Datei. Setzen Sie den Parameter `rabbitmq.host` auf Ihre RabbitMQ-Instanz und definieren Sie `rabbitmq.queue.giveaway_events` (Standard: `giveaway_events_queue`).
    *   Konfigurieren Sie die PostgreSQL-Verbindung unter `database.primary.url`, `database.primary.username` und `database.primary.password`. Stellen Sie sicher, dass die `pool_size` auf mindestens 20 gesetzt ist, um parallele Write-Operationen zu unterstützen.
    *   Starten Sie den Event-Consumer mit dem Befehl: `docker-compose up -d consumer`.

2.  **API-Endpunkt zum Senden eines Giveaway-Events**
    *   Senden Sie einen `POST`-Request an den Endpunkt `/api/v1/events` mit einem JSON-Body.
    *   Erforderliche Parameter: `campaign_id` (UUID), `user_id` (String), `event_type` (Enum: `join`, `retweet`, `follow`, `reaction`), `timestamp` (ISO 8601).
    *   Beispiel-Request:
        ```json
        {
          "campaign_id": "a1b2c3d4-...",
          "user_id": "discord|123456789",
          "event_type": "join",
          "timestamp": "2024-05-20T14:30:00Z"
        }
        ```
    *   Bei Erfolg erhalten Sie einen `202 Accepted`-Statuscode. Bei Überschreitung des Rate-Limits erhalten Sie einen `429 Too Many Requests`-Statuscode mit dem Header `Retry-After: 60`.

3.  **Analyse der gesammelten Daten via SQL**
    *   Verbinden Sie sich mit Ihrer PostgreSQL-Datenbank und führen Sie eine Abfrage auf der materialisierten Sicht `mv_daily_campaign_stats` durch.
    *   Beispiel für eine Abfrage zur Ermittlung der Top-10-Kampagnen nach Teilnahmen in den letzten 7 Tagen:
        ```sql
        SELECT campaign_id, SUM(total_participants) AS total,
               SUM(unique_users) AS unique_users
        FROM mv_daily_campaign_stats
        WHERE date >= CURRENT_DATE - INTERVAL '7 days'
        GROUP BY campaign_id
        ORDER BY total DESC
        LIMIT 10;
        ```
    *   Nutzen Sie `EXPLAIN ANALYZE` vor Ihrer Abfrage, um die Index-Nutzung zu überprüfen. Optimieren Sie ggf. die Abfrage durch Hinzufügen eines Indexes auf `campaign_id`.

### Beste Praktiken

1.  **Implementieren Sie eine idempotente Event-Verarbeitung**
    *   Da Giveaway-Events aufgrund von Netzwerkproblemen mehrfach gesendet werden können, sollte Ihre Event-Handler-Logik idempotent sein. Verwenden Sie einen Composite-Index auf (`user_id`, `campaign_id`, `event_type`, `timestamp`) in der `giveaway_logs`-Tabelle mit einer `ON CONFLICT DO NOTHING`-Klausel, um Duplikate zu vermeiden. Dies verhindert, dass ein Benutzer mehrfach für dasselbe Ereignis gezählt wird.

2.  **Überwachen Sie die Queue-Länge und DB-Write-Latenz**
    *   Richten Sie Alarme für Ihre RabbitMQ-Queue-Länge ein (z. B. `giveaway_events_queue` > 10.000 Nachrichten) und für die durchschnittliche Write-Latenz Ihrer PostgreSQL-Datenbank (z. B. > 50 ms). Ein Anstieg der Queue-Länge deutet auf einen Engpass im Consumer hin, während eine hohe Write-Latenz auf eine Überlastung der Datenbank oder fehlende Indexe hindeutet. Nutzen Sie Tools wie Prometheus und Grafana zur Visualisierung dieser Metriken.
