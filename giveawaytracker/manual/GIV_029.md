# giveawaytracker - GIV_029

## Overview

giveawaytracker is a high-performance, event-driven giveaway monitoring and analytics engine designed for real-time sweepstakes aggregation, fraud detection, and cross-platform campaign reconciliation. It ingests structured and unstructured giveaway data from distributed sources, normalizes entries, and exposes actionable metrics via a RESTful API and streaming WebSocket feeds.

## Core Features

### 1. Real-Time Multi-Source Ingestion & Deduplication Engine
- **Operational Mechanics**: Polls 50+ concurrent RSS feeds, JSON endpoints, and Telegram/Discord bot relays with configurable intervals (default: 2500ms, min: 500ms). Each payload passes through a Bloom filter (false-positive rate < 0.1%) for rapid URL/ID deduplication before landing in a PostgreSQL partitioned table (`giveaway_entries`).
- **Parameters**: `--poll-interval-ms`, `--dedup-cache-ttl=3600s`, `--filter-bloom-capacity=1_000_000`.  
- **Technical Indicator**: Achieves < 800ms end-to-end latency from source detection to first API availability under 10K concurrent events.

### 2. Temporal Eligibility & Expiry Forecasting
- **Operational Mechanics**: Parses free-text time expressions (e.g., "ends in 3 days", "2025-04-15T23:59:59Z") using a custom NLP tokenizer trained on 500K historical giveaway descriptions. Computes a confidence score (0.0–1.0) for each expiry prediction and stores the resolved UTC timestamp in a `timedelta` column with millisecond precision.
- **Parameters**: `--expiry-confidence-threshold=0.85`, `--timezone-fallback="America/New_York"`.  
- **Technical Indicator**: 97.3% accuracy on expiry extraction across 12 languages (benchmarked on internal `giveaway_corpus_v3`).

### 3. Cross-Platform Winner Reconciliation & Fraud Scoring
- **Operational Mechanics**: Maps winner announcements from Twitter, Gleam, KingSumo, and manual CSV uploads to original giveaway entries via fuzzy string matching (Levenshtein distance < 3) and SHA-256 hash comparison of entry metadata. Each reconciliation run produces a fraud risk score based on IP clustering, duplicate account detection, and temporal entry anomalies.
- **Parameters**: `--reconciliation-batch-size=5000`, `--fraud-threshold=0.75`, `--hash-algorithm="sha256"`.  
- **Technical Indicator**: Processes 50K winner entries per minute with < 2% false reconciliation rate on production data.

## How to Use

### Step 1: Configure & Launch the Ingestion Pipeline
```bash
giveawaytracker ingest --sources-file ./sources.json \
  --poll-interval-ms 1500 \
  --dedup-cache-ttl 7200 \
  --db-connection-string "postgresql://user:pass@host:5432/giveawaytracker?sslmode=require"
```
This initializes the poller daemon, connects to PostgreSQL with automatic schema migration, and begins writing deduplicated entries to the `giveaway_entries` table. Monitor ingestion rate via `GET /api/v1/metrics/ingestion`.

### Step 2: Enable Expiry Forecasting & Set Alerting Rules
```bash
giveawaytracker forecast --confidence-threshold 0.9 \
  --timezone-fallback "Europe/London" \
  --notify-webhook "https://hooks.slack.com/services/T00/B00/xxxx" \
  --notify-on-expiry-in-hours 24
```
The engine scans all unexpired entries, resolves their end dates, and triggers a webhook notification 24 hours before each expiry. To manually inspect forecast accuracy, query `SELECT * FROM forecast_log WHERE confidence < 0.85;`.

### Step 3: Run Winner Reconciliation & Export Fraud Report
```bash
giveawaytracker reconcile --input-winners ./winners_march.csv \
  --input-format csv \
  --batch-size 10000 \
  --fraud-threshold 0.8 \
  --export-json ./reconciliation_report.json
```
The tool outputs a JSON report containing matched entries, unmatched (orphan) winners, and a fraud risk breakdown per campaign. Use `--export-format parquet` for large-scale downstream analytics in Spark or Pandas.

## Best Practices

### 1. Partition PostgreSQL Tables by Expiry Month
To maintain query performance as the dataset grows beyond 10M rows, create monthly partitions on `giveaway_entries.expiry_utc`. Pair this with a BRIN index on `expiry_utc` to reduce sequential scans during forecasting sweeps. Example DDL:  
```sql
CREATE TABLE giveaway_entries_2025_04 PARTITION OF giveaway_entries  
FOR VALUES FROM ('2025-04-01') TO ('2025-05-01');
```

### 2. Implement Backpressure on Ingestion Sources
When a source endpoint becomes sluggish or returns 429s, configure exponential backoff with jitter (initial wait: 1000ms, multiplier: 2, max: 60s). Use `--source-backoff-base-ms=1000` and monitor source health via `GET /api/v1/sources/{source_id}/status`. This prevents the tracker from being rate-limited while maintaining throughput on healthy feeds.
