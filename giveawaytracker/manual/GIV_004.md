# giveawaytracker - GIV_004

## Overview

`giveawaytracker` is a high-performance, real-time giveaway monitoring and analytics engine designed for cross-platform sweepstakes, raffles, and contest aggregation. It ingests, deduplicates, and ranks giveaway entries using probabilistic data structures and temporal scoring algorithms, enabling precise win-rate optimization and fraud detection.

## Core Features

### 1. Probabilistic Deduplication & Entry Fingerprinting
- **Mechanism**: Uses a combination of **Bloom filters** (memory-optimized, configurable false-positive rate of 0.1%) and **MinHash signatures** (256-bit) to detect duplicate entries across multiple platforms (Twitter, Discord, Gleam) without storing raw user data.
- **Technical Specs**: Supports up to 10M unique entries per giveaway campaign with a deduplication latency of **< 5ms** per entry. Configurable `BLOOM_CAPACITY` and `HASH_FUNCTIONS` parameters exposed via `config.yaml`.
- **Output**: Generates a `dedup_score` (0.0–1.0) and a unique `fingerprint_hash` for each entry, logged to PostgreSQL with a composite index on `(giveaway_id, fingerprint_hash)`.

### 2. Temporal Win-Rate Prediction Engine
- **Mechanism**: Implements a **Poisson point process model** to estimate the probability of winning based on entry timestamps, historical win patterns, and platform-specific decay coefficients. Uses a sliding window of 72 hours with exponential time-decay weighting (`λ = 0.15`).
- **Technical Specs**: Prediction computation time per entry is **< 2ms** (C++ backend via PyO3 binding). Outputs a `win_probability` field (float, 0–100%) and a `recommended_entry_window` (ISO 8601 duration) for optimal entry timing.
- **Integration**: Feeds into a real-time dashboard via WebSocket (JSON payload, 200ms refresh interval). Supports multi-threaded batch processing with `WORKER_THREADS` parameter (default: 4).

### 3. Cross-Platform Webhook & API Aggregation Gateway
- **Mechanism**: A unified webhook receiver that normalizes entries from 12+ platforms (Twitter, Instagram, Discord, Telegram, Gleam, Rafflecopter, etc.) into a canonical JSON schema. Handles OAuth 2.0 token refresh and rate-limit backoff automatically.
- **Technical Specs**: Throughput: **1,200 requests/sec** (tested with k6 on a 4-core instance). Supports configurable `RATE_LIMIT_WINDOW` (default: 60s) and `MAX_RETRIES` (default: 3). Uses Redis Streams for buffering and PostgreSQL for persistent storage with `entry_timestamp` and `platform` as composite partition key.
- **Error Handling**: Logs malformed payloads to a dead-letter queue (Redis List) with full payload inspection via `webhook_inspector` CLI tool.

## How to Use

### Step 1: Configure the Aggregation Gateway
- Edit `config/gateway.yaml` to set your platform-specific API keys and webhook endpoints. Example:
  ```yaml
  platforms:
    twitter:
      bearer_token: "your_bearer_token"
      webhook_secret: "whsec_abc123"
    discord:
      bot_token: "your_bot_token"
      webhook_secret: "whsec_def456"
  ```
- Run the gateway initializer: `./giveawaytracker init --config config/gateway.yaml`. This validates all endpoints, tests OAuth token refresh, and creates the required PostgreSQL tables (`entries`, `fingerprints`, `predictions`) with auto-migration.
- Verify connectivity: `./giveawaytracker test-webhook --platform twitter`. Expect a 200 response with a `webhook_id` and `timestamp` in the payload.

### Step 2: Ingest & Deduplicate Entries
- Start the ingestion daemon: `./giveawaytracker ingest --workers 8 --bloom-capacity 1000000 --false-positive-rate 0.001`. This spawns 8 worker processes that listen on the webhook receiver port (default: 8080).
- Each incoming entry is hashed (SHA-256) and checked against the Bloom filter. If the filter returns a probable match, the MinHash signature is computed and compared against the PostgreSQL `fingerprints` table (indexed by `fingerprint_hash`). Duplicates are flagged with `is_duplicate: true` and stored in a separate `duplicates` table for audit.
- Monitor real-time stats: `./giveawaytracker stats --live`. Outputs entries/sec, dedup rate (%), and Bloom filter false-positive count. Example output:
  ```
  Ingest Rate: 342 entries/sec | Dedup Rate: 12.7% | False Positives: 0.02%
  ```

### Step 3: Run Win-Rate Prediction & Export Results
- Execute the prediction job: `./giveawaytracker predict --giveaway-id "GIV-2024-10-001" --window-hours 72 --decay-lambda 0.15`. This triggers the Poisson model on all non-duplicate entries for that giveaway.
- Results are written to the `predictions` table with fields: `entry_id`, `win_probability`, `recommended_window`, `model_version` (e.g., `v2.3.1`). Export to CSV: `./giveawaytracker export --giveaway-id "GIV-2024-10-001" --format csv --output predictions.csv`.
- For real-time dashboard integration, use the WebSocket endpoint: `ws://localhost:8080/ws/predictions?giveaway_id=GIV-2024-10-001`. Subscribe to receive JSON updates every 200ms.

## Best Practices

### 1. Optimize Bloom Filter Capacity for Large Campaigns
- For giveaways with >1M expected entries, set `BLOOM_CAPACITY` to at least 2x the expected count and `FALSE_POSITIVE_RATE` to 0.001 (0.1%). This ensures deduplication accuracy remains above 99.9% while keeping memory usage under 250MB. Monitor the `bloom_filter_load` metric in the stats dashboard; if it exceeds 80%, increase capacity or rotate the filter with a new seed.

### 2. Schedule Prediction Jobs During Off-Peak Hours
- The Poisson prediction engine is CPU-intensive for large datasets (e.g., 500K entries takes ~12 seconds on a 4-core machine). Schedule `predict` jobs via cron during low-traffic periods (e.g., 3:00 AM UTC) to avoid impacting webhook ingestion latency. Use the `--batch-size 10000` flag to process entries in chunks and reduce memory pressure.
