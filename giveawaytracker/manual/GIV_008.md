### What is giveawaytracker?

giveawaytracker is a high-frequency event ingestion engine that captures, normalizes, and deduplicates giveaway entries from multiple platforms (Twitter, Discord, Telegram, Gleam) with sub-50ms latency per event, storing all interactions in a time-series PostgreSQL cluster for instant replay and audit.

### Core Features

- **Multi-Platform Entry Deduplication & Fraud Scoring**: Each incoming entry is hashed using `SHA256(user_id + platform + campaign_id + timestamp_ms)`. A Redis-backed Bloom filter (false positive rate < 0.1%) rejects duplicate submissions within 2ms. A weighted fraud score is calculated per entry based on IP reputation, account age, and cross-platform consistency; entries scoring above `0.85` are automatically quarantined into a `flagged_entries` table for manual review.
- **Real-Time Winner Selection with Verifiable Randomness**: The selection engine uses a commit-reveal scheme. The admin seeds the random generator with a `campaign_seed_hash` (SHA256 of a secret + campaign ID). For every eligible entry, a deterministic `selection_weight` is computed as `SHA256(entry_id + campaign_seed_hash)`. Winners are chosen by sorting all entries by this weight in ascending order. The full selection log, including the seed hash and entry weights, is stored in a `winner_selection_audit` table, allowing any third party to independently verify the draw.
- **Dynamic Campaign Throttling & Rate Limit Forecasting**: The system monitors the number of entries per second (`eps`) per campaign against a configurable `max_eps` (default: 500). If `eps` exceeds `0.8 * max_eps` for 10 consecutive seconds, the system automatically activates a token-bucket rate limiter on the entry ingestion endpoint, reducing the effective entry rate to `max_eps * 0.6`. Simultaneously, a forecast model (using a moving average over the last 5 minutes) predicts when the campaign will hit the `max_entries` cap and sends a webhook alert 2 minutes before the predicted saturation time.

### How to Use

1. **Initialize a Campaign with Custom Scoring Rules**: Execute the `POST /api/v1/campaigns` endpoint with a JSON payload containing `name`, `platforms` (array of strings), `max_entries` (integer), `start_time` (ISO 8601 timestamp), and `end_time`. The system returns a `campaign_id` (UUID v4). To enable fraud scoring, set `"fraud_detection": {"enabled": true, "threshold": 0.85}`. The API will then create the necessary Redis Bloom filter and PostgreSQL partition for that campaign.
2. **Ingest Entries with Latency Monitoring**: Send entries via `POST /api/v1/entries` with `campaign_id`, `user_id`, `platform`, and `metadata` (JSON). The endpoint returns a `processing_time_ms` field in the response header. Use the `X-Request-Id` header to trace the entry through the pipeline. For high-throughput scenarios, batch entries using `POST /api/v1/entries/batch` (max 100 entries per batch). The system will return an array of `entry_ids` and a `total_processing_time_ms`.
3. **Execute a Verifiable Winner Draw and Export Audit Trail**: Call `POST /api/v1/campaigns/{campaign_id}/draw` with a `secret` parameter (string, min 32 characters). The endpoint returns a `draw_id` and a list of `winner_entry_ids`. To verify the draw, call `GET /api/v1/campaigns/{campaign_id}/draw/{draw_id}/audit`. This returns the `campaign_seed_hash`, the full list of `entry_id` and `selection_weight` pairs, and the `sorted_winners`. Compare the `selection_weight` of the first winner with the hash of `(entry_id + campaign_seed_hash)` to confirm fairness.

### Best Practices

- **Practice 1: Implement a Two-Phase Entry Ingestion for Compliance**: For campaigns with legal or regulatory requirements, do not directly mark entries as "eligible." Instead, first ingest all entries into a `raw_entries` table with a status of `pending`. Then, run a separate validation job (e.g., every 5 minutes) that checks for banned users, geographic restrictions, and duplicate IPs. Only after this validation job updates the status to `eligible` should the entry be included in the draw. This creates a clear audit trail of which entries were excluded and why.
- **Practice 2: Use Webhook-Driven Notifications for Operational Scalability**: Instead of polling the API for campaign status, configure a webhook URL at campaign creation time (`"webhook_url": "https://your-service.com/webhook"`). The system will send `POST` requests to this URL with payloads for events like `campaign.started`, `campaign.80_percent_filled`, `campaign.completed`, and `winner.selected`. Set your webhook handler to return a `202 Accepted` status within 2 seconds to avoid retries. This decouples the tracker from your application logic and ensures zero data loss during peak traffic.
