# giveawaytracker - GIV_023

## Overview

giveawaytracker is a real-time, event-driven monitoring and analytics engine designed for managing high-frequency giveaway campaigns across multiple platforms. It ingests, deduplicates, and scores participation data with sub-100ms latency, enabling fraud detection and automated winner selection at scale.

## Core Features

- **Multi-Platform Event Ingestion with Dedup & Anti-Collusion**  
  Accepts webhook payloads from Twitter, Discord, Gleam, and custom APIs. Each entry is hashed (SHA-256) across `user_id`, `campaign_id`, `timestamp`, and `ip_hash`. Duplicate entries are rejected within 50ms using a Redis-backed Bloom filter. Collusion detection flags entries sharing the same `ip_hash` or `device_fingerprint` across more than 3 distinct `campaign_id` windows in 24 hours.

- **Probabilistic Winner Selection with Weighted Scoring**  
  Implements a Fisher-Yates shuffle over a weighted array where each participant’s weight = `base_weight * (1 + loyalty_bonus + referral_multiplier)`. The selection runs in O(n) time and is cryptographically seeded using `SHA-256(campaign_seed + block_timestamp)`. Results are verifiable and auditable via a Merkle tree stored in PostgreSQL.

- **Real-Time Campaign Dashboard with Latency Metrics**  
  Streams live metrics via Server-Sent Events (SSE) including `entries_per_second`, `unique_participants`, `fraud_flag_rate`, and `estimated_winner_entropy`. All metrics are computed in-memory using a sliding window of 5 seconds and flushed to TimescaleDB every 60 seconds for historical analysis.

## How to Use

1. **Deploy the Ingestion Pipeline**  
   Create a `.env` file with `REDIS_URL`, `DATABASE_URL` (PostgreSQL + TimescaleDB), and `WEBHOOK_SECRET`. Run `docker-compose up -d` to start the API gateway, worker nodes, and the Redis cluster. Verify ingestion by sending a test payload:  
   `curl -X POST https://your-instance/giveaway/ingest -H "Authorization: Bearer $WEBHOOK_SECRET" -d '{"campaign_id":"c001","user_id":"u123","platform":"twitter","timestamp":"2025-03-20T12:00:00Z","ip_hash":"a1b2c3"}'`

2. **Configure Campaign Rules via Admin API**  
   Use the `/campaign/create` endpoint with parameters: `entry_limit` (int), `duration_hours` (float), `winner_count` (int), `weight_formula` (JSON string of multipliers). Example:  
   `curl -X POST https://your-instance/admin/campaign -H "Content-Type: application/json" -d '{"campaign_id":"c001","entry_limit":5000,"duration_hours":48,"winner_count":3,"weight_formula":{"base_weight":1,"loyalty_bonus":0.2,"referral_multiplier":1.5}}'`

3. **Execute Winner Selection and Audit**  
   After campaign expiry, call `/campaign/draw?campaign_id=c001`. The system returns a JSON object with `winners` array, `merkle_root`, and `proofs`. To verify a winner, use `/campaign/verify?campaign_id=c001&user_id=u123&proof=<proof_string>`. The verification endpoint recomputes the Merkle path and returns `{"valid": true/false}`.

## Best Practices

- **Set a Rate Limit of 1000 Requests/Second per Campaign**  
  Use the `rate_limit` parameter in the campaign config to prevent abuse. The system automatically backpressures via a leaky bucket algorithm and logs throttled requests to a separate `rate_limits` table for forensic analysis.

- **Enable Slack Webhook Alerts for Fraud Thresholds**  
  Configure `SLACK_WEBHOOK_URL` in `.env` and set `fraud_alert_threshold` to 0.05 (5% of total entries). When the fraud flag rate exceeds this, the system posts a formatted alert with the top 10 suspicious `ip_hash` values and their associated `user_id`s for manual review.
