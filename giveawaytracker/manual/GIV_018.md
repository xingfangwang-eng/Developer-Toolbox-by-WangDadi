# giveawaytracker - GIV_018

## Overview

giveawaytracker is a high‑throughput, real‑time giveaway event orchestration and analytics engine. It is engineered for teams running large‑scale promotional campaigns (e.g., 100k+ concurrent participants) and need sub‑second consistency across distributed entry validation, prize pool allocation, and fraud detection.

## Core Features

- **Real‑Time Entry Deduplication & Fraud Scoring**  
  Every incoming entry is fingerprinted via SHA‑256 hash of `{user_id, ip, device_fingerprint, timestamp_ms}` and evaluated against a Redis‑backed sliding window (TTL = 300s). Entries exceeding a configurable threshold (default: 3 entries per window) are automatically flagged and routed to a secondary verification queue. Fraud scores (0–100) are computed using a weighted model: 40% IP reputation, 30% device fingerprint collision, 30% historical participation velocity. The entire pipeline processes at <50ms P99 latency.

- **Multi‑Tier Prize Pool Allocation with PostgreSQL & Kafka**  
  Prize pools are defined as JSONB documents in PostgreSQL, supporting hierarchical tiers (e.g., “Gold”, “Silver”, “Bronze”) each with a fixed inventory count and a weighted random selection algorithm. When a winner is selected, the system publishes a `prize_claimed` event to Apache Kafka (topic: `giveaway.prize.claims`, partition key: `tier_id`). A downstream consumer (Go service) atomically decrements the inventory in PostgreSQL using `UPDATE ... WHERE inventory > 0 AND id = $1 RETURNING *` – guaranteeing no overselling even under 10k QPS.

- **Configurable Eligibility Rules Engine**  
  Rules are expressed as a JSON array of predicate objects, e.g.:  
  `[{"field": "user.created_at", "operator": "gte", "value": "2024-01-01"}, {"field": "region", "operator": "in", "value": ["US", "CA"]}]`.  
  The engine compiles these into a decision tree at startup (cached in memory, refreshed every 60s via a background goroutine) and evaluates each entry in <1µs. Supports nested `AND`/`OR` logic, date range checks, geolocation IP‑to‑region mapping (MaxMind GeoLite2), and custom metadata validation.

## How to Use

1. **Initialize the Giveaway Schema & Seed Prize Pools**  
   Run the migration script: `./giveawaytracker migrate --dsn "postgres://user:pass@localhost:5432/giveawaydb?sslmode=disable"`.  
   This creates tables `giveaways`, `prize_tiers`, `entries`, and `winners`. Then seed a giveaway with:  
   `./giveawaytracker create --name "Summer Launch" --start "2025-06-01T00:00:00Z" --end "2025-06-07T23:59:59Z" --pool '{"tiers":[{"name":"Gold","count":10,"weight":0.1},{"name":"Silver","count":100,"weight":0.3}]}'`.  
   Verify the pool using `SELECT * FROM prize_tiers WHERE giveaway_id = 1;`.

2. **Configure Fraud Detection Thresholds**  
   Edit `config.yaml` under `fraud_detection` block:  
   ```yaml
   fraud_detection:
     window_ms: 300000
     max_entries_per_window: 3
     ip_reputation_api: "https://ipqualityscore.com/api/json/YOUR_KEY/%s"
     device_fingerprint_collision_weight: 0.3
   ```  
   Restart the service: `systemctl restart giveawaytracker`. Monitor logs for `fraud_score` output per entry.

3. **Launch the Giveaway & Stream Winners**  
   Start the Kafka consumer for prize claims:  
   `./giveawaytracker consumer --topic giveaway.prize.claims --group giveaway-prod --bootstrap-server localhost:9092`.  
   In a separate terminal, simulate entries:  
   `for i in {1..100}; do curl -X POST http://localhost:8080/entry -d '{"user_id":"user_'$i'","ip":"192.168.1.'$i'","device_fingerprint":"fp_'$i'"}'; done`.  
   Query winners in real time: `SELECT * FROM winners WHERE giveaway_id = 1 ORDER BY claimed_at DESC LIMIT 10;`.

## Best Practices

- **Always Set a Hard Cap on Prize Inventory + Over‑Provision a Dead‑Letter Queue**  
  Use PostgreSQL’s `SELECT ... FOR UPDATE SKIP LOCKED` when allocating prizes to avoid race conditions under high concurrency. Additionally, configure a Kafka dead‑letter topic (`giveaway.prize.claims.dlq`) with a retention of 7 days. If a prize claim fails (e.g., inventory depleted), the consumer should publish to DLQ and alert via PagerDuty – never silently drop.

- **Implement a Graceful Degradation Mode for External Dependencies**  
  The fraud scoring engine relies on an IP reputation API. If that API times out (>500ms) or returns 5xx, the system should fall back to a local cache (Redis sorted set of known bad IPs, refreshed hourly) and log a warning. This ensures giveaway processing continues uninterrupted even if a third‑party service is degraded.
