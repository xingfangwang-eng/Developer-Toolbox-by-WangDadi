### What is giveawaytracker?

giveawaytracker is a real-time monitoring and analytics engine that ingests giveaway events from Discord, Twitter, Telegram, and custom webhook sources. It deduplicates entries using a combination of SHA-256 hashing and rule-based scoring, then feeds structured data into a Postgres-backed pipeline for latency-critical leaderboards and fraud detection. The system maintains sub-50ms write latency at 10,000 concurrent events per second.

### Core Features

- **Cross-Platform Event De-Duplication Engine**  
  Implements a two-phase dedup: first via a Bloom filter (in-memory, 1% false positive rate) for speed, then a PostgreSQL unique constraint on a composite key of `(platform_user_id, giveaway_id, entry_hash)`. Supports 20+ platform adapters with automatic schema evolution.

- **Real-Time Fraud Scoring & Threshold Alerts**  
  Each entry is scored by a weighted model that considers: account age (`account_age_days`), join-to-entry delay (`entry_delay_ms`), IP geolocation entropy (`geo_entropy`), and referral chain depth (`referral_depth`). Scores above 0.85 trigger an automatic block and push a WebSocket alert to the dashboard within 200ms.

- **Multi-Version Campaign Comparison & Rollback**  
  Every giveaway campaign configuration is versioned via a `campaign_versions` table (JSONB diff storage). Supports side-by-side comparison of any two versions, showing changes in `prize_pool`, `entry_window_start`, `entry_window_end`, and `max_entries_per_user`. Rollback to a previous version executes in under 1 second with a single API call.

### How to Use

1. **Initialize a Campaign with Custom Scoring Rules**  
   Send a POST request to `/api/v1/campaigns` with a JSON payload containing `name`, `platform_adapters` (array of strings like `["discord", "twitter"]`), `entry_window_start` (ISO 8601), `entry_window_end`, and `fraud_threshold` (float 0.0–1.0). The system returns a `campaign_id` and a `webhook_secret` for authenticating incoming events.

2. **Ingest Live Events via Webhook or SDK**  
   For each platform, configure the giveawaytracker webhook URL: `https://api.giveawaytracker.io/webhook/{campaign_id}`. Each event must include `platform_user_id`, `platform_username`, `entry_timestamp` (Unix ms), and optional `referrer_id`. The SDK (Node.js or Python) handles retry logic with exponential backoff (max 3 retries, 500ms base delay).

3. **Query Leaderboards and Export Fraud Reports**  
   Use the GraphQL endpoint (`/graphql`) to fetch real-time leaderboards:  
   ```graphql
   query {
     leaderboard(campaignId: "abc123", limit: 100, offset: 0) {
       rank
       userId
       score
       entriesCount
     }
   }
   ```  
   Export a full fraud report as CSV via `GET /api/v1/campaigns/{campaign_id}/fraud-report?format=csv`. The report includes `entry_id`, `fraud_score`, `blocked_flag`, and `reason_code`.

### Best Practices

- **Practice 1: Pre-configure Platform-Specific Rate Limits**  
  Each platform adapter has a default rate limit (e.g., Discord: 50 req/s, Twitter: 15 req/s). Before launching a campaign, adjust `rate_limit_per_second` in the campaign config to match your webhook provider’s capacity. Overloading the default limit can cause 429 responses and missed entries. Use the built-in `rate_limiter` middleware (token bucket algorithm) to smooth bursts.

- **Practice 2: Use Versioned Campaigns for A/B Testing**  
  Instead of creating separate campaigns for different entry rules, leverage the versioning system. Clone a campaign, modify `max_entries_per_user` or `entry_window_duration`, and run both versions simultaneously. Compare `conversion_rate` (entries / unique users) and `fraud_rate` (blocked entries / total entries) via the `/api/v1/campaigns/{id}/compare` endpoint. This reduces schema management overhead and simplifies rollback if a variant underperforms.
