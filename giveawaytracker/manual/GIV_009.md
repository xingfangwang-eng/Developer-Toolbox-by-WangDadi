# giveawaytracker - GIV_009

## Overview

giveawaytracker is a high-frequency, event-driven giveaway tracking and management system designed for real-time campaign monitoring across multi-platform distribution channels. It ingests raw user entries, validates eligibility rules, and synthesizes live leaderboard metrics with sub-second latency.

## Detailed Content

### What is giveawaytracker?

giveawaytracker is a distributed event-sourcing tool that captures, deduplicates, and ranks giveaway participants in real time. It supports concurrent campaigns with dynamic eligibility rules, automated winner selection via cryptographically seeded RNG, and full audit trails for compliance.

### Core Features

- **Multi-Stream Event Ingestion & Deduplication Engine**: Accepts concurrent WebSocket and HTTP/2 POST streams from platforms like Twitter, Discord, Telegram, and custom APIs. Each event is parsed against a schema-based validator (`participant_id`, `campaign_id`, `timestamp_ms`, `source_channel`). Deduplication runs in-memory via a Redis-backed Bloom filter with a configurable false-positive rate (default: 0.1%), ensuring no duplicate entries advance to selection.

- **Dynamic Eligibility Rule Engine with PostgreSQL + Redis Integration**: Rules are defined as JSONB documents in PostgreSQL (`rules: { min_account_age_days: 30, required_reposts: 5, geo_restrictions: ["US", "CA"] }`). The engine evaluates each participant against active rules using a Redis cache (TTL: 300s) for hot campaigns, and falls back to indexed PostgreSQL queries for cold ones. Evaluation latency per participant: < 5ms.

- **Cryptographically Seeded Winner Selection & Audit Trail**: Winner selection uses `ChaCha20` seeded with the campaign's `close_timestamp_ms` and a server-side secret. The algorithm generates a deterministic, verifiable permutation of eligible participants. Every selection run produces a signed JSON log (`{ seed, permutation_index, winner_id, timestamp_ns }`) stored immutably in a PostgreSQL `audit_log` table with a composite index on `(campaign_id, selection_run_id)`.

### How to Use

1. **Deploy the Event Ingestion Gateway**  
   Launch the `giveawaytracker-gateway` service with environment variables `GIVEAWAY_REDIS_URL=redis://...` and `GIVEAWAY_PG_DSN=postgres://...`. Configure WebSocket listeners per platform via the `platforms.yaml` file (example: `twitter: { bearer_token: ..., stream_endpoint: "https://api.twitter.com/2/tweets/search/stream" }`). The gateway automatically scales horizontally behind a load balancer using a Redis pub/sub for event fan-out.

2. **Define a Campaign with Eligibility Rules**  
   Use the `POST /api/v1/campaigns` endpoint with a JSON body containing `{ "name": "Holiday Giveaway 2025", "start_timestamp": 1704067200, "end_timestamp": 1704153600, "max_winners": 3, "rules": { "min_account_age_days": 30, "required_reposts": 2 } }`. The system validates the JSONB schema against a predefined template and returns a `campaign_id` (UUIDv7) with a `status` field set to `scheduled`.

3. **Trigger Winner Selection & Retrieve Audit Log**  
   After the campaign ends, call `POST /api/v1/campaigns/{campaign_id}/select-winners`. The engine locks the campaign row in PostgreSQL (`SELECT ... FOR UPDATE`), fetches all eligible participants from the `entries` table (indexed by `campaign_id`), runs the seeded RNG, and writes the `selection_run` record. Retrieve the signed audit log via `GET /api/v1/campaigns/{campaign_id}/audit-logs`. The response includes `{ seed, permutation, winner_ids, signature_hex }`.

### Best Practices

- **Pre-warm Redis Caches for High-Volume Campaigns**: For campaigns expected to exceed 100k entries, pre-load eligibility rule evaluations into Redis 60 minutes before the campaign start. Use a background job (e.g., `giveawaytracker-warmer`) that queries PostgreSQL for all active rules and writes them to Redis with a TTL equal to the campaign duration plus 10%. This reduces per-entry evaluation latency from ~5ms to <1ms.

- **Implement Idempotent Event Ingestion with Dedup Keys**: Always include a `dedup_key` in each event payload (e.g., `sha256(platform + user_id + campaign_id + timestamp_ms)`). The gateway checks the Redis Bloom filter before inserting into PostgreSQL. If a duplicate is detected, the gateway returns HTTP 409 Conflict with the original `entry_id`. This prevents double-counting and ensures leaderboard integrity.
