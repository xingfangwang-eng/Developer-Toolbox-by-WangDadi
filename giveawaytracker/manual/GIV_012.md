# giveawaytracker - GIV_012

## Overview

giveawaytracker is a high-performance, event-driven giveaway tracking and management system designed for real-time monitoring, fraud detection, and cross-platform campaign orchestration. It processes thousands of concurrent giveaway entries with sub-100ms latency and integrates directly with major social APIs and blockchain explorers for immutable proof-of-participation.

## Core Features

### 1. Real-Time Multi-Channel Entry Aggregator with Fraud Scoring
- **Parameter:** `--entry-source` (Twitter, Discord, Telegram, on-chain), `--fraud-threshold` (float, 0.0–1.0)
- **Metric:** Processes up to 10,000 entries per second with a 99.9% uptime SLI.
- **Behavior:** Parses raw mentions, webhook payloads, and smart contract events into a unified entry schema. Applies a weighted Bayesian fraud scoring model based on account age, IP reputation, and participation velocity. Entries exceeding the `--fraud-threshold` are automatically quarantined into a manual review queue.

### 2. Dynamic Winner Selection Engine with Verifiable Randomness
- **Parameter:** `--algorithm` (Fisher-Yates, Weighted Lottery, Chainlink VRF), `--seed` (hex string, optional)
- **Metric:** Selection execution time is consistently under 50ms for pools of 100,000 entries.
- **Behavior:** Supports configurable weighting (e.g., by follower count, referral count, or custom metadata). When using `--algorithm chainlink-vrf`, the engine fetches a verifiable random number from a Chainlink oracle, ensuring provably fair winner selection. The entire selection process is logged with a deterministic trace for post-event auditing.

### 3. Automated Compliance & Tax Reporting Module
- **Parameter:** `--jurisdiction` (US, EU, UK, CA), `--report-format` (CSV, JSON, IRS Form 1099-MISC template)
- **Metric:** Generates a fully compliant report for up to 5,000 winners in under 3 seconds.
- **Behavior:** Automatically calculates the fair market value (FMV) of each prize using a live price feed (via CoinGecko API for crypto, or a configurable fiat rate). Maps each winner's data to the required tax forms for the selected `--jurisdiction`. Outputs a downloadable ZIP containing individual winner tax packets and a master summary report.

## How to Use

### Step 1: Configure and Initialize the Campaign
```bash
giveawaytracker init --campaign "SummerNFTDrop" \
  --entry-source twitter,on-chain \
  --fraud-threshold 0.85 \
  --jurisdiction US
```
This command creates a new campaign directory with a `config.yaml` file. The system will immediately begin listening for entries from the specified sources. The fraud threshold of 0.85 means any entry with an 85% or higher probability of being fraudulent will be flagged.

### Step 2: Import Historical Entries (Optional) and Run Fraud Analysis
```bash
giveawaytracker import --file historical_entries_2024.csv --source legacy-db
giveawaytracker analyze --campaign "SummerNFTDrop" --output-format json
```
The `import` command allows you to backfill entries from a previous manual spreadsheet or database dump. The `analyze` command runs the fraud scoring algorithm across all entries (new and imported) and outputs a detailed JSON report. Use the `--output-format html` flag to generate a visual heatmap of suspicious entry clusters.

### Step 3: Execute the Draw and Generate Compliance Reports
```bash
giveawaytracker draw --campaign "SummerNFTDrop" \
  --algorithm chainlink-vrf \
  --prize-pool 100 NFT, 5000 USDC \
  --winners 50
giveawaytracker report --campaign "SummerNFTDrop" --format IRS
```
The `draw` command locks the entry pool, requests a random seed from Chainlink, and selects 50 winners. The results are immediately broadcast to a webhook URL (configured in `config.yaml`). The `report` command then generates the IRS-compliant 1099-MISC forms for all 50 winners, calculating the FMV of the NFT and USDC prizes at the time of the draw.

## Best Practices

### Practice 1: Implement a Two-Stage Fraud Review Pipeline
Never rely solely on automated fraud scoring. Configure `--fraud-threshold` at a moderate level (e.g., 0.7) and schedule a daily manual review of the quarantine queue. Use the `giveawaytracker review --campaign "X" --list-quarantined` command to view flagged entries with their full metadata (IP, wallet address, social profile). This hybrid approach catches sophisticated bots that pass the initial filter while maintaining high throughput.

### Practice 2: Use Immutable Logging for Legal Disputes
Enable the `--audit-log` flag in your campaign configuration. This logs every state change (entry received, fraud score updated, winner selected) to a separate append-only JSON file. For high-value giveaways, configure the audit log to also write to a write-once medium (e.g., IPFS or an AWS S3 bucket with Object Lock). This creates a tamper-proof chain of custody that can be presented in case of a legal challenge or community dispute.
