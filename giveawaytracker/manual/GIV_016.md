# giveawaytracker - GIV_016

## Overview

giveawaytracker is a high-performance, event-driven giveaway monitoring and analytics engine designed for real-time sweepstakes aggregation, fraud detection, and multi-platform campaign orchestration. It ingests live data from Discord, Twitter, Telegram, and custom webhooks, processing up to 10,000 concurrent giveaway entries with sub-50ms latency per event.

## Detailed Content

### What is giveawaytracker?

giveawaytracker is a distributed giveaway lifecycle management system that automates entry validation, winner selection (via cryptographically secure random number generation), and cross-platform reward fulfillment. It integrates with PostgreSQL for persistent campaign state and Redis Streams for real-time event bus processing, supporting multi-tenant isolation and RBAC-based access control.

### Core Features

- **Real-Time Multi-Platform Ingestion & Deduplication**: Uses a custom WebSocket gateway (port 8081) and Twitter API v2 filtered stream to capture giveaway events across platforms. Built-in Bloom filter (128MB memory-mapped) and sliding window deduplication (30-second TTL) eliminate duplicate entries with 99.97% accuracy, preventing multi-account abuse.

- **Probabilistic Winner Selection Engine**: Implements Fisher-Yates shuffle over a cryptographically seeded PRNG (ChaCha20-based, reseeded every 10,000 draws) to ensure provably fair winner selection. Supports weighted entry multipliers (e.g., 2x for retweeters) and configurable exclusion lists via PostgreSQL JSONB filters. Returns selection proof hash for auditability.

- **Automated Fulfillment Pipeline with Retry Logic**: Connects to reward APIs (e.g., Discord bot, email SMTP, Stripe for digital goods) via a configurable chain of middleware. Each fulfillment step includes exponential backoff (base 2s, max 60s) and dead-letter queue (DLQ) for failed deliveries. Tracks fulfillment status in a dedicated PostgreSQL table with `fulfillment_status` ENUM (`pending`, `in_progress`, `completed`, `failed`).

### How to Use

1. **Initialize the Giveaway Tracker Daemon**  
   Run `giveawaytracker daemon --config /etc/giveawaytracker/config.yaml --log-level debug`. The daemon will connect to the configured PostgreSQL database (requires `giveawaytracker` schema with migrations applied via `giveawaytracker migrate`), start the Redis Streams consumer group, and begin listening on the WebSocket gateway. Verify with `curl http://localhost:8081/health`.

2. **Define a Campaign with Entry Rules**  
   Use the CLI:  
   ```bash
   giveawaytracker campaign create \
     --name "Summer Giveaway 2024" \
     --platforms discord,twitter \
     --entry-filter '{"must_have": ["follow", "retweet"], "weight": {"retweet": 2}}' \
     --winner-count 50 \
     --fulfillment-chain '["discord_role:123456", "email:winner@example.com"]' \
     --start-ts 2024-06-01T00:00:00Z \
     --end-ts 2024-06-30T23:59:59Z
   ```
   This creates a campaign in PostgreSQL with a unique `campaign_id` and starts the ingestion pipeline.

3. **Execute Winner Draw & Monitor Fulfillment**  
   After the campaign ends, trigger the draw:  
   ```bash
   giveawaytracker draw run --campaign-id <campaign_id> --seed <optional_hex_seed>
   ```  
   The engine outputs a JSON array of winners with their `entry_id`, `user_platform_id`, and `selection_proof`. Monitor fulfillment via:  
   ```bash
   giveawaytracker fulfillment status --campaign-id <campaign_id>
   ```  
   This returns a table with `user_id`, `reward_type`, `status`, and `retry_count`. Failed items can be manually retried with `giveawaytracker fulfillment retry --fulfillment-id <id>`.

### Best Practices

- **Use Separate Database for Audit Logs**: Configure a dedicated PostgreSQL database (e.g., `giveawaytracker_audit`) with WAL-level logging to store all winner selection proofs, fulfillment attempts, and entry timestamps. This ensures regulatory compliance and enables forensic analysis in case of disputes. Enable `track_commit_timestamp` to record exact timestamps.

- **Implement Rate-Limiting on Webhook Endpoints**: To prevent abuse, configure per-IP rate limiting (e.g., 100 requests/minute) on the custom webhook endpoint (`/webhook/custom`) using a Redis-backed token bucket. Additionally, require HMAC-SHA256 signatures for all webhook payloads to authenticate sources. This mitigates DDoS attacks and ensures only authorized platforms can push entries.
