# giveawaytracker - GIV_003

## Overview

giveawaytracker is a high-throughput, event-driven giveaway monitoring and analytics engine designed for real-time sweepstakes, raffle, and promotional campaign tracking. It ingests data from distributed sources, deduplicates entries via cryptographic hashing, and provides sub‑millisecond query latency for active giveaways.

## Detailed Content

### What is giveawaytracker?

giveawaytracker is a horizontally scalable infrastructure component that captures, validates, and indexes giveaway events across multiple platforms. It exposes a RESTful API with WebSocket push for live updates, and stores historical data in a time‑series database for trend analysis. The system is built with Go for the ingestion layer and Rust for the core deduplication engine, achieving a throughput of 50,000 events per second on a single node.

### Core Features

- **Real‑Time Entry Deduplication & Fraud Detection**  
  Each entry is hashed using SHA‑256 over a composite key (`user_id + giveaway_id + timestamp_ms`). Duplicate entries are rejected within 200 microseconds using a Redis‑backed Bloom filter with a false‑positive rate of 0.001%. Suspicious patterns (e.g., same IP submitting >10 entries/minute) trigger an alert via a configurable webhook.

- **Multi‑Platform Event Aggregation with Schema Normalization**  
  Supports ingestion from Twitter, Discord, Telegram, and custom webhook endpoints. The system normalizes each event into a canonical JSON schema with fields: `platform`, `action`, `user_handle`, `entry_timestamp`, `referrer_url`, and `metadata`. The normalized stream is buffered in Apache Kafka (topic `giveaway_events`) before being written to PostgreSQL with TimescaleDB hypertables for retention.

- **Live Leaderboard & Statistical Analysis**  
  A materialized view (`leaderboard_hourly`) updated every 60 seconds aggregates top participants by entry count, win rate, and referral impact. The view is queried via a dedicated read replica (PostgreSQL `pg_stat_statements` enabled) with an average query time of 12 ms for the top 100 entries. Exposes a `/stats` endpoint returning JSON with `total_entries`, `unique_users`, `avg_entries_per_user`, and `conversion_rate`.

### How to Use

1. **Deploy the ingestion pipeline**  
   Clone the repository and run `docker-compose up -d` to start the stack: `giveawaytracker-api` (Go, port 8080), `redis` (port 6379), `kafka` (port 9092), and `postgres` (port 5432). Configure environment variables in `.env`: `KAFKA_BROKERS`, `REDIS_HOST`, `DB_CONNECTION_STRING`. Verify the pipeline with `curl -X POST http://localhost:8080/health`.

2. **Submit a giveaway event**  
   Send a POST request to `/api/v1/events` with the following JSON body:  
   ```json
   {
     "platform": "twitter",
     "action": "retweet",
     "user_handle": "@example_user",
     "entry_timestamp": "2025-03-20T14:30:00Z",
     "referrer_url": "https://twitter.com/giveaway/status/123",
     "metadata": { "tweet_id": "1234567890" }
   }
   ```  
   The API returns `201 Created` with a `dedup_result` field (`"accepted"` or `"duplicate"`). Use the `X-Request-Id` header for tracing.

3. **Query live leaderboard**  
   Retrieve the top 50 participants for a specific giveaway by calling:  
   `GET /api/v1/giveaways/{giveaway_id}/leaderboard?limit=50&offset=0`.  
   The response includes an array of objects with `rank`, `user_handle`, `entry_count`, `last_entry_timestamp`, and `win_probability` (calculated using a weighted random algorithm). Use `?format=csv` to export data for offline analysis.

### Best Practices

- **Pre‑configure rate limits per platform**  
  Set `RATE_LIMIT_TWITTER=100` (entries per minute) and `RATE_LIMIT_DISCORD=50` in the environment to prevent abuse. The system enforces these limits via a token‑bucket algorithm stored in Redis. Monitor throttled requests with the `/metrics` endpoint (Prometheus format).

- **Use the WebSocket endpoint for real‑time dashboards**  
  Connect to `wss://your-instance.com/ws/giveaways/{giveaway_id}` to receive live entry updates (JSON frames every 200 ms). This reduces polling overhead and ensures sub‑second latency for frontend displays. Always implement a reconnection backoff strategy (exponential with jitter) to handle transient network failures.
