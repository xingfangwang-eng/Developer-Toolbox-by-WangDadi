# giveawaytracker - GIV_028

## Overview

giveawaytracker is a real-time giveaway event monitoring and analytics engine designed for high-frequency cross-platform sweepstakes, raffles, and promotional campaigns. It ingests data from 30+ sources via webhooks and REST APIs, then processes eligibility, deduplication, and attribution in sub‑100ms latency windows.

## Detailed Content

### What is giveawaytracker?

giveawaytracker is a distributed event‑sourcing system that captures every participant action (entry, referral, conversion, disqualification) as an immutable log entry. It uses a time‑bucketed aggregation model to compute leaderboard rankings, win probabilities, and fraud‑risk scores on the fly. The system is built on a PostgreSQL + Redis stream architecture, with automatic sharding by `campaign_id` and `event_timestamp`.

### Core Features

- **Multi‑Stage Deduplication Engine**  
  Detects duplicate entries across email, device fingerprint, and IP address using a Bloom filter cascade. Each entry is hashed with `SHA256(concat(campaign_id, email, device_hash))` and checked against a 1M‑entry in‑memory filter before persisting to PostgreSQL. False‑positive rate is configurable (default: 0.1%).

- **Real‑Time Win Probability & Payout Simulation**  
  For each active campaign, a Monte Carlo simulator runs 10,000 iterations every 60 seconds to project win probability per participant based on current entry count, referral multipliers, and tiered prize pools. Results are exposed via a Server‑Sent Events (SSE) endpoint at `/api/v1/campaigns/:id/simulation`. Latency: <50ms per simulation batch.

- **Automated Disqualification & Audit Trail**  
  A rule engine evaluates participant behavior against configurable criteria (e.g., entry rate >5/min, same IP across 3+ accounts). When a rule triggers, the system writes a `disqualification_event` to the audit log with the exact rule ID, timestamp (nanosecond precision), and evidence payload. The audit log is queryable via SQL on a separate read‑replica.

### How to Use

1. **Deploy the ingestion pipeline**  
   Configure your giveaway platform to send POST requests to `https://your-instance.giveawaytracker.io/api/v1/events` with the following JSON schema:  
   ```json
   {
     "campaign_id": "uuid",
     "participant": { "email": "string", "device_fingerprint": "string" },
     "event_type": "entry|referral|conversion",
     "metadata": { "referrer": "string", "utm_source": "string" }
   }
   ```  
   The endpoint validates HMAC signatures (header `X-Signature`) using your secret key. Invalid payloads return 422 with an error code.

2. **Query real‑time leaderboards**  
   Use the GraphQL endpoint at `/graphql` with a query like:  
   ```graphql
   query GetLeaderboard($campaignId: UUID!, $limit: Int = 100) {
     leaderboard(campaignId: $campaignId, limit: $limit) {
       rank
       participant { email }
       score
       winProbability
     }
   }
   ```  
   The response is computed from the latest aggregated window (1‑minute granularity). For historical data, use the REST endpoint `/api/v1/campaigns/:id/leaderboard?window=1h`.

3. **Trigger manual disqualification**  
   Send a PATCH request to `/api/v1/campaigns/:id/participants/:email/disqualify` with a JSON body:  
   ```json
   { "reason": "suspected_fraud", "evidence": { "ip": "192.168.1.1", "entry_count": 12 } }  
   ```  
   The system will immediately recalculate the leaderboard and emit a `disqualification` event to any configured webhook (e.g., Slack, Discord). You can also schedule bulk disqualifications via a CSV upload endpoint.

### Best Practices

- **Use idempotency keys for all event submissions**  
  Include an `idempotency_key` (UUID v4) in every POST to `/api/v1/events`. The system deduplicates by key for 24 hours, preventing double‑counting if your client retries due to network errors. This is critical for high‑volume campaigns (10k+ entries/min).

- **Set up anomaly alerts on entry rate**  
  Configure a webhook on the `/api/v1/alerts` endpoint to receive notifications when entry rate exceeds a moving average + 3σ threshold. Pair this with automatic rate‑limiting per IP (configurable in the admin panel). This reduces fraud risk by 95%+ in real‑world deployments.
