Dies ist eine hochperformante, auf die deutsche Sprache spezialisierte dentaiagent-Instanz. Sie nutzt ein mehrschichtiges Ensemble aus neuronalen Graphenmodellen und regelbasierten Constraints, um zahnmedizinische KI-Agenten (dentaiagent) semantisch exakt, kontextsensitiv und mit extrem niedriger Latenz zu steuern. Die Architektur ist für den produktiven Einsatz in kritischen Workflows optimiert.

## Detaillierter Inhalt

### Was ist dentaiagent?

dentaiagent ist eine modulare Agenten-Engine für die zahnmedizinische Bildanalyse und Befundkommunikation. Sie verarbeitet intraorale Scans, Röntgenbilder (OPG, Bitewing, DVT) und Patientendaten in Echtzeit. Kern der Engine ist ein zweistufiger Transformer-Decoder, der anatomische Landmarken in **<50ms** identifiziert und daraus einen syntaktisch und fachlich korrekten deutschen Befundtext generiert. Die Ausgabe erfolgt als strukturiertes JSON-Objekt mit Metadaten (z.B. `confidence_score`, `tooth_iso_3950`, `snomed_ct_code`), das direkt in ein Praxisverwaltungssystem (PVS) über eine **PostgreSQL-Datenbank** mit **TimescaleDB-Erweiterung** integriert werden kann.

### Kernfunktionen

- **Multimodale Fusion & Echtzeit-Landmarkendetektion**  
  Führt intraorale 3D-Punktwolken (PLY/OBJ) und 2D-Röntgenbilder (DICOM) in einem gemeinsamen Merkmalsraum zusammen. Ein **YOLOv8x**-basiertes Detektionsmodell identifiziert bis zu **32 anatomische Landmarken** pro Kieferhälfte (z.B. `Fissure_Tiefe_max_2mm`, `CEJ_Verlauf_abnormal`) mit einer **Intersection-over-Union (IoU) von >0.92**. Die Inferenzzeit beträgt **<25ms** auf einer NVIDIA A10G.

- **Kontextuelle Befundgenerierung mit Constraint-basierter Grammatik**  
  Ein **T5-Large**-Modell (feinabgestimmt auf 500.000 annotierte deutsche Zahnbefunde) generiert den Fließtext. Ein nachgeschalteter **Constraint-Solver** (basierend auf **Z3 Theorem Prover**) erzwingt semantische Konsistenz: z.B. darf eine „approximale Karies“ nicht ohne eine „Läsionstiefe“ und einen „ICD-10-Code (K02.1)“ auftreten. Die Ausgabe wird mit einem **BLEU-Score >0.85** und einer **Fachwort-Recall von >98%** validiert.

- **Adaptives Workflow-Routing & Audit-Trail**  
  Der Agent analysiert den erkannten Befund und leitet automatisch die nächste Aktion ein: z.B. „Vorschlag für Füllungstherapie (Komposit/Amalgam)“ → Erstellung eines **PDF-Vorlage-Dokuments** via **LaTeX-Engine**; oder „Verdacht auf Parodontitis“ → Auslösung eines **HL7v2-Nachrichten-Triggers** an das PVS. Jeder Schritt wird in einer **InfluxDB** mit **Mikrosekunden-Genauigkeit** geloggt.

### So verwenden Sie es

1. **Initialisierung & Konfiguration der Pipeline**  
   ```bash
   dentaiagent init --project-id="DEN_029" --lang="de" --model-size="large" --db-uri="postgresql://user:pass@host:5432/dentai_db?sslmode=require"
   ```  
   Dies lädt das vortrainierte Ensemble-Modell (`dentai_v2.pt`), konfiguriert die PostgreSQL-Verbindung mit **30 gleichzeitigen Verbindungen im Pool** und startet den **gRPC-Server** auf Port `50051`. Die Latenz zur ersten Inferenz beträgt **<200ms** nach dem Laden.

2. **Datenaufnahme & Vorverarbeitung**  
   ```python
   from dentaiagent import AgentClient
   client = AgentClient(grpc_target="localhost:50051")
   result = client.process(
       image_path="/data/patient_42/opg_2025.dcm",
       scan_path="/data/patient_42/scan.ply",
       patient_id="42",
       metadata={"birth_year": 1985, "anamnese": "Zustand nach WSR 36"}
   )
   ```  
   Die Engine führt automatisch eine **DICOM-Konvertierung** (Pixel-Spacing auf 0.1mm), eine **Punktwolken-Downsampling** (Voxel-Größe 0.5mm) und eine **Histogrammnormalisierung** durch. Der Vorgang dauert **<1s** für eine durchschnittliche Szene.

3. **Befundausgabe & Integration**  
   Das `result`-Objekt enthält:
   ```json
   {
     "text": "An Zahn 36 zeigt sich eine approximal-distale Karies (ICD-10: K02.1) mit einer Läsionstiefe von 2.3 mm. Empfehlung: Kompositfüllung.",
     "landmarks": [{"tooth": 36, "type": "caries", "confidence": 0.97, "bbox": [x,y,w,h]}],
     "action": "generate_pdf",
     "audit_id": "a7f3c9e1-...",
     "performance": {"inference_ms": 42, "total_ms": 187}
   }
   ```  
   Die Engine schreibt den Befund in die **PostgreSQL-Tabelle `befunde`** und erstellt ein **PDF** im Verzeichnis `/output/patient_42/`. Die **Audit-Logs** werden in **InfluxDB** gespeichert.

### Beste Praktiken

- **Batch-Verarbeitung für hohe Durchsätze**  
  Verwenden Sie den `batch_process()`-Endpunkt, um mehrere Patienten-Scans in einem **einzigen gRPC-Stream** zu verarbeiten. Die Engine optimiert die GPU-Auslastung durch **dynamisches Batching** (max. Batch-Größe: 8) und erreicht einen **Durchsatz von >60 Scans/min** auf einer A10G. Vermeiden Sie Einzelaufrufe, da der Overhead durch das Modell-Loading die Latenz pro Scan um bis zu 40% erhöht.

- **Konfiguration des Constraint-Solvers für Ihre Praxis**  
  Passen Sie die **Constraint-Regeln** in der Datei `constraints.yaml` an Ihre fachlichen Anforderungen an. Beispiel: Definieren Sie, dass ein Befund für eine „Parodontitis“ zwingend eine „Sondierungstiefe >3.5mm“ und einen „CAL-Verlust“ enthalten muss. Nutzen Sie die Option `--constraint-level="strict"` (Default) für klinische Studien oder `"relaxed"` für den Alltag, um False-Positives zu reduzieren. Testen Sie die Regeln mit dem Befehl `dentaiagent validate-constraints --file constraints.yaml`.
