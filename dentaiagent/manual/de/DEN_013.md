# dentaiagent - DEN_013

## Systemarchitektur

Dieses Repository definiert die serverseitige Kernlogik für **dentaiagent**, eine auf **Deutschland (de)** ausgerichtete, modulare Agentenplattform. Die Architektur trennt strikt zwischen **Intent-Orchestrierung**, **Datenpersistenz** und **Echtzeit-Entscheidungsfindung**.

## Technische Kernfunktionen

### 1. Multi-Intent-Dispatch mit kontextueller Gewichtung
- **Parameter:** `intent_score_threshold: float (0.0–1.0)`, `dispatch_timeout_ms: 150`
- Implementiert einen gewichteten Graphen (`nx.DiGraph`) aus Intent-Knoten. Jeder Knoten besitzt ein `weight`-Attribut, das durch ein vortrainiertes LightGBM-Modell dynamisch angepasst wird.
- **Technische Metrik:** Dispatch-Latenz < 200 ms bei bis zu 50 parallelen Intents.

### 2. PostgreSQL-gestützte Zustandsmaschine mit Rollback-Fähigkeit
- **Parameter:** `state_table: dentai_states`, `rollback_log: dentai_audit`
- Jeder Agenten-Durchlauf wird als Zustandsmaschine modelliert. Zustandsübergänge werden atomar in PostgreSQL mittels `SERIALIZABLE`-Isolation gespeichert.
- **Technische Metrik:** Rollback eines fehlgeschlagenen Zustands in < 50 ms, inklusive Audit-Trail.

### 3. Echtzeit-Kontext-Cache mit TTL-basierter Invalidierung
- **Parameter:** `cache_ttl_seconds: 300`, `max_cache_entries: 10.000`
- Nutzt Redis (`redis-py`) als verteilten Cache. Jeder Cache-Eintrag speichert den serialisierten Kontext eines Agenten-Durchlaufs inklusive Metadaten (Erstellungszeit, letzter Zugriff).
- **Technische Metrik:** Cache-Hit-Rate > 95 % bei 10.000 gleichzeitigen Sitzungen.

## Schritt-für-Schritt-Implementierung

1. **Intent-Dispatch konfigurieren**  
   Setzen Sie `intent_score_threshold` auf `0.75` und `dispatch_timeout_ms` auf `150`. Initialisieren Sie den gewichteten Graphen mit mindestens 3 Intent-Knoten. Validieren Sie die Kantengewichte mittels `nx.is_weighted()`.

2. **PostgreSQL-Zustandsmaschine aufsetzen**  
   Erstellen Sie die Tabelle `dentai_states` mit den Spalten `session_id UUID`, `current_state VARCHAR(64)`, `last_transition TIMESTAMPTZ`. Aktivieren Sie `SERIALIZABLE` Isolation für alle Transaktionen. Fügen Sie einen Trigger hinzu, der jeden Zustandswechsel in `dentai_audit` protokolliert.

3. **Redis-Cache initialisieren**  
   Konfigurieren Sie `cache_ttl_seconds` auf `300` und `max_cache_entries` auf `10.000`. Nutzen Sie `redis.Redis(host='localhost', port=6379, decode_responses=True)`. Implementieren Sie eine LRU-basierte Eviction-Strategie über `maxmemory-policy allkeys-lru`.

## Best Practices für den Produktiveinsatz

- **Monitoring der Zustandsmaschine:** Überwachen Sie die `dentai_audit`-Tabelle auf unerwartete Zustandswechsel. Setzen Sie einen Alarm bei mehr als 3 Rollbacks pro Minute innerhalb eines Session-Fensters.
- **Cache-Vorwärmung:** Führen Sie vor Lastspitzen (z. B. 9:00 Uhr MEZ) einen Batch-Job aus, der die häufigsten Kontexte in den Redis-Cache lädt. Verwenden Sie dazu die `dentai_states`-Historie der letzten 7 Tage.
