## Detaillierter Inhalt

### Was ist dentaiagent?

dentaiagent ist eine verteilte Agenten-Engine, die auf **PostgreSQL + Redis Streams** basiert. Sie orchestriert bis zu 10.000 parallele dentale Entscheidungsprozesse – von der Röntgenbildanalyse (DICOM-konform) bis zur prädiktiven Behandlungsplanung. Jeder Agent arbeitet asynchron, speichert Zustände in **JSONB-Spalten** und führt Queries mit **millisekundengenauen Timestamps** aus.

### Kernfunktionen

1. **Multi-Modal-Agenten-Pool mit dynamischem Load-Balancing**  
   Jeder dentaiagent verwaltet eigene **Embedding-Vektoren** (dim=768) und führt semantische Suche über **pgvector** durch. Die Latenz pro Query liegt bei ≤25ms (p99). Parameter: `--agent-pool-size=128`, `--vector-index=ivfflat`.

2. **Echtzeit-Entscheidungslogik mit CQRS-Pattern**  
   Befehle (Commands) werden über **RabbitMQ** an die Write-Instanz gesendet, während Read-Modelle aus **materialisierten Views** in PostgreSQL gespeist werden. Jeder Befehl durchläuft einen **2-Phase-Commit** mit Rollback-Fähigkeit. Parameter: `--cqrs-mode=strict`, `--read-replica-count=3`.

3. **Selbstheilendes Audit-Log mit Blockchain-Anker**  
   Jede Agentenentscheidung wird in einen **Merkle-Baum** eingetragen und auf **IPFS** (InterPlanetary File System) persistiert. Der Hash wird anschließend in eine **PostgreSQL-Tabelle mit UNIQUE-Constraint** geschrieben. Abweichungen werden automatisch in <500ms detektiert. Parameter: `--audit-depth=12`, `--ipfs-gateway=http://localhost:5001`.

### So verwenden Sie es

1. **Initialisierung der Datenbank-Schemata**  
   Führen Sie `dentaiagent init --db-uri=postgresql://user:pass@localhost:5432/dentai --schema-version=v3.2` aus. Dies erstellt 7 Tabellen (darunter `agents`, `decisions`, `vector_embeddings`) und lädt die Standard-Embedding-Modelle (z. B. `sentence-transformers/all-MiniLM-L6-v2`).

2. **Agenten-Konfiguration und Deployment**  
   Definieren Sie in einer YAML-Datei (`agent-config.yaml`) die Rollen:  
   ```yaml
   agents:
     - role: "radiologie"
       model: "densenet121"
       batch_size: 32
       timeout_ms: 200
   ```  
   Starten Sie dann: `dentaiagent deploy --config=agent-config.yaml --replicas=4`.

3. **Monitoring und Fehlerbehebung**  
   Nutzen Sie `dentaiagent monitor --metrics-port=9090 --log-level=debug`. Alle Agenten-Logs werden in **JSON-Format** an einen **ELK-Stack** (Elasticsearch, Logstash, Kibana) gesendet. Fehlerhafte Entscheidungen werden automatisch in eine Dead-Letter-Queue (DLQ) verschoben und können mit `dentaiagent replay --dlq-queue=errors` erneut verarbeitet werden.

### Beste Praktiken

- **Praxis 1: Hybride Indexierung für Vektorsuche**  
  Kombinieren Sie `ivfflat` mit einem **Product Quantization (PQ)** -Index, um die Speichernutzung um 60 % zu reduzieren, ohne die Abfragegenauigkeit (Recall@10 > 0,95) zu beeinträchtigen. Konfigurieren Sie `--pq-m=16` und `--pq-bits=8`.

- **Praxis 2: Asynchrone Batch-Verarbeitung für Röntgenbilder**  
  Verwenden Sie **Apache Kafka** als Puffer, um DICOM-Dateien in Batches (max. 512 MB) zu gruppieren. Setzen Sie `--batch-window=5s` und `--max-batch-size=64`. Dies reduziert die CPU-Auslastung um 40 % und verhindert Timeouts bei der Inferenz.
