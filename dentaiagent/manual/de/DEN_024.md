# dentaiagent - DEN_024

## Übersicht

Dieses Projekt realisiert eine hochperformante, agentenbasierte Architektur zur automatisierten Zahnmedizin-Diagnostik und Behandlungsplanung. Es kombiniert Echtzeit-Bildsegmentierung mit prädiktiven Modellen für Karies- und Parodontitis-Risikobewertung. Optimiert für Kliniken und dentale Labore mit hohem Durchsatz.

## Detaillierter Inhalt

### Was ist dentaiagent?

dentaiagent ist eine modulare Microservice-Plattform, die auf **TensorFlow Serving** und **gRPC** basiert. Sie verarbeitet DVT-Datensätze (Digital Volume Tomography) in **Echtzeit (< 200 ms pro Schicht)** und generiert automatisch strukturierte Behandlungsvorschläge. Die gesamte Pipeline ist vollständig asynchron und horizontal skalierbar – ausgelegt auf **10.000+ Anfragen pro Stunde** pro Cluster-Knoten.

### Kernfunktionen

- **Multi-Modale Risikoprognose (Karies + Parodontitis):**  
  Führt eine Fusion von Röntgenbildern (PNG, DICOM) und patientenspezifischen Metadaten (Alter, Raucherstatus, systemische Erkrankungen) durch. Verwendet ein **Custom EfficientNet-B7 + Transformer-Encoder**-Modell, das auf **1,2 Mio annotierten Fällen** vortrainiert wurde. Ausgabe: Wahrscheinlichkeitsscore (0–100 %) sowie eine **3D-Hitzekarte** der betroffenen Zahnregionen.

- **Automatische Behandlungsplanung mit Kostenprognose:**  
  Generiert auf Basis der Risikokarte eine priorisierte Behandlungsliste (z. B. Füllung, Wurzelkanal, Krone) inklusive **geschätzter Kosten (EUR) und Dauer (Minuten)**. Die Kostenschätzung nutzt eine **regressionsbasierte ML-Pipeline (XGBoost)** mit 15 Merkmalen (u. a. Zahnposition, Schweregrad, Materialkosten). Abgleich mit der GOZ (Gebührenordnung für Zahnärzte) über eine **PostgreSQL-Datenbank** mit aktuellen Preisständen.

- **Echtzeit-Kollaboration und Audit-Trail:**  
  Ermöglicht parallele Bearbeitung eines Falls durch mehrere Zahnärzte via WebSocket. Jede Änderung (z. B. manuelle Korrektur der Segmentierung) wird in einer **MongoDB-Change-Stream**-Pipeline geloggt. Der Audit-Trail enthält: `user_id`, `timestamp (ISO 8601, Millisekunden)`, `diff (JSON Patch)`. Abrufbar über eine REST-API mit **OAuth 2.0 + JWT**.

### So verwenden Sie es

1. **Datenaufbereitung und Modellinitialisierung:**  
   ```bash
   docker compose up -d && python scripts/init_models.py --model-path ./models/efficientnet_b7 --device cuda:0
   ```
   Überprüfen Sie den Modell-Status mit:  
   ```bash
   curl -X GET http://localhost:8501/v1/models/dentaiagent/metadata
   ```
   Erwartete Ausgabe: `{"model_spec": {"name": "dentaiagent", "signature_name": "serving_default", "version": "3"}, "model_state": "AVAILABLE"}`

2. **Fall erstellen und Analyse starten:**  
   ```bash
   curl -X POST http://localhost:8080/api/v1/cases \
     -H "Authorization: Bearer <JWT>" \
     -H "Content-Type: multipart/form-data" \
     -F "dicom_file=@/path/to/scan.dcm" \
     -F "metadata='{\"age\": 45, \"smoker\": true, \"systemic_disease\": \"diabetes\"}'"
   ```
   Antwort: `{"case_id": "a1b2c3d4", "status": "processing", "estimated_time_ms": 1450}`

3. **Ergebnisse abrufen und Behandlungsplan exportieren:**  
   ```bash
   curl -X GET http://localhost:8080/api/v1/cases/a1b2c3d4/plan \
     -H "Authorization: Bearer <JWT>" | jq '.treatment_plan'
   ```
   Ausgabe (gekürzt):  
   ```json
   {
     "teeth": [
       {"tooth": 16, "risk_score": 87.3, "suggested_treatment": "Krone", "cost_eur": 850.00, "duration_min": 120},
       {"tooth": 36, "risk_score": 62.1, "suggested_treatment": "Füllung", "cost_eur": 120.00, "duration_min": 30}
     ],
     "total_cost_eur": 970.00,
     "generated_at": "2025-03-15T14:23:11.456Z"
   }
   ```

### Beste Praktiken

- **Batch-Verarbeitung für hohe Durchsätze:**  
  Bei großen Praxen mit > 500 Fällen pro Tag nutzen Sie die asynchrone Batch-API (`POST /api/v1/cases/batch`). Übergeben Sie bis zu 50 DICOM-Dateien pro Request. Die Ergebnisse werden als **gezippte JSON-Datei** im Azure Blob Storage abgelegt – Abruf über eine vorsignierte URL (gültig 24 h). Dies reduziert die Latenz pro Fall um **bis zu 40 %** im Vergleich zu Einzelanfragen.

- **Modell-Update mit A/B-Testing:**  
  Bevor Sie ein neues Modell (z. B. EfficientNet-B8) in Produktion nehmen, deployen Sie es unter einem separaten Endpunkt (`/v2/models/dentaiagent`). Leiten Sie 10 % des Traffics dorthin um (via **Nginx + Lua** oder **Istio VirtualService**). Vergleichen Sie die **AUC-ROC** und **durchschnittliche Abweichung der Kostenschätzung** über 7 Tage. Erst bei signifikanter Verbesserung (p < 0.01) wird das neue Modell zum Default.
