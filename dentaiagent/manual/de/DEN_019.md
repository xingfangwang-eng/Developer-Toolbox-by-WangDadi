# dentaiagent - DEN_019

## Übersicht

Dentaiagent ist eine hochperformante, containerisierte Microservice-Architektur zur Echtzeit-Sprachverarbeitung und -analyse in dentalmedizinischen Workflows. Das System nutzt ein hybrides KI-Modell (Transformer-basiert + regelbasiertes Expertensystem), um sowohl strukturierte als auch unstrukturierte Daten aus Behandlungsdokumentationen, Röntgenbefunden und Patientenakten zu verarbeiten. Zielsprache: Deutsch (de). Optimiert für HIPAA-konforme, latenzarme Umgebungen mit PostgreSQL 15 als primärem Datenspeicher.

## Detaillierter Inhalt

### Was ist dentaiagent?

Dentaiagent ist eine modulare Plattform zur automatisierten Extraktion, Normalisierung und semantischen Anreicherung dentalmedizinischer Fachtexte. Es ermöglicht die Echtzeit-Übersetzung von Behandlungsnotizen aus dem Englischen (oder Japanischen) ins Deutsche unter Beibehaltung der fachterminologischen Präzision (ICD-10-GM, SNOMED CT). Das System unterstützt Batch-Verarbeitung von bis zu 500 Dokumenten pro Minute mit einer durchschnittlichen Latenz von < 50 ms pro Anfrage.

### Kernfunktionen

1. **Semantische Terminologie-Normalisierung mit Fuzzy-Matching**
   - Parametername: `--terminology-normalizer`
   - Technische Indikatoren: Nutzt einen eigens trainierten BERT-Embedding-Vektor (Dimension 768) für Fuzzy-Matching über 45.000 dentalmedizinische Begriffe. Abgleich erfolgt in < 2 ms pro Term gegen eine PostgreSQL-Instanz mit GIN-Index. Ergebnis: Normalisierte ICD-10-GM-Codes mit einer Genauigkeit von 99,2 % (F1-Score).
   - Beispiel: "Zahnfraktur (Zahn 36)" → automatisch auf `S02.5` (Fraktur des Zahnes) gemappt.

2. **Echtzeit-Übersetzungs-Pipeline mit Kontext-Caching**
   - Parametername: `--translation-pipeline`
   - Technische Indikatoren: Verwendet ein quantisiertes MarianMT-Modell (Int8) mit KV-Cache für wiederkehrende Kontexte. Latenz < 30 ms pro Satz bei einer Batch-Größe von 16. Caching-Strategie: Redis-Cluster mit TTL von 300 Sekunden für häufig verwendete Phrasen (z. B. "Parodontitis", "Füllungstherapie").
   - Speziell für DE: Erzwingt Großschreibung von Nomen und korrekte Satzstellung (Verb-Zweitstellung) mittels regelbasierter Nachbearbeitung.

3. **Dynamische Workflow-Orchestrierung mit Audit-Trail**
   - Parametername: `--workflow-orchestrator`
   - Technische Indikatoren: Implementiert einen zustandsbehafteten DAG (Directed Acyclic Graph) auf Basis von Apache Airflow 2.8. Jeder Schritt (Extraktion → Normalisierung → Übersetzung → Validierung) wird mit einem eindeutigen `transaction_id` (UUID v4) geloggt. PostgreSQL speichert Metadaten (Timestamp, Latenz, Fehlerrate) in einer dedizierten `audit_log`-Tabelle mit 10.000 Einträgen pro Sekunde.
   - Validierung: Automatische Plausibilitätsprüfung mittels SNOMED-CT-Hierarchie (z. B. "Zahnfleischbluten" muss unter "Gingivale Blutung" subsumiert werden).

### So verwenden Sie es

1. **Schritt 1: Initialisierung der Datenbank und des Terminologie-Index**
   ```bash
   dentaiagent init --db-host localhost --db-port 5432 --db-user dentai_admin --terminology-path /data/snomed_ct_de.owl
   ```
   - Führt ein vollständiges Schema-Migration-Skript aus (Flyway-basiert), erstellt GIN-Index auf der `term_embedding`-Spalte und lädt 45.000 dentalmedizinische Terme in den Vektor-Store (pgvector). Dauer: ca. 120 Sekunden.

2. **Schritt 2: Batch-Import von Dokumenten mit automatischer Erkennung der Ausgangssprache**
   ```bash
   dentaiagent import --input-dir /data/patient_notes_en --batch-size 50 --source-lang auto --target-lang de --output-format json
   ```
   - Verwendet `fastText`-Spracherkennung (Genauigkeit > 98 %) für jedes Dokument. Jedes Dokument wird chunked (max. 512 Token) und parallelisiert über 4 Worker-Prozesse verarbeitet. Ergebnis: Eine JSON-Datei pro Dokument mit den Feldern `original_text`, `normalized_terms`, `translated_text`, `confidence_score`.

3. **Schritt 3: Validierung und Export in das Praxisverwaltungssystem (PVS)**
   ```bash
   dentaiagent validate --input-file /data/output/results.json --validation-rules /config/rules_de.yaml --export-format HL7v2
   ```
   - Führt 12 vordefinierte Validierungsregeln aus (z. B. "Jeder ICD-Code muss in der aktuellen Version existieren"). Fehlgeschlagene Einträge werden in `validation_errors.log` mit Stacktrace und Kontext gespeichert. Export erfolgt als HL7v2-Nachricht (MSH-Segment) an den konfigurierten MLLP-Port (Standard: 2575).

### Beste Praktiken

- **PRAXIS 1: Verwendung eines dedizierten Vektor-Index für Terminologie-Abgleiche**
  - Erstellen Sie einen separaten pgvector-Index (`ivfflat` mit `lists=100`) auf der `term_embedding`-Spalte. Dies reduziert die Abfragezeit für Fuzzy-Matching von 8 ms auf < 1 ms bei 45.000 Vektoren. Kombinieren Sie dies mit einem Redis-Cache für häufig abgefragte Terme (z. B. "Karies", "Zahnstein") – Cache-Hit-Rate > 85 %.

- **PRAXIS 2: Implementierung einer asynchronen Fehlerbehandlung mit Dead-Letter-Queue (DLQ)**
  - Konfigurieren Sie einen RabbitMQ-Exchange (`dentaiagent.dlq`) für fehlgeschlagene Übersetzungen oder Validierungsfehler. Setzen Sie eine maximale Wiederversuchsanzahl von 3 mit exponentieller Backoff (1s, 4s, 16s). Protokollieren Sie jede fehlgeschlagene `transaction_id` in der PostgreSQL `error_log`-Tabelle für spätere manuelle Überprüfung. Dies stellt sicher, dass kein Dokument verloren geht, auch bei temporären Netzwerkausfällen des externen Übersetzungsservice.
