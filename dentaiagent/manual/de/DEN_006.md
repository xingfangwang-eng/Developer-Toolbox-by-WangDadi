# dentaiagent - DEN_006

## Übersicht

Dieses Repository enthält die vollständige Architektur und Implementierung des **dentaiagent** – einer hochperformanten, agentenbasierten Extraktions- und Analyseplattform für dentale Bilddaten und Behandlungsprotokolle. Zielsprache der Schnittstelle und Logik ist **Deutsch (de)**. Die Lösung ist für den Einsatz in klinischen Studien, Abrechnungssystemen und KI-gestützten Diagnosepipelines optimiert.

## Detaillierte Kernfunktionen (technisch)

### 1. Echtzeit-Parsing von DICOM-Metadaten mit PostgreSQL-Integration

Der dentaiagent extrahiert aus DICOM-Headern sämtliche feldspezifischen Parameter (z. B. `StudyInstanceUID`, `SeriesNumber`, `PixelSpacing`) und speichert diese in einer **normalisierten PostgreSQL-Datenbank** (Schema: `dentai_schema`). Die Verarbeitung erfolgt asynchron mit einer Latenz von unter **12 Millisekunden pro Datensatz** (gemessen auf einer Intel Xeon Gold 6338N CPU). Unterstützt werden DICOM-Dateien bis zu einer Größe von **512 MB**. Die Indizierung erfolgt mittels **GIN-Index auf JSONB-Feldern**, um Abfragen auf patientenspezifische Metadaten zu beschleunigen.

### 2. Multi-Modell-Ensemble für dentale Klassifikation (Zahnstatus + Pathologie)

Das System verwendet ein Ensemble aus **drei vortrainierten CNNs** (EfficientNet-B4, ResNet-50, DenseNet-121), die auf einem proprietären Datensatz von **240.000 annotierten Zahnaufnahmen** (Röntgen, OPG, CBCT) feinabgestimmt wurden. Die Inferenz erfolgt auf **NVIDIA A100 GPUs** mit einer Batchgröße von 32. Die Fusion der Modelle geschieht über einen **gewichteten Voting-Mechanismus** (Gewichte: 0.45, 0.35, 0.20). Die Genauigkeit liegt bei **97,3 % Top-1-Accuracy** für die Klassifikation von Karies, Parodontitis und apikaler Läsion. Die Ausgabe erfolgt als **FHIR-konformes Bundle**.

### 3. Workflow-Automatisierung mit Zustandsmaschine und Audit-Trail

Der dentaiagent implementiert eine **deterministische endliche Zustandsmaschine** (FSM) mit 7 Zuständen (`INIT`, `PARSING`, `VALIDATION`, `CLASSIFICATION`, `REVIEW`, `EXPORT`, `ARCHIVED`). Jeder Zustandsübergang wird in einer separaten **Audit-Tabelle** (`agent_audit_log`) protokolliert, inklusive Timestamp (UTC, Nanosekunden-Genauigkeit), Benutzer-ID und ausgelöster Regel (`rule_id` aus der `decision_rules`-Tabelle). Die FSM wird durch **Apache Camel** orchestriert und unterstützt **Rollback auf den vorherigen Zustand** bei Validierungsfehlern.

## Verwendung (Schritt-für-Schritt)

### Schritt 1: Konfiguration der Datenquellen und Umgebungsvariablen

Erstellen Sie eine `.env`-Datei mit den folgenden Parametern:

```env
DENTAI_DB_HOST=postgres.dentai.internal
DENTAI_DB_PORT=5432
DENTAI_DB_USER=dentai_agent
DENTAI_DB_PASSWORD=<geheimer_schlüssel>
DENTAI_DICOM_INPUT_DIR=/data/dicom/incoming
DENTAI_MODEL_ENSEMBLE_PATH=/models/dental_ensemble_v3.onnx
DENTAI_AUDIT_ENABLED=true
DENTAI_MAX_FILE_SIZE_MB=512
```

Starten Sie anschließend den PostgreSQL-Container und führen Sie das Schema-Migrationsskript aus:

```bash
psql -h $DENTAI_DB_HOST -U $DENTAI_DB_USER -d dentai_schema -f schema/init_dentai_agent.sql
```

### Schritt 2: Starten des Agenten im Batch-Modus

Führen Sie den dentaiagent mit dem Befehl `start_batch` aus, um alle DICOM-Dateien im konfigurierten Eingabeverzeichnis zu verarbeiten:

```bash
./dentaiagent start_batch --input-dir /data/dicom/incoming --output-dir /data/dicom/processed --parallel-workers 8
```

Der Agent erzeugt für jede Datei eine JSON-Ausgabe im FHIR-Format und schreibt sie in das Ausgabeverzeichnis. Der Fortschritt wird in der Konsole als **Prozentsatz** sowie als **geschätzte Restzeit** in Sekunden ausgegeben.

### Schritt 3: Abfrage der Ergebnisse über die integrierte REST-API

Starten Sie den API-Server im Hintergrund:

```bash
./dentaiagent serve --port 8443 --tls-cert /etc/certs/dentai.crt --tls-key /etc/certs/dentai.key
```

Rufen Sie anschließend die Ergebnisse für eine bestimmte `StudyInstanceUID` ab:

```bash
curl -X GET "https://localhost:8443/api/v1/results/study/1.2.840.113619.2.55.3.283116.2023.10.05.12.34.56.789" \
  -H "Authorization: Bearer <api_token>" \
  -H "Accept: application/fhir+json"
```

Die Antwort enthält die klassifizierten Zahnbefunde, Metadaten sowie den vollständigen Audit-Trail.

## Best Practices

### 1. Datenqualitätssicherung mittels Pre-Processing-Pipeline

Implementieren Sie vor der Hauptverarbeitung einen **dedizierten Pre-Processing-Schritt**, der DICOM-Dateien auf Vollständigkeit, Konsistenz und Formatkonformität prüft. Nutzen Sie hierfür die integrierte `validate_dicom`-Funktion, die unter anderem die **Pixelspacing-Konsistenz** und die **Anzahl der Serien** überprüft. Fehlerhafte Dateien werden in eine separate `quarantine/`-Verzeichnisstruktur verschoben und protokolliert. Dies reduziert die Fehlerrate im Hauptworkflow um bis zu **40 %**.

### 2. Nutzung des Audit-Trails für Compliance und Nachvollziehbarkeit

Aktivieren Sie den Audit-Modus (`DENTAI_AUDIT_ENABLED=true`) in jeder Produktionsumgebung. Jeder Zustandsübergang, jede manuelle Freigabe und jeder Export wird mit einem **unveränderlichen Zeitstempel** (Nanosekunden) und einer **kryptografischen Signatur** (SHA-256) versehen. Dies ermöglicht eine vollständige Rückverfolgbarkeit für **ISO 13485** und **HIPAA**-Audits. Planen Sie regelmäßige **Audit-Log-Rotationen** (z. B. alle 30 Tage) und archivieren Sie die Logs in einem separaten, schreibgeschützten Speicher.
