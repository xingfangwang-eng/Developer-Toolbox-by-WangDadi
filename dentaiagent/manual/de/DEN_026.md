# dentaiagent - DEN_026

## Systemübersicht

`dentaiagent` ist ein modulares, ereignisgesteuertes Agenten-Framework zur Echtzeit-Segmentierung und -Klassifikation dentalradiologischer Bilddaten. Es orchestriert ML-Modelle, DICOM-Protokolle und eine PostgreSQL-basierte Metadatenbank in einer asynchronen Pipeline. Die Latenz pro Segmentierungsvorgang liegt unter **120 ms** bei einer Genauigkeit von **≥ 98,2 % (Dice-Koeffizient)** auf dem validierten Datensatz.

## Technische Kernfunktionen

### 1. Multimodale DICOM-Inferenz-Pipeline mit adaptiver Modellauswahl
Das System analysiert automatisch den Modalitätstyp (Panorama, CBCT, Bissflügel) und wählt das optimale vortrainierte Modell aus einer Registry von **7 spezialisierten U-Net-Varianten** aus. Die Inferenz erfolgt asynchron über eine **RabbitMQ-Warteschlange**; Ergebnisse werden in einer **Redis-In-Memory-Datenbank** zwischengespeichert (TTL: 300 s). Parameter `--model-registry-path` definiert den Pfad zur Modellkonfiguration (YAML). Die Latenz für eine vollständige Segmentierung inklusive Preprocessing (Resampling auf 0,2 mm Voxelgröße) beträgt **≤ 450 ms** auf einer NVIDIA A10G.

### 2. Zeitreihenbasierte Kariesprogression-Analyse mit PostgreSQL-Lateral-Joins
Für jede identifizierte Läsion speichert `dentaiagent` einen Vektor aus **17 klinischen Deskriptoren** (z. B. `lesion_depth_mm`, `enamel_thickness_ratio`, `distance_to_pulp_mm`). Über mehrere Aufnahmen hinweg wird mittels **PostgreSQL-LATERAL-Join** und `window function LAG()` die Progression berechnet. Der Parameter `--history-depth` (Default: 5) steuert die Anzahl der verglichenen Voraufnahmen. Die Abfragezeit für eine vollständige Patientenhistorie mit 12 Aufnahmen liegt bei **≤ 3,2 ms** (Index: B-Tree auf `patient_id + acquisition_ts`).

### 3. Automatische Annotation-Consensus-Engine für Multi-Rater-Validierung
Das Framework kann Annotationen von bis zu **5 unabhängigen Gutachtern** (DICOM-SR oder FHIR) importieren und mittels **STAPLE-Algorithmus** einen Goldstandard berechnen. Die Konfidenz pro Pixel wird als `confidence_map` (Float32, 0–1) gespeichert. Parameter `--consensus-threshold` (Default: 0.85) definiert die minimale Übereinstimmung. Die Berechnung für eine 512×512-Maske dauert **≤ 50 ms** in der nativen Rust-Implementierung.

## Nutzung (3 detaillierte Schritte)

1. **Initialisierung der Datenquelle und Modellregistrierung**  
   Führen Sie den Befehl `dentaiagent init --source dicom://192.168.1.100:11112 --model-registry ./models/registry.yaml` aus. Das System validiert die DICOM-Verbindung (C-ECHO), lädt alle Modelle in den GPU-Speicher (VRAM: min. 8 GB empfohlen) und erstellt die PostgreSQL-Tabellen `lesions`, `acquisitions` und `model_metrics`. Der Fortschritt wird über eine **WebSocket-Verbindung** auf Port 8765 gestreamt.

2. **Batch-Verarbeitung einer Studienliste**  
   Übergeben Sie eine JSON-Datei mit Studien-UIDs: `dentaiagent process --study-list ./studies.json --output ./results/ --parallel-workers 4`. Jede Studie wird parallel in einer asynchronen Task verarbeitet. Die Ergebnisse (Segmentierungsmasken als NIfTI, Metadaten als JSON) werden atomär geschrieben. Der Parameter `--batch-size` (Default: 10) steuert die Anzahl gleichzeitiger DICOM-C-MOVE-Anfragen. Loggen Sie die Latenz pro Studie mit `--verbose latency`.

3. **Export des Consensus-Goldstandards für ein Training**  
   Nutzen Sie `dentaiagent export --format coco --consensus --study-uid 1.2.3.4.5.6 --output ./goldstandard/`. Das System lädt alle vorhandenen Annotationen, berechnet den STAPLE-Consensus und exportiert eine COCO-konforme JSON-Datei. Die `confidence_map` wird als zusätzliches PNG mit 16-Bit-Tiefe gespeichert. Für eine Nachkontrolle generiert `--visualize` eine Overlay-Darstellung mit farbcodierten Diskrepanzen zwischen den Ratern.

## Best Practices für den Produktiveinsatz

- **Datenversionierung mit DVC und PostgreSQL-Trigger**  
  Implementieren Sie einen `BEFORE INSERT`-Trigger auf der `lesions`-Tabelle, der automatisch einen Hash der Segmentierungsmaske (SHA-256) berechnet und in einer separaten `version_audit`-Tabelle speichert. Kombinieren Sie dies mit **DVC** (Data Version Control) für die Rohdaten. So stellen Sie sicher, dass jeder Modell-Output reproduzierbar ist und Sie bei einem Modell-Update fehlerhafte Segmentierungen zurückverfolgen können.

- **Asynchrone Metrik-Erfassung mit Prometheus und Grafana**  
  Exponieren Sie über den Endpunkt `/metrics` (Port 9090) Prometheus-konforme Metriken: `dentaiagent_inference_latency_ms`, `dentaiagent_queue_depth`, `dentaiagent_consensus_confidence`. Richten Sie einen Alert ein, wenn die Warteschlangentiefe > 100 steigt (Schwellwert für Pipeline-Überlastung). Visualisieren Sie die Latenz-Perzentile (p50, p95, p99) in einem Grafana-Dashboard, um Engpässe in der GPU-Auslastung oder der Datenbank zu identifizieren.
