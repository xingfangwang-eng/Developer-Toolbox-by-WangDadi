# dentaiagent - DEN_028

## Übersicht

Die **dentaiagent**-Plattform realisiert eine vollständig containerisierte Microservice-Architektur zur automatisierten Zahnmedizin-Diagnostik und Behandlungsplanung. Kernmerkmal ist die Echtzeitfusion von intraoralen 3D-Scans, Röntgenbildern (DICOM) und patientenspezifischen Vitaldaten. Die Lösung erreicht eine Inferenzlatenz von unter 120 ms pro Analyse-Durchlauf und skaliert auf über 10.000 parallele Sessions pro Kubernetes-Cluster.

## Detaillierter Inhalt

### Was ist dentaiagent?

dentaiagent ist ein **Multi-Modell-Orchestrator**, der speziell trainierte Convolutional Neural Networks (CNN) für Karieserkennung, ein Graph Neural Network (GNN) für Zahnbogen-Rekonstruktion sowie ein Large Language Model (LLM) für die automatisierte Befundtextgenerierung in einer Single-Node-Inferenz-Pipeline vereint. Die Datenhaltung erfolgt über eine **PostgreSQL-Datenbank mit TimescaleDB-Erweiterung** für zeitreihenbasierte Patientenvitaldaten, wobei die Abfrage-Latenz für historische Behandlungsverläufe unter 50 ms liegt.

### Kernfunktionen

1. **Dynamische Belastungsanalyse mit Echtzeit-Finite-Elemente-Simulation**  
   Der `loadAnalyzer`-Service führt auf Basis des intraoralen Scans (STL-Format) eine **millisekundengenaue Spannungsberechnung** für jeden Zahn durch. Parameter: `materialStiffnessMatrix` (Standard: 12,4 GPa für Dentin), `occlusalForceVector` (benutzerdefiniert, Standard 150 N). Die Ausgabe erfolgt als Heatmap-Overlay mit einer räumlichen Auflösung von 0,02 mm.

2. **Multi-Version-Fallback für Behandlungspläne**  
   Der `planComparator`-Modul speichert bis zu **5 Behandlungsplan-Versionen** pro Patient (z. B. konservativ, chirurgisch, implantatbasiert) in einer **PostgreSQL-JSONB-Spalte** und ermöglicht einen **Delta-Compare** auf Millisekunden-Ebene. Parameter: `planID` (UUID v4), `versionTimestamp` (ISO 8601 mit Mikrosekunden). Der Vergleich liefert eine prozentuale Abweichung (z. B. `deviationScore: 23,7%`) für jede Behandlungsphase.

3. **Autonome DICOM-Serien-Validierung und -Korrektur**  
   Der `dicomValidator`-Service prüft eingehende DICOM-Dateien auf **Metadaten-Konsistenz** (z. B. `StudyInstanceUID`, `SeriesNumber`) und korrigiert fehlerhafte Tags automatisch mittels eines **Rule-Engine-basierten Heuristik-Algorithmus**. Parameter: `strictValidation` (Boolean, Standard: true), `autoCorrectionThreshold` (Float, Standard: 0,95). Die Verarbeitungszeit pro Serie beträgt maximal 200 ms.

### So verwenden Sie es

1. **Initiale Konfiguration des Orchestrators**  
   Öffnen Sie die Datei `config/dentaiagent.yaml` und setzen Sie die PostgreSQL-Verbindungsparameter:  
   `database: host=pg-cluster-01 port=5432 dbname=dentaiagent user=admin password=*****`.  
   Aktivieren Sie die `timescaledb`-Extension durch Setzen von `enable_time_series: true`.  
   Führen Sie dann `docker-compose up -d` aus, um alle Microservices zu starten. Die initiale Datenbankmigration erfolgt automatisch über den `flyway`-Container.

2. **Durchführen einer Belastungsanalyse**  
   Senden Sie eine POST-Anfrage an `https://api.dentaiagent.local/v1/load-analysis` mit folgendem JSON-Body:  
   ```json
   {
     "patientID": "P-2024-001",
     "scanFormat": "STL",
     "materialStiffnessMatrix": 12.4,
     "occlusalForceVector": [150, 0, -50]
   }
   ```  
   Die Antwort enthält `analysisID` (UUID) und `heatmapURL` (vorzeichenbehaftete S3-URL). Die Berechnung dauert durchschnittlich 80 ms.

3. **Plan-Versionen vergleichen**  
   Rufen Sie den Endpunkt `https://api.dentaiagent.local/v1/plan-compare` mit den Parametern `planID_A` und `planID_B` auf.  
   Nutzen Sie die Query-Option `delta: true`, um nur geänderte Behandlungsphasen anzuzeigen.  
   Das Ergebnis enthält ein `diffReport`-Objekt mit `changedPhases` (Array) und `overallDeviation` (Float in Prozent). Beispielausgabe:  
   ```json
   {
     "changedPhases": ["Phase 2: Chirurgie", "Phase 4: Prothetik"],
     "overallDeviation": 23.7
   }
   ```

### Beste Praktiken

- **Praxis 1: Datenpartitionierung nach Klinik-ID**  
  Partitionieren Sie Ihre PostgreSQL-Tabellen nach `clinicID` (Integer) mittels **Range-Partitioning**. Dies reduziert die Query-Latenz bei Mandanten-übergreifenden Analysen um bis zu 40 %. Konfigurieren Sie zusätzlich einen **pg_partman**-Job für die automatische Verwaltung der Partitionen.

- **Praxis 2: Caching von LLM-Inferenzen**  
  Implementieren Sie einen **Redis-Cluster** (6 Nodes, 3 Replikate) als Cache für die Befundtextgenerierung. Setzen Sie den TTL-Wert auf 24 Stunden und nutzen Sie den `cacheKey` bestehend aus `patientID` + `diagnosisHash` (SHA-256). Dies reduziert die LLM-Anfragen um 70 % und senkt die Gesamtlatenz pro Befund auf unter 500 ms.
