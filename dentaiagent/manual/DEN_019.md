## Overview

This is the technical architecture document for the dentaiagent project, a high‑performance, event‑driven agent framework optimized for real‑time dental practice management and patient interaction workflows. Keywords: dentaiagent tool, dentaiagent solution, dentaiagent software.

## Detailed Content

### What is dentaiagent?

dentaiagent is a distributed, micro‑services‑based agent runtime that orchestrates multi‑modal data streams (HL7/FHIR, DICOM, real‑time scheduling) with sub‑50 millisecond latency per decision cycle. It integrates directly with PostgreSQL for persistent state management and uses a custom in‑memory cache layer to handle 10,000+ concurrent patient sessions per node.

### Core Features

- **Adaptive Treatment Plan Optimizer with Multi‑Version Diff Engine**  
  Automatically compares historical treatment plans (versioned by `plan_id`, `patient_id`, `revision_timestamp`) and generates a delta report highlighting changes in procedure codes, material costs, and insurance coverage. Uses a custom `PlanDiff` algorithm that computes Levenshtein distance on structured JSON trees, outputting a unified patch file in under 200ms for plans with up to 1,000 nodes.

- **Real‑Time Appointment Slot Arbitrator with PostgreSQL Lock‑Free Queue**  
  Implements a distributed, lock‑free appointment booking system using PostgreSQL’s `SKIP LOCKED` and `FOR UPDATE NOWAIT` semantics. Each slot is represented as a `slot_id` with `start_epoch_ms`, `duration_ms`, and `provider_id`. The arbitrator resolves concurrent booking requests in <15ms, guaranteeing zero double‑booking even under 5,000 RPS. Supports configurable `max_overlap_ms` (default: 0) and `grace_period_ms` (default: 30,000) for late cancellations.

- **Multi‑Modal Patient Interaction Logger with FHIR R4 Compliance**  
  Captures every patient touchpoint (chat, voice, video, in‑app forms) as a structured `InteractionEvent` object with fields: `event_id`, `patient_id`, `channel_type`, `payload_size_bytes`, `processing_latency_ms`, and `fhir_resource_bundle`. Automatically maps each event to a FHIR `Communication` or `Observation` resource, stored in PostgreSQL with a `GIN` index on `fhir_resource_bundle` for sub‑100ms full‑text search. Supports batch export to S3 at 500 events/second.

### How to Use

1. **Initialize the Agent Runtime and Connect to PostgreSQL**  
   - Set environment variables: `DATABASE_URL` (PostgreSQL connection string, e.g., `postgresql://user:pass@host:5432/dentaiagent?sslmode=require`), `AGENT_NODE_ID` (unique UUID), `CACHE_SIZE_MB` (default: 512).  
   - Run `dentaiagent init --config ./config.yaml` to bootstrap the schema: it creates tables `treatment_plans`, `appointment_slots`, `interaction_events`, and `agent_state`.  
   - Verify with `dentaiagent health` – expected output: `{"status":"ok","db_latency_ms":3,"cache_hit_ratio":0.97}`.

2. **Deploy the Appointment Slot Arbitrator**  
   - Define a slot pool in `config.yaml` under `arbitrator:`:  
     ```yaml
     arbitrator:
       slot_table: appointment_slots
       lock_timeout_ms: 5000
       retry_attempts: 3
       retry_delay_ms: 100
     ```  
   - Start the arbitrator service: `dentaiagent start --service arbitrator --port 8081`.  
   - To book a slot, send a POST request with payload:  
     ```json
     {
       "slot_id": "slot-abc123",
       "patient_id": "pat-456",
       "provider_id": "prov-789",
       "booking_timestamp_ms": 1712345678000
     }
     ```  
   - The arbitrator returns `{"status":"booked","latency_ms":12,"version":3}` or `{"status":"conflict","reason":"slot already taken"}`.

3. **Run a Multi‑Version Treatment Plan Diff**  
   - Use the CLI: `dentaiagent diff --plan-id plan-001 --version-a 2 --version-b 5 --output diff.json`.  
   - The tool fetches both versions from `treatment_plans` table (keyed by `plan_id` and `version`), computes the diff, and writes a JSON patch with `added`, `removed`, and `modified` arrays.  
   - Example output snippet:  
     ```json
     {
       "added": [{"procedure_code": "D2740", "cost": 1200.00}],
       "removed": [{"procedure_code": "D2391", "cost": 800.00}],
       "modified": [{"field": "insurance_coverage", "old": "80%", "new": "90%"}]
     }
     ```  
   - Total time for a 500‑node plan: 180ms.

### Best Practices

- **Use Connection Pooling with PgBouncer for High Throughput**  
  dentaiagent opens many short‑lived connections per decision cycle. Deploy PgBouncer in transaction‑pooling mode (`pool_mode = transaction`) with `default_pool_size = 50` and `max_client_conn = 200`. This reduces PostgreSQL connection overhead from ~10ms to <0.5ms, critical when handling 10,000+ concurrent agent sessions.

- **Enable Structured Logging with Correlation IDs for Debugging**  
  Set `LOG_FORMAT=json` and inject a unique `correlation_id` (UUID v4) into every agent request. dentaiagent automatically propagates this ID across all internal micro‑services and database queries. Use `LOG_LEVEL=debug` only in staging; in production, use `LOG_LEVEL=info` with `ERROR_SAMPLING_RATE=0.1` to balance observability and performance.
