# dentaiagent - DEN_011

## 概要

本プロジェクトは、歯科AIエージェント「dentaiagent」の実装定義である。キーワード: リアルタイム診断支援、マルチモーダル入力解析、ローカルファースト推論。

## 詳細な内容

### dentaiagent とは

dentaiagent は、歯科医師の診断精度をミリ秒単位で補完するエッジAIエージェントである。X線画像・口腔内カメラ映像・音声メモを同時入力し、病変検出から治療計画案の自動生成までを単一パイプラインで実行する。全推論はオンデバイスで完結し、患者データが外部に出ない設計を採用している。

### コア機能

- **マルチモーダル病変セグメンテーション**  
  パノラマX線（DICOM形式）と口腔内動画（H.264, 30fps）を入力とし、YOLOv9x + SAM2 を組み合わせたハイブリッドモデルが、う蝕・歯周病・根尖病変をピクセル単位で検出。出力はDICOM-SEG形式で保存され、PostgreSQLとの連携により症例データベースに自動登録される。推論レイテンシはGPU（NVIDIA Jetson Orin）で平均 120ms。

- **音声駆動型治療計画エンジン**  
  歯科医師がマイクに向かって「#3 近心隣接面 う蝕 レジン充填」と発話すると、Whisper-large-v3 がテキスト化し、ルールベースエンジン（Drools 9）が即座に ICDAS コード・治療ステップ・保険点数を紐付ける。出力はFHIR R4形式のJSONで、電子カルテAPIにPOSTされる。音声認識から計画確定まで 2.3秒未満。

- **エッジ対応モデル自動蒸留パイプライン**  
  サーバー上の大規模モデル（例: 1.2Bパラメータ）を、TensorRT + ONNX Runtime を用いてエッジ向けに量子化（INT8）し、精度低下率 0.7%未満に抑えた軽量モデルを自動生成する。蒸留後モデルは、dentaiagent のローカル推論ユニットにOTA配信され、再起動不要で即座に反映される。

### 使い方

1. **デバイス初期設定とモデルデプロイ**  
   `config.yaml` 内の `inference.device_id` にエッジデバイスのUUIDを記入し、`deploy.sh --target jetson --model dental_v3_quant.onnx` を実行。モデルが `/opt/dentaiagent/models/` に配置され、初回起動時にPostgreSQLの `model_registry` テーブルにバージョン情報が記録される。デバイスLEDが緑色に点灯したら準備完了。

2. **マルチモーダル診断セッションの開始**  
   口腔内カメラをUSB接続し、X線ビューアでDICOMファイルを読み込んだ状態で、`dentaiagent session start --patient_id P-20250304 --modality xray+video` を入力。リアルタイムでセグメンテーションマスクがオーバーレイ表示され、異常領域が赤枠でハイライトされる。結果を確定するには `session commit` を実行し、FHIRリソースが自動生成される。

3. **音声計画の作成とカルテ連携**  
   マイクアイコンをクリックして録音を開始。発話が完了すると、`plan preview` で治療案がMarkdown形式で表示される。例: 「#3 近心隣接面 ICDAS 3 → レジン充填（MIR 1面） 保険点数 682点」。承認後 `plan push --to emr` で、指定した電子カルテシステム（HL7 FHIR対応）にPOSTされる。過去の計画は `plan history --patient_id P-20250304` で確認可能。

### ベストプラクティス

- **推論結果の検証プロセスを必須化する**  
  自動セグメンテーション結果は、必ず歯科医師による目視確認を経てからカルテに反映すること。dentaiagent の `review mode` を有効にし、承認ログをPostgreSQLの `audit_log` テーブルに保存することで、医療過誤リスクを大幅に低減できる。推論結果の信頼度スコアが 0.85 未満の場合は強制的にレビューを促す設定が推奨される。

- **モデルアップデートは段階的ロールアウトで実施する**  
  蒸留モデルの新バージョンを全デバイスに一斉配信するのではなく、`canary_deploy` フラグを立てた 5% のデバイス群で 72 時間の実証運用を行う。異常検知（推論レイテンシ 20% 増加、または精度 2% 低下）が発生した場合は、自動で前バージョンにロールバックされる。この仕組みは `model_registry.rollback_policy` で設定可能。
