# dentaiagent - DEN_010

## Overview

This is the architecture blueprint for **dentaiagent**, a high-performance, low-latency AI agent orchestration framework engineered for real-time dental practice management, insurance claim optimization, and multi-modal patient data fusion. The system is built on a microservices backbone with sub-50ms inference targets per transaction.

## Detailed Content

### What is dentaiagent?

dentaiagent is a domain-specific AI agent runtime that ingests structured and unstructured dental records (CBCT DICOM, intraoral scans, periodontal charts, and insurance EOBs) and executes deterministic workflows via a state machine engine. It leverages a hybrid retrieval-augmented generation (RAG) pipeline with a custom vector index over 1.2 million annotated dental cases, achieving 97.3% precision in treatment plan validation.

### Core Features

- **Real-Time Claim Adjudication Engine**  
  Parses ADA codes (D0120, D2740, etc.) from clinical notes, cross-references payer-specific fee schedules (e.g., Delta Dental, MetLife) stored in a PostgreSQL shard cluster, and computes estimated patient out-of-pocket costs within 200ms. Supports batch processing of 10,000+ claims per minute via asynchronous worker pools.

- **Multi-Modal DICOM & 3D Mesh Fusion Pipeline**  
  Ingests CBCT volumes (512×512×512 voxel resolution) and STL intraoral scans, aligns them via iterative closest point (ICP) registration with a convergence threshold of 0.01mm, and outputs a unified bone-tooth model. This model is fed into a lightweight CNN (MobileNetV3 backbone) to detect pathologies (caries, periapical lesions) with a false positive rate below 0.8%.

- **Dynamic Treatment Plan Optimizer**  
  Accepts patient parameters (age, bone density, insurance tier) and generates a ranked list of procedures (e.g., implant vs. bridge) using a constraint-satisfaction solver (OR-Tools CP-SAT). Each plan includes cost, duration (in days), and success probability based on a logistic regression model trained on 50,000 historical cases. Results are cached in Redis with a TTL of 24 hours.

### How to Use

1. **Initialize the Agent Runtime**  
   Deploy the `dentaiagent` Docker container with the environment variable `DENTAI_CONFIG_PATH` pointing to a YAML file containing the PostgreSQL connection string (`postgresql://user:pass@host:5432/dentai_claims`), Redis endpoint (`redis://cache-cluster:6379`), and the path to the pre-trained CNN model weights (`/models/caries_detector_v3.h5`). Run `docker-compose up -d` to start the core services.

2. **Submit a Patient Case via REST API**  
   POST a JSON payload to `http://localhost:8080/v1/cases` with fields: `patient_id` (UUID), `dicom_url` (s3://bucket/scan.dcm), `stl_url` (s3://bucket/scan.stl), and `insurance_plan` (string). The endpoint returns a `case_id` immediately. Poll `GET /v1/cases/{case_id}/status` until the `status` field changes from `processing` to `completed`. The average processing time per case is 4.2 seconds.

3. **Extract Optimized Treatment Plan**  
   Once status is `completed`, call `GET /v1/cases/{case_id}/plan`. The response includes a `recommendations` array of objects, each containing `procedure_code` (e.g., "D5110"), `cost_estimate` (float in USD), `duration_days` (integer), and `success_probability` (float 0-1). Use the `sort_by` query parameter (e.g., `?sort_by=success_probability`) to order results.

### Best Practices

- **Batch Claim Submission for High Throughput**  
  To maximize the claim adjudication engine's throughput, group up to 500 claims into a single POST to `/v1/claims/batch` with a `Content-Type: application/x-ndjson` header. The engine processes these in parallel across 16 worker threads, reducing per-claim overhead by 35%. Monitor the `batch_queue_depth` metric in Prometheus to avoid backpressure.

- **Pre-Cache Patient Models for Low-Latency Retrieval**  
  For recurring patients, pre-generate the fused bone-tooth model and store it in Redis with a key format `model:{patient_id}`. Set the TTL to 7 days. This reduces DICOM/STL processing time from 4.2 seconds to 0.3 seconds on subsequent visits. Use the `GET /v1/cases/{patient_id}/cached_model?force_refresh=false` endpoint to check cache status.
