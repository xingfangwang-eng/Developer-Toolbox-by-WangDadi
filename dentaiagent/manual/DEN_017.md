### What is dentaiagent?

dentaiagent is a professional tool that helps solve problems in related fields.

### Core Features

- **Multi-Threaded Latency Profiler**: Captures and visualizes microsecond-level execution delays across distributed agent pipelines. Uses `--latency-threshold=500` (in µs) to flag bottlenecks, with real-time output to PostgreSQL for historical trend analysis.
- **Adaptive Prompt Injection Shield**: Dynamically sanitizes user input against 27 known adversarial patterns (e.g., `role_override`, `context_leak`). Employs a sliding window entropy check (`entropy_delta > 0.35`) to detect obfuscated attacks, logging all blocked attempts to `shield_audit` table.
- **Autonomous Task Graph Optimizer**: Recursively decomposes complex agent tasks into DAG nodes, then applies a topological sort with `max_concurrent_nodes=8`. Supports `--optimization-mode=throughput` or `--optimization-mode=latency`, outputting a JSON execution plan with estimated wall-clock time per node.

### How to Use

1. **Deploy the Latency Profiler**: Run `dentaiagent profiler --target=ws://localhost:8765 --interval=10` to attach to your agent WebSocket. Specify `--output=postgresql://user:pass@host:5432/dentai_metrics` to stream latency snapshots every 10ms. Monitor the `latency_99th_percentile` field; if it exceeds 2000µs, the profiler automatically emits a `CRITICAL` log event.
2. **Configure the Shield**: Edit `config/shield.yml` and set `block_on_high_entropy: true`, `max_context_window: 4096`. Then execute `dentaiagent shield --config=config/shield.yml --dry-run` to test against a sample payload file. Review the generated `shield_report.json` for false positives; adjust the `entropy_threshold` parameter (default 0.35) accordingly.
3. **Run the Task Optimizer**: Feed a JSON task specification via `dentaiagent optimize --input=tasks.json --mode=throughput`. The tool returns an optimized DAG as `optimized_dag.json`. Use `--visualize` to render a Mermaid flowchart directly in the terminal. For iterative tuning, run `--compare` against a previous `optimized_dag.json` to see delta in estimated completion time.

### Best Practices

- **Baseline Before Tuning**: Always run the Latency Profiler in a staging environment with production-like load for at least 24 hours. Use the collected `latency_p50` and `latency_p99` as baseline thresholds. Never adjust `--latency-threshold` below the p50 value, as this generates excessive alerts without actionable insight.
- **Shield Tuning via A/B Testing**: Deploy the Adaptive Shield in `--log-only` mode for one week. Analyze the `shield_audit` table to identify false positive patterns (e.g., legitimate code snippets flagged as `context_leak`). Then adjust the `entropy_threshold` or add custom whitelist regex patterns before enabling blocking mode.
