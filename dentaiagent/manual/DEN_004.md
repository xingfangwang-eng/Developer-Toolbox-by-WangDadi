## Overview

This document details the architecture and operational workflow for `dentaiagent`, a distributed agent orchestration framework designed for real-time dental imaging inference, patient record fusion, and multi-modal diagnostic pipeline management. The system is optimized for sub-200ms inference-to-action latency in clinical environments.

## Detailed Content

### What is dentaiagent?

`dentaiagent` is a domain-specific agent runtime that coordinates edge-based CBCT segmentation, intraoral video stream processing, and FHIR-compliant record mutation through a declarative YAML pipeline. It decouples hardware I/O from diagnostic logic, enabling hot-swappable model backends (ONNX, TensorRT, CoreML) without downtime.

### Core Features

- **Real-Time CBCT Slice Fusion with Adaptive Tile Scheduling**  
  Accepts `.dcm` series via a gRPC stream, applies a 3D U-Net variant (`dentai-v3`) with mixed-precision inference (FP16), and outputs per-slice segmentation masks in under 180ms per volume (512x512x256). The `--tile-stride` parameter controls overlap ratio (default: 0.25) to minimize boundary artifacts during reconstruction.

- **Multi-Source Patient Record Reconciliation via FHIR R5 Delta Sync**  
  Maintains an in-memory delta graph (`dentai-graph`) that merges HL7v2 ADT messages, DICOM metadata, and clinical notes into a single FHIR `Patient` resource. Uses `--sync-interval-ms` (default: 5000) to poll PostgreSQL for record mutations, with automatic conflict resolution via timestamp-vector clocks.

- **Agentic Workflow Engine with Retry & Circuit Breaker**  
  Each diagnostic step (e.g., caries detection, implant planning) is defined as a stateful `AgentNode` in a DAG. The engine supports `--max-retries` (default: 3) and `--circuit-breaker-threshold` (default: 5 failures within 60s) to prevent cascade failures. Node outputs are cached in Redis with TTL `--cache-ttl-ms` (default: 300000).

### How to Use

1. **Initialise the Agent Runtime with Custom Model Path**  
   ```bash
   dentaiagent init --model-path ./models/dentai-v3.onnx --device cuda:0 --tile-stride 0.25
   ```
   This command loads the segmentation model into VRAM, configures the tile scheduler, and opens a gRPC listener on port `50051`. The runtime prints a checksum of the loaded graph (`SHA256: a3f2b...`) for audit logging.

2. **Run a Diagnostic Pipeline on a DICOM Series**  
   ```bash
   dentaiagent run --input-dir /data/cbct/session_20250315 --pipeline caries_segmentation --export-format nifti --sync-fhir
   ```
   The engine processes all `.dcm` files in the directory, applies the `dentai-v3` model with the configured tile stride, and writes segmentation masks to `/output/session_20250315/`. The `--sync-fhir` flag triggers a delta merge into the connected PostgreSQL FHIR store, updating the `Observation` resource with `performer=dentaiagent`.

3. **Monitor Agent Health & Adjust Circuit Breaker Parameters**  
   ```bash
   dentaiagent status --watch --circuit-breaker-threshold 10
   ```
   Opens a live TUI dashboard showing per-node latency (p50, p95, p99), cache hit ratio, and circuit breaker state (CLOSED/HALF_OPEN/OPEN). Use `Ctrl+C` to exit; the runtime persists the last known state to `/var/log/dentaiagent/state.json`.

### Best Practices

- **Pin GPU Memory for Latency-Sensitive Inference**  
  Set `CUDA_VISIBLE_DEVICES=0` and launch with `--device cuda:0 --cuda-malloc-heap 4G` to pre-allocate VRAM. This eliminates allocation jitter during multi-slice processing and reduces p99 latency by 22% in production benchmarks.

- **Use Staged FHIR Sync with Backpressure**  
  Configure `--sync-interval-ms 3000` and `--max-pending-records 500` to prevent the agent from overwhelming the FHIR server during bulk ingestion. Pair this with a PostgreSQL connection pool size of `10` (via `PGPOOL_SIZE` env var) to avoid deadlocks during concurrent delta writes.
