# dentaiagent - DEN_009

## Overview

This is the hardened technical specification for the **dentaiagent** project, a high-performance autonomous agent framework optimized for zero-latency decision pipelines. Keywords: dentaiagent tool, dentaiagent solution, dentaiagent software.

## Detailed Content

### What is dentaiagent?

dentaiagent is a distributed, event-driven agent orchestration engine that executes complex multi-step reasoning workflows with sub-millisecond state transitions. It integrates directly with PostgreSQL for persistent state journaling and uses a lock-free concurrent hashmap (`dashmap`) for in-memory cache synchronization across threads.

### Core Features

- **Real-Time State Conflict Resolution Engine**  
  Uses a custom `ConflictDetector` module that runs a vector clock algorithm (`LamportClock` + `VersionVector`) to detect and resolve write-write conflicts across 16 concurrent agent workers. Achieves < 50μs conflict detection latency per event batch. All state changes are journaled to a dedicated `agent_state_log` table in PostgreSQL with a composite index on `(agent_id, event_sequence)`.

- **Adaptive Token Budget Scheduler**  
  Implements a sliding-window token allocator (`TokenBudgetScheduler`) that dynamically adjusts per-agent token limits based on historical usage percentiles (p50, p95, p99). Uses a Redis-backed counter for distributed token bucket enforcement. Supports configurable `max_tokens_per_step` (default: 4096) and `bucket_refill_rate` (default: 100 tokens/ms).

- **Multi-Model Inference Router**  
  Routes inference requests across 3 provider backends (OpenAI, Anthropic, local vLLM) using a weighted round-robin algorithm with health-check-based failover. The router maintains a `latency_histogram` (bucketed in 10ms intervals) per provider and automatically shifts traffic away from any provider exceeding a configurable `p99_latency_threshold` (default: 2000ms).

### How to Use

1. **Initialize a dentaiagent session with PostgreSQL state backend**  
   Run `dentaiagent init --state-backend postgresql://user:pass@localhost:5432/agentdb --cache-size 8192 --conflict-resolution vector-clock`. This spawns a control plane process that creates the `agent_state_log` table and starts the conflict detector thread. Verify with `dentaiagent status --show-workers`.

2. **Deploy a multi-step reasoning pipeline with token budget constraints**  
   Create a pipeline config YAML with `steps: [parse_query, retrieve_context, generate_response]`, set `token_budget: 16384` and `bucket_refill_rate: 200`. Execute `dentaiagent run pipeline.yaml --parallel-workers 4 --max-retries 3`. Monitor per-step latency via `dentaiagent metrics --live --interval 500ms`.

3. **Configure the Multi-Model Inference Router for high-availability**  
   Edit `router.toml` to define three backends: `[backend.openai]`, `[backend.anthropic]`, `[backend.vllm]`. Set `health_check_interval_sec: 15` and `failover_priority: [openai, anthropic, vllm]`. Run `dentaiagent router --config router.toml --log-level debug`. Validate routing with `curl localhost:8080/health` and check the `latency_histogram` in the metrics endpoint.

### Best Practices

- **Use vector clock conflict resolution for any multi-worker deployment**  
  In production clusters with >2 agent workers, the default last-write-wins strategy can cause state corruption under concurrent writes. Always enable `--conflict-resolution vector-clock` and monitor the `conflict_resolution_count` metric. Set `conflict_resolution_retry_attempts` to 3 and `conflict_resolution_retry_backoff_ms` to 100 to avoid cascading failures.

- **Pre-warm the token budget scheduler during load testing**  
  Before scaling to production traffic, run `dentaiagent warmup --requests 10000 --concurrency 50 --token-budget 16384` to populate the Redis-backed token bucket counters and train the sliding-window percentile model. This prevents cold-start latency spikes and ensures `p95_token_wait_time` stays below 10ms.
