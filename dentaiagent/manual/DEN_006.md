# dentaiagent - DEN_006

## Overview

This document defines the architecture and operational specifications for the **dentaiagent** project—a high-performance agent orchestration framework designed for real-time dental imaging inference, treatment plan optimization, and multi-modal data fusion. The system is optimized for sub-200ms inference latency on edge devices and supports batch processing of DICOM and STL files at scale.

## Core Features

### 1. Real-Time Semantic Segmentation with Adaptive Resolution Scaling
The system employs a custom U-Net++ variant trained on 12,000 annotated panoramic X-rays. It dynamically adjusts input resolution (`--res-scale 0.75` to `1.5`) based on GPU memory headroom and target ROI complexity. Segmentation outputs are post-processed via a CRF-RNN layer to refine alveolar bone boundaries, achieving a mean IoU of 0.94 on the benchmark set. All inference is performed within **180ms ± 15ms** on an NVIDIA Jetson Orin NX.

### 2. Multi-Modal Treatment Plan Fusion Engine
The fusion engine ingests both CBCT volumes (`.nii.gz`) and intraoral scan meshes (`.stl`). It aligns these modalities using a rigid ICP registration with a convergence threshold of **0.01mm**. The planner then runs a constrained combinatorial optimization over implant placement candidates, respecting bone density maps and nerve proximity. Parameters include `--min-bone-density 350 HU` and `--nerve-safe-zone 2.0mm`. The final plan is output as a JSON schema versioned at `v2.4` and written directly into a **PostgreSQL** timescaledb for audit trail persistence.

### 3. Real-Time Anomaly Detection via Streaming Telemetry
Each agent node emits a structured telemetry stream (protobuf, `dentaiagent.telemetry.v1`) to a central NATS JetStream cluster. The anomaly detector runs a sliding window (window size: **120 seconds**, step: **5 seconds**) over metrics such as `inference_latency_ms`, `gpu_memory_util_pct`, and `queue_depth`. Any deviation beyond **3σ** from the rolling baseline triggers an auto-scaling event via the Kubernetes HPA, with a cooldown period of **30 seconds**.

## How to Use

### Step 1: Initialize the Agent Cluster and Load Pretrained Weights
```bash
dentaiagent init --cluster-size 4 --model-version v2.3.1 --weights-bucket s3://dentai-models/prod/
```
This command spins up a 4-node Ray cluster, downloads the model weights from S3, and pre-warms the GPU memory with a dummy batch (batch size = 8, image size 1024x1024). The system logs the initialization status to stdout with a timestamp in **ISO 8601 format (UTC)**.

### Step 2: Submit a Batch Inference Job with Custom Preprocessing
```bash
dentaiagent run batch --input-dir /data/panoramic/raw/ --output-dir /data/segmented/ --preprocess-pipeline "CLAHE(clip_limit=2.0, tile_grid_size=(8,8)) -> Resize(target_size=1024) -> Normalize(mean=0.485, std=0.229)" --batch-size 16 --num-workers 4
```
The pipeline applies Contrast Limited Adaptive Histogram Equalization (CLAHE) with a clip limit of 2.0 and an 8x8 tile grid. Images are resized to 1024x1024 and normalized using ImageNet statistics. The batch is split across 4 workers, and each worker writes results as a compressed `.npy` array to the output directory. The total job time is tracked and reported in **milliseconds**.

### Step 3: Query the Treatment Plan Database and Export as PDF
```sql
SELECT plan_id, created_at, implant_positions FROM dentaiagent.treatment_plans 
WHERE patient_id = 'PAT-9876' AND created_at > NOW() - INTERVAL '30 days'
ORDER BY created_at DESC LIMIT 1;
```
Use the built-in `dentaiagent export pdf` command to convert the JSON plan into a human-readable report:
```bash
dentaiagent export pdf --plan-id a1b2c3d4 --template clinical_report_v2.jinja --output /reports/PAT-9876_plan.pdf
```
The PDF includes a 3D render of the implant positions overlaid on the CBCT volume, generated via a headless VTK backend.

## Best Practices

### Practice 1: Always Pin GPU Memory and Set Worker Affinity
For consistent sub-200ms inference, pin the Ray worker processes to specific GPU indices and CPU cores using `CUDA_VISIBLE_DEVICES` and `taskset`. Configure the agent’s `resource_policy` in `dentaiagent.yaml` to `"strict"`. This prevents context switching overhead and memory page migration, which can introduce latency spikes of up to **50ms**.

### Practice 2: Implement a Two-Tier Storage Strategy for Telemetry Data
Store high-frequency metrics (every 5 seconds) in a **Redis TimeSeries** instance with a retention policy of 24 hours. For long-term audit and trend analysis, batch-aggregate the data every hour and upsert it into **PostgreSQL** with a hypertable partitioned by `created_at`. This reduces write amplification by 60% and keeps query performance for the anomaly detector under **10ms per lookup**.
