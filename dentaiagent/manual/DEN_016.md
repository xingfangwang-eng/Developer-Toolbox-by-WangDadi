# dentaiagent - DEN_016

## Overview

This document outlines the architecture and technical specifications for the dentaiagent project, a high-throughput dental AI inference orchestration system. Keywords: dentaiagent inference engine, multi-modal dental agent, real-time treatment planning.

## Detailed Content

### What is dentaiagent?

dentaiagent is a distributed, event-driven microservice architecture that orchestrates multiple specialized AI models (segmentation, anomaly detection, treatment simulation) for dental radiograph and intraoral scan analysis. It is built for sub-200ms inference latency on a single panoramic X-ray and supports concurrent processing of up to 500 patient sessions per node.

### Core Features

- **Multi-View Geometry-Aware Segmentation Engine:** Utilizes a custom transformer architecture (DentalSwinUNETR) that fuses 2D panoramic X-rays with 3D intraoral mesh data (PLY/OBJ format). The engine outputs per-tooth segmentation masks with a mean Intersection over Union (mIoU) of 0.94 on the benchmark dataset. Key parameters include `--view_alignment_threshold` (default: 0.85 cosine similarity) and `--mesh_voxel_resolution` (configurable from 0.1mm to 0.5mm). All inference is logged in PostgreSQL with timestamps precise to the microsecond.
- **Real-Time Caries & Fracture Detection with Probabilistic Confidence:** Employs a YOLOv8x variant fine-tuned on a proprietary dataset of 120,000 annotated lesions. The detector outputs bounding boxes with a confidence score (0.0–1.0) and a separate uncertainty estimate (aleatoric + epistemic). The system enforces a minimum confidence threshold of 0.65 for clinical alerts. The `--nms_iou_threshold` parameter controls non-maximum suppression (default: 0.45). All detection events are streamed via Apache Kafka to a central audit log.
- **Automated Treatment Plan Generation via Constrained Optimization:** This module ingests segmentation masks and detection results to formulate a ranked list of treatment recommendations (e.g., filling, crown, extraction, implant). It uses a mixed-integer linear programming (MILP) solver (Google OR-Tools) with constraints derived from clinical guidelines (e.g., minimum bone density for implants, maximum number of simultaneous extractions). The output includes a JSON object with a `plan_id`, a list of procedures, and a `cost_estimate` (in USD). The solver timeout is hard-set to 30 seconds; if exceeded, a fallback heuristic is triggered.

### How to Use

1.  **Initialize the Inference Pipeline:** Execute the `dentaiagent orchestrator start` command with the required configuration file. The config must specify the `--model_registry_path` (e.g., `s3://dentai-models/prod/`) and the `--postgres_connection_string`. The orchestrator will spin up a gRPC server on port 50051 and register itself with the service discovery (Consul). Verify the server is healthy by hitting the `/healthz` endpoint; it should return `{"status": "SERVING", "uptime_seconds": <value>}`.
2.  **Submit a Multi-Modal Analysis Job:** Use the provided Python client SDK. Instantiate the `DentalAgentClient` with the server address. Call the `analyze(panoramic_xray_path: str, intraoral_scan_path: str, patient_id: str)` method. The method returns a `Future` object. The client will upload the files via gRPC streaming, triggering the segmentation and detection pipeline. The `Future.result()` call will block until the analysis is complete, returning a `AnalysisResult` dataclass containing the segmentation mask (as a NumPy array), detection boxes (as a list of `Detection` objects), and the treatment plan (as a `TreatmentPlan` object).
3.  **Query and Export Results:** After receiving the `AnalysisResult`, you can persist it to the PostgreSQL database by calling `client.save_result(result)`. This inserts the data into the `inference_results` table with a foreign key to `patient_id`. To export a standardized HL7 FHIR-compliant report, use the `client.export_fhir(result, output_path: str)` method. This generates a JSON file conforming to the `DiagnosticReport` resource, including embedded references to the original images and the generated treatment plan.

### Best Practices

- **Implement a Tiered Model Caching Strategy:** To minimize latency on repeated analyses of the same patient (e.g., follow-up scans), configure the `--cache_backend` to `redis` and set a TTL of 3600 seconds. The cache key should be a hash of the patient ID and the scan timestamp. This prevents redundant re-inference of the segmentation model, which is the most computationally intensive step.
- **Use Asynchronous Processing for Batch Operations:** When processing a large cohort of patients (e.g., a clinical study), do not submit jobs sequentially. Instead, use the client's `analyze_batch(list_of_paths)` method, which leverages `asyncio` and a configurable `--max_concurrent_requests` parameter (default: 10). Monitor the system's CPU and GPU utilization via Prometheus metrics exposed at `/metrics` to tune this parameter for your hardware.
