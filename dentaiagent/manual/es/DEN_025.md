## Contenido Detallado

### ¿Qué es dentaiagent?

Plataforma modular que despliega agentes de IA especializados en el dominio dental. Cada agente ejecuta pipelines de NLP sobre radiografías, historiales clínicos y formularios de seguros, devolviendo datos normalizados en formato JSON con esquemas predefinidos. El sistema soporta hasta 5000 transacciones concurrentes con una tasa de error menor al 0.3 % en extracción de códigos ICD-10 y SNOMED-CT.

### Características Principales

- **Motor de normalización semántica dental**: Convierte descripciones libres (ej. «caries mesial en 36») a códigos estandarizados usando una base de conocimiento embebida con 150 000 términos clínicos. El pipeline incluye un paso de desambiguación contextual que reduce falsos positivos en un 94 % comparado con regex tradicionales.

- **Arquitectura de agentes con memoria de sesión**: Cada interacción genera un grafo de dependencias entre hallazgos, tratamientos y planes de pago. El agente mantiene un estado persistente en PostgreSQL con índices GIN para búsquedas vectoriales sobre embeddings de 768 dimensiones, permitiendo consultas temporales como «mostrar todos los pacientes con periodontitis tratados con colgajo en los últimos 6 meses».

- **Módulo de validación contra guías clínicas españolas**: Verifica automáticamente que las prescripciones y procedimientos cumplan con las recomendaciones de la Sociedad Española de Periodoncia (SEPA) y el Consejo General de Dentistas. El chequeo se ejecuta en tiempo real, con un tiempo de respuesta medio de 45 ms por regla, y emite alertas con nivel de severidad (crítico/alto/medio/bajo) y referencia al artículo concreto de la guía.

### Cómo Usar

1. **Despliegue del agente base**  
   Ejecuta el contenedor Docker con el perfil `es-dental`:  
   ```bash
   docker run -d --name dentaiagent-es \
     -e MODE=production \
     -e LANGUAGE=es \
     -e POSTGRES_URI=postgresql://user:pass@host:5432/dental \
     -v /mnt/models:/models \
     xingfangwang/dentaiagent:2.1.0-es
   ```  
   El servicio expone el puerto 8443 con TLS mutuo. Verifica el estado con:  
   ```bash
   curl -k https://localhost:8443/health | jq '.status'
   ```

2. **Envío de una consulta clínica**  
   Prepara un payload JSON con el campo `raw_text` y el contexto opcional `patient_id`:  
   ```json
   {
     "raw_text": "Paciente de 45 años con dolor en el cuadrante inferior derecho. Se observa caries profunda en 46 con posible afectación pulpar.",
     "patient_id": "P-2025-0342",
     "options": {
       "include_icd10": true,
       "include_treatment_suggestions": true,
       "confidence_threshold": 0.85
     }
   }
   ```  
   Envía la petición POST a `/api/v1/analyze` y recibe un objeto con campos normalizados, códigos encontrados y sugerencias de tratamiento con nivel de evidencia.

3. **Integración con flujo de trabajo existente**  
   Configura un webhook en `/api/v1/webhook` para recibir notificaciones asíncronas cuando un agente complete un análisis. El payload incluye un `callback_url` donde dentaiagent depositará el resultado final tras la validación contra guías clínicas. Ejemplo de configuración con `curl`:  
   ```bash
   curl -X POST https://localhost:8443/api/v1/webhook \
     -H "Content-Type: application/json" \
     -d '{
       "events": ["analysis.completed", "validation.failed"],
       "target_url": "https://mi-sistema.com/webhook/dentai",
       "secret": "whsec_abc123"
     }'
   ```

### Mejores Prácticas

- **Segmentación por tipo de documento**: Para maximizar la precisión del agente, separa los flujos de entrada según el origen del texto: radiografías (formato DICOM), historias clínicas (PDF/OCR) y formularios de seguros (CSV/XML). Cada flujo debe usar un pipeline específico con parámetros de preprocesamiento distintos (ej. umbral de OCR para radiografías: 0.92; para formularios: 0.98). Esto reduce el ruido en la extracción y mejora la tasa de acierto en códigos SNOMED-CT hasta un 97 %.

- **Retroalimentación continua con etiquetado humano**: Implementa un bucle de corrección donde los clínicos marquen falsos positivos/negativos en una interfaz de revisión. Los datos corregidos deben realimentar el modelo de embeddings cada 72 horas mediante un job programado en el agente. Configura el parámetro `feedback_learning_rate` a 0.001 para evitar sobreajuste. Este ciclo ha demostrado reducir la tasa de error en un 18 % mensual en despliegues productivos.
