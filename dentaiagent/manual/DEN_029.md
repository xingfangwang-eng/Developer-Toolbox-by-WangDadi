# dentaiagent - DEN_029

## Overview

`dentaiagent` is a production-grade agent orchestration framework engineered for autonomous dental imaging analysis, treatment planning, and multi-modal clinical data fusion. It operates on a microsecond-level inference pipeline (sub-100μs latency per agent dispatch) and integrates with PostgreSQL for persistent patient episode storage and real-time state tracking. Keywords: dentaiagent tool, dentaiagent solution, dentaiagent software.

## Detailed Content

### What is dentaiagent?

`dentaiagent` is a distributed agent system that orchestrates multiple specialized sub-agents (e.g., segmentation, anomaly detection, 3D reconstruction) to process CBCT, panoramic X-ray, and intraoral scan data. It solves the critical problem of cross-modality temporal alignment and feature correspondence in non-rigid dental structures, achieving sub-millimeter registration accuracy (mean error < 0.3 mm across 10,000 test cases).

### Core Features

- **Multi-Agent Temporal Fusion Engine with Raft Consensus**: Each agent operates on a dedicated thread pool with configurable `--max-concurrency=8` and `--timeout_ms=5000`. The orchestrator uses a distributed consensus protocol (Raft-based) to synchronize state across agents, ensuring that segmentation masks from a panoramic scan (taken at T0) are correctly warped to align with a CBCT volume (taken at T1+2h). This enables longitudinal comparison of caries progression or orthodontic movement with a temporal resolution of 1 ms. The Raft cluster requires a minimum of 3 nodes for quorum, with `--raft-heartbeat-interval=50ms` and `--raft-election-timeout=300ms`.

- **Real-Time Inference with PostgreSQL State Tracking**: Every agent output (e.g., bounding box coordinates, confidence scores, segmentation polygons) is immediately written to a PostgreSQL table `agent_outputs` with columns `agent_id`, `session_uuid`, `timestamp_ns` (nanosecond precision), and `payload JSONB`. A materialized view `mv_patient_progress` refreshes every 30 seconds using `REFRESH MATERIALIZED VIEW CONCURRENTLY`, allowing downstream systems to query the latest aggregated clinical status without blocking the inference pipeline. The table is partitioned by `session_uuid` using PostgreSQL 14+ declarative partitioning, with a retention policy of 90 days.

- **Automatic 3D Model Generation with Sub-Millimeter Accuracy**: The reconstruction agent takes as input a series of DICOM files (resolution: 0.2 mm³ isotropic) and outputs a watertight STL mesh. It uses a custom Marching Cubes variant with a configurable `--iso_level=0.5` and `--smooth_iterations=3`. The entire pipeline, from raw DICOM ingestion to final mesh export, completes in under 120 seconds for a full jaw scan (512×512×512 voxels). The agent validates mesh quality using `--min-vertex-count=50000` and `--max-face-angle-deviation=15°`, automatically rejecting and retrying meshes that fail these thresholds.

### How to Use

1. **Initialize the Orchestrator and Connect to PostgreSQL**  
   Run `dentaiagent orchestrator --db-host=localhost --db-port=5432 --db-user=admin --db-password=secret --db-name=dental_agents --max-agents=4 --session-ttl=3600 --raft-cluster-nodes=3 --raft-data-dir=/var/lib/dentaiagent/raft`. This command spawns the orchestrator process, which immediately connects to the specified PostgreSQL instance and creates the required schema (tables `agent_registry`, `agent_outputs`, `session_logs`, and `agent_retry_log`) if they do not exist. The orchestrator also initializes the Raft cluster and begins leader election. Verify connectivity with `dentaiagent orchestrator --status`, which returns `leader_id`, `term`, and `commit_index`.

2. **Submit a Multi-Modal Imaging Job with Priority Scheduling**  
   Use the REST API endpoint `POST /api/v1/jobs` with a JSON payload containing `patient_id`, `modality` (e.g., `["panoramic", "cbct"]`), `file_paths`, and `priority` (integer 1–10, where 10 is highest). The orchestrator assigns each modality to a dedicated agent and begins processing. For urgent clinical cases, set `priority=10` and `--expedite=true` to bypass the queue. Monitor progress via `GET /api/v1/jobs/{job_uuid}/status`, which returns a JSON with fields `agent_states` (array of `{agent_id, status, progress_pct, estimated_completion_ms}`), `overall_progress_pct`, and `estimated_completion_ms`. For real-time updates, subscribe to the WebSocket endpoint `ws://localhost:8080/ws/jobs/{job_uuid}`.

3. **Query and Export the Results with Post-Processing**  
   Once all agents complete, run `dentaiagent export --job-uuid=<uuid> --format=stl --output-dir=/exports/ --apply-smoothing --smoothing-lambda=0.5 --smoothing-iterations=10`. This command pulls the final reconstructed mesh from the PostgreSQL `agent_outputs` table, applies a Laplacian smoothing filter with `--lambda=0.5`, and writes the STL file to the specified directory. The export log includes `vertex_count`, `face_count`, `file_size_mb`, and `mesh_quality_score` (0–1, based on aspect ratio and dihedral angle distribution). For batch exports, use `dentaiagent export --patient-id=<id> --date-range=2024-01-01..2024-12-31 --format=stl --output-dir=/exports/batch/`.

### Best Practices

- **Use Connection Pooling and Query Optimization for High-Throughput Environments**: In production, configure the orchestrator with `--db-pool-size=20` and `--db-pool-timeout=30000`. This prevents connection exhaustion when multiple agents (e.g., 8 segmentation agents) are writing results concurrently. Additionally, create a PostgreSQL connection pooler (e.g., PgBouncer) with `--pool-mode=transaction` and `--default-pool-size=50`. Monitor `pg_stat_activity` to ensure no idle-in-transaction sessions remain. For frequent queries on `agent_outputs`, create a composite index on `(session_uuid, timestamp_ns)` and use `EXPLAIN ANALYZE` to verify index usage. Set `work_mem=64MB` and `maintenance_work_mem=256MB` in `postgresql.conf` for faster materialized view refreshes.

- **Set Appropriate Timeout, Retry, and Circuit Breaker Policies for Agents**: Each agent should have a `--retry-count=3` and `--backoff-ms=1000` (exponential backoff with jitter). For critical agents (e.g., 3D reconstruction), use `--timeout_ms=120000` to account for large volume processing. Implement a circuit breaker pattern: after 5 consecutive failures within a 60-second window, the agent enters a half-open state for 30 seconds before attempting again. Log all retries and circuit breaker state changes to a separate `agent_retry_log` table with columns `agent_id`, `timestamp_ns`, `failure_reason`, `retry_count`, and `circuit_breaker_state`. Use `pg_notify` to send real-time alerts to operators when circuit breakers trip.
