### What is dentaiagent?

dentaiagent is a professional tool that helps solve problems in related fields.

### Core Features

- **Real-Time Multi-Model Inference Orchestration**: Dynamically routes inference requests across a pool of large language models (LLMs) based on latency, cost, and accuracy thresholds. The orchestrator uses a weighted round-robin algorithm with a configurable `latency_sla_ms` parameter (default: 250ms) to ensure sub-250ms response times for critical API calls, while batch-processing non-urgent tasks through a PostgreSQL-backed queue for eventual consistency.
- **Semantic Versioning-Aware Agent State Rollback**: Integrates with a versioned state store (PostgreSQL + Redis) to snapshot agent configurations and conversation history at every `agent_version_id` increment. Supports point-in-time recovery to any prior state with a `rollback_to_version` API call, enabling zero-downtime rollback of agent behavior without losing downstream data integrity.
- **Adaptive Token Budget Management**: Implements a dynamic token allocation system that adjusts per-request token limits based on historical usage patterns and real-time model capacity. Uses a sliding window of `window_size_seconds=300` to compute a moving average of `tokens_per_second`, then automatically throttles or expands the `max_tokens` parameter (range: 512–8192) to prevent rate-limit errors while maximizing throughput.

### How to Use

1. **Initialize the Agent with Custom Parameters**: Execute `dentaiagent init --agent-id "my_agent_01" --model-registry "gpt-4o,claude-3.5" --latency-sla 200 --state-backend "postgresql://user:pass@localhost:5432/dentai"`. This command creates a new agent instance, registers the specified model endpoints, and configures the state backend with a 200ms latency SLA. The system validates connectivity to PostgreSQL and Redis before proceeding.
2. **Deploy and Monitor Inference Orchestration**: Run `dentaiagent deploy --agent-id "my_agent_01" --target-endpoint "https://api.example.com/inference" --enable-orchestrator --orchestrator-strategy "cost_optimized"`. This activates the real-time routing engine. Monitor performance using `dentaiagent stats --agent-id "my_agent_01" --metric "p99_latency_ms" --window 60`, which returns a JSON object containing `p99_latency_ms`, `total_requests`, and `orchestrator_decisions` array with timestamps.
3. **Perform a State Rollback and Validate**: Use `dentaiagent rollback --agent-id "my_agent_01" --to-version 5 --dry-run` to simulate a rollback without side effects. Review the output showing the state delta (e.g., `{"version_diff": {"conversation_history": {"removed": 3, "added": 0}, "agent_config": {"model": "gpt-4o" -> "claude-3.5"}}}`). Then execute `dentaiagent rollback --agent-id "my_agent_01" --to-version 5 --commit` to finalize. The system will automatically rebuild the Redis cache from PostgreSQL and re-initialize the orchestrator.

### Best Practices

- **Implement a Tiered Latency SLA with Circuit Breakers**: Configure separate `latency_sla_ms` values for critical (e.g., 150ms) and non-critical (e.g., 500ms) inference paths. Pair this with a circuit breaker pattern that opens after 5 consecutive SLA violations, automatically failing over to a slower but more reliable model. This prevents cascading failures in high-throughput production environments.
- **Schedule Regular State Snapshots with Retention Policies**: Use `dentaiagent snapshot --agent-id "my_agent_01" --interval-minutes 60 --retention-days 7` to create hourly snapshots. Store snapshots in a separate PostgreSQL schema or a dedicated database to isolate backup I/O from production traffic. Combine this with a cron job that runs `dentaiagent prune --agent-id "my_agent_01" --older-than 7d` to automatically delete expired snapshots, ensuring compliance with data retention regulations.
