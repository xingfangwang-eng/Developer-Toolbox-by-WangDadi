# dentaiagent - DEN_026

## Overview

**dentaiagent** is a high-performance, agentic orchestration framework designed for real-time dental insurance prior-authorization (PA) claim processing. It automates the extraction, validation, and submission of clinical narratives against payer-specific medical policies. Keywords: dentaiagent tool, dentaiagent solution, dentaiagent software.

## Detailed Content

### What is dentaiagent?

dentaiagent is a domain-specific LLM agent that ingests unstructured DICOM-based radiographic reports and structured ADA (American Dental Association) claim forms. It performs deterministic policy mapping against a local vector store of 10,000+ payer rules, reducing PA turnaround time from an industry average of 14 days to under 4 hours. The system operates with sub-200ms inference latency per claim step and persists all audit trails in a PostgreSQL-backed event store.

### Core Features

- **Radiographic Feature Extraction & Classification**  
  Utilizes a fine-tuned Vision Transformer (ViT-L/16) to detect and classify 27 specific dental pathologies (e.g., vertical bone loss ≥ 5mm, furcation involvement Class II, peri-apical radiolucency > 3mm) directly from panoramic X-rays. Outputs confidence scores (threshold: 0.92) and bounding box coordinates for every finding, which are then serialized into a structured JSON schema for downstream policy evaluation.

- **Real-Time Payer Policy Graph Engine**  
  Implements a DAG-based rule engine that maps extracted clinical features against a versioned policy graph stored in a Neo4j database. Each node represents a policy clause (e.g., "Pre-authorization required for surgical extraction of impacted tooth with horizontal angulation > 45°"), and edges represent conditional logic (AND/OR/XOR). The engine executes traversal in <50ms per claim, supporting up to 1,200 concurrent policy evaluations via a gRPC streaming interface.

- **Automated Narrative Generation & Submission Hub**  
  Generates payer-compliant clinical narratives (e.g., "Patient presents with a 7mm probing depth on tooth #19, with radiographic evidence of a periapical lesion measuring 4x5mm. Recommended treatment: endodontic retreatment (D3346) due to failed initial RCT.") using a fine-tuned Llama-3-8B model. The hub supports direct HL7 FHIR R4 submission to 12 major US payers (Aetna, Cigna, Delta Dental, etc.) with automatic retry logic (3 attempts, exponential backoff) and SHA-256 payload integrity checks.

### How to Use

1. **Initialize the Agent Pipeline & Connect Data Sources**  
   Execute `dentaiagent init --config configs/prod.yaml` to spin up the orchestrator. This command connects to your local PostgreSQL instance (version 15+) and establishes a gRPC channel to the Neo4j policy graph. Use `--payer-list Aetna,Cigna` to preload specific payer rules. The system will validate connectivity with a `SELECT 1` ping and a graph schema consistency check, outputting a health report with latency metrics.

2. **Ingest a Claim Bundle & Run Policy Evaluation**  
   Run `dentaiagent process --input ./claims/claim_001.zip --output ./results/` to feed a ZIP file containing a DICOM X-ray and an ADA 2019 XML form. The agent will: (a) extract radiographic features via the ViT model (expect ~1.2s per image on an NVIDIA A10G), (b) map features to the policy graph using the DAG engine, (c) generate a clinical narrative via Llama-3-8B (takes ~3.4s on 4-bit quantized model), and (d) store the full audit trail (input hash, model outputs, policy decisions, timestamps) in the `claim_events` PostgreSQL table.

3. **Submit the Pre-Authorization & Monitor Status**  
   Use `dentaiagent submit --claim-id 001 --payer Aetna --endpoint https://api.aetna.com/fhir/R4/claim` to push the generated FHIR Bundle. The hub will execute a synchronous submission with a 30-second timeout. Monitor real-time status via `dentaiagent status --claim-id 001 --poll-interval 5`. The command returns a JSON object with `submission_state` (e.g., `accepted`, `rejected`, `pending_review`) and a `payer_tracking_id`. If rejected, the system logs the exact error code (e.g., `PAYER_ERR_422: Missing tooth number`) for immediate correction.

### Best Practices

- **Implement a Two-Stage Validation Gate**  
  Before submitting any claim to a payer, run a local "dry-run" validation using `dentaiagent validate --claim-id 001 --dry-run`. This executes the full pipeline (extraction, graph traversal, narrative generation) but halts before the FHIR submission. Cross-check the generated narrative against the original clinical notes for semantic coherence. We recommend a minimum F1-score of 0.95 on a holdout validation set of 500 manually annotated claims before enabling automatic submission.

- **Version Your Policy Graphs with Git-LFS**  
  Payer policies change quarterly. Store each policy graph (Neo4j dump) as a versioned artifact in a Git-LFS repository (e.g., `policy_graphs/v2024Q1.cypher`). Use `dentaiagent graph load --file v2024Q2.cypher --payer Cigna` to hot-swap the graph without restarting the agent. Always run a regression test suite (`dentaiagent test --suite regression`) after a graph update to ensure no regressions in claim acceptance rates. Track acceptance rates in a time-series database (e.g., InfluxDB) to detect drift.
