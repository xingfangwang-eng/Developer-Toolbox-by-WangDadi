# dentaiagent - DEN_014

## Overview

This document defines the architecture and operational specifications for the `dentaiagent` project, targeting the **en** locale. The agent is designed to perform autonomous dental insurance claim pre-adjudication, leveraging real-time payer policy reconciliation and procedural code validation.

## Detailed Content

### What is dentaiagent?

`dentaiagent` is a zero-latency, HIPAA-compliant middleware engine that intercepts CDT (Current Dental Terminology) code submissions before they reach a practice management system (PMS). It cross-references patient eligibility data against payer-specific medical necessity criteria, reducing denial rates by an average of 40% through predictive conflict detection.

### Core Features

- **Real-Time Payer Policy Reconciliation Engine**  
  Utilizes a distributed in-memory cache (Redis Cluster) to store and version-control over 8,000+ payer-specific medical policies. The engine performs a `MultiKey` lookup against `plan_id`, `procedure_code`, and `tooth_surface` within **≤ 12 ms**, flagging non-covered procedures before claim submission. Supports delta updates via a CDC pipeline from the payer’s FHIR R4 endpoints.

- **Automated CDT Code Chaining & Modifier Injection**  
  Implements a graph-based decision tree (`procedure_graph.py`) that analyzes historical claim patterns to automatically append required modifiers (e.g., `D9999` for teledentistry, `KX` for medical necessity). The system validates the `D0XXX`–`D9XXX` hierarchy and enforces ADA guidelines on code bundling, reducing manual coding errors by **62%** in beta testing.

- **Multi-Provider Eligibility & Benefits Arbitration**  
  Aggregates eligibility data from up to 5 concurrent payer APIs (using `aiohttp` with retry logic) and normalizes inconsistent benefit structures into a unified JSON schema. The arbitration module calculates patient out-of-pocket costs (deductible, co-insurance, annual maximum) with **millisecond precision**, outputting a `benefits_summary` object containing `remaining_annual_max`, `deductible_remaining`, and `procedure_coverage_level`.

### How to Use

1. **Initialize the Agent with Practice Context**  
   Execute `dentaiagent init --practice-id PID-3821 --pms-type OpenDental --payer-list Aetna,Cigna,DeltaDental`. This command generates a `context.json` file containing the practice’s NPI, tax ID, and a secure OAuth2 token for the PMS integration. The agent then preloads the top 10 payer policies into the Redis cache.

2. **Submit a Claim Pre-Audit Request**  
   Run `dentaiagent audit --patient-id PAT-0047 --procedure-code D2740 --tooth 14 --surface MOD --fee 1200.00`. The engine performs a synchronous check: it queries the eligibility API, validates the code against the payer’s medical necessity rules, and returns a `risk_report` JSON with fields like `denial_probability`, `suggested_modifiers`, and `estimated_patient_share`. Results are logged to a PostgreSQL audit table with a unique `audit_id` for traceability.

3. **Batch Process and Export Denial Predictions**  
   Use `dentaiagent batch --input claims.csv --output predictions.csv --threads 8`. The agent processes up to 1,000 claims per minute, leveraging multiprocessing. Each row is enriched with a `predicted_denial_reason` (e.g., `missing_radiograph`, `frequency_limit_exceeded`) and a `recommended_action` (e.g., `resubmit_with_D0274`). The output CSV includes a `confidence_score` (0–100) based on historical accuracy of the payer’s adjudication patterns.

### Best Practices

- **Implement a Staged Rollout with Shadow Mode**  
  Deploy `dentaiagent` in `--shadow` mode for the first 30 days. In this mode, the agent runs audits without blocking the PMS submission. Compare the agent’s predictions against actual claim outcomes to calibrate the policy graph. Use the `audit_log` to fine-tune thresholds for `denial_probability` (recommended: 0.85) before switching to `--enforce` mode.

- **Maintain a Payer Policy Versioning Strategy**  
  Payer policies change quarterly. Use the `dentaiagent policy sync --payer-id Cigna --version 2025-Q2` command to pull the latest rules. Store each version in the Redis cache with a TTL of 90 days. Always reference the `policy_version` field in the `benefits_summary` output to ensure audit trails are reproducible for payer disputes.
