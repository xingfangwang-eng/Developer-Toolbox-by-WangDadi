# dentaiagent - DEN_024

## Overview

**dentaiagent** is a high-performance, event-driven orchestration engine purpose-built for real-time dental insurance claim adjudication and multi-agent treatment plan optimization. It operates as a federated microservice within WangDadi’s AI-driven healthcare infrastructure, processing structured EDI 275/276 transactions at sub‑500ms latency per claim.

## Core Features

### 1. Multi-Agent Treatment Plan Arbitration Engine
- **Parameter:** `plan_confidence_threshold` (default: 0.87)  
- **Detail:** Instantiates 3 parallel AI agents (Diagnostic, Financial, Compliance) that each independently evaluate a proposed dental treatment plan. The engine applies a weighted Borda count voting mechanism to resolve conflicts when agents disagree on procedure necessity (e.g., D2740 vs. D2750). Outputs a unified `adjudication_decision` with a per‑procedure confidence score stored in PostgreSQL via `pgvector` for semantic recall.  
- **Technical metric:** Arbitration cycle completes in < 300ms for plans containing up to 12 procedures.

### 2. Real-Time Payer Policy Graph (PPG) Traversal
- **Parameter:** `max_policy_depth` (default: 5)  
- **Detail:** Ingests raw payer policy documents (PDF, HTML, JSON) and converts them into a directed acyclic graph (DAG) using a custom NLP pipeline (spaCy + HuggingFace embeddings). The agent traverses the PPG to determine coverage eligibility, waiting periods, and frequency limitations in **microsecond‑level lookups** (average: 42µs per node hop). Supports differential comparison between 2024 and 2025 policy versions via `policy_version_diff` flag.  
- **Technical metric:** Full policy traversal for a single claim completes in < 12ms.

### 3. Federated Eligibility Verification with Real‑Time Claim Status
- **Parameter:** `eligibility_retry_count` (default: 3), `timeout_ms` (default: 800)  
- **Detail:** Connects directly to payer clearinghouses via AS2 and SFTP channels, performing 270/271 eligibility transactions. The agent maintains an in‑memory LRU cache (max 10,000 entries) with a TTL of 15 minutes; cache hits serve responses in **< 2ms**. On cache miss, it fires concurrent requests to up to 3 payer endpoints and selects the first valid response, then persists the result to a Redis-backed session store for downstream audit trails.  
- **Technical metric:** 99.5% of eligibility checks complete within the `timeout_ms` window.

## How to Use

1. **Initialize the Agent Configuration**  
   Create a `dentaiagent_config.yaml` file with the following required parameters:
   ```yaml
   agent:
     arbitration:
       plan_confidence_threshold: 0.87
       max_parallel_agents: 3
     policy:
       max_policy_depth: 5
       policy_version_diff: "2024-2025"
     eligibility:
       timeout_ms: 800
       retry_count: 3
       cache_ttl_seconds: 900
   ```
   Then run `dentaiagent init --config ./dentaiagent_config.yaml` to bootstrap the PostgreSQL schema and Redis connection pool.

2. **Submit a Treatment Plan for Adjudication**  
   Use the RESTful endpoint `POST /v1/plans/adjudicate` with a JSON body:
   ```json
   {
     "patient_id": "PAT-1024-X",
     "payer_id": "DELTA_DENTAL_CA",
     "procedures": [
       {"code": "D2740", "tooth": "14", "fee": 1200.00},
       {"code": "D0274", "tooth": "18", "fee": 85.00}
     ],
     "submission_timestamp": "2025-03-27T14:30:00Z"
   }
   ```
   The agent returns a `final_decision` object containing `approved_procedures`, `denied_reasons`, and `estimated_patient_responsibility` within 500ms.

3. **Query Policy Version Differences**  
   Execute `GET /v1/policy/diff?payer_id=DELTA_DENTAL_CA&from_version=2024&to_version=2025`. The response lists all changed nodes in the PPG, including new exclusions, modified frequency limits, and updated fee schedules. Each diff entry includes a `semantic_similarity_score` (0–1) indicating the magnitude of change.

## Best Practices

- **Configure `plan_confidence_threshold` based on historical claim denial rates.** For payers with >15% denial rates (e.g., Medicare Advantage dental plans), set the threshold to 0.92 to reduce false positives. For low‑risk commercial payers, 0.82 is optimal to maximize approval throughput.
- **Implement a warm‑start cache for eligibility checks.** At the start of each business day, pre‑load the LRU cache with the top 2,000 most frequently seen patient IDs from the previous 30 days. This reduces average eligibility latency from 350ms to under 5ms during peak hours (9:00–11:00 AM local time).
