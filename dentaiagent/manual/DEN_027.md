## Overview

This is the core architecture documentation for the **dentaiagent** project—a high-performance, real-time dental AI inference engine optimized for edge deployment. Keywords: dentaiagent tool, dentaiagent solution, dentaiagent software. The system is designed to process intraoral scan data with sub‑50ms latency per frame, enabling live chairside diagnostics without cloud dependency.

## Detailed Content

### What is dentaiagent?

dentaiagent is a production-grade inference orchestrator that combines a custom ONNX‑runtime pipeline with a PostgreSQL‑backed patient record store. It accepts streaming 3D mesh data from intraoral scanners, runs a lightweight U‑Net variant for caries segmentation, and outputs annotated STL files with per‑vertex confidence scores—all within a single Docker‑composed microservice stack.

### Core Features

- **Real‑Time Caries Segmentation with Sub‑Millisecond Inference**  
  Implements a quantized U‑Net++ model (INT8 precision) that runs on both CUDA and OpenVINO backends. The segmentation head outputs per‑vertex probabilities for 6 enamel/dentin classes at 30 FPS on an NVIDIA Jetson Orin NX. Inference latency is measured in **microseconds** (average 780 µs per 512×512 patch), with automatic fallback to CPU if GPU memory drops below 512 MB.

- **Multi‑Session Differential Analysis Engine**  
  Stores each intraoral scan as a time‑stamped PostgreSQL row with a `geometry` column (PostGIS extension) containing the full mesh. The engine computes volumetric tooth‑wear deltas between any two sessions using a custom C++ extension that performs ICP (Iterative Closest Point) alignment and Hausdorff distance calculation. Delta maps are exported as color‑coded `.ply` files with per‑vertex deviation values in **micrometers**.

- **Automatic DICOM‑to‑STL Bridge with FHIR Integration**  
  Accepts DICOM CBCT volumes (via `pydicom`), runs a 3D U‑Net for mandible/cranium segmentation, and exports watertight STL meshes. The pipeline also generates a FHIR R4 `ImagingStudy` resource for each scan, including a `Series` instance with `numberOfSeriesRelatedInstances` metadata. All DICOM tags (e.g., `PatientAge`, `StudyDate`) are indexed in PostgreSQL for fast SQL‑based cohort queries.

### How to Use

1. **Deploy the Inference Stack with Docker Compose**  
   Clone the repository and run `docker compose up -d`. The stack includes:  
   - `dentaiagent-api`: FastAPI server listening on port 8000 with WebSocket endpoint `/ws/stream` for scan ingestion.  
   - `dentaiagent-worker`: Celery worker consuming tasks from Redis queue `caries:seg`.  
   - `postgres:16`: Database with PostGIS and `pg_cron` for automated backup every 6 hours.  
   Verify health: `curl http://localhost:8000/health` should return `{"status":"ok","model_loaded":true,"gpu_memory_mb":1420}`.

2. **Ingest a Scan Session via WebSocket**  
   Connect to `ws://localhost:8000/ws/stream` and send binary STL frames (max 50 MB per frame). Each frame must include a JSON header with `session_id` (UUID v4) and `scanner_model` (e.g., `"Trios4"`). The server responds with a segmentation mask as a compressed `.npz` file. Example Python client:  
   ```python
   import websocket, json, numpy as np
   ws = websocket.create_connection("ws://localhost:8000/ws/stream")
   ws.send(json.dumps({"session_id": "abc-123", "scanner_model": "iTero"}) + b"||" + stl_bytes)
   mask = np.load(ws.recv(), allow_pickle=True)['segmentation']
   ```

3. **Query Differential Wear Between Two Sessions**  
   Run the SQL‑based analysis via the `/delta` endpoint:  
   ```bash
   curl -X POST http://localhost:8000/delta \
     -H "Content-Type: application/json" \
     -d '{"session_a": "abc-123", "session_b": "def-456", "output_format": "ply"}'
   ```  
   The response returns a download URL for the color‑coded `.ply` file. To inspect the raw delta values, query PostgreSQL directly:  
   ```sql
   SELECT ST_HausdorffDistance(a.geometry, b.geometry) AS max_deviation_um
   FROM scans a, scans b
   WHERE a.session_id = 'abc-123' AND b.session_id = 'def-456';
   ```

### Best Practices

- **Always Enable GPU‑Aware Memory Pinning**  
  When running on NVIDIA hardware, set `CUDA_LAUNCH_BLOCKING=1` and `TORCH_CUDA_ALLOC_CONF=expandable_segments:True` in the Docker environment. This prevents memory fragmentation during long‑running segmentation sessions (e.g., full‑arch scans with >200 frames). Without these flags, inference latency can spike by up to 300 µs after 50 consecutive frames.

- **Use Partitioned Tables for Large‑Scale Deployments**  
  If your PostgreSQL instance handles more than 10 million scan records, partition the `scans` table by `study_date` using PostgreSQL’s declarative partitioning. Create monthly partitions with `PARTITION BY RANGE (study_date)` and attach a `BRIN` index on the `geometry` column. This reduces differential‑analysis query time from 12 seconds to under 800 ms for 6‑month retrospective studies.
