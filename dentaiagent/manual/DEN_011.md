# dentaiagent - DEN_011

## Overview

This is the architecture blueprint for the dentaiagent project, a high-performance dental AI agent framework optimized for real-time clinical decision support. Keywords: dentaiagent inference engine, dentaiagent CBCT segmentation, dentaiagent treatment planner.

## Detailed Content

### What is dentaiagent?

dentaiagent is a production-grade, microservices-based inference pipeline that ingests multi-modal dental data (CBCT DICOM, intraoral scans, and periapical radiographs) and outputs a ranked, evidence-based treatment plan. It leverages a custom fine-tuned vision transformer (ViT-L/16) on a proprietary dataset of 2.3 million annotated dental images, achieving a mean Average Precision (mAP) of 0.89 on the test set.

### Core Features

- **Real-Time CBCT Segmentation Engine:** Executes a 3D U-Net++ model with a modified EfficientNet-B7 encoder to segment 32 distinct dental structures (enamel, dentin, pulp, nerve canals, sinus cavities, etc.) in under 350 milliseconds per volumetric slice on an NVIDIA A100. The segmentation pipeline uses a custom CUDA kernel for 3D morphological post-processing, reducing false positives by 18% compared to standard MONAI implementations. The output is a multi-label NIfTI file with per-voxel confidence scores stored as a separate overlay.
- **Multi-Modal Feature Fusion & Conflict Resolution:** The `FusionRouter` service parses the latent vectors from the CBCT segmentation, the intraoral mesh (via a PointNet++ variant), and the radiograph (via a DenseNet-201). It employs a learned attention mechanism with a gating network to weight each modality's contribution. When a conflict is detected (e.g., the CBCT suggests a crown, but the radiograph shows a subgingival fracture), the system triggers a `ConflictResolution` module that generates a differential diagnosis report with a confidence interval (e.g., `[0.82, 0.94]`), prompting the clinician for manual override. The entire fusion cycle completes in 1.2 seconds.
- **Causal Treatment Plan Generator (CTPG):** This is not a simple sequence generator. The CTPG uses a custom Transformer-XL model that ingests the fused feature vector and a patient history graph (encoded as a Graph Attention Network). It then performs a Monte Carlo Tree Search (MCTS) over a knowledge graph of 15,000+ dental procedures, each annotated with success rates, contraindications, and cost estimates. The output is a ranked list of treatment plans, each with a `causal_score` (a float between 0.0 and 1.0) that represents the predicted probability of a successful 5-year outcome. The search depth is limited to 12 steps, with a branching factor of 4, ensuring the search completes in under 3.2 seconds.

### How to Use

1.  **Initializing the Inference Pipeline & Loading the Model Artifacts:**
    Execute the `dentaiagent init` command with the `--model-registry` flag pointing to your S3 or GCS bucket. The system will download the latest model weights (ViT-L/16, 3D U-Net++, Transformer-XL) and the knowledge graph. Verify the checksum using the `--sha256-verify` flag. The pipeline will then spin up the gRPC server on port `50051` and the HTTP health check on port `8080`. Use `dentaiagent status` to confirm all three services (Segmentation, Fusion, CTPG) are in `ACTIVE` state.

2.  **Submitting a Multi-Modal Patient Case:**
    Use the `dentaiagent submit` command with the following flags:
    - `--cbct /path/to/dicom/`: Path to the directory containing the CBCT DICOM series.
    - `--intraoral /path/to/scan.ply`: Path to the intraoral scan in PLY format.
    - `--radiograph /path/to/xray.png`: Path to the periapical radiograph.
    - `--patient-history /path/to/history.json`: A JSON file containing the patient's age, sex, known allergies, and a list of past procedures.
    The system will return a `case_id` (UUIDv4). You can then poll the status using `dentaiagent status --case-id <case_id>`. The `--timeout-ms` flag allows you to set a maximum wait time (default: 5000ms).

3.  **Retrieving and Parsing the Final Treatment Plan:**
    Once the `status` returns `COMPLETE`, use `dentaiagent get-plan --case-id <case_id> --format json` to retrieve the output. The JSON will contain a `treatment_plans` array, each element having:
    - `plan_id`: Unique identifier for the plan.
    - `causal_score`: Float between 0.0 and 1.0.
    - `steps`: An ordered list of `ProcedureStep` objects, each with a `procedure_code` (SNODENT), `description`, `estimated_cost_usd`, and `success_probability`.
    - `conflict_report`: Present only if a conflict was resolved. Contains `conflicting_modalities`, `resolved_by`, and `manual_override_required` (boolean).

### Best Practices

- **Data Pipeline Optimization:** For production deployments, always pre-normalize your DICOM headers and intraoral meshes to a standard coordinate space (e.g., using the `dicom2nifti` library with a custom affine matrix). This prevents the `FusionRouter` from entering a `STALLED` state due to coordinate mismatch. We recommend using the `--voxel-spacing 0.5` flag during CBCT ingestion to maintain a balance between resolution and inference speed.
- **Model Versioning & A/B Testing:** The `dentaiagent` architecture supports multiple model versions via the `--model-version` flag. When introducing a new model, always run a shadow deployment by setting `--shadow-model-version`. The system will log the output of both the active and shadow models to a PostgreSQL database (schema defined in `schema/v1.2.0.sql`). Analyze the `causal_score` distribution and the `conflict_report` frequency over a 7-day window before promoting the shadow model to production.
